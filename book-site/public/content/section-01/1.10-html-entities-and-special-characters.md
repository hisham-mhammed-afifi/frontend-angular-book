# 1.10 HTML Entities and Special Characters

> HTML entities are coded representations of characters that either cannot be typed directly, would be misinterpreted by the parser, or are invisible and need to be self-documenting in source code.

---

## Character Reference Syntax

There are three ways to represent a character in HTML without typing it directly:

| Form | Syntax | Example | Result |
|------|--------|---------|--------|
| Named entity | `&name;` | `&amp;` | & |
| Decimal numeric | `&#nnnn;` | `&#38;` | & |
| Hexadecimal numeric | `&#xHHHH;` | `&#x26;` | & |

All character references start with `&` and end with `;`.

### Named Entities

HTML5 defines over 2,000 named entities (up from 252 in HTML4). Named entities are case-sensitive: `&Eacute;` produces E while `&eacute;` produces e.

```html
&copy;   <!-- (c) -->
&trade;  <!-- TM -->
&mdash;  <!-- --- -->
```

### Numeric References

Decimal and hexadecimal forms reference Unicode codepoints directly. The hex prefix `&#x` is case-insensitive, and leading zeros are optional.

```html
<!-- All three produce the Euro sign -->
&euro;      <!-- named -->
&#8364;     <!-- decimal (U+20AC = 8364 in decimal) -->
&#x20AC;    <!-- hex -->
&#x20ac;    <!-- hex, lowercase also works -->
```

Numeric references work for any Unicode codepoint, including characters that have no named entity (most emoji, rare symbols, CJK characters).

```html
&#x1F600;   <!-- grinning face emoji -->
&#128512;   <!-- same emoji in decimal -->
```

### Always Include the Semicolon

The semicolon is technically optional in some HTML parsing contexts, but omitting it creates "ambiguous ampersands" that parse inconsistently between browsers and break in XHTML.

```html
<!-- BAD: ambiguous -->
&amp without semicolon
&copy2025

<!-- GOOD: unambiguous -->
&amp; with semicolon
&copy;2025
```

## The Five Reserved Characters

Five characters have special meaning in HTML syntax. Using them literally in certain contexts will break the parser.

| Character | Named | Decimal | Hex | Why It Must Be Escaped |
|-----------|-------|---------|-----|----------------------|
| `&` | `&amp;` | `&#38;` | `&#x26;` | Starts entity references |
| `<` | `&lt;` | `&#60;` | `&#x3C;` | Opens HTML tags |
| `>` | `&gt;` | `&#62;` | `&#x3E;` | Closes HTML tags |
| `"` | `&quot;` | `&#34;` | `&#x22;` | Terminates double-quoted attributes |
| `'` | `&apos;` | `&#39;` | `&#x27;` | Terminates single-quoted attributes |

### Context-Dependent Rules

Not all five need escaping in every context:

- **`&`** must always be escaped in HTML content and attributes. Even in URLs inside `href`:

```html
<!-- BAD: &b starts an entity reference -->
<a href="page?a=1&b=2">Link</a>

<!-- GOOD: ampersand escaped -->
<a href="page?a=1&amp;b=2">Link</a>
```

- **`<`** must always be escaped in text content. A bare `<` is interpreted as a tag opening.

- **`>`** is technically safe unescaped in most text content, but must be escaped after `]]` to avoid the CDATA end sequence `]]>`. Best practice: always escape it.

- **`"`** only needs escaping inside double-quoted attribute values:

```html
<p title="He said &quot;hello&quot;">Text</p>
```

- **`'`** only needs escaping inside single-quoted attribute values. Note that `&apos;` was not part of HTML4's named entities (it came from XML). HTML5 adopted it, but `&#39;` is the safer choice for maximum compatibility.

### Security Implications

These five characters are the primary defense against Cross-Site Scripting (XSS). Encoding `<` prevents tag injection. Encoding `"` and `'` prevents attribute breakout. Server-side template engines should encode these automatically in all output contexts.

```html
<!-- Without encoding, user input could inject a script tag -->
<!-- User submits: <script>alert('xss')</script> -->

<!-- BAD: raw output -->
<p><script>alert('xss')</script></p>

<!-- GOOD: encoded output -->
<p>&lt;script&gt;alert(&#39;xss&#39;)&lt;/script&gt;</p>
```

## Entities vs Direct Unicode Characters

With UTF-8 encoding (used by 98%+ of the web), most characters can be typed directly. When should you use entities instead?

| Scenario | Entity | Direct Character | Rationale |
|----------|--------|-----------------|-----------|
| Reserved chars (`<`, `>`, `&`, `"`, `'`) | Always | Never | Parser would misinterpret |
| Invisible chars (nbsp, soft hyphen, zero-width) | Always | Avoid | Entities are self-documenting in source |
| Common symbols (copyright, trademark) | Either | Either | Entity aids readability |
| Typographic punctuation (em dash, curly quotes) | Either | Either | Direct is shorter |
| Emoji and CJK characters | Numeric ref | Direct | No named entities for most |
| Characters not on your keyboard | Numeric ref | Paste | Refs work regardless of editor encoding |

### File Encoding Mismatch

If you type a direct em dash character but your file is saved as Latin-1 instead of UTF-8, browsers expecting UTF-8 will show garbled text (mojibake). Entities always work regardless of file encoding because they use only ASCII characters.

```html
<!-- Safe regardless of file encoding -->
<p>Price&mdash;$50</p>

<!-- Only works if file is saved as UTF-8 -->
<p>Price---$50</p>
```

## Non-Breaking Space (`&nbsp;`)

The most commonly used (and misused) HTML entity. Unicode U+00A0.

`&nbsp;` does two things:
1. Renders as a space of normal width.
2. Prevents a line break at that position.

### Legitimate Use Cases

```html
<!-- Prevent break between value and unit -->
<p>The package weighs 2.5&nbsp;kg and measures 30&nbsp;cm.</p>

<!-- Keep titles and names together -->
<p>Contact Dr.&nbsp;Smith for details.</p>

<!-- Keep time formats together -->
<p>The meeting starts at 2:30&nbsp;PM.</p>

<!-- Keep short prepositions with the next word (typographic convention) -->
<p>This is a guide to&nbsp;HTML.</p>
```

### Bad Practice: Using `&nbsp;` for Layout

```html
<!-- BAD: using entities for visual spacing -->
<p>&nbsp;&nbsp;&nbsp;&nbsp;Indented paragraph</p>

<!-- GOOD: use CSS -->
<p style="text-indent: 2em;">Indented paragraph</p>
```

```html
<!-- BAD: spacing between elements -->
<button>Save</button>&nbsp;&nbsp;&nbsp;<button>Cancel</button>

<!-- GOOD: use CSS gap -->
<div style="display: flex; gap: 12px;">
  <button>Save</button>
  <button>Cancel</button>
</div>
```

### Gotchas

- `&nbsp;` prevents line breaks. On narrow screens, content joined by many `&nbsp;` entities can overflow its container and cause horizontal scrolling.
- Some screen readers announce "non-breaking space" instead of silently pausing, creating awkward output.
- Multiple consecutive `&nbsp;` entities are not collapsed by the browser (unlike regular spaces), which is why people use them for indentation. But this is fragile, non-semantic, and should always be replaced with CSS.

## The Full Space Character Family

HTML supports multiple space characters with different widths and behaviors:

| Space Type | Entity | Unicode | Width | Breaks? |
|-----------|--------|---------|-------|---------|
| Normal space | (space key) | U+0020 | Regular | Yes |
| Non-breaking space | `&nbsp;` | U+00A0 | Regular | No |
| En space | `&ensp;` | U+2002 | Width of "n" / half em | No |
| Em space | `&emsp;` | U+2003 | Width of "M" / full em | No |
| Thin space | `&thinsp;` | U+2009 | ~1/5 em | No |
| Hair space | `&#8202;` | U+200A | Thinnest visible space | No |
| Figure space | `&#8199;` | U+2007 | Width of a digit | No |
| Punctuation space | `&#8200;` | U+2008 | Width of a period | No |
| Narrow no-break space | `&#8239;` | U+202F | Narrower than nbsp | No |

### Practical Uses

```html
<!-- Thin space between number and unit (French typographic convention) -->
<p>100&thinsp;km/h</p>

<!-- Figure space for aligning tabular numbers -->
<pre>
  &#8199;42
  &#8199;&#8199;7
  100
</pre>

<!-- Em space for wide separation -->
<p>Chapter 1&emsp;Introduction</p>
```

## Typography Entities

### Dashes

There are distinct characters for different dash purposes. Using the wrong one is a common typographic error.

| Character | Name | Entity | Usage |
|-----------|------|--------|-------|
| - | Hyphen-minus | (keyboard `-`) | Compound words: `well-known` |
| -- | En dash | `&ndash;` | Ranges: `pages 1&ndash;10`, scores: `3&ndash;2` |
| --- | Em dash | `&mdash;` | Parenthetical breaks, attribution |
| - | Minus sign | `&minus;` | Mathematics: `5 &minus; 3 = 2` |

```html
<!-- En dash for ranges -->
<p>Open Monday&ndash;Friday, 9&ndash;5.</p>
<p>See chapters 3&ndash;7 for details.</p>

<!-- Em dash for parenthetical breaks -->
<p>The result&mdash;surprisingly&mdash;was positive.</p>

<!-- Minus sign for math (visually distinct from hyphen) -->
<p>The temperature dropped to &minus;15&deg;C.</p>
```

The hyphen-minus on your keyboard (`-`, U+002D) is technically none of these. It is a compromise character from the ASCII era that serves as hyphen, minus, and dash. Proper typography uses distinct characters for each.

### Quotation Marks

| Character | Name | Entity |
|-----------|------|--------|
| " | Left double quote | `&ldquo;` |
| " | Right double quote | `&rdquo;` |
| ' | Left single quote | `&lsquo;` |
| ' | Right single quote / apostrophe | `&rsquo;` |
| " | Straight double quote | `&quot;` |
| ' | Straight single quote | `&apos;` |
| << | Left guillemet | `&laquo;` |
| >> | Right guillemet | `&raquo;` |
| ,, | Double low-9 quote | `&bdquo;` |

```html
<!-- English curly quotes -->
<p>&ldquo;This is a quote,&rdquo; she said.</p>

<!-- Apostrophe (use right single quote, not straight quote) -->
<p>It&rsquo;s a beautiful day.</p>

<!-- French guillemets -->
<p>&laquo;&nbsp;Bonjour&nbsp;&raquo;</p>

<!-- German low-high quotes -->
<p>&bdquo;Guten Tag&ldquo;</p>
```

**Gotcha:** When pasting from Word processors, curly quotes may arrive as Windows-1252 encoded characters that display as garbled text if the page declares UTF-8 but the characters were not properly converted. Using named entities avoids this risk.

### Other Common Symbols

| Character | Name | Entity |
|-----------|------|--------|
| ... | Ellipsis | `&hellip;` |
| . | Middle dot | `&middot;` |
| * | Bullet | `&bull;` |
| (c) | Copyright | `&copy;` |
| (R) | Registered | `&reg;` |
| TM | Trademark | `&trade;` |
| S | Section | `&sect;` |
| ++ | Dagger | `&dagger;` |
| ++ | Double dagger | `&Dagger;` |
| o | Degree | `&deg;` |
| x | Multiplication | `&times;` |
| / | Division | `&divide;` |

### Three Periods vs Ellipsis

```html
<!-- BAD: three separate period characters -->
<p>Loading...</p>
<!-- Wider spacing, may wrap between periods -->

<!-- GOOD: single ellipsis glyph -->
<p>Loading&hellip;</p>
<!-- Tighter spacing, never splits across lines -->
```

The ellipsis entity `&hellip;` (U+2026) is a single character with properly kerned dots. Three periods are three separate characters with wider spacing. Screen readers may also handle them differently.

## Zero-Width and Invisible Characters

### Zero-Width Space (ZWSP)

Unicode U+200B. Invisible, zero-width. Creates a line-break opportunity without a visible space.

```html
<!-- Allow a long URL to wrap -->
<p>https://example.com/&#8203;very/&#8203;long/&#8203;path/&#8203;to/&#8203;resource</p>

<!-- Alternative: the <wbr> element does the same thing -->
<p>https://example.com/<wbr>very/<wbr>long/<wbr>path/<wbr>to/<wbr>resource</p>
```

**Gotcha:** ZWSP is invisible but present in string comparisons. Two visually identical strings can differ if one contains a ZWSP. This is a known vector for security attacks (homograph attacks) and subtle copy-paste bugs.

### Soft Hyphen (`&shy;`)

Unicode U+00AD. Invisible unless the browser needs to break the word at that point, in which case a visible hyphen is rendered.

```html
<p>Donau&shy;dampf&shy;schiff&shy;fahrts&shy;gesell&shy;schaft</p>
<!-- On wide screens: Donaudampfschifffahrtsgesellschaft -->
<!-- On narrow screens: Donaudampf- -->
<!--                    schifffahrts- -->
<!--                    gesellschaft  -->
```

Contrast with `<wbr>`: the `<wbr>` element allows a break but never shows a hyphen. `&shy;` shows a hyphen when the break is taken.

| Method | Visible on Break | Type |
|--------|-----------------|------|
| `&shy;` | Hyphen shown | Text character |
| `<wbr>` | Nothing shown | HTML element |
| `&#8203;` (ZWSP) | Nothing shown | Text character |

### Zero-Width Joiner (ZWJ) and Non-Joiner (ZWNJ)

These control ligature behavior in complex scripts and power compound emoji.

```html
<!-- ZWJ creates compound emoji -->
<!-- Person + ZWJ + Laptop = Technologist emoji -->
&#x1F9D1;&#x200D;&#x1F4BB;

<!-- ZWNJ prevents joining in Arabic/Persian scripts -->
<!-- Essential for correct rendering in these languages -->
```

### Directional Marks

`&lrm;` (Left-to-Right Mark, U+200E) and `&rlm;` (Right-to-Left Mark, U+200F) override the Unicode Bidirectional Algorithm inline.

```html
<!-- Without LRM, the period may appear on the wrong side in RTL context -->
<p>The word in Arabic is &#x0645;&#x0631;&#x062D;&#x0628;&#x0627;&lrm;.</p>
```

## Entities Inside `<script>` and `<style>`

HTML entities are not parsed inside `<script>` or `<style>` elements. The HTML parser treats their content as raw text.

```html
<script>
  // BAD: &amp; is NOT decoded here
  var x = "&amp;";
  console.log(x);        // prints "&amp;", NOT "&"
  console.log(x.length); // 5, not 1
</script>
```

To use special characters in JavaScript strings, use JavaScript's own escape syntax:

```html
<script>
  var copyright = "\u00A9";  // copyright sign
  var emoji = "\u{1F600}";   // grinning face (ES6 syntax)
  var ampersand = "&";       // just type it directly
</script>
```

## CSS `content` Property and Unicode

HTML entities do not work in CSS. CSS has its own escape syntax using backslash-hex.

```css
/* HTML entity syntax does NOT work in CSS */
.bad::before {
  content: "&copy;";  /* renders literally as "&copy;" */
}

/* CSS escape syntax: backslash + hex codepoint */
.good::before {
  content: "\00A9";   /* renders as (c) */
}

/* Leading zeros are optional */
.also-good::before {
  content: "\A9";     /* same result */
}
```

### Line Breaks in CSS Content

Use `\A` (U+000A, line feed) combined with `white-space: pre` or `pre-wrap`:

```css
.multiline::before {
  content: "Line 1\A Line 2";
  white-space: pre;
}
```

### Accessibility Warning

Content generated by CSS `::before` and `::after` is not reliably accessible to all screen readers. Some read it, others skip it. Never put essential information in CSS-generated content.

```css
/* BAD: essential info in CSS content */
.required::after {
  content: " (required)";
}

/* GOOD: use HTML for essential info, CSS for decoration only */
```

## Common Mistakes

- **Using `&nbsp;` for layout spacing.** Non-breaking spaces are for preventing line breaks between specific words, not for indentation or element spacing. Use CSS `margin`, `padding`, or `gap`.

- **Missing ampersand encoding in URLs.** `<a href="page?a=1&b=2">` is invalid. The `&b` starts an entity reference. Always write `&amp;` in HTML attribute URLs.

- **Double encoding.** When content passes through multiple encoding stages (database, server template, client framework), each stage may encode entities again. `&` becomes `&amp;` becomes `&amp;amp;`. Encode at the last possible moment, or use your framework's `double_encode: false` flag.

- **Assuming entities work in JavaScript.** `&amp;` inside a `<script>` block is the literal five-character string `"&amp;"`, not `"&"`. Use JavaScript's own `\u` escape syntax.

- **Case sensitivity errors.** `&Amp;` is not `&amp;`. `&COPY;` is not `&copy;`. Named entities are case-sensitive.

- **Missing semicolons.** `&copy2025` is ambiguous. Is it `&copy;2025` or the text `&copy2025`? Always include the semicolon: `&copy;2025`.

- **Using three periods instead of `&hellip;`.** Three periods have wider spacing, may split across lines, and are typographically incorrect. Use the single ellipsis character.

- **Overusing entities.** With UTF-8, there is no need to encode `e` as `&#101;` or copyright as `&#169;` when you can type `(c)` directly. Only encode what the parser requires or what is invisible.

## Best Practices

- **Always escape the five reserved characters in their respective contexts.** `&amp;` in all HTML content, `&lt;` and `&gt;` in text, `&quot;` in double-quoted attributes, `&#39;` in single-quoted attributes.

- **Declare UTF-8 encoding and type most characters directly.** With `<meta charset="utf-8">`, you can type accented letters, typographic quotes, and symbols without entities.

- **Use named entities for invisible characters.** `&nbsp;`, `&shy;`, `&#8203;` (ZWSP) are self-documenting in source code. A direct invisible character is indistinguishable from a regular space or empty string when reading the source.

- **Use proper typographic characters.** En dashes for ranges (`&ndash;`), em dashes for breaks (`&mdash;`), curly quotes for prose (`&ldquo;`/`&rdquo;`), ellipsis glyph (`&hellip;`), and the proper minus sign (`&minus;`) for math.

- **Do not rely on special characters alone to convey meaning.** Screen readers skip most symbols at default verbosity. If a symbol carries semantic meaning, provide text alternatives.

- **Use `&shy;` for long words that may need to break.** Particularly useful for German, Finnish, and other languages with compound words.

- **Use `<wbr>` or ZWSP for long unbreakable strings.** URLs, file paths, and hash strings benefit from explicit break opportunities.

## Real-World Use Cases

### Legal Footer with Proper Typography

```html
<footer>
  <p>&copy; 2024&ndash;2026 Acme Corp. All rights reserved.&reg;</p>
  <p>
    <a href="/terms">Terms</a> &middot;
    <a href="/privacy">Privacy</a> &middot;
    <a href="/cookies">Cookies</a>
  </p>
  <p>Acme Corp&trade; is a registered trademark.</p>
</footer>
```

### Technical Documentation with Code References

```html
<article>
  <h2>HTML Entity Syntax</h2>
  <p>
    To display a less-than sign in HTML, write <code>&amp;lt;</code>
    which renders as <code>&lt;</code>.
  </p>
  <p>
    To display the entity reference itself, double-encode:
    <code>&amp;amp;lt;</code> renders as <code>&amp;lt;</code>.
  </p>
  <p>
    The generic syntax is <code>&amp;name;</code> for named references
    and <code>&amp;#nnnn;</code> for numeric references.
  </p>
</article>
```

### Internationalized Content with Proper Quotes and Breaks

```html
<blockquote lang="en">
  <p>&ldquo;The only way to do great work is to love what you do.&rdquo;</p>
</blockquote>

<blockquote lang="fr">
  <p>&laquo;&nbsp;La vie est belle&nbsp;&raquo;</p>
</blockquote>

<blockquote lang="de">
  <p>&bdquo;Ich bin ein Berliner&ldquo;</p>
</blockquote>

<!-- Long German compound word with soft hyphens for graceful wrapping -->
<p lang="de">
  Die Kraft&shy;fahrzeug&shy;haft&shy;pflicht&shy;versicherung
  ist in Deutschland Pflicht.
</p>

<!-- Long URL with break opportunities -->
<p>
  Download from https://cdn.example.com/<wbr>releases/<wbr>v2.1.0/<wbr>package-name-<wbr>x86_64-<wbr>linux.tar.gz
</p>
```

---

> **Key Takeaway:** Always escape the five reserved characters (`&`, `<`, `>`, `"`, `'`) in their respective contexts. For everything else, declare UTF-8 and type characters directly, reserving entities for invisible characters where source code readability matters and for proper typographic marks where precision matters.
