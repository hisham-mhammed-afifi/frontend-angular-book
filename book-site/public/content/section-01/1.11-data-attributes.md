# 1.11 Data Attributes

> Data attributes (`data-*`) let you attach custom metadata to any HTML element, creating a bridge between your markup and JavaScript without inventing non-standard attributes or abusing classes and IDs.

---

## Syntax and Naming Rules

Any attribute whose name starts with `data-` is a data attribute. The part after the prefix is the custom name.

```html
<article
  data-post-id="42"
  data-category="tech"
  data-published="2026-02-25"
  data-featured="true"
>
  <h2>Article Title</h2>
</article>
```

### Naming Restrictions

- The name must have at least one character after `data-`.
- The name should not start with `xml` (case-insensitive), reserved for XML specifications.
- The name should not contain colons (`:`), which have special meaning in XML.
- The name should not contain uppercase ASCII letters (`A-Z`), as XML is lowercase-only.
- No spaces, `"`, `'`, `>`, `/`, or `=` characters.
- Numbers, hyphens, underscores, and periods are valid within the name.

These restrictions are recommendations, not hard errors. Browsers will still parse attributes that violate them, but the `dataset` API may behave unexpectedly.

```html
<!-- GOOD: lowercase with hyphens -->
<div data-user-id="42"></div>
<div data-created-at="2026-01-01"></div>

<!-- BAD: uppercase letters -->
<div data-userId="42"></div>

<!-- BAD: contains colons -->
<div data-my:value="42"></div>
```

## The `dataset` API

Every HTML element has a `dataset` property that returns a `DOMStringMap`, providing read and write access to all `data-*` attributes on that element.

### The Hyphen-to-CamelCase Conversion

The `dataset` API converts between hyphenated HTML attribute names and camelCase JavaScript property names.

**HTML to JavaScript:** Remove the `data-` prefix. For any hyphen immediately followed by a lowercase letter (`a-z`), remove the hyphen and uppercase the letter.

**JavaScript to HTML:** For any uppercase letter, insert a hyphen before it and lowercase it. Prepend `data-`.

| HTML Attribute | `dataset` Property |
|---|---|
| `data-user-id` | `dataset.userId` |
| `data-value` | `dataset.value` |
| `data-my-long-name` | `dataset.myLongName` |
| `data-item-1` | `dataset["item-1"]` |
| `data--double-dash` | `dataset["-doubleDash"]` |

### Edge Case: Hyphens Before Non-Lowercase Characters

The conversion only applies to hyphens followed by lowercase ASCII letters. Hyphens before digits, uppercase letters, or other hyphens are left as-is in the property name.

```html
<div data-item-1="first" data-item-2="second"></div>
```

```javascript
const el = document.querySelector('div');

// Dot notation does NOT work for these
el.dataset.item-1;     // SyntaxError (parsed as subtraction)

// Bracket notation required
el.dataset['item-1'];  // "first"
el.dataset['item-2'];  // "second"
```

### Reading, Writing, Deleting

```javascript
const card = document.querySelector('[data-post-id]');

// Reading
const id = card.dataset.postId;           // "42"
const category = card.dataset.category;   // "tech"

// Writing (creates or updates the attribute)
card.dataset.postId = '99';               // sets data-post-id="99"
card.dataset.newProp = 'hello';           // creates data-new-prop="hello"

// Deleting
delete card.dataset.category;             // removes data-category entirely

// Checking existence
'postId' in card.dataset;                 // true
'missing' in card.dataset;                // false

// Enumerating all data attributes
for (const [key, value] of Object.entries(card.dataset)) {
  console.log(key, value);
}
```

### `getAttribute` / `setAttribute` Alternative

The lower-level DOM API works with the full attribute name including the `data-` prefix. No camelCase conversion is involved.

```javascript
card.getAttribute('data-post-id');         // "42"
card.setAttribute('data-post-id', '99');
card.removeAttribute('data-post-id');
card.hasAttribute('data-post-id');         // boolean
```

Use `getAttribute`/`setAttribute` when you need the raw attribute name or when working inside tight loops where the marginal performance difference matters (see Performance section).

## All Values Are Strings

Both `dataset` and `getAttribute` always return strings. If you store `data-count="42"`, you get back the string `"42"`, not the number `42`. You must parse manually.

```javascript
const el = document.querySelector('[data-count]');

// BAD: string concatenation, not addition
el.dataset.count + 1;   // "421"

// GOOD: parse the value
parseInt(el.dataset.count, 10) + 1;  // 43

// Booleans are strings too
el.dataset.active === 'true';  // compare as string
el.dataset.active === true;    // always false

// JSON-encoded objects
const config = JSON.parse(el.dataset.config);
```

```html
<!-- Storing structured data -->
<div data-config='{"theme":"dark","fontSize":14}'></div>
```

```javascript
const el = document.querySelector('[data-config]');
const config = JSON.parse(el.dataset.config);
console.log(config.theme);    // "dark"
console.log(config.fontSize); // 14 (parsed from JSON)
```

## Querying Elements by Data Attributes

Use CSS attribute selectors with `querySelector` and `querySelectorAll`.

```javascript
// Element has the attribute (any value)
document.querySelector('[data-role]');

// Exact value match
document.querySelector('[data-role="admin"]');

// Multiple conditions
document.querySelectorAll('[data-type="button"][data-size="large"]');

// Starts with
document.querySelectorAll('[data-category^="tech"]');

// Contains
document.querySelectorAll('[data-tags*="javascript"]');
```

## CSS Integration

### Attribute Selectors

CSS attribute selectors work directly with data attributes, enabling styling based on state or configuration without toggling classes.

| Selector | Meaning |
|---|---|
| `[data-status]` | Has the attribute (any value or no value) |
| `[data-status="active"]` | Exact value match |
| `[data-status~="featured"]` | Whitespace-separated list contains "featured" |
| `[data-status\|="en"]` | Exactly "en" or starts with "en-" |
| `[data-status^="act"]` | Starts with "act" |
| `[data-status$="ive"]` | Ends with "ive" |
| `[data-status*="tiv"]` | Contains "tiv" |
| `[data-status="Active" i]` | Case-insensitive match |

```css
/* Style elements based on state */
[data-state="open"] {
  display: block;
}

[data-state="closed"] {
  display: none;
}

/* Style by priority level */
[data-priority="high"] {
  border-left: 4px solid red;
}

[data-priority="medium"] {
  border-left: 4px solid orange;
}

[data-priority="low"] {
  border-left: 4px solid green;
}

/* Case-insensitive matching */
[data-status="active" i] {
  opacity: 1;
}
```

### The `attr()` Function in CSS

The `attr()` function reads an attribute value for use in CSS. Traditionally, it only works with the `content` property:

```css
/* Show tooltip text from data attribute (widely supported) */
[data-tooltip]:hover::after {
  content: attr(data-tooltip);
  position: absolute;
  background: #333;
  color: white;
  padding: 4px 8px;
  border-radius: 4px;
  font-size: 0.875rem;
  white-space: nowrap;
}
```

### Advanced `attr()` (Chrome 133+)

Starting February 2025, Chrome supports an enhanced `attr()` that works with any CSS property and can parse typed values:

```css
/* Modern typed attr() - Chromium only as of early 2026 */
div {
  background-color: attr(data-color type(<color>), gray);
  width: attr(data-width type(<length>), 100px);
  opacity: attr(data-opacity type(<number>), 1);
}
```

The syntax is `attr(<name> type(<css-type>), <fallback>)`. Firefox and Safari are still implementing this, so treat it as progressive enhancement.

## Data Attributes vs Classes vs IDs

Each mechanism serves a different primary purpose. Mixing them creates maintenance problems.

| Mechanism | Primary Purpose | Example |
|---|---|---|
| `class` | Styling (CSS hooks) | `class="btn btn-primary"` |
| `id` | Fragment identifiers, label `for`, ARIA references | `id="main-nav"` |
| `data-*` | JavaScript behavior, state, configuration | `data-action="save"` |

### Why This Separation Matters

Using data attributes for JavaScript selection instead of classes means refactoring CSS (renaming classes, restructuring BEM) will never break your JavaScript, and vice versa.

```html
<!-- BAD: class serves double duty for styling AND behavior -->
<button class="btn save-btn js-save">Save</button>

<!-- GOOD: class for styling, data attribute for behavior -->
<button class="btn btn-primary" data-action="save">Save</button>
```

With the BAD approach, renaming `js-save` to `js-submit` during a CSS refactor silently breaks JavaScript that queries `.js-save`. With the GOOD approach, styles and behavior are decoupled.

## Event Delegation with Data Attributes

Attach a single event listener to a parent element. Use `event.target.closest()` to find the element with the data attribute and read its value to decide the action.

```html
<div class="toolbar">
  <button data-action="bold"><strong>B</strong></button>
  <button data-action="italic"><em>I</em></button>
  <button data-action="underline"><u>U</u></button>
  <button data-action="link">Link</button>
</div>
```

```javascript
const actions = {
  bold()      { document.execCommand('bold'); },
  italic()    { document.execCommand('italic'); },
  underline() { document.execCommand('underline'); },
  link()      { /* prompt for URL */ },
};

document.querySelector('.toolbar').addEventListener('click', (event) => {
  const btn = event.target.closest('[data-action]');
  if (!btn) return;

  const action = btn.dataset.action;
  actions[action]?.();
});
```

### Why `closest()` Is Essential

When the user clicks an icon `<svg>` or `<strong>` inside the button, `event.target` is the inner element, not the button. `closest('[data-action]')` walks up the DOM tree and finds the nearest ancestor (or self) matching the selector.

```javascript
// Without closest(): clicking <strong>B</strong> inside the button
// gives event.target = <strong>, which has no data-action
// Result: action is undefined, nothing happens

// With closest(): finds the <button data-action="bold"> ancestor
// Result: works correctly regardless of click target
```

### Dynamic Content

Event delegation handles dynamically added elements automatically. The listener is on the parent, so new children are covered without rebinding.

```javascript
// New button added later
toolbar.insertAdjacentHTML('beforeend',
  '<button data-action="strikethrough">S</button>'
);
// No need to attach a new listener. The existing one picks it up.
```

## Performance

### `dataset` vs `getAttribute`

`getAttribute()` is marginally faster for individual reads because it bypasses the DOMStringMap proxy. The difference is negligible for typical use (both sub-microsecond per call). In tight loops reading thousands of attributes, `getAttribute` can be measurably faster.

```javascript
// Slightly faster in tight loops
el.getAttribute('data-user-id');

// Slightly slower but more convenient
el.dataset.userId;
```

For typical application code, always prefer `dataset` for readability.

### CSS Selector Performance

Class selectors and ID selectors use highly optimized indexed lookups in the rendering engine. Attribute selectors require string comparison against the raw attribute value. Benchmarks show attribute selectors are roughly 2-3x slower than class selectors.

In practice, this difference is irrelevant unless your DOM has tens of thousands of nodes with complex stylesheets. For a few hundred elements, attribute selectors are perfectly fine.

### When to Avoid DOM-Based Data Storage

If your application frequently reads and writes the same data, store it in a JavaScript `Map` or `WeakMap` keyed by the element. Each `setAttribute` call can trigger style recalculation if CSS selectors match on that attribute. For read-heavy patterns like event delegation, data attributes are fine since reads are fast.

```javascript
// For frequent reads/writes: use a WeakMap
const elementData = new WeakMap();

function setData(el, key, value) {
  const data = elementData.get(el) || {};
  data[key] = value;
  elementData.set(el, data);
}

function getData(el, key) {
  return elementData.get(el)?.[key];
}
```

## Accessibility

### Data Attributes Are Invisible to Assistive Technology

Screen readers, voice assistants, and other assistive technologies do not read `data-*` attributes. They have zero semantic meaning in the accessibility tree.

| Attribute Type | Read by Screen Readers | Affects Accessibility Tree |
|---|---|---|
| `data-*` | No | No |
| `aria-*` | Yes | Yes |
| Semantic HTML (`<time>`, `<abbr>`) | Yes | Yes |

### The Rule

Data attributes are for your JavaScript logic. ARIA attributes are for the user's assistive technology. Never use one where the other belongs.

```html
<!-- BAD: storing accessible info in data attribute -->
<div data-label="Close dialog">X</div>

<!-- GOOD: using aria attribute for accessibility -->
<button aria-label="Close dialog">X</button>
```

```html
<!-- BAD: reinventing semantic HTML -->
<span data-datetime="2026-02-25">February 25</span>

<!-- GOOD: using the semantic element -->
<time datetime="2026-02-25">February 25</time>
```

## Security Considerations

### Data Attributes as an XSS Defense Strategy

OWASP recommends placing untrusted values in data attributes (properly encoded) and reading them in JavaScript, rather than injecting values directly into HTML content or script blocks. A properly quoted data attribute treats its value as an inert string that cannot execute code.

### The Dangerous Part: What You Do After Reading

Reading from a data attribute is safe. What you do with the value can be dangerous.

```javascript
// SAFE: using value as text content
el.textContent = el.dataset.userName;

// SAFE: using as a fetch parameter
fetch(`/api/users/${encodeURIComponent(el.dataset.userId)}`);

// DANGEROUS: inserting into innerHTML
el.innerHTML = el.dataset.content;  // XSS if content contains HTML

// DANGEROUS: passing to eval
eval(el.dataset.code);  // never do this
```

### Server-Side Encoding

When server-rendering data attributes with user input, always HTML-encode the value:

```html
<!-- BAD: raw user input can break out of the attribute -->
<div data-name="<?= userInput ?>">
<!-- If userInput is: "><script>alert('xss')</script> -->
<!-- Result: <div data-name=""><script>alert('xss')</script>"> -->

<!-- GOOD: HTML-entity-encoded -->
<div data-name="<?= htmlspecialchars(userInput, ENT_QUOTES) ?>">
```

## Common Mistakes

- **Using data attributes for visible content.** Search engines do not index data attributes. Screen readers do not read them. If content should be seen or heard, put it in the DOM.

- **Forgetting values are always strings.** `dataset.count + 1` is string concatenation, not addition. Always parse: `parseInt(dataset.count, 10) + 1`.

- **Using generic attribute names.** `data-id`, `data-type`, `data-value` will collide with third-party libraries, analytics tools, and browser extensions. Use a namespace prefix: `data-myapp-id`.

- **Using hyphens before digits in attribute names.** `data-item-1` requires bracket notation (`dataset["item-1"]`) because the camelCase conversion only applies to hyphens followed by lowercase letters. Use `data-item1` or `data-item-one` instead.

- **Storing large data payloads in the DOM.** Large JSON blobs or base64 strings in data attributes bloat the HTML and slow down parsing. Use a JavaScript-side data store for large payloads.

- **Inventing attributes without the `data-` prefix.** `<div myattr="value">` is invalid HTML and can conflict with future spec additions. Always use the `data-` prefix.

- **Using data attributes where semantic HTML or ARIA exists.** Do not use `data-label` when `aria-label` exists. Do not use `data-datetime` when `<time datetime="">` exists.

---

## Best Practices

- **Separate concerns: classes for styling, data attributes for behavior.** This decouples CSS refactoring from JavaScript logic.

- **Use `dataset` for normal code, `getAttribute` for hot paths.** `dataset` is more readable; `getAttribute` is marginally faster in tight loops.

- **Namespace your data attributes in shared contexts.** `data-myapp-*` prevents collisions with third-party code.

- **Keep values simple.** Prefer flat strings and simple values. If you need complex data, store a key in the data attribute and keep the actual data in JavaScript.

- **Use event delegation with `closest()`.** One listener on a parent using `data-action` is cleaner and more performant than individual listeners on each child.

- **Parse values explicitly.** Always convert strings to the appropriate type: `parseInt()`, `parseFloat()`, `=== 'true'`, or `JSON.parse()`.

- **Use data attributes for test selectors.** `data-testid` provides stable selectors for automated testing that are decoupled from styling and structure.

## Real-World Use Cases

### Tooltip System

```html
<button data-tooltip="Save your work" data-tooltip-position="top">
  Save
</button>
<button data-tooltip="Undo last action" data-tooltip-position="bottom">
  Undo
</button>
```

```css
[data-tooltip] {
  position: relative;
}

[data-tooltip]:hover::after {
  content: attr(data-tooltip);
  position: absolute;
  background: #1a1a2e;
  color: white;
  padding: 6px 10px;
  border-radius: 4px;
  font-size: 0.8rem;
  white-space: nowrap;
  pointer-events: none;
  z-index: 10;
}

[data-tooltip-position="top"]:hover::after {
  bottom: calc(100% + 8px);
  left: 50%;
  transform: translateX(-50%);
}

[data-tooltip-position="bottom"]:hover::after {
  top: calc(100% + 8px);
  left: 50%;
  transform: translateX(-50%);
}
```

### Analytics Tracking

```html
<nav>
  <a href="/pricing" data-track="click" data-track-category="navigation"
     data-track-label="pricing">Pricing</a>
  <a href="/docs" data-track="click" data-track-category="navigation"
     data-track-label="docs">Documentation</a>
</nav>
```

```javascript
document.addEventListener('click', (event) => {
  const tracked = event.target.closest('[data-track]');
  if (!tracked) return;

  const eventType = tracked.dataset.track;
  const category = tracked.dataset.trackCategory;
  const label = tracked.dataset.trackLabel;

  // Send to analytics service
  analytics.track(eventType, { category, label });
});
```

### Component Configuration from HTML

```html
<div class="carousel"
     data-autoplay="true"
     data-interval="5000"
     data-pause-on-hover="true"
     data-show-indicators="true">
  <div class="slide">Slide 1</div>
  <div class="slide">Slide 2</div>
  <div class="slide">Slide 3</div>
</div>
```

```javascript
document.querySelectorAll('.carousel').forEach((el) => {
  const config = {
    autoplay: el.dataset.autoplay === 'true',
    interval: parseInt(el.dataset.interval, 10) || 3000,
    pauseOnHover: el.dataset.pauseOnHover === 'true',
    showIndicators: el.dataset.showIndicators === 'true',
  };

  initCarousel(el, config);
});
```

This pattern lets you configure component instances declaratively in HTML, which is how libraries like Bootstrap operate (`data-bs-toggle`, `data-bs-target`, `data-bs-dismiss`).

---

> **Key Takeaway:** Data attributes bridge HTML and JavaScript cleanly: use them for behavior and configuration, not for content or styling. Keep the separation of concerns clear, always parse values from strings, and use event delegation with `closest()` for scalable, maintainable interactive patterns.
