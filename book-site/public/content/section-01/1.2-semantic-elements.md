# 1.2 Semantic Elements

> Semantic elements communicate meaning to browsers, screen readers, search engines, and other developers. Choosing the right element is not about styling. It is about what the content *is*.

---

## Why Semantics Matter

A `<div>` and a `<nav>` can look identical with CSS. The difference is invisible to sighted users but critical to everyone and everything else:

| Consumer | What It Gets from Semantics |
|---|---|
| Screen readers | Landmark navigation, element announcements, heading jumps |
| Search engines | Content hierarchy, main content identification, snippet selection |
| Browser features | Reader mode extraction, autofill heuristics, print formatting |
| AI assistants | Content structure for summarization and voice search |
| Other developers | Readable, self-documenting markup |

The rule: if a native HTML element exists for the content's purpose, use it instead of a `<div>` with ARIA attributes. Native semantics are more robust, require less code, and have broader support.

---

## Sectioning Elements

Four elements create sections in the document: `<article>`, `<section>`, `<nav>`, and `<aside>`. Each can contain its own `<header>`, `<footer>`, and heading hierarchy.

### `<article>`: Self-Contained Content

Content that makes sense on its own, independently of the rest of the page. If you could syndicate it (RSS feed, embed on another site, share on social media) and it would still make sense, it is an `<article>`.

```html
<article>
  <header>
    <h2>Understanding Closures in JavaScript</h2>
    <time datetime="2025-11-15">November 15, 2025</time>
  </header>
  <p>A closure is a function that retains access to its lexical scope...</p>
  <footer>
    <p>Written by <address><a href="mailto:author@example.com">Jane Doe</a></address></p>
  </footer>
</article>
```

Valid `<article>` content: blog posts, news articles, forum posts, user comments, product cards, interactive widgets, social media posts.

**Nested articles** represent related content. The classic example is blog comments:

```html
<article>
  <h2>My Blog Post</h2>
  <p>Post content here...</p>

  <section>
    <h3>Comments</h3>

    <article>
      <header>
        <b>Alice</b> <time datetime="2025-11-16T09:30">Nov 16, 9:30 AM</time>
      </header>
      <p>Great article!</p>
    </article>

    <article>
      <header>
        <b>Bob</b> <time datetime="2025-11-16T11:00">Nov 16, 11:00 AM</time>
      </header>
      <p>I disagree with the third point...</p>
    </article>
  </section>
</article>
```

### `<section>`: Thematic Grouping

A generic section of a document, grouped by theme. Unlike `<article>`, a `<section>` does not need to stand alone. It groups related content under a common heading.

```html
<section>
  <h2>Pricing Plans</h2>
  <div class="plans-grid">
    <article class="plan">
      <h3>Starter</h3>
      <p>$9/month</p>
    </article>
    <article class="plan">
      <h3>Pro</h3>
      <p>$29/month</p>
    </article>
  </div>
</section>
```

**Critical accessibility detail:** A `<section>` only becomes a landmark (`region` role) when it has an accessible name via `aria-label` or `aria-labelledby`. Without a name, it is semantically equivalent to a `<div>` for assistive technologies.

```html
<!-- This IS a landmark: screen readers announce "Pricing region" -->
<section aria-labelledby="pricing-heading">
  <h2 id="pricing-heading">Pricing</h2>
  <!-- ... -->
</section>

<!-- This is NOT a landmark: no accessible name -->
<section>
  <h2>Pricing</h2>
  <!-- ... -->
</section>
```

### The Decision Tree: `<article>` vs `<section>` vs `<div>`

```
Is the content semantically meaningful (not just a styling wrapper)?
  ├── No  --> <div>
  └── Yes
       └── Is it self-contained / syndicatable?
            ├── Yes --> <article>
            └── No  --> Does it represent a thematic group with a heading?
                 ├── Yes --> <section>
                 └── No  --> <div>
```

### `<nav>`: Major Navigation

Represents a section with navigation links. Only for **major** navigation blocks, not every group of links.

```html
<!-- Primary site navigation: use <nav> -->
<nav aria-label="Main">
  <ul>
    <li><a href="/">Home</a></li>
    <li><a href="/products">Products</a></li>
    <li><a href="/about">About</a></li>
  </ul>
</nav>

<!-- Breadcrumbs: use <nav> (major navigation aid) -->
<nav aria-label="Breadcrumb">
  <ol>
    <li><a href="/">Home</a></li>
    <li><a href="/products">Products</a></li>
    <li aria-current="page">Widget Pro</li>
  </ol>
</nav>

<!-- Footer links like "Privacy" and "Terms": usually NOT <nav> -->
<!-- These are secondary links, not major navigation -->
<footer>
  <ul>
    <li><a href="/privacy">Privacy Policy</a></li>
    <li><a href="/terms">Terms of Service</a></li>
  </ul>
</footer>
```

**When you have multiple `<nav>` elements, label each one** so screen reader users can distinguish them:

```html
<nav aria-label="Main">...</nav>
<nav aria-label="Footer">...</nav>
```

Without labels, a screen reader announces "navigation" for both, and the user has no way to tell them apart.

### `<aside>`: Tangentially Related Content

Content that is related to the surrounding content but could be removed without affecting the main flow.

**Outside `<article>` (page-level):** Site-wide sidebar content like blogrolls, advertising, additional navigation.

```html
<main>
  <article>
    <h1>Understanding Flexbox</h1>
    <p>Flexbox is a one-dimensional layout method...</p>
  </article>
</main>

<aside aria-label="Related articles">
  <h2>Related Articles</h2>
  <ul>
    <li><a href="/css-grid">CSS Grid Guide</a></li>
    <li><a href="/responsive">Responsive Design</a></li>
  </ul>
</aside>
```

**Inside `<article>`:** Content specifically related to that article, like pull quotes or glossary terms.

```html
<article>
  <h2>The State of WebAssembly</h2>
  <p>WebAssembly has seen massive adoption in 2025...</p>

  <aside>
    <p><strong>Did you know?</strong> WebAssembly was first announced in 2015
    and reached 1.0 status in 2017.</p>
  </aside>

  <p>Performance benchmarks show...</p>
</article>
```

**`<aside>` is a semantic concept, not a visual one.** A sidebar layout does not automatically mean `<aside>`. Choose based on the content's relationship to the main content, not its visual position.

## Structural Elements

### `<header>`: Introductory Content

Introductory content or navigational aids. Typically contains headings, logos, search forms, author info.

```html
<!-- Page-level header: gets implicit "banner" landmark role -->
<body>
  <header>
    <img src="/logo.svg" alt="Acme Corp">
    <nav aria-label="Main">...</nav>
  </header>
  <main>...</main>
</body>
```

```html
<!-- Article-level header: NO landmark role (generic) -->
<article>
  <header>
    <h2>Article Title</h2>
    <p>Published <time datetime="2025-06-15">June 15, 2025</time></p>
  </header>
  <p>Article content...</p>
</article>
```

**Cannot be nested inside:** `<footer>`, `<address>`, or another `<header>`.

### `<footer>`: Closing Content

Information about the nearest sectioning content or root element. Typically contains authorship, copyright, related links, contact info.

```html
<!-- Page-level footer: gets implicit "contentinfo" landmark role -->
<body>
  <header>...</header>
  <main>...</main>
  <footer>
    <p>&copy; 2025 Acme Corp. All rights reserved.</p>
  </footer>
</body>
```

```html
<!-- Article-level footer: NO landmark role -->
<article>
  <h2>Understanding Promises</h2>
  <p>Content...</p>
  <footer>
    <p>Tags: JavaScript, Async, ES6</p>
    <p>Last updated: <time datetime="2025-09-01">September 2025</time></p>
  </footer>
</article>
```

**Cannot contain:** `<header>` or another `<footer>`.

### `<main>`: Primary Content

The dominant content of the `<body>`. Content that is unique to this page, excluding repeated site-wide content like headers, footers, and navigation.

**Rules:**
- Only **one visible** `<main>` per document (additional `<main>` elements must have the `hidden` attribute)
- Must NOT be a descendant of `<article>`, `<aside>`, `<footer>`, `<header>`, or `<nav>`
- Should be a direct child of `<body>`

```html
<body>
  <header>...</header>
  <nav>...</nav>

  <main>
    <!-- This is where the unique page content goes -->
    <h1>Dashboard</h1>
    <section>...</section>
  </main>

  <footer>...</footer>
</body>
```

### `<search>`: Search Functionality

A container for form controls related to search or filtering. Added to the spec in 2023, with full browser support (Chrome 118+, Firefox 118+, Safari 17+).

```html
<!-- Replaces the old <form role="search"> pattern -->
<search>
  <form action="/search">
    <label for="q">Search articles</label>
    <input type="search" id="q" name="q">
    <button type="submit">Search</button>
  </form>
</search>
```

`<search>` is NOT for displaying search results. It wraps the search/filter controls only. Implicit ARIA role: `search`.

## Implicit ARIA Landmark Roles

Semantic elements automatically carry ARIA roles. Understanding this mapping prevents redundant ARIA and explains when landmarks appear to screen readers.

| HTML Element | Implicit Role | Condition |
|---|---|---|
| `<header>` | `banner` | Scoped to `<body>` (not inside sectioning content) |
| `<header>` | `generic` | Inside `<article>`, `<aside>`, `<main>`, `<nav>`, `<section>` |
| `<footer>` | `contentinfo` | Scoped to `<body>` |
| `<footer>` | `generic` | Inside sectioning content |
| `<nav>` | `navigation` | Always |
| `<main>` | `main` | Always |
| `<aside>` | `complementary` | Always |
| `<section>` | `region` | Only with an accessible name (`aria-label` / `aria-labelledby`) |
| `<section>` | `generic` | Without an accessible name |
| `<search>` | `search` | Always |
| `<form>` | `form` | Only with an accessible name |
| `<article>` | `article` | Always (but `article` is NOT a landmark role) |

**Anti-pattern: redundant ARIA roles:**

```html
<!-- BAD: redundant, the element already has this role -->
<nav role="navigation">...</nav>
<main role="main">...</main>
<aside role="complementary">...</aside>

<!-- GOOD: no role attribute needed -->
<nav>...</nav>
<main>...</main>
<aside>...</aside>
```

## The Document Outline (and Why It Failed)

The HTML5 specification originally defined a document outline algorithm. Sectioning elements (`<article>`, `<section>`, `<aside>`, `<nav>`) were supposed to create nested outline levels, allowing `<h1>` inside every section:

```html
<!-- This was SUPPOSED to create a proper hierarchy -->
<body>
  <h1>Site Title</h1>        <!-- outline level 1 -->
  <section>
    <h1>Section Title</h1>   <!-- supposed to be level 2 -->
    <section>
      <h1>Subsection</h1>    <!-- supposed to be level 3 -->
    </section>
  </section>
</body>
```

**This was never implemented.** No browser, no screen reader, no assistive technology ever supported it. The algorithm was officially removed from the WHATWG HTML spec in July 2022.

**What actually happens:** All three `<h1>` elements are announced as level 1 headings by screen readers. The document has a completely flat, broken heading structure.

**The correct approach:** Use traditional heading levels and maintain a strict hierarchy:

```html
<body>
  <h1>Site Title</h1>
  <section>
    <h2>Section Title</h2>
    <section>
      <h3>Subsection</h3>
    </section>
  </section>
</body>
```

**Rules for headings:**
- One `<h1>` per page
- Do not skip levels (`<h1>` to `<h3>` without `<h2>`)
- Headings should reflect the logical hierarchy of the content

## Text-Level Semantic Elements

### `<hgroup>`: Heading Group

Groups a heading with subtitle paragraphs. The element was removed from the spec in 2013, then restored with a new content model. It now contains a single heading element (`<h1>`-`<h6>`) plus `<p>` elements for subtitles.

```html
<hgroup>
  <h1>JavaScript: The Good Parts</h1>
  <p>A deep dive into the language's best features</p>
</hgroup>
```

The `<p>` elements inside `<hgroup>` are not exposed as separate headings. This prevents subtitles from appearing as headings in the document outline.

### `<address>`: Contact Information

Represents contact information for the nearest `<article>` or `<body>` ancestor. It is NOT for arbitrary postal addresses in body content.

```html
<!-- CORRECT: contact info for the page/site -->
<footer>
  <address>
    <p>Questions? Contact <a href="mailto:support@example.com">support@example.com</a></p>
    <p>Acme Corp, 123 Main St, Springfield</p>
  </address>
</footer>

<!-- CORRECT: contact info for an article's author -->
<article>
  <h2>My Article</h2>
  <p>Content...</p>
  <footer>
    <address>
      Written by <a href="mailto:jane@example.com">Jane Doe</a>
    </address>
  </footer>
</article>

<!-- BAD: using <address> for a shipping address in content -->
<p>Ship to:</p>
<address>
  123 Customer Lane, Anytown, USA
</address>
```

### `<time>`: Machine-Readable Dates

Represents a specific date, time, or duration. The `datetime` attribute provides the machine-readable value.

```html
<!-- Date -->
<time datetime="2025-12-25">Christmas Day</time>

<!-- Date and time with timezone -->
<time datetime="2025-12-25T20:00:00-05:00">December 25 at 8 PM EST</time>

<!-- Duration -->
<time datetime="PT2H30M">2 hours and 30 minutes</time>

<!-- If the text IS a valid datetime, the attribute is optional -->
<time>2025-12-25</time>

<!-- Week -->
<time datetime="2025-W51">Week 51 of 2025</time>
```

Valid `datetime` formats (ISO 8601 based):

| Format | Example |
|---|---|
| Year | `2025` |
| Year-month | `2025-12` |
| Full date | `2025-12-25` |
| Date without year | `12-25` |
| Week | `2025-W51` |
| Time | `20:00` or `20:00:00` |
| Date and time | `2025-12-25T20:00` or `2025-12-25 20:00` |
| With timezone | `20:00+05:30`, `20:00Z` |
| Duration | `P2DT3H30M` (2 days, 3 hours, 30 minutes) |

### `<mark>`: Highlighted Text

Marks text for reference or relevance to the user's current activity. The primary use case is highlighting search terms in results.

```html
<!-- Search results highlighting -->
<p>Results for "closure":</p>
<p>A <mark>closure</mark> is a function that retains access to
variables from its enclosing scope even after the outer function
has returned. <mark>Closures</mark> are fundamental to JavaScript.</p>
```

**`<mark>` is not `<strong>`.** `<strong>` indicates importance of the text itself. `<mark>` indicates relevance to the user's current context (search, reference).

**Accessibility gotcha:** Most screen readers do not announce `<mark>` by default. If the highlight carries meaning, add screen-reader-only text:

```css
mark::before {
  content: " [highlight start] ";
  clip-path: inset(50%);
  height: 1px;
  width: 1px;
  overflow: hidden;
  position: absolute;
  white-space: nowrap;
}

mark::after {
  content: " [highlight end] ";
  clip-path: inset(50%);
  height: 1px;
  width: 1px;
  overflow: hidden;
  position: absolute;
  white-space: nowrap;
}
```

### `<blockquote>` and `<cite>`

`<blockquote>` is for content quoted from another source. `<cite>` references the title of a creative work.

```html
<figure>
  <blockquote cite="https://www.w3.org/WAI/fundamentals/accessibility-intro/">
    <p>The Web is fundamentally designed to work for all people,
    whatever their hardware, software, language, location, or ability.</p>
  </blockquote>
  <figcaption>
    <cite>Introduction to Web Accessibility</cite>, W3C WAI
  </figcaption>
</figure>
```

Note: the `cite` **attribute** on `<blockquote>` is a URL (not visible to users). The `<cite>` **element** is for visible work titles. They serve different purposes.

### `<abbr>` and `<dfn>`

`<abbr>` marks an abbreviation. `<dfn>` marks the defining instance of a term.

```html
<!-- Abbreviation with expansion on hover -->
<p>The <abbr title="World Wide Web Consortium">W3C</abbr> maintains web standards.</p>

<!-- Defining a term -->
<p><dfn>Semantic HTML</dfn> is the practice of using HTML elements
that convey meaning about the content they contain.</p>

<!-- Combining both: defining an abbreviation -->
<p><dfn><abbr title="Application Programming Interface">API</abbr></dfn>
is a set of protocols for building software.</p>
```

### Code-Related Elements

```html
<!-- Inline code -->
<p>Use <code>Array.prototype.map()</code> to transform arrays.</p>

<!-- Code block -->
<pre><code>function greet(name) {
  return `Hello, ${name}!`;
}</code></pre>

<!-- Keyboard input -->
<p>Press <kbd><kbd>Ctrl</kbd>+<kbd>C</kbd></kbd> to copy.</p>

<!-- Sample output -->
<p>The console displays: <samp>Error: file not found</samp></p>

<!-- Variable -->
<p>Let <var>x</var> be the number of items in the array.</p>
```

**Nesting subtleties for `<kbd>` and `<samp>`:**

| Pattern | Meaning |
|---|---|
| `<kbd>` alone | User input (keyboard key) |
| `<kbd>` inside `<kbd>` | An actual key press within a key combination |
| `<kbd>` inside `<samp>` | Input echoed by the system |
| `<samp>` inside `<kbd>` | Input based on system output (like selecting a menu item) |

## Interactive Elements

### `<details>` and `<summary>`: Native Disclosure Widget

A collapsible section that requires no JavaScript.

```html
<details>
  <summary>System Requirements</summary>
  <ul>
    <li>Node.js 18+</li>
    <li>npm 9+</li>
    <li>4 GB RAM minimum</li>
  </ul>
</details>
```

**The `open` attribute** is a boolean attribute. Its presence means the content is visible.

```html
<!-- Expanded by default -->
<details open>
  <summary>Changelog</summary>
  <p>v2.0: Complete rewrite...</p>
</details>
```

**Gotcha:** `open="false"` still opens the details because boolean attributes are true when present, regardless of value. To close, remove the attribute entirely.

**Exclusive accordions with `name` (2024+):**

Multiple `<details>` elements with the same `name` form a group where only one can be open at a time. Browser support: Chrome 120+, Safari 17.2+, Firefox 130+.

```html
<!-- Only one can be open at a time -->
<details name="faq">
  <summary>What is your return policy?</summary>
  <p>30-day money-back guarantee on all products.</p>
</details>

<details name="faq">
  <summary>How long does shipping take?</summary>
  <p>Standard shipping takes 5-7 business days.</p>
</details>

<details name="faq">
  <summary>Do you ship internationally?</summary>
  <p>Yes, we ship to over 50 countries.</p>
</details>
```

The elements do not need to be siblings. The `name` attribute groups them regardless of DOM position.

**Styling the marker:**

```css
/* Remove the default triangle */
summary {
  list-style: none;
}
summary::-webkit-details-marker {
  display: none;
}

/* Custom indicator */
summary::after {
  content: '+';
  float: right;
}
details[open] summary::after {
  content: '-';
}
```

**Accessibility note:** `<details>` has known screen reader inconsistencies. Safari may not include `<details>` in keyboard focus order. JAWS ignores headings placed inside `<summary>`. Test with multiple screen readers.

### `<dialog>`: Native Modal

A dialog box or interactive component. Supports both modal and non-modal modes.

```html
<dialog id="confirm-dialog">
  <h2>Confirm Action</h2>
  <p>Are you sure you want to delete this item?</p>
  <form method="dialog">
    <button value="cancel">Cancel</button>
    <button value="confirm">Confirm</button>
  </form>
</dialog>

<button onclick="document.getElementById('confirm-dialog').showModal()">
  Delete Item
</button>
```

**`showModal()` vs `show()`:**

| Feature | `showModal()` | `show()` |
|---|---|---|
| Backdrop (`::backdrop`) | Yes | No |
| Rest of page inert | Yes | No |
| Escape key closes | Yes | No |
| Focus trapped | Yes | No |
| Placed in top layer | Yes | No |

**`form method="dialog"`** closes the dialog without submitting the form. The clicked button's `value` is stored in `dialog.returnValue`:

```javascript
const dialog = document.getElementById('confirm-dialog');

dialog.addEventListener('close', () => {
  if (dialog.returnValue === 'confirm') {
    deleteItem();
  }
});
```

**Styling the backdrop:**

```css
dialog::backdrop {
  background: rgba(0, 0, 0, 0.5);
  backdrop-filter: blur(4px);
}
```

**`::backdrop` only exists when opened via `showModal()`.** It does not appear with `show()`.

**Close on backdrop click (not built in):**

```javascript
dialog.addEventListener('click', (event) => {
  // The dialog element itself is the backdrop area
  if (event.target === dialog) {
    dialog.close();
  }
});
```

This works because clicks on the `::backdrop` bubble to the `<dialog>`. The dialog's content area is inside child elements, so `event.target === dialog` only matches backdrop clicks.

## Measurement and Output Elements

### `<progress>`: Task Progress

Displays completion progress for a task.

```html
<!-- Determinate: shows a filled bar -->
<label for="upload">Upload progress:</label>
<progress id="upload" max="100" value="70">70%</progress>

<!-- Indeterminate: shows an animated bar (no value attribute) -->
<label for="loading">Loading:</label>
<progress id="loading">Loading...</progress>
```

The text content inside `<progress>` is a fallback for browsers that do not support the element.

### `<meter>`: Scalar Measurement

Displays a value within a known range. NOT for task progress (use `<progress>`).

```html
<label for="disk">Disk usage:</label>
<meter id="disk" min="0" max="100" value="68" low="30" high="80" optimum="50">
  68 out of 100 GB
</meter>
```

The browser automatically color-codes the bar:

| Value Position | Color |
|---|---|
| In the optimum range | Green |
| In the suboptimal range | Yellow/orange |
| In the worst range | Red |

The `optimum` attribute tells the browser which end of the range is preferred. If `optimum` is below `low`, the browser considers lower values better. If `optimum` is above `high`, higher is better.

```html
<!-- Low is good (like response time): optimum < low -->
<meter min="0" max="1000" low="100" high="500" optimum="50" value="75">
  75ms response time
</meter>

<!-- High is good (like battery level): optimum > high -->
<meter min="0" max="100" low="20" high="80" optimum="100" value="90">
  90% battery
</meter>
```

### `<output>`: Calculation Result

A container for the result of a calculation or user action.

```html
<form oninput="result.value = parseInt(a.value) + parseInt(b.value)">
  <input type="number" id="a" name="a" value="0"> +
  <input type="number" id="b" name="b" value="0"> =
  <output name="result" for="a b">0</output>
</form>
```

The `for` attribute lists the IDs of elements that contribute to the calculation. The `<output>` value is NOT submitted with the form.

### `<data>`: Machine-Readable Values

Links human-readable content with a machine-readable `value`.

```html
<ul>
  <li><data value="UPC:022000159397">Mini Ketchup</data></li>
  <li><data value="UPC:048001211200">Jumbo Mustard</data></li>
</ul>
```

If the content is date/time related, use `<time>` instead.

## `<figure>` and `<figcaption>`

`<figure>` wraps self-contained content that is referenced from the main flow. It is NOT limited to images.

```html
<!-- Image with caption -->
<figure>
  <img src="chart.png" alt="Revenue growth Q1-Q4 2025">
  <figcaption>Figure 1: Revenue growth across all quarters of 2025.</figcaption>
</figure>

<!-- Code listing -->
<figure>
  <pre><code>const sum = (a, b) => a + b;</code></pre>
  <figcaption>Listing 3: A simple arrow function.</figcaption>
</figure>

<!-- Blockquote -->
<figure>
  <blockquote>
    <p>Talk is cheap. Show me the code.</p>
  </blockquote>
  <figcaption>Linus Torvalds</figcaption>
</figure>

<!-- Video -->
<figure>
  <video src="demo.mp4" controls>
    <track kind="captions" src="demo.vtt" srclang="en">
  </video>
  <figcaption>Demo: Setting up the development environment.</figcaption>
</figure>
```

**Rules:**
- `<figcaption>` must be the first or last child of `<figure>`
- Only one `<figcaption>` per `<figure>`
- The content of `<figure>` should be referenced from the main flow but could be moved to an appendix without disrupting the flow

## Common Mistakes

1. **Div soup.** Using `<div>` for everything when semantic elements exist. The page works visually but is unnavigable for screen reader users and provides no structural signals to search engines.

2. **Using semantic elements for styling.** Choosing `<aside>` because you want a sidebar layout, or `<section>` because you need a wrapper with padding. If the choice is driven by appearance rather than meaning, use `<div>`.

3. **`<section>` without a heading.** A `<section>` that cannot be described by a heading is probably not a section. Without an accessible name, it provides zero benefit over `<div>` to assistive technologies.

4. **Multiple `<h1>` elements.** The document outline algorithm that was supposed to support this was never implemented by any browser or screen reader. It was removed from the spec in 2022. Use one `<h1>` per page.

5. **Skipping heading levels.** Going from `<h2>` to `<h4>` breaks the logical hierarchy. Screen reader users navigate by headings and expect consistent nesting.

6. **Nesting violations.** Placing `<main>` inside `<aside>`, `<header>` inside `<footer>`, or block elements inside `<p>`. The parser will attempt error recovery, often producing unexpected DOM structures.

7. **Missing labels on duplicate landmarks.** Two `<nav>` elements without `aria-label` are both announced as "navigation" with no way to distinguish them.

8. **Using `<address>` for arbitrary postal addresses.** The element is specifically for contact information of the author or organization, not for any address appearing in content.

9. **Treating `<article>` as a layout container.** Only use `<article>` for self-contained, syndicatable content. A grid card that only has a title and "Read more" link is probably not an `<article>`.

10. **Redundant ARIA on semantic elements.** `<nav role="navigation">` adds nothing. The implicit role already exists. Extra ARIA attributes can introduce conflicts with future spec changes.

---

## Best Practices

- **Start with the document skeleton:** `<header>`, `<nav>`, `<main>`, `<footer>`. Add `<aside>`, `<article>`, and `<section>` as the content demands.
- **Maintain a strict heading hierarchy.** One `<h1>`, then `<h2>` for major sections, `<h3>` for subsections. Never skip levels.
- **Label duplicate landmarks.** If you have two `<nav>` elements, give each an `aria-label`.
- **Use `<section>` only when a heading describes the group.** If you are only grouping for styling, use `<div>`.
- **Prefer native HTML over ARIA.** A `<nav>` is better than `<div role="navigation">`. A `<dialog>` is better than `<div role="dialog" aria-modal="true">`.
- **Use `<dialog>` for modals.** It handles focus trapping, inert background, escape key, and top-layer stacking natively.
- **Use `<details>` for disclosures.** FAQ sections, expandable panels, and toggleable sections do not need JavaScript.
- **Test with a screen reader.** VoiceOver (macOS/iOS), NVDA (Windows), or TalkBack (Android). Navigate by landmarks and headings to verify the structure makes sense.

> **Key Takeaway:** Every HTML element is a signal. `<div>` says nothing. Semantic elements tell browsers, screen readers, search engines, and developers what the content means, enabling navigation, indexing, and accessibility without a single line of JavaScript or ARIA.
