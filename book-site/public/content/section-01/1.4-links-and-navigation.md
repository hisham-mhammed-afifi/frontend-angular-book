# 1.4 Links and Navigation

> The `<a>` element is the foundation of the web. Every URL, every page transition, every resource download flows through it. Getting links right means getting navigation, accessibility, security, and SEO right.

---

## The `<a>` Element

The anchor element creates hyperlinks. The `href` attribute defines the destination. Without `href`, the element is not a hyperlink, is not focusable, and is not announced as a link by screen readers.

### `href` Value Types

```html
<!-- Absolute URL -->
<a href="https://example.com/about">About Us</a>

<!-- Root-relative URL (from site root) -->
<a href="/about">About Us</a>

<!-- Relative URL (from current path) -->
<a href="about.html">About Us</a>

<!-- Fragment identifier (same page) -->
<a href="#pricing">Jump to Pricing</a>

<!-- Fragment on another page -->
<a href="/features#pricing">Features Pricing</a>

<!-- Email -->
<a href="mailto:support@example.com">Email Support</a>

<!-- Phone -->
<a href="tel:+14155551234">Call Support</a>

<!-- Download trigger -->
<a href="/files/report.pdf" download>Download Report</a>
```

### `mailto:` Links in Detail

```html
<!-- Basic -->
<a href="mailto:support@example.com">Email Support</a>

<!-- With subject and body -->
<a href="mailto:support@example.com?subject=Bug%20Report&body=Steps%20to%20reproduce%3A%0D%0A1.%20">
  Report a Bug
</a>

<!-- Multiple recipients, CC, and BCC -->
<a href="mailto:team@example.com,manager@example.com?cc=qa@example.com&bcc=log@example.com">
  Email the Team
</a>
```

Parameters: `subject`, `body`, `cc`, `bcc`. The first parameter uses `?`, subsequent ones use `&`. All values must be percent-encoded: spaces become `%20`, line breaks become `%0D%0A`, colons become `%3A`.

**Gotcha:** Not all users have a configured email client. Desktop users on web-only setups may get an error when clicking mailto links. Always provide an alternative contact method (a web form or visible email address).

### `tel:` Links

```html
<!-- International format in href, local display -->
<a href="tel:+14155551234">(415) 555-1234</a>
```

Use international format with `+` and country code in the `href`. The display text can use any formatting. On mobile, the link opens the dialer. On desktop, behavior varies by OS and installed applications.

### Anti-Pattern: `javascript:` in `href`

```html
<!-- BAD: never do this -->
<a href="javascript:void(0)" onclick="doSomething()">Click me</a>
```

Problems with `javascript:` URLs:
- The browser parses the JavaScript in the URL, which is costly and unnecessary
- Violates Content Security Policy (CSP), creating XSS vectors
- No fallback when JavaScript is disabled or fails to load
- Breaks right-click, middle-click, drag, and bookmark behaviors
- Screen readers announce it as a link but it navigates nowhere

### Anti-Pattern: `href="#"`

```html
<!-- BAD: scrolls to top, wrong semantics -->
<a href="#" onclick="toggleMenu()">Toggle Menu</a>

<!-- GOOD: use a button for actions -->
<button type="button" onclick="toggleMenu()">Toggle Menu</button>
```

`href="#"` scrolls the page to the top as a side effect. If the element performs an action rather than navigating somewhere, it is a button, not a link.

### Anti-Pattern: Empty `href=""`

```html
<!-- BAD: reloads the current page -->
<a href="">Do something</a>
```

An empty `href` resolves to the current page URL and triggers a reload.

### Protocol-Relative URLs (Obsolete)

```html
<!-- OBSOLETE: do not use -->
<a href="//example.com/page">Example</a>
```

These inherit the current page's protocol. In an HTTPS-everywhere world, they can downgrade to HTTP if the page is served over HTTP, opening the door for man-in-the-middle attacks. Always use explicit `https://`.

## Transparent Content Model

The `<a>` element has a **transparent** content model. It inherits the content rules of its parent:

```html
<!-- Parent is <div> (allows flow content), so <a> can wrap block elements -->
<div>
  <a href="/article">
    <article class="card">
      <h2>Article Title</h2>
      <p>Description text...</p>
      <img src="thumb.jpg" alt="Thumbnail">
    </article>
  </a>
</div>

<!-- Parent is <p> (allows phrasing content only), so <a> is inline only -->
<p>Read the <a href="/guide"><strong>complete guide</strong></a> for details.</p>
```

### Hard Restrictions

An `<a>` element must never contain:

| Forbidden Content | Why |
|---|---|
| Another `<a>` | Nested anchors are invalid; browser behavior is undefined |
| `<button>` | Interactive content cannot nest inside interactive content |
| `<input>`, `<select>`, `<textarea>` | Form controls are interactive content |
| `<details>` | Interactive content |
| `<iframe>`, `<embed>` | Interactive content |
| Any element with `tabindex` | Creates nested focusable elements |

```html
<!-- INVALID: button inside link -->
<a href="/page">
  <button>Click</button>  <!-- Two interactive elements stacked -->
</a>

<!-- INVALID: nested links -->
<a href="/outer">
  Text <a href="/inner">Inner link</a>
</a>

<!-- VALID: card link wrapping non-interactive content -->
<a href="/product">
  <div class="card">
    <img src="product.jpg" alt="Widget Pro">
    <h3>Widget Pro</h3>
    <p>$49.99</p>
  </div>
</a>
```

## The `target` Attribute

Controls where the linked document opens.

| Value | Behavior |
|---|---|
| `_self` | Same tab (default) |
| `_blank` | New tab or window |
| `_parent` | Parent frame; same as `_self` if no frames |
| `_top` | Topmost frame; same as `_self` if no frames |
| `name` | Opens in a named window/iframe; creates one if it does not exist |

### Security: The Reverse Tabnabbing Story

When `target="_blank"` opened a new tab, the opened page historically had access to `window.opener`, a reference back to the originating page. A malicious site could silently redirect the original tab:

```javascript
// On the malicious page that was opened in a new tab
window.opener.location = 'https://phishing-site.com/fake-login';
```

The user, returning to the original tab, would find what looks like a login page and unknowingly submit credentials to the attacker.

**The fix was `rel="noopener"`**, which sets `window.opener` to `null`.

**Modern browsers now imply `noopener` for all `target="_blank"` links by default.** This is part of the HTML spec since:
- Safari 12.2 (2018)
- Firefox 79 (2020)
- Chrome 88 (2021)

Explicitly adding `rel="noopener"` is now redundant but still commonly written for backward compatibility.

```html
<!-- Modern: noopener is implied, but explicit is fine -->
<a href="https://example.com" target="_blank" rel="noopener">
  External Site (opens in new tab)
</a>
```

## The `rel` Attribute

Defines the relationship between the current document and the linked resource. Accepts space-separated values.

### Values for `<a>` Elements

| Value | Purpose |
|---|---|
| `noopener` | Prevents `window.opener` access. Implied for `target="_blank"` in modern browsers. |
| `noreferrer` | Suppresses the `Referer` header. Also implies `noopener`. |
| `nofollow` | Tells search engines not to follow the link or pass PageRank. Used for user-generated content, ads, and untrusted links. |
| `external` | Indicates an external site. No browser behavior; semantic/SEO signal only. |
| `author` | Links to the author's page. |
| `bookmark` | Permalink to the nearest ancestor `<article>`. |
| `license` | Links to the applicable license. |
| `tag` | A keyword/tag for the document. |
| `help` | Context-sensitive help. |
| `prev` | Previous document in a sequence. |
| `next` | Next document in a sequence. |
| `alternate` | Alternative representation (different language, format). |
| `opener` | Explicitly restores `window.opener` for `target="_blank"` (overrides the default `noopener`). |

### Combining Values

```html
<!-- External untrusted link in new tab -->
<a href="https://user-submitted-site.com"
   target="_blank"
   rel="noopener noreferrer nofollow external">
  User's Website
</a>

<!-- Pagination -->
<a href="/blog?page=2" rel="next">Next Page</a>
<a href="/blog?page=0" rel="prev">Previous Page</a>
```

### SEO Note

`noopener` and `noreferrer` do **not** affect SEO. Only `nofollow` tells search engines not to pass link equity (PageRank). Google treats `nofollow` as a hint, not a directive.

## The `download` Attribute

Tells the browser to download the URL instead of navigating to it.

```html
<!-- Download with the server's filename -->
<a href="/api/export/report.csv" download>Download CSV</a>

<!-- Suggest a custom filename -->
<a href="/api/export/data" download="monthly-report-2025.csv">Download Report</a>
```

### Cross-Origin Restriction

The `download` attribute does **not** work for cross-origin URLs. This is a security measure to prevent malicious sites from tricking users into downloading files from domains they are authenticated with.

| Browser | Cross-origin behavior |
|---|---|
| Chrome | Ignores suggested filename but may still download |
| Firefox | Ignores the entire `download` attribute |
| Safari | Ignores the entire `download` attribute |

**Workaround for cross-origin downloads:**

```javascript
// Fetch the resource, create a blob URL (same-origin), then download
async function downloadCrossOrigin(url, filename) {
  const response = await fetch(url);
  const blob = await response.blob();
  const blobUrl = URL.createObjectURL(blob);

  const link = document.createElement('a');
  link.href = blobUrl;
  link.download = filename;
  link.click();

  URL.revokeObjectURL(blobUrl); // prevent memory leak
}
```

### Works with Blob and Data URLs

```html
<!-- Data URL download -->
<a href="data:text/csv;charset=utf-8,Name%2CAge%0AAlice%2C30%0ABob%2C25"
   download="people.csv">
  Download CSV
</a>
```

Blob URLs created via `URL.createObjectURL()` are same-origin by definition, so `download` works without restrictions.

## The `ping` Attribute

Sends POST requests to specified URLs when the link is clicked. Designed for lightweight click tracking without JavaScript.

```html
<a href="https://example.com"
   ping="https://analytics.example.com/track">
  Visit Example
</a>
```

When clicked, the browser sends a POST request with body `PING` and headers `Ping-From` (originating page) and `Ping-To` (destination).

**Browser support:** All major browsers except Firefox (behind flag `browser.send_pings` due to privacy concerns). Not widely used in production.

---

## The `referrerpolicy` Attribute

Controls how much referrer information is sent with the request.

```html
<!-- Send no referrer at all -->
<a href="https://example.com" referrerpolicy="no-referrer">Link</a>

<!-- Send only the origin, not the full path -->
<a href="https://example.com" referrerpolicy="origin">Link</a>
```

### Policy Values

| Value | Behavior |
|---|---|
| `no-referrer` | No `Referer` header sent |
| `origin` | Send scheme + host + port only, no path |
| `same-origin` | Full URL for same-origin; nothing for cross-origin |
| `strict-origin` | Origin only for same-security; nothing on HTTPS-to-HTTP |
| `strict-origin-when-cross-origin` | Full URL same-origin; origin cross-origin; nothing on downgrade. **Browser default.** |
| `no-referrer-when-downgrade` | Full URL unless HTTPS-to-HTTP downgrade |
| `unsafe-url` | Always full URL (leaks paths over HTTP) |

This attribute overrides the `Referrer-Policy` HTTP header for the specific link.

## The `hreflang` Attribute

Specifies the language of the linked document.

```html
<a href="https://example.com/es/pagina" hreflang="es">Version en espanol</a>
```

Most commonly used on `<link>` elements for SEO:

```html
<link rel="alternate" hreflang="en-us" href="https://example.com/en-us/page">
<link rel="alternate" hreflang="es" href="https://example.com/es/page">
<link rel="alternate" hreflang="x-default" href="https://example.com/">
```

`x-default` indicates the fallback page for users whose language is not otherwise matched. Each language version must reference itself and all other versions.

## Fragment Navigation

Any element with an `id` becomes a fragment target reachable via `#id` in the URL.

```html
<a href="#features">Jump to Features</a>

<!-- Elsewhere on the page -->
<section id="features">
  <h2>Features</h2>
  <p>Content here...</p>
</section>
```

### Smooth Scrolling

```css
html {
  scroll-behavior: smooth;
}

/* Respect user preferences for reduced motion */
@media (prefers-reduced-motion: reduce) {
  html {
    scroll-behavior: auto;
  }
}
```

### Handling Sticky Headers

When a page has a fixed or sticky header, fragment targets scroll behind it. Fix with `scroll-margin-top`:

```css
/* Offset all fragment targets by the header height */
[id] {
  scroll-margin-top: 5rem; /* match your sticky header height */
}
```

This is the modern, clean solution. Older hacks used `::before` pseudo-elements with negative margins.

### The `:target` Pseudo-Class

Styles the element targeted by the current URL fragment:

```css
/* Highlight the jumped-to section */
:target {
  outline: 2px solid #3b82f6;
  outline-offset: 0.5rem;
}

/* Animated highlight that fades out */
:target {
  animation: flash 2s ease-out;
}

@keyframes flash {
  from { background-color: #fef08a; }
  to { background-color: transparent; }
}
```

**Gotcha:** `:target` only activates when the fragment changes via URL navigation (clicking a link or editing the address bar). Programmatic `element.scrollIntoView()` does not trigger `:target`.

**Gotcha:** Each fragment link click adds an entry to the browser history. Pressing Back goes to the previous fragment position, not the previous page.

## Links vs. Buttons

This is one of the most common semantic mistakes on the web.

| | Link (`<a>`) | Button (`<button>`) |
|---|---|---|
| Purpose | Navigation | Action |
| Keyboard trigger | Enter | Enter AND Space |
| Screen reader | "link" | "button" |
| Right-click | "Open in new tab" context menu | No navigation menu |
| Middle-click | Opens in new tab | Nothing |
| Drag | Can drag the URL | Nothing |
| JS disabled | Navigates to `href` | Submits form (if in form) |

### The Rule

If clicking it takes the user somewhere (a URL, a page, a section, a file), it is a **link**. If clicking it does something (submit, delete, toggle, open modal), it is a **button**.

```html
<!-- Navigation: use a link -->
<a href="/settings">Settings</a>
<a href="/reports/annual.pdf" download>Download Report</a>
<a href="#contact">Jump to Contact</a>

<!-- Actions: use a button -->
<button type="button" onclick="toggleDarkMode()">Toggle Dark Mode</button>
<button type="button" onclick="deleteItem(42)">Delete</button>
<button type="submit">Submit Form</button>
```

```html
<!-- BAD: link pretending to be a button -->
<a href="#" onclick="deleteItem(); return false;">Delete</a>

<!-- BAD: button pretending to be a link -->
<button onclick="window.location='/about'">About Us</button>
```

## Styling Links: Pseudo-Class Order

Link pseudo-classes must be declared in a specific order because they have equal specificity and the cascade (source order) determines which wins when multiple states apply simultaneously.

### The LVHFA Order (LoVe HAFte)

```css
/* 1. Unvisited links */
a:link {
  color: #2563eb;
}

/* 2. Visited links */
a:visited {
  color: #7c3aed;
}

/* 3. Hover (mouse over) */
a:hover {
  color: #1d4ed8;
  text-decoration-thickness: 2px;
}

/* 4. Focus (keyboard navigation) */
a:focus-visible {
  outline: 2px solid #2563eb;
  outline-offset: 2px;
}

/* 5. Active (being clicked) */
a:active {
  color: #dc2626;
}
```

### Why Order Matters

A visited link being hovered is in both `:visited` and `:hover` states. If `:visited` is declared after `:hover`, the visited color always wins and hover effects are invisible on visited links. Similarly, `:active` must come after `:hover` because a link being clicked is also being hovered.

### `:visited` Privacy Restrictions

Browsers severely restrict `:visited` styles to prevent history-sniffing attacks.

**Allowed properties only:**
- `color`
- `background-color`
- `border-color` (and individual sides)
- `column-rule-color`
- `outline-color`
- `text-decoration-color`

**Restrictions:**
- Alpha channels are ignored (you cannot change opacity)
- `getComputedStyle()` always returns the unvisited style (JavaScript cannot detect visited links)
- Properties must have a base value on a non-`:visited` rule; `:visited` can only modify existing color values

```css
/* This works: changing an existing color */
a:link { color: blue; }
a:visited { color: purple; }

/* This does NOT work: adding a background that was not set */
a:link { /* no background-color set */ }
a:visited { background-color: yellow; } /* ignored */
```

## Accessibility

### Link Text Quality

Screen reader users often navigate by pulling up a list of all links on the page (VoiceOver Rotor, NVDA Elements List). Out of context, link text must describe the destination.

```html
<!-- BAD: meaningless out of context -->
<a href="/report">Click here</a>
<a href="/report">Read more</a>
<a href="/report">Link</a>
<a href="/report">Here</a>

<!-- GOOD: descriptive destination -->
<a href="/report">2025 Annual Report (PDF, 2.3 MB)</a>
<a href="/pricing">View pricing plans</a>
<a href="/docs/api">API documentation</a>
```

### Image Links Need Alt Text

When a link wraps an image, the `alt` attribute serves as the link's accessible name. An empty `alt=""` creates a link with no name, which is a WCAG failure.

```html
<!-- BAD: no accessible name -->
<a href="/"><img src="logo.svg" alt=""></a>

<!-- GOOD: alt describes the link destination -->
<a href="/"><img src="logo.svg" alt="Acme Corp homepage"></a>
```

### `aria-label` and Label in Name

`aria-label` overrides the visible text for screen readers. When used, the accessible name should start with or closely match the visible text (WCAG 2.5.3 Label in Name):

```html
<!-- BAD: aria-label doesn't match visible text -->
<a href="/settings" aria-label="Open configuration panel">Settings</a>

<!-- GOOD: aria-label starts with the visible text -->
<a href="/settings" aria-label="Settings page">Settings</a>

<!-- BEST: no aria-label needed, visible text is sufficient -->
<a href="/settings">Settings</a>
```

### `aria-current` for Navigation

Indicates the current page in a navigation menu:

```html
<nav aria-label="Main">
  <a href="/" aria-current="page">Home</a>
  <a href="/products">Products</a>
  <a href="/about">About</a>
</nav>
```

Screen readers announce "Home, current page, link." Also useful as a CSS hook:

```css
a[aria-current="page"] {
  font-weight: 700;
  border-bottom: 2px solid currentColor;
}
```

Values: `page`, `step`, `location`, `date`, `time`, `true`.

### Skip Links

WCAG 2.4.1 (Level A) requires a mechanism to bypass repeated blocks of content. A skip link lets keyboard users jump past the header and navigation directly to the main content.

```html
<body>
  <a href="#main" class="skip-link">Skip to main content</a>
  <header>
    <nav><!-- long navigation --></nav>
  </header>
  <main id="main" tabindex="-1">
    <!-- page content -->
  </main>
</body>
```

```css
.skip-link {
  position: absolute;
  left: -9999px;
  top: auto;
  z-index: 1000;
  padding: 0.5rem 1rem;
  background: #000;
  color: #fff;
  text-decoration: none;
}

.skip-link:focus {
  left: 1rem;
  top: 1rem;
}
```

The skip link must be:
- The **first focusable element** on the page
- Hidden off-screen by default
- Visible when focused via keyboard (Tab)
- Targeting an element with `tabindex="-1"` to ensure older browsers move focus correctly

### Focus Indicators

```css
/* Show focus ring for keyboard users, not mouse users */
a:focus-visible {
  outline: 2px solid #2563eb;
  outline-offset: 2px;
  border-radius: 2px;
}

/* Remove default outline only if focus-visible is supported */
a:focus:not(:focus-visible) {
  outline: none;
}
```

WCAG 2.4.7 (Level A) requires visible focus indicators. WCAG 2.4.13 (Level AAA) requires at least 2px thickness and 3:1 contrast ratio. **Never** use `outline: none` without a visible replacement.

### New Tab Warning

When a link opens a new tab, users with cognitive disabilities may be disoriented. Provide a warning:

```html
<a href="https://example.com" target="_blank" rel="noopener">
  External Documentation
  <span class="visually-hidden">(opens in new tab)</span>
</a>
```

```css
.visually-hidden {
  clip: rect(0 0 0 0);
  clip-path: inset(50%);
  height: 1px;
  width: 1px;
  overflow: hidden;
  position: absolute;
  white-space: nowrap;
}
```

### Touch Target Size

WCAG 2.5.8 (Level AA in WCAG 2.2) requires interactive elements to have a minimum target size of 24x24 CSS pixels. Links placed close together (like in a nav or footer) need sufficient padding or margin.

```css
nav a {
  display: inline-block;
  padding: 0.5rem 1rem; /* ensures 24px+ touch target */
  min-height: 44px;     /* Apple's recommended minimum */
  line-height: 44px;
}
```

## Image Maps (Brief)

Image maps make regions of an image clickable. They are largely obsolete but still in the spec.

```html
<img src="floor-plan.png"
     alt="Office floor plan with clickable rooms"
     usemap="#office"
     width="800"
     height="600">

<map name="office">
  <area shape="rect" coords="50,50,200,200"
        alt="Conference Room A" href="/rooms/conf-a">
  <area shape="circle" coords="400,300,50"
        alt="Break Room" href="/rooms/break">
  <area shape="poly" coords="600,100,750,100,750,250,600,250"
        alt="Server Room" href="/rooms/server">
</map>
```

| Shape | Coords Format |
|---|---|
| `rect` | x1, y1, x2, y2 (top-left, bottom-right) |
| `circle` | x, y, radius |
| `poly` | x1, y1, x2, y2, ..., xn, yn (polygon vertices) |
| `default` | No coords (entire image) |

Image maps are not responsive (coordinates are pixel-based), have poor accessibility, and are difficult to maintain. Modern alternatives: SVG with embedded links, CSS-positioned links over images, or separate linked elements.

## Common Mistakes

1. **`href="#"` or `javascript:void(0)` as button replacements.** If the element performs an action (toggle, delete, submit), use `<button>`. Links are for navigation.

2. **Empty `href=""`.** Resolves to the current page URL and causes a reload. Remove the attribute or use a button.

3. **"Click here" link text.** Meaningless when screen reader users browse a link list. Describe the destination.

4. **Image link without alt text.** Creates a link with no accessible name. The `alt` attribute on the image IS the link text for screen readers.

5. **Nested interactive content.** Placing `<button>`, `<input>`, or another `<a>` inside a link. The HTML spec forbids this and browser behavior is undefined.

6. **Opening new tabs without warning.** `target="_blank"` without "(opens in new tab)" text disorients users, especially those with cognitive disabilities.

7. **Removing focus indicators.** `outline: none` without a visible replacement makes the site unusable for keyboard navigators.

8. **Using protocol-relative URLs.** `//example.com` can downgrade to HTTP. Always specify `https://`.

9. **Links too close together without sufficient spacing.** Touch target sizes below 24x24px cause problems for users with motor control difficulties.

10. **`<button onclick="location='...'">` for navigation.** Loses right-click, middle-click, bookmark, and drag behaviors that users expect from links.

---

## Best Practices

- **Use descriptive link text.** Every link should make sense out of context.
- **Use `<a>` for navigation, `<button>` for actions.** No exceptions.
- **Add `rel="noopener"` to `target="_blank"` links.** Technically redundant in modern browsers but still commonly expected.
- **Add `rel="nofollow"` to untrusted/user-generated links.** Prevents passing PageRank to spam.
- **Always provide visible focus indicators.** Use `:focus-visible` to show them only for keyboard users.
- **Implement skip links** as the first focusable element for WCAG 2.4.1 compliance.
- **Use `aria-current="page"` on active navigation items.** Benefits screen readers and provides a CSS styling hook.
- **Use `scroll-margin-top` on fragment targets** to offset sticky headers.
- **Respect `prefers-reduced-motion`** when enabling smooth scroll.
- **Provide file type and size** in download link text: "Annual Report (PDF, 2.3 MB)".
