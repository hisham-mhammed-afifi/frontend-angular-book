# 2.17 Modern CSS Features

> CSS has undergone a radical transformation since 2022. Container queries, `:has()`, nesting, cascade layers, subgrid, view transitions, and native popovers are not experimental curiosities. They are production-ready tools that eliminate entire categories of JavaScript and preprocessor workarounds.

---

## Container Queries

Media queries respond to the viewport. Container queries respond to the size of a parent element. This is the difference between "how wide is the screen?" and "how much space does this component actually have?"

### Declaring a Container

```css
/* The parent must be explicitly declared as a container */
.card-wrapper {
  container-type: inline-size;
}

/* Shorthand: container: <name> / <type> */
.sidebar {
  container: sidebar / inline-size;
}
```

| `container-type` | Description |
|-------------------|-------------|
| `normal` | Not a query container (default) |
| `inline-size` | Containment on the inline axis only (width in horizontal writing). The safe default. |
| `size` | Containment on both axes. Can cause collapse. Use sparingly. |

### Querying a Container

```css
/* Unnamed: queries nearest ancestor container */
@container (min-width: 400px) {
  .card {
    flex-direction: row;
  }
}

/* Named container query */
@container sidebar (inline-size > 30em) {
  .nav-item {
    font-size: 1.2rem;
  }
}

/* Range syntax */
@container (300px <= width <= 600px) {
  .card {
    grid-template-columns: 1fr 1fr;
  }
}
```

### Why Not `container-type: size`?

`size` establishes containment on both axes, which means the browser cannot use the container's children to determine its block size. If nothing else sets the height, the container collapses to zero.

```css
/* Bad Practice: causes collapse unless height is set elsewhere */
.panel {
  container-type: size;
}

/* Good: inline-size only */
.panel {
  container-type: inline-size;
}
```

### The Fundamental Rule

You cannot query the container from itself. Only children of the container can use `@container` to query it. The container and the queried element must be different elements.

```html
<!-- WRONG: .card queries itself -->
<div class="card" style="container-type: inline-size;">
  <!-- @container rules targeting .card will NOT work here -->
</div>

<!-- CORRECT: wrapper is the container, card is the child -->
<div class="card-wrapper" style="container-type: inline-size;">
  <div class="card">
    <!-- @container rules targeting .card WILL work -->
  </div>
</div>
```

**Browser support:** Baseline Widely Available (Feb 2023). Chrome 105+, Firefox 110+, Safari 16+.

## The :has() Selector

`:has()` is the "parent selector" CSS never had. It selects an element based on what it contains or what follows it.

### Core Patterns

```css
/* Style a card that contains an image */
.card:has(img) {
  grid-template-rows: 300px 1fr;
}

/* Style a form when any input is invalid */
form:has(:invalid) {
  border: 2px solid red;
}

/* Style a label when its adjacent input is focused */
label:has(+ input:focus) {
  color: blue;
  font-weight: bold;
}

/* Card WITHOUT an image */
.card:not(:has(img)) {
  padding: 2rem;
}

/* Quantity query: style a list with 4+ items */
ul:has(li:nth-child(4)) {
  columns: 2;
}

/* Multiple conditions */
.card:has(img):has(.badge) {
  border: 2px solid gold;
}
```

### Performance Consideration

`:has()` forces the browser to look at descendants to determine whether to style an ancestor. Broad selectors are expensive:

```css
/* Bad Practice: checks every element in the DOM */
*:has(.error) {
  outline: 2px solid red;
}

/* Good: narrow the anchor element */
.form-field:has(.error) {
  outline: 2px solid red;
}

/* Better: use direct child combinator to limit subtree traversal */
.form-field:has(> .error) {
  outline: 2px solid red;
}
```

**Browser support:** Baseline Widely Available (Dec 2023). Chrome 105+, Firefox 121+, Safari 15.4+.

## Native CSS Nesting

No preprocessor needed. CSS now supports nesting selectors inside rule blocks.

```css
.card {
  background: white;
  border-radius: 8px;

  .title {
    font-size: 1.5rem;
    font-weight: bold;
  }

  .body {
    padding: 1rem;
    line-height: 1.6;
  }

  &:hover {
    box-shadow: 0 2px 8px rgb(0 0 0 / 0.15);
  }

  &::before {
    content: "";
    display: block;
  }

  /* Media queries nest directly */
  @media (width > 768px) {
    flex-direction: row;
  }
}
```

### Element Selectors Require `&`

Unlike SASS, you cannot nest a bare element selector. It must be prefixed with `&`.

```css
.card {
  /* INVALID in native CSS: */
  /* p { color: #333; } */

  /* Correct: */
  & p {
    color: #333;
  }
}
```

### Key Differences from SASS Nesting

| Behavior | Native CSS | SASS |
|----------|-----------|------|
| `&` suffix (`&-modifier`) | NOT string concatenation. `.card` + `&-dark` is invalid in the expected sense. | Concatenates to `.card-dark` |
| Specificity of grouped selectors | Wrapped in `:is()`, inheriting highest specificity | Each selector computed independently |
| Element selectors without `&` | Invalid. Must use `& p` | `p` works inside a block |

The specificity wrapping is the most impactful difference. When you nest inside a grouping selector, the parent is wrapped in `:is()`:

```css
/* Written: */
.foo, #bar {
  .baz { color: red; }
}

/* Native CSS equivalent: */
:is(.foo, #bar) .baz { color: red; }
/* Specificity is ALWAYS (1,0,1) because :is() takes the highest specificity (#bar) */

/* In SASS, it would produce: */
.foo .baz { color: red; }    /* specificity (0,1,1) */
#bar .baz { color: red; }    /* specificity (1,0,1) */
```

### BEM Does Not Work with Native Nesting

```css
.card {
  /* This does NOT produce .card__title */
  &__title { color: red; }

  /* It produces: .card__title -- wait, actually it IS valid
     but ONLY as a new selector, not string concatenation.
     The & represents the FULL selector .card, not a string.
     So &__title means elements matching .card that also match __title,
     which is not a valid class name without the dot. */
}
```

If you rely on BEM's `&__` and `&--` pattern, you still need SASS or a build tool.

**Browser support:** Baseline Widely Available (Aug 2024). Chrome 120+, Firefox 117+, Safari 17.2+.

## Cascade Layers (@layer)

Cascade layers give you explicit control over which styles win, independent of specificity.

### Declaring Layer Order

```css
/* Declare order upfront. This line locks the priority. */
@layer reset, base, components, utilities;

/* Now add styles to layers in any order */
@layer components {
  .btn {
    padding: 0.5rem 1rem;
    background: blue;
    color: white;
  }
}

@layer utilities {
  .bg-red { background: red; }
}

@layer reset {
  *, *::before, *::after {
    margin: 0;
    padding: 0;
    box-sizing: border-box;
  }
}

@layer base {
  body { font-family: system-ui; line-height: 1.6; }
}
```

### Priority Rules

| Priority (highest to lowest) | Source |
|-------------------------------|--------|
| 1 | `!important` in earliest layer |
| 2 | `!important` in latest layer |
| 3 | `!important` unlayered |
| 4 | Inline styles |
| 5 | Unlayered styles |
| 6 | Latest declared layer |
| ... | ... |
| N | Earliest declared layer |

The critical insight: **unlayered styles always beat layered styles** (for non-`!important` declarations). And **`!important` reverses the layer order**.

```css
@layer base, theme;

@layer base {
  .btn { color: gray; }
  .critical { color: red !important; } /* !important in FIRST layer wins */
}

@layer theme {
  .btn { color: blue; } /* Normal: theme layer wins over base */
  .critical { color: green !important; } /* !important: base layer wins over theme */
}
```

### Importing into Layers

```css
/* Put third-party CSS into a low-priority layer */
@import url("normalize.css") layer(reset);
@import url("bootstrap.css") layer(framework);

/* Your styles are unlayered, so they always win */
.btn {
  background: var(--brand-color); /* overrides bootstrap without specificity fights */
}
```

### Nested Layers

```css
@layer framework {
  @layer base, theme;

  @layer base {
    body { font-size: 16px; }
  }

  @layer theme {
    body { color: #333; }
  }
}

/* Reference nested layers with dot notation */
@layer framework.theme {
  body { color: #111; }
}
```

**Browser support:** Baseline Widely Available (Mar 2022). Chrome 99+, Firefox 97+, Safari 15.4+.

## Subgrid

Subgrid lets a grid child adopt the track definitions of its parent grid, so items across sibling elements align perfectly.

### The Problem Subgrid Solves

```css
/* Without subgrid: each card has its own independent grid */
.card-grid {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 1rem;
}

.card {
  display: grid;
  grid-template-rows: auto 1fr auto;
  /* Each card's rows are sized independently.
     Headers, bodies, and footers do NOT align across cards. */
}
```

### The Solution

```css
.card-grid {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  grid-template-rows: auto 1fr auto; /* Shared row tracks */
  gap: 1rem;
}

.card {
  display: grid;
  grid-row: span 3;              /* Must span the tracks being borrowed */
  grid-template-rows: subgrid;   /* Borrow parent's row tracks */
  gap: 0.5rem;                   /* Can override gap */
}
```

Now every card's header aligns with every other card's header, body content fills the middle row, and footers sit at the same baseline.

### Subgrid vs Regular Nested Grid

| Feature | Regular nested grid | Subgrid |
|---------|-------------------|---------|
| Track sizing | Independent | Inherited from parent |
| Cross-sibling alignment | No | Yes |
| Gap | Independent | Inherited (overridable) |
| Line names | Separate | Inherited from parent |

### One Axis at a Time

You can use `subgrid` on one axis and define your own tracks on the other:

```css
.card {
  display: grid;
  grid-row: span 3;
  grid-template-rows: subgrid;           /* Borrow row tracks */
  grid-template-columns: 80px 1fr;       /* Define own columns */
}
```

**Browser support:** Baseline Widely Available (Dec 2023). Chrome 117+, Firefox 71+, Safari 16+.

## @scope

`@scope` limits where styles apply in the DOM tree, with optional upper and lower boundaries.

### Basic Scope

```css
@scope (.card) {
  .title { font-size: 1.5rem; }
  .body { padding: 1rem; }
  img { border-radius: 8px; }
}

/* These styles only apply inside .card elements.
   A .title outside a .card is unaffected. */
```

### Donut Scope (Upper and Lower Boundaries)

```css
@scope (.article) to (.comments) {
  /* Styles apply inside .article but NOT inside .comments */
  p { line-height: 1.8; }
  a { color: blue; }
}

/* The .comments section and its descendants are excluded */
```

### Proximity-Based Specificity

When two `@scope` rules match the same element, the one whose scope root is closer in the DOM tree wins. This is a new cascade mechanism called **scoping proximity**.

```css
@scope (.light-theme) {
  p { color: #111; }
}

@scope (.dark-theme) {
  p { color: #eee; }
}
```

```html
<div class="light-theme">
  <div class="dark-theme">
    <p>This text is #eee because .dark-theme is closer.</p>
  </div>
</div>
```

No specificity hacks, no `!important`, no JavaScript. The closer scope root wins.

### Referencing the Scope Root

```css
@scope (.card) {
  :scope {
    /* Styles the .card element itself */
    border: 1px solid #ccc;
    border-radius: 8px;
  }
}
```

**Browser support:** Baseline Newly Available (mid-2025). Chrome 118+, Firefox 146+, Safari 17.4+.

## Anchor Positioning

Anchor positioning connects a positioned element (tooltip, popover, dropdown) to another element (its anchor) purely through CSS. No JavaScript positioning libraries needed.

### Setup

```css
/* Step 1: Name the anchor */
.trigger-btn {
  anchor-name: --my-trigger;
}

/* Step 2: Position relative to the anchor */
.tooltip {
  position: absolute;
  position-anchor: --my-trigger;

  /* Place it above the anchor, centered */
  position-area: top center;

  /* Fallback positions when it would overflow the viewport */
  position-try-fallbacks: flip-block, flip-inline;
}
```

### The position-area Grid

`position-area` uses a conceptual 3x3 grid around the anchor:

```
top left      | top center     | top right
center left   | center center  | center right
bottom left   | bottom center  | bottom right
```

### The anchor() Function (Fine Control)

```css
.dropdown {
  position: absolute;
  position-anchor: --toggle;

  /* Place the top of the dropdown at the bottom of the anchor */
  top: anchor(bottom);

  /* Align the left edge of the dropdown with the left edge of the anchor */
  left: anchor(left);

  /* Set width to match anchor width */
  width: anchor-size(width);
}
```

### Custom Fallback Positions

```css
@position-try --above-centered {
  position-area: top center;
}

@position-try --below-right {
  position-area: bottom right;
}

.tooltip {
  position-try-fallbacks: --above-centered, --below-right, flip-block;
}
```

**Browser support:** Limited. Chrome 125+, Edge 125+. **Not supported in Firefox or Safari** as of early 2026. Use progressive enhancement or a polyfill (OddBird provides one).

## View Transitions API

Smooth animated transitions between visual states, either within a single-page app or across full page navigations.

### SPA Transitions (JavaScript Trigger)

```js
document.startViewTransition(() => {
  // Update the DOM here
  updatePageContent();
});
```

### MPA Transitions (CSS-Only, Cross-Document)

```css
/* Opt in on both the source and destination pages */
@view-transition {
  navigation: auto;
}
```

### Naming Elements for Morphing

```css
/* On the list page */
.product-thumbnail {
  view-transition-name: product-hero;
}

/* On the detail page */
.product-hero-image {
  view-transition-name: product-hero;
}

/* The browser morphs between these two elements automatically */
```

### Customizing the Animation

```css
::view-transition-old(product-hero) {
  animation: fade-out 300ms ease;
}

::view-transition-new(product-hero) {
  animation: fade-in 300ms ease;
}

/* Whole-page cross-fade */
::view-transition-old(root) {
  animation: slide-to-left 300ms ease;
}

::view-transition-new(root) {
  animation: slide-from-right 300ms ease;
}
```

### Pseudo-Element Tree

```
::view-transition
  ::view-transition-group(name)
    ::view-transition-image-pair(name)
      ::view-transition-old(name)   /* Screenshot of old state */
      ::view-transition-new(name)   /* Live new state */
```

### Accessibility

```css
@media (prefers-reduced-motion: reduce) {
  ::view-transition-group(*),
  ::view-transition-old(*),
  ::view-transition-new(*) {
    animation: none !important;
  }
}
```

**Critical rule:** `view-transition-name` must be unique per page. Two elements with the same name causes the transition to fail.

**Browser support:** SPA transitions: Chrome 111+, Safari 18+. MPA transitions: Chrome 126+ only. Firefox does not support either as of early 2026.

## Popover API

Native HTML popovers without a single line of JavaScript.

### HTML-Only Popover

```html
<button popovertarget="info">Show Info</button>
<div id="info" popover>
  <p>Click outside or press Escape to close.</p>
</div>
```

That is it. No JavaScript, no ARIA attributes to manage, no focus-trapping code.

### Popover Types

| Attribute | Behavior |
|-----------|----------|
| `popover` or `popover="auto"` | Light dismiss (click outside or Esc). Only one `auto` popover open at a time. |
| `popover="manual"` | No light dismiss. Must be closed programmatically or via a toggle button. Multiple can coexist. |
| `popover="hint"` | For tooltips. Auto-dismissed. Chrome 135+ only. |

### Explicit Show/Hide/Toggle

```html
<button popovertarget="menu" popovertargetaction="show">Open</button>
<button popovertarget="menu" popovertargetaction="hide">Close</button>
<button popovertarget="menu" popovertargetaction="toggle">Toggle</button>

<div id="menu" popover>
  <nav>Menu content</nav>
</div>
```

### Styling Popovers

```css
/* Popovers live in the top layer. Default UA styles center them. */
[popover] {
  /* Override the default positioning */
  margin: 0;
  position: fixed;
  top: 4rem;
  right: 1rem;

  background: white;
  border: 1px solid #ccc;
  border-radius: 8px;
  padding: 1rem;
  box-shadow: 0 4px 16px rgb(0 0 0 / 0.15);
}

/* Style when open */
[popover]:popover-open {
  opacity: 1;
  transform: translateY(0);
}

/* Backdrop behind the popover */
[popover]::backdrop {
  background: rgb(0 0 0 / 0.3);
}
```

### Animating Popovers with @starting-style

```css
[popover] {
  opacity: 0;
  transform: translateY(-10px);
  transition: opacity 200ms ease,
              transform 200ms ease,
              display 200ms allow-discrete;

  @starting-style {
    &:popover-open {
      opacity: 0;
      transform: translateY(-10px);
    }
  }
}

[popover]:popover-open {
  opacity: 1;
  transform: translateY(0);
}
```

**Browser support:** Baseline Widely Available (Apr 2025). Chrome 114+, Firefox 125+, Safari 17+.

## @starting-style

`@starting-style` defines the initial state for CSS transitions when an element first appears. It solves the long-standing problem of animating elements into view.

### The Problem

Without `@starting-style`, an element that appears via `display: none` to `display: block` has no "from" state for transitions. The transition cannot happen because the element was not in the rendering tree.

### The Solution

```css
.notification {
  opacity: 1;
  transform: translateY(0);
  transition: opacity 300ms, transform 300ms;

  @starting-style {
    opacity: 0;
    transform: translateY(-20px);
  }
}
```

When the `.notification` element is inserted into the DOM (or changes from `display: none` to `display: block`), it starts at `opacity: 0; transform: translateY(-20px)` and transitions to its computed values.

### Entry AND Exit Animations

Entry and exit animations require four pieces working together:

```css
dialog[open] {
  opacity: 1;
  transform: scale(1);
  transition: opacity 300ms ease,
              transform 300ms ease,
              overlay 300ms ease allow-discrete,
              display 300ms ease allow-discrete;

  /* Entry: start from these values */
  @starting-style {
    opacity: 0;
    transform: scale(0.9);
  }
}

/* Exit: transition TO these values */
dialog:not([open]) {
  opacity: 0;
  transform: scale(0.9);
}
```

| Piece | Purpose |
|-------|---------|
| `@starting-style` | Defines the "from" state for entry transitions |
| `transition-behavior: allow-discrete` | Lets `display` and `overlay` participate in transitions |
| `display: none` on exit | The "to" state for exit |
| `overlay` in transition list | Keeps top-layer elements visible during exit animation |

**Browser support:** Baseline Newly Available (mid-2024). Chrome 117+, Firefox 129+, Safari 17.5+.

## Scroll-Driven Animations

Animations driven by scroll position instead of time. No JavaScript, no Intersection Observer.

### Scroll Progress Timeline

```css
/* A progress bar that fills as you scroll */
.progress-bar {
  position: fixed;
  top: 0;
  left: 0;
  height: 4px;
  background: var(--accent);
  transform-origin: left;
  animation: grow linear;
  animation-timeline: scroll();
}

@keyframes grow {
  from { transform: scaleX(0); }
  to { transform: scaleX(1); }
}
```

### View Progress Timeline

Animate elements as they enter or leave the viewport:

```css
.card {
  animation: slide-up linear;
  animation-timeline: view();
  animation-range: entry 0% entry 100%;
}

@keyframes slide-up {
  from {
    opacity: 0;
    transform: translateY(50px);
  }
  to {
    opacity: 1;
    transform: translateY(0);
  }
}
```

### animation-range Values

| Value | When |
|-------|------|
| `entry` | Element entering the scrollport |
| `exit` | Element exiting the scrollport |
| `contain` | Element fully within the scrollport |
| `cover` | First pixel entering to last pixel exiting |

### Named Scroll Timeline

```css
.scroll-container {
  overflow-y: auto;
  scroll-timeline: --main-scroll block;
}

.animated-child {
  animation: fade-in linear;
  animation-timeline: --main-scroll;
}
```

### Critical Gotcha

The `animation` shorthand resets `animation-timeline` to `auto`. Always declare `animation-timeline` AFTER the shorthand:

```css
/* WRONG: animation-timeline gets reset */
.element {
  animation-timeline: view();
  animation: fade-in linear;
}

/* CORRECT: animation-timeline after shorthand */
.element {
  animation: fade-in linear;
  animation-timeline: view();
}
```

**Browser support:** Limited. Chrome 115+, Edge 115+. Not in Firefox (behind flag) or Safari as of early 2026.

## color-mix() and light-dark()

### color-mix()

Mix two colors in any color space:

```css
:root {
  --brand: #3b82f6;

  /* Generate palette from one color */
  --brand-light: color-mix(in oklch, var(--brand), white 30%);
  --brand-dark: color-mix(in oklch, var(--brand), black 20%);

  /* Semi-transparent variant */
  --brand-50: color-mix(in srgb, var(--brand), transparent 50%);
}

.btn:hover {
  /* Darken on hover */
  background: color-mix(in oklch, var(--brand), black 15%);
}
```

Always prefer `oklch` for mixing. Mixing in `srgb` or `hsl` produces muddy results with complementary colors.

### light-dark()

Returns the first value in light mode, the second in dark mode:

```css
:root {
  color-scheme: light dark; /* REQUIRED for light-dark() to work */
}

body {
  background: light-dark(#ffffff, #1a1a1a);
  color: light-dark(#111111, #eeeeee);
}

.card {
  border: 1px solid light-dark(#ddd, #444);
  background: light-dark(#f9f9f9, #2a2a2a);
}
```

Without `color-scheme: light dark` on the element or an ancestor, `light-dark()` always returns the first (light) value.

**Browser support:** `color-mix()` Widely Available (May 2023). `light-dark()` Newly Available (2024).

## interpolate-size and calc-size()

The "animate to `height: auto`" problem, finally solved.

### The One-Line Fix

```css
:root {
  interpolate-size: allow-keywords;
}

/* Now this just works */
.accordion-panel {
  height: 0;
  overflow: hidden;
  transition: height 300ms ease;
}

.accordion-panel.open {
  height: auto; /* Smoothly animates from 0 to auto */
}
```

### calc-size() for Math with Intrinsic Sizes

```css
.element {
  /* 80% of fit-content width */
  width: calc-size(fit-content, size * 0.8);

  /* Auto height plus extra padding room */
  max-height: calc-size(auto, size + 2rem);
}
```

**Browser support:** Limited. Chrome 129+, Edge 129+. **Not in Firefox or Safari** as of early 2026. Use progressive enhancement.

## text-wrap: balance and pretty

### balance

Distributes text across lines to make them roughly equal in width. Ideal for headings.

```css
h1, h2, h3, h4 {
  text-wrap: balance;
}
```

Without `balance`, a heading might render as:
```
The Quick Brown Fox Jumps Over the
Lazy Dog
```

With `balance`:
```
The Quick Brown Fox
Jumps Over the Lazy Dog
```

Browsers limit `balance` to a small number of lines (Chrome: 6, Firefox: 10). It is not intended for body text.

### pretty

Prevents orphans (single words on the last line) in paragraphs:

```css
p {
  text-wrap: pretty;
}
```

Without `pretty`:
```
Lorem ipsum dolor sit amet consectetur adipiscing elit sed do eiusmod
tempor.
```

With `pretty`:
```
Lorem ipsum dolor sit amet consectetur
adipiscing elit sed do eiusmod tempor.
```

**Browser support:** `balance` Widely Available. `pretty` Chrome 117+, Safari 17.4+, not in Firefox.

## Feature Detection with @supports

Always wrap newer features in `@supports` when the fallback behavior is significantly different:

```css
/* Fallback: standard fixed positioning */
.tooltip {
  position: fixed;
  top: 50%;
  left: 50%;
  transform: translate(-50%, -50%);
}

/* Enhancement: anchor positioning */
@supports (position-anchor: --x) {
  .tooltip {
    position: absolute;
    position-anchor: --trigger;
    position-area: top center;
  }
}

/* Container query support check */
@supports (container-type: inline-size) {
  .card-wrapper {
    container-type: inline-size;
  }
}
```

## Common Mistakes

- **Using `container-type: size` by default**: This causes elements to collapse because it establishes block-axis containment. Use `inline-size` unless you explicitly need to query the block size and are setting a fixed height.

- **Not realizing `:has()` selects the element it's on, not the argument**: `div:has(p)` styles the `div`, not the `p`. The argument is the condition, not the target.

- **Expecting BEM `&__` and `&--` to work in native nesting**: Native CSS nesting does not concatenate strings. `&__title` does not produce `.card__title` like SASS does.

- **Forgetting unlayered styles beat all layers**: If you put a framework in `@layer` but leave your custom styles unlayered, the custom styles win. This is actually the intended behavior, but it confuses developers who expect layers to behave like specificity tiers.

- **Using identical `view-transition-name` on two visible elements**: The transition fails silently. Each name must be unique on the page at any given time.

- **Applying scroll-driven animation timeline before the shorthand**: The `animation` shorthand resets `animation-timeline`. Always put `animation-timeline` after `animation`.

- **Ignoring `prefers-reduced-motion`**: Every animation feature (view transitions, scroll-driven animations, `@starting-style`) must respect this preference.

## Best Practices

- **Use `@layer` to manage third-party CSS**: Import frameworks into a named layer so your unlayered styles automatically override them without specificity battles.

- **Prefer `:has(> child)` over `:has(descendant)`**: The direct child combinator limits the depth of subtree traversal, improving performance.

- **Limit CSS nesting to 3 levels**: Deeper nesting hurts readability and inflates specificity through the `:is()` wrapping mechanism.

- **Feature-detect before using limited-support features**: Wrap anchor positioning, scroll-driven animations, and `interpolate-size` in `@supports` blocks with reasonable fallbacks.

- **Use `oklch` as your default color space**: For `color-mix()`, gradients, and palette generation, `oklch` produces perceptually consistent results.

- **Set `interpolate-size: allow-keywords` on `:root`**: One line enables smooth height/width transitions to `auto` across your entire site (Chromium only for now, but harmless elsewhere).

- **Combine popover + anchor positioning + @starting-style**: This trio creates fully CSS-driven, animated, properly positioned popovers with no JavaScript.

## Real-World Use Cases

### 1. Responsive Card Component (Container Queries + Subgrid)

```css
.card-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(min(100%, 300px), 1fr));
  grid-template-rows: auto 1fr auto;
  gap: 1.5rem;
}

.card-wrapper {
  container-type: inline-size;
  display: grid;
  grid-row: span 3;
  grid-template-rows: subgrid;
}

.card {
  display: contents;
}

.card-header { /* Aligned across all cards via subgrid */ }
.card-body   { /* Fills middle row */ }
.card-footer { /* Aligned at bottom */ }

@container (min-width: 500px) {
  .card {
    display: grid;
    grid-template-columns: 200px 1fr;
  }
}
```

### 2. Theme System (light-dark + color-mix + @layer)

```css
@layer tokens, base, components;

@layer tokens {
  :root {
    color-scheme: light dark;
    --brand: oklch(0.6 0.2 250);
    --surface: light-dark(#ffffff, #1a1a2e);
    --text: light-dark(#111, #eee);
    --border: light-dark(
      color-mix(in oklch, var(--brand), white 70%),
      color-mix(in oklch, var(--brand), black 60%)
    );
    --hover: color-mix(in oklch, var(--brand), black 15%);
  }
}

@layer base {
  body {
    background: var(--surface);
    color: var(--text);
  }
}

@layer components {
  .btn-primary {
    background: var(--brand);
    color: white;
    border: 1px solid var(--border);
  }

  .btn-primary:hover {
    background: var(--hover);
  }
}
```

### 3. Animated Tooltip (Popover + Anchor + @starting-style)

```html
<button popovertarget="tip" class="trigger">Hover me</button>
<div id="tip" popover="hint" class="tooltip">Helpful information</div>
```

```css
.trigger {
  anchor-name: --trigger;
}

.tooltip {
  position: fixed;
  position-anchor: --trigger;
  position-area: top center;
  position-try-fallbacks: flip-block;

  margin: 0;
  padding: 0.5rem 1rem;
  background: #333;
  color: white;
  border-radius: 6px;
  font-size: 0.875rem;
  width: max-content;
  max-width: 250px;

  opacity: 1;
  transform: translateY(0);
  transition: opacity 150ms ease,
              transform 150ms ease,
              display 150ms allow-discrete;

  @starting-style {
    &:popover-open {
      opacity: 0;
      transform: translateY(4px);
    }
  }
}

/* Fallback for browsers without anchor positioning */
@supports not (position-anchor: --x) {
  .tooltip {
    position: fixed;
    top: auto;
    left: 50%;
    transform: translateX(-50%);
  }
}
```

> **Key Takeaway:** Modern CSS has absorbed capabilities that previously required JavaScript libraries, CSS preprocessors, and complex workarounds. Container queries, `:has()`, cascade layers, and native nesting are production-ready today. Anchor positioning, scroll-driven animations, and view transitions are the next wave. Build with progressive enhancement: use `@supports` for limited-support features and always respect `prefers-reduced-motion`.
