# 2.2 Specificity and the Cascade

> The cascade is the algorithm that decides which CSS declaration wins when multiple rules compete for the same property on the same element. Specificity is just one step in that algorithm, and misunderstanding either leads to `!important` wars and fragile stylesheets.

---

## The Cascade Algorithm

The cascade resolves conflicts through a strict sequence of steps. If a winner is found at any step, later steps are never consulted.

### Step 1: Relevance

Filter out all rules whose selectors do not match the element, or that are inside inapplicable `@media` or `@supports` blocks. Only matching rules proceed.

### Step 2: Origin and Importance

CSS has three origins:

| Origin | Description |
|---|---|
| User-Agent (UA) | Browser defaults (`<h1>` is bold, `<a>` is blue) |
| User | Styles from the user (OS settings, browser extensions, reader mode) |
| Author | Your CSS (the developer's stylesheets) |

For **normal** declarations, later origins win:

```
UA normal  <  User normal  <  Author normal
```

For **`!important`** declarations, the order is **reversed**:

```
Author !important  <  User !important  <  UA !important
```

This reversal is intentional. It ensures that user accessibility styles (large fonts, high contrast) can override even the most aggressive author `!important` declarations, and that browsers can enforce critical UA defaults (like focus outlines) that authors cannot break.

The full combined priority, from lowest to highest:

1. UA normal
2. User normal
3. Author normal
4. CSS Animations (override normal, not `!important`)
5. Author `!important`
6. User `!important`
7. UA `!important`
8. CSS Transitions (highest priority)

### Step 3: Cascade Layers (`@layer`)

Within the same origin-importance bucket, declarations are sorted by their cascade layer. Layers are covered in detail below.

### Step 4: Specificity

Within the same layer (or when no layers are used), the specificity of the selectors is compared.

### Step 5: Scoping Proximity (`@scope`)

When two selectors have equal specificity within the same layer, the one whose `@scope` root is closer to the matched element in the DOM wins. Covered below.

### Step 6: Order of Appearance

If everything above is tied, the last declaration in source order wins.

```css
p { color: red; }
p { color: blue; }  /* This wins -- same specificity, later in source */
```

## Specificity: The Three-Column Model

Specificity is expressed as a triple: **(A, B, C)**.

| Column | What Counts | Examples |
|---|---|---|
| A (ID) | `#id` selectors | `#nav`, `#main` |
| B (CLASS) | `.class`, `[attribute]`, `:pseudo-class` | `.card`, `[type="text"]`, `:hover` |
| C (TYPE) | `element`, `::pseudo-element` | `div`, `p`, `::before` |

### Comparison Is Column-by-Column

Each column is compared independently, left to right. A higher A column always wins, regardless of B and C values. You cannot accumulate lower-column selectors to beat a higher column.

```css
/* Specificity: 1-0-0 */
#header { color: blue; }

/* Specificity: 0-20-0 (twenty classes!) */
.a.b.c.d.e.f.g.h.i.j.k.l.m.n.o.p.q.r.s.t { color: red; }

/* #header wins. One ID always beats any number of classes. */
```

### Calculation Examples

| Selector | A | B | C | Total |
|---|---|---|---|---|
| `*` | 0 | 0 | 0 | 0-0-0 |
| `p` | 0 | 0 | 1 | 0-0-1 |
| `p::before` | 0 | 0 | 2 | 0-0-2 |
| `.card` | 0 | 1 | 0 | 0-1-0 |
| `p.card` | 0 | 1 | 1 | 0-1-1 |
| `[type="email"]` | 0 | 1 | 0 | 0-1-0 |
| `ul > li.active` | 0 | 1 | 2 | 0-1-2 |
| `a:hover` | 0 | 1 | 1 | 0-1-1 |
| `#nav` | 1 | 0 | 0 | 1-0-0 |
| `#nav .item a` | 1 | 1 | 1 | 1-1-1 |
| `#app #nav .list .item p` | 2 | 2 | 1 | 2-2-1 |

### What Contributes Zero Specificity

- **Combinators:** `>`, `+`, `~`, ` ` (space) contribute nothing.
- **Universal selector:** `*` contributes nothing.
- **`:where()`:** Always zero, regardless of its arguments.

### Inline Styles

Inline styles (`style="..."`) operate at a different level of the cascade. They beat any selector-based specificity. The only way to override an inline style from a stylesheet is with `!important`.

```html
<p style="color: red;">Hello</p>
```

```css
/* Does NOT override the inline style, even with high specificity */
#content .intro p { color: blue; }

/* This DOES override it */
p { color: blue !important; }
```

## Pseudo-Class Specificity: `:is()`, `:not()`, `:has()`, `:where()`

These functional pseudo-classes have special rules:

| Pseudo-class | Own weight | Argument weight |
|---|---|---|
| `:is()` | 0 | Most specific argument |
| `:not()` | 0 | Most specific argument |
| `:has()` | 0 | Most specific argument |
| `:where()` | 0 | Always zero |

```css
/* :is(.card, #hero) takes specificity from #hero = 1-0-0 */
:is(.card, #hero) p { }   /* 1-0-1 */

/* :where(.card, #hero) is always zero */
:where(.card, #hero) p { }  /* 0-0-1 */

/* :not(#main) takes specificity from #main = 1-0-0 */
div:not(#main) { }   /* 1-0-1 */

/* :has(> .active) takes specificity from .active = 0-1-0 */
ul:has(> .active) { }   /* 0-1-1 */
```

**Gotcha:** `:is(.card, #hero)` gives the entire selector specificity 1-0-0 even when it matches an element via `.card`. The highest argument always applies, regardless of which one actually matched.

### Nesting Specificity

Native CSS nesting treats `&` like `:is()` for specificity:

```css
.card {
  .title { font-weight: bold; }
}
/* Equivalent to: :is(.card) .title -- specificity 0-2-0 */

/* Danger with selector lists: */
.card, #hero {
  .title { font-weight: bold; }
}
/* Equivalent to: :is(.card, #hero) .title -- specificity 1-1-0 */
/* The #hero inflates specificity even for .card matches */
```

## Inheritance

Inheritance is separate from the cascade. When no rule targets a property on an element, the element inherits the property's value from its parent (if the property is inheritable).

### Properties That Inherit

Primarily text and typography-related:

| Category | Properties |
|---|---|
| Font | `font-family`, `font-size`, `font-weight`, `font-style`, `font-variant` |
| Text | `color`, `text-align`, `text-indent`, `text-transform`, `text-shadow`, `letter-spacing`, `word-spacing`, `line-height`, `white-space`, `word-break` |
| List | `list-style`, `list-style-type`, `list-style-position` |
| Other | `visibility`, `cursor`, `quotes`, `direction`, `writing-mode` |

### Properties That Do NOT Inherit

Box model and layout properties:

| Category | Properties |
|---|---|
| Box model | `margin`, `padding`, `border`, `width`, `height` |
| Layout | `display`, `position`, `float`, `overflow` |
| Background | `background`, `background-color`, `background-image` |
| Flex/Grid | `flex`, `grid-template-*`, `gap`, `align-items`, `justify-content` |

### Gotcha: Direct Targeting Beats Inheritance

Any directly-targeted rule beats an inherited value, regardless of the inherited rule's specificity:

```css
#header {
  color: blue;  /* specificity 1-0-0, but this is inherited by children */
}

* {
  color: red;  /* specificity 0-0-0, but directly targets every element */
}

/* Children of #header will be red, not blue.
   The universal selector directly targets them.
   The #header color is merely inherited. */
```

### Gotcha: Form Elements and Links

**`<a>` elements** do not inherit `color` by default. The UA stylesheet explicitly sets `color: -webkit-link` (blue/purple). You must force it:

```css
a { color: inherit; }
```

**Form elements** (`<input>`, `<button>`, `<select>`, `<textarea>`) do not inherit `font-family` or `font-size` by default. The UA stylesheet overrides them. Common fix:

```css
input, button, select, textarea {
  font: inherit;
}
```

## CSS-Wide Keywords

Five keywords can be applied to any CSS property:

| Keyword | Behavior |
|---|---|
| `inherit` | Takes the parent's computed value, even for non-inheriting properties |
| `initial` | Resets to the property's **spec-defined** initial value |
| `unset` | `inherit` for inheriting properties, `initial` for non-inheriting ones |
| `revert` | Rolls back to the value from the previous **origin** (usually the UA stylesheet) |
| `revert-layer` | Rolls back to the value from the previous **cascade layer** |

### The Critical `initial` vs `revert` Distinction

This is one of the most common CSS mistakes:

```css
div {
  display: initial;  /* "inline" -- the spec initial value for display */
  display: revert;   /* "block" -- the UA stylesheet value for <div> */
}

h1 {
  font-size: initial;  /* "medium" -- the spec initial value */
  font-size: revert;   /* "2em" -- the UA stylesheet value for <h1> */
}
```

`initial` resets to the CSS specification default, which is often not what you expect. `revert` resets to the browser default for that specific element, which is almost always what you actually want.

### `unset` for Bulk Resets

`unset` is useful with the `all` property to strip author styles:

```css
.reset-area * {
  all: unset;  /* Inherited properties -> inherit, non-inherited -> initial */
}
```

### `revert-layer` for Layer-Aware Rollbacks

```css
@layer base, components, overrides;

@layer base {
  .card { padding: 1rem; border: 1px solid #ddd; }
}

@layer components {
  .card { padding: 2rem; }
}

@layer overrides {
  .card {
    padding: revert-layer;  /* Rolls back to base layer's 1rem */
  }
}
```

## Cascade Layers (`@layer`)

Cascade layers give you explicit control over the cascade ordering, sitting above specificity in the algorithm. A less-specific selector in a higher-priority layer always beats a more-specific selector in a lower-priority layer.

### Declaring Layer Order

```css
/* Declare order first (optional but recommended) */
@layer reset, base, components, utilities;
```

The last declared layer has the highest priority for normal styles. In this example: `utilities` > `components` > `base` > `reset`.

### Adding Styles to Layers

```css
@layer reset {
  *, *::before, *::after {
    margin: 0;
    padding: 0;
    box-sizing: border-box;
  }
}

@layer base {
  body {
    font-family: system-ui, sans-serif;
    line-height: 1.6;
    color: #333;
  }

  a { color: #3498db; }
}

@layer components {
  .card {
    border: 1px solid #ddd;
    border-radius: 8px;
    padding: 1.5rem;
  }

  .btn {
    display: inline-flex;
    padding: 0.5rem 1rem;
    border-radius: 4px;
    cursor: pointer;
  }
}

@layer utilities {
  .hidden { display: none; }
  .sr-only { position: absolute; width: 1px; height: 1px; overflow: hidden; clip: rect(0,0,0,0); }
}
```

### Unlayered Styles Beat All Layers

Any CSS not inside an `@layer` is treated as being in an implicit "highest" layer:

```css
@layer base {
  p { color: blue; }  /* In a layer */
}

p { color: red; }  /* Unlayered -- wins */
```

This makes adoption incremental. You can start layering third-party CSS and your base styles without needing to layer everything at once.

### Importing into Layers

This is where layers shine for managing third-party CSS:

```css
@layer reset, framework, custom;

@import url("modern-normalize.css") layer(reset);
@import url("bootstrap.css") layer(framework);

@layer custom {
  /* Your styles here -- always beat the framework without !important */
  .card { border-radius: 12px; }
}
```

Before layers, overriding `#sidebar .widget .title { color: blue; }` from a framework required matching or exceeding that specificity. With layers, a simple `.title { color: red; }` in a higher layer wins.

### Nested Layers

```css
@layer framework {
  @layer base {
    body { font-size: 16px; }
  }
  @layer components {
    .btn { padding: 8px; }
  }
}

/* Reference with dot notation */
@layer framework.components {
  .btn { padding: 12px; }
}
```

### Anonymous Layers

```css
@layer {
  /* Cannot be referenced later -- styles are final */
  p { color: gray; }
}
```

Useful for one-off third-party imports you never need to extend.

### The `!important` Reversal in Layers

For normal declarations, later layers win:
```
reset < base < components < utilities
```

For `!important` declarations, the order **reverses**:
```
utilities !important < components !important < base !important < reset !important
```

This is the same pattern as the origin-level reversal. It ensures that foundational layers (resets) can use `!important` as a true last resort that later layers cannot override.

```css
@layer reset, components;

@layer reset {
  a { color: inherit !important; }  /* This WINS (earlier layer + !important) */
}

@layer components {
  a { color: blue !important; }     /* This LOSES */
}
```

This behavior is intentional and correct, but it surprises most developers the first time they encounter it.

## `@scope` -- Scoped Styles

`@scope` limits styles to a specific DOM subtree and introduces scoping proximity as a new cascade criterion.

### Basic Scoping

```css
@scope (.card) {
  h2 { color: navy; }
  p { line-height: 1.8; }
}
/* These rules only apply to h2 and p elements inside .card */
```

### Donut Scope (Upper and Lower Bounds)

Exclude a nested subtree:

```css
@scope (.article) to (.sidebar) {
  /* Matches descendants of .article but NOT those inside .sidebar */
  p { font-size: 1.1rem; }
  a { color: #2c3e50; }
}
```

This is called a "donut scope" because it has a hole in the middle. Useful for component-style isolation where nested sub-components should not be affected.

### Scoping Proximity

When two `@scope` rules match the same element with the same specificity in the same layer, the one whose scope root is closer in the DOM wins:

```css
@scope (.dark-theme) {
  p { color: white; }
}

@scope (.light-theme) {
  p { color: black; }
}
```

```html
<div class="dark-theme">
  <div class="light-theme">
    <p>This text is black.</p>
    <!-- .light-theme is closer, so its rule wins -->
  </div>
</div>
```

This eliminates the need for increasingly specific selectors or `!important` when dealing with nested theme contexts.

### Inline `@scope`

```html
<div class="widget">
  <style>
    @scope {
      /* Implicitly scoped to the parent element */
      p { color: navy; }
    }
  </style>
  <p>Scoped to this widget only.</p>
</div>
```

Browser support: Chrome 118+, Firefox 146+, Safari 17.4+. Baseline Newly Available as of early 2025.

## `!important`: When and When Not

### Legitimate Uses

1. **Utility classes that must override anything:**

```css
.hidden { display: none !important; }
.sr-only { position: absolute !important; overflow: hidden !important; }
```

2. **Overriding third-party CSS you cannot modify** (though cascade layers are the modern alternative).

3. **Accessibility styles** that must not be overridden by author CSS.

### When NOT to Use It

```css
/* BAD: using !important to "fix" a specificity problem */
.title { color: red !important; }

/* The real fix: understand why the specificity is wrong */
/* Maybe an #id selector is causing the issue -- remove it */
```

Adding `!important` to fix one rule leads to adding it to others, creating an escalation war where every declaration needs `!important` and the only way to override is inline `!important`.

### The Modern Alternative: Layers

Instead of:

```css
/* Third-party framework */
#sidebar .widget .title { color: blue; }

/* Your override -- forced to use !important */
.title { color: red !important; }
```

Use:

```css
@layer framework, custom;
@import url("framework.css") layer(framework);

@layer custom {
  .title { color: red; }  /* Wins without !important */
}
```

## Common Mistakes

1. **The base-10 myth.** Specificity is NOT a single number. (0,1,0) is not "10" and (0,0,11) is not "11." The columns are independent. 255 classes cannot beat 1 ID.

2. **Using `initial` expecting browser defaults.** `display: initial` gives `inline` (the spec default), not `block` for a `<div>`. Use `revert` for browser defaults.

3. **Forgetting that inheritance loses to direct targeting.** `#header { color: blue; }` is inherited by children, but even `* { color: red; }` (specificity 0-0-0) beats it because `*` directly targets each child.

4. **`:is()` specificity inflation.** `:is(.card, #hero)` gives 1-0-0 specificity to every match, even the `.card` one. Use `:where()` if you need zero specificity.

5. **`!important` in cascade layers.** Adding `!important` in a later layer (utilities) actually LOSES to `!important` in an earlier layer (reset). The reversal catches everyone.

6. **Form elements not inheriting fonts.** `<input>` and `<button>` ignore inherited `font-family` and `font-size`. Always set `font: inherit` on form elements.

7. **Links not inheriting color.** `<a>` has UA styles that override inherited `color`. Set `color: inherit` explicitly or target `a` directly.

8. **Over-qualifying selectors.** Writing `div#nav` when `#nav` alone is sufficient. This raises specificity from 1-0-0 to 1-0-1 for no benefit, making overrides harder.

9. **Specificity arms race.** Countering `.sidebar .widget h3` with `.main-content .article .header h3` is a sign of a broken architecture. Refactor to flat class selectors.

10. **Repeating selectors as a hack.** `.card.card` (0-2-0) works but is a code smell. If you need to beat another selector, restructure instead of repeating.

---

## Best Practices

1. **Use cascade layers for architectural CSS organization.** Declare `@layer reset, base, components, utilities` and put CSS in the appropriate layer. This makes specificity conflicts between layers impossible.

2. **Avoid ID selectors for styling.** Reserve `#id` for JavaScript hooks and URL fragments. One ID selector (1-0-0) requires another ID or `!important` to override.

3. **Keep selectors flat and class-based.** `.nav-link` beats `header nav ul li a` in maintainability. BEM naming (`.block__element--modifier`) keeps everything at 0-1-0.

4. **Use `:where()` for overridable base/framework styles.** Any code meant to be overridden by consumers should use `:where()` so even a single class can beat it.

5. **Use `revert` instead of `initial`.** When you want to reset a property to the browser default, `revert` gives you what you expect. `initial` gives the spec default.

6. **Import third-party CSS into layers.** `@import url("lib.css") layer(lib)` prevents any specificity battle with your own code.

7. **Never use `!important` to fix specificity.** Fix the selector architecture instead. If you must use `!important`, confine it to utility classes and document the reason.

8. **Set `font: inherit` on form elements.** Do this in your reset layer to prevent the UA stylesheet from overriding your typography.

## Real-World Use Cases

### 1. Layered Architecture with Third-Party Framework

```css
/* Declare the layer order upfront */
@layer reset, framework, components, utilities;

/* Import third-party CSS into controlled layers */
@import url("modern-normalize.css") layer(reset);
@import url("some-ui-lib.css") layer(framework);

/* Component styles -- always beat framework without !important */
@layer components {
  .card {
    padding: 2rem;
    border-radius: 12px;
    background: white;
  }

  .card .title {
    font-size: 1.25rem;
    font-weight: 600;
  }
}

/* Utility overrides -- highest layered priority */
@layer utilities {
  .hidden { display: none !important; }
  .text-center { text-align: center; }
  .mt-0 { margin-top: 0; }
}

/* Unlayered styles (e.g., page-specific) beat everything */
.homepage .hero {
  padding: 4rem 2rem;
}
```

### 2. Theming with `@scope` and Proximity

```css
/* Base theme */
@scope (.theme-light) {
  :scope {
    --bg: #ffffff;
    --text: #1a1a1a;
    --accent: #3498db;
  }

  .card {
    background: var(--bg);
    color: var(--text);
    border: 1px solid #e0e0e0;
  }
}

/* Dark theme */
@scope (.theme-dark) {
  :scope {
    --bg: #1a1a2e;
    --text: #e0e0e0;
    --accent: #64b5f6;
  }

  .card {
    background: var(--bg);
    color: var(--text);
    border: 1px solid #333;
  }
}
```

```html
<!-- Scoping proximity handles nesting correctly -->
<div class="theme-dark">
  <div class="card">Dark card</div>
  <div class="theme-light">
    <div class="card">Light card (closer scope wins)</div>
  </div>
</div>
```

No specificity tricks needed. The closer `@scope` root automatically wins.

### 3. Design System with `:where()` for Consumer Overrides

```css
/* Design system base -- all wrapped in :where() for zero specificity */
:where(.ds-btn) {
  display: inline-flex;
  align-items: center;
  gap: 0.5rem;
  padding: 0.625rem 1.25rem;
  border: 1px solid transparent;
  border-radius: 6px;
  font: inherit;
  font-weight: 500;
  cursor: pointer;
  transition: background 0.15s, border-color 0.15s;
}

:where(.ds-btn--primary) {
  background: var(--ds-color-primary, #3498db);
  color: white;
}

:where(.ds-btn--secondary) {
  background: transparent;
  border-color: var(--ds-color-primary, #3498db);
  color: var(--ds-color-primary, #3498db);
}

:where(.ds-btn:hover) {
  filter: brightness(0.9);
}
```

```css
/* Consumer project -- a single class easily overrides :where() styles */
.checkout-btn {
  padding: 1rem 2rem;
  border-radius: 999px;
  font-size: 1.125rem;
}
```

Because the design system uses `:where()`, `.checkout-btn` (specificity 0-1-0) beats every design system rule (specificity 0-0-0 from `:where()`). No `!important`, no specificity gymnastics.

---

> **Key Takeaway:** The cascade is a multi-step algorithm where origin, importance, cascade layers, and specificity are evaluated in strict order; mastering cascade layers (`@layer`) and specificity-zeroing (`:where()`) eliminates the need for `!important` and makes CSS architecturally predictable.
