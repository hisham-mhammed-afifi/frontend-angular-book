# 2.3 Box Model

> Every element in CSS generates a rectangular box composed of four areas: content, padding, border, and margin. How these areas are sized and how they interact determines every layout you build.

---

## The Four Areas

Every element's box consists of four concentric rectangles:

```
┌───────────────────────────── Margin ───────────────────────────────┐
│                                                                    │
│   ┌─────────────────────── Border ────────────────────────────┐    │
│   │                                                           │    │
│   │   ┌───────────────── Padding ───────────────────────┐     │    │
│   │   │                                                 │     │    │
│   │   │   ┌──────────── Content ───────────────┐        │     │    │
│   │   │   │                                    │        │     │    │
│   │   │   │   Text, images, child elements     │        │     │    │
│   │   │   │                                    │        │     │    │
│   │   │   └────────────────────────────────────┘        │     │    │
│   │   │                                                 │     │    │
│   │   └─────────────────────────────────────────────────┘     │    │
│   │                                                           │    │
│   └───────────────────────────────────────────────────────────┘    │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

| Area | Role | Visibility |
|---|---|---|
| Content | Where text, images, and children render | Visible |
| Padding | Space between content and border | Transparent (shows background) |
| Border | Wraps content + padding | Styled (color, width, style) |
| Margin | Space between this element and others | Always transparent |

Background color and images extend through the content and padding areas, but not through the border (unless `background-clip` is changed) and never through the margin.

## `box-sizing`: The Most Important Box Model Property

### `content-box` (Default)

`width` and `height` apply only to the content area. Padding and border are added on top:

```css
.box {
  box-sizing: content-box; /* default */
  width: 300px;
  padding: 20px;
  border: 5px solid #333;
}
/* Rendered width: 300 + 20 + 20 + 5 + 5 = 350px */
/* Content area: 300px */
```

### `border-box`

`width` and `height` include content + padding + border. The content area shrinks to fit:

```css
.box {
  box-sizing: border-box;
  width: 300px;
  padding: 20px;
  border: 5px solid #333;
}
/* Rendered width: 300px (exactly what you set) */
/* Content area: 300 - 20 - 20 - 5 - 5 = 250px */
```

With `border-box`, what you set is what you get. This makes layouts predictable, especially when combining percentage widths with fixed padding.

### `width: auto` vs `width: 100%`

This distinction matters more than most developers realize:

```css
/* width: auto (default) */
/* The element fills available space. Padding pushes INWARD. */
.auto-box {
  /* no explicit width */
  padding: 20px;
  border: 5px solid;
}
/* Total width = parent width. Content shrinks to accommodate padding + border. */

/* width: 100% with content-box */
/* The content area = 100% of parent. Padding pushes OUTWARD. */
.full-box {
  box-sizing: content-box;
  width: 100%;
  padding: 20px;
  border: 5px solid;
}
/* Total width = parent width + 40px padding + 10px border = OVERFLOW */
```

With `border-box`, `width: 100%` behaves identically to `width: auto` because padding is included in the 100%. But with `content-box`, `width: 100%` causes overflow. This is one of the most common layout bugs.

## The `box-sizing` Reset

### The Inheritance Pattern (Recommended)

```css
html {
  box-sizing: border-box;
}

*,
*::before,
*::after {
  box-sizing: inherit;
}
```

Why `inherit` instead of directly setting `border-box` on `*`? If a third-party component needs `content-box`, you can set it on the component's container and all its descendants inherit it automatically:

```css
.legacy-widget {
  box-sizing: content-box;
  /* All descendants inherit content-box */
}
```

### The Simple Pattern

```css
*,
*::before,
*::after {
  box-sizing: border-box;
}
```

Simpler, but any component requiring `content-box` must override every descendant individually. For most projects, this is fine.

## Margin Collapsing

Margin collapsing is one of the most misunderstood CSS behaviors. When two block-direction margins meet, they combine into a single margin equal to the larger of the two.

### Rule 1: Adjacent Siblings

When two block-level elements are stacked vertically, the bottom margin of the first and the top margin of the second collapse:

```css
.box-a { margin-bottom: 30px; }
.box-b { margin-top: 20px; }
/* Space between them: 30px (the larger value), NOT 50px */
```

### Rule 2: Parent and First/Last Child

A parent's `margin-top` collapses with its first child's `margin-top`. A parent's `margin-bottom` collapses with its last child's `margin-bottom`. The margin "escapes" the parent:

```css
.parent {
  margin-top: 0;
  /* No border, no padding */
}
.first-child {
  margin-top: 40px;
  /* This margin escapes above the parent */
}
```

The parent appears to have a 40px top margin even though it has `margin-top: 0`.

### Rule 3: Empty Blocks

If a block has no border, padding, inline content, height, or min-height, its own top and bottom margins collapse with each other. The element effectively disappears from a margin perspective:

```css
.empty { margin-top: 20px; margin-bottom: 30px; }
/* If the element is empty, these collapse to a single 30px margin */
```

### Rule 4: Negative Margins

| Scenario | Result |
|---|---|
| Both positive | Larger value |
| One positive, one negative | Sum (positive + negative) |
| Both negative | Most negative value |

```css
.box-a { margin-bottom: 30px; }
.box-b { margin-top: -10px; }
/* Space: 30 + (-10) = 20px */

.box-c { margin-bottom: -20px; }
.box-d { margin-top: -30px; }
/* Space: -30px (most negative) */
```

### What Prevents Margin Collapse

Margin collapsing only happens in block formatting context. These contexts prevent it:

| Prevention Method | How |
|---|---|
| `padding` or `border` on the collapsing edge | Separates the margins physically |
| `display: flex` on parent | Creates flex formatting context |
| `display: grid` on parent | Creates grid formatting context |
| `display: flow-root` on parent | Creates a new block formatting context |
| `overflow` other than `visible` on parent | Creates a new BFC |
| `display: inline-block` | Inline formatting |
| Floated elements | Out of normal flow |
| Absolutely/fixed positioned elements | Out of normal flow |
| Inline content between margins | Separates them |

```css
/* BAD: margin leaks out of parent */
.parent {
  background: #f0f0f0;
}
.child {
  margin-top: 40px;  /* Escapes above .parent */
}

/* FIX 1: Add padding */
.parent {
  padding-top: 1px;  /* Prevents collapse */
}

/* FIX 2: Use display: flow-root (purpose-built for this) */
.parent {
  display: flow-root;
}

/* FIX 3: Use flexbox (most common in modern CSS) */
.parent {
  display: flex;
  flex-direction: column;
}
```

### Only Block-Direction Margins Collapse

In horizontal writing modes (the default), only vertical margins (top/bottom) collapse. Horizontal margins (left/right) never collapse. In vertical writing modes, the block direction changes accordingly.

## Inline Elements and the Box Model

Inline elements (`<span>`, `<a>`, `<em>`, `<strong>`) follow different box model rules:

```css
span {
  width: 200px;        /* IGNORED on inline elements */
  height: 100px;       /* IGNORED on inline elements */
  margin-top: 20px;    /* Applied but does NOT push surrounding content */
  margin-bottom: 20px; /* Applied but does NOT push surrounding content */
  margin-left: 10px;   /* Works normally */
  margin-right: 10px;  /* Works normally */
  padding: 10px;       /* All sides applied, but vertical padding does NOT
                          push surrounding lines apart (only overlaps them) */
  border: 1px solid;   /* Same behavior as padding */
}
```

To give an inline element full box model behavior while keeping it in the inline flow, use `display: inline-block`:

```css
span.badge {
  display: inline-block;
  /* Now width, height, vertical margins, and vertical padding all work */
  padding: 4px 12px;
  margin: 2px;
  border-radius: 12px;
}
```

## Percentage Padding Gotcha

All percentage-based padding, including `padding-top` and `padding-bottom`, resolves relative to the **containing block's inline size (width)**, not its height:

```css
.container {
  width: 800px;
}
.child {
  padding-top: 10%;  /* 80px (10% of 800px width), NOT 10% of height */
}
```

This was historically exploited for aspect-ratio hacks before the `aspect-ratio` property existed:

```css
/* OLD: 16:9 aspect ratio hack using percentage padding */
.video-container {
  width: 100%;
  padding-top: 56.25%;  /* 9/16 = 0.5625 */
  position: relative;
}
.video-container iframe {
  position: absolute;
  top: 0; left: 0;
  width: 100%; height: 100%;
}

/* MODERN: Use aspect-ratio instead */
.video-container {
  width: 100%;
  aspect-ratio: 16 / 9;
}
```

## Negative Margins

Negative margins are valid CSS and have legitimate uses:

```css
/* Break out of a padded container */
.container {
  padding: 0 2rem;
}
.full-bleed {
  margin-left: -2rem;
  margin-right: -2rem;
}

/* Pull an element upward */
.overlap-up {
  margin-top: -40px;
}
```

Negative margins pull elements in the specified direction rather than pushing. They maintain document flow, unlike `position: relative` with offsets. Negative padding does not exist and is invalid.

## Overflow

The `overflow` property controls what happens when content exceeds its container's dimensions.

### All Values

| Value | Clips? | Scrollbars? | JS scrolling? | Creates BFC? |
|---|---|---|---|---|
| `visible` (default) | No | No | N/A | No |
| `hidden` | Yes | No | Yes | Yes |
| `scroll` | Yes | Always shown | Yes | Yes |
| `auto` | If needed | If needed | Yes | Yes |
| `clip` | Yes | No | No | No |

### `hidden` vs `clip`

This distinction is critical:

```css
/* overflow: hidden creates a scroll container */
.hidden-box {
  overflow: hidden;
}
/* Content IS clipped visually, but:
   - scrollIntoView() can scroll the element
   - Anchor links (#id) can scroll the element
   - Keyboard navigation to focusable children can scroll it
   - Creates a new BFC (affects margin collapsing) */

/* overflow: clip does NOT create a scroll container */
.clip-box {
  overflow: clip;
}
/* Content is clipped. No scrolling is possible at all.
   Does NOT create a BFC. */
```

Use `clip` when you truly want no scrolling. Use `hidden` when you want to clip visually but allow programmatic scrolling.

### `overflow-clip-margin`

Works only with `overflow: clip`. Defines how far overflowing content can extend beyond the clip edge:

```css
.element {
  overflow: clip;
  overflow-clip-margin: 20px;
  /* Content can extend 20px beyond the element's box before being clipped */
}
```

### Single-Axis Overflow Gotcha

Setting `overflow-x` and `overflow-y` to different values has restrictions:

```css
.element {
  overflow-x: clip;
  overflow-y: visible;
  /* This does NOT work as expected.
     The clip value behaves as hidden when the other axis is not clip or visible. */
}
```

If one axis is `clip` and the other is not `clip` or `visible`, the `clip` value silently becomes `hidden`.

## `aspect-ratio`

The `aspect-ratio` property sets a preferred aspect ratio for the element:

```css
.square { aspect-ratio: 1; }          /* 1:1 */
.widescreen { aspect-ratio: 16 / 9; } /* 16:9 */
.portrait { aspect-ratio: 3 / 4; }    /* 3:4 */
```

### How It Works

- Only takes effect when at least one dimension is `auto`.
- If both `width` and `height` are explicitly set, `aspect-ratio` is ignored.
- Setting one dimension lets `aspect-ratio` calculate the other.

```css
.card {
  width: 100%;
  aspect-ratio: 4 / 3;
  /* Height is automatically calculated to maintain 4:3 ratio */
}
```

### With Images

```css
img {
  aspect-ratio: auto;  /* default: uses the image's intrinsic ratio */
}

/* Prevent layout shift before images load */
img {
  width: 100%;
  aspect-ratio: 16 / 9;  /* reserves space while loading */
}
```

The `auto` keyword with a fallback ratio is useful for images that should use their intrinsic ratio when loaded but have a placeholder ratio before loading:

```css
img {
  aspect-ratio: auto 16 / 9;
  /* Use intrinsic ratio if available, otherwise 16/9 */
}
```

### Interaction with `min-*` / `max-*`

`min-height`, `max-height`, `min-width`, and `max-width` override `aspect-ratio` when they constrain the element. The aspect ratio is maintained only within those limits:

```css
.card {
  width: 100%;
  aspect-ratio: 1;
  max-height: 400px;
  /* Square until height would exceed 400px, then height is capped */
}
```

## Intrinsic Sizing Keywords

These keywords let elements size themselves based on their content rather than fixed or percentage values.

### `min-content`

The smallest the element can be without overflowing. For text, this is the width of the longest unbreakable word:

```css
.tag {
  width: min-content;
  /* Wraps text at every soft wrap opportunity.
     Width = longest single word. */
}
```

### `max-content`

The largest the element needs to fit all content without wrapping:

```css
.tag {
  width: max-content;
  /* No wrapping at all. Width = full text length on one line. */
}
```

### `fit-content`

A hybrid: grows with content like `max-content`, but never exceeds available space. Shrinks to `min-content` if necessary:

```css
.tooltip {
  width: fit-content;
  max-width: 300px;
  /* Content-sized up to 300px, then wraps */
}
```

The formula is: `min(max-content, max(min-content, available-space))`.

### Comparison

```css
.container { width: 200px; }

/* Text: "Hello beautiful world" */

.min { width: min-content; }
/* Width: ~72px ("beautiful" is longest word) */

.max { width: max-content; }
/* Width: ~175px (entire text on one line) */

.fit { width: fit-content; }
/* Width: ~175px (fits within 200px container) */
```

### `fit-content()` Function (Grid Only)

Different from the `fit-content` keyword. Only valid in grid track sizing:

```css
.grid {
  grid-template-columns: fit-content(200px) 1fr;
  /* First column: content-sized but capped at 200px */
}
```

## Outline vs Border

Outlines are drawn outside the border and do not take up space in the box model:

| Feature | Border | Outline |
|---|---|---|
| Part of box model | Yes (affects layout) | No (no layout impact) |
| Individual sides | Yes (`border-top`, etc.) | No (all sides only) |
| Follows `border-radius` | Yes | Yes (modern browsers) |
| Takes up space | Yes | No, may overlap content |
| Triggers reflow | Yes | No |
| Offset control | No | Yes (`outline-offset`) |

```css
/* Outline with offset for focus indicators */
button:focus-visible {
  outline: 3px solid #3498db;
  outline-offset: 3px;  /* gap between border and outline */
}

/* Negative offset creates an inset outline */
.inset-ring {
  outline: 2px solid rgba(0, 0, 0, 0.1);
  outline-offset: -8px;
}
```

Because outlines do not trigger reflow, changing outline properties during hover or focus animations is cheaper than changing border properties.

## Logical Properties

Logical properties map to physical directions based on the writing mode and text direction, enabling layouts that work across languages:

| Physical | Logical | Default mapping (LTR, horizontal-tb) |
|---|---|---|
| `margin-top` | `margin-block-start` | Top |
| `margin-bottom` | `margin-block-end` | Bottom |
| `margin-left` | `margin-inline-start` | Left |
| `margin-right` | `margin-inline-end` | Right |
| `width` | `inline-size` | Width |
| `height` | `block-size` | Height |

Shorthand properties make these practical:

```css
.element {
  margin-block: 1rem 2rem;    /* block-start  block-end */
  margin-inline: auto;        /* centers in both inline directions */
  padding-block: 1.5rem;      /* same on both block sides */
  padding-inline: 1rem 2rem;  /* inline-start  inline-end */
  border-block: 1px solid #ddd;
  border-inline-start: 3px solid #3498db;
}
```

In a right-to-left context, `margin-inline-start` maps to `margin-right` instead of `margin-left`. The same CSS works for both directions without changes.

## `box-decoration-break`

Controls how box decorations (background, border, padding, border-radius, box-shadow) render when an element fragments across lines, columns, or pages:

```css
.highlight {
  display: inline;
  background: #ffffcc;
  padding: 2px 6px;
  border-radius: 4px;
  box-decoration-break: clone;
  -webkit-box-decoration-break: clone; /* Safari */
}
```

| Value | Behavior |
|---|---|
| `slice` (default) | Element renders as if unbroken, then sliced at each break. Decorations are cut. |
| `clone` | Each fragment gets its own complete decorations (padding, border-radius, background). |

`clone` is essential for multi-line inline elements with backgrounds or borders. Without it, the border-radius and padding only appear at the start of the first line and end of the last line.

## Common Mistakes

1. **Forgetting the `box-sizing` reset.** Not setting `border-box` globally, then being surprised when `width: 100%` plus padding causes horizontal overflow.

2. **`width: 100%` vs `width: auto`.** With `content-box`, `width: 100%` makes the content area equal to the parent width, and padding/border overflow. `width: auto` (the default) accounts for padding and border automatically.

3. **Margin collapse surprise.** Two adjacent elements with `margin: 20px` have 20px between them, not 40px. Parent margins "leaking" through borderless, paddingless children.

4. **Vertical margin on inline elements.** Setting `margin-top` or `margin-bottom` on a `<span>` and expecting it to create space. Vertical margins on inline elements do not push surrounding content.

5. **Percentage padding direction.** `padding-top: 10%` is 10% of the containing block's **width**, not height. This catches everyone.

6. **`overflow: hidden` still scrolls.** Using `hidden` and being surprised when `scrollIntoView()` or anchor links cause scrolling. Use `overflow: clip` for true no-scroll clipping.

7. **Setting both dimensions with `aspect-ratio`.** If both `width` and `height` are explicitly set, `aspect-ratio` is completely ignored.

8. **`display: initial` for resetting.** `display: initial` gives `inline` (the spec default), not `block` for a `<div>`. This is a cascade/inheritance issue but manifests as a box model surprise.

9. **`outline: none` without replacement.** Removing the default outline on `:focus` without providing a `:focus-visible` alternative breaks keyboard accessibility.

10. **Single-axis `overflow: clip` with `visible`.** Setting `overflow-x: clip` and `overflow-y: visible` does not work. The `clip` silently becomes `hidden`.

---

## Best Practices

1. **Always set a `box-sizing: border-box` reset.** Use the inheritance pattern (`html { box-sizing: border-box }` + `*, *::before, *::after { box-sizing: inherit }`) for maximum flexibility.

2. **Prefer `gap` over margins for spacing in flex/grid layouts.** `gap` does not collapse, does not need `:first-child`/`:last-child` overrides, and clearly communicates intent.

3. **Use `display: flow-root` to prevent margin collapse.** It is purpose-built for creating a new block formatting context without side effects (unlike `overflow: hidden`).

4. **Use `aspect-ratio` instead of the padding hack.** The property is supported in all modern browsers and is far more readable.

5. **Use `overflow: clip` when you want true clipping.** Reserve `overflow: hidden` for cases where you need programmatic scrolling.

6. **Use logical properties for internationalization-ready CSS.** `margin-inline`, `padding-block`, `border-inline-start` work across all writing modes and directions.

7. **Prefer `outline` over `border` for focus indicators.** Outlines do not affect layout and do not trigger reflow.

8. **Use intrinsic sizing keywords for content-driven layouts.** `width: fit-content` for tooltips, `min-content` for tight wrapping, `max-content` for single-line labels.

## Real-World Use Cases

### 1. Full-Bleed Image Inside Padded Container

```css
.article {
  max-width: 720px;
  margin-inline: auto;
  padding-inline: 2rem;
}

/* Image breaks out of padding to span full container width */
.article .full-bleed {
  width: calc(100% + 4rem);
  margin-inline: -2rem;
}

/* Or using viewport units for true full-bleed */
.article .full-bleed-vw {
  width: 100vw;
  margin-inline: calc(-50vw + 50%);
}
```

### 2. Card Component with Predictable Sizing

```css
.card {
  /* border-box ensures width includes padding and border */
  box-sizing: border-box;
  width: 100%;
  padding: 1.5rem;
  border: 1px solid #e0e0e0;
  border-radius: 8px;
  overflow: clip; /* true clip, no accidental scrolling */
}

.card__image {
  /* Break out of card padding */
  margin: -1.5rem -1.5rem 1.5rem;
  width: calc(100% + 3rem);
  aspect-ratio: 16 / 9;
  object-fit: cover;
  border-radius: 8px 8px 0 0;
}

.card__title {
  width: fit-content; /* shrink-wrap to text */
  padding: 0.25rem 0.75rem;
  background: #3498db;
  color: white;
  border-radius: 4px;
  margin-block-end: 0.75rem;
}
```

### 3. Layout with Margin Collapse Prevention

```css
/* Vertical rhythm using a flow utility */
.flow {
  display: flow-root; /* prevents child margins from escaping */
}

.flow > * + * {
  margin-block-start: 1.5rem; /* consistent spacing between siblings */
}

/* Alternative: use flex for guaranteed no-collapse spacing */
.stack {
  display: flex;
  flex-direction: column;
  gap: 1.5rem; /* no collapse, no :first-child overrides needed */
}
```

---

> **Key Takeaway:** Use `box-sizing: border-box` globally so dimensions are predictable, understand that margin collapsing only applies in block formatting contexts (flex, grid, and `flow-root` prevent it), and prefer modern properties like `aspect-ratio`, `overflow: clip`, and `gap` over their legacy workarounds.
