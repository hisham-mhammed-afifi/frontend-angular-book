# 2.4 Display Types

> The `display` property controls two things simultaneously: how an element participates in its parent's layout (outer type) and what formatting context it creates for its children (inner type). Understanding both is essential for controlling layout behavior.

---

## The Two-Value Model

Every `display` value actually sets two things:

- **Outer display type**: How the element behaves in its parent's flow (`block` or `inline`).
- **Inner display type**: What layout rules apply to the element's children (`flow`, `flow-root`, `flex`, `grid`, `table`).

The modern CSS Display Module Level 3 formalizes this with multi-keyword syntax:

| Legacy (single-value) | Modern (two-value) | Outer | Inner |
|---|---|---|---|
| `block` | `block flow` | block | flow |
| `inline` | `inline flow` | inline | flow |
| `inline-block` | `inline flow-root` | inline | flow-root |
| `flex` | `block flex` | block | flex |
| `inline-flex` | `inline flex` | inline | flex |
| `grid` | `block grid` | block | grid |
| `inline-grid` | `inline grid` | inline | grid |
| `flow-root` | `block flow-root` | block | flow-root |
| `inline-table` | `inline table` | inline | table |

```css
/* These pairs are equivalent */
.a { display: flex; }
.a { display: block flex; }

.b { display: inline-block; }
.b { display: inline flow-root; }

.c { display: inline-flex; }
.c { display: inline flex; }
```

Rules:
- If only the outer value is specified, the inner defaults to `flow`.
- If only the inner value is specified, the outer defaults to `block` (except `ruby`, which defaults to `inline`).
- Legacy single-keyword values remain fully supported.

Browser support for multi-keyword syntax: Chrome 115+, Firefox 70+, Safari 15+.

---

## Block

Block-level elements start on a new line and stretch to fill the full width of their parent by default.

```css
.block-element {
  display: block;
  /* Equivalent to: display: block flow; */
}
```

Characteristics:
- Starts on a new line.
- Fills available width (when `width` is `auto`).
- Respects `width`, `height`, `margin`, `padding`, and `border` on all sides.
- Vertical margins between adjacent block boxes collapse (the larger margin wins).
- Default for `<div>`, `<p>`, `<h1>`-`<h6>`, `<section>`, `<ul>`, `<form>`, `<header>`, `<footer>`.

---

## Inline

Inline-level elements flow with surrounding text and do not start a new line.

```css
.inline-element {
  display: inline;
  /* Equivalent to: display: inline flow; */
}
```

Characteristics:
- Does not start a new line. Flows with adjacent text and inline elements.
- `width` and `height` are **ignored**.
- Horizontal `margin` and `padding` work normally and push adjacent inline content.
- Vertical `margin` is **completely ignored**.
- Vertical `padding` renders visually (background shows) but does **not** push surrounding lines apart. It overlaps them.
- Default for `<span>`, `<a>`, `<strong>`, `<em>`, `<code>`, `<abbr>`.

```css
/* BAD: these properties have no effect on inline elements */
span {
  width: 200px;   /* ignored */
  height: 50px;   /* ignored */
  margin-top: 20px;  /* ignored */
  margin-bottom: 20px;  /* ignored */
}
```

The vertical padding behavior is one of the most confusing CSS quirks:

```css
span.highlight {
  padding: 20px;
  background: lightblue;
  /* The background extends 20px above and below,
     but surrounding lines are NOT pushed away.
     The padding simply overlaps adjacent content. */
}
```

---

## Inline-Block

`inline-block` combines inline flow behavior with full block box model support.

```css
.badge {
  display: inline-block;
  /* Equivalent to: display: inline flow-root; */
  padding: 4px 12px;
  margin: 2px;
  width: 100px;     /* works */
  height: 30px;     /* works */
  vertical-align: middle;
}
```

Characteristics:
- Flows inline (sits on the same line as siblings).
- Respects `width`, `height`, `margin`, and `padding` on all sides.
- Creates a new block formatting context internally (contains floats, prevents margin collapse).
- Supports `vertical-align` for positioning relative to the line box.

### The Whitespace Gap Problem

HTML whitespace (spaces, newlines) between `inline-block` elements renders as a visible gap of approximately 4px:

```html
<!-- These have a visible gap between them -->
<div class="box"></div>
<div class="box"></div>
```

```css
.box {
  display: inline-block;
  width: 100px;
  height: 100px;
  background: #3498db;
}
```

Solutions:

```css
/* Solution 1: Use flexbox instead (preferred) */
.parent { display: flex; }

/* Solution 2: font-size trick (legacy) */
.parent { font-size: 0; }
.box { font-size: 16px; }

/* Solution 3: Remove HTML whitespace */
```

In modern CSS, use flexbox or grid instead of `inline-block` for horizontal layouts. They do not have the whitespace gap problem and offer far more layout control.

## Hiding Elements: `display: none` vs Alternatives

### Comparison

| Method | Takes space | Accessible | Clickable | Animatable | Performance cost |
|---|---|---|---|---|---|
| `display: none` | No | No | No | No | Reflow (expensive) |
| `visibility: hidden` | Yes | No | No | Yes | Repaint |
| `opacity: 0` | Yes | **Yes** | **Yes** | **Yes** (smooth) | Composite (cheap) |
| `content-visibility: hidden` | Configurable | No | No | No | Cached (fast toggle) |

### `display: none`

Removes the element from the layout entirely. No box is generated. The element is invisible to both sighted users and screen readers.

```css
.hidden { display: none; }
```

**Gotcha:** Toggling `display: none` on and off destroys and recreates the entire rendering state for the element and all its descendants. This is expensive for large subtrees.

### `visibility: hidden`

The element is invisible but still occupies space. Screen readers ignore it.

```css
.invisible { visibility: hidden; }
```

**Unique behavior:** Children can override this. A child with `visibility: visible` appears even when the parent is hidden:

```css
.parent { visibility: hidden; }
.parent .visible-child { visibility: visible; }
/* The child is visible. This does NOT work with display: none or opacity: 0. */
```

### `opacity: 0`

The element is fully transparent but still occupies space, remains in the accessibility tree, and is still interactive (clickable, focusable, tabbable).

```css
.transparent {
  opacity: 0;
  /* Users can still click this, screen readers still announce it */
}
```

**Use case:** Smooth fade transitions. Combine with `visibility` and `pointer-events` for proper hiding:

```css
.modal {
  opacity: 0;
  visibility: hidden;
  pointer-events: none;
  transition: opacity 0.3s, visibility 0.3s;
}

.modal.open {
  opacity: 1;
  visibility: visible;
  pointer-events: auto;
}
```

### The Visually Hidden Pattern

For content that should be accessible to screen readers but invisible to sighted users:

```css
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip-path: inset(50%);
  white-space: nowrap;
  border: 0;
}
```

This keeps the element in the accessibility tree and the tab order while removing it from visual display. Do not use `display: none` or `visibility: hidden` for this purpose.

## `display: flow-root`

Creates a new block formatting context (BFC) with zero side effects. This is the purpose-built way to establish a BFC.

```css
.container {
  display: flow-root;
  /* Equivalent to: display: block flow-root; */
}
```

### What a BFC Does

1. **Contains floats.** Floated children are contained within the BFC parent. The parent's height expands to wrap them.
2. **Prevents margin collapse.** Child margins do not escape through the parent's edges.
3. **Prevents float intrusion.** A BFC box does not overlap adjacent floats.

### Before `flow-root` (Legacy Hacks)

| Method | Side effect |
|---|---|
| `overflow: hidden` | Clips overflowing content (box-shadows, tooltips, positioned children) |
| `overflow: auto` | May create unwanted scrollbars |
| `display: inline-block` | Changes outer display to inline |
| `float: left` | Takes element out of flow |

```css
/* BAD: Using overflow to create BFC */
.clearfix {
  overflow: hidden; /* clips shadows, tooltips, etc. */
}

/* GOOD: Purpose-built BFC */
.clearfix {
  display: flow-root;
}
```

`display: flow-root` is the correct modern approach. It does exactly one thing (creates a BFC) with no side effects.

---

## `display: contents`

Makes the element's own box disappear. The element generates no boxes, but its children participate in the parent's layout as if the wrapper did not exist.

```css
.wrapper {
  display: contents;
  /* The wrapper's box vanishes. Its children become direct layout participants
     of the wrapper's parent. */
}
```

### Use Case: Grid Grandchild Participation

Grid can only place direct children. If semantic HTML requires a wrapper, `display: contents` lets grandchildren participate in the grid:

```html
<div class="grid">
  <div>Item 1</div>
  <div class="group" style="display: contents;">
    <div>Item 2</div>  <!-- participates in .grid's grid -->
    <div>Item 3</div>  <!-- participates in .grid's grid -->
  </div>
  <div>Item 4</div>
</div>
```

```css
.grid {
  display: grid;
  grid-template-columns: repeat(4, 1fr);
  gap: 1rem;
}
```

Without `display: contents`, `.group` would be a single grid item containing two children. With it, the group's children become direct grid items.

### What It Does NOT Do

- Does not reset styles. Inherited properties (`color`, `font`, etc.) still propagate through it.
- Does not remove the element from the DOM. Event listeners still fire, JavaScript still works.
- Does not remove `::before` and `::after` pseudo-elements.

### Accessibility Warning

`display: contents` historically removed elements from the accessibility tree in some browsers. This is being fixed but remains inconsistent:

- **Safe:** Using `display: contents` on `<div>` or `<span>` (generic wrappers with no semantic meaning).
- **Unsafe:** Using it on `<button>`, `<table>`, `<ul>`, `<ol>`, `<h1>`-`<h6>`, `<details>`, or any element with inherent semantics. Some browsers strip the element's role, breaking screen reader navigation.

```css
/* SAFE */
div.flex-wrapper { display: contents; }

/* DANGEROUS: may lose button semantics in some browsers */
button.contents { display: contents; }
```

## `display: list-item`

Generates a principal block box plus a marker box (the `::marker` pseudo-element). This is the default display of `<li>` and `<summary>`.

```css
/* Make any element behave like a list item */
.custom-list-item {
  display: list-item;
  list-style-type: disc;
}
```

### Three-Value Syntax

`list-item` can combine with outer and inner types:

```css
/* Default <li> behavior */
.item { display: block flow list-item; }

/* Inline list item (uncommon but valid) */
.inline-item { display: inline flow list-item; }
```

### Styling the Marker

```css
li::marker {
  color: #3498db;
  font-size: 1.2em;
  font-weight: bold;
}

/* Custom content replaces default bullet/number */
li::marker {
  content: ">> ";
}
```

The `::marker` pseudo-element supports a limited set of properties: `color`, `font-*`, `content`, `unicode-bidi`, `direction`, `animation-*`, `transition-*`, `white-space`.

## CSS Table Display Values

CSS can replicate HTML table layout on any elements:

| Value | HTML equivalent |
|---|---|
| `table` | `<table>` |
| `table-row` | `<tr>` |
| `table-cell` | `<td>` / `<th>` |
| `table-row-group` | `<tbody>` |
| `table-header-group` | `<thead>` |
| `table-footer-group` | `<tfoot>` |
| `table-column` | `<col>` |
| `table-column-group` | `<colgroup>` |
| `table-caption` | `<caption>` |
| `inline-table` | (inline-level table container) |

```css
/* Legacy vertical centering (pre-flexbox era) */
.parent {
  display: table;
  width: 100%;
  height: 300px;
}
.child {
  display: table-cell;
  vertical-align: middle;
  text-align: center;
}
```

### Anonymous Box Generation

If you use `display: table-cell` without a `table-row` parent, the browser generates anonymous boxes to complete the table structure. Text placed directly in a `display: table` element gets wrapped in anonymous `table-row` and `table-cell` boxes automatically.

In modern CSS, flexbox and grid have replaced nearly all CSS table layout use cases. CSS table display values persist primarily in legacy codebases.

## `content-visibility`

A performance optimization that controls whether the browser renders an element's content:

| Value | Renders? | Cached? | Searchable (Ctrl+F)? |
|---|---|---|---|
| `visible` (default) | Yes | N/A | Yes |
| `auto` | When near viewport | Yes | Yes |
| `hidden` | No | **Yes** | No |

### `content-visibility: auto`

The browser skips rendering of off-screen content entirely. When the element scrolls near the viewport, it renders on demand:

```css
.article-section {
  content-visibility: auto;
  contain-intrinsic-size: auto 500px;
}
```

Real-world impact: web.dev measured rendering time dropping from 232ms to 30ms (7x improvement) on a long-form page.

### `contain-intrinsic-size`

When `content-visibility: auto` skips rendering, the element has no intrinsic height. This causes layout shift when the content scrolls into view. Fix this with `contain-intrinsic-size`:

```css
.card {
  content-visibility: auto;
  contain-intrinsic-size: auto 300px;
  /* Uses 300px initially.
     "auto" remembers the actual rendered height after first paint. */
}
```

### `content-visibility: hidden` vs `display: none`

| Feature | `content-visibility: hidden` | `display: none` |
|---|---|---|
| Box generated | Yes | No |
| Rendering state cached | Yes (fast re-show) | No (full re-render) |
| Takes up space | Configurable | No |
| Best for | Cached SPA views, tab panels | Full removal |

```css
/* Tab panel: cached hidden content */
.tab-panel { content-visibility: hidden; }
.tab-panel.active { content-visibility: visible; }
```

### Cautions

- Do not apply `content-visibility: auto` to above-the-fold content. It delays rendering of initially visible elements, hurting Largest Contentful Paint (LCP).
- It implicitly applies `contain: layout style paint`, which creates a new stacking context and containing block.

Browser support: Chrome 85+, Firefox 125+, Safari 18+.

## Anonymous Boxes

Browsers automatically create invisible boxes to maintain valid formatting contexts. You cannot target these with CSS selectors.

### Mixed Block/Inline Children

A block container must contain only block-level boxes or only inline-level boxes. When you mix them, browsers wrap inline content in anonymous block boxes:

```html
<div>
  Bare text here         <!-- wrapped in anonymous block box -->
  <p>A paragraph</p>     <!-- explicit block box -->
  More bare text         <!-- wrapped in anonymous block box -->
</div>
```

### Flex/Grid Whitespace Items

Bare text nodes inside flex/grid containers become anonymous flex/grid items:

```html
<div style="display: flex;">
  Some text              <!-- becomes anonymous flex item -->
  <div>Explicit item</div>
</div>
```

This is why whitespace-only text nodes (spaces, newlines between elements) sometimes create unexpected empty flex/grid items.

## Replaced Elements and Display

Replaced elements (`<img>`, `<video>`, `<canvas>`, `<iframe>`, `<input>`, `<textarea>`, `<select>`) have special display behavior:

- `<img>` is `inline` by default but behaves like `inline-block`: it respects `width` and `height`.
- Setting only one dimension on a replaced element automatically scales the other proportionally (maintains aspect ratio). Non-replaced elements do not do this.
- `object-fit` and `object-position` control how the replaced content fits within its box.

### Blockification in Flex/Grid

When a replaced inline element becomes a flex or grid item, it undergoes **blockification**: its outer display type becomes `block`. However, replaced elements have special flex/grid sizing: they default to `align-self: start` (not `stretch`) to avoid distorting the content.

```css
.flex-container {
  display: flex;
  align-items: stretch; /* default */
}

/* <img> inside does NOT stretch vertically by default.
   It maintains its aspect ratio instead. */
```

## Common Mistakes

1. **Setting `width`/`height` on inline elements.** Both properties are completely ignored on `display: inline`. Use `inline-block` or `block`.

2. **Expecting vertical margin on inline elements.** Vertical margins have zero effect on inline elements. Vertical padding renders visually but does not push surrounding lines.

3. **Using `display: none` for accessible content.** Screen readers cannot see `display: none` content. Use the visually hidden pattern (`.sr-only`) for screen-reader-only content.

4. **Using `opacity: 0` as "hidden."** The element remains clickable, focusable, and announced by screen readers. Combine with `visibility: hidden` and `pointer-events: none` if you want true hiding.

5. **Using `overflow: hidden` to create a BFC.** Use `display: flow-root` instead. It was purpose-built for this with zero side effects.

6. **Applying `display: contents` to semantic elements.** Buttons, tables, lists, and headings may lose their accessibility semantics in some browsers.

7. **Forgetting `contain-intrinsic-size` with `content-visibility: auto`.** Without it, elements have zero height when off-screen, causing massive layout shift when they scroll into view.

8. **Using `inline-block` for horizontal layouts in new code.** Flexbox and grid do not have whitespace gap problems and offer far superior alignment and distribution.

9. **Toggling `display: none` on large DOM trees.** This destroys and recreates rendering state. For frequently toggled content, use `content-visibility: hidden` to cache the rendering state.

10. **Expecting `display: inline` on flex/grid items to work.** Flex and grid items are blockified automatically. Setting `display: inline` on a flex item has no effect.

---

## Best Practices

1. **Use the two-value syntax for clarity in new projects.** `display: inline flex` communicates intent better than `display: inline-flex`.

2. **Prefer `flow-root` over `overflow: hidden` for BFC creation.** It does exactly one thing with no side effects.

3. **Use `content-visibility: auto` for long pages.** Apply it to below-the-fold sections with `contain-intrinsic-size` to prevent layout shift. Significant performance gains with minimal effort.

4. **Combine `opacity`, `visibility`, and `pointer-events` for animated hiding.** This gives smooth transitions, removes interactivity, and removes from the accessibility tree.

5. **Reserve `display: contents` for generic wrappers only.** Only use it on `<div>` or `<span>`. Never on interactive or semantic elements.

6. **Use the visually hidden pattern for screen-reader-only content.** Never use `display: none` or `visibility: hidden` for content that should be accessible.

7. **Prefer flexbox/grid over `inline-block` for layouts.** They solve all the problems `inline-block` has (whitespace gaps, baseline quirks, limited alignment).

## Real-World Use Cases

### 1. Animated Modal with Proper Hiding

```css
.modal-overlay {
  position: fixed;
  inset: 0;
  background: rgba(0, 0, 0, 0.5);

  /* Hidden state */
  opacity: 0;
  visibility: hidden;
  pointer-events: none;
  transition: opacity 0.3s ease, visibility 0.3s ease;
}

.modal-overlay.active {
  opacity: 1;
  visibility: visible;
  pointer-events: auto;
}

.modal-content {
  transform: translateY(20px) scale(0.95);
  transition: transform 0.3s ease;
}

.modal-overlay.active .modal-content {
  transform: translateY(0) scale(1);
}
```

### 2. Long Article with Performance Optimization

```css
/* First section renders normally (above the fold) */
.article-section:first-of-type {
  content-visibility: visible;
}

/* Remaining sections use lazy rendering */
.article-section {
  content-visibility: auto;
  contain-intrinsic-size: auto 600px;
}

/* Images within lazy sections */
.article-section img {
  content-visibility: auto;
  contain-intrinsic-size: auto 400px 300px;
}
```

### 3. Grid Layout with Semantic Grouping

```css
.product-grid {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(250px, 1fr));
  gap: 1.5rem;
}

/* Semantic grouping wrapper disappears from layout */
.product-category {
  display: contents;
  /* Children participate directly in .product-grid's grid */
}

/* Each product card is a direct grid item */
.product-card {
  display: flex;
  flex-direction: column;
  border: 1px solid #e0e0e0;
  border-radius: 8px;
  overflow: clip;
}
```

```html
<div class="product-grid">
  <section class="product-category" aria-label="Electronics">
    <div class="product-card">Phone</div>
    <div class="product-card">Laptop</div>
  </section>
  <section class="product-category" aria-label="Clothing">
    <div class="product-card">Shirt</div>
    <div class="product-card">Jacket</div>
  </section>
</div>
<!-- All 4 cards flow in a single grid, despite being grouped semantically -->
```

Note: Using `display: contents` on `<section>` may affect accessibility in some browsers. Test with screen readers or use `<div>` with `role="group"` as an alternative.

---

> **Key Takeaway:** Every `display` value sets both an outer type (how the element participates in its parent's layout) and an inner type (what formatting context it creates for its children); understanding this two-value model, along with the practical differences between hiding methods and the performance benefits of `content-visibility`, is what separates layout control from layout guessing.
