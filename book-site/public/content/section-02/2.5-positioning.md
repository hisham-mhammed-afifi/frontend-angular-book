# 2.5 Positioning

> The `position` property controls how an element is placed in the document: in normal flow, offset from its normal position, relative to an ancestor, fixed to the viewport, or stuck during scroll. Combined with stacking contexts and z-index, it determines layering order.

---

## Position Values

| Value | In flow? | Positioned relative to | Creates stacking context? |
|---|---|---|---|
| `static` | Yes | N/A (normal flow) | No |
| `relative` | Yes (space preserved) | Its own normal position | Only with `z-index` other than `auto` |
| `absolute` | No (removed from flow) | Nearest positioned ancestor's padding box | Only with `z-index` other than `auto` |
| `fixed` | No (removed from flow) | Viewport (with exceptions) | Always |
| `sticky` | Yes (space preserved) | Nearest scrolling ancestor | Always |

The offset properties `top`, `right`, `bottom`, `left` (and the `inset` shorthand) only work on positioned elements. They have no effect on `static` elements. `z-index` also requires positioning (with exceptions for flex/grid children).

---

## `static` (Default)

The element is in normal flow. Offset properties and `z-index` are ignored.

```css
.element {
  position: static; /* default, usually not written */
  top: 20px;        /* ignored */
  z-index: 10;      /* ignored */
}
```

---

## `relative`

The element stays in flow (its original space is preserved), but it can be visually offset from its normal position. Other elements do not move to fill the gap.

```css
.shifted {
  position: relative;
  top: 10px;   /* shifts down 10px from normal position */
  left: 20px;  /* shifts right 20px from normal position */
}
```

Primary use cases:
- Establishing a containing block for `absolute` children.
- Minor visual adjustments without affecting surrounding layout.
- Enabling `z-index` on an element.

---

## `absolute`

The element is removed from flow entirely. No space is reserved. It is positioned against the padding box of its nearest positioned ancestor (any ancestor with `position` other than `static`).

```css
.parent {
  position: relative; /* establishes containing block */
}

.child {
  position: absolute;
  top: 0;
  right: 0;
  /* Sits at the top-right corner of .parent's padding box */
}
```

If no positioned ancestor exists, the element positions against the initial containing block (the viewport area of the `<html>` element).

### Stretching with `inset`

When opposing inset values are set and no explicit dimension is given, the element stretches to fill the space:

```css
.overlay {
  position: absolute;
  inset: 0;
  /* Equivalent to: top: 0; right: 0; bottom: 0; left: 0; */
  /* Stretches to fill the entire containing block */
}

.sidebar {
  position: absolute;
  top: 0;
  bottom: 0;
  left: 0;
  width: 250px;
  /* Stretches vertically, fixed width */
}
```

### `inset` Shorthand

`inset` sets `top`, `right`, `bottom`, `left` using the same multi-value syntax as `margin`:

```css
inset: 0;                     /* all sides 0 */
inset: 10px 20px;             /* top/bottom 10px, left/right 20px */
inset: 10px 20px 30px;        /* top 10px, left/right 20px, bottom 30px */
inset: 10px 20px 30px 40px;   /* top, right, bottom, left */
```

### `width: 100%` vs `inset: 0`

These are not equivalent on absolute elements:

```css
/* Uses containing block's CONTENT box width */
.a {
  position: absolute;
  width: 100%;
}

/* Stretches to containing block's PADDING box edges */
.b {
  position: absolute;
  inset: 0;
  /* or: left: 0; right: 0; */
}
```

`inset: 0` is usually what you want because it matches the containing block's padding box, which is the reference for absolute positioning.

---

## `fixed`

The element is removed from flow and positioned relative to the viewport. It stays in place during scroll.

```css
.fixed-header {
  position: fixed;
  top: 0;
  left: 0;
  right: 0;
  height: 60px;
  z-index: 100;
}

/* Prevent content from hiding behind the fixed header */
body {
  padding-top: 60px;
}
```

### The Transform/Filter Gotcha

`position: fixed` breaks when any ancestor has `transform`, `filter`, `backdrop-filter`, `perspective`, `contain: layout`, `contain: paint`, or `will-change` set to a non-default value. The fixed element repositions relative to that ancestor instead of the viewport.

```css
/* This BREAKS fixed positioning of children */
.parent {
  transform: translateX(0); /* even an identity transform */
}

.parent .modal {
  position: fixed;
  inset: 0;
  /* Now behaves like absolute, relative to .parent, NOT the viewport */
}
```

This is one of the most common CSS bugs. The fix is to move the fixed element outside the transformed ancestor in the DOM. There is no CSS-only workaround that keeps both the transform and true fixed behavior.

Properties that create this containing block override:
- `transform` (any value other than `none`)
- `filter` (any value other than `none`)
- `backdrop-filter` (any value other than `none`)
- `perspective` (any value other than `none`)
- `rotate`, `scale`, `translate` (any value)
- `contain: layout` or `contain: paint`
- `will-change` specifying any of the above

---

## `sticky`

A hybrid of `relative` and `fixed`. The element stays in flow (space preserved) and scrolls normally until it crosses a threshold, then "sticks" in place until it reaches the edge of its containing block.

```css
.sticky-header {
  position: sticky;
  top: 0; /* sticks when top edge reaches viewport top */
  background: white;
  z-index: 10;
}
```

### Requirements

1. At least one inset property (`top`, `bottom`, `left`, `right`) must be set to a non-`auto` value.
2. The sticky element must be shorter than its parent container.
3. The parent must have enough scrollable content for sticking to be observable.

### All the Ways Sticky Can Fail

**1. No inset value set:**

```css
/* BAD: nothing happens, indistinguishable from relative */
.header { position: sticky; }

/* GOOD: threshold defined */
.header { position: sticky; top: 0; }
```

**2. Ancestor has `overflow` other than `visible`:**

Any ancestor with `overflow: hidden`, `overflow: auto`, `overflow: scroll`, or even `overflow-x: hidden` creates a new scroll container. The sticky element sticks to that container instead of the viewport.

```css
/* BAD: the overflow: hidden ancestor breaks sticky */
.wrapper {
  overflow: hidden; /* sticky child sticks to THIS, not viewport */
}

/* FIX: use overflow: clip instead (does not create scroll container) */
.wrapper {
  overflow: clip;
}
```

**3. Parent is the same height as the sticky element:**

The sticky element can only stick within its parent. If the parent wraps tightly around the sticky child with no extra content, there is no room to stick.

**4. Flex/grid `align-items: stretch` (default):**

In flex and grid layouts, items stretch to fill their container by default. This makes the sticky element as tall as its container, so it reaches its threshold instantly and is always "stuck" (or never sticks visibly).

```css
/* BAD: sidebar stretches to full grid row height */
.layout {
  display: grid;
  grid-template-columns: 250px 1fr;
}

/* GOOD: align-items: start prevents stretching */
.layout {
  display: grid;
  grid-template-columns: 250px 1fr;
  align-items: start;
}

.sidebar {
  position: sticky;
  top: 1rem;
}
```

### Sticky in Both Directions

Sticky works on both axes simultaneously. This is essential for data tables with both sticky row headers and column headers:

```css
thead th {
  position: sticky;
  top: 0;
  z-index: 2;
  background: white;
}

tbody th {
  position: sticky;
  left: 0;
  z-index: 1;
  background: white;
}

/* Corner cell needs both */
thead th:first-child {
  position: sticky;
  top: 0;
  left: 0;
  z-index: 3;
  background: white;
}
```

## Containing Blocks

The containing block determines how percentage values for `width`, `height`, `padding`, `margin`, and offset properties are calculated:

| Element's `position` | Containing block |
|---|---|
| `static` or `relative` | Content box of nearest block-level ancestor |
| `absolute` | Padding box of nearest positioned ancestor |
| `fixed` | Viewport (initial containing block) |
| `sticky` | Nearest ancestor with a scrolling mechanism |

For `absolute` elements, if no positioned ancestor exists, the initial containing block (viewport) is used.

```css
.container {
  position: relative;
  padding: 20px;
  width: 400px;
}

.positioned {
  position: absolute;
  width: 50%; /* 50% of 400px = 200px (uses container's padding box for offset, content box for percentage resolution) */
  top: 0;
  right: 0;
}
```

## Stacking Contexts and `z-index`

### Default Painting Order (No z-index)

From back to front:
1. Background and borders of the root element
2. Non-positioned descendants, in DOM order
3. Positioned descendants, in DOM order

Positioned elements always paint on top of non-positioned elements, regardless of DOM order:

```css
/* Non-positioned: always behind positioned elements */
.background { /* position: static (default) */ }

/* Positioned: always in front of non-positioned */
.foreground { position: relative; }
```

### Full Stacking Order Within a Context

From back to front:
1. The context's own background and borders
2. Child stacking contexts with negative `z-index` (most negative first)
3. Non-positioned, non-floated block-level descendants
4. Non-positioned floated descendants
5. Non-positioned inline-level descendants
6. Positioned descendants with `z-index: auto` or `z-index: 0`
7. Child stacking contexts with positive `z-index` (lowest first)

### The Critical Rule

**z-index values are compared only within the same stacking context.** A `z-index: 99999` inside one stacking context cannot appear above a `z-index: 1` in a sibling stacking context if the sibling's context has a higher z-index on its root.

```css
.sidebar {
  position: relative;
  z-index: 1; /* stacking context with priority 1 */
}

.main {
  position: relative;
  z-index: 2; /* stacking context with priority 2 */
}

/* This dropdown can NEVER appear above .main */
.sidebar .dropdown {
  position: absolute;
  z-index: 99999; /* trapped inside sidebar's z-index: 1 context */
}
```

### What Creates a Stacking Context

| Trigger | Condition |
|---|---|
| Root element | `<html>` always creates one |
| `position: relative/absolute` + `z-index` | Any `z-index` value other than `auto` |
| `position: fixed` or `sticky` | Always (even without explicit z-index) |
| Flex/grid child + `z-index` | Any `z-index` value other than `auto` |
| `opacity` | Less than `1` |
| `transform` | Any value other than `none` |
| `filter` | Any value other than `none` |
| `backdrop-filter` | Any value other than `none` |
| `perspective` | Any value other than `none` |
| `clip-path` | Any value other than `none` |
| `mask` / `mask-image` | Any value other than `none` |
| `mix-blend-mode` | Any value other than `normal` |
| `isolation: isolate` | Always |
| `will-change` | Specifying a stacking-context-creating property |
| `contain: layout` or `contain: paint` | Always |
| `container-type: size` or `inline-size` | Always |

### `z-index` Without Positioning

`z-index` requires `position` to be something other than `static`. There are two exceptions:
- **Flex children** can use `z-index` without explicit positioning.
- **Grid children** can use `z-index` without explicit positioning.

```css
/* Works: flex child */
.flex-parent { display: flex; }
.flex-child { z-index: 10; } /* no position needed */

/* Does NOT work: normal flow */
.normal-child { z-index: 10; } /* ignored without position */
```

## The `isolation` Property

`isolation: isolate` creates a new stacking context with zero side effects. Unlike `opacity`, `transform`, or `filter`, it does not change appearance, layout, or performance characteristics.

```css
.component {
  isolation: isolate;
}
```

### Use Cases

**1. Scoping z-index within components:**

```css
.card {
  isolation: isolate;
  /* Internal z-index values cannot escape this context */
}

.card .badge { z-index: 1; }
.card .overlay { z-index: 2; }
/* These z-index values only compete with each other, not with the rest of the page */
```

**2. Preventing `mix-blend-mode` from leaking:**

```css
.blended-section {
  isolation: isolate;
  /* mix-blend-mode on children only blends within this container */
}
```

**3. Architectural z-index management:**

```css
/* Each major UI section gets its own stacking context */
.header { isolation: isolate; position: relative; z-index: 30; }
.main   { isolation: isolate; position: relative; z-index: 10; }
.footer { isolation: isolate; position: relative; z-index: 20; }
.modal-layer { isolation: isolate; position: relative; z-index: 100; }
```

## Centering with Absolute Positioning

### Transform Method (Unknown Dimensions)

```css
.parent { position: relative; }
.child {
  position: absolute;
  top: 50%;
  left: 50%;
  transform: translate(-50%, -50%);
}
```

The element is shifted 50% from the top and left of its containing block, then pulled back by 50% of its own width and height. Works with any content size.

### Inset + Auto Margin Method (Known Dimensions)

```css
.parent { position: relative; }
.child {
  position: absolute;
  inset: 0;
  margin: auto;
  width: 300px;
  height: 200px;
}
```

When all four inset values are 0 and margins are `auto`, the browser distributes the remaining space equally. Requires explicit `width` and `height`.

### Horizontal-Only Centering

```css
.child {
  position: absolute;
  left: 0;
  right: 0;
  margin-inline: auto;
  width: 300px; /* must have explicit width */
}
```

## CSS Anchor Positioning

The anchor positioning API allows positioning an element relative to another element (the "anchor") anywhere on the page, without a parent-child DOM relationship. This replaces JavaScript-based tooltip and popover positioning.

### Declaring an Anchor

```css
.trigger {
  anchor-name: --menu-trigger;
}
```

### Positioning Relative to the Anchor

```css
.menu {
  position: absolute;
  position-anchor: --menu-trigger;
  position-area: bottom; /* places menu below the trigger */
}
```

`position-area` values include: `top`, `bottom`, `left`, `right`, `center`, `top left`, `top right`, `bottom left`, `bottom right`, and logical equivalents (`block-start`, `inline-end`, etc.).

### Fallback Positioning

When the positioned element would overflow the viewport, fallbacks try alternative positions:

```css
.tooltip {
  position: absolute;
  position-anchor: --button;
  position-area: top;
  position-try-fallbacks: flip-block; /* try bottom if top overflows */
}
```

Built-in flip keywords:
- `flip-block`: flips in block axis (top to bottom)
- `flip-inline`: flips in inline axis (left to right)
- `flip-block flip-inline`: flips both axes

### Custom Fallbacks

```css
@position-try --below {
  position-area: bottom;
}

@position-try --right-side {
  position-area: right;
}

.tooltip {
  position: absolute;
  position-anchor: --trigger;
  position-area: top;
  position-try-fallbacks: --below, --right-side;
}
```

The browser tries each fallback in order until one fits within the viewport.

Browser support: Chrome 125+, Firefox 131+, Safari 18+, Edge 125+.

## Common Mistakes

1. **Overusing `absolute` for layout.** Absolute positioning removes elements from flow and breaks responsive design. Use flexbox/grid for layout. Reserve `absolute` for overlays, badges, and decorative elements.

2. **Forgetting the positioned ancestor.** Without a positioned parent, `absolute` elements position against the viewport area. Always set `position: relative` on the intended container.

3. **Using `fixed` when `sticky` is appropriate.** `fixed` removes the element from flow and requires manual content offset (e.g., `padding-top` on the body). `sticky` keeps the element in flow and scopes sticking to its parent.

4. **`transform`/`filter` breaking `fixed` children.** Any ancestor with `transform`, `filter`, `backdrop-filter`, or `perspective` turns `fixed` into `absolute`. Move the fixed element outside the transformed ancestor.

5. **z-index without positioning.** `z-index` is ignored on `static` elements (except flex/grid children). Always pair with `position: relative` (or another positioning value).

6. **z-index escalation.** Adding `z-index: 99999` is a symptom of not understanding stacking contexts. Use `isolation: isolate` to scope contexts and keep values low (1-10).

7. **`overflow` breaking `sticky`.** Any ancestor with `overflow: hidden`, `auto`, or `scroll` creates a new scroll container that captures the sticky element. Use `overflow: clip` instead if you need to clip content without affecting sticky behavior.

8. **Sticky in flex/grid without `align-items: start`.** The default `stretch` makes the sticky element fill its container height, leaving no room to stick.

9. **Sticky without an inset value.** `position: sticky` without `top`, `bottom`, `left`, or `right` behaves identically to `relative`. You must define the threshold.

10. **Accidental stacking contexts.** Setting `opacity: 0.99` or `transform: translateZ(0)` for "performance" creates a stacking context, trapping all descendant z-index values within it.

---

## Best Practices

1. **Use `isolation: isolate` on component roots.** This scopes z-index within each component and prevents cross-component z-index conflicts.

2. **Keep z-index values low and documented.** Define a z-index scale (e.g., dropdown: 10, sticky-header: 20, modal: 30, toast: 40) and stick to it.

3. **Prefer `sticky` over `fixed` for section-scoped sticking.** Sticky keeps the element in flow, works within its parent's bounds, and does not need manual content offsets.

4. **Use `inset: 0` instead of `top: 0; right: 0; bottom: 0; left: 0`.** It is shorter and communicates "fill the container" intent clearly.

5. **Use anchor positioning for tooltips and popovers.** The CSS anchor positioning API replaces complex JavaScript positioning libraries with declarative CSS, including automatic viewport-aware fallbacks.

6. **Test fixed elements with transformed ancestors.** If any parent uses `transform`, `filter`, or `will-change`, fixed children will misbehave. Restructure the DOM to avoid this.

7. **Use `overflow: clip` instead of `overflow: hidden` near sticky elements.** `clip` does not create a scroll container and does not break sticky positioning.

8. **Use flexbox/grid for layout, positioning for layering.** Positioning is for overlays, tooltips, badges, sticky headers, and stacking. It is not a layout system.

## Real-World Use Cases

### 1. Fixed Header with Sticky Sidebar

```css
.header {
  position: fixed;
  top: 0;
  left: 0;
  right: 0;
  height: 60px;
  background: white;
  border-bottom: 1px solid #e0e0e0;
  z-index: 20;
}

.layout {
  display: grid;
  grid-template-columns: 250px 1fr;
  align-items: start; /* critical for sticky sidebar */
  padding-top: 60px;  /* offset for fixed header */
}

.sidebar {
  position: sticky;
  top: 76px; /* 60px header + 16px gap */
  max-height: calc(100vh - 76px);
  overflow-y: auto;
}
```

### 2. Notification Badge on Avatar

```css
.avatar-container {
  position: relative;
  display: inline-block;
}

.avatar {
  width: 48px;
  height: 48px;
  border-radius: 50%;
  object-fit: cover;
}

.notification-badge {
  position: absolute;
  top: -2px;
  right: -2px;
  min-width: 18px;
  height: 18px;
  padding: 0 5px;
  border-radius: 9px;
  background: #e74c3c;
  color: white;
  font-size: 11px;
  font-weight: 600;
  line-height: 18px;
  text-align: center;
  border: 2px solid white;
}
```

### 3. Tooltip with Anchor Positioning

```css
.trigger-btn {
  anchor-name: --tooltip-anchor;
}

.tooltip {
  position: fixed;
  position-anchor: --tooltip-anchor;
  position-area: top;
  position-try-fallbacks: flip-block, flip-inline;

  margin-bottom: 8px;
  padding: 8px 12px;
  background: #1a1a2e;
  color: white;
  border-radius: 6px;
  font-size: 0.875rem;
  white-space: nowrap;
  pointer-events: none;

  /* Hidden by default */
  opacity: 0;
  transition: opacity 0.15s;
}

.trigger-btn:hover + .tooltip,
.trigger-btn:focus-visible + .tooltip {
  opacity: 1;
}
```

---

> **Key Takeaway:** Positioning controls layering and element placement outside normal flow; understand containing blocks (especially the transform/filter gotcha with fixed), use `isolation: isolate` to scope stacking contexts, prefer `sticky` over `fixed` when the element should stay within its parent, and use the CSS anchor positioning API to replace JavaScript-based tooltip and popover placement.
