# 3.11 Touch-Friendly Design

> Touch is the primary input method for over half of all web traffic. Designing for touch is not an afterthought applied to a desktop layout; it is a fundamental constraint that shapes sizing, spacing, event handling, scroll behavior, and feedback. This guide covers the full surface area: tap target requirements, the `touch-action` CSS property, interaction media features, pointer and touch event APIs, gesture support, navigation and form patterns, scroll mechanics, safe areas, accessibility, and performance. Every concept includes code. Zero Angular references.

## 1. Touch Target Sizing

A finger is not a pixel-precise cursor. The average adult fingertip covers roughly 10mm of screen surface, which translates to about 40 CSS pixels. Undersized targets cause rage taps, accidental activations, and user frustration.

### The Standards

| Source | Minimum Size | Level | Notes |
|---|---|---|---|
| WCAG 2.2 SC 2.5.8 | 24 x 24 CSS px | AA (minimum) | New in WCAG 2.2. Floor, not optimum. |
| WCAG 2.1 SC 2.5.5 | 44 x 44 CSS px | AAA (enhanced) | Original target size criterion. |
| Apple HIG | 44 x 44 pt | Platform guideline | 44pt = 44 CSS px at 1x. |
| Material Design | 48 x 48 dp | Platform guideline | dp maps to CSS px on web. |
| Microsoft Fluent | 40 x 40 px (min), 48 x 48 px (recommended) | Platform guideline | |

**Key distinction**: the target is the interactive hit area, not the visible graphic. An icon can be 24x24 visually while its tap area is 48x48.

### WCAG 2.2 SC 2.5.8 Details

This is a Level AA criterion, meaning it applies to most conformance targets. The rule:

> The size of the target for pointer inputs is at least 24 by 24 CSS pixels, except when:
> - **Spacing**: the undersized target has enough surrounding space so that a 24px-diameter circle centered on it does not overlap any other target or 24px circle of another target.
> - **Inline**: the target is within a sentence or block of text (e.g., inline links).
> - **User agent control**: the target's size is set by the user agent, not the author (e.g., default checkboxes).
> - **Essential**: the size is essential to the information being conveyed.

### Implementation Techniques

#### Technique 1: Minimum Size with min-width/min-height

```css
/* Base touch target */
.touch-target {
  min-width: 44px;
  min-height: 44px;
  display: inline-flex;
  align-items: center;
  justify-content: center;
}
```

#### Technique 2: Padding-Expanded Hit Area

When the visible element must stay small, expand the tap area with padding:

```css
/* Small icon, large tap area */
.icon-button {
  /* Reset native button styles */
  background: none;
  border: none;
  cursor: pointer;

  /* The icon itself is 24x24 */
  width: 24px;
  height: 24px;

  /* Padding expands the tappable area to 48x48 */
  padding: 12px;

  /* Prevent padding from inflating the visual size */
  box-sizing: content-box;
}
```

#### Technique 3: Pseudo-Element Hit Area Expansion

When you cannot add padding (layout constraints), use a pseudo-element:

```css
.compact-button {
  position: relative;
  /* Visual size can be anything */
}

.compact-button::after {
  content: "";
  position: absolute;
  /* Expand clickable area 12px in every direction */
  top: -12px;
  right: -12px;
  bottom: -12px;
  left: -12px;
}
```

#### Technique 4: Responsive Target Sizing

```css
/* Larger targets on coarse (touch) pointers */
.action-button {
  min-height: 36px; /* fine pointer default */
  padding: 6px 16px;
}

@media (pointer: coarse) {
  .action-button {
    min-height: 48px; /* touch devices */
    padding: 12px 24px;
  }
}
```

### Spacing Between Targets

Adequate spacing prevents accidental activation of adjacent targets.

```css
/* Toolbar with spaced buttons */
.toolbar {
  display: flex;
  gap: 8px; /* Minimum 8px between targets */
}

/* For dense mobile layouts, increase gap */
@media (pointer: coarse) {
  .toolbar {
    gap: 12px;
  }
}
```

WCAG 2.2 uses the "circle overlap" rule: center a 24px-diameter circle on each target. If the circles do not overlap other targets or circles, the spacing passes even if individual targets are under 24px.

### Common Mistakes

```css
/* BAD: icon button with no tap area expansion */
.icon-btn {
  width: 16px;
  height: 16px;
  /* User has a 16x16 hit area -- nearly impossible to tap */
}

/* BAD: buttons crammed together */
.button-row {
  display: flex;
  gap: 2px; /* Too tight for touch */
}

/* BAD: relying on visual size only */
.close-btn {
  font-size: 12px;
  /* The "X" is tiny and the tap area matches the text bounding box */
}
```

## 2. The touch-action CSS Property

The `touch-action` property controls which touch gestures the browser handles natively on an element. When you set `touch-action`, you tell the browser: "I will handle these gestures myself (via JavaScript); you handle the rest."

### All Values

| Value | Browser Handles | You Handle |
|---|---|---|
| `auto` | All gestures (pan, zoom, double-tap) | Nothing |
| `none` | Nothing | All gestures |
| `pan-x` | Horizontal panning only | Vertical pan, zoom, etc. |
| `pan-y` | Vertical panning only | Horizontal pan, zoom, etc. |
| `pan-left` | Left panning only | Everything else |
| `pan-right` | Right panning only | Everything else |
| `pan-up` | Upward panning only | Everything else |
| `pan-down` | Downward panning only | Everything else |
| `pinch-zoom` | Pinch-to-zoom only | Everything else |
| `manipulation` | Pan + pinch-zoom | Double-tap zoom, other non-standard gestures |

`manipulation` is shorthand for `pan-x pan-y pinch-zoom`. It is the most commonly used value because it preserves essential gestures (scroll and zoom) while eliminating the double-tap-to-zoom delay.

### Combinable Values

Values can be combined:

```css
/* Allow vertical scroll and pinch zoom, prevent horizontal pan */
.vertical-only {
  touch-action: pan-y pinch-zoom;
}

/* Allow horizontal scroll only (no zoom, no vertical scroll) */
.horizontal-carousel {
  touch-action: pan-x;
}
```

### Use Cases

#### Image Carousel

```css
/* User swipes horizontally (your JS). Browser handles vertical scroll. */
.carousel {
  touch-action: pan-y pinch-zoom;
  overflow: hidden;
}
```

#### Custom Map or Canvas

```css
/* You handle ALL gestures (pan, zoom, rotate) */
.map-container {
  touch-action: none;
}
```

#### Kill the Double-Tap Delay Globally

```css
/* Remove double-tap zoom everywhere. Keep scroll and pinch zoom. */
html {
  touch-action: manipulation;
}
```

#### Swipeable List Item

```css
/* Horizontal swipe (your JS) on each item. Browser handles vertical scroll. */
.swipe-item {
  touch-action: pan-y;
}
```

#### Custom Pull-to-Refresh

```css
/* Prevent browser pull-to-refresh. Keep horizontal pan and pinch zoom. */
.pull-to-refresh-container {
  touch-action: pan-x pan-down pinch-zoom;
  /* pan-down means browser only handles downward pan when scroll > 0 */
}

/* When scroll is at top, only allow horizontal and downward panning */
.pull-to-refresh-container.at-top {
  touch-action: pan-x pan-down;
}
```

### Gotchas

1. **`touch-action: none` disables ALL native gestures** including scrolling. Users cannot scroll the element or its children. Use it only on self-contained interactive regions (maps, games, drawing canvases).

2. **Inheritance trap**: `touch-action` applies to the element and its descendants unless overridden. Setting `touch-action: none` on a parent locks all children.

3. **Must be set before the touch starts**: the browser reads `touch-action` at `touchstart`/`pointerdown` time. Changing it dynamically after the gesture begins has no effect on that gesture.

4. **Direction-specific values (`pan-left`, `pan-right`, etc.) have limited browser support.** Stick to `pan-x`, `pan-y`, `pinch-zoom`, `manipulation`, `none`, and `auto` for broad compatibility.

### Browser Support

`touch-action` is supported in all major browsers. The `manipulation` value is universally supported. Directional values (`pan-left`, `pan-right`, `pan-up`, `pan-down`) have more limited support (Chrome/Edge only as of 2025).

## 3. Interaction Media Features

CSS Level 4 Media Queries introduced four interaction media features that let you adapt layouts based on the user's input capabilities.

### The Four Features

| Feature | What It Tests | Values |
|---|---|---|
| `hover` | Can the **primary** input hover? | `none`, `hover` |
| `any-hover` | Can **any** available input hover? | `none`, `hover` |
| `pointer` | What accuracy does the **primary** input have? | `none`, `coarse`, `fine` |
| `any-pointer` | What accuracy does **any** available input have? | `none`, `coarse`, `fine` |

### Primary vs Any

- `hover` and `pointer` reflect the **primary** input device only.
- `any-hover` and `any-pointer` reflect **all connected** input devices.

Example: a tablet with a connected mouse.

| Feature | Value | Reason |
|---|---|---|
| `hover` | `none` | Primary input is touch (no hover) |
| `any-hover` | `hover` | The mouse can hover |
| `pointer` | `coarse` | Primary input is a finger |
| `any-pointer` | `fine` | The mouse is a fine pointer |

### Device Mapping

| Device/Scenario | `hover` | `pointer` | `any-hover` | `any-pointer` |
|---|---|---|---|---|
| Desktop with mouse | `hover` | `fine` | `hover` | `fine` |
| Phone (touch only) | `none` | `coarse` | `none` | `coarse` |
| Tablet with stylus | `none` | `fine` | `none` | `fine` |
| Laptop with touchscreen | `hover` | `fine` | `hover` | `coarse` (touch is `any`) |
| Smart TV remote | `hover` | `coarse` | `hover` | `coarse` |
| Game console controller | `hover` | `coarse` | `hover` | `coarse` |

### Practical Usage

#### Increase Target Size on Coarse Pointers

```css
.checkbox {
  width: 18px;
  height: 18px;
}

@media (pointer: coarse) {
  .checkbox {
    width: 24px;
    height: 24px;
  }
}
```

#### Show Tooltips Only on Hover-Capable Devices

```css
/* Default: no tooltip (touch devices) */
.tooltip-trigger .tooltip {
  display: none;
}

/* Only show tooltip on hover if the device supports it */
@media (hover: hover) {
  .tooltip-trigger:hover .tooltip {
    display: block;
  }
}
```

#### Adapt Navigation Based on Input

```css
/* Touch: larger nav items, no dropdowns on hover */
@media (pointer: coarse) {
  .nav-item {
    padding: 16px 20px;
    min-height: 48px;
  }
}

/* Mouse: smaller nav items, hover-triggered dropdowns */
@media (hover: hover) and (pointer: fine) {
  .nav-item {
    padding: 8px 16px;
  }

  .nav-item:hover > .dropdown {
    display: block;
  }
}
```

### Gotchas

1. **Windows laptops with touchscreens**: `pointer` reports `fine` (the trackpad/mouse is primary). Touch is detectable only via `any-pointer: coarse`. Your hover styles will apply, which is correct since the primary input does hover.

2. **iPads**: Safari reports `pointer: coarse` and `hover: none` since touch is primary. This is accurate for iPads even with Magic Keyboard attached (Safari does not update these values dynamically).

3. **Do not use these to "detect mobile"**: these features describe input capability, not screen size or device type. A large-screen kiosk with a touchscreen is `pointer: coarse` but is not "mobile."

4. **`any-pointer` and `any-hover` can have multiple true values simultaneously**: a device can have both `any-pointer: fine` and `any-pointer: coarse` (laptop with touchscreen). The media query `(any-pointer: coarse)` will match even though `pointer` is `fine`.

### Browser Support

All four features are supported in every modern browser (Chrome 41+, Firefox 64+, Safari 9+, Edge 12+). Universal support.

## 4. Handling Hover on Touch Devices

The core problem: when a touch user taps an element with a `:hover` style, the browser applies the hover style on the first tap and may leave it "stuck" until the user taps somewhere else. This is called **sticky hover**.

### The Problem in Detail

```css
/* This causes sticky hover on touch */
.card:hover {
  background-color: #e0e0e0;
  transform: translateY(-2px);
  box-shadow: 0 4px 8px rgba(0, 0, 0, 0.1);
}
```

On a touch device:
1. User taps the card.
2. Browser fires `touchstart`, then applies `:hover` styles.
3. The card elevates and changes background.
4. User lifts finger. The `:hover` style remains.
5. The card stays "hovered" until the user taps elsewhere.

### Solution 1: @media (hover: hover) Guard

The simplest and most reliable fix:

```css
/* Hover effect ONLY on devices that actually support hover */
@media (hover: hover) {
  .card:hover {
    background-color: #e0e0e0;
    transform: translateY(-2px);
    box-shadow: 0 4px 8px rgba(0, 0, 0, 0.1);
  }
}
```

Touch devices with `hover: none` never receive this rule. No sticky hover.

### Solution 2: Combine hover + pointer for Precision

```css
/* Only apply hover on devices with a fine pointer that supports hover */
@media (hover: hover) and (pointer: fine) {
  .card:hover {
    background-color: #e0e0e0;
  }
}
```

This excludes smart TV remotes and game controllers (which report `hover: hover` but `pointer: coarse`).

### Solution 3: Provide Touch-Specific Active States

Instead of relying on hover, give touch users `:active` feedback:

```css
/* Desktop: hover feedback */
@media (hover: hover) {
  .button:hover {
    background-color: var(--color-hover);
  }
}

/* All devices: active (press) feedback */
.button:active {
  background-color: var(--color-active);
  transform: scale(0.97);
}
```

### Solution 4: JavaScript Removal of Stuck Hover

For cases where you cannot restructure your CSS:

```js
// Detect touch and add a class to <html>
// Then scope hover rules under .has-hover
document.addEventListener('pointerdown', function onFirstTouch(e) {
  if (e.pointerType === 'touch') {
    document.documentElement.classList.remove('has-hover');
  } else {
    document.documentElement.classList.add('has-hover');
  }
});
```

```css
/* Hover only when JS confirms hover-capable input */
.has-hover .card:hover {
  background-color: #e0e0e0;
}
```

### What About Hover-Dependent Content?

If hover reveals content (tooltips, dropdown menus), you need an alternative for touch:

```css
/* Desktop: show dropdown on hover */
@media (hover: hover) and (pointer: fine) {
  .nav-item:hover > .dropdown {
    display: block;
  }
}

/* Touch: show dropdown on tap (toggle via JS or :focus-within) */
@media (hover: none) {
  .nav-item:focus-within > .dropdown {
    display: block;
  }
}
```

Or use a CSS-only approach with `:focus-within`:

```html
<nav>
  <button class="nav-toggle" aria-expanded="false">Menu</button>
  <ul class="dropdown">
    <li><a href="/about">About</a></li>
    <li><a href="/contact">Contact</a></li>
  </ul>
</nav>
```

```css
.dropdown {
  display: none;
}

/* Works on both touch (tap to focus) and keyboard (tab to focus) */
.nav-toggle:focus + .dropdown,
.nav-toggle + .dropdown:focus-within {
  display: block;
}

/* Additionally, hover for mouse users */
@media (hover: hover) {
  nav:hover .dropdown {
    display: block;
  }
}
```

### Common Mistakes

```css
/* BAD: hover as the only interaction feedback */
.button:hover {
  opacity: 0.8;
}
/* Touch users get no feedback until the hover sticks */

/* BAD: using hover to reveal essential content */
.tooltip { display: none; }
.trigger:hover .tooltip { display: block; }
/* Touch users cannot access the tooltip content reliably */

/* BAD: matching on max-width to "detect touch" */
@media (max-width: 768px) {
  /* "Removing" hover. But iPads are wider than 768px. */
}
```

## 5. Touch Events vs Pointer Events

### Touch Events API

Touch events (`touchstart`, `touchmove`, `touchend`, `touchcancel`) are the original API for handling multi-touch input on the web. They only fire for touch input.

```js
element.addEventListener('touchstart', (e) => {
  // e.touches -- all current touch points on the screen
  // e.targetTouches -- touches on this element
  // e.changedTouches -- touches that changed in this event
  const touch = e.changedTouches[0];
  console.log(touch.clientX, touch.clientY);
});
```

### Pointer Events API

Pointer events (`pointerdown`, `pointermove`, `pointerup`, `pointercancel`, `pointerenter`, `pointerleave`, `pointerover`, `pointerout`, `gotpointercapture`, `lostpointercapture`) unify mouse, touch, and pen into a single event model.

```js
element.addEventListener('pointerdown', (e) => {
  // e.pointerId -- unique per finger/pen/mouse
  // e.pointerType -- "mouse", "touch", or "pen"
  // e.isPrimary -- true for the first finger, the mouse, etc.
  // e.pressure -- 0 to 1 (touch force, pen pressure)
  // e.width, e.height -- contact geometry (touch)
  // e.tiltX, e.tiltY -- pen tilt
  console.log(e.pointerType, e.clientX, e.clientY);
});
```

### Comparison

| Aspect | Touch Events | Pointer Events |
|---|---|---|
| Input types | Touch only | Mouse, touch, pen |
| Multi-touch | `e.touches` array | One event per pointer (use `pointerId`) |
| Code duplication | Need separate mouse handlers | Single handler for all inputs |
| Pointer capture | Not available | `element.setPointerCapture(pointerId)` |
| Contact geometry | Not available | `e.width`, `e.height` |
| Pressure | Not standardized | `e.pressure` (0-1) |
| Cancel event | `touchcancel` | `pointercancel` |
| Browser support | All modern browsers | All modern browsers |
| Passive by default | No (must opt-in) | No (but better defaults in newer browsers) |

### Which to Use

**Use Pointer Events.** They are the modern standard, handle all input types with one code path, and provide richer data. The W3C Pointer Events specification is a Recommendation. There is no reason to use touch events for new code unless you need the multi-touch `touches` array pattern (which you can replicate with a Map of `pointerId` values).

### Pointer Events: Practical Multi-Touch

```js
const pointers = new Map();

element.addEventListener('pointerdown', (e) => {
  pointers.set(e.pointerId, { x: e.clientX, y: e.clientY });
  element.setPointerCapture(e.pointerId);
});

element.addEventListener('pointermove', (e) => {
  if (!pointers.has(e.pointerId)) return;
  pointers.set(e.pointerId, { x: e.clientX, y: e.clientY });
  // Use pointers.size to check if multi-touch
  // Use the Map entries for pinch distance calculation, etc.
});

element.addEventListener('pointerup', (e) => {
  pointers.delete(e.pointerId);
});

element.addEventListener('pointercancel', (e) => {
  pointers.delete(e.pointerId);
});
```

### Pointer Capture

Pointer capture lets you redirect all pointer events for a specific pointer to a target element, even if the pointer moves outside the element. This is essential for drag operations:

```js
const slider = document.querySelector('.slider-thumb');

slider.addEventListener('pointerdown', (e) => {
  // All subsequent pointermove/pointerup events go to this element
  slider.setPointerCapture(e.pointerId);
});

slider.addEventListener('pointermove', (e) => {
  if (!slider.hasPointerCapture(e.pointerId)) return;
  // Update slider position even if finger moves outside the thumb
  updateSliderPosition(e.clientX);
});

slider.addEventListener('pointerup', (e) => {
  // Capture is released automatically on pointerup
  // Or manually: slider.releasePointerCapture(e.pointerId);
});
```

### CSS Companion: pointer-events

The CSS `pointer-events` property (not to be confused with the JS API) controls whether an element receives pointer events at all:

```css
/* Disable all pointer interaction */
.disabled-overlay {
  pointer-events: none;
}

/* Re-enable on a child inside a disabled parent */
.disabled-overlay .still-interactive {
  pointer-events: auto;
}
```

### Gotcha: Event Order on Touch Devices

When a user taps on a touch device, events fire in this order:

1. `pointerdown`
2. `touchstart`
3. `pointermove` (if finger moved)
4. `touchmove` (if finger moved)
5. `pointerup`
6. `touchend`
7. `mouseover`
8. `mouseenter`
9. `mousemove`
10. `mousedown`
11. `mouseup`
12. `click`

The mouse compatibility events (7-12) fire after a ~300ms delay (unless eliminated). If you call `preventDefault()` on `pointerdown`, the touch events still fire. If you call `preventDefault()` on `touchstart`, the mouse compatibility events do NOT fire.

## 6. The 300ms Tap Delay

### History

When Apple introduced the iPhone in 2007, they needed a way to let users zoom into non-mobile-optimized pages. They implemented double-tap-to-zoom: tap once to follow a link, tap twice quickly to zoom. To distinguish single-tap from double-tap, the browser waits 300ms after the first tap to see if a second tap follows. If not, the `click` event fires.

This 300ms delay made every tap on every link and button feel sluggish.

### Modern Browser Behavior

The delay is now automatically removed in modern browsers when:

1. **The page has a responsive viewport meta tag:**
   ```html
   <meta name="viewport" content="width=device-width">
   ```
   Chrome 32+ (2014), Firefox 55+, Edge, and Safari (iOS 9.3+, 2016) all remove the delay when the viewport is set to device-width because double-tap-to-zoom is unnecessary on a properly scaled page.

2. **`touch-action: manipulation` is set:**
   ```css
   html {
     touch-action: manipulation;
   }
   ```
   This explicitly disables double-tap-to-zoom while preserving pan and pinch-zoom.

### Belt-and-Suspenders Approach

For guaranteed elimination of the delay:

```html
<!-- In <head> -->
<meta name="viewport" content="width=device-width, initial-scale=1.0">
```

```css
/* In global CSS */
html {
  touch-action: manipulation;
}
```

Both together cover all modern browsers and edge cases.

### Legacy Solutions (Historical Context)

These are no longer needed but you will encounter them in older codebases:

- **FastClick.js** (FT Labs): Listened for `touchend` and fired a synthetic `click` event immediately. Deprecated. Do not use in new projects.
- **`touch-action: manipulation`**: Still the recommended CSS approach.
- **300ms polyfills**: No longer necessary.

### Testing for the Delay

```js
// Measure tap-to-click delay
let tapTime = 0;

document.addEventListener('pointerdown', () => {
  tapTime = performance.now();
});

document.addEventListener('click', () => {
  const delay = performance.now() - tapTime;
  console.log(`Tap to click: ${delay.toFixed(1)}ms`);
  // Should be < 50ms on modern browsers with proper viewport
});
```

### Gotcha: Missing Viewport Meta Tag

If you forget the viewport meta tag, mobile browsers may still impose the 300ms delay because they assume the page needs double-tap-to-zoom for a desktop layout. Always include the viewport meta tag.

## 7. Gesture Support

Custom gestures (swipe, pinch, long press) require JavaScript. CSS alone cannot detect these gestures. The key is using Pointer Events (or Touch Events) to track finger positions across the gesture lifecycle.

### Swipe Detection

```js
function addSwipeListener(element, callback, options = {}) {
  const threshold = options.threshold || 50;   // Minimum travel distance (px)
  const restraint = options.restraint || 100;   // Max perpendicular travel (px)
  const timeLimit = options.timeLimit || 300;   // Max gesture time (ms)

  let startX, startY, startTime;

  element.addEventListener('pointerdown', (e) => {
    startX = e.clientX;
    startY = e.clientY;
    startTime = Date.now();
    element.setPointerCapture(e.pointerId);
  });

  element.addEventListener('pointerup', (e) => {
    const dx = e.clientX - startX;
    const dy = e.clientY - startY;
    const elapsed = Date.now() - startTime;

    if (elapsed > timeLimit) return; // Too slow

    if (Math.abs(dx) >= threshold && Math.abs(dy) <= restraint) {
      // Horizontal swipe
      callback(dx > 0 ? 'right' : 'left');
    } else if (Math.abs(dy) >= threshold && Math.abs(dx) <= restraint) {
      // Vertical swipe
      callback(dy > 0 ? 'down' : 'up');
    }
  });
}

// Usage
const container = document.querySelector('.swipeable');

// Let the browser handle vertical scroll; we handle horizontal swipe
container.style.touchAction = 'pan-y';

addSwipeListener(container, (direction) => {
  console.log('Swiped:', direction);
});
```

### Pinch-to-Zoom Detection

```js
function addPinchListener(element, callback) {
  const pointers = new Map();

  // Prevent browser zoom so we handle it
  element.style.touchAction = 'none';

  element.addEventListener('pointerdown', (e) => {
    pointers.set(e.pointerId, { x: e.clientX, y: e.clientY });
  });

  element.addEventListener('pointermove', (e) => {
    if (!pointers.has(e.pointerId)) return;
    pointers.set(e.pointerId, { x: e.clientX, y: e.clientY });

    if (pointers.size === 2) {
      const [p1, p2] = [...pointers.values()];
      const distance = Math.hypot(p2.x - p1.x, p2.y - p1.y);
      callback({ distance, center: {
        x: (p1.x + p2.x) / 2,
        y: (p1.y + p2.y) / 2,
      }});
    }
  });

  element.addEventListener('pointerup', (e) => pointers.delete(e.pointerId));
  element.addEventListener('pointercancel', (e) => pointers.delete(e.pointerId));
}

// Usage: track scale factor
let initialDistance = null;
let currentScale = 1;

addPinchListener(document.querySelector('.zoomable'), ({ distance }) => {
  if (initialDistance === null) {
    initialDistance = distance;
    return;
  }
  currentScale = distance / initialDistance;
  document.querySelector('.zoomable').style.transform = `scale(${currentScale})`;
});
```

### Long Press Detection

```js
function addLongPressListener(element, callback, duration = 500) {
  let timerId = null;
  let startX, startY;
  const moveThreshold = 10; // Cancel if finger moves more than 10px

  element.addEventListener('pointerdown', (e) => {
    startX = e.clientX;
    startY = e.clientY;

    timerId = setTimeout(() => {
      callback(e);
      timerId = null;
    }, duration);
  });

  element.addEventListener('pointermove', (e) => {
    if (timerId === null) return;
    const moved = Math.hypot(e.clientX - startX, e.clientY - startY);
    if (moved > moveThreshold) {
      clearTimeout(timerId);
      timerId = null;
    }
  });

  element.addEventListener('pointerup', () => {
    if (timerId !== null) {
      clearTimeout(timerId);
      timerId = null;
    }
  });

  element.addEventListener('pointercancel', () => {
    if (timerId !== null) {
      clearTimeout(timerId);
      timerId = null;
    }
  });

  // Prevent context menu on long press (optional)
  element.addEventListener('contextmenu', (e) => e.preventDefault());
}

// Usage
addLongPressListener(document.querySelector('.list-item'), (e) => {
  console.log('Long pressed at', e.clientX, e.clientY);
  showContextMenu(e.clientX, e.clientY);
});
```

### Gesture Libraries

For production applications with complex gesture needs, consider a library:

| Library | Size | Gestures | Notes |
|---|---|---|---|
| Hammer.js | ~7KB gzip | Tap, pan, swipe, pinch, rotate, press | Widely used, not actively maintained |
| ZingTouch | ~3KB gzip | Tap, swipe, pan, pinch, rotate, expand | Modern, customizable |
| interact.js | ~10KB gzip | Drag, resize, gesture, inertia | Focuses on drag-and-drop |
| use-gesture (React) | ~5KB gzip | All common gestures | React-specific hooks |

### Gotcha: touch-action Coordination

Your `touch-action` CSS must match your gesture handling. If you handle horizontal swipe but `touch-action` is `auto`, the browser may intercept the horizontal pan before your JavaScript sees it. Set `touch-action: pan-y` (let browser handle vertical, you handle horizontal).

## 8. Touch-Friendly Navigation Patterns

### Thumb Zone Awareness

About 75% of interactions on phones happen with the thumb. The "thumb zone" model divides the screen into three regions:

- **Easy reach** (bottom third): Primary actions, main navigation
- **Stretch** (middle third): Secondary actions
- **Hard to reach** (top third): Infrequent actions, system UI

This is why bottom navigation bars became the dominant mobile pattern.

### Navigation Best Practices for Touch

1. **Limit bottom tab items to 3-5.** More than 5 makes targets too narrow.
2. **Use icons + labels.** Icons alone are ambiguous.
3. **Highlight the active item clearly.** Color, weight, or an indicator bar.
4. **Avoid hover-triggered dropdowns.** Use tap-to-toggle or dedicated pages.
5. **Support swipe-to-navigate** between major sections when appropriate (like tabs).
6. **Place destructive actions away from common tap zones** to prevent accidental triggering.

## 9. Touch-Friendly Forms

Forms are where touch design most often fails. Small inputs, wrong keyboard types, and awkward select elements create friction.

### Input Sizing

```css
/* Base form styles for touch */
input[type="text"],
input[type="email"],
input[type="password"],
input[type="number"],
input[type="tel"],
input[type="url"],
input[type="search"],
select,
textarea {
  /* Minimum height for comfortable tapping */
  min-height: 48px;
  padding: 12px 16px;

  /* Prevent iOS zoom on focus (requires >= 16px) */
  font-size: 16px;

  /* Full width on mobile for easy targeting */
  width: 100%;
  box-sizing: border-box;

  /* Clear borders for visibility */
  border: 1px solid #ccc;
  border-radius: 8px;

  /* Remove default appearance for consistent sizing */
  -webkit-appearance: none;
  appearance: none;
}

/* Labels above inputs (mandatory on mobile) */
label {
  display: block;
  margin-bottom: 6px;
  font-size: 14px;
  font-weight: 500;
}

/* Spacing between form groups */
.form-group {
  margin-bottom: 20px;
}
```

### The iOS Zoom Gotcha

iOS Safari zooms into the page when a user focuses an input with a font size smaller than 16px. This is a usability disaster because the user must manually zoom back out.

```css
/* BAD: triggers zoom on iOS */
input {
  font-size: 14px;
}

/* GOOD: no zoom */
input {
  font-size: 16px;
}

/* Alternative: prevent zoom entirely (not recommended, breaks accessibility) */
/* <meta name="viewport" content="..., maximum-scale=1"> */
/* This disables pinch-to-zoom which harms accessibility. Avoid. */
```

### Input Types and Virtual Keyboards

The `type` attribute triggers the appropriate virtual keyboard. Using the right type reduces errors and input time.

| Input Type | Keyboard Shown | Use For |
|---|---|---|
| `type="text"` | Standard keyboard | General text |
| `type="email"` | Keyboard with @ and .com | Email addresses |
| `type="tel"` | Numeric keypad | Phone numbers |
| `type="number"` | Numeric keyboard with +/- | Quantities, amounts |
| `type="url"` | Keyboard with / and .com | URLs |
| `type="search"` | Keyboard with Search button | Search fields |
| `type="password"` | Standard with show/hide | Passwords |

### The `inputmode` Attribute

When `type` does not match the desired keyboard (e.g., you want a numeric keypad but `type="text"` for formatting reasons), use `inputmode`:

```html
<!-- Numeric keyboard for a credit card input that needs text type for formatting -->
<input type="text" inputmode="numeric" pattern="[0-9]*"
       placeholder="1234 5678 9012 3456"
       autocomplete="cc-number">

<!-- Decimal keyboard for currency -->
<input type="text" inputmode="decimal" placeholder="0.00">

<!-- No autocorrect/autocapitalize for usernames -->
<input type="text" inputmode="email"
       autocorrect="off" autocapitalize="none"
       placeholder="username">
```

| `inputmode` Value | Keyboard |
|---|---|
| `none` | No keyboard (for custom input) |
| `text` | Standard text |
| `decimal` | Numeric with decimal point |
| `numeric` | Numeric only (0-9) |
| `tel` | Telephone keypad |
| `search` | Optimized for search |
| `email` | Email-optimized |
| `url` | URL-optimized |

### Touch-Friendly Checkboxes and Radios

Default checkboxes and radios are tiny. Replace them:

```css

  align-items: center;
  gap: 12px;

  padding: 8px 0;

}

  position: absolute;
  opacity: 0;
  width: 0;
  height: 0;

/* Visual replacement */
.custom-checkbox .check-mark {
  width: 24px;
  height: 24px;
  border: 2px solid #666;
  border-radius: 4px;
  flex-shrink: 0;
  display: flex;
  align-items: center;
  justify-content: center;
  transition: background-color 0.15s, border-color 0.15s;
}

/* Checked state */
.custom-checkbox input:checked + .check-mark {
  background-color: #1976d2;
  border-color: #1976d2;
}

/* Focus visible for keyboard users */
.custom-checkbox input:focus-visible + .check-mark {
  outline: 2px solid #1976d2;
  outline-offset: 2px;
}

/* The label text itself is part of the tap target */
.custom-checkbox span {
  font-size: 16px;
  user-select: none;
}
```

Native `<select>` elements trigger the OS-native picker on mobile, which is usually the best experience. Do not replace `<select>` with custom dropdowns on mobile unless you have a very good reason.

### Form Layout


## 10. Scroll Behavior on Touch

Touch scrolling has behaviors and properties that do not exist in mouse-driven scrolling.

### Momentum (Inertial) Scrolling

When a user flicks their finger and lifts, the content continues to scroll with decelerating momentum. This is handled natively by the browser. On modern iOS Safari (13+), momentum scrolling works automatically on all overflow elements without any special CSS.

**Historical note**: older iOS Safari required `-webkit-overflow-scrolling: touch` on `overflow: scroll` elements. This property is now deprecated and no longer necessary.

```css
/* Modern: momentum scrolling works automatically */
.scrollable {
  overflow-y: auto;
  /* No special property needed */
}

/* Legacy (still appears in old codebases, harmless but unnecessary) */
.scrollable-legacy {
  overflow-y: auto;
  -webkit-overflow-scrolling: touch; /* Deprecated, ignored on modern Safari */
}
```

### The overscroll-behavior Property

`overscroll-behavior` controls what happens when scrolling reaches the boundary of a scrollable element.

| Value | Effect |
|---|---|
| `auto` | Default. Scroll chains to parent. Browser overscroll effects (bounce, glow) active. |
| `contain` | No scroll chaining to parent. Local overscroll effects still shown (iOS bounce, Android glow). |
| `none` | No scroll chaining AND no local overscroll effects. |

Axis-specific variants: `overscroll-behavior-x`, `overscroll-behavior-y`.

#### Prevent Scroll Chaining in a Modal

```css
/* Without this, scrolling to the end of modal content
   causes the page behind it to scroll */
.modal-body {
  overflow-y: auto;
  max-height: 80vh;
  overscroll-behavior-y: contain;
}
```

#### Disable Pull-to-Refresh

```css
/* Prevent browser's native pull-to-refresh on the whole page */
body {
  overscroll-behavior-y: contain;
}
```

#### Disable All Overscroll Effects

```css
/* No bounce, no glow, no scroll chaining */
.fixed-scroll-area {
  overflow-y: auto;
  overscroll-behavior: none;
}
```

#### Chat Interface Pattern

```css
.chat-messages {
  overflow-y: auto;
  height: 100%;
  /* Prevent scrolling past the top from pulling down the page */
  overscroll-behavior-y: contain;

  /* Scroll new messages into view */
  overflow-anchor: auto;
}
```

### Scroll Snap for Touch

Scroll snap creates "magnetic" stop points that content snaps to, ideal for carousels and paginated layouts on touch:

```css
.snap-container {
  overflow-x: auto;
  scroll-snap-type: x mandatory;
  /* mandatory: always snaps. proximity: snaps only when close to a point. */

  display: flex;
  gap: 16px;
  padding: 16px;

  /* Hide scrollbar for cleaner touch UI */
  scrollbar-width: none;
}

.snap-container::-webkit-scrollbar {
  display: none;
}

.snap-item {
  scroll-snap-align: start;
  /* start: left edge aligns. center: center aligns. end: right edge aligns. */

  flex: 0 0 85%;  /* Each item takes 85% of container width */
  border-radius: 12px;
}
```

### Gotcha: Scroll Locking on iOS

Preventing body scroll while a modal is open is notoriously difficult on iOS:

```css
/* Step 1: When modal opens, lock body scroll */
body.modal-open {
  overflow: hidden;
  position: fixed;
  width: 100%;
  /* Preserve scroll position */
  top: calc(var(--scroll-y) * -1);
}
```

```js
// Step 2: Track and restore scroll position
function openModal() {
  const scrollY = window.scrollY;
  document.documentElement.style.setProperty('--scroll-y', scrollY + 'px');
  document.body.classList.add('modal-open');
}

function closeModal() {
  document.body.classList.remove('modal-open');
  const scrollY = document.documentElement.style.getPropertyValue('--scroll-y');
  window.scrollTo(0, parseInt(scrollY || '0'));
}
```

### Browser Support

- `overscroll-behavior`: All modern browsers (Chrome 63+, Firefox 59+, Safari 16+, Edge 18+).
- `scroll-snap-type`: All modern browsers (wide support since 2019+).

## 11. Touch Feedback

Touch users need immediate visual confirmation that their tap was registered. Without feedback, users tap again, causing double submissions, unintended navigation, and frustration.

### CSS :active State

The `:active` pseudo-class fires on press. It is the primary mechanism for tap feedback.

```css
.touch-button {
  background: #1976d2;
  color: white;
  padding: 14px 28px;
  border: none;
  border-radius: 8px;
  font-size: 16px;
  /* Transition for smooth feedback */
  transition: background-color 0.1s, transform 0.1s;
}

.touch-button:active {
  background: #1565c0;
  transform: scale(0.97);
}
```

**Gotcha on iOS**: Safari requires a `touchstart` event listener (even an empty one) on the element or a parent for `:active` to fire on touch. Modern workaround:

```css
/* Force :active to work on iOS Safari */
body {
  -webkit-tap-highlight-color: transparent;
}
```

```html
<!-- Or add an empty touchstart listener in the <body> tag -->
<body ontouchstart="">
```

### Tap Highlight Color

Browsers show a default tap highlight on touch. Customize or remove it:

```css
/* Remove default tap highlight (replace with your own :active style) */
* {
  -webkit-tap-highlight-color: transparent;
}

/* Or customize it */
.tappable {
  -webkit-tap-highlight-color: rgba(25, 118, 210, 0.2);
}
```

### CSS-Only Ripple Effect

A Material Design-style ripple using only CSS:

```css
  transition: width 0.4s ease, height 0.4s ease, opacity 0.4s ease;
  opacity: 0;
}

.ripple-button:active::after {
  width: 200%;
  height: 200%;
  opacity: 1;
  /* Reset transition on active for instant start */

}
```

### JavaScript Ripple (Position-Accurate)

For a ripple that starts from the exact tap point:

```js
function createRipple(event) {
  const button = event.currentTarget;
  const rect = button.getBoundingClientRect();

  const ripple = document.createElement('span');
  const diameter = Math.max(rect.width, rect.height) * 2;

  ripple.style.width = ripple.style.height = `${diameter}px`;
  ripple.style.left = `${event.clientX - rect.left - diameter / 2}px`;
  ripple.style.top = `${event.clientY - rect.top - diameter / 2}px`;
  ripple.className = 'ripple-effect';

  // Remove any existing ripple
  const existing = button.querySelector('.ripple-effect');
  if (existing) existing.remove();

  button.appendChild(ripple);

  // Clean up after animation
  ripple.addEventListener('animationend', () => ripple.remove());
}

document.querySelectorAll('.js-ripple').forEach((btn) => {
  btn.addEventListener('pointerdown', createRipple);
});
```

```css
  position: relative;
  overflow: hidden;
}

.ripple-effect {
  position: absolute;
  border-radius: 50%;
  background: rgba(255, 255, 255, 0.4);
  transform: scale(0);
  animation: ripple-expand 0.5s ease-out forwards;
  pointer-events: none;
}

  to {
    transform: scale(1);
    opacity: 0;
  }
}
```

### Vibration API (Haptic-Like Feedback)

The Vibration API provides tactile feedback on supported devices:

```js
// Simple vibration on tap
function hapticFeedback(duration = 10) {
  if ('vibrate' in navigator) {
    navigator.vibrate(duration);
  }
}

// Pattern vibration
function hapticPattern() {
  if ('vibrate' in navigator) {
    // vibrate 10ms, pause 50ms, vibrate 10ms
    navigator.vibrate([10, 50, 10]);
  }
}

// Use on critical actions
document.querySelector('.delete-btn').addEventListener('pointerdown', () => {
  hapticFeedback(15);
});
```

**Limitation**: The Vibration API is not supported on iOS Safari. It works on Android Chrome and some other Android browsers.

## 12. Safe Areas and Notch Handling

Modern phones have notches, rounded corners, dynamic islands, and home indicator bars that obscure content. CSS environment variables let you adapt.

### Enabling Full-Screen Layout

By default, the browser insets your content to avoid the notch. To extend to the edges (and handle safe areas yourself):

```html
<meta name="viewport"
      content="width=device-width, initial-scale=1.0, viewport-fit=cover">
```

Without `viewport-fit=cover`, the `env()` values are always `0`.

### Safe Area Environment Variables

```css
/* The four safe-area-inset variables */
.container {
  padding-top: env(safe-area-inset-top);
  padding-right: env(safe-area-inset-right);
  padding-bottom: env(safe-area-inset-bottom);
  padding-left: env(safe-area-inset-left);
}
```

| Variable | Where It Applies |
|---|---|
| `safe-area-inset-top` | Notch, status bar, dynamic island |
| `safe-area-inset-bottom` | Home indicator bar (iPhone), gesture bar |
| `safe-area-inset-left` | Rounded corners in landscape, side notch |
| `safe-area-inset-right` | Rounded corners in landscape, side notch |

### Practical Patterns

#### Fixed Bottom Bar with Safe Area

```css
.bottom-bar {
  position: fixed;
  bottom: 0;
  left: 0;
  right: 0;
  background: #fff;
  display: flex;
  align-items: center;
  /* Content padding + safe area */
  padding: 12px 16px calc(12px + env(safe-area-inset-bottom)) 16px;
  border-top: 1px solid #e0e0e0;
}
```

#### Full-Bleed Header with Safe Area

```css
.header {
  position: fixed;
  top: 0;
  left: 0;
  right: 0;
  /* Background extends behind the notch */
  background: #1976d2;
  color: white;
  /* Content stays below the notch */
  padding: calc(12px + env(safe-area-inset-top)) 16px 12px;
}
```

#### Using Fallback Values

`env()` accepts a second parameter as a fallback:

```css
.container {
  /* If safe-area-inset-bottom is not supported, use 0px */
  padding-bottom: env(safe-area-inset-bottom, 0px);
}

/* Combine with max() for minimum padding */
.bottom-bar {
  padding-bottom: max(12px, env(safe-area-inset-bottom));
}
```

#### Landscape Handling

In landscape on notched phones, the insets shift to the sides:

```css
.full-width-content {
  /* Safe padding on all sides adapts to orientation */
  padding-left: max(16px, env(safe-area-inset-left));
  padding-right: max(16px, env(safe-area-inset-right));
}
```

### Browser Support

`env()` and safe-area-inset variables are supported in all modern browsers (Safari 11.1+, Chrome 69+, Firefox 65+, Edge 79+). They were originally introduced by Apple as `constant()` (iOS 11.0-11.1) then standardized as `env()`.

## 13. Accessibility for Touch

Touch interfaces introduce unique accessibility challenges. Screen reader users on touch devices use completely different gesture models.

### VoiceOver (iOS) Gesture Model

When VoiceOver is active, the standard touch interactions change:

| Gesture | Action |
|---|---|
| Single tap anywhere | VoiceOver announces the element under finger |
| Double tap | Activates the focused element (equivalent to click) |
| Swipe right | Move to next element |
| Swipe left | Move to previous element |
| Three-finger swipe | Scroll the page |
| Two-finger tap | Pause/resume VoiceOver speech |
| Two-finger scrub (Z shape) | Go back / dismiss |

### TalkBack (Android) Gesture Model

| Gesture | Action |
|---|---|
| Tap | Announce element under finger |
| Double tap | Activate focused element |
| Swipe right | Next element |
| Swipe left | Previous element |
| Two-finger scroll | Scroll content |
| Three-finger tap | TalkBack menu |

### Key Implications for Developers

#### 1. Custom Gestures Must Have Alternatives

If your UI relies on swipe, pinch, or long press, provide button alternatives. Screen reader users cannot perform custom gestures because the screen reader intercepts them.

```html
<!-- BAD: swipe-only delete -->
<div class="swipe-to-delete">
  <p>Item content</p>
</div>

<!-- GOOD: swipe AND button -->
<div class="swipe-to-delete">
  <p>Item content</p>
  <button class="delete-btn" aria-label="Delete item">
    <svg aria-hidden="true"><!-- trash icon --></svg>
  </button>
</div>
```

#### 2. Minimum Target Size Matters Even More

Screen reader users use "explore by touch" (dragging a finger to discover elements). Small targets are harder to find and activate.

#### 3. Focus Order Must Be Logical

VoiceOver and TalkBack traverse the DOM in source order (unless overridden by `tabindex` or ARIA). Ensure the visual order matches the DOM order.

#### 4. Avoid Relying on Hover for Information

Screen reader users on touch devices have no hover state. Any content revealed only on hover is inaccessible.

```css
/* Accessible: visible by default, enhanced on hover */
.info-text {
  /* Always visible */
}

@media (hover: hover) {
  .info-text {
    /* Optionally enhanced on hover */
  }
}
```

#### 5. Touch-Specific ARIA Patterns

```html
<!-- Carousel: announce position -->
<div role="region" aria-label="Image carousel" aria-roledescription="carousel">
  <div role="group" aria-roledescription="slide" aria-label="1 of 5">
    <!-- slide content -->
  </div>
</div>

<!-- Swipe action with description -->
<div role="listitem" aria-label="Inbox message from Alice"
     aria-description="Swipe left to delete, swipe right to archive">
  <!-- message content -->
  <!-- Also provide buttons as alternatives -->
</div>
```

#### 6. Announce Dynamic Changes

```html
<!-- Live region for action confirmations -->
<div aria-live="polite" aria-atomic="true" class="visually-hidden" id="announcer">
</div>
```

```js
// When a swipe-to-delete completes
function announceAction(message) {
  document.getElementById('announcer').textContent = message;
}

// After deleting an item
announceAction('Message deleted. Undo available for 5 seconds.');
```

## 14. Performance on Touch Devices

Touch devices are typically less powerful than desktops. Scrolling smoothness, animation frame rates, and event handler efficiency directly impact the perceived quality of touch interactions.

### Passive Event Listeners

When you add a `touchstart`, `touchmove`, or `wheel` event listener, the browser does not know whether your handler will call `preventDefault()`. It must wait for your handler to finish before it can start scrolling. This blocks the compositor thread and causes scroll jank.

Passive listeners solve this:

```js
// BAD: blocks scrolling until handler completes
element.addEventListener('touchmove', handleMove);

// GOOD: tells browser "I will not call preventDefault()"
element.addEventListener('touchmove', handleMove, { passive: true });
```

When you mark a listener as `passive: true`, the browser scrolls immediately on the compositor thread without waiting for your JavaScript.

**When you CANNOT use passive**: if your handler needs to `preventDefault()` (e.g., custom gesture handling), you cannot use passive. In this case, set `touch-action` in CSS instead to tell the browser which gestures you will handle.

```css
/* Instead of calling preventDefault() in touchmove: */
.custom-drag-area {
  touch-action: none; /* Browser knows not to scroll here */
}
```

```js
// Now this listener does not need to call preventDefault()
// so it can be passive
element.addEventListener('pointermove', handleDrag, { passive: true });
```

### Chrome's Default Passive Behavior

Since Chrome 56, `touchstart` and `touchmove` listeners on `document`, `document.body`, and `window` are passive by default. If you call `preventDefault()` in these listeners, Chrome logs a warning and ignores it.

```js
// This will NOT prevent scrolling in Chrome 56+
document.addEventListener('touchmove', (e) => {
  e.preventDefault(); // Ignored -- listener is passive by default
});

// To override, explicitly set passive: false
document.addEventListener('touchmove', (e) => {
  e.preventDefault(); // Now works
}, { passive: false });
```

### GPU Acceleration with will-change

The `will-change` property promotes an element to its own compositor layer, enabling transforms and opacity changes without triggering layout or paint on the main thread.

```css
/* Promote an element that will be animated */
.animated-panel {
  will-change: transform;
}

/* Remove will-change after animation completes to free GPU memory */
.animated-panel.idle {
  will-change: auto;
}
```

#### Rules for will-change

1. **Do not apply to everything.** Each promoted layer consumes GPU memory. Overuse degrades performance.
2. **Apply before the animation starts.** The browser needs time to set up the layer.
3. **Remove when done.** Set `will-change: auto` after the animation completes.
4. **Do not use as a "performance hack."** Only use when you have a measurable jank problem.

```css
/* BAD: promoting everything */
* {
  will-change: transform;
}

/* BAD: permanent will-change on static elements */
.card {
  will-change: transform, opacity;
}

/* GOOD: apply only during interaction */
.card:active {
  will-change: transform;
  transform: scale(0.97);
}
```

### Compositor-Friendly Properties

Only `transform` and `opacity` can be animated entirely on the compositor thread (no layout, no paint). All other properties trigger main-thread work.

| Property | Layout | Paint | Composite |
|---|---|---|---|
| `transform` | No | No | Yes |
| `opacity` | No | No | Yes |
| `background-color` | No | Yes | Yes |
| `width`, `height` | Yes | Yes | Yes |
| `top`, `left` | Yes | Yes | Yes |
| `box-shadow` | No | Yes | Yes |

For smooth touch interactions, always animate `transform` and `opacity`:

```css
/* BAD: animating top/left causes layout thrashing */
.draggable {
  position: absolute;
  transition: top 0.1s, left 0.1s;
}

/* GOOD: animating transform is compositor-only */
.draggable {
  transition: transform 0.1s;
}
```

### Debouncing and Throttling Touch Handlers

`pointermove` and `touchmove` fire at the display refresh rate (60-120+ times per second). Expensive work in these handlers causes jank.

```js
// Throttle with requestAnimationFrame
let ticking = false;

element.addEventListener('pointermove', (e) => {
  if (ticking) return;
  ticking = true;

  requestAnimationFrame(() => {
    // Expensive DOM updates go here
    updatePosition(e.clientX, e.clientY);
    ticking = false;
  });
}, { passive: true });
```

### Reducing Paint Area

On touch devices, large paint areas during scroll cause dropped frames. Minimize the paint area:

```css
/* Promote fixed/sticky elements to their own layer */
.sticky-header {
  position: sticky;
  top: 0;
  z-index: 100;
  /* Own layer: scrolling the page does not repaint the header */
  will-change: transform;
  /* Or use the older hack: */
  /* transform: translateZ(0); */
}
```

### Image and Asset Optimization

Touch devices often have high-DPI screens but slower connections:

```html
<!-- Serve appropriately sized images -->
<img src="photo-400.jpg"
     srcset="photo-400.jpg 400w,
             photo-800.jpg 800w,
             photo-1200.jpg 1200w"
     sizes="(max-width: 600px) 100vw, 50vw"
     loading="lazy"
     decoding="async"
     alt="Description">
```

```css
/* Contain images to prevent layout shifts on slow loads */
.image-container {
  aspect-ratio: 16 / 9;
  overflow: hidden;
}

.image-container img {
  width: 100%;
  height: 100%;
  object-fit: cover;
}
```

### Testing Performance

Use Chrome DevTools on a real device via USB debugging, or use the Performance panel with CPU throttling (4x/6x slowdown) and Network throttling (Slow 3G) to simulate mobile conditions.

Key metrics to watch:
- **FPS during scroll**: should stay at 60fps (16.6ms per frame)
- **Total Blocking Time (TBT)**: long tasks that delay interaction
- **Interaction to Next Paint (INP)**: time from tap to visual response

## Summary: Touch-Friendly Design Checklist

| Area | Requirement |
|---|---|
| Tap targets | Minimum 44x44 CSS px (48x48 preferred). WCAG 2.2: 24x24 minimum. |
| Target spacing | 8px minimum gap. WCAG 2.2: 24px circle non-overlap. |
| touch-action | Set `manipulation` globally. Use specific values for custom gestures. |
| Hover handling | Guard `:hover` with `@media (hover: hover)`. Provide `:active` feedback. |
| Event handling | Use Pointer Events. Mark listeners as `passive: true` when possible. |
| 300ms delay | Viewport meta tag + `touch-action: manipulation`. |
| Forms | 48px min-height inputs. 16px+ font size. Correct `type`/`inputmode`. |
| Scroll | Use `overscroll-behavior: contain` for modals. `scroll-snap` for carousels. |
| Feedback | Always provide `:active` state. Consider ripple effects. |
| Safe areas | Use `viewport-fit=cover` + `env(safe-area-inset-*)` for notched devices. |
| Accessibility | Provide button alternatives for all gesture-based actions. |
| Performance | Passive listeners. Animate only `transform`/`opacity`. Throttle move handlers. |
