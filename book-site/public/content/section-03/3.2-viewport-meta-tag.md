# 3.2 Viewport Meta Tag

> The viewport meta tag tells mobile browsers how to size and scale the layout viewport. Without it, your responsive CSS is ignored and the page renders at a desktop-assumed 980px width, shrunk to fit a 375px screen.

---

## Why It Exists

In 2007, Apple shipped the first iPhone with Safari. The web was designed for 1000px desktop monitors. Apple's solution: assume every page is 980px wide, render it at that width, and shrink the result to fit the 320px screen. Users pinch-to-zoom to read.

Apple then invented the `<meta name="viewport">` tag to let developers opt out of this 980px assumption. It was never a formal standard. Android and other mobile browsers adopted it, and it became the universal mechanism for controlling the mobile viewport.

The 980px default persists in every mobile browser today. Without the viewport meta tag, your page is rendered at 980px and zoomed out. Text is tiny, tap targets are impossible, and media queries never trigger.

---

## The Recommended Tag

```html
<meta name="viewport" content="width=device-width, initial-scale=1">
```

This is the only viewport meta tag most websites need. It does two things:

1. **`width=device-width`** sets the layout viewport width to the device's CSS pixel width (393px on iPhone 15, 412px on Pixel 8) instead of the 980px default.
2. **`initial-scale=1`** sets the initial zoom to 100%, preventing the browser from zooming out to fit assumed desktop content.

---

## All Properties

| Property | Values | Default | Description |
|----------|--------|---------|-------------|
| `width` | `1` to `10000` (px) or `device-width` | ~980px on mobile | Layout viewport width |
| `height` | `1` to `10000` (px) or `device-height` | Calculated from width | Layout viewport height. Rarely used; broken in many browsers. |
| `initial-scale` | `0.1` to `10.0` | `1` | Zoom level on first page load |
| `minimum-scale` | `0.1` to `10.0` | `0.1` | Minimum zoom (how far the user can zoom out) |
| `maximum-scale` | `0.1` to `10.0` | `10.0` | Maximum zoom (how far the user can zoom in) |
| `user-scalable` | `yes` or `no` | `yes` | Whether pinch-to-zoom is allowed |
| `viewport-fit` | `auto`, `contain`, `cover` | `auto` | How the viewport handles non-rectangular displays (notches) |
| `interactive-widget` | `resizes-visual`, `resizes-content`, `overlays-content` | `resizes-visual` | How virtual keyboards affect the viewport |

### Parser Behavior

Browsers accept commas, semicolons, or spaces as separators and `=` or `:` between keys and values. Unrecognized keys are silently ignored. The standard format uses commas and `=`.

### Interaction Between width and initial-scale

When both are set, the browser takes the larger of the two computed values. In portrait, `width=device-width` and `initial-scale=1` typically resolve to the same value. In landscape, `initial-scale=1` may produce a wider value, and that one wins.

---

## Layout Viewport vs. Visual Viewport

Mobile browsers have two viewports, not one. Understanding the difference is critical for debugging layout issues.

| Property | Layout Viewport | Visual Viewport |
|----------|----------------|-----------------|
| What it defines | The canvas CSS is rendered against | The area the user currently sees |
| CSS uses it for | `width: 100%`, `vh`/`vw`, media queries | Nothing directly |
| Changes on zoom? | No | Yes (shrinks when zoomed in) |
| Changes with keyboard? | Depends on `interactive-widget` | Yes |
| JavaScript API | `document.documentElement.clientWidth` | `window.visualViewport.width` |

When a user pinch-zooms to 200%, the layout viewport stays the same size (your CSS layout does not reflow), but the visual viewport shrinks to half (they see less of the page). The visual viewport "pans" over the layout viewport.

### The Visual Viewport API

```js
const vv = window.visualViewport;

console.log(vv.width);       // Visual viewport width in CSS pixels
console.log(vv.height);      // Visual viewport height in CSS pixels
console.log(vv.offsetLeft);  // Horizontal offset from layout viewport
console.log(vv.offsetTop);   // Vertical offset from layout viewport
console.log(vv.scale);       // Current zoom level (1.0 = no zoom)

// Fires when the visual viewport resizes (keyboard, zoom)
vv.addEventListener('resize', () => {
  console.log('Resized:', vv.width, vv.height);
});
```

## viewport-fit and Safe Area Insets

Modern phones have notches, rounded corners, and home indicators that intrude into the display rectangle. The `viewport-fit` property controls whether the page extends behind these areas.

### viewport-fit Values

| Value | Behavior |
|-------|----------|
| `auto` / `contain` | Content stays within the safe rectangle. Areas behind the notch are left empty. This is the default. |
| `cover` | Content extends to fill the entire display, including behind the notch. You must handle safe areas manually. |

### Enabling Cover Mode

```html
<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">
```

Only use `viewport-fit=cover` if you are actively handling safe areas in CSS. Without safe area padding, content will be hidden behind the notch.

### The env() Safe Area Variables

```css
env(safe-area-inset-top)     /* Notch / status bar area */
env(safe-area-inset-right)   /* Right edge rounding */
env(safe-area-inset-bottom)  /* Home indicator area on iPhone */
env(safe-area-inset-left)    /* Left edge rounding */

/* Each accepts a fallback for devices without insets */
env(safe-area-inset-top, 0px)
```

On devices without notches, these values are `0`. They only produce non-zero values when `viewport-fit=cover` is set and the device has intrusions.

### Complete PWA Example

```css
/* The page extends behind the notch */
html {
  min-height: calc(100% + env(safe-area-inset-top));
  padding:
    env(safe-area-inset-top)
    env(safe-area-inset-right)
    env(safe-area-inset-bottom)
    env(safe-area-inset-left);
}

/* Fixed bottom bar clears the home indicator */
.bottom-nav {
  position: fixed;
  bottom: 0;
  left: 0;
  right: 0;
  height: calc(60px + env(safe-area-inset-bottom));
  padding-bottom: env(safe-area-inset-bottom);
  background: #1a1a1a;
}

/* Sticky header clears the notch */
.header {
  position: sticky;
  top: 0;
  padding-top: env(safe-area-inset-top);
  padding-left: env(safe-area-inset-left);
  padding-right: env(safe-area-inset-right);
}
```

## The 100vh Mobile Bug and New Viewport Units

### The Problem

On mobile browsers, the address bar and bottom navigation appear and disappear as the user scrolls. `100vh` equals the Large Viewport Height (the height when browser chrome is fully hidden). When the page loads with the address bar visible, `100vh` is taller than the visible area, causing overflow.

```
+--------------------+
| Address Bar (56px) |   <-- Visible on page load
+--------------------+
|                    |
|   Visible Area     |   <-- 700px (what the user sees)
|                    |
+--------------------+
| Bottom Nav (48px)  |
+--------------------+

100vh = 700 + 56 + 48 = 804px
Visible area = 700px
Result: 104px of content hidden below the fold
```

### The Solution: svh, lvh, dvh

| Unit | Viewport Size | When to Use |
|------|--------------|-------------|
| `svh` | Small (chrome visible) | Hero sections on load. Guaranteed to fit. |
| `lvh` | Large (chrome hidden) | Minimum page height when chrome is retracted. |
| `dvh` | Dynamic (adapts in real-time) | Modals, overlays that adapt as the user scrolls. |
| `vh` | Always equals Large | Legacy. Same as `lvh`. This is why it overflows. |

```css
/* Hero section: fills exactly the visible area, no overflow */
.hero {
  min-height: 100svh;
}

/* Modal overlay: adapts as address bar appears/disappears */
.modal-overlay {
  height: 100dvh;
}

/* Full-page background: fills screen when chrome is hidden */
.full-page {
  min-height: 100lvh;
}
```

### Width Units Too

The same pattern exists for width: `svw`, `lvw`, `dvw`. These matter less in practice because mobile browsers do not dynamically resize the viewport width. But they exist for consistency and for scrollbar differences on desktop.

### Legacy Fallback

```css
.hero {
  min-height: 100vh;                    /* Fallback for older browsers */
  min-height: -webkit-fill-available;   /* Safari legacy fix */
  min-height: 100svh;                   /* Modern solution */
}
```

**Browser support:** `svh`, `lvh`, `dvh` are supported in Chrome 108+, Firefox 101+, Safari 15.4+. All major browsers.

## interactive-widget: Virtual Keyboard Behavior

When a virtual keyboard appears on mobile, what happens to the page layout? The `interactive-widget` property controls this.

### The Three Modes

| Mode | Layout Viewport | Visual Viewport | Best For |
|------|----------------|-----------------|----------|
| `resizes-visual` | Unchanged | Shrinks | Default. General web pages. |
| `resizes-content` | Shrinks | Shrinks | Chat apps, forms where input must stay visible above keyboard. |
| `overlays-content` | Unchanged | Unchanged | Advanced: full control via Virtual Keyboard API. |

```html
<!-- Default (Chrome 108+): keyboard overlaps content -->
<meta name="viewport" content="width=device-width, initial-scale=1">

<!-- Chat app: keyboard pushes content up -->
<meta name="viewport" content="width=device-width, initial-scale=1, interactive-widget=resizes-content">

<!-- Advanced: handle keyboard manually via JS -->
<meta name="viewport" content="width=device-width, initial-scale=1, interactive-widget=overlays-content">
```

### The Fixed Bottom Bar Problem

If you have a `position: fixed` element at the bottom of the page, the default `resizes-visual` mode (Chrome 108+) means the keyboard covers it. The layout viewport does not shrink, so the fixed element stays behind the keyboard.

Solutions:
- Use `interactive-widget=resizes-content` to push the fixed bar above the keyboard.
- Or use the Virtual Keyboard API with `overlays-content` for granular control.
- Or use `dvh` units instead of fixed positioning to adapt dynamically.

**Browser support:** Chrome 108+, Firefox 132+. Safari does not support `interactive-widget` as of early 2026.

## Accessibility: Zoom and WCAG Compliance

### What WCAG Requires

**WCAG 1.4.4 (Resize Text, Level AA):** Text must be resizable up to 200% without loss of content or functionality.

**WCAG 1.4.10 (Reflow, Level AA):** Content must reflow at 400% zoom (equivalent to a 320px CSS viewport) without horizontal scrolling.

### What Fails

```html
<!-- FAILS WCAG 1.4.4: Disables zoom -->
<meta name="viewport" content="width=device-width, initial-scale=1, user-scalable=no">

<!-- FAILS WCAG 1.4.4: Prevents 200% zoom -->
<meta name="viewport" content="width=device-width, initial-scale=1, maximum-scale=1">

<!-- FAILS WCAG 1.4.4: maximum-scale below 2.0 -->
<meta name="viewport" content="width=device-width, initial-scale=1, maximum-scale=1.5">
```

### What Passes

```html
<!-- Best: no zoom restrictions -->
<meta name="viewport" content="width=device-width, initial-scale=1">

<!-- Also fine: generous maximum-scale -->
<meta name="viewport" content="width=device-width, initial-scale=1, maximum-scale=5">
```

### iOS Override

Since iOS 10, Safari **ignores** `user-scalable=no` and `maximum-scale` restrictions for user-initiated zoom. Apple made this as a deliberate accessibility protection. But this does not make the restriction acceptable:

- Android Chrome still enforces it (unless the user enables an accessibility setting).
- Automated WCAG audits (Lighthouse, axe, WAVE) flag it as a failure.
- It signals poor accessibility intent regardless of browser behavior.

### The Input Zoom Workaround

Developers often disable zoom to prevent Safari from auto-zooming into form inputs with font sizes below 16px. The correct fix:

```css
/* Bad Practice: disable zoom to prevent input zoom */
/* <meta name="viewport" content="..., maximum-scale=1"> */

/* Good: set input font size to 16px+ */
input,
select,
textarea {
  font-size: 1rem; /* 16px with default root size */
}
```

Safari auto-zooms into inputs when `font-size` is below 16px. Setting inputs to at least 16px prevents this without restricting zoom for all users.

## The Deprecated @viewport CSS Rule

The CSS Working Group proposed `@viewport` as a CSS-native replacement for the meta tag. It was part of the CSS Device Adaptation specification.

```css
/* DEPRECATED. Do not use. */
@viewport {
  width: device-width;
  zoom: 1;
}
```

It failed for three reasons:
1. Only IE10/11 and pre-Chromium Edge supported it (with `-ms-` prefix).
2. The meta tag was already universal. No incentive to migrate.
3. When both exist, the meta tag overrides `@viewport`.

The `@viewport` rule is fully deprecated and removed from all browser engines. The meta tag is the only mechanism for controlling the viewport.

## Recommended Configurations

| Scenario | Meta Tag |
|----------|----------|
| Standard responsive website | `<meta name="viewport" content="width=device-width, initial-scale=1">` |
| PWA on notched device | `<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">` |
| Chat app / keyboard-heavy forms | `<meta name="viewport" content="width=device-width, initial-scale=1, interactive-widget=resizes-content">` |
| PWA chat app on notched device | `<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover, interactive-widget=resizes-content">` |

## Common Mistakes

- **Omitting the tag entirely.** The page renders at 980px, shrunk to fit. Text is unreadable, tap targets are tiny, and media queries never activate. Google flags the page as "not mobile-friendly."

- **Using a fixed width.** `width=1024` forces a 1024px layout on all devices, causing horizontal scrolling on phones and disabling responsive behavior.

```html
<!-- Bad Practice -->
<meta name="viewport" content="width=1024">
```

- **Disabling zoom.** `user-scalable=no` or `maximum-scale=1` violates WCAG 1.4.4 (Resize Text, Level AA) and locks out users who rely on zoom for readability. The intent is usually to prevent Safari's input auto-zoom, which has a better CSS-only fix.

- **Including multiple viewport tags.** Only the last one is used. The first is silently ignored, which can cause confusing behavior if different tags set different values.

```html
<!-- Bad Practice: first tag is ignored -->
<meta name="viewport" content="width=device-width, initial-scale=1">
<meta name="viewport" content="width=500">
<!-- Result: viewport is 500px -->
```

- **Using `viewport-fit=cover` without handling safe areas.** The page extends behind the notch, but content is obscured because no `env(safe-area-inset-*)` padding is applied.

- **Using `height` in the viewport tag.** The `height` property is poorly implemented across browsers and produces inconsistent results. Do not use it.

- **Including `shrink-to-fit=no`.** This was a temporary iOS 9.0-9.3 workaround. It has been unnecessary since iOS 9.3.2 (2016). Harmless but dead weight.

---

## Best Practices

- **Use the canonical tag and nothing more.** `<meta name="viewport" content="width=device-width, initial-scale=1">` is sufficient for the vast majority of websites.

- **Never restrict zoom.** Do not add `user-scalable=no`, `maximum-scale=1`, or `minimum-scale=1`. Let users zoom.

- **Fix input zoom with CSS, not viewport restrictions.** Set `font-size: 1rem` (16px) on `input`, `select`, and `textarea` to prevent Safari's auto-zoom.

- **Only add `viewport-fit=cover` when you handle safe areas.** Pair it with `env(safe-area-inset-*)` padding on all edge-touching elements (headers, footers, fixed bars).

- **Use `svh` for hero sections, `dvh` for modals.** Stop using `100vh` for full-height sections on mobile. It overflows.

- **Use `interactive-widget=resizes-content` for form-heavy apps.** This keeps inputs visible above the virtual keyboard on Chrome.

- **Place the tag early in `<head>`.** The browser uses it before layout. Place it before any `<link>` or `<script>` tags to ensure it is parsed first.

## Real-World Use Cases

### 1. Standard Responsive Website

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>My Site</title>
  <link rel="stylesheet" href="styles.css">
</head>
<body>
  <!-- Responsive layout works correctly on all devices -->
</body>
</html>
```

```css
/* Base mobile styles (no media query) */
.container {
  width: 100%;
  padding: 1rem;
}

/* Tablet+ */
@media (min-width: 768px) {
  .container {
    max-width: 720px;
    margin: 0 auto;
    padding: 2rem;
  }
}
```

### 2. PWA with Notch Support

```html
<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">
```

```css
:root {
  --safe-top: env(safe-area-inset-top, 0px);
  --safe-right: env(safe-area-inset-right, 0px);
  --safe-bottom: env(safe-area-inset-bottom, 0px);
  --safe-left: env(safe-area-inset-left, 0px);
}

.app-header {
  position: fixed;
  top: 0;
  left: 0;
  right: 0;
  padding: calc(0.75rem + var(--safe-top)) var(--safe-right) 0.75rem var(--safe-left);
  background: var(--surface);
  z-index: 100;
}

.app-content {
  padding: calc(60px + var(--safe-top)) var(--safe-right) calc(60px + var(--safe-bottom)) var(--safe-left);
}

.app-tab-bar {
  position: fixed;
  bottom: 0;
  left: 0;
  right: 0;
  display: flex;
  height: calc(56px + var(--safe-bottom));
  padding-bottom: var(--safe-bottom);
  background: var(--surface);
}
```

### 3. Chat Application with Keyboard Handling

```html
<meta name="viewport" content="width=device-width, initial-scale=1, interactive-widget=resizes-content">
```

```css
.chat-layout {
  display: flex;
  flex-direction: column;
  height: 100dvh; /* Adapts to keyboard presence */
}

.chat-messages {
  flex: 1;
  overflow-y: auto;
  padding: 1rem;
}

.chat-input-bar {
  /* No position: fixed needed. The layout viewport shrinks,
     so this naturally sits above the keyboard. */
  display: flex;
  gap: 0.5rem;
  padding: 0.75rem;
  border-top: 1px solid var(--border);
  background: var(--surface);
}

.chat-input-bar input {
  flex: 1;
  font-size: 1rem; /* 16px: prevents Safari auto-zoom */
  padding: 0.5rem;
}
```

The combination of `interactive-widget=resizes-content` and `100dvh` means the entire chat layout shrinks when the keyboard appears, keeping the input field visible without JavaScript scroll hacks.

---

> **Key Takeaway:** The viewport meta tag is a single line of HTML that determines whether your responsive CSS works at all on mobile. Use `width=device-width, initial-scale=1`, never restrict zoom, and add `viewport-fit=cover` or `interactive-widget` only when you have a specific use case that demands it.
