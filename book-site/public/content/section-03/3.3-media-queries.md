# 3.3 Media Queries

> Media queries let you conditionally apply CSS based on the user's device characteristics, viewport dimensions, and personal preferences. They are the backbone of responsive design and the primary mechanism for adapting layouts, adjusting accessibility, and optimizing for print.

---

## Anatomy of a Media Query

Every media query follows the same structure:

```css
@media [media-type] [logical-operator] (media-feature: value) {
  /* Conditional styles */
}
```

Three places you can use media queries:

```css
/* 1. Inside a stylesheet with @media */
@media (width >= 768px) {
  .sidebar { display: block; }
}
```

```html
<!-- 2. On a <link> tag to conditionally load a stylesheet -->
<link rel="stylesheet" href="print.css" media="print">

<!-- 3. On a <style> tag -->
<style media="screen and (min-width: 768px)">
  .sidebar { display: block; }
</style>
```

```html
<!-- 4. Inside a <source> or <picture> element -->
<picture>
  <source srcset="wide.webp" media="(min-width: 1024px)">
  <img src="narrow.webp" alt="responsive image">
</picture>
```

---

## Media Types

Media types describe the broad category of the output device.

| Type | Description |
|------|-------------|
| `all` | Matches every device. Default when no type is specified. |
| `screen` | Screens: monitors, phones, tablets. |
| `print` | Printers and print preview mode. |

### Deprecated Types (Do Not Use)

The following were defined in earlier specifications and deprecated in Media Queries 4. Browsers treat them as unknown types (never match):

`tty`, `tv`, `projection`, `handheld`, `braille`, `embossed`, `aural`, `speech`

```css
/* BAD PRACTICE: these types are deprecated and will never match */
@media handheld {
  /* This code never executes in any modern browser */
}
```

If you omit the media type entirely, `all` is implied:

```css
/* These two are equivalent */
@media (min-width: 768px) { }
@media all and (min-width: 768px) { }
```

## Media Features: Complete Reference

### Viewport and Dimension Features

These features test the dimensions of the viewport (or page box, for print).

| Feature | Type | Accepts min/max | Description |
|---------|------|-----------------|-------------|
| `width` | range | Yes | Width of the viewport |
| `height` | range | Yes | Height of the viewport |
| `aspect-ratio` | range | Yes | Ratio of viewport width to height (e.g., `16/9`) |
| `orientation` | discrete | No | `portrait` or `landscape` |
| `overflow-block` | discrete | No | How the device handles content overflow in block axis: `none`, `scroll`, `paged` |
| `overflow-inline` | discrete | No | How the device handles content overflow in inline axis: `none`, `scroll` |

```css
/* Portrait phones */
@media (orientation: portrait) and (max-width: 480px) {
  .hero { height: 50vh; }
}

/* Widescreen monitors */
@media (min-aspect-ratio: 16/9) {
  .video-wrapper { padding-top: 56.25%; }
}
```

### Display Quality Features

| Feature | Type | Accepts min/max | Description |
|---------|------|-----------------|-------------|
| `resolution` | range | Yes | Pixel density: `dpi`, `dpcm`, `dppx`, or `x` |
| `color` | range | Yes | Bits per color channel. `0` for monochrome devices. |
| `color-index` | range | Yes | Number of entries in the color lookup table |
| `monochrome` | range | Yes | Bits per pixel in a monochrome frame buffer |
| `color-gamut` | discrete | No | Approximate color gamut: `srgb`, `p3`, `rec2020` |
| `dynamic-range` | discrete | No | Combination of brightness, contrast ratio, and color depth: `standard`, `high` |
| `update` | discrete | No | Refresh rate capability: `none` (print), `slow` (e-ink), `fast` (screens) |

```css
/* Retina / HiDPI displays */
@media (min-resolution: 2dppx) {
  .logo { background-image: url('logo@2x.png'); }
}

/* Wide gamut P3 display: serve richer colors */
@media (color-gamut: p3) {
  :root {
    --brand-red: color(display-p3 1 0.2 0.1);
  }
}

/* HDR display */
@media (dynamic-range: high) {
  .hero-image { filter: brightness(1.2); }
}

/* E-ink or slow-updating displays: disable animations */
@media (update: slow), (update: none) {
  * {
    animation-duration: 0s !important;
    transition-duration: 0s !important;
  }
}
```

### Interaction Features

These detect input device capabilities. Critical for adapting hover effects and tap targets.

| Feature | Tests | Values | Description |
|---------|-------|--------|-------------|
| `hover` | Primary input | `hover`, `none` | Can the primary input hover? |
| `any-hover` | Any input | `hover`, `none` | Can any available input hover? |
| `pointer` | Primary input | `fine`, `coarse`, `none` | Accuracy of the primary pointer |
| `any-pointer` | Any input | `fine`, `coarse`, `none` | Accuracy of any available pointer |

The distinction between `hover`/`pointer` and `any-hover`/`any-pointer` matters for multi-input devices. A tablet with a Bluetooth mouse has `pointer: coarse` (touchscreen is primary) but `any-pointer: fine` (mouse is also available).

```css
/* Only show hover effects on devices that can actually hover */
@media (hover: hover) {
  .card:hover {
    transform: translateY(-4px);
    box-shadow: 0 8px 24px rgba(0, 0, 0, 0.15);
  }
}

/* Enlarge tap targets for coarse pointers (fingers) */
@media (pointer: coarse) {
  .btn {
    min-height: 48px;
    min-width: 48px;
    padding: 12px 24px;
  }
}

/* Hide tooltips on touch-only devices */
@media (hover: none) and (pointer: coarse) {
  .tooltip { display: none; }
}
```

**Gotcha: Samsung devices.** Some Samsung phones with One UI report `(hover: hover)` despite being touch-only devices. This is a known browser bug. Never rely solely on `hover` media queries for critical functionality. Always ensure content remains accessible without hover.

```css
/* BAD PRACTICE: hiding content behind hover on assumption of mouse */
@media (hover: hover) {
  .dropdown-content {
    display: none;
  }
  .dropdown:hover .dropdown-content {
    display: block;
  }
}

/* BETTER: use hover as enhancement, keep a tap/click fallback */
.dropdown-content {
  display: none;
}
.dropdown.is-open .dropdown-content,
.dropdown:focus-within .dropdown-content {
  display: block;
}
@media (hover: hover) {
  .dropdown:hover .dropdown-content {
    display: block;
  }
}
```

## Logical Operators

### `and` - Both Conditions Must Be True

```css
/* Screen wider than 768px AND in landscape */
@media (min-width: 768px) and (orientation: landscape) {
  .sidebar { width: 300px; }
}

/* With a media type */
@media screen and (min-width: 768px) and (max-width: 1024px) {
  .container { padding: 2rem; }
}
```

### Comma (`,`) - Logical OR

A comma-separated list acts as OR. If any query in the list matches, the entire block applies.

```css
/* Narrow screens OR print */
@media (max-width: 600px), print {
  .sidebar { display: none; }
}
```

Since Media Queries Level 4, you can also use the `or` keyword inside parenthesized groups (see "Level 4 `or` keyword" below).

### `not` - Negate Entire Query

`not` negates the entire media query it belongs to, not individual features. This is the single biggest source of confusion.

```css
/* "not screen and (color)" means "not (screen and (color))" */
@media not screen and (color) {
  /* Applies to non-screen devices OR monochrome screens */
}

/* It does NOT mean "(not screen) and (color)" */
```

In a comma-separated list, `not` only negates the query it is attached to:

```css
/* "not screen and (color), print" means:
   - not (screen and (color))     <- negated
   - OR print                     <- not negated */
@media not screen and (color), print {
  body { background: white; }
}
```

**Level 4 `not` on individual features:** Media Queries Level 4 allows negating a single feature using parentheses:

```css
/* Negate a single feature (Level 4 syntax) */
@media (not (hover: hover)) {
  /* Devices where the primary input cannot hover */
}

/* Negate a group */
@media (not ((color) and (hover))) {
  /* Devices that are not (color AND hoverable) */
}
```

### `only` - Hide From Legacy Browsers

The `only` keyword prevents older browsers (IE9 and below) that do not support media features from applying the styles. Modern browsers ignore it. It requires a media type.

```css
@media only screen and (min-width: 768px) {
  /* Old browsers see "only" as an unknown media type and skip this block */
  /* Modern browsers ignore "only" and evaluate normally */
}
```

In practice, `only` is no longer necessary unless you support browsers from 2012 or earlier.

### Level 4 `or` Keyword

Media Queries Level 4 introduced an explicit `or` keyword that works inside grouped expressions:

```css
/* Level 4 syntax */
@media ((min-width: 768px) or (orientation: landscape)) {
  .content { columns: 2; }
}
```

**Mixing operators is illegal.** You cannot combine `and` and `or` at the same nesting level without parentheses:

```css
/* BAD: ambiguous, browsers will reject this */
@media (color) and (pointer: fine) or (hover: hover) { }

/* GOOD: use parentheses to clarify precedence */
@media (color) and ((pointer: fine) or (hover: hover)) { }
@media ((color) and (pointer: fine)) or (hover: hover) { }
```

## Range Syntax (Media Queries Level 4)

The range syntax replaces the `min-`/`max-` prefix pattern with mathematical comparison operators. It is more readable and less error-prone.

### Comparison Operators

| Operator | Meaning |
|----------|---------|
| `>` | Greater than |
| `>=` | Greater than or equal to |
| `<` | Less than |
| `<=` | Less than or equal to |
| `=` | Equal to |

### Side-by-Side Comparison

| Legacy Syntax | Range Syntax |
|---------------|-------------|
| `(min-width: 768px)` | `(width >= 768px)` |
| `(max-width: 1024px)` | `(width <= 1024px)` |
| `(min-width: 768px) and (max-width: 1024px)` | `(768px <= width <= 1024px)` |
| `(min-resolution: 2dppx)` | `(resolution >= 2dppx)` |

```css
/* Mobile-first breakpoints with range syntax */

/* Base: mobile (no query needed) */

/* Tablet and up */
@media (width >= 768px) {
  .grid { grid-template-columns: repeat(2, 1fr); }
}

/* Desktop and up */
@media (width >= 1024px) {
  .grid { grid-template-columns: repeat(3, 1fr); }
}

/* Wide desktop */
@media (width >= 1440px) {
  .grid { grid-template-columns: repeat(4, 1fr); }
}

/* Target a range */
@media (768px <= width <= 1024px) {
  .sidebar { width: 240px; }
}
```

### Why Range Syntax is Better

The legacy `min-`/`max-` syntax has a classic off-by-one problem:

```css
/* BAD PRACTICE: overlapping breakpoints */
@media (max-width: 768px) { /* 0-768 */ }
@media (min-width: 768px) { /* 768+ ... overlap at exactly 768px! */ }

/* GOOD: range syntax makes boundaries explicit */
@media (width < 768px) { /* 0 to 767.99... */ }
@media (width >= 768px) { /* 768+ */ }
```

### Browser Support

| Browser | Minimum Version |
|---------|----------------|
| Chrome | 104 |
| Firefox | 63 |
| Safari | 16.4 |
| Edge | 104 |

Range syntax is safe for production in 2025+. Global browser support exceeds 95%.

## User Preference Media Features

These features detect system-level preferences set by the user in their OS or browser settings. They are essential for accessibility and personalization.

### `prefers-color-scheme`

Detects whether the user has requested a light or dark theme.

| Value | Description |
|-------|-------------|
| `light` | User prefers a light theme (default in most systems) |
| `dark` | User prefers a dark theme |

```css
/* Define color variables with light-mode defaults */
:root {
  --bg: #ffffff;
  --text: #1a1a1a;
  --surface: #f5f5f5;
  --border: #e0e0e0;
}

/* Override for dark mode */
@media (prefers-color-scheme: dark) {
  :root {
    --bg: #121212;
    --text: #e0e0e0;
    --surface: #1e1e1e;
    --border: #333333;
  }
}

body {
  background-color: var(--bg);
  color: var(--text);
}
```

**Pattern: CSS `color-scheme` property as companion.**
The `color-scheme` property tells the browser which color schemes the page supports, so native form controls, scrollbars, and system colors automatically adapt:

```css
:root {
  color-scheme: light dark;
}
```

**Pattern: JavaScript detection and toggle.**

```js
// Check current system preference
const darkQuery = window.matchMedia('(prefers-color-scheme: dark)');
console.log(darkQuery.matches); // true if dark mode

// Listen for changes (user toggles system dark mode)
darkQuery.addEventListener('change', (e) => {
  document.documentElement.setAttribute(
    'data-theme',
    e.matches ? 'dark' : 'light'
  );
});
```

**Pattern: user override with a toggle button.**

```css
/* System preference as default */
@media (prefers-color-scheme: dark) {
  :root:not([data-theme="light"]) {
    --bg: #121212;
    --text: #e0e0e0;
  }
}

/* Explicit user override */
[data-theme="dark"] {
  --bg: #121212;
  --text: #e0e0e0;
}

[data-theme="light"] {
  --bg: #ffffff;
  --text: #1a1a1a;
}
```

**Common Mistake:** Not testing both modes during development. Use Chrome DevTools: open Rendering panel (Ctrl+Shift+P, search "Rendering"), then use "Emulate CSS media feature prefers-color-scheme" dropdown.

### `prefers-reduced-motion`

Detects whether the user has requested reduced animation. Critical for accessibility (vestibular disorders, epilepsy, ADHD).

| Value | Description |
|-------|-------------|
| `no-preference` | User has not indicated a preference |
| `reduce` | User wants minimized motion |

**Approach 1: Remove motion when requested (additive motion).**

```css
/* Animations defined normally */
.fade-in {
  animation: fadeSlideIn 0.5s ease-out;
}

@keyframes fadeSlideIn {
  from {
    opacity: 0;
    transform: translateY(20px);
  }
  to {
    opacity: 1;
    transform: translateY(0);
  }
}

/* Remove or simplify when user prefers reduced motion */
@media (prefers-reduced-motion: reduce) {
  .fade-in {
    animation: none;
    /* Element still appears, just without the slide */
  }

  /* Or replace with a subtle fade (no spatial movement) */
  .fade-in {
    animation: fadeOnly 0.2s ease-out;
  }

  @keyframes fadeOnly {
    from { opacity: 0; }
    to { opacity: 1; }
  }
}
```

**Approach 2: No-motion-first (subtractive motion).** Start with no animations and add them only when the user has not requested reduced motion. This approach is safer because it defaults to accessible behavior.

```css
/* Base: no animation */
.card {
  transition: none;
}

/* Add animation only when user is OK with motion */
@media (prefers-reduced-motion: no-preference) {
  .card {
    transition: transform 0.3s ease, box-shadow 0.3s ease;
  }

  .card:hover {
    transform: translateY(-4px);
    box-shadow: 0 8px 24px rgba(0, 0, 0, 0.15);
  }
}
```

**Important: reduce does not mean remove all motion.** Some motion aids comprehension (e.g., a list item sliding to make room for a new entry). The goal is to remove gratuitous, large-scale, or vestibular-triggering motion. Simple opacity fades are generally fine.

**Gotcha:** Even with `prefers-reduced-motion`, WCAG 2.2.2 requires a pause/stop mechanism for any animation lasting more than 5 seconds. The media query alone satisfies the technical requirement, but a visible pause button is better UX.

### `prefers-contrast`

Detects whether the user has requested higher or lower contrast.

| Value | Description |
|-------|-------------|
| `no-preference` | No preference indicated |
| `more` | User wants higher contrast |
| `less` | User wants lower contrast |
| `custom` | User has a custom palette (Windows High Contrast Mode) |

```css
/* Default styling */
.btn-secondary {
  border: 1px solid #ccc;
  color: #555;
}

/* Increase contrast when requested */
@media (prefers-contrast: more) {
  .btn-secondary {
    border: 2px solid #000;
    color: #000;
  }

  /* Increase text contrast */
  body {
    --text: #000000;
    --bg: #ffffff;
  }

  /* Make focus indicators more visible */
  :focus-visible {
    outline: 3px solid #000;
    outline-offset: 2px;
  }
}
```

### `forced-colors`

Detects when the user agent enforces a limited color palette (Windows High Contrast Mode). This is separate from `prefers-contrast`.

| Value | Description |
|-------|-------------|
| `none` | No forced colors active |
| `active` | Forced color mode is active |

When `forced-colors: active`, the browser overrides most color-related properties with system colors. Use this query for targeted fixes, not to create a separate design.

```css
@media (forced-colors: active) {
  /* Borders that rely on background color differences need explicit borders */
  .card {
    border: 1px solid ButtonText;
  }

  /* Custom SVG icons may lose visibility: force a system color */
  .icon svg {
    fill: ButtonText;
  }

  /* Prevent the browser from overriding specific elements */
  .brand-logo {
    forced-color-adjust: none;
  }
}
```

### `prefers-reduced-transparency`

Detects whether the user has requested reduced transparency effects.

| Value | Description |
|-------|-------------|
| `no-preference` | No preference indicated |
| `reduce` | User wants less transparency |

```css
.modal-overlay {
  background: rgba(0, 0, 0, 0.5);
}

@media (prefers-reduced-transparency: reduce) {
  .modal-overlay {
    background: rgb(0, 0, 0); /* Fully opaque */
  }

  .glass-card {
    backdrop-filter: none;
    background: var(--surface); /* Solid fallback */
  }
}
```

**Browser Support:** Chrome 118+, Edge 118+. Firefox requires a flag (`layout.css.prefers-reduced-transparency.enabled`). Safari does not support it yet. Use as progressive enhancement only.

## Scripting Media Feature

Detects whether JavaScript is available in the current document.

| Value | Description |
|-------|-------------|
| `none` | Scripting is not supported or not active |
| `initial-only` | Scripting is available only during initial page load (e.g., printed pages with scripts) |
| `enabled` | Scripting is fully supported and active |

```css
/* Show a no-JS warning */
@media (scripting: none) {
  .js-required-notice {
    display: block;
  }
  .js-only-widget {
    display: none;
  }
}

/* Progressive enhancement: show enhanced UI only when JS is available */
@media (scripting: enabled) {
  .dropdown {
    /* JS-powered dropdown styling */
  }
}
```

**Gotcha:** Browser extensions like NoScript or uBlock Origin that disable JavaScript are not detected. `scripting: enabled` still matches because the browser itself supports scripting; the extension blocks it at a different layer.

**Browser Support:** Chrome 120+, Firefox 119+, Safari 17+.

## Display Mode Media Feature

Detects how a web application is displayed, primarily for Progressive Web Apps (PWAs).

| Value | Description |
|-------|-------------|
| `browser` | Normal browser tab/window |
| `standalone` | Standalone window, no browser UI |
| `minimal-ui` | Standalone with minimal navigation controls |
| `fullscreen` | Full screen, no system UI |

```css
/* Show a "back" button only in standalone PWA mode (no browser back button) */
@media (display-mode: standalone) {
  .pwa-back-button {
    display: flex;
  }
}

/* Hide install prompt when already installed */
@media (display-mode: standalone), (display-mode: fullscreen) {
  .install-banner {
    display: none;
  }
}
```

Fallback chain when the browser does not support the requested mode: `fullscreen` -> `standalone` -> `minimal-ui` -> `browser`.

## Print Stylesheets

Print styles are applied when the user prints a page or enters print preview. They use the `print` media type.

### Two Approaches

**1. Separate file (loaded only for print):**

```html
<!-- The browser downloads this but does NOT render-block for screen -->
<link rel="stylesheet" href="print.css" media="print">
```

**2. Embedded in main stylesheet:**

```css
@media print {
  /* Print styles here */
}
```

### Essential Print Styles

```css
@media print {
  /* 1. Hide non-printable elements */
  nav,
  footer,
  .sidebar,
  .ad-banner,
  .cookie-consent,
  .no-print,
  button,
  video,
  audio {
    display: none !important;
  }

  /* 2. Reset backgrounds and colors for ink savings */
  body {
    background: white !important;
    color: black !important;
    font-size: 12pt;
    line-height: 1.5;
  }

  /* 3. Remove shadows and decorative effects */
  * {
    box-shadow: none !important;
    text-shadow: none !important;
  }

  /* 4. Kill animations and transitions */
  * {
    animation-duration: 0s !important;
    transition-duration: 0s !important;
  }

  /* 5. Make links useful on paper */
  a[href^="http"]::after {
    content: " (" attr(href) ")";
    font-size: 0.8em;
    color: #555;
  }

  /* 6. Avoid breaking inside critical elements */
  h1, h2, h3, h4, h5, h6 {
    break-after: avoid;
  }

  img, table, pre, blockquote {
    break-inside: avoid;
  }

  /* 7. Ensure images don't overflow the page */
  img {
    max-width: 100% !important;
  }

  /* 8. Set page margins */
  @page {
    margin: 2cm;
  }

  @page :first {
    margin-top: 3cm; /* Extra space on first page for title */
  }

  /* 9. Typography for print */
  body {
    font-family: Georgia, "Times New Roman", serif;
  }

  /* 10. Full-width content */
  .container, main, article {
    width: 100% !important;
    max-width: none !important;
    margin: 0 !important;
    padding: 0 !important;
  }
}
```

### Print-Specific CSS Properties

| Property | Description |
|----------|-------------|
| `break-before` | Force/avoid page break before element: `page`, `avoid`, `auto` |
| `break-after` | Force/avoid page break after element |
| `break-inside` | Prevent page breaks inside element: `avoid` |
| `orphans` | Minimum lines of a paragraph at the bottom of a page |
| `widows` | Minimum lines of a paragraph at the top of a page |
| `@page` | Define page margins, size, and named pages |

```css
@media print {
  p {
    orphans: 3; /* At least 3 lines before a break */
    widows: 3;  /* At least 3 lines after a break */
  }
}
```

## Nesting Media Queries

### Nesting Media Queries Inside Each Other

Media queries can be nested. The conditions combine with logical AND:

```css
@media (width >= 768px) {
  .sidebar { display: block; }

  @media (orientation: landscape) {
    /* This means: width >= 768px AND landscape */
    .sidebar { width: 300px; }
  }
}
```

### Native CSS Nesting with Media Queries

With CSS Nesting (supported in all modern browsers since 2023), you can nest `@media` inside a rule block. This keeps related styles together:

```css
/* Without nesting: scattered selectors */
.card {
  padding: 1rem;
  font-size: 0.875rem;
}

@media (width >= 768px) {
  .card {
    padding: 1.5rem;
    font-size: 1rem;
  }
}

@media (width >= 1024px) {
  .card {
    padding: 2rem;
    font-size: 1.125rem;
  }
}

/* With CSS nesting: everything in one block */
.card {
  padding: 1rem;
  font-size: 0.875rem;

  @media (width >= 768px) {
    padding: 1.5rem;
    font-size: 1rem;
  }

  @media (width >= 1024px) {
    padding: 2rem;
    font-size: 1.125rem;
  }
}
```

The `&` selector is implied in nested `@media` blocks. You can make it explicit:

```css
.header {
  font-size: 2rem;

  @media (width < 768px) {
    & {
      font-size: 1.25rem;
    }

    & .subtitle {
      display: none;
    }
  }
}
```

**Browser Support for CSS Nesting:** Chrome 112+, Firefox 117+, Safari 17.2+, Edge 112+.

## Media Queries in JavaScript

### `window.matchMedia()`

The `matchMedia()` method returns a `MediaQueryList` object that you can query and listen to:

```js
// One-time check
const isWide = window.matchMedia('(width >= 1024px)');
console.log(isWide.matches); // true or false

// Listen for changes
const darkMode = window.matchMedia('(prefers-color-scheme: dark)');

function handleThemeChange(e) {
  if (e.matches) {
    document.documentElement.classList.add('dark');
  } else {
    document.documentElement.classList.remove('dark');
  }
}

// Set initial state
handleThemeChange(darkMode);

// React to changes
darkMode.addEventListener('change', handleThemeChange);

// Clean up (when no longer needed)
darkMode.removeEventListener('change', handleThemeChange);
```

**Why `matchMedia` instead of `resize` event?** The `resize` event fires at every pixel of viewport change, potentially hundreds of times during a drag. `matchMedia` fires only when the query result changes (crosses a breakpoint), making it far more efficient.

```js
/* BAD PRACTICE: resize event for breakpoint detection */
window.addEventListener('resize', () => {
  if (window.innerWidth >= 768) {
    showDesktopNav();
  } else {
    showMobileNav();
  }
});

/* GOOD: matchMedia fires only at the boundary */
const tabletQuery = window.matchMedia('(width >= 768px)');
tabletQuery.addEventListener('change', (e) => {
  if (e.matches) {
    showDesktopNav();
  } else {
    showMobileNav();
  }
});
```

### Detecting Media Query Support in CSS

You can test if a browser supports a given media feature using `@supports` in combination with `@media`, or use `matchMedia` in JavaScript:

```js
// Check if the browser understands a media feature
function supportsMediaFeature(feature) {
  return window.matchMedia(`(${feature})`).media !== 'not all';
}

console.log(supportsMediaFeature('prefers-contrast: more')); // true/false
console.log(supportsMediaFeature('hover: hover'));           // true/false
```

## Media Queries vs. Container Queries

Media queries and container queries solve different problems. Understanding when to use each prevents overengineered CSS.

| Aspect | Media Queries | Container Queries |
|--------|---------------|-------------------|
| Responds to | Viewport dimensions, device/user settings | Parent container dimensions |
| Scope | Global (page-level) | Local (component-level) |
| Best for | Page layouts, breakpoints, user preferences | Reusable components that appear in different contexts |
| Syntax | `@media (width >= 768px)` | `@container (width >= 400px)` |
| Can detect | Width, height, orientation, color-scheme, motion prefs, etc. | Width, height (of the container) |
| Browser support | Universal | Chrome 105+, Firefox 110+, Safari 16+ |

### When to Use Media Queries

- Page-level layout changes (switching from single column to sidebar layout)
- User preference detection (dark mode, reduced motion, contrast)
- Print styles
- Targeting device capabilities (hover, pointer, resolution)
- `<picture>` and `<source>` `media` attributes (container queries cannot be used here)

### When to Use Container Queries

- A card component that appears in a sidebar (300px) and main content area (800px)
- A navigation component reused across pages with different layouts
- Any component whose layout should depend on its available space, not the viewport

```css
/* Container query example for comparison */
.card-container {
  container-type: inline-size;
  container-name: card;
}

.card {
  display: grid;
  grid-template-columns: 1fr;
}

@container card (width >= 400px) {
  .card {
    grid-template-columns: 200px 1fr;
  }
}
```

### Use Both Together

```css
/* Media query: page-level layout */
@media (width >= 1024px) {
  .page {
    display: grid;
    grid-template-columns: 300px 1fr;
  }
}

/* Container query: component-level adaptation */
.content-area {
  container-type: inline-size;
}

@container (width >= 500px) {
  .article-card {
    flex-direction: row;
  }
}

/* Media query: user preferences (cannot be replaced by container queries) */
@media (prefers-color-scheme: dark) {
  :root { --bg: #121212; }
}
```

## Performance Considerations

### Render Blocking Behavior

All stylesheets block rendering, but the browser is smart about prioritization:

```html
<!-- Render-blocks: browser waits for this before first paint -->
<link rel="stylesheet" href="main.css">

<!-- Does NOT render-block: browser knows it is not needed for screen -->
<link rel="stylesheet" href="print.css" media="print">

<!-- Does NOT render-block if viewport is narrower than 1024px -->
<link rel="stylesheet" href="desktop.css" media="(min-width: 1024px)">
```

**Critical detail:** The browser still downloads all linked stylesheets regardless of whether the media query matches. The `media` attribute only affects render-blocking, not downloading. This is by design: the user might rotate their phone or connect a printer at any time.

### Number of Media Queries

Having many media queries does not significantly impact runtime performance. The browser's CSS engine evaluates media queries efficiently. The real cost is file size: hundreds of inline `@media` blocks increase CSS byte size, which affects download time. Gzip compression mitigates this significantly because repeated `@media` patterns compress well.

### Optimization Strategies

**1. Mobile-first with `min-width`:**

```css
/* Mobile-first produces smaller initial payload.
   Mobile devices download less CSS before first paint. */

/* Small screen styles: no query, always loaded */
.grid { display: flex; flex-direction: column; }

/* Medium screens and up: additional rules */
@media (width >= 768px) {
  .grid { flex-direction: row; }
}
```

**2. Conditional loading for large differences:**

```html
<!-- Split CSS when desktop and mobile are drastically different -->
<link rel="stylesheet" href="core.css">
<link rel="stylesheet" href="desktop.css" media="(min-width: 1024px)">
```

**3. Avoid wrapping entire stylesheets in media queries:**

```css
/* BAD PRACTICE: duplicating all styles inside media queries */
@media (max-width: 767px) {
  .header { /* ... */ }
  .nav { /* ... */ }
  .card { /* ... */ }
  /* 200 more rules */
}
@media (min-width: 768px) {
  .header { /* ... */ }
  .nav { /* ... */ }
  .card { /* ... */ }
  /* 200 more rules */
}

/* GOOD: base styles + targeted overrides */
.header { /* base */ }
.nav { /* base */ }
.card { /* base */ }

@media (width >= 768px) {
  .nav { /* only what changes */ }
}
```

## Common Breakpoints

There is no universal "correct" set of breakpoints. The right answer is to set breakpoints where your content breaks, not where devices happen to be. That said, these are widely used starting points:

| Name | min-width (CSS) | Target |
|------|----------------|--------|
| Small phone | (default) | 0 - 479px |
| Phone | 480px | Large phones in portrait |
| Tablet | 768px | Tablets in portrait, large phones in landscape |
| Desktop | 1024px | Small desktops, tablets in landscape |
| Wide | 1280px | Standard desktops |
| Ultra-wide | 1536px | Large monitors |

```css
/* A practical mobile-first breakpoint system */
:root {
  --bp-sm: 480px;
  --bp-md: 768px;
  --bp-lg: 1024px;
  --bp-xl: 1280px;
}

/* NOTE: CSS custom properties cannot be used inside media queries.
   This is a limitation of the spec. The values above are for documentation only.
   You must use literal values in @media rules. */

@media (width >= 480px) { /* sm */ }
@media (width >= 768px) { /* md */ }
@media (width >= 1024px) { /* lg */ }
@media (width >= 1280px) { /* xl */ }
```

**Gotcha: CSS custom properties do not work in media queries.** `@media (width >= var(--bp-md))` is invalid and will be ignored. This is because media queries are evaluated before the cascade, and custom properties are resolved during the cascade. Use a preprocessor variable (SASS `$variable`) if you need DRY breakpoints, or define them as comments for documentation.

## Common Mistakes and Gotchas

### 1. Overlapping Breakpoints with Legacy Syntax

```css
/* BAD: both match at exactly 768px */
@media (max-width: 768px) { .a { color: red; } }
@media (min-width: 768px) { .a { color: blue; } }

/* FIX with legacy syntax: use 767.98px */
@media (max-width: 767.98px) { .a { color: red; } }
@media (min-width: 768px) { .a { color: blue; } }

/* FIX with range syntax: no ambiguity */
@media (width < 768px) { .a { color: red; } }
@media (width >= 768px) { .a { color: blue; } }
```

### 2. Forgetting the Viewport Meta Tag

Without `<meta name="viewport" content="width=device-width, initial-scale=1">`, mobile browsers render the page at ~980px width and your media queries never fire at the breakpoints you expect.

### 3. Designing for Specific Devices

```css
/* BAD: targeting specific devices */
@media (width: 375px) { /* iPhone SE */ }
@media (width: 390px) { /* iPhone 14 */ }
@media (width: 412px) { /* Pixel 8 */ }

/* GOOD: breakpoints where content needs adjustment */
@media (width >= 600px) { /* content needs 2 columns here */ }
```

### 4. Using Too Many Breakpoints

Every breakpoint is a maintenance burden. Most layouts need 3-4 breakpoints. If you need more than 5, reconsider your layout approach. Use fluid techniques (`clamp()`, `fr` units, `flex-wrap`) to reduce the number of breakpoints needed.

### 5. Desktop-First Cascade Issues

```css
/* BAD: desktop-first with max-width creates cascading overrides */
.sidebar { width: 300px; float: left; }

@media (max-width: 1024px) {
  .sidebar { width: 200px; }
}

@media (max-width: 768px) {
  .sidebar { width: 100%; float: none; }
}

/* GOOD: mobile-first, each breakpoint adds complexity */
.sidebar { width: 100%; }

@media (width >= 768px) {
  .sidebar { width: 200px; float: left; }
}

@media (width >= 1024px) {
  .sidebar { width: 300px; }
}
```

### 6. Misunderstanding `not` Scope

```css
/* This negates the ENTIRE query, not just "screen" */
@media not screen and (min-width: 768px) {
  /* Equivalent to: not (screen and (min-width: 768px)) */
  /* Matches: print, non-screen devices, AND screens under 768px */
}

/* If you want to negate just the media type: */
@media not screen {
  /* Only non-screen devices */
}
```

### 7. Assuming `hover: hover` Means Mouse

Some touch devices report `hover: hover`. Always provide fallbacks for tap/keyboard interaction alongside hover styles.

### 8. Not Testing Print Styles

Print styles are invisible unless you test them. Use the browser's print preview (Ctrl+P / Cmd+P) or toggle the CSS media type to `print` in DevTools:
- Chrome: DevTools -> Rendering -> Emulate CSS media type -> print
- Firefox: DevTools -> Inspector -> toggle print media simulation button

```css
/* ==========================================
   BASE STYLES (Mobile-first, no media query)
   ========================================== */
:root {
  --bg: #ffffff;
  --text: #1a1a1a;
  --surface: #f8f8f8;
  --primary: #0066cc;
  --gap: 1rem;

  color-scheme: light dark;
}

body {
  font-family: system-ui, -apple-system, sans-serif;
  background: var(--bg);
  color: var(--text);
  line-height: 1.6;
  padding: var(--gap);
}

.page {
  display: flex;
  flex-direction: column;
  gap: var(--gap);
}

.nav-list {
  display: flex;
  flex-direction: column;
  gap: 0.5rem;
}

.grid {
  display: grid;
  grid-template-columns: 1fr;
  gap: var(--gap);
}

/* ==========================================
   DARK MODE
   ========================================== */
@media (prefers-color-scheme: dark) {
  :root {
    --bg: #0d1117;
    --text: #e6edf3;
    --surface: #161b22;
    --primary: #58a6ff;
  }
}

/* ==========================================
   REDUCED MOTION
   ========================================== */
@media (prefers-reduced-motion: no-preference) {
  .card {
    transition: transform 0.2s ease, box-shadow 0.2s ease;
  }

  .fade-in {
    animation: fadeSlideUp 0.4s ease-out both;
  }

  @keyframes fadeSlideUp {
    from {
      opacity: 0;
      transform: translateY(16px);
    }
  }
}

/* ==========================================
   HIGH CONTRAST
   ========================================== */
@media (prefers-contrast: more) {
  :root {
    --text: #000000;
    --bg: #ffffff;
    --primary: #0000cc;
  }

  .card {
    border: 2px solid var(--text);
  }

  :focus-visible {
    outline: 3px solid var(--text);
    outline-offset: 2px;
  }
}

/* ==========================================
   FORCED COLORS (Windows High Contrast)
   ========================================== */
@media (forced-colors: active) {
  .card {
    border: 1px solid ButtonText;
  }

  .btn {
    border: 1px solid ButtonText;
  }
}

/* ==========================================
   REDUCED TRANSPARENCY
   ========================================== */
@media (prefers-reduced-transparency: reduce) {
  .overlay {
    background: var(--bg);
    opacity: 1;
  }
}

/* ==========================================
   HOVER-CAPABLE DEVICES
   ========================================== */

  .card:hover {
    transform: translateY(-2px);
    box-shadow: 0 4px 16px rgba(0, 0, 0, 0.1);
  }

  .nav-link:hover {
    text-decoration: underline;
    text-underline-offset: 4px;
  }
}

/* Enlarge touch targets for coarse pointers */
@media (pointer: coarse) {
  .btn, .nav-link {
    min-height: 48px;
    padding: 12px 20px;
  }
}

/* ==========================================
   TABLET (768px+)
   ========================================== */
@media (width >= 768px) {
  .nav-list {
    flex-direction: row;
    gap: 1.5rem;
  }

  .grid {
    grid-template-columns: repeat(2, 1fr);
  }
}

/* ==========================================
   DESKTOP (1024px+)
   ========================================== */
@media (width >= 1024px) {
  .page {
    flex-direction: row;
  }

  .sidebar {
    width: 280px;
    flex-shrink: 0;
  }

  .main {
    flex: 1;
  }

  .grid {
    grid-template-columns: repeat(3, 1fr);
  }
}

/* ==========================================
   WIDE DESKTOP (1440px+)
   ========================================== */
@media (width >= 1440px) {
  .page {
    max-width: 1400px;
    margin: 0 auto;
  }

  .grid {
    grid-template-columns: repeat(4, 1fr);
  }
}

/* ==========================================
   E-INK / SLOW DISPLAYS
   ========================================== */
@media (update: slow) {
  * {
    animation-duration: 0s !important;
    transition-duration: 0s !important;
  }
}

/* ==========================================
   PRINT
   ========================================== */
@media print {
  nav, .sidebar, .btn, footer, .no-print {
    display: none !important;
  }

  body {
    background: white !important;
    color: black !important;

  }

  .page {
    display: block;
  }

    font-size: 0.8em;
  }

  h2, h3 { break-after: avoid; }

  @page { margin: 2cm; }
}
```
