# 3.5 Fluid Typography

> Fluid typography scales font sizes (and spacing) smoothly between viewport widths without discrete breakpoint jumps. Instead of snapping from one size to another inside media queries, fluid type interpolates linearly across a defined viewport range. This guide covers the math, the `clamp()` function, modular type scales, accessibility traps, fluid spacing, container query units, and every mistake you will encounter along the way.

---

## The Problem with Breakpoint-Based Typography

Traditional responsive typography uses media queries to set font sizes at discrete widths:

```css
/* Breakpoint-based: sizes jump at each threshold */
h1 {
  font-size: 1.5rem;
}

@media (min-width: 768px) {
  h1 {
    font-size: 2rem;
  }
}

@media (min-width: 1200px) {
  h1 {
    font-size: 3rem;
  }
}
```

This creates abrupt jumps. At 767px the heading is 1.5rem. At 768px it snaps to 2rem. No intermediate sizes exist. Resize the browser and you see a jarring shift.

Fluid typography eliminates this by computing font size as a continuous function of the viewport width.

---

## The Foundation: `clamp()`

The CSS `clamp()` function is the modern tool for fluid typography. It takes three arguments:

```
clamp(MINIMUM, PREFERRED, MAXIMUM)
```

| Argument | Role | Unit recommendation |
|----------|------|---------------------|
| MINIMUM | Floor. Font size never goes below this. | `rem` |
| PREFERRED | The fluid value. Changes with viewport. | `rem` + `vw` (or `vi`) |
| MAXIMUM | Ceiling. Font size never exceeds this. | `rem` |

```css
h1 {
  /* Minimum 2rem, scales fluidly, caps at 4rem */
  font-size: clamp(2rem, 1rem + 3vw, 4rem);
}
```

The browser evaluates the preferred value against the bounds. If the preferred value is between the minimum and maximum, it wins. If it falls below the minimum, the minimum wins. If it exceeds the maximum, the maximum wins.

---

## The Math Behind the Preferred Value

The preferred value in `clamp()` is not a guess. It is derived from the equation of a line.

### The Linear Equation

You want font size to scale linearly from a minimum size at a minimum viewport width to a maximum size at a maximum viewport width. This is a straight line:

```
y = mx + b
```

Where:
- **y** = font size at any given viewport width
- **x** = current viewport width
- **m** = slope (rate of change)
- **b** = y-intercept (base size when viewport is 0)

### Step-by-Step Derivation

Given:
- `minFontSize` = 1rem (16px) at `minViewport` = 320px
- `maxFontSize` = 2rem (32px) at `maxViewport` = 1280px

**Step 1: Calculate the slope**

```
slope = (maxFontSize - minFontSize) / (maxViewport - minViewport)
slope = (32 - 16) / (1280 - 320)
slope = 16 / 960
slope = 0.01667
```

**Step 2: Calculate the y-intercept**

```
yIntercept = minFontSize - (slope * minViewport)
yIntercept = 16 - (0.01667 * 320)
yIntercept = 16 - 5.333
yIntercept = 10.667px
```

**Step 3: Convert to CSS units**

The slope must be expressed in `vw` (1vw = 1% of viewport width), so multiply by 100:

```
slopeVW = 0.01667 * 100 = 1.667vw
```

The y-intercept converts to `rem` by dividing by 16:

```
yInterceptRem = 10.667 / 16 = 0.667rem
```

**Step 4: Assemble the clamp**

```css
h1 {
  font-size: clamp(1rem, 0.667rem + 1.667vw, 2rem);
}
```

### Verification

At 320px viewport: `0.667rem + 1.667vw = 10.667px + (0.01667 * 320)px = 10.667 + 5.333 = 16px = 1rem` (hits minimum).

At 1280px viewport: `0.667rem + 1.667vw = 10.667px + (0.01667 * 1280)px = 10.667 + 21.333 = 32px = 2rem` (hits maximum).

At 800px viewport: `10.667 + (0.01667 * 800) = 10.667 + 13.333 = 24px = 1.5rem` (fluid middle).

### The Generic Formula

```
preferred = (yIntercept / 16)rem + (slope * 100)vw

Where:
  slope = (maxSize - minSize) / (maxViewport - minViewport)
  yIntercept = minSize - slope * minViewport
  (all values in px for calculation, convert output to rem)
```

## Why You Must Never Use `vw` Alone

Setting font size with only viewport units is an accessibility violation.

```css
/* BAD PRACTICE: pure vw for font-size */
h1 {
  font-size: 5vw;
}
```

### The Zoom Problem

Browsers do not scale viewport units when the user zooms. When a user triggers browser zoom:

- `1rem` at 200% zoom renders at 2x size (the `rem` value doubles because the root font size doubles)
- `5vw` at 200% zoom renders at the same size (the viewport has not changed)

WCAG Success Criterion 1.4.4 (Resize Text, Level AA) requires that text can be resized to 200% without loss of content or functionality. Pure `vw` sizing prevents this. The W3C documents this explicitly as **Failure Technique F94**: "Failure of Success Criterion 1.4.4 due to incorrect use of viewport units to resize text."

### The User Preference Problem

Users who set a larger default font size in their browser settings expect `rem`-based text to respect that. Pure `vw` ignores the user's preference entirely.

### The Fix

Always combine `vw` with a relative unit:

```css
/* GOOD: vw + rem allows zoom and respects user preferences */
h1 {
  font-size: clamp(2rem, 1.2rem + 2vw, 3.5rem);
}

/* BAD: pure vw ignores zoom and user settings */
h1 {
  font-size: 5vw;
}

/* BAD: vw without rem anchor in the preferred value */
h1 {
  font-size: clamp(1rem, 5vw, 3rem);
}
```

The third example looks safe because of `rem` bounds, but the preferred value `5vw` has no `rem` component. In the fluid range, zoom has zero effect on the computed size. Add a `rem` anchor to the preferred value so that zoom scales the base, even if `vw` does not respond.

## Modular Type Scales

A modular type scale is a sequence of font sizes generated by repeatedly multiplying a base size by a fixed ratio. This creates mathematically harmonious relationships between heading levels and body text.

### Common Ratios

| Name | Ratio | Character |
|------|-------|-----------|
| Minor Second | 1.067 | Barely perceptible difference |
| Major Second | 1.125 | Subtle, good for dense UIs |
| Minor Third | 1.200 | Moderate, comfortable for most sites |
| Major Third | 1.250 | Clear hierarchy, versatile default |
| Perfect Fourth | 1.333 | Strong contrast, good for editorial |
| Augmented Fourth | 1.414 | Dramatic, content-heavy layouts |
| Perfect Fifth | 1.500 | Bold jumps, use with restraint |
| Golden Ratio | 1.618 | Very large jumps, rarely practical for web |

### Generating a Scale

Starting from a base of `1rem` (16px) with a **Major Third** ratio (1.250):

| Step | Calculation | Size | Use |
|------|-------------|------|-----|
| -2 | 1 / 1.250^2 | 0.64rem | Small print, captions |
| -1 | 1 / 1.250 | 0.8rem | Footnotes, labels |
| 0 | 1.000 | 1rem | Body text |
| 1 | 1 * 1.250 | 1.25rem | Large body, lead text |
| 2 | 1 * 1.250^2 | 1.563rem | h4 / h5 |
| 3 | 1 * 1.250^3 | 1.953rem | h3 |
| 4 | 1 * 1.250^4 | 2.441rem | h2 |
| 5 | 1 * 1.250^5 | 3.052rem | h1 |

### Applying a Different Scale per Viewport

The key insight of fluid typography with modular scales: use a smaller ratio for small screens and a larger ratio for large screens. The base size can also change.

```
Small screen (320px):  base 1rem,   ratio 1.200 (Minor Third)
Large screen (1280px): base 1.125rem, ratio 1.333 (Perfect Fourth)
```

Each step then has a minimum and maximum size. You compute a `clamp()` for each step using the formula above.

```css
:root {
  /* Step -2 */
  --step--2: clamp(0.6944rem, 0.6546rem + 0.1991vw, 0.6328rem);
  /* Step -1 */
  --step--1: clamp(0.8333rem, 0.7963rem + 0.1852vw, 0.8438rem);
  /* Step 0 (body) */
  --step-0: clamp(1rem, 0.9583rem + 0.2083vw, 1.125rem);
  /* Step 1 */
  --step-1: clamp(1.2rem, 1.1417rem + 0.2917vw, 1.5rem);
  /* Step 2 */
  --step-2: clamp(1.44rem, 1.3479rem + 0.4606vw, 2rem);
  /* Step 3 */
  --step-3: clamp(1.728rem, 1.5764rem + 0.7579vw, 2.6667rem);
  /* Step 4 */
  --step-4: clamp(2.0736rem, 1.8229rem + 1.2537vw, 3.5556rem);
  /* Step 5 */
  --step-5: clamp(2.4883rem, 2.0787rem + 2.048vw, 4.7407rem);
}

body { font-size: var(--step-0); }
h1   { font-size: var(--step-5); }
h2   { font-size: var(--step-4); }
h3   { font-size: var(--step-3); }
h4   { font-size: var(--step-2); }
h5   { font-size: var(--step-1); }
small { font-size: var(--step--1); }
```

You do not calculate these by hand. Use a tool (covered below).

## Utopia: The Design System Approach

Utopia (utopia.fyi) is not a framework or library. It is a methodology and a set of calculators for generating fluid type and space scales.

### Core Philosophy

1. Define your design at two viewport widths (e.g., 320px and 1280px)
2. Pick a type scale ratio for each viewport
3. Let the browser interpolate everything in between
4. Think in **steps**, not pixel values ("use Step 3" instead of "use 24px")

### How the Type Calculator Works

You input:
- Minimum viewport width (e.g., 320px)
- Maximum viewport width (e.g., 1280px)
- Base font size at min viewport (e.g., 16px / 1rem)
- Base font size at max viewport (e.g., 18px / 1.125rem)
- Type scale ratio at min (e.g., 1.200 Minor Third)
- Type scale ratio at max (e.g., 1.333 Perfect Fourth)
- Number of steps above and below the base

It outputs CSS custom properties with `clamp()` values, ready to paste into your stylesheet.

## Fluid Spacing with `clamp()`

Fluid typography principles extend directly to spacing. Any property that accepts a length value can use `clamp()`.

### Padding

```css
.hero {
  /* Padding scales from 2rem (at 320px) to 6rem (at 1280px) */
  padding: clamp(2rem, 0.6667rem + 4.1667vw, 6rem);
}
```

### Margin

```css
.section + .section {
  /* Section gap scales from 3rem to 8rem */
  margin-block-start: clamp(3rem, 1.2917rem + 5.2083vw, 8rem);
}
```

### Gap

```css
.grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(min(100%, 20rem), 1fr));
  /* Gap scales from 1rem to 2.5rem */
  gap: clamp(1rem, 0.4875rem + 1.5625vw, 2.5rem);
}
```

### When Fluid Spacing Helps

- Section padding and margins on landing pages
- Grid and flex gaps that feel too tight on mobile or too loose on desktop
- Container max-width with fluid inline padding
- Vertical rhythm between content blocks

### When Fluid Spacing is Overkill

- Small spacing values where the difference between min and max is under 4px
- Component-internal spacing that should be consistent regardless of viewport
- Borders, outlines, and decorative details

## Container Query Units for Fluid Type

Viewport units tie font size to the browser window. Container query units tie font size to the size of a containing element. This is powerful for components that live in variable-width contexts (sidebars, card grids, modals).

### Setup

```css
/* Define the container */
.card {
  container-type: inline-size;
}

/* Fluid type based on container width */
.card h2 {
  font-size: clamp(1rem, 0.5rem + 3cqi, 2rem);
}

.card p {
  font-size: clamp(0.875rem, 0.7rem + 1.5cqi, 1.125rem);
}
```

### Container Query Units Reference

| Unit | Meaning |
|------|---------|
| `cqw` | 1% of the container's width |
| `cqh` | 1% of the container's height |
| `cqi` | 1% of the container's inline size |
| `cqb` | 1% of the container's block size |
| `cqmin` | Smaller of `cqi` and `cqb` |
| `cqmax` | Larger of `cqi` and `cqb` |

Use `cqi` for typography, as you want size to respond to the inline (horizontal in LTR) axis.

### Advantages Over Viewport Units

1. **Component portability**: A card component renders correctly whether placed in a full-width area or a narrow sidebar.
2. **Better zoom behavior**: `cqi` continues to increase as the container grows during zoom, unlike `vw` which stays fixed. When the user zooms, the layout adjusts, containers may reflow, and `cqi` responds to those changes.
3. **Nested layouts**: In complex layouts where components are nested inside flexible containers, `vw` gives you the global viewport size, but `cqi` gives you the local context.

### The Caveat

Container query units still need `rem` anchoring for accessibility, just like `vw`:

```css
/* GOOD: rem anchor + cqi */
.card p {
  font-size: clamp(1rem, 0.75rem + 2cqi, 1.5rem);
}

/* BAD: pure cqi */
.card p {
  font-size: 4cqi;
}
```

Always test zoom behavior. Container query units behave better than `vw` under zoom, but edge cases exist depending on how the container itself responds to zoom.

### Browser Support

Container query units are Baseline 2023 (widely available). All modern browsers support them. No polyfills needed for current projects.

## `text-size-adjust`: Preventing Mobile Text Inflation

Mobile browsers apply a **text inflation algorithm** that automatically enlarges text on pages designed for desktop. When a page is laid out at a wide viewport (800-1000px) and scaled down to fit a mobile screen, small text becomes unreadable, so the browser inflates it. This can interfere with fluid typography that already handles sizing.

### The Property

```css
/* Prevent mobile text inflation */
html {
  -webkit-text-size-adjust: 100%;
     -moz-text-size-adjust: 100%;
          text-size-adjust: 100%;
}
```

### Values

| Value | Effect |
|-------|--------|
| `auto` | Browser decides (default). Inflation may occur. |
| `none` | Disables inflation entirely. **Do not use.** |
| `100%` | Tells the browser the text is already sized correctly. No inflation. |

### Why `100%` and Not `none`

Setting `none` on older WebKit desktop browsers prevented users from zooming the page entirely. Using `100%` achieves the same practical effect (no inflation) without breaking zoom.

```css
/* BAD PRACTICE: disables zoom on older WebKit */
html {
  -webkit-text-size-adjust: none;
}

/* GOOD PRACTICE: prevents inflation without breaking zoom */
html {
  -webkit-text-size-adjust: 100%;
}
```

### When You Need It

Include `text-size-adjust: 100%` in your CSS reset if:
- You are using fluid typography (you are already sizing text correctly)
- You have a responsive layout with a proper viewport meta tag
- You are seeing unexpected font-size differences between portrait and landscape on iOS

Every major CSS reset (Normalize.css, modern-normalize, Andy Bell's reset) includes this declaration.

## Accessibility Deep Dive

Fluid typography introduces specific accessibility concerns that static sizing does not.

### WCAG 1.4.4: Resize Text (Level AA)

The requirement: text must be resizable up to 200% without loss of content or functionality. This means at the browser's maximum zoom (typically 500%), text should be at least 200% of its original size.

### How Zoom Interacts with `clamp()`

When a user zooms to 200%:
- `rem` values double (the root font size doubles)
- `vw` values stay the same (the viewport has not changed)
- The computed value of `0.5rem + 1.5vw` increases, but not by the full 200%

This means the **rem component** of your preferred value responds to zoom, but the **vw component** does not. The larger the `vw` portion relative to `rem`, the less zoom-responsive the text is.

### Testing for Compliance

Test at these zoom levels and viewport widths:

```
1. Set browser to 1280px wide
2. Zoom to 400% (this simulates a 320px-wide viewport at 100% zoom)
3. Check: is the text at least 200% of its original size at 1280px/100%?
4. Repeat for your actual min and max viewport breakpoints
```

### The Danger Zone

A `clamp()` declaration fails WCAG 1.4.4 when:
1. The fluid range is very wide (e.g., 320px to 2560px)
2. The size difference between min and max is large (e.g., 1rem to 5rem)
3. The `vw` component dominates the preferred value

```css
/* RISKY: large vw coefficient, small rem anchor */
h1 {
  font-size: clamp(1rem, 0.2rem + 5vw, 5rem);
}
/*
  At 1280px/100%: 0.2rem + 5vw = 3.2px + 64px = 67.2px
  At 1280px/200% zoom: 0.4rem + 5vw = 6.4px + 64px = 70.4px
  Increase: only 4.8% (needs to be 100% to hit 200% of original)
  FAILS WCAG 1.4.4
*/

/* SAFER: balanced rem + vw ratio */
h1 {
  font-size: clamp(1.5rem, 1rem + 1.5vw, 3rem);
}
/*
  At 1280px/100%: 1rem + 1.5vw = 16px + 19.2px = 35.2px
  At 1280px/200% zoom: 2rem + 1.5vw = 32px + 19.2px = 51.2px
  Increase: 45% (not 100%, but the min clamp of 3rem = 48px at 200%
  zoom becomes 96px via rem doubling, which IS 200% of 48px)
  The rem-based bounds save you.
*/
```

### The Safety Net: `rem` Bounds

When the user zooms, the `rem` minimum and maximum values in your `clamp()` also double. If zoom pushes the preferred value below the minimum, the minimum takes over, and that minimum responds to zoom correctly. This is why `rem` bounds are critical:

```css
/* The min value of 1.5rem becomes 3rem at 200% zoom */
/* This ensures text always reaches at least 200% at full zoom */
font-size: clamp(1.5rem, 1rem + 1.5vw, 3rem);
```

### Practical Accessibility Checklist

| Check | How |
|-------|-----|
| Text scales to 200% at 200% zoom | Zoom browser to 200%, verify all text doubled |
| Text does not overflow containers | Zoom to 200%, check for clipping or overlap |
| User font-size preference respected | Change browser default font size to 24px, verify text enlarges |
| No horizontal scroll at 400% zoom | Zoom to 400% at 1280px, verify reflow |
| Line length stays readable | At max viewport, verify lines do not exceed ~80 characters |

## Common Mistakes

### Mistake 1: Using `px` for `clamp()` Bounds

```css
/* BAD: px bounds ignore user font-size preferences */
h1 {
  font-size: clamp(24px, 1rem + 2vw, 48px);
}

/* GOOD: rem bounds respect user preferences and zoom */
h1 {
  font-size: clamp(1.5rem, 1rem + 2vw, 3rem);
}
```

When a user sets their browser's default font size to 20px, `1.5rem` becomes 30px. But `24px` stays 24px.

### Mistake 2: Assuming 1rem = 16px

Many fluid typography calculators assume the browser default of `1rem = 16px`. This is the default, but users can change it. If you hard-code pixel assumptions into your system, it breaks for users with non-default settings.

```css
/* The formula was derived assuming 16px root */
/* If the user's root is 20px, the breakpoints shift */
font-size: clamp(1rem, 0.667rem + 1.667vw, 2rem);
/* This still WORKS because rem is relative. */
/* The issue is when you mentally map "1rem = 16px" and */
/* make layout decisions based on that assumed pixel value */
```

The `clamp()` itself is fine because `rem` is relative. The mistake happens when you design your layout assuming exact pixel equivalents.

### Mistake 3: Not Including `rem` in the Preferred Value

```css
/* BAD: preferred value is pure vw */
p {
  font-size: clamp(1rem, 2.5vw, 1.5rem);
}

/* GOOD: preferred value includes rem anchor */
p {
  font-size: clamp(1rem, 0.85rem + 0.75vw, 1.5rem);
}
```

Without a `rem` component in the preferred value, zoom only affects the text when it hits the `rem` bounds. Within the fluid range, zoom has no effect.

### Mistake 4: Making Body Text Fluid

Body text rarely needs much fluid scaling. The difference between body text at 320px and 1280px is often just 1-3px. This tiny range makes fluid typography unnecessary, and it introduces accessibility risk for minimal visual benefit.

```css
/* OVERKILL: body text does not need to be fluid */
body {
  font-size: clamp(0.875rem, 0.8rem + 0.25vw, 1.125rem);
}

/* SIMPLER: just use rem, maybe one breakpoint */
body {
  font-size: 1rem;
}
```

Fluid typography shines for **headings** and **display text** where the size difference between small and large screens is substantial.

### Mistake 5: Forgetting the Safari `min-height: 0vw` Fix

Older versions of Safari had a bug where `clamp()` with `vw` units did not recalculate on viewport resize. The fix:

```css
/* Safari workaround for live resize recalculation */
body {
  min-height: 0vw; /* forces Safari to track viewport changes */
}
```

This is increasingly unnecessary as Safari updates, but worth knowing if you see stale font sizes in Safari during resize.

### Mistake 6: Fluid Type Everywhere

Not every text element benefits from fluid sizing. Use fluid typography selectively:

| Element | Fluid? | Reasoning |
|---------|--------|-----------|
| h1 | Yes | Large size difference between mobile and desktop |
| h2-h3 | Usually | Moderate size difference, scales well |
| h4-h6 | Rarely | Small size difference, not worth complexity |
| Body text | Rarely | 1-2px difference, use a single `rem` value |
| Captions | No | Too small to matter |
| Navigation | No | Fixed component, consistency matters more |
| Buttons | No | Touch target size should not shrink on mobile |

### Mistake 7: Ignoring Line Height and Measure

Fluid font size without fluid line height or max-width creates readability problems:

```css
h1 {
  font-size: clamp(2rem, 1.5rem + 1.5vw, 3.5rem);
  /* Line height should also adjust */
  line-height: clamp(1.1, 1.05 + 0.15vi, 1.3);
}

.content {
  font-size: var(--step-0);
  /* Constrain line length for readability */
  max-width: 65ch;
  /* Or use clamp for the container */
  max-width: clamp(45ch, 90%, 75ch);
}
```

As font size increases, line height typically needs to **decrease** (proportionally), and line length must be constrained. The optimal reading width is 45-75 characters per line.

## Building a Complete Fluid Type System

Here is a production-ready example combining fluid type scale, fluid spacing, and proper defaults.

### Step 1: CSS Reset Additions

```css
html {
  /* Prevent mobile text inflation */
  -webkit-text-size-adjust: 100%;
     -moz-text-size-adjust: 100%;
          text-size-adjust: 100%;
}

*,
*::before,
*::after {
  box-sizing: border-box;
}
```

### Step 2: Fluid Type Scale (Custom Properties)

```css
:root {
  /*
    Type scale generated for:
    Min viewport: 320px  | Base: 16px (1rem)    | Scale: 1.2 (Minor Third)
    Max viewport: 1280px | Base: 18px (1.125rem) | Scale: 1.333 (Perfect Fourth)
  */
  --font-size--2: clamp(0.6944rem, 0.6508rem + 0.2181vw, 0.8438rem);
  --font-size--1: clamp(0.8333rem, 0.7593rem + 0.3704vw, 1.125rem);
```

## Using the `vi` Unit Instead of `vw`

The `vi` unit (1% of the viewport inline size) is the logical-property equivalent of `vw`. In horizontal writing modes (LTR/RTL), `vi` equals `vw`. In vertical writing modes, `vi` maps to `vh`.

```css
/* Logical approach: works correctly in vertical writing modes */
h1 {
  font-size: clamp(2rem, 1rem + 3vi, 4rem);
}

/* Physical approach: always maps to viewport width */
h1 {
  font-size: clamp(2rem, 1rem + 3vw, 4rem);
}
```

For most projects (English, horizontal LTR), `vw` and `vi` are interchangeable. If you support vertical writing modes (CJK languages), use `vi`.

Browser support for `vi` is Baseline 2022, meaning all modern browsers support it.

## The OddBird Perspective: Rethinking Assumptions

Miriam Suzanne (OddBird) challenged common fluid typography assumptions in 2025. Key insights:

### Stop Thinking in Pixel Equivalents

Most developers think `1rem = 16px` and build their entire system on that equation. But `rem` is relative to the user's root font size, which may not be 16px. Building systems that work regardless of the root size is more robust.

### Start with `em` on the Root

```css
/* Let the root font-size be relative to user preference */
html {
  font-size: 1em; /* equals the user's chosen default */
}
```

Do not set `html { font-size: 16px; }` or `html { font-size: 62.5%; }`. Both override the user's preference. Using `1em` keeps the relationship intact.

### Fluid Typography is a Trade-Off

Fluid typography makes zoom less effective by design. You are deliberately tying font size to the viewport, which conflicts with the user's expectation that zoom enlarges everything proportionally. This is a trade-off worth acknowledging, not a bug to fix.

The recommendation: keep fluid ranges narrow, keep `rem` components large relative to `vw`, and test zoom thoroughly.

## Advanced: Fluid Line Height

Line height can also be fluid. Larger text typically needs tighter line height, smaller text needs looser. A `clamp()` on `line-height` with unitless values provides this:

```css
h1 {
  font-size: clamp(2rem, 1.5rem + 1.5vw, 3.5rem);
  /* Tighter line-height as size increases */
  line-height: clamp(1.1, 1rem + 0.2vi, 1.3);
}

p {
  font-size: var(--font-size-0);
  /* Slightly fluid line-height for body */
  line-height: clamp(1.5, 1.4rem + 0.1vi, 1.7);
}
```

Be careful: `line-height` with mixed units (unitless + viewport units inside `clamp`) can produce unexpected results. Test visually. Many teams keep `line-height` as a static unitless value per heading level and skip fluid line height entirely.

## Quick Reference: The Formulas

### Single Value Clamp

Given min size, max size, min viewport, max viewport (all in px):

```
slope = (maxSize - minSize) / (maxViewport - minViewport)
yIntercept = minSize - slope * minViewport
slopeVW = slope * 100
yInterceptRem = yIntercept / 16

clamp(minSizeRem, yInterceptRem + slopeVW vw, maxSizeRem)
```

### Sass Function for Fluid Values

```scss
/// Generate a clamp() value for fluid sizing
/// @param {Number} $min-size - Minimum size in px
/// @param {Number} $max-size - Maximum size in px
/// @param {Number} $min-vw - Minimum viewport width in px (default: 320)
/// @param {Number} $max-vw - Maximum viewport width in px (default: 1280)
/// @return {String} - CSS clamp() value
@function fluid($min-size, $max-size, $min-vw: 320, $max-vw: 1280) {
  $slope: ($max-size - $min-size) / ($max-vw - $min-vw);
  $y-intercept: $min-size - $slope * $min-vw;
  $slope-vw: $slope * 100;
  $y-intercept-rem: $y-intercept / 16;
  $min-rem: $min-size / 16;
  $max-rem: $max-size / 16;

  @return clamp(
    #{$min-rem}rem,
    #{$y-intercept-rem}rem + #{$slope-vw}vw,
    #{$max-rem}rem
  );
}

/* Usage */
h1 {
  font-size: fluid(24, 48); /* 24px at 320vw, 48px at 1280vw */
}

.section {
  padding: fluid(32, 96); /* 32px at 320vw, 96px at 1280vw */
}
```

### JavaScript Calculation (for build tools)

```js
/**
 * Generate a CSS clamp() value for fluid sizing.
 * @param {number} minSize  - Min size in px
 * @param {number} maxSize  - Max size in px
 * @param {number} minVw    - Min viewport in px (default 320)
 * @param {number} maxVw    - Max viewport in px (default 1280)
 * @param {number} rootSize - Root font size in px (default 16)
 * @returns {string} CSS clamp() declaration
 */
function fluidClamp(minSize, maxSize, minVw = 320, maxVw = 1280, rootSize = 16) {
  const slope = (maxSize - minSize) / (maxVw - minVw);
  const yIntercept = minSize - slope * minVw;
  const slopeVw = +(slope * 100).toFixed(4);
  const yInterceptRem = +(yIntercept / rootSize).toFixed(4);
  const minRem = +(minSize / rootSize).toFixed(4);
  const maxRem = +(maxSize / rootSize).toFixed(4);

  return `clamp(${minRem}rem, ${yInterceptRem}rem + ${slopeVw}vw, ${maxRem}rem)`;
}

// Example
console.log(fluidClamp(16, 32));
// "clamp(1rem, 0.6667rem + 1.6667vw, 2rem)"

console.log(fluidClamp(24, 64));
// "clamp(1.5rem, 0.6667rem + 4.1667vw, 4rem)"
```

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| `clamp(min, preferred, max)` | The preferred value mixes `rem` + `vw` for fluid scaling with zoom support |
| Formula | `slope = (maxSize - minSize) / (maxVw - minVw)`, `intercept = minSize - slope * minVw` |
| Accessibility | Always use `rem` for bounds AND in the preferred value. Test 200% zoom. |
| Pure `vw` | Never use alone for font-size. Fails WCAG 1.4.4 (F94 failure). |
| Modular scales | Pick two ratios (small screen, large screen). Generate steps with a tool. |
| Spacing | Same `clamp()` technique applies to padding, margin, and gap. |
| Container units | Use `cqi` instead of `vw` for component-level fluid sizing. |
| `text-size-adjust` | Set to `100%` in your reset. Never use `none`. |
| Tools | Use Utopia, Fluid Type Scale Calculator, or a Sass function. |
| Trade-off | Fluid type makes zoom less effective. Keep fluid ranges narrow. Test thoroughly. |
