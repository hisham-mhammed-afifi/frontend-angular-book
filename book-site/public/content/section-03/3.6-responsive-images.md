# 3.6 Responsive Images

> Responsive images serve appropriately sized and formatted files to each device, preventing wasted bandwidth on mobile and blurry images on retina displays. Getting this wrong is one of the most common causes of poor page performance.

---

## The Core Problem

A single 2000px-wide JPEG hero image might be perfect on a 4K desktop monitor. On a 375px phone, it wastes 80%+ of the downloaded bytes. On a 1x-DPR laptop, it downloads 4x the pixels needed. The solution: let the browser choose the right image from a set of candidates.

HTML provides two mechanisms:
- **`srcset` + `sizes`** on `<img>` for resolution switching (same image, different sizes).
- **`<picture>` + `<source>`** for art direction (different crops/compositions) and format switching (AVIF/WebP/JPEG fallback).

---

## srcset with Width Descriptors (w)

Use `w` descriptors when the image's rendered size changes with the viewport. Each candidate is tagged with its intrinsic pixel width.

```html
<img
  src="hero-800.jpg"
  srcset="
    hero-400.jpg   400w,
    hero-800.jpg   800w,
    hero-1200.jpg 1200w,
    hero-1600.jpg 1600w,
    hero-2000.jpg 2000w
  "
  sizes="(min-width: 1200px) 1100px,
         (min-width: 768px) 90vw,
         calc(100vw - 2rem)"
  width="2000"
  height="1000"
  alt="Mountain landscape"
>
```

### How the Browser Chooses

1. Parse `sizes` left-to-right. Use the length from the first matching media condition.
2. Resolve that length to CSS pixels (e.g., `90vw` on a 1000px viewport = 900 CSS pixels).
3. Multiply by the device pixel ratio (900px on a 2x screen = 1800 device pixels needed).
4. Pick the smallest `srcset` candidate whose `w` value is >= 1800. That is `hero-2000.jpg`.

The browser makes this choice **before layout**. It uses `sizes` as its only hint about the image's display width. If `sizes` is wrong, the browser downloads the wrong image.

### The sizes Attribute

```
sizes="(media-condition) length, (media-condition) length, default-length"
```

Valid units: `px`, `em`, `rem`, `vw`, `vh`, and `calc()` expressions. The last entry (no condition) is the fallback.

```html
sizes="(min-width: 1400px) 700px,
       (min-width: 1000px) 50vw,
       (min-width: 600px) calc(100vw - 4rem),
       calc(100vw - 2rem)"
```

If you omit `sizes` entirely with `w` descriptors, the browser assumes `sizes="100vw"`. On a 1920px desktop with DPR 2, it requests a ~3840w image even if the image only fills a sidebar. This is the most common responsive images mistake.

## srcset with Density Descriptors (x)

Use `x` descriptors when the image has a fixed CSS size across all viewports. The browser picks the file matching the device pixel ratio.

```html
<img
  src="logo-1x.png"
  srcset="
    logo-1x.png 1x,
    logo-2x.png 2x,
    logo-3x.png 3x
  "
  width="200"
  height="60"
  alt="Company logo"
>
```

- DPR 1 device loads `logo-1x.png` (200px file).
- DPR 2 device loads `logo-2x.png` (400px file).
- DPR 3 device loads `logo-3x.png` (600px file).

No `sizes` attribute is needed with `x` descriptors.

### w vs x Descriptors

| Feature | `w` descriptors | `x` descriptors |
|---------|----------------|-----------------|
| Use case | Fluid/responsive images | Fixed-size images |
| Requires `sizes` | Yes | No |
| Browser considers viewport | Yes | No |
| Common for | Hero images, content images, cards | Logos, avatars, icons |

**You cannot mix `w` and `x` descriptors in the same `srcset`.** The browser ignores the entire attribute if you do.

## The picture Element

`<picture>` wraps `<source>` elements and a required `<img>` fallback. It gives you explicit control over which image the browser loads.

### Art Direction (Different Crops Per Viewport)

```html
<picture>
  <!-- Mobile: tight portrait crop -->
  <source
    media="(max-width: 599px)"
    srcset="hero-portrait-400.jpg 400w,
            hero-portrait-800.jpg 800w"
    sizes="100vw"
  >
  <!-- Tablet: square crop -->
  <source
    media="(max-width: 1199px)"
    srcset="hero-square-600.jpg 600w,
            hero-square-1200.jpg 1200w"
    sizes="90vw"
  >
  <!-- Desktop: full landscape -->
  <source
    srcset="hero-landscape-1200.jpg 1200w,
            hero-landscape-2000.jpg 2000w"
    sizes="1100px"
  >
  <img src="hero-landscape-1200.jpg" alt="Person rowing on a lake" width="2000" height="1000">
</picture>
```

The browser stops at the first `<source>` whose `media` matches. Order matters: put the most specific (narrowest) conditions first.

### Format Switching

```html
<picture>
  <source type="image/avif" srcset="photo.avif">
  <source type="image/webp" srcset="photo.webp">
  <img src="photo.jpg" alt="Portrait" width="800" height="600">
</picture>
```

The `type` attribute specifies the MIME type. If the browser does not support that format, it skips the `<source>`. Order: most efficient format first (AVIF), then WebP, then JPEG fallback in the `<img>`.

### Combined Art Direction + Format Switching

```html
<picture>
  <!-- Mobile: AVIF, then WebP, then JPEG -->
  <source media="(max-width: 767px)" type="image/avif" srcset="hero-mobile.avif">
  <source media="(max-width: 767px)" type="image/webp" srcset="hero-mobile.webp">
  <source media="(max-width: 767px)" srcset="hero-mobile.jpg">

  <!-- Desktop: AVIF, then WebP, then JPEG -->
  <source type="image/avif" srcset="hero-desktop.avif">
  <source type="image/webp" srcset="hero-desktop.webp">

  <img src="hero-desktop.jpg" alt="Hero" width="1600" height="900">
</picture>
```

### picture vs srcset: When to Use Which

| Scenario | Use |
|----------|-----|
| Same image, different resolutions | `srcset` + `sizes` on `<img>` |
| Different crops/compositions per viewport | `<picture>` with `media` on `<source>` |
| Serve modern formats with fallbacks | `<picture>` with `type` on `<source>` |
| Both art direction AND format switching | `<picture>` with both `media` and `type` |

Key difference: `srcset` is a suggestion. The browser may pick any candidate based on conditions you cannot control (cache, network). `<source media="...">` is a command. The browser selects the first matching source unconditionally.

## Image Formats: AVIF, WebP, JPEG, PNG

| Feature | JPEG | PNG | WebP | AVIF |
|---------|------|-----|------|------|
| Lossy compression | Yes | No | Yes | Yes |
| Lossless compression | No | Yes | Yes | Yes |
| Transparency | No | Yes | Yes | Yes |
| Animation | No | APNG | Yes | Yes |
| HDR / wide gamut | No | No | Limited | Yes |
| Savings vs JPEG | baseline | n/a | ~30% | ~50% |
| Encoding speed | Fast | Fast | Fast | Slow (3-10x WebP) |
| Browser support (2025) | ~100% | ~100% | ~96% | ~94% |

**When to use each:**
- **AVIF**: First choice for photographs. Best compression. Use as the first `<source>` in `<picture>`.
- **WebP**: Safe modern default. Broader support than AVIF. Good balance of quality and size.
- **JPEG**: Universal fallback. Use as the `<img src>`.
- **PNG**: Only when lossless + transparency is required and WebP/AVIF lossless are not options.
- **SVG**: Icons, logos, illustrations. Resolution-independent.

AVIF encoding is significantly slower. Plan for longer build times or use CDN-side conversion (Cloudinary, imgix, Cloudflare).

## object-fit and object-position

These CSS properties control how an image fills its container.

### object-fit Values

```css
.fill    { object-fit: fill; }       /* Stretch. Ignore aspect ratio. Default. */
.contain { object-fit: contain; }    /* Fit inside. Letterbox. Preserve ratio. */
.cover   { object-fit: cover; }      /* Fill box. Crop overflow. Preserve ratio. */
.none    { object-fit: none; }       /* Use intrinsic size. Ignore container. */
.scale   { object-fit: scale-down; } /* contain or none, whichever is smaller. */
```

### object-position

Controls the alignment of the image within its box when `object-fit` causes cropping or letterboxing.

```css
/* Portrait photos: anchor to top so faces are visible */
.portrait {
  width: 100%;
  height: 300px;
  object-fit: cover;
  object-position: top center;
}

/* Custom focal point */
.hero-img {
  object-fit: cover;
  object-position: 30% 20%;
}
```

Default is `50% 50%` (centered).

### Card Image Pattern

```css
.card-image {
  width: 100%;
  height: 200px;
  object-fit: cover;
  object-position: center;
  border-radius: 8px 8px 0 0;
}
```

`object-fit` requires the image to have defined dimensions. If the `<img>` has no width/height (CSS or HTML), `object-fit` has no effect because the box matches the intrinsic image size.

## Loading, Priority, and Decoding

### loading Attribute

```html
<img src="hero.jpg" loading="eager" alt="...">    <!-- Default: load now -->
<img src="below.jpg" loading="lazy" alt="...">    <!-- Defer until near viewport -->
```

`lazy` defers the network request until the image is within a browser-defined distance of the viewport (typically 1250-2500px).

### fetchpriority Attribute

```html
<img src="hero.jpg" fetchpriority="high" alt="...">      <!-- Critical: boost priority -->
<img src="carousel-3.jpg" fetchpriority="low" alt="...">  <!-- Non-critical: lower priority -->
```

Use `high` for the LCP image. Use `low` for offscreen carousel slides or decorative images.

### decoding Attribute

```html
<img src="photo.jpg" decoding="async" alt="...">  <!-- Decode off main thread -->
<img src="photo.jpg" decoding="sync" alt="...">   <!-- Decode synchronously -->
```

`async` is good for below-the-fold images. For the LCP image, use `sync` or omit the attribute to avoid a visible pop-in.

### Recommended Combinations

| Image Position | loading | fetchpriority | decoding |
|----------------|---------|---------------|----------|
| LCP / hero (above fold) | omit (eager) | `high` | `sync` or omit |
| Above fold, not LCP | omit (eager) | omit | `async` |
| Below fold | `lazy` | omit or `low` | `async` |

**Never combine `loading="lazy"` with `fetchpriority="high"`.** These send contradictory signals.

## Preventing Layout Shift (CLS)

### The Problem

When an image loads without reserved space, content below it shifts down. This is Cumulative Layout Shift. The "good" threshold is CLS < 0.1.

### Solution: HTML width and height Attributes

Modern browsers compute `aspect-ratio` from HTML `width` and `height` attributes:

```html
<img src="photo.jpg" width="1600" height="900" alt="Landscape">
```

```css
img {
  max-width: 100%;
  height: auto;
}
```

The browser's UA stylesheet now includes:
```css
img[width][height] {
  aspect-ratio: attr(width) / attr(height);
}
```

This reserves the correct aspect ratio space before the image loads. The CSS `height: auto` overrides the HTML `height` attribute for rendering, but the browser still uses the ratio for space reservation.

### CSS aspect-ratio Property

For enforcing a specific design ratio regardless of the image's natural proportions:

```css
.hero-image {
  width: 100%;
  aspect-ratio: 16 / 9;
  object-fit: cover;
}
```

The `auto` keyword preserves natural proportions once loaded:

```css
img {
  aspect-ratio: auto 16 / 9; /* Use 16:9 as placeholder, natural ratio once loaded */
}
```

Set HTML `width` and `height` to the intrinsic resolution of the image file, not the CSS display size. These are hints for the browser's aspect ratio calculation.

## Preloading the LCP Image

For hero images that are the Largest Contentful Paint element, preload with responsive hints:

```html
<head>
  <link
    rel="preload"
    as="image"
    href="hero-1600.jpg"
    imagesrcset="hero-400.jpg 400w,
                 hero-800.jpg 800w,
                 hero-1200.jpg 1200w,
                 hero-1600.jpg 1600w"
    imagesizes="(min-width: 1200px) 1100px,
                (min-width: 768px) 90vw,
                calc(100vw - 2rem)"
    fetchpriority="high"
  >
</head>
```

The `imagesrcset` and `imagesizes` attributes mirror the `<img>` element's `srcset` and `sizes`. The browser uses the same selection algorithm to preload the correct variant.

Do not use CSS `background-image` for LCP elements. The browser cannot discover `background-image` URLs until CSS is parsed and the element is laid out. An `<img>` in HTML is discoverable by the preload scanner during HTML parsing.

## Responsive Background Images with image-set()

`image-set()` is the `srcset` equivalent for CSS `background-image`.

### Resolution-Based

```css
.hero {
  background-image: image-set(
    url("bg-1x.jpg") 1x,
    url("bg-2x.jpg") 2x,
    url("bg-3x.jpg") 3x
  );
  background-size: cover;
}
```

### Format-Based

```css
.hero {
  background-image: image-set(
    url("bg.avif") type("image/avif"),
    url("bg.webp") type("image/webp"),
    url("bg.jpg") type("image/jpeg")
  );
}
```

`image-set()` only supports density descriptors (`1x`, `2x`), not width descriptors. For viewport-based switching, use media queries:

```css
.hero { background-image: url("bg-small.jpg"); }

@media (min-width: 768px) {
  .hero { background-image: url("bg-medium.jpg"); }
}

@media (min-width: 1200px) {
  .hero { background-image: url("bg-large.jpg"); }
}
```

Safari may still need the `-webkit-image-set()` prefix for older versions.

## Responsive SVG

SVGs are resolution-independent, but responsive behavior requires the `viewBox` attribute.

### Making SVG Fluid

Remove `width` and `height` attributes. Keep `viewBox`.

```html
<svg viewBox="0 0 200 100" xmlns="http://www.w3.org/2000/svg">
  <circle cx="100" cy="50" r="40" fill="blue" />
</svg>
```

```css
svg {
  width: 100%;
  height: auto;
}
```

Without `viewBox`, a dimensionless SVG collapses to 0 height or defaults to 300x150.

### preserveAspectRatio

Controls how the SVG content fits its container when aspect ratios differ.

```html
<!-- Center and fit inside (like object-fit: contain) -->
<svg viewBox="0 0 400 200" preserveAspectRatio="xMidYMid meet">

<!-- Center and fill/crop (like object-fit: cover) -->
<svg viewBox="0 0 400 200" preserveAspectRatio="xMidYMid slice">

<!-- Stretch to fill (like object-fit: fill) -->
<svg viewBox="0 0 400 200" preserveAspectRatio="none">
```

### The currentColor Trick

```html
<svg viewBox="0 0 24 24" width="24" height="24" aria-label="Settings icon">
  <path d="M12 15.5A3.5 3.5 0 ..." fill="currentColor" />
</svg>
```

`fill="currentColor"` makes the SVG inherit the parent element's CSS `color`. The icon automatically matches text color across themes.

## content-visibility for Offscreen Images

`content-visibility: auto` tells the browser to skip rendering offscreen content until it approaches the viewport.

```css
.article-section {
  content-visibility: auto;
  contain-intrinsic-size: auto 800px;
}
```

`contain-intrinsic-size` provides a placeholder height to prevent scroll jumps. The `auto` keyword means: use the placeholder initially, but remember the actual size once rendered.

**`content-visibility` does NOT prevent image downloads.** It only skips rendering. To defer downloads, you still need `loading="lazy"` on the images. The two are complementary:
- `content-visibility: auto` saves rendering cost (CPU).
- `loading="lazy"` saves network cost (bandwidth).

Do not apply `content-visibility: auto` to elements containing the LCP image. It delays rendering and hurts LCP.

## Common Mistakes

- **Omitting `sizes` with `w` descriptors.** The browser assumes `100vw` and downloads images far larger than needed. This is the single most impactful responsive images mistake.

- **Inaccurate `sizes` values.** If `sizes` says `100vw` but the image occupies a 50% column, the browser downloads a 2x oversized image. Audit with RespImageLint or Lighthouse.

- **Using `loading="lazy"` on the LCP image.** This directly delays the Largest Contentful Paint. Use `fetchpriority="high"` instead.

- **Missing `width` and `height` attributes.** Without them, the browser cannot reserve space, causing layout shift (CLS) as images load.

- **Not providing format fallbacks.** Serving only AVIF means ~6% of users see nothing. Always include WebP and JPEG fallbacks via `<picture>`.

- **Placing `<source>` after `<img>` in `<picture>`.** Sources must precede the `<img>`. If placed after, they are ignored.

- **Using CSS `background-image` for the hero.** Background images are not discoverable by the preload scanner. Use `<img>` for any LCP-critical image.

- **Mixing `x` and `w` descriptors.** The entire `srcset` is silently ignored.

- **Setting `aspect-ratio` without `object-fit`.** Non-matching aspect ratios distort the image. Use `object-fit: cover` to crop gracefully.

---

## Best Practices

- **Always provide `srcset`, `sizes`, `width`, and `height`** on content images. This gives the browser everything it needs to select the right size and reserve space.

- **Use `<picture>` with `type` for format fallbacks.** Order: AVIF first, then WebP, then JPEG in the `<img>` fallback.

- **Mark the LCP image with `fetchpriority="high"`** and preload it in `<head>` with `imagesrcset` and `imagesizes`.

- **Use `loading="lazy"` on every below-the-fold image.** This is free performance.

- **Set a CSS baseline for all images:**

```css
img {
  max-width: 100%;
  height: auto;
  display: block;
}
```

- **Generate multiple sizes at build time.** Common widths: 400, 800, 1200, 1600, 2000px. Most build tools (Vite, webpack) have image optimization plugins.

- **Use CDN-based image optimization** (Cloudinary, imgix, Cloudflare Images) for automatic format conversion, resizing, and quality optimization.

- **Audit with Lighthouse.** Check for "Properly size images", "Serve images in next-gen formats", and "Image elements do not have explicit width and height."

    imagesizes="100vw"

  >

<body>

    <source

---

> **Key Takeaway:** Responsive images require two things: telling the browser what sizes you have (`srcset`) and how big the image will render (`sizes`). Get `sizes` wrong, and the browser downloads the wrong file. Get format fallbacks wrong, and users either see nothing or download bloated files. Always set `width`/`height` for CLS prevention, use `loading="lazy"` below the fold, and use `fetchpriority="high"` on the LCP image.
