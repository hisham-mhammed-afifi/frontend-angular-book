# 4.1 SASS vs SCSS Syntax

> Sass has two syntaxes: the indented syntax (.sass) and SCSS (.scss). They compile to the same CSS, but differ in how you write them. Understanding both is essential because you will encounter both in legacy codebases, but SCSS is the standard for new projects.

---

## The Two Syntaxes

Sass (Syntactically Awesome Style Sheets) offers two distinct syntaxes that compile to identical CSS output:

| Feature | SCSS (.scss) | Indented SASS (.sass) |
|---|---|---|
| Braces | Yes `{ }` | No (uses indentation) |
| Semicolons | Yes `;` | No (uses newlines) |
| CSS superset | Yes (any valid CSS is valid SCSS) | No (CSS must be converted) |
| File extension | `.scss` | `.sass` |
| Introduced | Sass v3 (2010) | Original syntax (2006) |
| Ecosystem adoption | Dominant | Niche |

The critical distinction: SCSS is a **superset of CSS**. Every valid CSS file is already valid SCSS. Indented SASS is not. This single fact drove the ecosystem's overwhelming adoption of SCSS.

## Side-by-Side Syntax Comparison

### Variables

```scss
// SCSS (.scss)
$font-stack: 'Helvetica Neue', Arial, sans-serif;
$primary-color: #3498db;

body {
  font-family: $font-stack;
  color: $primary-color;
}
```

```sass
// Indented SASS (.sass)
$font-stack: 'Helvetica Neue', Arial, sans-serif
$primary-color: #3498db

body
  font-family: $font-stack
  color: $primary-color
```

Variables work identically in both syntaxes. The difference is structural: no braces, no semicolons in `.sass`.

### Nesting

```scss
// SCSS
nav {
  background: #222;

  ul {
    margin: 0;
    padding: 0;
    list-style: none;

    li {
      display: inline-block;

      a {
        color: white;
        text-decoration: none;

        &:hover {
          color: #3498db;
        }
      }
    }
  }
}
```

```sass
// Indented SASS
nav
  background: #222

  ul
    margin: 0
    padding: 0
    list-style: none

    li
      display: inline-block

      a
        color: white
        text-decoration: none

        &:hover
          color: #3498db
```

Both produce the same CSS. The `&` parent selector works identically in both.

### Mixins

This is where the syntactic divergence is largest. Indented SASS has an additional shorthand form.

```scss
// SCSS
@mixin flex-center($direction: row) {
  display: flex;
  justify-content: center;
  align-items: center;
  flex-direction: $direction;
}

.card {
  @include flex-center(column);
  padding: 2rem;
}
```

```sass
// Indented SASS (standard form)
@mixin flex-center($direction: row)
  display: flex
  justify-content: center
  align-items: center
  flex-direction: $direction

.card
  @include flex-center(column)
  padding: 2rem
```

```sass
// Indented SASS (shorthand form)
=flex-center($direction: row)
  display: flex
  justify-content: center
  align-items: center
  flex-direction: $direction

.card
  +flex-center(column)
  padding: 2rem
```

The `=` (define mixin) and `+` (include mixin) shorthand is unique to the indented syntax. The Sass team now discourages this shorthand because it obscures intent. Prefer `@mixin` and `@include` even in `.sass` files.

### Functions

```scss
// SCSS
@use 'sass:math';

@function rem($px, $base: 16) {
  @return math.div($px, $base) * 1rem;
}

h1 {
  font-size: rem(32);    // 2rem
  margin-bottom: rem(24); // 1.5rem
}
```

```sass
// Indented SASS
@use 'sass:math'

@function rem($px, $base: 16)
  @return math.div($px, $base) * 1rem

h1
  font-size: rem(32)
  margin-bottom: rem(24)
```

### Control Flow

```scss
// SCSS
$breakpoints: (
  'sm': 576px,
  'md': 768px,
  'lg': 992px,
  'xl': 1200px,
);

@each $name, $value in $breakpoints {
  .container-#{$name} {
    max-width: $value;
  }
}

@for $i from 1 through 12 {
  .col-#{$i} {
    width: math.div(100%, 12) * $i;
  }
}

@mixin theme-color($theme) {
  @if $theme == 'dark' {
    background: #1a1a1a;
    color: #f0f0f0;
  } @else if $theme == 'light' {
    background: #ffffff;
    color: #333333;
  } @else {
    @error "Unknown theme: #{$theme}";
  }
}
```

```sass
// Indented SASS
$breakpoints: ('sm': 576px, 'md': 768px, 'lg': 992px, 'xl': 1200px)

@each $name, $value in $breakpoints
  .container-#{$name}
    max-width: $value

@for $i from 1 through 12
  .col-#{$i}
    width: math.div(100%, 12) * $i

@mixin theme-color($theme)
  @if $theme == 'dark'
    background: #1a1a1a
    color: #f0f0f0
  @else if $theme == 'light'
    background: #ffffff
    color: #333333
  @else
    @error "Unknown theme: #{$theme}"
```

### The Module System (`@use` and `@forward`)

```scss
// SCSS: _variables.scss
$primary: #3498db;
$secondary: #2ecc71;
$font-base: 16px;
```

```scss
// SCSS: _mixins.scss
@use 'variables' as vars;

@mixin button-variant($bg-color) {
  background-color: $bg-color;
  color: white;
  border: 2px solid darken($bg-color, 10%);
  padding: 0.5rem 1rem;
  border-radius: 4px;
}
```

```scss
// SCSS: styles.scss
@use 'variables' as vars;
@use 'mixins';

.btn-primary {
  @include mixins.button-variant(vars.$primary);
}
```

```sass
// SASS: _variables.sass
$primary: #3498db
$secondary: #2ecc71
$font-base: 16px
```

```sass
// SASS: _mixins.sass
@use 'variables' as vars

@mixin button-variant($bg-color)
  background-color: $bg-color
  color: white
  border: 2px solid darken($bg-color, 10%)
  padding: 0.5rem 1rem
  border-radius: 4px
```

```sass
// SASS: styles.sass
@use 'variables' as vars
@use 'mixins'

.btn-primary
  @include mixins.button-variant(vars.$primary)
```

### Comments

```scss
// SCSS
/* Multi-line comment:
   appears in compiled CSS */
// Single-line comment: stripped from output
```

```sass
// Indented SASS
/* Multi-line comment:
   subsequent lines must be indented
// Single-line comment: stripped from output
```

In indented SASS, multi-line `/* */` comments rely on indentation to determine where the comment ends, rather than an explicit `*/` closing marker.

## Why SCSS Dominates

### CSS Superset Advantage

The migration cost tells the story:

```
Adopting SCSS:    Rename .css to .scss. Done. Everything compiles.
Adopting SASS:    Rewrite every file. Remove all braces and semicolons. Fix indentation.
```

For a team with thousands of lines of existing CSS, SCSS adoption is near-zero effort. This is the single largest reason SCSS won.

### Framework and Library Ecosystem

Every major CSS framework and tool uses SCSS:

| Project | Syntax Used |
|---|---|
| Bootstrap | SCSS |
| Foundation | SCSS |
| Bulma | SCSS |
| Tailwind (config) | CSS/PostCSS (but Sass integration docs show SCSS) |
| Angular CLI | SCSS (default Sass option) |
| Vue CLI | SCSS (default Sass option) |
| Next.js | SCSS |
| Create React App | SCSS |

When you read documentation, copy code from Stack Overflow, or look at framework source code, it is overwhelmingly SCSS.

### Tooling Support

| Tool | SCSS Support | Indented SASS Support |
|---|---|---|
| Prettier | Built-in | Requires community plugin |
| Stylelint | Full (with `stylelint-scss`) | Partial (via `postcss-sass`) |
| VS Code IntelliSense | Excellent | Basic |
| Emmet | Full | Limited |
| Source maps | Full | Full |

The Prettier gap is particularly significant. SCSS gets automatic formatting out of the box. Indented SASS requires a third-party plugin that may not be actively maintained.

## When Indented SASS Makes Sense

Despite SCSS's dominance, there are legitimate reasons to use indented syntax:

### Enforced Clean Code

Indentation-based syntax **forces** proper nesting structure. You cannot write malformed nesting:

```sass
// This is impossible to write incorrectly in .sass
// because the indentation IS the structure
.card
  padding: 1rem

  &__title
    font-size: 1.5rem

  &__body
    line-height: 1.6
```

Compared to SCSS, where mismatched braces or missing semicolons can create subtle bugs.

### Reduced Visual Noise

```scss
// SCSS: 15 special characters (braces + semicolons)
.card {
  padding: 1rem;
  &__title {
    font-size: 1.5rem;
  }
  &__body {
    line-height: 1.6;
  }
}
```

```sass
// SASS: 0 special characters
.card
  padding: 1rem
  &__title
    font-size: 1.5rem
  &__body
    line-height: 1.6
```

For developers who use other indentation-based languages (Python, HAML, Slim, Pug), the indented syntax feels natural.

### Fewer Lines of Code

Indented SASS files are consistently shorter than their SCSS equivalents. For large stylesheets, this can mean 20-30% fewer lines.

## Sass Implementations: A Brief History

Understanding the implementation landscape matters because it affects which features are available.

| Implementation | Language | Status | npm Package |
|---|---|---|---|
| Ruby Sass | Ruby | **Dead** (EOL March 2019) | `sass` (old, removed) |
| LibSass | C/C++ | **Deprecated** (no new features) | `node-sass` |
| Dart Sass | Dart | **Active** (canonical reference) | `sass` |

### Dart Sass Is the Only Active Implementation

Dart Sass is the reference implementation. It gets new features first and is the only implementation that supports modern Sass features like `@use`, `@forward`, and the new color module.

```bash
# Install Dart Sass (the correct package)
npm install -D sass

# NOT this (deprecated LibSass wrapper)
npm install -D node-sass  # Do not use
```

If you are still using `node-sass`, migrate immediately. LibSass does not support `@use`/`@forward`, modern color functions, or any feature added after 2020.

### Performance Tiers

For build performance, there are two Dart Sass distribution options:

```bash
# Standard: Dart compiled to JavaScript (slower, but zero native dependencies)
npm install -D sass

# Embedded: Communicates with native Dart VM (faster, has native dependency)
npm install -D sass-embedded
```

The `sass-embedded` package is measurably faster for large projects. If your build times matter and you can tolerate a native dependency, use it.

For even faster builds in Vite or Webpack, use the Compiler API:

```js
// vite.config.js
import * as sass from 'sass';

export default {
  css: {
    preprocessorOptions: {
      scss: {
        // Reuses a single Sass compiler instance across all files
        // instead of spinning up a new one per file
        api: 'modern-compiler',
      },
    },
  },
};
```

## Critical Deprecations You Must Know

### `@import` Is Deprecated

This is the biggest active deprecation in the Sass ecosystem. If you see `@import` in Sass files, it needs to be migrated.

```scss
// BAD: Deprecated @import
@import 'variables';
@import 'mixins';
@import 'components/buttons';
```

```scss
// GOOD: Modern module system
@use 'variables';
@use 'mixins';
@use 'components/buttons';
```

Key differences between `@import` and `@use`:

| Behavior | `@import` | `@use` |
|---|---|---|
| Namespace | Global (everything dumped into one scope) | Namespaced by default |
| Multiple loads | Re-evaluates file each time | Evaluates once, caches |
| Member access | Direct: `$primary` | Namespaced: `vars.$primary` |
| Visibility | All members global | Members private by default (prefix with `-` or `_`) |

Automated migration tool:

```bash
# Install the migrator
npm install -g sass-migrator

# Migrate a single entry point and all its dependencies
sass-migrator module --migrate-deps src/styles/main.scss
```

`@import` will be removed in Dart Sass 3.0.0 (no sooner than two years after the deprecation in v1.80.0).

### Slash as Division Is Deprecated

```scss
// BAD: Slash is ambiguous (is it division or a CSS shorthand?)
.container {
  width: 100% / 3;
  font: 16px / 1.5 sans-serif;  // This is a CSS shorthand, not division
}
```

```scss
// GOOD: Use math.div() for division
@use 'sass:math';

.container {
  width: math.div(100%, 3);
  font: 16px / 1.5 sans-serif;  // This stays as-is (CSS shorthand)
}
```

### Global Built-in Functions Are Deprecated

```scss
// BAD: Global function calls
$dark-blue: darken(#3498db, 15%);
$light-blue: lighten(#3498db, 15%);
$adjusted: adjust-hue(#3498db, 30deg);
```

```scss
// GOOD: Namespaced module functions
@use 'sass:color';

$dark-blue: color.adjust(#3498db, $lightness: -15%);
$light-blue: color.adjust(#3498db, $lightness: 15%);
$adjusted: color.adjust(#3498db, $hue: 30deg);
```

## Mixing Syntaxes in One Project

You **can** mix `.scss` and `.sass` files in the same project. A `.scss` file can `@use` a `.sass` partial, and vice versa:

```scss
// styles.scss (SCSS file)
@use 'abstracts/variables';    // Could be _variables.sass or _variables.scss
@use 'components/card';        // Could be _card.sass or _card.scss
```

Sass resolves the file regardless of extension. However, mixing syntaxes is generally a bad idea:

- Developers need to mentally switch between two formatting styles
- Linting and formatting configurations need to handle both
- Code reviews become inconsistent
- New team members face unnecessary confusion

**Pick one syntax for the entire project and stick with it.** For nearly all teams, that should be SCSS.

## Common Mistakes

### Mistake 1: Choosing SASS syntax for team projects

Indented SASS is a valid personal preference, but it creates friction in team environments where most developers know CSS (and therefore SCSS) but not indented syntax.

### Mistake 2: Still using `@import` in new projects

Many tutorials and blog posts still show `@import`. Every new Sass file should use `@use` and `@forward`. The `@import` system has no namespace isolation and re-evaluates files on every import.

### Mistake 3: Installing `node-sass` instead of `sass`

```bash
# Wrong: deprecated, missing modern features
npm install -D node-sass

# Right: active, canonical implementation
npm install -D sass
```

`node-sass` wraps the deprecated LibSass. It does not support `@use`, `@forward`, modern color functions, or any Sass feature from the last several years.

### Mistake 4: Ignoring deprecation warnings

Dart Sass prints warnings to the console during compilation. These are not cosmetic. They indicate code that **will break** in a future major version. Address them now.

```bash
# In CI, treat specific deprecations as errors
sass --fatal-deprecation=import src/styles/main.scss dist/styles.css

# Silence warnings from third-party dependencies (while keeping your own)
sass --quiet-deps src/styles/main.scss dist/styles.css
```

### Mistake 5: Deep nesting

Both syntaxes support nesting, but nesting more than 3 levels deep produces high-specificity selectors that are hard to override:

```scss
// BAD: 5 levels deep produces .page .main .article .section .title
.page {
  .main {
    .article {
      .section {
        .title {
          font-size: 1.5rem; // Specificity: 0-5-0
        }
      }
    }
  }
}
```

```scss
// GOOD: Flat selectors with BEM
.article__title {
  font-size: 1.5rem; // Specificity: 0-1-0
}
```

### Mistake 6: Overusing `as *` with `@use`

```scss
// BAD: Dumps everything into global scope, defeating namespace benefits
@use 'variables' as *;
@use 'mixins' as *;
@use 'functions' as *;
// Now you have no idea where $primary or flex-center() comes from
```

```scss
// GOOD: Short, meaningful namespaces
@use 'variables' as vars;
@use 'mixins' as mix;
@use 'functions' as fn;
// vars.$primary, mix.flex-center(), fn.rem() - origin is clear
```

## Best Practices

### Practice 1: Use SCSS for all new projects

SCSS is the ecosystem default. It has better tooling support, more community resources, and zero migration cost from CSS. Use indented SASS only if your entire team prefers it and understands the trade-offs.

### Practice 2: Use `@use` and `@forward` exclusively

Never use `@import` in new code. Structure your project with `_index.scss` files that `@forward` the contents of each folder:

```scss
// abstracts/_index.scss
@forward 'variables';
@forward 'mixins';
@forward 'functions';
```

```scss
// main.scss
@use 'abstracts';

.button {
  color: abstracts.$primary;
  @include abstracts.flex-center;
}
```

### Practice 3: Use `sass` (Dart Sass), not `node-sass`

Check your `package.json`. If you see `node-sass`, replace it:

```bash
npm uninstall node-sass
npm install -D sass
```

The API is largely compatible, but test your build after switching.

### Practice 4: Configure your build tool for the modern compiler API

```js
// vite.config.js
export default {
  css: {
    preprocessorOptions: {
      scss: {
        api: 'modern-compiler',
      },
    },
  },
};
```

```js
// webpack.config.js (sass-loader v14+)
module.exports = {
  module: {
    rules: [
      {
        test: /\.scss$/,
        use: [
          'style-loader',
          'css-loader',
          {
            loader: 'sass-loader',
            options: {
              api: 'modern-compiler',
            },
          },
        ],
      },
    ],
  },
};
```

### Practice 5: Use `!default` for configurable values in shared code

```scss
// _config.scss (in a shared library or theme)
$primary: #3498db !default;
$border-radius: 4px !default;
$font-size-base: 1rem !default;
```

```scss
// Consumer can override before @use
// Unfortunately, @use doesn't support this directly.
// Use @use ... with () instead:
@use 'config' with (
  $primary: #e74c3c,
  $border-radius: 8px,
);
```

### Practice 6: Address deprecation warnings proactively

```bash
# Audit your project for deprecated patterns
npx sass --no-source-map --style=compressed src/main.scss /dev/null 2>&1 | grep "DEPRECATION"
```

## Real-World Use Cases

### Use Case 1: Migrating a CSS project to SCSS

You have 50 CSS files and want Sass features. With SCSS, the migration is trivial:

```bash
# Step 1: Rename files
for file in src/styles/*.css; do
  mv "$file" "${file%.css}.scss"
done

# Step 2: Create an entry point that uses them
# main.scss
```

```scss
// main.scss
@use 'reset';
@use 'typography';
@use 'layout';
@use 'components/buttons';
@use 'components/cards';
// Everything compiles immediately because CSS is valid SCSS
```

Now incrementally adopt Sass features (variables, mixins, nesting) file by file.

### Use Case 2: Converting between syntaxes

If you inherit a `.sass` codebase and need to convert to `.scss`:

```bash
# Use the sass-convert tool or sass-migrator
# Or use find-and-replace patterns:
# 1. Add semicolons at the end of each property line
# 2. Add opening braces after selectors
# 3. Add closing braces at dedent points
```

Dart Sass itself can help: compile the `.sass` files and inspect the structure, then recreate in `.scss`. For large codebases, automated tools are essential.

### Use Case 3: Setting up a new project with proper Sass architecture

```
src/
  styles/
    abstracts/
      _variables.scss     # Design tokens, colors, spacing
      _mixins.scss         # Reusable mixin definitions
      _functions.scss      # Custom Sass functions
      _index.scss          # @forward all abstracts
    base/
      _reset.scss          # CSS reset or normalize
      _typography.scss     # Base typography rules
      _index.scss          # @forward all base
    components/
      _button.scss
      _card.scss
      _modal.scss
      _index.scss          # @forward all components
    layout/
      _header.scss
      _footer.scss
      _grid.scss
      _index.scss          # @forward all layout
    main.scss              # Entry point: @use each folder
```

```scss
// abstracts/_index.scss
@forward 'variables';
@forward 'mixins';
@forward 'functions';

// main.scss
@use 'abstracts';
@use 'base';
@use 'layout';
@use 'components';
```

This structure works identically with `.sass` files. The only difference is file extensions and the absence of braces/semicolons inside each file.

---

> **Key Takeaway:** Use SCSS for all new projects. It is a superset of CSS with near-universal tooling support, while indented SASS is a niche choice with real trade-offs in team environments and tool compatibility.
