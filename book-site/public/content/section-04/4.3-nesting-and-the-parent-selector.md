# 4.3 Nesting and the `&` Parent Selector

> Nesting lets you write selectors that mirror your HTML structure, while the `&` parent selector gives you precise control over how nested selectors compile. Used well, nesting improves readability. Used poorly, it creates specificity nightmares.

---

## Basic Nesting

Sass nesting places one selector inside another. The inner selector compiles by prepending the outer selector with a descendant combinator (space).

```scss
// SCSS
.card {
  padding: 1rem;
  border: 1px solid #ddd;

  h2 {
    font-size: 1.5rem;
    margin-bottom: 0.5rem;
  }

  p {
    line-height: 1.6;
  }
}
```

```css
/* Compiled CSS */
.card {
  padding: 1rem;
  border: 1px solid #ddd;
}
.card h2 {
  font-size: 1.5rem;
  margin-bottom: 0.5rem;
}
.card p {
  line-height: 1.6;
}
```

Without `&`, nesting is shorthand for `& ` (ampersand + space). Writing `.child {}` inside `.parent {}` is identical to writing `& .child {}`.

---

## The `&` Parent Selector

The `&` is replaced with the fully compiled parent selector at that nesting level. It is not just the immediate parent; it is the entire selector chain built up to that point.

```scss
.grandparent {
  .parent {
    & .child {
      color: red;
    }
  }
}
// & here equals ".grandparent .parent"
// Output: .grandparent .parent .child { color: red; }
```

### Pseudo-classes and Pseudo-elements

The most common use of `&` is attaching pseudo-classes and pseudo-elements to the parent selector without inserting a space:

```scss
.button {
  background: #3498db;
  color: white;
  transition: background 0.2s;

  &:hover {
    background: #2980b9;
  }

  &:focus-visible {
    outline: 2px solid #3498db;
    outline-offset: 2px;
  }

  &:active {
    transform: scale(0.98);
  }

  &:disabled {
    opacity: 0.5;
    cursor: not-allowed;
  }

  &::before {
    content: '';
    display: inline-block;
    width: 1em;
    height: 1em;
  }
}
```

```css
/* Compiled CSS */
.button { background: #3498db; color: white; transition: background 0.2s; }
.button:hover { background: #2980b9; }
.button:focus-visible { outline: 2px solid #3498db; outline-offset: 2px; }
.button:active { transform: scale(0.98); }
.button:disabled { opacity: 0.5; cursor: not-allowed; }
.button::before { content: ''; display: inline-block; width: 1em; height: 1em; }
```

### Compound Selectors (Same Element)

The `&` directly adjacent to a class or attribute creates a compound selector, meaning "the same element that also has this class":

```scss
.btn {
  padding: 0.5rem 1rem;

  &.btn-primary {
    background: blue;
  }

  &.btn-danger {
    background: red;
  }

  &[disabled] {
    opacity: 0.5;
  }
}
```

```css
/* Compiled CSS */
.btn { padding: 0.5rem 1rem; }
.btn.btn-primary { background: blue; }
.btn.btn-danger { background: red; }
.btn[disabled] { opacity: 0.5; }
```

This is critically different from nesting without `&`:

```scss
.btn {
  .active { }    // .btn .active   -- a DESCENDANT element with class .active
  &.active { }   // .btn.active    -- the SAME element with both classes
}
```

Confusing these two is one of the most common nesting bugs.

### Suffix Concatenation

When `&` is followed by text (no space), Sass appends that text directly to the parent selector string:

```scss
.btn {
  &-primary { background: blue; }
  &-secondary { background: gray; }
  &-outline { border: 2px solid currentColor; }
}
```

```css
/* Compiled CSS */
.btn-primary { background: blue; }
.btn-secondary { background: gray; }
.btn-outline { border: 2px solid currentColor; }
```

This is how BEM patterns work with Sass (covered below).

### Reversing Context (Parent on the Right)

Placing `&` on the right side of a nested rule lets you style the element when it appears inside a different ancestor:

```scss
.nav-link {
  color: #333;

  .dark-theme & {
    color: #f0f0f0;
  }

  [dir="rtl"] & {
    text-align: right;
  }

  body.page-about & {
    font-weight: bold;
  }
}
```

```css
/* Compiled CSS */
.nav-link { color: #333; }
.dark-theme .nav-link { color: #f0f0f0; }
[dir="rtl"] .nav-link { text-align: right; }
body.page-about .nav-link { font-weight: bold; }
```

This pattern is useful for:
- Theme-based overrides (`.dark-theme &`, `.high-contrast &`)
- RTL layouts (`[dir="rtl"] &`)
- Page-specific styling (`body.page-home &`)
- Feature detection (`.no-js &`)

### Multiple `&` in One Selector

You can reference `&` more than once in a single nested selector:

```scss
.card {
  & + & {
    margin-top: 1rem;
  }

  & > &__body {
    flex: 1;
  }
}
```

```css
/* Compiled CSS */
.card + .card { margin-top: 1rem; }
.card > .card__body { flex: 1; }
```

### Specificity Doubling

Repeating `&` doubles the specificity without changing what it matches:

```scss
.widget {
  &#{&} {
    font-weight: bold;
  }
}
// Output: .widget.widget { font-weight: bold; }
// Specificity: (0, 2, 0) instead of (0, 1, 0)
```

This is sometimes used to override third-party library styles without `!important`, but should be used sparingly.

## BEM with Nesting

BEM (Block Element Modifier) is the most popular naming convention used with Sass nesting. The `&` suffix concatenation maps directly to BEM's naming pattern.

### The Standard BEM + Sass Pattern

```scss
.card {
  border: 1px solid #ddd;
  border-radius: 4px;
  overflow: hidden;

  // Elements
  &__header {
    padding: 1rem;
    background: #f5f5f5;
    border-bottom: 1px solid #ddd;
  }

  &__body {
    padding: 1rem;
  }

  &__footer {
    padding: 0.75rem 1rem;
    background: #fafafa;
    border-top: 1px solid #ddd;
  }

  &__title {
    font-size: 1.25rem;
    margin: 0;
  }

  // Modifiers
  &--featured {
    border-color: gold;
    box-shadow: 0 2px 8px rgba(0, 0, 0, 0.1);
  }

  &--compact {
    .card__body {
      padding: 0.5rem;
    }
  }
}
```

```css
/* Compiled CSS */
.card { border: 1px solid #ddd; border-radius: 4px; overflow: hidden; }
.card__header { padding: 1rem; background: #f5f5f5; border-bottom: 1px solid #ddd; }
.card__body { padding: 1rem; }
.card__footer { padding: 0.75rem 1rem; background: #fafafa; border-top: 1px solid #ddd; }
.card__title { font-size: 1.25rem; margin: 0; }
.card--featured { border-color: gold; box-shadow: 0 2px 8px rgba(0, 0, 0, 0.1); }
.card--compact .card__body { padding: 0.5rem; }
```

Notice that the compiled output uses flat, single-class selectors (specificity 0,1,0) for elements and modifiers. The nesting is purely organizational in the source code.

### The BEM Grandchild Problem

BEM convention dictates that `__` appears only once in a selector. Nesting BEM elements inside BEM elements violates this:

```scss
// BAD: Creates .nav__menu__item__link
.nav {
  &__menu {
    &__item {
      &__link {
        color: blue;
      }
    }
  }
}
```

The output `.nav__menu__item__link` is hard to read and implies a deep DOM dependency. Fix it by flattening the element names:

```scss
// GOOD: Flat BEM elements
.nav {
  &__menu {
    list-style: none;
  }

  &__item {
    display: inline-block;
  }

  &__link {
    color: blue;
    text-decoration: none;
  }
}
```

Output: `.nav__menu`, `.nav__item`, `.nav__link`. Each is a single-class selector with the same low specificity.

### Greppability Trade-off

One downside of `&__element` nesting: the string `.card__header` does not exist literally in your SCSS source. If you search your codebase for `.card__header`, you will not find the definition. You must search for `&__header` instead, which may match elements from different blocks.

Some teams accept this trade-off for the organizational benefit. Others prefer to write BEM selectors without nesting:

```scss
// Alternative: No nesting, fully greppable
.card { /* ... */ }
.card__header { /* ... */ }
.card__body { /* ... */ }
.card--featured { /* ... */ }
```

Both approaches are valid. Choose one and be consistent.

## Nesting At-Rules

### `@media` Inside Selectors

Sass lets you nest `@media` queries inside selectors. The at-rule "bubbles up" to the top level in compiled output, keeping the selector inside:

```scss
.sidebar {
  width: 300px;
  float: left;

  @media (max-width: 768px) {
    width: 100%;
    float: none;
  }
}
```

```css
/* Compiled CSS */
.sidebar { width: 300px; float: left; }
@media (max-width: 768px) {
  .sidebar { width: 100%; float: none; }
}
```

This co-locates responsive behavior with the base styles it modifies, which is significantly easier to maintain than grouping all media queries at the bottom of a file.

### Nested `@media` Merge

When you nest `@media` inside `@media`, Sass merges them with `and`:

```scss
@media (hover: hover) {
  .button {
    @media (min-width: 768px) {
      &:hover {
        transform: scale(1.05);
      }
    }
  }
}
```

```css
/* Compiled CSS */
@media (hover: hover) and (min-width: 768px) {
  .button:hover { transform: scale(1.05); }
}
```

### `@supports` Inside Selectors

```scss
.grid {
  display: flex;
  flex-wrap: wrap;

  @supports (display: grid) {
    display: grid;
    grid-template-columns: repeat(auto-fill, minmax(250px, 1fr));
  }
}
```

```css
/* Compiled CSS */
.grid { display: flex; flex-wrap: wrap; }
@supports (display: grid) {
  .grid { display: grid; grid-template-columns: repeat(auto-fill, minmax(250px, 1fr)); }
}
```

### The Duplicate `@media` Trade-off

Each selector with a nested `@media` produces its own `@media` block in the output. With many components, the compiled CSS can contain dozens of identical `@media` declarations:

```scss
.header { @media (max-width: 768px) { /* ... */ } }
.nav    { @media (max-width: 768px) { /* ... */ } }
.card   { @media (max-width: 768px) { /* ... */ } }
.footer { @media (max-width: 768px) { /* ... */ } }
```

This produces four separate `@media (max-width: 768px)` blocks. While browsers handle this efficiently and gzip compresses repetitive strings well, the uncompressed file size increases. If this concerns you, post-processing tools like cssnano can merge duplicate `@media` blocks.

## Nesting with Combinators

Combinators work naturally inside nested rules:

```scss
.list {
  // Direct child
  > li {
    padding: 0.5rem;
  }

  // Adjacent sibling
  + .list {
    margin-top: 1rem;
  }

  // General sibling
  ~ .description {
    color: #666;
  }
}
```

```css
/* Compiled CSS */
.list > li { padding: 0.5rem; }
.list + .list { margin-top: 1rem; }
.list ~ .description { color: #666; }
```

## `@at-root`: Escaping Nesting

The `@at-root` directive emits content at the document root, breaking out of all nesting:

```scss
.parent {
  color: black;

  @at-root .child {
    color: red;
  }
}
```

```css
/* Compiled CSS */
.parent { color: black; }
.child { color: red; }
```

### Block Form

Emit multiple rules at root level:

```scss
.parent {
  @at-root {
    .independent-a { color: red; }
    .independent-b { color: blue; }
  }
}
```

### Escaping At-Rules with `(without:)` and `(with:)`

You can selectively break out of at-rules while preserving others:

```scss
@media print {
  .page {
    width: 8in;

    @at-root (without: media) {
      // Escapes @media but keeps the .page selector context
      color: #111;
    }
  }
}
```

```css
/* Compiled CSS */
@media print {
  .page { width: 8in; }
}
.page { color: #111; }
```

### Practical Use: Keyframes Inside Components

A useful `@at-root` pattern places animation keyframes next to the component that uses them:

```scss
.notification {
  animation: slideIn 0.3s ease-out;

  @at-root {
    @keyframes slideIn {
      from { transform: translateY(-100%); opacity: 0; }
      to { transform: translateY(0); opacity: 1; }
    }
  }
}
```

The `@keyframes` ends up at the root level (where it must be), but stays co-located with the component in the source.

## Nesting Depth: The Three-Level Rule

The widely accepted guideline: **never nest more than three levels deep.**

### Why Deep Nesting Hurts

Deep nesting produces high-specificity selectors that are difficult to override:

```scss
// BAD: 5 levels deep
.page {
  .main {
    .article {
      .section {
        .title {
          font-size: 1.5rem;
        }
      }
    }
  }
}
// Output: .page .main .article .section .title
// Specificity: (0, 5, 0)
```

To override this, you need a selector with specificity greater than (0, 5, 0), which means an equally deep chain or `!important`. Both options make the problem worse.

```scss
// GOOD: Flat class selectors
.article-title {
  font-size: 1.5rem;
}
// Specificity: (0, 1, 0) -- trivially overridable
```

### What Belongs in Nesting

Nesting is appropriate for things that are genuinely part of the same selector:

```scss
// GOOD: Nesting for pseudo-states and context
.button {
  background: blue;            // Level 0

  &:hover { background: darkblue; }   // Level 1: pseudo-state
  &:focus-visible { outline: 2px solid; } // Level 1: pseudo-state
  &:disabled { opacity: 0.5; }        // Level 1: pseudo-state

  &--large { padding: 1rem 2rem; }    // Level 1: BEM modifier

  @media (max-width: 768px) {         // Level 1: responsive
    width: 100%;
  }
}
```

Nesting is not appropriate for mirroring HTML structure:

```scss
// BAD: Mirrors the DOM tree
nav {
  ul {
    li {
      a {
        color: blue;
      }
    }
  }
}
// Output: nav ul li a { color: blue; }
// Specificity: (0, 0, 4) -- fragile, tightly coupled to markup
```

### The Compiled Output Test

Before committing, check what your nesting produces. The golden rule: **your compiled CSS should look exactly the same as if you had written it by hand.** If the compiled output contains selectors you would never write manually, your nesting is too deep.

## Sass Nesting vs Native CSS Nesting

Native CSS now supports nesting in all modern browsers (Chrome 112+, Firefox 117+, Safari 16.5+). There are important differences from Sass nesting.

### Syntax Comparison

```scss
/* Sass nesting */
.card {
  color: black;
  .title { font-size: 1.5rem; }
  &:hover { border-color: blue; }
  &--featured { border-color: gold; }
}
```

```css
/* Native CSS nesting (identical syntax for these cases) */
.card {
  color: black;
  .title { font-size: 1.5rem; }
  &:hover { border-color: blue; }
  /* &--featured does NOT work in native CSS */
}
```

### Three Critical Differences

**1. Suffix concatenation does not exist in native CSS.**

Sass: `&--featured` compiles to `.card--featured`.

Native CSS: `&--featured` is invalid because the browser interprets `--featured` as a custom property fragment, not a class suffix. **BEM suffix patterns only work in Sass.**

```scss
// Sass: Works perfectly
.card {
  &__header { /* ... */ }    // .card__header
  &--primary { /* ... */ }   // .card--primary
}
```

```css
/* Native CSS: Does NOT work */
.card {
  &__header { /* ... */ }    /* Invalid */
  &--primary { /* ... */ }   /* Invalid */
}
```

**2. Specificity differs due to `:is()` wrapping.**

Native CSS implicitly wraps the parent selector in `:is()` when resolving nesting. Sass does direct text substitution.

```scss
// Source (same in both Sass and native CSS)
.foo, #bar {
  .baz { color: red; }
}
```

Sass output:
```css
.foo .baz, #bar .baz { color: red; }
/* .foo .baz specificity: (0, 2, 0) */
/* #bar .baz specificity: (1, 1, 0) */
```

Native CSS equivalent:
```css
:is(.foo, #bar) .baz { color: red; }
/* Both selectors have specificity: (1, 1, 0) */
/* :is() takes the highest specificity of its arguments */
```

The native CSS version gives `.foo .baz` higher specificity than you might expect because `:is()` inherits the specificity of `#bar`.

**3. `&` represents different things.**

In Sass, `&` is a text replacement. `.foo { &bar {} }` produces `.foobar`.

In native CSS, `&` represents the parent selector as a `:is()` expression. `.foo { &div {} }` means `div.foo`, not `.foodiv`. Native CSS cannot concatenate strings.

### When to Use Which

| Need | Use |
|---|---|
| BEM suffix patterns (`&__element`, `&--modifier`) | Sass nesting only |
| Co-located media queries | Either (both support it) |
| Pseudo-classes and pseudo-elements | Either |
| Zero build step | Native CSS nesting |
| Consistent specificity behavior | Check your specific case |
| Browser support below ~95% threshold | Sass nesting (compiles to flat CSS) |

## Selector List Expansion

When a parent is a selector list (comma-separated), nesting applies to each selector individually, creating a combinatorial expansion:

```scss
h1, h2, h3 {
  a {
    color: blue;

    &:hover {
      color: darkblue;
    }
  }
}
```

```css
/* Compiled CSS: 3 selectors for each nested rule */
h1 a, h2 a, h3 a { color: blue; }
h1 a:hover, h2 a:hover, h3 a:hover { color: darkblue; }
```

With more complex lists, this expansion can produce a lot of CSS. A parent with 3 selectors nested inside another parent with 3 selectors produces 9 selectors in the output.

## Common Mistakes

### Mistake 1: Confusing descendant and compound selectors

```scss
.btn {
  // Descendant: a child element with class .icon
  .icon {
    margin-right: 0.5rem;
  }

  // Compound: the same .btn element also has class .active
  &.active {
    background: green;
  }
}
```

Forgetting the `&` when you mean the same element is a frequent source of bugs. `.icon` selects a descendant; `&.icon` selects the button itself if it also has class `.icon`.

### Mistake 2: Nesting deeper than 3 levels

```scss
// BAD: Deeply coupled to markup, high specificity
.wrapper {
  .container {
    .sidebar {
      .widget {
        .widget-title {
          color: blue; // Specificity: (0, 5, 0)
        }
      }
    }
  }
}

// GOOD: Flat class selector
.widget-title {
  color: blue; // Specificity: (0, 1, 0)
}
```

### Mistake 3: Losing track of compiled output

What looks clean in SCSS can produce bloated CSS:

```scss
// 10 lines of SCSS
.card {
  .header {
    .title {
      .icon {
        svg {
          fill: red;
        }
      }
    }
  }
}
```

Compiles to: `.card .header .title .icon svg { fill: red; }`. Ask yourself: would I write this selector by hand? If not, refactor.

### Mistake 4: BEM grandchild nesting

```scss
// BAD: Double underscore
.nav {
  &__list {
    &__item {       // .nav__list__item -- violates BEM
      &__link { }   // .nav__list__item__link -- worse
    }
  }
}

// GOOD: Flat elements
.nav {
  &__list { }
  &__item { }
  &__link { }
}
```

### Mistake 5: Nesting everything inside a component

```scss
// BAD: Every child is nested inside .page
.page {
  .header { /* ... */ }
  .nav { /* ... */ }
  .main { /* ... */ }
  .sidebar { /* ... */ }
  .footer { /* ... */ }
}
```

This adds `.page` as a prefix to every selector, increasing specificity for no benefit. These children are likely independent components that should be top-level selectors.

### Mistake 6: Assuming native CSS `&` supports BEM suffixes

```css
/* Native CSS: This does NOT produce .card__header */
.card {
  &__header { color: red; }  /* INVALID */
}
```

Suffix concatenation (`&__header`, `&--modifier`) is a Sass-only feature. If you plan to migrate to native CSS nesting, BEM suffixes will need to be rewritten as full class selectors.

## Best Practices

### Practice 1: Limit nesting to 3 levels

Reserve nesting for pseudo-states, pseudo-elements, BEM modifiers, media queries, and context qualifiers. Avoid nesting based on HTML structure.

### Practice 2: Nest responsive styles with their base styles

```scss
.hero {
  padding: 4rem 2rem;
  font-size: 2rem;

  @media (max-width: 768px) {
    padding: 2rem 1rem;
    font-size: 1.25rem;
  }
}
```

This keeps all of `.hero`'s styles in one place instead of splitting base and responsive into separate locations.

### Practice 3: Check compiled output periodically

Open the compiled CSS file and verify the selectors are what you intended. Tools like Source Map Explorer or simply reading the output file catch nesting problems early.

### Practice 4: Use `&` for state variations, not hierarchy

```scss
// GOOD: States and variations on the same element
.input {
  border: 1px solid #ccc;

  &:focus { border-color: #3498db; }
  &:invalid { border-color: #e74c3c; }
  &.input--large { padding: 1rem; }
  &[readonly] { background: #f5f5f5; }
}
```

### Practice 5: Keep BEM elements flat

All `&__element` selectors should be at the same nesting level within a block:

```scss
.accordion {
  &__header { /* ... */ }
  &__body { /* ... */ }
  &__footer { /* ... */ }
  &__icon { /* ... */ }

  &--open {
    .accordion__body {
      display: block;
    }
    .accordion__icon {
      transform: rotate(180deg);
    }
  }
}
```

## Real-World Use Cases

### Use Case 1: Component with States and Responsive Behavior

```scss
.modal {
  position: fixed;
  inset: 0;
  display: flex;
  align-items: center;
  justify-content: center;
  background: rgba(0, 0, 0, 0.5);
  opacity: 0;
  visibility: hidden;
  transition: opacity 0.3s, visibility 0.3s;

  // State: open
  &.is-open {
    opacity: 1;
    visibility: visible;
  }

  // Elements
  &__content {
    background: white;
    border-radius: 8px;
    padding: 2rem;
    max-width: 600px;
    width: 90%;
    max-height: 80vh;
    overflow-y: auto;
  }

  &__header {
    display: flex;
    justify-content: space-between;
    align-items: center;
    margin-bottom: 1rem;
  }

  &__close {
    background: none;
    border: none;
    cursor: pointer;
    padding: 0.5rem;

    &:hover {
      opacity: 0.7;
    }
  }

  // Responsive
  @media (max-width: 480px) {
    align-items: flex-end;

    .modal__content {
      width: 100%;
      max-height: 90vh;
      border-radius: 8px 8px 0 0;
    }
  }
}
```

### Use Case 2: Navigation with Context Awareness

```scss
.nav {
  display: flex;
  gap: 1rem;

  &__link {
    color: #333;
    text-decoration: none;
    padding: 0.5rem 1rem;
    border-radius: 4px;
    transition: background 0.2s;

    &:hover {
      background: #f0f0f0;
    }

    &.is-active {
      background: #3498db;
      color: white;
    }

    // Context: dark header
    .header--dark & {
      color: #f0f0f0;

      &:hover {
        background: rgba(255, 255, 255, 0.1);
      }
    }

    // Context: sidebar
    .sidebar & {
      display: block;
      padding: 0.75rem 1rem;
      border-radius: 0;
      border-left: 3px solid transparent;

      &.is-active {
        border-left-color: #3498db;
        background: #f0f8ff;
      }
    }
  }
}
```

### Use Case 3: Form Field with Validation States

```scss
.form-field {
  margin-bottom: 1.5rem;

  &__label {
    display: block;
    font-weight: 600;
    margin-bottom: 0.25rem;
  }

  &__input {
    width: 100%;
    padding: 0.5rem 0.75rem;
    border: 1px solid #ccc;
    border-radius: 4px;
    transition: border-color 0.2s, box-shadow 0.2s;

    &:focus {
      border-color: #3498db;
      box-shadow: 0 0 0 3px rgba(52, 152, 219, 0.15);
      outline: none;
    }

    &::placeholder {

  // Validation states
  &--error {
```

> **Key Takeaway:** Nesting is an organizational tool, not a structural one. Use `&` for pseudo-states, BEM suffixes, modifiers, and context qualifiers. Keep nesting to 3 levels or fewer, and always verify the compiled CSS produces selectors you would write by hand.
