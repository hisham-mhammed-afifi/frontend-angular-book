# 4.5 Deprecation of `@import`

> Sass `@import` is deprecated as of Dart Sass 1.80.0 and will be removed in Dart Sass 3.0.0. Understanding why it was deprecated, what replaces it, and how to migrate is essential for every project using Sass today.

---

## Why `@import` Was Deprecated

`@import` was the original way to split Sass across files. It worked for years, but it had fundamental design flaws that could not be fixed without a replacement.

### Problem 1: Global Scope Pollution

Every `@import` dumps all variables, mixins, and functions into a single global namespace. Nothing is scoped to the file that defines it.

```scss
// _colors.scss
$primary: blue;

// _buttons.scss
$primary: red;  // Silently overwrites _colors.scss value

// main.scss
@import 'colors';
@import 'buttons';

.link { color: $primary; }  // red -- colors.$primary was silently overwritten
```

With dozens of files, tracking which `$primary` is active at any given point becomes impossible. Library authors were forced to prefix everything (`$bootstrap-primary`, `$susy-columns`) to avoid collisions.

### Problem 2: CSS Duplication

`@import` re-executes and re-emits a file's CSS every time it is imported:

```scss
// _reset.scss
*, *::before, *::after { box-sizing: border-box; margin: 0; }

// _header.scss
@import 'reset';

// _footer.scss
@import 'reset';

// main.scss
@import 'header';
@import 'footer';
```

The reset CSS appears **twice** in the compiled output. In large projects with deep dependency trees, this duplication inflates bundle sizes significantly.

### Problem 3: No Namespacing

You cannot tell where a member comes from by reading the code:

```scss
@import 'config';
@import 'utils';
@import 'helpers';
@import 'vendor/library';

.card {
  color: $primary;          // Which file defines this?
  padding: spacing(2);      // Which file defines this?
  @include respond-to(md);  // Which file defines this?
}
```

Every member could come from any imported file. IDE tooling struggles to trace origins because the import system provides no structural information.

### Problem 4: No Private Members

Every variable, mixin, and function in an imported file is globally accessible. There is no way to mark implementation details as internal:

```scss
// _grid.scss
$_internal-gutter: 16px;  // Intended to be private, but...

// consumer.scss
@import 'grid';
.box { padding: $_internal-gutter; }  // Accessible. No encapsulation.
```

### Problem 5: Compilation Speed

Because `@import` re-evaluates files on every inclusion, compilation time grows with redundant imports. A file imported by 20 components is parsed and executed 20 times.

### Problem 6: Untraceable Dependencies

With `@import`, a file can use members that it never directly imported, relying on a transitive import chain:

```scss
// main.scss
@import 'variables';  // Defines $primary
@import 'components'; // Uses $primary but never imports 'variables'
```

If the import order changes or `variables` is removed, `components` breaks with no clear indication of the dependency.

## The Deprecation Timeline

| Event | Version | Date |
|---|---|---|
| `@import` deprecated | Dart Sass 1.80.0 | October 2024 |
| Dart Sass 2.0.0 | TBD | Does NOT remove `@import` |
| `@import` removed | Dart Sass 3.0.0 | No sooner than October 2026 |

Key details:

- Dart Sass 2.0.0 includes other breaking changes (legacy JS API removal) but deliberately leaves `@import` untouched.
- Dart Sass 3.0.0 removes both `@import` and all global built-in functions simultaneously.
- After removal, any `@import` in Sass files will be treated as a plain CSS `@import` (which triggers a runtime HTTP request, not a compile-time inclusion). There may be an intermediate error phase before this transition.
- The Sass team committed to at least two years between deprecation and removal.

## What Was Deprecated Alongside `@import`

Global built-in functions that have namespaced equivalents in built-in modules are deprecated on the same timeline:

| Deprecated Global Function | Module Replacement |
|---|---|
| `darken($color, $amount)` | `color.adjust($color, $lightness: -$amount)` |
| `lighten($color, $amount)` | `color.adjust($color, $lightness: $amount)` |
| `saturate($color, $amount)` | `color.adjust($color, $saturation: $amount)` |
| `desaturate($color, $amount)` | `color.adjust($color, $saturation: -$amount)` |
| `mix($c1, $c2, $weight)` | `color.mix($c1, $c2, $weight)` |
| `adjust-color(...)` | `color.adjust(...)` |
| `scale-color(...)` | `color.scale(...)` |
| `change-color(...)` | `color.change(...)` |
| `red($color)` | `color.channel($color, 'red')` |
| `green($color)` | `color.channel($color, 'green')` |
| `blue($color)` | `color.channel($color, 'blue')` |
| `percentage($number)` | `math.percentage($number)` |
| `round($number)` | `math.round($number)` |
| `ceil($number)` | `math.ceil($number)` |
| `floor($number)` | `math.floor($number)` |
| `unit($number)` | `math.unit($number)` |
| `unitless($number)` | `math.is-unitless($number)` |
| `comparable($n1, $n2)` | `math.compatible($n1, $n2)` |

All module replacements require explicit `@use`:

```scss
@use 'sass:color';
@use 'sass:math';

$light: color.adjust(#3498db, $lightness: 20%);
$third: math.div(100%, 3);
$rounded: math.round(3.7);
```

## The Replacement: `@use` and `@forward`

Every problem with `@import` is solved by the module system:

| Problem | `@import` | `@use` / `@forward` |
|---|---|---|
| Global scope pollution | All members global | Members scoped to loading file |
| CSS duplication | Re-emits on every import | Evaluates once (singleton) |
| No namespacing | Flat global namespace | Automatic namespaces |
| No private members | Everything accessible | `-` or `_` prefix = private |
| Slow compilation | Re-parses per import | One-time evaluation |
| Untraceable dependencies | Transitive, implicit | Explicit per file |

A quick side-by-side:

```scss
// OLD: @import
@import 'variables';
@import 'mixins';

.card {
  color: $primary;
  @include respond-to(md) { width: 50%; }
}
```

```scss
// NEW: @use
@use 'variables' as vars;
@use 'mixins' as mix;

.card {
  color: vars.$primary;
  @include mix.respond-to(md) { width: 50%; }
}
```

## Migrating with `sass-migrator`

The `sass-migrator` tool automates most of the conversion.

### Installation

```bash
npm install -g sass-migrator
```

### Basic Usage

```bash
# Migrate a single entrypoint and all its dependencies
sass-migrator module --migrate-deps src/styles/main.scss

# Preview changes without writing (dry run)
sass-migrator module --migrate-deps --dry-run --verbose src/styles/main.scss

# Specify load paths (if your project uses them)
sass-migrator module --migrate-deps --load-path=src/styles src/styles/main.scss
```

### Key Flags

| Flag | Purpose |
|---|---|
| `--migrate-deps` | Migrate all transitively loaded files, not just the specified file |
| `--dry-run` | Preview changes without writing to disk |
| `--verbose` | Print detailed output (with `--dry-run`, prints new file contents) |
| `--load-path` | Specify additional load paths |
| `--remove-prefix=<prefix>` | Strip a prefix from member names (useful for libraries) |
| `--forward=all\|prefixed\|none` | Control `@forward` generation strategy |
| `--no-namespace` | Use `as *` instead of namespaces |

### What the Migrator Does

1. Converts `@import` to `@use` with appropriate namespaces
2. Adds namespace prefixes to all member references (`$primary` becomes `vars.$primary`)
3. Converts variable overrides (reassignment after `@import`) to `@use ... with()` clauses
4. Creates `@forward` rules for re-export files
5. Adds `@use` rules to files that used transitive dependencies

### What the Migrator Cannot Do

- Handle custom importers
- Handle the `pkg:` URL scheme
- Convert nested `@import` inside CSS rules (requires manual `meta.load-css()` conversion)
- Process dynamically generated import paths (interpolation in URLs)
- Reliably migrate complex third-party libraries like Bootstrap

Always run `--dry-run` first and verify the output.

## Manual Migration: Step by Step

For projects where the automated tool fails or for understanding the process.

### Step 1: Start from Leaf Files

Identify files that have no imports of their own. These are the safest to convert first.

```
_variables.scss    (no imports)     <- start here
_mixins.scss       (imports variables)
_buttons.scss      (imports variables, mixins)
main.scss          (imports everything)
```

### Step 2: Convert `@import` to `@use`

```scss
// BEFORE: _buttons.scss
@import 'variables';
@import 'mixins';

.button {
  color: $primary;
  @include respond-to(md) { width: auto; }
}
```

```scss
// AFTER: _buttons.scss
@use 'variables' as vars;
@use 'mixins' as mix;

.button {
  color: vars.$primary;
  @include mix.respond-to(md) { width: auto; }
}
```

### Step 3: Convert Re-export Files to `@forward`

```scss
// BEFORE: abstracts/_index.scss
@import 'variables';
@import 'mixins';
@import 'functions';
```

```scss
// AFTER: abstracts/_index.scss
@forward 'variables';
@forward 'mixins';
@forward 'functions';
```

### Step 4: Convert Variable Overrides to `with()`

```scss
// BEFORE
$primary: #e74c3c;
$font-size-base: 18px;
@import 'library';
```

```scss
// AFTER
@use 'library' with (
  $primary: #e74c3c,
  $font-size-base: 18px,
);
```

Only variables declared with `!default` in the library can be configured this way.

### Step 5: Replace Global Built-in Functions

```scss
// BEFORE
$light: lighten($primary, 20%);
$half: percentage(0.5);
$width: 100% / 3;

// AFTER
@use 'sass:color';
@use 'sass:math';

$light: color.adjust($primary, $lightness: 20%);
$half: math.percentage(0.5);
$width: math.div(100%, 3);
```

### Step 6: Handle Nested `@import`

`@use` must be at the top of a file. It cannot be nested inside rules. Convert nested `@import` to `meta.load-css()`:

```scss
// BEFORE
.dark-theme {
  @import 'themes/dark';
}
```

```scss
// AFTER
@use 'sass:meta';

.dark-theme {
  @include meta.load-css('themes/dark');
}
```

`meta.load-css()` includes the module's CSS output at the point of the `@include`. It does not make variables, mixins, or functions available. If you need those, wrap the module's content in a mixin.

## Managing Deprecation Warnings

After upgrading to Dart Sass 1.80.0+, you will see deprecation warnings for every `@import` and global built-in function call.

### Silence Warnings from Dependencies

```bash
# Show warnings from YOUR code, hide warnings from node_modules
sass --quiet-deps src/styles/main.scss dist/main.css
```

### Silence Specific Deprecation Types

```bash
# Completely silence @import warnings
sass --silence-deprecation=import src/styles/main.scss dist/main.css

# Silence both @import and global built-in function warnings
sass --silence-deprecation=import --silence-deprecation=global-builtin src/styles/main.scss dist/main.css
```

### Enforce Migration in CI

```bash
# Fail the build if @import is used
sass --fatal-deprecation=import src/styles/main.scss dist/main.css
```

Only use `--fatal-deprecation` when your entire codebase (and all dependencies) are fully migrated.

### Build Tool Configuration

```js
// webpack.config.js (sass-loader)
{
  loader: 'sass-loader',
  options: {
    sassOptions: {
      quietDeps: true,
      silenceDeprecations: ['import', 'global-builtin'],
    },
  },
}
```

```js
// vite.config.js
export default {
  css: {
    preprocessorOptions: {
      scss: {
        api: 'modern-compiler',
        silenceDeprecations: ['import', 'global-builtin'],
      },
    },
  },
};
```

### Terse Mode

By default, Dart Sass prints each type of deprecation warning a maximum of five times before silencing further occurrences. This prevents console flooding while still alerting you to the issue.

## Third-Party Library Compatibility

### The Problem

Many widely used libraries (notably Bootstrap 5.x) have not migrated to `@use`/`@forward`. They use `@import` internally and rely on global scope for variable overrides.

### Strategy: Use `--quiet-deps`

```bash
sass --quiet-deps src/main.scss dist/main.css
```

This shows deprecation warnings from your own code while hiding them from dependencies. You can fix your code now and wait for libraries to update.

### Strategy: Pin Sass Below 3.0.0

```json
{
  "devDependencies": {
    "sass": ">=1.80.0 <3.0.0"
  }
}
```

This ensures your project does not accidentally upgrade to the version that removes `@import`.

### Strategy: Import-Only Files (for Library Authors)

Library authors can support both old and new consumers by creating `.import.scss` companion files:

```scss
// _button.scss (module API for @use consumers)
$radius: 4px !default;
@mixin button { /* ... */ }
```

```scss
// _button.import.scss (legacy API for @import consumers)
@forward 'button' as btn-*;
```

When a consumer writes `@import 'button'`, Sass loads `_button.import.scss` (which re-exports with a prefix to match the old global naming). When a consumer writes `@use 'button'`, Sass loads `_button.scss` directly.

## Mixing `@import` and `@use` During Migration

During incremental migration, you may have files using both systems. Key behaviors:

- `@use` rules must appear before any `@import` rules in the same file.
- A file loaded via `@use` is a singleton. A file loaded via `@import` gets a fresh copy each time.
- If file A does `@use 'config'` and file B does `@import 'config'`, they get **different instances**. Variable overrides applied via `@import` are not visible to the `@use` instance.

```scss
// main.scss
@use 'config' with ($primary: red);  // Module instance: $primary = red
@import 'legacy-component';           // This @import of 'config' creates a new global copy

// The legacy component sees the default $primary, not the configured one
```

Minimize the mixed period. Migrate bottom-up: leaf files first, then files that depend on them, working toward the entrypoint.

## `meta.load-css()` for Dynamic Loading

`meta.load-css()` fills the gap left by nested `@import`. It loads a module's CSS output at any point in your stylesheet.

```scss
@use 'sass:meta';

// Load theme CSS inside a selector
.theme-dark {
  @include meta.load-css('themes/dark');
}

// Load with configuration
.theme-custom {
  @include meta.load-css('themes/base', $with: (
    $bg: #1a1a1a,
    $text: #f0f0f0,
  ));
}
```

Key characteristics:

- Only includes CSS output. Does not expose variables, mixins, or functions.
- The URL can use interpolation for partially dynamic paths.
- Each call compiles the module's CSS fresh (not singleton behavior).
- The `&` parent selector inside the loaded module does not see the outer nesting context.

## Common Mistakes

### Mistake 1: Ignoring deprecation warnings

The warnings exist because `@import` will stop working. Treating them as noise delays inevitable work and risks a broken build when Dart Sass 3.0 releases.

### Mistake 2: Using `--fatal-deprecation` with unmigrated dependencies

```bash
# This will fail if ANY dependency uses @import
sass --fatal-deprecation=import main.scss output.css
```

Use `--fatal-deprecation` only after all code and dependencies are migrated. During migration, use `--quiet-deps` instead.

### Mistake 3: Configuring a module after it was already loaded

```scss
// helpers.scss (loaded early, does @use 'config' without configuration)
@use 'config';

// main.scss (tries to configure later)
@use 'config' with ($primary: red);  // ERROR: already loaded
```

Configure shared modules at the very top of your entrypoint, before any other file loads them.

### Mistake 4: Expecting `@forward` to provide local access

```scss
// _index.scss
@forward 'variables';
.debug { color: $primary; }  // ERROR: $primary not available here
```

`@forward` only re-exports. Add `@use 'variables'` for local access.

### Mistake 5: Running `sass-migrator` without `--migrate-deps`

```bash
# Only migrates main.scss, not its dependencies
sass-migrator module main.scss

# Migrates main.scss AND all files it loads
sass-migrator module --migrate-deps main.scss
```

Without `--migrate-deps`, only the explicitly named files are converted. Dependencies remain as `@import`.

## Best Practices

### Practice 1: Migrate incrementally, bottom-up

Start with leaf files (no dependencies), then work up the tree. This minimizes the period where `@import` and `@use` coexist.

### Practice 2: Use `--quiet-deps` immediately

Add this flag to your build as soon as you upgrade to Dart Sass 1.80.0+. It separates your warnings from third-party noise.

### Practice 3: Run `--dry-run` before any automated migration

```bash
sass-migrator module --migrate-deps --dry-run --verbose src/main.scss
```

Review the proposed changes before applying them.

### Practice 4: Verify compiled output after migration

Diff the compiled CSS before and after migration. The module system may change CSS output order, which can affect cascade-dependent styles.

### Practice 5: Pin Sass version in `package.json`

```json
"sass": ">=1.80.0 <3.0.0"
```

This protects against unexpected breakage from a major version upgrade.

### Practice 6: Set a team deadline for migration

Do not let deprecation debt accumulate. Set a target date for completing the migration and track progress.

## Real-World Use Cases

### Use Case 1: Migrating a Bootstrap-Dependent Project

Bootstrap 5.x still uses `@import`. You cannot migrate Bootstrap's internals, but you can migrate your own code:

```scss
// BEFORE: main.scss
$primary: #e74c3c;
@import 'bootstrap/scss/bootstrap';
@import 'components/header';
@import 'components/card';
```

```scss
// AFTER: main.scss

// Bootstrap still requires @import (silence the warnings)
$primary: #e74c3c;
@import 'bootstrap/scss/bootstrap';

// Your own code uses @use
@use 'components/header';
@use 'components/card';
```

```bash
# Build command with warnings managed
sass --quiet-deps src/main.scss dist/main.css
```

Your code is migrated. Bootstrap's warnings are silenced until Bootstrap 6 ships with module system support.

### Use Case 2: Migrating a Design Token Library

```scss
// BEFORE: tokens/_index.scss
@import 'colors';
@import 'spacing';
@import 'typography';

// BEFORE: Consumer
@import 'tokens';
.card { color: $color-primary; padding: $space-md; }
```

```scss
// AFTER: tokens/_index.scss
@forward 'colors' as color-*;
@forward 'spacing' as space-*;
@forward 'typography' as type-*;

// AFTER: Consumer
@use 'tokens' as t;
.card { color: t.$color-primary; padding: t.$space-md; }
```

The prefix pattern (`as color-*`) avoids collisions between modules that might share generic names like `$base` or `$primary`.

### Use Case 3: Gradual CI Enforcement

Phase 1 (immediate):
```bash
# Suppress all @import warnings
sass --quiet-deps --silence-deprecation=import src/main.scss dist/main.css
```

Phase 2 (after migrating your code):
```bash
# Show warnings from your code, suppress from deps
sass --quiet-deps src/main.scss dist/main.css
```

Phase 3 (after all deps are migrated):
```bash
# Fail the build on any @import
sass --fatal-deprecation=import src/main.scss dist/main.css
```

---

> **Key Takeaway:** `@import` is deprecated because it creates global scope pollution, CSS duplication, and untraceable dependencies. Migrate to `@use`/`@forward` incrementally starting from leaf files, use `--quiet-deps` to manage third-party warnings, and pin your Sass version below 3.0.0 until your entire dependency tree is migrated.
