# 5.1 Variables and Scope

> JavaScript has three variable declaration keywords, four types of scope, and a set of hoisting rules that trip up even experienced developers. Understanding the mechanics behind `var`, `let`, and `const` is the foundation for writing predictable, bug-free code.

---

## Declaration Keywords: `var`, `let`, and `const`

### Quick Comparison

| Feature | `var` | `let` | `const` |
|---|---|---|---|
| Scope | Function | Block | Block |
| Hoisting | Yes (initialized to `undefined`) | Yes (uninitialized, TDZ) | Yes (uninitialized, TDZ) |
| Redeclaration in same scope | Allowed (silent) | SyntaxError | SyntaxError |
| Reassignment | Allowed | Allowed | TypeError |
| Must initialize at declaration | No | No | Yes |
| Creates `window` property (global) | Yes | No | No |

### `var`: Function-Scoped

`var` declarations are scoped to the nearest enclosing function, not to blocks. They are hoisted to the top of that function and initialized with `undefined`.

```javascript
function example() {
  console.log(x); // undefined (hoisted, initialized to undefined)
  if (true) {
    var x = 10;
  }
  console.log(x); // 10 (leaked out of the if-block)
}
```

`var` at the global level attaches to the `window` object:

```javascript
var globalVar = 'hello';
console.log(window.globalVar); // "hello"
```

`var` allows silent redeclaration:

```javascript
var count = 1;
var count = 2; // No error, silently overwrites
console.log(count); // 2
```

### `let`: Block-Scoped, Reassignable

`let` declarations are confined to the nearest `{}` block. They are hoisted but remain uninitialized until the declaration line is reached.

```javascript
function example() {
  if (true) {
    let x = 10;
    console.log(x); // 10
  }
  // console.log(x); // ReferenceError: x is not defined
}
```

`let` forbids redeclaration in the same scope:

```javascript
let count = 1;
// let count = 2; // SyntaxError: Identifier 'count' has already been declared
```

### `const`: Block-Scoped, Non-Reassignable Binding

`const` works like `let` but prevents reassignment of the binding. It must be initialized at declaration.

```javascript
const API_URL = 'https://api.example.com';
// API_URL = 'https://other.com'; // TypeError: Assignment to constant variable

// const x; // SyntaxError: Missing initializer in const declaration
```

**Critical distinction: `const` protects the binding, not the value.**

```javascript
const user = { name: 'Alice', age: 30 };
user.age = 31;          // Works fine. Mutating the object, not the binding.
user.email = 'a@b.com'; // Works fine. Adding a property.
// user = {};            // TypeError. Reassigning the binding.

const nums = [1, 2, 3];
nums.push(4);    // Works. The array reference is the same.
nums[0] = 99;   // Works.
// nums = [5, 6]; // TypeError.
```

## Hoisting

Hoisting is the process where the JavaScript engine processes declarations during a compile phase before executing code. Declarations are "moved" conceptually to the top of their scope.

### `var` Hoisting

The declaration is hoisted and initialized to `undefined`. The assignment stays in place.

```javascript
console.log(a); // undefined
var a = 5;
console.log(a); // 5

// The engine sees this as:
// var a = undefined;
// console.log(a);
// a = 5;
// console.log(a);
```

**Gotcha: `var` in a false conditional still hoists the declaration.**

```javascript
function test() {
  console.log(x); // undefined (not ReferenceError)
  if (false) {
    var x = 10; // Never executes, but the declaration is hoisted
  }
  console.log(x); // undefined
}
test();
```

### `let` and `const` Hoisting (The Temporal Dead Zone)

`let` and `const` are hoisted to the top of their block scope but are NOT initialized. The region between the start of the scope and the declaration line is the Temporal Dead Zone (TDZ). Accessing the variable in the TDZ throws a `ReferenceError`.

```javascript
{
  // --- TDZ for `x` starts here ---
  // console.log(x); // ReferenceError: Cannot access 'x' before initialization
  let x = 10;
  // --- TDZ for `x` ends here ---
  console.log(x); // 10
}
```

This proves `let`/`const` ARE hoisted:

```javascript
let x = 'outer';
{
  // If `let x` below were not hoisted, this would print 'outer'
  // Instead, it throws ReferenceError because `x` is in the TDZ
  // console.log(x); // ReferenceError
  let x = 'inner';
  console.log(x); // 'inner'
}
```

**Gotcha: `typeof` is NOT safe in the TDZ.**

```javascript
// typeof on a completely undeclared variable is safe
console.log(typeof neverDeclared); // "undefined" (no error)

// typeof on a TDZ variable throws
{
  // console.log(typeof myVar); // ReferenceError: Cannot access 'myVar' before initialization
  let myVar = 42;
}
```

### Function Declaration Hoisting

Function declarations are fully hoisted: both the name and the body. They can be called before their declaration.

```javascript
greet(); // "Hello!" (works because the entire function is hoisted)

function greet() {
  console.log('Hello!');
}
```

Function expressions assigned to `var` only hoist the variable, not the function body:

```javascript
// greet(); // TypeError: greet is not a function

var greet = function () {
  console.log('Hello!');
};

greet(); // "Hello!"
```

Function expressions with `let`/`const` hit the TDZ:

```javascript
// greet(); // ReferenceError: Cannot access 'greet' before initialization

const greet = function () {
  console.log('Hello!');
};
```

### Hoisting Priority

When both a `var` declaration and a function declaration exist for the same name, the function declaration takes priority during hoisting:

```javascript
console.log(foo); // [Function: foo] (not undefined)

var foo = 'bar';
function foo() {}

console.log(foo); // "bar" (assignment overwrites at runtime)
```

## Scope Types

### Global Scope

Variables declared outside any function or block. In browsers, `var` declarations become properties of `window`. `let` and `const` do not.

```javascript
var a = 1;
let b = 2;
const c = 3;

console.log(window.a); // 1
console.log(window.b); // undefined
console.log(window.c); // undefined

// All three are accessible globally
console.log(a, b, c); // 1 2 3
```

### Function Scope

Created by function declarations and function expressions. `var` is function-scoped.

```javascript
function process() {
  var result = 42;
  if (true) {
    var inner = 100; // Still function-scoped
  }
  console.log(result); // 42
  console.log(inner);  // 100 (accessible: var ignores block boundaries)
}
// console.log(result); // ReferenceError: result is not defined
```

### Block Scope

Created by any `{}` block: `if`, `for`, `while`, `switch`, or even a standalone block. `let` and `const` are block-scoped.

```javascript
{
  let blockScoped = 'only here';
  const alsoBlockScoped = 'only here too';
}
// console.log(blockScoped);     // ReferenceError
// console.log(alsoBlockScoped); // ReferenceError
```

**Standalone blocks are valid scope creators:**

```javascript
function calculate() {
  let total = 0;

  {
    // Temporary calculation scope
    let temp = 100;
    let discount = 0.2;
    total = temp - temp * discount;
  }

  // console.log(temp);     // ReferenceError
  // console.log(discount); // ReferenceError
  return total; // 80
}
```

### Module Scope

Each ES module has its own top-level scope. Variables are not added to the global object.

```javascript
// utils.js
const SECRET = 'abc123'; // Module-scoped, not global
export const publicValue = 42;

// main.js
import { publicValue } from './utils.js';
console.log(publicValue); // 42
// console.log(SECRET);   // ReferenceError (not exported)
```

ES modules run in strict mode by default, which eliminates accidental globals.

## The Scope Chain

When the engine encounters a variable reference, it searches the current scope first, then walks outward through each enclosing scope until it reaches the global scope. This lookup path is the scope chain.

```javascript
const global = 'G';

function outer() {
  const outerVar = 'O';

  function middle() {
    const middleVar = 'M';

    function inner() {
      const innerVar = 'I';

      // Scope chain lookup:
      // inner -> middle -> outer -> global
      console.log(innerVar);  // 'I' (found in inner)
      console.log(middleVar); // 'M' (found in middle)
      console.log(outerVar);  // 'O' (found in outer)
      console.log(global);    // 'G' (found in global)
    }

    inner();
  }

  middle();
}

outer();
```

The scope chain is determined at **author time** (lexical scoping), not at call time. Where you write the function determines its scope chain, not where you call it.

```javascript
const x = 'global';

function foo() {
  console.log(x); // Looks up x in foo's lexical scope chain
}

function bar() {
  const x = 'local';
  foo(); // Prints 'global', NOT 'local'
}

bar();
```

## Closures

A closure is a function that retains access to its lexical scope even after the outer function has returned. Every function in JavaScript creates a closure.

### Basic Closure

```javascript
function createCounter() {
  let count = 0; // Private variable

  return {
    increment() { count++; },
    decrement() { count--; },
    getCount() { return count; }
  };
}

const counter = createCounter();
counter.increment();
counter.increment();
counter.increment();
counter.decrement();
console.log(counter.getCount()); // 2

// count is inaccessible from outside
// console.log(count); // ReferenceError
```

### Closures Capture References, Not Values

This is the single most important thing to understand about closures. The closed-over variable is a live reference, not a snapshot.

```javascript
function makeGreeter() {
  let greeting = 'Hello';

  const greet = () => console.log(greeting);

  greeting = 'Goodbye'; // Mutate after creating the closure

  return greet;
}

makeGreeter()(); // "Goodbye" (not "Hello")
```

### Function Factories

```javascript
function makeMultiplier(factor) {
  return function (num) {
    return num * factor; // factor is captured from the outer scope
  };
}

const double = makeMultiplier(2);
const triple = makeMultiplier(3);

console.log(double(5));  // 10
console.log(triple(5));  // 15
console.log(double(10)); // 20
```

### The Module Pattern (IIFE + Closure)

Before ES modules, closures were the primary mechanism for encapsulation:

```javascript
const UserService = (function () {
  const users = []; // Private state
  let nextId = 1;   // Private state

  function findIndex(id) { // Private function
    return users.findIndex(u => u.id === id);
  }

  return {
    add(name) {
      users.push({ id: nextId++, name });
    },
    remove(id) {
      const idx = findIndex(id);
      if (idx !== -1) users.splice(idx, 1);
    },
    getAll() {
      return [...users]; // Return a copy
    }
  };
})();

UserService.add('Alice');
UserService.add('Bob');
console.log(UserService.getAll()); // [{ id: 1, name: 'Alice' }, { id: 2, name: 'Bob' }]
// console.log(UserService.users); // undefined (private)
```

## The Classic Closure-in-Loop Problem

This is the most frequently tested closure scenario and is rooted in how `var` and `let` handle loop iterations.

### The Problem with `var`

```javascript
// BAD: All callbacks print 5
for (var i = 0; i < 5; i++) {
  setTimeout(function () {
    console.log(i);
  }, i * 100);
}
// Output: 5, 5, 5, 5, 5
```

Why: `var i` is function-scoped. There is one `i` shared by all iterations. By the time the callbacks fire, the loop has finished and `i` is 5.

### Solution 1: Use `let` (Recommended)

```javascript
// GOOD: Each iteration gets its own binding
for (let i = 0; i < 5; i++) {
  setTimeout(function () {
    console.log(i);
  }, i * 100);
}
// Output: 0, 1, 2, 3, 4
```

This works because the ECMAScript spec defines a special behavior for `let` in `for` loops: a new lexical environment is created for each iteration, and the loop variable is re-initialized with the value from the end of the previous iteration. This is not merely "block scoping" but a per-iteration binding mechanism.

### Solution 2: IIFE (Historical)

```javascript
// Historical fix before let existed
for (var i = 0; i < 5; i++) {
  (function (j) {
    setTimeout(function () {
      console.log(j);
    }, j * 100);
  })(i);
}
// Output: 0, 1, 2, 3, 4
```

### Solution 3: setTimeout's Third Argument

```javascript
for (var i = 0; i < 5; i++) {
  setTimeout(function (j) {
    console.log(j);
  }, i * 100, i); // Third arg is passed to the callback
}
// Output: 0, 1, 2, 3, 4
```

## Variable Shadowing

Shadowing occurs when a variable in an inner scope has the same name as one in an outer scope. The inner variable "shadows" the outer one within its scope.

### Legal Shadowing

```javascript
const x = 'outer';

function example() {
  const x = 'function'; // Shadows outer x

  if (true) {
    const x = 'block'; // Shadows function x
    console.log(x);    // 'block'
  }

  console.log(x); // 'function'
}

example();
console.log(x); // 'outer'
```

### Illegal Shadowing

You cannot shadow a `let`/`const` variable with a `var` in a nested block within the same function scope. This is because `var` would hoist to the function scope, colliding with the `let`/`const` in that same scope.

```javascript
function illegal() {
  let x = 10;
  if (true) {
    // var x = 20; // SyntaxError: Identifier 'x' has already been declared
  }
}
```

The reverse is allowed: shadowing `var` with `let`/`const` in a nested block is fine.

```javascript
function legal() {
  var x = 10;
  if (true) {
    let x = 20; // Fine: let creates a new binding in the block scope
    console.log(x); // 20
  }
  console.log(x); // 10
}
```

## `switch` Statement Scope Gotcha

All `case` clauses in a `switch` share the same block scope. Declaring the same `let`/`const` in multiple cases causes a `SyntaxError`.

```javascript
// BAD: SyntaxError
function handleAction(action) {
  switch (action) {
    case 'create':
      let message = 'Created';
      break;
    case 'delete':
      // let message = 'Deleted'; // SyntaxError: Identifier 'message' has already been declared
      break;
  }
}
```

**Fix: Wrap each case in its own block.**

```javascript
// GOOD: Each case has its own scope
function handleAction(action) {
  switch (action) {
    case 'create': {
      let message = 'Created';
      console.log(message);
      break;
    }
    case 'delete': {
      let message = 'Deleted'; // No collision
      console.log(message);
      break;
    }
  }
}
```

## Object Immutability: `freeze`, `seal`, `preventExtensions`

Since `const` only prevents reassignment, JavaScript provides built-in methods to protect object values.

### Comparison Table

| Capability | `preventExtensions` | `seal` | `freeze` |
|---|---|---|---|
| Add new properties | No | No | No |
| Delete existing properties | Yes | No | No |
| Modify existing property values | Yes | Yes (if writable) | No |
| Reconfigure property descriptors | Yes | No | No |

### `Object.freeze()`

```javascript
const config = Object.freeze({
  apiUrl: 'https://api.example.com',
  timeout: 5000,
  retries: 3
});

config.timeout = 10000; // Silently fails in sloppy mode
console.log(config.timeout); // 5000 (unchanged)

// In strict mode:
'use strict';
// config.timeout = 10000; // TypeError: Cannot assign to read only property 'timeout'
```

### `Object.freeze()` is Shallow

```javascript
const settings = Object.freeze({
  theme: 'dark',
  notifications: {
    email: true,
    push: false
  }
});

// Top-level is frozen
// settings.theme = 'light'; // Fails

// Nested object is NOT frozen
settings.notifications.push = true; // Works!
console.log(settings.notifications.push); // true
```

### Deep Freeze Utility

```javascript
function deepFreeze(obj) {
  Object.freeze(obj);

  for (const key of Object.keys(obj)) {
    const value = obj[key];
    if (
      value !== null &&
      typeof value === 'object' &&
      !Object.isFrozen(value)
    ) {
      deepFreeze(value);
    }
  }

  return obj;
}

const config = deepFreeze({
  api: {
    url: 'https://api.example.com',
    headers: { 'Content-Type': 'application/json' }
  }
});

// config.api.url = 'https://other.com'; // TypeError in strict mode
// config.api.headers['Authorization'] = 'Bearer xyz'; // TypeError in strict mode
```

### `Object.seal()`

Prevents adding/removing properties but allows modifying existing writable properties:

```javascript
const user = Object.seal({ name: 'Alice', age: 30 });

user.age = 31;         // Works (modifying existing property)
// user.email = 'a@b'; // TypeError in strict mode (cannot add)
// delete user.name;   // TypeError in strict mode (cannot delete)
```

### Checking Object State

```javascript
const obj = { a: 1 };

console.log(Object.isExtensible(obj)); // true
Object.preventExtensions(obj);
console.log(Object.isExtensible(obj)); // false

const sealed = Object.seal({ b: 2 });
console.log(Object.isSealed(sealed)); // true

const frozen = Object.freeze({ c: 3 });
console.log(Object.isFrozen(frozen)); // true
```

## Strict Mode and Variables

Strict mode changes several variable behaviors. ES modules are strict by default.

### No Accidental Globals

```javascript
// Sloppy mode
function sloppy() {
  oops = 42; // Creates a global variable (window.oops)
}

// Strict mode
function strict() {
  'use strict';
  // oops = 42; // ReferenceError: oops is not defined
}
```

### No `var` Deletion

```javascript
'use strict';
var x = 10;
// delete x; // SyntaxError: Delete of an unqualified identifier in strict mode
```

### Reserved Words as Variable Names

```javascript
'use strict';
// let arguments = 5; // SyntaxError
// let eval = 10;     // SyntaxError
// let static = 20;   // SyntaxError (reserved in strict mode)
```

## Closures and Memory

Closures retain references to their enclosing lexical environment. This can prevent garbage collection if you are not careful.

### The Shared Scope Problem

All closures created in the same scope share the same lexical environment object. If one closure is kept alive, all variables in that scope are retained, even those used only by other closures.

```javascript
function createHandlers() {
  const largeData = new Array(1000000).fill('x'); // ~1MB
  const smallValue = 42;

  // Both closures share the same lexical environment
  const handler = () => console.log(smallValue);
  const unused = () => console.log(largeData.length);

  // Even if we never call `unused`, if `handler` is kept alive,
  // the engine may retain largeData because both closures share
  // the same scope.

  return handler;
}

const fn = createHandlers(); // largeData might be retained
```

### Mitigation: Null Out References

```javascript
function createHandlers() {
  let largeData = new Array(1000000).fill('x');
  const result = processData(largeData);

  largeData = null; // Allow garbage collection

  return () => console.log(result);
}
```

### Mitigation: Limit Closure Scope

```javascript
function createHandler() {
  const smallValue = (function () {
    const largeData = new Array(1000000).fill('x');
    return largeData.length; // Only extract what you need
  })();

  return () => console.log(smallValue);
}
```

## Common Mistakes

### 1. Using `var` in Modern Code

```javascript
// BAD: var leaks out of blocks, allows redeclaration
for (var i = 0; i < 10; i++) { /* ... */ }
console.log(i); // 10 (leaked)

// GOOD: let is block-scoped
for (let i = 0; i < 10; i++) { /* ... */ }
// console.log(i); // ReferenceError
```

### 2. Assuming `const` Means Immutable Value

```javascript
// BAD: Thinking const prevents mutation
const items = [1, 2, 3];
items.push(4); // Works! The array reference is the same.

// Use Object.freeze() if you need value immutability
const frozenItems = Object.freeze([1, 2, 3]);
// frozenItems.push(4); // TypeError
```

### 3. Forgetting TDZ with `typeof`

```javascript
// BAD: Assuming typeof is always safe
function check() {
  // console.log(typeof value); // ReferenceError (value is in TDZ)
  let value = 10;
}

// GOOD: Check after declaration, or check for truly undeclared vars
function checkSafe() {
  console.log(typeof neverDeclared); // "undefined" (safe for undeclared)
}
```

### 4. Implicit Globals in Sloppy Mode

```javascript
// BAD: Forgetting the declaration keyword
function calculate() {
  total = 0; // Creates window.total (implicit global)
  for (i = 0; i < 10; i++) { // Creates window.i
    total += i;
  }
  return total;
}
```

### 5. Confusing Hoisting Behavior

```javascript
// BAD: Assuming var in a function keeps outer value
var x = 1;
function confusing() {
  console.log(x); // undefined (not 1!)
  var x = 2;      // This var x is hoisted to function top
  console.log(x); // 2
}
confusing();
```

### 6. Closure Over Loop Variable with `var`

```javascript
// BAD: All handlers share the same i
const buttons = document.querySelectorAll('.btn');
for (var i = 0; i < buttons.length; i++) {
  buttons[i].addEventListener('click', function () {
    console.log('Button ' + i); // Always prints the final value of i
  });
}

// GOOD: Use let for per-iteration binding
for (let i = 0; i < buttons.length; i++) {
  buttons[i].addEventListener('click', function () {
    console.log('Button ' + i); // Correct value per button
  });
}
```

## Best Practices

1. **Default to `const`.** Only use `let` when you need to reassign. Never use `var` in new code.

2. **Use strict mode or ES modules.** Strict mode catches accidental globals and makes frozen object violations throw errors instead of failing silently.

3. **Declare variables close to their use.** Minimize the gap between declaration and first use.

4. **Initialize at declaration.** `let count = 0;` is better than declaring `let count;` and assigning later.

5. **Use ESLint rules.** Enable `no-var`, `prefer-const`, `no-shadow`, and `no-use-before-define` to catch issues automatically.

6. **Use `Object.freeze()` for config objects.** When you need true value immutability, `const` alone is not enough.

7. **Null out large references in closures.** When a closure only needs a derived value from a large dataset, extract the value and release the reference.

8. **Wrap `switch` cases in blocks.** Prevent `let`/`const` collisions across cases by giving each case its own `{}` scope.

## Real-World Use Cases

### Configuration Object with Enforced Immutability

```javascript
// config.js
const CONFIG = Object.freeze({
  API_BASE: 'https://api.example.com/v2',
  TIMEOUT_MS: 5000,
  MAX_RETRIES: 3,
  CACHE_TTL_MS: 60000,
  FEATURES: Object.freeze({
    darkMode: true,
    notifications: true,
    betaAccess: false
  })
});

export default CONFIG;

// usage.js
import CONFIG from './config.js';

async function fetchData(endpoint) {
  const url = `${CONFIG.API_BASE}/${endpoint}`;
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), CONFIG.TIMEOUT_MS);

  try {
    const response = await fetch(url, { signal: controller.signal });
    return response.json();
  } finally {
    clearTimeout(timeoutId);
  }
}
```

### Private State with Closures (Cache)

```javascript
function createCache(maxSize = 100) {
  const store = new Map();

  return {
    get(key) {
      return store.get(key) ?? null;
    },

    set(key, value) {
      if (store.size >= maxSize) {
        // Evict the oldest entry (first inserted)
        const firstKey = store.keys().next().value;
        store.delete(firstKey);
      }
      store.set(key, value);
    },

    has(key) {
      return store.has(key);
    },

    clear() {
      store.clear();
    },

    get size() {
      return store.size;
    }
  };
}

const cache = createCache(50);
cache.set('user:1', { name: 'Alice' });
cache.set('user:2', { name: 'Bob' });
console.log(cache.get('user:1')); // { name: 'Alice' }
console.log(cache.size);          // 2
// cache.store is undefined (private)
```

### Scoped Initialization with Block Scope

```javascript
async function initializeApp() {
  // Database connection scoped to initialization
  {
    const dbConfig = await loadConfig('database');
    const connection = await connectToDatabase(dbConfig);
    await runMigrations(connection);
    await connection.close();
    // dbConfig and connection are now out of scope and eligible for GC
  }

  // Server setup in its own scope
  {
    const serverConfig = await loadConfig('server');
    const app = createServer(serverConfig);
    app.listen(serverConfig.port);
    console.log(`Server running on port ${serverConfig.port}`);
  }
}
```

> **Key Takeaway:** Use `const` by default, `let` when reassignment is needed, and never `var`. Understand that hoisting applies to all declarations but only `var` initializes to `undefined` while `let`/`const` enter the Temporal Dead Zone. Closures capture variable references, not values, which is the root cause of most loop-related bugs and the foundation of encapsulation patterns.
