# 5.12 Classes

> Classes are syntactic sugar over JavaScript's prototypal inheritance, but they bring their own semantics, features, and gotchas. Understanding class fields, private members, static initialization blocks, inheritance rules, and composition patterns is essential for writing modern, maintainable object-oriented JavaScript.

---

## Class Declarations and Expressions

### Declaration

```js
class User {
  constructor(name) {
    this.name = name;
  }

  greet() {
    return `Hello, I'm ${this.name}`;
  }
}

const user = new User("Alice");
```

### Expression

```js
// Anonymous class expression
const User = class {
  constructor(name) { this.name = name; }
};

// Named class expression
const Admin = class AdminClass {
  constructor(name) { this.name = name; }
  whoAmI() { return AdminClass.name; } // "AdminClass" accessible inside
};

console.log(typeof AdminClass); // "undefined" (not accessible outside)
```

### Hoisting: Classes Are in the TDZ

Unlike function declarations, class declarations are subject to the Temporal Dead Zone. The name is hoisted but uninitialized.

```js
// ReferenceError: Cannot access 'User' before initialization
const u = new User("Alice");

class User {
  constructor(name) { this.name = name; }
}
```

```js
// Function declarations are fully hoisted (this works)
const result = greet();
function greet() { return "hello"; }
```

### Strict Mode by Default

Class bodies are always in strict mode, regardless of whether `"use strict"` appears.

```js
class Example {
  method() {
    // Strict mode is active here:
    // - 'this' is undefined in standalone calls (not window)
    // - Assigning to undeclared variables throws ReferenceError
    // - Duplicate parameter names are forbidden
  }
}
```

## Instance Fields

Fields declared in the class body are own properties set on each instance during construction.

```js
class Counter {
  count = 0;              // initialized to 0 per instance
  items = [];             // new array per instance (not shared)
  label;                  // initialized to undefined

  increment() {
    this.count++;
    return this.count;
  }
}

const a = new Counter();
const b = new Counter();
a.items.push("x");

console.log(a.items); // ["x"]
console.log(b.items); // [] (independent array)
```

### Field Initialization Order

Fields are initialized in **declaration order**, top to bottom. In a base class, fields initialize at the start of the constructor body. In a derived class, fields initialize immediately after `super()` returns.

```js
class Base {
  first = console.log("Base field 1");
  second = console.log("Base field 2");

  constructor() {
    console.log("Base constructor body");
  }
}

class Derived extends Base {
  third = console.log("Derived field");

  constructor() {
    console.log("Before super");
    super();
    // Derived fields initialize HERE, after super() returns
    console.log("After super");
  }
}

new Derived();
// "Before super"
// "Base field 1"
// "Base field 2"
// "Base constructor body"
// "Derived field"
// "After super"
```

### The Field Overwrite Gotcha

Derived class fields run after `super()`, which means they overwrite values set by the parent constructor.

```js
class Base {
  constructor() {
    this.value = 42; // set during base constructor
  }
}

class Derived extends Base {
  value = 0; // runs AFTER super(), overwrites 42
}

console.log(new Derived().value); // 0, not 42
```

### Base Constructor Cannot See Derived Fields

```js
class Base {
  constructor() {
    console.log(this.field); // undefined! Derived fields not yet initialized
  }
}

class Derived extends Base {
  field = "initialized";
}

new Derived(); // logs undefined, then field is set to "initialized"
```

### Field Property Descriptors

Public fields are own properties with `{ writable: true, enumerable: true, configurable: true }`. This means they appear in `Object.keys()`, unlike methods.

```js
class User {
  name = "Alice";
  greet() { return this.name; }
}

const u = new User();
console.log(Object.keys(u)); // ["name"] (field is enumerable)
// greet is on the prototype, NOT an own property, and is non-enumerable
```

## Private Fields and Methods

Private members use the `#` prefix. Access is enforced at the language level.

### Private Instance Fields

```js
class BankAccount {
  #balance = 0;
  #owner;

  constructor(owner, initialBalance) {
    this.#owner = owner;
    this.#balance = initialBalance;
  }

  deposit(amount) {
    this.#validateAmount(amount);
    this.#balance += amount;
  }

  withdraw(amount) {
    this.#validateAmount(amount);
    if (amount > this.#balance) {
      throw new Error("Insufficient funds");
    }
    this.#balance -= amount;
  }

  get balance() {
    return this.#balance;
  }

  // Private method
  #validateAmount(amount) {
    if (typeof amount !== "number" || amount <= 0) {
      throw new TypeError("Amount must be a positive number");
    }
  }
}

const acct = new BankAccount("Alice", 100);
acct.deposit(50);
console.log(acct.balance); // 150

// acct.#balance;     // SyntaxError (outside class body)
// acct.#validateAmount(10); // SyntaxError
```

### Private Getters and Setters

```js
class Temperature {
  #celsius;

  constructor(celsius) {
    this.#celsius = celsius;
  }

  get #fahrenheit() {
    return this.#celsius * 9 / 5 + 32;
  }

  set #fahrenheit(f) {
    this.#celsius = (f - 32) * 5 / 9;
  }

  toString() {
    return `${this.#celsius}C / ${this.#fahrenheit}F`;
  }
}
```

### Cross-Instance Private Access

One instance can access another instance's private fields if they are from the same class.

```js
class Vector {
  #x;
  #y;

  constructor(x, y) {
    this.#x = x;
    this.#y = y;
  }

  equals(other) {
    // Accessing another Vector instance's private fields
    return this.#x === other.#x && this.#y === other.#y;
  }

  add(other) {
    return new Vector(this.#x + other.#x, this.#y + other.#y);
  }
}

const a = new Vector(1, 2);
const b = new Vector(1, 2);
console.log(a.equals(b)); // true
```

### Key Rules for Private Members

- Must be declared up-front in the class body. Cannot be added dynamically.
- Not inherited by subclasses. A child class cannot access parent `#fields`.
- Not discoverable via reflection (`Object.keys`, `Object.getOwnPropertyNames`, `Reflect.ownKeys` all skip them).
- `delete this.#field` is a SyntaxError.
- Accessing `#field` on an object that does not have it throws TypeError.
- Incompatible with Proxy (accessing private fields through a proxy throws TypeError).

## Private Static Members

```js
class IdGenerator {
  static #nextId = 1;

  static generate() {
    return IdGenerator.#nextId++;
  }

  // Bad Practice: using 'this' for private static fields
  static generateBroken() {
    return this.#nextId++; // TypeError when called on a subclass!
  }
}

console.log(IdGenerator.generate()); // 1
console.log(IdGenerator.generate()); // 2
```

### The `this` Gotcha with Private Static Fields

```js
class Parent {
  static #value = 10;

  static getValue() {
    return this.#value; // 'this' is dynamic
  }
}

class Child extends Parent {}

Parent.getValue(); // 10 (this === Parent, works)
Child.getValue();  // TypeError! (this === Child, but Child doesn't have #value)
```

Always access private static fields via the class name directly, not via `this`.

```js
class Parent {
  static #value = 10;

  static getValue() {
    return Parent.#value; // safe: always refers to Parent
  }
}
```

## Static Fields, Methods, and Static Initialization Blocks

### Static Fields and Methods

```js
class MathUtils {
  static PI = 3.14159265359;
  static E = 2.71828182846;

  static circleArea(radius) {
    return MathUtils.PI * radius ** 2;
  }

  static factorial(n) {
    if (n <= 1) return 1;
    return n * MathUtils.factorial(n - 1);
  }
}

console.log(MathUtils.circleArea(5)); // 78.539...
console.log(MathUtils.factorial(5));  // 120

// Static methods are NOT on instances
const m = new MathUtils();
// m.circleArea(5); // TypeError: m.circleArea is not a function
```

### Static Inheritance

Static members are inherited through the constructor chain (`Child.__proto__ === Parent`).

```js
class Parent {
  static greet() { return "Hello from Parent"; }
}

class Child extends Parent {}

console.log(Child.greet()); // "Hello from Parent" (inherited)
```

### Static Initialization Blocks (ES2022)

Static blocks run once when the class is evaluated. They have privileged access to private fields.

```js
class Config {
  static #host;
  static #port;

  static {
    try {
      const rawConfig = readConfigFile("/etc/app.conf");
      Config.#host = rawConfig.host;
      Config.#port = rawConfig.port;
    } catch {
      Config.#host = "localhost";
      Config.#port = 3000;
    }
  }

  static getConnectionString() {
    return `${Config.#host}:${Config.#port}`;
  }
}
```

Multiple static blocks are allowed. They run in declaration order, interleaved with static field initializers.

```js
class Color {
  static RED = new Color(255, 0, 0);
  static GREEN = new Color(0, 255, 0);
  static BLUE = new Color(0, 0, 255);

  static ALL;
  static {
    Color.ALL = [Color.RED, Color.GREEN, Color.BLUE];
  }

  constructor(r, g, b) {
    this.r = r;
    this.g = g;
    this.b = b;
  }
}
```

### The "Friend" Pattern via Static Blocks

Static blocks can expose private field access to external code.

```js
let getSecret;

class Vault {
  #secret;

  constructor(secret) {
    this.#secret = secret;
  }

  static {
    // Expose private access to trusted code
    getSecret = (vault) => vault.#secret;
  }
}

const v = new Vault("password123");
console.log(getSecret(v)); // "password123"
```

### Static Block Restrictions

- Cannot use `await` (not an async context).
- Cannot use `return`.
- Cannot access `arguments`.
- `this` refers to the class constructor itself.

## Inheritance with `extends` and `super`

### Basic Inheritance

```js
class Animal {
  constructor(name) {
    this.name = name;
  }

  speak() {
    return `${this.name} makes a noise`;
  }
}

class Dog extends Animal {
  constructor(name, breed) {
    super(name); // must call before accessing 'this'
    this.breed = breed;
  }

  speak() {
    return `${this.name} barks`;
  }

  parentSpeak() {
    return super.speak(); // calls Animal.prototype.speak
  }
}

const rex = new Dog("Rex", "Labrador");
console.log(rex.speak());       // "Rex barks"
console.log(rex.parentSpeak()); // "Rex makes a noise"
```

### Constructor Rules in Derived Classes

1. If a derived class has a constructor, it **must** call `super()` before accessing `this`.
2. If a derived class omits the constructor, an implicit one is generated: `constructor(...args) { super(...args); }`.
3. Failing to call `super()` before `this` throws `ReferenceError`.

```js
class Broken extends Animal {
  constructor(name) {
    this.extra = true; // ReferenceError: Must call super constructor first
    super(name);
  }
}
```

### What `extends` Sets Up

Two prototype links:

```js
class Parent {}
class Child extends Parent {}

// Instance method inheritance
Object.getPrototypeOf(Child.prototype) === Parent.prototype; // true

// Static method inheritance
Object.getPrototypeOf(Child) === Parent; // true
```

### `extends` Accepts Expressions

The right side of `extends` can be any expression that evaluates to a constructor or `null`.

```js
function getBaseClass(type) {
  if (type === "array") return Array;
  if (type === "map") return Map;
  return Object;
}

class MyCollection extends getBaseClass("array") {
  first() {
    return this[0];
  }
}

const c = new MyCollection(1, 2, 3);
console.log(c.first()); // 1
console.log(c instanceof Array); // true
```

### `super` Is Statically Bound

`super` uses the internal `[[HomeObject]]` slot, which is set when the method is defined. Moving a method to a different object does not change what `super` refers to.

```js
class A {
  method() { return "A"; }
}

class B extends A {
  method() { return super.method(); } // super is always A, regardless of where this method is placed
}

const b = new B();
const extracted = b.method;

// Even if you call extracted differently, super.method still refers to A.prototype.method
```

## Brand Checking with Private Fields

The `in` operator can check for private field presence, providing reliable type checking.

```js
class Color {
  #red;
  #green;
  #blue;

  constructor(r, g, b) {
    this.#red = r;
    this.#green = g;
    this.#blue = b;
  }

  static isColor(value) {
    return #red in value; // true only if value has Color's #red field
  }

  mix(other) {
    if (!Color.isColor(other)) {
      throw new TypeError("Argument must be a Color instance");
    }
    return new Color(
      (this.#red + other.#red) / 2,
      (this.#green + other.#green) / 2,
      (this.#blue + other.#blue) / 2
    );
  }
}

const red = new Color(255, 0, 0);
const blue = new Color(0, 0, 255);

console.log(Color.isColor(red));    // true
console.log(Color.isColor({}));     // false
console.log(Color.isColor("red"));  // false
```

### Combining with `Symbol.hasInstance`

```js
class Color {
  #brand;

  constructor() {
    this.#brand = true;
  }

  static [Symbol.hasInstance](value) {
    return #brand in value;
  }
}

const c = new Color();
console.log(c instanceof Color); // true (uses brand check, not prototype chain)

// This is more reliable than default instanceof:
// - Works across realms
// - Cannot be fooled by prototype manipulation
```

## Abstract Class Patterns

JavaScript does not have a native `abstract` keyword, but you can simulate it.

### Using `new.target`

```js
class Shape {
  constructor() {
    if (new.target === Shape) {
      throw new Error("Shape is abstract and cannot be instantiated directly");
    }
  }

  // Abstract method
  area() {
    throw new Error(`${this.constructor.name} must implement area()`);
  }

  // Concrete method
  describe() {
    return `${this.constructor.name} with area ${this.area()}`;
  }
}

class Circle extends Shape {
  #radius;

  constructor(radius) {
    super();
    this.#radius = radius;
  }

  area() {
    return Math.PI * this.#radius ** 2;
  }
}

// new Shape();          // Error: Shape is abstract
new Circle(5).describe(); // "Circle with area 78.539..."
```

### Enforcing Method Implementation

```js
class Plugin {
  constructor() {
    if (new.target === Plugin) {
      throw new Error("Plugin is abstract");
    }

    // Verify required methods exist on the subclass
    const required = ["initialize", "execute", "cleanup"];
    for (const method of required) {
      if (typeof this[method] !== "function") {
        throw new Error(`${this.constructor.name} must implement ${method}()`);
      }
    }
  }
}

class LoggerPlugin extends Plugin {
  initialize() { /* ... */ }
  execute() { /* ... */ }
  cleanup() { /* ... */ }
}

class BrokenPlugin extends Plugin {
  initialize() { /* ... */ }
  // Missing execute and cleanup
}

new LoggerPlugin();  // works
new BrokenPlugin();  // Error: BrokenPlugin must implement execute()
```

## Mixins with Subclass Factories

The canonical pattern for multiple inheritance in JavaScript.

```js
const Serializable = (Base) =>
  class extends Base {
    serialize() {
      const obj = {};
      for (const key of Object.keys(this)) {
        obj[key] = this[key];
      }
      return JSON.stringify(obj);
    }

    static deserialize(json) {
      return Object.assign(new this(), JSON.parse(json));
    }
  };

const Validatable = (Base) =>
  class extends Base {
    #errors = [];

    get errors() { return [...this.#errors]; }
    get isValid() { return this.#errors.length === 0; }

    addError(message) {
      this.#errors.push(message);
    }

    clearErrors() {
      this.#errors = [];
    }
  };

const Timestamped = (Base) =>
  class extends Base {
    createdAt = new Date();
    updatedAt = new Date();

    touch() {
      this.updatedAt = new Date();
      return this;
    }
  };

// Compose: read inside-out, leftmost = closest in the chain
class User extends Timestamped(Validatable(Serializable(Object))) {
  constructor(name, email) {
    super();
    this.name = name;
    this.email = email;
  }
}

const user = new User("Alice", "alice@example.com");
user.touch();
user.addError("Email not verified");
console.log(user.isValid);      // false
console.log(user.serialize());  // JSON string with name, email, createdAt, updatedAt
```

### Mixin Order Matters

The last mixin applied is closest to the subclass in the prototype chain, so its methods take highest priority.

```js
const A = (Base) => class extends Base {
  method() { return "A"; }
};

const B = (Base) => class extends Base {
  method() { return "B"; }
};

class Test1 extends B(A(Object)) {}
console.log(new Test1().method()); // "B" (B is closer in the chain)

class Test2 extends A(B(Object)) {}
console.log(new Test2().method()); // "A" (A is closer in the chain)
```

## Class vs Factory Function

| Aspect | Class | Factory Function |
|---|---|---|
| Methods shared | Yes (prototype) | No (per-instance via closures) |
| True privacy | `#private` (ES2022) | Closure variables |
| `this` management | Must handle binding | No `this` issues |
| `instanceof` | Works | Does not work |
| `extends`/`super` | Built-in | Manual composition |
| V8 optimization | Hidden classes, inline caching | Less optimizable |
| Memory (many instances) | Efficient (shared methods) | Higher (per-instance methods) |

```js
// Class approach
class Counter {
  #count = 0;
  increment() { return ++this.#count; }
  getCount() { return this.#count; }
}

// Factory approach
function createCounter() {
  let count = 0;
  return {
    increment() { return ++count; },
    getCount() { return count; }
  };
}
```

Use classes when you need inheritance, `instanceof`, or are creating many instances. Use factories for simple encapsulated objects where `this` management is undesirable.

## Method Types

### Prototype Methods (Standard)

```js
class Example {
  // Lives on Example.prototype
  // Shared across all instances
  // Non-enumerable
  method() {
    return this;
  }
}
```

### Arrow Function Fields (Auto-Bound)

```js
class Example {
  // Created as an own property on each instance
  // Not shared (new function per instance)
  // Enumerable
  // 'this' is always the instance
  method = () => {
    return this;
  };
}
```

| Aspect | Prototype Method | Arrow Field |
|---|---|---|
| Location | Prototype (shared) | Instance (per-instance) |
| `this` binding | Dynamic (call-site dependent) | Lexical (always the instance) |
| Memory | O(1) | O(n) for n instances |
| Overridable by subclass | Yes | No (own property shadows prototype) |
| `super` usage | Yes | No (no `[[HomeObject]]`) |
| Enumerable | No | Yes |

```js
// When to use arrow fields: callbacks that need bound 'this'
class Button {
  label = "Click me";

  // This will be passed as a callback; arrow ensures 'this' is always the Button
  handleClick = () => {
    console.log(this.label);
  };
}

const btn = new Button();
document.querySelector("button").addEventListener("click", btn.handleClick);
// 'this' is correctly the Button instance, not the DOM element
```

### Generator and Async Methods

```js
class DataStream {
  #items;

  constructor(items) {
    this.#items = items;
  }

  // Generator method
  *[Symbol.iterator]() {
    for (const item of this.#items) {
      yield item;
    }
  }

  // Async method
  async fetchItem(id) {
    const response = await fetch(`/api/items/${id}`);
    return response.json();
  }

  // Async generator method
  async *fetchAll(ids) {
    for (const id of ids) {
      yield await this.fetchItem(id);
    }
  }
}

const stream = new DataStream([1, 2, 3]);
console.log([...stream]); // [1, 2, 3]
```

## Well-Known Symbols in Classes

### Making Instances Iterable

```js
class Range {
  #start;
  #end;

  constructor(start, end) {
    this.#start = start;
    this.#end = end;
  }

  *[Symbol.iterator]() {
    for (let i = this.#start; i <= this.#end; i++) {
      yield i;
    }
  }
}

const range = new Range(1, 5);
console.log([...range]); // [1, 2, 3, 4, 5]

for (const n of range) {
  console.log(n); // 1, 2, 3, 4, 5
}
```

### Custom String Tag

```js
class Money {
  #amount;
  #currency;

  constructor(amount, currency) {
    this.#amount = amount;
    this.#currency = currency;
  }

  get [Symbol.toStringTag]() {
    return "Money";
  }

  [Symbol.toPrimitive](hint) {
    if (hint === "number") return this.#amount;
    if (hint === "string") return `${this.#amount} ${this.#currency}`;
    return this.#amount;
  }
}

const price = new Money(29.99, "USD");
console.log(Object.prototype.toString.call(price)); // "[object Money]"
console.log(`${price}`);  // "29.99 USD"
console.log(+price);      // 29.99
```

## Decorators (TC39 Stage 3)

Decorators are a proposal for adding metadata and wrapping class elements. They are available via transpilation (Babel, TypeScript 5.0+) but not yet natively shipped in browsers.

```js
// A method decorator that logs calls
function logged(value, context) {
  const name = context.name;

  return function (...args) {
    console.log(`Calling ${name} with`, args);
    const result = value.call(this, ...args);
    console.log(`${name} returned`, result);
    return result;
  };
}

class Calculator {
  @logged
  add(a, b) {
    return a + b;
  }
}

new Calculator().add(2, 3);
// "Calling add with [2, 3]"
// "add returned 5"
```

Decorator functions receive `(value, context)` where `context` contains `kind`, `name`, `static`, `private`, `access`, `addInitializer`, and `metadata`.

Note: The standard decorators proposal differs significantly from the legacy TypeScript/Babel decorators. Legacy decorators receive `(target, name, descriptor)`, while standard decorators receive `(value, context)`.

## Common Mistakes

- **Using a class before its declaration**: Classes are in the TDZ, unlike function declarations. You will get a `ReferenceError`.

- **Forgetting `super()` in derived constructors**: Accessing `this` before `super()` throws `ReferenceError`. If you omit the constructor entirely, an implicit one calls `super()` for you.

- **Derived fields overwriting parent constructor values**: Fields declared on a derived class initialize after `super()` returns, silently overwriting values set during the parent's constructor.

- **Accessing private static fields via `this`**: When a static method is called on a subclass, `this` is the subclass, which does not have the parent's private static fields. Always use the class name directly.

- **Using arrow fields everywhere for "safety"**: Arrow fields are per-instance, not shared on the prototype. They consume more memory, cannot be overridden by subclasses, and do not support `super`. Use them only when you need auto-bound `this` (typically for event handlers).

- **Expecting private fields to be inherited**: Private fields are per-class, not inherited. A subclass cannot read or write the parent's `#fields`.

- **Relying on `instanceof` across realms**: `instanceof` fails for objects from different iframes or VM contexts. Use brand checks (`#field in obj`) or `Array.isArray()` for arrays.

---

## Best Practices

- Declare all instance fields at the top of the class body for clarity and V8 shape optimization.
- Use `#private` fields for internal state instead of the `_` naming convention.
- Prefer prototype methods over arrow function fields unless you specifically need auto-bound `this` for callbacks.
- Access private static fields by class name, not `this`.
- Use static initialization blocks for complex setup logic with try/catch or multi-step computation.
- Keep inheritance hierarchies shallow. Prefer composition via mixins over deep chains.
- Use `new.target` to simulate abstract classes when direct instantiation should be prevented.
- Use `#field in obj` for reliable type/brand checking instead of `instanceof`.
- Initialize fields in a consistent order across instances to help V8 maintain monomorphic hidden classes.

## Real-World Use Cases

### Event Emitter

```js
class EventEmitter {
  #listeners = new Map();

  on(event, callback) {
    if (!this.#listeners.has(event)) {
      this.#listeners.set(event, []);
    }
    this.#listeners.get(event).push(callback);
    return this;
  }

  off(event, callback) {
    const callbacks = this.#listeners.get(event);
    if (callbacks) {
      this.#listeners.set(
        event,
        callbacks.filter((cb) => cb !== callback)
      );
    }
    return this;
  }

  emit(event, ...args) {
    const callbacks = this.#listeners.get(event) || [];
    for (const callback of callbacks) {
      callback(...args);
    }
    return this;
  }

  once(event, callback) {
    const wrapper = (...args) => {
      this.off(event, wrapper);
      callback(...args);
    };
    return this.on(event, wrapper);
  }
}

const emitter = new EventEmitter();
emitter
```

### State Machine

```js
class StateMachine {
  #state;
  #transitions;
  #handlers = new Map();

  constructor(initialState, transitions) {
    this.#state = initialState;
    this.#transitions = new Map(
      transitions.map(({ from, event, to }) => [`${from}:${event}`, to])
    );
  }

  get state() {
    return this.#state;
  }

  onTransition(callback) {
    this.#handlers.set("transition", [
      ...(this.#handlers.get("transition") || []),
      callback
    ]);
    return this;
  }

  send(event) {
    const key = `${this.#state}:${event}`;
    const nextState = this.#transitions.get(key);

    if (!nextState) {
      throw new Error(`No transition for event "${event}" in state "${this.#state}"`);
    }

    const prevState = this.#state;
    this.#state = nextState;

    const handlers = this.#handlers.get("transition") || [];
    for (const handler of handlers) {
      handler({ from: prevState, to: nextState, event });
    }

    return this;
  }

  static #isMachine = true;
  static isMachine(value) {
    return value !== null && typeof value === "object" && #state in value;
  }
}

const light = new StateMachine("red", [
  { from: "red", event: "next", to: "green" },
  { from: "green", event: "next", to: "yellow" },
  { from: "yellow", event: "next", to: "red" }
]);

light.onTransition(({ from, to }) => console.log(`${from} -> ${to}`));
light.send("next"); // "red -> green"
light.send("next"); // "green -> yellow"
light.send("next"); // "yellow -> red"
```

### Repository Pattern with Composable Mixins

```js
const Cacheable = (Base) =>
  class extends Base {
    #cache = new Map();

    getCached(key) {
      return this.#cache.get(key);
    }

    setCached(key, value, ttl = 60000) {
      this.#cache.set(key, { value, expiry: Date.now() + ttl });
      return value;
    }

    isCacheValid(key) {
      const entry = this.#cache.get(key);
      return entry && entry.expiry > Date.now();
    }
  };

const Loggable = (Base) =>
  class extends Base {
    #logPrefix = this.constructor.name;

    log(level, message) {
      console.log(`[${level.toUpperCase()}] ${this.#logPrefix}: ${message}`);
    }
  };

class BaseRepository {
  #baseUrl;

  constructor(baseUrl) {
    this.#baseUrl = baseUrl;
  }

  async fetch(path) {
    const response = await fetch(`${this.#baseUrl}${path}`);
    if (!response.ok) throw new Error(`HTTP ${response.status}`);
    return response.json();
  }
}

class UserRepository extends Loggable(Cacheable(BaseRepository)) {
  constructor() {
    super("/api");
  }

  async getUser(id) {
    const cacheKey = `user:${id}`;

    if (this.isCacheValid(cacheKey)) {
      this.log("debug", `Cache hit for user ${id}`);
      return this.getCached(cacheKey).value;
    }

    this.log("info", `Fetching user ${id}`);
    const user = await this.fetch(`/users/${id}`);
    return this.setCached(cacheKey, user);
  }
}
```

Understand field initialization timing, the distinction between prototype methods and arrow fields, and prefer shallow hierarchies with mixin composition over deep inheritance chains.
