# 5.14 Modules (ES Modules)

> ES Modules are the official standard for organizing JavaScript into reusable, encapsulated files with explicit dependency declarations.

---

## Why Modules Exist

Before ES Modules, JavaScript had no built-in module system. Scripts shared a single global scope, leading to naming collisions, unpredictable load order, and tightly coupled code. Developers used patterns like IIFEs, the revealing module pattern, and third-party systems (CommonJS, AMD) to work around these limitations.

ES Modules (ESM), introduced in ES2015 and now fully supported in all modern environments, solve this at the language level.

```js
// Before modules: everything pollutes the global scope
var utils = {};
utils.add = function (a, b) { return a + b; };
// Any other script can overwrite utils

// With modules: explicit imports and exports
// math.js
export function add(a, b) { return a + b; }

// main.js
import { add } from './math.js';
add(2, 3); // 5
```

Key properties of ES Modules:
- Each file is its own module with its own scope
- Modules run in strict mode automatically
- Dependencies are declared statically at the top of the file
- Modules are evaluated only once (singleton)
- Bindings are live (more on this below)

## Export Syntax

### Named Exports

Named exports allow multiple values to be exported from a single module. Consumers must use the exact name (or rename with `as`).

```js
// utils.js

// Inline export
export const PI = 3.14159;

export function square(x) {
  return x * x;
}

export class Vector {
  constructor(x, y) {
    this.x = x;
    this.y = y;
  }
}
```

You can also declare first and export later using an export list:

```js
// utils.js
const PI = 3.14159;

function square(x) {
  return x * x;
}

class Vector {
  constructor(x, y) {
    this.x = x;
    this.y = y;
  }
}

// Export list at the bottom
export { PI, square, Vector };
```

Both styles are equivalent. The export list approach is useful when you want all exports visible in one place.

### Default Exports

A module can have exactly one default export. It represents the "main" thing the module provides.

```js
// logger.js
export default function log(message) {
  console.log(`[LOG] ${message}`);
}
```

The consumer can import it with any name:

```js
import log from './logger.js';
import myLogger from './logger.js'; // Same thing, different local name
```

Default exports are actually named exports with the special name `default`:

```js
// These are equivalent
export default function log(message) { /* ... */ }
export { log as default };
```

### Combining Named and Default Exports

A module can have both a default export and named exports:

```js
// http.js
export default class HttpClient {
  get(url) { /* ... */ }
  post(url, body) { /* ... */ }
}

export const HTTP_METHODS = ['GET', 'POST', 'PUT', 'DELETE'];
export function createHeaders(token) {
  return { Authorization: `Bearer ${token}` };
}
```

Importing both:

```js
import HttpClient, { HTTP_METHODS, createHeaders } from './http.js';
```

### Renaming Exports

```js
// internal naming
function internalValidate(data) { /* ... */ }

// Export under a different name
export { internalValidate as validate };
```

## Import Syntax

### Named Imports

```js
import { square, PI } from './utils.js';

console.log(square(5)); // 25
console.log(PI);        // 3.14159
```

### Import with Renaming

```js
import { square as sq, PI as pi } from './utils.js';

console.log(sq(5)); // 25
```

This is essential when two modules export the same name:

```js
import { parse as parseJSON } from './json-utils.js';
import { parse as parseXML } from './xml-utils.js';
```

### Default Import

```js
import Logger from './logger.js';
// The name "Logger" is your choice; it does not need to match the export
```

### Namespace Import

Import everything from a module as a single object:

```js
import * as utils from './utils.js';

console.log(utils.PI);        // 3.14159
console.log(utils.square(5)); // 25
console.log(utils.default);   // The default export, if one exists
```

The namespace object is sealed: you cannot add, delete, or reassign its properties.

### Side-Effect Import

Import a module purely for its side effects, without binding any names:

```js
import './polyfills.js';  // Executes the module but imports nothing
import './register-components.js';
```

This is common for polyfills, CSS imports in bundlers, or modules that register things globally.

### Combining Default and Named

```js
import HttpClient, { HTTP_METHODS } from './http.js';

// Or using namespace alongside default
import HttpClient, * as httpUtils from './http.js';
```

## Named vs Default Exports: The Debate

This is one of the most debated topics in JavaScript module design. Here are the trade-offs:

| Aspect | Named Exports | Default Exports |
|--------|--------------|-----------------|
| Rename required | No (but optional with `as`) | Always (consumer picks name) |
| Autocomplete/IDE | Excellent (exact names) | Poor (no name hint) |
| Refactoring | Rename propagates automatically | Consumer names never change |
| Tree shaking | Fully supported | Supported but less granular |
| Multiple per file | Unlimited | Exactly one |
| Re-export ergonomics | Easy | Naming conflicts possible |
| Discoverability | High (names are explicit) | Low (any name works) |

### The Case for Named Exports

```js
// Good: named exports are self-documenting
import { validateEmail, validatePhone } from './validators.js';

// Bad: default export gives no hint about content
import validators from './validators.js';
// Is it an object? A function? A class? What methods does it have?
```

Named exports enforce consistency. If 10 files import `validateEmail`, they all use the same name. With a default export, each file might use a different name (`validate`, `emailValidator`, `checkEmail`), making codebase searches harder.

### When Default Exports Make Sense

Default exports work well when a module has a single, obvious primary export:

```js
// React components (one component per file)
export default function UserProfile({ user }) { /* ... */ }

// Class-based modules
export default class EventEmitter { /* ... */ }
```

### Recommendation

Prefer named exports as the default choice. Use default exports sparingly, typically for single-class or single-component files where the module name already describes the export.

## Live Bindings

ES Module imports are **live bindings**, not copies. When the exporting module changes a value, the importing module sees the change.

```js
// counter.js
export let count = 0;

export function increment() {
  count++;
}
```

```js
// main.js
import { count, increment } from './counter.js';

console.log(count); // 0
increment();
console.log(count); // 1
increment();
console.log(count); // 2
```

This is fundamentally different from CommonJS, where `require()` returns a snapshot (a copy of the value at the time of import).

```js
// CommonJS: count is a COPY
const { count, increment } = require('./counter.js');

console.log(count); // 0
increment();
console.log(count); // Still 0! The local copy did not change.
```

### Imports Are Read-Only

You cannot reassign an imported binding:

```js
import { count } from './counter.js';

count = 10; // TypeError: Assignment to constant variable
// Only the exporting module can modify its own exports
```

This read-only constraint applies even if the export was declared with `let`. The importing module always gets a read-only view.

## Module Scope and Strict Mode

### Automatic Strict Mode

Every ES Module runs in strict mode automatically. You do not need `'use strict';`.

```js
// module.js (no 'use strict' needed)
x = 10; // ReferenceError: x is not defined (strict mode prevents implicit globals)

delete Object.prototype; // TypeError (strict mode prevents this)
```

### Module Scope

Each module has its own top-level scope. Variables declared at the top level of a module are not global:

```js
// moduleA.js
const secret = 'abc123';

// moduleB.js
console.log(secret); // ReferenceError: secret is not defined
// Even though both modules run in the same page
```

### Top-Level `this` is `undefined`

In regular scripts, `this` at the top level refers to the global object (`window` in browsers). In modules, it is `undefined`:

```js
// script.js (classic script)
console.log(this); // Window object

// module.js (ES module)
console.log(this); // undefined
```

## Dynamic `import()`

Static `import` declarations must appear at the top level and cannot be conditional. Dynamic `import()` solves this by returning a Promise that resolves to the module's namespace object.

### Basic Usage

```js
async function loadModule() {
  const module = await import('./heavy-feature.js');
  module.initialize();
}
```

### Conditional Loading

```js
async function loadLocale(lang) {
  // Load only the needed locale file
  const locale = await import(`./locales/${lang}.js`);
  return locale.translations;
}
```

### Lazy Loading on User Action

```js
document.getElementById('chart-btn').addEventListener('click', async () => {
  // Only load the charting library when the user actually needs it
  const { Chart } = await import('./chart-library.js');
  const chart = new Chart('#container');
  chart.render(data);
});
```

### Code Splitting with Bundlers

Bundlers like webpack, Rollup, and esbuild recognize `import()` as a split point. Each dynamic import becomes a separate chunk that loads on demand:

```js
// Webpack/Vite will create a separate chunk for this module
const { AdminPanel } = await import('./admin-panel.js');
```

You can provide bundler hints with magic comments (webpack-specific):

```js
const module = await import(
  /* webpackChunkName: "admin" */
  /* webpackPrefetch: true */
  './admin-panel.js'
);
```

### Importing Default Exports Dynamically

```js
const module = await import('./logger.js');
const Logger = module.default;

// Or with destructuring
const { default: Logger } = await import('./logger.js');
```

### Error Handling

```js
try {
  const module = await import('./optional-feature.js');
  module.init();
} catch (error) {
  console.warn('Optional feature not available:', error.message);
  // Fall back gracefully
}
```

## ES Modules vs CommonJS

Understanding the differences between ESM and CJS is critical for working in Node.js environments, where both systems coexist.

| Feature | ES Modules (ESM) | CommonJS (CJS) |
|---------|-------------------|-----------------|
| Syntax | `import` / `export` | `require()` / `module.exports` |
| Loading | Asynchronous | Synchronous |
| Binding | Live (read-only views) | Value copies |
| Top-level `this` | `undefined` | `module.exports` |
| File extension (Node) | `.mjs` or `"type": "module"` | `.cjs` or default `.js` |
| Static analysis | Yes (imports are hoisted) | No (require is a runtime function) |
| Tree shaking | Yes | Difficult |
| Conditional imports | Only via `import()` | `require()` anywhere |
| `__filename` / `__dirname` | Not available (use `import.meta`) | Available |
| Top-level await | Supported | Not supported |

### Interoperability in Node.js

ESM can import CJS modules:

```js
// ESM importing CJS
import cjsModule from './legacy.cjs';
// CJS module.exports becomes the default export
```

CJS cannot use static `import`, but can use dynamic `import()`:

```js
// CJS importing ESM (must be async)
async function loadESM() {
  const esmModule = await import('./modern.mjs');
  return esmModule;
}
```

### Dual Package Publishing

Library authors who need to support both ESM and CJS consumers use the `exports` field in `package.json`:

```json
{
  "name": "my-library",
  "type": "module",
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  }
}
```

### Node.js Module Resolution

Node.js determines whether a file is ESM or CJS based on:

1. File extension: `.mjs` is always ESM, `.cjs` is always CJS
2. The nearest `package.json` with `"type": "module"` makes `.js` files ESM
3. Without `"type": "module"`, `.js` files are CJS (the default)

```json
// package.json
{
  "type": "module"  // All .js files in this package are ESM
}
```

## `import.meta`

`import.meta` is an object that provides metadata about the current module. It is only available inside ES Modules.

### `import.meta.url`

Returns the full URL of the current module file:

```js
console.log(import.meta.url);
// Browser: "https://example.com/js/utils.js"
// Node.js: "file:///home/user/project/utils.js"
```

Common use: resolving paths relative to the current module:

```js
// Load a file relative to the current module
const dataUrl = new URL('./data.json', import.meta.url);
const response = await fetch(dataUrl);
```

### `import.meta.dirname` and `import.meta.filename` (Node.js 21+)

Node.js added these as ESM equivalents to `__dirname` and `__filename`:

```js
// Node.js 21+
console.log(import.meta.dirname);  // "/home/user/project/src"
console.log(import.meta.filename); // "/home/user/project/src/utils.js"
```

For older Node.js versions, you can derive them from `import.meta.url`:

```js
import { fileURLToPath } from 'node:url';
import { dirname } from 'node:path';

const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);
```

### `import.meta.resolve()` (Node.js 20.6+)

Resolves a module specifier to a URL without actually loading the module:

```js
const resolvedPath = import.meta.resolve('./utils.js');
// "file:///home/user/project/src/utils.js"

// Also works for packages
const lodashPath = import.meta.resolve('lodash');
```

## Module Loading: The Three Phases

The browser (and Node.js) loads ES Modules in three distinct phases:

### 1. Construction (Parsing)

The engine parses the module source, identifies all `import` and `export` statements, and builds a **module record**. It does NOT execute any code yet. It fetches all imported modules recursively to build the full dependency graph.

Because `import` is static, the engine can determine all dependencies at parse time without executing the module.

### 2. Instantiation (Linking)

The engine allocates memory for all exports and wires up the live bindings between importing and exporting modules. At this point, all modules know where their imports will come from, but the values have not been computed yet.

### 3. Evaluation (Execution)

The engine executes each module's top-level code in post-order (dependencies first, then dependents). Each module is evaluated exactly once. Subsequent imports of the same module reuse the already-evaluated instance.

```js
// This module runs only once, no matter how many files import it
console.log('Module evaluated!');
export const timestamp = Date.now();
```

```js
// fileA.js
import { timestamp } from './singleton.js'; // "Module evaluated!" printed

// fileB.js
import { timestamp } from './singleton.js'; // Nothing printed, same instance
```

This single-evaluation guarantee makes modules natural singletons.

## Circular Dependencies

Circular dependencies occur when module A imports module B, and module B imports module A (directly or through a chain).

### How ESM Handles Circular Dependencies

Because ESM uses live bindings and the three-phase loading process, circular dependencies can work, but with important caveats.

```js
// a.js
import { b } from './b.js';
console.log('a.js:', b); // "b from B"
export const a = 'a from A';

// b.js
import { a } from './a.js';
console.log('b.js:', a); // undefined! (a.js hasn't finished evaluating yet)
export const b = 'b from B';
```

What happens:
1. Engine starts evaluating `a.js`
2. `a.js` imports `b.js`, so engine pauses `a.js` and evaluates `b.js` first
3. `b.js` imports `a.js`, but `a.js` is already being evaluated (in progress), so the engine provides the partially initialized module
4. At this point, `a` has been allocated but not yet assigned (because `a.js` line with `export const a = ...` hasn't executed yet)
5. `b.js` sees `a` as `undefined`
6. `b.js` finishes, control returns to `a.js`
7. `a.js` can now see `b` correctly

### When Circular Dependencies Break

The problem occurs when you try to use an imported value during module evaluation, before the exporting module has assigned it:

```js
// BAD: Accessing a circular import at the top level
import { helper } from './other.js';
const result = helper(); // TypeError: helper is not a function
// helper hasn't been assigned yet during circular initialization
export function myFunc() { /* ... */ }
```

### Making Circular Dependencies Work

Use function declarations (they are hoisted) and defer access to after module evaluation:

```js
// WORKS: Function declarations are hoisted
// a.js
import { getB } from './b.js';
export function getA() { return 'A'; }

// Access circular import inside a function, not at top level
export function init() {
  console.log(getB()); // Called later, after both modules are fully loaded
}

// b.js
import { getA } from './a.js';
export function getB() { return 'B'; }

export function init() {
  console.log(getA()); // Safe: called after both modules are loaded
}
```

### Best Practice: Avoid Circular Dependencies

Circular dependencies are a code smell. They indicate tightly coupled modules that should be refactored:

```js
// BAD: A and B depend on each other
// a.js imports from b.js
// b.js imports from a.js

// GOOD: Extract shared logic into a third module
// shared.js (no circular dependency)
export function sharedHelper() { /* ... */ }

// a.js imports from shared.js
// b.js imports from shared.js
```

Tools like `eslint-plugin-import` (with the `no-cycle` rule) and `madge` can detect circular dependencies in your codebase.

## Re-exports and Barrel Files

### Re-export Syntax

Re-exports let a module forward exports from other modules without using them locally:

```js
// Named re-export
export { add, subtract } from './math.js';

// Re-export with renaming
export { add as sum } from './math.js';

// Re-export everything (all named exports)
export * from './math.js';

// Re-export a default export as a named export
export { default as MathUtils } from './math.js';

// Re-export a named export as the default
export { add as default } from './math.js';
```

### Barrel Files (Index Files)

A barrel file re-exports from multiple modules to provide a single entry point:

```js
// components/index.js (barrel file)
export { Button } from './Button.js';
export { Input } from './Input.js';
export { Modal } from './Modal.js';
export { Dropdown } from './Dropdown.js';
```

Consumers get a clean import:

```js
// Instead of four separate imports
import { Button } from './components/Button.js';
import { Input } from './components/Input.js';

// One import from the barrel
import { Button, Input } from './components/index.js';
```

### The Barrel File Problem

Barrel files can seriously hurt performance. When you import anything from a barrel file, the module system must evaluate ALL the modules it re-exports (or at minimum parse them to build the dependency graph).

```js
// You only need Button, but the barrel imports everything
import { Button } from './components/index.js';
// This also triggers loading Input.js, Modal.js, Dropdown.js,
// and all THEIR dependencies
```

This problem is called **barrel file bloat** and it affects:
- **Development server startup** (more files to parse)
- **Hot module replacement** (more modules to invalidate)
- **Bundle size** (if tree shaking cannot fully eliminate unused modules)

### Mitigating Barrel File Problems

1. **Keep barrel files small.** Only re-export the public API, not internal utilities.

2. **Use direct imports for performance-critical paths:**

```js
// Direct import bypasses the barrel
import { Button } from './components/Button.js';
```

3. **Configure your bundler.** Modern bundlers like Vite have optimizations for barrel files, but they are not perfect.

4. **Avoid `export * from`** because it re-exports everything, including things you may not intend to expose. Prefer explicit named re-exports.

```js
// BAD: exports everything, even internal helpers
export * from './math.js';

// GOOD: explicitly re-export only the public API
export { add, subtract, multiply } from './math.js';
```

## Tree Shaking

Tree shaking is the process of eliminating dead (unused) code from the final bundle. It relies on the static nature of ES Module `import`/`export` statements.

### How It Works

Because ESM imports are static and analyzable at build time, bundlers can determine which exports are actually used and remove the rest:

```js
// utils.js
export function used() { return 'I am used'; }
export function unused() { return 'I am never imported'; }

// main.js
import { used } from './utils.js';
used();
// After tree shaking, unused() is removed from the bundle
```

### The `sideEffects` Field

The `sideEffects` field in `package.json` tells bundlers whether a package's modules have side effects (code that runs at import time and affects global state):

```json
{
  "name": "my-library",
  "sideEffects": false
}
```

`"sideEffects": false` means the bundler can safely remove any module that is not imported. If some files DO have side effects (like CSS imports or polyfills), list them:

```json
{
  "sideEffects": [
    "*.css",
    "./src/polyfills.js"
  ]
}
```

### What Breaks Tree Shaking

1. **CommonJS modules.** `require()` is dynamic, so bundlers cannot statically analyze it.

2. **Side effects at the module level:**

```js
// This prevents tree shaking because the module has a side effect
export const config = {};
window.__myLib = config; // Side effect: modifies global state
```

3. **Re-exporting with `export *`** makes it harder for bundlers to determine what is used.

4. **Property access on namespace imports:**

```js
import * as utils from './utils.js';
// Bundler may not be able to tree-shake individual properties
utils[dynamicKey](); // Dynamic access prevents static analysis
```

5. **Classes with side effects in static initializers:**

```js
export class MyClass {
  static instance = new MyClass(); // Side effect: constructor runs at import
}
```

### Verifying Tree Shaking

Use your bundler's analysis tools to verify:

```bash
# Webpack
npx webpack --json=stats.json
npx webpack-bundle-analyzer stats.json

# Vite/Rollup
npx vite build --mode production
# Check the output size

# Source map explorer
npx source-map-explorer dist/bundle.js
```

## Import Attributes

Import attributes (formerly "import assertions") allow you to pass metadata to the module loader. This feature reached TC39 Stage 4 and is available in modern runtimes.

### Syntax

The syntax uses the `with` keyword (replacing the older `assert` keyword):

```js
// Import JSON
import data from './config.json' with { type: 'json' };
console.log(data.version); // Reads the JSON as a JavaScript object

// Import CSS (where supported)
import styles from './styles.css' with { type: 'css' };
document.adoptedStyleSheets = [styles];
```

### Dynamic Import with Attributes

```js
const data = await import('./config.json', { with: { type: 'json' } });
console.log(data.default); // The parsed JSON object
```

### Why Import Attributes Exist

Security. Without the `type` attribute, a server could respond to a `.json` request with JavaScript, potentially executing malicious code. The `type: 'json'` attribute tells the engine to ONLY accept JSON. If the response is anything else, it throws an error.

### Current Support

- **JSON modules**: Supported in Node.js 22+, Chrome 123+, Edge 123+
- **CSS modules**: Chrome 123+ (with `type: 'css'`)
- The `assert` keyword (older syntax) is deprecated in favor of `with`

```js
// OLD (deprecated)
import data from './config.json' assert { type: 'json' };

// NEW (current standard)
import data from './config.json' with { type: 'json' };
```

## Top-Level Await

ES2022 introduced top-level `await`, allowing modules to use `await` outside of `async` functions.

### Basic Usage

```js
// config.js
const response = await fetch('/api/config');
export const config = await response.json();
```

```js
// main.js
import { config } from './config.js';
// config is fully loaded and available here
console.log(config.apiUrl);
```

### How It Affects Module Loading

A module with top-level `await` pauses its own evaluation until the awaited promise resolves. Crucially, it also blocks any module that imports it:

```js
// slow-module.js
await new Promise(resolve => setTimeout(resolve, 5000)); // 5 second delay
export const data = 'ready';

// main.js
import { data } from './slow-module.js';
// main.js will not start evaluating until slow-module.js finishes (5 seconds)
```

However, sibling modules that do NOT depend on the awaiting module can still execute in parallel:

```js
// These two modules load in parallel if neither depends on the other
import './moduleA.js'; // Has top-level await (3 seconds)
import './moduleB.js'; // No top-level await (instant)
// moduleB.js can finish before moduleA.js
```

### Use Cases

```js
// 1. Loading configuration before the app starts
export const config = await (await fetch('/config.json')).json();

// 2. Feature detection
let storage;
try {
  const { indexedDB } = await import('./indexeddb-storage.js');
  storage = indexedDB;
} catch {
  const { localStorage } = await import('./localstorage-fallback.js');
  storage = localStorage;
}
export { storage };

// 3. Initializing a database connection
import { createConnection } from './db.js';
export const db = await createConnection('mongodb://localhost/mydb');
```

### Deadlock Risk

Top-level await combined with circular dependencies can cause deadlocks:

```js
// a.js
import { b } from './b.js';
export const a = await Promise.resolve('A');

// b.js
import { a } from './a.js';
export const b = await Promise.resolve('B');

// DEADLOCK: a.js waits for b.js, b.js waits for a.js
```

### Best Practices for Top-Level Await

- Use it sparingly. It blocks dependent modules.
- Never use it for operations that could be deferred.
- Be mindful of error handling: an unhandled rejection in a top-level await crashes the module graph.

```js
// BAD: No error handling
export const data = await fetch('/api/data').then(r => r.json());

// GOOD: Handle errors gracefully
let data;
try {
  const response = await fetch('/api/data');
  data = await response.json();
} catch (error) {
  console.error('Failed to load data:', error);
  data = { fallback: true };
}
export { data };
```

## Modules in the Browser

### `<script type="module">`

To use ES Modules in HTML, add `type="module"` to the script tag:

```html
<script type="module" src="./main.js"></script>
```

Key behaviors of `<script type="module">`:
- **Deferred by default.** Module scripts do not block HTML parsing. They execute after the document is parsed, equivalent to the `defer` attribute on classic scripts.
- **Executed once.** Even if you include the same module script twice, it only executes once.
- **CORS required.** Module scripts fetched from a different origin require proper CORS headers.
- **Strict mode.** Always runs in strict mode.
- **No `nomodule`.** The `nomodule` attribute is for the opposite purpose (see below).

### Inline Module Scripts

```html
<script type="module">
  import { greet } from './utils.js';
  greet('World');
</script>
```

### The `nomodule` Fallback

The `nomodule` attribute provides a fallback for browsers that do not support ES Modules:

```html
<!-- Modern browsers execute this -->
<script type="module" src="./app.modern.js"></script>

<!-- Legacy browsers execute this (modern browsers ignore it) -->
<script nomodule src="./app.legacy.js"></script>
```

In 2026, all major browsers support ES Modules, so `nomodule` is rarely needed for new projects.

### `async` on Module Scripts

Module scripts are deferred by default. Adding `async` makes them execute as soon as they are ready, without waiting for HTML parsing to complete:

```html
<!-- Deferred (default): executes after document is parsed -->
<script type="module" src="./analytics.js"></script>

<!-- Async: executes as soon as the module and its dependencies are loaded -->
<script type="module" async src="./analytics.js"></script>
```

### Import Maps

Import maps let you control module resolution in the browser without a bundler:

```html
<script type="importmap">
{
  "imports": {
    "lodash": "https://cdn.jsdelivr.net/npm/lodash-es@4.17.21/lodash.js",
    "utils/": "./src/utils/"
  }
}
</script>

<script type="module">
  // Now you can use bare specifiers
  import _ from 'lodash';
  import { format } from 'utils/format.js';
</script>
```

Import maps are supported in all modern browsers (Chrome 89+, Firefox 108+, Safari 16.4+).

## Module Design Patterns

### One Thing Per Module

Each module should have a single, clear responsibility:

```js
// BAD: kitchen sink module
// utils.js
export function formatDate() { /* ... */ }
export function validateEmail() { /* ... */ }
export function calculateTax() { /* ... */ }
export function parseCSV() { /* ... */ }

// GOOD: focused modules
// date-utils.js
export function formatDate() { /* ... */ }
export function parseDate() { /* ... */ }

// validators.js
export function validateEmail() { /* ... */ }
export function validatePhone() { /* ... */ }
```

### Facade Pattern

A facade module provides a simplified interface over complex subsystems:

```js
// api/facade.js
import { HttpClient } from './http-client.js';
import { AuthService } from './auth-service.js';
import { CacheService } from './cache-service.js';

const http = new HttpClient();
const auth = new AuthService();
const cache = new CacheService();

export async function fetchUser(id) {
  const cached = cache.get(`user:${id}`);
  if (cached) return cached;

  const token = await auth.getToken();
  const user = await http.get(`/users/${id}`, {
    headers: { Authorization: `Bearer ${token}` }
  });

  cache.set(`user:${id}`, user);
  return user;
}
```

### Configuration Module

```js
// config.js
const env = process.env.NODE_ENV || 'development';

const configs = {
  development: {
    apiUrl: 'http://localhost:3000',
    debug: true,
  },
  production: {
    apiUrl: 'https://api.example.com',
    debug: false,
  },
};

export const config = Object.freeze(configs[env]);
```

### Singleton via Module Scope

Because modules evaluate once, any state at the module level is a natural singleton:

```js
// store.js
const state = new Map();

export function get(key) {
  return state.get(key);
}

export function set(key, value) {
  state.set(key, value);
}

// Every file that imports from store.js shares the same Map
```

## Common Mistakes

### 1. Forgetting File Extensions in Browser/Node ESM

Bundlers resolve bare paths, but native ESM in browsers and Node.js requires full file extensions:

```js
// BAD (fails in native ESM)
import { add } from './math';

// GOOD
import { add } from './math.js';
```

### 2. Using `require()` in ESM Files

```js
// BAD: require is not available in ES Modules
const fs = require('fs');

// GOOD
import fs from 'node:fs';
// Or for dynamic needs:
import { createRequire } from 'node:module';
const require = createRequire(import.meta.url);
```

### 3. Expecting `import` to Work Like `require`

```js
// BAD: import is not a function you can call conditionally
if (condition) {
  import { feature } from './feature.js'; // SyntaxError
}

// GOOD: use dynamic import for conditional loading
if (condition) {
  const { feature } = await import('./feature.js');
}
```

### 4. Circular Dependencies at Evaluation Time

```js
// BAD: accessing import at top level in a circular dependency
import { helper } from './other.js';
const result = helper(); // May be undefined due to circular loading
export const value = result;

// GOOD: defer access to function call time
import { helper } from './other.js';
export function getValue() {
  return helper(); // Called after all modules are fully loaded
}
```

### 5. Overusing Barrel Files

```js
// BAD: importing one thing from a barrel that re-exports 50 modules
import { Button } from './components/index.js';

// GOOD: import directly when performance matters
import { Button } from './components/Button.js';
```

### 6. Exporting Mutable Objects Without Intention

```js
// BAD: consumers can mutate your module's state
export const config = { debug: true, apiUrl: 'https://api.com' };
// Another module: config.debug = false; (mutates the shared object)

// GOOD: freeze the object
export const config = Object.freeze({ debug: true, apiUrl: 'https://api.com' });
```

### 7. Not Handling Dynamic Import Errors

```js
// BAD: unhandled rejection
const module = await import('./missing-module.js');

// GOOD
try {
  const module = await import('./missing-module.js');
} catch (error) {
  console.error('Module failed to load:', error);
}
```

---

## Best Practices

- **Use named exports by default.** They provide better IDE support, autocomplete, and consistent naming across the codebase.

- **Include file extensions** in import paths when working outside a bundler (native ESM). Even with a bundler, including them improves clarity and compatibility.

- **Keep modules focused.** One responsibility per module. If a module exports 20+ things, it probably needs to be split.

- **Avoid circular dependencies.** Use tools like `eslint-plugin-import` with the `no-cycle` rule to catch them. Refactor shared logic into a separate module.

- **Be explicit with barrel file re-exports.** Use `export { name } from` instead of `export * from` to control the public API and help tree shaking.

- **Mark your package with `"sideEffects": false`** in `package.json` to enable optimal tree shaking. List files with real side effects explicitly.

- **Handle dynamic import failures.** Always wrap `import()` in try/catch or `.catch()`.

- **Use `Object.freeze()` for exported configuration objects** to prevent accidental mutation from consuming modules.

- **Prefer `import.meta.url` over `__dirname`** in ESM contexts. For Node.js 21+, use `import.meta.dirname` and `import.meta.filename`.

## Real-World Use Cases

### 1. Feature Flags with Dynamic Imports

```js
// feature-loader.js
const featureFlags = await (await fetch('/api/features')).json();

export async function loadFeature(name) {
  if (!featureFlags[name]) {
    return null;
  }

  try {
    const module = await import(`./features/${name}.js`);
    return module;
  } catch (error) {
    console.error(`Failed to load feature "${name}":`, error);
    return null;
  }
}

// Usage
const analytics = await loadFeature('analytics');
if (analytics) {
  analytics.init({ trackingId: 'UA-12345' });
}
```

### 2. Plugin System

```js
// plugin-manager.js
const plugins = new Map();

export async function registerPlugin(name, modulePath) {
  const module = await import(modulePath);

  if (typeof module.default?.init !== 'function') {
    throw new Error(`Plugin "${name}" must export a default object with an init() method`);
  }

  plugins.set(name, module.default);
  await module.default.init();
  console.log(`Plugin "${name}" registered`);
}

export function getPlugin(name) {
  return plugins.get(name);
}

// Usage
await registerPlugin('logger', './plugins/logger.js');
await registerPlugin('cache', './plugins/cache.js');
```

### 3. Internationalization with Lazy-Loaded Locales

```js
// i18n.js
let translations = {};
let currentLocale = 'en';

export async function setLocale(locale) {
  try {
    const module = await import(`./locales/${locale}.js`);
    translations = module.default;
    currentLocale = locale;
  } catch {
    console.warn(`Locale "${locale}" not found, falling back to English`);
    const fallback = await import('./locales/en.js');
    translations = fallback.default;
    currentLocale = 'en';
  }
}

export function t(key) {
  return translations[key] || key;
}

// Initialize with default locale
await setLocale('en');
```

```js
// locales/en.js
export default {
  'greeting': 'Hello',
  'farewell': 'Goodbye',
  'submit': 'Submit',
};

// locales/es.js
export default {
  'greeting': 'Hola',
  'farewell': 'Adios',
  'submit': 'Enviar',
};
```
