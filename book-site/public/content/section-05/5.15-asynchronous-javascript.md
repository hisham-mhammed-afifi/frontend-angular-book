# 5.15 Asynchronous JavaScript

> Asynchronous programming is how JavaScript handles operations that take time (network requests, file I/O, timers) without blocking the single thread that runs your code.

---

## Callbacks

A callback is a function passed as an argument to another function, to be called when an asynchronous operation completes. This was JavaScript's original async mechanism.

### The Callback Pattern

```js
function loadData(url, callback) {
  const xhr = new XMLHttpRequest();
  xhr.open('GET', url);
  xhr.onload = () => callback(null, JSON.parse(xhr.responseText));
  xhr.onerror = () => callback(new Error('Request failed'));
  xhr.send();
}

loadData('/api/users', (err, data) => {
  if (err) {
    console.error(err);
    return;
  }
  console.log(data);
});
```

### Error-First Convention

Node.js established the error-first callback convention: the first argument is always an error (or `null` if no error occurred), and subsequent arguments carry the result.

```js
fs.readFile('/path/to/file', 'utf8', (err, data) => {
  if (err) {
    console.error('Failed to read:', err.message);
    return; // MUST return to prevent executing the success path
  }
  console.log(data);
});
```

Forgetting to `return` after handling the error means the success code runs too. The callback pattern provides no enforcement; discipline is entirely on the developer.

### Callback Hell

Nested callbacks create a "pyramid of doom" that is difficult to read, debug, and maintain:

```js
// BAD: callback hell
getUser(userId, (err, user) => {
  if (err) return handleError(err);
  getOrders(user.id, (err, orders) => {
    if (err) return handleError(err);
    getOrderDetails(orders[0].id, (err, details) => {
      if (err) return handleError(err);
      getShippingInfo(details.trackingId, (err, shipping) => {
        if (err) return handleError(err);
        displayShipping(shipping);
      });
    });
  });
});
```

Callback hell is not just an indentation problem. The real issues are:
- Error handling must be repeated at every level
- Control flow (loops, conditionals) becomes extremely complex
- Shared state between callbacks leads to race conditions
- Debugging stack traces are unhelpful

Promises and async/await solve all of these problems.

## Promises

A Promise represents a value that may not be available yet but will be resolved at some point in the future (or rejected with an error).

### States

A Promise has exactly three states:

| State | Description | Transitions to |
|-------|-------------|----------------|
| **pending** | Initial state. Operation in progress. | fulfilled or rejected |
| **fulfilled** | Operation completed successfully. Has a value. | (terminal) |
| **rejected** | Operation failed. Has a reason (error). | (terminal) |

Once a Promise settles (becomes fulfilled or rejected), it can never change state again.

### Creating Promises

```js
const promise = new Promise((resolve, reject) => {
  // The executor function runs synchronously
  const success = performOperation();

  if (success) {
    resolve(value);              // Transitions to fulfilled
  } else {
    reject(new Error('failed')); // Transitions to rejected
  }
});
```

The executor runs synchronously inside the Promise constructor. The `resolve` and `reject` calls schedule microtasks for `.then()` and `.catch()` handlers.

```js
console.log('1 - before Promise');

const p = new Promise((resolve) => {
  console.log('2 - inside executor (synchronous)');
  resolve('done');
});

p.then((value) => {
  console.log('4 - then handler (microtask):', value);
});

console.log('3 - after Promise');

// Output:
// 1 - before Promise
// 2 - inside executor (synchronous)
// 3 - after Promise
// 4 - then handler (microtask): done
```

### Gotcha: Code After resolve/reject Still Runs

Calling `resolve()` or `reject()` does not stop execution of the executor:

```js
new Promise((resolve, reject) => {
  resolve('done');
  console.log('This still runs'); // Executes!
  throw new Error('This is silently ignored'); // Swallowed!
});
```

Use `return resolve(value)` to prevent this:

```js
new Promise((resolve, reject) => {
  if (valid) {
    return resolve('done'); // Exits the executor
  }
  reject(new Error('invalid'));
});
```

### Convenience Methods

```js
// Create an already-fulfilled Promise
const fulfilled = Promise.resolve(42);

// Create an already-rejected Promise
const rejected = Promise.reject(new Error('failed'));

// Resolve with a thenable (waits for it)
const fromThenable = Promise.resolve(fetch('/api/data'));
```

## Promise Chaining

Each `.then()` returns a new Promise, enabling flat chains instead of nested callbacks:

```js
fetch('/api/user')
  .then(response => response.json())
  .then(user => fetch(`/api/orders/${user.id}`))
  .then(response => response.json())
  .then(orders => console.log(orders))
  .catch(err => console.error(err));
```

### What .then() Returns

The behavior of the chain depends on what the `.then()` handler returns:

```js
// Return a value: next .then() receives it
Promise.resolve(1)
  .then(x => x + 1)   // Returns 2
  .then(x => x * 3)   // Returns 6
  .then(console.log);  // 6

// Return a Promise: chain waits for it to settle
Promise.resolve(1)
  .then(x => fetch(`/api/${x}`))  // Returns a Promise
  .then(response => response.json()) // Waits for fetch to complete first
  .then(console.log);

// Return nothing: next .then() receives undefined
Promise.resolve(1)
  .then(x => { console.log(x); }) // No explicit return
  .then(x => console.log(x));     // undefined

// Throw an error: jumps to the nearest .catch()
Promise.resolve(1)
  .then(x => { throw new Error('oops'); })
  .then(x => console.log('skipped'))
  .catch(err => console.error(err.message)); // "oops"
```

### .catch() Recovers the Chain

After `.catch()` handles an error, the chain continues as fulfilled:

```js
Promise.reject(new Error('initial failure'))
  .catch(err => {
    console.log('Caught:', err.message); // "Caught: initial failure"
    return 'recovered value';
  })
  .then(value => {
    console.log(value); // "recovered value" -- chain continues normally
  });
```

To keep the chain in a rejected state, re-throw or return a rejected Promise:

```js
Promise.reject(new Error('failure'))
  .catch(err => {
    console.error('Logging error:', err.message);
    throw err; // Re-throw to propagate
  })
  .then(value => {
    console.log('This never runs');
  })
  .catch(err => {
    console.log('Second catch:', err.message);
  });
```

### .finally()

Runs regardless of whether the Promise fulfilled or rejected. Does not receive any arguments and does not change the settled value (unless it throws):

```js
showLoadingSpinner();

fetch('/api/data')
  .then(response => response.json())
  .then(data => render(data))
  .catch(err => showError(err))
  .finally(() => {
    hideLoadingSpinner(); // Runs in both success and error cases
  });
```

### Where to Place .catch()

```js
// BAD: catch before then -- errors thrown in doWork() go uncaught
promise
  .catch(handleError)
  .then(doWork);

// GOOD: catch at the end catches errors from any point in the chain
promise
  .then(doWork)
  .catch(handleError);
```

## Promise Static Methods

### Promise.all

Fulfills when ALL promises fulfill. Rejects on the FIRST rejection (short-circuits).

```js
const [users, posts, settings] = await Promise.all([
  fetchUsers(),
  fetchPosts(),
  fetchSettings(),
]);
// All three run in parallel. Total time = slowest request.
```

When one rejects, the others continue executing (they are NOT cancelled), but their results are discarded:

```js
try {
  await Promise.all([
    fetch('/api/users'),    // Succeeds
    fetch('/api/missing'),  // Rejects (404)
    fetch('/api/settings'), // Succeeds, but result is discarded
  ]);
} catch (err) {
  // Only catches the first rejection
}
```

### Promise.allSettled

Always waits for ALL promises to settle. Never rejects. Returns an array of result objects:

```js
const results = await Promise.allSettled([
  fetch('/api/users'),
  fetch('/api/posts'),
  fetch('/api/comments'),
]);

for (const result of results) {
  if (result.status === 'fulfilled') {
    console.log('Success:', result.value);
  } else {
    console.log('Failed:', result.reason);
  }
}
```

Use `allSettled` when tasks are independent and you want partial success (loading dashboard widgets, fetching data from multiple services).

### Promise.race

Settles with the first promise to settle, whether fulfilled or rejected:

```js
// Classic timeout pattern
const result = await Promise.race([
  fetch('/api/data'),
  new Promise((_, reject) =>
    setTimeout(() => reject(new Error('Timeout')), 5000)
  ),
]);
```

The losing promises keep running. There is no cancellation mechanism built into `Promise.race`.

### Promise.any

Fulfills with the first promise to fulfill. Only rejects if ALL promises reject (with an `AggregateError`):

```js
try {
  const fastest = await Promise.any([
    fetch('https://cdn1.example.com/data'),
    fetch('https://cdn2.example.com/data'),
    fetch('https://cdn3.example.com/data'),
  ]);
  console.log('Fastest CDN responded:', fastest);
} catch (err) {
  console.log(err instanceof AggregateError); // true
  console.log(err.errors); // Array of all rejection reasons
}
```

### Comparison Table

| Method | Fulfills when | Rejects when | Short-circuits? |
|--------|---------------|--------------|-----------------|
| `Promise.all` | All fulfill | Any rejects | Yes (first rejection) |
| `Promise.allSettled` | All settle | Never | No |
| `Promise.race` | First settles (fulfilled) | First settles (rejected) | Yes (first settlement) |
| `Promise.any` | First fulfills | All reject | Yes (first fulfillment) |

## Promise.withResolvers() (ES2024)

This method extracts the `resolve` and `reject` functions from a Promise constructor, eliminating the old deferred pattern.

### Before (the workaround)

```js
let resolve, reject;
const promise = new Promise((res, rej) => {
  resolve = res;
  reject = rej;
});
```

### After (ES2024)

```js
const { promise, resolve, reject } = Promise.withResolvers();
```

### Use Cases

**Event-based resolution:**

```js
function waitForEvent(target, eventName) {
  const { promise, resolve } = Promise.withResolvers();
  target.addEventListener(eventName, resolve, { once: true });
  return promise;
}

const clickEvent = await waitForEvent(button, 'click');
```

**Request/response matching (Web Workers, message channels):**

```js
const pending = new Map();
let nextId = 0;

function callWorker(worker, data) {
  const id = nextId++;
  const { promise, resolve, reject } = Promise.withResolvers();
  pending.set(id, { resolve, reject });
  worker.postMessage({ id, ...data });
  return promise;
}

worker.addEventListener('message', (e) => {
  const { id, result, error } = e.data;
  const { resolve, reject } = pending.get(id);
  pending.delete(id);
  error ? reject(error) : resolve(result);
});
```

**Deduplicating concurrent requests (cache):**

```js
const cache = new Map();

function fetchOnce(url) {
  if (cache.has(url)) return cache.get(url);

  const { promise, resolve, reject } = Promise.withResolvers();
  cache.set(url, promise);

  fetch(url)
    .then(r => r.json())
    .then(resolve)
    .catch(reject);

  return promise;
}

// Multiple callers for the same URL share one in-flight request
const [a, b] = await Promise.all([fetchOnce('/api/data'), fetchOnce('/api/data')]);
// Only one fetch was made
```

## async/await

`async`/`await` is syntactic sugar over Promises that makes asynchronous code read like synchronous code.

### Basics

An `async` function always returns a Promise. The `await` keyword pauses execution of the function until the awaited Promise settles.

```js
async function fetchUser(id) {
  const response = await fetch(`/api/users/${id}`);
  if (!response.ok) {
    throw new Error(`HTTP ${response.status}`);
  }
  return response.json(); // Returns a Promise, auto-awaited by caller
}
```

### Error Handling with try/catch

```js
async function loadData() {
  try {
    const user = await fetchUser(1);
    const orders = await fetchOrders(user.id);
    return orders;
  } catch (err) {
    console.error('Failed:', err.message);
    throw err; // Re-throw to let caller handle it
  } finally {
    hideLoadingSpinner(); // Runs regardless of success or failure
  }
}
```

### return vs return await

Inside a `try/catch`, this distinction is critical:

```js
// BAD: rejection from fetchData() bypasses the local catch
async function bad() {
  try {
    return fetchData(); // Returns the Promise directly
  } catch (err) {
    console.log('Never catches fetchData rejections');
  }
}

// GOOD: await ensures rejection is caught locally
async function good() {
  try {
    return await fetchData(); // Await unwraps the rejection
  } catch (err) {
    console.log('Catches fetchData rejections');
    return fallbackData;
  }
}
```

Outside of `try/catch`, `return await` and `return` are functionally equivalent, though `return await` adds one extra microtask. The ESLint rule `no-return-await` was deprecated because `return await` is genuinely needed inside `try/catch/finally`.

### Parallel Execution

The most important performance pattern with async/await:

```js
// BAD: Sequential -- total time is 2s + 2s + 2s = 6s
async function sequential() {
  const a = await fetchA(); // 2s
  const b = await fetchB(); // 2s
  const c = await fetchC(); // 2s
  return [a, b, c];
}

// GOOD: Parallel -- total time is max(2s, 2s, 2s) = 2s
async function parallel() {
  const [a, b, c] = await Promise.all([
    fetchA(),
    fetchB(),
    fetchC(),
  ]);
  return [a, b, c];
}
```

An alternative "start-then-await" pattern exists but has a gotcha:

```js
// Start all promises first
const aPromise = fetchA();
const bPromise = fetchB();
const cPromise = fetchC();

// Then await them
const a = await aPromise;
const b = await bPromise; // If bPromise rejected, the rejection was unhandled
const c = await cPromise; // until this line runs

// Promise.all is safer because it handles all rejections immediately
```

### Async Arrow Functions and Methods

```js
// Arrow function
const fetchUser = async (id) => {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
};

// Object method
const api = {
  async getUser(id) {
    const res = await fetch(`/api/users/${id}`);
    return res.json();
  }
};

// Class method
class UserService {
  async getUser(id) {
    const res = await fetch(`/api/users/${id}`);
    return res.json();
  }
}
```

### Awaiting Non-Promise Values

`await` on a non-Promise value wraps it in `Promise.resolve()`:

```js
const x = await 42; // Same as: await Promise.resolve(42)
console.log(x);     // 42
```

This is rarely useful and adds unnecessary overhead. Do not mark a function `async` if it never uses `await`.

## Microtasks vs Macrotasks

Understanding the microtask and macrotask queues is essential for predicting execution order.

### Execution Order

1. Execute the current synchronous script to completion
2. Drain the entire microtask queue (every microtask, including ones added during this step)
3. Execute ONE macrotask
4. Drain the microtask queue again
5. Render (in browsers)
6. Repeat from step 3

### Classification

| Microtasks | Macrotasks |
|------------|------------|
| `Promise.then/catch/finally` | `setTimeout` / `setInterval` |
| `queueMicrotask()` | I/O callbacks (Node.js) |
| `MutationObserver` | `setImmediate` (Node.js) |
| `process.nextTick()` (Node.js) | `requestAnimationFrame` (before paint) |

### The Classic Ordering Example

```js
console.log('1 - sync');

setTimeout(() => console.log('2 - macrotask'), 0);

Promise.resolve().then(() => console.log('3 - microtask'));

queueMicrotask(() => console.log('4 - microtask'));

console.log('5 - sync');

// Output:
// 1 - sync
// 5 - sync
// 3 - microtask
// 4 - microtask
// 2 - macrotask
```

### Microtasks Nested Inside Microtasks

Microtasks scheduled during microtask processing run before ANY macrotask:

```js
Promise.resolve().then(() => {
  console.log('microtask 1');
  Promise.resolve().then(() => {
    console.log('microtask 2 (nested)');
  });
});

setTimeout(() => console.log('macrotask'), 0);

// Output:
// microtask 1
// microtask 2 (nested)
// macrotask
```

### Microtask Starvation

If microtasks keep scheduling more microtasks, macrotasks and rendering never get a turn:

```js
// BAD: infinite microtask loop blocks everything
function recurse() {
  queueMicrotask(recurse);
}
recurse();
// setTimeout, setInterval, rendering, user events -- all starved
```

### queueMicrotask()

Use `queueMicrotask()` when you need code to run after the current synchronous block but before the browser renders or any macrotasks fire:

```js
// Batching state updates
let needsUpdate = false;

function scheduleUpdate() {
  if (needsUpdate) return;
  needsUpdate = true;
  queueMicrotask(() => {
    needsUpdate = false;
    flushUpdates(); // One batched update per synchronous block
  });
}
```

Do NOT use `queueMicrotask` as a replacement for `setTimeout`. They have different scheduling semantics. Microtasks run before rendering; macrotasks run after.

## Async Iteration

### for-await-of

Iterates over async iterables, awaiting each value:

```js
async function* generateNumbers() {
  for (let i = 0; i < 5; i++) {
    await new Promise(resolve => setTimeout(resolve, 100));
    yield i;
  }
}

for await (const num of generateNumbers()) {
  console.log(num); // 0, 1, 2, 3, 4 (each 100ms apart)
}
```

### Async Generators

An async generator function combines `async function*` with `yield`. Its `.next()` method returns `Promise<{ value, done }>`.

```js
async function* paginate(baseUrl) {
  let page = 1;
  let hasMore = true;

  while (hasMore) {
    const response = await fetch(`${baseUrl}?page=${page}`);
    const data = await response.json();

    yield data.items; // Yield each page of results

    hasMore = data.hasNextPage;
    page++;
  }
}

// Consumer
for await (const items of paginate('/api/products')) {
  for (const item of items) {
    renderProduct(item);
  }
}
```

### Practical Patterns

**Reading a stream:**

```js
async function* readStream(stream) {
  const reader = stream.getReader();
  try {
    while (true) {
      const { done, value } = await reader.read();
      if (done) return;
      yield value;
    }
  } finally {
    reader.releaseLock();
  }
}

const response = await fetch('/api/large-file');
for await (const chunk of readStream(response.body)) {
  processChunk(chunk);
}
```

**Polling with timeout:**

```js
async function* poll(fn, intervalMs, signal) {
  while (!signal?.aborted) {
    yield await fn();
    await new Promise((resolve, reject) => {
      const timer = setTimeout(resolve, intervalMs);
      signal?.addEventListener('abort', () => {
        clearTimeout(timer);
        reject(new DOMException('Aborted', 'AbortError'));
      }, { once: true });
    });
  }
}

const controller = new AbortController();
for await (const status of poll(() => checkStatus(), 5000, controller.signal)) {
  updateUI(status);
  if (status.complete) {
    controller.abort();
  }
}
```

### Gotcha: for-await-of with Synchronous Iterables

Using `for await...of` with a regular synchronous iterable wraps each value in a Promise, adding unnecessary overhead:

```js
// Unnecessary overhead -- each value gets wrapped in Promise.resolve()
for await (const item of [1, 2, 3]) {
  console.log(item);
}

// Just use for...of for synchronous iterables
for (const item of [1, 2, 3]) {
  console.log(item);
}
```

## AbortController and Cancellation

`AbortController` provides a standard mechanism for cancelling asynchronous operations.

### Basic Usage

```js
const controller = new AbortController();
const { signal } = controller;

fetch('/api/data', { signal })
  .then(res => res.json())
  .then(data => console.log(data))
  .catch(err => {
    if (err.name === 'AbortError') {
      console.log('Request was cancelled');
    } else {
      throw err; // Re-throw non-cancellation errors
    }
  });

// Cancel after 5 seconds
setTimeout(() => controller.abort(), 5000);
```

### AbortSignal.timeout()

Creates a signal that automatically aborts after a specified duration:

```js
const response = await fetch('/api/data', {
  signal: AbortSignal.timeout(5000), // Aborts after 5 seconds
});
```

`AbortSignal.timeout()` throws a `TimeoutError`, not an `AbortError`:

```js
try {
  await fetch('/api/data', { signal: AbortSignal.timeout(5000) });
} catch (err) {
  if (err.name === 'TimeoutError') {
    console.log('Request timed out');
  } else if (err.name === 'AbortError') {
    console.log('Request was manually aborted');
  }
}
```

### AbortSignal.any()

Combines multiple signals. The resulting signal aborts when any of the input signals abort:

```js
const controller = new AbortController();
const timeoutSignal = AbortSignal.timeout(30000);

const response = await fetch('/api/data', {
  signal: AbortSignal.any([controller.signal, timeoutSignal]),
});
// Aborts on either user cancellation OR 30-second timeout
```

### Custom Abort Reasons

```js
const controller = new AbortController();
controller.abort(new Error('User navigated away'));

console.log(controller.signal.reason); // Error: User navigated away
```

### Removing Multiple Event Listeners at Once

```js
const controller = new AbortController();

element.addEventListener('click', onClick, { signal: controller.signal });
element.addEventListener('mousemove', onMove, { signal: controller.signal });
element.addEventListener('keydown', onKey, { signal: controller.signal });

// Remove all three listeners with one call
controller.abort();
```

### Building Cancellable Async Functions

```js
async function fetchWithRetry(url, { retries = 3, signal } = {}) {
  for (let attempt = 0; attempt <= retries; attempt++) {
    // Check before each attempt
    signal?.throwIfAborted();

    try {
      const response = await fetch(url, { signal });
      if (response.ok) return response;
      throw new Error(`HTTP ${response.status}`);
    } catch (err) {
      if (err.name === 'AbortError') throw err; // Never retry cancellations
      if (attempt === retries) throw err;
    }

    // Wait before retrying, but respect cancellation
    await new Promise((resolve, reject) => {
      const delay = 1000 * Math.pow(2, attempt);
      const timer = setTimeout(resolve, delay);
      signal?.addEventListener('abort', () => {
        clearTimeout(timer);
        reject(signal.reason);
      }, { once: true });
    });
  }
}
```

## Unhandled Rejections

An unhandled rejection occurs when a Promise is rejected and no `.catch()` or `try/catch` handles it.

### Consequences

- In Node.js 15+, unhandled rejections crash the process by default
- In browsers, unhandled rejections log a warning to the console
- Both environments fire events that you can listen to

### Global Handlers (Safety Net)

```js
// Browser
window.addEventListener('unhandledrejection', (event) => {
  console.error('Unhandled rejection:', event.reason);
  // Optionally send to error tracking service
  reportError(event.reason);
  event.preventDefault(); // Prevents default console logging
});

// Node.js
process.on('unhandledRejection', (reason, promise) => {
  console.error('Unhandled rejection at:', promise, 'reason:', reason);
  // Log and let the process crash (do not swallow)
});
```

### Common Sources of Unhandled Rejections

```js
// 1. Forgetting to catch
fetchData(); // No .catch(), no try/catch

// 2. Promise.all with multiple rejections
try {
  await Promise.all([failA(), failB(), failC()]);
} catch (err) {
  // Catches failA()'s rejection, but failB() and failC()
  // may also reject -- those rejections are unhandled
}

// 3. Floating promises in event handlers
button.addEventListener('click', async () => {
  await riskyOperation(); // If this rejects, nobody catches it
});
```

### Prevention

```js
// Always catch at the boundary
button.addEventListener('click', async () => {
  try {
    await riskyOperation();
  } catch (err) {
    showErrorToast(err.message);
  }
});

// Use Promise.allSettled when you expect partial failures
const results = await Promise.allSettled([failA(), failB(), failC()]);

// Use ESLint rule: @typescript-eslint/no-floating-promises
```

## Performance Considerations

### The Cost of async/await

- Each `async` function creates a Promise wrapper, allocating memory
- Modern engines (V8 7.2+) reduced the overhead from 3 microticks per await to 1
- For typical I/O-bound code, the overhead is negligible
- In tight loops processing millions of items, it is measurable

```js
// BAD: async wrapper around synchronous code adds overhead
async function add(a, b) {
  return a + b; // Wraps result in a Promise for no reason
}

// GOOD: keep synchronous functions synchronous
function add(a, b) {
  return a + b;
}
```

### Sequential vs Parallel

```js
// 10 sequential API calls at 200ms each = ~2000ms total
for (const id of ids) {
  const item = await fetchItem(id);
  results.push(item);
}

// 10 parallel API calls at 200ms each = ~200ms total
const results = await Promise.all(ids.map(id => fetchItem(id)));
```

When processing large arrays in parallel, limit concurrency to avoid overwhelming the server (see Real-World Use Cases).

## Common Mistakes

### 1. Forgetting await

```js
// BAD: logs a pending Promise, not the value
async function getUser() {
  const user = fetch('/api/user').then(r => r.json());
  console.log(user); // Promise { <pending> }
}

// GOOD
async function getUser() {
  const user = await fetch('/api/user').then(r => r.json());
  console.log(user); // { name: "Alice", ... }
}
```

### 2. async Callbacks in forEach

`Array.prototype.forEach` ignores the return value of its callback. If the callback is async, the returned Promise is silently discarded:

```js
// BAD: forEach does NOT await -- all iterations fire simultaneously
// and the function returns before any completes
async function processItems(items) {
  items.forEach(async (item) => {
    await saveItem(item);
  });
  console.log('Done!'); // Runs BEFORE any item is saved
}

// FIX: Sequential
async function processItems(items) {
  for (const item of items) {
    await saveItem(item);
  }
  console.log('Done!'); // Actually done
}

// FIX: Parallel
async function processItems(items) {
  await Promise.all(items.map(item => saveItem(item)));
  console.log('Done!');
}
```

This applies to `.map()` as well if you do not `await` the result:

```js
// BAD: results is an array of Promises, not values
const results = items.map(async item => {
  return await process(item);
});
console.log(results); // [Promise, Promise, Promise]

// GOOD
const results = await Promise.all(items.map(item => process(item)));
```

### 3. Sequential Awaits for Independent Operations

```js
// BAD: 6 seconds total (2 + 2 + 2)
const users = await fetchUsers();     // 2s
const posts = await fetchPosts();     // 2s
const comments = await fetchComments(); // 2s

// GOOD: 2 seconds total (all run in parallel)
const [users, posts, comments] = await Promise.all([
  fetchUsers(),
  fetchPosts(),
  fetchComments(),
]);
```

### 4. Swallowing Errors in .catch()

```js
// BAD: error is caught but not re-thrown, caller thinks everything is fine
async function getData() {
  return fetch('/api/data')
    .then(r => r.json())
    .catch(err => {
      console.error(err); // Logs error
      // Returns undefined -- caller gets undefined instead of an error
    });
}

// GOOD: either re-throw or return a meaningful fallback
async function getData() {
  try {
    const res = await fetch('/api/data');
    return await res.json();
  } catch (err) {
    console.error(err);
    throw err; // Re-throw so caller knows it failed
  }
}
```

### 5. Mixing .then() and await Incorrectly

```js
// BAD: rejection from .then() chain bypasses the catch block
async function mixed() {
  try {
    return fetch('/api/data').then(r => r.json());
    // If .json() throws, this catch does NOT fire
  } catch (err) {
    console.log('Not caught');
  }
}

// GOOD: use await consistently
async function clean() {
  try {
    const res = await fetch('/api/data');
    return await res.json();
  } catch (err) {
    console.log('Caught!');
  }
}
```

### 6. Creating Promises Unnecessarily

```js
// BAD: wrapping an existing Promise in a new one
function fetchUser(id) {
  return new Promise((resolve, reject) => {
    fetch(`/api/users/${id}`)
      .then(res => res.json())
      .then(resolve)
      .catch(reject);
  });
}

// GOOD: just return the existing Promise chain
function fetchUser(id) {
  return fetch(`/api/users/${id}`).then(res => res.json());
}

// Or use async/await
async function fetchUser(id) {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
}
```

This anti-pattern is called the **explicit Promise construction anti-pattern**.

### 7. Not Cancelling Async Operations on Cleanup

```js
// BAD: request continues even after component unmounts
function loadData() {
  fetch('/api/data').then(r => r.json()).then(setData);
}

// GOOD: cancel on cleanup
function loadData() {
  const controller = new AbortController();

  fetch('/api/data', { signal: controller.signal })
    .then(r => r.json())
    .then(setData)
    .catch(err => {
      if (err.name !== 'AbortError') throw err;
    });

  return () => controller.abort(); // Return cleanup function
}
```

## Best Practices

- **Prefer async/await over .then() chains** for readability, but understand both patterns thoroughly.

- **Always handle errors.** Use `try/catch` with async/await. Enable ESLint's `no-floating-promises` rule.

- **Use `return await` inside try/catch blocks** so rejections are caught locally.

- **Use `Promise.all()` for independent parallel operations.** Never sequentially await independent calls.

- **Use `Promise.allSettled()` when partial failure is acceptable** (dashboards, multi-source data fetching).

- **Use `AbortController` for cancellation.** Prefer `AbortSignal.timeout()` over manual setTimeout race patterns. Use `AbortSignal.any()` to combine cancellation conditions.

- **Never use async callbacks with `forEach`, `map`, `filter`, or `reduce`** without proper handling. Use `for...of` for sequential or `Promise.all(arr.map(...))` for parallel.

- **Do not mark functions `async` if they never use `await`.** It adds unnecessary Promise wrapping overhead.

- **Use `Promise.withResolvers()`** (ES2024) instead of the old deferred pattern.

- **Set up global unhandled rejection handlers** as a safety net in both browser and Node.js.

- **Limit concurrency** when processing large batches to avoid overwhelming servers or running out of memory.

## Real-World Use Cases

### 1. Retry with Exponential Backoff and Jitter

```js
async function retry(fn, { retries = 3, baseDelay = 1000, signal } = {}) {
  for (let attempt = 0; attempt <= retries; attempt++) {
    signal?.throwIfAborted();

    try {
      return await fn({ signal });
    } catch (err) {
      if (err.name === 'AbortError') throw err;
      if (attempt === retries) throw err;

      // Exponential backoff with jitter to prevent thundering herd
      const delay = baseDelay * Math.pow(2, attempt);
      const jitter = delay * (0.5 + Math.random() * 0.5);

      await new Promise((resolve, reject) => {
        const timer = setTimeout(resolve, jitter);
        signal?.addEventListener('abort', () => {
          clearTimeout(timer);
          reject(signal.reason);
        }, { once: true });
      });
    }
  }
}

// Usage
const data = await retry(
  ({ signal }) => fetch('/api/flaky-endpoint', { signal }).then(r => r.json()),
  { retries: 3, baseDelay: 1000 }
);
```

### 2. Concurrent Task Pool (Limit Parallelism)

When you need to process hundreds of items but only want N concurrent requests:

```js
async function pool(tasks, concurrency) {
  const results = [];
  const executing = new Set();

  for (const [index, task] of tasks.entries()) {
    const p = Promise.resolve()
      .then(() => task())
      .then(result => { results[index] = result; });

    executing.add(p);
    const clean = () => executing.delete(p);
    p.then(clean, clean);

    if (executing.size >= concurrency) {
      await Promise.race(executing);
    }
  }

  await Promise.all(executing);
  return results;
}

// Process 100 URLs, max 5 at a time
const tasks = urls.map(url => () => fetch(url).then(r => r.json()));
const results = await pool(tasks, 5);
```

### 3. Async Queue (Sequential Processing)

A mutex/serial queue ensures operations execute one at a time:

```js
function createQueue() {
  let chain = Promise.resolve();

  return function enqueue(fn) {
    const result = chain.then(fn);
    // Prevent rejection from breaking the chain
    chain = result.catch(() => {});
    return result;
  };
}

const queue = createQueue();

// These run one at a time, in order, regardless of when they are called
queue(() => saveToDB(record1));
queue(() => saveToDB(record2));
queue(() => saveToDB(record3));
```

### 4. Debounced Async Search

```js
function debounceAsync(fn, ms) {
  let timeoutId;
  let pending = [];

  return function (...args) {
    clearTimeout(timeoutId);

    return new Promise((resolve, reject) => {
      pending.push({ resolve, reject });

      timeoutId = setTimeout(async () => {
        const batch = pending;
        pending = [];
        try {
          const result = await fn.apply(this, args);
          batch.forEach(p => p.resolve(result));
        } catch (err) {
          batch.forEach(p => p.reject(err));
        }
      }, ms);
    });
  };
}

const search = debounceAsync(
  (query) => fetch(`/api/search?q=${encodeURIComponent(query)}`).then(r => r.json()),
  300
);

// Multiple rapid calls all receive the same debounced result
searchInput.addEventListener('input', async (e) => {
  const results = await search(e.target.value);
  renderResults(results);
});
```
