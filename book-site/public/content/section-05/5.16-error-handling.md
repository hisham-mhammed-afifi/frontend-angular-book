# 5.16 Error Handling

> Error handling is one of the most under-practiced skills in JavaScript development. Most developers learn the `try/catch` syntax and stop there. Real-world error handling requires understanding error propagation, custom error hierarchies, async error traps, global handlers, serialization edge cases, and deliberate patterns that separate expected failures from unexpected crashes.

---

## try / catch / finally

The `try/catch/finally` statement is JavaScript's primary synchronous error handling mechanism.

### Basic Syntax

```js
try {
  // Code that might throw
  const result = riskyOperation();
  console.log(result);
} catch (error) {
  // Runs ONLY if try block throws
  console.error('Something went wrong:', error.message);
} finally {
  // Runs ALWAYS, whether try succeeded or catch ran
  cleanup();
}
```

Key rules:
- The `catch` block is optional if `finally` is present
- The `finally` block is optional if `catch` is present
- You must have at least one of `catch` or `finally`

### try/finally Without catch

When you need cleanup but want the error to propagate:

```js
function readResource() {
  const resource = acquireResource();
  try {
    return processResource(resource);
  } finally {
    // Runs even though there is no catch
    // If processResource throws, the error propagates AFTER finally runs
    resource.release();
  }
}
```

### The catch Binding is Optional

Since ES2019, the catch binding (the error parameter) is optional:

```js
// ES2019+: catch binding is optional
try {
  JSON.parse(userInput);
} catch {
  // No (error) parameter needed if you do not use the error
  return defaultValue;
}
```

### The finally Return Gotcha

This is the single most dangerous edge case with `try/catch/finally`. A `return` statement inside `finally` overrides any `return` or `throw` from `try` or `catch`:

```js
function gotcha() {
  try {
    return 'from try';
  } finally {
    return 'from finally'; // This wins. Always.
  }
}

console.log(gotcha()); // 'from finally'
```

The same applies to `throw`:

```js
function swallowed() {
  try {
    throw new Error('Critical failure!');
  } finally {
    return 'everything is fine'; // Silently swallows the error
  }
}

console.log(swallowed()); // 'everything is fine' -- error is gone
```

And it also works with `catch`:

```js
function overridden() {
  try {
    throw new Error('Original error');
  } catch (e) {
    throw new Error('Catch error'); // This throw is abandoned
  } finally {
    return 'finally return'; // Overrides the throw from catch
  }
}

console.log(overridden()); // 'finally return'
```

**Rule: Never use `return`, `throw`, `break`, or `continue` inside `finally` blocks.** The `finally` block is for cleanup only.

### Nesting try/catch

Nested `try/catch` blocks work, but errors in the `catch` or `finally` blocks are not caught by the same `try/catch` construct:

```js
try {
  try {
    throw new Error('Inner error');
  } catch (innerErr) {
    // This throw is caught by the OUTER catch, not the inner one
    throw new Error('Error in inner catch');
  } finally {
    console.log('Inner finally runs');
  }
} catch (outerErr) {
  console.log(outerErr.message); // 'Error in inner catch'
}
```

### try/catch Does Not Catch Asynchronous Errors

This is a common misconception. `try/catch` only catches synchronous errors or errors in `await` expressions:

```js
// BAD: this does NOT work
try {
  setTimeout(() => {
    throw new Error('Async error'); // Uncaught!
  }, 1000);
} catch (e) {
  // This NEVER runs for the setTimeout callback
  console.error(e);
}

// BAD: this does NOT work either
try {
  fetch('/api/data').then(() => {
    throw new Error('Error inside .then'); // Uncaught promise rejection
  });
} catch (e) {
  // Never runs
}
```

The `try/catch` block finishes executing long before the asynchronous callback runs. The callback executes in a different call stack frame.

### Conditional catch Blocks

JavaScript does not have conditional catch syntax like Java's multi-catch. You use `instanceof` checks instead:

```js
try {
  performOperation();
} catch (error) {
  if (error instanceof TypeError) {
    handleTypeError(error);
  } else if (error instanceof RangeError) {
    handleRangeError(error);
  } else if (error instanceof CustomValidationError) {
    handleValidation(error);
  } else {
    // Re-throw unknown errors; do not swallow them
    throw error;
  }
}
```

**Always re-throw errors you do not know how to handle.** Swallowing unknown errors is one of the worst debugging anti-patterns.

## The Error Object

All built-in error types inherit from `Error.prototype`. An Error instance has three standard properties:

| Property  | Description |
|-----------|-------------|
| `message` | Human-readable error description |
| `name`    | The error type name (e.g., `"TypeError"`) |
| `stack`   | Non-standard but universally supported stack trace string |

```js
const err = new Error('Something failed');
console.log(err.name);    // 'Error'
console.log(err.message); // 'Something failed'
console.log(err.stack);   // 'Error: Something failed\n    at ...'
```

### Error Can Be Called Without `new`

Unlike most constructors, `Error()` without `new` still returns an Error instance:

```js
const err1 = new Error('with new');
const err2 = Error('without new');

console.log(err1 instanceof Error); // true
console.log(err2 instanceof Error); // true
```

This is true for all built-in error types. However, always use `new` for clarity.

### Built-in Error Types

JavaScript has eight built-in error constructors:

| Error Type       | When Thrown | Example |
|------------------|-------------|---------|
| `Error`          | Generic error; use as base class | `throw new Error('fail')` |
| `TypeError`      | Value is not the expected type | `null.property`, `undefined()` |
| `ReferenceError` | Non-existent variable is referenced | `console.log(undeclaredVar)` |
| `SyntaxError`    | Code cannot be parsed | `eval('if(')`, `JSON.parse('{bad}')` |
| `RangeError`     | Value outside allowed range | `new Array(-1)`, `(1).toFixed(200)` |
| `URIError`       | Malformed URI function argument | `decodeURIComponent('%')` |
| `EvalError`      | Legacy; related to `eval()` | Rarely thrown in modern JS |
| `AggregateError` | Multiple errors wrapped in one (ES2021) | `Promise.any([])` when all reject |

### TypeError in Detail

`TypeError` is by far the most common runtime error in JavaScript. It occurs when:

```js
// Accessing property of null or undefined
const obj = null;
obj.name; // TypeError: Cannot read properties of null

// Calling a non-function
const x = 42;
x(); // TypeError: x is not a function

// Assigning to a read-only property in strict mode
'use strict';
const frozen = Object.freeze({ a: 1 });
frozen.a = 2; // TypeError: Cannot assign to read only property

// Iterating a non-iterable
for (const item of 42) {} // TypeError: 42 is not iterable

// Modifying a const value
const y = 10;
y = 20; // TypeError: Assignment to constant variable
```

### RangeError in Detail

```js
// Invalid array length
new Array(-1);             // RangeError: Invalid array length
new Array(4294967296);     // RangeError: Invalid array length (> 2^32 - 1)

// Number method precision
(1.5).toFixed(200);        // RangeError: toFixed() digits argument must be between 0 and 100
(1.5).toPrecision(200);    // RangeError

// Stack overflow (recursive)
function recurse() { recurse(); }
recurse(); // RangeError: Maximum call stack size exceeded

// Invalid normalization form
'abc'.normalize('INVALID'); // RangeError: The normalization form should be one of NFC, NFD, NFKC, NFKD
```

### ReferenceError in Detail

```js
// Accessing undeclared variable
console.log(undeclaredVar); // ReferenceError: undeclaredVar is not defined

// Using let/const before declaration (temporal dead zone)
console.log(x); // ReferenceError: Cannot access 'x' before initialization
let x = 10;

// Assigning to an undeclared variable in strict mode
'use strict';
undeclaredVar = 5; // ReferenceError: undeclaredVar is not defined
```

### SyntaxError at Runtime

`SyntaxError` at parse time halts the entire script. But `SyntaxError` can also be thrown at runtime by functions that parse strings:

```js
// Runtime SyntaxError from JSON.parse
try {
  JSON.parse('{"key": undefined}');
} catch (e) {
  console.log(e instanceof SyntaxError); // true
  console.log(e.message); // Unexpected token u in JSON at position 8
}

// Runtime SyntaxError from eval
try {
  eval('if(');
} catch (e) {
  console.log(e instanceof SyntaxError); // true
}

// Runtime SyntaxError from Function constructor
try {
  new Function('return {');
} catch (e) {
  console.log(e instanceof SyntaxError); // true
}
```

### AggregateError (ES2021)

`AggregateError` represents multiple errors wrapped in a single error. It was introduced primarily for `Promise.any()`:

```js
// AggregateError from Promise.any when ALL promises reject
const promises = [
  Promise.reject(new Error('Service A failed')),
  Promise.reject(new Error('Service B failed')),
  Promise.reject(new Error('Service C failed')),
];

try {
  await Promise.any(promises);
} catch (e) {
  console.log(e instanceof AggregateError); // true
  console.log(e.message);                   // 'All promises were rejected'
  console.log(e.errors);                    // Array of 3 Error objects
  e.errors.forEach(err => console.log(err.message));
  // 'Service A failed'
  // 'Service B failed'
  // 'Service C failed'
}
```

You can create `AggregateError` manually for form validation or batch operations:

```js
function validateForm(data) {
  const errors = [];

  if (!data.email) errors.push(new Error('Email is required'));
  if (!data.name) errors.push(new Error('Name is required'));
  if (data.age < 0) errors.push(new RangeError('Age must be non-negative'));

  if (errors.length > 0) {
    throw new AggregateError(errors, 'Form validation failed');
  }

  return data;
}

try {
  validateForm({ email: '', name: '', age: -5 });
} catch (e) {
  if (e instanceof AggregateError) {
    console.log(`${e.message}:`);
    e.errors.forEach(err => console.log(`  - ${err.message}`));
  }
}
// Form validation failed:
//   - Email is required
//   - Name is required
//   - Age must be non-negative
```

### AggregateError for Parallel Operations

```js
async function fetchFromMultipleSources(urls) {
  const results = await Promise.allSettled(urls.map(url => fetch(url)));

  const failures = results
    .filter(r => r.status === 'rejected')
    .map(r => r.reason);

  const successes = results
    .filter(r => r.status === 'fulfilled')
    .map(r => r.value);

  if (failures.length > 0 && successes.length === 0) {
    throw new AggregateError(failures, 'All data sources failed');
  }

  if (failures.length > 0) {
    console.warn(`${failures.length} sources failed, continuing with partial data`);
  }

  return successes;
}
```

## Custom Error Classes

Extending `Error` is essential for production codebases. Custom errors enable type-safe error handling with `instanceof`, carry domain-specific context, and create clear error hierarchies.

### Basic Custom Error

```js
class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    this.name = 'ValidationError';
    this.field = field;
  }
}

try {
  throw new ValidationError('Email format is invalid', 'email');
} catch (e) {
  if (e instanceof ValidationError) {
    console.log(`Field "${e.field}": ${e.message}`);
    // Field "email": Email format is invalid
  }
}
```

### Using constructor.name for Automatic Name Setting

Setting `this.name` manually in every subclass is tedious and error-prone. Use `this.constructor.name` to set it automatically:

```js
class AppError extends Error {
  constructor(message, options) {
    super(message, options);
    // Automatically set name to the class name
    this.name = this.constructor.name;
  }
}

class NotFoundError extends AppError {
  constructor(resource, id, options) {
    super(`${resource} with id "${id}" not found`, options);
    this.resource = resource;
    this.resourceId = id;
    this.statusCode = 404;
  }
}

class AuthenticationError extends AppError {
  constructor(message = 'Authentication required', options) {
    super(message, options);
    this.statusCode = 401;
  }
}

const err = new NotFoundError('User', '123');
console.log(err.name);       // 'NotFoundError' (automatic)
console.log(err.message);    // 'User with id "123" not found'
console.log(err.statusCode); // 404
console.log(err instanceof NotFoundError); // true
console.log(err instanceof AppError);      // true
console.log(err instanceof Error);         // true
```

### Preserving Stack Trace with Error.captureStackTrace

`Error.captureStackTrace` is a V8-specific API (Chrome, Node.js, Edge) that cleans up the stack trace by removing the custom error's constructor frames:

```js
class HttpError extends Error {
  constructor(statusCode, message, options) {
    super(message, options);
    this.name = this.constructor.name;
    this.statusCode = statusCode;

    // V8 only: remove this constructor from the stack trace
    if (Error.captureStackTrace) {
      Error.captureStackTrace(this, this.constructor);
    }
  }
}
```

Without `captureStackTrace`, the stack trace starts inside the `HttpError` constructor. With it, the stack trace starts at the call site where `new HttpError(...)` was used. This makes debugging easier.

### Error Hierarchy Pattern

```js
// Base application error
class AppError extends Error {
  constructor(message, options) {
    super(message, options);
    this.name = this.constructor.name;
    this.timestamp = new Date().toISOString();
    if (Error.captureStackTrace) {
      Error.captureStackTrace(this, this.constructor);
    }
  }

  toJSON() {
    return {
      name: this.name,
      message: this.message,
      timestamp: this.timestamp,
      ...(this.cause && { cause: this.cause }),
    };
  }
}

// Operational errors (expected, recoverable)
class OperationalError extends AppError {
  constructor(message, options) {
    super(message, options);
    this.isOperational = true;
  }
}

// Network errors
class NetworkError extends OperationalError {
  constructor(message, statusCode, options) {
    super(message, options);
    this.statusCode = statusCode;
  }
}

// Validation errors
class ValidationError extends OperationalError {
  constructor(message, violations = [], options) {
    super(message, options);
    this.violations = violations;
  }
}

// Programming errors (bugs, should crash)
class ProgrammerError extends AppError {
  constructor(message, options) {
    super(message, options);
    this.isOperational = false;
  }
}
```

This hierarchy enables global error handling that differentiates between recoverable operational errors and unrecoverable programming bugs.

## Error.cause (ES2022)

ES2022 introduced the `cause` property, allowing you to chain errors and preserve the original error context. This is the standard way to wrap low-level errors with higher-level context.

### Basic Usage

```js
async function fetchUserProfile(userId) {
  try {
    const response = await fetch(`/api/users/${userId}`);
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}`);
    }
    return await response.json();
  } catch (error) {
    // Wrap the original error with business context
    throw new Error(`Failed to load profile for user ${userId}`, {
      cause: error,
    });
  }
}

try {
  await fetchUserProfile('abc-123');
} catch (error) {
  console.log(error.message);       // 'Failed to load profile for user abc-123'
  console.log(error.cause.message); // 'HTTP 404' (or whatever the original error was)
}
```

### Error Cause Chain

You can chain multiple levels of errors:

```js
async function getOrderSummary(orderId) {
  try {
    const order = await fetchOrder(orderId);
    try {
      const user = await fetchUserProfile(order.userId);
      return { order, user };
    } catch (error) {
      throw new Error('Failed to build order summary', { cause: error });
    }
  } catch (error) {
    throw new Error(`Order summary unavailable for ${orderId}`, { cause: error });
  }
}
```

### Logging the Full Cause Chain

By default, `console.error` does not display the full cause chain. You must walk it manually:

```js
function logErrorChain(error, depth = 0) {
  const indent = '  '.repeat(depth);
  console.error(`${indent}${error.name}: ${error.message}`);

  if (error.stack) {
    const stackLines = error.stack.split('\n').slice(1, 4); // first 3 frames
    stackLines.forEach(line => console.error(`${indent}  ${line.trim()}`));
  }

  if (error.cause) {
    console.error(`${indent}Caused by:`);
    logErrorChain(error.cause, depth + 1);
  }
}

// Usage
try {
  await getOrderSummary('order-456');
} catch (error) {
  logErrorChain(error);
}
// Error: Order summary unavailable for order-456
//   at getOrderSummary (app.js:15:11)
// Caused by:
//   Error: Failed to build order summary
//     at getOrderSummary (app.js:10:13)
//   Caused by:
//     Error: Failed to load profile for user user-789
//       at fetchUserProfile (api.js:8:11)
//     Caused by:
//       TypeError: Failed to fetch
```

### Using cause with Custom Errors

```js
class DatabaseError extends AppError {
  constructor(operation, table, options) {
    super(`Database ${operation} failed on table "${table}"`, options);
    this.operation = operation;
    this.table = table;
  }
}

async function createUser(userData) {
  try {
    return await db.query('INSERT INTO users ...', userData);
  } catch (dbError) {
    throw new DatabaseError('INSERT', 'users', { cause: dbError });
  }
}
```

### The cause Property is Non-Enumerable

```js
const err = new Error('outer', { cause: new Error('inner') });
console.log(Object.keys(err));                    // [] (no enumerable own properties)
console.log(err.cause);                           // Error: inner
console.log(JSON.stringify(err));                  // '{}'  -- cause is NOT serialized by default
console.log(Object.getOwnPropertyDescriptor(err, 'cause'));
// { value: Error: inner, writable: true, enumerable: false, configurable: true }
```

This is important for serialization: if you use `JSON.stringify` on an error, `cause` is lost. You need a custom `toJSON` method.

## The throw Statement

### You Can Throw Anything

JavaScript allows you to throw any value:

```js
throw 'string error';
throw 42;
throw true;
throw null;
throw undefined;
throw { code: 'INVALID', field: 'email' };
throw ['error1', 'error2'];
```

### Why You Should Always Throw Error Objects

Throwing non-Error values is a bad practice for several reasons:

```js
// BAD: Throwing a string
try {
  throw 'something went wrong';
} catch (e) {
  console.log(e.message);    // undefined -- strings do not have .message
  console.log(e.stack);      // undefined -- no stack trace
  console.log(e.name);       // undefined -- no error type
  console.log(typeof e);     // 'string'
  console.log(e instanceof Error); // false
}

// GOOD: Throwing an Error
try {
  throw new Error('something went wrong');
} catch (e) {
  console.log(e.message);    // 'something went wrong'
  console.log(e.stack);      // Full stack trace
  console.log(e.name);       // 'Error'
  console.log(e instanceof Error); // true
}
```

| Capability             | `throw new Error(msg)` | `throw 'string'` | `throw object` |
|------------------------|:----------------------:|:-----------------:|:--------------:|
| `.message` property    | Yes                    | No                | If added       |
| `.stack` trace         | Yes                    | No                | No             |
| `.name` property       | Yes                    | No                | If added       |
| `instanceof` checking  | Yes                    | No                | No             |
| `cause` chaining       | Yes                    | No                | No             |
| Caught by global handlers properly | Yes      | Partially         | Partially      |

### Re-throwing Errors

When you catch an error but cannot handle it, always re-throw:

```js
function processData(data) {
  try {
    return JSON.parse(data);
  } catch (error) {
    if (error instanceof SyntaxError) {
      // We know how to handle this
      console.warn('Invalid JSON, using default');
      return {};
    }
    // Unknown error -- let it propagate
    throw error;
  }
}
```

### Wrapping and Re-throwing

Add context while preserving the original error:

```js
async function loadConfig(path) {
  try {
    const raw = await fs.readFile(path, 'utf-8');
    return JSON.parse(raw);
  } catch (error) {
    // Do not just re-throw: add context
    throw new Error(`Failed to load config from "${path}"`, { cause: error });
  }
}
```

## Error Handling in Async Code

### try/catch with async/await

`try/catch` works naturally with `await`:

```js
async function fetchUser(id) {
  try {
    const response = await fetch(`/api/users/${id}`);
    if (!response.ok) {
      throw new Error(`HTTP error: ${response.status}`);
    }
    return await response.json();
  } catch (error) {
    console.error('Failed to fetch user:', error.message);
    return null;
  }
}
```

### Gotcha: Missing await

If you forget `await`, the `try/catch` will not catch the rejection:

```js
async function broken() {
  try {
    // BAD: missing await -- the returned promise rejection is NOT caught
    fetch('/api/data').then(r => r.json());
  } catch (e) {
    // Never reached for the fetch error
  }
}

async function fixed() {
  try {
    // GOOD: await the promise
    const response = await fetch('/api/data');
    return await response.json();
  } catch (e) {
    console.error('Caught:', e.message);
  }
}
```

### Gotcha: return vs return await

Inside a `try` block, `return promise` vs `return await promise` behaves differently:

```js
async function returnWithoutAwait() {
  try {
    // BAD: if fetchData() rejects, the catch block does NOT run
    // because the promise is returned before it settles
    return fetchData();
  } catch (e) {
    console.error('This will NOT catch fetchData rejections');
  }
}

async function returnWithAwait() {
  try {
    // GOOD: await forces the promise to settle inside the try block
    return await fetchData();
  } catch (e) {
    console.error('This WILL catch fetchData rejections');
  }
}
```

**Rule: Always use `return await` inside `try` blocks.**

### Promise .catch() Pattern

You can handle errors at the promise level instead of using `try/catch`:

```js
const data = await fetch('/api/data')
  .then(r => {
    if (!r.ok) throw new Error(`HTTP ${r.status}`);
    return r.json();
  })
  .catch(error => {
    console.error('Fetch failed:', error.message);
    return { fallback: true };
  });
```

### Go-Style Error Handling with await

A pattern inspired by Go's error handling where you wrap the promise to return a tuple:

```js
function to(promise) {
  return promise
    .then(data => [null, data])
    .catch(err => [err, null]);
}

// Usage
async function loadDashboard() {
  const [userErr, user] = await to(fetchUser(1));
  if (userErr) {
    showError('Failed to load user');
    return;
  }

  const [ordersErr, orders] = await to(fetchOrders(user.id));
  if (ordersErr) {
    showError('Failed to load orders');
    return;
  }

  render({ user, orders });
}
```

### Handling Multiple Concurrent Promises

```js
// Promise.all: fails fast on first rejection
try {
  const [users, orders, products] = await Promise.all([
    fetchUsers(),
    fetchOrders(),
    fetchProducts(),
  ]);
} catch (error) {
  // Only the FIRST rejection error is caught
  // Other pending promises continue running but their results are lost
  console.error('One of the fetches failed:', error.message);
}

// Promise.allSettled: never rejects, all results are captured
const results = await Promise.allSettled([
  fetchUsers(),
  fetchOrders(),
  fetchProducts(),
]);

const fulfilled = results.filter(r => r.status === 'fulfilled').map(r => r.value);
const rejected = results.filter(r => r.status === 'rejected').map(r => r.reason);

if (rejected.length > 0) {
  console.error(`${rejected.length} operations failed`);
  rejected.forEach(err => console.error('-', err.message));
}
```

### Unhandled Promise Rejections

An unhandled rejection occurs when a promise rejects and no `.catch()` or `try/catch` handles it:

```js
// BAD: unhandled rejection -- no catch anywhere
async function leaky() {
  const data = await fetch('/bad-url'); // If this rejects, it is unhandled
}
leaky(); // No .catch(), no try/catch wrapping the call

// BAD: fire-and-forget async call
async function doSomething() {
  throw new Error('oops');
}
doSomething(); // Unhandled rejection -- nobody awaits or catches

// GOOD: always handle or propagate
doSomething().catch(err => console.error('Handled:', err.message));
```

In modern Node.js (v15+), unhandled promise rejections cause the process to crash by default. In browsers, they log to the console and fire the `unhandledrejection` event.

## Global Error Handlers

### Browser: window.onerror

Catches uncaught synchronous errors:

```js
window.onerror = function (message, source, lineno, colno, error) {
  console.error('Global error:', {
    message, // Human-readable message
    source,  // Script URL
    lineno,  // Line number
    colno,   // Column number
    error,   // The Error object (may be null for cross-origin scripts)
  });

  // Returning true prevents the default browser error logging
  return true;
};
```

### Browser: window.addEventListener('error')

An alternative that uses the standard event model:

```js
window.addEventListener('error', (event) => {
  console.error('Caught:', event.error);
  // event.message, event.filename, event.lineno, event.colno
  event.preventDefault(); // Equivalent to returning true in onerror
});
```

**Key difference:** The event listener approach also catches resource loading errors (images, scripts, stylesheets) when used with the `true` (capture phase) parameter:

```js
// Catch resource loading errors (images, scripts, etc.)
window.addEventListener('error', (event) => {
  if (event.target !== window) {
    // This is a resource loading error, not a JS error
    console.error('Resource failed to load:', event.target.src || event.target.href);
  }
}, true); // Must use capture phase (third argument = true)
```

### Browser: unhandledrejection

Catches unhandled promise rejections:

```js
window.addEventListener('unhandledrejection', (event) => {
  console.error('Unhandled promise rejection:', event.reason);

  // Prevent the default browser logging
  event.preventDefault();

  // Send to error tracking service
  errorTracker.capture(event.reason);
});
```

### Browser: rejectionhandled

Fires when a previously unhandled rejection gets handled later:

```js
window.addEventListener('rejectionhandled', (event) => {
  console.log('Late-handled rejection:', event.reason);
  // You might want to remove this from your error tracking
});
```

### Node.js: uncaughtException

```js
process.on('uncaughtException', (error, origin) => {
  console.error('Uncaught exception:', error);
  console.error('Origin:', origin);

  // Log the error, then exit
  // WARNING: The process is in an undefined state after an uncaught exception.
  // You should NOT continue running. Clean up and exit.
  logToService(error).finally(() => process.exit(1));
});
```

### Node.js: unhandledRejection

```js
process.on('unhandledRejection', (reason, promise) => {
  console.error('Unhandled rejection:', reason);
  // In Node.js 15+, this will crash the process by default
});
```

### Cross-Origin Script Error Limitation

When an error occurs in a script loaded from a different origin, the browser reports it as `"Script error."` with no useful details (line 0, column 0, null error object). This is a security feature to prevent information leakage.

To get full error details for cross-origin scripts:

```html
<!-- 1. Add crossorigin attribute to the script tag -->
<script src="https://cdn.example.com/lib.js" crossorigin="anonymous"></script>
```

```
# 2. The server must respond with the CORS header
Access-Control-Allow-Origin: *
```

The `reportError()` global function dispatches an error event to the global error handler, similar to an unhandled throw:

```js
// Report an error to global handlers without actually throwing
try {
  riskyOperation();
```

## Stack Traces

### The stack Property

The `stack` property is a string containing the call stack at the point the error was created. It is non-standard but supported by all major engines:

```js
function innerFunction() {
  throw new Error('Deep error');
}

function middleFunction() {
  innerFunction();
}

function outerFunction() {
  middleFunction();
}

try {
  outerFunction();
} catch (e) {
  console.log(e.stack);
}
// Error: Deep error
//     at innerFunction (app.js:2:9)
//     at middleFunction (app.js:6:3)
//     at outerFunction (app.js:10:3)
//     at Object.<anonymous> (app.js:14:3)
```

### Error.captureStackTrace (V8 Only)

This V8-specific API creates the `stack` property on a target object. Its second argument (constructor opt) specifies a function; all frames above and including calls to this function are omitted from the trace:

```js
function createCustomStack() {
  const obj = {};
  Error.captureStackTrace(obj, createCustomStack);
  return obj.stack;
}

// The stack trace will NOT include createCustomStack itself
console.log(createCustomStack());
```

Use it in custom error classes to remove the constructor from the trace:

```js
class AppError extends Error {
  constructor(message) {
    super(message);
    this.name = this.constructor.name;

    // Remove AppError constructor from stack trace
    if (Error.captureStackTrace) {
      Error.captureStackTrace(this, this.constructor);
    }
  }
}
```

### Error.stackTraceLimit (V8 Only)

Controls how many stack frames are captured:

```js
// Default is 10 in V8
Error.stackTraceLimit = 10;

// Increase for deep call stacks
Error.stackTraceLimit = 50;

// Capture ALL frames (use with caution -- performance impact)
Error.stackTraceLimit = Infinity;

// Disable stack trace capture entirely (slight performance boost)
Error.stackTraceLimit = 0;
```

### Async Stack Traces

Modern engines provide async stack traces that show the chain of `await` calls:

```js
async function fetchData() {
  throw new Error('Network timeout');
}

async function processUser() {
  const data = await fetchData(); // async stack trace starts here
  return transform(data);
}

async function main() {
  await processUser();
}

main().catch(console.error);
// Error: Network timeout
//     at fetchData (app.js:2:9)
//     at async processUser (app.js:6:16)
//     at async main (app.js:11:3)
```

The `async` prefix in the stack trace indicates frames that crossed an `await` boundary. This works in Chrome DevTools and Node.js with the `--async-stack-traces` flag (enabled by default since Node 12).

### Stack Traces and Source Maps

In production, code is usually minified and bundled. Stack traces from minified code are unreadable:

```
Error: fail
    at a.b (app.min.js:1:45023)
    at c (app.min.js:1:47891)
```

Source maps (`.map` files) map minified positions back to original source code. Error tracking services (Sentry, Datadog, Bugsnag) use source maps to display readable stack traces. Upload source maps to your error tracking service but do NOT serve them publicly in production.

## Structured Error Handling Patterns

### The Result Pattern

Instead of throwing errors, return a result object that explicitly indicates success or failure. This is inspired by Rust's `Result<T, E>` type:

```js
// Result factory
function ok(value) {
  return { success: true, value };
}

function err(error) {
  return { success: false, error };
}

// Usage: a function that returns Result instead of throwing
function parseConfig(raw) {
  try {
    const config = JSON.parse(raw);
    if (!config.apiUrl) return err('Missing apiUrl');
    if (!config.timeout) return err('Missing timeout');
    return ok(config);
  } catch {
    return err('Invalid JSON');
  }
}

// Consumer: no try/catch needed
const result = parseConfig('{"apiUrl": "https://api.example.com", "timeout": 5000}');
if (result.success) {
  initApp(result.value);
} else {
  showConfigError(result.error);
}
```

### Result Pattern with TypeScript (Preview)

The Result pattern truly shines with TypeScript's type narrowing:

```js
// In TypeScript, this would be:
// type Result<T, E = Error> = { success: true; value: T } | { success: false; error: E };
//
// function parseConfig(raw: string): Result<Config, string> { ... }
//
// The compiler then forces you to check result.success before accessing .value or .error
// This makes error handling compile-time verified rather than runtime hoped-for
```

### Safe Wrapper Function

A utility that wraps any throwing function into a Result-returning function:

```js
function trySafe(fn, ...args) {
  try {
    const result = fn(...args);
    return { success: true, value: result };
  } catch (error) {
    return { success: false, error };
  }
}

// Async version
async function trySafeAsync(fn, ...args) {
  try {
    const result = await fn(...args);
    return { success: true, value: result };
  } catch (error) {
    return { success: false, error };
  }
}

// Usage
const { success, value, error } = trySafe(JSON.parse, userInput);
if (!success) {
  console.error('Parse failed:', error.message);
}
```

### Operational vs Programmer Errors

This is a critical distinction for production error handling:

| Category | Description | Response | Examples |
|----------|-------------|----------|----------|
| Operational | Expected failures; part of normal flow | Handle gracefully | Network timeout, file not found, invalid user input, permission denied |
| Programmer | Bugs in the code | Crash and fix | `undefined is not a function`, reading property of `null`, wrong argument types |

```js
class OperationalError extends Error {
  constructor(message, options) {
    super(message, options);
    this.name = 'OperationalError';
    this.isOperational = true;
  }
}

function globalErrorHandler(error) {
  if (error.isOperational) {
    // Operational: log, notify user, continue running
    logger.warn('Operational error:', error.message);
    notifyUser(error.message);
  } else {
    // Programmer error: log, alert on-call, restart
    logger.fatal('PROGRAMMER ERROR:', error);
    alertOnCall(error);
    process.exit(1); // Or in browsers: report and degrade gracefully
  }
}
```

## Error Serialization

### JSON.stringify Does Not Serialize Errors

By default, `JSON.stringify` produces an empty object for Error instances:

```js
const error = new Error('Something failed');
console.log(JSON.stringify(error)); // '{}'
```

This happens because `message`, `name`, and `stack` are non-enumerable properties.

### Custom toJSON Method

```js
class SerializableError extends Error {
  constructor(message, options) {
    super(message, options);
    this.name = this.constructor.name;
  }

  toJSON() {
    return {
      name: this.name,
      message: this.message,
      stack: this.stack,
      ...(this.cause && {
        cause: this.cause instanceof Error
          ? { name: this.cause.name, message: this.cause.message, stack: this.cause.stack }
          : this.cause,
      }),
    };
  }
}

const err = new SerializableError('Outer', {
  cause: new Error('Inner'),
});

console.log(JSON.stringify(err, null, 2));
// {
//   "name": "SerializableError",
//   "message": "Outer",
//   "stack": "SerializableError: Outer\n    at ...",
//   "cause": {
//     "name": "Error",
//     "message": "Inner",
//     "stack": "Error: Inner\n    at ..."
//   }
// }
```

### Generic Error Serializer

A replacer function for `JSON.stringify` that handles any Error:

```js
function errorReplacer(key, value) {
  if (value instanceof Error) {
    return {
      name: value.name,
      message: value.message,
      stack: value.stack,
      ...(value.cause && { cause: value.cause }),
      // Include any custom properties
      ...Object.getOwnPropertyNames(value).reduce((acc, prop) => {
        if (!['name', 'message', 'stack'].includes(prop)) {
          acc[prop] = value[prop];
        }
        return acc;
      }, {}),
    };
  }
  return value;
}

const error = new TypeError('Invalid input');
error.code = 'INVALID_INPUT';
console.log(JSON.stringify(error, errorReplacer, 2));
// {
//   "name": "TypeError",
//   "message": "Invalid input",
//   "stack": "TypeError: Invalid input\n    at ...",
//   "code": "INVALID_INPUT"
// }
```

### structuredClone and Errors

`structuredClone` supports Error objects, but with caveats:

```js
const original = new Error('test error');
original.customProp = 'custom';

const cloned = structuredClone(original);

console.log(cloned.message);    // 'test error'
console.log(cloned.name);       // 'Error'
console.log(cloned.stack);      // Preserved (browser-dependent)
console.log(cloned.customProp); // undefined -- custom properties are NOT cloned
console.log(cloned instanceof Error); // true
```

The structured clone algorithm serializes `name`, `message`, and `stack` for Error objects. However:
- Custom properties added to the error are **not** cloned
- Custom Error subclasses are cloned as plain `Error` objects (the subclass identity is lost)
- The `cause` property is preserved if present (in engines that support it)

```js
class CustomError extends Error {
  constructor(msg) {
    super(msg);
    this.name = 'CustomError';
    this.code = 42;
  }
}

const custom = new CustomError('test');
const cloned = structuredClone(custom);

console.log(cloned instanceof CustomError); // false -- subclass identity lost
console.log(cloned instanceof Error);       // true
console.log(cloned.name);                   // 'CustomError' (name is preserved)
console.log(cloned.code);                   // undefined (custom property lost)
```

### Sending Errors Over postMessage / Web Workers

Errors can be sent via `postMessage` (which uses structured clone internally):

```js
// In a Web Worker
try {
  processData();
} catch (error) {
  // Structured clone handles Error objects
  self.postMessage({ type: 'error', error });
}

// In main thread
worker.onmessage = (event) => {
  if (event.data.type === 'error') {
    const error = event.data.error;
    console.log(error instanceof Error); // true
    console.log(error.message);          // Preserved
    console.log(error.stack);            // Preserved
  }
};
```

## Validation vs Error Handling

A common question: when should you throw and when should you return an error value?

### When to Throw

Throw when the calling code has no reasonable way to continue:

```js
// Throw for programming errors (invalid arguments)
function divide(a, b) {
  if (typeof a !== 'number' || typeof b !== 'number') {
    throw new TypeError('Arguments must be numbers');
  }
  if (b === 0) {
    throw new RangeError('Division by zero');
  }
  return a / b;
}

// Throw for violated preconditions
function processOrder(order) {
  if (!order) throw new Error('Order is required');
  if (!order.items?.length) throw new Error('Order must have items');
  // ...
}

// Throw for unrecoverable states
function getRequiredConfig(key) {
  const value = config[key];
  if (value === undefined) {
    throw new Error(`Required config key "${key}" is missing`);
  }
  return value;
}
```

### When to Return Error Values

Return error values for expected, recoverable conditions:

```js
// Return null/undefined for "not found" scenarios
function findUser(id) {
  const user = users.get(id);
  return user || null; // Not an error; the user just does not exist
}

// Return Result for operations that commonly fail
function parseDate(input) {
  const date = new Date(input);
  if (isNaN(date.getTime())) {
    return { success: false, error: `Invalid date: "${input}"` };
  }
  return { success: true, value: date };
}

// Return validation results as data
function validateEmail(email) {
  const errors = [];
  if (!email) errors.push('Email is required');
  if (!email.includes('@')) errors.push('Email must contain @');
  if (email.length > 254) errors.push('Email too long');
  return { valid: errors.length === 0, errors };
}
```

### Decision Matrix

| Situation | Strategy |
|-----------|----------|
| Wrong argument type passed to a function | Throw `TypeError` |
| User input fails validation | Return error value (or `ValidationError`) |
| File/resource not found | Return `null` or throw depending on whether absence is normal |
| Network request fails | Throw (callers must handle network errors) |
| Business rule violated | Throw custom operational error |
| Bug in the code | Throw (let it crash, fix the bug) |
| Parsing failure (JSON, date, etc.) | Return `null` / Result pattern for user input, throw for programmatic input |

## Performance Implications of try/catch

### Modern Engine Optimization

Historical advice to "avoid try/catch in hot paths" is largely outdated. Modern V8 (TurboFan) and SpiderMonkey can optimize functions containing try/catch. However, some considerations remain:

```js
// The try/catch block itself has negligible overhead when no error is thrown
// The EXPENSIVE part is creating and throwing Error objects

// Benchmark: try/catch with no error (fast)
function withTryCatch(n) {
  let sum = 0;
  for (let i = 0; i < n; i++) {
    try {
      sum += i;
    } catch (e) {
      // never reached
    }
  }
  return sum;
}

// Benchmark: actually throwing errors (slow)
function withThrows(n) {
  let sum = 0;
  for (let i = 0; i < n; i++) {
    try {
      if (i < 0) throw new Error('negative'); // never thrown but engine must account for it
      sum += i;
    } catch (e) {
      sum -= 1;
    }
  }
  return sum;
}
```

### The Cost is in Throwing, Not Catching

```js
// SLOW: Using try/catch for flow control
function parseIntSlow(value) {
  try {
    const result = JSON.parse(value);
    if (typeof result !== 'number') throw new Error();
    return result;
  } catch {
    return NaN;
  }
}

// FAST: Using validation instead of exceptions
function parseIntFast(value) {
  const num = Number(value);
  return Number.isFinite(num) ? Math.trunc(num) : NaN;
}
```

Creating an Error object is expensive because the engine must capture the stack trace at the point of creation. If you are throwing thousands of errors per second (for example, using exceptions for input validation in a tight loop), you will see measurable performance degradation.

### When Performance Matters

```js
// BAD: Throwing for expected conditions in a hot loop
function processRecords(records) {
  let valid = 0;
  for (const record of records) {
    try {
      validate(record); // throws on invalid records
      valid++;
    } catch {
      // 30% of records are invalid = 30% of iterations pay the throw cost
    }
  }
  return valid;
}

// GOOD: Validation without throwing
function processRecords(records) {
  let valid = 0;
  for (const record of records) {
    if (isValid(record)) {
      valid++;
    }
  }
  return valid;
}
```

**Rule: Use exceptions for exceptional situations.** If a condition occurs in more than ~5% of calls, it is not exceptional and should not be handled with throw/catch.

## Common Mistakes

### 1. Swallowing Errors Silently

```js
// BAD: Error is caught and completely ignored
try {
  await saveToDatabase(data);
} catch (e) {
  // Empty catch block. The error disappears.
  // Nobody knows something went wrong.
}

// GOOD: At minimum, log the error
try {
  await saveToDatabase(data);
} catch (e) {
  console.error('Database save failed:', e);
  // Then decide: retry? show user message? re-throw?
}
```

### 2. Catching Too Broadly

```js
// BAD: Catches ALL errors, including bugs
async function handleRequest(req) {
  try {
    const data = processInput(req.body); // Could throw TypeError from a bug
    const result = await callApi(data);  // Could throw NetworkError (operational)
    return format(result);               // Could throw from a bug
  } catch (e) {
    return { error: 'Something went wrong' }; // Hides bugs as user-facing errors
  }
}

// GOOD: Catch only what you expect
async function handleRequest(req) {
  const data = processInput(req.body); // Let bugs crash
  try {
    const result = await callApi(data); // Only catch network-related errors
    return format(result);
  } catch (e) {
    if (e instanceof NetworkError) {
      return { error: 'Service unavailable, try again later' };
    }
    throw e; // Re-throw unexpected errors
  }
}
```

### 3. Losing the Original Error

```js
// BAD: Original error is completely lost
try {
  await fetchData();
} catch (e) {
  throw new Error('Data fetch failed'); // Original error with its stack trace is gone
}

// GOOD: Preserve the original error with cause
try {
  await fetchData();
} catch (e) {
  throw new Error('Data fetch failed', { cause: e });
}
```

### 4. Using Exceptions for Control Flow

```js
// BAD: Using try/catch to check if a key exists
function hasProperty(obj, key) {
  try {
    const val = obj[key];
    if (val === undefined) throw new Error();
    return true;
  } catch {
    return false;
  }
}

// GOOD: Use proper language features
function hasProperty(obj, key) {
  return key in obj;
}
```

### 5. Not Handling Promise Rejections

```js
// BAD: Fire and forget
async function syncData() {
  await fetchAndUpdate(); // If this rejects, it is unhandled
}
syncData(); // No .catch()

// BAD: Missing catch in promise chain
fetch('/api/data')
  .then(r => r.json())
  .then(data => render(data));
  // No .catch() -- if any .then() throws, it is unhandled

// GOOD: Always handle or propagate
syncData().catch(err => {
  console.error('Sync failed:', err);
  scheduleRetry();
});

// GOOD: Add .catch() to every promise chain
fetch('/api/data')
  .then(r => r.json())
  .then(data => render(data))
  .catch(err => showError(err.message));
```

### 6. Logging error Instead of error.message or error.stack

```js
// CONFUSING: Logs the error object, which may not display usefully
console.log('Error:', error); // In some environments, shows [object Error]

// BETTER: Log the message and stack separately
console.error('Error:', error.message);
console.error('Stack:', error.stack);

// BEST: Let console.error handle the Error object (it knows how to display it)
console.error(error); // Most environments display the message + stack
```

### 7. Async Errors in Constructors

Constructors cannot be `async`, so error handling for async initialization is tricky:

```js
// BAD: Cannot use async constructor
class Database {
  constructor(url) {
    // This promise is fire-and-forget -- errors are lost
    this.connection = connect(url); // Returns a promise, never awaited
  }
}

// GOOD: Use a static factory method
class Database {
  constructor(connection) {
    this.connection = connection;
  }

  static async create(url) {
    const connection = await connect(url); // Errors propagate naturally
    return new Database(connection);
  }
}

// Usage
const db = await Database.create('postgres://...');
```

### 8. Not Cleaning Up in finally

```js
// BAD: Cleanup only happens in the success path
async function processFile(path) {
  const handle = await fs.open(path);
  const data = await handle.read(); // If this throws, handle is never closed
  handle.close();
  return data;
}

// GOOD: Use finally for cleanup
async function processFile(path) {
  const handle = await fs.open(path);
  try {
    return await handle.read();
  } finally {
    await handle.close(); // Always runs, even if read() throws
  }
}
```

## Error Handling Patterns for Production

### Centralized Error Handler

```js
class ErrorHandler {
  constructor(options = {}) {
    this.logger = options.logger || console;
    this.onError = options.onError || null;
  }

  handle(error, context = {}) {
    // Normalize the error
    const normalizedError = this.normalize(error);

    // Add context
    normalizedError.context = {
      ...context,
      timestamp: new Date().toISOString(),
      url: typeof window !== 'undefined' ? window.location.href : undefined,
    };

    // Log it
    this.logger.error('[ErrorHandler]', normalizedError.message, normalizedError);

    // Notify callback
    if (this.onError) {
      this.onError(normalizedError);
    }

    return normalizedError;
  }

  normalize(error) {
    // Handle non-Error values that were thrown
    if (!(error instanceof Error)) {
      const normalized = new Error(
        typeof error === 'string' ? error : 'Non-error value thrown'
      );
      normalized.originalValue = error;
      return normalized;
    }
    return error;
  }
}

// Usage
const errorHandler = new ErrorHandler({
  onError: (error) => sendToSentry(error),
});

window.addEventListener('error', (event) => {
  errorHandler.handle(event.error, { source: 'window.onerror' });
});

window.addEventListener('unhandledrejection', (event) => {
  errorHandler.handle(event.reason, { source: 'unhandledrejection' });
});
```

### Retry with Exponential Backoff

```js
async function withRetry(fn, options = {}) {
  const {
    maxRetries = 3,
    baseDelay = 1000,
    maxDelay = 30000,
    shouldRetry = () => true,
  } = options;

  let lastError;

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await fn(attempt);
    } catch (error) {
      lastError = error;

      if (attempt === maxRetries || !shouldRetry(error, attempt)) {
        throw new Error(`Failed after ${attempt + 1} attempts`, { cause: error });
      }

      // Exponential backoff with jitter
      const delay = Math.min(
        baseDelay * Math.pow(2, attempt) + Math.random() * 1000,
        maxDelay
      );

      console.warn(`Attempt ${attempt + 1} failed, retrying in ${delay}ms...`);
      await new Promise(resolve => setTimeout(resolve, delay));
    }
  }
}

// Usage
const data = await withRetry(
  () => fetch('/api/data').then(r => {
    if (!r.ok) throw new Error(`HTTP ${r.status}`);
    return r.json();
  }),
  {
    maxRetries: 3,
    shouldRetry: (error) => {
      // Only retry on network or 5xx errors
      return error.message.includes('HTTP 5') ||
             error.message === 'Failed to fetch';
    },
  }
);
```

### Circuit Breaker Pattern

```js
class CircuitBreaker {
  constructor(options = {}) {
    this.failureThreshold = options.failureThreshold || 5;
    this.resetTimeout = options.resetTimeout || 60000;
    this.failureCount = 0;
    this.state = 'CLOSED'; // CLOSED = normal, OPEN = failing, HALF_OPEN = testing
    this.nextAttempt = 0;
  }

  async execute(fn) {
    if (this.state === 'OPEN') {
      if (Date.now() < this.nextAttempt) {
        throw new Error('Circuit breaker is OPEN: service unavailable');
      }
      this.state = 'HALF_OPEN';
    }

    try {
      const result = await fn();
      this.onSuccess();
      return result;
    } catch (error) {
      this.onFailure();
      throw error;
    }
  }

  onSuccess() {
    this.failureCount = 0;
    this.state = 'CLOSED';
  }

  onFailure() {
    this.failureCount++;
    if (this.failureCount >= this.failureThreshold) {
      this.state = 'OPEN';
      this.nextAttempt = Date.now() + this.resetTimeout;
      console.warn('Circuit breaker OPENED: too many failures');
    }
  }
}

// Usage
const apiBreaker = new CircuitBreaker({ failureThreshold: 3, resetTimeout: 30000 });

async function fetchWithBreaker(url) {
  return apiBreaker.execute(() =>
    fetch(url).then(r => {
      if (!r.ok) throw new Error(`HTTP ${r.status}`);
      return r.json();
    })
  );
}
```

### Safe JSON Parse Utility

```js
function safeJsonParse(text, fallback = null) {
  try {
    return JSON.parse(text);
  } catch {
    return fallback;
  }
}

// With validation
function safeJsonParseWith(text, validator) {
  try {
    const parsed = JSON.parse(text);
    return validator(parsed) ? { success: true, value: parsed } : { success: false, error: 'Validation failed' };
  } catch (e) {
    return { success: false, error: e.message };
  }
}

// Usage
const config = safeJsonParse(rawInput, { defaults: true });
const result = safeJsonParseWith('{"age": 25}', obj => typeof obj.age === 'number');
```

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| `try/catch/finally` | `finally` always runs; never `return` from `finally`; does not catch async errors |
| Built-in errors | 8 types; `TypeError` is most common; use `instanceof` to differentiate |
| Custom errors | Extend `Error`; set `this.name = this.constructor.name`; use `Error.captureStackTrace` in V8 |
| `Error.cause` (ES2022) | Chain errors to preserve context; non-enumerable; not logged by default |
| `throw` statement | Always throw Error objects, never primitives |
| Async error handling | Use `return await` in try blocks; always `.catch()` promises; handle `unhandledrejection` |
| Global handlers | `window.onerror` for sync, `unhandledrejection` for async; `process.on` in Node.js |
| Stack traces | `Error.captureStackTrace` (V8); `stackTraceLimit`; use source maps in production |
| Result pattern | Return `{ success, value/error }` for expected failures; reserve throw for unexpected ones |
| Serialization | `JSON.stringify(error)` gives `{}`; need custom `toJSON` or replacer; `structuredClone` preserves basics |
| Performance | try/catch is cheap when no error is thrown; creating + throwing Error objects is expensive |
| Validation vs throw | Throw for programmer errors; return for expected failures and user input |
| Common mistakes | Empty catch blocks, catching too broadly, losing original error, fire-and-forget async calls |
