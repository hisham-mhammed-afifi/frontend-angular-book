# 5.3 Operators

> JavaScript operators go well beyond arithmetic. Mastering precedence rules, short-circuit behavior, nullish coalescing, optional chaining, bitwise flags, and logical assignment separates developers who write predictable code from those who chase coercion bugs.

---

## Operator Precedence and Associativity

Precedence determines which operator binds its operands first. When two operators share the same precedence level, associativity determines evaluation order.

### Key Precedence Levels (Highest to Lowest)

| Precedence | Operator | Associativity |
|---|---|---|
| 18 | Grouping `()` | n/a |
| 17 | Member access `.`, `[]`, `?.`, `new` (with args) | Left |
| 16 | Function call `()`, `new` (no args) | Left |
| 15 | Postfix `++`, `--` | n/a |
| 14 | Prefix `++`, `--`, unary `+`, `-`, `!`, `~`, `typeof`, `void`, `delete`, `await` | Right |
| 13 | Exponentiation `**` | Right |
| 12 | Multiplicative `*`, `/`, `%` | Left |
| 11 | Additive `+`, `-` | Left |
| 10 | Bitwise shift `<<`, `>>`, `>>>` | Left |
| 9 | Relational `<`, `>`, `<=`, `>=`, `in`, `instanceof` | Left |
| 8 | Equality `==`, `!=`, `===`, `!==` | Left |
| 7 | Bitwise AND `&` | Left |
| 6 | Bitwise XOR `^` | Left |
| 5 | Bitwise OR `\|` | Left |
| 4 | Logical AND `&&` | Left |
| 3 | Logical OR `\|\|`, Nullish `??` | Left |
| 2 | Conditional `?:`, Assignment `=`, `+=`, `??=`, etc. | Right |
| 1 | Comma `,` | Left |

### Precedence Pitfalls

```javascript
// && binds tighter than ||
true || false && false   // true (reads as: true || (false && false))
(true || false) && false // false (explicit grouping changes result)

// typeof binds tighter than +
typeof "hello" + " world" // "string world" (reads as: (typeof "hello") + " world")
typeof ("hello" + " world") // "string"

// Assignment is right-associative
let a, b, c;
a = b = c = 5; // c=5, then b=5, then a=5

// Comma has the lowest precedence
let x = (1 + 2, 3 + 4); // x = 7 (comma returns last value)
let y = 1 + 2, 3 + 4;   // SyntaxError in let context
```

**Rule of thumb:** When in doubt, use parentheses. Explicit grouping costs nothing and prevents bugs.

## Short-Circuit Evaluation (`&&` and `||`)

Both `&&` and `||` return one of their operands, not necessarily a boolean. The right operand is skipped entirely if the left operand determines the result.

### Logical AND (`&&`)

Returns the first falsy operand, or the last operand if all are truthy.

```javascript
"hello" && 42       // 42 (both truthy, returns last)
0 && "hello"        // 0 (first is falsy, returns it)
null && undefined    // null (first is falsy)
1 && 2 && 3         // 3 (all truthy, returns last)
1 && 0 && 3         // 0 (first falsy encountered)
```

### Logical OR (`||`)

Returns the first truthy operand, or the last operand if all are falsy.

```javascript
"hello" || 42       // "hello" (first truthy)
0 || ""             // "" (both falsy, returns last)
null || 0 || "hi"   // "hi" (first truthy)
"" || 0 || false    // false (all falsy, returns last)
```

### Conditional Execution Pattern

```javascript
// Execute only if condition is truthy
isLoggedIn && renderDashboard();

// Provide a default (but beware of falsy values)
const name = user.name || "Anonymous";
```

### Side Effect Gotcha

```javascript
let count = 0;

// count++ returns 0 (the pre-increment value), which is falsy
// So doSomething() is never called
count++ && doSomething();
```

## Nullish Coalescing (`??`)

Returns the right-hand side only when the left is `null` or `undefined`. Unlike `||`, it does NOT trigger on `0`, `""`, `false`, or `NaN`.

```javascript
0 ?? "default"         // 0
"" ?? "default"        // ""
false ?? "default"     // false
NaN ?? "default"       // NaN
null ?? "default"      // "default"
undefined ?? "default" // "default"
```

### `??` vs `||` Comparison

| Left operand | `\|\| "fallback"` | `?? "fallback"` |
|---|---|---|
| `null` | `"fallback"` | `"fallback"` |
| `undefined` | `"fallback"` | `"fallback"` |
| `0` | `"fallback"` | `0` |
| `""` | `"fallback"` | `""` |
| `false` | `"fallback"` | `false` |
| `NaN` | `"fallback"` | `NaN` |

```javascript
// BAD: || treats 0 as "missing"
function getTimeout(ms) {
  return ms || 3000; // Timeout of 0 returns 3000!
}

// GOOD: ?? only falls through on null/undefined
function getTimeout(ms) {
  return ms ?? 3000; // Timeout of 0 returns 0
}
```

### Mixing `??` with `&&`/`||` Requires Parentheses

```javascript
// SyntaxError: cannot mix ?? with && or || without parentheses
// null || undefined ?? "foo"

// Must be explicit
(null || undefined) ?? "foo"  // "foo"
null || (undefined ?? "foo")  // "foo"
```

This restriction was intentional to prevent ambiguous precedence interpretations.

## Optional Chaining (`?.`)

Short-circuits to `undefined` when the value before `?.` is `null` or `undefined`. Three forms exist.

### Property Access

```javascript
const user = { profile: { address: { city: "Berlin" } } };

// Without optional chaining
const city = user && user.profile && user.profile.address && user.profile.address.city;

// With optional chaining
const city2 = user?.profile?.address?.city; // "Berlin"

// Missing intermediate
const missing = user?.settings?.theme; // undefined (no error)
```

### Dynamic Property Access

```javascript
const key = "name";
const value = user?.[key]; // user?.["name"]

const index = 0;
const first = arr?.[index];
```

### Method Calls

```javascript
const emitter = getEmitter();
emitter?.emit?.("event", data); // Safe even if emit does not exist
```

### Important Differences from `&&` Chaining

```javascript
// && short-circuits on ANY falsy value
const a = "" && "".length; // "" (short-circuits because "" is falsy)

// ?. only short-circuits on null/undefined
const b = ""?.length;      // 0 (does NOT short-circuit, "" is not nullish)

// This means converting && chains to ?. can change behavior
const c = 0 && 0.toString();  // 0 (short-circuits)
const d = (0)?.toString();    // "0" (does NOT short-circuit)
```

### Restrictions

```javascript
// Cannot assign through optional chaining
// obj?.prop = value; // SyntaxError

// Cannot use with new
// new Foo?.();       // SyntaxError

// Cannot use with tagged templates
// tag?.()``string``  // SyntaxError

// ?. on a non-function that exists still throws
const obj = { notAFn: 42 };
// obj.notAFn?.(); // TypeError: obj.notAFn is not a function
// (notAFn exists and is not null/undefined, so ?. does not short-circuit,
// then () tries to call 42 as a function)
```

### Works with `delete`

```javascript
delete user?.temporaryData; // Safe, no error if user is null
```

## Logical Assignment Operators (`??=`, `||=`, `&&=`)

Added in ES2021. These short-circuit the assignment itself: if the condition is not met, no assignment occurs (and no setter is triggered).

### `??=` (Nullish Assignment)

Assigns only if the left is `null` or `undefined`.

```javascript
let a = null;
a ??= 42;  // a is now 42

let b = 0;
b ??= 42;  // b is still 0 (0 is not nullish)

// Practical: lazy initialization
const cache = {};
cache.users ??= fetchUsers(); // Only fetches if cache.users is null/undefined
```

### `||=` (Logical OR Assignment)

Assigns only if the left is falsy.

```javascript
let title = "";
title ||= "Untitled"; // title is now "Untitled"

let count = 0;
count ||= 10; // count is now 10 (careful: 0 is falsy!)
```

### `&&=` (Logical AND Assignment)

Assigns only if the left is truthy.

```javascript
let user = { name: "  Alice  " };
user.name &&= user.name.trim(); // "Alice" (only trims if name is truthy)

let empty = "";
empty &&= empty.trim(); // "" (stays empty, assignment skipped)
```

### The Setter Difference

This is the key distinction from `x = x || y`:

```javascript
const obj = {
  _value: "original",
  get value() { return this._value; },
  set value(v) { console.log("Setter called!"); this._value = v; }
};

// This ALWAYS triggers the setter, even when the value does not change
obj.value = obj.value || "default"; // "Setter called!"

// This does NOT trigger the setter because the value is truthy
obj.value ||= "default"; // (no output, setter not called)
```

## Comparison Operators

### Relational Operators (`<`, `>`, `<=`, `>=`)

These always invoke ToNumber (or ToPrimitive) on both operands, except when both are strings (then lexicographic comparison is used).

```javascript
"10" > "9"    // false (lexicographic: "1" < "9")
"10" > 9      // true (string "10" coerced to number 10)
null >= 0     // true (null -> 0, 0 >= 0)
null == 0     // false (== has special null rules)
null > 0      // false (null -> 0, 0 > 0 is false)
```

The `null >= 0` but `null != 0` paradox exists because `>=` uses ToNumber (where `Number(null)` is `0`), but `==` has a special rule: null only equals null or undefined.

### `in` Operator

Checks if a property exists on an object or its prototype chain.

```javascript
"name" in { name: "Alice" }          // true
"toString" in {}                      // true (inherited from Object.prototype)
0 in ["a", "b", "c"]                 // true (index 0 exists)
3 in ["a", "b", "c"]                 // false (index 3 does not exist)
"length" in []                        // true (arrays have own length property)
```

### `in` vs `hasOwnProperty` vs `Object.hasOwn`

```javascript
const parent = { inherited: true };
const child = Object.create(parent);
child.own = true;

"inherited" in child                  // true (found in prototype)
child.hasOwnProperty("inherited")     // false (not own property)
Object.hasOwn(child, "inherited")     // false (not own property)

"own" in child                        // true
child.hasOwnProperty("own")           // true
Object.hasOwn(child, "own")           // true
```

**Prefer `Object.hasOwn()` (ES2022).** It handles edge cases that `hasOwnProperty` cannot:

```javascript
// Object.create(null) has no prototype, so no hasOwnProperty method
const bare = Object.create(null);
bare.key = "value";

// bare.hasOwnProperty("key"); // TypeError: bare.hasOwnProperty is not a function
Object.hasOwn(bare, "key");    // true (always works)

// hasOwnProperty can be overridden
const tricky = { hasOwnProperty: () => false };
tricky.hasOwnProperty("hasOwnProperty"); // false (wrong!)
Object.hasOwn(tricky, "hasOwnProperty"); // true (correct)
```

### `instanceof` Operator

Checks if an object's prototype chain contains a constructor's `prototype` property.

```javascript
[] instanceof Array    // true
[] instanceof Object   // true (Array extends Object)
"hi" instanceof String // false (primitives are not instances)

// Fails across realms (iframes)
// iframe.contentWindow.Array !== window.Array
// So an array from an iframe fails instanceof Array

// Use Array.isArray() instead for arrays
Array.isArray([])      // true (works across realms)
```

`instanceof` can be customized:

```javascript
class Even {
  static [Symbol.hasInstance](num) {
    return typeof num === "number" && num % 2 === 0;
  }
}

4 instanceof Even  // true
3 instanceof Even  // false
```

## Bitwise Operators

JavaScript converts numbers to 32-bit signed integers before bitwise operations. This means they only work correctly for integers in the range roughly -2.1 billion to 2.1 billion.

### Permission Flags (Bitmask Pattern)

This is the most practical use of bitwise operators in application code:

```javascript
const READ    = 0b0001; // 1
const WRITE   = 0b0010; // 2
const EXECUTE = 0b0100; // 4
const ADMIN   = 0b1000; // 8

// Combine permissions with OR
let userPerms = READ | WRITE; // 0b0011 = 3

// Check permission with AND
const canRead = (userPerms & READ) !== 0;     // true
const canExecute = (userPerms & EXECUTE) !== 0; // false

// Add permission with OR
userPerms |= EXECUTE; // 0b0111 = 7

// Remove permission with AND + NOT
userPerms &= ~WRITE; // 0b0101 = 5

// Toggle permission with XOR
userPerms ^= ADMIN; // 0b1101 = 13 (added)
userPerms ^= ADMIN; // 0b0101 = 5 (removed)

// Check multiple permissions at once
const required = READ | EXECUTE;
const hasAll = (userPerms & required) === required; // true
```

### Other Practical Uses

```javascript
// Even/odd check
function isEven(n) {
  return (n & 1) === 0;
}

// Truncate float to integer (toward zero)
3.7 | 0    // 3
-3.7 | 0   // -3 (NOT the same as Math.floor(-3.7) which is -4)
~~3.7      // 3
~~-3.7     // -3

// RGB to hex
function rgbToHex(r, g, b) {
  return "#" + ((r << 16) | (g << 8) | b).toString(16).padStart(6, "0");
}
rgbToHex(255, 128, 0) // "#ff8000"

// Swap without temp variable
let a = 5, b = 10;
a ^= b; b ^= a; a ^= b;
// a = 10, b = 5
```

### Bitwise NOT (`~`) and `indexOf`

```javascript
// ~x equals -(x + 1)
~0   // -1
~-1  // 0 (falsy!)
~1   // -2

// Historical pattern: ~indexOf as a boolean check
// ~(-1) is 0 (falsy), ~(anything else) is truthy
const str = "hello world";
if (~str.indexOf("world")) {
  // found
}

// Modern replacement: use .includes()
if (str.includes("world")) {
  // found (clearer intent)
}
```

## The Exponentiation Operator (`**`)

Right-associative. Equivalent to `Math.pow()` for Number operands, but also works with BigInt.

```javascript
2 ** 10      // 1024
2 ** 3 ** 2  // 512 (right-associative: 2 ** (3 ** 2) = 2 ** 9)
(2 ** 3) ** 2 // 64

// Works with BigInt
2n ** 100n   // 1267650600228229401496703205376n
// 2n ** -1n // RangeError (would produce a fraction)
```

### Unary Operator Restriction

```javascript
// -2 ** 2 // SyntaxError: Unary operator used immediately before exponentiation expression

-(2 ** 2)  // -4 (negate the result)
(-2) ** 2  // 4 (square negative two)
```

### Special Values

```javascript
NaN ** 0      // 1 (the only case where NaN does not propagate)
0 ** 0        // 1
Infinity ** 0 // 1
0 ** -1       // Infinity
(-1) ** 0.5   // NaN (square root of -1)
```

## The Comma Operator

Evaluates each operand left to right, returns the value of the last operand. Has the lowest precedence of all operators.

```javascript
const result = (1 + 2, 3 + 4, 5 + 6); // 11 (last expression)

// In for loops (most common legitimate use)
for (let i = 0, j = 10; i < j; i++, j--) {
  console.log(i, j); // 0 10, 1 9, 2 8, 3 7, 4 6
}

// One-liner arrow functions with side effects
const tap = (x) => (console.log(x), x); // Logs x, then returns x
const doubled = [1, 2, 3].map(tap).map(n => n * 2);
```

### Precedence Trap

```javascript
let x;

x = 1 + 2, 3 + 4;
console.log(x); // 3 (assignment binds tighter than comma)

x = (1 + 2, 3 + 4);
console.log(x); // 7 (parentheses force comma to complete first)
```

Note: Commas in array literals (`[1, 2, 3]`), object literals, and function arguments are separators, not the comma operator.

## The `void` Operator

Evaluates an expression and returns `undefined`.

```javascript
void 0          // undefined
void "hello"    // undefined
void (1 + 2)    // undefined
```

### Preventing Arrow Function Return Leakage

When an arrow function should not return a value, `void` ensures the return is `undefined`:

```javascript
// BAD: forEach callback accidentally returns a value
// (usually harmless, but can cause issues with certain APIs)
items.forEach(item => cache.delete(item)); // Returns boolean from delete

// GOOD: Explicitly discard the return value
items.forEach(item => void cache.delete(item));
```

### IIFE Without Parentheses

```javascript
void function () {
  console.log("IIFE");
}();

// Equivalent to:
(function () {
  console.log("IIFE");
})();
```

### Minification

Minifiers replace `undefined` (9 characters) with `void 0` (6 characters).

## The `delete` Operator

Removes a property from an object. Returns `true` if the deletion succeeds or the property does not exist.

```javascript
const user = { name: "Alice", age: 30 };
delete user.age;     // true
console.log(user);   // { name: "Alice" }
delete user.missing; // true (property did not exist, still "successful")
```

### Gotchas

```javascript
// delete on array creates a hole (does NOT change length)
const arr = [1, 2, 3];
delete arr[1];
console.log(arr);        // [1, empty, 3]
console.log(arr.length); // 3 (unchanged!)
console.log(arr[1]);     // undefined

// Use splice to remove and shift elements
arr.splice(1, 1);       // Removes element at index 1
console.log(arr);        // [1, 3]

// Non-configurable properties cannot be deleted
const obj = {};
Object.defineProperty(obj, "locked", { value: 42, configurable: false });
delete obj.locked;  // false in sloppy mode, TypeError in strict mode

// var/let/const declarations cannot be deleted
var x = 10;
delete x; // false (SyntaxError in strict mode)
```

## Tagged Template Literals

A tagged template is a function call with a special argument structure. The tag function receives string segments and interpolated values separately.

```javascript
function highlight(strings, ...values) {
  // strings: ["Name: ", ", Age: ", ""]
  // values: ["Alice", 30]
  return strings.reduce((result, str, i) => {
    return result + str + (values[i] !== undefined ? `<b>${values[i]}</b>` : "");
  }, "");
}

const name = "Alice";
const age = 30;
highlight`Name: ${name}, Age: ${age}`;
// "Name: <b>Alice</b>, Age: <b>30</b>"
```

### `String.raw`

Returns the raw string without processing escape sequences:

```javascript
String.raw`Hello\nWorld`  // "Hello\\nWorld" (literal backslash-n, not a newline)
String.raw`C:\Users\name` // "C:\\Users\\name" (no need to double-escape)

// Useful for regex patterns
const pattern = new RegExp(String.raw`^\d{3}\.\d{3}$`);
// Without String.raw you would need: new RegExp("^\\d{3}\\.\\d{3}$")
```

### The `.raw` Property

The `strings` parameter has a `.raw` property with unprocessed escape sequences:

```javascript
function showRaw(strings) {
  console.log(strings[0]);     // Actual newline character
  console.log(strings.raw[0]); // The two characters: \ and n
}

showRaw`hello\nworld`;
```

### Tag Functions Can Return Any Type

```javascript
function sql(strings, ...values) {
  return {
    text: strings.join("$"),     // Parameterized query
    params: values               // Separate parameters (safe from SQL injection)
  };
}

const id = 42;
const query = sql`SELECT * FROM users WHERE id = ${id}`;
// { text: "SELECT * FROM users WHERE id = $", params: [42] }
```

## Destructuring Assignment

Destructuring is an expression, not just a declaration syntax. It has several advanced patterns worth knowing.

### Default Values Only Trigger on `undefined`

```javascript
const { x = 5 } = { x: undefined }; // x = 5 (default triggered)
const { y = 5 } = { y: null };      // y = null (NOT 5, null is not undefined)
const { z = 5 } = { z: 0 };         // z = 0 (NOT 5)
const { w = 5 } = { w: "" };        // w = "" (NOT 5)
```

### Renaming with Defaults

```javascript
const { name: userName = "Anonymous", role: userRole = "guest" } = apiResponse;
```

### Nested Destructuring

```javascript
const response = {
  data: {
    users: [
      { id: 1, profile: { avatar: "pic.jpg" } }
    ]
  }
};

const { data: { users: [{ profile: { avatar } }] } } = response;
console.log(avatar); // "pic.jpg"

// Danger: missing intermediate throws TypeError
// const { data: { missing: { value } } } = response; // TypeError
```

### Computed Property Names

```javascript
const key = "name";
const { [key]: value } = { name: "Alice" };
console.log(value); // "Alice"
```

### Assignment Without Declaration

Requires parentheses to avoid parsing `{` as a block statement:

```javascript
let a, b;
// { a, b } = { a: 1, b: 2 }; // SyntaxError: { starts a block
({ a, b } = { a: 1, b: 2 });   // Works
```

### Array Destructuring Uses the Iterator Protocol

```javascript
// Swap variables
let x = 1, y = 2;
[x, y] = [y, x]; // x=2, y=1

// Skip elements
const [, , third] = [1, 2, 3]; // third = 3

// Rest element
const [first, ...rest] = [1, 2, 3, 4]; // first=1, rest=[2,3,4]

// Works with any iterable
const [a, b, c] = "abc"; // a="a", b="b", c="c"
const [d, e] = new Set([10, 20, 30]); // d=10, e=20
```

## Common Mistakes

### 1. Relying on Operator Precedence Instead of Parentheses

```javascript
// BAD: Requires knowing that && binds tighter than ||
if (isAdmin || isEditor && hasPermission) { /* ... */ }

// GOOD: Explicit intent
if (isAdmin || (isEditor && hasPermission)) { /* ... */ }
```

### 2. Using `||` for Defaults When `0` or `""` Are Valid

```javascript
// BAD: port 0 becomes 3000
const port = config.port || 3000;

// GOOD
const port = config.port ?? 3000;
```

### 3. Assuming `&&`/`||` Return Booleans

```javascript
// BAD: Storing the result expecting true/false
const isValid = user && user.verified;
// isValid could be null, undefined, the user object, or the boolean

// GOOD: Explicitly convert if you need a boolean
const isValid = Boolean(user?.verified);
// or
const isValid = !!user?.verified;
```

### 4. Confusing `?.` with `&&` When Migrating

```javascript
const val1 = 0 && 0 .toFixed(2);  // 0 (short-circuits on falsy 0)
const val2 = (0)?.toFixed(2);     // "0.00" (does NOT short-circuit, 0 is not nullish)
```

### 5. Using `delete` on Arrays

```javascript
// BAD: Creates a hole
const arr = [1, 2, 3];
delete arr[1]; // [1, empty, 3], length still 3

// GOOD: Use splice
arr.splice(1, 1); // [1, 3], length 2
```

### 6. Assuming `in` Checks Only Own Properties

```javascript
// BAD: in checks the entire prototype chain
if ("toString" in myObj) {
  // Always true for any object!
}

// GOOD: Check own properties only
if (Object.hasOwn(myObj, "toString")) {
  // Only true if myObj directly defines toString
}
```

### 7. Mixing `??` with `&&`/`||` Without Parentheses

```javascript
// SyntaxError
// const result = a || b ?? c;

// Must be explicit
const result = (a || b) ?? c;
```

### 8. Using `~~` as `Math.floor()`

```javascript
~~3.7   // 3 (same as Math.floor)
~~-3.7  // -3 (DIFFERENT from Math.floor(-3.7) which is -4)
// ~~ truncates toward zero; Math.floor rounds toward negative infinity
```

## Best Practices

1. **Use parentheses for complex expressions.** Do not rely on memorized precedence tables. `(a && b) || c` is clearer than `a && b || c`.

2. **Use `??` over `||` for defaults** when `0`, `""`, or `false` are valid values.

3. **Use `?.` for safe deep access** but understand it differs from `&&` chaining for falsy non-nullish values.

4. **Use logical assignment operators** (`??=`, `||=`, `&&=`) for initialization and conditional updates. They avoid unnecessary setter triggers.

5. **Use `Object.hasOwn()` over `in`** when checking own properties. Use `in` only when you intentionally want to check the prototype chain.

6. **Use `Array.isArray()` instead of `instanceof Array`.** It works across iframes and realms.

7. **Use `Number.isNaN()` and `Number.isFinite()`** instead of their global counterparts which coerce.

8. **Avoid bitwise tricks for readability-sensitive code.** Use `Math.trunc()` over `~~`, and `.includes()` over `~indexOf`. Reserve bitwise operations for actual flag/mask patterns.

## Real-World Use Cases

### Feature Flag System with Bitwise Permissions

```javascript
const Features = {
  DARK_MODE:    1 << 0, // 1
  NOTIFICATIONS: 1 << 1, // 2
  ANALYTICS:    1 << 2, // 4
  BETA_ACCESS:  1 << 3, // 8
  ADMIN_PANEL:  1 << 4, // 16
};

function createUser(name, features = 0) {
  return { name, features };
}

function hasFeature(user, feature) {
  return (user.features & feature) !== 0;
}

function enableFeature(user, feature) {
  user.features |= feature;
}

function disableFeature(user, feature) {
  user.features &= ~feature;
}

const user = createUser("Alice", Features.DARK_MODE | Features.NOTIFICATIONS);
hasFeature(user, Features.DARK_MODE);    // true
hasFeature(user, Features.BETA_ACCESS);  // false

enableFeature(user, Features.BETA_ACCESS);
hasFeature(user, Features.BETA_ACCESS);  // true

disableFeature(user, Features.DARK_MODE);
hasFeature(user, Features.DARK_MODE);    // false
```

### Safe Configuration Merging

```javascript
function mergeConfig(defaults, overrides) {
  return {
    host: overrides.host ?? defaults.host,
    port: overrides.port ?? defaults.port,
    debug: overrides.debug ?? defaults.debug,
    timeout: overrides.timeout ?? defaults.timeout,
    retries: overrides.retries ?? defaults.retries,
    basePath: overrides.basePath ?? defaults.basePath,
  };
}

const defaults = { host: "localhost", port: 3000, debug: false, timeout: 5000, retries: 3, basePath: "" };

// port: 0 and basePath: "" are preserved (unlike || which would use defaults)
const config = mergeConfig(defaults, { port: 0, debug: true, basePath: "" });
// { host: "localhost", port: 0, debug: true, timeout: 5000, retries: 3, basePath: "" }
```

### Lazy Cache with Logical Assignment

```javascript
class ApiClient {
  #cache = {};

  async getUser(id) {
    // Only fetches if cache entry is null/undefined
    return this.#cache[id] ??= await this.#fetchUser(id);
  }

  async #fetchUser(id) {
    const response = await fetch(`/api/users/${id}`);
    return response.json();
  }
}

const client = new ApiClient();
await client.getUser(1); // Fetches from API
await client.getUser(1); // Returns cached result (no fetch)
```

---

> **Key Takeaway:** Use `??` instead of `||` for defaults, `?.` for safe property access, parentheses for complex expressions, `Object.hasOwn()` for own-property checks, and logical assignment operators for conditional initialization. Know operator precedence exists, but write code that does not require your reader to memorize it.
