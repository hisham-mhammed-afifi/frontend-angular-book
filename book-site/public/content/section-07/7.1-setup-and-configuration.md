# 7.1 Setup and Configuration

> TypeScript adds static types to JavaScript. Getting the compiler and tsconfig.json right from the start prevents hours of debugging configuration issues later.

---

## Installing TypeScript

### Global Installation

```bash
npm install -g typescript

# Verify
tsc --version
# Version 5.8.x (or newer)
```

Global installation makes `tsc` available everywhere but pins your system to a single version. Not recommended for team projects where everyone needs the same version.

### Per-Project Installation (Recommended)

```bash
mkdir my-project && cd my-project
npm init -y
npm install --save-dev typescript

# Run with npx (uses the local version)
npx tsc --version
```

This ensures every developer on the team uses the exact same TypeScript version, locked by `package-lock.json`.

### Generate tsconfig.json

```bash
npx tsc --init
```

This creates a `tsconfig.json` with defaults and comments explaining each option. TypeScript 5.9+ generates a much simpler file with only the essential options.

## Running TypeScript

TypeScript files (`.ts`) cannot run directly in browsers or Node.js (with some exceptions). You need a compilation step.

### Option 1: tsc (TypeScript Compiler)

The official compiler. It type-checks AND transpiles.

```bash
# Compile a single file
npx tsc index.ts
# Produces index.js

# Compile entire project (uses tsconfig.json)
npx tsc

# Watch mode: recompile on file changes
npx tsc --watch
```

### Option 2: tsx (Recommended for Development)

`tsx` runs TypeScript files directly without a separate compile step. It uses esbuild under the hood, so it is fast. It does NOT type-check (use `tsc --noEmit` for that).

```bash
npm install --save-dev tsx

# Run a file directly
npx tsx src/index.ts

# Watch mode
npx tsx watch src/index.ts
```

### Option 3: ts-node

An older alternative to `tsx`. Slower and requires more configuration. Still used in some existing projects.

```bash
npm install --save-dev ts-node

npx ts-node src/index.ts
```

### Option 4: Node.js Native Type Stripping (Node 22.6+)

Node.js 22.6+ can run TypeScript by stripping type annotations at runtime. No compilation, no extra tools.

```bash
# Node.js 22.6 - 23.5 (explicit flag required)
node --experimental-strip-types src/index.ts

# Node.js 23.6+ (enabled by default)
node src/index.ts
```

**Important:** This only strips types. It does not perform type checking. It does not support TypeScript features that require code transformation (enums, namespaces, parameter properties, legacy decorators). Use it for quick scripts and development. For production, compile with `tsc` or a bundler.

### Running Strategy

| Context | Tool | Why |
|---------|------|-----|
| Development (run scripts) | `tsx` or `node --experimental-strip-types` | Fast, no build step |
| Type checking (CI, pre-commit) | `tsc --noEmit` | Full type checking, no output files |
| Production build | `tsc` or bundler (esbuild, Vite, webpack) | Optimized output |
| Library publishing | `tsc` with `declaration: true` | Generates `.d.ts` type definitions |

## tsconfig.json Structure

The `tsconfig.json` file controls how TypeScript behaves. It sits at the project root.

```jsonc
{
  "compilerOptions": {
    // How TypeScript compiles your code
  },
  "include": [
    // Which files to compile
  ],
  "exclude": [
    // Which files to skip
  ]
}
```

### File Inclusion

```jsonc
{
  "include": ["src/**/*"],         // All files under src/
  "exclude": ["node_modules", "dist", "**/*.spec.ts"]
}
```

- `include` uses glob patterns. `**/*` matches all files recursively.
- `exclude` defaults to `["node_modules", "bower_components", "jspm_packages"]` plus `outDir` if set.
- Files imported by included files are always compiled, even if they are in an excluded directory.

## Compiler Options: The Essential Ones

### target

What JavaScript version to compile down to.

```jsonc
{
  "compilerOptions": {
    "target": "ES2022"
  }
}
```

| Value | When to Use |
|-------|-------------|
| `ES5` | Must support IE11 (rare today) |
| `ES2015` / `ES6` | Older browser support |
| `ES2020` | Safe for all modern browsers |
| `ES2022` | Recommended for Node 18+ and modern browsers |
| `ESNext` | Latest features; changes with each TS release |

`ES2022` is the safest modern default. It includes `class` fields, `top-level await`, `Array.at()`, and `Object.hasOwn()`. Prefer a fixed target over `ESNext` for reproducible builds.

**What target does:**
```typescript
// Source (TypeScript)
const greeting = `Hello, ${name}!`;

// target: "ES5" output
var greeting = "Hello, " + name + "!";

// target: "ES2022" output (unchanged)
const greeting = `Hello, ${name}!`;
```

### module

What module system the output uses.

```jsonc
{
  "compilerOptions": {
    "module": "NodeNext"
  }
}
```

| Value | When to Use |
|-------|-------------|
| `commonjs` | Legacy Node.js (`require`/`module.exports`) |
| `NodeNext` | Modern Node.js (supports both ESM and CJS based on `package.json` `"type"`) |
| `ESNext` | Bundled apps (Vite, webpack, esbuild) |
| `preserve` | Let the bundler handle modules (TS 5.4+) |

For Node.js projects, `NodeNext` is the correct choice. It respects `package.json` `"type": "module"` and enforces `.js` extensions in imports.

For bundled frontend apps, `ESNext` or `preserve` both work. `preserve` is newer and tells TypeScript to leave import/export statements exactly as written.

### moduleResolution

How TypeScript finds modules when you write `import`.

```jsonc
{
  "compilerOptions": {
    "moduleResolution": "NodeNext"
  }
}
```

| Value | When to Use |
|-------|-------------|
| `node10` (formerly `node`) | Legacy. Does not understand `exports` in package.json |
| `NodeNext` | Modern Node.js. Respects `exports` maps, `.js` extensions required |
| `Bundler` | For bundled apps (Vite, webpack). No `.js` extension required |

**The `module` and `moduleResolution` pairing matters:**

| module | moduleResolution | Use Case |
|--------|-----------------|----------|
| `NodeNext` | `NodeNext` | Node.js apps and libraries |
| `ESNext` | `Bundler` | Frontend apps with a bundler |
| `preserve` | `Bundler` | Frontend apps (TS 5.4+) |

### lib

Which built-in type declarations to include.

```jsonc
{
  "compilerOptions": {
    "target": "ES2022",
    "lib": ["ES2022", "DOM", "DOM.Iterable"]
  }
}
```

- `target` automatically includes matching `lib` entries for JavaScript features.
- Add `DOM` and `DOM.Iterable` for browser code (gives you types for `document`, `window`, `fetch`, etc.).
- Omit `DOM` for pure Node.js code (use `@types/node` instead).

### outDir and rootDir

```jsonc
{
  "compilerOptions": {
    "rootDir": "src",     // Where your source files live
    "outDir": "dist"      // Where compiled JS goes
  }
}
```

The directory structure under `rootDir` is mirrored in `outDir`:
```
src/
  index.ts
  utils/helper.ts

dist/        (after tsc)
  index.js
  utils/helper.js
```

## Strict Mode

The `strict` flag is a shorthand that enables a group of strict type-checking options.

```jsonc
{
  "compilerOptions": {
    "strict": true
  }
}
```

This is equivalent to enabling all of these individually:

| Flag | What It Does |
|------|-------------|
| `noImplicitAny` | Error when TypeScript cannot infer a type and defaults to `any` |
| `strictNullChecks` | `null` and `undefined` are not assignable to other types |
| `strictFunctionTypes` | Enables contravariant function parameter checking |
| `strictBindCallApply` | Checks `bind`, `call`, `apply` argument types |
| `strictPropertyInitialization` | Class properties must be initialized or declared in constructor |
| `noImplicitThis` | Error when `this` has an implicit `any` type |
| `useUnknownInCatchVariables` | `catch(e)` types `e` as `unknown` instead of `any` |
| `alwaysStrict` | Emits `"use strict"` in output files |

**Always enable `strict: true` for new projects.** The most impactful flags are `noImplicitAny` and `strictNullChecks`. Without them, TypeScript catches far fewer bugs.

```typescript
// Without strict:
function greet(name) {         // name is implicitly `any` - no error
  return name.toUpperCase();   // runtime crash if name is a number
}

greet(42); // No compile error, crashes at runtime

// With strict:
function greet(name) {         // Error: Parameter 'name' implicitly has an 'any' type
  return name.toUpperCase();
}

function greet(name: string) { // Fixed: explicit type required
  return name.toUpperCase();
}

greet(42); // Error: Argument of type 'number' is not assignable to parameter of type 'string'
```

### Additional Strict-Like Flags

These are not included in `strict` but are worth enabling:

```jsonc
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "noFallthroughCasesInSwitch": true,
    "forceConsistentCasingInFileNames": true
  }
}
```

- `noUncheckedIndexedAccess`: Accessing an array element or object with index signature includes `undefined` in the type.

```typescript
const scores: number[] = [90, 85, 92];

// Without noUncheckedIndexedAccess:
const first = scores[0]; // type: number

// With noUncheckedIndexedAccess:
const first = scores[0]; // type: number | undefined
// Forces you to handle the undefined case
if (first !== undefined) {
  console.log(first.toFixed(2)); // safe
}
```

- `exactOptionalPropertyTypes`: Distinguishes between "property missing" and "property set to undefined".

```typescript
interface Config {
  debug?: boolean;
}

// Without exactOptionalPropertyTypes:
const config: Config = { debug: undefined }; // OK

// With exactOptionalPropertyTypes:
const config: Config = { debug: undefined }; // Error!
// Must be either { debug: true }, { debug: false }, or {}
```

## Recommended Configurations

### Node.js Application

```jsonc
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "outDir": "dist",
    "rootDir": "src",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "forceConsistentCasingInFileNames": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist"]
}
```

With `package.json`:
```json
{
  "type": "module",
  "scripts": {
    "build": "tsc",
    "dev": "tsx watch src/index.ts",
    "typecheck": "tsc --noEmit",
    "start": "node dist/index.js"
  }
}
```

### Frontend Application (Bundled with Vite/webpack)

```jsonc
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "lib": ["ES2022", "DOM", "DOM.Iterable"],
    "outDir": "dist",
    "rootDir": "src",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "forceConsistentCasingInFileNames": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "jsx": "react-jsx",
    "isolatedModules": true,
    "noEmit": true,
    "verbatimModuleSyntax": true
  },
  "include": ["src"]
}
```

Key differences from Node.js:
- `moduleResolution: "Bundler"` because the bundler resolves modules, not Node.
- `noEmit: true` because the bundler produces the output, not tsc.
- `isolatedModules: true` because bundlers compile files individually (no cross-file type features like `const enum` across files).
- `jsx` for JSX/TSX support.

### Library

```jsonc
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "outDir": "dist",
    "rootDir": "src",
    "strict": true,
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true,
    "esModuleInterop": true,
    "forceConsistentCasingInFileNames": true,
    "skipLibCheck": true
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist", "**/*.test.ts"]
}
```

- `declaration: true` generates `.d.ts` files so consumers get type information.
- `declarationMap: true` enables "Go to Definition" to jump to `.ts` source instead of `.d.ts`.
- Lower `target` for broader compatibility.

## Extending Configurations

Use `extends` to share a base configuration across projects or within a monorepo.

```jsonc
// tsconfig.base.json
{
  "compilerOptions": {
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  }
}

// tsconfig.json
{
  "extends": "./tsconfig.base.json",
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "outDir": "dist",
    "rootDir": "src"
  },
  "include": ["src"]
}
```

The community-maintained `@tsconfig/bases` package provides ready-made base configs:

```bash
npm install --save-dev @tsconfig/node22
```

```jsonc
{
  "extends": "@tsconfig/node22/tsconfig.json",
  "compilerOptions": {
    "outDir": "dist",
    "rootDir": "src"
  },
  "include": ["src"]
}
```

## Path Aliases

Map import paths to directories to avoid deep relative imports.

```jsonc
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@utils/*": ["src/utils/*"],
      "@models/*": ["src/models/*"]
    }
  }
}
```

```typescript
// Instead of:
import { formatDate } from "../../../utils/format";

// Write:
import { formatDate } from "@utils/format";
```

**Important:** Path aliases are resolved by TypeScript only at type-check time. The compiled JavaScript still contains the alias paths. You need a runtime resolver (like `tsconfig-paths` for Node.js, or bundler config for frontend) to make them work at runtime.

```bash
# For Node.js runtime
npm install --save-dev tsconfig-paths
node -r tsconfig-paths/register dist/index.js
```

For bundlers like Vite, configure `resolve.alias` in `vite.config.ts` to match.

## Project References

For large projects and monorepos, project references allow incremental builds across packages.

```jsonc
// packages/shared/tsconfig.json
{
  "compilerOptions": {
    "composite": true,      // Required for project references
    "declaration": true,
    "outDir": "dist",
    "rootDir": "src"
  },
  "include": ["src"]
}

// packages/app/tsconfig.json
{
  "compilerOptions": {
    "outDir": "dist",
    "rootDir": "src"
  },
  "references": [
    { "path": "../shared" }
  ],
  "include": ["src"]
}
```

Build with:
```bash
npx tsc --build          # Builds referenced projects in dependency order
npx tsc --build --watch  # Incremental watch mode
```

`composite: true` enables:
- Incremental builds (only recompile what changed).
- Dependency graph between packages.
- Faster builds in monorepos.

## Common Mistakes

- **Not enabling `strict: true`.** Without strict mode, TypeScript allows `any` types to silently propagate, null/undefined to bypass checks, and many bugs that types should catch. Always start with `strict: true`. Loosening individual flags later is preferable to enabling them one by one.

- **Mismatching `module` and `moduleResolution`.** Using `module: "ESNext"` with `moduleResolution: "NodeNext"` (or vice versa) causes confusing resolution errors. Pair `NodeNext`/`NodeNext` for Node.js and `ESNext`/`Bundler` for bundled apps.

- **Using `target: "ESNext"`.** `ESNext` is a moving target that changes with every TypeScript release. Builds that work today may produce different output tomorrow. Use a fixed target like `ES2022`.

- **Forgetting that path aliases need runtime support.** TypeScript resolves `@/utils/format` during type checking, but the compiled JavaScript still contains `@/utils/format` as a string. Without runtime resolution (bundler alias, `tsconfig-paths`, or `tsc-alias`), the app crashes with "Module not found".

- **Setting `skipLibCheck: true` without understanding it.** This skips type checking of `.d.ts` files (declaration files from `node_modules`). It speeds up compilation significantly and avoids errors from poorly typed third-party libraries. It is generally safe and recommended, but it means type errors in external declarations go undetected.

- **Not understanding `esModuleInterop`.** Without it, default imports from CommonJS modules (`import fs from "fs"`) fail. With it, TypeScript generates interop helpers. Enable it for compatibility. Alternatively, use `verbatimModuleSyntax` for stricter import/export behavior (this replaces `esModuleInterop` in modern setups).

## Best Practices

- **One tsconfig per concern.** Use `tsconfig.json` for the main build, `tsconfig.test.json` for tests (may include looser settings), and extend from a shared base. Do not put test files and source files in the same compilation unit if they have different requirements.

- **Use `tsc --noEmit` for type checking.** If a bundler handles transpilation, TypeScript's only job is type checking. Run `tsc --noEmit` in CI and pre-commit hooks.

- **Enable `sourceMap` and `declarationMap` for development.** Source maps let you debug TypeScript directly in the browser or Node.js. Declaration maps let "Go to Definition" in your editor jump to the original `.ts` file.

- **Add `isolatedModules: true` if using a bundler.** Bundlers like esbuild, swc, and Babel compile each file independently. `isolatedModules` makes TypeScript flag code patterns that break single-file compilation (like `const enum` across files or re-exporting types without `type` keyword).

- **Use `verbatimModuleSyntax` in modern projects.** This replaces `esModuleInterop` and `isolatedModules` with a single, stricter rule: import/export statements are emitted exactly as written. It forces you to use `import type` for type-only imports, making the intent explicit.

```typescript
// With verbatimModuleSyntax:
import type { User } from "./models"; // type-only, erased at compile
import { formatDate } from "./utils"; // value import, kept in output
```

- **Lock TypeScript version in `package.json`.** Use exact versions (`"typescript": "5.8.3"`) or at least pin the minor (`"~5.8.0"`) to prevent surprise changes when a new version releases.

## Real-World Use Cases

### 1. Monorepo with Shared Types

```
packages/
  shared/
    src/types.ts
    tsconfig.json
  api/
    src/index.ts
    tsconfig.json
  web/
    src/index.ts
    tsconfig.json
tsconfig.base.json
```

```jsonc
// tsconfig.base.json
{
  "compilerOptions": {
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true,
    "composite": true
  }
}

// packages/shared/tsconfig.json
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "outDir": "dist",
    "rootDir": "src"
  },
  "include": ["src"]
}

// packages/api/tsconfig.json
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "outDir": "dist",
    "rootDir": "src"
  },
  "references": [{ "path": "../shared" }],
  "include": ["src"]
}
```

### 2. CI Type Check Script

```json
{
  "scripts": {
    "typecheck": "tsc --noEmit",
    "typecheck:watch": "tsc --noEmit --watch",
    "lint": "eslint src/",
    "test": "vitest run",
    "ci": "npm run typecheck && npm run lint && npm run test"
  }
}
```

### 3. Gradual Migration from JavaScript

```jsonc
{
  "compilerOptions": {
    "allowJs": true,           // Allow .js files in the project
    "checkJs": false,          // Do not type-check .js files (yet)
    "strict": true,
    "outDir": "dist",
    "rootDir": "src"
  },
  "include": ["src"]
}
```

Rename files from `.js` to `.ts` one at a time. Each renamed file gets full type checking. The rest continue working as plain JavaScript. Increase strictness gradually:

1. Start with `allowJs: true`, `checkJs: false`.
2. Enable `checkJs: true` to get basic checking on `.js` files.
3. Rename `.js` to `.ts` file by file, fixing type errors as you go.
4. Once all files are `.ts`, remove `allowJs`.

---

> **Key Takeaway:** Start every TypeScript project with `strict: true`, pair `module` and `moduleResolution` correctly for your runtime (NodeNext/NodeNext for Node.js, ESNext/Bundler for frontend), and use `tsc --noEmit` for type checking when a bundler handles transpilation.
