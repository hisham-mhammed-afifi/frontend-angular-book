# 7.13 Conditional Types

> Conditional types express "if-then-else" logic at the type level. They enable types that change shape based on the type they receive, powering most of TypeScript's advanced type transformations including `Exclude`, `Extract`, `ReturnType`, and `Awaited`.

---

## Basic Syntax

A conditional type uses the same `extends` keyword used in constraints, combined with a ternary expression.

```typescript
type Condition = T extends U ? TrueType : FalseType;
```

If `T` is assignable to `U`, the result is `TrueType`. Otherwise it is `FalseType`.

```typescript
type IsString<T> = T extends string ? true : false;

type A = IsString<string>;  // true
type B = IsString<number>;  // false
type C = IsString<"hello">; // true ("hello" extends string)
```

The `extends` here means "is assignable to," not "inherits from." A literal `"hello"` is assignable to `string`, so the condition is true.

---

## How Conditional Types Work

### Simple Type Checks

```typescript
type IsNumber<T> = T extends number ? "yes" : "no";

type A = IsNumber<42>;      // "yes"
type B = IsNumber<string>;  // "no"
type C = IsNumber<number>;  // "yes"
```

### Selecting Between Types

```typescript
type TypeName<T> =
  T extends string ? "string" :
  T extends number ? "number" :
  T extends boolean ? "boolean" :
  T extends undefined ? "undefined" :
  T extends Function ? "function" :
  "object";

type A = TypeName<string>;      // "string"
type B = TypeName<42>;          // "number"
type C = TypeName<() => void>;  // "function"
type D = TypeName<Date>;        // "object"
```

### Conditional Return Types

```typescript
function process<T extends string | number>(
  value: T,
): T extends string ? string : number {
  if (typeof value === "string") {
    return value.toUpperCase() as any;
  }
  return (value * 2) as any;
}

const a = process("hello"); // string
const b = process(42);      // number
```

Note: the `as any` casts are needed because TypeScript cannot verify the implementation matches the conditional return type. This is a known limitation.

## Distributive Conditional Types

When a conditional type is applied to a **naked type parameter** (one that is not wrapped in another type), and that parameter receives a union, the conditional distributes over each member of the union individually.

### Distribution in Action

```typescript
type ToArray<T> = T extends any ? T[] : never;

type Result = ToArray<string | number>;
// Distributes:
// (string extends any ? string[] : never) | (number extends any ? number[] : never)
// = string[] | number[]
```

Each member of the union is processed independently, and the results form a new union.

### More Examples

```typescript
type Exclude<T, U> = T extends U ? never : T;

type Result = Exclude<"a" | "b" | "c", "a">;
// ("a" extends "a" ? never : "a") | ("b" extends "a" ? never : "b") | ("c" extends "a" ? never : "c")
// = never | "b" | "c"
// = "b" | "c"
```

```typescript
type Extract<T, U> = T extends U ? T : never;

type Result = Extract<string | number | boolean, string | number>;
// (string extends string | number ? string : never) | (number extends ...) | (boolean extends ...)
// = string | number | never
// = string | number
```

### What "Naked Type Parameter" Means

Distribution only happens when the checked type `T` appears directly, not wrapped in another type construct.

```typescript
// Naked T: distributes
type Dist<T> = T extends string ? "yes" : "no";
type A = Dist<string | number>; // "yes" | "no"

// Wrapped in tuple: does NOT distribute
type NoDist<T> = [T] extends [string] ? "yes" : "no";
type B = NoDist<string | number>; // "no" (the union as a whole is checked)
```

Wrapping contexts that prevent distribution:
- `[T]` (tuple)
- `{ value: T }` (object)
- `T[]` (array)
- `Promise<T>` (generic wrapper)

## Preventing Distribution

Sometimes you want to treat a union as a single unit rather than distributing over it.

### The Tuple Wrapping Trick

```typescript
// Distributive: each member checked individually
type IsStringDist<T> = T extends string ? true : false;
type A = IsStringDist<string | number>; // true | false (= boolean)

// Non-distributive: union checked as a whole
type IsStringAll<T> = [T] extends [string] ? true : false;
type B = IsStringAll<string | number>; // false (string | number does not extend string)
type C = IsStringAll<string>;          // true
```

### Practical Use: Check if Type IS a Union

```typescript
type IsUnion<T, Copy = T> =
  T extends unknown
    ? [Copy] extends [T]
      ? false
      : true
    : never;

type A = IsUnion<string>;          // false
type B = IsUnion<string | number>; // true
```

This works because the distributive branch processes each member, then checks if the original copy (as a whole) extends that single member. If `T` is not a union, `Copy` and `T` are the same, so the check passes and returns `false`.

## The `infer` Keyword

`infer` declares a type variable that TypeScript fills in from context. It can only appear in the `extends` clause of a conditional type, and the inferred variable is available in the true branch.

### Basic Inference

```typescript
// Extract element type from an array
type ElementOf<T> = T extends (infer E)[] ? E : never;

type A = ElementOf<string[]>;  // string
type B = ElementOf<number[]>;  // number
type C = ElementOf<string>;    // never (not an array)
```

### Inferring Function Parts

```typescript
// Return type
type MyReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

type A = MyReturnType<() => string>;           // string
type B = MyReturnType<(x: number) => boolean>; // boolean

// Parameter types
type MyParameters<T> = T extends (...args: infer P) => any ? P : never;

type C = MyParameters<(a: string, b: number) => void>; // [a: string, b: number]

// First parameter
type FirstParam<T> = T extends (first: infer F, ...rest: any[]) => any ? F : never;

type D = FirstParam<(name: string, age: number) => void>; // string
```

### Inferring from Object Types

```typescript
// Extract the value type of a Promise
type UnwrapPromise<T> = T extends Promise<infer V> ? V : T;

type A = UnwrapPromise<Promise<string>>;  // string
type B = UnwrapPromise<number>;           // number

// Extract a specific property type
type PropType<T, K extends string> = T extends { [key in K]: infer V } ? V : never;

type A2 = PropType<{ name: string; age: number }, "name">; // string
type B2 = PropType<{ name: string; age: number }, "age">;  // number
```

### Inferring from Template Literal Types

```typescript
type ExtractRoute<T extends string> =
  T extends `${infer Method} ${infer Path}`
    ? { method: Method; path: Path }
    : never;

type A = ExtractRoute<"GET /users">;
// { method: "GET"; path: "/users" }

type B = ExtractRoute<"POST /users/:id/posts">;
// { method: "POST"; path: "/users/:id/posts" }
```

```typescript
// Extract route parameters
type ExtractParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? Param | ExtractParams<`/${Rest}`>
    : T extends `${string}:${infer Param}`
      ? Param
      : never;

type Params = ExtractParams<"/users/:userId/posts/:postId">;
// "userId" | "postId"
```

## `infer` in Covariant vs Contravariant Positions

Where `infer` appears determines how TypeScript resolves multiple candidates.

### Covariant Position (Return Types): Union

When multiple candidates are inferred from covariant positions (output/return positions), the result is a **union**.

```typescript
type CovariantInfer<T> = T extends { a: infer U; b: infer U } ? U : never;

type Result = CovariantInfer<{ a: string; b: number }>;
// string | number (union of candidates)
```

### Contravariant Position (Parameters): Intersection

When multiple candidates are inferred from contravariant positions (input/parameter positions), the result is an **intersection**.

```typescript
type ContravariantInfer<T> =
  T extends { a: (x: infer U) => void; b: (x: infer U) => void } ? U : never;

type Result = ContravariantInfer<{ a: (x: string) => void; b: (x: number) => void }>;
// string & number = never (intersection of incompatible types)
```

This behavior is useful for extracting union vs intersection types from complex structures.

### Practical Application: Union to Intersection

```typescript
type UnionToIntersection<U> =
  (U extends any ? (k: U) => void : never) extends (k: infer I) => void
    ? I
    : never;

type Result = UnionToIntersection<{ a: 1 } | { b: 2 } | { c: 3 }>;
// { a: 1 } & { b: 2 } & { c: 3 }
```

This works by:
1. Distributing the union into function parameters (contravariant position)
2. Inferring from that contravariant position, which produces an intersection

## Recursive Conditional Types

Conditional types can reference themselves to process nested or variadic structures.

### Deep Promise Unwrapping

```typescript
type DeepAwaited<T> = T extends Promise<infer U> ? DeepAwaited<U> : T;

type A = DeepAwaited<Promise<Promise<Promise<string>>>>;
// string (unwraps all layers)
```

### Flatten Nested Arrays

```typescript
type Flatten<T> = T extends (infer E)[]
  ? Flatten<E>
  : T;

type A = Flatten<number[][][]>;  // number
type B = Flatten<string[]>;      // string
type C = Flatten<boolean>;       // boolean
```

### Deep Path Types

```typescript
type Paths<T, Prefix extends string = ""> = T extends object
  ? {
      [K in keyof T & string]:
        | `${Prefix}${K}`
        | Paths<T[K], `${Prefix}${K}.`>;
    }[keyof T & string]
  : never;

type User = {
  name: string;
  address: {
    street: string;
    city: string;
    coordinates: {
      lat: number;
      lng: number;
    };
  };
};

type UserPaths = Paths<User>;
// "name" | "address" | "address.street" | "address.city"
// | "address.coordinates" | "address.coordinates.lat" | "address.coordinates.lng"
```

### Recursion Depth Limits

TypeScript enforces recursion depth limits. Exceeding them produces:

```
Type instantiation is excessively deep and possibly infinite.
```

Use a depth counter to cap recursion:

```typescript
type DeepPartial<T, Depth extends unknown[] = []> =
  Depth["length"] extends 10
    ? T // stop recursing at depth 10
    : T extends object
      ? { [K in keyof T]?: DeepPartial<T[K], [...Depth, unknown]> }
      : T;
```

## Conditional Types with Mapped Types

Combining conditional types with mapped types creates powerful transformations.

### Filter Object Properties by Type

```typescript
type PickByType<T, V> = {
  [K in keyof T as T[K] extends V ? K : never]: T[K];
};

interface User {
  id: number;
  name: string;
  email: string;
  age: number;
  isActive: boolean;
}

type StringFields = PickByType<User, string>;
// { name: string; email: string }
```

### Make Specific Properties Optional

```typescript
type PartialBy<T, K extends keyof T> = Omit<T, K> & Partial<Pick<T, K>>;

type User = {
  id: string;
  name: string;
  email: string;
};

type CreateUser = PartialBy<User, "id">;
// { name: string; email: string; id?: string }
```

### Transform Method Signatures

```typescript
type AsyncMethods<T> = {
  [K in keyof T]: T[K] extends (...args: infer A) => infer R
    ? (...args: A) => Promise<R>
    : T[K];
};

interface UserService {
  getUser(id: string): User;
  deleteUser(id: string): boolean;
  name: string;
}

type AsyncUserService = AsyncMethods<UserService>;
// {
//   getUser(id: string): Promise<User>;
//   deleteUser(id: string): Promise<boolean>;
//   name: string; // non-function properties unchanged
// }
```

## The `never` Type in Conditional Types

`never` has special behavior with conditional types that you must understand.

### `never` as Input

When `never` is the input to a distributive conditional type, the result is always `never`. This is because `never` is an empty union, and distributing over zero members produces zero results.

```typescript
type IsString<T> = T extends string ? true : false;

type Result = IsString<never>;
// never (NOT false!)
// Because never is an empty union, distribution produces nothing
```

### Preventing `never` Propagation

```typescript
// Distributive: never in, never out
type Check<T> = T extends string ? "yes" : "no";
type A = Check<never>; // never

// Non-distributive: handles never correctly
type CheckSafe<T> = [T] extends [string] ? "yes" : "no";
type B = CheckSafe<never>; // "no" (never extends string is false)
```

### Check for `never`

```typescript
type IsNever<T> = [T] extends [never] ? true : false;

type A = IsNever<never>;  // true
type B = IsNever<string>; // false
type C = IsNever<"">;     // false
```

The tuple wrapper is essential here. Without it, `never` would short-circuit and return `never` instead of `true`.

## Conditional Types vs Overloads

Conditional types can sometimes replace function overloads, but each approach has trade-offs.

### With Overloads

```typescript
function process(value: string): string;
function process(value: number): number;
function process(value: string | number): string | number {
  if (typeof value === "string") return value.toUpperCase();
  return value * 2;
}

const a = process("hello"); // string
const b = process(42);      // number
```

### With Conditional Types

```typescript
type ProcessResult<T> = T extends string ? string : T extends number ? number : never;

function process<T extends string | number>(value: T): ProcessResult<T> {
  if (typeof value === "string") return value.toUpperCase() as ProcessResult<T>;
  return (value as number * 2) as ProcessResult<T>;
}

const a = process("hello"); // string
const b = process(42);      // number
```

| Feature | Overloads | Conditional Return Type |
|---------|-----------|------------------------|
| Type safety at call site | Strong | Strong |
| Implementation type safety | Weak (one implementation) | Weak (needs `as` casts) |
| Scalability | Each case needs a signature | Single generic signature |
| Readability | Verbose but explicit | Compact but harder to read |

Prefer overloads when there are 2-3 variants. Use conditional return types when the logic is systematic and has many cases.

## Common Mistakes

### Mistake 1: Forgetting About Distribution

```typescript
type ToArray<T> = T extends any ? T[] : never;

// Expected: (string | number)[]
// Actual: string[] | number[]
type Result = ToArray<string | number>;

// Fix: prevent distribution
type ToArrayNonDist<T> = [T] extends [any] ? T[] : never;
type Fixed = ToArrayNonDist<string | number>; // (string | number)[]
```

### Mistake 2: `never` Disappearing

```typescript
type Filter<T> = T extends string ? T : never;

type Result = Filter<never>;
// Expected: never (which is correct, but might surprise you)
// never as input always produces never as output in distributive types

// If you need to handle never explicitly:
type SafeFilter<T> = [T] extends [never] ? never : (T extends string ? T : never);
```

### Mistake 3: Complex Conditional Return Types Cannot Be Verified

```typescript
function getValue<T extends "string" | "number">(
  type: T,
): T extends "string" ? string : number {
  if (type === "string") {
    return "hello"; // Error: Type 'string' is not assignable to type 'T extends "string" ? string : number'
  }
  return 42; // Same error
}

// TypeScript cannot narrow generic conditional types inside the function body
// You must use 'as any' or 'as' casts in the implementation
function getValue<T extends "string" | "number">(
  type: T,
): T extends "string" ? string : number {
  if (type === "string") {
    return "hello" as any;
  }
  return 42 as any;
}
```

This is a known TypeScript limitation. The type system cannot verify that the implementation matches a generic conditional return type.

### Mistake 4: Inferring from Complex Structures Without Base Cases

```typescript
// Bad: no base case for non-matching types
type GetValue<T> = T extends { value: infer V } ? V : T;

// This works but can be confusing when T does not match:
type A = GetValue<string>; // string (falls through to the false branch)

// Better: use never for non-matching cases to make failures explicit
type GetValueStrict<T> = T extends { value: infer V } ? V : never;
type B = GetValueStrict<string>; // never (clearly indicates no match)
```

### Mistake 5: Nested Ternaries Becoming Unreadable

```typescript
// Bad: deeply nested, hard to follow
type Classify<T> =
  T extends string ? "string" :
  T extends number ? "number" :
  T extends boolean ? "boolean" :
  T extends null ? "null" :
  T extends undefined ? "undefined" :
  T extends Function ? "function" :
  T extends Array<any> ? "array" :
  "object";

// Better: break into helpers
type IsFunction<T> = T extends Function ? true : false;
type IsArray<T> = T extends Array<any> ? true : false;
type IsPrimitive<T> = T extends string | number | boolean | null | undefined ? true : false;
```

## Best Practices

### 1. Know When Distribution Matters

```typescript
// Distributive: needed for Exclude/Extract patterns
type NonNullable<T> = T extends null | undefined ? never : T;

// Non-distributive: needed for checking the type as a whole
type IsExactlyString<T> = [T] extends [string] ? true : false;
```

### 2. Use `infer` to Avoid Redundant Type Parameters

```typescript
// Bad: extra type parameter that could be inferred
type GetReturn<T extends (...args: any[]) => any, R = ReturnType<T>> = R;

// Good: infer directly
type GetReturn<T> = T extends (...args: any[]) => infer R ? R : never;
```

### 3. Add Depth Limits to Recursive Types

```typescript
type DeepFlatten<T, D extends unknown[] = []> =
  D["length"] extends 5
    ? T
    : T extends (infer E)[]
      ? DeepFlatten<E, [...D, unknown]>
      : T;
```

### 4. Document Complex Conditional Types

```typescript
/**
 * Extracts the resolved type from a Promise chain.
 * - Promise<string> -> string
 * - Promise<Promise<number>> -> number
 * - non-Promise types returned as-is
 */
type DeepAwaited<T> = T extends Promise<infer U> ? DeepAwaited<U> : T;
```

### 5. Test Conditional Types with Type Assertions

```typescript
// Verify your conditional type works as expected
type Assert<T extends true> = T;

type _test1 = Assert<IsString<"hello"> extends true ? true : false>;
type _test2 = Assert<IsString<42> extends false ? true : false>;
type _test3 = Assert<IsNever<never> extends true ? true : false>;
```

## Real-World Use Cases

### Type-Safe API Route Handler

```typescript
type RouteConfig = {
  "GET /users": { response: User[]; query: { page?: number; limit?: number } };
  "GET /users/:id": { response: User; params: { id: string } };
  "POST /users": { response: User; body: CreateUserInput };
  "PUT /users/:id": { response: User; params: { id: string }; body: UpdateUserInput };
  "DELETE /users/:id": { response: void; params: { id: string } };
};

type Route = keyof RouteConfig;

type RouteResponse<R extends Route> = RouteConfig[R]["response"];

type RouteInput<R extends Route> = RouteConfig[R] extends { params: infer P }
  ? RouteConfig[R] extends { body: infer B }
    ? { params: P; body: B }
    : { params: P }
  : RouteConfig[R] extends { body: infer B }
    ? { body: B }
    : RouteConfig[R] extends { query: infer Q }
      ? { query: Q }
      : {};

async function request<R extends Route>(
  route: R,
  input: RouteInput<R>,
): Promise<RouteResponse<R>> {
  // implementation
  return {} as RouteResponse<R>;
}

// Fully type-safe
const users = await request("GET /users", { query: { page: 1 } });
// users: User[]

const user = await request("GET /users/:id", { params: { id: "123" } });
// user: User

const created = await request("POST /users", {
  body: { name: "Alice", email: "alice@example.com", password: "secret" },
});
// created: User
```

### Smart Merge Utility

```typescript
// Merge two object types, with the second overriding the first
type Merge<A, B> = {
  [K in keyof A | keyof B]: K extends keyof B
    ? B[K]
    : K extends keyof A
      ? A[K]
      : never;
};

type Base = { id: number; name: string; active: boolean };
type Override = { name: string | null; active: number; role: string };

type Merged = Merge<Base, Override>;
// { id: number; name: string | null; active: number; role: string }
```

### Event System with Inferred Payloads

```typescript
type EventDefinitions = {
  click: { x: number; y: number };
  keypress: { key: string; code: number };
  resize: { width: number; height: number };
  custom: undefined;
};

type EventName = keyof EventDefinitions;

type EventPayload<E extends EventName> = EventDefinitions[E];

type HasPayload<E extends EventName> = EventPayload<E> extends undefined ? false : true;

// Handler type changes based on whether the event has a payload
type EventHandler<E extends EventName> = HasPayload<E> extends true
  ? (payload: EventPayload<E>) => void
  : () => void;

class EventBus {
  on<E extends EventName>(event: E, handler: EventHandler<E>): void {
    // ...
  }

  emit<E extends EventName>(
    ...args: HasPayload<E> extends true ? [event: E, payload: EventPayload<E>] : [event: E]
  ): void {
    // ...
  }
}

const bus = new EventBus();

bus.on("click", (payload) => {
  console.log(payload.x, payload.y); // fully typed
});

bus.on("custom", () => {
  console.log("custom event fired"); // no payload parameter
});

bus.emit("click", { x: 10, y: 20 });   // payload required
bus.emit("custom");                      // no payload needed
// bus.emit("click");                    // Error: missing payload
// bus.emit("custom", { x: 1 });        // Error: no payload expected
```

---

> **Key Takeaway:** Conditional types bring if-then-else logic to the type level. Master distributive behavior (and how to prevent it with tuple wrapping), `infer` for type extraction, and `never` propagation rules. Use conditional types to build type transformations that adapt to their inputs, but keep them readable by breaking complex logic into named helpers.
