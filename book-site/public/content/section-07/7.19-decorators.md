# 7.19 Decorators (Legacy and Stage 3)

> Decorators are a metaprogramming feature that let you annotate and modify classes and their members at definition time. TypeScript supports both the older experimental implementation and the newer TC39 Stage 3 standard.

---

## Two Decorator Systems

TypeScript has **two** distinct decorator implementations. They are not compatible with each other.

| Aspect | Legacy (Experimental) | Stage 3 (TC39 Standard) |
|--------|----------------------|------------------------|
| TypeScript version | 1.5+ | 5.0+ |
| Requires flag | `experimentalDecorators: true` | No flag needed |
| TC39 stage | Stage 2 (never advanced) | Stage 3 (approaching Stage 4) |
| Decorator arguments | `(target, propertyKey, descriptor)` | `(value, context)` |
| Parameter decorators | Yes | No (separate proposal) |
| `emitDecoratorMetadata` | Yes | No (use `Symbol.metadata`) |
| Auto-accessors | No | Yes (`accessor` keyword) |
| Private member decorators | No | Yes |
| Future-proof | No (will be deprecated) | Yes (aligns with JavaScript) |

**Recommendation:** New projects should use Stage 3 decorators (no flag). Legacy decorators are covered here because they are still heavily used in existing codebases.

---

## Stage 3 Decorators (The Standard)

Stage 3 decorators are the official ECMAScript proposal, supported in TypeScript 5.0+ without any compiler flag. They work by receiving the decorated value and a context object, and optionally returning a replacement.

### Decorator Signature

Every Stage 3 decorator has the same general shape:

```typescript
type Decorator = (value: DecoratedValue, context: DecoratorContext) => ReplacementValue | void;
```

The `context` object tells you what kind of thing is being decorated:

```typescript
interface DecoratorContext {
  kind: "class" | "method" | "getter" | "setter" | "field" | "accessor";
  name: string | symbol;
  static: boolean;
  private: boolean;
  addInitializer(initializer: () => void): void;
  access?: {
    get(): unknown;
    set(value: unknown): void;
  };
  metadata: Record<string | symbol, unknown>;
}
```

Not all properties exist on every kind. TypeScript provides specific context types for each.

---

### Class Decorators

A class decorator receives the class constructor and can return a replacement constructor (typically a subclass):

```typescript
type ClassDecorator = (
  value: Function,
  context: ClassDecoratorContext
) => Function | void;
```

```typescript
function sealed(target: Function, context: ClassDecoratorContext) {
  Object.seal(target);
  Object.seal(target.prototype);
}

@sealed
class User {
  name: string;
  constructor(name: string) {
    this.name = name;
  }
}
```

#### Replacing a Class

```typescript
function withTimestamp<T extends new (...args: any[]) => any>(
  target: T,
  context: ClassDecoratorContext
): T {
  return class extends target {
    createdAt = new Date();
  } as T;
}

@withTimestamp
class Order {
  id: string;
  constructor(id: string) {
    this.id = id;
  }
}

const order = new Order("ord_123");
console.log((order as any).createdAt); // Date object
```

### Method Decorators

A method decorator receives the method function and can return a replacement function:

```typescript
type MethodDecorator = (
  value: Function,
  context: ClassMethodDecoratorContext
) => Function | void;
```

```typescript
function log(
  target: Function,
  context: ClassMethodDecoratorContext
) {
  const methodName = String(context.name);

  return function (this: any, ...args: any[]) {
    console.log(`Called ${methodName} with`, args);
    const result = target.apply(this, args);
    console.log(`${methodName} returned`, result);
    return result;
  };
}

class Calculator {
  @log
  add(a: number, b: number): number {
    return a + b;
  }
}

const calc = new Calculator();
calc.add(2, 3);
// Called add with [2, 3]
// add returned 5
```

#### Using `addInitializer`

The `addInitializer` function registers a callback that runs when the class is instantiated. This is useful for binding methods:

```typescript
function bound(
  target: Function,
  context: ClassMethodDecoratorContext
) {
  const methodName = context.name;

  context.addInitializer(function (this: any) {
    this[methodName] = this[methodName].bind(this);
  });

  return target;
}

class Button {
  label = "Click me";

  @bound
  handleClick() {
    console.log(this.label); // Always refers to the Button instance
  }
}

const btn = new Button();
const handler = btn.handleClick;
handler(); // "Click me" (not undefined, because it was bound)
```

### Getter and Setter Decorators

Getter and setter decorators work like method decorators but target accessors specifically:

```typescript
function validate(
  target: () => any,  // For getters, the getter function
  context: ClassGetterDecoratorContext | ClassSetterDecoratorContext
) {
  if (context.kind === "setter") {
    return function (this: any, value: any) {
      if (value == null) {
        throw new Error(`${String(context.name)} cannot be null`);
      }
      return (target as Function).call(this, value);
    };
  }
  return target;
}

class Config {
  private _host = "localhost";

  get host(): string {
    return this._host;
  }

  @validate
  set host(value: string) {
    this._host = value;
  }
}

const config = new Config();
config.host = "example.com"; // OK
// config.host = null;       // Error at runtime: host cannot be null
```

### Field Decorators

Field decorators receive `undefined` as the first argument (fields have no function to wrap). They return an **initializer function** that transforms the field's initial value:

```typescript
type FieldDecorator = (
  value: undefined,
  context: ClassFieldDecoratorContext
) => ((initialValue: unknown) => unknown) | void;
```

```typescript
function defaultValue<T>(value: T) {
  return function (
    _target: undefined,
    context: ClassFieldDecoratorContext
  ) {
    return (initialValue: T | undefined) => initialValue ?? value;
  };
}

class Settings {
  @defaultValue(3000)
  port!: number;

  @defaultValue("localhost")
  host!: string;

  @defaultValue(false)
  debug!: boolean;
}

const s = new Settings();
console.log(s.port);  // 3000
console.log(s.host);  // "localhost"
console.log(s.debug); // false
```

### Auto-Accessor Decorators (The `accessor` Keyword)

Stage 3 introduces the `accessor` keyword, which creates a class field backed by a private slot with an automatic getter and setter. This is a new language feature, not just a TypeScript addition.

```typescript
class User {
  accessor name = "Anonymous";
}

// Equivalent to:
class User {
  #name = "Anonymous";
  get name() { return this.#name; }
  set name(value) { this.#name = value; }
}
```

Auto-accessor decorators can return an object with `get`, `set`, and `init` functions:

```typescript
function tracked<T extends object, V>(
  accessorTarget: { get: () => V; set: (value: V) => void },
  context: ClassAccessorDecoratorContext<T, V>
): {
  get?: () => V;
  set?: (value: V) => void;
  init?: (value: V) => V;
} {
  const fieldName = String(context.name);

  return {
    get() {
      const value = accessorTarget.get.call(this);
      console.log(`GET ${fieldName}:`, value);
      return value;
    },
    set(newValue: V) {
      console.log(`SET ${fieldName}:`, newValue);
      accessorTarget.set.call(this, newValue);
    },
    init(initialValue: V) {
      console.log(`INIT ${fieldName}:`, initialValue);
      return initialValue;
    },
  };
}

class UserProfile {
  @tracked
  accessor displayName = "Guest";

  @tracked
  accessor email = "";
}

const profile = new UserProfile();
// INIT displayName: Guest
// INIT email:

profile.displayName = "Alice";
// SET displayName: Alice

console.log(profile.displayName);
// GET displayName: Alice
```

Auto-accessors are the **most powerful** decorator target because they give you hooks into initialization, reads, and writes.

### Decorator Composition and Evaluation Order

When multiple decorators are stacked, they evaluate bottom-to-top (inner first):

```typescript
function first(target: Function, context: ClassMethodDecoratorContext) {
  console.log("first applied");
  return function (this: any, ...args: any[]) {
    console.log("first called");
    return (target as Function).apply(this, args);
  };
}

function second(target: Function, context: ClassMethodDecoratorContext) {
  console.log("second applied");
  return function (this: any, ...args: any[]) {
    console.log("second called");
    return (target as Function).apply(this, args);
  };
}

class Example {
  @first
  @second
  greet() {
    return "hello";
  }
}

// Application order (bottom-up):
// "second applied"
// "first applied"

// Call order (top-down, because first wraps second's result):
// "first called"
// "second called"
```

This is analogous to function composition: `first(second(greet))`.

### Decorator Factories

To pass arguments to a decorator, use a factory function (a function that returns a decorator):

```typescript
function throttle(ms: number) {
  return function (
    target: Function,
    context: ClassMethodDecoratorContext
  ) {
    let lastCall = 0;

    return function (this: any, ...args: any[]) {
      const now = Date.now();
      if (now - lastCall < ms) return;
      lastCall = now;
      return (target as Function).apply(this, args);
    };
  };
}

class ScrollHandler {
  @throttle(200)
  onScroll(event: Event) {
    console.log("Scroll handled");
  }
}
```

### Metadata with `Symbol.metadata`

The TC39 Decorator Metadata proposal (complementary to the decorators proposal) provides a standard way to attach metadata to classes via `Symbol.metadata`:

```typescript
function role(name: string) {
  return function (
    _target: Function,
    context: ClassDecoratorContext
  ) {
    context.metadata.role = name;
  };
}

function permission(perm: string) {
  return function (
    _target: Function,
    context: ClassMethodDecoratorContext
  ) {
    const perms = (context.metadata.permissions as string[]) ?? [];
    perms.push(perm);
    context.metadata.permissions = perms;
  };
}

@role("admin")
class AdminController {
  @permission("users:read")
  listUsers() { /* ... */ }

  @permission("users:write")
  createUser() { /* ... */ }
}

// Access metadata via Symbol.metadata
const meta = AdminController[Symbol.metadata];
console.log(meta?.role);        // "admin"
console.log(meta?.permissions); // ["users:read", "users:write"]
```

Metadata is inherited through the prototype chain. If a subclass does not define its own metadata, it inherits from its parent.

## Legacy (Experimental) Decorators

Legacy decorators require the `experimentalDecorators: true` flag in `tsconfig.json`. They use a different API based on `target`, `propertyKey`, and `PropertyDescriptor`.

```json
{
  "compilerOptions": {
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true
  }
}
```

### Legacy Class Decorator

```typescript
function sealed(constructor: Function) {
  Object.seal(constructor);
  Object.seal(constructor.prototype);
}

@sealed
class Greeter {
  greeting: string;
  constructor(message: string) {
    this.greeting = message;
  }
}
```

### Legacy Method Decorator

The method decorator receives three arguments:

```typescript
function enumerable(value: boolean) {
  return function (
    target: any,
    propertyKey: string,
    descriptor: PropertyDescriptor
  ) {
    descriptor.enumerable = value;
  };
}

class Greeter {
  greeting: string;
  constructor(message: string) {
    this.greeting = message;
  }

  @enumerable(false)
  greet() {
    return `Hello, ${this.greeting}`;
  }
}
```

### Legacy Property Decorator

```typescript
function format(formatString: string) {
  return function (target: any, propertyKey: string) {
    let value: string;

    const getter = () => value;
    const setter = (newVal: string) => {
      value = formatString.replace("%s", newVal);
    };

    Object.defineProperty(target, propertyKey, {
      get: getter,
      set: setter,
      enumerable: true,
      configurable: true,
    });
  };
}

class Greeter {
  @format("Hello, %s!")
  greeting!: string;
}

const g = new Greeter();
g.greeting = "World";
console.log(g.greeting); // "Hello, World!"
```

### Legacy Parameter Decorator

Parameter decorators are **unique to the legacy system**. They do not exist in Stage 3 (a separate proposal is in progress for them):

```typescript
function required(
  target: any,
  propertyKey: string,
  parameterIndex: number
) {
  const requiredParams: number[] =
    Reflect.getOwnMetadata("required", target, propertyKey) ?? [];
  requiredParams.push(parameterIndex);
  Reflect.defineMetadata("required", requiredParams, target, propertyKey);
}

class UserService {
  createUser(
    @required name: string,
    @required email: string,
    bio?: string
  ) {
    // ...
  }
}
```

Parameter decorators were heavily used by dependency injection frameworks to mark injectable parameters.

### Legacy `emitDecoratorMetadata`

When `emitDecoratorMetadata: true` is set, TypeScript emits design-time type information using the `reflect-metadata` library:

```typescript
import "reflect-metadata";

function logType(target: any, propertyKey: string) {
  const type = Reflect.getMetadata("design:type", target, propertyKey);
  console.log(`${propertyKey} type: ${type.name}`);
}

class Demo {
  @logType
  name!: string;
  // Logs: "name type: String"

  @logType
  count!: number;
  // Logs: "count type: Number"
}
```

This metadata emission is a TypeScript-only feature. It does not exist in the Stage 3 proposal (which uses `Symbol.metadata` instead).

## Comparison: The Same Decorator in Both Systems

### Logging Method Decorator

**Stage 3 (no flag):**

```typescript
function log(
  target: Function,
  context: ClassMethodDecoratorContext
) {
  const name = String(context.name);
  return function (this: any, ...args: any[]) {
    console.log(`[${name}] args:`, args);
    const result = (target as Function).apply(this, args);
    console.log(`[${name}] result:`, result);
    return result;
  };
}

class MathService {
  @log
  multiply(a: number, b: number): number {
    return a * b;
  }
}
```

**Legacy (`experimentalDecorators: true`):**

```typescript
function log(
  target: any,
  propertyKey: string,
  descriptor: PropertyDescriptor
) {
  const original = descriptor.value;
  descriptor.value = function (...args: any[]) {
    console.log(`[${propertyKey}] args:`, args);
    const result = original.apply(this, args);
    console.log(`[${propertyKey}] result:`, result);
    return result;
  };
  return descriptor;
}

class MathService {
  @log
  multiply(a: number, b: number): number {
    return a * b;
  }
}
```

Key differences:
- Stage 3 returns a **replacement function**. Legacy modifies the **property descriptor**.
- Stage 3 receives a **context object**. Legacy receives **three positional arguments**.
- Stage 3 has `context.name` as `string | symbol`. Legacy has `propertyKey` as `string`.

## Common Mistakes

### 1. Mixing Legacy and Stage 3 Decorators

```json
// Bad Practice: enabling experimentalDecorators with Stage 3 decorator code
{
  "compilerOptions": {
    "experimentalDecorators": true  // This switches to the legacy system!
  }
}
```

When `experimentalDecorators` is `true`, TypeScript uses the legacy decorator API. Stage 3 decorator code will not work correctly. You must choose one system per project:

- **Stage 3:** Remove `experimentalDecorators` from tsconfig (or set to `false`)
- **Legacy:** Set `experimentalDecorators: true`

### 2. Returning the Wrong Value from a Decorator

```typescript
// Bad Practice: Stage 3 method decorator returning undefined when it should return a function
function broken(target: Function, context: ClassMethodDecoratorContext) {
  console.log("decorated");
  // Forgot to return target or a replacement!
  // The method is now undefined.
}

// Good: always return a function (or the original target)
function working(target: Function, context: ClassMethodDecoratorContext) {
  console.log("decorated");
  return target; // Return the original if not modifying
}
```

### 3. Mutating State in Field Decorators Instead of Using the Initializer

```typescript
// Bad Practice: trying to set the field value directly
function defaultName(
  _target: undefined,
  context: ClassFieldDecoratorContext
) {
  // This does nothing to the field value
  console.log("Decorated", context.name);
}

// Good: return an initializer function
function defaultName(
  _target: undefined,
  context: ClassFieldDecoratorContext
) {
  return (value: unknown) => value ?? "Default Name";
}
```

Field decorators cannot directly set values. They must return a function that receives the initial value and returns the transformed value.

### 4. Assuming Parameter Decorators Exist in Stage 3

```typescript
// This ONLY works with experimentalDecorators: true
class Service {
  method(@inject("token") dependency: any) { }
}

// Stage 3 does NOT support parameter decorators.
// This is a separate TC39 proposal still in progress.
```

If your codebase relies on parameter decorators (common in DI frameworks), you must continue using legacy decorators until the parameter decorator proposal advances.

### 5. Forgetting That Decorator Evaluation Order Changed

Legacy decorators evaluate all instance decorators before all static decorators. Stage 3 decorators evaluate in **source order**, regardless of whether they are static or instance:

```typescript
// Stage 3: evaluated in source order (top to bottom)
class Example {
  @decorateA        // 1st
  instanceMethod() {}

  @decorateB        // 2nd
  static staticMethod() {}

  @decorateC        // 3rd
  anotherMethod() {}
}
```

## Best Practices

### 1. Use Stage 3 Decorators for New Projects

Stage 3 decorators are the future of JavaScript. They work without a flag, have better type safety, support private members, and will become a native JavaScript feature.

### 2. Keep Decorators Focused on a Single Responsibility

Each decorator should do one thing:

```typescript
// Good: separate concerns into separate decorators
@logged
@cached(60)
@authorized("admin")
getData() { /* ... */ }

// Bad: one decorator that does logging, caching, and auth
@doEverything({ log: true, cache: 60, role: "admin" })
getData() { /* ... */ }
```

### 3. Use Decorator Factories for Configuration

Always use a factory when the decorator needs parameters:

```typescript
// Factory pattern
function retry(maxAttempts: number) {
  return function (
    target: Function,
    context: ClassMethodDecoratorContext
  ) {
    return async function (this: any, ...args: any[]) {
      for (let i = 0; i < maxAttempts; i++) {
        try {
          return await (target as Function).apply(this, args);
        } catch (error) {
          if (i === maxAttempts - 1) throw error;
        }
      }
    };
  };
}

class ApiClient {
  @retry(3)
  async fetchData(url: string) {
    const response = await fetch(url);
    if (!response.ok) throw new Error("Failed");
    return response.json();
  }
}
```

### 4. Prefer `accessor` for Reactive Field Decoration

Auto-accessors give decorators hooks into both reads and writes, making them ideal for reactivity, validation, and observation patterns:

```typescript
function clamp(min: number, max: number) {
  return function (
    target: { get: () => number; set: (v: number) => void },
    context: ClassAccessorDecoratorContext
  ) {
    return {
      set(value: number) {
        target.set.call(this, Math.min(max, Math.max(min, value)));
      },
      get() {
        return target.get.call(this);
      },
    };
  };
}

class Volume {
  @clamp(0, 100)
  accessor level = 50;
}

const v = new Volume();
v.level = 150;
console.log(v.level); // 100 (clamped)
v.level = -10;
console.log(v.level); // 0 (clamped)
```

### 5. Document Decorators Thoroughly

Decorators are "magic" to consumers. Always document what they do:

```typescript
/**
 * Measures the execution time of the decorated method and logs it.
 *
 * @example
 * class Service {
 *   @measure
 *   async fetchData() { ... }
 * }
 * // Logs: "fetchData took 142ms"
 */
function measure(/* ... */) { /* ... */ }
```

## Real-World Use Cases

### 1. Method Timing and Performance Monitoring

```typescript
function measure(
  target: Function,
  context: ClassMethodDecoratorContext
) {
  const name = String(context.name);

  return async function (this: any, ...args: any[]) {
    const start = performance.now();
    try {
      const result = await (target as Function).apply(this, args);
      return result;
    } finally {
      const duration = performance.now() - start;
      console.log(`${name} took ${duration.toFixed(2)}ms`);
      // In production, send to your monitoring service instead
    }
  };
}

class DataService {
  @measure
  async fetchUsers(): Promise<User[]> {
    const response = await fetch("/api/users");
    return response.json();
  }

  @measure
  async searchProducts(query: string): Promise<Product[]> {
    const response = await fetch(`/api/products?q=${query}`);
    return response.json();
  }
}
```

### 2. Memoization Decorator

```typescript
function memoize(
  target: Function,
  context: ClassMethodDecoratorContext
) {
  const cache = new Map<string, unknown>();

  return function (this: any, ...args: any[]) {
    const key = JSON.stringify(args);
    if (cache.has(key)) {
      return cache.get(key);
    }
    const result = (target as Function).apply(this, args);
    cache.set(key, result);
    return result;
  };
}

class MathService {
  @memoize
  fibonacci(n: number): number {
    if (n <= 1) return n;
    return this.fibonacci(n - 1) + this.fibonacci(n - 2);
  }

  @memoize
  factorial(n: number): number {
    if (n <= 1) return 1;
    return n * this.factorial(n - 1);
  }
}

const math = new MathService();
console.log(math.fibonacci(40)); // Fast: result is cached
console.log(math.fibonacci(40)); // Instant: from cache
```

### 3. Validation with Auto-Accessors

```typescript
type Validator<T> = (value: T) => boolean;

function validate<T extends object, V>(
  predicate: Validator<V>,
  errorMessage: string
) {
  return function (
    target: { get: () => V; set: (v: V) => void },
    context: ClassAccessorDecoratorContext<T, V>
  ) {
    return {
      set(this: T, value: V) {
        if (!predicate(value)) {
          throw new Error(`${String(context.name)}: ${errorMessage}`);
        }
        target.set.call(this, value);
      },
      get(this: T) {
        return target.get.call(this);
      },
    };
  };
}

const isPositive = (n: number) => n > 0;
const isNonEmpty = (s: string) => s.trim().length > 0;
const isValidEmail = (s: string) => /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(s);

class UserForm {
  @validate(isNonEmpty, "Name must not be empty")
  accessor name = "";

  @validate(isValidEmail, "Invalid email format")
  accessor email = "";

  @validate(isPositive, "Age must be positive")
  accessor age = 1;
}

const form = new UserForm();
form.name = "Alice";                  // OK
form.email = "alice@example.com";     // OK
form.age = 25;                        // OK

// form.name = "";                    // Error: Name must not be empty
// form.email = "not-an-email";       // Error: Invalid email format
// form.age = -5;                     // Error: Age must be positive
```

---

> **Key Takeaway:** Stage 3 decorators are the standard going forward, using a context-object API that is cleaner and more powerful than the legacy three-argument pattern; use them for new code, and keep legacy decorators only for existing codebases that depend on parameter decorators or `emitDecoratorMetadata`.
