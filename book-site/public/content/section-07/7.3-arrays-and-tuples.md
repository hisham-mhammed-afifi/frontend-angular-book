# 7.3 Arrays and Tuples

> Arrays hold variable-length collections of one type. Tuples hold fixed-length collections where each position has its own type. Understanding both, along with readonly variants, is essential for writing safe TypeScript.

---

## Array Types

TypeScript provides two equivalent syntaxes for typed arrays.

### T[] Syntax

```typescript
const numbers: number[] = [1, 2, 3];
const names: string[] = ["Alice", "Bob"];
const flags: boolean[] = [true, false, true];
```

### Array\<T\> Generic Syntax

```typescript
const numbers: Array<number> = [1, 2, 3];
const names: Array<string> = ["Alice", "Bob"];
```

Both syntaxes are identical in behavior. TypeScript itself uses `T[]` in hover tooltips and error messages. Most codebases standardize on one. The `T[]` form is shorter and more common. The `Array<T>` form reads better for complex element types.

```typescript
// T[] can become hard to read with complex types
const handlers: (() => void)[] = [];         // array of functions
const pairs: [string, number][] = [];        // array of tuples
const mixed: (string | number)[] = [];       // array of union

// Array<T> is clearer for these cases
const handlers: Array<() => void> = [];
const pairs: Array<[string, number]> = [];
const mixed: Array<string | number> = [];
```

**Pick one style and be consistent.** The `@typescript-eslint/array-type` rule can enforce your choice.

### Type Inference with Arrays

```typescript
// TypeScript infers the element type from initial values
const scores = [90, 85, 92];          // number[]
const words = ["hello", "world"];     // string[]
const mixed = [1, "two", true];       // (string | number | boolean)[]

// Empty arrays need annotation (otherwise inferred as any[])
const items: string[] = [];
const cache: Map<string, number[]> = new Map();
```

### Multi-Dimensional Arrays

```typescript
const matrix: number[][] = [
  [1, 2, 3],
  [4, 5, 6],
  [7, 8, 9],
];

const grid: Array<Array<string>> = [
  ["X", "O", "X"],
  ["O", "X", "O"],
];

// 3D array
const cube: number[][][] = [[[1, 2], [3, 4]], [[5, 6], [7, 8]]];
```

### Typed Array Methods

Array methods carry types through the chain. TypeScript infers the return type automatically.

```typescript
const numbers = [1, 2, 3, 4, 5];

// map: number[] -> string[]
const strings = numbers.map((n) => String(n)); // string[]

// filter: number[] -> number[] (same type)
const evens = numbers.filter((n) => n % 2 === 0); // number[]

// reduce: number[] -> number
const sum = numbers.reduce((acc, n) => acc + n, 0); // number

// find: number[] -> number | undefined
const found = numbers.find((n) => n > 3); // number | undefined

// flatMap: produces a flat result
const nested = [[1, 2], [3, 4]];
const flat = nested.flatMap((arr) => arr); // number[]
```

### Narrowing with filter

A common gotcha: `filter` does not narrow the type by default.

```typescript
const values: (string | null)[] = ["hello", null, "world", null];

// BAD: filter returns (string | null)[] even though nulls are removed
const filtered = values.filter((v) => v !== null);
// type: (string | null)[]  <-- not what you expect

// GOOD: Use a type predicate to narrow
const filtered = values.filter((v): v is string => v !== null);
// type: string[]

// TypeScript 5.5+ can sometimes infer this automatically with:
const filtered = values.filter((v) => v !== null);
// type: string[] in TS 5.5+ (inferred narrowing)
```

## Readonly Arrays

A readonly array prevents mutations. You cannot `push`, `pop`, `splice`, or reassign elements.

### readonly T[]

```typescript
function printNames(names: readonly string[]): void {
  console.log(names.join(", "));
  // names.push("Charlie"); // Error: Property 'push' does not exist on type 'readonly string[]'
  // names[0] = "Dave";     // Error: Index signature in type 'readonly string[]' only permits reading
}

const people = ["Alice", "Bob"];
printNames(people); // OK: mutable array is assignable to readonly
```

### ReadonlyArray\<T\>

The generic equivalent.

```typescript
function sum(numbers: ReadonlyArray<number>): number {
  return numbers.reduce((acc, n) => acc + n, 0);
}
```

### Assignability Rules

A mutable array is assignable to a readonly array, but not the reverse.

```typescript
const mutable: number[] = [1, 2, 3];
const immutable: readonly number[] = mutable; // OK: widening to readonly

// const back: number[] = immutable; // Error: readonly -> mutable not allowed
// Workaround if you must:
const back: number[] = [...immutable]; // Create a new mutable copy
```

This is intentional. If readonly arrays could be assigned to mutable arrays, the readonly guarantee would be meaningless (someone could mutate through the mutable reference).

## Tuple Types

A tuple is a fixed-length array where each position has a specific type.

### Basic Tuples

```typescript
// A tuple with exactly two elements: string and number
let person: [string, number] = ["Alice", 30];

// Accessing elements
const name: string = person[0]; // string
const age: number = person[1];  // number

// Error: out of bounds
// const x = person[2]; // Error with noUncheckedIndexedAccess, or type is string | number

// Error: wrong types
// person = [30, "Alice"]; // Error: number not assignable to string at index 0
// person = ["Alice"];     // Error: source has 1 element, target requires 2
```

### Tuples vs Arrays

| Feature | Array (number[]) | Tuple ([number, string]) |
|---------|-----------------|-------------------------|
| Length | Variable | Fixed |
| Element types | All same | Each position different |
| Use case | Collections | Structured records |

### Destructuring Tuples

Tuples pair naturally with destructuring.

```typescript
type Coordinate = [number, number];

function getPosition(): Coordinate {
  return [40.7128, -74.006];
}

const [lat, lng] = getPosition();
// lat: number, lng: number

// With three elements
type RGB = [number, number, number];
const [r, g, b]: RGB = [255, 128, 0];
```

### Labeled Tuples

Labels add documentation to tuple positions. They appear in IDE tooltips and error messages but do not affect type compatibility.

```typescript
type Coordinate = [lat: number, lng: number];
type HTTPResponse = [status: number, body: string];
type Range = [start: number, end: number];

function parseRange(input: string): Range {
  const [start, end] = input.split("-").map(Number);
  return [start, end];
}

const [start, end] = parseRange("10-20");
// Hovering shows: start: number, end: number (from labels)
```

**Rule:** If one element has a label, all elements must have labels.

```typescript
// OK
type A = [name: string, age: number];

// Error: Tuple members must all have names or all not have names
// type B = [name: string, number];
```

### Optional Tuple Elements

```typescript
type FlexPoint = [number, number, number?];

const point2D: FlexPoint = [10, 20];        // OK, z is optional
const point3D: FlexPoint = [10, 20, 30];    // OK
// const bad: FlexPoint = [10];              // Error: need at least 2

// Length is 2 | 3
const len: 2 | 3 = point2D.length;
```

With labels:

```typescript
type FlexPoint = [x: number, y: number, z?: number];
```

### Rest Elements in Tuples

Tuples can have a rest element for variable-length trailing (or leading/middle) portions.

```typescript
// Fixed prefix, variable rest
type LogEntry = [timestamp: Date, level: string, ...messages: string[]];

const entry: LogEntry = [new Date(), "ERROR", "Disk full", "Retry in 5s"];
const short: LogEntry = [new Date(), "INFO"];  // zero rest elements is OK

// Fixed prefix and suffix
type Padded = [first: string, ...middle: number[], last: string];
const p: Padded = ["start", 1, 2, 3, "end"];

// Leading rest
type Tail = [...head: string[], tail: number];
const t: Tail = ["a", "b", "c", 42];
```

### Readonly Tuples

```typescript
type Point = readonly [number, number];

const origin: Point = [0, 0];
// origin[0] = 5; // Error: Cannot assign to '0' because it is a read-only property

// as const creates readonly tuples
const rgb = [255, 128, 0] as const;
// type: readonly [255, 128, 0]
// Both readonly AND literal types
```

### Tuple Inference

Without annotation, TypeScript infers arrays, not tuples.

```typescript
// Inferred as number[], not [number, number]
const point = [10, 20]; // number[]

// Force tuple inference:

// 1. Type annotation
const point1: [number, number] = [10, 20];

// 2. as const (makes it readonly with literal types)
const point2 = [10, 20] as const; // readonly [10, 20]

// 3. satisfies (preserves literal types while checking)
const point3 = [10, 20] satisfies [number, number]; // [number, number]

// 4. Helper function
function tuple<T extends unknown[]>(...args: T): T {
  return args;
}
const point4 = tuple(10, 20); // [number, number]
```

## as const with Arrays and Tuples

`as const` deeply freezes the type: arrays become readonly tuples, values become literal types.

### Arrays to Readonly Tuples

```typescript
// Without as const
const colors = ["red", "green", "blue"];
// type: string[]

// With as const
const colors = ["red", "green", "blue"] as const;
// type: readonly ["red", "green", "blue"]

// Each element is a literal type
type FirstColor = (typeof colors)[0]; // "red"
type AllColors = (typeof colors)[number]; // "red" | "green" | "blue"
```

### Deriving Union Types from Arrays

A powerful pattern: define values once, derive the type from them.

```typescript
const STATUS_CODES = ["active", "inactive", "pending", "archived"] as const;
type StatusCode = (typeof STATUS_CODES)[number];
// "active" | "inactive" | "pending" | "archived"

function isValidStatus(value: string): value is StatusCode {
  return (STATUS_CODES as readonly string[]).includes(value);
}

const input = "active";
if (isValidStatus(input)) {
  console.log(input); // type: StatusCode
}
```

### Nested Objects with as const

```typescript
const ROUTES = {
  home: "/",
  about: "/about",
  users: {
    list: "/users",
    detail: "/users/:id",
  },
} as const;

type Route = typeof ROUTES;
// {
//   readonly home: "/";
//   readonly about: "/about";
//   readonly users: {
//     readonly list: "/users";
//     readonly detail: "/users/:id";
//   };
// }
```

## Working with Array Types at the Type Level

### Extracting Element Types

```typescript
type Numbers = number[];
type Elem = Numbers[number]; // number

type Tuple = [string, number, boolean];
type First = Tuple[0];    // string
type Second = Tuple[1];   // number
type All = Tuple[number]; // string | number | boolean

// Extract the type of elements from any array/tuple
type ElementOf<T extends readonly unknown[]> = T[number];

const items = ["a", 1, true] as const;
type Item = ElementOf<typeof items>; // "a" | 1 | true
```

### Spreading Tuples in Function Parameters

```typescript
type Args = [string, number, boolean];

function example(...args: Args) {
  const [name, age, active] = args;
  // name: string, age: number, active: boolean
}

example("Alice", 30, true);

// Spread a tuple as arguments
const myArgs: Args = ["Bob", 25, false];
example(...myArgs);
```

### Variadic Tuple Types

Variadic tuples let you write generic functions that operate on tuple types of any length.

```typescript
// Concatenate two tuples
type Concat<A extends unknown[], B extends unknown[]> = [...A, ...B];

type Result = Concat<[string, number], [boolean]>;
// [string, number, boolean]

// Prepend an element
type Prepend<T, Arr extends unknown[]> = [T, ...Arr];
type WithId = Prepend<number, [string, boolean]>;
// [number, string, boolean]

// Practical: type-safe function that concatenates arrays
function concat<A extends unknown[], B extends unknown[]>(
  a: [...A],
  b: [...B]
): [...A, ...B] {
  return [...a, ...b];
}

const result = concat([1, "two"] as [number, string], [true] as [boolean]);
// type: [number, string, boolean]
// value: [1, "two", true]
```

## Common Patterns

### Typed Map and Filter Chains

```typescript
interface User {
  id: number;
  name: string;
  email: string | null;
}

const users: User[] = [
  { id: 1, name: "Alice", email: "alice@example.com" },
  { id: 2, name: "Bob", email: null },
  { id: 3, name: "Charlie", email: "charlie@example.com" },
];

// Chain preserves types through each step
const emailList: string[] = users
  .map((u) => u.email)                          // (string | null)[]
  .filter((email): email is string => email !== null); // string[]
```

### Fixed-Length Arrays via Tuples

```typescript
// Enforce exactly 3 elements
type RGB = [r: number, g: number, b: number];

function setColor(color: RGB): void {
  const [r, g, b] = color;
  console.log(`rgb(${r}, ${g}, ${b})`);
}

setColor([255, 128, 0]);     // OK
// setColor([255, 128]);     // Error: Source has 2 element(s) but target requires 3
// setColor([255, 128, 0, 1]); // Error: Source has 4 element(s) but target allows only 3
```

### Return Multiple Values

Tuples are the idiomatic way to return multiple values from a function.

```typescript
function divide(a: number, b: number): [quotient: number, remainder: number] {
  return [Math.floor(a / b), a % b];
}

const [q, r] = divide(17, 5);
// q: number (quotient), r: number (remainder)

// React-style useState pattern
function useState<T>(initial: T): [T, (value: T) => void] {
  let state = initial;
  const setState = (value: T) => { state = value; };
  return [state, setState];
}

const [count, setCount] = useState(0);
// count: number, setCount: (value: number) => void
```

### Key-Value Pairs

```typescript
type Entry<K, V> = [key: K, value: V];

function toEntries<T extends Record<string, unknown>>(
  obj: T
): Array<Entry<keyof T & string, T[keyof T]>> {
  return Object.entries(obj) as Array<Entry<keyof T & string, T[keyof T]>>;
}

const entries = toEntries({ name: "Alice", age: 30 });
// Array<[key: "name" | "age", value: string | number]>
```

## Common Mistakes

- **Forgetting that TypeScript infers arrays, not tuples.** Writing `const point = [10, 20]` gives `number[]`, not `[number, number]`. If you need a tuple, annotate explicitly or use `as const`.

```typescript
// BAD: inferred as number[]
const point = [10, 20];
point.push(30); // OK (not what you want for a coordinate)

// GOOD: explicitly a tuple
const point: [number, number] = [10, 20];
// point.push(30); // Still allowed at runtime! Tuples don't prevent push.
// Use readonly for true safety:
const point: readonly [number, number] = [10, 20];
```

- **Assuming tuples prevent push/pop.** TypeScript tuples are checked at assignment time, but `push()` and `pop()` still exist on the type because tuples extend `Array`. Only `readonly` tuples remove mutating methods.

```typescript
const pair: [string, number] = ["age", 30];
pair.push("extra"); // No compile error! Length is now 3.
// pair[2] is "extra" at runtime but TypeScript thinks pair.length is 2

// Fix: use readonly
const pair: readonly [string, number] = ["age", 30];
// pair.push("extra"); // Error: Property 'push' does not exist
```

- **Using `as const` when you need mutable data.** `as const` makes everything readonly and literal. If you later need to modify the array, you will get errors everywhere.

```typescript
const items = ["a", "b", "c"] as const;
// items.push("d"); // Error: push does not exist on readonly

// If you need mutability, use a type annotation instead:
const items: string[] = ["a", "b", "c"];
```

- **Not handling `undefined` from array access with `noUncheckedIndexedAccess`.** With this flag enabled (recommended), accessing `arr[i]` returns `T | undefined`. You must narrow before using.

```typescript
const nums: number[] = [1, 2, 3];

// With noUncheckedIndexedAccess:
const first = nums[0]; // number | undefined

// BAD: using without check
// first.toFixed(2); // Error: possibly undefined

// GOOD: narrow first
if (first !== undefined) {
  first.toFixed(2); // OK
}

// Or use non-null assertion if you are certain
const first = nums[0]!; // number (dangerous if array is empty)
```

- **Readonly tuple not assignable to mutable parameter.** A `readonly [number, number]` is not assignable to `[number, number]`. Functions that accept tuple parameters should use `readonly` if they do not mutate.

```typescript
function distance(a: [number, number], b: [number, number]): number {
  return Math.sqrt((a[0] - b[0]) ** 2 + (a[1] - b[1]) ** 2);
}

const p = [3, 4] as const; // readonly [3, 4]
// distance(p, [0, 0]); // Error: readonly [3, 4] is not assignable to [number, number]

// Fix: accept readonly tuples
function distance(
  a: readonly [number, number],
  b: readonly [number, number]
): number {
  return Math.sqrt((a[0] - b[0]) ** 2 + (a[1] - b[1]) ** 2);
}
distance(p, [0, 0]); // OK
```

## Best Practices

- **Use `readonly` in function parameters.** If a function does not mutate an array argument, type it as `readonly T[]` or `readonly [...]`. This accepts both mutable and immutable inputs and communicates that the function is safe.

```typescript
// GOOD: accepts both mutable and readonly arrays
function sum(numbers: readonly number[]): number {
  return numbers.reduce((acc, n) => acc + n, 0);
}
```

- **Prefer tuples for fixed-structure returns.** When a function returns two or three related values, a tuple with labels is cleaner than creating a one-off interface.

- **Derive union types from `as const` arrays.** Define values once as a runtime array, then extract the union type. This keeps types and values in sync automatically.

```typescript
const ROLES = ["admin", "editor", "viewer"] as const;
type Role = (typeof ROLES)[number]; // "admin" | "editor" | "viewer"
```

- **Use `satisfies` to check tuple types without widening.** The `satisfies` operator verifies the type while keeping the inferred literal types intact.

```typescript
const point = [10, 20] satisfies [number, number];
// type: [number, number] (not widened to number[])
// and TypeScript verifies it matches the shape
```

- **Be consistent with array syntax.** Pick `T[]` or `Array<T>` for your project and stick with it. Most teams use `T[]` for simple types and `Array<T>` for complex element types.

## Real-World Use Cases

### 1. Type-Safe CSV Parser

```typescript
type CSVRow = [id: string, name: string, email: string, role: string];

function parseCSV(raw: string): CSVRow[] {
  return raw
    .trim()
    .split("\n")
    .slice(1) // skip header
    .map((line) => {
      const parts = line.split(",");
      if (parts.length !== 4) {
        throw new Error(`Expected 4 columns, got ${parts.length}`);
      }
      return parts as unknown as CSVRow;
    });
}

const csv = `id,name,email,role
1,Alice,alice@example.com,admin
2,Bob,bob@example.com,editor`;

const rows = parseCSV(csv);
for (const [id, name, email, role] of rows) {
  console.log(`${name} (${role}): ${email}`);
}
```

### 2. Coordinate System with Readonly Tuples

```typescript
type Point2D = readonly [x: number, y: number];
type Point3D = readonly [x: number, y: number, z: number];

function translate(point: Point2D, dx: number, dy: number): Point2D {
  return [point[0] + dx, point[1] + dy];
}

function distance(a: Point2D, b: Point2D): number {
  return Math.sqrt((a[0] - b[0]) ** 2 + (a[1] - b[1]) ** 2);
}

function midpoint(a: Point2D, b: Point2D): Point2D {
  return [(a[0] + b[0]) / 2, (a[1] + b[1]) / 2];
}

const origin: Point2D = [0, 0];
const target: Point2D = [3, 4];

console.log(distance(origin, target));             // 5
console.log(translate(target, 10, 10));            // [13, 14]
console.log(midpoint(origin, target));             // [1.5, 2]

// Works with as const values too
const p = [5, 12] as const;
console.log(distance(origin, p)); // OK: readonly [5, 12] assignable to readonly [number, number]
```

### 3. Type-Safe Event System with Tuple Arguments

```typescript
type EventMap = {
  login: [userId: number, timestamp: Date];
  logout: [userId: number];
  purchase: [userId: number, itemId: string, amount: number];
  error: [code: number, message: string];
};

class TypedEmitter<Events extends Record<string, unknown[]>> {
  private listeners = new Map<string, Array<(...args: any[]) => void>>();

  on<K extends keyof Events & string>(
    event: K,
    listener: (...args: Events[K]) => void
  ): void {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, []);
    }
    this.listeners.get(event)!.push(listener);
  }

  emit<K extends keyof Events & string>(
    event: K,
    ...args: Events[K]
  ): void {
    const handlers = this.listeners.get(event) || [];
    for (const handler of handlers) {
      handler(...args);
    }
  }
}

const emitter = new TypedEmitter<EventMap>();

// Fully typed: listener params match the tuple definition
emitter.on("login", (userId, timestamp) => {
  // userId: number, timestamp: Date
  console.log(`User ${userId} logged in at ${timestamp.toISOString()}`);
});

emitter.on("purchase", (userId, itemId, amount) => {
  // userId: number, itemId: string, amount: number
  console.log(`User ${userId} bought ${itemId} for $${amount}`);
});

// Fully typed: emit args must match
emitter.emit("login", 42, new Date());
emitter.emit("purchase", 42, "item-123", 29.99);

// Error: Argument of type 'string' is not assignable to parameter of type 'number'
// emitter.emit("login", "not-a-number", new Date());
```

---

> **Key Takeaway:** Use arrays (`T[]`) for variable-length same-type collections and tuples (`[T1, T2, ...]`) for fixed-length multi-type structures. Always use `readonly` in function parameters that do not mutate, and leverage `as const` to derive union types from constant arrays while getting readonly tuple inference for free.
