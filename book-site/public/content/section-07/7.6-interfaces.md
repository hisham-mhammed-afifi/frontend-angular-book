# 7.6 Interfaces

> Interfaces define the shape of objects, function signatures, and class contracts. They are TypeScript's primary tool for structural typing and one of the most-used features in any TypeScript codebase.

---

## What Is an Interface?

An interface is a named type that describes the structure an object must have. TypeScript uses structural typing (also called "duck typing"), meaning any object that has the required shape satisfies the interface, regardless of whether it explicitly declares that it "implements" it.

```typescript
interface User {
  id: number;
  name: string;
  email: string;
}

// This object satisfies User because it has the right shape
const alice: User = {
  id: 1,
  name: "Alice",
  email: "alice@example.com",
};

// This also works - structural typing does not require an explicit relationship
function greet(user: User) {
  console.log(`Hello, ${user.name}`);
}

const bob = { id: 2, name: "Bob", email: "bob@example.com", role: "admin" };
greet(bob); // Works - bob has all required User properties (extra properties are fine here)
```

## Defining Interfaces

### Property Signatures

```typescript
interface Product {
  id: number;
  name: string;
  price: number;
  description: string;
}
```

Properties are separated by semicolons (`;`), commas (`,`), or newlines. Semicolons are the convention for interfaces.

### Optional Properties

Mark a property as optional with `?`. Its type becomes `T | undefined`.

```typescript
interface Config {
  host: string;
  port: number;
  ssl?: boolean;        // boolean | undefined
  timeout?: number;     // number | undefined
}

const config: Config = {
  host: "localhost",
  port: 3000,
  // ssl and timeout are not required
};
```

### Readonly Properties

Use `readonly` to prevent reassignment after initialization.

```typescript
interface Point {
  readonly x: number;
  readonly y: number;
}

const origin: Point = { x: 0, y: 0 };
// origin.x = 5; // Error: Cannot assign to 'x' because it is a read-only property

// Note: readonly is shallow. Nested objects can still be mutated.
interface Config {
  readonly settings: {
    theme: string;
  };
}

const config: Config = { settings: { theme: "dark" } };
// config.settings = { theme: "light" }; // Error: cannot reassign
config.settings.theme = "light"; // OK - mutating a nested property is allowed
```

### Method Signatures

Interfaces can describe methods in two syntaxes:

```typescript
interface Logger {
  // Method shorthand syntax
  log(message: string): void;

  // Property with function type syntax
  warn: (message: string) => void;
}

// Both are satisfied the same way
const logger: Logger = {
  log(message) {
    console.log(message);
  },
  warn: (message) => {
    console.warn(message);
  },
};
```

The two syntaxes are nearly identical, but there is a subtle difference under `strictFunctionTypes`: the method shorthand (`log(...)`) is bivariant (less strict), while the property syntax (`warn: (...)`) is contravariant (stricter). In practice, the property syntax is safer.

```typescript
interface Handler {
  // Method shorthand - bivariant (less safe)
  handle(event: MouseEvent): void;
}

interface StrictHandler {
  // Property syntax - contravariant (safer)
  handle: (event: MouseEvent) => void;
}

function processEvent(event: Event) {
  console.log(event.type);
}

// With method shorthand, this is allowed (even though it is unsound)
const handler: Handler = { handle: processEvent }; // No error

// With property syntax, TypeScript catches the problem
// const strictHandler: StrictHandler = { handle: processEvent };
// Error: Type '(event: Event) => void' is not assignable to type '(event: MouseEvent) => void'
```

## Index Signatures

When you do not know all property names ahead of time, use an index signature to describe the shape of values accessed by dynamic keys.

### String Index Signatures

```typescript
interface StringMap {
  [key: string]: string;
}

const headers: StringMap = {
  "Content-Type": "application/json",
  Authorization: "Bearer token123",
};

// Any string key works
headers["X-Custom"] = "value";
```

### Number Index Signatures

```typescript
interface NumberMap {
  [index: number]: string;
}

const list: NumberMap = {
  0: "first",
  1: "second",
  2: "third",
};

// Accessed like an array
console.log(list[0]); // "first"
```

### Combining Fixed and Dynamic Properties

```typescript
interface ApiResponse {
  status: number;
  message: string;
  [key: string]: unknown; // allows additional dynamic properties
}

const response: ApiResponse = {
  status: 200,
  message: "OK",
  data: [1, 2, 3],
  timestamp: Date.now(),
};
```

Important: when you have an index signature, all explicitly declared properties must have types compatible with the index signature's value type.

```typescript
interface Broken {
  [key: string]: number;
  // name: string; // Error: 'string' is not assignable to 'number'
  count: number; // OK - number is compatible
}
```

### `Record` as an Alternative

For simple key-value mappings, the `Record` utility type is often cleaner:

```typescript
// These are roughly equivalent
interface StringMap {
  [key: string]: string;
}

type StringMap = Record<string, string>;
```

## Extending Interfaces

### Single Inheritance

Use `extends` to create a new interface that inherits all members of a parent.

```typescript
interface Animal {
  name: string;
  age: number;
}

interface Dog extends Animal {
  breed: string;
  bark(): void;
}

const rex: Dog = {
  name: "Rex",
  age: 5,
  breed: "German Shepherd",
  bark() {
    console.log("Woof!");
  },
};
```

### Multiple Inheritance

An interface can extend multiple interfaces.

```typescript
interface Serializable {
  serialize(): string;
}

interface Loggable {
  log(): void;
}

interface Timestamped {
  createdAt: Date;
  updatedAt: Date;
}

// Combines all three
interface AuditableRecord extends Serializable, Loggable, Timestamped {
  id: string;
}

const record: AuditableRecord = {
  id: "abc-123",
  createdAt: new Date(),
  updatedAt: new Date(),
  serialize() {
    return JSON.stringify(this);
  },
  log() {
    console.log(`Record ${this.id}`);
  },
};
```

### Narrowing Inherited Properties

A child interface can narrow (but not widen) a parent property's type.

```typescript
interface Base {
  id: string | number;
  status: string;
}

interface Derived extends Base {
  id: string; // OK - string is narrower than string | number
  // id: boolean; // Error - boolean is not assignable to string | number
  status: "active" | "inactive"; // OK - union literal is narrower than string
}
```

### Extending Interfaces with Conflicting Properties

If two parent interfaces declare the same property with incompatible types, the child interface will produce an error.

```typescript
interface A {
  value: string;
}

interface B {
  value: number;
}

// Error: Interface 'C' cannot simultaneously extend types 'A' and 'B'.
// Named property 'value' of types 'A' and 'B' are not identical.
// interface C extends A, B {}

// Fix: override the property in the child
interface C extends A, B {
  value: string & number; // never, effectively - not useful
}

// Better fix: redesign so the types are compatible
interface A2 {
  value: string | number;
}

interface B2 {
  value: string | number;
}

interface C2 extends A2, B2 {
  value: string; // narrow to string, compatible with both parents
}
```

## Declaration Merging

Interfaces with the same name in the same scope are automatically merged. This is a feature unique to interfaces (type aliases cannot do this).

### Basic Merging

```typescript
interface Window {
  title: string;
}

interface Window {
  appVersion: string;
}

// The merged Window has both properties
const w: Window = {
  title: "My App",
  appVersion: "1.0.0",
};
```

### Merging Rules

1. Non-function members must have identical types if they share a name.
2. Function members with the same name become overloads.
3. Later declarations take higher priority for overload ordering.

```typescript
interface Box {
  height: number;
  width: number;
}

interface Box {
  height: number; // OK - same type
  // height: string; // Error - conflicts with the first declaration
  scale: number; // New property added
}

// Merged result: { height: number; width: number; scale: number; }
```

### Function Overload Merging

```typescript
interface Document {
  createElement(tagName: "div"): HTMLDivElement;
  createElement(tagName: "span"): HTMLSpanElement;
}

interface Document {
  createElement(tagName: "canvas"): HTMLCanvasElement;
  createElement(tagName: string): HTMLElement;
}

// Merged overloads (later declaration's members come first):
// createElement(tagName: "canvas"): HTMLCanvasElement;
// createElement(tagName: string): HTMLElement;
// createElement(tagName: "div"): HTMLDivElement;
// createElement(tagName: "span"): HTMLSpanElement;

// Exception: overloads with string literal parameters always go to the top
```

### Practical Use: Augmenting Third-Party Types

Declaration merging is most useful for extending types from libraries without modifying the original source.

```typescript
// Augment the global Window interface
declare global {
  interface Window {
    analytics: {
      track(event: string, data?: Record<string, unknown>): void;
    };
  }
}

// Now TypeScript knows about window.analytics
window.analytics.track("page_view", { page: "/home" });
```

```typescript
// Augment Express Request
declare module "express" {
  interface Request {
    userId?: string;
    sessionId?: string;
  }
}

// Now req.userId is available in Express handlers
```

## Callable and Constructable Interfaces

### Call Signatures

An interface can describe something that is callable (a function).

```typescript
interface Formatter {
  (input: string): string;
}

const uppercase: Formatter = (input) => input.toUpperCase();
const trim: Formatter = (input) => input.trim();

console.log(uppercase("hello")); // "HELLO"
```

### Construct Signatures

An interface can describe something that is constructable (a class constructor).

```typescript
interface ClockConstructor {
  new (hour: number, minute: number): ClockInstance;
}

interface ClockInstance {
  tick(): void;
}

function createClock(Ctor: ClockConstructor, h: number, m: number): ClockInstance {
  return new Ctor(h, m);
}
```

### Hybrid Types: Callable with Properties

Some JavaScript patterns produce objects that are both callable and have properties (like jQuery's `$`).

```typescript
interface Counter {
  // Call signature
  (start: number): string;
  // Properties
  interval: number;
  reset(): void;
}

function createCounter(): Counter {
  const counter = ((start: number) => {
    return `Started at ${start}`;
  }) as Counter;

  counter.interval = 1000;
  counter.reset = () => {
    console.log("Reset");
  };

  return counter;
}

const c = createCounter();
c(10);        // callable
c.interval;   // has properties
c.reset();    // has methods
```

## Generic Interfaces

### Basic Generic Interface

```typescript
interface Container<T> {
  value: T;
  getValue(): T;
  setValue(value: T): void;
}

const numberBox: Container<number> = {
  value: 42,
  getValue() {
    return this.value;
  },
  setValue(value) {
    this.value = value;
  },
};

const stringBox: Container<string> = {
  value: "hello",
  getValue() {
    return this.value;
  },
  setValue(value) {
    this.value = value;
  },
};
```

### Multiple Type Parameters

```typescript
interface KeyValuePair<K, V> {
  key: K;
  value: V;
}

const entry: KeyValuePair<string, number> = {
  key: "age",
  value: 30,
};
```

### Generic Constraints

```typescript
interface Repository<T extends { id: string }> {
  getById(id: string): T | undefined;
  getAll(): T[];
  save(entity: T): void;
  delete(id: string): boolean;
}

interface User {
  id: string;
  name: string;
  email: string;
}

// Works - User has an id: string property
const userRepo: Repository<User> = {
  getById(id) { /* ... */ return undefined; },
  getAll() { /* ... */ return []; },
  save(entity) { /* ... */ },
  delete(id) { /* ... */ return true; },
};

// Error - number does not have an id: string property
// const numberRepo: Repository<number> = { ... };
```

### Default Type Parameters

```typescript
interface ApiResponse<T = unknown> {
  data: T;
  status: number;
  timestamp: Date;
}

// T defaults to unknown if not specified
const generic: ApiResponse = {
  data: "anything",
  status: 200,
  timestamp: new Date(),
};

// Or specify explicitly
const typed: ApiResponse<User[]> = {
  data: [{ id: "1", name: "Alice", email: "alice@example.com" }],
  status: 200,
  timestamp: new Date(),
};
```

## Interfaces with Classes

### Implementing an Interface

A class can declare that it satisfies an interface using `implements`.

```typescript
interface Drawable {
  x: number;
  y: number;
  draw(): void;
}

class Circle implements Drawable {
  x: number;
  y: number;
  radius: number;

  constructor(x: number, y: number, radius: number) {
    this.x = x;
    this.y = y;
    this.radius = radius;
  }

  draw() {
    console.log(`Drawing circle at (${this.x}, ${this.y}) with radius ${this.radius}`);
  }
}
```

### Implementing Multiple Interfaces

```typescript
interface Serializable {
  serialize(): string;
}

interface Printable {
  print(): void;
}

class Report implements Serializable, Printable {
  constructor(private title: string, private content: string) {}

  serialize() {
    return JSON.stringify({ title: this.title, content: this.content });
  }

  print() {
    console.log(`${this.title}\n${this.content}`);
  }
}
```

### Important: `implements` Is a Check, Not a Source of Types

A common misconception is that `implements` provides type information to the class. It does not. It only checks that the class satisfies the interface.

```typescript
interface Checkable {
  check(name: string): boolean;
}

class NameChecker implements Checkable {
  // Bug: parameter 's' is not typed by the interface
  check(s) {
    // Error: Parameter 's' implicitly has an 'any' type
    return s.toLowerCase() === "ok";
  }
}
```

You still need to annotate parameters in the class method. The interface only verifies the final shape.

```typescript
class NameChecker implements Checkable {
  check(s: string) {
    // Now properly typed
    return s.toLowerCase() === "ok";
  }
}
```

## Excess Property Checking

TypeScript applies special rules when assigning object literals directly to an interface-typed variable.

### Fresh Object Literals Are Checked Strictly

```typescript
interface User {
  name: string;
  age: number;
}

// Direct assignment: excess property checking kicks in
const user: User = {
  name: "Alice",
  age: 30,
  email: "alice@example.com",
  // Error: Object literal may only specify known properties,
  // and 'email' does not exist in type 'User'
};
```

### Intermediate Variables Bypass the Check

```typescript
const data = {
  name: "Alice",
  age: 30,
  email: "alice@example.com",
};

// No error - excess property checking only applies to fresh object literals
const user: User = data;
```

This is by design. TypeScript assumes that if you store an object in a variable first, you might use those extra properties elsewhere, so it does not complain. But with a fresh literal, extra properties are almost always a typo.

### Bypassing with Type Assertion (Use Sparingly)

```typescript
const user = {
  name: "Alice",
  age: 30,
  email: "alice@example.com",
} as User;
// No error, but you lose the safety check
```

### Function Arguments Follow the Same Rule

```typescript
function createUser(user: User) { /* ... */ }

// Error: excess property
createUser({ name: "Alice", age: 30, emai: "typo@example.com" });

// No error: intermediate variable
const userData = { name: "Alice", age: 30, emai: "typo@example.com" };
createUser(userData); // the typo goes undetected
```

This is one of the most important gotchas to understand. Excess property checking is not part of TypeScript's structural type system; it is a separate heuristic that only applies to fresh object literals.

## Interfaces vs Type Aliases

This is one of the most debated topics in TypeScript. Here is a clear comparison.

### What Only Interfaces Can Do

1. **Declaration merging:**

```typescript
interface User {
  name: string;
}

interface User {
  age: number;
}

// Merged: { name: string; age: number; }
```

Type aliases cannot merge. Redeclaring a type alias is an error.

2. **Faster type checking (extends vs intersection):**

```typescript
// Interface extends - cached by name, faster checks
interface Dog extends Animal {
  breed: string;
}

// Type intersection - evaluated inline each time
type Dog = Animal & {
  breed: string;
};
```

TypeScript caches interfaces by name in an internal registry. Intersection types are computed every time they are encountered.

### What Only Type Aliases Can Do

1. **Primitives, unions, tuples:**

```typescript
type ID = string | number;        // union
type Pair = [string, number];     // tuple
type Name = string;               // primitive alias
// None of these can be expressed as an interface
```

2. **Mapped types:**

```typescript
type Readonly<T> = {
  readonly [K in keyof T]: T[K];
};
// Cannot write a mapped type with interface syntax
```

3. **Conditional types:**

```typescript
type IsString<T> = T extends string ? true : false;
// Cannot express conditional types with interfaces
```

### Comparison Table

| Feature | Interface | Type Alias |
|---------|-----------|------------|
| Object shapes | Yes | Yes |
| Function types | Yes | Yes |
| Declaration merging | Yes | No |
| `extends` keyword | Yes | No (uses `&`) |
| `implements` with classes | Yes | Yes (object types only) |
| Unions | No | Yes |
| Tuples | No | Yes |
| Primitive aliases | No | Yes |
| Mapped types | No | Yes |
| Conditional types | No | Yes |
| Performance (named cache) | Better | Slightly slower |
| Error messages | Named reference | Often expanded inline |

### When to Use Which

**Use `interface` when:**
- Defining the shape of objects, especially for public APIs
- You want declaration merging (module augmentation)
- You need `extends` for inheritance hierarchies
- Performance matters in a large codebase (interfaces are cached)

**Use `type` when:**
- You need unions, intersections, tuples, or primitive aliases
- You need mapped or conditional types
- You want to ensure the type cannot be accidentally merged

**Team convention matters most.** Pick one as the default and be consistent. Both the TypeScript handbook and popular style guides lean toward "use `interface` for object shapes, `type` for everything else."

## Common Mistakes

### Mistake 1: Using `I` Prefix

The Hungarian notation convention of prefixing interfaces with `I` (like `IUser`) is a C# pattern that is not idiomatic in TypeScript.

```typescript
// Bad: C# convention
interface IUser {
  name: string;
}

// Good: just use the name
interface User {
  name: string;
}
```

The TypeScript compiler team, the official handbook, and most major open-source projects do not use the `I` prefix.

### Mistake 2: Assuming `implements` Provides Type Inference

```typescript
interface Handler {
  handle(data: string): void;
}

class MyHandler implements Handler {
  // BAD: expecting the parameter type to come from the interface
  handle(data) {
    // Error: Parameter 'data' implicitly has an 'any' type
    console.log(data);
  }
}

// GOOD: annotate the parameter yourself
class MyHandler implements Handler {
  handle(data: string) {
    console.log(data);
  }
}
```

### Mistake 3: Forgetting That Excess Checking Only Applies to Literals

```typescript
interface Options {
  color: string;
  width: number;
}

// This catches the typo
// const opts: Options = { colour: "red", width: 10 }; // Error

// But this does NOT
const raw = { colour: "red", width: 10 };
const opts: Options = raw;
// No error - "colour" typo is invisible
// AND "color" is undefined at runtime
```

If you want strict checking even with intermediate variables, use `satisfies`:

```typescript
const raw = { colour: "red", width: 10 } satisfies Options;
// Error: 'colour' does not exist in type 'Options'. Did you mean 'color'?
```

### Mistake 4: Readonly Does Not Mean Immutable

```typescript
interface Config {
  readonly database: {
    host: string;
    port: number;
  };
}

const config: Config = {
  database: { host: "localhost", port: 5432 },
};

// config.database = { host: "other", port: 3306 }; // Error: readonly
config.database.host = "other"; // No error - readonly is shallow
```

For deep immutability, you need a recursive `DeepReadonly` type or use `Object.freeze` at runtime.

### Mistake 5: Accidental Declaration Merging

```typescript
// file: user.ts
interface Config {
  apiUrl: string;
}

// ... hundreds of lines later, or in another file that gets merged ...
interface Config {
  timeout: number;
}

// Now Config requires BOTH apiUrl and timeout
// This can be very confusing if unintentional
const config: Config = {
  apiUrl: "https://api.example.com",
  // Error: Property 'timeout' is missing
};
```

If you do not want merging, use `type` instead. Type aliases will produce a "Duplicate identifier" error on redeclaration.

## Best Practices

### 1. Prefer `interface` for Object Shapes in Public APIs

```typescript
// Public API types - interface gives better error messages and performance
export interface UserCreateInput {
  name: string;
  email: string;
  role?: "admin" | "user";
}

export interface UserResponse {
  id: string;
  name: string;
  email: string;
  createdAt: string;
}
```

### 2. Use `readonly` for Properties That Should Not Change

```typescript
interface Event {
  readonly type: string;
  readonly timestamp: number;
  readonly payload: unknown;
}
```

### 3. Keep Interfaces Focused (Interface Segregation)

```typescript
// Bad: one large interface with many optional properties
interface User {
  id: string;
  name: string;
  email: string;
  address?: string;
  phone?: string;
  avatar?: string;
  preferences?: Record<string, unknown>;
  billingInfo?: BillingInfo;
  subscriptionTier?: string;
}

// Good: split into focused interfaces
interface UserIdentity {
  id: string;
  name: string;
  email: string;
}

interface UserContact {
  address?: string;
  phone?: string;
}

interface UserProfile {
  avatar?: string;
  preferences?: Record<string, unknown>;
}

interface UserBilling {
  billingInfo: BillingInfo;
  subscriptionTier: string;
}

// Compose as needed
interface FullUser extends UserIdentity, UserContact, UserProfile, UserBilling {}
```

### 4. Use Generic Interfaces for Reusable Containers

```typescript
interface Result<T, E = Error> {
  success: boolean;
  data?: T;
  error?: E;
}

function fetchUser(id: string): Promise<Result<User>> {
  // ...
}

function parseConfig(raw: string): Result<Config, string> {
  // error is a string message instead of Error object
}
```

### 5. Prefer Property Function Syntax for Strict Checking

```typescript
// Prefer this (strict under strictFunctionTypes)
interface EventHandler {
  onClick: (event: MouseEvent) => void;
}

// Over this (bivariant, less safe)
interface EventHandler {
  onClick(event: MouseEvent): void;
}
```

## Real-World Use Cases

### Repository Pattern

```typescript
interface Entity {
  id: string;
  createdAt: Date;
  updatedAt: Date;
}

interface Repository<T extends Entity> {
  findById(id: string): Promise<T | null>;
  findAll(filter?: Partial<T>): Promise<T[]>;
  create(data: Omit<T, "id" | "createdAt" | "updatedAt">): Promise<T>;
  update(id: string, data: Partial<Omit<T, "id">>): Promise<T>;
  delete(id: string): Promise<boolean>;
}

interface User extends Entity {
  name: string;
  email: string;
  role: "admin" | "user";
}

interface Product extends Entity {
  name: string;
  price: number;
  stock: number;
}

// Both UserRepository and ProductRepository share the same contract
class UserRepository implements Repository<User> {
  async findById(id: string): Promise<User | null> { /* ... */ return null; }
  async findAll(filter?: Partial<User>): Promise<User[]> { /* ... */ return []; }
  async create(data: Omit<User, "id" | "createdAt" | "updatedAt">): Promise<User> {
    return { id: crypto.randomUUID(), createdAt: new Date(), updatedAt: new Date(), ...data };
  }
  async update(id: string, data: Partial<Omit<User, "id">>): Promise<User> { /* ... */ return {} as User; }
  async delete(id: string): Promise<boolean> { /* ... */ return true; }
}
```

### Plugin System

```typescript
interface Plugin {
  name: string;
  version: string;
  initialize(context: PluginContext): void;
  destroy?(): void;
}

interface PluginContext {
  config: Record<string, unknown>;
  logger: Logger;
  registerHook(name: string, callback: (...args: unknown[]) => void): void;
}

interface Logger {
  info(message: string): void;
  warn(message: string): void;
  error(message: string, error?: Error): void;
}

class PluginManager {
  private plugins: Plugin[] = [];

  register(plugin: Plugin) {
    this.plugins.push(plugin);
  }

  initializeAll(context: PluginContext) {
    for (const plugin of this.plugins) {
      console.log(`Initializing plugin: ${plugin.name} v${plugin.version}`);
      plugin.initialize(context);
    }
  }

  destroyAll() {
    for (const plugin of this.plugins) {
      plugin.destroy?.();
    }
    this.plugins = [];
  }
}

// Any object with the right shape can be a Plugin
const analyticsPlugin: Plugin = {
  name: "analytics",
  version: "2.0.0",
  initialize(ctx) {
    ctx.registerHook("pageView", (url) => {
      ctx.logger.info(`Page viewed: ${url}`);

  },
```

### Strategy Pattern with Interfaces

```typescript
interface PricingStrategy {
  calculate(basePrice: number, quantity: number): number;
  description: string;
}

const standardPricing: PricingStrategy = {
  description: "Standard pricing with no discounts",
  calculate(basePrice, quantity) {
    return basePrice * quantity;
  },
};

const bulkPricing: PricingStrategy = {
  description: "10% off for orders over 100 units",
  calculate(basePrice, quantity) {
    const discount = quantity > 100 ? 0.9 : 1;
    return basePrice * quantity * discount;
  },
};

const tieredPricing: PricingStrategy = {
  description: "Price per unit decreases with volume",
  calculate(basePrice, quantity) {
    if (quantity > 1000) return basePrice * 0.7 * quantity;
    if (quantity > 100) return basePrice * 0.85 * quantity;
    return basePrice * quantity;
  },
};

class Order {
  constructor(
    private items: { price: number; quantity: number }[],
    private strategy: PricingStrategy,
  ) {}

  getTotal(): number {
    return this.items.reduce(
      (total, item) => total + this.strategy.calculate(item.price, item.quantity),
      0,
    );
  }

  describe(): string {
    return `Order total: $${this.getTotal().toFixed(2)} (${this.strategy.description})`;
  }
}
```

---

> **Key Takeaway:** Use interfaces to define object shapes, especially at API boundaries. Leverage `extends` for composition, declaration merging for augmenting third-party types, and generics for reusable contracts. Use type aliases when you need unions, tuples, or mapped/conditional types.
