# 8.3 Liskov Substitution Principle (LSP)

> If code works correctly with a base type, it must also work correctly with any subtype of that base type. Subtypes must be behaviorally compatible with their parent types.

---

## The Principle Defined

The Liskov Substitution Principle is the "L" in SOLID. It was introduced by Barbara Liskov in her 1987 keynote *Data Abstraction and Hierarchy* and later formalized with Jeannette Wing in 1994.

### The Formal Definition

> If for each object `o1` of type `S` there is an object `o2` of type `T` such that for all programs `P` defined in terms of `T`, the behavior of `P` is unchanged when `o1` is substituted for `o2`, then `S` is a subtype of `T`.

### The Practical Translation

Any place in your code that accepts a value of type `T` should work correctly if given an instance of any subtype `S` of `T`. The subtype must honor the **behavioral contract** of the parent type, not just its structural signature.

```typescript
// If this function works with Bird, it must also work with
// every subtype of Bird without surprises.
function makeBirdFly(bird: Bird): void {
  bird.fly(); // This must work for ALL birds passed in.
}
```

If you create a `Penguin` subclass of `Bird` whose `fly()` method throws an error, you have violated LSP. The function's contract says "give me a Bird and I will make it fly." A Penguin breaks that contract.

## The Classic Violation: Rectangle and Square

This is the most famous LSP example. Mathematically, a square **is** a rectangle. But in object-oriented programming, a `Square` subclass of `Rectangle` violates LSP.

### The Naive Implementation

```typescript
class Rectangle {
  constructor(
    protected _width: number,
    protected _height: number
  ) {}

  get width(): number { return this._width; }
  set width(value: number) { this._width = value; }

  get height(): number { return this._height; }
  set height(value: number) { this._height = value; }

  area(): number {
    return this._width * this._height;
  }
}

class Square extends Rectangle {
  constructor(side: number) {
    super(side, side);
  }

  // Override to maintain the square invariant
  set width(value: number) {
    this._width = value;
    this._height = value; // Must keep sides equal
  }

  set height(value: number) {
    this._width = value; // Must keep sides equal
    this._height = value;
  }
}
```

### Why This Is a Violation

Consider code written against the `Rectangle` contract:

```typescript
function doubleWidth(rect: Rectangle): void {
  const originalHeight = rect.height;
  rect.width = rect.width * 2;

  // This assertion is part of Rectangle's behavioral contract:
  // setting width should not change height.
  console.assert(
    rect.height === originalHeight,
    `Expected height ${originalHeight}, got ${rect.height}`
  );
}

const rect = new Rectangle(5, 10);
doubleWidth(rect); // OK: width is 10, height is still 10

const square = new Square(5);
doubleWidth(square); // FAILS: width set to 10, but height also became 10
// Assertion fails: expected height 5, got 10
```

The `Rectangle` class has an implicit contract: setting `width` does not affect `height` and vice versa. `Square` violates this contract by coupling them. Any code that relies on the independence of width and height breaks when given a `Square`.

### The Fix

Do not force an inheritance relationship where the behaviors diverge. Use a shared abstraction instead:

```typescript
interface Shape {
  area(): number;
}

class Rectangle implements Shape {
  constructor(
    private _width: number,
    private _height: number
  ) {}

  get width(): number { return this._width; }
  set width(value: number) { this._width = value; }

  get height(): number { return this._height; }
  set height(value: number) { this._height = value; }

  area(): number {
    return this._width * this._height;
  }
}

class Square implements Shape {
  constructor(private _side: number) {}

  get side(): number { return this._side; }
  set side(value: number) { this._side = value; }

  area(): number {
    return this._side * this._side;
  }
}
```

Now `Rectangle` and `Square` are siblings, not parent and child. Code that needs a generic shape uses the `Shape` interface. Code that needs to manipulate width and height independently uses `Rectangle` directly. No substitution problem.

## The Rules of LSP

Barbara Liskov and Jeannette Wing formalized LSP using concepts from Design by Contract. These rules define what "behavioral compatibility" means:

### 1. Preconditions Cannot Be Strengthened

A subtype must accept **at least** everything the parent accepts. It cannot demand more.

```typescript
class BaseParser {
  // Accepts any string
  parse(input: string): object {
    return JSON.parse(input);
  }
}

// Bad Practice: strengthened precondition (rejects inputs the parent accepts)
class StrictParser extends BaseParser {
  parse(input: string): object {
    if (input.length > 1000) {
      throw new Error("Input too long"); // Parent has no such restriction
    }
    return JSON.parse(input);
  }
}
```

Code that uses `BaseParser` passes strings of any length. Substituting `StrictParser` breaks that code when the string is over 1000 characters. The subtype added a restriction the parent did not have.

```typescript
// Good: same or weaker preconditions
class LenientParser extends BaseParser {
  parse(input: string): object {
    // Accepts everything the parent accepts, plus handles empty strings
    if (input.trim().length === 0) {
      return {};
    }
    return JSON.parse(input);
  }
}
```

### 2. Postconditions Cannot Be Weakened

A subtype must deliver **at least** everything the parent promises. It cannot deliver less.

```typescript
interface Repository<T> {
  // Contract: returns the saved entity with a generated id
  save(entity: Omit<T, "id">): T;
}

class InMemoryRepository implements Repository<User> {
  private items: User[] = [];

  save(entity: Omit<User, "id">): User {
    const user: User = { ...entity, id: crypto.randomUUID() };
    this.items.push(user);
    return user; // Fulfills the contract: returns entity with id
  }
}

// Bad Practice: weakened postcondition
class BrokenRepository implements Repository<User> {
  save(entity: Omit<User, "id">): User {
    // Returns the entity WITHOUT generating an id
    return { ...entity, id: "" } as User; // Empty id violates the contract
  }
}
```

Code that calls `save()` expects a valid `id` on the returned entity. `BrokenRepository` breaks that expectation.

### 3. Invariants Must Be Preserved

Properties that are always true for the parent must remain true for the subtype.

```typescript
class BankAccount {
  protected _balance: number;

  constructor(initialBalance: number) {
    if (initialBalance < 0) throw new Error("Initial balance cannot be negative");
    this._balance = initialBalance;
  }

  // Invariant: balance is always >= 0
  get balance(): number { return this._balance; }

  withdraw(amount: number): void {
    if (amount > this._balance) {
      throw new Error("Insufficient funds");
    }
    this._balance -= amount;
  }

  deposit(amount: number): void {
    if (amount <= 0) throw new Error("Deposit must be positive");
    this._balance += amount;
  }
}

// Bad Practice: violates the non-negative balance invariant
class OverdraftAccount extends BankAccount {
  withdraw(amount: number): void {
    // Allows balance to go negative, breaking the parent's invariant
    this._balance -= amount;
  }
}
```

Any code that relies on `balance >= 0` (e.g., displaying account status, calculating interest) breaks with `OverdraftAccount`.

The fix: if overdraft behavior is needed, the invariant should be part of the contract from the start, or overdraft accounts should be a separate type with a clearly different contract.

### 4. No New Exceptions

A subtype should not throw exceptions that the parent does not throw (unless they are subtypes of exceptions the parent already throws).

```typescript
interface FileReader {
  // Contract: may throw FileNotFoundError
  read(path: string): string;
}

class LocalFileReader implements FileReader {
  read(path: string): string {
    if (!fs.existsSync(path)) {
      throw new FileNotFoundError(path);
    }
    return fs.readFileSync(path, "utf-8");
  }
}

// Bad Practice: throws a new kind of error the parent's contract doesn't mention
class NetworkFileReader implements FileReader {
  read(path: string): string {
    // Throws NetworkTimeoutError, which callers are not prepared for
    throw new NetworkTimeoutError("Connection timed out");
  }
}
```

Callers of `FileReader.read()` catch `FileNotFoundError`. They are not prepared for `NetworkTimeoutError`. If `NetworkFileReader` needs to throw network errors, the `FileReader` interface contract should be broadened, or `NetworkTimeoutError` should extend an error type the parent already declares.

### Summary Table

| Rule | Parent | Subtype |
|------|--------|---------|
| Preconditions | Require X | Must accept X or more (weaker) |
| Postconditions | Promise Y | Must deliver Y or more (stronger) |
| Invariants | Property P always holds | P must still always hold |
| Exceptions | Throws E | May only throw E or subtypes of E |

## Covariance and Contravariance in LSP

These terms describe how type relationships in the parent map to type relationships in the subtype.

### Covariant Return Types (Allowed)

A subtype can return a **more specific** type than the parent:

```typescript
class AnimalShelter {
  adopt(): Animal {
    return new Animal("generic");
  }
}

class CatShelter extends AnimalShelter {
  // Returns Cat, which is a subtype of Animal. This is covariant (more specific).
  adopt(): Cat {
    return new Cat("whiskers");
  }
}

// Safe: code expecting Animal gets a Cat, which is an Animal.
const shelter: AnimalShelter = new CatShelter();
const pet: Animal = shelter.adopt(); // Works fine
```

### Contravariant Parameters (The Ideal)

A subtype should accept **the same or broader** parameter types than the parent:

```typescript
class EventHandler {
  handle(event: MouseEvent): void {
    console.log(event.clientX);
  }
}

// Contravariant: accepts a broader type (Event instead of MouseEvent)
class BroadEventHandler extends EventHandler {
  handle(event: Event): void {
    console.log(event.type);
  }
}

// Safe: code passes MouseEvent, BroadEventHandler accepts any Event
```

**Note:** TypeScript's method parameter checking is **bivariant** by default for methods (accepts both broader and narrower types), which can mask LSP violations. Strict function types (`strictFunctionTypes: true`) enforce proper contravariance for function types (but not method syntax).

## Recognizing LSP Violations

### Sign 1: Type Checking the Subtype

If you find yourself checking `instanceof` to decide which subtype you have, LSP is likely violated:

```typescript
// Bad Practice: checking the subtype defeats polymorphism
function processPayment(processor: PaymentProcessor): void {
  if (processor instanceof CryptoProcessor) {
    // Special handling because CryptoProcessor behaves differently
    processor.connectToBlockchain();
  }
  processor.process();
}
```

If the subtype needs special handling, it does not truly substitute for the parent.

### Sign 2: Methods That Throw "Not Supported"

```typescript
// Bad Practice: subtype refuses to fulfill the contract
class ReadOnlyRepository extends Repository {
  save(entity: unknown): void {
    throw new Error("Not supported: this repository is read-only");
  }

  delete(id: string): void {
    throw new Error("Not supported: this repository is read-only");
  }
}
```

Code expecting a `Repository` will call `save()` and `delete()`. `ReadOnlyRepository` breaks at runtime. The fix is to split the interface (see ISP in 8.4):

```typescript
interface Readable<T> {
  find(id: string): T | null;
  findAll(): T[];
}

interface Writable<T> {
  save(entity: T): void;
  delete(id: string): void;
}

interface Repository<T> extends Readable<T>, Writable<T> {}

// Now a read-only store only implements Readable
class ReadOnlyStore<T> implements Readable<T> {
  find(id: string): T | null { /* ... */ return null; }
  findAll(): T[] { /* ... */ return []; }
  // No save() or delete() to violate
}
```

### Sign 3: Overriding Methods with Empty Bodies

```typescript
// Bad Practice: neutering inherited behavior
class Bird {
  fly(): void {
    console.log("Flying...");
  }
}

class Ostrich extends Bird {
  fly(): void {
    // Do nothing. Ostriches can't fly.
  }
}
```

Code that calls `bird.fly()` and expects the bird to actually fly gets an ostrich that silently does nothing. This is a subtle violation: the postcondition (the bird flies) is weakened to nothing.

## Common Mistakes

### 1. Modeling "Is-A" Relationships Based on Real-World Taxonomy

In the real world, a square is a rectangle and a penguin is a bird. But in software, subtyping is about **behavioral compatibility**, not taxonomic classification. Ask "Can I substitute this in all code that uses the parent?" not "Is this a kind of that in the real world?"

### 2. Using Inheritance for Code Reuse Alone

```typescript
// Bad Practice: inheriting just to reuse the logging method
class Logger {
  log(message: string): void {
    console.log(`[LOG] ${message}`);
  }
}

class UserService extends Logger {
  createUser(name: string): void {
    this.log(`Creating user: ${name}`);
    // ...
  }
}

// A UserService is NOT a Logger. You would never pass a UserService
// where a Logger is expected. This is inheritance for reuse, not substitution.
```

Use composition instead:

```typescript
class UserService {
  constructor(private readonly logger: Logger) {}

  createUser(name: string): void {
    this.logger.log(`Creating user: ${name}`);
  }
}
```

### 3. Weakening Error Handling in Subtypes

```typescript
// Parent guarantees: returns a non-empty array or throws
class DataFetcher {
  async fetch(url: string): Promise<string[]> {
    const response = await fetch(url);
    const data = await response.json();
    if (!Array.isArray(data) || data.length === 0) {
      throw new Error("No data found");
    }
    return data;
  }
}

// Bad Practice: subtype silently returns empty array instead of throwing
class CachedFetcher extends DataFetcher {
  async fetch(url: string): Promise<string[]> {
    try {
      return await super.fetch(url);
    } catch {
      return []; // Weakened postcondition: parent promises non-empty or error
    }
  }
}
```

Callers expect either data or an exception. Silently returning an empty array changes the contract.

### 4. Forgetting That LSP Applies to Interfaces, Not Just Classes

LSP is not limited to class inheritance. Any type that claims to implement an interface must honor that interface's full behavioral contract:

```typescript
interface Cache {
  get(key: string): string | null;
  set(key: string, value: string, ttlMs: number): void;
}

// Bad Practice: ignores the TTL, violating the contract
class InfiniteCache implements Cache {
  private store = new Map<string, string>();

  get(key: string): string | null {
    return this.store.get(key) ?? null;
  }

  set(key: string, value: string, ttlMs: number): void {
    this.store.set(key, value);
    // ttlMs is ignored. Items never expire.
    // Code that relies on TTL-based eviction will break.
  }
}
```

### 5. Confusing Structural Compatibility with Behavioral Compatibility

TypeScript uses structural typing: if two types have the same shape, they are compatible. But structural compatibility does not guarantee behavioral compatibility:

```typescript
interface Stack<T> {
  push(item: T): void;
  pop(): T | undefined;
  peek(): T | undefined;
}

// Structurally compatible, behaviorally wrong
class Queue<T> implements Stack<T> {
  private items: T[] = [];

  push(item: T): void { this.items.push(item); }
  pop(): T | undefined { return this.items.shift(); } // FIFO, not LIFO!
  peek(): T | undefined { return this.items[0]; }      // First, not last!
}
```

A `Queue` has the same method signatures as a `Stack`, but its behavior is fundamentally different (FIFO vs LIFO). Substituting a `Queue` where a `Stack` is expected produces incorrect results, even though TypeScript's type checker accepts it.

## Best Practices

### 1. Design by Contract

Think about what each method **requires** (preconditions) and what it **promises** (postconditions). Document these contracts. Ensure every subtype honors them:

```typescript
interface Sorter<T> {
  /**
   * Sorts the given array in ascending order.
   *
   * Precondition: items is a non-null array.
   * Postcondition: returned array contains exactly the same elements, in ascending order.
   * Postcondition: the original array is not modified.
   */
  sort(items: T[], compareFn: (a: T, b: T) => number): T[];
}
```

### 2. Prefer Composition Over Inheritance

Inheritance creates tight coupling and makes LSP violations easy. Composition lets you share behavior without the substitution constraint:

```typescript
// Instead of: class CachedRepository extends DatabaseRepository
// Use: class CachedRepository that wraps a Repository
class CachedRepository<T> implements Repository<T> {
  constructor(
    private readonly inner: Repository<T>,
    private readonly cache: Map<string, T> = new Map()
  ) {}

  async find(id: string): Promise<T | null> {
    if (this.cache.has(id)) return this.cache.get(id)!;
    const result = await this.inner.find(id);
    if (result) this.cache.set(id, result);
    return result;
  }

  async save(entity: T & { id: string }): Promise<void> {
    await this.inner.save(entity);
    this.cache.set(entity.id, entity);
  }
}
```

### 3. Use Interface Segregation to Prevent LSP Violations

Many LSP violations come from subtypes that cannot fulfill the full interface. Splitting the interface (ISP) removes the problem:

```typescript
// Instead of one fat interface that forces "not supported" implementations
interface FullFileSystem {
  read(path: string): string;
  write(path: string, content: string): void;
  delete(path: string): void;
}

// Split into capabilities
interface ReadableFS { read(path: string): string; }
interface WritableFS { write(path: string, content: string): void; }
interface DeletableFS { delete(path: string): void; }

// Each implementation only claims what it can actually do
class S3ReadOnlyFS implements ReadableFS {
  read(path: string): string { /* ... */ return ""; }
}

class LocalFS implements ReadableFS, WritableFS, DeletableFS {
  read(path: string): string { /* ... */ return ""; }
  write(path: string, content: string): void { /* ... */ }
  delete(path: string): void { /* ... */ }
}
```

### 4. Write Substitution Tests

For each subtype, write tests that exercise the parent's contract:

```typescript
// Shared test suite that any Sorter must pass
function testSorterContract(createSorter: () => Sorter<number>): void {
  test("returns same elements", () => {
    const sorter = createSorter();
    const input = [3, 1, 4, 1, 5];
    const result = sorter.sort(input, (a, b) => a - b);
    expect(result.sort()).toEqual(input.sort());
  });

  test("does not modify original array", () => {
    const sorter = createSorter();
    const input = [3, 1, 4];
    const copy = [...input];
    sorter.sort(input, (a, b) => a - b);
    expect(input).toEqual(copy);
  });

  test("returns elements in ascending order", () => {
    const sorter = createSorter();
    const result = sorter.sort([5, 2, 8, 1], (a, b) => a - b);
    for (let i = 1; i < result.length; i++) {
      expect(result[i]).toBeGreaterThanOrEqual(result[i - 1]);
    }
  });
}

// Run the same tests for every implementation
testSorterContract(() => new QuickSorter());
testSorterContract(() => new MergeSorter());
testSorterContract(() => new HeapSorter());
```

### 5. Ask the Substitution Question Before Inheriting

Before writing `class B extends A`, ask: "Can I put a `B` everywhere an `A` is used, and will all existing tests still pass?" If the answer is no, do not use inheritance.

## Real-World Use Cases

### 1. Payment Processing with Substitutable Processors

```typescript
interface PaymentResult {
  success: boolean;
  transactionId: string;
  message: string;
}

interface PaymentProcessor {
  /**
   * Precondition: amount > 0, currency is a valid ISO 4217 code
   * Postcondition: returns PaymentResult with a non-empty transactionId on success
   * Postcondition: returns PaymentResult with success=false and a message on failure
   * Never throws for business-level failures (declined, insufficient funds)
   * May throw for infrastructure failures (network down)
   */
  charge(amount: number, currency: string, token: string): Promise<PaymentResult>;
}

class StripeProcessor implements PaymentProcessor {
  async charge(amount: number, currency: string, token: string): Promise<PaymentResult> {
    const response = await stripe.charges.create({ amount, currency, source: token });
    return {
      success: response.status === "succeeded",
      transactionId: response.id,
      message: response.status === "succeeded" ? "Payment successful" : response.failure_message,
    };
  }
}

class PayPalProcessor implements PaymentProcessor {
  async charge(amount: number, currency: string, token: string): Promise<PaymentResult> {
    const response = await paypal.execute(token, { amount, currency });
    return {
      success: response.state === "approved",
      transactionId: response.id,
      message: response.state === "approved" ? "Payment successful" : "Payment declined",
    };
  }
}

// Any PaymentProcessor can be substituted here
class CheckoutService {
  constructor(private readonly processor: PaymentProcessor) {}

  async processOrder(order: Order, paymentToken: string): Promise<void> {
    const result = await this.processor.charge(order.total, order.currency, paymentToken);

    if (!result.success) {
      throw new PaymentFailedError(result.message);
    }

    await this.markOrderPaid(order.id, result.transactionId);
  }
}
```

`StripeProcessor` and `PayPalProcessor` are fully substitutable. The `CheckoutService` works identically regardless of which is injected.

### 2. Logging with Substitutable Transports

```typescript
interface LogEntry {
  level: "debug" | "info" | "warn" | "error";
  message: string;
  timestamp: Date;
  metadata?: Record<string, unknown>;
}

interface LogTransport {
  /**
   * Precondition: entry is a valid LogEntry with non-empty message
   * Postcondition: the entry is persisted/transmitted (best effort)
   * Invariant: write() never throws (swallows internal errors)
   */
  write(entry: LogEntry): void;
}

class ConsoleTransport implements LogTransport {
  write(entry: LogEntry): void {
    try {
      const line = `[${entry.timestamp.toISOString()}] ${entry.level}: ${entry.message}`;
      console.log(line);
    } catch {
      // Contract: never throw
    }
  }
}

class FileTransport implements LogTransport {
  constructor(private readonly filePath: string) {}

  write(entry: LogEntry): void {
    try {
      const line = JSON.stringify(entry) + "\n";
      fs.appendFileSync(this.filePath, line);
    } catch {
      // Contract: never throw
    }
  }
}

class HttpTransport implements LogTransport {
  constructor(private readonly endpoint: string) {}

  write(entry: LogEntry): void {
    try {
      // Fire and forget
      fetch(this.endpoint, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(entry),
      }).catch(() => {});
    } catch {
      // Contract: never throw
    }
  }
}

// All transports are substitutable. The logger works the same regardless.
class Logger {
  constructor(private readonly transports: LogTransport[]) {}

  log(level: LogEntry["level"], message: string, metadata?: Record<string, unknown>): void {
    const entry: LogEntry = { level, message, timestamp: new Date(), metadata };
    for (const transport of this.transports) {
      transport.write(entry);
    }
  }
}
```

### 3. Collection Types with Shared Contract Tests

```typescript
interface Collection<T> {
  add(item: T): void;
  remove(item: T): boolean;
  contains(item: T): boolean;
  size(): number;
  toArray(): T[];
}

class ArrayCollection<T> implements Collection<T> {
  private items: T[] = [];

  add(item: T): void { this.items.push(item); }
  remove(item: T): boolean {
    const idx = this.items.indexOf(item);
    if (idx === -1) return false;
    this.items.splice(idx, 1);
    return true;
  }
  contains(item: T): boolean { return this.items.includes(item); }
  size(): number { return this.items.length; }
  toArray(): T[] { return [...this.items]; }
}

class SetCollection<T> implements Collection<T> {
  private items = new Set<T>();

  add(item: T): void { this.items.add(item); }
  remove(item: T): boolean { return this.items.delete(item); }
  contains(item: T): boolean { return this.items.has(item); }
  size(): number { return this.items.size; }
  toArray(): T[] { return [...this.items]; }
}

// Any Collection<T> works here. The function does not care about the implementation.
function removeDuplicatesFromCollection<T>(
  source: Collection<T>,
  target: Collection<T>
): void {
  for (const item of source.toArray()) {
    if (!target.contains(item)) {
      target.add(item);
    }
  }
}
```

Both `ArrayCollection` and `SetCollection` honor the same behavioral contract. They can be substituted freely in any function that accepts `Collection<T>`.

---

> **Key Takeaway:** LSP is about behavioral compatibility, not structural compatibility or real-world taxonomy; a subtype must honor the parent's full contract (preconditions, postconditions, invariants, and exception behavior) so that code written against the parent works correctly with any subtype.
