# 8.5 Dependency Inversion Principle (DIP)

> High-level modules should not depend on low-level modules; both should depend on abstractions. This is the principle that turns rigid, tightly coupled systems into flexible, testable architectures.

---

## The Two Rules

Robert C. Martin introduced the Dependency Inversion Principle in his 1996 paper. It consists of two formal rules:

**Rule A:** High-level modules should not depend upon low-level modules. Both should depend upon abstractions.

**Rule B:** Abstractions should not depend upon details. Details should depend upon abstractions.

These two rules work together. Rule A tells you where to draw the dependency boundary. Rule B tells you which direction the dependency should point.

### What "High-Level" and "Low-Level" Mean

High-level modules contain business logic and policy decisions. They describe *what* your application does. Low-level modules contain implementation details. They describe *how* things get done.

| Layer | Examples | Concern |
|-------|----------|---------|
| High-level | Order processing, user authentication, pricing rules | Business policy |
| Mid-level | Data validation, formatting, caching | Coordination |
| Low-level | Database queries, HTTP calls, file system access, email sending | Infrastructure |

The traditional dependency structure looks like this:

```
OrderService -> MySQLDatabase
ReportGenerator -> PDFLibrary
NotificationService -> SendGridAPI
```

Every arrow represents a compile-time dependency. If `MySQLDatabase` changes its API, `OrderService` must change too. The high-level module is at the mercy of the low-level module.

### The "Inversion"

The word "inversion" refers to reversing the traditional direction of dependencies. Instead of high-level modules depending downward on low-level modules, you introduce an abstraction that the high-level module *owns*, and the low-level module implements.

```
OrderService -> OrderRepository (interface)
                     ^
                     |
              MySQLOrderRepository (implements)
```

Now `OrderService` depends on an abstraction it controls. `MySQLOrderRepository` also depends on that abstraction (by implementing it). Both depend on the abstraction. Neither depends on the other. The dependency direction has been *inverted* from the traditional flow.

## The Violation: Direct Dependencies

Here is a typical violation. A `UserService` directly instantiates and uses a concrete database class:

```typescript
// Low-level module
class MySQLDatabase {
  connect(): void {
    // MySQL-specific connection logic
  }

  query(sql: string): any[] {
    // MySQL-specific query execution
    return [];
  }
}

// High-level module - VIOLATES DIP
class UserService {
  private database: MySQLDatabase;

  constructor() {
    this.database = new MySQLDatabase(); // Direct dependency
  }

  getUser(id: string): any {
    return this.database.query(`SELECT * FROM users WHERE id = '${id}'`);
  }

  saveUser(user: any): void {
    this.database.query(
      `INSERT INTO users (name, email) VALUES ('${user.name}', '${user.email}')`
    );
  }
}
```

Problems with this code:

1. **Tight coupling.** `UserService` cannot work without `MySQLDatabase`. Switching to PostgreSQL means rewriting `UserService`.
2. **Untestable.** Unit testing `UserService` requires a real MySQL connection. You cannot substitute a fake database.
3. **Hidden dependency.** The constructor creates its own dependency internally. The caller has no idea that a database is involved or any ability to configure it.
4. **Change propagation.** If `MySQLDatabase` renames `query` to `execute`, every class that uses it must change.

## Applying DIP: Depend on Abstractions

The fix introduces an interface that the high-level module defines, and the low-level module implements:

```typescript
// Abstraction - owned by the high-level layer
interface UserRepository {
  findById(id: string): User | null;
  save(user: User): void;
}

interface User {
  id: string;
  name: string;
  email: string;
}

// Low-level module - implements the abstraction
class MySQLUserRepository implements UserRepository {
  findById(id: string): User | null {
    // MySQL-specific query
    const rows = this.executeQuery(
      `SELECT * FROM users WHERE id = ?`, [id]
    );
    return rows.length > 0 ? rows[0] : null;
  }

  save(user: User): void {
    this.executeQuery(
      `INSERT INTO users (id, name, email) VALUES (?, ?, ?)`,
      [user.id, user.name, user.email]
    );
  }

  private executeQuery(sql: string, params: string[]): any[] {
    // MySQL-specific implementation
    return [];
  }
}

// High-level module - depends only on the abstraction
class UserService {
  constructor(private readonly userRepo: UserRepository) {}

  getUser(id: string): User | null {
    return this.userRepo.findById(id);
  }

  createUser(name: string, email: string): User {
    const user: User = {
      id: crypto.randomUUID(),
      name,
      email,
    };
    this.userRepo.save(user);
    return user;
  }
}
```

Now you can swap implementations without touching `UserService`:

```typescript
class PostgresUserRepository implements UserRepository {
  findById(id: string): User | null {
    // PostgreSQL-specific query
    return null;
  }

  save(user: User): void {
    // PostgreSQL-specific insert
  }
}

class InMemoryUserRepository implements UserRepository {
  private users: Map<string, User> = new Map();

  findById(id: string): User | null {
    return this.users.get(id) ?? null;
  }

  save(user: User): void {
    this.users.set(user.id, user);
  }
}

// Production
const prodService = new UserService(new PostgresUserRepository());

// Testing
const testService = new UserService(new InMemoryUserRepository());
```

## Who Owns the Abstraction?

This is the most misunderstood part of DIP. The abstraction (interface) should be owned by the *high-level* module, not the low-level module.

### Bad: Low-Level Module Owns the Interface

```typescript
// database/mysql-repository.ts
export interface MySQLRepositoryInterface {
  connect(host: string, port: number): void;
  query(sql: string): any[];
  disconnect(): void;
}

export class MySQLRepository implements MySQLRepositoryInterface {
  connect(host: string, port: number): void { /* ... */ }
  query(sql: string): any[] { return []; }
  disconnect(): void { /* ... */ }
}
```

This interface is shaped by MySQL's concerns (`connect`, `query`, `sql`). If `UserService` depends on `MySQLRepositoryInterface`, it is still coupled to MySQL's vocabulary and mental model. Switching to a REST API would require a completely different interface. This is not true inversion.

### Good: High-Level Module Owns the Interface

```typescript
// domain/user-repository.ts
export interface UserRepository {
  findById(id: string): User | null;
  findByEmail(email: string): User | null;
  save(user: User): void;
  delete(id: string): void;
}

// infrastructure/mysql-user-repository.ts
import { UserRepository, User } from '../domain/user-repository';

export class MySQLUserRepository implements UserRepository {
  findById(id: string): User | null { /* translate to SQL */ return null; }
  findByEmail(email: string): User | null { /* translate to SQL */ return null; }
  save(user: User): void { /* translate to SQL */ }
  delete(id: string): void { /* translate to SQL */ }
}
```

The interface speaks the language of the domain (`findById`, `findByEmail`) not the language of the infrastructure (`query`, `connect`). The high-level module defines what it needs. The low-level module adapts to that contract.

### The Directory Structure Reflects Ownership

```
src/
  domain/
    user.ts                    # User type
    user-repository.ts         # Interface (owned here)
    user-service.ts            # High-level module
  infrastructure/
    mysql-user-repository.ts   # Implements domain interface
    postgres-user-repository.ts
    in-memory-user-repository.ts
```

Dependencies point inward: `infrastructure` depends on `domain`. Never the reverse.

## DIP Beyond Classes: Functions and Closures

DIP does not require classes or interfaces. In JavaScript and TypeScript, functions and closures achieve the same decoupling.

### Higher-Order Functions as Dependency Inversion

A function that accepts its dependencies as parameters is applying DIP:

```typescript
// The "abstraction" is a function signature
type FetchUser = (id: string) => Promise<User | null>;
type SaveUser = (user: User) => Promise<void>;

// High-level function - depends on abstractions (function types)
async function createUserAccount(
  name: string,
  email: string,
  fetchUser: FetchUser,
  saveUser: SaveUser,
): Promise<User> {
  const existing = await fetchUser(email);
  if (existing) {
    throw new Error('User already exists');
  }

  const user: User = { id: crypto.randomUUID(), name, email };
  await saveUser(user);
  return user;
}
```

The low-level details are provided at the call site:

```typescript
// MySQL implementation
const mysqlFetchUser: FetchUser = async (id) => {
  const row = await pool.query('SELECT * FROM users WHERE id = ?', [id]);
  return row ?? null;
};

const mysqlSaveUser: SaveUser = async (user) => {
  await pool.query('INSERT INTO users VALUES (?, ?, ?)',
    [user.id, user.name, user.email]);
};

// Usage
await createUserAccount('Alice', 'alice@test.com', mysqlFetchUser, mysqlSaveUser);

// Test usage
const fakeUsers = new Map<string, User>();

const fakeFetchUser: FetchUser = async (id) => fakeUsers.get(id) ?? null;
const fakeSaveUser: SaveUser = async (user) => { fakeUsers.set(user.id, user); };

await createUserAccount('Bob', 'bob@test.com', fakeFetchUser, fakeSaveUser);
```

### Factory Functions with Closures

A common pattern is a factory that closes over its dependencies:

```typescript
type Logger = (message: string) => void;
type HttpClient = (url: string, options?: RequestInit) => Promise<Response>;

function createApiClient(httpClient: HttpClient, logger: Logger) {
  return {
    async getUser(id: string): Promise<User> {
      logger(`Fetching user ${id}`);
      const response = await httpClient(`/api/users/${id}`);
      return response.json();
    },

    async updateUser(id: string, data: Partial<User>): Promise<User> {
      logger(`Updating user ${id}`);
      const response = await httpClient(`/api/users/${id}`, {
        method: 'PUT',
        body: JSON.stringify(data),
      });
      return response.json();
    },
  };
}

// Production
const apiClient = createApiClient(fetch, console.log);

// Test
const mockHttp: HttpClient = async () =>
  new Response(JSON.stringify({ id: '1', name: 'Test' }));
const silentLogger: Logger = () => {};
const testClient = createApiClient(mockHttp, silentLogger);
```

### Callback Parameters

Even simple callbacks apply DIP. The caller decides the implementation:

```typescript
function processItems<T>(
  items: T[],
  onSuccess: (item: T) => void,
  onError: (item: T, error: Error) => void,
): void {
  for (const item of items) {
    try {
      // processing logic
      onSuccess(item);
    } catch (error) {
      onError(item, error instanceof Error ? error : new Error(String(error)));
    }
  }
}

// The caller provides the concrete behavior
processItems(
  records,
  (item) => database.insert(item),   // could be anything
  (item, err) => logger.error(err),  // could be anything
);
```

## DIP, DI, and IoC: Three Distinct Concepts

These three terms are frequently confused. They are related but not interchangeable.

| Concept | What It Is | Level |
|---------|-----------|-------|
| Dependency Inversion Principle (DIP) | A design principle: depend on abstractions, not concretions | Architectural guideline |
| Dependency Injection (DI) | A technique: pass dependencies in from outside instead of creating them internally | Implementation pattern |
| Inversion of Control (IoC) | A broad principle: framework calls your code, not the other way around | Design philosophy |

### DIP Without DI

You can follow DIP without using dependency injection. Any pattern that introduces an abstraction between high and low-level modules satisfies DIP:

```typescript
// Using a service locator (satisfies DIP, but not DI)
class OrderService {
  processOrder(orderId: string): void {
    // Depends on abstraction (PaymentProcessor), not concrete implementation
    const processor = ServiceLocator.get<PaymentProcessor>('PaymentProcessor');
    processor.charge(orderId, 100);
  }
}
```

This is not dependency injection (the dependency is fetched, not injected), but it does satisfy DIP because `OrderService` depends on the `PaymentProcessor` abstraction, not a concrete `StripeProcessor`.

### DI Without DIP

You can use dependency injection while violating DIP:

```typescript
// Bad: DI used, but still depending on concrete types
class ReportService {
  // The dependency is injected, but it is a concrete class, not an abstraction
  constructor(private pdfGenerator: PDFKitGenerator) {}

  generateReport(data: ReportData): Buffer {
    return this.pdfGenerator.createPDF(data);
  }
}

// The dependency is injected from outside
const service = new ReportService(new PDFKitGenerator());
```

This uses DI (the dependency is passed in), but violates DIP because `ReportService` depends on the concrete `PDFKitGenerator` class, not an abstraction.

### All Three Together

When properly applied, DIP guides the architecture, DI provides the wiring mechanism, and IoC manages the lifecycle:

```typescript
// DIP: Abstraction defined by the high-level module
interface ReportGenerator {
  generate(data: ReportData): Buffer;
}

// Detail implements abstraction
class PDFReportGenerator implements ReportGenerator {
  generate(data: ReportData): Buffer {
    // PDF-specific logic
    return Buffer.from([]);
  }
}

// DI: Dependency passed from outside
class ReportService {
  constructor(private generator: ReportGenerator) {}

  createReport(data: ReportData): Buffer {
    return this.generator.generate(data);
  }
}

// IoC: A container manages creation and wiring
container.register<ReportGenerator>('ReportGenerator', PDFReportGenerator);
container.register('ReportService', (c) =>
  new ReportService(c.resolve<ReportGenerator>('ReportGenerator'))
);
```

## Rule B: Abstractions Should Not Depend on Details

Rule B is often overlooked. It says the abstraction itself must not leak implementation details.

### Violation: Leaky Abstraction

```typescript
// Bad: The interface leaks SQL details
interface UserRepository {
  findBySQL(query: string): User[];           // Leaks SQL
  save(user: User, tableName: string): void;  // Leaks table structure
  beginTransaction(): void;                   // Leaks database concept
  commit(): void;                             // Leaks database concept
}
```

This interface is shaped by a relational database. If you implement it with a REST API or an in-memory store, those methods make no sense. The abstraction depends on the details of a specific implementation.

### Fixed: Detail-Free Abstraction

```typescript
// Good: The interface speaks domain language
interface UserRepository {
  findById(id: string): User | null;
  findByEmail(email: string): User | null;
  save(user: User): void;
  delete(id: string): boolean;
}
```

Each implementation translates from domain concepts to its own specific mechanism:

```typescript
class SQLUserRepository implements UserRepository {
  // Translates domain operations to SQL internally
  findById(id: string): User | null {
    const result = this.db.query('SELECT * FROM users WHERE id = ?', [id]);
    return result ? this.mapToUser(result) : null;
  }
  // ... other methods handle transactions internally
}

class RestApiUserRepository implements UserRepository {
  // Translates domain operations to HTTP calls internally
  findById(id: string): User | null {
    const response = this.httpClient.get(`/users/${id}`);
    return response.ok ? response.json() : null;
  }
}
```

## Abstractions at Different Scales

DIP applies at every architectural layer, not just at the class level.

### Module Level

```typescript
// notification-service.ts (high-level module)
// Defines what it needs:
export interface MessageTransport {
  send(to: string, subject: string, body: string): Promise<boolean>;
}

export function createNotificationService(transport: MessageTransport) {
  return {
    async notifyUser(userId: string, message: string): Promise<void> {
      const user = await getUser(userId);
      await transport.send(user.email, 'Notification', message);
    },
  };
}

// email-transport.ts (low-level module, depends on high-level's interface)
import type { MessageTransport } from './notification-service';

export class SendGridTransport implements MessageTransport {
  async send(to: string, subject: string, body: string): Promise<boolean> {
    // SendGrid-specific API calls
    return true;
  }
}

// sms-transport.ts (another low-level module)
import type { MessageTransport } from './notification-service';

export class TwilioSmsTransport implements MessageTransport {
  async send(to: string, subject: string, body: string): Promise<boolean> {
    // Twilio-specific API calls
    return true;
  }
}
```

### Application Architecture Level

DIP is the foundation of layered architectures like Clean Architecture, Hexagonal Architecture (Ports and Adapters), and Onion Architecture. In all of these, the dependency rule states that dependencies point inward, toward the domain:

```
┌─────────────────────────────────────────────┐
│              Infrastructure                  │
│  (Database, HTTP, File System, Email)        │
│                                              │
│   ┌─────────────────────────────────────┐   │
│   │         Application Layer            │   │
│   │   (Use Cases, Orchestration)         │   │
│   │                                      │   │
│   │   ┌─────────────────────────────┐   │   │
│   │   │       Domain Layer           │   │   │
│   │   │  (Entities, Interfaces,      │   │   │
│   │   │   Business Rules)            │   │   │
│   │   └─────────────────────────────┘   │   │
│   │                                      │   │
│   └─────────────────────────────────────┘   │
│                                              │
└─────────────────────────────────────────────┘

Dependencies always point INWARD.
The domain never depends on infrastructure.
Infrastructure implements domain interfaces.
```

```typescript
// domain/ports.ts - the domain defines its needs
export interface OrderRepository {
  findById(id: string): Promise<Order | null>;
  save(order: Order): Promise<void>;
}

export interface PaymentGateway {
  charge(amount: number, currency: string, token: string): Promise<PaymentResult>;
  refund(paymentId: string): Promise<RefundResult>;
}

export interface EmailSender {
  sendOrderConfirmation(order: Order, recipient: string): Promise<void>;
}

// domain/order-service.ts - uses only domain-defined abstractions
export class OrderService {
  constructor(
    private orders: OrderRepository,
    private payments: PaymentGateway,
    private emails: EmailSender,
  ) {}

  async placeOrder(items: OrderItem[], paymentToken: string): Promise<Order> {
    const order = Order.create(items);
    const payment = await this.payments.charge(
      order.total, 'USD', paymentToken
    );
    order.markPaid(payment.id);
    await this.orders.save(order);
    await this.emails.sendOrderConfirmation(order, order.customerEmail);
    return order;
  }
}

// infrastructure/stripe-payment-gateway.ts
import type { PaymentGateway } from '../domain/ports';

export class StripePaymentGateway implements PaymentGateway {
  async charge(amount: number, currency: string, token: string) {
    // Stripe SDK calls
    return { id: 'pay_123', status: 'succeeded' as const };
  }

  async refund(paymentId: string) {
    // Stripe SDK calls
    return { id: 'ref_123', status: 'succeeded' as const };
  }
}
```

## Testing Benefits

DIP directly enables unit testing. When your high-level module depends on an abstraction, you substitute a test double without touching the module's code.

```typescript
// Test doubles are trivial to create
class StubOrderRepository implements OrderRepository {
  private stored: Order[] = [];

  async findById(id: string): Promise<Order | null> {
    return this.stored.find(o => o.id === id) ?? null;
  }

  async save(order: Order): Promise<void> {
    this.stored.push(order);
  }

  // Test helper - not part of the interface
  getStored(): Order[] {
    return [...this.stored];
  }
}

class StubPaymentGateway implements PaymentGateway {
  shouldFail = false;

  async charge(amount: number, currency: string, token: string) {
    if (this.shouldFail) {
      throw new Error('Payment declined');
    }
    return { id: 'pay_test', status: 'succeeded' as const };
  }

  async refund(paymentId: string) {
    return { id: 'ref_test', status: 'succeeded' as const };
  }
}

class SpyEmailSender implements EmailSender {
  sentEmails: Array<{ order: Order; recipient: string }> = [];

  async sendOrderConfirmation(order: Order, recipient: string): Promise<void> {
    this.sentEmails.push({ order, recipient });
  }
}

// Tests
describe('OrderService', () => {
  let orderRepo: StubOrderRepository;
  let payments: StubPaymentGateway;
  let emails: SpyEmailSender;
  let service: OrderService;

  beforeEach(() => {
    orderRepo = new StubOrderRepository();
    payments = new StubPaymentGateway();
    emails = new SpyEmailSender();
    service = new OrderService(orderRepo, payments, emails);
  });

  it('saves the order after successful payment', async () => {
    const order = await service.placeOrder(
      [{ product: 'Widget', price: 10, quantity: 2 }],
      'tok_valid'
    );

    const stored = orderRepo.getStored();
    expect(stored).toHaveLength(1);
    expect(stored[0].id).toBe(order.id);
  });

  it('sends confirmation email after placing order', async () => {
    await service.placeOrder(
      [{ product: 'Widget', price: 10, quantity: 1 }],
      'tok_valid'
    );

    expect(emails.sentEmails).toHaveLength(1);
  });

  it('does not save order when payment fails', async () => {
    payments.shouldFail = true;

    await expect(
      service.placeOrder(
        [{ product: 'Widget', price: 10, quantity: 1 }],
        'tok_invalid'
      )
    ).rejects.toThrow('Payment declined');

    expect(orderRepo.getStored()).toHaveLength(0);
  });
});
```

Without DIP, these tests would require a real Stripe account, a real database, and a real email server. With DIP, each test runs in milliseconds with no external dependencies.

## The Composition Root

When you apply DIP throughout your application, all the concrete wiring happens at one place: the composition root. This is the application's entry point where abstractions are mapped to implementations.

```typescript
// main.ts - the composition root
import { OrderService } from './domain/order-service';
import { PostgresOrderRepository } from './infrastructure/postgres-order-repository';
import { StripePaymentGateway } from './infrastructure/stripe-payment-gateway';
import { SendGridEmailSender } from './infrastructure/sendgrid-email-sender';

// All concrete choices are made here, in one place
function bootstrap() {
  const orderRepo = new PostgresOrderRepository(process.env.DATABASE_URL!);
  const payments = new StripePaymentGateway(process.env.STRIPE_KEY!);
  const emails = new SendGridEmailSender(process.env.SENDGRID_KEY!);

  const orderService = new OrderService(orderRepo, payments, emails);

  return { orderService };
}

const app = bootstrap();
```

The composition root is the only place in the application that knows about concrete implementations. Everything else depends only on abstractions.

Benefits of a single composition root:

- **One place to swap implementations.** Switching from Stripe to PayPal means changing one line.
- **Configuration is centralized.** Environment-specific wiring (dev, staging, production) is handled in one file.
- **Dependency graph is visible.** You can see the entire wiring of the application by reading one function.

## When NOT to Apply DIP

DIP adds indirection. Indirection has a cost: more files, more types, more ceremony. Apply DIP when the benefit outweighs the cost.

### Do Not Invert

**Stable dependencies.** If a module is unlikely to change, wrapping it in an abstraction adds complexity with no benefit:

```typescript
// No need to abstract Math or standard library functions
function calculateArea(radius: number): number {
  return Math.PI * radius ** 2; // Math is stable. Do not wrap it.
}

// No need to abstract built-in data structures
function uniqueItems<T>(items: T[]): T[] {
  return [...new Set(items)]; // Set is stable. Do not wrap it.
}
```

**Internal implementation details.** Private helpers within a single module do not need DIP. The abstraction boundary is at the module's public API:

```typescript
// Internal helpers are implementation details
function processOrder(order: Order): ProcessedOrder {
  const validated = validateOrder(order);    // private helper
  const priced = calculatePricing(validated); // private helper
  return formatOutput(priced);               // private helper
}

// No need to inject validateOrder, calculatePricing, formatOutput
// They are stable, internal, and change together with processOrder
```

**Simple scripts and utilities.** A one-off script that reads a file and transforms data does not need an abstraction layer:

```typescript
// This is fine for a simple utility
import { readFileSync, writeFileSync } from 'fs';

const data = readFileSync('input.csv', 'utf-8');
const transformed = transform(data);
writeFileSync('output.json', JSON.stringify(transformed));
```

### Do Invert

- When you have multiple implementations (or expect to)
- When you need to test in isolation
- When crossing architectural boundaries (domain to infrastructure)
- When a dependency is volatile (likely to change, or external)
- When a module should be reusable across different contexts

## Common Mistakes

- **Mistake 1: Confusing DIP with DI.** Dependency Injection is a technique for passing dependencies in from outside. DIP is a design principle about which direction dependencies should point. You can use DI without following DIP (injecting concrete classes), and you can follow DIP without DI (using a service locator or factory).

- **Mistake 2: Creating interfaces that mirror concrete classes.** If your interface has the same name as the class with "Interface" appended (`MySQLDatabaseInterface`), and exposes the same methods with the same signatures, the abstraction is shaped by the implementation. The interface should be shaped by the *consumer's* needs, not the provider's capabilities.

- **Mistake 3: Letting the low-level module own the abstraction.** If the interface lives in the same package or module as the concrete implementation, the dependency direction has not been inverted. The high-level module still depends on the low-level module's package to get the interface.

- **Mistake 4: Abstracting everything.** Not every dependency needs an interface. Wrapping `Math.random()`, `Date.now()`, or `JSON.parse()` in abstractions adds complexity with zero benefit for most applications. Apply DIP where there is actual volatility or a testability need.

- **Mistake 5: Leaky abstractions.** An interface that exposes `beginTransaction()`, `commit()`, and `rollback()` is leaking database concerns. An interface that takes `sql: string` as a parameter forces SQL on all implementations. The abstraction must be free of implementation-specific vocabulary.

- **Mistake 6: Pushing DIP too deep.** Applying DIP to internal helper functions within a single module adds indirection without benefit. DIP is for boundaries between modules and layers, not for every function call.

---

## Best Practices

- **Name interfaces after their role, not their implementation.** Use `UserRepository`, not `DatabaseUserInterface`. Use `MessageTransport`, not `EmailServiceInterface`. The name should describe the capability from the consumer's perspective.

- **Keep interfaces small and focused.** A `UserRepository` with 20 methods covering every possible query is hard to implement and hard to mock. Combine DIP with the Interface Segregation Principle to keep abstractions manageable.

- **Place interfaces near their consumers.** The interface belongs in the high-level module's package or domain layer, not alongside the implementation. This makes the ownership direction explicit.

- **Use a composition root for wiring.** Centralize all concrete-to-abstraction mapping in one place at the application entry point. This makes the wiring visible and easy to change.

- **Favor constructor injection.** Passing dependencies through the constructor makes them explicit, required, and visible. Property injection and setter injection hide dependencies and allow partially initialized objects.

- **Write integration tests at the boundary.** Unit test the high-level module with stubs. Write separate integration tests for each concrete implementation to verify it correctly satisfies the contract.

## Real-World Use Cases

### 1. Pluggable Storage Backend

An application that stores files locally in development but uses cloud storage in production:

```typescript
interface FileStorage {
  upload(key: string, data: Buffer): Promise<string>; // returns URL
  download(key: string): Promise<Buffer>;
  delete(key: string): Promise<void>;
  exists(key: string): Promise<boolean>;
}

class LocalFileStorage implements FileStorage {
  constructor(private basePath: string) {}

  async upload(key: string, data: Buffer): Promise<string> {
    const filePath = path.join(this.basePath, key);
    await fs.writeFile(filePath, data);
    return `file://${filePath}`;
  }

  async download(key: string): Promise<Buffer> {
    return fs.readFile(path.join(this.basePath, key));
  }

  async delete(key: string): Promise<void> {
    await fs.unlink(path.join(this.basePath, key));
  }

  async exists(key: string): Promise<boolean> {
    try {
      await fs.access(path.join(this.basePath, key));
      return true;
    } catch {
      return false;
    }
  }
}

class S3FileStorage implements FileStorage {
  constructor(private s3Client: S3Client, private bucket: string) {}

  async upload(key: string, data: Buffer): Promise<string> {
    await this.s3Client.send(new PutObjectCommand({
      Bucket: this.bucket,
      Key: key,
      Body: data,
    }));
    return `https://${this.bucket}.s3.amazonaws.com/${key}`;
  }

  async download(key: string): Promise<Buffer> {
    const response = await this.s3Client.send(new GetObjectCommand({
      Bucket: this.bucket,
      Key: key,
    }));
    return Buffer.from(await response.Body!.transformToByteArray());
  }

  async delete(key: string): Promise<void> {
    await this.s3Client.send(new DeleteObjectCommand({
      Bucket: this.bucket,
      Key: key,
    }));
  }

  async exists(key: string): Promise<boolean> {
    try {
      await this.s3Client.send(new HeadObjectCommand({
        Bucket: this.bucket,
        Key: key,
      }));
      return true;
    } catch {
      return false;
    }
  }
}

// The document service never knows or cares where files live
class DocumentService {
  constructor(private storage: FileStorage) {}

  async uploadDocument(name: string, content: Buffer): Promise<Document> {
    const key = `documents/${Date.now()}-${name}`;
    const url = await this.storage.upload(key, content);
    return { name, key, url, uploadedAt: new Date() };
  }
}

// Composition root decides the implementation
const storage = process.env.NODE_ENV === 'production'
  ? new S3FileStorage(new S3Client({}), process.env.S3_BUCKET!)
  : new LocalFileStorage('./uploads');

const documentService = new DocumentService(storage);
```

### 2. Multi-Channel Notification System

A system that sends notifications through different channels based on user preferences:

```typescript
interface NotificationChannel {
  send(recipient: string, message: NotificationMessage): Promise<boolean>;
  supports(type: NotificationType): boolean;
}

type NotificationType = 'email' | 'sms' | 'push' | 'slack';

interface NotificationMessage {
  title: string;
  body: string;
  priority: 'low' | 'normal' | 'high';
}

class EmailChannel implements NotificationChannel {
  constructor(private mailer: SmtpClient) {}

  send(recipient: string, message: NotificationMessage): Promise<boolean> {
    return this.mailer.sendMail({
      to: recipient,
      subject: message.title,
      html: message.body,
    });
  }

  supports(type: NotificationType): boolean {
    return type === 'email';
  }
}

class SlackChannel implements NotificationChannel {
  constructor(private webhookUrl: string) {}

  async send(recipient: string, message: NotificationMessage): Promise<boolean> {
    const response = await fetch(this.webhookUrl, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        channel: recipient,
        text: `*${message.title}*\n${message.body}`,
      }),
    });
    return response.ok;
  }

  supports(type: NotificationType): boolean {
    return type === 'slack';
  }
}

// High-level module depends only on the abstraction
class NotificationService {
  constructor(private channels: NotificationChannel[]) {}

  async notify(
    recipient: string,
    message: NotificationMessage,
    preferredType: NotificationType,
  ): Promise<boolean> {
    const channel = this.channels.find(c => c.supports(preferredType));
    if (!channel) {
      throw new Error(`No channel supports type: ${preferredType}`);
    }
    return channel.send(recipient, message);
  }

  async broadcast(
    recipient: string,
    message: NotificationMessage,
  ): Promise<void> {
    await Promise.all(
      this.channels.map(channel => channel.send(recipient, message))
    );
  }
}

// Wiring at the composition root
const notificationService = new NotificationService([
  new EmailChannel(smtpClient),
  new SlackChannel(process.env.SLACK_WEBHOOK!),
]);
```

### 3. Configurable Logging

A logging system where the destination is determined by the environment:

```typescript
interface LogTransport {
  log(level: LogLevel, message: string, context?: Record<string, unknown>): void;
}

type LogLevel = 'debug' | 'info' | 'warn' | 'error';

class ConsoleTransport implements LogTransport {
  log(level: LogLevel, message: string, context?: Record<string, unknown>): void {
    const timestamp = new Date().toISOString();
    const ctx = context ? ` ${JSON.stringify(context)}` : '';
    console[level](`[${timestamp}] [${level.toUpperCase()}] ${message}${ctx}`);
  }
}

class FileTransport implements LogTransport {
  constructor(private filePath: string) {}

  log(level: LogLevel, message: string, context?: Record<string, unknown>): void {
    const entry = JSON.stringify({
      timestamp: new Date().toISOString(),
      level,
      message,
      ...context,
    });
    appendFileSync(this.filePath, entry + '\n');
  }
}

class HttpTransport implements LogTransport {
  constructor(private endpoint: string) {}

  log(level: LogLevel, message: string, context?: Record<string, unknown>): void {
    // Fire and forget - do not await in logging
    fetch(this.endpoint, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ level, message, ...context }),
    }).catch(() => {}); // Swallow errors - logging should not crash the app
  }
}

// Logger depends only on the abstraction
class Logger {
  constructor(
    private transports: LogTransport[],
    private minLevel: LogLevel = 'info',
  ) {}

  private shouldLog(level: LogLevel): boolean {
    const levels: LogLevel[] = ['debug', 'info', 'warn', 'error'];
    return levels.indexOf(level) >= levels.indexOf(this.minLevel);
  }

  info(message: string, context?: Record<string, unknown>): void {
    if (this.shouldLog('info')) {
      this.transports.forEach(t => t.log('info', message, context));
    }
  }

  error(message: string, context?: Record<string, unknown>): void {
    if (this.shouldLog('error')) {
      this.transports.forEach(t => t.log('error', message, context));
    }
  }

  // ... debug, warn methods follow the same pattern
}

// Development: console only
const devLogger = new Logger([new ConsoleTransport()], 'debug');

// Production: file + remote monitoring
const prodLogger = new Logger([
  new FileTransport('/var/log/app.log'),
  new HttpTransport('https://logs.example.com/ingest'),
], 'info');
```

---

> **Key Takeaway:** The Dependency Inversion Principle is about ownership of abstractions. High-level modules define the interfaces they need, low-level modules implement those interfaces, and both depend on the abstraction rather than on each other. This inverts the traditional dependency flow and creates systems that are testable, flexible, and resilient to change.
