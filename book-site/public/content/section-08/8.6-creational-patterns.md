# 8.6 Creational Patterns (Singleton, Factory, Builder)

> Creational patterns abstract the object construction process, giving you control over how objects are created, which instances are shared, and how complex objects are assembled.

---

## Overview

The Gang of Four (GoF) defined five creational patterns: Singleton, Factory Method, Abstract Factory, Builder, and Prototype. This guide covers the three most commonly needed in frontend and Node.js JavaScript/TypeScript work:

| Pattern | Problem Solved | Key Mechanism |
|---------|---------------|---------------|
| Singleton | Ensure one instance exists and is shared | Private constructor + static instance |
| Factory (Method + Abstract) | Decouple object creation from usage; create families of related objects | Creator method returns interface, not concrete type |
| Builder | Construct complex objects step by step with many optional parts | Fluent method chaining, separate construction from representation |

---

## Singleton

### What It Is

The Singleton pattern ensures that a class has only one instance and provides a global access point to it. It is the simplest creational pattern and also the most frequently misused.

### The Classic Implementation

```typescript
class DatabaseConnection {
  private static instance: DatabaseConnection | null = null;
  private isConnected = false;

  // Private constructor prevents external instantiation
  private constructor(private readonly url: string) {}

  static getInstance(url: string): DatabaseConnection {
    if (!DatabaseConnection.instance) {
      DatabaseConnection.instance = new DatabaseConnection(url);
    }
    return DatabaseConnection.instance;
  }

  connect(): void {
    if (!this.isConnected) {
      console.log(`Connecting to ${this.url}`);
      this.isConnected = true;
    }
  }

  query(sql: string): unknown[] {
    if (!this.isConnected) throw new Error('Not connected');
    // ...execute query
    return [];
  }
}

const db1 = DatabaseConnection.getInstance('postgres://localhost/mydb');
const db2 = DatabaseConnection.getInstance('postgres://localhost/mydb');

console.log(db1 === db2); // true - same instance
```

### Thread Safety Caveat (Node.js)

JavaScript is single-threaded in the main event loop, so the classic double-checked locking pattern is unnecessary. However, Worker Threads each have their own memory space. A Singleton in the main thread is NOT shared with worker threads:

```typescript
// main.ts
const db = DatabaseConnection.getInstance('postgres://localhost/mydb');
// This instance is NOT accessible in worker threads
// Worker threads get their own fresh module scope
```

If you need shared state across workers, use `SharedArrayBuffer`, message passing, or a dedicated process (like Redis).

### ES Modules Are Singletons by Default

In modern JavaScript, every ES module is evaluated once and cached. The exports are shared across all imports within the same module graph. This means a plain module-level variable is effectively a singleton without any class ceremony:

```typescript
// config.ts
// This object is created once. Every file that imports it
// gets a reference to the same object.
export const config = {
  apiUrl: process.env.API_URL ?? 'http://localhost:3000',
  debug: process.env.NODE_ENV !== 'production',
};
```

```typescript
// service-a.ts
import { config } from './config';
config.apiUrl; // 'http://localhost:3000'

// service-b.ts
import { config } from './config';
config.apiUrl; // same reference, same value
```

This is why you should reach for a module-level export before writing a Singleton class. The module system gives you singleton semantics for free.

### When the Class Pattern Adds Value

A Singleton class is warranted when you need:

1. **Lazy initialization** - delay expensive setup until the instance is first needed.
2. **Constructor parameters** - the single instance needs configuration that is only available at runtime.
3. **Controlled teardown** - you need a `destroy()` or `disconnect()` method on the shared instance.

```typescript
// Lazy singleton with configuration
class Logger {
  private static instance: Logger | null = null;
  private readonly prefix: string;

  private constructor(config: { prefix: string; level: string }) {
    this.prefix = config.prefix;
    // Expensive setup: open file handle, establish connection, etc.
  }

  static getInstance(): Logger {
    if (!Logger.instance) {
      Logger.instance = new Logger({
        prefix: process.env.LOG_PREFIX ?? 'APP',
        level: process.env.LOG_LEVEL ?? 'info',
      });
    }
    return Logger.instance;
  }

  // For testing only - allows resetting global state between tests
  static reset(): void {
    Logger.instance = null;
  }

  log(message: string): void {
    console.log(`[${this.prefix}] ${message}`);
  }
}
```

### The Testability Problem

The Singleton's global state is its greatest weakness. Tests that depend on a Singleton share state between them, making tests order-dependent and fragile:

```typescript
// Bad: Tests share global state
describe('UserService', () => {
  it('creates a user', () => {
    const db = DatabaseConnection.getInstance(TEST_URL);
    db.query('INSERT INTO users ...');
    // Leaves state in the singleton that leaks into the next test
  });

  it('counts users', () => {
    const db = DatabaseConnection.getInstance(TEST_URL);
    // This test depends on the state left by the previous test
    const users = db.query('SELECT COUNT(*) FROM users');
    expect(users[0]).toBe(1); // Brittle - depends on test order
  });
});
```

The solution is to inject the singleton through the constructor (DIP + DI), then substitute a fake in tests:

```typescript
// Good: Inject the dependency, do not grab it globally
class UserService {
  constructor(private db: QueryExecutor) {} // Depends on interface, not singleton

  createUser(name: string): void {
    this.db.query(`INSERT INTO users (name) VALUES ('${name}')`);
  }
}

// Test uses a fresh stub every time
it('creates a user', () => {
  const stubDb: QueryExecutor = { query: jest.fn().mockReturnValue([]) };
  const service = new UserService(stubDb);
  service.createUser('Alice');
  expect(stubDb.query).toHaveBeenCalledWith(expect.stringContaining('Alice'));
});
```

### The Singleton as Anti-Pattern

The pattern earns its anti-pattern reputation when used for convenience rather than necessity. Signs you are abusing Singleton:

- The class holds mutable state that multiple parts of the app modify
- Unit tests cannot run in any order because they depend on the singleton's state
- You cannot create a second instance for a different context (e.g., two database schemas, two API environments)
- The singleton leaks into low-level modules that have no business knowing about it

## Factory Method

### What It Is

The Factory Method pattern defines an interface for creating an object, but lets subclasses (or a creator function) decide which class to instantiate. The key insight is that the caller asks for an object by type or specification, not by class name.

### The Problem Without Factory

```typescript
// Bad: Caller knows about every concrete class
function createNotification(type: string, message: string) {
  if (type === 'email') {
    return new EmailNotification(message, emailConfig); // knows about EmailNotification
  } else if (type === 'sms') {
    return new SmsNotification(message, twilioConfig);  // knows about SmsNotification
  } else if (type === 'push') {
    return new PushNotification(message, fcmConfig);    // knows about PushNotification
  }
  throw new Error(`Unknown type: ${type}`);
}
```

Adding a new notification type means modifying this function. Every caller is coupled to the concrete types.

### Simple Factory Function

The simplest factory is just a function that returns an interface:

```typescript
interface Notification {
  send(recipient: string): Promise<boolean>;
}

// Concrete implementations
class EmailNotification implements Notification {
  constructor(private message: string, private config: EmailConfig) {}

  async send(recipient: string): Promise<boolean> {
    // email-specific logic
    return true;
  }
}

class SmsNotification implements Notification {
  constructor(private message: string, private config: TwilioConfig) {}

  async send(recipient: string): Promise<boolean> {
    // SMS-specific logic
    return true;
  }
}

// Factory function - the only place that knows about concrete types
type NotificationType = 'email' | 'sms' | 'push';

function createNotification(
  type: NotificationType,
  message: string,
): Notification {
  switch (type) {
    case 'email': return new EmailNotification(message, emailConfig);
    case 'sms': return new SmsNotification(message, twilioConfig);
    case 'push': return new PushNotification(message, fcmConfig);
    default: throw new Error(`Unknown notification type: ${type}`);
  }
}

// Caller works with the interface, not concrete types
const notification = createNotification('email', 'Your order has shipped');
await notification.send('user@example.com');
```

### Registry-Based Factory (Open/Closed)

A map-based registry factory is open for extension without modification:

```typescript
type NotificationFactory = (message: string) => Notification;

const notificationRegistry = new Map<NotificationType, NotificationFactory>();

// Register factories - this is the only thing that changes when adding a type
notificationRegistry.set('email', (msg) => new EmailNotification(msg, emailConfig));
notificationRegistry.set('sms', (msg) => new SmsNotification(msg, twilioConfig));
notificationRegistry.set('push', (msg) => new PushNotification(msg, fcmConfig));

function createNotification(type: NotificationType, message: string): Notification {
  const factory = notificationRegistry.get(type);
  if (!factory) throw new Error(`No factory registered for: ${type}`);
  return factory(message);
}

// Adding a new type: just register, no other code changes
notificationRegistry.set('slack', (msg) => new SlackNotification(msg, slackConfig));
```

### Class-Based Factory Method (GoF Style)

The classic GoF pattern uses an abstract creator class with a factory method. Subclasses override the method to return different products:

```typescript
interface Transport {
  deliver(shipment: Shipment): void;
}

class TruckTransport implements Transport {
  deliver(shipment: Shipment): void {
    console.log(`Delivering by truck: ${shipment.id}`);
  }
}

class ShipTransport implements Transport {
  deliver(shipment: Shipment): void {
    console.log(`Delivering by ship: ${shipment.id}`);
  }
}

// Abstract creator
abstract class Logistics {
  // Factory method - subclasses override to produce different transports
  abstract createTransport(): Transport;

  // Template method that uses the factory method
  planDelivery(shipment: Shipment): void {
    const transport = this.createTransport();
    transport.deliver(shipment);
  }
}

// Concrete creators
class RoadLogistics extends Logistics {
  createTransport(): Transport {
    return new TruckTransport();
  }
}

class SeaLogistics extends Logistics {
  createTransport(): Transport {
    return new ShipTransport();
  }
}

// Usage
const logistics: Logistics = new RoadLogistics();
logistics.planDelivery({ id: 'SHP-001', weight: 500 });
```

### Abstract Factory

While Factory Method creates one product, Abstract Factory creates families of related products. Use it when you have multiple product hierarchies that must be used together:

```typescript
// Product interfaces
interface Button {
  render(): string;
  onClick(): void;
}

interface Dialog {
  render(): string;
  show(): void;
}

// Abstract Factory interface
interface UIFactory {
  createButton(label: string): Button;
  createDialog(title: string): Dialog;
}

// Light theme family
class LightButton implements Button {
  constructor(private label: string) {}
  render() { return `<button class="light">${this.label}</button>`; }
  onClick() { console.log(`Light button clicked: ${this.label}`); }
}

class LightDialog implements Dialog {
  constructor(private title: string) {}
  render() { return `<div class="dialog light">${this.title}</div>`; }
  show() { console.log(`Showing light dialog: ${this.title}`); }
}

// Dark theme family
class DarkButton implements Button {
  constructor(private label: string) {}
  render() { return `<button class="dark">${this.label}</button>`; }
  onClick() { console.log(`Dark button clicked: ${this.label}`); }
}

class DarkDialog implements Dialog {
  constructor(private title: string) {}
  render() { return `<div class="dialog dark">${this.title}</div>`; }
  show() { console.log(`Showing dark dialog: ${this.title}`); }
}

// Concrete factories
class LightThemeFactory implements UIFactory {
  createButton(label: string): Button { return new LightButton(label); }
  createDialog(title: string): Dialog { return new LightDialog(title); }
}

class DarkThemeFactory implements UIFactory {
  createButton(label: string): Button { return new DarkButton(label); }
  createDialog(title: string): Dialog { return new DarkDialog(title); }
}

// Client code - works with any factory
function renderLoginPage(factory: UIFactory): void {
  const submitBtn = factory.createButton('Log In');
  const errorDialog = factory.createDialog('Authentication Error');

  // Render the page
  console.log(submitBtn.render());
  console.log(errorDialog.render());
}

// Swap entire theme by swapping the factory
const isDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
const factory: UIFactory = isDark ? new DarkThemeFactory() : new LightThemeFactory();

renderLoginPage(factory);
```

The key property: all products from a factory are designed to work together. You cannot mix a `LightButton` with a `DarkDialog` accidentally, because you always go through the same factory instance.

## Builder

### What It Is

The Builder pattern separates the construction of a complex object from its representation. It is the right solution when:

- An object has many parameters, most of which are optional
- The constructor would become a telescope of arguments
- The same construction process must produce different representations
- Object construction involves multiple steps that must occur in sequence

### The Telescoping Constructor Problem

```typescript
// Bad: Telescoping constructors - which argument is which?
class QueryBuilder {
  constructor(
    table: string,
    conditions?: string[],
    orderBy?: string,
    orderDirection?: 'ASC' | 'DESC',
    limit?: number,
    offset?: number,
    joins?: string[],
    selectedFields?: string[],
  ) { /* ... */ }
}

// Caller cannot tell which argument is which
const query = new QueryBuilder(
  'users',
  ['age > 18'],
  'name',
  'ASC',
  10,
  0,
  [],
  ['id', 'name', 'email'],
);
```

This is unreadable. Adding a new parameter requires changes everywhere the constructor is called.

### The Builder Solution

```typescript
interface Query {
  table: string;
  fields: string[];
  conditions: string[];
  joins: JoinClause[];
  orderBy: string | null;
  orderDirection: 'ASC' | 'DESC';
  limit: number | null;
  offset: number;
}

interface JoinClause {
  type: 'INNER' | 'LEFT' | 'RIGHT';
  table: string;
  on: string;
}

class QueryBuilder {
  private query: Query = {
    table: '',
    fields: ['*'],
    conditions: [],
    joins: [],
    orderBy: null,
    orderDirection: 'ASC',
    limit: null,
    offset: 0,
  };

  from(table: string): this {
    this.query.table = table;
    return this; // return this enables chaining
  }

  select(...fields: string[]): this {
    this.query.fields = fields;
    return this;
  }

  where(condition: string): this {
    this.query.conditions.push(condition);
    return this;
  }

  join(type: JoinClause['type'], table: string, on: string): this {
    this.query.joins.push({ type, table, on });
    return this;
  }

  orderBy(field: string, direction: 'ASC' | 'DESC' = 'ASC'): this {
    this.query.orderBy = field;
    this.query.orderDirection = direction;
    return this;
  }

  limit(count: number): this {
    this.query.limit = count;
    return this;
  }

  offset(count: number): this {
    this.query.offset = count;
    return this;
  }

  build(): string {
    if (!this.query.table) throw new Error('Table is required');

    const parts: string[] = [];
    parts.push(`SELECT ${this.query.fields.join(', ')}`);
    parts.push(`FROM ${this.query.table}`);

    for (const join of this.query.joins) {
      parts.push(`${join.type} JOIN ${join.table} ON ${join.on}`);
    }

    if (this.query.conditions.length > 0) {
      parts.push(`WHERE ${this.query.conditions.join(' AND ')}`);
    }

    if (this.query.orderBy) {
      parts.push(`ORDER BY ${this.query.orderBy} ${this.query.orderDirection}`);
    }

    if (this.query.limit !== null) {
      parts.push(`LIMIT ${this.query.limit}`);
    }

    if (this.query.offset > 0) {
      parts.push(`OFFSET ${this.query.offset}`);
    }

    return parts.join('\n');
  }
}

// Usage is readable and self-documenting
const sql = new QueryBuilder()
  .from('users')
  .select('users.id', 'users.name', 'orders.total')
  .join('LEFT', 'orders', 'orders.user_id = users.id')
  .where('users.active = true')
  .where('users.age >= 18')
  .orderBy('users.name', 'ASC')
  .limit(25)
  .offset(50)
  .build();
```

### Immutable Builder (No Mutation)

The above builder mutates internal state. When immutability matters, return a new builder instance on each method call:

```typescript
class ImmutableRequestBuilder {
  constructor(
    private readonly config: Readonly<{
      url: string;
      method: string;
      headers: Record<string, string>;
      body: unknown | null;
      timeout: number;
    }>
  ) {}

  // Creates a new builder with the updated value - original is unchanged
  withMethod(method: string): ImmutableRequestBuilder {
    return new ImmutableRequestBuilder({ ...this.config, method });
  }

  withHeader(key: string, value: string): ImmutableRequestBuilder {
    return new ImmutableRequestBuilder({
      ...this.config,
      headers: { ...this.config.headers, [key]: value },
    });
  }

  withBody(body: unknown): ImmutableRequestBuilder {
    return new ImmutableRequestBuilder({ ...this.config, body });
  }

  withTimeout(ms: number): ImmutableRequestBuilder {
    return new ImmutableRequestBuilder({ ...this.config, timeout: ms });
  }

  build(): Request {
    return new Request(this.config.url, {
      method: this.config.method,
      headers: this.config.headers,
      body: this.config.body ? JSON.stringify(this.config.body) : null,
      signal: AbortSignal.timeout(this.config.timeout),
    });
  }
}

// Base builder - can be reused as a template
const baseRequest = new ImmutableRequestBuilder({
  url: '',
  method: 'GET',
  headers: { 'Content-Type': 'application/json' },
  body: null,
  timeout: 5000,
});

// Each extension creates a new builder, not mutating the base
const authRequest = baseRequest
  .withHeader('Authorization', `Bearer ${token}`);

const userRequest = authRequest
  .withMethod('GET')
  // Build with a different URL by extending the config
  .build();
```

### Director Pattern

When the same construction sequence is used repeatedly, extract it into a Director class:

```typescript
interface UserConfig {
  name: string;
  email: string;
  role: 'admin' | 'user' | 'guest';
  permissions: string[];
  notifications: boolean;
}

class UserBuilder {
  private config: Partial<UserConfig> = {};

  setName(name: string): this { this.config.name = name; return this; }
  setEmail(email: string): this { this.config.email = email; return this; }
  setRole(role: UserConfig['role']): this { this.config.role = role; return this; }
  setPermissions(permissions: string[]): this { this.config.permissions = permissions; return this; }
  enableNotifications(): this { this.config.notifications = true; return this; }
  disableNotifications(): this { this.config.notifications = false; return this; }

  build(): UserConfig {
    const { name, email, role, permissions } = this.config;
    if (!name || !email || !role || !permissions) {
      throw new Error('Missing required fields');
    }
    return {
      name,
      email,
      role,
      permissions,
      notifications: this.config.notifications ?? true,
    };
  }
}

// Director encapsulates common construction sequences
class UserDirector {
  static buildAdminUser(name: string, email: string): UserConfig {
    return new UserBuilder()
      .setName(name)
      .setEmail(email)
      .setRole('admin')
      .setPermissions(['read', 'write', 'delete', 'manage'])
      .enableNotifications()
      .build();
  }

  static buildGuestUser(name: string, email: string): UserConfig {
    return new UserBuilder()
      .setName(name)
      .setEmail(email)
      .setRole('guest')
      .setPermissions(['read'])
      .disableNotifications()
      .build();
  }
}

const admin = UserDirector.buildAdminUser('Alice', 'alice@company.com');
const guest = UserDirector.buildGuestUser('Bob', 'bob@example.com');
```

### Builder vs Object Spread

For simple objects with mostly optional properties, object spread with defaults is often cleaner than a Builder class:

```typescript
// For simple objects, this is enough
interface RequestOptions {
  method?: string;
  headers?: Record<string, string>;
  timeout?: number;
  retries?: number;
}

function createRequest(url: string, options: RequestOptions = {}): Request {
  const config = {
    method: 'GET',
    headers: { 'Content-Type': 'application/json' },
    timeout: 5000,
    retries: 3,
    ...options, // caller overrides defaults
  };
  return new Request(url, config);
}
```

Use a Builder class when:
- The construction process involves validation between steps
- You need the Director pattern to encode multiple standard configurations
- The object has more than 5-7 parameters with complex interdependencies
- Construction is multi-step and stateful

## Choosing the Right Creational Pattern

| Situation | Pattern |
|-----------|---------|
| You need one shared instance across the app | Singleton (or ES module export) |
| You need to decouple creation from usage | Factory Function or Factory Method |
| You need to create families of related objects | Abstract Factory |
| You have an object with many optional parameters | Builder |
| Construction requires multi-step assembly with validation | Builder with Director |
| You want to add a type without modifying existing code | Registry-based Factory |

---

## Common Mistakes

- **Mistake 1: Overusing Singleton.** Many developers reach for Singleton whenever they want shared state. The result is global mutable state that makes code untestable and hard to reason about. Before writing a Singleton class, ask: can a module-level export achieve the same thing?

- **Mistake 2: Singleton that cannot be reset.** If you must use a Singleton, provide a `reset()` or `destroy()` method (even if private) that tests can call to restore a clean state between test runs.

- **Mistake 3: Factory that returns concrete types.** A factory that returns `MySQLRepository` instead of `Repository` defeats the purpose. The return type should always be the interface.

- **Mistake 4: Builder that allows partially built objects.** If `build()` can return an invalid object because required fields were not set, the builder is broken. Required fields should be validated in `build()` and throw if missing.

- **Mistake 5: Mutable builder that returns the same instance via chaining.** If each builder method returns `this`, you cannot safely branch configurations. Setting a property on one reference mutates all references. Use immutable builders (new instance per method) if you need to branch.

- **Mistake 6: Abstract Factory that is too fine-grained.** An Abstract Factory that creates individual UI elements for every pixel-level variation becomes unmaintainable. Apply it at the theme or platform level, not for every minor variant.

---

## Best Practices

- **Prefer module-level exports over Singleton classes.** ES modules are singletons by design. Only write a Singleton class when you need lazy initialization, runtime configuration, or controlled teardown.

- **Return interfaces from factories, not concrete classes.** The whole value of the Factory pattern is decoupling. If the return type exposes the concrete class, you have added abstraction with no benefit.

- **Make Builder methods return `this` for fluent chaining.** In TypeScript, annotating the return type as `this` rather than the class name allows subclasses to maintain fluent interfaces without overriding every method.

- **Validate in `build()`, not in individual setter methods.** Cross-field validation (e.g., "if type is 'credit_card', then cardNumber is required") belongs in `build()` where you have the full picture.

- **Use the Director pattern for preset configurations.** If you find yourself writing the same builder chain in multiple places, extract it into a static director method. It gives the preset a name and a single place to update.

- **Combine patterns.** A Factory can return a Builder. A Builder can accept an Abstract Factory to create its sub-components. Patterns are not mutually exclusive.

## Real-World Use Cases

### 1. HTTP Client with Builder

An HTTP request builder that validates configuration before sending:

```typescript
type HttpMethod = 'GET' | 'POST' | 'PUT' | 'PATCH' | 'DELETE';

class HttpRequest {
  private _url = '';
  private _method: HttpMethod = 'GET';
  private _headers: Record<string, string> = {};
  private _body: unknown = null;
  private _timeout = 30_000;
  private _retries = 0;
  private _retryDelay = 1000;

  url(url: string): this { this._url = url; return this; }
  method(method: HttpMethod): this { this._method = method; return this; }
  header(key: string, value: string): this {
    this._headers[key] = value;
    return this;
  }
  json(body: unknown): this {
    this._body = body;
    this._headers['Content-Type'] = 'application/json';
    return this;
  }
  bearer(token: string): this {
    return this.header('Authorization', `Bearer ${token}`);
  }
  timeout(ms: number): this { this._timeout = ms; return this; }
  retry(count: number, delay = 1000): this {
    this._retries = count;
    this._retryDelay = delay;
    return this;
  }

  async send<T>(): Promise<T> {
    if (!this._url) throw new Error('URL is required');

    const controller = new AbortController();
    const timeoutId = setTimeout(
      () => controller.abort(),
      this._timeout
    );

    let lastError: Error | null = null;

    for (let attempt = 0; attempt <= this._retries; attempt++) {
      try {
        const response = await fetch(this._url, {
          method: this._method,
          headers: this._headers,
          body: this._body ? JSON.stringify(this._body) : null,
          signal: controller.signal,
        });

        if (!response.ok) {
          throw new Error(`HTTP ${response.status}: ${response.statusText}`);
        }

        clearTimeout(timeoutId);
        return response.json();
      } catch (error) {
        lastError = error instanceof Error ? error : new Error(String(error));
        if (attempt < this._retries) {
          await new Promise(r => setTimeout(r, this._retryDelay));
        }
      }
    }

    throw lastError!;
  }
}

// Usage: readable, fluent, self-documenting
const user = await new HttpRequest()
  .url('/api/users/123')
  .method('GET')
  .bearer(authToken)
  .timeout(10_000)
  .retry(3, 500)
  .send<User>();
```

### 2. Abstract Factory for Platform Abstraction

A storage factory that works in both browser and Node.js environments:

```typescript
interface StorageDriver {
  get(key: string): string | null;
  set(key: string, value: string): void;
  remove(key: string): void;
  clear(): void;
}

interface StorageFactory {
  createPersistentStorage(): StorageDriver;
  createSessionStorage(): StorageDriver;
}

// Browser factory
class BrowserStorageFactory implements StorageFactory {
  createPersistentStorage(): StorageDriver {
    return {
      get: (key) => localStorage.getItem(key),
      set: (key, value) => localStorage.setItem(key, value),
      remove: (key) => localStorage.removeItem(key),
      clear: () => localStorage.clear(),
    };
  }

  createSessionStorage(): StorageDriver {
    return {
      get: (key) => sessionStorage.getItem(key),
      set: (key, value) => sessionStorage.setItem(key, value),
      remove: (key) => sessionStorage.removeItem(key),
      clear: () => sessionStorage.clear(),
    };
  }
}

// Node.js / test factory
class MemoryStorageFactory implements StorageFactory {
  createPersistentStorage(): StorageDriver {
    const store = new Map<string, string>();
    return {
      get: (key) => store.get(key) ?? null,
      set: (key, value) => { store.set(key, value); },
      remove: (key) => { store.delete(key); },
      clear: () => { store.clear(); },
    };
  }

  createSessionStorage(): StorageDriver {
    // For tests, session and persistent behave the same
    return this.createPersistentStorage();
  }
}

// App code is platform-agnostic
function createAuthService(storageFactory: StorageFactory) {
  const tokenStore = storageFactory.createPersistentStorage();
  const sessionStore = storageFactory.createSessionStorage();

  return {
    saveToken(token: string): void {
      tokenStore.set('auth_token', token);
    },
    getToken(): string | null {
      return tokenStore.get('auth_token');
    },
    clearSession(): void {
      sessionStore.clear();
      tokenStore.remove('auth_token');
    },
  };
}

// Platform decided at entry point
const factory = typeof window !== 'undefined'
  ? new BrowserStorageFactory()
  : new MemoryStorageFactory();

const authService = createAuthService(factory);
```

### 3. Registry-Based Event Handler Factory

A factory that maps event types to their handlers, extensible without modification:

```typescript
interface DomainEvent {
  type: string;
  payload: unknown;
  occurredAt: Date;
}

interface EventHandler<T extends DomainEvent> {
  handle(event: T): Promise<void>;
}

interface OrderPlaced extends DomainEvent {
  type: 'OrderPlaced';
  payload: { orderId: string; userId: string; total: number };
}

interface UserRegistered extends DomainEvent {
  type: 'UserRegistered';
  payload: { userId: string; email: string };
}

class OrderPlacedHandler implements EventHandler<OrderPlaced> {
  async handle(event: OrderPlaced): Promise<void> {
    console.log(`Processing order ${event.payload.orderId}`);
    // Send confirmation email, update inventory, etc.
  }
}

class UserRegisteredHandler implements EventHandler<UserRegistered> {
  async handle(event: UserRegistered): Promise<void> {
    console.log(`Welcoming user ${event.payload.email}`);
    // Send welcome email, set up default preferences, etc.
  }
}

// Registry-based factory
class EventHandlerFactory {
  private registry = new Map<string, () => EventHandler<DomainEvent>>();

  register<T extends DomainEvent>(
    eventType: string,
    factory: () => EventHandler<T>,
  ): void {
    this.registry.set(eventType, factory as () => EventHandler<DomainEvent>);
  }

  create(eventType: string): EventHandler<DomainEvent> {
    const factory = this.registry.get(eventType);
    if (!factory) throw new Error(`No handler registered for: ${eventType}`);
    return factory();
  }
}

// Wire it at the composition root
const handlerFactory = new EventHandlerFactory();
handlerFactory.register('OrderPlaced', () => new OrderPlacedHandler());
handlerFactory.register('UserRegistered', () => new UserRegisteredHandler());

// Dispatch events without knowing handler types
async function dispatchEvent(event: DomainEvent): Promise<void> {
  const handler = handlerFactory.create(event.type);
  await handler.handle(event);
}
```

> **Key Takeaway:** Use Singleton sparingly (prefer module exports), Factory when you need to decouple creation from usage or support multiple implementations, and Builder when an object has so many optional parts that a constructor becomes unreadable.
