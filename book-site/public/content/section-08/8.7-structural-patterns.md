# 8.7 Structural Patterns (Adapter, Facade, Decorator, Proxy)

> Structural patterns deal with how objects and classes are composed to form larger structures, without tightly coupling them together.

---

## Overview

Structural patterns solve the problem of assembling objects and classes into larger, functional structures. All four patterns in this guide involve wrapping something in something else, but for different reasons:

| Pattern | Wraps | Purpose | Changes Interface? |
|---------|-------|---------|-------------------|
| Adapter | An incompatible object | Make it fit an expected interface | Yes |
| Facade | A subsystem | Simplify a complex API | Yes (hides complexity) |
| Decorator | An object with the same interface | Add behavior without subclassing | No |
| Proxy | An object with the same interface | Control access to the real object | No |

---

## Adapter

### What It Is

The Adapter converts the interface of a class or object into the interface your code expects. It is the pattern you reach for when integrating a third-party library, consuming a legacy API, or normalizing multiple external APIs behind a single interface.

The adapter does not add behavior. It translates. Think of it as a power plug adapter: the underlying electricity source is unchanged, but the plug now fits the socket you have.

### The Problem

Your application code depends on a `Logger` interface. A new team wants to use the `winston` logging library, which has a completely different API:

```typescript
// Your application's expected interface
interface Logger {
  log(level: 'info' | 'warn' | 'error', message: string): void;
}

// Third-party library: winston has a different API
// It has separate methods per level: winston.info(), winston.warn(), winston.error()
// and uses winston.log(level, message) with different arg shapes
import winston from 'winston';
```

Your code cannot call `winston.info()` directly everywhere because that couples your business logic to a specific logging library.

### Object Adapter (Composition)

The object adapter wraps the adaptee through composition:

```typescript
import winston from 'winston';

// The adapter implements the interface your code expects
class WinstonLoggerAdapter implements Logger {
  private readonly winstonLogger: winston.Logger;

  constructor() {
    this.winstonLogger = winston.createLogger({
      transports: [new winston.transports.Console()],
    });
  }

  // Translates the unified log() call into winston's API
  log(level: 'info' | 'warn' | 'error', message: string): void {
    this.winstonLogger[level](message);
  }
}

// Your application code never knows it is talking to winston
const logger: Logger = new WinstonLoggerAdapter();
logger.log('info', 'Server started');
logger.log('error', 'Database connection failed');
```

Swapping to `pino` later means writing a `PinoLoggerAdapter` without touching any business logic.

### Class Adapter (Inheritance)

TypeScript supports multiple inheritance via `implements`, so you can also adapt through inheritance. This is less common but useful when you can extend the adaptee:

```typescript
// Hypothetical: adaptee we can extend
class LegacyPaymentProcessor {
  processPayment(amount: number, cardNumber: string): string {
    // Legacy implementation
    return `txn_${Date.now()}`;
  }
}

// Modern interface
interface PaymentGateway {
  charge(amountInCents: number, token: string): Promise<{ id: string; status: string }>;
}

// Class adapter inherits from adaptee and implements target interface
class LegacyPaymentAdapter
  extends LegacyPaymentProcessor
  implements PaymentGateway
{
  async charge(
    amountInCents: number,
    token: string,
  ): Promise<{ id: string; status: string }> {
    // Convert cents to dollars for legacy system
    const amount = amountInCents / 100;
    const txnId = this.processPayment(amount, token);
    return { id: txnId, status: 'succeeded' };
  }
}
```

Prefer object adapters (composition) over class adapters (inheritance). Class adapters couple the adapter to the adaptee's inheritance hierarchy and limit flexibility.

### Two-Way Adapter

When you need to make two incompatible systems talk to each other in both directions, a two-way adapter implements both interfaces:

```typescript
interface MetricSystem {
  getTemperatureC(): number;
  getWeightKg(): number;
}

interface ImperialSystem {
  getTemperatureF(): number;
  getWeightLbs(): number;
}

class UnitsAdapter implements MetricSystem, ImperialSystem {
  private celsius: number;
  private kilograms: number;

  constructor(celsius: number, kilograms: number) {
    this.celsius = celsius;
    this.kilograms = kilograms;
  }

  getTemperatureC(): number { return this.celsius; }
  getWeightKg(): number { return this.kilograms; }

  getTemperatureF(): number { return this.celsius * 9/5 + 32; }
  getWeightLbs(): number { return this.kilograms * 2.20462; }
}
```

### Adapter vs Facade

- **Adapter** changes the interface to match what the caller expects. The intent is compatibility.
- **Facade** creates a new, simplified interface over a subsystem. The intent is simplification.

An Adapter makes a square peg fit a round hole. A Facade removes the need to know there is a hole at all.

---

## Facade

### What It Is

The Facade provides a simplified interface to a complex subsystem. It reduces the cognitive load of using the subsystem by exposing only what callers need.

The subsystem still exists in full behind the facade. Advanced users can bypass the facade and use the subsystem directly. The facade does not restrict access.

### The Problem Without Facade

Initializing and using a complex video conversion subsystem requires coordinating multiple low-level classes:

```typescript
// Without facade - caller must understand and coordinate the entire subsystem
const file = new VideoFile('movie.mp4');
const codec = CodecFactory.extract(file);
const bitrateReader = new BitrateReader();
const buffer = bitrateReader.read(file, codec);

const destinationCodec = new MPEG4CompressionCodec();
const outputBuffer = bitrateReader.convert(buffer, destinationCodec);
const mixer = new AudioMixer();
const result = mixer.fix(outputBuffer);
new VideoFile(result).save('movie.mp4');
```

The caller is burdened with knowing the entire pipeline. If any step changes, every caller must update.

### The Facade Solution

```typescript
// Subsystem classes (unchanged, not shown in full)
class VideoFile { /* ... */ }
class CodecFactory { static extract(file: VideoFile) { /* ... */ } }
class BitrateReader {
  read(file: VideoFile, codec: unknown) { /* ... */ }
  convert(buffer: unknown, codec: unknown) { /* ... */ }
}
class AudioMixer { fix(buffer: unknown) { /* ... */ } }
class MPEG4CompressionCodec { /* ... */ }

// Facade - single entry point for the most common operation
class VideoConverterFacade {
  convert(filename: string, format: 'mp4' | 'avi' | 'mov'): string {
    const file = new VideoFile(filename);
    const sourceCodec = CodecFactory.extract(file);
    const destinationCodec = new MPEG4CompressionCodec();
    const reader = new BitrateReader();
    const buffer = reader.read(file, sourceCodec);
    const converted = reader.convert(buffer, destinationCodec);
    const mixer = new AudioMixer();
    const output = mixer.fix(converted);

    const outputFilename = filename.replace(/\.[^.]+$/, `.${format}`);
    // save and return
    return outputFilename;
  }
}

// Caller needs to know nothing about the subsystem
const converter = new VideoConverterFacade();
const result = converter.convert('holiday.mp4', 'avi');
```

### Facade for Layered Architecture

Facades are ideal at layer boundaries. An Application Facade wraps multiple domain services and exposes coarse-grained operations to API controllers:

```typescript
// Domain services (fine-grained)
class UserService {
  findById(id: string): User | null { /* ... */ return null; }
  updateProfile(id: string, data: Partial<User>): User { /* ... */ return {} as User; }
}

class NotificationService {
  sendEmail(to: string, subject: string, body: string): Promise<void> {
    return Promise.resolve();
  }
}

class AuditService {
  record(actor: string, action: string, resource: string): void { /* ... */ }
}

class StorageService {
  upload(file: Buffer, path: string): Promise<string> {
    return Promise.resolve(`/uploads/${path}`);
  }
}

// Facade - coarse-grained operation for the controller
class UserAccountFacade {
  constructor(
    private users: UserService,
    private notifications: NotificationService,
    private audit: AuditService,
    private storage: StorageService,
  ) {}

  async updateAvatar(
    userId: string,
    avatarFile: Buffer,
    requestingUserId: string,
  ): Promise<{ avatarUrl: string }> {
    const user = this.users.findById(userId);
    if (!user) throw new Error('User not found');

    // Orchestrates multiple domain services
    const avatarUrl = await this.storage.upload(
      avatarFile,
      `avatars/${userId}.jpg`
    );
    this.users.updateProfile(userId, { avatarUrl });
    await this.notifications.sendEmail(
      user.email,
      'Your avatar was updated',
      `Your profile photo has been changed.`
    );
    this.audit.record(requestingUserId, 'UPDATE_AVATAR', userId);

    return { avatarUrl };
  }
}

// Controller stays thin - all orchestration is in the facade
class UserController {
  constructor(private facade: UserAccountFacade) {}

  async updateAvatar(req: Request, res: Response): Promise<void> {
    const result = await this.facade.updateAvatar(
      req.params.userId,
      req.body.file,
      req.user.id,
    );
    res.json(result);
  }
}
```

### When Facade Becomes Anti-Pattern

A facade that knows too much becomes a "God Object." Signs of a failing facade:

- The facade's methods take many parameters and make many decisions
- All classes in the subsystem depend on the facade (circular dependency)
- The facade is the only way to access the subsystem (access restriction)
- You are putting logic in the facade instead of the domain classes

A good facade is thin. It delegates immediately to domain classes. If you find yourself writing business logic in the facade, move it to the appropriate domain class.

## Decorator (Pattern)

### What It Is

The Decorator pattern attaches additional behavior to an object at runtime by wrapping it in a decorator object that has the same interface. Unlike subclassing (which adds behavior at compile time to all instances), the Decorator pattern adds behavior selectively to specific instances.

Note: This is the GoF Decorator *pattern*, not TypeScript/JavaScript decorator *syntax* (the `@` annotation syntax). They are related in concept but different things. This section covers the pattern.

### Why Not Subclassing?

Suppose you have a `DataSource` interface with `read()` and `write()`. You need combinations of:

- Plain file source
- Compressed file source
- Encrypted file source
- Compressed + Encrypted file source
- Cached file source
- Cached + Compressed file source

With inheritance, you need `n * m` subclasses for all combinations. With the Decorator pattern, you have `n + m` classes that compose freely.

### The Pattern

```typescript
// Component interface
interface DataSource {
  write(data: string): void;
  read(): string;
}

// Concrete component (the thing being decorated)
class FileDataSource implements DataSource {
  constructor(private filename: string) {}

  write(data: string): void {
    // Write data to file
    console.log(`Writing to ${this.filename}: ${data}`);
  }

  read(): string {
    // Read data from file
    return `Contents of ${this.filename}`;
  }
}

// Base decorator - delegates everything to the wrapped component
abstract class DataSourceDecorator implements DataSource {
  constructor(protected wrapped: DataSource) {}

  write(data: string): void {
    this.wrapped.write(data);
  }

  read(): string {
    return this.wrapped.read();
  }
}

// Concrete decorator: adds compression
class CompressionDecorator extends DataSourceDecorator {
  write(data: string): void {
    const compressed = this.compress(data);
    super.write(compressed);
  }

  read(): string {
    const data = super.read();
    return this.decompress(data);
  }

  private compress(data: string): string {
    return `[compressed:${data}]`;
  }

  private decompress(data: string): string {
    return data.replace(/^\[compressed:(.+)\]$/, '$1');
  }
}

// Concrete decorator: adds encryption
class EncryptionDecorator extends DataSourceDecorator {
  constructor(wrapped: DataSource, private key: string) {
    super(wrapped);
  }

  write(data: string): void {
    const encrypted = this.encrypt(data);
    super.write(encrypted);
  }

  read(): string {
    const data = super.read();
    return this.decrypt(data);
  }

  private encrypt(data: string): string {
    return `[encrypted:${data}]`; // simplified
  }

  private decrypt(data: string): string {
    return data.replace(/^\[encrypted:(.+)\]$/, '$1');
  }
}

// Concrete decorator: adds caching
class CachingDecorator extends DataSourceDecorator {
  private cache: string | null = null;

  read(): string {
    if (this.cache !== null) {
      console.log('Cache hit');
      return this.cache;
    }
    this.cache = super.read();
    return this.cache;
  }

  write(data: string): void {
    this.cache = null; // invalidate cache on write
    super.write(data);
  }
}

// Composing decorators freely at runtime
const plain = new FileDataSource('data.txt');

const compressed = new CompressionDecorator(plain);

const encrypted = new EncryptionDecorator(
  new CompressionDecorator(plain),
  'secret-key'
);

const cached = new CachingDecorator(
  new EncryptionDecorator(
    new CompressionDecorator(plain),
    'secret-key'
  )
);

// All have the same interface - caller does not know what is wrapped
cached.write('user data');
console.log(cached.read()); // Cache miss on first call
console.log(cached.read()); // Cache hit on second call
```

### Functional Decorator

Without class ceremony, a higher-order function achieves the same thing:

```typescript
type AsyncOperation<T> = (...args: unknown[]) => Promise<T>;

// Decorator: adds retry logic to any async function
function withRetry<T>(
  fn: AsyncOperation<T>,
  maxAttempts: number,
  delay: number,
): AsyncOperation<T> {
  return async (...args) => {
    let lastError: Error | null = null;
    for (let attempt = 1; attempt <= maxAttempts; attempt++) {
      try {
        return await fn(...args);
      } catch (error) {
        lastError = error instanceof Error ? error : new Error(String(error));
        if (attempt < maxAttempts) {
          await new Promise(r => setTimeout(r, delay));
        }
      }
    }
    throw lastError!;
  };
}

// Decorator: adds timing to any async function
function withTiming<T>(name: string, fn: AsyncOperation<T>): AsyncOperation<T> {
  return async (...args) => {
    const start = performance.now();
    const result = await fn(...args);
    const elapsed = performance.now() - start;
    console.log(`${name} took ${elapsed.toFixed(2)}ms`);
    return result;
  };
}

// Decorator: adds logging to any async function
function withLogging<T>(name: string, fn: AsyncOperation<T>): AsyncOperation<T> {
  return async (...args) => {
    console.log(`${name}: starting with args`, args);
    try {
      const result = await fn(...args);
      console.log(`${name}: succeeded`);
      return result;
    } catch (error) {
      console.log(`${name}: failed`, error);
      throw error;
    }
  };
}

// Compose decorators left-to-right using a helper
function pipe<T>(...decorators: Array<(fn: AsyncOperation<T>) => AsyncOperation<T>>) {
  return (fn: AsyncOperation<T>): AsyncOperation<T> =>
    decorators.reduce((acc, decorator) => decorator(acc), fn);
}

// Base function
async function fetchUser(id: string): Promise<User> {
  const response = await fetch(`/api/users/${id}`);
  return response.json();
}

// Composed with decorators
const robustFetchUser = pipe<User>(
  (fn) => withRetry(fn, 3, 1000),
  (fn) => withTiming('fetchUser', fn),
  (fn) => withLogging('fetchUser', fn),
)(fetchUser);

const user = await robustFetchUser('123');
```

### Decorator Order Matters

Decorator order changes behavior. Read from innermost (closest to the real object) to outermost:

```typescript
// Order 1: compress then encrypt
// write: data -> compress -> encrypt -> disk
// read:  disk -> decrypt -> decompress -> data
const source1 = new EncryptionDecorator(
  new CompressionDecorator(plain),
  'key'
);

// Order 2: encrypt then compress
// write: data -> encrypt -> compress -> disk
// read:  disk -> decompress -> decrypt -> data
const source2 = new CompressionDecorator(
  new EncryptionDecorator(plain, 'key')
);
```

Order 1 is generally preferred for security: you do not compress encrypted data (compression has poor effect on ciphertext and may leak information via side channels).

## Proxy

### What It Is

The Proxy provides a surrogate that controls access to the real object. Proxy and Decorator look identical from the outside (same interface), but their intents differ:

- **Decorator** adds behavior to enhance an object. It usually holds a reference passed in from outside.
- **Proxy** controls access to the object. It usually manages the object's lifecycle (creation, connection, destruction) itself.

### Proxy Types

| Type | Purpose |
|------|---------|
| Virtual Proxy | Lazy initialization: defers expensive creation until needed |
| Remote Proxy | Manages communication with a remote object |
| Protection Proxy | Checks access rights before forwarding requests |
| Caching Proxy | Caches results of expensive operations |
| Logging Proxy | Records all interactions for debugging or auditing |

### Virtual Proxy (Lazy Initialization)

```typescript
interface Image {
  render(): void;
  getDimensions(): { width: number; height: number };
}

class RealImage implements Image {
  private readonly data: Buffer;
  private readonly width: number;
  private readonly height: number;

  constructor(private filename: string) {
    // Expensive: loads and decodes the image immediately
    console.log(`Loading image: ${filename}`);
    this.data = this.loadFromDisk(filename);
    const dimensions = this.decodeDimensions(this.data);
    this.width = dimensions.width;
    this.height = dimensions.height;
  }

  render(): void {
    console.log(`Rendering image: ${this.filename}`);
  }

  getDimensions(): { width: number; height: number } {
    return { width: this.width, height: this.height };
  }

  private loadFromDisk(filename: string): Buffer {
    return Buffer.alloc(0); // simplified
  }

  private decodeDimensions(data: Buffer): { width: number; height: number } {
    return { width: 1920, height: 1080 }; // simplified
  }
}

// Proxy: defers the expensive load until render() is actually called
class LazyImageProxy implements Image {
  private realImage: RealImage | null = null;

  constructor(private filename: string) {
    // Nothing expensive happens here
  }

  // Only creates the real image when needed
  render(): void {
    if (!this.realImage) {
      this.realImage = new RealImage(this.filename);
    }
    this.realImage.render();
  }

  getDimensions(): { width: number; height: number } {
    if (!this.realImage) {
      this.realImage = new RealImage(this.filename);
    }
    return this.realImage.getDimensions();
  }
}

// Create hundreds of image proxies cheaply
const gallery: Image[] = [
  new LazyImageProxy('photo1.jpg'),
  new LazyImageProxy('photo2.jpg'),
  new LazyImageProxy('photo3.jpg'),
  // ... hundreds more
];

// Only the images that are actually rendered incur the loading cost
gallery[0].render(); // Loads and renders photo1 only
```

### Protection Proxy

```typescript
interface UserRepository {
  findAll(): User[];
  findById(id: string): User | null;
  save(user: User): void;
  delete(id: string): void;
}

type Role = 'admin' | 'manager' | 'viewer';

class UserRepositoryProxy implements UserRepository {
  constructor(
    private real: UserRepository,
    private currentRole: Role,
  ) {}

  findAll(): User[] {
    this.require('viewer', 'manager', 'admin');
    return this.real.findAll();
  }

  findById(id: string): User | null {
    this.require('viewer', 'manager', 'admin');
    return this.real.findById(id);
  }

  save(user: User): void {
    this.require('manager', 'admin');
    this.real.save(user);
  }

  delete(id: string): void {
    this.require('admin');
    this.real.delete(id);
  }

  private require(...roles: Role[]): void {
    if (!roles.includes(this.currentRole)) {
      throw new Error(
        `Access denied. Required: [${roles.join(', ')}], current: ${this.currentRole}`
      );
    }
  }
}

// Wrap the real repository with access control
const repo = new UserRepositoryProxy(realUserRepository, currentUser.role);
repo.findAll(); // ok for viewer, manager, admin
repo.delete('123'); // throws for viewer and manager
```

### JavaScript's Built-In Proxy

JavaScript has a native `Proxy` object that intercepts operations on any object using *traps*. This is the language-level mechanism; the GoF Proxy pattern can be implemented with it.

```typescript
// Common traps: get, set, has, deleteProperty, apply, construct

// Example 1: Validation proxy
function createValidatedUser(user: Partial<User>): User {
  return new Proxy(user as User, {
    set(target, prop, value) {
      if (prop === 'email' && typeof value === 'string') {
        if (!value.includes('@')) {
          throw new Error(`Invalid email: ${value}`);
        }
      }
      if (prop === 'age' && typeof value === 'number') {
        if (value < 0 || value > 150) {
          throw new Error(`Invalid age: ${value}`);
        }
      }
      target[prop as keyof User] = value;
      return true; // MUST return true, or strict mode throws TypeError
    },
  });
}

const user = createValidatedUser({ name: 'Alice' });
user.email = 'alice@example.com'; // ok
user.email = 'not-an-email'; // throws Error: Invalid email

// Example 2: Read-only proxy
function freeze<T extends object>(obj: T): Readonly<T> {
  return new Proxy(obj, {
    set() {
      throw new TypeError('Object is read-only');
    },
    deleteProperty() {
      throw new TypeError('Object is read-only');
    },
  });
}

// Example 3: Logging proxy - intercepts all property access
function createLoggedObject<T extends object>(target: T, name: string): T {
  return new Proxy(target, {
    get(obj, prop) {
      const value = Reflect.get(obj, prop);
      if (typeof value === 'function') {
        return function (...args: unknown[]) {
          console.log(`${name}.${String(prop)}(${args.map(a => JSON.stringify(a)).join(', ')})`);
          const result = value.apply(obj, args);
          console.log(`${name}.${String(prop)} returned:`, result);
          return result;
        };
      }
      console.log(`${name}.${String(prop)} accessed: ${JSON.stringify(value)}`);
      return value;
    },
  });
}

// Example 4: Default values proxy
function withDefaults<T extends object>(
  target: T,
  defaults: Partial<T>,
): T {
  return new Proxy(target, {
    get(obj, prop) {
      const value = Reflect.get(obj, prop);
      if (value === undefined && prop in defaults) {
        return defaults[prop as keyof T];
      }
      return value;
    },
  });
}

const config = withDefaults(
  { apiUrl: 'https://api.example.com' },
  { timeout: 5000, retries: 3, debug: false }
);
config.timeout; // 5000 (from defaults)
config.apiUrl;  // 'https://api.example.com' (from target)
```

### Proxy Gotchas

The native `Proxy` object has several non-obvious behaviors:

```typescript
// Gotcha 1: 'this' binding
class Counter {
  private count = 0;
  increment() { this.count++; }
  getCount() { return this.count; }
}

const counter = new Counter();
const proxied = new Proxy(counter, {
  get(target, prop) {
    const value = Reflect.get(target, prop);
    if (typeof value === 'function') {
      // Must bind to target, not proxy, or 'this' inside
      // the method refers to the proxy, not the real object
      return value.bind(target);
    }
    return value;
  },
});

// Gotcha 2: set must return true
const strictProxy = new Proxy({}, {
  set(target, prop, value) {
    target[prop as string] = value;
    // MUST return true. If you forget, strict mode throws a TypeError.
    return true;
  },
});

// Gotcha 3: Proxy does not work with Map/Set/WeakMap internals
// These built-ins use internal slots that proxies cannot intercept
const map = new Map();
const proxiedMap = new Proxy(map, {});
proxiedMap.set('key', 'value'); // TypeError: Method Map.prototype.set called on incompatible receiver
// Fix: bind the method to the original map
const proxiedMap2 = new Proxy(map, {
  get(target, prop) {
    const value = Reflect.get(target, prop);
    return typeof value === 'function' ? value.bind(target) : value;
  },
});
proxiedMap2.set('key', 'value'); // works
```

## Choosing the Right Structural Pattern

```
Is the interface incompatible?
  Yes -> ADAPTER (translate the interface)

Is the subsystem too complex to use directly?
  Yes -> FACADE (simplify with a higher-level API)

Do you need to add behavior to specific instances without subclassing?
  Yes -> DECORATOR (wrap with same interface, add behavior)

Do you need to control access to an object?
  Yes -> PROXY (control: lazy init, auth, caching, logging)
```

---

## Common Mistakes

- **Mistake 1: Adapter that leaks the adaptee's interface.** The adapter return type should be the *target* interface, not the adaptee. If callers know they are talking to an adapter (e.g., they call `adapter.getWinstonLogger()`), you have not really adapted anything.

- **Mistake 2: Facade that becomes a God Object.** A facade that grows to hundreds of lines, makes all the decisions, and forces everything through it is worse than the subsystem it was supposed to simplify. Facades should be thin delegators.

- **Mistake 3: Decorator that breaks the interface contract.** The decorator must behave consistently with the decorated interface. If the original `read()` method is synchronous, a decorator that makes it asynchronous is a contract violation, not a decoration.

- **Mistake 4: Proxy that does not use `Reflect`.** When implementing native JS `Proxy` traps, prefer `Reflect.get(target, prop, receiver)` over `target[prop]`. `Reflect` properly handles receiver binding, prototype chains, and exotic objects, whereas direct property access does not.

- **Mistake 5: Forgetting `return true` in `set` trap.** A `set` trap that does not return `true` causes a `TypeError` in strict mode. Always explicitly return `true` after a successful set, even if you did nothing special.

- **Mistake 6: Confusing Decorator with Proxy.** Both wrap with the same interface. The distinction is intent: Decorator *adds* behavior (you compose decorators knowingly). Proxy *controls access* to the real object (it manages the object's lifecycle and mediates access).

---

## Best Practices

- **Keep adapters focused on translation, not behavior.** If you are adding business logic in an adapter, that logic belongs in its own class. The adapter's job is interface translation only.

- **Use the Facade to define layer boundaries.** Every time you cross an architectural boundary (controller to domain, domain to infrastructure), consider a facade. It documents what the caller needs and hides what it does not.

- **Implement a base Decorator class.** When building class-based decorators, the base decorator class that delegates all methods to the wrapped component saves you from writing forwarding boilerplate in every concrete decorator. Each concrete decorator only overrides the methods it cares about.

- **Prefer functional decorators for functions.** Higher-order functions are cleaner than class-based decorators for functions. Reserve class-based decorators for scenarios with state or multiple interrelated methods.

- **Use `Reflect` methods in native Proxy traps.** `Reflect.get`, `Reflect.set`, `Reflect.has`, etc. are the correct counterparts for each trap and handle edge cases that direct property access does not.

- **Test decorators and proxies in isolation.** Test the real object without decorators, then test each decorator or proxy independently by injecting a stub for the wrapped object. This isolates behavior verification.

### 1. Adapter: Normalizing Multiple Payment APIs

An e-commerce application that supports multiple payment providers but exposes a single interface to the rest of the codebase:

```typescript
interface PaymentProvider {
  charge(
    amount: number,
    currency: string,
    paymentMethod: string,
  ): Promise<{ transactionId: string; status: 'succeeded' | 'failed' }>;

  refund(
    transactionId: string,
    amount?: number,
  ): Promise<{ refundId: string }>;
}

// Stripe uses a different shape
class StripeAdapter implements PaymentProvider {
  constructor(private stripe: StripeClient) {}

  async charge(amount: number, currency: string, paymentMethod: string) {
    const intent = await this.stripe.paymentIntents.create({
      amount,          // Stripe expects amount in cents
      currency,
      payment_method: paymentMethod,
      confirm: true,
    });
    return {
      transactionId: intent.id,
      status: intent.status === 'succeeded' ? 'succeeded' : 'failed',
    };
  }

  async refund(transactionId: string, amount?: number) {
    const refund = await this.stripe.refunds.create({
      payment_intent: transactionId,
      ...(amount !== undefined && { amount }),
    });
    return { refundId: refund.id };
  }
}

// PayPal has a completely different API
class PayPalAdapter implements PaymentProvider {
  constructor(private paypal: PayPalClient) {}

  async charge(amount: number, currency: string, paymentMethod: string) {
    // PayPal amounts are in decimal (not cents)
    const order = await this.paypal.orders.create({
      intent: 'CAPTURE',
      purchase_units: [{
        amount: {
          currency_code: currency,
          value: (amount / 100).toFixed(2),
        },
      }],
    });
    const capture = await this.paypal.orders.capture(order.id);
    return {
      transactionId: capture.id,
      status: capture.status === 'COMPLETED' ? 'succeeded' : 'failed',
    };
  }

  async refund(transactionId: string, amount?: number) {
    const refund = await this.paypal.captures.refund(transactionId, {
      amount: amount
        ? { value: (amount / 100).toFixed(2), currency_code: 'USD' }
        : undefined,
    });
    return { refundId: refund.id };
  }
}

// Business logic works with the interface, not the provider
class OrderService {
  constructor(private payment: PaymentProvider) {}

  async placeOrder(cart: Cart, paymentMethodId: string): Promise<Order> {
    const result = await this.payment.charge(
      cart.totalInCents,
      'USD',
      paymentMethodId,
    );
    if (result.status !== 'succeeded') {
      throw new Error('Payment failed');
    }
    return Order.create(cart, result.transactionId);
  }
}
```

### 2. Decorator: Composable Fetch Middleware

A fetch wrapper that composes cross-cutting concerns:

```typescript
type FetchFn = typeof fetch;

// Decorator: adds Authorization header
function withAuth(fetchFn: FetchFn, getToken: () => string): FetchFn {
  return (input, init = {}) => {
    const headers = new Headers(init.headers);
    headers.set('Authorization', `Bearer ${getToken()}`);
    return fetchFn(input, { ...init, headers });
  };
}

// Decorator: adds base URL prefix
function withBaseUrl(fetchFn: FetchFn, baseUrl: string): FetchFn {
  return (input, init?) => {
    const url = typeof input === 'string'
      ? `${baseUrl}${input}`
      : input;
    return fetchFn(url, init);
  };
}

// Decorator: adds retry logic
function withRetry(fetchFn: FetchFn, attempts: number): FetchFn {
  return async (input, init?) => {
    let lastError: Error | null = null;
    for (let i = 0; i < attempts; i++) {
      try {
        const response = await fetchFn(input, init);
        if (response.ok || response.status < 500) return response;
        throw new Error(`HTTP ${response.status}`);
      } catch (error) {
        lastError = error instanceof Error ? error : new Error(String(error));
        if (i < attempts - 1) {
          await new Promise(r => setTimeout(r, 1000 * (i + 1))); // exponential backoff
        }
      }
    }
    throw lastError!;
  };
}

// Decorator: adds caching for GET requests
function withCache(fetchFn: FetchFn, ttlMs: number): FetchFn {
  const cache = new Map<string, { data: Response; expiresAt: number }>();
  return async (input, init?) => {
    const method = (init?.method ?? 'GET').toUpperCase();
    if (method !== 'GET') return fetchFn(input, init);

    const key = String(input);
    const cached = cache.get(key);
    if (cached && cached.expiresAt > Date.now()) {
      return cached.data.clone();
    }

    const response = await fetchFn(input, init);
    if (response.ok) {
      cache.set(key, { data: response.clone(), expiresAt: Date.now() + ttlMs });
    }
    return response;
  };
}

// Compose: each decorator wraps the previous
const apiFetch = withRetry(
  withAuth(
    withCache(
      withBaseUrl(fetch, 'https://api.example.com'),
      60_000  // cache for 1 minute
    ),
    () => authStore.getToken()
  ),
  3  // retry up to 3 times
);

// Usage identical to native fetch
const response = await apiFetch('/users/123');
const user = await response.json();
```

### 3. Proxy: Caching Data Layer

A caching proxy that intercepts reads and stores results, invalidated on writes:

```typescript
interface ProductRepository {
  findById(id: string): Promise<Product | null>;
  findAll(filter?: ProductFilter): Promise<Product[]>;
  save(product: Product): Promise<void>;
  delete(id: string): Promise<void>;
}

class CachingProductRepository implements ProductRepository {
  private singleCache = new Map<string, Product>();
  private listCache: Product[] | null = null;
  private readonly TTL = 5 * 60 * 1000; // 5 minutes
  private cacheTime = 0;

  constructor(private real: ProductRepository) {}

  async findById(id: string): Promise<Product | null> {
    if (this.singleCache.has(id)) {
      console.log(`Cache hit: product ${id}`);
      return this.singleCache.get(id) ?? null;
    }
    const product = await this.real.findById(id);
    if (product) this.singleCache.set(id, product);
    return product;
  }

  async findAll(filter?: ProductFilter): Promise<Product[]> {
    const isFresh = Date.now() - this.cacheTime < this.TTL;
    if (this.listCache && isFresh && !filter) {
      console.log('Cache hit: product list');
      return this.listCache;
    }
    const products = await this.real.findAll(filter);
    if (!filter) {
      this.listCache = products;
      this.cacheTime = Date.now();
    }
    return products;
  }

  async save(product: Product): Promise<void> {
    await this.real.save(product);
    // Invalidate caches on write
    this.singleCache.delete(product.id);
    this.singleCache.set(product.id, product); // update with latest
    this.listCache = null; // invalidate list cache
  }

  async delete(id: string): Promise<void> {
    await this.real.delete(id);
    this.singleCache.delete(id);
    this.listCache = null;
  }
}

// Transparent to the application code
const products: ProductRepository = new CachingProductRepository(
  new PostgresProductRepository(db)
);

const product = await products.findById('prod_123'); // DB hit
const same = await products.findById('prod_123');    // Cache hit
```
