# 8.9 Patterns in Frontend Context (Module, Pub/Sub, Middleware, State Machine)

> Classic design patterns manifest differently in frontend JavaScript -- understanding how they translate gives you the vocabulary to solve real UI architecture problems.

---

## Overview

The GoF patterns were defined for object-oriented compiled languages. JavaScript's dynamic nature and the browser environment create distinct variations. This guide covers four patterns that appear constantly in frontend and Node.js codebases, in forms specific to JavaScript.

| Pattern | Core Idea | Frontend Manifestation |
|---------|----------|----------------------|
| Module | Encapsulate scope, expose a public API | IIFE (legacy), ES module files, revealing module |
| Pub/Sub | Decoupled broadcast/subscribe | `CustomEvent`, typed event bus, `BroadcastChannel` |
| Middleware | Composable pipeline with `next()` | Express/Koa, fetch interceptors, form processing |
| State Machine | Explicit states and transitions | UI component states, async flows, XState |

---

## Module Pattern

### Why It Existed

Before ES modules, JavaScript had no built-in scoping mechanism for files. Everything in a `<script>` tag lived in the global scope. The Module pattern used closures and IIFEs to create private scope.

### The IIFE Module

```javascript
// Pre-ES6 module pattern
const CartModule = (function () {
  // Private - not accessible from outside
  let items = [];
  let discount = 0;

  function recalculate() {
    const subtotal = items.reduce((sum, item) => sum + item.price * item.quantity, 0);
    return subtotal * (1 - discount);
  }

  // Public API returned as an object
  return {
    addItem(item) {
      const existing = items.find(i => i.id === item.id);
      if (existing) {
        existing.quantity += item.quantity;
      } else {
        items.push({ ...item });
      }
    },

    removeItem(id) {
      items = items.filter(i => i.id !== id);
    },

    applyDiscount(percent) {
      discount = percent / 100;
    },

    getTotal() {
      return recalculate();
    },

    getItems() {
      return [...items]; // return copy to prevent external mutation
    },
  };
})();

CartModule.addItem({ id: '1', name: 'Widget', price: 9.99, quantity: 2 });
CartModule.applyDiscount(10);
console.log(CartModule.getTotal()); // 17.982

// Private state is protected
console.log(CartModule.items); // undefined - not exposed
```

### Revealing Module Pattern

The Revealing Module is a variant where all logic is private by default and you explicitly choose what to reveal:

```javascript
const AuthModule = (function () {
  // Everything is defined privately
  let currentUser = null;
  let token = null;

  function validateToken(t) {
    return t && t.length > 20;
  }

  function setSession(user, t) {
    currentUser = user;
    token = t;
    localStorage.setItem('token', t);
  }

  function clearSession() {
    currentUser = null;
    token = null;
    localStorage.removeItem('token');
  }

  async function login(email, password) {
    const response = await fetch('/api/auth/login', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email, password }),
    });
    const data = await response.json();
    if (data.token) setSession(data.user, data.token);
    return data;
  }

  function logout() {
    clearSession();
    window.location.href = '/login';
  }

  function getUser() {
    return currentUser ? { ...currentUser } : null;
  }

  function isAuthenticated() {
    const storedToken = token || localStorage.getItem('token');
    return validateToken(storedToken);
  }

  // Reveal only what external code needs
  return {
    login,
    logout,
    getUser,
    isAuthenticated,
  };
  // validateToken, setSession, clearSession are private
})();
```

### ES Modules: The Modern Module Pattern

ES modules (`.mjs` / `import`/`export`) are the standard today. They provide the same encapsulation as the IIFE pattern but with native syntax:

```typescript
// cart.ts - each file is its own module scope
// Private: not exported, not accessible outside this file
let items: CartItem[] = [];
let discount = 0;

function recalculate(): number {
  const subtotal = items.reduce((sum, item) => sum + item.price * item.quantity, 0);
  return subtotal * (1 - discount);
}

// Public: explicitly exported
export function addItem(item: CartItem): void {
  const existing = items.find(i => i.id === item.id);
  if (existing) {
    existing.quantity += item.quantity;
  } else {
    items.push({ ...item });
  }
}

export function applyDiscount(percent: number): void {
  discount = percent / 100;
}

export function getTotal(): number {
  return recalculate();
}

export function getItems(): CartItem[] {
  return [...items];
}
```

ES module files are effectively singletons: the module is evaluated once and all importers share the same module scope. This means module-level variables like `items` and `discount` above are shared state.

### Namespace Module

When organizing a group of related utilities without a class, an exported namespace object works well:

```typescript
// string-utils.ts
function capitalize(str: string): string {
  return str.charAt(0).toUpperCase() + str.slice(1);
}

function truncate(str: string, maxLength: number, suffix = '...'): string {
  if (str.length <= maxLength) return str;
  return str.slice(0, maxLength - suffix.length) + suffix;
}

function slugify(str: string): string {
  return str
    .toLowerCase()
    .replace(/[^a-z0-9]+/g, '-')
    .replace(/^-|-$/g, '');
}

// Named exports for tree-shaking
export { capitalize, truncate, slugify };

// Or namespace object for grouped access
export const StringUtils = { capitalize, truncate, slugify };
```

### When the IIFE Pattern Still Applies

In 2026, you should reach for IIFE modules only in specific scenarios:

- **Browser scripts without a build step:** A single file that must work as a `<script>` tag
- **Polyfills and third-party snippets:** Code that must avoid polluting global scope in any environment
- **Immediately-executed initialization:** Code that must run once and has no exports

For everything else, use ES modules.

## Pub/Sub Pattern

### Pub/Sub vs Observer

Observer (covered in 8.8) has the subject hold references to its observers. Pub/Sub introduces a message broker: publishers and subscribers never know about each other. They only know the broker and the event channel name.

```
Observer:  Subject -> [Observer1, Observer2, Observer3]
Pub/Sub:   Publisher -> [Broker] -> [Subscriber1, Subscriber2, Subscriber3]
```

The distinction matters when components live in separate modules with no shared reference.

### Custom Event as Pub/Sub

The browser's built-in `CustomEvent` + `dispatchEvent` is a native Pub/Sub system. The `document` or any DOM element serves as the broker:

```typescript
// Publisher: dispatches events without knowing who listens
function publishUserLoggedIn(userId: string, role: string): void {
  document.dispatchEvent(
    new CustomEvent('user:logged_in', {
      detail: { userId, role },
      bubbles: true,
    })
  );
}

// Subscriber A: analytics module
document.addEventListener('user:logged_in', (event: Event) => {
  const { userId, role } = (event as CustomEvent).detail;
  analytics.identify(userId, { role });
});

// Subscriber B: UI module
document.addEventListener('user:logged_in', (event: Event) => {
  const { role } = (event as CustomEvent).detail;
  document.querySelector('.admin-nav')?.classList.toggle('hidden', role !== 'admin');
});

// Trigger from auth flow
publishUserLoggedIn('user_123', 'admin');
```

### Typed Pub/Sub Broker

A typed implementation prevents typos in event names and enforces payload shapes:

```typescript
type PubSubMap = {
  'cart:updated': { itemCount: number; total: number };
  'user:session_expired': { reason: 'timeout' | 'logout' | 'revoked' };
  'notification:received': { id: string; message: string; type: 'info' | 'error' | 'warn' };
  'ui:theme_changed': { theme: 'light' | 'dark' | 'system' };
};

type Subscriber<T> = (payload: T) => void;

class PubSubBroker {
  private channels = new Map<string, Set<Subscriber<unknown>>>();

  subscribe<K extends keyof PubSubMap>(
    channel: K,
    subscriber: Subscriber<PubSubMap[K]>,
  ): () => void {
    if (!this.channels.has(channel)) {
      this.channels.set(channel, new Set());
    }
    const subs = this.channels.get(channel)!;
    subs.add(subscriber as Subscriber<unknown>);

    // Return unsubscribe function
    return () => subs.delete(subscriber as Subscriber<unknown>);
  }

  publish<K extends keyof PubSubMap>(channel: K, payload: PubSubMap[K]): void {
    this.channels.get(channel)?.forEach(sub => {
      try {
        sub(payload);
      } catch (error) {
        // Isolate subscriber errors - one failing subscriber should not stop others
        console.error(`Subscriber error on channel "${channel}":`, error);
      }
    });
  }

  // Remove all subscribers from a channel
  clear(channel: keyof PubSubMap): void {
    this.channels.delete(channel);
  }
}

// Application-wide singleton broker
export const pubsub = new PubSubBroker();
```

### Using the Broker

```typescript
import { pubsub } from './pubsub-broker';

// In checkout module
function completePurchase(cart: Cart): void {
  processPayment(cart);
  pubsub.publish('cart:updated', { itemCount: 0, total: 0 });
}

// In navigation module (registered at startup)
const unsubCart = pubsub.subscribe('cart:updated', ({ itemCount }) => {
  document.querySelector('#cart-badge')!.textContent = String(itemCount);
});

// In analytics module (registered at startup)
const unsubCart2 = pubsub.subscribe('cart:updated', ({ total, itemCount }) => {
  analytics.track('cart_viewed', { total, itemCount });
});

// Clean up when a component is removed from the DOM
disconnectedCallback(): void {
  unsubCart();
  unsubCart2();
}
```

### BroadcastChannel for Cross-Tab Pub/Sub

The browser's `BroadcastChannel` API enables Pub/Sub across browser tabs and service workers:

```typescript
class CrossTabPubSub {
  private channel: BroadcastChannel;
  private localHandlers = new Map<string, Set<Function>>();

  constructor(channelName: string) {
    this.channel = new BroadcastChannel(channelName);
    this.channel.onmessage = (event) => {
      const { type, payload } = event.data;
      this.localHandlers.get(type)?.forEach(h => h(payload));
    };
  }

  publish(type: string, payload: unknown): void {
    // Post to other tabs
    this.channel.postMessage({ type, payload });
    // Also notify local handlers (same tab)
    this.localHandlers.get(type)?.forEach(h => h(payload));
  }

  subscribe(type: string, handler: Function): () => void {
    if (!this.localHandlers.has(type)) {
      this.localHandlers.set(type, new Set());
    }
    this.localHandlers.get(type)!.add(handler);
    return () => this.localHandlers.get(type)?.delete(handler);
  }
}

const tabBus = new CrossTabPubSub('app-events');

// Tab 1: user logs out
tabBus.publish('user:logged_out', {});

// Tab 2: receives the message and redirects
tabBus.subscribe('user:logged_out', () => {
  window.location.href = '/login';
});
```

## Middleware Pattern

### What It Is

Middleware is a pattern where a request (or any input) passes through a series of processing functions sequentially. Each function can inspect, transform, or terminate the flow. The key mechanism is a `next()` function each middleware calls to pass control to the next handler.

```
Request -> [mw1] -> [mw2] -> [mw3] -> Handler -> Response
                              ^
                              Calls next() to continue
                              Skips next() to terminate early
```

### Building a Middleware Pipeline

```typescript
type Context<T> = { request: T; response: unknown; [key: string]: unknown };
type Next = () => Promise<void>;
type Middleware<T> = (ctx: Context<T>, next: Next) => Promise<void>;

class Pipeline<T> {
  private middlewares: Middleware<T>[] = [];

  use(middleware: Middleware<T>): this {
    this.middlewares.push(middleware);
    return this;
  }

  async execute(request: T): Promise<unknown> {
    const ctx: Context<T> = { request, response: null };

    const dispatch = async (index: number): Promise<void> => {
      if (index >= this.middlewares.length) return;
      const middleware = this.middlewares[index];
      await middleware(ctx, () => dispatch(index + 1));
    };

    await dispatch(0);
    return ctx.response;
  }
}

// Usage
interface HttpRequest {
  method: string;
  path: string;
  headers: Record<string, string>;
  body: unknown;
  userId?: string;
}

const pipeline = new Pipeline<HttpRequest>();

// Logging middleware
pipeline.use(async (ctx, next) => {
  const start = Date.now();
  console.log(`--> ${ctx.request.method} ${ctx.request.path}`);
  await next(); // Call next before responding
  console.log(`<-- ${ctx.request.method} ${ctx.request.path} ${Date.now() - start}ms`);
});

// Authentication middleware
pipeline.use(async (ctx, next) => {
  const token = ctx.request.headers['authorization']?.replace('Bearer ', '');
  if (!token) {
    ctx.response = { status: 401, body: 'Unauthorized' };
    return; // Do NOT call next() - terminates the pipeline
  }
  ctx.userId = verifyToken(token); // Attaches data to context
  await next();
});

// Rate limiting middleware
pipeline.use(async (ctx, next) => {
  const key = `rate:${ctx.userId ?? ctx.request.headers['x-forwarded-for']}`;
  const count = await rateLimit.increment(key);
  if (count > 100) {
    ctx.response = { status: 429, body: 'Too Many Requests' };
    return;
  }
  await next();
});

// Route handler - last in the chain
pipeline.use(async (ctx) => {
  ctx.response = { status: 200, body: { userId: ctx.userId } };
});

await pipeline.execute(incomingRequest);
```

### Koa-Style Onion Model

Koa's middleware model is distinctive: middleware can run code both *before* and *after* calling `next()`. This creates an "onion" effect where the response passes back through each middleware:

```typescript
type KoaMiddleware<T> = (ctx: T, next: () => Promise<void>) => Promise<void>;

async function compose<T>(middlewares: KoaMiddleware<T>[], ctx: T): Promise<void> {
  let index = -1;

  async function dispatch(i: number): Promise<void> {
    if (i <= index) throw new Error('next() called multiple times');
    index = i;
    const fn = middlewares[i];
    if (!fn) return;
    await fn(ctx, () => dispatch(i + 1));
  }

  await dispatch(0);
}

interface ApiContext {
  path: string;
  method: string;
  status: number;
  body: unknown;
  startTime?: number;
}

const middlewares: KoaMiddleware<ApiContext>[] = [
  // Timing: wraps the entire request
  async (ctx, next) => {
    ctx.startTime = Date.now();
    await next(); // Downstream middlewares run here
    const elapsed = Date.now() - (ctx.startTime ?? 0);
    console.log(`${ctx.method} ${ctx.path} -> ${ctx.status} in ${elapsed}ms`);
  },

  // Error handling: catches errors from downstream
  async (ctx, next) => {
    try {
      await next();
    } catch (error) {
      ctx.status = 500;
      ctx.body = { error: 'Internal Server Error' };
      console.error('Unhandled error:', error);
    }
  },

  // CORS: sets headers on the way out
  async (ctx, next) => {
    await next();
    // This runs AFTER downstream, modifying the response
    // ctx.set('Access-Control-Allow-Origin', '*');
  },

  // Route handler
  async (ctx) => {
    ctx.status = 200;
    ctx.body = { message: 'Hello' };
  },
];

const ctx: ApiContext = { path: '/api/hello', method: 'GET', status: 200, body: null };
await compose(middlewares, ctx);
```

### Middleware for Data Validation

```typescript
type ValidatorMiddleware = (
  data: Record<string, unknown>,
  next: (errors: string[]) => void,
) => void;

function required(field: string): ValidatorMiddleware {
  return (data, next) => {
    const errors: string[] = [];
    if (!data[field]) errors.push(`${field} is required`);
    next(errors);
  };
}

function minLength(field: string, min: number): ValidatorMiddleware {
  return (data, next) => {
    const value = String(data[field] ?? '');
    const errors = value.length < min
      ? [`${field} must be at least ${min} characters`]
      : [];
    next(errors);
  };
}

function isEmail(field: string): ValidatorMiddleware {
  return (data, next) => {
    const value = String(data[field] ?? '');
    const errors = /^[^@]+@[^@]+\.[^@]+$/.test(value)
      ? []
      : [`${field} must be a valid email`];
    next(errors);
  };
}

class ValidationPipeline {
  private validators: ValidatorMiddleware[] = [];
  private allErrors: string[] = [];

  use(validator: ValidatorMiddleware): this {
    this.validators.push(validator);
    return this;
  }

  validate(data: Record<string, unknown>): string[] {
    this.allErrors = [];

    for (const validator of this.validators) {
      validator(data, (errors) => {
        this.allErrors.push(...errors);
      });
    }

    return this.allErrors;
  }
}

const signupValidator = new ValidationPipeline()
  .use(required('name'))
  .use(required('email'))
  .use(isEmail('email'))
  .use(minLength('password', 8));

const errors = signupValidator.validate({
  name: 'Alice',
  email: 'not-valid',
  password: 'short',
});
// ['email must be a valid email', 'password must be at least 8 characters']
```

## State Machine Pattern

### What It Is

A finite state machine (FSM) is a model where:
- A system is always in exactly **one** state from a finite set of states
- **Events** trigger **transitions** from one state to another
- **Actions** can occur on transitions or when entering/exiting states

State machines eliminate the most common class of UI bugs: "impossible states." Instead of scattered boolean flags like `isLoading`, `isError`, `isSuccess`, `isRetrying`, a state machine enforces that only the defined states and transitions are possible.

### The Problem State Machines Solve

```typescript
// Bad: multiple booleans create impossible states
interface FetchState {
  isLoading: boolean;
  isSuccess: boolean;
  isError: boolean;
  data: User | null;
  error: Error | null;
}

// These states are technically possible but should be impossible:
// { isLoading: true, isSuccess: true }
// { isLoading: true, isError: true }
// { isSuccess: true, isError: true }
// { isSuccess: true, data: null }
// { isError: true, error: null }
```

### Simple FSM Implementation

```typescript
type State = 'idle' | 'loading' | 'success' | 'error';
type Event = 'FETCH' | 'SUCCESS' | 'ERROR' | 'RESET';

interface Transition {
  target: State;
  action?: () => void;
}

type TransitionMap = Partial<Record<Event, Transition>>;
type StateMachineConfig = Record<State, TransitionMap>;

class StateMachine<S extends string, E extends string> {
  private current: S;

  constructor(
    private config: Record<S, Partial<Record<E, { target: S; action?: () => void }>>>,
    initial: S,
  ) {
    this.current = initial;
  }

  send(event: E): S {
    const transition = this.config[this.current]?.[event];
    if (!transition) {
      console.warn(`No transition from "${this.current}" on "${event}"`);
      return this.current;
    }
    transition.action?.();
    this.current = transition.target;
    return this.current;
  }

  getState(): S { return this.current; }

  matches(state: S): boolean { return this.current === state; }
}

// Define the fetch machine
const fetchMachine = new StateMachine<State, Event>(
  {
    idle: {
      FETCH: { target: 'loading' },
    },
    loading: {
      SUCCESS: { target: 'success' },
      ERROR: { target: 'error' },
    },
    success: {
      FETCH: { target: 'loading' },
      RESET: { target: 'idle' },
    },
    error: {
      FETCH: { target: 'loading' },
      RESET: { target: 'idle' },
    },
  },
  'idle',
);

// Usage
console.log(fetchMachine.getState()); // 'idle'
fetchMachine.send('FETCH');
console.log(fetchMachine.getState()); // 'loading'
fetchMachine.send('SUCCESS');
console.log(fetchMachine.getState()); // 'success'
fetchMachine.send('ERROR'); // Ignored - no transition from 'success' on 'ERROR'
console.log(fetchMachine.getState()); // 'success' - unchanged
```

### FSM with Context (Extended State)

A pure FSM only tracks the state. An extended state machine adds a *context* object for data that changes continuously (not just discretely):

```typescript
type FetchState = 'idle' | 'loading' | 'success' | 'error';
type FetchEvent =
  | { type: 'FETCH'; url: string }
  | { type: 'SUCCESS'; data: unknown }
  | { type: 'ERROR'; error: Error }
  | { type: 'RESET' };

interface FetchContext {
  url: string | null;
  data: unknown;
  error: Error | null;
  retries: number;
}

class FetchStateMachine {
  private state: FetchState = 'idle';
  private context: FetchContext = {
    url: null,
    data: null,
    error: null,
    retries: 0,
  };

  private listeners: Set<(state: FetchState, ctx: FetchContext) => void> = new Set();

  send(event: FetchEvent): void {
    const next = this.transition(this.state, event);
    if (next) {
      this.state = next.state;
      this.context = { ...this.context, ...next.context };
      this.notify();
    }
  }

  private transition(
    state: FetchState,
    event: FetchEvent,
  ): { state: FetchState; context: Partial<FetchContext> } | null {
    switch (state) {
      case 'idle':
        if (event.type === 'FETCH') {
          return { state: 'loading', context: { url: event.url, error: null } };
        }
        break;

      case 'loading':
        if (event.type === 'SUCCESS') {
          return { state: 'success', context: { data: event.data, retries: 0 } };
        }
        if (event.type === 'ERROR') {
          return {
            state: this.context.retries < 3 ? 'loading' : 'error',
            context: {
              error: event.error,
              retries: this.context.retries + 1,
            },
          };
        }
        break;

      case 'success':
      case 'error':
        if (event.type === 'RESET') {
          return { state: 'idle', context: { url: null, data: null, error: null, retries: 0 } };
        }
        if (event.type === 'FETCH') {
          return { state: 'loading', context: { url: event.url, error: null } };
        }
        break;
    }
    return null; // No valid transition
  }

  onChange(
    listener: (state: FetchState, ctx: FetchContext) => void,
  ): () => void {
    this.listeners.add(listener);
    return () => this.listeners.delete(listener);
  }

  private notify(): void {
    this.listeners.forEach(l => l(this.state, { ...this.context }));
  }

  getState(): FetchState { return this.state; }
  getContext(): Readonly<FetchContext> { return { ...this.context }; }
}
```

### Using the FSM for Async Data Fetching

```typescript
const machine = new FetchStateMachine();

machine.onChange((state, ctx) => {
  // Render based on state, not booleans
  switch (state) {
    case 'idle':
      renderIdle();
      break;
    case 'loading':
      renderLoading(ctx.url);
      break;
    case 'success':
      renderData(ctx.data);
      break;
    case 'error':
      renderError(ctx.error, ctx.retries);
      break;
  }
});

async function fetchUser(id: string): Promise<void> {
  machine.send({ type: 'FETCH', url: `/api/users/${id}` });
  try {
    const response = await fetch(`/api/users/${id}`);
    const data = await response.json();
    machine.send({ type: 'SUCCESS', data });
  } catch (error) {
    machine.send({ type: 'ERROR', error: error as Error });
  }
}
```

### Multi-Step Form State Machine

```typescript
type FormStep = 'personal' | 'address' | 'payment' | 'review' | 'submitted';
type FormEvent = 'NEXT' | 'BACK' | 'SUBMIT' | 'RESET';

interface FormContext {
  personal?: PersonalData;
  address?: AddressData;
  payment?: PaymentData;
}

const formTransitions: Record<FormStep, Partial<Record<FormEvent, FormStep>>> = {
  personal: { NEXT: 'address' },
  address: { NEXT: 'payment', BACK: 'personal' },
  payment: { NEXT: 'review', BACK: 'address' },
  review: { SUBMIT: 'submitted', BACK: 'payment' },
  submitted: { RESET: 'personal' },
};

class FormStateMachine {
  private step: FormStep = 'personal';
  private context: FormContext = {};

  send(event: FormEvent, data?: Partial<FormContext>): void {
    const nextStep = formTransitions[this.step][event];
    if (nextStep) {
      if (data) this.context = { ...this.context, ...data };
      this.step = nextStep;
    }
  }

  getStep(): FormStep { return this.step; }
  getContext(): Readonly<FormContext> { return { ...this.context }; }
  isOn(step: FormStep): boolean { return this.step === step; }

  // Computed properties based on state
  get canGoBack(): boolean { return this.step !== 'personal' && this.step !== 'submitted'; }
  get isComplete(): boolean { return this.step === 'submitted'; }
  get progress(): number {
    const steps: FormStep[] = ['personal', 'address', 'payment', 'review'];
    return Math.round((steps.indexOf(this.step) / (steps.length - 1)) * 100);
  }
}

const form = new FormStateMachine();
console.log(form.progress); // 0

form.send('NEXT', { personal: { name: 'Alice', email: 'alice@example.com' } });
console.log(form.getStep()); // 'address'
console.log(form.progress); // 33

form.send('BACK');
console.log(form.getStep()); // 'personal'
form.send('NEXT');
form.send('NEXT', { address: { street: '123 Main St' } });
console.log(form.getStep()); // 'payment'
```

### XState for Complex Machines

For production applications with complex state logic, XState provides a battle-tested implementation with visual tooling:

```typescript
import { createMachine, createActor } from 'xstate';

const authMachine = createMachine({
  id: 'auth',
  initial: 'unauthenticated',
  context: {
    user: null as User | null,
```

### When to Use a State Machine

State machines pay off when:

- The UI has more than 2-3 boolean states that interact
- Invalid state combinations keep appearing as bugs
- Transitions must be explicit and validated (form steps, auth flows, payment flows)
- You need to handle the same event differently depending on current state
- You want to make impossible states impossible by construction

Do not use them for simple toggle (open/closed), continuous values (scroll position, volume), or UI state that has no business logic.

---

## Common Mistakes

- **Mistake 1: Using IIFE modules in 2026.** Unless you are writing code for an environment without module support, use ES modules. IIFE modules predate native module support and require manual concatenation or namespace management that build tools make unnecessary.

- **Mistake 2: Forgetting to unsubscribe from Pub/Sub.** A subscriber that is never removed holds a reference to its callback in the broker's channel map, preventing garbage collection. Always return and call unsubscribe functions when components are destroyed.

- **Mistake 3: Middleware that calls `next()` multiple times.** Calling `next()` twice in the same middleware causes downstream middleware to execute twice. Build guards into your composition function to detect this, as Koa's implementation does.

- **Mistake 4: Using booleans instead of state machines for complex UI states.** Four booleans (`isLoading`, `isError`, `isSuccess`, `isEmpty`) yield 16 possible combinations, but only 4-5 are meaningful. A state machine with 4 named states makes the invalid combinations physically impossible to represent.

- **Mistake 5: State machine transitions that modify context during validation.** Transitions should be pure calculations. Compute the next state without side effects; apply side effects in `action` functions that run after the transition. Mixing calculation and mutation makes machines hard to test.

- **Mistake 6: Global Pub/Sub for everything.** If two components could be connected by a direct reference or prop, use a direct connection. Over-using Pub/Sub spreads event definitions across the codebase and makes it impossible to trace what happens when an event fires.

---

## Best Practices

- **In ES modules, export the minimum public surface.** Every export is part of your module's API. Keep implementation details unexported. Prefer named exports over default exports for discoverability and refactoring.

- **Type your Pub/Sub channels.** A `PubSubMap` type that maps channel names to their payload types catches typos and wrong payload shapes at compile time, not at runtime.

- **Implement the middleware pattern with error isolation.** A single subscriber or middleware handler that throws should not crash the pipeline or prevent other subscribers from running. Wrap each call in a try/catch at the broker level.

- **Model the full state space before coding the machine.** Draw the states and transitions on paper first. Every valid state, every event, every resulting state. The table or diagram becomes the spec; the code just implements it.

- **Use `send()` to trigger state transitions, never mutate state directly.** The entire value of a state machine is that transitions go through a central, validated mechanism. Bypassing it with direct state mutation defeats the purpose.

- **Scope Pub/Sub brokers to features, not to the entire application.** An application-wide event bus with hundreds of channels becomes impossible to reason about. Create separate brokers for logically distinct subsystems (auth, cart, notifications).

### 1. Module Pattern: Configuration Singleton

```typescript
// config.ts - module-level state is the singleton
type Environment = 'development' | 'staging' | 'production';

interface AppConfig {
  env: Environment;
  apiUrl: string;
  featureFlags: Record<string, boolean>;
  logLevel: 'debug' | 'info' | 'warn' | 'error';
}

// Private state
let config: AppConfig | null = null;

// Private function
function resolveEnv(): Environment {
  const env = process.env.NODE_ENV;
  if (env === 'production') return 'production';
  if (env === 'staging') return 'staging';
  return 'development';
}

// Public API
export function initConfig(overrides: Partial<AppConfig> = {}): void {
  const env = resolveEnv();
  config = {
    env,
    apiUrl: env === 'production'
      ? 'https://api.example.com'
      : 'http://localhost:3000',
    featureFlags: {
      newDashboard: env !== 'production',
      betaEditor: false,
    },
    logLevel: env === 'production' ? 'warn' : 'debug',
    ...overrides,
  };
}

export function getConfig(): Readonly<AppConfig> {
  if (!config) throw new Error('Config not initialized. Call initConfig() first.');
  return config;
}

export function isFeatureEnabled(flag: string): boolean {
  return getConfig().featureFlags[flag] ?? false;
}
```

### 2. Middleware: Form Processing Pipeline

```typescript
type FormData = Record<string, string>;
type ProcessorMiddleware = (
  data: FormData,
  errors: string[],
  next: () => void,
) => void;

class FormProcessor {
  private pipeline: ProcessorMiddleware[] = [];

  use(middleware: ProcessorMiddleware): this {
    this.pipeline.push(middleware);
    return this;
  }

  process(raw: FormData): { data: FormData; errors: string[] } {
    const data = { ...raw };
    const errors: string[] = [];

    const dispatch = (index: number) => {
      if (index < this.pipeline.length) {
        this.pipeline[index](data, errors, () => dispatch(index + 1));
      }
    };

    dispatch(0);
    return { data, errors };
  }
}

// Middleware functions
const trimStrings: ProcessorMiddleware = (data, _, next) => {
  Object.keys(data).forEach(key => {
    data[key] = data[key].trim();
  });
  next();
};

const normalizeEmail: ProcessorMiddleware = (data, _, next) => {
  if (data.email) data.email = data.email.toLowerCase();
  next();
};

const validateRequired = (fields: string[]): ProcessorMiddleware =>
  (data, errors, next) => {
    fields.forEach(field => {
      if (!data[field]) errors.push(`${field} is required`);
    });
    next();
  };

const validatePasswordMatch: ProcessorMiddleware = (data, errors, next) => {
  if (data.password !== data.passwordConfirm) {
    errors.push('Passwords do not match');
  }
  next();
};

// Build and use the pipeline
const signupPipeline = new FormProcessor()
  .use(trimStrings)
  .use(normalizeEmail)
  .use(validateRequired(['name', 'email', 'password']))
  .use(validatePasswordMatch);

const result = signupPipeline.process({
  name: '  Alice  ',
  email: 'ALICE@EXAMPLE.COM',
  password: 'pass123',
  passwordConfirm: 'pass456',
});
// { data: { name: 'Alice', email: 'alice@example.com', ... }, errors: ['Passwords do not match'] }
```

### 3. State Machine: Media Player

```typescript
type PlayerState = 'idle' | 'loading' | 'playing' | 'paused' | 'ended' | 'error';
type PlayerEvent = 'LOAD' | 'LOADED' | 'PLAY' | 'PAUSE' | 'END' | 'SEEK' | 'ERROR' | 'RESET';

const playerTransitions: Record<PlayerState, Partial<Record<PlayerEvent, PlayerState>>> = {
  idle: {
    LOAD: 'loading',
  },
  loading: {
    LOADED: 'paused', // Go to paused - wait for explicit PLAY
    ERROR: 'error',
  },
  playing: {
    PAUSE: 'paused',
    END: 'ended',
    SEEK: 'playing', // Seeking while playing stays playing
    ERROR: 'error',
  },
  paused: {
    PLAY: 'playing',
    SEEK: 'paused', // Seeking while paused stays paused
    RESET: 'idle',
  },
  ended: {
    PLAY: 'playing', // Replay from beginning
    RESET: 'idle',
    LOAD: 'loading', // Load a new track
  },
  error: {
    RESET: 'idle',
    LOAD: 'loading', // Try again
  },
};

class MediaPlayerMachine {
  private state: PlayerState = 'idle';
  private changeListeners = new Set<(state: PlayerState) => void>();

  send(event: PlayerEvent): void {
    const next = playerTransitions[this.state]?.[event];
    if (!next) return; // Invalid transition silently ignored
    this.state = next;
    this.changeListeners.forEach(l => l(this.state));
  }

  onStateChange(listener: (state: PlayerState) => void): () => void {
    this.changeListeners.add(listener);
    return () => this.changeListeners.delete(listener);
  }

  // UI helpers
  get canPlay(): boolean { return this.state === 'paused' || this.state === 'ended'; }
  get canPause(): boolean { return this.state === 'playing'; }
  get isBuffering(): boolean { return this.state === 'loading'; }
  get showReplay(): boolean { return this.state === 'ended'; }
}

const player = new MediaPlayerMachine();

const unsubscribe = player.onStateChange(state => {
  // Update UI based on valid states only
  document.querySelector('.play-btn')?.toggleAttribute('disabled', !player.canPlay);
  document.querySelector('.pause-btn')?.toggleAttribute('disabled', !player.canPause);
  document.querySelector('.spinner')?.classList.toggle('hidden', !player.isBuffering);
});

// Control flow is explicit and enforced
player.send('LOAD');       // idle -> loading
player.send('LOADED');     // loading -> paused
player.send('PLAY');       // paused -> playing
player.send('PAUSE');      // playing -> paused
player.send('PLAY');       // paused -> playing
player.send('END');        // playing -> ended
player.send('PAUSE');      // Invalid: no transition from ended on PAUSE - silently ignored
player.send('PLAY');       // ended -> playing (replay)
```

> **Key Takeaway:** The Module pattern controls what is public and what is private; Pub/Sub decouples publishers from subscribers through a named channel broker; Middleware composes cross-cutting concerns into a sequential pipeline with `next()`; State machines make implicit boolean logic explicit by naming all valid states and the only paths between them.
