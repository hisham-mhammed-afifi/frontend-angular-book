# 9.1 Git Basics

> Git is a distributed version control system that tracks changes to files over time, enabling collaboration, safe experimentation, and the ability to recover from any mistake.

---

## The Three Areas of Git

Every Git workflow revolves around three areas. Understanding where your files live at any moment is the foundation of everything else.

```
Working Tree         Staging Area (Index)     Local Repository
(your files on disk) (prepared snapshot)      (.git/objects)

[Modified files] --> git add --> [Staged files] --> git commit --> [Commits]
                  <-- git restore --         <-- git reset HEAD~1 --
```

### Working Tree

The working tree is the directory you edit in your editor. Files here are either:

- **Untracked**: Git has never seen them before (`git add` was never called)
- **Modified**: Tracked files that have changed since the last commit
- **Unmodified**: Tracked files with no changes since the last commit

### Staging Area (Index)

The staging area is an intermediate snapshot. When you `git add` a file, Git copies its current content into the index. The next commit will contain exactly what is in the index at commit time, not what is on disk.

This gives you fine-grained control: you can stage part of a file (`git add -p`), leave other changes unstaged, and create a precise, focused commit.

### Local Repository

The repository lives in the hidden `.git/` directory. Every `git commit` writes a permanent object into `.git/objects/`. The commit captures the full state of the index at that moment plus metadata (author, message, timestamp, parent commit reference).

## Initializing and Configuring Git

### Create a Repository

```bash
# Start a new repository in the current directory
git init

# Or clone an existing remote repository
git clone https://github.com/user/repo.git

# Clone into a specific directory name
git clone https://github.com/user/repo.git my-project
```

`git init` creates a `.git/` directory. No files are tracked yet.

### Global Configuration

Set your identity once globally. This metadata is embedded in every commit you make:

```bash
git config --global user.name "Alice Smith"
git config --global user.email "alice@example.com"

# Set your default editor (used for commit messages)
git config --global core.editor "code --wait"   # VS Code
git config --global core.editor "vim"            # Vim

# Set the default branch name for new repositories
git config --global init.defaultBranch main

# View all configuration
git config --list

# View where a specific setting comes from
git config --show-origin user.email
```

Configuration has three scopes, each overriding the previous:
1. **System** (`/etc/gitconfig`): Applies to all users on the machine
2. **Global** (`~/.gitconfig`): Applies to all repositories for your user
3. **Local** (`.git/config`): Applies only to the current repository

## Tracking Files

### The Status Lifecycle

```bash
git status          # Full status output
git status -s       # Short format: two-column codes
```

Short format codes:
```
?? untracked.txt      - untracked file
 M modified.txt       - modified, not staged (right column = working tree)
M  staged.txt         - modified and staged (left column = staging area)
MM both.txt           - modified, partially staged, then modified again
A  new-file.txt       - new file staged for addition
D  deleted.txt        - deletion staged
```

### Adding Files

```bash
git add file.txt              # Stage a specific file
git add src/                  # Stage all changes in a directory
git add .                     # Stage all changes in current directory and below
git add *.ts                  # Stage all .ts files using glob

# Stage parts of a file interactively (hunk-by-hunk)
git add -p file.txt
```

`git add -p` is one of the most underused Git features. It shows each changed block (hunk) and asks whether to stage it. Use `y` to stage, `n` to skip, `s` to split the hunk further, `e` to edit it manually.

```
diff --git a/src/user.ts b/src/user.ts
@@ -10,6 +10,10 @@ export class UserService {
   getUser(id: string): User {
+    // Added null check
+    if (!id) throw new Error('ID required');
     return this.db.findById(id);
   }
+
+  deleteUser(id: string): void { ... }

Stage this hunk [y,n,q,a,d,s,e,?]?
```

### Removing and Moving Files

```bash
# Remove a file from tracking AND from disk
git rm file.txt

# Remove from tracking only (keep file on disk - useful for files accidentally committed)
git rm --cached secret.env

# Move/rename a tracked file
git mv old-name.ts new-name.ts
# Equivalent to:
# mv old-name.ts new-name.ts
# git rm old-name.ts
# git add new-name.ts
```

## Making Commits

### The Commit Command

```bash
# Open editor for a detailed message
git commit

# Inline message (for short changes)
git commit -m "fix: validate email before saving"

# Stage all tracked modified files and commit in one step
# (does NOT add untracked files)
git commit -am "refactor: extract user validation logic"
```

### Writing Good Commit Messages

A commit message is documentation. Future you (or a colleague) will read it when trying to understand why a change was made.

**The Seven Rules (Chris Beams)**

1. Separate subject from body with a blank line
2. Limit the subject line to 50 characters
3. Capitalize the subject line
4. Do not end the subject line with a period
5. Use the imperative mood ("Add feature" not "Added feature")
6. Wrap the body at 72 characters
7. Use the body to explain *what* and *why*, not *how*

```
# Good commit message structure:
Fix email validation bypassing empty string check

The validator accepted empty strings as valid emails because
the regex anchors were not strict. This caused silent failures
when users submitted forms with blank email fields.

The fix tightens the regex to require at least one character
before and after the @ symbol.

Fixes: #472
```

### Bad Commit Messages

```bash
# Bad: no context - what was fixed?
git commit -m "fix bug"

# Bad: describes the diff, not the intent
git commit -m "changed line 47 in user.ts"

# Bad: umbrella commit - multiple unrelated changes
git commit -m "add user feature, fix login bug, update styles, add tests"

# Good: one logical change
git commit -m "feat: add email uniqueness check on registration"
```

### Conventional Commits

A widely adopted specification that makes commit messages machine-parseable (for changelogs, semantic versioning, release notes):

```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

**Types:**
| Type | When to use |
|------|-------------|
| `feat` | A new feature |
| `fix` | A bug fix |
| `docs` | Documentation only changes |
| `style` | Formatting, whitespace (no logic change) |
| `refactor` | Code restructuring without behavior change |
| `test` | Adding or correcting tests |
| `perf` | Performance improvements |
| `chore` | Build process, tooling, dependency updates |
| `ci` | CI/CD configuration changes |

```bash
# Examples:
git commit -m "feat(auth): add JWT refresh token rotation"
git commit -m "fix(cart): prevent negative quantity on decrement"
git commit -m "docs(readme): add Docker setup instructions"
git commit -m "refactor(api): extract validation into middleware"
git commit -m "chore(deps): upgrade TypeScript to 5.6"

# Breaking change (adds BREAKING CHANGE footer or ! after type)
git commit -m "feat!: change API response shape for /users endpoint

BREAKING CHANGE: The user object now uses camelCase keys.
Previously returned { first_name }, now returns { firstName }.
Clients must update their response parsers."
```

### Atomic Commits

An atomic commit does exactly one logical thing. It can be applied or reverted independently without breaking anything else.

```bash
# Bad: one commit does three things
git commit -m "add user profile page, fix signup bug, update button styles"

# Good: three separate atomic commits
git commit -m "fix(auth): prevent duplicate user creation on double-submit"
git commit -m "feat(profile): add user profile page with avatar upload"
git commit -m "style(buttons): align button padding with design system"
```

If you find yourself writing "and" or "also" in a commit message, the commit probably contains more than one logical change.

## Inspecting History and Changes

### git log

```bash
# Full log
git log

# Compact one-line format
git log --oneline

# Visual graph (useful for branches)
git log --oneline --graph --all

# Filter by author
git log --author="Alice"

# Filter by date
git log --since="2025-01-01" --until="2025-03-01"

# Filter by file
git log -- src/user.ts

# Search commit messages
git log --grep="fix(auth)"

# Show what changed in each commit
git log -p

# Limit to last 5 commits
git log -5

# Show stats (files changed, insertions, deletions)
git log --stat
```

### git diff

`git diff` answers different questions depending on the flags:

```bash
# What changed in the working tree that is NOT staged yet?
git diff

# What is staged and will go into the next commit?
git diff --staged
# Alias:
git diff --cached

# What changed between working tree and last commit (staged + unstaged)?
git diff HEAD

# What changed between two commits?
git diff abc123 def456

# What changed between two commits for a specific file?
git diff abc123 def456 -- src/user.ts

# What changed between two branches?
git diff main feature/new-login

# Summary: just file names and change type
git diff --name-status HEAD~3 HEAD
```

### git show

```bash
# Show the details of the most recent commit
git show

# Show a specific commit
git show abc1234

# Show a specific file at a specific commit
git show abc1234:src/user.ts

# Show only changed files for a commit
git show --name-only abc1234
```

### git blame

```bash
# See which commit last touched each line of a file
git blame src/user.ts

# Blame with short commit hash
git blame -s src/user.ts

# Blame a specific range of lines
git blame -L 10,25 src/user.ts

# Ignore whitespace changes
git blame -w src/user.ts
```

## The .gitignore File

`.gitignore` tells Git which files to never track. Patterns are matched against file paths relative to the `.gitignore` file's location.

```gitignore
# Ignore a specific file
.env
secret.key

# Ignore a directory
node_modules/
dist/
.cache/

# Ignore all .log files
*.log

# Ignore all .tmp files anywhere in the tree
**/*.tmp

# Ignore files in a specific location only
src/generated/

# Negate: track this file even though *.log is ignored
!important.log

# Ignore files with a specific extension in a specific directory
coverage/*.xml
```

Common patterns for JavaScript/TypeScript projects:

```gitignore
# Dependencies
node_modules/

# Build output
dist/
build/
.next/
out/

# Environment files (NEVER commit secrets)
.env
.env.local
.env.production

# Editor artifacts
.vscode/
.idea/
*.swp

# OS artifacts
.DS_Store
Thumbs.db

# Test coverage
coverage/
.nyc_output/

# TypeScript cache
*.tsbuildinfo
```

**Important gotcha:** `.gitignore` only ignores *untracked* files. If a file was already committed, adding it to `.gitignore` does nothing. You must first untrack it:

```bash
# Remove a tracked file from tracking (but keep it on disk)
git rm --cached .env

# Then commit the removal
git commit -m "chore: remove accidentally committed .env file"
```

## Undoing Changes

### Restoring Files (Non-Destructive)

```bash
# Discard unstaged changes in working tree (restore to staged version)
git restore file.txt

# Unstage a file (move from staging area back to working tree)
git restore --staged file.txt

# Both: unstage AND discard working tree changes
git restore --staged --worktree file.txt

# Restore to the version from a specific commit
git restore --source=HEAD~2 file.txt
```

`git restore` was introduced in Git 2.23 (2019) to give explicit, named commands for operations that were previously done with the confusing overloaded `git checkout`.

### Amending the Last Commit

```bash
# Fix the commit message of the last commit
git commit --amend -m "fix: correct the commit message"

# Add a forgotten file to the last commit
git add forgotten-file.ts
git commit --amend --no-edit  # keeps the existing message

# Change the author of the last commit
git commit --amend --author="Alice Smith <alice@example.com>"
```

**Critical rule:** Only amend commits that have NOT been pushed to a shared remote. Amending rewrites history. Pushing an amended commit over an existing shared commit requires a force push, which disrupts other collaborators.

### git reset

`git reset` moves the current branch pointer backward to a previous commit. It has three modes that differ in what happens to the changes that were "undone":

```bash
# --soft: undo commit(s), keep changes staged
# Use: when you committed too early and want to add more
git reset --soft HEAD~1
# Result: last commit removed, changes are in staging area

# --mixed (default): undo commit(s), keep changes unstaged
# Use: when you want to recommit differently
git reset HEAD~1
git reset --mixed HEAD~1
# Result: last commit removed, changes are in working tree (unstaged)

# --hard: undo commit(s), DISCARD all changes
# Use: when you want to completely throw away recent commits
git reset --hard HEAD~3
# Result: last 3 commits removed, changes are GONE from working tree

# Reset to a specific commit
git reset --hard abc1234
```

`HEAD~1` means "one commit before HEAD". `HEAD~3` means "three commits before HEAD". `HEAD^` is equivalent to `HEAD~1`.

**Warning:** `git reset --hard` discards uncommitted changes permanently. There is no undo (except via `git reflog` within 90 days). Always make sure you have nothing valuable in the working tree before a hard reset.

### git revert

`git revert` creates a new commit that undoes the changes of a previous commit. Unlike `reset`, it does not rewrite history -- the original commit remains, and a new "undo" commit is added on top.

```bash
# Revert the most recent commit
git revert HEAD

# Revert a specific commit by its hash
git revert abc1234

# Revert without immediately committing (review first)
git revert --no-commit abc1234
# Make further changes if needed
git commit -m "revert: undo broken auth middleware"

# Revert a range of commits
git revert HEAD~3..HEAD
```

Use `revert` for undoing changes on shared/public branches. Use `reset` only on local, private branches.

### Comparison Table

| Command | What It Undoes | Rewrites History? | Safe for Shared Branches? |
|---------|---------------|-------------------|--------------------------|
| `git restore file` | Working tree changes | No | Yes |
| `git restore --staged file` | Staged changes | No | Yes |
| `git commit --amend` | Last commit | Yes (local only) | No (before push) |
| `git reset --soft` | Commits (keeps staged) | Yes | No |
| `git reset --mixed` | Commits (keeps unstaged) | Yes | No |
| `git reset --hard` | Commits + all changes | Yes | No |
| `git revert` | Commits (new undo commit) | No | Yes |

## The Reflog: Git's Safety Net

The reflog (`reference log`) records every movement of HEAD and branch pointers. It is strictly local -- it is not pushed to remotes. Entries expire after 90 days by default.

```bash
# View the reflog
git reflog

# Example output:
# abc1234 HEAD@{0}: commit: feat: add user profile
# def5678 HEAD@{1}: reset: moving to HEAD~1
# ghi9012 HEAD@{2}: commit: fix: email validation
# jkl3456 HEAD@{3}: checkout: moving from feature to main

# Recover a "lost" commit after a hard reset
git reset --hard HEAD~3  # Uh oh
git reflog               # Find the commit hash
git reset --hard abc1234 # Restore to it

# Or create a new branch pointing to a lost commit
git branch recovery-branch abc1234
```

The reflog is your safety net. Before any scary operation (`reset --hard`, `rebase`, `force push`), know that the reflog can likely recover your work.

## Common Mistakes

- **Mistake 1: Committing sensitive files.** `.env` files, API keys, passwords, and certificates committed once are in the permanent history, even if later deleted. They must be considered compromised. Use `git rm --cached` to untrack them, add them to `.gitignore`, and rotate the credentials.

- **Mistake 2: Using `git add .` without reviewing.** `git add .` stages every change including files you may not have intended to include (debug code, test data, generated files). Use `git status` and `git diff` before staging, or use `git add -p` for granular control.

- **Mistake 3: Mega commits.** Committing a full day or week of work in a single commit ("WIP", "stuff", "updates") makes every Git operation harder: bisecting, reverting, reviewing, and understanding history. Commit frequently with focused changes.

- **Mistake 4: Amending pushed commits.** `git commit --amend` rewrites history. If you amend a commit that others have already pulled, their local histories diverge from yours. The result is merge conflicts and confusion when anyone tries to push or pull.

- **Mistake 5: Forgetting `--staged` in git diff.** `git diff` without flags shows unstaged changes. It is easy to think everything is correct and commit, only to discover staged content differs from what you inspected. Always run `git diff --staged` before committing.

- **Mistake 6: `.gitignore` not working on tracked files.** Adding a file to `.gitignore` after it has already been committed does nothing. You must `git rm --cached <file>` to remove it from tracking first.

## Best Practices

- **Commit often, push thoughtfully.** Make small commits locally as you work. Consolidate or clean up before pushing to a shared branch. Frequent commits give you fine-grained undo points.

- **Always run `git status` before committing.** A quick `git status` prevents accidental inclusion of unintended files. Make it a reflex before every commit.

- **Write commit messages for the future.** Ask yourself: "If I read this commit message in six months, will I know why this change was made?" The answer should be yes.

- **Use `git add -p` for logical commits.** If you made multiple logical changes in one editing session, `git add -p` lets you create separate, focused commits for each without needing to undo and redo work.

- **Keep `.gitignore` updated from the start.** Add a `.gitignore` before the first commit. Use `gitignore.io` or GitHub's language-specific templates as a starting point.

- **Use `git stash` for temporary context switching.** When you need to switch tasks before finishing current work, `git stash` saves your working tree and staging area to a stack without committing:

```bash
git stash          # Save current work
git stash pop      # Restore most recent stash
git stash list     # See all stashes
git stash push -m "half-finished user form" # Named stash
```

## Real-World Use Cases

### 1. Cleaning Up a Work Session Before Pushing

You have been working for hours and made many small saves. Before pushing, you want the history to tell a coherent story:

```bash
# See what you have
git log --oneline
# abc1234 WIP
# def5678 fix typo
# ghi9012 more stuff
# jkl3456 initial user form

# Review all changes since the last pushed commit
git diff origin/main HEAD

# Use interactive rebase to squash/reorder (covered in 9.5)
# For now, check that each commit is atomic:
git show abc1234 --name-only
git show def5678 --name-only

# If a commit message needs improvement:
# (only if not yet pushed)
git rebase -i HEAD~4  # Opens interactive editor for last 4 commits
```

### 2. Recovering from a Mistake

You ran `git reset --hard HEAD~2` and realized you needed one of those commits:

```bash
# Find the lost commits in the reflog
git reflog
# abc1234 HEAD@{0}: reset: moving to HEAD~2
# def5678 HEAD@{1}: commit: feat: user avatar upload  <- want this one
# ghi9012 HEAD@{2}: commit: fix: avatar size validation <- and this

# Option 1: restore the branch to before the reset
git reset --hard def5678

# Option 2: cherry-pick specific commits onto current branch
git cherry-pick def5678  # brings just that one commit
git cherry-pick ghi9012
```

### 3. Staging Only Part of a File

You added a debug `console.log` while fixing a bug. You want to commit only the fix, not the debug statement:

```bash
git add -p src/auth/login.ts
# Git shows each hunk (changed block)

# Hunk 1: the actual bug fix
# Stage this hunk [y,n,q,a,d,s,e,?]? y

# Hunk 2: the console.log you added
# Stage this hunk [y,n,q,a,d,s,e,?]? n

# Commit only the fix
git commit -m "fix(auth): prevent session token reuse after logout"

# Remove the debug log separately
git restore src/auth/login.ts  # discard unstaged changes
# Or keep it if you still need it for further debugging
```

```bash
# Verify the commit contains only the fix
git show --stat HEAD
git diff HEAD~1 HEAD -- src/auth/login.ts
```

---

> **Key Takeaway:** Git's three-area model (working tree, staging area, repository) gives you fine-grained control over what gets committed; mastering `git add -p`, `git diff --staged`, and the distinction between `restore`/`reset`/`revert` turns Git from a save button into a precision tool for narrating the story of your codebase.
