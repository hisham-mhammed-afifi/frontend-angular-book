# 9.4 Collaboration and Pull Requests

> Pull requests are the primary unit of collaboration in distributed teams -- they are where code gets reviewed, knowledge gets shared, and quality gets enforced.

---

## What a Pull Request Actually Is

A pull request (PR) is not a Git concept. Git itself only knows about branches, commits, and merges. A pull request is a feature provided by hosting platforms (GitHub, GitLab, Bitbucket) that wraps a proposed branch merge with tooling: a discussion thread, line-by-line comments, CI status checks, and approval gates.

The name comes from the original workflow: you ask the maintainer to pull your changes into their branch. In GitLab, the same concept is called a "merge request" (MR), which is arguably more accurate.

At its core, a PR is:
1. A comparison between a source branch (your feature) and a target branch (main, develop, etc.)
2. A forum for discussion and review
3. A gate through which changes must pass before merging

## The Fork and Pull Request Workflow

The fork workflow is the standard for contributing to open-source projects and is also used in organizations that want contributors to work in isolated repository copies.

### Step 1: Fork the repository

On GitHub, click "Fork" to create a copy under your account. This gives you full write access to your copy without needing permissions on the original.

```bash
# After forking on GitHub, clone YOUR fork
git clone https://github.com/your-username/project.git
cd project

# Add the original repository as "upstream"
git remote add upstream https://github.com/original-owner/project.git

# Verify
git remote -v
# origin    https://github.com/your-username/project.git (fetch)
# origin    https://github.com/your-username/project.git (push)
# upstream  https://github.com/original-owner/project.git (fetch)
# upstream  https://github.com/original-owner/project.git (push)
```

### Step 2: Sync your fork before starting work

Never start new work from a stale fork. Always sync first.

```bash
# Fetch changes from the original repository
git fetch upstream

# Update your local main to match upstream
git switch main
git merge upstream/main

# Push the updated main to your fork
git push origin main
```

### Step 3: Create a feature branch

Never work directly on main in your fork. Use a dedicated branch per feature or fix.

```bash
git switch -c feature/add-dark-mode
```

### Step 4: Do your work and push

```bash
# Make commits
git add -p
git commit -m "feat: add dark mode toggle to settings panel"

# Push to YOUR fork (origin), not upstream
git push origin feature/add-dark-mode
```

### Step 5: Open a pull request

Go to your fork on GitHub. GitHub will detect your recently pushed branch and show a "Compare & pull request" button. The PR targets `upstream/main` from `origin/feature/add-dark-mode`.

## Keeping Your Branch Up to Date

Long-lived branches drift from main. Conflicts accumulate. The longer you wait to sync, the worse it gets.

### Rebasing your branch onto the updated main

```bash
# Fetch latest from upstream
git fetch upstream

# Rebase your feature branch on top of the latest main
git switch feature/add-dark-mode
git rebase upstream/main

# If conflicts occur, resolve them, then:
git add <resolved-file>
git rebase --continue

# Force-push the rebased branch to your fork
# Use --force-with-lease for safety
git push origin feature/add-dark-mode --force-with-lease
```

Rebase rewrites your branch history so it appears as if you started from the latest main. This produces a clean, linear history when the PR is eventually merged.

### Merging main into your branch (alternative)

```bash
git switch feature/add-dark-mode
git fetch upstream
git merge upstream/main
git push origin feature/add-dark-mode
```

This adds a merge commit but avoids rewriting history. Easier, less risky, slightly messier history.

| Approach | When to use |
|----------|-------------|
| Rebase | Open-source, clean history required, branch not shared |
| Merge | Team branch shared with others, or when rebase conflicts are complex |

## Anatomy of a Good Pull Request

### Title

The title is the first thing reviewers see. It should be specific and summarize the change:

```
# Bad
Fix bug

# Bad
Update code

# Good
fix(auth): prevent token refresh race condition on concurrent requests

# Good
feat(dashboard): add CSV export for date-filtered reports
```

Using Conventional Commits format in PR titles makes changelogs easier to generate.

### Description

A good PR description answers three questions:
- **What** changed?
- **Why** did it change?
- **How** was it tested?

```markdown
## What

Adds a dark mode toggle to the user settings panel. The selected
preference is stored in localStorage and applied on page load.

## Why

Users requested dark mode in #342. The current white-only interface
causes eye strain in low-light environments.

## How to test

1. Navigate to Settings > Appearance
2. Toggle "Dark Mode"
3. Verify the page switches to the dark theme immediately
4. Refresh the page and verify the preference persists

## Screenshots

[Before] [After] (attach images or screen recordings)

## Checklist

- [x] Unit tests added/updated
- [x] No console.log left in code
- [x] Accessible (keyboard navigable)
- [ ] Design reviewed by @design-team
```

### Size

Small PRs get reviewed faster and more thoroughly. Large PRs get rubber-stamped or sit in review limbo.

Target: under 400 lines changed. If a feature requires more, split it into a sequence of PRs where each one builds on the previous.

```
# Splitting a large feature:
PR 1: feat(auth): add database schema for OAuth tokens
PR 2: feat(auth): implement token storage service
PR 3: feat(auth): add OAuth login flow
PR 4: feat(auth): add OAuth logout and token revocation
PR 5: feat(auth): add refresh token rotation
```

## Draft Pull Requests

A draft PR signals that the work is in progress and is not ready for full review. It is useful for:
- Getting early feedback on approach before completing the implementation
- Running CI checks on incomplete work
- Making work visible to the team without requesting review

```bash
# On GitHub: click the dropdown next to "Create pull request"
# and choose "Create draft pull request"

# Or via GitHub CLI:
gh pr create --draft --title "feat: OAuth integration" --body "WIP"

# When ready for review:
gh pr ready
```

On GitLab, prefix the title with `Draft:` to achieve the same effect.

## The Review Process

### Assigning reviewers

Who reviews a PR matters. Assign:
- Someone who owns or frequently modifies the affected code
- Someone with domain expertise for the type of change (security, performance, accessibility)
- No more than 2-3 reviewers -- more reviewers diffuses responsibility

### CODEOWNERS

`CODEOWNERS` is a file that automatically requests reviews from specified owners when files they own are changed in a PR.

```
# .github/CODEOWNERS (GitHub)
# or
# .gitlab/CODEOWNERS (GitLab)

# Format: pattern  owner(s)
*                  @lead-developer           # everyone must review by default
*.ts               @typescript-team
src/auth/          @security-team
src/payments/      @payments-team @security-team
docs/              @documentation-team
```

When a PR touches `src/auth/`, GitHub/GitLab automatically adds `@security-team` as required reviewers. This ensures the right people review the right code without having to remember who to tag.

### Review states

GitHub offers three review actions:

| Action | When to use |
|--------|-------------|
| **Comment** | Asking questions, leaving suggestions, non-blocking feedback. The PR can still be merged. |
| **Approve** | The code is ready to merge as-is. |
| **Request changes** | There are blockers that must be fixed before the PR can merge. |

**Bad practice:** Using "Comment" when you mean "Request changes" because you don't want to seem harsh. This is ambiguous -- the author doesn't know if the comment is blocking or optional. Be explicit.

### Writing good review comments

Be specific, not vague:

```
# Bad
This looks wrong.

# Good
This will throw when `user` is null (line 42). The API returns null
for deleted accounts. Add a null check here or handle it upstream
in the service layer.
```

Distinguish between blocking and non-blocking:

```
# Blocking (must fix before merge)
[BLOCKING] This introduces an N+1 query in the getUserList endpoint.
The `user.posts` call inside the loop will fire a separate database
query for each user. Use a JOIN or batch load instead.

# Suggestion (non-blocking, your call)
[NIT] Consider using `Array.from()` here instead of the spread
operator for slightly better readability.

# Question (informational, not requiring change)
[QUESTION] Why is this set to 60 seconds? Is that intentional or
a placeholder?
```

Suggest code changes directly:

On GitHub, reviewers can use the "suggestion" feature to propose exact code changes that the author can commit with one click:

```
# In a review comment, wrap the suggestion in:
```suggestion
const timeout = 30_000; // 30 seconds
```
```

This reduces round-trips. Instead of "please change X", you propose the exact change and the author applies it.

### Responding to review comments

As the author:
- Acknowledge every comment, even if you disagree
- If you make the requested change, mark the comment as resolved
- If you disagree, explain why with reasoning, not defensiveness

```
# Bad response to a blocking comment
This is fine, it works.

# Good response
Thanks -- good catch. I have added the null check in commit abc123.
The API docs say this field is always present, but you are right
that we should not trust the API contract without validation.
```

## Merge Strategies

When merging a PR, platforms offer different strategies. Each produces a different history.

### Merge commit (standard merge)

```
main:    A---B---C---M
                     |
feature:         D---E
```

Preserves the full branch history. `M` is a merge commit that combines the two branches. `git log` will show the branch structure.

```bash
# Equivalent command:
git merge feature-branch --no-ff
```

**When to use:** Feature branches where you want to see in history that this batch of commits was part of a single feature.

### Squash and merge

```
Before:
main:    A---B---C
feature:         D---E---F (three commits, including fixups)

After squash merge:
main:    A---B---C---S (single commit S containing all changes from D+E+F)
```

All commits from the feature branch are flattened into one commit on main. The individual commits (`D`, `E`, `F`) are discarded.

```bash
# Equivalent command:
git merge feature-branch --squash
git commit -m "feat: add dark mode toggle"
```

**When to use:** When the feature branch has noisy intermediate commits ("wip", "fix typo", "add logging") that you do not want polluting main history.

**Tradeoff:** You lose the individual commit granularity. Bisecting a regression becomes harder when a single squashed commit contains a large change.

### Rebase and merge

```
Before:
main:    A---B---C
feature:     D---E---F

After rebase merge:
main:    A---B---C---D'---E'---F'
```

The feature branch commits are replayed on top of main, one by one. Each commit gets a new hash (D', E', F') because their parent changed. Produces a linear history.

```bash
# Equivalent command:
git rebase main feature-branch
git switch main
git merge feature-branch --ff-only
```

**When to use:** When each commit in the feature branch is meaningful and clean, and you want a linear history on main.

### Choosing a strategy

| Strategy | History | Bisect | Revert PR | Common in |
|----------|---------|--------|-----------|-----------|
| Merge commit | Non-linear | Easy | Easy (revert merge commit) | Internal projects |
| Squash | Linear | Harder | Easy (revert squash commit) | Many open source projects |
| Rebase | Linear | Easy | Requires reverting N commits | Smaller teams |

Pick one and enforce it consistently. Mixing strategies makes history hard to read.

## Protected Branches

Protected branches prevent direct pushes to critical branches and enforce review requirements.

### GitHub branch protection rules

Configure via: Repository Settings > Branches > Branch protection rules

Common settings:
- **Require a pull request before merging:** No direct pushes to `main`
- **Require approvals:** Minimum N approvals before merge
- **Dismiss stale approvals when new commits are pushed:** Re-review required after changes
- **Require status checks to pass:** CI must pass before merge
- **Require branches to be up to date before merging:** Must rebase or merge main before merging
- **Restrict who can push:** Only certain teams can merge

```yaml
# GitHub Actions: enforce checks that count as "status checks"
# .github/workflows/ci.yml
name: CI
on: [pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
      - run: npm run build
```

After setting up this workflow, you can require the `test` job to pass before any PR can be merged.

### GitLab protected branches

GitLab's protected branch settings are in Repository > Settings > Repository > Protected branches.

GitLab additionally supports:
- **Merge request approval rules:** Define specific approval conditions (e.g., 2 approvals from `@senior-devs`)
- **Code Owner approval required:** CODEOWNERS approvals are mandatory
- **Require linear history:** No merge commits allowed

## Resolving Conflicts in Pull Requests

When your branch and the target branch have conflicting changes, the PR cannot be merged until conflicts are resolved.

### Resolving locally (recommended)

```bash
# Update your local main
git switch main
git pull

# Switch to your feature branch
git switch feature/add-dark-mode

# Rebase onto the updated main
git rebase main

# Git pauses at each conflicting commit
# CONFLICT (content): Merge conflict in src/settings.ts
# Fix the conflict in your editor, then:
git add src/settings.ts
git rebase --continue

# Push the resolved, rebased branch
git push origin feature/add-dark-mode --force-with-lease
```

What conflict markers look like:

```typescript
// src/settings.ts
function getTheme(): Theme {
<<<<<<< HEAD (main)
  return userPreferences.theme ?? 'light';
=======
  return localStorage.getItem('theme') as Theme ?? 'system';
>>>>>>> feature/add-dark-mode
}
```

Resolution: decide what the correct code should be. You may keep one side, the other, or write something entirely new:

```typescript
function getTheme(): Theme {
  // Feature branch approach: use localStorage
  // But also fall back to user preferences for server-side persistence
  const stored = localStorage.getItem('theme') as Theme;
  return stored ?? userPreferences.theme ?? 'system';
}
```

### Conflict markers reference

```
<<<<<<< HEAD
  Code from the branch you are merging INTO (main, in a rebase this is your branch)
=======
  Code from the branch being merged IN (your feature, in a rebase this is the incoming commit)
>>>>>>> branch-name
```

Note: during a rebase, `HEAD` refers to the upstream branch you are rebasing onto, and the incoming code is your commit. This is the opposite of a regular merge.

### Resolving via GitHub UI

For simple conflicts, GitHub offers an inline editor. Avoid this for complex conflicts -- it is too easy to make mistakes without proper editor support, syntax highlighting, or tests.

## Closing Issues Automatically

GitHub and GitLab can automatically close an issue when a PR that references it is merged.

```
# In the PR title or description, use keywords:
Fixes #123
Closes #456
Resolves #789

# After the PR is merged into the default branch, issue #123 is closed.
```

Supported keywords: `close`, `closes`, `closed`, `fix`, `fixes`, `fixed`, `resolve`, `resolves`, `resolved`.

For cross-repository references:
```
Fixes owner/repo#123
```

## GitHub CLI for Pull Requests

The `gh` CLI provides full PR management from the terminal.

```bash
# Create a PR
gh pr create --title "feat: dark mode" --body "Adds dark mode toggle"

# Create a draft PR
gh pr create --draft

# List open PRs
gh pr list

# View a specific PR
gh pr view 42

# View the PR for the current branch
gh pr view

# Check out a PR locally for review
gh pr checkout 42

# Review a PR
gh pr review 42 --approve
gh pr review 42 --request-changes --body "Please fix the null check"
gh pr review 42 --comment --body "LGTM but one question..."

# Merge a PR
gh pr merge 42 --squash
gh pr merge 42 --merge
gh pr merge 42 --rebase

# Close a PR without merging
gh pr close 42

# Reopen a closed PR
gh pr reopen 42
```

## Common Collaboration Patterns

### The Stacked PRs Pattern

When a feature requires multiple sequential changes, you can stack PRs:

```
main
 |
 +--- PR 1: Add database schema (target: main)
       |
       +--- PR 2: Add service layer (target: PR1's branch)
             |
             +--- PR 3: Add controller (target: PR2's branch)
```

Each PR targets the previous PR's branch, not main. Once PR1 merges, PR2 retargets to main, then PR3, and so on.

Benefit: reviewers can review smaller, focused slices. Early feedback on the schema design does not block the full feature.

Drawback: if an earlier PR requires significant rework, all subsequent PRs need rebasing.

### The Two-Reviewer Pattern

For security-sensitive code (authentication, payments, cryptography), require two independent reviewers, not just one:

```yaml
# .github/CODEOWNERS
src/auth/          @security-reviewer-1 @security-reviewer-2
src/payments/      @security-reviewer-1 @payments-lead
```

Combined with requiring all CODEOWNER reviews, this ensures no single person can approve sensitive changes.

### The Draft-to-Ready Workflow

```
1. Create draft PR immediately after first commit
   -- Makes work visible, allows early feedback
   -- CI runs on every push, catches issues early

2. Self-review before marking ready
   -- Read your own diff as if you were a reviewer
   -- Leave inline comments on non-obvious choices
   -- Remove debug code, console.log, TODO comments

3. Mark as "Ready for review" when complete
   -- All CI checks pass
   -- Self-review done
   -- PR description written

4. Address review comments within 24 hours
   -- Respect reviewers' time
   -- If you need more time, communicate it

5. Merge after approval
   -- Author typically merges, not reviewer
   -- Ensures author takes responsibility for the final state
```

## Common Mistakes

- **PRs that are too large:** A PR with 2,000 line changes will not get a proper review. Reviewers will tire and approve without truly understanding all the changes. Split the work.

- **Waiting until the feature is complete to open a PR:** Open a draft PR on the first commit. Continuous visibility prevents "surprise" PRs at the end of a sprint.

- **Force-pushing to a shared review branch:** If a teammate has checked out your branch to test it locally, force-pushing breaks their local branch. Use `--force-with-lease` and communicate the push.

- **Merging before CI passes:** "The tests only fail because of an unrelated flaky test" is not a justification. Fix or document the flaky test, then merge.

- **Responding to every nit with a commit:** Batching small fixes into a single commit is cleaner than five separate "address review comments" commits. Use `git commit --amend` or `git rebase -i` to clean up, then force-push.

- **Approving code you do not understand:** "LGTM!" after 30 seconds on a 500-line PR is not a review. If you lack context, ask questions rather than rubber-stamping.

- **Writing passive-aggressive review comments:** "Why would anyone write this?" or "This is obviously wrong." Code review is a teaching and collaboration tool, not a place to express frustration.

- **Ignoring the PR description:** Authors spend time writing descriptions and testing instructions. Reviewers who ignore them ask questions already answered, wasting everyone's time.

- **Leaving PRs in review limbo:** A PR that gets one comment, then sits for a week, then gets another comment, then sits again can take months to merge. Teams should establish SLAs: respond to reviews within N hours.

## Best Practices

- **One PR per concern:** A PR should do one thing. Fixing a bug while also refactoring unrelated code in the same PR makes it harder to understand, review, and revert if necessary.

- **Write the description before asking for review:** Reviewing your own PR description often reveals gaps in the implementation or the explanation.

- **Mark comments with priority:** Use prefixes like `[BLOCKING]`, `[NIT]`, `[QUESTION]`, `[SUGGESTION]` to make the severity of each comment immediately clear.

- **Request specific reviewers:** "Team, please review" is less effective than "@alice for the database changes and @bob for the API contract changes."

- **Use commit hygiene in the PR branch:** Even if you plan to squash, meaningful commits during the PR help reviewers follow the story of the change when reviewing commit by commit.

- **Test the changes yourself before requesting review:** Run the code, exercise the new behavior, check edge cases. The first review comment should not be "this crashes immediately when X."

- **Keep the CI pipeline fast:** A 20-minute CI run slows every iteration. Parallelize jobs, cache dependencies, run only affected tests where possible.

- **Establish a team review SLA:** Agree on a maximum time before responding to a review request (e.g., by end of next business day). Unreviewed PRs block shipping.

## Real-World Use Cases

### Contributing to an open-source project

```bash
# 1. Fork on GitHub, then clone your fork
git clone https://github.com/your-username/some-oss-project.git
cd some-oss-project

# 2. Add upstream remote
git remote add upstream https://github.com/original/some-oss-project.git

# 3. Sync before starting any work
git fetch upstream
git switch main
git rebase upstream/main
git push origin main

# 4. Create a branch that clearly names the contribution
git switch -c fix/memory-leak-in-connection-pool

# 5. Make changes following the project's contribution guidelines
# (check CONTRIBUTING.md)

# 6. Commit following the project's commit message format
git commit -m "fix(pool): release connection on timeout to prevent memory leak

Connections that timed out were not being returned to the pool,
causing the pool to exhaust its limit under sustained load.

Fixes #1823"

# 7. Push and open PR
git push origin fix/memory-leak-in-connection-pool
gh pr create --repo original/some-oss-project \
  --title "fix(pool): release connection on timeout" \
  --body "..."
```

### Team workflow with protected main branch

```bash
# Developer starts a feature
git switch main
git pull
git switch -c feat/user-profile-page

# Work, commit, push
git push origin feat/user-profile-page

# Open PR via CLI
gh pr create --title "feat(profile): add user profile page" \
  --body "Adds profile page with avatar upload and bio editing.

  Closes #245"

# Reviewer checks out to test locally
gh pr checkout 187

# After review, make changes requested
git add -A
git commit -m "fix: handle empty bio field validation"
git push origin feat/user-profile-page

# CI passes, approvals received
# Author merges (squash merge to keep main clean)
gh pr merge 187 --squash --delete-branch
```

### Handling a blocking review comment

```
Reviewer: [BLOCKING] The validateToken function does not handle
expired tokens. The `exp` claim is checked, but the clock skew
between services can be up to 5 seconds. This will cause valid
tokens to be rejected near the expiry boundary.

Author: Good catch. I have added a 10-second leeway to the expiry
check to account for clock skew. This is consistent with how
our other services handle it (see auth-service/src/jwt.ts:87).
Updated in commit a4c7d2e.
```

```typescript
// Before (flagged as blocking)
function validateToken(token: string): boolean {
  const decoded = jwt.decode(token);
  const now = Math.floor(Date.now() / 1000);
  return decoded.exp > now;
}

// After (addresses the comment)
const CLOCK_SKEW_SECONDS = 10;

function validateToken(token: string): boolean {
  const decoded = jwt.decode(token);
  const now = Math.floor(Date.now() / 1000);
  // Allow 10s leeway for clock skew between services
  return decoded.exp + CLOCK_SKEW_SECONDS > now;
}
```
