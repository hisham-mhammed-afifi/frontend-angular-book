# 9.5 Advanced Git

> Beyond commits and pushes: the tools that turn Git from a version tracker into a precision debugging and history-management instrument.

## Git Bisect: Binary Search for Regressions

`git bisect` automates the process of finding which commit introduced a bug. Instead of manually checking out commits one by one, Git uses binary search to narrow down the culprit in O(log n) steps.

For 1,000 commits, you need at most 10 tests. For 10,000 commits: 14 tests.

### Manual bisect

```bash
# 1. Start a bisect session
git bisect start

# 2. Mark the current state as bad (the bug exists here)
git bisect bad

# 3. Mark a known-good commit (before the bug existed)
git bisect good v2.3.0
# or by hash:
git bisect good a1b2c3d

# Git checks out the commit halfway between good and bad.
# Bisecting: 512 revisions left to test after this (roughly 9 steps)
# [f4e5d6c] feat: add user profile page

# 4. Test this commit manually. Does the bug exist?
#    If yes:
git bisect bad
#    If no:
git bisect good

# Git checks out the next midpoint. Repeat until:
# f4e5d6c is the first bad commit
# commit f4e5d6c
# Author: Alice <alice@example.com>
# Date:   Mon Jan 13 10:30:00 2025
#
#     feat: add user profile page

# 5. End the session and return to original HEAD
git bisect reset
```

### Automated bisect with `git bisect run`

If you can write a script that exits `0` when the bug is absent and non-zero when the bug is present, Git can run the entire binary search automatically without any interaction.

```bash
# Example: find the commit that broke a specific test
git bisect start
git bisect bad HEAD
git bisect good v2.3.0

# Run bisect automatically using an npm test command
git bisect run npm test -- --testPathPattern="src/auth/token.test.ts"

# Git will run the test at each midpoint and use the exit code
# to decide good (exit 0) or bad (exit non-zero).
# When done, it prints the first bad commit and resets.
```

Custom script example:

```bash
#!/bin/bash
# bisect-test.sh
# Exit 0 = good (bug absent), Exit 1 = bad (bug present)

# Build the project
npm run build --silent 2>/dev/null

# Exit with special code 125 to tell git bisect to skip this commit
# (e.g., the commit doesn't compile and can't be tested)
if [ $? -ne 0 ]; then
  exit 125
fi

# Run the specific failing test
npm test -- --testPathPattern="payment/checkout" --silent 2>/dev/null
exit $?
```

```bash
git bisect run ./bisect-test.sh
```

### Exit codes for `git bisect run`

| Exit code | Meaning |
|-----------|---------|
| 0 | Good -- bug not present |
| 1-124, 126-127 | Bad -- bug is present |
| 125 | Skip -- this commit cannot be tested |
| 128+ | Abort -- stop bisecting immediately |

### Bisect with terms

For regressions that are not bugs (e.g., a performance degradation or a feature being removed), you can use custom terms instead of "good" and "bad":

```bash
git bisect start --term-new=slow --term-old=fast
git bisect slow HEAD
git bisect fast v3.0.0
git bisect run ./perf-check.sh
```

## Git Rerere: Reuse Recorded Resolutions

`rerere` stands for "reuse recorded resolution." When enabled, Git remembers how you resolved a merge conflict and automatically applies the same resolution if the same conflict appears again.

This is invaluable in workflows where you frequently rebase a long-lived feature branch onto main, encountering the same conflicts repeatedly.

### Enable rerere

```bash
# Enable globally (recommended)
git config --global rerere.enabled true

# Enable per repository
git config rerere.enabled true
```

Git stores recorded resolutions in `.git/rr-cache/`.

### How it works in practice

```bash
# First encounter: rebase triggers a conflict
git rebase main

# CONFLICT (content): Merge conflict in src/api/config.ts
# Recorded preimage for 'src/api/config.ts'

# You resolve the conflict manually
vim src/api/config.ts
git add src/api/config.ts
git rebase --continue

# Rerere records the resolution.
# Recorded resolution for 'src/api/config.ts'.

# Second encounter: same conflict, different rebase
git rebase main

# CONFLICT (content): Merge conflict in src/api/config.ts
# Resolved 'src/api/config.ts' using previous resolution.

# Git applied the resolution automatically.
# You still need to stage and continue:
git add src/api/config.ts
git rebase --continue
```

Rerere does not automatically stage resolved files -- it applies the resolution and leaves the file in a resolved state, but you still need to `git add` it. This lets you verify the auto-resolution before committing.

### Rerere commands

```bash
# Show recorded resolutions
git rerere status

# Show the diff of what rerere would apply
git rerere diff

# Forget a recorded resolution (if it was wrong)
git rerere forget src/api/config.ts

# Manually trigger rerere (usually automatic on merge/rebase)
git rerere
```

### When rerere is particularly useful

- Long-lived feature branches that must track main continuously
- Repeated rebasing during code review iterations (fix, rebase, push, fix, rebase...)
- Maintaining release branches that regularly cherry-pick from main
- Open-source maintainers who apply the same patches across multiple version branches

## Git Attributes

`.gitattributes` controls how Git treats specific files: line endings, diff behavior, merge strategies, and binary/text classification.

### Line ending normalization

Mixed line endings (CRLF on Windows, LF on Unix) are a common source of spurious diffs. `.gitattributes` standardizes them:

```
# .gitattributes

# Normalize all text files to LF in the repository
# Check out with native line endings on each platform
* text=auto

# Force LF everywhere (recommended for cross-platform teams)
*.ts    text eol=lf
*.js    text eol=lf
*.json  text eol=lf
*.md    text eol=lf
*.css   text eol=lf
*.html  text eol=lf

# Force CRLF (Windows-specific files)
*.bat   text eol=crlf
*.cmd   text eol=crlf

# Treat as binary -- no line-ending conversion, no diff
*.png   binary
*.jpg   binary
*.gif   binary
*.ico   binary
*.woff  binary
*.woff2 binary
*.pdf   binary
*.zip   binary
```

### Custom diff drivers

By default, `git diff` shows raw diffs for all files. You can configure custom diff drivers for specific file types:

```
# .gitattributes
*.png diff=exif     # show image EXIF metadata instead of binary diff
*.docx diff=word    # extract text from Word documents for diff
```

```bash
# Register the "exif" diff driver
git config diff.exif.textconv exiftool

# Register the "word" diff driver (requires docx2txt)
git config diff.word.textconv docx2txt
```

### Merge strategies per file

```
# .gitattributes

# Always use "ours" strategy for this file on merge
# (e.g., auto-generated lockfile that should not merge)
package-lock.json merge=ours

# Mark as not mergeable -- conflicts must be resolved entirely from one side
*.generated.ts merge=binary
```

### Export ignore

Files marked with `export-ignore` are excluded from `git archive` outputs (e.g., when generating release tarballs):

```
# .gitattributes
.github/            export-ignore
.gitignore          export-ignore
.gitattributes      export-ignore
tests/              export-ignore
docs/               export-ignore
*.test.ts           export-ignore
```

## Git Submodules

A submodule is a separate Git repository embedded inside another Git repository. The parent repo tracks a specific commit of the submodule, not a branch.

### When to use submodules

- You depend on an external library that you might fork and modify
- You want to pin to a specific version of a dependency (outside of a package manager)
- You need multiple projects to share a common component repository

### Adding a submodule

```bash
# Add a submodule at a specified path
git submodule add https://github.com/example/shared-components.git lib/shared

# This creates:
# .gitmodules  -- configuration file for submodules
# lib/shared/  -- the submodule directory (contains the external repo)

# Commit the submodule addition
git add .gitmodules lib/shared
git commit -m "chore: add shared-components as submodule"
```

### Cloning a repository with submodules

```bash
# Option 1: clone and init separately
git clone https://github.com/your-org/project.git
git submodule init
git submodule update

# Option 2: clone with submodules in one step (recommended)
git clone --recurse-submodules https://github.com/your-org/project.git

# After pulling updates that change submodule commits:
git pull
git submodule update --init --recursive
```

### Updating a submodule

```bash
# Update a specific submodule to its latest remote commit
git submodule update --remote lib/shared

# Then commit the updated submodule reference
git add lib/shared
git commit -m "chore: update shared-components to latest"
```

### Submodule gotchas

```bash
# BAD: forgetting to commit after changing the submodule pointer
# If you run `git submodule update --remote` but don't commit,
# other developers will see a "dirty" submodule in git status:
# modified: lib/shared (new commits)

# BAD: entering the submodule directory and making changes
# without knowing you are in detached HEAD state
cd lib/shared
git log --oneline  # you are on a specific commit, not a branch
# If you make commits here, they are on no branch.
# When you run `git submodule update` later, those commits are lost.

# GOOD: if you need to make changes in the submodule,
# check out a branch first
cd lib/shared
git switch -c fix/button-color-contrast
# make changes, commit, push
cd ../..
git add lib/shared
git commit -m "chore: update shared-components (button contrast fix)"
```

### Git Subtree: the alternative

Subtree embeds the external repository's history directly into the parent repository. Contributors do not need to know the submodule workflow.

```bash
# Add a subtree (pulling in the external repo's content)
git subtree add --prefix=lib/shared \
  https://github.com/example/shared-components.git main --squash

# Pull updates from the external repo
git subtree pull --prefix=lib/shared \
  https://github.com/example/shared-components.git main --squash

# Push changes back to the external repo
git subtree push --prefix=lib/shared \
  https://github.com/example/shared-components.git main
```

| Feature | Submodule | Subtree |
|---------|-----------|---------|
| External repo is separate | Yes | No (embedded) |
| Requires special clone | Yes | No |
| Can push back to external | Yes | Yes (subtree push) |
| History visibility | Separate | Merged into parent |
| Complexity for contributors | High | Low |
| Disk usage | Less (shared) | More (copied) |

**Recommendation:** Use subtrees for most cases. Use submodules only when you need strict version pinning and contributors are comfortable with the workflow.

## Git Sparse Checkout

Sparse checkout lets you work with only a subset of the repository's files. This is useful for large monorepos where you only need files from one section.

### Enable sparse checkout

```bash
# Clone without checking out any files
git clone --no-checkout --filter=blob:none https://github.com/org/monorepo.git
cd monorepo

# Initialize sparse checkout
git sparse-checkout init --cone

# Specify which directories to check out
git sparse-checkout set apps/frontend packages/ui packages/utils

# Now check out
git checkout main

# Only apps/frontend, packages/ui, and packages/utils exist locally
# All other files are absent from the working tree
ls apps/
# backend/  (absent)
# frontend/ (present)
```

### Modify the sparse set

```bash
# Add more directories
git sparse-checkout add packages/testing

# Remove directories (reset to new set)
git sparse-checkout set apps/frontend packages/utils

# Disable sparse checkout (check out everything)
git sparse-checkout disable
```

### Cone mode vs non-cone mode

```bash
# Cone mode (default with --cone): works at directory boundaries
# Faster, simpler, recommended
git sparse-checkout init --cone

# Non-cone mode: supports file-level patterns but is slower
# Use only if you need to include/exclude specific files
git sparse-checkout init
git sparse-checkout set '/*' '!/docs'  # everything except docs/
```

## Git Filter-Repo: Rewriting History

`git filter-repo` is the modern replacement for the deprecated `git filter-branch`. Use it to:
- Remove sensitive data from history (passwords, keys, large files)
- Extract a subdirectory into its own repository
- Rename files or directories across all history

Install: `pip install git-filter-repo`

### Remove a file from all history

```bash
# Remove a committed secret from the entire repository history
git filter-repo --path secrets.env --invert-paths

# Remove multiple files
git filter-repo --path .env --path config/secrets.json --invert-paths

# After this, force-push all branches
# ALL collaborators must re-clone -- their local repos are now incompatible
git push --force-with-lease --all
```

**Important:** Removing a file from history does not invalidate secrets. Any exposed secret must be rotated immediately. History rewriting only prevents future clones from seeing the secret.

### Extract a subdirectory into a new repository

```bash
# Create a fresh clone to work with
git clone https://github.com/org/monorepo.git extracted-lib
cd extracted-lib

# Keep only the packages/auth subdirectory, rewriting history
# so that all commits only show changes to that subdirectory
git filter-repo --subdirectory-filter packages/auth

# The repo now contains only the packages/auth content
# with all relevant history preserved.
# Push to a new repository
git remote add origin https://github.com/org/auth-lib.git
git push -u origin main
```

### Rename paths across all history

```bash
# Rename src/ to lib/ in all commits
git filter-repo --path-rename src/:lib/
```

### Filter-repo vs BFG Repo Cleaner

| Tool | Language | Speed | Flexibility |
|------|----------|-------|-------------|
| `git filter-repo` | Python | Fast | High |
| BFG Repo Cleaner | Java | Faster for large repos | Limited to common cases |

BFG is simpler for the common case of removing large files:
```bash
bfg --strip-blobs-bigger-than 10M
bfg --delete-files '*.pem'
git reflog expire --expire=now --all
git gc --prune=now --aggressive
```

## Git Archive: Exporting Without History

`git archive` creates a tarball or zip of a tree without the `.git` directory. Useful for creating release artifacts.

```bash
# Create a tarball of the current HEAD
git archive --format=tar.gz HEAD -o release.tar.gz

# Create a tarball of a specific tag
git archive --format=tar.gz v2.3.0 -o release-v2.3.0.tar.gz

# Create a zip of HEAD
git archive --format=zip HEAD -o release.zip

# Archive only a subdirectory
git archive --format=tar.gz HEAD src/ -o src-only.tar.gz

# Add a prefix to all archived paths
git archive --format=tar.gz --prefix=myapp-v2.3.0/ v2.3.0 -o release.tar.gz

# List contents without extracting
git archive HEAD | tar -t
```

Files marked with `export-ignore` in `.gitattributes` are excluded from the archive.

## Git Bundle: Transferring Repositories Offline

A bundle is a single file that contains one or more refs and all the objects they reference. Use bundles to transfer repository data without a network connection.

```bash
# Create a complete bundle of the repository
git bundle create repo.bundle --all

# Create a bundle of only the main branch
git bundle create main.bundle main

# Create an incremental bundle (only changes since a specific commit)
git bundle create update.bundle main ^old-commit-hash
# or since a tag:
git bundle create update.bundle v2.0.0..main

# Verify a bundle before using it
git bundle verify repo.bundle

# Use a bundle like a remote repository
git clone repo.bundle cloned-from-bundle

# Fetch from a bundle
git remote add bundle-remote /path/to/repo.bundle
git fetch bundle-remote

# Pull a specific branch from a bundle
git pull repo.bundle main
```

Use cases for bundles:
- Working in an air-gapped environment
- Sending repository history via email or USB drive
- Bootstrapping a mirror without network access to the origin

## Advanced `git log` Techniques

Beyond `git log --oneline`, there are powerful filtering and formatting options:

```bash
# Show commits that changed a specific file
git log -- src/auth/token.ts

# Show commits by a specific author
git log --author="Alice"

# Show commits between two dates
git log --after="2024-01-01" --before="2024-06-30"

# Show commits matching a message pattern
git log --grep="fix.*token"

# Show commits that added or removed a specific string in any file
# (the "pickaxe" search)
git log -S "refreshToken"        # exact string match
git log -G "refresh.*Token"      # regex match

# Show the history of a function's changes (requires language support)
git log -L :validateToken:src/auth/token.ts

# Combine filters
git log --author="Alice" --after="2024-01-01" -S "refreshToken"

# Show only merge commits
git log --merges

# Show commits that are in branch A but not in branch B
git log main..feature/new-auth    # in feature, not in main
git log feature/new-auth..main    # in main, not in feature

# Show commits reachable from A or B but not their common ancestor
git log main...feature/new-auth   # symmetric difference

# Custom format
git log --pretty=format:"%h %ad %s [%an]" --date=short
```

### `git log -L` for function history

This tracks the change history of a specific function or line range:

```bash
# Track changes to the validateToken function
git log -L :validateToken:src/auth/token.ts

# Track changes to lines 42-67
git log -L 42,67:src/auth/token.ts
```

## `git stash` -- Advanced Usage

Basic stash was covered in 9.2. Here are the less-known capabilities:

```bash
# Stash only staged changes (leave unstaged changes in working tree)
git stash push --staged

# Stash with a descriptive message
git stash push -m "WIP: halfway through token refresh refactor"

# Stash untracked files too (by default, untracked files are not stashed)
git stash push --include-untracked

# Stash everything including ignored files
git stash push --all

# Apply a specific stash (not just the top)
git stash apply stash@{2}

# Apply and remove in one step
git stash pop stash@{2}

# Create a branch from a stash
# (useful when the stash can no longer apply cleanly to the current branch)
git stash branch feature/new-attempt stash@{0}

# Drop a specific stash
git stash drop stash@{1}

# Clear all stashes
git stash clear

# List stashes with full diff
git stash show -p stash@{0}
```

## `git notes`: Attaching Metadata to Commits

Git notes allow you to attach annotations to commits without modifying the commits themselves (which would change their hashes).

```bash
# Add a note to the current commit
git notes add -m "Reviewed by Alice on 2025-03-01"

# Add a note to a specific commit
git notes add -m "Performance: 340ms -> 12ms" abc1234

# Show notes alongside log
git log --show-notes

# Edit an existing note
git notes edit abc1234

# Remove a note
git notes remove abc1234
```

Notes are stored in `refs/notes/commits` and must be pushed/fetched explicitly:

```bash
git push origin refs/notes/commits
git fetch origin refs/notes/commits
```

Notes are rarely used in practice but are useful for build systems, CI pipelines, or audit tools that need to attach metadata to commits without rewriting history.

## Common Mistakes

- **Running `git bisect` without a test plan:** Starting bisect without knowing exactly what "good" and "bad" mean for your bug leads to inconsistent answers at each step. Define a precise, reproducible test before starting.

- **Not resetting after bisect:** If you forget `git bisect reset`, your HEAD stays on the bisect-found commit. All subsequent work goes onto a detached HEAD and can be lost.

- **Enabling rerere without verifying auto-resolutions:** Rerere's automatic resolutions are not always correct, especially if the conflicting code changed semantically since the resolution was recorded. Always verify auto-resolved files before staging.

- **Using submodules in a team without documenting the workflow:** New developers regularly forget to run `git submodule update --init --recursive` after pulling. This produces confusing "missing file" errors. Document this in your README and consider adding a post-merge hook.

- **Making commits inside a submodule while in detached HEAD:** Commits made in detached HEAD are reachable only by their hash. When `git submodule update` is run, those commits become unreachable. Always create a branch before committing in a submodule.

- **Using `git filter-branch` instead of `git filter-repo`:** `filter-branch` is deprecated, significantly slower, and has subtle correctness issues. Use `git filter-repo` for all history rewriting.

- **Rewriting shared history:** `git filter-repo`, interactive rebase, and `git commit --amend` all rewrite commits. Never rewrite commits that have been pushed to a shared branch and pulled by others. Coordinate history rewrites on main as a team event with all developers re-cloning.

- **`-S` (pickaxe) vs `-G` in `git log`:** `-S` finds commits that changed the count of a string (added or removed occurrences). `-G` finds commits where the diff matches a pattern. They answer different questions. Using `-S` to search for "any commit that touched this regex" will miss commits that both added and removed matching lines.

## Best Practices

- **Use `git bisect run` with an automated test whenever possible.** Manual bisect is error-prone. A failing unit test as the bisect script is the most reliable approach.

- **Enable `rerere` globally early.** The cost is near zero (a few kilobytes in `.git/rr-cache`), and it pays off the first time you encounter a repeated conflict.

- **Keep `.gitattributes` in every repository.** It prevents line-ending issues, ensures binary files are never corrupted by text normalization, and lets you customize diffs for non-text formats.

- **Document submodule workflows prominently.** Add a "Getting Started" section to your README that includes `git clone --recurse-submodules` and `git submodule update --init --recursive`.

- **Use `git log -S` before assuming a bug is recent.** The pickaxe search can tell you exactly when a specific string was added or removed, saving hours of manual investigation.

- **Never delete `.git/rr-cache` manually.** If a rerere resolution was wrong, use `git rerere forget <file>` to remove just that resolution, not the entire cache.

- **Create a backup branch before any history-rewriting operation.** Before running `git filter-repo` or an interactive rebase that touches many commits, tag the current state: `git tag backup/before-filter HEAD`. This makes recovery trivial if something goes wrong.

## Real-World Use Cases

### Finding a performance regression with bisect run

A page load time degraded from 800ms to 2,400ms over 200 commits. You have a Lighthouse CI script:

```bash
#!/bin/bash
# perf-bisect.sh

# Build the project
npm run build --silent 2>/dev/null || exit 125  # skip unbuildable commits

# Run Lighthouse and extract performance score
score=$(lighthouse http://localhost:3000 \
  --output=json --quiet \
  --chrome-flags="--headless" 2>/dev/null \
  | jq '.categories.performance.score')

# Good if score >= 0.90 (90/100), bad if lower
if (( $(echo "$score >= 0.90" | bc -l) )); then
  exit 0  # good
else
  exit 1  # bad
fi
```

```bash
git bisect start
git bisect bad HEAD
git bisect good v3.0.0  # known fast version
git bisect run ./perf-bisect.sh
# After ~8 tests, bisect identifies the commit that dropped the score
```

### Removing a committed secret from history

A developer accidentally committed an API key in `src/config.ts`. The key was pushed to the remote.

```bash
# Step 0: Rotate the API key immediately. History cleanup is secondary.

# Step 1: Remove from history using filter-repo
pip install git-filter-repo
git filter-repo --path src/config.ts --invert-paths

# If you only want to remove the key but keep the file:
# Write a replacement script
cat > replace-secret.py << 'EOF'
import re

def callback(commit, metadata):
    for change in commit.file_changes:
        if change.filename == b'src/config.ts':
            # Replace the leaked key with a placeholder
            change.blob_id = None  # forces git filter-repo to re-encode
            change.data = re.sub(
                b'AKIAIOSFODNN7EXAMPLE',
                b'<ROTATED_SEE_SECRETS_MANAGER>',
                change.data
            )

EOF
git filter-repo --commit-callback "$(cat replace-secret.py)"

# Step 2: Force-push all branches and tags
git push --force-with-lease --all
git push --force-with-lease --tags

# Step 3: Notify all collaborators to re-clone
# Their local histories still contain the secret.
# They cannot simply `git pull` -- they must re-clone.
```

### Using rerere in a long-lived release branch

Your team maintains a `release/v4` branch that cherry-picks hotfixes from `main`. A particular migration in `database/schema.ts` is always in conflict because `release/v4` uses the v3 schema format.

```bash
# Enable rerere (already done globally)
git config --global rerere.enabled true

# First cherry-pick: conflict occurs
git cherry-pick abc1234
# CONFLICT (content): Merge conflict in database/schema.ts

# Resolve manually (adapt the patch to the v3 schema format)
vim database/schema.ts
git add database/schema.ts
git cherry-pick --continue
# Recorded resolution for 'database/schema.ts'.

# Next cherry-pick that touches the same region:
git cherry-pick def5678
# Resolved 'database/schema.ts' using previous resolution.
git add database/schema.ts
git cherry-pick --continue

# The conflict is resolved automatically. You verify and move on.
# No manual editing required for the same conflict region.
```
