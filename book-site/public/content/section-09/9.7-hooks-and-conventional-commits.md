# 9.7 Hooks and Conventional Commits

> Git hooks automate quality gates at the exact moment work enters the repository, and Conventional Commits provide the structured message format that makes those gates -- and automated tooling -- possible.

---

## Git Hooks: What They Are

A Git hook is an executable script that Git runs automatically at specific points in its workflow. Every Git repository includes a `.git/hooks/` directory with sample hook scripts. When Git reaches a hook trigger point, it looks for an executable file with the hook's name in that directory and runs it.

```bash
ls .git/hooks/
# applypatch-msg.sample     post-update.sample     pre-merge-commit.sample
# commit-msg.sample         pre-applypatch.sample  pre-push.sample
# fsmonitor-watchman.sample pre-commit.sample      pre-rebase.sample
# post-checkout             pre-receive.sample     prepare-commit-msg.sample
# update.sample
```

To activate a sample hook, remove the `.sample` extension and ensure the file is executable:

```bash
cp .git/hooks/pre-commit.sample .git/hooks/pre-commit
chmod +x .git/hooks/pre-commit
```

Hooks are not committed to the repository by default (`.git/` is not tracked). This means hooks installed manually are local to one developer's machine. Tools like Husky and Lefthook solve the problem of sharing hooks across a team.

---

## Client-Side Hooks Reference

Client-side hooks run on the developer's local machine, triggered by local Git operations.

### Commit workflow hooks

| Hook | When it runs | Common uses | Can abort? |
|------|-------------|-------------|-----------|
| `pre-commit` | Before the commit message editor opens | Linting, formatting, running tests | Yes (exit non-zero) |
| `prepare-commit-msg` | After default message is created, before editor opens | Auto-populating message with ticket number | Yes |
| `commit-msg` | After the developer writes the commit message | Validating message format | Yes (exit non-zero) |
| `post-commit` | After the commit is finalized | Notifications, logging | No |

### Other client-side hooks

| Hook | When it runs | Common uses |
|------|-------------|-------------|
| `pre-push` | Before `git push` sends data | Running full test suite, preventing pushes to protected branches |
| `post-checkout` | After `git switch` or `git checkout` | Installing dependencies, running migrations |
| `post-merge` | After `git merge` or `git pull` | Reinstalling packages when `package.json` changed |
| `pre-rebase` | Before a rebase starts | Warning about rebasing shared branches |

### Server-side hooks

Server-side hooks run on the remote repository server. They are powerful because they enforce rules that cannot be bypassed by individual developers.

| Hook | When it runs | Common uses |
|------|-------------|-------------|
| `pre-receive` | Before refs are updated on the server | Rejecting malformed commits, enforcing policies |
| `update` | Once per ref being updated | Per-branch authorization rules |
| `post-receive` | After all refs are updated | Triggering CI, sending notifications, deployments |

Server-side hooks are configured by repository administrators and are outside the scope of most developer workflows. GitHub, GitLab, and Bitbucket expose them as "server hooks" or "webhooks."

## Writing a Raw Hook Script

Understanding what a hook script actually is demystifies all the tooling built on top.

### `pre-commit`: run ESLint before every commit

```bash
#!/bin/sh
# .git/hooks/pre-commit

# Run ESLint on all staged .ts and .js files
STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep -E '\.(ts|js)$')

if [ -z "$STAGED_FILES" ]; then
  exit 0  # no staged TypeScript/JavaScript files, nothing to check
fi

echo "Running ESLint on staged files..."
echo "$STAGED_FILES" | xargs npx eslint --max-warnings=0

if [ $? -ne 0 ]; then
  echo "ESLint failed. Fix the errors above before committing."
  exit 1
fi

exit 0
```

### `commit-msg`: validate the message format

The commit message file path is passed as the first argument (`$1`):

```bash
#!/bin/sh
# .git/hooks/commit-msg

COMMIT_MSG_FILE=$1
COMMIT_MSG=$(cat "$COMMIT_MSG_FILE")

# Pattern: type(optional-scope): description
# Examples: feat(auth): add OAuth login
#           fix: correct null check in token validator
PATTERN="^(feat|fix|docs|style|refactor|perf|test|build|ci|chore|revert)(\([a-z0-9-]+\))?: .{1,72}"

if ! echo "$COMMIT_MSG" | grep -qE "$PATTERN"; then
  echo ""
  echo "ERROR: Commit message does not follow Conventional Commits format."
  echo ""
  echo "Expected format:"
  echo "  type(scope): description"
  echo ""
  echo "Valid types: feat, fix, docs, style, refactor, perf, test, build, ci, chore, revert"
  echo ""
  echo "Your message: $COMMIT_MSG"
  exit 1
fi

exit 0
```

### `pre-push`: prevent pushes to main

```bash
#!/bin/sh
# .git/hooks/pre-push

REMOTE=$1
URL=$2

# Read refs being pushed from stdin
# Format: <local-ref> <local-sha> <remote-ref> <remote-sha>
while read local_ref local_sha remote_ref remote_sha; do
  # Block direct pushes to main or master
  if echo "$remote_ref" | grep -qE "refs/heads/(main|master)$"; then
    echo "ERROR: Direct pushes to '$remote_ref' are not allowed."
    echo "Open a pull request instead."
    exit 1
  fi
done

exit 0
```

## Managing Hooks with Husky

Writing raw hook scripts works, but sharing them across a team requires a solution to the problem that `.git/hooks/` is not committed. Husky solves this by storing hooks in a committed directory (`.husky/`) and using a Git core setting to redirect Git's hook directory.

### Setup (Husky v9)

```bash
# Install husky as a dev dependency
npm install --save-dev husky

# Initialize husky (creates .husky/ directory and sets core.hooksPath)
npx husky init
# This adds a "prepare" script to package.json:
# "prepare": "husky"
# And creates .husky/pre-commit with a placeholder
```

After running `npm install`, the `prepare` lifecycle script runs `husky`, which sets:

```bash
git config core.hooksPath .husky
```

This redirects Git to look for hooks in `.husky/` instead of `.git/hooks/`. Since `.husky/` is committed, all team members get the same hooks automatically when they run `npm install`.

### Creating hooks

Each hook is a file in `.husky/`:

```bash
# Create a pre-commit hook
echo "npx lint-staged" > .husky/pre-commit
chmod +x .husky/pre-commit

# Create a commit-msg hook
echo "npx --no -- commitlint --edit \$1" > .husky/commit-msg
chmod +x .husky/commit-msg

# Create a pre-push hook
echo "npm test" > .husky/pre-push
chmod +x .husky/pre-push
```

The `.husky/` files are simple shell scripts, not configuration. Keep them minimal:

```bash
# .husky/pre-commit
npx lint-staged
```

```bash
# .husky/commit-msg
npx --no -- commitlint --edit $1
```

### Bypassing hooks (use sparingly)

```bash
# Skip all hooks for a commit (use for emergency commits only)
git commit --no-verify -m "emergency: revert broken deployment"

# Skip pre-push hooks
git push --no-verify
```

Document in your team's CONTRIBUTING.md that `--no-verify` bypasses quality gates and should only be used for genuine emergencies.

## lint-staged: Run Checks Only on Staged Files

Running ESLint on the entire codebase before every commit is too slow. `lint-staged` runs checks only on the files that are staged for the current commit.

### Installation

```bash
npm install --save-dev lint-staged
```

### Configuration

```json
// package.json
{
  "lint-staged": {
    "*.{ts,tsx}": [
      "eslint --fix --max-warnings=0",
      "prettier --write"
    ],
    "*.{js,jsx}": [
      "eslint --fix --max-warnings=0",
      "prettier --write"
    ],
    "*.{css,scss}": [
      "prettier --write",
      "stylelint --fix"
    ],
    "*.{json,md,yml,yaml}": [
      "prettier --write"
    ]
  }
}
```

Or in a dedicated config file:

```js
// lint-staged.config.js
export default {
  '*.{ts,tsx,js,jsx}': (files) => [
    `eslint --fix --max-warnings=0 ${files.join(' ')}`,
    `prettier --write ${files.join(' ')}`,
  ],
  '*.{css,scss,md,json}': 'prettier --write',
};
```

### How lint-staged modifies the commit

When lint-staged runs `--fix` or `prettier --write`, it modifies files that are currently staged. Those modifications are NOT automatically re-staged. The commit will include the original staged content, not the fixed version.

To include fixes in the commit, add `git add` after the fix command:

```json
{
  "lint-staged": {
    "*.{ts,tsx}": [
      "eslint --fix",
      "git add"
    ]
  }
}
```

Modern versions of lint-staged handle this automatically. Verify with your version.

## Lefthook: The Alternative to Husky

Lefthook is a Git hooks manager written in Go. It runs as a binary with no Node.js dependency, supports parallel hook execution, and is well-suited for monorepos and non-JavaScript projects.

### Installation

```bash
# Via npm (for JS projects)
npm install --save-dev @evilmartians/lefthook

# Via binary (no Node required)
brew install lefthook         # macOS
# or download from GitHub releases
```

### Configuration

```yaml
# lefthook.yml

pre-commit:
  parallel: true              # run commands in parallel
  commands:
    lint:
      glob: "*.{ts,tsx,js,jsx}"
      run: npx eslint --fix {staged_files}
      stage_fixed: true       # re-stage fixed files automatically

    format:
      glob: "*.{ts,tsx,js,jsx,css,scss,json,md}"
      run: npx prettier --write {staged_files}
      stage_fixed: true

    type-check:
      run: npx tsc --noEmit

commit-msg:
  commands:
    validate:
      run: npx commitlint --edit {1}

pre-push:
  commands:
    test:
      run: npm test
```

### Husky vs Lefthook comparison

| Feature | Husky v9 | Lefthook |
|---------|----------|----------|
| Language | Node.js | Go (binary) |
| Config format | Shell scripts | YAML |
| Parallel execution | No (sequential) | Yes (built-in) |
| Non-JS projects | Awkward | Excellent |
| Monorepo support | Manual | Built-in |
| CI overhead | npm install required | Binary only |
| Learning curve | Low (familiar JS devs) | Low |
| Community size | Large | Growing |

For JavaScript-only projects, either works well. For polyglot or performance-sensitive setups, Lefthook has clear advantages.

## Conventional Commits Specification

Conventional Commits is a lightweight specification for writing structured commit messages. It provides a common language for humans and tools to understand what a commit does.

### Format

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

### Types

| Type | When to use | SemVer impact |
|------|------------|---------------|
| `feat` | A new feature | Minor (1.X.0) |
| `fix` | A bug fix | Patch (1.0.X) |
| `docs` | Documentation only | None |
| `style` | Formatting, whitespace (no logic change) | None |
| `refactor` | Code restructuring (no feature/fix) | None |
| `perf` | Performance improvement | Patch |
| `test` | Adding or correcting tests | None |
| `build` | Build system, external dependencies | None |
| `ci` | CI configuration and scripts | None |
| `chore` | Maintenance tasks, repo management | None |
| `revert` | Reverting a previous commit | Depends |

### Breaking changes

Breaking changes are indicated in two places:

```
feat(auth): replace JWT with session cookies

BREAKING CHANGE: The `Authorization: Bearer <token>` header is no
longer accepted. Clients must use session cookies. All existing
tokens are invalidated.
```

Or with an `!` after the type/scope:

```
feat(auth)!: replace JWT with session cookies
```

Both forms trigger a major version bump (X.0.0) in automated release tools.

### Scope

The scope is optional and narrows the area of the codebase affected:

```
feat(payments): add Stripe webhook handler
fix(auth): prevent token refresh race condition
docs(readme): add Docker setup instructions
refactor(database): extract connection pool to service
```

Scopes should be consistent within a project. Define them in your CONTRIBUTING.md.

### Examples

```
# Good -- clear type, clear description
feat(notifications): add email notification for order confirmation

# Good -- fix with context
fix(checkout): prevent crash when cart is empty during price calculation

# Good -- with body explaining the why
refactor(api): replace axios with native fetch

Removes the axios dependency (142KB) since the fetch API now has
full cross-browser support. The retry and interceptor logic has
been reimplemented using a custom wrapper.

# Good -- breaking change
feat(api)!: remove deprecated v1 endpoints

BREAKING CHANGE: All /api/v1/* endpoints are removed. Use /api/v2/*
equivalents. Migration guide: docs/migration/v1-to-v2.md

# Bad -- no type
update button color

# Bad -- vague
fix stuff

# Bad -- too long description
feat: add the ability to upload profile pictures with a new component
```

## commitlint: Enforcing the Specification

`commitlint` lints commit messages against a configured ruleset, most commonly the `@commitlint/config-conventional` preset that enforces the Conventional Commits spec.

### Installation

```bash
npm install --save-dev @commitlint/cli @commitlint/config-conventional
```

### Configuration

```js
// commitlint.config.js (or .commitlintrc.js)
export default {
  extends: ['@commitlint/config-conventional'],

  rules: {
    // Override or add rules
    'type-enum': [
      2,       // error level: 0=off, 1=warn, 2=error
      'always',
      ['feat', 'fix', 'docs', 'style', 'refactor', 'perf',
       'test', 'build', 'ci', 'chore', 'revert']
    ],
    'scope-case': [2, 'always', 'lower-case'],
    'subject-case': [2, 'never', ['sentence-case', 'start-case', 'pascal-case', 'upper-case']],
    'subject-max-length': [2, 'always', 72],
    'body-max-line-length': [2, 'always', 100],
  },
};
```

### Wire to Husky

```bash
# .husky/commit-msg
npx --no -- commitlint --edit $1
```

### Testing commitlint

```bash
# Test a message directly
echo "feat(auth): add OAuth login" | npx commitlint

# Lint the last N commits
npx commitlint --from HEAD~3 --to HEAD --verbose
```

### commitlint output

```bash
# Bad message
echo "update login" | npx commitlint

# Output:
⧗   input: update login
✖   subject may not be empty [subject-empty]
✖   type may not be empty [type-empty]
✖   found 2 problems, 0 warnings
```

```bash
# Good message
echo "fix(auth): prevent redirect loop on expired session" | npx commitlint
# Output:
⧗   input: fix(auth): prevent redirect loop on expired session
✔   found 0 problems, 0 warnings
```

## Automated Changelog Generation

Once commits follow the Conventional Commits format, changelogs can be generated automatically. Tools parse the commit history and group changes by type.

### `conventional-changelog-cli`

```bash
npm install --save-dev conventional-changelog-cli

# Generate changelog from all conventional commits
npx conventional-changelog -p conventional -i CHANGELOG.md -s

# Generate changelog for a new release (update CHANGELOG.md)
npx conventional-changelog -p conventional -i CHANGELOG.md -s -r 0
```

This produces a `CHANGELOG.md` like:

```markdown
# Changelog

## [2.4.0] - 2025-03-01

### Features

* **notifications:** add email notification for order confirmation ([a1b2c3d])
* **search:** add fuzzy search to product catalog ([e4f5g6h])

### Bug Fixes

* **checkout:** prevent crash when cart is empty ([i7j8k9l])
* **auth:** prevent token refresh race condition ([m1n2o3p])
```

### `semantic-release`: fully automated versioning

`semantic-release` reads the commit history since the last release, determines the version bump (patch/minor/major) from the commit types, generates a changelog, publishes to npm, creates a GitHub/GitLab release, and tags the commit -- all automatically in CI.

```bash
npm install --save-dev semantic-release \
  @semantic-release/changelog \
  @semantic-release/git \
  @semantic-release/github
```

```json
// .releaserc.json
{
  "branches": ["main"],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    ["@semantic-release/changelog", {
      "changelogFile": "CHANGELOG.md"
    }],
    "@semantic-release/npm",
    ["@semantic-release/git", {
      "assets": ["CHANGELOG.md", "package.json"],
      "message": "chore(release): ${nextRelease.version} [skip ci]\n\n${nextRelease.notes}"
    }],
    "@semantic-release/github"
  ]
}
```

```yaml
# .github/workflows/release.yml
name: Release
on:
  push:
    branches: [main]
jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      issues: write
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # semantic-release needs full history
      - uses: actions/setup-node@v4
        with:
          node-version: 22
      - run: npm ci
      - run: npx semantic-release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

With this setup, merging a `feat:` commit to `main` automatically publishes a new minor version. A `BREAKING CHANGE:` commit triggers a major version. No manual version bumping, no manual changelog editing.

## `prepare-commit-msg`: Auto-populating Commit Messages

This hook runs after the default message is prepared but before the editor opens. Use it to automatically insert information like the Jira ticket number from the branch name.

```bash
#!/bin/sh
# .git/hooks/prepare-commit-msg

COMMIT_MSG_FILE=$1
COMMIT_SOURCE=$2   # "message", "template", "merge", "squash", "commit"

# Only run on fresh commits (not amends, merges, squashes)
if [ "$COMMIT_SOURCE" = "merge" ] || [ "$COMMIT_SOURCE" = "squash" ]; then
  exit 0
fi

# Extract Jira ticket from branch name
# Branch format: feat/PROJ-1234-description or fix/PROJ-5678-something
BRANCH_NAME=$(git symbolic-ref --short HEAD 2>/dev/null)
TICKET=$(echo "$BRANCH_NAME" | grep -oE '[A-Z]+-[0-9]+')

if [ -n "$TICKET" ]; then
  # Prepend ticket to commit message
  # Existing content: "feat(auth): "
  # After hook: "feat(auth): [PROJ-1234] "
  EXISTING=$(cat "$COMMIT_MSG_FILE")
  echo "[${TICKET}] ${EXISTING}" > "$COMMIT_MSG_FILE"
fi
```

Result: when the developer types `feat(auth): add OAuth login`, the hook inserts the ticket number automatically, producing `[PROJ-1234] feat(auth): add OAuth login`.

## Full Setup: Recommended Configuration

A complete setup for a TypeScript project:

### 1. Install dependencies

```bash
npm install --save-dev \
  husky \
  lint-staged \
  @commitlint/cli \
  @commitlint/config-conventional \
  eslint \
  prettier

npx husky init
```

### 2. Configure lint-staged

```json
// package.json (relevant sections)
{
  "scripts": {
    "prepare": "husky"
  },
  "lint-staged": {
    "*.{ts,tsx,js,jsx}": ["eslint --fix --max-warnings=0", "prettier --write"],
    "*.{json,md,yml,css,scss}": ["prettier --write"]
  }
}
```

### 3. Configure commitlint

```js
// commitlint.config.js
export default { extends: ['@commitlint/config-conventional'] };
```

### 4. Create hooks

```bash
# .husky/pre-commit
npx lint-staged
```

```bash
# .husky/commit-msg
npx --no -- commitlint --edit $1
```

### 5. Commit and push the config

```bash
git add .husky/ package.json commitlint.config.js
git commit -m "chore: add husky, lint-staged, and commitlint"
```

From this point forward, every new developer who runs `npm install` automatically gets the hooks configured.

## Common Mistakes

- **Not committing the hook configuration.** Raw `.git/hooks/` scripts are invisible to other team members. Always use Husky or Lefthook to share hooks via committed files.

- **Making hooks too slow.** A `pre-commit` hook that runs the full test suite will be bypassed constantly with `--no-verify`. Keep `pre-commit` fast (under 10 seconds): run only linting and formatting on staged files. Move full test runs to `pre-push` or CI.

- **Using `eslint --fix` without re-staging.** When ESLint auto-fixes files, the fixes are not automatically included in the commit unless you re-stage them. Check whether your version of lint-staged handles this, or add `git add` explicitly.

- **Inconsistent type usage.** Teams that use `chore`, `misc`, and `maintenance` interchangeably get changelogs that are hard to parse and automated tooling that cannot determine version bumps. Define your allowed types and enforce them with commitlint.

- **Vague descriptions.** `fix: bug fix` and `feat: new feature` pass the format check but are useless in a changelog. Commitlint can enforce minimum description length (`subject-min-length` rule), but developer discipline matters more.

- **Forgetting `fetch-depth: 0` in CI.** `semantic-release` and `conventional-changelog` need the full Git history to determine what changed since the last release. A shallow clone (`fetch-depth: 1`, the default in many CI actions) will cause them to fail or generate incorrect changelogs.

- **Skipping the `[skip ci]` commit message tag.** When `semantic-release` commits the updated `CHANGELOG.md` and `package.json` back to the repository, that commit should not trigger another CI run. Include `[skip ci]` in the release commit message.

## Best Practices

- **Enforce at commit time, not review time.** A commitlint hook catches a bad message immediately. A PR reviewer catching it after the fact means a comment, a response, a fix, another commit, and more review time. Automate what can be automated.

- **Keep hooks fast.** The `pre-commit` hook budget is 5-10 seconds maximum. Lint-staged on changed files achieves this. Full test suites belong in `pre-push` or CI.

- **Define scopes explicitly.** Document your allowed scopes in CONTRIBUTING.md and optionally in `commitlint.config.js` under `scope-enum`. Consistent scopes produce readable changelogs (e.g., all auth changes grouped under `auth`).

- **Use `semantic-release` or an equivalent for libraries.** For applications, automated changelogs are informative. For published packages (npm libraries), automated versioning prevents human error in semver bumps.

- **Test your hooks before sharing them.** Run `npx husky` after changing hook files. Test edge cases: empty staged files, merge commits, rebase continuations.

- **Document bypass procedures.** Explicitly state when `--no-verify` is acceptable (genuine production emergency only) and require a follow-up commit that addresses the bypassed check.

## Real-World Use Cases

### Enforcing quality on a frontend team

A six-person team working on a TypeScript/React application sets up the following pipeline:

```
git commit
  --> pre-commit: lint-staged runs ESLint + Prettier on staged TS/TSX files
      --> ESLint auto-fixes import order and unused variables
      --> Prettier auto-formats code
      --> Changes are re-staged automatically
  --> commit-msg: commitlint validates the message format
      --> "Update stuff" is rejected immediately with a clear error
      --> "feat(dashboard): add date range filter for reports" passes
  --> Commit created
```

Result: every commit in the repository passes the linter, is consistently formatted, and has a meaningful message. Code review can focus on logic, not formatting debates.

### Automated npm package releases

An open-source utility library is hosted on GitHub and published to npm. The release process is fully automated:

```
Developer merges a PR with message "feat: add debounce utility"
  --> GitHub Actions workflow triggers on push to main
  --> CI: npm test (all tests pass)
  --> semantic-release runs:
      --> Analyzes commits since v1.3.2 (the last release)
      --> Finds 1 "feat:" -> minor bump: v1.4.0
      --> Generates CHANGELOG.md entry
      --> Bumps version in package.json to 1.4.0
      --> Publishes v1.4.0 to npm
      --> Creates GitHub Release with changelog
      --> Tags the commit with v1.4.0
      --> Commits updated CHANGELOG.md and package.json [skip ci]
  --> v1.4.0 is live on npm within minutes of the merge
```

No human decides the version number. No human writes the changelog. No human types `npm publish`.

### Preventing direct pushes to main

A `pre-push` hook blocks developers from pushing directly to `main`, ensuring all changes go through PR review:

```bash
#!/bin/sh
# .husky/pre-push

while read local_ref local_sha remote_ref remote_sha; do
  if echo "$remote_ref" | grep -qE "refs/heads/(main|master)$"; then
    echo ""
    echo "Direct push to '$(echo $remote_ref | sed 's|refs/heads/||')' is not allowed."
    echo "Create a feature branch and open a pull request instead."
    echo ""
    exit 1
  fi
done
```

Combined with a protected branch rule on GitHub (which blocks force pushes and requires PRs), this creates a two-layer defense: the hook prevents accidental pushes from command line, and the server-side rule prevents pushes from tools that bypass local hooks.

---

> **Key Takeaway:** Hooks enforce quality gates at the moment work enters the repository, and Conventional Commits provide the structured message format that makes those gates and automated release tooling possible -- together they remove entire categories of review comments and manual processes from your team's workflow.
