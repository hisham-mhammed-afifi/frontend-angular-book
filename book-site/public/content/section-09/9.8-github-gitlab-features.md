# 9.8 GitHub / GitLab Features

> GitHub and GitLab are more than Git hosts -- they are full development platforms that handle issue tracking, CI/CD, package registries, security scanning, and release management in one place.

---

## Platform Overview

Both GitHub and GitLab wrap Git hosting with a suite of tools that cover the entire software development lifecycle. Understanding what each platform provides prevents you from reinventing these tools manually.

| Feature area | GitHub | GitLab |
|---|---|---|
| Issue tracking | Issues + Projects | Issues + Boards |
| CI/CD | GitHub Actions | GitLab CI/CD |
| Package registry | GitHub Packages | GitLab Package Registry |
| Container registry | GitHub Container Registry | GitLab Container Registry |
| Security scanning | Code scanning (CodeQL) | Built-in SAST, DAST, dependency scan |
| Pages / static hosting | GitHub Pages | GitLab Pages |
| Wiki | Repository wiki | Repository wiki |
| Release management | Releases | Releases + Deployments |
| Self-hosting | GitHub Enterprise Server | GitLab CE / EE |
| Free CI minutes | 2,000 / month | 400 / month |

## GitHub Issues and Project Management

### Issues

Issues are the unit of work tracking on GitHub. They support markdown, file attachments, assignees, labels, milestones, and cross-references.

```
# Anatomy of a useful issue

Title: [BUG] Checkout crashes when cart contains a deleted product

## Description
Steps to reproduce:
1. Add a product to cart
2. Delete the product from admin panel
3. Attempt to checkout

Expected: Graceful error message
Actual: Uncaught TypeError: Cannot read properties of null

## Environment
- Browser: Chrome 123
- Environment: Production
- Affected users: ~3% based on Sentry data

## Related
- Closes #1234
- Blocked by #1200
```

Reference issues and PRs from commit messages and other issues:
- `Fixes #123` -- closes the issue when the commit/PR merges to the default branch
- `#123` -- creates a cross-reference without closing
- `org/repo#123` -- cross-repository reference

### Labels

Labels categorize issues and PRs. A consistent label system makes filtering and triage fast.

```
# Common label categories:

# Type
type: bug
type: feature
type: docs
type: chore

# Priority
priority: critical
priority: high
priority: medium
priority: low

# Status
status: in-progress
status: blocked
status: needs-review
status: wont-fix

# Area
area: auth
area: payments
area: frontend
area: infrastructure
```

Automate label creation across repositories with the GitHub CLI:

```bash
# Create labels programmatically
gh label create "priority: critical" --color "D93F0B" --description "Must fix before next release"
gh label create "type: bug" --color "FC2929" --description "Something is broken"

# List all labels
gh label list

# Apply a label to an issue
gh issue edit 42 --add-label "priority: critical"
```

### Milestones

A milestone groups issues and PRs into a named, optionally dated bucket and shows a progress bar.

```bash
# Create a milestone
gh api repos/owner/repo/milestones \
  --method POST \
  --field title="v2.4.0" \
  --field description="Q1 2025 release" \
  --field due_on="2025-03-31T23:59:59Z"

# Assign an issue to a milestone
gh issue edit 42 --milestone "v2.4.0"
```

Use milestones for version-scoped releases, not for sprints (use Projects for sprint tracking).

### GitHub Projects

GitHub Projects is a flexible spreadsheet-like view for organizing issues and PRs. It supports custom fields, multiple views (table, board, roadmap), and automation.

Key capabilities:
- **Custom fields:** priority (single select), story points (number), sprint (iteration), target date (date)
- **Views:** filter by assignee, label, milestone, or any custom field
- **Automation:** auto-add issues with specific labels, auto-archive closed issues
- **Roadmap view:** timeline-based visualization of iteration fields

```bash
# Add an issue to a project via CLI
gh project item-add 1 --owner org-name --url https://github.com/org/repo/issues/42
```

## GitHub Actions: CI/CD Built In

GitHub Actions is an event-driven automation platform. Workflows are YAML files in `.github/workflows/`.

### Anatomy of a workflow

```yaml
# .github/workflows/ci.yml
name: CI

# Triggers
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]
  workflow_dispatch:        # allows manual trigger from GitHub UI

# Environment variables available to all jobs
env:
  NODE_VERSION: '22'

jobs:
  # Each job runs in a fresh virtual machine
  test:
    name: Test
    runs-on: ubuntu-latest   # runner OS

    steps:
      # Step: check out the repository
      - name: Checkout
        uses: actions/checkout@v4

      # Step: set up Node.js
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'        # cache ~/.npm between runs

      # Step: install dependencies
      - name: Install dependencies
        run: npm ci

      # Step: run linting
      - name: Lint
        run: npm run lint

      # Step: run tests with coverage
      - name: Test
        run: npm test -- --coverage

      # Step: upload coverage artifact
      - name: Upload coverage
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: coverage/
          retention-days: 7
```

### Triggers

```yaml
on:
  # Push to specific branches
  push:
    branches: [main, 'release/**']
    paths:
      - 'src/**'             # only trigger when src/ changes
      - 'package.json'

  # Pull request events
  pull_request:
    types: [opened, synchronize, reopened]
    branches: [main]

  # On a schedule (cron syntax)
  schedule:
    - cron: '0 6 * * 1'     # every Monday at 6:00 UTC

  # Manual trigger with inputs
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        type: choice
        options: [staging, production]

  # Triggered by another workflow
  workflow_call:
    inputs:
      node-version:
        type: string
        required: false
        default: '22'
```

### Matrix builds

Matrix strategy runs a job multiple times with different combinations of variables. Essential for cross-platform or cross-version testing.

```yaml
jobs:
  test:
    strategy:
      fail-fast: false        # don't cancel other matrix jobs if one fails
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        node: ['20', '22']
        include:
          # Add extra combinations
          - os: ubuntu-latest
            node: '23'
        exclude:
          # Remove specific combinations
          - os: windows-latest
            node: '20'

    runs-on: ${{ matrix.os }}

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
```

This produces 5 jobs (3x2 minus the excluded combination, plus the extra).

### Caching dependencies

```yaml
- name: Cache node modules
  uses: actions/cache@v4
  id: cache-node-modules
  with:
    path: ~/.npm
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-node-

- name: Install dependencies
  # Skip install if cache hit
  if: steps.cache-node-modules.outputs.cache-hit != 'true'
  run: npm ci
```

### Secrets and environments

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production    # requires manual approval if configured

    steps:
      - name: Deploy
        run: ./deploy.sh
        env:
          # Reference repository or environment secrets
          API_KEY: ${{ secrets.PRODUCTION_API_KEY }}
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
```

Secrets are stored in:
- **Repository secrets:** available to all workflows in the repo
- **Environment secrets:** available only to jobs that target that environment
- **Organization secrets:** available across repositories in the org

Environment protection rules:
- Required reviewers: a named person must approve before the job runs
- Wait timer: delay before deployment starts
- Deployment branches: only specific branches can deploy to this environment

### OIDC: credential-free cloud authentication

Instead of storing long-lived cloud credentials as secrets, use OIDC to let GitHub Actions authenticate directly with cloud providers:

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      id-token: write   # required for OIDC
      contents: read

    steps:
      - uses: actions/checkout@v4

      # Authenticate to AWS without storing AWS_ACCESS_KEY_ID
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/GitHubActionsRole
          aws-region: us-east-1

      - name: Deploy to S3
        run: aws s3 sync ./dist s3://my-bucket/
```

The cloud provider trusts GitHub's OIDC token. No static credentials stored anywhere.

### Reusable workflows

Extract common workflow logic into a callable workflow to avoid duplication:

```yaml
# .github/workflows/reusable-test.yml
name: Reusable Test Workflow

on:
  workflow_call:
    inputs:
      node-version:
        required: false
        type: string
        default: '22'
    secrets:
      NPM_TOKEN:
        required: false

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm ci
      - run: npm test
```

```yaml
# .github/workflows/ci.yml -- calls the reusable workflow
name: CI

on:
  push:
    branches: [main]

jobs:
  run-tests:
    uses: ./.github/workflows/reusable-test.yml
    with:
      node-version: '22'
    secrets: inherit     # pass all secrets from this workflow to the called one
```

### Job dependencies and concurrency

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: npm run build

  test:
    needs: build          # runs after build completes
    runs-on: ubuntu-latest
    steps:
      - run: npm test

  deploy:
    needs: [build, test]  # runs after both build and test complete
    if: github.ref == 'refs/heads/main'  # only on main
    runs-on: ubuntu-latest
    steps:
      - run: ./deploy.sh

# Prevent concurrent deployments to the same environment
concurrency:
  group: deploy-production
  cancel-in-progress: false  # queue instead of canceling
```

### Workflow security

```yaml
# Restrict permissions (principle of least privilege)
permissions:
  contents: read
  pull-requests: write   # only if the workflow needs to comment on PRs

jobs:
  # Pin actions to a specific commit SHA, not just a tag
  # Tags can be moved; SHAs cannot
  - uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
```

Avoid using `pull_request_target` with untrusted code -- it runs with write permissions and can expose secrets. Use `pull_request` instead, which runs without write permissions.

## GitHub Releases

GitHub Releases pair a git tag with release notes and binary attachments.

```bash
# Create a release via CLI
gh release create v2.4.0 \
  --title "Release 2.4.0" \
  --notes "## What's Changed

### Features
- feat(notifications): add email notification for order confirmation (#234)
- feat(search): add fuzzy search to product catalog (#245)

### Bug Fixes
- fix(checkout): prevent crash when cart is empty (#256)

**Full Changelog**: https://github.com/org/repo/compare/v2.3.0...v2.4.0" \
  ./dist/app-linux-amd64 \
  ./dist/app-macos-amd64 \
  ./dist/app-windows-amd64.exe

# Create a draft release (not public until published)
gh release create v2.5.0-rc.1 --draft --prerelease

# List releases
gh release list

# Download a specific release asset
gh release download v2.4.0 --pattern "*.exe"
```

Automate release creation with GitHub Actions after `semantic-release` or a manual tag push:

```yaml
on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: npm run build
      - name: Create Release
        uses: softprops/action-gh-release@v2
        with:
          files: |
            dist/app-linux
            dist/app-macos
          generate_release_notes: true  # auto-generate from PRs merged since last release
```

## GitHub Packages

GitHub Packages hosts npm packages, Docker images, Maven artifacts, and more, directly in your GitHub organization.

### Publishing an npm package to GitHub Packages

```json
// package.json
{
  "name": "@your-org/your-package",
  "version": "1.0.0",
  "publishConfig": {
    "registry": "https://npm.pkg.github.com"
  }
}
```

```yaml
# .github/workflows/publish.yml
on:
  push:
    tags: ['v*']

jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      packages: write
      contents: read
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          registry-url: 'https://npm.pkg.github.com'
      - run: npm ci
      - run: npm publish
        env:
          NODE_AUTH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

### GitHub Container Registry

```bash
# Login to GitHub Container Registry
echo $GITHUB_TOKEN | docker login ghcr.io -u $GITHUB_USERNAME --password-stdin

# Build and push a Docker image
docker build -t ghcr.io/your-org/your-app:latest .
docker push ghcr.io/your-org/your-app:latest
```

```yaml
# GitHub Actions: build and push container
- name: Log in to GHCR
  uses: docker/login-action@v3
  with:
    registry: ghcr.io
    username: ${{ github.actor }}
    password: ${{ secrets.GITHUB_TOKEN }}

- name: Build and push
  uses: docker/build-push-action@v5
  with:
    push: true
    tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
```

## GitHub Pages

GitHub Pages hosts static sites directly from a repository.

```yaml
# .github/workflows/deploy-pages.yml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: ./dist

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## GitLab CI/CD

GitLab CI/CD is configured via `.gitlab-ci.yml` in the repository root.

### Anatomy of a GitLab pipeline

```yaml
# .gitlab-ci.yml

# Stages run in order. Jobs within a stage run in parallel.
stages:
  - build
  - test
  - deploy

variables:
  NODE_VERSION: "22"

# Template (hidden job with .)  that other jobs can extend
.node-base:
  image: node:${NODE_VERSION}
  before_script:
    - npm ci

# Build job
build:
  extends: .node-base
  stage: build
  script:
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 hour

# Test job (runs in parallel with lint because same stage)
test:
  extends: .node-base
  stage: test
  script:
    - npm test
  coverage: '/Lines\s*:\s*(\d+\.\d+)%/'  # regex to extract coverage %
  artifacts:
    reports:
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml
    expire_in: 1 week

# Lint job (same stage as test -- runs in parallel)
lint:
  extends: .node-base
  stage: test
  script:
    - npm run lint

# Deploy job
deploy-staging:
  stage: deploy
  environment:
    name: staging
    url: https://staging.example.com
  script:
    - ./deploy.sh staging
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
  needs:
    - build     # start as soon as build is done, no need to wait for test
```

### Rules and conditions

```yaml
deploy-production:
  stage: deploy
  environment: production
  script:
    - ./deploy.sh production
  rules:
    # Only run for version tags on main
    - if: '$CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+$/ && $CI_COMMIT_BRANCH == "main"'
      when: manual    # requires a click in the GitLab UI to proceed
    - when: never
```

### Merge request pipelines

GitLab can run pipelines specifically for merge requests, optionally simulating the result of the merge:

```yaml
# Run only on MR, not every branch push
test:
  stage: test
  script: npm test
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
```

### GitLab Environments and Deployments

```yaml
deploy-staging:
  environment:
    name: staging
    url: https://staging.example.com
    on_stop: stop-staging    # job to call when stopping the environment

stop-staging:
  environment:
    name: staging
    action: stop
  script:
    - ./undeploy.sh staging
  rules:
    - when: manual
```

Deployments track which version of the code is running in each environment. The GitLab UI shows deployment history, success/failure, and links to the running environment.

### GitLab Container Registry and Package Registry

```yaml
# Build and push Docker image to GitLab Container Registry
build-image:
  image: docker:latest

    IMAGE: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA
```

### GitLab built-in security scanning

GitLab includes security scanning as part of its CI/CD pipeline -- no external tools needed on most tiers:

```yaml
include:
  # SAST: static analysis for vulnerabilities in source code
  - template: Security/SAST.gitlab-ci.yml
  # Dependency scanning: vulnerable dependencies
  - template: Security/Dependency-Scanning.gitlab-ci.yml
  # Secret detection: leaked credentials in code
  - template: Security/Secret-Detection.gitlab-ci.yml
```

These templates add jobs to your pipeline automatically. Results appear in the MR as a security report, blocking merge if critical vulnerabilities are found (configurable).

## Common Mistakes

- **Storing secrets in environment variables set in workflow YAML:** Values assigned to `env:` at the workflow or job level are visible in the workflow definition and can be read in logs. Use `secrets` context for sensitive values.

- **Using `pull_request_target` for untrusted forks:** This trigger runs with write permissions and has access to secrets. If it checks out the PR's code and runs it, a malicious PR can exfiltrate secrets. Use `pull_request` (read-only) for untrusted code.

- **Not pinning action versions to commit SHAs:** A tag like `actions/checkout@v4` can be updated by the action author. A compromised action at that tag would run in your workflows. Pin to a specific commit SHA for security-sensitive workflows.

- **Running the entire test suite in `pre-commit` hooks:** This makes commits take minutes and trains developers to use `--no-verify`. Heavy checks belong in CI, not local hooks.

- **Uploading artifacts with no expiry:** Artifacts consume storage quota. Always set `retention-days` on artifacts that are not release artifacts.

- **Not using `needs:` for job ordering:** Jobs in the same stage do not share a filesystem. If a later job needs the build output from an earlier job, use `artifacts` + `needs:` (GitHub) or `artifacts` + `dependencies:` (GitLab).

- **Ignoring GitLab's built-in security templates:** Teams using GitLab often add external security scanning tools without knowing GitLab already includes SAST, secret detection, and dependency scanning as CI templates.

## Best Practices

- **Use `workflow_dispatch` inputs for manual deployments.** Adding environment selection as a manual trigger input prevents deploying to the wrong environment from a script error.

- **Set `fail-fast: false` in matrix builds.** By default, a failure in one matrix job cancels all others. You usually want to see all failures across all matrix combinations, not just the first one.

- **Use environments with required reviewers for production.** This creates a mandatory approval gate before any production deployment runs, regardless of who merged to main.

- **Cache dependencies keyed by lockfile hash.** `hashFiles('**/package-lock.json')` ensures the cache is invalidated when dependencies change, preventing stale dependency caches.

- **Use `concurrency` to prevent overlapping deployments.** Without it, a fast push of two commits can cause two simultaneous deployments that conflict in production.

- **Create issue templates.** `.github/ISSUE_TEMPLATE/` with markdown templates for bugs and features ensures issues contain the information you need before anyone starts working on them.

- **Use GitLab's `extends:` keyword to avoid YAML duplication.** Hidden jobs (prefixed with `.`) as base templates keep your pipeline DRY.

## Real-World Use Cases

### Full CI/CD pipeline for a web application

```yaml
# .github/workflows/pipeline.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: ${{ github.event_name == 'pull_request' }}

jobs:
  quality:
    name: Quality Checks
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm run type-check
      - run: npm test -- --coverage
      - uses: actions/upload-artifact@v4
        with:
          name: coverage
          path: coverage/
          retention-days: 7

  build:
    name: Build
    needs: quality
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
          retention-days: 1

  deploy-staging:
    name: Deploy to Staging
    needs: build
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment:
      name: staging
      url: https://staging.example.com
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: dist
          path: dist/
      - name: Deploy
        run: ./scripts/deploy.sh staging
        env:
          DEPLOY_TOKEN: ${{ secrets.STAGING_DEPLOY_TOKEN }}

  deploy-production:
    name: Deploy to Production
    needs: deploy-staging
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment:
      name: production      # has required reviewers configured
      url: https://example.com
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: dist
          path: dist/
      - name: Deploy
        run: ./scripts/deploy.sh production
        env:
          DEPLOY_TOKEN: ${{ secrets.PRODUCTION_DEPLOY_TOKEN }}
```

### GitLab review app per merge request

```yaml
# .gitlab-ci.yml

deploy-review:
  stage: deploy
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    url: https://$CI_COMMIT_REF_SLUG.review.example.com
    on_stop: stop-review
    auto_stop_in: 1 week    # automatically stop after 1 week
  script:
    - ./scripts/deploy-review.sh $CI_COMMIT_REF_SLUG
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"

stop-review:
  stage: deploy
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    action: stop
  script:
    - ./scripts/destroy-review.sh $CI_COMMIT_REF_SLUG
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
      when: manual
```

Each MR gets its own live URL (`feature-dark-mode.review.example.com`) for visual testing and stakeholder review, automatically destroyed when the MR is merged or closed.
