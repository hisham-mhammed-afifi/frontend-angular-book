# 10.2 Angular DevTools

> Angular DevTools is a Chrome/Firefox extension that exposes the component tree, dependency injection hierarchy, signal state, and change detection performance of Angular applications directly in the browser.

---

## Installation and Setup

Angular DevTools is a browser extension available for Chrome and Firefox.

```
Chrome: chrome.google.com/webstore  -> search "Angular DevTools"
Firefox: addons.mozilla.org         -> search "Angular DevTools"
```

Once installed, a new "Angular" tab appears in Chrome DevTools (F12) when you navigate to an Angular application running in development mode.

### Requirements

- The application must be compiled in **development mode**. Production builds strip out the debug hooks that Angular DevTools relies on.
- Angular version 9 or higher for the Component Explorer and Profiler.
- Angular version 17 or higher for the Injector Tree tab.
- Angular version 18+ for signal inspection capabilities.

If the Angular tab does not appear:
1. Verify the app is not a production build (`ng.getComponent` should be available in the console).
2. Hard refresh the page after installing the extension.
3. Close and reopen DevTools.

## Component Explorer

The Component Explorer is the primary view in Angular DevTools. It displays the live component tree mirroring the DOM structure, but showing Angular components and directives instead of raw HTML elements.

### Navigating the tree

The left panel shows the component hierarchy:

```
AppComponent
  HeaderComponent
    NavComponent
    SearchBarComponent
  MainContentComponent
    ProductListComponent
      ProductCardComponent   (x24)
    SidebarComponent
      FilterPanelComponent
  FooterComponent
```

Click any component to select it. The right panel immediately shows its properties.

### Inspecting component state

When a component is selected, the right panel displays:

- **Properties:** all class fields and their current values (including signals)
- **Inputs:** values received from the parent component
- **Outputs:** event emitters and their subscription status
- **View state:** template bindings and their evaluated values

```typescript
// Example: selecting ProductCardComponent in the explorer
// Right panel shows:
//
// Properties:
//   product: { id: 42, name: "Mechanical Keyboard", price: 149.99 }
//   isInCart: false           (signal value)
//   quantity: 1               (signal value)
//
// Inputs:
//   product: { id: 42, name: "Mechanical Keyboard", price: 149.99 }
//   featured: true
//
// Outputs:
//   addToCart: EventEmitter (0 subscribers)
```

### Editing values in real time

Double-click any property value to edit it live. The component re-renders with the new value immediately. This is useful for:
- Testing edge cases (empty strings, null values, extremely long text)
- Toggling boolean states without clicking through the UI
- Setting numeric values to boundary values (0, -1, Number.MAX_SAFE_INTEGER)

### Selecting a component from the page

Click the crosshair icon in the Angular DevTools toolbar, then click any element on the page. DevTools jumps to the Angular component that owns that element, similar to the Chrome Elements panel's inspect mode but at the component level.

### Accessing the component in the Console

After selecting a component in the Component Explorer, you can access it programmatically in the Console:

```javascript
// ng.getComponent() returns the component instance for a DOM element
// After selecting a component in Angular DevTools, its host element is $0
const comp = ng.getComponent($0);

// Now you can inspect or call methods
comp.product;
comp.addToCart.emit({ id: 42, quantity: 1 });

// For signal-based properties
comp.isInCart();      // read a signal value
comp.quantity.set(5); // set a signal value (triggers re-render)

// Access the component's injector
const injector = ng.getOwningNgModule($0);  // legacy
// or for standalone:
const injector = $0.__ngContext__;           // internal, use with caution
```

### Filtering the tree

The search bar at the top filters the component tree by name. Type `Product` to see only components containing "Product" in their name. This is essential in large applications with hundreds of components.

## Signal Inspection

Angular 18+ introduced signal-based reactivity, and Angular DevTools provides direct insight into signal state.

### Viewing signal values

Signal properties appear in the Component Explorer alongside regular properties. Their current value is displayed, and when a signal updates, the value refreshes in the panel.

```typescript
@Component({
  selector: 'app-counter',
  template: `
    <p>Count: {{ count() }}</p>
    <p>Double: {{ doubled() }}</p>
    <button (click)="increment()">+</button>
  `,
})
export class CounterComponent {
  count = signal(0);
  doubled = computed(() => this.count() * 2);

  increment() {
    this.count.update(c => c + 1);
  }
}

// In Angular DevTools Component Explorer:
//
// Properties:
//   count: 3                (signal)
//   doubled: 6              (computed)
//
// After clicking increment:
//   count: 4                (updated in real time)
//   doubled: 8              (updated in real time)
```

### Signal dependency visualization

Angular 19+ enhanced signal debugging with relationship visualization. DevTools can show:
- Which signals a `computed()` depends on
- Which `effect()` calls are triggered by a signal change
- The propagation path when a signal value changes

This makes it possible to trace unexpected re-computations: if a computed signal is updating too often, inspect its dependencies to find which source signal is changing.

### Debugging signal propagation

When a signal update triggers unexpected behavior:

1. Select the component that owns the signal in the Component Explorer
2. Note the signal's current value
3. Perform the action that triggers the unexpected behavior
4. Watch which signals update (values change in the panel)
5. Trace back: which signal changed first? Was it the right one?

You can also use `effect()` temporarily in your code for deeper tracing:

```typescript
// Temporary debugging effect (remove before committing)
effect(() => {
  console.log('count changed to:', this.count());
  console.trace();  // shows the call stack that triggered the change
});
```

## Profiler

The Profiler tab records change detection cycles and shows how long each component took to check. This is the primary tool for diagnosing slow rendering.

### Recording a profile

1. Switch to the **Profiler** tab in Angular DevTools
2. Click the **Record** button (circle icon)
3. Interact with the application (the action you want to profile)
4. Click **Stop**

DevTools displays a timeline of change detection cycles.

### Reading the profiler output

Each bar in the timeline represents one change detection cycle. Clicking a bar shows:

- **Which components were checked** during that cycle
- **How long each component took** to check (in milliseconds)
- **What triggered the cycle** (user event, timer, HTTP response, etc.)

```
Change Detection Cycle #12
Triggered by: click event on button.add-to-cart
Total time: 14.3ms

Components checked:
  AppComponent            0.2ms
  HeaderComponent         0.1ms
  MainContentComponent    0.3ms
  ProductListComponent    8.7ms    <-- bottleneck
    ProductCardComponent  0.3ms (x24 = 7.2ms)
  SidebarComponent        0.1ms
  FooterComponent         0.1ms
```

In this example, `ProductListComponent` and its 24 children account for most of the time. The fix might be:
- Use `OnPush` change detection strategy on `ProductCardComponent`
- Use `@defer` for off-screen product cards
- Use `trackBy` (or the `track` expression in `@for`) to avoid re-creating DOM nodes

### Identifying unnecessary change detection

Components highlighted in the profiler that did not change their output are being checked unnecessarily. Common causes:

```typescript
// BAD: default change detection checks every component on every cycle
@Component({
  selector: 'app-product-card',
  // no changeDetection specified = ChangeDetectionStrategy.Default
})

// GOOD: OnPush only checks when inputs change or signals update
@Component({
  selector: 'app-product-card',
  changeDetection: ChangeDetectionStrategy.OnPush,
})
```

With signals, Angular can skip change detection for components whose signal dependencies have not changed. The profiler confirms this: signal-based components appear with near-zero check times when their signals are unchanged.

### Flame chart view

The Profiler also offers a flame chart view that shows the call hierarchy during change detection. This reveals:
- Nested component checks (parent checking children)
- Time spent in lifecycle hooks (`ngOnInit`, `ngOnChanges`, `ngDoCheck`)
- Template expression evaluation time

Wide blocks in the flame chart indicate expensive operations that should be optimized.

## Injector Tree

The Injector Tree tab (Angular 17+) visualizes the dependency injection hierarchy. It shows two parallel trees:

### Environment injector hierarchy

```
NullInjector
  PlatformInjector
    RootInjector (ApplicationConfig providers)
      EnvironmentInjector (route-level providers)
```

### Element injector hierarchy

```
AppComponent
  HeaderComponent (provides: LoggerService)
    NavComponent
  MainContentComponent
    ProductListComponent (provides: ProductFilterService)
      ProductCardComponent
```

### Using the injector tree

1. Click any injector node to see its **configured providers** in the right panel
2. The resolution path highlights: when you select a specific injector, DevTools traces the path Angular walks up the tree to resolve a token
3. Find provider scope issues: if a service is accidentally provided at the component level instead of root, it creates multiple instances

```typescript
// Example: ProductFilterService is provided in ProductListComponent
// The injector tree shows it under ProductListComponent's element injector
// Every instance of ProductListComponent gets its own ProductFilterService

// If this is unintended (you wanted a singleton):
@Injectable({ providedIn: 'root' })  // move to root
export class ProductFilterService { }
```

### Debugging "No provider" errors

When Angular throws `NullInjectorError: No provider for SomeService`:

1. Open the Injector Tree tab
2. Search for `SomeService` in the providers
3. If it does not appear anywhere, the service is not provided
4. If it appears under a different injector than expected, the component requesting it may be outside that injector's scope

## Router State Inspection

The Component Explorer shows route-activated components in context. When navigating between routes, the tree updates to reflect the new component structure.

For deeper router debugging, use the Console:

```typescript
// Access the router instance
const router = ng.getComponent(
  document.querySelector('router-outlet')
)?.__ngContext__; // internal access

// Or inject Router in a component and inspect:
// In the Console after selecting a routed component:
const comp = ng.getComponent($0);
// If the component injects Router:
comp.router.url;                    // current URL
comp.router.config;                 // all route configurations
comp.activatedRoute.snapshot.params; // current route params
comp.activatedRoute.snapshot.data;   // route data
```

For route debugging, the best approach is adding a temporary service:

```typescript
// Temporary route debugging service (remove after debugging)
@Injectable({ providedIn: 'root' })
export class RouteDebugService {
  constructor(private router: Router) {
    this.router.events.subscribe(event => {
      if (event instanceof NavigationStart) {
        console.log('Navigation started:', event.url);
      }
      if (event instanceof NavigationEnd) {
        console.log('Navigation completed:', event.url);
      }
      if (event instanceof NavigationError) {
        console.error('Navigation failed:', event.error);
      }
    });
  }
}
```

## Console Debugging Utilities

Angular exposes several global functions in development mode that work independently of Angular DevTools.

### `ng.getComponent(element)`

Returns the component instance associated with a DOM element:

```javascript
// In the browser Console:
const el = document.querySelector('app-product-card');
const comp = ng.getComponent(el);
comp.product;           // inspect state
comp.addToCart.emit();  // trigger an output
```

### `ng.getDirectives(element)`

Returns an array of directive instances on a DOM element:

```javascript
const el = document.querySelector('[appHighlight]');
const directives = ng.getDirectives(el);
directives[0].color;  // inspect directive property
```

### `ng.getContext(element)`

Returns the context of an embedded view (e.g., inside `@for`, `@if`, or `*ngFor`):

```javascript
// For an element inside an @for loop:
const el = document.querySelector('.list-item');
const ctx = ng.getContext(el);
ctx.$implicit;  // the current iteration item
ctx.index;      // the iteration index
```

### `ng.applyChanges(component)`

Triggers change detection for a specific component:

```javascript
const comp = ng.getComponent(document.querySelector('app-counter'));
comp.count.set(99);       // update a signal
ng.applyChanges(comp);    // force re-render (signals usually auto-detect)
```

### Profiling from the Console

```javascript
// Enable debug tools (Angular 19+)
// In main.ts:
import { provideExperimentalDebugTools } from '@angular/core';

// Then in Console:
ng.profiler.timeChangeDetection();
// Output: ran 500 change detection cycles in 1432ms (2.86ms per cycle)
```

## Performance Debugging Workflow

A systematic approach to diagnosing performance issues with Angular DevTools:

### Step 1: Identify the symptom

Is the app slow on initial load, during navigation, or during interaction? Each has different debugging paths.

### Step 2: Profile change detection

```
1. Open Profiler tab
2. Record during the slow action
3. Find the longest change detection cycle
4. Identify the slowest components in that cycle
```

### Step 3: Check for unnecessary checks

```
In the profiler output, components that:
- Take > 0ms to check
- Did not actually change their rendered output
are being checked unnecessarily.

Fix: switch to OnPush or signal-based reactivity.
```

### Step 4: Check for expensive template expressions

```typescript
// BAD: method call in template re-evaluates every change detection cycle
@Component({
  template: `<div>{{ getExpensiveValue() }}</div>`,
})

// GOOD: compute once, store as signal
@Component({
  template: `<div>{{ expensiveValue() }}</div>`,
})
export class MyComponent {
  private data = input.required<Data>();
  expensiveValue = computed(() => this.computeExpensive(this.data()));
}
```

### Step 5: Check component count

In the Component Explorer, look at how many components are rendered. A page with 500+ components is likely over-componentized or missing virtualization.

```typescript
// For large lists, use virtual scrolling (CDK)
import { CdkVirtualScrollViewport, CdkVirtualFor } from '@angular/cdk/scrolling';

@Component({
  template: `
    <cdk-virtual-scroll-viewport itemSize="48" class="list-viewport">
      <div *cdkVirtualFor="let item of items" class="list-item">
        {{ item.name }}
      </div>
    </cdk-virtual-scroll-viewport>
  `,
  imports: [CdkVirtualScrollViewport, CdkVirtualFor],
})
```

## Common Mistakes

- **Running Angular DevTools against a production build.** Production builds remove Angular's debug metadata. DevTools will show a blank Angular tab or a message saying the application is not in development mode. Always test with `ng serve` (development mode).

- **Ignoring the profiler and relying on console.log for performance.** Adding `console.time` around template renders is imprecise and invasive. The Profiler gives exact per-component timing without modifying any code.

- **Not using OnPush after identifying unnecessary checks.** The Profiler shows which components are checked unnecessarily, but developers often note the issue without applying the fix. Switching to `ChangeDetectionStrategy.OnPush` or moving to signals eliminates the problem.

- **Confusing the element injector with the environment injector.** A service provided in a component's `providers` array creates a new instance for that component and its children (element injector). A service with `providedIn: 'root'` lives in the root environment injector. The Injector Tree tab makes this visible, but developers must understand what they are looking at.

- **Forgetting `ng.getComponent($0)` after selecting in the Elements panel.** Many developers select an element in Chrome's Elements panel and try to access its component, but forget to use `ng.getComponent()`. The `$0` reference alone gives you the DOM node, not the Angular component.

- **Leaving `debugger` statements and temporary `effect()` calls in committed code.** These debugging aids must be removed before committing. Use ESLint's `no-debugger` rule and code review to catch them.

## Best Practices

- **Start every performance investigation with the Profiler, not guesswork.** Record the slow interaction, find the bottleneck in the profiler output, then fix it. Guessing which component is slow leads to wasted optimization effort.

- **Use the Component Explorer's value editing for rapid testing.** Instead of building UI to test every edge case (empty strings, null, large numbers), select the component and edit the value directly in DevTools.

- **Use the Injector Tree to verify service scope.** Before debugging a "wrong data" issue, verify that the service is provided at the expected level. A service accidentally scoped to a component creates a separate instance that does not share state with sibling components.

- **Combine Angular DevTools with Chrome DevTools.** Angular DevTools shows the Angular-specific view (components, signals, DI). Chrome DevTools shows the underlying DOM, network requests, and JavaScript performance. Use both together: profile with Angular DevTools to find the slow component, then use Chrome's Performance panel to identify the exact JavaScript function causing the bottleneck.

- **Pin the Profiler to your development workflow.** Make it a habit to run a quick profile after implementing any list rendering or data-heavy component. Catching performance issues during development is far cheaper than after deployment.

## Real-World Use Cases

### Diagnosing slow list rendering

A product listing page with 200 items takes 400ms to render after filtering.

```
1. Open Angular DevTools Profiler tab
2. Record while applying a filter
3. Results show 3 change detection cycles:
   Cycle 1: 380ms
     ProductListComponent: 12ms
     ProductCardComponent: 1.5ms x 200 = 300ms
     FilterPanelComponent: 0.3ms

4. Problem: 200 ProductCardComponents are re-created on every filter change

5. Investigation in Component Explorer:
   - ProductListComponent uses @for without track expression
   - Every filter change destroys all 200 cards and creates new ones

6. Fix: add track expression
   @for (product of filteredProducts(); track product.id) {
     <app-product-card [product]="product" />
   }

7. Re-profile: Cycle 1 drops from 380ms to 45ms
   Only changed cards are re-rendered; unchanged cards are reused.
```

### Finding a service scope bug

A shopping cart badge shows "0 items" even after adding products. The cart service works correctly in the product page.

```
1. Open Angular DevTools Injector Tree tab
2. Search for CartService
3. Finding: CartService appears in TWO injectors:
   - Root environment injector (application-wide)
   - ProductPageComponent element injector

4. ProductPageComponent has:
   @Component({
     providers: [CartService],  // creates a LOCAL instance
   })

5. The product page uses its local CartService (correctly adds items).
   The header badge reads from the root CartService (always empty).

6. Fix: remove CartService from ProductPageComponent.providers.
   The root-level singleton is now used everywhere.

7. Verify in Injector Tree: CartService now appears only in the root injector.
```

### Profiling signal-based change detection

After migrating a dashboard from zone-based to signal-based reactivity, verify the improvement.

```
Before migration:
  Profiler shows: every WebSocket message triggers full change detection
  12 components checked per message, 8ms per cycle
  At 10 messages/second: 80ms/second spent in change detection

After migration:
  Profiler shows: WebSocket messages update only the affected signal
  Only 1-2 components checked per message, 0.5ms per cycle
  At 10 messages/second: 5ms/second spent in change detection

Signal-based approach: 16x reduction in change detection overhead.
```

The Profiler confirms the migration was effective by showing fewer components checked per cycle and shorter cycle times.

> **Key Takeaway:** Angular DevTools transforms invisible runtime behavior (change detection cycles, DI resolution, signal propagation) into visible, measurable data -- always start performance investigations with the Profiler tab, not with code changes based on guesswork.
