# 10.3 Debugging Strategies

> Debugging is not about finding bugs randomly -- it is the systematic process of forming hypotheses, testing them, and narrowing the search space until the root cause is isolated.

---

## The Scientific Method of Debugging

Effective debugging mirrors the scientific method. Developers who debug efficiently follow a structured process rather than changing code at random and hoping the bug disappears.

### The process

```
1. REPRODUCE the bug reliably
   - If you cannot reproduce it, you cannot verify a fix.
   - Define exact steps, exact inputs, exact environment.

2. OBSERVE the symptoms
   - What is the actual behavior?
   - What is the expected behavior?
   - What data is involved? What state is the application in?

3. FORM A HYPOTHESIS
   - Based on observations, what could cause this?
   - Where in the code path does the divergence happen?

4. TEST the hypothesis
   - Add a breakpoint, log, or assertion at the suspected location.
   - Does the data match your expectation at this point?

5. NARROW or PIVOT
   - If the hypothesis is confirmed: you found the area. Drill deeper.
   - If the hypothesis is wrong: eliminate this possibility and form a new one.

6. FIX and VERIFY
   - Apply the minimal fix.
   - Verify the original reproduction steps no longer trigger the bug.
   - Check for regressions (did the fix break something else?).
```

### The anti-pattern: shotgun debugging

```javascript
// BAD: changing random things until the bug goes away
// "Maybe if I add a setTimeout it'll work?"
setTimeout(() => {
  updateUI();
}, 100);

// "Still broken, let me try 500ms?"
setTimeout(() => {
  updateUI();
}, 500);

// This masks the timing issue without understanding the root cause.
// The bug will return under different conditions.
```

Shotgun debugging produces fragile fixes that break under different conditions. Always understand WHY a change fixes the bug before committing it.

## Debugging Technique: Binary Search (Wolf Fence)

When you know a bug exists somewhere in a large code path but do not know where, binary search narrows the location in O(log n) steps.

### In code

```javascript
// A function processes data through 8 steps.
// The output is wrong, but you do not know which step corrupts it.

function processOrder(order) {
  const step1 = validateOrder(order);
  const step2 = calculateTotals(step1);
  const step3 = applyDiscounts(step2);
  const step4 = calculateTax(step3);
  // Insert a check at the midpoint:
  console.log('After step 4:', JSON.stringify(step4));
  // Is the data correct here? If yes, the bug is in steps 5-8.
  // If no, the bug is in steps 1-4.
  const step5 = calculateShipping(step4);
  const step6 = formatLineItems(step5);
  const step7 = generateInvoice(step6);
  const step8 = sendConfirmation(step7);
  return step8;
}

// Round 1: check midpoint (step 4). Data is correct. Bug is in steps 5-8.
// Round 2: check step 6. Data is wrong. Bug is in step 5 or 6.
// Round 3: check step 5. Data is correct. Bug is in step 6.
// Found: formatLineItems() is the culprit. 3 checks instead of 8.
```

### In commits (git bisect)

```bash
# The bug exists in the current commit but not in v2.0.0.
# 500 commits between them. Binary search: ~9 checks.
git bisect start
git bisect bad HEAD
git bisect good v2.0.0
# Git checks out the midpoint. Test. Mark good or bad. Repeat.
git bisect run npm test
```

### In configuration

When a complex configuration causes unexpected behavior:

```
1. Start with a minimal configuration that works.
2. Add configuration options back one at a time.
3. When the bug reappears, the last added option is the cause.

# Or in reverse:
1. Start with the broken configuration.
2. Remove half the options.
3. Still broken? The cause is in the remaining half.
4. Fixed? The cause was in the removed half.
```

## Debugging Technique: Rubber Duck Debugging

Explaining a problem out loud (or in writing) forces you to articulate assumptions that you take for granted. Bugs often hide in those assumptions.

### How to do it

1. State the problem explicitly: "The shopping cart total shows $0 after adding an item."
2. Walk through the code path step by step, explaining each line to the duck (or a colleague).
3. At each step, state what you expect to happen and what actually happens.
4. When your explanation and the code diverge, you have found the bug.

```javascript
// "When the user clicks Add to Cart, handleAddToCart fires."
function handleAddToCart(product) {
  // "It adds the product to the cart array."
  cart.push(product);
  // "Then it calculates the new total."
  const total = cart.reduce((sum, item) => sum + item.price, 0);
  // "Wait -- I said 'item.price' but the product object uses 'item.cost'..."
  // BUG FOUND: property name mismatch. item.price is undefined.
  // undefined + undefined = NaN, displayed as $0.
}
```

The act of verbalizing "I expect this to be X" and then checking makes implicit assumptions explicit.

## Debugging Technique: Divide and Conquer with Isolation

When a bug only appears in a complex system, isolate the suspected component in a minimal reproduction.

### Creating a minimal reproduction

```javascript
// The bug: "Clicking submit sends duplicate requests."
// The component has 200 lines of code with form validation,
// analytics tracking, error handling, and retry logic.

// Step 1: Create a minimal version that only has the submit logic.
function minimalSubmit() {
  console.log('submit called');
  fetch('/api/submit', { method: 'POST', body: '{}' });
}

// Step 2: Wire the minimal version to the button.
document.querySelector('#submit').addEventListener('click', minimalSubmit);

// Step 3: Does the minimal version duplicate?
// No -> the bug is in the removed code (validation, retry, analytics).
// Yes -> the bug is in the event binding or the component lifecycle.

// Step 4: Add code back incrementally until the bug reappears.
```

This technique is also how you create a reproduction for bug reports. A minimal reproduction eliminates irrelevant context and focuses the investigation.

## Debugging Async Code

Asynchronous code is the #1 source of subtle bugs in JavaScript applications. Race conditions, unhandled rejections, and stale closures are difficult to reproduce and often appear only under specific timing conditions.

### Race conditions

A race condition occurs when the outcome depends on the timing of two or more asynchronous operations.

```javascript
// BUG: user types "react" into search. Results for "rea" arrive AFTER
// results for "react" because the network request for "rea" was slower.
// The UI shows stale results.

// BAD: no cancellation
async function search(query) {
  const response = await fetch(`/api/search?q=${query}`);
  const results = await response.json();
  renderResults(results);  // stale results overwrite correct results
}

input.addEventListener('input', (e) => search(e.target.value));

// GOOD: AbortController cancels the previous request
let currentController = null;

async function search(query) {
  // Cancel the previous in-flight request
  if (currentController) {
    currentController.abort();
  }
  currentController = new AbortController();

  try {
    const response = await fetch(`/api/search?q=${query}`, {
      signal: currentController.signal,
    });
    const results = await response.json();
    renderResults(results);
  } catch (err) {
    if (err.name === 'AbortError') {
      // Request was intentionally cancelled. Not an error.
      return;
    }
    throw err;
  }
}
```

### Debugging strategy for race conditions

```
1. Add timestamps to every async operation:
   console.log(`[${Date.now()}] Request sent for: ${query}`);
   console.log(`[${Date.now()}] Response received for: ${query}`);

2. Slow down the network in Chrome DevTools (Throttling: Slow 3G).
   Race conditions that happen 1% of the time on fast connections
   happen 90% of the time on slow connections.

3. Check: is the response being used still valid?
   (Is it for the current query, the current user, the current page?)
```

### Stale closures

```javascript
// BUG: the interval always logs "0" even after count changes.
let count = 0;

setInterval(() => {
  console.log(count);  // always 0 if count was reassigned with let
  // Wait -- `let count = 0` means `count` in the closure IS the
  // outer variable. If it updates, the closure sees the update.
  // But if this were React state:
  // const [count, setCount] = useState(0);
  // setInterval captures the INITIAL count value and never sees updates.
}, 1000);

// The stale closure pattern: the function captures a variable at
// creation time and never sees subsequent changes.

// FIX for React-like patterns: use a ref or move the interval
// setup into an effect that re-creates when count changes.
```

### Unhandled promise rejections

```javascript
// BAD: the fetch rejection is silently swallowed
async function loadData() {
  const data = await fetch('/api/data');  // what if this rejects?
  return data.json();
}

loadData();  // no .catch(), no try/catch around the caller

// When /api/data returns 500, fetch does NOT reject.
// fetch only rejects on network failure.
// A 500 response is a resolved promise with response.ok === false.

// GOOD: check both network errors and HTTP errors
async function loadData() {
  try {
    const response = await fetch('/api/data');

    if (!response.ok) {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }

    return await response.json();
  } catch (err) {
    // Network error OR HTTP error OR JSON parse error
    console.error('loadData failed:', err);
    throw err;  // re-throw so callers know about the failure
  }
}
```

### Catching all unhandled rejections globally

```javascript
// In development, log all unhandled rejections to the console
window.addEventListener('unhandledrejection', (event) => {
  console.error('Unhandled promise rejection:', event.reason);
  console.trace();
});
```

## Debugging Technique: Working Backwards from the Symptom

Instead of reading code top-down trying to find the bug, start from the observable symptom and trace backwards.

```
Symptom: "The price shows NaN on the checkout page."

Step 1: Find where the price is rendered.
  --> checkout.html line 42: {{ total }}

Step 2: Find where `total` is assigned.
  --> checkout.ts line 78: this.total = calculateTotal(this.items);

Step 3: Find what calculateTotal returns.
  --> Add breakpoint at line 78. Inspect this.items.
  --> items[2].price is undefined.

Step 4: Find where items[2] was created.
  --> cart-service.ts line 33: items are loaded from localStorage.

Step 5: Inspect localStorage.
  --> The stored item has "cost" not "price" (legacy data format).

Root cause: Migration from "cost" to "price" did not update
existing localStorage data. Old items use "cost", new code reads "price".

Fix: Add a migration step that renames "cost" to "price" on load.
```

Working backwards from the symptom is faster than reading code forward because it follows the data flow that produced the bug, skipping all irrelevant code paths.

## Debugging Technique: Assertions and Invariants

Assertions check conditions that must always be true at a specific point. When an assertion fails, it immediately surfaces the bug instead of letting it propagate.

```typescript
// Without assertions: NaN propagates silently through calculations
function calculateDiscount(price: number, percentage: number): number {
  return price * (1 - percentage / 100);
  // If percentage is undefined, returns NaN. No error thrown.
  // NaN flows into the total, then the UI. Hard to trace back.
}

// With assertions: the bug surfaces at its origin
function calculateDiscount(price: number, percentage: number): number {
  console.assert(
    typeof price === 'number' && !Number.isNaN(price),
    `Expected price to be a valid number, got: ${price}`
  );
  console.assert(
    typeof percentage === 'number' && percentage >= 0 && percentage <= 100,
    `Expected percentage between 0-100, got: ${percentage}`
  );

  return price * (1 - percentage / 100);
}
```

For production code, use a more robust assertion utility:

```typescript
function assert(condition: boolean, message: string): asserts condition {
  if (!condition) {
    throw new Error(`Assertion failed: ${message}`);
  }
}

function calculateDiscount(price: number, percentage: number): number {
  assert(!Number.isNaN(price), `price is NaN`);
  assert(percentage >= 0 && percentage <= 100, `percentage out of range: ${percentage}`);
  return price * (1 - percentage / 100);
}
```

TypeScript's `asserts condition` return type narrows the type after the assertion, so the compiler knows the values are valid.

## Debugging Technique: Logging Strategically

Effective logging is structured, filterable, and informative. Scattershot `console.log('here')` statements are the opposite.

### Structured logging

```typescript
// BAD: unstructured, unfilterable
console.log('data', data);
console.log('here 1');
console.log('made it');

// GOOD: structured with context
function logDebug(module: string, action: string, data?: unknown) {
  console.log(`[${module}] ${action}`, data ?? '');
}

logDebug('CartService', 'addItem', { productId: 42, quantity: 1 });
logDebug('CartService', 'calculateTotal', { itemCount: 3, total: 299.97 });
logDebug('AuthService', 'tokenRefresh', { expiresIn: 3600 });
```

### Conditional logging with namespaces

```typescript
// Create a debug logger that can be enabled per module
function createLogger(namespace: string) {
  const enabled = localStorage.getItem('debug')?.includes(namespace) ?? false;

  return {
    log: (...args: unknown[]) => {
      if (enabled) console.log(`[${namespace}]`, ...args);
    },
    warn: (...args: unknown[]) => {
      if (enabled) console.warn(`[${namespace}]`, ...args);
    },
    error: (...args: unknown[]) => {
      // Errors always log, regardless of debug setting
      console.error(`[${namespace}]`, ...args);
    },
  };
}

// Usage:
const log = createLogger('cart');
log.log('Item added', item);  // only logs if localStorage.debug includes 'cart'

// Enable in browser console:
// localStorage.setItem('debug', 'cart,auth,api');
// Reload the page. Only those namespaces will log.
```

### Console.table for structured data

```javascript
// Instead of logging arrays of objects as collapsed trees:
console.log(users);

// Use console.table for a readable view:
console.table(users, ['name', 'email', 'role']);
// Output:
// (index) | name    | email             | role
// 0       | Alice   | alice@example.com | admin
// 1       | Bob     | bob@example.com   | editor
// 2       | Carol   | carol@example.com | viewer
```

## Debugging Network Issues

### Verify the request before blaming the backend

```
1. Open Network panel
2. Find the request
3. Check:
   - Is the URL correct? (typo? wrong environment?)
   - Is the method correct? (GET vs POST?)
   - Are the headers correct? (missing Authorization?)
   - Is the payload correct? (correct JSON? correct field names?)
   - Is the response what you expect? (correct status code? correct body?)
```

### CORS errors

CORS errors appear in the Console but the actual error details are in the Network panel:

```
Console: "Access to fetch at 'https://api.example.com/data' from origin
'http://localhost:4200' has been blocked by CORS policy."

Network panel: the request shows as "failed" or "(blocked:cors)"

Common causes:
  - The backend does not include `Access-Control-Allow-Origin` header
  - The backend does not allow the specific HTTP method (OPTIONS preflight fails)
  - The request includes custom headers not listed in `Access-Control-Allow-Headers`

Quick check: Does the same request work from Postman or curl?
  - Yes -> CORS issue (browser-specific policy). Fix the backend headers.
  - No -> The API itself is broken. Not a CORS issue.
```

### Caching issues

```
Symptom: "I deployed a fix but the bug is still there."

Check:
1. Hard refresh: Ctrl+Shift+R (bypasses cache)
2. Open Network panel > check "Disable cache"
3. Application panel > Storage > Clear site data
4. Check for service worker caching: Application > Service Workers

If a service worker is caching responses, you may need to:
  - Unregister the service worker
  - Update the cache version in the service worker code
  - Use cache-busting query strings or filenames
```

## Debugging TypeScript-Specific Issues

### Trusting types too much

```typescript
// TypeScript guarantees types at compile time, not at runtime.
// Data from external sources (API, localStorage, URL params)
// can violate the type at runtime.

interface User {
  id: number;
  name: string;
  email: string;
}

// BAD: trusting the API response blindly
const user: User = await response.json();
// If the API returns { id: "42", name: null }, TypeScript
// does not catch this at runtime. user.name.toUpperCase() crashes.

// GOOD: validate at the boundary
import { z } from 'zod';

const UserSchema = z.object({
  id: z.number(),
  name: z.string(),
  email: z.string().email(),
});

const rawData = await response.json();
const user = UserSchema.parse(rawData);  // throws if validation fails
```

### `as` assertions hiding bugs

```typescript
// BAD: `as` assertion silences the compiler but the bug persists
const value = someFunction() as string;
value.toUpperCase();  // crashes if someFunction() returns undefined

// GOOD: runtime narrowing
const value = someFunction();
if (typeof value === 'string') {
  value.toUpperCase();  // safe
} else {
  throw new Error(`Expected string, got ${typeof value}`);
}
```

### `unknown` in catch blocks

```typescript
// BAD: accessing .message on an unknown error type
try {
  await doSomething();
} catch (err) {
  console.error(err.message);  // TypeScript error: err is unknown
}

// GOOD: narrow the error type
try {
  await doSomething();
} catch (err) {
  if (err instanceof Error) {
    console.error(err.message);
  } else {
    console.error('Non-error thrown:', err);
  }
}

// Utility function for consistent error handling
function toError(err: unknown): Error {
  if (err instanceof Error) return err;
  if (typeof err === 'string') return new Error(err);
  return new Error(JSON.stringify(err));
}
```

## Common Mistakes

- **Changing code before understanding the bug.** The fix must match the root cause. Applying a workaround without understanding why it works creates technical debt and hides the real issue.

- **Not reproducing the bug first.** If you cannot reproduce it, you cannot verify your fix. "It works on my machine" means the bug is environment-specific. Document exact reproduction steps: browser, OS, user data, network conditions, timing.

- **Debugging minified code.** Always ensure source maps are available in development. Debugging `a.b.c(d)` in minified code wastes time. If source maps are not loading, check that the build tool is configured to generate them and that the `.map` files are accessible.

- **Adding `console.log('here')` everywhere.** Unstructured logging produces noise. Use structured logging with module names, actions, and relevant data. Even better, use breakpoints -- they show all variables at once without code changes.

- **Ignoring the error message.** Error messages often contain the exact line number, file, and description of the problem. Read the full stack trace before diving into code.

- **Assuming the bug is in your code.** Sometimes the bug is in a dependency, a browser quirk, or the build tool. Check the issue tracker of the library. Search for the exact error message.

- **Not considering state as the bug source.** Many bugs are caused by unexpected application state, not by incorrect logic. Inspect the state (variables, localStorage, cookies, URL params) at the moment the bug occurs.

- **Fixing the symptom instead of the cause.** Adding a null check where a value should never be null masks the bug. Find out why the value is null and fix that.

---

## Best Practices

- **Reproduce first, fix second.** Write a failing test that reproduces the bug before touching the implementation. The test becomes a regression guard. If you cannot write a test, document the exact reproduction steps.

- **Use the scientific method.** Hypothesize, test, narrow. Do not randomly change code and check if the bug disappears. Each change should test a specific hypothesis.

- **Binary search large code paths.** When you do not know where the bug is, split the code path in half and check the midpoint. This is O(log n) instead of O(n).

- **Read the error message completely.** The line number in the stack trace is the starting point, not the root cause. Trace up the stack to find where the bad data originated.

- **Check external boundaries first.** When data is wrong, verify it at the boundary: the API response, the localStorage value, the URL parameter, the event payload. The bug is often in the data, not the logic.

- **Keep your debugging changes reversible.** Use breakpoints and logpoints instead of modifying source code. When you must modify code, use `git stash` or a dedicated debug branch so you can cleanly remove all debug code.

- **Pair debug with a colleague.** Explaining the problem to another person (or a rubber duck) forces you to articulate assumptions. The bug is often in an assumption you take for granted.

- **Time-box your investigation.** If you have been debugging for 30 minutes without progress, step back. Take a break. Explain the problem to someone. Change your approach. Continued staring at the same code with the same assumptions produces diminishing returns.

## Real-World Use Cases

### Debugging a race condition in search autocomplete

Users report that search results sometimes show results for a previous query.

```
1. REPRODUCE: Type "react" quickly, then backspace to "rea".
   Under normal conditions, hard to reproduce.
   Enable "Slow 3G" throttling in Network panel.
   Now it happens every time.

2. OBSERVE: Network panel shows:
   - Request for "react" sent at T+0ms
   - Request for "rea" sent at T+200ms
   - Response for "rea" arrives at T+800ms -> renders "rea" results
   - Response for "react" arrives at T+1200ms -> OVERWRITES with "react" results
   The later request returned first. The earlier request returned second.

3. HYPOTHESIS: No request cancellation. Stale responses overwrite current results.

4. TEST: Add console.log with timestamps to the fetch handler.
   Confirm that the response for "react" arrives after "rea".

5. FIX: Add AbortController. Cancel the previous request when a new one starts.

6. VERIFY: Under Slow 3G, results now always match the current input.
```

### Debugging a localStorage data migration issue

After a deployment, some users see "NaN" for prices on the checkout page.

```
1. REPRODUCE: Cannot reproduce with a fresh browser.
   Hypothesis: existing localStorage data causes the bug.
   Check a bug report: user had items in cart from before the deployment.

2. OBSERVE: Open Application panel > localStorage.
   Old items have { cost: 29.99 }. New code reads { price: 29.99 }.
   item.price is undefined. undefined in arithmetic produces NaN.

3. HYPOTHESIS: The codebase was refactored from "cost" to "price" but
   existing localStorage data was not migrated.

4. TEST: Manually set localStorage with old format. Refresh. NaN appears.
   Set localStorage with new format. Refresh. Correct prices.

5. FIX: Add a migration function that runs on app startup:
```

```typescript
function migrateCartData() {
  const raw = localStorage.getItem('cart');
  if (!raw) return;

  const items = JSON.parse(raw);
  let migrated = false;

  for (const item of items) {
    if ('cost' in item && !('price' in item)) {
      item.price = item.cost;
      delete item.cost;
      migrated = true;
    }
  }

  if (migrated) {
    localStorage.setItem('cart', JSON.stringify(items));
    console.log('Cart data migrated: "cost" -> "price"');
  }
}
```

```
6. VERIFY: Load the app with old localStorage format. Migration runs.
   Prices display correctly. No NaN.
```

### Debugging a "works locally, fails in CI" issue

Tests pass on the developer's machine but fail in CI with a timeout error.

```
1. REPRODUCE locally: tests pass in 8 seconds.
   REPRODUCE in CI: tests timeout after 30 seconds on one specific test.

2. OBSERVE: The failing test makes an API call to localhost:3000.
   Locally, the developer has a dev server running on port 3000.
   CI does not have this server.

3. HYPOTHESIS: The test depends on an external server that is not
   available in CI. The fetch hangs until timeout.

4. TEST: Stop the local dev server. Run tests locally.
   Same timeout error. Confirmed.

5. FIX: Mock the API call in the test:
```

```typescript
// Before: test depends on a running server
it('should load user data', async () => {
  const data = await userService.getUser(42);
  expect(data.name).toBe('Alice');
});

// After: test mocks the HTTP call
it('should load user data', async () => {
  vi.spyOn(global, 'fetch').mockResolvedValue(
    new Response(JSON.stringify({ name: 'Alice' }), { status: 200 })
  );

  const data = await userService.getUser(42);
  expect(data.name).toBe('Alice');
});
```

```
6. VERIFY: Tests pass both locally and in CI without a running server.
```

> **Key Takeaway:** Every debugging session should follow a structured process: reproduce the bug, form a hypothesis, test it, and narrow the search space -- random code changes are not debugging, they are gambling.
