# 10.5 Terminal Essentials

> The terminal is the unifying interface behind every build tool, version manager, package manager, and deployment script in web development -- fluency in it removes friction from every task.

---

## Shell Basics

A shell is a program that interprets commands. The most common shells are:

| Shell | Platform | Notes |
|-------|----------|-------|
| **Bash** | Linux, macOS (pre-Catalina), WSL | The de facto standard; most tutorials assume Bash |
| **Zsh** | macOS (default since Catalina), Linux | Bash-compatible with extra features (better completion, themes) |
| **PowerShell** | Windows | Not Bash-compatible; different syntax for many operations |
| **Git Bash** | Windows | Bash-like environment bundled with Git for Windows |

For cross-platform consistency, use Bash or Zsh. On Windows, use Git Bash, WSL (Windows Subsystem for Linux), or the VS Code integrated terminal configured to use Bash.

## Navigation

```bash
# Print the current working directory
pwd
# /home/user/projects/my-app

# List files and directories
ls              # basic listing
ls -la          # long format, include hidden files (dotfiles)
ls -lah         # long format, hidden files, human-readable sizes
ls -lt          # sorted by modification time (newest first)

# Change directory
cd src               # move into src/
cd ..                # move up one level
cd ../..             # move up two levels
cd ~                 # move to home directory
cd -                 # move to the previous directory (toggle between two)
cd ~/projects/app    # absolute path from home

# Tab completion: type the first few characters and press Tab
# cd sr<Tab> -> cd src/
# If ambiguous, press Tab twice to show options
```

### Path types

```bash
# Absolute path: starts from the root
/home/user/projects/my-app/src/app.ts

# Relative path: starts from the current directory
src/app.ts          # relative to pwd
./src/app.ts        # same (explicit current directory)
../shared/utils.ts  # relative, going up one level first

# Home shorthand
~/projects          # expands to /home/user/projects (Linux/macOS)
```

## File and Directory Operations

```bash
# Create files
touch index.html              # creates an empty file (or updates timestamp)
touch src/{app,main,styles}.ts  # creates three files using brace expansion

# Create directories
mkdir components              # single directory
mkdir -p src/app/features/auth  # create parent directories as needed

# Copy
cp file.txt backup.txt         # copy file
cp -r src/ src-backup/         # copy directory recursively

# Move / Rename
mv old-name.ts new-name.ts     # rename a file
mv component.ts src/app/       # move to another directory
mv src/old-dir/ src/new-dir/   # rename a directory

# Delete
rm file.txt                    # delete a file (no confirmation)
rm -r directory/               # delete a directory and all contents
rm -rf node_modules/           # force delete (no prompts, no errors if missing)

# WARNING: rm has no trash/recycle bin. Deleted files are gone permanently.
# Consider using `trash-cli` (npm install -g trash-cli) for recoverable deletes:
# trash file.txt
```

### Brace expansion

```bash
# Create multiple files/directories at once
mkdir -p src/{components,services,utils}
# Creates: src/components, src/services, src/utils

touch src/app.{ts,html,css,spec.ts}
# Creates: src/app.ts, src/app.html, src/app.css, src/app.spec.ts

# Rename with brace expansion
mv src/app.{ts,service.ts}
# Same as: mv src/app.ts src/app.service.ts
```

## Viewing and Searching File Contents

```bash
# Display entire file
cat package.json

# Display with line numbers
cat -n src/app.ts

# Display first/last lines
head -20 src/app.ts            # first 20 lines
tail -20 src/app.ts            # last 20 lines
tail -f logs/server.log        # follow: continuously show new lines (live log)

# Page through large files
less src/app.ts
# Navigation inside less:
#   Space    - next page
#   b        - previous page
#   /pattern - search forward
#   n        - next search result
#   q        - quit

# Search file contents with grep
grep "TODO" src/*.ts                    # search for "TODO" in all .ts files in src/
grep -r "console.log" src/              # recursive search in all files under src/
grep -rn "console.log" src/             # include line numbers
grep -ri "error" src/ --include="*.ts"  # case-insensitive, only .ts files
grep -rn "fetchUser" src/ --include="*.ts" --include="*.html"
grep -v "node_modules" output.txt       # invert match (exclude lines containing pattern)
grep -c "import" src/app.ts             # count matching lines
```

### Regular expressions in grep

```bash
# Find function declarations
grep -rn "function \w\+" src/ --include="*.ts"

# Find lines starting with "export"
grep -rn "^export" src/ --include="*.ts"

# Find lines ending with a semicolon
grep -rn ";$" src/ --include="*.ts"

# Extended regex (-E) for more patterns
grep -rEn "import .+ from '.+'" src/ --include="*.ts"
```

## Finding Files

```bash
# Find by name
find . -name "*.ts"                           # all .ts files from current directory
find src/ -name "*.spec.ts"                   # all spec files under src/
find . -name "*.ts" -not -path "*/node_modules/*"  # exclude node_modules

# Find by type
find . -type f -name "*.ts"                   # files only
find . -type d -name "components"             # directories only

# Find by modification time
find . -name "*.ts" -mtime -1                 # modified in the last 24 hours
find . -name "*.ts" -mmin -60                 # modified in the last 60 minutes

# Find by size
find . -type f -size +1M                      # files larger than 1MB
find . -type f -size +100k -name "*.js"       # JS files larger than 100KB

# Find and execute a command on results
find . -name "*.log" -delete                  # delete all .log files
find . -name "*.ts" -exec wc -l {} +          # count lines in all .ts files
```

### fd: a faster alternative to find

`fd` is a modern replacement for `find` with simpler syntax and better defaults (respects `.gitignore`, colored output, faster):

```bash
# Install: brew install fd (macOS), apt install fd-find (Ubuntu)

fd ".ts$"                      # find all .ts files
fd "component" --type f        # find files containing "component" in the name
fd ".spec.ts$" src/            # find spec files under src/
```

## Pipes and Redirection

Pipes (`|`) pass the output of one command as input to the next. Redirection (`>`, `>>`) sends output to files.

```bash
# Pipe: chain commands
ls -la | grep ".ts"                    # list files, then filter for .ts
cat package.json | grep "scripts" -A 20  # show scripts section of package.json
find . -name "*.ts" | wc -l           # count TypeScript files
history | grep "git"                   # find git commands from history

# Pipe to sort and deduplicate
grep -roh "import .* from '.*'" src/ --include="*.ts" | sort | uniq

# Redirection: write output to a file
ls -la > filelist.txt          # overwrite file with output
ls -la >> filelist.txt         # append to file
echo "Build started" > build.log

# Redirect stderr
npm run build 2> errors.log           # only errors go to file
npm run build > output.log 2>&1       # both stdout and stderr to file
npm run build > /dev/null 2>&1        # discard all output (silent)

# Here document: multi-line input
cat << 'EOF' > config.json
{
  "port": 3000,
  "host": "localhost"
}
EOF
```

### Useful pipe chains

```bash
# Find the 10 largest files in a project
find . -type f -not -path "*/node_modules/*" -exec du -h {} + | sort -rh | head -10

# Count lines of TypeScript code (excluding tests and node_modules)
find . -name "*.ts" -not -name "*.spec.ts" -not -path "*/node_modules/*" \
  | xargs wc -l | tail -1

# List all unique npm packages imported in the project
grep -roh "from '[^']*'" src/ --include="*.ts" \
  | sed "s/from '//;s/'//" \
  | grep -v "^\." \
  | sort | uniq

# Find TODO comments with file and line number
grep -rn "TODO\|FIXME\|HACK\|XXX" src/ --include="*.ts"
```

## Environment Variables

```bash
# View an environment variable
echo $PATH
echo $HOME
echo $NODE_ENV

# Set a variable for the current session
export NODE_ENV=production
export API_URL=https://api.example.com

# Set a variable for a single command
NODE_ENV=production npm run build

# View all environment variables
env
env | grep NODE

# Persist variables across sessions: add to ~/.bashrc or ~/.zshrc
echo 'export EDITOR="code --wait"' >> ~/.zshrc
source ~/.zshrc  # reload without restarting terminal
```

### The PATH variable

`PATH` is a colon-separated list of directories where the shell searches for executables.

```bash
echo $PATH
# /usr/local/bin:/usr/bin:/bin:/home/user/.local/bin

# When you type "node", the shell searches these directories in order:
# 1. /usr/local/bin/node
# 2. /usr/bin/node
# 3. /bin/node
# 4. /home/user/.local/bin/node
# It uses the first match.

# Add a directory to PATH
export PATH="$HOME/.local/bin:$PATH"

# Check where a command is located
which node        # /usr/local/bin/node
which npm         # /usr/local/bin/npm
```

## Node Version Management

Different projects require different Node.js versions. A version manager lets you switch between them instantly.

### fnm (Fast Node Manager)

fnm is written in Rust and is significantly faster than nvm (instant shell startup vs 2-3 seconds). It works on all platforms including Windows natively.

```bash
# Install fnm
# macOS: brew install fnm
# Windows: winget install Schniz.fnm
# Linux: curl -fsSL https://fnm.vercel.app/install | bash

# Install a Node version
fnm install 22
fnm install 20

# Use a specific version
fnm use 22

# Set a default version
fnm default 22

# List installed versions
fnm list

# List available versions
fnm list-remote

# Check current version
fnm current
node --version
```

### `.nvmrc` / `.node-version` for per-project versions

```bash
# Create a version file in the project root
echo "22" > .nvmrc
# or
echo "22" > .node-version

# fnm auto-switches when you cd into the project (with shell setup):
# Add to ~/.zshrc or ~/.bashrc:
eval "$(fnm env --use-on-cd)"

# Now:
cd ~/projects/legacy-app    # .nvmrc says 18 -> fnm switches to Node 18
cd ~/projects/new-app       # .nvmrc says 22 -> fnm switches to Node 22
```

### nvm (Node Version Manager)

nvm is the original Node version manager. It is Bash-based and does not run natively on Windows (requires WSL or Git Bash).

```bash
# Install nvm (Linux/macOS)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.0/install.sh | bash

# Same commands as fnm, slightly different syntax
nvm install 22
nvm use 22
nvm alias default 22
nvm ls
nvm ls-remote
```

| Feature | fnm | nvm |
|---------|-----|-----|
| Language | Rust | Bash |
| Shell startup impact | Near zero | 2-3 seconds |
| Windows native | Yes | No (WSL only) |
| `.nvmrc` support | Yes | Yes |
| Auto-switch on `cd` | Built-in | Requires manual shell config |

**Recommendation:** Use fnm for new setups. Use nvm if your team already uses it and migration is not worth the effort.

## npm Essentials

### Package installation

```bash
# Install all dependencies from package.json
npm install          # or: npm i

# Install a production dependency
npm install lodash

# Install a dev dependency
npm install --save-dev jest

# Install globally (avoid when possible; prefer npx)
npm install -g serve

# Install a specific version
npm install react@18.2.0

# Remove a package
npm uninstall lodash

# Clean install (delete node_modules and install from lockfile exactly)
npm ci               # faster and more deterministic than npm install

# Update packages
npm outdated         # show which packages have newer versions
npm update           # update within semver ranges in package.json
```

### `npm ci` vs `npm install`

| Feature | `npm install` | `npm ci` |
|---------|---------------|----------|
| Uses lockfile | Loosely | Strictly (exact versions) |
| Modifies lockfile | Yes (if needed) | Never (fails if mismatch) |
| Deletes node_modules | No | Yes (fresh install) |
| Speed | Slower | Faster |
| Use in CI | Avoid | Recommended |

Always use `npm ci` in CI pipelines for reproducible builds.

### npm scripts

```json
// package.json
{
  "scripts": {
    "start": "node server.js",
    "dev": "nodemon server.js",
    "build": "tsc && node dist/server.js",
    "test": "jest",
    "test:watch": "jest --watch",
    "test:coverage": "jest --coverage",
    "lint": "eslint src/",
    "lint:fix": "eslint src/ --fix",
    "format": "prettier --write 'src/**/*.{ts,html,css,json}'",
    "prepare": "husky",
    "precommit": "lint-staged"
  }
}
```

```bash
# Run a script
npm run build
npm run test:watch

# Built-in shortcuts (no "run" needed)
npm start            # same as npm run start
npm test             # same as npm run test

# Pass arguments to a script
npm run test -- --verbose --coverage
# The -- separates npm arguments from script arguments

# Run scripts sequentially
npm run lint && npm run test && npm run build

# List available scripts
npm run              # shows all scripts defined in package.json
```

### npx: run without installing

```bash
# Run a package without installing it globally
npx create-react-app my-app
npx serve dist/
npx eslint src/

# Run a specific version
npx -p node@18 node --version

# Run from a specific package
npx -p typescript tsc --init
```

`npx` checks `node_modules/.bin` first, then the global cache, then downloads temporarily. It is the preferred way to run CLI tools that you do not want installed globally.

## Process Management

```bash
# Run a command in the background
npm run dev &
# [1] 12345   (job number and PID)

# List background jobs
jobs
# [1]+  Running    npm run dev &

# Bring a background job to the foreground
fg %1

# Stop a foreground process
Ctrl+C              # sends SIGINT (interrupt)

# Suspend a foreground process and send to background
Ctrl+Z              # sends SIGTSTP (suspend)
bg %1               # resume in background

# Kill a process by PID
kill 12345           # graceful termination (SIGTERM)
kill -9 12345        # force kill (SIGKILL, last resort)

# Find a process by name
ps aux | grep "node"

# Find what is using a specific port
lsof -i :4200       # macOS/Linux
# or:
netstat -tlnp | grep 4200
```

### Killing a stuck process on a port

```bash
# Find the PID using port 4200
lsof -i :4200
# COMMAND  PID   USER  ... NAME
# node     12345 user  ... *:4200

# Kill it
kill 12345

# If it does not die:
kill -9 12345

# One-liner to find and kill:
lsof -ti :4200 | xargs kill
```

## Shell History and Shortcuts

```bash
# Search command history
history                       # show all history
history | grep "git push"     # filter history

# Reverse search (most useful)
Ctrl+R                        # start typing to search backwards through history
# Press Ctrl+R again to cycle through older matches
# Press Enter to execute, or Right arrow to edit before executing

# Re-run the last command
!!
sudo !!            # re-run last command with sudo

# Re-run the last command that started with a pattern
!git               # last command starting with "git"

# Navigation shortcuts (Bash/Zsh)
Ctrl+A             # move cursor to beginning of line
Ctrl+E             # move cursor to end of line
Ctrl+W             # delete word before cursor
Ctrl+U             # delete everything before cursor
Ctrl+K             # delete everything after cursor
Ctrl+L             # clear the screen (same as `clear`)
Alt+B              # move cursor back one word
Alt+F              # move cursor forward one word
```

### Aliases

```bash
# Add to ~/.bashrc or ~/.zshrc

# Navigation
alias ..="cd .."
alias ...="cd ../.."

# Git shortcuts
alias gs="git status"

# npm shortcuts
alias ni="npm install"
alias nr="npm run"
alias nci="npm ci"

# Project shortcuts
alias serve="npx serve"
alias dev="npm run dev"
```

### Custom functions

```bash
# Create a directory and cd into it
mkcd() {
  mkdir -p "$1" && cd "$1"
}

# Find and replace in files (requires sed)
replace() {
  grep -rl "$1" . --include="*.ts" | xargs sed -i "s/$1/$2/g"
}

# Kill process on a port
killport() {
  lsof -ti :"$1" | xargs kill -9 2>/dev/null
  echo "Killed process on port $1"
}
```

## Common Mistakes

- **Running `npm install` in CI instead of `npm ci`.** `npm install` may update the lockfile and install slightly different versions than other team members. `npm ci` installs exactly what the lockfile specifies.

- **Installing CLI tools globally.** Global installs (`npm install -g`) are machine-specific and version-locked. Use `npx` to run CLI tools, or add them as devDependencies and run via npm scripts.

- **Not using a `.nvmrc` or `.node-version` file.** Without it, team members may use different Node versions, causing "works on my machine" bugs. Pin the Node version per project.

- **Deleting with `rm -rf` without checking the path.** A typo like `rm -rf / home/user` (space before `home`) deletes everything from root. Always double-check `rm -rf` paths. Use `trash-cli` for recoverable deletes during development.

- **Not using `.gitignore` for generated directories.** `node_modules/`, `dist/`, `.angular/`, and `coverage/` should never be committed. If they are, `git rm -r --cached <dir>` removes them from tracking.

- **Ignoring shell startup time.** nvm adds 2-3 seconds to every new terminal. Over a day of opening/closing terminals, this adds up. Switch to fnm for instant startup.

- **Not knowing `Ctrl+R`.** Developers who retype long commands from memory waste time. Reverse search (`Ctrl+R`) finds any previous command by substring.

---

## Best Practices

- **Use `npm ci` in all CI/CD pipelines and after cloning.** It is faster, deterministic, and catches lockfile drift.

- **Pin the Node version per project with `.nvmrc`.** Run `node --version > .nvmrc` in the project root and commit it.

- **Define all project commands as npm scripts.** A new developer should only need `npm install` and `npm run <script>` to perform any task. Do not require them to know tool-specific CLI commands.

- **Learn 5 essential shortcuts: `Ctrl+R`, `Ctrl+A`, `Ctrl+E`, `Ctrl+W`, `Ctrl+C`.** These cover history search, cursor movement, word deletion, and process interruption.

- **Create aliases for commands you run more than 5 times per day.** `gs` for `git status`, `nr` for `npm run`, etc.

- **Use `&&` between commands that depend on each other.** `npm run lint && npm run test && npm run build` stops at the first failure. Using `;` would continue even if lint fails.

- **Redirect long output to files.** `npm run build > build.log 2>&1` lets you search the output with `grep` instead of scrolling through the terminal.

## Real-World Use Cases

### Setting up a new project from scratch

```bash
# Create project directory
mkdir -p ~/projects/my-app && cd ~/projects/my-app

# Pin Node version
echo "22" > .nvmrc
fnm use

# Initialize npm
npm init -y

# Install dependencies
npm install typescript
npm install --save-dev eslint prettier jest @types/jest ts-jest

# Initialize TypeScript
npx tsc --init

# Initialize ESLint
npx eslint --init

# Create directory structure
mkdir -p src/{components,services,utils}
touch src/index.ts

# Initialize git
git init
echo "node_modules/\ndist/\ncoverage/" > .gitignore
git add -A && git commit -m "chore: initial project setup"
```

### Investigating a production build issue

```bash
# Check Node/npm versions match CI
node --version      # v22.12.0
npm --version       # 10.9.2

# Clean install (match CI behavior)
rm -rf node_modules
npm ci

# Run the production build and capture output
npm run build -- --configuration=production > build.log 2>&1

# Search for errors in the build log
grep -i "error" build.log
grep -n "warning" build.log | head -20

# Check bundle sizes
find dist/ -name "*.js" -exec du -h {} + | sort -rh | head -10

# Find the largest file in the build output
find dist/ -type f | xargs du -h | sort -rh | head -1
```

### Cleaning up after switching branches

```bash
# After switching to a branch with different dependencies
git switch feature/new-dependencies

# Check if package.json changed
git diff HEAD~1 -- package.json

# If it changed, clean install
rm -rf node_modules .angular
npm ci

# Verify nothing is broken
npm run build
npm test
```

> **Key Takeaway:** Master `Ctrl+R` for history search, `npm ci` for deterministic installs, fnm for fast Node version switching, and npm scripts as the single entry point for all project commands -- these four habits eliminate the majority of terminal friction in web development.
