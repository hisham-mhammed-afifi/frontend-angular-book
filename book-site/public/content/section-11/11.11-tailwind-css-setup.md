# 11.11 Tailwind CSS Setup

> Tailwind CSS is a utility-first CSS framework that generates only the classes you use. Angular has first-class support for Tailwind, and v4's CSS-first configuration makes integration simpler than ever.

---

## Installation

Angular CLI provides built-in Tailwind support. There are two installation paths.

### Automated Setup (Recommended)

```bash
ng add @tailwindcss/schematics
```

This installs the required packages, creates the PostCSS configuration, and updates your global stylesheet automatically.

### Manual Setup

If you prefer explicit control:

```bash
npm install tailwindcss @tailwindcss/postcss postcss
```

Create a PostCSS configuration file in the project root:

```json
// .postcssrc.json
{
  "plugins": {
    "@tailwindcss/postcss": {}
  }
}
```

Add the Tailwind import to your global stylesheet:

```css
/* src/styles.css */
@import "tailwindcss";
```

That is the entire setup. No `tailwind.config.js` file is needed in v4. Tailwind scans your project files automatically, detects which utility classes you use, and generates only those styles.

---

## Tailwind v4 vs v3: What Changed

Tailwind CSS v4 is a ground-up rewrite. If you have worked with v3 before, the key differences matter:

| Aspect | v3 | v4 |
|--------|----|----|
| Configuration | `tailwind.config.js` (JavaScript) | CSS-first (`@theme` directive) |
| Build engine | JavaScript (PostCSS plugin) | Rust-powered Oxide engine |
| Content scanning | Manual `content` array in config | Automatic detection via `.gitignore` |
| Directives | `@tailwind base/components/utilities` | Single `@import "tailwindcss"` |
| Custom theme | `theme.extend` in JS config | `@theme { }` block in CSS |
| Custom utilities | `addUtilities()` plugin in JS | `@utility` directive in CSS |
| Custom variants | `addVariant()` plugin in JS | `@custom-variant` directive in CSS |
| Build speed | Baseline | Up to 5x faster |

### v3 Setup (Legacy Reference)

If you encounter an older Angular project with Tailwind v3:

```javascript
// tailwind.config.js (v3 only)
/** @type {import('tailwindcss').Config} */
module.exports = {
  content: [
    "./src/**/*.{html,ts}"
  ],
  theme: {
    extend: {
      colors: {
        brand: '#3b82f6'
      }
    }
  },
  plugins: []
};
```

```css
/* src/styles.css (v3) */
@tailwind base;
@tailwind components;
@tailwind utilities;
```

In v4, all of this is handled with a single `@import "tailwindcss"` line and the `@theme` directive.

## Content Detection

Tailwind v4 automatically scans your project to find which utility classes are used. It follows these rules:

1. Starts from the directory containing the CSS file that imports Tailwind
2. Scans all files recursively
3. Ignores anything in `.gitignore` (like `node_modules`, `dist`)
4. Ignores binary file types (images, videos, archives)

### The @source Directive

If Tailwind misses files (e.g., a shared UI library outside your project root), use `@source`:

```css
/* src/styles.css */
@import "tailwindcss";

/* Scan a monorepo sibling package for Tailwind classes */
@source "../../libs/shared-ui/src";

/* Scan a specific node_modules package */
@source "../node_modules/@company/ui-kit/dist";
```

### Excluding Paths

To prevent scanning large directories that do not contain Tailwind classes:

```css
/* src/styles.css */
@import "tailwindcss";

@source not "../../legacy-app";
```

## Customizing the Theme with @theme

The `@theme` directive defines design tokens that Tailwind uses to generate utility classes. Every variable you add creates corresponding utilities.

### Custom Colors

```css
/* src/styles.css */
@import "tailwindcss";

@theme {
  --color-brand-50: #eff6ff;
  --color-brand-100: #dbeafe;
  --color-brand-200: #bfdbfe;
  --color-brand-300: #93c5fd;
  --color-brand-400: #60a5fa;
  --color-brand-500: #3b82f6;
  --color-brand-600: #2563eb;
  --color-brand-700: #1d4ed8;
  --color-brand-800: #1e40af;
  --color-brand-900: #1e3a8a;
  --color-brand-950: #172554;
}
```

These generate utilities like `bg-brand-500`, `text-brand-700`, `border-brand-200`, etc.

### Custom Fonts

```css
@theme {
  --font-display: "Cal Sans", "Inter", sans-serif;
  --font-body: "Inter", system-ui, sans-serif;
  --font-mono: "JetBrains Mono", monospace;
}
```

Use as `font-display`, `font-body`, `font-mono`.

### Custom Spacing

```css
@theme {
  --spacing-18: 4.5rem;
  --spacing-88: 22rem;
  --spacing-128: 32rem;
}
```

Use as `w-88`, `p-18`, `mt-128`, etc.

### Custom Breakpoints

```css
@theme {
  --breakpoint-xs: 475px;
  --breakpoint-3xl: 1920px;
}
```

Use as `xs:flex`, `3xl:grid-cols-4`.

### Custom Animations

```css
@theme {
  --animate-fade-in: fade-in 0.3s ease-out;
  --animate-slide-up: slide-up 0.4s ease-out;
}

@keyframes fade-in {
  from { opacity: 0; }
  to { opacity: 1; }
}

@keyframes slide-up {
  from { opacity: 0; transform: translateY(8px); }
  to { opacity: 1; transform: translateY(0); }
}
```

Use as `animate-fade-in`, `animate-slide-up`.

### Removing Default Theme Values

To replace Tailwind's default colors entirely instead of extending them:

```css
@theme {
  /* Clear all default colors */
  --color-*: initial;

  /* Define only your palette */
  --color-primary: #2563eb;
  --color-secondary: #7c3aed;
  --color-neutral-50: #fafafa;
  --color-neutral-900: #171717;
}
```

The `--color-*: initial` syntax clears all values in that namespace.

## Tailwind and Angular View Encapsulation

Angular's default `ViewEncapsulation.Emulated` scopes component styles to that component. Tailwind classes applied in templates work perfectly with this because they are applied via the `class` attribute, not via component stylesheets.

### Where Styles Live

| Style Location | Works with Tailwind? | Notes |
|---------------|---------------------|-------|
| Template `class="..."` | Yes | This is the primary way to use Tailwind |
| Global `styles.css` | Yes | For `@theme`, `@utility`, `@layer` |
| Component `.css` file | Limited | Encapsulated; `@apply` needs import |
| Component inline `styles` | Limited | Same encapsulation constraints |

### Using @apply in Component Styles

If you want to use `@apply` inside a component's scoped stylesheet, you must import the Tailwind theme into that file:

```css
/* card.component.css */
@import "tailwindcss/theme" layer(theme);
@import "tailwindcss/utilities" layer(utilities);

:host {
  @apply block rounded-lg border border-neutral-200;
}

.card-title {
  @apply text-lg font-semibold text-neutral-900;
}
```

However, this adds overhead because each component that uses `@apply` must process the Tailwind imports. In most cases, using utility classes directly in templates is the better approach.

### When to Use ViewEncapsulation.None

Some teams set `ViewEncapsulation.None` on components that heavily use Tailwind to avoid any encapsulation issues:

```typescript
// BAD PRACTICE: Do not set None just for Tailwind
@Component({
  encapsulation: ViewEncapsulation.None,  // Avoid this
  // ...
})
```

This is unnecessary. Tailwind classes in templates work with the default encapsulation. Setting `None` removes style isolation for that component's CSS, which can cause unintended side effects.

## Custom Utilities with @utility

Define custom utility classes that work with Tailwind's variant system (hover, focus, responsive, etc.):

```css
/* src/styles.css */
@import "tailwindcss";

@utility content-auto {
  content-visibility: auto;
}

@utility scrollbar-thin {
  scrollbar-width: thin;
  scrollbar-color: var(--color-neutral-400) transparent;
}

@utility text-balance {
  text-wrap: balance;
}
```

Usage in templates:

```html
<main class="content-auto">
  <h1 class="text-balance text-4xl font-bold">
    A very long heading that should wrap nicely across multiple lines
  </h1>
  <div class="scrollbar-thin overflow-y-auto h-96">
    <!-- scrollable content -->
  </div>
</main>
```

Custom utilities defined with `@utility` automatically support variants:

```html
<div class="hover:scrollbar-thin lg:content-auto">
```

## Custom Variants with @custom-variant

Create reusable state selectors:

```css
/* src/styles.css */
@import "tailwindcss";

/* Match elements inside an .error parent */
@custom-variant error (&:is(.error *));

/* Match Angular's disabled state on form controls */
@custom-variant field-disabled (&:is([formcontrol]:disabled, [ng-reflect-is-disabled="true"]));

/* Match elements inside a right-to-left context */
@custom-variant rtl (&:is([dir="rtl"] *));
```

Usage:

```html
<input class="border-neutral-300 error:border-red-500 error:ring-red-200
              field-disabled:bg-neutral-100 field-disabled:cursor-not-allowed" />

<span class="ml-2 rtl:mr-2 rtl:ml-0">
  Label text
</span>
```

## Dark Mode

Tailwind v4 supports dark mode via the `dark:` variant. By default, it uses the CSS `prefers-color-scheme` media query.

### Class-Based Dark Mode

For manual toggle control, configure class-based dark mode:

```css
/* src/styles.css */
@import "tailwindcss";

@custom-variant dark (&:is(.dark *));
```

This tells Tailwind to activate `dark:` utilities when a `.dark` class exists on an ancestor element.

### Theme Service with Signals

```typescript
// src/app/core/theme.service.ts
import { Injectable, signal, effect } from '@angular/core';

export type Theme = 'light' | 'dark' | 'system';

@Injectable({ providedIn: 'root' })
export class ThemeService {
  private readonly storageKey = 'app-theme';

  theme = signal<Theme>(this.loadTheme());

  constructor() {
    // Apply the theme whenever it changes
    effect(() => {
      const current = this.theme();
      localStorage.setItem(this.storageKey, current);
      this.applyTheme(current);
    });
  }

  toggle() {
    this.theme.update(t => t === 'dark' ? 'light' : 'dark');
  }

  private loadTheme(): Theme {
    const stored = localStorage.getItem(this.storageKey) as Theme | null;
    return stored ?? 'system';
  }

  private applyTheme(theme: Theme) {
    const root = document.documentElement;
    const isDark = theme === 'dark' ||
      (theme === 'system' && window.matchMedia('(prefers-color-scheme: dark)').matches);

    root.classList.toggle('dark', isDark);
  }
}
```

### Theme Toggle Component


## Extracting Reusable Patterns

When the same set of utility classes repeats across components, extract them instead of duplicating.

### Option 1: Angular Components (Preferred)

The best way to avoid repetition in Angular is to create reusable components:

```typescript
@Component({
  selector: 'app-btn',
  template: `
    <button
      [type]="type()"
      [disabled]="disabled()"
      [ngClass]="{
        'bg-brand-600 text-white hover:bg-brand-700 focus:ring-brand-500': variant() === 'primary',
        'bg-white border border-neutral-300 text-neutral-700 hover:bg-neutral-50 focus:ring-neutral-500': variant() === 'secondary',
        'bg-red-600 text-white hover:bg-red-700 focus:ring-red-500': variant() === 'danger',
        'opacity-50 cursor-not-allowed': disabled()
      }"
      class="inline-flex items-center justify-center rounded-md px-4 py-2 text-sm
             font-medium transition-colors focus:outline-none focus:ring-2 focus:ring-offset-2">
      <ng-content />
    </button>
  `
})
export class ButtonComponent {
  variant = input<'primary' | 'secondary' | 'danger'>('primary');
  type = input<'button' | 'submit' | 'reset'>('button');
  disabled = input(false);
}
```

```html
<!-- Usage -->
<app-btn variant="primary">Save</app-btn>
<app-btn variant="secondary" [disabled]="isLoading()">Cancel</app-btn>
<app-btn variant="danger">Delete</app-btn>
```

### Option 2: @apply in Global Stylesheet

For base styles shared across the entire app:

```css
/* src/styles.css */
@import "tailwindcss";

@layer components {
  .btn {
    @apply inline-flex items-center justify-center rounded-md px-4 py-2
           text-sm font-medium transition-colors focus:outline-none
           focus:ring-2 focus:ring-offset-2;
  }

  .btn-primary {
    @apply bg-brand-600 text-white hover:bg-brand-700 focus:ring-brand-500;
  }

  .btn-secondary {
    @apply bg-white border border-neutral-300 text-neutral-700
           hover:bg-neutral-50 focus:ring-neutral-500;
  }

  .input-field {
    @apply block w-full rounded-md border border-neutral-300 px-3 py-2
           text-sm placeholder-neutral-400 focus:border-brand-500
           focus:outline-none focus:ring-1 focus:ring-brand-500
           dark:border-neutral-600 dark:bg-neutral-800 dark:text-white;
  }
}
```

Use `@apply` sparingly. It exists for cases where a CSS class is genuinely simpler than a component (e.g., base form field styles used across many native elements).

## Tailwind with SCSS

Angular projects may use SCSS for global styles. Tailwind v4 and SCSS have a compatibility constraint: you must use `@import` (not `@use`) for the Tailwind import because Tailwind is not a Sass module.

```scss
/* src/styles.scss */

/* Tailwind must use @import, not @use */
@import "tailwindcss";

/* Your SCSS partials can use @use normally */
@use 'variables' as vars;
@use 'mixins' as mix;

@theme {
  --color-brand-500: #{vars.$brand-primary};
  --color-brand-700: #{vars.$brand-dark};
}

/* SCSS features like nesting and mixins work alongside Tailwind */
.legacy-widget {
  @include mix.card-shadow;

  &__header {
    @apply text-lg font-semibold;
  }
}
```

### Caveats

- Sass processes `@import "tailwindcss"` before PostCSS. The Tailwind PostCSS plugin handles the actual class generation, so the order works correctly.
- Do not use `@use "tailwindcss"`. Tailwind is not a Sass module and will fail.
- Sass nesting and Tailwind's `@apply` can coexist, but prefer Tailwind utilities in templates over `@apply` in stylesheets.

## Integrating with Component Libraries

When using a component library (Angular Material, PrimeNG, etc.) alongside Tailwind, be aware of style conflicts.

### Preflight (Base Reset) Conflicts

Tailwind v4 applies Preflight (a CSS reset) by default. This can strip styles from third-party components. To disable it:

```css
/* src/styles.css */
@import "tailwindcss" layer(tailwind);

/* Override the Preflight base layer selectively */
@layer tailwind.base {
  /* Restore button appearance for third-party components */
  .mat-mdc-button,
  .p-button {
    all: revert;
  }
}
```

Or use Tailwind's `@layer` to scope your customizations:

```css
@import "tailwindcss";

/* Ensure third-party component styles take precedence */
@layer base {
  [class^="mat-"],
  [class^="p-"] {
    /* Let Material/PrimeNG styles win */
  }
}
```

### Specificity Management

If Tailwind utilities are being overridden by component library styles, use the `!` modifier (important):

```html
<!-- Force Tailwind to win over third-party styles -->
<button class="!bg-brand-600 !text-white">Override</button>
```

Use this sparingly. A better approach is to style wrappers and layout with Tailwind, and let the component library handle its own component internals.

## Build Output and Purging

Tailwind v4 automatically removes unused classes from the production build. No manual purge configuration is needed.

### Verifying Bundle Size

Check the generated CSS size:

```bash
ng build --configuration production

# Check the CSS output
ls -la dist/my-app/browser/*.css
```

A well-configured Tailwind project typically produces 10-30 KB of CSS (gzipped), regardless of how many utility classes the framework offers.

### Dynamic Class Names

Tailwind scans source code as plain text. It cannot detect dynamically constructed class names:

```typescript
// BAD: Tailwind cannot detect this
const color = 'red';
const className = `bg-${color}-500`;  // Not found by scanner

// GOOD: Use complete class names
const className = isError() ? 'bg-red-500' : 'bg-green-500';

// GOOD: Map values to complete classes
const colorMap: Record<string, string> = {
  success: 'bg-green-500 text-green-900',
  error: 'bg-red-500 text-red-900',
  warning: 'bg-yellow-500 text-yellow-900'
};
const className = colorMap[status()];
```

### Safelist

If you must use dynamic class names, add them to the safelist so Tailwind always includes them:

```css
/* src/styles.css */
@import "tailwindcss";

@source inline("bg-red-500 bg-green-500 bg-yellow-500 bg-blue-500
                text-red-900 text-green-900 text-yellow-900 text-blue-900");
```

The `@source inline()` directive tells Tailwind to scan the given string for class names, ensuring they are always generated.

## Common Mistakes

- **Constructing class names dynamically with string interpolation.** Tailwind scans source files as static text. It cannot evaluate expressions like `` `bg-${color}-500` ``. Always use complete, literal class names so the scanner finds them.

- **Using `@use "tailwindcss"` in SCSS files.** Tailwind is not a Sass module. The `@use` directive fails silently or produces errors. Always use `@import "tailwindcss"` even in SCSS projects.

- **Adding `@apply` in component-scoped stylesheets without importing the theme.** Angular's view encapsulation isolates component styles. The `@apply` directive cannot resolve Tailwind utilities unless you import `tailwindcss/theme` and `tailwindcss/utilities` into that component's stylesheet.

- **Setting `ViewEncapsulation.None` to "fix" Tailwind.** Tailwind classes applied via the `class` attribute in templates work with the default emulated encapsulation. Disabling encapsulation to solve a perceived Tailwind issue usually indicates a different underlying problem (missing global import, misconfigured PostCSS).

- **Overusing `@apply` to recreate components in CSS.** The `@apply` directive should be used sparingly for base-level reusable styles (form inputs, prose text). For component-level patterns (cards, buttons, modals), create Angular components instead. This gives you logic, inputs, outputs, and type safety.

- **Forgetting to configure `@source` for monorepo libraries.** In a monorepo, Tailwind scans from the CSS file's directory. If shared libraries live in a sibling folder, their Tailwind classes will be missing from the generated CSS unless you add `@source` paths.

---

## Best Practices

- **Use utility classes directly in templates as the primary approach.** This is the intended workflow. Angular components + Tailwind utility classes = reusable, type-safe UI with no CSS file maintenance.

- **Define your design system in `@theme`.** Establish brand colors, fonts, spacing scales, and breakpoints in a single `@theme` block. This centralizes design decisions and produces consistent utility classes project-wide.

- **Create Angular components for reusable patterns, not CSS classes.** A `<app-btn>` component with a `variant` input is more powerful, testable, and maintainable than a `.btn-primary` CSS class.

- **Keep the global stylesheet focused on Tailwind config and base styles.** The `styles.css` file should contain the Tailwind import, `@theme` customizations, `@utility` definitions, `@custom-variant` definitions, and minimal `@layer components` extractions. Avoid putting component-specific styles here.

- **Use the `@source inline()` safelist only when necessary.** If an API returns a color string that maps to a Tailwind class, safelist those specific classes. Do not safelist entire color palettes "just in case."

- **Integrate Tailwind's IntelliSense extension.** Install the "Tailwind CSS IntelliSense" VS Code extension. It provides autocomplete for classes in Angular templates, linting for invalid classes, and hover previews of the generated CSS.
