# 11.14 Migrating from NgModules to Standalone

> Angular provides an automated three-step schematic to convert NgModule-based projects to standalone components. Understanding the migration process, its limitations, and the manual cleanup required afterward is essential for modernizing existing codebases.

---

## Prerequisites

Before starting migration, verify:

1. **Angular version 15.2.0 or later.** The standalone migration schematic was introduced in 15.2. If your project is on an older version, update first with `ng update`.

2. **The project compiles without errors.** The schematic uses Angular's static analysis. If the project has TypeScript or template compilation errors, the schematic cannot analyze the code and will either fail or produce incorrect results.

3. **Clean Git state.** Commit all pending changes before running the schematic. You need to be able to revert if something goes wrong.

4. **Tests pass.** Run your test suite before migrating so you have a known-good baseline. After each step, run tests again to catch regressions.

```bash
# Verify before starting
ng build
ng test
git status  # Should be clean
```

---

## The Three-Step Migration

The official schematic runs in three sequential modes. Run them in order, verifying the build after each step.

### Overview

```
Step 1: Convert declarations to standalone
         |
         v
    Verify build and tests
         |
         v
Step 2: Remove unnecessary NgModules
         |
         v
    Verify build and tests
         |
         v
Step 3: Switch to standalone bootstrap
         |
         v
    Verify build, tests, and manual cleanup
```

### Running the Schematic

```bash
ng generate @angular/core:standalone
```

The CLI presents a menu with three options:
1. Convert all components, directives and pipes to standalone
2. Remove unnecessary NgModule classes
3. Bootstrap the project using standalone APIs

You can also specify the mode directly:

```bash
ng generate @angular/core:standalone --mode=convert-to-standalone
ng generate @angular/core:standalone --mode=prune-ng-modules
ng generate @angular/core:standalone --mode=standalone-bootstrap
```

## Step 1: Convert Declarations to Standalone

### What the Schematic Does

For every component, directive, and pipe in the project, the schematic:

1. Removes `standalone: false` (making it standalone, since `true` is the default)
2. Analyzes the template to determine which dependencies are used
3. Adds an `imports` array with those dependencies

### Before

```typescript
// user-list.component.ts (before)
@Component({
  selector: 'app-user-list',
  templateUrl: './user-list.component.html',
  styleUrl: './user-list.component.scss'
})
export class UserListComponent {
  users = signal<User[]>([]);
}
```

```typescript
// users.module.ts (before)
@NgModule({
  declarations: [
    UserListComponent,
    UserDetailComponent,
    UserCardComponent
  ],
  imports: [
    CommonModule,
    SharedModule,
    RouterModule.forChild(userRoutes)
  ],
  exports: [UserCardComponent]
})
export class UsersModule {}
```

### After Step 1

```typescript
// user-list.component.ts (after step 1)
@Component({
  selector: 'app-user-list',
  imports: [NgFor, NgIf, UserCardComponent, RouterLink],
  templateUrl: './user-list.component.html',
  styleUrl: './user-list.component.scss'
})
export class UserListComponent {
  users = signal<User[]>([]);
}
```

The module still exists but the component no longer depends on it for its template compilation scope.

### What to Check After Step 1

```bash
ng build
ng test
```

Common issues at this stage:

- **Missing imports.** The schematic may miss a dependency used in the template. If the build fails with "X is not a known element", add the missing import to the component's `imports` array.

- **Excessive CommonModule imports.** The schematic may add `CommonModule` to a component that only uses `NgIf`. This works but is not optimal. You can replace `CommonModule` with just `NgIf` later, or run the separate `CommonModule` migration schematic.

## Step 2: Remove Unnecessary NgModules

### What the Schematic Does

The schematic identifies NgModules that are no longer needed because all their declarations are now standalone. A module is removable if:

- It has no `providers` (or providers that are already available elsewhere)
- It has no `bootstrap` array
- Its only purpose was to group declarations

The schematic:
1. Deletes the module file
2. Updates any files that imported the deleted module
3. Replaces module references in route configurations

### Modules the Schematic Cannot Remove

Some modules require manual intervention:

| Module Type | Why It Stays | Manual Fix |
|-------------|-------------|------------|
| Module with `providers` | Services need to be moved | Move providers to route-level `providers` or `app.config.ts` |
| Module with `forRoot`/`forChild` imports | Complex provider configuration | Use `importProvidersFrom` |
| Module referenced in `loadChildren` | Lazy-loaded routing module | Convert to route file with `loadChildren` pointing to routes array |
| Module with custom constructor logic | Guard against double imports, etc. | Remove the guard; it is unnecessary in standalone |

### TODO Comments

When the schematic cannot fully clean up a reference, it leaves a comment:

```typescript
// TODO(standalone-migration): clean up removed NgModule reference manually
```

Search your codebase for these after running step 2:

```bash
grep -r "TODO(standalone-migration)" src/
```

### Manual Module Removal Example

A routing module that the schematic could not remove:

```typescript
// Before: users-routing.module.ts
@NgModule({
  imports: [RouterModule.forChild(userRoutes)],
  exports: [RouterModule]
})
export class UsersRoutingModule {}
```

This module only wraps `RouterModule.forChild`. Remove it and update the route configuration:

```typescript
// After: users.routes.ts (new file)
import { Routes } from '@angular/router';

export const userRoutes: Routes = [
  {
    path: '',
    loadComponent: () => import('./user-list.component')
      .then(m => m.UserListComponent)
  },
  {
    path: ':id',
    loadComponent: () => import('./user-detail.component')
      .then(m => m.UserDetailComponent)
  }
];
```

Update the parent route:

```typescript
// app.routes.ts
const routes: Routes = [
  {
    path: 'users',
    loadChildren: () => import('./features/users/users.routes')
      .then(m => m.userRoutes)
  }
];
```

## Step 3: Switch to Standalone Bootstrap

### What the Schematic Does

1. Converts `main.ts` from `bootstrapModule(AppModule)` to `bootstrapApplication(AppComponent, appConfig)`
2. Makes `AppComponent` standalone
3. Extracts providers and imports from `AppModule` into an `ApplicationConfig`
4. Uses `importProvidersFrom` for modules that have providers
5. Deletes `AppModule`

### Before

```typescript
// main.ts (before)
import { platformBrowserDynamic } from '@angular/platform-browser-dynamic';
import { AppModule } from './app/app.module';

platformBrowserDynamic().bootstrapModule(AppModule)
  .catch(err => console.error(err));
```

```typescript
// app.module.ts (before)
@NgModule({
  declarations: [AppComponent],
  imports: [
    BrowserModule,
    BrowserAnimationsModule,
    HttpClientModule,
    RouterModule.forRoot(routes),
    TranslateModule.forRoot({ defaultLanguage: 'en' })
  ],
  providers: [
    { provide: ErrorHandler, useClass: GlobalErrorHandler },
    { provide: HTTP_INTERCEPTORS, useClass: AuthInterceptor, multi: true }
  ],
  bootstrap: [AppComponent]
})
export class AppModule {}
```

### After Step 3

```typescript
// main.ts (after)
import { bootstrapApplication } from '@angular/platform-browser';
import { AppComponent } from './app/app.component';
import { appConfig } from './app/app.config';

bootstrapApplication(AppComponent, appConfig)
  .catch(err => console.error(err));
```

```typescript
// app.config.ts (generated)
import { ApplicationConfig, importProvidersFrom } from '@angular/core';
import { provideRouter } from '@angular/router';
import { provideHttpClient } from '@angular/common/http';
import { provideAnimations } from '@angular/platform-browser/animations';
import { TranslateModule } from '@ngx-translate/core';
import { routes } from './app.routes';
import { GlobalErrorHandler } from './core/global-error-handler';
import { ErrorHandler } from '@angular/core';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
    provideHttpClient(),
    provideAnimations(),
    importProvidersFrom(
      TranslateModule.forRoot({ defaultLanguage: 'en' })
    ),
    { provide: ErrorHandler, useClass: GlobalErrorHandler }
  ]
};
```

### Manual Cleanup After Step 3

The schematic converts `RouterModule.forRoot` to `provideRouter` and `HttpClientModule` to `provideHttpClient` when possible. But some conversions need manual refinement:

**HTTP Interceptors**

The schematic may use `importProvidersFrom(HttpClientModule)`. Manually convert to the functional interceptor pattern:

```typescript
// Before (class-based interceptor from NgModule era)
{ provide: HTTP_INTERCEPTORS, useClass: AuthInterceptor, multi: true }

// After (functional interceptor)
provideHttpClient(
  withInterceptors([authInterceptor])
)
```

Where `authInterceptor` is the functional equivalent:

```typescript
// auth.interceptor.ts
import { HttpInterceptorFn } from '@angular/common/http';
import { inject } from '@angular/core';

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const authService = inject(AuthService);
  const token = authService.token();

  if (token) {
    req = req.clone({
      setHeaders: { Authorization: `Bearer ${token}` }
    });
  }

  return next(req);
};
```

**Router Configuration Features**

If `RouterModule.forRoot` had options, map them to `provideRouter` features:

```typescript
// Before
RouterModule.forRoot(routes, {
  bindToComponentInputs: true,
  scrollPositionRestoration: 'enabled',
  preloadingStrategy: PreloadAllModules
})

// After
provideRouter(
  routes,
  withComponentInputBinding(),
  withInMemoryScrolling({ scrollPositionRestoration: 'enabled' }),
  withPreloading(PreloadAllModules)
)
```

## Migrating Specific Patterns

### SharedModule

The `SharedModule` pattern (a module that re-exports common components) dissolves naturally:

**Before:**

```typescript
@NgModule({
  declarations: [SpinnerComponent, ConfirmDialogComponent, TruncatePipe],
  imports: [CommonModule, FormsModule, ReactiveFormsModule],
  exports: [
    CommonModule, FormsModule, ReactiveFormsModule,
    SpinnerComponent, ConfirmDialogComponent, TruncatePipe
  ]
})
export class SharedModule {}
```

**After:** Delete `SharedModule`. Each component that used it now imports exactly what it needs:

```typescript
@Component({
  imports: [SpinnerComponent, ReactiveFormsModule],
  template: `
    @if (isLoading()) {
      <app-spinner />
    }
    <form [formGroup]="form">...</form>
  `
})
export class CheckoutComponent {}
```

No more importing 15 components when you only use 2.

### CoreModule

The `CoreModule` pattern (singleton services, app-shell components) also dissolves:

**Services:** Already use `providedIn: 'root'`. No change needed.

**App-shell components** (header, footer, sidebar): Convert to standalone and import them directly in `AppComponent`:

```typescript
@Component({
  selector: 'app-root',
  imports: [RouterOutlet, HeaderComponent, FooterComponent, SidebarComponent],
  template: `
    <app-header />
    <div class="layout">
      <app-sidebar />
      <main>
        <router-outlet />
      </main>
    </div>
    <app-footer />
  `
})
export class AppComponent {}
```

### Feature Modules with Providers

**Before:**

```typescript
@NgModule({
  declarations: [OrderListComponent, OrderDetailComponent],
  imports: [CommonModule, SharedModule, RouterModule.forChild(orderRoutes)],
  providers: [OrderService, OrderResolver]
})
export class OrdersModule {}
```

**After:** A route file with route-level providers:

```typescript
// orders.routes.ts
export const orderRoutes: Routes = [
  {
    path: '',
    providers: [OrderService, OrderResolver],
    children: [
      {
        path: '',
        loadComponent: () => import('./order-list.component')
          .then(m => m.OrderListComponent)
      },
      {
        path: ':id',
        loadComponent: () => import('./order-detail.component')
          .then(m => m.OrderDetailComponent),
        resolve: { order: OrderResolver }
      }
    ]
  }
];
```

### Routing Modules

**Before:**

```typescript
@NgModule({
  imports: [RouterModule.forChild(routes)],
  exports: [RouterModule]
})
export class OrdersRoutingModule {}
```

**After:** Delete the routing module entirely. Export a plain routes array:

```typescript
// orders.routes.ts
export const orderRoutes: Routes = [/* ... */];
```

## Incremental Migration with Path Targeting

For large codebases, migrate one feature at a time using the `--path` option:

```bash
# Migrate only the users feature
ng generate @angular/core:standalone --mode=convert-to-standalone --path=src/app/features/users

# Verify
ng build
ng test

# Migrate the orders feature
ng generate @angular/core:standalone --mode=convert-to-standalone --path=src/app/features/orders
```

The targeted folder must contain an NgModule for the schematic to work. After converting all features, run steps 2 and 3 on the entire project.

### Recommended Order

1. **Leaf features first.** Start with feature modules that are not imported by other modules.
2. **Shared module last (or skip it).** `SharedModule` is often imported everywhere. Converting it last minimizes cascading changes.
3. **App module last.** The bootstrap conversion should be the final step.

## The CommonModule Migration

After standalone migration, components may import `CommonModule` when they only need specific directives. Angular provides a separate schematic:

```bash
ng generate @angular/core:standalone --mode=common-module
```

Or from Angular 19+:

```bash
ng generate @angular/migrations:common-to-standalone
```

### What It Does

Replaces `CommonModule` with individual imports:

```typescript
// Before
@Component({
  imports: [CommonModule],
  template: `
    @if (show()) { <p>Hello</p> }
    @for (item of items(); track item.id) { <span>{{ item.name }}</span> }
  `
})
export class MyComponent {}

// After
@Component({
  imports: [],  // No imports needed: @if/@for are built-in, no directives required
  template: `
    @if (show()) { <p>Hello</p> }
    @for (item of items(); track item.id) { <span>{{ item.name }}</span> }
  `
})
export class MyComponent {}
```

With Angular's new control flow (`@if`, `@for`, `@switch`), many components no longer need any imports from `CommonModule`. Components using `NgIf`, `NgFor`, `NgSwitch` can be migrated to the new control flow syntax first, then `CommonModule` can be removed.

Angular also provides a control flow migration:

```bash
ng generate @angular/core:control-flow
```

## Handling Tests After Migration

### Unit Tests

The schematic does not perfectly update test files because tests are not AoT-compiled. After migration, you may see test failures from missing imports.

**Before (module-based test):**

```typescript
describe('UserListComponent', () => {
  beforeEach(async () => {
    await TestBed.configureTestingModule({
      declarations: [UserListComponent],
      imports: [SharedModule, HttpClientTestingModule]
    }).compileComponents();
  });
});
```

**After (standalone test):**

```typescript
describe('UserListComponent', () => {
  beforeEach(async () => {
    await TestBed.configureTestingModule({
      imports: [UserListComponent],  // The component itself, not a module
      providers: [
        provideHttpClient(),
        provideHttpClientTesting()
      ]
    }).compileComponents();
  });
});
```

Key changes:
- The component moves from `declarations` to `imports`
- `HttpClientTestingModule` becomes `provideHttpClient()` + `provideHttpClientTesting()`
- `SharedModule` is removed; the component imports its own dependencies

### E2E Tests

E2E tests typically do not need changes since they test the running application, not the module structure.

## Post-Migration Cleanup Checklist

After completing all three steps, run through this checklist:

| Task | Command / Action |
|------|-----------------|
| Search for TODO comments | `grep -r "TODO(standalone-migration)" src/` |
| Remove empty NgModule files | Search for files containing only an empty `@NgModule` |
| Replace `CommonModule` with specific imports | Run the CommonModule migration schematic |
| Convert class interceptors to functional | Manual refactor |
| Convert class guards/resolvers to functional | Manual refactor |
| Replace `RouterModule.forChild` with route arrays | Manual refactor |
| Replace `importProvidersFrom` where `provide*` functions exist | Manual refactor |
| Run linting | `ng lint --fix` |
| Run formatting | `npx prettier --write src/` |
| Run full test suite | `ng test` |
| Run production build | `ng build` |
| Manual smoke test | Verify critical user flows |

## Common Mistakes

- **Running the migration steps out of order.** The three steps must be run in sequence: convert, prune, bootstrap. Running prune before convert leaves declarations in non-standalone components that cannot compile without their module.

- **Not verifying the build between steps.** Each step can introduce issues. If you run all three without checking, you end up with many errors stacked together, making debugging harder.

- **Assuming the schematic handles everything.** The schematic is a best-effort tool. It cannot handle dynamic module imports, custom decorators, or code that does not follow standard Angular patterns. Plan for manual cleanup.

- **Leaving `CommonModule` imports after migrating to new control flow.** If your templates use `@if` and `@for` (not `*ngIf` and `*ngFor`), you do not need `CommonModule` at all. The schematic may add it unnecessarily.

- **Not updating test files.** The schematic focuses on application code. Test files using `TestBed.configureTestingModule` with `declarations` and module imports will break. Update them to use `imports: [TheComponent]` and standalone providers.

- **Trying to migrate a project with compilation errors.** The schematic uses the Angular compiler for static analysis. If the project does not compile, the schematic cannot determine which imports a component needs and will produce incorrect or incomplete results.

---

## Best Practices

- **Commit after each step.** Create a git commit after each of the three steps. This gives you a clean rollback point if a subsequent step causes problems.

- **Use `--path` for large projects.** Migrate one feature folder at a time. This keeps each change set small and reviewable.

- **Migrate to new control flow before or during the standalone migration.** Run `ng generate @angular/core:control-flow` to convert `*ngIf` to `@if`, `*ngFor` to `@for`, etc. This eliminates the need for `CommonModule` imports entirely.

- **Track migration progress with a checklist.** For a project with 20+ modules, create a tracking document listing every module and its migration status.

- **Pair the migration with dependency updates.** If you are also updating Angular versions, do the version update first (with `ng update`), verify everything works, then run the standalone migration. Do not combine both changes.

- **Review diffs carefully.** The schematic modifies many files at once. Review the changes to catch over-aggressive imports (importing an entire module when only one component is needed) or missing imports.

## Real-World Use Cases

### 1. Enterprise Application: Phased Migration Over Sprints

A team with a 400-component enterprise application plans migration across multiple sprints:

**Sprint 1:** Migrate 3 leaf feature modules using `--path`:

```bash
ng generate @angular/core:standalone --mode=convert-to-standalone --path=src/app/features/settings
ng generate @angular/core:standalone --mode=convert-to-standalone --path=src/app/features/notifications
ng generate @angular/core:standalone --mode=convert-to-standalone --path=src/app/features/help
ng build && ng test
git commit -m "feat: convert settings, notifications, help to standalone"
```

**Sprint 2:** Migrate core features and shared module:

```bash
ng generate @angular/core:standalone --mode=convert-to-standalone --path=src/app/features/dashboard
ng generate @angular/core:standalone --mode=convert-to-standalone --path=src/app/features/users
ng generate @angular/core:standalone --mode=convert-to-standalone --path=src/app/shared
ng build && ng test
git commit -m "feat: convert dashboard, users, shared to standalone"
```

**Sprint 3:** Prune modules and switch bootstrap:

```bash
ng generate @angular/core:standalone --mode=prune-ng-modules
ng build && ng test
git commit -m "refactor: remove unnecessary NgModules"

ng generate @angular/core:standalone --mode=standalone-bootstrap
ng build && ng test
git commit -m "refactor: switch to standalone bootstrap"
```

**Sprint 4:** Manual cleanup: remove CommonModule, convert interceptors, update tests.

### 2. Library Migration: Keeping NgModule Compatibility

A shared component library needs to support both NgModule-based and standalone consumers:

```typescript
// The component is standalone
@Component({
  selector: 'lib-datepicker',
  imports: [FormsModule],
  template: `...`
})
export class DatepickerComponent {}

// The NgModule still exists for backward compatibility
@NgModule({
  imports: [DatepickerComponent],
  exports: [DatepickerComponent]
})
export class DatepickerModule {}
```

NgModule consumers import `DatepickerModule`. Standalone consumers import `DatepickerComponent` directly. Both work. Over time, deprecate the module and remove it in a major version.

### 3. Converting a Feature Module with Route Providers

A payments feature with scoped services:

**Before:**

```typescript
// payments.module.ts
@NgModule({
  declarations: [
    PaymentListComponent,
    PaymentFormComponent,
    PaymentReceiptComponent
  ],
  imports: [
    CommonModule,
    ReactiveFormsModule,
    SharedModule,
    RouterModule.forChild([
      { path: '', component: PaymentListComponent },
      { path: 'new', component: PaymentFormComponent },
      { path: ':id/receipt', component: PaymentReceiptComponent }
    ])
  ],
  providers: [
    PaymentService,
    PaymentGatewayService,
    { provide: PAYMENT_API_URL, useValue: '/api/payments' }
  ]
})
export class PaymentsModule {}
```

**After:**

```typescript
// payments.routes.ts
import { Routes } from '@angular/router';
import { PaymentService } from './payment.service';
import { PaymentGatewayService } from './payment-gateway.service';
import { PAYMENT_API_URL } from './payment.tokens';

export const paymentRoutes: Routes = [
  {
    path: '',
    providers: [
      PaymentService,
      PaymentGatewayService,
      { provide: PAYMENT_API_URL, useValue: '/api/payments' }
    ],
    children: [
      {
        path: '',
        loadComponent: () => import('./payment-list.component')
          .then(m => m.PaymentListComponent)
      },
      {
        path: 'new',
        loadComponent: () => import('./payment-form.component')
          .then(m => m.PaymentFormComponent)
      },
      {
        path: ':id/receipt',
        loadComponent: () => import('./payment-receipt.component')
          .then(m => m.PaymentReceiptComponent)
      }
    ]
  }
];
```

The module file is deleted. The route file replaces it entirely. Services scoped to this route tree behave exactly like they did in the lazy-loaded module.

---

> **Key Takeaway:** Migrate from NgModules to standalone using the official three-step schematic (convert, prune, bootstrap), run it incrementally with `--path` for large projects, verify the build after each step, and plan for manual cleanup of providers, interceptors, route modules, and test files.
