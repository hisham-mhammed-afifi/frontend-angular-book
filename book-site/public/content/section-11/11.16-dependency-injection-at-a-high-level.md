# 11.16 Dependency Injection at a High Level

> Dependency injection (DI) is the mechanism Angular uses to supply classes with the things they need. Understanding how injectors resolve providers, where to register them, and how to use `inject()` is foundational to writing testable, modular Angular applications.

---

## What Is Dependency Injection?

Dependency injection is a design pattern where a class receives its dependencies from an external source rather than creating them itself. Angular has a built-in DI framework that manages the creation and delivery of dependencies throughout the application.

Without DI:

```typescript
// BAD: Tight coupling. Hard to test. Hard to swap implementations.
class OrderService {
  private http = new HttpClient(); // Creates its own dependency
  private logger = new LoggerService(); // Creates its own dependency

  getOrders() {
    this.logger.log('Fetching orders');
    return this.http.get('/api/orders');
  }
}
```

With DI:

```typescript
// GOOD: Dependencies are provided externally.
@Injectable({ providedIn: 'root' })
class OrderService {
  private http = inject(HttpClient);
  private logger = inject(LoggerService);

  getOrders() {
    this.logger.log('Fetching orders');
    return this.http.get('/api/orders');
  }
}
```

The DI framework handles creating `HttpClient` and `LoggerService`, finding the right instances, and delivering them. This makes classes testable (swap in mocks), loosely coupled (no hard-coded dependencies), and configurable (swap implementations per environment).

## The inject() Function

The `inject()` function is the modern, preferred way to request dependencies in Angular. It was introduced in Angular 14 and has largely replaced constructor injection.

```typescript
import { inject } from '@angular/core';

@Injectable({ providedIn: 'root' })
export class ProductService {
  private http = inject(HttpClient);
  private config = inject(APP_CONFIG);

  getProducts() {
    return this.http.get<Product[]>(`${this.config.apiUrl}/products`);
  }
}
```

### inject() vs Constructor Injection

Constructor injection still works and is valid, but `inject()` has advantages:

```typescript
// Constructor injection (legacy style, still works)
@Injectable({ providedIn: 'root' })
export class ProductService {
  constructor(
    private http: HttpClient,
    @Inject(APP_CONFIG) private config: AppConfig
  ) {}
}

// inject() function (modern style)
@Injectable({ providedIn: 'root' })
export class ProductService {
  private http = inject(HttpClient);
  private config = inject(APP_CONFIG);
}
```

| Aspect | `inject()` | Constructor Injection |
|--------|------------|----------------------|
| Type inference | Automatic from token | Manual annotation required |
| InjectionToken usage | No `@Inject()` decorator needed | Requires `@Inject()` decorator |
| Works in functions | Yes (guards, resolvers, factories) | No |
| Inheritance | No `super()` forwarding needed | Must pass deps through `super()` |
| Optional deps | `inject(Token, { optional: true })` | `@Optional()` decorator |
| Self/SkipSelf | `inject(Token, { self: true })` | `@Self()` decorator |
| Testability | Same | Same |

### Where inject() Works

`inject()` must be called within an **injection context**. These are the valid contexts:

```typescript
// 1. Field initializers in @Injectable, @Component, @Directive, @Pipe
@Component({ ... })
export class MyComponent {
  private service = inject(MyService); // Valid
}

// 2. Constructor body
@Component({ ... })
export class MyComponent {
  constructor() {
    const service = inject(MyService); // Valid
  }
}

// 3. Factory functions in providers
providers: [
  {
    provide: API_URL,
    useFactory: () => {
      const config = inject(APP_CONFIG); // Valid
      return config.apiUrl;
    },
  },
]

// 4. Functional route guards and resolvers
export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService); // Valid
  return auth.isAuthenticated();
};

// 5. InjectionToken factory
const LOGGER = new InjectionToken<Logger>('Logger', {
  providedIn: 'root',
  factory: () => {
    const env = inject(ENVIRONMENT); // Valid
    return env.production ? new SilentLogger() : new ConsoleLogger();
  },
});

// 6. runInInjectionContext()
runInInjectionContext(injector, () => {
  const service = inject(MyService); // Valid
});
```

Where `inject()` does NOT work:

```typescript
// BAD: Regular function outside injection context
function doSomething() {
  const service = inject(MyService); // ERROR: NG0203
}

// BAD: Lifecycle hooks (except constructor)
ngOnInit() {
  const service = inject(MyService); // ERROR: NG0203
}

// BAD: Event handlers
onClick() {
  const service = inject(MyService); // ERROR: NG0203
}
```

### inject() Options

```typescript
// Required (default): throws if not found
const service = inject(MyService);

// Optional: returns null if not found
const service = inject(MyService, { optional: true });
// Type: MyService | null

// Self: only check this component's injector, not parents
const service = inject(MyService, { self: true });

// SkipSelf: skip this component's injector, start from parent
const service = inject(MyService, { skipSelf: true });

// Host: stop searching at the host component boundary
const service = inject(MyService, { host: true });

// Combine flags
const service = inject(MyService, { optional: true, skipSelf: true });
```

## Providers and Registration

A **provider** tells Angular's DI system how to obtain a value for a given token. The token is what you inject; the provider is the recipe for creating it.

### Provider Types

#### Class Provider (useClass)

Creates a new instance of the specified class.

```typescript
providers: [
  // Shorthand: token and class are the same
  MyService,

  // Longhand equivalent
  { provide: MyService, useClass: MyService },

  // Swap implementation
  { provide: LoggerService, useClass: ProductionLoggerService },
]
```

#### Value Provider (useValue)

Provides a static value. No class instantiation.

```typescript
const APP_CONFIG: AppConfig = {
  apiUrl: 'https://api.example.com',
  pageSize: 20,
};

providers: [
  { provide: APP_CONFIG_TOKEN, useValue: APP_CONFIG },
  { provide: IS_PRODUCTION, useValue: true },
]
```

#### Factory Provider (useFactory)

Calls a function to produce the value. Can depend on other injected values.

```typescript
providers: [
  {
    provide: API_CLIENT,
    useFactory: () => {
      const config = inject(APP_CONFIG_TOKEN);
      const http = inject(HttpClient);
      return new ApiClient(http, config.apiUrl, config.timeout);
    },
  },
]
```

#### Existing Provider (useExisting)

Creates an alias to an existing provider. Both tokens resolve to the same instance.

```typescript
providers: [
  FullLoggerService,
  // MinimalLogger is an alias for the same FullLoggerService instance
  { provide: MinimalLogger, useExisting: FullLoggerService },
]
```

The difference between `useClass` and `useExisting` matters:

```typescript
// useClass: TWO separate instances
providers: [
  FullLoggerService,
  { provide: MinimalLogger, useClass: FullLoggerService },
]

// useExisting: ONE shared instance
providers: [
  FullLoggerService,
  { provide: MinimalLogger, useExisting: FullLoggerService },
]
```

### Multi Providers

When multiple providers should contribute values to the same token, use `multi: true`. The injected value becomes an array.

```typescript
const HTTP_INTERCEPTOR = new InjectionToken<HttpInterceptorFn[]>('interceptors');

providers: [
  { provide: HTTP_INTERCEPTOR, useValue: loggingInterceptor, multi: true },
  { provide: HTTP_INTERCEPTOR, useValue: authInterceptor, multi: true },
  { provide: HTTP_INTERCEPTOR, useValue: retryInterceptor, multi: true },
]

// Injected value: [loggingInterceptor, authInterceptor, retryInterceptor]
```

Angular uses multi providers internally for things like `HTTP_INTERCEPTORS`, `APP_INITIALIZER`, `ROUTES`, and `ENVIRONMENT_INITIALIZER`.

## InjectionToken

When the thing you want to inject is not a class (a string, number, object, function, or interface), use `InjectionToken` to create a unique token.

```typescript
import { InjectionToken } from '@angular/core';

// Simple token
export const API_URL = new InjectionToken<string>('API_URL');

// Token with a default factory (tree-shakable)
export const FEATURE_FLAGS = new InjectionToken<FeatureFlags>('FeatureFlags', {
  providedIn: 'root',
  factory: () => ({
    darkMode: false,
    experimentalSearch: false,
  }),
});

// Usage
@Component({ ... })
export class AppComponent {
  private apiUrl = inject(API_URL);
  private flags = inject(FEATURE_FLAGS);
}
```

Tokens with `factory` are tree-shakable: if nothing injects the token, the factory and its value are removed from the bundle.

## The Injector Hierarchy

Angular does not have a single flat container of providers. It has a hierarchy of injectors organized into two parallel trees.

### Element Injector Tree

Every component and directive has an associated **element injector**. When you add providers to a component's `providers` or `viewProviders` array, those providers are registered in that component's element injector.

```typescript
@Component({
  selector: 'app-checkout',
  providers: [CartService], // Registered in this component's element injector
  template: `...`,
})
export class CheckoutComponent {
  private cart = inject(CartService); // Gets the local instance
}
```

Child components inherit from their parent's element injector. If `ChildComponent` inside `CheckoutComponent` injects `CartService`, it gets the same instance registered in `CheckoutComponent`.

### Environment Injector Tree

Environment injectors are not tied to specific components. They exist at the application level and at route boundaries.

| Environment Injector | How to Register Providers |
|---------------------|--------------------------|
| Root | `providedIn: 'root'`, `bootstrapApplication(App, { providers: [...] })` |
| Route | `{ path: 'admin', providers: [...], loadComponent: ... }` |
| Platform | `providedIn: 'platform'` (rare, for multi-app pages) |

### Resolution Order

When a component calls `inject(SomeService)`, Angular resolves the dependency in this order:

1. **Element injector** of the requesting component
2. **Element injector** of each parent component, walking up the DOM tree
3. **Environment injector** of the route (if the component is routed)
4. **Root environment injector** (application-level providers)
5. **Platform environment injector**
6. **NullInjector** (throws `NullInjectorError` or returns `null` if optional)

```
Component Element Injector
        |
Parent Element Injector
        |
Grandparent Element Injector
        |
   ... up to root component ...
        |
Route Environment Injector (if route-loaded)
        |
Root Environment Injector
        |
Platform Environment Injector
        |
NullInjector (throws error)
```

## providedIn Options

The `@Injectable()` decorator's `providedIn` property controls where a service is registered and whether it can be tree-shaken.

### providedIn: 'root'

Registers the service as a singleton in the root environment injector. This is the default and most common option.

```typescript
@Injectable({ providedIn: 'root' })
export class AuthService {
  // Singleton across the entire application
  // Tree-shakable: removed from bundle if never injected
}
```

### providedIn: 'platform'

Registers the service in the platform injector. Shared across multiple Angular applications on the same page (micro-frontend scenario).

```typescript
@Injectable({ providedIn: 'platform' })
export class SharedAnalyticsService {
  // Same instance across multiple Angular apps on the page
}
```

### providedIn: 'any' (Deprecated)

Created a separate instance for each lazy-loaded module boundary. This is deprecated and rarely seen in modern Angular.

### No providedIn (Manual Registration)

Without `providedIn`, the service is not registered anywhere by default. You must manually add it to a `providers` array.

```typescript
@Injectable() // No providedIn
export class ScopedService {
  // Must be explicitly provided somewhere
}

// At component level
@Component({
  providers: [ScopedService], // New instance per component
})
export class MyComponent {}

// At route level
{
  path: 'feature',
  providers: [ScopedService], // New instance per route activation
  loadComponent: () => import('./feature.component'),
}
```

### Tree Shaking Comparison

| Registration Method | Tree-Shakable | When to Use |
|-------------------|---------------|-------------|
| `providedIn: 'root'` | Yes | App-wide singletons (most services) |
| `providedIn: 'platform'` | Yes | Multi-app pages |
| Component `providers` | N/A (always included) | Per-component instances |
| Route `providers` | N/A (lazy-loaded) | Per-route scoped services |
| `bootstrapApplication` providers | No | Third-party configuration, non-tree-shakable setup |

## Component-Level vs Route-Level Providers

### Component Providers

Every instance of the component gets its own instance of the service. The service is destroyed when the component is destroyed.

```typescript
@Component({
  selector: 'app-editor',
  providers: [UndoRedoService], // Each editor gets its own undo stack
  template: `
    <button (click)="undo()">Undo</button>
    <button (click)="redo()">Redo</button>
    <textarea (input)="onInput($event)"></textarea>
  `,
})
export class EditorComponent {
  private undoRedo = inject(UndoRedoService);

  undo() { this.undoRedo.undo(); }
  redo() { this.undoRedo.redo(); }
  onInput(event: Event) { /* ... */ }
}
```

If you have two `<app-editor>` instances on the page, each has its own `UndoRedoService`. Child components of each editor share the same instance.

### viewProviders

`viewProviders` works like `providers`, but the service is only visible to the component's own template children, not to content-projected children.

```typescript
@Component({
  selector: 'app-panel',
  viewProviders: [PanelStateService],
  template: `
    <div class="panel-body">
      <!-- PanelStateService is available here -->
      <app-panel-header />
      <ng-content></ng-content>
      <!-- Projected content CANNOT see PanelStateService -->
    </div>
  `,
})
export class PanelComponent {}
```

Use `viewProviders` when you want a "privacy boundary" that prevents projected content from accidentally using internal services.

### Route-Level Providers

Providers on a route create an environment injector scoped to that route. The services live as long as the route is active and are shared by all components in that route subtree.

```typescript
// app.routes.ts
export const routes: Routes = [
  {
    path: 'admin',
    providers: [
      AdminGuardService,
      AdminApiService,
      { provide: THEME, useValue: 'dark' },
    ],
    children: [
      { path: 'users', loadComponent: () => import('./admin-users.component') },
      { path: 'settings', loadComponent: () => import('./admin-settings.component') },
    ],
  },
];
```

Both `AdminUsersComponent` and `AdminSettingsComponent` share the same `AdminApiService` instance. When the user navigates away from `/admin`, all route-scoped services are destroyed.

This is the standalone replacement for what NgModule `providers` used to do for lazy-loaded feature modules.

## Injection Context and runInInjectionContext

An **injection context** is the runtime state where `inject()` is available. The most common contexts are constructors, field initializers, and factory functions. Outside these, `inject()` throws `NG0203`.

When you need to call `inject()` outside a natural injection context, use `runInInjectionContext()`:

```typescript
import { runInInjectionContext, EnvironmentInjector, inject } from '@angular/core';

@Component({ ... })
export class DynamicLoaderComponent {
  private envInjector = inject(EnvironmentInjector);

  loadWidget() {
    // This runs outside an injection context (event handler),
    // so we create one explicitly
    runInInjectionContext(this.envInjector, () => {
      const factory = inject(WidgetFactory);
      factory.create();
    });
  }
}
```

### DestroyRef

`DestroyRef` is an injectable token that lets you register cleanup callbacks tied to a component/service lifecycle:

```typescript
@Component({ ... })
export class SearchComponent {
  private destroyRef = inject(DestroyRef);

  constructor() {
    const subscription = someObservable$.subscribe(value => {
      // handle value
    });

    this.destroyRef.onDestroy(() => {
      subscription.unsubscribe();
    });
  }
}
```

`DestroyRef` is available in any injection context that has a lifecycle (components, directives, services). It replaces the `ngOnDestroy` lifecycle hook for cleanup in many cases.

## Functional Guards, Resolvers, and Interceptors

Angular's DI system powers functional (non-class) guards, resolvers, and interceptors. These are plain functions that run in an injection context.

### Functional Guard

```typescript
// auth.guard.ts
export const authGuard: CanActivateFn = (route, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (auth.isAuthenticated()) {
    return true;
  }
  return router.createUrlTree(['/login'], {
    queryParams: { returnUrl: state.url },
  });
};

// Usage in routes
{
  path: 'dashboard',
  canActivate: [authGuard],
  loadComponent: () => import('./dashboard.component'),
}
```

### Functional Resolver

```typescript
// user.resolver.ts
export const userResolver: ResolveFn<User> = (route) => {
  const userService = inject(UserService);
  const id = route.paramMap.get('id')!;
  return userService.getUser(id);
};

// Usage in routes
{
  path: 'users/:id',
  resolve: { user: userResolver },
  loadComponent: () => import('./user-detail.component'),
}
```

### Functional Interceptor

```typescript
// auth.interceptor.ts
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const auth = inject(AuthService);
  const token = auth.getToken();

  if (token) {
    req = req.clone({
      setHeaders: { Authorization: `Bearer ${token}` },
    });
  }
  return next(req);
};

// Registration
export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(withInterceptors([authInterceptor])),
  ],
};
```

All three patterns use `inject()` inside a function body. Angular ensures these functions run within an injection context so `inject()` works.

## forwardRef()

`forwardRef()` solves circular reference issues where a class is used before it is declared. This is rare but occurs with self-referencing or mutually dependent types.

```typescript
import { forwardRef } from '@angular/core';

// TreeNodeComponent references itself for recursive rendering
@Component({
  selector: 'app-tree-node',
  imports: [forwardRef(() => TreeNodeComponent)],
  template: `
    <span>{{ node().label }}</span>
    @if (node().children?.length) {
      @for (child of node().children!; track child.id) {
        <app-tree-node [node]="child" />
      }
    }
  `,
})
export class TreeNodeComponent {
  node = input.required<TreeNode>();
}
```

Another use case is when a class references itself in its own providers:

```typescript
@Component({
  selector: 'app-parent',
  providers: [
    {
      provide: PARENT_COMPONENT,
      useExisting: forwardRef(() => ParentComponent),
    },
  ],
  template: `<ng-content></ng-content>`,
})
export class ParentComponent {}
```

## Injection Tokens for Configuration

A common pattern is using `InjectionToken` with factory functions to create configurable, tree-shakable application settings.

```typescript
// tokens.ts
export interface AppConfig {
  apiUrl: string;
  retryAttempts: number;
  featureFlags: Record<string, boolean>;
}

export const APP_CONFIG = new InjectionToken<AppConfig>('AppConfig');

// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [
    {
      provide: APP_CONFIG,
      useValue: {
        apiUrl: 'https://api.example.com',
        retryAttempts: 3,
        featureFlags: { newCheckout: true },
      } satisfies AppConfig,
    },
  ],
};

// any-service.ts
@Injectable({ providedIn: 'root' })
export class ApiService {
  private config = inject(APP_CONFIG);
  private http = inject(HttpClient);

  get<T>(path: string) {
    return this.http.get<T>(`${this.config.apiUrl}${path}`);
  }
}
```

### Token with Default Factory

When a token has a reasonable default but can be overridden:

```typescript
export const RETRY_COUNT = new InjectionToken<number>('RetryCount', {
  providedIn: 'root',
  factory: () => 3, // Default: 3 retries
});

// Override in a specific route
{
  path: 'critical-feature',
  providers: [{ provide: RETRY_COUNT, useValue: 5 }],
  loadComponent: () => import('./critical.component'),
}
```

## Common Mistakes

### 1. Calling inject() Outside an Injection Context

```typescript
// BAD: inject() in ngOnInit
ngOnInit() {
  this.service = inject(MyService); // NG0203 error
}

// GOOD: inject() in field initializer or constructor
private service = inject(MyService);
```

### 2. Forgetting That Component Providers Create New Instances

```typescript
// BAD: Expecting a singleton but providing at component level
@Component({
  providers: [AuthService], // Every instance of this component gets a NEW AuthService
})
export class HeaderComponent {}

// GOOD: Use providedIn: 'root' for singletons
@Injectable({ providedIn: 'root' })
export class AuthService {}
```

### 3. Circular Dependencies

```typescript
// BAD: ServiceA injects ServiceB, ServiceB injects ServiceA
@Injectable({ providedIn: 'root' })
export class ServiceA {
  private b = inject(ServiceB); // ServiceB also injects ServiceA -> circular
}

// GOOD: Break the cycle with a mediator service, or use lazy injection
@Injectable({ providedIn: 'root' })
export class ServiceA {
  private injector = inject(Injector);

  doSomething() {
    // Resolve lazily at call time, not at construction time
    const b = this.injector.get(ServiceB);
    b.process();
  }
}
```

### 4. Using useClass When useExisting Was Intended

```typescript
// BAD: Creates two separate instances
providers: [
  RealLoggerService,
  { provide: AbstractLogger, useClass: RealLoggerService }, // Second instance!
]

// GOOD: Aliases to the same instance
providers: [
  RealLoggerService,
  { provide: AbstractLogger, useExisting: RealLoggerService },
]
```

### 5. Not Understanding the Resolution Order

```typescript
// Component provides ServiceA
@Component({
  providers: [{ provide: ServiceA, useClass: MockServiceA }],
  template: `<app-child />`,
})
export class ParentComponent {}

// Child expects the root-level ServiceA but gets MockServiceA
@Component({ ... })
export class ChildComponent {
  // Gets MockServiceA from parent's element injector, NOT the root singleton
  private service = inject(ServiceA);
}

// FIX: Use { skipSelf: true } or { self: true } to control resolution
private service = inject(ServiceA, { skipSelf: true }); // Skip parent, go higher
```

### 6. Providing Services in bootstrapApplication That Should Be Tree-Shakable

```typescript
// BAD: Not tree-shakable
export const appConfig: ApplicationConfig = {
  providers: [
    MyRarelyUsedService, // Always in the bundle, even if never injected
  ],
};

// GOOD: Tree-shakable via providedIn
@Injectable({ providedIn: 'root' })
export class MyRarelyUsedService {}
// Only in the bundle if actually injected somewhere
```

## Best Practices

### 1. Default to providedIn: 'root' for Services

Most services should be application-wide singletons. Use `providedIn: 'root'` for automatic tree shaking and simplicity. Only use component/route-level providers when you need scoped instances.

### 2. Use inject() Over Constructor Injection

`inject()` provides better type inference, works in functional contexts, simplifies inheritance, and does not require parameter decorators.

### 3. Expose Read-Only Interfaces

Combine `InjectionToken` with `useExisting` to expose narrow interfaces:

```typescript
// Full implementation
@Injectable({ providedIn: 'root' })
export class UserService {
  private users = signal<User[]>([]);
  readonly users$ = this.users.asReadonly();
  addUser(user: User) { /* ... */ }
  deleteUser(id: string) { /* ... */ }
}

// Read-only token for components that should not modify data
export const USER_READER = new InjectionToken<Pick<UserService, 'users$'>>('UserReader', {
  providedIn: 'root',
  factory: () => inject(UserService),
});
```

### 4. Use Route Providers for Feature Scoping

Instead of providing services globally when they are only needed in a feature area, scope them to routes:

```typescript
{
  path: 'checkout',
  providers: [CheckoutService, CartCalculationService],
  children: [
    { path: '', loadComponent: () => import('./checkout.component') },
    { path: 'confirm', loadComponent: () => import('./confirm.component') },
  ],
}
```

### 5. Prefer InjectionToken with Factory for Configuration

Tokens with factories are tree-shakable and self-documenting. They provide a default and can be overridden at any level.

### 6. Use DestroyRef for Cleanup Over ngOnDestroy

```typescript
// Modern cleanup pattern
export class MyComponent {
  private destroyRef = inject(DestroyRef);
  private subscription = inject(DataService).stream$.subscribe(data => {
    // handle
  });

  constructor() {
    this.destroyRef.onDestroy(() => this.subscription.unsubscribe());
  }
}
```

## Real-World Use Cases

### Multi-Tenant Configuration

```typescript
// tenant-config.token.ts
export interface TenantConfig {
  name: string;
  theme: 'light' | 'dark';
  logoUrl: string;
  apiBase: string;
  features: string[];
}

export const TENANT_CONFIG = new InjectionToken<TenantConfig>('TenantConfig');

// app.config.ts: Provide via factory that reads runtime config
export const appConfig: ApplicationConfig = {
  providers: [
    {
      provide: TENANT_CONFIG,
      useFactory: () => {
        // Loaded at runtime via a script tag or JSON fetch
        const config = (window as any).__TENANT_CONFIG__;
        if (!config) throw new Error('Tenant config not found');
        return config as TenantConfig;
      },
    },
    {
      provide: APP_INITIALIZER,
      useFactory: () => {
        const tenant = inject(TENANT_CONFIG);
        return () => {
          document.title = tenant.name;
          document.documentElement.setAttribute('data-theme', tenant.theme);
        };
      },
      multi: true,
    },
  ],
};

// Any component or service can inject the tenant config
@Component({ ... })
export class HeaderComponent {
  protected tenant = inject(TENANT_CONFIG);
  // template: <img [src]="tenant.logoUrl" [alt]="tenant.name" />
}
```

### Pluggable Logger with Multi Providers

```typescript
// logger.token.ts
export interface LogHandler {
  log(level: string, message: string, context?: Record<string, unknown>): void;
}
export const LOG_HANDLER = new InjectionToken<LogHandler[]>('LogHandlers');

// Implementations
export const consoleLogHandler: LogHandler = {
  log: (level, message, context) => console[level as 'log'](`[${level}] ${message}`, context),
};

export const sentryLogHandler: LogHandler = {
  log: (level, message, context) => {
    if (level === 'error') {
      // Send to Sentry
      Sentry.captureMessage(message, { extra: context });
    }
  },
};

// Registration
export const appConfig: ApplicationConfig = {
  providers: [
    { provide: LOG_HANDLER, useValue: consoleLogHandler, multi: true },
    { provide: LOG_HANDLER, useValue: sentryLogHandler, multi: true },
  ],
};

// Logger service aggregates all handlers
@Injectable({ providedIn: 'root' })
export class LoggerService {
  private handlers = inject(LOG_HANDLER);

  error(message: string, context?: Record<string, unknown>) {
    this.handlers.forEach(h => h.log('error', message, context));
  }

  info(message: string, context?: Record<string, unknown>) {
    this.handlers.forEach(h => h.log('info', message, context));
  }
}
```

### Scoped State per Route with Component Providers

```typescript
// form-state.service.ts
@Injectable() // No providedIn: must be explicitly provided
export class FormStateService {
  private _dirty = signal(false);
  private _data = signal<Record<string, unknown>>({});

  readonly dirty = this._dirty.asReadonly();
  readonly data = this._data.asReadonly();

  patch(key: string, value: unknown) {
    this._data.update(d => ({ ...d, [key]: value }));
    this._dirty.set(true);
  }

  reset() {
    this._data.set({});
    this._dirty.set(false);
  }
}

// Each wizard step shares the same FormStateService because the parent provides it
@Component({
  selector: 'app-wizard',
  providers: [FormStateService], // Scoped to this wizard instance
  template: `
    <nav>
      <a routerLink="step1">Step 1</a>
      <a routerLink="step2">Step 2</a>
      <a routerLink="step3">Step 3</a>
    </nav>
    <router-outlet />
    <footer>
      <span>Unsaved changes: {{ formState.dirty() }}</span>
    </footer>
  `,
})
export class WizardComponent {
  protected formState = inject(FormStateService);
}

// step1.component.ts
@Component({ ... })
export class Step1Component {
  // Gets the same FormStateService instance from the parent WizardComponent
  private formState = inject(FormStateService);

  onNameChange(name: string) {
    this.formState.patch('name', name);
  }
}
```

Multiple `<app-wizard>` instances on the same page each have their own isolated `FormStateService`.

---

> **Key Takeaway:** Angular's DI system resolves dependencies by walking up two parallel trees (element injectors then environment injectors). Use `inject()` for requesting dependencies, `providedIn: 'root'` for tree-shakable singletons, component/route `providers` for scoped instances, and `InjectionToken` for non-class values.
