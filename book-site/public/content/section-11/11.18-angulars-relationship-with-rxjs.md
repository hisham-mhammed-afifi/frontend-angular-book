# 11.18 Angular's Relationship with RxJS

> RxJS is deeply embedded in Angular's internals (HttpClient, Router, Reactive Forms) and remains essential for complex async workflows. Signals now handle most local UI state, but knowing when to use each and how to bridge them is critical for building modern Angular applications.

---

## How Angular Uses RxJS Internally

Angular does not simply recommend RxJS. It uses RxJS as a core dependency in several subsystems. Even in a fully signal-based application, you will encounter observables in these areas:

| Angular Subsystem | RxJS Usage |
|-------------------|-----------|
| `HttpClient` | All HTTP methods (`get`, `post`, `put`, `delete`) return `Observable<T>` |
| `Router` | `Router.events`, `ActivatedRoute.params`, `ActivatedRoute.queryParams`, `ActivatedRoute.data` are all observables |
| `Reactive Forms` | `valueChanges` and `statusChanges` on `FormControl`, `FormGroup`, and `FormArray` are observables |
| `EventEmitter` | Extends `Subject<T>` (though `output()` now replaces it without RxJS) |
| `AsyncPipe` | Subscribes to observables and signals in templates |
| `Interceptors` | `HttpInterceptorFn` receives and returns `Observable<HttpEvent>` |
| `Guards/Resolvers` | Can return `Observable<boolean>` or `Observable<T>` |

```typescript
// HttpClient returns Observable
const users$ = this.http.get<User[]>('/api/users');

// Router params are Observable
const id$ = this.route.paramMap.pipe(
  map(params => params.get('id'))
);

// Reactive forms expose Observable
const search$ = this.searchControl.valueChanges;
```

## Signals vs Observables: When to Use Each

Signals and observables solve different problems. They are complementary, not competing.

### Use Signals For

- **Local component state**: Counters, toggles, form field values, UI flags
- **Derived/computed state**: Filtered lists, totals, formatted values
- **Template rendering**: Anything the template reads synchronously
- **State shared across components**: Via services with `signal()` + `asReadonly()`

```typescript
// Signal-based component state
@Component({ ... })
export class ProductListComponent {
  private productService = inject(ProductService);

  filter = signal('');
  sortBy = signal<'name' | 'price'>('name');
  products = this.productService.products; // Signal from service

  filteredProducts = computed(() => {
    const term = this.filter().toLowerCase();
    let items = this.products();
    if (term) {
      items = items.filter(p => p.name.toLowerCase().includes(term));
    }
    return items.sort((a, b) =>
      this.sortBy() === 'price' ? a.price - b.price : a.name.localeCompare(b.name)
    );
  });
}
```

### Use Observables For

- **Complex async coordination**: Multiple streams that depend on each other, race conditions, retries
- **Event streams**: WebSocket messages, SSE, DOM events that need debouncing/throttling
- **Cancellation logic**: Aborting in-flight requests when a new one starts
- **Time-based operations**: Delays, intervals, timeouts, polling
- **Backpressure handling**: Throttling, buffering, sampling high-frequency events

```typescript
// Observable-based: autocomplete with debounce, distinct, and cancellation
@Component({ ... })
export class SearchComponent {
  private http = inject(HttpClient);
  searchTerm = signal('');

  // Signal -> Observable -> RxJS operators -> Signal
  private results$ = toObservable(this.searchTerm).pipe(
    debounceTime(300),
    distinctUntilChanged(),
    filter(term => term.length >= 2),
    switchMap(term => this.http.get<Result[]>(`/api/search?q=${term}`)),
  );

  results = toSignal(this.results$, { initialValue: [] });
}
```

### Decision Table

| Scenario | Use |
|----------|-----|
| Toggle a sidebar open/closed | `signal(false)` |
| Compute a filtered list from other signals | `computed()` |
| Selected item that resets when list changes | `linkedSignal()` |
| Fetch data when a signal changes | `resource()` / `rxResource()` |
| Autocomplete with debounce | Observable + `toSignal()` |
| WebSocket real-time feed | Observable + `toSignal()` |
| Retry failed HTTP requests with backoff | Observable with `retry()` |
| Poll an API every 30 seconds | Observable with `timer()` + `switchMap()` |
| Track form validity in a reactive form | Observable (`statusChanges`) |
| Coordinate multiple parallel API calls | `forkJoin()` or `combineLatest()` |
| Event from child component | `output()` (no RxJS) |
| Read route parameters | Observable or `input()` with `withComponentInputBinding()` |

## The Interop Bridge: toSignal() and toObservable()

The `@angular/core/rxjs-interop` package provides the bridge between the two worlds.

### toSignal(): Observable to Signal

Subscribes to an observable and exposes the latest emitted value as a signal. The subscription is automatically managed and cleaned up.

```typescript
import { toSignal } from '@angular/core/rxjs-interop';

@Component({ ... })
export class DashboardComponent {
  private http = inject(HttpClient);
  private route = inject(ActivatedRoute);

  // Observable -> Signal with initial value
  stats = toSignal(
    this.http.get<DashboardStats>('/api/stats'),
    { initialValue: null }
  );

  // Route params as a signal
  userId = toSignal(
    this.route.paramMap.pipe(map(p => p.get('id')!)),
    { initialValue: '' }
  );
}
```

#### toSignal Options

| Option | Effect |
|--------|--------|
| `initialValue: T` | Sets the starting value. Signal type becomes `Signal<T>` (no `undefined`). |
| `requireSync: true` | Asserts synchronous first emission (e.g., `BehaviorSubject`). Removes `undefined` from type. |
| `manualCleanup: true` | Prevents auto-unsubscribe on destroy. Rarely needed. |
| `rejectErrors: false` | Swallows errors instead of throwing. |
| `equal: (a, b) => boolean` | Custom equality for the resulting signal. |

```typescript
// BehaviorSubject emits synchronously: no undefined in type
const subject = new BehaviorSubject<number>(0);
const count = toSignal(subject, { requireSync: true });
// count: Signal<number>

// Regular Observable: needs initialValue or type includes undefined
const data = toSignal(this.http.get<Data>('/api/data'));
// data: Signal<Data | undefined>

const data2 = toSignal(this.http.get<Data>('/api/data'), { initialValue: null });
// data2: Signal<Data | null>
```

### toObservable(): Signal to Observable

Converts a signal into an observable via an internal `effect()` and `ReplaySubject(1)`.

```typescript
import { toObservable } from '@angular/core/rxjs-interop';

@Component({ ... })
export class SearchComponent {
  searchTerm = signal('');

  // Use RxJS operators on a signal's value stream
  private searchResults$ = toObservable(this.searchTerm).pipe(
    debounceTime(300),
    distinctUntilChanged(),
    switchMap(term =>
      term.length < 2
        ? of([])
        : this.http.get<SearchResult[]>(`/api/search?q=${term}`)
    ),
    catchError(() => of([])),
  );

  // Convert back to signal for the template
  searchResults = toSignal(this.searchResults$, { initialValue: [] });
}
```

**Key behavior:** `toObservable()` emits asynchronously (via a microtask) because it uses an `effect()` internally. This means the observable does not emit synchronously when the signal changes.

## RxJS Operators You Will Still Use

Even in a signal-heavy application, certain RxJS operators remain indispensable when working with Angular's observable-based APIs.

### Higher-Order Mapping Operators

These operators map each source value to an inner observable and manage the subscription strategy.

| Operator | Inner Sub Strategy | Use Case |
|----------|--------------------|----------|
| `switchMap` | Cancels previous, subscribes to new | Search/autocomplete, route-based data loading |
| `concatMap` | Queues, processes sequentially | Sequential writes, ordered operations |
| `mergeMap` | Runs all in parallel | Parallel independent requests |
| `exhaustMap` | Ignores new until current completes | Form submission (prevent double submit) |

```typescript
// switchMap: Cancel previous search when new term arrives
searchControl.valueChanges.pipe(
  debounceTime(300),
  distinctUntilChanged(),
  switchMap(term => this.http.get<Result[]>(`/api/search?q=${term}`))
);

// concatMap: Process file uploads one at a time
this.uploadQueue$.pipe(
  concatMap(file => this.http.post('/api/upload', file))
);

// mergeMap: Fetch details for multiple items in parallel
this.selectedIds$.pipe(
  mergeMap(id => this.http.get<Item>(`/api/items/${id}`))
);

// exhaustMap: Ignore rapid clicks on submit button
this.submitClick$.pipe(
  exhaustMap(() => this.http.post('/api/order', this.formData()))
);
```

### Filtering and Timing

```typescript
// debounceTime: Wait for pause in input
searchControl.valueChanges.pipe(
  debounceTime(300),  // Wait 300ms after last keystroke
);

// throttleTime: Limit frequency of events
fromEvent(window, 'scroll').pipe(
  throttleTime(100),  // At most one event per 100ms
);

// distinctUntilChanged: Skip duplicate consecutive values
searchControl.valueChanges.pipe(
  distinctUntilChanged(),  // Only emit when value actually changes
);

// filter: Skip values that don't meet criteria
searchControl.valueChanges.pipe(
  filter(term => term.length >= 3),  // Only search with 3+ characters
);
```

### Combination Operators

```typescript
// combineLatest: React when ANY source emits (after all have emitted once)
combineLatest([
  this.userId$,
  this.dateRange$,
  this.filter$,
]).pipe(
  switchMap(([userId, range, filter]) =>
    this.http.get('/api/logs', { params: { userId, ...range, filter } })
  )
);

// forkJoin: Wait for ALL observables to complete, emit final values
forkJoin({
  user: this.http.get<User>('/api/user/1'),
  posts: this.http.get<Post[]>('/api/user/1/posts'),
  settings: this.http.get<Settings>('/api/user/1/settings'),
}).subscribe(({ user, posts, settings }) => {
  // All three requests completed
});

// merge: Combine multiple event sources into one stream
merge(
  fromEvent(this.saveBtn.nativeElement, 'click'),
  fromEvent(document, 'keydown').pipe(filter(e => e.ctrlKey && e.key === 's')),
).pipe(
  exhaustMap(() => this.saveDocument())
);
```

### Error Handling and Retry

```typescript
// catchError: Handle errors gracefully
this.http.get<Data>('/api/data').pipe(
  catchError(error => {
    console.error('Request failed', error);
    return of(null); // Return fallback value
  })
);

// retry with delay: Retry failed requests
this.http.get<Data>('/api/data').pipe(
  retry({
    count: 3,
    delay: (error, retryCount) => timer(retryCount * 1000), // 1s, 2s, 3s
  }),
  catchError(error => {
    this.notificationService.error('Failed after 3 retries');
    return EMPTY;
  })
);
```

## Replacing BehaviorSubject with Signals

The most common RxJS-to-signals migration is replacing `BehaviorSubject` patterns in services.

### Before: BehaviorSubject Pattern

```typescript
// BAD (verbose): BehaviorSubject-based service
@Injectable({ providedIn: 'root' })
export class CartService {
  private itemsSubject = new BehaviorSubject<CartItem[]>([]);
  readonly items$ = this.itemsSubject.asObservable();

  readonly total$ = this.items$.pipe(
    map(items => items.reduce((sum, i) => sum + i.price * i.qty, 0))
  );

  readonly count$ = this.items$.pipe(
    map(items => items.length)
  );

  addItem(item: CartItem) {
    const current = this.itemsSubject.getValue();
    this.itemsSubject.next([...current, item]);
  }

  removeItem(id: string) {
    const current = this.itemsSubject.getValue();
    this.itemsSubject.next(current.filter(i => i.id !== id));
  }
}

// Component had to use async pipe or subscribe
@Component({
  template: `
    <span>{{ cartService.count$ | async }} items</span>
    <span>{{ cartService.total$ | async | currency }}</span>
  `,
})
export class HeaderComponent {
  cartService = inject(CartService);
}
```

### After: Signal Pattern

```typescript
// GOOD (concise): Signal-based service
@Injectable({ providedIn: 'root' })
export class CartService {
  private _items = signal<CartItem[]>([]);
  readonly items = this._items.asReadonly();

  readonly total = computed(() =>
    this._items().reduce((sum, i) => sum + i.price * i.qty, 0)
  );

  readonly count = computed(() => this._items().length);

  addItem(item: CartItem) {
    this._items.update(items => [...items, item]);
  }

  removeItem(id: string) {
    this._items.update(items => items.filter(i => i.id !== id));
  }
}

// Component: no async pipe, no subscription management
@Component({
  template: `
    <span>{{ cartService.count() }} items</span>
    <span>{{ cartService.total() | currency }}</span>
  `,
})
export class HeaderComponent {
  cartService = inject(CartService);
}
```

### Mapping BehaviorSubject Patterns to Signals

| BehaviorSubject Pattern | Signal Equivalent |
|------------------------|-------------------|
| `new BehaviorSubject(initialValue)` | `signal(initialValue)` |
| `subject.getValue()` | `mySignal()` |
| `subject.next(newValue)` | `mySignal.set(newValue)` |
| `subject.asObservable()` | `mySignal.asReadonly()` |
| `subject.pipe(map(...))` | `computed(() => ...)` |
| `combineLatest([a$, b$]).pipe(map(...))` | `computed(() => fn(a(), b()))` |
| `subject.pipe(distinctUntilChanged())` | Built-in (signals use equality checks) |
| `subject.pipe(filter(...))` | Logic inside `computed()` |

## Subscription Management

When you do use observables, managing subscriptions is critical to prevent memory leaks.

### takeUntilDestroyed() (Preferred)

The simplest modern approach. Automatically completes the observable when the component/directive/service is destroyed.

```typescript
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';

@Component({ ... })
export class LiveFeedComponent {
  private ws = inject(WebSocketService);

  constructor() {
    // Automatically unsubscribes on component destroy
    this.ws.messages$.pipe(
      takeUntilDestroyed(),
    ).subscribe(msg => {
      this.messages.update(msgs => [...msgs, msg]);
    });
  }

  messages = signal<Message[]>([]);
}
```

`takeUntilDestroyed()` must be called in an injection context (constructor or field initializer). To use it elsewhere, pass a `DestroyRef`:

```typescript
@Component({ ... })
export class MyComponent {
  private destroyRef = inject(DestroyRef);

  ngOnInit() {
    someObservable$.pipe(
      takeUntilDestroyed(this.destroyRef),
    ).subscribe(value => { ... });
  }
}
```

### toSignal() (Also Manages Subscriptions)

When the goal is to display observable data in a template, `toSignal()` manages the subscription automatically:

```typescript
// No manual subscribe, no async pipe, no cleanup needed
protected userData = toSignal(this.http.get<User>('/api/me'), { initialValue: null });
```

### Subscription Hierarchy

Prefer these approaches in this order:

1. **toSignal()**: If you need the value in templates or computed signals
2. **takeUntilDestroyed()**: If you need side effects from a subscription
3. **DestroyRef.onDestroy()**: If you need fine-grained manual control
4. **Manual unsubscribe**: Last resort, for edge cases

```typescript
// 1. toSignal (best for template consumption)
data = toSignal(this.service.data$, { initialValue: [] });

// 2. takeUntilDestroyed (best for side-effect subscriptions)
constructor() {
  this.events$.pipe(takeUntilDestroyed()).subscribe(e => this.handleEvent(e));
}

// 3. DestroyRef (for non-injection contexts)
private destroyRef = inject(DestroyRef);
startPolling() {
  const sub = interval(30000).pipe(
    switchMap(() => this.http.get('/api/status'))
  ).subscribe(status => this.status.set(status));
  this.destroyRef.onDestroy(() => sub.unsubscribe());
}
```

### takeUntilDestroyed Placement

Always place `takeUntilDestroyed()` **after** higher-order mapping operators to avoid inner observable leaks:

```typescript
// BAD: takeUntilDestroyed before switchMap
// Inner observable from switchMap can leak
source$.pipe(
  takeUntilDestroyed(),
  switchMap(id => this.http.get(`/api/items/${id}`)), // This inner observable is NOT cleaned up
);

// GOOD: takeUntilDestroyed after all operators
source$.pipe(
  switchMap(id => this.http.get(`/api/items/${id}`)),
  takeUntilDestroyed(),
);
```

## Resource APIs: The Signal-Native Async Pattern

For the most common async pattern (fetch data when parameters change), Angular's resource APIs eliminate the need for manual observable plumbing.

### resource() Replaces switchMap + BehaviorSubject

```typescript
// BEFORE: Observable-based data fetching
@Component({ ... })
export class UserProfileComponent implements OnInit, OnDestroy {
  private destroy$ = new Subject<void>();
  private userId$ = new BehaviorSubject<string>('');
  user: User | null = null;

  ngOnInit() {
    this.route.paramMap.pipe(
      map(p => p.get('id')!),
      tap(id => this.userId$.next(id)),
      switchMap(id => this.http.get<User>(`/api/users/${id}`)),
      takeUntil(this.destroy$),
    ).subscribe(user => this.user = user);
  }

  ngOnDestroy() {
    this.destroy$.next();
    this.destroy$.complete();
  }
}

// AFTER: Signal + resource
@Component({ ... })
export class UserProfileComponent {
  private route = inject(ActivatedRoute);
  userId = toSignal(this.route.paramMap.pipe(map(p => p.get('id')!)), { initialValue: '' });

  userResource = resource({
    request: () => this.userId(),
    loader: async ({ request: id, abortSignal }) => {
      const res = await fetch(`/api/users/${id}`, { signal: abortSignal });
      return res.json() as Promise<User>;
    },
  });
}
```

### rxResource() for RxJS-Based Loaders

```typescript
import { rxResource } from '@angular/core/rxjs-interop';

@Component({ ... })
export class UserProfileComponent {
  private http = inject(HttpClient);
  userId = signal('1');

  userResource = rxResource({
    request: () => this.userId(),
    loader: ({ request: id }) => this.http.get<User>(`/api/users/${id}`),
  });

  // Template uses: userResource.value(), userResource.status(), userResource.error()
}
```

### httpResource() for Simple HTTP Fetches

```typescript
import { httpResource } from '@angular/common/http';

@Component({ ... })
export class UserProfileComponent {
  userId = signal('1');

  // Minimal boilerplate
  userResource = httpResource<User>(() => `/api/users/${this.userId()}`);
}
```

## Common Mistakes

### 1. Using Observables for Simple Local State

```typescript
// BAD: Overengineering with BehaviorSubject
private isMenuOpen$ = new BehaviorSubject<boolean>(false);
toggleMenu() { this.isMenuOpen$.next(!this.isMenuOpen$.getValue()); }

// GOOD: Signal is simpler
isMenuOpen = signal(false);
toggleMenu() { this.isMenuOpen.update(v => !v); }
```

### 2. Subscribing in Components Without Cleanup

```typescript
// BAD: Memory leak
ngOnInit() {
  this.dataService.data$.subscribe(data => {
    this.data = data;
  });
  // Never unsubscribed!
}

// GOOD: Use takeUntilDestroyed
constructor() {
  this.dataService.data$.pipe(
    takeUntilDestroyed(),
  ).subscribe(data => this.data.set(data));
}

// BETTER: Use toSignal (no subscribe at all)
data = toSignal(this.dataService.data$, { initialValue: [] });
```

### 3. Using switchMap When concatMap or exhaustMap Is Needed

```typescript
// BAD: switchMap cancels in-flight save requests
this.saveClick$.pipe(
  switchMap(() => this.http.post('/api/save', payload)) // Previous save can be cancelled!
);

// GOOD: exhaustMap ignores clicks while save is in progress
this.saveClick$.pipe(
  exhaustMap(() => this.http.post('/api/save', payload))
);

// GOOD: concatMap queues saves sequentially
this.saveClick$.pipe(
  concatMap(() => this.http.post('/api/save', payload))
);
```

### 4. Placing takeUntilDestroyed Before Higher-Order Operators

```typescript
// BAD: Inner observable from switchMap leaks
source$.pipe(
  takeUntilDestroyed(),
  switchMap(id => longRunningRequest$(id)), // Leak!
);

// GOOD: takeUntilDestroyed goes last
source$.pipe(
  switchMap(id => longRunningRequest$(id)),
  takeUntilDestroyed(),
);
```

### 5. Calling toSignal Outside an Injection Context

```typescript
// BAD: toSignal in ngOnInit
ngOnInit() {
  this.data = toSignal(this.http.get('/api/data')); // NG0203 error
}

// GOOD: toSignal in field initializer or constructor
data = toSignal(this.http.get<Data>('/api/data'), { initialValue: null });
```

### 6. Forgetting forkJoin Requires Completion

```typescript
// BAD: forkJoin with a Subject that never completes
const data$ = new Subject<string>();
forkJoin([data$, this.http.get('/api/other')]).subscribe(/* Never fires */);

// GOOD: forkJoin with completing observables
forkJoin([
  this.http.get('/api/a'), // HTTP observables complete after emitting
  this.http.get('/api/b'),
]).subscribe(([a, b]) => { ... });
```

## Best Practices

### 1. Signals for State, Observables for Events

Use signals to hold and derive application state. Use observables for event streams, async coordination, and time-based operations. Do not mix the patterns within a single concern.

### 2. Convert at the Boundary

Convert observables to signals as close to the consumer as possible. Services can expose observables; components convert them via `toSignal()` or `rxResource()`.

```typescript
// Service exposes Observable (standard RxJS API)
@Injectable({ providedIn: 'root' })
export class NotificationService {
  readonly notifications$ = this.ws.on('notification');
}

// Component converts to signal at the boundary
@Component({ ... })
export class NotificationBellComponent {
  private notifService = inject(NotificationService);
  notifications = toSignal(this.notifService.notifications$, { initialValue: [] });
}
```

### 3. Use resource/rxResource/httpResource for Data Fetching

For the pattern "fetch data when parameters change," prefer resource APIs over manual `switchMap` + `subscribe`. They handle cancellation, status tracking, and cleanup automatically.

### 4. Use takeUntilDestroyed for All Manual Subscriptions

Never use the old `destroy$ = new Subject()` pattern. `takeUntilDestroyed()` is shorter, safer, and does not require implementing `OnDestroy`.

### 5. Choose the Right Higher-Order Mapping Operator

- Read requests (search, navigation): `switchMap` (cancel stale)
- Write requests (save, delete): `exhaustMap` (ignore duplicates) or `concatMap` (queue)
- Independent parallel operations: `mergeMap`

### 6. Do Not Subscribe in Services Just to Set a Signal

```typescript
// BAD: Subscribing just to bridge observable -> signal
@Injectable({ providedIn: 'root' })
export class DataService {
  data = signal<Data | null>(null);
  constructor() {
    this.http.get<Data>('/api/data').subscribe(d => this.data.set(d)); // Leak potential, no cleanup
  }
}

// GOOD: Let the component decide how to consume
@Injectable({ providedIn: 'root' })
export class DataService {
  private http = inject(HttpClient);
  getData() { return this.http.get<Data>('/api/data'); }
}

// In component:
data = toSignal(inject(DataService).getData(), { initialValue: null });
```

## Real-World Use Cases

### Real-Time Dashboard with WebSocket

```typescript
@Injectable({ providedIn: 'root' })
export class StockFeedService {
  private socket$ = webSocket<StockTick>('wss://feeds.example.com/stocks');

  // Observable: real-time event stream, needs RxJS operators
  getPriceFeed(symbol: string): Observable<StockTick> {
    return this.socket$.pipe(
      filter(tick => tick.symbol === symbol),
      distinctUntilChanged((a, b) => a.price === b.price),
      retry({ delay: 3000 }), // Reconnect on failure
    );
  }
}

@Component({
  selector: 'app-stock-ticker',
  template: `
    @let tick = currentTick();
    @if (tick) {
      <div class="ticker" [class.up]="tick.change > 0" [class.down]="tick.change < 0">
        <span class="symbol">{{ tick.symbol }}</span>
        <span class="price">{{ tick.price | currency }}</span>
        <span class="change">{{ tick.change | number:'1.2-2' }}%</span>
      </div>
    } @else {
      <app-spinner />
    }
  `,
})
export class StockTickerComponent {
  symbol = input.required<string>();
  private feedService = inject(StockFeedService);

  // Observable (WebSocket) -> Signal (template)
  currentTick = toSignal(
    toObservable(toSignal(toObservable(this.symbol), { requireSync: true })).pipe(
      switchMap(sym => this.feedService.getPriceFeed(sym))
    )
  );
}
```

A simpler version using `computed()` + `rxResource()`:

```typescript
@Component({
  selector: 'app-stock-ticker',
  template: `
    @let tick = tickResource.value();
    @if (tick) {
      <div class="ticker">{{ tick.symbol }} {{ tick.price | currency }}</div>
    }
  `,
})
export class StockTickerComponent {
  symbol = input.required<string>();
  private feedService = inject(StockFeedService);

  tickResource = rxResource({
    request: () => this.symbol(),
    loader: ({ request: sym }) => this.feedService.getPriceFeed(sym),
  });
}
```

### Search with Debounce and Error Recovery

```typescript
@Component({
  selector: 'app-global-search',
  template: `
    <input
      [value]="query()"
      (input)="query.set(getValue($event))"
      placeholder="Search..."
    />

    @switch (resultsResource.status()) {
      @case ('loading') { <app-spinner /> }
      @case ('error') { <p class="error">Search failed. Please try again.</p> }
      @case ('resolved') {
        @let items = resultsResource.value()!;
        @if (items.length) {
          @for (item of items; track item.id) {
            <app-search-result [item]="item" />
          }
        } @else {
          <p>No results for "{{ query() }}"</p>
        }
      }
    }
  `,
})
export class GlobalSearchComponent {
  private http = inject(HttpClient);
  query = signal('');

  // Signal -> Observable (for debounce) -> back to Signal (via rxResource)
  private debouncedQuery = toSignal(
    toObservable(this.query).pipe(
      debounceTime(300),
      distinctUntilChanged(),
    ),
    { initialValue: '' }
  );

  resultsResource = rxResource({
    request: () => this.debouncedQuery(),
    loader: ({ request: q }) => {
      if (q.length < 2) return of([]);
      return this.http.get<SearchResult[]>(`/api/search?q=${q}`).pipe(
        retry({ count: 2, delay: 1000 }),
        catchError(() => of([])),
      );
    },
  });

  getValue(event: Event): string {
    return (event.target as HTMLInputElement).value;
  }
}
```

### Parallel Data Loading with forkJoin

```typescript
@Component({
  selector: 'app-admin-dashboard',
  template: `
    @if (dashboardData(); as data) {
      <app-stats-cards [stats]="data.stats" />
      <app-recent-orders [orders]="data.orders" />
      <app-user-activity [activity]="data.activity" />
    } @else {
      <app-skeleton-dashboard />
    }
  `,
})
export class AdminDashboardComponent {
  private http = inject(HttpClient);

  // forkJoin: load all dashboard data in parallel, wait for all
  dashboardData = toSignal(
    forkJoin({
      stats: this.http.get<Stats>('/api/admin/stats'),
      orders: this.http.get<Order[]>('/api/admin/recent-orders'),
      activity: this.http.get<Activity[]>('/api/admin/activity'),
    }).pipe(
      catchError(error => {
        console.error('Dashboard load failed:', error);
        return of(null);
      })
    ),
    { initialValue: null }
  );
}
```

---

> **Key Takeaway:** Signals and RxJS are complementary: use signals for synchronous state and derivations, use observables for complex async coordination (debounce, retry, WebSockets, parallel requests), and bridge them with `toSignal()`, `toObservable()`, and `rxResource()` at component boundaries.
