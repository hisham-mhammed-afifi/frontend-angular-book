# 11.19 Angular DevTools

> Angular DevTools is a browser extension that lets you inspect component trees, profile change detection, debug dependency injection, and monitor signal state directly in your browser's developer tools.

---

## Installation and Setup

Angular DevTools is available as a browser extension for Chrome, Firefox, and Edge.

**Install from:**
- Chrome Web Store: search "Angular DevTools"
- Firefox Add-ons: search "Angular DevTools"
- Edge Add-ons: search "Angular DevTools" (or install the Chrome version)

Once installed, open the browser's developer tools (`F12` or `Ctrl+Shift+I` on Windows/Linux, `Cmd+Option+I` on macOS) and look for the **Angular** tab.

### Requirements

- Angular v12 or higher (v17+ for injector tree, v20+ for Chrome Performance panel integration)
- **Development builds only.** Angular DevTools does not work with production builds because Angular strips out the debug hooks at build time.

```bash
# DevTools works with the dev server
ng serve

# DevTools does NOT work with production builds
ng build
# Serving the dist/ output will show "Angular application not detected"
```

If you see the message "We detected an application built with production configuration" or "Angular application not detected," you are either viewing a production build or a non-Angular page.

## The Three Tabs

Angular DevTools has three main tabs:

| Tab | Purpose | Angular Version |
|-----|---------|-----------------|
| **Components** | Inspect component tree, view/edit properties, inputs, outputs, signal values | v12+ |
| **Profiler** | Record and analyze change detection cycles, identify performance bottlenecks | v12+ |
| **Injector Tree** | Visualize the dependency injection hierarchy and provider resolution paths | v17+ |

---

## Component Explorer

The Components tab shows your application's component tree in the left panel. Selecting a component reveals its details in the right panel.

### Navigating the Tree

The component tree mirrors your application's DOM hierarchy. Each node shows the component's selector name. You can:

- **Expand/collapse** nodes to explore the hierarchy
- **Search** for components by name using the filter bar at the top
- **Select** a component to view its details in the right panel
- **Right-click** a component and choose "Scroll into view" to find it on the page

### Inspecting Component State

When you select a component, the right panel shows:

- **Properties**: All public and private fields of the component class
- **Inputs**: Current values of all `input()` / `@Input()` bindings
- **Outputs**: Registered `output()` / `@Output()` emitters
- **Signal values**: Current values of `signal()`, `computed()`, `linkedSignal()` instances
- **View providers and providers**: Services scoped to this component

```typescript
// Given this component:
@Component({
  selector: 'app-product-card',
  providers: [PriceCalculatorService],
  template: `...`,
})
export class ProductCardComponent {
  product = input.required<Product>();
  showDetails = input(false);
  addedToCart = output<Product>();

  quantity = signal(1);
  totalPrice = computed(() => this.product().price * this.quantity());
  isInStock = signal(true);
}

// In DevTools, selecting <app-product-card> shows:
// Inputs:
//   product: { id: '123', name: 'Widget', price: 29.99 }
//   showDetails: false
//
// Outputs:
//   addedToCart: OutputEmitterRef
//
// Properties:
//   quantity: Signal(1)
//   totalPrice: Signal(29.99)
//   isInStock: Signal(true)
```

### Editing Properties at Runtime

You can modify component properties directly in DevTools to test behavior:

1. Select a component in the tree
2. Find the property in the right panel
3. Right-click the value and select "Edit value" (or double-click if supported)
4. Type the new value and press Enter

This triggers change detection and updates the view immediately. Useful for testing edge cases (what happens if `quantity` is `0`? What if `product.name` is an empty string?).

### Selecting an Element on the Page

Use the "Select an element" button (crosshair icon) in the Angular DevTools toolbar. Click any element on the page, and DevTools navigates to that component in the tree. This works like the standard "Select element" in Chrome's Elements tab, but jumps to the Angular component.

### Console Access

When you select a component in DevTools, it becomes accessible as `$ng0` in the browser console:

```javascript
// In the browser console after selecting a component:
$ng0
// Returns the component instance

$ng0.quantity()
// Reads the signal value: 1

$ng0.quantity.set(5)
// Updates the signal, triggers change detection
```

This lets you call methods, set signals, and interact with the component programmatically during debugging.

## Profiler

The Profiler tab helps you understand how long Angular spends in change detection and which components are the most expensive.

### Recording a Profile

1. Open the **Profiler** tab
2. Click the **Record** button (circle icon)
3. Interact with your application (click buttons, navigate, type in forms)
4. Click the **Stop** button
5. Analyze the recorded change detection cycles

### Understanding the Timeline

After recording, you see a **timeline bar chart** at the top. Each bar represents one change detection cycle.

- **Height** of the bar = total time spent in that cycle
- **Color** indicates severity: green (fast), yellow (moderate), red (slow)
- Click a bar to see details about that cycle

### Bar Chart View

When you select a cycle, a bar chart appears showing every component checked during that cycle:

- Each bar represents a component
- The width indicates how much time Angular spent checking that component
- Components are ordered by render time (slowest first)
- The display shows both the component name and the time in milliseconds

```
// Example profiler output for a selected cycle:
// Change detection cycle #14
// Total time: 12.3ms
// Source: Timer (setInterval)
//
// Component breakdown:
//   DataTableComponent     - 8.2ms
//   ChartComponent         - 2.1ms
//   HeaderComponent        - 0.8ms
//   SidebarComponent       - 0.6ms
//   FooterComponent        - 0.4ms
```

This tells you `DataTableComponent` is the bottleneck consuming 66% of the cycle time.

### Flame Graph View

Switch to the flame graph view for a hierarchical visualization:

- **X-axis**: Time span of the change detection cycle
- **Y-axis**: Component hierarchy (parent at top, children below)
- **Tile width**: How long Angular spent in that component
- **Color intensity**: Relative time spent (darker = slower)

The flame graph is useful for seeing which branches of the component tree are expensive and whether the cost is concentrated at the top (parent) or spread across many children.

### What the Profiler Tells You

| Observation | Likely Issue | Fix |
|-------------|-------------|-----|
| Many bars per interaction | Too many change detection cycles | Move to signals, use `OnPush` strategy for legacy components |
| One component dominates | Expensive template expression or heavy computation | Move logic to `computed()`, avoid function calls in templates |
| Cycles with no user interaction | Uncontrolled `setInterval`, WebSocket, or observable emissions | Use `takeUntilDestroyed()`, reduce polling frequency |
| Every component checked every cycle | Zone.js global change detection | Migrate to zoneless with `provideZonelessChangeDetection()` |

### Import and Export Profiles

You can export a recorded profile as JSON and import it later or share it with teammates. This is useful for:

- Comparing performance before and after optimization
- Sharing profiling data in bug reports
- Reproducing performance issues across machines

## Injector Tree

The Injector Tree tab (Angular 17+) visualizes the dependency injection hierarchy. It shows two parallel trees:

### Environment Injector Tree

Shows the application-level injectors:

- **Platform Injector**: Providers shared across multiple apps on the page
- **Root Injector**: Providers from `bootstrapApplication()` and `providedIn: 'root'` services
- **Route Injectors**: Providers from route-level `providers` arrays

### Element Injector Tree

Shows the component-level injectors:

- Each component with a `providers` or `viewProviders` array appears as a node
- Child components inherit from parent injectors

### Inspecting a Provider

Click on any injector node to see:

- All tokens registered at that level
- The provider type (class, value, factory, existing)
- The resolution path Angular follows to satisfy a dependency

This is invaluable for debugging:

- **"Why is my service a different instance?"** Check if a component provides it locally, creating a new instance instead of using the root singleton.
- **"Where is my provider registered?"** Search for the token in the injector tree to find which level provides it.
- **"Why do I get NullInjectorError?"** Walk the tree to see which injectors Angular checks and where the search fails.

```typescript
// Scenario: Two components inject LoggerService but get different instances
@Component({
  selector: 'app-panel-a',
  providers: [LoggerService], // Panel A has its own LoggerService
})
export class PanelAComponent {
  logger = inject(LoggerService);
}

@Component({
  selector: 'app-panel-b',
  // No providers array; gets LoggerService from root
})
export class PanelBComponent {
  logger = inject(LoggerService);
}

// The Injector Tree makes this visible:
// Root Injector
//   - LoggerService (providedIn: 'root')
//
// PanelAComponent Element Injector
//   - LoggerService (overrides root)
//
// PanelBComponent Element Injector
//   - (empty, inherits from root)
```

## Chrome Performance Panel Integration

Starting with Angular 20, Angular instruments the Chrome DevTools Performance panel with framework-specific traces. This gives you Angular context directly in Chrome's built-in profiling tool.

### Enabling Angular Profiling

```typescript
// Option 1: In your application code (during development)
import { enableProfiling } from '@angular/core';
enableProfiling();

// Option 2: In the browser console at runtime
ng.enableProfiling();
```

### What You See

After enabling and recording a performance trace in Chrome's Performance tab:

- A dedicated **Angular** track appears alongside the standard Main, Network, and Frames tracks
- Angular-specific entries include:
  - Component rendering (which component, how long)
  - Change detection cycles
  - Lifecycle hook execution
  - Template rendering time

### Benefits Over Angular DevTools Profiler

| Aspect | Angular DevTools Profiler | Chrome Performance Panel |
|--------|--------------------------|-------------------------|
| Scope | Change detection only | Full application: JS execution, layout, paint, Angular internals |
| Correlation | Standalone timeline | Correlated with network requests, layout shifts, long tasks |
| Overhead | Moderate | Minimal (uses zero-cost extensibility API when not recording) |
| Production | No | No (development builds only) |
| Export | JSON (DevTools format) | Standard Chrome trace format |

The Chrome Performance panel integration is best for diagnosing issues where Angular performance interacts with browser layout, painting, or network timing. The Angular DevTools Profiler is best for focused change detection analysis.

## Debugging Techniques with DevTools

### Technique 1: Finding Components That Render Too Often

1. Open the Profiler tab
2. Record while performing a single user action (e.g., clicking a button)
3. Count how many change detection cycles were triggered
4. Check if components unrelated to the action were re-checked

If unrelated components appear in every cycle, they are being checked unnecessarily. Solutions:

```typescript
// Solution 1: Use signals (zoneless detects changes per-component)
// Before: zone-based, entire tree checked
count = 0;
increment() { this.count++; }

// After: signal-based, only this component re-renders
count = signal(0);
increment() { this.count.update(n => n + 1); }

// Solution 2: Use OnPush change detection (for legacy code)
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,
  // ...
})
export class LegacyComponent {}
```

### Technique 2: Debugging "NullInjectorError"

When you see `NullInjectorError: No provider for SomeService`:

1. Open the Injector Tree tab
2. Search for `SomeService` in the token search
3. If not found anywhere, the service is missing a provider
4. If found at a level above the requesting component, check for `self: true` flags

```typescript
// Error: NullInjectorError: No provider for FeatureService
// Check: Is FeatureService providedIn: 'root'?
// Check: Is it in a route's providers array?
// Check: Is it in the component's providers array?

// Fix: Add providedIn or register in providers
@Injectable({ providedIn: 'root' }) // Simplest fix
export class FeatureService {}
```

### Technique 3: Identifying Expensive Template Expressions

1. Record a profile with the Profiler
2. Find the component with the longest render time
3. Open the component's template and look for:
   - Function calls: `{{ getFilteredItems() }}` (runs every cycle)
   - Complex pipes: chained or impure pipes
   - Large `@for` loops without proper `track`

```typescript
// BAD: Function in template re-runs on every check
@Component({
  template: `
    @for (item of getExpensiveList(); track item.id) {
      <app-item [data]="item" />
    }
  `,
})
export class ListComponent {
  getExpensiveList() {
    return this.items.filter(/* ... */).sort(/* ... */); // Runs every cycle
  }
}

// GOOD: Computed signal, only recomputes when dependencies change
@Component({
  template: `
    @for (item of expensiveList(); track item.id) {
      <app-item [data]="item" />
    }
  `,
})
export class ListComponent {
  private items = signal<Item[]>([]);
  expensiveList = computed(() =>
    this.items().filter(/* ... */).sort(/* ... */)
  );
}
```

### Technique 4: Verifying Signal Dependencies

In the Components tab, select a component and check its signal values. Change an input to the component and verify:

- The signal values update as expected
- `computed()` signals derive correct values
- No stale values remain

```typescript
// If you see a computed signal stuck at an old value:
// 1. Check if the source signals are updating
// 2. Check if the computed reads the signals inside its function
// 3. Check for untracked() calls that might exclude dependencies
```

## ng.getComponent() and Console Utilities

Angular exposes global debugging utilities on the `ng` namespace in development builds.

```javascript
// In the browser console:

// Get a component instance from a DOM element
const el = document.querySelector('app-header');
const component = ng.getComponent(el);
console.log(component);

// Get a directive instance
const directives = ng.getDirectives(el);

// Get the owning component of a nested element
const owner = ng.getOwningComponent(someNestedElement);

// Trigger change detection manually
ng.applyChanges(component);

// Enable profiling for Chrome Performance panel
ng.enableProfiling();
```

These utilities work without Angular DevTools installed and are useful for quick debugging sessions or automated testing in the console.

## Common Mistakes

### 1. Trying to Use DevTools on Production Builds

Angular strips debug information from production builds. There is no way around this. If you need to profile a production-like app, build with:

```bash
# Development build with production-like optimizations but debug info intact
ng build --configuration development
```

### 2. Ignoring the Profiler During Development

Many developers only profile when they notice performance issues. By then, the problem may be deeply embedded. Profile regularly, especially after adding new features.

### 3. Not Using the Injector Tree for DI Debugging

When facing `NullInjectorError` or unexpected service instances, developers often resort to `console.log` and trial-and-error. The Injector Tree tab shows the exact resolution path and makes DI issues immediately visible.

### 4. Confusing Angular DevTools Profiler with Chrome Performance Tab

The Angular DevTools Profiler focuses on change detection cycles and component-level timing. The Chrome Performance tab shows the full picture (JavaScript execution, layout, paint, compositing). Use both together for comprehensive analysis.

### 5. Not Enabling Chrome Performance Panel Integration

The Angular-specific track in Chrome's Performance panel (Angular 20+) provides the most detailed performance data with minimal overhead. Many developers do not know about `ng.enableProfiling()` and miss this feature.

## Best Practices

### 1. Make Profiling Part of Your Workflow

Record a quick profile after any significant UI change. Check that:
- Only expected components re-render
- No new performance regressions appeared
- Change detection cycle count is reasonable for the interaction

### 2. Use the Component Explorer to Verify Data Flow

Before writing console.log statements, check the component's state in DevTools. You can see inputs, signal values, and computed results without modifying code.

### 3. Use $ng0 for Interactive Debugging

Select a component in DevTools, then switch to the Console tab. `$ng0` gives you direct access to the component instance. Call methods, update signals, and observe the effects in real time.

### 4. Export and Share Profiles

When reporting performance issues to your team, export the profile JSON. Include it in the issue tracker so others can import it and see exactly what you saw.

### 5. Combine Angular DevTools with Chrome Performance Panel

For deep performance analysis:
1. Enable Angular profiling: `ng.enableProfiling()`
2. Record in Chrome's Performance tab (not Angular DevTools Profiler)
3. Analyze both the Angular track (component timing) and the standard traces (layout, paint, network)

### 6. Use the Injector Tree Before Adding Random providers Arrays

When a service is not resolving correctly, inspect the Injector Tree first. Understand where Angular is looking before adding providers to random components.

## Real-World Use Cases

### Diagnosing a Slow Table Rendering

**Symptom:** A data table with 500 rows takes 800ms to render after filtering.

**Investigation:**
1. Open Angular DevTools Profiler, record while applying a filter
2. See a single change detection cycle taking 800ms
3. Click the cycle bar; `DataTableComponent` takes 780ms
4. Open the template: `@for (row of getFilteredRows(); track $index)`

**Root causes found:**
- `getFilteredRows()` is a method call (re-runs every check)
- `track $index` causes full DOM recreation when order changes
- Each row component performs its own date formatting in the template

**Fixes:**

```typescript
// Before (slow)
@Component({
  template: `
    @for (row of getFilteredRows(); track $index) {
      <tr>
        <td>{{ formatDate(row.date) }}</td>
        <td>{{ row.name }}</td>
      </tr>
    }
  `,
})
export class DataTableComponent {
  getFilteredRows() { /* expensive filter + sort */ }
  formatDate(d: Date) { return new Intl.DateTimeFormat('en').format(d); }
}

// After (fast)
@Component({
  template: `
    @for (row of filteredRows(); track row.id) {
      <tr>
        <td>{{ row.date | date:'mediumDate' }}</td>
        <td>{{ row.name }}</td>
      </tr>
    }
  `,
})
export class DataTableComponent {
  private rows = input.required<Row[]>();
  private filter = signal('');

  filteredRows = computed(() => {
    const term = this.filter().toLowerCase();
    return this.rows()
      .filter(r => r.name.toLowerCase().includes(term))
      .sort((a, b) => a.name.localeCompare(b.name));
  });
}
```

Re-profile: cycle drops from 800ms to 15ms.

### Debugging an Unexpected Service Instance

**Symptom:** `AuthService.isAuthenticated()` returns `false` in a child component, but the user is logged in.

**Investigation:**
1. Open the Injector Tree tab
2. Search for `AuthService`
3. Find it is provided at:
   - Root Injector (the correct singleton)
   - `AdminLayoutComponent`'s element injector (an accidental duplicate)

**Root cause:** `AdminLayoutComponent` has `providers: [AuthService]`, creating a new instance that has never seen the login action.

**Fix:**

```typescript
// Before (broken): Component provides its own AuthService
@Component({
  selector: 'app-admin-layout',
  providers: [AuthService], // Remove this!
  template: `...`,
})
export class AdminLayoutComponent {}

// After (fixed): Use the root singleton
@Component({
  selector: 'app-admin-layout',
  // No providers array; inherits AuthService from root
  template: `...`,
})
export class AdminLayoutComponent {}
```

### Identifying Unnecessary Change Detection Cycles

**Symptom:** The app feels sluggish during idle periods.

**Investigation:**
1. Open Angular DevTools Profiler
2. Record for 10 seconds without any user interaction
3. See 150+ change detection cycles triggered by a `setInterval`

**Root cause:** A forgotten polling service runs every 100ms and triggers zone-based change detection each time.

**Fix:**

```typescript
// Before (zone-aware polling triggers CD every 100ms)
@Injectable({ providedIn: 'root' })
export class StatusPollingService {
  constructor() {
    setInterval(() => {
      fetch('/api/status').then(r => r.json()).then(data => {
        this.status = data; // Triggers zone.js change detection
      });
    }, 100);
  }
}

// After: Use signals + resource with reasonable interval
@Injectable({ providedIn: 'root' })
export class StatusPollingService {
  private _status = signal<Status | null>(null);
  readonly status = this._status.asReadonly();

  constructor() {
    // Poll every 30 seconds, not 100ms
    effect((onCleanup) => {
      const id = setInterval(async () => {
        const res = await fetch('/api/status');
        this._status.set(await res.json());
      }, 30_000);
      onCleanup(() => clearInterval(id));
    });
  }
}
```

Or better, use the resource API which handles cleanup automatically:

```typescript
@Injectable({ providedIn: 'root' })
export class StatusPollingService {
  private tick = signal(0);

  constructor() {
    // Increment tick every 30s to trigger resource reload
    effect((onCleanup) => {
      const id = setInterval(() => this.tick.update(n => n + 1), 30_000);
      onCleanup(() => clearInterval(id));
    });
  }

  statusResource = resource({
    request: () => this.tick(),
    loader: async ({ abortSignal }) => {
      const res = await fetch('/api/status', { signal: abortSignal });
      return res.json() as Promise<Status>;
    },
  });
}
```

Re-profile: idle change detection cycles drop to zero (zoneless) or only when signal-driven.

> **Key Takeaway:** Angular DevTools gives you three essential debugging tools: the Component Explorer for inspecting state and signal values, the Profiler for identifying change detection bottlenecks, and the Injector Tree for understanding dependency resolution. Combined with Chrome's Performance panel integration (`ng.enableProfiling()`), these tools make performance and DI issues visible without resorting to console.log.
