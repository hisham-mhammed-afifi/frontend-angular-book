# 11.3 Standalone Components Architecture

> Standalone components are the default in Angular 21. They eliminate the need for NgModules by declaring their own dependencies directly, resulting in simpler, more self-contained, and more tree-shakable code.

---

## What Are Standalone Components?

A standalone component is a component that manages its own template dependencies. Instead of being declared inside an NgModule and relying on that module's `imports` and `declarations` arrays, a standalone component lists everything it needs in its own `imports` array.

```typescript
import { Component } from '@angular/core';
import { RouterOutlet } from '@angular/router';
import { HeaderComponent } from './layout/header.component';

@Component({
  selector: 'app-root',
  imports: [RouterOutlet, HeaderComponent],
  template: `
    <app-header />
    <router-outlet />
  `
})
export class AppComponent {}
```

There is no `standalone: true` property needed in Angular 21. Components are standalone by default. In older Angular versions (14-18), you had to explicitly mark `standalone: true` in the decorator. In Angular 19+, the default flipped: all components, directives, and pipes are standalone unless you explicitly set `standalone: false`.

### The Old Way (NgModules)

Before standalone components, you had to:

1. Create the component.
2. Create or find an NgModule.
3. Add the component to the module's `declarations` array.
4. Import the module wherever the component was needed.

```typescript
// Legacy: NgModule-based approach
@NgModule({
  declarations: [
    UserListComponent,
    UserDetailComponent,
    UserFormComponent,
  ],
  imports: [
    CommonModule,
    FormsModule,
    RouterModule.forChild(routes),
  ],
  exports: [UserListComponent],
})
export class UsersModule {}
```

### The New Way (Standalone)

Each component declares its own dependencies. No module wrapper needed.

```typescript
// Modern: standalone component
@Component({
  selector: 'app-user-list',
  imports: [DatePipe, RouterLink, UserCardComponent],
  templateUrl: './user-list.component.html',
})
export class UserListComponent {
  private userService = inject(UserService);
  users = this.userService.getUsers();
}
```

The component is self-contained. You can see exactly what it depends on by looking at its `imports` array.

## The imports Array

The `imports` array on a standalone component accepts:

1. **Other standalone components**
2. **Standalone directives**
3. **Standalone pipes**
4. **NgModules** (for backward compatibility)

```typescript
import { Component } from '@angular/core';
import { NgClass } from '@angular/common';
import { FormsModule } from '@angular/forms';
import { RouterLink } from '@angular/router';
import { ButtonComponent } from '@shared/components/button.component';
import { TruncatePipe } from '@shared/pipes/truncate.pipe';
import { TooltipDirective } from '@shared/directives/tooltip.directive';

@Component({
  selector: 'app-product-card',
  imports: [
    // Standalone directive from Angular
    NgClass,
    // NgModule (legacy interop)
    FormsModule,
    // Standalone component
    ButtonComponent,
    // Standalone pipe
    TruncatePipe,
    // Standalone directive
    TooltipDirective,
    // Router directive
    RouterLink,
  ],
  template: `
    <div [ngClass]="{ 'featured': product().featured }">
      <h3>{{ product().name | truncate:50 }}</h3>
      <p appTooltip="Click for details">
        {{ product().description }}
      </p>
      <app-button (clicked)="addToCart()">Add to Cart</app-button>
      <a [routerLink]="['/products', product().id]">Details</a>
    </div>
  `
})
export class ProductCardComponent {
  product = input.required<Product>();
  addedToCart = output<Product>();

  addToCart() {
    this.addedToCart.emit(this.product());
  }
}
```

### Importing Individual Items vs. Entire Modules

You can import either individual standalone items or entire NgModules. Prefer individual imports for better tree shaking.

```typescript
// Bad Practice: importing entire CommonModule when you only need NgClass
@Component({
  imports: [CommonModule], // Pulls in everything: NgIf, NgFor, AsyncPipe, etc.
  template: `<div [ngClass]="classes">...</div>`
})
export class MyComponent {}

// Good Practice: import only what you use
@Component({
  imports: [NgClass],
  template: `<div [ngClass]="classes">...</div>`
})
export class MyComponent {}
```

In Angular 21 with the new control flow (`@if`, `@for`, `@switch`), you rarely need `NgIf`, `NgFor`, or `NgSwitch` imports at all. The new control flow syntax is built into the template compiler and requires no imports.

```typescript
// Angular 21: no imports needed for control flow
@Component({
  imports: [], // No NgIf, NgFor needed
  template: `
    @if (users().length > 0) {
      @for (user of users(); track user.id) {
        <app-user-card [user]="user" />
      }
    } @else {
      <p>No users found.</p>
    }
  `
})
export class UserListComponent {
  users = input.required<User[]>();
}
```

## Standalone Directives

Directives follow the same standalone pattern. They are standalone by default in Angular 21.

```typescript
import { Directive, ElementRef, inject, input, effect } from '@angular/core';

@Directive({
  selector: '[appHighlight]'
})
export class HighlightDirective {
  private el = inject(ElementRef);
  color = input('yellow', { alias: 'appHighlight' });

  constructor() {
    effect(() => {
      this.el.nativeElement.style.backgroundColor = this.color();
    });
  }
}
```

Use in a component by adding it to `imports`:

```typescript
@Component({
  imports: [HighlightDirective],
  template: `<p appHighlight="lightblue">Highlighted text</p>`
})
export class MyComponent {}
```

## Standalone Pipes

Pipes are also standalone by default.

```typescript
import { Pipe, PipeTransform } from '@angular/core';

@Pipe({
  name: 'truncate'
})
export class TruncatePipe implements PipeTransform {
  transform(value: string, maxLength: number = 100): string {
    if (!value) return '';
    if (value.length <= maxLength) return value;
    return value.substring(0, maxLength) + '...';
  }
}
```

Use in a component:

```typescript
@Component({
  imports: [TruncatePipe],
  template: `<p>{{ description() | truncate:200 }}</p>`
})
export class ArticleCardComponent {
  description = input.required<string>();
}
```

## Dependency Injection Without NgModules

With standalone components, dependency injection works at three levels:

### 1. Application-Level Providers (Singletons)

Registered in `app.config.ts` via `bootstrapApplication`. Available everywhere.

```typescript
// app.config.ts
import { ApplicationConfig, provideZonelessChangeDetection } from '@angular/core';
import { provideRouter } from '@angular/router';
import { provideHttpClient } from '@angular/common/http';
import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
    provideRouter(routes),
    provideHttpClient(),
  ]
};
```

Services with `providedIn: 'root'` are also application-level singletons, but they do not need to be listed in `app.config.ts`. They are tree-shakable: if no component injects them, they are excluded from the bundle.

```typescript
@Injectable({ providedIn: 'root' })
export class AuthService {
  // Singleton, tree-shakable, available everywhere
}
```

### 2. Route-Level Providers (Environment Injectors)

Registered in route configurations. Available to all components, directives, guards, and resolvers under that route. This replaces the old pattern of providing services in a lazy-loaded NgModule.

```typescript
// features/users/users.routes.ts
import { Routes } from '@angular/router';
import { UserService } from './user.service';

export const USERS_ROUTES: Routes = [
  {
    path: '',
    providers: [UserService],
    children: [
      {
        path: '',
        loadComponent: () =>
          import('./user-list.component').then(m => m.UserListComponent),
      },
      {
        path: ':id',
        loadComponent: () =>
          import('./user-detail.component').then(m => m.UserDetailComponent),
      },
    ],
  },
];
```

`UserService` is instantiated once when the route is activated. All components under this route share the same instance. When the user navigates away, the instance is destroyed (if the route is no longer active and the injector is garbage collected).

### 3. Component-Level Providers

Registered in a component's `providers` array. Each component instance gets its own instance of the service.

```typescript
@Component({
  selector: 'app-user-form',
  providers: [FormValidationService],
  template: `...`
})
export class UserFormComponent {
  private validation = inject(FormValidationService);
  // Each UserFormComponent instance gets its own FormValidationService
}
```

This is useful when you need isolated state per component instance, such as form state, undo/redo stacks, or local caches.

### Provider Scope Comparison

| Scope | Where Registered | Lifetime | Use Case |
|---|---|---|---|
| Application | `providedIn: 'root'` or `app.config.ts` | App lifetime | Auth, HTTP, global state |
| Route | Route `providers` array | Route activation | Feature services, feature state |
| Component | Component `providers` array | Component instance | Per-instance state, isolated services |

## Lazy Loading Standalone Components

Standalone components enable two lazy loading patterns without NgModules.

### loadComponent: Lazy Load a Single Component

```typescript
export const routes: Routes = [
  {
    path: 'dashboard',
    loadComponent: () =>
      import('./features/dashboard/dashboard.component')
        .then(m => m.DashboardComponent),
  },
];
```

The `DashboardComponent` is in its own chunk. It is downloaded only when the user navigates to `/dashboard`.

### loadChildren: Lazy Load a Route Group

```typescript
export const routes: Routes = [
  {
    path: 'users',
    loadChildren: () =>
      import('./features/users/users.routes')
        .then(m => m.USERS_ROUTES),
  },
];
```

The route file exports an array of `Routes`:

```typescript
// features/users/users.routes.ts
import { Routes } from '@angular/router';

export const USERS_ROUTES: Routes = [
  {
    path: '',
    loadComponent: () =>
      import('./user-list.component').then(m => m.UserListComponent),
  },
  {
    path: ':id',
    loadComponent: () =>
      import('./user-detail.component').then(m => m.UserDetailComponent),
  },
];
```

No NgModule. No `RouterModule.forChild()`. Just an array of routes with lazy-loaded standalone components.

### Default Exports Shorthand

If your component uses a default export, Angular's router unwraps it automatically:

```typescript
// user-list.component.ts
@Component({ /* ... */ })
export default class UserListComponent {}

// In routes:
{
  path: '',
  loadComponent: () => import('./user-list.component'),
  // No .then() needed
}
```

## Interoperability with NgModules

Standalone components can coexist with NgModule-based code. This is critical for gradual migration.

### Using an NgModule in a Standalone Component

Add the module to the standalone component's `imports`:

```typescript
@Component({
  imports: [ReactiveFormsModule], // NgModule imported directly
  template: `
    <form [formGroup]="form">
      <input formControlName="name" />
    </form>
  `
})
export class UserFormComponent {
  form = new FormGroup({
    name: new FormControl(''),
  });
}
```

### Using a Standalone Component in an NgModule

Add the standalone component to the module's `imports` (not `declarations`):

```typescript
// Legacy module that needs to use a standalone component
@NgModule({
  declarations: [LegacyPageComponent],
  imports: [
    CommonModule,
    NewStandaloneButtonComponent, // Standalone component goes in imports
  ],
})
export class LegacyModule {}
```

### importProvidersFrom for Legacy Module Providers

Some NgModules provide services through their `forRoot()` or `forChild()` methods. To use these in a standalone app:

```typescript
// app.config.ts
import { importProvidersFrom } from '@angular/core';
import { SomeLegacyModule } from 'legacy-library';

export const appConfig: ApplicationConfig = {
  providers: [
    importProvidersFrom(SomeLegacyModule.forRoot({ apiKey: '...' })),
  ]
};
```

`importProvidersFrom` extracts the providers from an NgModule and registers them in the application injector. It is only used in `bootstrapApplication` or route `providers`, never inside a component's `providers` array.

## Testing Standalone Components

Testing is simpler because you do not need to configure an NgModule in `TestBed`.

```typescript
import { ComponentFixture, TestBed } from '@angular/core/testing';
import { UserListComponent } from './user-list.component';
import { UserService } from './user.service';

describe('UserListComponent', () => {
  let component: UserListComponent;
  let fixture: ComponentFixture<UserListComponent>;

  beforeEach(async () => {
    await TestBed.configureTestingModule({
      // No declarations needed. The component manages its own imports.
      imports: [UserListComponent],
      providers: [
        { provide: UserService, useValue: mockUserService },
      ],
    }).compileComponents();

    fixture = TestBed.createComponent(UserListComponent);
    component = fixture.componentInstance;
  });

  it('should create', () => {
    expect(component).toBeTruthy();
  });
});
```

Notice: the standalone component goes in `imports`, not `declarations`. You can override its dependencies using `TestBed.overrideComponent` if needed.

### Overriding Imports in Tests

```typescript
TestBed.configureTestingModule({
  imports: [UserListComponent],
})
.overrideComponent(UserListComponent, {
  remove: { imports: [RealApiService] },
  add: { imports: [MockApiService] },
});
```

## Component Composition Patterns

### Smart/Container and Dumb/Presentational

The pattern still applies with standalone components. The difference is that each component explicitly imports what it needs.

```typescript
// Smart component: handles data and logic
@Component({
  selector: 'app-user-page',
  imports: [UserListComponent, LoadingSpinnerComponent],
  template: `
    @if (loading()) {
      <app-loading-spinner />
    } @else {
      <app-user-list [users]="users()" (userSelected)="onSelect($event)" />
    }
  `
})
export class UserPageComponent {
  private userService = inject(UserService);

  users = signal<User[]>([]);
  loading = signal(true);

  constructor() {
    this.userService.getAll().subscribe(users => {
      this.users.set(users);
      this.loading.set(false);
    });
  }
}

// Dumb component: pure presentation, no services injected
@Component({
  selector: 'app-user-list',
  imports: [UserCardComponent],
  template: `
    @for (user of users(); track user.id) {
      <app-user-card [user]="user" (click)="userSelected.emit(user)" />
    }
  `
})
export class UserListComponent {
  users = input.required<User[]>();
  userSelected = output<User>();
}
```

### Composing Shared Dependencies

When many components in a feature share the same imports, you can create a shared array:

```typescript
// features/users/shared-imports.ts
import { DatePipe, CurrencyPipe } from '@angular/common';
import { RouterLink } from '@angular/router';
import { ButtonComponent } from '@shared/components/button.component';
import { AvatarComponent } from '@shared/components/avatar.component';

export const USER_SHARED_IMPORTS = [
  DatePipe,
  CurrencyPipe,
  RouterLink,
  ButtonComponent,
  AvatarComponent,
] as const;
```

```typescript
// features/users/user-list.component.ts
import { USER_SHARED_IMPORTS } from './shared-imports';

@Component({
  imports: [...USER_SHARED_IMPORTS, SortHeaderComponent],
  template: `...`
})
export class UserListComponent {}
```

This reduces repetition without reintroducing NgModules.

## How Standalone Affects Bundle Size

Standalone components improve tree shaking. With NgModules, if you import a module, everything declared and exported by that module is pulled into the bundle, even if you only use one component from it.

With standalone components, each import is explicit and direct. The bundler knows exactly which components, directives, and pipes are used.

```typescript
// NgModule approach: imports entire SharedModule
@NgModule({
  imports: [SharedModule], // Pulls in 30 components, 10 pipes, 5 directives
})
export class FeatureModule {}

// Standalone approach: imports only what is needed
@Component({
  imports: [ButtonComponent, TruncatePipe], // Only 2 items in the bundle
})
export class FeatureComponent {}
```

The result: smaller initial bundles and more effective code splitting.

## Common Mistakes

- **Importing `CommonModule` instead of specific items.** `CommonModule` re-exports `NgIf`, `NgFor`, `NgClass`, `NgStyle`, `AsyncPipe`, `DatePipe`, `CurrencyPipe`, and many more. In Angular 21 with built-in control flow, you rarely need any of these. Import individual items when you do.

- **Forgetting to add a component to the `imports` array.** The Angular compiler will report an error if you use a component selector in a template without importing it. Read the error message carefully: it will tell you exactly which import is missing.

- **Putting standalone components in `declarations` instead of `imports`.** When using a standalone component inside an NgModule (legacy code), it goes in the module's `imports` array, not `declarations`. Declaring a standalone component throws a compile error.

- **Using `importProvidersFrom` inside a component's `providers`.** `importProvidersFrom` is designed for `bootstrapApplication` and route `providers` only. Inside a component, provide services directly in the `providers` array.

- **Over-scoping services.** Not everything needs `providedIn: 'root'`. If a service is only used within one feature, provide it at the route level. If it is only used by one component, provide it at the component level. Over-scoping creates unnecessary singletons that live for the app's lifetime.

- **Creating "barrel modules" to group standalone imports.** Some developers create an NgModule purely to group common standalone imports together. This reintroduces the NgModule pattern you are trying to avoid. Instead, use a shared array of imports (see "Composing Shared Dependencies" above).

- **Not lazy loading features.** Standalone components make lazy loading trivial with `loadComponent` and `loadChildren`. If your entire app loads in one bundle, you are missing the primary performance benefit of standalone architecture.

---

## Best Practices

- **Import only what the template uses.** Each component's `imports` array should be a minimal, explicit list. The Angular Language Service in your IDE will auto-suggest missing imports.

- **Use the built-in control flow (`@if`, `@for`, `@switch`).** These require no imports and are more performant than the structural directive equivalents (`*ngIf`, `*ngFor`).

- **Prefer `providedIn: 'root'` for application-wide services.** It is tree-shakable and requires no registration in `app.config.ts`.

- **Use route-level providers for feature-scoped services.** This replaces the old pattern of providing services in a lazy-loaded NgModule's `providers` array.

- **Use `inject()` instead of constructor injection.** It is more concise, works in functions (guards, interceptors, resolvers), and plays well with standalone components.

- **Lazy load every feature.** Use `loadComponent` for single-page features and `loadChildren` for multi-page features.

- **Keep components small and focused.** Standalone architecture makes it easy to compose components. Avoid bloated components with large `imports` arrays. If the imports list is long, the component is probably doing too much.

## Real-World Use Cases

### 1. Feature Module Replacement

Old NgModule-based feature:

```typescript
// OLD: users.module.ts (deprecated pattern)
@NgModule({
  declarations: [UserListComponent, UserDetailComponent, UserFormComponent],
  imports: [CommonModule, ReactiveFormsModule, RouterModule.forChild(routes)],
  providers: [UserService],
})
export class UsersModule {}
```

Modern standalone equivalent:

```typescript
// NEW: users.routes.ts (no module needed)
import { Routes } from '@angular/router';
import { UserService } from './user.service';

export const USERS_ROUTES: Routes = [
  {
    path: '',
    providers: [UserService],
    children: [
      {
        path: '',
        loadComponent: () =>
          import('./user-list.component').then(m => m.UserListComponent),
      },
      {
        path: 'new',
        loadComponent: () =>
          import('./user-form.component').then(m => m.UserFormComponent),
      },
      {
        path: ':id',
        loadComponent: () =>
          import('./user-detail.component').then(m => m.UserDetailComponent),
      },
      {
        path: ':id/edit',
        loadComponent: () =>
          import('./user-form.component').then(m => m.UserFormComponent),
      },
    ],
  },
];
```

Each component is self-contained. `UserService` is scoped to the route. No module file at all.

### 2. Shared UI Library with Standalone Components

```typescript
// shared/components/confirm-dialog.component.ts
@Component({
  selector: 'app-confirm-dialog',
  imports: [ButtonComponent],
  template: `
    <div class="dialog-backdrop" (click)="cancel()">
      <div class="dialog" (click)="$event.stopPropagation()">
        <h2>{{ title() }}</h2>
        <p>{{ message() }}</p>
        <div class="actions">
          <app-button (clicked)="cancel()">Cancel</app-button>
          <app-button (clicked)="confirm()">{{ confirmLabel() }}</app-button>
        </div>
      </div>
    </div>
  `,
  styleUrl: './confirm-dialog.component.scss'
})
export class ConfirmDialogComponent {
  title = input('Confirm');
  message = input.required<string>();
  confirmLabel = input('OK');

  confirmed = output<void>();
  cancelled = output<void>();

  confirm() { this.confirmed.emit(); }
  cancel() { this.cancelled.emit(); }
}
```

Any feature can use this by adding `ConfirmDialogComponent` to its `imports`. No shared module needed.

### 3. Dynamic Component Loading

Standalone components work seamlessly with Angular's `ViewContainerRef` for dynamic loading:

```typescript
@Component({
  selector: 'app-widget-host',
  template: `<ng-container #container />`
})
export class WidgetHostComponent {
  @ViewChild('container', { read: ViewContainerRef })
  container!: ViewContainerRef;

  async loadWidget(widgetType: string) {
    this.container.clear();

    let component: Type<unknown>;

    switch (widgetType) {
      case 'chart':
        component = (await import('./widgets/chart-widget.component'))
          .ChartWidgetComponent;
        break;
      case 'table':
        component = (await import('./widgets/table-widget.component'))
          .TableWidgetComponent;
        break;
      default:
        return;
    }

    this.container.createComponent(component);
  }
}
```

Each widget is a standalone component. They are loaded on demand, not bundled upfront.

---

> **Key Takeaway:** Standalone components are the default and recommended architecture in Angular 21. Every component, directive, and pipe declares its own dependencies in its `imports` array, eliminating NgModules for dependency management. Combined with route-level providers and lazy loading via `loadComponent`/`loadChildren`, standalone architecture delivers smaller bundles, simpler code, and easier testing.
