# 11.4 Bootstrapping with `bootstrapApplication`

> `bootstrapApplication` is the entry point for every Angular 21 application. It replaces the legacy `platformBrowserDynamic().bootstrapModule()` pattern and wires up the root component, providers, and platform-level configuration in a single function call.

---

## How Angular Starts

Every Angular application starts with a single function call in `main.ts`. This function creates the platform (browser DOM bindings), sets up the root injector with all registered providers, instantiates the root component, and inserts it into the DOM at the location matched by the root component's selector.

```typescript
// main.ts (Angular 21)
import { bootstrapApplication } from '@angular/platform-browser';
import { appConfig } from './app/app.config';
import { AppComponent } from './app/app.component';

bootstrapApplication(AppComponent, appConfig)
  .catch((err) => console.error(err));
```

That is the entire bootstrap file. Two arguments:
1. **Root component**: The standalone component that serves as the shell of your application.
2. **Application config**: An `ApplicationConfig` object containing the `providers` array.

---

## The ApplicationConfig Object

`ApplicationConfig` is a simple interface with one property:

```typescript
interface ApplicationConfig {
  providers: Array<Provider | EnvironmentProviders>;
}
```

The config is typically defined in `app.config.ts`:

```typescript
// app.config.ts
import { ApplicationConfig, provideZonelessChangeDetection } from '@angular/core';
import { provideRouter } from '@angular/router';
import { provideHttpClient } from '@angular/common/http';
import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
    provideRouter(routes),
    provideHttpClient(),
  ]
};
```

Separating the config from `main.ts` keeps the bootstrap clean and makes the config reusable for testing or server-side rendering.

## The Legacy Approach (for Context)

Before standalone components, Angular applications bootstrapped through an NgModule:

```typescript
// main.ts (legacy)
import { platformBrowserDynamic } from '@angular/platform-browser-dynamic';
import { AppModule } from './app/app.module';

platformBrowserDynamic()
  .bootstrapModule(AppModule)
  .catch((err) => console.error(err));
```

```typescript
// app.module.ts (legacy)
@NgModule({
  declarations: [AppComponent],
  imports: [
    BrowserModule,
    HttpClientModule,
    AppRoutingModule,
    BrowserAnimationsModule,
  ],
  providers: [],
  bootstrap: [AppComponent],
})
export class AppModule {}
```

The module imported other modules that each carried their own providers and declarations. The migration to `bootstrapApplication` eliminates this entire layer.

| Aspect | Legacy (`bootstrapModule`) | Modern (`bootstrapApplication`) |
|---|---|---|
| Entry function | `platformBrowserDynamic().bootstrapModule()` | `bootstrapApplication()` |
| Package | `@angular/platform-browser-dynamic` | `@angular/platform-browser` |
| Root unit | NgModule (`AppModule`) | Standalone component (`AppComponent`) |
| Provider registration | Module `imports` and `providers` | `ApplicationConfig.providers` |
| Routing setup | `RouterModule.forRoot(routes)` | `provideRouter(routes)` |
| HTTP setup | `HttpClientModule` | `provideHttpClient()` |
| Animations | `BrowserAnimationsModule` | `provideAnimations()` |

## Provider Functions

Angular provides a set of `provide*` functions that replace the old NgModule imports. Each function returns an `EnvironmentProviders` object that configures a specific subsystem.

### Core Providers

#### provideZonelessChangeDetection()

The default in Angular 21. Configures change detection without Zone.js. Angular uses its own scheduling based on signals and explicit change detection triggers.

```typescript
import { provideZonelessChangeDetection } from '@angular/core';

export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
  ]
};
```

In Angular 21, new projects are zoneless by default. You do not need to explicitly add this provider unless you are migrating an older project. If your application still depends on Zone.js (typically because it uses legacy patterns that rely on automatic change detection from async operations like `setTimeout`, `addEventListener`, or raw Promises), use `provideZoneChangeDetection()` instead:

```typescript
import { provideZoneChangeDetection } from '@angular/core';

export const appConfig: ApplicationConfig = {
  providers: [
    provideZoneChangeDetection({ eventCoalescing: true }),
  ]
};
```

The `eventCoalescing: true` option reduces unnecessary change detection cycles by batching multiple DOM events into a single check.

### Router Providers

#### provideRouter(routes, ...features)

Replaces `RouterModule.forRoot()`. Registers the router and its dependencies.

```typescript
import { provideRouter } from '@angular/router';
import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
  ]
};
```

`provideRouter` accepts optional feature functions that enable additional router capabilities:

```typescript
import {
  provideRouter,
  withComponentInputBinding,
  withViewTransitions,
  withInMemoryScrolling,
  withPreloading,
  PreloadAllModules,
  withNavigationErrorHandler,
  withHashLocation,
  withRouterConfig,
} from '@angular/router';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(
      routes,
      // Bind route params, query params, and resolve data to component inputs
      withComponentInputBinding(),
      // Enable View Transitions API for route changes
      withViewTransitions(),
      // Restore scroll position on back navigation
      withInMemoryScrolling({
        scrollPositionRestoration: 'enabled',
        anchorScrolling: 'enabled',
      }),
      // Preload all lazy routes after initial load
      withPreloading(PreloadAllModules),
      // Handle navigation errors globally
      withNavigationErrorHandler((error) => {
        console.error('Navigation error:', error);
      }),
    ),
  ]
};
```

#### Router Feature Reference

| Feature | Purpose |
|---|---|
| `withComponentInputBinding()` | Binds route/query params and resolve data to component `input()` signals |
| `withViewTransitions()` | Enables the View Transitions API for animated route changes |
| `withInMemoryScrolling()` | Configures scroll position restoration and anchor scrolling |
| `withPreloading(strategy)` | Preloads lazy routes using a strategy (`PreloadAllModules`, custom) |
| `withNavigationErrorHandler(fn)` | Global error handler for navigation failures |
| `withHashLocation()` | Uses hash-based URLs (`/#/path`) instead of HTML5 pushState |
| `withRouterConfig(options)` | Configures `paramsInheritanceStrategy`, `onSameUrlNavigation`, etc. |
| `withDebugTracing()` | Logs all router events to the console (development only) |
| `withDisabledInitialNavigation()` | Prevents router from navigating on bootstrap |
| `withEnabledBlockingInitialNavigation()` | Blocks rendering until initial navigation completes (useful for SSR) |

### HTTP Providers

#### provideHttpClient(...features)

Replaces `HttpClientModule`. Configures the `HttpClient` service.

```typescript
import {
  provideHttpClient,
  withInterceptors,
  withFetch,
  withJsonpSupport,
  withRequestsMadeViaParent,
} from '@angular/common/http';
import { authInterceptor } from './core/auth.interceptor';
import { loggingInterceptor } from './core/logging.interceptor';

export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(
      // Register functional interceptors (order matters)
      withInterceptors([authInterceptor, loggingInterceptor]),
      // Use the Fetch API instead of XMLHttpRequest
      withFetch(),
    ),
  ]
};
```

| Feature | Purpose |
|---|---|
| `withInterceptors([...fns])` | Registers functional HTTP interceptors |
| `withInterceptorsFromDi()` | Uses class-based interceptors from DI (legacy interop) |
| `withFetch()` | Uses the Fetch API backend instead of `XMLHttpRequest` |
| `withJsonpSupport()` | Enables JSONP requests |
| `withRequestsMadeViaParent()` | Routes requests through the parent injector's `HttpClient` |
| `withNoXsrfProtection()` | Disables XSRF token handling |

### Animation Providers

#### provideAnimations()

Replaces `BrowserAnimationsModule`. Enables the Angular animations system.

```typescript
import { provideAnimations } from '@angular/platform-browser/animations';

export const appConfig: ApplicationConfig = {
  providers: [
    provideAnimations(),
  ]
};
```

If you do not use animations, skip this provider entirely. This keeps the animations module out of the bundle.

To disable animations (useful for testing or reduced motion preferences):

```typescript
import { provideNoopAnimations } from '@angular/platform-browser/animations';

export const appConfig: ApplicationConfig = {
  providers: [
    provideNoopAnimations(),
  ]
};
```

### SSR and Hydration Providers

#### provideClientHydration(...features)

Enables client-side hydration for server-side rendered applications.

```typescript
import { provideClientHydration, withEventReplay } from '@angular/platform-browser';

export const appConfig: ApplicationConfig = {
  providers: [
    provideClientHydration(
      withEventReplay(), // Replay user events that happened before hydration
    ),
  ]
};
```

## Application Initializers

Sometimes you need to run async setup before the application renders. Configuration loading, authentication token validation, and feature flag fetching are common examples.

### provideAppInitializer()

The modern way to register initializer functions. Replaces the `APP_INITIALIZER` injection token.

```typescript
import { provideAppInitializer, inject } from '@angular/core';
import { ConfigService } from './core/config.service';

export const appConfig: ApplicationConfig = {
  providers: [
    provideAppInitializer(() => {
      const configService = inject(ConfigService);
      return configService.load();
      // Angular waits for this Promise/Observable to complete
      // before rendering the application
    }),
  ]
};
```

The initializer function can return a `Promise`, an `Observable`, or `void`. If it returns a `Promise` or `Observable`, Angular waits for it to resolve before rendering.

### Multiple Initializers

You can register multiple initializers. They run in parallel by default.

```typescript
export const appConfig: ApplicationConfig = {
  providers: [
    provideAppInitializer(() => {
      const config = inject(ConfigService);
      return config.load();
    }),
    provideAppInitializer(() => {
      const auth = inject(AuthService);
      return auth.validateToken();
    }),
  ]
};
```

Both initializers start simultaneously. The app renders only after both complete.

### Legacy: APP_INITIALIZER Token

Older codebases use the `APP_INITIALIZER` injection token. It still works but `provideAppInitializer` is preferred.

```typescript
// Legacy approach
import { APP_INITIALIZER } from '@angular/core';

export const appConfig: ApplicationConfig = {
  providers: [
    {
      provide: APP_INITIALIZER,
      useFactory: (config: ConfigService) => () => config.load(),
      deps: [ConfigService],
      multi: true,
    },
  ]
};
```

The `multi: true` flag allows multiple providers for the same token. Each factory function runs during initialization.

### ENVIRONMENT_INITIALIZER

For route-level initialization (running code when a lazy route is first activated):

```typescript
import { ENVIRONMENT_INITIALIZER, inject } from '@angular/core';

export const FEATURE_ROUTES: Routes = [
  {
    path: '',
    providers: [
      {
        provide: ENVIRONMENT_INITIALIZER,
        multi: true,
        useValue: () => {
          const analytics = inject(AnalyticsService);
          analytics.trackFeatureAccess('users');
        },
      },
    ],
    children: [/* ... */],
  },
];
```

This runs once when the route's environment injector is created, not on every navigation to the route.

## A Complete app.config.ts

Here is a production-ready configuration covering the most common scenarios:

```typescript
// app.config.ts
import { ApplicationConfig, provideZonelessChangeDetection } from '@angular/core';
import {
  provideRouter,
  withComponentInputBinding,
  withViewTransitions,
  withInMemoryScrolling,
} from '@angular/router';
import { provideHttpClient, withInterceptors, withFetch } from '@angular/common/http';
import { provideAnimations } from '@angular/platform-browser/animations';
import { provideAppInitializer, inject } from '@angular/core';

import { routes } from './app.routes';
import { authInterceptor } from './core/auth.interceptor';
import { errorInterceptor } from './core/error.interceptor';
import { ConfigService } from './core/config.service';

export const appConfig: ApplicationConfig = {
  providers: [
    // Change detection
    provideZonelessChangeDetection(),

    // Routing
    provideRouter(
      routes,
      withComponentInputBinding(),
      withViewTransitions(),
      withInMemoryScrolling({ scrollPositionRestoration: 'enabled' }),
    ),

    // HTTP
    provideHttpClient(
      withInterceptors([authInterceptor, errorInterceptor]),
      withFetch(),
    ),

    // Animations
    provideAnimations(),

    // App initialization
    provideAppInitializer(() => {
      const config = inject(ConfigService);
      return config.load();
    }),
  ]
};
```

## Server-Side Rendering Configuration

SSR applications need a separate server config that includes server-specific providers.

```typescript
// app.config.server.ts
import { mergeApplicationConfig, ApplicationConfig } from '@angular/core';
import { provideServerRendering } from '@angular/platform-server';
import { provideServerRouting } from '@angular/ssr';
import { appConfig } from './app.config';
import { serverRoutes } from './app.routes.server';

const serverConfig: ApplicationConfig = {
  providers: [
    provideServerRendering(),
    provideServerRouting(serverRoutes),
  ]
};

export const config = mergeApplicationConfig(appConfig, serverConfig);
```

The `mergeApplicationConfig` function combines the browser config with server-specific providers. The server entry point uses this merged config:

```typescript
// main.server.ts
import { bootstrapApplication } from '@angular/platform-browser';
import { AppComponent } from './app/app.component';
import { config } from './app/app.config.server';

const bootstrap = () => bootstrapApplication(AppComponent, config);
export default bootstrap;
```

## Testing Configuration

Tests can reuse the same `ApplicationConfig` or create a test-specific version:

```typescript
// test-config.ts
import { ApplicationConfig } from '@angular/core';
import { provideRouter } from '@angular/router';
import { provideHttpClient } from '@angular/common/http';
import { provideNoopAnimations } from '@angular/platform-browser/animations';
import { routes } from './app.routes';

export const testConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
    provideHttpClient(),
    provideNoopAnimations(), // Disable animations in tests for speed
  ]
};
```

Use it in tests:

```typescript
beforeEach(async () => {
  await TestBed.configureTestingModule({
    imports: [AppComponent],
    providers: testConfig.providers,
  }).compileComponents();
});
```

## importProvidersFrom: Legacy Module Interop

When a third-party library only provides an NgModule (not a `provide*` function), use `importProvidersFrom` to extract its providers:

```typescript
import { importProvidersFrom } from '@angular/core';
import { SomeLegacyModule } from 'legacy-library';
import { AnotherModule } from 'another-library';

export const appConfig: ApplicationConfig = {
  providers: [
    importProvidersFrom(
      SomeLegacyModule.forRoot({ apiKey: 'abc123' }),
      AnotherModule,
    ),
  ]
};
```

`importProvidersFrom` collects all providers from the module and its transitively imported modules. Use it as a bridge until the library provides a standalone `provide*` function.

## Error Handling During Bootstrap

The `bootstrapApplication` function returns a `Promise<ApplicationRef>`. Always handle errors:

```typescript
// main.ts
bootstrapApplication(AppComponent, appConfig)
  .catch((err) => {
    console.error('Application bootstrap failed:', err);
    // Optionally show a fallback UI
    document.body.innerHTML = `
      <div style="padding: 2rem; font-family: sans-serif;">
        <h1>Application Error</h1>
        <p>The application failed to start. Please refresh the page.</p>
      </div>
    `;
  });
```

Common bootstrap failures:
- Missing provider for a required service.
- App initializer that rejects or hangs.
- Template compilation errors (shown at build time with AOT but can surface here with JIT).
- Network errors if an initializer fetches remote configuration.

### Handling Initializer Failures Gracefully

If an app initializer fails, the entire bootstrap fails. Handle errors inside the initializer to degrade gracefully instead of crashing:

```typescript
provideAppInitializer(() => {
  const config = inject(ConfigService);
  return config.load().catch((err) => {
    console.warn('Config load failed, using defaults:', err);
    config.useDefaults();
    // Returning normally (not re-throwing) allows bootstrap to continue
  });
}),
```

## Common Mistakes

- **Forgetting to add a required provider.** If your component injects `HttpClient` but `provideHttpClient()` is not in `app.config.ts`, you get a runtime `NullInjectorError`. Read the error message: it tells you exactly which token is missing.

- **Adding `provideZonelessChangeDetection()` in Angular 21 when it is already the default.** New Angular 21 projects are zoneless by default. Adding the provider explicitly is harmless but redundant. Only add it when migrating an older project to zoneless.

- **Registering the same provider twice.** Adding `provideRouter(routes)` in both `app.config.ts` and a route's `providers` array creates two router instances. Framework providers like `provideRouter`, `provideHttpClient`, and `provideAnimations` should be registered exactly once at the application level.

- **Blocking bootstrap with a hanging initializer.** If an `APP_INITIALIZER` or `provideAppInitializer` function never resolves its Promise, the application never renders. Always add error handling and timeouts to initializer functions that make network requests.

- **Using `importProvidersFrom` for modules that have standalone equivalents.** If a library provides `provideX()`, use that instead of `importProvidersFrom(XModule)`. The `provide*` function is tree-shakable; importing the full module is not.

- **Not providing animations when using Angular Material.** Angular Material requires the animations system. If you use Material components without `provideAnimations()`, animations silently fail or throw errors. Add `provideAnimations()` or `provideNoopAnimations()` to your config.

---

## Best Practices

- **Keep `app.config.ts` as the single source of truth for application providers.** All `provide*` functions should live here. Route-level providers go in route configurations.

- **Use feature functions to configure subsystems.** Instead of relying on defaults, explicitly opt into features like `withComponentInputBinding()`, `withViewTransitions()`, and `withFetch()`. This documents your application's capabilities.

- **Separate server config from browser config.** Use `mergeApplicationConfig` to compose them. The server config adds server-specific providers without duplicating browser providers.

- **Handle bootstrap errors.** Always `.catch()` the `bootstrapApplication` promise. Show a user-friendly fallback if the app fails to start.

- **Make initializers resilient.** Catch errors inside initializer functions and fall back to defaults. A failed config fetch should not prevent the app from rendering.

- **Order interceptors deliberately.** In `withInterceptors([...])`, interceptors execute in order for requests (first to last) and in reverse for responses. Put auth first, logging last.

## Real-World Use Cases

### 1. Multi-Environment Configuration Loading

```typescript
// core/config.service.ts
@Injectable({ providedIn: 'root' })
export class ConfigService {
  private config = signal<AppConfig | null>(null);
  readonly apiUrl = computed(() => this.config()?.apiUrl ?? '');
  readonly featureFlags = computed(() => this.config()?.featureFlags ?? {});

  private http = inject(HttpClient);

  load(): Promise<void> {
    return firstValueFrom(
      this.http.get<AppConfig>('/assets/config.json')
    ).then(config => {
      this.config.set(config);
    });
  }

  useDefaults() {
    this.config.set({
      apiUrl: '/api',
      featureFlags: {},
    });
  }
}

// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(withFetch()),
    provideRouter(routes, withComponentInputBinding()),
    provideAppInitializer(() => {
      const config = inject(ConfigService);
      return config.load().catch(() => config.useDefaults());
    }),
  ]
};
```

The app loads `/assets/config.json` before rendering. Different environments (staging, production) deploy different `config.json` files without rebuilding the application.

### 2. Authentication Token Validation on Startup

```typescript
// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
    provideHttpClient(withInterceptors([authInterceptor])),
    provideAppInitializer(() => {
      const auth = inject(AuthService);
      const router = inject(Router);

      return auth.validateStoredToken().then(isValid => {
        if (!isValid) {
          // Token expired or missing. Redirect to login after bootstrap.
          router.navigateByUrl('/login');
        }
      }).catch(() => {
        // Network error. Let the app start; guards will handle protection.
      });
    }),
  ]
};
```

### 3. Shared Config Between Browser and SSR

```typescript
// app.config.ts (shared browser config)
export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
    provideRouter(
      routes,
      withComponentInputBinding(),
      withViewTransitions(),
      withInMemoryScrolling({ scrollPositionRestoration: 'enabled' }),
    ),
    provideHttpClient(
      withInterceptors([authInterceptor, errorInterceptor]),
      withFetch(),
    ),
    provideAnimations(),
  ]
};

// app.config.server.ts (server additions)
import { mergeApplicationConfig } from '@angular/core';
import { provideServerRendering } from '@angular/platform-server';
import { provideServerRouting } from '@angular/ssr';
import { appConfig } from './app.config';
import { serverRoutes } from './app.routes.server';

export const config = mergeApplicationConfig(appConfig, {
  providers: [
    provideServerRendering(),
    provideServerRouting(serverRoutes),
  ]
});
```

The browser and server share the same base config. The server config adds SSR-specific providers. This pattern ensures both environments have identical routing, HTTP, and animation configuration.

---

> **Key Takeaway:** `bootstrapApplication` with `ApplicationConfig` is the foundation of every Angular 21 application. It replaces NgModule-based bootstrapping with a flat list of `provide*` functions that configure routing, HTTP, animations, change detection, and initialization. Keep everything in `app.config.ts`, use feature functions for explicit opt-in behavior, and always handle bootstrap errors.
