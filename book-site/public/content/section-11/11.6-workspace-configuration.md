# 11.6 Workspace Configuration (`angular.json` Deep Dive)

> `angular.json` is the central configuration file for every Angular workspace. It controls how your application is built, served, tested, and linted. Understanding every section gives you full control over the build pipeline.

---

## Top-Level Structure

The `angular.json` file has this structure:

```json
{
  "$schema": "./node_modules/@angular/cli/lib/config/schema.json",
  "version": 1,
  "cli": { /* ... */ },
  "newProjectRoot": "projects",
  "projects": {
    "my-app": { /* project configuration */ }
  }
}
```

### $schema

Points to the JSON schema that your IDE uses for autocompletion and validation. When you type a property name, VS Code suggests valid options because of this schema.

### version

Always `1`. This is the configuration file format version, not the Angular version.

### cli

Workspace-wide CLI settings that apply to all projects:

```json
{
  "cli": {
    "analytics": false,
    "cache": {
      "enabled": true,
      "path": ".angular/cache",
      "environment": "all"
    },
    "schematicCollections": ["@angular/material"],
    "packageManager": "npm"
  }
}
```

| Property | Description |
|---|---|
| `analytics` | Enable/disable CLI usage analytics |
| `cache.enabled` | Toggle the build cache |
| `cache.path` | Directory for cached build artifacts |
| `cache.environment` | `"all"`, `"ci"`, or `"local"` to control when caching is active |
| `schematicCollections` | Additional schematic collections for `ng generate` |
| `packageManager` | `"npm"`, `"yarn"`, `"pnpm"`, or `"cnpm"` |

### newProjectRoot

The directory where new projects are created when using `ng generate application` or `ng generate library` in a multi-project workspace. Default: `"projects"`.

---

## Project Configuration

Each entry in the `projects` object defines one application or library:

```json
{
  "projects": {
    "my-app": {
      "projectType": "application",
      "root": "",
      "sourceRoot": "src",
      "prefix": "app",
      "schematics": { /* ... */ },
      "architect": { /* ... */ }
    }
  }
}
```

### projectType

Either `"application"` or `"library"`. Applications can be built and served; libraries are compiled into distributable packages.

### root

The root folder of the project, relative to the workspace root. For single-project workspaces this is `""` (the workspace root). For multi-project workspaces it is `"projects/my-lib"`.

### sourceRoot

The root folder for the project's source files. Usually `"src"` for applications and `"projects/my-lib/src"` for libraries.

### prefix

The string prepended to generated component and directive selectors. Default is `"app"`, producing selectors like `app-header`, `app-user-list`.

```json
{
  "prefix": "acme"
}
```

Now `ng generate component header` creates a component with selector `acme-header`.

## Schematics Defaults

The `schematics` section sets default options for `ng generate` commands. This replaces the need to pass flags every time.

```json
{
  "schematics": {
    "@schematics/angular:component": {
      "style": "scss",
      "changeDetection": "OnPush",
      "skipTests": false
    },
    "@schematics/angular:service": {
      "skipTests": false
    },
    "@schematics/angular:pipe": {
      "skipTests": false
    },
    "@schematics/angular:directive": {
      "skipTests": false
    },
    "@schematics/angular:guard": {
      "functional": true
    }
  }
}
```

With this config, every `ng g c my-component` automatically uses SCSS and OnPush change detection. No flags needed.

### Available Schematic Options

| Schematic | Key Options |
|---|---|
| `@schematics/angular:component` | `style`, `changeDetection`, `viewEncapsulation`, `inlineStyle`, `inlineTemplate`, `skipTests`, `flat`, `prefix` |
| `@schematics/angular:service` | `skipTests`, `flat` |
| `@schematics/angular:pipe` | `skipTests`, `flat` |
| `@schematics/angular:directive` | `skipTests`, `flat` |
| `@schematics/angular:guard` | `functional` |
| `@schematics/angular:interceptor` | `functional` |
| `@schematics/angular:resolver` | `functional` |

## The Architect Section

The `architect` section contains build targets. Each target maps to a builder (an npm package that performs the work) and has options and configurations.

```json
{
  "architect": {
    "build":  { /* ... */ },
    "serve":  { /* ... */ },
    "test":   { /* ... */ },
    "lint":   { /* ... */ },
    "deploy": { /* ... */ }
  }
}
```

Standard target names correspond to CLI commands: `ng build` runs the `build` target, `ng serve` runs `serve`, etc. You can also define custom targets and run them with `ng run my-app:custom-target`.

---

## The Build Target

The most important and complex target. Controls how the application is compiled and bundled.

```json
{
  "build": {
    "builder": "@angular/build:application",
    "options": {
      "outputPath": "dist/my-app",
      "index": "src/index.html",
      "browser": "src/main.ts",
      "polyfills": [],
      "tsConfig": "tsconfig.app.json",
      "assets": [
        { "glob": "**/*", "input": "public" }
      ],
      "styles": [
        "src/styles.scss"
      ],
      "scripts": []
    },
    "configurations": {
      "production": { /* ... */ },
      "development": { /* ... */ }
    },
    "defaultConfiguration": "production"
  }
}
```

### Core Options

| Option | Description |
|---|---|
| `outputPath` | Directory for built files. String or object for SSR: `{ "base": "dist/my-app", "browser": "", "server": "" }` |
| `index` | Path to the main HTML file |
| `browser` | Entry point for the browser bundle (usually `src/main.ts`) |
| `polyfills` | Array of polyfill entry points or package names |
| `tsConfig` | Path to the TypeScript configuration file |
| `assets` | Static files copied to the output directory |
| `styles` | Global stylesheets applied to the entire application |
| `scripts` | Global scripts included in the HTML |

### SSR-Specific Options

When SSR is enabled, additional options appear:

```json
{
  "options": {
    "outputPath": "dist/my-app",
    "browser": "src/main.ts",
    "server": "src/main.server.ts",
    "ssr": {
      "entry": "server.ts"
    },
    "prerender": true
  }
}
```

| Option | Description |
|---|---|
| `server` | Entry point for the server bundle |
| `ssr.entry` | Server entry point for the SSR Express/Node server |
| `prerender` | Enable build-time prerendering of routes |
| `outputMode` | `"static"` for full prerendering, `"server"` for SSR with live server |

### Assets Configuration

Assets can be a simple glob or a detailed object:

```json
{
  "assets": [
    { "glob": "**/*", "input": "public" },
    {
      "glob": "**/*.svg",
      "input": "node_modules/@my-company/icons/assets",
      "output": "/icons"
    },
    {
      "glob": "manifest.webmanifest",
      "input": "src",
      "output": "/"
    }
  ]
}
```

| Property | Description |
|---|---|
| `glob` | File matching pattern |
| `input` | Source directory (relative to workspace root) |
| `output` | Destination directory (relative to output path). Defaults to `""` |
| `ignore` | Array of glob patterns to exclude |
| `followSymlinks` | Follow symbolic links when resolving assets |

### Styles Configuration

Global styles applied before component styles:

```json
{
  "styles": [
    "src/styles.scss",
    {
      "input": "src/themes/dark-theme.scss",
      "bundleName": "dark-theme",
      "inject": false
    }
  ]
}
```

| Property | Description |
|---|---|
| `input` | Path to the stylesheet |
| `bundleName` | Name of the output bundle (used for lazy-loaded themes) |
| `inject` | `true` (default): inject into HTML. `false`: generate a separate CSS file without injecting it |

Setting `inject: false` creates a CSS file that can be loaded on demand (useful for theme switching).

### Scripts Configuration

Global scripts loaded before the application:

```json
{
  "scripts": [
    "node_modules/some-library/dist/lib.min.js",
    {
      "input": "src/scripts/analytics.js",
      "bundleName": "analytics",
      "inject": false
    }
  ]
}
```

Scripts are concatenated and included as `<script>` tags. Use `inject: false` to produce a separate file without automatic inclusion. Scripts do not go through tree shaking.

## Build Configurations

Configurations override `options` values for different environments. They are selected with the `--configuration` flag.

```json
{
  "configurations": {
    "production": {
      "budgets": [
        {
          "type": "initial",
          "maximumWarning": "500kB",
          "maximumError": "1MB"
        },
        {
          "type": "anyComponentStyle",
          "maximumWarning": "4kB",
          "maximumError": "8kB"
        }
      ],
      "outputHashing": "all"
    },
    "development": {
      "optimization": false,
      "extractLicenses": false,
      "sourceMap": true
    },
    "staging": {
      "optimization": true,
      "sourceMap": {
        "scripts": true,
        "styles": false,
        "hidden": true
      },
      "fileReplacements": [
        {
          "replace": "src/environments/environment.ts",
          "with": "src/environments/environment.staging.ts"
        }
      ]
    }
  },
  "defaultConfiguration": "production"
}
```

### Optimization Options

The `optimization` field accepts a boolean or a detailed object:

```json
{
  "optimization": {
    "scripts": true,
    "styles": {
      "minify": true,
      "inlineCritical": true
    },
    "fonts": {
      "inline": true
    }
  }
}
```

| Option | Default (prod) | Description |
|---|---|---|
| `scripts` | `true` | Minify and tree-shake JavaScript |
| `styles.minify` | `true` | Minify CSS |
| `styles.inlineCritical` | `true` | Inline critical CSS for faster first paint |
| `fonts.inline` | `true` | Inline Google Fonts and other font URLs |

### Source Map Options

```json
{
  "sourceMap": {
    "scripts": true,
    "styles": true,
    "hidden": true,
    "vendor": false
  }
}
```

| Option | Description |
|---|---|
| `scripts` | Generate source maps for JavaScript files |
| `styles` | Generate source maps for CSS files |
| `hidden` | Generate source maps but do not reference them in the output (no `//# sourceMappingURL`). Useful for error tracking services like Sentry |
| `vendor` | Include source maps for third-party dependencies |

### File Replacements

Swap files based on configuration. Commonly used for environment files:

```json
{
  "fileReplacements": [
    {
      "replace": "src/environments/environment.ts",
      "with": "src/environments/environment.staging.ts"
    }
  ]
}
```

When building with `--configuration=staging`, every import of `src/environments/environment.ts` resolves to `src/environments/environment.staging.ts` instead.

### Output Hashing

```json
{
  "outputHashing": "all"
}
```

| Value | Effect |
|---|---|
| `"none"` | No hashing. File names are static |
| `"all"` | Hash all output files (JS, CSS, assets) |
| `"bundles"` | Hash only JavaScript bundles |
| `"media"` | Hash only asset files (images, fonts) |

Content hashing enables aggressive browser caching. When a file changes, its hash changes, forcing browsers to download the new version. `"all"` is the standard for production.

## The Serve Target

Configures the development server:

```json
{
  "serve": {
    "builder": "@angular/build:dev-server",
    "configurations": {
      "production": {
        "buildTarget": "my-app:build:production"
      },
      "development": {
        "buildTarget": "my-app:build:development"
      }
    },
    "defaultConfiguration": "development",
    "options": {
      "port": 4200,
      "host": "localhost",
      "open": false,
      "hmr": true,
      "proxyConfig": "proxy.conf.json",
      "ssl": false,
      "sslCert": "",
      "sslKey": ""
    }
  }
}
```

| Option | Description |
|---|---|
| `buildTarget` | Reference to the build target and configuration to use |
| `port` | Port number for the dev server |
| `host` | Host to listen on (`"0.0.0.0"` for network access) |
| `open` | Open browser automatically |
| `hmr` | Enable Hot Module Replacement |
| `proxyConfig` | Path to a proxy configuration file |
| `ssl` | Enable HTTPS |
| `sslCert` / `sslKey` | Paths to SSL certificate and key files |
| `headers` | Custom HTTP headers for the dev server |

The `buildTarget` property uses the format `project:target:configuration`. The serve target delegates actual building to the build target.

## The Test Target

Configures the test runner:

```json
{
  "test": {
    "builder": "@angular/build:unit-test",
    "options": {
      "tsConfig": "tsconfig.spec.json",
      "polyfills": [],
      "assets": [
        { "glob": "**/*", "input": "public" }
      ],
      "styles": ["src/styles.scss"]
    }
```

For Angular 21 projects using Vitest, the builder is `@angular/build:unit-test`. Older projects using Karma use `@angular-devkit/build-angular:karma`.

## Custom Targets

You can define any target name and attach a builder to it:

```json
{
  "architect": {
    "analyze": {
      "builder": "@angular/build:application",
      "options": {
        "outputPath": "dist/analyze",
        "browser": "src/main.ts",
        "tsConfig": "tsconfig.app.json",
        "sourceMap": true,
        "namedChunks": true,
        "optimization": true
      }
    }
  }
}
```

Run it with:

```bash
ng run my-app:analyze
```

## Multi-Project Workspace

A workspace can contain multiple applications and libraries:

```json
{
  "projects": {
    "customer-app": {
      "projectType": "application",
      "root": "projects/customer-app",
      "sourceRoot": "projects/customer-app/src",
      "prefix": "cust",
      "architect": { /* ... */ }
    },
    "admin-app": {
      "projectType": "application",
      "root": "projects/admin-app",
      "sourceRoot": "projects/admin-app/src",
      "prefix": "admin",
      "architect": { /* ... */ }
    },
    "shared-ui": {
      "projectType": "library",
      "root": "projects/shared-ui",
      "sourceRoot": "projects/shared-ui/src",
      "prefix": "ui",
      "architect": {
        "build": {
          "builder": "@angular/build:ng-packagr",
          "options": {
            "project": "projects/shared-ui/ng-package.json"
          }
        }
      }
    }
  }
}
```

Target a specific project:

```bash
ng serve customer-app
ng build admin-app --configuration=production
ng test shared-ui
```

Libraries use the `ng-packagr` builder, which compiles them into the Angular Package Format (APF) for distribution or local consumption.

## Configuration Inheritance and Override Order

Values are resolved in this order (later overrides earlier):

1. **Builder defaults** (built into the builder package)
2. **Workspace-level `cli` and `schematics`** (top-level in `angular.json`)
3. **Project-level `options`** (in the `architect.build.options` section)
4. **Configuration overrides** (in `architect.build.configurations.production`)
5. **Command-line flags** (highest priority)

```bash
# angular.json has port: 4200
# staging config has port: 4300
# Command line overrides everything:
ng serve --port=5000 --configuration=staging
# Result: port 5000, staging file replacements
```

## Common Mistakes

- **Editing `angular.json` while `ng serve` is running without restarting.** Some changes (like assets, styles, and scripts arrays) require restarting the dev server. Changes to `options` and `configurations` do not hot-reload.

- **Using relative paths in the wrong context.** Paths in `angular.json` are relative to the workspace root, not to the project root. For a multi-project setup, `"src/main.ts"` refers to `workspace-root/src/main.ts`, not `workspace-root/projects/my-app/src/main.ts`. Use `"projects/my-app/src/main.ts"` for multi-project workspaces.

- **Confusing `assets` with `styles`.** Files in `assets` are copied as-is to the output. Files in `styles` are compiled, minified, and bundled. Putting a CSS file in `assets` means it is copied but not processed. Putting an image in `styles` causes a build error.

- **Forgetting to add the staging configuration.** The CLI only generates `production` and `development` configurations. If your deployment has a staging environment, you must add the `staging` configuration manually with appropriate `fileReplacements` and optimization settings.

- **Setting `optimization: true` in development.** This slows down rebuilds dramatically because every change triggers minification, tree shaking, and critical CSS inlining. Keep `optimization: false` for development.

- **Enabling source maps in production without `hidden: true`.** Regular source maps append a `//# sourceMappingURL` comment to the output, which browsers use to load the maps. This exposes your source code to anyone who opens DevTools. Use `hidden: true` to generate maps without the reference, then upload them to your error tracking service separately.

- **Omitting build budgets.** Without budgets, bundle size grows unchecked. One developer adds a large library, another adds another, and soon the initial bundle is 2MB. Budgets catch this at build time.

- **Adding everything to `scripts`.** The `scripts` array is for libraries that cannot be imported as ES modules (legacy jQuery plugins, analytics snippets). Modern libraries should be `npm install`-ed and imported in your TypeScript code, where they benefit from tree shaking.

## Best Practices

- **Set schematic defaults on day one.** Configure `style`, `changeDetection`, and `functional` defaults in the `schematics` section so every generated artifact follows team conventions.

- **Use separate configurations for each environment.** At minimum: `production` (optimized, no source maps), `development` (unoptimized, source maps), and optionally `staging` (optimized, hidden source maps, staging environment file).

- **Configure build budgets in production.** Set `maximumWarning` and `maximumError` for both `initial` and `anyComponentStyle` bundle types. This fails the build before oversized bundles reach production.

- **Use `inject: false` for theme stylesheets.** Generate separate CSS files for themes or optional styles. Load them on demand using `<link>` tags or the Angular CDK overlay system.

- **Use `hidden: true` for production source maps if you need them.** This generates maps for services like Sentry without exposing them to end users.

- **Keep `angular.json` version-controlled.** It is not a generated file. Changes to build options, budgets, and configurations should go through code review.

- **Validate with the schema.** The `$schema` property enables IDE validation. If you see red squiggles under a property, the value is incorrect.

## Real-World Use Cases

### 1. Multi-Environment Build Pipeline

```json
{
  "configurations": {
    "production": {
      "optimization": true,
      "outputHashing": "all",
      "sourceMap": false,
      "extractLicenses": true,
      "budgets": [
        { "type": "initial", "maximumWarning": "500kB", "maximumError": "1MB" },
        { "type": "anyComponentStyle", "maximumWarning": "4kB", "maximumError": "8kB" }
      ],
      "fileReplacements": [
        {
          "replace": "src/environments/environment.ts",
          "with": "src/environments/environment.production.ts"
        }
      ]
    },
    "staging": {
      "optimization": true,
      "outputHashing": "all",
      "sourceMap": { "scripts": true, "hidden": true },
      "fileReplacements": [
        {
          "replace": "src/environments/environment.ts",
          "with": "src/environments/environment.staging.ts"
        }
      ]
    },
    "development": {
      "optimization": false,
      "sourceMap": true,
      "extractLicenses": false
    }
  }
}
```

Build for each environment:

```bash
ng build --configuration=production   # Optimized, no source maps
ng build --configuration=staging      # Optimized, hidden source maps
ng build --configuration=development  # Fast, full source maps
```

### 2. Theme Switching with Non-Injected Styles

```json
{
  "styles": [
    "src/styles.scss",
    {
      "input": "src/themes/light-theme.scss",
      "bundleName": "light-theme",
      "inject": true
    },
    {
      "input": "src/themes/dark-theme.scss",
      "bundleName": "dark-theme",
      "inject": false
    }
  ]
}
```

The light theme is injected by default. The dark theme is built as a separate CSS file (`dark-theme.css`). Switch themes at runtime by toggling `<link>` tags:

```typescript
@Injectable({ providedIn: 'root' })
export class ThemeService {
  private linkElement: HTMLLinkElement | null = null;

  setTheme(theme: 'light' | 'dark') {
    if (theme === 'dark') {
      if (!this.linkElement) {
        this.linkElement = document.createElement('link');
        this.linkElement.rel = 'stylesheet';
        this.linkElement.href = 'dark-theme.css';
        document.head.appendChild(this.linkElement);
      }
    } else {
      this.linkElement?.remove();
      this.linkElement = null;
    }
  }
}
```

### 3. Incorporating Third-Party Assets from node_modules

```json
{
  "assets": [
    { "glob": "**/*", "input": "public" },
    {
      "glob": "**/*.woff2",
      "input": "node_modules/@fontsource/inter/files",
      "output": "/fonts/inter"
    },
    {
      "glob": "*.svg",
      "input": "node_modules/heroicons/24/outline",
      "output": "/icons/heroicons"
    }
  ],
  "styles": [
    "src/styles.scss",
    "node_modules/@fontsource/inter/index.css"
  ]
}
```

This copies font files and icon SVGs from npm packages into the build output, and includes the font CSS as a global stylesheet. The fonts are then available at `/fonts/inter/` and icons at `/icons/heroicons/` in the running application.

---

> **Key Takeaway:** `angular.json` is the backbone of your build pipeline. Set schematic defaults, configure per-environment builds with appropriate optimization and source map settings, add build budgets to prevent bundle bloat, and use asset/style configuration to manage static resources without manual copying.
