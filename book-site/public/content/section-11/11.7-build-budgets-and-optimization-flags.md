# 11.7 Build Budgets and Optimization Flags

> Build budgets enforce bundle size limits at compile time, and optimization flags control what the build system does to shrink your output. Together, they prevent performance regressions and ensure fast load times in production.

---

## What Are Build Budgets?

Build budgets are size thresholds configured in `angular.json`. When a build exceeds a budget, the CLI emits a warning or an error. Budgets run during `ng build` and fail the build if the `maximumError` threshold is crossed.

Without budgets, bundle size grows silently. A developer adds a date library (70KB), another adds a charting library (200KB), and within a few sprints the initial bundle is over 1MB. Budgets catch this at build time, before it reaches production.

```json
{
  "configurations": {
    "production": {
      "budgets": [
        {
          "type": "initial",
          "maximumWarning": "500kB",
          "maximumError": "1MB"
        },
        {
          "type": "anyComponentStyle",
          "maximumWarning": "4kB",
          "maximumError": "8kB"
        }
      ]
    }
  }
}
```

When the initial bundle exceeds 500KB, the build outputs:

```
Warning: bundle initial exceeded maximum budget. Budget 500.00 kB was not met by 42.31 kB with a total of 542.31 kB.
```

When it exceeds 1MB, the build fails:

```
Error: bundle initial exceeded maximum budget. Budget 1.00 MB was not met by 128.45 kB with a total of 1.13 MB.
```

## Budget Types

Angular supports several budget types that target different parts of the build output.

| Type | What It Measures |
|---|---|
| `initial` | The combined size of all JavaScript and CSS that loads on the initial page (before lazy loading) |
| `allScript` | The total size of all JavaScript files |
| `all` | The total size of the entire application (JS, CSS, HTML, assets) |
| `anyScript` | The size of any single JavaScript file |
| `anyComponentStyle` | The size of any single component's CSS |
| `any` | The size of any single file in the output |

### initial

The most important budget. It measures everything the browser must download before the user sees the first meaningful paint: the main bundle, polyfills, and global styles. Lazy-loaded chunks are excluded.

```json
{
  "type": "initial",
  "maximumWarning": "500kB",
  "maximumError": "1MB"
}
```

### anyComponentStyle

Catches bloated component styles. A component with 10KB of CSS is a sign that styles are not properly shared through global styles or that the component is doing too much.

```json
{
  "type": "anyComponentStyle",
  "maximumWarning": "4kB",
  "maximumError": "8kB"
}
```

### allScript

The total JavaScript output. Useful for catching overall growth even when individual bundles stay under `anyScript` limits.

```json
{
  "type": "allScript",
  "maximumWarning": "1MB",
  "maximumError": "2MB"
}
```

### anyScript

The size of any single JavaScript chunk. Catches a single lazy-loaded feature that has grown too large.

```json
{
  "type": "anyScript",
  "maximumWarning": "300kB",
  "maximumError": "500kB"
}
```

### all and any

`all` measures the entire build output (including images and fonts copied from assets). `any` measures any single output file. These are less commonly used but useful for specific constraints.

---

## Budget Threshold Properties

Each budget entry supports four threshold properties:

| Property | Trigger | Behavior |
|---|---|---|
| `maximumWarning` | Build output exceeds this value | Prints a warning, build succeeds |
| `maximumError` | Build output exceeds this value | Prints an error, build fails |
| `minimumWarning` | Build output is below this value | Prints a warning (rarely used) |
| `minimumError` | Build output is below this value | Prints an error (rarely used) |
| `baseline` | Reference size for percentage-based budgets | Used with `warning`/`error` deviation fields |

### Size Units

Budgets accept these formats:

| Format | Example | Meaning |
|---|---|---|
| Number | `512000` | Bytes |
| `b` | `512000b` | Bytes |
| `kB` | `500kB` | Kilobytes |
| `MB` | `1MB` | Megabytes |
| `%` | `10%` | Percentage of baseline |

### Baseline and Percentage Budgets

Instead of absolute sizes, you can set a baseline and use percentage-based deviations:

```json
{
  "type": "initial",
  "baseline": "400kB",
  "warning": "50kB",
  "error": "100kB"
}
```

This warns if the initial bundle grows more than 50KB beyond 400KB (i.e., exceeds 450KB) and errors if it grows more than 100KB beyond 400KB (i.e., exceeds 500KB). This approach focuses on drift rather than absolute size.

## Recommended Budget Configuration

For a typical Angular 21 application:

```json
{
  "budgets": [
    {
      "type": "initial",
      "maximumWarning": "500kB",
      "maximumError": "1MB"
    },
    {
      "type": "anyComponentStyle",
      "maximumWarning": "4kB",
      "maximumError": "8kB"
    },
    {
      "type": "anyScript",
      "maximumWarning": "300kB",
      "maximumError": "500kB"
    }
  ]
}
```

For performance-critical applications (e-commerce, public-facing):

```json
{
  "budgets": [
    {
      "type": "initial",
      "maximumWarning": "250kB",
      "maximumError": "500kB"
    },
    {
      "type": "anyComponentStyle",
      "maximumWarning": "2kB",
      "maximumError": "4kB"
    },
    {
      "type": "anyScript",
      "maximumWarning": "150kB",
      "maximumError": "300kB"
    }
  ]
}
```

Google recommends that the total compressed JavaScript for the initial load be under 170KB for a fast experience on 3G networks. Budgets in `angular.json` measure uncompressed sizes. Gzip typically achieves 60-70% compression, so 500KB uncompressed is roughly 150-200KB compressed.

## Optimization Flags

The `optimization` option in `angular.json` controls what the build system does to reduce bundle size. It is enabled by default in production configurations.

### Boolean Form

```json
{
  "optimization": true
}
```

Enables all optimizations: script minification, style minification, tree shaking, dead code elimination, critical CSS inlining, and font inlining.

### Object Form

For granular control:

```json
{
  "optimization": {
    "scripts": true,
    "styles": {
      "minify": true,
      "inlineCritical": true
    },
    "fonts": {
      "inline": true
    }
  }
}
```

### What Each Optimization Does

#### Scripts Optimization (`scripts: true`)

- **Minification**: Shortens variable names, removes whitespace, collapses expressions. Handled by esbuild's native minifier.
- **Tree Shaking**: Eliminates unused exports. If you import `{ map }` from RxJS, only the `map` operator is bundled, not the entire library.
- **Dead Code Elimination**: Removes code paths that can never execute (e.g., `if (false) { ... }`).
- **Scope Hoisting**: Flattens module scopes to reduce closure overhead.

```typescript
// Before optimization:
import { map, filter, switchMap, debounceTime, distinctUntilChanged } from 'rxjs';

// If only map and filter are actually used in the code:
// After tree shaking: switchMap, debounceTime, distinctUntilChanged are removed
```

#### Style Minification (`styles.minify: true`)

Removes whitespace, comments, and shortens color values in CSS:

```css
/* Before */
.user-card {
  background-color: #ffffff;
  margin-top: 16px;
  margin-bottom: 16px;
  /* Card styling */
}

/* After */
.user-card{background-color:#fff;margin-top:16px;margin-bottom:16px}
```

#### Critical CSS Inlining (`styles.inlineCritical: true`)

Extracts CSS needed for the initial viewport and inlines it into the HTML `<head>`. The remaining CSS loads asynchronously. This eliminates the render-blocking CSS request, improving First Contentful Paint (FCP).

```html
<!-- Generated HTML with critical CSS inlined -->
<head>
  <style>
    /* Critical CSS for above-the-fold content */
    body{margin:0;font-family:sans-serif}
    .header{height:64px;background:#1a1a2e}
    .hero{padding:48px 24px}
  </style>
  <link rel="stylesheet" href="styles.abc123.css" media="print" onload="this.media='all'">
</head>
```

The full stylesheet loads in the background via the `media="print"` trick, which prevents it from blocking rendering.

#### Font Inlining (`fonts.inline: true`)

Downloads Google Fonts (and other external fonts referenced in CSS) at build time and inlines the `@font-face` declarations directly into the HTML. This eliminates the extra network round trip to the font provider.

```css
/* Before (in your stylesheet) */
@import url('https://fonts.googleapis.com/css2?family=Inter:wght@400;600&display=swap');

/* After (inlined into index.html at build time) */
/* The @font-face declarations are extracted and placed in a <style> tag */
```

## Analyzing Bundle Size

Budgets tell you when something is too big. Bundle analysis tells you what is too big and why.

### source-map-explorer

The most accurate tool for Angular applications.

```bash
# Install
npm install --save-dev source-map-explorer

# Build with source maps
ng build --source-map

# Analyze
npx source-map-explorer dist/my-app/browser/*.js
```

This opens an interactive treemap in your browser. Each rectangle represents a module, sized by its contribution to the bundle. You can identify:
- Large third-party libraries
- Code that should be lazy-loaded but is not
- Duplicate dependencies
- Unused imports that escaped tree shaking

### Adding an Analyze Script

```json
{
  "scripts": {
    "analyze": "ng build --source-map && npx source-map-explorer dist/my-app/browser/*.js"
  }
}
```

Run with `npm run analyze`.

### Reading the Treemap

Look for:
- **Large rectangles in the main bundle**: candidates for lazy loading or lighter alternatives.
- **Duplicate libraries**: two versions of the same package bundled (e.g., `lodash` and `lodash-es`).
- **node_modules dominating the output**: most of your bundle should be your application code, not third-party code.

## Reducing Bundle Size

When budgets are exceeded, here are the strategies to bring the size back down, ordered by impact.

### 1. Lazy Load Features

The single most effective technique. Move feature components and their dependencies into lazy-loaded routes.

```typescript
// Before: eagerly loaded
import { DashboardComponent } from './features/dashboard/dashboard.component';

const routes: Routes = [
  { path: 'dashboard', component: DashboardComponent },
];

// After: lazy loaded
const routes: Routes = [
  {
    path: 'dashboard',
    loadComponent: () =>
      import('./features/dashboard/dashboard.component')
        .then(m => m.DashboardComponent),
  },
];
```

Every lazy-loaded route becomes a separate chunk. The initial bundle only includes the code needed for the first route.

### 2. Use @defer for Heavy Components

Angular's `@defer` block delays rendering (and loading) of components until a trigger condition is met:

```html
@defer (on viewport) {
  <app-heavy-chart [data]="chartData()" />
} @placeholder {
  <div class="chart-skeleton"></div>
}
```

The `<app-heavy-chart>` component and its charting library are only downloaded when the element scrolls into the viewport. This removes them from the initial bundle entirely.

### 3. Replace Heavy Libraries

| Heavy Library | Lighter Alternative | Savings |
|---|---|---|
| `moment` (300KB) | `date-fns` (tree-shakable, ~10KB per function) | ~280KB |
| `lodash` (70KB) | `lodash-es` (tree-shakable) or native JS | ~60KB |
| `chart.js` (200KB) | Lazy load it with `@defer` | Removes from initial |
| `@angular/material` (full) | Import individual components | Varies |

### 4. Import Individual Items

```typescript
// Bad Practice: imports entire lodash
import _ from 'lodash';
const result = _.cloneDeep(obj);

// Good Practice: import only what you need
import cloneDeep from 'lodash-es/cloneDeep';
const result = cloneDeep(obj);
```

```typescript
// Bad Practice: import entire icon set
import { MatIconModule } from '@angular/material/icon';
// If you register 500 icons but use 10, all 500 are bundled

// Good Practice: register only the icons you use
import { MatIconRegistry } from '@angular/material/icon';
// Register individual SVG icons
```

### 5. Check for Duplicate Dependencies

```bash
npm ls moment
npm ls lodash
```

If two packages depend on different versions of the same library, both versions end up in the bundle. Use `npm dedupe` or update packages to align versions.

### 6. Use providedIn: 'root' for Tree-Shakable Services

Services with `providedIn: 'root'` are tree-shakable. If no component injects the service, it is excluded from the bundle.

```typescript
// Tree-shakable: removed from bundle if unused
@Injectable({ providedIn: 'root' })
export class AnalyticsService { }

// NOT tree-shakable: always included if the module is imported
@NgModule({
  providers: [AnalyticsService],
})
export class CoreModule { }
```

## Development vs. Production Optimization

| Flag | Development | Production |
|---|---|---|
| `optimization` | `false` | `true` |
| `sourceMap` | `true` | `false` (or `{ hidden: true }`) |
| `extractLicenses` | `false` | `true` |
| `outputHashing` | `"none"` | `"all"` |
| `budgets` | Not enforced | Enforced |

Never enable optimization in development. It slows down rebuilds dramatically because every file change triggers minification, tree shaking, and critical CSS processing.

```json
{
  "configurations": {
    "production": {
      "optimization": true,
      "sourceMap": false,
      "outputHashing": "all",
      "budgets": [
        { "type": "initial", "maximumWarning": "500kB", "maximumError": "1MB" }
      ]
    },
    "development": {
      "optimization": false,
      "sourceMap": true,
      "outputHashing": "none"
    }
  }
}
```

## Monitoring Bundle Size Over Time

### CI Integration

Add budget checks to your CI pipeline. Since `ng build` exits with a non-zero code when a budget error is exceeded, CI automatically fails:

```yaml
# GitHub Actions
- name: Build
  run: ng build --configuration=production
  # Exits with error code if budgets are exceeded
```

### Bundle Size Tracking

Track bundle sizes across commits to spot trends:

```bash
# After build, record the size
BUNDLE_SIZE=$(du -sb dist/my-app/browser | cut -f1)
echo "Bundle size: ${BUNDLE_SIZE} bytes"
```

Tools like [Bundlewatch](https://bundlewatch.io/) or [size-limit](https://github.com/ai/size-limit) can compare sizes against a baseline and post results as PR comments.

### package.json Script for Size Reporting

```json
{
  "scripts": {
    "build:prod": "ng build --configuration=production",
    "size-report": "ng build --configuration=production && du -sh dist/my-app/browser/*.js | sort -rh"
  }
}
```

## Common Mistakes

- **Raising budgets instead of fixing the problem.** When a budget warning fires, the fix is to reduce bundle size (lazy load, replace heavy libraries, remove unused code), not to increase the budget limit. Raising the limit just delays the problem.

- **Not setting budgets at all.** Without budgets, there is no automated guard against bundle growth. Every Angular project should have at least an `initial` and `anyComponentStyle` budget in the production configuration.

- **Setting optimization to true in development.** This makes `ng serve` rebuilds extremely slow. Keep `optimization: false` for development and only enable it in production and staging configurations.

- **Ignoring `anyComponentStyle` budgets.** A single component with 10KB of CSS indicates either duplicated global styles inside a component, an entire library imported in component styles, or a component that should be split.

- **Analyzing uncompressed sizes only.** Budget values are uncompressed. The actual network transfer size is 60-70% smaller with gzip/brotli. A 500kB uncompressed budget translates to roughly 150-200kB over the wire.

- **Not lazy loading features.** If your entire app loads in one chunk, optimization flags can only do so much. Lazy loading is the primary mechanism for keeping the initial bundle small. Every feature behind a route should be lazy loaded.

- **Including large libraries in the main bundle when they are only used on one page.** A charting library used only on the analytics page should be imported in that page's component, lazy loaded via the router. If it appears in the main bundle, it loads on every page.

---

## Best Practices

- **Start with strict budgets and relax only when justified.** Begin with 500kB warning / 1MB error for initial. If a legitimate dependency pushes you past the limit, document why in a commit message before adjusting.

- **Run `source-map-explorer` after every dependency change.** When you add or update an npm package, check how it affects the bundle. Catch bloat immediately, not at the next production deploy.

- **Lazy load aggressively.** Every feature behind a route should use `loadComponent` or `loadChildren`. Use `@defer` for heavy components within pages.

- **Prefer tree-shakable alternatives.** Choose `date-fns` over `moment`, `lodash-es` over `lodash`, individual Material component imports over the full module.

- **Use `hidden: true` source maps for production error tracking.** Generate source maps for Sentry or similar tools without exposing them to users.

- **Automate bundle size monitoring in CI.** Make the build fail on budget violations. Post bundle size diffs on pull requests.

- **Review the treemap quarterly.** Even without new features, dependency updates can change bundle sizes. A periodic treemap review catches regressions.

## Real-World Use Cases

### 1. Diagnosing a Budget Violation

The CI build fails:

```
Error: bundle initial exceeded maximum budget.
Budget 500.00 kB was not met by 156.23 kB with a total of 656.23 kB.
```

Investigation steps:

```bash
# Step 1: Build with source maps
ng build --source-map

# Step 2: Analyze the bundle
npx source-map-explorer dist/my-app/browser/main-*.js

# Step 3: The treemap shows:
#   - @angular/core: 95KB (expected)
#   - @angular/router: 42KB (expected)
#   - chart.js: 210KB (problem!)
#   - your-app: 180KB (expected)

# Step 4: chart.js is only used on the analytics page.
# Move it to a lazy-loaded route.
```

After lazy loading the analytics feature, the initial bundle drops to 440KB and the budget passes.

### 2. Progressive Budget Tightening

Start with a lenient budget for a new project and tighten it as the application matures:

```json
// Sprint 1: establish baseline
{ "type": "initial", "maximumWarning": "750kB", "maximumError": "1.5MB" }

// Sprint 5: after initial features are built, tighten
{ "type": "initial", "maximumWarning": "500kB", "maximumError": "1MB" }

// Sprint 10: after optimizations, tighten further
{ "type": "initial", "maximumWarning": "350kB", "maximumError": "700kB" }
```

Each tightening is accompanied by optimization work: lazy loading new features, replacing heavy libraries, and splitting oversized components.

### 3. CI Pipeline with Bundle Size Reporting

```yaml
# .github/workflows/build.yml
name: Build and Size Check
on: [pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: 'npm'

      - run: npm ci

      - name: Build production
        run: ng build --configuration=production

      - name: Report bundle size
        run: |
          echo "## Bundle Size Report" >> $GITHUB_STEP_SUMMARY
          echo '```' >> $GITHUB_STEP_SUMMARY
          du -sh dist/my-app/browser/*.js | sort -rh >> $GITHUB_STEP_SUMMARY
          echo '```' >> $GITHUB_STEP_SUMMARY

      - name: Cache Angular build
        uses: actions/cache@v4
        with:
          path: .angular/cache
          key: ng-cache-${{ hashFiles('**/package-lock.json') }}
```

This pipeline builds the app (budget violations fail the build), reports individual chunk sizes as a GitHub Actions step summary, and caches the Angular build cache for faster subsequent runs.

---

> **Key Takeaway:** Build budgets are your automated guard against bundle bloat. Set them in every project's production configuration, enforce them in CI, and when they fire, use `source-map-explorer` to find what grew and lazy loading to move it out of the initial bundle.
