# 11.8 Custom Builders and Esbuild/Vite Configuration

> The Angular CLI delegates all build, serve, and test work to "builders" registered in angular.json. Understanding how the built-in builders work, how to extend them with esbuild plugins, and how to create your own builders gives you full control over the build pipeline.

---

## How Builders Work in Angular

The Angular CLI uses an internal orchestration layer called **Architect**. When you run `ng build`, Architect reads `angular.json`, resolves the builder specified for the `build` target, creates an instance of that builder, and passes it the merged configuration.

```
ng build --configuration=production
    |
    v
Architect reads angular.json
    |
    v
Resolves builder: @angular/build:application
    |
    v
Loads builder handler function
    |
    v
Passes merged options + BuilderContext
    |
    v
Builder executes and returns BuilderOutput
```

Every architect target in `angular.json` maps to a builder:

```json
{
  "architect": {
    "build": {
      "builder": "@angular/build:application",
      "options": { ... }
    },
    "serve": {
      "builder": "@angular/build:dev-server",
      "options": { ... }
    },
    "test": {
      "builder": "@angular-devkit/build-angular:karma",
      "options": { ... }
    }
  }
}
```

The builder string format is `package-name:builder-name`. The package exposes a `builders.json` that maps builder names to their implementations.

## Built-in Builders Reference

Angular ships with two packages of builders:

| Package | Builder | Purpose |
|---------|---------|---------|
| `@angular/build` | `application` | Production builds (esbuild) |
| `@angular/build` | `dev-server` | Development server (Vite) |
| `@angular-devkit/build-angular` | `karma` | Unit testing with Karma |
| `@angular-devkit/build-angular` | `extract-i18n` | Extract i18n messages |
| `@angular-devkit/build-angular` | `browser` | Legacy Webpack builds |
| `@angular-devkit/build-angular` | `browser-esbuild` | Transitional esbuild builder |
| `@angular-devkit/build-angular` | `dev-server` | Legacy Webpack dev server |

Angular 21 defaults to `@angular/build:application` and `@angular/build:dev-server`. The `browser` and `browser-esbuild` builders are legacy paths.

## The Application Builder Architecture

The `@angular/build:application` builder uses a two-stage pipeline:

### Stage 1: esbuild (Bundling)

esbuild handles:
- TypeScript transpilation (via the Angular compiler `ngtsc`)
- Tree shaking and dead code elimination
- Code splitting and chunk generation
- Asset handling (fonts, images via loaders)
- CSS/SASS compilation
- Minification (production mode)

### Stage 2: Vite (Dev Server Only)

When running `ng serve`, the output from esbuild is passed to Vite which provides:
- Fast cold starts through dependency pre-bundling
- Hot Module Replacement (HMR) for styles and templates
- Efficient file serving via native ESM

```
ng serve
    |
    v
esbuild compiles Angular app (in-memory)
    |
    v
Vite receives build artifacts
    |
    v
Vite serves via dev server with HMR
```

This is different from typical Vite projects where Vite handles both bundling and serving. Angular uses Vite **only** as a dev server. The actual compilation and bundling is always done by esbuild with the Angular compiler.

## Extending the Application Builder with Plugins

The application builder supports custom esbuild plugins directly in `angular.json` without needing third-party packages.

### Plugin Configuration in angular.json

```json
{
  "architect": {
    "build": {
      "builder": "@angular/build:application",
      "options": {
        "plugins": [
          "./esbuild/my-plugin.js"
        ]
      }
    }
  }
}
```

### Writing a Basic esbuild Plugin

Create the file at `esbuild/my-plugin.js`:

```javascript
// esbuild/my-plugin.js
// A simple esbuild plugin that logs build start and end

const myPlugin = {
  name: 'my-plugin',
  setup(build) {
    build.onStart(() => {
      console.log('Build started at:', new Date().toISOString());
    });

    build.onEnd((result) => {
      if (result.errors.length > 0) {
        console.error('Build failed with', result.errors.length, 'errors');
      } else {
        console.log('Build completed successfully');
      }
    });
  }
};

module.exports = myPlugin;
```

### Plugin Export Formats

The plugin file can export in several formats:

```javascript
// Format 1: Single plugin object
module.exports = {
  name: 'single-plugin',
  setup(build) { /* ... */ }
};

// Format 2: Array of plugins
module.exports = [
  { name: 'plugin-a', setup(build) { /* ... */ } },
  { name: 'plugin-b', setup(build) { /* ... */ } }
];

// Format 3: Factory function returning a plugin
module.exports = (options) => ({
  name: 'factory-plugin',
  setup(build) {
    // Use options from angular.json
    console.log('Plugin options:', options);
  }
});

// Format 4: Factory function returning multiple plugins
module.exports = (options) => [
  { name: 'plugin-a', setup(build) { /* ... */ } },
  { name: 'plugin-b', setup(build) { /* ... */ } }
];
```

### Passing Options to Plugins

When a plugin needs configuration, use the object form in `angular.json`:

```json
{
  "plugins": [
    {
      "path": "./esbuild/env-plugin.js",
      "options": {
        "prefix": "APP_",
        "envFile": ".env"
      }
    }
  ]
}
```

The plugin receives options as an argument to the factory function:

```javascript
// esbuild/env-plugin.js
module.exports = (options) => ({
  name: 'env-plugin',
  setup(build) {
    const prefix = options.prefix || 'NG_';
    const envVars = {};

    // Collect environment variables matching the prefix
    for (const [key, value] of Object.entries(process.env)) {
      if (key.startsWith(prefix)) {
        envVars[`process.env.${key}`] = JSON.stringify(value);
      }
    }

    // Inject as build-time constants
    build.initialOptions.define = {
      ...build.initialOptions.define,
      ...envVars
    };
  }
});
```

To use these variables in your Angular code without TypeScript errors, declare them:

```typescript
// src/env.d.ts
declare const process: {
  env: {
    APP_API_URL: string;
    APP_VERSION: string;
    [key: string]: string;
  };
};
```

## Practical esbuild Plugin Examples

### Build Timestamp Plugin

Inject a build timestamp as a compile-time constant:

```javascript
// esbuild/build-info-plugin.js
const { execSync } = require('child_process');

module.exports = {
  name: 'build-info',
  setup(build) {
    let gitHash = 'unknown';
    try {
      gitHash = execSync('git rev-parse --short HEAD').toString().trim();
    } catch {
      // Not in a git repo or git not available
    }

    build.initialOptions.define = {
      ...build.initialOptions.define,
      '__BUILD_TIMESTAMP__': JSON.stringify(new Date().toISOString()),
      '__GIT_HASH__': JSON.stringify(gitHash),
      '__BUILD_MODE__': JSON.stringify(build.initialOptions.minify ? 'production' : 'development')
    };
  }
};
```

Usage in Angular:

```typescript
// src/app/build-info.ts
declare const __BUILD_TIMESTAMP__: string;
declare const __GIT_HASH__: string;
declare const __BUILD_MODE__: string;

export const BUILD_INFO = {
  timestamp: __BUILD_TIMESTAMP__,
  gitHash: __GIT_HASH__,
  mode: __BUILD_MODE__
} as const;
```

### Custom File Resolver Plugin

Redirect imports to platform-specific files:

```javascript
// esbuild/platform-resolver.js
const path = require('path');
const fs = require('fs');

module.exports = (options) => ({
  name: 'platform-resolver',
  setup(build) {
    const platform = options.platform || 'web';

    // When importing ./service, check for ./service.web.ts first
    build.onResolve({ filter: /\.\/.*/ }, (args) => {
      if (args.kind !== 'import-statement') return;

      const dir = path.dirname(args.resolveDir + '/' + args.path);
      const base = path.basename(args.path);
      const platformPath = path.join(
        args.resolveDir,
        `${args.path}.${platform}.ts`
      );

      if (fs.existsSync(platformPath)) {
        return { path: platformPath };
      }

      // Fall through to default resolution
      return undefined;
    });
  }
});
```

### SVG Inline Plugin

Inline SVG files as strings at build time:

```javascript
// esbuild/svg-inline-plugin.js
const fs = require('fs');

module.exports = {
  name: 'svg-inline',
  setup(build) {
    build.onLoad({ filter: /\.svg$/ }, async (args) => {
      const svg = await fs.promises.readFile(args.path, 'utf8');
      // Minify by removing newlines and extra whitespace
      const minified = svg.replace(/\n/g, '').replace(/\s{2,}/g, ' ').trim();

      return {
        contents: `export default ${JSON.stringify(minified)};`,
        loader: 'js'
      };
    });
  }
};
```

## Loader Configuration

The application builder provides a `loader` option for handling non-standard file types without writing a full plugin:

```json
{
  "architect": {
    "build": {
      "builder": "@angular/build:application",
      "options": {
        "loader": {
          ".txt": "text",
          ".bin": "binary",
          ".png": "file",
          ".svg": "text"
        }
      }
    }
  }
}
```

### Available Loaders

| Loader | Behavior | Import Result |
|--------|----------|---------------|
| `text` | Inlines file content as a string | `string` default export |
| `binary` | Inlines file content as binary | `Uint8Array` default export |
| `file` | Copies file to output directory | Runtime file path as default export |
| `empty` | Resolves import but provides no content | Empty module |

### Using Loaded Files in TypeScript

```typescript
// With "loader": { ".txt": "text" } in angular.json

// Declare module for TypeScript
declare module '*.txt' {
  const content: string;
  export default content;
}

// Use in a component
import licenseText from './LICENSE.txt';

@Component({
  selector: 'app-license',
  template: `<pre>{{ license }}</pre>`
})
export class LicenseComponent {
  license = licenseText;
}
```

```typescript
// With "loader": { ".csv": "text" } in angular.json

declare module '*.csv' {
  const content: string;
  export default content;
}

import rawCsv from '../data/countries.csv';

function parseCsv(raw: string): string[][] {
  return raw.split('\n').map(line => line.split(','));
}

const countries = parseCsv(rawCsv);
```

## The Define Option

The `define` option replaces global identifiers with constant expressions at build time:

```json
{
  "architect": {
    "build": {
      "builder": "@angular/build:application",
      "options": {
        "define": {
          "API_BASE_URL": "'https://api.example.com'",
          "FEATURE_FLAGS.newDashboard": "true",
          "APP_VERSION": "'2.5.0'"
        }
      },
      "configurations": {
        "staging": {
          "define": {
            "API_BASE_URL": "'https://staging-api.example.com'"
          }
        }
      }
    }
  }
}
```

Values in `define` are raw JavaScript expressions, not JSON strings. For string values, wrap in single quotes inside double quotes: `"'my-string'"`.

Usage in code:

```typescript
// Declare the constants for TypeScript
declare const API_BASE_URL: string;
declare const FEATURE_FLAGS: { newDashboard: boolean };
declare const APP_VERSION: string;

// Use directly -- esbuild replaces these at compile time
const apiUrl = API_BASE_URL; // Replaced with 'https://api.example.com'
const showDashboard = FEATURE_FLAGS.newDashboard; // Replaced with true
```

The `define` option is simpler than `fileReplacements` for injecting environment-specific constants. Use `define` for simple key-value pairs and `fileReplacements` for swapping entire configuration files.

## Creating a Custom Builder from Scratch

When the built-in builders and plugins are not enough, you can create a fully custom builder.

### Project Structure

```
my-builder/
  package.json
  builders.json
  src/
    my-builder/
      index.ts
      schema.json
      schema.d.ts
```

### Step 1: Define the Schema

The schema validates options passed from `angular.json`:

```json
// my-builder/src/my-builder/schema.json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "My Builder Options",
  "description": "Options for the custom deploy builder",
  "type": "object",
  "properties": {
    "outputPath": {
      "type": "string",
      "description": "Path to the build output directory"
    },
    "target": {
      "type": "string",
      "description": "Deployment target",
      "enum": ["staging", "production"],
      "default": "staging"
    },
    "dryRun": {
      "type": "boolean",
      "description": "Simulate deployment without making changes",
      "default": false
    }
  },
  "required": ["outputPath"],
  "additionalProperties": false
}
```

Generate a TypeScript interface from the schema (or write one manually):

```typescript
// my-builder/src/my-builder/schema.d.ts
export interface MyBuilderSchema {
  outputPath: string;
  target: 'staging' | 'production';
  dryRun: boolean;
}
```

### Step 2: Implement the Builder

```typescript
// my-builder/src/my-builder/index.ts
import {
  BuilderContext,
  BuilderOutput,
  createBuilder
} from '@angular-devkit/architect';
import { MyBuilderSchema } from './schema';
import { promises as fs } from 'fs';
import { join } from 'path';

async function runBuilder(
  options: MyBuilderSchema,
  context: BuilderContext
): Promise<BuilderOutput> {
  // Resolve the workspace root
  const workspaceRoot = context.workspaceRoot;
  const outputPath = join(workspaceRoot, options.outputPath);

  // Log with the builder's logger
  context.logger.info(`Deploying to ${options.target}...`);
  context.logger.info(`Output path: ${outputPath}`);

  // Report progress (current, total, status)
  context.reportProgress(0, 3, 'Checking output...');

  // Verify build output exists
  try {
    await fs.access(outputPath);
  } catch {
    return {
      success: false,
      error: `Output path does not exist: ${outputPath}. Run ng build first.`
    };
  }

  context.reportProgress(1, 3, 'Validating files...');

  // Check for index.html
  const indexPath = join(outputPath, 'browser', 'index.html');
  try {
    await fs.access(indexPath);
  } catch {
    return {
      success: false,
      error: 'index.html not found in build output.'
    };
  }

  context.reportProgress(2, 3, 'Deploying...');

  if (options.dryRun) {
    context.logger.warn('Dry run: skipping actual deployment.');
    return { success: true };
  }

  // Perform the actual deployment
  try {
    await deploy(outputPath, options.target, context);
  } catch (err) {
    return {
      success: false,
      error: `Deployment failed: ${(err as Error).message}`
    };
  }

  context.reportProgress(3, 3, 'Done.');
  context.logger.info('Deployment complete.');

  return { success: true };
}

async function deploy(
  outputPath: string,
  target: string,
  context: BuilderContext
): Promise<void> {
  // Your deployment logic here
  // e.g., upload to S3, rsync to server, etc.
  context.logger.info(`Uploading ${outputPath} to ${target} environment...`);
}

export default createBuilder(runBuilder);
```

### Step 3: Register in builders.json

```json
// my-builder/builders.json
{
  "$schema": "@angular-devkit/architect/src/builders-schema.json",
  "builders": {
    "deploy": {
      "implementation": "./src/my-builder/index",
      "schema": "./src/my-builder/schema.json",
      "description": "Deploy the Angular application"
    }
  }
}
```

### Step 4: Register in package.json

```json
// my-builder/package.json
{
  "name": "my-builder",
  "version": "1.0.0",
  "builders": "builders.json",
  "dependencies": {
    "@angular-devkit/architect": "^0.2100.0"
  }
}
```

### Step 5: Use in angular.json

```json
{
  "architect": {
    "deploy": {
      "builder": "my-builder:deploy",
      "options": {
        "outputPath": "dist/my-app",
        "target": "staging",
        "dryRun": false
      },
      "configurations": {
        "production": {
          "target": "production"
        }
      }
    }
  }
}
```

Run with:

```bash
ng run my-app:deploy
ng run my-app:deploy:production
```

## BuilderContext API Reference

The `BuilderContext` object provided to every builder has the following members:

| Member | Type | Description |
|--------|------|-------------|
| `workspaceRoot` | `string` | Absolute path to the workspace root |
| `currentDirectory` | `string` | Directory where `ng` was invoked |
| `target` | `Target` | The current target being executed (project, target, configuration) |
| `logger` | `Logger` | Logging methods: `debug`, `info`, `warn`, `error`, `fatal` |
| `reportStatus(status)` | `void` | Report a status string for the running builder |
| `reportProgress(current, total?, status?)` | `void` | Report numeric progress |
| `reportRunning()` | `void` | Report that the builder is running (for long tasks) |
| `scheduleTarget(target, overrides?)` | `Promise<BuilderRun>` | Schedule another architect target |
| `scheduleBuilder(builderName, options?)` | `Promise<BuilderRun>` | Schedule a builder by name |
| `getTargetOptions(target)` | `Promise<JsonObject>` | Get options for a target |
| `getBuilderNameForTarget(target)` | `Promise<string>` | Resolve builder name for a target |
| `validateOptions(options, builderName)` | `Promise<JsonObject>` | Validate options against schema |
| `addTeardown(teardown)` | `void` | Register a cleanup callback |

### Scheduling Other Builders

A powerful feature: builders can schedule other builders and compose workflows:

```typescript
async function buildAndDeploy(
  options: BuildAndDeploySchema,
  context: BuilderContext
): Promise<BuilderOutput> {
  // First, run the build
  const buildTarget = {
    target: 'build',
    project: context.target?.project ?? '',
    configuration: 'production'
  };

  context.logger.info('Starting build...');
  const buildRun = await context.scheduleTarget(buildTarget);
  const buildResult = await buildRun.result;

  if (!buildResult.success) {
    return { success: false, error: 'Build failed.' };
  }

  // Then deploy
  context.logger.info('Build succeeded. Starting deployment...');
  const deployRun = await context.scheduleBuilder('my-builder:deploy', {
    outputPath: options.outputPath,
    target: options.deployTarget
  });
  const deployResult = await deployRun.result;

  return deployResult;
}
```

### Watching with Observable Return

For builders that watch for changes (like `ng serve`), return an Observable:

```typescript
import { Observable } from 'rxjs';

function watchBuilder(
  options: WatchOptions,
  context: BuilderContext
): Observable<BuilderOutput> {
  return new Observable(subscriber => {
    const watcher = chokidar.watch(options.watchPath);

    watcher.on('change', async (path) => {
      context.logger.info(`File changed: ${path}`);
      try {
        await processFile(path);
        subscriber.next({ success: true });
      } catch (err) {
        subscriber.next({
          success: false,
          error: (err as Error).message
        });
      }
    });

    // Emit initial success
    subscriber.next({ success: true });

    // Cleanup when unsubscribed
    return () => watcher.close();
  });
}

export default createBuilder(watchBuilder);
```

## Third-Party Builder: @angular-builders/custom-esbuild

When you need deeper access to the esbuild and Vite configuration than the built-in `plugins` option provides, the `@angular-builders/custom-esbuild` package offers extended builders.

### Installation

```bash
npm install @angular-builders/custom-esbuild --save-dev
```

### Available Builders

| Builder | Extends | Extra Capabilities |
|---------|---------|-------------------|
| `@angular-builders/custom-esbuild:application` | `@angular/build:application` | Custom plugins, indexHtmlTransformer |
| `@angular-builders/custom-esbuild:dev-server` | `@angular/build:dev-server` | Custom Vite middlewares |

### Configuration

```json
{
  "architect": {
    "build": {
      "builder": "@angular-builders/custom-esbuild:application",
      "options": {
        "plugins": ["./esbuild/my-plugin.js"],
        "indexHtmlTransformer": "./esbuild/index-html-transformer.js",
        "outputPath": "dist/my-app",
        "index": "src/index.html",
        "browser": "src/main.ts",
        "tsConfig": "tsconfig.app.json"
      }
    },
    "serve": {
      "builder": "@angular-builders/custom-esbuild:dev-server",
      "options": {
        "middlewares": ["./vite/api-mock-middleware.js"]
      }
    }
  }
}
```

### Index HTML Transformer

Transform `index.html` during the build:

```typescript
// esbuild/index-html-transformer.ts
export default (indexHtml: string): string => {
  // Inject a CSP meta tag
  const cspTag = `<meta http-equiv="Content-Security-Policy" content="default-src 'self'">`;
  return indexHtml.replace('</head>', `  ${cspTag}\n</head>`);
};
```

Async transformations are also supported:

```typescript
// esbuild/index-html-transformer.ts
import { readFile } from 'fs/promises';

export default async (indexHtml: string): Promise<string> => {
  const version = JSON.parse(
    await readFile('package.json', 'utf-8')
  ).version;

  return indexHtml.replace(
    '</body>',
    `<script>window.__APP_VERSION__="${version}";</script>\n</body>`
  );
};
```

### Custom Vite Middleware

Add mock API endpoints or middleware to the dev server:

```javascript
// vite/api-mock-middleware.js
module.exports = (req, res, next) => {
  if (req.url === '/api/health') {
    res.setHeader('Content-Type', 'application/json');
    res.end(JSON.stringify({ status: 'ok' }));
    return;
  }
  next();
};
```

## Custom Build Pipelines (Without Angular CLI)

For rare cases where you need a fully custom pipeline, Angular supports two community tools:

### Analog (Vite Plugin)

Analog provides a Vite plugin that integrates the Angular compiler:

```typescript
// vite.config.ts
import { defineConfig } from 'vite';
import angular from '@analogjs/vite-plugin-angular';

export default defineConfig({
  plugins: [
    angular()
  ],
  build: {
    target: 'es2022'
  }
});
```

Use cases for Analog's Vite plugin:
- Existing Vite-based projects adding Angular
- File-based routing requirements
- Deep Vite ecosystem integration (Vitest, SSR patterns)

### Rspack Plugin

For projects coupled to the Webpack ecosystem:

```javascript
// rspack.config.js
const { createAngularRspackPlugin } = require('@aspect-build/angular-rspack');

module.exports = {
  plugins: [
    createAngularRspackPlugin({
      tsconfig: './tsconfig.app.json',
      entryPoints: ['./src/main.ts']
    })
  ]
};
```

Use cases for Rspack:
- Heavy reliance on Webpack-specific plugins
- Module Federation requirements
- Gradual migration from Webpack

### When NOT to Use Custom Pipelines

The Angular CLI's built-in builders cover the vast majority of use cases. Avoid custom pipelines unless you have:
- An existing non-Angular app adding Angular incrementally
- Hard requirements for Module Federation that native federation cannot meet
- An experimental project deliberately exploring alternative tooling

Custom pipelines lose:
- `ng update` automated migrations
- Schematics support
- Official Angular builder improvements
- Community alignment

## Migration from Webpack Builders

If your project still uses the legacy `@angular-devkit/build-angular:browser` (Webpack) builder, here is the migration path.

### Builder Mapping

| Legacy Builder | New Builder |
|---------------|-------------|
| `@angular-devkit/build-angular:browser` | `@angular/build:application` |
| `@angular-devkit/build-angular:browser-esbuild` | `@angular/build:application` |
| `@angular-devkit/build-angular:dev-server` | `@angular/build:dev-server` |

### Key Option Changes

| Old Option | New Option | Notes |
|-----------|------------|-------|
| `main` | `browser` | Entry point renamed |
| `ngswConfigPath` | `serviceWorker` | Now a string path directly |
| `resourcesOutputPath` | Removed | Handled automatically |
| `vendorChunk` | Removed | esbuild handles splitting |
| `commonChunk` | Removed | esbuild handles splitting |
| `buildOptimizer` | Removed | Always active in production |
| `deployUrl` | Removed | Use `<base href>` or `APP_BASE_HREF` |
| `scripts[].lazy` | `scripts[].inject: false` | Different syntax for lazy scripts |

### Common Migration Issues

**1. Tilde Imports in SASS**

```scss
/* BAD: Tilde import (Webpack-only) */
@use '~bootstrap/scss/bootstrap';

/* GOOD: Direct path (esbuild-compatible) */
@use 'bootstrap/scss/bootstrap';
```

**2. CommonJS Dependencies**

esbuild warns about CommonJS modules. If you see warnings:

```
Warning: Module 'some-package' used a CommonJS pattern
```

Fix by enabling `esModuleInterop` in `tsconfig.json`:

```json
{
  "compilerOptions": {
    "esModuleInterop": true
  }
}
```

Or suppress the warning if the package has no ESM build:

```json
{
  "build": {
    "options": {
      "allowedCommonJsDependencies": [
        "some-package"
      ]
    }
  }
}
```

**3. Custom Webpack Configuration**

If you used `@angular-builders/custom-webpack`, switch to `@angular-builders/custom-esbuild` or use the built-in `plugins` option:

```json
// Before (Webpack)
```

## Testing Custom Builders

Use the Architect test harness for integration testing:

```typescript
// my-builder/src/my-builder/index.spec.ts
import { Architect } from '@angular-devkit/architect';
import { TestingArchitectHost } from '@angular-devkit/architect/testing';
import { schema } from '@angular-devkit/core';
import { join } from 'path';

describe('My Builder', () => {
  let architect: Architect;
  let architectHost: TestingArchitectHost;

  beforeEach(async () => {
    const registry = new schema.CoreSchemaRegistry();
    registry.addPostTransform(schema.transforms.addUndefinedDefaults);

    architectHost = new TestingArchitectHost(
      join(__dirname, '../../test-workspace'),  // workspace root
      join(__dirname, '../../test-workspace')   // current dir
    );

    architect = new Architect(architectHost, registry);

    // Register the builder
    await architectHost.addBuilderFromPackage(join(__dirname, '../..'));
  });

  it('should deploy successfully', async () => {
    const run = await architect.scheduleBuilder(
      'my-builder:deploy',
      {
        outputPath: 'dist/test-app',
        target: 'staging',
        dryRun: true
      }
    );

    const output = await run.result;
    expect(output.success).toBe(true);

    await run.stop();
  });

  it('should fail when output path does not exist', async () => {
    const run = await architect.scheduleBuilder(
      'my-builder:deploy',
      {
        outputPath: 'nonexistent/path',
        target: 'staging',
        dryRun: false
      }
    );

    const output = await run.result;
    expect(output.success).toBe(false);
    expect(output.error).toContain('does not exist');

    await run.stop();
  });
});
```

## Common Mistakes

- **Writing esbuild plugins that depend on Webpack APIs.** esbuild plugins have a completely different API from Webpack loaders and plugins. The `onResolve`, `onLoad`, `onStart`, and `onEnd` hooks are the primary extension points. There are no equivalents to Webpack's `module.rules` or `plugins` array in esbuild.

- **Assuming Vite configuration is directly accessible.** Angular's use of Vite is encapsulated within the CLI dev server builder. You cannot drop a `vite.config.ts` in your project root and expect Angular to pick it up. To customize Vite behavior, use the `@angular-builders/custom-esbuild:dev-server` builder with middleware options, or switch to a fully custom pipeline with Analog.

- **Using `define` values without proper quoting.** Values in the `define` option are raw JavaScript expressions. If you write `"API_URL": "https://example.com"`, esbuild interprets `https://example.com` as JavaScript (a labeled statement with a comment). You must write `"API_URL": "'https://example.com'"` with inner quotes for string values.

- **Forgetting TypeScript declarations for defined constants.** If you use `define` or inject constants via plugins, TypeScript will report errors unless you declare the types with `declare const` statements or ambient module declarations.

- **Reaching for third-party builders when built-in options suffice.** The native `plugins`, `loader`, and `define` options in the application builder cover most customization needs. Only install `@angular-builders/custom-esbuild` if you need `indexHtmlTransformer` or Vite middleware.

---

## Best Practices

- **Use the built-in `plugins` option first.** Before installing third-party packages, check if the native application builder's `plugins`, `loader`, and `define` options meet your needs. They handle 80% of customization requirements.

- **Keep esbuild plugins simple and focused.** Each plugin should do one thing. If you need multiple modifications, create separate plugins rather than one monolithic plugin that handles everything.

- **Use factory functions for configurable plugins.** Export a factory function that accepts options rather than hardcoding values. This lets you pass configuration through `angular.json` and reuse the plugin across projects.

- **Declare all build-time constants in a single type declaration file.** Create a `src/build-env.d.ts` or `src/env.d.ts` that declares all `define` constants. This keeps TypeScript happy and documents what is available at build time.

- **Test custom builders with the Architect test harness.** Do not test builders by running `ng build` in a child process. The `TestingArchitectHost` provides a proper testing environment that validates schemas and simulates the full builder lifecycle.

- **Version and publish reusable builders as separate packages.** If a builder is used across multiple projects, publish it to a private registry. Include proper `builders.json`, JSON schemas, and documentation.

## Real-World Use Cases

### 1. Feature Flag Injection at Build Time

A SaaS product builds separate bundles for different customer tiers:

```json
// angular.json
{
  "configurations": {
    "enterprise": {
      "define": {
        "FEATURE_AUDIT_LOG": "true",
        "FEATURE_SSO": "true",
        "FEATURE_CUSTOM_BRANDING": "true",
        "MAX_USERS": "999999"
      }
    },
    "starter": {
      "define": {
        "FEATURE_AUDIT_LOG": "false",
        "FEATURE_SSO": "false",
        "FEATURE_CUSTOM_BRANDING": "false",
        "MAX_USERS": "10"
      }
    }
  }
}
```

```typescript
// src/app/features/feature-gate.ts
declare const FEATURE_AUDIT_LOG: boolean;
declare const FEATURE_SSO: boolean;

export function isFeatureEnabled(feature: string): boolean {
  const flags: Record<string, boolean> = {
    'audit-log': FEATURE_AUDIT_LOG,
    'sso': FEATURE_SSO
  };
  return flags[feature] ?? false;
}
```

```bash
ng build --configuration=enterprise
ng build --configuration=starter
```

Tree shaking eliminates dead code branches. If `FEATURE_SSO` is `false`, esbuild removes the entire SSO module from the starter bundle.

### 2. CI/CD Deploy Builder

A custom builder that handles the build-and-deploy workflow in a single `ng` command:

```typescript
// builders/deploy-s3/index.ts
import { createBuilder, BuilderContext, BuilderOutput } from '@angular-devkit/architect';
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';
import { readdir, readFile } from 'fs/promises';
import { join } from 'path';

interface DeployS3Options {
  buildTarget: string;
  bucket: string;
  region: string;
  prefix: string;
}

async function deployS3Builder(
  options: DeployS3Options,
  context: BuilderContext
): Promise<BuilderOutput> {
  // Schedule the build first
  const buildRun = await context.scheduleTarget({
    target: 'build',
    project: context.target!.project,
    configuration: 'production'
  });
  const buildResult = await buildRun.result;

  if (!buildResult.success) {
    return { success: false, error: 'Build failed' };
  }

  // Upload to S3
  const s3 = new S3Client({ region: options.region });
  const outputDir = join(context.workspaceRoot, 'dist', context.target!.project, 'browser');
  const files = await readdir(outputDir, { recursive: true, withFileTypes: true });

  let uploaded = 0;
  const fileEntries = files.filter(f => f.isFile());

  for (const file of fileEntries) {
    const filePath = join(file.parentPath, file.name);
    const key = `${options.prefix}/${filePath.replace(outputDir, '').replace(/\\/g, '/')}`;

    await s3.send(new PutObjectCommand({
      Bucket: options.bucket,
      Key: key,
      Body: await readFile(filePath)
    }));

    uploaded++;
    context.reportProgress(uploaded, fileEntries.length, `Uploading ${file.name}`);
  }

  context.logger.info(`Uploaded ${uploaded} files to s3://${options.bucket}/${options.prefix}`);
  return { success: true };
}

export default createBuilder(deployS3Builder);
```

### 3. Build Analytics Plugin

Track build times and bundle sizes in a central dashboard:

```javascript
// esbuild/analytics-plugin.js
const https = require('https');

module.exports = (options) => ({
  name: 'build-analytics',
  setup(build) {
    let startTime;

    build.onStart(() => {
      startTime = Date.now();
    });

    build.onEnd((result) => {
      const duration = Date.now() - startTime;
      const outputFiles = result.outputFiles || [];
      const totalSize = outputFiles.reduce((sum, f) => sum + f.contents.length, 0);

      const payload = JSON.stringify({
        project: options.project,
        timestamp: new Date().toISOString(),
        duration,
        totalSize,
        errors: result.errors.length,
        warnings: result.warnings.length,
        ci: !!process.env.CI,
        branch: process.env.GITHUB_REF_NAME || 'unknown'
      });

      // Fire and forget to analytics endpoint
      const req = https.request(options.endpoint, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' }
      });
      req.write(payload);
      req.end();
    });
  }
});
```
