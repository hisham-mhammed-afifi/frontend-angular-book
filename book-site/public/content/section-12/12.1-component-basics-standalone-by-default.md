# 12.1 Component Basics (Standalone by Default)

> Components are the fundamental building blocks of Angular applications. Every piece of UI you see is rendered by a component. In Angular 19+, all components are standalone by default, meaning they manage their own dependencies without NgModules.

---

## What Is a Component?

A component is a TypeScript class decorated with `@Component` that controls a piece of the UI. It combines:

- **A TypeScript class** that holds state and behavior
- **An HTML template** that defines the view
- **Optional CSS styles** scoped to that component
- **Metadata** (the decorator) that tells Angular how to process it

```typescript
import { Component, signal } from '@angular/core';

@Component({
  selector: 'app-greeting',
  template: `<h1>Hello, {{ name() }}!</h1>`,
  styles: `h1 { color: #333; font-family: sans-serif; }`,
})
export class GreetingComponent {
  name = signal('World');
}
```

This component can be used in any other component's template:

```html
<app-greeting />
```

---

## The @Component Decorator

The `@Component` decorator accepts a metadata object that configures the component. Here is every property you will commonly use:

| Property | Required | Description |
|----------|----------|-------------|
| `selector` | Yes | CSS selector that identifies this component in templates |
| `template` | Yes* | Inline HTML template |
| `templateUrl` | Yes* | Path to an external HTML template file |
| `styles` | No | Inline CSS styles (string or array of strings) |
| `styleUrl` | No | Path to an external CSS file |
| `styleUrls` | No | Array of paths to external CSS files |
| `imports` | No | Standalone dependencies (components, directives, pipes) used in the template |
| `providers` | No | Dependency injection providers scoped to this component |
| `viewProviders` | No | Providers visible only to the component's view children (not projected content) |
| `changeDetection` | No | `ChangeDetectionStrategy.Default` or `OnPush` |
| `encapsulation` | No | `ViewEncapsulation.Emulated` (default), `ShadowDom`, or `None` |
| `host` | No | Host element bindings (properties, attributes, events) |
| `standalone` | No | Defaults to `true` in Angular 19+. Set to `false` only for NgModule components. |
| `exportAs` | No | Name used to reference this component in template reference variables |
| `animations` | No | List of animation trigger definitions |
| `schemas` | No | Schemas allowing non-Angular elements/properties (e.g., `CUSTOM_ELEMENTS_SCHEMA`) |

*Either `template` or `templateUrl` must be provided, but not both.

## Generating a Component with the CLI

The Angular CLI generates components with all necessary files:

```bash
# Generate a component (standalone by default in Angular 19+)
ng generate component user-profile

# Shorthand
ng g c user-profile
```

This creates:

```
src/app/user-profile/
  user-profile.component.ts      # Component class
  user-profile.component.html    # Template
  user-profile.component.css     # Styles
  user-profile.component.spec.ts # Test file
```

### Useful CLI Flags

```bash
# Inline template and styles (no separate HTML/CSS files)
ng g c user-avatar --inline-template --inline-style

# Skip test file
ng g c user-avatar --skip-tests

# Generate in a specific directory
ng g c features/dashboard/widgets/stats-card

# Flat (no subdirectory)
ng g c shared/button --flat

# With a custom prefix
ng g c admin-header --prefix=admin
```

### Configuring Defaults in angular.json

Set project-wide defaults for component generation:

```json
{
  "schematics": {
    "@schematics/angular:component": {
      "style": "scss",
      "skipTests": false,
      "changeDetection": "OnPush",
      "prefix": "app"
    }
  }
}
```

## The Selector

The selector is a CSS selector that Angular uses to find and instantiate the component in templates.

```typescript
@Component({
  selector: 'app-user-card',  // Element selector (most common)
  template: `...`,
})
export class UserCardComponent {}
```

### Selector Types

```typescript
// Element selector (recommended for components)
selector: 'app-user-card'
// Usage: <app-user-card />

// Attribute selector (common for directives, rare for components)
selector: '[appHighlight]'
// Usage: <div appHighlight>...</div>

// Class selector (rarely used)
selector: '.app-tooltip'
// Usage: <div class="app-tooltip">...</div>
```

### Naming Conventions

- Use a **custom prefix** for all selectors to avoid conflicts with HTML elements and third-party libraries
- The prefix is configured in `angular.json` under `prefix` (default: `"app"`)
- Use **kebab-case**: `app-user-card`, not `appUserCard` or `AppUserCard`
- The component class name uses **PascalCase** with a `Component` suffix: `UserCardComponent`

```typescript
// GOOD: Custom prefix, kebab-case
@Component({ selector: 'app-user-card' })
export class UserCardComponent {}

// GOOD: Feature-area prefix
@Component({ selector: 'admin-user-list' })
export class UserListComponent {}

// BAD: No prefix (risks collision with HTML or third-party elements)
@Component({ selector: 'user-card' })
export class UserCardComponent {}

// BAD: PascalCase selector
@Component({ selector: 'UserCard' })
export class UserCardComponent {}
```

## Templates: Inline vs External

### Inline Template

Use the `template` property with a template literal:

```typescript
@Component({
  selector: 'app-badge',
  template: `
    <span class="badge" [class]="variant()">
      <ng-content />
    </span>
  `,
})
export class BadgeComponent {
  variant = input<'success' | 'warning' | 'error'>('success');
}
```

**When to use inline:** Small components with fewer than ~10 lines of HTML. Keeps everything in one file for quick scanning.

### External Template

Use the `templateUrl` property pointing to an HTML file:

```typescript
@Component({
  selector: 'app-dashboard',
  templateUrl: './dashboard.component.html',
  styleUrl: './dashboard.component.scss',
})
export class DashboardComponent {}
```

**When to use external:** Any component with non-trivial HTML. Separates concerns and makes templates easier to edit.

### Rule of Thumb

If the template fits on a screen without scrolling, inline is fine. If you need to scroll, use a separate file. Most real-world components use external templates.

## Styles: Scoping and Options

### Inline Styles

```typescript
@Component({
  selector: 'app-card',
  template: `<div class="card"><ng-content /></div>`,
  styles: `
    .card {
      border: 1px solid #ddd;
      border-radius: 8px;
      padding: 16px;
    }
  `,
})
export class CardComponent {}
```

### External Styles

```typescript
@Component({
  selector: 'app-card',
  templateUrl: './card.component.html',
  styleUrl: './card.component.scss', // Single file
})
export class CardComponent {}
```

### Multiple Style Files

```typescript
@Component({
  styleUrls: [
    './card.component.scss',
    './card-animations.scss',
  ],
})
export class CardComponent {}
```

### ViewEncapsulation

Angular scopes component styles by default so they do not leak to other components.

| Mode | Behavior |
|------|----------|
| `Emulated` (default) | Angular adds unique attributes (`_ngcontent-abc123`) to elements and rewrites CSS selectors to match. Styles are scoped but no Shadow DOM is used. |
| `ShadowDom` | Uses the browser's native Shadow DOM. Full style isolation. Some third-party CSS may not penetrate the shadow boundary. |
| `None` | Styles are added to the global `<head>` with no scoping. Affects the entire application. |

```typescript
import { ViewEncapsulation } from '@angular/core';

@Component({
  selector: 'app-global-styles',
  template: `<p>These styles are global</p>`,
  styles: `p { color: red; }`,
  encapsulation: ViewEncapsulation.None, // Careful: affects all <p> elements
})
export class GlobalStylesComponent {}
```

In practice, keep the default `Emulated` encapsulation. Use `None` sparingly (e.g., for a theme component that intentionally sets global CSS variables). Use `ShadowDom` for web component interop or strict isolation requirements.

## Imports: Declaring Template Dependencies

Standalone components must explicitly import every component, directive, and pipe they use in their template.

```typescript
import { Component } from '@angular/core';
import { DatePipe, CurrencyPipe } from '@angular/common';
import { RouterLink } from '@angular/router';
import { UserAvatarComponent } from '../user-avatar/user-avatar.component';
import { HighlightDirective } from '../../shared/highlight.directive';

@Component({
  selector: 'app-user-card',
  imports: [
    DatePipe,
    CurrencyPipe,
    RouterLink,
    UserAvatarComponent,
    HighlightDirective,
  ],
  template: `
    <div appHighlight>
      <app-user-avatar [src]="user().avatar" />
      <h3>
        <a [routerLink]="['/users', user().id]">{{ user().name }}</a>
      </h3>
      <p>Joined: {{ user().joinDate | date:'mediumDate' }}</p>
      <p>Balance: {{ user().balance | currency }}</p>
    </div>
  `,
})
export class UserCardComponent {
  user = input.required<User>();
}
```

### What Goes in Imports

| Item | Example |
|------|---------|
| Other standalone components | `UserAvatarComponent` |
| Standalone directives | `RouterLink`, `RouterLinkActive`, `NgClass` |
| Standalone pipes | `DatePipe`, `CurrencyPipe`, `AsyncPipe` |
| NgModules (for legacy interop) | `FormsModule`, `ReactiveFormsModule` |

### What Does NOT Go in Imports

- **Services** (use `inject()` or constructor injection)
- **Built-in control flow** (`@if`, `@for`, `@switch` need no import)
- **Signal APIs** (`signal`, `computed`, `effect`, `input`, `output`, `model` are imported in TypeScript, not in the decorator)

```typescript
// BAD: Importing CommonModule just for @if/@for (not needed since Angular 17)
@Component({
  imports: [CommonModule], // Unnecessary if using @if/@for/@switch
  template: `
    @if (isVisible()) {
      <p>Hello</p>
    }
  `,
})

// GOOD: No imports needed for built-in control flow
@Component({
  template: `
    @if (isVisible()) {
      <p>Hello</p>
    }
  `,
})
```

## Component Class: State and Behavior

The component class is where you define state (signals, properties) and behavior (methods).

### Signal-Based State (Modern)

```typescript
@Component({
  selector: 'app-counter',
  template: `
    <p>Count: {{ count() }}</p>
    <p>Doubled: {{ doubled() }}</p>
    <button (click)="increment()">+1</button>
    <button (click)="reset()">Reset</button>
  `,
})
export class CounterComponent {
  // Writable signal for local state
  count = signal(0);

  // Computed signal for derived state
  doubled = computed(() => this.count() * 2);

  // Methods for behavior
  increment() {
    this.count.update(n => n + 1);
  }

  reset() {
    this.count.set(0);
  }
}
```

### Injecting Dependencies

Use the `inject()` function in field initializers or the constructor:

```typescript
@Component({
  selector: 'app-product-list',
  template: `
    @for (product of products(); track product.id) {
      <app-product-card [product]="product" />
    }
  `,
  imports: [ProductCardComponent],
})
export class ProductListComponent {
  private productService = inject(ProductService);
  private router = inject(Router);

  products = toSignal(
    this.productService.getAll(),
    { initialValue: [] }
  );

  navigateToProduct(id: string) {
    this.router.navigate(['/products', id]);
  }
}
```

## Self-Closing Component Tags

Since Angular 15.1, you can use self-closing tags for components that do not project content:

```html
<!-- Self-closing (preferred when no content is projected) -->
<app-spinner />
<app-user-avatar [src]="avatarUrl()" />
<app-divider />

<!-- Equivalent long form -->
<app-spinner></app-spinner>
<app-user-avatar [src]="avatarUrl()"></app-user-avatar>
<app-divider></app-divider>

<!-- Components WITH projected content must use open/close tags -->
<app-card>
  <h2>Card Title</h2>
  <p>Card body content goes here.</p>
</app-card>
```

Self-closing tags reduce template noise and improve readability. Use them whenever a component has no children.

## Host Element

Every component has a **host element**: the DOM element that matches the component's selector. You can bind properties, attributes, classes, styles, and events on the host element using the `host` metadata property.

```typescript
@Component({
  selector: 'app-alert',
  host: {
    'role': 'alert',
    '[class.alert-success]': 'type() === "success"',
    '[class.alert-error]': 'type() === "error"',
    '[class.alert-warning]': 'type() === "warning"',
    '[attr.aria-live]': '"polite"',
  },
  template: `<ng-content />`,
  styles: `
    :host {
      display: block;
      padding: 12px 16px;
      border-radius: 4px;
      margin-bottom: 8px;
    }
    :host(.alert-success) { background: #d4edda; color: #155724; }
    :host(.alert-error) { background: #f8d7da; color: #721c24; }
    :host(.alert-warning) { background: #fff3cd; color: #856404; }
  `,
})
export class AlertComponent {
  type = input<'success' | 'error' | 'warning'>('success');
}

// Usage:
// <app-alert type="error">Something went wrong!</app-alert>
// Renders: <app-alert role="alert" class="alert-error" aria-live="polite">...</app-alert>
```

### The :host Selector

In component styles, `:host` targets the host element itself:

```css
/* Targets the component's host element */
:host {
  display: block;
  border: 1px solid #ccc;
}

/* Conditional styling based on host classes */
:host(.compact) {
  padding: 4px;
}

/* When a parent has a specific class */
:host-context(.dark-theme) {
  background: #333;
  color: #fff;
}
```

## Providers: Component-Scoped Services

Adding a service to a component's `providers` creates a new instance of that service for each instance of the component. This instance is shared with all child components.

```typescript
@Component({
  selector: 'app-editor',
  providers: [UndoRedoService], // Each editor gets its own instance
  template: `
    <div class="toolbar">
      <button (click)="undo()">Undo</button>
      <button (click)="redo()">Redo</button>
    </div>
    <textarea (input)="onInput($event)"></textarea>
  `,
})
export class EditorComponent {
  private undoRedo = inject(UndoRedoService);

  undo() { this.undoRedo.undo(); }
  redo() { this.undoRedo.redo(); }
  onInput(event: Event) {
    const text = (event.target as HTMLTextAreaElement).value;
    this.undoRedo.push(text);
  }
}
```

Use component-level providers when:
- Each component instance needs its own service instance (undo/redo, form state, local cache)
- The service lifecycle should match the component lifecycle (created when the component is created, destroyed when it is destroyed)

Do NOT use component-level providers for application-wide singletons. Use `providedIn: 'root'` for those.

## Change Detection Strategy

```typescript
import { ChangeDetectionStrategy } from '@angular/core';

@Component({
  selector: 'app-static-card',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <div class="card">
      <h2>{{ title() }}</h2>
      <ng-content />
    </div>
  `,
})
export class StaticCardComponent {
  title = input.required<string>();
}
```

| Strategy | Behavior |
|----------|----------|
| `Default` | Component is checked in every change detection cycle |
| `OnPush` | Component is checked only when inputs change (by reference), signals update, events fire inside the component, or `markForCheck()` is called |

With **zoneless** Angular (default in Angular 21), change detection is driven by signals. The `OnPush` strategy becomes less critical because Angular already knows exactly which components need checking. However, `OnPush` is still beneficial for:
- Legacy codebases still using Zone.js
- Third-party library components that may trigger zone-based change detection

## Component Communication Summary

Components communicate through several mechanisms (each covered in depth in later subtopics):

| Mechanism | Direction | API |
|-----------|-----------|-----|
| Signal Inputs | Parent to child | `input()`, `input.required()` |
| Signal Outputs | Child to parent | `output()` |
| Two-Way Binding | Both | `model()` |
| Content Projection | Parent to child (DOM) | `<ng-content>` |
| Template Queries | Parent reads child | `viewChild()`, `viewChildren()`, `contentChild()`, `contentChildren()` |
| Dependency Injection | Any to any | `inject()` with shared services |

## Common Mistakes

### 1. Forgetting to Add Dependencies to Imports

```typescript
// BAD: Uses RouterLink in template but doesn't import it
@Component({
  // imports: [], <-- missing RouterLink
  template: `<a [routerLink]="['/home']">Home</a>`,
})
export class NavComponent {} // RouterLink silently ignored or error thrown

// GOOD: Declare all template dependencies
@Component({
  imports: [RouterLink],
  template: `<a [routerLink]="['/home']">Home</a>`,
})
export class NavComponent {}
```

### 2. Using standalone: false Unnecessarily

```typescript
// BAD: Setting standalone: false for new components
@Component({
  standalone: false, // Only do this for legacy NgModule-based components
})
export class NewComponent {}

// GOOD: Omit standalone entirely (defaults to true in Angular 19+)
@Component({
  selector: 'app-new',
  template: `...`,
})
export class NewComponent {}
```

### 3. Importing CommonModule for Built-In Control Flow

```typescript
// BAD: CommonModule imported but only using @if/@for
@Component({
  imports: [CommonModule],
  template: `@if (show()) { <p>Visible</p> }`,
})

// GOOD: No imports needed for @if/@for/@switch
@Component({
  template: `@if (show()) { <p>Visible</p> }`,
})
```

Only import `CommonModule` (or individual pipes/directives from `@angular/common`) when you actually use something from it, like `DatePipe`, `CurrencyPipe`, `NgClass`, or `AsyncPipe`.

### 4. Massive Component Files

```typescript
// BAD: 500-line component doing everything
@Component({
  selector: 'app-dashboard',
  template: `...300 lines of HTML...`,
})
export class DashboardComponent {
  // 200 lines of logic covering charts, tables, filters, modals...
}

// GOOD: Break into focused child components
// app-dashboard (orchestrator)
//   app-stats-cards (display stats)
//   app-recent-orders (display orders table)
//   app-activity-feed (display activity)
```

### 5. Using ViewEncapsulation.None Without Awareness

```typescript
// BAD: Global styles that affect the whole app
@Component({
  encapsulation: ViewEncapsulation.None,
  styles: `
    p { margin: 0; }       /* Affects ALL <p> elements */
    .btn { color: red; }   /* Affects ALL .btn elements */
  `,
})
```

### 6. Not Using a Selector Prefix

```typescript
// BAD: No prefix, might conflict with HTML or third-party components
@Component({ selector: 'button' })      // Overrides native <button>!
@Component({ selector: 'card' })         // Might conflict with a UI library

// GOOD: Unique prefix
@Component({ selector: 'app-button' })
@Component({ selector: 'app-card' })
```

## Best Practices

### 1. Keep Components Small and Focused

A component should do one thing well. If it manages a form, a table, and a modal, split it into three components. The parent orchestrates; children handle specific UI concerns.

### 2. Use Signal-Based APIs for All New Code

Use `input()` instead of `@Input()`, `output()` instead of `@Output()`, `model()` for two-way binding, and `signal()`/`computed()` for internal state.

### 3. Use External Templates for Non-Trivial Components

If your template has more than a few lines, put it in a separate `.html` file. It makes both the template and the class easier to read and work with.

### 4. Name Selectors with a Prefix

Configure the prefix in `angular.json` and enforce it with linting rules. This prevents selector collisions across libraries and native HTML.

### 5. Default to Standalone

Never set `standalone: false` on new components. The standalone model is simpler, better for tree-shaking, and aligns with Angular's future direction.

### 6. Use the host Property for Host Element Bindings

Keep all host element bindings (classes, styles, attributes, events) in the `host` metadata property rather than using `@HostBinding`/`@HostListener` decorators.

## Real-World Use Cases

### Reusable Button Component

```typescript
@Component({
  selector: 'app-button',
  host: {
    '[class]': '"btn btn-" + variant()',
    '[class.btn-loading]': 'loading()',
    '[attr.disabled]': 'disabled() || loading() ? "" : null',
    '[attr.aria-busy]': 'loading()',
    '(click)': 'handleClick($event)',
  },
  template: `
    @if (loading()) {
      <span class="spinner"></span>
    }
    <ng-content />
  `,
  styles: `
    :host {
      display: inline-flex;
      align-items: center;
      gap: 8px;
      cursor: pointer;
    }
    :host([disabled]) {
      opacity: 0.6;
      cursor: not-allowed;
      pointer-events: none;
    }
    .spinner {
      width: 16px;
      height: 16px;
      border: 2px solid currentColor;
      border-right-color: transparent;
      border-radius: 50%;
      animation: spin 0.6s linear infinite;
    }
    @keyframes spin { to { transform: rotate(360deg); } }
  `,
})
export class ButtonComponent {
  variant = input<'primary' | 'secondary' | 'danger'>('primary');
  disabled = input(false);
  loading = input(false);
  clicked = output<MouseEvent>();

  handleClick(event: MouseEvent) {
    if (!this.disabled() && !this.loading()) {
      this.clicked.emit(event);
    }
  }
}

// Usage:
// <app-button variant="primary" [loading]="isSaving()" (clicked)="save()">
//   Save Changes
// </app-button>
```

### Page Container with Layout Slots

```typescript
@Component({
  selector: 'app-page',
  template: `
    <header class="page-header">
      <ng-content select="[page-title]" />
      <div class="page-actions">
        <ng-content select="[page-actions]" />
      </div>
    </header>
    <main class="page-content">
      <ng-content />
    </main>
  `,
  styles: `
    :host { display: flex; flex-direction: column; height: 100%; }
    .page-header {
      display: flex; justify-content: space-between; align-items: center;
      padding: 16px 24px; border-bottom: 1px solid #eee;
    }
    .page-content { flex: 1; padding: 24px; overflow-y: auto; }
  `,
})
export class PageComponent {}

// Usage:
// <app-page>
//   <h1 page-title>User Management</h1>
//   <div page-actions>
//     <app-button variant="primary" (clicked)="addUser()">Add User</app-button>
//   </div>
//
//   <!-- Default slot: page content -->
//   <app-user-table [users]="users()" />
// </app-page>
```

### Feature Component with Scoped State

```typescript
@Component({
  selector: 'app-shopping-cart',
  providers: [CartStateService], // Each cart widget gets its own state
  imports: [CurrencyPipe, CartItemComponent],
  template: `
    @let items = cartState.items();
    @let total = cartState.total();

    <h2>Shopping Cart ({{ items.length }})</h2>

    @for (item of items; track item.productId) {
      <app-cart-item
        [item]="item"
        (removed)="cartState.removeItem(item.productId)"
        (quantityChanged)="cartState.updateQuantity(item.productId, $event)"
      />
    } @empty {
      <p class="empty">Your cart is empty.</p>
    }

    @if (items.length) {
      <footer class="cart-footer">
        <span class="total">Total: {{ total | currency }}</span>
        <app-button variant="primary" (clicked)="checkout()">Checkout</app-button>
      </footer>
    }
  `,
})
export class ShoppingCartComponent {
  protected cartState = inject(CartStateService);
  private router = inject(Router);

  checkout() {
    this.router.navigate(['/checkout']);
  }
}
```

> **Key Takeaway:** An Angular component combines a TypeScript class, an HTML template, and scoped styles into a self-contained UI building block. In Angular 19+, components are standalone by default: they declare their template dependencies via `imports`, need no NgModule, and work seamlessly with signals, the new control flow, and zoneless change detection.
