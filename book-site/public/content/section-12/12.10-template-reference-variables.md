# 12.10 Template Reference Variables

> Template reference variables (`#ref`) give you direct access to DOM elements, component instances, and directive instances within the template, enabling element-to-element communication without involving the component class.

---

## Declaring a Template Reference Variable

A template reference variable is declared with the hash symbol (`#`) followed by a name, placed as an attribute on any template element.

```html
<input #nameInput type="text" />
<button (click)="greet(nameInput.value)">Greet</button>
```

In this example, `#nameInput` creates a variable called `nameInput` that refers to the `<input>` DOM element. You can use `nameInput.value` elsewhere in the same template without involving the component class.

### Alternative syntax: ref-

Angular also supports the `ref-` prefix as an alternative to `#`:

```html
<input ref-nameInput type="text" />
```

This is rarely used. The `#` syntax is the community standard.

---

## What a Template Reference Variable Points To

The value assigned to the variable depends on the element it is placed on:

| Element Type | Variable Value |
|---|---|
| Plain HTML element (`<div>`, `<input>`, etc.) | The `HTMLElement` instance (DOM node) |
| Angular component (`<app-chart>`) | The component class instance |
| `<ng-template>` | A `TemplateRef` instance |
| Element with a directive that has `exportAs` | The directive instance (when using `#ref="exportAs"`) |

### On a plain HTML element

```html
<input #emailInput type="email" />
<p>Value: {{ emailInput.value }}</p>
<button (click)="emailInput.focus()">Focus</button>
```

`emailInput` is the native `HTMLInputElement`. You can call any DOM method or read any DOM property directly.

### On a component

```html
<app-counter #counter />
<p>Current count: {{ counter.count() }}</p>
<button (click)="counter.increment()">Increment from parent template</button>
```

`counter` is the `CounterComponent` instance. You can access its public properties, signals, and methods.

```typescript
@Component({
  selector: 'app-counter',
  standalone: true,
  template: `<span>{{ count() }}</span>`
})
export class CounterComponent {
  count = signal(0);

  increment(): void {
    this.count.update(n => n + 1);
  }
}
```

### On ng-template

```html
<ng-template #greetingTpl let-name>
  <p>Hello, {{ name }}!</p>
</ng-template>
```

`greetingTpl` is a `TemplateRef`. This is used with structural patterns like `ngTemplateOutlet` or signal queries for dynamic rendering.

### On a directive with exportAs

```html
<form #myForm="ngForm">
  <input name="email" ngModel required #email="ngModel" />
  <p>Form valid: {{ myForm.valid }}</p>
  <p>Email valid: {{ email.valid }}</p>
  <p>Email errors: {{ email.errors | json }}</p>
</form>
```

When you write `#ref="directiveName"`, Angular looks for a directive on that element whose `exportAs` metadata matches the string. The variable then points to that directive instance instead of the DOM element.

The `exportAs` property is defined in the directive decorator:

```typescript
@Directive({
  selector: '[ngModel]',
  exportAs: 'ngModel', // This name is used with #ref="ngModel"
})
export class NgModel { /* ... */ }
```

## exportAs in Detail

### How exportAs works

Any directive or component can define an `exportAs` name in its metadata. This allows template authors to grab the directive instance via a template reference variable.

```typescript
@Directive({
  selector: '[appTooltip]',
  standalone: true,
  exportAs: 'tooltip',
})
export class TooltipDirective {
  message = input.required<string>();

  show(): void { /* display tooltip */ }
  hide(): void { /* hide tooltip */ }
  isVisible = signal(false);
}
```

```html
<button appTooltip message="Click to save" #tip="tooltip">
  Save
</button>
<p>Tooltip visible: {{ tip.isVisible() }}</p>
<button (click)="tip.show()">Show tooltip programmatically</button>
```

Without `exportAs`, `#tip` on a `<button>` would be the `HTMLButtonElement`. With `#tip="tooltip"`, it becomes the `TooltipDirective` instance.

### Components with exportAs

Components can also use `exportAs`:

```typescript
@Component({
  selector: 'app-dropdown',
  standalone: true,
  exportAs: 'dropdown',
  template: `
    @if (isOpen()) {
      <div class="dropdown-panel">
        <ng-content />
      </div>
    }
  `
})
export class DropdownComponent {
  isOpen = signal(false);
  toggle(): void { this.isOpen.update(v => !v); }
  open(): void { this.isOpen.set(true); }
  close(): void { this.isOpen.set(false); }
}
```

```html
<app-dropdown #dd="dropdown">
  <ul>
    <li>Option 1</li>
    <li>Option 2</li>
  </ul>
</app-dropdown>
<button (click)="dd.toggle()">Toggle Dropdown</button>
```

Note: When `#ref` is placed on a component without `="exportName"`, it defaults to the component instance. `exportAs` is only needed when the element could match multiple directives and you need to be explicit, or when used on a plain HTML element with a directive.

## Scope Rules

### Template-wide scope (with caveats)

A template reference variable is accessible anywhere in the same template, even before its declaration in the DOM order:

```html
<!-- Using nameInput BEFORE its declaration in DOM order works -->
<p>Name: {{ nameInput.value }}</p>
<input #nameInput type="text" value="Angular" />
```

Angular processes the entire template before rendering, so forward references are valid.

### Structural directive scope boundaries

When a template reference variable is declared inside a structural block (`@if`, `@for`, `@switch`, or legacy `*ngIf`, `*ngFor`), it is scoped to that block. It is not accessible outside of it.

```html
@if (showInput()) {
  <input #conditionalInput type="text" />
  <!-- conditionalInput is accessible here -->
  <p>{{ conditionalInput.value }}</p>
}

<!-- ERROR: conditionalInput is NOT accessible here -->
<!-- <p>{{ conditionalInput.value }}</p> -->
```

Each structural block creates its own embedded view with its own scope. Variables declared inside cannot leak out.

### @for loop variables

Inside `@for`, each iteration has its own scope. A template reference variable declared in the loop body refers to the element in that specific iteration:

```html
@for (item of items(); track item.id) {
  <input #itemInput [value]="item.name" />
  <!-- itemInput refers to THIS iteration's input -->
  <button (click)="updateName(item.id, itemInput.value)">Save</button>
}
```

This works because each `@for` iteration creates its own embedded view. The `#itemInput` in each iteration is a separate variable pointing to a separate DOM element.

### Nested templates

`<ng-template>` creates a new scope. Variables declared inside are not accessible outside:

```html
<ng-template #tpl>
  <div #innerDiv>Inside template</div>
</ng-template>
<!-- innerDiv is NOT accessible here -->
```

## Template Reference Variables vs @let

Angular 18.1 introduced the `@let` syntax for declaring template-local variables. These serve different purposes:

| Feature | Template Reference Variable (`#ref`) | `@let` Variable |
|---|---|---|
| Syntax | `#name` on an element | `@let name = expression;` |
| What it references | DOM element, component, directive, or TemplateRef | Any expression result |
| Tied to an element | Yes (must be on an element) | No (standalone declaration) |
| Read-only | Reference itself is read-only | Value is read-only |
| Can hold computed values | No (always an element reference) | Yes |
| Forward reference | Yes (usable before declaration) | No (must be declared before use) |

### When to use which

```html
<!-- Template ref: access a DOM element -->
<input #searchInput type="text" />
<button (click)="searchInput.focus()">Focus</button>

<!-- @let: store a computed value for reuse -->
@let fullName = firstName() + ' ' + lastName();
<h2>{{ fullName }}</h2>
<p>Welcome back, {{ fullName }}!</p>
```

Use `#ref` when you need access to an element, component, or directive. Use `@let` when you need to store and reuse a computed expression.

## Template Reference Variables with viewChild

Template reference variables are the primary way to connect template elements to the component class via `viewChild`:

```typescript
@Component({
  selector: 'app-editor',
  standalone: true,
  template: `
    <textarea #editor rows="10"></textarea>
    <button (click)="clear()">Clear</button>
  `
})
export class EditorComponent {
  // viewChild queries by the template reference variable name
  editor = viewChild.required<ElementRef<HTMLTextAreaElement>>('editor');

  clear(): void {
    this.editor().nativeElement.value = '';
    this.editor().nativeElement.focus();
  }
}
```

The string `'editor'` in `viewChild('editor')` matches the `#editor` template reference variable. This is the bridge between the template and the component class.

### Multiple variables on different elements

Each template reference variable must have a unique name within its scope:

```typescript
@Component({
  template: `
    <input #firstName />
    <input #lastName />
    <canvas #chart></canvas>
  `
})
export class MyComponent {
  firstNameInput = viewChild.required<ElementRef>('firstName');
  lastNameInput = viewChild.required<ElementRef>('lastName');
  chartCanvas = viewChild.required<ElementRef>('chart');
}
```

## Using Template Reference Variables for Element-to-Element Communication

One of the most practical uses of template reference variables is passing information between elements without going through the component class:

### Passing input value to a method

```html
<input #taskInput placeholder="New task" (keydown.enter)="addTask(taskInput.value); taskInput.value = ''" />
<button (click)="addTask(taskInput.value); taskInput.value = ''">Add</button>
```

The input value is read directly from the DOM element and passed to the component method. The input is also cleared directly via the DOM reference.

### Triggering focus on another element

```html
<input #search type="text" />
<button (click)="search.focus()">Focus Search</button>
```

### Reading form control state

```html
<form #loginForm="ngForm" (ngSubmit)="onLogin(loginForm.value)">
  <input name="email" ngModel required #email="ngModel" />
  <input name="password" ngModel required type="password" />

  @if (email.invalid && email.touched) {
    <p class="error">Email is required.</p>
  }

  <button [disabled]="loginForm.invalid">Login</button>
</form>
```

Here, `#loginForm="ngForm"` accesses the `NgForm` directive and `#email="ngModel"` accesses the `NgModel` directive for the email input. These provide `.valid`, `.invalid`, `.touched`, `.dirty`, `.value`, and `.errors` properties.

### Passing a component reference to a sibling

```html
<app-data-source #source />
<app-data-table [dataProvider]="source" />
```

The `#source` variable is the `DataSourceComponent` instance. It is passed directly as an input to `DataTableComponent`. This avoids routing the data through the parent component class.

## Template Reference Variables on ng-container

`<ng-container>` is a logical grouping element that does not render a DOM node. You can place a template reference variable on it, but since there is no DOM element, the variable is mainly useful with directives:

```html
<ng-container #group="ngModelGroup" ngModelGroup="address">
  <input name="street" ngModel />
  <input name="city" ngModel />
</ng-container>
<p>Address group valid: {{ group.valid }}</p>
```

## Common Mistakes

### 1. Duplicate variable names in the same scope

```html
<!-- BAD: Two elements with the same #ref name -->
<input #field type="text" />
<input #field type="email" />
<p>{{ field.value }}</p> <!-- Which field? Unpredictable! -->
```

Angular does not throw a compile error for duplicate names, but the behavior is undefined. The second declaration may silently shadow the first.

### 2. Accessing a scoped variable outside its structural block

```html
@if (showSearch()) {
  <input #searchInput type="text" />
}

<!-- ERROR: searchInput is not accessible here -->
<button (click)="searchInput.focus()">Focus</button>
```

**Fix**: Move the button inside the `@if` block, or use `viewChild` in the component class to access the element reactively.

### 3. Expecting #ref on a component to return the DOM element

```html
<app-card #card />
<!-- card is the CardComponent instance, NOT the host HTMLElement -->
<p>{{ card.nativeElement }}</p> <!-- undefined! -->
```

To get the host element of a component, use `viewChild` with `read: ElementRef`:

```typescript
cardElement = viewChild('card', { read: ElementRef });
```

### 4. Confusing #ref with @let for storing values

```html
<!-- BAD: Trying to store a computed value in a template ref -->
<!-- Template refs must be on elements -->

<!-- GOOD: Use @let for computed values -->
@let total = price() * quantity();
<p>Total: {{ total }}</p>
```

### 5. Using template reference variables for complex logic

```html
<!-- BAD: Complex chaining through template refs -->
<app-form #form />
<app-validator #validator [form]="form" />
<app-submit [valid]="validator.isValid" [data]="form.data" />
```

When the template starts threading multiple refs between many components, it becomes hard to understand data flow. Move this coordination to the component class or a service.

### 6. Forgetting the exportAs name for directive access

```html
<!-- BAD: Without ="ngModel", #email is the HTMLInputElement, not NgModel -->
<input ngModel name="email" #email />
<p>{{ email.valid }}</p> <!-- ERROR: valid is not a property of HTMLInputElement -->

<!-- GOOD: Specify the directive export name -->
<input ngModel name="email" #email="ngModel" />
<p>{{ email.valid }}</p> <!-- Works: email is the NgModel instance -->
```

## Best Practices

### 1. Use descriptive names that reflect the element or purpose

```html
<!-- GOOD: Clear what each ref points to -->
<input #searchInput type="text" />
<canvas #chartCanvas></canvas>
<app-sidebar #sidebar />

<!-- BAD: Generic or abbreviated names -->
<input #x type="text" />
<canvas #c></canvas>
<app-sidebar #s />
```

### 2. Prefer template refs for simple, template-only interactions

If you only need to read an input value on a button click or focus an element, template reference variables keep the logic in the template and avoid unnecessary component class code:

```html
<!-- Simple: no component class involvement needed -->
<input #tagInput placeholder="Add tag" />
<button (click)="addTag(tagInput.value); tagInput.value = ''">Add</button>
```

### 3. Use viewChild for component-class access

When you need to access a DOM element from TypeScript (for third-party library initialization, measurements, etc.), use `viewChild` with a template reference variable:

```typescript
canvas = viewChild.required<ElementRef<HTMLCanvasElement>>('chartCanvas');
```

### 4. Always specify exportAs when accessing directive instances

Without `="exportName"`, the ref defaults to the host element (for plain HTML) or the component instance (for components). Be explicit when you need a directive.

### 5. Avoid passing template refs deeply through component trees

Template reference variables should be used locally. Passing a `#ref` as an input through multiple levels of components creates tight coupling. Use services or signals for cross-component communication.

### 6. Use @let instead of template refs for computed values

If you need to store a derived value (not an element reference), `@let` is the correct tool:

```html
@let discountedPrice = price() * (1 - discount());
<p>{{ discountedPrice | currency }}</p>
```

## Real-World Use Cases

### Quick-Add Form with Template Refs

```typescript
@Component({
  selector: 'app-quick-add',
  standalone: true,
  template: `
    <div class="quick-add">
      <input
        #taskInput
        type="text"
        placeholder="What needs to be done?"
        (keydown.enter)="add(taskInput)"
      />
      <select #prioritySelect>
        <option value="low">Low</option>
        <option value="medium" selected>Medium</option>
        <option value="high">High</option>
      </select>
      <button (click)="add(taskInput)">Add Task</button>
    </div>
  `
})
export class QuickAddComponent {
  taskAdded = output<{ text: string; priority: string }>();

  add(input: HTMLInputElement): void {
    const text = input.value.trim();
    if (!text) return;

    // Access sibling element through viewChild would also work,
    // but here we receive the DOM element directly via template ref
    this.taskAdded.emit({ text, priority: 'medium' });
    input.value = '';
    input.focus();
  }
}
```

### Video Player Controls

```typescript
@Component({
  selector: 'app-video-player',
  standalone: true,
  template: `
    <div class="player">
      <video
        #video
        [src]="src()"
        (timeupdate)="onTimeUpdate(video)"
        (loadedmetadata)="onLoaded(video)"
      ></video>

      <div class="controls">
        <button (click)="video.paused ? video.play() : video.pause()">
          {{ video.paused ? 'Play' : 'Pause' }}
        </button>

        <input
          type="range"
          min="0"
          [max]="duration()"
          [value]="currentTime()"
          (input)="video.currentTime = +seekBar.value"
          #seekBar
        />

        <span>{{ currentTime() | number:'1.0-0' }}s / {{ duration() | number:'1.0-0' }}s</span>

        <input
          type="range"
          min="0"
          max="1"
          step="0.1"
          [value]="video.volume"
          (input)="video.volume = +volumeSlider.value"
          #volumeSlider
        />
      </div>
    </div>
  `
})
export class VideoPlayerComponent {
  src = input.required<string>();

  currentTime = signal(0);
  duration = signal(0);

  onTimeUpdate(video: HTMLVideoElement): void {
    this.currentTime.set(video.currentTime);
  }

  onLoaded(video: HTMLVideoElement): void {
    this.duration.set(video.duration);
  }
}
```

This example uses template reference variables to interact with the native `<video>` element directly in the template (play, pause, volume, seek) while using signals for values that need to update the template display.

### Form Validation with NgModel Exports

```typescript
@Component({
  selector: 'app-contact-form',
  standalone: true,
  imports: [FormsModule],
  template: `
    <form #contactForm="ngForm" (ngSubmit)="onSubmit(contactForm)">
      <div class="field">
        <label for="name">Name</label>
        <input
          id="name"
          name="name"
          ngModel
          required
          minlength="2"
          #name="ngModel"
        />
        @if (name.invalid && name.touched) {
          <div class="errors">
            @if (name.errors?.['required']) {
              <p>Name is required.</p>
            }
            @if (name.errors?.['minlength']) {
              <p>Name must be at least 2 characters.</p>
            }
          </div>
        }
      </div>

      <div class="field">
        <label for="email">Email</label>
        <input
          id="email"
          name="email"
          ngModel
          required
          email
          #emailCtrl="ngModel"
        />
        @if (emailCtrl.invalid && emailCtrl.touched) {
          <div class="errors">
            @if (emailCtrl.errors?.['required']) {
              <p>Email is required.</p>
            }
            @if (emailCtrl.errors?.['email']) {
              <p>Enter a valid email address.</p>
            }
          </div>
        }
      </div>

      <div class="field">
        <label for="message">Message</label>
        <textarea
          id="message"
          name="message"
          ngModel
          required
          #message="ngModel"
          rows="4"
        ></textarea>
        @if (message.invalid && message.touched) {
          <p class="error">Message is required.</p>
        }
      </div>

      <button [disabled]="contactForm.invalid">Send</button>
      <p>Form status: {{ contactForm.valid ? 'Valid' : 'Invalid' }}</p>
    </form>
  `
})
export class ContactFormComponent {
  submitted = output<{ name: string; email: string; message: string }>();

  onSubmit(form: NgForm): void {
    if (form.valid) {
      this.submitted.emit(form.value as { name: string; email: string; message: string });
      form.resetForm();
    }
  }
}
```

This demonstrates `exportAs` usage extensively: `#contactForm="ngForm"` accesses the form directive to check overall validity, and each `#ctrl="ngModel"` accesses individual control state for per-field validation messages.

---

> **Key Takeaway:** Template reference variables (`#ref`) provide direct, in-template access to DOM elements, component instances, and directive instances (via `exportAs`), enabling simple element-to-element interactions without component class code, while `viewChild` bridges these references into the component class for more complex scenarios.
