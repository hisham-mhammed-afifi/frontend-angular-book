# 12.14 Template Expressions and Safe Navigation

> Angular template expressions are the mini-language inside `{{ }}`, `[ ]`, and `( )` bindings. Understanding what is allowed, what is forbidden, how operators interact, and how to handle null safely determines whether your templates are clean or full of subtle bugs.

---

## Template Expression Basics

A template expression is any JavaScript-like expression that Angular evaluates and converts to a string (for interpolation) or a value (for property/event bindings).

```html
<!-- Interpolation: expression inside {{ }} -->
<h1>{{ title() }}</h1>
<p>Total: {{ price() * quantity() }}</p>
<span>{{ user().firstName + ' ' + user().lastName }}</span>

<!-- Property binding: expression inside [ ] -->
<img [src]="imageUrl()" [alt]="imageAlt()" />
<div [class.active]="isActive()"></div>
<input [disabled]="!canEdit()" />

<!-- Event binding: statement inside ( ) -->
<button (click)="onSave()">Save</button>
<input (keyup.enter)="onSearch(query())" />
```

Angular evaluates these expressions in the context of the component instance. The expression can access:
- Component properties and methods.
- Signal reads (calling the signal as a function).
- Template reference variables (`#ref`).
- Template input variables (`@let`, `@for` implicit variables).

---

## What Is Allowed in Template Expressions

Most JavaScript expressions work in Angular templates:

```html
<!-- Arithmetic -->
<p>{{ basePrice() + tax() }}</p>
<p>{{ total() * 1.1 }}</p>
<p>{{ Math.round is NOT allowed, but custom methods are }}</p>

<!-- String concatenation and template literals (Angular 19+) -->
<p>{{ firstName() + ' ' + lastName() }}</p>
<p>{{ `${firstName()} ${lastName()}` }}</p>

<!-- Comparison operators -->
<div [class.highlight]="score() > 90"></div>
<div [class.warning]="score() >= 50 && score() < 70"></div>

<!-- Ternary operator -->
<span>{{ isOnline() ? 'Online' : 'Offline' }}</span>
<img [src]="darkMode() ? '/assets/logo-dark.svg' : '/assets/logo-light.svg'" />

<!-- Logical operators -->
<p>{{ displayName() || 'Anonymous' }}</p>
<p>{{ username() ?? 'Guest' }}</p>

<!-- Optional chaining -->
<p>{{ user()?.address?.city }}</p>

<!-- Array and object access -->
<p>{{ items()[0] }}</p>
<p>{{ config()['theme'] }}</p>

<!-- Method calls -->
<p>{{ formatCurrency(price()) }}</p>
<p>{{ items().length }}</p>

<!-- Pipe operator -->
<p>{{ birthday() | date:'longDate' }}</p>
<p>{{ price() | currency:'USD' }}</p>
```

## What Is Forbidden in Template Expressions

Angular's template parser rejects or ignores several JavaScript constructs. These restrictions enforce the principle that template expressions should be side-effect-free and idempotent.

### Assignment Operators

```html
<!-- BAD: Assignment operators are not allowed -->
<p>{{ count = 5 }}</p>              <!-- Error -->
<p>{{ count += 1 }}</p>             <!-- Error -->
<p>{{ count -= 1 }}</p>             <!-- Error -->
<p>{{ count *= 2 }}</p>             <!-- Error -->
```

Templates are for reading state, not writing it. Use event bindings to modify state.

### Increment and Decrement

```html
<!-- BAD: ++ and -- are not allowed -->
<p>{{ count++ }}</p>                <!-- Error -->
<p>{{ --index }}</p>                <!-- Error -->
```

### `new` Operator

```html
<!-- BAD: Cannot create objects with new -->
<p>{{ new Date() }}</p>             <!-- Error -->
<p>{{ new Map() }}</p>              <!-- Error -->
```

If you need to create a value, do it in the component class and expose it as a signal or property.

### `typeof` and `instanceof`

```html
<!-- BAD: Type operators are not allowed -->
<p>{{ typeof value }}</p>           <!-- Error -->
<p>{{ value instanceof Array }}</p> <!-- Error -->
```

### Bitwise Operators (Except `|`)

```html
<!-- BAD: Bitwise operators are not allowed -->
<p>{{ flags & 0x01 }}</p>           <!-- Error -->
<p>{{ a ^ b }}</p>                  <!-- Error -->
<p>{{ ~mask }}</p>                  <!-- Error -->
<p>{{ a << 2 }}</p>                 <!-- Error -->
```

The `|` symbol is reserved for pipes. It cannot be used as the bitwise OR operator.

### Chaining with `;` or `,`

```html
<!-- BAD: Cannot chain multiple expressions -->
<p>{{ doA(); doB() }}</p>           <!-- Error -->
<p>{{ a, b }}</p>                   <!-- Error -->
```

### Global Objects

```html
<!-- BAD: Cannot access globals directly -->
<p>{{ Math.round(3.7) }}</p>        <!-- Error: Math is not defined -->
<p>{{ JSON.stringify(data()) }}</p>  <!-- Error: JSON is not defined -->
<p>{{ console.log('debug') }}</p>   <!-- Error: console is not defined -->
<p>{{ window.innerWidth }}</p>      <!-- Error: window is not defined -->
<p>{{ document.title }}</p>         <!-- Error: document is not defined -->
```

The expression context is the component instance, not the global scope. To use `Math.round`, expose it through the component:

```typescript
@Component({
  selector: 'app-stats',
  standalone: true,
  template: `<p>{{ round(averageScore()) }}</p>`
})
export class StatsComponent {
  averageScore = input.required<number>();

  // Expose Math.round to the template
  round = Math.round;
}
```

Or better, use a `computed` signal:

```typescript
export class StatsComponent {
  averageScore = input.required<number>();
  roundedScore = computed(() => Math.round(this.averageScore()));
}
```

```html
<p>{{ roundedScore() }}</p>
```

## Template Literals (Angular 19+)

Angular 19 introduced support for untagged template literals in template expressions. This allows backtick strings with `${}` interpolation inside Angular template bindings.

```html
<!-- Before Angular 19: string concatenation -->
<img [alt]="'Photo of ' + user().name + ' taken on ' + date()" />
<p>{{ 'Hello, ' + name() + '!' }}</p>

<!-- Angular 19+: template literals -->
<img [alt]="`Photo of ${user().name} taken on ${date()}`" />
<p>{{ `Hello, ${name()}!` }}</p>

<!-- Multi-expression composition -->
<div [attr.aria-label]="`${itemCount()} items, ${selectedCount()} selected`" />

<!-- With method calls inside -->
<p>{{ `Total: ${formatCurrency(subtotal() + tax())}` }}</p>
```

Template literals make complex string compositions much more readable than concatenation chains.

**Restriction:** Tagged template literals (like `html\`...\``) are not supported. Only plain backtick strings with `${}` placeholders work.

## The Safe Navigation Operator (`?.`)

The safe navigation operator (also called optional chaining) prevents errors when accessing properties on null or undefined values. If any segment in the chain is nullish, the entire expression short-circuits and returns `undefined` instead of throwing.

### Basic Usage

```html
<!-- Without safe navigation: throws if user() is null -->
<p>{{ user().name }}</p>

<!-- With safe navigation: renders nothing if user() is null -->
<p>{{ user()?.name }}</p>

<!-- Deep chaining -->
<p>{{ order()?.shipping?.address?.city }}</p>

<!-- Method calls -->
<p>{{ user()?.getDisplayName() }}</p>

<!-- Array access -->
<p>{{ items()?.[0]?.name }}</p>
```

### How Angular Compiles It

The Angular compiler translates `user()?.name` into something equivalent to:

```typescript
user() == null ? null : user().name
```

This is similar to JavaScript's native optional chaining (`?.`), and since Angular 16+ with strict templates, the behavior aligns with the TC39 specification: it returns `undefined` when short-circuited.

### Chaining Multiple Levels

```html
<!-- Each ?. independently guards against null/undefined at that level -->
<p>{{ company()?.departments?.[0]?.manager?.email }}</p>
```

If `company()` returns null, the result is `undefined`. If `company()` exists but `departments` is null, the result is still `undefined`. Each `?.` in the chain is an independent null check.

### With Signals

Signals that may hold null or undefined values are a perfect use case for `?.`:

```typescript
export class ProfileComponent {
  // This signal may be null before data loads
  user = signal<User | null>(null);

  // httpResource returns null before the first response
  profile = httpResource<Profile>({
    url: () => `/api/users/${this.userId()}/profile`
  });
}
```

```html
<div class="profile">
  <h2>{{ profile.value()?.displayName }}</h2>
  <p>{{ profile.value()?.bio }}</p>
  <p>{{ profile.value()?.location?.city }}, {{ profile.value()?.location?.country }}</p>
</div>
```

## The Nullish Coalescing Operator (`??`)

Available since Angular 12, the nullish coalescing operator provides a default value when an expression is `null` or `undefined` (but not other falsy values like `0`, `''`, or `false`).

### Basic Usage

```html
<!-- Provides a fallback for null/undefined -->
<p>{{ username() ?? 'Guest' }}</p>
<p>{{ score() ?? 'N/A' }}</p>
<p>{{ user()?.avatar ?? '/assets/default-avatar.png' }}</p>
```

### `??` vs `||`

This distinction is critical:

```typescript
export class SettingsComponent {
  volume = signal(0);       // Legitimate value: 0
  nickname = signal('');    // Legitimate value: empty string
  darkMode = signal(false); // Legitimate value: false
}
```

```html
<!-- || treats 0, '', false as falsy and returns the fallback -->
<p>Volume: {{ volume() || 'Default' }}</p>       <!-- Shows: "Default" (WRONG) -->
<p>Nick: {{ nickname() || 'Anonymous' }}</p>      <!-- Shows: "Anonymous" (WRONG) -->
<p>Dark: {{ darkMode() || 'Light mode' }}</p>     <!-- Shows: "Light mode" (WRONG) -->

<!-- ?? only triggers on null/undefined -->
<p>Volume: {{ volume() ?? 'Default' }}</p>        <!-- Shows: 0 (CORRECT) -->
<p>Nick: {{ nickname() ?? 'Anonymous' }}</p>       <!-- Shows: "" (CORRECT) -->
<p>Dark: {{ darkMode() ?? 'Light mode' }}</p>      <!-- Shows: false (CORRECT) -->
```

| Expression | `null` | `undefined` | `0` | `''` | `false` |
|---|---|---|---|---|---|
| `value \|\| fallback` | fallback | fallback | fallback | fallback | fallback |
| `value ?? fallback` | fallback | fallback | 0 | '' | false |

Use `??` when the value could legitimately be `0`, `''`, or `false`. Use `||` only when all falsy values should trigger the fallback.

## Operator Precedence with Pipes

The pipe operator (`|`) has the lowest precedence in Angular template expressions, lower than any JavaScript operator. This creates counterintuitive behavior when combined with other operators.

### The Pipe Precedence Trap

```html
<!-- What you might write -->
{{ birthday() | date:'longDate' ?? 'Unknown' }}

<!-- What Angular actually evaluates -->
{{ birthday() | date:('longDate' ?? 'Unknown') }}
```

The `??` binds tighter than `|`, so it evaluates `'longDate' ?? 'Unknown'`, which is always `'longDate'` (a non-null string). The `??` operator never gets a chance to provide the fallback for the entire expression.

### The Fix: Parentheses

```html
<!-- CORRECT: Parentheses force the pipe to evaluate first -->
{{ (birthday() | date:'longDate') ?? 'Unknown' }}

<!-- More examples -->
{{ (price() | currency:'USD') ?? 'Free' }}
{{ (timestamp() | date:'short') ?? 'No date' }}
```

### Ternary Operator with Pipes

The same precedence issue applies to the ternary operator:

```html
<!-- BAD: Pipe binds to the wrong part of the ternary -->
{{ condition() ? value1() | uppercase : value2() }}
<!-- Angular reads this as: condition() ? value1() | uppercase:(value2()) -->

<!-- CORRECT: Use parentheses -->
{{ condition() ? (value1() | uppercase) : value2() }}

<!-- CORRECT: Parentheses on both branches for clarity -->
{{ condition() ? (value1() | uppercase) : (value2() | lowercase) }}
```

### General Rule

Whenever you combine a pipe with any other operator (ternary, nullish coalescing, logical OR/AND), wrap the pipe expression in parentheses.

```html
<!-- Always safe with parentheses -->
{{ (value() | someTransform) ?? defaultValue }}
{{ condition() ? (a() | pipeA) : (b() | pipeB) }}
{{ (items() | filter:criteria()).length > 0 }}
```

## The Non-Null Assertion Operator (`!`)

The non-null assertion operator tells TypeScript's type checker (during template type checking) that a value is guaranteed to be non-null, even though its type allows null.

```html
<!-- user() has type User | null, but we know it's loaded by this point -->
<p>{{ user()!.name }}</p>
```

### When to Use It

The non-null assertion should be rare. It is a type-checker override, not a runtime guard. If the value actually is null, you get a runtime error.

```typescript
export class UserDetailComponent {
  user = input.required<User>();

  // required inputs are never null/undefined after initialization
  // but sometimes the type checker needs help
}
```

```html
<!-- The input is required, so it's always present -->
<!-- But some complex expressions may confuse the type checker -->
<p>{{ user()!.address!.city }}</p>
```

### `!` vs `?.`

| Operator | Behavior | Runtime Safety |
|---|---|---|
| `?.` | Returns undefined if null | Safe. No runtime error. |
| `!` | Tells type checker "trust me, this is not null" | Unsafe. Throws if actually null. |

```html
<!-- SAFE: If user is null, renders nothing -->
<p>{{ user()?.name }}</p>

<!-- UNSAFE: If user is null, throws TypeError at runtime -->
<p>{{ user()!.name }}</p>
```

Almost always prefer `?.` over `!`. Use `!` only when:
1. You have a runtime guarantee (e.g., inside an `@if` block that already checked for null).
2. The type checker cannot infer the narrowed type from the control flow.

```html
@if (user(); as u) {
  <!-- u is guaranteed non-null here, but if the type is still User | null -->
  <!-- you might need ! in rare edge cases -->
  <p>{{ u.name }}</p>
}
```

In practice, Angular's template type narrowing inside `@if` handles most cases, making `!` unnecessary.

## The `$any()` Type Cast Function

`$any()` casts an expression to `any`, disabling all type checking for that expression. It is Angular's escape hatch when the type checker is wrong or too strict.

```html
<!-- When the type checker complains about a property -->
<p>{{ $any(someObject).dynamicProperty }}</p>

<!-- When accessing a property that exists at runtime but not in the type -->
<p>{{ $any(this).legacyProperty }}</p>
```

### When to Use It

```html
<!-- Accessing a property added by a third-party library at runtime -->
<div>{{ $any(element).somePluginData }}</div>

<!-- Working with a loosely-typed API response before you add proper types -->
<p>{{ $any(response()).nested.value }}</p>
```

### Why to Avoid It

`$any()` is a code smell. It silences the type checker entirely, which means:
- Typos in property names go undetected.
- Refactoring tools cannot track the expression.
- If the data shape changes, you get runtime errors instead of compile-time errors.

```html
<!-- BAD PRACTICE: Using $any to silence legitimate errors -->
<p>{{ $any(user()).nmae }}</p>  <!-- Typo: "nmae" instead of "name" -->
<!-- No compile error! This fails silently or shows "undefined" at runtime -->
```

The correct fix is almost always to update your types, use `?.`, or restructure the template logic.

## Expression Context and `this`

Inside a template expression, the context is the component instance. You do not need `this.` to access component members.

```typescript
@Component({
  selector: 'app-demo',
  standalone: true,
  template: `
    <!-- These are equivalent -->
    <p>{{ title }}</p>
    <p>{{ this.title }}</p>

    <!-- Signal reads -->
    <p>{{ count() }}</p>
    <p>{{ this.count() }}</p>

    <!-- Method calls -->
    <p>{{ formatDate(today) }}</p>
    <p>{{ this.formatDate(this.today) }}</p>
  `
})
export class DemoComponent {
  title = 'Hello';
  count = signal(0);
  today = new Date();

  formatDate(date: Date): string {
    return date.toLocaleDateString();
  }
}
```

Using `this` is valid but unnecessary and unconventional. Omit it.

Template input variables shadow component properties:

```html
@for (item of items(); track item.id) {
  <!-- "item" here refers to the loop variable, not a component property -->
  <p>{{ item.name }}</p>
}
```

## Expression Complexity and Performance

Template expressions are evaluated on every change detection cycle. Complex expressions that do heavy computation cause performance problems.

### Bad: Complex Expressions in Templates

```html
<!-- BAD PRACTICE: Filtering and sorting in the template expression -->
<ul>
  @for (item of items().filter(i => i.active).sort((a, b) => a.name.localeCompare(b.name)); track item.id) {
    <li>{{ item.name }}</li>
  }
</ul>

<!-- BAD PRACTICE: Complex formatting logic -->
<p>{{ value() > 1000 ? (value() / 1000).toFixed(1) + 'k' : value().toString() }}</p>
```

These expressions run on every change detection cycle, recreating arrays and performing string operations repeatedly.

### Good: Move Logic to Computed Signals

```typescript
export class ItemListComponent {
  items = input.required<Item[]>();

  // Computed only re-evaluates when items() changes
  activeItems = computed(() =>
    this.items()
      .filter(i => i.active)
      .sort((a, b) => a.name.localeCompare(b.name))
  );
}
```

```html
<ul>
  @for (item of activeItems(); track item.id) {
    <li>{{ item.name }}</li>
  }
</ul>
```

```typescript
export class StatsComponent {
  value = input.required<number>();

  formattedValue = computed(() => {
    const v = this.value();
    return v > 1000 ? `${(v / 1000).toFixed(1)}k` : v.toString();
  });
}
```

```html
<p>{{ formattedValue() }}</p>
```

Computed signals are memoized. They only recalculate when their dependencies change, not on every change detection cycle.

### When Simple Expressions Are Fine

Not every expression needs to be extracted. Simple property reads and trivial operations are perfectly acceptable:

```html
<!-- These are fine in templates -->
<p>{{ firstName() }} {{ lastName() }}</p>
<p>{{ items().length }}</p>
<p>{{ price() * quantity() }}</p>
<span [class.active]="isSelected()"></span>
<div [style.opacity]="disabled() ? 0.5 : 1"></div>
```

The rule of thumb: if the expression does not create new objects, iterate collections, or perform multiple function calls, it is fine in the template.

## Extended Diagnostics for Template Expressions

Angular's compiler includes extended diagnostics that catch common expression mistakes at compile time.

### NG8102: Nullish Coalescing Not Nullable

```html
<!-- WARNING: username is type string (never null/undefined) -->
<p>{{ username() ?? 'Guest' }}</p>
<!-- The ?? is useless because username() can never be null -->
```

Fix: Either remove the `??` or update the type to allow null.

### NG8107: Optional Chain Not Nullable

```html
<!-- WARNING: user() is type User (never null/undefined) -->
<p>{{ user()?.name }}</p>
<!-- The ?. is useless because user() is always defined -->
```

Fix: If the signal truly cannot be null, use `.` instead of `?.`. If it can be null, update the type.

### NG8103: Missing Control Flow Directive

Catches cases where you write `*ngIf` or `*ngFor` but forgot to import the directive. With standalone components and new control flow, this is less common.

## Common Mistakes

- **Mistake 1: Using `||` when `??` is needed.**
  `||` treats `0`, `''`, and `false` as falsy. If your signal legitimately holds zero or an empty string, `||` replaces it with the fallback. Use `??` to only catch `null` and `undefined`.

- **Mistake 2: Forgetting parentheses around pipe expressions.**
  Because pipes have the lowest precedence, `value | pipe ?? fallback` evaluates as `value | pipe:(fallback)`, not `(value | pipe) ?? fallback`. Always wrap pipe expressions in parentheses when combining with other operators.

- **Mistake 3: Putting complex logic in template expressions.**
  Filtering, sorting, mapping, or formatting inside `{{ }}` runs on every change detection cycle. Extract it to a `computed` signal for memoization and testability.

- **Mistake 4: Using `!` (non-null assertion) instead of `?.` (safe navigation).**
  The `!` operator only suppresses the type checker; it provides zero runtime protection. If the value is actually null, you get a `TypeError`. Use `?.` for runtime safety. Reserve `!` for cases where you have a structural guarantee of non-null (like inside `@if` narrowing).

- **Mistake 5: Using `$any()` to silence type errors.**
  `$any()` disables type checking for the expression entirely, hiding typos, missing properties, and wrong types. Fix the underlying type issue instead. If you must use `$any()`, add a comment explaining why.

- **Mistake 6: Accessing global objects directly in templates.**
  `Math`, `JSON`, `console`, `window`, and `document` are not available in template expressions. Expose specific functions through the component class or use computed signals to pre-compute values.

---

## Best Practices

- **Use `?.` liberally for nullable data.** Signals fetching data from APIs often start as null. Chain `?.` through the property path to gracefully handle the loading state without verbose `@if` checks for every sub-property.

- **Use `??` for meaningful defaults.** When displaying data that might be null, provide a human-readable fallback with `??` rather than showing "undefined" or blank space.

- **Parenthesize pipe expressions in compound operators.** Make it a team rule: any pipe combined with `??`, `||`, `&&`, or `? :` must be wrapped in parentheses. This eliminates an entire class of subtle bugs.

- **Keep template expressions simple.** A good template expression is a signal read, a property access, a simple comparison, or a pipe transformation. Anything more complex belongs in a `computed` signal.

- **Prefer `computed` signals over template methods for derived values.** Methods called in templates run on every change detection cycle. Computed signals are memoized and only recalculate when dependencies change.

- **Let Angular's type narrowing work for you.** Inside `@if (user(); as u)`, the variable `u` is automatically non-null. Use this pattern instead of sprinkling `!` assertions throughout the template.

## Real-World Use Cases

### User Profile Card with Safe Navigation

A profile card that handles loading state, missing optional fields, and display formatting all through template expressions.

```typescript
@Component({
  selector: 'app-profile-card',
  standalone: true,
  imports: [DatePipe],
  template: `
    @if (profile.value(); as p) {
      <div class="card">
        <img
          [src]="p.avatarUrl ?? '/assets/default-avatar.png'"
          [alt]="`${p.firstName} ${p.lastName}`"
        />
        <h2>{{ p.firstName }} {{ p.lastName }}</h2>
        <p class="title">{{ p.jobTitle ?? 'No title' }}</p>
        <p class="location">
          {{ p.location?.city ?? 'Unknown city' }},
          {{ p.location?.country ?? 'Unknown country' }}
        </p>
        <p class="joined">
          Joined {{ (p.joinDate | date:'mediumDate') ?? 'Unknown' }}
        </p>
        <p class="bio">{{ p.bio ?? 'This user has not written a bio yet.' }}</p>

        @if (p.socialLinks; as links) {
          <div class="socials">
            @if (links.twitter) {
              <a [href]="links.twitter">Twitter</a>
            }
            @if (links.github) {
              <a [href]="links.github">GitHub</a>
            }
            @if (links.linkedin) {
              <a [href]="links.linkedin">LinkedIn</a>
            }
          </div>
        }
      </div>
    } @else {
      <div class="card skeleton">Loading profile...</div>
    }
  `
})
export class ProfileCardComponent {
  userId = input.required<string>();

  profile = httpResource<UserProfile>({
    url: () => `/api/users/${this.userId()}`
  });
}
```

### Data Table with Computed Derived Values

A table that displays formatted, filtered, and sorted data using computed signals instead of complex template expressions.

```typescript
@Component({
  selector: 'app-employee-table',
  standalone: true,
  imports: [CurrencyPipe, DatePipe],
  template: `
    <div class="table-header">
      <input
        [value]="searchTerm()"
        (input)="searchTerm.set($event.target.value)"
        placeholder="Search employees..."
      />
      <span class="count">
        {{ filteredEmployees().length }} of {{ employees().length }} employees
      </span>
    </div>

    <table>
      <thead>
        <tr>
          <th>Name</th>
          <th>Department</th>
          <th>Salary</th>
          <th>Start Date</th>
          <th>Status</th>
        </tr>
      </thead>
      <tbody>
        @for (emp of filteredEmployees(); track emp.id) {
          <tr [class.inactive]="emp.status !== 'active'">
            <td>{{ emp.firstName }} {{ emp.lastName }}</td>
            <td>{{ emp.department ?? 'Unassigned' }}</td>
            <td>{{ emp.salary | currency:'USD':'symbol':'1.0-0' }}</td>
            <td>{{ emp.startDate | date:'mediumDate' }}</td>
            <td>
              <span [class]="'badge badge-' + emp.status">
                {{ emp.status }}
              </span>
            </td>
          </tr>
        } @empty {
          <tr>
            <td colspan="5">No employees match "{{ searchTerm() }}"</td>
          </tr>
        }
      </tbody>
    </table>

    <div class="summary">
      Average salary: {{ averageSalary() | currency:'USD':'symbol':'1.0-0' }}
    </div>
  `
})
export class EmployeeTableComponent {
  employees = input.required<Employee[]>();
  searchTerm = signal('');

  // Filtering and sorting done in computed, not in the template
  filteredEmployees = computed(() => {
    const term = this.searchTerm().toLowerCase();
    return this.employees()
      .filter(e =>
        e.firstName.toLowerCase().includes(term) ||
        e.lastName.toLowerCase().includes(term) ||
        (e.department?.toLowerCase().includes(term) ?? false)
      )
      .sort((a, b) => a.lastName.localeCompare(b.lastName));
  });

  averageSalary = computed(() => {
    const emps = this.filteredEmployees();
    if (emps.length === 0) return 0;
    return emps.reduce((sum, e) => sum + e.salary, 0) / emps.length;
  });
}
```

### Configuration Display with Deep Optional Chaining

A settings panel that displays deeply nested configuration objects with multiple levels of null safety.

```typescript
@Component({
  selector: 'app-config-viewer',
  standalone: true,
  imports: [JsonPipe],
  template: `
    <div class="config-panel">
      <h2>Server Configuration</h2>

      <dl class="config-list">
        <dt>Host</dt>
        <dd>{{ config()?.server?.host ?? 'localhost' }}</dd>

        <dt>Port</dt>
        <dd>{{ config()?.server?.port ?? 3000 }}</dd>

        <dt>SSL Enabled</dt>
        <dd>{{ config()?.server?.ssl?.enabled ?? false }}</dd>

        <dt>Certificate</dt>
        <dd>{{ config()?.server?.ssl?.certPath ?? 'Not configured' }}</dd>

        <dt>Database</dt>
        <dd>{{ config()?.database?.type ?? 'sqlite' }}://{{ config()?.database?.host ?? 'localhost' }}</dd>

        <dt>Max Connections</dt>
        <dd>{{ config()?.database?.pool?.max ?? 10 }}</dd>

        <dt>Cache TTL</dt>
        <dd>{{ config()?.cache?.ttl ?? 3600 }}s</dd>

        <dt>Log Level</dt>
        <dd>{{ config()?.logging?.level ?? 'info' }}</dd>

        <dt>Features</dt>
        <dd>
          @for (feature of config()?.features ?? []; track feature) {
            <span class="tag">{{ feature }}</span>
          } @empty {
            <span class="none">No features enabled</span>
          }
        </dd>
      </dl>

      @if (config()?.raw) {
        <details>
          <summary>Raw Config</summary>
          <pre>{{ config()?.raw | json }}</pre>
        </details>
      }
    </div>
  `
})
export class ConfigViewerComponent {
  config = input<AppConfig | null>(null);
}
```

This demonstrates how `?.` and `??` work together to create a template that gracefully handles missing configuration at every level, showing sensible defaults without any `@if` checks for individual properties.

---

> **Key Takeaway:** Keep template expressions simple and side-effect-free. Use `?.` for null-safe property access, `??` for meaningful defaults, always parenthesize pipes in compound expressions, and move any non-trivial logic into `computed` signals where it is memoized and testable.
