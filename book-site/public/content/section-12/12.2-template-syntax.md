# 12.2 Template Syntax (Interpolation, Property/Event/Two-Way Binding)

> Angular templates use a declarative binding syntax to connect component state to the DOM. This guide covers every binding type in depth: how it works internally, edge cases, type safety, security, and the signal-based two-way binding model.

---

## The Four Binding Directions

Every binding in Angular moves data in one of four directions:

| Binding | Syntax | Direction | Example |
|---------|--------|-----------|---------|
| Interpolation | `{{ expression }}` | Component to DOM (text) | `{{ user().name }}` |
| Property binding | `[property]="expression"` | Component to DOM (property) | `[disabled]="isLoading()"` |
| Event binding | `(event)="statement"` | DOM to component | `(click)="save()"` |
| Two-way binding | `[(prop)]="signal"` | Both directions | `[(value)]="searchTerm"` |

All four share the same foundation: Angular evaluates the expression or statement in the component's context and synchronizes data with the DOM.

## Interpolation Deep Dive

Interpolation evaluates a template expression and inserts the result as text.

```html
<h1>{{ title() }}</h1>
<p>{{ firstName() }} {{ lastName() }}</p>
<span>{{ items().length }} items in cart</span>
```

### How Interpolation Works Internally

1. Angular evaluates the expression in the component's context
2. Converts the result to a string using `String()` coercion
3. Inserts the string as a text node in the DOM
4. On subsequent change detection, Angular compares the new string to the previous one. If different, it updates the text node.

### String Coercion Rules

Because interpolation always converts to a string, some values produce unexpected results:

```html
<!-- Numbers: fine -->
{{ 42 }}            <!-- "42" -->

<!-- Booleans: fine but be aware -->
{{ true }}          <!-- "true" -->
{{ false }}         <!-- "false" -->

<!-- null and undefined: rendered as empty string -->
{{ null }}          <!-- "" (empty) -->
{{ undefined }}     <!-- "" (empty) -->

<!-- Objects: not useful -->
{{ { name: 'A' } }} <!-- "[object Object]" -->

<!-- Arrays: joined with commas -->
{{ [1, 2, 3] }}     <!-- "1,2,3" -->
```

### Security: Automatic Sanitization

Angular automatically sanitizes interpolated values. If a value contains HTML tags or script elements, they are escaped, not rendered.

```typescript
@Component({
  template: `<p>{{ userInput() }}</p>`,
})
export class SafeComponent {
  // Even if the user inputs: <script>alert('XSS')</script>
  // Angular renders it as literal text, not as an HTML script tag
  userInput = signal('<script>alert("XSS")</script>');
}
// Renders: <p>&lt;script&gt;alert("XSS")&lt;/script&gt;</p>
```

This is one of Angular's core XSS protections. Interpolation never inserts raw HTML into the DOM. To render trusted HTML, use `[innerHTML]` with extreme caution (and Angular still sanitizes that binding by default).

### Expression Restrictions Revisited

Template expressions are a subset of JavaScript. They are evaluated in the component's scope and have these restrictions:

```html
<!-- ALLOWED -->
{{ a + b }}                   <!-- Arithmetic -->
{{ cond ? 'yes' : 'no' }}    <!-- Ternary -->
{{ obj?.prop }}               <!-- Optional chaining -->
{{ val ?? 'default' }}        <!-- Nullish coalescing -->
{{ val | pipeName }}          <!-- Pipe operator -->
{{ fn() }}                    <!-- Method call -->
{{ arr[0] }}                  <!-- Index access -->

<!-- NOT ALLOWED -->
{{ x = 5 }}                   <!-- Assignment -->
{{ i++ }}                     <!-- Increment/decrement -->
{{ new Date() }}              <!-- new keyword -->
{{ typeof x }}                <!-- typeof operator -->
{{ a | b }}                   <!-- Bitwise OR (conflicts with pipe) -->
{{ `hello ${name}` }}         <!-- Template literals (Angular 19 and below) -->
```

**Note on template literals:** As of Angular 20, JavaScript template literals are supported in template expressions. In Angular 19 and below, use concatenation instead.

```html
<!-- Angular 19 and below -->
{{ 'Hello, ' + name() + '!' }}

<!-- Angular 20+ -->
{{ `Hello, ${name()}!` }}
```

## Property Binding Deep Dive

Property binding sets a DOM property, component input, or directive property to the result of a template expression.

```html
<img [src]="imageUrl()" [alt]="imageAlt()">
<button [disabled]="!isValid()">Submit</button>
<app-card [title]="cardTitle()" [data]="cardData()">...</app-card>
```

### Properties vs Attributes

This is the most important distinction in Angular binding and the source of many bugs.

**HTML attributes** initialize DOM properties and are defined in the HTML source. **DOM properties** are the live values on the DOM element object.

```html
<!-- The HTML attribute "value" sets the initial DOM property -->
<input value="initial">
```

```javascript
// After the page loads:
const input = document.querySelector('input');
input.value;          // "initial" (DOM property, live value)
input.getAttribute('value'); // "initial" (HTML attribute, static)

input.value = 'changed';
input.value;          // "changed" (DOM property updated)
input.getAttribute('value'); // "initial" (HTML attribute unchanged!)
```

Angular's `[property]` syntax binds to the **DOM property**, not the HTML attribute. This is the correct behavior for most cases because DOM properties reflect the current state.

### When Properties and Attributes Differ

| Attribute | Property | Notes |
|-----------|----------|-------|
| `class` | `className` | Property is `className`, not `class` |
| `for` | `htmlFor` | On `<label>` elements |
| `colspan` | `colSpan` | camelCase in DOM |
| `rowspan` | `rowSpan` | camelCase in DOM |
| `tabindex` | `tabIndex` | camelCase in DOM |
| `readonly` | `readOnly` | camelCase in DOM |
| `maxlength` | `maxLength` | camelCase in DOM |
| `href` | `href` | Same, but property returns resolved URL |

Angular handles most of these automatically. When you write `[colspan]="span()"`, Angular binds to the correct DOM property.

### Attribute Binding: When No Property Exists

Some HTML attributes have no corresponding DOM property. For these, use `[attr.name]`:

```html
<!-- ARIA attributes (no DOM property) -->
<button [attr.aria-label]="label()" [attr.aria-expanded]="isOpen()">Menu</button>

<!-- SVG attributes -->
<svg><rect [attr.x]="x()" [attr.y]="y()" [attr.width]="w()" [attr.height]="h()" /></svg>

<!-- data-* attributes -->
<div [attr.data-testid]="testId()">...</div>

<!-- role attribute -->
<nav [attr.role]="'navigation'">...</nav>
```

### Null Removes the Attribute

When an attribute binding evaluates to `null` or `undefined`, Angular removes the attribute entirely from the DOM element. This is useful for conditional attributes:

```html
<!-- disabled attribute is removed when isDisabled() is false/null -->
<button [attr.disabled]="isDisabled() ? '' : null">Click</button>

<!-- aria-hidden removed when not needed -->
<div [attr.aria-hidden]="shouldHide() ? 'true' : null">...</div>
```

For DOM properties, use `null` carefully:

```html
<!-- DOM property: disabled is a boolean property -->
<button [disabled]="isDisabled()">Click</button>
<!-- isDisabled() = true  -> button is disabled -->
<!-- isDisabled() = false -> button is enabled -->
<!-- No need for null tricks with boolean properties -->
```

### Class Binding

```html
<!-- Single class: add/remove based on truthiness -->
<div [class.active]="isActive()">Tab</div>
<div [class.highlighted]="isNew()" [class.dimmed]="isArchived()">Item</div>

<!-- Multi-class with object -->
<div [class]="{ active: isActive(), disabled: isDisabled() }">...</div>

<!-- Static + dynamic classes together -->
<div class="card" [class.card-elevated]="isElevated()">...</div>
```

**Precedence:** When both `class` (static) and `[class]` (dynamic) exist, the dynamic binding wins for overlapping classes. Static classes that do not conflict are preserved.

### Style Binding

```html
<!-- Single style property -->
<div [style.background-color]="bgColor()">...</div>

<!-- With unit suffix -->
<div [style.width.px]="width()">...</div>
<div [style.font-size.em]="fontSize()">...</div>
<div [style.height.%]="heightPercent()">...</div>

<!-- Multi-style with object -->
<div [style]="{ 'max-width.px': maxWidth(), 'opacity': opacity() }">...</div>
```

**Sanitization:** Angular sanitizes style values. If you try to bind a URL to a style that expects one (e.g., `background-image`), Angular may block it if the URL is not trusted.

```typescript
// Angular may sanitize this:
bgImage = signal('url(https://evil.com/tracker.png)');
// Template: [style.background-image]="bgImage()"
// Result: style may be empty due to sanitization

// If you trust the URL, use DomSanitizer (carefully):
bgImage = this.sanitizer.bypassSecurityTrustStyle('url(https://cdn.example.com/bg.png)');
```

## Event Binding Deep Dive

Event binding listens for DOM events or component outputs and runs a template statement when they fire.

```html
<button (click)="handleClick()">Save</button>
<input (input)="onInput($event)" (keydown.enter)="submit()">
<app-item (deleted)="onItemDeleted($event)">...</app-item>
```

### The $event Object

`$event` is a special variable available only inside event binding expressions. Its type depends on the source:

| Event Source | $event Type | Example |
|-------------|-------------|---------|
| Native DOM event | The DOM event type (e.g., `MouseEvent`, `KeyboardEvent`, `Event`) | `(click)="onClick($event)"` |
| Component `output()` | The emitted value type | `(deleted)="onDeleted($event)"` where `$event` is `string` if output is `output<string>()` |
| Directive output | The emitted value type | Same as component |

### Type-Safe Event Handling

```typescript
@Component({
  template: `
    <input
      (input)="onInput($event)"
      (keydown)="onKeyDown($event)"
      (focus)="onFocus($event)"
    />
    <app-color-picker (colorSelected)="onColor($event)" />
  `,
  imports: [ColorPickerComponent],
})
export class FormComponent {
  // Type the parameter for the specific DOM event
  onInput(event: Event) {
    const target = event.target as HTMLInputElement;
    this.searchTerm.set(target.value);
  }

  onKeyDown(event: KeyboardEvent) {
    if (event.key === 'Escape') {
      this.searchTerm.set('');
    }
  }

  onFocus(event: FocusEvent) {
    // FocusEvent-specific properties available
    console.log('Related target:', event.relatedTarget);
  }

  // Component output: $event is the emitted type
  onColor(color: string) {
    this.selectedColor.set(color);
  }
}
```

### Key Event Filtering

Angular provides built-in key event filtering so you do not need to check `event.key` manually:

```html
<!-- Single key -->
<input (keydown.enter)="submit()">
<input (keyup.escape)="cancel()">
<input (keydown.tab)="nextField()">

<!-- Arrow keys -->
<div (keydown.arrowUp)="selectPrevious()">
<div (keydown.arrowDown)="selectNext()">

<!-- Modifier keys -->
<input (keydown.control.s)="save()">
<input (keydown.shift.enter)="submitAndContinue()">
<input (keydown.alt.backspace)="clearAll()">
<input (keydown.meta.z)="undo()">

<!-- Modifier + specific key -->
<textarea (keydown.control.shift.z)="redo()">
```

Modifier names: `control`, `shift`, `alt`, `meta` (Cmd on Mac, Win on Windows).

### Event Statement Restrictions

Event binding uses **template statements**, which are slightly different from template expressions:

```html
<!-- ALLOWED in statements -->
(click)="doSomething()"                  <!-- Method call -->
(click)="count.set(count() + 1)"        <!-- Signal set -->
(click)="x = 5"                         <!-- Assignment (in statements only!) -->
(click)="a(); b()"                      <!-- Chained statements with ; -->
(click)="handleClick($event)"           <!-- $event access -->
(click)="items.update(i => [...i, x])"  <!-- Complex expression -->

<!-- NOT ALLOWED -->
(click)="doSomething() | uppercase"     <!-- Pipes in statements -->
(click)="new MyClass()"                 <!-- new keyword -->
(click)="i++"                           <!-- Increment/decrement -->
```

### Preventing Default and Propagation

```typescript
@Component({
  template: `
    <form (submit)="onSubmit($event)">
      <a href="https://external.com" (click)="onLinkClick($event)">Link</a>
      <button type="submit">Submit</button>
    </form>
  `,
})
export class FormComponent {
  onSubmit(event: Event) {
    event.preventDefault(); // Prevent form from navigating
    // Handle submission
  }

  onLinkClick(event: MouseEvent) {
    event.preventDefault();    // Prevent navigation
    event.stopPropagation();   // Prevent bubbling to parent handlers
    // Handle click
  }
}
```

## Two-Way Binding Deep Dive

Two-way binding synchronizes a value between a parent and child component in both directions.

### The Banana-in-a-Box Syntax

`[(prop)]` is syntactic sugar that desugars to a property binding plus an event binding:

```html
<!-- Shorthand -->
<app-slider [(value)]="volume" />

<!-- Desugared equivalent -->
<app-slider [value]="volume()" (valueChange)="volume.set($event)" />
```

The naming convention is strict: the event name must be the property name plus `Change`.

### model(): Signal-Based Two-Way Binding

The `model()` function creates a `ModelSignal` that is a writable signal shared between parent and child. When the child writes to it, the parent's bound signal is updated automatically.

```typescript
// child: slider.component.ts
@Component({
  selector: 'app-slider',
  template: `
    <input
      type="range"
      [min]="min()"
      [max]="max()"
      [value]="value()"
      (input)="onInput($event)"
    />
    <span>{{ value() }}</span>
  `,
})
export class SliderComponent {
  min = input(0);
  max = input(100);
  value = model(50); // ModelSignal: writable, two-way bindable

  onInput(event: Event) {
    const val = +(event.target as HTMLInputElement).value;
    this.value.set(val); // Updates both child and parent
  }
}

// parent
@Component({
  imports: [SliderComponent],
  template: `
    <app-slider [(value)]="volume" [min]="0" [max]="100" />
    <p>Volume: {{ volume() }}</p>
  `,
})
export class PlayerComponent {
  volume = signal(75);
}
```

### model.required()

Like `input.required()`, you can require a two-way binding:

```typescript
@Component({
  selector: 'app-toggle',
  template: `
    <button (click)="checked.set(!checked())">
      {{ checked() ? 'ON' : 'OFF' }}
    </button>
  `,
})
export class ToggleComponent {
  checked = model.required<boolean>(); // Parent MUST bind this
}

// Usage: <app-toggle [(checked)]="isEnabled" />
// Omitting [(checked)] causes a compile-time error
```

### How model() Differs from input() + output()

| Feature | `model()` | `input()` + `output()` |
|---------|-----------|------------------------|
| Signal type | `ModelSignal` (writable) | `InputSignal` (read-only) + `OutputEmitterRef` |
| Binding syntax | `[(prop)]="signal"` | `[prop]="signal()" (propChange)="signal.set($event)"` |
| Auto output name | Yes (`propChange` generated) | Manual: must name output `propChange` yourself |
| Use case | Bidirectional state sync | Explicit data down, events up |
| Readability | Concise | More explicit about data flow |

### Two-Way Binding with ngModel (Forms)

For native form elements, `ngModel` from `FormsModule` provides two-way binding:

```typescript
import { FormsModule } from '@angular/forms';

@Component({
  imports: [FormsModule],
  template: `
    <input [(ngModel)]="username" placeholder="Username" />
    <textarea [(ngModel)]="bio" placeholder="Bio"></textarea>
    <select [(ngModel)]="role">
      <option value="admin">Admin</option>
      <option value="user">User</option>
    </select>

    <p>{{ username() }}, {{ role() }}</p>
  `,
})
export class ProfileFormComponent {
  username = signal('');
  bio = signal('');
  role = signal('user');
}
```

`ngModel` works with signals directly in Angular 19+. The signal's value is read for the input, and `ngModel` calls `.set()` when the user types.

### Legacy Two-Way Binding Pattern

Before `model()`, two-way binding required pairing `@Input()` with `@Output()` following the naming convention:

```typescript
// LEGACY: Manual @Input + @Output pair
@Component({
  selector: 'app-toggle',
  template: `
    <button (click)="toggle()">{{ checked ? 'ON' : 'OFF' }}</button>
  `,
})
export class LegacyToggleComponent {
  @Input() checked = false;
  @Output() checkedChange = new EventEmitter<boolean>();

  toggle() {
    this.checked = !this.checked;
    this.checkedChange.emit(this.checked);
  }
}

// Usage was the same: <app-toggle [(checked)]="isEnabled" />
```

This pattern still works but is more verbose. Prefer `model()` for new code.

## Binding Syntax Quick Reference

| What You Want | Syntax | Example |
|---------------|--------|---------|
| Display text | `{{ expr }}` | `{{ name() }}` |
| Set a DOM property | `[prop]="expr"` | `[src]="url()"` |
| Set an HTML attribute | `[attr.name]="expr"` | `[attr.aria-label]="label()"` |
| Toggle a CSS class | `[class.name]="expr"` | `[class.active]="isActive()"` |
| Set multiple classes | `[class]="expr"` | `[class]="classMap()"` |
| Set an inline style | `[style.prop]="expr"` | `[style.color]="color()"` |
| Set a style with unit | `[style.prop.unit]="expr"` | `[style.width.px]="w()"` |
| Listen to a DOM event | `(event)="stmt"` | `(click)="save()"` |
| Listen with key filter | `(keydown.key)="stmt"` | `(keydown.enter)="submit()"` |
| Two-way bind | `[(prop)]="signal"` | `[(value)]="volume"` |
| Two-way with ngModel | `[(ngModel)]="signal"` | `[(ngModel)]="username"` |

## Template Type Checking

Angular's compiler type-checks template expressions against the component class. Errors in expressions are caught at build time.

### Strict Mode

Angular's default template type checking mode (`strictTemplates: true` in `tsconfig.json`) catches:

```typescript
@Component({
  template: `
    <!-- Error: 'nmae' does not exist on type 'User' -->
    <p>{{ user().nmae }}</p>

    <!-- Error: Type 'number' is not assignable to type 'string' -->
    <app-label [text]="42" />

    <!-- Error: 'nonExistentMethod' does not exist -->
    <button (click)="nonExistentMethod()">Click</button>
  `,
})
export class MyComponent {
  user = signal<User>({ name: 'Alice', email: 'alice@example.com' });
}
```

### $any() Type-Cast Escape Hatch

When the type checker is too restrictive (rare), use `$any()` to bypass it:

```html
<!-- Suppress type errors for dynamic property access -->
<p>{{ $any(user()).dynamicProp }}</p>

<!-- Use sparingly. This disables type safety. -->
```

### strictNullChecks in Templates

With strict null checks, signals that can be `undefined` must be handled:

```html
<!-- user() might return null/undefined -->
@if (user(); as u) {
  <p>{{ u.name }}</p>  <!-- u is guaranteed non-null here -->
}

<!-- Or use optional chaining -->
<p>{{ user()?.name }}</p>
<p>{{ user()?.address?.city ?? 'Unknown' }}</p>
```

## Security in Bindings

### Sanitization Contexts

Angular applies context-sensitive sanitization to all bindings:

| Context | What Gets Sanitized | Binding Example |
|---------|-------------------|-----------------|
| HTML | Tags, attributes, scripts | `[innerHTML]="value"` |
| Style | `url()` values, `expression()` | `[style.background]="value"` |
| URL | `javascript:` protocol, data URIs | `[href]="value"` |
| Resource URL | Full URL to loadable resources | `[src]="value"` on `<script>`, `<iframe>` |

```typescript
@Component({
  template: `
    <!-- Safe: Angular sanitizes the innerHTML -->
    <div [innerHTML]="htmlContent()"></div>

    <!-- Safe: Angular blocks javascript: URLs -->
    <a [href]="userProvidedUrl()">Link</a>
  `,
})
export class ContentComponent {
  // If this contains <script> tags, Angular strips them
  htmlContent = signal('<p>Hello</p><script>alert("XSS")</script>');

  // If this is "javascript:alert('XSS')", Angular replaces it with "unsafe:javascript:..."
  userProvidedUrl = signal('https://example.com');
}
```

### Bypassing Security (Use with Extreme Caution)

```typescript
import { DomSanitizer, SafeHtml } from '@angular/platform-browser';

@Component({
  template: `<div [innerHTML]="trustedHtml()"></div>`,
})
export class RichContentComponent {
  private sanitizer = inject(DomSanitizer);

  // Only do this with content you completely control
  trustedHtml = computed(() =>
    this.sanitizer.bypassSecurityTrustHtml(this.rawHtml())
  );

  private rawHtml = signal('<p class="styled">Trusted content with <em>formatting</em></p>');
}
```

**Never bypass security for user-provided content.** Only use `bypassSecurityTrust*` methods for content from your own CMS or build pipeline.

## Common Mistakes

### 1. Confusing Property Binding with Attribute Binding

```html
<!-- BAD: Trying to set colspan as a property (works, but not for all attributes) -->
<td [colspan]="span()">...</td>

<!-- GOOD: Use attribute binding for HTML attributes without DOM properties -->
<td [attr.colspan]="span()">...</td>
```

In practice, Angular maps `[colspan]` to the `colSpan` DOM property correctly for `<td>`. But for attributes like `aria-*`, `data-*`, and SVG attributes, you must use `[attr.*]`.

### 2. Passing Strings Where Booleans Are Expected

```html
<!-- BAD: Passes the string "false", which is truthy! -->
<app-toggle disabled="false"></app-toggle>

<!-- GOOD: Property binding passes the boolean false -->
<app-toggle [disabled]="false"></app-toggle>
```

### 3. Forgetting $event in Event Bindings

```html
<!-- BAD: onInput receives no argument -->
<input (input)="onInput()">

<!-- GOOD: Pass $event to access the event -->
<input (input)="onInput($event)">
```

### 4. Complex Expressions in Event Bindings

```html
<!-- BAD: Too much logic in the template -->
<button (click)="items.update(i => i.filter(x => x.id !== item.id).sort((a,b) => a.name.localeCompare(b.name)))">
  Remove
</button>

<!-- GOOD: Move logic to a method -->
<button (click)="removeItem(item.id)">Remove</button>
```

### 5. Using Two-Way Binding When One-Way Suffices

```html
<!-- BAD: Two-way binding when the child never modifies the value -->
<app-display [(data)]="data"></app-display>

<!-- GOOD: One-way input is sufficient -->
<app-display [data]="data()" />
```

### 6. Not Handling null in Chained Property Access

```html
<!-- BAD: Throws if user() is null -->
<p>{{ user().address.city }}</p>

<!-- GOOD: Optional chaining -->
<p>{{ user()?.address?.city }}</p>

<!-- BETTER: Use @if to guard the block -->
@if (user(); as u) {
  <p>{{ u.address.city }}</p>
}
```

## Best Practices

### 1. Use Property Binding for Non-String Values

Interpolation converts everything to strings. Use `[property]` for booleans, numbers, objects, and arrays.

### 2. Type Your Event Handlers

Always type the `$event` parameter in your component methods. This catches mismatches between the DOM event type and your handler.

### 3. Use model() Only When True Two-Way Sync Is Needed

For most parent-child communication, `input()` + `output()` is clearer. Reserve `model()` for form-like controls where the child genuinely needs to modify a shared value (sliders, toggles, date pickers).

### 4. Use Signal Reads in Bindings

Bindings to signals (`count()`) integrate with Angular's change detection natively. Avoid wrapping signals in getter methods that create new references on each call.

### 5. Keep Template Expressions Simple

If an expression requires more than basic property access, arithmetic, or a pipe, move it to a `computed()` signal.

### 6. Trust Angular's Sanitization

Do not bypass Angular's security unless you have a verified, trusted content source. The built-in sanitization covers HTML, URLs, styles, and resource URLs.

## Real-World Use Cases

### Configurable Data Grid Header

```typescript
@Component({
  selector: 'app-grid-header',
  template: `
    <th
      [class.sortable]="sortable()"
      [class.sorted]="isSorted()"
      [class.asc]="isSorted() && sortDir() === 'asc'"
      [class.desc]="isSorted() && sortDir() === 'desc'"
      [attr.aria-sort]="ariaSortValue()"
      [style.width.px]="width()"
      [style.min-width.px]="minWidth()"
      (click)="onHeaderClick()"
      (keydown.enter)="onHeaderClick()"
      [attr.tabindex]="sortable() ? '0' : null"
      [attr.role]="sortable() ? 'button' : null"
    >
      <ng-content />
      @if (isSorted()) {
        <span class="sort-indicator">{{ sortDir() === 'asc' ? '\u25B2' : '\u25BC' }}</span>
      }
    </th>
  `,
})
export class GridHeaderComponent {
  column = input.required<string>();
  sortable = input(true);
  width = input<number | null>(null);
  minWidth = input(80);

  currentSort = input<{ column: string; direction: 'asc' | 'desc' } | null>(null);
  sortChanged = output<{ column: string; direction: 'asc' | 'desc' }>();

  isSorted = computed(() => this.currentSort()?.column === this.column());
  sortDir = computed(() => this.currentSort()?.direction ?? 'asc');

  ariaSortValue = computed(() => {
    if (!this.isSorted()) return this.sortable() ? 'none' : null;
    return this.sortDir() === 'asc' ? 'ascending' : 'descending';
  });

  onHeaderClick() {
    if (!this.sortable()) return;
    const newDir = this.isSorted() && this.sortDir() === 'asc' ? 'desc' : 'asc';
    this.sortChanged.emit({ column: this.column(), direction: newDir });
  }
}
```

```typescript
@Component({

  `,
  styles: `
    .slider-container { display: flex; align-items: center; gap: 12px; }
    .value-display { min-width: 48px; text-align: right; font-variant-numeric: tabular-nums; }
  `,
})
export class RangeSliderComponent {
  label = input('');
  min = input(0);
  max = input(100);
  step = input(1);
  unit = input('');
  value = model(0); // Two-way bindable

  onSlide(event: Event) {
    this.value.set(+(event.target as HTMLInputElement).value);
  }
}

// Usage with two-way binding:
// <app-range-slider label="Volume" [(value)]="volume" [max]="100" unit="%" />
// <app-range-slider label="Brightness" [(value)]="brightness" [min]="10" [max]="100" unit="%" />
```

### Dynamic Form Field with Event Handling

```typescript
@Component({
  selector: 'app-search-bar',
  imports: [FormsModule],
  template: `
    <div
      class="search-bar"
      [class.focused]="isFocused()"
      [class.has-value]="query().length > 0"
    >
      <input
        type="search"
        [(ngModel)]="query"
        [placeholder]="placeholder()"
        [attr.aria-label]="placeholder()"
        (focus)="isFocused.set(true)"
        (blur)="isFocused.set(false)"
        (keydown.enter)="onSearch()"
        (keydown.escape)="onClear()"
        #searchInput
      />
      @if (query().length > 0) {
        <button
          class="clear-btn"
          (click)="onClear(); searchInput.focus()"
          [attr.aria-label]="'Clear search'"
        >
          X
        </button>
      }
      <button
        class="search-btn"
        (click)="onSearch()"
        [disabled]="query().length < minLength()"
        [attr.aria-label]="'Search'"
      >
        Search
      </button>
    </div>
  `,
})
export class SearchBarComponent {
  placeholder = input('Search...');
  minLength = input(2);
  searched = output<string>();

  query = signal('');
  isFocused = signal(false);

  onSearch() {
    const q = this.query().trim();
    if (q.length >= this.minLength()) {
      this.searched.emit(q);
    }
  }

  onClear() {
    this.query.set('');
  }
}

// Usage:
// <app-search-bar placeholder="Search products..." (searched)="onSearch($event)" />
```

---

> **Key Takeaway:** Angular's binding syntax moves data between component and DOM through four mechanisms: interpolation (text), property binding (DOM properties), event binding (user actions), and two-way binding (bidirectional sync with `model()`). Always bind to DOM properties (not attributes) unless no property exists, type your event handlers, and let Angular's automatic sanitization protect against XSS.
