# 12.4 Component Lifecycle Hooks

> Lifecycle hooks let you tap into key moments of a component's existence, from creation to destruction, to run initialization logic, react to input changes, clean up resources, and interact with the DOM after rendering.

---

## Overview of the Lifecycle

Every Angular component goes through a predictable sequence of phases. Angular provides hooks (methods and functions) that let you execute code at each phase. Some hooks are class methods you implement via interfaces. Others are standalone functions you call during construction.

### Execution Order

The hooks fire in this order during a component's life:

| Order | Hook | Fires When | Frequency |
|---|---|---|---|
| 1 | `constructor` | Class is instantiated | Once |
| 2 | `ngOnChanges` | Input properties change | Once (initial) + on every change |
| 3 | `ngOnInit` | Component is initialized (after first `ngOnChanges`) | Once |
| 4 | `ngDoCheck` | Change detection runs | Every CD cycle |
| 5 | `ngAfterContentInit` | Projected content (`ng-content`) is initialized | Once |
| 6 | `ngAfterContentChecked` | Projected content is checked | Every CD cycle |
| 7 | `ngAfterViewInit` | Component's view and child views are initialized | Once |
| 8 | `ngAfterViewChecked` | Component's view and child views are checked | Every CD cycle |
| 9 | `ngOnDestroy` | Component is about to be destroyed | Once |

Additionally, Angular provides standalone functions that run after the rendering pipeline completes:

| Function | Fires When | Frequency |
|---|---|---|
| `afterNextRender` | After the next render cycle completes | Once |
| `afterEveryRender` | After every render cycle completes | Every render |
| `afterRenderEffect` | After render, reactively when tracked signals change | Reactive |

---

## Constructor vs ngOnInit

This is the most commonly confused distinction in Angular lifecycle.

### Constructor

The constructor is a TypeScript/JavaScript class feature, not an Angular feature. It runs when the class is instantiated. At this point:

- The component instance exists
- `inject()` works (the injection context is active)
- Input properties have NOT been set yet
- The DOM does NOT exist yet
- Signal inputs return their default/initial values

```typescript
@Component({ /* ... */ })
export class UserProfileComponent {
  private userService = inject(UserService); // inject() works here

  name = input.required<string>(); // NOT yet populated with parent's value

  constructor() {
    console.log(this.name()); // Throws: required input not yet set
  }
}
```

### ngOnInit

`ngOnInit` runs once after Angular has set all input properties for the first time. This is where you should place initialization logic that depends on inputs.

```typescript
@Component({ /* ... */ })
export class UserProfileComponent implements OnInit {
  private userService = inject(UserService);

  userId = input.required<number>();
  user = signal<User | null>(null);

  ngOnInit(): void {
    // Inputs are available here
    console.log(this.userId()); // Has the value from the parent

    // Fetch data based on input
    this.userService.getUser(this.userId()).subscribe(u => {
      this.user.set(u);
    });
  }
}
```

### When signal inputs reduce the need for ngOnInit

With signal-based reactive patterns, you can often eliminate `ngOnInit` entirely:

```typescript
@Component({ /* ... */ })
export class UserProfileComponent {
  private userService = inject(UserService);

  userId = input.required<number>();

  // Reactive: automatically re-fetches when userId changes
  userResource = httpResource<User>(() => `/api/users/${this.userId()}`);

  // Or with computed for derived state
  displayName = computed(() => {
    const user = this.userResource.value();
    return user ? `${user.firstName} ${user.lastName}` : 'Loading...';
  });

  // No ngOnInit needed
}
```

The `httpResource` and `computed` patterns react to signal changes automatically. There is no need to manually wire up initialization in `ngOnInit`.

**Rule of thumb**: If your `ngOnInit` only subscribes to data based on inputs, consider replacing it with `resource`, `httpResource`, `computed`, or `effect`.

## ngOnChanges

Fires every time one or more input properties change, including the initial setting. Receives a `SimpleChanges` object.

```typescript
@Component({ /* ... */ })
export class PriceDisplayComponent implements OnChanges {
  price = input.required<number>();
  currency = input<string>('USD');

  formattedPrice = '';

  ngOnChanges(changes: SimpleChanges): void {
    // changes is keyed by input property name
    if (changes['price'] || changes['currency']) {
      const priceChange = changes['price'];

      if (priceChange) {
        console.log('Previous:', priceChange.previousValue);
        console.log('Current:', priceChange.currentValue);
        console.log('First change:', priceChange.firstChange);
      }

      this.formattedPrice = this.formatPrice(this.price(), this.currency());
    }
  }

  private formatPrice(price: number, currency: string): string {
    return new Intl.NumberFormat('en-US', { style: 'currency', currency }).format(price);
  }
}
```

### SimpleChanges object structure

Each key in `SimpleChanges` is a `SimpleChange` with three properties:

| Property | Type | Description |
|---|---|---|
| `previousValue` | `any` | Previous value (undefined on first change) |
| `currentValue` | `any` | Current value |
| `firstChange` | `boolean` | `true` if this is the initial setting |

### When to prefer computed or effect over ngOnChanges

`ngOnChanges` is a legacy pattern. With signal inputs, derived state is better handled reactively:

```typescript
// Legacy: ngOnChanges
@Component({ /* ... */ })
export class LegacyComponent implements OnChanges {
  price = input.required<number>();
  currency = input<string>('USD');
  formattedPrice = '';

  ngOnChanges(): void {
    this.formattedPrice = new Intl.NumberFormat('en-US', {
      style: 'currency',
      currency: this.currency(),
    }).format(this.price());
  }
}

// Modern: computed signal (preferred)
@Component({ /* ... */ })
export class ModernComponent {
  price = input.required<number>();
  currency = input<string>('USD');

  formattedPrice = computed(() =>
    new Intl.NumberFormat('en-US', {
      style: 'currency',
      currency: this.currency(),
    }).format(this.price())
  );
}
```

The `computed` version is declarative, lazy (only recalculates when read and dependencies changed), and does not require implementing an interface.

## ngDoCheck

Fires on every change detection cycle, immediately after `ngOnChanges` (or after `ngOnInit` on the first cycle). This hook gives you a chance to implement custom change detection logic.

```typescript
@Component({ /* ... */ })
export class DeepCheckComponent implements DoCheck {
  data = input<ComplexObject>();

  private previousData: ComplexObject | undefined;

  ngDoCheck(): void {
    const current = this.data();
    if (current && !this.deepEqual(current, this.previousData)) {
      this.previousData = structuredClone(current);
      this.handleDeepChange();
    }
  }

  private deepEqual(a: unknown, b: unknown): boolean {
    return JSON.stringify(a) === JSON.stringify(b);
  }

  private handleDeepChange(): void {
    // React to deep changes
  }
}
```

**Warning**: `ngDoCheck` runs on every change detection cycle, even when no inputs changed. Placing expensive logic here will degrade performance. Use it only when Angular's default reference-based change detection is insufficient and you need deep comparison.

## Content and View Init/Checked Hooks

### ngAfterContentInit and ngAfterContentChecked

These hooks relate to **projected content** (content passed through `<ng-content>`).

```typescript
@Component({
  selector: 'app-card',
  template: `
    <div class="card">
      <ng-content select="[card-header]" />
      <div class="card-body">
        <ng-content />
      </div>
    </div>
  `
})
export class CardComponent implements AfterContentInit, AfterContentChecked {
  headerContent = contentChild<ElementRef>('cardHeader');

  ngAfterContentInit(): void {
    // Projected content is now available
    // contentChild/contentChildren queries are resolved
    console.log('Header projected:', !!this.headerContent());
  }

  ngAfterContentChecked(): void {
    // Runs after every check of projected content
    // Use sparingly due to frequency
  }
}
```

- `ngAfterContentInit`: Runs once after projected content is initialized. Content queries (`contentChild`, `contentChildren`) are resolved by this point.
- `ngAfterContentChecked`: Runs after every change detection check of the projected content. Avoid expensive operations here.

### ngAfterViewInit and ngAfterViewChecked

These hooks relate to the **component's own view** (its template and child components).

```typescript
@Component({
  selector: 'app-chart',
  template: `<canvas #chartCanvas></canvas>`
})
export class ChartComponent implements AfterViewInit {
  canvas = viewChild.required<ElementRef<HTMLCanvasElement>>('chartCanvas');

  ngAfterViewInit(): void {
    // The view is fully rendered. DOM elements are available.
    // viewChild/viewChildren queries are resolved.
    const ctx = this.canvas().nativeElement.getContext('2d');
    this.initChart(ctx);
  }

  private initChart(ctx: CanvasRenderingContext2D | null): void {
    // Initialize a chart library
  }
}
```

- `ngAfterViewInit`: Runs once after the view (template + child components) is initialized. View queries (`viewChild`, `viewChildren`) are resolved. This is the classic place to access DOM elements.
- `ngAfterViewChecked`: Runs after every check of the view. Same performance caveat as `ngAfterContentChecked`.

### Key difference: Content vs View

| Aspect | Content hooks | View hooks |
|---|---|---|
| What they observe | Projected content (`ng-content`) | Component's own template |
| Query type | `contentChild` / `contentChildren` | `viewChild` / `viewChildren` |
| Runs before | View hooks | Nothing (last in init sequence) |
| Common use case | Reading projected elements | DOM manipulation, chart init |

## ngOnDestroy and DestroyRef

### ngOnDestroy

Fires once when a component is about to be removed from the DOM.

```typescript
@Component({ /* ... */ })
export class PollingComponent implements OnInit, OnDestroy {
  private intervalId: ReturnType<typeof setInterval> | null = null;

  ngOnInit(): void {
    this.intervalId = setInterval(() => this.poll(), 5000);
  }

  ngOnDestroy(): void {
    if (this.intervalId) {
      clearInterval(this.intervalId);
    }
  }

  private poll(): void {
    // Fetch updated data
  }
}
```

### DestroyRef (Modern Alternative)

`DestroyRef` is an injectable service that lets you register cleanup callbacks without implementing the `OnDestroy` interface. It was introduced in Angular 16 and is the preferred approach in modern Angular.

```typescript
@Component({ /* ... */ })
export class PollingComponent {
  private destroyRef = inject(DestroyRef);

  constructor() {
    const intervalId = setInterval(() => this.poll(), 5000);

    this.destroyRef.onDestroy(() => {
      clearInterval(intervalId);
    });
  }

  private poll(): void {
    // Fetch updated data
  }
}
```

### Why DestroyRef is better than ngOnDestroy

| Feature | `ngOnDestroy` | `DestroyRef` |
|---|---|---|
| Requires interface | Yes (`implements OnDestroy`) | No |
| Can be used in services | Yes | Yes |
| Enables reusable cleanup logic | Only via inheritance | Via composition (extract to utility) |
| Works with `inject()` pattern | Awkward (method on class) | Natural (inject + register) |
| Multiple cleanup callbacks | Must coordinate in one method | Register multiple independently |

### Reusable cleanup with DestroyRef

Because `DestroyRef` is injected, you can create utility functions that encapsulate cleanup logic:

```typescript
// Utility function: auto-clear interval on destroy
function autoInterval(callback: () => void, ms: number): void {
  const destroyRef = inject(DestroyRef);
  const id = setInterval(callback, ms);
  destroyRef.onDestroy(() => clearInterval(id));
}

// Usage in a component
@Component({ /* ... */ })
export class DashboardComponent {
  constructor() {
    autoInterval(() => this.refreshData(), 10000);
  }

  private refreshData(): void { /* ... */ }
}
```

This is not possible with `ngOnDestroy` without inheritance or mixins.

### takeUntilDestroyed

For RxJS subscriptions, `takeUntilDestroyed` is the preferred pattern:

```typescript
@Component({ /* ... */ })
export class SearchComponent {
  private searchService = inject(SearchService);

  constructor() {
    // Automatically unsubscribes when component is destroyed
    this.searchService.results$
      .pipe(takeUntilDestroyed())
      .subscribe(results => {
        // handle results
      });
  }
}
```

`takeUntilDestroyed()` without arguments only works inside an injection context (constructor or field initializer). If you need it elsewhere, pass a `DestroyRef`:

```typescript
@Component({ /* ... */ })
export class SearchComponent {
  private destroyRef = inject(DestroyRef);

  setupLater(): void {
    someObservable$
      .pipe(takeUntilDestroyed(this.destroyRef))
      .subscribe(/* ... */);
  }
}
```

## afterNextRender

`afterNextRender` registers a callback to run once after the next rendering cycle completes. It runs in the browser only (not during SSR). This is the modern replacement for most `ngAfterViewInit` DOM-access patterns.

```typescript
@Component({
  selector: 'app-map',
  template: `<div #mapContainer class="map"></div>`
})
export class MapComponent {
  mapContainer = viewChild.required<ElementRef<HTMLDivElement>>('mapContainer');

  constructor() {
    afterNextRender(() => {
      // DOM is ready. Initialize a third-party library.
      const map = new mapboxgl.Map({
        container: this.mapContainer().nativeElement,
        style: 'mapbox://styles/mapbox/streets-v12',
        center: [-74.5, 40],
        zoom: 9,
      });
    });
  }
}
```

### Key properties of afterNextRender

- Runs once, after the next render cycle
- Runs only in the browser (skipped during SSR/prerendering)
- Must be called in an injection context (constructor or field initializer)
- Does not require implementing any interface

### Phases

`afterNextRender` accepts a phase configuration for optimizing DOM reads and writes:

```typescript
constructor() {
  afterNextRender({
    earlyRead: () => {
      // Read DOM layout before any writes
      return this.element().nativeElement.getBoundingClientRect();
    },
    write: (rect) => {
      // rect is the return value from earlyRead
      // Write to the DOM based on the measurement
      this.tooltip().nativeElement.style.top = `${rect.bottom + 8}px`;
      this.tooltip().nativeElement.style.left = `${rect.left}px`;
    },
  });
}
```

The four phases execute in order: `earlyRead` -> `write` -> `mixedReadWrite` -> `read`. Each phase callback receives the return value of the previous phase.

| Phase | Purpose | Rule |
|---|---|---|
| `earlyRead` | Read DOM layout before writes | Do NOT write to the DOM |
| `write` | Mutate the DOM | Do NOT read layout properties |
| `mixedReadWrite` | Both read and write (escape hatch) | Avoid if possible |
| `read` | Final DOM reads after all writes | Do NOT write to the DOM |

Separating reads and writes prevents layout thrashing, where the browser is forced to recalculate layout between interleaved reads and writes.

## afterEveryRender

`afterEveryRender` registers a callback to run after every rendering cycle. Use it for ongoing DOM synchronization.

```typescript
@Component({
  selector: 'app-scroll-spy',
  template: `<div #content class="scrollable"><ng-content /></div>`
})
export class ScrollSpyComponent {
  content = viewChild.required<ElementRef<HTMLDivElement>>('content');
  scrollProgress = signal(0);

  constructor() {
    afterEveryRender({
      read: () => {
        const el = this.content().nativeElement;
        const progress = el.scrollTop / (el.scrollHeight - el.clientHeight);
        this.scrollProgress.set(Math.round(progress * 100));
      },
    });
  }
}
```

**Caution**: Like `ngAfterViewChecked`, this runs frequently. Keep the callback lightweight.

---

## afterRenderEffect

`afterRenderEffect` is the reactive version of `afterEveryRender`. It automatically tracks signal dependencies and only re-runs when those signals change and a render occurs. Introduced in Angular 19 (developer preview).

```typescript
@Component({
  selector: 'app-highlight',
  template: `<pre><code #codeBlock>{{ code() }}</code></pre>`
})
export class HighlightComponent {
  codeBlock = viewChild.required<ElementRef<HTMLElement>>('codeBlock');
  code = input.required<string>();
  language = input<string>('typescript');

  constructor() {
    afterRenderEffect({
      write: () => {
        // Automatically re-runs when code() or language() signals change
        const el = this.codeBlock().nativeElement;
        hljs.highlightElement(el);
      },
    });
  }
}
```

### afterRenderEffect vs afterEveryRender vs effect

| Feature | `effect()` | `afterEveryRender()` | `afterRenderEffect()` |
|---|---|---|---|
| When it runs | After signal change (timing varies) | After every render | After render, when tracked signals change |
| DOM access safe | No (may run before render) | Yes | Yes |
| Signal tracking | Automatic | None (runs always) | Automatic |
| Phase support | No | Yes | Yes |
| Use case | Non-DOM side effects | Unconditional DOM sync | Reactive DOM sync |

**Choose `afterRenderEffect` when**: You need to interact with the DOM in response to signal changes. It is the most efficient option because it combines signal tracking with post-render timing.

**Choose `afterEveryRender` when**: You need to run DOM logic unconditionally after every render, regardless of which signals changed.

**Choose `effect()` when**: Your side effect does not involve the DOM (logging, analytics, localStorage sync).

## Lifecycle Hooks in Directives

Directives have access to the same lifecycle hooks as components, except for view-related hooks (`ngAfterViewInit`, `ngAfterViewChecked`) since directives do not have views.

```typescript
@Directive({
  selector: '[appAutoFocus]',
  standalone: true,
})
export class AutoFocusDirective {
  private el = inject(ElementRef);

  constructor() {
    afterNextRender(() => {
      this.el.nativeElement.focus();
    });
  }
}
```

Directives do have content hooks (`ngAfterContentInit`, `ngAfterContentChecked`) and can use content queries.

## Signal-Based Replacements for Lifecycle Hooks

Many traditional lifecycle hook patterns can be replaced with signal-based alternatives:

| Legacy Pattern | Signal-Based Replacement |
|---|---|
| `ngOnInit` to fetch data based on input | `httpResource` / `resource` with signal input |
| `ngOnChanges` to derive state from inputs | `computed()` from signal inputs |
| `ngOnChanges` to trigger side effects on input change | `effect()` reading signal inputs |
| `ngOnDestroy` for cleanup | `DestroyRef.onDestroy()` |
| `ngAfterViewInit` for one-time DOM access | `afterNextRender()` |
| `ngAfterViewChecked` for continuous DOM sync | `afterRenderEffect()` or `afterEveryRender()` |
| `ngDoCheck` for deep change detection | `computed()` with custom equality |

```typescript
// Before: Multiple lifecycle hooks
@Component({ /* ... */ })
export class LegacyUserComponent implements OnInit, OnChanges, OnDestroy, AfterViewInit {
  @Input() userId!: number;
  user: User | null = null;
  @ViewChild('avatar') avatarEl!: ElementRef;
  private subscription?: Subscription;

  ngOnInit(): void {
    this.loadUser();
  }

  ngOnChanges(changes: SimpleChanges): void {
    if (changes['userId'] && !changes['userId'].firstChange) {
      this.loadUser();
    }
  }

  ngAfterViewInit(): void {
    this.initAvatarWidget(this.avatarEl.nativeElement);
  }

  ngOnDestroy(): void {
    this.subscription?.unsubscribe();
  }

  private loadUser(): void {
    this.subscription?.unsubscribe();
    this.subscription = this.userService.getUser(this.userId).subscribe(u => {
      this.user = u;
    });
  }

  private initAvatarWidget(el: HTMLElement): void { /* ... */ }
}

// After: Signal-based, no lifecycle hooks
@Component({ /* ... */ })
export class ModernUserComponent {
  userId = input.required<number>();

  // Replaces ngOnInit + ngOnChanges for data fetching
  userResource = httpResource<User>(() => `/api/users/${this.userId()}`);

  avatarEl = viewChild.required<ElementRef>('avatar');

  constructor() {
    // Replaces ngAfterViewInit
    afterNextRender(() => {
      this.initAvatarWidget(this.avatarEl().nativeElement);
    });
    // No ngOnDestroy needed: httpResource manages its own cleanup
  }

  private initAvatarWidget(el: HTMLElement): void { /* ... */ }
}
```

## Common Mistakes

### 1. Putting logic in the constructor that depends on inputs

```typescript
// BAD: Inputs are not set yet in the constructor
@Component({ /* ... */ })
export class BadComponent {
  userId = input.required<number>();

  constructor() {
    console.log(this.userId()); // ERROR: required input not set
    this.fetchData(this.userId()); // Will fail
  }
}

// GOOD: Use ngOnInit, or better, use resource/computed
@Component({ /* ... */ })
export class GoodComponent {
  userId = input.required<number>();

  userResource = httpResource<User>(() => `/api/users/${this.userId()}`);
}
```

### 2. Modifying the DOM in ngAfterViewInit without considering SSR

```typescript
// BAD: document is not available during SSR
@Component({ /* ... */ })
export class BadComponent implements AfterViewInit {
  ngAfterViewInit(): void {
    document.getElementById('chart')?.setAttribute('data-ready', 'true');
  }
}

// GOOD: afterNextRender is automatically skipped during SSR
@Component({ /* ... */ })
export class GoodComponent {
  constructor() {
    afterNextRender(() => {
      document.getElementById('chart')?.setAttribute('data-ready', 'true');
    });
  }
}
```

### 3. Forgetting to clean up subscriptions

```typescript
// BAD: Memory leak
@Component({ /* ... */ })
export class BadComponent implements OnInit {
  ngOnInit(): void {
    this.dataService.stream$.subscribe(data => {
      this.data = data;
    });
    // Never unsubscribed!
  }
}

// GOOD: Use takeUntilDestroyed
@Component({ /* ... */ })
export class GoodComponent {
  constructor() {
    inject(DataService).stream$
      .pipe(takeUntilDestroyed())
      .subscribe(data => {
        this.data.set(data);
      });
  }
}
```

### 4. Expensive operations in frequently-called hooks

```typescript
// BAD: Deep comparison every change detection cycle
@Component({ /* ... */ })
export class BadComponent implements DoCheck {
  ngDoCheck(): void {
    // This runs on EVERY change detection cycle
    const result = this.expensiveDeepComparison(this.data);
    if (result) { /* ... */ }
  }
}

// GOOD: Use computed with custom equality
@Component({ /* ... */ })
export class GoodComponent {
  data = input.required<ComplexObject>();

  derivedState = computed(
    () => this.processData(this.data()),
    { equal: (a, b) => JSON.stringify(a) === JSON.stringify(b) }
  );
}
```

### 5. Using ngAfterViewInit instead of afterNextRender for DOM library init

`afterNextRender` is SSR-safe and runs only in the browser. `ngAfterViewInit` runs during SSR too, where DOM APIs like `document`, `window`, and `getBoundingClientRect()` are not available. Always prefer `afterNextRender` for DOM interaction.

### 6. Modifying state in ngAfterViewChecked causing infinite loops

```typescript
// BAD: Can trigger ExpressionChangedAfterItHasBeenCheckedError
@Component({ /* ... */ })
export class BadComponent implements AfterViewChecked {
  counter = 0;

  ngAfterViewChecked(): void {
    this.counter++; // Modifies state during check = infinite loop risk
  }
}
```

The "checked" hooks (`ngAfterContentChecked`, `ngAfterViewChecked`) run during change detection. Modifying state that is bound in the template triggers another change detection cycle, leading to the `ExpressionChangedAfterItHasBeenCheckedError`.

---

## Best Practices

### 1. Prefer signal-based patterns over lifecycle hooks

Use `computed` instead of `ngOnChanges`, `httpResource`/`resource` instead of `ngOnInit` data fetching, `DestroyRef` instead of `ngOnDestroy`, and `afterNextRender` instead of `ngAfterViewInit`.

### 2. Use afterNextRender for all one-time DOM initialization

It is SSR-safe, does not require implementing an interface, and clearly communicates intent.

### 3. Use separate phases in afterNextRender to avoid layout thrashing

If you need to read layout properties and then write to the DOM, use `earlyRead` and `write` phases rather than doing both in a single callback.

### 4. Keep "checked" hooks lightweight

`ngDoCheck`, `ngAfterContentChecked`, and `ngAfterViewChecked` run on every change detection cycle. If you must use them, keep the logic minimal with early returns.

### 5. Use DestroyRef.onDestroy for cleanup in utility functions

This pattern enables reusable cleanup logic that cannot be achieved with `ngOnDestroy` alone.

### 6. Never modify template-bound state in "checked" hooks

Any modification to state that the template reads will trigger another change detection cycle and potentially an infinite loop or the `ExpressionChangedAfterItHasBeenCheckedError`.

## Real-World Use Cases

### Third-Party Chart Library Integration

```typescript
@Component({
  selector: 'app-sales-chart',
  standalone: true,
  template: `<canvas #chartCanvas></canvas>`
})
export class SalesChartComponent {
  private salesService = inject(SalesService);

  canvas = viewChild.required<ElementRef<HTMLCanvasElement>>('chartCanvas');
  period = input<'week' | 'month' | 'year'>('month');

  salesData = httpResource<SalesData[]>(() => `/api/sales?period=${this.period()}`);

  private chart: Chart | null = null;

  constructor() {
    // One-time chart initialization after DOM is ready
    afterNextRender(() => {
      const ctx = this.canvas().nativeElement.getContext('2d')!;
      this.chart = new Chart(ctx, {
        type: 'bar',
        data: { labels: [], datasets: [] },
        options: { responsive: true },
      });
    });

    // Reactively update chart data when salesData changes
    afterRenderEffect({
      write: () => {
        const data = this.salesData.value();
        if (this.chart && data) {
          this.chart.data.labels = data.map(d => d.label);
          this.chart.data.datasets[0] = {
            label: 'Sales',
            data: data.map(d => d.amount),
          };
          this.chart.update();
        }
      },
    });

    // Cleanup on destroy
    inject(DestroyRef).onDestroy(() => {
      this.chart?.destroy();
    });
  }
}
```

### WebSocket Connection with Automatic Reconnection

```typescript
@Component({
  selector: 'app-live-feed',
  standalone: true,
  template: `
    @if (connectionStatus() === 'connected') {
      <span class="badge online">Live</span>
    } @else {
      <span class="badge offline">Reconnecting...</span>
    }

    @for (message of messages(); track message.id) {
      <app-feed-item [message]="message" />
    } @empty {
      <p>No messages yet.</p>
    }
  `
})
export class LiveFeedComponent {
  private destroyRef = inject(DestroyRef);
  private feedUrl = input.required<string>();

  messages = signal<FeedMessage[]>([]);
  connectionStatus = signal<'connected' | 'disconnected'>('disconnected');

  private ws: WebSocket | null = null;

  constructor() {
    // Use effect to react to feedUrl changes
    effect(() => {
      const url = this.feedUrl();
      this.connect(url);
    });

    // Cleanup WebSocket on destroy
    this.destroyRef.onDestroy(() => {
      this.ws?.close();
    });
  }

  private connect(url: string): void {
    this.ws?.close();
    this.ws = new WebSocket(url);

    this.ws.onopen = () => this.connectionStatus.set('connected');
    this.ws.onclose = () => {
      this.connectionStatus.set('disconnected');
      // Reconnect after 3 seconds
      setTimeout(() => this.connect(url), 3000);
    };
    this.ws.onmessage = (event) => {
      const msg: FeedMessage = JSON.parse(event.data);
      this.messages.update(list => [msg, ...list].slice(0, 100));
    };
  }
}
```

### Responsive Element with ResizeObserver

```typescript
@Component({
  selector: 'app-responsive-grid',
  standalone: true,
  template: `
    <div #gridContainer class="grid" [class.compact]="isCompact()">
      <ng-content />
    </div>
  `
})
export class ResponsiveGridComponent {
  container = viewChild.required<ElementRef<HTMLDivElement>>('gridContainer');
  breakpoint = input<number>(600);

  isCompact = signal(false);

  constructor() {
    const destroyRef = inject(DestroyRef);

    afterNextRender(() => {
      const el = this.container().nativeElement;

      const observer = new ResizeObserver(entries => {
        const width = entries[0].contentRect.width;
        this.isCompact.set(width < this.breakpoint());
      });

      observer.observe(el);

      destroyRef.onDestroy(() => {
        observer.disconnect();
      });
    });
  }
}
```

This pattern combines `afterNextRender` (SSR-safe DOM access), `DestroyRef` (cleanup), and signals (reactive state update) without any lifecycle hook interfaces.

---

> **Key Takeaway:** Modern Angular minimizes the need for class-based lifecycle hooks by providing signal-based alternatives (`computed`, `effect`, `resource`) and standalone functions (`afterNextRender`, `afterRenderEffect`, `DestroyRef`). Reach for lifecycle hooks only when the signal-based approach does not fit, and always prefer `afterNextRender` over `ngAfterViewInit` for DOM interaction.
