# 12.6 Signal Outputs

> The `output()` function provides a lightweight, type-safe way to declare component outputs that emit custom events to parent components, replacing the `@Output()` decorator and `EventEmitter` with a simpler API that has zero RxJS overhead.

---

## Why output() Over @Output and EventEmitter

The legacy pattern for emitting events from a child component required two imports and a decorator:

```typescript
// Legacy
import { Output, EventEmitter } from '@angular/core';

@Component({ /* ... */ })
export class LegacyComponent {
  @Output() saved = new EventEmitter<string>();

  save(): void {
    this.saved.emit('data');
  }
}
```

This approach has several drawbacks:

1. **EventEmitter extends RxJS Subject.** Every output carries the full RxJS machinery, even if you never use operators on it.
2. **EventEmitter has broken types.** `EventEmitter<string>` does not prevent you from calling `this.saved.emit()` without arguments at runtime, despite the generic type.
3. **The `@Output()` decorator is verbose.** Two imports, a decorator, and a class instantiation for a simple event.

The `output()` function solves all of these:

```typescript
// Modern
import { output } from '@angular/core';

@Component({ /* ... */ })
export class ModernComponent {
  saved = output<string>();

  save(): void {
    this.saved.emit('data');
  }
}
```

- No RxJS dependency. The `OutputEmitterRef` is a lightweight emitter.
- Type-safe. `output<string>()` requires `emit(value: string)`. Calling `emit()` without an argument is a compile error.
- Less boilerplate. One import, one function call.

## Declaring Outputs

### Basic Output (with payload)

```typescript
import { Component, output } from '@angular/core';

@Component({
  selector: 'app-search-bar',
  standalone: true,
  template: `
    <input
      type="text"
      [value]="query"
      (input)="onInput($event)"
      (keydown.enter)="search()"
    />
    <button (click)="search()">Search</button>
  `
})
export class SearchBarComponent {
  searchSubmitted = output<string>();  // OutputEmitterRef<string>

  query = '';

  onInput(event: Event): void {
    this.query = (event.target as HTMLInputElement).value;
  }

  search(): void {
    this.searchSubmitted.emit(this.query);
  }
}
```

```html
<!-- Parent template -->
<app-search-bar (searchSubmitted)="onSearch($event)" />
```

The `$event` in the parent template is typed as `string` because the output was declared as `output<string>()`.

### Void Output (no payload)

When an output signals that something happened without carrying data, use `output()` without a type argument (or explicitly `output<void>()`):

```typescript
@Component({
  selector: 'app-modal',
  standalone: true,
  template: `
    <div class="backdrop" (click)="closed.emit()"></div>
    <div class="modal">
      <ng-content />
      <button (click)="closed.emit()">Close</button>
    </div>
  `
})
export class ModalComponent {
  closed = output();  // OutputEmitterRef<void>
}
```

```html
<!-- Parent -->
<app-modal (closed)="showModal.set(false)" />
```

For void outputs, `emit()` takes no arguments.

### Output with Alias

Like signal inputs, outputs support an `alias` option that changes the public name of the output as seen by the parent:

```typescript
@Component({
  selector: 'app-color-picker',
  standalone: true,
  template: `<!-- ... -->`
})
export class ColorPickerComponent {
  // Parent binds to (colorChange), component uses colorSelected internally
  colorSelected = output<string>({ alias: 'colorChange' });

  pickColor(hex: string): void {
    this.colorSelected.emit(hex);
  }
}
```

```html
<!-- Parent uses the alias -->
<app-color-picker (colorChange)="onColorChange($event)" />
```

Use aliases sparingly, primarily for backward compatibility during renames.

## OutputEmitterRef API

The `output()` function returns an `OutputEmitterRef<T>`, which has two methods:

| Method | Signature | Description |
|---|---|---|
| `emit` | `emit(value: T): void` | Emits a value to all subscribers (parent template binding + programmatic) |
| `subscribe` | `subscribe(callback: (value: T) => void): OutputRefSubscription` | Registers a callback that runs when the output emits |

### emit()

The primary way to fire an output. Call it from component methods, event handlers, or effects.

```typescript
@Component({ /* ... */ })
export class CounterComponent {
  countChanged = output<number>();

  count = signal(0);

  increment(): void {
    this.count.update(n => n + 1);
    this.countChanged.emit(this.count());
  }
}
```

### subscribe()

The `subscribe` method enables programmatic subscription, primarily useful with dynamically created components.

```typescript
@Component({ /* ... */ })
export class DynamicHostComponent {
  private vcr = inject(ViewContainerRef);

  createDialog(): void {
    const ref = this.vcr.createComponent(ConfirmDialogComponent);

    // Programmatic subscription to the output
    const sub = ref.instance.confirmed.subscribe((result) => {
      console.log('Dialog confirmed:', result);
      ref.destroy();
    });
  }
}
```

The `OutputRefSubscription` returned by `subscribe` has an `unsubscribe()` method:

```typescript
const sub = ref.instance.confirmed.subscribe(handleConfirm);

// Later, if needed
sub.unsubscribe();
```

**Important**: Angular automatically cleans up subscriptions created by template bindings `(outputName)="handler()"` when the component is destroyed. Programmatic subscriptions via `subscribe()` are also cleaned up when the owning component is destroyed. Manual unsubscription is only needed if you want to stop listening before the component is destroyed.

## OutputEmitterRef is NOT a Signal

Despite being part of Angular's modern API alongside signal inputs, `output()` does not create a signal. It creates an emitter. This is a deliberate design choice:

- **Signals** represent state (a value that changes over time and can be read at any moment).
- **Outputs** represent events (discrete occurrences that happen at specific points in time).

You cannot read the "current value" of an output. You can only emit values through it and subscribe to its emissions.

```typescript
@Component({ /* ... */ })
export class ExampleComponent {
  clicked = output<void>();

  // This does NOT work. Outputs are not signals.
  // console.log(this.clicked());  // ERROR: not callable
}
```

## Emitting from Effects and Computed

### Emitting from effect()

You can emit outputs inside an `effect()` to fire events when signals change:

```typescript
@Component({ /* ... */ })
export class SliderComponent {
  value = model(0);
  valueChanged = output<number>();

  constructor() {
    effect(() => {
      // Emit whenever value changes
      this.valueChanged.emit(this.value());
    });
  }
}
```

Be cautious with this pattern. The effect runs on every change to the tracked signal, which may fire more often than intended.

### Do NOT wrap outputs in computed

```typescript
// This makes no sense. Computed is for derived values, not events.
// Outputs are imperative emissions, not reactive derivations.
myOutput = computed(() => this.someOutput.emit(this.data())); // WRONG
```

## RxJS Interop

Angular provides two interop functions in `@angular/core/rxjs-interop` for bridging outputs with RxJS.

### outputFromObservable

Converts an RxJS Observable into an Angular output. The output emits whenever the observable emits a `next` notification.

```typescript
import { Component } from '@angular/core';
import { outputFromObservable } from '@angular/core/rxjs-interop';
import { interval, map } from 'rxjs';

@Component({
  selector: 'app-ticker',
  standalone: true,
  template: `<p>Tick count emitted as output</p>`
})
export class TickerComponent {
  // Observable source
  private tick$ = interval(1000).pipe(map(i => i + 1));

  // Expose as Angular output
  ticked = outputFromObservable(this.tick$);
}
```

```html
<!-- Parent subscribes via template binding -->
<app-ticker (ticked)="onTick($event)" />
```

**Behavior details:**
- `next` notifications are forwarded as output emissions
- `error` notifications are NOT handled by Angular (they propagate as uncaught errors)
- `complete` notifications stop the output from emitting further values
- Angular unsubscribes from the observable when the component is destroyed

**Important**: `outputFromObservable` must be called in a component or directive property initializer, just like `input()` and `output()`.

### outputToObservable

Converts an `OutputRef` (from `output()` or `outputFromObservable()`) into an RxJS Observable. Useful when a parent component needs to apply RxJS operators to a child's output.

```typescript
import { Component, viewChild } from '@angular/core';
import { outputToObservable } from '@angular/core/rxjs-interop';
import { debounceTime, distinctUntilChanged } from 'rxjs';

@Component({
  selector: 'app-search-page',
  standalone: true,
  imports: [SearchBarComponent],
  template: `
    <app-search-bar #searchBar />
    <app-results [query]="debouncedQuery()" />
  `
})
export class SearchPageComponent {
  searchBar = viewChild.required(SearchBarComponent);
  debouncedQuery = signal('');

  constructor() {
    afterNextRender(() => {
      outputToObservable(this.searchBar().searchSubmitted)
        .pipe(
          debounceTime(300),
          distinctUntilChanged(),
          takeUntilDestroyed(inject(DestroyRef))
        )
        .subscribe(query => this.debouncedQuery.set(query));
    });
  }
}
```

However, this pattern is rarely needed. In most cases, the child component should handle debouncing internally, and the parent should receive already-processed events.

## Naming Conventions

### Use past tense for event outputs

Output names should describe what happened, not what should happen. They are events, not commands.

```typescript
// GOOD: Past tense describes what happened
saved = output<Item>();
deleted = output<string>();
formSubmitted = output<FormData>();
selectionChanged = output<string[]>();
dialogClosed = output<void>();

// BAD: Imperative/command style
save = output<Item>();        // Sounds like a command
delete = output<string>();    // Sounds like a command
onSubmit = output<FormData>();// "on" prefix is redundant with (event) binding
```

### Match the Angular convention: `verbNoun` or `nounVerbed`

```typescript
// Common patterns
itemSelected = output<Item>();
pageChanged = output<number>();
filterApplied = output<FilterCriteria>();
rowClicked = output<RowData>();
```

### Avoid collisions with native DOM events

Do not name outputs `click`, `change`, `input`, `submit`, `focus`, `blur`, or other native event names. The parent template would have ambiguity between the component output and the native DOM event.

```typescript
// BAD: Collides with native 'change' event
change = output<string>();

// GOOD: Specific to the component
valueChanged = output<string>();
```

## @Output and EventEmitter (Legacy)

You will encounter this pattern in existing codebases.

### Legacy syntax

```typescript
import { Component, Output, EventEmitter } from '@angular/core';

@Component({ /* ... */ })
export class LegacyComponent {
  @Output() itemSelected = new EventEmitter<Item>();
  @Output('publicName') internalName = new EventEmitter<void>(); // Aliased

  selectItem(item: Item): void {
    this.itemSelected.emit(item);
  }
}
```

### EventEmitter quirks

`EventEmitter` extends RxJS `Subject`, which means it has `pipe()`, `subscribe()`, `next()`, `error()`, and `complete()`. However:

- **Calling `next()` instead of `emit()`**: Both work because `emit` delegates to `next`, but `next` bypasses the type safety that `emit` provides.
- **Calling `complete()` or `error()`**: Permanently closes the emitter. Any subsequent `emit()` calls are silently ignored. This is almost always a bug.
- **Using `pipe()` on an output**: While technically possible, it creates a new Observable that is disconnected from the output. The parent template still subscribes to the original emitter.

### Comparison table

| Feature | `output()` | `@Output() + EventEmitter` |
|---|---|---|
| Import | `output` from `@angular/core` | `Output`, `EventEmitter` |
| Return type | `OutputEmitterRef<T>` | `EventEmitter<T>` |
| RxJS dependency | None | Full (extends Subject) |
| Type safety of `emit()` | Strict | Loose (allows missing args) |
| `pipe()` support | No (use `outputToObservable`) | Yes (but misleading) |
| `complete()` / `error()` | Not available | Available (usually a bug) |
| Bundle impact | Minimal | Includes RxJS Subject internals |
| Status | Current, recommended | Legacy |

## Migration from @Output to output()

### Automatic Migration Schematic

```bash
ng generate @angular/core:output-migration
```

This schematic:
- Converts `@Output() name = new EventEmitter<T>()` to `name = output<T>()`
- Converts `@Output('alias') name = new EventEmitter<T>()` to `name = output<T>({ alias: 'alias' })`
- Preserves emit calls (the `.emit()` method name is the same)
- Updates imports

### What the schematic does NOT migrate

- Outputs used with `.pipe()` (requires manual conversion to `outputFromObservable` or restructuring)
- Outputs where `.complete()` or `.error()` is called (these patterns must be removed)
- Outputs used with `.subscribe()` in non-standard ways inside the component itself

### Manual migration examples

**Basic output:**

```typescript
// Before
@Output() clicked = new EventEmitter<void>();

// After
clicked = output<void>();
// or simply:
clicked = output();
```

**Output with alias:**

```typescript
// Before
@Output('itemClicked') clicked = new EventEmitter<Item>();

// After
clicked = output<Item>({ alias: 'itemClicked' });
```

**Output with pipe (requires restructuring):**

```typescript
// Before: pipe on EventEmitter (anti-pattern anyway)
@Output() search = new EventEmitter<string>();

ngOnInit() {
  this.search.pipe(debounceTime(300)).subscribe(term => {
    this.doSearch(term);
  });
}

// After: Handle debounce separately
searchSubmitted = output<string>();

private searchSubject = new Subject<string>();

constructor() {
  this.searchSubject.pipe(
    debounceTime(300),
    takeUntilDestroyed()
  ).subscribe(term => {
    this.searchSubmitted.emit(term);
  });
}

onSearch(term: string): void {
  this.searchSubject.next(term);
}
```

## Common Mistakes

### 1. Trying to read an output like a signal

```typescript
// BAD: Outputs are emitters, not signals
@Component({ /* ... */ })
export class BadComponent {
  clicked = output<void>();

  checkIfClicked(): void {
    const value = this.clicked(); // ERROR: not callable as a function
  }
}
```

Outputs do not hold state. They are fire-and-forget emitters. If you need to track whether an event has occurred, maintain a separate signal.

### 2. Naming outputs with the "on" prefix

```typescript
// BAD: Redundant "on" prefix
onSaved = output<Item>();
// In template: (onSaved)="handle($event)" reads as "on on saved"

// GOOD: Past-tense verb
saved = output<Item>();
// In template: (saved)="handle($event)" reads naturally
```

### 3. Emitting in a constructor or field initializer

```typescript
// BAD: No parent is listening yet at construction time
@Component({ /* ... */ })
export class BadComponent {
  ready = output<void>();

  constructor() {
    this.ready.emit(); // No one receives this
  }
}
```

The parent's template binding `(ready)="onReady()"` is not connected until the component is fully initialized and inserted into the view. Emitting in the constructor fires before the parent can listen. Use `afterNextRender` if you need to emit after the component is rendered:

```typescript
@Component({ /* ... */ })
export class GoodComponent {
  ready = output<void>();

  constructor() {
    afterNextRender(() => {
      this.ready.emit();
    });
  }
}
```

### 4. Using EventEmitter's complete() or error()

```typescript
// BAD: Calling complete() kills the output permanently
@Component({ /* ... */ })
export class BadComponent {
  @Output() saved = new EventEmitter<string>();

  save(data: string): void {
    this.saved.emit(data);
    this.saved.complete(); // Output is now dead. Future emit() calls are silently ignored.
  }
}
```

The `output()` function does not expose `complete()` or `error()`, preventing this mistake entirely.

### 5. Emitting complex objects by reference without cloning

```typescript
// CAUTION: Parent receives a reference to the same object
@Component({ /* ... */ })
export class FormComponent {
  formSubmitted = output<FormData>();

  submit(): void {
    this.formSubmitted.emit(this.formState); // Parent gets same reference
    this.resetForm(); // If this mutates formState, parent's reference is affected
  }
}

// SAFER: Emit a copy
submit(): void {
  this.formSubmitted.emit({ ...this.formState });
  this.resetForm();
}
```

### 6. Colliding with native DOM event names

```typescript
// BAD: 'click' collides with native click event
click = output<void>();

// GOOD: Specific name
buttonClicked = output<void>();
```

## Best Practices

### 1. Use output() for all new components

There is no reason to use `@Output()` and `EventEmitter` in new code. The `output()` function is simpler, lighter, and type-safe.

### 2. Keep outputs as events, not commands

Outputs should describe what happened in the child. The parent decides what to do about it. This keeps the child reusable and decoupled.

```typescript
// Child reports an event
itemSelected = output<Item>();

// Parent decides the action
// (itemSelected)="addToCart($event)"
// (itemSelected)="showDetail($event)"
// (itemSelected)="navigateToItem($event)"
```

### 3. Emit the minimum necessary data

Emit only the data the parent needs to respond. Avoid emitting the entire component state.

```typescript
// BAD: Emitting everything
saved = output<{ item: Item; formState: FormState; timestamp: Date; user: User }>();

// GOOD: Emit only what the parent needs
saved = output<Item>();
```

### 4. Use outputFromObservable for observable-based sources

When your event source is naturally an Observable (e.g., a WebSocket message stream or an interval), use `outputFromObservable` rather than manually subscribing and calling `emit`.

```typescript
// Instead of this:
tick = output<number>();
constructor() {
  interval(1000).pipe(takeUntilDestroyed()).subscribe(n => this.tick.emit(n));
}

// Do this:
tick = outputFromObservable(interval(1000));
```

### 5. Handle debouncing and throttling inside the child

The parent should not need to debounce outputs. If an output fires too frequently, throttle it in the child before emitting.

```typescript
@Component({ /* ... */ })
export class SearchInputComponent {
  searchChanged = output<string>();

  private inputSubject = new Subject<string>();

  constructor() {
    this.inputSubject.pipe(
      debounceTime(300),
      distinctUntilChanged(),
      takeUntilDestroyed()
    ).subscribe(term => {
      this.searchChanged.emit(term);
    });
  }

  onInput(event: Event): void {
    this.inputSubject.next((event.target as HTMLInputElement).value);
  }
}
```

### 6. Prefer void outputs for simple notifications

If the parent does not need data, just notification that something happened, use `output()` (void) rather than emitting `true` or `null`.

```typescript
// GOOD
closed = output();          // void: "the modal closed"
refreshRequested = output(); // void: "user wants a refresh"

// BAD
closed = output<boolean>();  // What does true/false mean here?
```

## Real-World Use Cases

### Confirmation Dialog

```typescript
@Component({
  selector: 'app-confirm-dialog',
  standalone: true,
  template: `
    <div class="overlay" (click)="canceled.emit()"></div>
    <div class="dialog" role="dialog">
      <h2>{{ title() }}</h2>
      <p>{{ message() }}</p>
      <div class="actions">
        <button (click)="canceled.emit()">Cancel</button>
        <button class="primary" (click)="confirmed.emit()">
          {{ confirmLabel() }}
        </button>
      </div>
    </div>
  `
})
export class ConfirmDialogComponent {
  title = input('Are you sure?');
  message = input.required<string>();
  confirmLabel = input('Confirm');

  confirmed = output();
  canceled = output();
}
```

```html
<!-- Parent usage -->
@if (showDeleteDialog()) {
  <app-confirm-dialog
    title="Delete Item"
    [message]="'Delete ' + selectedItem().name + '?'"
    confirmLabel="Delete"
    (confirmed)="deleteItem(selectedItem().id); showDeleteDialog.set(false)"
    (canceled)="showDeleteDialog.set(false)"
  />
}
```

### Sortable Table Header

```typescript
interface SortEvent {
  column: string;
  direction: 'asc' | 'desc';
}

@Component({
  selector: 'app-sortable-header',
  standalone: true,
  template: `
    <th (click)="toggleSort()" [class.active]="isActive()">
      <ng-content />
      @if (isActive()) {
        <span class="sort-icon">{{ currentDirection() === 'asc' ? '↑' : '↓' }}</span>
      }
    </th>
  `
})
export class SortableHeaderComponent {
  column = input.required<string>();
  activeSort = input<SortEvent | null>(null);

  sortChanged = output<SortEvent>();

  isActive = computed(() => this.activeSort()?.column === this.column());

  currentDirection = computed(() =>
    this.isActive() ? this.activeSort()!.direction : 'asc'
  );

  toggleSort(): void {
    const newDir = this.isActive() && this.currentDirection() === 'asc' ? 'desc' : 'asc';
    this.sortChanged.emit({ column: this.column(), direction: newDir });
  }
}
```

```html
<!-- Parent: data table -->
<table>
  <thead>
    <tr>
      <app-sortable-header
        column="name"
        [activeSort]="currentSort()"
        (sortChanged)="currentSort.set($event)"
      >
        Name
      </app-sortable-header>
      <app-sortable-header
        column="email"
        [activeSort]="currentSort()"
        (sortChanged)="currentSort.set($event)"
      >
        Email
      </app-sortable-header>
    </tr>
  </thead>
  <!-- rows -->
</table>
```

### File Upload with Progress

```typescript
interface UploadProgress {
  file: string;
  percent: number;
}

@Component({
  selector: 'app-file-upload',
  standalone: true,
  template: `
    <input
      type="file"
      [multiple]="multiple()"
      [accept]="accept()"
      (change)="onFilesSelected($event)"
    />

    @for (upload of activeUploads(); track upload.file) {
      <div class="upload-row">
        <span>{{ upload.file }}</span>
        <progress [value]="upload.percent" max="100"></progress>
        <span>{{ upload.percent }}%</span>
      </div>
    }
  `
})
export class FileUploadComponent {
  private http = inject(HttpClient);

  accept = input('.pdf,.jpg,.png');
  multiple = input(false, { transform: booleanAttribute });
  uploadUrl = input.required<string>();

  uploadStarted = output<string[]>();
  uploadProgress = output<UploadProgress>();
  uploadCompleted = output<string>();
  uploadFailed = output<{ file: string; error: string }>();

  activeUploads = signal<UploadProgress[]>([]);

  onFilesSelected(event: Event): void {
    const files = Array.from((event.target as HTMLInputElement).files ?? []);
    if (files.length === 0) return;

    this.uploadStarted.emit(files.map(f => f.name));

    for (const file of files) {
      this.uploadFile(file);
    }
  }

  private uploadFile(file: File): void {
    const formData = new FormData();
    formData.append('file', file);

    this.activeUploads.update(list => [...list, { file: file.name, percent: 0 }]);

    this.http.post(this.uploadUrl(), formData, {
      reportProgress: true,
      observe: 'events',
    }).subscribe({
      next: (event) => {
        if (event.type === HttpEventType.UploadProgress && event.total) {
          const percent = Math.round((event.loaded / event.total) * 100);
          this.activeUploads.update(list =>
            list.map(u => u.file === file.name ? { ...u, percent } : u)
          );
          this.uploadProgress.emit({ file: file.name, percent });
        }
        if (event.type === HttpEventType.Response) {
          this.activeUploads.update(list => list.filter(u => u.file !== file.name));
          this.uploadCompleted.emit(file.name);
        }
      },
      error: (err) => {
        this.activeUploads.update(list => list.filter(u => u.file !== file.name));
        this.uploadFailed.emit({ file: file.name, error: err.message });
      },
    });
  }
}
```

```html
<!-- Parent usage -->
<app-file-upload
  uploadUrl="/api/upload"
  multiple
  accept=".pdf,.docx"
  (uploadStarted)="onUploadStart($event)"
  (uploadCompleted)="onFileUploaded($event)"
  (uploadFailed)="onUploadError($event)"
/>
```

---

> **Key Takeaway:** The `output()` function creates lightweight, type-safe emitters (`OutputEmitterRef`) that replace `@Output()` and `EventEmitter` with zero RxJS overhead, strict typing on `emit()`, and interop functions (`outputFromObservable`, `outputToObservable`) for bridging with observables when needed.
