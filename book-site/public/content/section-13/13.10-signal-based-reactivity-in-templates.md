# 13.10 Signal-Based Reactivity in Templates

> Signals give Angular templates fine-grained reactivity: only the parts of the view that depend on changed state get re-evaluated, replacing zone-based "check everything" with targeted updates.

---

## How Signals Interact with Templates

When you read a signal inside a component template, Angular registers that read as a dependency. When the signal's value changes, Angular marks that specific component for re-rendering. This is the foundation of signal-based reactivity in templates.

```typescript
import { Component, signal, computed } from '@angular/core';

@Component({
  selector: 'app-counter',
  standalone: true,
  template: `
    <p>Count: {{ count() }}</p>
    <p>Double: {{ doubleCount() }}</p>
    <button (click)="increment()">+1</button>
  `,
})
export class CounterComponent {
  count = signal(0);
  doubleCount = computed(() => this.count() * 2);

  increment() {
    this.count.update(n => n + 1);
  }
}
```

When `count` changes:
1. Angular knows the template reads `count()` and `doubleCount()`.
2. `doubleCount` recalculates because its dependency (`count`) changed.
3. Angular marks the component dirty and re-renders only the affected bindings.

### The Function Call Syntax

Every signal read in a template uses parentheses: `{{ count() }}`, `[value]="count()"`, `(click)="handler(count())"`. This is not a method call in the traditional sense. It is a signal read that establishes a reactive dependency.

```html
<!-- Each () is a signal read that creates a dependency -->
<input [value]="searchTerm()" (input)="searchTerm.set($event.target.value)" />
<p>Results for: {{ searchTerm() }}</p>
<ul>
  @for (item of filteredItems(); track item.id) {
    <li>{{ item.name }}</li>
  }
</ul>
```

## Signals and Change Detection

### The Zone.js Model (Legacy)

In traditional Angular, Zone.js patches all async APIs (setTimeout, Promise, event listeners). After any async operation, Angular runs change detection from the root component downward, checking every binding in every component. This is wasteful when most components have not changed.

### The Signal Model

With signals, Angular knows exactly which components depend on which state. When a signal changes, only components that read that signal are marked dirty. The rest of the tree is skipped.

```typescript
@Component({
  selector: 'app-parent',
  standalone: true,
  imports: [ChildAComponent, ChildBComponent],
  template: `
    <app-child-a [name]="name()" />
    <app-child-b [count]="count()" />
    <button (click)="count.update(n => n + 1)">Increment</button>
  `,
})
export class ParentComponent {
  name = signal('Alice');
  count = signal(0);
}
```

When `count` changes, `ChildBComponent` is marked dirty. `ChildAComponent` is NOT marked dirty because it reads `name`, which did not change. In the Zone.js model, both children would be checked.

### OnPush and Signals

Signals work with `OnPush` change detection. When a signal read in an `OnPush` component's template changes, Angular automatically marks the component dirty. You never need to call `markForCheck()`.

```typescript
@Component({
  selector: 'app-status',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<span class="badge">{{ status() }}</span>`,
})
export class StatusComponent {
  status = input.required<string>();
  // Signal input: when the parent passes a new value,
  // this component is automatically marked dirty.
  // No markForCheck() needed.
}
```

### Zoneless Angular

Signals are the path to zoneless Angular. In a zoneless application, there is no Zone.js patching. Change detection runs only when signals change or when you explicitly call `ChangeDetectorRef.markForCheck()`.

```typescript
// app.config.ts
import { provideZonelessChangeDetection } from '@angular/core';

export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
  ],
};
```

In zoneless mode, signals are not optional. They are the primary mechanism for telling Angular something changed. Plain property assignments will NOT trigger re-renders:

```typescript
// Bad Practice in zoneless: plain property assignment
@Component({
  template: `<p>{{ message }}</p>`,
})
export class BrokenComponent {
  message = 'Hello';

  update() {
    this.message = 'World'; // Template will NOT update in zoneless mode
  }
}

// Correct: use a signal
@Component({
  template: `<p>{{ message() }}</p>`,
})
export class WorkingComponent {
  message = signal('Hello');

  update() {
    this.message.set('World'); // Template updates correctly
  }
}
```

## `computed()` in Templates

Computed signals derive state from other signals. They are lazy (only calculated when read) and memoized (cached until a dependency changes).

### Using Computed for Template Logic

Move complex logic out of template expressions and into computed signals:

```typescript
// Bad Practice: complex logic in the template
@Component({
  template: `
    <p>{{ items().filter(i => i.active).length }} active items</p>
    <p>Total: {{ items().reduce((sum, i) => sum + i.price, 0) | currency }}</p>
  `,
})
export class BadComponent {
  items = signal<Item[]>([]);
}
```

Every change detection cycle re-evaluates both `filter()` and `reduce()`. With computed signals, the calculation only runs when `items` actually changes:

```typescript
// Correct: derived state in computed signals
@Component({
  template: `
    <p>{{ activeCount() }} active items</p>
    <p>Total: {{ totalPrice() | currency }}</p>
  `,
})
export class GoodComponent {
  items = signal<Item[]>([]);

  activeCount = computed(() => this.items().filter(i => i.active).length);
  totalPrice = computed(() => this.items().reduce((sum, i) => sum + i.price, 0));
}
```

### Computed Dependency Tracking Is Dynamic

Only signals that are actually read during the computation are tracked. If your computed has conditional logic, the dependencies change based on which branch executes.

```typescript
showDetails = signal(false);
user = signal<User | null>(null);
orders = signal<Order[]>([]);

// When showDetails() is false, orders() is NOT a dependency.
// When showDetails() is true, orders() IS a dependency.
summary = computed(() => {
  const u = this.user();
  if (!u) return 'No user selected';

  if (this.showDetails()) {
    return `${u.name}: ${this.orders().length} orders`;
  }

  return u.name;
});
```

### Computed Signals Must Be Pure

Do not put side effects inside `computed()`. No DOM manipulation, no HTTP calls, no console.log for debugging purposes.

```typescript
// Bad Practice: side effects in computed
fullName = computed(() => {
  console.log('Recalculating...'); // side effect
  this.analytics.track('name-read'); // side effect
  return `${this.firstName()} ${this.lastName()}`;
});

// Correct: pure derivation
fullName = computed(() => `${this.firstName()} ${this.lastName()}`);
```

## `@let` Template Variables

Angular's `@let` syntax (stable since Angular 18.1) declares local variables within templates. Combined with signals, it eliminates repeated signal reads and simplifies complex template expressions.

### Basic Syntax

```html
@let name = user().name;
@let greeting = 'Hello, ' + name;

<h1>{{ greeting }}</h1>
<p>Welcome back, {{ name }}.</p>
```

Each `@let` declares exactly one variable, terminated by a semicolon. The expression is re-evaluated whenever its dependencies change.

### Reducing Repeated Signal Reads

Without `@let`, you might call a signal multiple times:

```html
<!-- Without @let: user() called three times -->
<h2>{{ user().name }}</h2>
<p>{{ user().email }}</p>
<span>Joined: {{ user().joinDate | date }}</span>
```

With `@let`, read once:

```html
@let u = user();

<h2>{{ u.name }}</h2>
<p>{{ u.email }}</p>
<span>Joined: {{ u.joinDate | date }}</span>
```

### Type Narrowing with `@let`

`@let` shines when combined with signals that may be `null` or `undefined`:

```html
@let data = asyncData();

@if (data) {
  <!-- data is narrowed to non-null here -->
  <app-chart [values]="data.values" [labels]="data.labels" />
} @else {
  <app-skeleton />
}
```

### Using `@let` with Pipes

Pipe a value once and reuse the result:

```html
@let formattedDate = order().createdAt | date:'longDate';

<h3>Order placed on {{ formattedDate }}</h3>
<p>Confirmation for {{ formattedDate }} has been sent.</p>
```

Without `@let`, the pipe would execute twice.

### Scoping Rules

`@let` variables are scoped to the current view and its descendants. A variable declared inside an `@if` block is not accessible outside it.

```html
@if (showDetails()) {
  @let detail = computedDetail();
  <p>{{ detail }}</p>
}

<!-- ERROR: detail is not accessible here -->
<!-- <p>{{ detail }}</p> -->
```

Variables declared at the top level of the template are accessible everywhere within that template.

### `@let` Cannot Be Reassigned

Unlike JavaScript's `let`, template `@let` variables cannot be reassigned. Their value is controlled entirely by the expression on the right side.

```html
<!-- This is NOT valid -->
<!-- @let count = 0; -->
<!-- (click)="count = count + 1" -->

<!-- Use a signal instead for mutable state -->
```

## Signal Inputs in Templates

Signal inputs (`input()`, `input.required()`) are signals. Reading them in the template creates reactive dependencies just like any other signal.

```typescript
@Component({
  selector: 'app-user-card',
  standalone: true,
  template: `
    @let u = user();

    <div class="card" [class.vip]="u.isVip">
      <h3>{{ u.name }}</h3>
      <p>{{ u.email }}</p>
      @if (showBadge()) {
        <span class="badge">{{ u.role }}</span>
      }
    </div>
  `,
})
export class UserCardComponent {
  user = input.required<User>();
  showBadge = input(true);
}
```

When the parent passes a new `user` object, Angular detects the signal change and re-renders only this component. If only `showBadge` changes, the `user` bindings are not re-evaluated (they read the same cached signal value).

---

## Model Inputs in Templates

Model inputs (`model()`) are writable signals that enable two-way binding. In templates, they read like signals and write like signals.

```typescript
@Component({
  selector: 'app-slider',
  standalone: true,
  template: `
    <input
      type="range"
      [min]="min()"
      [max]="max()"
      [value]="value()"
      (input)="value.set(+$event.target.value)"
    />
    <span>{{ value() }}</span>
  `,
})
export class SliderComponent {
  min = input(0);
  max = input(100);
  value = model(50);
}
```

```html
<!-- Parent: two-way binding with banana-in-a-box -->
<app-slider [(value)]="brightness" />
```

When the user drags the slider, `value.set()` updates the model signal. The parent's `brightness` signal updates automatically through two-way binding.

## Signal Queries in Templates

Signal-based view and content queries return signals, making queried elements reactive.

```typescript
@Component({
  selector: 'app-tabs',
  standalone: true,
  template: `
    <div class="tab-headers">
      @for (tab of tabPanels(); track tab) {
        <button
          [class.active]="tab === activeTab()"
          (click)="activeTab.set(tab)"
        >
          {{ tab.label }}
        </button>
      }
    </div>

    <div class="tab-content">
      <ng-content />
    </div>
  `,
})
export class TabsComponent {
  tabPanels = contentChildren(TabPanelComponent);
  activeTab = signal<TabPanelComponent | null>(null);

  constructor() {
    // Set first tab as active when tabs are projected
    effect(() => {
      const panels = this.tabPanels();
      if (panels.length > 0 && !this.activeTab()) {
        this.activeTab.set(panels[0]);
      }
    });
  }
}
```

`contentChildren()` returns a `Signal<ReadonlyArray<TabPanelComponent>>`. When tabs are added or removed, the signal updates, the template re-renders the tab headers, and the effect adjusts the active tab.

## `toSignal()` for Observable-to-Signal Conversion

When working with observables (HTTP calls, router params, form value changes), convert them to signals for template consumption.

```typescript
import { toSignal } from '@angular/core/rxjs-interop';

@Component({
  selector: 'app-product-list',
  standalone: true,
  template: `
    @let products = productList();

    @if (products.length === 0) {
      <p>No products found.</p>
    } @else {
      @for (product of products; track product.id) {
        <app-product-card [product]="product" />
      }
    }
  `,
})
export class ProductListComponent {
  private http = inject(HttpClient);
  private category = input.required<string>();

  productList = toSignal(
    toObservable(this.category).pipe(
      switchMap(cat => this.http.get<Product[]>(`/api/products?category=${cat}`))
    ),
    { initialValue: [] }
  );
}
```

`toSignal` subscribes to the observable and exposes the latest emitted value as a signal. The template reads `productList()` just like any other signal.

### `toSignal` Options

| Option | Type | Description |
|---|---|---|
| `initialValue` | `T` | Value before the first emission. Makes the signal type `Signal<T>` instead of `Signal<T \| undefined>`. |
| `requireSync` | `boolean` | Asserts the observable emits synchronously (e.g., `BehaviorSubject`). Throws if it does not. |
| `manualCleanup` | `boolean` | Prevents automatic unsubscription on component destroy. Rarely needed. |

## `resource()` and `rxResource()` in Templates

For async data loading, `resource()` (promise-based) and `rxResource()` (observable-based) provide signal-based state management with loading, error, and value states.

```typescript
import { resource } from '@angular/core';

@Component({
  selector: 'app-user-profile',
  standalone: true,
  template: `
    @if (userResource.isLoading()) {
      <app-skeleton />
    } @else if (userResource.error()) {
      <p class="error">Failed to load user.</p>
    } @else {
      @let user = userResource.value();
      @if (user) {
        <h2>{{ user.name }}</h2>
        <p>{{ user.email }}</p>
      }
    }
  `,
})
export class UserProfileComponent {
  userId = input.required<string>();

  userResource = resource({
    request: () => this.userId(),
    loader: async ({ request: id }) => {
      const response = await fetch(`/api/users/${id}`);
      return response.json() as Promise<User>;
    },
  });
}
```

The resource automatically re-fetches when `userId` changes. The template reactively displays loading, error, or data states through signal reads.

## Combining Signals for Complex UIs

Real templates often combine multiple signal types. Here is a pattern for a filterable, sortable list:

```typescript
@Component({
  selector: 'app-task-board',
  standalone: true,
  template: `
    <div class="controls">
      <input
        [value]="searchTerm()"
        (input)="searchTerm.set($event.target.value)"
        placeholder="Search tasks..."
      />

      <select [value]="sortBy()" (change)="sortBy.set($event.target.value)">
        <option value="name">Name</option>
        <option value="date">Date</option>
        <option value="priority">Priority</option>
      </select>

      <label>
        <input
          type="checkbox"
          [checked]="showCompleted()"
          (change)="showCompleted.set($event.target.checked)"
        />
        Show completed
      </label>
    </div>

    <p class="summary">
      Showing {{ visibleTasks().length }} of {{ allTasks().length }} tasks
    </p>

    @for (task of visibleTasks(); track task.id) {
      <app-task-card
        [task]="task"
        (statusChange)="updateTaskStatus(task.id, $event)"
      />
    } @empty {
      <p>No tasks match your filters.</p>
    }
  `,
})
export class TaskBoardComponent {
  // Source data
  allTasks = signal<Task[]>([]);

  // UI state
  searchTerm = signal('');
  sortBy = signal('date');
  showCompleted = signal(false);

  // Derived state: filter, then sort
  filteredTasks = computed(() => {
    const tasks = this.allTasks();
    const term = this.searchTerm().toLowerCase();
    const includeCompleted = this.showCompleted();

    return tasks.filter(t => {
      if (!includeCompleted && t.status === 'completed') return false;
      if (term && !t.name.toLowerCase().includes(term)) return false;
      return true;
    });
  });

  visibleTasks = computed(() => {
    const tasks = this.filteredTasks();
    const sort = this.sortBy();

    return [...tasks].sort((a, b) => {
      if (sort === 'name') return a.name.localeCompare(b.name);
      if (sort === 'date') return b.createdAt.getTime() - a.createdAt.getTime();
      if (sort === 'priority') return b.priority - a.priority;
      return 0;
    });
  });

  updateTaskStatus(id: string, status: string) {
    this.allTasks.update(tasks =>
      tasks.map(t => t.id === id ? { ...t, status } : t)
    );
  }
}
```

The dependency graph:

```
allTasks ──────────┐
searchTerm ────────┤
showCompleted ─────┼──> filteredTasks ──> visibleTasks
sortBy ────────────────────────────────────────┘
```

When `searchTerm` changes, `filteredTasks` recalculates, then `visibleTasks` recalculates. When `sortBy` changes, only `visibleTasks` recalculates (because `filteredTasks` does not depend on `sortBy`). Computed signals handle this granularity automatically.

## `effect()` and Templates

Effects run in response to signal changes but should NOT be used to update template state. Effects are for side effects: logging, analytics, DOM APIs, localStorage, etc.

```typescript
// Correct: effect for a side effect (saving to localStorage)
searchTerm = signal('');

constructor() {
  effect(() => {
    localStorage.setItem('lastSearch', this.searchTerm());
  });
}
```

```typescript
// Bad Practice: using effect to derive state
count = signal(0);
doubleCount = signal(0); // should be computed

constructor() {
  effect(() => {
    this.doubleCount.set(this.count() * 2); // WRONG: use computed() instead
  });
}

// Correct
count = signal(0);
doubleCount = computed(() => this.count() * 2);
```

Setting signals inside `effect()` is allowed but requires the `allowSignalWrites` flag (removed in Angular 19+). Angular discourages this pattern because it can create circular dependencies and is harder to reason about.

## `untracked()` for Selective Dependency Tracking

Sometimes you need to read a signal inside a reactive context without creating a dependency. Use `untracked()`.

```typescript
import { untracked } from '@angular/core';

searchTerm = signal('');
pageSize = signal(20);

constructor() {
  // This effect should only re-run when searchTerm changes,
  // not when pageSize changes.
  effect(() => {
    const term = this.searchTerm();
    const size = untracked(() => this.pageSize());
    this.analyticsService.trackSearch(term, size);
  });
}
```

`untracked()` reads the signal value but does not register it as a dependency of the surrounding reactive context.

## Signal Equality and Template Updates

By default, signals use `Object.is()` for equality checks. Setting the same primitive value twice does not trigger a re-render:

```typescript
name = signal('Alice');
name.set('Alice'); // No re-render: Object.is('Alice', 'Alice') is true
```

For objects and arrays, `Object.is()` compares references. A new object with identical content triggers a re-render:

```typescript
config = signal({ theme: 'dark' });
config.set({ theme: 'dark' }); // Re-renders: different object reference
```

### Custom Equality

Provide a custom equality function to prevent unnecessary re-renders for objects:

```typescript
import _ from 'lodash';

config = signal({ theme: 'dark', fontSize: 14 }, {
  equal: _.isEqual, // deep comparison
});

config.set({ theme: 'dark', fontSize: 14 }); // No re-render: deeply equal
```

For arrays of items, consider checking length and IDs rather than deep equality:

```typescript
items = signal<Item[]>([], {
  equal: (a, b) => {
    if (a.length !== b.length) return false;
    return a.every((item, i) => item.id === b[i].id && item.updatedAt === b[i].updatedAt);
  },
});
```

## Common Mistakes

- **Using plain properties instead of signals in zoneless mode.** Without Zone.js, Angular has no way to detect that a plain property changed. The template will not update. Use `signal()` for any mutable state that appears in the template.

- **Putting side effects in `computed()`.** Computed signals must be pure. They can be called multiple times or not at all depending on memoization. HTTP calls, console.log, and DOM manipulation inside `computed()` lead to unpredictable behavior.

- **Calling signal reads after `await` in an `effect()`.** The reactive context is synchronous. Any signal reads after an `await` boundary are NOT tracked as dependencies:

```typescript
// Bad Practice: signal read after await
effect(async () => {
  const data = await fetchData();
  const format = this.displayFormat(); // NOT tracked as a dependency
  this.render(data, format);
});

// Correct: read signals before await
effect(async () => {
  const format = this.displayFormat(); // tracked
  const data = await fetchData();
  this.render(data, format);
});
```

- **Using `effect()` to set derived state instead of `computed()`.** If one signal's value is always a function of another signal's value, use `computed()`. Effects are for side effects, not state derivation.

- **Overusing signals for non-reactive state.** Constants, configuration that never changes, and internal flags that do not affect the template do not need to be signals. Use plain properties for truly static values.

- **Creating new array/object references on every update without checking for actual changes.** Each new reference triggers re-computation in all dependent computed signals and effects.

---

## Best Practices

- **Use `computed()` for every piece of derived state.** If a value can be calculated from other signals, it should be a computed signal, not a manually synchronized writable signal.

- **Use `@let` to reduce repeated signal reads.** If you read `user()` five times in a template, declare `@let u = user();` once and use `u` throughout.

- **Keep effects minimal and focused.** Each effect should do one thing: save to localStorage, track analytics, or sync with a non-Angular API. Do not use effects as a substitute for computed signals.

- **Provide custom equality for frequently updated object/array signals.** This prevents unnecessary re-renders when the actual content has not changed.

- **Convert observables to signals at the boundary.** Use `toSignal()` where the observable enters the component. Then work exclusively with signals throughout the template and component logic.

- **Design your signal graph like a spreadsheet.** Source signals (cells you type into) flow through computed signals (formulas) to the template (display). Data flows in one direction, and every derived value is automatically consistent.

## Real-World Use Cases

### Shopping Cart with Reactive Totals

```typescript
@Component({
  selector: 'app-cart',
  standalone: true,
  template: `
    @for (item of cartItems(); track item.productId) {
      <div class="cart-item">
        <span>{{ item.name }}</span>
        <input
          type="number"
          [value]="item.quantity"
          (change)="updateQuantity(item.productId, +$event.target.value)"
          min="1"
        />
        <span>{{ item.price * item.quantity | currency }}</span>
        <button (click)="removeItem(item.productId)">Remove</button>
      </div>
    } @empty {
      <p>Your cart is empty.</p>
    }

    @if (cartItems().length > 0) {
      <div class="cart-summary">
        <p>Subtotal: {{ subtotal() | currency }}</p>
        <p>Tax ({{ taxRate() * 100 }}%): {{ tax() | currency }}</p>
        @if (discount()) {
          <p class="discount">Discount: -{{ discount() | currency }}</p>
        }
        <p class="total">Total: {{ total() | currency }}</p>
      </div>
    }
  `,
})
export class CartComponent {
  cartItems = signal<CartItem[]>([]);
  taxRate = signal(0.08);
  couponDiscount = signal(0);

  subtotal = computed(() =>
    this.cartItems().reduce((sum, item) => sum + item.price * item.quantity, 0)
  );

  discount = computed(() => {
    const sub = this.subtotal();
    const coupon = this.couponDiscount();
    return Math.min(coupon, sub); // discount cannot exceed subtotal
  });

  tax = computed(() => (this.subtotal() - this.discount()) * this.taxRate());

  total = computed(() => this.subtotal() - this.discount() + this.tax());

  updateQuantity(productId: string, quantity: number) {
    this.cartItems.update(items =>
      items.map(item =>
        item.productId === productId ? { ...item, quantity } : item
      )
    );
  }

  removeItem(productId: string) {
    this.cartItems.update(items =>
      items.filter(item => item.productId !== productId)
    );
  }
}
```

The entire pricing chain (subtotal, discount, tax, total) automatically recalculates when any cart item changes. No manual wiring needed.

### Real-Time Search with Debounce

```typescript
@Component({
  selector: 'app-search',
  standalone: true,
  template: `
    <input
      [value]="rawInput()"
      (input)="rawInput.set($event.target.value)"
      placeholder="Search..."
    />

    @if (searchResults.isLoading()) {
      <div class="spinner">Searching...</div>
    } @else if (searchResults.error()) {
      <p class="error">Search failed. Try again.</p>
    } @else {
      @let results = searchResults.value() ?? [];
      <p>{{ results.length }} results for "{{ debouncedTerm() }}"</p>
      @for (result of results; track result.id) {
        <app-search-result [item]="result" />
      } @empty {
        <p>No results found.</p>
      }
    }
  `,
})
export class SearchComponent {
  private http = inject(HttpClient);

  rawInput = signal('');

  // Debounce using toSignal + RxJS
  debouncedTerm = toSignal(
    toObservable(this.rawInput).pipe(
      debounceTime(300),
      distinctUntilChanged(),
      filter(term => term.length >= 2),
    ),
    { initialValue: '' }
  );

  searchResults = rxResource({
    request: () => this.debouncedTerm(),
    loader: ({ request: term }) => {
      if (!term) return of([]);
      return this.http.get<SearchResult[]>(`/api/search?q=${encodeURIComponent(term)}`);
    },
  });
}
```

Signal graph: `rawInput` -> `debouncedTerm` (via RxJS debounce) -> `searchResults` (via rxResource). The template reactively displays loading, error, and result states.

### Theme-Aware Component with localStorage Persistence

```typescript
@Component({
  selector: 'app-theme-toggle',
  standalone: true,
  template: `
    <button (click)="toggleTheme()">
      {{ isDark() ? 'Light Mode' : 'Dark Mode' }}
    </button>
  `,
})
export class ThemeToggleComponent {
  private themeService = inject(ThemeService);
  isDark = this.themeService.isDark;

  toggleTheme() {
    this.themeService.toggle();
  }
}

// theme.service.ts
@Injectable({ providedIn: 'root' })
export class ThemeService {
  private theme = signal<'light' | 'dark'>(
    (localStorage.getItem('theme') as 'light' | 'dark') ?? 'light'
  );

  isDark = computed(() => this.theme() === 'dark');

  constructor() {
    // Side effect: persist theme changes to localStorage and update DOM
    effect(() => {
      const current = this.theme();
      localStorage.setItem('theme', current);
      document.documentElement.setAttribute('data-theme', current);
    });
  }

  toggle() {
    this.theme.update(t => t === 'light' ? 'dark' : 'light');
  }
}
```

The service holds the source signal. `isDark` is a computed consumed by any component. The `effect` handles side effects (localStorage and DOM attribute). No component needs to know about localStorage.

---

> **Key Takeaway:** Read signals in templates to create reactive dependencies, derive state with `computed()` instead of complex template expressions, use `@let` to reduce repeated reads, and reserve `effect()` strictly for side effects. This signal-first approach enables fine-grained reactivity that works with OnPush today and zoneless Angular tomorrow.
