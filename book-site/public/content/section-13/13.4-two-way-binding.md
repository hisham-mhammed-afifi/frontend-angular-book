# 13.4 Two-Way Binding

> Two-way binding synchronizes a value between a parent and child component (or between a component and a form element) so that changes in either direction are reflected immediately. Angular provides the `[()]` "banana-in-a-box" syntax to express this pattern concisely.

---

## The Banana-in-a-Box Syntax

Two-way binding combines property binding (`[ ]`) and event binding (`( )`) into a single notation: `[( )]`.

```html
<!-- Two-way binding -->
<app-slider [(value)]="volume" />

<!-- Desugars to: -->
<app-slider [value]="volume" (valueChange)="volume = $event" />
```

The name comes from the visual appearance: square brackets (the box) surrounding parentheses (the banana). Data flows in to the child through the property binding, and changes flow back out through the event binding.

---

## How Desugaring Works

Every `[(prop)]="expr"` is syntactic sugar for `[prop]="expr" (propChange)="expr = $event"`. Angular enforces a naming convention: the event name must be the property name with a `Change` suffix.

```html
<!-- These are identical -->
<app-color-picker [(color)]="selectedColor" />
<app-color-picker [color]="selectedColor" (colorChange)="selectedColor = $event" />

<!-- These are also identical -->
<app-toggle [(checked)]="isEnabled" />
<app-toggle [checked]="isEnabled" (checkedChange)="isEnabled = $event" />
```

With signals, the desugared form uses `.set()`:

```html
<!-- With a signal -->
<app-slider [(value)]="volume" />

<!-- Desugars to: -->
<app-slider [value]="volume()" (valueChange)="volume.set($event)" />
```

Angular handles this automatically. When `volume` is a `WritableSignal`, `[(value)]="volume"` reads from `volume()` for the property binding and calls `volume.set($event)` for the event binding.

---

## The `model()` API (Signal-Based Two-Way Binding)

The modern approach to two-way binding in Angular uses `model()` in the child component. Introduced in Angular 17.2 and stable since Angular 18, `model()` creates a `ModelSignal` that is writable from both the parent and the child.

### Basic Usage

```typescript
// Child component
@Component({
  selector: 'app-toggle',
  standalone: true,
  template: `
    <button
      class="toggle"
      [class.on]="checked()"
      (click)="toggle()"
      role="switch"
      [attr.aria-checked]="checked()"
    >
      {{ checked() ? 'ON' : 'OFF' }}
    </button>
  `,
  styles: [`
    .toggle { padding: 8px 16px; border: 2px solid #ccc; border-radius: 20px; cursor: pointer; }
    .toggle.on { background: #4caf50; color: white; border-color: #4caf50; }
  `]
})
export class ToggleComponent {
  // model() creates a two-way bindable signal
  checked = model(false);

  toggle() {
    this.checked.update(v => !v);
    // The parent's bound value is automatically updated
  }
}
```

```html
<!-- Parent template -->
<app-toggle [(checked)]="darkMode" />
<p>Dark mode is {{ darkMode() ? 'enabled' : 'disabled' }}</p>
```

```typescript
// Parent component
export class SettingsComponent {
  darkMode = signal(false);
}
```

When the child calls `this.checked.update(v => !v)`, the parent's `darkMode` signal is automatically updated because they share the same writable signal.

### Required Model

```typescript
export class SliderComponent {
  // Required: parent MUST provide a two-way binding
  value = model.required<number>();

  min = input(0);
  max = input(100);
  step = input(1);
}
```

```html
<!-- Parent must provide [(value)] -->
<app-slider [(value)]="volume" [min]="0" [max]="100" [step]="5" />
```

### Model with Default

```typescript
export class SearchFieldComponent {
  // Optional with a default value
  query = model('');
  placeholder = input('Search...');
}
```

```html
<!-- Two-way binding (full sync) -->
<app-search-field [(query)]="searchTerm" />

<!-- One-way binding (parent pushes, child changes don't propagate back) -->
<app-search-field [query]="initialTerm" />

<!-- No binding (uses default value) -->
<app-search-field />
```

### The ModelSignal API

`model()` returns a `ModelSignal<T>`, which extends `WritableSignal<T>`. You can use all signal operations:

```typescript
export class CounterComponent {
  count = model(0);

  increment() {
    this.count.update(c => c + 1);
  }

  decrement() {
    this.count.update(c => c - 1);
  }

  reset() {
    this.count.set(0);
  }

  // Use as a readonly signal in computed
  doubled = computed(() => this.count() * 2);

  // Use in effects
  constructor() {
    effect(() => {
      console.log('Count changed to:', this.count());
    });
  }
}
```

## Two-Way Binding with `ngModel` (Form Elements)

For native HTML form elements, Angular provides the `ngModel` directive from `FormsModule`. This is the standard way to two-way bind form inputs.

### Setup

```typescript
import { FormsModule } from '@angular/forms';

@Component({
  selector: 'app-profile-form',
  standalone: true,
  imports: [FormsModule], // Required for ngModel
  template: `
    <form>
      <input [(ngModel)]="name" name="name" />
      <input [(ngModel)]="email" name="email" type="email" />
      <textarea [(ngModel)]="bio" name="bio"></textarea>
      <select [(ngModel)]="role" name="role">
        <option value="admin">Admin</option>
        <option value="editor">Editor</option>
        <option value="viewer">Viewer</option>
      </select>
      <input type="checkbox" [(ngModel)]="isActive" name="active" />
    </form>

    <pre>
      Name: {{ name() }}
      Email: {{ email() }}
      Bio: {{ bio() }}
      Role: {{ role() }}
      Active: {{ isActive() }}
    </pre>
  `
})
export class ProfileFormComponent {
  name = signal('');
  email = signal('');
  bio = signal('');
  role = signal('viewer');
  isActive = signal(true);
}
```

### FormsModule is Required

Without importing `FormsModule`, you get the error:

```
Can't bind to 'ngModel' since it isn't a known property of 'input'.
```

For standalone components, import `FormsModule` directly in the component's `imports` array. For module-based components, import it in the NgModule.

### How `ngModel` Works

`ngModel` is a directive that:
1. Reads the bound value and sets the element's `value` property.
2. Listens for `input` (or `change`) events on the element.
3. When the user types, updates the bound value with the new input.

It handles the differences between element types:
- `<input type="text">`: binds to `value`, listens to `input`.
- `<input type="checkbox">`: binds to `checked`, listens to `change`.
- `<input type="radio">`: binds to `checked`, listens to `change`.
- `<select>`: binds to `value`, listens to `change`.
- `<textarea>`: binds to `value`, listens to `input`.

### `ngModel` with Signals

Since Angular 17.1, `ngModel` can bind directly to signals:

```html
<!-- signal-based: Angular reads the signal and calls .set() on changes -->
<input [(ngModel)]="username" name="username" />
```

```typescript
username = signal(''); // WritableSignal works with ngModel
```

The directive detects that `username` is a `WritableSignal` and uses `.set()` to update it. You do not need to call `.set()` manually.

## Building Custom Two-Way Binding (Legacy Pattern)

Before `model()`, the convention was to pair an `@Input()` with an `@Output()` that follows the `Change` suffix naming convention.

### The `input()` + `output()` Pattern

```typescript
@Component({
  selector: 'app-rating',
  standalone: true,
  template: `
    <div class="stars">
      @for (star of starsArray; track star) {
        <button
          [class.filled]="star <= value()"
          (click)="rate(star)"
        >
          ★
        </button>
      }
    </div>
  `
})
export class RatingComponent {
  value = input(0);
  valueChange = output<number>();

  starsArray = [1, 2, 3, 4, 5];

  rate(star: number) {
    this.valueChange.emit(star);
  }
}
```

```html
<!-- Parent can use banana-in-a-box because value + valueChange convention is met -->
<app-rating [(value)]="userRating" />
```

This works because Angular sees `[(value)]` and desugars it to `[value]="userRating" (valueChange)="userRating.set($event)"`.

### When to Use `input()` + `output()` vs `model()`

| Feature | `model()` | `input()` + `output()` |
|---|---|---|
| Signal type | `ModelSignal` (writable) | `InputSignal` (read-only) + `OutputEmitterRef` |
| Child can write | Yes, directly via `.set()`/`.update()` | No, must call `output.emit()` |
| Boilerplate | One declaration | Two declarations |
| Parent notification | Automatic when signal changes | Only when `emit()` is explicitly called |
| Custom emit logic | Less control (any `.set()` propagates) | Full control (emit only when you choose) |
| Reactivity | Full signal reactivity | Input is reactive, output is imperative |

**Use `model()`** when the child owns the state and both sides should be able to read and write freely (sliders, toggles, pickers).

**Use `input()` + `output()`** when you need control over when changes propagate (validation before emitting, transforming the value, batching changes).

## Two-Way Binding Without `ngModel` (Manual Pattern)

You can implement two-way binding on native elements without `ngModel` by combining property and event bindings manually:

```html
<!-- Without ngModel -->
<input
  [value]="searchTerm()"
  (input)="searchTerm.set($any($event.target).value)"
/>

<!-- With ngModel (cleaner) -->
<input [(ngModel)]="searchTerm" name="search" />
```

The manual pattern is useful when:
- You do not want to import `FormsModule`.
- You need to intercept or transform the value before setting it.
- You want to respond to a different event than the default.

```typescript
@Component({
  selector: 'app-debounced-input',
  standalone: true,
  template: `
    <input
      [value]="displayValue()"
      (input)="onInput($event)"
      (keydown.enter)="onCommit()"
      (blur)="onCommit()"
    />
  `
})
export class DebouncedInputComponent {
  value = model('');

  private pendingValue = signal('');
  displayValue = computed(() => this.pendingValue() || this.value());

  onInput(event: Event) {
    const val = (event.target as HTMLInputElement).value;
    this.pendingValue.set(val);
  }

  onCommit() {
    if (this.pendingValue()) {
      this.value.set(this.pendingValue());
      this.pendingValue.set('');
    }
  }
}
```

## Two-Way Binding with `linkedSignal`

For cases where you want a local mutable copy of a parent-provided value, `linkedSignal` pairs well with two-way binding:

```typescript
@Component({
  selector: 'app-editable-field',
  standalone: true,
  template: `
    @if (isEditing()) {
      <input
        [value]="localValue()"
        (input)="localValue.set($any($event.target).value)"
        (keydown.enter)="save()"
        (keydown.escape)="cancel()"
      />
      <button (click)="save()">Save</button>
      <button (click)="cancel()">Cancel</button>
    } @else {
      <span (dblclick)="startEdit()">{{ value() }}</span>
    }
  `
})
export class EditableFieldComponent {
  value = model('');

  isEditing = signal(false);

  // Local copy that resets when the parent value changes
  localValue = linkedSignal(() => this.value());

  startEdit() {
    this.isEditing.set(true);
  }

  save() {
    this.value.set(this.localValue());
    this.isEditing.set(false);
  }

  cancel() {
    this.localValue.set(this.value()); // Reset to original
    this.isEditing.set(false);
  }
}
```

## Desugaring Details with Signals

Understanding exactly how Angular desugars `[()]` with signals clarifies the behavior:

### With a Plain Property (Legacy)

```html
<app-slider [(value)]="volume" />

<!-- Becomes: -->
<app-slider [value]="volume" (valueChange)="volume = $event" />
```

Simple assignment. `volume` is a class property.

### With a WritableSignal

```html
<app-slider [(value)]="volume" />

<!-- Becomes: -->
<app-slider [value]="volume()" (valueChange)="volume.set($event)" />
```

Angular detects that `volume` is a `WritableSignal` and:
1. Reads via `volume()` for the property binding.
2. Writes via `volume.set($event)` for the event binding.

### With a `model()` in the Child

When the child uses `model()`, there is no separate `output()`. The `ModelSignal` itself acts as both the readable input and the writable output. Angular wires it so that calling `.set()` or `.update()` on the child's model signal propagates the change to the parent's bound signal.

```typescript
// Child
export class SliderComponent {
  value = model(0); // Single declaration handles both directions
}

// Parent
export class PlayerComponent {
  volume = signal(50);
}
```

```html
<app-slider [(value)]="volume" />
```

Now `this.value()` in the child reads the same value as `this.volume()` in the parent. When either side calls `.set()`, both are updated.

## Common Patterns

### Form with Multiple Two-Way Bound Fields

```typescript
@Component({
  selector: 'app-address-form',
  standalone: true,
  imports: [FormsModule],
  template: `
    <div class="form-group">
      <label for="street">Street</label>
      <input id="street" [(ngModel)]="street" name="street" />
    </div>
    <div class="form-group">
      <label for="city">City</label>
      <input id="city" [(ngModel)]="city" name="city" />
    </div>
    <div class="form-group">
      <label for="state">State</label>
      <select id="state" [(ngModel)]="state" name="state">
        @for (s of states; track s) {
          <option [value]="s">{{ s }}</option>
        }
      </select>
    </div>
    <div class="form-group">
      <label for="zip">ZIP Code</label>
      <input id="zip" [(ngModel)]="zip" name="zip" maxlength="10" />
    </div>
  `
})
export class AddressFormComponent {
  street = model('');
  city = model('');
  state = model('');
  zip = model('');

  states = ['CA', 'NY', 'TX', 'FL', 'WA'];
}
```

```html
<!-- Parent gets two-way sync on all fields -->
<app-address-form
  [(street)]="address().street"
  [(city)]="address().city"
  [(state)]="address().state"
  [(zip)]="address().zip"
/>
```

### Controlled vs Uncontrolled Components

Two-way binding creates a "controlled" component where the parent owns the source of truth.

```html
<!-- Controlled: parent owns the value -->
<app-search [(query)]="searchTerm" />

<!-- Uncontrolled: child owns the value, parent can read initial state -->
<app-search [query]="initialTerm" />

<!-- Fully uncontrolled: child manages everything internally -->
<app-search />
```

With `model()`, all three patterns work from the same component definition:

```typescript
export class SearchComponent {
  query = model(''); // Default empty string

  // Works whether parent provides [(query)], [query], or nothing
  results = computed(() => this.filterResults(this.query()));
}
```

## Common Mistakes

- **Mistake 1: Forgetting to import `FormsModule` for `ngModel`.**
  The most common Angular error. `ngModel` is a directive from `@angular/forms`. Without `FormsModule` in the component's `imports` array (or the NgModule), you get `Can't bind to 'ngModel' since it isn't a known property`. Always import `FormsModule` when using `ngModel`.

- **Mistake 2: Missing the `name` attribute when using `ngModel` in a form.**
  Inside a `<form>` element, Angular requires a `name` attribute on every `ngModel` element for form registration. Without it, you get a runtime error. Outside a `<form>`, the `name` attribute is optional but recommended.

- **Mistake 3: Misspelling the `Change` suffix in custom two-way binding.**
  For `[(value)]` to work, the output must be named exactly `valueChange`, not `valueChanged`, `onValueChange`, or `valueUpdate`. Angular's desugaring is based on this exact convention.

- **Mistake 4: Using `model()` when `input()` + `output()` is more appropriate.**
  `model()` propagates every `.set()` call to the parent. If you need validation or transformation before notifying the parent, use `input()` + `output()` so you can call `emit()` only when the value is valid.

- **Mistake 5: Two-way binding to a non-writable value.**
  `[(value)]="readOnlyComputed"` will fail because a `computed` signal is not writable. The target of `[()]` must be a `WritableSignal`, a plain class property, or a setter.

- **Mistake 6: Expecting `ngModel` to work without `FormsModule` on standalone components.**
  Standalone components do not inherit imports from any module. You must explicitly add `FormsModule` to the component's `imports` array, even if it was available in a parent module previously.

---

## Best Practices

- **Use `model()` for custom component two-way binding.** It is the signal-based, single-declaration approach that replaces the `input()` + `output()` pair. Less boilerplate, full signal reactivity, automatic parent sync.

- **Use `ngModel` for native form elements.** `ngModel` handles the differences between input types (`text`, `checkbox`, `radio`, `select`, `textarea`) and provides form validation integration. Do not re-implement what `ngModel` already does.

- **Prefer `[(ngModel)]` with signals.** Since Angular 17.1, `ngModel` supports `WritableSignal` directly. Your component properties can be signals and `ngModel` will call `.set()` on changes.

- **Keep two-way bound components focused on a single value.** If a component needs to expose multiple two-way bound values, consider whether it should be multiple components instead. Each `model()` should represent one clear, cohesive piece of state.

- **Use `linkedSignal` for local editable copies.** When you need to edit a value locally before committing it (like a form with save/cancel), create a `linkedSignal` from the model input and write back to the model on save.

- **Add `name` attributes to all `ngModel` elements in forms.** This is required for Angular's form tracking and validation system to work correctly.

## Real-World Use Cases

### Color Picker with Two-Way Bound Model

A color picker component that exposes its selected color through `model()` for two-way binding.

```typescript
@Component({
  selector: 'app-color-picker',
  standalone: true,
  template: `
    <div class="color-picker">
      <div
        class="preview"
        [style.background-color]="color()"
        (click)="isOpen.set(!isOpen())"
      ></div>

      @if (isOpen()) {
        <div class="palette">
          @for (c of palette; track c) {
            <button
              class="swatch"
              [style.background-color]="c"
              [class.selected]="c === color()"
              (click)="selectColor(c)"
              [attr.aria-label]="c"
            ></button>
          }
          <div class="custom">
            <label>Custom:</label>
            <input
              type="color"
              [value]="color()"
              (input)="selectColor($any($event.target).value)"
            />
          </div>
        </div>
      }
    </div>
  `,
  styles: [`
    .preview { width: 40px; height: 40px; border-radius: 4px; border: 2px solid #ccc; cursor: pointer; }
    .palette { display: flex; flex-wrap: wrap; gap: 4px; padding: 8px; background: white; border: 1px solid #ddd; border-radius: 4px; }
    .swatch { width: 28px; height: 28px; border: 2px solid transparent; border-radius: 4px; cursor: pointer; }
    .swatch.selected { border-color: #333; }
  `]
})
export class ColorPickerComponent {
  color = model('#1976d2');

  isOpen = signal(false);

  palette = [
    '#f44336', '#e91e63', '#9c27b0', '#673ab7',
    '#3f51b5', '#1976d2', '#03a9f4', '#00bcd4',
    '#009688', '#4caf50', '#8bc34a', '#cddc39',
    '#ffeb3b', '#ffc107', '#ff9800', '#ff5722',
    '#795548', '#9e9e9e', '#607d8b', '#000000',
  ];

  selectColor(c: string) {
    this.color.set(c);
    this.isOpen.set(false);
  }
}
```

```html
<!-- Parent uses two-way binding -->
<app-color-picker [(color)]="themeColor" />
<div [style.background-color]="themeColor()">
  Themed content: {{ themeColor() }}
</div>
```

### Range Slider with Model and Validation

A slider that uses `model()` for the value and validates bounds internally.

```typescript
@Component({
  selector: 'app-range-slider',
  standalone: true,
  template: `
    <div class="slider-container">
      <input
        type="range"
        [min]="min()"
        [max]="max()"
        [step]="step()"
        [value]="value()"
        (input)="onSlide($event)"
      />
      <div class="slider-labels">
        <span>{{ min() }}</span>
        <span class="current-value">{{ value() }}</span>
        <span>{{ max() }}</span>
      </div>
    </div>
  `,
  styles: [`
    .slider-container { width: 100%; }
    input[type="range"] { width: 100%; }
    .slider-labels { display: flex; justify-content: space-between; font-size: 0.75rem; color: #666; }
    .current-value { font-weight: 600; color: #1976d2; }
  `]
})
export class RangeSliderComponent {
  value = model(50);
  min = input(0);
  max = input(100);
  step = input(1);

  onSlide(event: Event) {
    const input = event.target as HTMLInputElement;
    const num = Number(input.value);
    // Clamp to bounds
    const clamped = Math.max(this.min(), Math.min(this.max(), num));
    this.value.set(clamped);
  }
}
```

```html
<h3>Volume: {{ volume() }}</h3>
<app-range-slider [(value)]="volume" [min]="0" [max]="100" [step]="5" />

<h3>Brightness: {{ brightness() }}%</h3>
<app-range-slider [(value)]="brightness" [min]="10" [max]="100" />
```

### Settings Form with `ngModel` and Signals

A complete settings form using `ngModel` with signal-backed properties and `FormsModule`.

```typescript
@Component({
  selector: 'app-settings',
  standalone: true,
  imports: [FormsModule],
  template: `
    <form class="settings-form">
      <section>
        <h3>Profile</h3>
        <div class="field">
          <label for="displayName">Display Name</label>
          <input id="displayName" [(ngModel)]="displayName" name="displayName" />
        </div>
        <div class="field">
          <label for="email">Email</label>
          <input id="email" [(ngModel)]="email" name="email" type="email" />
        </div>
        <div class="field">
          <label for="bio">Bio</label>
          <textarea id="bio" [(ngModel)]="bio" name="bio" rows="3"></textarea>
        </div>
      </section>

      <section>
        <h3>Preferences</h3>
        <div class="field">
          <label for="language">Language</label>
          <select id="language" [(ngModel)]="language" name="language">
            <option value="en">English</option>
            <option value="es">Spanish</option>
            <option value="fr">French</option>
            <option value="de">German</option>
          </select>
        </div>
        <div class="field">
          <label for="theme">Theme</label>
          <select id="theme" [(ngModel)]="theme" name="theme">
            <option value="light">Light</option>
            <option value="dark">Dark</option>
            <option value="system">System</option>
          </select>
        </div>
        <div class="field checkbox">
          <input
            id="notifications"
            type="checkbox"
            [(ngModel)]="emailNotifications"
            name="notifications"
          />
          <label for="notifications">Email Notifications</label>
        </div>
        <div class="field checkbox">
          <input
            id="marketing"
            type="checkbox"
            [(ngModel)]="marketingEmails"
            name="marketing"
          />
          <label for="marketing">Marketing Emails</label>
        </div>
      </section>

      <div class="actions">
        <button type="button" (click)="save()">Save Settings</button>
        <button type="button" (click)="reset()">Reset</button>
      </div>

      <pre class="debug">{{ debugInfo() }}</pre>
    </form>
  `
})
export class SettingsComponent {
  displayName = signal('');
  email = signal('');
  bio = signal('');
  language = signal('en');
  theme = signal('system');
  emailNotifications = signal(true);
  marketingEmails = signal(false);

  debugInfo = computed(() => JSON.stringify({
    displayName: this.displayName(),
    email: this.email(),
    bio: this.bio(),
    language: this.language(),
    theme: this.theme(),
    emailNotifications: this.emailNotifications(),
    marketingEmails: this.marketingEmails(),
  }, null, 2));

  private settingsService = inject(SettingsService);

  save() {
    this.settingsService.update({
      displayName: this.displayName(),
      email: this.email(),
      bio: this.bio(),
      language: this.language(),
      theme: this.theme(),
      emailNotifications: this.emailNotifications(),
      marketingEmails: this.marketingEmails(),
    });
  }

  reset() {
    const defaults = this.settingsService.getDefaults();
    this.displayName.set(defaults.displayName);
    this.email.set(defaults.email);
    this.bio.set(defaults.bio);
    this.language.set(defaults.language);
    this.theme.set(defaults.theme);
    this.emailNotifications.set(defaults.emailNotifications);
    this.marketingEmails.set(defaults.marketingEmails);
  }
}
```

> **Key Takeaway:** Use `model()` for signal-based two-way binding in custom components (single declaration, automatic parent sync) and `[(ngModel)]` with `FormsModule` for native form elements. The banana-in-a-box syntax `[()]` always desugars to a property binding plus a `Change`-suffixed event binding.
