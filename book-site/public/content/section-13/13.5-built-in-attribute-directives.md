# 13.5 Built-in Attribute Directives

> Angular ships with several attribute directives that modify the appearance or behavior of DOM elements. Knowing what each one does, when to use it, and when a simpler alternative exists prevents unnecessary imports and keeps templates lean.

---

## What Are Attribute Directives?

Attribute directives are classes decorated with `@Directive` that attach to elements and modify their behavior or appearance. Unlike structural directives (which add/remove elements from the DOM), attribute directives change properties of existing elements.

Angular's built-in attribute directives include:

| Directive | Purpose | Import From |
|---|---|---|
| `NgClass` | Add/remove CSS classes conditionally | `@angular/common` |
| `NgStyle` | Set inline styles dynamically | `@angular/common` |
| `NgModel` | Two-way bind form element values | `@angular/forms` |
| `RouterLink` | Navigate to routes | `@angular/router` |
| `RouterLinkActive` | Add CSS class when a route is active | `@angular/router` |
| `NgOptimizedImage` | Optimize image loading | `@angular/common` |

This guide covers `NgClass`, `NgStyle`, and `NgModel` as the core attribute directives. `RouterLink`/`RouterLinkActive` are covered in the routing section. `NgOptimizedImage` has its own dedicated guide (13.8).

---

## NgClass

`NgClass` adds and removes CSS classes on an element based on an expression.

### Import

```typescript
import { NgClass } from '@angular/common';

@Component({
  standalone: true,
  imports: [NgClass],
  // ...
})
```

### Object Syntax

Keys are class names, values are boolean expressions:

```html
<div [ngClass]="{
  'card': true,
  'card-active': isActive(),
  'card-disabled': isDisabled(),
  'card-highlighted': isHighlighted()
}">
  Content
</div>
```

### Array Syntax

Each element is a class name string to add unconditionally:

```html
<div [ngClass]="['base', 'layout', activeClass()]">Content</div>
```

```typescript
activeClass = computed(() => this.isActive() ? 'active' : 'inactive');
```

### String Syntax

A space-separated string of class names:

```html
<div [ngClass]="'card card-lg shadow'">Content</div>
```

### Space-Separated Keys (Unique to NgClass)

`NgClass` supports multiple space-separated classes in a single object key. This is the one feature that `[class]` binding cannot do:

```html
<!-- This adds BOTH "text-bold" and "text-large" when the condition is true -->
<div [ngClass]="{ 'text-bold text-large': isImportant() }">
  Important message
</div>
```

With plain `[class]`, you would need separate bindings:

```html
<div [class.text-bold]="isImportant()" [class.text-large]="isImportant()">
  Important message
</div>
```

### NgClass vs `[class]` Binding

In most cases, the built-in `[class]` and `[class.name]` syntax is preferred over `NgClass`:

```html
<!-- Preferred: no import needed, more performant -->
<div [class.active]="isActive()" [class.disabled]="isDisabled()">A</div>
<div [class]="classObject()">B</div>

<!-- NgClass: only needed for space-separated keys or mutation detection -->
<div [ngClass]="{ 'class-a class-b': condition() }">C</div>
```

| Feature | `[class]` / `[class.name]` | `NgClass` |
|---|---|---|
| Performance | Direct DOM manipulation | Directive overhead |
| Import required | No | Yes (`NgClass` from `@angular/common`) |
| Single class toggle | `[class.name]="bool"` | Overkill |
| Object input | Yes | Yes |
| Array input | Yes | Yes |
| String input | Yes | Yes |
| Space-separated keys | No | Yes |
| Mutation detection | No (needs new reference) | Yes |

**Recommendation:** Use `[class]` and `[class.name]` for new code. Use `NgClass` only when you need space-separated class keys or when mutating an existing object without changing its reference.

## NgStyle

`NgStyle` sets one or more inline style properties based on an expression.

### Import

```typescript
import { NgStyle } from '@angular/common';

@Component({
  standalone: true,
  imports: [NgStyle],
  // ...
})
```

### Object Syntax

```html
<div [ngStyle]="{
  'color': textColor(),
  'font-size.px': fontSize(),
  'background-color': isHighlighted() ? '#fff3e0' : null,
  'padding.px': padding()
}">
  Styled content
</div>
```

Keys are CSS property names (hyphenated or camelCase). Values are the CSS values. The `.unit` suffix works the same as with `[style]` binding. `null` removes the property.

### NgStyle vs `[style]` Binding

The Angular team has stated that `NgStyle` no longer provides significant value over the built-in `[style]` binding syntax and may eventually be removed.

```html
<!-- Preferred: no import needed, more performant -->
<div [style.color]="textColor()" [style.font-size.px]="fontSize()">A</div>
<div [style]="styleObject()">B</div>

<!-- NgStyle: no significant advantage -->
<div [ngStyle]="{ 'color': textColor(), 'font-size.px': fontSize() }">C</div>
```

| Feature | `[style]` / `[style.prop]` | `NgStyle` |
|---|---|---|
| Performance | Direct DOM manipulation | Directive overhead |
| Import required | No | Yes (`NgStyle` from `@angular/common`) |
| Single property | `[style.prop]="value"` | Overkill |
| Object input | Yes | Yes |
| String input | Yes | No |
| Unit suffix | Yes (`.px`, `.rem`, `.%`) | Yes |

**Recommendation:** Use `[style]` and `[style.prop]` exclusively. There is no scenario where `NgStyle` is strictly necessary.

## NgModel

`NgModel` creates a two-way data binding between a form element and a component property. It is an attribute directive from the Forms module.

### Import

```typescript
import { FormsModule } from '@angular/forms';

@Component({
  standalone: true,
  imports: [FormsModule], // NgModel is part of FormsModule
  // ...
})
```

### Basic Usage

```html
<input [(ngModel)]="username" name="username" />
<textarea [(ngModel)]="bio" name="bio"></textarea>
<select [(ngModel)]="role" name="role">
  <option value="admin">Admin</option>
  <option value="user">User</option>
</select>
<input type="checkbox" [(ngModel)]="isActive" name="active" />
```

### With Signals

Since Angular 17.1, `ngModel` works directly with `WritableSignal`:

```typescript
export class FormComponent {
  username = signal('');
  bio = signal('');
  role = signal('user');
  isActive = signal(true);
}
```

`ngModel` detects writable signals and uses `.set()` to update them. No extra syntax needed.

### One-Way Read with `ngModel`

You can use `ngModel` in one-way mode:

```html
<!-- One-way: sets initial value, does NOT sync changes back -->
<input [ngModel]="initialValue()" name="field" />

<!-- Two-way: full synchronization -->
<input [(ngModel)]="currentValue" name="field" />
```

### ngModel with Change Events

You can split the binding and add custom handling:

```html
<input
  [ngModel]="searchTerm()"
  (ngModelChange)="onSearchChange($event)"
  name="search"
/>
```

```typescript
onSearchChange(value: string) {
  this.searchTerm.set(value);
  this.performSearch(value);
}
```

This is equivalent to the two-way binding but gives you an explicit handler for the change event.

### ngModel Inside a Form

When `ngModel` is inside a `<form>` element, a `name` attribute is required:

```html
<form #myForm="ngForm" (ngSubmit)="onSubmit(myForm)">
  <!-- name is REQUIRED inside a form -->
  <input [(ngModel)]="email" name="email" type="email" required />
  <input [(ngModel)]="password" name="password" type="password" required />

  <button type="submit" [disabled]="myForm.invalid">Submit</button>
</form>
```

```typescript
onSubmit(form: NgForm) {
  if (form.valid) {
    console.log('Form value:', form.value);
    // { email: '...', password: '...' }
  }
}
```

### ngModel with Validation

`ngModel` integrates with Angular's template-driven validation:

```html
<input
  [(ngModel)]="email"
  name="email"
  required
  email
  #emailField="ngModel"
/>

@if (emailField.invalid && emailField.touched) {
  <div class="error">
    @if (emailField.errors?.['required']) {
      <span>Email is required</span>
    }
    @if (emailField.errors?.['email']) {
      <span>Invalid email format</span>
    }
  </div>
}
```

Exporting `ngModel` as a template reference variable (`#emailField="ngModel"`) gives you access to its validation state: `valid`, `invalid`, `pristine`, `dirty`, `touched`, `untouched`, and `errors`.

## Legacy Structural Directives (Deprecated in Angular 20)

While not attribute directives, it is important to note the deprecation status of the structural directives, as they are the other category of built-in directives you will encounter in existing codebases.

### Deprecated Structural Directives

| Deprecated Directive | Modern Replacement | Status |
|---|---|---|
| `*ngIf` / `NgIf` | `@if` / `@else` | Deprecated in v20, removal planned v22 |
| `*ngFor` / `NgFor` | `@for` with `track` | Deprecated in v20, removal planned v22 |
| `*ngSwitch` / `NgSwitch` | `@switch` / `@case` | Deprecated in v20, removal planned v22 |
| `NgSwitchCase` | `@case` | Deprecated in v20, removal planned v22 |
| `NgSwitchDefault` | `@default` | Deprecated in v20, removal planned v22 |

### Migration

Angular CLI provides a migration schematic:

```bash
ng generate @angular/core:control-flow-migration
```

This converts templates from the directive syntax to the new control flow syntax automatically.

### Why They Were Deprecated

1. **Performance.** The new control flow is compiled at build time, not interpreted at runtime by directives. This eliminates directive overhead.
2. **Bundle size.** No need to import `NgIf`, `NgFor`, or `NgSwitch` from `@angular/common`.
3. **Memory.** `*ngFor` has a subtle memory leak when using `trackBy` that the new `@for` with mandatory `track` avoids.
4. **Readability.** `@if`, `@for`, `@switch` read more naturally and do not require the `*` microsyntax.

### Legacy NgSwitch Example (For Reference)

You will see this in existing codebases:

```html
<!-- Legacy NgSwitch (deprecated) -->
<div [ngSwitch]="status()">
  <span *ngSwitchCase="'active'" class="badge-active">Active</span>
  <span *ngSwitchCase="'inactive'" class="badge-inactive">Inactive</span>
  <span *ngSwitchCase="'pending'" class="badge-pending">Pending</span>
  <span *ngSwitchDefault class="badge-unknown">Unknown</span>
</div>

<!-- Modern equivalent -->
@switch (status()) {
  @case ('active') {
    <span class="badge-active">Active</span>
  }
  @case ('inactive') {
    <span class="badge-inactive">Inactive</span>
  }
  @case ('pending') {
    <span class="badge-pending">Pending</span>
  }
  @default {
    <span class="badge-unknown">Unknown</span>
  }
}
```

## Attribute Directives on Host Elements

Attribute directives can also be applied to component host elements:

```html
<!-- NgClass on a component -->
<app-card [ngClass]="{ 'featured': isFeatured() }">Content</app-card>

<!-- Class binding on a component (preferred) -->
<app-card [class.featured]="isFeatured()">Content</app-card>

<!-- Style binding on a component -->
<app-card [style.--card-bg]="cardColor()">Content</app-card>
```

The directive or binding modifies the host element's class list or style properties.

---

## Combining Multiple Directives

An element can have multiple attribute directives:

```html
<input
  [(ngModel)]="searchTerm"
  name="search"
  [ngClass]="{ 'input-error': hasError(), 'input-focused': isFocused() }"
  [ngStyle]="{ 'width.px': inputWidth() }"
  appAutoFocus
  appDebounceInput
/>
```

However, the modern approach avoids `NgClass` and `NgStyle` here:

```html
<input
  [(ngModel)]="searchTerm"
  name="search"
  [class.input-error]="hasError()"
  [class.input-focused]="isFocused()"
  [style.width.px]="inputWidth()"
  appAutoFocus
  appDebounceInput
/>
```

Less imports, same result, better performance.

## Common Mistakes

- **Mistake 1: Forgetting to import the directive.**
  In standalone components, `NgClass`, `NgStyle`, and `FormsModule` (for `NgModel`) must be explicitly imported. Without the import, the directive silently does not apply (or throws a template error for `ngModel`). Prefer `[class]`/`[style]` bindings that require no imports.

- **Mistake 2: Using `NgClass` or `NgStyle` when `[class]`/`[style]` bindings suffice.**
  The built-in bindings are recommended by the Angular team, need no imports, and have less overhead. Reaching for `NgClass`/`NgStyle` is unnecessary in most cases.

- **Mistake 3: Missing the `name` attribute on `ngModel` inside forms.**
  Inside a `<form>` tag, every `[(ngModel)]` element must have a `name` attribute. Without it, Angular throws a runtime error because it cannot register the control with the form.

- **Mistake 4: Using `*ngIf`, `*ngFor`, or `*ngSwitch` in new code.**
  These structural directives are deprecated since Angular 20. Use `@if`, `@for`, and `@switch` control flow blocks instead. They are faster, smaller, and do not need imports.

- **Mistake 5: Using `NgStyle` with the `.unit` suffix incorrectly.**
  Writing `[ngStyle]="{ 'width': 200 }"` sets `width: 200` (no unit, invalid CSS). Use `[ngStyle]="{ 'width.px': 200 }"` or `[ngStyle]="{ 'width': '200px' }"`. The same applies to `[style]` binding.

- **Mistake 6: Expecting `NgClass` to react to object mutations.**
  While `NgClass` is documented as supporting mutation detection, relying on it is fragile. Prefer computed signals that return new objects, which work consistently with both `NgClass` and `[class]` binding.

---

## Best Practices

- **Prefer `[class]` and `[class.name]` over `NgClass`.** They are faster, require no imports, and handle 95% of use cases. Only use `NgClass` for space-separated class keys.

- **Prefer `[style]` and `[style.prop]` over `NgStyle`.** There is no practical reason to use `NgStyle` in new code. The built-in binding syntax does everything `NgStyle` does with less overhead.

- **Use `@if`/`@for`/`@switch` instead of `*ngIf`/`*ngFor`/`*ngSwitch`.** The control flow blocks are the modern replacement. Run the migration schematic on existing projects.

- **Import `FormsModule` for `ngModel`, not individual directives.** `NgModel` is part of `FormsModule`. Import the module, not the directive directly.

- **Export `ngModel` as a template reference for validation.** Use `#field="ngModel"` to access the control's validation state (`valid`, `invalid`, `touched`, `errors`).

- **Keep directive usage minimal in templates.** Every directive is a class instance that Angular manages. Fewer directives means less overhead. The built-in bindings and control flow blocks achieve the same results without directive instances.

## Real-World Use Cases

### Dynamic Form with ngModel and Validation

A registration form using `ngModel` for two-way binding with template-driven validation.

```typescript
@Component({
  selector: 'app-registration',
  standalone: true,
  imports: [FormsModule],
  template: `
    <form #regForm="ngForm" (ngSubmit)="onSubmit(regForm)" class="form">
      <div class="field">
        <label for="name">Full Name</label>
        <input
          id="name"
          [(ngModel)]="name"
          name="name"
          required
          minlength="2"
          #nameField="ngModel"
          [class.input-error]="nameField.invalid && nameField.touched"
        />
        @if (nameField.invalid && nameField.touched) {
          <span class="error">
            @if (nameField.errors?.['required']) { Name is required. }
            @if (nameField.errors?.['minlength']) { At least 2 characters. }
          </span>
        }
      </div>

      <div class="field">
        <label for="email">Email</label>
        <input
          id="email"
          [(ngModel)]="email"
          name="email"
          type="email"
          required
          email
          #emailField="ngModel"
          [class.input-error]="emailField.invalid && emailField.touched"
        />
        @if (emailField.invalid && emailField.touched) {
          <span class="error">
            @if (emailField.errors?.['required']) { Email is required. }
            @if (emailField.errors?.['email']) { Invalid email format. }
          </span>
        }
      </div>

      <div class="field">
        <label for="password">Password</label>
        <input
          id="password"
          [(ngModel)]="password"
          name="password"
          type="password"
          required
          minlength="8"
          #pwField="ngModel"
          [class.input-error]="pwField.invalid && pwField.touched"
        />
        @if (pwField.invalid && pwField.touched) {
          <span class="error">
            @if (pwField.errors?.['required']) { Password is required. }
            @if (pwField.errors?.['minlength']) { At least 8 characters. }
          </span>
        }
      </div>

      <div class="field">
        <label for="role">Role</label>
        <select
          id="role"
          [(ngModel)]="role"
          name="role"
          required
          #roleField="ngModel"
        >
          <option value="">Select a role</option>
          <option value="developer">Developer</option>
          <option value="designer">Designer</option>
          <option value="manager">Manager</option>
        </select>
      </div>

      <div class="field checkbox">
        <input
          id="terms"
          type="checkbox"
          [(ngModel)]="agreedToTerms"
          name="terms"
          required
          #termsField="ngModel"
        />
        <label for="terms">I agree to the terms and conditions</label>
      </div>

      <button
        type="submit"
        [disabled]="regForm.invalid || isSubmitting()"
        [class.btn-loading]="isSubmitting()"
      >
        {{ isSubmitting() ? 'Registering...' : 'Register' }}
      </button>
    </form>
  `,
  styles: [`
    .form { max-width: 400px; display: flex; flex-direction: column; gap: 16px; }
    .field { display: flex; flex-direction: column; gap: 4px; }
    .field.checkbox { flex-direction: row; align-items: center; gap: 8px; }
    .error { color: #f44336; font-size: 0.8rem; }
    .input-error { border-color: #f44336; }
    button:disabled { opacity: 0.6; cursor: not-allowed; }
  `]
})
export class RegistrationComponent {
  name = signal('');
  email = signal('');
  password = signal('');
  role = signal('');
  agreedToTerms = signal(false);
  isSubmitting = signal(false);

  private authService = inject(AuthService);

  onSubmit(form: NgForm) {
    if (form.invalid) return;

    this.isSubmitting.set(true);
    this.authService.register({
      name: this.name(),
      email: this.email(),
      password: this.password(),
      role: this.role(),
    }).subscribe({
      next: () => this.isSubmitting.set(false),
      error: () => this.isSubmitting.set(false),
    });
  }
}
```

### Conditional Styling with Class Binding (No NgClass)

A notification list where each item has different styling based on type and read status, using only built-in class bindings.

```typescript
@Component({
  selector: 'app-notification-list',
  standalone: true,
  imports: [DatePipe],
  template: `
    <ul class="notifications">
      @for (item of notifications(); track item.id) {
        <li
          class="notification"
          [class.notification-unread]="!item.read"
          [class.notification-info]="item.type === 'info'"
          [class.notification-warning]="item.type === 'warning'"
          [class.notification-error]="item.type === 'error'"
          [class.notification-success]="item.type === 'success'"
          (click)="markAsRead(item.id)"
        >
          <span class="notification-icon">
            @switch (item.type) {
              @case ('info') { i }
              @case ('warning') { ! }
              @case ('error') { x }
              @case ('success') { ✓ }
            }
          </span>
          <div class="notification-body">
            <p class="notification-message">{{ item.message }}</p>
            <time
              class="notification-time"
              [class.recent]="isRecent(item.timestamp)"
            >
              {{ item.timestamp | date:'short' }}
            </time>
          </div>
        </li>
      } @empty {
        <li class="no-notifications">No notifications</li>
      }
    </ul>
  `,
  styles: [`
    .notifications { list-style: none; padding: 0; margin: 0; }
    .notification { display: flex; gap: 12px; padding: 12px; border-bottom: 1px solid #eee; cursor: pointer; }
    .notification:hover { background: #fafafa; }
    .notification-unread { background: #f5f5ff; font-weight: 500; }
    .notification-info .notification-icon { color: #1976d2; }
    .notification-warning .notification-icon { color: #ff9800; }
    .notification-error .notification-icon { color: #f44336; }
    .notification-success .notification-icon { color: #4caf50; }
    .notification-time { font-size: 0.75rem; color: #999; }
    .notification-time.recent { color: #1976d2; }
  `]
})
export class NotificationListComponent {
  notifications = input.required<Notification[]>();
  read = output<string>();

  markAsRead(id: string) {
    this.read.emit(id);
  }

  isRecent(timestamp: Date): boolean {
    return Date.now() - new Date(timestamp).getTime() < 3600000; // 1 hour
  }
}
```

This example achieves complex conditional class logic using only `[class.name]` bindings. No `NgClass` import needed.

### Settings Panel with ngModel and Computed Signals

A settings panel that demonstrates `ngModel` with signals and computed derived values.

```typescript
@Component({
  selector: 'app-display-settings',
  standalone: true,
  imports: [FormsModule],
  template: `
    <div class="settings-panel">
      <h3>Display Settings</h3>

      <div class="setting">
        <label for="fontSize">Font Size: {{ fontSize() }}px</label>
        <input
          id="fontSize"
          type="range"
          [(ngModel)]="fontSize"
          name="fontSize"
          min="12"
          max="24"
          step="1"
        />
      </div>

      <div class="setting">
        <label for="lineHeight">Line Height: {{ lineHeight() }}</label>
        <input
          id="lineHeight"
          type="range"
          [(ngModel)]="lineHeight"
          name="lineHeight"
          min="1"
          max="2.5"
          step="0.1"
        />
      </div>

      <div class="setting">
        <label for="fontFamily">Font Family</label>
        <select id="fontFamily" [(ngModel)]="fontFamily" name="fontFamily">
          <option value="system-ui">System Default</option>
          <option value="'Inter', sans-serif">Inter</option>
          <option value="'Georgia', serif">Georgia</option>
          <option value="'Fira Code', monospace">Fira Code</option>
        </select>
      </div>

      <div class="setting">
        <input
          id="darkMode"
          type="checkbox"
          [(ngModel)]="darkMode"
          name="darkMode"
        />
        <label for="darkMode">Dark Mode</label>
      </div>

      <div
        class="preview"
        [style.font-size.px]="fontSize()"
        [style.line-height]="lineHeight()"
        [style.font-family]="fontFamily()"
        [class.dark]="darkMode()"
      >
        <p>The quick brown fox jumps over the lazy dog.</p>
        <p>{{ previewDetails() }}</p>
      </div>

      <button (click)="resetDefaults()">Reset to Defaults</button>
    </div>
  `,
  styles: [`
    .settings-panel { max-width: 500px; }
    .setting { margin-bottom: 12px; }
    .preview { padding: 16px; border: 1px solid #ddd; border-radius: 4px; margin: 16px 0; }
    .preview.dark { background: #1e1e1e; color: #e0e0e0; border-color: #444; }
  `]
})
export class DisplaySettingsComponent {
  fontSize = signal(16);
  lineHeight = signal(1.5);
  fontFamily = signal('system-ui');
  darkMode = signal(false);

  previewDetails = computed(() =>
    `${this.fontSize()}px / ${this.lineHeight()} / ${this.fontFamily()} / ${this.darkMode() ? 'dark' : 'light'}`
  );

  resetDefaults() {
    this.fontSize.set(16);
    this.lineHeight.set(1.5);
    this.fontFamily.set('system-ui');
    this.darkMode.set(false);
  }
}
```

---

> **Key Takeaway:** Prefer the built-in `[class]`/`[style]` bindings and `@if`/`@for`/`@switch` control flow over the `NgClass`, `NgStyle`, and structural directives. The main built-in attribute directive you will continue to use regularly is `NgModel` (via `FormsModule`) for form element two-way binding. Everything else has a lighter-weight native alternative.
