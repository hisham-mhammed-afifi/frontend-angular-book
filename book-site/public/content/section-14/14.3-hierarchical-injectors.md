# 14.3 Hierarchical Injectors

> Angular maintains two parallel injector hierarchies that work together to resolve dependencies: the ElementInjector tree (tied to the component tree) and the EnvironmentInjector tree (tied to the application and route configuration). Understanding how they interact is critical for controlling service scope, lifetime, and sharing.

---

## The Two Injector Hierarchies

Angular's DI system is not a single flat container. It consists of two separate trees of injectors that are consulted in a specific order when resolving a dependency.

### EnvironmentInjector Hierarchy

This hierarchy is independent of the component tree. It is configured through:

- `@Injectable({ providedIn: 'root' })` on service classes.
- The `providers` array in `ApplicationConfig` (passed to `bootstrapApplication()`).
- The `providers` array on route definitions.

```
NullInjector
    |
PlatformInjector
    |
Root EnvironmentInjector  (bootstrapApplication providers + providedIn: 'root')
    |
Route EnvironmentInjector  (route-level providers)
    |
Child Route EnvironmentInjector  (child route providers)
```

### ElementInjector Hierarchy

This hierarchy mirrors the component tree. Every component and directive creates an ElementInjector, though most are empty unless you specify `providers` or `viewProviders`.

```
AppComponent ElementInjector
    |
LayoutComponent ElementInjector
    |
SidebarComponent ElementInjector
    |
MenuItemComponent ElementInjector
```

### How They Work Together

When a component calls `inject(SomeService)`, Angular resolves in two phases:

**Phase 1: Walk up the ElementInjector tree.**
Starting at the requesting component's ElementInjector, Angular checks each ancestor's ElementInjector moving upward through the component tree. If found, resolution stops and returns the instance.

**Phase 2: Fall back to the EnvironmentInjector tree.**
If no ElementInjector provides the token, Angular switches to the EnvironmentInjector associated with the component's location. It walks up through route-level injectors to the root EnvironmentInjector. If still not found, it reaches the PlatformInjector, then the NullInjector, which throws `NullInjectorError` (unless `{ optional: true }` was specified).

```
Component requests inject(ServiceX)
         |
         v
[ElementInjector: current component] --> not found
         |
[ElementInjector: parent component]  --> not found
         |
[ElementInjector: grandparent]       --> not found
         |
         v  (switch to EnvironmentInjector tree)
[Route EnvironmentInjector]           --> not found
         |
[Root EnvironmentInjector]            --> FOUND, return instance
```

---

## The NullInjector

At the very top of the EnvironmentInjector tree sits the `NullInjector`. It contains no providers. When resolution reaches it, Angular throws:

```
NullInjectorError: No provider for SomeService!
```

Unless you used `{ optional: true }`, in which case it returns `null`.

```typescript
// Throws NullInjectorError if SomeService is not provided anywhere
private service = inject(SomeService);

// Returns null instead of throwing
private service = inject(SomeService, { optional: true });
```

---

## The Platform Injector

Below the NullInjector is the PlatformInjector, created by `platformBrowserDynamic()` (or `platformBrowser()` for AOT). It provides platform-specific services like `DomSanitizer`. You rarely interact with it directly.

Services registered with `@Injectable({ providedIn: 'platform' })` live here. This scope matters only when multiple Angular applications run on the same page (e.g., Angular Elements).

---

## The Root EnvironmentInjector

This is where most application-wide services live. It is configured in two ways:

### 1. `providedIn: 'root'` (Preferred)

```typescript
@Injectable({ providedIn: 'root' })
export class AuthService {
  // Singleton for the entire application
  // Tree-shakable: removed from bundle if never injected
}
```

### 2. `ApplicationConfig` Providers

```typescript
// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(),
    provideRouter(routes),
    { provide: API_BASE_URL, useValue: 'https://api.example.com' },
  ],
};
```

Services in the `providers` array are NOT tree-shakable because the explicit reference prevents dead code elimination.

## ElementInjector: Component-Level Providers

When you add a `providers` array to a component, Angular registers those providers in that component's ElementInjector. Each component instance gets its own copy of the service.

```typescript
@Injectable()
export class FormStateService {
  fields = signal<Record<string, string>>({});
  isDirty = signal(false);

  setField(name: string, value: string) {
    this.fields.update(f => ({ ...f, [name]: value }));
    this.isDirty.set(true);
  }

  reset() {
    this.fields.set({});
    this.isDirty.set(false);
  }
}

@Component({
  selector: 'app-edit-form',
  standalone: true,
  providers: [FormStateService], // new instance per component
  template: `
    <input (input)="formState.setField('name', $event.target.value)" />
    <p>Dirty: {{ formState.isDirty() }}</p>
  `,
})
export class EditFormComponent {
  protected formState = inject(FormStateService);
}
```

If you render two `<app-edit-form>` elements, each gets its own `FormStateService`. They do not share state.

### Children Inherit the Parent's Provider

Child components that do not provide their own instance receive the parent's instance:

```typescript
@Component({
  selector: 'app-parent',
  standalone: true,
  providers: [CounterService],
  imports: [ChildAComponent, ChildBComponent],
  template: `
    <app-child-a />
    <app-child-b />
  `,
})
export class ParentComponent {}

@Component({
  selector: 'app-child-a',
  standalone: true,
  template: `<p>{{ counter.value() }}</p>`,
})
export class ChildAComponent {
  protected counter = inject(CounterService);
  // Gets the SAME instance as ChildB, provided by Parent
}

@Component({
  selector: 'app-child-b',
  standalone: true,
  template: `<button (click)="counter.increment()">+1</button>`,
})
export class ChildBComponent {
  protected counter = inject(CounterService);
  // Gets the SAME instance as ChildA, provided by Parent
}
```

When ChildB clicks the button, ChildA's template updates because they share the same `CounterService` instance from the parent's ElementInjector.

## `viewProviders` vs `providers`

Components have two separate provider arrays:

### `providers`

Available to the component, its view children, AND its content children (projected via `<ng-content>`).

### `viewProviders`

Available only to the component's own view. NOT available to content children.

```typescript
@Injectable()
export class ThemeService {
  theme = signal('light');
}

@Component({
  selector: 'app-card',
  standalone: true,
  viewProviders: [ThemeService], // only visible in card's own template
  template: `
    <!-- This component CAN inject ThemeService -->
    <app-card-header />

    <!-- Content children CANNOT inject ThemeService -->
    <ng-content />
  `,
})
export class CardComponent {}

// Used as:
<app-card>
  <!-- This is projected content (content child) -->
  <!-- inject(ThemeService) here would NOT find the viewProvider -->
  <app-card-actions />
</app-card>
```

`app-card-header` is defined in CardComponent's template (view child), so it CAN access `ThemeService`. `app-card-actions` is projected content, so it CANNOT access `ThemeService` from `viewProviders`.

### When to Use `viewProviders`

Use `viewProviders` when you want to encapsulate a service so that projected content cannot access it. This is uncommon but useful for component libraries that need strict internal boundaries.

In practice, most developers use `providers` exclusively. `viewProviders` is an advanced tool for library authors.

## Route-Level Providers

Routes can define their own providers, creating an EnvironmentInjector scoped to that route and its children.

```typescript
export const routes: Routes = [
  {
    path: 'admin',
    providers: [AdminService, AdminGuardService],
    children: [
      { path: 'users', component: AdminUsersComponent },
      { path: 'settings', component: AdminSettingsComponent },
    ],
  },
  {
    path: 'shop',
    providers: [CartService],
    loadComponent: () => import('./shop/shop.component').then(m => m.ShopComponent),
  },
];
```

`AdminService` is available to `AdminUsersComponent`, `AdminSettingsComponent`, and all their descendants. It is NOT available to components outside the `/admin` route.

### Route Providers and Lazy Loading

When a route is lazy-loaded, Angular creates a new EnvironmentInjector for that route. Services provided at that level are isolated from the rest of the application.

```typescript
{
  path: 'reports',
  loadChildren: () => import('./reports/routes').then(m => m.REPORT_ROUTES),
  providers: [ReportDataService],
}
```

`ReportDataService` exists only while the `/reports` route is active. When the user navigates away, the EnvironmentInjector (and the service instance) is destroyed.

### Singleton vs Route-Scoped

```typescript
// Singleton: same instance everywhere
@Injectable({ providedIn: 'root' })
export class UserService {}

// Route-scoped: new instance per route activation
// (no providedIn, provided in route config)
@Injectable()
export class ReportFilterService {}
```

## Resolution Walkthrough

Consider this component tree with providers at various levels:

```
bootstrapApplication providers: [LoggerService]

Route /dashboard providers: [DashboardDataService]

AppComponent (no providers)
  └── LayoutComponent providers: [SidebarService]
        └── DashboardComponent (no providers)
              └── WidgetComponent providers: [WidgetConfigService]
```

When `WidgetComponent` calls `inject(X)`, here is the resolution path for different tokens:

| Token | Resolution Path |
|---|---|
| `WidgetConfigService` | Found in WidgetComponent's ElementInjector |
| `SidebarService` | Not in Widget -> found in LayoutComponent's ElementInjector |
| `DashboardDataService` | Not in any ElementInjector -> found in /dashboard route EnvironmentInjector |
| `LoggerService` | Not in any ElementInjector -> not in route -> found in root EnvironmentInjector |
| `UnprovidedService` | Not found anywhere -> NullInjector throws error |

### Shadowing (Provider Override)

If both LayoutComponent and WidgetComponent provide `SidebarService`:

```typescript
@Component({
  providers: [SidebarService], // instance A
})
export class LayoutComponent {}

@Component({
  providers: [SidebarService], // instance B (shadows instance A)
})
export class WidgetComponent {
  private sidebar = inject(SidebarService); // gets instance B
}
```

WidgetComponent gets its own instance (B). LayoutComponent and its other children get instance A. This is called **shadowing**: a closer provider overrides a more distant one.

## Visualizing the Injector Tree

Angular DevTools (browser extension) provides an Injector Tree view that shows:

- The full hierarchy of ElementInjectors and EnvironmentInjectors.
- Which providers are registered at each level.
- The resolution path for any specific injection.

To use it:
1. Open Chrome DevTools.
2. Go to the Angular tab.
3. Select "Injector Tree" view.
4. Click on any component to see its injector chain.

This is the fastest way to debug "wrong instance" or "no provider" issues.

## Multi-Instance Services with ElementInjectors

A common pattern is creating per-component service instances for isolated state:

```typescript
@Injectable()
export class PanelStateService {
  isExpanded = signal(false);
  title = signal('');

  toggle() {
    this.isExpanded.update(v => !v);
  }
}

@Component({
  selector: 'app-accordion-panel',
  standalone: true,
  providers: [PanelStateService],
  template: `
    <div class="panel-header" (click)="state.toggle()">
      {{ state.title() }}
      <span>{{ state.isExpanded() ? 'Collapse' : 'Expand' }}</span>
    </div>
    @if (state.isExpanded()) {
      <div class="panel-body">
        <ng-content />
      </div>
    }
  `,
})
export class AccordionPanelComponent {
  protected state = inject(PanelStateService);
  title = input.required<string>();

  constructor() {
    effect(() => this.state.title.set(this.title()));
  }
}
```

```html
<!-- Each panel has its own PanelStateService instance -->
<app-accordion-panel title="Section 1">Content 1</app-accordion-panel>
<app-accordion-panel title="Section 2">Content 2</app-accordion-panel>
<app-accordion-panel title="Section 3">Content 3</app-accordion-panel>
```

Expanding Section 1 does not affect Section 2 or 3.

## Accessing Parent-Provided Services with `skipSelf`

When a component provides its own instance but also needs the parent's instance:

```typescript
@Injectable()
export class BreadcrumbService {
  label = signal('');
  parent = signal<BreadcrumbService | null>(null);

  fullPath = computed((): string[] => {
    const parentPath = this.parent()?.fullPath() ?? [];
    const current = this.label();
    return current ? [...parentPath, current] : parentPath;
  });
}

@Component({
  selector: 'app-section',
  standalone: true,
  providers: [BreadcrumbService],
  template: `
    <nav>{{ breadcrumb.fullPath().join(' > ') }}</nav>
    <ng-content />
  `,
})
export class SectionComponent {
  private breadcrumb = inject(BreadcrumbService);
  private parentBreadcrumb = inject(BreadcrumbService, { skipSelf: true, optional: true });
  label = input.required<string>();

  constructor() {
    effect(() => {
      this.breadcrumb.label.set(this.label());
      this.breadcrumb.parent.set(this.parentBreadcrumb);
    });
  }
}
```

```html
<app-section label="Home">
  <app-section label="Products">
    <app-section label="Electronics">
      <!-- Breadcrumb: Home > Products > Electronics -->
    </app-section>
  </app-section>
</app-section>
```

Each `SectionComponent` provides its own `BreadcrumbService` and uses `skipSelf` to find the parent's instance.

## Service Lifetime and Destruction

### Root Services

Created once when first injected. Never destroyed (live as long as the application).

### Component-Scoped Services

Created when the component is instantiated. Destroyed when the component is removed from the DOM. Use `DestroyRef` for cleanup.

### Route-Scoped Services

Created when the route is activated. Destroyed when the route is deactivated (user navigates away). Services in route `providers` follow the route lifecycle.

```typescript
@Injectable()
export class EditorAutoSaveService {
  private destroyRef = inject(DestroyRef);
  private http = inject(HttpClient);
  private content = signal('');

  private autoSaveInterval: ReturnType<typeof setInterval>;

  constructor() {
    this.autoSaveInterval = setInterval(() => {
      this.save();
    }, 30_000);

    this.destroyRef.onDestroy(() => {
      clearInterval(this.autoSaveInterval);
      this.save(); // final save on navigation away
    });
  }

  updateContent(text: string) {
    this.content.set(text);
  }

  private save() {
    const text = this.content();
    if (text) {
      this.http.post('/api/draft', { content: text }).subscribe();
    }
  }
}

// Route config
{
  path: 'editor/:id',
  component: EditorComponent,
  providers: [EditorAutoSaveService],
}
```

## Common Mistakes

- **Accidentally creating multiple instances of a service intended to be a singleton.** This happens when you add a `providedIn: 'root'` service to a component's `providers` array. The component gets its own instance that does not share state with the root singleton. Remove it from `providers` if you want the root singleton.

```typescript
// Bad Practice: re-providing a root singleton
@Injectable({ providedIn: 'root' })
export class CartService { ... }

@Component({
  providers: [CartService], // WRONG: creates a second instance
})
export class CheckoutComponent { ... }
```

- **Expecting route-level providers to persist across navigations.** When the user navigates away from a route and comes back, the EnvironmentInjector is destroyed and recreated. Any state in route-provided services is lost. If you need persistent state, use `providedIn: 'root'`.

- **Confusing `providers` and `viewProviders`.** Services in `viewProviders` are invisible to content children (projected content). If a projected component needs a service, it must be in `providers`, not `viewProviders`.

- **Not understanding shadowing.** When a child component provides the same service as a parent, the child's instance shadows the parent's. This is by design, but unexpected if you intended to share the parent's instance. Use `skipSelf` to explicitly reach the parent's instance.

- **Providing non-singleton services without `@Injectable()`.** If a service is meant to be component-scoped (no `providedIn`), it still needs `@Injectable()` if it injects other services. Without the decorator, Angular cannot resolve its dependencies.

---

## Best Practices

- **Default to `providedIn: 'root'` for application-wide services.** This is the simplest, most tree-shakable approach. Only use component or route providers when you specifically need scoped instances.

- **Use component-level providers for per-instance state.** When each component instance needs isolated state (form state, panel state, undo history), provide the service in the component's `providers` array.

- **Use route-level providers for feature-scoped services.** When a service should exist only while a feature route is active, declare it in the route's `providers`. This prevents memory leaks and ensures clean state on re-entry.

- **Use `skipSelf` + `optional` for parent-child communication through services.** This pattern allows each level in a component tree to provide its own instance while maintaining access to the parent's instance.

- **Use Angular DevTools Injector Tree for debugging.** When a service resolves to the wrong instance or throws `NullInjectorError`, the Injector Tree view shows exactly which injectors are checked and where providers are registered.

- **Document provider scope in service files.** When a service omits `providedIn`, add a comment explaining where it should be provided and why.

```typescript
/**
 * Manages form state for a single form instance.
 * Provide in the component that owns the form.
 */
@Injectable()
export class FormStateService { ... }
```

## Real-World Use Cases

### Nested Tab Groups with Isolated State

```typescript
@Injectable()
export class TabGroupService {
  tabs = signal<TabConfig[]>([]);
  activeIndex = signal(0);

  register(tab: TabConfig) {
    this.tabs.update(list => [...list, tab]);
  }

  activate(index: number) {
    this.activeIndex.set(index);
  }
}

@Component({
  selector: 'app-tab-group',
  standalone: true,
  providers: [TabGroupService], // each tab-group gets its own instance
  template: `
    <div class="tab-headers">
      @for (tab of tabGroup.tabs(); track tab.id; let i = $index) {
        <button
          [class.active]="i === tabGroup.activeIndex()"
          (click)="tabGroup.activate(i)"
        >
          {{ tab.label }}
        </button>
      }
    </div>
    <div class="tab-content">
      <ng-content />
    </div>
  `,
})
export class TabGroupComponent {
  protected tabGroup = inject(TabGroupService);
}

@Component({
  selector: 'app-tab',
  standalone: true,
  template: `
    @if (isActive()) {
      <ng-content />
    }
  `,
})
export class TabComponent {
  private tabGroup = inject(TabGroupService); // gets parent's instance
  label = input.required<string>();

  private index = -1;

  isActive = computed(() => this.tabGroup.activeIndex() === this.index);

  constructor() {
    this.index = this.tabGroup.tabs().length;
    this.tabGroup.register({ id: crypto.randomUUID(), label: this.label() });
  }
}
```

```html
<!-- Outer tab group -->
<app-tab-group>
  <app-tab label="Overview">
    <!-- Nested tab group with its OWN TabGroupService -->
    <app-tab-group>
      <app-tab label="Summary">Summary content</app-tab>
      <app-tab label="Details">Details content</app-tab>
    </app-tab-group>
  </app-tab>
  <app-tab label="Settings">Settings content</app-tab>
</app-tab-group>
```

Each `<app-tab-group>` creates its own `TabGroupService`. The inner and outer tab groups are completely independent.

### Feature Module with Route-Scoped State

```typescript
// shopping/cart.service.ts
@Injectable() // no providedIn: scoped to route
export class CartService {
  private _items = signal<CartItem[]>([]);
  readonly items = this._items.asReadonly();
  readonly total = computed(() =>
    this._items().reduce((sum, i) => sum + i.price * i.quantity, 0)
  );

  addItem(item: CartItem) {
    this._items.update(list => [...list, item]);
  }
}

// shopping/routes.ts
export const SHOPPING_ROUTES: Routes = [
  {
    path: '',
    providers: [CartService], // single instance for the entire shopping feature
    children: [
      { path: 'browse', component: BrowseComponent },
      { path: 'cart', component: CartComponent },
      { path: 'checkout', component: CheckoutComponent },
    ],
  },
];

// app routes
{
  path: 'shop',
  loadChildren: () => import('./shopping/routes').then(m => m.SHOPPING_ROUTES),
}
```

`BrowseComponent`, `CartComponent`, and `CheckoutComponent` all share the same `CartService` instance. When the user navigates away from `/shop`, the service is destroyed and the cart is emptied. If the cart needs to persist, move it to `providedIn: 'root'`.

### Theme Override Through the Injector Hierarchy

```typescript
@Injectable({ providedIn: 'root' })
export class ThemeService {
  primaryColor = signal('#1976d2');
  mode = signal<'light' | 'dark'>('light');
}

@Component({
  selector: 'app-dark-section',
  standalone: true,
  providers: [
    {
      provide: ThemeService,
      useFactory: () => {
        const theme = new ThemeService();
        theme.mode.set('dark');
        theme.primaryColor.set('#bb86fc');
        return theme;
      },
    },
  ],
  template: `<ng-content />`,
  host: {
    '[class.dark]': 'true',
  },
})
export class DarkSectionComponent {}
```

```html
<app-header />  <!-- uses root ThemeService (light) -->

<app-dark-section>
  <app-sidebar /> <!-- uses DarkSection's ThemeService (dark) -->
  <app-content /> <!-- uses DarkSection's ThemeService (dark) -->
</app-dark-section>

<app-footer /> <!-- uses root ThemeService (light) -->
```

Components inside `<app-dark-section>` automatically receive the dark theme. Components outside receive the root light theme. No prop drilling, no context APIs. The injector hierarchy handles scoping.

---

> **Key Takeaway:** Angular resolves dependencies by first walking up the ElementInjector tree (component hierarchy), then falling back to the EnvironmentInjector tree (application and route providers). Place services at the right level in these hierarchies to control whether they are singletons, feature-scoped, or per-component instances.
