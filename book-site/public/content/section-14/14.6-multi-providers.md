# 14.6 Multi Providers

> Multi providers let you register multiple values under the same token, collecting them into an array at injection time. This powers extensible systems like initializers, interceptors, validators, and plugin architectures.

---

## The "Last Wins" Problem

By default, when two providers use the same token, the last one silently replaces the first:

```typescript
providers: [
  { provide: Logger, useClass: ConsoleLogger },
  { provide: Logger, useClass: FileLogger },   // this wins, ConsoleLogger is gone
]

// inject(Logger) returns FileLogger
```

There is no warning. `ConsoleLogger` is silently dropped. This is Angular's normal behavior for single-value tokens.

---

## `multi: true` Collects All Values

Adding `multi: true` to a provider tells Angular to collect all providers for that token into an array instead of replacing previous ones:

```typescript
export const VALIDATOR = new InjectionToken<Validator>('VALIDATOR');

providers: [
  { provide: VALIDATOR, useClass: RequiredValidator, multi: true },
  { provide: VALIDATOR, useClass: EmailValidator, multi: true },
  { provide: VALIDATOR, useClass: MinLengthValidator, multi: true },
]

// inject(VALIDATOR) returns [RequiredValidator, EmailValidator, MinLengthValidator]
```

Every provider marked `multi: true` for the same token contributes to the array. The injection result is always an array, even with a single provider.

---

## Rules for `multi: true`

### All Providers Must Agree

Either ALL providers for a token use `multi: true`, or NONE of them do. Mixing causes the Angular error `NG0209`:

```typescript
// Bad Practice: mixing multi and non-multi
providers: [
  { provide: VALIDATOR, useClass: RequiredValidator, multi: true },
  { provide: VALIDATOR, useClass: EmailValidator },  // ERROR: NG0209
]
```

### Order Matters

The array preserves registration order. Providers listed first appear first in the array:

```typescript
providers: [
  { provide: HOOK, useValue: 'first', multi: true },
  { provide: HOOK, useValue: 'second', multi: true },
  { provide: HOOK, useValue: 'third', multi: true },
]

// inject(HOOK) returns ['first', 'second', 'third']
```

### All Provider Strategies Work

You can use `useClass`, `useValue`, `useFactory`, and `useExisting` with `multi: true`:

```typescript
providers: [
  { provide: PLUGIN, useClass: AnalyticsPlugin, multi: true },
  { provide: PLUGIN, useValue: { name: 'static-plugin', init: () => {} }, multi: true },
  { provide: PLUGIN, useFactory: () => new CustomPlugin(inject(HttpClient)), multi: true },
  { provide: PLUGIN, useExisting: LegacyPlugin, multi: true },
]
```

### Component-Level Multi Providers Do Not Merge with Parent

When a component provides a multi-token, its array replaces (does not extend) the parent's array for that token within that component's scope:

```typescript
// Root providers
providers: [
  { provide: VALIDATOR, useClass: RequiredValidator, multi: true },
  { provide: VALIDATOR, useClass: EmailValidator, multi: true },
]
// inject(VALIDATOR) at root returns [RequiredValidator, EmailValidator]

// Component overrides the entire array
@Component({
  providers: [
    { provide: VALIDATOR, useClass: PatternValidator, multi: true },
  ],
})
export class SpecialFormComponent {
  // inject(VALIDATOR) returns [PatternValidator] ONLY
  // RequiredValidator and EmailValidator are NOT included
}
```

This is a critical gotcha. Component-level multi providers create a new array, not an extension of the parent array.

## Built-in Multi-Provider Tokens

Angular uses multi providers internally for several extensibility points.

### `APP_INITIALIZER` (Deprecated) and `provideAppInitializer`

`APP_INITIALIZER` runs functions before the application renders. If any function returns a `Promise` or `Observable`, Angular waits for it to resolve before completing bootstrapping.

**Legacy pattern (deprecated in Angular 19):**

```typescript
// Deprecated: APP_INITIALIZER with multi: true
function loadConfig(http: HttpClient, configService: ConfigService) {
  return () => http.get('/api/config').pipe(
    tap(config => configService.setConfig(config)),
  );
}

providers: [
  {
    provide: APP_INITIALIZER,
    useFactory: loadConfig,
    deps: [HttpClient, ConfigService],
    multi: true,
  },
]
```

**Modern pattern: `provideAppInitializer`**

```typescript
import { provideAppInitializer } from '@angular/core';

providers: [
  provideAppInitializer(() => {
    const http = inject(HttpClient);
    const configService = inject(ConfigService);
    return firstValueFrom(
      http.get<AppConfig>('/api/config').pipe(
        tap(config => configService.setConfig(config)),
      ),
    );
  }),
]
```

`provideAppInitializer` internally uses `multi: true`. You can call it multiple times to register multiple initializers:

```typescript
providers: [
  provideAppInitializer(() => {
    // Load configuration
    const http = inject(HttpClient);
    return firstValueFrom(http.get('/api/config'));
  }),
  provideAppInitializer(() => {
    // Initialize analytics
    const analytics = inject(AnalyticsService);
    analytics.initialize();
  }),
]
```

All initializers run concurrently. Angular waits for all returned Promises/Observables before rendering.

### `ENVIRONMENT_INITIALIZER` (Deprecated) and `provideEnvironmentInitializer`

Runs when an EnvironmentInjector is created. Unlike `APP_INITIALIZER`, it is non-blocking (Angular does not wait for it) and runs at the injector level, not just at app bootstrap.

**Legacy pattern:**

```typescript
// Deprecated
providers: [
  {
    provide: ENVIRONMENT_INITIALIZER,
    multi: true,
    useValue: () => {
      const logger = inject(LoggerService);
      logger.log('Environment injector created');
    },
  },
]
```

**Modern pattern:**

```typescript
import { provideEnvironmentInitializer } from '@angular/core';

providers: [
  provideEnvironmentInitializer(() => {
    const logger = inject(LoggerService);
    logger.log('Environment injector created');
  }),
]
```

This is useful for route-level initialization. When a lazy-loaded route creates a new EnvironmentInjector, any `provideEnvironmentInitializer` in that route's providers runs automatically.

```typescript
export const adminRoutes: Routes = [
  {
    path: '',
    providers: [
      AdminService,
      provideEnvironmentInitializer(() => {
        const admin = inject(AdminService);
        admin.loadPermissions(); // runs when /admin route is activated
      }),
    ],
    children: [ ... ],
  },
];
```

### HTTP Interceptors

Angular's HTTP client uses a multi-provider system for interceptors. The modern functional interceptor API wraps this:

```typescript
import { provideHttpClient, withInterceptors } from '@angular/common/http';

export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(
      withInterceptors([
        loggingInterceptor,
        authInterceptor,
        retryInterceptor,
      ]),
    ),
  ],
};
```

Under the hood, `withInterceptors` registers each function as a multi provider. The order in the array determines the execution order.

## Creating Your Own Multi-Provider Token

### Step 1: Define the Token

```typescript
import { InjectionToken } from '@angular/core';

export interface FormValidator {
  name: string;
  validate(value: unknown): string | null; // returns error message or null
}

export const FORM_VALIDATOR = new InjectionToken<FormValidator>('FORM_VALIDATOR');
```

### Step 2: Register Multiple Providers

```typescript
providers: [
  {
    provide: FORM_VALIDATOR,
    multi: true,
    useValue: {
      name: 'required',
      validate: (value: unknown) => value ? null : 'Field is required',
    },
  },
  {
    provide: FORM_VALIDATOR,
    multi: true,
    useValue: {
      name: 'minLength',
      validate: (value: unknown) =>
        typeof value === 'string' && value.length >= 3 ? null : 'Minimum 3 characters',
    },
  },
]
```

### Step 3: Inject and Use the Array

```typescript
@Injectable({ providedIn: 'root' })
export class ValidationService {
  private validators = inject(FORM_VALIDATOR, { optional: true }) ?? [];

  validate(value: unknown): string[] {
    return this.validators
      .map(v => v.validate(value))
      .filter((msg): msg is string => msg !== null);
  }
}
```

### Step 4: Let Consumers Extend

```typescript
// A feature module adds its own validator
export const featureRoutes: Routes = [
  {
    path: 'checkout',
    providers: [
      {
        provide: FORM_VALIDATOR,
        multi: true,
        useValue: {
          name: 'creditCard',
          validate: (value: unknown) =>
            typeof value === 'string' && /^\d{16}$/.test(value) ? null : 'Invalid card number',
        },
      },
    ],
    component: CheckoutComponent,
  },
];
```

Note: this route-level provider creates a new array containing only the `creditCard` validator. It does NOT include the root-level `required` and `minLength` validators within that route's scope.

## `provide` Helper Functions for Multi Tokens

Follow Angular's convention of exporting `provideXxx` functions:

```typescript
export function provideValidator(validator: FormValidator): Provider {
  return {
    provide: FORM_VALIDATOR,
    useValue: validator,
    multi: true,
  };
}

export function provideValidatorFactory(factory: () => FormValidator): Provider {
  return {
    provide: FORM_VALIDATOR,
    useFactory: factory,
    multi: true,
  };
}

// Consumer usage
providers: [
  provideValidator({
    name: 'required',
    validate: (v) => v ? null : 'Required',
  }),
  provideValidator({
    name: 'email',
    validate: (v) => typeof v === 'string' && v.includes('@') ? null : 'Invalid email',
  }),
  provideValidatorFactory(() => {
    const config = inject(APP_CONFIG);
    return {
      name: 'maxLength',
      validate: (v) =>
        typeof v === 'string' && v.length <= config.maxFieldLength
          ? null
          : `Maximum ${config.maxFieldLength} characters`,
    };
  }),
]
```

## Multi Providers Across the Injector Hierarchy

Understanding how multi providers interact with the hierarchy is essential.

### Root Level

```typescript
// app.config.ts
providers: [
  { provide: PLUGIN, useClass: CorePlugin, multi: true },
  { provide: PLUGIN, useClass: AnalyticsPlugin, multi: true },
]
// inject(PLUGIN) anywhere returns [CorePlugin, AnalyticsPlugin]
```

### Route Level Replaces Root

```typescript
// admin.routes.ts
{
  path: 'admin',
  providers: [
    { provide: PLUGIN, useClass: AdminPlugin, multi: true },
  ],
  children: [ ... ],
}
// inject(PLUGIN) inside /admin returns [AdminPlugin] only
// CorePlugin and AnalyticsPlugin are NOT included
```

### Component Level Replaces Everything Above

```typescript
@Component({
  providers: [
    { provide: PLUGIN, useClass: LocalPlugin, multi: true },
  ],
})
export class SpecialComponent {
  // inject(PLUGIN) returns [LocalPlugin] only
}
```

This replacement behavior is by design. If you need additive behavior, inject the parent's array and combine:

```typescript
@Component({
  providers: [
    {
      provide: PLUGIN,
      useFactory: () => {
        const parentPlugins = inject(PLUGIN, { skipSelf: true, optional: true }) ?? [];
        return [...parentPlugins, new LocalPlugin()];
      },
    },
    // Note: this is NOT a multi provider. It replaces the token entirely.
  ],
})
export class AdditiveComponent {
  private plugins = inject(PLUGIN); // combined parent + local
}
```

This is advanced and fragile. Prefer structuring your application so that multi providers are registered at a single level.

## Typed Multi-Provider Tokens

When defining a multi-provider token, the generic type should be the single item type, not an array:

```typescript
// Correct: generic is the ITEM type
export const HOOK = new InjectionToken<LifecycleHook>('HOOK');

// When injected with multi: true, Angular returns LifecycleHook[]
const hooks = inject(HOOK); // type: LifecycleHook[]
```

Do not define the token as `InjectionToken<LifecycleHook[]>` because Angular wraps multi results in an array automatically. Defining it as an array would result in `LifecycleHook[][]` at injection.

---

## Common Mistakes

- **Forgetting `multi: true` on one of the providers.** If any provider for the token omits `multi: true`, Angular throws `NG0209`. Every provider for a multi token must consistently include `multi: true`.

```typescript
// Bad Practice: inconsistent multi flag
providers: [
  { provide: HOOK, useValue: hookA, multi: true },
  { provide: HOOK, useValue: hookB },  // ERROR: NG0209
]
```

- **Expecting component-level multi providers to merge with parent providers.** They do not merge. A component-level multi provider creates a fresh array containing only its own entries. The parent's entries are not included.

- **Using the deprecated `APP_INITIALIZER` / `ENVIRONMENT_INITIALIZER` tokens.** Since Angular 19, use `provideAppInitializer()` and `provideEnvironmentInitializer()` instead. The old tokens still work but produce deprecation warnings.

- **Defining the `InjectionToken` generic as an array type.** `new InjectionToken<Plugin[]>('PLUGIN')` is wrong for multi providers. Define it as `new InjectionToken<Plugin>('PLUGIN')`. Angular wraps the result in an array when `multi: true` is used.

- **Assuming multi-provider order matches module import order.** The array order follows the order providers are listed in the `providers` array. When providers come from different sources (imported modules, route configs), the order depends on how Angular flattens them. If order matters, be explicit about the registration sequence.

- **Not handling the case where no provider exists.** If no `multi: true` provider is registered for a token, `inject(TOKEN)` throws `NullInjectorError`. Use `inject(TOKEN, { optional: true })` to get `null` instead, and default to an empty array.

```typescript
private plugins = inject(PLUGIN, { optional: true }) ?? [];
```

---

## Best Practices

- **Export `provideXxx()` helper functions.** This hides the `multi: true` detail from consumers and follows Angular conventions.

- **Use `provideAppInitializer()` for startup tasks.** Return a Promise or Observable if the app should wait for initialization to complete.

- **Use `provideEnvironmentInitializer()` for route-level setup.** Lazy-loaded routes get their own EnvironmentInjector, so initializers scoped to routes run only when the route activates.

- **Keep multi-provider arrays at a single level when possible.** The non-additive behavior across hierarchy levels is a common source of bugs. If all providers are registered at the root, the array is predictable.

- **Always use `{ optional: true }` with `?? []` when injecting multi tokens.** This prevents crashes when no provider is registered and provides a sensible default.

- **Document the expected interface for multi-provider consumers.** Since multiple independent parts of the application contribute to the array, clear documentation prevents mismatched contracts.

## Real-World Use Cases

### Plugin System

```typescript
export interface AppPlugin {
  name: string;
  priority: number;
  initialize(): void | Promise<void>;
  destroy?(): void;
}

export const APP_PLUGIN = new InjectionToken<AppPlugin>('APP_PLUGIN');

export function providePlugin(plugin: AppPlugin): Provider {
  return { provide: APP_PLUGIN, useValue: plugin, multi: true };
}

export function providePluginFactory(factory: () => AppPlugin): Provider {
  return { provide: APP_PLUGIN, useFactory: factory, multi: true };
}

// app.config.ts
providers: [
  providePlugin({
    name: 'analytics',
    priority: 1,
    initialize() {
      console.log('Analytics initialized');
    },
  }),
  providePluginFactory(() => {
    const http = inject(HttpClient);
    return {
      name: 'remote-config',
      priority: 0,
      async initialize() {
        const config = await firstValueFrom(http.get('/api/remote-config'));
        console.log('Remote config loaded', config);
      },
    };
  }),
]

// Plugin runner
@Injectable({ providedIn: 'root' })
export class PluginRunner {
  private plugins = inject(APP_PLUGIN, { optional: true }) ?? [];
  private destroyRef = inject(DestroyRef);

  async initializeAll() {
    const sorted = [...this.plugins].sort((a, b) => a.priority - b.priority);
    for (const plugin of sorted) {
      await plugin.initialize();
    }
  }

  constructor() {
    this.destroyRef.onDestroy(() => {
      this.plugins.forEach(p => p.destroy?.());
    });
  }
}

// Bootstrap with app initializer
providers: [
  provideAppInitializer(() => {
    const runner = inject(PluginRunner);
    return runner.initializeAll();
  }),
]
```

### Extensible Validation System

```typescript
export type ValidatorFn = (value: unknown) => string | null;

export const FIELD_VALIDATOR = new InjectionToken<ValidatorFn>('FIELD_VALIDATOR');

export function provideFieldValidator(fn: ValidatorFn): Provider {
  return { provide: FIELD_VALIDATOR, useValue: fn, multi: true };
}

// Core validators
export function provideCoreValidators(): Provider[] {
  return [
    provideFieldValidator((v) => (v == null || v === '') ? 'Required' : null),
    provideFieldValidator((v) =>
      typeof v === 'string' && v.length > 255 ? 'Max 255 characters' : null
    ),
  ];
}

// Email-specific validator added at feature level
export function provideEmailValidator(): Provider {
  return provideFieldValidator((v) =>
    typeof v === 'string' && /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(v)
      ? null
      : 'Invalid email address'
  );
}

// Usage
@Component({
  selector: 'app-email-field',
  standalone: true,
  providers: [
    // In this component, only email validation applies
    provideEmailValidator(),
  ],
  template: `
    <input [value]="value()" (input)="onInput($event)" />
    @for (error of errors(); track error) {
      <p class="error">{{ error }}</p>
    }
  `,
})
export class EmailFieldComponent {
  private validators = inject(FIELD_VALIDATOR, { optional: true }) ?? [];

  value = signal('');
  errors = computed(() =>
    this.validators
      .map(fn => fn(this.value()))
      .filter((msg): msg is string => msg !== null)
  );

  onInput(event: Event) {
    this.value.set((event.target as HTMLInputElement).value);
  }
}
```

### Application Startup with Multiple Initializers

```typescript
// config.initializer.ts
export function provideConfigInitializer(): Provider {
  return provideAppInitializer(() => {
    const http = inject(HttpClient);
    const configService = inject(ConfigService);
    return firstValueFrom(
      http.get<AppConfig>('/api/config').pipe(
        tap(config => configService.setConfig(config)),
      ),
    );
  });
}

// auth.initializer.ts
export function provideAuthInitializer(): Provider {
  return provideAppInitializer(() => {
    const auth = inject(AuthService);
    return auth.tryRestoreSession(); // returns Promise<void>
  });
}

// i18n.initializer.ts
export function provideI18nInitializer(): Provider {
  return provideAppInitializer(() => {
    const i18n = inject(I18nService);
    const config = inject(ConfigService);
    return i18n.loadLocale(config.defaultLocale); // returns Promise<void>
  });
}

// app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(),
    provideRouter(routes),
    provideConfigInitializer(),
    provideAuthInitializer(),
    provideI18nInitializer(),
  ],
};
```

All three initializers run concurrently during bootstrapping. The application does not render until all three complete. The order in the array does not determine execution order since they run in parallel, but the config initializer should complete before i18n if i18n depends on `ConfigService`. In that case, handle the dependency within the initializer itself (e.g., await config loading before loading locale data) rather than relying on array ordering.

---

> **Key Takeaway:** Use `multi: true` to collect multiple values under one token into an array, enabling extensible patterns like plugins, validators, and initializers; always use `provideAppInitializer()` and `provideEnvironmentInitializer()` instead of the deprecated `APP_INITIALIZER`/`ENVIRONMENT_INITIALIZER` tokens; and remember that component-level multi providers replace (not extend) parent arrays.
