# 14.7 DI Resolution Modifiers

> Resolution modifiers let you control where Angular looks for a dependency, where it stops looking, and what happens when it cannot find one.

---

## The Default Resolution Behavior

Without any modifiers, Angular resolves a dependency using a two-phase search:

1. **ElementInjector phase** -- starts at the current component or directive and walks up through parent components in the template tree.
2. **EnvironmentInjector phase** -- if no ElementInjector provides the token, Angular checks the environment injector hierarchy (route injectors, root injector, platform injector).
3. **NullInjector** -- if still unresolved, the `NullInjector` throws a `NullInjectorError`.

```
Current Element -> Parent Element -> ... -> Root Element
                                                |
                                         EnvironmentInjector (route)
                                                |
                                         EnvironmentInjector (root)
                                                |
                                         PlatformInjector
                                                |
                                          NullInjector (throws)
```

Resolution modifiers alter this default flow.

---

## The Four Modifiers

Angular provides four resolution modifiers. Each can be used as a parameter decorator (legacy constructor injection) or as an option in the `inject()` function.

| Modifier | Decorator | `inject()` Option | Effect |
|----------|-----------|-------------------|--------|
| Optional | `@Optional()` | `{ optional: true }` | Returns `null` instead of throwing when not found |
| Self | `@Self()` | `{ self: true }` | Only checks the current ElementInjector |
| SkipSelf | `@SkipSelf()` | `{ skipSelf: true }` | Skips the current ElementInjector, starts from parent |
| Host | `@Host()` | `{ host: true }` | Stops at the host element boundary |

---

## Optional

The `optional` modifier prevents Angular from throwing when a dependency cannot be found. Instead, it injects `null`.

### With `inject()`

```typescript
import { Component, inject } from '@angular/core';
import { AnalyticsService } from './analytics.service';

@Component({
  selector: 'app-dashboard',
  template: `<h1>Dashboard</h1>`,
})
export class DashboardComponent {
  // Returns null if AnalyticsService is not provided anywhere
  private analytics = inject(AnalyticsService, { optional: true });

  trackEvent(name: string) {
    // Must null-check since the service might not exist
    this.analytics?.track(name);
  }
}
```

### With Constructor Injection (Legacy)

```typescript
import { Component, Optional } from '@angular/core';
import { AnalyticsService } from './analytics.service';

@Component({
  selector: 'app-dashboard',
  template: `<h1>Dashboard</h1>`,
})
export class DashboardComponent {
  constructor(@Optional() private analytics: AnalyticsService | null) {}

  trackEvent(name: string) {
    this.analytics?.track(name);
  }
}
```

### When to Use Optional

- **Plugin systems**: A component checks for an optional feature service that may or may not be provided by a parent.
- **Library components**: A component in a shared library that can integrate with an app-level service if present, but works without it.
- **Graceful degradation**: A logging or analytics service that should not crash the app if missing.

### The Null Check Trap

```typescript
// BAD: Forgetting that optional injection returns null
private logger = inject(LoggerService, { optional: true });

ngOnInit() {
  // TypeError: Cannot read properties of null
  this.logger.log('initialized');
}

// GOOD: Always null-check optional dependencies
ngOnInit() {
  this.logger?.log('initialized');
}
```

## Self

The `self` modifier restricts the search to the current component's or directive's own `providers` array. Angular will not walk up the injector tree.

### With `inject()`

```typescript
@Component({
  selector: 'app-panel',
  providers: [PanelStateService],
  template: `<p>Panel content</p>`,
})
export class PanelComponent {
  // Only resolves if PanelStateService is in THIS component's providers
  private state = inject(PanelStateService, { self: true });
}
```

### What Happens Without a Local Provider

```typescript
@Component({
  selector: 'app-child',
  // No providers array -- PanelStateService is NOT provided here
  template: `<p>Child</p>`,
})
export class ChildComponent {
  // Throws NullInjectorError even if a parent provides PanelStateService
  private state = inject(PanelStateService, { self: true });
}
```

To avoid the error, combine with `optional`:

```typescript
private state = inject(PanelStateService, { self: true, optional: true });
// Returns null if not provided locally
```

### When to Use Self

- **Enforcing local scope**: When a component must have its own instance and should never accidentally inherit a parent's instance.
- **Directive isolation**: When a directive requires a provider to be on the same element.

### Self with Directives

Directives frequently use `self` to ensure they only interact with providers on their own host element:

```typescript
@Directive({
  selector: '[appTooltip]',
  providers: [TooltipPositionService],
})
export class TooltipDirective {
  // Only resolves the TooltipPositionService provided on this directive's element
  private position = inject(TooltipPositionService, { self: true });
}
```

## SkipSelf

The `skipSelf` modifier is the opposite of `self`. It tells Angular to skip the current ElementInjector and start the search from the parent.

### With `inject()`

```typescript
@Component({
  selector: 'app-child',
  providers: [{ provide: CounterService, useFactory: () => new CounterService('child') }],
  template: `<p>Child counter</p>`,
})
export class ChildComponent {
  // Skips the child's own CounterService, resolves from parent
  private parentCounter = inject(CounterService, { skipSelf: true });
}
```

### The Classic Use Case: Nested Tree Structures

`skipSelf` is essential when building recursive or hierarchical components like tree views, nested menus, or accordion panels:

```typescript
@Component({
  selector: 'app-tree-node',
  providers: [TreeNodeService],
  template: `
    <div class="node">
      {{ node.label }}
      @for (child of node.children; track child.id) {
        <app-tree-node [node]="child" />
      }
    </div>
  `,
})
export class TreeNodeComponent {
  node = input.required<TreeNode>();

  // This node's own service (provided locally)
  private self = inject(TreeNodeService);

  // The parent node's service (skipping this node's provider)
  private parent = inject(TreeNodeService, { skipSelf: true, optional: true });

  ngOnInit() {
    if (this.parent) {
      this.parent.registerChild(this.self);
    }
  }
}
```

The root node has no parent, so `optional: true` prevents the error at the top of the tree.

### SkipSelf for Provider Overriding

A common pattern is providing a default service at a higher level and letting child components override it while still accessing the parent version:

```typescript
// Parent provides a "base" theme
@Component({
  selector: 'app-page',
  providers: [{ provide: ThemeService, useValue: { color: 'blue' } }],
  template: `<app-card />`,
})
export class PageComponent {}

// Child overrides the theme but can still reference the parent's
@Component({
  selector: 'app-card',
  providers: [{ provide: ThemeService, useValue: { color: 'red' } }],
  template: `<p>Card</p>`,
})
export class CardComponent {
  // Gets { color: 'red' } -- the local provider
  private theme = inject(ThemeService);

  // Gets { color: 'blue' } -- the parent's provider
  private parentTheme = inject(ThemeService, { skipSelf: true });
}
```

## Host

The `host` modifier is the most nuanced. It stops the ElementInjector search at the host element boundary. The behavior depends on where the injection happens:

- **In a component**: `host` restricts to the component's own ElementInjector (behaves like `self`).
- **In a directive**: `host` searches the directive's element injector and then checks the host component's `viewProviders` (but NOT `providers`).

### Understanding the Host Boundary

The "host" of a directive is the component whose template contains the element the directive is applied to. The key insight: `host` allows a directive to access the host component's `viewProviders` but not its `providers`.

```typescript
@Component({
  selector: 'app-container',
  // viewProviders are accessible via { host: true } from child directives
  viewProviders: [ContainerService],
  template: `<div appChild></div>`,
})
export class ContainerComponent {}

@Directive({
  selector: '[appChild]',
})
export class ChildDirective {
  // Resolves ContainerService from the host's viewProviders
  private container = inject(ContainerService, { host: true });
}
```

If `ContainerService` were in `providers` instead of `viewProviders`, the `{ host: true }` injection in the directive would fail with a `NullInjectorError`.

### The viewProviders Distinction

This `providers` vs `viewProviders` distinction with `host` is a critical concept:

```typescript
@Component({
  selector: 'app-form-field',
  // Providers -- NOT visible to directives via { host: true }
  providers: [ValidationService],
  // viewProviders -- visible to directives via { host: true }
  viewProviders: [FormFieldService],
  template: `<input appInput />`,
})
export class FormFieldComponent {}

@Directive({
  selector: '[appInput]',
})
export class InputDirective {
  // Works -- FormFieldService is in viewProviders
  private formField = inject(FormFieldService, { host: true });

  // Fails -- ValidationService is in providers, not viewProviders
  private validation = inject(ValidationService, { host: true });
}
```

### Host in Angular's Own Codebase

Angular's forms module uses `host` extensively. `NgModel` uses `@Host()` to locate its parent `NgForm` or `FormGroupDirective`:

```typescript
// Simplified from Angular source
@Directive({ selector: '[ngModel]' })
export class NgModel {
  constructor(
    @Optional() @Host() parent: ControlContainer
  ) {
    // parent is the NgForm or FormGroupDirective on an ancestor element
    // @Host ensures ngModel only finds forms within the same view
  }
}
```

This prevents an `ngModel` inside a nested component from accidentally connecting to a form in a parent component's template.

### Host vs Self

| Scenario | `self` | `host` |
|----------|--------|--------|
| Component injecting from its own providers | Yes | Yes |
| Directive injecting from its own providers | Yes | Yes |
| Directive injecting from host's viewProviders | No | Yes |
| Directive injecting from host's providers | No | No |
| Searching parent components | No | No |

In a component (not a directive), `host` and `self` produce identical results because the component IS the host.

## Combining Modifiers

Certain combinations are common and useful. Others are logically contradictory.

### Valid Combinations

#### `optional` + `self`

"Look only here. If not found, return null."

```typescript
private logger = inject(LoggerService, { self: true, optional: true });
```

#### `optional` + `skipSelf`

"Look only in ancestors. If not found, return null."

```typescript
private parentLogger = inject(LoggerService, { skipSelf: true, optional: true });
```

#### `optional` + `host`

"Look up to the host boundary. If not found, return null."

```typescript
private container = inject(ContainerService, { host: true, optional: true });
```

#### `skipSelf` + `host`

"Start from the parent, but stop at the host boundary."

```typescript
// In a directive applied to an element inside a host component
private service = inject(SomeService, { skipSelf: true, host: true });
```

#### `skipSelf` + `optional`

The most common combination. Used in recursive structures where the root has no parent:

```typescript
private parentNode = inject(TreeNodeComponent, { skipSelf: true, optional: true });
```

### Invalid or Contradictory Combinations

#### `self` + `skipSelf`

"Look only here" + "Skip here" is a contradiction. Angular throws:

```typescript
// ERROR: Cannot use both self and skipSelf
private service = inject(MyService, { self: true, skipSelf: true });
```

#### `self` + `host`

While technically not always an error at runtime, these are redundant in components and potentially contradictory in directives. Avoid this combination.

## inject() Options vs Parameter Decorators

The `inject()` function with options is the modern approach. Parameter decorators are the legacy approach. Both achieve the same resolution behavior, but `inject()` has advantages.

### Syntax Comparison

```typescript
// Modern: inject() with options
@Component({ /* ... */ })
export class ModernComponent {
  private required = inject(MyService);
  private optional = inject(MyService, { optional: true });
  private localOnly = inject(MyService, { self: true });
  private fromParent = inject(MyService, { skipSelf: true });
  private hostBound = inject(MyService, { host: true });
  private combined = inject(MyService, { skipSelf: true, optional: true });
}

// Legacy: parameter decorators
@Component({ /* ... */ })
export class LegacyComponent {
  constructor(
    private required: MyService,
    @Optional() private optional: MyService | null,
    @Self() private localOnly: MyService,
    @SkipSelf() private fromParent: MyService,
    @Host() private hostBound: MyService,
    @SkipSelf() @Optional() private combined: MyService | null,
  ) {}
}
```

### Why Prefer inject()

| Aspect | `inject()` | Parameter Decorators |
|--------|-----------|---------------------|
| Works in standalone functions | Yes (guards, interceptors, etc.) | No |
| TypeScript ES decorators compatible | Yes | No (parameter decorators not in TC39 spec) |
| Readable at declaration site | Yes (inline options) | Moderate (stacked decorators can be noisy) |
| Testable | Same | Same |
| Tree-shakable | Same | Same |
| Future-proof | Yes | Uncertain (TC39 has no parameter decorator proposal) |

### Migration Path

Converting from decorators to `inject()` is straightforward:

```typescript
// Before
constructor(
  @Optional() @SkipSelf() private parentMenu: MenuComponent | null,
  @Self() private menuState: MenuStateService,
) {}

// After
private parentMenu = inject(MenuComponent, { skipSelf: true, optional: true });
private menuState = inject(MenuStateService, { self: true });
```

Note the parameter order does not matter in the options object. `{ skipSelf: true, optional: true }` and `{ optional: true, skipSelf: true }` are identical.

## Resolution Flow Diagrams

### Default (No Modifiers)

```
inject(ServiceA)

[Current Element] -> found? return
         |
  [Parent Element] -> found? return
         |
  [Grandparent...] -> found? return
         |
  [Root Element]   -> found? return
         |
[EnvironmentInjector (route)] -> found? return
         |
[EnvironmentInjector (root)]  -> found? return
         |
[PlatformInjector]            -> found? return
         |
[NullInjector]                -> throw NullInjectorError
```

### With `{ self: true }`

```
inject(ServiceA, { self: true })

[Current Element] -> found? return
         |
       not found -> throw NullInjectorError
```

### With `{ skipSelf: true }`

```
inject(ServiceA, { skipSelf: true })

[Current Element] -> SKIPPED
         |
  [Parent Element] -> found? return
         |
  [Grandparent...] -> found? return
         |
       ... continues normally to NullInjector
```

### With `{ host: true }` (from a directive)

```
inject(ServiceA, { host: true })

[Directive Element] -> found? return
         |
[Host Component viewProviders] -> found? return
         |
       not found -> throw NullInjectorError
```

### With `{ optional: true }`

```
inject(ServiceA, { optional: true })

... same search path as default ...
         |
[NullInjector] -> return null (instead of throwing)
```

## Practical Patterns

### Pattern 1: Optional Parent Registration

A child component that registers itself with a parent if the parent exists:

```typescript
@Component({
  selector: 'app-tab',
  template: `<ng-content />`,
})
export class TabComponent implements OnInit, OnDestroy {
  private tabGroup = inject(TabGroupComponent, { skipSelf: true, optional: true });

  label = input.required<string>();

  ngOnInit() {
    this.tabGroup?.addTab(this);
  }

  ngOnDestroy() {
    this.tabGroup?.removeTab(this);
  }
}
```

### Pattern 2: Self-Contained Component with Isolated State

A component that must always create its own service instance:

```typescript
@Component({
  selector: 'app-modal',
  providers: [ModalStateService],
  template: `
    @if (state.isOpen()) {
      <div class="overlay">
        <div class="modal-content">
          <ng-content />
        </div>
      </div>
    }
  `,
})
export class ModalComponent {
  // Guaranteed to be this modal's own instance, never a parent's
  protected state = inject(ModalStateService, { self: true });
}
```

### Pattern 3: Directive Accessing Host Component

A directive that enhances its host component with additional behavior:

```typescript
@Directive({
  selector: '[appAutoFocus]',
})
export class AutoFocusDirective implements AfterViewInit {
  private elementRef = inject(ElementRef);
  // Access the host component's form field service if available
  private formField = inject(FormFieldService, { host: true, optional: true });

  ngAfterViewInit() {
    this.elementRef.nativeElement.focus();
    this.formField?.markAsTouched();
  }
}
```

### Pattern 4: Layered Configuration with SkipSelf

Building a configuration system where child components extend parent configuration:

```typescript
import { InjectionToken, inject } from '@angular/core';

export const CONFIG = new InjectionToken<Record<string, unknown>>('CONFIG');

@Component({
  selector: 'app-section',
  providers: [
    {
      provide: CONFIG,
      useFactory: () => {
        const parentConfig = inject(CONFIG, { skipSelf: true, optional: true });
        return {
          ...parentConfig,  // Inherit parent config
          sectionId: crypto.randomUUID(),  // Add section-specific config
        };
      },
    },
  ],
  template: `<ng-content />`,
})
export class SectionComponent {
  protected config = inject(CONFIG);
}
```

Each nested `<app-section>` inherits its parent's config and adds its own values.

### Pattern 5: Form Control Directive

Mimicking how Angular's own form directives use `host`:

```typescript
@Directive({
  selector: '[appFormControl]',
})
export class FormControlDirective implements OnInit {
  // Find the closest form group, but only within the host component's view
  private formGroup = inject(FormGroupDirective, { host: true, optional: true });
  private elementRef = inject(ElementRef);

  controlName = input.required<string>({ alias: 'appFormControl' });

  ngOnInit() {
    if (this.formGroup) {
      const control = this.formGroup.getControl(this.controlName());
      // Bind the element to the control...
    }
  }
}
```

## Common Mistakes

### Mistake 1: Using `self` When You Mean `skipSelf`

```typescript
// WRONG: "I want the parent's service, not my own"
private parentService = inject(MyService, { self: true });
// This gets YOUR service, not the parent's

// CORRECT
private parentService = inject(MyService, { skipSelf: true });
```

### Mistake 2: Forgetting `optional` With `skipSelf` in Recursive Components

```typescript
// WRONG: Crashes at the root node because there is no parent
@Component({
  selector: 'app-tree-node',
  providers: [NodeService],
  template: `...`,
})
export class TreeNodeComponent {
  private parent = inject(NodeService, { skipSelf: true }); // NullInjectorError at root!
}

// CORRECT: Root node gets null
private parent = inject(NodeService, { skipSelf: true, optional: true });
```

### Mistake 3: Expecting `host` to Check `providers`

```typescript
@Component({
  selector: 'app-wrapper',
  providers: [WrapperService], // In providers, NOT viewProviders
  template: `<div appChild></div>`,
})
export class WrapperComponent {}

@Directive({ selector: '[appChild]' })
export class ChildDirective {
  // FAILS: host does not check the host component's providers
  private wrapper = inject(WrapperService, { host: true });
}

// FIX: Move WrapperService to viewProviders
@Component({
  selector: 'app-wrapper',
  viewProviders: [WrapperService], // Now accessible via { host: true }
  template: `<div appChild></div>`,
})
export class WrapperComponent {}
```

### Mistake 4: Confusing `host` With `self` in Components

In a component (not a directive), `host` and `self` behave identically because the component is its own host. Using `host` in a component does not add any extra search scope:

```typescript
@Component({
  selector: 'app-panel',
  template: `...`,
})
export class PanelComponent {
  // These two lines produce identical behavior in a component
  private a = inject(MyService, { self: true });
  private b = inject(MyService, { host: true });
}
```

Use `self` in components for clarity. Reserve `host` for directives where the distinction matters.

### Mistake 5: Stacking Contradictory Decorators

```typescript
// WRONG: Logically contradictory
constructor(@Self() @SkipSelf() private service: MyService) {}

// Angular will throw an error at runtime
```

### Mistake 6: Assuming `optional` Changes the Search Path

`optional` does not change WHERE Angular looks. It only changes what happens when the search ends without finding a provider:

```typescript
// These search the same path, the only difference is the fallback
inject(MyService);                      // throws if not found
inject(MyService, { optional: true });  // returns null if not found
```

## Best Practices

### 1. Default to No Modifiers

Most injections need no modifiers at all. The default resolution (search up through all ancestors, then environment injectors) is correct for the vast majority of cases. Only add modifiers when you have a specific reason.

### 2. Use `inject()` Over Parameter Decorators

The `inject()` function is the modern API. It works in all injection contexts (components, directives, services, guards, interceptors, pipes), is compatible with TypeScript's standard decorator proposal, and keeps dependency declarations at the field level.

### 3. Pair `skipSelf` With `optional` in Recursive Structures

Any recursive component or directive that uses `skipSelf` to find its parent should include `optional` to handle the root case gracefully.

### 4. Use `self` to Enforce Isolation

When a component must have its own service instance and should never inherit from a parent, use `{ self: true }`. This makes the architectural intent explicit and catches configuration errors early.

### 5. Prefer `viewProviders` When Using `host` in Directives

If you design a component that provides services for child directives to consume via `{ host: true }`, use `viewProviders` instead of `providers`. This is a deliberate design choice in Angular's DI system to control visibility boundaries.

### 6. Document Non-Obvious Modifier Usage

When you use a modifier for architectural reasons, add a brief comment:

```typescript
// skipSelf: get the parent accordion's service, not this panel's own
private accordion = inject(AccordionService, { skipSelf: true, optional: true });
```

## Real-World Use Cases

### Nested Accordion Component

An accordion with expandable panels where each panel registers with its parent accordion:

```typescript
@Component({
  selector: 'app-accordion',
  providers: [AccordionService],
  template: `<ng-content />`,
})
export class AccordionComponent {
  private accordion = inject(AccordionService, { self: true });

  // The accordion service manages which panel is expanded
}

@Component({
  selector: 'app-accordion-panel',
  template: `
    <button (click)="toggle()">{{ title() }}</button>
    @if (isExpanded()) {
      <div class="panel-body">
        <ng-content />
      </div>
    }
  `,
})
export class AccordionPanelComponent implements OnInit, OnDestroy {
  title = input.required<string>();

  // Find the accordion service from the parent, not from self
  private accordion = inject(AccordionService, { skipSelf: true });

  isExpanded = signal(false);

  ngOnInit() {
    this.accordion.registerPanel(this);
  }

  ngOnDestroy() {
    this.accordion.unregisterPanel(this);
  }

  toggle() {
    this.accordion.togglePanel(this);
  }
}
```

### Scoped Error Handler

Different sections of an app provide different error handling strategies:

```typescript
export abstract class ErrorHandler {
  abstract handle(error: Error): void;
}

@Component({
  selector: 'app-checkout',
  providers: [
    {
      provide: ErrorHandler,
      useFactory: () => {
        const router = inject(Router);
        return {
          handle(error: Error) {
            console.error('Checkout error:', error);
            router.navigate(['/checkout/error']);
          },
        };
      },
    },
  ],
  template: `<router-outlet />`,
})
export class CheckoutComponent {}

@Component({
  selector: 'app-payment-form',
  template: `<form (submit)="submit()">...</form>`,
})
export class PaymentFormComponent {
  // Gets the checkout-specific error handler, not the root one
  // No modifier needed here -- default resolution finds the closest provider
  private errorHandler = inject(ErrorHandler);

  // But if we wanted to guarantee it comes from a parent scope:
  private scopedHandler = inject(ErrorHandler, { skipSelf: true, optional: true });
}
```

### Feature Flag Service with Layered Overrides

A feature flag system where different parts of the app can override flags:

```typescript
import { InjectionToken, inject, Component, signal, computed } from '@angular/core';

export const FEATURE_FLAGS = new InjectionToken<Record<string, boolean>>('FEATURE_FLAGS');

// Root flags
export function provideFeatureFlags(flags: Record<string, boolean>) {
  return { provide: FEATURE_FLAGS, useValue: flags };
}

// Component that overrides specific flags for its subtree
@Component({
  selector: 'app-beta-section',
  providers: [
    {
      provide: FEATURE_FLAGS,
      useFactory: () => {
        const parentFlags = inject(FEATURE_FLAGS, { skipSelf: true, optional: true }) ?? {};
        return {
          ...parentFlags,
          newCheckout: true,    // Override for this section
          betaBanner: true,
        };
      },
    },
  ],
  template: `<ng-content />`,
})
export class BetaSectionComponent {}

// Any child component gets the merged flags
@Component({
  selector: 'app-product-page',
  template: `
    @if (flags['betaBanner']) {
      <div class="beta-banner">Beta Feature</div>
    }
  `,
})
export class ProductPageComponent {
  protected flags = inject(FEATURE_FLAGS);
}
```

---

> **Key Takeaway:** Resolution modifiers give you precise control over Angular's dependency lookup. Use `optional` for graceful fallbacks, `self` for strict isolation, `skipSelf` for parent access in hierarchical structures, and `host` for directive-to-host-component communication. Prefer the `inject()` function with options over parameter decorators.
