# 14.8 Environment Injector and `runInInjectionContext`

> Environment injectors operate outside the component tree. `runInInjectionContext` lets you call `inject()` anywhere by temporarily establishing an injection context.

---

## The Two Injector Hierarchies

Angular maintains two parallel injector hierarchies:

| Hierarchy | Created By | Lifetime | Configured Via |
|-----------|-----------|----------|----------------|
| **ElementInjector** | Each component/directive instance | Component instance | `providers` / `viewProviders` on `@Component` / `@Directive` |
| **EnvironmentInjector** | Bootstrap, route config, or `createEnvironmentInjector` | Application or scope lifetime | `providedIn: 'root'`, `ApplicationConfig.providers`, route `providers` |

When Angular resolves a token, it searches the ElementInjector chain first, then falls back to the EnvironmentInjector chain. Environment injectors are the backbone of application-wide services, route-scoped services, and platform-level providers.

### The EnvironmentInjector Chain

```
NullInjector (throws)
       ^
PlatformInjector
       ^
Root EnvironmentInjector  (bootstrapApplication providers)
       ^
Route EnvironmentInjector (route-level providers)
       ^
Child Route EnvironmentInjector (child route providers)
```

Each level inherits from its parent. A token not found at the current level bubbles up.

---

## EnvironmentInjector API

The `EnvironmentInjector` class extends `Injector` and adds the ability to run code in an injection context.

### Getting a Reference

You can inject the `EnvironmentInjector` itself:

```typescript
import { Component, EnvironmentInjector, inject } from '@angular/core';

@Component({
  selector: 'app-root',
  template: `<router-outlet />`,
})
export class AppComponent {
  private envInjector = inject(EnvironmentInjector);
}
```

This gives you the closest `EnvironmentInjector` in the hierarchy. In a root component, this is the root environment injector. Inside a lazily loaded route, it is the route's environment injector.

### Core Methods

```typescript
abstract class EnvironmentInjector extends Injector {
  // Resolve a token
  abstract get<T>(token: ProviderToken<T>, notFoundValue?: T, flags?: InjectFlags): T;

  // Run a function in this injector's context (DEPRECATED -- use runInInjectionContext)
  abstract runInContext<ReturnT>(fn: () => ReturnT): ReturnT;

  // Destroy this injector and all services it created
  abstract destroy(): void;

  // Register a destroy callback
  abstract onDestroy(callback: () => void): () => void;
}
```

> **Important:** `runInContext` on the `EnvironmentInjector` instance is deprecated since Angular 16. Use the standalone `runInInjectionContext` function instead.

## `createEnvironmentInjector`

This function creates a new `EnvironmentInjector` with custom providers, linked to a parent:

```typescript
import { createEnvironmentInjector, EnvironmentInjector } from '@angular/core';

const childInjector = createEnvironmentInjector(
  [
    SomeService,
    { provide: CONFIG_TOKEN, useValue: { debug: true } },
  ],
  parentEnvironmentInjector,
  'MyChildInjector' // optional debug name
);
```

### Signature

```typescript
function createEnvironmentInjector(
  providers: (Provider | EnvironmentProviders)[],
  parent: EnvironmentInjector,
  debugName?: string | null,
): EnvironmentInjector;
```

### When to Use It

1. **Dynamic component creation with isolated services** -- when you load a component programmatically and need it to have its own service scope.
2. **Plugin systems** -- when loading third-party code that needs its own DI container.
3. **Multi-tenant scenarios** -- when different parts of the app need different service configurations.

### Dynamic Component with Custom Injector

```typescript
import {
  Component,
  ViewContainerRef,
  EnvironmentInjector,
  createEnvironmentInjector,
  inject,
  Type,
} from '@angular/core';

@Component({
  selector: 'app-plugin-host',
  template: `<button (click)="loadPlugin()">Load Plugin</button>`,
})
export class PluginHostComponent {
  private vcr = inject(ViewContainerRef);
  private parentInjector = inject(EnvironmentInjector);

  async loadPlugin() {
    // Lazy-load the plugin module
    const { PluginComponent, pluginProviders } = await import('./plugin/plugin');

    // Create a scoped injector for the plugin
    const pluginInjector = createEnvironmentInjector(
      pluginProviders,
      this.parentInjector,
      'PluginInjector',
    );

    // Create the component with the scoped injector
    const componentRef = this.vcr.createComponent(PluginComponent, {
      environmentInjector: pluginInjector,
    });

    // Clean up when the component is destroyed
    componentRef.onDestroy(() => pluginInjector.destroy());
  }
}
```

The plugin barrel file:

```typescript
// plugin/plugin.ts
import { Component, Injectable, inject } from '@angular/core';

@Injectable()
export class PluginDataService {
  getData() {
    return ['plugin-item-1', 'plugin-item-2'];
  }
}

export const pluginProviders = [PluginDataService];

@Component({
  selector: 'app-plugin',
  template: `<ul>@for (item of items; track item) { <li>{{ item }}</li> }</ul>`,
})
export class PluginComponent {
  private dataService = inject(PluginDataService);
  items = this.dataService.getData();
}
```

### Destroying Custom Injectors

Every `EnvironmentInjector` you create must be destroyed when no longer needed. Failing to destroy it leaks the services it holds:

```typescript
// WRONG: Creating without cleanup
const injector = createEnvironmentInjector(providers, parent);
// ... use it ...
// Memory leak: services in the injector are never destroyed

// CORRECT: Destroy when done
const injector = createEnvironmentInjector(providers, parent);
// ... use it ...
injector.destroy(); // All services with ngOnDestroy or DestroyRef callbacks are cleaned up
```

You can also register cleanup callbacks:

```typescript
const injector = createEnvironmentInjector(providers, parent);

const unregister = injector.onDestroy(() => {
  console.log('Injector destroyed, cleaning up resources');
});

// Later, if you need to remove the callback before destroy:
unregister();
```

## `makeEnvironmentProviders`

The `makeEnvironmentProviders` function wraps an array of providers into an `EnvironmentProviders` type. This prevents those providers from being accidentally used in a component's `providers` array.

```typescript
import { makeEnvironmentProviders, InjectionToken } from '@angular/core';

export const API_URL = new InjectionToken<string>('API_URL');

export function provideApi(url: string) {
  return makeEnvironmentProviders([
    { provide: API_URL, useValue: url },
    ApiService,
    HttpCacheInterceptor,
  ]);
}
```

### Type Safety Enforcement

```typescript
// In bootstrapApplication -- works
bootstrapApplication(AppComponent, {
  providers: [
    provideApi('https://api.example.com'),
  ],
});

// In a route -- works
const routes: Routes = [
  {
    path: 'dashboard',
    component: DashboardComponent,
    providers: [provideApi('https://api.example.com/v2')],
  },
];

// In a component -- TypeScript ERROR
@Component({
  providers: [provideApi('...')],  // Type 'EnvironmentProviders' is not assignable to type 'Provider'
})
export class SomeComponent {}
```

This is the same pattern Angular uses internally for `provideRouter`, `provideHttpClient`, `provideAnimations`, and similar configuration functions.

### Building a `provideXxx` Function

The convention for library authors and application architects:

```typescript
import {
  makeEnvironmentProviders,
  InjectionToken,
  Provider,
  EnvironmentProviders,
  inject,
} from '@angular/core';

// Configuration interface
export interface FeatureFlagConfig {
  endpoint: string;
  refreshIntervalMs: number;
  defaultFlags: Record<string, boolean>;
}

// Tokens
export const FEATURE_FLAG_CONFIG = new InjectionToken<FeatureFlagConfig>('FEATURE_FLAG_CONFIG');

// Provider function
export function provideFeatureFlags(config: FeatureFlagConfig): EnvironmentProviders {
  return makeEnvironmentProviders([
    { provide: FEATURE_FLAG_CONFIG, useValue: config },
    FeatureFlagService,
    {
      provide: FeatureFlagLoader,
      useFactory: () => {
        const http = inject(HttpClient);
        const cfg = inject(FEATURE_FLAG_CONFIG);
        return new FeatureFlagLoader(http, cfg.endpoint);
      },
    },
  ]);
}
```

## Injection Context

An **injection context** is a runtime scope where `inject()` is valid. Outside of this scope, calling `inject()` throws `NG0203`.

### Valid Injection Contexts

| Context | Example |
|---------|---------|
| Constructor of an `@Injectable` or `@Component` class | `constructor() { this.service = inject(MyService); }` |
| Field initializer of such a class | `private service = inject(MyService);` |
| `useFactory` function in a provider | `{ provide: TOKEN, useFactory: () => inject(Dep) }` |
| `InjectionToken` factory | `new InjectionToken('T', { factory: () => inject(Dep) })` |
| Functional guard, resolver, or interceptor | `const guard: CanActivateFn = () => inject(Auth).isLoggedIn()` |
| `afterRender` / `afterNextRender` callbacks | `afterRender(() => { inject(Renderer2); })` |
| Any function called from within one of the above | Stack frames inherit context |

### Invalid Injection Contexts

```typescript
// WRONG: setTimeout callback is not in an injection context
@Component({ ... })
export class BadComponent {
  constructor() {
    setTimeout(() => {
      const svc = inject(MyService); // NG0203!
    }, 1000);
  }
}

// WRONG: Event handler is not in an injection context
onClick() {
  const svc = inject(MyService); // NG0203!
}

// WRONG: RxJS operator callback after async boundary
ngOnInit() {
  this.http.get('/api').pipe(
    map(data => {
      const svc = inject(OtherService); // NG0203!
      return svc.transform(data);
    }),
  ).subscribe();
}
```

## `runInInjectionContext`

The `runInInjectionContext` function establishes a temporary injection context, allowing `inject()` to work in places where it normally cannot.

### Signature

```typescript
function runInInjectionContext<ReturnT>(
  injector: Injector,
  fn: () => ReturnT,
): ReturnT;
```

Note that it accepts any `Injector`, not just `EnvironmentInjector`. This means you can use it with element injectors too.

### Basic Usage

```typescript
import { Component, Injector, inject, runInInjectionContext } from '@angular/core';
import { Router } from '@angular/router';

@Component({
  selector: 'app-dashboard',
  template: `<button (click)="navigate()">Go</button>`,
})
export class DashboardComponent {
  private injector = inject(Injector);

  navigate() {
    // Event handlers are NOT injection contexts, so we need runInInjectionContext
    runInInjectionContext(this.injector, () => {
      const router = inject(Router);
      router.navigate(['/home']);
    });
  }
}
```

### The Synchronous Rule

`inject()` only works synchronously within the callback. Any async operation inside the callback loses the injection context:

```typescript
runInInjectionContext(this.injector, () => {
  const svc = inject(MyService); // Works

  setTimeout(() => {
    const other = inject(OtherService); // NG0203! Async boundary
  }, 0);
});
```

If you need a service in an async callback, capture it synchronously first:

```typescript
runInInjectionContext(this.injector, () => {
  const svc = inject(MyService);       // Capture synchronously
  const other = inject(OtherService);  // Capture synchronously

  setTimeout(() => {
    svc.doSomething();   // Use the already-captured reference
    other.process();     // Works fine
  }, 1000);
});
```

## `assertInInjectionContext`

This utility function throws `NG0203` with a helpful error message if called outside an injection context. It is designed for building custom inject functions:

```typescript
import { assertInInjectionContext, inject, ElementRef } from '@angular/core';

export function injectNativeElement<T extends HTMLElement>(): T {
  // Throws NG0203 with message pointing to injectNativeElement
  assertInInjectionContext(injectNativeElement);
  return inject(ElementRef).nativeElement as T;
}
```

Without `assertInInjectionContext`, the error would point to the internal `inject(ElementRef)` call, making debugging harder. With it, the error clearly says: "injectNativeElement() must be called from an injection context."

### Building Custom Inject Functions

Custom inject functions are a powerful pattern for encapsulating reusable injection logic:

```typescript
import {
  assertInInjectionContext,
  inject,
  DestroyRef,
  signal,
  WritableSignal,
} from '@angular/core';

/**
 * Creates a signal that tracks window width, with automatic cleanup.
 */
export function injectWindowWidth(): WritableSignal<number> {
  assertInInjectionContext(injectWindowWidth);

  const width = signal(window.innerWidth);
  const destroyRef = inject(DestroyRef);

  const handler = () => width.set(window.innerWidth);
  window.addEventListener('resize', handler);

  destroyRef.onDestroy(() => {
    window.removeEventListener('resize', handler);
  });

  return width;
}

// Usage in a component
@Component({
  selector: 'app-responsive',
  template: `<p>Width: {{ width() }}px</p>`,
})
export class ResponsiveComponent {
  width = injectWindowWidth(); // Works because field initializer is an injection context
}
```

### Custom Inject Function with `runInInjectionContext` Fallback

Some custom inject functions accept an optional `Injector` for use outside injection contexts:

```typescript
import {
  assertInInjectionContext,
  inject,
  Injector,
  runInInjectionContext,
  DestroyRef,
  signal,
} from '@angular/core';

export interface InjectMediaQueryOptions {
  injector?: Injector;
}

export function injectMediaQuery(
  query: string,
  options?: InjectMediaQueryOptions,
) {
  // If an injector is provided, use it. Otherwise, assert we are in context.
  if (!options?.injector) {
    assertInInjectionContext(injectMediaQuery);
  }

  const injector = options?.injector ?? inject(Injector);

  // Use runInInjectionContext to ensure inject() works
  return runInInjectionContext(injector, () => {
    const matches = signal(window.matchMedia(query).matches);
    const destroyRef = inject(DestroyRef);

    const mql = window.matchMedia(query);
    const handler = (e: MediaQueryListEvent) => matches.set(e.matches);
    mql.addEventListener('change', handler);

    destroyRef.onDestroy(() => mql.removeEventListener('change', handler));

    return matches.asReadonly();
  });
}
```

Usage:

```typescript
@Component({ ... })
export class LayoutComponent {
  // Option 1: In field initializer (injection context available)
  isMobile = injectMediaQuery('(max-width: 768px)');

  // Option 2: Outside injection context (e.g., in a method)
  private injector = inject(Injector);

  ngAfterViewInit() {
    const isLandscape = injectMediaQuery('(orientation: landscape)', {
      injector: this.injector,
    });
  }
}
```

## Dynamic Component Loading with Environment Injectors

One of the most practical uses of `createEnvironmentInjector` is loading components with their own service scope.

### Lazy-Loaded Feature with Scoped Services

```typescript
import {
  Component,
  ViewContainerRef,
  EnvironmentInjector,
  createEnvironmentInjector,
  inject,
  signal,
} from '@angular/core';

@Component({
  selector: 'app-shell',
  template: `
    <nav>
      <button (click)="loadFeature('analytics')">Analytics</button>
      <button (click)="loadFeature('settings')">Settings</button>
    </nav>
    <div #outlet></div>
  `,
})
export class ShellComponent {
  private vcr = inject(ViewContainerRef);
  private parentInjector = inject(EnvironmentInjector);
  private activeInjector: EnvironmentInjector | null = null;

  async loadFeature(name: string) {
    // Destroy previous feature's injector
    this.cleanUp();

    // Dynamically import the feature
    const feature = await this.resolveFeature(name);

    // Create a scoped injector for this feature
    this.activeInjector = createEnvironmentInjector(
      feature.providers,
      this.parentInjector,
      `${name}FeatureInjector`,
    );

    // Render the component
    this.vcr.clear();
    this.vcr.createComponent(feature.component, {
      environmentInjector: this.activeInjector,
    });
  }

  private async resolveFeature(name: string) {
    switch (name) {
      case 'analytics':
        return import('./features/analytics/analytics.feature');
      case 'settings':
        return import('./features/settings/settings.feature');
      default:
        throw new Error(`Unknown feature: ${name}`);
    }
  }

  private cleanUp() {
    this.vcr.clear();
    this.activeInjector?.destroy();
    this.activeInjector = null;
  }

  ngOnDestroy() {
    this.cleanUp();
  }
}
```

Feature barrel:

```typescript
// features/analytics/analytics.feature.ts
import { Component, Injectable, inject, makeEnvironmentProviders } from '@angular/core';

@Injectable()
export class AnalyticsDataService {
  // Scoped to the analytics feature, destroyed when the feature unloads
  private cache = new Map<string, unknown>();

  fetchMetrics() { /* ... */ }
}

export const providers = [AnalyticsDataService];

@Component({
  selector: 'app-analytics',
  template: `<h2>Analytics Dashboard</h2>`,
})
export class AnalyticsComponent {
  private data = inject(AnalyticsDataService);
}

export const component = AnalyticsComponent;
```

### How the Router Does It

The Angular router uses the same mechanism internally. When a route has a `providers` array:

```typescript
const routes: Routes = [
  {
    path: 'admin',
    loadComponent: () => import('./admin/admin.component'),
    providers: [
      AdminService,
      { provide: ADMIN_CONFIG, useValue: { role: 'superadmin' } },
    ],
  },
];
```

The router calls `createEnvironmentInjector` with the route's providers and the parent route's injector. This is why services provided at the route level are:

1. **Scoped** to that route and its children.
2. **Destroyed** when the user navigates away.
3. **Recreated** on each navigation to that route.

## `EnvironmentInjector` vs `Injector`

You will encounter both types when working with injection contexts:

| Aspect | `Injector` | `EnvironmentInjector` |
|--------|-----------|----------------------|
| Base type | Abstract base for all injectors | Extends `Injector` |
| Includes element injectors | Yes | No |
| Used with `runInInjectionContext` | Yes | Yes |
| Used with `createEnvironmentInjector` | No (needs `EnvironmentInjector` parent) | Yes |
| Inject via DI | `inject(Injector)` gives closest injector | `inject(EnvironmentInjector)` gives closest env injector |

When calling `runInInjectionContext`, you can pass either type:

```typescript
// With the general Injector (includes element-level providers)
private injector = inject(Injector);

runInInjectionContext(this.injector, () => {
  // Can resolve both element-level and environment-level providers
  const elementRef = inject(ElementRef); // Works if in a component context
  const router = inject(Router);         // Works
});

// With EnvironmentInjector (only environment-level providers)
private envInjector = inject(EnvironmentInjector);

runInInjectionContext(this.envInjector, () => {
  const router = inject(Router);          // Works
  const elementRef = inject(ElementRef);  // Fails -- not in an element context
});
```

**Rule of thumb:** Use `inject(Injector)` when you need access to both element and environment providers. Use `inject(EnvironmentInjector)` when you only need environment-level services and want to be explicit.

Testing code that uses `inject()` requires establishing an injection context. Angular's `TestBed` provides one:

```typescript

describe('injectWindowWidth', () => {
  it('should return current window width', () => {

    const width = TestBed.runInInjectionContext(() => {
      return injectWindowWidth();

```

## Common Mistakes

### Mistake 1: Calling `inject()` in Async Callbacks

```typescript
// WRONG: inject() is invalid after an async boundary
async loadData() {
  const data = await fetch('/api/data');
  const parser = inject(DataParserService); // NG0203!
}

// CORRECT: Capture the service reference before the async boundary
private parser = inject(DataParserService);

async loadData() {
  const data = await fetch('/api/data');
  this.parser.parse(data); // Works
}
```

### Mistake 2: Forgetting to Destroy Custom Injectors

```typescript
// WRONG: Leaks services
loadFeature() {
  const injector = createEnvironmentInjector(providers, this.parentInjector);
  this.vcr.createComponent(FeatureComponent, { environmentInjector: injector });
  // injector is never destroyed
}

// CORRECT: Track and destroy
private featureInjector: EnvironmentInjector | null = null;

loadFeature() {
  this.featureInjector?.destroy(); // Clean up previous
  this.featureInjector = createEnvironmentInjector(providers, this.parentInjector);
  this.vcr.createComponent(FeatureComponent, {
    environmentInjector: this.featureInjector,
  });
}

ngOnDestroy() {
  this.featureInjector?.destroy();
}
```

### Mistake 3: Using Deprecated `runInContext`

```typescript
// DEPRECATED (Angular 16+)
this.envInjector.runInContext(() => {
  inject(MyService);
});

// CORRECT: Use the standalone function
runInInjectionContext(this.envInjector, () => {
  inject(MyService);
});
```

### Mistake 4: Using `EnvironmentInjector` When You Need Element Providers

```typescript
@Component({ ... })
export class MyComponent {
  private envInjector = inject(EnvironmentInjector);

  someMethod() {
    runInInjectionContext(this.envInjector, () => {
      // FAILS: ElementRef is an element-level provider, not an environment one
      const el = inject(ElementRef);
    });
  }

  // CORRECT: Use the general Injector for element-level access
  private injector = inject(Injector);

  someMethodFixed() {
    runInInjectionContext(this.injector, () => {
      const el = inject(ElementRef); // Works
    });
  }
}
```

### Mistake 5: Overusing `runInInjectionContext`

If you find yourself calling `runInInjectionContext` frequently within a component, the code probably needs restructuring. Most dependencies should be injected at initialization time:

```typescript
// BAD: Using runInInjectionContext everywhere
@Component({ ... })
export class OverusingComponent {
  private injector = inject(Injector);

  methodA() {
    runInInjectionContext(this.injector, () => {
      inject(ServiceA).doA();
    });
  }

  methodB() {
    runInInjectionContext(this.injector, () => {
      inject(ServiceB).doB();
    });
  }

  // GOOD: Inject once, use everywhere
  private serviceA = inject(ServiceA);
  private serviceB = inject(ServiceB);

  methodA() { this.serviceA.doA(); }
  methodB() { this.serviceB.doB(); }
}
```

Reserve `runInInjectionContext` for cases where the service is truly needed conditionally or dynamically, not for standard injection.

## Best Practices

### 1. Prefer Field Initializers Over `runInInjectionContext`

The simplest and most readable way to use `inject()` is at the field level. Only reach for `runInInjectionContext` when you cannot inject at initialization time.

### 2. Always Destroy Custom Environment Injectors

Treat `createEnvironmentInjector` like `new` -- every creation needs a corresponding cleanup. Link destruction to the component lifecycle or route deactivation.

### 3. Use `makeEnvironmentProviders` for Library Provider Functions

When building `provideXxx()` functions, wrap your providers with `makeEnvironmentProviders`. This prevents consumers from accidentally adding environment-only providers to component-level injectors.

### 4. Accept an Optional `Injector` in Custom Inject Functions

Design custom inject functions to work both in injection contexts (default) and outside them (via an explicit injector):

```typescript
export function injectFoo(options?: { injector?: Injector }) {
  if (!options?.injector) {
    assertInInjectionContext(injectFoo);
  }
  const injector = options?.injector ?? inject(Injector);
  return runInInjectionContext(injector, () => {
    // ...
  });
}
```

### 5. Use `inject(Injector)` Over `inject(EnvironmentInjector)` for `runInInjectionContext`

Unless you specifically need to exclude element-level providers, use the general `Injector` to get the broadest resolution scope.

### 6. Use `assertInInjectionContext` in Reusable Functions

Always guard custom inject functions with `assertInInjectionContext` so errors point to the correct API.

## Real-World Use Cases

### Use Case 1: Third-Party Library Integration

Integrating a charting library that needs Angular services in its event callbacks:

```typescript
import { Component, inject, Injector, runInInjectionContext, AfterViewInit } from '@angular/core';

@Component({
  selector: 'app-chart',
  template: `<div #chartContainer></div>`,
})
export class ChartComponent implements AfterViewInit {
  private injector = inject(Injector);
  private analytics = inject(AnalyticsService);

  ngAfterViewInit() {
    const chart = new ThirdPartyChart(document.getElementById('chart'));

    chart.onDataPointClick((point) => {
      // This callback is outside injection context, but we captured analytics above
      this.analytics.trackClick('chart-point', point.label);
    });

    chart.onRenderComplete(() => {
      // If we need a service we did not capture at init time:
      runInInjectionContext(this.injector, () => {
        inject(PerformanceMonitor).recordRender('chart', performance.now());
      });
    });
  }
}
```

### Use Case 2: Micro-Frontend Shell with Scoped Injectors

A shell application that loads micro-frontends, each with isolated services:

```typescript
@Component({
  selector: 'app-micro-frontend-host',
  template: `<ng-container #mfContainer></ng-container>`,
})
export class MicroFrontendHostComponent {
  private vcr = inject(ViewContainerRef);
  private parentInjector = inject(EnvironmentInjector);
  private activeInjectors = new Map<string, EnvironmentInjector>();

  async mount(mfName: string, config: MfConfig) {
    // Each micro-frontend gets its own injector
    const mfInjector = createEnvironmentInjector(
      [
        { provide: MF_CONFIG, useValue: config },
        { provide: MF_NAME, useValue: mfName },
        ...config.providers,
      ],
      this.parentInjector,
      `MF:${mfName}`,
    );

    this.activeInjectors.set(mfName, mfInjector);

    const componentType = await config.loadComponent();
    this.vcr.createComponent(componentType, {
      environmentInjector: mfInjector,
    });
  }

  unmount(mfName: string) {
    const injector = this.activeInjectors.get(mfName);
    if (injector) {
      injector.destroy();
      this.activeInjectors.delete(mfName);
    }
  }

  ngOnDestroy() {
    for (const injector of this.activeInjectors.values()) {
      injector.destroy();
    }
    this.activeInjectors.clear();
  }
}
```

### Use Case 3: RxJS Operator with `takeUntilDestroyed`

Using `takeUntilDestroyed` inside an RxJS pipe requires an injection context or an explicit `DestroyRef`:

```typescript
import { Component, inject, DestroyRef } from '@angular/core';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';
import { interval } from 'rxjs';

@Component({
  selector: 'app-ticker',
  template: `{{ count }}`,
})
export class TickerComponent {
  count = 0;

  // Option A: Call takeUntilDestroyed in field initializer (injection context available)
  private tick$ = interval(1000).pipe(
    takeUntilDestroyed(),
  );

  // Option B: Pass DestroyRef explicitly when outside injection context
  private destroyRef = inject(DestroyRef);

  ngOnInit() {
    interval(1000).pipe(
      takeUntilDestroyed(this.destroyRef), // Explicit DestroyRef, no injection context needed
    ).subscribe(n => this.count = n);
  }
}
```

---

> **Key Takeaway:** `EnvironmentInjector` and `runInInjectionContext` bridge the gap between Angular's DI system and code that runs outside normal injection contexts. Use `createEnvironmentInjector` for scoped service lifetimes in dynamic scenarios, `runInInjectionContext` for calling `inject()` in event handlers or callbacks, and `makeEnvironmentProviders` to build type-safe provider functions.
