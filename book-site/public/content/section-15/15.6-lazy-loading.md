# 15.6 Lazy Loading

> Lazy loading splits your application into smaller JavaScript chunks that are loaded on demand, reducing the initial bundle size and improving time-to-interactive. Angular provides three lazy loading mechanisms: route-based (`loadComponent`/`loadChildren`), template-based (`@defer`), and programmatic (dynamic `import()`).

---

## Why Lazy Loading Matters

Without lazy loading, every component, service, and dependency ships in a single JavaScript bundle. For a non-trivial app, this means the user downloads megabytes of code before seeing anything.

| Metric | Eager Loading | Lazy Loading |
|--------|-------------|-------------|
| Initial bundle | Everything | Only what the landing page needs |
| Time to interactive | Slow (proportional to total app size) | Fast (proportional to landing page size) |
| Subsequent navigation | Instant (already loaded) | Small delay on first visit (chunk download) |
| Memory usage | High (all components parsed) | Lower (chunks loaded on demand) |

The goal: load the minimum code needed for the first screen, then load the rest in the background or on demand.

## Route-Based Lazy Loading

### `loadComponent`

Lazily loads a single standalone component when its route is activated:

```typescript
const routes: Routes = [
  {
    path: 'settings',
    loadComponent: () =>
      import('./settings/settings.component').then(m => m.SettingsComponent),
  },
];
```

The `import()` call tells the bundler (Webpack/esbuild/Vite) to create a separate chunk for `SettingsComponent` and everything it imports. The chunk is fetched only when the user navigates to `/settings`.

### Default Export Shorthand

If the component is the default export, skip the `.then()`:

```typescript
// settings.component.ts
@Component({ ... })
export default class SettingsComponent {}

// Route
{
  path: 'settings',
  loadComponent: () => import('./settings/settings.component'),
}
```

### `loadChildren`

Lazily loads an entire set of child routes:

```typescript
const routes: Routes = [
  {
    path: 'admin',
    loadChildren: () =>
      import('./admin/admin.routes').then(m => m.ADMIN_ROUTES),
  },
];
```

```typescript
// admin/admin.routes.ts
import { Routes } from '@angular/router';

export const ADMIN_ROUTES: Routes = [
  {
    path: '',
    loadComponent: () => import('./admin-layout.component'),
    children: [
      { path: '', loadComponent: () => import('./dashboard.component') },
      { path: 'users', loadComponent: () => import('./users.component') },
      { path: 'reports', loadComponent: () => import('./reports.component') },
    ],
  },
];
```

This creates a separate chunk for the admin routes file and each lazily loaded component within it.

### `loadComponent` vs `loadChildren`

| Use Case | Use |
|----------|-----|
| Single component with no children | `loadComponent` |
| Feature with multiple child routes | `loadChildren` |
| Feature with route-level providers | `loadChildren` (providers on the parent route) |

```typescript
// loadChildren with route-level providers
{
  path: 'editor',
  loadChildren: () => import('./editor/editor.routes').then(m => m.EDITOR_ROUTES),
  providers: [EditorStateService, AutoSaveService],
}
```

## What Gets Chunked

The bundler performs static analysis on `import()` calls. Everything reachable from the imported file becomes part of that chunk, unless it is already in the main bundle or another chunk.

### The Import Chain

```
main.js (initial bundle)
  └── imports AppComponent, HomeComponent, provideRouter, etc.

chunk-admin.js (lazy)
  └── imports AdminLayout, AdminDashboard, AdminService
        └── imports ChartLibrary (only if not used elsewhere)
```

### Shared Dependencies

If a lazy chunk and the main bundle both import the same library (e.g., `lodash`), the bundler places it in a shared/common chunk loaded with the main bundle. This prevents duplication but increases the initial load.

```
main.js          -- AppComponent + shared deps
common.js        -- lodash (used by both main and admin)
chunk-admin.js   -- AdminComponent (uses lodash from common.js)
chunk-reports.js -- ReportsComponent (uses lodash from common.js)
```

## Preloading Strategies

Preloading downloads lazy chunks in the background after the initial page load, so they are ready when the user navigates.

### `PreloadAllModules`

Downloads all lazy chunks immediately after the initial load:

```typescript
import { PreloadAllModules } from '@angular/router';

provideRouter(
  routes,
  withPreloading(PreloadAllModules),
);
```

Good for small-to-medium apps where total bundle size is manageable. Bad for large apps where preloading everything wastes bandwidth.

### `NoPreloading` (Default)

No background loading. Chunks are fetched only when the user navigates:

```typescript
provideRouter(routes, withPreloading(NoPreloading));
```

### Custom Preloading Strategy

Preload only routes marked with `data: { preload: true }`:

```typescript
import { Injectable } from '@angular/core';
import { PreloadingStrategy, Route } from '@angular/router';
import { Observable, of, EMPTY } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class SelectivePreloadStrategy implements PreloadingStrategy {
  preload(route: Route, load: () => Observable<unknown>): Observable<unknown> {
    return route.data?.['preload'] ? load() : EMPTY;
  }
}
```

```typescript
const routes: Routes = [
  {
    path: 'dashboard',
    loadComponent: () => import('./dashboard/dashboard.component'),
    data: { preload: true },  // Will be preloaded
  },
  {
    path: 'admin',
    loadChildren: () => import('./admin/admin.routes').then(m => m.ADMIN_ROUTES),
    // No preload flag -- loaded only on navigation
  },
];

provideRouter(routes, withPreloading(SelectivePreloadStrategy));
```

### Network-Aware Preloading

Preload only on fast connections:

```typescript
@Injectable({ providedIn: 'root' })
export class NetworkAwarePreloadStrategy implements PreloadingStrategy {
  preload(route: Route, load: () => Observable<unknown>): Observable<unknown> {
    const connection = (navigator as any).connection;

    // Only preload on fast connections
    if (connection) {
      const dominated = connection.saveData || connection.effectiveType === '2g';
      if (dominated) return EMPTY;
    }

    return route.data?.['preload'] ? load() : EMPTY;
  }
}
```

## Template-Based Lazy Loading with `@defer`

`@defer` lazily loads components within a template, independent of routing. It is ideal for heavy components that are not immediately visible or needed.

### Basic Syntax

```typescript
@Component({
  template: `
    <h1>Dashboard</h1>
    <app-summary-cards />

    @defer (on viewport) {
      <app-analytics-chart />
    } @placeholder {
      <div class="chart-skeleton">Loading chart...</div>
    } @loading (after 200ms; minimum 500ms) {
      <app-spinner />
    } @error {
      <p>Failed to load chart.</p>
    }
  `,
})
export class DashboardComponent {}
```

### Trigger Types

| Trigger | Fires When | Use Case |
|---------|-----------|----------|
| `on idle` | Browser is idle (`requestIdleCallback`) | Default. Non-critical below-the-fold content |
| `on viewport` | Placeholder enters the viewport | Long scrollable pages, infinite scroll |
| `on interaction` | User clicks or presses a key on the placeholder | Tabs, expandable sections |
| `on hover` | User hovers over or focuses the placeholder | Tooltips, preview panels |
| `on timer(ms)` | After the specified milliseconds | Delayed secondary content |
| `on immediate` | During template rendering | Components needed soon but not critical path |
| `when condition` | When the expression becomes truthy | Conditional feature flags, auth state |

### Combining Triggers

```typescript
// Load on viewport OR when the user explicitly requests it
@defer (on viewport; when showChart()) {
  <app-chart />
} @placeholder {
  <button (click)="showChart.set(true)">Show Chart</button>
}
```

The first trigger that fires wins. Once loaded, the component stays loaded.

### Prefetching

Prefetch downloads the chunk without rendering the component:

```typescript
// Prefetch after 2 seconds, render when scrolled into view
@defer (on viewport; prefetch on timer(2000)) {
  <app-heavy-widget />
} @placeholder {
  <div class="widget-placeholder"></div>
}
```

Prefetching is especially useful for content the user is likely to scroll to. The chunk downloads in the background, so when the viewport trigger fires, the component renders instantly.

### `@placeholder` Block

Shown before the trigger fires. Content here is in the initial bundle, so keep it lightweight:

```typescript
@defer (on viewport) {
  <app-data-grid [data]="gridData()" />
} @placeholder (minimum 100ms) {
  <div class="grid-skeleton">
    @for (i of skeletonRows; track i) {
      <div class="skeleton-row"></div>
    }
  </div>
}
```

The `minimum` option ensures the placeholder shows for at least the specified time, avoiding a flash of content.

### `@loading` Block

Shown while the chunk is downloading:

```typescript
@defer (on interaction) {
  <app-rich-text-editor />
} @placeholder {
  <button>Click to edit</button>
} @loading (after 150ms; minimum 300ms) {
  <app-spinner />
}
```

- `after`: Do not show the loading block until this time has passed (avoids flash for fast loads).
- `minimum`: Show the loading block for at least this time (prevents jarring flash).

### `@error` Block

Shown if the chunk fails to download:

```typescript
@defer (on viewport) {
  <app-map />
} @error {
  <p>Could not load the map. <button (click)="retry()">Retry</button></p>
}
```

## `@defer` vs Route-Based Lazy Loading

| Aspect | Route-Based | `@defer` |
|--------|------------|----------|
| Granularity | Entire page/feature | Individual component within a page |
| Trigger | Navigation event | Viewport, interaction, hover, timer, condition |
| URL change | Yes | No |
| Use case | Feature-level code splitting | Component-level code splitting |
| SEO/deep-linking | Yes (URL-driven) | No |
| Complementary? | Yes | Yes |

Use route-based lazy loading for features (pages, sections). Use `@defer` for heavy components within those features (charts, editors, maps, third-party widgets).

## Programmatic Lazy Loading

For cases where neither routing nor `@defer` fits, use dynamic `import()` directly:

```typescript
@Component({
  template: `
    <button (click)="loadEditor()">Open Editor</button>
    @if (editorComponent) {
      <ng-container *ngComponentOutlet="editorComponent" />
    }
  `,
})
export class DocumentComponent {
  editorComponent: Type<unknown> | null = null;

  async loadEditor() {
    const { RichTextEditorComponent } = await import('./rich-text-editor.component');
    this.editorComponent = RichTextEditorComponent;
  }
}
```

### With `createEnvironmentInjector` for Scoped Services

```typescript
async loadPlugin() {
  const { PluginComponent, pluginProviders } = await import('./plugin/plugin');
  const injector = createEnvironmentInjector(
    pluginProviders,
    this.parentInjector,
    'PluginInjector',
  );
  this.vcr.createComponent(PluginComponent, { environmentInjector: injector });
}
```

## Handling Chunk Load Failures

When a new deployment happens while users have the old app open, lazy-loaded chunk URLs become invalid. The user clicks a link and gets a `ChunkLoadError`.

### Using `withNavigationErrorHandler`

```typescript
provideRouter(
  routes,
  withNavigationErrorHandler((error) => {
    // Detect chunk load failure
    if (error?.message?.includes('Loading chunk') ||
        error?.message?.includes('Failed to fetch dynamically imported module')) {
      // Option 1: Reload the page to get the new deployment
      window.location.reload();
    }
  }),
);
```

### Retry Logic with Service Worker

If using a service worker, the cached navigation response serves the latest `index.html`, which references the new chunk filenames:

```typescript
// ngsw-config.json
{
  "index": "/index.html",
  "assetGroups": [
    {
      "name": "app",
      "installMode": "prefetch",
      "resources": {
        "files": ["/index.html", "/*.css", "/main.*.js"]
      }
    },
    {
      "name": "lazy",
      "installMode": "lazy",
      "resources": {
        "files": ["/chunk-*.js"]
      }
    }
  ]
}
```

## Analyzing Bundle Size

### Using `ng build` with Source Maps

```bash
ng build --source-map
npx source-map-explorer dist/my-app/browser/main.js
```

This visualizes what is in each chunk and helps identify:
- Dependencies that should be lazy-loaded but are in the main bundle.
- Shared libraries inflating the common chunk.
- Unused code that should be tree-shaken.

### Using the Angular CLI Budget

```json
// angular.json
{
  "budgets": [
    {
      "type": "initial",
      "maximumWarning": "500kB",
      "maximumError": "1MB"
    },
    {
      "type": "anyComponentStyle",
      "maximumWarning": "2kB",
      "maximumError": "4kB"
    }
  ]
}
```

The build fails if a lazy chunk or the initial bundle exceeds the budget.

## Common Mistakes

### Mistake 1: Importing Lazy Components Eagerly

```typescript
// BAD: Importing the component at the top defeats lazy loading
import { AdminComponent } from './admin/admin.component';

const routes: Routes = [
  { path: 'admin', component: AdminComponent }, // In the main bundle!
];

// GOOD: Use dynamic import
const routes: Routes = [
  {
    path: 'admin',
    loadComponent: () => import('./admin/admin.component').then(m => m.AdminComponent),
  },
];
```

Even one static `import` statement pulls the component into the main bundle, regardless of `loadComponent`.

### Mistake 2: Importing Lazy Routes in `app.config.ts`

```typescript
// BAD: Importing the route file eagerly
import { ADMIN_ROUTES } from './admin/admin.routes';

const routes: Routes = [
  { path: 'admin', children: ADMIN_ROUTES }, // Not lazy!
];

// GOOD: Dynamic import in loadChildren
const routes: Routes = [
  {
    path: 'admin',
    loadChildren: () => import('./admin/admin.routes').then(m => m.ADMIN_ROUTES),
  },
];
```

### Mistake 3: Not Lazy Loading Heavy Third-Party Libraries

```typescript
// BAD: Chart library loaded on every page
import { ChartComponent } from './chart.component'; // Imports chart.js

@Component({
  imports: [ChartComponent],
  template: `<app-chart />`,
})
export class DashboardComponent {}

// GOOD: Defer the heavy chart component
@Component({
  template: `
    @defer (on viewport) {
      <app-chart />
    } @placeholder {
      <div class="chart-skeleton"></div>
    }
  `,
})
export class DashboardComponent {}
```

### Mistake 4: Using `@defer` for Everything

```typescript
// BAD: Deferring tiny, lightweight components adds overhead
@defer {
  <app-icon name="settings" />
} @placeholder {
  <span>...</span>
}

// GOOD: Only defer components that are heavy (charts, editors, maps)
// or not immediately visible (below the fold)
<app-icon name="settings" />
```

Each `@defer` block creates a separate chunk request. For small components, the HTTP overhead exceeds the bundle savings.

### Mistake 5: Forgetting Preloading

```typescript
// BAD: No preloading -- every navigation to a lazy route has a visible delay
provideRouter(routes);

// GOOD: Preload important routes in the background
provideRouter(routes, withPreloading(SelectivePreloadStrategy));
```

### Mistake 6: Shared Module Bloating Common Chunk

```typescript
// BAD: SharedModule imports everything, inflating the common chunk
@NgModule({
  imports: [DataGridComponent, ChartComponent, RichEditorComponent, ...],
  exports: [DataGridComponent, ChartComponent, RichEditorComponent, ...],
})
export class SharedModule {}

// GOOD: Import only what each feature needs directly
// In feature A:
import { DataGridComponent } from '@shared/data-grid.component';

// In feature B:
import { ChartComponent } from '@shared/chart.component';
```

With standalone components, you import individual components directly, avoiding the shared module bloat problem entirely.

## Best Practices

### 1. Lazy Load Everything Except the Landing Page

The first screen the user sees should load eagerly. Everything else should be lazy.

### 2. Use `@defer` for Below-the-Fold Content

Heavy components that are not visible on initial render (charts, maps, data grids below the scroll) should use `@defer (on viewport)`.

### 3. Combine Route Lazy Loading with `@defer`

Route-level lazy loading splits features. `@defer` splits components within features. Use both for maximum code splitting.

### 4. Use a Custom Preloading Strategy

`PreloadAllModules` is too aggressive for large apps. Use a selective strategy that preloads based on data flags, user behavior, or network conditions.

### 5. Set Bundle Budgets

Configure budgets in `angular.json` to catch bundle regressions in CI.

### 6. Handle Chunk Failures Gracefully

Use `withNavigationErrorHandler` to detect chunk load failures and reload the page or show a user-friendly error.

### 7. Analyze Bundles Regularly

Run `source-map-explorer` after builds to verify that lazy chunks contain only what they should.

## Real-World Use Cases

### E-Commerce with Route and Template Lazy Loading

```typescript
// app.routes.ts -- route-level lazy loading
export const routes: Routes = [
  {
    path: '',
    component: HomeComponent,  // Eagerly loaded (landing page)
  },
  {
    path: 'products',
    loadChildren: () => import('./products/product.routes').then(m => m.PRODUCT_ROUTES),
    data: { preload: true },  // Preload since users likely visit this
  },
  {
    path: 'cart',
    loadComponent: () => import('./cart/cart.component'),
  },
  {
    path: 'checkout',
    loadComponent: () => import('./checkout/checkout.component'),
    canActivate: [authGuard],
  },
  {
    path: 'account',
    loadChildren: () => import('./account/account.routes').then(m => m.ACCOUNT_ROUTES),
  },
];
```

```typescript
// products/product-detail.component.ts -- template-level lazy loading
@Component({
  template: `
    <app-product-header [product]="product()" />
    <app-product-gallery [images]="product()?.images ?? []" />

    <!-- Reviews are below the fold, defer until visible -->
    @defer (on viewport; prefetch on timer(3000)) {
      <app-product-reviews [productId]="product()?.id" />
    } @placeholder {
      <div class="reviews-skeleton">
        <h3>Customer Reviews</h3>
        <div class="skeleton-lines"></div>
      </div>
    }

    <!-- Recommendation engine is heavy, defer until visible -->
    @defer (on viewport) {
      <app-recommended-products [category]="product()?.category" />
    } @placeholder {
      <div class="recommendations-skeleton"></div>
    }
  `,
})
export class ProductDetailComponent {
  product = input<Product>();
}
```

### Admin Dashboard with Selective Preloading

```typescript
// admin.routes.ts
export const ADMIN_ROUTES: Routes = [
  {
    path: '',
    loadComponent: () => import('./admin-layout.component'),
    children: [
      {
        path: '',
        redirectTo: 'dashboard',
        pathMatch: 'full',
      },
      {
        path: 'dashboard',
        loadComponent: () => import('./dashboard.component'),
        data: { preload: true },
      },
      {
        path: 'users',
        loadComponent: () => import('./users/user-list.component'),
        data: { preload: true },
      },
      {
        path: 'users/:id',
        loadComponent: () => import('./users/user-detail.component'),
      },
      {
        path: 'analytics',
        loadComponent: () => import('./analytics/analytics.component'),
        // Not preloaded -- heavy chart dependencies
      },
      {
        path: 'audit-log',
        loadComponent: () => import('./audit/audit-log.component'),
        // Not preloaded -- rarely visited
      },
    ],
  },
];
```

```typescript
// analytics.component.ts -- heavy charts deferred within the feature
@Component({
  template: `
    <h1>Analytics</h1>
    <app-date-range-picker (rangeChange)="onRangeChange($event)" />

    @defer (on idle) {
      <app-revenue-chart [range]="dateRange()" />
    } @placeholder {
      <div class="chart-placeholder" style="height: 400px;"></div>
    } @loading (after 100ms) {
      <app-spinner />
    }

    @defer (on viewport) {
      <app-user-activity-chart [range]="dateRange()" />
    } @placeholder {
      <div class="chart-placeholder" style="height: 300px;"></div>
    }

    @defer (on viewport) {
      <app-conversion-funnel [range]="dateRange()" />
    } @placeholder {
      <div class="chart-placeholder" style="height: 350px;"></div>
    }
  `,
})
export class AnalyticsComponent {
  dateRange = signal<DateRange>({ start: subDays(new Date(), 30), end: new Date() });

  onRangeChange(range: DateRange) {
    this.dateRange.set(range);
  }
}
```

### Progressive Loading with `@defer` Waterfall

```typescript
// Complex page that loads sections progressively
@Component({
  template: `
    <!-- Critical: always in initial bundle -->
    <app-hero-banner />
    <app-quick-stats [data]="stats()" />

    <!-- Second priority: load after initial render is done -->
    @defer (on idle; prefetch on immediate) {
      <app-activity-feed />
    } @placeholder {
      <app-feed-skeleton />
    }

    <!-- Third priority: load when scrolled into view -->
    @defer (on viewport; prefetch on idle) {
      <app-team-calendar />
    } @placeholder {
      <div class="calendar-skeleton" style="height: 500px;"></div>
    }

    <!-- Low priority: load only when user scrolls far down -->
    @defer (on viewport) {
      <app-recent-documents />
    } @placeholder {
      <div class="documents-skeleton"></div>
    }

    @defer (on viewport) {
      <app-integrations-panel />
    } @placeholder {
      <div class="integrations-skeleton"></div>
    }
  `,
})
export class DashboardPageComponent {
  stats = inject(StatsService).quickStats;
}
```

This creates a progressive loading waterfall:
1. Hero and stats render immediately.
2. Activity feed downloads during idle time, renders when ready.
3. Calendar prefetches during idle, renders when scrolled into view.
4. Documents and integrations load only when scrolled far enough.

---

> **Key Takeaway:** Use route-based lazy loading (`loadComponent`/`loadChildren`) to split features into separate chunks, `@defer` to lazy load heavy components within features, and preloading strategies to download important chunks before the user needs them. The combination of all three gives you the fastest initial load with the smoothest subsequent navigation.
