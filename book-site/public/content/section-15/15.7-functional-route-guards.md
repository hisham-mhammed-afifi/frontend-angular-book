# 15.7 Functional Route Guards

> Route guards are functions that control whether a user can navigate to, away from, or even match a route. Functional guards replaced class-based guards as the recommended approach in Angular 14.2+.

---

## Guard Types Overview

Angular provides four guard types, each with a specific purpose:

| Guard | Type Signature | Purpose |
|-------|---------------|---------|
| `canActivate` | `CanActivateFn` | Can the user navigate TO this route? |
| `canActivateChild` | `CanActivateChildFn` | Can the user access child routes? |
| `canDeactivate` | `CanDeactivateFn<T>` | Can the user navigate AWAY from this route? |
| `canMatch` | `CanMatchFn` | Should this route even be considered during URL matching? |

All guards run in an injection context, so `inject()` is available.

---

## `CanActivateFn`

The most commonly used guard. It determines whether a user can access a route.

### Basic Authentication Guard

```typescript
import { inject } from '@angular/core';
import { CanActivateFn, Router, UrlTree } from '@angular/router';
import { AuthService } from './auth.service';

export const authGuard: CanActivateFn = (route, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (auth.isAuthenticated()) {
    return true;
  }

  // Redirect to login with return URL
  return router.createUrlTree(['/login'], {
    queryParams: { returnUrl: state.url },
  });
};
```

### Parameters

```typescript
const myGuard: CanActivateFn = (
  route: ActivatedRouteSnapshot,  // The route being navigated to
  state: RouterStateSnapshot,      // The full router state after navigation
) => {
  // route.paramMap, route.data, route.queryParamMap are available
  // state.url is the full target URL
  return true;
};
```

### Return Types

A guard can return any of the following:

| Return | Effect |
|--------|--------|
| `true` | Navigation proceeds |
| `false` | Navigation is cancelled |
| `UrlTree` | Navigation redirects to the given URL |
| `RedirectCommand` | Navigation redirects with navigation options |
| `Observable<boolean \| UrlTree>` | Router uses the first emitted value |
| `Promise<boolean \| UrlTree>` | Router awaits the resolved value |

### Using Route Data in Guards

```typescript
export const roleGuard: CanActivateFn = (route) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  const requiredRole = route.data['role'] as string;

  if (auth.hasRole(requiredRole)) {
    return true;
  }

  return router.createUrlTree(['/unauthorized']);
};

// Route configuration
{
  path: 'admin',
  loadComponent: () => import('./admin/admin.component'),
  canActivate: [roleGuard],
  data: { role: 'admin' },
}
```

## `RedirectCommand` (Angular 18+)

`RedirectCommand` is a class that provides more control over redirect behavior than `UrlTree`. It accepts `NavigationBehaviorOptions` for fine-grained control.

### Basic Usage

```typescript
import { RedirectCommand, Router } from '@angular/router';

export const authGuard: CanActivateFn = (route, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (auth.isAuthenticated()) {
    return true;
  }

  const loginUrl = router.parseUrl('/login');
  return new RedirectCommand(loginUrl, {
    skipLocationChange: false,  // Update the browser URL
    // state: { returnUrl: state.url },  // Pass navigation state
  });
};
```

### `UrlTree` vs `RedirectCommand`

| Feature | `UrlTree` | `RedirectCommand` |
|---------|----------|-------------------|
| Simple redirect | Yes | Yes |
| `skipLocationChange` | No | Yes |
| `replaceUrl` | No | Yes |
| Navigation state (`state`) | No | Yes |
| When to use | Simple redirects | When you need navigation options |

```typescript
// UrlTree: simple redirect
return router.createUrlTree(['/login']);

// RedirectCommand: redirect with options
return new RedirectCommand(
  router.parseUrl('/login'),
  { replaceUrl: true }, // Replace history entry instead of pushing
);
```

## `CanActivateChildFn`

Protects all child routes of a parent. Runs once per navigation, regardless of how many child routes are activated.

```typescript
export const adminChildGuard: CanActivateChildFn = (childRoute, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (auth.hasRole('admin')) {
    return true;
  }

  return router.createUrlTree(['/unauthorized']);
};
```

```typescript
// Route configuration
{
  path: 'admin',
  loadComponent: () => import('./admin/admin-layout.component'),
  canActivateChild: [adminChildGuard],
  children: [
    { path: 'users', loadComponent: () => import('./admin/users.component') },
    { path: 'settings', loadComponent: () => import('./admin/settings.component') },
    { path: 'logs', loadComponent: () => import('./admin/logs.component') },
  ],
}
```

All three child routes are protected by `adminChildGuard` without repeating the guard on each child.

### `canActivate` vs `canActivateChild`

| Aspect | `canActivate` | `canActivateChild` |
|--------|-------------|-------------------|
| Protects | The route itself | All children of the route |
| Runs when | Navigating to this route | Navigating to any child route |
| Placed on | The route to protect | The parent route |
| Use case | Protect a specific page | Protect an entire section |

You can combine both:

```typescript
{
  path: 'admin',
  canActivate: [authGuard],           // Must be logged in to access admin
  canActivateChild: [adminRoleGuard], // Must be admin to access any child
  children: [ ... ],
}
```

## `CanDeactivateFn`

Prevents the user from leaving a route. The most common use case is warning about unsaved changes.

### Unsaved Changes Guard

```typescript
// Define an interface for components with unsaved changes
export interface HasUnsavedChanges {
  hasUnsavedChanges(): boolean;
}

export const unsavedChangesGuard: CanDeactivateFn<HasUnsavedChanges> = (
  component,
  currentRoute,
  currentState,
  nextState,
) => {
  if (component.hasUnsavedChanges()) {
    return confirm('You have unsaved changes. Leave this page?');
  }
  return true;
};
```

### Parameters

```typescript
const guard: CanDeactivateFn<MyComponent> = (
  component: MyComponent,            // The component instance being left
  currentRoute: ActivatedRouteSnapshot, // Current route info
  currentState: RouterStateSnapshot,    // Current full state
  nextState: RouterStateSnapshot,       // Where the user is trying to go
) => boolean | UrlTree | Observable<boolean | UrlTree> | Promise<boolean | UrlTree>;
```

### Implementing the Interface

```typescript
@Component({
  template: `
    <form>
      <input [(ngModel)]="name" />
      <button (click)="save()">Save</button>
    </form>
  `,
})
export class ProfileEditorComponent implements HasUnsavedChanges {
  private originalName = '';
  name = '';
  private saved = false;

  constructor() {
    // Load original data
    effect(() => {
      const user = this.userService.currentUser();
      if (user) {
        this.originalName = user.name;
        this.name = user.name;
      }
    });
  }

  hasUnsavedChanges(): boolean {
    return !this.saved && this.name !== this.originalName;
  }

  save() {
    this.userService.updateName(this.name);
    this.saved = true;
  }

  private userService = inject(UserService);
}
```

```typescript
// Route
{
  path: 'profile/edit',
  component: ProfileEditorComponent,
  canDeactivate: [unsavedChangesGuard],
}
```

### Signal-Based Unsaved Changes

A more modern approach using signals:

```typescript
export interface CanLeave {
  canLeave: Signal<boolean>;
}

export const canLeaveGuard: CanDeactivateFn<CanLeave> = (component) => {
  if (!component.canLeave()) {
    return confirm('You have unsaved changes. Leave this page?');
  }
  return true;
};
```

```typescript
@Component({ ... })
export class EditorComponent implements CanLeave {
  private dirty = signal(false);
  canLeave = computed(() => !this.dirty());

  onContentChange() {
    this.dirty.set(true);
  }

  save() {
    // ... save logic
    this.dirty.set(false);
  }
}
```

### Custom Confirmation Dialog

Instead of the browser `confirm()`, use a custom dialog:

```typescript
export const unsavedChangesGuard: CanDeactivateFn<HasUnsavedChanges> = (component) => {
  if (!component.hasUnsavedChanges()) {
    return true;
  }

  const dialog = inject(DialogService);
  return dialog.confirm({
    title: 'Unsaved Changes',
    message: 'You have unsaved changes. Do you want to leave without saving?',
    confirmText: 'Leave',
    cancelText: 'Stay',
  });
  // Returns Observable<boolean>
};
```

## `CanMatchFn`

Determines whether a route should be considered during URL matching. Unlike other guards, returning `false` does not cancel navigation. It tells the router to skip this route and try the next one.

### Feature Flag Routing

```typescript
export const featureGuard: CanMatchFn = (route, segments) => {
  const features = inject(FeatureFlagService);
  const flag = route.data?.['featureFlag'] as string;
  return features.isEnabled(flag);
};

const routes: Routes = [
  {
    path: 'checkout',
    loadComponent: () => import('./checkout-v2/checkout.component'),
    canMatch: [featureGuard],
    data: { featureFlag: 'newCheckout' },
  },
  {
    path: 'checkout',
    loadComponent: () => import('./checkout-v1/checkout.component'),
    // Fallback when feature flag is off
  },
];
```

### Role-Based Component Selection

```typescript
const isAdmin: CanMatchFn = () => inject(AuthService).hasRole('admin');
const isManager: CanMatchFn = () => inject(AuthService).hasRole('manager');

const routes: Routes = [
  {
    path: 'dashboard',
    loadComponent: () => import('./admin-dashboard.component'),
    canMatch: [isAdmin],
  },
  {
    path: 'dashboard',
    loadComponent: () => import('./manager-dashboard.component'),
    canMatch: [isManager],
  },
  {
    path: 'dashboard',
    loadComponent: () => import('./user-dashboard.component'),
    // Default for everyone else
  },
];
```

### `canMatch` Parameters

```typescript
const guard: CanMatchFn = (
  route: Route,          // The route configuration object (NOT ActivatedRouteSnapshot)
  segments: UrlSegment[], // The remaining URL segments to match
) => boolean | UrlTree | Observable<boolean | UrlTree> | Promise<boolean | UrlTree>;
```

Note: `canMatch` receives `Route` (the configuration), not `ActivatedRouteSnapshot`. This is because the route has not been activated yet.

## Guard Composition and Chaining

### Multiple Guards on One Route

Guards execute in array order. If any guard returns `false` or a `UrlTree`, subsequent guards do not run:

```typescript
{
  path: 'admin/settings',
  component: AdminSettingsComponent,
  canActivate: [authGuard, adminGuard, verifiedEmailGuard],
  // 1. authGuard runs first
  // 2. If authGuard returns true, adminGuard runs
  // 3. If adminGuard returns true, verifiedEmailGuard runs
  // 4. If all return true, navigation proceeds
}
```

### Higher-Order Guard Factory

Create parameterized guards using factory functions:

```typescript
export function requireRole(...roles: string[]): CanActivateFn {
  return (route, state) => {
    const auth = inject(AuthService);
    const router = inject(Router);

    if (roles.some(role => auth.hasRole(role))) {
      return true;
    }

    return router.createUrlTree(['/unauthorized']);
  };
}

// Usage
{
  path: 'reports',
  component: ReportsComponent,
  canActivate: [requireRole('admin', 'analyst')],
}

{
  path: 'billing',
  component: BillingComponent,
  canActivate: [requireRole('admin', 'finance')],
}
```

### Permission-Based Guard Factory

```typescript
export function requirePermission(permission: string): CanActivateFn {
  return () => {
    const permissions = inject(PermissionService);
    const router = inject(Router);

    if (permissions.has(permission)) {
      return true;
    }

    return router.createUrlTree(['/forbidden']);
  };
}

// Route configuration
{
  path: 'users',
  loadComponent: () => import('./users.component'),
  canActivate: [authGuard, requirePermission('users.read')],
},
{
  path: 'users/create',
  loadComponent: () => import('./create-user.component'),
  canActivate: [authGuard, requirePermission('users.create')],
},
```

## Async Guards

Guards can return `Promise` or `Observable` for async operations like API calls.

### Promise-Based Guard

```typescript
export const subscriptionGuard: CanActivateFn = async (route) => {
  const subscription = inject(SubscriptionService);
  const router = inject(Router);

  const plan = await subscription.getCurrentPlan();
  const requiredPlan = route.data['requiredPlan'] as string;

  if (plan.level >= getPlanLevel(requiredPlan)) {
    return true;
  }

  return router.createUrlTree(['/upgrade'], {
    queryParams: { plan: requiredPlan },
  });
};
```

### Observable-Based Guard

```typescript
import { map, take } from 'rxjs';

export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);
  const router = inject(Router);

  return auth.authState$.pipe(
    take(1), // Important: take only the first emission
    map(user => {
      if (user) return true;
      return router.createUrlTree(['/login']);
    }),
  );
};
```

> **Critical:** When returning an `Observable`, the router subscribes and uses the first emitted value. Always use `take(1)` or ensure the observable completes. An observable that never emits will block navigation indefinitely.

## Componentless Guard Application

Apply guards to componentless routes to protect entire sections:

```typescript
const routes: Routes = [
  // All authenticated routes grouped under a componentless parent
  {
    path: '',
    canActivate: [authGuard],
    children: [
      { path: 'dashboard', loadComponent: () => import('./dashboard.component') },
      { path: 'profile', loadComponent: () => import('./profile.component') },
      {
        path: 'admin',
        canActivate: [requireRole('admin')],
        children: [
          { path: '', loadComponent: () => import('./admin/dashboard.component') },
          { path: 'users', loadComponent: () => import('./admin/users.component') },
        ],
      },
    ],
  },
  // Public routes (no guard)
  { path: 'login', loadComponent: () => import('./login.component') },
  { path: 'register', loadComponent: () => import('./register.component') },
];
```

## Testing Functional Guards

### Basic Guard Test

```typescript
import { TestBed } from '@angular/core/testing';
import { authGuard } from './auth.guard';
import { AuthService } from './auth.service';
import { ActivatedRouteSnapshot, RouterStateSnapshot, Router } from '@angular/router';

describe('authGuard', () => {
  let authServiceSpy: jasmine.SpyObj<AuthService>;

  beforeEach(() => {
    authServiceSpy = jasmine.createSpyObj('AuthService', ['isAuthenticated']);

    TestBed.configureTestingModule({
      providers: [
        { provide: AuthService, useValue: authServiceSpy },
      ],
    });
  });

  it('should allow access when authenticated', () => {
    authServiceSpy.isAuthenticated.and.returnValue(true);

    const result = TestBed.runInInjectionContext(() => {
      return authGuard(
        {} as ActivatedRouteSnapshot,
        { url: '/dashboard' } as RouterStateSnapshot,
      );
    });

    expect(result).toBe(true);
  });

  it('should redirect to login when not authenticated', () => {
    authServiceSpy.isAuthenticated.and.returnValue(false);

    const result = TestBed.runInInjectionContext(() => {
      return authGuard(
        {} as ActivatedRouteSnapshot,
        { url: '/dashboard' } as RouterStateSnapshot,
      );
    });

    expect(result).toBeInstanceOf(UrlTree);
    const router = TestBed.inject(Router);
    expect((result as UrlTree).toString()).toBe('/login?returnUrl=%2Fdashboard');
  });
});
```

### Testing a Guard Factory

```typescript
describe('requireRole', () => {
  it('should allow access for matching role', () => {
    const authSpy = jasmine.createSpyObj('AuthService', ['hasRole']);
    authSpy.hasRole.and.returnValue(true);

    TestBed.configureTestingModule({
      providers: [{ provide: AuthService, useValue: authSpy }],
    });

    const guard = requireRole('admin');

    const result = TestBed.runInInjectionContext(() =>
      guard({} as ActivatedRouteSnapshot, {} as RouterStateSnapshot)
    );

    expect(result).toBe(true);
    expect(authSpy.hasRole).toHaveBeenCalledWith('admin');
  });
});
```

## Common Mistakes

### Mistake 1: Navigating Instead of Returning a UrlTree

```typescript
// BAD: Navigate programmatically and return false
export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (!auth.isAuthenticated()) {
    router.navigate(['/login']); // Race condition!
    return false;
  }
  return true;
};

// GOOD: Return a UrlTree
export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (!auth.isAuthenticated()) {
    return router.createUrlTree(['/login']);
  }
  return true;
};
```

Returning `false` and navigating separately creates a race condition where the cancellation and the new navigation compete.

### Mistake 2: Forgetting `take(1)` on Observable Guards

```typescript
// BAD: Observable never completes, navigation hangs
export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);
  return auth.user$.pipe(
    map(user => !!user),
  );
  // user$ is a BehaviorSubject that never completes
};

// GOOD: Take only the first emission
export const authGuard: CanActivateFn = () => {
  const auth = inject(AuthService);
  return auth.user$.pipe(
    take(1),
    map(user => !!user),
  );
};
```

### Mistake 3: Using Class-Based Guards (Deprecated)

```typescript
// DEPRECATED: Class-based guard
@Injectable({ providedIn: 'root' })
export class AuthGuard implements CanActivate {
  constructor(private auth: AuthService, private router: Router) {}

  canActivate(route: ActivatedRouteSnapshot, state: RouterStateSnapshot) {
    // ...
  }
}

// MODERN: Functional guard
export const authGuard: CanActivateFn = (route, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);
  // ...
};
```

Class-based guards still work but are deprecated. Functional guards are simpler, more composable, and align with Angular's functional API direction.

### Mistake 4: Relying Solely on Client-Side Guards

```typescript
// BAD: Guard is the only access control
export const adminGuard: CanActivateFn = () => {
  return inject(AuthService).isAdmin();
};
// User can bypass this by disabling JavaScript or modifying the app

// GOOD: Guard is a UX convenience, server enforces authorization
// The guard prevents the user from seeing the admin UI
// The API returns 403 if the user is not an admin
```

Guards are a user experience feature, not a security boundary. Always enforce authorization on the server.

### Mistake 5: Overusing Guards

```typescript
// BAD: Guard on every single route
{
  path: 'settings',
  canActivate: [authGuard],
  children: [
    { path: 'profile', canActivate: [authGuard], component: ProfileComponent },
    { path: 'billing', canActivate: [authGuard], component: BillingComponent },
    { path: 'notifications', canActivate: [authGuard], component: NotificationsComponent },
  ],
}

// GOOD: Guard on the parent protects all children
{
  path: 'settings',
  canActivate: [authGuard],
  children: [
    { path: 'profile', component: ProfileComponent },
    { path: 'billing', component: BillingComponent },
    { path: 'notifications', component: NotificationsComponent },
  ],
}
```

### Mistake 6: Conflicting Guards on `canMatch`

```typescript
// BAD: canMatch returns UrlTree (confusing behavior)
const guard: CanMatchFn = () => {
  return router.createUrlTree(['/login']);
  // canMatch with UrlTree redirects but is semantically odd
};

// GOOD: canMatch returns boolean -- let canActivate handle redirects
const matchGuard: CanMatchFn = () => inject(AuthService).isAuthenticated();
const activateGuard: CanActivateFn = () => {
  if (!inject(AuthService).isAuthenticated()) {
    return inject(Router).createUrlTree(['/login']);
  }
  return true;
};
```

## Best Practices

### 1. Return `UrlTree` or `RedirectCommand` Instead of `false` + `navigate`

This is the most important pattern. Let the router handle the redirect atomically.

### 2. Use Guard Factories for Parameterized Logic

`requireRole('admin')`, `requirePermission('users.write')` are cleaner and more reusable than creating separate guard functions for each role.

### 3. Place Guards at the Highest Applicable Level

If all children require auth, put the guard on the parent. Do not repeat it on every child route.

### 4. Use `canActivateChild` for Section-Wide Protection

When an entire section (admin, settings) needs the same guard, `canActivateChild` on the parent is cleaner than `canActivate` on each child.

### 5. Use `canMatch` for Conditional Route Selection

When the same URL should load different components based on state (feature flags, user roles), `canMatch` is the right tool.

### 6. Keep Guards Focused

Each guard should check one thing. Compose multiple single-purpose guards rather than building one guard that checks everything.

## Real-World Use Cases

### Authentication with Return URL

```typescript
export const authGuard: CanActivateFn = (route, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (auth.isAuthenticated()) {
    return true;
  }

  return router.createUrlTree(['/login'], {
    queryParams: { returnUrl: state.url },
  });
};

// Login component reads returnUrl after successful login
@Component({ ... })
export class LoginComponent {
  private router = inject(Router);
  returnUrl = input('/', { alias: 'returnUrl' });

  async onLogin(credentials: Credentials) {
    await this.authService.login(credentials);
    this.router.navigateByUrl(this.returnUrl());
  }
}
```

### Subscription Tier Gating

```typescript
export function requirePlan(minimumPlan: 'free' | 'pro' | 'enterprise'): CanActivateFn {
  const planLevels = { free: 0, pro: 1, enterprise: 2 };

  return async () => {
    const subscription = inject(SubscriptionService);
    const router = inject(Router);

    const currentPlan = await subscription.getCurrentPlan();

    if (planLevels[currentPlan] >= planLevels[minimumPlan]) {
      return true;
    }

    return router.createUrlTree(['/upgrade'], {
      queryParams: { required: minimumPlan },
    });
  };
}

// Routes
{
  path: 'analytics',
  loadComponent: () => import('./analytics.component'),
  canActivate: [authGuard, requirePlan('pro')],
},
{
  path: 'api-access',
  loadComponent: () => import('./api-access.component'),
  canActivate: [authGuard, requirePlan('enterprise')],
},
```

### Unsaved Changes with Custom Dialog

```typescript
export const unsavedChangesGuard: CanDeactivateFn<HasUnsavedChanges> = (
  component,
  currentRoute,
  currentState,
  nextState,
) => {
  if (!component.hasUnsavedChanges()) {
    return true;
  }

  const dialog = inject(DialogService);
  return dialog.open(ConfirmDialogComponent, {
    data: {
      title: 'Unsaved Changes',
      message: 'You have unsaved changes that will be lost.',
      actions: [
        { label: 'Discard Changes', value: true, style: 'warn' },
        { label: 'Keep Editing', value: false, style: 'primary' },
      ],
    },
  }).afterClosed();
};
```

---

> **Key Takeaway:** Functional guards are simple functions that run in an injection context. Return `UrlTree` or `RedirectCommand` for redirects, use factory functions for parameterized guards, place guards at the highest applicable route level, and always enforce authorization on the server -- client-side guards are a UX feature, not a security boundary.
