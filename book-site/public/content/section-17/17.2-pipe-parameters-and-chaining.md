# 17.2 Pipe Parameters and Chaining

> Pipes accept parameters to customize their behavior, and multiple pipes can be chained together to create transformation pipelines. Understanding the syntax, execution order, and operator precedence is essential for writing correct and efficient templates.

---

## Pipe Parameter Syntax

Pipes accept parameters separated by colons (`:`). The value to the left of the pipe operator is always the first argument. Additional arguments follow the pipe name, each preceded by a colon.

```html
<!-- No parameters -->
{{ value | pipeName }}

<!-- One parameter -->
{{ value | pipeName:param1 }}

<!-- Two parameters -->
{{ value | pipeName:param1:param2 }}

<!-- Three parameters -->
{{ value | pipeName:param1:param2:param3 }}
```

### How Parameters Map to the Transform Method

Every pipe implements a `transform()` method. The template value becomes the first argument, and colon-separated parameters become subsequent arguments.

```typescript
// Pipe definition
@Pipe({ name: 'myPipe', standalone: true })
export class MyPipe implements PipeTransform {
  transform(value: string, prefix: string, suffix: string): string {
    return `${prefix}${value}${suffix}`;
  }
}
```

```html
<!-- Usage: value is 'World', prefix is 'Hello ', suffix is '!' -->
{{ 'World' | myPipe:'Hello ':'!' }}
<!-- Output: Hello World! -->
```

The mapping is straightforward:

| Template Expression | `transform()` Call |
|---|---|
| `value \| pipeName` | `transform(value)` |
| `value \| pipeName:a` | `transform(value, a)` |
| `value \| pipeName:a:b` | `transform(value, a, b)` |
| `value \| pipeName:a:b:c` | `transform(value, a, b, c)` |

### Parameter Types

Parameters can be any valid template expression: literals, component properties, method calls, or even nested expressions.

```typescript
@Component({
  standalone: true,
  imports: [DatePipe, CurrencyPipe, SlicePipe],
  template: `
    <!-- String literal parameter -->
    {{ today | date:'yyyy-MM-dd' }}

    <!-- Component property as parameter -->
    {{ today | date:dateFormat }}

    <!-- Expression as parameter -->
    {{ price | currency:(isEuro ? 'EUR' : 'USD') }}

    <!-- Numeric literal -->
    {{ items | slice:0:pageSize }}

    <!-- Computed property or signal -->
    {{ items | slice:0:itemsPerPage() }}
  `
})
export class ExampleComponent {
  today = new Date();
  dateFormat = 'fullDate';
  price = 42.99;
  isEuro = false;
  items = ['a', 'b', 'c', 'd', 'e'];
  pageSize = 3;
  itemsPerPage = signal(3);
}
```

### Parentheses in Parameters

When a parameter expression contains operators that could be ambiguous, wrap it in parentheses:

```html
<!-- Ternary operator as parameter needs parentheses -->
{{ amount | currency:(useEuro ? 'EUR' : 'USD'):'symbol' }}

<!-- Without parentheses, this is ambiguous -->
<!-- BAD: Angular may misinterpret the colons -->
{{ amount | currency:useEuro ? 'EUR' : 'USD':'symbol' }}
```

### Optional Parameters

Pipes can define optional parameters with default values:

```typescript
@Pipe({ name: 'truncate', standalone: true })
export class TruncatePipe implements PipeTransform {
  transform(value: string, limit: number = 50, trail: string = '...'): string {
    if (value.length <= limit) {
      return value;
    }
    return value.substring(0, limit) + trail;
  }
}
```

```html
<!-- Uses defaults: limit=50, trail='...' -->
{{ longText | truncate }}

<!-- Custom limit, default trail -->
{{ longText | truncate:20 }}

<!-- Custom limit and trail -->
{{ longText | truncate:20:'---' }}
```

### Passing Objects and Arrays as Parameters

You can pass object literals and arrays, but the syntax requires care:

```typescript
@Pipe({ name: 'highlight', standalone: true })
export class HighlightPipe implements PipeTransform {
  transform(text: string, config: { color: string; bold: boolean }): string {
    // Implementation
    return text;
  }
}
```

```html
<!-- Object literal as parameter - must use a component property -->
<!-- This does NOT work inline due to template expression limitations -->
<!-- BAD: {{ text | highlight:{ color: 'red', bold: true } }} -->

<!-- GOOD: Define the config in the component class -->
{{ text | highlight:highlightConfig }}
```

```typescript
// In component class
highlightConfig = { color: 'red', bold: true };
```

Inline object literals with curly braces are not supported in Angular template expressions. Always define complex parameter objects as component properties or signals.

## Pipe Chaining

Multiple pipes can be applied sequentially using the `|` operator. Each pipe receives the output of the previous pipe as its input.

```html
{{ value | pipe1 | pipe2 | pipe3 }}
```

### Execution Order

Pipes execute left to right. The output of each pipe becomes the input of the next:

```html
{{ 'hello world' | titlecase | uppercase }}
```

Step-by-step:
1. Input: `'hello world'`
2. After `titlecase`: `'Hello World'`
3. After `uppercase`: `'HELLO WORLD'`

```html
{{ 'ANGULAR PIPES' | lowercase | titlecase }}
```

1. Input: `'ANGULAR PIPES'`
2. After `lowercase`: `'angular pipes'`
3. After `titlecase`: `'Angular Pipes'`

### Chaining with Parameters

Each pipe in the chain can have its own parameters:

```html
{{ birthday | date:'fullDate' | uppercase }}
<!-- Output: SUNDAY, MARCH 15, 2026 -->

{{ price | currency:'EUR':'symbol':'1.2-2' | uppercase }}
<!-- Output: EUR42.99 (uppercase has no effect on numbers/symbols) -->

{{ longDescription | slice:0:100 | uppercase }}
<!-- First 100 characters, then uppercased -->
```

### Chaining Order Matters

The order of pipes affects the result. Consider:

```html
<!-- Slice first, then format -->
{{ items | slice:0:5 | json }}
<!-- Shows JSON of first 5 items -->

<!-- JSON first, then slice = slicing a string! -->
{{ items | json | slice:0:50 }}
<!-- Shows first 50 characters of the JSON string -->
```

Another example demonstrating order significance:

```html
{{ 'hello beautiful world' | uppercase | slice:0:5 }}
<!-- Result: HELLO -->

{{ 'hello beautiful world' | slice:0:5 | uppercase }}
<!-- Result: HELLO -->
<!-- Same result here, but not always the case -->

{{ 'hello beautiful world' | slice:6:15 | titlecase }}
<!-- Result: Beautiful -->

{{ 'hello beautiful world' | titlecase | slice:6:15 }}
<!-- Result: Beautiful -->
<!-- Same here, but consider: -->

{{ 'hello-world' | titlecase | slice:0:5 }}
<!-- Result: Hello (titlecase does not capitalize after hyphen) -->
```

### Chaining with @if and @for

Pipes work seamlessly with Angular's control flow:

```html
@if (users | slice:0:1 | json; as firstUser) {
  <!-- Note: this is the JSON string of the first user -->
}

@for (user of users | slice:start:end; track user.id) {
  <div>{{ user.name | titlecase }}</div>
}
```

### Using the `as` Keyword with Pipes

When using `@if` with `async`, you can alias the result to avoid repeated pipe usage:

```html
@if (data$ | async; as data) {
  <h1>{{ data.title | titlecase }}</h1>
  <p>{{ data.description }}</p>
  <span>{{ data.date | date:'mediumDate' }}</span>
}
```

## Operator Precedence

The pipe operator (`|`) sits at a specific point in Angular's operator precedence hierarchy. Binary operators like `+`, `-`, `*`, `/` have **higher** precedence than the pipe, so they evaluate first. But the pipe has **higher** precedence than the ternary operator (`?:`), which creates a major gotcha.

### Binary Operators Evaluate Before Pipes

```html
<!-- Addition happens first, then pipe -->
{{ 1 + 2 | number }}
<!-- Evaluates as: {{ (1 + 2) | number }} = "3" -->

<!-- String concatenation happens first -->
{{ 'hello' + ' ' + 'world' | uppercase }}
<!-- Evaluates as: {{ ('hello' + ' ' + 'world') | uppercase }} = "HELLO WORLD" -->

<!-- Property access happens first -->
{{ user.name | uppercase }}
<!-- Evaluates as: {{ (user.name) | uppercase }} -->

<!-- Method call happens first -->
{{ getName() | uppercase }}
<!-- Evaluates as: {{ (getName()) | uppercase }} -->
```

### The Ternary Operator Gotcha

The pipe operator has **higher** precedence than the ternary operator. This means the pipe binds to the value immediately to its left, not to the entire ternary expression:

```html
<!-- BAD: Pipe binds tighter than ternary -->
{{ isActive ? 'active' : 'inactive' | uppercase }}
<!-- Evaluates as: {{ isActive ? 'active' : ('inactive' | uppercase) }} -->
<!-- When isActive=true:  "active" (not uppercased!) -->
<!-- When isActive=false: "INACTIVE" -->
```

```html
<!-- GOOD: Use parentheses to apply pipe to the entire expression -->
{{ (isActive ? 'active' : 'inactive') | uppercase }}
<!-- Output: "ACTIVE" or "INACTIVE" -->
```

This is one of the most common bugs with pipes. Always wrap ternary expressions in parentheses when piping the result. This behavior is different from AngularJS, where the pipe applied to the entire ternary expression without parentheses.

### Nullish Coalescing and Pipes

The nullish coalescing operator (`??`) has higher precedence than the pipe, but this creates its own trap:

```html
<!-- ?? evaluates before | so this works as expected -->
{{ user?.name ?? 'Anonymous' | uppercase }}
<!-- Evaluates as: {{ (user?.name ?? 'Anonymous') | uppercase }} -->
<!-- Both branches get uppercased -->

<!-- But be careful with ?? AFTER a pipe -->
{{ value | date:'YYYY-MM-dd' ?? 'Unknown' }}
<!-- Evaluates as: {{ value | date:('YYYY-MM-dd' ?? 'Unknown') }} -->
<!-- The ?? is being applied to the PARAMETER, not the pipe output! -->
<!-- 'YYYY-MM-dd' is never null, so ?? is useless here -->

<!-- GOOD: Use parentheses to apply ?? to the pipe result -->
{{ (value | date:'YYYY-MM-dd') ?? 'Unknown' }}
```

### Logical Operators and Pipes

```html
<!-- || has higher precedence than pipe -->
{{ value || 'default' | uppercase }}
<!-- Evaluates as: {{ (value || 'default') | uppercase }} -->
<!-- This actually works as expected because || evaluates first -->

<!-- But be explicit with parentheses for clarity -->
{{ (value || 'default') | uppercase }}
```

### Comparison Operators and Pipes

You cannot use the pipe operator inside expressions that Angular interprets as part of bindings:

```html
<!-- This does NOT work - pipe in a property binding expression -->
<!-- BAD: <div [class.active]="status | uppercase === 'ACTIVE'"> -->

<!-- GOOD: Use a computed signal or getter instead -->
<div [class.active]="isActive()">
```

```typescript
status = signal('active');
isActive = computed(() => this.status().toUpperCase() === 'ACTIVE');
```

## Null and Undefined Through Pipe Chains

When `null` or `undefined` flows through a chain, behavior depends on each pipe's implementation:

```html
{{ nullValue | date:'fullDate' | uppercase }}
```

1. `nullValue` is `null`
2. `DatePipe` receives `null`, returns `null`
3. `UpperCasePipe` receives `null`, returns `null`
4. Template renders nothing

Most built-in pipes handle `null`/`undefined` gracefully by returning `null`. But custom pipes may not:

```typescript
// BAD: Custom pipe that crashes on null
@Pipe({ name: 'double', standalone: true })
export class DoublePipe implements PipeTransform {
  transform(value: number): number {
    return value * 2; // TypeError if value is null/undefined
  }
}

// GOOD: Null-safe custom pipe
@Pipe({ name: 'double', standalone: true })
export class DoublePipe implements PipeTransform {
  transform(value: number | null | undefined): number | null {
    if (value == null) return null;
    return value * 2;
  }
}
```

### Safe Navigation with Pipes

```html
<!-- Safe navigation operator works before the pipe -->
{{ user?.address?.city | uppercase }}
<!-- If any part is null/undefined, pipe receives null -->

<!-- Combining with fallback -->
{{ (user?.address?.city ?? 'Unknown') | uppercase }}
```

## Dynamic Pipe Parameters

Parameters can be reactive, using signals or component properties that change:

```typescript
@Component({
  standalone: true,
  imports: [DatePipe],
  template: `
    <select (change)="onFormatChange($event)">
      <option value="short">Short</option>
      <option value="medium">Medium</option>
      <option value="full">Full</option>
    </select>
    <p>{{ today | date:selectedFormat() }}</p>
  `
})
export class DynamicFormatComponent {
  today = new Date();
  selectedFormat = signal('medium');

  onFormatChange(event: Event) {
    const value = (event.target as HTMLSelectElement).value;
    this.selectedFormat.set(value);
  }
}
```

Since `DatePipe` is pure, it re-evaluates when either the date value or the format parameter changes. Signals trigger change detection, which causes the pipe to check if its inputs have changed.

### Dynamic Currency Switching

```typescript
@Component({
  standalone: true,
  imports: [CurrencyPipe],
  template: `
    <div class="currency-toggle">
      <button (click)="currency.set('USD')">USD</button>
      <button (click)="currency.set('EUR')">EUR</button>
      <button (click)="currency.set('GBP')">GBP</button>
    </div>
    <p class="price">{{ price | currency:currency():'symbol' }}</p>
  `
})
export class PriceComponent {
  price = 99.99;
  currency = signal('USD');
}
```

## Performance Considerations

### Pure Pipe Caching

Pure pipes (the default) use reference equality to decide whether to re-execute. When chaining pure pipes, each pipe in the chain independently checks if its input has changed:

```html
{{ data | pipeA | pipeB | pipeC }}
```

On each change detection cycle:
1. Angular checks if `data` reference changed
2. If yes, `pipeA` re-executes, producing a new output
3. `pipeB` sees a new input reference, re-executes
4. `pipeC` sees a new input reference, re-executes
5. If `data` has not changed, none of the pipes re-execute

This makes chaining pure pipes efficient. The entire chain is skipped when the source value has not changed.

### Impure Pipes in Chains

If any pipe in a chain is impure, it runs on every change detection cycle. This forces all downstream pipes to also re-evaluate because the impure pipe produces a new reference each time:

```html
<!-- AsyncPipe is impure - it runs every cycle -->
<!-- SlicePipe is pure but gets new input from async each emission -->
@for (item of items$ | async | slice:0:5; track item.id) {
  <div>{{ item.name }}</div>
}
```

### Avoid Long Chains for Complex Transformations

When you find yourself chaining more than 2-3 pipes, consider moving the logic to a `computed()` signal or a dedicated custom pipe:

```html
<!-- BAD: Too many chained pipes, hard to read and debug -->
{{ rawText | stripHtml | truncate:100 | titlecase | highlight:searchTerm }}
```

```typescript
// GOOD: Use a computed signal for complex transformations
formattedText = computed(() => {
  let text = this.stripHtml(this.rawText());
  text = text.substring(0, 100);
  text = this.toTitleCase(text);
  return this.highlight(text, this.searchTerm());
});
```

```html
{{ formattedText() }}
```

Or create a single custom pipe that encapsulates the entire transformation.

## Pipes in Different Template Contexts

### In Property Bindings

```html
<!-- Pipes work in property bindings -->
<img [alt]="productName | uppercase" [src]="imageUrl">

<!-- Pipes in attribute bindings -->
<div [attr.aria-label]="description | truncate:50"></div>

<!-- Pipes in style bindings -->
<div [style.width]="(progress | percent) + '%'">
  <!-- Note: PercentPipe returns a string like "75%",
       so you may need a custom approach -->
</div>
```

### In Event Bindings

Pipes cannot be used in event binding expressions:

```html
<!-- BAD: This does NOT work -->
<!-- <button (click)="handleClick($event | myPipe)">Click</button> -->

<!-- Pipes are for display transformations, not event handling -->
```

### In Two-Way Bindings

Pipes cannot be used with two-way binding because pipes are one-directional transformations:

```html
<!-- BAD: This does NOT work -->
<!-- <input [(ngModel)]="name | uppercase"> -->

<!-- GOOD: Use a setter or signal effect for transformation -->
<input [ngModel]="name | uppercase" (ngModelChange)="name = $event">
```

### In Template Variables

```html
<!-- Pipe result can be captured with @if...as -->
@if (longText | truncate:50; as truncated) {
  <p>{{ truncated }}</p>
  <button (click)="showFull = true">Read more</button>
}
```

## Common Mistakes

- **Forgetting parentheses around ternary expressions before pipes.** `condition ? a : b | pipe` only pipes `b`, not the whole expression. Always use `(condition ? a : b) | pipe`.

- **Assuming pipe chains are evaluated right to left.** Chains evaluate left to right. `value | a | b` means "apply `a` first, then `b` to the result."

- **Passing inline object literals as pipe parameters.** Angular template expressions do not support `{}` object literals. Define complex parameters as component properties.

- **Creating overly long pipe chains.** More than 3 pipes chained together becomes hard to read and debug. Use a `computed()` signal or a custom pipe that combines the transformations.

- **Using pipes in event binding expressions.** Pipes transform display values only. They cannot be used in `(click)`, `(change)`, or other event handlers.

- **Ignoring null safety in custom pipe chains.** If a pipe in the middle of a chain returns `null`, the next pipe must handle `null` input or the chain breaks with a runtime error.

---

## Best Practices

- **Always use parentheses when mixing pipes with ternary, nullish coalescing, or logical operators.** This prevents subtle bugs from operator precedence.

- **Keep chains short (2-3 pipes maximum).** For more complex transformations, use `computed()` signals or a single custom pipe.

- **Pass dynamic parameters via signals.** This ensures the pipe re-evaluates reactively when configuration changes.

- **Make custom pipes null-safe.** Check for `null`/`undefined` before processing, especially when the pipe may appear in a chain after other pipes that can return `null`.

- **Prefer property/signal references over inline expressions for complex parameters.** This keeps templates readable and makes parameters easier to test.

## Real-World Use Cases

### Configurable Data Table Cells

```typescript
@Component({
  standalone: true,
  imports: [DatePipe, CurrencyPipe, DecimalPipe, UpperCasePipe, SlicePipe],
  template: `
    <table>
      <thead>
        <tr>
          <th>Name</th>
          <th>Revenue</th>
          <th>Growth</th>
          <th>Joined</th>
          <th>Status</th>
        </tr>
      </thead>
      <tbody>
        @for (row of data | slice:pageStart():pageEnd(); track row.id) {
          <tr>
            <td>{{ row.name | titlecase }}</td>
            <td>{{ row.revenue | currency:currency():'symbol':'1.0-0' }}</td>
            <td [class.positive]="row.growth > 0">
              {{ row.growth | percent:'1.1-1' }}
            </td>
            <td>{{ row.joinedAt | date:dateFormat() }}</td>
            <td>{{ row.status | uppercase }}</td>
          </tr>
        }
      </tbody>
    </table>
  `
})
export class DataTableComponent {
  data = input.required<Row[]>();
  currency = signal('USD');
  dateFormat = signal('mediumDate');
  page = signal(0);
  pageSize = signal(10);

  pageStart = computed(() => this.page() * this.pageSize());
  pageEnd = computed(() => this.pageStart() + this.pageSize());
}
```

### User Profile Display with Fallbacks

```typescript
@Component({
  standalone: true,
  imports: [TitleCasePipe, DatePipe, UpperCasePipe, SlicePipe],
  template: `
    <div class="profile">
      <h2>{{ (user().displayName ?? user().email) | titlecase }}</h2>
      <p class="bio">
        {{ (user().bio ?? 'No bio provided') | slice:0:200 }}
        @if ((user().bio?.length ?? 0) > 200) {
          <button (click)="showFullBio.set(true)">Read more</button>
        }
      </p>
      <div class="meta">
        <span>Role: {{ user().role | uppercase }}</span>
        <span>Member since {{ user().createdAt | date:'MMMM yyyy' }}</span>
      </div>
    </div>
  `
})
export class ProfileComponent {
  user = input.required<UserProfile>();
  showFullBio = signal(false);
}
```

### Search Results with Highlighted Term

```typescript
@Pipe({ name: 'highlight', standalone: true })
export class HighlightPipe implements PipeTransform {
  transform(text: string | null, term: string): string {
    if (!text || !term) return text ?? '';
    const regex = new RegExp(`(${term})`, 'gi');
    return text.replace(regex, '<mark>$1</mark>');
  }
}

@Component({
  standalone: true,
  imports: [HighlightPipe, SlicePipe, TitleCasePipe],
  template: `
    @for (result of results() | slice:0:20; track result.id) {
      <div class="search-result">
        <h3 [innerHTML]="result.title | titlecase | highlight:searchTerm()"></h3>
        <p [innerHTML]="result.snippet | highlight:searchTerm()"></p>
      </div>
    }
  `
})
export class SearchResultsComponent {
  results = input.required<SearchResult[]>();
  searchTerm = input.required<string>();
}
```

In this example, the chain `titlecase | highlight` applies title casing first, then wraps matching terms in `<mark>` tags. The order matters because `highlight` injects HTML that should not be altered by subsequent text pipes.

---

> **Key Takeaway:** Pipe parameters are colon-separated values that map directly to `transform()` method arguments, and pipes chain left to right. The most critical rule: always wrap ternary and logical expressions in parentheses before piping, because the pipe operator has lower precedence than you might expect.
