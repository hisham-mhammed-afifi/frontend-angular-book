# 17.6 Pipes vs Methods in Templates

> Choosing between pipes, methods, getters, and computed signals for template transformations directly impacts performance. Understanding how each interacts with change detection is essential for building fast Angular applications.

---

## The Problem with Methods in Templates

When you call a method in a template expression, Angular executes that method on **every change detection cycle**. Angular has no way to know whether the method's return value has changed without calling it.

```typescript
@Component({
  standalone: true,
  template: `
    <p>{{ getFullName() }}</p>
    <p>{{ formatPrice(product.price) }}</p>
    @for (item of getFilteredItems(); track item.id) {
      <div>{{ item.name }}</div>
    }
  `
})
export class ProductComponent {
  user = { firstName: 'John', lastName: 'Doe' };
  product = { price: 42.99 };
  items: Item[] = [];

  getFullName(): string {
    console.log('getFullName called'); // Called EVERY CD cycle
    return `${this.user.firstName} ${this.user.lastName}`;
  }

  formatPrice(price: number): string {
    console.log('formatPrice called'); // Called EVERY CD cycle
    return `$${price.toFixed(2)}`;
  }

  getFilteredItems(): Item[] {
    console.log('getFilteredItems called'); // Called EVERY CD cycle
    return this.items.filter(i => i.active);
  }
}
```

Open the console and interact with the page. You will see these methods called repeatedly, even when the underlying data has not changed. Every click, every keystroke, every mouse movement that triggers change detection causes all three methods to re-execute.

### Why This Matters

For simple string concatenation, the cost is negligible. But for operations involving:
- Array filtering or sorting
- Complex string formatting
- Object transformation
- Calculations with iteration

The cost compounds. If change detection runs 20 times per second (common during user interaction), and you have 5 method calls in the template, each doing a non-trivial operation, that is 100 unnecessary function executions per second.

---

## How Pure Pipes Solve This

Pure pipes cache their result and only re-execute when the input value changes (by reference for objects, by value for primitives).

```typescript
@Pipe({ name: 'fullName', standalone: true })
export class FullNamePipe implements PipeTransform {
  transform(user: { firstName: string; lastName: string }): string {
    console.log('fullName pipe called'); // Only when user reference changes
    return `${user.firstName} ${user.lastName}`;
  }
}
```

```html
<p>{{ user | fullName }}</p>
```

The pipe's `transform()` is called once when `user` is first rendered, and then only again when the `user` reference changes. During all other change detection cycles, Angular returns the cached result without calling `transform()`.

### Side-by-Side Comparison

```typescript
@Component({
  standalone: true,
  imports: [FullNamePipe],
  template: `
    <!-- Method: called every CD cycle -->
    <p>{{ getFullName() }}</p>

    <!-- Pipe: called only when user reference changes -->
    <p>{{ user | fullName }}</p>

    <button (click)="onUnrelatedClick()">Click me</button>
  `
})
export class ComparisonComponent {
  user = { firstName: 'John', lastName: 'Doe' };

  getFullName(): string {
    console.log('method called');
    return `${this.user.firstName} ${this.user.lastName}`;
  }

  onUnrelatedClick() {
    // This triggers change detection
    // Method: re-executes (unnecessary)
    // Pipe: skipped (input unchanged)
  }
}
```

Clicking the button triggers change detection. The method re-executes. The pipe does not.

---

## How Computed Signals Solve This

`computed()` signals are the modern Angular 21 approach. They track their reactive dependencies and only recalculate when a dependency changes.

```typescript
@Component({
  standalone: true,
  template: `
    <p>{{ fullName() }}</p>
    <button (click)="onUnrelatedClick()">Click me</button>
  `
})
export class SignalComponent {
  firstName = signal('John');
  lastName = signal('Doe');

  fullName = computed(() => {
    console.log('computed evaluated'); // Only when firstName or lastName changes
    return `${this.firstName()} ${this.lastName()}`;
  });

  onUnrelatedClick() {
    // Does NOT trigger fullName recalculation
    // The computed signal knows its dependencies haven't changed
  }
}
```

Computed signals are:
- **Lazy:** Only computed when read.
- **Memoized:** Cached until a dependency changes.
- **Dependency-tracked:** Angular knows exactly which signals affect the result.
- **Not tied to change detection cycles:** They recalculate based on signal graph changes, not CD timing.

## Getters in Templates

TypeScript getters behave exactly like methods in templates. Angular calls the getter on every change detection cycle.

```typescript
@Component({
  standalone: true,
  template: `
    <p>{{ fullName }}</p>
  `
})
export class GetterComponent {
  firstName = 'John';
  lastName = 'Doe';

  // Called EVERY change detection cycle, just like a method
  get fullName(): string {
    console.log('getter called');
    return `${this.firstName} ${this.lastName}`;
  }
}
```

Getters look like properties in the template (`fullName` not `fullName()`), which makes them deceptive. Developers often assume they behave like cached properties, but they do not.

### Getter vs Method: No Difference in Performance

```typescript
// These are functionally identical in Angular's change detection:
get total(): number { return this.items.reduce((sum, i) => sum + i.price, 0); }
getTotal(): number { return this.items.reduce((sum, i) => sum + i.price, 0); }
```

Both execute on every CD cycle. The getter is not cached. The only difference is template syntax: `{{ total }}` vs `{{ getTotal() }}`.

---

## Full Comparison Table

| Approach | Re-evaluates | Caching | Reusability | Template Syntax | Recommended |
|----------|-------------|---------|-------------|-----------------|-------------|
| Method call | Every CD cycle | None | Component only | `{{ method() }}` | Avoid for expensive ops |
| Getter | Every CD cycle | None | Component only | `{{ getter }}` | Avoid for expensive ops |
| Pure pipe | When input changes | Yes (reference) | Across components | `{{ value \| pipe }}` | Good for reusable transforms |
| Impure pipe | Every CD cycle | None (by default) | Across components | `{{ value \| pipe }}` | Avoid |
| `computed()` | When dependency changes | Yes (value) | Component (or service) | `{{ signal() }}` | Best for component logic |

### Execution Frequency in Practice

Consider a component with a text input and a list display:

```
User types 10 characters in an input field.
Each keystroke triggers change detection.
```

| Approach | Executions |
|----------|-----------|
| Method in template | 10 |
| Getter in template | 10 |
| Pure pipe (input unchanged) | 0 |
| Computed signal (deps unchanged) | 0 |
| Impure pipe | 10 |

For a method that filters a 1000-item array taking 2ms each call, that is 20ms of wasted computation just from typing. With a pure pipe or computed signal, it is 0ms.

## When Methods in Templates Are Acceptable

Methods in templates are not always bad. They are acceptable when:

### 1. The Operation Is Trivially Cheap

```html
<!-- Simple boolean check: negligible cost -->
<button [disabled]="isFormInvalid()">Submit</button>
```

```typescript
isFormInvalid(): boolean {
  return !this.name || !this.email; // Two property checks
}
```

### 2. The Method Is an Event Handler (Not a Binding)

```html
<!-- Event handlers are fine: they run once per event, not per CD cycle -->
<button (click)="handleSubmit()">Submit</button>
<input (input)="onSearchChange($event)">
```

Event bindings `(click)`, `(input)`, etc. are fundamentally different from interpolation or property bindings. They execute once when the event fires, not on every change detection cycle.

### 3. You Are Using OnPush with Controlled Change Detection

With `OnPush`, change detection runs less frequently. Methods still execute on every CD cycle for that component, but the cycles themselves are less frequent:

```typescript
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<p>{{ computeValue() }}</p>`
})
export class OnPushComponent {
  // Runs less often because OnPush limits CD cycles
  // But still runs unnecessarily when CD triggers for other reasons
  computeValue(): string { ... }
}
```

Even with OnPush, a computed signal or pipe is still better.

## Migration Patterns

### Method to Pipe

```typescript
// BEFORE: Method in template
@Component({
  template: `<p>{{ formatDate(order.createdAt) }}</p>`
})
export class OrderComponent {
  formatDate(date: Date): string {
    return date.toLocaleDateString('en-US', {
      year: 'numeric', month: 'short', day: 'numeric'
    });
  }
}

// AFTER: Built-in pipe
@Component({
  imports: [DatePipe],
  template: `<p>{{ order.createdAt | date:'mediumDate' }}</p>`
})
export class OrderComponent {}
```

### Method to Computed Signal

```typescript
// BEFORE: Method for filtering
@Component({
  template: `
    @for (item of getActiveItems(); track item.id) {
      <div>{{ item.name }}</div>
    }
  `
})
export class ListComponent {
  items: Item[] = [];

  getActiveItems(): Item[] {
    return this.items.filter(i => i.active);
  }
}

// AFTER: Computed signal
@Component({
  template: `
    @for (item of activeItems(); track item.id) {
      <div>{{ item.name }}</div>
    }
  `
})
export class ListComponent {
  items = signal<Item[]>([]);

  activeItems = computed(() => this.items().filter(i => i.active));
}
```

### Getter to Computed Signal

```typescript
// BEFORE: Getter
@Component({
  template: `<p>{{ totalPrice }}</p>`
})
export class CartComponent {
  cartItems: CartItem[] = [];

  get totalPrice(): number {
    return this.cartItems.reduce((sum, item) => sum + item.price * item.quantity, 0);
  }
}

// AFTER: Computed signal
@Component({
  template: `<p>{{ totalPrice() }}</p>`
})
export class CartComponent {
  cartItems = signal<CartItem[]>([]);

  totalPrice = computed(() =>
    this.cartItems().reduce((sum, item) => sum + item.price * item.quantity, 0)
  );
}
```

### Complex Method to Custom Pipe

```typescript
// BEFORE: Complex transformation method
@Component({
  template: `<p>{{ formatFileSize(file.bytes) }}</p>`
})
export class FileComponent {
  formatFileSize(bytes: number): string {
    if (bytes === 0) return '0 B';
    const k = 1024;
    const units = ['B', 'KB', 'MB', 'GB'];
    const i = Math.floor(Math.log(bytes) / Math.log(k));
    return `${(bytes / Math.pow(k, i)).toFixed(1)} ${units[i]}`;
  }
}

// AFTER: Reusable pipe
@Pipe({ name: 'fileSize', standalone: true })
export class FileSizePipe implements PipeTransform {
  transform(bytes: number | null, decimals: number = 1): string {
    if (bytes == null || bytes === 0) return '0 B';
    const k = 1024;
    const units = ['B', 'KB', 'MB', 'GB'];
    const i = Math.floor(Math.log(bytes) / Math.log(k));
    return `${(bytes / Math.pow(k, i)).toFixed(decimals)} ${units[i]}`;
  }
}

@Component({
  imports: [FileSizePipe],
  template: `<p>{{ file.bytes | fileSize }}</p>`
})
export class FileComponent {}
```

## Decision Guide

Use this to choose the right approach:

### Use a Built-in Pipe When:
- Angular already provides the transformation (date, currency, number, percent, case, slice, json)
- No custom logic needed

### Use a Custom Pipe When:
- The transformation is reusable across multiple components
- It takes a single value and returns a transformed value (pure function)
- You want declarative, readable template syntax

### Use a Computed Signal When:
- The transformation depends on multiple reactive values
- The result is specific to one component
- You need the value in both template and TypeScript logic
- You want reactive dependency tracking

### Use a Method/Getter Only When:
- The operation is trivially cheap (simple boolean check, property access)
- It is used in a single place and not worth extracting
- You understand the performance implications

## Profiling Template Performance

### Using Angular DevTools

Angular DevTools (browser extension) shows:
- Change detection frequency for each component
- Time spent in change detection per component
- Which components are checked most often

### Manual Instrumentation

```typescript
// Add console.count to measure execution frequency
getFilteredItems(): Item[] {
  console.count('getFilteredItems');
  return this.items.filter(i => i.active);
}
```

Interact with the page and check the console. If you see the count increasing rapidly without data changes, the method is running unnecessarily.

### Performance Comparison Test

```typescript
@Component({
  standalone: true,
  imports: [FileSizePipe],
  template: `
    <!-- Test 1: Method (check console for call count) -->
    @for (file of files; track file.id) {
      <span>{{ formatSize(file.bytes) }}</span>
    }

    <!-- Test 2: Pipe (check console for call count) -->
    @for (file of files; track file.id) {
      <span>{{ file.bytes | fileSize }}</span>
    }

    <!-- Test 3: Computed (no unnecessary calls) -->
    @for (file of formattedFiles(); track file.id) {
      <span>{{ file.size }}</span>
    }
  `
})
export class BenchmarkComponent {
  files = Array.from({ length: 100 }, (_, i) => ({
    id: i,
    bytes: Math.floor(Math.random() * 1_000_000_000)
  }));

  formattedFiles = computed(() =>
    this.files.map(f => ({
      ...f,
      size: this.formatSizeInternal(f.bytes)
    }))
  );

  formatSize(bytes: number): string {
    console.count('method');
    return this.formatSizeInternal(bytes);
  }

  private formatSizeInternal(bytes: number): string {
    const k = 1024;
    const units = ['B', 'KB', 'MB', 'GB'];
    const i = Math.floor(Math.log(bytes) / Math.log(k));
    return `${(bytes / Math.pow(k, i)).toFixed(1)} ${units[i]}`;
  }
}
```

With 100 files and a mouse movement triggering 10 CD cycles:
- **Method:** 1000 calls (100 files x 10 cycles)
- **Pipe:** 100 calls (once per file, cached thereafter)
- **Computed:** 0 calls after initial (files signal did not change)

## Impact with OnPush and Zoneless

### Default Change Detection + Method

Worst case. Every event anywhere in the app triggers CD, and every method in the template executes.

### OnPush + Method

Better. CD runs only when inputs change or events fire in this component. Methods still run on every CD cycle for this component, but cycles are less frequent.

### OnPush + Pure Pipe

Good. Fewer CD cycles, and pipes skip execution when inputs are unchanged.

### Signals + Computed

Best. No CD cycle dependency at all. Computed signals recalculate only when their signal dependencies change. Works optimally in both zone-based and zoneless Angular.

### Zoneless + Method

In zoneless Angular (default in Angular 21), CD runs less frequently because Zone.js is not patching async operations. Methods in templates still execute on every CD cycle, but those cycles are triggered only by signals, events, and explicit `markForCheck()` calls. The impact is reduced compared to zone-based apps, but computed signals and pipes remain the better choice.

---

## Common Mistakes

- **Using getters as a "cheaper" alternative to methods.** Getters execute on every change detection cycle, exactly like methods. They offer no caching. The only difference is syntax.

- **Assuming OnPush eliminates the problem.** OnPush reduces how often change detection runs, but methods and getters still execute on every cycle within that component. A computed signal or pure pipe is still better.

- **Using methods for array filtering in `@for` loops.** This is the most expensive pattern: the filter runs for every item on every CD cycle. Always use `computed()` or a pure pipe with immutable data.

- **Creating a pipe for a one-off transformation.** If the transformation is unique to one component and depends on multiple component properties, a `computed()` signal is simpler and more appropriate than a pipe.

- **Calling `console.log` or expensive operations inside template methods during development.** These calls persist in production if not removed and can cause significant performance degradation.

---

## Best Practices

- **Use `computed()` signals as the default for derived values.** They are the most performant option in Angular 21 and work naturally with the signal-based architecture.

- **Use built-in pipes for standard formatting** (dates, numbers, currency, casing). Do not recreate what Angular already provides.

- **Create custom pipes for reusable transformations** that are used across multiple components. Pipes are the right abstraction for "take a value, return a transformed value."

- **Reserve methods for event handlers only.** Methods bound to `(click)`, `(input)`, and other events run once per event, not per CD cycle, so they are not a performance concern.

- **Profile before optimizing.** Use `console.count()` or Angular DevTools to measure actual execution frequency. Optimize the methods that run most often and cost the most.

- **Migrate getters to computed signals.** When refactoring, replace `get x()` patterns with `x = computed(() => ...)` for automatic caching and dependency tracking.

## Real-World Use Cases

### Data Table with Formatting

```typescript
@Component({
  standalone: true,
  imports: [DatePipe, CurrencyPipe, DecimalPipe],
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <table>
      @for (order of orders(); track order.id) {
        <tr>
          <!-- GOOD: Pipes for standard formatting -->
          <td>{{ order.date | date:'shortDate' }}</td>
          <td>{{ order.total | currency:'USD' }}</td>
          <td>{{ order.itemCount | number }}</td>

          <!-- GOOD: Computed signal for derived state -->
          <td>{{ orderStatus(order) }}</td>
        </tr>
      }
      <tr class="totals">
        <!-- GOOD: Computed signal for aggregation -->
        <td colspan="2">Total</td>
        <td>{{ grandTotal() | currency:'USD' }}</td>
      </tr>
    </table>
  `
})
export class OrderTableComponent {
  orders = input.required<Order[]>();

  grandTotal = computed(() =>
    this.orders().reduce((sum, o) => sum + o.total, 0)
  );

  // Simple, cheap lookup - acceptable as a method
  orderStatus(order: Order): string {
    return order.shipped ? 'Shipped' : 'Pending';
  }
}
```

### User Profile Card

```typescript
// BAD: Methods for everything
@Component({
  template: `
    <div class="profile">
      <h2>{{ getDisplayName() }}</h2>
      <p>{{ getMembershipDuration() }}</p>
      <p>{{ getFormattedBio() }}</p>
      <span class="badge">{{ getRoleBadge() }}</span>
    </div>
  `
})
export class BadProfileComponent {
  user: User = { ... };

  getDisplayName() { return `${this.user.firstName} ${this.user.lastName}`; }
  getMembershipDuration() { /* date calculation */ }
  getFormattedBio() { return this.user.bio?.substring(0, 200) + '...'; }
  getRoleBadge() { return this.user.role.toUpperCase(); }
}

// GOOD: Signals and pipes
@Component({
  imports: [UpperCasePipe, TruncatePipe],
  template: `
    <div class="profile">
      <h2>{{ displayName() }}</h2>
      <p>{{ membershipDuration() }}</p>
      <p>{{ user().bio | truncate:200 }}</p>
      <span class="badge">{{ user().role | uppercase }}</span>
    </div>
  `
})
export class GoodProfileComponent {
  user = input.required<User>();

  displayName = computed(() => {
    const u = this.user();
    return `${u.firstName} ${u.lastName}`;
  });

  membershipDuration = computed(() => {
    const joined = new Date(this.user().joinedAt);
    const months = Math.floor(
      (Date.now() - joined.getTime()) / (1000 * 60 * 60 * 24 * 30)
    );
    return months < 12 ? `${months} months` : `${Math.floor(months / 12)} years`;
  });
}
```

### Search Results with Performance

```typescript
@Component({
  standalone: true,
  imports: [HighlightPipe, TruncatePipe, DatePipe],
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <input (input)="onSearch($event)" placeholder="Search...">

    @for (result of filteredResults(); track result.id) {
      <article class="result">
        <!-- Pipes for display transformations -->
        <h3 [innerHTML]="result.title | highlight:searchTerm()"></h3>
        <p>{{ result.excerpt | truncate:150 }}</p>
        <time>{{ result.publishedAt | date:'mediumDate' }}</time>
      </article>
    }

    <p class="count">{{ resultCount() }} results found</p>
  `
})
export class SearchResultsComponent {
  private results = input.required<SearchResult[]>();
  searchTerm = signal('');

  // Computed: recalculates only when results or searchTerm changes
  filteredResults = computed(() => {
    const term = this.searchTerm().toLowerCase();
    if (!term) return this.results();
    return this.results().filter(r =>
      r.title.toLowerCase().includes(term) ||
      r.excerpt.toLowerCase().includes(term)
    );
  });

  resultCount = computed(() => this.filteredResults().length);

  onSearch(event: Event) {
    this.searchTerm.set((event.target as HTMLInputElement).value);
  }
}
```

In this example:
- `filteredResults()` uses `computed()` because it depends on two signals and involves filtering.
- `highlight`, `truncate`, and `date` use pipes because they are reusable display transformations.
- No methods are called in template bindings.

---

> **Key Takeaway:** Methods and getters in templates execute on every change detection cycle with no caching, making them a hidden performance tax. Pure pipes cache by input reference, and `computed()` signals cache by dependency tracking. In Angular 21, use `computed()` for component-specific derived state and pipes for reusable display formatting.
