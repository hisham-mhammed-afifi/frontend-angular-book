# 17.7 Internationalization with Pipes

> Angular's locale-aware pipes automatically format dates, numbers, currencies, and percentages according to regional conventions. Configuring locale data correctly is the foundation for serving a global audience.

---

## How Angular Pipes Use Locale

Angular's formatting pipes (`DatePipe`, `DecimalPipe`, `CurrencyPipe`, `PercentPipe`) use the `LOCALE_ID` injection token to determine formatting rules. The default locale is `en-US`.

Each locale defines:
- Decimal separators (`.` vs `,`)
- Thousands grouping separators (`,` vs `.` vs ` `)
- Currency symbol and position
- Date format patterns (month/day order, day names, month names)
- AM/PM indicators
- First day of the week

These rules come from the Unicode CLDR (Common Locale Data Repository), which Angular pre-packages as locale data files.

```typescript
// Default behavior: en-US
{{ 1234567.89 | number }}      // "1,234,567.89"
{{ 0.42 | percent }}           // "42%"
{{ 99.99 | currency }}         // "$99.99"
{{ today | date:'mediumDate' }} // "Mar 15, 2026"
```

## Setting the Application Locale

### Step 1: Import Locale Data

Angular includes locale data for `en-US` by default. For any other locale, you must import and register the data.

```typescript
// app.config.ts
import { LOCALE_ID } from '@angular/core';
import { registerLocaleData } from '@angular/common';
import localeFr from '@angular/common/locales/fr';
import localeFrExtra from '@angular/common/locales/extra/fr';

// Register the locale data
registerLocaleData(localeFr, 'fr-FR', localeFrExtra);

export const appConfig: ApplicationConfig = {
  providers: [
    { provide: LOCALE_ID, useValue: 'fr-FR' }
  ]
};
```

The `registerLocaleData()` function accepts:
1. The locale data module (required)
2. A locale ID string (optional, defaults to the one in the data)
3. Extra locale data for extended date/time formats (optional)

### Step 2: Provide LOCALE_ID

```typescript
// app.config.ts
import { LOCALE_ID } from '@angular/core';

export const appConfig: ApplicationConfig = {
  providers: [
    { provide: LOCALE_ID, useValue: 'de-DE' }
  ]
};
```

### What Changes with Locale

| Expression | `en-US` | `de-DE` | `fr-FR` | `ar-EG` |
|-----------|---------|---------|---------|---------|
| `1234.56 \| number` | `1,234.56` | `1.234,56` | `1 234,56` | `١٬٢٣٤٫٥٦` |
| `0.75 \| percent` | `75%` | `75 %` | `75 %` | `٧٥٪` |
| `42 \| currency:'USD'` | `$42.00` | `42,00 $` | `42,00 $US` | `٤٢٫٠٠ US$` |
| `date \| date:'mediumDate'` | `Mar 15, 2026` | `15.03.2026` | `15 mars 2026` | `١٥ مارس ٢٠٢٦` |

Notice that the locale affects not just the format, but also:
- Number digit characters (Arabic-Indic numerals in `ar-EG`)
- Symbol position (`$42` vs `42 $`)
- Currency symbol (`$` vs `$US`)
- Date component order and separators

## Locale Data Structure

Angular's locale data files contain arrays of formatting rules. Here is what a locale file provides:

```typescript
// Simplified structure of locale data
{
  localeId: 'fr-FR',
  // Number formatting
  NumberSymbols: {
    decimal: ',',
    group: '\u202f',  // narrow no-break space
    percentSign: '%',
    minusSign: '-',
    // ...
  },
  // Currency
  CurrencySymbols: {
    'EUR': ['€', 'EUR'],
    'USD': ['$US', '$'],
    // ...
  },
  // Date/time formats
  DateTimePatterns: {
    short: 'dd/MM/y',
    medium: 'd MMM y',
    long: 'd MMMM y',
    full: 'EEEE d MMMM y',
    // ...
  },
  // Day and month names
  DayNames: ['dimanche', 'lundi', 'mardi', ...],
  MonthNames: ['janvier', 'février', 'mars', ...],
  // ...
}
```

## Per-Pipe Locale Override

Every locale-aware pipe accepts a `locale` parameter that overrides `LOCALE_ID` for that specific usage.

### DecimalPipe

```html
<!-- App locale is en-US, but format this number as German -->
{{ 1234.56 | number:'1.2-2':'de-DE' }}
<!-- Output: 1.234,56 -->

<!-- Force French formatting -->
{{ 1234.56 | number:'1.0-0':'fr-FR' }}
<!-- Output: 1 235 (rounded, narrow no-break space as grouping) -->
```

**Syntax:** `{{ value | number[:digitsInfo[:locale]] }}`

### CurrencyPipe

```html
<!-- App locale is en-US, but format as Japanese Yen for Japanese users -->
{{ 1500 | currency:'JPY':'symbol':'1.0-0':'ja-JP' }}
<!-- Output: ¥1,500 -->

<!-- Euro in German format -->
{{ 42.50 | currency:'EUR':'symbol':'1.2-2':'de-DE' }}
<!-- Output: 42,50 € -->
```

**Syntax:** `{{ value | currency[:currencyCode[:display[:digitsInfo[:locale]]]] }}`

### PercentPipe

```html
{{ 0.1234 | percent:'1.1-2':'fr-FR' }}
<!-- Output: 12,34 % (French uses comma for decimal, space before %) -->
```

**Syntax:** `{{ value | percent[:digitsInfo[:locale]] }}`

### DatePipe

```html
<!-- Force German date format regardless of app locale -->
{{ today | date:'fullDate':undefined:'de-DE' }}
<!-- Output: Sonntag, 15. März 2026 -->

<!-- Japanese date -->
{{ today | date:'mediumDate':undefined:'ja-JP' }}
<!-- Output: 2026/03/15 -->
```

**Syntax:** `{{ value | date[:format[:timezone[:locale]]] }}`

Note: To specify locale without timezone, pass `undefined` for the timezone parameter.

## DEFAULT_CURRENCY_CODE

The `DEFAULT_CURRENCY_CODE` injection token sets the default currency code used by `CurrencyPipe` when no code is explicitly provided.

```typescript
import { DEFAULT_CURRENCY_CODE } from '@angular/core';

export const appConfig: ApplicationConfig = {
  providers: [
    { provide: DEFAULT_CURRENCY_CODE, useValue: 'EUR' }
  ]
};
```

```html
<!-- Without DEFAULT_CURRENCY_CODE: defaults to 'USD' -->
{{ 42 | currency }}
<!-- Output: $42.00 -->

<!-- With DEFAULT_CURRENCY_CODE set to 'EUR': -->
{{ 42 | currency }}
<!-- Output: €42.00 -->

<!-- Explicit code always overrides the default -->
{{ 42 | currency:'GBP' }}
<!-- Output: £42.00 -->
```

### Currency Display Modes

The `display` parameter controls how the currency is shown:

```html
{{ 42 | currency:'EUR':'code' }}
<!-- Output: EUR42.00 -->

{{ 42 | currency:'EUR':'symbol' }}
<!-- Output: €42.00 -->

{{ 42 | currency:'EUR':'symbol-narrow' }}
<!-- Output: €42.00 (narrow variant, same for EUR) -->

{{ 42 | currency:'CAD':'symbol' }}
<!-- Output: CA$42.00 (distinguishes from USD $) -->

{{ 42 | currency:'CAD':'symbol-narrow' }}
<!-- Output: $42.00 (narrow: just $, may be ambiguous) -->
```

### Currency Fraction Digits

Different currencies have different default fraction digits based on ISO 4217:

```html
{{ 42 | currency:'USD' }}   <!-- $42.00 (2 fraction digits) -->
{{ 42 | currency:'JPY' }}   <!-- ¥42 (0 fraction digits) -->
{{ 42 | currency:'BHD' }}   <!-- BHD42.000 (3 fraction digits) -->
{{ 42 | currency:'KRW' }}   <!-- ₩42 (0 fraction digits) -->
```

Angular automatically uses the correct number of fraction digits for each currency unless you override with `digitsInfo`.

## DATE_PIPE_DEFAULT_OPTIONS

Configure application-wide defaults for `DatePipe`:

```typescript
import { DATE_PIPE_DEFAULT_OPTIONS } from '@angular/common';

export const appConfig: ApplicationConfig = {
  providers: [
    {
      provide: DATE_PIPE_DEFAULT_OPTIONS,
      useValue: {
        dateFormat: 'dd/MM/yyyy',  // Default format
        timezone: '+0000'          // Force UTC
      }
    }
  ]
};
```

```html
<!-- Uses the configured default format -->
{{ today | date }}
<!-- Output: 15/03/2026 (instead of default 'mediumDate') -->

<!-- Explicit format overrides the default -->
{{ today | date:'fullDate' }}
<!-- Output: Sunday, March 15, 2026 -->
```

This is useful when your application consistently uses a specific date format and you want to avoid repeating it in every template.

## Multi-Locale Application

For applications that support multiple languages, you need a strategy for switching locales.

### Build-Time Locale (Angular i18n)

Angular's built-in i18n compiles a separate bundle per locale. Each bundle has the correct `LOCALE_ID` baked in at build time.

```json
// angular.json
{
  "projects": {
    "my-app": {
      "i18n": {
        "sourceLocale": "en-US",
        "locales": {
          "fr": "src/locale/messages.fr.xlf",
          "de": "src/locale/messages.de.xlf"
        }
      }
    }
  }
}
```

```bash
# Build for all locales
ng build --localize
# Produces: dist/my-app/en-US/, dist/my-app/fr/, dist/my-app/de/
```

Each build has the correct locale data registered and pipes format according to that locale automatically.

### Runtime Locale Switching

For single-build applications that switch locale at runtime (using libraries like Transloco or ngx-translate):

```typescript
import { LOCALE_ID } from '@angular/core';
import { registerLocaleData } from '@angular/common';
import localeEn from '@angular/common/locales/en';
import localeFr from '@angular/common/locales/fr';
import localeDe from '@angular/common/locales/de';

// Register all supported locales at startup
registerLocaleData(localeEn);
registerLocaleData(localeFr);
registerLocaleData(localeDe);

// Dynamic locale provider
export const appConfig: ApplicationConfig = {
  providers: [
    {
      provide: LOCALE_ID,
      useFactory: () => {
        // Read from localStorage, cookie, or URL
        return localStorage.getItem('locale') ?? 'en-US';
      }
    }
  ]
};
```

**Important limitation:** `LOCALE_ID` is provided at application bootstrap. Changing it requires reloading the application. For true runtime switching without reload, you must override the locale parameter on each pipe usage or use a custom pipe that reads the current locale dynamically.

### Dynamic Locale Pipe

```typescript
@Pipe({ name: 'localizedDate', standalone: true })
export class LocalizedDatePipe implements PipeTransform {
  private datePipe = inject(DatePipe);
  private localeService = inject(LocaleService);

  transform(value: Date | string | number, format?: string): string | null {
    return this.datePipe.transform(value, format, undefined, this.localeService.currentLocale);
  }
}
```

```typescript
// locale.service.ts
@Injectable({ providedIn: 'root' })
export class LocaleService {
  currentLocale = 'en-US';

  setLocale(locale: string) {
    this.currentLocale = locale;
    // Note: You'd need to trigger a re-render
  }
}
```

This approach is limited because pure pipes will not re-run when the service's locale changes (same pipe inputs). See section 17.4 for workarounds.

## I18nPluralPipe for Locale-Aware Pluralization

`I18nPluralPipe` maps a numeric value to a string based on ICU pluralization rules:

```typescript
@Component({
  standalone: true,
  imports: [I18nPluralPipe],
  template: `
    <p>{{ itemCount | i18nPlural:itemMapping }}</p>
  `
})
export class CartSummaryComponent {
  itemCount = 0;

  itemMapping: { [k: string]: string } = {
    '=0': 'Your cart is empty',
    '=1': 'You have 1 item',
    'other': 'You have # items'
  };
}
```

### ICU Plural Categories

The ICU standard defines six plural categories, though most languages use only a subset:

| Category | When Used | Languages |
|----------|-----------|-----------|
| `zero` | For 0 | Arabic, Latvian |
| `one` | For 1 (or locale-specific singular) | English, German, French |
| `two` | For 2 | Arabic, Hebrew |
| `few` | For small numbers | Russian (2-4), Polish (2-4), Czech |
| `many` | For larger numbers | Russian (5-20), Polish (5-21) |
| `other` | Catch-all (required) | All languages |

Exact matches (`=0`, `=1`, `=42`) take priority over category matches.

### Pluralization Example for Multiple Languages

```typescript
// English
itemMappingEn = {
  '=0': 'No items',
  '=1': '1 item',
  'other': '# items'
};

// Russian (has complex plural rules)
itemMappingRu = {
  '=0': 'Нет элементов',
  '=1': '1 элемент',
  'few': '# элемента',      // 2-4, 22-24, etc.
  'many': '# элементов',     // 5-20, 25-30, etc.
  'other': '# элемента'
};

// Arabic (has dual form)
itemMappingAr = {
  '=0': 'لا عناصر',
  '=1': 'عنصر واحد',
  'two': 'عنصران',
  'few': '# عناصر',
  'many': '# عنصرًا',
  'other': '# عنصر'
};
```

## I18nSelectPipe for Gender and Category Selection

`I18nSelectPipe` chooses a string based on a string key. Commonly used for gender-dependent text:

```typescript
@Component({
  standalone: true,
  imports: [I18nSelectPipe],
  template: `
    <p>{{ user.gender | i18nSelect:inviteMapping }}</p>
  `
})
export class InviteComponent {
  user = { name: 'Alex', gender: 'other' };

  inviteMapping = {
    'male': 'Invite him to the event.',
    'female': 'Invite her to the event.',
    'other': 'Invite them to the event.'
  };
}
```

### Combining with I18nPluralPipe

```html
<p>
  {{ user.gender | i18nSelect:pronounMapping }}
  {{ user.notificationCount | i18nPlural:notificationMapping }}
</p>
```

```typescript
pronounMapping = {
  'male': 'He has',
  'female': 'She has',
  'other': 'They have'
};

notificationMapping = {
  '=0': 'no new notifications.',
  '=1': '1 new notification.',
  'other': '# new notifications.'
};
// Output: "She has 5 new notifications."
```

## Using the Native Intl API

Modern browsers provide the `Intl` API, which Angular's pipes use internally. You can also use `Intl` directly in custom pipes for more control:

### Intl.NumberFormat

```typescript
@Pipe({ name: 'compactNumber', standalone: true })
export class CompactNumberPipe implements PipeTransform {
  private locale = inject(LOCALE_ID);

  transform(value: number | null, notation: 'compact' | 'standard' = 'compact'): string {
    if (value == null) return '';
    return new Intl.NumberFormat(this.locale, {
      notation,
      compactDisplay: 'short'
    }).format(value);
  }
}
```

```html
{{ 1500 | compactNumber }}        <!-- "1.5K" (en-US) -->
{{ 1500 | compactNumber }}        <!-- "1,5 Tsd." (de-DE) -->
{{ 2500000 | compactNumber }}     <!-- "2.5M" (en-US) -->
```

### Intl.RelativeTimeFormat

```typescript
@Pipe({ name: 'relativeTime', standalone: true })
export class RelativeTimePipe implements PipeTransform {
  private locale = inject(LOCALE_ID);

  transform(value: Date | string | number | null): string {
    if (!value) return '';
    const date = new Date(value);
    const diffMs = date.getTime() - Date.now();
    const diffSec = Math.round(diffMs / 1000);

    const rtf = new Intl.RelativeTimeFormat(this.locale, { numeric: 'auto' });

    const abs = Math.abs(diffSec);
    if (abs < 60) return rtf.format(diffSec, 'second');
    if (abs < 3600) return rtf.format(Math.round(diffSec / 60), 'minute');
    if (abs < 86400) return rtf.format(Math.round(diffSec / 3600), 'hour');
    return rtf.format(Math.round(diffSec / 86400), 'day');
  }
}
```

```html
{{ comment.createdAt | relativeTime }}
<!-- en-US: "3 hours ago" -->
<!-- fr-FR: "il y a 3 heures" -->
<!-- de-DE: "vor 3 Stunden" -->
<!-- ja-JP: "3 時間前" -->
```

### Intl.ListFormat

```typescript
@Pipe({ name: 'listFormat', standalone: true })
export class ListFormatPipe implements PipeTransform {
  private locale = inject(LOCALE_ID);

  transform(
    items: string[] | null,
    type: 'conjunction' | 'disjunction' = 'conjunction'
  ): string {
    if (!items || items.length === 0) return '';
    return new Intl.ListFormat(this.locale, {
      style: 'long',
      type
    }).format(items);
  }
}
```

```html
{{ ['Alice', 'Bob', 'Charlie'] | listFormat }}
<!-- en-US: "Alice, Bob, and Charlie" -->
<!-- fr-FR: "Alice, Bob et Charlie" -->
<!-- de-DE: "Alice, Bob und Charlie" -->
<!-- ja-JP: "Alice、Bob、Charlie" -->

{{ ['red', 'blue'] | listFormat:'disjunction' }}
<!-- en-US: "red or blue" -->
<!-- fr-FR: "rouge ou bleu" -->
```

## Configuring Locale at Different Levels

### Application-Wide

```typescript
// app.config.ts
providers: [
  { provide: LOCALE_ID, useValue: 'fr-FR' },
  { provide: DEFAULT_CURRENCY_CODE, useValue: 'EUR' },
  {
    provide: DATE_PIPE_DEFAULT_OPTIONS,
    useValue: { dateFormat: 'dd/MM/yyyy' }
  }
]
```

### Per-Component Override

A component can override the locale for itself and its children:

```typescript
@Component({
  standalone: true,
  imports: [DatePipe, CurrencyPipe],
  providers: [
    { provide: LOCALE_ID, useValue: 'ja-JP' },
    { provide: DEFAULT_CURRENCY_CODE, useValue: 'JPY' }
  ],
  template: `
    <p>{{ today | date:'mediumDate' }}</p>  <!-- 2026/03/15 -->
    <p>{{ price | currency }}</p>            <!-- ¥1,500 -->
  `
})
export class JapaneseWidgetComponent {
  today = new Date();
  price = 1500;
}
```

This component and its children use Japanese formatting, even if the app-wide locale is different.

### Per-Pipe Usage Override

```html
<!-- Override just this one pipe call -->
{{ price | currency:'EUR':'symbol':'1.2-2':'de-DE' }}
```

## Common Mistakes

- **Forgetting to call `registerLocaleData()`.** Setting `LOCALE_ID` without registering the locale data causes pipes to fall back to `en-US` silently or throw a "Missing locale data" warning. Always import and register the data before providing the `LOCALE_ID`.

- **Assuming `LOCALE_ID` can be changed at runtime.** `LOCALE_ID` is injected at bootstrap time. Changing it after the application starts has no effect on pipes. For runtime locale switching, you need to either reload the application or use per-pipe locale parameters.

- **Using narrow currency symbols in multi-currency contexts.** `'symbol-narrow'` uses `$` for both USD and CAD, making them indistinguishable. Use `'symbol'` (which gives `CA$` vs `$`) or `'code'` when displaying multiple currencies.

- **Not registering extra locale data for extended date formats.** The `long` and `full` date/time formats and day period names (morning, afternoon) require extra locale data. Without it, these formats fall back to defaults.

- **Hardcoding format strings that conflict with locale conventions.** Using `'MM/dd/yyyy'` as a custom date format violates the day-first convention of most non-US locales. Prefer predefined formats (`'mediumDate'`, `'shortDate'`) that adapt to the locale.

- **Ignoring the `'other'` key in `I18nPluralPipe` mappings.** The `'other'` category is required as a fallback. Omitting it causes the pipe to return an empty string for unmatched values.

- **Not considering RTL text direction.** Arabic, Hebrew, and other RTL locales may need additional CSS handling for formatted text. Pipes handle the text content, but layout direction (`dir="rtl"`) must be set manually; Angular does not set it from `LOCALE_ID`.

- **Parsing ISO date strings without timezone info.** `new Date('2026-03-15')` is parsed as UTC midnight, so `DatePipe` in a non-UTC timezone may display the previous day. Use `'2026-03-15T00:00:00'` (local time) or `'2026/03/15'` (slash format) to parse in local time.

- **Using IANA timezone names with DatePipe.** DatePipe only understands `UTC`, `GMT`, and continental US abbreviations. IANA names like `'America/Chicago'` silently fall back to the local timezone. Use offset strings like `'+0530'` instead.

- **Registering locale data inside a lazy-loaded module.** `registerLocaleData()` must be called at the global scope (typically `main.ts` or `app.config.ts`). Calling it inside a lazy-loaded module has no effect.

---

## Best Practices

- **Use predefined date formats (`'short'`, `'medium'`, `'long'`, `'full'`) instead of custom format strings.** Predefined formats adapt to the locale automatically. Custom strings like `'MM/dd/yyyy'` ignore locale conventions.

- **Set `LOCALE_ID` and `DEFAULT_CURRENCY_CODE` at the application level** in `app.config.ts`. This ensures consistent formatting throughout the application.

- **Use `DATE_PIPE_DEFAULT_OPTIONS` for application-wide date format consistency.** This eliminates repeated format strings in templates.

- **Register all supported locale data at startup.** Import and register locale data for every language your application supports to avoid runtime surprises.

- **Leverage the native `Intl` API for advanced formatting** (compact notation, relative time, list formatting) that Angular's built-in pipes do not cover. Inject `LOCALE_ID` to stay consistent with the app locale.

- **Test your application in all supported locales.** Number formatting, date ordering, and currency positioning vary dramatically between locales. Visual testing in each locale catches issues that unit tests miss.

- **Watch for the future `Intl` API migration.** Angular has plans (GitHub #54470) to migrate from shipping CLDR data files to using the browser's native `Intl` API. This would eliminate the need for `registerLocaleData()` and reduce bundle size. Design custom pipes around `Intl` to be future-compatible.

## Real-World Use Cases

### Multi-Currency E-Commerce

```typescript
@Component({
  standalone: true,
  imports: [CurrencyPipe, DecimalPipe],
  template: `
    <div class="price-card">
      <p class="price">
        {{ product.price | currency:userCurrency():'symbol':'1.2-2':userLocale() }}
      </p>
      @if (product.originalPrice > product.price) {
        <p class="original">
          {{ product.originalPrice | currency:userCurrency():'symbol':'1.2-2':userLocale() }}
        </p>
        <p class="savings">
          Save {{ savings() | currency:userCurrency():'symbol':'1.0-0':userLocale() }}
        </p>
      }
      <p class="rating">{{ product.rating | number:'1.1-1':userLocale() }}</p>
    </div>
  `
})
export class PriceCardComponent {
  product = input.required<Product>();
  userCurrency = input<string>('USD');
  userLocale = input<string>('en-US');

  savings = computed(() =>
    this.product().originalPrice - this.product().price
  );
}
```

### Locale-Aware Dashboard

```typescript
@Component({
  standalone: true,
  imports: [DatePipe, DecimalPipe, PercentPipe, CompactNumberPipe],
  template: `
    <div class="dashboard">
      <div class="stat">
        <label>Revenue</label>
        <span>{{ stats().revenue | compactNumber }}</span>
      </div>
      <div class="stat">
        <label>Growth</label>
        <span [class.positive]="stats().growth > 0">
          {{ stats().growth | percent:'1.1-1' }}
        </span>
      </div>
      <div class="stat">
        <label>Users</label>
        <span>{{ stats().activeUsers | number }}</span>
      </div>
      <div class="stat">
        <label>Last Updated</label>
        <span>{{ stats().updatedAt | date:'medium' }}</span>
      </div>
    </div>
  `
})
export class DashboardComponent {
  stats = input.required<DashboardStats>();
}
```

All formatting adapts automatically based on the configured `LOCALE_ID`. No explicit locale parameters needed when the app-wide setting is correct.

### User Profile with Pluralization

```typescript
@Component({
  standalone: true,
  imports: [I18nPluralPipe, I18nSelectPipe, DatePipe, RelativeTimePipe],
  template: `
    <div class="profile">

      <p>{{ user().followerCount | i18nPlural:followerMapping }}</p>

      <p>Last active {{ user().lastActiveAt | relativeTime }}</p>
    </div>
  `
})
export class ProfileComponent {
  user = input.required<UserProfile>();

  bioMapping = {
    'male': 'He is a software developer.',
    'female': 'She is a software developer.',
    'other': 'They are a software developer.'
  };

  followerMapping = {
    '=0': 'No followers yet',
    '=1': '1 follower',
    'other': '# followers'
  };

  postMapping = {
    '=0': 'No posts published',
    '=1': '1 post published',
    'other': '# posts published'
  };
}
```

> **Key Takeaway:** Angular's locale-aware pipes automatically adapt to regional formatting conventions through `LOCALE_ID` and `registerLocaleData()`. Set the locale and currency defaults at the application level, use predefined format strings that respect locale rules, and leverage the native `Intl` API for advanced formatting needs like compact numbers and relative time.
