# 17.8 Standalone Pipes

> Standalone pipes are self-contained, directly importable transformation units that do not require NgModule declarations. In Angular 21, all pipes are standalone by default, enabling better tree-shaking, simpler imports, and cleaner architecture.

---

## What Are Standalone Pipes

A standalone pipe is a pipe that declares itself as independently importable via `standalone: true` in the `@Pipe` decorator. It does not need to be part of an NgModule's `declarations` array.

```typescript
import { Pipe, PipeTransform } from '@angular/core';

@Pipe({
  name: 'truncate',
  standalone: true  // Default in Angular 21, can be omitted
})
export class TruncatePipe implements PipeTransform {
  transform(value: string, limit: number = 50): string {
    if (!value) return '';
    return value.length > limit ? value.substring(0, limit) + '...' : value;
  }
}
```

Components import standalone pipes directly:

```typescript
@Component({
  standalone: true,
  imports: [TruncatePipe],
  template: `<p>{{ description | truncate:100 }}</p>`
})
export class ArticleComponent {
  description = '...';
}
```

No NgModule involved. The pipe is imported where it is used, creating an explicit dependency graph.

---

## Standalone Is the Default in Angular 21

Since Angular 19, `standalone: true` is the default for components, directives, and pipes. You no longer need to specify it.

```typescript
// Angular 21: standalone is implied
@Pipe({ name: 'truncate' })
export class TruncatePipe implements PipeTransform {
  transform(value: string, limit: number = 50): string {
    if (!value) return '';
    return value.length > limit ? value.substring(0, limit) + '...' : value;
  }
}
```

To create a non-standalone pipe (for legacy NgModule usage), you must explicitly opt out:

```typescript
// Explicitly non-standalone (legacy pattern)
@Pipe({
  name: 'truncate',
  standalone: false
})
export class TruncatePipe implements PipeTransform { ... }
```

The CLI schematic `ng generate pipe` produces standalone pipes by default:

```bash
ng generate pipe pipes/truncate
# Creates standalone pipe at src/app/pipes/truncate.pipe.ts
```

## Standalone Pipes vs NgModule Pipes

### NgModule Approach (Legacy)

In the NgModule world, pipes had to be declared in a module and that module had to be imported by other modules:

```typescript
// BAD: Legacy NgModule approach
@Pipe({ name: 'truncate', standalone: false })
export class TruncatePipe implements PipeTransform { ... }

@NgModule({
  declarations: [TruncatePipe],
  exports: [TruncatePipe]        // Must export for other modules to use
})
export class SharedPipesModule {}

// Consumer module
@NgModule({
  imports: [SharedPipesModule],   // Import the whole module
  declarations: [ArticleComponent]
})
export class ArticleModule {}
```

Problems with this approach:
- **Indirect dependency:** The component depends on the module, not the pipe directly. It is unclear which pipes from `SharedPipesModule` the component actually uses.
- **Bundling overhead:** Importing the module brings all its declared pipes into the bundle, even if the component uses only one.
- **Boilerplate:** Declaring, exporting, and importing through modules adds configuration noise.

### Standalone Approach (Angular 21)

```typescript
// GOOD: Standalone approach
@Pipe({ name: 'truncate' })
export class TruncatePipe implements PipeTransform { ... }

@Component({
  standalone: true,
  imports: [TruncatePipe],  // Direct import, no module needed
  template: `<p>{{ text | truncate:100 }}</p>`
})
export class ArticleComponent {}
```

Benefits:
- **Explicit dependencies:** The component's `imports` array shows exactly which pipes it uses.
- **Tree-shakable:** Only the pipes actually imported are included in the bundle.
- **No module boilerplate:** No `declarations`, `exports`, or intermediary modules.
- **Lazy loading friendly:** Pipes imported only in lazy-loaded components are bundled with the lazy chunk.

## Importing Built-in Pipes

All Angular built-in pipes are standalone. You can import them individually or use `CommonModule`.

### Individual Imports (Recommended)

```typescript
import { DatePipe, CurrencyPipe, UpperCasePipe } from '@angular/common';

@Component({
  standalone: true,
  imports: [DatePipe, CurrencyPipe, UpperCasePipe],
  template: `
    <p>{{ order.date | date:'mediumDate' }}</p>
    <p>{{ order.total | currency }}</p>
    <p>{{ order.status | uppercase }}</p>
  `
})
export class OrderComponent {}
```

### CommonModule Import (Not Recommended)

```typescript
import { CommonModule } from '@angular/common';

@Component({
  standalone: true,
  imports: [CommonModule],  // Imports ALL common pipes, directives, etc.
  template: `...`
})
export class OrderComponent {}
```

`CommonModule` includes all built-in pipes, all built-in directives (`NgIf`, `NgFor`, `NgClass`, etc.), and other utilities. Importing it when you only need one or two pipes brings unnecessary code into the component's dependency graph.

### Impact on Bundle Size

With individual imports, the build tooling can tree-shake unused pipes:

```typescript
// Only DatePipe and CurrencyPipe code is included in the bundle
imports: [DatePipe, CurrencyPipe]

// vs.

// All of CommonModule is referenced (though modern bundlers
// may still tree-shake unreferenced pipes in practice)
imports: [CommonModule]
```

In practice, Angular's Ivy compiler and esbuild/Vite bundler are aggressive about tree-shaking. The difference may be minimal for small applications, but individual imports make the dependency graph explicit and are the recommended practice.

## Tree-Shaking and Bundle Optimization

Standalone pipes enable precise tree-shaking because each pipe is a distinct importable unit.

### How Tree-Shaking Works with Pipes

```
Application
  ├── ComponentA imports [DatePipe, TruncatePipe]
  ├── ComponentB imports [CurrencyPipe]
  └── ComponentC imports [DatePipe, UpperCasePipe]

Bundle includes: DatePipe, TruncatePipe, CurrencyPipe, UpperCasePipe
Not included: PercentPipe, LowerCasePipe, SlicePipe, etc.
```

With NgModules:

```
Application
  ├── ModuleA imports [SharedModule]  // SharedModule declares 10 pipes
  ├── ModuleB imports [SharedModule]
  └── ModuleC imports [SharedModule]

Bundle includes: All 10 pipes from SharedModule
Actually used: 4 pipes
```

### Lazy Loading with Standalone Pipes

Standalone pipes imported only in lazy-loaded components are bundled in the lazy chunk:

```typescript
// Route configuration
export const routes: Routes = [
  {
    path: 'reports',
    loadComponent: () => import('./reports/report.component')
      .then(m => m.ReportComponent)
  }
];

// report.component.ts (lazy-loaded)
@Component({
  standalone: true,
  imports: [
    ChartPipe,        // Only bundled in the reports chunk
    StatsPipe,        // Only bundled in the reports chunk
    DatePipe           // May be shared if used elsewhere
  ],
  template: `...`
})
export class ReportComponent {}
```

`ChartPipe` and `StatsPipe` are only loaded when the user navigates to `/reports`.

## Creating and Organizing Standalone Pipes

### File Structure

```
src/app/
  pipes/
    truncate.pipe.ts
    truncate.pipe.spec.ts
    file-size.pipe.ts
    file-size.pipe.spec.ts
    time-ago.pipe.ts
    time-ago.pipe.spec.ts
    index.ts              // Barrel export
```

### Barrel Export

```typescript
// src/app/pipes/index.ts
export { TruncatePipe } from './truncate.pipe';
export { FileSizePipe } from './file-size.pipe';
export { TimeAgoPipe } from './time-ago.pipe';
```

```typescript
// In a component
import { TruncatePipe, FileSizePipe } from '../pipes';

@Component({
  standalone: true,
  imports: [TruncatePipe, FileSizePipe],
  template: `...`
})
export class FileListComponent {}
```

### Shared Pipe Library

For monorepos or multi-application projects, create a shared pipe library:

```
libs/shared/pipes/
  src/
    lib/
      truncate.pipe.ts
      file-size.pipe.ts
      time-ago.pipe.ts
      highlight.pipe.ts
      index.ts
    index.ts
    public-api.ts
```

```typescript
// libs/shared/pipes/src/public-api.ts
export { TruncatePipe } from './lib/truncate.pipe';
export { FileSizePipe } from './lib/file-size.pipe';
export { TimeAgoPipe } from './lib/time-ago.pipe';
export { HighlightPipe } from './lib/highlight.pipe';
```

```typescript
// Consumer application
import { TruncatePipe } from '@myorg/shared/pipes';

@Component({
  standalone: true,
  imports: [TruncatePipe],
  template: `{{ text | truncate:100 }}`
})
export class MyComponent {}
```

Each application only bundles the pipes it imports from the library.

## Migrating from NgModule Pipes to Standalone

### The Migration Schematic

Angular provides an automated migration schematic:

```bash
ng generate @angular/core:standalone
```

This schematic:
1. Adds `standalone: true` to components, directives, and pipes
2. Updates `imports` arrays to include direct dependencies
3. Removes unnecessary NgModule declarations
4. Preserves the application's behavior

There is also a dedicated schematic to replace `CommonModule` with individual imports:

```bash
ng generate @angular/core:common-module-to-standalone
```

This analyzes templates and replaces `CommonModule` imports with only the specific pipes and directives actually used (e.g., `DatePipe`, `AsyncPipe`, `NgClass`).

### Manual Migration Steps

**Step 1:** Add `standalone: true` to the pipe (or remove `standalone: false` in Angular 19+):

```typescript
// BEFORE
@Pipe({ name: 'truncate' })
export class TruncatePipe implements PipeTransform { ... }

// In Angular 21, this is already standalone by default
// No change needed unless it was explicitly standalone: false
```

**Step 2:** Remove the pipe from NgModule declarations:

```typescript
// BEFORE
@NgModule({
  declarations: [TruncatePipe],
  exports: [TruncatePipe]
})
export class SharedModule {}

// AFTER: Delete the module if it only contained pipes
// Or remove the pipe from declarations/exports if other things remain
```

**Step 3:** Import the pipe directly in consuming components:

```typescript
// BEFORE (component in an NgModule that imports SharedModule)
@Component({
  template: `{{ text | truncate }}`
})
export class MyComponent {}

// AFTER
@Component({
  standalone: true,
  imports: [TruncatePipe],
  template: `{{ text | truncate }}`
})
export class MyComponent {}
```

### Incremental Migration

You do not have to migrate everything at once. Standalone and NgModule-based code can coexist:

```typescript
// A standalone component can import an NgModule
@Component({
  standalone: true,
  imports: [SharedPipesModule],  // NgModule import in standalone component
  template: `{{ text | truncate }}`
})
export class HybridComponent {}

// An NgModule can import a standalone pipe
@NgModule({
  imports: [TruncatePipe],       // Standalone pipe in NgModule imports
  declarations: [LegacyComponent]
})
export class LegacyModule {}
```

## Standalone Pipes with Dependency Injection

Standalone pipes support dependency injection through `inject()`, just like any other Angular injectable:

```typescript
@Pipe({ name: 'translate' })
export class TranslatePipe implements PipeTransform {
  private translationService = inject(TranslationService);

  transform(key: string): string {
    return this.translationService.translate(key);
  }
}
```

The pipe resolves its dependencies from the component's injector hierarchy. This means:

```typescript
@Component({
  standalone: true,
  imports: [TranslatePipe],
  providers: [
    // This TranslationService instance is available to TranslatePipe
    { provide: TranslationService, useClass: FrenchTranslationService }
  ],
  template: `<p>{{ 'hello' | translate }}</p>`
})
export class FrenchComponent {}
```

### Providing a Pipe in the Injector

To use a pipe programmatically (outside the template), you must provide it:

```typescript
@Component({
  standalone: true,
  imports: [DatePipe],
  providers: [DatePipe],  // Required for inject() usage
  template: `<p>{{ formattedDate }}</p>`
})
export class ReportComponent {
  private datePipe = inject(DatePipe);
  formattedDate = this.datePipe.transform(new Date(), 'fullDate');
}
```

Importing a pipe in `imports` makes it available in templates. Adding it to `providers` makes it available for injection. These are separate concerns.

## Testing Standalone Pipes

Standalone pipes are simpler to test because they have no module dependencies.

### Without Dependencies

```typescript
import { TruncatePipe } from './truncate.pipe';

describe('TruncatePipe', () => {
  const pipe = new TruncatePipe();

  it('should truncate long strings', () => {
    expect(pipe.transform('Hello World', 5)).toBe('Hello...');
  });

  it('should not truncate short strings', () => {
    expect(pipe.transform('Hi', 50)).toBe('Hi');
  });

  it('should handle null', () => {
    expect(pipe.transform(null as any)).toBe('');
  });
});
```

No `TestBed`, no module configuration. Just instantiate and test.

### With Dependencies

```typescript
import { TestBed } from '@angular/core/testing';
import { TranslatePipe } from './translate.pipe';
import { TranslationService } from '../services/translation.service';

describe('TranslatePipe', () => {
  let pipe: TranslatePipe;

  beforeEach(() => {
    TestBed.configureTestingModule({
      providers: [
        TranslatePipe,
        {
          provide: TranslationService,
          useValue: { translate: (key: string) => `translated:${key}` }
        }
      ]
    });
    pipe = TestBed.inject(TranslatePipe);
  });

  it('should translate keys', () => {
    expect(pipe.transform('hello')).toBe('translated:hello');
  });
});
```

### Testing a Component That Uses a Standalone Pipe

```typescript
import { ComponentFixture, TestBed } from '@angular/core/testing';
import { ArticleComponent } from './article.component';

describe('ArticleComponent', () => {
  let fixture: ComponentFixture<ArticleComponent>;

  beforeEach(async () => {
    await TestBed.configureTestingModule({
      imports: [ArticleComponent]  // Standalone component brings its own pipe imports
    }).compileComponents();

    fixture = TestBed.createComponent(ArticleComponent);
  });

  it('should render truncated text', () => {
    fixture.componentRef.setInput('description', 'A'.repeat(200));
    fixture.detectChanges();
    const text = fixture.nativeElement.querySelector('p').textContent;
    expect(text.length).toBeLessThan(200);
  });
});
```

When testing a standalone component, its `imports` (including pipes) are automatically available. No additional configuration needed.

### Mocking a Pipe in Tests

To replace a pipe with a mock in tests:

```typescript
@Pipe({ name: 'translate' })
class MockTranslatePipe implements PipeTransform {
  transform(key: string): string {
    return key; // Return the key as-is
  }
}

beforeEach(async () => {
  await TestBed.configureTestingModule({
    imports: [ArticleComponent]
  })
  .overrideComponent(ArticleComponent, {
    remove: { imports: [TranslatePipe] },
    add: { imports: [MockTranslatePipe] }
  })
  .compileComponents();
});
```

## CommonModule vs Individual Imports

A side-by-side of both approaches:

```typescript
// Approach 1: CommonModule (quick but imprecise)
import { CommonModule } from '@angular/common';

@Component({
  standalone: true,
  imports: [CommonModule],
  template: `{{ name | uppercase }}`
})
export class QuickComponent {}

// Approach 2: Individual imports (precise and explicit)
import { UpperCasePipe } from '@angular/common';

@Component({
  standalone: true,
  imports: [UpperCasePipe],
  template: `{{ name | uppercase }}`
})
export class PreciseComponent {}
```

| Aspect | CommonModule | Individual Imports |
|--------|-------------|-------------------|
| Simplicity | One import covers everything | Must know exact pipe names |
| Bundle size | Slightly larger (references all) | Minimal (only what's used) |
| Readability | Unclear which pipes are used | Explicit dependency list |
| Refactoring | Harder to track pipe usage | Easy to find all usages |
| IDE support | Generic autocomplete | Precise import suggestions |

**Recommendation:** Use individual imports. The explicit dependency list serves as documentation and enables better tooling support.

## Common Mistakes

- **Importing `CommonModule` instead of specific pipes.** This brings all common directives and pipes into scope, making dependencies opaque. Import only what you need.

- **Forgetting to import a standalone pipe in the component.** Unlike NgModule pipes (which are available to all components in the module), standalone pipes must be imported in each component that uses them. Missing imports cause template compilation errors.

- **Adding a standalone pipe to `declarations` instead of `imports` in an NgModule.** Standalone pipes go in the `imports` array of an NgModule, not `declarations`. `declarations` is for non-standalone items only.

- **Not removing pipes from NgModule declarations after migration.** When migrating to standalone, remove the pipe from the module's `declarations` and `exports`. Leaving it in both places causes duplicate declaration errors.

- **Providing a pipe for injection but forgetting to add it to `imports`.** Adding a pipe to `providers` makes it available for `inject()`, but it is not available in the template unless it is also in `imports`. These are separate mechanisms.

- **Assuming `inject()` and constructor injection behave identically in pipes.** There is a known inconsistency (GitHub #50952): constructor injection in pipes uses the directive injector (can access component-level providers), but `inject()` uses the environment injector and may not find services provided at the component level. Test carefully when migrating pipes from constructor to `inject()`.

---

## Best Practices

- **Let standalone be the default.** In Angular 21, do not add `standalone: true` explicitly. It is the default. Only specify `standalone: false` if you are maintaining legacy NgModule code.

- **Import pipes individually, not via CommonModule.** This makes dependencies explicit, improves tree-shaking, and makes it easy to find which components use which pipes.

- **Use barrel exports for shared pipe collections.** Create an `index.ts` that re-exports all pipes from a directory or library. Consumers import specific pipes from the barrel.

- **Keep pipe files focused.** One pipe per file, with its test alongside: `truncate.pipe.ts` and `truncate.pipe.spec.ts`.

- **Use the migration schematic for large codebases.** `ng generate @angular/core:standalone` automates the tedious parts of converting NgModule pipes to standalone.

- **Test standalone pipes as plain classes when possible.** No TestBed overhead for pipes without dependencies.

## Real-World Use Cases

### Feature Module with Scoped Pipes

```typescript
// A lazy-loaded feature that uses its own pipes
// These pipes are only bundled when this route is loaded

@Pipe({ name: 'orderStatus' })
export class OrderStatusPipe implements PipeTransform {
  transform(status: string): string {
    const labels: Record<string, string> = {
      pending: 'Pending Review',
      processing: 'In Progress',
      shipped: 'Shipped',
      delivered: 'Delivered',
      cancelled: 'Cancelled'
    };
    return labels[status] ?? status;
  }
}

@Pipe({ name: 'trackingUrl' })
export class TrackingUrlPipe implements PipeTransform {
  transform(trackingNumber: string, carrier: string): string {
    const urls: Record<string, string> = {
      ups: `https://www.ups.com/track?tracknum=${trackingNumber}`,
      fedex: `https://www.fedex.com/fedextrack/?trknbr=${trackingNumber}`,
      usps: `https://tools.usps.com/go/TrackConfirmAction?tLabels=${trackingNumber}`
    };
    return urls[carrier] ?? '#';
  }
}

@Component({
  standalone: true,
  imports: [
    DatePipe,
    CurrencyPipe,
    OrderStatusPipe,
    TrackingUrlPipe
  ],
  template: `
    @for (order of orders(); track order.id) {
      <div class="order-card">
        <span class="status">{{ order.status | orderStatus }}</span>
        <span class="date">{{ order.createdAt | date:'mediumDate' }}</span>
        <span class="total">{{ order.total | currency }}</span>
        @if (order.trackingNumber) {
          <a [href]="order.trackingNumber | trackingUrl:order.carrier"
             target="_blank">
            Track Package
          </a>
        }
      </div>
    }
  `
})
export class OrderListComponent {
  orders = input.required<Order[]>();
}
```

### Shared Library Used Across Applications

```typescript
// @myorg/shared-pipes library

// truncate.pipe.ts
@Pipe({ name: 'truncate' })
export class TruncatePipe implements PipeTransform {
  transform(value: string | null, limit: number = 50, trail: string = '...'): string {
    if (!value) return '';
    return value.length > limit ? value.substring(0, limit) + trail : value;
  }
}

// file-size.pipe.ts
@Pipe({ name: 'fileSize' })
export class FileSizePipe implements PipeTransform {
  transform(bytes: number | null, decimals: number = 1): string {
    if (bytes == null || bytes === 0) return '0 B';
    const k = 1024;
    const units = ['B', 'KB', 'MB', 'GB', 'TB'];
    const i = Math.floor(Math.log(Math.abs(bytes)) / Math.log(k));
    return `${(bytes / Math.pow(k, i)).toFixed(decimals)} ${units[i]}`;
  }
}

// public-api.ts
export { TruncatePipe } from './lib/truncate.pipe';
export { FileSizePipe } from './lib/file-size.pipe';
```

```typescript
// App 1: Blog application
import { TruncatePipe } from '@myorg/shared-pipes';

@Component({
  standalone: true,
  imports: [TruncatePipe],  // Only TruncatePipe is bundled
  template: `<p>{{ article.excerpt | truncate:200 }}</p>`
})
export class ArticlePreviewComponent {}

// App 2: File manager application
import { FileSizePipe } from '@myorg/shared-pipes';

@Component({
  standalone: true,
  imports: [FileSizePipe],  // Only FileSizePipe is bundled
  template: `<span>{{ file.size | fileSize }}</span>`
})
export class FileRowComponent {}
```

Each application bundles only the pipes it actually uses from the shared library.

### Gradual Migration from NgModule to Standalone

```typescript
// Phase 1: Make the pipe standalone (backward compatible)
@Pipe({ name: 'highlight' })  // standalone by default in Angular 21
export class HighlightPipe implements PipeTransform {
  transform(text: string, term: string): string { ... }
}

// Phase 2: Legacy module can still use it via imports
@NgModule({
  imports: [HighlightPipe],  // Standalone pipe in NgModule imports
  declarations: [LegacySearchComponent],
  exports: [LegacySearchComponent]
})
export class SearchModule {}

// Phase 3: New standalone components import directly
@Component({
  standalone: true,
  imports: [HighlightPipe],
  template: `<span [innerHTML]="text | highlight:query"></span>`
})
export class NewSearchComponent {}

// Both approaches work simultaneously during migration
```

> **Key Takeaway:** Standalone pipes are the default in Angular 21, eliminating NgModule boilerplate and enabling explicit, tree-shakable dependencies. Import pipes individually in each component that uses them, organize shared pipes in barrel-exported libraries, and use the migration schematic to convert legacy NgModule pipes.
