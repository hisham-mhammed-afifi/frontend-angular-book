# 18.12 RxJS Interop with Signals (`toSignal()`, `toObservable()`)

> Angular's rxjs-interop package bridges the reactive world of Observables with the synchronous world of Signals, letting you use each where it works best.

---

## Why Interop Exists

Signals and Observables solve different problems:

| Aspect | Signals | Observables |
|--------|---------|-------------|
| Value access | Synchronous (`signal()`) | Asynchronous (`.subscribe()`) |
| Push/Pull | Pull (read when needed) | Push (values arrive when ready) |
| Change detection | Granular, automatic | Requires `async` pipe or manual markForCheck |
| Async operations | Not designed for them | Built for them |
| Operators | `computed()`, `effect()` | 100+ RxJS operators |
| Lifecycle | Garbage collected | Must unsubscribe |

Neither replaces the other. HTTP calls, WebSocket streams, and complex async coordination remain RxJS territory. Template state, derived values, and component inputs are Signal territory. The interop layer connects them.

```typescript
import { toSignal, toObservable } from '@angular/core/rxjs-interop';
```

---

## `toSignal()` -- Observable to Signal

`toSignal()` subscribes to an Observable and returns a Signal that reflects the latest emitted value.

### Basic Usage

```typescript
@Component({
  selector: 'app-clock',
  template: `<p>Seconds: {{ seconds() }}</p>`
})
export class ClockComponent {
  // Converts an Observable into a Signal
  // The template reads seconds() like any other signal
  readonly seconds = toSignal(interval(1000), { initialValue: 0 });
}
```

The subscription is created immediately when `toSignal()` is called. It is automatically cleaned up when the component's `DestroyRef` fires.

### The Full Options Object

```typescript
toSignal<T>(source: Observable<T>, options?: ToSignalOptions<T>): Signal<T>

interface ToSignalOptions<T> {
  initialValue?: T;           // Value before first emission
  requireSync?: boolean;      // Enforce synchronous first emission
  manualCleanup?: boolean;    // Disable automatic DestroyRef teardown
  injector?: Injector;        // Custom injector for DestroyRef lookup
  equal?: (a: T, b: T) => boolean;  // Custom equality function
  debugName?: string;          // Label for Angular DevTools
}
```

### `initialValue` -- Handling the "Before First Emission" Gap

Observables are asynchronous. Between the moment `toSignal()` is called and the first emission, the Signal needs a value. Without `initialValue`, the Signal's type includes `undefined`.

```typescript
// Without initialValue: Signal<User | undefined>
readonly user = toSignal(this.userService.user$);
// user() might be undefined until the HTTP call completes

// With initialValue: Signal<User>
readonly user = toSignal(this.userService.user$, {
  initialValue: { name: 'Guest', role: 'anonymous' } as User
});
// user() is always a User, even before the Observable emits
```

Type implications:

```typescript
const a = toSignal(source$);
// Type: Signal<T | undefined>
// a() could be undefined

const b = toSignal(source$, { initialValue: defaultVal });
// Type: Signal<T>
// b() is always T

const c = toSignal(source$, { initialValue: undefined });
// Type: Signal<T | undefined>
// Explicitly typed as possibly undefined

const d = toSignal(source$, { requireSync: true });
// Type: Signal<T>
// d() is always T (Observable MUST emit synchronously)
```

### `requireSync` -- For Synchronous Sources

When the Observable is guaranteed to emit synchronously on subscription (like `BehaviorSubject`, `of()`, or `startWith()`), use `requireSync: true` to get a cleaner type without `undefined`.

```typescript
// BehaviorSubject always emits its current value synchronously
private readonly count$ = new BehaviorSubject(0);

// Type: Signal<number> (no undefined!)
readonly count = toSignal(this.count$, { requireSync: true });
```

If the Observable does NOT emit synchronously, `requireSync: true` throws a runtime error:

```typescript
// RUNTIME ERROR: Observable did not emit synchronously
const broken = toSignal(this.http.get('/api/data'), { requireSync: true });
```

Use `requireSync` with:
- `BehaviorSubject`
- Observables starting with `startWith()`
- `of()`, `from([...])`
- Any Observable that emits immediately on subscribe

### `manualCleanup` -- Opting Out of Auto-Teardown

By default, `toSignal()` requires an injection context to obtain a `DestroyRef` for automatic cleanup. Setting `manualCleanup: true` disables this.

```typescript
// No injection context needed, but YOU must ensure cleanup
const data = toSignal(source$, { manualCleanup: true });
```

Use cases for `manualCleanup`:
- The Observable completes naturally (e.g., `take(5)`, HTTP calls)
- You are creating the Signal outside an injection context
- The Signal should outlive the component (rare)

**Warning:** If the Observable never completes and `manualCleanup: true` is set, the subscription leaks.

### `injector` -- Custom Injection Context

When calling `toSignal()` outside a constructor or field initializer, you can pass an explicit injector:

```typescript
@Component({ selector: 'app-demo', template: `...` })
export class DemoComponent {
  private readonly injector = inject(Injector);

  ngOnInit() {
    // Would throw without injector option (no injection context)
    const data = toSignal(this.dataService.stream$, {
      injector: this.injector,
      initialValue: []
    });
  }
}
```

### `equal` -- Custom Equality

By default, Signals use `Object.is` to determine if a value has changed. You can override this:

```typescript
readonly position = toSignal(this.gps.position$, {
  initialValue: { lat: 0, lng: 0 },
  equal: (a, b) => a.lat === b.lat && a.lng === b.lng
});
// Signal only updates (and triggers change detection) when lat or lng actually change
```

### Error Handling in `toSignal()`

When the source Observable errors, `toSignal()` throws the error when the Signal is read:

```typescript
const data = toSignal(this.http.get('/api/data'));

// If the HTTP request fails:
// data() throws the error (not returns undefined)
```

This means errors propagate through the template and trigger Angular's error handler. To handle errors gracefully, catch them in the Observable:

```typescript
// Handle errors in the Observable, not the Signal
readonly data = toSignal(
  this.http.get<Data>('/api/data').pipe(
    catchError(() => of(null)) // Signal gets null instead of throwing
  ),
  { initialValue: null }
);

// Or use a result wrapper
readonly result = toSignal(
  this.http.get<Data>('/api/data').pipe(
    map(data => ({ status: 'success' as const, data })),
    catchError(err => of({ status: 'error' as const, error: err }))
  ),
  { initialValue: { status: 'loading' as const } }
);
```

**Note:** Angular 19 introduced a `rejectErrors` option that silently froze the Signal at the last good value on error. This was **removed in Angular 20** because it was considered a poor practice (it hid errors). The recommended approach is always explicit `catchError()` in the pipe.

### Completion Behavior

When the source Observable completes, the Signal retains its last value. It does not reset to `undefined` or throw.

```typescript
const value = toSignal(of(42));
// After of(42) completes, value() returns 42 forever
```

## `toObservable()` -- Signal to Observable

`toObservable()` converts a Signal into an Observable that emits whenever the Signal's value changes.

### Basic Usage

```typescript
import { toObservable } from '@angular/core/rxjs-interop';

@Component({
  selector: 'app-search',
  template: `<input (input)="query.set($event.target.value)" />`
})
export class SearchComponent {
  readonly query = signal('');

  // Observable that emits whenever the query signal changes
  private readonly query$ = toObservable(this.query);

  readonly results = toSignal(
    this.query$.pipe(
      debounceTime(300),
      distinctUntilChanged(),
      switchMap(q => q.length < 2 ? of([]) : this.searchService.search(q))
    ),
    { initialValue: [] }
  );
}
```

### How `toObservable()` Works Internally

`toObservable()` uses an `effect()` under the hood that watches the Signal. When the Signal value changes, the effect runs and pushes the new value to subscribers via a `Subject`.

Key behaviors:
1. Emits the current value of the Signal on subscription (like a `BehaviorSubject`)
2. Subsequent emissions happen asynchronously (scheduled via microtask)
3. Intermediate values are coalesced (if the Signal changes multiple times before the microtask runs, only the latest value is emitted)

```typescript
const count = signal(0);
const count$ = toObservable(count);

count$.subscribe(v => console.log('Received:', v));
// Received: 0 (initial value, emitted asynchronously via microtask)

count.set(1);
count.set(2);
count.set(3);
// Received: 3 (coalesced: only the latest value is emitted)
```

### Injection Context Requirement

`toObservable()` requires an injection context because it creates an internal `effect()`:

```typescript
// WORKS: Field initializer (injection context)
readonly query$ = toObservable(this.querySignal);

// WORKS: Constructor (injection context)
constructor() {
  const query$ = toObservable(this.querySignal);
}

// DOES NOT WORK: ngOnInit (no injection context)
ngOnInit() {
  const query$ = toObservable(this.querySignal); // THROWS NG0203
}

// FIX: Pass injector explicitly
private readonly injector = inject(Injector);

ngOnInit() {
  const query$ = toObservable(this.querySignal, { injector: this.injector });
}
```

### `toObservable()` Does NOT Emit Synchronously

Unlike `BehaviorSubject`, `toObservable()` emits asynchronously. This matters for `requireSync`:

```typescript
const sig = signal(42);
const obs$ = toObservable(sig);

// This will NOT work because toObservable emits asynchronously
const backToSignal = toSignal(obs$, { requireSync: true }); // THROWS!

// FIX: Use initialValue instead
const backToSignal = toSignal(obs$, { initialValue: 42 });
```

**Testing gotcha:** Because `toObservable()` emits via microtask, synchronous unit tests will not see emissions immediately. Use `fakeAsync` + `tick()` or `await fixture.whenStable()` to flush microtasks:

```typescript
it('should emit signal changes', fakeAsync(() => {
  const sig = signal(0);
  const values: number[] = [];
  TestBed.runInInjectionContext(() => {
    toObservable(sig).subscribe(v => values.push(v));
  });

  tick(); // Flush microtask for initial value
  expect(values).toEqual([0]);

  sig.set(42);
  tick(); // Flush microtask for new value
  expect(values).toEqual([0, 42]);
}));
```

## The Signal-RxJS Bridge Pattern

The most common pattern in Angular 21 apps is the "Signal in, RxJS process, Signal out" pattern:

```
Signal (input) -> toObservable() -> RxJS pipeline -> toSignal() -> Signal (output)
```

### Pattern: Debounced Search

```typescript
@Component({
  selector: 'app-search',
  template: `
    <input [ngModel]="query()" (ngModelChange)="query.set($event)" />
    @for (result of results(); track result.id) {
      <div>{{ result.name }}</div>
    }
  `
})
export class SearchComponent {
  private readonly searchService = inject(SearchService);

  // Input signal
  readonly query = signal('');

  // RxJS processing pipeline
  readonly results = toSignal(
    toObservable(this.query).pipe(
      debounceTime(300),
      distinctUntilChanged(),
      filter(q => q.length >= 2),
      switchMap(q => this.searchService.search(q))
    ),
    { initialValue: [] }
  );
}
```

### Pattern: Route Parameter Processing

```typescript
@Component({
  selector: 'app-product',
  template: `
    @if (product(); as product) {
      <h1>{{ product.name }}</h1>
      <p>{{ product.description }}</p>
    } @else {
      <p>Loading...</p>
    }
  `
})
export class ProductComponent {
  private readonly route = inject(ActivatedRoute);
  private readonly productService = inject(ProductService);

  readonly product = toSignal(
    this.route.paramMap.pipe(
      map(params => params.get('id')!),
      distinctUntilChanged(),
      switchMap(id => this.productService.getById(id))
    )
  );
}
```

### Pattern: Combining Signals with Observables

```typescript
@Component({
  selector: 'app-dashboard',
  template: `
    <select [ngModel]="selectedRegion()" (ngModelChange)="selectedRegion.set($event)">
      @for (r of regions; track r) { <option [value]="r">{{ r }}</option> }
    </select>
    <app-chart [data]="chartData()" />
  `
})
export class DashboardComponent {
  private readonly http = inject(HttpClient);

  readonly selectedRegion = signal('us-east');
  readonly regions = ['us-east', 'us-west', 'eu-west', 'ap-south'];

  readonly chartData = toSignal(
    toObservable(this.selectedRegion).pipe(
      switchMap(region => this.http.get<ChartData>(`/api/metrics/${region}`))
    ),
    { initialValue: { labels: [], values: [] } }
  );
}
```

## `toSignal()` vs `async` Pipe

Both convert Observables for template consumption. Here is when to use each:

| Aspect | `toSignal()` | `async` pipe |
|--------|-------------|--------------|
| Location | TypeScript class | HTML template |
| Type safety | Full (Signal type) | Requires null checks |
| Multiple template uses | One subscription, read signal() multiple times | One subscription per `\| async` usage (use `@if as` to avoid) |
| Composability | Can use in `computed()`, `effect()` | Template-only |
| Change detection | Signal-based (granular) | Zone-based (less granular) |
| Error handling | Throws on read | Swallows errors silently |
| Initial value | Configurable | `null` until first emission |
| Cleanup | Automatic via DestroyRef | Automatic via pipe lifecycle |

**General guidance:** Prefer `toSignal()` in Angular 21 for consistency with the signal-based architecture. Use `async` pipe when migrating incrementally or when the Observable is only used once in the template.

```typescript
// BEFORE: async pipe (still works but less composable)
@Component({
  template: `
    @if (user$ | async; as user) {
      <h1>{{ user.name }}</h1>
      <p>{{ user.email }}</p>
    }
  `
})
export class OldComponent {
  readonly user$ = this.userService.currentUser$;
}

// AFTER: toSignal (composable, type-safe)
@Component({
  template: `
    @if (user(); as user) {
      <h1>{{ user.name }}</h1>
      <p>{{ user.email }}</p>
    }
  `
})
export class NewComponent {
  readonly user = toSignal(inject(UserService).currentUser$);

  // Can now use user() in computed signals
  readonly greeting = computed(() => {
    const u = this.user();
    return u ? `Hello, ${u.name}` : 'Welcome, guest';
  });
}
```

## `toObservable()` vs `effect()` vs `computed()`

When a Signal changes and you need to react, you have three options:

| Tool | Use When |
|------|----------|
| `computed()` | Deriving a synchronous value from signals |
| `effect()` | Performing side effects (logging, DOM manipulation, imperative API calls) |
| `toObservable()` | You need RxJS operators (debounce, switchMap, retry, etc.) |

```typescript
// computed: synchronous derivation
readonly fullName = computed(() => `${this.firstName()} ${this.lastName()}`);

// effect: side effect without RxJS
effect(() => {
  console.log('User changed:', this.user());
  localStorage.setItem('lastUser', JSON.stringify(this.user()));
});

// toObservable: when you need RxJS operators
readonly searchResults = toSignal(
  toObservable(this.searchQuery).pipe(
    debounceTime(300),           // Cannot do this with computed or effect
    switchMap(q => this.search(q)) // Cannot do this with computed or effect
  ),
  { initialValue: [] }
);
```

**Do NOT use `toObservable()` just to watch a signal change.** If you do not need RxJS operators, `effect()` or `computed()` is simpler.

```typescript
// BAD: Using toObservable when effect would suffice
toObservable(this.theme).subscribe(theme => {
  document.body.className = theme;
});

// GOOD: Use effect for simple side effects
effect(() => {
  document.body.className = this.theme();
});
```

## Output Interop Utilities

Angular provides utilities to bridge component outputs with Observables:

### `outputToObservable()`

Converts an Angular `output()` to an Observable for parent components that need RxJS processing:

```typescript
// Child component with output
@Component({
  selector: 'app-color-picker',
  template: `...`
})
export class ColorPickerComponent {
  readonly colorSelected = output<string>();
}

// Parent component consuming the output as an Observable
@Component({
  selector: 'app-parent',
  template: `<app-color-picker #picker />`
})
export class ParentComponent {
  @ViewChild('picker') picker!: ColorPickerComponent;

  ngAfterViewInit() {
    // Convert the output to an Observable for RxJS processing
    outputToObservable(this.picker.colorSelected).pipe(
      debounceTime(500),
      distinctUntilChanged(),
      takeUntilDestroyed(this.destroyRef)
    ).subscribe(color => {
      this.applyThemeColor(color);
    });
  }
}
```

### `outputFromObservable()`

Creates an Angular `output()` from an Observable. Useful in directive-like patterns:

```typescript
@Directive({
  selector: '[appIntersection]'
})
export class IntersectionDirective {
  private readonly elementRef = inject(ElementRef);

  // Create an output from an Observable
  readonly visible = outputFromObservable(
    new Observable<boolean>(subscriber => {
      const observer = new IntersectionObserver(entries => {
        subscriber.next(entries[0].isIntersecting);
      });
      observer.observe(this.elementRef.nativeElement);
      return () => observer.disconnect();
    })
  );
}

// Usage in template:
// <div appIntersection (visible)="onVisible($event)">...</div>
```

## `resource()` and `rxResource()` -- Signal-Based Async (Experimental)

Angular 19+ introduced `resource()` and `rxResource()` as experimental APIs for managing async data loading with signals. While not part of the stable rxjs-interop package, they represent the direction Angular is heading.

### `rxResource()` -- RxJS-Based

```typescript
@Component({
  selector: 'app-user-profile',
  template: `
    @if (userResource.isLoading()) {
      <spinner />
    }
    @if (userResource.value(); as user) {
      <h1>{{ user.name }}</h1>
    }
    @if (userResource.error(); as err) {
      <p>Error: {{ err.message }}</p>
    }
  `
})
export class UserProfileComponent {
  private readonly http = inject(HttpClient);
  readonly userId = input.required<string>();

  // Angular 20+ renamed: request -> params, loader -> stream
  readonly userResource = rxResource({
    params: () => ({ id: this.userId() }),
    stream: ({ params }) => this.http.get<User>(`/api/users/${params.id}`)
  });
}
```

### `resource()` -- Promise-Based

```typescript
// Angular 20+ renamed: request -> params, loader uses params
readonly userResource = resource({
  params: () => ({ id: this.userId() }),
  loader: async ({ params, abortSignal }) => {
    const response = await fetch(`/api/users/${params.id}`, { signal: abortSignal });
    return response.json();
  }
});
```

### Resource API Surface

```typescript
interface ResourceRef<T> {
  value: Signal<T | undefined>;       // The loaded data
  isLoading: Signal<boolean>;         // Whether a request is in flight
  error: Signal<unknown>;             // Error from the last request
  status: Signal<ResourceStatus>;     // 'idle' | 'loading' | 'resolved' | 'error' | 'reloading'
  reload: () => void;                 // Manually trigger a re-fetch
  hasValue: Signal<boolean>;          // Whether value is available
}
```

**Status as of Angular 21:** `resource()` and `rxResource()` are still experimental. The API may change. Use `toSignal()` for stable production code.

## Coalescing and Timing

Understanding the timing of `toObservable()` emissions is critical for avoiding subtle bugs.

### Coalescing Behavior

`toObservable()` uses Angular's microtask scheduling. Multiple synchronous Signal changes are coalesced into a single emission.

```typescript
const count = signal(0);
const count$ = toObservable(count);

count$.subscribe(v => console.log(v));

// All three changes happen synchronously
count.set(1);
count.set(2);
count.set(3);

// Output:
// 0 (initial value, via microtask)
// 3 (coalesced: only the final value)
// NOT: 1, 2, 3
```

This is intentional. It mirrors Angular's change detection batching. If you need every intermediate value, RxJS Observables (not Signals) are the right primitive.

### `toSignal()` Timing

`toSignal()` updates the Signal synchronously when the Observable emits. There is no coalescing on the `toSignal()` side.

```typescript
const subject = new Subject<number>();
const sig = toSignal(subject, { initialValue: 0 });

console.log(sig()); // 0 (initial value)

subject.next(1);
console.log(sig()); // 1 (updated synchronously)

subject.next(2);
console.log(sig()); // 2 (updated synchronously)
```

### The Round-Trip Problem

Converting Signal to Observable and back introduces a microtask delay:

```typescript
const original = signal(0);
const roundTrip = toSignal(toObservable(original), { initialValue: 0 });

original.set(42);
console.log(roundTrip()); // Still 0! The microtask hasn't fired yet.

// After microtask:
// roundTrip() === 42
```

Avoid unnecessary round-trips. If you already have a Signal and just need a derived value, use `computed()` instead of `toObservable().pipe(...).pipe(toSignal())` for synchronous transformations.

## Common Mistakes

1. **Using `toSignal()` without `initialValue` and not handling `undefined`.** The Signal's type is `T | undefined`. Accessing properties without null checking causes template errors.

    ```typescript
    // BAD
    readonly user = toSignal(this.userService.user$);
    // user().name -- TypeError if user() is undefined

    // GOOD
    readonly user = toSignal(this.userService.user$, {
      initialValue: { name: '', email: '' }
    });
    // Or: @if (user(); as u) { ... } in template
    ```

2. **Using `requireSync: true` with asynchronous Observables.** This throws at runtime. Only use with `BehaviorSubject`, `of()`, or Observables starting with `startWith()`.

3. **Calling `toSignal()` or `toObservable()` inside `ngOnInit()` without passing an injector.** Both require an injection context. Pass `{ injector: this.injector }` when outside constructor/field initializer scope.

4. **Creating `toSignal()` in a loop or conditional.** Unlike React hooks, `toSignal()` creates a subscription. Calling it in a loop creates multiple subscriptions. Create the Signal once as a class field.

    ```typescript
    // BAD: Creates new subscription on every call
    getUser(): Signal<User | undefined> {
      return toSignal(this.http.get<User>('/api/user'));
    }

    // GOOD: Single subscription, created once
    readonly user = toSignal(this.http.get<User>('/api/user'));
    ```

5. **Expecting `toObservable()` to emit synchronously.** It emits via microtask. Code that reads the Observable value synchronously after a Signal change will get the old value.

6. **Using `toObservable()` when `computed()` or `effect()` would suffice.** If you do not need RxJS operators, avoid the Observable overhead.

7. **Not catching errors before `toSignal()`.** If the Observable errors, reading the Signal throws. Always use `catchError()` in the pipe before `toSignal()` if errors are expected.

---

## Best Practices

1. **Prefer `toSignal()` over `async` pipe for new Angular 21 code.** Signals integrate better with the framework's change detection and are composable with `computed()`.

2. **Always provide an `initialValue` or use `requireSync`.** Avoid `Signal<T | undefined>` when you can provide a meaningful default.

3. **Handle errors in the Observable pipe, not in the Signal consumer.** Use `catchError()` before `toSignal()` to transform errors into fallback values.

4. **Use `shareReplay(1)` on Observables shared by multiple `toSignal()` calls.** Each `toSignal()` creates its own subscription. Without sharing, each triggers a separate execution.

5. **Reserve `toObservable()` for when you genuinely need RxJS operators.** For synchronous derivations, use `computed()`. For side effects, use `effect()`.

6. **Use `rxResource()` for data-fetching patterns that depend on signal inputs.** It handles loading states, errors, and refetching more elegantly than manual `toObservable()` + `switchMap` + `toSignal()` chains.

## Real-World Use Cases

### 1. Paginated Data Table

```typescript
@Component({
  selector: 'app-data-table',
  template: `
    <table>
      @for (row of rows(); track row.id) {
        <tr><td>{{ row.name }}</td><td>{{ row.status }}</td></tr>
      }
    </table>
    <app-paginator
      [page]="page()"
      [pageSize]="pageSize()"
      [total]="total()"
      (pageChange)="page.set($event)"
      (pageSizeChange)="pageSize.set($event)"
    />
    @if (loading()) { <spinner /> }
  `
})
export class DataTableComponent {
  private readonly http = inject(HttpClient);

  readonly page = signal(1);
  readonly pageSize = signal(25);
  readonly sortField = signal('name');
  readonly sortDir = signal<'asc' | 'desc'>('asc');

  // Combine all query parameters into a single Observable
  private readonly params$ = toObservable(
    computed(() => ({
      page: this.page(),
      pageSize: this.pageSize(),
      sort: this.sortField(),
      dir: this.sortDir()
    }))
  );

  readonly loading = signal(false);

  private readonly response = toSignal(
    this.params$.pipe(
      tap(() => this.loading.set(true)),
      debounceTime(100), // Coalesce rapid parameter changes
      switchMap(params =>
        this.http.get<PagedResponse>('/api/data', { params: params as any })
      ),
      tap(() => this.loading.set(false)),
      catchError(() => {
        this.loading.set(false);
        return of({ rows: [], total: 0 });
      })
    ),
    { initialValue: { rows: [], total: 0 } }
  );

  readonly rows = computed(() => this.response().rows);
  readonly total = computed(() => this.response().total);
}
```

### 2. Real-Time Collaborative Editor Status

```typescript
@Component({
  selector: 'app-editor-status',
  template: `
    <div class="status-bar">
      @for (user of activeUsers(); track user.id) {
        <app-avatar [user]="user" [style.border-color]="user.color" />
      }
      <span>{{ connectionStatus() }}</span>
    </div>
  `
})
export class EditorStatusComponent {
  private readonly ws = inject(WebSocketService);

  readonly activeUsers = toSignal(
    this.ws.messages$.pipe(
      filter(msg => msg.type === 'presence'),
      scan((users, msg) => {
        if (msg.action === 'join') return [...users, msg.user];
        return users.filter(u => u.id !== msg.user.id);
      }, [] as User[])
    ),
    { initialValue: [] }
  );

  readonly connectionStatus = toSignal(
    this.ws.connectionState$.pipe(
      map(state => {
        switch (state) {
          case 'connected': return 'Connected';
          case 'reconnecting': return 'Reconnecting...';
          case 'disconnected': return 'Disconnected';
        }
      })
    ),
    { initialValue: 'Connecting...' }
  );
}
```

### 3. Form with Cross-Field Validation via API

```typescript
@Component({
  selector: 'app-registration',
  template: `
    <form [formGroup]="form">
      <input formControlName="username" />
      @if (usernameStatus() === 'taken') {
        <span class="error">Username is already taken</span>
      }
      @if (usernameStatus() === 'checking') {
        <span class="hint">Checking availability...</span>
      }
      @if (usernameStatus() === 'available') {
        <span class="success">Username is available</span>
      }
    </form>
  `
})
export class RegistrationComponent {
  private readonly http = inject(HttpClient);
  readonly form = inject(FormBuilder).group({
    username: ['', Validators.required]
  });

  readonly usernameStatus = toSignal(
    this.form.get('username')!.valueChanges.pipe(
      debounceTime(400),
      distinctUntilChanged(),
      filter(v => v.length >= 3),
      switchMap(username =>
        concat(
          of('checking' as const),
          this.http.get<{ available: boolean }>(`/api/check-username/${username}`).pipe(
            map(res => res.available ? 'available' as const : 'taken' as const),
            catchError(() => of('error' as const))
          )
        )
      )
    ),
    { initialValue: 'idle' as const }
  );
}
```

---

> **Key Takeaway:** Use `toSignal()` to bring Observable data into the Signal world for templates and computed values, `toObservable()` to leverage RxJS operators on Signal changes, and always handle errors in the Observable pipeline before converting to a Signal.
