# 18.7 Combination Operators

> Combination operators merge, join, or coordinate multiple Observables into a single output stream, enabling reactive composition of independent data sources.

---

## Overview

Combination operators take two or more Observables and produce a single Observable. They differ in how they handle timing, ordering, and completion.

| Category | Operators |
|----------|----------|
| Merge (interleave) | `merge`, `mergeWith`, `mergeAll` |
| Concatenate (sequential) | `concat`, `concatWith`, `concatAll` |
| Latest values | `combineLatest`, `combineLatestWith`, `combineLatestAll` |
| Pair by index | `zip`, `zipWith`, `zipAll` |
| Wait for all | `forkJoin` |
| Side input | `withLatestFrom` |
| Race | `race`, `raceWith` |
| Prepend/Append | `startWith`, `endWith` |
| Pairing | `pairwise` |
| Splitting | `partition` |

Most combination operators come in two forms:
- **Static function:** `merge(a$, b$)` (creation-style, used at the start of a pipeline)
- **Pipeable operator:** `a$.pipe(mergeWith(b$))` (used inside `pipe()`)

---

## merge / mergeWith

Subscribes to all input Observables concurrently and forwards their emissions into a single stream as they arrive. Output order depends on emission timing, not input order.

```typescript
import { merge, interval, map, take } from 'rxjs';

const clicks$ = fromEvent(document, 'click').pipe(map(() => 'click'));
const keys$ = fromEvent(document, 'keydown').pipe(map(() => 'key'));
const timer$ = interval(5000).pipe(map(() => 'tick'));

// Static form
merge(clicks$, keys$, timer$).subscribe(v => console.log(v));

// Pipeable form
clicks$.pipe(
  mergeWith(keys$, timer$)
).subscribe(v => console.log(v));
```

```
A$:     ──1────3────5────|
B$:     ───2────4────6───|

merge(A$, B$):
        ──12───34───56───|
          (interleaved by timing)
```

**Concurrency limit:** The static `merge` accepts an optional `concurrent` parameter:

```typescript
// Subscribe to at most 2 Observables at a time
merge(obs1$, obs2$, obs3$, obs4$, 2);
// obs3$ subscribes only after obs1$ or obs2$ completes
```

**Completion:** `merge` completes only when ALL input Observables have completed. If any input never completes, the merged stream never completes.

Angular use case: merging multiple event sources:

```typescript
// Multiple ways to trigger a save
const autoSave$ = this.form.valueChanges.pipe(debounceTime(5000));
const manualSave$ = fromEvent(this.saveBtn.nativeElement, 'click');
const keyboardSave$ = fromEvent<KeyboardEvent>(document, 'keydown').pipe(
  filter(e => e.ctrlKey && e.key === 's'),
  tap(e => e.preventDefault())
);

merge(autoSave$, manualSave$, keyboardSave$).pipe(
  exhaustMap(() => this.saveData()),
  takeUntilDestroyed()
).subscribe();
```

### `mergeAll(concurrent?)`

Flattens a higher-order Observable (an Observable of Observables) by subscribing to all inner Observables concurrently.

```typescript
import { of, mergeAll, delay } from 'rxjs';

// Higher-order: Observable<Observable<number>>
of(
  of(1).pipe(delay(300)),
  of(2).pipe(delay(100)),
  of(3).pipe(delay(200))
).pipe(
  mergeAll()
).subscribe(v => console.log(v));
// 2 (100ms), 3 (200ms), 1 (300ms) -- ordered by completion time
```

`mergeAll()` is equivalent to `mergeMap(x => x)`. It is useful when you have a stream that produces Observables and you want to subscribe to all of them.

## concat / concatWith

Subscribes to Observables one at a time, in order. Each must complete before the next is subscribed to.

```typescript
import { concat, of, delay } from 'rxjs';

const first$ = of('A', 'B').pipe(delay(100));
const second$ = of('C', 'D').pipe(delay(100));
const third$ = of('E');

// Static form
concat(first$, second$, third$).subscribe(v => console.log(v));
// A, B (after 100ms), C, D (after 200ms), E (immediately)

// Pipeable form
first$.pipe(
  concatWith(second$, third$)
).subscribe(v => console.log(v));
```

```
A$:     ──1──2──|
B$:              ──3──4──|
C$:                       ──5──|

concat(A$, B$, C$):
        ──1──2──3──4──5──|
          (sequential, B waits for A, C waits for B)
```

**Critical gotcha:** If any source never completes, subsequent sources are never subscribed to:

```typescript
// Bad Practice: interval never completes
concat(
  interval(1000),  // Runs forever
  of('done')       // Never reached
);
```

### `concatAll()`

Flattens a higher-order Observable by subscribing to inner Observables sequentially.

```typescript
of(
  of(1, 2),
  of(3, 4),
  of(5, 6)
).pipe(
  concatAll()
).subscribe(v => console.log(v));
// 1, 2, 3, 4, 5, 6 (strict order)
```

## combineLatest / combineLatestWith

Emits an array (or object) of the latest values from each source whenever any source emits. Requires every source to have emitted at least once before producing the first output.

```typescript
import { combineLatest, interval, map, take } from 'rxjs';

const a$ = interval(1000).pipe(map(v => `A${v}`), take(3));
const b$ = interval(1500).pipe(map(v => `B${v}`), take(3));

// Object syntax (preferred)
combineLatest({
  a: a$,
  b: b$
}).subscribe(v => console.log(v));
// { a: 'A0', b: 'B0' } at 1.5s (both have emitted)
// { a: 'A1', b: 'B0' } at 2s
// { a: 'A1', b: 'B1' } at 3s
// { a: 'A2', b: 'B1' } at 3s
// { a: 'A2', b: 'B2' } at 4.5s

// Array syntax
combineLatest([a$, b$]).subscribe(([a, b]) => console.log(a, b));

// Pipeable form
a$.pipe(
  combineLatestWith(b$)
).subscribe(([a, b]) => console.log(a, b));
```

```
A$:     ──A0──A1──A2──|
B$:     ───────B0───────B1───────B2──|

combineLatest([A$, B$]):
        ───────[A0,B0]──[A1,B0]──[A2,B1]──[A2,B2]──|
                         [A1,B1]
                         [A2,B1]
```

**Key behaviors:**
- Does NOT emit until all sources have emitted at least once.
- After that, emits on every source emission with the latest from all others.
- Completes when all sources complete.

**The "initial value" problem:** If one source waits for user interaction (like a form control), `combineLatest` stalls. Use `startWith` to provide an initial value:

```typescript
combineLatest({
  search: this.searchControl.valueChanges.pipe(startWith('')),
  category: this.categoryControl.valueChanges.pipe(startWith('all')),
  page: this.page$.pipe(startWith(1))
}).pipe(
  switchMap(params => this.loadProducts(params))
);
```

**Completion:** Completes when ALL sources complete. If one source completes but the others keep emitting, `combineLatest` continues to produce values using the last value from the completed source.

**The glitch problem:** When multiple synchronous sources derived from the same parent emit in the same microtask, `combineLatest` can produce intermediate inconsistent values:

```typescript
const subject = new BehaviorSubject('A');
const mapped = subject.pipe(map(v => v.toLowerCase()));

combineLatest([subject, mapped]).subscribe(console.log);
subject.next('B');
// Produces: ['A', 'a'], ['B', 'a'], ['B', 'b']
//                        ^^^^^^^^ glitch: inconsistent intermediate state
```

The fix is `debounceTime(0)` to defer to the next microtask, so only the final consistent state survives:

```typescript
combineLatest([subject, mapped]).pipe(
  debounceTime(0)
).subscribe(console.log);
// Produces: ['A', 'a'], ['B', 'b'] -- no glitch
```

This is rarely an issue with async sources (HTTP, user events) but matters when deriving multiple streams from the same synchronous source.

**Deprecation details:**

| Version | Status |
|---------|--------|
| RxJS 6 | Pipeable `combineLatest` operator deprecated in favor of static function |
| RxJS 7 | `combineLatestWith` introduced as pipeable replacement; rest-argument signatures deprecated |
| RxJS 8+ | Pipeable `combineLatest` removed; use static `combineLatest()` or pipeable `combineLatestWith()` |

### `combineLatestAll()`

Flattens a higher-order Observable by collecting all inner Observables, then applying `combineLatest` to them.

```typescript
// Wait for all inner observables to emit, then combine latest
source$.pipe(
  map(id => this.http.get(`/api/${id}`)),
  combineLatestAll()
);
```

## forkJoin

Waits for ALL input Observables to complete, then emits an array (or object) containing the last value from each.

```typescript
import { forkJoin } from 'rxjs';

// Object syntax (preferred)
forkJoin({
  user: this.http.get<User>('/api/user'),
  settings: this.http.get<Settings>('/api/settings'),
  notifications: this.http.get<Notification[]>('/api/notifications')
}).subscribe(result => {
  console.log(result.user, result.settings, result.notifications);
});

// Array syntax
forkJoin([
  this.http.get<User>('/api/user'),
  this.http.get<Settings>('/api/settings')
]).subscribe(([user, settings]) => {
  console.log(user, settings);
});
```

```
A$:     ──1──2──3──|
B$:     ────A────B──|

forkJoin([A$, B$]):
        ────────────[3, B]|
                    ^ emits only on completion, with last values
```

**Key behaviors:**
- Emits exactly once, then completes.
- Only the LAST value from each source is included.
- If ANY source errors, forkJoin errors immediately and unsubscribes from all remaining sources. All successful results are lost.
- If ANY source completes without emitting, forkJoin completes without emitting (silent empty).

**Error handling pattern:** Catch errors at the inner level to prevent one failure from killing everything:

```typescript
forkJoin({
  critical: this.http.get<Data>('/api/data'),
  optional: this.http.get<Extra>('/api/extra').pipe(
    catchError(() => of(null))  // fallback so forkJoin doesn't fail
  )
}).subscribe(({ critical, optional }) => {
  // critical is guaranteed, optional might be null
});
```

**`forkJoin` vs `Promise.all`:**

| Feature | `forkJoin` | `Promise.all` |
|---------|-----------|---------------|
| Multiple values | Uses last value | N/A (Promises resolve once) |
| Lazy | Yes (executes on subscribe) | No (Promises are eager) |
| Cancellation | Unsubscribe cancels all | Not cancellable |
| Empty input | Emits `{}` or `[]` immediately | Resolves with `[]` |

**Empty source gotcha:**

```typescript
// EMPTY completes without emitting: forkJoin silently completes
forkJoin({
  data: this.http.get('/api/data'),
  extra: EMPTY  // No value!
}).subscribe({
  next: v => console.log(v),   // Never called!
  complete: () => console.log('Done') // Called immediately
});
```

To handle potentially empty sources, pipe them through `defaultIfEmpty`:

```typescript
forkJoin({
  data: this.http.get('/api/data'),
  extra: this.maybeEmpty$.pipe(defaultIfEmpty(null))
});
```

**Never-completing source gotcha:**

```typescript
// Bad Practice: BehaviorSubject never completes on its own
const subject = new BehaviorSubject('hello');
forkJoin({
  fromSubject: subject, // Hangs forever!
  fromHttp: this.http.get('/api/data')
});

// Fix: take(1) or first() to force completion
forkJoin({
  fromSubject: subject.pipe(take(1)),
  fromHttp: this.http.get('/api/data')
});
```

## withLatestFrom

When the source emits, combines that value with the latest values from other Observables. The other Observables do NOT trigger emissions on their own.

```typescript
import { interval, withLatestFrom, map } from 'rxjs';

const source$ = interval(2000); // Drives emissions
const other$ = interval(500);   // Only sampled, never triggers

source$.pipe(
  withLatestFrom(other$),
  map(([sourceVal, otherVal]) => `source: ${sourceVal}, other: ${otherVal}`)
).subscribe(v => console.log(v));
// At 2s: "source: 0, other: 3"
// At 4s: "source: 1, other: 7"
// At 6s: "source: 2, other: 11"
```

```
Source: ──────S0──────S1──────S2──|
Other:  ──O0──O1──O2──O3──O4──O5──O6──O7──...

withLatestFrom(Other):
        ──────[S0,O3]──[S1,O7]──[S2,O11]──|
              ^ only emits when Source emits
```

**Key difference from `combineLatest`:**
- `combineLatest`: ANY input emission triggers output.
- `withLatestFrom`: ONLY the source (left side) triggers output. The other inputs are passively sampled.

**Important:** If the other Observable has not emitted yet when the source emits, that source emission is silently dropped:

```typescript
const source$ = timer(0, 1000);    // Emits immediately at 0ms
const other$ = timer(500, 1000);   // First emit at 500ms

source$.pipe(
  withLatestFrom(other$)
).subscribe(v => console.log(v));
// First emission from source (at 0ms) is DROPPED because other$ hasn't emitted
// Output starts at 1000ms: [1, 0]
```

Angular use case: combining a user action with current state:

```typescript
this.saveButton$.pipe(
  withLatestFrom(this.form.valueChanges, this.currentUser$),
  filter(([_, formValues, user]) => user != null),
  switchMap(([_, formValues, user]) =>
    this.http.post(`/api/users/${user!.id}`, formValues)
  ),
  takeUntilDestroyed()
).subscribe();
```

## zip / zipWith

Combines values from multiple sources by index. Waits until all sources have emitted their Nth value, then emits the Nth combined value.

```typescript
import { zip, of, interval, take } from 'rxjs';

const names$ = of('Alice', 'Bob', 'Carol');
const ages$ = of(30, 25, 35);

// Static form
zip(names$, ages$).subscribe(([name, age]) => {
  console.log(`${name} is ${age}`);
});
// Alice is 30, Bob is 25, Carol is 35

// Pipeable form
names$.pipe(
  zipWith(ages$)
).subscribe(([name, age]) => console.log(`${name} is ${age}`));
```

```
A$:     ──1────2────3────|
B$:     ───A──────B──────C──|

zip(A$, B$):
        ───[1,A]──[2,B]──[3,C]──|
           ^ waits for both to have emitted index N
```

**Key behaviors:**
- Strictly index-based: emission N from source A pairs with emission N from source B.
- Completes when any source completes and all pairs have been emitted.
- If sources emit at very different rates, the faster source is buffered in memory.

**Memory warning:** If one source emits much faster than the other, `zip` buffers the fast source's values. This can cause unbounded memory growth on long-lived streams.

```typescript
// Bad Practice: huge memory buffer
zip(
  interval(1),     // Emits every 1ms
  interval(10000)  // Emits every 10s
);
// After 10s: 10,000 values from the first source are buffered
```

Use `combineLatest` or `withLatestFrom` instead when you want the latest values rather than index-paired values.

## race / raceWith

Subscribes to all sources, mirrors the first one to emit, and unsubscribes from all others permanently.

```typescript
import { race, timer, map, of, delay } from 'rxjs';

const primary$ = this.http.get('/api/data').pipe(delay(2000));
const fallback$ = timer(3000).pipe(map(() => cachedData));

// Use whichever responds first
race(primary$, fallback$).subscribe(v => console.log(v));
// If primary responds in 2s, uses primary
// If primary takes > 3s, uses fallback
```

```
A$:     ────────A1──A2──|
B$:     ──B1──B2──B3──|

race(A$, B$):
        ──B1──B2──B3──|
          ^ B emitted first, A is unsubscribed permanently
```

**Key behavior:** Once a winner is chosen, it is permanent for the lifetime of the subscription. The losing Observables are unsubscribed and never reconsidered.

Angular timeout pattern:

```typescript
// Timeout with fallback
const dataWithTimeout$ = race(
  this.http.get<Config>('/api/config'),
  timer(5000).pipe(map(() => this.defaultConfig))
);
```

## startWith / endWith

### `startWith(...values)`

Prepends synchronous values before the source Observable starts emitting.

```typescript
import { of, startWith } from 'rxjs';

of(3, 4, 5).pipe(
  startWith(1, 2)
).subscribe(v => console.log(v));
// 1, 2, 3, 4, 5
```

```
Source:  ──3──4──5──|
startWith(1, 2)
Output:  (12)──3──4──5──|
          ^ synchronous, before source
```

This is one of the most used operators in Angular for providing initial values to `combineLatest` and `switchMap` pipelines:

```typescript
// Provide initial value for combineLatest
combineLatest({
  search: this.searchControl.valueChanges.pipe(startWith('')),
  sort: this.sortControl.valueChanges.pipe(startWith('name'))
});

// Provide loading state before HTTP response
this.http.get<User[]>('/api/users').pipe(
  map(users => ({ loading: false, data: users })),
  startWith({ loading: true, data: [] as User[] })
);
```

### `endWith(...values)`

Appends synchronous values after the source Observable completes.

```typescript
import { of, endWith } from 'rxjs';

of(1, 2, 3).pipe(
  endWith(4, 5)
).subscribe(v => console.log(v));
// 1, 2, 3, 4, 5
```

```typescript
// Mark operation as complete
this.http.post('/api/save', data).pipe(
  ignoreElements(),
  endWith('saved')
);
```

**Note:** `endWith` values are emitted only if the source completes. If the source errors, `endWith` does not run.

## pairwise

Emits the previous and current value as a 2-element array. Does not emit until the source has emitted at least two values.

```typescript
import { pairwise } from 'rxjs';

of(1, 2, 3, 4).pipe(
  pairwise()
).subscribe(console.log);
// [1, 2], [2, 3], [3, 4]
```

```
Source:  --1--2--3--4--|

pairwise()

Output:  -----[1,2]--[2,3]--[3,4]--|
              ^ first output requires 2 source emissions
```

Angular use cases:

```typescript
// Track previous and current route for analytics
inject(Router).events.pipe(
  filter((e): e is NavigationEnd => e instanceof NavigationEnd),
  map(e => e.urlAfterRedirects),
  pairwise(),
  takeUntilDestroyed()
).subscribe(([previousUrl, currentUrl]) => {
  this.analytics.trackNavigation(previousUrl, currentUrl);
});

// Detect scroll direction
fromEvent(window, 'scroll').pipe(
  map(() => window.scrollY),
  pairwise(),
  map(([prev, curr]) => curr > prev ? 'down' : 'up'),
  distinctUntilChanged(),
  takeUntilDestroyed()
).subscribe(direction => this.headerService.setScrollDirection(direction));
```

**Combining `startWith` with `pairwise`:** Since `pairwise` swallows the first emission, prepend a known initial value to get output for every input:

```typescript
// Without startWith: first movement from origin is lost
this.position$.pipe(pairwise());

// With startWith: [0, firstPosition] emitted immediately
this.position$.pipe(
  startWith(0),
  pairwise()
);
```

## partition

Splits a single Observable into two Observables based on a predicate: one for values that match, one for values that do not.

```typescript
import { partition, from } from 'rxjs';

const source$ = from([1, 2, 3, 4, 5, 6]);
const [even$, odd$] = partition(source$, x => x % 2 === 0);

even$.subscribe(v => console.log('Even:', v)); // 2, 4, 6
odd$.subscribe(v => console.log('Odd:', v));   // 1, 3, 5
```

```
Source:  --1--2--3--4--5--6--|

partition(source$, x => x % 2 === 0)

even$:   -----2-----4-----6--|
odd$:    --1-----3-----5-----|
```

Angular use case:

```typescript
// Split WebSocket messages into data updates and control messages
const [data$, control$] = partition(
  this.wsService.messages$.pipe(share()),
  (msg): msg is DataMessage => msg.type === 'data'
);

data$.pipe(takeUntilDestroyed()).subscribe(msg => this.store.update(msg.payload));
control$.pipe(takeUntilDestroyed()).subscribe(msg => this.handleControl(msg));
```

**Gotcha: double subscription.** Each output subscribes to the source independently. If the source is cold (like HTTP), it executes twice. Always `share()` the source first.

## Choosing the Right Combination Operator

This is the most common source of confusion with RxJS. Here is a decision guide:

### Decision Table

| Question | Answer | Use |
|----------|--------|-----|
| Do all sources complete (like HTTP)? | Yes | `forkJoin` |
| Do you need ongoing updates when any source changes? | Yes | `combineLatest` |
| Should only one source trigger output? | Yes | `withLatestFrom` |
| Must sources be processed in strict order? | Yes | `concat` |
| Should all sources run concurrently, output interleaved? | Yes | `merge` |
| Should values be paired by emission index? | Yes | `zip` |
| Do you want the first source to respond? | Yes | `race` |

### `combineLatest` vs `withLatestFrom` vs `forkJoin`

These three are the most frequently confused:

```typescript
// combineLatest: emits when ANY source emits (after all have emitted once)
combineLatest([a$, b$]).subscribe(([a, b]) => ...);
// Use for: reactive filters, multi-input forms, dashboards

// withLatestFrom: emits only when the SOURCE emits, sampling others
source$.pipe(withLatestFrom(other$)).subscribe(([s, o]) => ...);
// Use for: button click + current state, action + context

// forkJoin: emits ONCE when ALL complete, with their last values
forkJoin([a$, b$]).subscribe(([a, b]) => ...);
// Use for: parallel HTTP requests, initialization data loading
```

| Feature | `combineLatest` | `withLatestFrom` | `forkJoin` |
|---------|----------------|------------------|-----------|
| Emissions | On any input change | Only on source change | Once, on all complete |
| Requires all to emit first | Yes | Yes (others only) | Yes (must complete) |
| Works with infinite streams | Yes | Yes | No (needs completion) |
| Number of outputs | Many (ongoing) | Many (source-driven) | One |
| Typical use | Reactive state | Action + context | Parallel requests |

## Flattening Higher-Order Observables

When an operator produces Observables of Observables, you need a flattening strategy:

| Flattening Operator | Strategy |
|--------------------|----------|
| `mergeAll()` | Subscribe to all concurrently |
| `concatAll()` | Subscribe sequentially, wait for each to complete |
| `switchAll()` | Cancel previous, subscribe to latest |
| `exhaustAll()` | Ignore new while current is active |
| `combineLatestAll()` | Apply `combineLatest` to collected inner Observables |
| `zipAll()` | Apply `zip` to collected inner Observables |

```typescript
// switchAll example: cancel previous
source$.pipe(
  map(id => this.http.get(`/api/${id}`)), // Observable<Observable<Data>>
  switchAll()  // Equivalent to switchMap(id => this.http.get(...))
);
```

In practice, the `*Map` variants (`switchMap`, `mergeMap`, `concatMap`, `exhaustMap`) are more commonly used than the `*All` variants because they combine mapping and flattening in one step.

## The Pipeable "With" Operators (RxJS 7+)

RxJS 7 introduced pipeable alternatives to the static creation functions for use inside `.pipe()`:

| Static Function | Pipeable Equivalent | Notes |
|----------------|---------------------|-------|
| `merge(a$, b$)` | `a$.pipe(mergeWith(b$))` | Identical behavior |
| `concat(a$, b$)` | `a$.pipe(concatWith(b$))` | Identical behavior |
| `combineLatest([a$, b$])` | `a$.pipe(combineLatestWith(b$))` | Replaces deprecated pipeable `combineLatest` |
| `zip(a$, b$)` | `a$.pipe(zipWith(b$))` | Identical behavior |
| `race(a$, b$)` | `a$.pipe(raceWith(b$))` | Identical behavior |

Use the static form when creating a new stream from multiple sources at the top level. Use the pipeable form when adding combination to an existing pipeline chain. Both produce identical results.

## Common Mistakes

- **Using `forkJoin` with Observables that never complete.** `forkJoin` waits for all sources to complete. `BehaviorSubject`, `interval`, `fromEvent`, and other long-lived Observables never complete on their own. Use `take(1)` or `first()` if you need just the current value.

- **Forgetting `startWith` when using `combineLatest`.** `combineLatest` does not emit until ALL sources have emitted at least once. If a source depends on user interaction, the entire pipeline stalls until the user acts. Always provide `startWith` for sources that may not emit immediately.

- **Confusing `combineLatest` with `withLatestFrom`.** `combineLatest` emits when ANY source changes. `withLatestFrom` emits ONLY when the source (left side) changes. Using `combineLatest` when you mean `withLatestFrom` causes unexpected emissions from passive data sources.

- **Silent empty from `forkJoin` with `EMPTY` sources.** If any input completes without emitting a value, `forkJoin` completes without emitting anything. This is silent and hard to debug. Use `defaultIfEmpty(null)` on potentially empty sources.

- **Using `zip` with mismatched emission rates.** `zip` buffers the faster source in memory. With long-lived streams emitting at very different rates, this can cause unbounded memory growth. Prefer `combineLatest` or `withLatestFrom` unless you specifically need index-based pairing.

- **Using deprecated `combineLatest` operator form.** The pipeable `combineLatest` operator (from `rxjs/operators`) was deprecated in RxJS 7. Use the static `combineLatest()` function or the `combineLatestWith` pipeable operator instead:

```typescript
// Deprecated: pipeable combineLatest
import { combineLatest } from 'rxjs/operators'; // DO NOT
source$.pipe(combineLatest(other$));

// Good: static function
import { combineLatest } from 'rxjs';
combineLatest([source$, other$]);

// Good: pipeable combineLatestWith
import { combineLatestWith } from 'rxjs';
source$.pipe(combineLatestWith(other$));
```

- **Not using object syntax for `combineLatest` and `forkJoin`.** With 3+ sources, positional arrays are error-prone and hard to read. Object syntax names each source:

```typescript
// Bad Practice: which is which?
forkJoin([getUserReq, getOrderReq, getPrefsReq])
  .subscribe(([a, b, c]) => ...); // What is a? b? c?

// Good: self-documenting
forkJoin({
  user: getUserReq,
  orders: getOrderReq,
  prefs: getPrefsReq
}).subscribe(({ user, orders, prefs }) => ...);
```

- **Using `merge` when `concat` is intended (or vice versa).** `merge` runs sources concurrently (output order depends on timing). `concat` runs sources sequentially (guaranteed order). Mixing them up causes ordering bugs.

---

## Best Practices

- **Use `combineLatest` with object syntax** for multi-source reactive state. It is the backbone of reactive Angular: combining filters, sort orders, pagination, and search terms into a single data-loading pipeline.

- **Use `forkJoin` for parallel HTTP requests** that are independent and all needed before proceeding. It is the Observable equivalent of `Promise.all()`.

- **Use `withLatestFrom` for action-driven patterns** where a user event (click, submit) should be combined with the current state but the state itself should not trigger the action.

- **Provide `startWith` for every `combineLatest` input** that might not emit immediately. This prevents the pipeline from stalling.

- **Use `merge` to combine independent event streams** that should trigger the same behavior (save button, keyboard shortcut, auto-save timer).

- **Prefer `concat` for ordered sequential operations** where the next step depends on the previous completing (multi-step forms, sequential API calls).

- **Use `toSignal()` at the end of combination pipelines** in Angular 21 to convert the final Observable to a Signal for template consumption.

### Save Button with Current Form State

```typescript

  template: `
    <form [formGroup]="form">
      <input formControlName="name" />
      <input formControlName="email" />
    </form>
    <button #saveBtn>Save</button>
    <span>{{ saveStatus() }}</span>
  `

export class ProfileEditorComponent {
  private http = inject(HttpClient);
  private destroyRef = inject(DestroyRef);
  private saveBtn = viewChild<ElementRef>('saveBtn');

  form = new FormGroup({
    name: new FormControl(''),
    email: new FormControl('')
  });

  saveStatus = signal('');

  ngAfterViewInit() {
    fromEvent(this.saveBtn()!.nativeElement, 'click').pipe(
      withLatestFrom(this.form.valueChanges.pipe(startWith(this.form.value))),
      filter(([_, values]) => this.form.valid),
      exhaustMap(([_, values]) => {
        this.saveStatus.set('Saving...');
        return this.http.put('/api/profile', values).pipe(
          tap(() => this.saveStatus.set('Saved')),
          catchError(err => {
            this.saveStatus.set('Error: ' + err.message);
            return EMPTY;
          })
        );
      }),
      takeUntilDestroyed(this.destroyRef)
    ).subscribe();
  }
}
```

## Quick Reference Table

| Operator | Emits When | Output | Completes When | Use For |
|----------|-----------|--------|---------------|---------|
| `combineLatest` | Any source emits (after all have emitted once) | Latest from all | All sources complete | Live data combination |
| `forkJoin` | All sources complete | Last from all (once) | Immediately after emit | Parallel HTTP requests |
| `withLatestFrom` | Primary emits (after secondaries have emitted) | Primary + latest secondaries | Primary completes | Event + context snapshot |
| `merge` | Any source emits | Single value from that source | All sources complete | Combining event streams |
| `concat` | Current source emits | Single value from current source | Last source completes | Sequential operations |
| `zip` | All sources have emitted Nth value | Nth values from all | Shortest source completes | Index-paired coordination |
| `race` | First source to emit | Values from winning source only | Winning source completes | Fastest response wins |
| `partition` | Source emits | Two separate streams | Source completes | Splitting by condition |
| `startWith` | Immediately on subscribe | Prepended value(s) | Source completes | Initial/default values |
| `endWith` | Source completes | Appended value(s) | After appended values | Completion signal |
| `pairwise` | Source emits (after 2nd emission) | [previous, current] | Source completes | Change detection |

---

> **Key Takeaway:** Choose combination operators based on how you need sources coordinated: `combineLatest` for reactive state (any change triggers update), `forkJoin` for parallel one-shot operations (all must complete), `withLatestFrom` for action-driven patterns (source triggers, others sampled), `merge` for interleaving independent event streams, and `concat` for ordered sequential processing. Always provide `startWith` values for `combineLatest` inputs and use object syntax for readability.
