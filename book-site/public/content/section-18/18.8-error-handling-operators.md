# 18.8 Error Handling Operators

> Error handling in RxJS determines whether a failed Observable recovers with a fallback, retries the operation, or propagates the error to subscribers. Proper error handling keeps reactive pipelines alive and applications resilient.

---

## How Errors Work in Observables

An Observable can terminate in two ways:
1. **Complete:** The stream ends successfully (no more values).
2. **Error:** The stream ends with an error (no more values, no completion).

Once an error occurs, the Observable is dead. No more `next` or `complete` notifications are delivered. This is a critical difference from try/catch in imperative code where execution can continue after a caught error.

```typescript
import { Observable } from 'rxjs';

const source$ = new Observable<number>(subscriber => {
  subscriber.next(1);
  subscriber.next(2);
  subscriber.error(new Error('Something broke'));
  subscriber.next(3);     // Never delivered
  subscriber.complete();  // Never delivered
});

source$.subscribe({
  next: v => console.log(v),       // 1, 2
  error: err => console.error(err), // Error: Something broke
  complete: () => console.log('Done') // Never called
});
```

Error handling operators intercept the error notification and either replace the errored Observable with a new one, retry the original, or transform the error before it reaches the subscriber.

| Operator | Purpose |
|----------|---------|
| `catchError` | Replace errored Observable with a fallback |
| `retry` | Resubscribe to the source on error |
| `throwError` | Create an Observable that immediately errors |
| `finalize` | Run cleanup logic on completion, error, or unsubscribe |
| `timeout` | Error if the source does not emit within a time limit |
| `tap` (error handler) | Side effects on error without modifying the stream |

---

## catchError

The primary error handling operator. Intercepts an error and returns a replacement Observable.

```typescript
import { of, catchError, map, throwError } from 'rxjs';

// Strategy 1: Return fallback data
this.http.get<User[]>('/api/users').pipe(
  catchError(err => {
    console.error('Request failed:', err);
    return of([]); // Return empty array as fallback
  })
);

// Strategy 2: Rethrow a transformed error
this.http.get<User[]>('/api/users').pipe(
  catchError(err => {
    return throwError(() => new AppError('Failed to load users', err));
  })
);

// Strategy 3: Return EMPTY (swallow error, complete silently)
this.http.get<void>('/api/audit-log').pipe(
  catchError(() => EMPTY) // Silently ignore failures
);
```

### Signature

```typescript
catchError<T, O extends ObservableInput<any>>(
  selector: (err: any, caught: Observable<T>) => O
): OperatorFunction<T, T | ObservedValueOf<O>>
```

The `selector` function receives:
- `err`: The error that occurred.
- `caught`: The original source Observable (useful for retry-like behavior).

### Fallback Patterns

**Return static data:**

```typescript
this.http.get<Config>('/api/config').pipe(
  catchError(() => of(DEFAULT_CONFIG))
);
```

**Return a different Observable:**

```typescript
this.http.get<Data>('/api/primary').pipe(
  catchError(() => this.http.get<Data>('/api/fallback'))
);
```

**Rethrow with more context:**

```typescript
this.http.get<User>('/api/user').pipe(
  catchError(err => throwError(() => ({
    code: err.status,
    message: `User API failed: ${err.statusText}`,
    original: err
  })))
);
```

**Swallow the error:**

```typescript
// Use EMPTY to silently discard the error and complete
this.analytics.track(event).pipe(
  catchError(() => EMPTY)
);
```

### The `caught` Parameter for Retry

The second parameter is the source Observable, allowing you to retry by returning it:

```typescript
let attempts = 0;

this.http.get('/api/data').pipe(
  catchError((err, caught) => {
    attempts++;
    if (attempts < 3) {
      return caught; // Resubscribe to the source (retry)
    }
    return throwError(() => err); // Give up after 3 attempts
  })
);
```

However, the `retry` operator is better suited for retry logic. Use `caught` only for unusual retry patterns.

### Placement: Inner vs Outer

Where you place `catchError` determines what happens after the error.

**Outer catchError (kills the pipeline on error):**

```typescript
// Bad Practice for ongoing streams: outer catchError kills the pipeline
this.searchControl.valueChanges.pipe(
  switchMap(term => this.http.get(`/api/search?q=${term}`)),
  catchError(() => of([])) // Entire pipeline completes after first error!
).subscribe(results => this.results.set(results));
// After one failed search, no future searches will work
```

**Inner catchError (keeps the pipeline alive):**

```typescript
// Good: inner catchError recovers per-request, pipeline stays alive
this.searchControl.valueChanges.pipe(
  switchMap(term =>
    this.http.get(`/api/search?q=${term}`).pipe(
      catchError(() => of([])) // Only this request fails, pipeline continues
    )
  )
).subscribe(results => this.results.set(results));
// Failed searches return [], future searches still work
```

This is one of the most important error handling concepts in RxJS: errors propagate outward through the operator chain. Once an error reaches the outermost Observable, the entire subscription is dead. Catch errors as close to the source as possible to keep outer pipelines alive.

## retry

Resubscribes to the source Observable when an error occurs. In RxJS 7+, `retry` accepts a configuration object for fine-grained control.

### Basic Retry

```typescript
import { retry } from 'rxjs';

// Retry up to 3 times
this.http.get('/api/data').pipe(
  retry(3) // Shorthand for retry({ count: 3 })
);
```

```
Source attempt 1: ──1──X (error)
Source attempt 2: ──1──X (error)
Source attempt 3: ──1──X (error)
Source attempt 4: ──1──X (error propagated to subscriber)
                  ^ 1 original + 3 retries = 4 total attempts
```

### RetryConfig (RxJS 7+)

```typescript
import { retry, timer } from 'rxjs';

this.http.get('/api/data').pipe(
  retry({
    count: 3,                    // Maximum number of retries
    delay: 1000,                 // Fixed delay between retries (ms)
    resetOnSuccess: false        // Reset retry count on successful emission
  })
);
```

### Configurable Delay

The `delay` property accepts a number (fixed ms), an Observable factory, or a function:

```typescript
// Fixed delay
retry({ count: 3, delay: 2000 });

// Exponential backoff
retry({
  count: 4,
  delay: (error, retryCount) => timer(Math.pow(2, retryCount) * 1000)
  // 2s, 4s, 8s, 16s
});

// Delay with jitter (prevents thundering herd)
retry({
  count: 4,
  delay: (error, retryCount) => {
    const base = Math.pow(2, retryCount) * 1000;
    const jitter = Math.random() * 1000;
    return timer(base + jitter);
  }
});

// Conditional retry (only retry certain errors)
retry({
  count: 3,
  delay: (error, retryCount) => {
    if (error.status === 404) {
      // Don't retry 404s
      return throwError(() => error);
    }
    return timer(1000 * retryCount);
  }
});
```

### `resetOnSuccess`

When `true`, the retry counter resets whenever the source emits a value successfully. This is useful for long-lived Observables like WebSocket connections:

```typescript
this.webSocket$.pipe(
  retry({
    count: 10,
    delay: 3000,
    resetOnSuccess: true // Reset counter after each successful message
  })
);
// If the WebSocket reconnects and receives messages, the retry count
// resets to 0. Only fails after 10 consecutive retries with no success.
```

### Deprecated: `retryWhen`

`retryWhen` was deprecated in RxJS 7 and removed in RxJS 8. It was replaced by the `delay` function in `retry`:

```typescript
// Deprecated: retryWhen
source$.pipe(
  retryWhen(errors$ =>
    errors$.pipe(delay(1000), take(3))
  )
);

// Modern replacement: retry with delay
source$.pipe(
  retry({
    count: 3,
    delay: 1000
  })
);

// Complex retryWhen replacement
source$.pipe(
  retry({
    count: 3,
    delay: (error, retryCount) => {
      console.log(`Retry ${retryCount} after error:`, error);
      return timer(retryCount * 1000);
    }
  })
);
```

## throwError

Creates an Observable that immediately emits an error notification. Used inside `catchError` to rethrow or transform errors.

```typescript
import { throwError } from 'rxjs';

// Factory function (RxJS 7+ required)
throwError(() => new Error('Something went wrong'));

// Used in catchError to rethrow
source$.pipe(
  catchError(err => {
    if (err.status === 401) {
      this.authService.logout();
      return throwError(() => new UnauthorizedError());
    }
    return throwError(() => err); // Rethrow unhandled errors
  })
);
```

**Always use the factory form:** The direct value form `throwError(new Error(...))` is deprecated in RxJS 7. The factory ensures a fresh stack trace per subscription.

## finalize

Runs a callback when the Observable terminates for any reason: complete, error, or unsubscribe. It is the Observable equivalent of `finally` in try/catch/finally.

```typescript
import { finalize } from 'rxjs';

this.http.get('/api/data').pipe(
  finalize(() => {
    this.loading.set(false); // Always runs: success, error, or cancel
  })
).subscribe({
  next: data => this.data.set(data),
  error: err => this.error.set(err.message)
});
```

### finalize vs tap

```typescript
// tap: separate handlers for next, error, complete
source$.pipe(
  tap({
    next: v => console.log('Value:', v),
    error: err => console.error('Error:', err),
    complete: () => console.log('Complete')
  })
);

// finalize: single handler for ANY termination
source$.pipe(
  finalize(() => console.log('Stream ended'))
);
```

| | `tap({ error })` | `tap({ complete })` | `finalize` |
|---|---|---|---|
| On error | Yes | No | Yes |
| On complete | No | Yes | Yes |
| On unsubscribe | No | No | Yes |

`finalize` is the only way to run cleanup when the subscriber unsubscribes before completion.

### Execution Order

`finalize` runs after the error or complete notification has been delivered to the subscriber:

```typescript
source$.pipe(
  finalize(() => console.log('3: finalize'))
).subscribe({
  error: err => console.log('1: subscriber error'),
  complete: () => console.log('2: subscriber complete')
});
// On error: "1: subscriber error", "3: finalize"
// On complete: "2: subscriber complete", "3: finalize"
```

Angular loading state pattern:

```typescript
@Component({ ... })
export class DataComponent {
  loading = signal(false);
  data = signal<Item[]>([]);
  error = signal<string | null>(null);

  loadData(): void {
    this.loading.set(true);
    this.error.set(null);

    this.http.get<Item[]>('/api/items').pipe(
      finalize(() => this.loading.set(false))
    ).subscribe({
      next: items => this.data.set(items),
      error: err => this.error.set(err.message)
    });
  }
}
```

## timeout

Errors if the source Observable does not emit within a specified time.

### `timeout(config)`

```typescript
import { timeout } from 'rxjs';

// Timeout after 5 seconds of no emission
this.http.get('/api/slow-endpoint').pipe(
  timeout(5000) // Throws TimeoutError after 5s
);

// With configuration
this.http.get('/api/slow-endpoint').pipe(
  timeout({
    first: 5000,  // Time allowed for first emission
    each: 3000,   // Time allowed between subsequent emissions
    with: () => throwError(() => new Error('Request timed out'))
  })
);
```

### Timeout with Fallback

```typescript
// Switch to a fallback Observable on timeout
this.http.get<Data>('/api/primary').pipe(
  timeout({
    first: 3000,
    with: () => this.http.get<Data>('/api/fallback')
  })
);

// Or use catchError after timeout
this.http.get<Data>('/api/data').pipe(
  timeout(5000),
  catchError(err => {
    if (err.name === 'TimeoutError') {
      return of(DEFAULT_DATA);
    }
    return throwError(() => err);
  })
);
```

### `timeout` Configuration

| Option | Type | Description |
|--------|------|-------------|
| `first` | `number \| Date` | Maximum wait for the first emission |
| `each` | `number` | Maximum wait between emissions |
| `with` | `() => ObservableInput` | Factory returning fallback Observable |
| `meta` | `any` | Custom metadata attached to `TimeoutError` |

If only a number is passed, it sets `first` (timeout for first emission).

## tap for Error Side Effects

`tap` can observe errors without modifying the stream. Use it for logging, analytics, or side effects.

```typescript
import { tap } from 'rxjs';

this.http.get('/api/data').pipe(
  tap({
    next: data => console.log('Received:', data),
    error: err => {
      console.error('API Error:', err);
      this.analytics.trackError('data_load_failed', err);
    },
    complete: () => console.log('Request complete')
  }),
  catchError(() => of(DEFAULT_DATA))
);
```

`tap` does not modify or intercept the error. The error continues to propagate after `tap` processes it. To handle the error (stop propagation), use `catchError`.

Since RxJS 7.5, `tap` also accepts `subscribe`, `unsubscribe`, and `finalize` callbacks in its observer:

```typescript
source$.pipe(
  tap({
    subscribe: () => console.log('Subscribed'),
    next: v => console.log('Value:', v),
    error: err => console.error('Error:', err),
    complete: () => console.log('Complete'),
    unsubscribe: () => console.log('Unsubscribed'),
    finalize: () => console.log('Finalized (any termination)')
  })
);
```

The `finalize` callback inside `tap` behaves identically to the standalone `finalize` operator.

## Error Handling Strategies

### Strategy 1: Catch and Replace

Replace the failed Observable with fallback data. The subscriber receives the fallback as a normal `next` emission.

```typescript
this.http.get<User[]>('/api/users').pipe(
  catchError(() => of([]))
).subscribe(users => {
  // users is either real data or empty array
  this.users.set(users);
});
```

**When to use:** When you have a reasonable default and the user should not see an error state.

### Strategy 2: Catch and Rethrow

Transform the error into a domain-specific error, then let it propagate.

```typescript
this.http.get<User>('/api/user').pipe(
  catchError(err => {
    if (err.status === 404) {
      return throwError(() => new UserNotFoundError(err));
    }
    if (err.status === 403) {
      return throwError(() => new ForbiddenError(err));
    }
    return throwError(() => new UnknownApiError(err));
  })
).subscribe({
  next: user => this.user.set(user),
  error: err => {
    if (err instanceof UserNotFoundError) {
      this.router.navigate(['/not-found']);
    } else if (err instanceof ForbiddenError) {
      this.router.navigate(['/forbidden']);
    }
  }
});
```

**When to use:** When different error types require different handling by the consumer.

### Strategy 3: Retry Then Fallback

Attempt the operation multiple times before falling back.

```typescript
this.http.get<Data>('/api/data').pipe(
  retry({
    count: 3,
    delay: (error, retryCount) => {
      // Only retry server errors (5xx)
      if (error.status < 500) {
        return throwError(() => error);
      }
      return timer(retryCount * 1000);
    }
  }),
  catchError(() => of(CACHED_DATA)) // Final fallback after all retries
);
```

**When to use:** For transient errors (network glitches, server overload) where a retry might succeed.

### Strategy 4: Notify and Recover

Show an error notification to the user but keep the pipeline alive.

```typescript
this.searchControl.valueChanges.pipe(
  debounceTime(300),
  switchMap(term =>
    this.http.get<Result[]>(`/api/search?q=${term}`).pipe(
      catchError(err => {
        this.notificationService.showError('Search failed. Please try again.');
        return of([]); // Return empty results, pipeline stays alive
      })
    )
  ),
  takeUntilDestroyed()
).subscribe(results => this.results.set(results));
```

**When to use:** For user-facing features where errors should be visible but not block further interaction.

### Strategy 5: Redirect on Auth Errors

```typescript
this.http.get<Data>('/api/protected-data').pipe(
  catchError(err => {
    if (err.status === 401) {
      this.authService.logout();
      this.router.navigate(['/login']);
      return EMPTY; // Don't emit, just redirect
    }
    return throwError(() => err);
  })
);
```

## Error Handling in Angular HTTP Interceptors

Angular's `HttpInterceptor` (or functional interceptor in Angular 21) provides a centralized place for error handling across all HTTP requests.

### Functional Interceptor (Angular 21)

```typescript
import { HttpInterceptorFn, HttpErrorResponse } from '@angular/common/http';
import { catchError, throwError } from 'rxjs';

export const errorInterceptor: HttpInterceptorFn = (req, next) => {
  return next(req).pipe(
    catchError((error: HttpErrorResponse) => {
      if (error.status === 401) {
        inject(AuthService).logout();
        inject(Router).navigate(['/login']);
      }

      if (error.status === 0) {
        // Network error
        console.error('Network error:', error);
      }

      if (error.status >= 500) {
        inject(NotificationService).showError('Server error. Please try later.');
      }

      return throwError(() => error);
    })
  );
};

// Register in app config
export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(
      withInterceptors([errorInterceptor])
    )
  ]
};
```

### Retry Interceptor

```typescript
export const retryInterceptor: HttpInterceptorFn = (req, next) => {
  // Only retry GET requests (idempotent)
  if (req.method !== 'GET') {
    return next(req);
  }

  return next(req).pipe(
    retry({
      count: 2,
      delay: (error, retryCount) => {
        // Only retry on server errors or network failures
        if (error instanceof HttpErrorResponse && error.status < 500 && error.status !== 0) {
          return throwError(() => error);
        }
        return timer(retryCount * 1000);
      }
    })
  );
};
```

### Timeout Interceptor

```typescript
export const timeoutInterceptor: HttpInterceptorFn = (req, next) => {
  const timeoutMs = Number(req.headers.get('X-Timeout')) || 30000;

  return next(req).pipe(
    timeout(timeoutMs),
    catchError(err => {
      if (err.name === 'TimeoutError') {
        return throwError(() => new HttpErrorResponse({
          error: 'Request timed out',
          status: 408,
          statusText: 'Request Timeout',
          url: req.url
        }));
      }
      return throwError(() => err);
    })
  );
};
```

## Common Mistakes

- **Placing `catchError` outside `switchMap` and expecting the pipeline to survive.** An outer `catchError` replaces the entire Observable, and the pipeline completes or switches to the fallback permanently. Place `catchError` inside the inner Observable to handle errors per-emission while keeping the outer stream alive.

```typescript
// Bad Practice: pipeline dies after first error
source$.pipe(
  switchMap(id => this.http.get(`/api/${id}`)),
  catchError(() => of(null)) // Stream completes after one error
);

// Good: each request handles its own errors
source$.pipe(
  switchMap(id =>
    this.http.get(`/api/${id}`).pipe(
      catchError(() => of(null)) // Only this request fails
    )
  )
);
```

- **Returning a non-Observable from `catchError`.** The return value of `catchError` must be an Observable. Returning a plain value throws a runtime error:

```typescript
// Bad Practice: returns a plain value, not an Observable
catchError(() => []);

// Good: wrap in of()
catchError(() => of([]));
```

- **Using `retry` without a delay.** Immediate retries can overwhelm a struggling server. Always add a delay between retries, especially for network requests:

```typescript
// Bad Practice: hammers the server immediately
retry(3);

// Good: exponential backoff
retry({
  count: 3,
  delay: (_, retryCount) => timer(Math.pow(2, retryCount) * 1000)
});
```

- **Retrying non-idempotent requests.** Retrying POST, PUT, or DELETE requests can cause duplicate operations (double charges, duplicate records). Only retry GET requests automatically. For write operations, let the user decide whether to retry.

- **Not handling errors at all.** An unhandled error in an Observable kills the stream silently. In Angular, an unhandled HTTP error logs to the console but the component may show stale or empty data with no feedback to the user.

- **Using `catchError` to swallow all errors.** Catching every error with `catchError(() => EMPTY)` hides bugs. At minimum, log the error. Better yet, show user-facing feedback or track the error in analytics.

- **Confusing `finalize` with `catchError`.** `finalize` runs a side effect but does not change the stream. It does not catch or swallow errors. If you need to both log and recover, use both:

```typescript
source$.pipe(
  catchError(err => {
    this.logger.error(err);
    return of(fallbackData);
  }),
  finalize(() => this.loading.set(false))
);
```

- **Using the deprecated `throwError(value)` form.** Always use the factory: `throwError(() => new Error(...))`. The factory produces fresh stack traces per subscription.

---

## Best Practices

- **Catch errors at the innermost possible level.** Place `catchError` inside `switchMap`, `mergeMap`, or `concatMap` to handle errors per-operation without killing the outer pipeline.

- **Use `retry` with exponential backoff and jitter** for network requests. This prevents thundering herd problems where many clients retry simultaneously.

- **Only retry idempotent operations automatically.** GET requests are safe to retry. POST/PUT/DELETE should be retried with caution or not at all.

- **Centralize common error handling in HTTP interceptors.** Handle 401 (redirect to login), 500 (show notification), and network errors (show offline banner) in an interceptor. Let component-level `catchError` handle business-specific error cases.

- **Use `finalize` for cleanup that must happen regardless of outcome.** Loading states, spinners, and resource cleanup belong in `finalize`, not in separate error and complete handlers.

- **Transform API errors into domain errors.** Create custom error classes (`UserNotFoundError`, `ValidationError`) in `catchError` so consumers can make decisions based on error type, not HTTP status codes.

- **Always provide user feedback on error.** Silent failures are the worst user experience. Either show an error message, provide a retry button, or display cached/default data with an indicator.

- **Use `timeout` for external API calls.** HTTP requests can hang indefinitely (especially behind slow proxies). Set reasonable timeouts and handle `TimeoutError` with fallback data or user notification.

## Real-World Use Cases

### Resilient API Service

```typescript
@Injectable({ providedIn: 'root' })
export class ApiService {
  private http = inject(HttpClient);
  private notification = inject(NotificationService);

  get<T>(url: string, fallback?: T): Observable<T> {
    return this.http.get<T>(url).pipe(
      timeout(10000),
      retry({
        count: 2,
        delay: (error, retryCount) => {
          if (error instanceof HttpErrorResponse && error.status < 500) {
            return throwError(() => error); // Don't retry client errors
          }
          return timer(Math.pow(2, retryCount) * 1000);
        }
      }),
      catchError(err => {
        if (fallback !== undefined) {
          this.notification.showWarning('Using cached data');
          return of(fallback);
        }
        this.notification.showError(`Failed to load data from ${url}`);
        return throwError(() => err);
      })
    );
  }
}
```

### Form Submission with Error Recovery

```typescript
@Component({ ... })
export class OrderFormComponent {
  private http = inject(HttpClient);
  private notification = inject(NotificationService);

  submitting = signal(false);
  error = signal<string | null>(null);

  submitOrder(order: Order): void {
    this.submitting.set(true);
    this.error.set(null);

    this.http.post<OrderConfirmation>('/api/orders', order).pipe(
      timeout(15000),
      finalize(() => this.submitting.set(false))
    ).subscribe({
      next: confirmation => {
        this.notification.showSuccess(`Order ${confirmation.id} placed!`);
        this.router.navigate(['/orders', confirmation.id]);
      },
      error: err => {
        if (err.name === 'TimeoutError') {
          this.error.set('Request timed out. Please check your order status before resubmitting.');
        } else if (err.status === 409) {
          this.error.set('This order was already submitted.');
        } else if (err.status === 422) {
          this.error.set('Invalid order data: ' + err.error?.message);
        } else {
          this.error.set('An unexpected error occurred. Please try again.');
        }
      }
    });
  }
}
```

### WebSocket with Auto-Reconnection

```typescript
@Injectable({ providedIn: 'root' })
export class LiveDataService {
  private notification = inject(NotificationService);

  private socket$ = webSocket<ServerEvent>('wss://api.example.com/live');

  readonly events$ = this.socket$.pipe(
    tap({
      error: err => {
        console.error('WebSocket error:', err);
        this.notification.showWarning('Connection lost. Reconnecting...');
      }
    }),
    retry({
      count: Infinity,       // Always retry
      delay: (error, retryCount) => {
        // Cap backoff at 30 seconds
        const delay = Math.min(Math.pow(2, retryCount) * 1000, 30000);
        return timer(delay);
      },
      resetOnSuccess: true   // Reset counter when messages arrive
    }),
    share()                  // Share connection among subscribers
  );

  send(message: ClientEvent): void {
    this.socket$.next(message);
  }
}
```

---

> **Key Takeaway:** Always place `catchError` inside inner Observables (within `switchMap`, `mergeMap`, etc.) to keep outer pipelines alive. Use `retry` with exponential backoff for transient failures, centralize cross-cutting error handling in HTTP interceptors, and use `finalize` for cleanup that must run regardless of outcome.
