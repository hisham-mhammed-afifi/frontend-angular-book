# 1.1 Document Structure

> The HTML document structure is the skeleton every browser parses before a single pixel is painted. Getting it wrong triggers quirks mode, breaks accessibility, kills SEO, and tanks performance.

---

## The Minimal Valid Document

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Page Title</title>
</head>
<body>
  <!-- content -->
</body>
</html>
```

Five things are non-negotiable:

| Element | Why |
|---|---|
| `<!DOCTYPE html>` | Activates standards mode |
| `<html lang="...">` | WCAG Level A requirement; screen reader pronunciation |
| `<meta charset="utf-8">` | Must be within the first 1024 bytes |
| `<meta name="viewport">` | Without it, mobile browsers render at ~980px width |
| `<title>` | Required by the spec; one per document |

Every item above has a hard failure mode if omitted. The rest of this guide explains each one.

---

## DOCTYPE: Standards Mode vs. Quirks Mode

`<!DOCTYPE html>` is not an HTML element. It is a processing instruction that tells the browser which rendering mode to use.

### The Three Rendering Modes

| Mode | Trigger | Behavior |
|---|---|---|
| No-quirks (standards) | `<!DOCTYPE html>` | Follows modern HTML/CSS specs |
| Quirks | Missing or malformed doctype | Emulates IE5/Navigator 4 layout bugs |
| Limited-quirks (almost-standards) | Certain legacy doctypes (e.g., XHTML Transitional) | Standards mode with minor table cell line-height differences |

### What Actually Changes in Quirks Mode

Quirks mode is not just "old rendering." It changes concrete CSS behavior:

```css
/* In quirks mode, table cells with display: table-cell
   behave as if box-sizing: border-box for height properties */
.cell {
  height: 200px;
  padding: 20px;
  /* Quirks: 200px total including padding */
  /* Standards: 200px content + 20px padding on each side = 240px total */
}
```

```css
/* Quirks mode auto-corrects unitless values */
.box {
  margin: 10;
  /* Quirks: interpreted as margin: 10px */
  /* Standards: invalid, ignored entirely */
}
```

```css
/* Quirks mode auto-corrects bare hex colors */
.text {
  color: ff0000;
  /* Quirks: interpreted as #ff0000 */
  /* Standards: invalid, ignored entirely */
}
```

### Detecting the Mode Programmatically

```javascript
console.log(document.compatMode);
// "CSS1Compat" -> standards mode
// "BackCompat" -> quirks mode
```

### Rules for the DOCTYPE

- It must be the **very first thing** in the document. No whitespace, no comments, no BOM before it.
- It is case-insensitive: `<!DOCTYPE html>`, `<!doctype html>`, and `<!Doctype Html>` all work.
- Lighthouse flags missing doctypes as a performance and compatibility issue.

```html
<!-- BAD: comment before doctype can trigger quirks mode in some legacy browsers -->
<!-- This is my page -->
<!DOCTYPE html>

<!-- GOOD: doctype is the absolute first thing -->
<!DOCTYPE html>
<!-- This is my page -->
```

## The `<html>` Element

The root element wraps everything. Its most critical attribute is `lang`.

### Why `lang` Matters

Screen readers use `lang` to load the correct pronunciation engine. Without it, a screen reader configured for Spanish will attempt to read English text with Spanish phonetics. The result is unintelligible.

```html
<!-- Screen reader loads English pronunciation rules -->
<html lang="en">

<!-- American English specifically -->
<html lang="en-US">

<!-- Brazilian Portuguese -->
<html lang="pt-BR">
```

`lang` also affects:

| Consumer | How It Uses `lang` |
|---|---|
| Screen readers | Pronunciation and braille tables |
| Search engines | Language detection and regional ranking |
| Browser translation | Determines source language for auto-translate prompts |
| CSS `:lang()` | Language-specific styling (quotation marks, hyphenation) |
| Spell checkers | Dictionary selection |

### Multi-Language Content

Override `lang` on specific elements when the language changes:

```html
<html lang="en">
<body>
  <p>This paragraph is in English.</p>
  <blockquote lang="fr">
    <p>Ce paragraphe est en francais.</p>
  </blockquote>
  <p>Back to English.</p>
</body>
</html>
```

The `:lang()` CSS pseudo-class responds to these overrides:

```css
/* Different quotation marks per language */
:lang(en) > q { quotes: '\201C' '\201D' '\2018' '\2019'; } /* curly double, curly single */
:lang(fr) > q { quotes: '\00AB' '\00BB' '\201C' '\201D'; } /* guillemets, curly double */
```

### Tag Omission (Know It, Avoid It)

The HTML spec allows omitting the `<html>`, `<head>`, and `<body>` start/end tags under certain conditions. This is technically valid:

```html
<!DOCTYPE html>
<title>Valid</title>
<p>This works.
```

The browser infers the missing elements. **Do not rely on this.** Omitting tags makes it impossible to set `lang` on `<html>`, makes the document harder to read, and can produce unexpected DOM structures if content does not conform to content model rules.

## The `<head>` Element

The `<head>` contains metadata. Only these elements are permitted inside it: `<title>`, `<base>`, `<link>`, `<style>`, `<meta>`, `<script>`, `<noscript>`, and `<template>`.

### Element Order Matters

The order of elements in `<head>` has real performance and correctness implications. Use this order:

```html
<head>
  <!-- 1. Character encoding: MUST be in the first 1024 bytes -->
  <meta charset="utf-8">

  <!-- 2. Viewport: must come early for mobile rendering -->
  <meta name="viewport" content="width=device-width, initial-scale=1">

  <!-- 3. Title: required, exactly one -->
  <title>Page Title - Site Name</title>

  <!-- 4. Meta tags: description, robots, theme-color -->
  <meta name="description" content="A concise page description under 160 characters.">
  <meta name="theme-color" content="#1a1a2e">

  <!-- 5. Base (if needed): must come before any URL-containing elements -->
  <!-- <base href="https://example.com/"> -->

  <!-- 6. Preconnect: establish early connections to critical third-party origins -->
  <link rel="preconnect" href="https://fonts.googleapis.com">
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

  <!-- 7. CSS: render-blocking, order determines cascade precedence -->
  <link rel="stylesheet" href="/css/main.css">

  <!-- 8. Preload: critical resources the browser has not discovered yet -->
  <link rel="preload" href="/fonts/inter.woff2" as="font" type="font/woff2" crossorigin>

  <!-- 9. Scripts: defer or module to avoid blocking the parser -->
  <script src="/js/app.js" defer></script>

  <!-- 10. Everything else: Open Graph, icons, prefetch -->
  <meta property="og:title" content="Page Title">
  <link rel="icon" href="/favicon.ico">
  <link rel="prefetch" href="/next-page.html">
</head>
```

### Why Charset Must Be First

The HTML spec requires `<meta charset="utf-8">` to be **entirely within the first 1024 bytes** of the document. If the browser has not determined the encoding by that point, it may:

1. Default to a legacy encoding (like Windows-1252)
2. Need to re-parse the entire document after discovering the correct encoding
3. Display garbled characters (mojibake)

```html
<!-- BAD: charset comes after a large block of meta tags -->
<head>
  <title>My Page</title>
  <meta name="description" content="...">
  <meta name="keywords" content="...">
  <!-- ... 50 more meta tags ... -->
  <meta charset="utf-8"> <!-- too late, may be past 1024 bytes -->
</head>

<!-- GOOD: charset is the very first element in <head> -->
<head>
  <meta charset="utf-8">
  <title>My Page</title>
</head>
```

### Encoding Detection Priority

Browsers determine character encoding in this order:

| Priority | Source | Example |
|---|---|---|
| 1 (highest) | Byte Order Mark (BOM) | Leading bytes `EF BB BF` in the file |
| 2 | HTTP header | `Content-Type: text/html; charset=utf-8` |
| 3 | `<meta>` element | `<meta charset="utf-8">` |
| 4 (lowest) | Encoding sniffing | Browser heuristics (unreliable) |

A BOM overrides everything, including the HTTP header. The HTTP header overrides the meta tag. Always set both the HTTP header and the meta tag for defense in depth.

### The Two Valid Charset Syntaxes

```html
<!-- Short form (preferred) -->
<meta charset="utf-8">

<!-- Long form (legacy, still valid) -->
<meta http-equiv="Content-Type" content="text/html; charset=utf-8">
```

UTF-8 is the **only valid encoding** for HTML5 documents per the spec.

## The `<title>` Element

Required. Exactly one per document. Used by browser tabs, window lists, history entries, bookmarks, search engine results, and social media cards.

### SEO Implications

```html
<!-- BAD: generic, keyword-free -->
<title>Home</title>

<!-- BAD: keyword stuffed -->
<title>Buy Shoes Online Cheap Shoes Best Shoes Free Shipping Shoes</title>

<!-- GOOD: specific, keyword-rich, under 60 characters -->
<title>Running Shoes for Trail Racing - TrailGear</title>
```

| Rule | Why |
|---|---|
| Unique per page | Duplicate titles confuse search engines |
| Primary keyword near the start | Search engines weight early words more |
| Under ~60 characters | Google truncates beyond this in SERPs |
| Pattern: "Page - Site" or "Page \| Site" | Consistent branding without wasting keyword space |

## The Viewport Meta Tag

```html
<meta name="viewport" content="width=device-width, initial-scale=1">
```

Without this tag, mobile browsers render the page at a virtual viewport width of ~980px and then scale down to fit the screen. All media queries and responsive units break.

### What the Values Mean

| Property | Effect |
|---|---|
| `width=device-width` | Viewport width matches the device's CSS pixel width |
| `initial-scale=1` | No zoom on initial load |

### Anti-Pattern: Disabling Zoom

```html
<!-- BAD: violates WCAG 1.4.4 (Resize Text) -->
<meta name="viewport"
  content="width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no">

<!-- GOOD: allows user zooming -->
<meta name="viewport" content="width=device-width, initial-scale=1">
```

Users with low vision depend on zoom. Disabling it is an accessibility violation that Lighthouse also flags.

## The `<base>` Element

`<base>` sets the base URL for all relative URLs in the document. It is rarely needed and frequently causes bugs.

```html
<head>
  <base href="https://cdn.example.com/assets/">
</head>
<body>
  <!-- This resolves to https://cdn.example.com/assets/images/logo.png -->
  <img src="images/logo.png" alt="Logo">
</body>
```

### Critical Gotchas

**Fragment links break:**

```html
<base href="https://example.com/page/">

<!-- EXPECTED: scroll to #section on the current page -->
<!-- ACTUAL: navigates to https://example.com/page/#section (full navigation) -->
<a href="#section">Jump to section</a>
```

**Only the first `<base>` counts:**

```html
<!-- Only the first href and first target are used -->
<base href="https://a.com/">
<base href="https://b.com/"> <!-- ignored -->
```

**Must come before URL-containing elements:**

```html
<!-- BAD: base after link, the link resolved before base was parsed -->
<link rel="stylesheet" href="style.css">
<base href="https://cdn.example.com/">

<!-- GOOD: base before any URL-bearing elements -->
<base href="https://cdn.example.com/">
<link rel="stylesheet" href="style.css">
```

**Recommendation:** Avoid `<base>` unless you have a specific, well-understood use case. Use absolute URLs or a consistent path convention instead.

## How Browsers Parse HTML

Understanding HTML parsing explains why malformed markup "works" and why certain patterns produce surprising results.

### The Two-Stage Pipeline

1. **Tokenization**: Raw bytes are converted into tokens (start tags, end tags, character data, comments, DOCTYPE).
2. **Tree construction**: Tokens are consumed to build the DOM. Each token is processed immediately as it is emitted. The tree builder also **drives** the tokenizer: the tokenizer needs to know context (e.g., inside `<script>` vs. `<div>`) to tokenize correctly. You cannot tokenize HTML without a DOM.

### Error Recovery Is Spec-Defined

The HTML5 spec defines **exact recovery behavior** for every type of malformed HTML. This is not browser-specific guesswork. Three behaviors worth knowing:

**1. Automatic `<p>` closure:**

A `<p>` is auto-closed when the parser encounters certain elements:

```html
<!-- What you write -->
<p>First paragraph
<p>Second paragraph

<!-- What the DOM contains -->
<p>First paragraph</p>
<p>Second paragraph</p>
```

The full list of elements that close `<p>`: `<address>`, `<article>`, `<aside>`, `<blockquote>`, `<details>`, `<dialog>`, `<div>`, `<dl>`, `<fieldset>`, `<figcaption>`, `<figure>`, `<footer>`, `<form>`, `<h1>`-`<h6>`, `<header>`, `<hgroup>`, `<hr>`, `<main>`, `<menu>`, `<nav>`, `<ol>`, `<p>`, `<pre>`, `<search>`, `<section>`, `<table>`, `<ul>`.

**2. Foster parenting:**

Content placed directly inside `<table>` that is not valid table content gets moved **before** the table:

```html
<!-- What you write -->
<table>
  <p>Misplaced paragraph</p>
  <tr><td>Cell</td></tr>
</table>

<!-- What the DOM contains -->
<p>Misplaced paragraph</p>
<table>
  <tbody>
    <tr><td>Cell</td></tr>
  </tbody>
</table>
```

Note also the implicit `<tbody>`. The parser inserts it automatically.

**3. Block inside `<p>` causes splitting:**

```html
<!-- What you write -->
<p>Before <div>Inside</div> After</p>

<!-- What the DOM contains -->
<p>Before </p>
<div>Inside</div>
 After
<p></p>
```

The parser closes `<p>` when it sees `<div>` (because `<p>` only allows phrasing content). The "After" text and empty `<p>` are artifacts of the recovery process.

### Void Elements

These elements cannot have children and must not have closing tags:

`<area>`, `<base>`, `<br>`, `<col>`, `<embed>`, `<hr>`, `<img>`, `<input>`, `<link>`, `<meta>`, `<param>`, `<source>`, `<track>`, `<wbr>`

```html
<!-- VALID: no closing tag -->
<br>
<img src="photo.jpg" alt="Photo">
<input type="text" name="query">

<!-- ALSO VALID but meaningless: trailing slash is silently ignored -->
<br />
<img src="photo.jpg" alt="Photo" />

<!-- INVALID: void elements cannot have closing tags -->
<br></br>  <!-- parsed as TWO <br> elements -->
```

Self-closing syntax (`<br />`) is an XHTML artifact. In HTML, the trailing slash has no effect. The parser ignores it.

### The Preload Scanner

While the main parser is blocked by a synchronous `<script>`, a secondary parser (the **preload scanner**) continues scanning the raw HTML ahead of the main parser to discover resources it can fetch early.

```html
<head>
  <!-- Main parser blocks here while downloading and executing app.js -->
  <script src="/js/app.js"></script>

  <!-- Preload scanner finds these while main parser is blocked -->
  <link rel="stylesheet" href="/css/main.css">
  <script src="/js/utils.js"></script>
</head>
<body>
  <!-- Preload scanner also discovers this image -->
  <img src="/images/hero.jpg" alt="Hero">
</body>
```

**Key implication:** Resources injected by JavaScript (dynamic `<script>`, `<link>`, or `<img>` elements) cannot be discovered by the preload scanner. This is one reason why server-rendered HTML outperforms client-rendered HTML for initial load performance.

## Content Model Rules

HTML elements are organized into content categories that define what can contain what.

### The Main Categories

| Category | Description | Examples |
|---|---|---|
| Flow | Most elements usable in `<body>` | `<div>`, `<p>`, `<ul>`, `<table>`, `<h1>` |
| Phrasing | Inline/text-level elements | `<span>`, `<a>`, `<em>`, `<strong>`, `<code>`, `<img>` |
| Sectioning | Create sections in the document outline | `<article>`, `<aside>`, `<nav>`, `<section>` |
| Heading | Headings | `<h1>` through `<h6>`, `<hgroup>` |
| Embedded | External content | `<img>`, `<video>`, `<audio>`, `<iframe>`, `<canvas>` |
| Interactive | User-interactable elements | `<a>`, `<button>`, `<input>`, `<select>`, `<details>` |
| Metadata | Document metadata | `<link>`, `<meta>`, `<script>`, `<style>`, `<title>` |

All phrasing content is also flow content, but not vice versa.

### Rules That Catch People Off Guard

**`<p>` only accepts phrasing content:**

```html
<!-- INVALID: div is flow content, not phrasing content -->
<p><div>This breaks</div></p>

<!-- VALID -->
<p><span>This works</span></p>
<p><strong>This too</strong></p>
<p><a href="#">And this</a></p>
```

**`<a>` has a transparent content model:**

```html
<!-- <a> inside <p>: inherits <p>'s content model (phrasing only) -->
<p><a href="#"><em>Phrasing content is fine</em></a></p>

<!-- <a> inside <div>: inherits <div>'s content model (flow content) -->
<div>
  <a href="#">
    <h2>Heading inside a link</h2>
    <p>Paragraph inside a link</p>
  </a>
</div>
```

**Interactive content cannot nest:**

```html
<!-- INVALID: button inside a link -->
<a href="#"><button>Click</button></a>

<!-- INVALID: link inside a button -->
<button><a href="#">Click</a></button>
```

## Script Loading Strategies

How and where you load scripts directly impacts parsing, rendering, and time-to-interactive.

### The Four Modes

```html
<!-- 1. Classic: blocks parsing, blocks rendering -->
<script src="app.js"></script>

<!-- 2. defer: downloads in parallel, executes after parsing, preserves order -->
<script src="app.js" defer></script>

<!-- 3. async: downloads in parallel, executes ASAP (may interrupt parsing), no order guarantee -->
<script src="analytics.js" async></script>

<!-- 4. module: deferred by default, strict mode, evaluated once -->
<script type="module" src="app.js"></script>
```

### Comparison Table

| Attribute | Download | Execution | Order Guaranteed | Blocks Parser |
|---|---|---|---|---|
| (none) | Blocks | Immediate | Yes | Yes |
| `defer` | Parallel | After parsing | Yes | No |
| `async` | Parallel | When ready | No | Briefly, during execution |
| `type="module"` | Parallel | After parsing | Yes | No |

### When to Use Each

| Strategy | Use Case |
|---|---|
| `defer` | Main application scripts that depend on the DOM |
| `async` | Independent scripts (analytics, ads) that do not depend on DOM or other scripts |
| `type="module"` | Modern ES module code |
| No attribute | Almost never. Only if a script must execute at that exact parser position |

### `defer` Only Works on External Scripts

```html
<!-- defer works: external script -->
<script src="app.js" defer></script>

<!-- defer is IGNORED: inline script -->
<script defer>
  console.log('This runs immediately, blocking the parser');
</script>
```

### Document Lifecycle Events

```
Parser starts
  |
  v
readyState = "loading"
  |
  v
HTML fully parsed, deferred scripts executed
  |
  v
readyState = "interactive"
DOMContentLoaded fires
  |
  v
All sub-resources loaded (images, stylesheets, iframes)
  |
  v
readyState = "complete"
window.load fires
```

```javascript
// Runs when DOM is ready (after deferred scripts)
document.addEventListener('DOMContentLoaded', () => {
  console.log('DOM ready');
});

// Runs when everything is loaded
window.addEventListener('load', () => {
  console.log('All resources loaded');
});

// Check current state
console.log(document.readyState); // "loading", "interactive", or "complete"
```

## Resource Hints

Resource hints in `<head>` tell the browser to prepare connections or fetch resources before they are needed.

### `preconnect`: Early Connection Setup

```html
<!-- Establishes DNS + TCP + TLS connection early -->
<link rel="preconnect" href="https://fonts.googleapis.com">

<!-- CORS resources (like fonts) need the crossorigin attribute -->
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
```

Without `crossorigin` on CORS resources, the browser opens a **separate** connection and the preconnect is wasted.

Limit preconnect to 2-4 critical origins. Each connection costs memory and CPU.

### `dns-prefetch`: Lightweight Alternative

```html
<!-- Only resolves DNS (cheaper than preconnect) -->
<link rel="dns-prefetch" href="https://analytics.example.com">
```

Use for less-critical third-party domains. Often paired with preconnect as a fallback:

```html
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="dns-prefetch" href="https://fonts.googleapis.com">
```

### `preload`: Fetch Critical Resources Early

```html
<!-- Preload a font the browser has not discovered yet -->
<link rel="preload" href="/fonts/inter.woff2" as="font" type="font/woff2" crossorigin>

<!-- Preload a critical CSS file -->
<link rel="preload" href="/css/above-fold.css" as="style">

<!-- Preload the LCP image -->
<link rel="preload" href="/images/hero.webp" as="image">
```

The `as` attribute is required. It tells the browser the resource type so it can set priority correctly and apply the right Content Security Policy.

**Warning:** Chrome logs a console warning if you preload a resource but do not use it within ~3 seconds.

### `prefetch`: Future Navigation Resources

```html
<!-- Low-priority fetch for resources needed on the next page -->
<link rel="prefetch" href="/next-page.html">
<link rel="prefetch" href="/js/next-page-bundle.js">
```

### `fetchpriority`: Fine-Tune Priority

```html
<!-- Boost LCP image priority -->
<img src="hero.jpg" alt="Hero" fetchpriority="high">

<!-- Lower priority for below-the-fold images -->
<img src="footer-logo.jpg" alt="Footer Logo" fetchpriority="low">

<!-- Works on scripts too -->
<script src="critical.js" fetchpriority="high"></script>
```

## The Critical Rendering Path

The sequence from receiving HTML bytes to painting pixels:

```
HTML bytes
  -> Tokenize + Parse -> DOM
CSS bytes
  -> Parse -> CSSOM
                        DOM + CSSOM -> Render Tree -> Layout -> Paint
```

### What Blocks Rendering

| Resource | Blocks Parsing? | Blocks Rendering? |
|---|---|---|
| CSS (`<link rel="stylesheet">`) | No | Yes (render-blocking) |
| Synchronous `<script>` | Yes | Yes |
| `<script defer>` | No | No |
| `<script async>` | No | Briefly during execution |
| Images | No | No |

### CSS Blocks JavaScript

If a stylesheet `<link>` appears before a `<script>` in `<head>`, the script will not execute until the stylesheet is loaded. Reason: the script might query computed styles.

```html
<head>
  <!-- Browser must download and parse this first... -->
  <link rel="stylesheet" href="styles.css">

  <!-- ...before this script can execute, even though parsing could continue -->
  <script src="app.js"></script>
</head>
```

### Optimization Techniques

**Non-critical CSS with media queries:**

```html
<!-- Always loaded, but only render-blocking for screen -->
<link rel="stylesheet" href="print.css" media="print">

<!-- Only render-blocking at matching viewport widths -->
<link rel="stylesheet" href="desktop.css" media="(min-width: 1024px)">
```

**Inline critical CSS:**

```html
<head>
  <style>
    /* Above-the-fold styles inlined to avoid network round-trip */
    body { margin: 0; font-family: system-ui, sans-serif; }
    .hero { height: 100vh; display: grid; place-items: center; }
  </style>
  <!-- Non-critical CSS loaded without blocking render -->
  <link rel="preload" href="/css/main.css" as="style" onload="this.rel='stylesheet'">
</head>
```

**Avoid `document.write()`:**

```javascript
// BAD: forces the parser to restart, kills performance
document.write('<script src="analytics.js"><\/script>');

// GOOD: dynamic script insertion
const script = document.createElement('script');
script.src = 'analytics.js';
document.head.appendChild(script);
```

## HTML Attributes vs. DOM Properties

A frequent source of confusion, especially in frameworks.

### Core Difference

| | HTML Attributes | DOM Properties |
|---|---|---|
| Defined in | Markup | JavaScript object |
| Type | Always strings | Any type (boolean, number, object) |
| Access | `getAttribute()` / `setAttribute()` | `element.propertyName` |
| Reflects | Initial value from markup | Current live value |

### They Are Not Always 1:1

```javascript
const div = document.querySelector('div');

// Attribute name != property name
div.getAttribute('class');  // "my-class"
div.className;              // "my-class"  (property is className, not class)

// <label for="email">
label.getAttribute('for');  // "email"
label.htmlFor;              // "email"  (property is htmlFor, not for)
```

### Attributes Reflect Initial State, Properties Reflect Current State

```html
<input type="checkbox" checked>
```

```javascript
const checkbox = document.querySelector('input');

// Before user interaction:
checkbox.getAttribute('checked');  // "" (attribute exists, value is empty string)
checkbox.checked;                  // true (DOM property)

// User unchecks the checkbox:
checkbox.getAttribute('checked');  // "" (STILL the same -- attribute never changes)
checkbox.checked;                  // false (property reflects current state)
```

```html
<input type="text" value="initial">
```

```javascript
const input = document.querySelector('input');

// User types "changed" into the input:
input.getAttribute('value');  // "initial" (attribute is frozen)
input.value;                  // "changed" (property is live)
input.defaultValue;           // "initial" (maps to the attribute)
```

### Non-Standard Attributes

```html
<div data-id="42" custom-attr="test"></div>
```

```javascript
const div = document.querySelector('div');

div.getAttribute('data-id');     // "42"
div.dataset.id;                  // "42" (data-* attributes have a dedicated API)

div.getAttribute('custom-attr'); // "test"
div.customAttr;                  // undefined (non-standard attributes do not create properties)
```

## Common Mistakes

1. **Missing DOCTYPE.** No visible error, but layout and CSS behavior silently change. Box model calculations, inline element spacing, and unitless value handling all break.

2. **Charset declared too late.** If `<meta charset="utf-8">` is not within the first 1024 bytes, the browser may re-parse the document or display garbled characters. Always place it as the first element in `<head>`.

3. **Missing `lang` on `<html>`.** WCAG Level A violation. Screen readers use the wrong pronunciation engine. Search engines lose a language signal. Browser translation prompts break.

4. **Disabling zoom in the viewport meta.** `user-scalable=no` and `maximum-scale=1` prevent visually impaired users from zooming. This violates WCAG 1.4.4.

5. **Block elements inside `<p>`.** Writing `<p><div>text</div></p>` produces a split DOM with empty `<p>` tags. The parser auto-closes `<p>` when it encounters block-level elements.

6. **Relying on tag omission.** The spec allows omitting `<html>`, `<head>`, and `<body>` tags, but this prevents setting `lang` and makes the document fragile. Always write explicit tags.

7. **Preconnect without `crossorigin` for CORS resources.** Fonts served cross-origin need `crossorigin` on the preconnect link, or the browser opens a second connection and the hint is wasted.

8. **Using `<base>` without understanding fragment link behavior.** All `#anchor` links resolve against the base URL, causing full page navigations instead of scrolling.

9. **Synchronous scripts in `<head>` without `defer`.** Blocks both HTML parsing and rendering. Use `defer` for application scripts, `async` for independent scripts.

10. **Multiple `<h1>` elements.** The HTML5 outline algorithm that was supposed to support multiple `<h1>` per section was **never implemented** by any browser or assistive technology. Use one `<h1>` per page.

---

## Best Practices

- **Always start with the five non-negotiables:** DOCTYPE, `<html lang>`, charset, viewport, title.
- **Place `<meta charset="utf-8">` as the first element in `<head>`**, before everything else.
- **Use `defer` on all application scripts.** There is almost no reason to use parser-blocking scripts.
- **Limit preconnect hints to 2-4 critical origins.** More than that wastes connections.
- **Preload only late-discovered critical resources.** Fonts referenced in CSS are a prime candidate. Do not preload everything.
- **Set both the HTTP `Content-Type` header and the `<meta charset>` tag.** Defense in depth against encoding issues.
- **Validate your HTML.** The W3C validator catches content model violations, missing required elements, and accessibility issues that browsers silently recover from.
- **Use explicit closing tags even where the spec allows omission.** Clarity over cleverness.

## Real-World Use Cases

### 1. Production-Ready Document Shell

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Dashboard - Acme Corp</title>
  <meta name="description" content="Real-time analytics dashboard for Acme Corp.">
  <meta name="theme-color" content="#0f172a">

  <!-- Preconnect to API and CDN -->
  <link rel="preconnect" href="https://api.acme.com">
  <link rel="preconnect" href="https://cdn.acme.com" crossorigin>

  <!-- Critical CSS inlined -->
  <style>
    *, *::before, *::after { box-sizing: border-box; margin: 0; }
    body { font-family: system-ui, -apple-system, sans-serif; }
    .app-shell { display: grid; grid-template-rows: auto 1fr auto; min-height: 100vh; }
  </style>

  <!-- Main stylesheet -->
  <link rel="stylesheet" href="/css/main.css">

  <!-- Preload late-discovered font -->
  <link rel="preload" href="/fonts/inter-var.woff2" as="font" type="font/woff2" crossorigin>

  <!-- Application script -->
  <script src="/js/app.js" type="module"></script>

  <!-- Open Graph -->
  <meta property="og:title" content="Dashboard - Acme Corp">
  <meta property="og:type" content="website">
  <meta property="og:image" content="https://acme.com/og-dashboard.png">

  <!-- Favicon -->
  <link rel="icon" href="/favicon.svg" type="image/svg+xml">
</head>
<body>
  <div class="app-shell">
    <header><!-- nav --></header>
    <main><!-- content --></main>
    <footer><!-- footer --></footer>
  </div>
</body>
</html>
```

### 2. Multi-Language Document

### 3. Performance-Optimized Landing Page

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Fast Landing Page</title>

  <!-- Preconnect to image CDN -->
  <link rel="preconnect" href="https://images.example.com" crossorigin>

  <!-- Inline critical CSS for instant first paint -->
  <style>
    body { margin: 0; }
    .hero { height: 100vh; display: flex; align-items: center; justify-content: center; }
    .hero img { width: 100%; height: 100%; object-fit: cover; }
  </style>

  <!-- Preload the LCP image -->
  <link rel="preload" href="https://images.example.com/hero.webp" as="image"
        fetchpriority="high">

  <!-- Non-critical CSS loaded asynchronously -->
  <link rel="preload" href="/css/below-fold.css" as="style"
        onload="this.rel='stylesheet'">
  <noscript><link rel="stylesheet" href="/css/below-fold.css"></noscript>

  <!-- Analytics: async because it is independent -->
  <script src="/js/analytics.js" async></script>

  <!-- App code: deferred to not block parsing -->
  <script src="/js/app.js" defer></script>

  <!-- Prefetch the next likely navigation -->
  <link rel="prefetch" href="/pricing">
</head>
<body>
  <section class="hero">
    <img src="https://images.example.com/hero.webp"
         alt="Product showcase"
         fetchpriority="high"
         width="1920"
         height="1080">
  </section>
  <!-- Below-the-fold content -->
</body>
</html>
```

---

> **Key Takeaway:** The document structure is not boilerplate to copy-paste. Every element in `<head>` has performance, accessibility, or correctness implications. Get the order right, never skip the five non-negotiables, and understand how the parser recovers from your mistakes so you stop making them.
