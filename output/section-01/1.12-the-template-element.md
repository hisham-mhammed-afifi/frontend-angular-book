# 1.12 The Template Element

> The `<template>` element holds HTML fragments that the browser parses but does not render, providing a performant and safe mechanism for stamping out reusable DOM structures with JavaScript.

---

## How `<template>` Works

The `<template>` element is unique in HTML. Its contents are parsed by the browser into a DOM tree, but that tree is completely inert: scripts do not execute, images do not load, styles are not applied, audio does not play, and nothing is rendered on screen.

```html
<template id="greeting-tpl">
  <div class="greeting">
    <h2>Hello, <span class="name"></span>!</h2>
    <p>Welcome to the site.</p>
  </div>
</template>

<!-- Nothing visible appears on the page from the above -->
```

The template's content exists in a separate, inert document fragment. You access it through the `.content` property, clone it, populate it with data, and insert it into the live DOM.

---

## The `content` Property

`HTMLTemplateElement.content` is a read-only property that returns a `DocumentFragment` containing the template's DOM subtree.

```javascript
const template = document.getElementById('greeting-tpl');

// The template element itself has no child nodes
console.log(template.childNodes.length);         // 0
console.log(template.children.length);            // 0

// Content lives in the DocumentFragment
console.log(template.content.childNodes.length);  // 1+ (the div and whitespace text nodes)
console.log(template.content.querySelector('.name')); // <span class="name">
```

### Key Behavior: Content Belongs to a Separate Document

The `content` fragment's `ownerDocument` is a separate, inert Document, not the page's main document. This matters for custom elements (covered later) and means that queries on the main document will not find elements inside templates.

```javascript
// WRONG: returns null (content is in a separate document)
document.getElementById('something-inside-template');
document.querySelector('.name');  // if .name only exists inside the template

// CORRECT: query through the content property
template.content.querySelector('.name');
```

---

## Cloning Template Content

You never insert `template.content` directly. Doing so moves the content out of the template permanently, leaving it empty for future use. Always clone first.

### The Critical Mistake: Direct Insertion

```javascript
const template = document.getElementById('greeting-tpl');
const container = document.getElementById('output');

// BAD: empties the template permanently
container.appendChild(template.content);
// template.content is now empty, cannot be reused

// GOOD: clone first, template stays intact
container.appendChild(template.content.cloneNode(true));
// template.content is unchanged, can clone again
```

This happens because `DocumentFragment` moves its children (not copies) when passed to `appendChild()`. After insertion, the fragment is empty.

### `cloneNode(true)` vs `document.importNode()`

There are two ways to clone template content:

```javascript
// Approach 1: cloneNode
const clone = template.content.cloneNode(true);

// Approach 2: importNode
const clone = document.importNode(template.content, true);
```

The `true` argument means "deep clone" (include all descendants).

| Aspect | `cloneNode(true)` | `document.importNode(content, true)` |
|---|---|---|
| Document context | Clone belongs to template's inert document | Clone belongs to the active document |
| Custom elements | Constructors not called in current registry | Constructors are called in current registry |
| Standard HTML elements | Identical result | Identical result |
| Event listeners | Not copied | Not copied |
| Performance | Marginally faster | Marginally slower |

For templates containing only standard HTML elements, both produce the same practical result. When your templates contain custom elements, use `document.importNode()` to ensure custom element constructors fire correctly in the current document's registry.

**Best practice:** Default to `document.importNode(template.content, true)`. It is technically more correct and handles all cases.

### Populating the Clone

After cloning, modify the clone's content before inserting it into the live DOM:

```javascript
const template = document.getElementById('greeting-tpl');
const clone = document.importNode(template.content, true);

// Populate with data
clone.querySelector('.name').textContent = 'Alice';

// Insert into the live DOM
document.getElementById('output').appendChild(clone);
```

## Performance: Template vs innerHTML vs createElement

| Approach | Parse Cost | Repeated Use | Security |
|---|---|---|---|
| `innerHTML` | Re-parses HTML string every time | Slow for repeated inserts | XSS-vulnerable with user input |
| `createElement` + manual DOM | No parsing | Moderate (verbose code) | Safe by default |
| `<template>` + `cloneNode` | Parsed once at page load | Fast for repeated cloning | Inert content, safe |

### Why `<template>` Wins for Repeated Structures

`innerHTML` causes the browser to parse an HTML string, construct a DOM tree, and insert it every single time it is assigned. With `<template>`, the HTML is parsed once when the page loads. Each subsequent `cloneNode(true)` call is a fast binary copy of the already-parsed tree, avoiding the parsing step entirely.

```javascript
// SLOW: parses HTML string 1000 times
for (let i = 0; i < 1000; i++) {
  tbody.innerHTML += `<tr><td>${i}</td><td>Name ${i}</td></tr>`;
  // Even worse: also re-parses ALL existing rows each iteration
}

// FAST: parses once, clones 1000 times
const template = document.getElementById('row-tpl');
const fragment = document.createDocumentFragment();

for (let i = 0; i < 1000; i++) {
  const clone = document.importNode(template.content, true);
  clone.querySelector('.cell-id').textContent = i;
  clone.querySelector('.cell-name').textContent = `Name ${i}`;
  fragment.appendChild(clone);
}

tbody.appendChild(fragment); // single DOM insertion
```

The second approach also batches all insertions into a single `appendChild` call via an intermediate `DocumentFragment`, avoiding layout recalculation on every iteration.

### When `<template>` Is Not Worth It

For one-off content that will never be reused, `<template>` adds complexity without performance benefit. In those cases, `createElement` or even `innerHTML` (with proper sanitization) is fine.

## Event Listeners Are Not Cloned

`cloneNode()` and `importNode()` do not copy event listeners added via `addEventListener()`. Only inline event handler attributes (like `onclick="..."`) are preserved in the clone, but inline handlers are a bad practice.

```html
<template id="btn-tpl">
  <button class="action-btn">Click me</button>
</template>
```

```javascript
const template = document.getElementById('btn-tpl');
const clone = document.importNode(template.content, true);

// Attach listener to the clone, not the template
clone.querySelector('.action-btn').addEventListener('click', () => {
  console.log('Button clicked');
});

document.body.appendChild(clone);
```

For templates that produce many cloned elements, use event delegation on a parent container instead of attaching individual listeners to each clone:

```javascript
document.getElementById('button-container').addEventListener('click', (e) => {
  const btn = e.target.closest('.action-btn');
  if (!btn) return;
  console.log('Button clicked:', btn.textContent);
});
```

## Nested Templates

Cloning an outer template does not activate inner templates. Each nested `<template>` must be cloned separately.

```html
<template id="outer">
  <div class="section">
    <h2 class="title"></h2>
    <template id="inner-item">
      <li class="item"></li>
    </template>
    <ul class="list"></ul>
  </div>
</template>
```

```javascript
const outerTemplate = document.getElementById('outer');
const outerClone = document.importNode(outerTemplate.content, true);

outerClone.querySelector('.title').textContent = 'My List';

// The inner template must be accessed from the clone, not the original
const innerTemplate = outerClone.querySelector('#inner-item');
const list = outerClone.querySelector('.list');

['Apple', 'Banana', 'Cherry'].forEach((fruit) => {
  const itemClone = document.importNode(innerTemplate.content, true);
  itemClone.querySelector('.item').textContent = fruit;
  list.appendChild(itemClone);
});

// Remove the inner template element before inserting into the DOM
innerTemplate.remove();

document.body.appendChild(outerClone);
```

## The `<slot>` Element

`<slot>` is a companion element used inside Shadow DOM to define insertion points where light DOM content (children of the host element) should be projected.

### Default Slot

A `<slot>` without a `name` attribute captures all light DOM children that are not assigned to a named slot.

```html
<template id="card-tpl">
  <style>
    .card { border: 1px solid #ccc; padding: 16px; }
  </style>
  <div class="card">
    <slot></slot>  <!-- default slot: captures unassigned children -->
  </div>
</template>
```

### Named Slots

Named slots capture light DOM children with a matching `slot` attribute.

```html
<template id="card-tpl">
  <div class="card">
    <header>
      <slot name="title">Default Title</slot>
    </header>
    <div class="body">
      <slot></slot>  <!-- default slot -->
    </div>
    <footer>
      <slot name="actions"></slot>
    </footer>
  </div>
</template>
```

```html
<!-- Usage with a custom element that uses the template above -->
<my-card>
  <h2 slot="title">Product Name</h2>
  <p>Product description goes here.</p>
  <div slot="actions">
    <button>Buy</button>
    <button>Save</button>
  </div>
</my-card>
```

The `<h2>` projects into the "title" slot, the `<div>` with buttons projects into the "actions" slot, and the `<p>` (with no `slot` attribute) projects into the default slot.

### Slot Fallback Content

Content inside a `<slot>` tag is displayed only when no light DOM content is assigned to that slot:

```html
<slot name="title">Untitled</slot>
<!-- Shows "Untitled" only if no element has slot="title" -->
```

Slots do not move DOM nodes. They project them: the light DOM children remain in the light DOM tree but are rendered at the slot's position in the shadow DOM.

## Declarative Shadow DOM

Declarative Shadow DOM allows creating shadow roots directly in HTML, without JavaScript. A `<template>` element with the `shadowrootmode` attribute is detected by the HTML parser and immediately applied as the shadow root of its parent.

```html
<my-component>
  <template shadowrootmode="open">
    <style>
      p { color: navy; font-weight: bold; }
    </style>
    <p>This is shadow DOM content.</p>
    <slot></slot>
  </template>
  <span>This is light DOM content, projected via slot.</span>
</my-component>
```

### Key Behaviors

- The `<template>` element is **removed** from the DOM after parsing. It is replaced by a `ShadowRoot` attached to the parent.
- Works even with JavaScript disabled.
- Accepts `"open"` or `"closed"` as values. Any other value causes it to be treated as a normal template.

### Additional Attributes

| Attribute | Purpose |
|---|---|
| `shadowrootmode` | `"open"` or `"closed"`, creates the shadow root |
| `shadowrootclonable` | Makes the shadow root clonable via `cloneNode()` |
| `shadowrootdelegatesfocus` | Delegates focus to the first focusable element in the shadow tree |
| `shadowrootserializable` | Allows `Element.getHTML()` to serialize the shadow root |

### Browser Support

All three major engines support `shadowrootmode` as of August 2024 (Chrome 111+, Firefox 123+, Safari 16.4+).

### Server-Side Rendering

Declarative Shadow DOM enables SSR of web components. The shadow DOM structure is expressed as HTML that can be sent from the server and parsed by the browser without client-side JavaScript.

```html
<!-- Server sends this HTML -->
<search-box>
  <template shadowrootmode="open">
    <style>
      input { border: 2px solid #3498db; padding: 8px; border-radius: 4px; }
    </style>
    <input type="search" placeholder="Search...">
    <slot name="button"></slot>
  </template>
  <button slot="button">Go</button>
</search-box>
<!-- Browser creates shadow DOM during HTML parsing, no JS needed -->
```

## Feature Detection

```javascript
if ('content' in document.createElement('template')) {
  // Browser supports <template>
} else {
  // Fallback: use innerHTML or other approach
}
```

For Declarative Shadow DOM:

```javascript
function supportsDeclarativeShadowDOM() {
  const tpl = document.createElement('template');
  return 'shadowRootMode' in tpl;
}
```

All modern browsers support `<template>` (since 2013+). Feature detection is only needed if you must support very old browsers.

## Avoiding Duplicate IDs

Templates that contain `id` attributes produce duplicate IDs when cloned multiple times. This violates the HTML spec and breaks `getElementById()`.

```html
<!-- BAD: id inside a reusable template -->
<template id="user-tpl">
  <div id="user-card">  <!-- duplicate IDs when cloned multiple times -->
    <span id="user-name"></span>
  </div>
</template>
```

```html
<!-- GOOD: use classes or data attributes inside templates -->
<template id="user-tpl">
  <div class="user-card">
    <span class="user-name"></span>
  </div>
</template>
```

If you absolutely need unique IDs (for `<label for="">` or ARIA references), generate them dynamically:

```javascript
let idCounter = 0;

function cloneWithUniqueIds(template) {
  const clone = document.importNode(template.content, true);
  const id = `user-${++idCounter}`;

  const input = clone.querySelector('.user-input');
  const label = clone.querySelector('.user-label');

  input.id = id;
  label.setAttribute('for', id);

  return clone;
}
```

## Common Mistakes

- **Directly appending `template.content` without cloning.** This empties the template permanently. Always use `template.content.cloneNode(true)` or `document.importNode(template.content, true)`.

- **Querying the main document for elements inside templates.** `document.querySelector()` does not search inside template content. Use `template.content.querySelector()` instead.

- **Expecting nested templates to activate automatically.** Cloning an outer template does not clone inner templates. Each must be activated separately.

- **Forgetting that event listeners are not cloned.** `cloneNode` and `importNode` do not copy listeners added with `addEventListener()`. Attach listeners after cloning, or use event delegation.

- **Using `id` attributes inside templates that will be cloned multiple times.** This produces duplicate IDs. Use classes or data attributes instead.

- **Assuming template content is rendered.** `<template>` content is invisible and inert. It will not appear on screen until cloned and inserted into the live DOM.

- **Using `cloneNode` instead of `importNode` with custom elements.** `cloneNode` does not trigger custom element constructors in the current document. Use `document.importNode()` when templates contain custom elements.

---

## Best Practices

- **Always clone, never directly append** `template.content`. Use `document.importNode(template.content, true)` as the default.

- **Use `<template>` for repeated DOM structures.** Lists, table rows, cards, notifications. The one-time parse cost amortizes across many clones.

- **Prefer `<template>` over `innerHTML` for dynamic content.** It avoids reparsing HTML strings and eliminates XSS risk from the template content itself.

- **Batch insertions with `DocumentFragment`.** When cloning many templates, append clones to a fragment first, then append the fragment to the DOM once. This avoids triggering layout recalculation on every insertion.

- **Use semantic HTML inside templates.** When cloned into the live DOM, semantic elements provide meaning to assistive technology.

- **Use event delegation for cloned elements.** One listener on a parent is more maintainable and performant than attaching listeners to each clone.

- **Use Declarative Shadow DOM for server-rendered components.** The `shadowrootmode` attribute enables SSR without client-side JavaScript.

## Real-World Use Cases

### Dynamic Table with Batch Rendering

```html
<template id="row-tpl">
  <tr>
    <td class="cell-name"></td>
    <td class="cell-email"></td>
    <td class="cell-role"></td>
    <td>
      <button class="edit-btn" data-action="edit">Edit</button>
      <button class="delete-btn" data-action="delete">Delete</button>
    </td>
  </tr>
</template>

<table>
  <thead>
    <tr><th>Name</th><th>Email</th><th>Role</th><th>Actions</th></tr>
  </thead>
  <tbody id="user-table-body"></tbody>
</table>
```

```javascript
const template = document.getElementById('row-tpl');
const tbody = document.getElementById('user-table-body');
const fragment = document.createDocumentFragment();

users.forEach((user) => {
  const clone = document.importNode(template.content, true);
  clone.querySelector('.cell-name').textContent = user.name;
  clone.querySelector('.cell-email').textContent = user.email;
  clone.querySelector('.cell-role').textContent = user.role;

  // Store user ID for event delegation
  clone.querySelector('tr').dataset.userId = user.id;

  fragment.appendChild(clone);
});

tbody.appendChild(fragment); // single DOM operation

// Event delegation for all action buttons
tbody.addEventListener('click', (e) => {
  const btn = e.target.closest('[data-action]');
  if (!btn) return;

  const row = btn.closest('tr');
  const userId = row.dataset.userId;
  const action = btn.dataset.action;

  if (action === 'edit') editUser(userId);
  if (action === 'delete') deleteUser(userId);
});
```

### Notification System

```html
<template id="notification-tpl">
  <div class="notification" role="alert">
    <span class="notification-message"></span>
    <button class="notification-close" aria-label="Dismiss">&times;</button>
  </div>
</template>

<div id="notifications" aria-live="polite"></div>
```

```javascript
const template = document.getElementById('notification-tpl');
const container = document.getElementById('notifications');

function showNotification(message, type = 'info', duration = 5000) {
  const clone = document.importNode(template.content, true);
  const el = clone.querySelector('.notification');

  el.dataset.type = type; // used for CSS styling: [data-type="error"], etc.
  clone.querySelector('.notification-message').textContent = message;

  clone.querySelector('.notification-close').addEventListener('click', () => {
    el.remove();
  });

  container.appendChild(clone);

  // Auto-dismiss
  if (duration > 0) {
    setTimeout(() => el.remove(), duration);
  }
}

// Usage
showNotification('Settings saved successfully.', 'success');
showNotification('Failed to connect to server.', 'error', 0); // no auto-dismiss
```

### Declarative Shadow DOM Component

```html
<!-- Works without any JavaScript -->
<alert-banner>
  <template shadowrootmode="open">
    <style>
      :host {
        display: block;
        padding: 12px 16px;
        border-radius: 4px;
        font-family: system-ui, sans-serif;
      }
      :host([type="warning"]) {
        background: #fff3cd;
        border: 1px solid #ffc107;
        color: #856404;
      }
      :host([type="error"]) {
        background: #f8d7da;
        border: 1px solid #f5c6cb;
        color: #721c24;
      }
      :host([type="success"]) {
        background: #d4edda;
        border: 1px solid #c3e6cb;
        color: #155724;
      }
      ::slotted(strong) {
        display: block;
        margin-bottom: 4px;
      }

</alert-banner>

<!-- Usage -->
<alert-banner type="warning">
  <strong slot="title">Warning</strong>

</alert-banner>

<alert-banner type="success">
  <strong slot="title">Success</strong>
  Your changes have been saved.
</alert-banner>
```

---

> **Key Takeaway:** The `<template>` element gives you a browser-parsed, inert DOM fragment that can be cloned efficiently for repeated use. Always clone (never directly append), use `document.importNode()` for safety with custom elements, and leverage event delegation for cloned interactive content.
