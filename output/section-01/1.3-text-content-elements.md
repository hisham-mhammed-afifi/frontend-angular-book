# 1.3 Text Content Elements

> Text content elements organize blocks of content between the opening and closing `<body>` tags. They define paragraphs, headings, lists, quotations, preformatted text, and thematic breaks, giving structure to the actual content of a page.

---

## The `<p>` Element

The paragraph element groups related sentences into a single block of text. It is the most common text content element and the one most frequently misused.

### Content Model

`<p>` accepts **phrasing content only**. That means text and inline elements: `<em>`, `<strong>`, `<a>`, `<span>`, `<code>`, `<img>`, `<br>`, `<input>`, `<abbr>`, `<time>`, `<mark>`, `<sub>`, `<sup>`, and similar.

It cannot contain block-level elements: no `<div>`, `<h1>`-`<h6>`, `<ul>`, `<ol>`, `<blockquote>`, `<table>`, `<pre>`, `<figure>`, `<section>`, or another `<p>`.

```html
<!-- VALID: phrasing content only -->
<p>This paragraph has <strong>bold text</strong>, an <a href="#">inline link</a>,
   and an <img src="icon.svg" alt="icon" width="16" height="16"> inline image.</p>

<!-- VALID: br is phrasing content -->
<p>Line one<br>Line two</p>
```

### The Parser Splitting Gotcha

When you place a block element inside a `<p>`, the browser does not nest it. The parser auto-closes the `<p>`, renders the block element, then creates stray text nodes and possibly an empty `<p>`:

```html
<!-- What you write -->
<p>Before <div>Block content</div> After</p>

<!-- What the DOM actually contains -->
<p>Before </p>
<div>Block content</div>
 After
<p></p>
```

This is not a browser bug. The HTML spec defines this exact recovery behavior. The list of elements that trigger auto-closure of `<p>`:

`address`, `article`, `aside`, `blockquote`, `details`, `dialog`, `div`, `dl`, `fieldset`, `figcaption`, `figure`, `footer`, `form`, `h1`-`h6`, `header`, `hgroup`, `hr`, `main`, `menu`, `nav`, `ol`, `p`, `pre`, `search`, `section`, `table`, `ul`

**This matters in practice.** If a CMS, template engine, or framework injects a `<div>` inside a `<p>`, the DOM structure silently breaks. Styles targeting `p > *` or JavaScript querying `p` children will not find what you expect.

```html
<!-- BAD: common CMS mistake -->
<p>Some introductory text <div class="embed">Embedded widget</div> continues here.</p>

<!-- GOOD: restructure to avoid the nesting -->
<p>Some introductory text</p>
<div class="embed">Embedded widget</div>
<p>Continues here.</p>
```

## Heading Elements (`<h1>` through `<h6>`)

Headings define a six-level hierarchy for document sections. They are the primary navigation mechanism for screen reader users.

### Content Model

Headings accept phrasing content only. You can put inline elements inside headings but not block elements.

```html
<!-- VALID -->
<h2>Understanding <code>Array.prototype.reduce()</code></h2>

<!-- VALID -->
<h2><a href="/guide">Complete Guide</a></h2>

<!-- INVALID: div inside heading -->
<h2><div>Section Title</div></h2>
```

### Implicit ARIA Role

Every heading carries an implicit `heading` role with an `aria-level` matching the number:

| Element | ARIA Role | aria-level |
|---|---|---|
| `<h1>` | `heading` | `1` |
| `<h2>` | `heading` | `2` |
| `<h3>` | `heading` | `3` |
| `<h4>` | `heading` | `4` |
| `<h5>` | `heading` | `5` |
| `<h6>` | `heading` | `6` |

### Hierarchy Rules

Screen reader users rely on heading levels to understand page structure. Nearly 70% of screen reader users navigate pages primarily by headings (WebAIM survey). Violating the hierarchy makes content harder to understand.

```html
<!-- CORRECT: no skipped levels going down -->
<h1>JavaScript Deep Dive</h1>
  <h2>Variables and Scope</h2>
    <h3>var vs let vs const</h3>
    <h3>Block Scope vs Function Scope</h3>
  <h2>Functions</h2>
    <h3>Arrow Functions</h3>
      <h4>Implicit Return</h4>

<!-- BAD: skipped from h1 to h3 -->
<h1>JavaScript Deep Dive</h1>
  <h3>Variables and Scope</h3>  <!-- Where is h2? Screen reader users think they missed content -->

<!-- ACCEPTABLE: skipping levels going back UP -->
<h1>Page Title</h1>
  <h2>Section A</h2>
    <h3>Subsection A.1</h3>
      <h4>Detail A.1.1</h4>
  <h2>Section B</h2>  <!-- Jumping from h4 back to h2 is fine: closing subsections -->
```

### One `<h1>` Per Page

The HTML5 document outline algorithm was supposed to allow multiple `<h1>` elements (one per sectioning element). No browser or assistive technology ever implemented it. The algorithm was removed from the spec in July 2022.

```html
<!-- BAD: multiple h1 elements -->
<h1>Site Name</h1>
<article>
  <h1>Article Title</h1>  <!-- Screen readers announce this as another level 1 heading -->
</article>

<!-- GOOD: single h1, proper hierarchy -->
<h1>Article Title</h1>
<article>
  <h2>First Section</h2>
  <h2>Second Section</h2>
</article>
```

### Do Not Use Headings for Visual Sizing

```html
<!-- BAD: h4 chosen for its smaller font size -->
<h4>This is not really a subsection, I just wanted small text</h4>

<!-- GOOD: use the correct heading level, style with CSS -->
<h2 class="text-sm">This is a real subsection</h2>
```

```css
.text-sm {
  font-size: 0.875rem;
  font-weight: 600;
}
```

## Lists: `<ul>`, `<ol>`, `<li>`

Lists group related items. Screen readers announce the list type and item count, giving users a mental model of the content before they read it.

### Unordered Lists (`<ul>`)

Use when item order does not matter. A shopping list, a set of features, a group of links.

```html
<h2>Required Tools</h2>
<ul>
  <li>Node.js 18+</li>
  <li>npm or yarn</li>
  <li>A code editor</li>
</ul>
```

Screen readers announce: "List, 3 items." Then for each item: "Bullet, Node.js 18 plus."

### Ordered Lists (`<ol>`)

Use when the **sequence is meaningful**: steps in a process, rankings, legal clauses, instructions.

```html
<h2>Setup Steps</h2>
<ol>
  <li>Clone the repository</li>
  <li>Install dependencies</li>
  <li>Copy the environment file</li>
  <li>Start the development server</li>
</ol>
```

An alphabetically sorted list is NOT necessarily ordered. Alphabetical order is a convenience, not a meaningful sequence. Use `<ul>` for alphabetical lists unless the position carries meaning (like a ranked list).

### `<ol>` Attributes

| Attribute | Type | Description |
|---|---|---|
| `type` | `1`, `a`, `A`, `i`, `I` | Numbering style: decimal, lowercase alpha, uppercase alpha, lowercase roman, uppercase roman |
| `start` | Integer | Starting number. Always Arabic numerals, even when type is letters or roman |
| `reversed` | Boolean | Counts down from the start value |

```html
<!-- Lowercase letters starting at "d" -->
<ol type="a" start="4">
  <li>Fourth item (d)</li>
  <li>Fifth item (e)</li>
</ol>

<!-- Reversed countdown -->
<ol reversed>
  <li>Bronze</li>  <!-- 3 -->
  <li>Silver</li>  <!-- 2 -->
  <li>Gold</li>    <!-- 1 -->
</ol>

<!-- Reversed with explicit start -->
<ol reversed start="5">
  <li>Fifth</li>   <!-- 5 -->
  <li>Fourth</li>  <!-- 4 -->
  <li>Third</li>   <!-- 3 -->
  <li>Second</li>  <!-- 2 -->
  <li>First</li>   <!-- 1 -->
</ol>
```

**Edge case:** If a reversed list has more items than the start value, numbering goes to zero and into negatives.

```html
<ol reversed start="2">
  <li>Item (2)</li>
  <li>Item (1)</li>
  <li>Item (0)</li>
  <li>Item (-1)</li>
</ol>
```

### The `value` Attribute on `<li>`

Inside an `<ol>`, the `value` attribute on `<li>` resets the counter at that point. Subsequent items continue from the new value.

```html
<ol>
  <li>Item 1</li>
  <li>Item 2</li>
  <li value="10">Item 10</li>
  <li>Item 11</li>  <!-- continues from 10 -->
  <li>Item 12</li>
</ol>
```

The `value` must always be an integer, regardless of the display `type`. It has no effect inside `<ul>`.

### Nesting Lists

Nested lists go **inside** the `<li>`, not as siblings:

```html
<!-- CORRECT: nested list inside li -->
<ul>
  <li>Fruits
    <ul>
      <li>Apple</li>
      <li>Banana</li>
    </ul>
  </li>
  <li>Vegetables
    <ul>
      <li>Carrot</li>
      <li>Broccoli</li>
    </ul>
  </li>
</ul>

<!-- WRONG: nested list as sibling of li -->
<ul>
  <li>Fruits</li>
  <ul>  <!-- INVALID: ul cannot be a direct child of ul -->
    <li>Apple</li>
  </ul>
</ul>
```

Nested `<ol>` elements restart their counter from 1 (or from `start`). The parent's numbering is not inherited. For hierarchical numbering (1.1, 1.2, etc.), use CSS counters:

```css
ol {
  counter-reset: item;
  list-style-type: none;
}

ol li::before {
  counter-increment: item;
  content: counters(item, ".") " ";
}
```

### VoiceOver and `list-style: none`

Safari's VoiceOver will not announce a `<ul>` or `<ol>` as a list when `list-style: none` is applied. This is an intentional Apple design decision (to prevent navigation menus from being announced as lists). If you need the list semantics preserved after removing bullets:

```css
ul.no-bullets {
  list-style: none;
  padding-left: 0;
}
```

```html
<!-- Add role="list" to restore list semantics in VoiceOver -->
<ul class="no-bullets" role="list">
  <li>Item one</li>
  <li>Item two</li>
</ul>
```

## Description Lists: `<dl>`, `<dt>`, `<dd>`

Description lists group name-value pairs. They are underused but appropriate for glossaries, metadata, key-value displays, and FAQs.

### Content Model

`<dl>` contains groups of one or more `<dt>` elements followed by one or more `<dd>` elements. Groups can optionally be wrapped in `<div>` (the only element allowed as a direct wrapper).

### Patterns

**One term, one description:**

```html
<dl>
  <dt>Latency</dt>
  <dd>The time delay between a request and its response.</dd>
</dl>
```

**Multiple terms sharing one description (synonyms):**

```html
<dl>
  <dt>CSS</dt>
  <dt>Cascading Style Sheets</dt>
  <dd>A stylesheet language used for describing the presentation of a document.</dd>
</dl>
```

**One term with multiple descriptions:**

```html
<dl>
  <dt>Port</dt>
  <dd>A number identifying a network endpoint (e.g., port 443 for HTTPS).</dd>
  <dd>A physical connector on a device (e.g., USB port).</dd>
</dl>
```

**Key-value metadata:**

```html
<dl>
  <dt>Author</dt>
  <dd>Jane Doe</dd>

  <dt>Published</dt>
  <dd><time datetime="2025-11-15">November 15, 2025</time></dd>

  <dt>Category</dt>
  <dd>JavaScript</dd>

  <dt>Reading Time</dt>
  <dd>8 minutes</dd>
</dl>
```

### The `<div>` Wrapper

Each `<dt>`/`<dd>` group can be wrapped in a `<div>` for styling or microdata purposes. This is the only element allowed as a direct child wrapper.

```html
<dl class="metadata">
  <div class="meta-row">
    <dt>Status</dt>
    <dd>Published</dd>
  </div>
  <div class="meta-row">
    <dt>Version</dt>
    <dd>2.1.0</dd>
  </div>
</dl>
```

```css
.metadata {
  display: grid;
  grid-template-columns: max-content 1fr;
}

.meta-row {
  display: contents; /* Allows children to participate in the parent grid */
}

.meta-row dt {
  font-weight: 700;
  padding-right: 1rem;
}
```

**Rules for `<div>` wrapping:**
- Either ALL groups are wrapped or NONE are. No mixing.
- Only `<div>` is allowed. Not `<span>`, not `<section>`, not any other element.

### `<dt>` Content Restrictions

`<dt>` allows flow content but explicitly forbids:
- Heading content (`<h1>`-`<h6>`)
- Sectioning content (`<article>`, `<aside>`, `<nav>`, `<section>`)
- `<header>`, `<footer>`, `<hgroup>`

`<dd>` has no such restrictions and allows all flow content.

### Accessibility Notes

Screen reader support for `<dl>` is generally good but inconsistent across platforms:
- Some announce "description list" and a count of items when entering
- Others announce "term" and "definition" on individual parts
- VoiceOver tends to have weaker support than JAWS and NVDA
- Despite inconsistencies, using `<dl>` still provides better structure than alternatives like `<ul>` or plain `<div>`

## The `<blockquote>` Element

Represents a section quoted from another source. Contains flow content (paragraphs, headings, lists, etc.).

### The `cite` Attribute vs. the `<cite>` Element

These are different things that serve different purposes:

| | `cite` attribute | `<cite>` element |
|---|---|---|
| What it holds | A URL pointing to the source | The title of a creative work |
| Visibility | Not rendered by any browser | Rendered (italic by default) |
| Placed on/in | The `<blockquote>` tag itself | Outside or inside the attribution text |
| Consumed by | Search engines, scrapers | Screen readers, visual readers |

### Attribution Patterns

Attribution must be placed **outside** the `<blockquote>` element. Putting it inside makes it part of the quoted text, which is semantically wrong.

**Pattern 1: Simple attribution with `<p>`**

```html
<blockquote cite="https://www.brainyquote.com/quotes/linus_torvalds_367308">
  <p>Talk is cheap. Show me the code.</p>
</blockquote>
<p>-- Linus Torvalds</p>
```

**Pattern 2: Attribution with `<figure>` and `<figcaption>` (recommended)**

```html
<figure>
  <blockquote cite="https://tools.ietf.org/html/rfc1925">
    <p>With sufficient thrust, pigs fly just fine. However, this is not
    necessarily a good idea.</p>
  </blockquote>
  <figcaption>
    RFC 1925, <cite>The Twelve Networking Truths</cite>
  </figcaption>
</figure>
```

**Pattern 3: Footer inside blockquote (conforming per evolved spec)**

The spec was updated to allow a `<footer>` or `<cite>` inside `<blockquote>` for attribution:

```html
<blockquote>
  <p>The Web is fundamentally designed to work for all people.</p>
  <footer>-- <cite>W3C, Introduction to Web Accessibility</cite></footer>
</blockquote>
```

This is technically valid but semantically ambiguous because `<footer>` applies to its nearest sectioning ancestor. The `<figure>`/`<figcaption>` pattern is cleaner.

### `<blockquote>` vs. `<q>`

| | `<blockquote>` | `<q>` |
|---|---|---|
| Display | Block-level | Inline |
| Use | Extended quotations | Short, inline quotations |
| Quotes added | No (just indentation) | Yes, browser adds quotation marks |

```html
<!-- Block quote for extended text -->
<blockquote>
  <p>Programs must be written for people to read, and only incidentally
  for machines to execute.</p>
</blockquote>

<!-- Inline quote for short phrases -->
<p>As Dijkstra said, <q>simplicity is a great virtue.</q></p>
```

### Do Not Use `<blockquote>` for Indentation

```html
<!-- BAD: using blockquote for visual indentation -->
<blockquote>
  <p>This is not a quote, I just wanted indented text.</p>
</blockquote>

<!-- GOOD: use CSS for indentation -->
<p class="indented">This is not a quote, I just wanted indented text.</p>
```

```css
.indented {
  margin-left: 2rem;
}
```

## The `<pre>` Element

Represents preformatted text where whitespace (spaces, tabs, newlines) is preserved exactly as written in the source. Browsers apply `white-space: pre` and a monospaced font by default.

### Content Model

`<pre>` accepts phrasing content only. Common children: `<code>`, `<samp>`, `<kbd>`, `<var>`.

### The Standard Code Block Pattern

```html
<pre><code class="language-javascript">function fibonacci(n) {
  if (n &lt;= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}</code></pre>
```

Note: `<` and `>` must be escaped as `&lt;` and `&gt;` inside `<pre>` when displaying code that contains HTML or comparison operators.

### First Newline Stripping

The HTML parser strips the **first** newline character immediately after the `<pre>` opening tag. This is spec-defined behavior.

```html
<!-- One newline after <pre>: stripped -->
<pre>
Hello</pre>
<!-- Renders: "Hello" (no leading blank line) -->

<!-- Two newlines after <pre>: first stripped, second preserved -->
<pre>

Hello</pre>
<!-- Renders: blank line + "Hello" -->
```

**Indentation gotcha:** If you indent the `</pre>` closing tag, the spaces before it become part of the content:

```html
<!-- BAD: indentation adds unwanted spaces -->
<div>
  <pre>
function greet() {
  return 'hello';
}
  </pre>  <!-- The spaces before </pre> create trailing whitespace -->
</div>

<!-- GOOD: closing tag flush with content -->
<div>
  <pre>
function greet() {
  return 'hello';
}</pre>
</div>
```

### Controlling Tab Width

```css
pre {
  tab-size: 2;    /* Display tabs as 2 spaces wide */
}
```

The `tab-size` property only works when `white-space` is `pre`, `pre-wrap`, or `break-spaces`. The default tab width is 8.

### Handling Long Lines

By default, `<pre>` content does not wrap, causing horizontal scrolling. Two common approaches:

```css
/* Option 1: Allow wrapping (breaks visual alignment) */
pre {
  white-space: pre-wrap;
  word-break: break-all;
}

/* Option 2: Scroll container (preserves alignment) */
pre {
  overflow-x: auto;
  max-width: 100%;
}
```

### Accessibility Concerns

Screen readers do not announce `<pre>`, `<code>`, or `<samp>` elements differently by default. Browsers do not expose these elements in accessibility APIs.

**ASCII art** is a significant problem. Screen readers read individual characters, making ASCII art incomprehensible:

```html
<!-- BAD: screen reader reads "slash backslash underscore slash backslash..." -->
<pre>
 /\_/\
( o.o )
 > ^ <
</pre>

<!-- GOOD: provide accessible alternative -->
<figure>
  <pre role="img" aria-label="ASCII art of a cat face">
 /\_/\
( o.o )
 > ^ <
  </pre>
  <figcaption>ASCII art of a cat face</figcaption>
</figure>
```

**Code blocks** benefit from `<figure>` and `<figcaption>`:

```html
<figure>
  <figcaption>Listing 1: Calculating the factorial of a number</figcaption>
  <pre><code class="language-javascript">function factorial(n) {
  if (n === 0) return 1;
  return n * factorial(n - 1);
}</code></pre>
</figure>
```

## The `<hr>` Element

A thematic break between paragraph-level elements. In HTML5, the name changed from "horizontal rule" (visual) to "thematic break" (semantic).

### Semantic Meaning

`<hr>` signals a change of topic or scene, not just a decorative line. Screen readers announce it as a "separator." NVDA users can jump between separators with the `S` key.

```html
<article>
  <h2>Chapter 12: The Discovery</h2>
  <p>She opened the door and stepped into the dark room...</p>
  <p>The flashlight flickered and died.</p>

  <hr>  <!-- Scene change -->

  <p>Three hours later, rescue arrived.</p>
  <p>The team found her sitting calmly by the window.</p>
</article>
```

### When to Use `<hr>` vs. CSS Borders

| Need | Solution |
|---|---|
| Thematic/topic break in content | `<hr>` |
| Decorative line between sections | CSS `border-bottom` |
| Visual separator in a card or panel | CSS `border-top` on the next section |

```html
<!-- GOOD: thematic break (semantic) -->
<p>End of one topic.</p>
<hr>
<p>Start of a different topic.</p>

<!-- GOOD: decorative line (CSS, no semantic meaning) -->
<section class="card">
  <h2>Card Title</h2>
  <p>Card content.</p>
</section>
```

```css
.card {
  border-bottom: 1px solid #e5e7eb;
  padding-bottom: 1.5rem;
  margin-bottom: 1.5rem;
}
```

If a heading already marks the transition between sections, you do not need an `<hr>`. The heading itself communicates the shift.

### Decorative `<hr>` Hidden from Screen Readers

```html
<!-- Hide purely decorative lines from assistive technology -->
<hr aria-hidden="true">
```

## The `<div>` Element

The generic container element. It has no semantic meaning. Its implicit ARIA role is `generic`.

### When to Use `<div>`

- As a **styling hook** when no semantic element applies
- As a **layout container** (flex parent, grid parent)
- As a **JavaScript target** for DOM manipulation
- As a **grouping wrapper** for CSS positioning

```html
<!-- CORRECT: div as a layout container -->
<div class="grid grid-cols-3 gap-4">
  <article>...</article>
  <article>...</article>
  <article>...</article>
</div>

<!-- CORRECT: div as a styling wrapper -->
<div class="max-w-prose mx-auto">
  <h1>Article Title</h1>
  <p>Article content...</p>
</div>
```

### When NOT to Use `<div>`

If a semantic element exists for the content's purpose, use it instead:

```html
<!-- BAD -->
<div class="header">...</div>
<div class="navigation">...</div>
<div class="main-content">...</div>
<div class="sidebar">...</div>
<div class="footer">...</div>

<!-- GOOD -->
<header>...</header>
<nav>...</nav>
<main>...</main>
<aside>...</aside>
<footer>...</footer>
```

```html
<!-- BAD: div as a button -->
<div class="btn" onclick="handleClick()">Submit</div>
<!-- Missing: keyboard access, focus, screen reader role, form submission -->

<!-- GOOD: actual button -->
<button type="button" onclick="handleClick()">Submit</button>
```

The HTML5 spec calls `<div>` "an element of last resort, for when no other element is suitable."

## Content Model Summary

| Element | Accepts | Implicit ARIA Role | Void? |
|---|---|---|---|
| `<p>` | Phrasing content only | (none) | No |
| `<h1>`-`<h6>` | Phrasing content only | `heading` (aria-level N) | No |
| `<ul>` | `<li>` elements | `list` | No |
| `<ol>` | `<li>` elements | `list` | No |
| `<li>` | Flow content | `listitem` | No |
| `<dl>` | `<dt>`/`<dd>` groups, optionally in `<div>` | (none) | No |
| `<dt>` | Flow content (restricted) | `term` | No |
| `<dd>` | Flow content | `definition` | No |
| `<blockquote>` | Flow content | `blockquote` | No |
| `<pre>` | Phrasing content | (none) | No |
| `<hr>` | Empty | `separator` | Yes |
| `<figure>` | Flow content + optional `<figcaption>` | `figure` | No |
| `<figcaption>` | Flow content | (none) | No |
| `<div>` | Flow content | `generic` | No |

## Common Mistakes

1. **Block elements inside `<p>`.** Writing `<p><div>...</div></p>` produces a split DOM with orphaned text and empty paragraph elements. The parser cannot nest block content inside `<p>`.

2. **Fake lists with `<br>` and dashes.** Screen readers do not recognize `<p>- Item<br>- Item</p>` as a list. No item count is announced, no list navigation is available. Use `<ul>` or `<ol>`.

3. **Skipping heading levels.** Jumping from `<h2>` to `<h4>` breaks the navigable document structure. Screen reader users assume they missed a level.

4. **Headings chosen for font size.** Using `<h4>` because you want smaller text instead of using the correct level and adjusting with CSS.

5. **`<blockquote>` for indentation.** The element means "quoted from another source." For visual indentation, use CSS `margin-left`.

6. **Nested list outside `<li>`.** Placing `<ul>` as a sibling of `<li>` inside a parent `<ul>` is invalid. Nested lists must be children of `<li>`.

7. **Missing `role="list"` after `list-style: none`.** Safari VoiceOver stops announcing the element as a list when bullets are removed via CSS. Add `role="list"` to restore the semantics.

8. **Using `<hr>` as decoration.** If you just want a line, use a CSS border. `<hr>` carries semantic meaning and is announced by screen readers as a separator.

9. **Raw text outside `<p>`.** Placing text directly inside `<body>`, `<section>`, or `<div>` without wrapping in `<p>`. While technically valid, unwrapped text nodes are handled inconsistently by screen readers.

10. **Using `<address>` for arbitrary postal addresses.** The element represents contact information for the nearest `<article>` or `<body>` ancestor, not a shipping address or venue location.

---

## Best Practices

- **Wrap all text in appropriate elements.** Paragraphs in `<p>`, code in `<code>` or `<pre><code>`, quotes in `<blockquote>` or `<q>`. Avoid anonymous text nodes.
- **Maintain a strict heading hierarchy.** One `<h1>`, sequential levels, no skipping.
- **Use lists for groups of related items.** `<ul>` for unordered, `<ol>` for sequential, `<dl>` for name-value pairs.
- **Use `<figure>` for referenced content.** Code listings, images, tables, and quotes that are referenced from the main flow.
- **Use `<hr>` only for thematic breaks.** Decorative lines are CSS borders.
- **Add `role="list"` when removing list markers** if you need the list semantics preserved in Safari VoiceOver.
- **Keep nesting depth reasonable.** Lists nested more than two levels deep become difficult for all users.
- **Escape special characters in `<pre>`.** Use `&lt;` for `<` and `&amp;` for `&` when displaying code.

## Real-World Use Cases

### 1. Technical Documentation Page

```html
<main>
  <h1>API Reference: Authentication</h1>

  <section>
```

### 2. Blog Post with Quotes and Code

```html
<article>
  <h1>Why Immutability Matters</h1>

  <p>Immutability is not a constraint. It is a design choice that eliminates
     an entire category of bugs.</p>

  <figure>
    <blockquote cite="https://www.infoq.com/presentations/Value-Values/">
      <p>Place-oriented programming means mutable state, and mutable state
         is the new spaghetti code.</p>
    </blockquote>
    <figcaption>Rich Hickey, <cite>The Value of Values</cite></figcaption>
  </figure>

  <h2>The Problem with Mutation</h2>
  <p>Consider this common pattern:</p>

  <figure>
    <figcaption>Bad practice: mutating an array parameter</figcaption>
    <pre><code class="language-javascript">// BAD: the caller's array is modified
function addItem(items, newItem) {
  items.push(newItem); // mutation!
  return items;
}

const original = ['a', 'b'];
const updated = addItem(original, 'c');
console.log(original); // ['a', 'b', 'c'] -- original was mutated!</code></pre>
  </figure>

  <figure>
    <figcaption>Good practice: returning a new array</figcaption>
    <pre><code class="language-javascript">// GOOD: original array is untouched
function addItem(items, newItem) {
  return [...items, newItem]; // new array
}

const original = ['a', 'b'];
const updated = addItem(original, 'c');
console.log(original); // ['a', 'b'] -- untouched
console.log(updated);  // ['a', 'b', 'c']</code></pre>
  </figure>

  <h2>Benefits</h2>
  <ol>
    <li>Predictable state: no hidden side effects from shared references</li>
    <li>Easier debugging: state at any point in time can be inspected</li>
    <li>Thread safety: no race conditions on shared data</li>
    <li>Change detection: reference comparison replaces deep equality checks</li>
  </ol>

  <hr>

  <h2>When Mutation Is Acceptable</h2>
  <p>Immutability has a cost. Creating new objects and arrays consumes memory.
     In performance-critical loops processing thousands of items, controlled
     mutation within a local scope may be the right tradeoff.</p>
</article>
```

> **Key Takeaway:** Text content elements are not interchangeable wrappers. Each carries specific semantics that affect how screen readers navigate content, how search engines index it, and how browsers recover from nesting errors. Use `<p>` for paragraphs, real list elements for lists, `<blockquote>` only for actual quotes, `<pre>` for whitespace-sensitive content, `<hr>` for topic shifts, and `<div>` only when nothing else fits.
