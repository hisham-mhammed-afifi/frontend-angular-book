# 1.5 Images and Media

> Images, video, and audio are the heaviest resources on most web pages. How you embed them determines page load speed, layout stability, accessibility, and bandwidth costs.

---

## The `<img>` Element

### Core Attributes

```html
<img
  src="photo.jpg"
  alt="Golden retriever playing fetch in a park"
  width="800"
  height="600"
  loading="lazy"
  decoding="async"
  fetchpriority="auto"
>
```

| Attribute | Purpose |
|---|---|
| `src` | URL of the image. Required for the element to render anything. |
| `alt` | Text alternative. Required on every `<img>`. |
| `width` / `height` | Intrinsic dimensions in pixels (no units). Prevents layout shift. |
| `loading` | `"lazy"` or `"eager"` (default). Controls when the browser fetches the image. |
| `decoding` | `"async"`, `"sync"`, or `"auto"`. Controls decode timing. |
| `fetchpriority` | `"high"`, `"low"`, or `"auto"`. Signals relative fetch importance. |
| `srcset` | Multiple image sources for resolution or width switching. |
| `sizes` | Media conditions describing how wide the image renders at each breakpoint. |
| `crossorigin` | CORS mode: `"anonymous"` or `"use-credentials"`. Needed for canvas usage. |
| `referrerpolicy` | Controls `Referer` header when fetching. |

`<img>` is a void element. It cannot have children or a closing tag.

## Alt Text

Every `<img>` must have an `alt` attribute. This is WCAG 1.1.1 (Level A). There are no exceptions.

### Three Categories of Alt Text

**1. Informative images: describe the content.**

```html
<img src="chart.png"
     alt="Bar chart showing revenue growing from $2M in Q1 to $5M in Q4 2025">
```

**2. Functional images (links, buttons): describe the action or destination.**

```html
<a href="/">
  <img src="logo.svg" alt="Acme Corp homepage">
</a>

<button type="submit">
  <img src="search-icon.svg" alt="Search">
</button>
```

**3. Decorative images: use empty alt.**

```html
<!-- Decorative flourish, adds no information -->
<img src="divider.svg" alt="">

<!-- Icon next to text that already conveys the meaning -->
<img src="checkmark.svg" alt=""> Saved successfully
```

### `alt=""` vs. Missing `alt`

| Scenario | Screen Reader Behavior |
|---|---|
| `alt="Golden retriever"` | Announces "Golden retriever, image" |
| `alt=""` | Skips the image entirely (correct for decorative) |
| No `alt` attribute | Announces the **file name**: "IMG_20230415_143022.jpg, image" |

A missing `alt` attribute is a WCAG violation that makes the page noisy and confusing for screen reader users.

### Alt Text Rules

- Keep it concise. Aim for under 125 characters (some screen readers truncate beyond this).
- Describe the **purpose**, not just the appearance. "Submit button" is better than "green rectangle."
- Do not start with "image of" or "picture of." The screen reader already announces it as an image.
- Do not duplicate surrounding text. If a caption says "Figure 1: Revenue chart," the alt should describe what the chart shows, not repeat "Revenue chart."
- For complex images (charts, infographics), provide a brief alt plus a longer description via adjacent text, `<figcaption>`, or `aria-describedby`.

```html
<!-- BAD -->
<img src="photo.jpg" alt="photo">
<img src="photo.jpg" alt="image">
<img src="photo.jpg" alt="photo.jpg">
<img src="graph.png" alt="graph">

<!-- GOOD -->
<img src="photo.jpg" alt="Team celebrating the product launch at the NYC office">
<img src="graph.png" alt="Line graph showing 40% reduction in page load time after optimization">
```

## Width, Height, and Layout Shift (CLS)

When a browser encounters an `<img>` without `width` and `height`, it does not know how much vertical space to reserve. The image loads, then pushes content down. This is Cumulative Layout Shift (CLS), a Core Web Vital that impacts search rankings.

### The Fix

```html
<img src="photo.jpg" alt="Description" width="800" height="600">
```

The browser reads these attributes, computes the aspect ratio (800/600 = 4:3), and reserves the correct space before the image loads. Zero layout shift.

### How It Interacts with CSS

The `width` and `height` attributes set the **intrinsic** aspect ratio. CSS controls the rendered size:

```css
img {
  max-width: 100%;
  height: auto; /* preserves aspect ratio from attributes */
}
```

Modern browsers automatically derive `aspect-ratio` from the `width`/`height` attributes. Setting `width="800" height="600"` is equivalent to the browser applying `aspect-ratio: 800 / 600`.

### When CSS Overrides the Aspect Ratio

If CSS forces a different aspect ratio than the attributes define, the image distorts. Use `object-fit` to handle this:

```css
.hero-image {
  width: 100%;
  height: 300px; /* forces a different aspect ratio */
  object-fit: cover; /* crops to fill without distortion */
}
```

| `object-fit` Value | Behavior |
|---|---|
| `fill` | Stretches to fill (distorts). Default. |
| `contain` | Scales to fit within the box, preserving aspect ratio. May show gaps. |
| `cover` | Scales to cover the box, preserving aspect ratio. Crops overflow. |
| `none` | No resizing. Shows at intrinsic size, clips if too large. |
| `scale-down` | Like `contain` but never scales up, only down. |

## Loading: Lazy vs. Eager

### `loading="lazy"`

Defers fetching until the image approaches the viewport. The browser uses an internal threshold (Chrome: ~1250px on fast connections, ~2500px on slow connections).

```html
<!-- Below-the-fold images: lazy load -->
<img src="gallery-photo.jpg" alt="..." width="400" height="300" loading="lazy">
```

### `loading="eager"` (Default)

Fetches the image immediately, regardless of its position on the page. This is the default when the attribute is omitted.

```html
<!-- LCP / above-the-fold images: eager load (or omit the attribute) -->
<img src="hero.jpg" alt="..." width="1200" height="600" fetchpriority="high">
```

### When NOT to Lazy Load

| Image Type | Why Eager |
|---|---|
| Hero / banner image | Almost always the LCP element |
| Above-the-fold content | Visible on initial load |
| Logo in the header | Visible on every page load |
| First image in a carousel | The one the user sees first |

Lazy loading the LCP image delays the most important visual on the page and hurts Core Web Vitals scores.

---

## `fetchpriority`

Signals relative importance to the browser's resource fetcher.

```html
<!-- Hero image: high priority -->
<img src="hero.jpg" alt="..." fetchpriority="high">

<!-- Carousel image not currently visible: low priority -->
<img src="slide-3.jpg" alt="..." loading="lazy" fetchpriority="low">
```

| Value | When to Use |
|---|---|
| `high` | LCP image, hero image, critical above-the-fold content |
| `low` | Below-the-fold images, carousel slides not in view, thumbnails |
| `auto` | Default. Let the browser decide. |

**Gotcha:** `fetchpriority="high"` combined with `loading="lazy"` still defers the fetch while off-screen. The high priority kicks in only once the image approaches the viewport. This combination is rarely useful.

---

## `decoding`

Controls whether the browser decodes the image synchronously or asynchronously relative to rendering other content.

```html
<!-- Decode off the main thread (good for below-the-fold) -->
<img src="photo.jpg" alt="..." decoding="async">

<!-- Decode synchronously (guarantees image is ready when painted) -->
<img src="logo.png" alt="..." decoding="sync">
```

| Value | Behavior |
|---|---|
| `async` | Decode off the main thread. Page renders faster but the image may pop in. |
| `sync` | Decode blocks rendering. Image is guaranteed visible on first paint. |
| `auto` | Browser decides. Default. |

**Gotcha:** `decoding="async"` on above-the-fold LCP images can cause a flash where text renders first and the image pops in a moment later.

## Responsive Images with `srcset` and `sizes`

### Resolution Switching (Pixel Density Descriptors)

For **fixed-size** images that need higher resolution on HiDPI screens:

```html
<img src="logo.png"
     srcset="logo.png 1x, logo@2x.png 2x, logo@3x.png 3x"
     alt="Company Logo"
     width="200"
     height="100">
```

The browser picks the image matching the device pixel ratio (DPR). A 2x Retina display gets `logo@2x.png`. No `sizes` attribute needed.

### Width Switching (Width Descriptors)

For **responsive** images that change size with the viewport:

```html
<img src="photo-800.jpg"
     srcset="photo-400.jpg 400w,
             photo-800.jpg 800w,
             photo-1200.jpg 1200w,
             photo-1600.jpg 1600w"
     sizes="(max-width: 600px) 100vw,
            (max-width: 1200px) 50vw,
            33vw"
     alt="Landscape photograph">
```

The `w` value tells the browser the intrinsic pixel width of each image file. The browser then:

1. Evaluates `sizes` media conditions left-to-right, finds the first match
2. Gets the slot width (e.g., `50vw` at 1000px viewport = 500px)
3. Multiplies by device pixel ratio (e.g., 500px * 2 DPR = 1000px)
4. Picks the smallest `srcset` image whose width >= 1000px (selects `photo-1200.jpg`)

### Rules

- You **cannot mix** `x` and `w` descriptors in the same `srcset`. Both get ignored.
- When using `w` descriptors, the `sizes` attribute is **required**. Without it, the browser assumes `100vw`.
- The `src` attribute serves as the fallback for browsers that do not support `srcset`.

### `sizes` Syntax

```
sizes="[media-condition] [length], [media-condition] [length], [default-length]"
```

```html
<!-- Full width on mobile, half on tablet, third on desktop -->
sizes="(max-width: 600px) 100vw, (max-width: 1200px) 50vw, 33vw"

<!-- Fixed width at all sizes -->
sizes="200px"

<!-- Using calc() -->
sizes="(max-width: 600px) calc(100vw - 2rem), 600px"
```

## The `<picture>` Element

`<picture>` gives the author explicit control over which image the browser loads. The browser **must** use the first matching `<source>`, unlike `srcset` where the browser has discretion.

Two use cases: **art direction** and **format switching**.

### Art Direction

Serve differently cropped images for different viewport sizes:

```html
<picture>
  <!-- Wide crop for desktop -->
  <source media="(min-width: 1200px)" srcset="hero-wide.jpg" width="1200" height="400">

  <!-- Square crop for tablet -->
  <source media="(min-width: 600px)" srcset="hero-square.jpg" width="600" height="600">

  <!-- Tall crop for mobile -->
  <img src="hero-tall.jpg" alt="Product showcase" width="400" height="600"
       loading="eager" fetchpriority="high">
</picture>
```

### Format Switching

Serve modern formats with fallbacks:

```html
<picture>
  <source type="image/avif" srcset="photo.avif">
  <source type="image/webp" srcset="photo.webp">
  <img src="photo.jpg" alt="Description" width="800" height="600">
</picture>
```

The browser checks if it supports the `type` MIME type **before** making any network request. If it does not support `image/avif`, it skips to the next `<source>` without downloading anything.

### Combining Art Direction, Format Switching, and Responsive Sizes

```html
<picture>
  <!-- Desktop: wide crop, AVIF with responsive widths -->
  <source media="(min-width: 1024px)"
          type="image/avif"
          srcset="hero-wide-800.avif 800w,
                  hero-wide-1200.avif 1200w,
                  hero-wide-1600.avif 1600w"
          sizes="100vw">

  <!-- Desktop: wide crop, WebP fallback -->
  <source media="(min-width: 1024px)"
          type="image/webp"
          srcset="hero-wide-800.webp 800w,
                  hero-wide-1200.webp 1200w,
                  hero-wide-1600.webp 1600w"
          sizes="100vw">

  <!-- Mobile: tall crop, AVIF -->
  <source type="image/avif"
          srcset="hero-tall-400.avif 400w,
                  hero-tall-800.avif 800w"
          sizes="100vw">

  <!-- Mobile: tall crop, WebP -->
  <source type="image/webp"
          srcset="hero-tall-400.webp 400w,
                  hero-tall-800.webp 800w"
          sizes="100vw">

  <!-- Final fallback: JPEG -->
  <img src="hero-tall-800.jpg" alt="Product hero" width="800" height="1000">
</picture>
```

### Rules for `<picture>`

- `<source>` order matters. The browser uses the **first match**. Put most specific conditions first.
- The `<img>` element inside `<picture>` is **required**. Without it, nothing renders.
- All attributes go on the `<img>`: `alt`, `width`, `height`, `loading`, `fetchpriority`, `class`, etc.
- `<picture>` itself renders nothing. It is a wrapper that provides sources to the `<img>`.

### `<picture>` vs. `<img srcset>`

| Feature | `<img srcset>` | `<picture>` |
|---|---|---|
| Use case | Same image, different resolutions | Different images or formats |
| Browser discretion | Yes (may use cached smaller image) | No (must use first match) |
| Art direction | No | Yes (via `media` on `<source>`) |
| Format switching | No | Yes (via `type` on `<source>`) |
| `sizes` needed | Yes (with `w` descriptors) | On `<source>` if using `w` descriptors |

## Image Formats

| Format | Type | Transparency | Animation | Compression vs JPEG | Best For |
|---|---|---|---|---|---|
| JPEG | Lossy | No | No | Baseline | Photographs, complex color images |
| PNG | Lossless | Yes (alpha) | No (APNG does) | Much larger | Logos, icons, screenshots, transparency |
| GIF | Lossless (256 colors) | Binary only | Yes | Poor | Simple animations (legacy) |
| WebP | Lossy or lossless | Yes (alpha) | Yes | ~25-35% smaller | General-purpose replacement for JPEG/PNG |
| AVIF | Lossy or lossless | Yes (alpha) | Yes | ~50% smaller | Best compression, photographs, HDR |
| SVG | Vector | Yes | Yes (CSS/SMIL) | N/A (scalable) | Logos, icons, illustrations, UI graphics |

### Browser Support (2025)

| Format | Support |
|---|---|
| JPEG / PNG / GIF | 100% |
| WebP | ~97% |
| AVIF | ~93% |
| SVG | ~98% |

### Recommended Strategy

```html
<picture>
  <source type="image/avif" srcset="photo.avif">
  <source type="image/webp" srcset="photo.webp">
  <img src="photo.jpg" alt="Description">
</picture>
```

### Format Gotchas

- **AVIF encoding is slow.** Server-side encoding takes significantly longer than JPEG or WebP. Budget for this in build pipelines.
- **AVIF progressive decoding is weak.** Images may appear to "pop in" rather than gradually sharpen.
- **Lossy WebP can show banding** in smooth gradients (skies, vignettes).
- **Lossless WebP can be larger than optimized PNG** for certain image types (flat-color illustrations).
- **PNG for photographs** is a common waste. A PNG photo can be 5-10x larger than an equivalent JPEG.

## SVG: Inline vs. `<img src>`

| | `<img src="icon.svg">` | Inline `<svg>` |
|---|---|---|
| Cacheable | Yes (separate HTTP request, cached) | No (embedded in HTML, re-downloaded per page) |
| CSS styling | Cannot style internals | Full CSS control (colors, hover, animations) |
| JS manipulation | Cannot access internals | Full DOM access |
| HTTP requests | One per image (or zero if cached) | Zero (inline) |
| HTML size | Small | Increases with SVG complexity |
| Lazy loadable | Yes | No |
| Best for | Static, reusable, decorative SVGs | Interactive, color-changing, critical icons |

### Accessible Inline SVG

```html
<!-- Informative SVG -->
<svg role="img" aria-labelledby="cart-title" viewBox="0 0 24 24">
  <title id="cart-title">Shopping cart with 3 items</title>
  <!-- paths -->
</svg>

<!-- Decorative SVG -->
<svg aria-hidden="true" focusable="false" viewBox="0 0 24 24">
  <!-- paths -->
</svg>
```

## The `<video>` Element

```html
<video
  controls
  width="640"
  height="360"
  poster="thumbnail.jpg"
  preload="metadata"
  playsinline
>
  <source src="video.webm" type="video/webm">
  <source src="video.mp4" type="video/mp4">
  <track kind="captions" src="captions-en.vtt" srclang="en" label="English" default>
  <track kind="subtitles" src="subtitles-es.vtt" srclang="es" label="Spanish">
  <p>Your browser does not support HTML video.
     <a href="video.mp4">Download the video</a>.</p>
</video>
```

### Key Attributes

| Attribute | Purpose |
|---|---|
| `controls` | Shows native playback controls. Always include unless building custom UI. |
| `poster` | Image shown before playback starts. Without it, the browser shows the first frame (or blank). |
| `preload` | `"none"`, `"metadata"`, or `"auto"`. Hint for how much to preload. |
| `autoplay` | Starts playback automatically. Subject to browser restrictions. |
| `muted` | Starts muted. **Required** for autoplay to work in most browsers. |
| `loop` | Restarts when playback ends. |
| `playsinline` | Prevents iOS Safari from forcing fullscreen. **Required** for autoplay on iOS. |
| `width` / `height` | Player dimensions. Prevents CLS. |

### `preload` Values

| Value | Behavior | When to Use |
|---|---|---|
| `none` | Downloads nothing until play is requested | Videos users may never play; many videos on one page |
| `metadata` | Downloads metadata (duration, dimensions, first frame) | Best default for most cases |
| `auto` | Browser may download the entire video | Only when you are confident the user will play it |

`preload` is ignored when `autoplay` is present. The browser preloads enough to start playback.

### Autoplay Restrictions

Browsers enforce these rules (as of 2025, all major browsers):

| Scenario | Allowed? |
|---|---|
| Muted autoplay | Always allowed |
| Autoplay with sound | Blocked by default |
| Autoplay with sound + prior user interaction on domain | Allowed (Chrome) |
| Autoplay with sound + installed PWA | Allowed |

**iOS Safari** requires **both** `muted` and `playsinline` for autoplay.

```html
<!-- Muted background video: works everywhere -->
<video autoplay muted loop playsinline poster="poster.jpg">
  <source src="background.webm" type="video/webm">
  <source src="background.mp4" type="video/mp4">
</video>
```

### Detecting Autoplay Capability

```javascript
const video = document.querySelector('video');
const playPromise = video.play();

if (playPromise !== undefined) {
  playPromise
    .then(() => {
      // Autoplay started successfully
    })
    .catch(() => {
      // Autoplay was blocked, show a play button
      video.controls = true;
    });
}
```

## The `<audio>` Element

```html
<audio controls preload="metadata">
  <source src="podcast.ogg" type="audio/ogg; codecs=opus">
  <source src="podcast.mp3" type="audio/mpeg">
  <p>Your browser does not support HTML audio.
     <a href="podcast.mp3">Download the episode</a>.</p>
</audio>
```

Attributes are the same as `<video>` minus `poster`, `playsinline`, and `width`/`height`.

**Never autoplay audio with sound.** It is a hostile user experience and is blocked by browsers unless the user has previously interacted with the domain.

Always include `controls`. Even if you build custom UI, native controls serve as an accessible fallback.

## The `<source>` Element

`<source>` provides alternative media resources inside `<picture>`, `<video>`, and `<audio>`. Its allowed attributes depend on the parent:

### Inside `<picture>`

| Attribute | Purpose |
|---|---|
| `srcset` | Image URLs with optional width or density descriptors (required) |
| `sizes` | Media conditions with slot widths |
| `media` | Media query for art direction |
| `type` | MIME type for format switching |
| `width` / `height` | CLS prevention |

`src` is NOT allowed on `<source>` inside `<picture>`.

### Inside `<video>` or `<audio>`

| Attribute | Purpose |
|---|---|
| `src` | URL of the media resource (required) |
| `type` | MIME type, optionally with codecs |

`srcset`, `sizes`, and `media` are NOT allowed on `<source>` inside `<video>` or `<audio>`.

### Why `type` Matters

When `type` is specified, the browser checks format support **before** making any network request. If it does not support the format, it skips to the next `<source>` without downloading anything. Without `type`, the browser must request the file to determine its format, wasting bandwidth.

```html
<!-- GOOD: type lets browser skip unsupported formats without downloading -->
<source src="video.webm" type="video/webm; codecs=vp9,opus">
<source src="video.mp4" type="video/mp4">

<!-- BAD: browser must request each file to check support -->
<source src="video.webm">
<source src="video.mp4">
```

## The `<track>` Element

Provides timed text tracks for `<video>` and `<audio>`.

```html
<video controls>
  <source src="lecture.mp4" type="video/mp4">
  <track kind="captions" src="cc-en.vtt" srclang="en" label="English (CC)" default>
  <track kind="subtitles" src="sub-es.vtt" srclang="es" label="Spanish">
  <track kind="descriptions" src="desc-en.vtt" srclang="en" label="Audio Descriptions">
  <track kind="chapters" src="chapters.vtt" srclang="en" label="Chapters">
</video>
```

### Track Kinds

| Kind | Purpose | For Whom |
|---|---|---|
| `captions` | Dialog + non-speech sounds: "[door slams]", "[music]" | Deaf / hard of hearing |
| `subtitles` | Translation or transcription of dialog only | Language translation |
| `descriptions` | Text descriptions of visual content, synthesized as audio | Blind users |
| `chapters` | Chapter titles shown in the seek bar | All users |
| `metadata` | Machine-readable data, not displayed | JavaScript consumption |

### Attributes

| Attribute | Required? | Purpose |
|---|---|---|
| `kind` | Yes | Track type (see above) |
| `src` | Yes | URL to a WebVTT (`.vtt`) file |
| `srclang` | Required for `subtitles` | BCP 47 language tag (e.g., `"en"`, `"es"`) |
| `label` | Recommended | Human-readable name shown in the track picker UI |
| `default` | No | Enables this track by default (one per kind) |

### WebVTT Format

```
WEBVTT

00:00:01.000 --> 00:00:04.000
Welcome to this tutorial on HTML media elements.

00:00:05.000 --> 00:00:08.500
We will cover images, video, and audio.

00:00:09.000 --> 00:00:12.000
[upbeat music playing]
```

**Gotcha:** Cross-origin `.vtt` files require CORS headers on the server.

**Gotcha:** `<track>` must be placed after all `<source>` elements and before fallback content.

## Performance: LCP Image Optimization

The Largest Contentful Paint (LCP) image is the single most impactful element for perceived load speed. Optimizing it:

### Checklist

```html
<!-- 1. Do NOT lazy load -->
<!-- 2. Set fetchpriority="high" -->
<!-- 3. Set width and height for CLS -->
<!-- 4. Use modern format with fallback -->
<picture>
  <source type="image/avif" srcset="hero.avif">
  <source type="image/webp" srcset="hero.webp">
  <img src="hero.jpg"
       alt="Product showcase"
       width="1200"
       height="600"
       fetchpriority="high">
</picture>
```

### Preloading the LCP Image

If the LCP image is not discoverable in the initial HTML (CSS background image, JavaScript-loaded image), preload it:

```html
<head>
  <link rel="preload" as="image" href="hero.jpg" fetchpriority="high">
</head>
```

For responsive images:

```html
<head>
  <link rel="preload" as="image"
        href="hero-1200.jpg"
        imagesrcset="hero-400.jpg 400w, hero-800.jpg 800w, hero-1200.jpg 1200w"
        imagesizes="100vw"
        fetchpriority="high">
</head>
```

Do not preload everything. When everything is high priority, nothing is.

## Common Mistakes

1. **Missing `alt` attribute.** Screen readers announce the file name. Every `<img>` needs `alt`, even if empty.

2. **Lazy loading the LCP image.** Delays the most important visual element. Never use `loading="lazy"` on above-the-fold images.

3. **Missing `width` and `height`.** Causes layout shift (CLS) when the image loads, pushing content down.

4. **Using PNG for photographs.** PNGs are massive for photo content. A PNG photo can be 5-10x larger than an equivalent JPEG.

5. **Serving oversized images.** A 4000px-wide image in a 400px container wastes bandwidth. Use `srcset` and `sizes`.

6. **Mixing `x` and `w` descriptors in `srcset`.** Both get silently ignored. Use one type per `srcset`.

7. **Using `w` descriptors without `sizes`.** The browser assumes `100vw`, which often selects the wrong image.

8. **Wrong `<source>` order in `<picture>`.** The browser uses the first match. Put the most specific conditions first.

9. **Missing `<img>` inside `<picture>`.** Nothing renders. The `<img>` is required and is where `alt`, `loading`, etc. go.

10. **Missing `type` on media `<source>` elements.** Forces the browser to download files to check format support.

11. **Autoplay without `muted`.** Browsers silently block it. The video or audio does not play and no error is shown.

12. **Missing `playsinline` on iOS.** Videos hijack into fullscreen mode on iPhone.

13. **`preload="auto"` on multiple videos.** Downloads entire video files even if the user never plays them.

14. **`decoding="async"` on LCP images.** Can cause a pop-in flash where text renders first.

15. **Using images for what CSS can do.** Shadows, gradients, rounded corners, and borders should be CSS.

---

## Best Practices

- **Every `<img>` gets an `alt` attribute.** Descriptive for informative images, empty for decorative.
- **Always set `width` and `height`** on `<img>` and `<video>` to prevent layout shift.
- **Use `fetchpriority="high"` on the LCP image** and do not lazy load it.
- **Use `loading="lazy"` on all below-the-fold images.**
- **Serve modern formats** via `<picture>`: AVIF first, WebP second, JPEG/PNG fallback.
- **Use `srcset` with `w` descriptors and `sizes`** for responsive images that change size with the viewport.
- **Use pixel density descriptors** (`1x`, `2x`) for fixed-size images (logos, icons).
- **Set `preload="metadata"`** as the default for video and audio.
- **Always include `muted` and `playsinline`** for autoplay video.
- **Provide captions** (`kind="captions"`) for all video content and transcripts for all audio content.
- **Include `type` on every `<source>`** to prevent unnecessary downloads.

## Real-World Use Cases

### 1. Responsive Hero Image with Format Switching

```html
<section class="hero">
```
