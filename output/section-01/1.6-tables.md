# 1.6 Tables

> HTML tables exist for one purpose: presenting data that has a logical relationship between rows and columns. Getting the structure right determines whether screen readers can navigate the data and whether CSS can style it predictably.

---

## Table Element Hierarchy

A valid table follows this element order:

```
<table>
  <caption>         (optional, must be first)
  <colgroup>/<col>  (optional, zero or more)
  <thead>           (optional, maximum one)
  <tbody>           (zero or more, or bare <tr> elements)
  <tfoot>           (optional, maximum one, after all tbody/tr)
</table>
```

You cannot mix `<tbody>` elements and bare `<tr>` elements as direct children of `<table>`. Use one approach or the other.

### Minimal Valid Table

```html
<table>
  <tr>
    <th>Name</th>
    <th>Score</th>
  </tr>
  <tr>
    <td>Alice</td>
    <td>95</td>
  </tr>
</table>
```

### Fully Structured Table

```html
<table>
  <caption>Student Test Scores</caption>
  <colgroup>
    <col>
    <col class="scores">
  </colgroup>
  <thead>
    <tr>
      <th scope="col">Name</th>
      <th scope="col">Score</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Alice</td>
      <td>95</td>
    </tr>
    <tr>
      <td>Bob</td>
      <td>87</td>
    </tr>
  </tbody>
  <tfoot>
    <tr>
      <th scope="row">Average</th>
      <td>91</td>
    </tr>
  </tfoot>
</table>
```

## The `<caption>` Element

The caption is the accessible name of the table. Screen readers announce it when a user encounters the table, helping them decide whether to explore it.

```html
<table>
  <caption>Monthly Revenue by Product Line (USD)</caption>
  <!-- ... -->
</table>
```

**Rules:**
- Must be the **first child** of `<table>`
- Visible to all users (unlike `aria-label`, which is screen-reader-only)
- WCAG requires tables to have an accessible name; `<caption>` is the preferred method
- Can be styled with CSS: font, color, padding, `text-align`
- `caption-side: bottom` moves it visually below the table, but it stays first in the DOM

```css
table {
  caption-side: bottom; /* renders below the table visually */
}

caption {
  font-weight: 700;
  text-align: left;
  padding: 0.5rem 0;
}
```

## `<colgroup>` and `<col>`

These elements define column groups for structural and (limited) styling purposes.

```html
<table>
  <colgroup>
    <col>                          <!-- first column -->
    <col span="4" class="data">   <!-- next four columns -->
  </colgroup>
  <!-- ... -->
</table>
```

### CSS Properties That Actually Work on `<col>`

Only four categories of CSS properties have any effect:

| Property Category | Notes |
|---|---|
| `background` | Paints behind cell backgrounds (invisible if cells have their own background) |
| `border` | Only with `border-collapse: collapse` on the table |
| `width` | Sets default column width |
| `visibility: collapse` | Supposed to hide the column. Firefox implements it correctly. Chrome/Safari treat it as `visibility: hidden` (space not reclaimed). |

**Everything else is silently ignored.** You cannot set `color`, `font-size`, `padding`, `text-align`, or any other property on `<col>`.

```css
/* This works */
col.highlight {
  background: #fef9c3;
  width: 200px;
}

/* This is silently ignored */
col.highlight {
  color: red;        /* ignored */
  font-weight: bold; /* ignored */
  text-align: right; /* ignored */
}
```

### Background Paint Order (Back to Front)

`table` > `colgroup` > `col` > `thead`/`tbody`/`tfoot` > `tr` > `td`/`th`

A column background is only visible through cells that have `background: transparent` (the default).

## `<thead>`, `<tbody>`, `<tfoot>`

### None Are Required

The spec does not mandate any of these. However, using them provides:
- Semantic grouping for screen readers
- Independent scrolling of the body (with CSS `overflow` on a `display: block` tbody)
- Print behavior: `<thead>` repeats at the top of each printed page (in supporting browsers)
- Required structure for `scope="rowgroup"` and `scope="colgroup"` associations

### The Implicit `<tbody>` (Critical Gotcha)

Browsers **always** insert a `<tbody>` around `<tr>` elements that are direct children of `<table>`:

```html
<!-- What you write -->
<table>
  <tr><td>Data</td></tr>
</table>

<!-- What the DOM actually contains -->
<table>
  <tbody>
    <tr><td>Data</td></tr>
  </tbody>
</table>
```

**Consequences:**
- CSS `table > tr` **never matches**. Use `table > tbody > tr` or `table tr`.
- `document.querySelector('table').children[0]` returns the `<tbody>`, not the `<tr>`.
- `table.rows` works correctly regardless (it spans all sections), but direct DOM traversal is affected.

### Print Behavior

| Section | Expected Print Behavior | Browser Reality |
|---|---|---|
| `<thead>` | Repeats at the top of each printed page | Chrome: yes. Firefox: yes. Safari: varies. |
| `<tfoot>` | Repeats at the bottom of each printed page | Chrome: no. Firefox: partially. Safari: no. |

Use `break-inside: avoid` on `<tr>` to prevent rows from splitting across pages.

## The `<th>` Element and `scope`

The `<th>` element defines a header cell. The `scope` attribute tells screen readers which data cells the header applies to.

### `scope` Values

| Value | Direction | Use Case |
|---|---|---|
| `col` | Header applies to all cells **below** it in the column | Column headers in `<thead>` |
| `row` | Header applies to all cells **to the right** in the row | Row headers in the first column |
| `colgroup` | Header applies to all cells in the column group | Multi-level column group headers (with `colspan`) |
| `rowgroup` | Header applies to all cells in the row group | Multi-level row group headers (with `rowspan`) |

### Simple Table with Column and Row Headers

```html
<table>
  <caption>Server Response Times (ms)</caption>
  <thead>
    <tr>
      <th scope="col">Endpoint</th>
      <th scope="col">P50</th>
      <th scope="col">P95</th>
      <th scope="col">P99</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">/api/users</th>
      <td>45</td>
      <td>120</td>
      <td>350</td>
    </tr>
    <tr>
      <th scope="row">/api/orders</th>
      <td>62</td>
      <td>180</td>
      <td>520</td>
    </tr>
  </tbody>
</table>
```

A screen reader user navigating to the cell "180" hears: "P95, /api/orders, 180." Both the column and row headers are announced.

### Multi-Level Headers with `scope="colgroup"`

```html
<table>
  <caption>Product Availability by Store</caption>
  <thead>
    <tr>
      <td></td>
      <th scope="colgroup" colspan="2">Downtown</th>
      <th scope="colgroup" colspan="2">Suburbs</th>
    </tr>
    <tr>
      <td></td>
      <th scope="col">Main St</th>
      <th scope="col">2nd Ave</th>
      <th scope="col">Mall</th>
      <th scope="col">Plaza</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">Widget A</th>
      <td>12</td>
      <td>5</td>
      <td>8</td>
      <td>15</td>
    </tr>
    <tr>
      <th scope="row">Widget B</th>
      <td>3</td>
      <td>0</td>
      <td>22</td>
      <td>7</td>
    </tr>
  </tbody>
</table>
```

### When `scope` Is Not Enough: `headers` and `id`

For irregular tables where headers do not align in clean rows or columns, use the `headers` attribute with `id` references on each `<th>`:

```html
<table>
  <caption>Exam Results by Student and Subject</caption>
  <thead>
    <tr>
      <td></td>
      <th id="math" scope="col">Math</th>
      <th id="science" scope="col">Science</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th id="alice" scope="row">Alice</th>
      <td headers="alice math">95</td>
      <td headers="alice science">88</td>
    </tr>
    <tr>
      <th id="bob" scope="row">Bob</th>
      <td headers="bob math">72</td>
      <td headers="bob science">91</td>
    </tr>
  </tbody>
</table>
```

The `headers` attribute takes a space-separated list of `id` values for all headers that apply to that cell.

**Use `scope` whenever possible.** Reserve `headers`/`id` for genuinely irregular tables. The `headers` approach is more fragile (IDs must be unique document-wide) and more verbose.

## `colspan` and `rowspan`

### `colspan`: Spanning Columns

```html
<table>
  <thead>
    <tr>
      <th scope="col">Task</th>
      <th scope="col">Mon</th>
      <th scope="col">Tue</th>
      <th scope="col">Wed</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Planning</td>
      <td colspan="2">All day meeting</td>
      <!-- only 1 more td needed (3 columns - colspan 2 = 1) -->
      <td>Review</td>
    </tr>
  </tbody>
</table>
```

### `rowspan`: Spanning Rows

```html
<table>
  <thead>
    <tr>
      <th scope="col">Category</th>
      <th scope="col">Item</th>
      <th scope="col">Price</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row" rowspan="2">Drinks</th>
      <td>Coffee</td>
      <td>$3.50</td>
    </tr>
    <tr>
      <!-- no th here: "Drinks" spans into this row -->
      <td>Tea</td>
      <td>$2.50</td>
    </tr>
    <tr>
      <th scope="row" rowspan="2">Food</th>
      <td>Sandwich</td>
      <td>$8.00</td>
    </tr>
    <tr>
      <td>Salad</td>
      <td>$7.00</td>
    </tr>
  </tbody>
</table>
```

### Counting Rules

- If a row has N columns and one cell uses `colspan="X"`, that row needs `N - X + 1` cell elements.
- When using `rowspan="Y"` on a cell, the Y-1 rows below it need one fewer `<td>` to avoid misalignment.
- `rowspan="0"` spans to the end of the current section (`<thead>`, `<tbody>`, or `<tfoot>`), not the entire table.
- `colspan="0"` spans the remaining columns in its column group (browser support varies).
- Maximum `colspan`: 1000. Maximum `rowspan`: 65534.

## How Screen Readers Navigate Tables

Screen reader users navigate tables cell-by-cell using keyboard shortcuts:

| Action | NVDA / JAWS | VoiceOver (Mac) |
|---|---|---|
| Next cell in row | Ctrl+Alt+Right | Ctrl+Option+Right |
| Previous cell in row | Ctrl+Alt+Left | Ctrl+Option+Left |
| Next cell in column | Ctrl+Alt+Down | Ctrl+Option+Down |
| Previous cell in column | Ctrl+Alt+Up | Ctrl+Option+Up |

### What Gets Announced

When a user enters a table:
- "Table with X rows and Y columns"
- Caption text

When moving between cells:
- Column header (when entering a new column)
- Row header (when entering a new row)
- Cell content
- Cell position ("Row X, Column Y")

This navigation **only works** when `<th>` elements with `scope` (or `headers`) are correctly coded. Without them, the screen reader cannot announce headers.

### `aria-sort` for Sortable Columns

```html
<thead>
  <tr>
    <th scope="col" aria-sort="ascending">
      <button>Name</button>
    </th>
    <th scope="col">
      <button>Score</button>
    </th>
  </tr>
</thead>
```

- `aria-sort` goes **only** on the currently sorted column
- Values: `ascending`, `descending`, `none`, `other`
- Other sortable columns should have **no** `aria-sort` attribute (not even `none`)
- Screen readers announce "sorted ascending" (or similar) when the header is encountered

## Tables vs. CSS Grid/Flexbox

### Use `<table>` When:

- Data has a logical relationship between rows and columns
- Users need to compare values across rows or down columns
- Screen reader table navigation is needed (Ctrl+Alt+Arrow keys)
- Examples: financial data, schedules, statistics, comparison charts, spreadsheet data

### Use CSS Grid/Flexbox When:

- Arranging layout components (cards, sidebars, navigation)
- Content is not semantically tabular
- Visual grid does not represent data relationships

### The Modern Anti-Pattern

The old mistake was using `<table>` for page layout. The **new** mistake is using `<div>` with CSS Grid for tabular data. If data belongs in a spreadsheet, it belongs in a `<table>`. CSS Grid on `<div>` elements conveys zero semantic information about data relationships.

```html
<!-- BAD: tabular data in divs -->
<div class="grid">
  <div class="header">Name</div>
  <div class="header">Score</div>
  <div>Alice</div>
  <div>95</div>
</div>

<!-- GOOD: tabular data in a table -->
<table>
  <thead>
    <tr>
      <th scope="col">Name</th>
      <th scope="col">Score</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Alice</td>
      <td>95</td>
    </tr>
  </tbody>
</table>
```

## CSS Styling

### `border-collapse`

```css
/* Cells share borders (most common for data tables) */
table { border-collapse: collapse; }

/* Cells have separate borders with gaps */
table { border-collapse: separate; }
```

With `collapse`, adjacent cells merge into a single border. The wider or more specific border wins conflicts. With `separate` (the default), each cell has independent borders.

### `border-spacing`

Only works with `border-collapse: separate`. Ignored when collapsed.

```css
table {
  border-collapse: separate;
  border-spacing: 0;        /* no gaps between cells */
  border-spacing: 8px;      /* uniform spacing */
  border-spacing: 12px 4px; /* horizontal, vertical */
}
```

### `table-layout`

```css
/* auto: column widths based on ALL content (slow for large tables) */
table { table-layout: auto; }

/* fixed: column widths based on first row only (fast, predictable) */
table { table-layout: fixed; width: 100%; }
```

| Mode | Width Calculation | Rendering Speed | Overflow |
|---|---|---|---|
| `auto` | Scans all rows to determine widths | Slow (entire table must download first) | Content always fits |
| `fixed` | Uses first row (or `<col>` widths) only | Fast (renders as first row arrives) | Content may overflow; use `overflow: hidden` or `text-overflow` |

`table-layout: fixed` requires an explicit `width` on the table. Without it, the browser falls back to `auto`.

### `empty-cells`

Only works with `border-collapse: separate`.

```css
table {
  border-collapse: separate;
  empty-cells: hide; /* hides borders and backgrounds of empty cells */
}
```

A cell with only whitespace is NOT considered empty. It must have truly no content.

### `caption-side`

```css
table { caption-side: top; }    /* default */
table { caption-side: bottom; } /* visually below the table */
```

### Sticky Headers

```css
thead th {
  position: sticky;
  top: 0;
  background: #fff; /* required: content scrolls through without it */
  z-index: 1;
}
```

**Critical gotchas:**

1. `position: sticky` does **not** work on `<thead>` or `<tr>`. Apply it to `<th>` cells directly.
2. `border-collapse: collapse` **breaks sticky borders**. Collapsed borders disappear as cells scroll behind the sticky header.
3. **Workaround:** Use `border-collapse: separate; border-spacing: 0;` and simulate borders with `box-shadow`:

```css
table {
  border-collapse: separate;
  border-spacing: 0;
}

thead th {
  position: sticky;
  top: 0;
  background: #fff;
  box-shadow: inset 0 -2px 0 #d1d5db; /* simulates bottom border */
  z-index: 1;
}

td, th {
  border-bottom: 1px solid #e5e7eb;
  padding: 0.75rem 1rem;
}
```

### Sticky First Column

```css
td:first-child,
th:first-child {
  position: sticky;
  left: 0;
  background: #fff;
  z-index: 1;
}

/* Corner cell (sticky both ways) needs highest z-index */
thead th:first-child {
  z-index: 2;
}
```

## Responsive Table Patterns

### Pattern 1: Horizontal Scroll (Safest)

Wraps the table in a scrollable container. Preserves all table semantics and screen reader navigation.

```html
<div class="table-container" role="region" aria-labelledby="table-caption" tabindex="0">
  <table>
    <caption id="table-caption">Quarterly Revenue</caption>
    <!-- ... -->
  </table>
</div>
```

```css
.table-container {
  overflow-x: auto;
  -webkit-overflow-scrolling: touch;
}

/* Visual scroll indicator */
.table-container {
  background:
    linear-gradient(to right, white 30%, transparent),
    linear-gradient(to left, white 30%, transparent),
    linear-gradient(to right, rgba(0,0,0,0.1), transparent 20px),
    linear-gradient(to left, rgba(0,0,0,0.1), transparent 20px);
  background-position: left center, right center, left center, right center;
  background-repeat: no-repeat;
  background-size: 40px 100%, 40px 100%, 20px 100%, 20px 100%;
  background-attachment: local, local, scroll, scroll;
}
```

Add `tabindex="0"` and `role="region"` with `aria-labelledby` so keyboard users can scroll the container.

### Pattern 2: Priority Columns (Hide Non-Essential)

```html
<table>
  <thead>
    <tr>
      <th scope="col">Name</th>
      <th scope="col">Status</th>
      <th scope="col" class="priority-low">Created</th>
      <th scope="col" class="priority-low">Updated</th>
    </tr>
  </thead>
  <!-- ... -->
</table>
```

```css
@media (max-width: 600px) {
  .priority-low {
    display: none;
  }
}
```

Provide a toggle button to reveal hidden columns on demand.

### Pattern 3: Stacked Cards (Accessibility Warning)

```css
@media (max-width: 600px) {
  table, thead, tbody, th, td, tr {
    display: block;
  }

  thead {
    position: absolute;
    width: 1px;
    height: 1px;
    overflow: hidden;
    clip: rect(0, 0, 0, 0);
  }

  tr {
    margin-bottom: 1rem;
    border: 1px solid #e5e7eb;
    border-radius: 0.5rem;
    padding: 0.75rem;
  }

  td::before {
    content: attr(data-label);
    font-weight: 700;
    display: block;
    margin-bottom: 0.25rem;
  }
}
```

```html
<tr>
  <td data-label="Name">Alice</td>
  <td data-label="Score">95</td>
  <td data-label="Grade">A</td>
</tr>
```

**Warning:** Setting `display: block` on table elements **strips implicit ARIA roles** in some browsers. Screen reader table navigation breaks completely. Use this pattern only when the horizontal scroll approach is not feasible, and test with screen readers.

## Foster Parenting (Parser Behavior)

When the HTML parser encounters non-table content directly inside `<table>`, it moves it **before** the table in the DOM:

```html
<!-- What you write -->
<table>
  <p>Note: all values in USD.</p>
  <tr><td>100</td></tr>
</table>

<!-- What the DOM contains -->
<p>Note: all values in USD.</p>
<table>
  <tbody>
    <tr><td>100</td></tr>
  </tbody>
</table>
```

This applies to any non-table content: text nodes, `<p>`, `<div>`, `<span>`. Only whitespace-only text nodes are silently dropped instead of foster-parented.

If you need to add a note above a table, place it **outside** the `<table>` element intentionally, or use `<caption>`.

## Common Mistakes

1. **Using tables for page layout.** Destroys semantic meaning. Screen readers announce "table with X rows and Y columns" for every layout table. If you must support legacy layout tables, add `role="presentation"` to strip all table semantics.

2. **Using `<td>` for header cells.** Bold CSS styling does not make a cell a header. Screen readers cannot announce column/row context without `<th>`.

3. **Missing `scope` on `<th>`.** Without `scope`, screen readers must guess header direction. This works unreliably for simple tables and fails entirely for complex ones.

4. **Missing `<caption>`.** Tables without a caption have no accessible name. Screen readers announce "table" with no description of what the data represents.

5. **Building data tables with `<div>`.** The reverse of the layout-table mistake. CSS Grid on divs provides zero semantic table information.

6. **Mismatched `rowspan`/`colspan` cell counts.** Using `colspan="3"` but not reducing the number of other cells in the row. Or using `rowspan` but not removing cells from the spanned rows.

7. **CSS `table > tr` selector.** Never matches because of the implicit `<tbody>`. Use `table > tbody > tr` or just `table tr`.

8. **`border-collapse: collapse` with `position: sticky`.** Borders disappear on sticky headers. Use `border-collapse: separate; border-spacing: 0;` with `box-shadow` instead.

9. **Setting CSS properties on `<col>` that do not work.** Only `background`, `border` (collapsed), `width`, and `visibility` have any effect. Everything else is silently ignored.

10. **Responsive stacked cards without testing screen readers.** `display: block` on table elements strips ARIA roles. Table navigation breaks.

---

## Best Practices

- **Use `<table>` for tabular data only.** Not for layout, not for alignment, not for spacing.
- **Always include a `<caption>`.** Every data table needs an accessible name.
- **Use `<th>` with `scope` for all headers.** `scope="col"` for column headers, `scope="row"` for row headers.
- **Use `<thead>`, `<tbody>`, `<tfoot>`** for semantic grouping, print behavior, and sticky headers.
- **Always set `width` and `height` attributes** or explicit CSS dimensions to prevent CLS.
- **Use `table-layout: fixed`** for large tables to improve rendering speed.
- **Use horizontal scroll** as the primary responsive strategy. It preserves all table semantics.
- **Use `border-collapse: separate; border-spacing: 0;`** when combining with `position: sticky`.
- **Keep tables simple.** If a table requires deep `headers`/`id` markup, consider splitting it into multiple simpler tables.
