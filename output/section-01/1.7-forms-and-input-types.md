# 1.7 Forms and Input Types

> Forms are the primary mechanism for collecting user input on the web. Understanding form elements, input types, and the submission process is essential for building interactive applications.

---

## The `<form>` Element

The `<form>` element is a container that groups interactive controls for submitting data. It is not just a wrapper; it controls how data is collected, encoded, and sent.

### Key Attributes

| Attribute | Values | Purpose |
|-----------|--------|---------|
| `action` | URL | Where to send form data. Defaults to the current page URL if omitted. |
| `method` | `get`, `post`, `dialog` | HTTP method for submission. `get` appends data to the URL. `post` sends it in the request body. `dialog` closes a parent `<dialog>`. |
| `enctype` | `application/x-www-form-urlencoded`, `multipart/form-data`, `text/plain` | How form data is encoded before sending. Only applies to `method="post"`. |
| `novalidate` | Boolean | Disables built-in constraint validation on submission. |
| `autocomplete` | `on`, `off` | Hint for browsers to auto-fill fields. |
| `name` | String | Identifies the form. Used by `document.forms.name`. |
| `target` | `_self`, `_blank`, `_parent`, `_top` | Where to display the response. |
| `rel` | `noopener`, `noreferrer`, etc. | Relationship for forms with `target="_blank"`. |

```html
<form action="/api/register" method="post" enctype="multipart/form-data">
  <!-- Controls go here -->
</form>
```

### When to Use `multipart/form-data`

The default `application/x-www-form-urlencoded` encodes data as key=value pairs separated by `&`. This works for text data but corrupts binary data. You must use `multipart/form-data` when the form includes file uploads.

```html
<!-- BAD: file data will be corrupted -->
<form method="post">
  <input type="file" name="avatar">
  <button type="submit">Upload</button>
</form>

<!-- GOOD: binary data preserved -->
<form method="post" enctype="multipart/form-data">
  <input type="file" name="avatar">
  <button type="submit">Upload</button>
</form>
```

### The `method="dialog"` Value

When a form is inside a `<dialog>`, using `method="dialog"` closes the dialog on submission without sending any HTTP request. The submit button's `value` becomes the dialog's `returnValue`.

```html
<dialog id="confirm">
  <form method="dialog">
    <p>Are you sure?</p>
    <button value="cancel">Cancel</button>
    <button value="confirm">Confirm</button>
  </form>
</dialog>

<script>
  const dialog = document.getElementById('confirm');
  dialog.showModal();
  dialog.addEventListener('close', () => {
    console.log(dialog.returnValue); // "cancel" or "confirm"
  });
</script>
```

## Form Submission Process

When a form is submitted, the browser performs these steps in order:

1. **Build the entry list** - Collects all submittable elements with a `name` attribute that are not disabled.
2. **Run constraint validation** - Unless `novalidate` is set, each field is checked against its validation constraints. If any field is invalid, submission is aborted and `invalid` events fire.
3. **Encode the data** - The entry list is encoded according to the `enctype`.
4. **Send the request** - An HTTP request is sent to the `action` URL using the specified `method`.

### Elements That Participate in Submission

Only elements with a `name` attribute are included in the submitted data. Elements without `name` are ignored entirely.

```html
<form action="/search" method="get">
  <!-- Included: has name -->
  <input type="text" name="query" value="html forms">

  <!-- NOT included: no name attribute -->
  <input type="text" value="this is ignored">

  <!-- NOT included: disabled -->
  <input type="text" name="filter" value="recent" disabled>

  <!-- Included: readonly fields are still submitted -->
  <input type="text" name="source" value="web" readonly>
</form>
<!-- Submits as: /search?query=html+forms&source=web -->
```

### Implicit Submission

Pressing Enter in a single-line input triggers form submission if the form has a submit button. This is called implicit submission. If a form has no submit button, pressing Enter does nothing.

```html
<!-- Enter submits this form -->
<form>
  <input type="text" name="search">
  <button type="submit">Go</button>
</form>

<!-- Enter does NOT submit this form (no submit button) -->
<form>
  <input type="text" name="search">
  <button type="button">Not a submit button</button>
</form>
```

## Text Input Types

### `type="text"`

The default input type. Accepts any text.

```html
<input type="text" name="username" placeholder="Enter username"
       maxlength="30" minlength="3" size="20">
```

### `type="password"`

Obscures the entered text. Browsers may offer to save the password. The value is still sent as plain text over the network, so always use HTTPS.

```html
<input type="password" name="pass" autocomplete="new-password"
       minlength="8" required>
```

The `autocomplete` value matters here:
- `current-password` - for login forms (autofill existing password)
- `new-password` - for registration/change forms (generate a new password)

### `type="email"`

Validates that the input contains an email address. On mobile, brings up a keyboard with `@` and `.` keys.

```html
<!-- Single email -->
<input type="email" name="email" required>

<!-- Multiple emails (comma-separated) -->
<input type="email" name="recipients" multiple>
```

The built-in validation accepts `user@domain` without a TLD. If you need stricter validation, add a `pattern` attribute or use server-side validation.

### `type="url"`

Validates that the input is a valid URL. Requires the protocol (e.g., `https://`). Mobile keyboards include `/` and `.com` keys.

```html
<input type="url" name="website" placeholder="https://example.com">
```

### `type="tel"`

Does not validate the format (phone number formats vary too much worldwide). Its primary purpose is triggering a numeric keyboard on mobile devices.

```html
<input type="tel" name="phone" pattern="[0-9]{3}-[0-9]{3}-[0-9]{4}"
       placeholder="123-456-7890">
```

### `type="search"`

Functionally identical to `type="text"`, but may be styled differently by the browser (e.g., rounded corners on Safari, a clear button). Some screen readers announce it as a search field.

```html
<input type="search" name="q" placeholder="Search..."
       aria-label="Search the site">
```

## Numeric and Range Input Types

### `type="number"`

Provides a numeric input with optional spinner buttons. Values are always floating-point numbers.

```html
<input type="number" name="quantity" min="1" max="100" step="1" value="1">
```

| Attribute | Purpose |
|-----------|---------|
| `min` | Minimum allowed value |
| `max` | Maximum allowed value |
| `step` | Increment/decrement step. Use `"any"` for any decimal value. |
| `value` | Default value |

### `type="number"` vs `inputmode="numeric"`

This is one of the most misunderstood distinctions in HTML forms.

`type="number"` is for values that are mathematically numeric, where spinner buttons and step constraints make sense. It is wrong for values that happen to be digits but are not quantities: phone numbers, credit card numbers, zip codes, PINs.

```html
<!-- GOOD: actual quantity -->
<input type="number" name="quantity" min="1" max="99" step="1">

<!-- BAD: credit card number is not a mathematical quantity -->
<input type="number" name="card">
<!-- The value "4111111111111111" is too large for floating-point precision -->

<!-- GOOD: use text with inputmode for digit-only non-numeric fields -->
<input type="text" inputmode="numeric" pattern="[0-9]{16}" name="card"
       autocomplete="cc-number">
```

Problems with `type="number"` for non-quantity values:
- Leading zeros are stripped (`007` becomes `7`)
- Large numbers lose precision due to IEEE 754 floating point
- Spinner buttons are meaningless for IDs and codes
- Some browsers allow `e` (for exponent notation) as input
- `value` returns an empty string if the input contains non-numeric characters, even though the user typed something

### `type="range"`

Renders a slider control. Does not show the current value by default.

```html
<label for="volume">Volume</label>
<input type="range" id="volume" name="volume" min="0" max="100" step="5" value="50">
<output for="volume">50</output>

<script>
  const range = document.getElementById('volume');
  const output = document.querySelector('output[for="volume"]');
  range.addEventListener('input', () => {
    output.textContent = range.value;
  });
</script>
```

## Date and Time Input Types

These types provide native date/time pickers. Browser support and appearance vary significantly.

| Type | Format | Example Value |
|------|--------|---------------|
| `date` | `YYYY-MM-DD` | `2026-02-25` |
| `time` | `HH:MM` | `14:30` |
| `datetime-local` | `YYYY-MM-DDTHH:MM` | `2026-02-25T14:30` |
| `month` | `YYYY-MM` | `2026-02` |
| `week` | `YYYY-Www` | `2026-W09` |

```html
<input type="date" name="birthday" min="1900-01-01" max="2026-12-31">

<input type="time" name="alarm" min="06:00" max="22:00" step="900">
<!-- step="900" = 15-minute intervals (900 seconds) -->

<input type="datetime-local" name="meeting" value="2026-02-25T09:00">

<input type="month" name="expiry" min="2026-01">

<input type="week" name="sprint" value="2026-W09">
```

### Caveats

- `month` and `week` have no native picker in Firefox.
- `datetime-local` has no timezone information. It is a local date and time.
- The old `type="datetime"` (with timezone) was removed from the spec. Use `datetime-local` plus separate timezone handling.
- `step` for time-based inputs is in seconds. `step="60"` means 1-minute intervals, `step="3600"` means 1-hour intervals.
- `min` and `max` restrict the picker but a user can still type an out-of-range value. Validation catches it on submission.

## Color Input

Renders a color picker. The value is always a 7-character hex string like `#ff0000`.

```html
<input type="color" name="theme-color" value="#3498db">
```

It does not support alpha (transparency). If you need RGBA, use a custom color picker or a text input with validation.

## File Input

```html
<!-- Single file -->
<input type="file" name="document">

<!-- Multiple files -->
<input type="file" name="photos" multiple>

<!-- Accept specific types -->
<input type="file" name="avatar" accept="image/*">
<input type="file" name="data" accept=".csv,.xlsx">
<input type="file" name="video" accept="video/mp4,video/webm">
```

The `accept` attribute is a hint for the file picker dialog. It does not enforce the restriction; users can switch to "All Files" and select anything. Always validate the file type on the server.

### Accessing Selected Files

```javascript
const fileInput = document.querySelector('input[type="file"]');
fileInput.addEventListener('change', () => {
  for (const file of fileInput.files) {
    console.log(file.name);     // "photo.jpg"
    console.log(file.size);     // 204800 (bytes)
    console.log(file.type);     // "image/jpeg"
    console.log(file.lastModified); // 1708876800000 (timestamp)
  }
});
```

## Hidden Input

Carries data that the user does not see or interact with. Commonly used for CSRF tokens, IDs, or tracking values.

```html
<form action="/update" method="post">
  <input type="hidden" name="csrf_token" value="abc123xyz">
  <input type="hidden" name="user_id" value="42">
  <input type="text" name="display_name">
  <button type="submit">Update</button>
</form>
```

Hidden inputs are still visible in the page source and DevTools. Never use them for secrets on the client side.

## Checkbox and Radio

### `type="checkbox"`

Independent boolean controls. Each checked checkbox sends its `value` (or `"on"` if no `value` is specified). Unchecked checkboxes send nothing.

```html
<fieldset>
  <legend>Notifications</legend>
  <label>
    <input type="checkbox" name="notify_email" value="yes" checked>
    Email notifications
  </label>
  <label>
    <input type="checkbox" name="notify_sms" value="yes">
    SMS notifications
  </label>
</fieldset>
```

The fact that unchecked checkboxes are omitted from submission is a common source of bugs in server-side processing. The server cannot distinguish "unchecked" from "field not present."

### `type="radio"`

Mutually exclusive selection within a group. All radio buttons with the same `name` form a group. Exactly one can be selected at a time.

```html
<fieldset>
  <legend>Payment Method</legend>
  <label>
    <input type="radio" name="payment" value="credit" required>
    Credit Card
  </label>
  <label>
    <input type="radio" name="payment" value="debit">
    Debit Card
  </label>
  <label>
    <input type="radio" name="payment" value="paypal">
    PayPal
  </label>
</fieldset>
```

Place `required` on at least one radio button in the group. The browser will enforce that one must be selected before submission. There is no way to deselect a radio button without JavaScript (or providing an explicit "None" option).

## Button Types

### `type="submit"` (default)

Submits the form. If a `<button>` has no `type` attribute, it defaults to `submit`. This is a frequent source of unintended form submissions.

```html
<!-- These are identical -->
<button type="submit">Send</button>
<button>Send</button>

<!-- BAD: unintended submission inside a form -->
<form>
  <button onclick="doSomething()">Click me</button>
  <!-- This ALSO submits the form because type defaults to "submit" -->
</form>

<!-- GOOD: explicitly set type="button" for non-submit actions -->
<form>
  <button type="button" onclick="doSomething()">Click me</button>
</form>
```

### `type="reset"`

Resets all form controls to their initial values (their `value` attributes, not empty). Rarely used because it can accidentally erase user input.

```html
<button type="reset">Clear form</button>
```

### `type="button"`

Does nothing by default. Use it for buttons that trigger JavaScript without submitting the form.

```html
<button type="button" onclick="togglePreview()">Preview</button>
```

### `<button>` vs `<input type="submit">`

Both submit forms, but `<button>` can contain HTML content (icons, spans), while `<input>` can only display plain text via its `value` attribute.

```html
<!-- Limited to text only -->
<input type="submit" value="Send">

<!-- Can contain rich content -->
<button type="submit">
  <svg><!-- icon --></svg>
  Send Message
</button>
```

Always prefer `<button>` over `<input type="submit">`.

### Form Override Attributes on Buttons

Submit buttons can override the parent form's attributes for that specific submission.

```html
<form action="/save" method="post">
  <input type="text" name="data">

  <!-- Default: saves as draft -->
  <button type="submit">Save Draft</button>

  <!-- Override: submits to a different endpoint -->
  <button type="submit" formaction="/publish" formmethod="post">Publish</button>

  <!-- Override: opens in new tab with GET -->
  <button type="submit" formaction="/preview" formmethod="get"
          formtarget="_blank">Preview</button>

  <!-- Override: skip validation for this button -->
  <button type="submit" formnovalidate>Save without validation</button>
</form>
```

| Attribute | Overrides |
|-----------|-----------|
| `formaction` | `action` |
| `formmethod` | `method` |
| `formenctype` | `enctype` |
| `formtarget` | `target` |
| `formnovalidate` | `novalidate` |

## The `<select>` Element

Creates a dropdown menu or a scrollable list.

```html
<!-- Dropdown -->
<label for="country">Country</label>
<select id="country" name="country">
  <option value="">-- Select --</option>
  <option value="us">United States</option>
  <option value="gb">United Kingdom</option>
  <option value="de" selected>Germany</option>
</select>
```

### Option Groups

```html
<select name="car">
  <optgroup label="Swedish Cars">
    <option value="volvo">Volvo</option>
    <option value="saab">Saab</option>
  </optgroup>
  <optgroup label="German Cars">
    <option value="mercedes">Mercedes</option>
    <option value="audi">Audi</option>
  </optgroup>
</select>
```

`<optgroup>` elements cannot be nested. There is no native way to create multi-level grouped options.

### Multiple Selection

```html
<select name="skills" multiple size="6">
  <option value="html">HTML</option>
  <option value="css">CSS</option>
  <option value="js">JavaScript</option>
  <option value="ts">TypeScript</option>
  <option value="angular">Angular</option>
  <option value="react">React</option>
</select>
```

With `multiple`, users hold Ctrl/Cmd to select multiple options. The UX is poor; checkboxes are usually a better alternative for multi-select.

### `<option>` Value vs Text

If an `<option>` has no `value` attribute, its text content is used as the value. This is often undesirable because display text and submitted value should be independent.

```html
<!-- value="United States" is sent (the display text) -->
<option>United States</option>

<!-- value="us" is sent (the explicit value) -->
<option value="us">United States</option>
```

## The `<textarea>` Element

Multi-line text input. Unlike `<input>`, the value is placed between the opening and closing tags, not in a `value` attribute.

```html
<textarea name="bio" rows="5" cols="40" maxlength="500"
          placeholder="Tell us about yourself"></textarea>
```

### The `wrap` Attribute

| Value | Behavior |
|-------|----------|
| `soft` (default) | Text wraps visually but no line breaks are added to the submitted value. |
| `hard` | Line breaks are inserted into the submitted value where the text wraps. Requires `cols` to be set. |
| `off` | No wrapping. A horizontal scrollbar appears if the line exceeds the width. |

### Content Gotcha

Any whitespace between the tags becomes part of the value:

```html
<!-- BAD: value starts with a newline and spaces -->
<textarea name="notes">
  </textarea>

<!-- GOOD: no whitespace inside -->
<textarea name="notes"></textarea>
```

## The `<label>` Element

Labels associate descriptive text with a form control. This is critical for accessibility: screen readers announce the label when a user focuses the control, and clicking the label focuses/activates the control.

### Explicit Association

Uses the `for` attribute to match the control's `id`.

```html
<label for="email">Email address</label>
<input type="email" id="email" name="email">
```

### Implicit Association

Wraps the control inside the label. No `for` or `id` needed.

```html
<label>
  Email address
  <input type="email" name="email">
</label>
```

### Which to Use

Explicit association is more robust. Some assistive technologies have historically had issues with implicit labels. The explicit pattern also allows placing the label and input anywhere in the DOM (they do not have to be adjacent).

One label can only be associated with one control. If a control needs multiple labels, use `aria-labelledby` instead.

```html
<!-- BAD: label can only reference one control -->
<label for="first">Name</label>
<input id="first" name="first">
<input id="last" name="last">

<!-- GOOD: each input has its own label -->
<label for="first">First name</label>
<input id="first" name="first">
<label for="last">Last name</label>
<input id="last" name="last">
```

## The `<fieldset>` and `<legend>` Elements

`<fieldset>` groups related controls. `<legend>` provides a caption for the group. This pairing is essential for accessibility: screen readers announce the legend text when entering the group.

```html
<fieldset>
  <legend>Shipping Address</legend>
  <label for="street">Street</label>
  <input type="text" id="street" name="street">
  <label for="city">City</label>
  <input type="text" id="city" name="city">
  <label for="zip">Zip Code</label>
  <input type="text" id="zip" name="zip" inputmode="numeric">
</fieldset>
```

### Disabling Entire Groups

Setting `disabled` on a `<fieldset>` disables all controls inside it. This is the only way to disable a group of controls with a single attribute.

```html
<fieldset disabled>
  <legend>Payment (disabled until checkout)</legend>
  <input type="text" name="card">
  <input type="text" name="expiry">
  <input type="text" name="cvv">
</fieldset>
```

The `<legend>` element's styling is notoriously difficult. It has special rendering behavior that breaks normal CSS box model expectations (margin, padding, and border behave differently than on other elements).

## The `<output>` Element

Represents the result of a calculation or user action. It is associated with inputs via the `for` attribute.

```html
<form oninput="result.value = parseInt(a.value) + parseInt(b.value)">
  <input type="number" id="a" name="a" value="0"> +
  <input type="number" id="b" name="b" value="0"> =
  <output name="result" for="a b">0</output>
</form>
```

`<output>` has an implicit `aria-live="polite"` role, so changes to its content are announced by screen readers.

## The `<datalist>` Element

Provides autocomplete suggestions for an input. The user can type freely or pick from the list.

```html
<label for="browser">Browser</label>
<input type="text" id="browser" name="browser" list="browsers">
<datalist id="browsers">
  <option value="Chrome">
  <option value="Firefox">
  <option value="Safari">
  <option value="Edge">
</datalist>
```

`<datalist>` works with text, email, url, tel, number, range, date, time, datetime-local, month, week, and color inputs. For `type="range"`, it renders tick marks on the slider.

```html
<input type="range" min="0" max="100" step="25" list="quarters">
<datalist id="quarters">
  <option value="0" label="0%">
  <option value="25" label="25%">
  <option value="50" label="50%">
  <option value="75" label="75%">
  <option value="100" label="100%">
</datalist>
```

## The `inputmode` Attribute

Controls which virtual keyboard is shown on mobile devices. It does not validate or constrain input.

| Value | Keyboard |
|-------|----------|
| `none` | No keyboard. Use for custom input interfaces. |
| `text` | Standard text keyboard (default). |
| `decimal` | Numeric keyboard with decimal separator. |
| `numeric` | Numeric keyboard without decimal. |
| `tel` | Telephone keypad (0-9, *, #). |
| `search` | Search-optimized keyboard (may include a Search/Go button). |
| `email` | Text keyboard with `@` and `.` easily accessible. |
| `url` | Text keyboard with `/` and `.com` easily accessible. |

```html
<!-- Show numeric keyboard for a zip code field -->
<input type="text" inputmode="numeric" pattern="[0-9]{5}" name="zip">

<!-- Show keyboard with decimal point for currency -->
<input type="text" inputmode="decimal" name="price">
```

## The `enterkeyhint` Attribute

Customizes the label on the Enter key of virtual keyboards.

| Value | Typical Label |
|-------|---------------|
| `enter` | Enter / Return |
| `done` | Done |
| `go` | Go |
| `next` | Next |
| `previous` | Previous |
| `search` | Search |
| `send` | Send |

```html
<input type="text" name="search" enterkeyhint="search">
<input type="text" name="first_name" enterkeyhint="next">
<textarea name="message" enterkeyhint="send"></textarea>
```

This is purely cosmetic for mobile keyboards. It does not change the behavior of the Enter key.

## The `autocomplete` Attribute

Tells the browser what type of data a field expects, enabling autofill from stored user data.

### Common Values

| Value | Purpose |
|-------|---------|
| `name` | Full name |
| `given-name` | First name |
| `family-name` | Last name |
| `email` | Email address |
| `tel` | Phone number |
| `street-address` | Street address |
| `address-line1` | Address line 1 |
| `address-line2` | Address line 2 |
| `address-level2` | City |
| `address-level1` | State/Province |
| `postal-code` | Zip/postal code |
| `country` | Country code |
| `cc-number` | Credit card number |
| `cc-exp` | Card expiration date |
| `cc-csc` | Card security code |
| `new-password` | New password (triggers password generator) |
| `current-password` | Existing password |
| `one-time-code` | OTP/2FA code (triggers SMS autofill on mobile) |
| `username` | Username |

### Sectioning with Prefixes

You can prefix autocomplete values with `section-*` and `shipping`/`billing` to create separate autofill groups.

```html
<fieldset>
  <legend>Billing Address</legend>
  <input type="text" name="billing_street" autocomplete="billing street-address">
  <input type="text" name="billing_city" autocomplete="billing address-level2">
</fieldset>

<fieldset>
  <legend>Shipping Address</legend>
  <input type="text" name="shipping_street" autocomplete="shipping street-address">
  <input type="text" name="shipping_city" autocomplete="shipping address-level2">
</fieldset>
```

## The `form` Attribute on Controls

Any form control can be associated with a form that it is not nested inside, using the `form` attribute.

```html
<form id="checkout">
  <input type="text" name="item">
  <button type="submit">Buy</button>
</form>

<!-- This input is outside the form but still submitted with it -->
<input type="text" name="coupon" form="checkout" placeholder="Coupon code">
```

This is useful when the visual layout does not match the logical form structure. The control participates in the form's submission and validation as if it were inside the form.

## The `FormData` API

The `FormData` API extracts form data programmatically, matching the browser's native submission behavior.

```javascript
const form = document.querySelector('form');

form.addEventListener('submit', (event) => {
  event.preventDefault();

  const data = new FormData(form);

  // Iterate over entries
  for (const [key, value] of data) {
    console.log(`${key}: ${value}`);
  }

  // Get a single value
  console.log(data.get('email'));

  // Get all values for a multi-select or checkboxes
  console.log(data.getAll('skills'));

  // Convert to URLSearchParams (for GET-style queries)
  const params = new URLSearchParams(data);
  console.log(params.toString()); // "email=test%40example.com&skills=html&skills=css"

  // Send as fetch request
  fetch('/api/submit', {
    method: 'POST',
    body: data // automatically sets Content-Type to multipart/form-data
  });
});
```

### Building FormData Manually

```javascript
const data = new FormData();
data.append('username', 'alice');
data.append('avatar', fileInput.files[0]); // File object
data.append('tags', 'html');
data.append('tags', 'css'); // Multiple values for same key

fetch('/api/profile', { method: 'POST', body: data });
```

When passing `FormData` as the `body` of a `fetch` request, do not set the `Content-Type` header manually. The browser sets it to `multipart/form-data` with the correct boundary string automatically. Manually setting it will break the request.

## Accessibility for Forms

### Descriptive Labels Are Non-Negotiable

Every form control must have an accessible name. The primary mechanism is `<label>`. For cases where a visible label is not part of the design, use `aria-label` or `aria-labelledby`.

```html
<!-- GOOD: visible label -->
<label for="search">Search</label>
<input type="search" id="search" name="q">

<!-- ACCEPTABLE: visually hidden label -->
<label for="search" class="visually-hidden">Search</label>
<input type="search" id="search" name="q">

<!-- ACCEPTABLE: aria-label when no visible label exists -->
<input type="search" name="q" aria-label="Search the site">

<!-- BAD: placeholder is NOT a label replacement -->
<input type="search" name="q" placeholder="Search...">
```

### Error Messages with `aria-describedby`

Link error messages to their controls so screen readers announce the error when the field is focused.

```html
<label for="email">Email</label>
<input type="email" id="email" name="email"
       aria-describedby="email-error" aria-invalid="true">
<span id="email-error" role="alert">Please enter a valid email address.</span>
```

### Required Fields

The `required` attribute is announced by screen readers. You can also visually indicate required fields, but do not rely solely on color or asterisks.

```html
<label for="name">Name <span aria-hidden="true">*</span></label>
<input type="text" id="name" name="name" required>
```

## Common Mistakes

- **Forgetting `type="button"` on non-submit buttons.** Inside a form, any `<button>` without an explicit `type` defaults to `submit`. This causes accidental form submissions.

- **Using `type="number"` for non-quantity digit strings.** Credit card numbers, zip codes, phone numbers, and PINs are not mathematical quantities. Use `type="text"` with `inputmode="numeric"` instead.

- **Missing `name` attributes.** Controls without `name` are not included in the submitted data. The form appears to work but the server receives no data.

- **Missing labels.** Using `placeholder` as a label is an accessibility failure. Placeholder text disappears when the user starts typing, and most screen readers do not reliably treat it as a label.

- **Forgetting `enctype="multipart/form-data"` for file uploads.** Without it, only the filename is sent, not the file content.

- **Manually setting `Content-Type` with `FormData` in `fetch`.** The browser sets the correct boundary; overriding it corrupts the request.

- **Not handling unchecked checkboxes on the server.** Unchecked checkboxes are simply absent from the submitted data. The server must handle missing keys, not just check for `"on"` or `"off"`.

---

## Best Practices

- **Always use semantic form elements.** Use `<form>`, `<fieldset>`, `<legend>`, and `<label>` to give structure and meaning to your forms.

- **Set explicit `type` on every `<button>` inside a form.** Never rely on the default `type="submit"`.

- **Use `autocomplete` values generously.** Correct autocomplete values dramatically improve user experience by enabling browser autofill.

- **Use `inputmode` for mobile keyboards.** Pair `type="text"` with `inputmode` to show the right keyboard without the validation constraints of specialized types.

- **Group related controls with `<fieldset>` and `<legend>`.** This is the only way to semantically group radio buttons and checkboxes for screen readers.

- **Use the `form` attribute to associate disconnected controls.** When layout constraints prevent nesting a control inside its form, use the `form` attribute rather than restructuring the DOM.

- **Validate on the server.** Client-side validation improves UX but can always be bypassed. Never trust client-side validation alone.

## Real-World Use Cases

### Registration Form

```html
<form action="/api/register" method="post">
  <fieldset>
    <legend>Account Details</legend>

    <label for="reg-email">Email</label>
    <input type="email" id="reg-email" name="email"
           autocomplete="email" required>

    <label for="reg-pass">Password</label>
    <input type="password" id="reg-pass" name="password"
           autocomplete="new-password" minlength="8" required
           aria-describedby="pass-hint">
    <p id="pass-hint">Minimum 8 characters.</p>
  </fieldset>

  <fieldset>
    <legend>Profile</legend>

    <label for="reg-name">Full Name</label>
    <input type="text" id="reg-name" name="full_name"
           autocomplete="name" required>

    <label for="reg-phone">Phone (optional)</label>
    <input type="tel" id="reg-phone" name="phone"
           autocomplete="tel">
  </fieldset>

  <button type="submit">Create Account</button>

```

### Payment Form with Proper Input Types

```html
<form action="/api/pay" method="post">
  <fieldset>
    <legend>Card Information</legend>

    <label for="card-number">Card Number</label>
    <input type="text" id="card-number" name="card_number"
           inputmode="numeric" pattern="[0-9]{13,19}"

    <input type="text" id="card-cvc" name="card_cvc"
           inputmode="numeric" pattern="[0-9]{3,4}"

  <label for="amount">Amount</label>
  <input type="number" id="amount" name="amount"
         min="1" step="0.01" required>

```

### Multi-Step Survey with `method="dialog"`

```html
<dialog id="survey">
  <form method="dialog">
    <fieldset>
      <legend>How did you hear about us?</legend>
      <label><input type="radio" name="source" value="search"> Search engine</label>
      <label><input type="radio" name="source" value="social"> Social media</label>
      <label><input type="radio" name="source" value="friend"> A friend</label>
      <label><input type="radio" name="source" value="other"> Other</label>
    </fieldset>

    <label for="feedback">Additional feedback</label>
    <textarea id="feedback" name="feedback" rows="4"></textarea>

    <button type="button" onclick="document.getElementById('survey').close('skip')">
      Skip
    </button>
    <button value="submitted">Submit</button>
  </form>
</dialog>

<script>
  const dialog = document.getElementById('survey');
  dialog.showModal();
  dialog.addEventListener('close', () => {
    if (dialog.returnValue === 'submitted') {
      const data = new FormData(dialog.querySelector('form'));
      fetch('/api/survey', { method: 'POST', body: data });
    }
  });
</script>
```

> **Key Takeaway:** Choose the right input type for the job: `type="number"` for quantities, `inputmode="numeric"` for digit strings, always pair controls with `<label>`, and use `<fieldset>`/`<legend>` for grouping. The difference between a usable form and a frustrating one lies in these semantic choices.
