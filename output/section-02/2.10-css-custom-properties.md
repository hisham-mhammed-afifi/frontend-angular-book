# 2.10 CSS Custom Properties (Variables)

> CSS custom properties are runtime variables that participate in the cascade, inherit through the DOM, and can be manipulated with JavaScript. Combined with `@property`, they unlock typed values, animation, and performance optimization.

---

## Declaration and Usage

Custom properties are declared with a `--` prefix and consumed with `var()`:

```css
:root {
  --brand-color: #3b82f6;
  --spacing: 16px;
  --font-stack: "Inter", system-ui, sans-serif;
}

.card {
  background: var(--brand-color);
  padding: var(--spacing);
  font-family: var(--font-stack);
}
```

They can hold any valid token stream: colors, lengths, strings, shorthand fragments, even entire property values with commas.

---

## Scope and Inheritance

Custom properties are **dynamically scoped** through the DOM tree, not lexically scoped like preprocessor variables. They inherit from parent to child, just like `color` or `font-family`.

### Global Scope

Declaring on `:root` makes a property available everywhere:

```css
:root {
  --accent: #3b82f6;
}

/* Available to every element in the document */
.button { background: var(--accent); }
.link   { color: var(--accent); }
```

### Local Scope

Declaring on a specific selector limits the property to that element and its descendants:

```css
.sidebar {
  --accent: #10b981;  /* Only .sidebar and children see this */
}

.sidebar .button {
  background: var(--accent);  /* #10b981 */
}

.main .button {
  background: var(--accent);  /* #3b82f6, from :root */
}
```

### Cascade Resolution

Custom properties participate in specificity and the cascade like any other property:

```css
:root       { --color: blue; }    /* specificity: 0-1-0 */
.card       { --color: green; }   /* specificity: 0-1-0, wins by source order */
#hero .card { --color: red; }     /* specificity: 1-1-0, wins by specificity */
```

### Contextual Theming

Because custom properties inherit through the DOM, components automatically adapt to their context:

```css
.theme-light { --bg: #ffffff; --text: #1a1a1a; }
.theme-dark  { --bg: #0f172a; --text: #e2e8f0; }

/* This component needs no modification to work in either theme */
.card {
  background: var(--bg);
  color: var(--text);
}
```

```html
<div class="theme-light">
  <div class="card">Light card</div>
</div>
<div class="theme-dark">
  <div class="card">Dark card</div>
</div>
```

## Fallback Values

`var()` accepts a second argument as a fallback:

```css
.element {
  color: var(--text-color, #333);
}
```

### Nested Fallbacks (Custom Property Stacks)

Fallbacks can themselves contain `var()`, creating a chain:

```css
.button {
  /* Component-level -> Theme-level -> System default */
  background: var(--button-bg, var(--action-color, var(--brand-color, #0066cc)));
}
```

This lets component authors provide customization hooks while maintaining sensible defaults. A consumer can set `--button-bg` to override just the button, or `--action-color` to affect all interactive elements.

### Comma-Separated Fallbacks

Everything after the first comma is the fallback, including additional commas:

```css
.heading {
  /* Fallback is: Georgia, "Times New Roman", serif */
  font-family: var(--heading-font, Georgia, "Times New Roman", serif);
}
```

### The `initial` Trick

Setting a custom property to `initial` makes it guaranteed-invalid, which triggers the fallback in `var()`:

```css
.card {
  --card-shadow: initial;  /* guaranteed-invalid */
  box-shadow: var(--card-shadow, 0 2px 8px rgba(0,0,0,0.1));
  /* Fallback is used */
}
```

This is useful for conditionally "disabling" a custom property so a downstream fallback takes effect.

## Invalid at Computed-Value Time (IACVT)

This is the single most important gotcha of custom properties. Understanding it prevents hours of debugging.

### The Problem

When CSS is parsed, the browser does not know where a custom property will be used, so it accepts nearly anything as valid:

```css
:root {
  --accent: 42px;  /* Valid as a custom property value */
}
```

The actual validation happens when `var()` is substituted:

```css
.text {
  color: red;
  color: var(--accent);  /* 42px substituted, invalid for color */
}
```

You might expect the result to be `red` (the previous declaration). It is not. Here is what happens:

1. The cascade sees two `color` declarations and keeps the last one (`var(--accent)`) because it has equal specificity and comes later.
2. At computed-value time, `42px` is substituted for `var(--accent)`.
3. `42px` is invalid for the `color` property.
4. The declaration becomes **IACVT** (Invalid At Computed-Value Time).
5. The property resolves to `unset`, which means the inherited value (not `red`).

The first `color: red` was already discarded during cascade resolution. It is gone.

### IACVT vs. Regular Invalid Declarations

```css
/* Regular invalid: discarded at parse time, red is used */
.text {
  color: red;
  color: 42px;       /* parser discards this immediately */
  /* Result: red */
}

/* IACVT: var() kept by cascade, substitution fails */
.text {
  color: red;
  color: var(--size); /* --size is 42px */
  /* Result: unset (inherited or initial), NOT red */
}
```

### Key Rule

A var() fallback only triggers when the custom property is **undefined** or **guaranteed-invalid** (from `initial`, circular references, or missing definitions). If the property has a defined value that happens to be the wrong type, the fallback is NOT used.

```css
:root {
  --accent: 42px;  /* defined, but wrong type for color */
}

.text {
  /* Fallback "red" is NOT used because --accent IS defined */
  color: var(--accent, red);
  /* Result: IACVT -> unset */
}
```

## The `@property` Rule

`@property` registers a custom property with a type, initial value, and inheritance behavior. This is part of the CSS Houdini specifications and reached Baseline in July 2024 (Chrome 85+, Safari 16.4+, Firefox 128+).

### Syntax

```css
@property --hue {
  syntax: "<number>";
  inherits: true;
  initial-value: 200;
}

@property --card-radius {
  syntax: "<length>";
  inherits: false;
  initial-value: 8px;
}

@property --gradient-color {
  syntax: "<color>";
  inherits: false;
  initial-value: #3b82f6;
}
```

Three descriptors are required:
- **`syntax`**: The type. Determines how the browser validates and interpolates the value.
- **`inherits`**: Whether the property inherits through the DOM (`true` or `false`).
- **`initial-value`**: The default value. Required unless syntax is `*`.

### Available Syntax Types

| Type | Description | Example Values |
|------|-------------|----------------|
| `<color>` | Any CSS color | `#ff0000`, `oklch(50% 0.2 250)` |
| `<length>` | Length with unit | `16px`, `2rem` |
| `<number>` | Any number | `1.5`, `42` |
| `<integer>` | Whole number | `0`, `3` |
| `<percentage>` | Percentage | `50%`, `100%` |
| `<length-percentage>` | Length or percentage | `16px`, `50%` |
| `<angle>` | Angle | `45deg`, `0.5turn` |
| `<time>` | Duration | `300ms`, `1s` |
| `<image>` | Image value | `url(...)`, `linear-gradient(...)` |
| `<transform-function>` | Single transform | `rotate(45deg)` |
| `<transform-list>` | Transform chain | `rotate(45deg) scale(2)` |
| `<custom-ident>` | Custom identifier | Any keyword |
| `*` | Universal (any token stream) | Anything |

Combinators: `<length> | <percentage>` for alternatives. `<length>+` for space-separated lists. `<color>#` for comma-separated lists.

### initial-value Constraints

The initial-value must be **computationally independent**. It cannot use relative units that depend on context:

```css
/* Valid initial-values */
@property --size { syntax: "<length>"; inherits: false; initial-value: 16px; }
@property --angle { syntax: "<angle>"; inherits: false; initial-value: 0deg; }
@property --color { syntax: "<color>"; inherits: false; initial-value: black; }

/* INVALID initial-values (depend on context) */
/* initial-value: 2em;   -- depends on parent font-size */
/* initial-value: 50%;   -- depends on containing block */
/* initial-value: 3rem;  -- depends on root font-size */
```

### Type Safety

When a typed property receives an invalid value, it silently falls back to the initial-value:

```css
@property --radius {
  syntax: "<length>";
  inherits: false;
  initial-value: 8px;
}

.card {
  --radius: banana;  /* Invalid for <length> */
  border-radius: var(--radius);  /* Uses initial-value: 8px */
}
```

No error appears in the console. This can make debugging difficult.

## Animating Custom Properties

Without `@property`, custom properties are stored as strings. The browser cannot interpolate between strings, so transitions and animations do not work:

```css
/* Does NOT animate -- custom property is a string */
:root { --color: #3b82f6; }
.box {
  background: var(--color);
  transition: --color 0.3s;  /* no effect */
}
.box:hover { --color: #ef4444; }
```

### Registering Enables Animation

```css
@property --color {
  syntax: "<color>";
  inherits: false;
  initial-value: #3b82f6;
}

.box {
  background: var(--color);
  transition: --color 0.3s;  /* NOW it works */
}
.box:hover { --color: #ef4444; }
```

### Gradient Animation

This is the killer use case. You cannot normally animate gradients because they are `<image>` values with no defined interpolation. With `@property`, you animate the individual stops:

```css
@property --gradient-angle {
  syntax: "<angle>";
  inherits: false;
  initial-value: 0deg;
}

.animated-gradient {
  background: linear-gradient(var(--gradient-angle), #3b82f6, #8b5cf6);
  animation: spin-gradient 3s linear infinite;
}

@keyframes spin-gradient {
  to { --gradient-angle: 360deg; }
}
```

### Animating Gradient Color Stops

```css
@property --stop-1 {
  syntax: "<color>";
  inherits: false;
  initial-value: #3b82f6;
}

@property --stop-2 {
  syntax: "<color>";
  inherits: false;
  initial-value: #8b5cf6;
}

.hero {
  background: linear-gradient(135deg, var(--stop-1), var(--stop-2));
  transition: --stop-1 0.6s, --stop-2 0.6s;
}

.hero:hover {
  --stop-1: #ef4444;
  --stop-2: #f97316;
}
```

### Hue Cycle Animation

```css
@property --hue {
  syntax: "<number>";
  inherits: false;
  initial-value: 0;
}

.rainbow {
  background: hsl(var(--hue) 80% 60%);
  animation: hue-cycle 5s linear infinite;
}

@keyframes hue-cycle {
  to { --hue: 360; }
}
```

## Disabling Inheritance with @property

By default, custom properties inherit. `@property` with `inherits: false` changes this:

```css
@property --panel-bg {
  syntax: "<color>";
  inherits: false;
  initial-value: white;
}

.sidebar {
  --panel-bg: #f0f0f0;  /* Only .sidebar gets this */
}

.sidebar .card {
  background: var(--panel-bg);  /* Gets initial-value: white, NOT #f0f0f0 */
}
```

This is valuable for component-scoped properties that should not leak to children.

### Performance Impact

Properties with `inherits: false` are significantly faster when updated. The browser does not need to propagate changes through the subtree. Benchmarks from web.dev show an **848% performance improvement** for non-inheriting properties over inheriting ones during animation.

Always set `inherits: false` on properties used in animations.

## JavaScript Interaction

### Reading Custom Properties

```javascript
const root = document.documentElement;
const brand = getComputedStyle(root).getPropertyValue('--brand-color').trim();
// Always .trim() -- getPropertyValue returns whitespace padding
```

### Writing Custom Properties

```javascript
// On :root (global)
document.documentElement.style.setProperty('--brand-color', '#ff6600');

// On a specific element
const card = document.querySelector('.card');
card.style.setProperty('--card-elevation', '3');
```

### Removing Custom Properties

```javascript
card.style.removeProperty('--card-elevation');
// Property reverts to inherited or initial value
```

### Theme Switching

```javascript
function setTheme(theme) {
  const root = document.documentElement;
  const themes = {
    light: { '--bg': '#ffffff', '--text': '#1a1a1a', '--accent': '#3b82f6' },
    dark:  { '--bg': '#0f172a', '--text': '#e2e8f0', '--accent': '#60a5fa' },
  };
  for (const [prop, value] of Object.entries(themes[theme])) {
    root.style.setProperty(prop, value);
  }
}
```

### Registering via JavaScript

```javascript
CSS.registerProperty({
  name: '--progress',
  syntax: '<percentage>',
  inherits: false,
  initialValue: '0%',
});
```

`CSS.registerProperty()` throws if called twice for the same property name. Prefer `@property` in CSS for easier management, or wrap in try/catch.

## Partial Values and Unit Appending

Custom properties can store partial values, not just complete property values.

### Unit Appending with calc()

Direct concatenation does not work. CSS does not support string concatenation:

```css
:root {
  --size: 16;
}

.box {
  /* BAD: produces "16 px" (space between), which is invalid */
  width: var(--size)px;

  /* GOOD: multiply by 1 of the desired unit */
  width: calc(var(--size) * 1px);
}
```

### Spacing Scale from a Unitless Base

```css
:root {
  --space: 8;
}

.card {
  padding: calc(var(--space) * 2 * 1px);   /* 16px */
  margin-bottom: calc(var(--space) * 3 * 1px); /* 24px */
  gap: calc(var(--space) * 1px);            /* 8px */
}
```

### HSL Channel Decomposition

Store individual color channels as separate properties for fine-grained manipulation:

```css
:root {
  --primary-h: 220;
  --primary-s: 90%;
  --primary-l: 56%;
}

.button {
  background: hsl(var(--primary-h) var(--primary-s) var(--primary-l));
}

.button:hover {
  /* Darken: only change lightness */
  background: hsl(var(--primary-h) var(--primary-s) calc(var(--primary-l) - 10%));
}
```

Note: With modern CSS, `color-mix()` and relative color syntax (covered in 2.9) often replace this pattern.

### Shorthand Fragments

```css
:root {
  --shadow-color: 220 3% 15%;
  --shadow-strength: 1%;
}

.card {
  box-shadow:
    0 1px 2px hsl(var(--shadow-color) / calc(var(--shadow-strength) + 9%)),
    0 2px 4px hsl(var(--shadow-color) / calc(var(--shadow-strength) + 6%)),
    0 4px 8px hsl(var(--shadow-color) / calc(var(--shadow-strength) + 3%));
}
```

## The Space Toggle Trick

CSS allows a single space as a valid custom property value: `--flag: ;` (one space character). Combined with `initial` (guaranteed-invalid), this enables a form of boolean logic in pure CSS.

### How It Works

```css
:root {
  --dark: ;         /* space = "on" */
  --light: initial; /* guaranteed-invalid = "off" */
}

body {
  /* When --dark is a space, "black" is kept (space is invisible) */
  /* When --dark is initial, fallback "white" is used */
  background: var(--dark, white) var(--light, white);
  color: var(--dark, #e0e0e0) var(--light, #1a1a1a);
}

@media (prefers-color-scheme: light) {
  :root {
    --dark: initial;
    --light: ;
  }
}
```

### Numeric Boolean Approach

A simpler variant uses 0 and 1 with `calc()`:

```css
:root {
  --is-large: 0;  /* 0 = false, 1 = true */
}

@media (min-width: 1024px) {
  :root { --is-large: 1; }
}

.element {
  font-size: calc(16px + var(--is-large) * 8px);
  /* 16px when small, 24px when large */

  padding: calc(8px + var(--is-large) * 8px);
  /* 8px when small, 16px when large */
}
```

### Future: CSS if()

The CSS `if()` function is in development (Chrome 137+ behind a flag). It will replace the space toggle trick:

```css
/* Future syntax -- not production-ready */
property: if(style(--theme: dark), #000, #fff);
```

As of early 2026, this is Chromium-only and experimental. Do not use in production.

## CSS Custom Properties vs. SASS Variables

| Feature | CSS Custom Properties | SASS Variables |
|---------|----------------------|----------------|
| Resolved at | Runtime (browser) | Compile time (build) |
| Cascade-aware | Yes | No |
| DOM inheritance | Yes | No (lexical scope) |
| JS accessible | Yes | No |
| Responsive (media queries) | Yes (values change at runtime) | No (compiled to static) |
| Type checking | With `@property` | With SASS type functions |
| Math | `calc()` only | Full SASS math |
| Loops/conditionals | Limited (space toggle) | Full `@if`, `@for`, `@each` |
| String concatenation | No | Yes |
| Build step required | No | Yes |

### When to Use Each

**SASS variables** for build-time constants: breakpoint values, z-index scales, utility class generation, values that never change at runtime.

**CSS custom properties** for runtime flexibility: theming, dark mode, user preferences, component customization, responsive adjustments, anything JavaScript needs to read or write.

### Use Both Together

The most common production pattern: SASS generates CSS custom properties.

```scss
$colors: (
  "primary": #3b82f6,
  "secondary": #8b5cf6,
  "danger": #ef4444,
);

:root {
  @each $name, $value in $colors {
    --color-#{$name}: #{$value};
  }
}
```

## Design Tokens with Custom Properties

### Three-Layer Architecture

```css
/* Layer 1: Primitive tokens (raw scale values) */
:root {
  --color-blue-50: #eff6ff;
  --color-blue-100: #dbeafe;
  --color-blue-500: #3b82f6;
  --color-blue-700: #1d4ed8;
  --color-blue-900: #1e3a5a;

  --space-1: 4px;
  --space-2: 8px;
  --space-3: 12px;
  --space-4: 16px;
  --space-6: 24px;
  --space-8: 32px;

  --radius-sm: 4px;
  --radius-md: 8px;
  --radius-lg: 16px;
}

/* Layer 2: Semantic tokens (purpose-driven) */
:root {
  --color-bg: var(--color-blue-50);
  --color-text: var(--color-blue-900);
  --color-action: var(--color-blue-500);
  --color-action-hover: var(--color-blue-700);

  --space-inline: var(--space-4);
  --space-stack: var(--space-2);
  --radius-interactive: var(--radius-md);
}

/* Layer 3: Component tokens */
.button {
  --button-bg: var(--color-action);
  --button-bg-hover: var(--color-action-hover);
  --button-radius: var(--radius-interactive);
  --button-padding-x: var(--space-inline);
  --button-padding-y: var(--space-2);

  background: var(--button-bg);
  border-radius: var(--button-radius);
  padding: var(--button-padding-y) var(--button-padding-x);
}

.button:hover {
  background: var(--button-bg-hover);
}
```

### Theme Switching at the Semantic Layer

```css
/* Dark theme: only override semantic tokens */
[data-theme="dark"] {
  --color-bg: #0f172a;
  --color-text: #e2e8f0;
  --color-action: #60a5fa;
  --color-action-hover: #93bbfd;
}
```

Primitive tokens never change. Semantic tokens change per theme. Component tokens reference semantic tokens and adapt automatically.

### Naming Convention

Use the `category-attribute-variant` pattern:

```
--color-text-primary
--color-text-secondary
--color-bg-surface
--font-size-lg
--font-weight-bold
--space-inline-md
--shadow-elevation-2
--radius-interactive
```

## Responsive Patterns

Custom properties change at runtime, making them ideal for responsive design:

```css
:root {
  --content-width: 90%;
  --gutter: 16px;
  --columns: 1;
}

@media (min-width: 768px) {
  :root {
    --content-width: 80%;
    --gutter: 24px;
    --columns: 2;
  }
}

@media (min-width: 1200px) {
  :root {
    --content-width: 1140px;
    --gutter: 32px;
    --columns: 3;
  }
}

.grid {
  display: grid;
  grid-template-columns: repeat(var(--columns), 1fr);
  gap: var(--gutter);
  max-width: var(--content-width);
  margin: 0 auto;
}
```

Change the properties once; every usage updates automatically.

## Performance Considerations

### General Performance

Custom properties at normal scale are fast. Benchmarks show 5,000 custom properties on 10,000 elements add only ~0.7% overhead compared to hardcoded values. Not a concern for typical applications.

### When Performance Matters

1. **Updating `:root` properties triggers full-document recalculation.** Scope property changes to the narrowest possible element:

   ```css
   /* Bad: triggers recalc for entire document */
   /* document.documentElement.style.setProperty('--card-bg', 'red'); */

   /* Good: only .card and descendants recalculate */
   /* card.style.setProperty('--card-bg', 'red'); */
   ```

2. **`inherits: false` is critical for animation.** Non-inheriting properties are 8x faster because the browser skips subtree propagation:

   ```css
   /* Fast: change is scoped to the element */
   @property --rotation {
     syntax: "<angle>";
     inherits: false;
     initial-value: 0deg;
   }
   ```

3. **Batch JavaScript updates.** Multiple `setProperty()` calls trigger multiple style recalculations. Group them:

   ```javascript
   // Batch updates in a single frame
   requestAnimationFrame(() => {
     root.style.setProperty('--bg', newBg);
     root.style.setProperty('--text', newText);
     root.style.setProperty('--accent', newAccent);
   });
   ```

## Common Mistakes

1. **Expecting fallback on type mismatch.** If `--accent` is `42px` and you use `color: var(--accent, red)`, the fallback `red` is NOT used. The property substitutes `42px`, finds it invalid for `color`, and becomes `unset`. The fallback only triggers when the property is undefined or guaranteed-invalid.

2. **Using custom properties in media query conditions.** `@media (min-width: var(--bp))` is invalid and silently ignored. Custom properties cannot appear in media query conditions. Use SASS variables or hardcode breakpoint values.

3. **Circular references.** `--a: calc(var(--a) + 1px)` creates a cycle. Both properties become guaranteed-invalid. There is no self-referencing in CSS custom properties.

4. **Evaluation scope confusion.** Defining `--color: rgb(var(--r), var(--g), var(--b))` on `:root` and then changing `--r` on a child element does NOT change `--color` for that child. The `rgb()` call was already resolved at `:root`. Move the expression to the point of use:
   ```css
   /* BAD: resolved at :root, changing --r on child has no effect */
   :root {
     --r: 59; --g: 130; --b: 246;
     --color: rgb(var(--r), var(--g), var(--b));
   }

   /* GOOD: expression at point of use */
   :root { --r: 59; --g: 130; --b: 246; }
   .card { background: rgb(var(--r), var(--g), var(--b)); }
   ```

5. **Unit concatenation.** `var(--size)px` produces `16 px` with a space, which is invalid. Use `calc(var(--size) * 1px)`.

6. **Forgetting `.trim()` in JavaScript.** `getPropertyValue()` returns the raw string with leading whitespace. Always trim.

7. **Inline specificity from JavaScript.** `element.style.setProperty('--x', value)` sets at inline-style specificity, overriding all stylesheet declarations. Remember to `removeProperty()` when the override is no longer needed.

8. **Empty vs. space.** `--foo:;` (no space) is invalid syntax. `--foo: ;` (with a space) is valid. The value must contain at least one token.

---

## Best Practices

1. **Use semantic naming, not raw values.** Map primitives to semantic tokens. Consumers reference `--color-action`, not `--color-blue-500`. This allows theme changes without touching component code.

2. **Provide fallbacks for critical properties.** Even if you control all the CSS, fallbacks protect against typos and refactoring mistakes:
   ```css
   color: var(--text-color, #333);
   ```

3. **Register animated properties with `@property`.** Always set `inherits: false` for animation performance:
   ```css
   @property --progress {
     syntax: "<percentage>";
     inherits: false;
     initial-value: 0%;
   }
   ```

4. **Scope properties as narrowly as possible.** Global `:root` properties are convenient but slow to update and easy to accidentally override. Put component-specific properties on the component selector.

5. **Use custom property stacks for component APIs.** Let consumers override at multiple levels:
   ```css
   .button {
     background: var(--button-bg, var(--action-color, var(--brand, #0066cc)));
   }
   ```

6. **Do not use custom properties as property names.** `var(--prop): value;` is invalid CSS. Custom properties can only appear in values.

7. **Prefer `@property` in CSS over `CSS.registerProperty()` in JavaScript.** CSS declarations are easier to maintain, override, and debug.

## Real-World Use Cases

### 1. Component API with Customization Hooks

```css
/* Component defines its custom property API */
.avatar {
  --avatar-size: 40px;
  --avatar-radius: 50%;
  --avatar-border: 2px solid transparent;

  width: var(--avatar-size);
  height: var(--avatar-size);
  border-radius: var(--avatar-radius);
  border: var(--avatar-border);
  object-fit: cover;
}

/* Consumers customize without touching component internals */
.avatar-lg {
```

### 2. Animated Progress Ring

```css
@property --progress {
  syntax: "<percentage>";
  inherits: false;
  initial-value: 0%;
}

.progress-ring {
  --progress: 0%;
  width: 120px;
  height: 120px;
  border-radius: 50%;
  background: conic-gradient(
    #3b82f6 var(--progress),
```

### 3. Responsive Spacing System

```css
:root {
  --base-space: 4px;
  --scale: 1;
}

@media (min-width: 768px)  { :root { --scale: 1.25; } }
@media (min-width: 1200px) { :root { --scale: 1.5; } }

:root {
  --space-xs: calc(var(--base-space) * 1 * var(--scale));   /* 4-6px */
  --space-sm: calc(var(--base-space) * 2 * var(--scale));   /* 8-12px */
  --space-md: calc(var(--base-space) * 4 * var(--scale));   /* 16-24px */
  --space-lg: calc(var(--base-space) * 8 * var(--scale));   /* 32-48px */
  --space-xl: calc(var(--base-space) * 16 * var(--scale));  /* 64-96px */
}

.section {
  padding: var(--space-lg) var(--space-md);
}

.card {
  padding: var(--space-md);
  gap: var(--space-sm);
}
```

Change `--scale` at one breakpoint and every spacing value in the system adjusts proportionally.

---

> **Key Takeaway:** CSS custom properties are runtime cascade-aware variables that enable theming, component APIs, and responsive design; register them with `@property` when you need type safety, animation, or inheritance control, and always be aware of IACVT behavior when debugging fallback failures.
