# 2.13 Transitions

> CSS transitions smoothly interpolate property values between states. Understanding which properties to animate, how easing works, and the new entry/exit primitives separates janky UIs from polished ones.

---

## Transition Fundamentals

### The Four Longhand Properties

| Property | Default | Purpose |
|----------|---------|---------|
| `transition-property` | `all` | Which property to animate |
| `transition-duration` | `0s` | How long the transition takes |
| `transition-timing-function` | `ease` | The easing curve |
| `transition-delay` | `0s` | Wait before the transition starts |

### The Shorthand

```css
transition: <property> <duration> <timing-function> <delay>;
```

The first time value is always `duration`. The second time value is `delay`:

```css
/* 300ms duration, no delay */
transition: opacity 300ms ease;

/* 300ms duration, 100ms delay */
transition: opacity 300ms ease 100ms;
```

### Multiple Properties

Comma-separate each property with its own duration, timing, and delay:

```css
.card {
  transition:
    opacity 0.3s ease-out,
    transform 0.5s cubic-bezier(0.4, 0, 0.2, 1);
}
```

Each group is independent. You cannot combine shared values across groups.

### Reversal Behavior

When a property changes back to its original value mid-transition, the browser reverses from the current interpolated position, not from the end state. The CSS Transitions Level 2 spec defines a "reversing shortening factor" so that mid-flight reversals feel proportional rather than restarting the full duration.

---

## transition-property: Avoid `all`

`transition: all 0.3s ease` is one of the most common CSS mistakes. Problems:

1. **Performance**: The browser must check every computed property for changes on every frame.
2. **Unintended animations**: Future CSS additions can accidentally trigger transitions on `padding`, `border`, `box-shadow`, or other properties.
3. **Unmaintainable**: When reading `all`, you cannot determine which properties the author intended to animate.

```css
/* Bad Practice */
.card {
  transition: all 0.3s ease;
}

/* Good Practice */
.card {
  transition: opacity 0.3s ease, transform 0.3s ease;
}
```

### Animatable Property Performance Tiers

| Tier | Properties | Cost | GPU Accelerated |
|------|-----------|------|-----------------|
| Compositor-only | `transform`, `opacity` | Very low | Yes |
| Paint-only | `color`, `background-color`, `border-color` | Medium | No |
| Layout-triggering | `width`, `height`, `margin`, `padding`, `top`, `left` | High | No |
| Very expensive | `box-shadow`, `filter` (complex) | High | Partial |

**Rule of thumb**: Stick to `transform` and `opacity` for 60fps performance. If you need to animate size or position, use `transform: scale()` and `transform: translate()` instead of `width`/`height`/`top`/`left`.

## Easing Functions

### Built-in Keywords

| Keyword | cubic-bezier() | Character |
|---------|---------------|-----------|
| `linear` | `(0, 0, 1, 1)` | Constant speed |
| `ease` (default) | `(0.25, 0.1, 0.25, 1)` | Slow start, fast middle, slow end |
| `ease-in` | `(0.42, 0, 1, 1)` | Starts slow, accelerates |
| `ease-out` | `(0, 0, 0.58, 1)` | Starts fast, decelerates |
| `ease-in-out` | `(0.42, 0, 0.58, 1)` | Symmetric slow-fast-slow |

### Custom cubic-bezier()

`cubic-bezier(x1, y1, x2, y2)` defines a Bezier curve with two control points. The x values must be between 0 and 1. The y values can exceed this range to create overshoot effects:

```css
/* Overshoot (bouncy feeling) */
transition-timing-function: cubic-bezier(0.34, 1.56, 0.64, 1);

/* Material Design standard curve */
transition-timing-function: cubic-bezier(0.4, 0, 0.2, 1);

/* Snappy deceleration */
transition-timing-function: cubic-bezier(0, 0, 0.2, 1);
```

### The linear() Function

`linear()` creates piecewise linear easing with multiple stops. This enables bounce, elastic, and spring effects that `cubic-bezier()` cannot express:

```css
/* Bounce effect */
transition-timing-function: linear(
  0, 0.063, 0.25, 0.563, 1 36.4%,
  0.812, 0.75, 0.813, 1 72.7%,
  0.953, 0.938, 0.953, 1 90.9%,
  0.984, 1 100%
);
```

Stops are equidistant by default. Explicit percentage positions let you control the hold and speed of each segment:

```css
/* Hold at 0.25 from 25% to 75%, then jump to 1 */
transition-timing-function: linear(0, 0.25 25% 75%, 1);
```

### Step Functions

`steps(n, position)` divides the transition into discrete jumps:

```css
/* 6 frames, each jump at the end of its interval */
transition-timing-function: steps(6);

/* Immediate first jump */
transition-timing-function: steps(6, jump-start);
```

| Position | Behavior |
|----------|----------|
| `jump-end` (default) | Last jump at 100% |
| `jump-start` | First jump at 0% |
| `jump-both` | Jumps at both 0% and 100% |
| `jump-none` | No jump at endpoints |

Shorthand keywords:
- `step-start` = `steps(1, jump-start)`: instantly jumps to the final value.
- `step-end` = `steps(1, jump-end)`: stays at the initial value until the end.

Step functions are primarily useful for sprite sheet animations and typewriter effects.

## Transition Delay Techniques

### Negative Delay

A negative delay starts the transition immediately, but partway through:

```css
/* 1s transition that begins at the 30% point */
.element {
  transition: opacity 1s ease -0.3s;
}
```

This is useful when you want an element to appear already partially transitioned.

### Staggered Transitions

Use CSS custom properties with `calc()` to create staggered effects:

```css
.list-item {
  opacity: 0;
  transform: translateY(20px);
  transition: opacity 0.4s ease, transform 0.4s ease;
  transition-delay: calc(var(--i) * 100ms);
}

.list.visible .list-item {
  opacity: 1;
  transform: translateY(0);
}
```

```html
<ul class="list">
  <li class="list-item" style="--i: 0">First</li>
  <li class="list-item" style="--i: 1">Second</li>
  <li class="list-item" style="--i: 2">Third</li>
  <li class="list-item" style="--i: 3">Fourth</li>
</ul>
```

### Per-Property Delays for Sequenced Effects

```css
.card:hover {
  transition:
    transform 0.3s ease 0s,         /* starts immediately */
    opacity 0.3s ease 0.15s,        /* starts after 150ms */
    box-shadow 0.3s ease 0.3s;      /* starts after 300ms */
}
```

### Different Enter and Exit Timing

Place the exit transition on the base state and the enter transition on the active state:

```css
/* Exit: fast collapse */
.dropdown {
  opacity: 0;
  transform: translateY(-10px);
  transition: opacity 0.15s ease, transform 0.15s ease;
}

/* Enter: slower, with delay */
.trigger:hover .dropdown {
  opacity: 1;
  transform: translateY(0);
  transition: opacity 0.3s ease 0.1s, transform 0.3s ease;
}
```

The base state's `transition` controls the mouse-out behavior. The hover state's `transition` controls the mouse-in behavior.

## Transitioning display and visibility

### The Historical Problem

`display: none` instantly removes an element from rendering. Any running transition is killed immediately. There was no way to smoothly hide an element with `display: none`.

### transition-behavior: allow-discrete

`transition-behavior: allow-discrete` enables transitions on discrete properties like `display`. The value flips at a specific point:

- **Animating TO `display: none`**: The visible value is maintained for the full duration, then flips to `none` at 100%.
- **Animating FROM `display: none`**: The value flips to visible at 0%, so the element is visible throughout.

```css
.panel {
  opacity: 1;
  transition:
    opacity 0.3s ease,
    display 0.3s allow-discrete;
}

.panel.hidden {
  opacity: 0;
  display: none;
}
```

### The overlay Property

When animating top-layer elements (dialogs, popovers), also transition `overlay` to keep the element in the top layer during the exit animation:

```css
dialog[open] {
  opacity: 1;
  transform: scale(1);
}

dialog {
  opacity: 0;
  transform: scale(0.9);
  transition:
    opacity 0.3s ease,
    transform 0.3s ease,
    display 0.3s allow-discrete,
    overlay 0.3s allow-discrete;
}
```

Without `overlay`, the dialog is immediately removed from the top layer when closed, causing it to be clipped behind other content before the fade-out finishes.

## @starting-style: Entry Transitions

### The Problem

When an element appears (changes from `display: none` to visible, or is inserted into the DOM), there is no "before" state to transition from. The element simply pops into existence.

### The Solution

`@starting-style` defines the initial values to transition from when an element first receives styles:

```css
.card {
  opacity: 1;
  transform: translateY(0);
  transition: opacity 0.5s ease, transform 0.5s ease;
}

@starting-style {
  .card {
    opacity: 0;
    transform: translateY(20px);
  }
}
```

### Nested Syntax

`@starting-style` can be nested inside the rule block:

```css
.card {
  opacity: 1;
  transform: translateY(0);
  transition: opacity 0.5s ease, transform 0.5s ease;

  @starting-style {
    opacity: 0;
    transform: translateY(20px);
  }
}
```

### Full Entry + Exit Pattern for Popovers

```css
.popover {
  /* Exit state (closed) */
  opacity: 0;
  transform: translateY(-10px);
  transition:
    opacity 0.3s ease,
    transform 0.3s ease,
    display 0.3s allow-discrete,
    overlay 0.3s allow-discrete;
}

.popover:popover-open {
  /* Open state */
  opacity: 1;
  transform: translateY(0);

  /* Entry state (where the transition starts FROM) */
  @starting-style {
    opacity: 0;
    transform: translateY(10px);
  }
}
```

### The Specificity Gotcha

`@starting-style` follows normal specificity rules. It is NOT promoted like `@keyframes` styles. This means:

1. `@starting-style` must appear AFTER the element's regular styles, or the regular styles override it.
2. A more specific selector that sets the same property will beat the starting style.

```css
/* Bug: @starting-style has lower specificity */
.container .card {
  opacity: 1;  /* specificity: 0-2-0 */
}

@starting-style {
  .card {
    opacity: 0;  /* specificity: 0-1-0 -- LOSES, never applied */
  }
}

/* Fix: match or exceed the specificity */
@starting-style {
  .container .card {
    opacity: 0;  /* specificity: 0-2-0 -- works */
  }
}
```

## Transitioning to height: auto

### Why It Does Not Work

The browser cannot interpolate between a fixed length and `auto` because `auto` is content-dependent. Computing it for every animation frame would require a full layout reflow at each step.

### Workaround 1: CSS Grid 0fr/1fr

Grid track `fr` units are animatable. Wrap the content in a grid container:

```css
.accordion {
  display: grid;
  grid-template-rows: 0fr;
  transition: grid-template-rows 0.4s ease;
}

.accordion.open {
  grid-template-rows: 1fr;
}

.accordion > .content {
  overflow: hidden;
}
```

This is the best cross-browser solution as of 2026. Supported in Chrome 107+, Firefox, and Safari.

### Workaround 2: max-height Hack

```css
.panel {
  max-height: 0;
  overflow: hidden;
  transition: max-height 0.3s ease;
}

.panel.open {
  max-height: 500px;  /* must be larger than actual content */
}
```

The problem: if `max-height` is much larger than the content, the visible animation ends well before the transition duration. The easing feels wrong because the element stops growing while the transition is still running.

### Workaround 3: interpolate-size (Chromium Only)

`interpolate-size: allow-keywords` enables direct transitions to `auto`:

```css
:root {
  interpolate-size: allow-keywords;
}

.panel {
  height: 0;
  overflow: hidden;
  transition: height 0.3s ease;
}

.panel.open {
  height: auto;  /* smoothly transitions */
}
```

As of early 2026, this is only supported in Chromium browsers (Chrome 129+, Edge 129+). Not yet in Firefox or Safari.

### calc-size()

`calc-size()` enables calculations on intrinsic size values and automatically enables interpolation:

```css
.panel.open {
  height: calc-size(auto);  /* enables interpolation */
}

/* Math with intrinsic sizes */
.element {
  width: calc-size(fit-content, size + 2rem);
}
```

Same Chromium-only limitation as `interpolate-size`.

### Progressive Enhancement

```css
/* Cross-browser fallback: CSS Grid */
.accordion {
  display: grid;
  grid-template-rows: 0fr;
  transition: grid-template-rows 0.4s ease;
}
.accordion.open {
  grid-template-rows: 1fr;
}
.accordion > .content {
  overflow: hidden;
}

/* Enhancement for Chromium */
@supports (interpolate-size: allow-keywords) {
  .accordion {
    display: block;
  }
  .accordion > .content {
    height: 0;
    overflow: hidden;
    interpolate-size: allow-keywords;
    transition: height 0.4s ease;
  }
  .accordion.open > .content {
    height: auto;
  }
}
```

## JavaScript Transition Events

Four events fire on `Element` during a transition:

| Event | When |
|-------|------|
| `transitionrun` | Transition created (even before delay) |
| `transitionstart` | Transition begins animating (after delay) |
| `transitionend` | Transition completes |
| `transitioncancel` | Transition is interrupted |

### Firing Order

- No delay: `transitionrun` and `transitionstart` fire nearly simultaneously, then `transitionend`.
- With delay: `transitionrun`, then after delay `transitionstart`, then `transitionend`.
- If canceled (e.g., `display: none` set mid-transition): `transitioncancel` fires instead of `transitionend`.

### The Multiple-Event Gotcha

`transitionend` fires once PER PROPERTY PER ELEMENT. Transitioning `opacity` and `transform` produces two events:

```javascript
element.addEventListener('transitionend', (e) => {
  // Filter to act only once
  if (e.propertyName === 'opacity') {
    element.remove();
  }
});
```

### transitionend Never Fires If Canceled

If a transition is interrupted (element removed, `display` set to `none`, transition-property removed), `transitionend` does NOT fire. Use `transitioncancel` as a backup:

```javascript
element.addEventListener('transitionend', handleDone);
element.addEventListener('transitioncancel', handleDone);
```

## Performance: The Rendering Pipeline

The browser renders in three stages:

1. **Layout**: Calculate geometry and positions of every element.
2. **Paint**: Fill pixels (text, colors, images, shadows).
3. **Composite**: Combine painted layers in the correct order.

### Compositor-Only Properties

`transform` and `opacity` skip layout and paint entirely. They run on the GPU compositor thread, achieving 60fps even when the main thread is busy with JavaScript.

How it works:
1. The browser promotes the element to its own compositing layer.
2. The layer texture is uploaded to the GPU.
3. During the transition, only the layer's transform/opacity changes on the GPU.
4. No layout or paint recalculation occurs.

### will-change

`will-change` hints the browser to prepare an element for animation:

```css
.card {
  will-change: transform;
}
```

Rules for `will-change`:
- Do NOT apply it to everything. Each promoted layer consumes GPU memory.
- Apply it BEFORE the animation starts (e.g., on hover of a parent, or via JS before triggering the change).
- Remove it after the animation ends.
- Do not use it for premature optimization. Only solve actual observed jank.

Toggling via JavaScript:

```javascript
card.addEventListener('mouseenter', () => {
  card.style.willChange = 'transform';
});
card.addEventListener('transitionend', () => {
  card.style.willChange = 'auto';
});
```

## Accessibility: prefers-reduced-motion

Motion can trigger vestibular disorders, migraines, and seizures. Users who enable "Reduce motion" in their OS settings expect the web to respect that preference.

### Basic Implementation

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    transition-duration: 0.01ms !important;
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
  }
}
```

### Why 0.01ms Instead of 0s

Setting `transition-duration: 0.01ms` ensures `transitionend` events still fire. Setting `transition: none` or `0s` means `transitionend` never fires, breaking JavaScript workflows that depend on it.

### Reduced Motion Does Not Mean No Motion

Replace large sweeping motions with subtle fades. State changes should still be visible:

```css
.card {
  transition: transform 0.3s ease, opacity 0.3s ease;
}

@media (prefers-reduced-motion: reduce) {
  .card {
    /* Remove the motion, keep the fade */
    transition: opacity 0.15s ease;
  }
}
```

### JavaScript Detection

```javascript
const reduced = window.matchMedia('(prefers-reduced-motion: reduce)');

if (reduced.matches) {
  // Skip JS-driven animations
}

// React to live changes
reduced.addEventListener('change', (e) => {
  // User toggled the OS setting
});
```

## Common Mistakes

1. **Using `transition: all`.** Triggers unintended animations, hurts performance, and is impossible to audit. Always list specific properties.

2. **Animating layout properties.** Transitioning `width`, `height`, `margin`, `padding`, `top`, or `left` triggers a full layout recalculation per frame. Use `transform` and `opacity` instead.

3. **Transitions that are too long.** Hover transitions should be 150-300ms. Page-level transitions can go up to 500ms. Anything longer feels sluggish.

4. **Placing `transition` only on `:hover`.** The transition plays on mouse-in but snaps back instantly on mouse-out because the base state has no transition declaration. Always put the transition on the base state unless you intentionally want asymmetric behavior.

5. **Not filtering `transitionend` events.** The event fires once per property. If you transition two properties and run cleanup code on `transitionend`, your code runs twice.

6. **Forgetting that transitions go both ways.** A transition declaration triggers on any state change, including reversal. If you want different enter/exit timing, set different transitions on the base and active selectors.

7. **Using `transition: none` for reduced motion.** This prevents `transitionend` from firing, breaking JS-dependent workflows. Use `transition-duration: 0.01ms` instead.

8. **Not handling the `height: auto` problem.** Directly transitioning to `height: auto` does not work. Use the CSS Grid `0fr`/`1fr` technique for cross-browser support.

---

## Best Practices

1. **Animate only `transform` and `opacity`** for guaranteed 60fps performance. These are compositor-only and GPU-accelerated.

2. **Define a motion design token system.** Consistent durations and easings across your application:
   ```css
   :root {
     --duration-fast: 150ms;
     --duration-normal: 300ms;
     --duration-slow: 500ms;
     --ease-standard: cubic-bezier(0.4, 0, 0.2, 1);
     --ease-decelerate: cubic-bezier(0, 0, 0.2, 1);
     --ease-accelerate: cubic-bezier(0.4, 0, 1, 1);
   }
   ```

3. **Use `ease-out` for enter transitions** (elements appearing) and **`ease-in` for exit transitions** (elements disappearing). This matches how physical objects accelerate into view and decelerate as they leave.

4. **Respect `prefers-reduced-motion`.** Replace motion with fades, never fully disable state changes. Use `0.01ms` duration instead of `none`.

5. **Use `@starting-style` for entry transitions** instead of JavaScript `requestAnimationFrame` hacks.

6. **Use `transition-behavior: allow-discrete`** for smoothly toggling `display: none`.

7. **Use the CSS Grid `0fr`/`1fr` technique** for `height: auto` transitions until `interpolate-size` reaches cross-browser support.

8. **Apply `will-change` only when needed** and remove it after the transition ends.

## Real-World Use Cases

### 1. Smooth Dialog Entry and Exit

```css
dialog[open] {
  opacity: 1;
  transform: translateY(0);
}

dialog {
  opacity: 0;
  transform: translateY(20px);
  transition:
    opacity 0.3s var(--ease-decelerate),
    transform 0.3s var(--ease-decelerate),
    display 0.3s allow-discrete,
    overlay 0.3s allow-discrete;
}

@starting-style {
  dialog[open] {
    opacity: 0;
    transform: translateY(-20px);
  }
}
```

Entry: slides down and fades in. Exit: slides down and fades out. `display` and `overlay` keep the dialog visible during exit.

### 2. Accordion with CSS Grid Height Transition

```css
.accordion-item {
  border-bottom: 1px solid hsl(220 13% 90%);
}

.accordion-panel {
  display: grid;
  grid-template-rows: 0fr;
  transition: grid-template-rows 0.4s var(--ease-standard);
}

.accordion-item.open .accordion-panel {
  grid-template-rows: 1fr;
}

.accordion-panel > .inner {
  overflow: hidden;
}
```

### 3. Staggered List Reveal

```css
.reveal-list .item {
  opacity: 0;
  transform: translateY(16px);
  transition:
    opacity 0.4s var(--ease-decelerate),
    transform 0.4s var(--ease-decelerate);
  transition-delay: calc(var(--i) * 80ms);
}

.reveal-list.visible .item {
  opacity: 1;
  transform: translateY(0);
}

@media (prefers-reduced-motion: reduce) {
  .reveal-list .item {
    transition: opacity 0.15s ease;
    transform: none;
  }
}
```

```html
<ul class="reveal-list">
  <li class="item" style="--i: 0">Dashboard</li>
  <li class="item" style="--i: 1">Analytics</li>
  <li class="item" style="--i: 2">Reports</li>
  <li class="item" style="--i: 3">Settings</li>
</ul>
```

---

> **Key Takeaway:** Always transition specific properties (not `all`), stick to `transform` and `opacity` for performance, use `@starting-style` with `transition-behavior: allow-discrete` for modern entry/exit animations, and always respect `prefers-reduced-motion`.
