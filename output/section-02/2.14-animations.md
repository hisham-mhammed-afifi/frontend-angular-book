# 2.14 Animations

> CSS @keyframes animations provide multi-step, self-starting, loopable motion sequences. Unlike transitions (which need a state change trigger), animations can run on load, loop infinitely, and define as many intermediate steps as needed.

---

## Animation vs. Transition: When to Use Which

| Feature | Transitions | Animations |
|---------|-------------|------------|
| Trigger | Requires state change (`:hover`, class toggle) | Self-starting, or via class addition |
| Complexity | A to B only | Unlimited keyframe steps |
| Looping | Not built-in | `animation-iteration-count: infinite` |
| Reversal | Automatic on state removal | Manual via `animation-direction` |
| Intermediate states | None | As many `%` stops as needed |
| Mid-flight reversal | Built-in proportional reversal | Must manage manually |

**Use transitions** for simple A-to-B state changes (hover, focus, toggle). **Use animations** for multi-step sequences, on-load effects, continuous loops, or anything that requires intermediate states.

---

## @keyframes Syntax

The `@keyframes` at-rule defines a named animation sequence:

```css
@keyframes slide-in {
  from {
    transform: translateX(-100%);
    opacity: 0;
  }
  to {
    transform: translateX(0);
    opacity: 1;
  }
}
```

`from` equals `0%`. `to` equals `100%`. You can use any percentage between:

```css
@keyframes bounce {
  0%   { transform: translateY(0); }
  30%  { transform: translateY(-30px); }
  50%  { transform: translateY(0); }
  70%  { transform: translateY(-15px); }
  100% { transform: translateY(0); }
}
```

### Multiple Percentages in One Block

```css
@keyframes hold-then-fade {
  0%, 80% { opacity: 1; }
  100%    { opacity: 0; }
}
```

### Omitting 0% or 100%

If you omit `0%` or `100%`, the browser generates it from the element's existing computed styles:

```css
@keyframes fade-out {
  /* 0% is generated from the element's current opacity */
  to { opacity: 0; }
}
```

### Duplicate Percentage Merging

Within the same `@keyframes` block, duplicate percentage selectors merge their properties. If the same property appears at the same percentage more than once, the last one wins:

```css
@keyframes example {
  50% { top: 10px; }
  50% { left: 20px; }
  /* Result at 50%: top: 10px AND left: 20px (merged) */
}
```

## The Animation Shorthand

```css
animation: <name> <duration> <timing-function> <delay> <iteration-count> <direction> <fill-mode> <play-state>;
```

### All Longhand Properties

| Property | Default | Purpose |
|----------|---------|---------|
| `animation-name` | `none` | Which `@keyframes` to use |
| `animation-duration` | `0s` | Length of one cycle |
| `animation-timing-function` | `ease` | Easing curve per keyframe interval |
| `animation-delay` | `0s` | Wait before starting |
| `animation-iteration-count` | `1` | How many cycles (or `infinite`) |
| `animation-direction` | `normal` | Forward, reverse, alternating |
| `animation-fill-mode` | `none` | Styles applied before/after |
| `animation-play-state` | `running` | Running or paused |
| `animation-timeline` | `auto` | Time-based or scroll-driven |
| `animation-composition` | `replace` | How values compose with underlying styles |

### Shorthand Parsing Rules

The first time value is `duration`. The second time value is `delay`:

```css
/* 300ms duration, 100ms delay */
animation: slide-in 300ms ease 100ms;
```

The `animation-name` should come last in the shorthand. If a name collides with a CSS keyword (like `reverse`, `infinite`, `none`), the parser consumes it as a property value instead of a name:

```css
/* Bug: "reverse" is parsed as animation-direction, not a name */
animation: reverse 1s;

/* Safe: name last */
animation: 1s ease-in-out infinite alternate slide-in;
```

### Multiple Animations

Comma-separate multiple animations on one element:

```css
.hero {
  animation:
    fade-in 0.5s ease-out,
    slide-up 0.5s ease-out,
    pulse 2s ease-in-out 0.5s infinite;
}
```

When two animations target the same property, the last animation in the list wins (default `replace` composition).

## animation-timing-function

The timing function applies **between keyframes**, not across the entire animation. Each pair of adjacent keyframes is a separate interpolation with its own easing:

```css
@keyframes move {
  0%   { transform: translateX(0); }
  50%  { transform: translateX(200px); }  /* easing applies 0% -> 50% */
  100% { transform: translateX(400px); }  /* easing applies 50% -> 100% */
}

.box {
  animation: move 2s ease-in-out;
  /* ease-in-out applies to EACH segment, not the whole duration */
}
```

You can set different easings per keyframe:

```css
@keyframes complex {
  0%   { transform: translateX(0); animation-timing-function: ease-in; }
  50%  { transform: translateX(200px); animation-timing-function: ease-out; }
  100% { transform: translateX(400px); }
}
```

### Step Functions for Sprite Sheets

`steps(n)` divides each keyframe interval into discrete jumps with no interpolation:

```css
.character {
  width: 64px;
  height: 64px;
  background: url('spritesheet.png') no-repeat;
  animation: walk 0.8s steps(8) infinite;
}

@keyframes walk {
  from { background-position: 0 0; }
  to   { background-position: -512px 0; }  /* 8 frames * 64px */
}
```

Step positions control where jumps occur:

| Keyword | Behavior |
|---------|----------|
| `jump-end` (default) | Jumps at the end of each step |
| `jump-start` | Jumps at the start of each step |
| `jump-both` | Jumps at both endpoints |
| `jump-none` | No jumps at endpoints |

## animation-direction

| Value | Odd Cycles | Even Cycles |
|-------|-----------|-------------|
| `normal` | 0% to 100% | 0% to 100% |
| `reverse` | 100% to 0% | 100% to 0% |
| `alternate` | 0% to 100% | 100% to 0% |
| `alternate-reverse` | 100% to 0% | 0% to 100% |

`alternate` is the most useful for creating smooth back-and-forth motion:

```css
.pulse {
  animation: pulse 1s ease-in-out infinite alternate;
}

@keyframes pulse {
  from { transform: scale(1); }
  to   { transform: scale(1.05); }
}
```

### Easing Reversal Gotcha

When playing in reverse, easing functions are also reversed. `ease-in` effectively becomes `ease-out` during reverse cycles. This is by design but can be surprising.

## animation-fill-mode

Fill mode controls what styles apply before and after the animation runs.

| Value | During Delay | After Completion |
|-------|-------------|------------------|
| `none` | Element's own styles | Element's own styles |
| `forwards` | Element's own styles | Last keyframe's styles retained |
| `backwards` | First keyframe's styles applied | Element's own styles |
| `both` | First keyframe's styles applied | Last keyframe's styles retained |

### Which Keyframe Is "First" and "Last"?

This depends on `animation-direction`:

| Direction | First Keyframe | Last Keyframe |
|-----------|---------------|---------------|
| `normal` | 0% | 100% |
| `reverse` | 100% | 0% |
| `alternate` (odd count) | 0% | 100% |
| `alternate` (even count) | 0% | 0% |

### The forwards Danger

`forwards` creates a "sticky" style that sits above normal cascade rules. Only `!important` declarations can override it. This is a frequent source of bugs:

```css
@keyframes fade-in {
  from { opacity: 0; }
  to   { opacity: 1; }
}

.card {
  animation: fade-in 0.5s ease forwards;
}

/* This does NOT work -- animated value overrides normal cascade */
.card.hidden {
  opacity: 0;  /* IGNORED while fill mode is active */
}
```

The fix is to remove the animation entirely or use `animation: none` when you want to override:

```css
.card.hidden {
  animation: none;
  opacity: 0;
}
```

### When backwards Is Useful

`backwards` only matters when `animation-delay` is set. It applies the first keyframe's styles during the delay period instead of showing the element's default state:

```css
.item {
  opacity: 1;
  animation: fade-in 0.5s ease 0.3s backwards;
  /* During the 0.3s delay, opacity is 0 (from the 0% keyframe) */
  /* Without backwards, opacity would be 1 during the delay */
}
```

## animation-composition

When an animation composes with the element's underlying styles:

| Value | Behavior | Example: base `translateX(50px)`, keyframe `translateX(100px)` |
|-------|----------|--------------------------------------------------------------|
| `replace` (default) | Keyframe value replaces base | `translateX(100px)` |
| `add` | Keyframe value appended | `translateX(50px) translateX(100px)` |
| `accumulate` | Values mathematically combined | `translateX(150px)` |

```css
.element {
  transform: translateX(50px);
  animation: move 1s ease;
  animation-composition: accumulate;
}

@keyframes move {
  to { transform: translateX(100px); }
  /* Result: 50px + 100px = 150px */
}
```

The distinction between `add` and `accumulate` matters for transform lists. `add` creates two separate transforms. `accumulate` sums the numeric values.

## @keyframes and the Cascade

Animated values follow special cascade rules:

### Override Hierarchy (strongest to weakest)

1. `!important` declarations on the element
2. Actively transitioning properties
3. **Animated properties (from @keyframes)**
4. Normal styles (regardless of specificity)

This means animated values override ALL normal CSS, no matter how specific the selector. This is why `fill-mode: forwards` can be so dangerous.

### Additional Rules

- `!important` inside `@keyframes` is silently ignored
- Duplicate `@keyframes` with the same name do NOT merge; the last one wins entirely
- All percentage selectors within a `@keyframes` block have equal specificity; order resolves conflicts

## animation-play-state

Pause and resume animations without losing their current position:

```css
.spinner {
  animation: rotate 1s linear infinite;
}

.spinner.paused {
  animation-play-state: paused;
}
```

### JavaScript Control

```javascript
const spinner = document.querySelector('.spinner');

// Pause
spinner.style.animationPlayState = 'paused';

// Resume from where it left off
spinner.style.animationPlayState = 'running';
```

### Custom Property Approach

```css
.animated {
  --play-state: running;
  animation: spin 2s linear infinite;
  animation-play-state: var(--play-state);
}
```

```javascript
el.style.setProperty('--play-state', 'paused');
```

## Scroll-Driven Animations

Instead of running on a time-based timeline, animations can be driven by scroll position.

### scroll() Timeline

Tracks how far a scroll container has been scrolled (0% to 100%):

```css
.progress-bar {
  animation: grow linear;
  animation-timeline: scroll();
}

@keyframes grow {
  from { width: 0%; }
  to   { width: 100%; }
}
```

The `scroll()` function accepts arguments for which scroller and which axis:

```css
animation-timeline: scroll();                /* nearest scrollable ancestor, block axis */
animation-timeline: scroll(root);            /* document root scroller */
animation-timeline: scroll(root inline);     /* root scroller, inline (horizontal) axis */
```

### view() Timeline

Tracks an element's visibility within its scroll container:

```css
.card {
  animation: reveal linear;
  animation-timeline: view();
}

@keyframes reveal {
  from { opacity: 0; transform: translateY(50px); }
  to   { opacity: 1; transform: translateY(0); }
}
```

### animation-range

Controls when within the timeline the animation plays:

```css
.card {
  animation: reveal linear;
  animation-timeline: view();
  animation-range: entry;  /* animate only during viewport entry */
}
```

Named ranges for view timelines:

| Range | Description |
|-------|-------------|
| `cover` | Full range while element is visible at all |
| `contain` | Element fully within viewport |
| `entry` | Element entering the viewport |
| `exit` | Element leaving the viewport |
| `entry-crossing` | Element crossing the viewport end border |
| `exit-crossing` | Element crossing the viewport start border |

Precise range control:

```css
/* Start at 25% of entry, end at 75% of entry */
animation-range: entry 25% entry 75%;
```

### Performance

Scroll-driven animations on `transform` and `opacity` run entirely on the compositor thread. Zero main-thread impact, smooth 60fps regardless of JavaScript activity. This makes them far more performant than JavaScript scroll listeners or Intersection Observer-based solutions.

### Browser Support

Chrome 115+, Safari 18+, Edge 115+. Firefox has partial support behind a flag. Not yet fully cross-browser, but progressive enhancement is straightforward since the animation simply does not apply in unsupported browsers.

## The Web Animations API (WAAPI)

The JavaScript counterpart to CSS animations. Provides programmatic control over animations created from JavaScript or CSS.

### Element.animate()

```javascript
const animation = element.animate(
  [
    { transform: 'translateX(0)', opacity: 1 },
    { transform: 'translateX(300px)', opacity: 0 }
  ],
  {
    duration: 1000,
    easing: 'ease-in-out',
    iterations: 1,
    fill: 'forwards',
    delay: 0,
    direction: 'normal'
  }
);
```

Property names use camelCase: `background-color` becomes `backgroundColor`.

### Animation Object Controls

```javascript
animation.pause();
animation.play();
animation.reverse();
animation.cancel();
animation.finish();

animation.playbackRate = 2;     // 2x speed
animation.playbackRate = 0.5;   // half speed
animation.currentTime = 500;    // jump to 500ms

// Wait for completion
await animation.finished;
console.log('Done');
```

### commitStyles()

Commits the animation's current state to the element's inline styles, then allows canceling the animation cleanly:

```javascript
const anim = el.animate(
  [{ transform: 'translateX(200px)' }],
  { duration: 500, fill: 'forwards' }
);

await anim.finished;
anim.commitStyles();  // writes transform to inline style
anim.cancel();        // removes animation without visual snap
```

### getAnimations()

```javascript
// All animations on an element (CSS and WAAPI)
const anims = element.getAnimations();

// All animations in the entire document
const allAnims = document.getAnimations();

// Pause everything
allAnims.forEach(a => a.pause());
```

## Animation Events

### CSS Animation Events

| Event | When |
|-------|------|
| `animationstart` | After `animation-delay` expires, animation begins |
| `animationiteration` | One cycle ends and the next begins |
| `animationend` | All cycles complete |

`animationiteration` does NOT fire for `animation-iteration-count: 1` (there is no "next" cycle).

```javascript
element.addEventListener('animationend', (e) => {
  console.log(e.animationName);   // @keyframes name
  console.log(e.elapsedTime);     // seconds since start (excluding delay)
  console.log(e.pseudoElement);   // '::before', '::after', or ''
});
```

### Event Gotcha

`animationend` does NOT fire if the element is removed from the DOM or if the animation CSS is removed before completion. If you rely on this event for cleanup, always handle the case where it never fires (use a timeout fallback or `animationcancel` as backup).

## Common Mistakes

1. **Forgetting to set `animation-duration`.** The default is `0s`. The animation runs instantly and appears to do nothing.

2. **Animating layout properties.** `width`, `height`, `margin`, `padding`, `top`, `left` trigger full layout recalculation per frame. Use `transform` and `opacity` instead.

3. **Using `animation-fill-mode: forwards` without understanding cascade implications.** The filled value overrides all normal CSS. Subsequent class changes that set the same property will be ignored. Use `animation: none` to break out of it.

4. **Naming keyframes with reserved words.** `none`, `inherit`, `initial`, `reverse`, `infinite` are parsed as property values in the shorthand, not as animation names. Choose descriptive names like `fade-in`, `slide-up`.

5. **Animating inline elements.** Per spec, animation behavior on inline-level elements is undefined. Always animate block, inline-block, or flex/grid children.

6. **Infinite animations without purpose.** Continuous looping wastes CPU/battery and distracts users. Only loop when the animation serves a clear purpose (loading spinners, background ambiance).

7. **Ignoring `prefers-reduced-motion`.** Approximately 8% of users experience motion sensitivity. Large movements, spinning, and parallax are the most problematic.

8. **Applying `will-change` to everything.** Each promoted layer consumes GPU memory. On mobile devices with limited GPU memory, too many layers degrades performance or crashes the browser.

9. **Relying on `animationend` for cleanup.** The event does not fire if the animation is canceled or the element is removed. Always have a fallback.

---

## Best Practices

1. **Animate only compositor properties** (`transform`, `opacity`, `filter`, `clip-path`) for guaranteed 60fps.

2. **Always respect `prefers-reduced-motion`:**
   ```css
   @media (prefers-reduced-motion: reduce) {
     *,
     *::before,
     *::after {
       animation-duration: 0.01ms !important;
       animation-iteration-count: 1 !important;
     }
   }
   ```
   Or replace large motion with subtle opacity fades.

3. **Use `animation-fill-mode: both` only when you fully understand which keyframe is "first" and "last"** based on your direction and iteration count. Prefer removing animations cleanly over relying on `forwards`.

4. **Use CSS custom properties for animation timing tokens:**
   ```css
   :root {
     --duration-fast: 150ms;
     --duration-normal: 300ms;
     --ease-bounce: cubic-bezier(0.34, 1.56, 0.64, 1);
   }
   ```

5. **Use scroll-driven animations** (`animation-timeline: view()`) instead of JavaScript Intersection Observer for scroll-reveal effects. They run on the compositor thread with zero main-thread cost.

6. **Use the Web Animations API** when you need dynamic control (playback rate, seek, reverse, completion promises) that CSS alone cannot provide.

7. **Keep animations purposeful.** Every animation should communicate something: spatial relationships, state changes, feedback, or attention direction.

## Real-World Use Cases

### 1. Scroll-Driven Reveal

```css
.section-card {
  animation: scroll-reveal linear both;
  animation-timeline: view();
  animation-range: entry 10% entry 90%;
}

@keyframes scroll-reveal {
  from {
    opacity: 0;
    transform: translateY(40px) scale(0.95);
  }
  to {
    opacity: 1;
    transform: translateY(0) scale(1);
  }
}

@media (prefers-reduced-motion: reduce) {
  .section-card {
    animation: none;
    opacity: 1;
  }
}
```

No JavaScript needed. Runs on the compositor thread. Falls back gracefully in unsupported browsers (elements simply appear without animation).

### 2. Staggered Loading Skeleton

```css
@keyframes shimmer {
  from { background-position: -200% 0; }
  to   { background-position: 200% 0; }
}

.skeleton {
  background: linear-gradient(
    90deg,
    hsl(220 10% 92%) 25%,
    hsl(220 10% 96%) 50%,
    hsl(220 10% 92%) 75%
  );
  background-size: 200% 100%;
  animation: shimmer 1.5s ease-in-out infinite;
  border-radius: 4px;
}

.skeleton-text {
  height: 1em;
  margin-bottom: 0.5em;
}

.skeleton-avatar {
  width: 48px;
  height: 48px;
  border-radius: 50%;
}
```

### 3. Page Scroll Progress Bar

```css
.progress-bar {
  position: fixed;
  top: 0;
  left: 0;
  height: 3px;
  background: oklch(55% 0.2 250);
  transform-origin: left;
  animation: progress-grow linear;
  animation-timeline: scroll(root);
  z-index: 9999;
}

@keyframes progress-grow {
  from { transform: scaleX(0); }
  to   { transform: scaleX(1); }
}
```

Uses `scaleX` instead of `width` for compositor-only performance. The `scroll(root)` timeline tracks the document scroll position.

---

> **Key Takeaway:** Use `@keyframes` for multi-step, self-starting, or looping animations; always animate compositor-only properties (`transform`, `opacity`) for performance; be cautious with `animation-fill-mode: forwards` since it overrides normal cascade rules; and use scroll-driven `animation-timeline` for scroll-reveal effects instead of JavaScript.
