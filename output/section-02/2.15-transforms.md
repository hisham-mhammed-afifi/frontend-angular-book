# 2.15 Transforms

> CSS transforms move, rotate, scale, and skew elements without affecting layout. They are GPU-accelerated, making them the foundation of performant animations, but they carry hidden side effects that break `position: fixed` and create stacking contexts.

---

## 2D Transform Functions

All 2D transforms are applied via the `transform` property:

```css
.element {
  transform: translate(100px, 50px) rotate(45deg) scale(1.5) skew(10deg);
}
```

### translate()

Moves an element from its current position:

```css
transform: translate(100px, 50px);   /* X and Y */
transform: translateX(100px);        /* X only */
transform: translateY(50px);         /* Y only */
```

**Percentage values refer to the element's own size, not the parent.** This is different from most CSS percentage properties:

```css
/* Moves the element up by exactly its own height */
transform: translateY(-100%);
```

This is the basis of the classic absolute centering trick:

```css
.centered {
  position: absolute;
  top: 50%;
  left: 50%;
  transform: translate(-50%, -50%);
}
```

### rotate()

Rotates around a fixed point (default: center). Accepts `deg`, `rad`, `grad`, or `turn`:

```css
transform: rotate(45deg);     /* clockwise */
transform: rotate(-45deg);    /* counter-clockwise */
transform: rotate(0.5turn);   /* 180 degrees */
```

### scale()

Resizes relative to the original size. `1` = no change, `2` = doubled, `0.5` = halved:

```css
transform: scale(1.5);          /* uniform 150% */
transform: scale(2, 0.5);       /* 200% width, 50% height */
transform: scaleX(1.5);         /* width only */
transform: scaleY(0.5);         /* height only */
```

Negative values create mirror/flip effects:

```css
transform: scaleX(-1);          /* horizontal flip */
transform: scaleY(-1);          /* vertical flip */
transform: scale(-1, -1);       /* both (equivalent to rotate(180deg)) */
```

### skew()

Shears/slants an element on the 2D plane:

```css
transform: skew(15deg);         /* X axis only */
transform: skew(15deg, 5deg);   /* X and Y */
transform: skewX(15deg);        /* X axis only */
transform: skewY(5deg);         /* Y axis only */
```

---

## Function Order Matters

Transform functions are applied **right to left** (like mathematical function composition). The order produces different results:

```css
/* First rotates, THEN translates along the rotated axis */
.a { transform: translateX(200px) rotate(45deg); }

/* First translates along horizontal, THEN rotates in place */
.b { transform: rotate(45deg) translateX(200px); }
```

In `.a`, the element rotates first, then moves 200px along the now-tilted X axis (diagonally). In `.b`, the element moves 200px to the right, then rotates in place.

**Key insight:** After rotation, the element's local coordinate system is rotated too. A subsequent `translateX()` moves along the rotated X axis, not the original horizontal.

---

## transform-origin

The pivot point for all transforms. Default is `50% 50%` (center):

```css
/* Keywords */
transform-origin: center;            /* 50% 50% */
transform-origin: left top;          /* 0% 0% */
transform-origin: right bottom;      /* 100% 100% */
transform-origin: center bottom;     /* 50% 100% */

/* Lengths and percentages */
transform-origin: 0 0;               /* top-left corner */
transform-origin: 25% 75%;           /* custom point */
transform-origin: 90px 30px;         /* exact pixel offset */
```

### Practical Examples

```css
/* Pendulum: swings from top-center */
.pendulum {
  transform-origin: center top;
  animation: swing 2s ease-in-out infinite alternate;
}

/* Door hinge on the left edge */
.door {
  transform-origin: left center;
  transition: transform 0.4s;
}
.door:hover {
  transform: perspective(600px) rotateY(-30deg);
}

/* Scale from top-left corner (like expanding a tooltip) */
.tooltip {
  transform-origin: 0 0;
  transform: scale(0);
  transition: transform 0.2s;
}
.trigger:hover .tooltip {
  transform: scale(1);
}
```

### Three-Value Syntax for 3D

The third value sets the Z offset. It must be a length, not a percentage:

```css
transform-origin: center center 50px;
```

## Individual Transform Properties

Modern CSS provides separate `translate`, `rotate`, and `scale` properties that solve the transform overwrite problem.

### Syntax

```css
.element {
  translate: 100px 50px;       /* x, y */
  translate: 100px 50px 20px;  /* x, y, z */
  rotate: 45deg;               /* z-axis */
  rotate: x 45deg;             /* around x-axis */
  rotate: 1 1 0 45deg;         /* custom axis vector */
  scale: 1.5;                  /* uniform */
  scale: 1.5 0.8;              /* x, y */
}
```

There is no individual `skew` property. Use `transform: skew()` for that.

### Fixed Application Order

Individual properties are always applied in this order, regardless of declaration order in CSS:

1. `translate` (outermost, applied first)
2. `rotate`
3. `scale` (innermost, applied last)

```css
/* These produce the SAME result */
.a { translate: 50px 0; rotate: 30deg; scale: 1.2; }
.b { scale: 1.2; rotate: 30deg; translate: 50px 0; }
```

### Solving the Overwrite Problem

With the traditional `transform` property, different classes or states overwrite each other:

```css
/* Bad: hover completely replaces the translate */
.card {
  transform: translateY(-10px);
}
.card:hover {
  transform: scale(1.05);  /* translateY is LOST */
}
```

Individual properties are independent:

```css
/* Good: each property is separate, no overwriting */
.card {
  translate: 0 -10px;
}
.card:hover {
  scale: 1.05;  /* translate is preserved */
}
```

This is especially valuable in animations where different `@keyframes` can target different properties without conflict:

```css
.element {
  animation: float 3s ease-in-out infinite, spin 10s linear infinite;
}

@keyframes float {
  0%, 100% { translate: 0 0; }
  50%      { translate: 0 -20px; }
}

@keyframes spin {
  to { rotate: 1turn; }
}
```

With the traditional `transform` property, these two animations would conflict because both write to the same property.

### Additive with transform

When both individual properties and `transform` are set, they are additive. Individual properties apply first, then `transform` functions:

```css
.element {
  translate: 100px 0;           /* applied first */
  transform: rotate(45deg);     /* applied after translate */
}
```

## 3D Transforms

### 3D Rotation

```css
transform: rotateX(45deg);      /* tilt forward/backward */
transform: rotateY(45deg);      /* turn left/right */
transform: rotateZ(45deg);      /* same as rotate(45deg) */
transform: rotate3d(1, 1, 0, 45deg);  /* custom axis */
```

`rotate3d(x, y, z, angle)` rotates around a vector defined by the x/y/z components. `rotate3d(1, 0, 0, 45deg)` equals `rotateX(45deg)`.

### 3D Translation

```css
transform: translateZ(50px);           /* toward viewer */
transform: translateZ(-50px);          /* away from viewer */
transform: translate3d(100px, 50px, 30px);  /* x, y, z */
```

`translateZ()` requires `perspective` to have any visible effect. Elements with positive Z values appear larger; negative Z values appear smaller.

### 3D Scaling

```css
transform: scaleZ(2);                  /* scale along Z axis */
transform: scale3d(1.5, 1, 2);        /* x, y, z */
```

`scaleZ()` alone has no visible effect. It only becomes visible when combined with other 3D transforms like `rotateX` or `rotateY`.

---

## perspective

Perspective defines how far the viewer is from the z=0 plane. Lower values create more extreme 3D effects; higher values are subtler.

### Property vs. Function

| Approach | Scope | Vanishing Point |
|----------|-------|-----------------|
| `perspective` property (on parent) | Shared 3D space for all children | Single shared vanishing point |
| `perspective()` function (on element) | Per-element | Each element has its own |

### perspective Property (Parent)

```css
.scene {
  perspective: 800px;
}

.scene .card {
  transform: rotateY(30deg);  /* rendered in the shared 3D space */
}
```

All children share the same vanishing point, so they appear to exist in the same 3D scene.

### perspective() Function (Element)

```css
.card {
  transform: perspective(800px) rotateY(30deg);
}
```

Each element gets its own vanishing point. Sibling elements will NOT appear to exist in the same 3D space.

**The function must come FIRST in the transform chain:**

```css
/* Correct */
transform: perspective(800px) rotateY(45deg);

/* Wrong: perspective applied after rotation, unexpected result */
transform: rotateY(45deg) perspective(800px);
```

### perspective-origin

Shifts the vanishing point for the `perspective` property:

```css
.scene {
  perspective: 800px;
  perspective-origin: 25% 50%;  /* viewer looking from the left */
}
```

Default is `50% 50%` (centered). Only works with the property, not the function.

### Choosing Values

| Value | Effect | Use Case |
|-------|--------|----------|
| 100-300px | Extreme, dramatic 3D | Artistic effects |
| 400-800px | Moderate, natural 3D | Card flips, UI elements |
| 800-1500px | Subtle, barely noticeable | Gentle depth cues |

## transform-style

Controls whether children render in 3D or are flattened:

| Value | Behavior |
|-------|----------|
| `flat` (default) | Children are projected onto the parent's 2D plane |
| `preserve-3d` | Children maintain their own 3D positions |

```css
.cube {
  transform-style: preserve-3d;
  transform: rotateX(-20deg) rotateY(30deg);
}

.cube .face {
  position: absolute;
  /* Each face exists in 3D space relative to the cube */
}
```

### The Flattening Properties Gotcha

Several CSS properties force `transform-style` to `flat` even if you explicitly set `preserve-3d`:

| Property | Value That Forces Flat |
|----------|-----------------------|
| `overflow` | Anything except `visible` |
| `opacity` | Less than `1` |
| `filter` | Anything except `none` |
| `clip-path` | Anything except `none` |
| `mask` / `mask-image` | Anything except `none` |
| `mix-blend-mode` | Anything except `normal` |
| `isolation` | `isolate` |
| `contain` | `paint` or `layout` |

This is one of the most common 3D CSS debugging headaches. You cannot use `overflow: hidden` and `preserve-3d` on the same element:

```css
/* Bug: overflow kills 3D rendering */
.container {
  transform-style: preserve-3d;
  overflow: hidden;  /* Forces flat! 3D is broken. */
}

/* Fix: move overflow to a wrapper */
.overflow-wrapper {
  overflow: hidden;
}
.overflow-wrapper .container {
  transform-style: preserve-3d;
}
```

`transform-style` is NOT inherited. Set it on every intermediate element in a nested 3D structure.

## backface-visibility

Controls whether the back of an element is visible when rotated to face away from the viewer:

```css
backface-visibility: visible;  /* default: back face shows (mirrored) */
backface-visibility: hidden;   /* back face is invisible */
```

### Card Flip Pattern

The classic 3D card flip:

```css
.scene {
  perspective: 600px;
}

.card {
  position: relative;
  width: 300px;
  height: 200px;
  transform-style: preserve-3d;
  transition: transform 0.6s;
}

.card:hover {
  transform: rotateY(180deg);
}

.card-face {
  position: absolute;
  inset: 0;
  backface-visibility: hidden;
}

.card-front {
  /* Faces viewer by default */
}

.card-back {
  transform: rotateY(180deg);  /* Pre-rotated to face away */
}
```

When `.card` rotates 180deg, `.card-front` faces away (hidden) and `.card-back` faces the viewer.

Safari may still require `-webkit-backface-visibility: hidden` for full compatibility.

## Transform Side Effects

### Creates a Stacking Context

Any element with `transform` (other than `none`) creates a new stacking context. `z-index` on descendants only competes within this context, not the parent's context. Even `transform: translateX(0)` triggers this.

### Breaks position: fixed

**This is one of the most notorious CSS gotchas.** Any element with `transform`, `filter`, or `perspective` becomes the containing block for ALL descendants, including `position: fixed` elements:

```css
.parent {
  transform: translateZ(0);  /* or any non-none transform */
}

/* .modal is supposed to be fixed to the viewport */
.parent .modal {
  position: fixed;
  top: 0;
  left: 0;
  /* BUG: positions relative to .parent, NOT the viewport */
  /* Scrolls with the page instead of staying fixed */
}
```

This frequently catches developers who apply `transform: translateZ(0)` or `will-change: transform` to wrapper elements for GPU acceleration. The fix is to move the fixed element outside the transformed ancestor in the DOM.

### Breaks position: absolute Containment

A transformed element also becomes the containing block for `position: absolute` children, even if the transformed element itself is `position: static`.

## matrix() and matrix3d()

Every transform ultimately computes to a matrix. Browser DevTools show the computed `matrix()` or `matrix3d()` value.

### matrix() (2D)

Six values representing a 2x3 affine transformation:

```css
transform: matrix(a, b, c, d, tx, ty);
```

| Parameter | Maps To |
|-----------|---------|
| `a` | scaleX |
| `b` | skewY |
| `c` | skewX |
| `d` | scaleY |
| `tx` | translateX |
| `ty` | translateY |

```css
/* These are equivalent */
.a { transform: translate(100px, 50px) scale(2, 1.5); }
.b { transform: matrix(2, 0, 0, 1.5, 100, 50); }
```

Identity (no transform): `matrix(1, 0, 0, 1, 0, 0)`.

### matrix3d() (3D)

Sixteen values representing a 4x4 homogeneous transformation matrix in column-major order. Rarely hand-written; primarily used for programmatic generation via JavaScript or for reading computed values in DevTools.

## transform-box (SVG)

Defines the reference box for `transform` and `transform-origin` calculations:

```css
transform-box: content-box;   /* HTML default */
transform-box: border-box;
transform-box: fill-box;      /* SVG object bounding box */
transform-box: stroke-box;    /* SVG stroke bounding box */
transform-box: view-box;      /* SVG viewport (SVG default) */
```

For SVG elements, the default `view-box` means `transform-origin: center` refers to the center of the SVG viewport, not the individual shape. To make transforms relative to the shape itself:

```css
.svg-icon path {
  transform-box: fill-box;
  transform-origin: center;
  animation: spin 2s linear infinite;
}
```

## Performance

### Why Transforms Are Fast

`transform` and `opacity` are the only properties that can be composited on the GPU without triggering layout or paint. Animating `transform` does not affect surrounding elements (no reflow).

### Layer Promotion

When the browser detects `transform` will be animated, it promotes the element to its own GPU compositing layer. `will-change: transform` tells the browser to promote in advance:

```css
.card {
  will-change: transform;  /* pre-promote to GPU layer */
}
```

Rules for `will-change`:
- Apply only to elements that will actually animate.
- Remove after animation completes when possible.
- Never apply to all elements. Each layer consumes GPU memory.

### Use Transforms Instead of Layout Properties

```css
/* Bad: triggers layout recalculation every frame */
@keyframes bad-move {
  to { left: 200px; top: 100px; }
}

/* Good: GPU-composited, no layout recalculation */
@keyframes good-move {
  to { transform: translate(200px, 100px); }
}
```

## Common Mistakes

1. **Transforms on inline elements.** `transform` has no effect on `display: inline` elements. Use `inline-block`, `block`, or a flex/grid child instead.

2. **Transform overwrite in multiple classes.** Adding a class with `transform: scale(1.1)` wipes out any existing `transform: translateY(-10px)`. Use individual properties (`translate`, `rotate`, `scale`) to avoid this.

3. **`position: fixed` inside a transformed ancestor.** The fixed element positions relative to the transformed parent, not the viewport. Move the fixed element outside the transformed tree.

4. **`overflow: hidden` killing `preserve-3d`.** These cannot coexist on the same element. Move `overflow: hidden` to a separate wrapper.

5. **`perspective()` not first in the transform chain.** `transform: rotateY(45deg) perspective(500px)` does not produce the expected result. Always put `perspective()` first.

6. **Percentage translate referring to the element, not the parent.** `translate(50%)` moves by 50% of the element's own size, not the containing block.

7. **Forgetting that rotation changes the coordinate system.** After `rotate(45deg)`, a subsequent `translateX()` moves along the rotated axis, not horizontal.

8. **Animating with `@keyframes` and losing transform functions.** If one keyframe has `transform: translateX(100px) scale(1.5)` and another has only `transform: rotate(45deg)`, the translate and scale are lost. Each keyframe must redeclare all functions, or use individual properties.

9. **Applying `will-change: transform` everywhere.** Each promoted layer consumes GPU VRAM. On mobile, too many layers degrades performance or crashes the browser.

---

## Best Practices

1. **Use individual `translate`, `rotate`, `scale` properties** in new projects to avoid the overwrite problem and enable independent animation of each.

2. **Use `transform` and `opacity` for all animations.** They are the only truly compositor-only properties, guaranteed to run at 60fps.

3. **Use the `perspective` property on a parent** when multiple children share a 3D scene. Use `perspective()` function only for single-element 3D effects.

4. **Set `transform-box: fill-box` on SVG elements** to make transforms relative to the shape instead of the viewport.

5. **Audit for side effects** when adding transforms. Check for `position: fixed` descendants and existing stacking context dependencies.

6. **Prefer `will-change: transform` over `translateZ(0)` hacks** for GPU layer promotion, and only apply it to elements that need it.

## Real-World Use Cases

### 1. Card Flip with 3D Transforms

```css
.flip-scene {
  perspective: 600px;
  width: 300px;
  height: 200px;
}

.flip-card {
  position: relative;
  width: 100%;
  height: 100%;
  transform-style: preserve-3d;
  transition: transform 0.6s ease;
}

.flip-scene:hover .flip-card {
  transform: rotateY(180deg);
}

.flip-face {
  position: absolute;
  inset: 0;
  backface-visibility: hidden;
  border-radius: 12px;
  display: flex;
  align-items: center;
  justify-content: center;
}

.flip-front {
  background: oklch(55% 0.2 250);
  color: white;
}

.flip-back {
  background: oklch(55% 0.2 150);
  color: white;
  transform: rotateY(180deg);
}
```

### 2. Independent Animations on Individual Properties

```css
.floating-icon {
  animation:
    bob 3s ease-in-out infinite alternate,
    gentle-spin 12s linear infinite;
}

@keyframes bob {
  from { translate: 0 0; }
  to   { translate: 0 -15px; }
}

@keyframes gentle-spin {
  to { rotate: 1turn; }
}

@media (prefers-reduced-motion: reduce) {
  .floating-icon {
    animation: none;
  }
}
```

No conflict between the two animations because they target different individual properties.

### 3. Performant Hover Lift with Shadow

```css
.product-card {
  translate: 0 0;
  transition: translate 0.25s ease;
  position: relative;
}

.product-card::after {
  content: '';
  position: absolute;
  inset: 0;
  border-radius: inherit;
  box-shadow: 0 12px 24px hsl(220 60% 15% / 0.2);
  opacity: 0;
  transition: opacity 0.25s ease;
  z-index: -1;
  pointer-events: none;
}

.product-card:hover {
  translate: 0 -4px;
}

.product-card:hover::after {
  opacity: 1;
}
```

Only `translate` (compositor) and `opacity` (compositor) are animated. No layout or paint cost.

---

> **Key Takeaway:** Use individual `translate`, `rotate`, and `scale` properties to avoid the overwrite problem, always animate transforms (not layout properties) for GPU performance, and watch for the hidden side effects: stacking context creation, `position: fixed` breakage, and `preserve-3d` flattening by `overflow: hidden`.
