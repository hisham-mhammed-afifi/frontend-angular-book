# 2.9 Colors

> CSS color is no longer just hex codes. Modern CSS provides perceptually uniform color spaces, color mixing, relative color manipulation, and dark mode primitives that eliminate the need for preprocessor color functions.

---

## Color Formats: The Full Landscape

CSS supports more color formats than most developers realize. Here is every format you can use today, with syntax and tradeoffs.

### Hex

```css
color: #3b82f6;         /* 6-digit hex (RGB) */
color: #3b82f680;       /* 8-digit hex (RGBA) */
color: #38f;            /* 3-digit shorthand */
color: #38f8;           /* 4-digit shorthand with alpha */
```

Hex is compact but unreadable. You cannot tell at a glance what `#3b82f6` looks like, and you cannot adjust it without a tool.

### rgb() and rgba()

```css
color: rgb(59, 130, 246);          /* legacy comma syntax */
color: rgb(59 130 246);            /* modern space syntax */
color: rgb(59 130 246 / 0.5);     /* alpha with modern syntax */
```

The modern syntax uses spaces instead of commas and a slash for alpha. The `rgba()` function still works but is redundant since `rgb()` now accepts alpha.

### hsl() and hsla()

```css
color: hsl(217, 91%, 60%);        /* legacy comma syntax */
color: hsl(217 91% 60%);          /* modern space syntax */
color: hsl(217 91% 60% / 0.5);   /* alpha */
```

HSL was the first human-readable format. Hue is a degree on the color wheel (0-360), saturation is intensity (0%-100%), lightness is brightness (0%-100%). But HSL has a fundamental flaw: its lightness channel lies.

### hwb()

```css
color: hwb(217 10% 4%);           /* hue whiteness blackness */
color: hwb(217 10% 4% / 0.5);    /* with alpha */
```

HWB stands for Hue, Whiteness, Blackness. It describes how you would mix paint: start with a pure hue, add white, add black. Some people find this more intuitive than HSL.

### oklch() -- The Modern Default

```css
color: oklch(65% 0.2 250);        /* lightness chroma hue */
color: oklch(65% 0.2 250 / 0.5); /* with alpha */
```

OKLCH is the cylindrical form of the OKLab color space. Three channels:
- **L** (Lightness): 0% to 100%. Perceptually uniform.
- **C** (Chroma): 0 to ~0.4. Saturation/vibrancy.
- **H** (Hue): 0 to 360 degrees.

### oklab()

```css
color: oklab(65% -0.1 0.15);      /* lightness a-axis b-axis */
```

OKLab uses Cartesian coordinates instead of polar. OKLCH is preferred for human authoring because hue and chroma are more intuitive than a/b axes.

### color() -- Predefined Color Spaces

```css
color: color(display-p3 1 0.5 0);          /* Display-P3 */
color: color(srgb 0.23 0.51 0.96);         /* sRGB (0-1 scale) */
color: color(srgb-linear 0.04 0.21 0.91);  /* linear-light sRGB */
```

The `color()` function accesses predefined color spaces: `srgb`, `srgb-linear`, `display-p3`, `a98-rgb`, `prophoto-rgb`, `rec2020`, `xyz`, `xyz-d50`, `xyz-d65`. Channel values range from 0 to 1.

## Format Comparison Table

| Format | Gamut | Human-Readable | Perceptually Uniform | Wide Gamut | Best For |
|--------|-------|----------------|---------------------|------------|----------|
| `#hex` | sRGB | No | No | No | Legacy code, design tool copy-paste |
| `rgb()` | sRGB | Somewhat | No | No | JS interop, programmatic manipulation |
| `hsl()` | sRGB | Yes | No (L lies) | No | Quick manual tweaks |
| `hwb()` | sRGB | Yes | No | No | Paint-mixing mental model |
| `oklch()` | Wide | Yes | Yes | Yes | New projects, design systems, palettes |
| `oklab()` | Wide | No | Yes | Yes | Programmatic blending |
| `color(display-p3)` | P3 | Somewhat | No | P3 only | Targeting P3 displays specifically |

## Why HSL's Lightness Lies

This is the single most important reason to move beyond HSL. Consider these two HSL colors:

```css
.yellow { background: hsl(60, 100%, 50%); }   /* bright yellow */
.blue   { background: hsl(240, 100%, 50%); }  /* dark blue */
```

Both claim 50% lightness. But yellow looks far brighter than blue to the human eye. HSL's lightness is mathematically derived from RGB channel values, not from how humans perceive brightness.

Now compare in OKLCH:

```css
.yellow { background: oklch(50% 0.2 100); }  /* actually mid-brightness */
.blue   { background: oklch(50% 0.2 260); }  /* actually mid-brightness */
```

Both look equally bright. OKLCH's L channel maps to perceived luminance, making it reliable for building consistent color palettes.

## OKLCH Deep Dive

### Palette Generation

The killer feature of OKLCH is that you can build an entire color scale by adjusting only lightness:

```css
:root {
  /* Same hue and chroma, only lightness changes */
  --blue-50:  oklch(95% 0.03 250);
  --blue-100: oklch(90% 0.06 250);
  --blue-200: oklch(80% 0.12 250);
  --blue-300: oklch(70% 0.18 250);
  --blue-400: oklch(65% 0.25 250);
  --blue-500: oklch(55% 0.25 250);
  --blue-600: oklch(45% 0.22 250);
  --blue-700: oklch(35% 0.18 250);
  --blue-800: oklch(25% 0.12 250);
  --blue-900: oklch(15% 0.06 250);
}
```

Notice that chroma decreases at the extremes. This is because very light and very dark colors cannot physically sustain high chroma. Trying to force high chroma at extreme lightness values triggers gamut clipping.

### Maximum Chroma Varies by Hue

There is no single safe maximum chroma value. Yellow can sustain higher chroma than blue at the same lightness:

```css
/* This works -- yellow at high chroma */
color: oklch(85% 0.25 95);

/* This gets clipped -- blue at high chroma */
color: oklch(85% 0.25 260);  /* browser clips to nearest displayable color */
```

When the browser clips, the result may not be what you expect. Keep chroma values moderate (0.1-0.2) for predictable results, and push higher only when you have verified the output visually.

### Gradient Interpolation in OKLCH

By default, gradients interpolate in sRGB, which produces a "gray dead zone" between vivid colors:

```css
/* Bad: sRGB interpolation creates muddy gray midpoint */
.gradient-bad {
  background: linear-gradient(to right, blue, yellow);
}

/* Good: OKLCH interpolation stays vibrant */
.gradient-good {
  background: linear-gradient(in oklch to right, blue, yellow);
}
```

You can also control the hue interpolation direction:

```css
/* Short arc (fewer degrees between hues) */
background: linear-gradient(in oklch shorter hue to right, red, blue);

/* Long arc (more degrees, passes through intermediate hues) */
background: linear-gradient(in oklch longer hue to right, red, blue);
```

### OKLCH Gradient Bug with Neutrals

Gradients from a chromatic color to white, black, or transparent in OKLCH can produce unexpected color fringing (teal or purple artifacts):

```css
/* Bug: may show color artifacts near white */
.broken {
  background: linear-gradient(in oklch, oklch(65% 0.25 250), white);
}

/* Fix: give the neutral endpoint the same hue with near-zero chroma */
.fixed {
  background: linear-gradient(
    in oklch,
    oklch(65% 0.25 250),
    oklch(100% 0.001 250)
  );
}
```

The issue occurs because `white` has no hue information, so the browser interpolates through an arbitrary hue on the way to zero chroma.

## Wide Gamut Colors

### What Wide Gamut Means

Display-P3 is a color space roughly 50% larger in volume than sRGB. Modern Apple displays (since iPhone 7, MacBook Pro 2016), Google Pixel devices, and most HDR monitors support P3. This means they can show colors more vivid than anything sRGB can express.

Hex, `rgb()`, and `hsl()` are locked to sRGB. To access wider gamut colors, use `oklch()`, `oklab()`, `lch()`, `lab()`, or `color(display-p3 ...)`.

### Using Display-P3

```css
/* Vivid orange impossible in sRGB */
.hero-bg {
  /* sRGB fallback */
  background: rgb(255, 100, 0);
  /* Wide gamut override */
  background: color(display-p3 1 0.4 0);
}
```

The `color()` function takes channel values from 0 to 1. To convert from RGB 0-255: divide by 255.

### Feature Detection

```css
/* Detect if the display supports P3 */
@media (color-gamut: p3) {
  :root {
    --accent: color(display-p3 0 0.9 0.25);
  }
}

/* Detect if the browser understands the syntax */
@supports (color: color(display-p3 0 0 0)) {
  .badge {
    background: color(display-p3 0.9 0.2 0.5);
  }
}
```

`@media (color-gamut: p3)` checks the display hardware. `@supports` checks the CSS parser.

### OKLCH Covers Wide Gamut Automatically

If you are already using OKLCH, you get wide gamut access without `color()`:

```css
/* This OKLCH color falls outside sRGB but within Display-P3 */
color: oklch(70% 0.3 150);
```

The browser automatically maps the color to the widest gamut the display can handle. On an sRGB-only display, it gamut-maps to the nearest sRGB equivalent.

## color-mix()

`color-mix()` blends two colors in a specified color space. It is the CSS equivalent of Sass's `mix()`, `darken()`, and `lighten()` functions combined.

### Syntax

```css
color-mix(in <color-space>, <color> <percentage>?, <color> <percentage>?)
```

### Basic Mixing

```css
/* 50/50 mix (default) */
background: color-mix(in oklch, blue, red);

/* 25% blue, 75% red */
background: color-mix(in oklch, blue 25%, red);

/* Equivalent: explicit both percentages */
background: color-mix(in oklch, blue 25%, red 75%);
```

### Darken and Lighten

```css
.btn {
  --color: oklch(60% 0.2 250);
  background: var(--color);
}

/* Darken by mixing with black */
.btn:hover {
  background: color-mix(in oklch, var(--color), black 20%);
}

/* Lighten by mixing with white */
.btn:disabled {
  background: color-mix(in oklch, var(--color), white 60%);
}
```

### The Transparency Trick

One of the most useful patterns: add opacity to any color, including CSS custom properties:

```css
:root {
  --brand: oklch(55% 0.2 250);

  /* Opacity variants without decomposing the color */
  --brand-a10: color-mix(in srgb, var(--brand), transparent 90%);
  --brand-a20: color-mix(in srgb, var(--brand), transparent 80%);
  --brand-a50: color-mix(in srgb, var(--brand), transparent 50%);
}

/* Usage */
.overlay {
  background: var(--brand-a20);
}
```

This is significant because there was previously no way to add alpha to a custom property color. You could not do `rgba(var(--brand), 0.5)` since `var(--brand)` is not three separate numbers.

### Color Space Choice Matters

```css
/* Muddy gray midpoint in sRGB */
background: color-mix(in srgb, blue, yellow);

/* Vibrant midpoint in OKLCH */
background: color-mix(in oklch, blue, yellow);
```

Default to `in oklch` for perceptually uniform results.

### With currentColor

```css
.icon {
  /* 50% transparent version of the parent text color */
  fill: color-mix(in srgb, currentColor 50%, transparent);
}
```

## Relative Color Syntax

Relative color syntax lets you derive a new color from an existing one by decomposing it into channels and modifying them with `calc()`.

### Syntax

```css
color-function(from <origin-color> channel1 channel2 channel3 / alpha)
```

The origin color is converted to the target color function's space, and the channel variables become available.

### Lighten and Darken

```css
.lighter {
  /* Increase lightness by 25% */
  background: oklch(from var(--brand) calc(l * 1.25) c h);
}

.darker {
  /* Decrease lightness by 25% */
  background: oklch(from var(--brand) calc(l * 0.75) c h);
}
```

### Adjust Saturation

```css
.muted {
  /* Halve the chroma */
  background: oklch(from var(--brand) l calc(c * 0.5) h);
}

.vivid {
  /* Double the chroma (may get clipped) */
  background: oklch(from var(--brand) l calc(c * 2) h);
}
```

### Shift Hue

```css
.complement {
  /* Complementary color: rotate 180 degrees */
  background: oklch(from var(--brand) l c calc(h + 180));
}

.analogous {
  /* Analogous color: rotate 30 degrees */
  background: oklch(from var(--brand) l c calc(h + 30));
}
```

### Adjust Only Opacity

```css
.semi-transparent {
  /* Keep all channels, change only alpha */
  background: rgb(from var(--brand) r g b / 0.5);
}
```

### Build a Full Palette from One Base

```css
:root {
  --base: oklch(60% 0.2 250);

  --light:   oklch(from var(--base) calc(l + 0.2) calc(c * 0.5) h);
  --dark:    oklch(from var(--base) calc(l - 0.2) c h);
  --accent:  oklch(from var(--base) l c calc(h + 30));
  --muted:   oklch(from var(--base) l calc(c * 0.3) h);
  --surface: oklch(from var(--base) 95% calc(c * 0.1) h);
}
```

### Gotchas

Channel names belong to the output function, not the input color. When you write `oklch(from red l c h)`, red is first converted to OKLCH, then `l`, `c`, `h` become available. You cannot reference OKLCH channels inside an `rgb()` function.

## currentColor

`currentColor` resolves to the computed value of the `color` property on the same element (or inherited from a parent). It is effectively a built-in variable.

### SVG Icons Inheriting Text Color

```css
.nav-link {
  color: #3b82f6;
}

.nav-link svg {
  fill: currentColor;  /* automatically #3b82f6 */
}

.nav-link:hover {
  color: #1d4ed8;
  /* SVG fill updates automatically */
}
```

### Properties That Already Default to currentColor

You do not need to write `currentColor` explicitly for these properties:

| Property | Default |
|----------|---------|
| `border-color` | `currentColor` |
| `outline-color` | `currentColor` |
| `text-decoration-color` | `currentColor` |
| `column-rule-color` | `currentColor` |
| `caret-color` | `currentColor` |
| `text-emphasis-color` | `currentColor` |

This means `border: 1px solid` already uses the text color for the border. No need for `border: 1px solid currentColor`.

### Where currentColor Is Useful

```css
.tag {
  color: var(--tag-color);
  border: 1px solid;               /* already uses currentColor */
  box-shadow: 0 0 0 2px currentColor;  /* explicit, not default */
  background: color-mix(in srgb, currentColor 10%, transparent);
}
```

`box-shadow` color does NOT default to `currentColor` in all browsers. Always specify it explicitly.

## Dark Mode: color-scheme and light-dark()

### color-scheme Property

`color-scheme` tells the browser which color schemes an element supports. It affects browser-provided UI: form controls, scrollbars, the default canvas background, and default text color.

```css
:root {
  /* Support both, prefer light */
  color-scheme: light dark;
}
```

Without this declaration, the browser assumes light mode only. Users who prefer dark mode will still see white form controls and scrollbars.

### Per-Element Override

```css
.code-block {
  /* Always render this widget in dark mode */
  color-scheme: dark;
  background: #1e1e1e;
  color: #e0e0e0;
}
```

### color-scheme vs prefers-color-scheme

| Feature | `color-scheme` | `prefers-color-scheme` |
|---------|---------------|----------------------|
| Controls | Browser/UA defaults (forms, scrollbars, canvas) | Your custom CSS rules |
| Mechanism | CSS property declaration | Media query |
| Scope | Per-element (inherited) | Global, based on OS setting |

They are complementary. Use both:

```css
:root {
  color-scheme: light dark;  /* browser defaults follow OS */
}

@media (prefers-color-scheme: dark) {
  .card {
    background: #1e1e1e;     /* your custom dark styles */
  }
}
```

### light-dark() Function

`light-dark()` returns one of two color values based on the active color scheme. It eliminates the need for separate media query blocks for color declarations.

```css
:root {
  color-scheme: light dark;

  --bg:      light-dark(#ffffff, #121212);
  --text:    light-dark(#1a1a1a, #e0e0e0);
  --primary: light-dark(#2563eb, #60a5fa);
  --surface: light-dark(#f3f4f6, #1e1e1e);
  --border:  light-dark(#d1d5db, #374151);
}

body {
  background: var(--bg);
  color: var(--text);
}
```

### light-dark() Requirements

1. `color-scheme: light dark` must be set on `:root` or an ancestor. Without it, `light-dark()` always returns the first (light) value.
2. `light-dark()` only works for `<color>` values. You cannot use it for `padding`, `font-size`, or other non-color properties. For those, you still need `@media (prefers-color-scheme: dark)`.
3. It cannot be nested: `light-dark(light-dark(...), ...)` is invalid.

### The Meta Tag

For optimal dark mode, add this to your HTML `<head>`:

```html
<meta name="color-scheme" content="light dark">
```

This tells the browser to apply dark defaults immediately, before CSS loads. Without it, users may see a flash of white before your dark styles apply.

## accent-color

`accent-color` re-tints browser-styled form controls without fully replacing them with custom CSS.

### Affected Controls

- `<input type="checkbox">`
- `<input type="radio">`
- `<input type="range">`
- `<progress>`

### Usage

```css
/* Global accent for all form controls */
:root {
  accent-color: oklch(55% 0.2 250);
}

/* Per-element override */
input[type="range"].danger {
  accent-color: oklch(55% 0.2 25);
}
```

The browser automatically picks a contrasting foreground color (white or black checkmark) for accessibility. You cannot control this foreground color.

### What accent-color Does NOT Affect

`<select>`, `<button>`, `<input type="text">`, `<textarea>`, and dropdown menus are unaffected. For those, you need full custom styling.

## System Colors and Forced Colors

### System Colors

CSS system color keywords map to the user's OS color preferences. They are critical for accessibility, particularly Windows High Contrast Mode.

| Keyword | Maps To |
|---------|---------|
| `Canvas` | Default page background |
| `CanvasText` | Default text color |
| `LinkText` | Unvisited link color |
| `VisitedText` | Visited link color |
| `ActiveText` | Active link color |
| `ButtonFace` | Button background |
| `ButtonText` | Button text |
| `ButtonBorder` | Button border |
| `Field` | Input field background |
| `FieldText` | Input field text |
| `Highlight` | Selection background |
| `HighlightText` | Selection text |
| `Mark` | Highlighted text background |
| `MarkText` | Highlighted text color |
| `GrayText` | Disabled text |
| `AccentColor` | System accent color |
| `AccentColorText` | Text on accent color |

### Forced Colors Mode

Windows High Contrast Mode triggers forced-colors mode. The browser overrides your colors with the user's chosen system colors.

```css
@media (forced-colors: active) {
  .card {
    /* box-shadow is forced to none in this mode */
    box-shadow: none;
    border: 2px solid ButtonBorder;
  }

  :focus-visible {
    outline: 2px solid Highlight;
    outline-offset: 2px;
  }

  /* Ensure custom indicators remain visible */
  .badge::before {
    background: Highlight;
  }
}
```

In forced-colors mode, the browser forces `box-shadow` to `none`, `background-image` (non-URL) to `none`, and overrides `color`, `background-color`, and `border-color` to system colors.

Do NOT design a separate layout for forced-colors users. Use the media query only for small fixes: replacing shadows with borders, ensuring custom indicators remain visible.

### forced-color-adjust

```css
/* Opt out of forced color adjustment for a specific element */
.brand-logo {
  forced-color-adjust: none;
}
```

Use this sparingly. Only opt out when you are certain the element will remain accessible without forced color overrides.

## contrast-color() (Future)

`contrast-color()` automatically picks either black or white, whichever has higher contrast against a given background:

```css
/* Not yet shipping in stable browsers */
.badge {
  background: var(--badge-color);
  color: contrast-color(var(--badge-color));
}
```

### Current Status

- Originally proposed as `color-contrast()` in CSS Color Level 5.
- Renamed to `contrast-color()` and moved to CSS Color Level 6.
- Part of Interop 2026, meaning browser vendors are actively implementing it.
- Currently only available in Safari Technology Preview.

### Workaround Until It Ships

Use `color-mix()` with relative color syntax, or compute contrast ratios in JavaScript. Some CSS-only approximations use OKLCH lightness thresholds:

```css
/* Approximate: if lightness > 60%, use dark text; otherwise light text */
/* This requires per-element logic and is not a true replacement */
```

For now, manually set text colors against backgrounds.

## Named Color Gotchas

CSS named colors carry historical baggage from the X11 color system. Some names are misleading:

| Name | Hex | Actual Brightness |
|------|-----|------------------|
| `gray` | `#808080` | 50% |
| `darkgray` | `#A9A9A9` | 66% (lighter than `gray`) |
| `lightgray` | `#D3D3D3` | 83% |
| `dimgray` | `#696969` | 41% (actually the darkest) |

The reason: the W3C had `gray` as `#808080`, but adopted X11's `darkgray` (`#A9A9A9`) and `lightgray` (`#D3D3D3`) unchanged. X11 had its own `gray` at `#BEBEBE`, relative to which `darkgray` was indeed darker.

Other oddities:
- `lime` is `#00FF00` (pure green). `limegreen` is `#32CD32` (duller).
- `rebeccapurple` (`#663399`) was added to CSS Color Level 4 in memory of Eric Meyer's daughter Rebecca.
- There is `mediumvioletred` but no `violetred`.

Avoid named colors in production code. Use them only for quick prototyping.

## Common Mistakes

1. **Using HSL lightness for consistent palettes.** HSL's L channel is not perceptually uniform. Two colors at `hsl(_, _, 50%)` with different hues look drastically different in brightness. Use OKLCH instead.

2. **Setting `color` without `background-color` (or vice versa).** If you set only the foreground color, user stylesheets or browser defaults may create unreadable combinations. If you set one, set both.

3. **Using `opacity` when you want transparent color.** `opacity` makes the entire element and all its children transparent. Use alpha channels or `color-mix(in srgb, var(--color), transparent 50%)` for color-only transparency.

4. **Relying on color alone to convey meaning.** Links distinguished only by color are invisible to colorblind users. Error states shown only with red text fail for red-green color blindness. Always add underlines, icons, patterns, or other non-color cues.

5. **Not providing sRGB fallbacks for wide gamut colors.** Older browsers do not understand `oklch()` or `color(display-p3 ...)`. Stack declarations:
   ```css
   .hero {
     background: #3b82f6;                    /* sRGB fallback */
     background: oklch(55% 0.2 250);        /* modern override */
   }
   ```

6. **Forgetting `color-scheme` when using `light-dark()`.** Without `color-scheme: light dark` on `:root`, `light-dark()` always returns the first (light mode) value, regardless of the user's OS preference.

7. **Pushing high chroma at extreme lightness in OKLCH.** Very light or very dark colors cannot sustain high chroma. The browser clips to the nearest displayable color, producing unpredictable results.

8. **OKLCH gradients to pure neutrals.** Gradients from a chromatic OKLCH color to `white`, `black`, or `transparent` can produce hue artifacts. Match the hue and use near-zero chroma for neutral endpoints.

## Best Practices

1. **Default to OKLCH for new projects.** It is perceptually uniform, supports wide gamut, and is human-readable. Use hex/rgb only when interfacing with legacy tools or JS APIs.

2. **Use semantic color tokens, not raw values.**
   ```css
   :root {
     /* Primitive scale */
     --blue-500: oklch(55% 0.2 250);
     --blue-600: oklch(45% 0.22 250);

     /* Semantic tokens */
     --color-primary: var(--blue-500);
     --color-primary-hover: var(--blue-600);
     --color-text: light-dark(oklch(15% 0.02 250), oklch(90% 0.02 250));
     --color-surface: light-dark(oklch(98% 0.005 250), oklch(15% 0.01 250));
   }
   ```

3. **Use `color-mix()` for state variants** (hover, active, disabled) instead of hardcoding separate values. This keeps your palette mathematically consistent and reduces the number of tokens.

4. **Use `light-dark()` for dark mode colors.** It is cleaner than duplicating every custom property in a `prefers-color-scheme` media query.

5. **Always set `color-scheme: light dark` on `:root`** and add `<meta name="color-scheme" content="light dark">` in your HTML head.

6. **Test contrast ratios.** WCAG AA requires 4.5:1 for normal text and 3:1 for large text. WCAG AAA requires 7:1 for normal text. Chrome DevTools shows contrast ratios in the color picker.

7. **Interpolate gradients in OKLCH.** Add `in oklch` to `linear-gradient()`, `radial-gradient()`, and `conic-gradient()` to avoid the gray dead zone.

## Real-World Use Cases

### 1. Design System Color Scale

Generate an entire brand color scale from one OKLCH base color:

```css
:root {
  --brand-hue: 250;
  --brand-chroma: 0.2;

  --brand-50:  oklch(97% 0.02 var(--brand-hue));
  --brand-100: oklch(93% 0.04 var(--brand-hue));
  --brand-200: oklch(85% 0.08 var(--brand-hue));
  --brand-300: oklch(75% 0.14 var(--brand-hue));
  --brand-400: oklch(65% var(--brand-chroma) var(--brand-hue));
  --brand-500: oklch(55% var(--brand-chroma) var(--brand-hue));
  --brand-600: oklch(45% 0.18 var(--brand-hue));
  --brand-700: oklch(35% 0.14 var(--brand-hue));
  --brand-800: oklch(25% 0.10 var(--brand-hue));
  --brand-900: oklch(15% 0.06 var(--brand-hue));
}
```

Change `--brand-hue` to a different value and the entire scale updates.

### 2. Dark Mode with light-dark() and color-mix()

```css
:root {
  color-scheme: light dark;

  --brand: oklch(55% 0.2 250);

  --bg:           light-dark(oklch(99% 0.005 250), oklch(12% 0.01 250));
  --text:         light-dark(oklch(15% 0.02 250),  oklch(92% 0.02 250));
  --surface:      light-dark(oklch(97% 0.01 250),  oklch(18% 0.015 250));
  --border:       light-dark(oklch(85% 0.02 250),  oklch(30% 0.02 250));
  --text-muted:   light-dark(oklch(45% 0.02 250),  oklch(65% 0.02 250));
}

body {
  background: var(--bg);
  color: var(--text);
}

.card {
  background: var(--surface);
  border: 1px solid var(--border);
}

.btn-primary {
  background: var(--brand);
  color: white;
}

.btn-primary:hover {
  background: color-mix(in oklch, var(--brand), black 15%);
}

.btn-primary:active {
  background: color-mix(in oklch, var(--brand), black 25%);
}
```

### 3. Accessible High-Contrast Fixes

```css
/* Base styles */
.notification {
  background: var(--surface);
  box-shadow: 0 2px 8px oklch(0% 0 0 / 0.15);
  border-radius: 8px;
}

.notification.error {
  border-left: 4px solid oklch(55% 0.22 25);
}

/* Forced-colors mode fixes */
@media (forced-colors: active) {
  .notification {
    box-shadow: none;               /* forced to none anyway */
    border: 1px solid ButtonBorder; /* visible border replacement */
  }

  .notification.error {
    border-left: 4px solid LinkText; /* system color for emphasis */
  }

  .notification .icon {
    forced-color-adjust: none; /* preserve icon colors if needed */
  }
}
```

---

> **Key Takeaway:** Use OKLCH as your default color format for perceptually uniform palettes, combine it with `color-mix()` and relative color syntax for dynamic color manipulation, and use `light-dark()` with `color-scheme` to implement dark mode with minimal code duplication.
