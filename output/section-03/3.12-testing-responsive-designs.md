Every concept includes working code. Zero Angular references.

## 1. Browser DevTools Overview

Every major browser ships a responsive testing mode. The core idea is the same: resize the viewport, override the user agent, and optionally simulate touch, network conditions, or device pixel ratio. The implementations differ in meaningful ways.

| Browser | Shortcut | Mode Name | Unique Strength |
|---|---|---|---|
| Chrome | `Ctrl+Shift+M` (with DevTools open) | Device Mode | Media query bars, sensor emulation, CSS media feature overrides |
| Firefox | `Ctrl+Shift+M` | Responsive Design Mode | Overflow badge, touch simulation toggle, UA spoofing per device |
| Safari | `Ctrl+Cmd+R` (Develop menu) | Responsive Design Mode | Direct "Open with Simulator" for iOS/iPadOS, pixel ratio presets |
| Edge | `Ctrl+Shift+M` (with DevTools open) | Device Emulation | Same Chromium engine as Chrome, plus Edge-specific device presets |

### What They All Share

- Viewport resizing (pixel width and height)
- Device presets (iPhone, Pixel, iPad, etc.)
- Device pixel ratio override
- Touch event simulation
- Network throttling (Fast 3G, Slow 3G, custom profiles)
- User agent string override

### What They Do Not Share

- Rendering engines. Chrome and Edge use Blink. Firefox uses Gecko. Safari uses WebKit. A CSS property can render differently across engines even at the same viewport size.
- Touch behavior. Firefox translates mouse events into touch events when touch simulation is on. Chrome uses separate touch event dispatch.
- Container query tooling. Chrome has container badges in the Elements panel. Firefox is catching up.

## 2. Chrome DevTools Deep Dive

Chrome DevTools is the most feature-rich responsive testing tool built into a browser. Its Device Mode goes far beyond simple resizing.

### Enabling Device Mode

1. Open DevTools (`F12` or `Ctrl+Shift+I`).
2. Click the **Toggle Device Toolbar** icon (phone/tablet icon in the top-left of DevTools) or press `Ctrl+Shift+M`.

### The Device Toolbar

When Device Mode is active, the toolbar appears above the viewport with these controls:

| Control | Purpose |
|---|---|
| Dimensions dropdown | Select a specific device (iPhone 14, Pixel 7, iPad Air, etc.) or "Responsive" for free-form |
| Width x Height inputs | Type exact pixel dimensions |
| Zoom selector | Scale the preview (does not change CSS pixel calculations) |
| DPR (Device Pixel Ratio) | Override the `window.devicePixelRatio` value |
| Rotate button | Switch between portrait and landscape |
| Three-dot menu | Show media queries, show rulers, add custom device |

### Responsive Mode (Free-Form)

Select **Responsive** from the device dropdown. Drag the viewport handles or type dimensions. This is the most flexible mode for testing breakpoints.

### Media Query Bars

Click the three-dot menu in the device toolbar and select **Show media queries**. Colored bars appear above the viewport:

- **Blue bars**: `max-width` queries
- **Green bars**: queries with both `min-width` and `max-width`
- **Orange bars**: `min-width` queries

Click any bar to snap the viewport to that breakpoint width. Right-click a bar to jump to the CSS source of that query.

### Network Throttling

In the device toolbar or the Network panel:

```
No throttling  ->  Fast 3G  ->  Slow 3G  ->  Offline  ->  Custom
```

Fast 3G profile: 1.6 Mbps download, 0.75 Mbps upload, 562ms latency.
Slow 3G profile: 0.5 Mbps download, 0.75 Mbps upload, 2000ms latency.

### CPU Throttling

In the Performance panel, use the CPU dropdown:

- No throttling
- 4x slowdown
- 6x slowdown

This simulates slower processors but uses your desktop CPU architecture, which differs from mobile ARM chips.

### Touch Simulation

When Device Mode targets a mobile device type, the cursor becomes a circle simulating a fingertip. Chrome dispatches `touchstart`, `touchmove`, and `touchend` events. There are four device types available:

| Type | Mouse Events | Touch Events |
|---|---|---|
| Mobile | No | Yes |
| Mobile (no touch) | Yes | No |
| Desktop | Yes | No |
| Desktop (touch) | Yes | Yes |

**Gotcha**: Chrome touch emulation is single-finger only. Multi-touch gestures like pinch-to-zoom cannot be emulated. You need real devices for that.

### Sensor Emulation

Open the **Sensors** panel (`Ctrl+Shift+P` -> type "Sensors"):

- **Location**: Override geolocation coordinates
- **Orientation**: Set device orientation (alpha, beta, gamma)
- **Touch**: Force touch events on desktop-type viewports

### CSS Media Feature Overrides

Open the **Rendering** panel (`Ctrl+Shift+P` -> type "Rendering"):

| Override | Options | Tests |
|---|---|---|
| `prefers-color-scheme` | light, dark | Dark mode layouts |
| `prefers-reduced-motion` | no-preference, reduce | Animation removal |
| `prefers-contrast` | no-preference, more, less, custom | High contrast layouts |
| `forced-colors` | none, active | Windows High Contrast Mode |
| `prefers-reduced-data` | no-preference, reduce | Data-saving layouts |
| `color-gamut` | srgb, p3, rec2020 | Wide-gamut color support |

These overrides apply immediately without a page refresh. The Styles panel updates to show which media queries are now active.

### Custom Devices

To add a device not in Chrome's preset list:

1. Open DevTools Settings (`F1`).
2. Go to the **Devices** tab.
3. Click **Add custom device**.
4. Enter: name, width, height, device pixel ratio, user agent string, and device type (Mobile/Desktop).

## 3. Firefox Responsive Design Mode

Firefox's Responsive Design Mode has distinct features that make it worth using alongside Chrome.

### Activation

Press `Ctrl+Shift+M` or go to **Tools > Browser Tools > Responsive Design Mode**. Unlike Chrome, Firefox does not require DevTools to be open first.

### Unique Features

**Overflow Debugging Badges**: Firefox's Inspector places an `overflow` badge on elements that cause overflow and a `scroll` badge on scrollable ancestors. This is invaluable for finding the source of horizontal scrollbars on mobile viewports.

**Touch Simulation Toggle**: A dedicated touch icon in the Responsive Design Mode toolbar enables/disables touch event translation. When enabled, all mouse events become touch events. Toggle it off to test mouse behavior at the same viewport size.

**User Agent Spoofing**: When you select a device like iPhone, Firefox automatically sets `navigator.userAgent` to Safari's UA string. This triggers UA-sniffing code paths that you might otherwise miss.

**Custom Device Presets**: Click the device dropdown, then **Edit List** to add custom devices with specific width, height, DPR, user agent, and touch support.

**Screenshot Capture**: The camera icon in the toolbar saves a screenshot of the current viewport. If "Screenshot to clipboard" is enabled in settings, it copies directly to the clipboard.

**No Auto-Reload**: Firefox does not reload the page when you change viewport dimensions. This preserves application state during testing. Chrome reloads when switching between device presets.

**DPR Control**: A dropdown lets you select 1x, 2x, or 3x pixel ratios to test `@media (resolution: 2dppx)` queries and responsive image selection.

### Firefox DevTools Overflow Detection

Firefox marks overflowing elements directly in the DOM tree:

```
<div class="container">       overflow
  <div class="wide-child">
    ...
  </div>
</div>
```

Click the `overflow` badge to highlight the element. This instantly reveals which element breaks your mobile layout.

## 4. Safari Responsive Design Mode

Safari's mode is accessible from the **Develop** menu (enable it in **Safari > Settings > Advanced > Show features for web developers**).

### Activation

**Develop > Enter Responsive Design Mode** or `Ctrl+Cmd+R`.

### Key Features

- **Pixel Ratio Toggle**: Switch between 1x, 2x, and 3x to test Retina rendering.
- **Device Presets**: iPhone, iPad, and custom dimensions.
- **Open with Simulator**: The standout feature. From Responsive Design Mode, click a device and choose "Open with Simulator" to launch an Xcode iOS/iPadOS simulator running real WebKit. This is the closest you get to real iOS testing without a physical device.

### Limitations

- Safari Responsive Design Mode runs macOS WebKit, not iOS WebKit. iOS has different rendering behaviors (font rendering, scroll momentum, safe area handling).
- Available only on macOS.
- No network throttling built into Responsive Design Mode (use the Network tab instead).

### Using Web Inspector with iOS Simulator

1. Install Xcode (free from the App Store).
2. Open Responsive Design Mode in Safari.
3. Select a device and click **Open with Simulator**.
4. In the simulator, navigate to the URL.
5. In Safari on macOS, go to **Develop > Simulator** to inspect and debug the page running in the simulator.

- The **Dual Screen Emulation** feature simulates foldable and dual-screen devices. Select the Surface Duo preset to test layouts that span two screens with a hinge.
- Edge DevTools has a **3D View** that can help visualize z-index stacking across responsive layouts.

## 6. Device Emulation vs Real Device Testing

Emulation is fast and convenient. It is also incomplete. Understanding exactly what it misses prevents false confidence.

### What Emulation Covers

| Aspect | Emulated Accurately? |
|---|---|
| Viewport dimensions | Yes |
| CSS media queries based on width/height | Yes |
| Device pixel ratio for CSS | Yes |
| Touch event dispatch (single finger) | Mostly |
| User agent string | Yes |
| Network throttling | Approximate |

### What Emulation Misses

| Aspect | Why It Matters |
|---|---|
| **Rendering engine differences** | Chrome emulating an iPhone still uses Blink, not WebKit. Safari-specific bugs will not appear. |
| **Mobile GPU rendering** | Font anti-aliasing, subpixel rendering, gradient banding, and compositing differ on mobile GPUs. |
| **Mobile CPU architecture** | ARM chips handle JavaScript differently than x86/x64. A script that runs in 50ms on desktop may take 200ms on a mid-range phone. CPU throttling in DevTools is a rough approximation. |
| **Native UI elements** | `<select>`, `<input type="date">`, file pickers, and other native controls look and behave differently on mobile. Emulation shows desktop versions. |
| **Scroll physics** | Momentum scrolling, rubber-banding, scroll snapping, and overscroll behavior differ between iOS and Android. |
| **Safe areas and notches** | `env(safe-area-inset-top)` returns 0 in emulation because there is no actual notch or Dynamic Island. |
| **Multi-touch gestures** | Pinch, rotate, and multi-finger swipe cannot be tested in emulation. |
| **Keyboard behavior** | The virtual keyboard on mobile pushes the viewport, triggers `visualViewport` resize events, and may obscure fixed-position elements. Emulation does not simulate this. |
| **Real network conditions** | Throttling is synthetic. Real mobile networks have packet loss, jitter, and variable latency that throttling profiles do not replicate. |
| **Battery and thermal throttling** | Mobile devices throttle CPU/GPU when the battery is low or the device is hot. This cannot be emulated. |
| **Browser-specific quirks** | Samsung Internet, UC Browser, Opera Mini, and other mobile browsers have their own rendering quirks that no desktop emulator covers. |

### The Rule of Thumb

Use emulation for rapid iteration during development. Use real devices for sign-off before release. Never ship a responsive design tested only in emulation.

## 7. Testing with Real Devices

### USB Remote Debugging: Android

1. **Enable Developer Options** on the Android device:
   - Go to **Settings > About Phone**.
   - Tap **Build Number** seven times.
   - Go back to **Settings > Developer Options**.
   - Enable **USB Debugging**.

2. **Connect the device** to your computer via USB.

3. **Open Chrome** on your desktop and navigate to `chrome://inspect/#devices`.

4. Your device appears in the list. Click **Inspect** on the page you want to debug.

5. A full DevTools window opens, connected to the mobile browser. You can:
   - Inspect and edit the DOM
   - Debug JavaScript with breakpoints
   - Profile performance
   - View console logs
   - Modify CSS live

**Port Forwarding**: If your dev server runs on `localhost:3000`, click **Port Forwarding** in chrome://inspect and forward port 3000. The mobile device can then access `localhost:3000` directly.

### USB Remote Debugging: iOS Safari

1. On the iOS device, go to **Settings > Safari > Advanced** and enable **Web Inspector**.

2. Connect the device to a Mac via USB cable.

3. On the Mac, open Safari. Go to **Develop** menu. The connected device appears as a submenu.

4. Open a page in Safari on the iOS device. It appears under the device submenu in Mac Safari.

5. Click the page URL to open Web Inspector, connected to the mobile Safari instance.

**Limitation**: iOS remote debugging requires a Mac and Safari. There is no way to inspect iOS Safari from Windows or Linux without third-party proxies like `ios-webkit-debug-proxy`.

### Android Wireless Debugging

Starting with Android 11, you can use wireless ADB:

```bash
# On the device: Settings > Developer Options > Wireless Debugging > Pair device
# Note the IP:port and pairing code

# On desktop:
adb pair 192.168.1.100:39547
# Enter pairing code when prompted

adb connect 192.168.1.100:43721
# Now chrome://inspect shows the wireless device
```

### Testing on Budget Devices

Testing only on flagship phones gives a false sense of performance. Most global web traffic comes from mid-range and budget devices. Keep a test lab with:

- A budget Android phone (2-3GB RAM, mid-range chipset)
- An older iPhone (iPhone SE or iPhone 8 era)
- A tablet (any size)

If physical devices are impractical, use cloud device labs.

## 9. Automated Responsive Testing

Manual viewport testing does not scale. Automated tests verify responsive behavior on every commit.

### Playwright

Playwright is the strongest option for automated responsive testing. It supports Chromium, Firefox, and WebKit (Safari's engine) from a single API.

#### Setting Viewports

```js
// playwright.config.js
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  projects: [
    {
      name: 'Mobile Chrome',
      use: {
        ...devices['Pixel 7'],
        // Pixel 7: 412x915, DPR 2.625, touch, mobile UA
      },
    },
    {
      name: 'Mobile Safari',
      use: {
        ...devices['iPhone 14'],
        // iPhone 14: 390x844, DPR 3, touch, Safari UA
      },
    },
    {
      name: 'Tablet',
      use: {
        ...devices['iPad Pro 11'],
      },
    },
    {
      name: 'Desktop Chrome',
      use: {
        ...devices['Desktop Chrome'],
        viewport: { width: 1280, height: 720 },
      },
    },
  ],
});
```

#### Testing Layout Changes at Breakpoints

```js
// tests/responsive.spec.js
import { test, expect } from '@playwright/test';

test.describe('Navigation responsiveness', () => {
  test('shows hamburger menu on mobile', async ({ page }) => {
    // Set a mobile viewport
    await page.setViewportSize({ width: 375, height: 667 });
    await page.goto('/');

    // The desktop nav should be hidden
    const desktopNav = page.locator('nav.desktop-nav');
    await expect(desktopNav).toBeHidden();

    // The hamburger button should be visible
    const hamburger = page.locator('button.hamburger');
    await expect(hamburger).toBeVisible();
  });

  test('shows full nav on desktop', async ({ page }) => {
    await page.setViewportSize({ width: 1280, height: 800 });
    await page.goto('/');

    const desktopNav = page.locator('nav.desktop-nav');
    await expect(desktopNav).toBeVisible();

    const hamburger = page.locator('button.hamburger');
    await expect(hamburger).toBeHidden();
  });
});
```

#### Testing Across Multiple Viewports in One Test

```js
const viewports = [
  { name: 'mobile', width: 375, height: 667 },
  { name: 'tablet', width: 768, height: 1024 },
  { name: 'desktop', width: 1440, height: 900 },
];

for (const vp of viewports) {
  test(`card grid is correct at ${vp.name} (${vp.width}px)`, async ({ page }) => {
    await page.setViewportSize({ width: vp.width, height: vp.height });
    await page.goto('/products');

    const cards = page.locator('.product-card');
    const count = await cards.count();
    expect(count).toBeGreaterThan(0);

    // Verify no horizontal overflow
    const bodyWidth = await page.evaluate(() => document.body.scrollWidth);
    expect(bodyWidth).toBeLessThanOrEqual(vp.width);
  });
}
```

#### Playwright Screenshot Comparison (Built-in Visual Regression)

```js
test('homepage visual at mobile', async ({ page }) => {
  await page.setViewportSize({ width: 375, height: 667 });
  await page.goto('/');

  // First run: creates baseline screenshot
  // Subsequent runs: compares against baseline
  await expect(page).toHaveScreenshot('homepage-mobile.png', {
    maxDiffPixels: 100,  // Allow small rendering differences
    fullPage: true,
  });
});

test('homepage visual at desktop', async ({ page }) => {
  await page.setViewportSize({ width: 1440, height: 900 });
  await page.goto('/');

  await expect(page).toHaveScreenshot('homepage-desktop.png', {
    maxDiffPixels: 100,
    fullPage: true,
  });
});
```

**Gotcha**: Playwright screenshots differ across operating systems due to font rendering differences. Run screenshot tests in CI on a consistent OS (Linux Docker containers are the standard approach). Use `--update-snapshots` to regenerate baselines when intentional changes are made.

### Cypress

Cypress uses the `cy.viewport()` command to resize the test browser.

#### Setting Viewports

```js
// cypress.config.js
module.exports = {
  viewportWidth: 1280,  // Default viewport
  viewportHeight: 720,
};
```

```js
// cypress/e2e/responsive.cy.js
describe('Responsive navigation', () => {
  const sizes = [
    // [width, height]
    [375, 667],   // iPhone SE
    [768, 1024],  // iPad
    [1280, 720],  // Desktop
  ];

  sizes.forEach((size) => {
    it(`displays correctly at ${size[0]}x${size[1]}`, () => {
      cy.viewport(size[0], size[1]);
      cy.visit('/');

      if (size[0] < 768) {
        cy.get('nav.desktop-nav').should('not.be.visible');
        cy.get('button.hamburger').should('be.visible');
      } else {
        cy.get('nav.desktop-nav').should('be.visible');
      }
    });
  });
});
```

#### Using Device Presets

```js
cy.viewport('iphone-6');      // 375 x 667
cy.viewport('ipad-2');        // 768 x 1024
cy.viewport('macbook-15');    // 1440 x 900

// Landscape orientation
cy.viewport('iphone-6', 'landscape');  // 667 x 375
```

#### Checking for Horizontal Overflow

```js
it('has no horizontal scroll on mobile', () => {
  cy.viewport(375, 667);
  cy.visit('/');

  cy.document().then((doc) => {
    const bodyScrollWidth = doc.body.scrollWidth;
    const viewportWidth = doc.documentElement.clientWidth;
    expect(bodyScrollWidth).to.be.at.most(viewportWidth);
  });
});
```

**Limitations of Cypress for Responsive Testing**:

- Cypress runs tests only in the browser it launches (Chrome, Firefox, Edge, or Electron). It cannot run WebKit/Safari.
- `cy.viewport()` resizes the viewport but does not emulate mobile user agents, touch events, or device pixel ratios by default. You must set these separately.
- Cypress does not support native mobile testing.

### Puppeteer

Puppeteer controls headless Chrome/Chromium. It is lower-level than Playwright but effective for responsive screenshot scripts.

```js
import puppeteer from 'puppeteer';

const viewports = [
  { name: 'mobile',  width: 375,  height: 667,  deviceScaleFactor: 2, isMobile: true, hasTouch: true },
  { name: 'tablet',  width: 768,  height: 1024, deviceScaleFactor: 2, isMobile: true, hasTouch: true },
  { name: 'desktop', width: 1440, height: 900,  deviceScaleFactor: 1, isMobile: false, hasTouch: false },
];

(async () => {
  const browser = await puppeteer.launch();
  const page = await browser.newPage();

  for (const vp of viewports) {
    await page.setViewport({
      width: vp.width,
      height: vp.height,
      deviceScaleFactor: vp.deviceScaleFactor,
      isMobile: vp.isMobile,
      hasTouch: vp.hasTouch,
    });

    // Set viewport BEFORE navigating
    await page.goto('https://example.com', { waitUntil: 'networkidle2' });
    await page.screenshot({
      path: `screenshots/${vp.name}.png`,
      fullPage: true,
    });

    console.log(`Captured ${vp.name} at ${vp.width}x${vp.height}`);
  }

  await browser.close();
})();
```

**Gotcha**: Always call `setViewport()` before `page.goto()`. Setting it after navigation may not trigger media query recalculation correctly.

### Comparison: Playwright vs Cypress vs Puppeteer

| Feature | Playwright | Cypress | Puppeteer |
|---|---|---|---|
| Multi-browser | Chromium, Firefox, WebKit | Chrome, Firefox, Edge, Electron | Chromium only |
| Built-in visual regression | Yes (`toHaveScreenshot`) | No (needs plugin) | No (manual) |
| Device emulation presets | Yes (extensive library) | Yes (limited preset names) | Yes (via `setViewport` options) |
| Touch emulation | Yes | No | Yes |
| Mobile UA emulation | Yes (per device) | Manual | Manual |
| Parallel execution | Native | With Cypress Cloud | Manual |
| Test runner | Built-in | Built-in | None (library only) |

## 10. Visual Regression Testing

Functional tests verify behavior. Visual regression tests verify appearance. A button can be functionally correct (it clicks, it navigates) while being visually broken (overlapping text, wrong color, clipped content). Visual regression catches what functional tests miss.

### How It Works

1. **Baseline capture**: Take screenshots of your UI at defined viewports.
2. **Change detection**: After code changes, take new screenshots.
3. **Pixel comparison**: Diff the new screenshots against baselines.
4. **Review**: Highlight differences. Accept intentional changes as new baselines. Flag regressions.

### Percy (by BrowserStack)

Percy is a cloud-based visual testing platform. It captures DOM snapshots (not screenshots) and renders them server-side at specified widths.

#### Setup with Cypress

```bash
npm install --save-dev @percy/cli @percy/cypress
```

```js
// cypress/support/e2e.js
import '@percy/cypress';
```

```js
// cypress/e2e/visual.cy.js
describe('Visual regression', () => {
  it('captures homepage at multiple widths', () => {
    cy.visit('/');
    cy.percySnapshot('Homepage', {
      widths: [375, 768, 1280],
    });
  });

  it('captures product page', () => {
    cy.visit('/products/1');
    cy.percySnapshot('Product Detail', {
      widths: [375, 768, 1280],
    });
  });
});
```

```bash
# Run tests through Percy
npx percy exec -- cypress run
```

#### Setup with Playwright

```bash
npm install --save-dev @percy/cli @percy/playwright
```

```js
import { test } from '@playwright/test';
import percySnapshot from '@percy/playwright';

test('homepage visual', async ({ page }) => {
  await page.goto('/');
  await percySnapshot(page, 'Homepage', {
    widths: [375, 768, 1280],
  });
});
```

#### Percy Gotchas

- Each width counts as a separate screenshot toward your monthly quota.
- Percy renders DOM snapshots server-side. If your app changes the DOM based on viewport width (e.g., different components for mobile vs desktop), you must capture snapshots at each width from a browser already set to that width.
- Firefox snapshots in Percy cannot be captured below 450px width.
- Dynamic content (timestamps, ads, random data) causes false positives. Use Percy's `data-percy-css` or hide dynamic elements.

### Chromatic (by Storybook)

Chromatic is designed for component-level visual testing integrated with Storybook.

#### Setup

```bash
npm install --save-dev chromatic
npx chromatic --project-token=<your-token>
```

#### Responsive Testing in Storybook

Define viewport decorators in your stories:

```js
// Button.stories.js
export default {
  title: 'Components/Button',
  parameters: {
    chromatic: {
      viewports: [375, 768, 1280],
    },
  },
};

export const Primary = {
  args: { label: 'Click me', variant: 'primary' },
};
```

Chromatic captures each story at each specified viewport and compares against baselines. It supports Chrome, Firefox, Safari, and Edge rendering.

### BackstopJS

BackstopJS is a free, open-source tool for visual regression testing. It runs locally using headless Chromium via Puppeteer.

#### Installation and Setup

```bash
npm install -g backstopjs
backstop init
```

This creates a `backstop.json` configuration file:

```json
{
  "id": "my-project",
  "viewports": [
    { "label": "phone",   "width": 320,  "height": 480 },
    { "label": "tablet",  "width": 768,  "height": 1024 },
    { "label": "desktop", "width": 1920, "height": 1080 }
  ],
  "scenarios": [
    {
```

## 11. Testing Media Queries

### Verifying Media Queries in DevTools

#### Chrome Media Query Bars

1. Open Device Mode.
2. Click the three-dot menu > **Show media queries**.
3. Colored bars appear above the viewport.
4. Drag the viewport width. When you cross a breakpoint, the corresponding bar highlights.
5. Right-click a bar to reveal the source CSS file and line number.

#### DevTools "Computed" Styles

Select an element in the Elements panel. In the **Computed** tab, scroll down to see which CSS rules apply. As you resize the viewport, rules inside media queries appear and disappear.

### Testing Media Queries in JavaScript

The `window.matchMedia()` API lets you programmatically check and react to media query changes.

#### Checking a Query

```js
const isMobile = window.matchMedia('(max-width: 767px)');

console.log(isMobile.matches);
// true if viewport <= 767px, false otherwise
```

#### Listening for Changes

```js
const query = window.matchMedia('(min-width: 768px)');

function handleBreakpointChange(event) {
  if (event.matches) {
    console.log('Entered tablet/desktop layout');
    // Show desktop navigation, etc.
  } else {
    console.log('Entered mobile layout');
    // Show hamburger menu, etc.
  }
}

// Modern API: addEventListener
query.addEventListener('change', handleBreakpointChange);

// Call once on load
handleBreakpointChange(query);
```

#### Automated Testing of matchMedia Behavior

```js
// Playwright test verifying JavaScript responds to viewport changes
test('matchMedia responds to viewport resize', async ({ page }) => {
  await page.setViewportSize({ width: 375, height: 667 });
  await page.goto('/');

  // Check that the page considers itself "mobile"
  const isMobile = await page.evaluate(
    () => window.matchMedia('(max-width: 767px)').matches
  );
  expect(isMobile).toBe(true);

  // Resize to desktop
  await page.setViewportSize({ width: 1280, height: 800 });

  const isDesktop = await page.evaluate(
    () => window.matchMedia('(min-width: 768px)').matches
  );
  expect(isDesktop).toBe(true);
});
```

### Testing Media Feature Queries

Beyond width, test feature queries that affect layout:

```css
/* Hover capability */
@media (hover: hover) {
  .card:hover { transform: scale(1.02); }
}

/* Pointer precision */
@media (pointer: coarse) {
  .button { min-height: 48px; padding: 12px 24px; }
}

/* Reduced motion */
@media (prefers-reduced-motion: reduce) {
  * { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; }
}

/* Dark mode */
@media (prefers-color-scheme: dark) {
  :root { --bg: #1a1a1a; --text: #e0e0e0; }
}
```

Test these in Chrome by opening the Rendering panel and toggling each override. In automated tests:

```js
// Playwright: emulate reduced motion
test('animations are disabled with reduced motion', async ({ browser }) => {
  const context = await browser.newContext({
    reducedMotion: 'reduce',
  });
  const page = await context.newPage();
  await page.goto('/');

  // Verify that animation-duration is effectively 0
  const duration = await page.evaluate(() => {
    const el = document.querySelector('.animated-element');
    return getComputedStyle(el).animationDuration;
  });
  expect(duration).toBe('0.01ms');
});

// Playwright: emulate dark mode
test('dark mode styles apply', async ({ browser }) => {
  const context = await browser.newContext({
    colorScheme: 'dark',
  });
  const page = await context.newPage();
  await page.goto('/');

  const bgColor = await page.evaluate(() => {
    return getComputedStyle(document.documentElement).getPropertyValue('--bg').trim();
  });
  expect(bgColor).toBe('#1a1a1a');
});
```

## 12. Testing Container Queries

Container queries respond to the size of a parent container, not the viewport. This makes them harder to test with viewport-based tools.

### Chrome DevTools Container Query Support

1. Open the **Elements** panel.
2. Elements with `container-type` set show a **container** badge next to the tag name.
3. Click the badge to toggle a dotted overlay showing the container boundaries.
4. In the **Styles** pane, `@container` rules appear and highlight when active.

### Visual Debugging

Add a temporary outline to all containers during development:

```css
/* Debug-only: visualize all container query contexts */
[style*="container-type"],
[class] {
  /* Target elements with container-type set */
}

/* More reliable: target the property directly */
*:where([style*="container-type"]) {
  outline: 2px dashed rebeccapurple;
}
```

Or use this catch-all in your dev stylesheet:

```css
/* Add to a debug stylesheet, never ship this */
* {
  container-type: unset; /* Don't use this -- just for illustration */
}

/* Instead, highlight known containers */
.card-container,
.sidebar,
.main-content {
  outline: 2px dashed oklch(0.5 0.2 300);
}
```

### Testing Container Queries Programmatically

Since container queries depend on element width, not viewport width, you cannot just resize the viewport. You need to control the container's width:

```js
// Playwright: test a container query by controlling parent width
test('card layout changes at narrow container', async ({ page }) => {
  await page.goto('/components');

  // Inject a style that forces the container to a narrow width
  await page.addStyleTag({
    content: `.card-container { width: 300px !important; }`,
  });

  // Wait for layout recalculation
  await page.waitForTimeout(100);

  const cardLayout = await page.evaluate(() => {
    const card = document.querySelector('.card');
    return getComputedStyle(card).flexDirection;
  });

  // At 300px container width, the card should stack vertically
  expect(cardLayout).toBe('column');
});

test('card layout is horizontal at wide container', async ({ page }) => {
  await page.goto('/components');

  await page.addStyleTag({
    content: `.card-container { width: 800px !important; }`,
  });

  await page.waitForTimeout(100);

  const cardLayout = await page.evaluate(() => {
    const card = document.querySelector('.card');
    return getComputedStyle(card).flexDirection;
  });

  expect(cardLayout).toBe('row');
});
```

## 13. Testing Responsive Images

Responsive images using `srcset` and `sizes` attributes load different files based on viewport width and device pixel ratio. Verifying which image actually loaded requires specific techniques.

### Checking the Loaded Source in DevTools

1. Open the **Network** panel.
2. Filter by **Img**.
3. **Disable cache** (checkbox at top of Network panel). Browser caching prevents `srcset` re-evaluation on resize.
4. Reload the page at each viewport width.
5. Observe which image URL was requested.

Alternatively, in the **Elements** panel:

1. Select the `<img>` element.
2. Look at the `currentSrc` property in the **Properties** pane (not the HTML `src` attribute, which is the fallback).
3. Or hover over the `srcset` values in the HTML; Chrome shows the resolved URL.

### JavaScript Verification

```js
// Check which image source was selected by the browser
const img = document.querySelector('.hero-image');
console.log('currentSrc:', img.currentSrc);
console.log('naturalWidth:', img.naturalWidth);
console.log('naturalHeight:', img.naturalHeight);
```

### Automated Test

```js
// Playwright: verify correct responsive image is loaded
test('loads small image on mobile', async ({ page }) => {
  await page.setViewportSize({ width: 375, height: 667 });

  // Disable cache to force fresh image selection
  const context = page.context();
  await page.route('**/*', (route) => {
    route.continue({
      headers: { ...route.request().headers(), 'Cache-Control': 'no-cache' },
    });
  });

  await page.goto('/');

  const currentSrc = await page.evaluate(() => {
    const img = document.querySelector('.hero-image');
    return img.currentSrc;
  });

  // Mobile should load the small variant
  expect(currentSrc).toContain('hero-small');
});

test('loads large image on desktop', async ({ page }) => {
  await page.setViewportSize({ width: 1440, height: 900 });
  await page.goto('/');

  const currentSrc = await page.evaluate(() => {
    const img = document.querySelector('.hero-image');
    return img.currentSrc;
  });

  expect(currentSrc).toContain('hero-large');
});
```

### Common `srcset` Testing Gotchas

| Gotcha | Explanation |
|---|---|
| Browser caching | Once the browser loads a larger image, it may reuse it even at smaller viewports. Always disable cache when testing. |
| `sizes` attribute required | When using `w` descriptors in `srcset`, you must include `sizes`. Without it, the browser defaults to `sizes="100vw"` which may select the wrong source. |
| DPR matters | A 375px-wide phone with 3x DPR needs a 1125px-wide image for sharp rendering. Check with different DPR settings in DevTools. |
| `<picture>` element | `<picture>` with `<source>` tags uses `media` attributes, not `sizes`. Test each `<source>` media condition independently. |
| Art direction | `<picture>` art direction means different images at different breakpoints (not just different sizes). Verify composition and cropping, not just resolution. |

## 14. Network Throttling and Mobile Performance

Responsive design includes responsive performance. A layout that works on mobile but takes 12 seconds to load on a 3G connection has failed.

### Chrome DevTools Throttling

**Network Throttling** (Network panel or Device toolbar):

| Profile | Download | Upload | Latency |
|---|---|---|---|
| Fast 3G | 1.6 Mbps | 0.75 Mbps | 562ms |
| Slow 3G | 0.5 Mbps | 0.75 Mbps | 2000ms |
| Offline | 0 | 0 | N/A |

**Custom profiles**: Click "Add..." to create profiles matching real network conditions in your target markets.

**CPU Throttling** (Performance panel):

| Setting | Effect |
|---|---|
| No throttling | Full desktop CPU speed |
| 4x slowdown | Simulates a mid-range phone |
| 6x slowdown | Simulates a budget phone |

### Lighthouse Mobile Audit

Run Lighthouse in Chrome DevTools with the **Mobile** preset:

1. Open the **Lighthouse** panel.
2. Select **Mobile** as the device.
3. Check **Performance**.
4. Click **Analyze page load**.

Lighthouse throttles the CPU by 4x and the network to a simulated "typical mobile connection" (1.6 Mbps, 150ms RTT). It reports:

- **First Contentful Paint (FCP)**: When the first content pixel renders.
- **Largest Contentful Paint (LCP)**: When the largest content element renders.
- **Cumulative Layout Shift (CLS)**: Visual stability during load.
- **Total Blocking Time (TBT)**: How long the main thread was blocked.

### Testing Performance Programmatically

```js
// Playwright: measure LCP on mobile
test('LCP is under 2.5s on mobile', async ({ page }) => {
  await page.setViewportSize({ width: 375, height: 667 });
  await page.goto('/', { waitUntil: 'networkidle' });

  const lcp = await page.evaluate(() => {
    return new Promise((resolve) => {
      new PerformanceObserver((list) => {
        const entries = list.getEntries();
        const last = entries[entries.length - 1];
        resolve(last.startTime);
      }).observe({ type: 'largest-contentful-paint', buffered: true });
    });
  });

  console.log(`LCP: ${lcp}ms`);
  expect(lcp).toBeLessThan(2500);
});
```

### What Synthetic Throttling Misses

- **Packet loss**: Real mobile networks drop packets. Synthetic throttling just adds latency.
- **Variable latency**: Real connections fluctuate. Throttling applies a constant delay.
- **DNS resolution**: Mobile DNS can be slower. Throttling does not affect DNS.
- **Radio state transitions**: Mobile radios transition between idle, active, and high-power states, adding variable delays.

For real-world performance data, combine synthetic testing with field data from the Chrome User Experience Report (CrUX) or Real User Monitoring (RUM) tools.

## 15. Accessibility Testing in Responsive Context

Responsive design and accessibility intersect at several points. A layout that reflows at small viewports must remain usable for people who rely on zoom, screen readers, or keyboard navigation.

### WCAG 1.4.10 Reflow (Level AA)

This criterion requires that content reflow without horizontal scrolling at 320 CSS pixels wide (for vertical scrolling content) and 256 CSS pixels tall (for horizontal scrolling content).

**Why 320px?** It corresponds to a 1280px desktop viewport zoomed to 400%. Users with low vision frequently zoom to 400% or more.

#### How to Test Reflow

**Manual method:**

1. Open Chrome.
2. Set the browser window to 1280px wide.
3. Zoom to 400% (`Ctrl` + `+` four times from 100%, or type 400 in the DevTools device toolbar width as 320px).
4. Navigate every page.
5. Verify:
   - No horizontal scrollbar appears (except for data tables, maps, diagrams, toolbars).
   - All text is visible without horizontal scrolling.
   - All interactive elements are reachable.
   - Content order remains logical.

**DevTools method:**

1. Open Device Mode.
2. Set width to 320px.
3. This simulates the same condition as 400% zoom at 1280px.

#### Automated Test

```js
// Playwright: verify no horizontal overflow at 320px (WCAG 1.4.10)
test('WCAG 1.4.10: no horizontal scroll at 320px width', async ({ page }) => {
  await page.setViewportSize({ width: 320, height: 568 });
  await page.goto('/');

  const hasHorizontalScroll = await page.evaluate(() => {
    return document.documentElement.scrollWidth > document.documentElement.clientWidth;
  });

  expect(hasHorizontalScroll).toBe(false);
});
```

### Exceptions to Reflow

Content that inherently requires two-dimensional layout is exempt:

- Data tables
- Images, maps, and diagrams
- Video players
- Games
- Presentations
- Interfaces where toolbars must remain visible while editing (e.g., code editors, drawing tools)

### Testing Zoom Behavior

Browsers handle zoom differently:

| Browser | Zoom Behavior |
|---|---|
| Chrome | Increases CSS pixel size. `window.innerWidth` decreases. Media queries respond. |
| Firefox | Same as Chrome. |
| Safari | Same, but smooth-zooms on iOS with pinch. |
| iOS Safari | Pinch zoom does not trigger media queries. Only viewport meta tag `initial-scale` affects layout. |

**Gotcha**: On iOS, pinch zoom does not change the CSS viewport. It magnifies the rendered page. This means media queries do not fire during pinch zoom on iOS. Users who zoom on iOS rely on the layout already being correct at the device's CSS viewport width.

### Testing Touch Target Sizes After Reflow

When content reflows to a narrower layout, touch targets may shrink or overlap. Verify:

```js
// Check that all buttons meet 44x44px minimum after reflow
test('touch targets meet minimum size at 320px', async ({ page }) => {
  await page.setViewportSize({ width: 320, height: 568 });
  await page.goto('/');

  const smallTargets = await page.evaluate(() => {
    const interactives = document.querySelectorAll(
      'a, button, input, select, textarea, [role="button"], [tabindex]'
    );
    const violations = [];
    for (const el of interactives) {
      const rect = el.getBoundingClientRect();
      if (rect.width > 0 && rect.height > 0) {
        if (rect.width < 44 || rect.height < 44) {
          violations.push({
            tag: el.tagName,
            text: el.textContent.trim().slice(0, 30),
            width: Math.round(rect.width),
            height: Math.round(rect.height),
          });
        }
      }
    }
    return violations;
  });

  if (smallTargets.length > 0) {
    console.warn('Small touch targets:', smallTargets);
  }
  // Warning, not hard failure: inline links are exempt
});
```

### Focus Visibility on Mobile

Ensure `:focus-visible` styles remain visible at all viewports. A focus ring that is 2px on desktop but gets clipped by `overflow: hidden` on a mobile container is an accessibility failure.

## 16. Cross-Browser Testing Strategy

You cannot test every browser-device-viewport combination. The strategy is to cover the most impactful combinations efficiently.

### Step 1: Know Your Audience

Check your analytics for:

- Browser distribution (Chrome, Safari, Firefox, Edge, Samsung Internet, etc.)
- Device distribution (iPhone models, Android models, iPad, desktop)
- Viewport distribution (most common widths)
- Operating system distribution

### Step 2: Define Tier Coverage

| Tier | Criteria | Testing Level |
|---|---|---|
| Tier 1 | Top 3 browsers, top 5 devices by traffic | Full manual + automated testing every release |
| Tier 2 | Next 5 browser/device combos | Automated testing + spot-check manual |
| Tier 3 | Long-tail browsers (UC Browser, Opera Mini, etc.) | Periodic manual testing, address blocking bugs only |

### Step 3: Select Viewport Breakpoints

Do not test 50 viewports. Test at your actual breakpoints plus one pixel above and below.

```
Common test widths:
320px   - Smallest supported (iPhone SE, WCAG 1.4.10 reflow)
375px   - iPhone 12/13/14/15 standard
390px   - iPhone 14/15 Pro
412px   - Pixel 7/8, Samsung Galaxy S-series
768px   - iPad portrait
1024px  - iPad landscape, small laptops
1280px  - Standard desktop
1440px  - Large desktop
1920px  - Full HD
```

Test at your breakpoints:

```css
/* If your breakpoints are: */

/* Test at: 767px, 768px, 1023px, 1024px */
```

## 17. Common Debugging Techniques

### Finding Horizontal Overflow

The most common responsive bug is an element that extends beyond the viewport width, causing a horizontal scrollbar.

#### JavaScript Console Snippet

Paste this in the browser console to find every element wider than the viewport:

```js
// Find all elements causing horizontal overflow
const docWidth = document.documentElement.offsetWidth;

document.querySelectorAll('*').forEach((el) => {
  const rect = el.getBoundingClientRect();
  if (rect.right > docWidth || rect.left < 0) {
    console.log(
      '%c OVERFLOW ',
      'background: red; color: white;',
      el.tagName,
      el.className || el.id || '',
      `left: ${Math.round(rect.left)}, right: ${Math.round(rect.right)}, viewport: ${docWidth}`
    );
    el.style.outline = '3px solid red';
  }
});
```

#### CSS Outline Debugging

```js
// Add outlines to every element to visualize boxes
// Uses outline (not border) to avoid affecting layout
document.querySelectorAll('*').forEach((el) => {
  el.style.outline = '1px solid rgba(255, 0, 0, 0.3)';
});
```

#### Common Overflow Causes

| Cause | Fix |
|---|---|
| Fixed-width element (`width: 500px`) | Use `max-width: 100%` or `width: min(500px, 100%)` |
| `100vw` includes scrollbar width | Use `100%` instead of `100vw` for full-width elements |
| Missing `flex-wrap: wrap` | Add `flex-wrap: wrap` to flex containers that should wrap on narrow screens |
| Images without `max-width` | Add `img { max-width: 100%; height: auto; }` |
| Pre-formatted text or code blocks | Add `overflow-x: auto` to `<pre>` and `<code>` containers |
| Long unbreakable strings (URLs, email addresses) | Add `overflow-wrap: break-word` or `word-break: break-all` |
| Negative margins pulling content outside container | Ensure negative margins are contained by parent overflow |
| Absolute/fixed positioned elements | Ensure they respect viewport bounds |
| Tables wider than viewport | Wrap in a container with `overflow-x: auto` |

### Firefox Overflow Badges

Firefox DevTools automatically marks overflowing elements:

1. Open Inspector.
2. Look for `overflow` badges next to element tags in the DOM tree.
3. Click the badge to highlight the overflowing element.
4. The parent scrollable container gets a `scroll` badge.

### Debugging Layout Shifts

Layout shifts during responsive transitions (e.g., images loading without explicit dimensions) cause poor CLS scores.

```js
// Monitor layout shifts in the console
new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.hadRecentInput) continue; // Ignore user-initiated shifts
    console.log('Layout shift:', entry.value, entry.sources);
    entry.sources?.forEach((source) => {
      console.log('  Shifted element:', source.node);
      source.node?.style && (source.node.style.outline = '3px solid orange');
    });
  }
}).observe({ type: 'layout-shift', buffered: true });
```

### Debugging `z-index` Stacking at Different Viewports

Elements that stack correctly on desktop may overlap incorrectly on mobile when `position: relative` or `position: absolute` contexts change.

```js
// Log all positioned elements and their z-index
document.querySelectorAll('*').forEach((el) => {
  const style = getComputedStyle(el);
  if (style.position !== 'static' && style.zIndex !== 'auto') {
    console.log(
      el.tagName,
      el.className,
      `position: ${style.position}`,
      `z-index: ${style.zIndex}`
    );
  }
});
```

### Debugging Media Query Conflicts

When multiple media queries overlap or contradict each other:

1. Open DevTools Elements panel.
2. Select the problematic element.
3. In the **Styles** pane, look for struck-through rules. These are overridden.
4. Check the **Computed** tab to see the final applied values.
5. Use the **Media Query Bars** (Chrome) to identify which queries are active at the current width.

### Debugging Touch Target Issues

```js
// Highlight all interactive elements smaller than 44x44px
document.querySelectorAll('a, button, [role="button"], input, select, textarea').forEach((el) => {
  const rect = el.getBoundingClientRect();

    el.title = `${Math.round(rect.width)}x${Math.round(rect.height)}`;

});
```

## 18. Responsive Testing Checklist

Use this checklist for every responsive page before release. It is organized by testing phase.

### Phase 1: Development (Every Save)

```
[ ] No horizontal scrollbar at any width from 320px to 1920px
[ ] Content is readable without zooming on mobile
[ ] Images are not stretched or pixelated
[ ] Text does not overflow its container
[ ] No overlapping elements
[ ] Navigation is accessible at all widths
```

### Phase 2: Breakpoint Verification

```
[ ] Layout changes correctly at each defined breakpoint
[ ] Test at breakpoint-1px and breakpoint+1px (e.g., 767px and 768px)
[ ] No content disappears between breakpoints
[ ] Transitions between breakpoints are smooth (if animated)
[ ] No "dead zones" where layout is broken between breakpoints
```

### Phase 3: Interactive Elements

```
[ ] Touch targets are at least 44x44px on mobile
[ ] Touch targets have at least 8px spacing between them
[ ] Hover states degrade gracefully on touch devices
[ ] Dropdown menus and modals are usable on mobile
[ ] Form inputs are not obscured by virtual keyboard
[ ] Autocomplete and datepickers work on mobile browsers
```

### Phase 4: Media and Images

```
[ ] Responsive images load correct source (verify currentSrc in DevTools)
[ ] Images have explicit width and height to prevent layout shift
[ ] <picture> art direction serves correct crops at each breakpoint
[ ] Video players resize without breaking aspect ratio
[ ] SVGs scale correctly
[ ] Lazy-loaded images load when scrolled into view on all viewports
```

### Phase 5: Typography and Content

```
[ ] Font sizes are readable on small screens (minimum 16px body text)
[ ] Line lengths are comfortable (45-75 characters per line)
[ ] Long words/URLs break correctly (overflow-wrap: break-word)
[ ] Tables are either responsive (stacked) or horizontally scrollable
[ ] Code blocks do not overflow their containers
```

### Phase 6: Performance

```
[ ] Page loads in under 3s on Fast 3G throttling
[ ] LCP is under 2.5s on mobile
[ ] CLS is under 0.1 on mobile
[ ] No render-blocking resources for above-the-fold content on mobile
[ ] Images are appropriately compressed for mobile bandwidth
[ ] Fonts are optimized (font-display: swap, subset, preload critical)
```

### Phase 7: Accessibility

```
[ ] WCAG 1.4.10 Reflow: no horizontal scroll at 320px width
[ ] Zoom to 200%: content reflows correctly
[ ] Zoom to 400%: content remains usable
[ ] Focus order is logical at all viewport sizes
[ ] Focus indicators are visible at all viewport sizes
[ ] Screen reader announces content in logical order at mobile layout
[ ] Skip links work at mobile viewport
[ ] Color contrast passes at all viewport sizes (check dark mode too)
```

### Phase 8: Cross-Browser

```
[ ] Chrome on Android: tested on real device or BrowserStack
[ ] Safari on iOS: tested on real device or BrowserStack
[ ] Firefox: tested at mobile and desktop widths
[ ] Edge: tested at desktop width
[ ] Samsung Internet: tested if audience warrants (check analytics)
```

### Phase 9: Orientation and Special Cases

```
[ ] Portrait and landscape orientations on mobile
[ ] Portrait and landscape orientations on tablet
[ ] Foldable device modes (if targeting, test with Edge DevTools Surface Duo emulation)
[ ] Print styles do not break responsive layout
[ ] Right-to-left (RTL) layout (if supporting)
```

### Phase 10: Automated Regression

```
[ ] Visual regression tests capture all key pages at 3+ viewports
[ ] Viewport automated tests check for horizontal overflow
[ ] CI pipeline runs responsive tests on every PR
[ ] Screenshot baselines are up to date
```

---

## Summary

Testing responsive designs requires a layered approach. No single tool or technique covers everything.

| Layer | Tools | What It Catches |
|---|---|---|
| **DevTools** | Chrome, Firefox, Safari, Edge | Rapid layout verification, media query debugging, overflow detection |
| **Emulation** | DevTools device mode, Responsively App, Polypane | Viewport-based layout issues, breakpoint behavior |
| **Real Devices** | USB debugging, wireless ADB, iOS Web Inspector | Rendering engine differences, scroll physics, native UI, touch behavior, performance |
| **Cloud Devices** | BrowserStack, Sauce Labs, LambdaTest | Wide device/browser coverage without a physical lab |
| **Automated Tests** | Playwright, Cypress, Puppeteer | Regression prevention, CI integration, viewport assertions |
| **Visual Regression** | Percy, Chromatic, BackstopJS, Playwright screenshots | Pixel-level layout regressions across viewports |
| **Performance** | Lighthouse, DevTools throttling, WebPageTest | Load times and interaction responsiveness on mobile connections |
| **Accessibility** | WCAG 1.4.10 reflow testing, axe-core, manual zoom testing | Zoom usability, reflow, focus management, touch target sizes |

Start every session in DevTools. Automate the checks you repeat. Test on real devices before you ship. Never trust emulation alone.
