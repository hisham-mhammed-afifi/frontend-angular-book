> Mobile devices account for over 63% of global web traffic, yet most sites are built and tested on powerful desktops connected to fast Wi-Fi. Every concept includes working code. Zero Angular references.

## 1. The Mobile Performance Landscape

Mobile is not a smaller desktop. It is a fundamentally different execution environment with four hard constraints that compound each other.

### 1.1 CPU Constraints

A typical mid-range phone (the global median device) has a CPU that is 3-5x slower than a current-generation laptop. JavaScript that takes 200ms on your MacBook takes 600-1000ms on a Samsung Galaxy A14. The gap is not closing: flagship phones improve each year, but the median device that real users carry stays 3-4 years behind.

| Device Tier | Example | CPU Relative Speed | Typical RAM |
|---|---|---|---|
| Flagship 2025 | iPhone 16 Pro, Pixel 9 Pro | 1x (baseline) | 8-12 GB |
| Mid-range 2025 | Samsung A55, Pixel 8a | 2-3x slower | 4-6 GB |
| Budget / Older | Samsung A14, Redmi 12 | 4-6x slower | 3-4 GB |
| Global median | Mixed, 3-4 years old | ~4x slower | 3-4 GB |

### 1.2 Memory Constraints

Mobile browsers get a fraction of available RAM. Chrome on Android typically gets 150-300 MB for a tab on a 4 GB device. Exceed that, and the tab gets killed and has to reload from scratch. Desktop Chrome can comfortably use 1-2 GB per tab.

```
Memory pressure chain reaction:
1. Page allocates too many DOM nodes, images, or JS objects
2. OS memory pressure triggers garbage collection pauses (jank)
3. If pressure continues, the browser kills background tabs
4. If the foreground tab exceeds limits, the entire tab crashes
```

### 1.3 Network Constraints

The P75 (75th percentile) mobile connection provides roughly 9 Mbps download, 3 Mbps upload, and 100ms round-trip latency. That is the reality, not the theoretical peak of 4G or 5G.

| Connection | Download | Latency (RTT) | Real-World Notes |
|---|---|---|---|
| 3G | 1.5 Mbps | 300-400ms | Still common in developing regions |
| 4G LTE | 10-50 Mbps | 30-70ms | Typical in urban areas |
| 4G congested | 2-5 Mbps | 100-200ms | Stadiums, trains, rush hour |
| 5G (Sub-6) | 50-200 Mbps | 10-30ms | Coverage gaps cause fallback to 4G |
| 5G (mmWave) | 500+ Mbps | 1-10ms | Requires line-of-sight, very limited coverage |
| Global P75 | ~9 Mbps | ~100ms | What you should design for |

Latency matters more than bandwidth for web performance. A page with 20 small requests at 100ms RTT per request pays 2 full seconds just in network round-trips, regardless of bandwidth.

### 1.4 Battery Constraints

Every CPU cycle, network request, and screen wake-up drains battery. Users notice. Apps and sites that drain battery get uninstalled or avoided. The main battery drains from web pages are:

- Continuous JavaScript execution (timers, animations, polling)
- Frequent network requests (keep-alive, analytics pings)
- GPS/sensor access
- High-frequency DOM updates forcing repaints

### 1.5 The Compound Effect

These constraints multiply. On a desktop with a fast CPU and Wi-Fi, a 2 MB JavaScript bundle downloads in 200ms and parses in 100ms. On a mid-range phone over 4G:

```
Download:   2 MB / 9 Mbps = ~1.8 seconds
Parse:      4x slower CPU  = ~400ms
Execute:    4x slower CPU  = ~800ms
Total:      ~3.0 seconds (just for JS, before any rendering)
```

This is why mobile performance optimization is not optional. It is the difference between a usable site and a site that loses 40-70% of its potential conversions.

## 2. Core Web Vitals

Core Web Vitals are Google's standardized performance metrics that directly affect search ranking. They measure what users actually experience: loading speed, interactivity, and visual stability.

### 2.1 The Three Metrics

| Metric | Measures | Good | Needs Improvement | Poor |
|---|---|---|---|---|
| **LCP** (Largest Contentful Paint) | Perceived load speed | <= 2.5s | 2.5s - 4.0s | > 4.0s |
| **INP** (Interaction to Next Paint) | Responsiveness | <= 200ms | 200ms - 500ms | > 500ms |
| **CLS** (Cumulative Layout Shift) | Visual stability | <= 0.1 | 0.1 - 0.25 | > 0.25 |

To pass Core Web Vitals assessment, at least 75% of page visits must meet the "Good" threshold for each metric.

### 2.2 LCP (Largest Contentful Paint)

LCP measures when the largest visible element in the viewport finishes rendering. This is usually a hero image, a large heading, or a video poster. It represents the moment the user perceives the page as "loaded."

**What counts as LCP:**
- `<img>` elements
- `<image>` inside SVG
- `<video>` poster images
- Elements with `background-image` loaded via CSS
- Block-level text elements (`<h1>`, `<p>`, etc.)

**Common LCP killers on mobile:**
1. Unoptimized hero images (large files, wrong format)
2. Render-blocking CSS or JS delaying first paint
3. Web fonts blocking text rendering (FOIT)
4. Server response time (TTFB > 600ms)
5. Client-side rendering with no server content

**Optimization strategies:**

```html
<!-- Preload the LCP image so the browser fetches it immediately -->
<link rel="preload" as="image" href="/hero.webp"
      fetchpriority="high">

<!-- Use fetchpriority to signal importance -->
<img src="/hero.webp"
     alt="Hero banner"
     width="1200" height="600"
     fetchpriority="high">

<!-- DO NOT lazy-load the LCP element -->
<!-- BAD: This delays the most important image -->
<img src="/hero.webp" loading="lazy" alt="Hero">
```

### 2.3 INP (Interaction to Next Paint)

INP replaced First Input Delay (FID) in March 2024. While FID only measured the delay of the first interaction, INP measures the latency of all interactions throughout the page's lifecycle and reports the worst one (at the 98th percentile).

**What INP measures:**
1. **Input delay** - time from user action to event handler start (main thread busy)
2. **Processing time** - time the event handler takes to run
3. **Presentation delay** - time from handler completion to next paint

```
INP = Input Delay + Processing Time + Presentation Delay
```

**Why 43% of sites fail INP:**
- Long JavaScript tasks block the main thread
- Heavy event handlers (sorting, filtering, DOM manipulation)
- Layout thrashing inside event handlers
- Third-party scripts competing for main thread time

**Optimization with `scheduler.yield()`:**

```javascript
// BAD: One long task that blocks the main thread for 300ms+
function processLargeDataset(items) {
  items.forEach(item => {
    // Heavy computation per item
    computeExpensiveResult(item);
    updateDOM(item);
  });
}

// GOOD: Break into chunks, yielding to the main thread between each
async function processLargeDataset(items) {
  for (const item of items) {
    computeExpensiveResult(item);
    updateDOM(item);

    // Yield to the main thread so user interactions can be processed
    // scheduler.yield() puts remaining work at the FRONT of the queue
    // (unlike setTimeout which puts it at the back)
    if (scheduler.yield) {
      await scheduler.yield();
    }
  }
}
```

**Fallback for browsers without `scheduler.yield()`:**

```javascript
function yieldToMain() {
  // scheduler.yield() is preferred: it resumes work at front of queue
  if (globalThis.scheduler?.yield) {
    return scheduler.yield();
  }
  // Fallback: setTimeout puts work at back of queue (less ideal)
  return new Promise(resolve => setTimeout(resolve, 0));
}

async function handleClick(event) {
  // Phase 1: Visual feedback (fast)
  button.classList.add('processing');
  await yieldToMain();

  // Phase 2: Data processing (slower)
  const result = computeResult(data);
  await yieldToMain();

  // Phase 3: DOM update
  renderResult(result);
}
```

### 2.4 CLS (Cumulative Layout Shift)

CLS measures how much visible content shifts unexpectedly during the page lifecycle. A CLS of 0.1 means 10% of the viewport shifted. Layout shifts triggered by user interaction (clicking a button that reveals content) do not count.

**Common CLS causes on mobile:**
1. Images without `width` and `height` attributes
2. Ads, embeds, or iframes without reserved space
3. Dynamically injected content above the fold
4. Web fonts causing text reflow (different metrics than fallback)
5. Late-loading CSS that changes layout

**Fixing CLS:**

```html
<!-- ALWAYS set width and height on images -->
<!-- The browser calculates aspect ratio before download -->
<img src="/photo.webp"
     width="800" height="600"
     alt="Product photo">

<!-- For responsive images, width/height still prevent CLS -->
<!-- CSS can override the dimensions, but the ratio is locked in -->
<style>
  img {
    max-width: 100%;
    height: auto; /* Maintains aspect ratio from width/height attrs */
  }
</style>

<!-- Reserve space for ads and embeds -->
<div style="min-height: 250px;">
  <!-- Ad loads here asynchronously -->
  <div id="ad-slot"></div>
</div>

<!-- Use aspect-ratio for containers that load dynamic content -->
<div class="video-embed" style="aspect-ratio: 16 / 9;">
  <!-- iframe loads here -->
</div>
```

**Font-related CLS mitigation:**

```css
/* Match fallback font metrics to custom font to minimize shift */
@font-face {
  font-family: 'CustomFont';
  src: url('/fonts/custom.woff2') format('woff2');
  font-display: swap;
  /* size-adjust and ascent/descent-override reduce layout shift */
  size-adjust: 105%;
  ascent-override: 90%;
  descent-override: 20%;
  line-gap-override: 0%;
}

body {
  /* System font stack as fallback, tuned to match custom font */
  font-family: 'CustomFont', -apple-system, BlinkMacSystemFont, sans-serif;
}
```

## 3. The Critical Rendering Path

The Critical Rendering Path (CRP) is the sequence of steps the browser takes to convert HTML, CSS, and JavaScript into pixels on screen. Understanding this pipeline is essential because every optimization technique targets one or more of these steps.

### 3.1 The Pipeline

```
HTML bytes
  |
  v
[Parse HTML] --> DOM tree
  |
  |  (CSS discovered)
  v
[Parse CSS] --> CSSOM tree
  |
  |  (JS discovered - may block both)
  v
[Execute JS] --> May modify DOM and CSSOM
  |
  v
[Combine DOM + CSSOM] --> Render Tree
  |
  v
[Layout] --> Compute geometry (position, size) of every element
  |
  v
[Paint] --> Fill in pixels (colors, text, images, borders)
  |
  v
[Composite] --> Combine layers and send to GPU for display
```

### 3.2 Key Concepts

**DOM Construction** is incremental. The browser starts building the DOM as soon as HTML bytes arrive. This is why content at the top of the document renders first.

**CSSOM Construction** is render-blocking. The browser will not render anything until it has built the CSSOM, because rendering without styles would cause a flash of unstyled content. This means CSS in the `<head>` blocks first paint.

**JavaScript Execution** blocks HTML parsing by default. When the parser encounters a `<script>` tag without `async` or `defer`, it must:
1. Pause HTML parsing
2. Wait for the script to download
3. Execute the script
4. Resume HTML parsing

**Layout (Reflow)** is expensive on mobile. The browser must calculate the position and size of every visible element. Triggering layout multiple times per frame (layout thrashing) causes visible jank.

**Paint** converts layout information into actual pixels. Complex CSS (shadows, gradients, filters) increases paint time.

**Composite** is the cheapest step. It combines pre-painted layers. Animations that only trigger compositing (`transform`, `opacity`) are smooth even on slow devices.

### 3.3 What Triggers Each Step

| CSS Property Changed | Triggers |
|---|---|
| `width`, `height`, `margin`, `padding`, `top`, `left` | Layout + Paint + Composite |
| `color`, `background-color`, `box-shadow`, `border-color` | Paint + Composite |
| `transform`, `opacity` | Composite only |
| `filter` (on composited layer) | Composite only |

```css
/* BAD: Animating left triggers layout on every frame */
.slide-bad {
  transition: left 0.3s ease;
  position: absolute;
  left: 0;
}
.slide-bad.active {
  left: 200px;
}

/* GOOD: Animating transform only triggers composite */
.slide-good {
  transition: transform 0.3s ease;
  transform: translateX(0);
}
.slide-good.active {
  transform: translateX(200px);
}
```

### 3.4 Measuring the CRP

```javascript
// Use the Performance API to measure CRP milestones
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.log(`${entry.name}: ${entry.startTime.toFixed(0)}ms`);
  }
});

// These capture key CRP moments
observer.observe({ type: 'paint', buffered: true });
// Output:
// first-paint: 342ms
// first-contentful-paint: 345ms

// Navigation Timing API for detailed breakdown
window.addEventListener('load', () => {
  const timing = performance.getEntriesByType('navigation')[0];
  console.log({
    dns:        timing.domainLookupEnd - timing.domainLookupStart,
    tcp:        timing.connectEnd - timing.connectStart,
    ttfb:       timing.responseStart - timing.requestStart,
    download:   timing.responseEnd - timing.responseStart,
    domParse:   timing.domInteractive - timing.responseEnd,
    domReady:   timing.domContentLoadedEventEnd - timing.domInteractive,
    fullLoad:   timing.loadEventEnd - timing.navigationStart,
  });
});
```

## 4. Render-Blocking Resources

A render-blocking resource is anything that prevents the browser from rendering pixels to the screen. The three main culprits are CSS, synchronous JavaScript, and web fonts.

### 4.1 CSS is Render-Blocking by Default

Every `<link rel="stylesheet">` in the `<head>` blocks rendering until it downloads and parses. The browser does this intentionally to prevent a flash of unstyled content.

**Strategy 1: Inline Critical CSS**

Extract the CSS needed for above-the-fold content and inline it directly in the HTML. Load the rest asynchronously.

```html
<head>
  <!-- Critical CSS inlined: renders immediately, no network request -->
  <style>
    /* Only styles needed for the initial viewport */
    body { margin: 0; font-family: system-ui, sans-serif; }
    .header { background: #1a1a2e; color: white; padding: 1rem; }
    .hero { padding: 2rem; text-align: center; }
    .hero img { max-width: 100%; height: auto; }
  </style>

  <!-- Non-critical CSS loaded asynchronously -->
  <!-- This pattern: media="print" loads without blocking, then swaps -->
  <link rel="stylesheet"
        href="/styles/full.css"
        media="print"
        onload="this.media='all'">
  <noscript>
    <link rel="stylesheet" href="/styles/full.css">
  </noscript>
</head>
```

**Strategy 2: Split CSS by Media Query**

```html
<!-- Only the matching stylesheet blocks rendering -->
<!-- On mobile, desktop.css downloads but does NOT block rendering -->
<link rel="stylesheet" href="/mobile.css" media="(max-width: 767px)">
<link rel="stylesheet" href="/desktop.css" media="(min-width: 768px)">
<link rel="stylesheet" href="/print.css" media="print">
```

The browser downloads all stylesheets regardless, but only render-blocks on the one that matches the current media. On a phone, `desktop.css` downloads at low priority without blocking paint.

**Strategy 3: Reduce CSS Size**

```css
/* Remove unused CSS with tools like PurgeCSS, UnCSS, or Tailwind's purge */
/* Minify CSS: remove whitespace, shorten values */

/* BEFORE: 847 bytes */
.navigation-container {
  background-color: #ffffff;
  padding-top: 16px;
  padding-right: 24px;
  padding-bottom: 16px;
  padding-left: 24px;
}

/* AFTER: 87 bytes */
.nav{background:#fff;padding:16px 24px}
```

### 4.2 JavaScript is Parser-Blocking by Default

```html
<!-- BAD: Blocks HTML parsing until script downloads and executes -->
<head>
  <script src="/analytics.js"></script>   <!-- Parser stops here -->
  <script src="/app.js"></script>          <!-- Parser still stopped -->
</head>

<!-- GOOD: defer maintains order, executes after parsing -->
<head>
  <script src="/analytics.js" defer></script>
  <script src="/app.js" defer></script>
</head>
```

See [Section 9: JavaScript Performance](#9-javascript-performance) for the full async/defer/module breakdown.

### 4.3 Fonts are Render-Blocking (by Default)

When a web font is declared in CSS and used in visible text, browsers hide that text until the font downloads (FOIT). On slow connections, users see blank text for seconds.

See [Section 7: Web Font Optimization](#7-web-font-optimization) for solutions.

## 5. Resource Hints

Resource hints tell the browser about resources it will need soon, allowing it to start fetching them before it discovers them naturally during parsing. On mobile, where latency is high, saving even one round-trip can mean hundreds of milliseconds.

### 5.1 Quick Reference

| Hint | What It Does | When to Use | Cost |
|---|---|---|---|
| `dns-prefetch` | Resolves DNS for a domain | Third-party domains you will use | Very low |
| `preconnect` | DNS + TCP + TLS handshake | Critical third-party origins | Low-medium |
| `preload` | Fetches a specific resource with high priority | Resources needed for current page (LCP image, critical font) | Medium |
| `prefetch` | Fetches a resource for future navigation | Resources for the next likely page | Low (may be ignored on slow connections) |
| `modulepreload` | Preloads an ES module and its dependencies | Critical JS modules | Medium |
| `fetchpriority` | Adjusts relative priority of a fetch | Fine-tune which images/scripts load first | None (metadata only) |

### 5.2 `dns-prefetch`

```html
<!-- Resolve DNS for third-party origins early -->
<!-- Cost: one DNS lookup (~20-120ms saved later) -->
<link rel="dns-prefetch" href="https://fonts.googleapis.com">
<link rel="dns-prefetch" href="https://cdn.analytics.example.com">
```

### 5.3 `preconnect`

```html
<!-- Full connection setup: DNS + TCP + TLS -->
<!-- Saves 100-300ms on mobile for each connection -->
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link rel="preconnect" href="https://api.example.com">

<!-- GOTCHA: Do not preconnect to more than 4-6 origins -->
<!-- Each open connection consumes memory and CPU -->
<!-- Unused connections waste resources and may be closed by the OS -->
```

### 5.4 `preload`

```html
<!-- Preload the LCP image -->
<link rel="preload" as="image" href="/hero-mobile.webp"
      fetchpriority="high"
      media="(max-width: 767px)">
<link rel="preload" as="image" href="/hero-desktop.webp"
      fetchpriority="high"
      media="(min-width: 768px)">

<!-- Preload a critical font -->
<link rel="preload" as="font" href="/fonts/body.woff2"
      type="font/woff2" crossorigin>

<!-- Preload critical CSS that is loaded via @import (otherwise late-discovered) -->
<link rel="preload" as="style" href="/critical-component.css">
```

**Common preload mistakes:**

```html
<!-- BAD: Preloading everything defeats the purpose -->
<link rel="preload" as="image" href="/icon1.svg">
<link rel="preload" as="image" href="/icon2.svg">
<link rel="preload" as="image" href="/icon3.svg">
<link rel="preload" as="image" href="/icon4.svg">
<!-- When everything is high priority, nothing is -->

<!-- BAD: Preloading without using the resource -->
<!-- Chrome will warn: "The resource was preloaded but not used within
     a few seconds. Make sure it has an appropriate 'as' value and
     it is preloaded intentionally." -->
<link rel="preload" as="script" href="/unused-module.js">

<!-- BAD: Wrong 'as' value causes double download -->
<link rel="preload" as="script" href="/styles.css"> <!-- Wrong! -->
<link rel="preload" as="style" href="/styles.css">  <!-- Correct -->
```

### 5.5 `prefetch`

```html
<!-- Prefetch resources for the next likely navigation -->
<!-- Low priority: browser may skip on slow/metered connections -->
<link rel="prefetch" href="/about.html">
<link rel="prefetch" as="script" href="/about-page-bundle.js">

<!-- GOTCHA: Prefetched resources go into the HTTP cache -->
<!-- If the cache is full, they may evict more important resources -->
<!-- Use sparingly: 1-3 resources maximum -->
```

### 5.6 `modulepreload`

```html
<!-- Preloads an ES module AND its static import tree -->
<!-- Unlike preload, this also compiles the module -->
<link rel="modulepreload" href="/js/app.mjs">
<link rel="modulepreload" href="/js/router.mjs">

<!-- Without modulepreload, the browser discovers imports sequentially:
     app.mjs -> import router.mjs -> import utils.mjs
     Each step waits for download + parse of the previous step.
     modulepreload starts all of them in parallel. -->
```

### 5.7 `fetchpriority`

```html
<!-- Tell the browser which images matter most -->
<!-- HIGH: LCP image should load before other images -->
<img src="/hero.webp" fetchpriority="high" alt="Hero">

<!-- LOW: Below-the-fold images, carousel slides not yet visible -->
<img src="/slide-3.webp" fetchpriority="low" loading="lazy" alt="Slide 3">

<!-- Works on scripts too -->
<script src="/critical-app.js" fetchpriority="high"></script>
<script src="/analytics.js" fetchpriority="low" defer></script>

<!-- Works on fetch() API calls -->
<script>
  // High priority: data needed for initial render
  fetch('/api/products', { priority: 'high' });

  // Low priority: prefetching data for later
  fetch('/api/recommendations', { priority: 'low' });
</script>
```

## 6. Image Optimization

Images are typically the largest payload on mobile pages. A single unoptimized hero image can exceed the size of all JavaScript on the page. Getting images right is the highest-impact optimization for most sites.

### 6.1 Format Comparison

| Format | Compression | Transparency | Animation | Browser Support | Best For |
|---|---|---|---|---|---|
| JPEG | Lossy | No | No | 100% | Photos (legacy) |
| PNG | Lossless | Yes | No | 100% | Icons, screenshots (legacy) |
| WebP | Lossy + Lossless | Yes | Yes | ~97% | Photos, graphics (default choice) |
| AVIF | Lossy + Lossless | Yes | Yes | ~92% | Photos (best compression) |
| SVG | N/A (vector) | Yes | Yes | 100% | Icons, logos, illustrations |

**Size comparison for a typical photo at equivalent visual quality:**

| Format | File Size | Savings vs JPEG |
|---|---|---|
| JPEG (quality 80) | 200 KB | -- |
| WebP (quality 75) | 130 KB | 35% smaller |
| AVIF (quality 60) | 100 KB | 50% smaller |

### 6.2 The `<picture>` Element for Format Fallback

```html
<!-- Browser picks the first <source> it supports, falls back to <img> -->
<picture>
  <!-- AVIF: best compression, ~92% support -->
  <source srcset="/hero.avif" type="image/avif">
  <!-- WebP: great compression, ~97% support -->
  <source srcset="/hero.webp" type="image/webp">
  <!-- JPEG: universal fallback -->
  <img src="/hero.jpg"
       alt="Product showcase"
       width="1200" height="800"
       fetchpriority="high">
</picture>
```

### 6.3 Responsive Images with `srcset` and `sizes`

```html
<!-- Serve different sizes based on viewport width -->
<img
  srcset="
    /product-400.webp   400w,
    /product-800.webp   800w,
    /product-1200.webp 1200w,
    /product-1600.webp 1600w
  "
  sizes="
    (max-width: 600px) 100vw,
    (max-width: 1200px) 50vw,
    33vw
  "
  src="/product-800.webp"
  alt="Product photo"
  width="1200" height="800"
  loading="lazy">

<!--
  How the browser decides:
  1. On a 375px phone (2x DPR): needs 375 * 2 = 750px image
     -> picks product-800.webp (closest >= 750)
  2. On a 1440px desktop: sizes says 33vw = 480px, 1x DPR
     -> picks product-800.webp (closest >= 480)
-->
```

### 6.4 Lazy Loading

```html
<!-- Native lazy loading: browser handles intersection detection -->
<img src="/photo.webp"
     loading="lazy"
     alt="Below the fold content"
     width="600" height="400">

<!-- Also works on iframes -->
<iframe src="https://youtube.com/embed/xyz"
        loading="lazy"
        width="560" height="315"
        title="Video"></iframe>
```

**When NOT to lazy load:**
- The LCP element (hero image, main banner)
- Images in the first viewport (they should load immediately)
- Small icons and logos (overhead of lazy loading exceeds benefit)

**Custom lazy loading with IntersectionObserver:**

```javascript
// For background images or more complex lazy loading scenarios
const lazyImages = document.querySelectorAll('[data-src]');

const imageObserver = new IntersectionObserver((entries, observer) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      const img = entry.target;
      // Swap data-src to src to trigger load
      img.src = img.dataset.src;
      if (img.dataset.srcset) {
        img.srcset = img.dataset.srcset;
      }
      img.removeAttribute('data-src');
      observer.unobserve(img); // Stop watching once loaded
    }
  });
}, {
  // Start loading 200px before the image enters the viewport
  rootMargin: '200px 0px',
  threshold: 0
});

lazyImages.forEach(img => imageObserver.observe(img));
```

### 6.5 Mobile Image Size Targets

| Image Type | Target Size (mobile) | Notes |
|---|---|---|
| Hero / Banner | 150-300 KB | The most impactful image to optimize |
| Product photo | 50-150 KB | Use responsive srcset |
| Thumbnail | 10-30 KB | Aggressively compress |
| Icon / Logo | 1-5 KB | Use SVG when possible |
| Decorative | 0 KB | Consider CSS gradients or remove |

### 6.6 Image Performance Gotchas

```html
<!-- BAD: Serving desktop images to mobile -->
<img src="/hero-3840x2160.jpg" alt="Hero">
<!-- A 3840px image on a 375px phone wastes 90%+ of the data -->

<!-- BAD: No width/height causes CLS -->
<img src="/photo.webp" alt="Photo">
<!-- Browser doesn't know dimensions until download = layout shift -->

<!-- BAD: Lazy loading the LCP image -->
<img src="/hero.webp" loading="lazy" alt="Hero">
<!-- Delays the most important image on the page -->

<!-- BAD: Using PNG for photos -->
<img src="/photo.png" alt="Photo">
<!-- PNG is lossless: 5x-10x larger than WebP for photos -->

<!-- GOOD: Complete optimized image -->
<img
  srcset="/photo-400.avif 400w, /photo-800.avif 800w"
  sizes="(max-width: 600px) 100vw, 50vw"
  src="/photo-800.webp"
  alt="Descriptive alt text"
  width="800" height="600"
  loading="lazy"
  decoding="async">
```

### 6.7 The `decoding` Attribute

```html
<!-- decoding="async" tells the browser it can decode the image
     off the main thread, avoiding jank during decode -->
<img src="/large-photo.webp"
     decoding="async"
     alt="Large photo"
     width="1200" height="800">

<!-- decoding="sync" forces synchronous decode (default for most browsers) -->
<!-- Only use sync for the LCP image if you need guaranteed immediate display -->
```

## 7. Web Font Optimization

Web fonts are a hidden performance killer on mobile. A single font file can be 50-200 KB, and browsers hide text until the font loads (Flash of Invisible Text). On a 3G connection, users may see blank text for 3-5 seconds.

### 7.1 FOIT vs FOUT

| Behavior | What Happens | User Impact |
|---|---|---|
| **FOIT** (Flash of Invisible Text) | Browser hides text until font loads | Users see blank space, cannot read content |
| **FOUT** (Flash of Unstyled Text) | Browser shows fallback font, swaps when font loads | Users can read immediately, slight visual shift |

FOUT is almost always better for performance and usability. Users can start reading immediately.

### 7.2 `font-display` Property

```css
@font-face {
  font-family: 'CustomFont';
  src: url('/fonts/custom.woff2') format('woff2');

  /* swap: FOUT. Shows fallback immediately, swaps when ready.
     Best for body text. May cause CLS if metrics differ. */
  font-display: swap;
}

@font-face {
  font-family: 'HeadingFont';
  src: url('/fonts/heading.woff2') format('woff2');

  /* fallback: Short FOIT (~100ms block), then fallback.
     If font doesn't load within swap period (~3s), fallback stays.
     Best balance of design and performance. */
  font-display: fallback;
}

@font-face {
  font-family: 'OptionalFont';
  src: url('/fonts/optional.woff2') format('woff2');

  /* optional: Very short FOIT (~100ms block). If font isn't ready,
     fallback stays for the ENTIRE page session.
     Font is still cached for next navigation.
     Best for non-critical decorative fonts. */
  font-display: optional;
}
```

| `font-display` Value | Block Period | Swap Period | Best For |
|---|---|---|---|
| `auto` | Browser decides | Browser decides | Never use explicitly |
| `block` | 3 seconds | Infinite | Icon fonts (only) |
| `swap` | ~0ms | Infinite | Body text, headings |
| `fallback` | ~100ms | ~3 seconds | Good compromise |
| `optional` | ~100ms | None | Decorative, non-essential |

### 7.3 Font Preloading

```html
<!-- Preload critical fonts to avoid late discovery -->
<!-- Without preload, the browser must:
     1. Download HTML
     2. Parse CSS
     3. Find @font-face declaration
     4. Determine the font is needed for visible text
     5. THEN start downloading the font
     Preload skips steps 2-4 -->
<link rel="preload"
      href="/fonts/body-regular.woff2"
      as="font"
      type="font/woff2"
      crossorigin>

<!-- crossorigin is REQUIRED even for same-origin fonts -->
<!-- Without it, the font downloads twice -->
```

### 7.4 Font Subsetting

```css
/* Full Latin font: ~80 KB */
/* Subset with only characters you use: ~15 KB */

@font-face {
  font-family: 'CustomFont';
  src: url('/fonts/custom-latin.woff2') format('woff2');
  /* unicode-range tells the browser to only download this file
     if characters in this range are actually used on the page */
  unicode-range: U+0000-00FF, U+0131, U+0152-0153, U+02BB-02BC;
}

/* Google Fonts does this automatically with per-range CSS files */
```

**Command-line subsetting with `pyftsubset`:**

```bash
# Install: pip install fonttools brotli
# Subset to Latin characters only, output as WOFF2
pyftsubset CustomFont.ttf \
  --output-file=CustomFont-latin.woff2 \
  --flavor=woff2 \
  --layout-features='kern,liga' \
  --unicodes="U+0000-00FF,U+0131,U+0152-0153"
```

### 7.5 Variable Fonts

```css
/* Instead of loading 4 separate files (regular, bold, italic, bold-italic),
   load ONE variable font file that contains all weights and styles */
@font-face {
  font-family: 'InterVariable';
  src: url('/fonts/Inter-Variable.woff2') format('woff2-variations');
  font-weight: 100 900; /* Supports any weight in this range */
  font-display: swap;
}

h1 { font-weight: 800; }   /* All from the same file */
body { font-weight: 400; }
.light { font-weight: 300; }
.bold { font-weight: 700; }

/* Size savings: 4 static files (~240 KB) vs 1 variable file (~150 KB) */
/* Plus: only 1 network request instead of 4 */
```

### 7.6 Self-Hosting vs CDN

| Approach | Pros | Cons |
|---|---|---|
| Google Fonts CDN | Easy setup, auto-subsetting | Extra DNS + connection, no cache sharing (since Chrome 86), privacy concerns |
| Self-hosted | Full control, no third-party connection, can preload | Must manage subsetting, compression yourself |

**Self-hosting is now the recommended approach** because modern browsers partition their caches by top-level domain. Two sites using the same Google Fonts URL no longer share the cached font file.

```html
<!-- BEFORE: Two extra connections (googleapis.com + gstatic.com) -->
<link href="https://fonts.googleapis.com/css2?family=Inter:wght@400;700"
      rel="stylesheet">

<!-- AFTER: Self-hosted, preloaded, no third-party connections -->
<link rel="preload" href="/fonts/inter-var.woff2"
      as="font" type="font/woff2" crossorigin>
<style>
  @font-face {
    font-family: 'Inter';
    src: url('/fonts/inter-var.woff2') format('woff2-variations');
    font-weight: 100 900;
    font-display: swap;
  }
</style>
```

### 7.7 Reducing Font CLS with Metric Overrides

```css
/* The biggest source of font-related CLS is the difference in
   character metrics between the fallback font and the web font.
   Use @font-face descriptors to align them. */

@font-face {
  font-family: 'Inter';
  src: url('/fonts/inter.woff2') format('woff2');
  font-display: swap;

  /* These descriptors adjust the fallback font's metrics
     to match Inter's metrics, minimizing layout shift */
  ascent-override: 90%;
  descent-override: 22%;
  line-gap-override: 0%;
  size-adjust: 107%;
}

/* Tools to calculate these values:
   - https://screenspan.net/fallback
   - https://seek-oss.github.io/capsize/
*/
```

## 8. CSS Performance

CSS affects performance at every stage of the rendering pipeline: download, parsing, style calculation, layout, paint, and composite. On mobile, where CPU is limited, inefficient CSS is measurably slower.

### 8.1 Critical CSS

Critical CSS is the minimum CSS required to render the above-the-fold content. Inline it in the `<head>` so the browser can render without waiting for external stylesheets.

**Tools for extracting critical CSS:**
- `critical` (npm package by Addy Osmani)
- `critters` (used by many build tools)
- `penthouse` (headless Chrome-based)

```javascript
// Build script example using the 'critical' package
const critical = require('critical');

critical.generate({
  inline: true,               // Inline critical CSS into HTML
  base: 'dist/',
  src: 'index.html',
  target: 'index.html',
  width: 375,                 // Mobile viewport width
  height: 812,                // Mobile viewport height
  penthouse: {
    timeout: 60000
  }
});
```

### 8.2 Removing Unused CSS

```bash
# Check how much CSS is unused using Chrome DevTools:
# 1. Open DevTools > More Tools > Coverage
# 2. Reload the page
# 3. Red bars = unused CSS bytes

# Typical findings: 60-90% of framework CSS is unused on any given page
```

**PurgeCSS integration:**

```javascript
// postcss.config.js
module.exports = {
  plugins: [
```

### 8.3 CSS Containment

The `contain` property tells the browser that an element's internals are independent from the rest of the page. This lets the browser skip expensive recalculations.

```css
/* contain: layout - element's layout doesn't affect siblings */
/* contain: paint  - nothing inside paints outside bounds */
/* contain: size   - element's size doesn't depend on children */
/* contain: style  - counters/quotes scoped to this subtree */

/* Practical example: a card in a grid */
.card {
  contain: layout paint;
  /* Browser knows:
     - Changes inside .card won't affect layout of other cards
     - Nothing inside .card paints outside its border box
     - So it can skip re-laying-out siblings when .card changes */
}

/* contain: strict = layout + paint + size */
/* Use when you know the exact size */
.fixed-widget {
  contain: strict;
  width: 300px;
  height: 250px;
}

/* contain: content = layout + paint (most common shorthand) */
.article-section {
  contain: content;
}
```

### 8.4 `content-visibility`

`content-visibility: auto` is one of the highest-impact CSS performance properties available. It tells the browser to skip rendering of off-screen content entirely: no layout, no paint, no style calculation.

```css
/* Apply to sections that start below the fold */
.content-section {
  content-visibility: auto;

  /* CRITICAL: Always set contain-intrinsic-size */
  /* Without it, the scrollbar jumps as sections render */
  /* This gives the browser a placeholder size estimate */
  contain-intrinsic-size: auto 500px;
}

/* Real-world example: a long article with many sections */
article > section {
  content-visibility: auto;
  contain-intrinsic-size: auto 300px;
}

/* Real-world example: a product list */
.product-card {
  content-visibility: auto;
  contain-intrinsic-size: auto 400px;
}
```

**Impact:** In benchmarks, `content-visibility: auto` provides a 7x rendering performance improvement on pages with lots of off-screen content. The improvement is largest on mobile where CPU is the bottleneck.

**Browser support:** Baseline since September 2025 (Chrome, Firefox, Safari all support it).

**Gotchas:**
- Elements with `content-visibility: auto` are not searchable with `Ctrl+F` until they render (browser may render them for search)
- Accessibility tree is still built, so screen readers work correctly
- Anchor links to hidden content may not scroll correctly without browser intervention

### 8.5 `will-change`

`will-change` hints to the browser that a property will animate, allowing it to set up optimizations (like promoting to a GPU layer) in advance.

```css
/* GOOD: Apply before the animation starts, remove after */
.dropdown-trigger:hover + .dropdown {
  will-change: transform, opacity;
}
.dropdown.animating {
  will-change: transform, opacity;
  transition: transform 0.3s, opacity 0.3s;
}

/* BAD: Applying will-change to everything */
* {
  will-change: transform; /* Creates a GPU layer for EVERY element */
  /* Each layer consumes VRAM: 100 elements = potential crash on mobile */
}

/* BAD: Permanent will-change */
.card {
  will-change: transform; /* Wastes GPU memory forever */
}

/* GOOD: Apply via JS only when needed */
```

```javascript
// Apply will-change just before animation, remove after
const element = document.querySelector('.panel');

element.addEventListener('mouseenter', () => {
  element.style.willChange = 'transform';
});

element.addEventListener('transitionend', () => {
  element.style.willChange = 'auto'; // Release GPU memory
});
```

**Rules for `will-change`:**
1. Do not apply it to more than a handful of elements
2. Do not apply it permanently via CSS (apply via JS, remove after animation)
3. Do not apply it to elements that are not going to change
4. If performance is fine without it, do not add it

### 8.6 Selector Performance

Modern browsers are fast at matching selectors, but extremely complex selectors still matter on mobile with thousands of elements.

```css
/* Browsers match selectors RIGHT to LEFT */

/* BAD: Universal selector forces checking every element */
.sidebar > * > .icon { }

/* BAD: Deep descendant with tag qualifier */
body .main .content .list li a span { }

/* GOOD: Specific class, shallow nesting */
.nav-icon { }
.list-item-link { }

/* GOTCHA: :nth-child() with complex formulas is expensive
   in style recalculation when DOM changes frequently */
.grid-item:nth-child(3n + 1):not(.featured):last-of-type { }
```

In practice, selector performance is rarely the bottleneck. Focus on reducing total CSS size and leveraging containment first.

## 9. JavaScript Performance

JavaScript is the single largest threat to mobile performance. It must be downloaded, parsed, compiled, and executed, and all of this happens on a CPU that is 3-5x slower than the developer's machine.

### 9.1 `async` vs `defer` vs `module`

```
NORMAL <script>:
  HTML: ===X---------X========>
  JS:       [download][execute]
  (Parser blocks at X, resumes after execution)

ASYNC <script>:
  HTML: ==================>       (may pause briefly for execution)
  JS:   [download]
                   [execute]      (executes as soon as downloaded)
  (Non-deterministic order if multiple async scripts)

DEFER <script>:
  HTML: ==================>
  JS:   [download]
                          [execute]  (after HTML parsing, before DOMContentLoaded)
  (Preserves script order)

MODULE <script type="module">:
  HTML: ==================>
  JS:   [download + deps]
                          [execute]  (deferred by default, strict mode)
  (Also resolves import tree in parallel with modulepreload)
```

| Attribute | Download | Execution | Order Preserved | Use Case |
|---|---|---|---|---|
| (none) | Blocks parsing | Blocks parsing | Yes | Legacy, avoid |
| `async` | Parallel | ASAP after download | No | Analytics, ads, independent scripts |
| `defer` | Parallel | After HTML parse | Yes | App scripts that need DOM |
| `type="module"` | Parallel | After HTML parse | Yes (import order) | Modern ES module apps |

```html
<!-- Recommended pattern for most sites -->
<head>
  <!-- Critical: preload the main bundle -->
  <link rel="modulepreload" href="/js/app.mjs">

  <!-- Analytics: async because it's independent -->
  <script src="/js/analytics.js" async></script>

  <!-- App code: defer to maintain order, execute after parse -->
  <script src="/js/vendor.js" defer></script>
  <script src="/js/app.js" defer></script>
</head>
```

**Measurable impact:** Using `defer` reduces First Contentful Paint by approximately 19% on mobile compared to synchronous loading.

### 9.2 Code Splitting

Do not ship a single massive bundle. Split code by route and load only what is needed.

```javascript
// Static import: included in the main bundle
import { heavyLibrary } from './heavy-library.js';

// Dynamic import: creates a separate chunk, loaded on demand
button.addEventListener('click', async () => {
  // This chunk only downloads when the user clicks
  const { heavyLibrary } = await import('./heavy-library.js');
  heavyLibrary.doWork();
});
```

**Route-based splitting (vanilla JS):**

```javascript
// Simple router with code splitting
const routes = {
  '/': () => import('./pages/home.js'),
  '/products': () => import('./pages/products.js'),
  '/cart': () => import('./pages/cart.js'),
};

async function navigate(path) {
  const loader = routes[path];
  if (loader) {
    const module = await loader();
    module.render(document.getElementById('app'));
  }
}

// Each page is a separate bundle, downloaded only when navigated to
```

### 9.3 Tree Shaking

Tree shaking eliminates unused exports from bundles. It works with ES module `import`/`export` syntax.

```javascript
// math-utils.js
export function add(a, b) { return a + b; }
export function subtract(a, b) { return a - b; }
export function multiply(a, b) { return a * b; }
export function divide(a, b) { return a / b; }

// app.js - only imports add
import { add } from './math-utils.js';
console.log(add(2, 3));

// After tree shaking: subtract, multiply, divide are removed from bundle
```

**What breaks tree shaking:**

```javascript
// BAD: Importing the entire library
import _ from 'lodash';           // Imports ALL of lodash (~70 KB)
_.get(obj, 'path');

// GOOD: Import only what you need
import get from 'lodash-es/get';  // Imports only get (~1 KB)
get(obj, 'path');

// BAD: Side effects prevent tree shaking
// If a module has top-level code that runs on import, bundlers
// cannot safely remove it
import './polyfills.js';  // Must run, cannot be removed

// sideEffects: false in package.json tells bundlers
// that all files in the package are safe to tree-shake
```

### 9.4 Long Tasks and the Main Thread

Any task that runs for more than 50ms is a "long task." During a long task, the browser cannot respond to user input, update animations, or perform garbage collection.

```javascript
// Detecting long tasks
const longTaskObserver = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    // entry.duration is the task length in ms
    console.warn(`Long task detected: ${entry.duration.toFixed(0)}ms`);
    // entry.attribution tells you what script caused it
    if (entry.attribution) {
      console.warn('Source:', entry.attribution[0]?.name);
    }
  }
});
longTaskObserver.observe({ type: 'longtask', buffered: true });
```

**Breaking up long tasks:**

```javascript
// BAD: Sorting 10,000 items in one task = 200ms+ on mobile
function renderSortedList(items) {
  items.sort((a, b) => expensiveCompare(a, b));  // 150ms
  items.forEach(item => createDOMNode(item));      // 100ms
  // Total: 250ms long task, UI frozen the entire time
}

// GOOD: Chunk the work
async function renderSortedList(items) {
  // Phase 1: Sort (CPU-bound, still one task but isolated)
  items.sort((a, b) => expensiveCompare(a, b));
  await yieldToMain();  // Let browser process any pending input

  // Phase 2: Render in chunks of 100
  const CHUNK_SIZE = 100;
  for (let i = 0; i < items.length; i += CHUNK_SIZE) {
    const chunk = items.slice(i, i + CHUNK_SIZE);
    chunk.forEach(item => createDOMNode(item));
    await yieldToMain();  // Yield between chunks
  }
}

function yieldToMain() {
  if (globalThis.scheduler?.yield) return scheduler.yield();
  return new Promise(resolve => setTimeout(resolve, 0));
}
```

### 9.5 Web Workers

Offload CPU-intensive work to a background thread so the main thread stays responsive.

```javascript
// worker.js - runs in a separate thread
self.addEventListener('message', (event) => {
  const { data, action } = event.data;

  if (action === 'sort') {
    // This sort runs on the worker thread
    // Main thread stays responsive during the entire operation
    const sorted = data.sort((a, b) => a.name.localeCompare(b.name));
    self.postMessage({ action: 'sorted', result: sorted });
  }

  if (action === 'filter') {
    const filtered = data.filter(item =>
      item.name.toLowerCase().includes(event.data.query)
    );
    self.postMessage({ action: 'filtered', result: filtered });
  }
});

// main.js
const worker = new Worker('/worker.js');

worker.addEventListener('message', (event) => {
  if (event.data.action === 'sorted') {
    renderList(event.data.result);
  }
});

// Kick off sorting without blocking the main thread
worker.postMessage({ action: 'sort', data: largeArray });
```

**When to use Web Workers:**
- Sorting/filtering large datasets (1000+ items)
- Image processing (canvas manipulation)
- Data parsing (CSV, JSON of 1 MB+)
- Complex calculations (crypto, compression)
- Markdown/text rendering

**When NOT to use Web Workers:**
- DOM manipulation (workers have no DOM access)
- Small operations (postMessage overhead exceeds computation time)
- Operations that need synchronous results

### 9.6 `requestIdleCallback`

Schedule low-priority work during idle periods when the browser has nothing else to do.

```javascript
// Schedule non-critical work during idle time
function sendAnalytics(data) {
  if ('requestIdleCallback' in window) {
    requestIdleCallback((deadline) => {
      // deadline.timeRemaining() tells you how many ms you have
      // before the browser needs the main thread back
      if (deadline.timeRemaining() > 5) {
        navigator.sendBeacon('/analytics', JSON.stringify(data));
      }
    }, { timeout: 2000 }); // Force execution after 2s max
  } else {
    // Fallback for Safari (lacks requestIdleCallback)
    setTimeout(() => {
      navigator.sendBeacon('/analytics', JSON.stringify(data));
    }, 0);
  }
}
```

## 10. Network Optimization

Network is the first bottleneck users hit. Before the browser can parse, render, or execute anything, it must download bytes over the network. On mobile, every round-trip costs 50-300ms.

### 10.1 HTTP/2 vs HTTP/3

| Feature | HTTP/1.1 | HTTP/2 | HTTP/3 (QUIC) |
|---|---|---|---|
| Multiplexing | No (6 conn. per origin) | Yes (single connection) | Yes (single connection) |
| Header compression | No | HPACK | QPACK |
| Head-of-line blocking | Per connection | Per connection (TCP) | Per stream (none) |
| Connection setup | TCP + TLS (3 RTTs) | TCP + TLS (2 RTTs) | 0-1 RTT |
| Connection migration | No | No | Yes (handles network switch) |
| Protocol | TCP | TCP | UDP (QUIC) |

**HTTP/3 reduces latency by approximately 30% on mobile connections** because:
1. 0-RTT connection resumption (no handshake on repeat visits)
2. No head-of-line blocking: a lost packet in one stream does not block others
3. Connection migration: switching from Wi-Fi to cellular does not drop the connection

### 10.2 Compression

```
                    GZIP        Brotli      Savings (typical HTML/CSS/JS)
Compression ratio:  70-80%      75-85%      Brotli ~15-20% smaller than gzip
CPU cost:           Low         Medium      Use Brotli level 4-6 (not 11)
Browser support:    ~100%       ~97%        Always serve Brotli with gzip fallback
```

**Server configuration (nginx):**

```nginx
# Enable Brotli (requires ngx_brotli module)
brotli on;
brotli_comp_level 6;        # 4-6 is the sweet spot (11 is too slow)
brotli_types
  text/html
  text/css
  text/javascript
  application/javascript
  application/json
  image/svg+xml;

# Gzip fallback
gzip on;
gzip_comp_level 5;
gzip_types
  text/html
  text/css
  text/javascript
  application/javascript
  application/json
  image/svg+xml;
```

**What to compress and what to skip:**

| Compress | Skip |
|---|---|
| HTML, CSS, JS | JPEG, PNG, WebP, AVIF (already compressed) |
| JSON, XML, SVG | WOFF2 (already Brotli-compressed internally) |
| Plain text, CSV | Video files (MP4, WebM) |

### 10.3 Caching Headers

```
                       Cache-Control Header                    When to Use
Static assets         Cache-Control: public, max-age=31536000  CSS/JS with hash in filename
  (hashed filenames)  (1 year, immutable)                      /app.a1b2c3.js

HTML documents        Cache-Control: no-cache                  Always revalidated
                      (or max-age=0, must-revalidate)

API responses         Cache-Control: private, max-age=60       User-specific data, short cache
                      (or no-store for sensitive data)

Images (hashed)       Cache-Control: public, max-age=31536000  /hero.a1b2c3.webp
Images (unhashed)     Cache-Control: public, max-age=86400     /avatar.webp (changes occasionally)
```

```
# Ideal caching setup for a production site:

# index.html - always check for updates
Cache-Control: no-cache
ETag: "abc123"

# app.f8e2a1.js - cached forever (filename changes on update)
Cache-Control: public, max-age=31536000, immutable

# styles.c4d5e6.css - cached forever
Cache-Control: public, max-age=31536000, immutable

# api/products - cache for 5 minutes
Cache-Control: private, max-age=300
Vary: Accept-Encoding
```

### 10.4 CDN

A CDN (Content Delivery Network) serves your assets from edge servers geographically close to the user. Instead of a 200ms RTT to a single origin server, the user gets a 20-50ms RTT to a nearby edge.

**CDN impact on mobile:**
- Reduces TTFB by 100-300ms on average
- Edge caching means popular resources are served without hitting origin
- Modern CDNs handle Brotli compression at the edge
- Some CDNs (Cloudflare, Fastly) support HTTP/3 out of the box

### 10.5 Reducing Requests

Each HTTP request on mobile carries overhead: DNS lookup, TCP connection, TLS handshake, and latency. While HTTP/2 multiplexing helps, reducing total requests still matters.

```html
<!-- BAD: Many small files = many requests -->
<link rel="stylesheet" href="/reset.css">
<link rel="stylesheet" href="/typography.css">
<link rel="stylesheet" href="/layout.css">
<link rel="stylesheet" href="/components.css">
<link rel="stylesheet" href="/utilities.css">

<!-- GOOD: Bundle into fewer files -->
<link rel="stylesheet" href="/styles.css">

<!-- For icons: use an SVG sprite instead of individual files -->
<svg class="icon" aria-hidden="true">
  <use href="/icons.svg#search"></use>
</svg>

<!-- Inline small assets (< 1 KB) as data URIs or inline SVG -->
<img src="data:image/svg+xml,<svg>...</svg>" alt="">
```

## 11. Service Workers and Caching

Service workers intercept network requests and serve responses from cache, making pages load instantly on repeat visits and work offline. On mobile, where network is unreliable, this is transformative.

### 11.1 Service Worker Lifecycle

```javascript
// sw.js - the service worker file

// 1. INSTALL: Cache critical assets
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open('static-v1').then((cache) => {
      return cache.addAll([
        '/',
        '/styles.css',
        '/app.js',
        '/offline.html',
      ]);
    })
  );
  // Skip waiting to activate immediately
  self.skipWaiting();
});

// 2. ACTIVATE: Clean up old caches
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then((keys) => {
      return Promise.all(
        keys
          .filter(key => key !== 'static-v1' && key !== 'dynamic-v1')
          .map(key => caches.delete(key))
      );
    })
  );
  // Take control of all tabs immediately
  self.clients.claim();
});

// 3. FETCH: Intercept requests and serve from cache
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then((cached) => {
      return cached || fetch(event.request);
    })
  );
});
```

### 11.2 Caching Strategies

| Strategy | How It Works | Best For |
|---|---|---|
| **Cache First** | Check cache, fall back to network | Static assets (CSS, JS, images) |
| **Network First** | Try network, fall back to cache | API data, HTML pages |
| **Stale While Revalidate** | Serve cache immediately, update in background | Frequently updated content |
| **Cache Only** | Only serve from cache | Offline-critical assets |
| **Network Only** | Only use network | Real-time data, auth tokens |

**Cache First implementation:**

```javascript
// Best for static assets that rarely change
self.addEventListener('fetch', (event) => {
  if (event.request.destination === 'image' ||
      event.request.destination === 'style' ||
      event.request.destination === 'script') {
    event.respondWith(
      caches.match(event.request).then((cached) => {
        if (cached) return cached;  // Cache hit: instant response

        return fetch(event.request).then((response) => {
          // Clone the response (it can only be consumed once)
          const clone = response.clone();
          caches.open('static-v1').then((cache) => {
            cache.put(event.request, clone);
          });
          return response;
        });
      })
    );
  }
});
```

**Stale While Revalidate implementation:**

```javascript
// Best for content that should be fresh but where speed matters
self.addEventListener('fetch', (event) => {
  if (event.request.url.includes('/api/products')) {
    event.respondWith(
      caches.open('api-cache').then((cache) => {
        return cache.match(event.request).then((cached) => {
          // Always fetch fresh data in background
          const fetchPromise = fetch(event.request).then((response) => {
            cache.put(event.request, response.clone());
            return response;
          });

          // Return cached immediately, fresh data updates next load
          return cached || fetchPromise;
        });
      })
    );
  }
});
```

### 11.3 Precaching vs Runtime Caching

```javascript
// PRECACHING: Resources cached during install
// Use for: app shell, critical CSS/JS, offline page
const PRECACHE_URLS = [
  '/',
  '/styles.css',
  '/app.js',
  '/offline.html',
  '/manifest.json',
];

// RUNTIME CACHING: Resources cached on first access
// Use for: images, API responses, third-party resources
self.addEventListener('fetch', (event) => {
  // Only runtime-cache GET requests
  if (event.request.method !== 'GET') return;

  event.respondWith(
    caches.match(event.request).then((cached) => {
      return cached || fetch(event.request).then((response) => {
        // Only cache successful responses
        if (!response || response.status !== 200) return response;

        const clone = response.clone();
        caches.open('runtime-v1').then(cache => {
          cache.put(event.request, clone);
        });
        return response;
      });
    }).catch(() => {
      // Offline fallback for navigation requests
      if (event.request.mode === 'navigate') {
        return caches.match('/offline.html');
      }
    })
  );
});
```

### 11.4 Cache Size Management

```javascript
// Mobile devices have limited storage
// Trim caches to prevent filling up storage

async function trimCache(cacheName, maxItems) {
  const cache = await caches.open(cacheName);
  const keys = await cache.keys();
  if (keys.length > maxItems) {
    // Delete oldest entries (FIFO)
    await cache.delete(keys[0]);
    await trimCache(cacheName, maxItems);  // Recurse until under limit
  }
}

// Call during activate or periodically
self.addEventListener('activate', (event) => {
  event.waitUntil(
    Promise.all([
      trimCache('images', 50),   // Keep max 50 cached images
      trimCache('api-cache', 30) // Keep max 30 API responses
    ])
  );
});
```

| Aspect | Lab Data | Field Data |
|---|---|---|
| Source | Lighthouse, WebPageTest | CrUX (Chrome User Experience Report) |

| Reproducible | Yes | No (varies by user) |

## 14. Responsive-Specific Performance

Responsive design can hurt mobile performance when implemented carelessly. A common mistake is building for desktop and then hiding things on mobile. The hidden content still downloads, parses, and affects performance.

### 14.1 `display: none` Does Not Prevent Downloads

```html
<!-- BAD: Desktop sidebar hidden on mobile, but STILL downloaded -->
<style>
  @media (max-width: 767px) {
    .sidebar { display: none; }
  }
</style>
<div class="sidebar">
  <!-- These images STILL download on mobile -->
  <img src="/ad-banner.jpg" alt="">
  <img src="/recommended-1.jpg" alt="">
  <img src="/recommended-2.jpg" alt="">
  <!-- This iframe STILL loads on mobile -->
  <iframe src="/widget.html"></iframe>
</div>

<!-- display: none hides the element visually and removes it from layout,
     but the browser has already downloaded all its resources -->
```

### 14.2 Conditional Loading with `<picture>` and Media Queries

```html
<!-- Load different images for different viewports -->
```

## 15. Battery and Memory Considerations

Mobile users care about battery life. A website that drains 5% battery in 10 minutes of reading will be abandoned. Battery optimization requires conscious choices about JavaScript execution, network usage, and rendering.

### 15.1 Reducing CPU Wake-Ups

```javascript
// BAD: setInterval polls every second, keeping CPU awake
setInterval(() => {
  fetch('/api/notifications')
    .then(r => r.json())
    .then(data => updateBadge(data.count));
}, 1000);

// GOOD: Use longer intervals and exponential backoff
let pollInterval = 5000;
const MAX_INTERVAL = 60000;

function pollNotifications() {
  fetch('/api/notifications')
    .then(r => r.json())
    .then(data => {
      updateBadge(data.count);
      if (data.count === 0) {
        // No new notifications: poll less frequently
        pollInterval = Math.min(pollInterval * 2, MAX_INTERVAL);
      } else {
        pollInterval = 5000; // Reset when there's activity
      }
      setTimeout(pollNotifications, pollInterval);
    })
    .catch(() => {
      // Network error: back off aggressively
      pollInterval = MAX_INTERVAL;
      setTimeout(pollNotifications, pollInterval);
    });
}

// BEST: Use Server-Sent Events or WebSockets instead of polling
const eventSource = new EventSource('/api/notifications/stream');
eventSource.addEventListener('notification', (event) => {
  const data = JSON.parse(event.data);
  updateBadge(data.count);
});
```

### 15.2 Respecting Background Tabs

```javascript
// Browsers throttle timers in background tabs:
// - setTimeout/setInterval: capped at 1 call per second (background)
// - requestAnimationFrame: paused entirely in background tabs
// - Chrome 133+: may freeze CPU-intensive background tabs entirely

// Use Page Visibility API to stop unnecessary work
document.addEventListener('visibilitychange', () => {
  if (document.hidden) {
    // Tab is hidden: pause animations, stop polling
    pauseAnimations();
    clearInterval(pollingTimer);
    analytics.flush(); // Send pending analytics before freeze
  } else {
    // Tab is visible again: resume
    resumeAnimations();
    pollingTimer = setInterval(poll, pollInterval);
  }
});

// GOTCHA: Do not rely on visibilitychange for critical saves
// The tab might be killed without firing this event on mobile
// Use navigator.sendBeacon() for critical data
window.addEventListener('pagehide', () => {
  navigator.sendBeacon('/api/save-state', JSON.stringify(appState));
});
```

### 15.3 Animation Power Consumption

```css
/* EXPENSIVE: Triggers layout + paint on every frame */
@keyframes pulse-bad {
  0%   { width: 100px; height: 100px; }
  50%  { width: 120px; height: 120px; }
  100% { width: 100px; height: 100px; }
}

/* CHEAP: Only triggers composite (GPU layer, no layout/paint) */

  0%   { transform: scale(1); }
  50%  { transform: scale(1.2); }
  100% { transform: scale(1); }
}

/* VERY CHEAP: opacity is also composite-only */
@keyframes fade {
  0%   { opacity: 0; }
  100% { opacity: 1; }
}

/* Pause animations when user prefers reduced motion */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

### 15.4 Memory Leak Prevention

Memory leaks are especially harmful on mobile because the memory ceiling is low. A leak that takes 5 minutes to cause problems on desktop can crash a mobile tab in 60 seconds.

```javascript
// LEAK: Event listeners on removed elements
function createComponent(container) {
  const handler = () => { /* ... */ };
  window.addEventListener('resize', handler);

  // When the component is removed from DOM,
  // the resize listener keeps a reference to handler,
  // which keeps a reference to the component's closure.
  // This is a memory leak.
}

// FIX: Use AbortController for cleanup
function createComponent(container) {
  const controller = new AbortController();

  window.addEventListener('resize', () => {
    // Component logic here
  }, { signal: controller.signal });

  // When removing the component:
  function destroy() {
    controller.abort();  // Removes ALL listeners registered with this signal
    container.remove();
  }
  return { destroy };
}

// LEAK: IntersectionObserver / MutationObserver not disconnected
const observer = new IntersectionObserver(callback);
observer.observe(element);
// If element is removed but observer is not disconnected, leak

// FIX: Disconnect when done
function cleanup() {
  observer.disconnect();
}

// LEAK: Closures holding large data
function processData(hugeArray) {
  return function getCount() {
    // This closure keeps hugeArray alive in memory forever
    return hugeArray.length;
  };
}

// FIX: Extract what you need, release the reference
function processData(hugeArray) {
  const count = hugeArray.length;  // Extract the value
  return function getCount() {
    return count;  // Only the number is kept, array can be GC'd
  };
}
```

### 15.5 Detecting Memory Issues

```javascript
// Use Performance.measureUserAgentSpecificMemory() to track JS heap
if (performance.measureUserAgentSpecificMemory) {
  // Must be called in a cross-origin isolated context
  setInterval(async () => {
    const result = await performance.measureUserAgentSpecificMemory();
    console.log(`JS heap: ${(result.bytes / 1024 / 1024).toFixed(1)} MB`);

    if (result.bytes > 100 * 1024 * 1024) {
      console.warn('Memory usage exceeds 100 MB - potential leak');
    }

}

```

## 16. Performance Checklist

Use this checklist to audit mobile performance. Items are ordered by typical impact.

### Critical (Do First)

- [ ] Images use modern formats (WebP/AVIF) with fallbacks
- [ ] Images have `width` and `height` attributes (prevents CLS)
- [ ] LCP image has `fetchpriority="high"` and is NOT lazy-loaded
- [ ] Below-fold images use `loading="lazy"`
- [ ] Responsive images use `srcset` and `sizes`
- [ ] CSS is inlined for above-the-fold content (critical CSS)
- [ ] Non-critical CSS loads asynchronously
- [ ] JavaScript uses `defer` or `async` (no parser-blocking scripts)
- [ ] Web fonts use `font-display: swap` or `font-display: optional`
- [ ] Web fonts are in WOFF2 format and preloaded

### Important (Do Next)

- [ ] Third-party origins use `preconnect` or `dns-prefetch`
- [ ] JavaScript is code-split by route (dynamic imports)
- [ ] Unused CSS is removed (PurgeCSS or equivalent)
- [ ] Text assets use Brotli compression (with gzip fallback)
- [ ] Static assets have long cache headers with hashed filenames
- [ ] Long JavaScript tasks are broken into chunks (< 50ms each)
- [ ] Animations use `transform` and `opacity` only (composite)
- [ ] `content-visibility: auto` used on off-screen sections
- [ ] Service worker caches critical assets for repeat visits
- [ ] Performance budgets are set and enforced in CI

### Advanced

- [ ] Fonts are subsetted to required character ranges
- [ ] Variable fonts replace multiple static font files
- [ ] `contain: content` or `contain: layout paint` on independent components
- [ ] Web Workers used for CPU-intensive computations
- [ ] `matchMedia` gates desktop-only JS from loading on mobile
- [ ] Network Information API adapts content for slow connections
- [ ] `requestIdleCallback` schedules non-critical work
- [ ] Page Visibility API pauses background work
- [ ] Memory leaks audited (no orphaned listeners/observers)
- [ ] HTTP/3 enabled on the server/CDN

### Measurement

- [ ] Lighthouse mobile score >= 90
- [ ] LCP <= 2.5 seconds (field data, 75th percentile)
- [ ] INP <= 200ms (field data, 75th percentile)
- [ ] CLS <= 0.1 (field data, 75th percentile)
- [ ] Total page weight < 500 KB compressed
- [ ] JavaScript < 200 KB compressed
- [ ] Web Vitals library deployed to track real-user metrics
- [ ] CrUX data monitored monthly
