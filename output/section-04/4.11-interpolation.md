# 4.11 Interpolation (`#{}`)

> Interpolation embeds the result of a SASS expression into CSS as an unquoted string. It is required for dynamic selectors, property names, and at-rule names, but should never be used where a plain variable works.

---

## What Interpolation Does

The `#{}` syntax evaluates a SassScript expression at compile time, converts the result to an unquoted string, and inserts it into the surrounding context.

```scss
$prefix: "app";
$size: 16px;

.#{$prefix}-header {
  font-size: $size;
}
```

Compiles to:

```css
.app-header {
  font-size: 16px;
}
```

The critical fact: interpolation always produces an **unquoted string**. It strips type information, unit awareness, and quotation marks from the value. This is both its power and its primary source of bugs.

---

## Where Interpolation Is Required

Interpolation is needed in contexts where SASS cannot evaluate a variable directly.

| Context | Example |
|---------|---------|
| Selectors | `.icon-#{$name} { }` |
| Property names | `#{$prop}: value;` |
| Custom property values | `--color: #{$var};` |
| At-rule names | `@#{$layer} { }` |
| At-rule dynamic strings | `@media #{$query} { }` |
| `@extend` targets | `@extend %#{$placeholder};` |
| Inside quoted strings | `"path/#{$file}.png"` |
| Unquoted string construction | `icon-#{$name}` |
| CSS `@import` URLs | `@import #{$url};` |
| Comments | `/* Version: #{$ver} */` |

### Where Interpolation Is NOT Needed

Plain variables work directly in property values. Adding `#{}` there is wasteful and can cause bugs.

```scss
$color: #ff0000;
$size: 16px;

// BAD: unnecessary interpolation
.element {
  color: #{$color};
  font-size: #{$size};
  margin: #{$size} auto;
}

// GOOD: plain variables
.element {
  color: $color;
  font-size: $size;
  margin: $size auto;
}
```

Both produce identical CSS, but the plain variable version preserves type information, allowing SASS to perform arithmetic and type checking.

## Where Interpolation Cannot Be Used

### Dynamic Variable Names

You cannot construct a variable name dynamically. `$margin-#{$side}` is a syntax error.

```scss
$side: "left";

// SYNTAX ERROR
// $margin-#{$side}: 20px;

// WORKAROUND: use a map
$margins: (
  "left": 20px,
  "right": 10px,
  "top": 0,
  "bottom": 16px,
);

.element {
  margin-left: map-get($margins, $side);
}
```

### Dynamic Mixin Names

You cannot call a mixin by interpolated name.

```scss
$theme: "dark";

// SYNTAX ERROR
// @include apply-#{$theme}-theme;

// WORKAROUND: use @if or a parameterized mixin
@mixin apply-theme($name) {
  @if $name == "dark" {
    background: #333;
    color: #fff;
  } @else if $name == "light" {
    background: #fff;
    color: #333;
  } @else {
    @error "Unknown theme: #{$name}";
  }
}
```

### Dynamic Sass Function Calls

Interpolation cannot call Sass functions dynamically. It does work for CSS function names, but not SassScript functions.

```scss
// Works: CSS function name
$fn: "rotate";
transform: #{$fn}(45deg);
// Output: transform: rotate(45deg);

// Does NOT work: calling a Sass function dynamically
// $result: #{$fn-name}($arg); // Not a valid Sass function call
```

For dynamic Sass function calls, use `meta.call()`:

```scss
@use "sass:meta";

$fn: meta.get-function("darken");
$result: meta.call($fn, blue, 10%);
```

## Interpolation in Selectors

This is the most common use of interpolation. Selectors cannot contain bare variables.

### Dynamic Class Names

```scss
$component: "card";

.#{$component} {
  padding: 16px;

  &__header {
    font-weight: bold;
  }

  &__body {
    line-height: 1.5;
  }

  &--highlighted {
    border: 2px solid gold;
  }
}
```

### Generating BEM Modifiers from Maps

```scss
$variants: (
  "primary": #007bff,
  "secondary": #6c757d,
  "danger": #dc3545,
);

.btn {
  padding: 8px 16px;
  border: none;
  cursor: pointer;

  @each $name, $color in $variants {
    &--#{$name} {
      background-color: $color;
      color: #fff;

      &:hover {
        background-color: darken($color, 10%);
      }
    }
  }
}
```

### Utility Class Generation

```scss
@each $side in (top, right, bottom, left) {
  @for $i from 0 through 5 {
    .m-#{$side}-#{$i} {
      margin-#{$side}: $i * 8px;
    }
  }
}
```

### Dynamic Pseudo-Classes

```scss
$state: "hover";

.link {
  &:#{$state} {
    text-decoration: underline;
  }
}
```

## Interpolation in Property Names

Variables cannot appear as property names without interpolation.

### Dynamic Property Mixin

```scss
@mixin spacing($type, $side, $value) {
  #{$type}-#{$side}: $value;
}

.box {
  @include spacing(margin, top, 20px);    // margin-top: 20px;
  @include spacing(padding, left, 10px);  // padding-left: 10px;
}
```

### Applying Properties from a Map

```scss
$declarations: (
  "font-size": 16px,
  "line-height": 1.5,
  "letter-spacing": 0.02em,
  "color": #333,
);

.text {
  @each $property, $value in $declarations {
    #{$property}: $value;
  }
}
```

### Vendor Prefixes (Legacy Pattern)

```scss
@mixin prefix($property, $value) {
  -webkit-#{$property}: $value;
  -moz-#{$property}: $value;
  #{$property}: $value;
}

.element {
  @include prefix(transform, rotate(45deg));
}
```

Note: Autoprefixer is the modern solution for vendor prefixes. This pattern is shown for understanding interpolation, not as a recommended approach to prefixing.

### Logical Properties

```scss
@mixin logical-margin($direction, $value) {
  margin-#{$direction}: $value;
}

.element {
  @include logical-margin(inline-start, 16px);
  @include logical-margin(block-end, 24px);
}
```

## Interpolation in At-Rules

### `@keyframes` with Dynamic Names

```scss
@mixin make-animation($name, $from, $to) {
  @keyframes #{$name} {
    from { opacity: $from; }
    to { opacity: $to; }
  }
}

@include make-animation(fade-in, 0, 1);
@include make-animation(fade-out, 1, 0);
```

### Unique Animation Names with `unique-id()`

```scss
@mixin inline-animation($duration) {
  $name: anim-#{unique-id()};

  @keyframes #{$name} {
    @content;
  }

  animation-name: $name;
  animation-duration: $duration;
}

.pulse {
  @include inline-animation(2s) {
    from { transform: scale(1); }
    to { transform: scale(1.05); }
  }
}
```

### `@media` with Dynamic Queries

Simple feature queries do not need interpolation because SASS evaluates variables inside parentheses:

```scss
$breakpoint: 768px;

// No interpolation needed here
@media (min-width: $breakpoint) {
  .container { max-width: 720px; }
}
```

But full dynamic query strings require it:

```scss
$query: "(min-width: 768px) and (max-width: 1024px)";

@media #{$query} {
  .tablet-only { display: block; }
}
```

### `@layer` and Other At-Rule Names

Dart Sass 1.15.0+ supports interpolation in at-rule names:

```scss
$layer: "components";

@layer #{$layer} {
  .btn { padding: 8px 16px; }
}
```

## Interpolation in Strings

Inside quoted strings, `#{}` works like template literals in JavaScript:

```scss
$name: "card";
$version: 3;

$message: "Component: #{$name}, version #{$version}";
// Result: "Component: card, version 3"
```

### URL Construction

```scss
$base-path: "/assets/images";
$filename: "hero";
$ext: "webp";

.hero {
  background-image: url("#{$base-path}/#{$filename}.#{$ext}");
  // Output: background-image: url("/assets/images/hero.webp");
}
```

### Quote Stripping Behavior

Interpolation strips quotes from quoted strings when embedding them. This is important to understand:

```scss
$font: "Helvetica Neue";

// Inside another quoted string, the inner quotes are removed
.element {
  content: "Font is #{$font}";
  // Output: content: "Font is Helvetica Neue";
  // The variable's quotes are stripped; the outer quotes remain
}

// As a standalone value, the quotes are also stripped
$unquoted: #{$font};
// $unquoted is: Helvetica Neue (unquoted string)
```

This matters for font stacks:

```scss
$family: "Open Sans";

// BAD: interpolation strips the quotes
.bad {
  font-family: #{$family}, sans-serif;
  // Output: font-family: Open Sans, sans-serif;
  // "Open Sans" without quotes is INVALID CSS
}

// GOOD: plain variable preserves quotes
.good {
  font-family: $family, sans-serif;
  // Output: font-family: "Open Sans", sans-serif;
}
```

## Interpolation and Custom Properties (CSS Variables)

Custom property values are a special case. SASS requires interpolation to inject variables into custom properties because SASS does not parse custom property values as SassScript.

```scss
$primary: #007bff;
$spacing: 16px;

:root {
  // REQUIRED: interpolation for Sass variables in custom properties
  --color-primary: #{$primary};
  --spacing-md: #{$spacing};
}
```

Without interpolation, SASS outputs the literal text `$primary` instead of the value.

### The `meta.inspect()` Problem

Interpolation strips quotes, which breaks quoted values in custom properties:

```scss
$font-stack: "Segoe UI", Roboto, sans-serif;

:root {
  // BAD: interpolation strips quotes from "Segoe UI"
  --font-family: #{$font-stack};
  // Output: --font-family: Segoe UI, Roboto, sans-serif;
  // "Segoe UI" without quotes is invalid

  // GOOD: meta.inspect() preserves quotes
  --font-family: #{meta.inspect($font-stack)};
  // Output: --font-family: "Segoe UI", Roboto, sans-serif;
}
```

Use `meta.inspect()` when the value being interpolated into a custom property contains quoted strings that must remain quoted.

```scss
@use "sass:meta";

@mixin custom-prop($name, $value) {
  --#{$name}: #{meta.inspect($value)};
}

:root {
  @include custom-prop("font-family", ("Georgia", serif));
  // Output: --font-family: "Georgia", serif;
}
```

## Interpolation and `calc()`

Modern Dart Sass parses `calc()` natively, so you can use variables directly without interpolation.

```scss
$sidebar: 250px;
$gap: 20px;

// PREFERRED: direct variable usage
.content {
  width: calc(100% - $sidebar - $gap);
}

// LEGACY: interpolation (still works but unnecessary)
.content-legacy {
  width: calc(100% - #{$sidebar} - #{$gap});
}
```

Direct usage is better because SASS can type-check and simplify the expression. Interpolation bypasses this, outputting the expression verbatim.

### `#{$var}%` Is No Longer Valid in Calculations

Recent Dart Sass versions reject interpolation followed by a unit inside calculation functions:

```scss
$pct: 50;

// BAD: will error in recent Dart Sass
// width: calc(#{$pct}%);

// GOOD: multiply to add units
width: calc($pct * 1%);

// BETTER: define with units from the start
$pct: 50%;
width: calc($pct);
```

## Interpolation and the `/` Operator

The `/` operator for division is deprecated in Sass (removed in Dart Sass 2.0). Interpolation was historically used as a workaround to output a literal slash.

```scss
$font-size: 16px;
$line-height: 1.5;

// OLD workaround: interpolation to prevent division
.text {
  font: #{$font-size}/#{$line-height} sans-serif;
  // Output: font: 16px/1.5 sans-serif;
}

// MODERN: use list.slash()
@use "sass:list";

.text-modern {
  font: list.slash($font-size, $line-height) sans-serif;
}

// Grid shorthand
$col-start: 1;
$col-end: 3;

.grid-item {
  grid-column: list.slash($col-start, $col-end);
  // Output: grid-column: 1 / 3;
}
```

## How Interpolation Destroys Type Information

This is the most important gotcha. Interpolation converts any value to an unquoted string, losing all type metadata.

```scss
$width: 100px;

// $width is a number with unit px
@debug meta.type-of($width);      // "number"

// After interpolation, it is an unquoted string
@debug meta.type-of(#{$width});   // "string"
```

### Arithmetic After Interpolation Fails

```scss
$base: 10px;

// WORKS: arithmetic on numbers
$result: $base + 5px;   // 15px

// BROKEN: concatenation on strings
$broken: #{$base} + 5px;
// Result: the string "10px + 5px" (not 15px)
```

### Adding Units via Interpolation Creates Strings, Not Numbers

```scss
$raw: 16;

// BAD: creates the string "16px", not the number 16px
$bad: #{$raw}px;
// $bad + 4px would fail or concatenate to "16px4px"

// GOOD: arithmetic to attach units
$good: $raw * 1px;   // 16px as a number
// $good + 4px = 20px
```

### Comparisons Fail on Interpolated Values

```scss
$value: #{100px};

// ERROR: cannot compare string "100px" to number 50px
// @if $value > 50px { ... }

// The original number works fine
$value: 100px;
@if $value > 50px { ... }   // true
```

## `string.unquote()` vs Interpolation for Unquoting

Both strip quotes, but `string.unquote()` is more explicit about intent:

```scss
@use "sass:string";

$quoted: "hello";

// Both produce the unquoted string: hello
$via-interpolation: #{$quoted};
$via-unquote: string.unquote($quoted);

// Prefer string.unquote() when the goal is specifically to remove quotes
// Prefer #{} when embedding into a larger context (selectors, strings, etc.)
```

## Common Mistakes

- **Using interpolation in property values where a plain variable works.** `color: #{$color}` is equivalent to `color: $color` but loses type safety. Always use plain variables in values.

- **Attaching units via interpolation.** `#{$n}px` creates a string, not a number. Use `$n * 1px` or define variables with units from the start.

- **Performing arithmetic on interpolated values.** `#{$a} + #{$b}` concatenates strings instead of adding numbers. Keep values as numbers and interpolate only at the final output point.

- **Forgetting that interpolation strips quotes.** `font-family: #{$font}` outputs `Open Sans` without quotes when `$font` is `"Open Sans"`. Use the plain variable or `meta.inspect()` for custom properties.

- **Using `#{$var}%` inside `calc()`.** Recent Dart Sass rejects this pattern. Multiply to add units: `$var * 1%`.

- **Trying to create dynamic variable or mixin names.** SASS does not support `$margin-#{$side}` or `@include #{$name}`. Use maps for dynamic values and `@if` chains or `meta.call()` for dynamic dispatch.

- **Interpolating inside `calc()` unnecessarily.** Modern Dart Sass supports direct variable usage in `calc()`. Prefer `calc(100% - $var)` over `calc(100% - #{$var})`.

---

## Best Practices

- **Use interpolation only where required.** Selectors, property names, custom property values, at-rule names, and string embedding. Never in plain property values.

- **Keep interpolation expressions simple.** If the expression inside `#{}` is complex, extract it to a variable first.

```scss
// Hard to read
.#{str-slice($component, 1, 3)}-#{$variant}--#{$state} { }

// Better: extract to a variable
$selector: "#{str-slice($component, 1, 3)}-#{$variant}--#{$state}";
.#{$selector} { }
```

- **Use `meta.inspect()` for custom properties containing quoted strings.** This preserves the quotes that interpolation would strip.

- **Use `math.div()` for division and `list.slash()` for slash-separated values.** Do not use interpolation as a workaround for the deprecated `/` operator.

- **Document generated selectors.** When dynamically generating class names with interpolation and loops, add a comment listing the generated names so developers can search for them.

```scss
// Generates: .btn--primary, .btn--secondary, .btn--danger
@each $name, $color in $variants {
  .btn--#{$name} { ... }
}
```

- **Use maps instead of dynamic variable names.** This is cleaner, type-safe, and works with all SASS features.

- **Prefer direct variables in `calc()`.** Modern Dart Sass handles this natively with full type-checking and simplification.

## Real-World Use Cases

### 1. Component Style Generator

```scss
@use "sass:map";

$components: (
  "card": (
    "padding": 16px,
    "border-radius": 8px,
    "background": #fff,
    "shadow": 0 2px 8px rgba(0, 0, 0, 0.1),
  ),
  "badge": (
    "padding": 4px 8px,
    "border-radius": 12px,
    "background": #e0e0e0,
    "shadow": none,
  ),
  "tag": (
    "padding": 2px 6px,
    "border-radius": 4px,
    "background": #f5f5f5,
    "shadow": none,
  ),
);

@each $name, $props in $components {
  .#{$name} {
    @each $prop, $value in $props {
      #{$prop}: $value;
    }
  }
}
```

### 2. Design Tokens to CSS Custom Properties

```scss
@use "sass:map";
@use "sass:meta";

$tokens: (
  "color-primary": #1976d2,
  "color-danger": #d32f2f,
  "color-success": #388e3c,
  "spacing-xs": 4px,
  "spacing-sm": 8px,
  "spacing-md": 16px,
  "spacing-lg": 24px,
  "font-family-sans": ("Inter", sans-serif),
  "font-family-mono": ("Fira Code", monospace),
  "border-radius-sm": 2px,
  "border-radius-md": 4px,
  "border-radius-lg": 8px,
);

:root {
  @each $name, $value in $tokens {
    // Use meta.inspect() to preserve quotes in font family lists
    @if type-of($value) == "list" {
      --#{$name}: #{meta.inspect($value)};
    } @else {
      --#{$name}: #{$value};
    }
  }
}

// Output:
// :root {
//   --color-primary: #1976d2;
//   --color-danger: #d32f2f;
//   ...
//   --font-family-sans: "Inter", sans-serif;
//   --font-family-mono: "Fira Code", monospace;
//   --border-radius-sm: 2px;
//   ...
// }
```

### 3. Responsive Utility Framework

```scss
@use "sass:map";

$breakpoints: ("": null, "sm": 576px, "md": 768px, "lg": 992px);

$utilities: (
  "d": (
    "property": "display",
    "values": (none, block, flex, inline-flex, grid),
  ),
  "text": (
    "property": "text-align",
    "values": (left, center, right),
  ),
);

@each $abbrev, $config in $utilities {
  $property: map.get($config, "property");
  $values: map.get($config, "values");

  @each $bp-name, $bp-value in $breakpoints {
    $prefix: if($bp-name == "", "", "#{$bp-name}\\:");

    @if $bp-value {
      @media (min-width: $bp-value) {
        @each $value in $values {
          // Generates: .sm\:d-flex, .md\:text-center, etc.
          .#{$prefix}#{$abbrev}-#{$value} {
            #{$property}: $value;
          }
        }
      }
    } @else {
      @each $value in $values {
        // Generates: .d-flex, .text-center, etc.
        .#{$abbrev}-#{$value} {
          #{$property}: $value;
        }
      }
    }
  }
}
```

---

> **Key Takeaway:** Use `#{}` only where SASS cannot accept a plain variable: selectors, property names, custom properties, at-rule names, and inside strings. Never use it in property values where `$var` works directly, because interpolation destroys type information, prevents arithmetic, and strips quotes.
