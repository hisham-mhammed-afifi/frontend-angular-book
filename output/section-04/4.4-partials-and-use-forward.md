# 4.4 Partials and `@use` / `@forward` (Modern Module System)

> The Sass module system replaces the deprecated `@import` with `@use` and `@forward`, giving you namespaced access, singleton evaluation, and explicit dependency management. This is the foundation of maintainable Sass architecture.

---

## Partials

Partials are Sass files prefixed with an underscore (`_`). The underscore tells Sass that the file should not be compiled into a standalone CSS file. It exists only to be loaded by other files.

```
styles/
  _variables.scss      # Partial: not compiled alone
  _mixins.scss         # Partial: not compiled alone
  main.scss            # Entrypoint: compiled to main.css
```

When loading a partial, omit the underscore and the extension:

```scss
// Loads _variables.scss
@use 'variables';
```

### File Resolution Algorithm

When Sass encounters `@use 'foo'`, it searches for files in this order within each directory:

1. `foo.scss`, `foo.sass`, `foo.css`
2. `_foo.scss`, `_foo.sass`, `_foo.css`
3. `foo/_index.scss`, `foo/_index.sass`, `foo/_index.css`
4. `foo/index.scss`, `foo/index.sass`, `foo/index.css`

Sass checks the current file's directory first, then any configured load paths.

Never have both `_foo.scss` and `foo.scss` in the same directory. Sass will either error or produce unpredictable results.

## `@use`: Loading Modules

`@use` loads a Sass file as a module and makes its members (variables, mixins, functions) available under a namespace.

### Default Namespace

The namespace defaults to the last component of the URL:

```scss
@use 'abstracts/variables';
// Namespace: "variables"
// Access: variables.$primary, variables.$spacing
```

```scss
@use 'utils/breakpoints';
// Namespace: "breakpoints"
// Access: @include breakpoints.respond-to('md')
```

### Custom Namespace with `as`

```scss
@use 'abstracts/variables' as vars;
// Access: vars.$primary, vars.$spacing

@use 'utils/breakpoints' as bp;
// Access: @include bp.respond-to('md')
```

Short, meaningful aliases keep code readable when you reference modules frequently.

### Wildcard with `as *`

```scss
@use 'abstracts/variables' as *;
// Access: $primary, $spacing (no namespace prefix)
```

The wildcard dumps all public members into the current scope. Use this only for your own files where you control the names. For third-party libraries, wildcards risk name collisions when the library adds new members.

### Accessing Members

Variables, functions, and mixins each use namespace syntax:

```scss
@use 'config';
@use 'utils';
@use 'mixins';

.card {
  max-width: config.$max-width;             // Variable
  padding: utils.spacing(2);                 // Function
  @include mixins.respond-to('tablet') {     // Mixin
    max-width: 100%;
  }
}
```

### Placement Rules

`@use` rules must appear at the top of a file, before any style rules, selectors, or other statements. Only `@forward` rules and other `@use` rules may appear before them.

```scss
// CORRECT: @use at the top
@use 'variables' as vars;
@use 'mixins';

.button {
  color: vars.$primary;
}
```

```scss
// ERROR: @use cannot appear after style rules
.button {
  color: red;
}

@use 'variables';  // Sass throws an error here
```

## Module Configuration with `@use ... with()`

Modules can expose configurable variables using `!default`. Consumers override them with `with()`:

```scss
// _config.scss (library module)
$primary: #3498db !default;
$border-radius: 4px !default;
$font-size-base: 1rem !default;
$enable-shadows: true !default;
```

```scss
// consumer.scss
@use 'config' with (
  $primary: #e74c3c,
  $border-radius: 8px,
);

// config.$primary is now #e74c3c
// config.$border-radius is now 8px
// config.$font-size-base stays 1rem (not overridden)
// config.$enable-shadows stays true (not overridden)
```

Only variables marked with `!default` can be configured. Attempting to configure a variable without `!default` is a compile error.

### Configuration Happens Once

A module can only be configured the first time it is loaded. If two files try to configure the same module differently, the second one throws an error:

```scss
// a.scss
@use 'config' with ($primary: red);    // Works: first load

// b.scss
@use 'config' with ($primary: blue);   // ERROR: already loaded/configured

// c.scss
@use 'config';                          // Works: no configuration, gets whatever was set first
```

Configure shared libraries at the top of your main entrypoint file, before anything else loads them.

## Singleton Evaluation

This is the most important behavioral difference between `@use` and the deprecated `@import`.

`@use` evaluates a module exactly once. Every file that `@use`s the same module gets a reference to the same singleton instance. CSS output from the module appears only once in the compiled stylesheet.

```scss
// _reset.scss
*,
*::before,
*::after {
  box-sizing: border-box;
  margin: 0;
  padding: 0;
}
```

```scss
// _header.scss
@use 'reset';  // reset.scss is loaded

// _footer.scss
@use 'reset';  // Same module; NOT loaded again; no duplicate CSS
```

With the old `@import`, every `@import 'reset'` would re-emit the CSS, producing duplicates. With `@use`, the reset CSS appears exactly once in the output, regardless of how many files load it.

## `@forward`: Re-exporting Modules

`@forward` makes a module's public members available to anyone who `@use`s the forwarding file. It does NOT make them available in the current file.

```scss
// abstracts/_index.scss
@forward 'variables';
@forward 'mixins';
@forward 'functions';
```

```scss
// component.scss
@use 'abstracts';

.card {
  color: abstracts.$primary;                   // From _variables.scss
  @include abstracts.respond-to('tablet') {    // From _mixins.scss
    width: 100%;
  }
  padding: abstracts.spacing(2);               // From _functions.scss
}
```

The consumer sees a unified `abstracts` namespace. They do not need to know that variables, mixins, and functions live in separate files.

### `@forward` Does NOT Give Local Access

A critical gotcha: `@forward` only re-exports. If the file that contains the `@forward` also needs to use the forwarded members, it must separately `@use` them:

```scss
// abstracts/_index.scss
@forward 'variables';
@forward 'mixins';

// WRONG: Cannot use variables.$primary here
// $primary is forwarded to consumers, not available locally

// If you need local access:
@use 'variables';
// Now you can use variables.$primary in this file
```

When a file has both `@forward` and `@use` for the same module, always put `@forward` first. This ensures that downstream configuration via `@use ... with()` is applied before your local `@use` loads the module.

### Adding Prefixes

Prefixes prevent name collisions when forwarding multiple modules:

```scss
// _index.scss
@forward 'colors' as color-*;
@forward 'spacing' as space-*;
@forward 'typography' as type-*;
```

```scss
// consumer.scss
@use 'tokens';

.card {
  color: tokens.$color-primary;        // Was $primary in _colors.scss
  padding: tokens.$space-md;           // Was $md in _spacing.scss
  font-size: tokens.$type-base-size;   // Was $base-size in _typography.scss
}
```

The prefix is prepended to all forwarded member names: variables, functions, and mixins.

### Controlling Visibility with `show` and `hide`

Forward only specific members:

```scss
@forward 'utils' show spacing, $max-width, respond-to;
```

Forward everything except specific members:

```scss
@forward 'internal' hide $-debug-mode, -log-warning;
```

Member names include the `$` for variables. A single `@forward` rule can use either `show` or `hide`, but not both.

### Configuring Through `@forward`

`@forward` can set default values for the forwarded module, creating a configuration layer:

```scss
// _opinionated.scss
@forward 'base-library' with (
  $primary: #222222 !default,
  $border-radius: 0 !default,
);
```

```scss
// consumer.scss
@use 'opinionated' with (
  $primary: #e74c3c,
  // $border-radius uses the opinionated default: 0
);
```

The `!default` in `@forward ... with()` is critical. Without it, the value is locked and downstream consumers cannot override it.

## Index Files (Barrel Pattern)

The `_index.scss` convention lets you load an entire folder with a single `@use`:

```scss
// @use 'abstracts' loads abstracts/_index.scss
@use 'abstracts';
```

The index file typically contains only `@forward` rules:

```scss
// abstracts/_index.scss
@forward 'variables';
@forward 'mixins';
@forward 'functions';
```

### Full Project Structure

```
src/
  styles/
    abstracts/
      _variables.scss
      _mixins.scss
      _functions.scss
      _index.scss
    base/
      _reset.scss
      _typography.scss
      _index.scss
    components/
      _button.scss
      _card.scss
      _modal.scss
      _index.scss
    layout/
      _header.scss
      _footer.scss
      _grid.scss
      _index.scss
    main.scss
```

Each `_index.scss`:

```scss
// base/_index.scss
@forward 'reset';
@forward 'typography';

// components/_index.scss
@forward 'button';
@forward 'card';
@forward 'modal';

// layout/_index.scss
@forward 'header';
@forward 'footer';
@forward 'grid';
```

The main entrypoint:

```scss
// main.scss
@use 'base';
@use 'layout';
@use 'components';
```

### Every File Declares Its Own Dependencies

This is the fundamental shift from `@import`. Under `@import`, loading variables in `main.scss` made them globally available. Under `@use`, every file is self-contained:

```scss
// components/_button.scss
@use '../abstracts' as abs;

.button {
  color: abs.$primary;
  border-radius: abs.$border-radius;
  @include abs.respond-to('tablet') {
    width: 100%;
  }
}
```

```scss
// components/_card.scss
@use '../abstracts' as abs;

.card {
  padding: abs.$spacing-md;
  border: 1px solid abs.$border-color;
}
```

Each component file explicitly declares that it depends on `abstracts`. This makes dependencies visible, auditable, and refactorable.

## Private Members

Members whose names start with `-` or `_` are private to their module:

```scss
// _colors.scss
$primary: #3498db;          // Public
$secondary: #2ecc71;        // Public
$-mix-amount: 15%;          // Private
$_internal-base: #f0f0f0;   // Private (same as $-internal-base)

$primary-light: color.scale($primary, $lightness: $-mix-amount);
```

```scss
// consumer.scss
@use 'colors';

.box {
  color: colors.$primary;         // Works
  // color: colors.$-mix-amount;  // ERROR: private member
}
```

Private members are also excluded from `@forward`. They never leak into the public API.

Note: The underscore in `_colors.scss` (partial naming) and the underscore in `$_internal` (private member) are completely separate concepts. The file prefix prevents standalone compilation. The member prefix prevents external access.

## Built-in Modules

Sass ships built-in modules that must be explicitly loaded. They use the `sass:` URL scheme.

### `sass:math`

```scss
@use 'sass:math';

.container {
  width: math.div(100%, 3);           // 33.3333% (replaces / for division)
  padding: math.max(1rem, 16px);
  border-radius: math.clamp(4px, 2vw, 16px);
}

$is-unitless: math.is-unitless(42);    // true
$unit: math.unit(42px);               // "px"
```

### `sass:color`

```scss
@use 'sass:color';

$brand: #3498db;
$brand-light: color.scale($brand, $lightness: 30%);
$brand-dark: color.adjust($brand, $lightness: -20%);
$brand-muted: color.change($brand, $saturation: 20%);
$mixed: color.mix($brand, white, 50%);
```

The global functions `darken()`, `lighten()`, `saturate()`, and `desaturate()` are deprecated. Use `color.adjust()` or `color.scale()` instead.

### `sass:map`

```scss
@use 'sass:map';

$themes: (
  light: (bg: #fff, text: #333),
  dark: (bg: #1a1a1a, text: #f0f0f0),
);

$light-bg: map.get($themes, light, bg);  // #fff (deep access)
$merged: map.merge($themes, (high-contrast: (bg: #000, text: #fff)));
$has-dark: map.has-key($themes, dark);    // true
```

### `sass:list`

```scss
@use 'sass:list';

$sizes: 12px 14px 16px 18px 24px;
$first: list.nth($sizes, 1);           // 12px
$count: list.length($sizes);           // 5
$appended: list.append($sizes, 32px);  // 12px 14px 16px 18px 24px 32px
$idx: list.index($sizes, 16px);        // 3
```

### `sass:string`

```scss
@use 'sass:string';

$name: 'Hello World';
$upper: string.to-upper-case($name);   // "HELLO WORLD"
$slice: string.slice($name, 7);        // "World"
$len: string.length($name);            // 11
$parts: string.split($name, ' ');      // ("Hello", "World")
```

### `sass:meta`

```scss
@use 'sass:meta';

// Get a function reference
$fn: meta.get-function('darken', $module: 'color');
$result: meta.call($fn, #3498db, 10%);

// Check types
$type: meta.type-of(42px);   // "number"
$type: meta.type-of(red);    // "color"
$type: meta.type-of("hi");   // "string"

// Inspect values (preserves quotes, useful for debugging)
$debug: meta.inspect("hello");  // '"hello"'
```

`meta.load-css()` dynamically loads a module's CSS as a mixin. The URL can be a variable, enabling conditional loading:

```scss
@use 'sass:meta';

@mixin load-theme($name) {
  @include meta.load-css('themes/#{$name}');
}

.app {
  @include load-theme('dark');
}
```

### `sass:selector`

```scss
@use 'sass:selector';

$unified: selector.unify('.card', '.featured');  // .card.featured
$nested: selector.nest('.parent', '.child');      // .parent .child
```

## `@use` vs `@import`: The Complete Comparison

| Aspect | `@import` (deprecated) | `@use` (current) |
|---|---|---|
| Scope | Global: everything in one flat namespace | Local: members scoped to the loading file |
| Namespacing | None | Automatic, based on filename |
| Multiple loads | Re-executes and re-emits CSS every time | Evaluates once, singleton |
| CSS duplication | Same file imported 5 times = 5x CSS | Same file used 5 times = 1x CSS |
| Private members | Not supported | Members starting with `-` or `_` are private |
| Configuration | Reassign variable after import | Explicit `with()` syntax |
| Built-in functions | Available globally (`darken()`, `map-get()`) | Require explicit module load (`color.adjust()`, `map.get()`) |
| Nesting | Can be nested inside rules | Must be at top of file |
| Deprecation | Deprecated (Dart Sass 1.80.0) | Recommended approach |

## Circular Dependencies

Sass does not allow circular dependencies. If module A `@use`s module B, and module B `@use`s module A, Sass throws an error:

```
Error: Module loop: this module is already being loaded.
```

Prevent this by structuring dependencies as a directed acyclic graph (DAG):

```
main.scss
  -> layout/_index.scss
       -> abstracts/_index.scss (leaf node: no dependencies)
  -> components/_index.scss
       -> abstracts/_index.scss (same singleton, no conflict)
```

Variables, mixins, and functions should live in "leaf node" modules that depend on nothing else. Components and layouts depend on abstracts, never the reverse.

## Common Mistakes

### Mistake 1: Expecting `@forward` to give local access

```scss
// _index.scss
@forward 'variables';

// WRONG: $primary is NOT available here
.debug { color: $primary; }

// FIX: Add @use for local access (after @forward)
@forward 'variables';
@use 'variables';

.debug { color: variables.$primary; }
```

### Mistake 2: Configuring a module that was already loaded

```scss
// a.scss (loaded first)
@use 'config';

// b.scss (loaded second)
@use 'config' with ($primary: red);  // ERROR: already loaded
```

Configure shared modules in your main entrypoint, before any other file loads them.

### Mistake 3: Using `as *` for everything

```scss
// BAD: No idea where anything comes from
@use 'colors' as *;
@use 'spacing' as *;
@use 'typography' as *;
@use 'breakpoints' as *;

.card {
  color: $primary;      // Which module?
  padding: $md;         // Which module?
  font-size: $base;     // Which module?
}
```

```scss
// GOOD: Clear namespaces
@use 'colors' as c;
@use 'spacing' as sp;
@use 'typography' as type;

.card {
  color: c.$primary;
  padding: sp.$md;
  font-size: type.$base;
}
```

### Mistake 4: Forgetting that every file must declare its dependencies

Under `@import`, loading variables in `main.scss` made them globally available. Under `@use`, each file is isolated:

```scss
// BAD: Assuming globals exist
// components/_card.scss
.card {
  color: $primary;  // ERROR: $primary is not defined in this file
}
```

```scss
// GOOD: Explicit dependency declaration
// components/_card.scss
@use '../abstracts' as abs;

.card {
  color: abs.$primary;
}
```

### Mistake 5: Using deprecated global built-in functions

```scss
// BAD: Global functions are deprecated
$light: lighten(#3498db, 20%);
$width: 100% / 3;
$key: map-get($map, primary);
```

```scss
// GOOD: Module-namespaced functions
@use 'sass:color';
@use 'sass:math';
@use 'sass:map';

$light: color.adjust(#3498db, $lightness: 20%);
$width: math.div(100%, 3);
$key: map.get($map, primary);
```

### Mistake 6: Wrong order of `@forward` and `@use`

```scss
// BAD: @use before @forward
@use 'variables';
@forward 'variables';  // Configuration from downstream won't work properly

// GOOD: @forward first
@forward 'variables';
@use 'variables';
```

## Best Practices

### Practice 1: Use short, meaningful namespace aliases

```scss
@use 'abstracts/variables' as vars;
@use 'abstracts/mixins' as mix;
@use 'abstracts/functions' as fn;
@use 'sass:math';
@use 'sass:color';
```

Or load the folder-level index for a single namespace:

```scss
@use 'abstracts' as abs;
```

### Practice 2: Create `_index.scss` files for every folder

```scss
// components/_index.scss
@forward 'button';
@forward 'card';
@forward 'modal';
@forward 'form-field';
```

This lets consumers load the entire folder with `@use 'components'`.

### Practice 3: Mark implementation details as private

```scss
// _spacing.scss
$_base-unit: 8px;  // Private: consumers should not depend on this

$sm: $_base-unit;          // Public
$md: $_base-unit * 2;      // Public
$lg: $_base-unit * 3;      // Public
$xl: $_base-unit * 4;      // Public
```

### Practice 4: Use `!default` for all configurable values

```scss
// _config.scss
$primary: #3498db !default;
$secondary: #2ecc71 !default;
$border-radius: 4px !default;
$font-family: 'Inter', sans-serif !default;
$enable-shadows: true !default;
$enable-transitions: true !default;
```

### Practice 5: Configure libraries at the entrypoint

```scss
// main.scss (entrypoint)
@use 'third-party-lib' with (
  $primary: #e74c3c,
  $font-size-base: 18px,
);

// All subsequent @use of 'third-party-lib' get this configuration
@use 'components';
@use 'layout';
```

### Practice 6: Use prefixes when forwarding multiple modules from one index

```scss
// tokens/_index.scss
@forward 'colors' as color-*;
@forward 'spacing' as space-*;
@forward 'typography' as type-*;
@forward 'breakpoints' as bp-*;
```

This prevents collisions if multiple source modules export members with the same name (e.g., `$base`).

## Real-World Use Cases

### Use Case 1: Design System Token Architecture

```scss
// tokens/_colors.scss
$primary: #3498db !default;
$secondary: #2ecc71 !default;
$danger: #e74c3c !default;
$warning: #f39c12 !default;
$info: #1abc9c !default;
$-mix-ratio: 12%;  // Private

$primary-light: color.scale($primary, $lightness: 30%);
$primary-dark: color.scale($primary, $lightness: -30%);
```

```scss
// tokens/_spacing.scss
$_unit: 8px !default;  // Private base

$xs: $_unit * 0.5;     // 4px
$sm: $_unit;            // 8px
$md: $_unit * 2;        // 16px
$lg: $_unit * 3;        // 24px
$xl: $_unit * 4;        // 32px
$2xl: $_unit * 6;       // 48px
```

```scss
// tokens/_index.scss
@forward 'colors' as color-*;
@forward 'spacing' as space-*;
```

```scss
// components/_card.scss
@use '../tokens' as t;

.card {
  padding: t.$space-md;
  border: 1px solid t.$color-primary-light;
  border-radius: 4px;

  &__header {
    padding: t.$space-sm t.$space-md;
    background: t.$color-primary;
    color: white;
  }
}
```

### Use Case 2: Configurable Component Library

```scss
// lib/_config.scss
$prefix: 'ui' !default;
$primary: #0066cc !default;
$border-radius: 4px !default;
$transition-speed: 200ms !default;
```

```scss
// lib/_button.scss
@use 'config';

.#{config.$prefix}-button {
  background: config.$primary;
  border-radius: config.$border-radius;
  transition: background config.$transition-speed;
  border: none;
  color: white;
  padding: 0.5rem 1rem;
  cursor: pointer;

  &:hover {
    background: color.adjust(config.$primary, $lightness: -10%);
  }
}
```

```scss
// lib/_index.scss
@forward 'config';
@forward 'button';
@forward 'input';
@forward 'modal';
```

```scss
// app/main.scss (consumer)
@use 'lib' with (
  $prefix: 'app',
  $primary: #e74c3c,
  $border-radius: 8px,
);
// All lib components now use the custom configuration
```

### Use Case 3: Multi-theme Setup with `@forward` Configuration Layers

```scss
// themes/_base.scss
$bg: #ffffff !default;
$text: #333333 !default;
$surface: #f5f5f5 !default;
$border: #dddddd !default;
```

```scss
// themes/_dark.scss
@forward 'base' with (
  $bg: #1a1a1a !default,
  $text: #f0f0f0 !default,
  $surface: #2d2d2d !default,
  $border: #444444 !default,
);
```

```scss
// themes/_high-contrast.scss
@forward 'base' with (
  $bg: #000000 !default,
  $text: #ffffff !default,
  $surface: #1a1a1a !default,
  $border: #ffffff !default,
);
```

```scss
// main-dark.scss
@use 'themes/dark' as theme;

body {
  background: theme.$bg;
  color: theme.$text;
}

.card {
  background: theme.$surface;
  border: 1px solid theme.$border;
}
```

Each theme file is a configuration layer built on top of the base theme. The `!default` flags in the `@forward` allow further customization by the final consumer if needed.

## Migrating from `@import`

The `sass-migrator` tool automates most of the migration:

```bash
# Install
npm install -g sass-migrator

# Preview changes (dry run)
sass-migrator module --migrate-deps --dry-run src/styles/main.scss

# Run migration
sass-migrator module --migrate-deps src/styles/main.scss

# With verbose output
sass-migrator module --migrate-deps --verbose src/styles/main.scss
```

Key flags:

| Flag | Purpose |
|---|---|
| `--migrate-deps` | Also migrate all files loaded by the entrypoint |
| `--dry-run` | Preview without writing changes |
| `--verbose` | Show detailed output |
| `--forward=all` | Forward all public members from migrated files |
| `--remove-prefix` | Remove a prefix from member names (e.g., strip `mat-` after namespacing) |

After running the migrator, verify your compiled CSS output matches what it produced before migration. The module system may change CSS output order, which can affect cascade-dependent styles.

---

> **Key Takeaway:** Every Sass file should explicitly declare its dependencies with `@use`, and library-facing files should expose their public API through `@forward`. This module system eliminates global scope pollution, prevents CSS duplication, and makes dependency relationships visible and auditable.
