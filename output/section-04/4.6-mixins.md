# 4.6 Mixins (`@mixin`, `@include`, Parameters, Default Values)

> Mixins are reusable blocks of CSS declarations that accept arguments, emit styles, and can wrap content blocks. They are the primary tool for DRY stylesheets without the pitfalls of `@extend`.

---

## Basic Syntax

A mixin is declared with `@mixin` and included with `@include`:

```scss
@mixin reset-list {
  margin: 0;
  padding: 0;
  list-style: none;
}

nav ul {
  @include reset-list;
}
```

```css
/* Compiled CSS */
nav ul {
  margin: 0;
  padding: 0;
  list-style: none;
}
```

Mixins can contain any CSS that would be valid at the point of inclusion: property declarations, nested rules, at-rules, and even other `@include` statements.

```scss
@mixin card-base {
  border: 1px solid #ddd;
  border-radius: 4px;
  padding: 1rem;

  &:hover {
    box-shadow: 0 2px 8px rgba(0, 0, 0, 0.1);
  }
}

.card {
  @include card-base;
  background: white;
}
```

Note: mixin names treat hyphens and underscores as identical. `reset-list` and `reset_list` refer to the same mixin.

## Parameters

### Positional Arguments

```scss
@mixin size($width, $height) {
  width: $width;
  height: $height;
}

.avatar {
  @include size(48px, 48px);
}

.banner {
  @include size(100%, 300px);
}
```

### Default Values

Default values make arguments optional:

```scss
@mixin square($size, $radius: 0) {
  width: $size;
  height: $size;

  @if $radius != 0 {
    border-radius: $radius;
  }
}

.icon { @include square(32px); }           // No radius
.avatar { @include square(48px, 50%); }    // Circle
```

Default values can reference earlier parameters:

```scss
@mixin rectangle($width, $height: $width) {
  width: $width;
  height: $height;
}

.square { @include rectangle(100px); }          // 100px x 100px
.rect { @include rectangle(200px, 100px); }     // 200px x 100px
```

### The `null` Default Pattern

When a property should be omitted entirely if no value is passed, use `null` as the default. Sass does not output properties with `null` values.

```scss
@mixin box($width, $height, $bg: null, $border: null, $radius: null) {
  width: $width;
  height: $height;
  background: $bg;
  border: $border;
  border-radius: $radius;
}

.card {
  @include box(300px, 200px, $bg: white, $radius: 8px);
}
```

```css
/* Compiled CSS: border is omitted because it was null */
.card {
  width: 300px;
  height: 200px;
  background: white;
  border-radius: 8px;
}
```

This is cleaner than using empty strings or `false` as defaults, which would output invalid or empty CSS properties.

### Keyword (Named) Arguments

Arguments can be passed by name instead of position:

```scss
@mixin button($bg, $color: white, $padding: 0.5rem 1rem, $radius: 4px) {
  background: $bg;
  color: $color;
  padding: $padding;
  border-radius: $radius;
  border: none;
  cursor: pointer;
}

.btn-primary {
  @include button(#3498db);
}

.btn-danger {
  @include button(
    $bg: #e74c3c,
    $radius: 8px,
    $padding: 0.75rem 1.5rem,
  );
}
```

Keyword arguments are particularly valuable when a mixin has many optional parameters. They make the call self-documenting and order-independent.

Trailing commas in argument lists are permitted, which improves version control diffs.

### Rest Arguments (`...`)

Capture an arbitrary number of arguments:

```scss
@mixin transition($properties...) {
  $result: ();
  @each $prop in $properties {
    $result: append($result, $prop 0.3s ease, comma);
  }
  transition: $result;
}

.card {
  @include transition(background, color, box-shadow);
}
```

```css
/* Compiled CSS */
.card {
  transition: background 0.3s ease, color 0.3s ease, box-shadow 0.3s ease;
}
```

You can also expand a list when calling a mixin:

```scss
$shadow-args: 0 2px 8px rgba(0, 0, 0, 0.1);
.card {
  box-shadow: $shadow-args;
}

// Expanding a list into positional arguments
$sizes: 200px, 100px;
.box {
  @include size($sizes...);
}
```

### Arbitrary Keyword Arguments

Using `meta.keywords()`, you can accept any number of named arguments:

```scss
@use 'sass:meta';

@mixin define-colors($args...) {
  @each $name, $color in meta.keywords($args) {
    --color-#{$name}: #{$color};
  }
}

:root {
  @include define-colors(
    $primary: #3498db,
    $secondary: #2ecc71,
    $danger: #e74c3c,
    $warning: #f39c12,
  );
}
```

```css
/* Compiled CSS */
:root {
  --color-primary: #3498db;
  --color-secondary: #2ecc71;
  --color-danger: #e74c3c;
  --color-warning: #f39c12;
}
```

Important: if `meta.keywords()` is never called on an argument list, Sass rejects extra keyword arguments. This acts as validation against misspelled argument names.

## Content Blocks (`@content`)

Mixins can accept a block of CSS from the caller using `@content`:

```scss
@mixin on-hover {
  &:not([disabled]):hover {
    @content;
  }
}

.button {
  background: #3498db;

  @include on-hover {
    background: #2980b9;
    transform: translateY(-1px);
  }
}
```

```css
/* Compiled CSS */
.button { background: #3498db; }
.button:not([disabled]):hover {
  background: #2980b9;
  transform: translateY(-1px);
}
```

### Responsive Breakpoint Mixin

The most common `@content` pattern:

```scss
@use 'sass:map';

$breakpoints: (
  sm: 576px,
  md: 768px,
  lg: 992px,
  xl: 1200px,
  xxl: 1400px,
);

@mixin respond-to($name) {
  $value: map.get($breakpoints, $name);
  @if $value == null {
    @error "Unknown breakpoint: `#{$name}`. Available: #{map.keys($breakpoints)}";
  }
  @media (min-width: $value) {
    @content;
  }
}

.sidebar {
  display: none;

  @include respond-to(lg) {
    display: block;
    width: 280px;
  }
}
```

### `@content` with Arguments

Mixins can pass arguments back to the content block. The caller receives them with `using`:

```scss
@mixin responsive-grid($columns...) {
  @each $col in $columns {
    @media (min-width: map.get($breakpoints, $col)) {
      @content($col);
    }
  }
}

.grid {
  display: grid;
  grid-template-columns: 1fr;

  @include responsive-grid(md, lg, xl) using ($bp) {
    @if $bp == md {
      grid-template-columns: repeat(2, 1fr);
    } @else if $bp == lg {
      grid-template-columns: repeat(3, 1fr);
    } @else {
      grid-template-columns: repeat(4, 1fr);
    }
  }
}
```

### Content Block Scoping

Content blocks are **lexically scoped**. They see variables from the caller's scope, not from inside the mixin:

```scss
$color: red;

@mixin example {
  $color: blue;  // This is NOT visible to @content
  @content;
}

.box {
  @include example {
    color: $color;  // red (caller's scope), NOT blue
  }
}
```

This is a common source of confusion. The content block behaves as if it were written at the `@include` site, not inside the mixin body.

## Mixins vs `@extend`

### Output Differences

```scss
// Mixin approach
@mixin shared-styles {
  padding: 1rem;
  border: 1px solid #ddd;
  border-radius: 4px;
}

.card { @include shared-styles; background: white; }
.alert { @include shared-styles; background: #fff3cd; }
```

```css
/* Mixin output: declarations are duplicated */
.card { padding: 1rem; border: 1px solid #ddd; border-radius: 4px; background: white; }
.alert { padding: 1rem; border: 1px solid #ddd; border-radius: 4px; background: #fff3cd; }
```

```scss
// @extend approach
%shared-styles {
  padding: 1rem;
  border: 1px solid #ddd;
  border-radius: 4px;
}

.card { @extend %shared-styles; background: white; }
.alert { @extend %shared-styles; background: #fff3cd; }
```

```css
/* @extend output: selectors are grouped */
.card, .alert { padding: 1rem; border: 1px solid #ddd; border-radius: 4px; }
.card { background: white; }
.alert { background: #fff3cd; }
```

### Why Mixins Usually Win

Despite producing larger uncompressed CSS, mixins are preferred because:

1. **Gzip favors repetition.** Benchmarks show mixin output is 33% smaller after gzip than `@extend` output, because identical repeated strings compress extremely well while long unique selector lists do not.
2. **Predictable output.** `@extend` can produce unexpected selector combinations, especially in nested contexts.
3. **No cross-media-query limitation.** `@extend` cannot cross `@media` boundaries. Mixins work everywhere.
4. **Arguments.** Mixins accept parameters. `@extend` does not.

Use `@extend` only for semantic IS-A relationships (`.error--serious` extends `.error`). Use mixins for everything else.

## Mixins vs Functions

| Aspect | `@mixin` | `@function` |
|---|---|---|
| Purpose | Emit CSS declarations and rules | Compute and return a value |
| Usage | `@include mixin-name()` | `property: function-name()` |
| Output | Produces CSS directly | Returns a single value |
| Can contain rules | Yes | No |
| Can use `@content` | Yes | No |

```scss
// Function: computes a value
@function spacing($multiplier) {
  @return 8px * $multiplier;
}

// Mixin: emits CSS
@mixin flex-center($direction: row) {
  display: flex;
  justify-content: center;
  align-items: center;
  flex-direction: $direction;
}

.card {
  @include flex-center(column);
  padding: spacing(2);        // 16px
  margin-bottom: spacing(3);  // 24px
}
```

Rule of thumb: if you need a **value**, write a function. If you need **CSS output**, write a mixin.

## Mixins in the Module System

With `@use`, mixins are accessed through namespaces:

```scss
// _layout.scss
@mixin container($max-width: 1200px) {
  max-width: $max-width;
  margin-inline: auto;
  padding-inline: 1rem;
}

@mixin flex-center {
  display: flex;
  justify-content: center;
  align-items: center;
}
```

```scss
// component.scss
@use 'layout';

.page {
  @include layout.container;
}

.hero {
  @include layout.flex-center;
  min-height: 80vh;
}
```

Custom namespace:

```scss
@use 'layout' as l;

.page { @include l.container(960px); }
```

Private mixins (prefixed with `-` or `_`) are not accessible outside their module:

```scss
// _layout.scss
@mixin -reset-spacing {  // Private
  margin: 0;
  padding: 0;
}

@mixin container($max-width: 1200px) {
  @include -reset-spacing;  // Internal use is fine
  max-width: $max-width;
  margin-inline: auto;
}
```

## First-Class Mixins (`meta.get-mixin` and `meta.apply`)

Dart Sass 1.69.0+ supports treating mixins as values that can be stored in variables and passed to other mixins:

```scss
@use 'sass:meta';

@mixin button-base {
  padding: 0.5rem 1rem;
  border: none;
  border-radius: 4px;
  cursor: pointer;
}

@mixin apply-variants($base-mixin, $variants) {
  @include meta.apply($base-mixin);

  @each $name, $bg in $variants {
    &--#{$name} {
      @include meta.apply($base-mixin);
      background: $bg;
    }
  }
}

.btn {
  @include apply-variants(
    meta.get-mixin('button-base'),
    (primary: #3498db, danger: #e74c3c, success: #2ecc71)
  );
}
```

Key functions:

- `meta.get-mixin($name, $module: null)` returns a mixin as a value.
- `meta.apply($mixin, $args...)` includes a mixin value. Content blocks are forwarded.
- `meta.mixin-exists($name, $module: null)` checks if a mixin exists.

This replaces the need for interpolation-based dynamic mixin calls (which Sass does not support: `@include #{$name}` is invalid).

## Argument Forwarding

Pass all arguments from one mixin to another:

```scss
// Alias pattern for deprecation
@mixin btn($args...) {
  @warn "btn() is deprecated. Use button() instead.";
  @include button($args...);
}
```

This forwards both positional and keyword arguments intact.

## Common Mixin Patterns

### Visually Hidden (Accessibility)

```scss
@mixin visually-hidden {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}

.sr-only {
  @include visually-hidden;
}
```

### Truncate Text

```scss
@mixin truncate($max-width: 100%) {
  max-width: $max-width;
  overflow: hidden;
  text-overflow: ellipsis;
  white-space: nowrap;
}

@mixin line-clamp($lines: 2) {
  display: -webkit-box;
  -webkit-line-clamp: $lines;
  -webkit-box-orient: vertical;
  overflow: hidden;
}
```

### Aspect Ratio (Pre-`aspect-ratio` Fallback)

```scss
@mixin aspect-ratio($width, $height) {
  aspect-ratio: #{$width} / #{$height};

  // Fallback for older browsers
  @supports not (aspect-ratio: 1) {
    &::before {
      content: '';
      display: block;
      padding-top: math.div($height, $width) * 100%;
    }
  }
}
```

### Focus Visible Ring

```scss
@mixin focus-ring($color: #3498db, $offset: 2px, $width: 2px) {
  &:focus-visible {
    outline: $width solid $color;
    outline-offset: $offset;
  }
}

.button { @include focus-ring; }
.link { @include focus-ring(currentColor, 4px); }
```

### Container with Breakpoint Variants

```scss
@mixin container($max-widths: (sm: 540px, md: 720px, lg: 960px, xl: 1140px)) {
  width: 100%;
  margin-inline: auto;
  padding-inline: 1rem;

  @each $bp, $width in $max-widths {
    @include respond-to($bp) {
      max-width: $width;
    }
  }
}
```

### Custom Property Setter

```scss
@mixin define-tokens($tokens) {
  @each $name, $value in $tokens {
    --#{$name}: #{$value};
  }
}

:root {
  @include define-tokens((
    color-primary: #3498db,
    color-secondary: #2ecc71,
    spacing-sm: 8px,
    spacing-md: 16px,
    radius-default: 4px,
  ));
}
```

## Common Mistakes

### Mistake 1: Argument-less mixins used everywhere

```scss
// BAD: This mixin adds no parametric value
@mixin card-shadow {
  box-shadow: 0 2px 8px rgba(0, 0, 0, 0.1);
}

// Used in 50 places -- produces 50 identical declarations
// A utility class would be simpler:
.shadow-sm { box-shadow: 0 2px 8px rgba(0, 0, 0, 0.1); }
```

If a mixin has no parameters and no `@content`, consider whether a CSS class or placeholder selector is more appropriate.

### Mistake 2: Mixins as CSS knowledge avoidance

```scss
// BAD: Hiding standard CSS behind a mixin
@mixin center-flex {
  display: flex;
  justify-content: center;
  align-items: center;
}
```

If the mixin name is just a description of the CSS it contains, and it takes no arguments, it adds indirection without value. Write the CSS directly.

### Mistake 3: Not using `null` for optional properties

```scss
// BAD: Empty string creates invalid CSS
@mixin box($border: '') {
  border: $border;  // Outputs: border: ;
}

// GOOD: null omits the property entirely
@mixin box($border: null) {
  border: $border;  // Omitted from output when null
}
```

### Mistake 4: Expecting `@content` to see mixin variables

```scss
@mixin wrapper {
  $inner-padding: 2rem;  // NOT visible to @content
  .wrapper {
    @content;
  }
}

@include wrapper {
  padding: $inner-padding;  // ERROR: undefined variable
}
```

Content blocks are lexically scoped to the include site. Use `@content` with arguments to pass values from the mixin to the caller.

### Mistake 5: Deep nesting inside mixins

```scss
// BAD: Mixin produces deeply nested, high-specificity output
@mixin nav-styles {
  nav {
    ul {
      li {
        a {
          color: blue;
        }
      }
    }
  }
}
```

The same nesting rules apply inside mixins. Keep it flat.

### Mistake 6: Vendor prefix mixins in 2025+

```scss
// UNNECESSARY: Autoprefixer handles this at build time
@mixin transform($value) {
  -webkit-transform: $value;
  -ms-transform: $value;
  transform: $value;
}
```

Use Autoprefixer in your build pipeline instead. Hand-written prefix mixins produce unnecessary CSS for browsers that do not need the prefix.

## Best Practices

### Practice 1: Use arguments to justify the mixin's existence

A mixin should do something a CSS class cannot: accept parameters, use `@content`, or generate dynamic CSS. If it does none of these, a plain class is likely better.

### Practice 2: Use `null` defaults for optional properties

```scss
@mixin text-style($size, $weight: null, $transform: null, $decoration: null) {
  font-size: $size;
  font-weight: $weight;
  text-transform: $transform;
  text-decoration: $decoration;
}
```

Only the properties with non-null values appear in the output.

### Practice 3: Use keyword arguments for mixins with 3+ optional parameters

```scss
// Clear and self-documenting
@include button(
  $bg: #3498db,
  $radius: 8px,
  $padding: 0.75rem 1.5rem,
);
```

### Practice 4: Provide error messages for invalid arguments

```scss
@mixin respond-to($name) {
  $value: map.get($breakpoints, $name);
  @if $value == null {
    @error "Unknown breakpoint `#{$name}`. Available: #{map.keys($breakpoints)}";
  }
  @media (min-width: $value) {
    @content;
  }
}
```

`@error` stops compilation with a clear message. `@warn` prints a warning but continues.

### Practice 5: Keep mixins focused

One mixin should do one thing. Prefer composing small mixins over building monolithic ones:

```scss
// Focused, composable mixins
@mixin flex-center { display: flex; justify-content: center; align-items: center; }
@mixin text-truncate { overflow: hidden; text-overflow: ellipsis; white-space: nowrap; }

// Composed at the include site
.header-title {
  @include flex-center;
  @include text-truncate;
  max-width: 300px;
}
```

## Real-World Use Cases

### Use Case 1: Responsive Breakpoint System

```scss
// _breakpoints.scss
@use 'sass:map';

$breakpoints: (
  sm: 576px,
  md: 768px,
  lg: 992px,
  xl: 1200px,
  xxl: 1400px,
) !default;

@mixin respond-to($name) {
  $value: map.get($breakpoints, $name);
  @if $value == null {
    @error "Unknown breakpoint: #{$name}";
  }
  @media (min-width: $value) {
    @content;
  }
}

@mixin respond-below($name) {
  $value: map.get($breakpoints, $name);
  @if $value == null {
    @error "Unknown breakpoint: #{$name}";
  }
  @media (max-width: $value - 0.02px) {
    @content;
  }
}

@mixin respond-between($lower, $upper) {
  $min: map.get($breakpoints, $lower);
  $max: map.get($breakpoints, $upper);
  @media (min-width: $min) and (max-width: $max - 0.02px) {
    @content;
  }
}
```

```scss
// Usage
@use 'breakpoints' as bp;

.grid {
  display: grid;
  grid-template-columns: 1fr;
  gap: 1rem;

  @include bp.respond-to(md) {
    grid-template-columns: repeat(2, 1fr);
  }

  @include bp.respond-to(xl) {
    grid-template-columns: repeat(4, 1fr);
  }
}
```

### Use Case 2: Component Variant Generator

```scss
// _buttons.scss
@use 'sass:color';

@mixin button-variant($bg, $text: white) {
  background-color: $bg;
  color: $text;
  border: 1px solid $bg;

  &:hover {
    background-color: color.adjust($bg, $lightness: -8%);
    border-color: color.adjust($bg, $lightness: -12%);
  }

  &:active {
    background-color: color.adjust($bg, $lightness: -15%);
  }

  &:disabled {
    background-color: color.scale($bg, $saturation: -50%, $lightness: 30%);
    border-color: transparent;
    cursor: not-allowed;
  }
}

$button-variants: (
  primary: #3498db,
  secondary: #6c757d,
  success: #2ecc71,
  danger: #e74c3c,
  warning: #f39c12,
);

@each $name, $color in $button-variants {
  .btn-#{$name} {
    @include button-variant($color);
  }
}
```

### Use Case 3: Theme-Aware Component Mixin

```scss
// _theme.scss
@mixin themed($property, $light-value, $dark-value) {
  #{$property}: $light-value;

  @media (prefers-color-scheme: dark) {
    #{$property}: $dark-value;
  }
}

@mixin themed-block {
  @content(light);

  @media (prefers-color-scheme: dark) {
    @content(dark);
  }
}
```

```scss
// Usage
@use 'theme';

.card {
  @include theme.themed(background, #ffffff, #1a1a1a);
  @include theme.themed(color, #333333, #f0f0f0);
  @include theme.themed(border-color, #dddddd, #444444);
  border: 1px solid;
  border-radius: 4px;
  padding: 1rem;
}

.hero {
  @include theme.themed-block using ($mode) {
    @if $mode == light {
      background: linear-gradient(135deg, #667eea, #764ba2);
      color: white;
    } @else {
      background: linear-gradient(135deg, #1a1a2e, #16213e);
      color: #e0e0e0;
    }
  }
}
```

---

> **Key Takeaway:** Mixins earn their place when they accept arguments, use `@content`, or generate dynamic CSS. If a mixin has no parameters and no content block, a plain CSS class is usually simpler and produces less output.
