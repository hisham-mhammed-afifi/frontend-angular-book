# 5.10 Scope and Closures

> Scope determines where variables are accessible, and closures allow functions to remember the scope in which they were created. Together, they form the foundation of data privacy, state management, and most advanced patterns in JavaScript.

---

## Execution Context and the Call Stack

Every piece of JavaScript code runs inside an **execution context**. Understanding how these contexts are created and managed is the key to understanding scope.

### Two Phases of Execution Context Creation

1. **Creation Phase**: Memory is allocated. `var` declarations are hoisted and initialized to `undefined`. `let`/`const` declarations are hoisted but placed in the Temporal Dead Zone (uninitialized). Function declarations are stored fully in memory (both name and body).
2. **Execution Phase**: Code runs line by line. Assignments happen. Functions are invoked.

```js
console.log(a); // undefined (var is hoisted and initialized to undefined)
console.log(b); // ReferenceError (let is hoisted but in TDZ)

var a = 1;
let b = 2;
```

### Execution Context Components

Each execution context contains three internal components:

| Component | Purpose |
|---|---|
| **LexicalEnvironment** | Holds `let`, `const`, and function declaration bindings. Used for identifier resolution. |
| **VariableEnvironment** | Holds `var` bindings. Stays fixed for the function. |
| **ThisBinding** | The value of `this` for this context. |

### The Call Stack

The call stack is a LIFO (Last-In, First-Out) structure. When a function is called, a new execution context is pushed onto the stack. When it returns, the context is popped off.

```js
function first() {
  console.log("first start");
  second();
  console.log("first end");
}

function second() {
  console.log("second start");
  third();
  console.log("second end");
}

function third() {
  console.log("third");
}

first();
// Call stack growth: Global -> first -> second -> third
// Call stack shrink: third pops -> second pops -> first pops -> back to Global
```

If the stack grows too deep (infinite recursion), you get a `RangeError: Maximum call stack size exceeded`.

## Lexical Environments and Environment Records

The ECMAScript specification defines scope through **Lexical Environments**, which consist of two parts:

1. **Environment Record**: The actual storage for identifiers (variables, functions, parameters).
2. **Outer Environment Reference**: A pointer to the parent lexical environment, forming the scope chain.

### Types of Environment Records

| Type | Used By | Stores |
|---|---|---|
| **Declarative** | Function scopes, block scopes | `let`, `const`, `class`, function params |
| **Object** | Global scope (partially), `with` statements | Bindings as properties on a backing object |
| **Function** | Function calls | Everything Declarative does, plus `this`, `arguments`, `super`, `new.target` |
| **Module** | ES modules | Everything Declarative does, plus immutable `import` bindings |
| **Global** | Global scope | Composite: Declarative (for `let`/`const`) + Object (for `var`/function declarations on `globalThis`) |

### Why `var` and `let` Differ at Global Scope

The Global Environment Record is a composite. `var` and function declarations go to the Object record (becoming properties of `globalThis`). `let` and `const` go to the Declarative record (NOT properties of `globalThis`).

```js
var x = 1;
let y = 2;

console.log(globalThis.x); // 1 (var is on the Object record)
console.log(globalThis.y); // undefined (let is on the Declarative record)
console.log(y);            // 2 (still accessible via scope chain)
```

## The Scope Chain and Variable Resolution

When code references a variable, the engine searches for it using a well-defined algorithm.

### The Resolution Algorithm

The spec defines `GetIdentifierReference(lex, name, strict)`:

1. If `lex` is `null`, the variable is not found anywhere. Throw `ReferenceError`.
2. Check if `lex`'s Environment Record has a binding for `name`.
3. If found, return a reference to that binding.
4. If not found, set `lex` to the outer environment reference and go to step 1.

This recursive walk from inner to outer lexical environments IS the scope chain.

```js
const global = "global";

function outer() {
  const outerVar = "outer";

  function inner() {
    const innerVar = "inner";

    // Resolution chain for each variable:
    // innerVar: found in inner's Environment Record
    // outerVar: not in inner -> found in outer's Environment Record
    // global:   not in inner -> not in outer -> found in Global Environment Record
    console.log(innerVar, outerVar, global);
  }

  inner();
}

outer(); // "inner outer global"
```

### Scope Chain Is Lexical, Not Dynamic

The chain is determined by where the function is **defined** in the source code, not where it is **called**.

```js
const value = "global";

function printValue() {
  console.log(value); // looks up value in the scope where printValue was DEFINED
}

function wrapper() {
  const value = "wrapper";
  printValue(); // still prints "global", NOT "wrapper"
}

wrapper(); // "global"
```

This is the core principle of **lexical scoping**: the scope chain is fixed at definition time.

## Five Types of Scope

### 1. Global Scope

Variables declared outside any function or block. Accessible everywhere.

```js
var globalVar = "I am global";

function anywhere() {
  console.log(globalVar); // accessible
}
```

### 2. Module Scope

Top-level declarations in an ES module are scoped to that module, not the global scope.

```js
// module.js
const moduleVar = "I am module-scoped";
export function getVar() {
  return moduleVar;
}

// other.js
import { getVar } from "./module.js";
console.log(getVar());      // "I am module-scoped"
console.log(typeof moduleVar); // "undefined" (not in this module's scope)
```

Important: `var` at the top level of an ES module does NOT create a property on `globalThis` (unlike in a script).

### 3. Function Scope

Created by function declarations, function expressions, and arrow functions. `var`, `let`, and `const` are all contained within function scope.

```js
function example() {
  var funcScoped = "function";
  let blockScoped = "also function-scoped here";

  if (true) {
    var stillFuncScoped = "var ignores block";
    let blockOnly = "block-scoped";
  }

  console.log(stillFuncScoped); // "var ignores block"
  console.log(typeof blockOnly); // "undefined" (block scope ended)
}
```

### 4. Block Scope

Created by `{}` blocks (if, for, while, switch, bare blocks). Only `let` and `const` are block-scoped. `var` ignores block scope entirely.

```js
{
  let blockLet = 1;
  const blockConst = 2;
  var notBlockScoped = 3;
}

console.log(notBlockScoped); // 3 (var escapes the block)
console.log(blockLet);       // ReferenceError
```

### 5. Catch Clause Scope

The `catch` parameter has its own block scope since ES2015.

```js
try {
  throw new Error("test");
} catch (error) {
  console.log(error.message); // "test"
}

console.log(typeof error); // "undefined" (error is scoped to the catch block)
```

### Scope Comparison Table

| Feature | Global | Module | Function | Block |
|---|---|---|---|---|
| `var` contained | No (becomes global property) | Yes | Yes | No (leaks out) |
| `let`/`const` contained | Yes (Declarative record) | Yes | Yes | Yes |
| Function declarations | Hoisted globally | Hoisted to module top | Hoisted to function top | Block-scoped (strict) |
| Creates new `this` | N/A | N/A | Yes (regular functions) | No |

## The Temporal Dead Zone (TDZ)

The TDZ is the period between entering a scope and reaching the declaration line for `let`, `const`, or `class`.

### TDZ Is Temporal, Not Positional

```js
// This works because the function is CALLED after the declaration
function safe() {
  return x; // reference to x is fine here
}

let x = 10;
console.log(safe()); // 10
```

```js
// This fails because the function is CALLED before the declaration
function unsafe() {
  return y;
}

console.log(unsafe()); // ReferenceError: Cannot access 'y' before initialization
let y = 10;
```

The TDZ is about when the code runs, not where it appears in the source.

### `typeof` in the TDZ

```js
// Undeclared variable: typeof returns "undefined" (no error)
console.log(typeof undeclaredVar); // "undefined"

// TDZ variable: typeof throws ReferenceError
let declared;
console.log(typeof laterDeclared); // ReferenceError
let laterDeclared = 1;
```

This is a significant difference from `var` behavior, where `typeof` never throws.

### TDZ in Default Parameters

Default parameters create their own scope, and TDZ applies within it.

```js
// ReferenceError: Cannot access 'b' before initialization
function broken(a = b, b = 1) {
  return a + b;
}

// Works: a is already initialized when b's default is evaluated
function fixed(a = 1, b = a * 2) {
  return a + b; // 1 + 2 = 3
}
```

### Class Declarations Are in the TDZ

```js
// ReferenceError: Cannot access 'MyClass' before initialization
const instance = new MyClass();

class MyClass {
  constructor() {
    this.value = 42;
  }
}
```

## How Closures Work Under the Hood

A closure is a function combined with a reference to the lexical environment where it was created. Every function has an internal `[[Environment]]` property that points to its defining scope.

### The Mechanism

```js
function createGreeter(greeting) {
  // A new Lexical Environment is created with: { greeting: "Hello" }

  return function (name) {
    // This function's [[Environment]] points to createGreeter's Lexical Environment
    // When called, a new Lexical Environment is created for this function,
    // with its outer reference pointing to createGreeter's environment
    return `${greeting}, ${name}`;
  };
}

const hello = createGreeter("Hello");
// createGreeter has returned, but its Lexical Environment is NOT garbage collected
// because hello's [[Environment]] still references it

console.log(hello("Alice")); // "Hello, Alice"
// Variable resolution: name -> found locally. greeting -> not local -> found in outer env.
```

### What the Engine Retains

When a closure is created, the engine links the function to the Lexical Environment of the enclosing scope. The key question is: which variables from that scope are retained?

```js
function outer() {
  const kept = "I am referenced";
  const discarded = "I am NOT referenced";
  const huge = new Array(1_000_000);

  return function inner() {
    return kept; // only 'kept' is referenced
  };
}

const fn = outer();
```

V8 performs static analysis and only retains variables that are actually referenced by inner functions. In the example above, `discarded` and `huge` are eligible for garbage collection after `outer` returns, because `inner` does not reference them.

However, this optimization is defeated by `eval`:

```js
function outer() {
  const kept = "referenced";
  const shouldBeDiscarded = "not referenced directly";

  return function inner() {
    return eval("shouldBeDiscarded"); // eval forces ALL variables to be retained
  };
}
```

The presence of `eval` prevents the engine from statically analyzing which variables are needed, so it retains everything.

## The Shared Scope Problem

This is the most important closure memory pitfall, and it is subtle enough to cause production memory leaks.

When multiple closures are created in the same scope, they share a single internal Context object (the Lexical Environment). Even if one closure does not reference a particular variable, it keeps that variable alive if another closure in the same scope does.

```js
function createHandlers() {
  const hugeData = new Array(1_000_000).fill("x");

  // This closure references hugeData
  function processData() {
    return hugeData.length;
  }

  // This closure does NOT reference hugeData
  function getTimestamp() {
    return Date.now();
  }

  // Both closures share the same scope
  // hugeData cannot be GC'd as long as EITHER closure is alive
  return { processData, getTimestamp };
}

const handlers = createHandlers();
// Even if we only use getTimestamp and never call processData,
// hugeData is still retained in memory
```

### The Fix: Scope Separation

```js
function createHandlers() {
  const processData = createProcessor();
  const getTimestamp = () => Date.now();

  return { processData, getTimestamp };
}

function createProcessor() {
  const hugeData = new Array(1_000_000).fill("x");
  return function () {
    return hugeData.length;
  };
}

// Now getTimestamp and processData are in different scopes
// If processData is GC'd, hugeData can be too
```

### The Fix: Nullify When Done

```js
function createHandlers() {
  let hugeData = new Array(1_000_000).fill("x");

  const result = processArray(hugeData);
  hugeData = null; // allow GC

  function getResult() {
    return result;
  }

  function getTimestamp() {
    return Date.now();
  }

  return { getResult, getTimestamp };
}
```

## Closures Over Loop Variables

### The Classic Bug with `var`

```js
for (var i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 100);
}
// Output: 5, 5, 5, 5, 5
// All closures share the same 'i' (function-scoped)
```

### The Fix with `let`

The spec defines special behavior for `let` in `for` loops: a new Lexical Environment is created for each iteration. The spec calls this `CreatePerIterationEnvironment`. Each iteration's environment copies the loop variable's current value.

```js
for (let i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 100);
}
// Output: 0, 1, 2, 3, 4
// Each iteration has its own 'i' binding
```

### The Pre-ES6 Fix with IIFE

```js
for (var i = 0; i < 5; i++) {
  (function (j) {
    setTimeout(() => console.log(j), 100);
  })(i);
}
// Output: 0, 1, 2, 3, 4
// The IIFE creates a new function scope per iteration
```

### `const` in Loops

```js
// 'const' in a standard for loop: error on increment
for (const i = 0; i < 5; i++) {} // TypeError: Assignment to constant variable

// 'const' in for...of: works, because a new binding is created per iteration
for (const item of [1, 2, 3]) {
  setTimeout(() => console.log(item), 100);
}
// Output: 1, 2, 3 (each iteration gets its own const binding)
```

## Stale Closures

A stale closure captures a variable's value at a point in time and continues to use that outdated value even after the variable has been updated. This is correct closure behavior, but it creates bugs when you expect the closure to see the latest value.

### The Pattern

```js
function createLogger() {
  let count = 0;

  // This closure captures 'count' by reference
  const logCount = () => console.log(`Count: ${count}`);

  // Take a snapshot (stale closure)
  const snapshot = logCount;

  count = 10;

  snapshot(); // "Count: 10" -- NOT stale, because closures capture by reference
}
```

Wait, that is not stale. Closures capture the **variable**, not the **value**. The staleness problem occurs when the variable itself is replaced (as happens with React state) or when a new scope is created:

```js
function setupTimers() {
  for (let i = 0; i < 3; i++) {
    const value = i; // new 'value' each iteration
    setTimeout(() => {
      console.log(value); // each closure has its own 'value'
    }, 1000);
  }

  // But what if we want the LATEST value?
  let latest = 0;
  const timer = setInterval(() => {
    console.log(latest); // always logs whatever 'latest' was when the closure was created
    // If 'latest' is in the same scope and mutated, the closure sees the mutation
  }, 1000);

  latest = 42;
  // The interval callback WILL see 42, because 'latest' is a mutable let binding
}
```

### Where Stale Closures Actually Bite

The real problem is when the closure is created once but the "variable" is conceptually replaced (not mutated). This is common in functional patterns and React hooks where state values are immutable snapshots:

```js
function createApp() {
  let state = { count: 0 };

  const handler = () => {
    // This closure captured 'state' by reference
    // But if state is REPLACED (not mutated), the closure sees the replacement
    console.log(state.count);
  };

  // Replacing state works fine because the closure holds a reference to the variable
  state = { count: 10 };
  handler(); // 10

  // The staleness problem occurs when a closure is created in one "cycle"
  // and the reference it needs is in a DIFFERENT scope that has since been replaced
}
```

### The Ref Pattern (Escaping Staleness)

Use a mutable container object to always access the latest value:

```js
function createUpdatableHandler() {
  const ref = { current: null };

  function setHandler(fn) {
    ref.current = fn;
  }

  function callHandler(...args) {
    // Always calls the LATEST handler, not a stale closure
    return ref.current?.(...args);
  }

  return { setHandler, callHandler };
}
```

## Closure Patterns

### The Module Pattern

Uses an IIFE to create a private scope, returning an object with public methods that close over private state. Creates a singleton.

```js
const counter = (function () {
  let count = 0; // private

  return {
    increment() { return ++count; },
    decrement() { return --count; },
    getCount() { return count; }
  };
})();

counter.increment();
counter.increment();
console.log(counter.getCount()); // 2
console.log(counter.count);      // undefined (truly private)
```

### The Revealing Module Pattern

All functions are defined privately. The return object maps public names to private references.

```js
const calculator = (function () {
  let result = 0;

  function add(n) { result += n; }
  function subtract(n) { result -= n; }
  function getResult() { return result; }
  function reset() { result = 0; }

  // Reveal only what should be public
  return {
    add,
    subtract,
    getResult,
    // reset is NOT exposed (private)
  };
})();

calculator.add(10);
calculator.subtract(3);
console.log(calculator.getResult()); // 7
```

### The Factory Pattern

Like the module pattern but not an IIFE. Called multiple times to create multiple independent instances.

```js
function createStack() {
  const items = []; // private per instance

  return {
    push(item) { items.push(item); },
    pop() { return items.pop(); },
    peek() { return items[items.length - 1]; },
    get size() { return items.length; },
    isEmpty() { return items.length === 0; }
  };
}

const stack1 = createStack();
const stack2 = createStack();

stack1.push("a");
stack1.push("b");

console.log(stack1.size);   // 2
console.log(stack2.size);   // 0 (completely independent)
console.log(stack1.items);  // undefined (private)
```

### Memoization

Uses a closure to maintain a persistent cache between calls.

```js
function memoize(fn) {
  const cache = new Map();

  return function (...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const result = fn.apply(this, args);
    cache.set(key, result);
    return result;
  };
}

const expensiveSquare = memoize((n) => {
  console.log("Computing...");
  return n * n;
});

expensiveSquare(4); // "Computing..." -> 16
expensiveSquare(4); // 16 (cached, no "Computing...")
expensiveSquare(5); // "Computing..." -> 25
```

### Partial Application and Currying

Closures remember previously supplied arguments.

```js
function partial(fn, ...presetArgs) {
  return function (...laterArgs) {
    return fn(...presetArgs, ...laterArgs);
  };
}

function log(level, timestamp, message) {
  console.log(`[${level}] ${timestamp}: ${message}`);
}

const logError = partial(log, "ERROR");
const logInfo = partial(log, "INFO");

logError(Date.now(), "Database connection failed");
logInfo(Date.now(), "Server started on port 3000");
```

## Closures vs Classes: Memory and Performance

When you need to create many instances with private state, the choice between closures and classes has measurable implications.

### Closure-Based (Factory)

```js
function createCounter(initial = 0) {
  let count = initial;
  return {
    increment() { return ++count; },
    getCount() { return count; }
  };
}
```

Each call creates new function objects for `increment` and `getCount`. For 1,000 instances, that is 2,000 function objects.

### Class-Based

```js
class Counter {
  #count; // truly private (ES2022 private fields)

  constructor(initial = 0) {
    this.#count = initial;
  }

  increment() { return ++this.#count; }
  getCount() { return this.#count; }
}
```

Methods live on the prototype and are shared. For 1,000 instances, there are still only 2 function objects total.

### Comparison

| Aspect | Closure Factory | Class |
|---|---|---|
| Memory per instance | Higher (new function objects) | Lower (shared prototype methods) |
| Creation speed | Slightly slower | Slightly faster |
| V8 optimization | Megamorphic call sites possible | Monomorphic inline caches |
| True privacy | Yes (closure scope) | Yes (`#` private fields) |
| Inheritance | Manual | Built-in (`extends`, `super`) |

For single instances (singletons, module pattern), closures are fine. For many instances (hundreds or thousands), prefer classes.

## Memory Leaks from Closures

### Common Leak Patterns

**1. Forgotten timers:**

```js
function startPolling(url) {
  const data = [];

  // This interval keeps 'data' alive forever
  setInterval(async () => {
    const response = await fetch(url);
    const result = await response.json();
    data.push(result); // data grows indefinitely
  }, 5000);

  // No way to stop the interval or clear data
}
```

**2. Event listeners not removed:**

```js
function setupHandler(element) {
  const heavyState = new Array(100_000).fill("data");

  element.addEventListener("click", () => {
    console.log(heavyState.length);
  });

  // If element is removed from the DOM but the listener is not removed,
  // both the element and heavyState remain in memory
}
```

**3. Accumulating closures in arrays:**

```js
const handlers = [];

function registerHandler() {
  const localData = fetchSomeData(); // potentially large

  handlers.push(() => {
    process(localData);
  });

  // Every call adds a closure that retains its own localData
  // The handlers array grows forever
}
```

### Prevention Checklist

- Always clear `setInterval` and `setTimeout` when they are no longer needed.
- Remove event listeners when elements are removed from the DOM.
- Set references to `null` when large data is no longer needed inside a closure's scope.
- Use `WeakMap` or `WeakRef` when associating data with objects that should be garbage-collectible.
- Separate scopes so long-lived closures do not share scope with closures that hold large data.
- Profile memory with Chrome DevTools Heap Snapshots to identify retained closures.

## `eval` and Dynamic Scope Interaction

JavaScript uses lexical scope exclusively. Two features can disrupt this: `eval()` and `with`.

### Direct `eval` Accesses the Local Scope

```js
function example() {
  const secret = 42;

  // Direct eval runs in the current scope
  console.log(eval("secret")); // 42
}
```

### Indirect `eval` Runs in Global Scope

```js
function example() {
  const secret = 42;
  const indirectEval = eval;

  // Indirect eval runs in the global scope
  console.log(indirectEval("typeof secret")); // "undefined"
}
```

### `eval` Defeats Closure Optimization

```js
function outer() {
  const a = 1;
  const b = 2;
  const huge = new Array(1_000_000);

  return function inner() {
    // Without eval: V8 only retains variables inner actually references
    // With eval: V8 retains ALL variables because it cannot predict what eval accesses
    return eval("a + b");
  };
}
// 'huge' is retained even though inner never references it directly
```

### `with` Statement (Avoid)

```js
// Bad Practice: 'with' creates dynamic scope behavior
const obj = { x: 10, y: 20 };

with (obj) {
  console.log(x + y); // 30 (resolves x and y from obj)
}
// 'with' is forbidden in strict mode for good reason
```

## Common Mistakes

- **Shared scope memory retention**: Multiple closures in the same scope preventing garbage collection of large objects that only one closure references. Restructure into separate scopes or nullify large references.

- **Forgetting to clean up timers and listeners**: Closures passed to `setInterval`, `setTimeout`, or `addEventListener` keep their scope alive indefinitely if not cleared.

- **Using `var` in loops and expecting per-iteration closures**: `var` is function-scoped. Use `let` for per-iteration bindings.

- **Assuming V8 retains only referenced variables when `eval` is present**: The presence of `eval` anywhere in a function body forces the engine to retain all variables in the scope.

- **Not understanding that the scope chain is lexical**: The chain follows where the function is defined in the code, not where it is called from.

- **Creating closures in hot loops unnecessarily**: Each closure creates new function objects. If the closure does not need per-iteration state, hoist it outside the loop.

---

## Best Practices

- Prefer `let` and `const` over `var`. Block scoping eliminates most accidental closure bugs.
- Be aware of the shared scope problem. If a long-lived closure shares scope with a closure that captures large data, restructure the code into separate scopes.
- Always clean up: clear timers, remove event listeners, unsubscribe from observables when closures are no longer needed.
- Avoid `eval`. It defeats engine optimizations and creates security vulnerabilities.
- Use closures deliberately for encapsulation: module pattern for singletons, factory pattern for multiple instances, memoization for caching.
- Prefer classes over closure factories when creating many instances (hundreds or more) for better memory efficiency.
- Use `WeakMap` and `WeakRef` when you need to associate data with objects without preventing garbage collection.
- Profile memory with Chrome DevTools Heap Snapshots to detect retained closures in production.

## Real-World Use Cases

### Private State with Closure-Based Service

```js
function createAuthService() {
  let token = null;
  let refreshTimer = null;

  async function login(credentials) {
    const response = await fetch("/api/login", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(credentials)
    });

    const data = await response.json();
    token = data.token;

    // Schedule token refresh
    clearTimeout(refreshTimer);
    refreshTimer = setTimeout(refreshToken, data.expiresIn - 60000);
  }

  async function refreshToken() {
    const response = await fetch("/api/refresh", {
      headers: { Authorization: `Bearer ${token}` }
    });
    const data = await response.json();
    token = data.token;
    refreshTimer = setTimeout(refreshToken, data.expiresIn - 60000);
  }

  function getToken() {
    return token;
  }

  function logout() {
    token = null;
    clearTimeout(refreshTimer);
    refreshTimer = null;
  }

  return { login, getToken, logout };
}

const auth = createAuthService();
// token and refreshTimer are completely private
// No external code can tamper with them
```

### Event Handler with Scope Isolation

```js
function createDragHandler(element) {
  let isDragging = false;
  let startX = 0;
  let startY = 0;
  let offsetX = 0;
  let offsetY = 0;

  function onMouseDown(e) {
    isDragging = true;
    startX = e.clientX - offsetX;
    startY = e.clientY - offsetY;
    document.addEventListener("mousemove", onMouseMove);
    document.addEventListener("mouseup", onMouseUp);
  }

  function onMouseMove(e) {
    if (!isDragging) return;
    offsetX = e.clientX - startX;
    offsetY = e.clientY - startY;
    element.style.transform = `translate(${offsetX}px, ${offsetY}px)`;
  }

  function onMouseUp() {
    isDragging = false;
    document.removeEventListener("mousemove", onMouseMove);
    document.removeEventListener("mouseup", onMouseUp);
  }

  element.addEventListener("mousedown", onMouseDown);

  // Return a cleanup function
  return function destroy() {
    element.removeEventListener("mousedown", onMouseDown);
    document.removeEventListener("mousemove", onMouseMove);
    document.removeEventListener("mouseup", onMouseUp);
  };
}

const cleanup = createDragHandler(document.getElementById("draggable"));
// All drag state is private. Call cleanup() to remove all listeners.
```

### Configuration Pipeline with Closures

```js
function createApiClient(baseConfig) {
  const interceptors = [];

  function use(interceptor) {
    interceptors.push(interceptor);
  }

  async function request(path, options = {}) {
    let config = {
      ...baseConfig,
      ...options,
      url: `${baseConfig.baseUrl}${path}`,
      headers: { ...baseConfig.headers, ...options.headers }
    };

    // Apply interceptors (each is a closure that may capture its own state)
    for (const interceptor of interceptors) {
      config = await interceptor(config);
    }

    const response = await fetch(config.url, config);
    return response.json();
  }

  return { use, request };
}

const api = createApiClient({
  baseUrl: "https://api.example.com",
  headers: { "Content-Type": "application/json" }
});

// Each interceptor is a closure with its own captured state
api.use((config) => {
  const token = localStorage.getItem("token");
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

api.use((config) => {
  console.log(`[${new Date().toISOString()}] ${config.url}`);
  return config;
});

api.request("/users"); // both interceptors are applied
```

> **Key Takeaway:** Scope defines where variables live, the scope chain determines how they are found, and closures allow functions to carry their defining scope with them. Understanding the interplay between lexical environments, the shared scope problem, and garbage collection is what separates developers who use closures from those who master them.
