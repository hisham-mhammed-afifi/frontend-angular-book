# 5.2 Data Types and Type Coercion

> JavaScript is dynamically and weakly typed. Understanding the eight data types, how the engine converts between them, and where implicit coercion hides bugs is critical to writing predictable code.

---

## The Eight Data Types

JavaScript has 7 primitive types and 1 structural type:

| Type | Example | `typeof` Result |
|---|---|---|
| String | `"hello"` | `"string"` |
| Number | `42`, `3.14`, `NaN`, `Infinity` | `"number"` |
| BigInt | `42n` | `"bigint"` |
| Boolean | `true`, `false` | `"boolean"` |
| Symbol | `Symbol("id")` | `"symbol"` |
| undefined | `undefined` | `"undefined"` |
| null | `null` | `"object"` (bug) |
| Object | `{}`, `[]`, `function(){}`, `new Date()` | `"object"` or `"function"` |

Primitives are immutable. When you call `"hello".toUpperCase()`, JavaScript creates a temporary wrapper object, calls the method, and discards the wrapper. The original string is never modified.

## `typeof` Operator

`typeof` returns one of exactly 8 strings:

```javascript
typeof "hello"        // "string"
typeof 42             // "number"
typeof 42n            // "bigint"
typeof true           // "boolean"
typeof Symbol("x")    // "symbol"
typeof undefined      // "undefined"
typeof {}             // "object"
typeof function() {}  // "function"
```

### `typeof` Quirks

**The null bug:**

```javascript
typeof null // "object"
```

This is a historical bug from the first JavaScript implementation in 1995. Values were stored as 32-bit units with small type tags. Objects had the tag `000`. `null` was represented as the NULL pointer (`0x00`), so its type tag bits were all zeros, matching the object tag. A fix was proposed but rejected because it would break existing websites.

**Check for null explicitly:**

```javascript
// BAD: typeof cannot distinguish null from objects
if (typeof value === "object") { /* could be null! */ }

// GOOD: Explicit null check
if (value !== null && typeof value === "object") { /* safe */ }
```

**NaN and Infinity are "number":**

```javascript
typeof NaN       // "number"
typeof Infinity  // "number"
typeof -Infinity // "number"
```

**Arrays have no distinct typeof:**

```javascript
typeof []         // "object"
typeof new Date() // "object"
typeof /regex/    // "object"

// Use Array.isArray() for arrays
Array.isArray([])         // true
Array.isArray({length:0}) // false
```

**Classes are functions:**

```javascript
class Foo {}
typeof Foo // "function"
```

**Wrapper objects are objects, not primitives:**

```javascript
typeof "hello"            // "string"
typeof new String("hello") // "object" (wrapper object)

// Never use wrapper constructors with new
// BAD
const str = new String("hello");
const num = new Number(42);
const bool = new Boolean(true);

// GOOD (these are type conversion functions, not constructors)
const str2 = String(42);     // "42"
const num2 = Number("42");   // 42
const bool2 = Boolean(1);    // true
```

## Number Type Deep Dive

JavaScript uses IEEE 754 double-precision 64-bit floating point for all numbers. This gives approximately 15-17 significant decimal digits of precision.

### Special Numeric Values

```javascript
Number.MAX_SAFE_INTEGER  // 9007199254740991 (2^53 - 1)
Number.MIN_SAFE_INTEGER  // -9007199254740991
Number.MAX_VALUE         // ~1.7976931348623157e+308
Number.MIN_VALUE         // ~5e-324 (smallest positive value)
Number.POSITIVE_INFINITY // Infinity
Number.NEGATIVE_INFINITY // -Infinity
Number.NaN               // NaN
Number.EPSILON           // ~2.22e-16 (smallest difference between two representable numbers)
```

### Floating Point Precision

```javascript
0.1 + 0.2                // 0.30000000000000004
0.1 + 0.2 === 0.3        // false
0.1 * 3 === 0.3          // false
0.1 * 10 === 1           // true (rounding happens to cancel out)
```

This is not a JavaScript bug. It affects every language using IEEE 754. The number `0.1` cannot be represented exactly in binary, just as `1/3` cannot be represented exactly in decimal.

**Comparison using epsilon (for values near magnitude 1):**

```javascript
function nearlyEqual(a, b, epsilon = Number.EPSILON) {
  const diff = Math.abs(a - b);
  if (diff < epsilon) return true;

  // Scaled epsilon for larger numbers
  return diff <= epsilon * Math.max(Math.abs(a), Math.abs(b));
}

nearlyEqual(0.1 + 0.2, 0.3); // true
```

**Integer arithmetic for currency:**

```javascript
// BAD: Floating point in financial calculations
const price = 0.1 + 0.2; // 0.30000000000000004

// GOOD: Compute in cents (integers), display in dollars
const priceInCents = 10 + 20; // 30
const display = (priceInCents / 100).toFixed(2); // "0.30"
```

### Safe Integer Boundaries

```javascript
Number.MAX_SAFE_INTEGER + 1 === Number.MAX_SAFE_INTEGER + 2 // true (!)
9007199254740993 // stored as 9007199254740992 (precision lost)

// Guard against this with large API IDs
Number.isSafeInteger(9007199254740991) // true
Number.isSafeInteger(9007199254740992) // false
```

Large IDs from APIs (like Snowflake IDs) lose precision when parsed as JSON numbers. Solutions: transmit them as strings or use BigInt.

### Negative Zero

```javascript
const negZero = -0;
negZero === 0              // true (indistinguishable with ===)
String(negZero)            // "0" (sign is lost)
JSON.stringify(negZero)    // "0"
negZero.toString()         // "0"

// Detection methods
Object.is(negZero, -0)    // true
Object.is(negZero, 0)     // false
1 / negZero === -Infinity  // true (classic detection trick)
```

## NaN

NaN stands for "Not-a-Number" but `typeof NaN === "number"`. It represents an undefined or unrepresentable numeric result.

### How NaN is Produced

```javascript
0 / 0              // NaN
parseInt("abc")    // NaN
Math.sqrt(-1)      // NaN
Number(undefined)  // NaN
Number("hello")    // NaN
undefined + 1      // NaN
```

### NaN is Not Equal to Itself

```javascript
NaN === NaN // false
NaN == NaN  // false
NaN !== NaN // true (the only value where this is true)
```

This follows the IEEE 754 specification: NaN represents an indeterminate value, and two indeterminate values are not necessarily equal.

### `isNaN()` vs `Number.isNaN()`

The global `isNaN()` coerces its argument to a number first, then checks if the result is NaN. This produces misleading results. `Number.isNaN()` does NOT coerce and only returns `true` for actual NaN values.

| Expression | `isNaN()` | `Number.isNaN()` |
|---|---|---|
| `NaN` | `true` | `true` |
| `"hello"` | `true` | `false` |
| `undefined` | `true` | `false` |
| `{}` | `true` | `false` |
| `"NaN"` | `true` | `false` |
| `null` | `false` | `false` |
| `""` | `false` | `false` |
| `"37"` | `false` | `false` |
| `true` | `false` | `false` |
| `37` | `false` | `false` |

```javascript
// BAD: Global isNaN coerces, giving false positives
isNaN("hello")     // true (Number("hello") is NaN)
isNaN(undefined)   // true (Number(undefined) is NaN)

// GOOD: Number.isNaN does not coerce
Number.isNaN("hello")   // false (not a number type at all)
Number.isNaN(undefined) // false
Number.isNaN(NaN)       // true
Number.isNaN(0 / 0)     // true
```

**Self-inequality trick (pre-ES6 NaN check):**

```javascript
// NaN is the only value not equal to itself
function isReallyNaN(value) {
  return value !== value;
}
```

## BigInt

BigInt provides arbitrary-precision integer arithmetic. Created with the `n` suffix or `BigInt()` constructor.

```javascript
const big = 9007199254740993n; // Precise (unlike Number)
typeof big // "bigint"
```

### Mixing Restrictions

BigInt and Number cannot be mixed in arithmetic operations:

```javascript
// 1n + 1;    // TypeError: Cannot mix BigInt and other types
// 1n * 2;    // TypeError

// Explicit conversion required
1n + BigInt(1)  // 2n
Number(1n) + 1  // 2 (loses precision for very large values)
```

### Comparison Works Across Types

```javascript
1n === 1   // false (different types)
1n == 1    // true (loose equality coerces)
2n > 1     // true
1n < 1.5   // true
0n === 0   // false
0n == 0    // true
```

### Key Limitations

```javascript
// Division truncates (integer division)
5n / 2n    // 2n (not 2.5n)

// No unary +
// +1n      // TypeError

// No Math support
// Math.max(1n, 2n) // TypeError

// No JSON support
// JSON.stringify(1n) // TypeError

// Cannot create from non-integers
// BigInt(1.5)    // RangeError
// BigInt(NaN)    // RangeError
// BigInt("1.5")  // SyntaxError

// Precision loss if converting from Number
BigInt(Number.MAX_SAFE_INTEGER + 1) // 9007199254740992n (already lost precision)
BigInt("9007199254740993")          // 9007199254740993n (correct: use string)
```

### JSON Serialization Workaround

```javascript
const data = { id: 9007199254740993n, name: "Alice" };

// Serialize with replacer
const json = JSON.stringify(data, (key, value) =>
  typeof value === "bigint" ? value.toString() : value
);
// '{"id":"9007199254740993","name":"Alice"}'

// Deserialize with reviver (if you know which fields are BigInt)
const parsed = JSON.parse(json, (key, value) =>
  key === "id" ? BigInt(value) : value
);
```

### Boolean Context

```javascript
if (0n)  { /* does NOT execute, 0n is falsy */ }
if (1n)  { /* executes, non-zero BigInt is truthy */ }
Boolean(0n)  // false
Boolean(1n)  // true
```

## Symbol

Every `Symbol()` call returns a guaranteed unique value. Symbols serve as collision-free property keys and metaprogramming hooks.

```javascript
const s1 = Symbol("description");
const s2 = Symbol("description");
s1 === s2 // false (always unique)
```

### Symbol Registry

```javascript
// Global registry: shared across realms
const s1 = Symbol.for("app.id");
const s2 = Symbol.for("app.id");
s1 === s2 // true (same key in the global registry)

Symbol.keyFor(s1) // "app.id"
Symbol.keyFor(Symbol("local")) // undefined (not in registry)
```

### Symbols as Property Keys

```javascript
const ID = Symbol("id");
const user = {
  [ID]: 123,
  name: "Alice"
};

user[ID]          // 123
user.name         // "Alice"

// Symbols are NOT enumerable
Object.keys(user)                   // ["name"]
JSON.stringify(user)                // '{"name":"Alice"}'
for (const key in user) { /* only "name" */ }

// Access symbol properties explicitly
Object.getOwnPropertySymbols(user)  // [Symbol(id)]
Reflect.ownKeys(user)               // ["name", Symbol(id)]
```

### Well-Known Symbols

```javascript
// Symbol.toPrimitive: customize type conversion
const price = {
  amount: 99.99,
  currency: "USD",

  [Symbol.toPrimitive](hint) {
    if (hint === "number") return this.amount;
    if (hint === "string") return `${this.currency} ${this.amount}`;
    return this.amount; // default hint
  }
};

+price          // 99.99 (number hint)
`${price}`      // "USD 99.99" (string hint)
price + 0       // 99.99 (default hint)

// Symbol.iterator: make any object iterable
const range = {
  start: 1,
  end: 5,

  [Symbol.iterator]() {
    let current = this.start;
    const end = this.end;
    return {
      next() {
        return current <= end
          ? { value: current++, done: false }
          : { done: true };
      }
    };
  }
};

[...range]           // [1, 2, 3, 4, 5]
for (const n of range) { /* 1, 2, 3, 4, 5 */ }
```

### Coercion Rules

```javascript
// Symbols CANNOT be implicitly coerced
// "" + Symbol("x")   // TypeError
// +Symbol("x")       // TypeError

// Explicit conversion works
String(Symbol("x"))         // "Symbol(x)"
Symbol("x").toString()      // "Symbol(x)"
Symbol("x").description     // "x" (ES2019+)
Boolean(Symbol("x"))        // true (all symbols are truthy)
```

## Falsy and Truthy Values

### The Complete Falsy List

There are exactly 9 falsy values in JavaScript. Everything else is truthy.

| Value | Type | Notes |
|---|---|---|
| `false` | Boolean | |
| `0` | Number | Positive zero |
| `-0` | Number | Negative zero |
| `0n` | BigInt | BigInt zero |
| `""` | String | Empty string only |
| `null` | null | |
| `undefined` | undefined | |
| `NaN` | Number | |
| `document.all` | Object | The only falsy object (legacy browser compat) |

### Truthy Surprises

```javascript
// All of these are TRUTHY
Boolean([])             // true (empty array)
Boolean({})             // true (empty object)
Boolean("0")            // true (non-empty string)
Boolean("false")        // true (non-empty string)
Boolean(" ")            // true (space is non-empty)
Boolean(new Boolean(false)) // true (wrapper object!)
Boolean(new Number(0))      // true (wrapper object!)
Boolean(new String(""))     // true (wrapper object!)
Boolean(-1)             // true (non-zero)
Boolean(Infinity)       // true
Boolean(-Infinity)      // true
Boolean([0])            // true (non-empty array)
```

### The `[]` Paradox

```javascript
// [] is truthy
if ([]) { console.log("truthy"); } // "truthy"

// But [] == false is true (coercion chain)
[] == false   // true
// Step 1: false -> 0 (ToNumber on boolean)
// Step 2: [] -> "" (ToPrimitive: valueOf returns [], toString returns "")
// Step 3: "" -> 0 (ToNumber on string)
// Step 4: 0 == 0 -> true
```

### `||` vs `??` (Nullish Coalescing)

```javascript
// || returns the first truthy operand (or the last operand)
0 || "default"      // "default" (0 is falsy)
"" || "default"     // "default" ("" is falsy)
false || "default"  // "default" (false is falsy)
null || "default"   // "default"

// ?? returns the first non-nullish operand (null/undefined only)
0 ?? "default"      // 0 (0 is not null/undefined)
"" ?? "default"     // "" ("" is not null/undefined)
false ?? "default"  // false
null ?? "default"   // "default"
undefined ?? "default" // "default"
```

```javascript
// BAD: || treats 0 and "" as "missing"
function getPort(port) {
  return port || 3000; // Port 0 would return 3000!
}

// GOOD: ?? only falls through on null/undefined
function getPort(port) {
  return port ?? 3000;
}

getPort(0)         // 0 (correct)
getPort(undefined) // 3000
getPort(null)      // 3000
```

## Implicit Type Coercion

### The `+` Operator: Addition or Concatenation?

If either operand is a string (or coerces to a string via ToPrimitive), `+` performs string concatenation. Otherwise, it performs numeric addition.

```javascript
1 + 2          // 3 (both numbers: addition)
"1" + 2        // "12" (string + anything: concatenation)
1 + "2"        // "12"
"1" + "2"      // "12"
true + true    // 2 (both coerce to numbers)
true + "1"     // "true1" (string wins)
null + 1       // 1 (null -> 0)
undefined + 1  // NaN (undefined -> NaN)
[] + []        // "" (both coerce to empty string)
[] + {}        // "[object Object]"
{} + []        // 0 (in console: {} is parsed as empty block, becomes +[])
({} + [])      // "[object Object]" (grouped as expression)
```

### Arithmetic Operators (Always Numeric)

`-`, `*`, `/`, `%` always invoke ToNumber on both operands:

```javascript
"5" - 3     // 2
"5" * "3"   // 15
"10" / "2"  // 5
"10" % "3"  // 1
true - 1    // 0
null * 5    // 0 (null -> 0)
```

### Unary `+` (ToNumber)

```javascript
+"42"       // 42
+""         // 0
+" "        // 0 (whitespace -> 0)
+true       // 1
+false      // 0
+null       // 0
+undefined  // NaN
+[]         // 0 ([] -> "" -> 0)
+[1]        // 1 ([1] -> "1" -> 1)
+[1,2]      // NaN ([1,2] -> "1,2" -> NaN)
+{}         // NaN ({} -> "[object Object]" -> NaN)
```

### Template Literals (Always ToString)

```javascript
`${null}`      // "null"
`${undefined}` // "undefined"
`${true}`      // "true"
`${42}`        // "42"
`${[1,2]}`     // "1,2"
`${{a:1}}`     // "[object Object]"
`${-0}`        // "0" (sign is lost)
```

## The Abstract Operations Behind Coercion

When the engine needs to convert a value, it invokes internal abstract operations.

### ToNumber

| Input | Result |
|---|---|
| `undefined` | `NaN` |
| `null` | `0` |
| `true` | `1` |
| `false` | `0` |
| `""` (empty string) | `0` |
| `"  "` (whitespace) | `0` |
| `"123"` | `123` |
| `"123abc"` | `NaN` |
| `"0x1A"` | `26` (hex) |
| `Symbol()` | `TypeError` |
| Object | ToPrimitive("number"), then ToNumber |

### ToString

| Input | Result |
|---|---|
| `undefined` | `"undefined"` |
| `null` | `"null"` |
| `true` | `"true"` |
| `false` | `"false"` |
| `0` | `"0"` |
| `-0` | `"0"` (sign lost) |
| `NaN` | `"NaN"` |
| `Infinity` | `"Infinity"` |
| Object | ToPrimitive("string"), then ToString |

### ToBoolean

ToBoolean is a direct lookup. It does NOT call ToPrimitive. The 9 falsy values produce `false`. Everything else produces `true`. No conversion chain occurs.

### ToPrimitive

When an operation expects a primitive but receives an object, ToPrimitive is invoked with a "hint":

1. If the object has `[Symbol.toPrimitive](hint)`, call it. Done.
2. If hint is `"string"`: try `toString()`, then `valueOf()`.
3. If hint is `"number"` or `"default"`: try `valueOf()`, then `toString()`.

```javascript
// Custom ToPrimitive
const temp = {
  celsius: 100,
  [Symbol.toPrimitive](hint) {
    if (hint === "number") return this.celsius;
    if (hint === "string") return `${this.celsius}C`;
    return this.celsius; // default
  }
};

+temp          // 100
`${temp}`      // "100C"
temp + 0       // 100 (default hint -> number)
temp + ""      // "100" (default hint -> number -> string concat)
```

**Why `+[]` is `0`:**

```javascript
// Step 1: Unary + invokes ToNumber on []
// Step 2: ToNumber on object calls ToPrimitive([], "number")
// Step 3: ToPrimitive: [].valueOf() returns [] (not primitive, skip)
// Step 4: ToPrimitive: [].toString() returns "" (primitive, use it)
// Step 5: ToNumber("") returns 0
+[] // 0
```

**Why `+{}` is `NaN`:**

```javascript
// Step 1: Unary + invokes ToNumber on {}
// Step 2: ToPrimitive({}, "number")
// Step 3: {}.valueOf() returns {} (not primitive, skip)
// Step 4: {}.toString() returns "[object Object]" (primitive, use it)
// Step 5: ToNumber("[object Object]") returns NaN
+{} // NaN
```

## Equality Comparisons

### The Four Equality Algorithms

| Algorithm | Coerces? | NaN === NaN? | +0 === -0? | Used By |
|---|---|---|---|---|
| `==` (Abstract) | Yes | No | Yes | `==`, `!=` |
| `===` (Strict) | No | No | Yes | `===`, `!==`, `indexOf`, `switch` |
| `Object.is` (SameValue) | No | Yes | No | `Object.is()` |
| SameValueZero | No | Yes | Yes | `Set`, `Map`, `includes()` |

### Abstract Equality (`==`) Algorithm

The key rules:
1. Same type: use `===`.
2. `null == undefined`: `true`. Neither coerces to anything else.
3. Number vs String: convert String to Number.
4. Boolean vs anything: convert Boolean to Number, then restart.
5. Object vs primitive: call ToPrimitive on Object, then restart.

```javascript
null == undefined   // true (special rule)
null == 0           // false (null only equals null or undefined)
null == ""          // false
null == false       // false
undefined == 0      // false
undefined == ""     // false
undefined == false  // false

"" == 0             // true (ToNumber("") is 0)
"  " == 0           // true (ToNumber("  ") is 0)
"0" == false        // true (false -> 0, "0" -> 0)
"1" == true         // true (true -> 1, "1" -> 1)
"2" == true         // false (true -> 1, "2" -> 2, 2 !== 1)
```

### The `[] == ![]` Puzzle

```javascript
[] == ![]  // true (!)
```

Step-by-step:
1. `![]` is evaluated first. `[]` is truthy, so `![]` is `false`.
2. Now we have `[] == false`.
3. Boolean on one side: `false` -> `0`. Now: `[] == 0`.
4. Object vs Number: ToPrimitive(`[]`) -> `""`. Now: `"" == 0`.
5. String vs Number: `ToNumber("")` -> `0`. Now: `0 == 0`.
6. Result: `true`.

### The `null >= 0` Paradox

```javascript
null == 0   // false (null only equals null/undefined)
null > 0    // false (null -> 0, 0 > 0 is false)
null >= 0   // true (null -> 0, 0 >= 0 is true)
```

`==` has special rules for null. But `>`, `<`, `>=`, `<=` use ToNumber directly, where `Number(null)` is `0`.

### `Object.is()`

Differs from `===` in exactly two cases:

```javascript
Object.is(NaN, NaN)  // true (=== returns false)
Object.is(+0, -0)    // false (=== returns true)
Object.is(-0, +0)    // false

// All other cases match ===
Object.is(1, 1)           // true
Object.is("a", "a")       // true
Object.is(null, null)     // true
Object.is(null, undefined) // false
```

### SameValueZero in Built-ins

```javascript
// Array.includes uses SameValueZero (NaN-aware)
[NaN].includes(NaN)   // true

// Array.indexOf uses Strict Equality (not NaN-aware)
[NaN].indexOf(NaN)    // -1

// Set and Map use SameValueZero
const set = new Set([NaN, NaN]);
set.size               // 1
set.has(NaN)           // true

const map = new Map([[NaN, "a"], [NaN, "b"]]);
map.size               // 1
map.get(NaN)           // "b" (second set overwrites first)
```

## `parseInt()` vs `Number()`

These are different conversion functions with different behavior:

| Input | `Number()` | `parseInt()` | `parseFloat()` |
|---|---|---|---|
| `"123"` | `123` | `123` | `123` |
| `"123abc"` | `NaN` | `123` | `123` |
| `""` | `0` | `NaN` | `NaN` |
| `" "` | `0` | `NaN` | `NaN` |
| `"0x1A"` | `26` | `26` | `0` |
| `"010"` | `10` | `10` | `10` |
| `true` | `1` | `NaN` | `NaN` |
| `null` | `0` | `NaN` | `NaN` |
| `undefined` | `NaN` | `NaN` | `NaN` |
| `"3.14"` | `3.14` | `3` | `3.14` |
| `"  42  "` | `42` | `42` | `42` |

```javascript
// parseInt parses character by character, stops at first non-numeric
parseInt("100px")    // 100
parseInt("$100")     // NaN (first character is not a digit)

// Always specify the radix
parseInt("08")       // 8 (modern engines default to base 10)
parseInt("08", 10)   // 8 (explicit: always do this)
parseInt("0xFF", 16) // 255

// Number() is stricter: the entire string must be valid
Number("100px")      // NaN
Number("  42  ")     // 42 (trims whitespace)
```

## Common Mistakes

### 1. Using `==` Instead of `===`

```javascript
// BAD: Coercion produces surprising results
"" == 0       // true
"0" == false  // true
[] == false   // true
"2" == true   // false (would expect true)

// GOOD: Strict equality, no surprises
"" === 0      // false
"0" === false // false
```

**Exception:** `value == null` is a legitimate shorthand for `value === null || value === undefined`.

### 2. Using Global `isNaN()` Instead of `Number.isNaN()`

```javascript
// BAD: Coerces before checking
isNaN("hello")  // true (but "hello" is not NaN, it is a string)

// GOOD: No coercion
Number.isNaN("hello") // false
Number.isNaN(NaN)     // true
```

### 3. Trusting `typeof` for Null and Arrays

```javascript
// BAD
typeof null === "object"  // true (bug)
typeof [] === "array"     // false (no such typeof result)

// GOOD
value === null
Array.isArray(value)
```

### 4. Using `||` When `??` is Needed

```javascript
// BAD: Valid falsy values are treated as missing
const count = input.count || 10; // count of 0 returns 10

// GOOD: Only null/undefined trigger the default
const count = input.count ?? 10; // count of 0 returns 0
```

### 5. Mixing BigInt and Number

```javascript
// BAD: Throws TypeError
// const result = 1n + 1;

// GOOD: Explicit conversion
const result = 1n + BigInt(1); // 2n
```

### 6. Losing Precision with Large IDs

```javascript
// BAD: Precision lost when parsing JSON
const json = '{"id": 9007199254740993}';
JSON.parse(json).id // 9007199254740992 (wrong!)

// GOOD: Transmit large IDs as strings
const json2 = '{"id": "9007199254740993"}';
BigInt(JSON.parse(json2).id) // 9007199254740993n (correct)
```

### 7. The `+` Operator Ambiguity

```javascript
// BAD: Concatenation when you wanted addition
const total = "5" + 3;  // "53"
const count = prompt("Enter number:") + 1; // string concatenation!

// GOOD: Explicit conversion
const total = Number("5") + 3;  // 8
const count = Number(prompt("Enter number:")) + 1;
```

---

## Best Practices

1. **Use `===` by default.** Only use `==` intentionally for `value == null` checks.

2. **Use `Number.isNaN()` and `Number.isFinite()`.** Avoid the global `isNaN()` and `isFinite()` which coerce.

3. **Use `??` over `||` for defaults** when `0`, `""`, or `false` are valid values.

4. **Prefer explicit conversion.** `Number(x)`, `String(x)`, `Boolean(x)` signal intent clearly.

5. **Use `Array.isArray()` for array checks.** `typeof` returns `"object"` for arrays.

6. **Always pass a radix to `parseInt()`.** `parseInt(str, 10)` prevents ambiguity.

7. **Use integer arithmetic for money.** Store amounts in cents/pence as integers.

8. **Use BigInt or string IDs for large numbers.** Anything beyond `Number.MAX_SAFE_INTEGER` will lose precision silently.

## Real-World Use Cases

### Input Validation Utility

```javascript
function validateNumericInput(value) {
  // Reject null, undefined, empty string
  if (value == null || value === "") {
    return { valid: false, error: "Value is required" };
  }

  const num = Number(value);

  if (Number.isNaN(num)) {
    return { valid: false, error: "Not a valid number" };
  }

  if (!Number.isFinite(num)) {
    return { valid: false, error: "Value must be finite" };
  }

  return { valid: true, value: num };
}

validateNumericInput("42")        // { valid: true, value: 42 }
validateNumericInput("3.14")      // { valid: true, value: 3.14 }
validateNumericInput("abc")       // { valid: false, error: "Not a valid number" }
validateNumericInput("")          // { valid: false, error: "Value is required" }
validateNumericInput(null)        // { valid: false, error: "Value is required" }
validateNumericInput("Infinity")  // { valid: false, error: "Value must be finite" }
```

### Type-Safe API Response Parser

```javascript
function parseApiResponse(raw) {
  return {
    id: typeof raw.id === "string" ? raw.id : String(raw.id),
    name: String(raw.name ?? ""),
    count: Number(raw.count) || 0,
    price: Math.round(Number(raw.price) * 100), // Store as cents
    isActive: raw.isActive === true, // Strict boolean, no coercion
    tags: Array.isArray(raw.tags) ? raw.tags.map(String) : [],
    metadata: raw.metadata != null && typeof raw.metadata === "object"
      ? raw.metadata
      : {}
  };
}

const response = parseApiResponse({
  id: 12345,
  name: null,
  count: "7",
  price: "29.99",
  isActive: 1,        // Will be false (strict check)
  tags: ["js", 42],
  metadata: undefined
});

// { id: "12345", name: "", count: 7, price: 2999, isActive: false,
//   tags: ["js", "42"], metadata: {} }
```

### Robust Equality Comparison Utility

```javascript
function strictEqual(a, b) {
  // Handle NaN (the only value not equal to itself)
  if (a !== a && b !== b) return true;
  return a === b;
}

// Or for the strictest comparison (distinguishes -0 from +0)
function sameValue(a, b) {
  return Object.is(a, b);
}

// For collections/deduplication (matches Set/Map behavior)
function sameValueZero(a, b) {
  if (a === 0 && b === 0) return true; // +0 and -0 are same
  if (a !== a && b !== b) return true;  // NaN equals NaN
  return a === b;
}
```

> **Key Takeaway:** JavaScript has 8 data types and a complex implicit coercion system driven by abstract operations (ToPrimitive, ToNumber, ToString, ToBoolean). Use `===` by default, `Number.isNaN()` over `isNaN()`, `??` over `||` for defaults, and always convert types explicitly at system boundaries rather than relying on implicit coercion in business logic.
