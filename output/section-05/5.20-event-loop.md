# 5.20 Event Loop

> The event loop is the mechanism that allows JavaScript to perform non-blocking operations on a single thread by offloading work and scheduling callbacks through a system of task queues and microtask queues.

---

## The Single-Threaded Model

JavaScript executes on a single thread. The runtime consists of three core structures:

- **Call Stack**: A LIFO stack that tracks function execution. Every function call creates a stack frame (execution context). When a function returns, its frame is popped. Only one function ever executes at a time.
- **Heap**: Unstructured memory where objects are allocated and managed by the garbage collector.
- **Task Queues**: FIFO queues where callbacks from completed async operations wait to be moved to the call stack.

JavaScript is single-threaded, but the runtime is not. The browser (or Node.js) uses multiple threads for networking, file I/O, timers, and rendering. Only your JavaScript code runs on one thread.

### Run-to-Completion

Every task (function invocation from a queue) runs to completion before the next task starts. No other code can interrupt a function mid-execution:

```js
let x = 0;

function task() {
  x = 1;
  // No async callback can interrupt here, even if one is ready.
  x = 2;
}

task();
console.log(x); // Always 2, never 1
```

This guarantee simplifies reasoning about shared state but means any long-running synchronous code blocks everything else.

## The Event Loop Algorithm

The WHATWG HTML specification defines the event loop processing model. Simplified:

```
LOOP FOREVER:
  1. Pick the oldest task from a task queue (macrotask queue).
     If no tasks, skip to step 4.
  2. Execute that task (your JS code runs here).
  3. Set "currently running task" to null.
  4. Drain the microtask queue:
     - While the microtask queue is not empty:
       - Dequeue and execute the oldest microtask.
       - (If that microtask enqueues more microtasks, they run NOW too.)
  5. If this is a "rendering opportunity":
     a. Run requestAnimationFrame callbacks.
     b. Run style calculations, layout, paint, composite.
     c. Run ResizeObserver callbacks.
     d. Run IntersectionObserver callbacks.
  6. If the event loop has nothing to do:
     - Run requestIdleCallback callbacks.
  7. Go to step 1.
```

The initial `<script>` execution is itself a macrotask. That is the very first thing the event loop processes.

The spec says "rendering opportunity" does NOT happen after every task. Browsers typically target 60fps (~16.67ms per frame) but skip rendering if nothing changed or if the tab is in the background.

## The Call Stack

Each function call creates an **execution context** pushed onto the call stack. It contains:
- The function's local variables and arguments
- The `this` binding
- A reference to the outer lexical environment (for closures)
- The return address (where to resume after the function returns)

```js
function multiply(a, b) {
  return a * b;
}

function square(n) {
  return multiply(n, n); // Pushes multiply's frame on top
}

function printSquare(n) {
  const result = square(n); // Pushes square's frame
  console.log(result);
}

printSquare(5);
// Call stack at deepest point:
// | multiply(5, 5) |  <-- top
// | square(5)      |
// | printSquare(5)  |
// | (global)        |  <-- bottom
```

### Stack Overflow

The call stack has a finite size (roughly 10,000-25,000 frames in V8). Exceeding it throws `RangeError: Maximum call stack size exceeded`:

```js
function recurse() {
  recurse(); // No base case
}
recurse(); // RangeError: Maximum call stack size exceeded
```

For deep recursion, use a trampoline pattern to convert recursive calls into iteration:

```js
function trampoline(fn) {
  return function (...args) {
    let result = fn(...args);
    while (typeof result === 'function') {
      result = result();
    }
    return result;
  };
}

const safeSum = trampoline(function sum(n, acc = 0) {
  if (n === 0) return acc;
  return () => sum(n - 1, acc + n); // Returns a thunk, not a recursive call
});

safeSum(100000); // Works fine
```

## Task Queue (Macrotask Queue)

The task queue holds callbacks from:

| Source | Example |
|--------|---------|
| Timers | `setTimeout`, `setInterval` |
| I/O | Network responses, file reads |
| UI Events | `click`, `keydown`, `scroll`, `resize` |
| MessageChannel | `port.postMessage()` |
| `postMessage` | `window.postMessage()` |
| `setImmediate` | Node.js only |

**Only ONE macrotask is dequeued per event loop iteration.** After executing it, the engine drains all microtasks, potentially renders, and only then picks the next macrotask.

```js
setTimeout(() => console.log('task 1'), 0);
setTimeout(() => console.log('task 2'), 0);

// Between task 1 and task 2:
// - All microtasks are drained
// - A render may occur
```

The spec allows multiple task queues with different priorities. Browsers give user input events higher priority than timer callbacks. A click handler can run before a `setTimeout(fn, 0)` that was scheduled earlier.

---

## Microtask Queue

The microtask queue has higher priority than the task queue. After every task and after every microtask that generates new microtasks, the engine drains the entire microtask queue.

| Source | Environment |
|--------|-------------|
| `Promise.then/catch/finally` | Browser + Node.js |
| `queueMicrotask()` | Browser + Node.js |
| `MutationObserver` | Browser only |
| `process.nextTick()` | Node.js only (even higher priority) |
| `await` continuations | Browser + Node.js |

```js
console.log('script start');

setTimeout(() => console.log('setTimeout'), 0);

Promise.resolve()
  .then(() => console.log('promise 1'))
  .then(() => console.log('promise 2'));

queueMicrotask(() => console.log('queueMicrotask'));

console.log('script end');

// Output:
// script start
// script end
// promise 1
// queueMicrotask
// promise 2
// setTimeout
```

### queueMicrotask vs Promise.resolve().then()

Both create microtasks, but `queueMicrotask` is cleaner and avoids creating a throwaway Promise:

```js
// Preferred
queueMicrotask(() => {
  batchUpdate();
});

// Wasteful: creates an unnecessary Promise object
Promise.resolve().then(() => {
  batchUpdate();
});
```

## Execution Order

The complete priority order for one event loop iteration:

```
1. Synchronous code (the currently executing task)
2. Microtasks (ALL of them, including newly enqueued ones)
3. requestAnimationFrame callbacks (if rendering opportunity)
4. Style calculation, Layout, Paint, Composite (if rendering opportunity)
5. ResizeObserver / IntersectionObserver callbacks
6. requestIdleCallback (if idle time remains)
7. Next macrotask (back to step 1)
```

```js
console.log('1: sync');

setTimeout(() => console.log('6: macrotask'), 0);

requestAnimationFrame(() => console.log('5: rAF'));

Promise.resolve().then(() => {
  console.log('3: microtask 1');
  queueMicrotask(() => console.log('4: microtask 2 (nested)'));
});

console.log('2: sync');

// Typical output: 1, 2, 3, 4, 5, 6
// (rAF timing relative to macrotask depends on rendering opportunity)
```

## requestAnimationFrame

`requestAnimationFrame` (rAF) registers a callback to run before the next repaint. It is neither a macrotask nor a microtask. It runs in its own queue during the rendering step.

```
[Task] -> [Microtasks] -> [rAF callbacks] -> [Style/Layout/Paint] -> [Next Task]
```

| Aspect | Behavior |
|--------|----------|
| Frequency | Matches display refresh rate (60Hz = ~16.67ms) |
| Background tabs | Paused or heavily throttled |
| Cancellation | `cancelAnimationFrame(id)` |
| Callback argument | `DOMHighResTimeStamp` |
| Batching | All rAFs registered in one frame run in the SAME frame |

```js
function animate(timestamp) {
  element.style.transform = `translateX(${timestamp * 0.1}px)`;
  requestAnimationFrame(animate);
}
requestAnimationFrame(animate);
```

**Gotcha:** rAF callbacks scheduled inside a rAF callback run in the NEXT frame, not the current one:

```js
requestAnimationFrame(() => {
  console.log('frame 1');
  requestAnimationFrame(() => {
    console.log('frame 2'); // Next frame, not this one
  });
});
```

### rAF for Coalescing Updates

```js
let ticking = false;
window.addEventListener('scroll', () => {
  if (!ticking) {
    requestAnimationFrame(() => {
      element.style.transform = `translateY(${window.scrollY}px)`;
      ticking = false;
    });
    ticking = true;
  }
});
```

## requestIdleCallback

`requestIdleCallback` queues a function to run during idle periods, after all tasks and rendering are complete.

```js
requestIdleCallback((deadline) => {
  // deadline.timeRemaining() returns ms left in this idle period (max 50ms)
  // deadline.didTimeout is true if the timeout option expired

  while (deadline.timeRemaining() > 0 && workQueue.length > 0) {
    processWorkItem(workQueue.shift());
  }

  // Reschedule if more work remains
  if (workQueue.length > 0) {
    requestIdleCallback(processWork);
  }
});

// With a maximum timeout (forces execution even during busy periods)
requestIdleCallback(doAnalytics, { timeout: 2000 });
```

Use cases: analytics reporting, lazy-loading non-critical resources, pre-caching, logging.

Not suitable for animations or time-sensitive work. It runs at the lowest priority and may be delayed indefinitely.

## setTimeout(fn, 0) Behavior

`setTimeout(fn, 0)` does NOT mean "run immediately." It means "run after the current task, all microtasks, and any rendering, subject to clamping rules."

### The 4ms Clamping Rule

After 5 levels of nesting, `setTimeout` is clamped to a minimum of 4ms:

```js
setTimeout(() => {                  // Level 1: ~0ms
  setTimeout(() => {                // Level 2: ~0ms
    setTimeout(() => {              // Level 3: ~0ms
      setTimeout(() => {            // Level 4: ~0ms
        setTimeout(() => {          // Level 5+: CLAMPED to 4ms
          console.log('delayed');
        }, 0);
      }, 0);
    }, 0);
  }, 0);
}, 0);
```

### Additional Throttling

| Condition | Minimum Delay |
|-----------|---------------|
| Foreground tab, nesting < 5 | ~0-1ms |
| Foreground tab, nesting >= 5 | 4ms |
| Background tab (Chrome, after 30s) | 10,000ms (10 seconds) |
| Cross-origin iframe | May be throttled to 100ms |
| Page hidden (`visibilitychange`) | Heavily throttled |

### MessageChannel: Avoiding the 4ms Clamp

`MessageChannel` creates macrotasks without the nested timer clamping. This is what React uses internally for scheduling:

```js
function scheduleTask(callback) {
  const channel = new MessageChannel();
  channel.port1.onmessage = callback;
  channel.port2.postMessage(null);
}

// Each call creates a macrotask with NO 4ms clamp
scheduleTask(() => console.log('immediate macrotask'));
```

## The Rendering Pipeline

The browser's rendering pipeline runs during the "update the rendering" step:

```
JavaScript -> Style -> Layout -> Paint -> Composite
```

| Stage | What Happens | Thread |
|-------|-------------|--------|
| JavaScript | rAF callbacks, event handlers | Main thread |
| Style | Match CSS selectors, compute final styles | Main thread |
| Layout (Reflow) | Calculate element sizes and positions | Main thread |
| Paint | Fill in pixels (text, colors, borders, shadows) | Main thread |
| Composite | Combine layers, apply transforms, send to GPU | Compositor thread |

### Layout Thrashing

Reading layout properties after writing style changes forces the browser to run layout synchronously:

```js
// BAD: Layout thrashing (forced synchronous layout)
for (let i = 0; i < 1000; i++) {
  element.style.width = element.offsetWidth + 1 + 'px';
  // Each iteration: write -> FORCE layout -> read -> repeat
}

// GOOD: Batch reads, then batch writes
const width = element.offsetWidth; // Read once
for (let i = 0; i < 1000; i++) {
  element.style.width = (width + i) + 'px'; // Write only
}
```

### What Triggers Each Stage

| Layout (expensive) | Paint (moderate) | Composite only (cheap) |
|-------------------|-----------------|----------------------|
| `width`, `height` | `color` | `transform` |
| `margin`, `padding` | `background` | `opacity` |
| `top`, `left` | `box-shadow` | `will-change` |
| `display` | `border-radius` | `filter` (GPU) |
| `font-size` | `visibility` | |
| `position` | `outline` | |

Animate `transform` and `opacity` whenever possible. They skip layout and paint entirely and run on the compositor thread.

## Microtask Starvation

Because the event loop drains ALL microtasks before moving to the next macrotask or rendering, an infinite stream of microtasks blocks everything:

```js
// DANGER: Infinite microtask loop
function starve() {
  queueMicrotask(starve);
}
starve();
// Page is permanently frozen. No rendering, no macrotasks, no user input.

// Same problem with Promises
function promiseStarve() {
  Promise.resolve().then(promiseStarve);
}
promiseStarve(); // Permanent freeze
```

This is worse than an infinite `while(true)` because the engine IS making progress (executing microtasks), so the browser may not detect the hang as quickly.

**Gotcha:** `async/await` does NOT prevent starvation:

```js
// Still starves: await Promise.resolve() creates microtasks, not macrotasks
async function stillStarves() {
  while (true) {
    await Promise.resolve(); // Microtask continuation
    doWork();
  }
}
// The microtask queue never empties.
```

**To yield properly, use a macrotask:**

```js
async function yieldsCorrectly() {
  for (let i = 0; i < 100000; i++) {
    doWork(i);
    if (i % 100 === 0) {
      // Yield via setTimeout (macrotask) to let rendering and input process
      await new Promise(resolve => setTimeout(resolve, 0));
    }
  }
}
```

## Node.js Event Loop

Node.js uses **libuv** for its event loop, which has a phase-based architecture different from the browser.

### The 6 Phases

```
   ┌───────────────────────────┐
┌─>│        timers              │  setTimeout, setInterval
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │     pending callbacks      │  Deferred I/O callbacks
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare        │  Internal use only
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           poll             │  Retrieve new I/O events
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           check            │  setImmediate callbacks
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │      close callbacks       │  socket.on('close', ...)
│  └───────────────────────────┘
```

Between EVERY phase transition, Node.js drains the microtask queue (both `process.nextTick` and Promise microtasks).

### process.nextTick vs queueMicrotask vs setImmediate

| Feature | `process.nextTick()` | `queueMicrotask()` | `setImmediate()` |
|---------|---------------------|--------------------|--------------------|
| Queue | nextTick queue | Microtask queue | Check phase |
| Priority | Highest async | High | Normal (macrotask) |
| Starvation risk | HIGH | HIGH | LOW |
| Spec | Node.js only | Web standard | Node.js only |

```js
process.nextTick(() => console.log('1: nextTick'));
queueMicrotask(() => console.log('2: queueMicrotask'));
setTimeout(() => console.log('3 or 4: setTimeout'), 0);
setImmediate(() => console.log('3 or 4: setImmediate'));

// Output:
// 1: nextTick
// 2: queueMicrotask
// 3 or 4: setTimeout / setImmediate (order is non-deterministic at top level)
```

**Inside an I/O callback, `setImmediate` always runs before `setTimeout`:**

```js
const fs = require('fs');
fs.readFile(__filename, () => {
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
  // Always: immediate, timeout
  // Because we are in the poll phase, check phase (setImmediate) runs next.
});
```

**Node.js recommendation:** Prefer `setImmediate()` over `process.nextTick()` in most cases. `process.nextTick` can starve the event loop with recursive calls.

## Web Workers

Web Workers run JavaScript in a separate OS thread with their own event loop, call stack, and memory heap.

```
Main Thread                    Worker Thread
┌──────────────┐              ┌──────────────┐
│  Event Loop  │              │  Event Loop  │
│  Call Stack  │  postMessage │  Call Stack  │
│  Task Queue  │ ◄──────────► │  Task Queue  │
│  DOM Access  │  (copy data) │  NO DOM      │
└──────────────┘              └──────────────┘
```

### Communication via postMessage

```js
// main.js
const worker = new Worker('worker.js');
worker.postMessage({ data: largeArray }); // Data is COPIED (structured clone)
worker.onmessage = (event) => {
  console.log('Result:', event.data);
};

// worker.js
self.onmessage = (event) => {
  const result = heavyComputation(event.data);
  self.postMessage(result);
};
```

### Transferable Objects (Zero-Copy)

For large data, use transferable objects to move ownership without copying:

```js
const buffer = new ArrayBuffer(1024 * 1024); // 1MB
console.log(buffer.byteLength); // 1048576

worker.postMessage(buffer, [buffer]); // Transfer ownership
console.log(buffer.byteLength); // 0 (buffer is neutered)
```

### SharedArrayBuffer (Shared Memory)

`SharedArrayBuffer` allows both threads to access the same memory. Requires `Atomics` for synchronization:

```js
// main.js
const shared = new SharedArrayBuffer(1024);
const view = new Int32Array(shared);
worker.postMessage(shared);

// worker.js
self.onmessage = (event) => {
  const view = new Int32Array(event.data);
  Atomics.add(view, 0, 1);   // Thread-safe increment
  Atomics.notify(view, 0);    // Wake up waiting threads
};
```

Requires these HTTP headers:

```
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
```

## The Scheduler API

The Prioritized Task Scheduling API provides fine-grained control over task priority.

### scheduler.postTask()

```js
// Three priority levels
scheduler.postTask(() => console.log('critical'), { priority: 'user-blocking' });
scheduler.postTask(() => console.log('normal'),   { priority: 'user-visible' });
scheduler.postTask(() => console.log('low'),       { priority: 'background' });

// Output: critical, normal, low
```

With cancellation:

```js
const controller = new AbortController();

scheduler.postTask(() => {
  console.log('may be cancelled');
}, { signal: controller.signal, priority: 'background' });

controller.abort(); // Cancel before execution
```

### scheduler.yield()

Yields to the main thread and resumes at the same priority. Unlike `setTimeout(0)`, no 4ms clamping:

```js
async function longTask() {
  for (let i = 0; i < 1000; i++) {
    doWork(i);
    if (i % 100 === 0) {
      await scheduler.yield(); // Yield every 100 iterations
    }
  }
}
```

Feature detection:

```js
async function yieldToMain() {
  if (globalThis.scheduler?.yield) {
    await scheduler.yield();
  } else {
    await new Promise(resolve => setTimeout(resolve, 0));
  }
}
```

Browser support (early 2026): `scheduler.postTask()` in Chrome 94+, Edge 94+, Firefox 135+. `scheduler.yield()` in Chrome 129+, Edge 129+.

## Tricky Execution Order Examples

### Example 1: The Basics

```js
console.log('A');
setTimeout(() => console.log('B'), 0);
Promise.resolve().then(() => console.log('C'));
console.log('D');

// Output: A, D, C, B
// Sync first (A, D), then microtasks (C), then macrotasks (B)
```

### Example 2: Nested Promises and Timers

```js
console.log('1');

setTimeout(() => {
  console.log('2');
  Promise.resolve().then(() => console.log('3'));
}, 0);

Promise.resolve().then(() => {
  console.log('4');
  setTimeout(() => console.log('5'), 0);
});

console.log('6');

// Output: 1, 6, 4, 2, 3, 5
// Sync: 1, 6
// Microtask: 4 (also schedules setTimeout for '5')
// Macrotask: 2 (enqueues microtask for '3')
// Microtask: 3
// Macrotask: 5
```

### Example 3: async/await

```js
async function foo() {
  console.log('foo start');
  await bar();
  console.log('foo end'); // Microtask (continuation after await)
}

async function bar() {
  console.log('bar');
}

console.log('script start');
setTimeout(() => console.log('setTimeout'), 0);
foo();
new Promise((resolve) => {
  console.log('promise executor'); // Promise constructor is synchronous
  resolve();
}).then(() => console.log('promise then'));
console.log('script end');

// Output:
// script start
// foo start
// bar
// promise executor
// script end
// foo end
// promise then
// setTimeout
```

### Example 4: Click Handler (User vs Programmatic)

```js
button.addEventListener('click', () => {
  console.log('listener 1');
  Promise.resolve().then(() => console.log('microtask 1'));
});

button.addEventListener('click', () => {
  console.log('listener 2');
  Promise.resolve().then(() => console.log('microtask 2'));
});

// USER clicks the button:
// listener 1, microtask 1, listener 2, microtask 2
// Each listener dispatched as a separate task; microtasks drain between them.

// button.click() (programmatic):
// listener 1, listener 2, microtask 1, microtask 2
// Synchronous dispatch within one task; microtasks drain after both listeners.
```

## Common Mistakes

### 1. Assuming setTimeout(fn, 0) Is Immediate

```js
// BAD: expecting synchronous-like timing
setTimeout(() => updateUI(), 0);
console.log('UI updated'); // This runs BEFORE updateUI()
```

### 2. Using Microtasks When You Need to Yield

```js
// BAD: queueMicrotask does NOT yield to rendering or user input
queueMicrotask(() => {
  heavyWork(); // Still blocks rendering
});

// GOOD: use setTimeout or scheduler.yield to actually yield
setTimeout(() => {
  heavyWork();
}, 0);
```

### 3. Blocking the Event Loop with Long Synchronous Code

```js
// BAD: freezes the page for 500ms+
items.forEach(item => heavyComputation(item));

// GOOD: chunk the work
async function processChunked(items) {
  for (let i = 0; i < items.length; i++) {
    heavyComputation(items[i]);
    if (i % 100 === 0) {
      await new Promise(resolve => setTimeout(resolve, 0));
    }
  }
}
```

### 4. Layout Thrashing in Loops

```js
// BAD: read-write-read-write cycle forces layout per iteration
for (const el of elements) {
  el.style.height = el.offsetHeight * 2 + 'px';
}

// GOOD: batch reads, then batch writes
const heights = [...elements].map(el => el.offsetHeight);
elements.forEach((el, i) => {
  el.style.height = heights[i] * 2 + 'px';
});
```

### 5. Expecting Consistent setTimeout vs setImmediate Order in Node.js

```js
// Non-deterministic at the top level
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));
// Order depends on system clock resolution
```

## Best Practices

- **Keep tasks under 50ms.** Anything longer is a "long task" that blocks rendering and input. Break up heavy work with `scheduler.yield()`, `setTimeout`, or `requestIdleCallback`.

- **Use `requestAnimationFrame` for visual updates.** Coalesce multiple DOM writes into a single rAF callback to avoid redundant layout and paint.

- **Batch DOM reads and writes.** Never interleave reading layout properties with writing styles in a loop.

- **Animate `transform` and `opacity` only.** They run on the compositor thread and skip layout and paint entirely.

- **Use `MessageChannel` instead of `setTimeout(fn, 0)`** when you need to schedule macrotasks without the 4ms nesting clamp.

- **Use `scheduler.yield()` (with fallback)** for yielding to the main thread in modern browsers.

- **Move heavy computation to Web Workers.** The main thread stays responsive while workers process data on separate threads.

- **Use `requestIdleCallback` for non-critical background work** like analytics, pre-caching, and telemetry.

## Real-World Use Cases

### 1. Progressive Rendering with Time Budgeting

```js
async function renderList(items) {
  const BUDGET_MS = 8; // Half a frame at 60fps

  for (let i = 0; i < items.length; i++) {
    const start = performance.now();

    while (i < items.length && performance.now() - start < BUDGET_MS) {
      renderItem(items[i]);
      i++;
    }

    // Yield to let the browser render what we have so far
    if (globalThis.scheduler?.yield) {
      await scheduler.yield();
    } else {
      await new Promise(resolve => requestAnimationFrame(resolve));
    }
  }
}
```

### 2. Offloading to a Web Worker

```js
// image-processor.js (worker)
self.onmessage = ({ data: { imageData, filter } }) => {
  const pixels = new Uint8ClampedArray(imageData.data);

  for (let i = 0; i < pixels.length; i += 4) {
    if (filter === 'grayscale') {
      const avg = (pixels[i] + pixels[i+1] + pixels[i+2]) / 3;
      pixels[i] = pixels[i+1] = pixels[i+2] = avg;
    }
  }

  self.postMessage({ data: pixels }, [pixels.buffer]); // Transfer back
};

// main.js
const worker = new Worker('image-processor.js');
const imageData = ctx.getImageData(0, 0, canvas.width, canvas.height);

worker.postMessage(
  { imageData, filter: 'grayscale' },
  [imageData.data.buffer] // Transfer to avoid copy
);

worker.onmessage = ({ data }) => {
  const processed = new ImageData(new Uint8ClampedArray(data.data), canvas.width);
  ctx.putImageData(processed, 0, 0);
};
```

### 3. Non-Blocking Analytics with requestIdleCallback

```js
const analyticsQueue = [];

function trackEvent(event) {
  analyticsQueue.push({ ...event, timestamp: Date.now() });
  scheduleFlush();
}

let flushScheduled = false;
function scheduleFlush() {
  if (flushScheduled) return;
  flushScheduled = true;

  requestIdleCallback((deadline) => {
    flushScheduled = false;

    while (deadline.timeRemaining() > 5 && analyticsQueue.length > 0) {
      const batch = analyticsQueue.splice(0, 10);
      navigator.sendBeacon('/analytics', JSON.stringify(batch));
    }

    if (analyticsQueue.length > 0) {
      scheduleFlush(); // More events remain, schedule another idle callback
    }
  }, { timeout: 10000 }); // Force flush after 10s even if busy
}
```

---

> **Key Takeaway:** The event loop processes one macrotask at a time, drains all microtasks after each task, renders when the browser decides it is time, and then repeats; understanding this sequence is essential for writing responsive, non-blocking JavaScript.
