# 5.24 WebSockets and Server-Sent Events (SSE)

> Two protocols for real-time communication between client and server: WebSocket for full-duplex bidirectional messaging, and SSE for simple unidirectional server-to-client streaming.

---

## The Need for Real-Time Communication

Traditional HTTP is request-response: the client asks, the server answers. For real-time features (chat, live dashboards, notifications, collaborative editing), you need the server to push data to the client without the client constantly asking "anything new?" (polling).

There are two native browser APIs for this:

| Feature | WebSocket | Server-Sent Events (SSE) |
|---------|-----------|--------------------------|
| Direction | Full-duplex (bidirectional) | Unidirectional (server to client) |
| Protocol | `ws://` / `wss://` (upgraded from HTTP) | Standard HTTP (`text/event-stream`) |
| Data format | Text and binary | Text only (UTF-8) |
| Auto reconnection | No | Yes, built-in |
| Stream resumption | No | Yes, via `Last-Event-ID` |
| Custom headers | No (during handshake) | No (native API), yes (via fetch) |
| HTTP/2 multiplexing | No (uses its own TCP connection) | Yes |
| Browser support | Universal | Universal (except IE) |

## WebSocket API

### Creating a Connection

```javascript
// Basic connection
const ws = new WebSocket("wss://example.com/chat");

// With subprotocol negotiation
const ws = new WebSocket("wss://example.com/chat", ["json", "xml"]);

// With a single subprotocol
const ws = new WebSocket("wss://example.com/chat", "json");
```

The constructor immediately begins the opening handshake. It sends an HTTP `Upgrade` request, and the server responds with `101 Switching Protocols` to establish the connection.

The URL must use `ws://` or `wss://` (or `http://`/`https://`, which are automatically mapped). It cannot contain a URL fragment (`#`).

### readyState

The connection goes through four states:

| Constant | Value | Meaning |
|----------|-------|---------|
| `WebSocket.CONNECTING` | `0` | Handshake in progress |
| `WebSocket.OPEN` | `1` | Connection established, ready to communicate |
| `WebSocket.CLOSING` | `2` | `close()` called, closing handshake in progress |
| `WebSocket.CLOSED` | `3` | Connection closed (or failed to open) |

```javascript
if (ws.readyState === WebSocket.OPEN) {
  ws.send("safe to send");
}
```

### Properties

```javascript
ws.url;             // "wss://example.com/chat" (resolved absolute URL)
ws.protocol;        // "json" (subprotocol selected by server, or "")
ws.extensions;      // e.g., "permessage-deflate" (server-selected extensions)
ws.binaryType;      // "blob" (default) or "arraybuffer"
ws.bufferedAmount;  // bytes queued by send() but not yet transmitted
```

### Events

Four events cover the entire lifecycle:

```javascript
const ws = new WebSocket("wss://example.com/chat");

// 1. Connection established
ws.addEventListener("open", (event) => {
  console.log("Connected");
  ws.send("Hello server");
});

// 2. Message received
ws.addEventListener("message", (event) => {
  console.log("Received:", event.data);
  // event.data is a string (text frame) or ArrayBuffer/Blob (binary frame)
});

// 3. Error occurred
ws.addEventListener("error", (event) => {
  // Intentionally vague for security. No useful error details here.
  // Check the close event for code and reason.
  console.error("WebSocket error");
});

// 4. Connection closed
ws.addEventListener("close", (event) => {
  console.log("Closed:", event.code, event.reason);
  console.log("Clean close:", event.wasClean);
});
```

You can also use the `on*` property shorthand (`ws.onopen`, `ws.onmessage`, etc.), but `addEventListener` is preferred since it allows multiple listeners.

### Sending Data

```javascript
// Send text
ws.send("plain text message");

// Send JSON (most common pattern)
ws.send(JSON.stringify({ type: "chat", text: "hello" }));

// Send an ArrayBuffer
const buffer = new ArrayBuffer(4);
new Uint32Array(buffer)[0] = 42;
ws.send(buffer);

// Send a typed array
ws.send(new Uint8Array([0x48, 0x65, 0x6C, 0x6C, 0x6F]));

// Send a Blob
ws.send(new Blob(["binary data"], { type: "application/octet-stream" }));
```

`send()` is asynchronous. It queues data in an internal buffer and returns immediately. It does NOT wait for transmission. Use `bufferedAmount` to check how much is queued.

```javascript
// BAD PRACTICE: Calling send() when not connected
ws.send("will throw"); // InvalidStateError if readyState is CONNECTING

// GOOD: Always check readyState
function safeSend(ws, data) {
  if (ws.readyState === WebSocket.OPEN) {
    ws.send(data);
    return true;
  }
  return false;
}
```

### Closing the Connection

```javascript
// Normal close
ws.close();

// With a close code and reason
ws.close(1000, "Done chatting");

// Custom application-level close code (3000-4999)
ws.close(4001, "Session expired");
```

The `code` must be `1000` (normal) or in the range `3000-4999`. The `reason` string must be at most 123 bytes of UTF-8.

### Close Codes

| Code | Name | Meaning |
|------|------|---------|
| 1000 | Normal Closure | Connection fulfilled its purpose |
| 1001 | Going Away | Server shutting down or page navigating away |
| 1002 | Protocol Error | Protocol-level error |
| 1003 | Unsupported Data | Received unsupported data type |
| 1005 | No Status Received | No close code was present (set locally, cannot be sent) |
| 1006 | Abnormal Closure | Connection dropped without close frame (set locally, cannot be sent) |
| 1007 | Invalid Payload | Data inconsistent with message type |
| 1008 | Policy Violation | Message violates policy |
| 1009 | Message Too Big | Message too large to process |
| 1011 | Internal Error | Server encountered unexpected condition |
| 1012 | Service Restart | Server is restarting |
| 1013 | Try Again Later | Server temporarily unavailable |
| 1015 | TLS Failure | TLS handshake failed (set locally, cannot be sent) |
| 3000-3999 | Registered | Reserved for libraries/frameworks (IANA registered) |
| 4000-4999 | Private Use | Available for application-specific use |

Codes `1005`, `1006`, and `1015` are never sent over the wire. The browser sets them locally to indicate specific conditions.

## Binary Data with WebSockets

### Receiving Binary Data

The `binaryType` property controls how binary frames are delivered to `onmessage`:

```javascript
const ws = new WebSocket("wss://example.com/binary");

// Option 1: Receive as ArrayBuffer (recommended)
ws.binaryType = "arraybuffer";

ws.onmessage = (event) => {
  if (typeof event.data === "string") {
    // text frame
    console.log("Text:", event.data);
  } else {
    // binary frame -- event.data is an ArrayBuffer
    const view = new DataView(event.data);
    const messageType = view.getUint8(0);
    const payload = view.getFloat64(1);
    console.log("Binary:", messageType, payload);
  }
};

// Option 2: Receive as Blob (default)
ws.binaryType = "blob";

ws.onmessage = async (event) => {
  if (event.data instanceof Blob) {
    // Must convert asynchronously
    const buffer = await event.data.arrayBuffer();
    const view = new DataView(buffer);
    // ...process
  }
};
```

| Aspect | ArrayBuffer | Blob |
|--------|-------------|------|
| Processing | Synchronous via DataView/TypedArray | Async via `blob.arrayBuffer()` |
| Performance | Faster for immediate processing | Slower (async conversion step) |
| Best for | Parsing protocols, real-time data | Large files, creating object URLs |

### Sending Binary Data

```javascript
// Custom binary protocol: [type: u8] [id: u32] [value: f64]
function sendBinaryMessage(ws, type, id, value) {
  const buffer = new ArrayBuffer(13); // 1 + 4 + 8 bytes
  const view = new DataView(buffer);
  view.setUint8(0, type);
  view.setUint32(1, id);
  view.setFloat64(5, value);
  ws.send(buffer);
}

sendBinaryMessage(ws, 1, 42, 3.14);
```

## Backpressure and bufferedAmount

`bufferedAmount` tells you how many bytes are queued by `send()` but not yet transmitted. It is your only tool for detecting send-side congestion.

```javascript
const MAX_BUFFER = 65536; // 64 KB

function throttledSend(ws, data) {
  if (ws.bufferedAmount > MAX_BUFFER) {
    // Too much data queued. Drop, defer, or slow down.
    console.warn("Send buffer full, deferring");
    return false;
  }
  ws.send(data);
  return true;
}
```

### Gotcha: No Receive-Side Backpressure

The standard WebSocket API provides no mechanism to slow down incoming messages. If the server sends faster than the client processes, messages accumulate in memory. There is no event when `bufferedAmount` changes. You must poll it if you need to monitor it.

### WebSocketStream (Experimental, Chrome Only)

The `WebSocketStream` API integrates with the Streams API to solve backpressure on both sides:

```javascript
// Chrome 124+ only, not widely supported
const wss = new WebSocketStream("wss://example.com");
const { readable, writable } = await wss.opened;

// Reading with automatic backpressure
const reader = readable.getReader();
while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  await processMessage(value); // reader pauses until we call read() again
}

// Writing with backpressure
const writer = writable.getWriter();
await writer.write("hello"); // awaits until buffer has capacity
```

This is not production-ready for cross-browser applications yet.

## WebSocket Subprotocols

Subprotocols define an application-level protocol layered on top of WebSocket frames. They are negotiated during the handshake.

```javascript
// Client requests one or more subprotocols
const ws = new WebSocket("wss://example.com", ["mqtt", "wamp", "json"]);

ws.addEventListener("open", () => {
  // Server selected one (or none)
  console.log("Negotiated protocol:", ws.protocol); // e.g., "json"

  if (ws.protocol === "json") {
    ws.send(JSON.stringify({ action: "subscribe", topic: "news" }));
  }
});
```

The negotiation works as follows:

1. Client sends `Sec-WebSocket-Protocol: mqtt, wamp, json` in the upgrade request.
2. Server picks the one it supports (e.g., `Sec-WebSocket-Protocol: json`) and includes it in the response.
3. If the server supports none, it omits the header.
4. After the handshake, `ws.protocol` reflects the selected subprotocol (empty string if none).

Common subprotocols include `mqtt` (IoT messaging), `stomp` (messaging), `wamp` (RPC + pub/sub), and `graphql-ws` (GraphQL subscriptions).

## WebSocket Reconnection

WebSocket has zero built-in reconnection. You must implement it yourself. The standard pattern uses exponential backoff with jitter:

```javascript
class ReconnectingWebSocket {
  #url;
  #protocols;
  #ws = null;
  #attempt = 0;
  #maxAttempts = 10;
  #baseDelay = 1000;   // 1 second
  #maxDelay = 30000;   // 30 seconds
  #queue = [];
  #listeners = { open: [], message: [], close: [], error: [] };
  #forceClosed = false;

  constructor(url, protocols) {
    this.#url = url;
    this.#protocols = protocols;
    this.#connect();
  }

  #connect() {
    this.#ws = new WebSocket(this.#url, this.#protocols);

    this.#ws.addEventListener("open", (e) => {
      this.#attempt = 0; // reset backoff on success

      // Flush queued messages
      while (this.#queue.length > 0) {
        this.#ws.send(this.#queue.shift());
      }

      this.#listeners.open.forEach((fn) => fn(e));
    });

    this.#ws.addEventListener("message", (e) => {
      this.#listeners.message.forEach((fn) => fn(e));
    });

    this.#ws.addEventListener("close", (e) => {
      this.#listeners.close.forEach((fn) => fn(e));

      // Only reconnect on abnormal closure
      if (!this.#forceClosed && e.code !== 1000) {
        this.#scheduleReconnect();
      }
    });

    this.#ws.addEventListener("error", (e) => {
      this.#listeners.error.forEach((fn) => fn(e));
      // The close event always fires after error, so reconnection
      // is handled there.
    });
  }

  #scheduleReconnect() {
    if (this.#attempt >= this.#maxAttempts) {
      console.error("Max reconnection attempts reached");
      return;
    }

    // Exponential backoff: 1s, 2s, 4s, 8s, 16s, 30s, 30s, ...
    const delay = Math.min(
      this.#baseDelay * Math.pow(2, this.#attempt),
      this.#maxDelay
    );

    // Add random jitter (0% to 50% of delay) to avoid thundering herd
    const jitter = delay * 0.5 * Math.random();

    setTimeout(() => {
      this.#attempt++;
      this.#connect();
    }, delay + jitter);
  }

  send(data) {
    if (this.#ws && this.#ws.readyState === WebSocket.OPEN) {
      this.#ws.send(data);
    } else {
      this.#queue.push(data); // queue for when connection restores
    }
  }

  close(code = 1000, reason) {
    this.#forceClosed = true;
    this.#ws?.close(code, reason);
  }

  on(event, callback) {
    this.#listeners[event]?.push(callback);
  }

  get readyState() {
    return this.#ws?.readyState ?? WebSocket.CLOSED;
  }
}
```

Key design decisions:

- **Exponential backoff** prevents hammering the server after a crash (1s, 2s, 4s, 8s, ...).
- **Jitter** prevents a "thundering herd" where thousands of clients reconnect at exactly the same time.
- **Max attempts** prevents infinite loops.
- **Message queue** prevents data loss during brief disconnections.
- **Force-closed flag** distinguishes intentional closes from failures.

## WebSocket Heartbeat

The WebSocket protocol defines Ping/Pong control frames, but the browser JavaScript API does not expose them. Browsers handle Pong responses automatically at the protocol level. For detecting dead connections from JavaScript, implement an application-level heartbeat:

```javascript
class HeartbeatWebSocket {
  #ws;
  #pingInterval;
  #pongTimeout;
  #intervalMs = 25000;  // send ping every 25 seconds
  #timeoutMs = 10000;   // expect pong within 10 seconds

  constructor(url) {
    this.#ws = new WebSocket(url);
    this.#ws.addEventListener("open", () => this.#startHeartbeat());
    this.#ws.addEventListener("close", () => this.#stopHeartbeat());
    this.#ws.addEventListener("message", (event) => {
      const data = JSON.parse(event.data);
      if (data.type === "pong") {
        clearTimeout(this.#pongTimeout);
        return;
      }
      this.onMessage(data);
    });
  }

  #startHeartbeat() {
    this.#pingInterval = setInterval(() => {
      if (this.#ws.readyState !== WebSocket.OPEN) return;

      this.#ws.send(JSON.stringify({ type: "ping" }));

      this.#pongTimeout = setTimeout(() => {
        // No pong received. Connection is dead.
        this.#ws.close(4000, "Heartbeat timeout");
      }, this.#timeoutMs);
    }, this.#intervalMs);
  }

  #stopHeartbeat() {
    clearInterval(this.#pingInterval);
    clearTimeout(this.#pongTimeout);
  }

  onMessage(data) {
    // Override in application code
  }
}
```

The ping interval (25 seconds) should be shorter than any intermediary timeout. Load balancers and proxies typically close idle connections after 60 seconds.

## Server-Sent Events (SSE)

### The EventSource API

```javascript
// Basic connection
const source = new EventSource("/api/events");

// Cross-origin with credentials (cookies)
const source = new EventSource("https://api.example.com/events", {
  withCredentials: true,
});
```

The constructor opens a GET request to the URL. The server must respond with `Content-Type: text/event-stream`.

### readyState

| Constant | Value | Meaning |
|----------|-------|---------|
| `EventSource.CONNECTING` | `0` | Not yet open, or reconnecting |
| `EventSource.OPEN` | `1` | Connection is open |
| `EventSource.CLOSED` | `2` | Connection closed, no more reconnection |

### Listening to Events

```javascript
const source = new EventSource("/api/events");

// Default "message" event (no event: field, or event: message)
source.addEventListener("message", (event) => {
  console.log("Data:", event.data);
  console.log("Last ID:", event.lastEventId);
  console.log("Origin:", event.origin);
});

// Named events (server sends event: notification)
source.addEventListener("notification", (event) => {
  const data = JSON.parse(event.data);
  showNotification(data.title, data.body);
});

source.addEventListener("priceUpdate", (event) => {
  const { symbol, price } = JSON.parse(event.data);
  updateTicker(symbol, price);
});

// Connection opened
source.addEventListener("open", (event) => {
  console.log("SSE connection established");
});

// Error handling
source.addEventListener("error", (event) => {
  if (source.readyState === EventSource.CONNECTING) {
    console.log("Reconnecting...");
  } else if (source.readyState === EventSource.CLOSED) {
    console.log("Connection closed permanently");
  }
});
```

### Closing the Connection

```javascript
source.close(); // Stops reconnection attempts permanently
```

After calling `close()`, `readyState` becomes `EventSource.CLOSED` and no further reconnection occurs.

## SSE Protocol Format

The server sends a stream of UTF-8 text events. Each event consists of one or more field lines, separated by blank lines (`\n\n`).

### Fields

| Field | Purpose |
|-------|---------|
| `data:` | The event payload. Multiple lines are joined with `\n`. |
| `event:` | Names the event type. Defaults to `"message"` if omitted. |
| `id:` | Sets the last event ID. Sent back as `Last-Event-ID` on reconnection. |
| `retry:` | Sets reconnection delay in milliseconds. |
| `:` (comment) | Ignored by the client. Used as a heartbeat/keep-alive. |

### Example Stream

```
: heartbeat (comment, keeps connection alive)

data: Simple unnamed message

event: userLogin
data: {"user":"alice","time":1709123456}
id: 1

data: first line
data: second line
id: 2

event: priceUpdate
data: {"symbol":"AAPL","price":185.42}
id: 3

retry: 5000
```

What the client receives:

1. The comment is ignored.
2. `"Simple unnamed message"` dispatches as a `"message"` event.
3. `{"user":"alice",...}` dispatches as a `"userLogin"` event.
4. `"first line\nsecond line"` dispatches as a `"message"` event (multi-line `data:` fields are joined with `\n`).
5. `{"symbol":"AAPL",...}` dispatches as a `"priceUpdate"` event.
6. `retry: 5000` tells the browser to wait 5 seconds before reconnecting.

### Server Response Headers

The server must include:

```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

For Nginx proxying, also add `X-Accel-Buffering: no` to prevent buffering.

## SSE Automatic Reconnection and Stream Resumption

This is one of SSE's biggest advantages over WebSocket.

### How It Works

1. The server sends `id:` fields with each event.
2. The client tracks the last received ID internally.
3. When the connection drops, the browser automatically reconnects.
4. On reconnection, the browser sends the `Last-Event-ID` HTTP header with the last received ID.
5. The server reads this header and resumes the stream from that point.

```javascript
const source = new EventSource("/api/events");

source.addEventListener("message", (event) => {
  console.log("ID:", event.lastEventId); // The current event's ID
  console.log("Data:", event.data);
});

// If connection drops after receiving id: 42:
// Browser reconnects with header: Last-Event-ID: 42
// Server resumes from event 43
```

### Server-Controlled Retry Interval

```
retry: 3000
data: reconnect in 3 seconds if disconnected
```

The `retry:` field sets the delay (in milliseconds) that the browser waits before reconnecting. Without it, browsers use a default (typically 3-5 seconds).

### When Reconnection Does NOT Happen

- After calling `source.close()`.
- If the server responds with any status other than 200.
- If the `Content-Type` is not `text/event-stream`.
- If the server sends an HTTP 204 No Content (this is the recommended way for the server to tell the client to stop reconnecting).

## SSE Connection Limit Problem

With HTTP/1.1, browsers limit connections to 6 per domain. Each `EventSource` consumes one. Opening multiple tabs with SSE connections quickly exhausts this limit, blocking all other HTTP requests to the same domain.

### Solutions

**1. Use HTTP/2**: HTTP/2 multiplexes up to 100 streams over a single TCP connection, effectively eliminating this problem. This is the recommended fix.

**2. SharedWorker**: Share a single SSE connection across tabs:

```javascript
// shared-worker.js
let source = null;
const ports = [];

self.addEventListener("connect", (event) => {
  const port = event.ports[0];
  ports.push(port);

  if (!source) {
    source = new EventSource("/api/events");
    source.addEventListener("message", (event) => {
      // Broadcast to all connected tabs
      for (const p of ports) {
        p.postMessage({ type: "message", data: event.data });
      }
    });
  }

  port.start();
});

// main.js (in each tab)
const worker = new SharedWorker("shared-worker.js");
worker.port.start();
worker.port.addEventListener("message", (event) => {
  console.log("Received:", event.data);
});
```

**3. BroadcastChannel**: One tab owns the SSE connection, others listen via `BroadcastChannel`:

```javascript
// Leader tab
const source = new EventSource("/api/events");
const channel = new BroadcastChannel("sse-events");

source.addEventListener("message", (event) => {
  channel.postMessage({ type: "message", data: event.data });
});

// Other tabs
const channel = new BroadcastChannel("sse-events");
channel.addEventListener("message", (event) => {
  console.log("Received:", event.data);
});
```

## Authentication Challenges

Neither WebSocket nor native EventSource support custom HTTP headers during the initial request. This creates authentication challenges.

### WebSocket Authentication Strategies

**1. Cookie-based** (simplest, but has CSWSH risk):

```javascript
// If user is authenticated via session cookie, it is sent automatically
const ws = new WebSocket("wss://example.com/chat");
// Server MUST validate the Origin header to prevent CSWSH
```

**2. Token in query string** (common, but token appears in logs):

```javascript
const ws = new WebSocket(`wss://example.com/chat?token=${accessToken}`);
// Use short-lived tokens to limit exposure
```

**3. First-message authentication** (most common in practice):

```javascript
const ws = new WebSocket("wss://example.com/chat");

ws.addEventListener("open", () => {
  // Send auth as the first message
  ws.send(JSON.stringify({
    type: "auth",
    token: accessToken,
  }));
});

ws.addEventListener("message", (event) => {
  const msg = JSON.parse(event.data);
  if (msg.type === "auth_success") {
    console.log("Authenticated, ready to chat");
  } else if (msg.type === "auth_failed") {
    ws.close(4001, "Authentication failed");
  }
});
```

**4. Ticket-based two-step** (most secure):

```javascript
// Step 1: Get a short-lived ticket via authenticated HTTP
const response = await fetch("/api/ws-ticket", {
  method: "POST",
  headers: { Authorization: `Bearer ${accessToken}` },
});
const { ticket } = await response.json();

// Step 2: Connect with the one-time ticket
const ws = new WebSocket(`wss://example.com/chat?ticket=${ticket}`);
// Server validates and invalidates the ticket (single use)
```

### EventSource Authentication Strategies

**Cookie-based** (native):

```javascript
const source = new EventSource("/api/events", {
  withCredentials: true, // sends cookies cross-origin
});
```

**Token in query string** (native):

```javascript
const source = new EventSource(`/api/events?token=${accessToken}`);
```

For anything more sophisticated, use fetch-based streaming (next section).

## Fetch-Based SSE (Alternative to EventSource)

The native `EventSource` API has hard limitations: GET-only, no custom headers, no request body. Using `fetch()` with a `ReadableStream` overcomes all of them.

```javascript
async function streamEvents(url, options = {}) {
  const { signal, onMessage, onError, headers = {}, body } = options;

  const response = await fetch(url, {
    method: body ? "POST" : "GET",
    headers: {
      Accept: "text/event-stream",
      ...headers,
    },
    body: body ? JSON.stringify(body) : undefined,
    signal,
  });

  if (!response.ok) {
    throw new Error(`HTTP ${response.status}: ${response.statusText}`);
  }

  const reader = response.body.getReader();
  const decoder = new TextDecoder();
  let buffer = "";

  while (true) {
    const { value, done } = await reader.read();
    if (done) break;

    buffer += decoder.decode(value, { stream: true });

    // Split on double newline (event boundary)
    const parts = buffer.split("\n\n");
    buffer = parts.pop(); // keep the last incomplete chunk

    for (const part of parts) {
      if (!part.trim()) continue;

      const event = parseSSEEvent(part);
      if (event) {
        onMessage?.(event);
      }
    }
  }
}

function parseSSEEvent(raw) {
  const lines = raw.split("\n");
  let data = "";
  let eventType = "message";
  let id = "";

  for (const line of lines) {
    if (line.startsWith("data: ")) {
      data += (data ? "\n" : "") + line.slice(6);
    } else if (line.startsWith("data:")) {
      data += (data ? "\n" : "") + line.slice(5);
    } else if (line.startsWith("event: ")) {
      eventType = line.slice(7);
    } else if (line.startsWith("id: ")) {
      id = line.slice(4);
    } else if (line.startsWith(":")) {
      // comment, ignore
    }
  }

  if (!data) return null;
  return { type: eventType, data, lastEventId: id };
}
```

### Usage

```javascript
const controller = new AbortController();

streamEvents("/api/chat", {
  signal: controller.signal,
  headers: {
    Authorization: "Bearer my-token",
  },
  body: { prompt: "Explain closures in JavaScript" },
  onMessage(event) {
    if (event.type === "token") {
      appendToOutput(event.data);
    } else if (event.type === "done") {
      console.log("Stream complete");
    }
  },
});

// Cancel the stream
controller.abort();
```

### EventSource vs. Fetch Streaming

| Feature | EventSource | fetch + ReadableStream |
|---------|-------------|------------------------|
| Auto reconnection | Yes | No (implement manually) |
| Last-Event-ID resume | Yes | No (track manually) |
| Custom headers | No | Yes |
| POST/PUT/PATCH | No | Yes |
| Request body | No | Yes |
| Named event dispatch | Automatic | Manual parsing |
| AbortController | No (`close()` instead) | Yes |

This is the pattern used by LLM APIs (ChatGPT, Claude) for token-by-token streaming.

---

## WebSocket vs. SSE: When to Use Which

### Use WebSocket When:

- **Bidirectional communication is required**: Chat, collaborative editing, multiplayer games.
- **Binary data is involved**: Audio/video streams, file transfer, binary protocols.
- **Low-latency bidirectional messaging matters**: Real-time gaming, trading platforms.
- **The server needs to receive frequent messages from the client**: Each client message over SSE would be a separate HTTP request.

### Use SSE When:

- **Data flows primarily server-to-client**: Live dashboards, notifications, stock tickers, log streaming.
- **You want automatic reconnection and stream resumption**: SSE handles this out of the box.
- **HTTP/2 compatibility matters**: SSE multiplexes cleanly over HTTP/2.
- **Simplicity is valued**: No handshake upgrade, no special server infrastructure, works behind standard proxies and CDNs.
- **AI/LLM response streaming**: Token-by-token output from language models.

### When Neither is Right:

- **Low-frequency updates**: Simple polling or long-polling may be simpler.
- **Peer-to-peer**: Use WebRTC.
- **Very high throughput bidirectional with backpressure**: Consider WebTransport (emerging standard).

## Common Mistakes

### WebSocket Mistakes

- **Not checking readyState before send()**: Calling `send()` during `CONNECTING` throws `InvalidStateError`. Always guard with `if (ws.readyState === WebSocket.OPEN)`.

- **Closing immediately after send()**: `send()` is async. Calling `close()` right after may discard queued data. Wait for `bufferedAmount === 0` first:

```javascript
// BAD PRACTICE
ws.send(bigPayload);
ws.close(); // might discard the payload

// GOOD
ws.send(bigPayload);
const waitForFlush = setInterval(() => {
  if (ws.bufferedAmount === 0) {
    clearInterval(waitForFlush);
    ws.close(1000, "Done");
  }
}, 50);
```

- **Expecting error details from the error event**: The `error` event provides no useful information by design (security). Inspect the `close` event's `code` and `reason` instead.

- **Not parsing JSON in a try-catch**: Incoming messages may not always be valid JSON.

```javascript
// BAD PRACTICE
ws.onmessage = (e) => {
  const data = JSON.parse(e.data); // throws on invalid JSON
};

// GOOD
ws.onmessage = (e) => {
  try {
    const data = JSON.parse(e.data);
    handleMessage(data);
  } catch (err) {
    console.error("Invalid message:", e.data);
  }
};
```

- **Ignoring bufferedAmount**: Sending data in a tight loop without checking can cause memory exhaustion.

- **Using ws:// in production**: Always use `wss://` (TLS). Browsers may block mixed content (insecure WebSocket on an HTTPS page).

- **Not handling page navigation**: Open WebSockets prevent bfcache (back/forward cache). Close on `pagehide`:

```javascript
window.addEventListener("pagehide", () => ws.close());
```

- **Reconnecting without backoff**: Immediate reconnection after failure causes a "thundering herd" and hammers the server. Always use exponential backoff with jitter.

### SSE Mistakes

- **Not using HTTP/2**: With HTTP/1.1, SSE connections count toward the 6-connections-per-domain limit. Multiple tabs with SSE can deadlock all requests.

- **Omitting `id:` fields**: Without IDs, the browser cannot send `Last-Event-ID` on reconnection, so the server cannot resume the stream.

- **Expecting reconnection on non-200 responses**: If the server returns 401, 500, etc., EventSource fires an error and does NOT reconnect. Only a 200 with `text/event-stream` content type triggers normal streaming and reconnection.

- **Trying to send custom headers**: The native `EventSource` does not support custom headers. Use cookies, query string tokens, or switch to fetch-based streaming.

- **Not sending heartbeats from the server**: Without periodic data (even comments like `: heartbeat\n\n`), proxies and load balancers may close idle connections.

---

## Best Practices

- **WebSocket: Always implement reconnection with exponential backoff and jitter.** This is not optional. Connections will drop.

- **WebSocket: Use application-level heartbeats** (ping/pong messages) every 20-30 seconds to detect dead connections before intermediary timeouts (typically 60 seconds).

- **WebSocket: Define a message protocol.** Use a consistent structure like `{ type: "...", payload: ... }` so both sides know how to interpret each message.

- **SSE: Always include `id:` fields** in server events to enable stream resumption on reconnection.

- **SSE: Send periodic heartbeat comments** (`: heartbeat\n\n`) from the server to keep the connection alive through proxies.

- **SSE: Use HTTP/2** to avoid the 6-connection-per-domain limit.

- **Both: Validate and sanitize all incoming data.** WebSocket and SSE payloads are untrusted input. Never use `innerHTML` with received data.

- **Both: Use TLS** (`wss://` and `https://`). Always.

- **Both: Implement graceful degradation.** If the connection fails permanently, fall back to polling or show the user a "connection lost" indicator.

- **Authentication: Prefer the ticket-based two-step pattern** for WebSocket. For SSE, use cookie-based auth or fetch-based streaming with `Authorization` headers.

### 1. Chat Application (WebSocket)

```javascript
class ChatClient {
  #ws;
  #reconnect;
  #messageCallback;

  constructor(url) {
    this.#reconnect = new ReconnectingWebSocket(url);

    this.#reconnect.on("message", (event) => {
      try {
        const msg = JSON.parse(event.data);
        this.#handleServerMessage(msg);
      } catch (e) {
        console.error("Bad message:", event.data);
      }
    });

    this.#reconnect.on("open", () => {
      console.log("Chat connected");
    });
  }

  #handleServerMessage(msg) {
    switch (msg.type) {
      case "chat":
        this.#messageCallback?.(msg.from, msg.text, msg.timestamp);
        break;
      case "typing":
        showTypingIndicator(msg.user);
        break;
      case "presence":
        updateUserList(msg.users);
        break;
      case "pong":
        // heartbeat response, handled by ReconnectingWebSocket
        break;
    }
  }

  sendMessage(text) {
    this.#reconnect.send(JSON.stringify({
      type: "chat",
      text,
      timestamp: Date.now(),
    }));
  }

  sendTyping() {
    this.#reconnect.send(JSON.stringify({ type: "typing" }));
  }

  onMessage(callback) {
    this.#messageCallback = callback;
  }

  disconnect() {
    this.#reconnect.close();
  }
}

// Usage
const chat = new ChatClient("wss://chat.example.com");
chat.onMessage((from, text, ts) => {
  appendMessage(from, text, new Date(ts));
});
chat.sendMessage("Hello everyone!");
```

### 3. AI Response Streaming (Fetch-Based SSE)

```javascript
async function streamAIResponse(prompt, outputElement) {
  const controller = new AbortController();

  try {
    const response = await fetch("/api/generate", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        "Authorization": `Bearer ${getToken()}`,
        "Accept": "text/event-stream",
      },
      body: JSON.stringify({ prompt, model: "gpt-4" }),
      signal: controller.signal,
    });

    if (!response.ok) {
      throw new Error(`API error: ${response.status}`);
    }

    const reader = response.body.getReader();
    const decoder = new TextDecoder();
    let buffer = "";

    while (true) {
      const { value, done } = await reader.read();
      if (done) break;

      buffer += decoder.decode(value, { stream: true });
      const events = buffer.split("\n\n");
      buffer = events.pop();

      for (const event of events) {
        if (!event.trim()) continue;

        for (const line of event.split("\n")) {
          if (line.startsWith("data: ")) {
            const data = line.slice(6);
            if (data === "[DONE]") return;

            try {
              const parsed = JSON.parse(data);
              const token = parsed.choices?.[0]?.delta?.content;
              if (token) {
                outputElement.textContent += token;
              }
            } catch {
              // Non-JSON data line, skip
            }
          }
        }
      }
    }
  } catch (err) {
    if (err.name === "AbortError") {
      console.log("Stream cancelled by user");
    } else {
      throw err;
    }
  }

  // Return abort function so caller can cancel
  return () => controller.abort();
}

// Usage
const cancel = await streamAIResponse(
  "Explain closures",
  document.getElementById("output")
);

// User clicks "Stop"
document.getElementById("stop-btn").addEventListener("click", cancel);
```

## Security Considerations

### WebSocket Security

- **Always use `wss://`**: Browsers may block `ws://` on HTTPS pages. Data is plaintext without TLS.
- **Validate Origin on the server**: The `Origin` header is sent during the handshake, but non-browser clients can spoof it. Combine with proper authentication.
- **Cross-Site WebSocket Hijacking (CSWSH)**: If using cookie auth, browsers send cookies to any origin opening a WebSocket to your domain. The server MUST check the `Origin` header.
- **Sanitize all incoming messages**: WebSocket payloads are untrusted input. Never insert them into the DOM with `innerHTML`.
- **Set message size limits**: Prevent memory exhaustion from oversized payloads.
- **Rate-limit connections and messages**: Per-IP and per-user limits prevent abuse.

### SSE Security

- **Use HTTPS**: Encrypt the stream.
- **CORS headers**: Required for cross-origin SSE. Server must send `Access-Control-Allow-Origin`.
- **Token rotation**: If using query string tokens, make them short-lived and single-use.
- **Do not embed sensitive data in SSE payloads** unless the connection is properly authenticated.

---

> **Key Takeaway:** Use WebSocket for bidirectional, real-time communication (chat, games, collaborative editing) and SSE for unidirectional server-to-client streaming (dashboards, notifications, AI streaming) where its automatic reconnection and HTTP/2 compatibility provide significant advantages over WebSocket's more complex setup.
