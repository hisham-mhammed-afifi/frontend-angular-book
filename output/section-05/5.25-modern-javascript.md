# 5.25 Modern JavaScript (ES2022-ES2025)

> A comprehensive tour of every major JavaScript feature finalized from ES2022 through ES2025, organized by year, with code examples, gotchas, and migration guidance.

---

## How ECMAScript Versions Work

The TC39 committee manages JavaScript's evolution. Proposals go through five stages:

| Stage | Name | Meaning |
|-------|------|---------|
| 0 | Strawperson | Just an idea |
| 1 | Proposal | Problem and solution described |
| 2 | Draft | Formal spec text written |
| 2.7 | (Informal) | Spec complete, awaiting implementations |
| 3 | Candidate | Spec finalized, implementations underway |
| 4 | Finished | Two shipping implementations, included in next spec |

Features reach Stage 4 and are included in the yearly snapshot (ES2022, ES2023, etc.). All features in this guide are Stage 4 (finalized) and supported in modern browsers.

## ES2022

ES2022 was finalized in June 2022.

### Top-Level `await`

Use `await` directly at the module level without wrapping in an `async` function.

```javascript
// Before ES2022: awkward async IIFE
(async () => {
  const config = await fetch("/api/config").then((r) => r.json());
  init(config);
})();

// ES2022: clean top-level await
const config = await fetch("/api/config").then((r) => r.json());
init(config);
export { config };
```

Dynamic import with fallback:

```javascript
let translations;
try {
  translations = await import(`./i18n/${navigator.language}.mjs`);
} catch {
  translations = await import("./i18n/en.mjs");
}
```

**Gotchas:**
- Only works in ES modules (`<script type="module">` or `.mjs` files), not in classic scripts or CommonJS.
- A module using top-level `await` blocks all modules that import it until the `await` resolves.
- Circular dependencies combined with top-level `await` can deadlock.

### Class Fields and Methods

ES2022 formalized class fields (public and private), static blocks, and private methods that were already widely used.

```javascript
class User {
  // Public instance field with default value
  role = "viewer";

  // Private instance field (truly private, not convention)
  #password;

  // Private static field
  static #instanceCount = 0;

  // Static block for complex initialization
  static {
    // Runs once when the class is evaluated
    console.log("User class loaded");
    User.#instanceCount = 0;
  }

  constructor(name, password) {
    this.name = name;
    this.#password = password;
    User.#instanceCount++;
  }

  // Private method
  #hashPassword(raw) {
    return raw.split("").reverse().join(""); // simplified
  }

  // Private getter
  get #maskedPassword() {
    return "*".repeat(this.#password.length);
  }

  changePassword(oldPw, newPw) {
    if (this.#password !== oldPw) {
      throw new Error("Wrong password");
    }
    this.#password = this.#hashPassword(newPw);
  }

  toString() {
    return `${this.name} (${this.#maskedPassword})`;
  }

  static getCount() {
    return User.#instanceCount;
  }
}

const user = new User("Alice", "secret123");
console.log(user.role);       // "viewer"
console.log(user.name);       // "Alice"
// console.log(user.#password); // SyntaxError: Private field
```

**Private field checks with `in`:**

```javascript
class Color {
  #red;
  #green;
  #blue;

  constructor(r, g, b) {
    this.#red = r;
    this.#green = g;
    this.#blue = b;
  }

  static isColor(obj) {
    // Check if obj has the private field (brand check)
    return #red in obj;
  }

  equals(other) {
    if (!Color.isColor(other)) return false;
    return this.#red === other.#red
      && this.#green === other.#green
      && this.#blue === other.#blue;
  }
}

Color.isColor(new Color(0, 0, 0)); // true
Color.isColor({ red: 0 });         // false
```

**Static blocks** are useful for complex one-time initialization:

```javascript
class Database {
  static connection;
  static isReady;

  static {
    try {
      Database.connection = connectSync("localhost:5432");
      Database.isReady = true;
    } catch {
      Database.connection = null;
      Database.isReady = false;
    }
  }
}
```

### `at()` Method for Indexable Values

Access elements by index, supporting negative indices to count from the end.

```javascript
const arr = [10, 20, 30, 40, 50];

// Positive index (same as bracket notation)
arr.at(0);  // 10
arr.at(2);  // 30

// Negative index (count from end)
arr.at(-1); // 50  (last element)
arr.at(-2); // 40  (second to last)

// Compare with the old way
arr[arr.length - 1]; // 50 (verbose)
arr.at(-1);          // 50 (clean)
```

Works on strings and TypedArrays too:

```javascript
"hello".at(-1);                    // "o"
new Uint8Array([1, 2, 3]).at(-1);  // 3
```

**Gotcha:** `at()` returns `undefined` for out-of-range indices, just like bracket notation. It does NOT throw.

### `Object.hasOwn()`

A static replacement for `Object.prototype.hasOwnProperty.call()`.

```javascript
const obj = { name: "Alice", age: 30 };

// Old way (verbose, required .call for safety)
Object.prototype.hasOwnProperty.call(obj, "name"); // true

// ES2022 way
Object.hasOwn(obj, "name"); // true
Object.hasOwn(obj, "email"); // false
```

Why not just `obj.hasOwnProperty()`? It fails on objects with no prototype:

```javascript
const bare = Object.create(null);
bare.key = "value";

// bare.hasOwnProperty("key"); // TypeError: not a function
Object.hasOwn(bare, "key"); // true (always works)
```

### Error `cause`

Chain errors with a `cause` property to preserve the original error.

```javascript
async function fetchUser(id) {
  try {
    const response = await fetch(`/api/users/${id}`);
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}`);
    }
    return await response.json();
  } catch (err) {
    // Wrap with context, preserve original cause
    throw new Error(`Failed to fetch user ${id}`, { cause: err });
  }
}

try {
  await fetchUser(42);
} catch (err) {
  console.log(err.message);       // "Failed to fetch user 42"
  console.log(err.cause.message); // "HTTP 404" (or network error)

  // Causes can be chained indefinitely
  let e = err;
  while (e) {
    console.log(e.message);
    e = e.cause;
  }
}
```

The `cause` can be any value, not just an `Error`:

```javascript
throw new Error("Validation failed", { cause: { field: "email", rule: "format" } });
```

### `RegExp` Match Indices (`d` Flag)

The `d` flag adds start and end indices for each captured group.

```javascript
const re = /(?<year>\d{4})-(?<month>\d{2})-(?<day>\d{2})/d;
const match = re.exec("Date: 2025-03-15");

// match.indices[0] = [6, 16]  (full match start/end)
// match.indices[1] = [6, 10]  (year group)
// match.indices[2] = [11, 13] (month group)
// match.indices[3] = [14, 16] (day group)

// Named group indices
// match.indices.groups.year  = [6, 10]
// match.indices.groups.month = [11, 13]
// match.indices.groups.day   = [14, 16]
```

Useful for syntax highlighters and editors where you need exact character positions.

## ES2023

ES2023 was finalized in June 2023.

### Array Find from Last

`findLast()` and `findLastIndex()` search from the end of the array.

```javascript
const transactions = [
  { id: 1, type: "credit", amount: 100 },
  { id: 2, type: "debit", amount: 50 },
  { id: 3, type: "credit", amount: 200 },
  { id: 4, type: "debit", amount: 75 },
];

// Find last credit transaction
const lastCredit = transactions.findLast((t) => t.type === "credit");
// { id: 3, type: "credit", amount: 200 }

const lastCreditIndex = transactions.findLastIndex((t) => t.type === "credit");
// 2
```

Compare with the old approach:

```javascript
// BAD PRACTICE: reverse + find (mutates or copies the array)
const lastCredit = [...transactions].reverse().find((t) => t.type === "credit");

// GOOD: findLast (no copy, no mutation, efficient)
const lastCredit = transactions.findLast((t) => t.type === "credit");
```

### Immutable Array Methods

Four new methods that return new arrays instead of mutating the original: `toSorted()`, `toReversed()`, `toSpliced()`, and `with()`.

```javascript
const original = [3, 1, 4, 1, 5, 9];

// toSorted() - like sort() but returns a new array
const sorted = original.toSorted((a, b) => a - b);
console.log(sorted);   // [1, 1, 3, 4, 5, 9]
console.log(original); // [3, 1, 4, 1, 5, 9] (unchanged)

// toReversed() - like reverse() but returns a new array
const reversed = original.toReversed();
console.log(reversed); // [9, 5, 1, 4, 1, 3]
console.log(original); // [3, 1, 4, 1, 5, 9] (unchanged)

// toSpliced() - like splice() but returns a new array
const spliced = original.toSpliced(2, 1, 99, 100);
console.log(spliced);  // [3, 1, 99, 100, 1, 5, 9]
console.log(original); // [3, 1, 4, 1, 5, 9] (unchanged)

// with() - replace element at index, returns new array
const replaced = original.with(0, 42);
console.log(replaced); // [42, 1, 4, 1, 5, 9]
console.log(original); // [3, 1, 4, 1, 5, 9] (unchanged)
```

`with()` supports negative indices:

```javascript
const arr = [1, 2, 3, 4, 5];
arr.with(-1, 99); // [1, 2, 3, 4, 99]
```

**Why this matters:** Immutable operations are essential for state management patterns and frameworks that rely on reference equality checks to detect changes. Before ES2023, you had to spread-copy before mutating:

```javascript
// BAD PRACTICE: Old way, verbose and error-prone
const sorted = [...arr].sort((a, b) => a - b);

// GOOD: ES2023 way
const sorted = arr.toSorted((a, b) => a - b);
```

### Hashbang (`#!`) Grammar

The spec formally allows `#!` (hashbang/shebang) at the start of a file, so JavaScript files can be directly executable:

```javascript
#!/usr/bin/env node
console.log("I can be run directly: ./script.js");
```

This was already supported by engines but is now formally part of the spec. The `#!` comment is stripped before parsing.

### WeakRef and Symbol as WeakMap Keys

Symbols (non-registered, i.e. not created by `Symbol.for()`) can be used as `WeakMap` and `WeakSet` keys:

```javascript
const key = Symbol("myKey");
const weakMap = new WeakMap();

weakMap.set(key, { data: "associated value" });
weakMap.get(key); // { data: "associated value" }

// Symbol.for() keys are NOT allowed (they are globally registered and never GC'd)
// weakMap.set(Symbol.for("global"), "value"); // TypeError
```

### Change Array by Copy: Full Comparison

| Mutating Method | Non-Mutating ES2023 Equivalent |
|----------------|-------------------------------|
| `sort()` | `toSorted()` |
| `reverse()` | `toReversed()` |
| `splice()` | `toSpliced()` |
| `arr[i] = value` | `with(index, value)` |

All four new methods work on any array-like with a `length` property (via `Array.prototype`).

## ES2024

ES2024 was finalized in June 2024.

### `Object.groupBy()` and `Map.groupBy()`

Group iterable elements by a key returned from a callback.

```javascript
const inventory = [
  { name: "apples", type: "fruit", quantity: 5 },
  { name: "bananas", type: "fruit", quantity: 12 },
  { name: "carrots", type: "vegetable", quantity: 8 },
  { name: "broccoli", type: "vegetable", quantity: 3 },
  { name: "salmon", type: "protein", quantity: 2 },
];

// Object.groupBy returns a null-prototype plain object
const grouped = Object.groupBy(inventory, (item) => item.type);

console.log(grouped);
// {
//   fruit: [{ name: "apples", ... }, { name: "bananas", ... }],
//   vegetable: [{ name: "carrots", ... }, { name: "broccoli", ... }],
//   protein: [{ name: "salmon", ... }],
// }

// Access groups directly
grouped.fruit.length; // 2
```

`Map.groupBy()` returns a `Map`, useful when keys are not strings:

```javascript
const items = [
  { name: "shirt", price: 25 },
  { name: "pants", price: 65 },
  { name: "hat", price: 15 },
  { name: "shoes", price: 90 },
];

const byRange = Map.groupBy(items, (item) => {
  if (item.price < 30) return "budget";
  if (item.price < 70) return "mid";
  return "premium";
});

byRange.get("budget");  // [{ name: "shirt", ... }, { name: "hat", ... }]
byRange.get("premium"); // [{ name: "shoes", ... }]
```

Using objects as keys (only possible with `Map.groupBy`):

```javascript
const lowStock = { label: "low" };
const highStock = { label: "high" };

const byStock = Map.groupBy(inventory, (item) =>
  item.quantity < 5 ? lowStock : highStock
);

byStock.get(lowStock);  // items with quantity < 5
byStock.get(highStock); // items with quantity >= 5
```

**Gotchas:**
- `Object.groupBy()` returns a null-prototype object (no `hasOwnProperty`, `toString`, etc.).
- The callback receives `(element, index)` just like `map()`.
- The callback's return value is coerced to a property key (string for `Object.groupBy`, any value for `Map.groupBy`).

Compare with the old approach:

```javascript
// BAD PRACTICE: Manual grouping with reduce (verbose, error-prone)
const grouped = items.reduce((acc, item) => {
  const key = item.type;
  (acc[key] ??= []).push(item);
  return acc;
}, {});

// GOOD: Object.groupBy
const grouped = Object.groupBy(items, (item) => item.type);
```

### `Promise.withResolvers()`

Extract `resolve` and `reject` from a Promise without the executor callback pattern.

```javascript
// Before ES2024: The deferred pattern was ugly
let resolve, reject;
const promise = new Promise((res, rej) => {
  resolve = res;
  reject = rej;
});

// ES2024: Clean extraction
const { promise, resolve, reject } = Promise.withResolvers();

// Now you can resolve/reject from anywhere
setTimeout(() => resolve("done!"), 1000);
const result = await promise; // "done!"
```

Real-world use: event-to-promise adapter:

```javascript
function waitForEvent(target, eventName) {
  const { promise, resolve, reject } = Promise.withResolvers();

  const cleanup = () => {
    target.removeEventListener(eventName, onEvent);
    target.removeEventListener("error", onError);
  };

  const onEvent = (e) => { cleanup(); resolve(e); };
  const onError = (e) => { cleanup(); reject(e); };

  target.addEventListener(eventName, onEvent, { once: true });
  target.addEventListener("error", onError, { once: true });

  return promise;
}

const event = await waitForEvent(button, "click");
```

### `String.isWellFormed()` and `String.toWellFormed()`

Detect and fix lone surrogates in strings.

```javascript
// Well-formed: valid Unicode
"hello".isWellFormed();     // true
"hello\uD800".isWellFormed(); // false (lone surrogate)

// Fix lone surrogates by replacing with U+FFFD (replacement character)
"hello\uD800".toWellFormed(); // "hello\uFFFD"

// Why it matters: APIs like encodeURIComponent throw on lone surrogates
try {
  encodeURIComponent("hello\uD800"); // URIError
} catch (e) {
  const safe = "hello\uD800".toWellFormed();
  encodeURIComponent(safe); // works: "hello%EF%BF%BD"
}
```

### `Atomics.waitAsync()`

An async version of `Atomics.wait()` for use in the main thread (since `Atomics.wait()` blocks and is only allowed in workers).

```javascript
const sab = new SharedArrayBuffer(1024);
const int32 = new Int32Array(sab);

// Non-blocking wait on the main thread
const result = Atomics.waitAsync(int32, 0, 0);

if (result.async) {
  result.value.then((status) => {
    console.log(status); // "ok" when value changed, "timed-out", or "not-equal"
  });
} else {
  console.log(result.value); // synchronous result ("not-equal")
}
```

This is primarily used in multi-threaded applications with Web Workers and SharedArrayBuffer.

### `ArrayBuffer.prototype.resize()` and `transfer()`

Resizable ArrayBuffers and zero-copy transfers:

```javascript
// Resizable ArrayBuffer
const buffer = new ArrayBuffer(8, { maxByteLength: 64 });
console.log(buffer.byteLength);    // 8
console.log(buffer.maxByteLength); // 64
console.log(buffer.resizable);     // true

buffer.resize(16); // grow to 16 bytes
buffer.resize(4);  // shrink to 4 bytes

// Transfer ownership (zero-copy move)
const newBuffer = buffer.transfer();
// buffer is now detached (zero-length, unusable)
// newBuffer has the data

// Transfer with new size
const bigger = buffer.transfer(32);
```

### `RegExp` `v` Flag (Unicode Sets)

The `v` flag is an upgraded `u` flag with set operations on character classes:

```javascript
// Intersection: characters that are BOTH Greek AND letters
const greekLetters = /[\p{Script=Greek}&&\p{Letter}]/v;

// Difference: letters that are NOT ASCII
const nonAsciiLetters = /[\p{Letter}--[a-zA-Z]]/v;

// Union (already possible, but cleaner with v flag)
const vowels = /[[aeiou][AEIOU]]/v;

// String properties (multi-character matches)
const emoji = /\p{RGI_Emoji}/v;
emoji.test("👨‍👩‍👧‍👦"); // true (matches the full emoji sequence)
```

The `v` flag replaces `u` and cannot be used together with it.

## ES2025

ES2025 was finalized in June 2025.

### Set Methods

Seven new methods for `Set` that perform common set theory operations, returning new Sets (non-mutating):

```javascript
const frontEnd = new Set(["HTML", "CSS", "JavaScript", "TypeScript"]);
const backEnd = new Set(["JavaScript", "TypeScript", "Python", "Go"]);

// union() - all elements from both sets
frontEnd.union(backEnd);
// Set {"HTML", "CSS", "JavaScript", "TypeScript", "Python", "Go"}

// intersection() - elements in both sets
frontEnd.intersection(backEnd);
// Set {"JavaScript", "TypeScript"}

// difference() - elements in first but not second
frontEnd.difference(backEnd);
// Set {"HTML", "CSS"}

// symmetricDifference() - elements in either but not both
frontEnd.symmetricDifference(backEnd);
// Set {"HTML", "CSS", "Python", "Go"}

// isSubsetOf() - all elements of first are in second
new Set(["JavaScript"]).isSubsetOf(frontEnd); // true

// isSupersetOf() - first contains all elements of second
frontEnd.isSupersetOf(new Set(["HTML", "CSS"])); // true

// isDisjointFrom() - no elements in common
new Set(["Rust", "Zig"]).isDisjointFrom(frontEnd); // true
```

All methods accept any iterable (not just Sets) as the argument:

```javascript
const mySet = new Set([1, 2, 3]);
mySet.union([3, 4, 5]);           // Set {1, 2, 3, 4, 5}
mySet.intersection([2, 3, 4]);    // Set {2, 3}
```

**Comparison with the old approach:**

```javascript
// BAD PRACTICE: Manual set operations (verbose and error-prone)
function union(a, b) {
  return new Set([...a, ...b]);
}
function intersection(a, b) {
  return new Set([...a].filter((x) => b.has(x)));
}
function difference(a, b) {
  return new Set([...a].filter((x) => !b.has(x)));
}

// GOOD: ES2025 built-in methods
const result = setA.union(setB);
const common = setA.intersection(setB);
const diff = setA.difference(setB);
```

### Iterator Helpers

New methods on iterator objects for lazy transformation and consumption, similar to array methods but without creating intermediate arrays:

```javascript
// Iterator.prototype methods:
// .map(), .filter(), .take(), .drop(), .flatMap()
// .reduce(), .toArray(), .forEach(), .some(), .every(), .find()

function* naturals() {
  let n = 1;
  while (true) yield n++;
}

// Lazy chain: no intermediate arrays, works with infinite iterators
const result = naturals()
  .filter((n) => n % 2 === 0)  // even numbers: 2, 4, 6, ...
  .map((n) => n ** 2)           // squared: 4, 16, 36, ...
  .take(5)                      // first 5: 4, 16, 36, 64, 100
  .toArray();                   // materialize

console.log(result); // [4, 16, 36, 64, 100]
```

Each helper returns a new iterator (lazy), except for the "consuming" methods:

| Method | Returns | Lazy? |
|--------|---------|-------|
| `.map(fn)` | Iterator | Yes |
| `.filter(fn)` | Iterator | Yes |
| `.take(n)` | Iterator | Yes |
| `.drop(n)` | Iterator | Yes |
| `.flatMap(fn)` | Iterator | Yes |
| `.reduce(fn, init)` | Value | No (consumes) |
| `.toArray()` | Array | No (consumes) |
| `.forEach(fn)` | undefined | No (consumes) |
| `.some(fn)` | boolean | No (consumes, short-circuits) |
| `.every(fn)` | boolean | No (consumes, short-circuits) |
| `.find(fn)` | value/undefined | No (consumes, short-circuits) |

**Working with existing iterables:**

```javascript
// Maps, Sets, and other iterables have iterator helpers
const userMap = new Map([
  ["alice", { age: 30 }],
  ["bob", { age: 25 }],
  ["charlie", { age: 35 }],
]);

const youngUsers = userMap
  .values()
  .filter((user) => user.age < 32)
  .toArray();
// [{ age: 30 }, { age: 25 }]

// DOM NodeLists (when iterable)
const visibleDivs = document.querySelectorAll("div")
  .values()
  .filter((el) => el.offsetHeight > 0)
  .take(10)
  .toArray();
```

**`Iterator.from()` wraps any iterable or iterator-like object:**

```javascript
// Wrap a plain iterable
const iter = Iterator.from([1, 2, 3]);
iter.map((n) => n * 2).toArray(); // [2, 4, 6]

// Wrap an iterator-like object (has next() but no Symbol.iterator)
const iterLike = { next() { return { value: 1, done: false }; } };
const proper = Iterator.from(iterLike);
proper.take(3).toArray(); // [1, 1, 1]
```

**`flatMap` example:**

```javascript
function* words(sentence) {
  yield* sentence.split(" ");
}

const sentences = ["hello world", "foo bar baz"];
const allWords = sentences
  .values()
  .flatMap((s) => words(s))
  .toArray();
// ["hello", "world", "foo", "bar", "baz"]
```

### `RegExp.escape()`

Safely escape a string for use inside a regular expression.

```javascript
const userInput = "price is $9.99 (USD)";

// Before ES2025: manual escaping (easy to get wrong)
function escapeRegExp(str) {
  return str.replace(/[.*+?^${}()|[\]\\]/g, "\\$&");
}

// ES2025: built-in
const escaped = RegExp.escape(userInput);
// "price\\ is\\ \\$9\\.99\\ \\(USD\\)"

const re = new RegExp(escaped);
re.test("price is $9.99 (USD)"); // true
```

This eliminates a common source of bugs and security issues when building regexes from user input.

### Duplicate Named Capture Groups

Named capture groups can reuse the same name in different alternatives of a regex:

```javascript
// Before ES2025: SyntaxError to reuse group names
// /(?<year>\d{4})-\d{2}|\d{2}-(?<year>\d{4})/  // ERROR

// ES2025: allowed in different alternatives (separated by |)
const dateRe = /(?<year>\d{4})-(?<month>\d{2})|(?<month>\d{2})\/(?<year>\d{4})/;

"2025-03".match(dateRe).groups;
// { year: "2025", month: "03" }

"03/2025".match(dateRe).groups;
// { year: "2025", month: "03" }
```

Both formats produce the same named groups, simplifying parsing logic when input formats vary.

### Pattern Modifiers (Inline Regex Flags)

Apply flags to specific parts of a regex pattern:

```javascript
// (?i:...) makes a section case-insensitive
const re = /hello (?i:world)/; // "world" is case-insensitive, "hello" is not

re.test("hello World"); // true
re.test("hello WORLD"); // true
re.test("Hello world"); // false ("hello" is case-sensitive)

// (?-i:...) turns off case-insensitivity in a section
const re2 = /(?i:hello) (?-i:world)/;
re2.test("HELLO world"); // true
re2.test("HELLO WORLD"); // false
```

Supported modifiers: `i` (case-insensitive), `m` (multiline), `s` (dotAll).

### `Promise.try()`

Wrap a function call in a promise, catching synchronous errors too:

```javascript
// Problem: sync errors are not caught by .catch()
function mightThrowSync() {
  throw new Error("sync error");
}

// BAD PRACTICE: sync throw escapes the promise chain
Promise.resolve()
  .then(() => mightThrowSync())
  .catch((err) => console.log(err)); // works but awkward

// GOOD: Promise.try catches both sync and async errors
Promise.try(() => mightThrowSync())
  .catch((err) => console.log(err.message)); // "sync error"

// Works with async functions too
Promise.try(async () => {
  const data = await fetchData();
  return processData(data); // sync throw here is also caught
}).catch(handleError);
```

This unifies error handling for functions that might be sync or async:

```javascript
function getFromCache(key) {
  // Might return a value synchronously from cache
  // or fetch it asynchronously
  const cached = cache.get(key);
  if (cached) return cached;
  return fetch(`/api/${key}`).then((r) => r.json());
}

// Promise.try handles both paths
Promise.try(() => getFromCache("user"))
  .then((data) => render(data))
  .catch((err) => showError(err));
```

### Import Attributes

Specify metadata about imported modules (replaces the older "import assertions" syntax that used `assert`):

```javascript
// Import JSON
import config from "./config.json" with { type: "json" };
console.log(config.apiUrl);

// Import CSS (where supported)
import styles from "./styles.css" with { type: "css" };
document.adoptedStyleSheets.push(styles);

// Dynamic import with attributes
const data = await import("./data.json", { with: { type: "json" } });
```

**Why `with` instead of `assert`?** The keyword changed from `assert` to `with` because "assert" implied the import would succeed and only be type-checked afterward. `with` correctly indicates it affects how the module is loaded.

**Gotcha:** Browser and runtime support varies. JSON imports with `with { type: "json" }` are widely supported. CSS imports are less broadly available.

### `Float16Array` and `Math.f16round()`

Half-precision (16-bit) floating-point support:

```javascript
// Float16Array for memory-efficient storage
const f16 = new Float16Array([1.5, 2.5, 3.5]);
console.log(f16[0]);          // 1.5
console.log(f16.BYTES_PER_ELEMENT); // 2

// Math.f16round - round to float16 precision
Math.f16round(1.337); // 1.3369140625

// DataView methods
const buffer = new ArrayBuffer(2);
const view = new DataView(buffer);
view.setFloat16(0, 3.14);
view.getFloat16(0); // 3.140625 (float16 precision)
```

Useful for GPU data, machine learning tensors, and other scenarios where memory is constrained and full 32/64-bit precision is unnecessary.

## Common Mistakes

- **Using `sort()` when you need `toSorted()`**: Forgetting that `sort()` mutates the original array. This is a frequent source of bugs in state management where the original array reference matters.

```javascript
// BAD PRACTICE: mutates state
const sorted = state.items.sort((a, b) => a.price - b.price);
// state.items is now also sorted (same reference!)

// GOOD: returns a new array
const sorted = state.items.toSorted((a, b) => a.price - b.price);
// state.items is unchanged
```

- **Confusing `Object.groupBy()` result with a regular object**: It returns a null-prototype object, so methods like `hasOwnProperty()` and `toString()` do not exist on it. Use `Object.hasOwn()` or `in`.

- **Using `at()` expecting it to throw on out-of-range access**: `at()` returns `undefined` for out-of-range indices, not an error. It behaves just like bracket notation for positive indices.

- **Forgetting that top-level `await` blocks importers**: A module using top-level `await` delays execution of every module that imports it. Do not use it for long-running operations that block app startup.

- **Using `with { type: "json" }` in environments that do not support it yet**: Import attributes are part of the spec but not universally shipped. Check your target environment.

- **Not chaining Error causes**: When rethrowing errors, always use the `cause` option. Otherwise the original error and its stack trace are lost.

```javascript
// BAD PRACTICE: original error is lost
catch (err) {
  throw new Error("Operation failed");
}

// GOOD: preserve the cause chain
catch (err) {
  throw new Error("Operation failed", { cause: err });
}
```

- **Consuming an iterator helper chain multiple times**: Iterator helpers produce one-shot iterators. Once consumed (by `toArray()`, `reduce()`, `forEach()`, etc.), they are exhausted.

```javascript
const iter = [1, 2, 3].values().map((n) => n * 2);
iter.toArray(); // [2, 4, 6]
iter.toArray(); // [] (already consumed!)
```

---

## Best Practices

- **Prefer immutable array methods** (`toSorted`, `toReversed`, `toSpliced`, `with`) over their mutating counterparts in any code where the original array should not change.

- **Use `Object.hasOwn()` over `hasOwnProperty()`** in all new code. It works on null-prototype objects and is shorter.

- **Always provide Error `cause`** when wrapping exceptions. This creates debuggable error chains.

- **Use `Object.groupBy()` instead of `reduce()` for grouping.** It is more readable, less error-prone, and communicates intent.

- **Use `Promise.withResolvers()`** whenever you need to resolve/reject a promise from outside its constructor. This is cleaner than the old "deferred" pattern.

- **Use iterator helpers for large or infinite data sets.** They avoid creating intermediate arrays, which reduces memory usage and can improve performance.

- **Use `RegExp.escape()` whenever building regexes from dynamic strings.** Never hand-roll regex escaping.

- **Use `findLast()` and `findLastIndex()`** instead of reversing arrays to find the last matching element.

## Real-World Use Cases

### 1. Data Processing Pipeline with Iterator Helpers

```javascript
// Process a large CSV file line-by-line without loading it all into memory
async function* readLines(url) {
  const response = await fetch(url);
  const reader = response.body.getReader();
  const decoder = new TextDecoder();
  let buffer = "";

  while (true) {
    const { value, done } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split("\n");
    buffer = lines.pop();
    for (const line of lines) yield line;
  }
  if (buffer) yield buffer;
}

// Lazy pipeline: filter, transform, take only what we need
const topExpensiveProducts = await readLines("/data/products.csv")
  .drop(1)                                      // skip header
  .map((line) => line.split(","))                // parse CSV
  .map(([name, , price]) => ({ name, price: +price }))
  .filter((p) => p.price > 100)                 // only expensive items
  .take(10)                                     // first 10 matches
  .toArray();
```

### 2. Robust Error Handling with cause Chains

```javascript
class ApiError extends Error {
  constructor(message, statusCode, options) {
    super(message, options); // passes { cause } through
    this.statusCode = statusCode;
  }
}

async function fetchJSON(url) {
  try {
    const response = await fetch(url);
    if (!response.ok) {
      throw new ApiError(
        `API returned ${response.status}`,
        response.status
      );
    }
    return await response.json();
  } catch (err) {
    throw new ApiError("Network request failed", 0, { cause: err });
  }
}

async function getUser(id) {
  try {
    return await fetchJSON(`/api/users/${id}`);
  } catch (err) {
    throw new Error(`Failed to load user ${id}`, { cause: err });
  }
}

// Full error chain for debugging
try {
  await getUser(42);
} catch (err) {
  // Walk the cause chain
  let current = err;
  while (current) {
    console.error(current.message);
    if (current instanceof ApiError) {
      console.error("  Status:", current.statusCode);
    }
    current = current.cause;
  }
}
```

### 3. State Management with Immutable Methods and Grouping

```javascript
const state = {
  tasks: [
    { id: 1, title: "Write docs", status: "done", priority: 3 },
    { id: 2, title: "Fix bug", status: "todo", priority: 1 },
    { id: 3, title: "Add tests", status: "in-progress", priority: 2 },
    { id: 4, title: "Deploy", status: "todo", priority: 1 },
    { id: 5, title: "Review PR", status: "done", priority: 2 },
  ],
};

// Sort by priority without mutating state
const sortedTasks = state.tasks.toSorted((a, b) => a.priority - b.priority);

// Group by status
const byStatus = Object.groupBy(state.tasks, (task) => task.status);
// { done: [...], todo: [...], "in-progress": [...] }

// Update a single task immutably
const index = state.tasks.findIndex((t) => t.id === 3);
const updatedTasks = state.tasks.with(index, {
  ...state.tasks[index],
  status: "done",
});

// Find the last completed task
const lastDone = state.tasks.findLast((t) => t.status === "done");
// { id: 5, title: "Review PR", status: "done", priority: 2 }

// Deduplicate tags using Set methods
const myTags = new Set(["javascript", "typescript", "css"]);
const trending = new Set(["rust", "typescript", "go"]);

const allTags = myTags.union(trending);
// Set {"javascript", "typescript", "css", "rust", "go"}

const shared = myTags.intersection(trending);
// Set {"typescript"}

const uniqueToMe = myTags.difference(trending);
// Set {"javascript", "css"}
```
