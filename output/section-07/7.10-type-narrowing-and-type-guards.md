# 7.10 Type Narrowing and Type Guards

> Type narrowing is how TypeScript refines a broad type to a more specific one based on runtime checks in your code. Type guards are the conditions that trigger this refinement. Mastering narrowing is the key to writing type-safe code without excessive type assertions.

---

## What Is Narrowing?

Narrowing is the process by which TypeScript analyzes your control flow (if statements, switch cases, loops, assignments) and automatically refines a type from a wider type to a more specific one.

```typescript
function process(value: string | number) {
  // Here, value is string | number

  if (typeof value === "string") {
    // TypeScript narrows value to string
    console.log(value.toUpperCase());
  } else {
    // TypeScript narrows value to number
    console.log(value.toFixed(2));
  }
}
```

TypeScript does this through **control flow analysis** (CFA). It follows every possible execution path through your code and tracks what the type of each variable must be at each point.

## `typeof` Guards

The `typeof` operator returns a string identifying the runtime type of a value. TypeScript uses it to narrow primitives.

### Recognized `typeof` Results

| Check | Narrows to |
|-------|-----------|
| `typeof x === "string"` | `string` |
| `typeof x === "number"` | `number` |
| `typeof x === "bigint"` | `bigint` |
| `typeof x === "boolean"` | `boolean` |
| `typeof x === "symbol"` | `symbol` |
| `typeof x === "undefined"` | `undefined` |
| `typeof x === "function"` | Function types |
| `typeof x === "object"` | `object \| null` |

### Basic Usage

```typescript
function stringify(value: string | number | boolean): string {
  if (typeof value === "string") {
    return value; // value is string
  }
  if (typeof value === "number") {
    return value.toString(); // value is number
  }
  return value ? "true" : "false"; // value is boolean
}
```

### The `typeof null` Gotcha

JavaScript's `typeof null` returns `"object"`. TypeScript knows this and does NOT narrow `null` out when checking `typeof x === "object"`.

```typescript
function process(value: string | object | null) {
  if (typeof value === "object") {
    // value is object | null (NOT just object!)
    // value.toString(); // Possible error if value is null

    // Must handle null separately
    if (value !== null) {
      console.log(value.toString()); // Now value is object
    }
  }
}
```

### Negated `typeof` Checks

```typescript
function process(value: string | number | boolean) {
  if (typeof value !== "string") {
    // value is number | boolean
    if (typeof value !== "number") {
      // value is boolean
      console.log(value ? "yes" : "no");
    }
  }
}
```

## `instanceof` Guards

The `instanceof` operator checks whether an object's prototype chain contains a specific constructor. TypeScript narrows accordingly.

```typescript
function handleError(error: Error | string) {
  if (error instanceof Error) {
    // error is Error
    console.log(error.message);
    console.log(error.stack);
  } else {
    // error is string
    console.log(error);
  }
}
```

### Subclass Narrowing

```typescript
class HttpError extends Error {
  constructor(
    public statusCode: number,
    message: string,
  ) {
    super(message);
  }
}

class ValidationError extends Error {
  constructor(
    public fields: string[],
    message: string,
  ) {
    super(message);
  }
}

function handleError(error: Error) {
  if (error instanceof HttpError) {
    // error is HttpError
    console.log(`HTTP ${error.statusCode}: ${error.message}`);
  } else if (error instanceof ValidationError) {
    // error is ValidationError
    console.log(`Invalid fields: ${error.fields.join(", ")}`);
  } else {
    // error is Error (base class)
    console.log(error.message);
  }
}
```

### `instanceof` with Built-in Types

```typescript
function processValue(value: unknown) {
  if (value instanceof Date) {
    console.log(value.toISOString()); // value is Date
  } else if (value instanceof RegExp) {
    console.log(value.source); // value is RegExp
  } else if (value instanceof Map) {
    console.log(value.size); // value is Map<unknown, unknown>
  } else if (value instanceof Array) {
    console.log(value.length); // value is any[]
  }
}
```

## Truthiness Narrowing

JavaScript's truthiness rules narrow out `null`, `undefined`, `0`, `""`, `NaN`, and `false`.

```typescript
function printName(name: string | null | undefined) {
  if (name) {
    // name is string (null and undefined are falsy)
    console.log(name.toUpperCase());
  }
}
```

### Narrowing in Logical Operators

```typescript
function getLength(value: string | null): number {
  // && narrows the left side before evaluating the right
  return value && value.length || 0;
}

// Optional chaining does NOT narrow (it is not a type guard)
function getLength2(value: string | null): number | undefined {
  return value?.length; // value is still string | null
}
```

### The Truthiness Trap with Falsy Values

```typescript
function processCount(count: number | null) {
  if (count) {
    // count is number, but 0 is EXCLUDED!
    console.log(count);
  }
  // If count is 0, this branch runs. But 0 is a valid count!
}

// Fix: check for null explicitly
function processCount(count: number | null) {
  if (count !== null) {
    // count is number (including 0)
    console.log(count);
  }
}
```

This is one of the most common narrowing bugs. Truthiness checks eliminate all falsy values, not just `null`/`undefined`. Use explicit equality checks when `0`, `""`, or `false` are valid values.

## Equality Narrowing

TypeScript uses equality checks (`===`, `!==`, `==`, `!=`) to narrow types.

### Strict Equality (`===` / `!==`)

```typescript
function process(value: string | number | null) {
  if (value === null) {
    // value is null
    return;
  }
  // value is string | number (null eliminated)

  if (value === "special") {
    // value is "special" (literal type)
    return;
  }
  // value is string | number
}
```

### Loose Equality (`==` / `!=`) with `null`

This is a useful trick. In JavaScript, `== null` catches both `null` and `undefined`.

```typescript
function process(value: string | null | undefined) {
  if (value == null) {
    // value is null | undefined
    return;
  }
  // value is string (both null and undefined eliminated)
  console.log(value.toUpperCase());
}

// Equivalent to:
function process2(value: string | null | undefined) {
  if (value != null) {
    // value is string
    console.log(value.toUpperCase());
  }
}
```

This is the one case where `==` is idiomatic in TypeScript: checking for both `null` and `undefined` in a single comparison.

### Comparing Two Variables

```typescript
function example(a: string | number, b: string | boolean) {
  if (a === b) {
    // The only type that overlaps: string
    // a is string, b is string
    console.log(a.toUpperCase());
    console.log(b.toUpperCase());
  }
}
```

## The `in` Operator

The `in` operator checks if a property exists on an object. TypeScript uses it to narrow union types of objects.

```typescript
type Fish = { swim: () => void; name: string };
type Bird = { fly: () => void; name: string };

function move(animal: Fish | Bird) {
  if ("swim" in animal) {
    // animal is Fish
    animal.swim();
  } else {
    // animal is Bird
    animal.fly();
  }
}
```

### `in` with Optional Properties

The `in` operator narrows to types that have the property as **required or optional**. If the property is optional, the value could still be `undefined`.

```typescript
type A = { x: number };
type B = { x?: number; y: number };

function process(value: A | B) {
  if ("y" in value) {
    // value is B
    console.log(value.y);
  } else {
    // value is A
    console.log(value.x);
  }
}
```

### `in` vs `typeof` for Property Checks

```typescript
type Circle = { kind: "circle"; radius: number };
type Square = { kind: "square"; side: number };

function area(shape: Circle | Square) {
  // Using discriminant (preferred)
  if (shape.kind === "circle") {
    return Math.PI * shape.radius ** 2;
  }

  // Using "in" (works but less precise)
  if ("radius" in shape) {
    return Math.PI * shape.radius ** 2;
  }
}
```

Discriminant checks are more explicit and self-documenting than `in` checks. Use `in` when there is no shared discriminant property.

## Discriminated Union Narrowing

Discriminated unions narrow through a shared literal property. This was introduced in 7.8 but is critical to narrowing, so here is the narrowing mechanics in detail.

```typescript
type Result<T> =
  | { status: "success"; data: T }
  | { status: "error"; message: string }
  | { status: "loading" };

function handleResult(result: Result<string>) {
  switch (result.status) {
    case "success":
      // result is { status: "success"; data: string }
      console.log(result.data.toUpperCase());
      break;

    case "error":
      // result is { status: "error"; message: string }
      console.error(result.message);
      break;

    case "loading":
      // result is { status: "loading" }
      console.log("Loading...");
      break;
  }
}
```

### Narrowing with `if` Chains

```typescript
function handleResult(result: Result<string>) {
  if (result.status === "success") {
    console.log(result.data);
    return;
  }

  if (result.status === "error") {
    console.error(result.message);
    return;
  }

  // result is { status: "loading" } here
  console.log("Loading...");
}
```

After each `return`, TypeScript removes the handled variant from the union. By the time you reach the end, only the remaining variant is possible.

## Assignment Narrowing

TypeScript narrows a variable's type when you assign a value to it.

```typescript
let value: string | number;

value = "hello";
// value is string here
console.log(value.toUpperCase());

value = 42;
// value is number here
console.log(value.toFixed(2));
```

### Assignment After Declaration

```typescript
let result: string | number | boolean;

// Initially, result is string | number | boolean
result = Math.random() > 0.5 ? "yes" : 42;
// After assignment, result is string | number (boolean eliminated)
```

TypeScript tracks the actual assigned value's type, not the declared type. But it will never narrow beyond the declared type.

## User-Defined Type Guards (Type Predicates)

When built-in narrowing is not enough, you can write your own type guard functions using the `is` keyword.

### Syntax

```typescript
function isString(value: unknown): value is string {
  return typeof value === "string";
}
```

The return type `value is string` is a **type predicate**. It tells TypeScript: "If this function returns `true`, then `value` is a `string`."

### Basic Type Guards

```typescript
type Cat = { meow(): void; whiskers: number };
type Dog = { bark(): void; breed: string };

function isCat(animal: Cat | Dog): animal is Cat {
  return "meow" in animal;
}

function isDog(animal: Cat | Dog): animal is Dog {
  return "bark" in animal;
}

function interact(animal: Cat | Dog) {
  if (isCat(animal)) {
    animal.meow(); // animal is Cat
  } else {
    animal.bark(); // animal is Dog
  }
}
```

### Guard for `unknown` Input

Type guards are especially useful for validating external data of type `unknown`.

```typescript
type User = {
  id: string;
  name: string;
  email: string;
};

function isUser(value: unknown): value is User {
  return (
    typeof value === "object" &&
    value !== null &&
    "id" in value &&
    "name" in value &&
    "email" in value &&
    typeof (value as User).id === "string" &&
    typeof (value as User).name === "string" &&
    typeof (value as User).email === "string"
  );
}

function processInput(data: unknown) {
  if (isUser(data)) {
    // data is User, fully type-safe
    console.log(data.name, data.email);
  }
}
```

### Array Filtering with Type Guards

The `.filter()` method accepts type predicates, allowing you to narrow array types.

```typescript
type Animal = Cat | Dog;

const animals: Animal[] = [/* ... */];

// Without type guard: result is Animal[]
const filtered = animals.filter((a) => "meow" in a);

// With type guard: result is Cat[]
const cats = animals.filter(isCat);
// cats is Cat[]

cats.forEach((cat) => cat.meow()); // fully typed
```

```typescript
// Filtering out null/undefined
const values: (string | null | undefined)[] = ["a", null, "b", undefined, "c"];

function isNotNullish<T>(value: T): value is NonNullable<T> {
  return value != null;
}

const strings = values.filter(isNotNullish);
// strings is string[]
```

### Generic Type Guards

```typescript
function isInstanceOf<T>(
  ctor: new (...args: any[]) => T,
): (value: unknown) => value is T {
  return (value): value is T => value instanceof ctor;
}

const isDate = isInstanceOf(Date);
const isError = isInstanceOf(Error);

const items: unknown[] = [new Date(), "hello", new Error("oops"), 42];

const dates = items.filter(isDate);
// dates is Date[]

const errors = items.filter(isError);
// errors is Error[]
```

## Assertion Functions

Assertion functions are functions that throw an error if a condition is not met. Unlike type predicates (which return `boolean`), assertion functions either return normally (narrowing the type) or throw.

### `asserts condition`

```typescript
function assert(condition: unknown, message?: string): asserts condition {
  if (!condition) {
    throw new Error(message ?? "Assertion failed");
  }
}

function process(value: string | null) {
  assert(value !== null, "Value must not be null");
  // After the assertion, value is string
  console.log(value.toUpperCase());
}
```

### `asserts value is Type`

```typescript
function assertIsString(value: unknown): asserts value is string {
  if (typeof value !== "string") {
    throw new TypeError(`Expected string, got ${typeof value}`);
  }
}

function assertIsUser(value: unknown): asserts value is User {
  if (
    typeof value !== "object" ||
    value === null ||
    !("id" in value) ||
    !("name" in value)
  ) {
    throw new TypeError("Invalid user object");
  }
}

function processInput(data: unknown) {
  assertIsUser(data);
  // data is User from this point forward (no if/else needed)
  console.log(data.name);
}
```

### Type Predicates vs Assertion Functions

| Feature | Type Predicate (`is`) | Assertion Function (`asserts`) |
|---------|----------------------|-------------------------------|
| Return type | `boolean` | `void` |
| When false | Returns `false` | Throws an error |
| Usage pattern | Inside `if` conditions | Standalone statement |
| Narrowing scope | Inside the `if` block | Rest of the scope |
| Error handling | Caller handles | Exception propagates |

```typescript
// Type predicate: caller decides what to do
function isString(value: unknown): value is string {
  return typeof value === "string";
}

if (isString(input)) {
  // narrowed here
} else {
  // handle the non-string case
}

// Assertion function: fail fast
function assertString(value: unknown): asserts value is string {
  if (typeof value !== "string") throw new TypeError("Not a string");
}

assertString(input);
// narrowed from here to end of scope
// no else branch needed
```

**Use type predicates** when you want conditional branching.
**Use assertion functions** when invalid data is a programming error that should crash early.

## Control Flow Analysis Details

### Narrowing Persists Through the Scope

```typescript
function example(value: string | number) {
  if (typeof value === "string") {
    // value is string from here...
    const upper = value.toUpperCase();
    const len = value.length;
    // ...to the end of this block
  }
  // value is back to string | number
}
```

### Early Returns Narrow the Remainder

```typescript
function process(value: string | number | null) {
  if (value === null) return;
  // value is string | number (null eliminated)

  if (typeof value === "string") return value.toUpperCase();
  // value is number

  return value.toFixed(2);
}
```

This is the "guard clause" pattern. Each early return eliminates a type from the union for all code that follows.

### Narrowing in Loops

```typescript
function firstString(values: (string | number)[]): string | undefined {
  for (const value of values) {
    if (typeof value === "string") {
      return value; // value is string
    }
  }
  return undefined;
}
```

### Narrowing Does NOT Persist Across Callbacks

This is a critical limitation.

```typescript
function process(value: string | null) {
  if (value !== null) {
    // value is string here

    setTimeout(() => {
      // value is string | null again!
      // TypeScript cannot guarantee value hasn't been reassigned
      // before this callback executes
      // value.toUpperCase(); // Error if value is let-declared
    }, 1000);
  }
}
```

**Why?** The callback might execute later, after `value` could have been reassigned. TypeScript is conservative and resets the narrowing.

**Fix: capture the narrowed value in a `const`:**

```typescript
function process(value: string | null) {
  if (value !== null) {
    const narrowed = value; // const, cannot be reassigned
    setTimeout(() => {
      // narrowed is string (narrowing preserved because it is const)
      console.log(narrowed.toUpperCase());
    }, 1000);
  }
}
```

Since TypeScript 5.4, narrowing is preserved for `const` variables in closures. But for `let` variables, narrowing is still lost.

## Narrowing `unknown`

The `unknown` type is the type-safe alternative to `any`. You must narrow it before using it.

```typescript
function processValue(value: unknown) {
  // Cannot do anything with value without narrowing
  // value.toString(); // Error: 'value' is of type 'unknown'

  // Narrow step by step
  if (typeof value === "string") {
    console.log(value.toUpperCase());
  } else if (typeof value === "number") {
    console.log(value.toFixed(2));
  } else if (value instanceof Date) {
    console.log(value.toISOString());
  } else if (Array.isArray(value)) {
    console.log(value.length);
  } else if (typeof value === "object" && value !== null) {
    console.log(Object.keys(value));
  }
}
```

### Validating External Data

```typescript
function parseApiResponse(raw: unknown): User {
  if (
    typeof raw !== "object" ||
    raw === null ||
    !("id" in raw) ||
    !("name" in raw) ||
    !("email" in raw)
  ) {
    throw new Error("Invalid API response");
  }

  // raw is narrowed to object & Record<"id" | "name" | "email", unknown>
  const { id, name, email } = raw as { id: unknown; name: unknown; email: unknown };

  if (typeof id !== "string" || typeof name !== "string" || typeof email !== "string") {
    throw new Error("Invalid field types");
  }

  return { id, name, email };
}
```

For complex validation, consider using a schema validation library (Zod, Valibot, ArkType) that generates TypeScript types from schemas.

## Common Mistakes

### Mistake 1: Using Truthiness to Check for `null` When Falsy Values Are Valid

```typescript
function getDiscount(discount: number | null): number {
  // Bad: 0 is a valid discount but is falsy
  if (discount) {
    return discount; // misses 0
  }
  return 0;

  // Good: explicit null check
  if (discount !== null) {
    return discount; // includes 0
  }
  return 0;
}
```

### Mistake 2: Forgetting `typeof null === "object"`

```typescript
function process(value: object | null) {
  // Bad: does not eliminate null
  if (typeof value === "object") {
    // value is still object | null!
  }

  // Good: check for null explicitly
  if (value !== null) {
    // value is object
  }
}
```

### Mistake 3: Type Guard Functions That Lie

TypeScript does NOT validate the logic inside a type guard function. It trusts your `is` predicate.

```typescript
// This compiles but is WRONG
function isString(value: unknown): value is string {
  return typeof value === "number"; // BUG: checking for number, not string
}

const value: unknown = 42;
if (isString(value)) {
  // TypeScript thinks value is string, but it is actually 42
  value.toUpperCase(); // Runtime crash!
}
```

Always ensure your type guard logic actually validates what the predicate claims.

### Mistake 4: Narrowing Lost in Callbacks

```typescript
let value: string | null = "hello";

if (value !== null) {
  // value is string here

  someAsyncFunction().then(() => {
    // value is string | null again (for let variables)
    // TypeScript cannot guarantee value was not reassigned
  });
}

// Fix: use const
const value2: string | null = getValue();
if (value2 !== null) {
  someAsyncFunction().then(() => {
    // value2 is still string (const cannot be reassigned)
    console.log(value2.toUpperCase());
  });
}
```

### Mistake 5: Narrowing the Wrong Variable

```typescript
type Response = { data: string } | { error: string };

function process(res: Response) {
  // Bad: checking a property that exists on both variants
  if ("data" in res) {
    // This narrows, but only if "data" is NOT on the other type
  }

  // When both types share property names, use discriminated unions
  type BetterResponse =
    | { ok: true; data: string }
    | { ok: false; error: string };
}
```

### Mistake 6: Over-Relying on `as` Instead of Narrowing

```typescript
function process(value: unknown) {
  // Bad: type assertion bypasses safety
  const str = value as string;
  console.log(str.toUpperCase()); // crashes if value is not a string

  // Good: narrow first
  if (typeof value === "string") {
    console.log(value.toUpperCase()); // safe
  }
}
```

## Best Practices

### 1. Prefer Built-in Guards Over Custom Guards When Possible

```typescript
// Good: typeof is simple and reliable
if (typeof value === "string") { /* ... */ }

// Only use custom guards when built-in guards are insufficient
function isUser(value: unknown): value is User {
  // Complex validation that typeof/instanceof cannot handle
}
```

### 2. Use Guard Clauses (Early Returns) for Clean Narrowing

```typescript
function processUser(user: User | null | undefined): string {
  if (user == null) return "No user";
  // user is User from here on
  return user.name;
}
```

### 3. Use `assertNever` for Exhaustive Switches

```typescript
function assertNever(value: never): never {
  throw new Error(`Unhandled value: ${JSON.stringify(value)}`);
}

function handle(action: Action) {
  switch (action.type) {
    case "add": /* ... */ break;
    case "remove": /* ... */ break;
    default: assertNever(action); // compile error if cases are missing
  }
}
```

### 4. Validate External Data at System Boundaries

```typescript
// Validate once at the boundary, then use typed values everywhere
function handleRequest(body: unknown): Response {
  assertIsCreateUserRequest(body); // throws if invalid
  // body is CreateUserRequest from here
  return createUser(body);
}
```

### 5. Use `const` Variables to Preserve Narrowing in Closures

```typescript
function setupHandler(config: Config | null) {
  if (config === null) return;

  // Capture in const to preserve narrowing in callbacks
  const validConfig = config;

  button.addEventListener("click", () => {
    // validConfig is Config (narrowing preserved)
    console.log(validConfig.apiUrl);
  });
}
```

### 6. Combine Multiple Narrow Checks with Logical Operators

```typescript
function process(value: unknown) {
  // Chain checks in a single if
  if (
    typeof value === "object" &&
    value !== null &&
    "name" in value &&
    typeof (value as any).name === "string"
  ) {
    // value is narrowed through each check
  }
}
```

## Real-World Use Cases

### API Response Handler with Discriminated Unions

```typescript
type ApiResult<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T; timestamp: number }
  | { status: "error"; error: string; code: number };

function renderResult<T>(result: ApiResult<T>, render: (data: T) => string): string {
  switch (result.status) {
    case "idle":
      return "Ready to fetch";
    case "loading":
      return "Loading...";
    case "success":
      return `${render(result.data)} (fetched at ${new Date(result.timestamp).toLocaleTimeString()})`;
    case "error":
      return `Error ${result.code}: ${result.error}`;
    default:
      return assertNever(result);
  }
}

function assertNever(value: never): never {
  throw new Error(`Unhandled: ${JSON.stringify(value)}`);
}
```

### Runtime Validation Layer

```typescript
// Reusable type guard builder
type Validator<T> = (value: unknown) => value is T;

function isString(value: unknown): value is string {
  return typeof value === "string";
}

function isNumber(value: unknown): value is number {
  return typeof value === "number" && !Number.isNaN(value);
}

function isObject(value: unknown): value is Record<string, unknown> {
  return typeof value === "object" && value !== null && !Array.isArray(value);
}

function hasShape<T extends Record<string, Validator<any>>>(
  validators: T,
): Validator<{ [K in keyof T]: T[K] extends Validator<infer V> ? V : never }> {
  return (value: unknown): value is any => {
    if (!isObject(value)) return false;
    for (const [key, validate] of Object.entries(validators)) {
      if (!validate(value[key])) return false;
    }
    return true;
  };
}

// Compose validators
const isUser = hasShape({
  id: isString,
  name: isString,
  age: isNumber,
});

// Usage
function handleData(raw: unknown) {
  if (isUser(raw)) {
    // raw is { id: string; name: string; age: number }
    console.log(`${raw.name} is ${raw.age} years old`);
  }
}
```

### Command Pattern with Assertion Guards

```typescript
type Command =
  | { type: "CREATE_USER"; payload: { name: string; email: string } }
  | { type: "DELETE_USER"; payload: { id: string } }
  | { type: "UPDATE_ROLE"; payload: { id: string; role: "admin" | "user" } };

function assertCommand(value: unknown): asserts value is Command {
  if (!isObject(value)) throw new Error("Command must be an object");
  if (typeof (value as any).type !== "string") throw new Error("Command must have a type");
  if (typeof (value as any).payload !== "object") throw new Error("Command must have a payload");

  const type = (value as any).type;
  const validTypes = ["CREATE_USER", "DELETE_USER", "UPDATE_ROLE"];
  if (!validTypes.includes(type)) {
    throw new Error(`Unknown command type: ${type}`);
  }
}

function isObject(value: unknown): value is Record<string, unknown> {
  return typeof value === "object" && value !== null;
}

// Process commands from external source (e.g., message queue)
function processRawCommand(raw: unknown) {
  assertCommand(raw);
  // raw is Command from here

  switch (raw.type) {
    case "CREATE_USER":
      console.log(`Creating user: ${raw.payload.name}`);
      break;
    case "DELETE_USER":
      console.log(`Deleting user: ${raw.payload.id}`);
      break;
    case "UPDATE_ROLE":
      console.log(`Updating role to ${raw.payload.role} for ${raw.payload.id}`);
      break;
  }
}
```

---

> **Key Takeaway:** TypeScript narrows types automatically through `typeof`, `instanceof`, `in`, equality checks, truthiness, and discriminated unions. When built-in narrowing is not enough, write user-defined type guards (`is`) for conditional branching or assertion functions (`asserts`) for fail-fast validation. Always narrow instead of asserting with `as`.
