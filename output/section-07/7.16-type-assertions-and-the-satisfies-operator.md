# 7.16 Type Assertions and the `satisfies` Operator

> Type assertions let you override what the compiler infers; `satisfies` lets you validate a value against a type without losing the inferred specificity. Knowing when to use each (and when to avoid both) is critical for writing safe TypeScript.

---

## Type Assertions with `as`

A type assertion tells the compiler "treat this value as this type." It performs **zero runtime checks**. It is purely a compile-time instruction that is erased in the emitted JavaScript.

```typescript
const input = document.getElementById("email");
// input is HTMLElement | null

const emailInput = document.getElementById("email") as HTMLInputElement;
// emailInput is HTMLInputElement (you asserted it)
```

### Two Syntax Forms

TypeScript supports two syntax forms for type assertions. They are functionally identical:

```typescript
// "as" syntax (preferred)
const a = someValue as SomeType;

// Angle-bracket syntax (cannot be used in .tsx files)
const b = <SomeType>someValue;
```

The angle-bracket syntax conflicts with JSX, so the `as` syntax is the universal standard. Use `as` everywhere for consistency.

### What Assertions Actually Do

Assertions do not convert, cast, or transform data. They change what type the **compiler sees**, nothing more:

```typescript
interface User {
  id: number;
  name: string;
  email: string;
}

const data = { id: 1, name: "Alice" };

// This compiles. But email is missing at runtime.
const user = data as User;

console.log(user.email); // undefined at runtime, no compile error
```

This is why assertions are inherently unsafe. The compiler trusts you unconditionally.

### The Assignability Rule

TypeScript does not allow completely unrelated assertions. The source and target types must overlap (one must be assignable to the other, or they must share a common supertype):

```typescript
const x = "hello" as number;
// Error: Conversion of type 'string' to type 'number' may be a mistake
// because neither type sufficiently overlaps with the other.
```

This guard catches obvious mistakes but can be bypassed with a **double assertion** through `unknown`:

```typescript
const x = "hello" as unknown as number;
// No error. This is a deliberate escape hatch.
```

## The Double Assertion Escape Hatch

Double assertion (`as unknown as T` or `as any as T`) bypasses all type checking. It exists for genuinely exceptional situations.

```typescript
// Via unknown (preferred over any)
const value = someString as unknown as ComplexType;

// Via any (also works, but any is more contagious)
const value2 = someString as any as ComplexType;
```

### When Double Assertion Is Acceptable

1. **Test mocks**: Creating partial mock objects that satisfy an interface for testing purposes.

```typescript
// In a test file
const mockService = {
  getUser: vi.fn().mockResolvedValue({ id: 1, name: "Test" }),
} as unknown as UserService;
```

2. **Library type bugs**: Working around incorrect type definitions in third-party libraries while waiting for a fix.

3. **Migration**: Temporarily bridging incompatible types during a large-scale refactor.

### When Double Assertion Is a Code Smell

If you reach for `as unknown as T` in application code (not tests, not library workarounds), that is almost always a sign that your type model is wrong. Fix the types instead.

## The Non-Null Assertion Operator (`!`)

The postfix `!` operator tells TypeScript that a value is not `null` or `undefined`, without any runtime check:

```typescript
function getLength(value: string | null): number {
  return value!.length; // "Trust me, it's not null"
}
```

This is equivalent to `(value as string).length` when the type is `string | null`. The `!` is removed entirely in the emitted JavaScript.

### Legitimate Uses

```typescript
// After a guard that TypeScript cannot track
const map = new Map<string, number>();
map.set("count", 42);

if (map.has("count")) {
  // TypeScript does not narrow Map.get after Map.has
  const count = map.get("count")!; // Safe here: we just checked
}
```

```typescript
// DOM elements that are guaranteed to exist (e.g., in index.html)
const root = document.getElementById("root")!;
```

### The Problem

Non-null assertions hide bugs silently. If the assumption is wrong, you get a runtime crash with no TypeScript warning:

```typescript
// Bad Practice: blindly asserting non-null
function processUser(users: Map<string, User>, id: string): void {
  const user = users.get(id)!; // Crashes if id is not in the map
  console.log(user.name);
}
```

### Safer Alternatives

```typescript
// Option 1: Explicit null check
const user = users.get(id);
if (!user) {
  throw new Error(`User ${id} not found`);
}
// user is narrowed to User

// Option 2: Optional chaining + nullish coalescing
const name = users.get(id)?.name ?? "Unknown";

// Option 3: Assertion function (runtime + type narrowing)
function assertDefined<T>(value: T | null | undefined, msg: string): asserts value is T {
  if (value == null) throw new Error(msg);
}

const user2 = users.get(id);
assertDefined(user2, `User ${id} not found`);
// user2 is narrowed to User with a meaningful error message
```

## `as const` Assertions

The `as const` assertion tells TypeScript to infer the **narrowest possible type** for a value. It makes all properties `readonly` and infers literal types instead of widened primitives.

```typescript
// Without as const
const config = {
  api: "https://api.example.com",
  timeout: 5000,
  retries: 3,
};
// Type: { api: string; timeout: number; retries: number }

// With as const
const config2 = {
  api: "https://api.example.com",
  timeout: 5000,
  retries: 3,
} as const;
// Type: {
//   readonly api: "https://api.example.com";
//   readonly timeout: 5000;
//   readonly retries: 3;
// }
```

### What `as const` Does

| Without `as const` | With `as const` |
|---------------------|-----------------|
| `"hello"` inferred as `string` | `"hello"` stays as `"hello"` |
| `42` inferred as `number` | `42` stays as `42` |
| `[1, 2, 3]` is `number[]` | `[1, 2, 3]` is `readonly [1, 2, 3]` |
| Object properties are mutable | Object properties are `readonly` |

### Common Patterns

**Deriving union types from arrays:**

```typescript
const ROLES = ["admin", "editor", "viewer"] as const;
type Role = (typeof ROLES)[number];
// "admin" | "editor" | "viewer"
```

**Deriving types from config objects:**

```typescript
const HTTP_METHODS = {
  GET: "GET",
  POST: "POST",
  PUT: "PUT",
  DELETE: "DELETE",
} as const;

type HttpMethod = (typeof HTTP_METHODS)[keyof typeof HTTP_METHODS];
// "GET" | "POST" | "PUT" | "DELETE"
```

### `as const` Is Not Runtime Immutability

`as const` is a compile-time only concept. It does not call `Object.freeze()`. At runtime, the value is still mutable:

```typescript
const arr = [1, 2, 3] as const;
// TypeScript: readonly [1, 2, 3]
// Runtime: a regular mutable array

// TypeScript error: Property 'push' does not exist on type 'readonly [1, 2, 3]'
arr.push(4);

// But if you bypass TypeScript (e.g., from untyped JS code), it works at runtime
(arr as any).push(4); // No runtime error
```

## The `satisfies` Operator

Introduced in TypeScript 4.9, `satisfies` validates that a value conforms to a type **without widening the inferred type**. This is its key differentiator from type annotations and `as` assertions.

### The Problem `satisfies` Solves

Consider a color palette:

```typescript
type Color = "red" | "green" | "blue";
type ColorValue = string | [number, number, number];

type Palette = Record<Color, ColorValue>;
```

**With a type annotation**, you lose literal types:

```typescript
const palette: Palette = {
  red: "#ff0000",
  green: [0, 255, 0],
  blue: "#0000ff",
};

palette.red.toUpperCase();
// Error: Property 'toUpperCase' does not exist on type 'string | [number, number, number]'
// TypeScript widened the type. It forgot that red is specifically a string.
```

**With `satisfies`**, you keep the inferred types:

```typescript
const palette = {
  red: "#ff0000",
  green: [0, 255, 0],
  blue: "#0000ff",
} satisfies Palette;

palette.red.toUpperCase();     // OK: red is inferred as string
palette.green[0];              // OK: green is inferred as [number, number, number]

// And you still get validation:
const bad = {
  red: "#ff0000",
  green: [0, 255, 0],
  blue: "#0000ff",
  purple: "#800080",         // Error: 'purple' is not in type 'Palette'
} satisfies Palette;
```

### How `satisfies` Differs from `: Type` and `as Type`

```typescript
interface Config {
  environment: "development" | "staging" | "production";
  port: number;
  debug: boolean;
}
```

**Type annotation (`: Config`)**:

```typescript
const config1: Config = {
  environment: "development",
  port: 3000,
  debug: true,
};

config1.environment; // Type: "development" | "staging" | "production" (widened)
config1.port;        // Type: number (widened)
```

**Type assertion (`as Config`)**:

```typescript
const config2 = {
  environment: "development",
  port: 3000,
  debug: true,
  extra: "oops",      // No error! Excess properties are not checked with `as`
} as Config;

config2.environment;   // Type: "development" | "staging" | "production"
```

**`satisfies Config`**:

```typescript
const config3 = {
  environment: "development",
  port: 3000,
  debug: true,
} satisfies Config;

config3.environment; // Type: "development" (literal preserved!)
config3.port;        // Type: 3000 (literal preserved!)
config3.debug;       // Type: true (literal preserved!)
```

### Comparison Table

| Behavior | `: Type` | `as Type` | `satisfies Type` |
|----------|----------|-----------|------------------|
| Validates structure | Yes | No | Yes |
| Catches excess properties | Yes | No | Yes |
| Catches missing properties | Yes | No | Yes |
| Preserves literal types | No | No | Yes |
| Changes inferred type | Yes (widens) | Yes (overrides) | No (preserves) |
| Runtime effect | None | None | None |

## `as const satisfies` Pattern

The combination of `as const` and `satisfies` gives you the best of both worlds: immutable literal types **and** type validation.

The order matters: `as const satisfies Type` is valid. `satisfies Type as const` is **not** valid.

```typescript
type Route = {
  path: string;
  method: "GET" | "POST" | "PUT" | "DELETE";
};

const routes = {
  getUsers: { path: "/users", method: "GET" },
  createUser: { path: "/users", method: "POST" },
  getUser: { path: "/users/:id", method: "GET" },
} as const satisfies Record<string, Route>;

// All values are readonly with literal types:
routes.getUsers.method;  // Type: "GET" (not "GET" | "POST" | "PUT" | "DELETE")
routes.getUsers.path;    // Type: "/users" (not string)

// But the structure is validated:
const badRoutes = {
  getUsers: { path: "/users", method: "PATCH" }, // Error: "PATCH" not assignable
} as const satisfies Record<string, Route>;
```

### Validated Constant Arrays

```typescript
type Permission = "read" | "write" | "admin";

const DEFAULT_PERMISSIONS = ["read", "write"] as const satisfies readonly Permission[];
// Type: readonly ["read", "write"]
// Validated: every element must be a valid Permission

// Derive the exact union type from the array
type DefaultPerm = (typeof DEFAULT_PERMISSIONS)[number]; // "read" | "write"
```

### Validated Configuration Maps

```typescript
interface FeatureFlag {
  enabled: boolean;
  description: string;
}

const FLAGS = {
  darkMode: { enabled: true, description: "Dark theme support" },
  newDashboard: { enabled: false, description: "Redesigned dashboard" },
  betaFeatures: { enabled: true, description: "Enable beta features" },
} as const satisfies Record<string, FeatureFlag>;

// Exact key union is preserved
type FlagName = keyof typeof FLAGS; // "darkMode" | "newDashboard" | "betaFeatures"

// Literal values are preserved
type DarkModeEnabled = (typeof FLAGS)["darkMode"]["enabled"]; // true (not boolean)
```

## Practical Decision Framework

### When to Use Type Annotations (`: Type`)

Use annotations when the **declared type is the contract** and you do not need to preserve literal specificity:

```typescript
// Function parameters and return types
function createUser(name: string, email: string): User {
  return { id: crypto.randomUUID(), name, email };
}

// Variables that will be reassigned
let currentUser: User = getDefaultUser();
currentUser = fetchedUser;

// When you want TypeScript to enforce the wider type
const handler: EventHandler = (event) => {
  // event is typed by the annotation
};
```

### When to Use `satisfies`

Use `satisfies` when you want **validation and inference** simultaneously:

```typescript
// Configuration objects where literal values matter
const config = {
  apiUrl: "https://api.example.com",
  timeout: 5000,
} satisfies AppConfig;

// Discriminated unions where the discriminant must stay literal
const action = { type: "INCREMENT", payload: 1 } satisfies Action;
// action.type is "INCREMENT", not string

// Record types where you want key autocomplete
const translations = {
  en: "Hello",
  fr: "Bonjour",
  de: "Hallo",
} satisfies Record<string, string>;
// keys are "en" | "fr" | "de", not string
```

### When to Use `as`

Use `as` only when **you know more than the compiler**, and there is no safer alternative:

```typescript
// DOM element narrowing (when you know the element type)
const canvas = document.getElementById("game") as HTMLCanvasElement;

// API response parsing (when you trust the source)
const data = await response.json() as ApiResponse;

// Working around library type bugs
const result = brokenLibFunction() as unknown as CorrectType;
```

### When to Use `as const`

Use `as const` when you need **readonly literal types**:

```typescript
// Deriving union types from values
const STATUSES = ["pending", "active", "archived"] as const;
type Status = (typeof STATUSES)[number];

// Ensuring tuple inference
const point = [10, 20] as const; // readonly [10, 20], not number[]

// Enum-like objects
const Direction = { Up: 0, Down: 1, Left: 2, Right: 3 } as const;
```

### When to Use `as const satisfies`

Use the combination when you need **readonly literals with validation**:

```typescript
// Validated constant tables
const HTTP_CODES = {
  OK: 200,
  CREATED: 201,
  BAD_REQUEST: 400,
  NOT_FOUND: 404,
  SERVER_ERROR: 500,
} as const satisfies Record<string, number>;

// Validated route definitions
const ROUTES = [
  { path: "/", component: "Home" },
  { path: "/about", component: "About" },
] as const satisfies readonly { path: string; component: string }[];
```

## Type Assertions in Control Flow

Type assertions do **not** affect control flow analysis. TypeScript's narrowing engine ignores them:

```typescript
function example(value: string | number) {
  const str = value as string;
  // str is typed as string, but...

  if (typeof value === "number") {
    // TypeScript still knows value might be a number here
    // The assertion did not narrow value itself
    console.log(value.toFixed(2));
  }
}
```

This is another reason to prefer type guards over assertions. Type guards actually narrow the type in subsequent code; assertions just slap a label on a single expression.

## Common Mistakes

### 1. Using `as` to Silence Errors Instead of Fixing Types

```typescript
// Bad Practice: hiding a real type error
interface Product {
  id: string;
  name: string;
  price: number;
}

const product = {
  id: 123,          // Wrong type: should be string
  name: "Widget",
  price: 9.99,
} as Product;       // No error! But id is wrong at runtime.

// Good: fix the data or the type
const product2: Product = {
  id: "123",        // Fixed
  name: "Widget",
  price: 9.99,
};
```

### 2. Using Non-Null Assertion Instead of Proper Null Handling

```typescript
// Bad Practice: spraying ! everywhere
function getUser(id: string): User {
  const user = userMap.get(id)!;       // Crash if not found
  const email = user.email!;           // Crash if email is null
  const domain = email.split("@")[1]!; // Crash if malformed
  return user;
}

// Good: handle each case
function getUser(id: string): User | undefined {
  const user = userMap.get(id);
  if (!user) return undefined;
  return user;
}
```

### 3. Using `as` When `satisfies` Would Work

```typescript
type Theme = "light" | "dark" | "system";

// Bad Practice: using as loses validation
const defaultTheme = "light" as Theme;
// If you typo "ligt" as Theme, no error

// Good: satisfies catches typos and preserves the literal
const defaultTheme2 = "light" satisfies Theme;
// Typo "ligt" satisfies Theme -> Error!
// And defaultTheme2 is still typed as "light", not Theme
```

### 4. Confusing `as const` with Runtime Immutability

```typescript
const config = { debug: true } as const;

// TypeScript prevents this:
// config.debug = false; // Error: Cannot assign to 'debug' because it is a read-only property

// But at runtime, nothing stops mutation from untyped code:
function untypedMiddleware(obj: any) {
  obj.debug = false; // No error, mutates at runtime
}
untypedMiddleware(config); // config.debug is now false at runtime

// If you need runtime immutability, use Object.freeze:
const frozen = Object.freeze({ debug: true });
```

### 5. Double Asserting in Application Code

```typescript
// Bad Practice: double assertion to force incompatible types
function processData(input: string): void {
  const data = input as unknown as DataModel; // Completely unchecked
  data.items.forEach(item => item.process());
  // Runtime crash: input is a string, not DataModel
}

// Good: parse and validate
function processData(input: string): void {
  const parsed = JSON.parse(input);
  if (!isDataModel(parsed)) {
    throw new Error("Invalid data format");
  }
  parsed.items.forEach(item => item.process());
}
```

### 6. Using `satisfies` Where a Type Annotation Is Simpler

```typescript
// Unnecessary: satisfies on a variable that will be reassigned
let count = 0 satisfies number; // Just use: let count: number = 0;

// Unnecessary: satisfies on a function parameter (not possible anyway)
// satisfies only works on expressions, not parameter declarations

// satisfies adds value when the inferred type is richer than the checked type
// If there is no difference, just use an annotation
```

## Best Practices

### 1. Default to Annotations, Reach for `satisfies` When Needed

Type annotations are the standard. Use `satisfies` when you specifically need literal type preservation or when the annotation would widen a type you want to keep narrow.

### 2. Ban `as any` in Production Code

Configure your linter to flag `as any`. Every `as any` is a hole in your type system. If you must escape the type system, use `as unknown` with a type guard:

```typescript
// Bad
const value = something as any;

// Better: explicit unknown with subsequent narrowing
const value = something as unknown;
if (typeof value === "string") {
  // Now safely narrowed
}
```

### 3. Prefer Assertion Functions Over Non-Null Assertions

When you need to assert that something is defined, an assertion function gives you both the type narrowing and a meaningful runtime error:

```typescript
function assertDefined<T>(
  value: T | null | undefined,
  name: string
): asserts value is T {
  if (value == null) {
    throw new Error(`Expected ${name} to be defined`);
  }
}

// Instead of: const el = document.getElementById("app")!;
const el = document.getElementById("app");
assertDefined(el, "app element");
// el is now HTMLElement with a descriptive error if missing
```

### 4. Use `as const satisfies` for Validated Constants

Any constant table, configuration object, or enum-like value that needs both validation and literal types should use this pattern:

```typescript
const ENDPOINTS = {
  users: "/api/v1/users",
  posts: "/api/v1/posts",
  comments: "/api/v1/comments",
} as const satisfies Record<string, `/${string}`>;
```

### 5. Document Every `as` Assertion with a Comment

If you cannot avoid an `as` assertion, explain **why** it is safe:

```typescript
// Safe: createElement("canvas") always returns HTMLCanvasElement
const canvas = document.createElement("canvas") as HTMLCanvasElement;

// Safe: JSON schema validation guarantees this shape (validated by ajv above)
const config = parsed as ServerConfig;
```

## Real-World Use Cases

### 1. API Response Handling with Runtime Validation

```typescript
interface ApiResponse<T> {
  data: T;
  status: number;
  message: string;
}

interface User {
  id: string;
  name: string;
  email: string;
}

// Type guard for runtime validation
function isUser(value: unknown): value is User {
  return (
    typeof value === "object" &&
    value !== null &&
    "id" in value &&
    "name" in value &&
    "email" in value &&
    typeof (value as User).id === "string" &&
    typeof (value as User).name === "string" &&
    typeof (value as User).email === "string"
  );
}

async function fetchUser(id: string): Promise<User> {
  const response = await fetch(`/api/users/${id}`);
  const json: unknown = await response.json();

  // Runtime validation, not just an assertion
  if (
    typeof json === "object" &&
    json !== null &&
    "data" in json &&
    isUser((json as { data: unknown }).data)
  ) {
    return (json as ApiResponse<User>).data;
  }

  throw new Error("Invalid API response");
}
```

### 2. Type-Safe Configuration with `as const satisfies`

```typescript
type LogLevel = "debug" | "info" | "warn" | "error";

interface AppConfig {
  name: string;
  version: string;
  logLevel: LogLevel;
  features: Record<string, boolean>;
  endpoints: Record<string, string>;
}

const CONFIG = {
  name: "MyApp",
  version: "2.1.0",
  logLevel: "info",
  features: {
    darkMode: true,
    notifications: true,
    analytics: false,
  },
  endpoints: {
    auth: "https://auth.example.com",
    api: "https://api.example.com",
    cdn: "https://cdn.example.com",
  },
} as const satisfies AppConfig;

// Type-safe key access with full literal types
type FeatureName = keyof typeof CONFIG.features;
// "darkMode" | "notifications" | "analytics"

type EndpointName = keyof typeof CONFIG.endpoints;
// "auth" | "api" | "cdn"

function isFeatureEnabled(feature: FeatureName): boolean {
  return CONFIG.features[feature];
}

function getEndpoint(name: EndpointName): string {
  return CONFIG.endpoints[name];
}

// Full autocomplete and type checking:
isFeatureEnabled("darkMode");       // OK
isFeatureEnabled("nonexistent");    // Error

getEndpoint("auth");                // OK
getEndpoint("unknown");             // Error
```

### 3. Discriminated Union Action Creators

```typescript
type Action =
  | { type: "ADD_TODO"; payload: { text: string } }
  | { type: "TOGGLE_TODO"; payload: { id: number } }
  | { type: "REMOVE_TODO"; payload: { id: number } }
  | { type: "SET_FILTER"; payload: { filter: "all" | "active" | "done" } };

// Using satisfies preserves the literal discriminant
function addTodo(text: string) {
  return { type: "ADD_TODO", payload: { text } } satisfies Action;
  // Return type: { type: "ADD_TODO"; payload: { text: string } }
  // NOT the full Action union
}

function toggleTodo(id: number) {
  return { type: "TOGGLE_TODO", payload: { id } } satisfies Action;
  // Return type: { type: "TOGGLE_TODO"; payload: { id: number } }
}

// With a type annotation, the return type would be the full Action union,
// losing the discriminant specificity:
function addTodoBad(text: string): Action {
  return { type: "ADD_TODO", payload: { text } };
  // Return type: Action (all four variants)
}

// The satisfies version enables precise type narrowing downstream:
const action = addTodo("Buy milk");
if (action.type === "ADD_TODO") {
  // Already narrowed without extra checks, because the type is
  // { type: "ADD_TODO"; payload: { text: string } }
  console.log(action.payload.text);
}
```

### 4. DOM Element Selection with Safe Assertions

```typescript
// A pattern that combines runtime checking with type assertions
function querySelector<T extends HTMLElement>(
  selector: string,
  expectedType: new () => T
): T {
  const element = document.querySelector(selector);

  if (!element) {
    throw new Error(`Element not found: ${selector}`);
  }

  if (!(element instanceof expectedType)) {
    throw new Error(
      `Element ${selector} is ${element.constructor.name}, expected ${expectedType.name}`
    );
  }

  return element; // Already narrowed to T by instanceof
}

// Usage: type-safe with runtime validation
const canvas = querySelector("canvas#game", HTMLCanvasElement);
// canvas is HTMLCanvasElement, guaranteed at runtime

const input = querySelector("input#email", HTMLInputElement);
// input is HTMLInputElement, guaranteed at runtime

// Compare with raw assertion:
// const canvas = document.querySelector("canvas#game") as HTMLCanvasElement;
// No runtime check. Crashes silently if element is missing or wrong type.
```

---

> **Key Takeaway:** Prefer type annotations for declaring contracts, use `satisfies` when you need validation without losing type precision, reserve `as` for the rare cases where you genuinely know more than the compiler, and always favor runtime checks over compile-time assertions when safety matters.
