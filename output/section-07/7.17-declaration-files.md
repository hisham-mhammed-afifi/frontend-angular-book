# 7.17 Declaration Files

> Declaration files (`.d.ts`) describe the shape of JavaScript code for the TypeScript compiler, enabling type checking and autocompletion for libraries and code that do not have built-in types.

---

## What Are Declaration Files?

A declaration file is a file with the `.d.ts` extension that contains **only type information**. It has no runtime code, produces no JavaScript output, and exists solely to tell the TypeScript compiler what types are available. Every time you use a library with full autocomplete and type checking, declaration files are what make that possible.

```typescript
// math-utils.d.ts
// Describes a JavaScript module without implementing it
export declare function add(a: number, b: number): number;
export declare function multiply(a: number, b: number): number;
export declare const PI: number;
```

When you import from the corresponding JavaScript module, TypeScript uses these declarations to type-check your code.

---

## Scripts vs Modules in TypeScript

TypeScript classifies every file as either a **script** or a **module**. This distinction is critical for understanding how declaration files work.

**Module:** Any file that contains at least one `import` or `export` statement. All declarations are scoped to that file.

**Script:** A file with no `import` or `export` statements. All declarations are treated as **global**, available everywhere in the project without importing.

```typescript
// This is a MODULE (has export)
export interface User {
  id: string;
  name: string;
}

// This is a SCRIPT (no import/export)
// Everything declared here is global
interface GlobalConfig {
  apiUrl: string;
  debug: boolean;
}
```

### The `moduleDetection` Setting

TypeScript's `moduleDetection` compiler option controls how this classification works:

| Value | Behavior |
|-------|----------|
| `"auto"` (default) | Files with `import`/`export` are modules; others are scripts |
| `"force"` | All `.ts` files are treated as modules (recommended for modern projects) |
| `"legacy"` | Same as `auto` but for older TypeScript compatibility |

**Important:** `.d.ts` files **always** use `"auto"` detection, even if your project sets `moduleDetection: "force"`. A `.d.ts` file without `import`/`export` is always a global script.

## The `declare` Keyword

The `declare` keyword tells TypeScript "this thing exists at runtime, but I am not implementing it here." It creates an **ambient declaration** that describes something without providing an implementation.

### Declaring Variables and Functions

```typescript
// globals.d.ts (script - no import/export, so these are global)

declare const API_BASE_URL: string;
declare let currentLocale: string;
declare function formatCurrency(amount: number, currency: string): string;
```

After this declaration, you can use `API_BASE_URL`, `currentLocale`, and `formatCurrency` anywhere in your project without importing them. TypeScript trusts that they exist at runtime.

### Declaring Classes

```typescript
declare class EventEmitter {
  on(event: string, listener: (...args: unknown[]) => void): this;
  emit(event: string, ...args: unknown[]): boolean;
  removeListener(event: string, listener: (...args: unknown[]) => void): this;
}
```

### Declaring Namespaces

Namespaces group related declarations. This is common for libraries that expose a single global object:

```typescript
// lodash.d.ts
declare namespace _ {
  function chunk<T>(array: T[], size: number): T[][];
  function compact<T>(array: (T | null | undefined | false | 0 | "")[]): T[];
  function uniq<T>(array: T[]): T[];
}
```

### What You Cannot Do

Declaration files cannot contain implementation code:

```typescript
// Bad: implementation in a .d.ts file
declare function add(a: number, b: number): number {
  return a + b; // Error: An implementation cannot be declared in ambient contexts
}
```

## Generating Declaration Files

The most reliable way to get correct declaration files is to let TypeScript generate them from your source code. Configure `tsconfig.json`:

```json
{
  "compilerOptions": {
    "declaration": true,
    "declarationDir": "./dist/types",
    "declarationMap": true,
    "emitDeclarationOnly": false
  }
}
```

| Option | Purpose |
|--------|---------|
| `declaration` | Generate `.d.ts` files alongside `.js` output |
| `declarationDir` | Output directory for declaration files |
| `declarationMap` | Generate `.d.ts.map` files for "Go to Definition" in editors |
| `emitDeclarationOnly` | Only emit `.d.ts` files, no `.js` (useful for type-only packages) |

Given this source file:

```typescript
// src/utils.ts
export function clamp(value: number, min: number, max: number): number {
  return Math.min(Math.max(value, min), max);
}

export interface Range {
  min: number;
  max: number;
}
```

TypeScript generates:

```typescript
// dist/types/utils.d.ts
export declare function clamp(value: number, min: number, max: number): number;
export interface Range {
  min: number;
  max: number;
}
```

## How TypeScript Finds Declaration Files

TypeScript resolves type declarations through several mechanisms, checked in order:

### 1. Bundled Types

If a package includes its own `.d.ts` files (indicated by the `types` or `typings` field in `package.json`), TypeScript uses those directly:

```json
{
  "name": "my-library",
  "main": "./dist/index.js",
  "types": "./dist/index.d.ts"
}
```

Modern packages also use the `exports` field for more granular control:

```json
{
  "name": "my-library",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    },
    "./utils": {
      "types": "./dist/utils.d.ts",
      "import": "./dist/utils.mjs"
    }
  }
}
```

**Important:** The `"types"` condition should always come **first** in each export entry. TypeScript resolves conditions top-to-bottom and stops at the first match.

### 2. DefinitelyTyped (`@types` packages)

For libraries written in JavaScript that do not bundle their own types, the community-maintained [DefinitelyTyped](https://github.com/DefinitelyTyped/DefinitelyTyped) repository provides type definitions published as `@types` packages on npm:

```bash
npm install --save-dev @types/lodash
npm install --save-dev @types/express
npm install --save-dev @types/node
```

TypeScript **automatically** picks up any `@types` package installed in `node_modules/@types/`. No configuration needed. The package name always mirrors the library name: `@types/lodash` provides types for `lodash`.

### 3. The `typeRoots` and `types` Compiler Options

By default, TypeScript looks for `@types` packages in all `node_modules/@types/` directories up the folder tree. You can override this:

```json
{
  "compilerOptions": {
    "typeRoots": ["./node_modules/@types", "./custom-types"]
  }
}
```

To restrict which `@types` packages are auto-included:

```json
{
  "compilerOptions": {
    "types": ["node", "jest"]
  }
}
```

This includes **only** `@types/node` and `@types/jest`. All other `@types` packages are excluded from automatic inclusion. Note that this does not prevent them from being used if explicitly imported.

### 4. Local Declaration Files

Any `.d.ts` file inside your project that is included by `tsconfig.json` (via `include` or `files`) is picked up automatically.

```
project/
  src/
    index.ts
    types/
      global.d.ts      <-- Included if src/ is in the include array
  custom.d.ts           <-- Included if root is in the include array
  tsconfig.json
```

## Writing Module Declaration Files

### Typing an Untyped npm Package

When you install a JavaScript package that has no types and no `@types` package, TypeScript will report an error. You can create your own declaration file:

```typescript
// src/types/untyped-lib.d.ts
declare module "untyped-lib" {
  export function doSomething(input: string): Promise<string>;
  export function transform(data: unknown[]): unknown[];

  export interface Options {
    verbose?: boolean;
    timeout?: number;
  }

  export default class Client {
    constructor(options?: Options);
    connect(): Promise<void>;
    disconnect(): void;
  }
}
```

Now you can import from `"untyped-lib"` with full type checking.

### Wildcard Module Declarations

For non-JavaScript imports (images, CSS files, JSON, etc.), use wildcard patterns:

```typescript
// src/types/assets.d.ts
declare module "*.png" {
  const src: string;
  export default src;
}

declare module "*.svg" {
  const src: string;
  export default src;
}

declare module "*.css" {
  const classes: Record<string, string>;
  export default classes;
}

declare module "*.module.css" {
  const classes: Record<string, string>;
  export default classes;
}
```

```typescript
// Now you can import assets with types
import logo from "./logo.png";       // string
import styles from "./app.module.css"; // Record<string, string>
```

### Quick Escape: `any` Module Declaration

When you need a quick fix and plan to add proper types later, you can declare a module as `any`:

```typescript
// src/types/temp.d.ts
declare module "problematic-library"; // Everything from this module is `any`
```

This silences all errors for that module but provides no type safety. Treat it as a temporary measure.

## Writing Global Declaration Files

### Adding Global Types

A `.d.ts` file with no `import`/`export` statements adds all its declarations to the global scope:

```typescript
// src/types/global.d.ts
interface AppConfig {
  apiUrl: string;
  version: string;
  features: Record<string, boolean>;
}

type Nullable<T> = T | null;

type AsyncResult<T> = {
  data: T | null;
  error: Error | null;
  loading: boolean;
};
```

These types are available everywhere without importing.

### Declaring Global Variables

If your application injects global variables (e.g., via a script tag or build tool):

```typescript
// src/types/globals.d.ts
declare const __APP_VERSION__: string;
declare const __BUILD_TIMESTAMP__: number;
declare const __DEV__: boolean;
```

```typescript
// Usage anywhere in the project
console.log(`Version: ${__APP_VERSION__}`);
if (__DEV__) {
  console.log("Development mode");
}
```

## Module Augmentation

Module augmentation lets you **add new types** to an existing module's declarations. This is powerful for extending third-party libraries.

### Augmenting a Module from a Module File

When your file has `import`/`export` statements (making it a module), `declare module` **augments** the existing module rather than replacing it:

```typescript
// src/types/express-augment.d.ts
import "express"; // This import makes this file a module

declare module "express" {
  interface Request {
    userId?: string;
    sessionId?: string;
    permissions?: string[];
  }
}
```

Now every Express `Request` object has the additional properties. The original Express types are preserved and extended.

### Augmenting from a Script File (Override)

If your file has **no** `import`/`export` statements (making it a script), `declare module` **overrides** the entire module declaration:

```typescript
// src/types/override.d.ts
// No import/export - this is a script

declare module "some-module" {
  // This REPLACES all existing types for "some-module"
  export function only(): void;
}
```

This is rarely what you want. To augment rather than override, ensure your file is a module by adding at least one `import` or `export`:

```typescript
// Force module mode with an empty export
export {};

declare module "some-module" {
  // Now this AUGMENTS instead of replacing
  export function additional(): void;
}
```

### Augmenting Global Scope from a Module

Use `declare global` to add declarations to the global scope from within a module file:

```typescript
// src/setup.ts (this is a module because it has exports)
export function initApp(): void {
  // ...
}

declare global {
  interface Window {
    analytics: {
      track(event: string, data?: Record<string, unknown>): void;
      identify(userId: string): void;
    };
  }

  // Add a global function
  function reportError(error: Error): void;
}
```

**Note:** Inside a `declare global` block, you do **not** use the `declare` keyword again. The block itself is already an ambient context.

```typescript
// Bad Practice
declare global {
  declare const MY_VAR: string; // Error or redundant
}

// Good
declare global {
  const MY_VAR: string;
}
```

## Augmenting Built-in Types

You can extend built-in JavaScript types by augmenting their interfaces.

### Extending `Array`

```typescript
// src/types/array-extensions.d.ts
export {};

declare global {
  interface Array<T> {
    first(): T | undefined;
    last(): T | undefined;
    unique(): T[];
  }
}
```

This tells TypeScript that every array has these methods. You must still implement them at runtime:

```typescript
// src/polyfills.ts
Array.prototype.first = function () {
  return this[0];
};

Array.prototype.last = function () {
  return this[this.length - 1];
};

Array.prototype.unique = function () {
  return [...new Set(this)];
};
```

### Extending `Window`

```typescript
// src/types/window.d.ts
export {};

declare global {
  interface Window {
    __INITIAL_STATE__: Record<string, unknown>;
    dataLayer: Array<Record<string, unknown>>;
  }
}
```

### Extending Node.js `process.env`

```typescript
// src/types/env.d.ts
export {};

declare global {
  namespace NodeJS {
    interface ProcessEnv {
      NODE_ENV: "development" | "production" | "test";
      DATABASE_URL: string;
      API_KEY: string;
      PORT?: string;
    }
  }
}
```

Now `process.env.DATABASE_URL` is typed as `string` (not `string | undefined`), and `process.env.NODE_ENV` only accepts the three specified values.

## Triple-Slash Directives

Triple-slash directives are special comments at the very top of a file that instruct the compiler to include additional files or type packages.

### `/// <reference path="..." />`

Declares a dependency on another file, including it in the compilation:

```typescript
/// <reference path="./legacy-globals.d.ts" />

// Now types from legacy-globals.d.ts are available
```

This is a legacy pattern. In modern TypeScript, use `import` statements or `tsconfig.json` `include` instead.

### `/// <reference types="..." />`

Declares a dependency on an `@types` package:

```typescript
/// <reference types="node" />
/// <reference types="jest" />

// Now types from @types/node and @types/jest are available
```

This is occasionally necessary in declaration files that need to reference types from other packages without importing them (which would change the file from a script to a module).

### `/// <reference lib="..." />`

Includes a specific built-in library:

```typescript
/// <reference lib="es2022" />
/// <reference lib="dom" />
```

### Rules for Triple-Slash Directives

- Must appear at the **very top** of the file
- Only single-line or multi-line comments can precede them
- Any directive that appears after a statement is treated as a regular comment

```typescript
// This works:
/// <reference types="node" />
import { readFile } from "fs";

// This does NOT work (directive after a statement):
import { readFile } from "fs";
/// <reference types="node" /> // Treated as a comment, ignored
```

## Publishing Type Declarations

When publishing a library, you have two options for providing types to consumers.

### Option 1: Bundle Types with the Package (Recommended)

Include generated `.d.ts` files in your package and point to them in `package.json`:

```json
{
  "name": "my-library",
  "version": "1.0.0",
  "main": "./dist/index.js",
  "module": "./dist/index.mjs",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.js"
    }
  },
  "files": ["dist"]
}
```

Benefits:
- Types are always in sync with the code (generated from the same source)
- No separate install step for consumers
- Version is always correct

### Option 2: Publish to DefinitelyTyped

If you do not control the library (or it is written in plain JavaScript), contribute types to DefinitelyTyped. Consumers install them with:

```bash
npm install --save-dev @types/your-library
```

Drawback: Types can drift out of sync with the library since they are maintained separately.

### The `typesVersions` Field

For libraries that need different types for different TypeScript versions:

```json
{
  "typesVersions": {
    ">=5.0": {
      "*": ["./dist/types-5.0/*"]
    },
    ">=4.5": {
      "*": ["./dist/types-4.5/*"]
    }
  }
}
```

TypeScript checks the version constraints top-to-bottom and uses the first match.

## The `skipLibCheck` Option

```json
{
  "compilerOptions": {
    "skipLibCheck": true
  }
}
```

When `true`, TypeScript skips type checking of **all** `.d.ts` files (both third-party and your own). This is common in production projects because:

1. It speeds up compilation significantly
2. It prevents build failures from conflicting or buggy third-party type definitions
3. It avoids issues where different `@types` packages have incompatible versions of shared types

**The tradeoff:** Errors in your own `.d.ts` files will not be caught. This is one reason to avoid storing application types in `.d.ts` files when regular `.ts` files would work.

## Common Mistakes

### 1. Storing Application Types in `.d.ts` Files

```typescript
// Bad Practice: putting app types in a .d.ts file
// src/types/user.d.ts
export interface User {
  id: string;
  name: string;
  email: string;
}

// Problem: skipLibCheck: true will skip this file entirely.
// If you make a mistake here, you will not get an error.
```

```typescript
// Good: use a regular .ts file for application types
// src/types/user.ts
export interface User {
  id: string;
  name: string;
  email: string;
}
```

Use `.d.ts` files only for their intended purpose: describing external JavaScript, declaring globals, and module augmentations.

### 2. Forgetting That `.d.ts` Files Are Not Checked Against JavaScript

Declaration files are the type system's "trust me" mechanism. TypeScript does not verify that a `.d.ts` file accurately describes the JavaScript it claims to type:

```typescript
// math.d.ts
export declare function add(a: number, b: number): number;
```

```javascript
// math.js - the actual implementation is wrong
export function add(a, b) {
  return String(a) + String(b); // Returns a string, not a number!
}
```

TypeScript will happily trust the declaration file. The mismatch only shows up at runtime. This is why generating declarations from TypeScript source is safer than writing them by hand.

### 3. Accidentally Overriding Instead of Augmenting

```typescript
// Bad Practice: missing export makes this a script, which OVERRIDES the module
// src/types/express.d.ts
declare module "express" {
  interface Request {
    userId?: string;
  }
}
// All original express types are gone!

// Good: add an import or export to make it a module (AUGMENTS)
import "express";

declare module "express" {
  interface Request {
    userId?: string;
  }
}
```

### 4. Using `declare` Inside `declare global`

```typescript
// Bad Practice
declare global {
  declare const CONFIG: { debug: boolean }; // Redundant/error
  declare function setup(): void;           // Redundant/error
}

// Good
declare global {
  const CONFIG: { debug: boolean };
  function setup(): void;
}
```

The `declare global` block is already an ambient context. Adding `declare` inside it is either redundant or an error.

### 5. Polluting Global Scope Unnecessarily

```typescript
// Bad Practice: dumping everything into global scope
// global.d.ts
type ID = string;
type Timestamp = number;
type Email = string;
type UserRole = "admin" | "editor" | "viewer";
interface User { /* ... */ }
interface Post { /* ... */ }
interface Comment { /* ... */ }
// These are now global, creating potential naming conflicts

// Good: keep types in modules, import explicitly
// src/types/user.ts
export type UserRole = "admin" | "editor" | "viewer";
export interface User { /* ... */ }
```

Global types should be reserved for things that are genuinely global: environment variables, global augmentations, and polyfill declarations.

### 6. Wrong Triple-Slash Directive Placement

```typescript
import { something } from "somewhere";

/// <reference types="node" />  // IGNORED: appears after a statement

export function doStuff(): void { /* ... */ }
```

Triple-slash directives must be at the very top of the file, before any statements.

## Best Practices

### 1. Generate Declaration Files from TypeScript Source

Do not write `.d.ts` files by hand for code you control. Let the compiler generate them:

```json
{
  "compilerOptions": {
    "declaration": true,
    "declarationMap": true
  }
}
```

This guarantees that types always match the implementation.

### 2. Use `declarationMap` for Better Editor Experience

Declaration maps link generated `.d.ts` files back to the original `.ts` source. When a consumer of your library does "Go to Definition," they see the actual source code instead of the stripped-down declaration file.

### 3. Keep Global Declarations Minimal

Reserve global `.d.ts` files for:
- Environment variables (`process.env`, `import.meta.env`)
- Build-time constants (`__DEV__`, `__APP_VERSION__`)
- Global augmentations (`Window`, `Array`)
- Wildcard module declarations (`*.png`, `*.css`)

Everything else should be in `.ts` files with explicit imports.

### 4. Version Your Augmentations

When augmenting third-party libraries, note the library version you are targeting:

```typescript
// src/types/express-augment.d.ts
// Augmentations for express@4.x
// Review when upgrading to express@5
import "express";

declare module "express" {
  interface Request {
    userId?: string;
  }
}
```

### 5. Use `exports` with `types` Condition in `package.json`

For library authors, always include the `types` condition first in each export entry:

```json
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  }
}
```

## Real-World Use Cases

### 1. Typing a Third-Party JavaScript Library

You are using a small JavaScript utility library (`string-magic`) that has no types and no `@types` package:

```typescript
// src/types/string-magic.d.ts
declare module "string-magic" {
  /**
   * Truncates a string to the specified length, adding ellipsis if truncated.
   */
  export function truncate(str: string, maxLength: number): string;

  /**
   * Converts a string to title case.
   */
  export function titleCase(str: string): string;

  /**
   * Slugifies a string (lowercase, hyphens instead of spaces).
   */
  export function slugify(str: string): string;

  export interface FormatOptions {
    locale?: string;
    uppercase?: boolean;
    trim?: boolean;
  }

  export function format(str: string, options?: FormatOptions): string;
}
```

```typescript
// src/app.ts
import { truncate, slugify } from "string-magic";

const title = truncate("A very long article title that needs shortening", 30);
const slug = slugify("My Blog Post Title"); // "my-blog-post-title"
```

### 2. Extending Express with Custom Properties

```typescript
// src/types/express.d.ts
import "express";

declare module "express" {
  interface Request {
    /** The authenticated user, set by auth middleware */
    user?: {
      id: string;
      email: string;
      roles: string[];
    };
    /** Request correlation ID for distributed tracing */
    correlationId: string;
    /** Start time for request duration measurement */
    startTime: number;
  }
}
```

```typescript
// src/middleware/auth.ts
import { Request, Response, NextFunction } from "express";

export function authMiddleware(req: Request, res: Response, next: NextFunction): void {
  const token = req.headers.authorization?.split(" ")[1];
  if (!token) {
    res.status(401).json({ error: "Unauthorized" });
    return;
  }

  // req.user is typed as the augmented interface
  req.user = verifyToken(token);
  next();
}

export function requireRole(role: string) {
  return (req: Request, res: Response, next: NextFunction): void => {
    if (!req.user?.roles.includes(role)) {
      res.status(403).json({ error: "Forbidden" });
      return;
    }
    next();
  };
}
```

### 3. Environment Type Safety

```typescript
// src/types/env.d.ts
export {};

declare global {
  namespace NodeJS {
    interface ProcessEnv {
      NODE_ENV: "development" | "production" | "test";
      PORT: string;
      DATABASE_URL: string;
      REDIS_URL: string;
      JWT_SECRET: string;
      LOG_LEVEL?: "debug" | "info" | "warn" | "error";
      SENTRY_DSN?: string;
    }
  }
}
```

```typescript
// src/config.ts
// All env vars are typed. Required ones are `string`, optional ones are `string | undefined`.
const config = {
  port: parseInt(process.env.PORT, 10),        // PORT is string (required)
  dbUrl: process.env.DATABASE_URL,             // string (required)
  logLevel: process.env.LOG_LEVEL ?? "info",   // string | undefined -> defaulted
  sentryDsn: process.env.SENTRY_DSN,           // string | undefined (optional)
  isDev: process.env.NODE_ENV === "development",
};

export default config;
```

**Caveat:** This typing is aspirational. TypeScript believes these variables exist, but at runtime they might still be `undefined` if your deployment is misconfigured. For truly safe env handling, combine declaration files with a runtime validation layer:

```typescript
// src/env.ts
function requireEnv(name: string): string {
  const value = process.env[name];
  if (!value) {
    throw new Error(`Missing required environment variable: ${name}`);
  }
  return value;
}

export const env = {
  port: parseInt(requireEnv("PORT"), 10),
  dbUrl: requireEnv("DATABASE_URL"),
  jwtSecret: requireEnv("JWT_SECRET"),
  logLevel: (process.env.LOG_LEVEL ?? "info") as "debug" | "info" | "warn" | "error",
};
```

---

> **Key Takeaway:** Use `.d.ts` files to describe external JavaScript and declare globals; use regular `.ts` files for your own application types; generate declarations from source for libraries you publish; and always remember that declaration files are trusted by the compiler but never verified against runtime code.
