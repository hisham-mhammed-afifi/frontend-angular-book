# 7.2 Basic Types

> TypeScript's type system starts with the same primitives as JavaScript, then adds special types like `any`, `unknown`, `never`, and `void` that let you express constraints JavaScript cannot.

---

## Primitive Types

TypeScript has a type for every JavaScript primitive. Always use the lowercase versions.

### string

```typescript
let firstName: string = "Alice";
let greeting: string = `Hello, ${firstName}`;
let empty: string = "";

// TypeScript infers the type - no annotation needed
let lastName = "Smith"; // inferred as string
```

### number

All numbers are floating-point. There is no separate `int` or `float` type.

```typescript
let age: number = 30;
let price: number = 19.99;
let hex: number = 0xff;
let binary: number = 0b1010;
let octal: number = 0o744;
let negative: number = -42;
let notANumber: number = NaN;       // NaN is a valid number
let infinity: number = Infinity;     // Infinity is a valid number
```

### boolean

```typescript
let isActive: boolean = true;
let hasPermission: boolean = false;

// Inferred
let isDone = true; // boolean
```

### bigint

For integers larger than `Number.MAX_SAFE_INTEGER` (2^53 - 1). Requires `target: "ES2020"` or higher in tsconfig.

```typescript
let huge: bigint = 9007199254740991n;
let alsoHuge: bigint = BigInt(9007199254740991);

// bigint and number are NOT interchangeable
let sum = huge + 1n;     // OK: bigint + bigint
// let bad = huge + 1;   // Error: cannot mix bigint and number
```

### symbol

Unique identifiers. Every `Symbol()` call creates a value that is not equal to any other.

```typescript
let id: symbol = Symbol("id");
let anotherId: symbol = Symbol("id");

console.log(id === anotherId); // false (always unique)

// unique symbol: a subtype of symbol tied to a specific declaration
const ID: unique symbol = Symbol("ID");
// let ID2: unique symbol = Symbol("ID"); // only const allowed with unique symbol
```

### null and undefined

```typescript
let nothing: null = null;
let notDefined: undefined = undefined;

// With strictNullChecks (default in strict mode):
let name: string = "Alice";
// name = null;       // Error: Type 'null' is not assignable to type 'string'
// name = undefined;  // Error: Type 'undefined' is not assignable to type 'string'

// Explicitly allow null:
let nickname: string | null = "Al";
nickname = null; // OK
```

With `strictNullChecks` enabled (the default in `strict: true`), `null` and `undefined` are only assignable to their own types and to `void`/`any`/`unknown`. This is one of the most important safety features in TypeScript.

## Lowercase vs Uppercase: string vs String

Always use lowercase primitive types. The uppercase versions (`String`, `Number`, `Boolean`, `Symbol`, `BigInt`) refer to the object wrapper types, not the primitives.

```typescript
// BAD: Object wrapper types
let a: String = "hello";    // String object, not string primitive
let b: Number = 42;
let c: Boolean = true;

// GOOD: Primitive types
let x: string = "hello";
let y: number = 42;
let z: boolean = true;

// The wrapper types are almost never what you want:
let s: String = new String("hello");
typeof s; // "object", not "string"
```

This is a common beginner mistake. TypeScript will let you use `String` but it causes subtle issues with type compatibility.

## Special Types

### any

`any` disables all type checking. The variable can hold anything, and you can do anything with it without errors.

```typescript
let data: any = 42;
data = "hello";        // OK
data = true;           // OK
data.nonExistent();    // OK at compile time, crashes at runtime
data.foo.bar.baz;      // OK at compile time, crashes at runtime
```

`any` is contagious. If a function returns `any`, everything that uses the result also loses type safety.

```typescript
function parseData(raw: any) {
  return raw.data; // return type is any
}

const result = parseData({ data: 42 });
result.toUpperCase(); // No error! But crashes because 42 is not a string
```

**When `any` is acceptable:**
- Migrating JavaScript to TypeScript incrementally.
- Working with truly dynamic third-party data where typing is impossible.
- Prototyping code you will type properly later.

**In all other cases, use `unknown` instead.**

### unknown

`unknown` is the type-safe version of `any`. You can assign anything to it, but you cannot use it until you narrow the type.

```typescript
let data: unknown = 42;
data = "hello";        // OK (assignment is fine)

// data.toUpperCase(); // Error: Object is of type 'unknown'
// data + 1;           // Error: Object is of type 'unknown'

// Must narrow the type first
if (typeof data === "string") {
  console.log(data.toUpperCase()); // OK: TypeScript knows data is string here
}

if (typeof data === "number") {
  console.log(data + 1); // OK: TypeScript knows data is number here
}
```

`unknown` forces you to validate before using the value. This is the correct type for external data (API responses, user input, JSON parsing).

```typescript
// Parsing external JSON
function parseJSON(raw: string): unknown {
  return JSON.parse(raw);
}

const data = parseJSON('{"name": "Alice"}');
// data is unknown - must validate before using

if (
  typeof data === "object" &&
  data !== null &&
  "name" in data &&
  typeof (data as Record<string, unknown>).name === "string"
) {
  console.log((data as { name: string }).name); // "Alice"
}
```

### void

`void` means a function does not return a meaningful value. It may return `undefined` implicitly, but the caller should not use the return value.

```typescript
function logMessage(msg: string): void {
  console.log(msg);
  // No return statement (or `return;` or `return undefined;`)
}

const result = logMessage("hello"); // result is void
// result.toString(); // Error: cannot use void
```

**`void` vs `undefined` as return type:**

```typescript
// void: caller should ignore the return value
function log(msg: string): void {
  console.log(msg);
}

// undefined: function explicitly returns undefined
function getNothing(): undefined {
  return undefined;
  // return; // also OK in TS 5.1+
}
```

A function typed as returning `void` can technically return any value when used in a callback context. This is by design to allow callbacks that ignore return values:

```typescript
const nums = [1, 2, 3];

// Array.forEach expects a callback returning void
// But push returns a number - this is OK because void says "ignore the return"
nums.forEach((n) => nums.push(n)); // No error
```

### never

`never` represents a value that never occurs. It is the bottom type (the empty set).

**Functions that never return:**

```typescript
// Throws an error - execution never completes normally
function throwError(message: string): never {
  throw new Error(message);
}

// Infinite loop - execution never completes
function infiniteLoop(): never {
  while (true) {
    // ...
  }
}
```

**Exhaustiveness checking:**

The most practical use of `never` is ensuring you handle all cases in a union type.

```typescript
type Shape = "circle" | "square" | "triangle";

function getArea(shape: Shape): number {
  switch (shape) {
    case "circle":
      return Math.PI * 10 * 10;
    case "square":
      return 10 * 10;
    case "triangle":
      return (10 * 10) / 2;
    default:
      // If all cases are handled, shape is `never` here
      const _exhaustive: never = shape;
      return _exhaustive;
  }
}

// If someone adds "pentagon" to Shape but forgets to add a case:
// Error: Type '"pentagon"' is not assignable to type 'never'
```

**`never` in type operations:**

```typescript
// never is the identity for unions (like 0 for addition)
type A = string | never; // string (never disappears)

// never absorbs in intersections (like 0 for multiplication)
type B = string & never; // never

// Filtering types with never
type NonNullable<T> = T extends null | undefined ? never : T;
type Result = NonNullable<string | null | undefined>; // string
```

## The Type Hierarchy

Understanding how types relate to each other prevents confusion.

```
                    unknown          (top type: everything is assignable to unknown)
                   /   |   \
            string  number  boolean  object  ...  (all concrete types)
                   \   |   /
                    never            (bottom type: never is assignable to everything)
```

| Rule | Example |
|------|---------|
| Everything is assignable to `unknown` | `let x: unknown = 42` OK |
| Nothing is assignable from `unknown` without narrowing | `let y: number = x` Error |
| `never` is assignable to everything | `let z: string = throwError("!")` OK |
| Nothing is assignable to `never` | `let n: never = 42` Error |
| `any` breaks all rules (bypasses the type system) | `let a: any = 42; let b: string = a;` OK (unsafe) |

## object, Object, and {}

Three object-related types that look similar but behave very differently.

### object (lowercase)

Represents any non-primitive value. Use this when you need "any object but not a primitive."

```typescript
function logObject(obj: object): void {
  console.log(obj);
}

logObject({ name: "Alice" });  // OK
logObject([1, 2, 3]);          // OK (arrays are objects)
logObject(new Date());         // OK
// logObject("hello");         // Error: string is not an object
// logObject(42);              // Error: number is not an object
// logObject(null);            // Error: null is not an object
```

### Object (uppercase)

The `Object` interface from `lib.es5.d.ts`. Describes methods available on all JavaScript objects (`toString`, `valueOf`, etc.). Includes primitives because they auto-box.

```typescript
let a: Object = "hello"; // OK! (string auto-boxes to String object)
let b: Object = 42;      // OK!
let c: Object = true;    // OK!
// let d: Object = null;  // Error with strictNullChecks
```

**Do not use `Object`.** It is too broad and misleading.

### {} (empty object literal)

Represents any value that is not `null` or `undefined`. This is almost the same as `Object`.

```typescript
let a: {} = "hello";  // OK
let b: {} = 42;       // OK
let c: {} = true;     // OK
let d: {} = { x: 1 }; // OK
// let e: {} = null;   // Error with strictNullChecks
```

**Do not use `{}` as a type.** It does not mean "empty object." It means "anything that is not null/undefined."

### What to Use

| Need | Type |
|------|------|
| Any non-primitive value | `object` |
| Object with known keys | `{ name: string; age: number }` |
| Object with string keys, same value type | `Record<string, ValueType>` |
| Any value that is not null/undefined | `NonNullable<unknown>` or just use proper types |
| Avoid | `Object` and `{}` |

## Type Inference

TypeScript infers types when you do not write them explicitly. It is often better to let inference work than to annotate everything.

### When Inference Works Well

```typescript
// Variable initialization
let name = "Alice";          // inferred: string
let age = 30;                // inferred: number
let scores = [90, 85, 92];  // inferred: number[]

// Return types
function add(a: number, b: number) {
  return a + b; // return type inferred: number
}

// Array methods
const doubled = [1, 2, 3].map((n) => n * 2); // number[]
const names = ["Alice", "Bob"].filter((n) => n.startsWith("A")); // string[]

// Destructuring
const { x, y } = { x: 10, y: 20 }; // x: number, y: number
```

### When to Add Explicit Types

```typescript
// 1. Function parameters (ALWAYS annotate)
function greet(name: string): string {
  return `Hello, ${name}`;
}

// 2. Empty collections
const items: string[] = [];          // without annotation: any[] or never[]
const cache: Map<string, number> = new Map();

// 3. Public API / library boundaries
export function fetchUser(id: number): Promise<User> {
  // ...
}

// 4. When inference produces a wider type than you want
let status = "active";                    // inferred: string (too wide)
let status: "active" | "inactive" = "active"; // narrower, correct

// 5. When the inferred type is complex and unclear
const result: ParsedConfig = parseConfig(rawInput);
```

### const vs let Inference

`const` produces narrower types because the value cannot be reassigned.

```typescript
let x = "hello";     // type: string (could be reassigned to any string)
const y = "hello";   // type: "hello" (literal type, cannot change)

let n = 42;          // type: number
const m = 42;        // type: 42

let flag = true;     // type: boolean
const on = true;     // type: true
```

### as const (const assertion)

Forces the widest possible literal inference on an entire value.

```typescript
// Without as const
const colors = ["red", "green", "blue"];
// type: string[]

// With as const
const colors = ["red", "green", "blue"] as const;
// type: readonly ["red", "green", "blue"]
// Each element is a literal type, and the array is readonly

const config = {
  env: "production",
  port: 3000,
  debug: false,
} as const;
// type: { readonly env: "production"; readonly port: 3000; readonly debug: false }
// All properties are readonly with literal types
```

## Type Assertions

Type assertions tell TypeScript "I know more about this type than you do." They do NOT perform runtime conversion.

### as syntax (preferred)

```typescript
const input = document.getElementById("name") as HTMLInputElement;
input.value = "Alice"; // OK: TypeScript trusts it is HTMLInputElement

const data = JSON.parse('{"name":"Alice"}') as { name: string };
console.log(data.name); // OK
```

### Angle bracket syntax (not usable in JSX/TSX files)

```typescript
const input = <HTMLInputElement>document.getElementById("name");
```

### Double assertion (escape hatch)

When TypeScript refuses a direct assertion because the types are too different:

```typescript
// Error: Cannot assert string to number directly
// const x = "hello" as number;

// Double assertion through unknown (use with extreme caution)
const x = "hello" as unknown as number; // compiles, but meaningless
```

**Type assertions are dangerous.** They override TypeScript's analysis. If the assertion is wrong, you get runtime crashes with no compile-time warning.

```typescript
// BAD: lying to the compiler
const data = JSON.parse("42") as { name: string };
console.log(data.name); // undefined at runtime, no compile error

// GOOD: validate at runtime
const raw: unknown = JSON.parse("42");
if (typeof raw === "object" && raw !== null && "name" in raw) {
  const data = raw as { name: string };
  console.log(data.name);
}
```

### Non-null assertion (!)

Tells TypeScript a value is not `null` or `undefined`. Use sparingly.

```typescript
const el = document.getElementById("app");
// el is HTMLElement | null

el!.textContent = "Hello"; // Asserts el is not null
// If el IS null, this crashes at runtime

// Prefer null checking instead:
if (el) {
  el.textContent = "Hello"; // TypeScript knows el is HTMLElement here
}

// Or optional chaining:
el?.textContent; // safe, returns undefined if el is null
```

## Common Mistakes

- **Using `any` instead of `unknown`.** `any` silences all errors, including real bugs. Use `unknown` and narrow with type guards. The only time `any` is justified is during migration from untyped JavaScript.

- **Using uppercase `String`, `Number`, `Boolean` as types.** These are the object wrapper types, not the primitives. Always use lowercase `string`, `number`, `boolean`. The uppercase versions cause subtle compatibility issues.

```typescript
// BAD
function greet(name: String): void { }

// GOOD
function greet(name: string): void { }
```

- **Over-annotating when inference is sufficient.** Adding type annotations to every variable adds noise without adding safety.

```typescript
// BAD: redundant annotations
const name: string = "Alice";
const age: number = 30;
const isActive: boolean = true;
const items: number[] = [1, 2, 3];

// GOOD: let inference work
const name = "Alice";
const age = 30;
const isActive = true;
const items = [1, 2, 3];
```

- **Not handling `null` and `undefined` with `strictNullChecks`.** Without checking, accessing properties on a potentially null value crashes at runtime. Always narrow or use optional chaining.

- **Confusing type assertions with type casting.** `as SomeType` does not convert the value. It only tells the compiler to treat it as that type. `"42" as number` does not convert the string to a number. It is a compile-time lie.

- **Using `{}` to mean "empty object".** The type `{}` matches any non-nullish value, including strings, numbers, and booleans. To describe an object with no known properties, use `Record<string, never>` or `object`.

---

## Best Practices

- **Enable `strict: true` and never disable `strictNullChecks`.** Null/undefined bugs are the most common runtime errors in JavaScript. TypeScript catches them at compile time, but only with `strictNullChecks` enabled.

- **Use `unknown` for external data.** API responses, JSON parsing, third-party callbacks. Narrow to the expected type with type guards before using.

- **Let TypeScript infer when it can.** Annotate function parameters and public APIs. Let inference handle local variables, return types of simple functions, and array method chains.

- **Use `as const` for configuration objects and constant arrays.** This gives you literal types and `readonly` protection, making your constants truly constant at the type level.

- **Use `never` for exhaustiveness checks.** In switch statements and conditional types, assign the default/else branch to `never` to catch unhandled cases at compile time.

- **Prefer type narrowing over type assertions.** Narrowing (typeof, instanceof, in, custom type guards) is verified at runtime. Assertions are compile-time promises that TypeScript cannot verify.

## Real-World Use Cases

### 1. Safe API Response Handling

```typescript
interface User {
  id: number;
  name: string;
  email: string;
}

function isUser(data: unknown): data is User {
  return (
    typeof data === "object" &&
    data !== null &&
    "id" in data &&
    typeof (data as Record<string, unknown>).id === "number" &&
    "name" in data &&
    typeof (data as Record<string, unknown>).name === "string" &&
    "email" in data &&
    typeof (data as Record<string, unknown>).email === "string"
  );
}

async function fetchUser(id: number): Promise<User> {
  const response = await fetch(`/api/users/${id}`);
  const data: unknown = await response.json();

  if (!isUser(data)) {
    throw new Error("Invalid user data from API");
  }

  return data; // TypeScript knows this is User
}
```

### 2. Configuration with Literal Types and const Assertion

```typescript
const LOG_LEVELS = ["debug", "info", "warn", "error"] as const;
type LogLevel = (typeof LOG_LEVELS)[number]; // "debug" | "info" | "warn" | "error"

const config = {
  level: "info" as LogLevel,
  output: "console",
  maxSize: 1024,
} as const;

function log(level: LogLevel, message: string): void {
  const levelIndex = LOG_LEVELS.indexOf(level);
  const configIndex = LOG_LEVELS.indexOf(config.level);
  if (levelIndex >= configIndex) {
    console.log(`[${level.toUpperCase()}] ${message}`);
  }
}

log("debug", "Starting up");  // suppressed (below info)
log("warn", "Disk space low"); // printed
// log("verbose", "...");      // Error: "verbose" is not assignable to LogLevel
```

### 3. Exhaustive Event Handler

```typescript
type AppEvent =
  | { type: "LOGIN"; userId: number }
  | { type: "LOGOUT" }
  | { type: "PURCHASE"; itemId: number; amount: number };

function handleEvent(event: AppEvent): string {
  switch (event.type) {
    case "LOGIN":
      return `User ${event.userId} logged in`;
    case "LOGOUT":
      return "User logged out";
    case "PURCHASE":
      return `Purchased item ${event.itemId} for $${event.amount}`;
    default:
      // Compile-time guarantee: all cases handled
      const _exhaustive: never = event;
      return _exhaustive;
  }
}

// If a new event type is added to AppEvent (e.g., "REFUND"),
// this function will fail to compile until the new case is handled.
```

---

> **Key Takeaway:** Use lowercase primitive types (`string`, `number`, `boolean`), prefer `unknown` over `any` for untyped data, leverage type inference for local variables, and use `never` for exhaustiveness checks. TypeScript's type system is most effective when you let inference handle the obvious cases and add explicit types where they communicate intent or catch real bugs.
