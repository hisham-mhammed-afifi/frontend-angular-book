# 7.5 Type Annotations

> Type annotations are the explicit labels you attach to variables, parameters, and return values to tell TypeScript what type something should be. Knowing when to annotate and when to let inference do the work is what separates noisy TypeScript from clean TypeScript.

---

## What Are Type Annotations?

A type annotation is syntax that explicitly declares the type of a binding. TypeScript uses a colon (`:`) followed by a type after a variable name, parameter name, or function signature.

```typescript
// Variable annotation
let username: string = "alice";

// Parameter annotation
function greet(name: string) {
  console.log(`Hello, ${name}`);
}

// Return type annotation
function getAge(): number {
  return 30;
}
```

Without annotations, TypeScript infers types from initialization values and usage context. Annotations override or constrain what inference would otherwise determine.

---

## Variable Annotations

### Basic Variable Annotations

```typescript
let count: number = 0;
let name: string = "Bob";
let isActive: boolean = true;
let data: null = null;
let id: symbol = Symbol("id");
let big: bigint = 100n;
```

### When Variable Annotations Are Redundant

TypeScript infers the type from the initializer. These annotations add nothing:

```typescript
// Redundant - TypeScript already infers these types
let count: number = 0;        // inferred as number
let name: string = "Bob";     // inferred as string
let isActive: boolean = true;  // inferred as boolean

// Clean - let inference do the work
let count = 0;
let name = "Bob";
let isActive = true;
```

### When Variable Annotations Are Necessary

**1. Delayed initialization:**

```typescript
// TypeScript cannot infer the type without an initializer
let result: string;

if (someCondition) {
  result = "success";
} else {
  result = "failure";
}
```

Without the annotation, `result` would be `any` (in non-strict mode) or an error (in strict mode with `noImplicitAny`).

**2. Wider type than the initializer:**

```typescript
// Inferred as 0 (literal type with const) or number (with let)
// But you want to allow string too
let id: string | number = 0;
id = "abc-123"; // valid because of the annotation

// Without annotation, this would fail:
let id2 = 0;
// id2 = "abc-123"; // Error: Type 'string' is not assignable to type 'number'
```

**3. Empty collections:**

```typescript
// Inferred as never[] - you cannot push anything into it
const items = [];
// items.push("hello"); // Error in strict mode: Argument of type 'string' is not assignable to parameter of type 'never'

// Annotate to specify what the array will hold
const items: string[] = [];
items.push("hello"); // works
```

**4. Functions that return `any`:**

```typescript
// JSON.parse returns any
const parsed = JSON.parse('{"name": "Alice"}');
// parsed is any - no type safety

// Annotate to get type checking
const parsed: { name: string } = JSON.parse('{"name": "Alice"}');
// Now parsed.name is string, and parsed.age would be an error
```

## Function Parameter Annotations

Function parameters are never inferred from their usage inside the function body. You must annotate them (unless contextual typing applies, covered later).

```typescript
// Parameters MUST be annotated
function add(a: number, b: number) {
  return a + b;
}

// Without annotations in strict mode:
// function add(a, b) { } // Error: Parameter 'a' implicitly has an 'any' type
```

### Optional Parameters

Use `?` to mark a parameter as optional. Optional parameters receive `undefined` when not provided.

```typescript
function greet(name: string, greeting?: string) {
  // greeting is string | undefined inside the function
  const g = greeting ?? "Hello";
  return `${g}, ${name}!`;
}

greet("Alice");          // "Hello, Alice!"
greet("Alice", "Hey");   // "Hey, Alice!"
```

### Default Parameters

Default values provide both a fallback and a type. You do not need an annotation when the default value makes the type obvious.

```typescript
// No annotation needed - TypeScript infers greeting as string
function greet(name: string, greeting = "Hello") {
  return `${greeting}, ${name}!`;
}

// The parameter is still optional at the call site
greet("Alice"); // works
```

However, if the default value does not represent the full range of acceptable types:

```typescript
// You want to accept string | null, but default is string
function greet(name: string, greeting: string | null = "Hello") {
  if (greeting === null) return name;
  return `${greeting}, ${name}!`;
}
```

### Rest Parameters

```typescript
function sum(...numbers: number[]): number {
  return numbers.reduce((acc, n) => acc + n, 0);
}

// With tuple types for fixed-then-rest patterns
function log(level: string, ...messages: string[]) {
  console.log(`[${level}]`, ...messages);
}
```

## Return Type Annotations

### Inferred Return Types

TypeScript infers the return type from the `return` statements. For simple functions, this is usually sufficient.

```typescript
// Inferred return type: number
function double(n: number) {
  return n * 2;
}

// Inferred return type: string
function greet(name: string) {
  return `Hello, ${name}`;
}
```

### When to Annotate Return Types

**1. Public API boundaries (exported functions):**

```typescript
// Good practice: annotate exports
export function parseConfig(raw: string): Config {
  return JSON.parse(raw);
}

// If parseConfig accidentally returns the wrong shape,
// the error appears at the function definition, not at every call site
```

This is the single most important rule for return type annotations. When a function is exported, an explicit return type:
- Catches errors at the function definition instead of at distant call sites
- Serves as documentation for consumers
- Prevents accidental API changes when refactoring the function body

**2. Recursive functions:**

```typescript
// TypeScript cannot always infer recursive return types
function fibonacci(n: number): number {
  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}
```

**3. Functions with multiple return paths that should unify:**

```typescript
// Without annotation, inferred as string | number
function process(input: unknown) {
  if (typeof input === "string") return input.toUpperCase();
  if (typeof input === "number") return input * 2;
  return null;
}

// With annotation, you enforce the contract
function process(input: unknown): string | null {
  if (typeof input === "string") return input.toUpperCase();
  if (typeof input === "number") return String(input * 2);
  return null;
}
```

**4. Enforcing `void` or specific promise types:**

```typescript
// Enforce that the function does not return anything meaningful
function logMessage(msg: string): void {
  console.log(msg);
  // return msg; // Error: Type 'string' is not assignable to type 'void'
}

// Enforce the resolved type of an async function
async function fetchUser(id: string): Promise<User> {
  const response = await fetch(`/api/users/${id}`);
  return response.json();
}
```

## Object Type Annotations

### Inline Object Types

```typescript
function printCoord(pt: { x: number; y: number }) {
  console.log(`(${pt.x}, ${pt.y})`);
}

printCoord({ x: 3, y: 7 });
```

Properties are separated by either `;` or `,`. The trailing separator is optional.

### Optional Properties

```typescript
function printName(name: { first: string; last?: string }) {
  console.log(name.first);
  // name.last is string | undefined
  if (name.last !== undefined) {
    console.log(name.last);
  }
}

printName({ first: "Alice" });
printName({ first: "Alice", last: "Smith" });
```

### Readonly Properties

```typescript
function freeze(point: { readonly x: number; readonly y: number }) {
  // point.x = 10; // Error: Cannot assign to 'x' because it is a read-only property
  console.log(point.x, point.y);
}
```

### Index Signatures

When you do not know all property names ahead of time but know the shape of the values:

```typescript
function processHeaders(headers: { [key: string]: string }) {
  for (const [name, value] of Object.entries(headers)) {
    console.log(`${name}: ${value}`);
  }
}

// Equivalent with Record utility type
function processHeaders(headers: Record<string, string>) {
  // ...
}
```

## Destructuring with Type Annotations

Destructuring annotations are a common source of confusion. You cannot annotate individual destructured bindings inline. Instead, you annotate the entire pattern.

### Object Destructuring

```typescript
// WRONG - this is JavaScript label syntax, not type annotations
// const { name: string, age: number } = person;
// This renames `name` to `string` and `age` to `number`

// CORRECT - annotate the whole destructured object
const { name, age }: { name: string; age: number } = person;
```

### Function Parameter Destructuring

```typescript
// Annotate after the entire destructuring pattern
function greet({ name, age }: { name: string; age: number }) {
  console.log(`${name} is ${age}`);
}

// With a type alias (much cleaner for complex objects)
type Person = {
  name: string;
  age: number;
};

function greet({ name, age }: Person) {
  console.log(`${name} is ${age}`);
}
```

### Array Destructuring

```typescript
// Annotate with a tuple type
const [first, second]: [string, number] = ["Alice", 30];

// In function parameters
function processEntry([key, value]: [string, number]) {
  console.log(`${key}: ${value}`);
}
```

### Destructuring with Renaming and Types

```typescript
// Renaming AND typing
const { name: userName, age: userAge }: { name: string; age: number } = person;
// userName is string, userAge is number

// In function parameters
function display({ name: displayName }: { name: string }) {
  console.log(displayName);
}
```

### Destructuring with Defaults and Types

```typescript
// Defaults with type annotation
function createUser({ name, role = "user" }: { name: string; role?: string }) {
  console.log(`${name} (${role})`);
}

createUser({ name: "Alice" });           // "Alice (user)"
createUser({ name: "Bob", role: "admin" }); // "Bob (admin)"
```

## Contextual Typing

Contextual typing is when TypeScript infers a type based on the position where an expression appears. This is the main scenario where function parameters do not need explicit annotations.

### Callback Parameters

```typescript
const names = ["Alice", "Bob", "Charlie"];

// No annotations needed - TypeScript knows the callback signature from Array.forEach
names.forEach(function (name) {
  // name is inferred as string
  console.log(name.toUpperCase());
});

// Arrow function version - same inference
names.forEach((name) => {
  console.log(name.toUpperCase());
});
```

### Event Handlers

```typescript
document.addEventListener("click", (event) => {
  // event is inferred as MouseEvent
  console.log(event.clientX, event.clientY);
});

document.addEventListener("keydown", (event) => {
  // event is inferred as KeyboardEvent
  console.log(event.key);
});
```

### Where Contextual Typing Applies

- Arguments to function calls (callbacks)
- Right-hand side of assignments when the left side has a type
- Return statements when the function has a return type annotation
- Type assertions
- Object and array literals in typed positions

### When Contextual Typing Breaks

```typescript
// Extracting a callback to a named function loses context
const names = ["Alice", "Bob"];

// Contextual typing works inline
names.forEach((name) => console.log(name.toUpperCase())); // name is string

// But not when extracted
function logUpperCase(name) {
  // Error: Parameter 'name' implicitly has an 'any' type
  console.log(name.toUpperCase());
}
names.forEach(logUpperCase);

// Fix: annotate the extracted function
function logUpperCase(name: string) {
  console.log(name.toUpperCase());
}
names.forEach(logUpperCase);
```

This is a common source of confusion. The moment you extract an inline callback into a standalone function, you lose contextual typing and must add parameter annotations.

## Function Type Expressions

When you need to describe the type of a function itself (not just its parameters and return), use a function type expression.

### Basic Function Types

```typescript
// Type alias for a function
type MathOp = (a: number, b: number) => number;

const add: MathOp = (a, b) => a + b;
const subtract: MathOp = (a, b) => a - b;
const multiply: MathOp = (a, b) => a * b;

// No parameter annotations needed - contextual typing from MathOp
```

### Callback Type Annotations

```typescript
// Function that accepts a callback
function fetchData(url: string, onSuccess: (data: string) => void) {
  // ... fetch and call onSuccess
}

// Using a type alias
type SuccessHandler = (data: string) => void;
type ErrorHandler = (error: Error) => void;

function fetchData(url: string, onSuccess: SuccessHandler, onError: ErrorHandler) {
  // ...
}
```

### Callable Signatures in Object Types

```typescript
// Object type with a call signature
type Logger = {
  (message: string): void;
  level: string;
};

// Or using interface
interface Logger {
  (message: string): void;
  level: string;
}

function createLogger(): Logger {
  const log = ((message: string) => {
    console.log(`[${log.level}] ${message}`);
  }) as Logger;
  log.level = "info";
  return log;
}
```

### Construct Signatures

```typescript
type UserConstructor = {
  new (name: string, age: number): User;
};

function createUser(ctor: UserConstructor, name: string, age: number): User {
  return new ctor(name, age);
}
```

## Type Annotations vs `as` Assertions vs `satisfies`

These three mechanisms serve different purposes. Choosing the wrong one is a common mistake.

### Type Annotations (`: Type`)

Annotations declare the type upfront. The value must conform to the annotation.

```typescript
// The annotation constrains what can be assigned
const config: Record<string, string> = {
  host: "localhost",
  port: "3000",
};

// config.host is typed as string (widened)
// config.host.toUpperCase(); // works
// But you lose the knowledge of which specific keys exist
// config.unknownKey; // no error - string index allows any key
```

### Type Assertions (`as Type`)

Assertions tell TypeScript "trust me, I know the type." They bypass normal checking.

```typescript
const input = document.getElementById("username") as HTMLInputElement;
// TypeScript trusts you that it is an HTMLInputElement

// Dangerous: assertions can lie
const value = "hello" as unknown as number;
// No error, but value is actually a string at runtime
```

### The `satisfies` Operator

`satisfies` validates that a value matches a type without widening it. You get the best of both worlds: type checking and precise inference.

```typescript
// With annotation: type is Record<string, string>, losing literal key info
const config: Record<string, string> = {
  host: "localhost",
  port: "3000",
};
// config.host is string (OK but wide)
// config.database // no error, any string key is allowed

// With satisfies: type-checked but precise
const config = {
  host: "localhost",
  port: "3000",
} satisfies Record<string, string>;
// config.host is "localhost" (literal type preserved)
// config.database // Error: Property 'database' does not exist
```

### Comparison Table

| Feature | `: Type` | `as Type` | `satisfies Type` |
|---------|----------|-----------|-------------------|
| Type checking | Yes | No (override) | Yes |
| Narrows the type | Widens to annotation | Forces to assertion | Keeps inferred type |
| Catches extra properties | Yes (fresh objects) | No | Yes |
| Catches missing properties | Yes | No | Yes |
| Use case | Contracts, APIs | DOM casts, `any` escape | Validated literals |

### When to Use Each

```typescript
type Theme = "light" | "dark" | "system";

// `: Type` - when you want the DECLARED type
let currentTheme: Theme = "light";
currentTheme = "dark"; // allowed, type is Theme

// `satisfies` - when you want VALIDATION but keep the LITERAL type
const defaultTheme = "light" satisfies Theme;
// defaultTheme is "light" (not Theme), but TypeScript verified it is valid

// `as` - only when you know more than TypeScript
const el = document.querySelector(".btn") as HTMLButtonElement;
```

## Annotating Complex Patterns

### Async Functions

```typescript
// Return type is automatically wrapped in Promise
async function fetchUser(id: string): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
}

// You annotate the unwrapped type, not Promise<...>, when inference handles it
async function fetchUser(id: string) {
  const res = await fetch(`/api/users/${id}`);
  return res.json() as Promise<User>;
}
```

### Generator Functions

```typescript
// Generator<YieldType, ReturnType, NextType>
function* range(start: number, end: number): Generator<number, void, unknown> {
  for (let i = start; i < end; i++) {
    yield i;
  }
}

// Often, letting TypeScript infer is sufficient
function* range(start: number, end: number) {
  for (let i = start; i < end; i++) {
    yield i;
  }
}
// Inferred as Generator<number, void, unknown>
```

### Generic Function Annotations

```typescript
// Generic function with explicit type parameter constraint
function first<T>(arr: T[]): T | undefined {
  return arr[0];
}

// The caller can explicitly pass the type or let inference work
const n = first([1, 2, 3]);       // T inferred as number
const s = first<string>(["a"]);   // T explicitly string

// Generic with constraint annotation
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}
```

### Variables Holding Functions

```typescript
// Arrow function with full annotation
const double: (n: number) => number = (n) => n * 2;

// Using a type alias (preferred for readability)
type Transform = (value: number) => number;
const double: Transform = (n) => n * 2;
const triple: Transform = (n) => n * 3;

// Generic arrow function (note the trailing comma in TSX files)
const identity = <T,>(value: T): T => value;
// The comma after T prevents JSX ambiguity in .tsx files
```

### Class Property Annotations

```typescript
class User {
  // Property annotations
  name: string;
  age: number;
  email: string | null;

  // Definite assignment assertion (use sparingly)
  role!: string; // "I promise this will be assigned before use"

  constructor(name: string, age: number) {
    this.name = name;
    this.age = age;
    this.email = null;
    // role is set by an init method, not the constructor
  }

  // Method annotations follow function annotation rules
  getDisplayName(): string {
    return `${this.name} (${this.age})`;
  }
}
```

## The Annotation Decision Framework

Use this framework to decide whether to annotate:

### Always Annotate

1. **Function parameters** (unless contextual typing applies)
2. **Exported function return types** (API contract)
3. **Variables initialized later** (no initializer to infer from)
4. **Empty arrays** (otherwise `never[]`)
5. **Values from `JSON.parse`** or other `any`-returning functions
6. **Class properties** (constructor-assigned properties can use `strictPropertyInitialization`)

### Let Inference Work

1. **Variables with obvious initializers** (`const x = 5` is clearly a number)
2. **Callback parameters** (contextual typing handles these)
3. **Return types of simple, private functions** (inference is correct and the function is not part of a public API)
4. **`const` declarations with literal values** (inference gives the literal type)
5. **Derived values** (`const doubled = n * 2` is clearly a number)

### Use `satisfies` Instead of Annotation

1. **Configuration objects** where you want both validation and literal types
2. **Lookup tables** where you want keys to be known precisely
3. **Constant values** that must conform to a type but should keep their literal type

```typescript
// Configuration with satisfies
type Config = {
  env: "development" | "staging" | "production";
  port: number;
  debug: boolean;
};

const config = {
  env: "development",
  port: 3000,
  debug: true,
} satisfies Config;

// config.env is "development" (not the union type)
// config.port is 3000 (not number)
// But all values are validated against Config
```

## Common Mistakes

### Mistake 1: Annotating Everything

Over-annotation adds noise without adding safety.

```typescript
// Bad: every obvious type annotated
const name: string = "Alice";
const count: number = 42;
const active: boolean = true;
const items: string[] = ["a", "b", "c"];

// Good: let inference handle the obvious
const name = "Alice";
const count = 42;
const active = true;
const items = ["a", "b", "c"];
```

### Mistake 2: Using `as` When You Should Use an Annotation

```typescript
// Bad: assertion bypasses checking
const config = {
  port: "not-a-number",
} as { port: number };
// No error! config.port is typed as number but is actually a string

// Good: annotation catches the error
const config: { port: number } = {
  port: "not-a-number",
  // Error: Type 'string' is not assignable to type 'number'
};
```

### Mistake 3: Confusing Destructuring Syntax

```typescript
// Bug: this renames, not annotates
const { name: string } = user;
// Creates a variable called `string` with the value of user.name

// Fix: annotate the whole pattern
const { name }: { name: string } = user;
```

### Mistake 4: Optional Parameters in Callbacks

```typescript
// Bad: marking callback parameters as optional
type Handler = (data?: string) => void;
// This means the callback might be called WITHOUT data

// Good: if data is always provided, do not make it optional
type Handler = (data: string) => void;

// If the data truly might be missing:
type Handler = (data: string | undefined) => void;
```

### Mistake 5: Losing Contextual Typing by Extracting Functions

```typescript
// Works: inline callback gets contextual typing
const doubled = [1, 2, 3].map((n) => n * 2);

// Broken: extracted function loses context
function doubleIt(n) {
  // Error: Parameter 'n' implicitly has an 'any' type
  return n * 2;
}
const doubled = [1, 2, 3].map(doubleIt);

// Fix: annotate the extracted function
function doubleIt(n: number): number {
  return n * 2;
}
```

### Mistake 6: Annotating Variables with `satisfies` Instead of Using `satisfies`

```typescript
// Annotation widens the type
const routes: Record<string, { path: string }> = {
  home: { path: "/" },
  about: { path: "/about" },
};
routes.nonexistent; // no error because any string key is valid

// satisfies preserves specific keys
const routes = {
  home: { path: "/" },
  about: { path: "/about" },
} satisfies Record<string, { path: string }>;
routes.nonexistent; // Error: Property 'nonexistent' does not exist
routes.home.path;   // works and is typed as "/"
```

## Best Practices

### 1. Annotate at Boundaries, Infer Internally

Module boundaries (exported functions, class public methods) deserve explicit annotations. Internal implementation details do not.

```typescript
// Public API - annotate
export function calculateTax(amount: number, rate: number): number {
  // Internal - let inference work
  const base = amount * rate;
  const rounded = Math.round(base * 100) / 100;
  return rounded;
}
```

### 2. Prefer Type Aliases for Complex Object Annotations

```typescript
// Hard to read
function process(data: { id: number; name: string; tags: string[]; metadata: { created: Date; updated: Date } }) {}

// Clean
type ProcessData = {
  id: number;
  name: string;
  tags: string[];
  metadata: {
    created: Date;
    updated: Date;
  };
};

function process(data: ProcessData) {}
```

### 3. Use `satisfies` for Configuration and Lookup Objects

```typescript
type Route = {
  path: string;
  requiresAuth: boolean;
};

const routes = {
  home: { path: "/", requiresAuth: false },
  dashboard: { path: "/dashboard", requiresAuth: true },
  profile: { path: "/profile", requiresAuth: true },
} satisfies Record<string, Route>;

// TypeScript knows exactly which keys exist
// routes.settings would be an error
```

### 4. Enable `noImplicitAny` and Let It Guide You

With `noImplicitAny` (included in `strict` mode), TypeScript forces you to annotate wherever inference cannot determine a type. This is the best teacher for learning where annotations are actually needed.

```typescript
// tsconfig.json
{
  "compilerOptions": {
    "strict": true // includes noImplicitAny
  }
}
```

### 5. Use Return Type Annotations for Error Localization

```typescript
// Without return annotation: error appears at call site
function getUser(id: number) {
  // ... complex logic that accidentally returns wrong type
  return { name: "Alice", age: "thirty" }; // age should be number
}
const user = getUser(1);
const nextYear = user.age + 1; // Error HERE, far from the source

// With return annotation: error appears at the function
function getUser(id: number): { name: string; age: number } {
  return { name: "Alice", age: "thirty" };
  // Error HERE: Type 'string' is not assignable to type 'number'
}
```

## Real-World Use Cases

### API Response Typing

```typescript
// Define the expected shape
type ApiResponse<T> = {
  data: T;
  status: number;
  message: string;
};

type User = {
  id: number;
  name: string;
  email: string;
};

// Annotate the return type to enforce the contract
async function fetchUsers(): Promise<ApiResponse<User[]>> {
  const res = await fetch("/api/users");
  // JSON returns any, annotation ensures we catch mismatches early
  const body: ApiResponse<User[]> = await res.json();
  return body;
}

// Callers get full type safety
const { data: users } = await fetchUsers();
users[0].name; // string
// users[0].phone; // Error: Property 'phone' does not exist
```

### Event System with Typed Handlers

```typescript
// Map event names to their payload types
type EventMap = {
  login: { userId: string; timestamp: number };
  logout: { userId: string };
  purchase: { itemId: string; amount: number };
};

// Handler type uses the map
type EventHandler<K extends keyof EventMap> = (payload: EventMap[K]) => void;

class EventBus {
  private handlers: Partial<{ [K in keyof EventMap]: EventHandler<K>[] }> = {};

  on<K extends keyof EventMap>(event: K, handler: EventHandler<K>) {
    if (!this.handlers[event]) {
      this.handlers[event] = [];
    }
    (this.handlers[event] as EventHandler<K>[]).push(handler);
  }

  emit<K extends keyof EventMap>(event: K, payload: EventMap[K]) {
    const eventHandlers = this.handlers[event] as EventHandler<K>[] | undefined;
    eventHandlers?.forEach((handler) => handler(payload));
  }
}

const bus = new EventBus();

// Contextual typing gives full autocomplete on the payload
bus.on("login", (payload) => {
  // payload is { userId: string; timestamp: number }
  console.log(payload.userId);
});

bus.on("purchase", (payload) => {
  // payload is { itemId: string; amount: number }
  console.log(payload.amount.toFixed(2));
});
```

### Form Validation with Precise Types

```typescript
type ValidationRule = {
  validate: (value: string) => boolean;
  message: string;
};

type FormSchema<T extends Record<string, unknown>> = {
  [K in keyof T]: {
    initialValue: T[K];
    rules: ValidationRule[];
  };
};

// satisfies validates the schema while preserving field names
const loginSchema = {
  email: {
    initialValue: "",
    rules: [
      { validate: (v) => v.length > 0, message: "Email is required" },
      { validate: (v) => v.includes("@"), message: "Invalid email" },
    ],
  },
  password: {
    initialValue: "",
    rules: [
      { validate: (v) => v.length > 0, message: "Password is required" },
      { validate: (v) => v.length >= 8, message: "At least 8 characters" },
    ],
  },
} satisfies FormSchema<{ email: string; password: string }>;

// TypeScript knows the exact fields
loginSchema.email.initialValue; // string
// loginSchema.username; // Error: Property 'username' does not exist
```

---

> **Key Takeaway:** Annotate at module boundaries (exported functions, class public APIs, delayed initializations) and let TypeScript infer everywhere else. Use `satisfies` when you need both type validation and preserved literal types, and reserve `as` assertions for cases where you genuinely know more than the compiler.
