# 7.7 Type Aliases

> Type aliases give a name to any type, from simple primitives to complex computed types. They are the foundation for unions, intersections, mapped types, conditional types, template literal types, and most of TypeScript's advanced type-level programming.

---

## What Is a Type Alias?

A type alias creates a new name for an existing type. It does not create a new, distinct type. Wherever the alias is used, TypeScript substitutes the underlying type.

```typescript
type UserID = string;
type Age = number;

// UserID is interchangeable with string
let id: UserID = "abc-123";
let name: string = id; // no error, UserID IS string
```

The `type` keyword followed by a name, an equals sign, and any valid type expression defines an alias.

## Basic Type Aliases

### Primitive Aliases

Give semantic names to primitive types for readability.

```typescript
type Milliseconds = number;
type Pixels = number;
type Email = string;
type UUID = string;

function delay(ms: Milliseconds): Promise<void> {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

function setWidth(el: HTMLElement, width: Pixels) {
  el.style.width = `${width}px`;
}
```

Note: these are purely documentation. TypeScript does not distinguish `Milliseconds` from `Pixels` since both resolve to `number`. If you need truly distinct types, you must use branded types (covered later).

### Object Type Aliases

```typescript
type User = {
  id: string;
  name: string;
  email: string;
};

type Point = {
  x: number;
  y: number;
};

function distance(a: Point, b: Point): number {
  return Math.sqrt((b.x - a.x) ** 2 + (b.y - a.y) ** 2);
}
```

Object type aliases look almost identical to interfaces. The key behavioral differences were covered in 7.6 (interfaces cannot be redeclared/merged, type aliases cannot use `extends`). For object shapes, either works. Choose one convention and be consistent.

### Function Type Aliases

```typescript
type Predicate<T> = (value: T) => boolean;
type Comparator<T> = (a: T, b: T) => number;
type EventHandler = (event: Event) => void;

const isPositive: Predicate<number> = (n) => n > 0;
const byAge: Comparator<{ age: number }> = (a, b) => a.age - b.age;
```

Function type aliases are cleaner than writing out the full signature every time, especially for callbacks that appear in multiple places.

### Tuple Type Aliases

```typescript
type Coordinate = [x: number, y: number];
type RGBColor = [red: number, green: number, blue: number];
type KeyValue<K, V> = [key: K, value: V];

const origin: Coordinate = [0, 0];
const red: RGBColor = [255, 0, 0];
const entry: KeyValue<string, number> = ["age", 30];
```

## Union Types (Preview)

Union types are covered in depth in 7.8, but they are central to type aliases because interfaces cannot express them.

```typescript
// Only possible with type, not interface
type StringOrNumber = string | number;
type Status = "pending" | "active" | "inactive";
type Result<T> = T | Error;

// Discriminated union (tagged union)
type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "rectangle"; width: number; height: number }
  | { kind: "triangle"; base: number; height: number };

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "rectangle":
      return shape.width * shape.height;
    case "triangle":
      return (shape.base * shape.height) / 2;
  }
}
```

## Intersection Types (Preview)

Intersection types combine multiple types into one. Again, covered in depth in 7.8, but previewed here since type aliases are the primary vehicle.

```typescript
type HasId = { id: string };
type HasTimestamps = { createdAt: Date; updatedAt: Date };
type HasName = { name: string };

// Combine with &
type Entity = HasId & HasTimestamps;
type NamedEntity = Entity & HasName;

const user: NamedEntity = {
  id: "abc",
  name: "Alice",
  createdAt: new Date(),
  updatedAt: new Date(),
};
```

## Generic Type Aliases

Type aliases can be parameterized with generics, making them reusable across different types.

### Basic Generics

```typescript
type Nullable<T> = T | null;
type Optional<T> = T | undefined;
type Maybe<T> = T | null | undefined;

let name: Nullable<string> = "Alice";
name = null; // valid

type Pair<A, B> = {
  first: A;
  second: B;
};

const pair: Pair<string, number> = { first: "age", second: 30 };
```

### Generic Constraints

```typescript
type WithId<T extends { id: string }> = T & { id: string };

// Works
type UserWithId = WithId<{ name: string; id: string }>;

// Error: Type '{ name: string }' does not satisfy constraint '{ id: string }'
// type Bad = WithId<{ name: string }>;
```

### Default Type Parameters

```typescript
type ApiResponse<T = unknown, E = string> = {
  data: T;
  error: E | null;
  status: number;
};

// Using defaults
const res1: ApiResponse = { data: "hello", error: null, status: 200 };

// Overriding
const res2: ApiResponse<User[], Error> = {
  data: [],
  error: null,
  status: 200,
};
```

### Recursive Type Aliases

Type aliases can reference themselves. This is essential for modeling nested data structures.

```typescript
// JSON value type
type JsonValue =
  | string
  | number
  | boolean
  | null
  | JsonValue[]
  | { [key: string]: JsonValue };

const config: JsonValue = {
  name: "app",
  version: 1,
  features: ["auth", "logging"],
  database: {
    host: "localhost",
    port: 5432,
    ssl: true,
  },
};
```

```typescript
// Tree structure
type TreeNode<T> = {
  value: T;
  children: TreeNode<T>[];
};

const fileSystem: TreeNode<string> = {
  value: "root",
  children: [
    {
      value: "src",
      children: [
        { value: "index.ts", children: [] },
        { value: "utils.ts", children: [] },
      ],
    },
    {
      value: "package.json",
      children: [],
    },
  ],
};
```

```typescript
// Linked list
type LinkedList<T> = {
  value: T;
  next: LinkedList<T> | null;
};

const list: LinkedList<number> = {
  value: 1,
  next: { value: 2, next: { value: 3, next: null } },
};
```

```typescript
// Deep readonly (recursive utility)
type DeepReadonly<T> = {
  readonly [K in keyof T]: T[K] extends object ? DeepReadonly<T[K]> : T[K];
};

type Config = {
  database: {
    host: string;
    port: number;
    credentials: {
      user: string;
      password: string;
    };
  };
};

type FrozenConfig = DeepReadonly<Config>;
// All nested properties are now readonly
```

## Mapped Types

Mapped types iterate over the keys of an existing type and transform each property. They can only be expressed as type aliases.

### Basic Mapped Types

```typescript
// Make all properties optional
type MyPartial<T> = {
  [K in keyof T]?: T[K];
};

// Make all properties required
type MyRequired<T> = {
  [K in keyof T]-?: T[K];
};

// Make all properties readonly
type MyReadonly<T> = {
  readonly [K in keyof T]: T[K];
};

// Remove readonly from all properties
type Mutable<T> = {
  -readonly [K in keyof T]: T[K];
};
```

### Mapped Types with Key Remapping (`as`)

TypeScript 4.1 introduced key remapping with `as`.

```typescript
// Prefix all keys with "get"
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};

type User = {
  name: string;
  age: number;
  email: string;
};

type UserGetters = Getters<User>;
// {
//   getName: () => string;
//   getAge: () => number;
//   getEmail: () => string;
// }
```

```typescript
// Filter out certain keys
type RemoveKind<T> = {
  [K in keyof T as Exclude<K, "kind">]: T[K];
};

type Circle = {
  kind: "circle";
  radius: number;
};

type CircleWithoutKind = RemoveKind<Circle>;
// { radius: number }
```

### Mapped Types over Unions

```typescript
// Create an object type from a union of string literals
type Flags<T extends string> = {
  [K in T]: boolean;
};

type Permission = "read" | "write" | "execute";
type PermissionFlags = Flags<Permission>;
// { read: boolean; write: boolean; execute: boolean }
```

## Conditional Types

Conditional types express "if-then-else" logic at the type level. They use the `extends` keyword as a type-level ternary.

### Basic Syntax

```typescript
type IsString<T> = T extends string ? true : false;

type A = IsString<string>;  // true
type B = IsString<number>;  // false
type C = IsString<"hello">; // true (literal extends string)
```

### Extracting Types with `infer`

The `infer` keyword declares a type variable within a conditional type, allowing you to "capture" part of a type.

```typescript
// Extract the return type of a function
type MyReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

type A = MyReturnType<() => string>;           // string
type B = MyReturnType<(x: number) => boolean>; // boolean

// Extract array element type
type ElementType<T> = T extends (infer E)[] ? E : T;

type C = ElementType<string[]>;  // string
type D = ElementType<number>;    // number (not an array, returns T)

// Extract Promise resolved type
type Awaited<T> = T extends Promise<infer U> ? Awaited<U> : T;

type E = Awaited<Promise<string>>;           // string
type F = Awaited<Promise<Promise<number>>>;  // number (recursive unwrap)
```

### Distributive Conditional Types

When a conditional type acts on a union, it distributes over each member of the union individually.

```typescript
type ToArray<T> = T extends any ? T[] : never;

type Result = ToArray<string | number>;
// string[] | number[]   (NOT (string | number)[])

// Distribution happens because T is a "naked" type parameter
// To prevent distribution, wrap in a tuple:
type ToArrayNonDist<T> = [T] extends [any] ? T[] : never;

type Result2 = ToArrayNonDist<string | number>;
// (string | number)[]
```

### Practical Conditional Types

```typescript
// Extract only function properties from an object
type FunctionPropertyNames<T> = {
  [K in keyof T]: T[K] extends (...args: any[]) => any ? K : never;
}[keyof T];

type FunctionProperties<T> = Pick<T, FunctionPropertyNames<T>>;

type Mixed = {
  name: string;
  age: number;
  greet(): void;
  calculate(x: number): number;
};

type Fns = FunctionProperties<Mixed>;
// { greet(): void; calculate(x: number): number }
```

## Template Literal Types

Template literal types use the same backtick syntax as JavaScript template literals, but in type positions. They build string literal types from other string literal types.

### Basic Template Literals

```typescript
type Greeting = `Hello, ${string}`;

const a: Greeting = "Hello, World";    // valid
const b: Greeting = "Hello, TypeScript"; // valid
// const c: Greeting = "Hi, World";    // Error: does not start with "Hello, "
```

### Union Distribution in Template Literals

When a union is interpolated, TypeScript computes the cross product.

```typescript
type Color = "red" | "green" | "blue";
type Size = "small" | "medium" | "large";

type ColorSize = `${Color}-${Size}`;
// "red-small" | "red-medium" | "red-large"
// | "green-small" | "green-medium" | "green-large"
// | "blue-small" | "blue-medium" | "blue-large"
```

### Intrinsic String Manipulation Types

TypeScript provides four built-in string transformation types:

```typescript
type Upper = Uppercase<"hello">;       // "HELLO"
type Lower = Lowercase<"HELLO">;       // "hello"
type Cap = Capitalize<"hello">;        // "Hello"
type Uncap = Uncapitalize<"Hello">;    // "hello"
```

### Event Handler Pattern

A classic use case: derive event handler names from event types.

```typescript
type EventName = "click" | "focus" | "blur";

type HandlerName = `on${Capitalize<EventName>}`;
// "onClick" | "onFocus" | "onBlur"

type Handlers = {
  [K in HandlerName]: (event: Event) => void;
};
// {
//   onClick: (event: Event) => void;
//   onFocus: (event: Event) => void;
//   onBlur: (event: Event) => void;
// }
```

### Parsing Strings with Template Literals and `infer`

```typescript
// Extract parts of a string type
type ParseRoute<T extends string> =
  T extends `${infer Start}/:${infer Param}/${infer Rest}`
    ? { start: Start; param: Param; rest: ParseRoute<`/${Rest}`> }
    : T extends `${infer Start}/:${infer Param}`
      ? { start: Start; param: Param }
      : T;

type Route = ParseRoute<"/users/:id/posts/:postId">;
// { start: "/users"; param: "id"; rest: { start: "/posts"; param: "postId" } }
```

### Autocomplete-Friendly Patterns

A useful trick: allow any string but suggest specific values for autocomplete.

```typescript
// This gives autocomplete for known values while still accepting any string
type Color = "red" | "blue" | "green" | (string & {});

let color: Color = "red";   // autocomplete suggests "red", "blue", "green"
color = "purple";            // still valid, no error
```

The `(string & {})` trick prevents TypeScript from collapsing the union into just `string`, preserving the literal suggestions in IDE autocomplete.

## Built-in Utility Types (Type Alias Based)

TypeScript ships with utility types that are all implemented as type aliases. Understanding how they work demystifies the type system.

### Property Modifiers

```typescript
// Partial<T> - all properties optional
type UserUpdate = Partial<User>;
// { id?: string; name?: string; email?: string; }

// Required<T> - all properties required
type StrictConfig = Required<Config>;

// Readonly<T> - all properties readonly
type FrozenUser = Readonly<User>;

// How Partial is implemented:
// type Partial<T> = { [K in keyof T]?: T[K] };
```

### Property Selection

```typescript
type User = {
  id: string;
  name: string;
  email: string;
  password: string;
};

// Pick<T, K> - select specific properties
type PublicUser = Pick<User, "id" | "name" | "email">;
// { id: string; name: string; email: string }

// Omit<T, K> - remove specific properties
type UserWithoutPassword = Omit<User, "password">;
// { id: string; name: string; email: string }
```

### Union Filtering

```typescript
type Status = "active" | "inactive" | "pending" | "deleted";

// Exclude<T, U> - remove members from union
type ActiveStatuses = Exclude<Status, "deleted">;
// "active" | "inactive" | "pending"

// Extract<T, U> - keep only matching members
type InactiveStatuses = Extract<Status, "inactive" | "deleted">;
// "inactive" | "deleted"
```

### Function-Related Utilities

```typescript
function createUser(name: string, age: number, email: string): User {
  return { id: crypto.randomUUID(), name, email, password: "" };
}

// Parameters<T> - extract parameter types as a tuple
type CreateUserParams = Parameters<typeof createUser>;
// [name: string, age: number, email: string]

// ReturnType<T> - extract return type
type CreateUserResult = ReturnType<typeof createUser>;
// User

// ConstructorParameters<T> - extract constructor parameter types
type DateArgs = ConstructorParameters<typeof Date>;
```

### Record

```typescript
// Record<K, V> - create an object type with keys K and values V
type UserRoles = Record<string, "admin" | "user" | "moderator">;

// Practical: enum-like exhaustive mapping
type Status = "active" | "inactive" | "pending";
type StatusConfig = Record<Status, { label: string; color: string }>;

const statusMap: StatusConfig = {
  active: { label: "Active", color: "green" },
  inactive: { label: "Inactive", color: "gray" },
  pending: { label: "Pending", color: "yellow" },
  // Missing any status would be a compile error
};
```

### NonNullable

```typescript
type MaybeString = string | null | undefined;
type DefiniteString = NonNullable<MaybeString>;
// string
```

## Branded Types (Nominal Typing with Type Aliases)

TypeScript's structural type system means `Milliseconds` and `Pixels` are both just `number`. Branded types create distinct types that cannot be accidentally mixed.

```typescript
// The brand pattern
type Brand<T, B extends string> = T & { readonly __brand: B };

type USD = Brand<number, "USD">;
type EUR = Brand<number, "EUR">;
type Pixels = Brand<number, "Pixels">;
type Milliseconds = Brand<number, "Milliseconds">;

// Constructor functions that "brand" the value
function usd(amount: number): USD {
  return amount as USD;
}

function eur(amount: number): EUR {
  return amount as EUR;
}

function pixels(value: number): Pixels {
  return value as Pixels;
}

// Now you cannot accidentally mix currencies
function addUSD(a: USD, b: USD): USD {
  return (a + b) as USD;
}

const price = usd(10);
const tax = usd(2);
const total = addUSD(price, tax); // OK

const euroPrice = eur(10);
// addUSD(price, euroPrice); // Error: EUR is not assignable to USD

// Cannot accidentally pass pixels where milliseconds are expected
function delay(ms: Milliseconds): Promise<void> {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

const width = pixels(100);
// delay(width); // Error: Pixels is not assignable to Milliseconds
```

This pattern is widely used in production codebases to prevent unit confusion bugs.

## Type Aliases Cannot Do (What Interfaces Can)

### No Declaration Merging

```typescript
type User = { name: string };
// type User = { age: number }; // Error: Duplicate identifier 'User'
```

This is actually an advantage when you want to prevent accidental merging.

### No `extends` Keyword

Type aliases use `&` (intersection) for composition instead of `extends`.

```typescript
// Interface way
interface Animal {
  name: string;
}
interface Dog extends Animal {
  breed: string;
}

// Type alias way
type Animal = {
  name: string;
};
type Dog = Animal & {
  breed: string;
};
```

Both produce the same shape, but the `extends` approach has slightly better performance in the type checker because TypeScript caches interface relationships.

### Slightly Worse Error Messages (Sometimes)

Interfaces are shown by name in error messages. Complex type aliases are sometimes expanded inline, making error messages harder to read.

```typescript
// Interface error: "Type '...' is not assignable to type 'User'"
// Type alias error might expand: "Type '...' is not assignable to type '{ id: string; name: string; email: string; }'"
```

TypeScript has improved this significantly in recent versions, but it is still worth knowing.

## Common Mistakes

### Mistake 1: Thinking Type Aliases Create New Types

```typescript
type UserID = string;
type ProductID = string;

function getUser(id: UserID) { /* ... */ }

const productId: ProductID = "prod-123";
getUser(productId); // No error! Both are just string

// If you need distinct types, use branded types (shown above)
```

### Mistake 2: Overly Complex Nested Types

```typescript
// Bad: deeply nested inline type is unreadable
type Response = {
  data: {
    users: {
      id: string;
      profile: {
        name: string;
        settings: {
          theme: "light" | "dark";
          notifications: {
            email: boolean;
            push: boolean;
          };
        };
      };
    }[];
  };
};

// Good: break into named types
type NotificationSettings = {
  email: boolean;
  push: boolean;
};

type UserSettings = {
  theme: "light" | "dark";
  notifications: NotificationSettings;
};

type UserProfile = {
  name: string;
  settings: UserSettings;
};

type User = {
  id: string;
  profile: UserProfile;
};

type Response = {
  data: {
    users: User[];
  };
};
```

### Mistake 3: Using Type Aliases When Declaration Merging Is Needed

```typescript
// If you need to augment a third-party library's types:

// This works with interfaces
declare module "express" {
  interface Request {
    userId?: string;
  }
}

// This does NOT work with type aliases
// You cannot merge into a type alias from another module
```

### Mistake 4: Intersection Instead of Extending Creates Confusing Conflicts

```typescript
type A = { value: string };
type B = { value: number };

// Intersection creates an impossible type (string & number = never)
type C = A & B;
// C.value is string & number, which is never

// With interfaces, this would be caught as an error immediately:
// interface C extends A, B {} // Error: incompatible types for 'value'
```

### Mistake 5: Hitting Recursion Limits

```typescript
// Too-deep recursion can crash the compiler
type DeepNest<T, Depth extends unknown[] = []> =
  Depth["length"] extends 100
    ? T
    : { value: T; nested: DeepNest<T, [...Depth, unknown]> };

// Error: Type instantiation is excessively deep and possibly infinite.

// Fix: add a reasonable depth cap or restructure
```

## Best Practices

### 1. Use Type Aliases for Non-Object Types

Interfaces cannot express unions, tuples, or primitive aliases. Use type aliases for these.

```typescript
type Result<T, E = Error> = { ok: true; value: T } | { ok: false; error: E };
type Coordinate = [number, number];
type EventName = "click" | "hover" | "focus";
```

### 2. Extract Reusable Generic Utilities

```typescript
// Build a library of project-specific utility types
type Nullable<T> = T | null;
type AsyncResult<T> = Promise<Result<T>>;
type DeepPartial<T> = {
  [K in keyof T]?: T[K] extends object ? DeepPartial<T[K]> : T[K];
};
```

### 3. Name Intermediate Types for Readability

```typescript
// Instead of
type Handlers = {
  [K in keyof EventMap as `on${Capitalize<string & K>}`]: (
    payload: EventMap[K],
  ) => void;
};

// Break it into steps
type EventName = keyof EventMap;
type HandlerName<K extends string> = `on${Capitalize<K>}`;
type HandlerFn<K extends EventName> = (payload: EventMap[K]) => void;

type Handlers = {
  [K in EventName as HandlerName<string & K>]: HandlerFn<K>;
};
```

### 4. Use `satisfies` with Type Aliases for Validated Constants

```typescript
type Route = {
  path: string;
  auth: boolean;
};

type Routes = Record<string, Route>;

const routes = {
  home: { path: "/", auth: false },
  dashboard: { path: "/dashboard", auth: true },
} satisfies Routes;

// routes.home.path is "/" (literal), not string
// routes.settings would be an error (key does not exist)
```

### 5. Prefer Explicit Type Aliases Over Inline Types for Reuse

```typescript
// Bad: same shape repeated
function createUser(data: { name: string; email: string }) { /* ... */ }
function updateUser(id: string, data: { name: string; email: string }) { /* ... */ }
function validateUser(data: { name: string; email: string }): boolean { return true; }

// Good: named once, used everywhere
type UserInput = { name: string; email: string };
function createUser(data: UserInput) { /* ... */ }
function updateUser(id: string, data: UserInput) { /* ... */ }
function validateUser(data: UserInput): boolean { return true; }
```

## Real-World Use Cases

### API Layer with Generic Response Types

```typescript
type ApiError = {
  code: string;
  message: string;
  details?: Record<string, string[]>;
};

type ApiResponse<T> =
  | { success: true; data: T; meta?: { page: number; total: number } }
  | { success: false; error: ApiError };

type PaginatedResponse<T> = ApiResponse<T> & {
  meta: { page: number; total: number; perPage: number };
};

async function fetchApi<T>(url: string): Promise<ApiResponse<T>> {
  const response = await fetch(url);
  const json = await response.json();

  if (!response.ok) {
    return {
      success: false,
      error: { code: String(response.status), message: json.message ?? "Unknown error" },
    };
  }

  return { success: true, data: json as T };
}

// Usage
type User = { id: string; name: string };
const result = await fetchApi<User[]>("/api/users");

if (result.success) {
  // result.data is User[]
  console.log(result.data.length);
} else {
  // result.error is ApiError
  console.error(result.error.message);
}
```

### State Machine Types

```typescript
type State =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: string }
  | { status: "error"; error: Error; retryCount: number };

type Action =
  | { type: "FETCH" }
  | { type: "SUCCESS"; payload: string }
  | { type: "ERROR"; error: Error }
  | { type: "RETRY" };

function reducer(state: State, action: Action): State {
  switch (state.status) {
    case "idle":
      if (action.type === "FETCH") return { status: "loading" };
      return state;

    case "loading":
      if (action.type === "SUCCESS") return { status: "success", data: action.payload };
      if (action.type === "ERROR") return { status: "error", error: action.error, retryCount: 0 };
      return state;

    case "error":
      if (action.type === "RETRY") return { status: "loading" };
      return state;

    case "success":
      if (action.type === "FETCH") return { status: "loading" };
      return state;
  }
}
```

### Type-Safe Event System with Template Literals

```typescript
type EventMap = {
  user: {
    login: { userId: string; timestamp: number };
    logout: { userId: string };
    update: { userId: string; fields: string[] };
  };
  order: {
    create: { orderId: string; total: number };
    cancel: { orderId: string; reason: string };
  };
};

// Build dot-notation event names: "user.login" | "user.logout" | ...
type EventNames = {
  [Domain in keyof EventMap]: {
    [Action in keyof EventMap[Domain]]: `${string & Domain}.${string & Action}`;
  }[keyof EventMap[Domain]];
}[keyof EventMap];

// "user.login" | "user.logout" | "user.update" | "order.create" | "order.cancel"

// Extract payload type for a given event name
type EventPayload<T extends EventNames> =
  T extends `${infer D}.${infer A}`
    ? D extends keyof EventMap
      ? A extends keyof EventMap[D]
        ? EventMap[D][A]
        : never
      : never
    : never;

type LoginPayload = EventPayload<"user.login">;
// { userId: string; timestamp: number }

// Type-safe emit function
function emit<T extends EventNames>(event: T, payload: EventPayload<T>): void {
  console.log(`[${event}]`, payload);
}

emit("user.login", { userId: "abc", timestamp: Date.now() }); // OK
// emit("user.login", { orderId: "xyz" }); // Error: missing userId and timestamp
// emit("user.unknown", {}); // Error: "user.unknown" is not in EventNames
```

---

> **Key Takeaway:** Type aliases are TypeScript's most versatile type construct. Use them for unions, tuples, mapped types, conditional types, template literal types, and any type expression that interfaces cannot express. For plain object shapes, either interfaces or type aliases work. Pick a convention and stay consistent.
