# 7.8 Union and Intersection Types

> Union types (`|`) represent values that can be one of several types. Intersection types (`&`) combine multiple types into one that has all their properties. Together they are the fundamental building blocks of type composition in TypeScript.

---

## Union Types

A union type describes a value that can be one of several types. The `|` operator separates each member of the union.

```typescript
type StringOrNumber = string | number;

let value: StringOrNumber;
value = "hello"; // valid
value = 42;      // valid
// value = true; // Error: Type 'boolean' is not assignable to type 'StringOrNumber'
```

### Property Access on Unions

You can only access properties and methods that exist on **every** member of the union. This is the fundamental rule.

```typescript
type StringOrNumber = string | number;

function process(value: StringOrNumber) {
  // Both string and number have .toString()
  value.toString(); // OK

  // .toUpperCase() only exists on string
  // value.toUpperCase(); // Error: Property 'toUpperCase' does not exist on type 'number'

  // .toFixed() only exists on number
  // value.toFixed(2); // Error: Property 'toFixed' does not exist on type 'string'
}
```

This makes sense: if TypeScript does not know whether `value` is a `string` or `number`, allowing `.toUpperCase()` would crash at runtime when `value` is `42`.

### Narrowing Unions

To access type-specific members, you must narrow the union first. TypeScript tracks control flow to determine which branch you are in.

```typescript
function process(value: string | number) {
  if (typeof value === "string") {
    // TypeScript knows value is string here
    console.log(value.toUpperCase());
  } else {
    // TypeScript knows value is number here
    console.log(value.toFixed(2));
  }
}
```

Common narrowing techniques (covered in depth in 7.10):

```typescript
// typeof
if (typeof x === "string") { /* x is string */ }

// instanceof
if (x instanceof Date) { /* x is Date */ }

// in operator
if ("name" in x) { /* x has a name property */ }

// truthiness
if (x) { /* x is not null/undefined/0/""/false */ }

// equality
if (x === null) { /* x is null */ }
```

### Union of Object Types

```typescript
type Cat = {
  name: string;
  purr(): void;
};

type Dog = {
  name: string;
  bark(): void;
};

type Pet = Cat | Dog;

function interact(pet: Pet) {
  // name exists on both Cat and Dog
  console.log(pet.name); // OK

  // purr() only exists on Cat
  // pet.purr(); // Error

  // Use 'in' to narrow
  if ("purr" in pet) {
    pet.purr(); // TypeScript knows pet is Cat
  } else {
    pet.bark(); // TypeScript knows pet is Dog
  }
}
```

## Discriminated Unions (Tagged Unions)

Discriminated unions are the most powerful pattern for working with union types. Each member of the union has a common property (the "discriminant" or "tag") with a unique literal value.

### The Pattern

```typescript
type Circle = {
  kind: "circle";
  radius: number;
};

type Rectangle = {
  kind: "rectangle";
  width: number;
  height: number;
};

type Triangle = {
  kind: "triangle";
  base: number;
  height: number;
};

type Shape = Circle | Rectangle | Triangle;
```

The `kind` property is the discriminant. Each shape has a unique `kind` value.

### Narrowing with the Discriminant

```typescript
function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      // TypeScript knows shape is Circle here
      return Math.PI * shape.radius ** 2;

    case "rectangle":
      // TypeScript knows shape is Rectangle here
      return shape.width * shape.height;

    case "triangle":
      // TypeScript knows shape is Triangle here
      return (shape.base * shape.height) / 2;
  }
}
```

TypeScript narrows `shape` to the correct member based on the `kind` value. You get full autocomplete for the properties of each specific shape.

### Exhaustiveness Checking with `never`

The `never` type represents something that should not exist. If you handle all union members in a switch, the default case has type `never`. This lets you create a compile-time check that forces updates when new members are added.

```typescript
function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "rectangle":
      return shape.width * shape.height;
    case "triangle":
      return (shape.base * shape.height) / 2;
    default:
      // If all cases are handled, shape is never
      const _exhaustive: never = shape;
      return _exhaustive;
  }
}
```

Now if you add a new shape to the union:

```typescript
type Ellipse = {
  kind: "ellipse";
  rx: number;
  ry: number;
};

type Shape = Circle | Rectangle | Triangle | Ellipse;
```

The `area` function immediately produces a compile error at the `default` case because `Ellipse` is not handled, making `shape` of type `Ellipse` rather than `never`. This forces you to add the missing case.

### Helper Function for Exhaustive Checks

```typescript
function assertNever(value: never, message?: string): never {
  throw new Error(message ?? `Unexpected value: ${value}`);
}

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "rectangle":
      return shape.width * shape.height;
    case "triangle":
      return (shape.base * shape.height) / 2;
    default:
      return assertNever(shape, `Unhandled shape: ${(shape as any).kind}`);
  }
}
```

This provides both a compile-time error for unhandled cases and a runtime error if somehow a value slips through.

### Multiple Discriminant Properties

The discriminant does not have to be a single property named `kind`. Any common property with literal types works, and you can even use multiple properties together.

```typescript
type Success = { status: "success"; code: 200; data: string };
type Redirect = { status: "redirect"; code: 301 | 302; location: string };
type NotFound = { status: "error"; code: 404; message: string };
type ServerError = { status: "error"; code: 500; message: string; stack?: string };

type HttpResponse = Success | Redirect | NotFound | ServerError;

function handleResponse(res: HttpResponse) {
  switch (res.status) {
    case "success":
      console.log(res.data);
      break;
    case "redirect":
      console.log(`Redirect to: ${res.location}`);
      break;
    case "error":
      // Still a union of NotFound | ServerError
      // Narrow further with code
      if (res.code === 404) {
        console.log("Not found:", res.message);
      } else {
        console.log("Server error:", res.message, res.stack);
      }
      break;
  }
}
```

## Union Type Mechanics

### Union of Literal Types

The most common use of unions is combining literal types to create constrained sets of values.

```typescript
type Direction = "north" | "south" | "east" | "west";
type HttpMethod = "GET" | "POST" | "PUT" | "DELETE" | "PATCH";
type DiceRoll = 1 | 2 | 3 | 4 | 5 | 6;
type Falsy = false | 0 | "" | null | undefined;

function move(direction: Direction, steps: number) {
  // direction can only be one of the four values
}

move("north", 5); // OK
// move("up", 5); // Error: Argument of type '"up"' is not assignable to parameter of type 'Direction'
```

### Union with `null` and `undefined`

TypeScript's `strictNullChecks` mode (recommended) removes `null` and `undefined` from all types by default. You must explicitly include them in a union.

```typescript
// With strictNullChecks: true
type MaybeString = string | null;
type OptionalNumber = number | undefined;

function findUser(id: string): User | null {
  // explicitly model the "not found" case
  const user = db.get(id);
  return user ?? null;
}

const user = findUser("abc");
if (user !== null) {
  // user is User here, not User | null
  console.log(user.name);
}
```

### Distributing Operations Over Unions

Many TypeScript operations distribute over unions automatically.

```typescript
// keyof distributes
type A = { x: number; y: number };
type B = { y: number; z: number };

type KeysOfUnion = keyof (A | B);
// "y" - only the common key

type KeysOfIntersection = keyof (A & B);
// "x" | "y" | "z" - all keys

// Conditional types distribute
type ToArray<T> = T extends any ? T[] : never;
type Result = ToArray<string | number>;
// string[] | number[]
```

### Union Reduction and Widening

TypeScript reduces unions by removing subtypes.

```typescript
// string literal is a subtype of string
type A = "hello" | string;
// Reduced to: string (the literal is absorbed)

// number literal is a subtype of number
type B = 42 | number;
// Reduced to: number

// This matters when you want to keep literals for autocomplete
// Use the (string & {}) trick:
type Color = "red" | "blue" | "green" | (string & {});
// Literals are preserved for autocomplete, but any string is accepted
```

## Intersection Types

An intersection type combines multiple types into one. The resulting type has **all** properties from **all** constituent types. Think of it as "AND": the value must satisfy every type simultaneously.

```typescript
type HasName = { name: string };
type HasAge = { age: number };
type HasEmail = { email: string };

type Person = HasName & HasAge & HasEmail;

const alice: Person = {
  name: "Alice",
  age: 30,
  email: "alice@example.com",
  // All three properties are required
};
```

### Intersection of Object Types

Intersecting object types merges their properties.

```typescript
type Printable = {
  print(): void;
};

type Serializable = {
  serialize(): string;
};

type Loggable = {
  log(level: string): void;
};

type PrintableSerializableLoggable = Printable & Serializable & Loggable;
// Has: print(), serialize(), log()

const obj: PrintableSerializableLoggable = {
  print() { console.log("Printing..."); },
  serialize() { return JSON.stringify(this); },
  log(level) { console.log(`[${level}]`); },
};
```

### Intersection vs `extends`

For object types, intersection (`&`) and interface `extends` produce similar results, but they behave differently when properties conflict.

```typescript
// extends: conflicts produce a compile error
interface A { value: string }
// interface B extends A { value: number } // Error: incompatible types

// intersection: conflicts silently create 'never'
type A2 = { value: string };
type B2 = { value: number };
type C2 = A2 & B2;
// C2.value is string & number, which is never
// No error at the type definition! The error appears when you try to use it.

const obj: C2 = {
  value: "hello", // Error: Type 'string' is not assignable to type 'never'
};
```

This is a critical difference. Interfaces catch conflicts immediately. Intersections defer the error.

## Intersection Mechanics

### Intersection of Primitives

Intersecting incompatible primitives always results in `never`.

```typescript
type Impossible = string & number; // never
type AlsoImpossible = boolean & string; // never
```

No value can be both a `string` and a `number` simultaneously. The result is `never`, meaning no value can satisfy this type.

### Intersection of Compatible Types

When types overlap, the intersection is the overlap.

```typescript
type A = string & (string | number);
// string (string is the overlap)

type B = "hello" & string;
// "hello" (the literal is more specific and satisfies both)

type C = { a: number } & { a: number; b: string };
// { a: number; b: string } (the more complete type satisfies both)
```

### Intersection of Union Types

Distributing intersections over unions follows set theory.

```typescript
type A = ("a" | "b") & ("b" | "c");
// "b" - only the common member

type B = (string | number) & (number | boolean);
// number - the overlap between the two unions
```

### Property Merging in Detail

When two object types share a property name, the property types are intersected (not unioned).

```typescript
type A = { shared: string | number; onlyA: boolean };
type B = { shared: string | boolean; onlyB: Date };

type AB = A & B;
// {
//   shared: (string | number) & (string | boolean); // simplifies to string
//   onlyA: boolean;
//   onlyB: Date;
// }

// shared becomes string because:
// (string | number) & (string | boolean)
// = (string & string) | (string & boolean) | (number & string) | (number & boolean)
// = string | never | never | never
// = string
```

### Method Intersection Creates Overloads

When two object types both have a method with the same name, the intersection creates an overloaded signature.

```typescript
type A = {
  handle(input: string): string;
};

type B = {
  handle(input: number): number;
};

type AB = A & B;

declare const ab: AB;
ab.handle("hello"); // returns string
ab.handle(42);      // returns number
```

## Intersection for Mixins and Composition

Intersections are the primary tool for composing object types outside of `extends`.

### Adding Metadata to Entities

```typescript
type WithId = { id: string };
type WithTimestamps = { createdAt: Date; updatedAt: Date };
type WithSoftDelete = { deletedAt: Date | null };

type BaseEntity = WithId & WithTimestamps;

type User = BaseEntity & {
  name: string;
  email: string;
};

type Post = BaseEntity & WithSoftDelete & {
  title: string;
  content: string;
  authorId: string;
};

const user: User = {
  id: "u1",
  createdAt: new Date(),
  updatedAt: new Date(),
  name: "Alice",
  email: "alice@example.com",
};
```

### Augmenting Function Types

```typescript
type Logger = (message: string) => void;
type WithLevel = { level: "info" | "warn" | "error" };

type LeveledLogger = Logger & WithLevel;

const logger: LeveledLogger = Object.assign(
  (message: string) => console.log(`[${logger.level}] ${message}`),
  { level: "info" as const },
);

logger("Hello");     // callable
logger.level;        // has properties
```

## Union and Intersection Duality

Unions and intersections have an inverse relationship when applied to object types. This is a common source of confusion.

### Values vs Properties

```typescript
type A = { x: number; y: number };
type B = { y: number; z: number };

// Union: value is A OR B
type AorB = A | B;
// Accessible properties: only y (common to both)
// More values are accepted (either shape works)

// Intersection: value is A AND B
type AandB = A & B;
// Accessible properties: x, y, and z (all from both)
// Fewer values are accepted (must have everything)
```

The intuition:
- **Union** = more possible values, fewer guaranteed properties
- **Intersection** = fewer possible values, more guaranteed properties

### The Set Theory View

Think of types as sets of values:

| Operation | Values | Properties |
|-----------|--------|------------|
| `A \| B` (union) | Values in A **or** B (larger set) | Properties in A **and** B (smaller set) |
| `A & B` (intersection) | Values in A **and** B (smaller set) | Properties in A **or** B (larger set) |

This is why:
- `keyof (A | B)` produces the **intersection** of keys
- `keyof (A & B)` produces the **union** of keys

```typescript
type A = { x: number; y: number };
type B = { y: number; z: number };

type UnionKeys = keyof (A | B);     // "y"
type IntersectionKeys = keyof (A & B); // "x" | "y" | "z"
```

## Unions with Functions

### Overload-Like Behavior with Unions

```typescript
type StringTransform = (input: string) => string;
type NumberTransform = (input: number) => number;

// Union of function types: must work with BOTH input types
type Transform = StringTransform | NumberTransform;

// You can only call it with arguments common to ALL signatures
// Both accept their respective types, so the safe call type is:
// (input: string & number) => string | number
// which is (input: never) => string | number
// Essentially, you cannot call a union of functions unless their parameters overlap

declare const transform: Transform;
// transform("hello"); // Error: Argument of type 'string' is not assignable to parameter of type 'never'
```

This is counterintuitive but correct. A union of function types requires arguments that satisfy ALL parameter types simultaneously, which for incompatible parameters means `never`.

### Intersection of Function Types: Overloads

```typescript
type StringTransform = (input: string) => string;
type NumberTransform = (input: number) => number;

// Intersection of function types creates overloads
type Transform = StringTransform & NumberTransform;

declare const transform: Transform;
transform("hello"); // OK, returns string
transform(42);      // OK, returns number
```

This is the opposite of unions. Intersection of function types means the function can handle EITHER input, acting as overloads.

## Distributive Behavior

### Conditional Types Distribute Over Unions

```typescript
type Wrap<T> = T extends any ? { value: T } : never;

type Result = Wrap<string | number>;
// { value: string } | { value: number }
// NOT { value: string | number }
```

Each member of the union is processed independently, and the results form a new union.

### Preventing Distribution

Wrap the type parameter in a tuple to prevent distribution.

```typescript
type WrapAll<T> = [T] extends [any] ? { value: T } : never;

type Result = WrapAll<string | number>;
// { value: string | number }
// The union is processed as a whole
```

### Distributing Over Mapped Types

```typescript
type PickByType<T, V> = {
  [K in keyof T as T[K] extends V ? K : never]: T[K];
};

type User = {
  id: number;
  name: string;
  email: string;
  age: number;
  isActive: boolean;
};

type StringFields = PickByType<User, string>;
// { name: string; email: string }

type NumberFields = PickByType<User, number>;
// { id: number; age: number }
```

## Common Mistakes

### Mistake 1: Accessing Properties Before Narrowing

```typescript
type Cat = { name: string; purr(): void };
type Dog = { name: string; bark(): void };
type Pet = Cat | Dog;

function play(pet: Pet) {
  // Bad: accessing purr() on a union without narrowing
  // pet.purr(); // Error

  // Good: narrow first
  if ("purr" in pet) {
    pet.purr();
  }
}
```

### Mistake 2: Silent `never` from Conflicting Intersections

```typescript
type A = { status: string };
type B = { status: number };
type C = A & B;

// No error at the type level!
// But C.status is string & number = never
// Any attempt to create a C value will fail

// const obj: C = { status: "active" }; // Error: string is not assignable to never
// const obj: C = { status: 200 };      // Error: number is not assignable to never

// This is a silent trap. With interfaces + extends, you get an immediate error.
```

### Mistake 3: Forgetting Exhaustive Checks

```typescript
type Action = { type: "add"; value: number } | { type: "remove"; id: string };

function reduce(state: number[], action: Action): number[] {
  switch (action.type) {
    case "add":
      return [...state, action.value];
    // Forgot "remove" case!
    // No error without exhaustive check
  }
  return state;
}

// Fix: add exhaustive check
function reduce(state: number[], action: Action): number[] {
  switch (action.type) {
    case "add":
      return [...state, action.value];
    case "remove":
      return state.filter((_, i) => String(i) !== action.id);
    default:
      const _exhaustive: never = action;
      return _exhaustive;
  }
}
```

### Mistake 4: Union Widening with `let`

```typescript
// const preserves the literal type
const status = "active"; // type: "active"

// let widens to string
let status2 = "active"; // type: string

// This means status2 cannot be assigned to a literal union without annotation
type Status = "active" | "inactive";
// const s: Status = status2; // Error: string is not assignable to Status

// Fix: annotate the variable
let status3: Status = "active"; // type: Status
```

### Mistake 5: Thinking Union of Functions Is Callable with Either Argument

```typescript
type Fn = ((x: string) => void) | ((x: number) => void);

declare const fn: Fn;
// fn("hello"); // Error! Cannot call with string
// fn(42);      // Error! Cannot call with number

// The parameter type becomes string & number = never
// Use intersection (&) instead to create overloads
type Fn2 = ((x: string) => void) & ((x: number) => void);
declare const fn2: Fn2;
fn2("hello"); // OK
fn2(42);      // OK
```

### Mistake 6: Excess Properties Leak Through Union Members

```typescript
type A = { kind: "a"; value: string };
type B = { kind: "b"; count: number };
type AB = A | B;

// TypeScript allows extra properties that exist on OTHER union members
const x: AB = { kind: "a", value: "hello", count: 42 };
// No error! count exists on B, so TypeScript allows it

// This can lead to unexpected objects. Use strictObjectChecks or
// branded types if you need truly exclusive unions.
```

## Best Practices

### 1. Prefer Discriminated Unions for Complex Object Unions

```typescript
// Bad: unstructured union
type Response = { data: string } | { error: string };
// Which is it? No clean way to check.

// Good: discriminated union
type Response =
  | { success: true; data: string }
  | { success: false; error: string };

function handle(res: Response) {
  if (res.success) {
    console.log(res.data); // TypeScript narrows to the success variant
  } else {
    console.error(res.error); // TypeScript narrows to the error variant
  }
}
```

### 2. Always Add Exhaustive Checks to Switch Statements

```typescript
function assertNever(value: never, msg?: string): never {
  throw new Error(msg ?? `Unhandled value: ${JSON.stringify(value)}`);
}

// Use in every switch over a discriminated union
switch (action.type) {
  case "add": /* ... */ break;
  case "remove": /* ... */ break;
  default: assertNever(action);
}
```

### 3. Use Intersections for Composition, Not Deep Merging

```typescript
// Good: composing flat types
type WithId = { id: string };
type WithMeta = { createdAt: Date };
type User = WithId & WithMeta & { name: string };

// Risky: intersecting types with overlapping nested properties
type Config1 = { db: { host: string; port: number } };
type Config2 = { db: { ssl: boolean } };
type MergedConfig = Config1 & Config2;
// MergedConfig.db is { host: string; port: number } & { ssl: boolean }
// This works for non-conflicting properties, but if they clash, you get never
```

### 4. Keep Union Members Consistent

```typescript
// Bad: inconsistent shapes
type Result =
  | { ok: true; data: User }
  | { error: string; code: number };

// Good: consistent discriminant and structure
type Result =
  | { ok: true; data: User; error: null }
  | { ok: false; data: null; error: { message: string; code: number } };
```

### 5. Use `Extract` and `Exclude` to Manipulate Unions

```typescript
type Status = "active" | "inactive" | "pending" | "deleted" | "archived";

type ActiveStatuses = Exclude<Status, "deleted" | "archived">;
// "active" | "inactive" | "pending"

type DangerStatuses = Extract<Status, "deleted" | "archived">;
// "deleted" | "archived"
```

## Real-World Use Cases

### Result Type (Error Handling Without Exceptions)

```typescript
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

function divide(a: number, b: number): Result<number, string> {
  if (b === 0) {
    return { ok: false, error: "Division by zero" };
  }
  return { ok: true, value: a / b };
}

function safeJsonParse<T>(json: string): Result<T> {
  try {
    return { ok: true, value: JSON.parse(json) };
  } catch (e) {
    return { ok: false, error: e instanceof Error ? e : new Error(String(e)) };
  }
}

// Usage with exhaustive narrowing
const result = divide(10, 3);
if (result.ok) {
  console.log(`Result: ${result.value.toFixed(2)}`);
} else {
  console.error(`Error: ${result.error}`);
}
```

### State Machine with Discriminated Unions

```typescript
type AuthState =
  | { status: "unauthenticated" }
  | { status: "authenticating"; provider: string }
  | { status: "authenticated"; user: User; token: string }
  | { status: "error"; message: string; retryCount: number };

type AuthAction =
  | { type: "LOGIN_START"; provider: string }
  | { type: "LOGIN_SUCCESS"; user: User; token: string }
  | { type: "LOGIN_FAILURE"; message: string }
  | { type: "LOGOUT" }
  | { type: "RETRY" };

function authReducer(state: AuthState, action: AuthAction): AuthState {
  switch (action.type) {
    case "LOGIN_START":
      return { status: "authenticating", provider: action.provider };

    case "LOGIN_SUCCESS":
      return { status: "authenticated", user: action.user, token: action.token };

    case "LOGIN_FAILURE":
      return {
        status: "error",
        message: action.message,
        retryCount: state.status === "error" ? state.retryCount + 1 : 0,
      };

    case "LOGOUT":
      return { status: "unauthenticated" };

    case "RETRY":
      if (state.status === "error") {
        return { status: "authenticating", provider: "retry" };
      }
      return state;

    default:
      return assertNever(action);
  }
}

function assertNever(value: never): never {
  throw new Error(`Unhandled action: ${JSON.stringify(value)}`);
}
```

### API Layer with Intersection Composition

```typescript
type WithPagination = {
  page: number;
  perPage: number;
  total: number;
  totalPages: number;
};

type WithSorting = {
  sortBy: string;
  sortOrder: "asc" | "desc";
};

type WithFilters<T> = {
  filters: Partial<T>;
  activeFilterCount: number;
};

type ApiListResponse<T> = {
  items: T[];
} & WithPagination;

type AdvancedListResponse<T> = ApiListResponse<T> & WithSorting & WithFilters<T>;

type User = { id: string; name: string; role: "admin" | "user" };

async function fetchUsers(
  page: number,
  sortBy: string,
): Promise<AdvancedListResponse<User>> {
  const response = await fetch(`/api/users?page=${page}&sort=${sortBy}`);
  return response.json();
}

const result = await fetchUsers(1, "name");
// result has: items, page, perPage, total, totalPages, sortBy, sortOrder, filters, activeFilterCount
console.log(`Page ${result.page} of ${result.totalPages}`);
console.log(`Sorted by ${result.sortBy} (${result.sortOrder})`);
console.log(`${result.activeFilterCount} active filters`);
result.items.forEach((user) => console.log(user.name));
```

---

> **Key Takeaway:** Use union types (`|`) when a value can be one of several types, and always narrow before accessing type-specific properties. Use intersection types (`&`) to compose multiple types into one that has all their properties. Discriminated unions with exhaustive checks are the single most important pattern for type-safe branching logic in TypeScript.
