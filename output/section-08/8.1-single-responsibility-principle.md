# 8.1 Single Responsibility Principle (SRP)

> A module should be responsible to one, and only one, actor. SRP is not about doing "one thing" but about having one reason to change, driven by one group of stakeholders.

---

## The Principle, Correctly Understood

The Single Responsibility Principle is the "S" in SOLID. It was coined by Robert C. Martin and is one of the most cited, and most misunderstood, principles in software design.

### The Common (Wrong) Interpretation

> "A class should do only one thing."

This interpretation leads to absurd decomposition: a class that adds numbers, a class that subtracts numbers, a class that formats the result. That is not what SRP means.

### The Correct Definition

Robert Martin clarified SRP over the years. The most precise formulation, from *Clean Architecture* (2017), is:

> "A module should be responsible to one, and only one, actor."

An **actor** is a group of stakeholders or users who want the system to change in a particular way. The principle says: gather together the things that change for the same reasons (same actor), and separate the things that change for different reasons (different actors).

### The Classic Employee Example

```typescript
// Bad Practice: three different actors in one class
class Employee {
  constructor(
    public name: string,
    public hoursWorked: number,
    public hourlyRate: number
  ) {}

  // The CFO cares about this (finance/payroll)
  calculatePay(): number {
    return this.hoursWorked * this.hourlyRate;
  }

  // The COO cares about this (operations/reporting)
  reportHours(): string {
    return `${this.name} worked ${this.hoursWorked} hours`;
  }

  // The CTO cares about this (technology/persistence)
  save(): void {
    // Save to database
  }
}
```

This class has **three reasons to change**:

1. The CFO requests a change to how pay is calculated (overtime rules, bonuses).
2. The COO requests a change to hour reporting format (new compliance regulation).
3. The CTO requests a change to the persistence layer (migrate from SQL to NoSQL).

A change for one actor risks breaking functionality another actor depends on. SRP says: separate these into distinct modules.

```typescript
// Good: each class serves one actor

class PayCalculator {
  calculatePay(hoursWorked: number, hourlyRate: number): number {
    return hoursWorked * hourlyRate;
  }

  calculateOvertime(hoursWorked: number, hourlyRate: number): number {
    const regularHours = Math.min(hoursWorked, 40);
    const overtimeHours = Math.max(hoursWorked - 40, 0);
    return (regularHours * hourlyRate) + (overtimeHours * hourlyRate * 1.5);
  }
}

class HourReporter {
  generateReport(name: string, hoursWorked: number): string {
    return `${name} worked ${hoursWorked} hours`;
  }

  generateWeeklyReport(name: string, dailyHours: number[]): string {
    const total = dailyHours.reduce((sum, h) => sum + h, 0);
    return `${name}: ${total} hours across ${dailyHours.length} days`;
  }
}

class EmployeeRepository {
  save(employee: { name: string; hoursWorked: number; hourlyRate: number }): void {
    // Persist to database
  }

  findByName(name: string): Employee | null {
    // Query database
    return null;
  }
}
```

Now, a change to overtime calculation only touches `PayCalculator`. A database migration only touches `EmployeeRepository`. The blast radius of every change is contained.

## What "Reason to Change" Really Means

A "reason to change" is **not** a bug fix. It is **not** a refactoring. Those are developer activities that happen for technical reasons. A "reason to change" is a **business requirement** from a specific actor:

| Actor | Their Concern | What They Might Request |
|-------|--------------|------------------------|
| Finance team | Revenue calculations | Change tax formula |
| Operations team | Reporting compliance | Add new report columns |
| Infrastructure team | Data storage | Migrate database engine |
| Security team | Access control | Add audit logging |
| UX team | Display logic | Change date formatting |

If two of these concerns live in the same module, a request from one actor can accidentally break the other's functionality. SRP prevents this coupling.

## SRP Beyond Classes: Functions, Modules, and Files

SRP applies at every level of abstraction, not just classes.

### Functions

A function should do one thing at one level of abstraction:

```typescript
// Bad Practice: one function doing parsing, validation, and persistence
function processUserInput(raw: string): void {
  // Parse
  const data = JSON.parse(raw);

  // Validate
  if (!data.name || data.name.length < 2) {
    throw new Error("Invalid name");
  }
  if (!data.email || !data.email.includes("@")) {
    throw new Error("Invalid email");
  }

  // Persist
  const db = connectToDatabase();
  db.collection("users").insertOne({
    name: data.name,
    email: data.email,
    createdAt: new Date(),
  });

  // Notify
  sendWelcomeEmail(data.email, data.name);
}
```

```typescript
// Good: each function has one job at one level of abstraction
function processUserInput(raw: string): void {
  const data = parseUserInput(raw);
  validateUser(data);
  saveUser(data);
  notifyUser(data);
}

function parseUserInput(raw: string): UserInput {
  return JSON.parse(raw) as UserInput;
}

function validateUser(data: UserInput): void {
  if (!data.name || data.name.length < 2) {
    throw new Error("Invalid name");
  }
  if (!data.email || !data.email.includes("@")) {
    throw new Error("Invalid email");
  }
}

function saveUser(data: UserInput): void {
  const db = connectToDatabase();
  db.collection("users").insertOne({
    name: data.name,
    email: data.email,
    createdAt: new Date(),
  });
}

function notifyUser(data: UserInput): void {
  sendWelcomeEmail(data.email, data.name);
}
```

### Modules and Files

In a TypeScript project, each file or module should group related functionality for one actor:

```
// Bad: one mega-file
src/
  userService.ts  (600 lines: auth + CRUD + emails + reports)

// Good: separated by responsibility
src/
  user/
    userRepository.ts     (persistence)
    userValidator.ts       (validation rules)
    userNotifier.ts        (email notifications)
    userReportGenerator.ts (reporting)
    userService.ts         (orchestration)
```

## Identifying Responsibilities

### The "Describe It" Test

Try to describe what a class does in one sentence **without using the words "and" or "or"**:

- "This class manages user authentication." (Single responsibility)
- "This class manages user authentication **and** sends emails **and** generates reports." (Multiple responsibilities)

### The "Who Cares?" Test

For each method, ask: "Which actor (person, team, department) would request a change to this?" If different methods are owned by different actors, the class has multiple responsibilities.

### Code Smells That Signal SRP Violations

| Smell | What It Suggests |
|-------|-----------------|
| Class has 10+ methods | Likely serving multiple actors |
| Constructor has 5+ dependencies | Coordinating too many concerns |
| Hard to name the class (ends in "Manager", "Handler", "Processor", "Helper") | Unclear or multiple responsibilities |
| A change in one method requires changes in unrelated methods | Coupled responsibilities |
| Test file is 500+ lines | Testing too many behaviors in one unit |
| Merge conflicts in the same file from different features | Different actors touching the same module |

## SRP in Practice: Layered Architecture

The most common application of SRP in real projects is layering. Each layer has a single responsibility:

```typescript
// Data layer: responsible for persistence
class OrderRepository {
  async findById(id: string): Promise<Order | null> {
    // Database query
    return null;
  }

  async save(order: Order): Promise<void> {
    // Database insert/update
  }

  async findByCustomer(customerId: string): Promise<Order[]> {
    // Database query
    return [];
  }
}

// Business logic layer: responsible for domain rules
class OrderService {
  constructor(
    private readonly repository: OrderRepository,
    private readonly pricingEngine: PricingEngine,
    private readonly inventoryChecker: InventoryChecker
  ) {}

  async placeOrder(items: OrderItem[], customerId: string): Promise<Order> {
    // Validate inventory
    for (const item of items) {
      const available = await this.inventoryChecker.isAvailable(
        item.productId,
        item.quantity
      );
      if (!available) {
        throw new Error(`Product ${item.productId} is out of stock`);
      }
    }

    // Calculate pricing
    const total = this.pricingEngine.calculateTotal(items);

    // Create and save order
    const order: Order = {
      id: generateId(),
      customerId,
      items,
      total,
      status: "pending",
      createdAt: new Date(),
    };

    await this.repository.save(order);
    return order;
  }
}

// Presentation layer: responsible for HTTP concerns
class OrderController {
  constructor(private readonly orderService: OrderService) {}

  async handleCreateOrder(req: Request, res: Response): Promise<void> {
    try {
      const { items, customerId } = req.body;
      const order = await this.orderService.placeOrder(items, customerId);
      res.status(201).json(order);
    } catch (error) {
      if (error instanceof Error) {
        res.status(400).json({ error: error.message });
      }
    }
  }
}
```

Each layer changes for different reasons:
- `OrderRepository` changes when the database schema or technology changes.
- `OrderService` changes when business rules change (pricing, validation, workflow).
- `OrderController` changes when the API contract changes (endpoints, response format).

## The Over-Separation Trap

SRP does not mean "every function gets its own class." Over-applying the principle creates its own problems:

### Class Explosion

```typescript
// Over-separated: too granular, too many tiny classes
class NameValidator {
  validate(name: string): boolean {
    return name.length >= 2;
  }
}

class EmailValidator {
  validate(email: string): boolean {
    return email.includes("@");
  }
}

class AgeValidator {
  validate(age: number): boolean {
    return age >= 0 && age <= 150;
  }
}

// This is 3 trivial classes that could be one
```

```typescript
// Better: group related validations that change for the same reason
class UserValidator {
  validateName(name: string): boolean {
    return name.length >= 2;
  }

  validateEmail(email: string): boolean {
    return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
  }

  validateAge(age: number): boolean {
    return age >= 0 && age <= 150;
  }

  validate(user: { name: string; email: string; age: number }): string[] {
    const errors: string[] = [];
    if (!this.validateName(user.name)) errors.push("Invalid name");
    if (!this.validateEmail(user.email)) errors.push("Invalid email");
    if (!this.validateAge(user.age)) errors.push("Invalid age");
    return errors;
  }
}
```

All user validation rules are owned by the same actor (the product team that defines user requirements). They change together. Splitting them into separate classes adds indirection without benefit.

### The Cohesion Principle

SRP is closely related to **cohesion**: the degree to which elements within a module belong together. High cohesion means all parts of a module are related and work toward the same purpose.

**Signs of good cohesion:**
- Methods use most of the class's fields
- Methods are called together in common workflows
- A change to one method often requires a change to another in the same class

**Signs of low cohesion (SRP violation):**
- Groups of methods that never interact with each other
- Fields that are only used by a subset of methods
- Methods that could be moved to another class without any change to the remaining methods

## SRP Applied to Functions

Functions deserve the same scrutiny as classes. The key rule: **a function should operate at one level of abstraction**.

```typescript
// Bad Practice: mixing levels of abstraction
async function registerUser(name: string, email: string, password: string): Promise<void> {
  // Low-level: hashing
  const salt = crypto.randomBytes(16).toString("hex");
  const hash = crypto.pbkdf2Sync(password, salt, 10000, 64, "sha512").toString("hex");

  // Mid-level: building object
  const user = {
    id: crypto.randomUUID(),
    name,
    email,
    passwordHash: `${salt}:${hash}`,
    createdAt: new Date(),
  };

  // Low-level: SQL
  await pool.query(
    "INSERT INTO users (id, name, email, password_hash, created_at) VALUES ($1, $2, $3, $4, $5)",
    [user.id, user.name, user.email, user.passwordHash, user.createdAt]
  );

  // Low-level: SMTP
  const transporter = nodemailer.createTransport({ /* config */ });
  await transporter.sendMail({
    to: email,
    subject: "Welcome!",
    html: `<h1>Welcome, ${name}!</h1>`,
  });
}
```

```typescript
// Good: each function at one level of abstraction
async function registerUser(name: string, email: string, password: string): Promise<void> {
  const passwordHash = await hashPassword(password);
  const user = createUser(name, email, passwordHash);
  await saveUser(user);
  await sendWelcomeEmail(user);
}

// Each of these is a focused, testable unit
async function hashPassword(password: string): Promise<string> {
  const salt = crypto.randomBytes(16).toString("hex");
  const hash = crypto.pbkdf2Sync(password, salt, 10000, 64, "sha512").toString("hex");
  return `${salt}:${hash}`;
}

function createUser(name: string, email: string, passwordHash: string): User {
  return {
    id: crypto.randomUUID(),
    name,
    email,
    passwordHash,
    createdAt: new Date(),
  };
}

async function saveUser(user: User): Promise<void> {
  await pool.query(
    "INSERT INTO users (id, name, email, password_hash, created_at) VALUES ($1, $2, $3, $4, $5)",
    [user.id, user.name, user.email, user.passwordHash, user.createdAt]
  );
}

async function sendWelcomeEmail(user: User): Promise<void> {
  await emailService.send({
    to: user.email,
    subject: "Welcome!",
    html: `<h1>Welcome, ${user.name}!</h1>`,
  });
}
```

The top-level function reads like a recipe. Each step is a single responsibility that can be tested, replaced, or modified independently.

## Common Mistakes

### 1. Interpreting SRP as "One Method Per Class"

SRP is about having one **reason to change** (one actor), not one method. A class with ten methods that all serve the same actor is perfectly fine.

### 2. Naming Classes "Manager", "Handler", or "Helper"

These names are a code smell because they describe a vague role, not a responsibility. If you cannot name a class after its single responsibility, it probably has more than one:

```typescript
// Bad: vague name, unclear responsibility
class UserManager {
  authenticate(email: string, password: string): boolean { /* ... */ }
  updateProfile(userId: string, data: ProfileData): void { /* ... */ }
  generateReport(userId: string): Report { /* ... */ }
  sendNotification(userId: string, message: string): void { /* ... */ }
}

// Good: specific names reflecting specific responsibilities
class Authenticator { /* ... */ }
class ProfileEditor { /* ... */ }
class UserReportGenerator { /* ... */ }
class NotificationSender { /* ... */ }
```

### 3. Separating by Technical Layer Instead of by Actor

Grouping all validators in one file, all formatters in another, and all API calls in a third is separation by **technical concern**, not by **actor**. This can be worse than no separation at all:

```typescript
// Bad: organized by technical type
src/
  validators/
    userValidator.ts
    orderValidator.ts
    productValidator.ts
  formatters/
    userFormatter.ts
    orderFormatter.ts
    productFormatter.ts
  repositories/
    userRepository.ts
    orderRepository.ts
    productRepository.ts
```

```
// Better: organized by domain/actor
src/
  user/
    userValidator.ts
    userFormatter.ts
    userRepository.ts
    userService.ts
  order/
    orderValidator.ts
    orderFormatter.ts
    orderRepository.ts
    orderService.ts
```

When you need to change "user" functionality, everything is in one folder. When you need to change "order" functionality, you go to a different folder. Each domain module serves a specific set of actors.

### 4. Ignoring SRP at the Module/Package Level

SRP does not stop at classes. A package or library that bundles unrelated features has a module-level SRP violation. Users who only need feature A are forced to install and update code for features B, C, and D.

### 5. Confusing SRP with DRY

Do not merge two classes just because they share similar code. If they serve different actors, they should remain separate, even if some code is duplicated:

```typescript
// These serve different actors. The "calculate" logic might look similar today,
// but will diverge as each actor's requirements evolve.

class PayrollCalculator {
  // Finance team controls this
  calculateGrossPay(hours: number, rate: number): number {
    return hours * rate;
  }
}

class BillingCalculator {
  // Sales team controls this
  calculateClientBilling(hours: number, rate: number): number {
    return hours * rate; // Same formula TODAY, but different owner
  }
}
```

If you merge these, a change for the sales team's billing formula risks breaking the finance team's payroll. The duplication is intentional.

## Best Practices

### 1. Ask "Who Requests Changes?" Before Structuring Code

Before creating a class or module, identify the actor. If you cannot name a single person or team who would request changes to this code, it might be too broad.

### 2. Start With Larger Modules, Split When You Feel Pain

Do not pre-emptively split everything into micro-classes. Start with a reasonable grouping. When a module starts accumulating unrelated changes (different PRs touching different methods for different reasons), that is the signal to split.

### 3. Use the Facade Pattern for Orchestration

When multiple single-responsibility classes need to work together, create a thin orchestrator that delegates:

```typescript
class UserRegistrationFacade {
  constructor(
    private readonly validator: UserValidator,
    private readonly hasher: PasswordHasher,
    private readonly repository: UserRepository,
    private readonly notifier: UserNotifier
  ) {}

  async register(input: RegistrationInput): Promise<User> {
    this.validator.validate(input);
    const hash = await this.hasher.hash(input.password);
    const user = await this.repository.create({
      name: input.name,
      email: input.email,
      passwordHash: hash,
    });
    await this.notifier.sendWelcome(user);
    return user;
  }
}
```

The facade itself has a single responsibility: orchestrating the registration workflow. It does not contain validation logic, hashing logic, or persistence logic.

### 4. Write Tests as a Litmus Test

If your test file requires extensive mocking of unrelated dependencies, the class likely has too many responsibilities. A well-focused class needs few mocks and its tests are straightforward.

### 5. Align Code Structure with Team Structure (Conway's Law)

If your organization has separate teams for payments, inventory, and shipping, your code modules should reflect that boundary. This is not a coincidence: Conway's Law states that systems mirror the communication structure of the organizations that build them. SRP formalizes this at the code level.

## Real-World Use Cases

### 1. Notification System

```typescript
// Each notification channel is its own responsibility
interface NotificationChannel {
  send(recipient: string, message: string): Promise<void>;
}

class EmailChannel implements NotificationChannel {
  constructor(private readonly transporter: EmailTransporter) {}

  async send(recipient: string, message: string): Promise<void> {
    await this.transporter.sendMail({
      to: recipient,
      subject: "Notification",
      text: message,
    });
  }
}

class SmsChannel implements NotificationChannel {
  constructor(private readonly smsClient: SmsClient) {}

  async send(recipient: string, message: string): Promise<void> {
    await this.smsClient.sendMessage({ to: recipient, body: message });
  }
}

class PushChannel implements NotificationChannel {
  constructor(private readonly pushService: PushService) {}

  async send(recipient: string, message: string): Promise<void> {
    await this.pushService.notify({ deviceToken: recipient, body: message });
  }
}

// Orchestrator delegates to the correct channel
class NotificationService {
  constructor(private readonly channels: Map<string, NotificationChannel>) {}

  async notify(channel: string, recipient: string, message: string): Promise<void> {
    const handler = this.channels.get(channel);
    if (!handler) {
      throw new Error(`Unknown notification channel: ${channel}`);
    }
    await handler.send(recipient, message);
  }
}
```

Each channel changes independently: the email team can change SMTP settings without touching SMS logic. Adding a new channel (Slack, webhook) means adding a new class, not modifying existing ones.

### 2. File Processing Pipeline

```typescript
// Each step in the pipeline has one responsibility

class FileReader {
  read(path: string): string {
    return fs.readFileSync(path, "utf-8");
  }
}

class CsvParser {
  parse(raw: string): string[][] {
    return raw
      .split("\n")
      .filter(line => line.trim().length > 0)
      .map(line => line.split(",").map(cell => cell.trim()));
  }
}

class DataValidator {
  validate(rows: string[][]): ValidatedRow[] {
    return rows.map((row, index) => {
      if (row.length < 3) {
        throw new Error(`Row ${index} has insufficient columns`);
      }
      return { name: row[0], email: row[1], role: row[2] };
    });
  }
}

class DataImporter {
  constructor(private readonly repository: UserRepository) {}

  async import(users: ValidatedRow[]): Promise<number> {
    let count = 0;
    for (const user of users) {
      await this.repository.upsert(user);
      count++;
    }
    return count;
  }
}

// Orchestrator composes the pipeline
class ImportPipeline {
  constructor(
    private readonly reader: FileReader,
    private readonly parser: CsvParser,
    private readonly validator: DataValidator,
    private readonly importer: DataImporter
  ) {}

  async run(filePath: string): Promise<number> {
    const raw = this.reader.read(filePath);
    const rows = this.parser.parse(raw);
    const validated = this.validator.validate(rows);
    return this.importer.import(validated);
  }
}
```

Each stage can be tested in isolation, reused in other pipelines, or replaced without affecting the others.

### 3. Logging Service with Separated Concerns

```typescript
// Formatting is one responsibility
interface LogFormatter {
  format(level: string, message: string, meta?: Record<string, unknown>): string;
}

class JsonFormatter implements LogFormatter {
  format(level: string, message: string, meta?: Record<string, unknown>): string {
    return JSON.stringify({ level, message, timestamp: new Date().toISOString(), ...meta });
  }
}

class TextFormatter implements LogFormatter {
  format(level: string, message: string, meta?: Record<string, unknown>): string {
    const ts = new Date().toISOString();
    const metaStr = meta ? ` ${JSON.stringify(meta)}` : "";
    return `[${ts}] ${level.toUpperCase()}: ${message}${metaStr}`;
  }
}

// Transport is another responsibility
interface LogTransport {
  write(output: string): void;
}

class ConsoleTransport implements LogTransport {
  write(output: string): void {
    console.log(output);
  }
}

class FileTransport implements LogTransport {
  constructor(private readonly path: string) {}

  write(output: string): void {
    fs.appendFileSync(this.path, output + "\n");
  }
}

// The logger orchestrates formatting and transport
class Logger {
  constructor(
    private readonly formatter: LogFormatter,
    private readonly transports: LogTransport[]
  ) {}

  log(level: string, message: string, meta?: Record<string, unknown>): void {
    const output = this.formatter.format(level, message, meta);
    for (const transport of this.transports) {
      transport.write(output);
    }
  }

  info(message: string, meta?: Record<string, unknown>): void {
    this.log("info", message, meta);
  }

  error(message: string, meta?: Record<string, unknown>): void {
    this.log("error", message, meta);
  }
}
```

The operations team can request a new log format (JSON to logfmt) without touching transport code. The infrastructure team can add a new transport (Elasticsearch, CloudWatch) without touching formatting code.

---

> **Key Takeaway:** SRP is about aligning code modules with organizational actors so that a change requested by one stakeholder never risks breaking functionality owned by another; it is not about making everything as small as possible.
