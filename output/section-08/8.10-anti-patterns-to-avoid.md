# 8.10 Anti-Patterns to Avoid

> Anti-patterns are recurring solutions that seem reasonable but create more problems than they solve -- knowing them by name lets you recognize and eliminate them before they accumulate.

---

## What Is an Anti-Pattern?

An anti-pattern is a common response to a recurring problem that initially looks like a reasonable solution but ultimately makes things worse. It is the counterpart to a design pattern.

Anti-patterns in software share three traits:

1. They appear frequently in the wild
2. They appear to be a valid solution at first glance
3. They create hidden costs: poor maintainability, low testability, fragility under change

This guide covers the anti-patterns most commonly encountered in JavaScript and TypeScript codebases, organized by category.

---

## Architecture Anti-Patterns

### God Object / God Class

A class or object that knows too much or does too much. It accumulates unrelated responsibilities over time because it is convenient to add new behavior to an already-imported class.

```typescript
// Bad: UserManager does everything related to users
class UserManager {
  createUser(data: CreateUserData): User { /* ... */ }
  deleteUser(id: string): void { /* ... */ }
  updateProfile(id: string, data: Partial<User>): User { /* ... */ }
  sendWelcomeEmail(user: User): void { /* email logic */ }
  sendPasswordResetEmail(user: User): void { /* email logic */ }
  hashPassword(plain: string): string { /* crypto logic */ }
  validatePassword(plain: string, hash: string): boolean { /* crypto logic */ }
  generateToken(userId: string): string { /* JWT logic */ }
  verifyToken(token: string): JwtPayload { /* JWT logic */ }
  checkPermission(userId: string, resource: string): boolean { /* auth logic */ }
  logActivity(userId: string, action: string): void { /* audit logic */ }
  uploadAvatar(userId: string, file: Buffer): string { /* file logic */ }
  generateReport(userId: string): UserReport { /* reporting logic */ }
}
```

The symptom: if you change a JWT library, you touch `UserManager`. If you change an email provider, you touch `UserManager`. Every team's changes conflict.

```typescript
// Good: responsibilities split by actor/concern
class UserService {
  constructor(
    private repo: UserRepository,
    private auth: AuthService,
    private notifications: NotificationService,
  ) {}

  createUser(data: CreateUserData): User { /* ... */ }
  deleteUser(id: string): void { /* ... */ }
  updateProfile(id: string, data: Partial<User>): User { /* ... */ }
}

class AuthService {
  hashPassword(plain: string): string { /* ... */ }
  validatePassword(plain: string, hash: string): boolean { /* ... */ }
  generateToken(userId: string): string { /* ... */ }
  verifyToken(token: string): JwtPayload { /* ... */ }
  checkPermission(userId: string, resource: string): boolean { /* ... */ }
}

class NotificationService {
  sendWelcomeEmail(user: User): void { /* ... */ }
  sendPasswordResetEmail(user: User): void { /* ... */ }
}
```

### Shotgun Surgery

When a single logical change requires edits in many unrelated files. This is the opposite of the God Object: instead of everything being in one place, related code is scattered across many places.

```typescript
// Bad: the HTTP status codes are scattered
// user-controller.ts
res.status(404).json({ error: 'Not found' });

// product-controller.ts
res.status(404).json({ error: 'Not Found' });  // different casing

// order-controller.ts
res.status(404).send('not found');  // different format

// Changing the 404 response shape requires touching every controller
```

```typescript
// Good: centralized
const HttpStatus = {
  NOT_FOUND: 404,
  UNAUTHORIZED: 401,
  FORBIDDEN: 403,
  OK: 200,
} as const;

function notFound(res: Response, message = 'Not found'): void {
  res.status(HttpStatus.NOT_FOUND).json({ error: message });
}

// Controllers call the helper - one change updates everywhere
notFound(res, 'User not found');
```

### The Lava Layer

Layers of old code that nobody dares to remove because nobody understands what it does or whether anything still uses it. New features are built on top of old code that contradicts them.

Signs of lava layers:
- Comments like `// do not touch this`
- Functions called `doThingNew` and `doThingNew2` alongside the original `doThing`
- Dead branches that are never reached but nobody removes
- Configuration files with thousands of lines where most settings are unknown

The fix is aggressive deletion combined with version control confidence. If the code is covered by tests and the tests pass after deletion, delete it.

## Code-Level Anti-Patterns

### Magic Numbers and Magic Strings

Literal values that appear in code without explanation of what they represent or where they come from.

```typescript
// Bad: what do these numbers mean?
function calculateShipping(weight: number, zone: number): number {
  if (weight > 32) return 45.99;        // 32? 45.99?
  if (zone === 3) return weight * 1.25;  // zone 3? 1.25?
  return weight * 0.75;
}

// Bad: magic string
if (user.role === 'admin_super_level_2') { /* ... */ }
```

```typescript
// Good: named constants make intent clear
const MAX_STANDARD_WEIGHT_LBS = 32;
const OVERSEAS_ZONE = 3;
const HEAVY_ITEM_FLAT_RATE = 45.99;
const OVERSEAS_RATE_MULTIPLIER = 1.25;
const DOMESTIC_RATE_MULTIPLIER = 0.75;

const UserRole = {
  SUPER_ADMIN: 'admin_super_level_2',
  ADMIN: 'admin',
  USER: 'user',
} as const;

function calculateShipping(weight: number, zone: number): number {
  if (weight > MAX_STANDARD_WEIGHT_LBS) return HEAVY_ITEM_FLAT_RATE;
  if (zone === OVERSEAS_ZONE) return weight * OVERSEAS_RATE_MULTIPLIER;
  return weight * DOMESTIC_RATE_MULTIPLIER;
}

if (user.role === UserRole.SUPER_ADMIN) { /* ... */ }
```

### Primitive Obsession

Using primitive types (strings, numbers, booleans) to represent domain concepts that deserve their own type with validation and behavior.

```typescript
// Bad: email is just a string - no validation, no type safety
function sendEmail(to: string, subject: string, body: string): void { /* ... */ }

// Can accidentally pass arguments in wrong order:
sendEmail(subject, email, body); // TypeScript can't catch this

// Bad: user ID is just a string - what format? UUID? integer? prefix?
function getUser(id: string): User { /* ... */ }
// Can pass any string - order ID, product ID, etc.
function getOrder(id: string): Order { /* ... */ }

const userId = 'order_123';
getUser(userId); // TypeScript silently accepts a wrong ID type
```

```typescript
// Good: branded types give primitives a distinct identity
type Brand<T, B extends string> = T & { readonly __brand: B };
type UserId = Brand<string, 'UserId'>;
type OrderId = Brand<string, 'OrderId'>;
type Email = Brand<string, 'Email'>;

function createUserId(raw: string): UserId {
  if (!raw.startsWith('usr_')) throw new Error('Invalid user ID format');
  return raw as UserId;
}

function createEmail(raw: string): Email {
  if (!/^[^@]+@[^@]+\.[^@]+$/.test(raw)) throw new Error('Invalid email');
  return raw.toLowerCase() as Email;
}

function getUser(id: UserId): User { /* ... */ }
function getOrder(id: OrderId): Order { /* ... */ }
function sendEmail(to: Email, subject: string, body: string): void { /* ... */ }

const userId = createUserId('usr_123');
const orderId = createOrderId('order_456');

getUser(orderId); // TypeScript error: OrderId is not assignable to UserId
```

### Boolean Trap

Using boolean parameters to control behavior inside a function, creating ambiguous call sites where true/false has no clear meaning.

```typescript
// Bad: what does 'true' mean at the call site?
function updateUser(id: string, data: Partial<User>, notify: boolean): void {
  // ...
}

updateUser('usr_123', { name: 'Alice' }, true);   // notify? what else could it mean?
updateUser('usr_123', { name: 'Alice' }, false);  // no notify?

// Worse: multiple booleans
function renderList(
  items: Item[],
  showHeader: boolean,
  showFooter: boolean,
  reversed: boolean,
): string {
  // ...
}
renderList(items, true, false, true); // ??? cannot tell at the call site
```

```typescript
// Good: option object with named properties
interface UpdateUserOptions {
  notify?: boolean;
}

function updateUser(id: string, data: Partial<User>, options: UpdateUserOptions = {}): void {
  const { notify = true } = options;
  // ...
}

updateUser('usr_123', { name: 'Alice' }, { notify: false }); // self-documenting

// Good: separate functions when behavior truly differs
interface RenderOptions {
  showHeader?: boolean;
  showFooter?: boolean;
  reversed?: boolean;
}

function renderList(items: Item[], options: RenderOptions = {}): string {
  // ...
}
renderList(items, { reversed: true, showHeader: true });
```

### Long Parameter List

Functions with more than 3-4 parameters are hard to call correctly, hard to read, and often indicate that the function is doing too much.

```typescript
// Bad: easy to transpose arguments, hard to remember order
function createUser(
  name: string,
  email: string,
  password: string,
  role: string,
  departmentId: string,
  managerId: string | null,
  sendWelcomeEmail: boolean,
  locale: string,
): User { /* ... */ }

// Hard to tell if this is correct:
createUser('Alice', 'alice@co.com', 'secret', 'admin', 'dept_1', null, true, 'en-US');
```

```typescript
// Good: use a typed object parameter
interface CreateUserInput {
  name: string;
  email: string;
  password: string;
  role: 'admin' | 'user' | 'guest';
  departmentId: string;
  managerId?: string;
  sendWelcomeEmail?: boolean;
  locale?: string;
}

function createUser(input: CreateUserInput): User { /* ... */ }

// Self-documenting, order-independent
createUser({
  name: 'Alice',
  email: 'alice@co.com',
  password: 'secret',
  role: 'admin',
  departmentId: 'dept_1',
  locale: 'en-US',
});
```

### Feature Envy

A method that uses data and behavior from another class more than its own. It is a sign that the method is in the wrong class.

```typescript
// Bad: OrderItem method envies Order's data
class OrderItem {
  constructor(
    public productId: string,
    public quantity: number,
    public unitPrice: number,
    public order: Order,
  ) {}

  getLineTotal(): number {
    return this.unitPrice * this.quantity;
  }

  // Feature envy: this method uses Order's data, not its own
  getOrderSummary(): string {
    return `Order ${this.order.id} placed on ${this.order.createdAt} ` +
      `by ${this.order.customer.name} for ${this.order.items.length} items`;
  }
}
```

```typescript
// Good: move the method to the class it envies
class Order {
  constructor(
    public id: string,
    public createdAt: Date,
    public customer: Customer,
    public items: OrderItem[],
  ) {}

  getSummary(): string {
    return `Order ${this.id} placed on ${this.createdAt} ` +
      `by ${this.customer.name} for ${this.items.length} items`;
  }
}

class OrderItem {
  constructor(
    public productId: string,
    public quantity: number,
    public unitPrice: number,
  ) {}

  getLineTotal(): number {
    return this.unitPrice * this.quantity;
  }
}
```

## TypeScript-Specific Anti-Patterns

### The `any` Escape Hatch

Using `any` to silence the TypeScript compiler is the most common TypeScript anti-pattern. It disables type checking for that variable and any values derived from it, spreading type unsafety silently.

```typescript
// Bad: any defeats the purpose of TypeScript
async function fetchUser(id: string): Promise<any> {
  const response = await fetch(`/api/users/${id}`);
  return response.json(); // returns any
}

const user = await fetchUser('123');
user.nme; // typo - TypeScript does not catch it because user is 'any'
user.role.toUpperCase(); // might throw at runtime if role is undefined
```

```typescript
// Good: validate at the boundary, trust inside
import { z } from 'zod';

const UserSchema = z.object({
  id: z.string(),
  name: z.string(),
  email: z.string().email(),
  role: z.enum(['admin', 'user', 'guest']),
});

type User = z.infer<typeof UserSchema>;

async function fetchUser(id: string): Promise<User> {
  const response = await fetch(`/api/users/${id}`);
  const raw = await response.json();
  return UserSchema.parse(raw); // throws if shape is wrong
}

const user = await fetchUser('123');
user.nme;           // TypeScript error: Property 'nme' does not exist
user.role.toUpperCase(); // safe: role is known to be a string
```

### Overusing Type Assertions (`as`)

Using `as SomeType` to convince TypeScript you know better than the compiler when you actually do not.

```typescript
// Bad: asserting without verifying
const response = await fetch('/api/config');
const config = await response.json() as AppConfig; // no runtime validation

// Later code assumes config is valid:
console.log(config.apiUrl.toLowerCase()); // crashes if apiUrl is undefined

// Bad: double assertion to bypass the type system entirely
const x = someUnknownValue as unknown as User; // this always "works" and is always wrong
```

```typescript
// Good: type assertions are for compiler information, not runtime safety
// Use them only when you have more information than TypeScript does,
// AND when you are certain of the type:

// Acceptable: narrowing after a runtime check
const element = document.getElementById('my-input');
if (element instanceof HTMLInputElement) {
  // TypeScript knows element is HTMLInputElement here - no assertion needed
  console.log(element.value);
}

// Acceptable: asserting with a const enum that TypeScript cannot infer
const direction = 'north' as const; // narrows to literal type 'north'

// Acceptable: DOM casting you have verified by reading the HTML
const canvas = document.getElementById('canvas') as HTMLCanvasElement;
// You know it's a canvas because you wrote the HTML, but add a guard:
if (!(canvas instanceof HTMLCanvasElement)) throw new Error('Canvas not found');
```

### Widening Types Unnecessarily

Declaring variables with wider types than needed, losing the specificity TypeScript inferred.

```typescript
// Bad: explicit type annotation loses the literal
const status: string = 'active'; // TypeScript infers string, not 'active'
// Now you cannot use status as a discriminant in a union

// Bad: returning a wide type from a function that produces a narrow value
function getDirection(): string {
  return 'north'; // TypeScript cannot tell callers this is always 'north' | 'south' | ...
}

// Good: let TypeScript infer, or use const assertion
const status = 'active' as const; // type is 'active'
const directions = ['north', 'south', 'east', 'west'] as const;
// type is readonly ['north', 'south', 'east', 'west']

// Good: explicit union type
function getDirection(): 'north' | 'south' | 'east' | 'west' {
  return 'north';
}
```

### Ignoring `unknown` in Catch Blocks

TypeScript 4.0+ types caught errors as `unknown`. Code that blindly casts to `Error` is unsound.

```typescript
// Bad: assumes the caught value is always an Error
try {
  await doSomething();
} catch (error) {
  console.log(error.message); // TypeScript error in strict mode
  // error.message crashes if error is a string, number, or null
}

// Also bad: casting without verification
try {
  await doSomething();
} catch (error) {
  const e = error as Error; // wrong if error is a string rejection
  console.log(e.message);
}
```

```typescript
// Good: narrow the type
try {
  await doSomething();
} catch (error) {
  if (error instanceof Error) {
    console.log(error.message); // safe
  } else {
    console.log('Unknown error:', String(error));
  }
}

// Good: utility function
function toError(value: unknown): Error {
  if (value instanceof Error) return value;
  return new Error(String(value));
}

try {
  await doSomething();
} catch (error) {
  const err = toError(error);
  logger.error(err.message, { stack: err.stack });
}
```

## Async Anti-Patterns

### Callback Hell (Pyramid of Doom)

Deeply nested callbacks where each level depends on the result of the previous:

```javascript
// Bad: hard to follow, errors are missed, impossible to test individual steps
getUser(userId, function(err, user) {
  if (err) {
    handleError(err);
    return;
  }
  getOrders(user.id, function(err, orders) {
    if (err) {
      handleError(err);
      return;
    }
    getProducts(orders[0].id, function(err, products) {
      if (err) {
        handleError(err);
        return;
      }
      // Now 4 levels deep - this is the pyramid of doom
      renderPage(user, orders, products, function(err, html) {
        if (err) handleError(err);
        res.send(html);
      });
    });
  });
});
```

```typescript
// Good: async/await is flat and readable
async function renderUserPage(userId: string): Promise<string> {
  const user = await getUser(userId);
  const orders = await getOrders(user.id);
  const products = await getProducts(orders[0].id);
  return renderPage(user, orders, products);
}
```

### The Unhandled Promise Rejection

Forgetting to `await` a Promise or failing to `.catch()` it allows errors to disappear silently.

```typescript
// Bad: errors from saveUser are silently swallowed
function updateProfile(userId: string, data: Partial<User>): void {
  saveUser(userId, data); // Fire-and-forget: no await, no catch
  sendAnalyticsEvent('profile_updated'); // runs before save completes
}

// Bad: catch exists but does not rethrow
async function processOrder(id: string): Promise<Order> {
  try {
    return await db.findOrder(id);
  } catch (error) {
    console.log('Error:', error); // Logs but returns undefined implicitly
    // Caller expects an Order but gets undefined
  }
}
```

```typescript
// Good: always await or explicitly handle
async function updateProfile(userId: string, data: Partial<User>): Promise<void> {
  await saveUser(userId, data); // Wait for completion
  sendAnalyticsEvent('profile_updated'); // Runs after save
}

// Good: rethrow or return a meaningful fallback
async function processOrder(id: string): Promise<Order | null> {
  try {
    return await db.findOrder(id);
  } catch (error) {
    logger.error('Failed to find order', { id, error });
    return null; // Explicit fallback
  }
}
```

### Nested Promises (Promise Hell)

Using `.then()` inside `.then()`, reintroducing the nesting problem that promises were meant to solve:

```typescript
// Bad: nested .then() - same pyramid shape, different syntax
getUser(userId)
  .then(user => {
    return getOrders(user.id).then(orders => {  // nested .then inside .then
      return getProducts(orders[0].id).then(products => {
        return { user, orders, products }; // have to close all the brackets
      });
    });
  })
  .catch(handleError);

// Good: chain flat, use Promise.all for parallel
async function loadDashboard(userId: string) {
  const user = await getUser(userId);
  // These two are independent - run in parallel
  const [orders, preferences] = await Promise.all([
    getOrders(user.id),
    getPreferences(user.id),
  ]);
  const products = await getProducts(orders[0].id);
  return { user, orders, preferences, products };
}
```

### Async in forEach

`Array.prototype.forEach` does not await async callbacks. Each callback fires and the `forEach` call returns before any of them complete.

```typescript
// Bad: forEach does not wait for async operations
async function processOrders(orders: Order[]): Promise<void> {
  orders.forEach(async (order) => {
    await saveToDatabase(order); // forEach ignores the returned promise
  });
  // This line runs BEFORE any order is saved
  console.log('All orders processed'); // LIE: none are processed yet
}

// Good: use for...of to maintain sequential async execution
async function processOrders(orders: Order[]): Promise<void> {
  for (const order of orders) {
    await saveToDatabase(order); // Sequential: each waits for the previous
  }
  console.log('All orders processed'); // True
}

// Good: use Promise.all for parallel execution
async function processOrdersParallel(orders: Order[]): Promise<void> {
  await Promise.all(orders.map(order => saveToDatabase(order)));
  console.log('All orders processed'); // True
}
```

## State Management Anti-Patterns

### Mutable Shared State

Multiple parts of the application modifying the same object in place, making it impossible to trace which code is responsible for a given state.

```typescript
// Bad: shared mutable object modified from many places
const appState = {
  user: null,
  cart: [],
  isLoading: false,
};

// userModule.ts
appState.user = { id: '123', name: 'Alice' };

// cartModule.ts
appState.cart.push({ productId: 'p1', quantity: 1 });

// productPage.ts
appState.isLoading = true;
// Who resets isLoading to false? When? Where?
```

```typescript
// Good: encapsulate state with explicit update mechanisms
class AppStore {
  private state = {
    user: null as User | null,
    cart: [] as CartItem[],
    isLoading: false,
  };

  private listeners = new Set<() => void>();

  getState() { return { ...this.state }; }

  setUser(user: User | null): void {
    this.state = { ...this.state, user };
    this.notify();
  }

  addToCart(item: CartItem): void {
    this.state = { ...this.state, cart: [...this.state.cart, item] };
    this.notify();
  }

  setLoading(isLoading: boolean): void {
    this.state = { ...this.state, isLoading };
    this.notify();
  }

  subscribe(listener: () => void): () => void {
    this.listeners.add(listener);
    return () => this.listeners.delete(listener);
  }

  private notify(): void {
    this.listeners.forEach(l => l());
  }
}
```

### Prop Drilling

Passing data and callbacks through many layers of components when only deeply nested components need them.

```typescript
// Bad: cartCount is passed through components that don't use it
function Page({ user, cartCount }: PageProps) {
  return <Header user={user} cartCount={cartCount} />;
}

function Header({ user, cartCount }: HeaderProps) {
  return (
    <nav>
      <UserMenu user={user} />
      <CartIcon count={cartCount} />  {/* Finally used here */}
    </nav>
  );
}

// UserMenu also receives cartCount just to pass it down if needed later
```

The fix depends on context: module-level state stores, context/dependency injection, or lifting state to the nearest common ancestor.

## Performance Anti-Patterns

### Premature Optimization

Optimizing code before you have measured that it is a bottleneck. Premature optimization adds complexity and obscures intent for speculative gains.

```typescript
// Bad: hand-rolled caching before any evidence of performance issue
class UserService {
  private _cache: Map<string, User> = new Map();
  private _cacheTimestamps: Map<string, number> = new Map();
  private _cacheTTL = 300_000;

  async getUser(id: string): Promise<User> {
    const cached = this._cache.get(id);
    const ts = this._cacheTimestamps.get(id);
    if (cached && ts && Date.now() - ts < this._cacheTTL) {
      return cached;
    }
    const user = await this.db.findById(id);
    this._cache.set(id, user);
    this._cacheTimestamps.set(id, Date.now());
    return user;
  }
}
// This complexity is justified only after profiling shows getUser is a bottleneck
```

The rule: write clear, correct code first. Profile to find actual bottlenecks. Optimize only those bottlenecks with evidence.

### N+1 Query Problem

Loading one record then making N additional queries to load related records for each, instead of fetching the related data in bulk.

```typescript
// Bad: N+1 queries
async function getOrdersWithCustomers(): Promise<OrderWithCustomer[]> {
  const orders = await db.query('SELECT * FROM orders'); // 1 query

  const result = [];
  for (const order of orders) {
    // 1 query PER ORDER - if there are 100 orders, this runs 100 times
    const customer = await db.query(
      'SELECT * FROM customers WHERE id = ?', [order.customerId]
    );
    result.push({ ...order, customer: customer[0] });
  }
  return result; // Total: 1 + N queries
}
```

```typescript
// Good: batch fetch then stitch in memory
async function getOrdersWithCustomers(): Promise<OrderWithCustomer[]> {
  const orders = await db.query('SELECT * FROM orders'); // 1 query

  const customerIds = [...new Set(orders.map(o => o.customerId))];
  const customers = await db.query(
    `SELECT * FROM customers WHERE id IN (${customerIds.map(() => '?').join(',')})`,
    customerIds,
  ); // 1 query total

  const customerMap = new Map(customers.map(c => [c.id, c]));

  return orders.map(order => ({
    ...order,
    customer: customerMap.get(order.customerId)!,
  }));
  // Total: 2 queries regardless of how many orders exist
}
```

## Common Mistakes (Summary Table)

| Anti-Pattern | Symptom | Fix |
|-------------|---------|-----|
| God Object | Class has 30+ methods, 10+ dependencies | Split by SRP |
| Magic Numbers | `if (x > 32)` with no context | Named constants |
| Boolean Trap | `doThing(true, false, true)` | Options object |
| Primitive Obsession | `userId: string` accepts any string | Branded types |
| `any` overuse | TypeScript type errors disappear | Validate at boundaries with Zod |
| Async in forEach | `console.log` runs before async work | `for...of` or `Promise.all` |
| Unhandled promise | Errors disappear, state is inconsistent | Always `await` or `.catch()` |
| N+1 queries | 100 orders = 101 queries | Batch fetch + in-memory join |
| Shotgun surgery | 1 change = 10 file edits | Centralize the thing that changes |
| Feature envy | Method uses other class's data | Move the method to the right class |

## Best Practices

- **Name things by what they mean, not what they are.** `MAX_UPLOAD_SIZE_BYTES` is better than `maxSize`. `Email` branded type is better than `string`.

- **Treat TypeScript's type system as a tool, not an obstacle.** When you reach for `as` or `any`, pause and ask why TypeScript is unhappy. The answer usually reveals a real problem.

- **Always handle Promise rejections.** Use `await` inside `try/catch` or `.catch()` on every promise chain. Never let a rejection propagate silently.

- **Measure before optimizing.** Use the browser DevTools profiler, Node.js `--prof`, or `console.time()` before adding caching, memoization, or other optimizations. Optimize the actual bottleneck, not the imagined one.

- **Prefer flat over nested.** Flat `async/await` chains, flat directory structures, flat class hierarchies, and flat configuration all beat the pyramid equivalents in readability and maintainability.

- **Delete dead code.** Commented-out code, unreachable branches, unused exports, and "just in case" utilities add cognitive overhead with zero value. Delete them and let version control serve as the backup.

## Real-World Use Cases

### 1. Fixing Primitive Obsession in an E-Commerce System

```typescript
// Before: primitives everywhere - easy to mix up arguments
interface Product {
  id: string;
```

### 3. Fixing Async in forEach with Real Data Processing

```typescript
// Before: silent data loss in batch processing
async function importUsers(csvRows: CsvRow[]): Promise<ImportResult> {
  const results: { success: number; failed: number } = { success: 0, failed: 0 };

  csvRows.forEach(async (row) => {
    // These all fire at once and forEach returns before any complete
    try {
      await db.insertUser(transformRow(row));
      results.success++; // May or may not have run by the time we return
    } catch {
      results.failed++;
    }
  });

  // Returns immediately with { success: 0, failed: 0 } in all cases
  return results;
}

// After: controlled concurrency with sequential processing
async function importUsers(
  csvRows: CsvRow[],
  options = { concurrency: 5 },
): Promise<ImportResult> {
  const results = { success: 0, failed: 0 };
  const chunks: CsvRow[][] = [];

  // Split into chunks for controlled concurrency
  for (let i = 0; i < csvRows.length; i += options.concurrency) {
    chunks.push(csvRows.slice(i, i + options.concurrency));
  }

  for (const chunk of chunks) {
    // Process each chunk in parallel, chunks sequentially
    const outcomes = await Promise.allSettled(
      chunk.map(row => db.insertUser(transformRow(row)))
    );
    for (const outcome of outcomes) {
      if (outcome.status === 'fulfilled') results.success++;
      else results.failed++;
    }
  }

  return results;
}
```
