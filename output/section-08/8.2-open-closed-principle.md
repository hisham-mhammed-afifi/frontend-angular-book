# 8.2 Open/Closed Principle (OCP)

> Software entities should be open for extension but closed for modification: you should be able to add new behavior without changing existing, tested code.

---

## The Principle Explained

The Open/Closed Principle (OCP) is the "O" in SOLID. It was first articulated by Bertrand Meyer in *Object-Oriented Software Construction* (1988) and later refined by Robert C. Martin to emphasize abstract interfaces over implementation inheritance.

### The Core Idea

A module is **open for extension** if new behavior can be added to it. A module is **closed for modification** if its existing source code does not need to change when new behavior is added.

At first, this sounds contradictory. How can something be extended without being modified? The answer is **abstraction**. When a module depends on an abstraction (an interface, an abstract class, or a callback), new behavior can be introduced by providing a new implementation of that abstraction, without touching the module itself.

### Meyer vs Martin

Bertrand Meyer's original OCP relied on **implementation inheritance**: extend a class by subclassing it. Robert Martin reframed it around **abstract interfaces and polymorphism**: depend on abstractions, and swap in new implementations. Martin's version is the one used in modern practice.

---

## The Problem OCP Solves

Consider a shipping cost calculator:

```typescript
// Bad Practice: closed for extension, open for modification
class ShippingCalculator {
  calculate(method: string, weight: number): number {
    if (method === "standard") {
      return weight * 1.5;
    } else if (method === "express") {
      return weight * 3.0;
    } else if (method === "overnight") {
      return weight * 5.0;
    }
    throw new Error(`Unknown shipping method: ${method}`);
  }
}
```

Every time a new shipping method is added (drone delivery, same-day, freight), you must **modify** the `ShippingCalculator` class. You must touch existing code, re-test it, and risk introducing regressions in the existing branches.

This is a textbook OCP violation. The growing `if`/`else` or `switch` statement is the most common symptom.

---

## Achieving OCP with Polymorphism

The fix is to invert the dependency: instead of the calculator knowing about every method, define an abstraction that each method implements.

```typescript
// Define the abstraction
interface ShippingStrategy {
  calculate(weight: number): number;
}

// Each shipping method is its own implementation
class StandardShipping implements ShippingStrategy {
  calculate(weight: number): number {
    return weight * 1.5;
  }
}

class ExpressShipping implements ShippingStrategy {
  calculate(weight: number): number {
    return weight * 3.0;
  }
}

class OvernightShipping implements ShippingStrategy {
  calculate(weight: number): number {
    return weight * 5.0;
  }
}

// The calculator depends on the abstraction, not the implementations
class ShippingCalculator {
  calculate(strategy: ShippingStrategy, weight: number): number {
    return strategy.calculate(weight);
  }
}
```

Now adding a new shipping method is pure extension:

```typescript
// New requirement: drone delivery. No existing code is modified.
class DroneShipping implements ShippingStrategy {
  calculate(weight: number): number {
    if (weight > 5) {
      throw new Error("Drone shipping only supports packages up to 5kg");
    }
    return weight * 8.0;
  }
}

// Usage
const calculator = new ShippingCalculator();
calculator.calculate(new DroneShipping(), 2.5); // 20
```

`ShippingCalculator` is **closed** (its source code never changes). The set of shipping methods is **open** (anyone can add a new implementation).

## Extension Mechanisms in TypeScript

OCP can be achieved through several mechanisms. Each has a different level of flexibility.

### 1. Interfaces and Polymorphism (Most Common)

The strategy pattern shown above. This is the primary way to achieve OCP in TypeScript.

```typescript
interface Renderer {
  render(data: unknown): string;
}

class HtmlRenderer implements Renderer {
  render(data: unknown): string {
    return `<div>${JSON.stringify(data)}</div>`;
  }
}

class JsonRenderer implements Renderer {
  render(data: unknown): string {
    return JSON.stringify(data, null, 2);
  }
}

class CsvRenderer implements Renderer {
  render(data: unknown): string {
    if (!Array.isArray(data)) throw new Error("CSV requires array data");
    const headers = Object.keys(data[0]);
    const rows = data.map(row => headers.map(h => String(row[h])).join(","));
    return [headers.join(","), ...rows].join("\n");
  }
}

// Closed for modification. Open for extension (add MarkdownRenderer, XmlRenderer, etc.)
function generateReport(renderer: Renderer, data: unknown): string {
  return renderer.render(data);
}
```

### 2. Higher-Order Functions and Callbacks

In functional-style TypeScript, OCP is achieved by passing behavior as arguments:

```typescript
type Predicate<T> = (item: T) => boolean;
type Transformer<T, U> = (item: T) => U;

// This function is closed. Its behavior is extended by passing different functions.
function processItems<T, U>(
  items: T[],
  filter: Predicate<T>,
  transform: Transformer<T, U>
): U[] {
  return items.filter(filter).map(transform);
}

// Extension 1: get names of active users
const activeUserNames = processItems(
  users,
  user => user.isActive,
  user => user.name
);

// Extension 2: get emails of admins
const adminEmails = processItems(
  users,
  user => user.role === "admin",
  user => user.email
);
```

No need to modify `processItems`. New filtering and transformation logic is plugged in from outside.

### 3. Composition and Decorator Pattern

Wrap existing behavior to extend it without modification:

```typescript
interface Logger {
  log(message: string): void;
}

class ConsoleLogger implements Logger {
  log(message: string): void {
    console.log(message);
  }
}

// Extends logging behavior without modifying ConsoleLogger
class TimestampLogger implements Logger {
  constructor(private readonly inner: Logger) {}

  log(message: string): void {
    const timestamp = new Date().toISOString();
    this.inner.log(`[${timestamp}] ${message}`);
  }
}

class PrefixLogger implements Logger {
  constructor(
    private readonly inner: Logger,
    private readonly prefix: string
  ) {}

  log(message: string): void {
    this.inner.log(`${this.prefix}: ${message}`);
  }
}

// Compose layers of behavior
const logger = new PrefixLogger(
  new TimestampLogger(new ConsoleLogger()),
  "APP"
);
logger.log("Server started");
// Console output: APP: [2026-03-03T10:00:00.000Z] Server started
```

Each wrapper is an extension. `ConsoleLogger` is never modified.

### 4. Plugin/Registry Pattern

For systems where extensions are discovered or registered at runtime:

```typescript
interface Plugin {
  name: string;
  execute(context: AppContext): void;
}

class PluginManager {
  private plugins: Plugin[] = [];

  register(plugin: Plugin): void {
    this.plugins.push(plugin);
  }

  executeAll(context: AppContext): void {
    for (const plugin of this.plugins) {
      plugin.execute(context);
    }
  }
}

// The manager is closed. New plugins are added without modifying it.
const manager = new PluginManager();
manager.register(new AnalyticsPlugin());
manager.register(new LoggingPlugin());
manager.register(new CachePlugin());
```

### 5. Maps/Registries Instead of Switch Statements

When you need to dispatch by a key, use a map instead of a switch:

```typescript
// Bad Practice: switch that must be modified for every new type
function formatValue(type: string, value: unknown): string {
  switch (type) {
    case "currency": return `$${Number(value).toFixed(2)}`;
    case "percent": return `${Number(value)}%`;
    case "date": return new Date(value as string).toLocaleDateString();
    default: return String(value);
  }
}

// Good: registry that can be extended without modifying the function
type Formatter = (value: unknown) => string;

const formatters = new Map<string, Formatter>();
formatters.set("currency", (v) => `$${Number(v).toFixed(2)}`);
formatters.set("percent", (v) => `${Number(v)}%`);
formatters.set("date", (v) => new Date(v as string).toLocaleDateString());

function formatValue(type: string, value: unknown): string {
  const formatter = formatters.get(type);
  if (!formatter) return String(value);
  return formatter(value);
}

// Extension: add a new formatter without modifying formatValue
formatters.set("phone", (v) => {
  const s = String(v).replace(/\D/g, "");
  return `(${s.slice(0, 3)}) ${s.slice(3, 6)}-${s.slice(6)}`;
});
```

## When Modification Is Acceptable

OCP does not mean "never edit a file." It is a design goal, not a religion. There are legitimate reasons to modify existing code:

1. **Bug fixes:** Fixing incorrect behavior is not "modification" in the OCP sense. The module's intended behavior has not changed.

2. **Refactoring:** Restructuring code for clarity without changing behavior is maintenance, not modification.

3. **Changing the abstraction itself:** If the interface needs a new method, all implementations must be updated. This is unavoidable, but a well-designed abstraction minimizes this need.

4. **Simple, bounded enums:** If a type only ever has 3-4 variants and that is inherent to the domain (e.g., traffic light colors), a switch statement is fine. OCP is about code that you **expect to grow**.

The key question is: **Will this code need to handle new variants regularly?** If yes, design for extension. If no, a simple conditional is perfectly acceptable.

## Recognizing OCP Violations

### The Switch/If-Else Smell

Any time you see a conditional that dispatches based on a type or category, ask: "Will this grow?"

```typescript
// OCP violation: new payment methods require modifying this function
function processPayment(method: string, amount: number): void {
  if (method === "credit_card") {
    chargeCreditCard(amount);
  } else if (method === "paypal") {
    chargePayPal(amount);
  } else if (method === "bank_transfer") {
    chargeBankTransfer(amount);
  }
  // Next week: Apple Pay, Google Pay, crypto...
}
```

### The "Shotgun Surgery" Smell

When adding a new feature requires changes in multiple files that all have the same switch/if structure:

```
// Adding "drone" shipping requires changes in:
// - ShippingCalculator.ts (add if branch)
// - ShippingValidator.ts (add if branch)
// - ShippingFormatter.ts (add if branch)
// - ShippingIcon.tsx (add if branch)
// - shipping.test.ts (add test branch)
```

If the same new case must be added in 5 places, those 5 places are all OCP violations. A polymorphic approach centralizes each variant in one place.

### The Growing Import List

If a file imports more and more concrete classes over time, it is likely depending on implementations rather than abstractions.

## Common Mistakes

### 1. Premature Abstraction

```typescript
// Bad Practice: creating an interface for something that will never have a second implementation
interface UserNameFormatter {
  format(name: string): string;
}

class DefaultUserNameFormatter implements UserNameFormatter {
  format(name: string): string {
    return name.trim();
  }
}
// If there will never be another way to format a user name, this is over-engineering.
```

OCP is about code that you **expect** to change. If the behavior is stable and has only one implementation, an interface adds indirection without value. Add the abstraction when the second use case appears, not before.

### 2. Using Inheritance Instead of Composition

Meyer's original OCP relied on subclassing, but deep inheritance hierarchies are fragile:

```typescript
// Bad Practice: deep inheritance chain
class BaseProcessor { /* ... */ }
class ValidatingProcessor extends BaseProcessor { /* ... */ }
class LoggingValidatingProcessor extends ValidatingProcessor { /* ... */ }
class CachingLoggingValidatingProcessor extends LoggingValidatingProcessor { /* ... */ }
```

This creates tight coupling between layers. Prefer composition (the decorator pattern, strategy pattern, or dependency injection) over inheritance for extension.

### 3. Leaking Implementation Details Through the Interface

```typescript
// Bad Practice: the interface is too specific to one implementation
interface DataStore {
  connectToPostgres(connectionString: string): void;
  runSqlQuery(sql: string): Promise<unknown[]>;
  closePostgresConnection(): void;
}

// This cannot be extended with a MongoDB implementation
// because the interface assumes SQL and Postgres
```

```typescript
// Good: abstract the interface to represent the capability, not the technology
interface DataStore {
  connect(): Promise<void>;
  find(collection: string, query: Record<string, unknown>): Promise<unknown[]>;
  save(collection: string, data: unknown): Promise<void>;
  disconnect(): Promise<void>;
}
```

### 4. Modifying the Interface Every Time a New Feature Is Needed

If you find yourself adding methods to an interface regularly, the abstraction is too narrow or at the wrong level:

```typescript
// Every new report type adds a method. This is modification, not extension.
interface ReportService {
  generateSalesReport(): Report;
  generateInventoryReport(): Report;
  generateUserReport(): Report;
  // ... keeps growing
}

// Better: parameterize the report type
interface ReportGenerator {
  generate(data: unknown): Report;
}

// Each report type is a separate implementation
class SalesReportGenerator implements ReportGenerator { /* ... */ }
class InventoryReportGenerator implements ReportGenerator { /* ... */ }
```

### 5. Confusing "Closed for Modification" with "Immutable Code"

OCP does not mean code is frozen forever. It means that **adding new behavior should not require editing existing behavior.** You can still refactor, fix bugs, and improve performance in existing code. The principle targets feature additions, not maintenance.

## Best Practices

### 1. Identify the Axis of Change

Before designing for extension, identify **what is likely to change**. A payment system will likely add new payment methods. A rendering system will likely add new output formats. An analytics system will likely add new event types. These are the axes of change that benefit from OCP.

### 2. Use the Strategy Pattern for Behavioral Variation

When different variants share the same interface but differ in behavior, the strategy pattern is the natural fit:

```typescript
interface SortStrategy<T> {
  sort(items: T[]): T[];
}

class QuickSort<T> implements SortStrategy<T> {
  sort(items: T[]): T[] {
    // Quick sort implementation
    return [...items].sort();
  }
}

class MergeSort<T> implements SortStrategy<T> {
  sort(items: T[]): T[] {
    // Merge sort implementation
    return [...items].sort();
  }
}

class Sorter<T> {
  constructor(private strategy: SortStrategy<T>) {}

  sort(items: T[]): T[] {
    return this.strategy.sort(items);
  }

  setStrategy(strategy: SortStrategy<T>): void {
    this.strategy = strategy;
  }
}
```

### 3. Depend on Abstractions, Not Concretions

This is the Dependency Inversion Principle (DIP), which enables OCP. When a module depends on an interface, any new implementation automatically extends that module's behavior:

```typescript
// Depends on the interface, not on EmailNotifier specifically
class OrderProcessor {
  constructor(private readonly notifier: Notifier) {}

  async process(order: Order): Promise<void> {
    await this.saveOrder(order);
    await this.notifier.notify(order.customerEmail, `Order ${order.id} confirmed`);
  }
}
```

### 4. Apply OCP Progressively

Do not pre-design abstractions for every possible future requirement. Instead:

1. Write the simplest solution first.
2. When the second variant appears, refactor to introduce the abstraction.
3. The third and subsequent variants slot in through extension.

This is sometimes called the "Rule of Three": abstract after you see the pattern three times, not before.

### 5. Make the Closed Parts Small and the Open Parts Clear

The closed core should be as small and stable as possible. The extension points should be clearly documented so consumers know where to plug in:

```typescript
/**
 * Add new validation rules by implementing the ValidationRule interface
 * and registering them with the Validator.
 */
interface ValidationRule<T> {
  name: string;
  validate(value: T): ValidationResult;
}

class Validator<T> {
  private rules: ValidationRule<T>[] = [];

  addRule(rule: ValidationRule<T>): void {
    this.rules.push(rule);
  }

  validate(value: T): ValidationResult[] {
    return this.rules.map(rule => rule.validate(value));
  }
}
```

## Real-World Use Cases

### 1. Extensible Validation Pipeline

```typescript
interface ValidationRule<T> {
  name: string;
  validate(value: T): { valid: boolean; message?: string };
}

// Each rule is a self-contained unit
class RequiredRule implements ValidationRule<string> {
  name = "required";
  validate(value: string) {
    return value.trim().length > 0
      ? { valid: true }
      : { valid: false, message: "Field is required" };
  }
}

class MinLengthRule implements ValidationRule<string> {
  name: string;
  constructor(private min: number) {
    this.name = `minLength(${min})`;
  }
  validate(value: string) {
    return value.length >= this.min
      ? { valid: true }
      : { valid: false, message: `Must be at least ${this.min} characters` };
  }
}

class PatternRule implements ValidationRule<string> {
  name: string;
  constructor(private pattern: RegExp, private message: string) {
    this.name = `pattern(${pattern.source})`;
  }
  validate(value: string) {
    return this.pattern.test(value)
      ? { valid: true }
      : { valid: false, message: this.message };
  }
}

// The validator is closed for modification
class FieldValidator<T> {
  private rules: ValidationRule<T>[] = [];

  addRule(rule: ValidationRule<T>): this {
    this.rules.push(rule);
    return this;
  }

  validate(value: T): { valid: boolean; errors: string[] } {
    const errors: string[] = [];
    for (const rule of this.rules) {
      const result = rule.validate(value);
      if (!result.valid && result.message) {
        errors.push(result.message);
      }
    }
    return { valid: errors.length === 0, errors };
  }
}

// Usage: compose validators from rules
const emailValidator = new FieldValidator<string>()
  .addRule(new RequiredRule())
  .addRule(new MinLengthRule(5))
  .addRule(new PatternRule(/^[^\s@]+@[^\s@]+\.[^\s@]+$/, "Invalid email format"));

const result = emailValidator.validate("bad");
// { valid: false, errors: ["Must be at least 5 characters", "Invalid email format"] }

// Extension: add a custom rule without modifying any existing code
class BlacklistRule implements ValidationRule<string> {
  name = "blacklist";
  constructor(private blocked: string[]) {}
  validate(value: string) {
    return !this.blocked.includes(value)
      ? { valid: true }
      : { valid: false, message: "This value is not allowed" };
  }
}

emailValidator.addRule(new BlacklistRule(["test@test.com", "admin@admin.com"]));
```

### 2. Pluggable Export System

```typescript
interface Exporter<T> {
  readonly format: string;
  readonly mimeType: string;
  export(data: T[]): string;
}

class CsvExporter<T extends Record<string, unknown>> implements Exporter<T> {
  format = "csv";
  mimeType = "text/csv";

  export(data: T[]): string {
    if (data.length === 0) return "";
    const headers = Object.keys(data[0]);
    const rows = data.map(row =>
      headers.map(h => String(row[h] ?? "")).join(",")
    );
    return [headers.join(","), ...rows].join("\n");
  }
}

class JsonExporter<T> implements Exporter<T> {
  format = "json";
  mimeType = "application/json";

  export(data: T[]): string {
    return JSON.stringify(data, null, 2);
  }
}

class XmlExporter<T extends Record<string, unknown>> implements Exporter<T> {
  format = "xml";
  mimeType = "application/xml";

  export(data: T[]): string {
    const rows = data.map(row => {
      const fields = Object.entries(row)
        .map(([key, val]) => `    <${key}>${val}</${key}>`)
        .join("\n");
      return `  <row>\n${fields}\n  </row>`;
    });
    return `<?xml version="1.0"?>\n<data>\n${rows.join("\n")}\n</data>`;
  }
}

// The export service is closed for modification
class ExportService<T extends Record<string, unknown>> {
  private exporters = new Map<string, Exporter<T>>();

  register(exporter: Exporter<T>): void {
    this.exporters.set(exporter.format, exporter);
  }

  getSupportedFormats(): string[] {
    return Array.from(this.exporters.keys());
  }

  export(format: string, data: T[]): { content: string; mimeType: string } {
    const exporter = this.exporters.get(format);
    if (!exporter) {
      throw new Error(
        `Unsupported format: ${format}. Supported: ${this.getSupportedFormats().join(", ")}`
      );
    }
    return {
      content: exporter.export(data),
      mimeType: exporter.mimeType,
    };
  }
}

// Setup
const service = new ExportService();
service.register(new CsvExporter());
service.register(new JsonExporter());
service.register(new XmlExporter());

// Adding a new format later: no modification to ExportService
class MarkdownExporter<T extends Record<string, unknown>> implements Exporter<T> {
  format = "markdown";
  mimeType = "text/markdown";

  export(data: T[]): string {
    if (data.length === 0) return "";
    const headers = Object.keys(data[0]);
    const headerRow = `| ${headers.join(" | ")} |`;
    const separator = `| ${headers.map(() => "---").join(" | ")} |`;
    const rows = data.map(row =>
      `| ${headers.map(h => String(row[h] ?? "")).join(" | ")} |`
    );
    return [headerRow, separator, ...rows].join("\n");
  }
}

service.register(new MarkdownExporter());
```

### 3. Middleware Pipeline

```typescript
type Context = {
  request: Request;
  response: Partial<Response>;
  state: Record<string, unknown>;
};

type Next = () => Promise<void>;
type Middleware = (ctx: Context, next: Next) => Promise<void>;

// The pipeline runner is closed for modification
class Pipeline {
  private middlewares: Middleware[] = [];

  use(middleware: Middleware): void {
    this.middlewares.push(middleware);
  }

  async execute(ctx: Context): Promise<void> {
    let index = 0;

    const next = async (): Promise<void> => {
      if (index < this.middlewares.length) {
        const mw = this.middlewares[index++];
        await mw(ctx, next);
      }
    };

    await next();
  }
}

// Extension through new middleware (no modification to Pipeline)
const logging: Middleware = async (ctx, next) => {
  const start = Date.now();
  await next();
  console.log(`${ctx.request.method} ${ctx.request.url} - ${Date.now() - start}ms`);
};

const auth: Middleware = async (ctx, next) => {
  const token = ctx.request.headers.get("Authorization");
  if (!token) {
    ctx.response.status = 401;
    return;
  }
  ctx.state.userId = verifyToken(token);
  await next();
};

const rateLimit: Middleware = async (ctx, next) => {
  const ip = ctx.request.headers.get("X-Forwarded-For") ?? "unknown";
  if (isRateLimited(ip)) {
    ctx.response.status = 429;
    return;
  }
  await next();
};

// Compose the pipeline
const pipeline = new Pipeline();
pipeline.use(logging);
pipeline.use(rateLimit);
pipeline.use(auth);

declare function verifyToken(token: string): string;
declare function isRateLimited(ip: string): boolean;
```

The `Pipeline` class never changes. New behaviors (CORS, compression, caching) are added by writing new middleware functions and calling `pipeline.use()`.

---

> **Key Takeaway:** Design modules that can be extended with new behavior through abstractions (interfaces, callbacks, registries) rather than requiring edits to existing, tested code; identify the axes of change in your system and build extension points there.
