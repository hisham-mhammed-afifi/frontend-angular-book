# 8.4 Interface Segregation Principle (ISP)

> No client should be forced to depend on methods it does not use. Prefer many small, focused interfaces over one large, general-purpose interface.

---

## The Principle Explained

The Interface Segregation Principle is the "I" in SOLID. It was formulated by Robert C. Martin while consulting for Xerox on a printer system where a single, monolithic interface was causing cascading recompilation and coupling problems.

### The Core Idea

When an interface grows too large, the classes that implement it are forced to provide implementations for methods they do not need. This leads to:

- **Empty or dummy implementations** that do nothing or throw "not supported" errors
- **Unnecessary coupling** where a change to one method forces recompilation and retesting of classes that never use that method
- **Confused consumers** who see methods on an object that do not apply to their use case

ISP says: **split large interfaces into smaller, role-specific ones** so that each client depends only on the methods it actually uses.

## The Fat Interface Problem

### A Classic Violation

```typescript
// Bad Practice: one interface with everything
interface Machine {
  print(document: Document): void;
  scan(document: Document): Image;
  fax(document: Document, number: string): void;
  staple(document: Document): void;
  copy(document: Document, copies: number): void;
}
```

A high-end office printer can implement all five methods. But what about a basic home printer?

```typescript
// Forced to implement methods it cannot perform
class HomePrinter implements Machine {
  print(document: Document): void {
    // Works fine
  }

  scan(document: Document): Image {
    throw new Error("This printer does not have a scanner");
  }

  fax(document: Document, number: string): void {
    throw new Error("This printer does not have a fax modem");
  }

  staple(document: Document): void {
    throw new Error("This printer does not have a stapler");
  }

  copy(document: Document, copies: number): void {
    throw new Error("This printer cannot copy");
  }
}
```

Four out of five methods throw errors. Any code that receives a `Machine` and calls `scan()` will crash when given a `HomePrinter`. This is both an ISP violation and an LSP violation (the subtype cannot fulfill the contract).

### The Fix: Segregated Interfaces

```typescript
interface Printer {
  print(document: Document): void;
}

interface Scanner {
  scan(document: Document): Image;
}

interface Fax {
  fax(document: Document, number: string): void;
}

interface Stapler {
  staple(document: Document): void;
}

interface Copier {
  copy(document: Document, copies: number): void;
}

// Home printer only implements what it can do
class HomePrinter implements Printer {
  print(document: Document): void {
    // Print the document
  }
}

// Office machine implements everything it supports
class OfficeMachine implements Printer, Scanner, Copier, Fax, Stapler {
  print(document: Document): void { /* ... */ }
  scan(document: Document): Image { /* ... */ return {} as Image; }
  copy(document: Document, copies: number): void { /* ... */ }
  fax(document: Document, number: string): void { /* ... */ }
  staple(document: Document): void { /* ... */ }
}

// Scanner-printer combo
class ScannerPrinter implements Printer, Scanner {
  print(document: Document): void { /* ... */ }
  scan(document: Document): Image { /* ... */ return {} as Image; }
}
```

Now each client depends only on what it needs:

```typescript
// This function only needs printing capability
function printReport(printer: Printer, report: Document): void {
  printer.print(report);
}

// This function only needs scanning capability
function digitizeDocument(scanner: Scanner, doc: Document): Image {
  return scanner.scan(doc);
}

// Both work with HomePrinter, ScannerPrinter, or OfficeMachine
// depending on which interfaces they implement
printReport(new HomePrinter(), report);       // OK
printReport(new OfficeMachine(), report);     // OK
digitizeDocument(new ScannerPrinter(), doc);  // OK
// digitizeDocument(new HomePrinter(), doc);  // Compile error: HomePrinter is not a Scanner
```

The compile error is the goal. ISP turns runtime "not supported" errors into compile-time type errors.

## Role Interfaces vs Header Interfaces

### Role Interfaces

A **role interface** defines a specific capability or role that a type plays. Each interface represents one thing the type can do:

```typescript
interface Readable {
  read(): string;
}

interface Writable {
  write(data: string): void;
}

interface Closable {
  close(): void;
}
```

These are small, focused, and composable. A type can implement any combination of roles.

### Header Interfaces

A **header interface** mirrors the full public API of a class. It is the opposite of a role interface:

```typescript
// Header interface: mirrors the entire class
interface FileStream {
  read(): string;
  write(data: string): void;
  close(): void;
  seek(position: number): void;
  flush(): void;
  getPosition(): number;
  getSize(): number;
}
```

Header interfaces are fat interfaces by definition. They force every consumer to depend on every method, even if they only need one. Prefer role interfaces.

### Composing Role Interfaces

TypeScript's intersection types and interface extension make composing role interfaces natural:

```typescript
// Compose specific combinations
type ReadWriteStream = Readable & Writable & Closable;
type ReadOnlyStream = Readable & Closable;

// Or use interface extension
interface ReadWriteClosable extends Readable, Writable, Closable {}

// Functions declare exactly what they need
function processInput(source: Readable): string {
  return source.read();
}

function saveOutput(target: Writable, data: string): void {
  target.write(data);
}

function transferData(source: Readable, target: Writable): void {
  const data = source.read();
  target.write(data);
}
```

Each function depends on the **minimum** interface it needs. A `Readable` source could be a file, a network stream, a string buffer, or a mock in tests. The function does not care.

## ISP Applied to TypeScript Types

ISP is not limited to `interface` declarations. It applies to any type boundary: function parameters, generic constraints, object shapes.

### Function Parameters

```typescript
// Bad Practice: function requires a full User object but only uses email
interface User {
  id: string;
  name: string;
  email: string;
  avatar: string;
  preferences: UserPreferences;
  createdAt: Date;
}

function sendWelcomeEmail(user: User): void {
  // Only uses email and name
  mailer.send(user.email, `Welcome, ${user.name}!`);
}

// Good: function declares exactly what it needs
function sendWelcomeEmail(user: { email: string; name: string }): void {
  mailer.send(user.email, `Welcome, ${user.name}!`);
}
```

The second version is easier to test (pass a simple object), easier to reuse (works with any object that has `email` and `name`), and communicates its actual dependencies clearly.

### Generic Constraints

```typescript
// Bad Practice: overly broad constraint
function getFullName<T extends User>(entity: T): string {
  return `${entity.name}`;
}

// Good: constraint specifies only what is needed
function getFullName<T extends { name: string }>(entity: T): string {
  return entity.name;
}
```

### Pick and Utility Types

TypeScript's built-in utility types support ISP naturally:

```typescript
interface Product {
  id: string;
  name: string;
  description: string;
  price: number;
  stock: number;
  category: string;
  images: string[];
  rating: number;
}

// Only the fields needed for display
type ProductCard = Pick<Product, "id" | "name" | "price" | "images">;

// Only the fields needed for inventory
type InventoryItem = Pick<Product, "id" | "name" | "stock">;

function renderProductCard(product: ProductCard): void {
  // Only has access to id, name, price, images
}

function updateStock(item: InventoryItem, newStock: number): void {
  // Only has access to id, name, stock
}
```

## ISP and the Repository Pattern

The repository pattern is a common place where ISP violations occur.

### The Fat Repository

```typescript
// Bad Practice: one repository interface for everything
interface Repository<T> {
  find(id: string): Promise<T | null>;
  findAll(): Promise<T[]>;
  findByQuery(query: Partial<T>): Promise<T[]>;
  save(entity: T): Promise<void>;
  update(id: string, data: Partial<T>): Promise<void>;
  delete(id: string): Promise<void>;
  count(): Promise<number>;
  exists(id: string): Promise<boolean>;
  bulkInsert(entities: T[]): Promise<void>;
  bulkDelete(ids: string[]): Promise<void>;
}
```

A read-only analytics dashboard only needs `find`, `findAll`, and `findByQuery`. A simple form only needs `find` and `save`. Forcing every consumer to depend on all ten methods is an ISP violation.

### Segregated Repository

```typescript
interface ReadRepository<T> {
  find(id: string): Promise<T | null>;
  findAll(): Promise<T[]>;
  findByQuery(query: Partial<T>): Promise<T[]>;
}

interface WriteRepository<T> {
  save(entity: T): Promise<void>;
  update(id: string, data: Partial<T>): Promise<void>;
  delete(id: string): Promise<void>;
}

interface BulkRepository<T> {
  bulkInsert(entities: T[]): Promise<void>;
  bulkDelete(ids: string[]): Promise<void>;
}

interface CountRepository {
  count(): Promise<number>;
  exists(id: string): Promise<boolean>;
}

// Full repository for code that needs everything
type FullRepository<T> = ReadRepository<T> & WriteRepository<T> & BulkRepository<T> & CountRepository;
```

Now each consumer declares exactly what it needs:

```typescript
// Dashboard only reads
class AnalyticsDashboard {
  constructor(private readonly repo: ReadRepository<Order>) {}

  async getRecentOrders(): Promise<Order[]> {
    return this.repo.findAll();
  }
}

// Admin panel reads and writes
class OrderAdmin {
  constructor(private readonly repo: ReadRepository<Order> & WriteRepository<Order>) {}

  async updateOrder(id: string, data: Partial<Order>): Promise<void> {
    await this.repo.update(id, data);
  }
}

// Import service needs bulk operations
class DataImporter {
  constructor(private readonly repo: BulkRepository<Order>) {}

  async importOrders(orders: Order[]): Promise<void> {
    await this.repo.bulkInsert(orders);
  }
}
```

## How ISP Relates to Other SOLID Principles

### ISP and SRP

SRP says a class should have one reason to change. ISP says an interface should not force clients to depend on methods they do not use. They are complementary: SRP focuses on the implementation, ISP focuses on the contract exposed to consumers.

A class can follow SRP (single responsibility internally) but violate ISP if its interface exposes too many methods to clients that only need a subset.

### ISP and LSP

ISP prevents LSP violations. When an interface is too large, implementors are tempted to throw "not supported" errors or leave methods empty, which violates LSP. Smaller interfaces mean every implementor can genuinely fulfill the contract.

### ISP and OCP

Segregated interfaces make the system more open for extension. New capabilities can be added as new interfaces without modifying existing ones. Existing code that depends on the smaller interfaces is unaffected.

## Common Mistakes

### 1. Creating One Interface Per Method (Over-Segregation)

```typescript
// Bad Practice: taken too far
interface HasId { id: string; }
interface HasName { name: string; }
interface HasEmail { email: string; }
interface HasAge { age: number; }
interface HasAddress { address: string; }
// ... one interface per field

type User = HasId & HasName & HasEmail & HasAge & HasAddress;
```

This is excessive. ISP does not mean "one method per interface." It means "group methods by the roles that clients actually use together." If `name` and `email` are always used together by consumers, they belong in the same interface.

### 2. Splitting Interfaces by Implementation Rather Than by Client Need

```typescript
// Bad Practice: split based on what the class does internally
interface DatabaseMethods {
  connect(): void;
  disconnect(): void;
  query(sql: string): unknown[];
}

interface CacheMethods {
  getFromCache(key: string): unknown;
  setInCache(key: string, value: unknown): void;
}
```

ISP says split based on **what clients need**, not on how the implementation is structured internally. If a client needs both `query` and caching, they need both interfaces regardless of the internal split.

### 3. Forcing All Implementors to Use the Same Segregation

```typescript
// Not every consumer needs the same split
// Sometimes one client needs Read + Write, another needs Read + Count
// The segregation should match client usage patterns, not a fixed template
```

Analyze your actual consumers. Different applications of the same domain may need different interface groupings.

### 4. Using Marker Interfaces Instead of Real Segregation

```typescript
// Bad Practice: empty interfaces used as markers
interface Printable {}
interface Scannable {}

class Printer implements Printable {
  print(): void { /* ... */ }
}

// The marker interface tells you nothing about the capability.
// A function accepting Printable cannot call print() because
// Printable has no methods.
```

Marker interfaces (empty interfaces) do not provide ISP benefits. The interface must declare the methods the role provides.

### 5. Not Segregating Callback/Event Interfaces

```typescript
// Bad Practice: one handler interface with every event
interface AppEventHandler {
  onUserLogin(user: User): void;
  onUserLogout(userId: string): void;
  onOrderCreated(order: Order): void;
  onOrderShipped(orderId: string): void;
  onPaymentReceived(payment: Payment): void;
  onError(error: Error): void;
}

// Every listener must implement all six methods even if it only cares about one

// Good: separate handler interfaces
interface LoginHandler {
  onUserLogin(user: User): void;
}

interface OrderHandler {
  onOrderCreated(order: Order): void;
  onOrderShipped(orderId: string): void;
}

interface ErrorHandler {
  onError(error: Error): void;
}
```

## Best Practices

### 1. Design Interfaces from the Client's Perspective

Do not start by asking "What can this class do?" Start by asking "What does each consumer need?"

```typescript
// Start from the consumer:
// - The checkout page needs: getCart, addItem, removeItem, getTotal
// - The analytics page needs: getCart, getTotal, getItemCount
// - The admin page needs: clearCart, setDiscount

interface CartReader {
  getCart(): CartItem[];
  getTotal(): number;
}

interface CartWriter {
  addItem(item: CartItem): void;
  removeItem(itemId: string): void;
}

interface CartAdmin {
  clearCart(): void;
  setDiscount(percent: number): void;
}

interface CartAnalytics extends CartReader {
  getItemCount(): number;
}
```

### 2. Use TypeScript's Structural Typing to Your Advantage

In TypeScript, you do not need explicit `implements` to satisfy an interface. Any object with the right shape works:

```typescript
interface Loggable {
  toString(): string;
}

// No need for "implements Loggable"
class Order {
  constructor(public id: string, public total: number) {}
  toString(): string {
    return `Order #${this.id}: $${this.total}`;
  }
}

function logItem(item: Loggable): void {
  console.log(item.toString());
}

logItem(new Order("123", 49.99)); // Works without explicit implements
```

This means ISP in TypeScript is even more natural than in nominally-typed languages. You can define the exact shape you need inline.

### 3. Prefer Inline Type Narrowing for One-Off Functions

For functions used in a single context, you do not always need a named interface:

```typescript
// Perfectly fine for a utility function
function formatDisplayName(user: { firstName: string; lastName: string }): string {
  return `${user.firstName} ${user.lastName}`;
}
```

Name the interface when it is used by multiple functions or represents a meaningful domain role.

### 4. Group Methods That Change Together

Methods that are always updated together belong in the same interface. Methods that change for different reasons belong in separate interfaces:

```typescript
// These methods change together (both relate to display rendering)
interface Renderable {
  render(): HTMLElement;
  getStyles(): CSSStyleDeclaration;
}

// These methods change for different reasons (data vs display)
// and should be in separate interfaces
interface DataProvider {
  getData(): unknown;
}
```

### 5. Apply ISP Progressively

Like all SOLID principles, apply ISP when you feel the pain. If an interface has three methods and all consumers use all three, it is fine. When a new consumer appears that only needs one method, that is the signal to segregate.

## Real-World Use Cases

### 1. Authentication System with Role-Based Interfaces

```typescript
interface Authenticatable {
  verifyCredentials(email: string, password: string): Promise<AuthResult>;
}

interface TokenProvider {
  generateToken(userId: string): string;
  verifyToken(token: string): TokenPayload | null;
  revokeToken(token: string): void;
}

interface PasswordManager {
  hashPassword(password: string): Promise<string>;
  comparePassword(plain: string, hashed: string): Promise<boolean>;
}

interface SessionManager {
  createSession(userId: string): Session;
  getSession(sessionId: string): Session | null;
  destroySession(sessionId: string): void;
}

// The login endpoint only needs authentication and token generation
class LoginController {
  constructor(
    private readonly auth: Authenticatable,
    private readonly tokens: TokenProvider
  ) {}

  async login(email: string, password: string): Promise<{ token: string }> {
    const result = await this.auth.verifyCredentials(email, password);
    if (!result.success) throw new Error("Invalid credentials");
    const token = this.tokens.generateToken(result.userId);
    return { token };
  }
}

// The password reset flow only needs password management
class PasswordResetController {
  constructor(private readonly passwords: PasswordManager) {}

  async resetPassword(newPassword: string): Promise<string> {
    return this.passwords.hashPassword(newPassword);
  }
}

// The auth middleware only needs token verification
class AuthMiddleware {
  constructor(private readonly tokens: TokenProvider) {}

  verify(token: string): TokenPayload | null {
    return this.tokens.verifyToken(token);
  }
}
```

Each controller depends on exactly the capability it needs. Testing the `LoginController` requires mocking only `Authenticatable` and `TokenProvider`, not the entire auth system.

### 2. Notification Service with Segregated Channels

```typescript
interface MessageComposer {
  compose(template: string, data: Record<string, unknown>): string;
}

interface EmailSender {
  sendEmail(to: string, subject: string, body: string): Promise<void>;
}

interface SmsSender {
  sendSms(to: string, body: string): Promise<void>;
}

interface PushSender {
  sendPush(deviceToken: string, title: string, body: string): Promise<void>;
}

// Order confirmation only needs email
class OrderConfirmation {
  constructor(
    private readonly composer: MessageComposer,
    private readonly email: EmailSender
  ) {}

  async send(order: Order): Promise<void> {
    const body = this.composer.compose("order-confirmation", { order });
    await this.email.sendEmail(order.customerEmail, "Order Confirmed", body);
  }
}

// Security alert needs SMS and push (not email)
class SecurityAlert {
  constructor(
    private readonly sms: SmsSender,
    private readonly push: PushSender
  ) {}

  async alert(user: { phone: string; deviceToken: string }, message: string): Promise<void> {
    await Promise.all([
      this.sms.sendSms(user.phone, message),
      this.push.sendPush(user.deviceToken, "Security Alert", message),
    ]);
  }
}
```

### 3. Data Table Component with Segregated Feature Interfaces

```typescript
interface Sortable<T> {
  sort(data: T[], column: keyof T, direction: "asc" | "desc"): T[];
}

interface Filterable<T> {
  filter(data: T[], predicate: (item: T) => boolean): T[];
}

interface Pageable<T> {
  paginate(data: T[], page: number, pageSize: number): { items: T[]; total: number };
}

interface Exportable<T> {
  exportToCsv(data: T[]): string;
}

// A simple list only needs filtering
class SimpleList<T> {
  constructor(private readonly filterer: Filterable<T>) {}

  render(data: T[], searchTerm: string): T[] {
    return this.filterer.filter(data, (item) =>
      JSON.stringify(item).toLowerCase().includes(searchTerm.toLowerCase())
    );
  }
}

// A full data table needs sorting, filtering, and pagination
class DataTable<T> {
  constructor(
    private readonly sorter: Sortable<T>,
    private readonly filterer: Filterable<T>,
    private readonly pager: Pageable<T>
  ) {}

  render(
    data: T[],
    sortColumn: keyof T,
    sortDir: "asc" | "desc",
    filterFn: (item: T) => boolean,
    page: number,
    pageSize: number
  ) {
    const filtered = this.filterer.filter(data, filterFn);
    const sorted = this.sorter.sort(filtered, sortColumn, sortDir);
    return this.pager.paginate(sorted, page, pageSize);
  }
}

// An export button only needs the export capability
class ExportButton<T> {
  constructor(private readonly exporter: Exportable<T>) {}

  handleClick(data: T[]): string {
    return this.exporter.exportToCsv(data);
  }
}
```

Each component declares exactly the capabilities it requires. A `SimpleList` does not depend on sorting or pagination. An `ExportButton` does not depend on filtering or sorting. Adding a new feature (e.g., grouping) means adding a new interface without modifying existing ones.

---

> **Key Takeaway:** Design interfaces from the consumer's perspective, grouping only the methods each client actually needs; this turns runtime "not supported" errors into compile-time type errors and keeps your codebase loosely coupled and easy to test.
