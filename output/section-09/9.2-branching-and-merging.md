# 9.2 Branching and Merging

> Branches let you develop features, fix bugs, and experiment in isolation; merging and rebasing are how you bring that work back together -- choosing the wrong strategy at the wrong time causes history chaos that is painful to untangle.

---

## How Branches Work Internally

A branch in Git is nothing more than a lightweight pointer to a commit. Creating a branch does not copy files -- it creates a 41-byte file in `.git/refs/heads/` containing a commit SHA.

```
main    ->  c7f2a1e  (commit)
                |
           parent commits
```

When you commit on a branch, Git moves the pointer forward:

```
              feat/login
                  |
main  ->  c7f2a1e  ->  d8b3f4a  ->  e9c4g5b
```

`HEAD` is a special pointer that tracks which branch you are currently on. When you switch branches, Git moves `HEAD` to point to the new branch and updates the working tree to match that branch's tip commit.

## Branch Commands

### Creating and Switching

```bash
# Create a new branch (does not switch to it)
git branch feature/user-profile

# Create and switch in one step
git switch -c feature/user-profile
# (older equivalent: git checkout -b feature/user-profile)

# Switch to an existing branch
git switch main
# (older equivalent: git checkout main)

# Create a branch from a specific commit or tag
git switch -c hotfix/payment-crash v2.1.0
git switch -c recovery abc1234  # from a specific commit hash
```

Use `git switch` (added in Git 2.23) instead of `git checkout` for branch operations. `checkout` does too many things; `switch` has a single, clear purpose.

### Listing and Inspecting

```bash
# List local branches
git branch

# List all branches (local + remote tracking)
git branch -a

# List remote branches only
git branch -r

# Show last commit on each branch
git branch -v

# Show which branches have been merged into current branch
git branch --merged

# Show which branches have NOT been merged into current branch
git branch --no-merged
```

### Renaming and Deleting

```bash
# Rename the current branch
git branch -m new-name

# Rename a specific branch
git branch -m old-name new-name

# Delete a branch that has been merged
git branch -d feature/done

# Force-delete a branch regardless of merge status
git branch -D feature/abandoned
# Use with caution: unmerged commits become unreachable (recoverable via reflog for 90 days)

# Delete a remote tracking branch
git push origin --delete feature/old-branch
```

## Merging

Merging integrates changes from one branch into another. Git performs different types of merges depending on the commit history.

### Fast-Forward Merge

A fast-forward merge happens when the target branch has no new commits since the source branch diverged. Git simply moves the branch pointer forward -- no merge commit is created.

```
Before:
main: A - B - C
                \
feature:         D - E - F

After fast-forward merge:
main: A - B - C - D - E - F
feature: (same, or deleted)
```

```bash
# Switch to the branch you want to merge INTO
git switch main

# Merge the feature branch (fast-forward if possible)
git merge feature/quick-fix

# Force a merge commit even when fast-forward is possible
git merge --no-ff feature/quick-fix
```

When to use fast-forward:
- Short-lived branches with a single logical change
- When you want a perfectly linear history
- Automated hotfix branches that never diverge

### Three-Way Merge

When both branches have diverged (both have new commits since their common ancestor), Git performs a three-way merge using three commits: the two branch tips and their common ancestor.

```
Before:
main:    A - B - C - G
                  \
feature:           D - E - F

After three-way merge:
main:    A - B - C - G - M   (M is the merge commit)
                  \       /
feature:           D - E - F
```

```bash
git switch main
git merge feature/new-dashboard
# Git creates a merge commit M automatically

# If there are conflicts, resolve them, then:
git add resolved-file.ts
git merge --continue
# (or git commit if --continue is not available)

# Abort the merge and return to the pre-merge state
git merge --abort
```

When to use `--no-ff` (force merge commit):
- Feature branches -- preserves the fact that a feature existed as a separate branch
- Makes `git log --graph` more readable
- Allows reverting the entire feature with one `git revert <merge-commit>`

### Squash Merge

A squash merge takes all the commits from the source branch, condenses them into a single set of changes, and stages them in your working tree. You then make one final commit. No branch history is preserved in the target branch.

```bash
git switch main
git merge --squash feature/messy-wip
# Git stages all changes but does NOT commit
git commit -m "feat: add user avatar upload"
# All 15 WIP commits from the feature branch become one clean commit on main
```

When to use squash merge:
- Feature branches with many messy WIP commits
- When the branch history is irrelevant and you only care about the final result
- GitHub/GitLab "Squash and merge" button uses this strategy

Downside: the individual commits on the feature branch are no longer in the target branch's history. `git branch --merged` will NOT show the feature branch as merged (because no merge commit was created), so you must delete the branch manually after.

## Rebasing

Rebase moves or replays a sequence of commits onto a new base commit, creating brand-new commits with the same changes but different parents (and thus different SHA hashes).

```
Before rebase:
main:    A - B - C - G
                  \
feature:           D - E - F

After: git switch feature && git rebase main
main:    A - B - C - G
                      \
feature:               D' - E' - F'  (new commits, same changes)
```

The feature branch is now "based on" G instead of C. A subsequent merge of feature into main will be a fast-forward.

```bash
# Switch to the branch you want to rebase
git switch feature/new-api

# Rebase onto main (replay this branch's commits on top of main's tip)
git rebase main

# If conflicts occur:
# 1. Resolve the conflict in the file
git add resolved-file.ts
git rebase --continue
# 2. Or skip this commit
git rebase --skip
# 3. Or abort entirely
git rebase --abort
```

### The Golden Rule of Rebasing

**Never rebase commits that exist on a shared remote branch.**

Rebase creates new commits (different SHAs) from existing ones. If you rebase commits that others have already pulled, their local histories contain the old commits while the remote now has new ones. When they try to push or pull, Git sees the divergent histories and creates a mess of duplicate commits or requires a force push that destroys their work.

```bash
# Safe: rebasing a local feature branch before merging
git switch feature/my-work
git rebase main
git switch main
git merge feature/my-work  # fast-forward

# DANGEROUS: rebasing a branch that teammates have pulled
git rebase main  # on a shared branch like 'develop'
git push --force  # overwrites the remote's history
```

The rule: if you `push`, do not `rebase`. Use merge instead for shared branches.

### Interactive Rebase

Interactive rebase (`-i`) lets you rewrite history by reordering, editing, squashing, or dropping commits. It opens an editor listing commits oldest-first, each with an action word you can change.

```bash
# Interactively rebase the last 4 commits
git rebase -i HEAD~4

# Or rebase everything since the branch diverged from main
git rebase -i main
```

The editor opens:

```
pick d4e5f6a fix: add null check in getUser
pick 7b8c9d0 WIP: half-done refactor
pick 1a2b3c4 fixup: forgot to save file
pick 5e6f7a8 feat: extract user validation

# Commands:
# p, pick   = use commit
# r, reword = use commit, but edit the commit message
# e, edit   = use commit, but stop for amending
# s, squash = use commit, meld into previous commit (keeps message)
# f, fixup  = like squash, but discard this commit's log message
# d, drop   = remove commit entirely
# x, exec   = run command after this commit
```

Change the actions and save:

```
pick d4e5f6a fix: add null check in getUser
squash 1a2b3c4 fixup: forgot to save file  <- squash into previous
pick 5e6f7a8 feat: extract user validation
reword 7b8c9d0 WIP: half-done refactor     <- edit this message
drop   ...                                 <- remove a commit
```

Common interactive rebase workflows:

```bash
# Fix a commit message from 3 commits ago
git rebase -i HEAD~3
# Change 'pick' to 'reword' on the target commit, save, edit message in next editor

# Squash last 5 WIP commits into one clean commit
git rebase -i HEAD~5
# Change all except first 'pick' to 'squash' or 'fixup'

# Add a forgotten file to a past commit
git add forgotten.ts
git commit --fixup=HEAD~2  # creates a commit named "fixup! <original message>"
git rebase -i --autosquash HEAD~4  # automatically places fixup! commits in the right position
```

## Resolving Merge Conflicts

A conflict occurs when the same region of a file was changed differently in both branches being merged. Git cannot decide which version is correct and asks you to resolve it manually.

### Reading Conflict Markers

```
<<<<<<< HEAD
// Our version (current branch)
function getUser(id: string): User {
  return db.findById(id);
}
=======
// Incoming version (branch being merged)
async function getUser(id: string): Promise<User | null> {
  const user = await db.findById(id);
  return user ?? null;
}
>>>>>>> feature/async-user-service
```

Three sections:
- `<<<<<<< HEAD` to `=======`: your current branch's version
- `=======` to `>>>>>>>`: the incoming branch's version
- You must choose one, combine both, or write something entirely new

### Resolving Manually

```bash
# See which files have conflicts
git status
# Both modified: src/user.ts

# Open the file, find conflict markers, edit to the desired result
# Delete ALL conflict markers (<<<, ===, >>>)
# Save the file

# Mark as resolved by staging
git add src/user.ts

# If more conflicts remain, repeat for each file, then:
git merge --continue
# Or if rebasing:
git rebase --continue
```

### Conflict Resolution Shortcuts

```bash
# Accept entirely "ours" (current branch) for a specific file
git checkout --ours src/generated-schema.ts
git add src/generated-schema.ts

# Accept entirely "theirs" (incoming branch) for a specific file
git checkout --theirs package-lock.json
git add package-lock.json
```

`--ours` and `--theirs` semantics differ during rebase: "ours" is the branch you are rebasing onto, "theirs" is the branch being rebased. This is the opposite of the intuitive expectation.

### Using a Merge Tool

```bash
# Open the configured merge tool for each conflicted file
git mergetool

# Configure a specific tool
git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait $MERGED'
```

VS Code's built-in merge editor shows four panels: incoming, current, base (common ancestor), and result. It lets you accept changes with buttons rather than manually editing markers.

### Preventing Conflicts

- **Pull frequently.** The longer a branch lives without syncing with main, the more divergence accumulates.
- **Keep branches short-lived.** A branch that exists for two days has far fewer conflicts than one that exists for two weeks.
- **Communicate about shared files.** If two people are editing the same file simultaneously, coordinate.
- **Use a formatter.** Whitespace and style differences generate conflicts with no semantic value. Enforce consistent formatting with Prettier/ESLint and a pre-commit hook.

## Cherry-Pick

Cherry-pick applies the changes from a specific commit onto the current branch, creating a new commit with those same changes. It copies a commit, not a branch.

```bash
# Apply a specific commit to the current branch
git cherry-pick abc1234

# Apply multiple commits
git cherry-pick abc1234 def5678

# Apply a range of commits (exclusive of first, inclusive of last)
git cherry-pick abc1234..def5678

# Apply without immediately committing (stage only)
git cherry-pick --no-commit abc1234
# Make further edits if needed
git commit -m "cherry-pick and adjust user fix"

# If a conflict occurs during cherry-pick:
# Resolve conflict, then:
git add resolved-file.ts
git cherry-pick --continue
# Or abort:
git cherry-pick --abort
```

### When to Use Cherry-Pick

- **Hotfixes:** A bug was fixed on `develop` and needs to go to `main` immediately without merging all of `develop`.
- **Selective backporting:** A new feature includes both a bug fix and new code; cherry-pick just the fix commit to a maintenance branch.
- **Recovering a lost commit:** A commit was on a deleted branch; find it in `reflog` and cherry-pick it.

```bash
# Hotfix workflow: fix was committed on develop as abc1234
git switch main
git cherry-pick abc1234  # Applies just that fix to main
git push origin main
```

### Cherry-Pick Pitfalls

Cherry-pick creates a **new commit** with a different SHA. If you later merge the original branch, Git may apply those changes twice, causing duplicate diffs. Prefer merging over cherry-picking when possible.

Avoid cherry-picking merge commits. Git cannot determine which parent to diff against without the `-m` flag:

```bash
# Cherry-picking a merge commit requires specifying the mainline parent
git cherry-pick -m 1 <merge-commit-sha>  # -m 1 = use the first parent
```

## Detached HEAD State

A detached HEAD occurs when HEAD points directly to a commit rather than to a branch. This happens when you checkout a specific commit, tag, or remote branch directly.

```bash
# These all create a detached HEAD:
git checkout abc1234         # checkout a specific commit
git checkout v2.1.0          # checkout a tag
git checkout origin/main     # checkout a remote tracking branch

git status
# HEAD detached at abc1234
```

In detached HEAD state you can:
- Look around and run tests
- Make experimental commits

But those commits are not on any branch. If you switch away without creating a branch, the commits become unreachable (orphaned). Git will warn you and eventually garbage-collect them.

```bash
# Experiment in detached HEAD
git checkout abc1234
# make changes, commit...

# Save your work by creating a branch BEFORE switching away
git switch -c experiment/from-old-commit

# Or if you already switched away and lost the commits:
git reflog  # Find the SHA of the commit you made
git branch recover-work <sha>
```

## Stash

`git stash` temporarily saves your working tree and staging area without committing, so you can switch context.

```bash
# Stash all tracked changes (staged and unstaged)
git stash

# Stash with a descriptive message
git stash push -m "half-finished login form validation"

# Also include untracked files
git stash push -u

# List all stashes
git stash list
# stash@{0}: On feature/login: half-finished login form validation
# stash@{1}: WIP on main: quick config experiment

# Apply the most recent stash (keeps it in the stash list)
git stash apply

# Apply and remove from the stash list
git stash pop

# Apply a specific stash
git stash apply stash@{2}

# Drop a specific stash
git stash drop stash@{1}

# Clear all stashes
git stash clear

# Create a branch from a stash
git stash branch new-feature stash@{0}
```

Stash is not a substitute for proper branching. For work in progress that you will return to, create a branch. Use stash for brief context switches (reviewing someone's PR, quick hotfix).

## Common Mistakes

- **Mistake 1: Rebasing shared branches.** If teammates have pulled from a branch and you rebase it, their local history diverges from the new remote history. The result is duplicate commits and force-push conflicts. Only rebase commits that exist solely on your local machine.

- **Mistake 2: Merging the wrong direction.** `git merge feature` while on `main` merges feature into main. `git merge main` while on `feature` merges main into feature (updates the feature branch). These are easy to transpose. Always confirm `git status` shows you are on the correct branch before merging.

- **Mistake 3: Forgetting to delete merged branches.** After merging, the branch pointer still exists. Old branches accumulate, clutter the branch list, and confuse `git branch -a` output. Delete merged branches promptly: `git branch -d` locally and `git push origin --delete` remotely.

- **Mistake 4: Losing work in detached HEAD.** Making commits in detached HEAD without immediately creating a branch is a common source of "lost" work. The commits exist in the reflog for 90 days, but only if you catch the situation quickly. Always create a branch from detached HEAD commits before switching away.

- **Mistake 5: Over-relying on cherry-pick.** Cherry-pick duplicates commits. If you cherry-pick a commit into main and then later merge the source branch, Git may apply the same changes twice, resulting in duplicate entries in the log and potential conflicts. Use merge when you need all commits from a branch; use cherry-pick only for surgical selection.

- **Mistake 6: Conflicts in generated files.** `package-lock.json`, `yarn.lock`, migration files, and auto-generated code produce conflicts that are impossible to resolve by reading the diff. Use `--ours` or `--theirs` on these files and regenerate them properly afterward.

## Best Practices

- **Prefer `git switch` over `git checkout` for branches.** `checkout` is overloaded (it also restores files). `switch` is explicit and harder to misuse.

- **Merge feature branches with `--no-ff` to preserve history.** A merge commit documents when a feature was integrated and makes the feature atomic: reversible with a single `git revert`.

- **Rebase before merging, not instead of merging.** Rebase your feature branch onto the latest main to resolve conflicts before creating the merge commit. This keeps the merge clean without giving up the merge commit's structural information.

- **Use `git log --oneline --graph --all` to visualize the branch state** before any merge or rebase operation. Understanding the current history prevents surprises.

- **Keep branches short-lived.** The longer a branch exists, the more divergence it accumulates and the harder merging becomes. Aim for branches that live hours to days, not weeks to months.

- **Name branches descriptively.** `feature/user-avatar-upload`, `fix/cart-quantity-negative`, `chore/upgrade-node-22` are infinitely more useful than `feature1`, `johns-branch`, `temp`.

## Real-World Use Cases

### 1. Feature Branch Workflow

```bash
# Start from an up-to-date main
git switch main
git pull origin main

# Create a feature branch
git switch -c feature/order-tracking

# Work, commit frequently
git add src/order-tracking/
git commit -m "feat(orders): add tracking number display"

git add src/order-tracking/map.component.ts
git commit -m "feat(orders): integrate map view for shipping status"

# Before merging, sync with main to resolve any divergence
git fetch origin
git rebase origin/main  # or: git merge origin/main

# Resolve any conflicts, then merge into main
git switch main
git merge --no-ff feature/order-tracking -m "feat: order tracking (#142)"

# Delete the feature branch
git branch -d feature/order-tracking
git push origin --delete feature/order-tracking
```

### 2. Hotfix on a Production Tag

A critical bug is on production (tagged `v3.2.1`). The fix must go to production immediately, not wait for the next planned release:

```bash
# Create a hotfix branch from the production tag
git switch -c hotfix/payment-null-pointer v3.2.1

# Fix the bug
git add src/payment/processor.ts
git commit -m "fix(payment): handle null payment method gracefully"

# Merge to main (or release branch)
git switch main
git merge --no-ff hotfix/payment-null-pointer

# Also merge to develop so it doesn't regress
git switch develop
git merge --no-ff hotfix/payment-null-pointer

# Tag the new production release
git switch main
git tag -a v3.2.2 -m "hotfix: payment null pointer (CVE-2025-1234)"

# Push everything
git push origin main develop --tags

# Clean up
git branch -d hotfix/payment-null-pointer
```

### 3. Cleaning Up a Messy Branch with Interactive Rebase

Before creating a PR, a developer cleans up their noisy commit history:

```bash
# Current branch history (last 6 commits)
git log --oneline
# a1b2c3d WIP
# b2c3d4e fix typo
# c3d4e5f more WIP
# d4e5f6a actually working now
# e5f6a7b forgot to add test file
# f6a7b8c feat: add product search with filters

# Interactively rebase the last 6 commits
git rebase -i HEAD~6

# In the editor, reorder and combine:
# pick f6a7b8c feat: add product search with filters
# squash e5f6a7b forgot to add test file
# squash d4e5f6a actually working now
# squash c3d4e5f more WIP
# squash b2c3d4e fix typo
# squash a1b2c3d WIP

# Git opens another editor for the final squashed message:
# feat(search): add product search with filters
#
# - Supports filtering by category, price range, and rating
# - Results are paginated with cursor-based pagination
# - Debounced input prevents excessive API calls

# Result: one clean, atomic commit ready for review
git log --oneline
# g7h8i9j feat(search): add product search with filters
```

---

> **Key Takeaway:** Branches are just pointers to commits -- cheap to create and delete; merge preserves history while rebase creates a linear history, and the golden rule is to never rebase commits that have already been shared with others.
