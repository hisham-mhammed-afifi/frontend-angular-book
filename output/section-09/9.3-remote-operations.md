# 9.3 Remote Operations

> A remote is a version of your repository hosted elsewhere; remote operations synchronize your local history with that shared source and form the backbone of all team collaboration.

---

## How Remotes Work

A remote is a named reference to another copy of your repository, stored at a URL. Git stores remote references in `.git/config`. When you clone a repository, Git automatically creates a remote named `origin` pointing to the URL you cloned from.

```
Local Repository          Remote Repository (GitHub, GitLab, etc.)
.git/refs/heads/main  ->  (push/fetch)  ->  refs/heads/main
.git/refs/remotes/    <-  (fetch)       <-  remote tracking branches
  origin/main
  origin/develop
  origin/feature/x
```

Remote tracking branches (e.g., `origin/main`) are read-only local snapshots of the remote's branches. They are updated by `git fetch` and `git pull`. You never commit directly to them.

---

## Managing Remotes

```bash
# List all configured remotes
git remote -v
# origin  https://github.com/user/repo.git (fetch)
# origin  https://github.com/user/repo.git (push)

# Add a new remote
git remote add upstream https://github.com/original/repo.git

# Change a remote's URL (e.g., switching from HTTPS to SSH)
git remote set-url origin git@github.com:user/repo.git

# Rename a remote
git remote rename origin old-origin

# Remove a remote
git remote remove upstream

# Show detailed info about a remote (branches, tracking status)
git remote show origin
```

### Multiple Remotes

Multiple remotes are common when:
- Working on a fork (`origin` = your fork, `upstream` = the original project)
- Deploying to multiple environments (a separate `heroku` or `production` remote)
- Mirroring repositories

```bash
# Fork workflow setup
git remote add upstream https://github.com/original/project.git

# Sync your fork with upstream changes
git fetch upstream
git switch main
git merge upstream/main   # or: git rebase upstream/main
git push origin main       # push updated main to your fork
```

## git fetch

`git fetch` downloads commits, branches, and tags from a remote into your local remote tracking branches **without modifying your working tree or local branches**. It is always safe to run.

```bash
# Fetch all branches from origin
git fetch origin

# Fetch a specific branch
git fetch origin feature/new-api

# Fetch from all remotes
git fetch --all

# Fetch and delete local tracking branches for remote branches that no longer exist
git fetch --prune
git fetch -p  # shorthand

# Fetch all remotes and prune
git fetch --all --prune

# Fetch tags
git fetch --tags
```

After fetching, you can inspect what changed before integrating:

```bash
# See what commits arrived since last fetch
git log origin/main..main       # commits on local main not on remote
git log main..origin/main       # commits on remote not yet on local

# Compare working tree to remote
git diff main origin/main

# See all commits that came in
git log --oneline origin/main
```

This is the key advantage of fetch over pull: you can review incoming changes before deciding how to integrate them.

---

## git pull

`git pull` is `git fetch` followed immediately by either `git merge` or `git rebase`. It updates your current branch with remote changes in one step.

```bash
# Pull (fetch + merge) from the tracked remote branch
git pull

# Pull with rebase instead of merge (creates linear history)
git pull --rebase

# Pull from a specific remote and branch
git pull origin main

# Pull with rebase from a specific branch
git pull --rebase origin main
```

### pull.rebase Configuration

Setting `pull.rebase` globally makes `git pull` always rebase instead of merge, keeping a linear history:

```bash
# Prefer rebase when pulling (recommended for feature branches)
git config --global pull.rebase true

# Use merge when pulling (Git's traditional default)
git config --global pull.rebase false

# Use fast-forward only -- fail if a merge commit would be needed
git config --global pull.rebase false
git config --global pull.ff only
```

`pull.ff only` is the strictest setting: `git pull` will only succeed if it can fast-forward (no merge commit). If the branches have diverged, it fails with an error, forcing you to explicitly choose between merge and rebase. This prevents accidental merge commits on feature branches.

### fetch vs pull: When to Use Each

| Scenario | Use |
|---------|-----|
| Checking if the remote has new work | `git fetch` then inspect |
| Updating main from the remote after a review | `git pull` (or `pull --rebase`) |
| Before rebasing your feature branch onto main | `git fetch`, then `git rebase origin/main` |
| Shared branch where others push frequently | `git fetch --prune` then review, then merge |
| Automated scripts / CI | `git fetch` -- safer, no implicit merge |

The general rule: **fetch is always safe; pull is convenient but has side effects**. In team environments, prefer `fetch` + explicit `merge` or `rebase` so you are in control of when and how integration happens.

## git push

`git push` uploads your local branch commits to the remote.

```bash
# Push current branch to its tracked remote branch
git push

# Push a specific branch to a specific remote
git push origin feature/new-login

# Push and set the upstream tracking relationship in one step
git push -u origin feature/new-login
# After this, plain 'git push' works for this branch

# Push all local branches
git push --all origin

# Push tags
git push --tags

# Push a specific tag
git push origin v2.1.0
```

### Setting Upstream Automatically

Configure Git to always set the upstream on first push, so you never need `-u` again:

```bash
git config --global push.autoSetupRemote true
# Now 'git push' on a new branch automatically sets --set-upstream
```

### push.default

Controls which branches `git push` (without arguments) pushes:

```bash
# Only push the current branch to its tracked upstream (recommended)
git config --global push.default current

# Push only when the local and remote branch names match exactly
git config --global push.default simple  # Git's current default

# Push all branches that have matching remote branches
git config --global push.default matching  # old default, avoid this
```

Use `push.default current` in combination with `push.autoSetupRemote true` for the most friction-free experience.

## Force Pushing Safely

Force pushing rewrites the remote's history. It is necessary after rebasing or amending a pushed commit, but dangerous when done carelessly.

### `--force` vs `--force-with-lease`

```bash
# DANGEROUS: overwrites remote branch unconditionally
# If a teammate pushed since your last fetch, their commits are gone
git push --force origin feature/my-work

# SAFER: only force-pushes if the remote hasn't changed since your last fetch
# If someone else pushed in the meantime, the push fails with an error
git push --force-with-lease origin feature/my-work

# SAFEST: also verifies you have integrated any auto-fetched remote changes
git push --force-with-lease --force-if-includes origin feature/my-work
```

`--force-with-lease` checks that the remote tracking branch (`origin/feature/my-work`) matches what is on the remote server. If they differ, it refuses to push and asks you to fetch first.

**The weakness:** if your editor or IDE auto-fetches in the background, it silently updates `origin/feature/my-work` to match the remote. Then `--force-with-lease` passes even though you haven't reviewed the remote changes. `--force-if-includes` closes this loophole.

### When Force Pushing Is Acceptable

- After rebasing a personal feature branch that has not yet been reviewed
- After amending the most recent commit on a personal branch
- After `git rebase -i` cleanup on a private branch

### When Force Pushing Is Never Acceptable

- On `main`, `master`, `develop`, or any shared integration branch
- On any branch other people are actively using
- Without communicating with the team first

### Protecting Branches from Force Push

On GitHub/GitLab, enable branch protection rules on `main`:
- Require pull request reviews before merging
- Require status checks to pass
- Disallow force pushes (even for admins)

This is a required configuration for any team project.

## Tracking Branches and Upstream

A tracking relationship links your local branch to a remote branch. Once established, `git push` and `git pull` without arguments use the tracked remote.

```bash
# See tracking relationships for all branches
git branch -vv
# * main           abc1234 [origin/main] Latest commit message
#   feature/login  def5678 [origin/feature/login: ahead 2] My commit

# The output shows:
# [origin/main]              = tracking remote, in sync
# [origin/feature/login: ahead 2]  = tracking remote, 2 commits ahead (need to push)
# [origin/feature: behind 3]       = 3 remote commits not yet pulled
# (no tracking info shown)          = no upstream set

# Set an upstream tracking branch
git branch --set-upstream-to=origin/main main
# Or shorthand:
git branch -u origin/main main

# Remove the upstream tracking
git branch --unset-upstream

# Check the upstream of the current branch
git rev-parse --abbrev-ref --symbolic-full-name @{upstream}
```

### Remote Tracking Branches

Remote tracking branches live in `.git/refs/remotes/`. They are read-only snapshots updated by fetch:

```bash
# List remote tracking branches
git branch -r
# origin/main
# origin/develop
# origin/feature/user-profile

# Check out a remote branch locally (creates a local tracking branch)
git switch --track origin/feature/user-profile
# Creates: local branch 'feature/user-profile' tracking 'origin/feature/user-profile'

# Or explicitly:
git switch -c feature/user-profile origin/feature/user-profile

# Delete a remote branch (removes it from the remote server)
git push origin --delete feature/old-work

# Delete stale remote tracking references (after remote branches are deleted)
git remote prune origin
# Or during fetch:
git fetch --prune
```

## Cloning Options

```bash
# Standard clone
git clone https://github.com/user/repo.git

# Clone into a specific directory
git clone https://github.com/user/repo.git my-dir

# Clone a specific branch
git clone --branch develop https://github.com/user/repo.git

# Shallow clone: only the last N commits (faster, less disk)
git clone --depth 1 https://github.com/user/repo.git
# Useful for CI/CD pipelines that don't need full history

# Convert a shallow clone to a full clone
git fetch --unshallow

# Clone without downloading large objects (partial clone)
git clone --filter=blob:none https://github.com/user/repo.git
# Objects are downloaded on demand as you checkout files
```

### Shallow Clone Gotchas

Shallow clones are missing history. Operations that need history (`git blame`, `git log` on old commits, `git bisect`) may fail or give incomplete results. For CI/CD, a depth of 1 is usually fine. For development work, use a full clone.

## Tags: Versioning Releases

Tags mark specific commits, usually for releases. Unlike branches, tags do not move when new commits are made.

```bash
# Create a lightweight tag (just a pointer to a commit)
git tag v2.1.0

# Create an annotated tag (includes author, date, message -- recommended for releases)
git tag -a v2.1.0 -m "Release version 2.1.0 - adds user avatar upload"

# Tag a specific past commit
git tag -a v2.0.5 abc1234 -m "Hotfix backport"

# List all tags
git tag
git tag -l "v2.*"  # filter by pattern

# Show tag details
git show v2.1.0

# Push a specific tag to remote
git push origin v2.1.0

# Push all tags
git push origin --tags

# Delete a local tag
git tag -d v2.1.0-rc1

# Delete a remote tag
git push origin --delete v2.1.0-rc1

# Check out a tag (enters detached HEAD state)
git checkout v2.1.0
# Better: create a branch from a tag
git switch -c release/v2.1.0 v2.1.0
```

### Semantic Versioning with Tags

```bash
# Annotated tags for production releases
git tag -a v1.0.0 -m "Initial stable release"
git tag -a v1.1.0 -m "Add user authentication"
git tag -a v1.1.1 -m "Fix session expiry bug"
git tag -a v2.0.0 -m "Breaking: new API response shape"

# Pre-release tags
git tag -a v2.0.0-alpha.1 -m "Alpha 1 of v2.0"
git tag -a v2.0.0-rc.1 -m "Release candidate 1"

# Push all tags with commits
git push origin main --tags
```

## Worktrees

`git worktree` lets you check out multiple branches simultaneously in separate directories, all sharing the same `.git` repository. This avoids stashing and switching when you need to work on two things at once.

```bash
# Create a new worktree for a branch
git worktree add ../hotfix-branch hotfix/payment-crash

# Create a worktree and a new branch
git worktree add -b feature/experiment ../experiment-dir main

# List all worktrees
git worktree list
# /Users/alice/project        abc1234 [main]
# /Users/alice/hotfix-branch  def5678 [hotfix/payment-crash]

# Remove a worktree (after you are done)
git worktree remove ../hotfix-branch

# Prune stale worktree references
git worktree prune
```

Worktrees are ideal for:
- Reviewing a PR while continuing work on your current feature
- Running tests on two branches simultaneously
- Hot-fixing production while a long-running feature branch is in progress

## Common Mistakes

- **Mistake 1: Using `git push --force` on shared branches.** A plain force push to `main` or `develop` overwrites any commits teammates have pushed since your last fetch, permanently destroying their work. Always use `--force-with-lease` on personal branches, and never force push to shared branches.

- **Mistake 2: Pulling without understanding the merge strategy.** `git pull` by default creates a merge commit when local and remote have diverged. On a feature branch, this litters the history with "Merge branch 'main' into feature/x" commits. Use `git pull --rebase` or set `pull.rebase true` globally.

- **Mistake 3: Pushing to `main` directly.** All commits to shared branches should go through pull requests (or merge requests). Direct pushes bypass code review, CI, and branch protection. Enable branch protection rules on the remote to prevent this.

- **Mistake 4: Not pruning stale remote tracking branches.** When remote branches are deleted (after PRs are merged), their local tracking references (`origin/feature/old`) remain until you prune. Over time, `git branch -r` becomes cluttered with dead references. Run `git fetch --prune` regularly or set `remote.origin.prune = true` in config.

- **Mistake 5: Using lightweight tags for releases.** Lightweight tags are just pointers -- they have no author, date, or message. Annotated tags are full Git objects with that metadata. For anything version-controlled (releases, changelogs, semantic versioning), always use `git tag -a`.

- **Mistake 6: Shallow cloning in development.** `--depth 1` speeds up CI but breaks `git blame`, `git bisect`, `git log` on older commits, and `git describe`. Only use shallow clones in CI pipelines where full history is not needed.

---

## Best Practices

- **Fetch frequently, pull deliberately.** Run `git fetch --prune` often to keep remote tracking branches current. Integrate (pull/merge/rebase) only when you are ready to deal with any conflicts.

- **Always set `push.autoSetupRemote true`.** This eliminates the "fatal: The current branch has no upstream branch" error on first push of a new branch and removes the need to remember `-u`.

- **Use `--force-with-lease` instead of `--force`.** Create a git alias: `git config --global alias.fpush "push --force-with-lease"` so `git fpush` is always the safe version.

- **Delete branches after merging.** Both locally (`git branch -d`) and on the remote (`git push origin --delete`). Merged branches are clutter.

- **Set branch protection on `main` on day one.** Require PRs, status checks, and disallow force pushes. This cannot be set up "later" -- by then, bad habits have formed.

- **Use annotated tags for every release.** Annotated tags appear in `git describe`, show up differently in GitHub's release UI, and carry meaningful metadata. Never use lightweight tags for releases.

## Real-World Use Cases

### 1. Keeping a Fork in Sync with Upstream

When contributing to an open source project, your fork quickly diverges from the original. Keep it synchronized:

```bash
# One-time setup after forking
git remote add upstream https://github.com/original/project.git
git remote -v
# origin   https://github.com/yourname/project.git (fetch)
# upstream https://github.com/original/project.git (fetch)

# Daily sync workflow
git fetch upstream                    # Download upstream changes
git switch main                       # Switch to local main

# Option A: merge (preserves merge commit showing sync point)
git merge upstream/main

# Option B: rebase (linear history)
git rebase upstream/main

git push origin main                  # Update your fork's main
git fetch --prune upstream            # Remove stale tracking branches
```

### 2. CI/CD Optimized Clone

CI pipelines need speed. Use partial and shallow cloning:

```bash
# GitHub Actions example
# actions/checkout@v4 handles this, but here's the raw git equivalent:

# Shallow clone for fastest CI (only last commit)
git clone --depth 1 --branch main https://github.com/user/repo.git

# For pipelines that need tag history (semantic release, changelog generation)
git clone --depth 50 --branch main https://github.com/user/repo.git
git fetch --tags  # Fetch all tags without fetching all history

# For monorepos where you only need a subdirectory
git clone --filter=blob:none --sparse https://github.com/user/monorepo.git
git sparse-checkout set packages/my-app
```

### 3. Hotfix with Worktrees

A critical bug is discovered in production while a long-running feature is in progress. Using worktrees avoids stashing and context switching:

```bash
# Current work on feature branch -- do NOT stash
# Working directory has unstaged changes

# Create a hotfix worktree without disturbing the current directory
git worktree add ../hotfix-dir -b hotfix/payment-null main

# Open the hotfix in a new terminal/editor session
cd ../hotfix-dir
code .

# Fix the bug, commit, push
git add src/payment/
git commit -m "fix(payment): handle null card token gracefully"
git push -u origin hotfix/payment-null

# Create PR from hotfix branch via GitHub CLI
gh pr create --title "fix: payment null token crash" --base main

# After PR is merged, clean up
cd ../project                          # Back to your feature work
git worktree remove ../hotfix-dir      # Remove the worktree
git branch -d hotfix/payment-null      # Remove local branch
git push origin --delete hotfix/payment-null  # Remove remote branch
git fetch --prune                      # Clean up tracking refs
```

### 4. Release Tagging Workflow

After merging a release branch or a release PR:

```bash
# Ensure you are on the latest main
git switch main
git pull origin main

# Verify the state before tagging
git log --oneline -5
git diff HEAD~1 HEAD --stat

# Create an annotated release tag
git tag -a v3.0.0 -m "$(cat <<'EOF'
Release v3.0.0

Changes:
- feat: user avatar upload with S3 integration
- feat: real-time notifications via WebSocket
- fix: session timeout on mobile browsers
- BREAKING: API response format now uses camelCase

Migration guide: docs/migration/v2-to-v3.md
EOF
)"

# Verify the tag
git show v3.0.0

# Push the tag
git push origin v3.0.0

# GitHub automatically creates a release from an annotated tag
# (or use: gh release create v3.0.0 --notes-from-tag)
```

---

> **Key Takeaway:** `git fetch` is always safe -- it only updates remote tracking references; `git pull` is `fetch + merge/rebase` and changes your local branch; and `git push --force-with-lease` is the only acceptable form of force push on non-shared branches.
