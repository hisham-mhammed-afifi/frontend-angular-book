# 9.6 Branching Strategies

> A branching strategy is a set of rules for how branches are created, named, merged, and deleted -- it determines your team's entire development and release rhythm.

---

## Why Branching Strategy Matters

The way you organize branches has cascading effects on:
- How fast code moves from developer to production
- How often merge conflicts occur and how bad they are
- Whether your `main` branch is always deployable
- How you manage hotfixes without disrupting in-progress work
- Whether CI/CD is easy or impossible to implement

There is no universally correct strategy. The right one depends on your team size, release frequency, and product type.

---

## Strategy 1: GitFlow

GitFlow was introduced by Vincent Driessen in 2010 and became the dominant strategy for versioned software. It defines a strict branching model with specific roles for each branch type.

### The branch structure

```
main        -- production-ready code only. Tagged with version numbers.
develop     -- integration branch for completed features
feature/*   -- individual features, branched from develop
release/*   -- release stabilization, branched from develop
hotfix/*    -- urgent production fixes, branched from main
```

### The develop branch

`develop` is the primary working branch. Features are merged into it continuously. It represents the "next release" state.

```
main:     A-----------B-----------C
           \           \           \
develop:    D---E---F---G---H---I---J
                |           |
feature/*:     (work)       (work)
```

### Feature branch lifecycle

```bash
# Start a feature from develop
git switch develop
git pull
git switch -c feature/user-notifications

# Work, commit
git commit -m "feat(notifications): add email notification service"
git commit -m "feat(notifications): add in-app notification UI"

# Merge back into develop when complete
git switch develop
git pull
git merge feature/user-notifications --no-ff
git push origin develop

# Delete the feature branch
git branch -d feature/user-notifications
git push origin --delete feature/user-notifications
```

### Release branch lifecycle

When `develop` has all features for the next release, cut a release branch. This freezes feature development for the release and allows only bug fixes.

```bash
# Cut a release branch from develop
git switch develop
git pull
git switch -c release/2.4.0

# Only bug fixes go here -- no new features
git commit -m "fix: correct email template encoding"
git commit -m "fix: update version number to 2.4.0"

# When ready, merge into BOTH main and develop
git switch main
git merge release/2.4.0 --no-ff
git tag -a v2.4.0 -m "Release 2.4.0"
git push origin main --tags

git switch develop
git merge release/2.4.0 --no-ff
git push origin develop

# Delete the release branch
git branch -d release/2.4.0
git push origin --delete release/2.4.0
```

The release branch protects `develop` from being frozen. While `release/2.4.0` stabilizes, work on features for `2.5.0` continues in `develop`.

### Hotfix branch lifecycle

A critical bug is discovered in production. You cannot take features from `develop` -- you need to fix only the production version.

```bash
# Branch from main (production), not develop
git switch main
git pull
git switch -c hotfix/fix-payment-crash

# Apply the minimal fix
git commit -m "fix(payments): prevent crash when card expiry is nil"

# Merge into BOTH main and develop
git switch main
git merge hotfix/fix-payment-crash --no-ff
git tag -a v2.3.1 -m "Hotfix 2.3.1: payment crash"
git push origin main --tags

git switch develop
git merge hotfix/fix-payment-crash --no-ff
git push origin develop

# If a release branch exists, merge there too
git switch release/2.4.0
git merge hotfix/fix-payment-crash --no-ff
git push origin release/2.4.0

# Delete hotfix branch
git branch -d hotfix/fix-payment-crash
git push origin --delete hotfix/fix-payment-crash
```

### GitFlow diagram (complete)

```
main:     A-----------B(v1.0)-----------C(v1.1)-----------D(v2.0)
           \           |                  |                  |
hotfix:     |           `--hotfix/1.0.1--'                  |
            |                                               |
develop:    E---F---G---H---I---J---K---L---M---N---O---P---Q
                |       |           |           |
feature:       (A-feat) (B-feat)   (C-feat)    (D-feat)
                                |
release:                    release/2.0--bug-fix--bug-fix
                                                          |
                                            merged -> main (v2.0) + develop
```

### GitFlow: when to use it

- **Versioned software** with multiple live versions (v1.x, v2.x)
- **Scheduled, infrequent releases** (monthly, quarterly)
- **Large teams** with separate release management responsibilities
- **Compliance-heavy environments** requiring sign-off before any production change

### GitFlow: drawbacks

- **Long-lived branches accumulate conflicts.** A feature branch open for two weeks will have significant divergence from `develop` and from other features.
- **Slow feedback loop.** Features sit in `develop` for weeks before reaching users.
- **`develop` is not deployable** until the release branch finalizes it. You cannot deploy `develop` directly.
- **Double merging.** Hotfixes and releases must be merged into both `main` and `develop`, creating opportunities for mistakes.
- **Not compatible with continuous delivery.** You cannot deploy multiple times per day with GitFlow.

## Strategy 2: GitHub Flow

GitHub Flow was described by Scott Chacon in 2011 as a simpler alternative to GitFlow. It assumes that `main` is always deployable and that you deploy continuously.

### The branch structure

```
main        -- always deployable, represents production
feature/*   -- any branch for any change (feature, fix, chore)
```

That is all. There is no `develop`, no `release/*`, no `hotfix/*`.

### The workflow

```bash
# 1. Create a branch from main
git switch main
git pull
git switch -c feature/dark-mode

# 2. Make commits
git commit -m "feat: add dark mode CSS variables"
git commit -m "feat: implement dark mode toggle"

# 3. Open a pull request early (use draft PR)
git push origin feature/dark-mode
gh pr create --draft --title "feat: dark mode" --body "WIP"

# 4. Get review, make changes, mark ready
gh pr ready

# 5. Merge to main after approval and CI passes
gh pr merge --squash

# 6. Deploy main to production immediately after merge
# (automated by CI/CD pipeline)
```

### GitHub Flow diagram

```
main:       A---B---C---D---E---F (deploys happen at each point)
                |           |
feature:       (dark-mode)  (search-improvements)
```

No `develop` branch. Features merge directly to `main`. Every merge triggers a deployment.

### GitHub Flow: when to use it

- **Web applications** that deploy continuously (SaaS, web products)
- **Small to medium teams** (2-20 developers)
- **Single version** in production (no need to support v1.x and v2.x simultaneously)
- **Fast iteration** is more valuable than structured release gates

### GitHub Flow: drawbacks

- **No release staging.** There is no buffer between a merged feature and production. If your deployment pipeline is not fast and reliable, merges cause unpredictable production changes.
- **Hotfix complexity.** Since there are no hotfix branches, a production fix is just another feature branch. This is fine operationally, but the process looks identical to any other change.
- **Feature flags required for large features.** If a feature takes weeks to build, you either need a long-lived branch (bad) or feature flags to hide incomplete work in `main` (better, but adds complexity).

## Strategy 3: Trunk-Based Development

Trunk-Based Development (TBD) takes GitHub Flow further. In its strictest form, all developers commit directly to `main` (the trunk) multiple times per day, with no feature branches at all. In practice, most teams use very short-lived feature branches (1-2 days maximum).

### The branch structure

```
main (trunk) -- always green, always deployable
               -- feature flags control what users see
feature/*    -- maximum lifetime: 1-2 days
```

### The workflow

```bash
# Option A: commit directly to main (for very small changes)
git switch main
git pull
git commit -m "fix: correct pluralization in error message"
git push origin main
# CI runs, deploys automatically if green

# Option B: short-lived feature branch (for any change > 1 hour)
git switch main
git pull
git switch -c feat/add-search-bar

# Work quickly -- this branch must merge within 2 days
git commit -m "feat: add search bar component"
git commit -m "feat: wire search bar to API"

# Open PR, get rapid review (hours, not days)
git push origin feat/add-search-bar
gh pr create --title "feat: add search bar"

# Merge (squash or rebase-merge) immediately after approval
gh pr merge --squash
git branch -d feat/add-search-bar
```

### Feature flags: the key enabler

TBD requires deploying incomplete features hidden behind flags. Without feature flags, you cannot merge an unfinished 3-week feature into main without exposing it to users.

```typescript
// Simple feature flag check
function renderDashboard(user: User) {
  if (featureFlags.isEnabled('new-dashboard', user)) {
    return <NewDashboard />;
  }
  return <LegacyDashboard />;
}
```

The flag is toggled externally (via a config file, environment variable, or a service like LaunchDarkly). The code lives in `main` but is hidden from most users until the feature is complete.

Types of feature flags:

| Type | Lifetime | Purpose |
|------|----------|---------|
| Release toggle | Days to weeks | Hide incomplete work in production |
| Experiment toggle | Weeks to months | A/B testing |
| Ops toggle | Indefinite | Kill switch for risky functionality |
| Permission toggle | Indefinite | Feature access by user tier/role |

### Continuous integration requirements

TBD only works if your CI pipeline is fast and reliable:

```
Push to main
  --> CI triggers automatically
  --> All tests run (unit, integration, e2e)
  --> Build succeeds
  --> Deploy to staging
  --> Smoke tests pass
  --> Deploy to production (automated)
Total time: ideally under 10 minutes
```

If CI is slow (30+ minutes), developers stop waiting for it and either push over failures or accumulate changes into big batches -- defeating the purpose of TBD.

### TBD diagram

```
main:   A-B-C-D-E-F-G-H-I-J-K-L-M-N (many small commits per day)
                |   |           |
short branches: |   +--feat--+  +--fix--+
                |            |          |
                (merged      (merged    (merged
                same day)    next day)  same day)
```

### TBD: when to use it

- **High-performing engineering teams** with strong test coverage and fast CI
- **Web products** where continuous deployment is the norm
- **Feature development** can be broken into daily-mergeable increments
- **Teams with code review culture** that can review small PRs within hours

### TBD: drawbacks

- **Discipline required.** A single developer who leaves a branch open for a week undermines the whole model.
- **Feature flags are overhead.** Someone must manage, test, and eventually remove every feature flag.
- **Test coverage must be high.** You cannot merge to main multiple times daily without confidence that your tests catch regressions.
- **Code review speed must match commit speed.** Slow reviewers create a bottleneck.

## Strategy 4: GitLab Flow

GitLab Flow is a pragmatic middle ground between GitHub Flow and GitFlow. It adds environment branches to GitHub Flow.

```
main (development)
  |
  v
pre-production (staging)
  |
  v
production
```

Changes flow downstream only. You never cherry-pick from `production` back to `main`.

```bash
# Feature merges into main (same as GitHub Flow)
git merge feature/new-feature into main

# Periodically, main is merged forward to pre-production
git switch pre-production
git merge main

# After QA passes, pre-production merges to production
git switch production
git merge pre-production
```

This model is useful when you have distinct staging and production environments that diverge intentionally (different configs, different data), and you want the branch structure to mirror the environment pipeline.

## Choosing a Strategy

| Factor | GitFlow | GitHub Flow | Trunk-Based |
|--------|---------|-------------|-------------|
| Release frequency | Weekly/monthly | Daily/continuous | Multiple times daily |
| Multiple live versions | Yes | No | No |
| Team size | Large | Small-medium | Any (with discipline) |
| CI/CD maturity required | Low | Medium | High |
| Feature flag usage | Rare | Occasional | Required |
| Merge conflict risk | High | Medium | Low |
| Onboarding complexity | High | Low | Medium |
| Hotfix process | Dedicated branch | Regular PR | Regular PR |

### Decision guide

```
Do you need to support multiple live versions?
  YES -> GitFlow (or a simplified variant)
  NO  -> continue

Do you deploy continuously (multiple times per week)?
  NO  -> GitHub Flow
  YES -> continue

Do you have high test coverage and fast CI (< 15 min)?
  NO  -> GitHub Flow until you improve CI
  YES -> Trunk-Based Development
```

## Naming Conventions

Consistent branch names make automation easier and the repository easier to navigate.

```bash
# Feature branches
feature/add-dark-mode
feat/JIRA-1234-user-notifications
feature/search-improvements

# Bug fix branches
fix/payment-crash-on-nil-card
bugfix/login-redirect-loop

# Hotfix branches (GitFlow only)
hotfix/v2.3.1-payment-crash

# Release branches (GitFlow only)
release/2.4.0
release/2024-03

# Chore / infrastructure branches
chore/upgrade-node-18
refactor/extract-auth-service
ci/add-playwright-tests
docs/update-api-readme

# Personal/experimental (deleted before PR)
wip/alice/dark-mode-experiment
experiment/new-search-algorithm
```

Automate enforcement with a pre-receive hook or branch protection naming pattern:

```bash
# Example: reject branches that don't match naming pattern
# .git/hooks/pre-push
branch=$(git symbolic-ref HEAD | sed 's|refs/heads/||')
pattern="^(feature|feat|fix|bugfix|hotfix|release|chore|refactor|ci|docs|wip|experiment)/"

if ! echo "$branch" | grep -qE "$pattern"; then
  echo "ERROR: Branch name '$branch' does not match naming convention."
  echo "Use: feature/, fix/, hotfix/, release/, chore/, refactor/, etc."
  exit 1
fi
```

## Environment Branches and Deployment Pipelines

Regardless of the branching strategy, most teams need a way to map branches to deployment environments.

### Common environment mapping

```
Strategy A (GitFlow style):
  develop   -> deploys to development environment automatically
  release/* -> deploys to staging environment automatically
  main      -> deploys to production manually or after approval

Strategy B (GitHub Flow / TBD style):
  main      -> deploys to staging automatically after merge
  main      -> deploys to production after manual approval (or automatically)

Strategy C (GitLab Flow style):
  main          -> development
  pre-production -> staging (promoted by merging main -> pre-production)
  production     -> production (promoted by merging pre-production -> production)
```

### CI/CD integration example

```yaml
# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]
  workflow_dispatch:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test

  deploy-staging:
    needs: test
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - run: ./scripts/deploy.sh staging

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production  # requires manual approval in GitHub
    steps:
      - uses: actions/checkout@v4
      - run: ./scripts/deploy.sh production
```

## Common Mistakes

- **Choosing GitFlow because it sounds "professional":** GitFlow adds significant process overhead. If your team deploys a web app multiple times a week, GitFlow will slow you down without providing benefits. Use the simplest strategy that meets your actual requirements.

- **Long-lived feature branches regardless of strategy:** No strategy benefits from branches that live weeks or months. The longer a branch lives, the worse the eventual merge conflict. In GitFlow, features should still merge to `develop` within days, not weeks.

- **Merging without deleteing the branch:** Branches that have been merged should be deleted. Hundreds of stale branches make the repository hard to navigate and CI lists cluttered.

- **Not merging hotfixes back to develop in GitFlow:** A hotfix applied only to `main` will reappear as a bug the next time `develop` is released. Always apply hotfixes to both `main` and `develop`.

- **Using TBD without feature flags:** Incomplete code in `main` without a flag hiding it exposes users to broken features. "We will just be careful" is not a strategy -- flag everything that is not done.

- **Inconsistent branch naming:** Different developers using `feature/`, `feat/`, `features/`, and no prefix at all makes automation impossible and the branch list unreadable.

- **Conflating deploy and release:** In trunk-based development especially, "deploying" means putting code in production infrastructure. "Releasing" means making a feature visible to users. These are separate events. Feature flags are what separate them.

---

## Best Practices

- **Always keep `main` deployable.** Regardless of strategy, if `main` can break, you lose the ability to do emergency releases. Protected branches and required CI are the guardrails.

- **Delete merged branches immediately.** Configure your merge button to delete branches automatically. A repository with 5 active branches is manageable. One with 300 is not.

- **Match your strategy to your release frequency.** If you deploy weekly, GitHub Flow is probably enough. If you deploy monthly to versioned software, GitFlow's ceremony is justified.

- **Never work directly on shared branches.** Even in TBD, the discipline is to merge quickly, not to bypass code review. Committing directly to `main` without review is only appropriate for solo projects.

- **Automate branch naming enforcement.** A pre-receive hook or branch protection rule that rejects improperly named branches costs almost nothing to set up and prevents naming chaos.

- **Write a CONTRIBUTING.md** that documents your branching strategy, naming conventions, and PR process. New team members should not have to guess.

- **Start simple, evolve as needed.** Most teams should start with GitHub Flow. If you discover you need release branches (you released a `v2.0` and need to patch `v1.x` simultaneously), then evolve to GitFlow for the release management parts. Do not adopt complexity you do not yet need.

## Real-World Use Cases

### SaaS web application: GitHub Flow

A five-person team building a B2B SaaS product deploys multiple times per week.

```bash
# Developer picks up a Jira ticket: JIRA-2341 (add CSV export)
git switch main
git pull
git switch -c feat/JIRA-2341-csv-export

# Two days of work
git commit -m "feat(reports): add CSV export service"
git commit -m "feat(reports): add export button to report header"
git commit -m "test(reports): add unit tests for CSV service"

# Open PR, get review from one teammate
gh pr create \
  --title "feat(reports): add CSV export for date-filtered reports" \
  --body "Implements JIRA-2341. Closes #241."

# CI passes, one approval received
# Squash-merge to main
gh pr merge --squash --delete-branch

# Deployment pipeline automatically deploys main to production
# Feature is live within 10 minutes of merge
```

### Versioned library: GitFlow

An open-source library that publishes to npm is on `v3.x`. A critical security vulnerability is found in `v2.x`.

```bash
# Branches in play:
# main     -- v3.x production code
# develop  -- next v3.x release
# v2.x     -- maintained v2.x branch (long-lived support branch)

# Fix the vulnerability in v2.x
git switch v2.x
git pull
git switch -c hotfix/v2.8.1-security-fix

git commit -m "security: patch XSS vulnerability in template renderer (CVE-2025-1234)"

# Merge back to v2.x branch and tag
git switch v2.x
git merge hotfix/v2.8.1-security-fix --no-ff
git tag -a v2.8.1 -m "Security patch: CVE-2025-1234"
git push origin v2.x --tags

# Cherry-pick or port the fix to v3.x develop as well
git switch develop
git cherry-pick <fix-commit-hash>
git push origin develop

# Publish v2.8.1 to npm
npm publish
```

### Trunk-based development: e-commerce platform

A 15-person team at an e-commerce company builds a new checkout redesign. The feature will take four weeks but they need to keep shipping other work.

```typescript
// Feature flag wraps all new checkout code
// lib/feature-flags.ts
export const flags = {
  NEW_CHECKOUT_FLOW: process.env.NEXT_PUBLIC_FF_NEW_CHECKOUT === 'true',
};

// pages/checkout/index.tsx
import { flags } from '@/lib/feature-flags';

export default function CheckoutPage() {
  if (flags.NEW_CHECKOUT_FLOW) {
    return <NewCheckoutFlow />;
  }
  return <LegacyCheckoutFlow />;
}
```

```bash
# Week 1: Checkout team merges cart redesign components behind flag
git switch -c feat/checkout-cart-step
# ... work for 1 day ...
gh pr create --title "feat(checkout): cart step for new flow [behind flag]"
gh pr merge --squash  # merged, but flag is OFF in production

# Week 2: Checkout team merges payment step behind flag
git switch -c feat/checkout-payment-step
# ... work for 1 day ...
gh pr merge --squash

# Meanwhile, other teams ship unrelated features to main
# No conflicts because checkout work is isolated in components,
# not in the main checkout page routing (which uses the flag)

# Week 4: All steps complete. Enable flag for 5% of users in staging.
# Test, fix issues (small PRs, merged same day)
# Enable for 100% of users in production.

# Week 5: Remove legacy checkout code and the feature flag.
git switch -c chore/remove-legacy-checkout
# Delete LegacyCheckoutFlow, remove flag checks
gh pr merge --squash
```

The four-week feature ships in incremental commits merged to `main` continuously. No long-lived branch. No monster merge conflict at the end.

---

> **Key Takeaway:** Choose the simplest branching strategy that satisfies your actual release requirements -- most web teams need GitHub Flow, not GitFlow, and the teams with the highest deployment frequency use Trunk-Based Development backed by feature flags and fast CI.
