# 10.4 VS Code Setup for Angular

> A properly configured VS Code environment eliminates entire categories of errors before you run the code, enforces team consistency, and accelerates every editing operation.

---

## Essential Extensions

These extensions are non-negotiable for Angular development. Install them before writing any Angular code.

### Angular Language Service

The most important extension. It provides:
- Autocompletion inside Angular templates (component inputs, outputs, pipes, structural directives)
- Go-to-definition from template to component class and vice versa
- Inline error checking in templates (type mismatches, missing properties, incorrect bindings)
- Hover information showing types and documentation
- Rename symbol across template and class files simultaneously

```
Extension ID: Angular.ng-template
```

Without this extension, templates are treated as plain HTML with no type safety.

### ESLint

Static analysis for TypeScript and Angular-specific rules. Catches code smells, unused variables, incorrect patterns, and enforces team coding standards.

```
Extension ID: dbaeumer.vscode-eslint
```

### Prettier

Opinionated code formatter. Removes all formatting debates from code review by enforcing a consistent style automatically.

```
Extension ID: esbenp.prettier-vscode
```

### EditorConfig for VS Code

Reads `.editorconfig` files and applies settings (indentation, line endings, trailing whitespace) so all team members use the same basic formatting regardless of personal VS Code settings.

```
Extension ID: EditorConfig.EditorConfig
```

### Sharing recommended extensions with the team

Create an `.vscode/extensions.json` file in the repository root:

```json
// .vscode/extensions.json
{
  "recommendations": [
    "Angular.ng-template",
    "dbaeumer.vscode-eslint",
    "esbenp.prettier-vscode",
    "EditorConfig.EditorConfig",
    "johnpapa.Angular2",
    "cyrilletuzi.angular-schematics",
    "usernamehw.errorlens"
  ],
  "unwantedRecommendations": [
    "ms-vscode.vscode-typescript-tslint-plugin"
  ]
}
```

When a team member opens the project, VS Code prompts them to install the recommended extensions. The `unwantedRecommendations` list prevents deprecated or conflicting extensions from being suggested.

## Workspace Settings

Workspace settings (`.vscode/settings.json`) override user settings for the project. These should be committed to the repository so every developer has the same configuration.

### Complete workspace settings file

```json
// .vscode/settings.json
{
  // ── Formatting ──────────────────────────────────────────────
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "explicit",
    "source.organizeImports": "explicit"
  },

  // ── TypeScript ──────────────────────────────────────────────
  "typescript.preferences.importModuleSpecifier": "relative",
  "typescript.updateImportsOnFileMove.enabled": "always",
  "typescript.suggest.autoImports": true,
  "typescript.tsdk": "node_modules/typescript/lib",

  // ── Angular ─────────────────────────────────────────────────
  "angular.enable.strictTemplates": true,

  // ── ESLint ──────────────────────────────────────────────────
  "eslint.validate": [
    "javascript",
    "typescript",
    "html"
  ],
  "eslint.useFlatConfig": true,

  // ── File Associations ───────────────────────────────────────
  "files.associations": {
    "*.html": "html"
  },

  // ── File Exclusions (performance) ───────────────────────────
  "files.exclude": {
    "**/node_modules": true,
    "**/dist": true,
    "**/.angular": true
  },
  "search.exclude": {
    "**/node_modules": true,
    "**/dist": true,
    "**/.angular": true,
    "**/coverage": true
  },
  "files.watcherExclude": {
    "**/node_modules/**": true,
    "**/dist/**": true,
    "**/.angular/**": true
  },

  // ── Editor ──────────────────────────────────────────────────
  "editor.tabSize": 2,
  "editor.insertSpaces": true,
  "editor.rulers": [100],
  "editor.bracketPairColorization.enabled": true,
  "editor.guides.bracketPairs": "active",
  "editor.linkedEditing": true,
  "editor.inlineSuggest.enabled": true,

  // ── Files ───────────────────────────────────────────────────
  "files.trimTrailingWhitespace": true,
  "files.insertFinalNewline": true,
  "files.trimFinalNewlines": true,
  "files.eol": "\n"
}
```

### Key settings explained

**`editor.codeActionsOnSave`:** Runs ESLint auto-fix and organizes imports every time you save. This eliminates manual cleanup and ensures consistent import order.

**`typescript.preferences.importModuleSpecifier: "relative"`:** When auto-importing, VS Code uses relative paths (`./services/auth.service`) instead of non-relative paths. Angular projects typically use relative imports.

**`typescript.updateImportsOnFileMove.enabled: "always"`:** When you rename or move a file, VS Code automatically updates all import paths that reference it. Without this, moving a file breaks imports silently.

**`files.watcherExclude`:** Excludes `node_modules`, `dist`, and `.angular` from the file watcher. This significantly reduces CPU usage and memory consumption on large projects. Without this exclusion, VS Code watches tens of thousands of files unnecessarily.

**`angular.enable.strictTemplates`:** Enables strict type checking in Angular templates. Template expressions are type-checked against the component class, catching type mismatches at edit time instead of runtime.

## Debugging Configuration

### launch.json for Chrome debugging

Create `.vscode/launch.json`:

```json
// .vscode/launch.json
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Launch Chrome (ng serve)",
      "type": "chrome",
      "request": "launch",
      "url": "http://localhost:4200",
      "webRoot": "${workspaceFolder}",
      "sourceMapPathOverrides": {
        "webpack:/*": "${webRoot}/*"
      }
    },
    {
      "name": "Attach to Chrome",
      "type": "chrome",
      "request": "attach",
      "port": 9222,
      "webRoot": "${workspaceFolder}",
      "sourceMapPathOverrides": {
        "webpack:/*": "${webRoot}/*"
      }
    },
    {
      "name": "Debug Tests (Karma)",
      "type": "chrome",
      "request": "launch",
      "url": "http://localhost:9876/debug.html",
      "webRoot": "${workspaceFolder}"
    }
  ]
}
```

### How to debug

1. Start `ng serve` in the terminal
2. Press `F5` (or Run > Start Debugging)
3. VS Code opens Chrome and connects to your running app
4. Set breakpoints directly in your TypeScript files in VS Code
5. Breakpoints hit in VS Code with full access to the Scope pane, Watch, and Call Stack

### Compound configurations

Run multiple debug targets simultaneously:

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Frontend",
      "type": "chrome",
      "request": "launch",
      "url": "http://localhost:4200",
      "webRoot": "${workspaceFolder}/apps/frontend"
    },
    {
      "name": "Backend",
      "type": "node",
      "request": "launch",
      "program": "${workspaceFolder}/apps/api/src/main.ts",
      "outFiles": ["${workspaceFolder}/apps/api/dist/**/*.js"]
    }
  ],
  "compounds": [
    {
      "name": "Full Stack",
      "configurations": ["Frontend", "Backend"]
    }
  ]
}
```

Select "Full Stack" in the debug dropdown to launch both the frontend and backend debuggers simultaneously.

## Tasks Configuration

VS Code tasks automate common commands and integrate with the terminal.

```json
// .vscode/tasks.json
{
  "version": "2.0.0",
  "tasks": [
    {
      "label": "ng serve",
      "type": "shell",
      "command": "npx ng serve",
      "group": "build",
      "isBackground": true,
      "problemMatcher": {
        "owner": "typescript",
        "pattern": {
          "regexp": "^(.*)\\((\\d+),(\\d+)\\):\\s+(error|warning)\\s+TS(\\d+):\\s+(.*)$",
          "file": 1,
          "line": 2,
          "column": 3,
          "severity": 4,
          "code": 5,
          "message": 6
        },
        "background": {
          "activeOnStart": true,
          "beginsPattern": "Compiling",
          "endsPattern": "Compiled"
        }
      }
    },
    {
      "label": "ng test",
      "type": "shell",
      "command": "npx ng test --watch=false --code-coverage",
      "group": "test",
      "problemMatcher": []
    },
    {
      "label": "ng lint",
      "type": "shell",
      "command": "npx ng lint",
      "group": "test",
      "problemMatcher": ["$eslint-stylish"]
    },
    {
      "label": "ng build (production)",
      "type": "shell",
      "command": "npx ng build --configuration=production",
      "group": {
        "kind": "build",
        "isDefault": true
      },
      "problemMatcher": ["$tsc"]
    }
  ]
}
```

Run tasks with `Ctrl+Shift+B` (build tasks) or `Ctrl+Shift+P` > "Tasks: Run Task".

The `problemMatcher` configuration parses compiler output and shows errors in the Problems panel with clickable file/line references.

## ESLint Configuration for Angular

Angular 20+ uses the ESLint flat config system (`eslint.config.js`).

```javascript
// eslint.config.js
import angular from 'angular-eslint';
import tseslint from 'typescript-eslint';
import prettier from 'eslint-config-prettier';

export default tseslint.config(
  {
    files: ['**/*.ts'],
    extends: [
      ...tseslint.configs.recommended,
      ...tseslint.configs.stylistic,
      ...angular.configs.tsRecommended,
    ],
    processor: angular.processInlineTemplates,
    rules: {
      '@angular-eslint/directive-selector': [
        'error',
        { type: 'attribute', prefix: 'app', style: 'camelCase' },
      ],
      '@angular-eslint/component-selector': [
        'error',
        { type: 'element', prefix: 'app', style: 'kebab-case' },
      ],
      '@typescript-eslint/no-unused-vars': [
        'error',
        { argsIgnorePattern: '^_' },
      ],
      '@typescript-eslint/explicit-function-return-type': 'off',
      '@typescript-eslint/no-floating-promises': 'error',
    },
  },
  {
    files: ['**/*.html'],
    extends: [
      ...angular.configs.templateRecommended,
      ...angular.configs.templateAccessibility,
    ],
  },
  prettier, // must be last to disable conflicting rules
);
```

### Key ESLint rules for Angular

| Rule | What it catches |
|------|-----------------|
| `@angular-eslint/component-selector` | Components not using the correct prefix or naming style |
| `@angular-eslint/no-empty-lifecycle-method` | Empty `ngOnInit()` and similar lifecycle hooks |
| `@angular-eslint/prefer-on-push-component-change-detection` | Components not using `OnPush` change detection |
| `@angular-eslint/template/no-negated-async` | Negated async pipe in templates (`!obs$ \| async`) |
| `@angular-eslint/template/accessibility-alt-text` | Images missing alt text in templates |
| `@typescript-eslint/no-floating-promises` | Promises without `await` or `.catch()` |

```json
// .prettierrc
{
  "singleQuote": true,
  "trailingComma": "all",
  "printWidth": 100,
  "tabWidth": 2,
  "semi": true,
  "bracketSpacing": true,
  "arrowParens": "always",
  "endOfLine": "lf",
  "htmlWhitespaceSensitivity": "css",
  "overrides": [
    {
      "files": "*.html",
      "options": {
        "parser": "angular"
      }
    }
  ]
}
```

```
# .prettierignore
dist
node_modules
.angular
coverage
package-lock.json
```

The `"parser": "angular"` override ensures Prettier uses the Angular parser for HTML template files, which correctly handles Angular template syntax (`@if`, `@for`, `[binding]`, `(event)`, etc.).

### Resolving ESLint + Prettier conflicts

`eslint-config-prettier` disables all ESLint rules that conflict with Prettier. It must be the last entry in your ESLint config's `extends` array. With this setup:
- **Prettier** handles formatting (whitespace, semicolons, quotes)
- **ESLint** handles code quality (unused variables, type errors, Angular patterns)

There is no conflict because each tool owns a separate concern.

## Performance Optimization

Large Angular projects (monorepos with hundreds of components) can make VS Code slow. These settings help.

### Exclude unnecessary files from watching

```json
// .vscode/settings.json
{
  "files.watcherExclude": {
    "**/node_modules/**": true,
    "**/dist/**": true,
    "**/.angular/**": true,
    "**/coverage/**": true,
    "**/.git/objects/**": true,
    "**/.git/subtree-cache/**": true
  }
}
```

### Limit TypeScript memory

```json
// .vscode/settings.json
{
  "typescript.tsserver.maxTsServerMemory": 4096
}
```

Default is 3072 MB. Increase for very large projects. Decrease if your machine has limited RAM.

### Disable minimap for large files

```json
{
  "editor.minimap.enabled": false
}
```

The minimap is rarely useful for Angular development and consumes rendering resources on large files.

## Common Mistakes

- **Not installing the Angular Language Service.** Without it, templates have no type checking, no autocompletion, and no go-to-definition. Every other extension is optional; this one is mandatory.

- **Using user settings instead of workspace settings for team conventions.** User settings are personal preferences. Workspace settings (`.vscode/settings.json`) are committed to the repo and shared. Format-on-save, ESLint auto-fix, and import organization belong in workspace settings.

- **Not excluding `node_modules` from the file watcher.** Without the exclusion, VS Code watches hundreds of thousands of files, consuming CPU and slowing everything. Add `files.watcherExclude` with `node_modules`, `dist`, and `.angular`.

- **Conflicting Prettier and ESLint rules.** If both tools try to enforce formatting, they fight on every save. Use `eslint-config-prettier` to disable all ESLint formatting rules, and let Prettier handle formatting exclusively.

- **Not using `editor.codeActionsOnSave`.** Manually running ESLint fix and organizing imports is tedious and inconsistent. Configure them to run on save automatically.

- **Missing `sourceMapPathOverrides` in launch.json.** Without correct source map path mappings, breakpoints in VS Code will not hit because the debugger cannot map the compiled JavaScript back to your TypeScript source files.

- **Using TSLint instead of ESLint.** TSLint has been deprecated since 2019. Angular migrated to ESLint starting with Angular 12. If your project still uses TSLint, migrate with `ng add @angular-eslint/schematics`.

## Best Practices

- **Commit `.vscode/settings.json`, `.vscode/extensions.json`, and `.vscode/tasks.json`.** These files standardize the development environment for every team member. Add `.vscode/launch.json` as well if the debug configuration is not user-specific.

- **Use `"source.organizeImports": "explicit"` on save.** This removes unused imports and sorts the remaining ones alphabetically. Clean imports make diffs smaller and more focused.

- **Set `"typescript.updateImportsOnFileMove.enabled": "always"`.** When you refactor by moving or renaming files, VS Code automatically updates all import paths. This saves significant manual effort.

- **Create project-specific snippets for signal-based patterns.** If your team uses signals extensively, snippets for `signal()`, `computed()`, `effect()`, and `input()` save keystrokes and enforce consistent patterns.

- **Set `"angular.enable.strictTemplates": true`.** This enables the strictest template type checking. Combined with the Angular Language Service, it catches binding errors at edit time rather than at runtime.

- **Keep extensions minimal.** Every extension adds startup time and memory overhead. Install only what you actually use. Periodically review installed extensions and remove ones you have not used in weeks.

## Real-World Use Cases

### Setting up a new team member

A new developer joins the team and clones the Angular repository.

```bash
# Clone and install
git clone https://github.com/org/app.git
cd app
npm install

# Open in VS Code
code .

# VS Code detects .vscode/extensions.json
# Prompt: "This workspace has extension recommendations..."
# Developer clicks "Install All"

# VS Code applies .vscode/settings.json automatically
# ESLint, Prettier, Angular Language Service are all configured

# Developer opens any .ts file:
# - Autocompletion works for Angular APIs
# - Errors appear inline (red squiggles)
# - Saving auto-formats and auto-fixes

# Developer presses F5:
# - Chrome opens to localhost:4200
# - Breakpoints set in VS Code hit correctly
# - Full debugging experience
```

Zero manual configuration. The workspace files handle everything.

### Debugging a failing template binding

The template shows an empty list even though the service returns data.

```typescript
// product-list.component.ts
@Component({
  template: `
    @for (product of products(); track product.id) {
      <app-product-card [product]="product" />
    }
  `,
})
export class ProductListComponent {
  private productService = inject(ProductService);
  products = this.productService.getProducts; // BUG: missing () call
}
```

With the Angular Language Service, VS Code shows a red squiggle on `products()` in the template because `products` is a `Signal<Product[]>` getter (a function reference), not a `Signal`. The tooltip says: "Type `() => Signal<Product[]>` is not assignable to type `Signal<Product[]>`."

Without the Angular Language Service, this error only appears at runtime as an empty list with no error message.

### Multi-root workspace for a monorepo

A monorepo with a frontend app, a backend API, and a shared library benefits from a multi-root workspace:

```json
// angular-project.code-workspace
{
  "folders": [
    { "path": "apps/frontend", "name": "Frontend" },
    { "path": "apps/api", "name": "Backend API" },
    { "path": "libs/shared", "name": "Shared Library" }
  ],
  "settings": {
    "files.watcherExclude": {
      "**/node_modules/**": true,
      "**/dist/**": true,
      "**/.angular/**": true
    }
  }
}
```

Open with `code angular-project.code-workspace`. Each folder appears as a separate root in the Explorer. Search, go-to-definition, and refactoring work across all three roots. The shared library changes are immediately reflected in both the frontend and backend via TypeScript's project references.
