# 10.6 Browser Console Techniques

> The browser console is your most accessible debugging tool. Mastering its full API turns `console.log` from a crutch into a precision instrument.

---

## The Console Object

The `console` object is available globally in every browser environment (`window.console`) and in Node.js. It exposes far more than just `log`. Every method writes to the same output stream but with different formatting, filtering, and behavior.

All console methods accept multiple arguments. The browser joins them with spaces and renders objects as expandable trees.

```javascript
// Multiple arguments are joined
console.log("User:", user, "Score:", score);

// Objects are rendered as expandable trees
console.log({ name: "Alice", scores: [98, 87, 93] });
```

## Output Methods

### console.log()

The most common method. Outputs general informational messages.

```javascript
console.log("Application started");
console.log("Value is", 42);
console.log("Multiple", "strings", "concatenated");
```

### console.info()

Functionally identical to `console.log` in most browsers. In Firefox, it shows a small "i" icon. The real value is semantic: use it to signal informational messages that are distinct from debug noise.

```javascript
console.info("Server connected on port 3000");
```

### console.warn()

Outputs a warning with a yellow background and triangle icon. Browsers allow filtering by warning level, so this helps separate "something is off" from "here is data."

```javascript
console.warn("Deprecated API used: fetchLegacy()");
console.warn("Cache size approaching limit:", cacheSize);
```

### console.error()

Outputs an error with a red background and cross icon. Includes a stack trace by default in most browsers.

```javascript
console.error("Failed to load configuration");
console.error("HTTP error:", response.status, response.statusText);
```

### console.debug()

Identical to `console.log` but categorized as "Verbose" in Chrome. Hidden by default unless you enable verbose logging in the console filter dropdown.

```javascript
console.debug("Internal state:", internalState);
```

This makes `console.debug` useful for messages you want available but not cluttering normal output.

### console.dir()

Displays an interactive list of the properties of a JavaScript object. The key difference from `console.log`:

```javascript
const el = document.querySelector("#app");

// console.log shows the HTML representation
console.log(el);
// <div id="app">...</div>

// console.dir shows the JavaScript object with all properties
console.dir(el);
// div#app { accessKey: "", align: "", ariaAtomic: null, ... }
```

Use `console.dir` when you need to inspect the JavaScript properties of a DOM element rather than its HTML structure.

### console.dirxml()

Displays an XML/HTML element representation of the specified object. In practice, `console.log` already does this for DOM elements, so `console.dirxml` is rarely needed directly.

```javascript
console.dirxml(document.body);
```

### console.clear()

Clears all previous console output. Also clears the console history in some browsers.

```javascript
console.clear();
// Console is now empty
```

Note: If "Preserve log" is enabled in DevTools settings, `console.clear()` does nothing.

## Assertion

### console.assert()

Logs an error message only when the first argument is falsy. Does NOT throw an error or halt execution. Think of it as a conditional `console.error`.

```javascript
const age = 15;
console.assert(age >= 18, "User is underage:", age);
// Assertion failed: User is underage: 15

const name = "Alice";
console.assert(name, "Name is required");
// No output (name is truthy)
```

This is useful for sanity checks during development:

```javascript
function processItems(items) {
  console.assert(Array.isArray(items), "Expected array, got:", typeof items);
  console.assert(items.length > 0, "Empty array passed to processItems");

  // ... process items
}
```

Common mistake: relying on `console.assert` for production validation. It only logs; it does not throw. Use proper error handling for real validation.

## Counting

### console.count() and console.countReset()

Tracks how many times a labeled call site is reached. The label defaults to `"default"` if omitted.

```javascript
function handleClick(type) {
  console.count(type);
}

handleClick("save");    // save: 1
handleClick("save");    // save: 2
handleClick("cancel");  // cancel: 1
handleClick("save");    // save: 3

console.countReset("save");
handleClick("save");    // save: 1
```

Real use case: tracking how many times a function fires during a render cycle to detect unnecessary re-renders:

```javascript
function renderList(items) {
  console.count("renderList called");
  // If this logs more than expected, something is triggering extra renders
  return items.map(item => renderItem(item));
}
```

## Timing

### console.time(), console.timeLog(), and console.timeEnd()

Measures elapsed time with millisecond precision. Each timer needs a unique label.

```javascript
console.time("data-fetch");

const response = await fetch("/api/users");
console.timeLog("data-fetch", "fetch complete");

const data = await response.json();
console.timeLog("data-fetch", "parse complete");

renderUsers(data);
console.timeEnd("data-fetch");

// data-fetch: 245.3ms fetch complete
// data-fetch: 247.1ms parse complete
// data-fetch: 312.8ms
```

Key behaviors:
- Up to 10,000 simultaneous timers per page.
- `console.timeLog` logs intermediate values without stopping the timer.
- `console.timeEnd` logs the final time and removes the timer.
- Calling `console.timeEnd` with a non-existent label produces a warning.

Nesting timers:

```javascript
console.time("total");

console.time("step-1");
await loadConfig();
console.timeEnd("step-1"); // step-1: 89.2ms

console.time("step-2");
await connectDB();
console.timeEnd("step-2"); // step-2: 203.5ms

console.timeEnd("total"); // total: 294.1ms
```

## Grouping

### console.group(), console.groupCollapsed(), and console.groupEnd()

Creates collapsible, nested sections in the console output. Invaluable for organizing related logs.

```javascript
console.group("User Authentication");
console.log("Checking credentials...");
console.log("Token valid:", true);
console.log("Permissions:", ["read", "write"]);
console.groupEnd();
```

Output appears as a collapsible tree:

```
v User Authentication
    Checking credentials...
    Token valid: true
    Permissions: (2) ["read", "write"]
```

`console.groupCollapsed` starts the group in a collapsed state, which is better for verbose output:

```javascript
console.groupCollapsed("HTTP Request: GET /api/users");
console.log("Headers:", headers);
console.log("Status:", 200);
console.log("Body:", responseBody);
console.groupEnd();
```

Nesting groups for complex operations:

```javascript
function debugRequest(method, url, options) {
  console.group(`${method} ${url}`);

  console.groupCollapsed("Request Headers");
  Object.entries(options.headers).forEach(([key, val]) => {
    console.log(`${key}: ${val}`);
  });
  console.groupEnd();

  console.groupCollapsed("Request Body");
  console.log(options.body);
  console.groupEnd();

  console.groupEnd();
}
```

## Table Display

### console.table()

Renders arrays and objects as sortable, filterable tables. This is one of the most underused console methods.

```javascript
const users = [
  { name: "Alice", role: "admin", active: true },
  { name: "Bob", role: "editor", active: false },
  { name: "Carol", role: "viewer", active: true },
];

console.table(users);
```

Output is a formatted table with clickable column headers for sorting:

```
| (index) | name    | role     | active |
|---------|---------|----------|--------|
| 0       | Alice   | admin    | true   |
| 1       | Bob     | editor   | false  |
| 2       | Carol   | viewer   | true   |
```

You can select which columns to display:

```javascript
// Only show name and role columns
console.table(users, ["name", "role"]);
```

Works with objects too:

```javascript
const config = {
  apiUrl: "https://api.example.com",
  timeout: 5000,
  retries: 3,
  debug: false,
};

console.table(config);
```

```
| (index)  | Value                    |
|----------|--------------------------|
| apiUrl   | https://api.example.com  |
| timeout  | 5000                     |
| retries  | 3                        |
| debug    | false                    |
```

Nested objects display `[Object]` in cells, so flatten data before tabling:

```javascript
// Bad: nested objects show as [Object]
console.table(complexNestedData);

// Good: flatten first
const flat = data.map(d => ({
  name: d.name,
  city: d.address.city,
  zip: d.address.zip,
}));
console.table(flat);
```

## Stack Traces

### console.trace()

Outputs a stack trace from the point where it is called. Shows the entire call path.

```javascript
function innerFunction() {
  console.trace("Trace from innerFunction");
}

function middleFunction() {
  innerFunction();
}

function outerFunction() {
  middleFunction();
}

outerFunction();

// Trace from innerFunction
//   innerFunction      @ script.js:2
//   middleFunction     @ script.js:6
//   outerFunction      @ script.js:10
//   (anonymous)        @ script.js:13
```

Use `console.trace` to answer "who called this function?" without setting breakpoints:

```javascript
class EventBus {
  emit(event, data) {
    console.trace(`Event emitted: ${event}`);
    // Now you can see exactly which code path triggered each event
    this.listeners[event]?.forEach(fn => fn(data));
  }
}
```

## Format Specifiers

Console methods support C-style format specifiers for controlling output.

| Specifier | Description                              |
|-----------|------------------------------------------|
| `%s`      | Format as string                         |
| `%d`/`%i` | Format as integer                       |
| `%f`      | Format as floating-point number          |
| `%o`      | Format as expandable DOM element         |
| `%O`      | Format as expandable JavaScript object   |
| `%c`      | Apply CSS styles to the output           |

```javascript
console.log("String: %s", "hello");          // String: hello
console.log("Integer: %d", 42.7);            // Integer: 42
console.log("Float: %f", 3.14159);           // Float: 3.14159
console.log("DOM: %o", document.body);       // Expandable DOM element
console.log("Object: %O", { a: 1, b: 2 });  // Expandable JS object
```

### CSS Styling with %c

The `%c` specifier applies CSS rules to everything after it until the next `%c` or end of string.

```javascript
console.log(
  "%cSuccess!%c Operation completed in %c142ms",
  "color: green; font-weight: bold;",
  "color: inherit;",
  "color: blue; font-style: italic;"
);
```

Practical uses:

```javascript
// Application banner
console.log(
  "%c MyApp v2.3.0 %c Production ",
  "background: #333; color: #fff; padding: 2px 6px; border-radius: 3px 0 0 3px;",
  "background: #28a745; color: #fff; padding: 2px 6px; border-radius: 0 3px 3px 0;"
);

// Warning banner
console.log(
  "%c WARNING ",
  "background: #ff6b35; color: white; font-size: 16px; padding: 4px 8px; font-weight: bold;"
);

// Categorized log levels
function styledLog(category, message) {
  const styles = {
    auth:    "background: #6c5ce7; color: white; padding: 1px 4px;",
    network: "background: #00b894; color: white; padding: 1px 4px;",
    render:  "background: #e17055; color: white; padding: 1px 4px;",
  };

  console.log(`%c${category}%c ${message}`, styles[category] || "", "");
}

styledLog("auth", "Token refreshed");
styledLog("network", "GET /api/users 200 OK");
styledLog("render", "Component re-rendered");
```

### Supported CSS Properties in Console

Not all CSS works. Browsers support a subset:

- `color`, `background`, `background-color`
- `font-size`, `font-weight`, `font-style`, `font-family`
- `padding`, `margin`
- `border`, `border-radius`
- `text-decoration`, `text-transform`
- `line-height`, `letter-spacing`

Properties like `width`, `height`, `display`, and layout-related rules are ignored.

## Labeled Logging Pattern

When `console.log` output gets crowded, adding labels manually is tedious. Use computed property shorthand to label values automatically.

```javascript
const username = "alice";
const userId = 42;
const isActive = true;

// Bad: no labels
console.log(username, userId, isActive);
// alice 42 true
// Which value is which?

// Good: wrap in object for automatic labeling
console.log({ username, userId, isActive });
// { username: "alice", userId: 42, isActive: true }
```

This works because of ES6 shorthand property names. The browser renders the object with property names as labels.

For arrays, wrap in a labeled object:

```javascript
const items = [1, 2, 3, 4, 5];
console.log({ items, length: items.length });
// { items: [1, 2, 3, 4, 5], length: 5 }
```

## Console Utilities API (Chrome/Edge Only)

These are built-in functions available only in the DevTools console. They do not work in your source code.

### Element References: $0 through $4

The `$0` through `$4` shortcuts reference the last five elements selected in the Elements panel.

```javascript
// Select an element in the Elements panel, then in Console:
$0              // The currently selected element
$0.textContent  // Its text
$0.classList    // Its CSS classes
$1              // The previously selected element
```

### Last Evaluated Expression: $_

The `$_` shortcut holds the result of the last evaluated expression.

```javascript
2 + 2
// 4

$_ * 10
// 40

$_.toString()
// "40"
```

### Query Selectors: $() and $$()

Shorthand for `document.querySelector` and `document.querySelectorAll` (returns a real array, not a NodeList).

```javascript
$(".header")         // document.querySelector(".header")
$$("li.active")      // Array.from(document.querySelectorAll("li.active"))

// Since $$() returns an array, you can use array methods directly
$$("img").map(img => img.src)
$$("a").filter(a => a.href.includes("external"))
```

### copy()

Copies any value to the clipboard as a JSON string.

```javascript
copy($0)                           // Copy selected element's outerHTML
copy($$("a").map(a => a.href))     // Copy all link URLs as JSON array
copy(JSON.parse(responseText))     // Copy formatted JSON to clipboard
```

### monitor() and unmonitor()

Logs every call to a function with its arguments.

```javascript
function calculateTotal(items, tax) {
  return items.reduce((sum, i) => sum + i.price, 0) * (1 + tax);
}

monitor(calculateTotal);

// Now every call logs:
// function calculateTotal called with arguments: [{...}, {...}], 0.08

unmonitor(calculateTotal);  // Stop monitoring
```

### monitorEvents() and unmonitorEvents()

Logs all events dispatched to a DOM element.

```javascript
// Monitor all events on an element
monitorEvents($0);

// Monitor specific event types
monitorEvents($0, "click");

// Monitor event categories
monitorEvents($0, "mouse");    // mousedown, mouseup, click, mouseover, etc.
monitorEvents($0, "key");      // keydown, keyup, keypress
monitorEvents($0, "touch");    // touchstart, touchmove, touchend
monitorEvents($0, "control");  // resize, scroll, zoom, focus, blur, select

// Monitor multiple specific events
monitorEvents($0, ["click", "keydown"]);

// Stop monitoring
unmonitorEvents($0);
unmonitorEvents($0, "click");
```

### getEventListeners()

Returns all event listeners registered on an element.

```javascript
getEventListeners($0);
// { click: [{listener: f, useCapture: false, passive: false, once: false, type: "click"}], ... }

// Check if an element has a click handler
getEventListeners($0).click;

// Get all event types registered
Object.keys(getEventListeners($0));
```

### queryObjects()

Returns all objects created with a specific constructor. Useful for detecting memory leaks.

```javascript
class MyComponent {
  constructor(name) {
    this.name = name;
  }
}

// After running code that creates instances:
queryObjects(MyComponent);
// Shows all live MyComponent instances
```

### keys() and values()

Shorthand for `Object.keys()` and `Object.values()`.

```javascript
const config = { host: "localhost", port: 3000, debug: true };

keys(config);   // ["host", "port", "debug"]
values(config); // ["localhost", 3000, true]
```

## Live Expressions

Chrome DevTools supports "Live Expressions" that evaluate continuously without cluttering the console log.

How to use:
1. Click the eye icon above the Console input.
2. Type a JavaScript expression.
3. The expression evaluates in real time and updates automatically.

Use cases:
- `document.activeElement` to track which element has focus
- `performance.memory.usedJSHeapSize` to watch memory usage
- `document.querySelectorAll('.item').length` to track DOM element count
- Custom expressions like `window.appState.currentUser`

## Console Filtering

Browsers provide built-in filtering for console output:

### By Level

The console toolbar has buttons for filtering by message level:
- **Verbose** (includes `console.debug`)
- **Info** (includes `console.log` and `console.info`)
- **Warnings** (`console.warn`)
- **Errors** (`console.error`)

### By Text

The filter input accepts text to search through messages:

```
// In the console filter box:
fetch          // Shows only messages containing "fetch"
-error         // Excludes messages containing "error"
url:api        // Filter by source URL (Chrome)
/regex/        // Regular expression filter (Chrome)
```

### By Source

```
// Filter by source file
source:app.js

// Exclude third-party noise
-source:vendor
-source:node_modules
```

## Structured Logging Patterns

### Namespace-Based Logging

Create a lightweight logging utility that mirrors the `debug` npm package pattern:

```javascript
function createLogger(namespace) {
  const color = stringToColor(namespace);

  return {
    log: (...args) =>
      console.log(`%c[${namespace}]`, `color: ${color}; font-weight: bold;`, ...args),
    warn: (...args) =>
      console.warn(`%c[${namespace}]`, `color: ${color}; font-weight: bold;`, ...args),
    error: (...args) =>
      console.error(`%c[${namespace}]`, `color: ${color}; font-weight: bold;`, ...args),
    time: (label) => console.time(`[${namespace}] ${label}`),
    timeEnd: (label) => console.timeEnd(`[${namespace}] ${label}`),
  };
}

// Generate deterministic color from string
function stringToColor(str) {
  let hash = 0;
  for (let i = 0; i < str.length; i++) {
    hash = str.charCodeAt(i) + ((hash << 5) - hash);
  }
  return `hsl(${hash % 360}, 70%, 45%)`;
}

// Usage
const authLog = createLogger("auth");
const apiLog = createLogger("api");
const uiLog = createLogger("ui");

authLog.log("User logged in:", userId);       // [auth] in unique color
apiLog.log("GET /users", { status: 200 });    // [api] in different color
uiLog.warn("Component re-rendered 5 times");  // [ui] in another color
```

### Conditional Logging with Log Levels

```javascript
const LogLevel = {
  DEBUG: 0,
  INFO: 1,
  WARN: 2,
  ERROR: 3,
  SILENT: 4,
};

function createLeveledLogger(currentLevel = LogLevel.INFO) {
  return {
    debug: (...args) => {
      if (currentLevel <= LogLevel.DEBUG) console.debug(...args);
    },
    info: (...args) => {
      if (currentLevel <= LogLevel.INFO) console.info(...args);
    },
    warn: (...args) => {
      if (currentLevel <= LogLevel.WARN) console.warn(...args);
    },
    error: (...args) => {
      if (currentLevel <= LogLevel.ERROR) console.error(...args);
    },
    setLevel: (level) => { currentLevel = level; },
  };
}

const logger = createLeveledLogger(LogLevel.DEBUG);
logger.debug("Verbose detail");  // Shows in development
logger.info("Normal operation");
logger.warn("Something unexpected");
logger.error("Something broke");

// In production
logger.setLevel(LogLevel.ERROR);
logger.debug("This is now silent");
logger.error("This still shows");
```

## Stripping Console Statements from Production

Leaving `console.log` in production code wastes bandwidth, leaks internal details, and slows down rendering. Remove them automatically.

### ESLint: no-console Rule

```javascript
// eslint.config.js
export default [
  {
    rules: {
      "no-console": ["warn", { allow: ["warn", "error"] }],
    },
  },
];
```

This warns on `console.log`, `console.info`, `console.debug`, etc., but allows `console.warn` and `console.error` which are more likely intentional.

### Terser (Built Into Webpack 5 and Vite)

```javascript
// vite.config.js
export default defineConfig({
  esbuild: {
    drop: ["console", "debugger"],
  },
});
```

```javascript
// webpack.config.js
const TerserPlugin = require("terser-webpack-plugin");

module.exports = {
  optimization: {
    minimizer: [
      new TerserPlugin({
        terserOptions: {
          compress: {
            drop_console: true,   // Remove all console.* calls
            drop_debugger: true,  // Remove debugger statements
          },
        },
      }),
    ],
  },
};
```

For selective removal (keep errors and warnings):

```javascript
// Keep console.error and console.warn, remove everything else
compress: {
  pure_funcs: ["console.log", "console.info", "console.debug"],
}
```

### Babel Plugin

```bash
npm install --save-dev babel-plugin-transform-remove-console
```

```json
{
  "env": {
    "production": {
      "plugins": ["transform-remove-console"]
    }
  }
}
```

## Snippet Execution

The browser console is a full JavaScript REPL. You can run multi-line code, define functions, and interact with the page.

### Multi-Line Input

Press `Shift+Enter` for new lines without executing:

```javascript
// Shift+Enter after each line, Enter on last line to execute
function fibonacci(n) {
  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}
fibonacci(10);
```

// Snippet: Highlight all images without alt text

// Snippet: List all event listeners on the page
$$("*").forEach(el => {

## Console in Different Contexts

### Web Workers

Web Workers have their own console that outputs to the main thread's DevTools console. Messages are prefixed with the worker name.

```javascript
// worker.js
console.log("Worker initialized");  // Shows in main console with worker label
```

### Service Workers

Service Worker console output appears in:
- The main page's DevTools console.
- The Application panel under "Service Workers."
- `chrome://serviceworker-internals` for detailed logs.

### iframes

Each iframe has its own console context. In Chrome, use the context dropdown at the top of the Console panel to switch between the main page and iframe contexts.

## Overriding Console Methods

You can wrap console methods for additional behavior like remote logging or timestamp injection.

```javascript
// Add timestamps to all console output
const originalLog = console.log;
console.log = function (...args) {
  const timestamp = new Date().toISOString().slice(11, 23);
  originalLog.call(console, `[${timestamp}]`, ...args);
};

console.log("Hello");
// [14:32:05.123] Hello
```

```javascript
// Collect all logs into an array for later inspection or sending to a server
const logBuffer = [];

["log", "warn", "error", "info"].forEach(method => {
  const original = console[method];
  console[method] = function (...args) {
    logBuffer.push({
      method,
      args,
      timestamp: Date.now(),
      stack: new Error().stack,
    });
    original.apply(console, args);
  };
});

// Later: send logBuffer to your logging service
```

Important: Always preserve the original method reference and use `.call` or `.apply` to maintain correct `this` context. Overriding without preserving breaks the original behavior.

## Common Mistakes

- **Using only `console.log` for everything.** Use `console.warn`, `console.error`, `console.table`, and `console.group` to categorize output. This enables console-level filtering and makes scanning output faster.

- **Leaving `console.log` in production.** Every log statement runs, serializes objects, and outputs to the console. This costs CPU time and leaks internal details to anyone who opens DevTools. Strip them with Terser, ESLint, or a Babel plugin.

- **Logging full objects when you need one property.** `console.log(response)` dumps the entire object. If you only need the status, log `console.log("Status:", response.status)`. Less noise, faster to read.

- **Not using `console.table` for arrays of objects.** Scrolling through dozens of expanded objects is slow. `console.table(array, ["name", "status"])` gives you a sortable, scannable view instantly.

- **Ignoring console filters.** When debugging in a noisy codebase, use the built-in level and text filters instead of scrolling through hundreds of lines. Type a filter term or toggle warning/error buttons.

- **Confusing DevTools utilities with JavaScript API.** Functions like `$0`, `$$()`, `copy()`, `monitor()`, and `getEventListeners()` only work in the DevTools console. They are not part of JavaScript and will throw `ReferenceError` in your source code.

- **Forgetting that `console.assert` does not throw.** It only logs when the condition is false. It does not halt execution or break on errors. Do not use it as a replacement for error handling.

## Best Practices

- **Match console method to severity.** Use `console.error` for errors, `console.warn` for warnings, `console.debug` for verbose details. This enables filtering and makes scanning instant.

- **Use object shorthand for labels.** `console.log({ count, total, ratio })` is faster to write and read than `console.log("count:", count, "total:", total, "ratio:", ratio)`.

- **Group related output.** Wrap sequences of related logs in `console.group` / `console.groupCollapsed` to keep the console organized. Especially useful in loops or multi-step processes.

- **Use `console.table` for collections.** Any time you have an array of objects or a key-value map, `console.table` is almost always more readable than `console.log`.

- **Use `console.time` instead of manual timestamps.** It is built-in, higher precision, and does not require date arithmetic.

- **Create namespace loggers in complex applications.** Color-coded, prefixed logs let you visually scan output and filter by subsystem.

- **Automate console stripping.** Configure your build tool to remove console statements in production. Do not rely on developers remembering to delete them.

## Real-World Use Cases

### 1. API Response Debugging

When integrating with a new API, structured logging beats scrolling through raw objects.

```javascript
async function fetchUsers() {
  console.group("fetchUsers");
  console.time("fetchUsers");

  try {
    const response = await fetch("/api/users");

    console.groupCollapsed("Response Headers");
    response.headers.forEach((value, key) => console.log(`${key}: ${value}`));
    console.groupEnd();

    const data = await response.json();
    console.log("Status:", response.status);
    console.table(data.users, ["id", "name", "email", "role"]);
    console.timeEnd("fetchUsers");

    return data;
  } catch (error) {
    console.error("fetchUsers failed:", error.message);
    console.trace("Call stack");
    console.timeEnd("fetchUsers");
    throw error;
  } finally {
    console.groupEnd();
  }
}
```

### 2. Performance Profiling a Render Pipeline

Identifying which step in a multi-step process is the bottleneck.

```javascript
function renderDashboard(rawData) {
  console.group("Dashboard Render");

  console.time("transform");
  const transformed = transformData(rawData);
  console.timeEnd("transform");

  console.time("sort");
  const sorted = sortByPriority(transformed);
  console.timeEnd("sort");

  console.time("filter");
  const filtered = applyFilters(sorted, activeFilters);
  console.timeEnd("filter");

  console.time("dom-update");
  updateDOM(filtered);
  console.timeEnd("dom-update");

  console.log("Items rendered:", filtered.length);
  console.groupEnd();
}

// Output:
// v Dashboard Render
//     transform: 12.4ms
//     sort: 3.1ms
//     filter: 1.8ms
//     dom-update: 87.2ms    <-- bottleneck is clear
//     Items rendered: 245
```

### 3. Debugging Event Handler Conflicts

When multiple event handlers fire unexpectedly, use DevTools utilities to audit and monitor.

```javascript
// Step 1: In DevTools Console, find all listeners on a button
const btn = document.querySelector("#submit-btn");
getEventListeners(btn);
// { click: [{listener: f, ...}, {listener: f, ...}] }
// Two click handlers? That might be the bug.

// Step 2: Monitor events to see what fires and in what order
monitorEvents(btn, "click");
// Click the button and watch the log

// Step 3: Track function calls
monitor(handleSubmit);
monitor(validateForm);
// Now you see the exact order and arguments

// Step 4: Clean up when done
unmonitorEvents(btn);
unmonitor(handleSubmit);
unmonitor(validateForm);
```

---

> **Key Takeaway:** The console API has 20+ methods designed for specific debugging scenarios. Using `console.table` for data, `console.group` for organization, `console.time` for performance, and DevTools utilities for DOM inspection eliminates the need to litter code with raw `console.log` statements.
