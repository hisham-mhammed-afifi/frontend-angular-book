# 11.1 Angular CLI Installation and Commands

> The Angular CLI is the primary tool for creating, developing, building, testing, and deploying Angular applications. Mastering its commands and options eliminates manual configuration and enforces consistent project conventions.

---

## Installing the Angular CLI

The CLI is distributed as an npm package. Install it globally so the `ng` command is available system-wide.

```bash
npm install -g @angular/cli
```

Verify the installation:

```bash
ng version
```

Output shows the CLI version, Node.js version, and the versions of underlying packages (TypeScript, esbuild, etc.).

### Version Management

You can install a specific version:

```bash
# Install a specific major version
npm install -g @angular/cli@21

# Install an exact version
npm install -g @angular/cli@21.2.0
```

When you run `ng` commands inside a project, the CLI uses the **local** version from `node_modules/.bin/ng`, not the global one. This means different projects can use different CLI versions. The global install is only used for `ng new` and `ng version` outside a project directory.

### npx Alternative

You can skip the global install entirely and use `npx`:

```bash
npx @angular/cli@21 new my-app
```

This downloads and runs the specified CLI version without installing it globally. Useful for one-off project creation or when working across multiple Angular versions.

## ng new: Creating a Project

The `ng new` command scaffolds a complete Angular workspace with a ready-to-run application.

```bash
ng new my-app
```

### Interactive Prompts (Angular 21)

When you run `ng new`, the CLI asks several questions:

1. **Which stylesheet format?** Options: CSS, SCSS, Sass, Less. SCSS is the most popular choice in the Angular ecosystem.
2. **Server-Side Rendering (SSR)?** Yes/No. Adds SSR and Static Site Generation (SSG) support.

### Angular 21 Defaults

New projects in Angular 21 ship with these defaults out of the box:

| Feature | Default |
|---|---|
| Component architecture | Standalone (no NgModules) |
| Change detection | Zoneless (no Zone.js) |
| Test framework | Vitest |
| Build system | esbuild + Vite |
| Style encapsulation | Emulated |
| SSR | Prompted (not default) |
| Strict mode | Enabled |

### Common ng new Flags

```bash
# Skip installing node_modules (useful for CI or custom setups)
ng new my-app --skip-install

# Skip git initialization
ng new my-app --skip-git

# Set stylesheet format without prompt
ng new my-app --style=scss

# Enable SSR without prompt
ng new my-app --ssr

# Skip SSR without prompt
ng new my-app --ssr=false

# Set a prefix for component selectors (default: "app")
ng new my-app --prefix=myco

# Skip the interactive prompts and use all defaults
ng new my-app --defaults

# Create a minimal project (no tests, inline styles/templates)
ng new my-app --minimal

# See what would be generated without writing files
ng new my-app --dry-run
```

### Generated Project Structure

After `ng new my-app`, you get:

```
my-app/
  .vscode/                    # VS Code workspace settings
    extensions.json
    launch.json
    tasks.json
  public/                     # Static assets (served as-is)
    favicon.ico
  src/
    app/
      app.component.ts        # Root component (standalone)
      app.component.html      # Root template
      app.component.css        # Root styles
      app.component.spec.ts   # Root component tests
      app.config.ts            # Application configuration (providers)
      app.routes.ts            # Route definitions
    index.html                 # Entry HTML page
    main.ts                    # Bootstrap entry point
    styles.css                 # Global styles
  angular.json                 # Workspace and build configuration
  package.json                 # Dependencies and scripts
  tsconfig.json                # TypeScript base configuration
  tsconfig.app.json            # App-specific TypeScript config
  tsconfig.spec.json           # Test-specific TypeScript config
```

Notice there is no `app.module.ts`. Angular 21 projects are standalone by default. The application bootstraps through `main.ts` using `bootstrapApplication()`.

## ng generate: Scaffolding Code

The `ng generate` command (alias: `ng g`) creates files following Angular conventions. It generates the boilerplate, registers the artifact where needed, and creates test files.

### Components

```bash
# Generate a standalone component (default in Angular 21)
ng generate component users/user-list

# Alias
ng g c users/user-list
```

This creates:

```
src/app/users/user-list/
  user-list.component.ts
  user-list.component.html
  user-list.component.css
  user-list.component.spec.ts
```

The generated component:

```typescript
import { Component } from '@angular/core';

@Component({
  selector: 'app-user-list',
  templateUrl: './user-list.component.html',
  styleUrl: './user-list.component.css'
})
export class UserListComponent {
}
```

Common component flags:

```bash
# Inline template (no separate HTML file)
ng g c header --inline-template
# Alias: -t

# Inline styles (no separate CSS file)
ng g c header --inline-style
# Alias: -s

# Skip test file generation
ng g c header --skip-tests

# Flat (no subdirectory)
ng g c header --flat

# Custom selector prefix
ng g c header --prefix=shared

# Specific style format
ng g c header --style=scss

# Change detection strategy
ng g c header --change-detection=OnPush

# View encapsulation
ng g c header --view-encapsulation=None

# Dry run (preview without creating files)
ng g c header --dry-run
```

### Services

```bash
ng generate service users/user
# Alias: ng g s users/user
```

Creates:

```
src/app/users/
  user.service.ts
  user.service.spec.ts
```

Generated service:

```typescript
import { Injectable } from '@angular/core';

@Injectable({
  providedIn: 'root'
})
export class UserService {
}
```

The `providedIn: 'root'` makes the service a singleton available application-wide without explicitly listing it in a providers array.

### Pipes

```bash
ng generate pipe shared/truncate
# Alias: ng g p shared/truncate
```

```typescript
import { Pipe, PipeTransform } from '@angular/core';

@Pipe({
  name: 'truncate'
})
export class TruncatePipe implements PipeTransform {
  transform(value: unknown, ...args: unknown[]): unknown {
    return null;
  }
}
```

### Directives

```bash
ng generate directive shared/highlight
# Alias: ng g d shared/highlight
```

### Guards (Functional by Default)

```bash
ng generate guard auth/auth
# Alias: ng g guard auth/auth
```

The CLI prompts which guard interfaces to implement: `CanActivate`, `CanActivateChild`, `CanDeactivate`, `CanMatch`.

In Angular 21, guards are generated as **functions** by default:

```typescript
import { CanActivateFn } from '@angular/router';

export const authGuard: CanActivateFn = (route, state) => {
  return true;
};
```

### Resolvers (Functional by Default)

```bash
ng generate resolver users/user-detail
# Alias: ng g r users/user-detail
```

```typescript
import { ResolveFn } from '@angular/router';

export const userDetailResolver: ResolveFn<boolean> = (route, state) => {
  return true;
};
```

### Interceptors (Functional by Default)

```bash
ng generate interceptor core/auth
```

```typescript
import { HttpInterceptorFn } from '@angular/common/http';

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  return next(req);
};
```

### Other Schematics

```bash
# Class
ng g class models/user

# Interface
ng g interface models/user

# Enum
ng g enum models/status

# Library (in a multi-project workspace)
ng g library shared-ui

# Environment files

# Web Worker
ng g web-worker data-processor
```

## ng serve: Development Server

The `ng serve` command compiles the application and starts a development server with live reload.

```bash
ng serve
```

Default: `http://localhost:4200`

### Key Flags

```bash
# Custom port
ng serve --port=3000

# Open browser automatically
ng serve --open
# Alias: -o

# Specific host (for network access)
ng serve --host=0.0.0.0

# Enable Hot Module Replacement
ng serve --hmr

# Disable live reload
ng serve --live-reload=false

# Use SSL
ng serve --ssl
ng serve --ssl-cert=./ssl/cert.pem --ssl-key=./ssl/key.pem

# Proxy API requests to a backend
ng serve --proxy-config=proxy.conf.json
```

### Proxy Configuration

Create `proxy.conf.json` in the project root to forward API requests to a backend server:

```json
{
  "/api": {
    "target": "http://localhost:3000",
    "secure": false,
    "changeOrigin": true,
    "pathRewrite": {
      "^/api": ""
    }
  }
}
```

This forwards `http://localhost:4200/api/users` to `http://localhost:3000/users`.

Register it in `angular.json` for permanent use:

```json
{
  "serve": {
    "options": {
      "proxyConfig": "proxy.conf.json"
    }
  }
}
```

Or add a script in `package.json`:

```json
{
  "scripts": {
    "start": "ng serve --proxy-config=proxy.conf.json"
  }
}
```

### HMR in Angular 21

Angular 21 uses Vite-powered HMR. Template and style changes apply instantly without a full page reload. Component TypeScript changes trigger a fast re-render. Enable it with:

```bash
ng serve --hmr
```

Or set it permanently in `angular.json`:

```json
{
  "serve": {
    "options": {
      "hmr": true
    }
  }
}
```

## ng build: Production Builds

The `ng build` command compiles the application into an output directory (`dist/` by default).

```bash
ng build
```

### Build Configurations

Angular uses the `application` builder (powered by esbuild + Vite) by default in Angular 21:

```bash
# Production build (default)
ng build

# Development build (no optimization, source maps)
ng build --configuration=development
```

The production build automatically applies:
- Tree shaking (removes unused code)
- Minification (scripts and styles)
- Dead code elimination
- Critical CSS inlining
- Font inlining
- Content hashing for cache busting

### Key Build Flags

```bash
# Custom output path
ng build --output-path=./build

# Generate source maps (for debugging production issues)
ng build --source-map

# Named chunks (for easier bundle analysis)
ng build --named-chunks

# Skip bundling (for debugging)
ng build --optimization=false

# Verbose output
ng build --verbose

# Specific configuration
ng build --configuration=staging
```

### Analyzing Bundle Size

```bash
# Install the bundle analyzer
npm install --save-dev source-map-explorer

# Build with source maps
ng build --source-map

# Analyze
npx source-map-explorer dist/my-app/browser/*.js
```

This opens an interactive treemap showing what contributes to your bundle size. It identifies large dependencies and dead code that slipped through tree shaking.

## ng test: Running Tests

Angular 21 defaults to **Vitest** for new projects.

```bash
ng test
```

### Key Flags

```bash
# Run once (no watch mode, useful for CI)
ng test --watch=false

# Run specific test files
ng test --include='**/user.service.spec.ts'

# Code coverage
ng test --code-coverage

# Specific browsers (if using Karma on older projects)
ng test --browsers=ChromeHeadless
```

### Vitest in Angular 21

New projects use `@analogjs/vitest-angular` by default. The configuration lives in `vitest.config.ts` or the `test` section of `angular.json`:

```typescript
// vitest.config.ts (generated for Angular 21 projects)
import { defineConfig } from 'vitest/config';
import angular from '@analogjs/vitest-angular';

export default defineConfig({
  plugins: [angular()],
  test: {
    globals: true,
    environment: 'jsdom',
    include: ['src/**/*.spec.ts'],
    setupFiles: ['src/test-setup.ts'],
  },
});
```

## ng add: Adding Libraries

The `ng add` command installs an npm package and runs its schematic to configure your project.

```bash
# Add Angular Material
ng add @angular/material

# Add PWA support
ng add @angular/pwa

# Add ESLint
ng add @angular-eslint/schematics

# Add SSR support
ng add @angular/ssr

# Add Tailwind CSS
ng add @angular/tailwindcss
```

`ng add` differs from `npm install` because it also:
- Updates `angular.json` with any required builder or asset configuration.
- Modifies source files (adds imports, updates `app.config.ts`).
- Installs peer dependencies.

### Bad Practice: Using npm install for Angular libraries

```bash
# Bad: installs the package but does not configure it
npm install @angular/material

# Good: installs AND configures the package
ng add @angular/material
```

Always check if a library supports `ng add` before falling back to `npm install` with manual configuration.

## ng update: Updating Angular

The `ng update` command updates Angular packages and runs migration schematics that automatically refactor your code for breaking changes.

```bash
# Check what can be updated
ng update

# Update Angular core and CLI to latest within current major
ng update @angular/core @angular/cli

# Update to next major version
ng update @angular/core@22 @angular/cli@22

# Force update (skip version compatibility check)
ng update @angular/core --force

# Migrate specific collection
ng update @angular/core --migrate-only
```

### Update Workflow

1. **Check** what needs updating:
   ```bash
   ng update
   ```
   Shows a table of packages and their available updates.

2. **Update** the framework:
   ```bash
   ng update @angular/core @angular/cli
   ```
   Runs migration schematics that handle breaking changes automatically (renames, API changes, config updates).

3. **Update** third-party libraries:
   ```bash
   ng update @angular/material
   ```

4. **Test** the application:
   ```bash
   ng test --watch=false && ng build
   ```

The official [Angular Update Guide](https://angular.dev/update-guide) provides step-by-step instructions for every version transition, including manual steps that schematics cannot handle.

## ng cache: Build Cache

Angular CLI caches build artifacts to speed up subsequent builds. The cache lives in `.angular/cache/` by default.

```bash
# Check cache status
ng cache info

# Enable cache
ng cache enable

# Disable cache
ng cache disable

# Clear cache
ng cache clean
```

The cache is enabled by default. On rebuild, esbuild reuses previously compiled modules, which can cut rebuild times significantly (often 50-80% faster).

Add `.angular/` to your `.gitignore` to avoid committing cache files.

## package.json Scripts

The CLI commands are typically wrapped in `package.json` scripts for consistency across a team:

```json
{
  "scripts": {
    "start": "ng serve",
    "build": "ng build",
    "test": "ng test",
    "lint": "ng lint",
    "build:prod": "ng build --configuration=production",
    "build:staging": "ng build --configuration=staging",
    "test:ci": "ng test --watch=false --code-coverage",
    "serve:proxy": "ng serve --proxy-config=proxy.conf.json",
    "analyze": "ng build --source-map && npx source-map-explorer dist/my-app/browser/*.js"
  }
}
```

Team members run `npm start`, `npm test`, etc. without needing to remember CLI flags.

## The --dry-run Flag

Every generation and creation command supports `--dry-run` (alias: `-d`). This shows what files would be created or modified without actually writing anything.

```bash
ng g c users/user-profile --dry-run
```

Output:

```
CREATE src/app/users/user-profile/user-profile.component.ts (245 bytes)
CREATE src/app/users/user-profile/user-profile.component.html (30 bytes)
CREATE src/app/users/user-profile/user-profile.component.css (0 bytes)
CREATE src/app/users/user-profile/user-profile.component.spec.ts (640 bytes)

NOTE: The "--dry-run" flag means no changes were made.
```

Always use `--dry-run` when you are unsure about a command's output or when working with unfamiliar schematics.

## Common Mistakes

- **Installing the CLI only globally and expecting version consistency.** The global CLI is only for `ng new` and `ng version`. Inside a project, the local version from `node_modules` is used. If you have CLI v21 globally but a project uses v19, the v19 CLI runs. To update the local CLI: `ng update @angular/cli`.

- **Using `npm install` instead of `ng add` for Angular libraries.** `ng add` runs schematics that configure `angular.json`, update `app.config.ts`, and install peer dependencies. `npm install` only downloads the package. You then have to configure everything manually.

- **Not using `--dry-run` before generating in unfamiliar paths.** The CLI places generated files relative to the current directory or the project root. Running `ng g c some-component` from the wrong directory creates files in unexpected locations. Preview with `--dry-run` first.

- **Running `ng update` across multiple major versions at once.** Angular migration schematics are designed for one major version jump at a time. Jumping from v18 to v21 directly can skip intermediate migrations. Update one major version at a time: v18 to v19, test, v19 to v20, test, v20 to v21, test.

- **Forgetting to commit before `ng update`.** Migration schematics modify source files. If something goes wrong, you need a clean state to revert to. Always commit (or stash) before running `ng update`.

- **Ignoring the build cache.** The `.angular/cache/` directory can grow large over time. If you encounter strange build issues, `ng cache clean` often resolves them. Also add `.angular/` to `.gitignore`.

- **Running `ng serve` in production.** `ng serve` is a development server. It does not optimize, minify, or apply security headers. Use `ng build` to produce production assets, then serve them with a proper web server (Nginx, Caddy, Node.js, etc.) or a platform like Firebase, Vercel, or Netlify.

## Best Practices

- **Use `ng add` for first-party and well-known libraries.** It handles configuration automatically. Check the library's documentation for `ng add` support before resorting to manual `npm install`.

- **Wrap CLI commands in package.json scripts.** Team members should not need to memorize flags. `npm start`, `npm test`, `npm run build:staging` are self-documenting.

- **Use `--dry-run` liberally.** Preview before generating. It costs nothing and prevents file cleanup.

- **Update one major version at a time.** Follow the official update guide at `angular.dev/update-guide`. Commit between each update step.

- **Configure default schematics in angular.json.** Instead of passing flags every time, set project-wide defaults:

```json
{
  "schematics": {
    "@schematics/angular:component": {
      "style": "scss",
      "changeDetection": "OnPush",
      "skipTests": false
    },
    "@schematics/angular:service": {
      "skipTests": false
    }
  }
}
```

Now every `ng g c` uses SCSS and OnPush without specifying flags.

- **Use the proxy config for backend API calls during development.** This avoids CORS issues and keeps the frontend configuration clean. Do not hardcode `http://localhost:3000` in your services.

- **Enable shell completion.** Run `ng completion` once to enable tab completion. It saves keystrokes and helps discover available commands and flags.

## Real-World Use Cases

### 1. Setting Up a New Team Project

```bash
# Create the project with team-standard options
ng new acme-dashboard --style=scss --ssr=false --prefix=acme

cd acme-dashboard

# Add standard libraries
ng add @angular-eslint/schematics
ng add @angular/material

# Configure default schematics in angular.json
# (see Best Practices section above)

# Set up proxy for backend
cat > proxy.conf.json << 'EOF'
{
  "/api": {
    "target": "http://localhost:8080",
    "secure": false,
    "changeOrigin": true
  }
}
EOF

# Update start script
# In package.json: "start": "ng serve --proxy-config=proxy.conf.json -o"

# Generate initial structure
ng g c layout/header --skip-tests
ng g c layout/sidebar --skip-tests
ng g c layout/footer --skip-tests
ng g c pages/dashboard
ng g c pages/users/user-list
ng g c pages/users/user-detail
ng g s core/api
ng g s core/auth
ng g guard core/auth
ng g interceptor core/auth

# Commit the initial scaffold
git add -A && git commit -m "chore: initial project scaffold"
```

### 2. Upgrading an Existing Project from Angular 20 to 21

```bash
# Step 1: Ensure clean working tree
git status

# Step 2: Check available updates
ng update

# Step 3: Update core and CLI together
ng update @angular/core@21 @angular/cli@21

# Step 4: The migration schematic automatically:
#   - Removes Zone.js from polyfills
#   - Adds provideZonelessChangeDetection() to app.config.ts
#   - Updates test configuration to Vitest (if opted in)
#   - Updates angular.json builder references

# Step 5: Test
ng test --watch=false
ng build

# Step 6: Update third-party libraries
ng update @angular/material@21

# Step 7: Final verification
ng serve
```

### 3. Building and Deploying to Multiple Environments

Define environment-specific configurations in `angular.json`:

```json
{
  "configurations": {
    "production": {
      "budgets": [
        {
          "type": "initial",
          "maximumWarning": "500kB",
          "maximumError": "1MB"
        }
      ],
      "outputHashing": "all"
    },
    "staging": {
      "optimization": true,
      "sourceMap": true,
      "outputHashing": "all",
      "fileReplacements": [
        {
          "replace": "src/environments/environment.ts",
          "with": "src/environments/environment.staging.ts"
        }
      ]
    }
  }
}
```

Build and deploy:

```bash
# Staging
ng build --configuration=staging
# Deploy dist/ to staging server

# Production
ng build --configuration=production
# Deploy dist/ to production server

# Or use ng deploy with a configured builder
ng deploy --configuration=production
```

> **Key Takeaway:** The Angular CLI is not just a project generator. It is the primary interface for every development task: scaffolding, serving, building, testing, linting, updating, and deploying. Learn its flags, configure defaults in `angular.json`, and use `ng add` and `ng update` to let the CLI do the heavy lifting.
