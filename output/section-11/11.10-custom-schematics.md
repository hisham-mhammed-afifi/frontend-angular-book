# 11.10 Custom Schematics

> Schematics are template-based code generators that transform a project's file tree. They power every `ng generate` command and can be customized to enforce team conventions, scaffold boilerplate, and automate repetitive tasks.

---

## What Schematics Are

A schematic is a set of instructions for transforming a software project by generating or modifying code. The Angular CLI uses schematics internally for every code generation command (`ng generate component`, `ng generate service`, etc.). You can create your own schematics to:

- Scaffold components, services, or features with your team's conventions baked in
- Add boilerplate that the default schematics do not produce (state management, testing setup, barrel files)
- Automate library installation and configuration via `ng add`
- Run code migrations when updating library versions via `ng update`

Schematics operate on a **virtual file system** called a `Tree`. All changes (creating, modifying, deleting files) are recorded against this virtual tree. They are only applied to the real file system after validation succeeds. This means a failed schematic never leaves partial changes on disk.

## Setting Up a Schematics Project

### Install the Schematics CLI

```bash
npm install -g @angular-devkit/schematics-cli
```

### Create a Blank Collection

```bash
schematics blank --name=my-schematics
cd my-schematics
```

This generates:

```
my-schematics/
  package.json
  tsconfig.json
  src/
    collection.json
    my-schematics/
      index.ts
      index_spec.ts
```

### The collection.json File

This is the entry point that maps schematic names to their implementations:

```json
// src/collection.json
{
  "$schema": "../node_modules/@angular-devkit/schematics/collection-schema.json",
  "schematics": {
    "my-schematics": {
      "description": "A blank schematic.",
      "factory": "./my-schematics/index#mySchematics"
    }
  }
}
```

The `factory` field points to the exported function: `./path/to/file#exportedFunctionName`.

### package.json Schematics Entry

```json
{
  "name": "my-schematics",
  "version": "1.0.0",
  "schematics": "./src/collection.json",
  "dependencies": {
    "@angular-devkit/schematics": "^19.0.0",
    "@angular-devkit/core": "^19.0.0"
  },
  "devDependencies": {
    "typescript": "~5.8.0"
  }
}
```

The `"schematics"` field in `package.json` tells the Angular CLI where to find the collection metadata.

## Core Concepts

### Tree

The `Tree` is a virtual representation of the file system. Every schematic receives a `Tree` as input and returns a (potentially modified) `Tree` as output.

```typescript
import { Tree } from '@angular-devkit/schematics';

// Reading a file
const content = tree.read('src/app/app.config.ts');
if (content) {
  const text = content.toString('utf-8');
}

// Checking if a file exists
if (tree.exists('src/app/app.routes.ts')) {
  // File exists
}

// Creating a file
tree.create('src/app/models/user.model.ts', `
export interface User {
  id: string;
  name: string;
  email: string;
}
`);

// Overwriting a file
tree.overwrite('src/app/app.config.ts', newContent);

// Deleting a file
tree.delete('src/app/old-file.ts');
```

### Rule

A `Rule` is a function that takes a `Tree` and returns a `Tree` (or an `Observable<Tree>`). Rules are the building blocks of schematics.

```typescript
import { Rule, Tree, SchematicContext } from '@angular-devkit/schematics';

export function mySchematic(): Rule {
  return (tree: Tree, context: SchematicContext) => {
    context.logger.info('Running my schematic...');
    tree.create('hello.txt', 'Hello from schematics!');
    return tree;
  };
}
```

### SchematicContext

The `context` object provides utilities:

| Member | Purpose |
|--------|---------|
| `context.logger` | Logging: `.info()`, `.warn()`, `.error()`, `.debug()` |
| `context.addTask()` | Schedule post-generation tasks (e.g., `npm install`) |
| `context.schematic` | Metadata about the currently executing schematic |
| `context.engine` | Access to the schematics engine |

### RuleFactory

A `RuleFactory` is a function that takes options and returns a `Rule`. This is what the `factory` field in `collection.json` points to:

```typescript
// The function signature of a RuleFactory
export function mySchematic(options: MySchematicOptions): Rule {
  return (tree: Tree, context: SchematicContext) => {
    // Transform the tree based on options
    return tree;
  };
}
```

## Creating a Generate Schematic

Let's build a schematic that generates a typed API service for a given resource.

### Step 1: Define the Schema

```json
// src/api-service/schema.json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "id": "MyApiServiceSchema",
  "title": "API Service Schematic",
  "type": "object",
  "properties": {
    "name": {
      "type": "string",
      "description": "The name of the resource (e.g., 'user', 'product')",
      "$default": {
        "$source": "argv",
        "index": 0
      },
      "x-prompt": "What is the resource name?"
    },
    "path": {
      "type": "string",
      "description": "The path to create the service in",
      "default": "src/app/services"
    },
    "skipTests": {
      "type": "boolean",
      "description": "Skip generating a test file",
      "default": false
    }
  },
  "required": ["name"]
}
```

The `$default` with `$source: "argv"` means the first positional argument becomes `name`. The `x-prompt` triggers an interactive prompt if the value is not provided.

### Step 2: Create the TypeScript Interface

```typescript
// src/api-service/schema.ts
export interface ApiServiceSchema {
  name: string;
  path: string;
  skipTests: boolean;
}
```

### Step 3: Create Template Files

Templates use EJS-like syntax (`<%= %>` for interpolation, `<% %>` for logic) and `__variable__` placeholders in file and folder names.

```
src/api-service/
  files/
    __name@dasherize__-api.service.ts.template
    __name@dasherize__-api.service.spec.ts.template
```

The service template:

```typescript
// src/api-service/files/__name@dasherize__-api.service.ts.template
import { inject, Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Observable } from 'rxjs';

export interface <%= classify(name) %> {
  id: string;
  // TODO: Add properties for <%= classify(name) %>
}

@Injectable({ providedIn: 'root' })
export class <%= classify(name) %>ApiService {
  private http = inject(HttpClient);
  private readonly baseUrl = '/api/<%= dasherize(name) %>s';

  getAll(): Observable<<%= classify(name) %>[]> {
    return this.http.get<<%= classify(name) %>[]>(this.baseUrl);
  }

  getById(id: string): Observable<<%= classify(name) %>> {
    return this.http.get<<%= classify(name) %>>(`${this.baseUrl}/${id}`);
  }

  create(payload: Omit<<%= classify(name) %>, 'id'>): Observable<<%= classify(name) %>> {
    return this.http.post<<%= classify(name) %>>(this.baseUrl, payload);
  }

  update(id: string, payload: Partial<<%= classify(name) %>>): Observable<<%= classify(name) %>> {
    return this.http.patch<<%= classify(name) %>>(`${this.baseUrl}/${id}`, payload);
  }

  delete(id: string): Observable<void> {
    return this.http.delete<void>(`${this.baseUrl}/${id}`);
  }
}
```

The test template:

```typescript
// src/api-service/files/__name@dasherize__-api.service.spec.ts.template
import { TestBed } from '@angular/core/testing';
import { provideHttpClient } from '@angular/common/http';

  beforeEach(() => {
    TestBed.configureTestingModule({

        provideHttpClient(),

      ]

  afterEach(() => {
    httpMock.verify();

```

### Step 4: Implement the Rule Factory

```typescript
// src/api-service/index.ts
import {
  Rule,
  SchematicContext,
  Tree,
  apply,
  url,
  template,
  move,
  mergeWith,
  filter,
  noop
} from '@angular-devkit/schematics';
import { strings } from '@angular-devkit/core';
import { ApiServiceSchema } from './schema';

export function apiService(options: ApiServiceSchema): Rule {
  return (tree: Tree, context: SchematicContext) => {
    context.logger.info(`Generating API service for "${options.name}"...`);

    // Apply templates from the files/ directory
    const templateSource = apply(
      url('./files'),  // Load template files from ./files directory
      [
        // Conditionally exclude test files
        options.skipTests
          ? filter(path => !path.endsWith('.spec.ts.template'))
          : noop(),

        // Process templates: replace <%= %> and __name__ placeholders
        template({
          ...strings,         // dasherize, classify, camelize, capitalize
          ...options           // name, path, skipTests
        }),

        // Move generated files to the target path
        move(options.path)
      ]
    );

    // Merge the generated files into the existing tree
    return mergeWith(templateSource)(tree, context);
  };
}
```

### Step 5: Register in collection.json

```json
{
  "$schema": "../node_modules/@angular-devkit/schematics/collection-schema.json",
  "schematics": {
    "api-service": {
      "description": "Generate a typed API service with CRUD operations",
      "factory": "./api-service/index#apiService",
      "schema": "./api-service/schema.json"
    }
  }
}
```

### Step 6: Build and Run

```bash
# Build the schematics
npm run build

# Run locally (from the schematics project)
schematics .:api-service --name=product

# Or link and use with ng generate
npm link
cd /path/to/angular-project
npm link my-schematics
ng generate my-schematics:api-service product
```

## String Utility Functions

The `strings` object from `@angular-devkit/core` provides name transformations used in templates and file paths:

| Function | Input | Output | Use Case |
|----------|-------|--------|----------|
| `dasherize` | `UserProfile` | `user-profile` | File names, selectors |
| `classify` | `user-profile` | `UserProfile` | Class names |
| `camelize` | `user-profile` | `userProfile` | Variable names |
| `capitalize` | `user` | `User` | First letter uppercase |
| `underscore` | `UserProfile` | `user_profile` | Snake case |
| `decamelize` | `UserProfile` | `user profile` | Human-readable labels |

### File Name Syntax

In file and folder names, use the `@function` pipe syntax:

```
__name@dasherize__.service.ts.template
```

For multiple transformations, chain them:

```
__name@dasherize@if-flat__/
```

The `@if-flat` function removes the folder if the `flat` option is `true`.

### Template Content Syntax

Inside template files, call functions directly:

```typescript
// In a .template file
export class <%= classify(name) %>Component {
  readonly title = '<%= capitalize(name) %>';
}
```

## Modifying Existing Files

Generating new files is straightforward. Modifying existing files requires reading, parsing, and updating them.

### Simple String Manipulation

```typescript
import { Rule, Tree } from '@angular-devkit/schematics';

export function addRouteEntry(options: { name: string }): Rule {
  return (tree: Tree) => {
    const routesPath = 'src/app/app.routes.ts';
    const content = tree.read(routesPath);

    if (!content) {
      throw new Error(`Could not find ${routesPath}`);
    }

    const text = content.toString('utf-8');
    const routeName = strings.dasherize(options.name);
    const className = strings.classify(options.name);

    // Find the routes array and add a new entry
    const newRoute = `
  {
    path: '${routeName}',
    loadComponent: () => import('./features/${routeName}/${routeName}.component')
      .then(m => m.${className}Component)
  },`;

    // Insert before the closing bracket of the routes array
    const routesArrayEnd = text.lastIndexOf('];');
    if (routesArrayEnd === -1) {
      throw new Error('Could not find Routes array in app.routes.ts');
    }

    const updatedContent = text.slice(0, routesArrayEnd) + newRoute + '\n' + text.slice(routesArrayEnd);
    tree.overwrite(routesPath, updatedContent);

    return tree;
  };
}
```

### Using the UpdateRecorder

For precise insertions, use the `UpdateRecorder` API:

```typescript
import { Rule, Tree } from '@angular-devkit/schematics';

export function addImport(options: { name: string; path: string }): Rule {
  return (tree: Tree) => {
    const filePath = 'src/app/app.config.ts';
    const content = tree.read(filePath);

    if (!content) {
      throw new Error(`File not found: ${filePath}`);
    }

    const text = content.toString('utf-8');
    const recorder = tree.beginUpdate(filePath);

    // Add an import statement at the top of the file
    const importStatement = `import { ${options.name} } from '${options.path}';\n`;
    const lastImportEnd = text.lastIndexOf("import ");
    const insertPosition = text.indexOf('\n', lastImportEnd) + 1;

    recorder.insertLeft(insertPosition, importStatement);

    tree.commitUpdate(recorder);
    return tree;
  };
}
```

### Using TypeScript AST (Advanced)

For robust modifications, parse the file with TypeScript's compiler API:

```typescript
import { Rule, Tree } from '@angular-devkit/schematics';
import * as ts from 'typescript';

export function addProviderToConfig(options: { providerName: string; importPath: string }): Rule {
  return (tree: Tree) => {
    const configPath = 'src/app/app.config.ts';
    const content = tree.read(configPath);

    if (!content) {
      throw new Error(`File not found: ${configPath}`);
    }

    const text = content.toString('utf-8');
    const sourceFile = ts.createSourceFile(
      configPath,
      text,
      ts.ScriptTarget.Latest,
      true
    );

    // Find the providers array in the ApplicationConfig
    const providersArray = findProvidersArray(sourceFile);

    if (!providersArray) {
      throw new Error('Could not find providers array in app.config.ts');
    }

    const recorder = tree.beginUpdate(configPath);

    // Add import
    const importStatement = `import { ${options.providerName} } from '${options.importPath}';\n`;
    recorder.insertLeft(0, importStatement);

    // Add provider to the array
    const lastElement = providersArray.elements[providersArray.elements.length - 1];
    const insertPos = lastElement ? lastElement.getEnd() : providersArray.getStart() + 1;
    const comma = lastElement ? ',\n    ' : '\n    ';
    recorder.insertRight(insertPos, `${comma}${options.providerName}()`);

    tree.commitUpdate(recorder);
    return tree;
  };
}

function findProvidersArray(sourceFile: ts.SourceFile): ts.ArrayLiteralExpression | undefined {
  let result: ts.ArrayLiteralExpression | undefined;

  function visit(node: ts.Node) {
    if (
      ts.isPropertyAssignment(node) &&
      ts.isIdentifier(node.name) &&
      node.name.text === 'providers' &&
      ts.isArrayLiteralExpression(node.initializer)
    ) {
      result = node.initializer;
      return;
    }
    ts.forEachChild(node, visit);
  }

  visit(sourceFile);
  return result;
}
```

## Composing Rules

Schematics supports composing multiple rules using utility functions:

### chain

Execute rules sequentially:

```typescript
import { chain, Rule } from '@angular-devkit/schematics';

export function featureModule(options: FeatureOptions): Rule {
  return chain([
    generateComponent(options),
    generateService(options),
    addRoute(options),
    updateBarrelFile(options)
  ]);
}
```

### schematic and externalSchematic

Call other schematics from within a schematic:

```typescript
import { schematic, externalSchematic, Rule } from '@angular-devkit/schematics';

export function scaffoldFeature(options: FeatureOptions): Rule {
  return chain([
    // Call another schematic in the same collection
    schematic('api-service', { name: options.name }),

    // Call a schematic from a different package
    externalSchematic('@schematics/angular', 'component', {
      name: options.name,
      standalone: true,
      changeDetection: 'OnPush',
      style: 'scss'
    })
  ]);
}
```

### mergeWith

Merge generated template files into the tree:

```typescript
import { apply, url, template, move, mergeWith } from '@angular-devkit/schematics';

const templateSource = apply(url('./files'), [
  template({ ...strings, ...options }),
  move(options.path)
]);

return mergeWith(templateSource);
```

### filter

Conditionally include files:

```typescript
import { filter } from '@angular-devkit/schematics';

// Exclude spec files
filter(path => !path.endsWith('.spec.ts'))

// Include only TypeScript files
filter(path => path.endsWith('.ts'))
```

## Scheduling Tasks

Schematics can schedule post-execution tasks, such as running `npm install`:

```typescript
import { Rule, SchematicContext, Tree } from '@angular-devkit/schematics';
import { NodePackageInstallTask, RunSchematicTask } from '@angular-devkit/schematics/tasks';

export function ngAdd(options: NgAddOptions): Rule {
  return (tree: Tree, context: SchematicContext) => {
    // Add a dependency to package.json
    const packageJson = JSON.parse(tree.read('package.json')!.toString());
    packageJson.dependencies['some-library'] = '^2.0.0';
    tree.overwrite('package.json', JSON.stringify(packageJson, null, 2));

    // Schedule npm install to run after the schematic completes
    const installTaskId = context.addTask(new NodePackageInstallTask());

    // Schedule another schematic to run after npm install finishes
    context.addTask(
      new RunSchematicTask('setup', { project: options.project }),
      [installTaskId]  // Dependencies: wait for install to complete
    );

    return tree;
  };
}
```

## ng-add Schematics

The `ng-add` schematic runs automatically when a user installs your library with `ng add your-library`. It is the standard place to configure a project for your library.

### Registering ng-add

```json
// package.json of your library
{
  "name": "my-ui-lib",
  "schematics": "./schematics/collection.json",
  "ng-add": {
    "save": "dependencies"
  }
}
```

```json
// schematics/collection.json
{
  "schematics": {
    "ng-add": {
      "description": "Configure the project to use my-ui-lib",
      "factory": "./ng-add/index#ngAdd",
      "schema": "./ng-add/schema.json"
    }
  }
}
```

### Typical ng-add Implementation

```typescript
// schematics/ng-add/index.ts
import { Rule, Tree, SchematicContext, chain } from '@angular-devkit/schematics';
import { NodePackageInstallTask } from '@angular-devkit/schematics/tasks';

export function ngAdd(): Rule {
  return chain([
    addPeerDependencies(),
    addStyles(),
    scheduleInstall()
  ]);
}

function addPeerDependencies(): Rule {
  return (tree: Tree) => {
    const packageJson = JSON.parse(tree.read('package.json')!.toString());

    const dependencies: Record<string, string> = {
      '@popperjs/core': '^2.11.0'
    };

    for (const [pkg, version] of Object.entries(dependencies)) {
      if (!packageJson.dependencies[pkg]) {
        packageJson.dependencies[pkg] = version;
      }
    }

    tree.overwrite('package.json', JSON.stringify(packageJson, null, 2));
    return tree;
  };
}

function addStyles(): Rule {
  return (tree: Tree) => {
    const angularJson = JSON.parse(tree.read('angular.json')!.toString());
    const projectName = Object.keys(angularJson.projects)[0];
    const buildOptions = angularJson.projects[projectName].architect.build.options;

    const stylePath = 'node_modules/my-ui-lib/styles/theme.css';
    if (!buildOptions.styles.includes(stylePath)) {
      buildOptions.styles.push(stylePath);
    }

    tree.overwrite('angular.json', JSON.stringify(angularJson, null, 2));
    return tree;
  };
}

function scheduleInstall(): Rule {
  return (_tree: Tree, context: SchematicContext) => {
    context.addTask(new NodePackageInstallTask());
  };
}
```

## ng-update Schematics (Migrations)

Migration schematics run when users update your library with `ng update`. They automate breaking changes across versions.

### Registering Migrations

```json
// package.json
{
  "ng-update": {
    "migrations": "./schematics/migrations.json"
  }
}
```

```json
// schematics/migrations.json
{
  "$schema": "../node_modules/@angular-devkit/schematics/collection-schema.json",
  "schematics": {
    "rename-component-selector": {
      "version": "3.0.0",
      "description": "Rename <my-button> to <ui-button>",
      "factory": "./migrations/v3/rename-selector#renameSelector"
    },
    "update-import-paths": {
      "version": "4.0.0",
      "description": "Update import paths from my-lib/core to my-lib",
      "factory": "./migrations/v4/update-imports#updateImports"
    }
  }
}
```

The `version` field specifies when this migration applies. When a user runs `ng update my-lib` and crosses from <3.0.0 to >=3.0.0, the `rename-component-selector` migration executes.

### Migration Implementation

```typescript
// schematics/migrations/v3/rename-selector.ts
import { Rule, Tree } from '@angular-devkit/schematics';

export function renameSelector(): Rule {
  return (tree: Tree) => {
    tree.visit((filePath) => {
      if (!filePath.endsWith('.html') && !filePath.endsWith('.ts')) return;
      if (filePath.includes('node_modules')) return;

      const content = tree.read(filePath);
      if (!content) return;

      let text = content.toString('utf-8');
      const original = text;

      // Replace in templates
      text = text.replace(/<my-button/g, '<ui-button');
      text = text.replace(/<\/my-button>/g, '</ui-button>');

      // Replace in component selectors
      text = text.replace(/selector:\s*['"]my-button['"]/g, "selector: 'ui-button'");

      if (text !== original) {
        tree.overwrite(filePath, text);
      }
    });

    return tree;
  };
}
```

## Testing Schematics

Use `SchematicTestRunner` for integration tests:

```typescript
// src/api-service/index.spec.ts
import { SchematicTestRunner, UnitTestTree } from '@angular-devkit/schematics/testing';
import { join } from 'path';

const collectionPath = join(__dirname, '../collection.json');

describe('api-service schematic', () => {
  let runner: SchematicTestRunner;
  let appTree: UnitTestTree;

  beforeEach(async () => {
    runner = new SchematicTestRunner('my-schematics', collectionPath);

    // Create a minimal Angular workspace tree to run against
    appTree = await runner.runExternalSchematic(
      '@schematics/angular',
      'workspace',
      { name: 'test-workspace', version: '21.0.0' }
    );
    appTree = await runner.runExternalSchematic(
      '@schematics/angular',
      'application',
      { name: 'test-app', projectRoot: '' },
      appTree
    );
  });

  it('should create the service file', async () => {
    const tree = await runner.runSchematic(
      'api-service',
      { name: 'product', path: 'src/app/services' },
      appTree
    );

    expect(tree.files).toContain('/src/app/services/product-api.service.ts');
  });

  it('should create the spec file by default', async () => {
    const tree = await runner.runSchematic(
      'api-service',
      { name: 'product', path: 'src/app/services' },
      appTree
    );

    expect(tree.files).toContain('/src/app/services/product-api.service.spec.ts');
  });

  it('should skip spec file when skipTests is true', async () => {
    const tree = await runner.runSchematic(
      'api-service',
      { name: 'product', path: 'src/app/services', skipTests: true },
      appTree
    );

    expect(tree.files).not.toContain('/src/app/services/product-api.service.spec.ts');
  });

  it('should use the correct class name', async () => {
    const tree = await runner.runSchematic(
      'api-service',
      { name: 'user-profile', path: 'src/app/services' },
      appTree
    );

    const content = tree.readContent('/src/app/services/user-profile-api.service.ts');
    expect(content).toContain('export class UserProfileApiService');
    expect(content).toContain("baseUrl = '/api/user-profiles'");
  });

  it('should generate a valid test file', async () => {
    const tree = await runner.runSchematic(
      'api-service',
      { name: 'order', path: 'src/app/services' },
      appTree
    );

    const content = tree.readContent('/src/app/services/order-api.service.spec.ts');
    expect(content).toContain('OrderApiService');
    expect(content).toContain('provideHttpClient()');
    expect(content).toContain('provideHttpClientTesting()');
  });
});
```

Run tests:

```bash
npm test
```

## Overriding Default Schematics

You can redirect the default `ng generate component` to use your custom schematic by configuring `angular.json`:

```json
{
  "cli": {
    "schematicCollections": ["my-schematics", "@schematics/angular"]
  }
}
```

When a user runs `ng generate component`, Angular CLI looks for a `component` schematic in `my-schematics` first. If not found, it falls back to `@schematics/angular`.

Alternatively, configure defaults for the built-in schematics:

```json
{
  "schematics": {
    "@schematics/angular:component": {
      "standalone": true,
      "changeDetection": "OnPush",
      "style": "scss",
      "skipTests": false
    }
  }
}
```

## Common Mistakes

- **Mutating the Tree directly instead of using templates or recorders.** Always use `tree.beginUpdate()` / `tree.commitUpdate()` for modifications, or use `apply()` / `mergeWith()` for new files. Direct string manipulation of `tree.overwrite()` without recorders can produce hard-to-debug issues with overlapping changes.

- **Forgetting to handle the case where a file does not exist.** Always check `tree.exists()` or handle `null` from `tree.read()` before operating on files. Schematics should fail with a clear error message rather than throw a cryptic null reference.

- **Not testing with a real Angular workspace tree.** Testing against an empty `Tree` misses path resolution issues. Use `runExternalSchematic` to create a proper workspace and application tree in your test setup.

- **Hardcoding paths instead of using the project configuration.** Do not assume the source root is always `src/`. Read the project configuration from `angular.json` to resolve the correct `sourceRoot`:

```typescript
function getSourceRoot(tree: Tree, projectName: string): string {
  const angularJson = JSON.parse(tree.read('angular.json')!.toString());
  return angularJson.projects[projectName].sourceRoot || 'src';
}
```

- **Publishing without building.** Schematics are authored in TypeScript but must be published as compiled JavaScript. Ensure your `prepublishOnly` script compiles TypeScript and copies template files and JSON schemas to the output directory.

## Best Practices

- **Use JSON schemas with x-prompt for interactive options.** The `x-prompt` field in your schema gives users a guided experience when they forget required flags. This matches the UX of Angular's built-in schematics.

- **Compose schematics using `chain` and `externalSchematic`.** Break complex generators into small, focused rules. A feature scaffold might chain a component schematic, a service schematic, a route registration rule, and a barrel file update.

- **Include a sandbox app for development.** Keep a test Angular project inside your schematics repo. Add npm scripts that run the schematic against the sandbox and reset it with `git checkout` so you can iterate quickly.

- **Copy non-TypeScript files in your build step.** Template files (`.template`), JSON schemas, and `collection.json` are not compiled by `tsc`. Add a `postbuild` script that copies them to the output directory:

```json
{
  "scripts": {
    "build": "tsc -p tsconfig.json",
    "postbuild": "copyfiles -u 1 \"src/**/*.json\" \"src/**/*.template\" dist/"
  }
}
```

- **Version your migration schematics carefully.** Each migration's `version` in `migrations.json` must match the library version that introduced the breaking change. Users only run migrations for versions they cross during update.

## Real-World Use Cases

### 1. Feature Scaffold Schematic

A team creates a schematic that generates a complete feature folder matching their architecture:

```bash
ng generate my-schematics:feature user-management
```

Generates:

```
src/app/features/user-management/
  user-management.component.ts
  user-management.component.html
  user-management.component.scss

  user-management.routes.ts
  services/
    user-management-api.service.ts
    user-management-api.service.spec.ts
  models/
    user-management.model.ts
```

And automatically registers the route in `app.routes.ts`.

### 2. ng-add for a Design System Library

A company publishes an internal component library. The `ng-add` schematic configures everything:

```bash
ng add @company/ui-kit
```

The schematic:
1. Adds peer dependencies to `package.json`
2. Adds the theme stylesheet to `angular.json` styles
3. Adds `provideAnimations()` to `app.config.ts` if not already present
4. Creates a `src/styles/_variables.scss` with the company's design tokens
5. Runs `npm install`

### 3. Migration Schematic for Breaking API Changes

Version 5.0 of a library renames its primary service. The migration schematic:

```typescript
export function migrateToV5(): Rule {
  return (tree: Tree) => {
    tree.visit((filePath) => {
      if (!filePath.endsWith('.ts') || filePath.includes('node_modules')) return;

      const content = tree.read(filePath)?.toString('utf-8');
      if (!content) return;

      let updated = content;

      // Rename imports
      updated = updated.replace(
        /from\s+['"]@company\/ui-kit\/old-module['"]/g,
        "from '@company/ui-kit'"
      );

      // Rename class references
      updated = updated.replace(/OldDataService/g, 'DataService');

      if (updated !== content) {
        tree.overwrite(filePath, updated);
      }
    });

    return tree;
  };
}
```

When users run `ng update @company/ui-kit`, the migration automatically updates all import paths and class references across their entire project.
