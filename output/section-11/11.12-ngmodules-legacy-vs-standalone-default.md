# 11.12 NgModules (Legacy) vs Standalone (Default)

> NgModules were the original way to organize Angular applications. Standalone components, now the default since Angular 19, eliminate the need for NgModules in most cases. Understanding both is essential because you will encounter NgModules in existing codebases and third-party libraries.

---

## The NgModule System

An NgModule is a class decorated with `@NgModule` that serves as a compilation context and dependency container. It groups components, directives, and pipes into cohesive blocks and configures dependency injection for that scope.

### The @NgModule Decorator

```typescript
// Legacy: A feature module
import { NgModule } from '@angular/core';
import { CommonModule } from '@angular/common';
import { RouterModule } from '@angular/router';
import { UserListComponent } from './user-list.component';
import { UserDetailComponent } from './user-detail.component';
import { UserCardComponent } from './user-card.component';
import { UserService } from './user.service';
import { FullNamePipe } from './full-name.pipe';
import { HighlightDirective } from './highlight.directive';
import { userRoutes } from './user.routes';

@NgModule({
  declarations: [
    UserListComponent,
    UserDetailComponent,
    UserCardComponent,
    FullNamePipe,
    HighlightDirective
  ],
  imports: [
    CommonModule,
    RouterModule.forChild(userRoutes)
  ],
  exports: [
    UserCardComponent,
    FullNamePipe
  ],
  providers: [
    UserService
  ]
})
export class UserModule {}
```

### NgModule Metadata Properties

| Property | Purpose | Scope |
|----------|---------|-------|
| `declarations` | Components, directives, and pipes that belong to this module | Local to the module |
| `imports` | Other modules whose exported declarations this module needs | Makes their exports available here |
| `exports` | Declarations from this module that other modules can use | Public API of the module |
| `providers` | Services available to this module's injector | Global (eager) or module-scoped (lazy) |
| `bootstrap` | The root component (only in the root AppModule) | Application entry point |

### How Declarations Scope Works

A component declared in `UserModule` is only usable in templates of other components within `UserModule`. To make it available outside, it must be in the `exports` array.

```typescript
// UserModule exports UserCardComponent
// Other modules that import UserModule can use <app-user-card>

@NgModule({
  imports: [UserModule],  // Now this module can use UserCardComponent
  declarations: [DashboardComponent]
})
export class DashboardModule {}
```

A component, directive, or pipe can only be declared in **one** module. Declaring it in two modules causes a compilation error.

## NgModule Categories

In the NgModule era, projects typically organized modules into these categories:

### Root Module (AppModule)

```typescript
// Legacy: app.module.ts
@NgModule({
  declarations: [AppComponent],
  imports: [
    BrowserModule,
    AppRoutingModule,
    CoreModule,
    SharedModule
  ],
  bootstrap: [AppComponent]
})
export class AppModule {}
```

### Core Module

Singleton services and components used once (header, footer, layout):

```typescript
// Legacy: core/core.module.ts
@NgModule({
  declarations: [
    HeaderComponent,
    FooterComponent,
    NotFoundComponent
  ],
  imports: [CommonModule, RouterModule],
  exports: [
    HeaderComponent,
    FooterComponent
  ]
})
export class CoreModule {
  // Guard against multiple imports
  constructor(@Optional() @SkipSelf() parentModule: CoreModule) {
    if (parentModule) {
      throw new Error('CoreModule is already loaded. Import it only in AppModule.');
    }
  }
}
```

### Shared Module

Reusable components, directives, and pipes imported by feature modules:

```typescript
// Legacy: shared/shared.module.ts
@NgModule({
  declarations: [
    LoadingSpinnerComponent,
    ConfirmDialogComponent,
    TruncatePipe,
    AutoFocusDirective
  ],
  imports: [
    CommonModule,
    FormsModule,
    ReactiveFormsModule
  ],
  exports: [
    // Re-export Angular modules so feature modules do not need to import them
    CommonModule,
    FormsModule,
    ReactiveFormsModule,
    // Export shared declarations
    LoadingSpinnerComponent,
    ConfirmDialogComponent,
    TruncatePipe,
    AutoFocusDirective
  ]
})
export class SharedModule {}
```

### Feature Modules

Domain-specific functionality, often lazy-loaded:

```typescript
// Legacy: features/orders/orders.module.ts
@NgModule({
  declarations: [
    OrderListComponent,
    OrderDetailComponent,
    OrderStatusBadgeComponent
  ],
  imports: [
    SharedModule,
    RouterModule.forChild(orderRoutes)
  ]
})
export class OrdersModule {}
```

## The forRoot / forChild Pattern

Some modules need to provide services globally (singleton) while also exporting declarations to multiple modules. The `forRoot` / `forChild` pattern solves this.

### How It Works

```typescript
// Legacy: A module with configurable providers
@NgModule({
  declarations: [TranslatePipe],
  exports: [TranslatePipe]
})
export class TranslateModule {
  // Called once in AppModule: provides the singleton service + declarations
  static forRoot(config: TranslateConfig): ModuleWithProviders<TranslateModule> {
    return {
      ngModule: TranslateModule,
      providers: [
        TranslateService,
        { provide: TRANSLATE_CONFIG, useValue: config }
      ]
    };
  }

  // Called in feature modules: provides only declarations (no duplicate service)
  static forChild(): ModuleWithProviders<TranslateModule> {
    return {
      ngModule: TranslateModule,
      providers: []  // No providers, reuses the root service
    };
  }
}
```

```typescript
// AppModule
@NgModule({
  imports: [TranslateModule.forRoot({ defaultLang: 'en' })]
})
export class AppModule {}

// Feature module
@NgModule({
  imports: [TranslateModule.forChild()]
})
export class FeatureModule {}
```

### Why This Existed

Without `forRoot`, importing a module with providers in a lazy-loaded feature module would create a new instance of the service in the lazy module's injector. The `forRoot` / `forChild` convention ensured only the root module provided the singleton service.

### The Standalone Equivalent

With standalone components and `providedIn: 'root'`, the `forRoot` / `forChild` pattern is unnecessary:

```typescript
// Modern: Service is automatically a singleton
@Injectable({ providedIn: 'root' })
export class TranslateService {
  // Singleton by default, no forRoot needed
}
```

## The Standalone Model

Standalone components, directives, and pipes manage their own dependencies directly. No NgModule required.

### Standalone Component

```typescript
// Modern: Angular 21 default
@Component({
  selector: 'app-user-list',
  imports: [UserCardComponent, FullNamePipe, DatePipe],
  template: `
    @for (user of users(); track user.id) {
      <app-user-card [user]="user" />
      <span>{{ user | fullName }}</span>
    }
  `
})
export class UserListComponent {
  users = input.required<User[]>();
}
```

Since Angular 19, `standalone: true` is the default. You do not need to write it explicitly. The `imports` array on the component replaces the NgModule's `declarations` and `imports`.

### Standalone Directive

```typescript
@Directive({
  selector: '[appHighlight]'
})
export class HighlightDirective {
  private el = inject(ElementRef);
  color = input('yellow', { alias: 'appHighlight' });

  @HostListener('mouseenter') onMouseEnter() {
    this.el.nativeElement.style.backgroundColor = this.color();
  }

  @HostListener('mouseleave') onMouseLeave() {
    this.el.nativeElement.style.backgroundColor = '';
  }
}
```

### Standalone Pipe

```typescript
@Pipe({ name: 'fullName' })
export class FullNamePipe implements PipeTransform {
  transform(user: { firstName: string; lastName: string }): string {
    return `${user.firstName} ${user.lastName}`;
  }
}
```

## Side-by-Side Comparison

| Aspect | NgModule | Standalone |
|--------|----------|------------|
| Dependency declaration | Centralized in `@NgModule.declarations` | Per-component `imports` array |
| Dependency visibility | Shared across the module | Explicit per component |
| Boilerplate | High (module class + metadata) | Low (just the component) |
| Lazy loading | `loadChildren` pointing to a module | `loadComponent` or `loadChildren` with route file |
| Service provision | `@NgModule.providers` or `forRoot` | `providedIn: 'root'` or route-level `providers` |
| Template compilation scope | All declarations in the module | Only what the component imports |
| Tree shaking | Modules can prevent tree shaking | Each import is individually shakeable |
| Testing setup | Must configure TestBed with the module or mock it | Import only what the component needs |
| Default since | Angular 2 | Angular 19 |

### Bundle Size Impact

NgModules can hurt tree shaking. If a module declares 20 components and you import that module, the bundler may include all 20 even if you only use one. With standalone components, you import exactly what you need:

```typescript
// NgModule approach: imports the entire SharedModule
@NgModule({
  imports: [SharedModule]  // Pulls in everything SharedModule exports
})
export class FeatureModule {}

// Standalone approach: imports only what is used
@Component({
  imports: [LoadingSpinnerComponent, TruncatePipe]  // Only these two
})
export class FeatureComponent {}
```

## Lazy Loading: Modules vs Standalone

### NgModule Lazy Loading

```typescript
// Legacy: Lazy load an entire module
const routes: Routes = [
  {
    path: 'orders',
    loadChildren: () => import('./features/orders/orders.module')
      .then(m => m.OrdersModule)
  }
];
```

The `OrdersModule` must define its child routes using `RouterModule.forChild(routes)`.

### Standalone Lazy Loading

```typescript
// Modern: Lazy load a single component
const routes: Routes = [
  {
    path: 'orders',
    loadComponent: () => import('./features/orders/order-list.component')
      .then(m => m.OrderListComponent)
  }
];

// Modern: Lazy load a route file with child routes
const routes: Routes = [
  {
    path: 'orders',
    loadChildren: () => import('./features/orders/order.routes')
      .then(m => m.orderRoutes)
  }
];
```

```typescript
// features/orders/order.routes.ts
import { Routes } from '@angular/router';

export const orderRoutes: Routes = [
  {
    path: '',
    loadComponent: () => import('./order-list.component')
      .then(m => m.OrderListComponent)
  },
  {
    path: ':id',
    loadComponent: () => import('./order-detail.component')
      .then(m => m.OrderDetailComponent)
  }
];
```

With standalone, there is no module wrapping the routes. Each component is independently lazy-loaded, and the route file is just an exported array.

### Route-Level Providers (Replaces Module Providers)

In the NgModule world, lazy-loaded modules got their own injector. With standalone, you achieve the same with route-level `providers`:

```typescript
// Modern: Route-level providers replace module-scoped providers
export const orderRoutes: Routes = [
  {
    path: '',
    providers: [
      OrderService,
      { provide: ORDER_API_URL, useValue: '/api/orders' }
    ],
    children: [
      {
        path: '',
        loadComponent: () => import('./order-list.component')
          .then(m => m.OrderListComponent)
      },
      {
        path: ':id',
        loadComponent: () => import('./order-detail.component')
          .then(m => m.OrderDetailComponent)
      }
    ]
  }
];
```

Services provided at the route level are instantiated when the route is activated and shared among all child routes, exactly like a lazy-loaded module's injector.

## Bootstrapping: Modules vs Standalone

### NgModule Bootstrap (Legacy)

```typescript
// Legacy: main.ts
import { platformBrowserDynamic } from '@angular/platform-browser-dynamic';
import { AppModule } from './app/app.module';

platformBrowserDynamic().bootstrapModule(AppModule)
  .catch(err => console.error(err));
```

```typescript
// Legacy: app.module.ts
@NgModule({
  declarations: [AppComponent],
  imports: [
    BrowserModule,
    AppRoutingModule,
    HttpClientModule,
    BrowserAnimationsModule
  ],
  providers: [],
  bootstrap: [AppComponent]
})
export class AppModule {}
```

### Standalone Bootstrap (Modern)

```typescript
// Modern: main.ts
import { bootstrapApplication } from '@angular/platform-browser';
import { AppComponent } from './app/app.component';
import { appConfig } from './app/app.config';

bootstrapApplication(AppComponent, appConfig)
  .catch(err => console.error(err));
```

```typescript
// Modern: app.config.ts
import { ApplicationConfig, provideZonelessChangeDetection } from '@angular/core';
import { provideRouter, withComponentInputBinding } from '@angular/router';
import { provideHttpClient, withInterceptors } from '@angular/common/http';
import { provideAnimations } from '@angular/platform-browser/animations';
import { routes } from './app.routes';
import { authInterceptor } from './core/auth.interceptor';

export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
    provideRouter(routes, withComponentInputBinding()),
    provideHttpClient(withInterceptors([authInterceptor])),
    provideAnimations()
  ]
};
```

Every `@NgModule.imports` entry has a `provide*` function equivalent:

| NgModule Import | Standalone Provider |
|-----------------|-------------------|
| `BrowserModule` | Handled by `bootstrapApplication` |
| `HttpClientModule` | `provideHttpClient()` |
| `RouterModule.forRoot(routes)` | `provideRouter(routes)` |
| `BrowserAnimationsModule` | `provideAnimations()` |
| `NoopAnimationsModule` | `provideNoopAnimations()` |

## Interoperability: Using NgModules in Standalone Projects

You will encounter third-party libraries that still export NgModules. Standalone components can import NgModules directly:

```typescript
// Importing an NgModule into a standalone component
@Component({
  imports: [
    MatButtonModule,      // Angular Material still uses NgModules
    MatIconModule,
    ReactiveFormsModule   // Angular's forms module is still an NgModule
  ],
  template: `
    <button mat-raised-button color="primary">
      <mat-icon>save</mat-icon>
      Save
    </button>
  `
})
export class SaveButtonComponent {}
```

For application-level NgModule providers (like those from `forRoot`), use `importProvidersFrom` in `app.config.ts`:

```typescript
// app.config.ts
import { importProvidersFrom } from '@angular/core';
import { TranslateModule } from '@ngx-translate/core';

export const appConfig: ApplicationConfig = {
  providers: [
    importProvidersFrom(
      TranslateModule.forRoot({ defaultLanguage: 'en' })
    )
  ]
};
```

## When You Still See NgModules

Even in Angular 21, NgModules appear in specific contexts:

1. **Third-party libraries** that have not migrated (Angular Material, ngx-translate, some UI kits)
2. **Legacy codebases** undergoing incremental migration
3. **Testing modules** like `HttpClientTestingModule` (though `provideHttpClientTesting()` is the modern equivalent)
4. **Angular's own modules** like `CommonModule`, `FormsModule`, `ReactiveFormsModule` (these are stable NgModules that standalone components can import directly)

You do not need to create new NgModules. But you must know how to read and interact with existing ones.

---

## Migration Strategy

Angular provides an automated schematic for migrating from NgModules to standalone:

```bash
ng generate @angular/core:standalone
```

The schematic offers three steps to run in sequence:

### Step 1: Convert Declarations to Standalone

```bash
ng generate @angular/core:standalone --mode=convert-to-standalone
```

This adds `imports` arrays to components and removes `standalone: false`.

Before:

```typescript
@Component({
  selector: 'app-order-list',
  templateUrl: './order-list.component.html'
})
export class OrderListComponent {}
```

After:

```typescript
@Component({
  selector: 'app-order-list',
  imports: [CommonModule, OrderCardComponent, DatePipe],
  templateUrl: './order-list.component.html'
})
export class OrderListComponent {}
```

### Step 2: Remove Empty NgModules

```bash
ng generate @angular/core:standalone --mode=prune-modules
```

Deletes NgModules that only served as containers for declarations (no meaningful providers or configuration).

### Step 3: Switch to Standalone Bootstrap

```bash
ng generate @angular/core:standalone --mode=standalone-bootstrap
```

Replaces `bootstrapModule(AppModule)` with `bootstrapApplication(AppComponent, appConfig)`.

### Incremental Migration

You do not have to migrate all at once. Standalone components and NgModule-based components coexist:

```typescript
// A standalone component can be declared in a module for backward compatibility
@NgModule({
  // This still works: import standalone components into an NgModule
  imports: [NewStandaloneComponent],
  declarations: [LegacyComponent],
  exports: [NewStandaloneComponent, LegacyComponent]
})
export class HybridModule {}
```

## Common Mistakes

- **Declaring a component in multiple NgModules.** A component, directive, or pipe can only belong to one NgModule's `declarations`. If two modules need it, declare it in one and export it, then import that module in the other.

- **Confusing `imports` and `declarations` in NgModules.** `declarations` is for your own components/directives/pipes. `imports` is for other modules. Putting a component in `imports` or a module in `declarations` causes a compilation error.

- **Providing services in SharedModule without forRoot.** If `SharedModule` includes a service in its `providers` and is imported by a lazy-loaded module, that lazy module gets its own instance of the service. This breaks the singleton pattern. Use `providedIn: 'root'` instead, or use the `forRoot` / `forChild` pattern.

- **Importing the entire SharedModule when only one component is needed (in standalone projects).** In a standalone project, import the specific component rather than a module containing 30 components. This preserves tree shaking.

```typescript
// BAD: Importing more than needed
@Component({
  imports: [SharedModule]  // Pulls in everything
})
export class MyComponent {}

// GOOD: Import only what is used
@Component({
  imports: [LoadingSpinnerComponent]
})
export class MyComponent {}
```

- **Not understanding provider scope in lazy-loaded modules.** Services provided in a lazy-loaded NgModule get a child injector. Components outside that lazy-loaded module receive a different (or no) instance. This is intentional but confuses developers who expect global singletons.

- **Creating new NgModules in Angular 21 projects.** There is no reason to create new NgModules in a new project. Use standalone components with route-level providers for scoped services and `providedIn: 'root'` for global services.

---

## Best Practices

- **Default to standalone for all new code.** Every new component, directive, and pipe should be standalone. Only interact with NgModules when consuming legacy libraries.

- **Use `providedIn: 'root'` for singleton services.** This eliminates the need for `forRoot`, `CoreModule`, or manual provider registration in any module or config.

- **Use route-level `providers` for scoped services.** If a service should only exist within a feature route tree, provide it in the route configuration. This replaces the lazy-loaded module injector pattern.

- **Migrate incrementally.** Use the `@angular/core:standalone` schematic to convert one feature at a time. Start with leaf components (those not used by many others) and work inward.

- **Import individual items, not entire modules.** In standalone components, import `MatButtonModule` only if you cannot import the standalone `MatButton` directly. As libraries migrate, prefer granular imports.

- **Do not re-export modules from a "shared" barrel in standalone projects.** The NgModule-era pattern of a `SharedModule` that re-exports `CommonModule`, `FormsModule`, etc. is an anti-pattern in standalone. Each component should declare its own imports.

## Real-World Use Cases

### 1. Legacy Codebase with Hybrid Approach

A large enterprise app has 200+ components in NgModules. The team migrates incrementally:

```typescript
// New feature: fully standalone
@Component({
  imports: [ReactiveFormsModule, MatInputModule, MatButtonModule],
  template: `
    <form [formGroup]="form">
      <mat-form-field>
        <input matInput formControlName="name" />
      </mat-form-field>
      <button mat-raised-button type="submit">Save</button>
    </form>
  `
})
export class NewFeatureComponent {
  private fb = inject(FormBuilder);
  form = this.fb.group({ name: [''] });
}

// Legacy feature module still exists, untouched
@NgModule({
  declarations: [OldDashboardComponent, OldWidgetComponent],
  imports: [CommonModule, SharedModule]
})
export class LegacyDashboardModule {}

// Routes mix both approaches
const routes: Routes = [
  {
    path: 'new-feature',
    loadComponent: () => import('./new-feature.component')
      .then(m => m.NewFeatureComponent)
  },
  {
    path: 'dashboard',
    loadChildren: () => import('./legacy/dashboard.module')
      .then(m => m.LegacyDashboardModule)
  }
];
```

### 2. Third-Party Library Consumption

Using a library that only exports NgModules:

```typescript
// The library exports an NgModule
// import { ChartModule } from 'third-party-charts';

// Standalone component imports the NgModule directly
@Component({
  imports: [ChartModule],
  template: `<tp-chart [data]="chartData()" [options]="chartOptions" />`
})
export class AnalyticsComponent {
  chartData = input.required<ChartData>();
  chartOptions = { responsive: true, type: 'line' };
}

// If the library uses forRoot, use importProvidersFrom in app.config.ts
export const appConfig: ApplicationConfig = {
  providers: [
    importProvidersFrom(ChartModule.forRoot({ license: 'abc123' }))
  ]
};
```

### 3. Scoped Service with Route Providers (Replacing Module Providers)

A checkout flow needs a `CartService` that resets when the user navigates away:

```typescript
// cart.service.ts
@Injectable()  // NOT providedIn: 'root', so it is scoped
export class CartService {
  items = signal<CartItem[]>([]);

  addItem(item: CartItem) {
    this.items.update(current => [...current, item]);
  }

  clear() {
    this.items.set([]);
  }

  total = computed(() =>
    this.items().reduce((sum, item) => sum + item.price * item.quantity, 0)
  );
}

// checkout.routes.ts
export const checkoutRoutes: Routes = [
  {
    path: '',
    providers: [CartService],  // New instance for this route tree
    children: [
      {
        path: '',
        loadComponent: () => import('./cart.component').then(m => m.CartComponent)
      },
      {
        path: 'shipping',
        loadComponent: () => import('./shipping.component').then(m => m.ShippingComponent)
      },
      {
        path: 'payment',
        loadComponent: () => import('./payment.component').then(m => m.PaymentComponent)
      }
    ]
  }
];
```

When the user navigates to `/checkout`, a new `CartService` is created. All child routes share that instance. When the user navigates away, the service is destroyed. This is the standalone equivalent of the lazy-loaded module injector pattern.

---

> **Key Takeaway:** Standalone components are the default and recommended approach in Angular 21, eliminating the complexity of NgModules for new code. Understand NgModules so you can work with legacy code and third-party libraries, but do not create new ones.
