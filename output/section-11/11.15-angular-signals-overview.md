# 11.15 Angular Signals Overview

> Signals are Angular's synchronous, pull-based reactivity primitive. They replace zone-based dirty checking with explicit, fine-grained change tracking that makes your application faster and your code easier to reason about.

---

## What Are Signals?

A signal is a wrapper around a value that notifies consumers when that value changes. Unlike observables (push-based, asynchronous), signals are pull-based and synchronous. You read a signal's current value by calling it as a getter function, and the reactivity system tracks which signals were read during a computation.

```typescript
import { signal } from '@angular/core';

// Create a writable signal with an initial value
const count = signal(0);

// Read the value by calling the signal as a function
console.log(count()); // 0

// Set a new value
count.set(5);
console.log(count()); // 5

// Update based on the current value
count.update(current => current + 1);
console.log(count()); // 6
```

Signals integrate directly with Angular's change detection. When a signal used in a template changes, Angular knows exactly which component needs re-rendering without traversing the entire component tree.

---

## The Signal Primitives

Angular's signal system consists of five core primitives:

| Primitive | Purpose | Writable | Reactive Source |
|-----------|---------|----------|-----------------|
| `signal()` | Holds mutable state | Yes | Self |
| `computed()` | Derives values from other signals | No (read-only) | Other signals |
| `effect()` | Runs side effects when signals change | N/A | Other signals |
| `linkedSignal()` | Writable signal that resets when a source changes | Yes | Other signals + manual |
| `resource()` / `rxResource()` | Bridges async operations into signal world | Partially (via `.set()`) | Other signals + async |

---

## WritableSignal in Depth

`signal()` returns a `WritableSignal<T>`. It has three methods:

### set()

Replaces the current value entirely.

```typescript
const name = signal('Alice');
name.set('Bob');
```

### update()

Derives the next value from the current value. Useful when the new value depends on the old.

```typescript
const items = signal<string[]>([]);

// Add an item immutably
items.update(current => [...current, 'new item']);
```

### asReadonly()

Returns a `Signal<T>` (read-only) that mirrors the writable signal. Use this to expose state to consumers without giving them write access.

```typescript
@Component({
  selector: 'app-counter',
  template: `<span>{{ count() }}</span>`,
})
export class CounterComponent {
  private _count = signal(0);

  // Public API: read-only
  readonly count = this._count.asReadonly();

  increment() {
    this._count.update(n => n + 1);
  }
}
```

### The mutate() Method Was Removed

Earlier developer previews included a `mutate()` method that allowed in-place mutation of objects/arrays. It was removed from the public API because signal change notification is lost when the value is read outside a reactive context. The team decided immutable updates via `update()` are the correct pattern.

```typescript
// BAD: This API no longer exists
// items.mutate(arr => arr.push('new'));

// GOOD: Use update() with immutable operations
items.update(arr => [...arr, 'new']);
```

## Equality Functions

By default, signals use `Object.is()` for equality checking. If the new value is equal to the old value, dependents are not notified. You can provide a custom equality function.

```typescript
import { signal } from '@angular/core';

// Default: Object.is() comparison
const count = signal(0);
count.set(0); // No notification, same value

// Custom equality for objects
const user = signal(
  { name: 'Alice', age: 30 },
  { equal: (a, b) => a.name === b.name && a.age === b.age }
);

user.set({ name: 'Alice', age: 30 }); // No notification, equal by custom check
user.set({ name: 'Alice', age: 31 }); // Notifies dependents
```

**When to use custom equality:**

- Rarely. The default `Object.is()` works correctly in the vast majority of cases.
- Custom equality functions introduce complexity and potential for subtle bugs.
- If you find yourself needing custom equality often, reconsider your data structures (use primitives or immutable patterns instead).

```typescript
// BAD: Overly clever equality that hides bugs
const config = signal(
  { theme: 'dark', lang: 'en' },
  { equal: (a, b) => a.theme === b.theme } // Ignores lang changes!
);

// GOOD: Just let Object.is() do its job with immutable updates
const config = signal({ theme: 'dark', lang: 'en' });
```

## computed()

A `computed()` signal derives its value from one or more source signals. It is lazy (does not compute until first read), cached (recomputes only when dependencies change), and read-only.

```typescript
import { signal, computed } from '@angular/core';

const firstName = signal('John');
const lastName = signal('Doe');

// Computed derives from firstName and lastName
const fullName = computed(() => `${firstName()} ${lastName()}`);

console.log(fullName()); // "John Doe"

firstName.set('Jane');
console.log(fullName()); // "Jane Doe"
```

### Dependency Tracking Is Automatic

Angular tracks which signals are read during the computation function. If a signal is not read in a given execution (e.g., due to a conditional), it is not tracked for that run.

```typescript
const showFullName = signal(true);
const firstName = signal('John');
const lastName = signal('Doe');
const nickname = signal('JD');

const displayName = computed(() => {
  if (showFullName()) {
    return `${firstName()} ${lastName()}`; // Tracks: showFullName, firstName, lastName
  }
  return nickname(); // Tracks: showFullName, nickname
});
```

### Rules for computed()

1. **Must be pure.** No side effects. No DOM manipulation. No HTTP calls. No writing to other signals.
2. **Must be synchronous.** No `setTimeout`, no Promises, no async/await.
3. **Should not throw.** An error in a computed propagates to every consumer.

```typescript
// BAD: Side effect in computed
const total = computed(() => {
  const t = price() * quantity();
  console.log('Total changed!'); // Side effect
  return t;
});

// BAD: Async in computed
const userData = computed(async () => {
  return await fetch('/api/user'); // Signals are synchronous
});

// GOOD: Pure derivation
const total = computed(() => price() * quantity());
```

### Computed Signals with Custom Equality

Like `signal()`, `computed()` also accepts a custom equality function. The computed value is only propagated to dependents if the new result is not equal to the previous one.

```typescript
const items = signal([1, 2, 3]);

// Only notify dependents if the count actually changes
const itemCount = computed(
  () => items().length,
  { equal: (a, b) => a === b } // Numbers already use Object.is(), this is redundant but illustrative
);
```

## effect()

An `effect()` runs a callback whenever any signal it reads changes. Effects are for side effects: DOM manipulation, logging, syncing with external systems, making API calls.

```typescript
import { signal, effect } from '@angular/core';

const currentUser = signal('Alice');

// Runs once immediately, then whenever currentUser changes
effect(() => {
  console.log(`Current user: ${currentUser()}`);
});

currentUser.set('Bob'); // Logs: "Current user: Bob"
```

### Where Effects Can Be Created

Effects require an injection context. The most common places:

```typescript
@Component({
  selector: 'app-dashboard',
  template: `...`,
})
export class DashboardComponent {
  private userService = inject(UserService);
  private count = signal(0);

  // 1. In a field initializer (has injection context)
  private logger = effect(() => {
    console.log(`Count is ${this.count()}`);
  });

  constructor() {
    // 2. In the constructor
    effect(() => {
      document.title = `Dashboard - ${this.userService.name()}`;
    });
  }
}

// 3. In a service
@Injectable({ providedIn: 'root' })
export class ThemeService {
  private theme = signal<'light' | 'dark'>('light');

  constructor() {
    effect(() => {
      document.documentElement.setAttribute('data-theme', this.theme());
    });
  }
}
```

To create an effect outside an injection context, pass an `Injector` explicitly:

```typescript
export class SomeComponent {
  private injector = inject(Injector);

  ngOnInit() {
    effect(() => {
      // ...
    }, { injector: this.injector });
  }
}
```

### Effect Cleanup

The callback receives an `onCleanup` function to register teardown logic. The cleanup runs before the next execution and when the effect is destroyed.

```typescript
effect((onCleanup) => {
  const query = searchTerm();
  const controller = new AbortController();

  fetch(`/api/search?q=${query}`, { signal: controller.signal })
    .then(res => res.json())
    .then(data => results.set(data));

  onCleanup(() => {
    controller.abort(); // Cancel the previous request
  });
});
```

Common cleanup scenarios:
- Aborting HTTP requests
- Clearing `setTimeout` / `setInterval`
- Removing event listeners
- Closing WebSocket connections

### Writing to Signals Inside Effects

As of Angular 19, effects can write to signals by default. The `allowSignalWrites` option was removed entirely.

```typescript
const source = signal(0);
const doubled = signal(0);

effect(() => {
  // This works without any special flag in Angular 19+
  doubled.set(source() * 2);
});
```

However, this pattern is often a code smell. If you are deriving a value, prefer `computed()` or `linkedSignal()`:

```typescript
// BETTER: Use computed for pure derivation
const doubled = computed(() => source() * 2);

// BETTER: Use linkedSignal if you need it writable + derived
const doubled = linkedSignal(() => source() * 2);
```

**When to actually write signals in effects:**
- Syncing with external non-reactive APIs (localStorage, cookies, third-party libraries)
- Bridging imperative code that cannot be refactored

### untracked()

The `untracked()` function reads a signal's value without adding it as a dependency.

```typescript
import { untracked } from '@angular/core';

const userId = signal(1);
const logCount = signal(0);

effect(() => {
  // userId is tracked: effect re-runs when userId changes
  const id = userId();

  // logCount is NOT tracked: changes to logCount do NOT re-trigger the effect
  const currentLogCount = untracked(() => logCount());

  console.log(`Logging user ${id}, log entry #${currentLogCount}`);
  logCount.update(n => n + 1);
});
```

Use `untracked()` when you need to read a signal inside an effect or computed without creating a dependency on it.

## linkedSignal()

`linkedSignal()` creates a writable signal whose value resets whenever a source signal changes. It bridges the gap between `computed()` (read-only, always derived) and `signal()` (writable, never derived).

### Shorthand Syntax

The simplest form takes a computation function:

```typescript
import { signal, linkedSignal } from '@angular/core';

const options = signal(['Standard', 'Express', 'Overnight']);
const selectedOption = linkedSignal(() => options()[0]);

console.log(selectedOption()); // "Standard"

// User manually selects a different option
selectedOption.set('Express');
console.log(selectedOption()); // "Express"

// When the source changes, linkedSignal resets
options.set(['Economy', 'Priority']);
console.log(selectedOption()); // "Economy" (reset to first item)
```

### Full Syntax

For more control, pass `source` and `computation` separately. The `computation` receives the new source value and the previous value of the linked signal:

```typescript
const page = signal(1);
const pageSize = signal(10);

const offset = linkedSignal({
  source: () => ({ page: page(), size: pageSize() }),
  computation: (source, previous) => {
    // If only pageSize changed, try to keep the current offset
    if (previous && source.page === previous.source.page) {
      return previous.value;
    }
    // Otherwise, calculate from page and size
    return (source.page - 1) * source.size;
  },
});
```

### When to Use linkedSignal vs computed vs signal

| Scenario | Use |
|----------|-----|
| Read-only derived value | `computed()` |
| Writable, no derived reset | `signal()` |
| Writable, resets when source changes | `linkedSignal()` |
| Writable, needs complex reset logic | `linkedSignal()` with full syntax |

Common use cases for `linkedSignal()`:
- Selected item in a list that resets when the list changes
- Form defaults that reset when the entity being edited changes
- Pagination offset that resets when filters change
- Sort direction that resets when the sort column changes

```typescript
// Pagination example
const currentFilter = signal('');
const currentPage = linkedSignal(() => {
  currentFilter(); // Track the filter
  return 1;        // Reset to page 1 whenever filter changes
});

// User navigates to page 3
currentPage.set(3);

// User changes filter: page resets to 1
currentFilter.set('angular');
console.log(currentPage()); // 1
```

## Resource APIs: Async Data with Signals

### resource()

The `resource()` function bridges async operations into the signal world. It runs a loader function whenever its request signal changes and exposes the result as a signal.

```typescript
import { signal, resource } from '@angular/core';

const userId = signal(1);

const userResource = resource({
  request: () => userId(),
  loader: async ({ request: id, abortSignal }) => {
    const response = await fetch(`/api/users/${id}`, { signal: abortSignal });
    return response.json();
  },
});

// Access the data synchronously via signals
console.log(userResource.value());   // User object or undefined
console.log(userResource.status());  // 'idle' | 'loading' | 'resolved' | 'error' | 'reloading' | 'local'
console.log(userResource.error());   // Error object or undefined
```

Key features of `resource()`:
- Automatic request cancellation when the request signal changes (via `abortSignal`)
- Race condition prevention: only the latest request's result is used
- Status tracking: idle, loading, resolved, error, reloading
- `reload()` method to force a re-fetch with the same parameters

### rxResource()

For RxJS-based loaders, `rxResource()` accepts an Observable-returning loader:

```typescript
import { signal } from '@angular/core';
import { rxResource } from '@angular/core/rxjs-interop';

const userId = signal(1);

const userResource = rxResource({
  request: () => userId(),
  loader: ({ request: id }) => {
    return inject(HttpClient).get<User>(`/api/users/${id}`);
  },
});
```

### httpResource() (Experimental)

`httpResource()` simplifies HTTP-based resources by wrapping `HttpClient` internally:

```typescript
import { httpResource } from '@angular/common/http';

const userId = signal(1);

// Simple GET request
const userResource = httpResource<User>(() => `/api/users/${userId()}`);

// With full request configuration
const userResource = httpResource<User>(() => ({
  url: `/api/users/${userId()}`,
  method: 'GET',
  headers: { 'Accept': 'application/json' },
}));
```

### Resource Status Pattern

Use the status signal to drive template rendering:

```typescript
@Component({
  template: `
    @switch (userResource.status()) {
      @case ('loading') {
        <app-spinner />
      }
      @case ('error') {
        <app-error [error]="userResource.error()" />
      }
      @case ('resolved') {
        <app-user-profile [user]="userResource.value()!" />
      }
    }
  `,
})
export class UserProfileContainer {
  private userId = signal(1);

  protected userResource = resource({
    request: () => this.userId(),
    loader: async ({ request: id, abortSignal }) => {
      const res = await fetch(`/api/users/${id}`, { signal: abortSignal });
      if (!res.ok) throw new Error(`Failed to load user ${id}`);
      return res.json() as Promise<User>;
    },
  });
}
```

## RxJS Interop: toSignal() and toObservable()

The `@angular/core/rxjs-interop` package provides bridges between signals and observables.

### toSignal()

Converts an Observable into a read-only Signal. The subscription is managed automatically and unsubscribes when the component/service is destroyed.

```typescript
import { toSignal } from '@angular/core/rxjs-interop';
import { interval } from 'rxjs';

@Component({
  template: `<p>Seconds: {{ seconds() }}</p>`,
})
export class TimerComponent {
  // Observable -> Signal
  protected seconds = toSignal(interval(1000), { initialValue: 0 });
}
```

Options for `toSignal()`:

| Option | Description |
|--------|-------------|
| `initialValue` | Starting value before first emission. Without this, the signal type is `T \| undefined`. |
| `requireSync` | Asserts the Observable emits synchronously (e.g., BehaviorSubject). Removes `undefined` from the type. |
| `manualCleanup` | Prevents automatic unsubscription on destroy. |
| `rejectErrors` | When `false`, errors are swallowed. Default is `true` (errors throw). |
| `equal` | Custom equality function for the resulting signal. |

```typescript
// With BehaviorSubject: no undefined in the type
const subject = new BehaviorSubject(42);
const value = toSignal(subject, { requireSync: true });
// value: Signal<number> (not Signal<number | undefined>)

// With initialValue
const data = toSignal(someObservable$, { initialValue: [] });
// data: Signal<SomeType[]> (not Signal<SomeType[] | undefined>)
```

### toObservable()

Converts a Signal into an Observable. Uses an internal `effect()` and a `ReplaySubject` to emit values.

```typescript
import { toObservable } from '@angular/core/rxjs-interop';
import { signal } from '@angular/core';
import { debounceTime, distinctUntilChanged, switchMap } from 'rxjs';

@Component({
  template: `<input (input)="onSearch($event)" />`,
})
export class SearchComponent {
  private searchTerm = signal('');
  private http = inject(HttpClient);

  // Signal -> Observable for RxJS operator chains
  private results$ = toObservable(this.searchTerm).pipe(
    debounceTime(300),
    distinctUntilChanged(),
    switchMap(term => this.http.get<Result[]>(`/api/search?q=${term}`))
  );

  // Convert back to signal for the template
  protected results = toSignal(this.results$, { initialValue: [] });

  onSearch(event: Event) {
    this.searchTerm.set((event.target as HTMLInputElement).value);
  }
}
```

### When to Use toSignal vs toObservable

| Direction | When |
|-----------|------|
| Observable -> Signal (`toSignal`) | Reading observable data in templates. Consuming existing RxJS services in signal-based components. |
| Signal -> Observable (`toObservable`) | Applying RxJS operators (debounce, switchMap, combineLatest) to signal values. Integrating with existing RxJS pipelines. |

## Signal-Based Component APIs

Angular has introduced signal-based replacements for most decorator-based component APIs:

| Decorator | Signal Replacement | Status |
|-----------|-------------------|--------|
| `@Input()` | `input()` / `input.required()` | Stable |
| `@Output()` | `output()` | Stable |
| `@Input() + @Output()` (two-way) | `model()` | Stable |
| `@ViewChild()` | `viewChild()` / `viewChild.required()` | Stable |
| `@ViewChildren()` | `viewChildren()` | Stable |
| `@ContentChild()` | `contentChild()` / `contentChild.required()` | Stable |
| `@ContentChildren()` | `contentChildren()` | Stable |

These are covered in depth in Sections 12.5 through 12.9, but here is a quick preview:

```typescript
@Component({
  selector: 'app-user-card',
  template: `
    <h2>{{ name() }}</h2>
    @if (showDetails()) {
      <p>Email: {{ email() }}</p>
    }

export class UserCardComponent {
```

## Signals and Change Detection

### Zone-Based (Legacy)

With Zone.js, Angular patches async APIs (setTimeout, Promise, fetch, etc.) and runs change detection on the entire component tree after any async operation. This is wasteful: most components have not changed.

### Signal-Based (Zoneless)

With signals, Angular knows exactly which components have changed because signals notify the framework directly. Starting with Angular 21, zoneless is the default for new applications.

```typescript
// Angular 21: New projects are zoneless by default
// For Angular 19-20, opt in explicitly:
export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
    // ...
  ],
};
```

What signals do for change detection:
- A signal read in a template marks that component as potentially dirty
- When the signal's value changes, the component is marked for check
- Only marked components are re-rendered

This eliminates:
- The need for `OnPush` change detection strategy (all components are effectively OnPush)
- The need for Zone.js (~13-15 KB bundle savings)
- The need for `markForCheck()` / `detectChanges()` calls in most cases

```typescript
@Component({
  selector: 'app-user',
  template: `
    <h1>{{ user().name }}</h1>
    <p>Visits: {{ visitCount() }}</p>
    <button (click)="incrementVisits()">+1</button>
  `,
})
export class UserComponent {
  user = input.required<User>();
  visitCount = signal(0);

  incrementVisits() {
    // Angular knows this component needs re-rendering
    // because visitCount changed
    this.visitCount.update(n => n + 1);
  }
}
```

## The Signal Graph

Signals form a directed acyclic graph (DAG) of dependencies:

```
signal(count)  ------>  computed(doubled)  ------>  effect(log)
                  |
signal(factor) --/

signal(name)  ------>  computed(greeting)
```

Key properties:
- **Glitch-free:** All dependents see a consistent state. If `count` and `factor` both change, `doubled` sees both new values, not a mix of old and new.
- **Lazy evaluation:** `computed()` values are only recalculated when read, not when dependencies change.
- **Push-pull hybrid:** Notifications that something changed are pushed, but the actual recomputation is pulled (deferred until the value is read).

```typescript
const count = signal(0);
const factor = signal(2);
const doubled = computed(() => count() * factor());
const label = computed(() => `Result: ${doubled()}`);

// Even if we update both count and factor,
// label sees a consistent snapshot
count.set(5);
factor.set(3);

// When label is finally read (e.g., in a template render),
// it computes: "Result: 15" -- never "Result: 10" (stale factor)
console.log(label()); // "Result: 15"
```

## Common Mistakes

### 1. Using effect() When computed() or linkedSignal() Would Do

```typescript
// BAD: Unnecessary effect to derive state
const firstName = signal('John');
const lastName = signal('Doe');
const fullName = signal('');
effect(() => {
  fullName.set(`${firstName()} ${lastName()}`);
});

// GOOD: Use computed for derived values
const fullName = computed(() => `${firstName()} ${lastName()}`);
```

### 2. Side Effects in computed()

```typescript
// BAD: Logging, HTTP calls, or DOM manipulation in computed
const total = computed(() => {
  const t = price() * quantity();
  analyticsService.track('total_calculated', t); // Side effect!
  return t;
});

// GOOD: Keep computed pure, use effect for side effects
const total = computed(() => price() * quantity());
effect(() => {
  analyticsService.track('total_calculated', total());
});
```

### 3. Forgetting That Signals Are Functions

```typescript
// BAD: Passing the signal itself instead of calling it
const name = signal('Alice');
console.log(name);   // [Function: signal] -- not "Alice"

// GOOD: Call the signal to get its value
console.log(name()); // "Alice"
```

### 4. Mutating Objects In Place

```typescript
// BAD: Mutating the same reference
const users = signal<User[]>([]);
const currentUsers = users();
currentUsers.push(newUser); // Mutating in place
users.set(currentUsers);    // Same reference, Object.is() returns true, NO notification

// GOOD: Create a new reference
users.update(current => [...current, newUser]);
```

### 5. Creating Effects Without an Injection Context

```typescript
// BAD: effect() outside injection context
ngAfterViewInit() {
  effect(() => { ... }); // ERROR: NG0203 inject() must be called from injection context
}

// GOOD: Pass an injector
private injector = inject(Injector);
ngAfterViewInit() {
  effect(() => { ... }, { injector: this.injector });
}
```

### 6. Overusing Custom Equality Functions

```typescript
// BAD: Complex equality that ignores important fields
const config = signal(
  { theme: 'dark', fontSize: 14, lang: 'en' },
  { equal: (a, b) => a.theme === b.theme } // Ignores fontSize and lang changes!
);

// GOOD: Use default equality with immutable updates
const config = signal({ theme: 'dark', fontSize: 14, lang: 'en' });
```

### 7. Reading Signals Outside Reactive Context and Expecting Updates

```typescript
// BAD: Reading once, expecting it to stay updated
const count = signal(0);
const currentCount = count(); // Reads once: 0
// currentCount is just a number, not reactive

// GOOD: Read inside a reactive context
computed(() => count()); // Re-evaluates when count changes
effect(() => console.log(count())); // Re-runs when count changes
// In templates: {{ count() }} -- tracked by Angular's template renderer
```

## Best Practices

### 1. Prefer computed() Over effect() for Derived State

`computed()` is lazy, cached, glitch-free, and side-effect-free. `effect()` runs eagerly and is harder to reason about. Always reach for `computed()` first.

### 2. Keep Signals Close to Their Consumers

Define signals in the component or service that owns the state. Expose read-only versions to consumers via `asReadonly()`.

```typescript
@Injectable({ providedIn: 'root' })
export class CartService {
  private _items = signal<CartItem[]>([]);
  readonly items = this._items.asReadonly();
  readonly total = computed(() =>
    this._items().reduce((sum, item) => sum + item.price * item.quantity, 0)
  );

  addItem(item: CartItem) {
    this._items.update(items => [...items, item]);
  }
}
```

### 3. Use Immutable Update Patterns

Signals rely on reference equality by default. Always create new objects/arrays when updating.

```typescript
// Objects
user.update(u => ({ ...u, name: 'New Name' }));

// Arrays: add
items.update(arr => [...arr, newItem]);

// Arrays: remove
items.update(arr => arr.filter(item => item.id !== targetId));

// Arrays: update one item
items.update(arr =>
  arr.map(item => item.id === targetId ? { ...item, done: true } : item)
);
```

### 4. Use untracked() to Prevent Unnecessary Re-runs

When an effect needs to read a signal for a one-time value but should not re-run when that signal changes, use `untracked()`.

### 5. Use linkedSignal() for Resettable State

Whenever you have local state that should reset when some source changes (selected item, pagination, form defaults), `linkedSignal()` is the right tool.

### 6. Hierarchy of Preference

When deciding which signal primitive to use, follow this order:

1. `computed()` -- for read-only derived values
2. `linkedSignal()` -- for writable values that reset when a source changes
3. `signal()` -- for independent writable state
4. `effect()` -- for side effects (last resort)

## Real-World Use Cases

### Shopping Cart State Management

```typescript
@Injectable({ providedIn: 'root' })
export class CartService {
  private _items = signal<CartItem[]>([]);
  readonly items = this._items.asReadonly();

  readonly itemCount = computed(() => this._items().length);
  readonly subtotal = computed(() =>
    this._items().reduce((sum, i) => sum + i.price * i.qty, 0)
  );
  readonly tax = computed(() => this.subtotal() * 0.08);
  readonly total = computed(() => this.subtotal() + this.tax());

  // Sync cart to localStorage
  private persist = effect(() => {
    const items = this._items();
    localStorage.setItem('cart', JSON.stringify(items));
  });

  addItem(product: Product) {
    this._items.update(items => {
      const existing = items.find(i => i.productId === product.id);
      if (existing) {
        return items.map(i =>
          i.productId === product.id ? { ...i, qty: i.qty + 1 } : i
        );
      }
      return [...items, { productId: product.id, name: product.name, price: product.price, qty: 1 }];
    });
  }

  removeItem(productId: string) {
    this._items.update(items => items.filter(i => i.productId !== productId));
  }

  clear() {
    this._items.set([]);
  }
}
```

### Paginated Data Table with Filters

```typescript
@Component({
  selector: 'app-data-table',
  template: `
    <input (input)="onSearch($event)" placeholder="Search..." />

    <select (change)="onCategoryChange($event)">
      @for (cat of categories(); track cat) {
        <option [value]="cat">{{ cat }}</option>
      }
    </select>

    <p>Page {{ currentPage() }} of {{ totalPages() }}</p>

    @switch (dataResource.status()) {
      @case ('loading') { <app-spinner /> }
      @case ('error') { <p>Error loading data</p> }
      @case ('resolved') {
        <table>
          @for (row of dataResource.value()!.items; track row.id) {
            <tr>
              <td>{{ row.name }}</td>
              <td>{{ row.category }}</td>
            </tr>
          }
        </table>
      }
    }

    <button (click)="prevPage()" [disabled]="currentPage() <= 1">Previous</button>
    <button (click)="nextPage()" [disabled]="currentPage() >= totalPages()">Next</button>
  `,
})
export class DataTableComponent {
  private http = inject(HttpClient);

  categories = signal(['All', 'Electronics', 'Books', 'Clothing']);
  searchTerm = signal('');
  selectedCategory = signal('All');
  pageSize = signal(20);

  // Reset to page 1 when filters change
  currentPage = linkedSignal(() => {
    this.searchTerm();
    this.selectedCategory();
    return 1;
  });

  protected dataResource = rxResource({
    request: () => ({
      search: this.searchTerm(),
      category: this.selectedCategory(),
      page: this.currentPage(),
      size: this.pageSize(),
    }),
    loader: ({ request: params }) =>
      this.http.get<PaginatedResponse>('/api/items', { params: params as any }),
  });

  totalPages = computed(() => {
    const data = this.dataResource.value();
    return data ? Math.ceil(data.total / this.pageSize()) : 0;
  });

  onSearch(event: Event) {
    this.searchTerm.set((event.target as HTMLInputElement).value);
  }

  onCategoryChange(event: Event) {
    this.selectedCategory.set((event.target as HTMLSelectElement).value);
  }

  nextPage() {
    this.currentPage.update(p => p + 1);
  }

  prevPage() {
    this.currentPage.update(p => Math.max(1, p - 1));
  }
}
```

### Theme Service with System Preference Detection

```typescript
@Injectable({ providedIn: 'root' })
export class ThemeService {
  private _theme = signal<'light' | 'dark'>(this.getInitialTheme());
  readonly theme = this._theme.asReadonly();
  readonly isDark = computed(() => this._theme() === 'dark');

  constructor() {
    // Sync theme to DOM and localStorage
    effect(() => {
      const theme = this._theme();
      document.documentElement.setAttribute('data-theme', theme);
      localStorage.setItem('theme', theme);
    });

    // Listen to system preference changes
    const mediaQuery = window.matchMedia('(prefers-color-scheme: dark)');
    mediaQuery.addEventListener('change', (e) => {
      if (!localStorage.getItem('theme')) {
        this._theme.set(e.matches ? 'dark' : 'light');
      }
    });
  }

  toggle() {
    this._theme.update(t => t === 'light' ? 'dark' : 'light');
  }

  private getInitialTheme(): 'light' | 'dark' {
    const stored = localStorage.getItem('theme') as 'light' | 'dark' | null;
    if (stored) return stored;
    return window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
  }
}
```

---

> **Key Takeaway:** Angular's signal system (signal, computed, effect, linkedSignal, resource) provides synchronous, fine-grained reactivity that replaces zone-based change detection. Prefer computed() for derived state, linkedSignal() for resettable writable state, and treat effect() as the last resort for bridging to non-reactive external systems.
