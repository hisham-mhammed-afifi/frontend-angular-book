# 11.17 Template Syntax Overview

> Angular templates are HTML enhanced with binding syntax, control flow blocks, pipes, and local variables. Understanding every binding type and template construct is essential for building Angular UIs.

---

## Text Interpolation

Double curly braces `{{ }}` evaluate a TypeScript expression and render the result as text. Angular converts the result to a string.

```html
<h1>{{ title }}</h1>
<p>Welcome, {{ user().name }}</p>
<span>Total: {{ price() * quantity() | currency }}</span>
```

Interpolation works in text content and attribute values:

```html
<!-- Text node -->
<p>Order #{{ orderId() }}</p>

<!-- Inside an attribute (less common, prefer property binding) -->
<img src="https://avatars.example.com/{{ userId() }}.png" alt="{{ username() }}">
```

Expressions are evaluated in the component's context. You can call methods, access signals, use operators, and chain pipes. However, template expressions have restrictions:

| Allowed | Not Allowed |
|---------|-------------|
| Property access: `user().name` | Assignments: `x = 5` |
| Method calls: `getTotal()` | Increment/decrement: `i++`, `--count` |
| Math: `a + b`, `a * b` | `new` keyword |
| Ternary: `active ? 'Yes' : 'No'` | Chained expressions: `a; b` |
| Nullish coalescing: `value ?? 'default'` | Bitwise operators: `a \| b` (conflicts with pipe) |
| Optional chaining: `user?.address?.city` | `typeof` |
| Pipe operator: `value \| pipeName` | Template literals: `` `hello ${name}` `` |

## Property Binding

Square brackets bind a component/directive/DOM property to an expression. The data flows one way: from the component to the template.

```html
<!-- DOM property binding -->
<img [src]="imageUrl()" [alt]="imageAlt()">
<button [disabled]="isSubmitting()">Submit</button>
<input [value]="searchTerm()">

<!-- Component input binding -->
<app-user-card [user]="selectedUser()" [showActions]="true"></app-user-card>

<!-- Directive property binding -->
<div [ngClass]="{ 'active': isActive(), 'disabled': isDisabled() }"></div>
```

### Property Binding vs Interpolation

For string properties, both work but property binding is safer:

```html
<!-- These are equivalent for string values -->
<img src="assets/{{ filename() }}" />
<img [src]="'assets/' + filename()" />

<!-- Property binding is required for non-string values -->
<app-chart [data]="chartData()"></app-chart>       <!-- Object -->
<app-toggle [enabled]="isEnabled()"></app-toggle>   <!-- Boolean -->
<app-list [items]="filteredItems()"></app-list>      <!-- Array -->

<!-- BAD: Interpolation converts to string, breaking non-string inputs -->
<app-chart data="{{ chartData() }}"></app-chart>     <!-- Passes "[object Object]" string -->
<app-toggle enabled="{{ isEnabled() }}"></app-toggle> <!-- Passes "true" string, not boolean -->
```

**Rule:** Use property binding for everything except pure text display. Use interpolation only for rendering text.

## Attribute Binding

Some HTML attributes do not have corresponding DOM properties (e.g., `colspan`, `aria-*`, `data-*`, SVG attributes). For these, use `[attr.name]`:

```html
<!-- Table colspan has no DOM property -->
<td [attr.colspan]="columnSpan()">Total</td>

<!-- ARIA attributes -->
<button [attr.aria-label]="buttonLabel()" [attr.aria-expanded]="isOpen()">
  Menu
</button>

<!-- Data attributes -->
<div [attr.data-testid]="'user-' + user().id">{{ user().name }}</div>

<!-- SVG attributes -->
<svg>
  <circle [attr.cx]="x()" [attr.cy]="y()" [attr.r]="radius()" />
</svg>
```

When to use attribute binding vs property binding:

| Use Case | Syntax |
|----------|--------|
| Standard DOM properties (`src`, `disabled`, `value`) | `[property]="expr"` |
| HTML-only attributes (`colspan`, `role`) | `[attr.name]="expr"` |
| ARIA attributes | `[attr.aria-*]="expr"` |
| Custom data attributes | `[attr.data-*]="expr"` |
| SVG attributes | `[attr.*]="expr"` |

## Class Binding

Bind CSS classes to elements conditionally.

### Single Class Binding

```html
<!-- Adds 'active' class when isActive() is truthy -->
<div [class.active]="isActive()">Tab</div>

<!-- Multiple individual class bindings -->
<button
  [class.primary]="isPrimary()"
  [class.disabled]="isDisabled()"
  [class.loading]="isLoading()">
  Save
</button>
```

### Multi-Class Binding

```html
<!-- String: space-separated class names -->
<div [class]="'card card-large'">...</div>

<!-- Array of class names -->
<div [class]="['card', 'card-large', isActive() ? 'active' : '']">...</div>

<!-- Object: keys are class names, values are booleans -->
<div [class]="{ card: true, active: isActive(), highlighted: isNew() }">...</div>
```

**Prefer `[class.name]` over `ngClass`.** The built-in class binding syntax is more performant than the `NgClass` directive because it avoids the overhead of the directive layer.

```html
<!-- GOOD: Direct class binding -->
<div [class.expanded]="isExpanded()">...</div>

<!-- Avoid when possible: NgClass directive (slower) -->
<div [ngClass]="{ expanded: isExpanded() }">...</div>
```

## Style Binding

Bind inline CSS styles to elements.

### Single Style Property

```html
<!-- Simple value -->
<div [style.background-color]="bgColor()">Colored box</div>

<!-- With unit suffix -->
<div [style.width.px]="containerWidth()">Resizable</div>
<div [style.font-size.rem]="fontSize()">Scaled text</div>
<div [style.height.%]="heightPercent()">Flexible</div>

<!-- Conditional style -->
<div [style.opacity]="isVisible() ? '1' : '0'">Fade in/out</div>
```

### Multi-Style Binding

```html
<!-- Object: keys are CSS property names, values are the styles -->
<div [style]="{ width: cardWidth(), height: cardHeight(), border: '1px solid #ccc' }">
  ...
</div>
```

Like class binding, prefer `[style.property]` over the `NgStyle` directive for better performance.

## Event Binding

Parentheses bind a DOM event or component output to a statement that runs when the event fires.

```html
<!-- DOM events -->
<button (click)="onSave()">Save</button>
<input (input)="onSearch($event)" (keydown.enter)="submitSearch()">
<div (mouseenter)="onHover(true)" (mouseleave)="onHover(false)">Hover me</div>
<form (submit)="onSubmit($event)">...</form>

<!-- Component output events -->
<app-color-picker (colorChange)="onColorChange($event)"></app-color-picker>
<app-modal (closed)="onModalClosed()"></app-modal>
```

### The $event Object

`$event` contains the event payload. For DOM events, it is the native `Event` object. For component outputs, it is whatever value the output emitted.

```typescript
@Component({
  template: `
    <input (input)="onInput($event)" />
    <app-rating (ratingChange)="onRating($event)" />
  `,
})
export class FormComponent {
  onInput(event: Event) {
    const value = (event.target as HTMLInputElement).value;
    this.searchTerm.set(value);
  }

  onRating(stars: number) {
    this.rating.set(stars);
  }
}
```

### Key Event Filtering

Angular supports filtering keyboard events by key name:

```html
<!-- Only fires on Enter key -->
<input (keydown.enter)="submit()">

<!-- Modifier keys -->
<input (keydown.control.s)="save()">
<input (keydown.shift.tab)="focusPrevious()">
<input (keydown.alt.enter)="submitAndContinue()">

<!-- Specific keys -->
<input (keyup.escape)="cancel()">
<input (keydown.arrowDown)="selectNext()">
```

## Two-Way Binding

Two-way binding combines property binding and event binding. The "banana in a box" syntax `[()]` binds a value in both directions: component to template and template to component.

### With model() Signal Inputs

The modern approach uses `model()` in child components:

```typescript
// child component
@Component({
  selector: 'app-toggle',
  template: `
    <button (click)="enabled.set(!enabled())">
      {{ enabled() ? 'ON' : 'OFF' }}
    </button>
  `,
})
export class ToggleComponent {
  enabled = model(false); // Two-way bindable signal
}

// parent template
<app-toggle [(enabled)]="isFeatureEnabled" />
```

When the child calls `enabled.set(newValue)`, the parent's `isFeatureEnabled` signal is automatically updated.

### With ngModel (Forms)

For native form elements, `ngModel` provides two-way binding:

```typescript
import { FormsModule } from '@angular/forms';

@Component({
  imports: [FormsModule],
  template: `
    <input [(ngModel)]="username" placeholder="Username" />
    <p>Hello, {{ username() }}</p>
  `,
})
export class LoginComponent {
  username = signal('');
}
```

### How Two-Way Binding Works Under the Hood

`[(prop)]="expr"` desugars to:

```html
<!-- This shorthand: -->
<app-toggle [(enabled)]="isEnabled" />

<!-- Is equivalent to: -->
<app-toggle [enabled]="isEnabled()" (enabledChange)="isEnabled.set($event)" />
```

For `model()` inputs, Angular generates the `enabledChange` output automatically. For legacy `@Input()` / `@Output()` pairs, you must follow the naming convention: if the input is `value`, the output must be `valueChange`.

## Control Flow Blocks

Angular's built-in control flow replaces the structural directives `*ngIf`, `*ngFor`, and `[ngSwitch]`. The new syntax is faster, requires no imports, and has a cleaner visual structure.

### @if / @else if / @else

```html
@if (user()) {
  <h1>Welcome, {{ user()!.name }}</h1>
} @else if (isLoading()) {
  <app-spinner />
} @else {
  <a routerLink="/login">Sign in</a>
}
```

#### Saving the Conditional Result with `as`

```html
<!-- Store the result of user() in a local variable -->
@if (user(); as u) {
  <p>{{ u.name }} ({{ u.email }})</p>
}

<!-- Useful for async pipe results -->
@if (data$ | async; as data) {
  <app-chart [data]="data" />
}
```

### @for / @empty

```html
@for (product of products(); track product.id) {
  <app-product-card [product]="product" />
} @empty {
  <p>No products found.</p>
}
```

The `track` expression is **required**. It tells Angular how to identify each item for efficient DOM reuse.

#### Track Expression Options

```html
<!-- Track by a unique property (most common) -->
@for (item of items(); track item.id) { ... }

<!-- Track by index (use only for static lists that never reorder) -->
@for (item of items(); track $index) { ... }

<!-- Track by the item reference itself (for primitive arrays) -->
@for (name of names(); track name) { ... }

<!-- Track by a computed value -->
@for (item of items(); track item.type + '-' + item.id) { ... }
```

#### Implicit Variables in @for

| Variable | Type | Description |
|----------|------|-------------|
| `$index` | `number` | Zero-based index of the current item |
| `$count` | `number` | Total number of items |
| `$first` | `boolean` | True if this is the first item |
| `$last` | `boolean` | True if this is the last item |
| `$even` | `boolean` | True if the index is even |
| `$odd` | `boolean` | True if the index is odd |

```html
@for (item of items(); track item.id; let idx = $index; let isLast = $last) {
  <div [class.last-item]="isLast">
    {{ idx + 1 }}. {{ item.name }}
  </div>
}
```

### @switch / @case / @default

```html
@switch (status()) {
  @case ('pending') {
    <app-badge color="yellow">Pending</app-badge>
  }
  @case ('active') {
    <app-badge color="green">Active</app-badge>
  }
  @case ('cancelled') {
    <app-badge color="red">Cancelled</app-badge>
  }
  @default {
    <app-badge color="gray">Unknown</app-badge>
  }
}
```

`@switch` uses strict equality (`===`) for comparison. Unlike JavaScript `switch`, there is no fall-through between cases.

### Comparison: New Control Flow vs Legacy Directives

| Feature | New Syntax | Legacy Directive |
|---------|-----------|-----------------|
| Conditional | `@if (cond) {}` | `*ngIf="cond"` |
| Else clause | `@else {}` | `*ngIf="cond; else tmpl"` with `<ng-template #tmpl>` |
| Else-if | `@else if (cond2) {}` | Not directly possible; nest `*ngIf` inside `else` template |
| Loop | `@for (item of list; track item.id) {}` | `*ngFor="let item of list; trackBy: trackFn"` |
| Empty state | `@empty {}` | Manual `*ngIf="list.length === 0"` |
| Switch | `@switch (value) { @case (x) {} }` | `[ngSwitch]="value"` + `*ngSwitchCase="x"` |
| Import needed | None | `CommonModule` or individual directives |
| Performance | Better (optimized internals) | Directive overhead |

## @let: Template Local Variables

The `@let` syntax (Angular 18.1+) lets you declare read-only local variables in templates.

```html
<!-- Simple expression -->
@let fullName = user().firstName + ' ' + user().lastName;
<h1>{{ fullName }}</h1>

<!-- Computed value reused multiple times -->
@let total = cart().items.reduce((sum, i) => sum + i.price, 0);
<p>Subtotal: {{ total | currency }}</p>
<p>Tax: {{ total * 0.08 | currency }}</p>
<p>Total: {{ total * 1.08 | currency }}</p>

<!-- With async pipe -->
@let data = dataStream$ | async;
@if (data) {
  <app-dashboard [data]="data" />
}

<!-- Signal unwrapping for cleaner templates -->
@let u = user();
@if (u) {
  <p>{{ u.name }} - {{ u.email }}</p>
}
```

### @let Rules

- One variable per `@let` declaration.
- The variable is read-only (cannot be reassigned).
- The variable is scoped to the current view and its descendants (not hoisted).
- The value is kept up to date reactively.

```html
<!-- BAD: Cannot declare multiple variables -->
@let a = 1, b = 2;

<!-- GOOD: Separate declarations -->
@let a = 1;
@let b = 2;
```

## Template Reference Variables

A `#` prefix creates a reference to an element or component in the template. The reference is available anywhere in the same template.

```html
<!-- Reference to a native element -->
<input #searchInput type="text" />
<button (click)="onSearch(searchInput.value)">Search</button>

<!-- Reference to a component instance -->
<app-color-picker #picker></app-color-picker>
<button (click)="picker.reset()">Reset Color</button>

<!-- Reference to a directive -->
<form #myForm="ngForm" (submit)="onSubmit(myForm)">
  <input name="email" ngModel required />
  <button [disabled]="myForm.invalid">Submit</button>
</form>
```

Template reference variables are distinct from `@let` variables:

| Feature | `#ref` | `@let` |
|---------|--------|--------|
| What it references | DOM element, component, or directive instance | Any expression result |
| Scope | Entire template | Current view and descendants |
| Mutable | The reference itself is constant | Read-only |
| Use case | Access element APIs, call component methods | Store computed values, unwrap async/signals |

## Pipes

Pipes transform displayed values in templates. They use the `|` operator.

```html
<!-- Built-in pipes -->
<p>{{ birthday() | date:'longDate' }}</p>
<p>{{ price() | currency:'USD':'symbol':'1.2-2' }}</p>
<p>{{ name() | uppercase }}</p>
<p>{{ description() | slice:0:100 }}...</p>
<p>{{ fileSize() | number:'1.0-0' }} bytes</p>

<!-- Pipe chaining -->
<p>{{ name() | lowercase | titlecase }}</p>

<!-- Pipe with async data -->
<p>{{ serverTime$ | async | date:'HH:mm:ss' }}</p>

<!-- Pipe in property bindings -->
<img [alt]="title() | uppercase" [src]="imageUrl()" />
```

### Pipe Precedence

The pipe operator has **lower precedence** than most operators except the ternary `? :`. This can cause unexpected behavior:

```html
<!-- BAD: Pipe captures the entire ternary expression -->
<!-- Parsed as: (isSpecial() ? 'Special Price' : price()) | currency -->
{{ isSpecial() ? 'Special Price' : price() | currency }}

<!-- GOOD: Use parentheses to be explicit -->
{{ isSpecial() ? 'Special Price' : (price() | currency) }}

<!-- Also need parens here -->
{{ (value() | uppercase) + ' suffix' }}
```

### Signals vs Async Pipe

With signals, you often do not need the `async` pipe:

```html
<!-- Observable approach (requires async pipe) -->
@if (user$ | async; as user) {
  <p>{{ user.name }}</p>
}

<!-- Signal approach (no pipe needed) -->
@let user = userSignal();
@if (user) {
  <p>{{ user.name }}</p>
}
```

## Safe Navigation and Nullish Coalescing

### Optional Chaining (?.)

Prevents errors when accessing properties on potentially null/undefined values.

```html
<p>{{ user()?.address?.city }}</p>
<p>{{ order()?.items?.[0]?.name }}</p>
```

If any part of the chain is nullish, the expression evaluates to `undefined` and nothing is rendered.

### Nullish Coalescing (??)

Provides a fallback value when the expression is `null` or `undefined`.

```html
<p>{{ user()?.name ?? 'Anonymous' }}</p>
<p>{{ config()?.pageSize ?? 10 }}</p>
<p>{{ description() ?? 'No description available' }}</p>
```

Unlike `||`, the `??` operator does not treat `0`, `''`, or `false` as falsy:

```html
<!-- || treats 0 as falsy -->
{{ count() || 'N/A' }}  <!-- Shows "N/A" when count is 0 -->

<!-- ?? only triggers for null/undefined -->
{{ count() ?? 'N/A' }}  <!-- Shows "0" when count is 0 -->
```

## Host Bindings

Components can bind to their own host element using the `host` property in the `@Component` decorator. This replaces the legacy `@HostBinding` and `@HostListener` decorators.

```typescript
@Component({
  selector: 'app-tooltip',
  host: {
    // Attribute
    'role': 'tooltip',

    // Property binding
    '[id]': 'tooltipId()',

    // Class binding
    '[class.visible]': 'isVisible()',
    '[class.position-top]': 'position() === "top"',

    // Style binding
    '[style.left.px]': 'x()',
    '[style.top.px]': 'y()',

    // Event binding
    '(mouseenter)': 'onMouseEnter()',
    '(mouseleave)': 'onMouseLeave()',

    // Keyboard event with filter
    '(document:keydown.escape)': 'hide()',
  },
  template: `<ng-content />`,
})
export class TooltipComponent {
  tooltipId = input.required<string>();
  position = input<'top' | 'bottom'>('top');
  isVisible = signal(false);
  x = signal(0);
  y = signal(0);

  onMouseEnter() { this.isVisible.set(true); }
  onMouseLeave() { this.isVisible.set(false); }
  hide() { this.isVisible.set(false); }
}
```

**Prefer the `host` property over `@HostBinding` and `@HostListener` decorators.** The `host` property keeps all host bindings in one place and is the recommended approach.

```typescript
// BAD: Decorator-based (legacy)
@Component({ ... })
export class MyComponent {
  @HostBinding('class.active') isActive = false;
  @HostListener('click') onClick() { ... }
}

// GOOD: host property (modern)
@Component({
  host: {
    '[class.active]': 'isActive()',
    '(click)': 'onClick()',
  },
})
export class MyComponent {
  isActive = signal(false);
  onClick() { ... }
}
```

## ng-content and Content Projection

`<ng-content>` defines a slot where parent-provided content is rendered inside a child component.

```typescript
// card.component.ts
@Component({
  selector: 'app-card',
  template: `
    <div class="card">
      <div class="card-header">
        <ng-content select="[card-header]" />
      </div>
      <div class="card-body">
        <ng-content />
      </div>
      <div class="card-footer">
        <ng-content select="[card-footer]" />
      </div>
    </div>
  `,
})
export class CardComponent {}

// Usage
<app-card>
  <h2 card-header>Title</h2>
  <p>This goes into the default slot (card-body).</p>
  <button card-footer>Save</button>
</app-card>
```

The `select` attribute uses CSS selectors to target specific projected content. Content without a matching selector goes to the default (unnamed) `<ng-content>`.

## ng-template and ng-container

### ng-template

Defines a block of HTML that is not rendered by default. It is used with structural directives, `@defer`, and programmatic template rendering.

```html
<!-- Used with legacy structural directives -->
<ng-template #loading>
  <app-spinner />
</ng-template>

<!-- Programmatic rendering via ViewContainerRef -->
<ng-template #itemTmpl let-item let-idx="index">
  <div>{{ idx }}: {{ item.name }}</div>
</ng-template>
```

### ng-container

A grouping element that does not render a DOM element. Useful for applying logic without adding extra wrapper elements.

```html
<!-- Grouping without extra DOM nodes -->
@if (isLoggedIn()) {
  <ng-container>
    <app-header />
    <app-sidebar />
  </ng-container>
}

<!-- Applying multiple things to a single logical block -->
<ng-container *ngTemplateOutlet="headerTmpl; context: { $implicit: title }"></ng-container>
```

With the new control flow syntax, `ng-container` is less necessary because `@if`, `@for`, and `@switch` do not produce wrapper elements. But it remains useful for template outlets and content projection selectors.

## Common Mistakes

### 1. Using Interpolation for Non-String Bindings

```html
<!-- BAD: Boolean becomes the string "true" -->
<app-toggle enabled="{{ isEnabled() }}"></app-toggle>

<!-- GOOD: Property binding passes the actual boolean -->
<app-toggle [enabled]="isEnabled()"></app-toggle>
```

### 2. Calling Expensive Functions in Templates

```html
<!-- BAD: getFilteredItems() runs on every change detection cycle -->
@for (item of getFilteredItems(); track item.id) {
  <p>{{ item.name }}</p>
}

<!-- GOOD: Use a computed signal -->
<!-- In component: filteredItems = computed(() => ...); -->
@for (item of filteredItems(); track item.id) {
  <p>{{ item.name }}</p>
}
```

### 3. Forgetting the track Expression in @for

```html
<!-- ERROR: track is required -->
@for (item of items()) {
  <p>{{ item.name }}</p>
}

<!-- GOOD -->
@for (item of items(); track item.id) {
  <p>{{ item.name }}</p>
}
```

### 4. Using $index as track for Dynamic Lists

```html
<!-- BAD: Tracking by index defeats DOM reuse when items are added/removed/reordered -->
@for (item of items(); track $index) {
  <app-item [data]="item" />
}

<!-- GOOD: Track by a stable unique identifier -->
@for (item of items(); track item.id) {
  <app-item [data]="item" />
}
```

### 5. Pipe Precedence Confusion

```html
<!-- BAD: Pipe binds tighter than + -->
{{ firstName() | uppercase + ' ' + lastName() | uppercase }}
<!-- Parsed as: (firstName() | (uppercase + ' ' + lastName())) | uppercase -->

<!-- GOOD: Parentheses make intent clear -->
{{ (firstName() | uppercase) + ' ' + (lastName() | uppercase) }}
```

### 6. Mixing || with ?? Without Parentheses

```html
<!-- ERROR: Cannot mix || and ?? without parentheses -->
{{ a() || b() ?? c() }}

<!-- GOOD -->
{{ (a() || b()) ?? c() }}
```

## Best Practices

### 1. Use Property Binding for All Non-Text Values

Reserve interpolation for rendering text. Use `[property]` for booleans, objects, arrays, and numbers.

### 2. Use Signals in Templates Instead of Async Pipe

Signals provide synchronous access and do not need the `async` pipe. Convert observables to signals at the component boundary with `toSignal()`.

### 3. Use the New Control Flow Syntax

Prefer `@if`, `@for`, `@switch` over `*ngIf`, `*ngFor`, `[ngSwitch]`. The new syntax is faster, needs no imports, supports `@else if`, `@empty`, and does not add wrapper elements.

### 4. Keep Template Expressions Simple

If a template expression needs more than one line of logic, move it to a `computed()` signal or a method in the component.

### 5. Use @let to Avoid Repeated Signal Calls

```html
<!-- Without @let: signal called multiple times -->
<p>{{ user().firstName }} {{ user().lastName }} ({{ user().email }})</p>

<!-- With @let: signal called once, result reused -->
@let u = user();
<p>{{ u.firstName }} {{ u.lastName }} ({{ u.email }})</p>
```

### 6. Use the host Property for Host Bindings

Keep `[class.*]`, `[style.*]`, `[attr.*]`, and event listeners on the host element in the `host` metadata property, not in decorator-based `@HostBinding` / `@HostListener`.

## Real-World Use Cases

### Data Table with Sorting and Filtering

```typescript
@Component({
  selector: 'app-data-table',
  template: `
    <div class="toolbar">
      <input
        [value]="filter()"
        (input)="filter.set(getValue($event))"
        placeholder="Filter..."
      />
    </div>

    <table>
      <thead>
        <tr>
          @for (col of columns(); track col.key) {
            <th (click)="toggleSort(col.key)" [class.sorted]="sortKey() === col.key">
              {{ col.label }}
              @if (sortKey() === col.key) {
                <span>{{ sortDir() === 'asc' ? '\u25B2' : '\u25BC' }}</span>
              }
            </th>
          }
        </tr>
      </thead>
      <tbody>
        @for (row of sortedRows(); track row.id; let isOdd = $odd) {
          <tr [class.odd]="isOdd">
            @for (col of columns(); track col.key) {
              <td>{{ row[col.key] }}</td>
            }
          </tr>
        } @empty {
          <tr>
            <td [attr.colspan]="columns().length">No data matches your filter.</td>
          </tr>
        }
      </tbody>
    </table>
  `,
})
export class DataTableComponent {
  columns = input.required<Column[]>();
  rows = input.required<Record<string, unknown>[]>();

  filter = signal('');
  sortKey = signal('');
  sortDir = signal<'asc' | 'desc'>('asc');

  filteredRows = computed(() => {
    const term = this.filter().toLowerCase();
    if (!term) return this.rows();
    return this.rows().filter(row =>
      Object.values(row).some(val => String(val).toLowerCase().includes(term))
    );
  });

  sortedRows = computed(() => {
    const key = this.sortKey();
    if (!key) return this.filteredRows();
    const dir = this.sortDir() === 'asc' ? 1 : -1;
    return [...this.filteredRows()].sort((a, b) => {
      const aVal = String(a[key] ?? '');
      const bVal = String(b[key] ?? '');
      return aVal.localeCompare(bVal) * dir;
    });
  });

  toggleSort(key: string) {
    if (this.sortKey() === key) {
      this.sortDir.update(d => d === 'asc' ? 'desc' : 'asc');
    } else {
      this.sortKey.set(key);
      this.sortDir.set('asc');
    }
  }

  getValue(event: Event): string {
    return (event.target as HTMLInputElement).value;
  }
}
```

### Form with Two-Way Binding and Validation Feedback

```typescript
@Component({
  selector: 'app-contact-form',
  imports: [FormsModule],
  template: `
    <form #form="ngForm" (submit)="onSubmit(form)">
      <div class="field">
        <label for="name">Name</label>
        <input
          id="name"
          name="name"
          [(ngModel)]="name"
          required
          minlength="2"
          #nameCtrl="ngModel"
        />
        @if (nameCtrl.invalid && nameCtrl.touched) {
          <span class="error">
            @if (nameCtrl.errors?.['required']) {
              Name is required.
            } @else if (nameCtrl.errors?.['minlength']) {
              @let req = nameCtrl.errors?.['minlength']?.requiredLength;
              Name must be at least {{ req }} characters.
            }
          </span>
        }
      </div>

      <div class="field">
        <label for="email">Email</label>
        <input
          id="email"
          name="email"
          type="email"
          [(ngModel)]="email"
          required
          email
          #emailCtrl="ngModel"
        />
        @if (emailCtrl.invalid && emailCtrl.touched) {
          <span class="error">Please enter a valid email.</span>
        }
      </div>

      <button type="submit" [disabled]="form.invalid">
        Submit
      </button>
    </form>
  `,
})
export class ContactFormComponent {
  name = signal('');
  email = signal('');

  onSubmit(form: NgForm) {
    if (form.valid) {
      console.log('Submitting:', { name: this.name(), email: this.email() });
    }
  }
}
```

> **Key Takeaway:** Angular's template syntax provides interpolation for text, property/attribute/class/style bindings for dynamic values, event binding for user interaction, two-way binding with `model()`, built-in control flow (`@if`, `@for`, `@switch`) for structure, `@let` for local variables, and pipes for value transformation. Use signals in templates, prefer the new control flow over legacy directives, and keep expressions simple.
