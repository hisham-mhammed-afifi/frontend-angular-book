# 11.2 Project Structure and File Conventions

> A well-organized project structure keeps code discoverable, maintainable, and scalable. Angular provides conventions and tooling that enforce consistency across teams and projects of any size.

---

## The Default Angular 21 Project

Running `ng new my-app` generates this structure:

```
my-app/
  .angular/                   # CLI build cache (gitignored)
  .vscode/                    # VS Code workspace settings
    extensions.json
    launch.json
    tasks.json
  node_modules/               # Installed packages (gitignored)
  public/                     # Static assets served as-is
    favicon.ico
  src/
    app/
      app.component.ts
      app.component.html
      app.component.css
      app.component.spec.ts
      app.config.ts
      app.routes.ts
    index.html
    main.ts
    styles.css
  angular.json
  package.json
  package-lock.json
  tsconfig.json
  tsconfig.app.json
  tsconfig.spec.json
```

There is no `app.module.ts`. Angular 21 projects are standalone by default and bootstrap through `bootstrapApplication()` in `main.ts`.

## Root-Level Configuration Files

Each file in the workspace root serves a specific purpose.

### angular.json

The workspace configuration file. Controls:
- Build targets (what builder to use, optimization flags, asset paths)
- Serve options (port, proxy, HMR)
- Test configuration
- Lint configuration
- Default schematics options
- Project definitions (for multi-project workspaces)

```json
{
  "$schema": "./node_modules/@angular/cli/lib/config/schema.json",
  "version": 1,
  "newProjectRoot": "projects",
  "projects": {
    "my-app": {
      "projectType": "application",
      "root": "",
      "sourceRoot": "src",
      "prefix": "app",
      "architect": {
        "build": { /* ... */ },
        "serve": { /* ... */ },
        "test": { /* ... */ }
      }
    }
  }
}
```

### tsconfig.json

The base TypeScript configuration. All other `tsconfig.*.json` files extend this one. Sets compiler strictness, module resolution, and Angular-specific options.

```json
{
  "compileOnSave": false,
  "compilerOptions": {
    "outDir": "./dist/out-tsc",
    "strict": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "sourceMap": true,
    "declaration": false,
    "downlevelIteration": true,
    "experimentalDecorators": true,
    "moduleResolution": "bundler",
    "importHelpers": true,
    "target": "ES2022",
    "module": "ES2022",
    "lib": ["ES2022", "dom"],
    "skipLibCheck": true
  },
  "angularCompilerOptions": {
    "strictInjectionParameters": true,
    "strictInputAccessModifiers": true,
    "strictTemplates": true
  }
}
```

### tsconfig.app.json

Extends the base config for the application build. Specifies which files to include and exclude.

```json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "outDir": "./out-tsc/app",
    "types": []
  },
  "files": ["src/main.ts"],
  "include": ["src/**/*.d.ts"]
}
```

### tsconfig.spec.json

Extends the base config for test files. Includes test-specific types.

```json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "outDir": "./out-tsc/spec",
    "types": ["vitest/globals"]
  },
  "include": [
    "src/**/*.spec.ts",
    "src/**/*.d.ts"
  ]
}
```

### package.json

Standard npm configuration. The `scripts` section wraps CLI commands:

```json
{
  "scripts": {
    "ng": "ng",
    "start": "ng serve",
    "build": "ng build",
    "test": "ng test",
    "watch": "ng build --watch --configuration development"
  }
}
```

## The src/ Directory

All application source code lives under `src/`.

### main.ts

The application entry point. Bootstraps the root component with providers.

```typescript
import { bootstrapApplication } from '@angular/platform-browser';
import { appConfig } from './app/app.config';
import { AppComponent } from './app/app.component';

bootstrapApplication(AppComponent, appConfig)
  .catch((err) => console.error(err));
```

### index.html

The single HTML page. The root component's selector (`<app-root>`) is placed here.

```html
<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>MyApp</title>
  <base href="/">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <link rel="icon" type="image/x-icon" href="favicon.ico">
</head>
<body>
  <app-root></app-root>
</body>
</html>
```

### styles.css (or styles.scss)

Global styles that apply across the entire application. Component-specific styles go in component style files.

### public/

Static assets served at the root URL. Files in `public/` are copied as-is to the build output. Place images, fonts, JSON files, and other static resources here.

```
public/
  favicon.ico
  images/
    logo.png
  data/
    countries.json
```

Reference them with root-relative paths: `/images/logo.png`, `/data/countries.json`.

## The app/ Directory

The `app/` folder is where all application code lives. How you organize it determines how maintainable the project is as it grows.

### app.config.ts

Application-level configuration. Registers providers that the entire app needs.

```typescript
import { ApplicationConfig, provideZonelessChangeDetection } from '@angular/core';
import { provideRouter } from '@angular/router';
import { provideHttpClient, withInterceptors } from '@angular/common/http';
import { routes } from './app.routes';
import { authInterceptor } from './core/auth.interceptor';

export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
    provideRouter(routes),
    provideHttpClient(withInterceptors([authInterceptor])),
  ]
};
```

### app.routes.ts

Top-level route definitions. Feature routes are lazy-loaded.

```typescript
import { Routes } from '@angular/router';

export const routes: Routes = [
  { path: '', redirectTo: 'dashboard', pathMatch: 'full' },
  {
    path: 'dashboard',
    loadComponent: () =>
      import('./features/dashboard/dashboard.component')
        .then(m => m.DashboardComponent),
  },
  {
    path: 'users',
    loadChildren: () =>
      import('./features/users/users.routes')
        .then(m => m.USERS_ROUTES),
  },
];
```

### app.component.ts

The root component. Typically contains the layout shell (header, sidebar, router outlet, footer).

```typescript
import { Component } from '@angular/core';
import { RouterOutlet } from '@angular/router';
import { HeaderComponent } from './layout/header.component';
import { SidebarComponent } from './layout/sidebar.component';

@Component({
  selector: 'app-root',
  imports: [RouterOutlet, HeaderComponent, SidebarComponent],
  template: `
    <app-header />
    <div class="layout">
      <app-sidebar />
      <main>
        <router-outlet />
      </main>
    </div>
  `,
  styleUrl: './app.component.scss'
})
export class AppComponent {}
```

## Scalable Folder Structure

For anything beyond a trivial application, organize `app/` into three top-level directories: `core/`, `shared/`, and `features/`. Optionally add `layout/` for shell components.

```
src/app/
  core/                       # Singleton services, interceptors, guards
    auth.interceptor.ts
    auth.guard.ts
    auth.service.ts
    error-handler.service.ts
    api.service.ts
  shared/                     # Reusable components, pipes, directives
    components/
      button/
        button.component.ts
        button.component.html
        button.component.scss
      modal/
        modal.component.ts
        modal.component.html
        modal.component.scss
    pipes/
      truncate.pipe.ts
      date-format.pipe.ts
    directives/
      tooltip.directive.ts
      click-outside.directive.ts
    models/
      user.model.ts
      pagination.model.ts
    utils/
      date-utils.ts
      string-utils.ts
  layout/                     # Application shell components
    header.component.ts
    sidebar.component.ts
    footer.component.ts
  features/                   # Feature areas (lazy-loaded)
    dashboard/
      dashboard.component.ts
      dashboard.component.html
      widgets/
        stats-card.component.ts
        activity-feed.component.ts
    users/
      users.routes.ts
      user-list/
        user-list.component.ts
        user-list.component.html
      user-detail/
        user-detail.component.ts
        user-detail.component.html
      user-form/
        user-form.component.ts
        user-form.component.html
      user.service.ts
    products/
      products.routes.ts
      product-list.component.ts
      product-detail.component.ts
      product.service.ts
  app.component.ts
  app.config.ts
  app.routes.ts
```

### core/

Contains singleton services, interceptors, guards, and anything that should exist exactly once in the application. Services in `core/` use `providedIn: 'root'` and are never imported by `shared/` or feature modules.

Rules:
- No UI components.
- Only application-wide services.
- Imported only in `app.config.ts` or `app.routes.ts`.
- Services here are singletons by design.

```typescript
// core/auth.service.ts
import { Injectable, signal, computed } from '@angular/core';

@Injectable({ providedIn: 'root' })
export class AuthService {
  private currentUser = signal<User | null>(null);

  readonly isAuthenticated = computed(() => this.currentUser() !== null);
  readonly user = this.currentUser.asReadonly();

  login(credentials: Credentials) { /* ... */ }
  logout() { this.currentUser.set(null); }
}
```

### shared/

Contains reusable components, pipes, directives, models, and utility functions used across multiple features. Nothing in `shared/` should depend on a specific feature.

Rules:
- No services with application state.
- No feature-specific logic.
- Every export should be usable by any feature.
- Components here should be presentational (dumb), not smart.

```typescript
// shared/components/button/button.component.ts
import { Component, input, output } from '@angular/core';

@Component({
  selector: 'app-button',
  template: `
    <button
      [type]="type()"
      [disabled]="disabled()"
      (click)="clicked.emit()">
      <ng-content />
    </button>
  `,
  styleUrl: './button.component.scss'
})
export class ButtonComponent {
  type = input<'button' | 'submit' | 'reset'>('button');
  disabled = input(false);
  clicked = output<void>();
}
```

### features/

Each subdirectory represents a distinct feature area. Features are self-contained: they have their own components, services, routes, and models. Features are typically lazy-loaded through the router.

Rules:
- Each feature has its own route file if it has multiple pages.
- Feature services are scoped to the feature (not `providedIn: 'root'`).
- Features can import from `core/` and `shared/` but never from other features.
- If two features need the same service, move it to `core/`.

```typescript
// features/users/users.routes.ts
import { Routes } from '@angular/router';

export const USERS_ROUTES: Routes = [
  {
    path: '',
    loadComponent: () =>
      import('./user-list/user-list.component')
        .then(m => m.UserListComponent),
  },
  {
    path: ':id',
    loadComponent: () =>
      import('./user-detail/user-detail.component')
        .then(m => m.UserDetailComponent),
  },
  {
    path: ':id/edit',
    loadComponent: () =>
      import('./user-form/user-form.component')
        .then(m => m.UserFormComponent),
  },
];
```

### layout/

Shell components that form the application frame. Header, sidebar, footer, and navigation. These are imported directly by `AppComponent` and are not lazy-loaded.

## File Naming Conventions

### Traditional Convention (Still Common)

Angular has historically used dot-separated type suffixes:

```
user-list.component.ts
user-list.component.html
user-list.component.scss
user-list.component.spec.ts
auth.service.ts
auth.guard.ts
truncate.pipe.ts
highlight.directive.ts
auth.interceptor.ts
user.model.ts
users.routes.ts
```

Pattern: `{feature-name}.{type}.ts`

### Updated Convention (Angular 20+)

The Angular style guide RFC (2024-2025) decided to drop the type suffix requirement. The new position: Angular's own documentation will not use `.component`, `.service`, etc. in file names. The reasoning is that modern IDEs make type suffixes unnecessary for navigation, and descriptive names are more valuable than formulaic ones.

```
# New style (no type suffix)
user-list.ts          # instead of user-list.component.ts
user-list.html
user-list.scss
user-list.spec.ts
auth-store.ts         # instead of auth.service.ts
auth.guard.ts         # guards/interceptors still commonly suffixed
truncate.pipe.ts      # pipes still commonly suffixed
```

### Which Convention to Use

Both are valid. The key rule: **be consistent within a project**. If your team uses type suffixes, use them everywhere. If you adopt the new suffix-free convention, apply it everywhere.

| Convention | Pros | Cons |
|---|---|---|
| Type suffixed (`.component.ts`) | Instantly identifies file purpose. Familiar to most Angular developers. Easy to glob (`*.component.ts`). | Verbose. Redundant when IDE provides the same info. |
| Suffix-free (`.ts`) | Shorter names. Encourages descriptive naming over generic suffixes. Matches Angular's future direction. | Requires more descriptive names. Harder to glob by type. Unfamiliar to teams on older codebases. |

For this guide, we use the traditional suffixed convention because it is still the most widely used and what the CLI generates by default as of Angular 21.

### General Naming Rules

1. **Use kebab-case for file names**: `user-profile.component.ts`, not `UserProfile.component.ts` or `userProfile.component.ts`.

2. **Use PascalCase for class names**: `UserProfileComponent`, `AuthService`, `TruncatePipe`.

3. **Use camelCase for selectors**: `appHighlight` (directives), `truncate` (pipes).

4. **Use kebab-case with prefix for component selectors**: `app-user-profile`, `acme-dashboard-widget`.

5. **Match file names to class names**: The class `UserListComponent` lives in `user-list.component.ts`.

6. **One artifact per file**: Do not put multiple components, services, or pipes in the same file. Exception: small, tightly-coupled items like a component and its model interface.

## Barrel Files (index.ts)

A barrel file re-exports multiple modules from a directory, creating a public API:

```typescript
// shared/pipes/index.ts
export { TruncatePipe } from './truncate.pipe';
export { DateFormatPipe } from './date-format.pipe';
export { CurrencyFormatPipe } from './currency-format.pipe';
```

Consumers import from the barrel:

```typescript
// Instead of three separate imports:
import { TruncatePipe } from '../shared/pipes/truncate.pipe';
import { DateFormatPipe } from '../shared/pipes/date-format.pipe';

// Import from the barrel:
import { TruncatePipe, DateFormatPipe } from '../shared/pipes';
```

### Barrel File Risks

Barrel files have a significant gotcha: **they can break lazy loading and tree shaking**.

When you import one item from a barrel, the bundler must evaluate the entire barrel file and all its re-exports. If the barrel re-exports a heavy module alongside the lightweight pipe you need, that heavy module gets pulled into the bundle.

```typescript
// shared/index.ts (dangerous mega-barrel)
export * from './components/heavy-chart/heavy-chart.component';
export * from './components/button/button.component';
export * from './pipes/truncate.pipe';
export * from './directives/tooltip.directive';
```

```typescript
// You only need TruncatePipe, but the bundler evaluates everything
import { TruncatePipe } from '../shared';
// heavy-chart.component (and its chart library) may get pulled in
```

### Barrel File Rules

1. **Use small, focused barrels per subdirectory**, not one mega-barrel for all of `shared/`.
2. **Never use barrel files across lazy-loaded feature boundaries.**
3. **Never use `export *` in barrels.** Explicitly list each export.
4. **Consider skipping barrels entirely** for small directories where direct imports are clear enough.

```typescript
// Good: focused barrel for a specific subdirectory
// shared/pipes/index.ts
export { TruncatePipe } from './truncate.pipe';
export { DateFormatPipe } from './date-format.pipe';

// Bad: mega-barrel for all of shared
// shared/index.ts
export * from './components';
export * from './pipes';
export * from './directives';
```

## Path Aliases

TypeScript path aliases prevent deep relative imports like `../../../core/auth.service`. Configure them in `tsconfig.json`:

```json
{
  "compilerOptions": {
    "paths": {
      "@core/*": ["src/app/core/*"],
      "@shared/*": ["src/app/shared/*"],
      "@features/*": ["src/app/features/*"],
      "@env/*": ["src/environments/*"]
    }
  }
}
```

Now imports are clean regardless of file depth:

```typescript
// Before: relative paths (fragile, hard to read)
import { AuthService } from '../../../core/auth.service';
import { ButtonComponent } from '../../shared/components/button/button.component';

// After: path aliases (stable, self-documenting)
import { AuthService } from '@core/auth.service';
import { ButtonComponent } from '@shared/components/button/button.component';
```

Path aliases work with the Angular compiler, VS Code IntelliSense, and ESLint. No additional bundler configuration is needed when using the Angular CLI.

## Multi-Project Workspaces

For large organizations, a single Angular workspace can contain multiple applications and libraries:

```
workspace/
  projects/
    admin-app/
      src/
      tsconfig.app.json
    customer-app/
      src/
      tsconfig.app.json
    shared-lib/
      src/
      ng-package.json
  angular.json             # Defines all projects
  tsconfig.json            # Base config
  package.json
```

Create additional projects:

```bash
# New application in the workspace
ng generate application admin-app

# New library
ng generate library shared-lib
```

Build or serve a specific project:

```bash
ng serve admin-app
ng build shared-lib
ng test customer-app
```

Libraries are built as Angular packages and consumed by applications within the workspace using TypeScript path mappings configured automatically by the CLI.

### When to Use Multi-Project Workspaces

| Scenario | Recommendation |
|---|---|
| Single application | Single-project workspace (default `ng new`) |
| 2-3 related apps sharing code | Multi-project workspace |
| 5+ apps or independent teams | Nx workspace with enforced module boundaries |
| Micro frontends | Separate repositories or Nx with Module Federation |

## Feature-Based vs. Type-Based Organization

There are two approaches to organizing files. Feature-based is the Angular-recommended approach.

### Type-Based (Not Recommended)

Groups files by their technical type:

```
src/app/
  components/
    user-list.component.ts
    user-detail.component.ts
    product-list.component.ts
    dashboard.component.ts
  services/
    user.service.ts
    product.service.ts
    auth.service.ts
  pipes/
    truncate.pipe.ts
  guards/
    auth.guard.ts
```

Problems:
- To work on the "users" feature, you open files scattered across `components/`, `services/`, `guards/`, and `pipes/`.
- As the project grows, each directory becomes a flat list of dozens of files.
- Deleting a feature requires searching multiple directories.
- Lazy loading by feature is awkward.

### Feature-Based (Recommended)

Groups files by the feature they belong to:

```
src/app/
  features/
    users/
      user-list.component.ts
      user-detail.component.ts
      user.service.ts
      users.routes.ts
    products/
      product-list.component.ts
      product-detail.component.ts
      product.service.ts
      products.routes.ts
  core/
    auth.service.ts
    auth.guard.ts
  shared/
    pipes/
      truncate.pipe.ts
```

Benefits:
- Everything related to "users" is in one place.
- Deleting a feature means deleting one directory.
- Lazy loading maps directly to feature directories.
- New team members find related code immediately.

## Scaling Patterns by Project Size

### Small Project (1-5 features, 1-2 developers)

Keep it simple. A flat `app/` structure with minimal nesting is fine.

```
src/app/
  dashboard.component.ts
  user-list.component.ts
  user-detail.component.ts
  auth.service.ts
  api.service.ts
  app.component.ts
  app.config.ts
  app.routes.ts
```

Do not over-engineer with `core/`, `shared/`, and `features/` when the entire app has five components.

### Medium Project (5-15 features, 2-5 developers)

Introduce the `core/shared/features` pattern:

```
src/app/
  core/
  shared/
  features/
  layout/
  app.component.ts
  app.config.ts
  app.routes.ts
```

### Large Project (15+ features, 5+ developers)

Add domain grouping within features, enforce boundaries with ESLint, and consider Nx for monorepo tooling:

```
src/app/
  core/
  shared/
  domains/
    sales/
      orders/
      invoices/
      customers/
    inventory/
      products/
      warehouses/
    hr/
      employees/
      departments/
  layout/
  app.component.ts
  app.config.ts
  app.routes.ts
```

Each domain has its own route tree, lazy-loaded at the domain level.

## Common Mistakes

- **Organizing by type instead of by feature.** Putting all components in one folder and all services in another makes it harder to work on a specific feature, harder to delete features, and harder to lazy-load. Organize by feature.

- **Creating a mega-barrel `index.ts` for all of `shared/`.** This defeats tree shaking and lazy loading. Use small, focused barrel files per subdirectory, or skip barrels entirely.

- **Cross-feature imports.** Feature A should never import from Feature B. If two features need the same code, move it to `shared/` (for UI) or `core/` (for services). Cross-feature imports create circular dependencies and tight coupling.

- **Putting feature-specific services in `core/`.** A service used only by the "users" feature belongs in `features/users/`, not in `core/`. Reserve `core/` for truly application-wide singletons.

- **Not using path aliases.** Deep relative imports like `../../../../core/auth.service` are fragile and hard to read. Set up `@core/`, `@shared/`, and `@features/` aliases in `tsconfig.json`.

- **Mixing infrastructure and feature logic in components.** Components should handle presentation. HTTP calls, state management, and business logic belong in services. A component that fetches data, transforms it, and renders it is doing too much.

- **Ignoring the generated .vscode/ directory.** The CLI generates VS Code workspace settings (`launch.json`, `extensions.json`, `tasks.json`) that benefit the whole team. Commit them to version control.

---

## Best Practices

- **Follow the core/shared/features pattern.** It scales from small to large projects and provides clear boundaries for imports and lazy loading.

- **Keep features self-contained.** Each feature directory should contain everything it needs: components, services, routes, and models. A feature should be deletable by removing its directory and its route entry.

- **Lazy-load features.** Use `loadComponent` or `loadChildren` in routes to split features into separate bundles.

- **Use `providedIn: 'root'` for application-wide services.** For feature-scoped services, provide them in the feature's route configuration or component.

- **Set up path aliases early.** Configure `@core/`, `@shared/`, `@features/` in `tsconfig.json` on project creation. Retrofitting them later requires updating many imports.

- **Enforce boundaries.** Use ESLint rules (like `@nx/enforce-module-boundaries` or `eslint-plugin-boundaries`) to prevent cross-feature imports programmatically. Code reviews alone cannot catch every violation.

- **Commit .vscode/ and .editorconfig.** Shared editor settings reduce formatting conflicts across the team.

- **Use the CLI to generate files.** `ng generate` follows naming conventions automatically. Manual file creation risks inconsistent names and missed registrations.

### 2. Route Configuration Matching Feature Structure

```typescript
// app.routes.ts
import { Routes } from '@angular/router';
import { authGuard } from '@core/auth.guard';

export const routes: Routes = [
  { path: '', redirectTo: 'catalog', pathMatch: 'full' },
  {
    path: 'catalog',
    loadChildren: () =>
      import('./features/catalog/catalog.routes')
        .then(m => m.CATALOG_ROUTES),
  },
  {
    path: 'checkout',
    canActivate: [authGuard],
    loadChildren: () =>
      import('./features/checkout/checkout.routes')
        .then(m => m.CHECKOUT_ROUTES),
  },
  {
    path: 'account',
    canActivate: [authGuard],
    loadChildren: () =>
      import('./features/account/account.routes')
        .then(m => m.ACCOUNT_ROUTES),
  },
  {
    path: 'admin',
    canActivate: [authGuard],
    loadChildren: () =>
      import('./features/admin/admin.routes')
        .then(m => m.ADMIN_ROUTES),
  },
];
```

```typescript
// features/catalog/catalog.routes.ts
import { Routes } from '@angular/router';

export const CATALOG_ROUTES: Routes = [
  {
    path: '',
    loadComponent: () =>
      import('./product-list.component')
        .then(m => m.ProductListComponent),
  },
  {
    path: ':id',
    loadComponent: () =>
      import('./product-detail.component')
        .then(m => m.ProductDetailComponent),
  },
];
```

Each feature owns its routes. The top-level `app.routes.ts` only knows about feature entry points, not individual pages within features.

### 3. Enforcing Boundaries with ESLint

Using `eslint-plugin-boundaries` to prevent cross-feature imports:

```javascript
// eslint.config.js
import boundaries from 'eslint-plugin-boundaries';

export default [
  {
    plugins: { boundaries },
    settings: {
      'boundaries/elements': [
        { type: 'core', pattern: 'src/app/core/*' },
        { type: 'shared', pattern: 'src/app/shared/*' },
        { type: 'feature', pattern: 'src/app/features/*' },
        { type: 'layout', pattern: 'src/app/layout/*' },
        { type: 'app', pattern: 'src/app/*' },
      ],
    },
    rules: {
      'boundaries/element-types': [
        'error',
        {
          default: 'disallow',
          rules: [
            // Features can import from core and shared
            { from: 'feature', allow: ['core', 'shared'] },
            // Shared can only import from shared
            { from: 'shared', allow: ['shared'] },
            // Core can only import from core
            { from: 'core', allow: ['core'] },
            // Layout can import from core and shared
            { from: 'layout', allow: ['core', 'shared'] },
            // App root can import from anything
            { from: 'app', allow: ['core', 'shared', 'feature', 'layout'] },
          ],
        },
      ],
    },
  },
];
```

This makes cross-feature imports a lint error, caught in CI before code reaches main.

---

> **Key Takeaway:** Organize by feature, not by file type. Use `core/` for singletons, `shared/` for reusable UI, and `features/` for self-contained, lazy-loaded feature areas. Set up path aliases and boundary enforcement early because restructuring a large project later is painful.
