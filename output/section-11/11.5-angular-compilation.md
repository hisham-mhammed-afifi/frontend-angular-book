# 11.5 Angular Compilation (AOT, Esbuild)

> Angular transforms your TypeScript and HTML templates into optimized JavaScript through a multi-phase compilation pipeline. Understanding how this pipeline works helps you write code that compiles faster, catches errors earlier, and produces smaller bundles.

---

## Two Compilation Modes

Angular has two compilation strategies: Ahead-of-Time (AOT) and Just-in-Time (JIT). AOT is the default for all builds in Angular 21.

### Ahead-of-Time (AOT) Compilation

AOT compiles templates and TypeScript during the build, before the browser downloads the application. The output is plain JavaScript with no template compiler needed at runtime.

```
Source Code -> Angular Compiler (ngc) -> Optimized JS -> Browser
              (build time)
```

### Just-in-Time (JIT) Compilation

JIT compiles templates in the browser at runtime. The application ships with the Angular compiler, and templates are compiled when components are first rendered.

```
Source Code + Angular Compiler -> Browser -> Compile on demand -> Render
                                  (runtime)
```

### Comparison

| Aspect | AOT | JIT |
|---|---|---|
| When it compiles | Build time | Runtime (in browser) |
| Bundle includes compiler | No | Yes (+~100KB) |
| Template errors caught | At build time | At runtime |
| First render speed | Faster (pre-compiled) | Slower (compile first) |
| Build time | Longer | Shorter |
| Security | Templates pre-compiled, harder to inject | Compiler in browser, wider attack surface |
| Default in Angular 21 | Yes (`ng build`, `ng serve`) | No (opt-in with `--configuration=development` in rare cases) |

AOT is the only compilation mode used in production. JIT is effectively deprecated for normal development. Angular 21's `ng serve` uses AOT with incremental compilation, making JIT unnecessary even during development.

## The Angular Compiler (ngtsc)

The Angular compiler, internally called `ngtsc`, is a TypeScript transformer. It plugs into the TypeScript compiler (`tsc`) and adds Angular-specific processing. When you run `ng build`, the process is:

1. TypeScript parses all `.ts` files into an Abstract Syntax Tree (AST).
2. `ngtsc` scans for Angular decorators (`@Component`, `@Directive`, `@Pipe`, `@Injectable`, `@NgModule`).
3. `ngtsc` processes each decorator through multiple phases.
4. TypeScript emits JavaScript files, with Angular-generated code replacing the decorators.

### Phase 1: Analysis

The compiler scans every file for decorated classes and builds a model of the application. For each `@Component`, it records:
- The selector
- The template (inline or from a separate file)
- The styles
- The `imports` array (other components, directives, pipes)
- Input and output bindings
- Change detection strategy
- View encapsulation

```typescript
// What you write:
@Component({
  selector: 'app-greeting',
  imports: [UpperCasePipe],
  template: `<h1>{{ name() | uppercase }}</h1>`
})
export class GreetingComponent {
  name = input.required<string>();
}
```

The analysis phase extracts: selector is `app-greeting`, template uses an `uppercase` pipe, the component has one required input called `name`.

### Phase 2: Resolution

The compiler connects decorated classes to each other. It resolves:
- Which directives and pipes are available in each component's template (from the `imports` array).
- Which services are injectable and where they are provided.
- Route configurations and lazy-loaded boundaries.

### Phase 3: Template Type Checking

The compiler generates a synthetic TypeScript file (called a Type Check Block, or TCB) for each component template. This file represents the template as TypeScript code, allowing the TypeScript type checker to verify all bindings.

```typescript
// Your template:
// <h1>{{ user().name }}</h1>
// <button (click)="save(user())">Save</button>

// Generated TCB (simplified concept):
function _tcb_GreetingComponent(ctx: GreetingComponent) {
  const _t1 = ctx.user().name;   // Type checks user() returns an object with .name
  ctx.save(ctx.user());           // Type checks save() accepts the user type
}
```

If `user()` returns `User | null`, the type checker flags `user().name` as potentially unsafe (cannot read `.name` of `null`). This is how `strictTemplates` catches errors at build time.

### Phase 4: Code Generation (Emit)

The compiler generates an "Angular definition" for each decorated class. Components get a `componentDef` that contains:
- A compiled template function (instructions for the Angular runtime to create and update DOM).
- Metadata about inputs, outputs, host bindings, and queries.

```typescript
// Conceptual output (simplified):
GreetingComponent.ɵcmp = defineComponent({
  type: GreetingComponent,
  selectors: [['app-greeting']],
  inputs: { name: 'name' },
  template: function GreetingComponent_Template(rf, ctx) {
    if (rf & 1) {  // Creation mode
      elementStart(0, 'h1');
      pipe(1, 'uppercase');
      text(2);
      elementEnd();
    }
    if (rf & 2) {  // Update mode
      advance(2);
      textInterpolate(pipeBind1(1, 3, ctx.name()));
    }
  },
  dependencies: [UpperCasePipe],
});
```

You never write or see this code. The compiler generates it, and esbuild bundles it into the final output.

## Template Type Checking

Template type checking is one of the most valuable features of AOT compilation. It catches bugs that would otherwise only surface at runtime.

### strictTemplates

Enabled by default in Angular 21 projects (`ng new --strict` has been the default since Angular 12). Configured in `tsconfig.json`:

```json
{
  "angularCompilerOptions": {
    "strictTemplates": true,
    "strictInjectionParameters": true,
    "strictInputAccessModifiers": true
  }
}
```

What `strictTemplates` checks:

| Check | Example Error |
|---|---|
| Binding type mismatch | `[disabled]="userName"` where `userName` is `string`, not `boolean` |
| Missing properties | `{{ user.naem }}` (typo) on an object that has no `naem` property |
| Null safety | `{{ user().name }}` when `user()` can return `null` |
| Wrong event payload | `(click)="handle($event)"` where `$event` type does not match handler |
| Incorrect pipe arguments | `{{ value \| date:'invalid-format' }}` |
| Wrong directive inputs | `<div [ngClass]="42">` where `ngClass` expects string/array/object |

### Example: Catching Errors at Build Time

```typescript
interface User {
  id: number;
  name: string;
  email: string;
}

@Component({
  template: `
    <!-- Error: Property 'fullName' does not exist on type 'User' -->
    <h1>{{ user().fullName }}</h1>

    <!-- Error: Type 'string' is not assignable to type 'boolean' -->
    <button [disabled]="user().name">Save</button>

    <!-- Error: Object is possibly 'null' -->
    <span>{{ nullableUser()?.address.city }}</span>
  `
})
export class ProfileComponent {
  user = input.required<User>();
  nullableUser = input<User | null>(null);
}
```

All three errors are caught during `ng build`, not when a user clicks the wrong button in production.

### Opting Out for Specific Expressions

Use `$any()` to bypass type checking for a specific expression:

```html
<!-- Bypass when you know better than the compiler -->
<span>{{ $any(legacyData).customField }}</span>
```

Use this sparingly. It should be a last resort for third-party libraries with incorrect types or genuinely dynamic data.

## The Build System: esbuild + Vite

Angular 21 uses esbuild for bundling and Vite for the development server. This replaced the legacy Webpack-based system starting in Angular 17.

### Architecture

```
Development (ng serve):
  TypeScript + Templates -> ngtsc (AOT) -> esbuild (bundle) -> Vite (dev server + HMR)

Production (ng build):
  TypeScript + Templates -> ngtsc (AOT) -> esbuild (bundle + minify + tree shake) -> dist/
```

### Why esbuild?

esbuild is written in Go and compiles to native machine code. It is 10-100x faster than Webpack for bundling operations.

| Operation | Webpack | esbuild |
|---|---|---|
| Initial build (medium app) | 15-30 seconds | 2-5 seconds |
| Rebuild (file change) | 2-5 seconds | <500ms |
| Production build | 30-60 seconds | 5-15 seconds |
| Code splitting | Plugin-based | Native |
| Tree shaking | Plugin-based | Native |
| Minification | Terser (JS-based) | Native (Go-based) |

Actual times vary by project size, but the ratio is consistent.

### The Application Builder

Angular 21 uses the `@angular/build:application` builder (previously `@angular-devkit/build-angular:application`). This builder handles:

- Client-side bundling with esbuild
- Server-side rendering builds (optional)
- Static site generation / prerendering (optional)
- CSS/Sass processing
- Asset copying
- Index HTML generation
- Service worker generation

Configured in `angular.json`:

```json
{
  "architect": {
    "build": {
      "builder": "@angular/build:application",
      "options": {
        "outputPath": "dist/my-app",
        "index": "src/index.html",
        "browser": "src/main.ts",
        "tsConfig": "tsconfig.app.json",
        "assets": [
          { "glob": "**/*", "input": "public" }
        ],
        "styles": ["src/styles.scss"],
        "scripts": []
      },
      "configurations": {
        "production": {
          "budgets": [
            {
              "type": "initial",
              "maximumWarning": "500kB",
              "maximumError": "1MB"
            }
          ],
          "outputHashing": "all"
        },
        "development": {
          "optimization": false,
          "extractLicenses": false,
          "sourceMap": true
        }
      },
      "defaultConfiguration": "production"
    }
  }
}
```

### Vite Dev Server

The Vite dev server provides:
- **Hot Module Replacement (HMR)**: Template and style changes apply instantly without a full reload.
- **Prebundling**: Third-party dependencies are prebundled by Vite for faster module resolution.
- **Native ES Modules**: During development, code is served as ES modules, skipping full bundling.

When you run `ng serve`, Angular compiles with AOT, esbuild bundles the result in memory, and Vite serves it with HMR support. You do not need to configure Vite directly.

## The Build Cache

Angular CLI caches build artifacts in `.angular/cache/`. The cache stores:

- TypeScript incremental compilation state
- esbuild bundler graph
- Sass/PostCSS compilation results
- Angular AOT compilation output

On rebuild, only changed files are recompiled. Unchanged modules are loaded from cache.

```bash
# Cache location
.angular/cache/

# Check cache status
ng cache info

# Clear cache (if builds behave unexpectedly)
ng cache clean

# Disable cache (not recommended)
ng cache disable
```

The cache can grow to several hundred MB for large projects. It is gitignored by default.

## Migration from Webpack to esbuild

If you are maintaining an older project that still uses the Webpack-based `browser` builder, migrate to the `application` builder:

```bash
ng update @angular/cli --name use-application-builder
```

This schematic automatically:
- Changes the builder in `angular.json` from `@angular-devkit/build-angular:browser` to `@angular/build:application`.
- Renames `main` to `browser` in the build options.
- Merges `tsconfig.server.json` into `tsconfig.app.json` if SSR is configured.
- Adds `"esModuleInterop": true` to `tsconfig.json` for CommonJS compatibility.

### Common Migration Issues

**1. Tilde imports in Sass**

The `~` prefix for importing from `node_modules` in Sass is not supported by esbuild.

```scss
// Bad: tilde imports (Webpack-only)
@use '~@angular/material' as mat;

// Good: bare package specifier
@use '@angular/material' as mat;
```

**2. CommonJS default imports**

esbuild handles ES modules natively. CommonJS interop requires `esModuleInterop`:

```typescript
// Before (Webpack tolerated this)
import * as moment from 'moment';

// After (esbuild with esModuleInterop: true)
import moment from 'moment';
```

**3. Custom Webpack plugins**

If your project uses custom Webpack plugins (for fonts, SVGs, or other asset processing), these do not work with esbuild. Replace them with esbuild plugins or move asset processing to build scripts.

**4. Global scripts order**

The `scripts` array in `angular.json` works the same way, but the bundling behavior may differ for scripts that depend on global variable order. Test thoroughly after migration.

## Angular Compiler Options Reference

These options go in `angularCompilerOptions` in `tsconfig.json`.

### Type Checking Options

| Option | Default | Description |
|---|---|---|
| `strictTemplates` | `true` (in strict projects) | Full template type checking |
| `strictInjectionParameters` | `true` | Verify DI injection token types |
| `strictInputAccessModifiers` | `true` | Enforce input access modifiers in templates |

### Compilation Behavior Options

| Option | Default | Description |
|---|---|---|
| `compilationMode` | `'full'` | `'full'` for apps, `'partial'` for libraries |
| `preserveWhitespaces` | `false` | Keep whitespace in templates (increases bundle) |
| `disableExpressionLowering` | `false` | Disable metadata expression lowering |
| `enableI18nLegacyMessageIdFormat` | `true` | Generate legacy i18n message IDs |
| `extendedDiagnostics` | `{}` | Configure extended template diagnostics |

### Extended Diagnostics

Angular provides additional template checks beyond basic type checking:

```json
{
  "angularCompilerOptions": {
    "strictTemplates": true,
    "extendedDiagnostics": {
      "defaultCategory": "warning",
      "checks": {
        "invalidBananaInBox": "error",
        "nullishCoalescingNotNullable": "warning",
        "textAttributeNotBinding": "warning",
        "missingNgForOfLet": "error",
        "suffixNotSupported": "warning",
        "interpolatedSignalNotInvoked": "error"
      }
    }
  }
}
```

Key extended diagnostics:

| Diagnostic | Catches |
|---|---|
| `invalidBananaInBox` | `([ngModel])` instead of `[(ngModel)]` |
| `nullishCoalescingNotNullable` | `value ?? fallback` where `value` is never null |
| `textAttributeNotBinding` | `disabled="true"` (string attribute) instead of `[disabled]="true"` (binding) |
| `interpolatedSignalNotInvoked` | `{{ mySignal }}` instead of `{{ mySignal() }}` |
| `missingNgForOfLet` | `*ngFor="item of items"` missing `let` keyword |

The `interpolatedSignalNotInvoked` diagnostic is particularly important in Angular 21 where signals are pervasive. Writing `{{ count }}` instead of `{{ count() }}` renders the signal function itself, not its value.

## Partial Compilation for Libraries

Libraries published to npm use `compilationMode: 'partial'` instead of `'full'`. Partial compilation pre-processes Angular decorators but leaves the final code generation to the consuming application's compiler. This ensures libraries work across Angular versions.

```json
// tsconfig.lib.json (for a library)
{
  "angularCompilerOptions": {
    "compilationMode": "partial"
  }
}
```

When an application imports a partially compiled library, Angular's "linker" processes it during the application's build. This happens transparently.

Application projects always use `compilationMode: 'full'` (the default). Do not set partial compilation on an application.

## Common Mistakes

- **Ignoring template type-check errors.** Disabling `strictTemplates` or using `$any()` everywhere defeats the purpose of AOT. Template type checking catches real bugs. Fix the types instead of silencing the compiler.

- **Using tilde imports in Sass after migrating to esbuild.** The `~` prefix was a Webpack convention. esbuild resolves bare package specifiers (`@angular/material`) directly from `node_modules`. Remove all `~` prefixes.

- **Not adding `esModuleInterop: true` when importing CommonJS packages.** Without this flag, default imports from CommonJS packages fail. The migration schematic adds it automatically, but manual migrations may miss it.

- **Confusing AOT errors with runtime errors.** If `ng build` fails with a template error, the bug is in your template, not in Angular. Read the error message: it includes the file, line, column, and the expected type. These are compile-time errors, not runtime exceptions.

- **Skipping the build cache on CI.** The `.angular/cache/` directory significantly speeds up rebuilds. On CI, cache this directory between builds (similar to caching `node_modules`). For GitHub Actions:

```yaml
- name: Cache Angular build
  uses: actions/cache@v4
  with:
    path: .angular/cache
    key: angular-cache-${{ hashFiles('**/angular.json') }}
```

- **Setting `preserveWhitespaces: true` without reason.** This adds unnecessary whitespace nodes to the DOM, increasing bundle size and slowing rendering. Keep it `false` (the default) unless you have a specific layout requirement.

- **Using JIT compilation in production.** JIT requires shipping the Angular compiler to the browser (~100KB). It is slower, less secure, and provides no type checking for templates. AOT is the only mode suitable for production.

---

## Best Practices

- **Keep `strictTemplates: true`.** It is the single most effective setting for catching bugs early. Fix type errors in templates rather than working around them.

- **Enable extended diagnostics.** Set `interpolatedSignalNotInvoked` to `error` in Angular 21 projects. Forgetting to call a signal is a common mistake that produces confusing output.

- **Use the `application` builder.** If your project still uses the `browser` builder with Webpack, migrate to `application`. The speed improvement is substantial and the migration is mostly automated.

- **Cache builds on CI.** Caching `.angular/cache/` between CI runs saves minutes on large projects.

- **Monitor bundle size with budgets.** Set `maximumWarning` and `maximumError` in the build configuration. The compiler fails the build if bundles exceed the threshold, preventing accidental size regressions.

- **Use partial compilation for libraries.** Published libraries must use `compilationMode: 'partial'` so they work with any Angular version's linker.

## Real-World Use Cases

### 1. Diagnosing a Template Type Error

The build fails with:

```
Error: src/app/features/users/user-detail.component.html:12:15 - error TS2339:
Property 'fullName' does not exist on type 'User'.

12   <h1>{{ user().fullName }}</h1>
                  ~~~~~~~~
```

Fix: check the `User` interface. Either add `fullName` or use the correct property:

```typescript
// The interface
interface User {
  id: number;
  firstName: string;
  lastName: string;
}

// Fix the template
// Before: {{ user().fullName }}
// After:
{{ user().firstName }} {{ user().lastName }}
```

The compiler tells you exactly what is wrong. Trust it.

### 2. Catching a Signal Invocation Bug

With `interpolatedSignalNotInvoked` set to `error`:

```
Error: src/app/features/dashboard/stats.component.html:5:10 - error NG8109:
'count' is a function and should be invoked: 'count()'

5   <span>{{ count }}</span>
            ~~~~~
```

Fix:

```html
<!-- Before (displays function reference) -->
<span>{{ count }}</span>

<!-- After (displays the signal's value) -->
<span>{{ count() }}</span>
```

Without this diagnostic, the template would render `[Function]` or `function() { ... }` instead of the number, and the bug would only be discovered during testing or in production.

### 3. Migrating a Webpack Project to esbuild

```bash
# Step 1: Update Angular
ng update @angular/core @angular/cli

# Step 2: Run the migration schematic
ng update @angular/cli --name use-application-builder

# Step 3: Fix Sass tilde imports
# Find: @use '~
# Replace with: @use '

# Step 4: Fix CommonJS imports if needed
# Find: import * as X from 'package'
# Replace with: import X from 'package'

# Step 5: Build and test
ng build
ng test --watch=false

# Step 6: Compare bundle sizes
# Before migration: note the dist/ output size
# After migration: compare (usually similar or smaller)

# Step 7: Verify dev server
ng serve
# Check that HMR works for templates and styles
```

The entire migration typically takes 30-60 minutes for a medium project, with most of that time spent on fixing Sass imports and testing edge cases.

---

> **Key Takeaway:** Angular's AOT compiler catches template errors at build time, and the esbuild-powered build system delivers those checks with dramatically faster build speeds. Keep `strictTemplates` enabled, use the `application` builder, and treat compiler errors as bugs in your code, not obstacles to work around.
