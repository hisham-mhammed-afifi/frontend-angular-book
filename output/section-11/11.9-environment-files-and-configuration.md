# 11.9 Environment Files and Configuration

> Angular provides build-time environment files and file replacements so you can ship different configuration per target (development, staging, production). For more dynamic needs, runtime configuration patterns let you build once and deploy everywhere.

---

## Setting Up Environment Files

Angular 21 projects do not include environment files by default. Generate them with:

```bash
ng generate environments
```

This creates:

```
src/
  environments/
    environment.ts              # Default (production)
    environment.development.ts  # Development overrides
```

It also adds the necessary `fileReplacements` entry to `angular.json` so the CLI knows which file to swap during builds.

### The Default Environment File

```typescript
// src/environments/environment.ts
// This is the production (default) configuration.
// Values here are used unless a build configuration swaps this file.

export const environment = {
  production: true,
  apiUrl: 'https://api.example.com',
  featureFlags: {
    newDashboard: true,
    betaReports: false
  }
};
```

### The Development Override

```typescript
// src/environments/environment.development.ts
export const environment = {
  production: false,
  apiUrl: 'http://localhost:3000',
  featureFlags: {
    newDashboard: true,
    betaReports: true
  }
};
```

### Importing in Application Code

Always import from the base file. The build system replaces it at compile time:

```typescript
// src/app/services/api.service.ts
import { environment } from '../environments/environment';

// At build time, this import resolves to whichever file
// the active configuration specifies via fileReplacements.

const API_URL = environment.apiUrl;
```

## How fileReplacements Work

The `fileReplacements` option in `angular.json` tells the builder to substitute one file for another during compilation. The original file is never modified; esbuild simply resolves imports of the original to the replacement.

```json
{
  "architect": {
    "build": {
      "builder": "@angular/build:application",
      "configurations": {
        "development": {
          "fileReplacements": [
            {
              "replace": "src/environments/environment.ts",
              "with": "src/environments/environment.development.ts"
            }
          ]
        }
      }
    }
  }
}
```

When you run:

```bash
ng build                           # Uses environment.ts (production default)
ng build --configuration development  # Swaps in environment.development.ts
ng serve                           # Uses development by default (configured via defaultConfiguration)
```

The `ng generate environments` command sets `"defaultConfiguration": "development"` on the `serve` target, so `ng serve` uses the development environment automatically.

### File Replacement Is Not Limited to Environments

You can replace any file, not just environment files:

```json
{
  "fileReplacements": [
    {
      "replace": "src/app/services/analytics.service.ts",
      "with": "src/app/services/analytics.service.mock.ts"
    }
  ]
}
```

This is useful for swapping mock services, feature implementations, or platform-specific code.

## Adding Custom Environments

Most real projects need more than two environments.

### Step 1: Create the Environment File

```typescript
// src/environments/environment.staging.ts
export const environment = {
  production: true,
  apiUrl: 'https://staging-api.example.com',
  featureFlags: {
    newDashboard: true,
    betaReports: true
  }
};
```

### Step 2: Add the Configuration to angular.json

```json
{
  "configurations": {
    "production": {
      "budgets": [/* ... */],
      "outputHashing": "all"
    },
    "staging": {
      "budgets": [/* ... */],
      "outputHashing": "all",
      "fileReplacements": [
        {
          "replace": "src/environments/environment.ts",
          "with": "src/environments/environment.staging.ts"
        }
      ]
    },
    "development": {
      "optimization": false,
      "extractLicenses": false,
      "sourceMap": true,
      "fileReplacements": [
        {
          "replace": "src/environments/environment.ts",
          "with": "src/environments/environment.development.ts"
        }
      ]
    }
  }
}
```

### Step 3: Build with the New Configuration

```bash
ng build --configuration staging
```

### Typical Multi-Environment Setup

| Environment | `production` | API URL | Source Maps | Optimization |
|-------------|-------------|---------|-------------|--------------|
| development | `false` | `http://localhost:3000` | Yes | No |
| staging | `true` | `https://staging-api.example.com` | Hidden | Yes |
| qa | `true` | `https://qa-api.example.com` | Hidden | Yes |
| production | `true` | `https://api.example.com` | No | Yes |

## Type Safety for Environment Files

Without proper typing, nothing prevents environment files from drifting out of sync. One file might add a property that another forgets.

### Define an Interface

```typescript
// src/environments/environment.interface.ts
export interface Environment {
  production: boolean;
  apiUrl: string;
  featureFlags: {
    newDashboard: boolean;
    betaReports: boolean;
  };
  sentryDsn?: string;
}
```

### Use It in Every Environment File

```typescript
// src/environments/environment.ts
import { Environment } from './environment.interface';

export const environment: Environment = {
  production: true,
  apiUrl: 'https://api.example.com',
  featureFlags: {
    newDashboard: true,
    betaReports: false
  },
  sentryDsn: 'https://abc@sentry.io/123'
};
```

```typescript
// src/environments/environment.development.ts
import { Environment } from './environment.interface';

export const environment: Environment = {
  production: false,
  apiUrl: 'http://localhost:3000',
  featureFlags: {
    newDashboard: true,
    betaReports: true
  }
  // sentryDsn omitted intentionally (optional)
};
```

Now TypeScript will catch missing required properties at compile time if any environment file falls out of sync.

## The `define` Option (Build-Time Injection)

Angular 17.2+ introduced the `define` option in the application builder. This injects compile-time constants via esbuild's define feature, providing an alternative to file replacements for simple values.

### Configuration in angular.json

```json
{
  "architect": {
    "build": {
      "builder": "@angular/build:application",
      "options": {
        "define": {
          "APP_VERSION": "'1.5.0'"
        }
      },
      "configurations": {
        "production": {
          "define": {
            "API_URL": "'https://api.example.com'",
            "ENABLE_LOGGING": "false"
          }
        },
        "staging": {
          "define": {
            "API_URL": "'https://staging-api.example.com'",
            "ENABLE_LOGGING": "true"
          }
        },
        "development": {
          "define": {
            "API_URL": "'http://localhost:3000'",
            "ENABLE_LOGGING": "true"
          }
        }
      }
    }
  }
}
```

### CLI Override with --define

You can override or add define values from the command line:

```bash
ng build --define "API_URL='https://custom-api.example.com'"
```

This is powerful for CI/CD pipelines where the API URL comes from a pipeline variable:

```yaml
# GitHub Actions example
- name: Build
  run: ng build --define "API_URL='${{ vars.API_URL }}'" --define "SENTRY_DSN='${{ secrets.SENTRY_DSN }}'"
```

### TypeScript Declarations

Define a type declaration file so TypeScript knows about the injected constants:

```typescript
// src/build-env.d.ts
declare const API_URL: string;
declare const APP_VERSION: string;
declare const ENABLE_LOGGING: boolean;
```

### Using Defined Constants

```typescript
// src/app/app.config.ts
import { ApplicationConfig } from '@angular/core';

declare const API_URL: string;

export const appConfig: ApplicationConfig = {
  providers: [
    { provide: 'API_URL', useValue: API_URL }
  ]
};
```

```typescript
// src/app/services/api.service.ts
import { inject, Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';

declare const API_URL: string;

@Injectable({ providedIn: 'root' })
export class ApiService {
  private http = inject(HttpClient);

  // API_URL is replaced with the literal string at build time.
  // The declare statement above tells TypeScript the variable exists.
  getUsers() {
    return this.http.get(`${API_URL}/users`);
  }
}
```

### define vs fileReplacements

| Aspect | `define` | `fileReplacements` |
|--------|---------|-------------------|
| Granularity | Individual values | Entire files |
| Setup complexity | Low (key-value pairs) | Medium (separate files per env) |
| Type safety | Manual `declare const` | Interface-backed exports |
| CLI override | Yes (`--define`) | No |
| Tree shaking | Excellent (dead code from `false` flags removed) | Good |
| Best for | Simple constants, feature flags, URLs | Complex config objects, service swaps |

## Runtime Configuration (Build Once, Deploy Everywhere)

Build-time configuration requires a separate build per environment. Runtime configuration loads settings after the app starts, so one build artifact works in every environment.

### Pattern 1: JSON Configuration File

#### Step 1: Create the Config File

```json
// src/assets/config.json
{
  "apiUrl": "https://api.example.com",
  "authDomain": "auth.example.com",
  "sentryDsn": "https://abc@sentry.io/123",
  "featureFlags": {
    "newDashboard": true
  }
}
```

#### Step 2: Create a Configuration Service

```typescript
// src/app/core/config.service.ts
import { Injectable } from '@angular/core';

export interface AppConfig {
  apiUrl: string;
  authDomain: string;
  sentryDsn: string;
  featureFlags: {
    newDashboard: boolean;
  };
}

@Injectable({ providedIn: 'root' })
export class ConfigService {
  private config!: AppConfig;

  async load(): Promise<void> {
    const response = await fetch('/assets/config.json');
    if (!response.ok) {
      throw new Error(`Failed to load config: ${response.status}`);
    }
    this.config = await response.json();
  }

  get apiUrl(): string {
    return this.config.apiUrl;
  }

  get authDomain(): string {
    return this.config.authDomain;
  }

  get sentryDsn(): string {
    return this.config.sentryDsn;
  }

  get featureFlags(): AppConfig['featureFlags'] {
    return this.config.featureFlags;
  }

  // Provide the entire config if needed
  get all(): Readonly<AppConfig> {
    return this.config;
  }
}
```

#### Step 3: Load Before Bootstrap

```typescript
// src/main.ts
import { bootstrapApplication } from '@angular/platform-browser';
import { AppComponent } from './app/app.component';
import { appConfig } from './app/app.config';
import { ConfigService } from './app/core/config.service';

async function main() {
  // Load config BEFORE Angular bootstraps
  const configService = new ConfigService();
  await configService.load();

  await bootstrapApplication(AppComponent, {
    ...appConfig,
    providers: [
      ...appConfig.providers,
      { provide: ConfigService, useValue: configService }
    ]
  });
}

main().catch(err => console.error('Bootstrap failed:', err));
```

#### Alternative: Using provideAppInitializer

```typescript
// src/app/app.config.ts
import { ApplicationConfig, provideAppInitializer, inject } from '@angular/core';
import { ConfigService } from './core/config.service';

export const appConfig: ApplicationConfig = {
  providers: [
    provideAppInitializer(() => {
      const configService = inject(ConfigService);
      return configService.load();
    })
  ]
};
```

With `provideAppInitializer`, Angular waits for the returned Promise to resolve before rendering any component. This is cleaner than the pre-bootstrap approach but means the `ConfigService` is created by Angular's DI, so it starts without data and loads it during initialization.

### Pattern 2: Window Object Injection (Docker/Kubernetes)

This pattern injects configuration into a script tag at container startup.

#### Step 1: Create a Config Template

```html
<!-- src/assets/env-config.template.js -->
(function(window) {
  window.__env = {
    apiUrl: '${API_URL}',
    authDomain: '${AUTH_DOMAIN}',
    sentryDsn: '${SENTRY_DSN}'
  };
})(window);
```

#### Step 2: Add to index.html

```html
<!-- src/index.html -->
<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>MyApp</title>
  <base href="/">
  <script src="assets/env-config.js"></script>
</head>
<body>
  <app-root></app-root>
</body>
</html>
```

#### Step 3: Docker Entrypoint Script

```bash
#!/bin/sh
# docker-entrypoint.sh
# Replace placeholders with actual environment variables

envsubst < /usr/share/nginx/html/assets/env-config.template.js \
         > /usr/share/nginx/html/assets/env-config.js

exec nginx -g 'daemon off;'
```

#### Step 4: Dockerfile

```dockerfile
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npx ng build --configuration production

FROM nginx:alpine
COPY --from=build /app/dist/my-app/browser /usr/share/nginx/html
COPY docker-entrypoint.sh /docker-entrypoint.sh
RUN chmod +x /docker-entrypoint.sh

ENTRYPOINT ["/docker-entrypoint.sh"]
```

#### Step 5: Read from Window in Angular

```typescript
// src/app/core/env-config.service.ts
import { Injectable } from '@angular/core';

interface EnvConfig {
  apiUrl: string;
  authDomain: string;
  sentryDsn: string;
}

@Injectable({ providedIn: 'root' })
export class EnvConfigService {
  private config: EnvConfig;

  constructor() {
    const windowEnv = (window as any).__env;
    if (!windowEnv) {
      throw new Error('Runtime environment configuration not found. Is env-config.js loaded?');
    }
    this.config = windowEnv;
  }

  get apiUrl(): string {
    return this.config.apiUrl;
  }

  get authDomain(): string {
    return this.config.authDomain;
  }

  get sentryDsn(): string {
    return this.config.sentryDsn;
  }
}
```

#### Step 6: Run with Environment Variables

```bash
docker run -e API_URL=https://api.example.com \
           -e AUTH_DOMAIN=auth.example.com \
           -e SENTRY_DSN=https://abc@sentry.io/123 \
           -p 8080:80 my-app
```

The same Docker image works for every environment. Only the environment variables change.

## Providing Configuration via Dependency Injection

Rather than accessing the `ConfigService` directly, you can use injection tokens for cleaner architecture:

```typescript
// src/app/core/config.tokens.ts
import { InjectionToken } from '@angular/core';

export const API_URL = new InjectionToken<string>('API_URL');
export const AUTH_DOMAIN = new InjectionToken<string>('AUTH_DOMAIN');
export const FEATURE_FLAGS = new InjectionToken<Record<string, boolean>>('FEATURE_FLAGS');
```

```typescript
// src/app/app.config.ts
import { ApplicationConfig, provideAppInitializer, inject } from '@angular/core';
import { ConfigService } from './core/config.service';
import { API_URL, AUTH_DOMAIN, FEATURE_FLAGS } from './core/config.tokens';

export const appConfig: ApplicationConfig = {
  providers: [
    provideAppInitializer(() => {
      const configService = inject(ConfigService);
      return configService.load();
    }),
    {
      provide: API_URL,
      useFactory: () => inject(ConfigService).apiUrl
    },
    {
      provide: AUTH_DOMAIN,
      useFactory: () => inject(ConfigService).authDomain
    },
    {
      provide: FEATURE_FLAGS,
      useFactory: () => inject(ConfigService).featureFlags
    }
  ]
};
```

```typescript
// src/app/services/api.service.ts
import { inject, Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { API_URL } from '../core/config.tokens';

@Injectable({ providedIn: 'root' })
export class ApiService {
  private http = inject(HttpClient);
  private apiUrl = inject(API_URL);

  getUsers() {
    return this.http.get(`${this.apiUrl}/users`);
  }
}
```

This makes testing trivial since you can override injection tokens in tests without touching the config service.

## Choosing the Right Strategy

| Criterion | Build-Time Files | `define` Flag | Runtime JSON | Window Injection |
|-----------|-----------------|---------------|-------------|------------------|
| Build per environment | Yes | Yes | No | No |
| CI/CD friendly | Moderate | Excellent | Excellent | Excellent |
| Docker compatible | Requires rebuild | Requires rebuild | Yes | Yes |
| SSR compatible | Yes | Yes | Needs server route | Needs template |
| Setup complexity | Low | Low | Medium | Medium-High |
| Secret safety | Code-level | CLI-level | File-level | Container-level |
| Tree shaking | Good | Excellent | None | None |
| Best for | Small projects, few environments | Feature flags, CI injection | Multi-tenant SaaS | Docker/Kubernetes |

### Decision Flowchart

1. **Do you deploy the same build to multiple environments?**
   - No: Use build-time files or `define`
   - Yes: Use runtime configuration (JSON or window injection)

2. **Do you need dead code elimination for feature flags?**
   - Yes: Use `define` (esbuild removes unreachable branches)
   - No: Any approach works

3. **Are you running in Docker/Kubernetes?**
   - Yes: Window injection or runtime JSON with config map
   - No: Build-time approaches are simpler

4. **Do you have more than 4 environments?**
   - Yes: Runtime configuration avoids maintaining many environment files
   - No: Build-time files are manageable

## Common Mistakes

- **Storing secrets in environment files.** Environment files are compiled into the JavaScript bundle. Anyone who opens DevTools can read them. API keys, database credentials, and tokens must never go in `environment.ts`. Use server-side proxies or backend APIs to handle secrets.

- **Forgetting to create a staging/qa environment file.** Teams add new properties to `environment.development.ts` and `environment.ts` but forget `environment.staging.ts`. Without a shared interface, this silently ships `undefined` values to staging.

- **Importing environment.development.ts directly.** Always import from `environment.ts`. If you import a specific environment file, file replacement does not apply, and you get the wrong values in other configurations.

```typescript
// BAD: Hardcoded to development, ignores file replacement
import { environment } from '../environments/environment.development';

// GOOD: Replaced by the build system based on configuration
import { environment } from '../environments/environment';
```

- **Using define values without inner quotes for strings.** The `define` option takes raw JavaScript expressions. Writing `"API_URL": "https://example.com"` is interpreted as JavaScript, not a string literal. You need `"API_URL": "'https://example.com'"` with inner single quotes.

- **Loading runtime config inside a component instead of before bootstrap.** If you fetch `/assets/config.json` inside a component, the app renders before the config is available. Use `provideAppInitializer` or load before `bootstrapApplication` to guarantee config is ready before any component runs.

- **Not handling fetch failures for runtime config.** If `/assets/config.json` fails to load (404, network error), the app should fail loudly rather than silently running with undefined values. Always check the response status and throw on failure.

---

## Best Practices

- **Use a shared `Environment` interface.** Define a TypeScript interface for your environment shape and type every environment file with it. The compiler catches missing or mistyped properties.

- **Keep environment files minimal.** Only put values that genuinely differ per environment. Shared constants that never change belong in a regular constants file, not in environment files.

- **Use `define` for feature flags in CI/CD.** The `--define` CLI flag lets pipeline variables flow into the build without maintaining separate environment files. Combined with esbuild's dead code elimination, disabled features are removed from the bundle entirely.

- **Prefer runtime config for containerized deployments.** If you build Docker images in CI and deploy to multiple environments, runtime configuration (JSON fetch or window injection) avoids separate builds per target.

- **Add environment files to .gitignore selectively.** If environment files contain sensitive values (even non-secret ones like internal staging URLs), consider using `.env` files with a build script that generates `environment.*.ts` from templates, rather than committing values directly.

- **Validate runtime config at load time.** When using runtime JSON or window injection, validate the loaded config against a schema or type guard before the app uses it. Catch misconfiguration early.

```typescript
function validateConfig(config: unknown): asserts config is AppConfig {
  const c = config as Record<string, unknown>;
  if (typeof c.apiUrl !== 'string' || !c.apiUrl) {
    throw new Error('Invalid config: apiUrl is required');
  }
  if (typeof c.authDomain !== 'string' || !c.authDomain) {
    throw new Error('Invalid config: authDomain is required');
  }
}
```

## Real-World Use Cases

### 1. Multi-Tenant SaaS with Runtime Config

A SaaS product deploys the same Angular build to multiple customer tenants, each with their own API and branding:

```json
// /assets/config.json (served per-tenant by the CDN or reverse proxy)
{
  "apiUrl": "https://tenant-a.api.example.com",
  "tenantName": "Acme Corp",
  "theme": "blue",
  "logoUrl": "/assets/tenants/acme/logo.svg",
  "features": {
    "billing": true,
    "reports": true,
    "adminPanel": false
  }
}
```

```typescript
// src/app/core/tenant.service.ts
import { Injectable, signal, computed } from '@angular/core';

export interface TenantConfig {
  apiUrl: string;
  tenantName: string;
  theme: string;
  logoUrl: string;
  features: Record<string, boolean>;
}

@Injectable({ providedIn: 'root' })
export class TenantService {
  private _config = signal<TenantConfig | null>(null);

  config = this._config.asReadonly();
  tenantName = computed(() => this._config()?.tenantName ?? 'App');
  theme = computed(() => this._config()?.theme ?? 'default');

  async load(): Promise<void> {
    const response = await fetch('/assets/config.json');
    if (!response.ok) throw new Error('Tenant config load failed');
    this._config.set(await response.json());
  }

  isFeatureEnabled(feature: string): boolean {
    return this._config()?.features[feature] ?? false;
  }
}
```

### 2. Feature Flag Gating with define

A product team uses compile-time feature flags to gate unreleased features:

```json
// angular.json (partial)
{
  "configurations": {
    "production": {
      "define": {
        "FEATURE_NEW_EDITOR": "false",
        "FEATURE_AI_ASSIST": "false",
        "FEATURE_DARK_MODE": "true"
      }
    },
    "canary": {
      "define": {
        "FEATURE_NEW_EDITOR": "true",
        "FEATURE_AI_ASSIST": "true",
        "FEATURE_DARK_MODE": "true"
      }
    }
  }
}
```

```typescript
// src/app/features/editor/editor-page.component.ts
declare const FEATURE_NEW_EDITOR: boolean;

@Component({
  selector: 'app-editor-page',
  template: `
    @if (useNewEditor) {
      <app-new-editor />
    } @else {
      <app-legacy-editor />
    }
  `
})
export class EditorPageComponent {
  // esbuild replaces FEATURE_NEW_EDITOR with true or false.
  // When false, the entire @if block and the NewEditorComponent import
  // are removed from the production bundle via tree shaking.
  useNewEditor = FEATURE_NEW_EDITOR;
}
```

Build for canary users: `ng build --configuration canary`
Build for production: `ng build --configuration production`

The production bundle physically does not contain the new editor code.

### 3. Kubernetes ConfigMap with Window Injection

A DevOps team manages Angular configuration through Kubernetes ConfigMaps:

```yaml
# k8s/configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: frontend-config
data:
  env-config.js: |
    (function(window) {
      window.__env = {
        apiUrl: 'https://api.prod.internal',
        authDomain: 'auth.prod.internal',
        logLevel: 'warn'
      };
    })(window);
```

```yaml
# k8s/deployment.yaml
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      containers:
        - name: frontend
          image: my-app:latest
          volumeMounts:
            - name: config-volume
              mountPath: /usr/share/nginx/html/assets/env-config.js
              subPath: env-config.js
      volumes:
        - name: config-volume
          configMap:
            name: frontend-config
```

The ConfigMap mounts the JavaScript file directly into the Nginx container, overwriting the placeholder. No entrypoint script needed. Changing the ConfigMap and restarting the pods updates the configuration without rebuilding the image.

---

> **Key Takeaway:** Use build-time environment files (`fileReplacements`) or `define` for simple projects with few environments; switch to runtime configuration (JSON fetch or window injection) when you need to build once and deploy the same artifact to multiple environments.
