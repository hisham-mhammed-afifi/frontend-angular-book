# 12.12 Deferrable Views

> `@defer` lets you lazily load parts of a template on demand, splitting heavy components into separate JavaScript chunks that are fetched only when needed. This reduces initial bundle size and improves Core Web Vitals without manual lazy loading plumbing.

---

## What Deferrable Views Solve

Traditional Angular applications eagerly load every component referenced in a template. If your dashboard page imports a charting library, a rich text editor, and a comments section, all of that JavaScript ships in the initial bundle even if the user never scrolls down to see the comments.

Before `@defer`, you had two options:
1. Route-level lazy loading (works only at the route boundary).
2. Manual dynamic imports with `ViewContainerRef.createComponent()` (verbose and error-prone).

`@defer` solves this at the template level. Wrap any section of your template in `@defer` and Angular automatically:
- Splits the dependencies into a separate chunk at build time.
- Loads that chunk only when the trigger condition is met.
- Renders placeholder, loading, and error states.
- Handles prefetching for optimal perceived performance.

---

## Basic Syntax

```html
@defer {
  <app-heavy-chart [data]="chartData()" />
}
```

This is the simplest form. By default, the deferred block uses an `idle` trigger, meaning the browser loads the chunk when it reaches an idle state (via `requestIdleCallback`).

The full syntax with all optional sub-blocks:

```html
@defer (on viewport) {
  <app-comments [postId]="postId()" />
} @placeholder (minimum 500ms) {
  <div class="comments-skeleton">
    <div class="skeleton-line"></div>
    <div class="skeleton-line"></div>
    <div class="skeleton-line"></div>
  </div>
} @loading (after 300ms; minimum 500ms) {
  <div class="loading-spinner">Loading comments...</div>
} @error {
  <div class="error-message">
    Failed to load comments.
    <button (click)="retry()">Retry</button>
  </div>
}
```

## Sub-Blocks

### `@placeholder`

The `@placeholder` block is what the user sees before the deferred content has started loading. Its contents are eagerly loaded (part of the initial bundle).

```html
@defer (on interaction) {
  <app-rich-editor [content]="content()" />
} @placeholder {
  <div class="editor-placeholder" tabindex="0">
    Click to load editor...
  </div>
}
```

**Parameters:**

| Parameter | Description | Example |
|---|---|---|
| `minimum` | Minimum time to show the placeholder, preventing a flash if loading is fast | `@placeholder (minimum 500ms)` |

The `minimum` parameter is useful when the deferred chunk loads almost instantly (from cache or because it is tiny). Without it, the user sees a brief flash of the placeholder before the real content appears. Setting `minimum 500ms` ensures the placeholder stays visible for at least half a second.

```html
@defer {
  <app-dashboard-widget />
} @placeholder (minimum 300ms) {
  <div class="widget-skeleton">Loading widget...</div>
}
```

### `@loading`

The `@loading` block appears while the JavaScript chunk is being fetched. Like `@placeholder`, its contents are eagerly loaded.

```html
@defer (on viewport) {
  <app-analytics-panel />
} @loading (after 200ms; minimum 500ms) {
  <app-spinner />
}
```

**Parameters:**

| Parameter | Description | Example |
|---|---|---|
| `after` | Wait this long after loading starts before showing the loading block | `after 200ms` |
| `minimum` | Keep the loading block visible for at least this long | `minimum 500ms` |

The `after` parameter prevents a loading indicator from appearing for fast loads. If the chunk arrives within 200ms, the user never sees a spinner. The `minimum` parameter prevents the spinner from flashing briefly. Together, they produce smooth loading UX:

- Chunk loads in <200ms: User goes directly from placeholder to content (no spinner).
- Chunk loads in 200ms-700ms: Spinner appears after 200ms and stays for at least 500ms.
- Chunk loads in >700ms: Spinner appears after 200ms and stays until loading completes.

### `@error`

The `@error` block appears if the JavaScript chunk fails to load (network error, server error, etc.).

```html
@defer {
  <app-third-party-widget />
} @error {
  <div class="error-state">
    <p>Could not load widget.</p>
    <button (click)="reloadPage()">Reload page</button>
  </div>
}
```

The `@error` block's dependencies are also eagerly loaded. Keep it lightweight. Angular does not provide a built-in retry mechanism for `@defer`, so if you want retry behavior, you need to trigger a page reload or re-render the parent.

## Trigger Conditions

Triggers determine when Angular starts fetching the deferred chunk. You specify them with the `on` keyword.

### `on idle` (Default)

```html
@defer {
  <app-recommendations />
}

<!-- Equivalent to: -->
@defer (on idle) {
  <app-recommendations />
}
```

Uses `requestIdleCallback` to load the chunk when the browser's main thread is idle. This is the default trigger when you do not specify one. Good for content that should load soon but is not critical for the initial paint.

### `on viewport`

```html
@defer (on viewport) {
  <app-footer-content />
} @placeholder {
  <div style="height: 200px;"></div>
}
```

Uses the `IntersectionObserver` API to load the chunk when the placeholder element enters the viewport. This is the most common trigger for below-the-fold content.

**Important:** The viewport trigger observes the placeholder element by default. If there is no `@placeholder` block, it observes a comment node, which has no dimensions and may trigger immediately. Always include a `@placeholder` with actual height when using `on viewport`.

You can also target a specific template reference variable:

```html
<div #triggerArea class="scroll-sentinel"></div>

<!-- The defer block triggers when #triggerArea enters the viewport -->
@defer (on viewport(triggerArea)) {
  <app-lazy-section />
} @placeholder {
  <div>Scroll down to load more</div>
}
```

### `on interaction`

```html
@defer (on interaction) {
  <app-rich-text-editor [content]="document()" />
} @placeholder {
  <div class="editor-preview" tabindex="0">
    {{ document().substring(0, 200) }}...
    <span class="hint">Click to edit</span>
  </div>
}
```

Loads the chunk when the user clicks or presses a key on the placeholder element (uses `click` and `keydown` events). You can target a specific element:

```html
<button #editBtn>Edit Document</button>

@defer (on interaction(editBtn)) {
  <app-rich-text-editor [content]="document()" />
} @placeholder {
  <div class="document-preview">{{ document() }}</div>
}
```

### `on hover`

```html
@defer (on hover) {
  <app-tooltip-content [data]="tooltipData()" />
} @placeholder {
  <span class="hoverable">Hover for details</span>
}
```

Triggers on `mouseenter` and `focusin` events. Useful for content that appears on hover, like tooltips, previews, or expanded card details. Like `interaction`, you can target a specific element with a template reference variable.

### `on timer`

```html
@defer (on timer(3s)) {
  <app-promotional-banner />
} @placeholder {
  <div style="height: 80px;"></div>
}
```

Loads after the specified duration. The timer starts when the `@defer` block is first rendered in the template. Accepts milliseconds (`500ms`) or seconds (`3s`).

### `on immediate`

```html
@defer (on immediate) {
  <app-above-fold-widget />
}
```

Starts loading immediately during template rendering, without waiting for idle. The chunk is still separate (code-split), but the fetch begins right away. Use this when you want code splitting but not deferred loading.

### `when` (Imperative Boolean Trigger)

```html
@defer (when showComments()) {
  <app-comments [postId]="postId()" />
} @placeholder {
  <button (click)="showComments.set(true)">Show Comments</button>
}
```

```typescript
export class PostComponent {
  postId = input.required<string>();
  showComments = signal(false);
}
```

The `when` condition is checked every change detection cycle. Once it becomes `true`, loading starts. The condition is one-shot: once it triggers, the deferred content stays loaded even if the condition becomes `false` again.

### Combining Triggers

You can combine multiple triggers. They act as OR conditions, meaning the first trigger that fires initiates loading.

```html
@defer (on viewport; on timer(5s)) {
  <app-comments [postId]="postId()" />
} @placeholder {
  <div class="comment-area" style="min-height: 100px;">
    Comments will appear here
  </div>
}
```

This loads comments when the user scrolls to them OR after 5 seconds, whichever comes first.

You can also combine `when` with `on` triggers:

```html
@defer (when userLoggedIn(); on interaction) {
  <app-account-panel />
} @placeholder {
  <div>Log in or click to see your account</div>
}
```

## Prefetching

Prefetching separates when the chunk is fetched from when it is rendered. The chunk downloads in advance, and when the actual trigger fires, the content renders instantly from the already-loaded code.

```html
@defer (on interaction; prefetch on hover) {
  <app-settings-panel />
} @placeholder {
  <button>Open Settings</button>
}
```

In this example:
1. When the user hovers over the placeholder, Angular starts downloading the chunk.
2. When the user clicks, the chunk is already cached, so `<app-settings-panel>` renders immediately.

### Prefetch Triggers

Prefetch supports the same trigger types as the main block:

```html
<!-- Prefetch when idle, render on viewport -->
@defer (on viewport; prefetch on idle) {
  <app-below-fold />
} @placeholder {
  <div style="height: 400px;"></div>
}

<!-- Prefetch on timer, render on interaction -->
@defer (on interaction; prefetch on timer(2s)) {
  <app-editor />
} @placeholder {
  <div>Click to edit</div>
}

<!-- Prefetch with when condition -->
@defer (on interaction; prefetch when prefetchReady()) {
  <app-dashboard />
} @placeholder {
  <div>Loading dashboard...</div>
}
```

### Prefetch `on idle` Pattern

This is arguably the most useful pattern. Prefetch the chunk as soon as the browser is idle, but only show the content when the user scrolls to it:

```html
@defer (on viewport; prefetch on idle) {
  <app-feature-comparison-table />
} @placeholder (minimum 300ms) {
  <div class="table-skeleton" style="height: 400px;">
    Loading comparison...
  </div>
}
```

The chunk starts downloading during idle time. By the time the user scrolls down, the JavaScript is already cached and the component renders instantly.

## Requirements and Constraints

### Components Must Be Standalone

`@defer` only works with standalone components, directives, and pipes. Non-standalone dependencies are eagerly loaded even when placed inside `@defer`.

```typescript
// WORKS: standalone component
@Component({
  selector: 'app-chart',
  standalone: true,
  template: `<canvas #chart></canvas>`,
  styles: [`:host { display: block; }`]
})
export class ChartComponent {}
```

```html
<!-- This correctly lazy-loads ChartComponent -->
@defer {
  <app-chart [data]="chartData()" />
}
```

```typescript
// BAD PRACTICE: Non-standalone component in @defer
// This will NOT be lazy-loaded. It will be included in the main bundle.
@Component({
  selector: 'app-legacy-chart',
  template: `<canvas #chart></canvas>`
  // No standalone: true
})
export class LegacyChartComponent {}
```

### No References Outside `@defer`

If a component is referenced anywhere outside a `@defer` block in the same file, Angular includes it in the main bundle. The compiler detects that the dependency is needed eagerly.

```html
<!-- BAD PRACTICE: Same component used both inside and outside @defer -->
<app-chart [data]="summaryData()" />

@defer (on viewport) {
  <!-- This will NOT be lazy-loaded because app-chart is also used above -->
  <app-chart [data]="detailedData()" />
}
```

The fix is to ensure the deferred component appears only inside `@defer` blocks, or split it into two separate components.

### No `viewChild` / `contentChild` References

Using `viewChild()` or `contentChild()` to reference a deferred component creates a static import, pulling it into the main bundle.

```typescript
// BAD PRACTICE: viewChild defeats @defer
export class DashboardComponent {
  // This creates a static reference, preventing lazy loading
  chart = viewChild(ChartComponent);
}
```

```html
@defer {
  <app-chart [data]="chartData()" />
}
```

Even though `<app-chart>` is inside `@defer`, the `viewChild(ChartComponent)` import in the class file forces the bundler to include `ChartComponent` eagerly.

### No Shared File Exports

If the deferred component's file also exports types, constants, or tokens that are imported by eager parts of the application, the entire file gets pulled into the main bundle.

```typescript
// chart.component.ts

// BAD PRACTICE: Exporting a utility alongside the component
export function formatChartLabel(value: number): string {
  return value.toFixed(2);
}

@Component({
  selector: 'app-chart',
  standalone: true,
  template: `...`
})
export class ChartComponent {}
```

If any eager component imports `formatChartLabel`, the bundler includes the entire `chart.component.ts` file, defeating `@defer`. Move shared utilities to separate files.

## How It Works Under the Hood

When Angular's compiler encounters a `@defer` block, it:

1. **Identifies dependencies**: Finds all components, directives, and pipes used inside the block.
2. **Creates a separate chunk**: Moves those dependencies into a separate JavaScript file during the build.
3. **Generates a loading function**: Replaces the direct import with a dynamic `import()` call.
4. **Sets up the trigger**: Registers the appropriate browser API (IntersectionObserver, requestIdleCallback, event listeners, etc.).
5. **Manages lifecycle**: Transitions between placeholder, loading, content, and error states.

The resulting chunk is a standard dynamic import, so it benefits from the browser's HTTP cache. If the user navigates away and comes back, the chunk may already be cached.

## Nesting `@defer` Blocks

You can nest `@defer` blocks, but be aware of cascading loads.

```html
@defer (on viewport) {
  <div class="comments-section">
    <h2>Comments</h2>
    <app-comment-list [comments]="comments()" />

    @defer (on interaction) {
      <app-comment-form [postId]="postId()" />
    } @placeholder {
      <button>Write a comment</button>
    }
  </div>
} @placeholder {
  <div style="height: 200px;">Comments section</div>
}
```

This is fine because the inner `@defer` has a different trigger. The outer block loads when the user scrolls to it, and the inner block loads only when the user clicks "Write a comment".

**Bad:** Nesting `@defer` blocks that both trigger immediately creates a waterfall where the inner block cannot load until the outer block finishes loading and renders.

```html
<!-- BAD PRACTICE: Cascading immediate triggers -->
@defer (on immediate) {
  <div>
    @defer (on immediate) {
      <app-nested-widget />
    }
  </div>
}
```

The inner component cannot start loading until the outer chunk is fetched, parsed, and executed. This creates a request waterfall that slows down loading.

## Testing Deferrable Views

Angular provides testing utilities for controlling `@defer` block states in unit tests.

```typescript
import { ComponentFixture, TestBed, DeferBlockState, DeferBlockFixture } from '@angular/core/testing';

describe('PostPageComponent', () => {
  let fixture: ComponentFixture<PostPageComponent>;

  beforeEach(async () => {
    await TestBed.configureTestingModule({
      imports: [PostPageComponent]
    }).compileComponents();

    fixture = TestBed.createComponent(PostPageComponent);
    fixture.detectChanges();
  });

  it('should show placeholder initially', async () => {
    // Get the first @defer block in the template
    const deferBlock = (await fixture.getDeferBlocks())[0];

    // Default state is Placeholder
    expect(fixture.nativeElement.textContent).toContain('Loading comments...');
  });

  it('should show loading state', async () => {
    const deferBlock = (await fixture.getDeferBlocks())[0];

    // Manually transition to loading state
    await deferBlock.render(DeferBlockState.Loading);
    fixture.detectChanges();

    expect(fixture.nativeElement.querySelector('.spinner')).toBeTruthy();
  });

  it('should render deferred content', async () => {
    const deferBlock = (await fixture.getDeferBlocks())[0];

    // Transition to complete (loaded) state
    await deferBlock.render(DeferBlockState.Complete);
    fixture.detectChanges();

    expect(fixture.nativeElement.querySelector('app-comments')).toBeTruthy();
  });

  it('should show error state on failure', async () => {
    const deferBlock = (await fixture.getDeferBlocks())[0];

    // Transition to error state
    await deferBlock.render(DeferBlockState.Error);
    fixture.detectChanges();

    expect(fixture.nativeElement.textContent).toContain('Failed to load');
  });
});
```

The `DeferBlockState` enum has four values:

| State | Description |
|---|---|
| `Placeholder` | The initial state before loading starts |
| `Loading` | The chunk is being fetched |
| `Complete` | The chunk loaded successfully and the content is rendered |
| `Error` | The chunk failed to load |

## Common Mistakes

- **Mistake 1: Deferring above-the-fold content.**
  Wrapping components that are visible on the initial viewport in `@defer` hurts Core Web Vitals. The user sees a placeholder flash or layout shift as the deferred content loads. Only defer content below the fold or behind user interaction.

- **Mistake 2: Using `on viewport` without a placeholder that has actual dimensions.**
  The viewport trigger observes the placeholder element. If the placeholder has zero height (or no placeholder is provided), the observer may fire immediately because a zero-height element is technically "in the viewport." Always give your placeholder explicit dimensions that approximate the final content size.

- **Mistake 3: Referencing the deferred component outside the `@defer` block.**
  Using the same component both inside and outside `@defer` in the same template, or referencing it via `viewChild()`, creates a static import that pulls the component into the main bundle. The `@defer` block still renders correctly, but you get zero code-splitting benefit.

- **Mistake 4: Using non-standalone components inside `@defer`.**
  The compiler silently falls back to eager loading for non-standalone components. Your template works, but the component is bundled upfront. Check the build output to verify chunks are actually being created.

- **Mistake 5: Creating cascading nested `@defer` blocks with immediate triggers.**
  Nesting `@defer (on immediate)` inside another `@defer (on immediate)` creates a request waterfall. The inner chunk cannot start loading until the outer chunk finishes. Use different triggers for nested blocks or flatten the structure.

- **Mistake 6: Exporting shared utilities from the same file as the deferred component.**
  If any eager code imports anything from the same file as the deferred component, the entire file is included in the main bundle. Keep deferred components in their own files with no additional exports.

---

## Best Practices

- **Use `prefetch on idle` for below-the-fold content.** Combine `on viewport` with `prefetch on idle` so the chunk downloads during idle time and renders instantly when the user scrolls to it. This is the ideal pattern for most lazy-loaded sections.

- **Match placeholder dimensions to content dimensions.** Prevent Cumulative Layout Shift (CLS) by making the placeholder the same approximate height as the deferred content. Use skeleton screens or fixed-height containers.

- **Always provide `@placeholder` and `@error` blocks.** The placeholder gives users visual feedback about what will appear. The error block prevents a blank space if loading fails. These are not technically required but should be treated as mandatory in production.

- **Use `minimum` on `@loading` and `@placeholder`.** Without `minimum`, fast loads cause a brief flash of the loading or placeholder state. Setting `minimum 300ms` to `500ms` ensures smooth transitions.

- **Use `after` on `@loading` to skip spinners for fast loads.** `@loading (after 200ms)` prevents the spinner from appearing at all if the chunk loads in under 200ms. This avoids unnecessary visual noise.

- **Verify code splitting in the build output.** After adding `@defer`, inspect `ng build` output to confirm that new chunks were created. If you do not see additional chunks, one of the constraints above is being violated.

## Real-World Use Cases

### Blog Post with Deferred Comments Section

The comments section is heavy (includes a rich text editor, avatar loading, nested replies). Users who just want to read the article should not pay the cost of loading all that JavaScript.

```typescript
@Component({
  selector: 'app-blog-post',
  standalone: true,
  imports: [DatePipe],
  template: `
    <article class="post">
      <h1>{{ post().title }}</h1>
      <time>{{ post().publishedAt | date:'longDate' }}</time>
      <div class="content" [innerHTML]="post().htmlContent"></div>
    </article>

    <section class="comments-section">
      @defer (on viewport; prefetch on idle) {
        <app-comments
          [postId]="post().id"
          [commentCount]="post().commentCount"
        />
      } @placeholder (minimum 300ms) {
        <div class="comments-skeleton">
          <h3>Comments ({{ post().commentCount }})</h3>
          <div class="skeleton-block" style="height: 120px;"></div>
          <div class="skeleton-block" style="height: 120px;"></div>
        </div>
      } @loading (after 300ms; minimum 500ms) {
        <div class="comments-loading">
          <div class="spinner"></div>
          <p>Loading comments...</p>
        </div>
      } @error {
        <div class="comments-error">
          <p>Could not load comments. Please refresh the page.</p>
        </div>
      }
    </section>
  `
})
export class BlogPostComponent {
  post = input.required<BlogPost>();
}
```

### Admin Dashboard with Tab-Based Deferred Panels

An admin dashboard with multiple tabs. Only the active tab's content needs to load. Each tab is a heavy component with its own dependencies.

```typescript
@Component({
  selector: 'app-admin-dashboard',
  standalone: true,
  template: `
    <nav class="tabs">
      <button
        [class.active]="activeTab() === 'overview'"
        (click)="activeTab.set('overview')"
      >Overview</button>
      <button
        [class.active]="activeTab() === 'analytics'"
        (click)="activeTab.set('analytics')"
      >Analytics</button>
      <button
        [class.active]="activeTab() === 'users'"
        (click)="activeTab.set('users')"
      >Users</button>
      <button
        [class.active]="activeTab() === 'settings'"
        (click)="activeTab.set('settings')"
      >Settings</button>
    </nav>

    <div class="tab-content">
      <!-- Overview loads immediately (above the fold, always visible first) -->
      <app-overview-panel [class.hidden]="activeTab() !== 'overview'" />

      <!-- Analytics defers until user selects the tab -->
      @defer (when activeTab() === 'analytics'; prefetch on idle) {
        <app-analytics-panel [class.hidden]="activeTab() !== 'analytics'" />
      } @placeholder {
        @if (activeTab() === 'analytics') {
          <div class="panel-skeleton">Loading analytics...</div>
        }
      }

      <!-- Users panel -->
      @defer (when activeTab() === 'users'; prefetch on idle) {
        <app-users-panel [class.hidden]="activeTab() !== 'users'" />
      } @placeholder {
        @if (activeTab() === 'users') {
          <div class="panel-skeleton">Loading user management...</div>
        }
      }

      <!-- Settings panel -->
      @defer (when activeTab() === 'settings'; prefetch on idle) {
        <app-settings-panel [class.hidden]="activeTab() !== 'settings'" />
      } @placeholder {
        @if (activeTab() === 'settings') {
          <div class="panel-skeleton">Loading settings...</div>
        }
      }
    </div>
  `,
  styles: [`
    .hidden { display: none; }
    .tabs { display: flex; gap: 4px; border-bottom: 2px solid #e0e0e0; }
    .tabs button {
      padding: 8px 16px;
      border: none;
      background: none;
      cursor: pointer;
    }
    .tabs button.active {
      border-bottom: 2px solid #1976d2;
      color: #1976d2;
      font-weight: 600;
    }
  `]
})
export class AdminDashboardComponent {
  activeTab = signal<'overview' | 'analytics' | 'users' | 'settings'>('overview');
}
```

### E-Commerce Product Page with Hover-Triggered Preview

A product listing where hovering over a product card prefetches a detailed preview, and clicking opens it instantly.

```typescript
@Component({
  selector: 'app-product-list',
  standalone: true,
  template: `
    <div class="product-grid">
      @for (product of products(); track product.id) {
        <div class="product-card">
          <img [src]="product.thumbnail" [alt]="product.name" />
          <h3>{{ product.name }}</h3>
          <p class="price">{{ product.price | currency }}</p>

          <div #previewTrigger class="preview-trigger">
            @defer (on interaction(previewTrigger); prefetch on hover(previewTrigger)) {
              <app-product-detail-preview [productId]="product.id" />
            } @placeholder {
              <button class="preview-btn">Quick View</button>
            } @loading (after 100ms) {
              <div class="preview-loading">
                <div class="mini-spinner"></div>
              </div>
            } @error {
              <p class="preview-error">Preview unavailable</p>
            }
          </div>
        </div>
      }
    </div>
  `
})
export class ProductListComponent {
  products = input.required<Product[]>();
}
```

In this pattern:
1. The user sees a grid of product cards with a "Quick View" button.
2. On hover, Angular prefetches the `ProductDetailPreviewComponent` chunk.
3. On click, the preview renders instantly because the code is already cached.
4. If the user never hovers or clicks, no extra JavaScript is loaded.

---

> **Key Takeaway:** `@defer` brings template-level lazy loading to Angular. Use it with `on viewport` and `prefetch on idle` for below-the-fold content, keep deferred components standalone and isolated in their own files, and always provide `@placeholder` and `@error` blocks for a polished user experience.
