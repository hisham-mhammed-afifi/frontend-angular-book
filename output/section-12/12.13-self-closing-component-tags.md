# 12.13 Self-Closing Component Tags

> Angular lets you write `<app-widget />` instead of `<app-widget></app-widget>` for components with no projected content. This shorthand reduces template noise and makes it immediately clear that an element is empty.

---

## The Syntax

Since Angular 15.1, the template compiler accepts self-closing tag syntax for components, directives, and structural elements. The trailing slash `/` tells the compiler that the element has no children.

```html
<!-- Traditional syntax -->
<app-icon name="settings"></app-icon>
<app-divider></app-divider>
<app-spinner></app-spinner>
<app-avatar [user]="currentUser()"></app-avatar>

<!-- Self-closing syntax -->
<app-icon name="settings" />
<app-divider />
<app-spinner />
<app-avatar [user]="currentUser()" />
```

Both forms produce identical output. The Angular compiler transforms self-closing tags into their expanded form during compilation. There is zero runtime difference.

---

## Why It Matters

### Reduced Visual Noise

In a component with many child components that take no projected content, the closing tags add significant clutter:

```html
<!-- Without self-closing tags: 20 lines -->
<div class="dashboard">
  <app-header></app-header>
  <app-breadcrumb [items]="breadcrumbs()"></app-breadcrumb>
  <div class="metrics">
    <app-metric-card [value]="revenue()" label="Revenue"></app-metric-card>
    <app-metric-card [value]="users()" label="Users"></app-metric-card>
    <app-metric-card [value]="orders()" label="Orders"></app-metric-card>
    <app-metric-card [value]="conversion()" label="Conversion"></app-metric-card>
  </div>
  <app-chart [data]="chartData()" type="line"></app-chart>
  <app-data-table [rows]="tableRows()" [columns]="columns()"></app-data-table>
  <app-footer></app-footer>
</div>

<!-- With self-closing tags: same 20 lines but less horizontal noise -->
<div class="dashboard">
  <app-header />
  <app-breadcrumb [items]="breadcrumbs()" />
  <div class="metrics">
    <app-metric-card [value]="revenue()" label="Revenue" />
    <app-metric-card [value]="users()" label="Users" />
    <app-metric-card [value]="orders()" label="Orders" />
    <app-metric-card [value]="conversion()" label="Conversion" />
  </div>
  <app-chart [data]="chartData()" type="line" />
  <app-data-table [rows]="tableRows()" [columns]="columns()" />
  <app-footer />
</div>
```

The self-closing version is easier to scan because your eye does not need to match opening and closing tags for elements that have no children.

### Intent Communication

Self-closing syntax signals to anyone reading the template: "This element has no content." When you see `<app-card>...</app-card>`, you expect content between the tags. When you see `<app-icon name="check" />`, you know instantly there is nothing inside.

### Consistency with Other Frameworks

React, Vue, Svelte, and Solid all support self-closing tags for components. Developers coming from those ecosystems expect this syntax. It also makes Angular templates feel more modern and less verbose.

---

## Where Self-Closing Tags Work

### Angular Components

Any standalone or module-based component can use self-closing syntax when no content is projected:

```html
<app-logo />
<app-search-bar [placeholder]="'Search...'" />
<app-notification-bell [count]="unreadCount()" />
```

### Directives on Elements

Elements with directives can also be self-closed, as long as the element itself supports it:

```html
<!-- Custom component with a directive -->
<app-tooltip message="Click to save" appHighlight />
```

### `ng-container` and `ng-template`

These Angular structural elements support self-closing when they have no content:

```html
<!-- ng-container with a directive but no children -->
<ng-container *ngTemplateOutlet="headerTpl" />

<!-- ng-template self-closed (unusual but valid) -->
<ng-template #empty />
```

However, `ng-template` is almost always used with content inside, so self-closing it is rare.

### SVG and Foreign Elements

Self-closing syntax works for SVG elements within Angular templates:

```html
<svg width="24" height="24">
  <circle cx="12" cy="12" r="10" fill="currentColor" />
  <line x1="4" y1="12" x2="20" y2="12" stroke="white" />
</svg>
```

SVG elements have always supported self-closing syntax in HTML because they are "foreign" elements (parsed in the SVG namespace, not the HTML namespace).

## Where Self-Closing Tags Do NOT Work

### Standard HTML Elements

The HTML spec does not support self-closing syntax for non-void elements. The browser's HTML parser ignores the trailing slash and treats the element as open.

```html
<!-- BAD PRACTICE: Do not self-close standard HTML elements -->
<div />           <!-- Browser sees: <div> (open, never closed) -->
<span />          <!-- Browser sees: <span> (open, never closed) -->
<p />             <!-- Browser sees: <p> (open, never closed) -->
<button />        <!-- Browser sees: <button> (open, never closed) -->
<ul />            <!-- Browser sees: <ul> (open, never closed) -->
<textarea />      <!-- Browser sees: <textarea> (open, never closed) -->
<script />        <!-- Browser sees: <script> (open, eats everything after it) -->
```

Angular compiles templates before they reach the browser, so `<div />` in an Angular template technically works at runtime. However, it is misleading and inconsistent with HTML semantics. The angular-eslint rule `prefer-self-closing-tags` intentionally skips native HTML elements.

### HTML Void Elements

Void elements like `<br>`, `<hr>`, `<img>`, `<input>` cannot have children by definition. They do not need a closing tag at all.

```html
<!-- These are void elements. The trailing slash is optional and cosmetic. -->
<br>
<br />    <!-- Both are valid HTML -->
<hr>
<hr />    <!-- Both are valid HTML -->
<img src="photo.jpg" alt="Photo">
<img src="photo.jpg" alt="Photo" />   <!-- Both are valid HTML -->
<input type="text" name="email">
<input type="text" name="email" />     <!-- Both are valid HTML -->
```

The complete list of HTML void elements:

| Element | Purpose |
|---|---|
| `<area>` | Image map area |
| `<base>` | Document base URL |
| `<br>` | Line break |
| `<col>` | Table column |
| `<embed>` | External content |
| `<hr>` | Thematic break |
| `<img>` | Image |
| `<input>` | Form input |
| `<link>` | External resource link |
| `<meta>` | Metadata |
| `<param>` | Object parameter (deprecated) |
| `<source>` | Media source |
| `<track>` | Text track |
| `<wbr>` | Word break opportunity |

For void elements, the slash is purely cosmetic. The HTML parser ignores it either way. Whether you write `<br>` or `<br />` is a team convention, not a functional choice.

### Components That Accept Content Projection

You technically can self-close a component that has `<ng-content>` in its template. The component just receives no projected content. But this can be confusing if the component was designed to always receive children.

```typescript
@Component({
  selector: 'app-card',
  standalone: true,
  template: `
    <div class="card">
      <div class="card-header">{{ title() }}</div>
      <div class="card-body">
        <ng-content />
      </div>
    </div>
  `
})
export class CardComponent {
  title = input<string>('');
}
```

```html
<!-- This is valid but the card body will be empty -->
<app-card title="Empty Card" />

<!-- This is what you usually want -->
<app-card title="User Info">
  <p>Some content here</p>
</app-card>
```

The self-closing version renders a card with an empty body. This is not a bug, but it may indicate a mistake if the card was meant to always have content.

## Migration Schematic

Angular provides an official migration schematic to convert existing templates to use self-closing tags:

```bash
ng generate @angular/core:self-closing-tags
```

This schematic:
1. Scans all component templates in the project.
2. Finds elements that have no children and no projected content.
3. Converts them from `<tag></tag>` to `<tag />`.
4. Skips native HTML elements (only processes Angular components and structural elements).

You can also use the ngxtension community migration:

```bash
npx ng generate ngxtension:convert-self-closing-tags
```

Both schematics only transform components and custom elements, not standard HTML elements.

### Running It Safely

```bash
# Preview changes without applying them
ng generate @angular/core:self-closing-tags --dry-run

# Apply changes
ng generate @angular/core:self-closing-tags

# Check that everything still compiles
ng build
```

## ESLint Rule: `prefer-self-closing-tags`

The `@angular-eslint/template/prefer-self-closing-tags` rule enforces consistent use of self-closing tags across your project. It was added in angular-eslint v16.2.

### Configuration

```json
{
  "rules": {
    "@angular-eslint/template/prefer-self-closing-tags": "error"
  }
}
```

Or in the flat config format:

```typescript
// eslint.config.js
import angular from '@angular-eslint/eslint-plugin-template';

export default [
  {
    files: ['**/*.html'],
    plugins: { '@angular-eslint/template': angular },
    rules: {
      '@angular-eslint/template/prefer-self-closing-tags': 'error'
    }
  }
];
```

### What It Does

The rule flags any component or structural element that:
1. Has no children (no text content, no child elements).
2. Is not a native HTML element.
3. Is not already self-closed.

```html
<!-- FLAGGED: empty component should be self-closed -->
<app-spinner></app-spinner>

<!-- OK: already self-closed -->
<app-spinner />

<!-- OK: has content, cannot be self-closed -->
<app-card>
  <p>Content</p>
</app-card>

<!-- NOT FLAGGED: native HTML element (skipped by the rule) -->
<div></div>
```

### Auto-Fix

The rule supports `--fix`, so you can automatically convert your entire project:

```bash
npx eslint --fix "src/**/*.html"
```

This is the easiest way to adopt self-closing tags across a large codebase.

## Angular Template Compiler Behavior

Understanding how the compiler handles self-closing tags helps dispel confusion.

### Compilation Step

Angular templates are parsed by the Angular compiler, not by the browser's HTML parser. The compiler:

1. Reads the template source (string or file).
2. Parses it into an Abstract Syntax Tree (AST).
3. Recognizes `<tag />` as an element with no children.
4. Generates the same instructions as `<tag></tag>`.

The output JavaScript (the component's template function) is identical for both forms.

### Browser Never Sees the Self-Closing Syntax

Because Angular compiles templates ahead of time (AOT) or during build, the browser's HTML parser never processes your template source directly. This is why `<div />` technically works in Angular templates even though it is invalid in HTML. The compiler handles it before the browser is involved.

However, just because it works does not mean you should do it. The convention is clear: self-close Angular components and structural elements, not native HTML elements.

## Self-Closing Tags with Control Flow

Self-closing components work naturally inside Angular's control flow blocks:

```html
@if (isLoading()) {
  <app-spinner />
} @else if (hasError()) {
  <app-error-message [error]="error()" />
} @else {
  <app-data-table [rows]="data()" />
}

@for (user of users(); track user.id) {
  <app-user-card [user]="user" />
}

@defer (on viewport) {
  <app-heavy-chart [data]="chartData()" />
} @placeholder {
  <app-chart-skeleton />
}
```

This reads much cleaner than the non-self-closed equivalent, especially inside `@for` loops where repetition amplifies the noise.

## Formatting and Style Conventions

### Single-Line vs Multi-Line

For components with few or no bindings, keep it on one line:

```html
<app-divider />
<app-spinner />
<app-icon name="check" />
```

For components with multiple bindings, break across lines with the self-closing slash on the last attribute line or its own line:

```html
<!-- Slash on the last attribute line -->
<app-metric-card
  [value]="revenue()"
  label="Revenue"
  icon="dollar"
  trend="up" />

<!-- Slash on its own line (Prettier default for Angular) -->
<app-metric-card
  [value]="revenue()"
  label="Revenue"
  icon="dollar"
  trend="up"
/>
```

Both are acceptable. Pick one convention and enforce it with your formatter (Prettier or similar). Prettier for Angular HTML uses the second style by default when the tag breaks across multiple lines.

### Whitespace Before the Slash

The space before `/>` is optional but conventional:

```html
<app-icon name="check" />   <!-- space before slash: conventional -->
<app-icon name="check"/>    <!-- no space: also valid -->
```

Most formatters add the space. Stick with your formatter's output.

## Common Mistakes

- **Mistake 1: Self-closing native HTML elements in Angular templates.**
  Writing `<div />` or `<span />` works in Angular because the compiler processes it before the browser, but it violates HTML semantics and confuses developers who expect standard HTML behavior. The ESLint rule intentionally skips native elements for this reason. Only self-close Angular components and structural elements.

- **Mistake 2: Self-closing a component that should have projected content.**
  If a component is designed to receive children via `<ng-content>`, self-closing it means the body is empty. This is valid but often unintentional. Review the component's API before self-closing it. If it has `<ng-content>` and you are deliberately providing no content, add a comment explaining why.

- **Mistake 3: Inconsistent usage across templates.**
  Mixing `<app-icon></app-icon>` and `<app-icon />` in the same project (or the same file) creates visual inconsistency. Adopt the ESLint rule or run the migration schematic to enforce one style everywhere.

- **Mistake 4: Confusing HTML void elements with self-closing tags.**
  `<br>`, `<img>`, `<input>` are void elements. They never had closing tags. Adding `/>` to them is cosmetic, not functional. Do not conflate this with Angular's self-closing component syntax, which is a compiler feature. The slash on `<app-widget />` tells Angular's parser to close the element; the slash on `<br />` is ignored by the browser's parser.

- **Mistake 5: Assuming self-closing syntax affects the DOM.**
  `<app-widget />` and `<app-widget></app-widget>` produce identical DOM output. There is no performance difference, no structural difference, and no behavioral difference. It is purely a source-level convenience.

---

## Best Practices

- **Enable the ESLint rule project-wide.** Add `@angular-eslint/template/prefer-self-closing-tags` to your lint configuration with auto-fix. This ensures consistency without relying on manual discipline.

- **Run the migration schematic when adopting the convention.** Use `ng generate @angular/core:self-closing-tags` to convert the entire codebase at once, then enable the lint rule to prevent regression.

- **Only self-close Angular components and structural elements.** Do not self-close native HTML elements like `<div>`, `<span>`, or `<button>`. This keeps templates aligned with HTML standards and avoids confusing developers.

- **Self-close components in `@for` loops for maximum readability.** Loops amplify template noise. Self-closing tags inside `@for` make the repeated structure easier to scan.

- **Use self-closing as a signal for "no content."** When reviewing templates, treat `<app-card />` as a deliberate statement that no content is projected. If this seems wrong for a given component, it probably is.

## Real-World Use Cases

### Icon System with Self-Closing Tags

An icon component is the most common candidate for self-closing syntax because icons never have projected content.

```typescript
@Component({
  selector: 'app-icon',
  standalone: true,
  template: `
    <svg
      [attr.width]="size()"
      [attr.height]="size()"
      [attr.viewBox]="'0 0 24 24'"
      [class]="'icon icon-' + name()"
    >
      <use [attr.href]="'#icon-' + name()" />
    </svg>
  `,
  styles: [`
    :host { display: inline-flex; align-items: center; }
    svg { fill: currentColor; }
  `]
})
export class IconComponent {
  name = input.required<string>();
  size = input<number>(24);
}
```

Usage throughout the app:

```html
<nav class="toolbar">
  <button class="nav-btn">
    <app-icon name="menu" />
  </button>
  <h1>Dashboard</h1>
  <div class="actions">
    <button class="action-btn">
      <app-icon name="search" size="20" />
    </button>
    <button class="action-btn">
      <app-icon name="notifications" size="20" />
    </button>
    <app-avatar [user]="currentUser()" />
  </div>
</nav>
```

Every `<app-icon ... />` is immediately recognizable as a self-contained element with no children.

### Form with Mixed Self-Closing and Content Components

A form that uses both self-closing (leaf) components and content-projecting (container) components:

```html
<form (ngSubmit)="onSubmit()">
  <app-form-section title="Personal Information">
    <app-text-field
      label="First Name"
      [control]="form.controls.firstName" />
    <app-text-field
      label="Last Name"
      [control]="form.controls.lastName" />
    <app-date-picker
      label="Date of Birth"
      [control]="form.controls.dob" />
  </app-form-section>

  <app-form-section title="Contact">
    <app-text-field
      label="Email"
      type="email"
      [control]="form.controls.email" />
    <app-phone-field
      label="Phone"
      [control]="form.controls.phone" />
  </app-form-section>

  <app-form-section title="Preferences">
    <app-checkbox
      label="Subscribe to newsletter"
      [control]="form.controls.newsletter" />
    <app-select
      label="Language"
      [options]="languages()"
      [control]="form.controls.language" />
  </app-form-section>

  <div class="form-actions">
    <app-button variant="outline" type="button" (clicked)="onCancel()">
      Cancel
    </app-button>
    <app-button variant="primary" type="submit">
      Save
    </app-button>
  </div>
</form>
```

Notice the pattern:
- `<app-text-field ... />`, `<app-date-picker ... />`, `<app-checkbox ... />`, `<app-select ... />` are self-closed because they are leaf components with no projected content.
- `<app-form-section>...</app-form-section>` and `<app-button>...</app-button>` use opening/closing tags because they project content (form fields and button labels respectively).

This visual distinction makes it easy to identify which components are containers and which are standalone widgets.

### Data Table with Declarative Column Definitions

A table component where column definitions are self-closing configuration components:

```html
<app-data-table [data]="employees()" [paginated]="true" [pageSize]="25">
  <app-column field="name" header="Full Name" sortable />
  <app-column field="email" header="Email" sortable />
  <app-column field="department" header="Department" filterable />
  <app-column field="startDate" header="Start Date" type="date" sortable />
  <app-column field="salary" header="Salary" type="currency" align="right" />
  <app-column field="status" header="Status" filterable />
</app-data-table>
```

Each `<app-column ... />` is a declarative configuration element. Self-closing syntax makes the column definitions read almost like a configuration file, with each line clearly defining one column. The closing `</app-column>` tags would add noise without communicating anything useful.

---

> **Key Takeaway:** Self-closing tags (`<app-widget />`) are a syntactic convenience that reduces template clutter for components with no projected content. Enable the `prefer-self-closing-tags` ESLint rule, run the migration schematic, and reserve self-closing syntax for Angular components only, not native HTML elements.
