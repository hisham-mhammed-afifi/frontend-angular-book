# 12.3 Control Flow

> Angular's built-in control flow blocks (`@if`, `@for`, `@switch`) replace the legacy structural directives with faster, type-safe, and more readable template syntax that compiles away at build time.

---

## The Shift from Directives to Built-in Syntax

Angular 17 introduced built-in control flow as a replacement for `*ngIf`, `*ngFor`, and `ngSwitch`. As of Angular 20, the old structural directives are officially deprecated. The new syntax is not a directive at all. It is processed by the Angular compiler at build time, which means:

- No directive class is instantiated at runtime
- No hidden `<ng-template>` wrappers are created
- Smaller bundle size and faster rendering
- Better type narrowing and type checking
- Syntax closer to JavaScript/TypeScript itself

You do not need to import anything into your component's `imports` array to use `@if`, `@for`, or `@switch`. They are built into the template compiler.

```typescript
// Nothing to import for control flow
@Component({
  selector: 'app-example',
  standalone: true,
  imports: [], // No CommonModule, no NgIf, no NgFor
  template: `
    @if (items().length > 0) {
      <ul>
        @for (item of items(); track item.id) {
          <li>{{ item.name }}</li>
        }
      </ul>
    } @else {
      <p>No items available.</p>
    }
  `
})
export class ExampleComponent {
  items = signal<Item[]>([]);
}
```

## @if - Conditional Rendering

The `@if` block conditionally renders a section of the template based on a truthy expression.

### Basic Syntax

```html
@if (condition) {
  <!-- rendered when condition is truthy -->
}
```

The expression inside the parentheses follows the same rules as JavaScript truthiness. Zero, empty string, `null`, `undefined`, `NaN`, and `false` are all falsy.

### @else and @else if

```html
@if (status() === 'loading') {
  <app-spinner />
} @else if (status() === 'error') {
  <p class="error">Failed to load data.</p>
} @else {
  <app-content [data]="data()" />
}
```

You can chain as many `@else if` blocks as needed. The `@else` block is optional. Only one branch renders at a time.

### Variable Aliasing with `as`

The `as` keyword stores the result of the `@if` expression in a local template variable. This is especially useful when the expression involves a function call or a signal you do not want to evaluate multiple times.

```html
@if (currentUser(); as user) {
  <h2>Welcome, {{ user.name }}</h2>
  <p>Email: {{ user.email }}</p>
  <p>Role: {{ user.role }}</p>
}
```

Without `as`, you would either call `currentUser()` multiple times or declare a `@let` variable. The `as` syntax keeps it concise and evaluates the expression exactly once.

**Important**: The `as` alias is only available in the first `@if` condition. You cannot use `as` in `@else if` blocks.

```html
<!-- This does NOT work -->
@if (conditionA) {
  ...
} @else if (getUser(); as user) {  <!-- ERROR: as not supported here -->
  ...
}
```

**Workaround**: Use `@let` before the `@if` block or nest an `@if` inside the `@else` block.

```html
<!-- Option 1: @let before @if -->
@let user = getUser();
@if (conditionA) {
  ...
} @else if (user) {
  <p>{{ user.name }}</p>
}

<!-- Option 2: Nested @if in @else -->
@if (conditionA) {
  ...
} @else {
  @if (getUser(); as user) {
    <p>{{ user.name }}</p>
  }
}
```

### Type Narrowing

The `@if` block provides automatic type narrowing, similar to TypeScript's control flow analysis.

```typescript
interface AdminUser {
  name: string;
  permissions: string[];
}

@Component({
  template: `
    @if (isAdmin()) {
      <!-- TypeScript narrows the type inside this block -->
      <p>Admin: {{ adminUser().name }}</p>
    }
  `
})
```

However, there are known limitations with optional chaining in templates:

```html
<!-- TypeScript narrows this correctly in .ts files, but templates have limitations -->
@if (user()?.address) {
  <!-- user() might still be considered possibly null here in some cases -->
  <p>{{ user()!.address!.city }}</p>  <!-- may need non-null assertion -->
}
```

**Better approach**: Use `as` or `@let` to store the resolved value.

```html
@if (user()?.address; as address) {
  <p>{{ address.city }}</p>  <!-- address is narrowed to non-null -->
}
```

### Comparing @if to *ngIf

| Feature | `@if` | `*ngIf` |
|---|---|---|
| Import required | No | Yes (`CommonModule` or `NgIf`) |
| Runtime overhead | None (compiled away) | Directive instantiation |
| `else` support | `@else if` chains | Only `else` with `ng-template` ref |
| Type narrowing | Better | Limited |
| Variable alias | `as` keyword | `as` keyword |
| Status | Current | Deprecated in Angular 20 |

The `*ngIf` `else` syntax was notoriously awkward:

```html
<!-- Legacy: required a separate ng-template -->
<div *ngIf="isLoaded; else loading">
  Content here
</div>
<ng-template #loading>
  <app-spinner></app-spinner>
</ng-template>
```

```html
<!-- Modern: inline and readable -->
@if (isLoaded()) {
  <div>Content here</div>
} @else {
  <app-spinner />
}
```

## @for - Iterating Over Collections

The `@for` block renders a template for each item in an iterable collection.

### Basic Syntax

```html
@for (item of items; track item.id) {
  <div>{{ item.name }}</div>
}
```

Three parts are required:
1. **Loop variable**: `item of items` declares the iteration variable and source collection
2. **Track expression**: `track item.id` tells Angular how to identify each element (mandatory)
3. **Template body**: The content inside `{ }` is rendered for each item

### The Track Expression (Mandatory)

Unlike the old `*ngFor` where `trackBy` was optional, the `track` expression is required for `@for`. This is a deliberate design decision to prevent performance problems by default.

The track expression tells Angular how to map data items to DOM nodes. When the collection changes, Angular uses this mapping to determine which DOM elements to create, destroy, or move, instead of destroying and recreating the entire list.

```html
<!-- Track by a unique property (preferred for dynamic lists) -->
@for (user of users(); track user.id) {
  <app-user-card [user]="user" />
}

<!-- Track by index (acceptable for static lists) -->
@for (tab of tabs; track $index) {
  <button>{{ tab.label }}</button>
}
```

#### What to use for track

| Scenario | Track Expression | Why |
|---|---|---|
| Items have unique IDs | `track item.id` | Optimal. Preserves DOM on reorder/update. |
| Items are primitives | `track $index` | No unique property available. |
| Static list (never changes) | `track $index` | Cheapest comparison. |
| Items from API with UUIDs | `track item.uuid` | Stable identifier across fetches. |
| Items added/removed frequently | `track item.id` | Avoids destroying/recreating all DOM. |
| Composite key needed | `track item.type + '-' + item.id` | When single field is not unique. |

#### Track expression rules

The track expression can reference:
- The loop variable (`item`, `user`, etc.)
- `$index`
- Properties of the component class (fields, methods)

The track expression cannot:
- Call functions with side effects
- Use complex logic (keep it a pure expression)

```typescript
// Track by a component method is allowed but rarely needed
@for (item of items(); track getId(item)) {
  <div>{{ item.name }}</div>
}
```

#### Performance impact of incorrect tracking

```html
<!-- BAD: Tracking by object reference with new objects from API -->
<!-- If the API returns new object references each time, Angular recreates all DOM -->
@for (item of items(); track item) {
  <app-card [data]="item" />
}

<!-- GOOD: Track by stable ID -->
@for (item of items(); track item.id) {
  <app-card [data]="item" />
}
```

When Angular receives a new array from an API, every object is a new reference even if the data is identical. Tracking by object reference causes Angular to destroy and recreate every DOM element. Tracking by a stable ID like `item.id` allows Angular to reuse existing DOM elements, preserving scroll position, CSS animations, and user input state (like text in input fields).

### Implicit Context Variables

The `@for` block provides six implicit variables accessible within the loop body:

| Variable | Type | Description |
|---|---|---|
| `$index` | `number` | Zero-based index of the current item |
| `$first` | `boolean` | `true` for the first item |
| `$last` | `boolean` | `true` for the last item |
| `$even` | `boolean` | `true` if `$index` is even |
| `$odd` | `boolean` | `true` if `$index` is odd |
| `$count` | `number` | Total number of items in the collection |

```html
@for (user of users(); track user.id; let i = $index, let isLast = $last) {
  <div class="user-row" [class.striped]="$odd">
    <span>{{ i + 1 }} of {{ $count }}</span>
    <span>{{ user.name }}</span>
    @if (!isLast) {
      <hr />
    }
  </div>
}
```

#### Aliasing implicit variables

You can alias any implicit variable using `let`. This is essential when nesting `@for` blocks, because inner loops shadow the outer loop's implicit variables.

```html
@for (category of categories(); track category.id; let catIndex = $index) {
  <h3>{{ catIndex + 1 }}. {{ category.name }}</h3>

  @for (product of category.products; track product.id) {
    <!-- $index here refers to the inner loop -->
    <!-- catIndex refers to the outer loop -->
    <p>{{ catIndex + 1 }}.{{ $index + 1 }} {{ product.name }}</p>
  }
}
```

Without aliasing, there is no way to access the outer `$index` from the inner loop. The inner `$index` shadows the outer one.

### @empty - Handling Empty Collections

The `@empty` block renders when the collection passed to `@for` has zero items.

```html
@for (notification of notifications(); track notification.id) {
  <app-notification-card [notification]="notification" />
} @empty {
  <div class="empty-state">
    <p>No notifications yet.</p>
  </div>
}
```

The `@empty` block must appear immediately after the closing `}` of the `@for` block. It cannot be separated by other content.

#### The async data gotcha with @empty

When data comes from an API, the collection starts as an empty array (or `undefined`). The `@empty` block will flash briefly before data arrives.

```typescript
@Component({
  template: `
    <!-- BAD: @empty flashes during loading -->
    @for (item of items(); track item.id) {
      <app-item [item]="item" />
    } @empty {
      <p>No items found.</p>
    }
  `
})
export class ItemListComponent {
  items = signal<Item[]>([]); // Empty initially, filled after API call
}
```

**Solution**: Guard with a loading state so `@empty` only shows when data has actually loaded and the list is genuinely empty.

```typescript
@Component({
  template: `
    @if (loading()) {
      <app-spinner />
    } @else {
      @for (item of items(); track item.id) {
        <app-item [item]="item" />
      } @empty {
        <p>No items found.</p>
      }
    }
  `
})
export class ItemListComponent {
  private itemService = inject(ItemService);
  items = signal<Item[]>([]);
  loading = signal(true);

  constructor() {
    this.itemService.getItems().subscribe(data => {
      this.items.set(data);
      this.loading.set(false);
    });
  }
}
```

Alternatively, use the `resource` API which provides built-in status tracking:

```typescript
@Component({
  template: `
    @if (itemsResource.isLoading()) {
      <app-spinner />
    } @else {
      @for (item of itemsResource.value() ?? []; track item.id) {
        <app-item [item]="item" />
      } @empty {
        <p>No items found.</p>
      }
    }
  `
})
export class ItemListComponent {
  itemsResource = httpResource<Item[]>('/api/items');
}
```

### Comparing @for to *ngFor

| Feature | `@for` | `*ngFor` |
|---|---|---|
| Import required | No | Yes (`CommonModule` or `NgForOf`) |
| Track function | Required (`track`) | Optional (`trackBy`) |
| Empty state | Built-in `@empty` block | Manual `*ngIf` check needed |
| Implicit variables | `$index`, `$first`, `$last`, `$even`, `$odd`, `$count` | `index`, `first`, `last`, `even`, `odd` (no count) |
| Aliasing | `let i = $index` | `let i = index` |
| Runtime overhead | None (compiled away) | Directive instantiation |
| Status | Current | Deprecated in Angular 20 |

Note the addition of `$count` which was not available in `*ngFor`. Also note the `$` prefix convention that distinguishes implicit variables from the loop variable.

## @switch - Multi-branch Matching

The `@switch` block renders content based on which `@case` matches the expression, using strict equality (`===`).

### Basic Syntax

```html
@switch (status()) {
  @case ('pending') {
    <app-pending-badge />
  }
  @case ('active') {
    <app-active-badge />
  }
  @case ('suspended') {
    <app-suspended-badge />
  }
  @default {
    <app-unknown-badge />
  }
}
```

### Key Characteristics

1. **Strict equality**: Cases are matched with `===`. Type coercion does not happen.
2. **No fallthrough**: Unlike JavaScript's `switch`, there is no `break` statement and no fallthrough between cases. When a case matches, only that case's content renders.
3. **Optional @default**: The `@default` block renders when no `@case` matches. It is optional.
4. **No duplicate @default**: Only one `@default` block is allowed.

### Matching with Enums and Constants

```typescript
enum UserRole {
  Admin = 'ADMIN',
  Editor = 'EDITOR',
  Viewer = 'VIEWER',
}

@Component({
  template: `
    @switch (role()) {
      @case (UserRole.Admin) {
        <app-admin-dashboard />
      }
      @case (UserRole.Editor) {
        <app-editor-panel />
      }
      @case (UserRole.Viewer) {
        <app-viewer-panel />
      }
      @default {
        <p>Unknown role. Contact support.</p>
      }
    }
  `
})
export class DashboardComponent {
  role = input.required<UserRole>();
  protected readonly UserRole = UserRole; // Expose enum to template
}
```

You must expose the enum to the template by assigning it to a component property. Templates cannot directly access TypeScript enums from external imports.

### Comparing @switch to ngSwitch

| Feature | `@switch` | `ngSwitch` |
|---|---|---|
| Import required | No | Yes (`CommonModule` or individually) |
| Syntax | Single `@switch` block | Three directives: `[ngSwitch]`, `*ngSwitchCase`, `*ngSwitchDefault` |
| Fallthrough | No | No |
| Comparison | `===` | `===` |
| Status | Current | Deprecated in Angular 20 |

The legacy `ngSwitch` required three separate directives on different elements:

```html
<!-- Legacy: verbose and split across elements -->
<div [ngSwitch]="status">
  <app-active *ngSwitchCase="'active'"></app-active>
  <app-pending *ngSwitchCase="'pending'"></app-pending>
  <app-unknown *ngSwitchDefault></app-unknown>
</div>
```

```html
<!-- Modern: self-contained and readable -->
@switch (status()) {
  @case ('active') { <app-active /> }
  @case ('pending') { <app-pending /> }
  @default { <app-unknown /> }
}
```

## Nesting Control Flow Blocks

Control flow blocks can be freely nested. This is where the readability advantage over legacy directives becomes significant.

```html
@if (categories().length > 0) {
  @for (category of categories(); track category.id) {
    <section>
      <h2>{{ category.name }}</h2>

      @if (category.products.length > 0) {
        <ul>
          @for (product of category.products; track product.id; let pi = $index) {
            <li [class.featured]="$first">
              @switch (product.status) {
                @case ('in-stock') {
                  <span class="badge green">In Stock</span>
                }
                @case ('low-stock') {
                  <span class="badge yellow">Low Stock ({{ product.quantity }})</span>
                }
                @case ('out-of-stock') {
                  <span class="badge red">Out of Stock</span>
                }
              }
              {{ product.name }} - {{ product.price | currency }}
            </li>
          }
        </ul>
      } @else {
        <p>No products in this category.</p>
      }
    </section>
  } @empty {
    <p>No categories available.</p>
  }
} @else {
  <app-empty-catalog />
}
```

This would have been far more complex with the legacy directive syntax, requiring multiple `ng-template` references and `ng-container` wrappers.

## Control Flow with Signals

Control flow blocks work seamlessly with signals. Because signals are functions, you call them with `()` inside the expressions.

```typescript
@Component({
  template: `
    @if (isAuthenticated()) {
      @switch (userRole()) {
        @case ('admin') {
          <app-admin-nav />
        }
        @case ('user') {
          <app-user-nav />
        }
      }

      @for (item of menuItems(); track item.id) {
        <app-menu-item [item]="item" />
      } @empty {
        <p>No menu items configured.</p>
      }
    } @else {
      <app-login-form />
    }
  `
})
export class AppShellComponent {
  private authService = inject(AuthService);

  isAuthenticated = this.authService.isAuthenticated;
  userRole = this.authService.userRole;
  menuItems = computed(() => {
    const role = this.userRole();
    return this.getMenuItemsForRole(role);
  });

  private getMenuItemsForRole(role: string): MenuItem[] {
    // return role-specific menu items
  }
}
```

When any signal read inside a control flow expression changes, Angular re-evaluates only that block. Combined with `OnPush` change detection (or zoneless), this is highly efficient.

## Control Flow and @let

The `@let` declaration (Angular 18.1+) pairs naturally with control flow. Declare template-local variables to simplify complex expressions.

```html
@let total = cartItems().reduce((sum, item) => sum + item.price * item.qty, 0);
@let hasDiscount = total > 100;
@let finalTotal = hasDiscount ? total * 0.9 : total;

@if (cartItems().length > 0) {
  <table>
    <thead>
      <tr>
        <th>Product</th>
        <th>Qty</th>
        <th>Price</th>
      </tr>
    </thead>
    <tbody>
      @for (item of cartItems(); track item.id) {
        <tr>
          <td>{{ item.name }}</td>
          <td>{{ item.qty }}</td>
          <td>{{ item.price * item.qty | currency }}</td>
        </tr>
      }
    </tbody>
  </table>

  @if (hasDiscount) {
    <p>10% discount applied!</p>
  }
  <p class="total">Total: {{ finalTotal | currency }}</p>
} @else {
  <p>Your cart is empty.</p>
}
```

`@let` variables are read-only and scoped to the current template and its descendants. They are re-evaluated whenever their dependencies change.

## Migration from Legacy Directives

### Automatic Migration Schematic

Angular provides a migration schematic that automatically converts legacy directive syntax to the new control flow:

```bash
ng generate @angular/core:control-flow-migration
```

Options:
- `--path`: Target a specific directory or file
- `--format`: Apply formatting after migration

The schematic handles most conversions automatically, including:
- `*ngIf` with `else` template references to `@if` / `@else`
- `*ngFor` with `trackBy` to `@for` with `track`
- `[ngSwitch]` with `*ngSwitchCase` and `*ngSwitchDefault` to `@switch`

### Manual migration patterns

**ngIf with else template:**

```html
<!-- Before -->
<div *ngIf="isLoaded; else loadingTpl">
  {{ data }}
</div>
<ng-template #loadingTpl>
  <app-spinner></app-spinner>
</ng-template>

<!-- After -->
@if (isLoaded()) {
  <div>{{ data() }}</div>
} @else {
  <app-spinner />
}
```

**ngIf with then/else:**

```html
<!-- Before -->
<ng-container *ngIf="condition; then thenTpl; else elseTpl"></ng-container>
<ng-template #thenTpl>Then content</ng-template>
<ng-template #elseTpl>Else content</ng-template>

<!-- After -->
@if (condition()) {
  Then content
} @else {
  Else content
}
```

**ngFor with trackBy:**

```html
<!-- Before -->
<div *ngFor="let item of items; trackBy: trackById; let i = index; let isEven = even">
  {{ i }}: {{ item.name }} {{ isEven ? '(even)' : '' }}
</div>

<!-- After -->
@for (item of items(); track item.id; let i = $index) {
  <div>
    {{ i }}: {{ item.name }} {{ $even ? '(even)' : '' }}
  </div>
}
```

Notice that `trackBy` was a method reference in the old syntax, but `track` is an inline expression in the new syntax.

**ngSwitch:**

```html
<!-- Before -->
<div [ngSwitch]="color">
  <span *ngSwitchCase="'red'" class="red">Red</span>
  <span *ngSwitchCase="'blue'" class="blue">Blue</span>
  <span *ngSwitchDefault>Other</span>
</div>

<!-- After -->
@switch (color()) {
  @case ('red') { <span class="red">Red</span> }
  @case ('blue') { <span class="blue">Blue</span> }
  @default { <span>Other</span> }
}
```

### What the schematic does not handle

- Converting observable-based code to signals (you still need to manually migrate to signal-based patterns)
- Removing `CommonModule` imports if it is only used for directives (you need to check if pipes like `DatePipe` are still needed)
- Complex `*ngIf` patterns with multiple template references that have unusual nesting

## Common Mistakes

### 1. Forgetting the track expression in @for

```html
<!-- Compilation error: track is required -->
@for (item of items()) {
  <div>{{ item.name }}</div>
}

<!-- Fix: always provide track -->
@for (item of items(); track item.id) {
  <div>{{ item.name }}</div>
}
```

This is a compiler error, not a warning. Your template will not compile without `track`.

### 2. Using $index as track for dynamic lists

```html
<!-- BAD: For lists that change (add, remove, reorder), $index causes full re-renders -->
@for (item of searchResults(); track $index) {
  <app-result-card [result]="item" />
}

<!-- GOOD: Use a stable unique identifier -->
@for (item of searchResults(); track item.id) {
  <app-result-card [result]="item" />
}
```

When you add an item at the beginning of a list tracked by `$index`, every item's index shifts, causing Angular to update every DOM node. Use `$index` only for truly static lists like navigation tabs or dropdown options that never change.

### 3. Using two separate @if blocks instead of @if/@else

```html
<!-- BAD: Two separate conditions that could conflict -->
@if (isLoggedIn()) {
  <app-dashboard />
}
@if (!isLoggedIn()) {
  <app-login />
}

<!-- GOOD: Use @else for mutually exclusive conditions -->
@if (isLoggedIn()) {
  <app-dashboard />
} @else {
  <app-login />
}
```

Two separate `@if` blocks are evaluated independently. If `isLoggedIn()` changes between evaluations, you could briefly see both or neither. `@else` guarantees mutual exclusion.

### 4. Not handling the @empty flash with async data

See the async data gotcha section above. Always guard `@empty` with a loading check when data comes from an API.

### 5. Trying to use break or continue in @for

```html
<!-- This does NOT work. @for does not support flow control statements. -->
@for (item of items(); track item.id) {
  @if (item.hidden) {
    <!-- There is no "continue" or "skip" -->
  }
  <div>{{ item.name }}</div>
}

<!-- Solution: Filter the collection before iterating -->
```

```typescript
// Filter in a computed signal
filteredItems = computed(() => this.items().filter(item => !item.hidden));
```

```html
@for (item of filteredItems(); track item.id) {
  <div>{{ item.name }}</div>
}
```

### 6. Importing CommonModule or NgIf/NgFor alongside control flow

```typescript
// BAD: Unnecessary import
@Component({
  imports: [CommonModule], // Not needed for @if, @for, @switch
  template: `
    @if (show()) {
      <p>Hello</p>
    }
  `
})
```

If your template only uses the built-in control flow and no pipes from `CommonModule` (like `DatePipe`, `CurrencyPipe`, `AsyncPipe`), remove the import entirely.

## Best Practices

### 1. Always use a stable unique ID for track in dynamic lists

Use `item.id`, `item.uuid`, or any property that remains the same for a given item across data refreshes. Reserve `$index` for static, never-changing lists.

### 2. Use @else instead of opposite @if conditions

When two states are mutually exclusive, always use `@if / @else` or `@if / @else if / @else`. Separate `@if` blocks for opposite conditions are harder to read and can have subtle timing issues.

### 3. Pre-filter and pre-sort collections in computed signals

Do not put filtering or sorting logic in the template. Compute the filtered/sorted list in a `computed` signal and iterate over the result.

```typescript
// In the component
private allItems = signal<Item[]>([]);
searchTerm = signal('');

filteredItems = computed(() => {
  const term = this.searchTerm().toLowerCase();
  return this.allItems().filter(item =>
    item.name.toLowerCase().includes(term)
  );
});
```

```html
<!-- In the template -->
@for (item of filteredItems(); track item.id) {
  <app-item [item]="item" />
} @empty {
  <p>No items match "{{ searchTerm() }}".</p>
}
```

### 4. Keep control flow expressions simple

Avoid complex logic inside `@if` or `@switch` expressions. Extract complex conditions into computed signals or component methods.

```html
<!-- BAD: Complex inline logic -->
@if (user() && user()!.roles.includes('admin') && !user()!.suspended && features().adminPanel) {
  <app-admin-panel />
}

<!-- GOOD: Computed signal with descriptive name -->
@if (canAccessAdminPanel()) {
  <app-admin-panel />
}
```

```typescript
canAccessAdminPanel = computed(() => {
  const u = this.user();
  return u != null
    && u.roles.includes('admin')
    && !u.suspended
    && this.features().adminPanel;
});
```

### 5. Use @let to avoid repeated signal calls

```html
<!-- BAD: Calls user() multiple times -->
@if (user()) {
  <h2>{{ user()!.name }}</h2>
  <p>{{ user()!.email }}</p>
  <p>{{ user()!.phone }}</p>
}

<!-- GOOD: Use as or @let -->
@if (user(); as u) {
  <h2>{{ u.name }}</h2>
  <p>{{ u.email }}</p>
  <p>{{ u.phone }}</p>
}
```

### 6. Leverage @empty for declarative empty states

Instead of wrapping `@for` in an `@if` to check `length > 0`, use the `@empty` block. It is more declarative and keeps the empty state logic co-located with the list.

```html
<!-- Less ideal: separate @if check -->
@if (items().length > 0) {
  @for (item of items(); track item.id) {
    <app-item [item]="item" />
  }
} @else {
  <p>No items.</p>
}

<!-- Better: @empty block -->
@for (item of items(); track item.id) {
  <app-item [item]="item" />
} @empty {
  <p>No items.</p>
}
```

interface Employee {

@Component({
  selector: 'app-employee-table',

  imports: [FormsModule],
  template: `
    <div class="controls">
      <input
        type="text"
        placeholder="Search employees..."
        [ngModel]="searchTerm()"
        (ngModelChange)="searchTerm.set($event)"
      />
      <select [ngModel]="departmentFilter()" (ngModelChange)="departmentFilter.set($event)">
        <option value="">All Departments</option>
        @for (dept of departments(); track dept) {
          <option [value]="dept">{{ dept }}</option>

          <th>#</th>

        @for (emp of filteredEmployees(); track emp.id; let i = $index) {
          <tr [class.striped]="$odd">
            <td>{{ i + 1 }}</td>
            <td>{{ emp.name }}</td>
            <td>{{ emp.department }}</td>
            <td>
              @switch (emp.status) {
                @case ('active') {
                  <span class="badge active">Active</span>
                }
                @case ('on-leave') {
                  <span class="badge leave">On Leave</span>
                }
                @case ('terminated') {
                  <span class="badge terminated">Terminated</span>
                }
              }
            </td>
          </tr>
        } @empty {
          <tr>
            <td colspan="4" class="empty">
              No employees match your criteria.
            </td>
          </tr>
        }
      </tbody>
    </table>

    <p class="count">Showing {{ filteredEmployees().length }} of {{ $count }} employees</p>
  `
})
export class EmployeeTableComponent {
  private employeeService = inject(EmployeeService);

  allEmployees = signal<Employee[]>([]);
  searchTerm = signal('');
  departmentFilter = signal('');

  departments = computed(() => {
    const depts = new Set(this.allEmployees().map(e => e.department));
    return [...depts].sort();
  });

  filteredEmployees = computed(() => {
    let result = this.allEmployees();
    const term = this.searchTerm().toLowerCase();
    const dept = this.departmentFilter();

    if (term) {
      result = result.filter(e => e.name.toLowerCase().includes(term));
    }
    if (dept) {
      result = result.filter(e => e.department === dept);
    }
    return result;
  });

  constructor() {
    this.employeeService.getAll().subscribe(data => {
      this.allEmployees.set(data);
    });
  }
}

          }
          @case (2) {
            <app-payment-info (next)="goNext()" (back)="goBack()" />
          }
          @case (3) {
            <app-review (back)="goBack()" (submit)="onSubmit()" />
          }

    </div>

  }
