# 12.5 Signal Inputs

> Signal inputs declare component inputs as read-only signals, replacing the `@Input()` decorator with a reactive, type-safe, and composable API that integrates naturally with `computed`, `effect`, and Angular's rendering pipeline.

---

## Why Signal Inputs

The `@Input()` decorator served Angular well for years, but it has fundamental limitations:

1. **Not reactive by default.** To react to input changes, you need `ngOnChanges` or setter-based `@Input()` with manual logic.
2. **Mutable.** A child component can overwrite its own `@Input` property, creating confusing data flow.
3. **No integration with signals.** In a signal-based architecture, decorator inputs require manual bridging to participate in reactive computations.

Signal inputs solve all three problems. They are:

- **Read-only signals**: The parent sets the value; the child reads it reactively.
- **Automatically tracked**: `computed()`, `effect()`, and templates subscribe to changes without any lifecycle hooks.
- **Type-safe**: Required inputs produce compile-time errors when omitted, and transform functions widen the accepted type while narrowing the output type.

Signal inputs were introduced in Angular 17.1 (developer preview) and stabilized in Angular 17.2.

## Declaring Signal Inputs

### Optional Inputs with Default Values

Use `input()` to create an optional input. The first argument is the default value.

```typescript
import { Component, input } from '@angular/core';

@Component({
  selector: 'app-badge',
  standalone: true,
  template: `<span [class]="'badge badge-' + variant()">{{ label() }}</span>`
})
export class BadgeComponent {
  label = input('Default');      // InputSignal<string>, default: 'Default'
  variant = input('primary');    // InputSignal<string>, default: 'primary'
}
```

The type is inferred from the default value. `input('Default')` creates an `InputSignal<string>`.

### Optional Inputs without Default Values

If you do not provide a default, you must specify the type explicitly. The signal's value will be `undefined` until the parent sets it.

```typescript
@Component({ /* ... */ })
export class UserCardComponent {
  avatarUrl = input<string>();    // InputSignal<string | undefined>
  subtitle = input<string>();     // InputSignal<string | undefined>
}
```

In the template, `avatarUrl()` returns `string | undefined`, so you must handle the undefined case:

```html
@if (avatarUrl(); as url) {
  <img [src]="url" alt="Avatar" />
} @else {
  <img src="/assets/default-avatar.png" alt="Default" />
}
```

### Required Inputs

Use `input.required()` to declare inputs that the parent component must provide. Omitting them causes a compile-time error.

```typescript
@Component({
  selector: 'app-product-card',
  standalone: true,
  template: `
    <div class="product">
      <h3>{{ name() }}</h3>
      <p>{{ price() | currency }}</p>
    </div>
  `
})
export class ProductCardComponent {
  name = input.required<string>();    // InputSignal<string>
  price = input.required<number>();   // InputSignal<number>
}
```

```html
<!-- Parent template -->
<app-product-card [name]="product.name" [price]="product.price" />

<!-- Compile error: Missing required input 'price' -->
<app-product-card [name]="product.name" />
```

Required inputs do not accept a default value. Their type does not include `undefined` because Angular guarantees they will be set before the component initializes.

**Important**: Do not read a required input in the constructor. At construction time, Angular has not yet set the input values:

```typescript
constructor() {
  console.log(this.name()); // ERROR: NG0950 - required input not yet set
}
```

Read required inputs in `ngOnInit`, `computed`, `effect`, `afterNextRender`, or the template. These all run after Angular has initialized inputs.

## Input Options

The `input()` and `input.required()` functions accept an options object as the second argument (or first for `input.required()`).

### alias

Changes the name of the input as seen by the parent template while keeping a different property name inside the component.

```typescript
@Component({
  selector: 'app-slider',
  standalone: true,
  template: `<input type="range" [value]="value()" />`
})
export class SliderComponent {
  // Parent binds as [sliderValue], component reads as value()
  value = input(50, { alias: 'sliderValue' });
}
```

```html
<!-- Parent template -->
<app-slider [sliderValue]="75" />
```

Use aliases sparingly. They make the mapping between parent and child less obvious. The primary use case is maintaining backward compatibility during a rename.

### transform

A function that transforms the bound value before it is stored in the signal. The transform can accept a wider type than the signal's output type.

```typescript
@Component({
  selector: 'app-toggle',
  standalone: true,
  template: `<div [class.active]="enabled()">{{ enabled() ? 'ON' : 'OFF' }}</div>`
})
export class ToggleComponent {
  // Accepts string | boolean from parent, stores boolean
  enabled = input(false, {
    transform: (value: boolean | string) => {
      return value === '' || value === true || value === 'true';
    }
  });
}
```

```html
<!-- All of these work -->
<app-toggle [enabled]="true" />
<app-toggle enabled />           <!-- attribute presence = empty string -->
<app-toggle [enabled]="'true'" />
```

The transform function:
- Runs every time the parent sets a new value
- Can widen the accepted type (e.g., accept `string | boolean`, output `boolean`)
- Must be a pure function (no side effects)
- Runs synchronously

### TypeScript types with transform

When a transform is present, the signal type and the accepted binding type differ. Angular uses `InputSignalWithTransform<T, TransformT>`:

```typescript
// Without transform
name = input('');           // InputSignal<string>

// With transform
enabled = input(false, {    // InputSignalWithTransform<boolean, string | boolean>
  transform: (v: string | boolean) => Boolean(v)
});
```

- `T` (first generic): The type returned by reading the signal. Here, `boolean`.
- `TransformT` (second generic): The type the parent can bind. Here, `string | boolean`.

## Built-in Transform Functions

Angular provides two built-in transforms for common coercion scenarios.

### booleanAttribute

Converts attribute-style values to booleans. Useful for inputs that mimic HTML attributes like `disabled`, `readonly`, or `hidden`.

```typescript
import { Component, input, booleanAttribute } from '@angular/core';

@Component({
  selector: 'app-button',
  standalone: true,
  template: `<button [disabled]="disabled()"><ng-content /></button>`
})
export class ButtonComponent {
  disabled = input(false, { transform: booleanAttribute });
}
```

```html
<!-- All of these set disabled() to true -->
<app-button disabled />             <!-- '' -> true -->
<app-button disabled="" />          <!-- '' -> true -->
<app-button [disabled]="true" />    <!-- true -> true -->

<!-- This sets disabled() to false -->
<app-button [disabled]="false" />   <!-- false -> false -->
```

`booleanAttribute` follows DOM conventions: the presence of the attribute (even with an empty string value) means `true`. Only an explicit `false` binding means `false`.

### numberAttribute

Converts string values to numbers. Provides a fallback for invalid numbers.

```typescript
import { Component, input, numberAttribute } from '@angular/core';

@Component({
  selector: 'app-pagination',
  standalone: true,
  template: `<p>Page {{ page() }} of {{ totalPages() }}</p>`
})
export class PaginationComponent {
  page = input(1, { transform: numberAttribute });
  totalPages = input(1, { transform: numberAttribute });
}
```

```html
<!-- Attribute syntax passes strings; numberAttribute converts them -->
<app-pagination page="3" totalPages="10" />
```

`numberAttribute` calls `Number()` on the value. If the result is `NaN`, it falls back to the second argument (default: `NaN`). To provide a custom fallback:

```typescript
// numberAttribute signature: (value: unknown, fallback?: number) => number
// To use a custom fallback, wrap it:
page = input(1, {
  transform: (value: unknown) => numberAttribute(value, 1)
});
```

## Reading Signal Inputs

Signal inputs are `Signal<T>`. You read them by calling the signal function.

### In Templates

```html
<h2>{{ title() }}</h2>
<p [class.active]="isActive()">Content</p>
@if (subtitle()) {
  <p class="subtitle">{{ subtitle() }}</p>
}
```

### In Computed Signals

```typescript
@Component({ /* ... */ })
export class PricingComponent {
  price = input.required<number>();
  quantity = input(1);
  taxRate = input(0.1);

  subtotal = computed(() => this.price() * this.quantity());
  tax = computed(() => this.subtotal() * this.taxRate());
  total = computed(() => this.subtotal() + this.tax());
}
```

When `price`, `quantity`, or `taxRate` change, the computed signals automatically recalculate. No `ngOnChanges` needed.

### In Effects

```typescript
@Component({ /* ... */ })
export class SearchComponent {
  query = input('');

  constructor() {
    effect(() => {
      // Automatically re-runs when query() changes
      console.log('Search query changed to:', this.query());
    });
  }
}
```

### With Resource APIs

```typescript
@Component({ /* ... */ })
export class UserDetailComponent {
  userId = input.required<number>();

  // httpResource reads userId() reactively
  userResource = httpResource<User>(() => `/api/users/${this.userId()}`);
}
```

Whenever the parent changes `userId`, the resource automatically fetches new data.

## Signal Inputs Are Read-Only

A critical difference from `@Input()`: signal inputs cannot be written to by the child component. They are `InputSignal<T>`, not `WritableSignal<T>`.

```typescript
@Component({ /* ... */ })
export class CounterComponent {
  count = input(0);

  increment(): void {
    this.count.set(1);     // ERROR: Property 'set' does not exist on type 'InputSignal<number>'
    this.count.update(n => n + 1); // ERROR: same
  }
}
```

This enforces unidirectional data flow: data flows from parent to child. If the child needs to modify a value, it should:

1. **Emit an event** so the parent updates the binding.
2. **Use `model()`** for two-way binding (covered in 12.7).
3. **Create a local writable signal** initialized from the input.

```typescript
@Component({ /* ... */ })
export class CounterComponent {
  initialCount = input(0);

  // Local mutable copy, initialized once
  count = signal(0);

  constructor() {
    // Sync initial value after inputs are set
    effect(() => {
      // Only set on first run or when parent explicitly changes
      this.count.set(this.initialCount());
    });
  }

  increment(): void {
    this.count.update(n => n + 1); // Works fine
  }
}
```

However, be cautious with this pattern. If the parent changes `initialCount`, the effect will reset the local `count`. This might or might not be what you want. Use `linkedSignal` for a more declarative version of this pattern:

```typescript
@Component({ /* ... */ })
export class CounterComponent {
  initialCount = input(0);

  // Resets to initialCount whenever it changes, but writable locally
  count = linkedSignal(() => this.initialCount());

  increment(): void {
    this.count.update(n => n + 1);
  }
}
```

## Reacting to Input Changes

### Replacing ngOnChanges with computed

The most common use of `ngOnChanges` was to derive state whenever inputs changed. `computed()` handles this declaratively.

```typescript
// Legacy: ngOnChanges
@Component({ /* ... */ })
export class LegacyComponent implements OnChanges {
  @Input() firstName = '';
  @Input() lastName = '';
  fullName = '';

  ngOnChanges(): void {
    this.fullName = `${this.firstName} ${this.lastName}`;
  }
}

// Modern: computed
@Component({ /* ... */ })
export class ModernComponent {
  firstName = input('');
  lastName = input('');
  fullName = computed(() => `${this.firstName()} ${this.lastName()}`);
}
```

### Triggering side effects with effect

When a change should trigger a side effect (not just compute a value), use `effect()`:

```typescript
@Component({ /* ... */ })
export class ThemeComponent {
  theme = input<'light' | 'dark'>('light');

  constructor() {
    effect(() => {
      // Update document attribute when theme input changes
      document.documentElement.setAttribute('data-theme', this.theme());
    });
  }
}
```

### Replacing ngOnChanges for data fetching

```typescript
// Legacy: ngOnChanges + manual subscription management
@Component({ /* ... */ })
export class LegacyDetailComponent implements OnChanges, OnDestroy {
  @Input() itemId!: number;
  item: Item | null = null;
  private sub?: Subscription;

  ngOnChanges(changes: SimpleChanges): void {
    if (changes['itemId']) {
      this.sub?.unsubscribe();
      this.sub = this.itemService.getItem(this.itemId).subscribe(i => {
        this.item = i;
      });
    }
  }

  ngOnDestroy(): void {
    this.sub?.unsubscribe();
  }
}

// Modern: httpResource reacts to signal input automatically
@Component({ /* ... */ })
export class ModernDetailComponent {
  itemId = input.required<number>();

  itemResource = httpResource<Item>(() => `/api/items/${this.itemId()}`);
  // No ngOnChanges, no ngOnDestroy, no manual subscription
}
```

## InputSignal Type Hierarchy

Understanding the type system helps when working with generics or wrapper components.

```
Signal<T>
  └── InputSignal<T>                        (no transform)
        extends InputSignalWithTransform<T, T>

InputSignalWithTransform<T, TransformT>     (with transform)
```

| Type | Generic Parameters | When Used |
|---|---|---|
| `InputSignal<T>` | `T` = read type, write type is also `T` | `input()` and `input.required()` without transform |
| `InputSignalWithTransform<T, TransformT>` | `T` = read type, `TransformT` = accepted bind type | `input()` and `input.required()` with transform |

Both types extend `Signal<T>`, so they can be passed anywhere a `Signal<T>` is expected:

```typescript
function formatDisplayName(name: Signal<string>): string {
  return name().toUpperCase();
}

// Works with both InputSignal and regular Signal
class MyComponent {
  name = input.required<string>();  // InputSignal<string> extends Signal<string>

  formatted = computed(() => formatDisplayName(this.name));
}
```

## @Input Decorator (Legacy)

You will encounter `@Input()` in existing codebases. Understanding the differences helps with migration.

### Basic legacy syntax

```typescript
@Component({ /* ... */ })
export class LegacyComponent {
  @Input() name = '';                          // Optional with default
  @Input() title?: string;                     // Optional, possibly undefined
  @Input({ required: true }) id!: number;      // Required (Angular 16+)
  @Input({ alias: 'myColor' }) color = 'blue'; // Aliased
  @Input({ transform: booleanAttribute }) disabled = false; // Transform (Angular 16+)
}
```

### Key differences between @Input and input()

| Feature | `@Input()` | `input()` / `input.required()` |
|---|---|---|
| Return type | Plain value | `InputSignal<T>` (Signal) |
| Reactive | No (need ngOnChanges) | Yes (works with computed/effect) |
| Mutable by child | Yes | No (read-only) |
| Required | `@Input({ required: true })` | `input.required()` |
| Transform | `@Input({ transform })` | `input(default, { transform })` |
| Alias | `@Input('alias')` or `@Input({ alias })` | `input(default, { alias })` |
| Status | Legacy (will be deprecated) | Current and recommended |

## Migration from @Input to input()

### Automatic Migration Schematic

Angular provides a migration schematic:

```bash
ng generate @angular/core:signal-input-migration
```

Options:
- `--path=src/app/feature`: Target a specific directory
- `--best-effort-mode`: Migrate as much as possible, adding TODO comments for inputs that cannot be safely migrated

### What the schematic handles

- Converts `@Input() prop = value` to `prop = input(value)`
- Converts `@Input({ required: true }) prop!: Type` to `prop = input.required<Type>()`
- Converts `@Input({ alias: 'x' })` to `input(value, { alias: 'x' })`
- Converts `@Input({ transform })` to `input(value, { transform })`
- Updates all template references to add `()` (e.g., `{{ prop }}` becomes `{{ prop() }}`)
- Updates TypeScript references to add `()` where the value is read

### What the schematic does NOT handle

- Inputs that are written to inside the component (signal inputs are read-only)
- Inputs used with `@ViewChild` or `@ContentChild` in ways that conflict
- Complex patterns involving `ngOnChanges` with `SimpleChanges` (requires manual refactoring to `computed` or `effect`)

For inputs that are written to, you need to decide:
1. Convert to `model()` if the parent needs the updated value (two-way binding)
2. Create a local `linkedSignal` if only the child needs a mutable copy

## Common Mistakes

### 1. Reading required inputs in the constructor

```typescript
// BAD: Required input not yet set at construction time
@Component({ /* ... */ })
export class BadComponent {
  id = input.required<number>();

  constructor() {
    console.log(this.id()); // NG0950: input 'id' is required but not set
  }
}

// GOOD: Read in computed, effect, or ngOnInit
@Component({ /* ... */ })
export class GoodComponent {
  id = input.required<number>();
  label = computed(() => `Item #${this.id()}`); // Evaluated lazily, after inputs are set
}
```

### 2. Trying to write to signal inputs

```typescript
// BAD: Signal inputs are read-only
@Component({ /* ... */ })
export class BadComponent {
  name = input('');

  reset(): void {
    this.name.set(''); // ERROR: 'set' does not exist on InputSignal<string>
  }
}

// GOOD: Use a local writable signal or linkedSignal
@Component({ /* ... */ })
export class GoodComponent {
  nameInput = input('');
  name = linkedSignal(() => this.nameInput());

  reset(): void {
    this.name.set('');
  }
}
```

### 3. Forgetting parentheses in templates

```html
<!-- BAD: Binds the signal function itself, not its value -->
<p>{{ name }}</p>            <!-- Displays: function() { ... } -->
<p [title]="name">...</p>   <!-- Binds the function reference -->

<!-- GOOD: Call the signal to get the value -->
<p>{{ name() }}</p>
<p [title]="name()">...</p>
```

This is the single most common mistake when transitioning from `@Input()` to `input()`. The schematic handles template updates, but manual conversions often miss the parentheses.

### 4. Using effect() instead of computed() for derived state

```typescript
// BAD: Effect for derived state is an anti-pattern
@Component({ /* ... */ })
export class BadComponent {
  price = input.required<number>();
  quantity = input(1);
  total = signal(0);

  constructor() {
    effect(() => {
      this.total.set(this.price() * this.quantity()); // Anti-pattern
    });
  }
}

// GOOD: Use computed for derived values
@Component({ /* ... */ })
export class GoodComponent {
  price = input.required<number>();
  quantity = input(1);
  total = computed(() => this.price() * this.quantity());
}
```

`computed` is synchronous, lazy, cached, and guaranteed to be consistent (glitch-free). `effect` is asynchronous, eagerly scheduled, and can create timing issues when used for derived state.

### 5. Over-using transform for complex logic

```typescript
// BAD: Complex parsing in a transform
items = input<ParsedItem[]>([], {
  transform: (raw: string) => {
    // Complex JSON parsing, validation, error handling in a transform
    try {
      const parsed = JSON.parse(raw);
      return parsed.filter(isValid).map(normalize);
    } catch {
      return [];
    }
  }
});

// GOOD: Keep transforms simple (type coercion only). Use computed for complex logic.
rawItems = input<string>('');
items = computed(() => {
  try {
    const parsed = JSON.parse(this.rawItems());
    return parsed.filter(isValid).map(normalize);
  } catch {
    return [];
  }
});
```

Transforms should do simple type coercion (string to boolean, string to number). Complex parsing belongs in `computed` or a service.

### 6. Declaring inputs with the wrong type when no default is provided

```typescript
// Common confusion: what type does this have?
title = input<string>();        // InputSignal<string | undefined>
title = input('');              // InputSignal<string> (no undefined)
title = input.required<string>(); // InputSignal<string> (no undefined, guaranteed set)
```

| Declaration | Type | Includes undefined? |
|---|---|---|
| `input<string>()` | `InputSignal<string \| undefined>` | Yes |
| `input('')` | `InputSignal<string>` | No |
| `input.required<string>()` | `InputSignal<string>` | No |

If you need to distinguish "not set" from "set to empty string", use `input<string>()` without a default. If you just need a safe default, provide one.

## Best Practices

### 1. Use input.required for inputs that the component cannot function without

If a component makes no sense without a particular input, declare it as required. This catches errors at compile time rather than runtime.

```typescript
// A user card without a user is meaningless
@Component({ /* ... */ })
export class UserCardComponent {
  user = input.required<User>();
}
```

### 2. Provide meaningful defaults for optional inputs

Instead of `input<string>()` which yields `string | undefined`, consider whether a default makes the component simpler to use:

```typescript
// Consumer must always handle undefined
variant = input<string>();      // 'primary' | 'secondary' | undefined

// Consumer can omit, and the component works out of the box
variant = input('primary');     // Always 'primary' | 'secondary'
```

### 3. Use booleanAttribute and numberAttribute for attribute-style binding

When a component will be used with plain HTML attributes (no square brackets), these transforms ensure correct coercion:

```html
<!-- Without transform, disabled is the string "true", not boolean true -->
<app-button disabled="true" />

<!-- With booleanAttribute, any presence of the attribute = true -->
<app-button disabled />
```

### 4. Derive state with computed, not effect

Always prefer `computed()` over `effect()` for values derived from inputs. `computed` is synchronous and glitch-free. `effect` is for side effects only (logging, DOM manipulation, external API calls).

### 5. Use readonly for all input declarations

This is enforced by the type system (signal inputs return `InputSignal`, not `WritableSignal`), but adding `readonly` as a convention makes intent clear and prevents accidental reassignment of the signal itself:

```typescript
readonly name = input.required<string>();
readonly count = input(0);
```

### 6. Keep transforms pure and simple

Transforms should convert types, not perform business logic. Use `computed` for anything beyond basic coercion.

## Real-World Use Cases

### Configurable Data Table Column

```typescript
interface ColumnDef<T> {
  key: keyof T;
  header: string;
  sortable?: boolean;
}

@Component({
  selector: 'app-data-table',
  standalone: true,
  template: `
    <table>
      <thead>
        <tr>
          @for (col of columns(); track col.key) {
            <th
              [class.sortable]="col.sortable"
              (click)="col.sortable ? toggleSort(col.key) : null"
            >
              {{ col.header }}
              @if (col.sortable && sortKey() === col.key) {
                <span>{{ sortDir() === 'asc' ? '↑' : '↓' }}</span>
              }
            </th>
          }
        </tr>
      </thead>
      <tbody>
        @for (row of sortedData(); track trackFn()(row)) {
          <tr>
            @for (col of columns(); track col.key) {
              <td>{{ row[col.key] }}</td>
            }
          </tr>
        } @empty {
          <tr><td [attr.colspan]="columns().length">{{ emptyMessage() }}</td></tr>
        }
      </tbody>
    </table>
  `
})
export class DataTableComponent<T extends Record<string, unknown>> {
  data = input.required<T[]>();
  columns = input.required<ColumnDef<T>[]>();
  trackFn = input<(item: T) => unknown>(() => (item: T) => item);
  emptyMessage = input('No data available.');

  sortKey = signal<keyof T | null>(null);
  sortDir = signal<'asc' | 'desc'>('asc');

  sortedData = computed(() => {
    const key = this.sortKey();
    const dir = this.sortDir();
    const items = [...this.data()];

    if (!key) return items;

    return items.sort((a, b) => {
      const aVal = a[key];
      const bVal = b[key];
      const cmp = aVal < bVal ? -1 : aVal > bVal ? 1 : 0;
      return dir === 'asc' ? cmp : -cmp;
    });
  });

  toggleSort(key: keyof T): void {
    if (this.sortKey() === key) {
      this.sortDir.update(d => d === 'asc' ? 'desc' : 'asc');
    } else {
      this.sortKey.set(key);
      this.sortDir.set('asc');
    }
  }
}
```

### Reusable Pagination Component

```typescript
@Component({
  selector: 'app-pagination',
  standalone: true,
  template: `
    <nav aria-label="Pagination">
      <button [disabled]="currentPage() <= 1" (click)="goToPage(currentPage() - 1)">
        Previous
      </button>

      @for (page of visiblePages(); track page) {
        <button
          [class.active]="page === currentPage()"
          (click)="goToPage(page)"
        >
          {{ page }}
        </button>
      }

      <button [disabled]="currentPage() >= totalPages()" (click)="goToPage(currentPage() + 1)">
        Next
      </button>
    </nav>
  `
})
export class PaginationComponent {
  totalItems = input.required<number>();
  pageSize = input(10, { transform: numberAttribute });
  currentPage = input(1, { transform: numberAttribute });
  maxVisiblePages = input(5, { transform: numberAttribute });

  pageChange = output<number>();

  totalPages = computed(() => Math.ceil(this.totalItems() / this.pageSize()));

  visiblePages = computed(() => {
    const total = this.totalPages();
    const current = this.currentPage();
    const max = this.maxVisiblePages();

    let start = Math.max(1, current - Math.floor(max / 2));
    let end = Math.min(total, start + max - 1);
    start = Math.max(1, end - max + 1);

    const pages: number[] = [];
    for (let i = start; i <= end; i++) {
      pages.push(i);
    }
    return pages;
  });

  goToPage(page: number): void {
    if (page >= 1 && page <= this.totalPages()) {
      this.pageChange.emit(page);
    }
  }
}
```

```html
<!-- Usage with attribute syntax (numberAttribute coerces strings) -->
<app-pagination
  [totalItems]="searchResults().length"
  pageSize="20"
  [currentPage]="page()"
  (pageChange)="page.set($event)"
/>
```

---

> **Key Takeaway:** Signal inputs (`input()` and `input.required()`) replace `@Input()` with read-only, reactive signals that integrate directly with `computed`, `effect`, and `resource`, eliminating the need for `ngOnChanges` and enforcing unidirectional data flow by making inputs immutable from the child's perspective.
