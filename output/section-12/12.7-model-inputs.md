# 12.7 Model Inputs

> The `model()` function creates a writable signal that is automatically exposed as an input/output pair, enabling two-way binding between parent and child components with a single declaration.

---

## What Problem Does model() Solve

Two-way binding is the ability for a child component to both receive a value from its parent and push changes back. Before `model()`, achieving this required declaring a separate `input()` and `output()`:

```typescript
// Before model(): two declarations for one concept
@Component({
  selector: 'app-slider',
  standalone: true,
  template: `<input type="range" [value]="value()" (input)="onInput($event)" />`
})
export class SliderComponent {
  value = input(0);
  valueChange = output<number>(); // Must follow the "nameChange" convention

  onInput(event: Event): void {
    const newVal = +(event.target as HTMLInputElement).value;
    this.valueChange.emit(newVal);  // Manually emit
  }
}
```

```html
<!-- Parent: banana-in-a-box syntax desugars to input + output -->
<app-slider [(value)]="sliderValue" />
```

With `model()`, this collapses into a single declaration:

```typescript
// After model(): one declaration, same behavior
@Component({
  selector: 'app-slider',
  standalone: true,
  template: `<input type="range" [value]="value()" (input)="onInput($event)" />`
})
export class SliderComponent {
  value = model(0); // ModelSignal<number>: writable, input + output in one

  onInput(event: Event): void {
    this.value.set(+(event.target as HTMLInputElement).value); // Just set the signal
  }
}
```

```html
<!-- Parent: same syntax -->
<app-slider [(value)]="sliderValue" />
```

The child calls `value.set()` or `value.update()`. Angular automatically notifies the parent through the implicit `valueChange` output. No manual `emit()` needed.

## Declaring Model Inputs

### Optional with Default Value

```typescript
import { Component, model } from '@angular/core';

@Component({
  selector: 'app-toggle',
  standalone: true,
  template: `
    <button (click)="checked.set(!checked())">
      {{ checked() ? 'ON' : 'OFF' }}
    </button>
  `
})
export class ToggleComponent {
  checked = model(false); // ModelSignal<boolean>, default: false
}
```

The type is inferred from the default value. `model(false)` creates a `ModelSignal<boolean>`.

### Optional without Default Value

```typescript
@Component({ /* ... */ })
export class DatePickerComponent {
  selectedDate = model<Date>(); // ModelSignal<Date | undefined>
}
```

Without a default, the type includes `undefined`. The parent may or may not bind to it.

### Required Model Inputs

Use `model.required()` when the parent must provide a value:

```typescript
@Component({
  selector: 'app-color-picker',
  standalone: true,
  template: `<!-- ... -->`
})
export class ColorPickerComponent {
  color = model.required<string>(); // ModelSignal<string>, no undefined
}
```

```html
<!-- Parent must bind -->
<app-color-picker [(color)]="selectedColor" />

<!-- Compile error: missing required model input -->
<app-color-picker />
```

### Model Input with Alias

```typescript
@Component({
  selector: 'app-rating',
  standalone: true,
  template: `<!-- ... -->`
})
export class RatingComponent {
  // Internal name: score. Parent binds as: [(rating)]
  score = model(0, { alias: 'rating' });
}
```

```html
<app-rating [(rating)]="userRating" />
```

The alias affects both the input name and the auto-generated output name. With `alias: 'rating'`, the output becomes `ratingChange`.

## How Two-Way Binding Desugars

The banana-in-a-box syntax `[()]` is syntactic sugar. Angular desugars it into a property binding and an event binding.

### With model()

```html
<!-- This -->
<app-slider [(value)]="sliderValue" />

<!-- Desugars to -->
<app-slider [value]="sliderValue" (valueChange)="sliderValue = $event" />
```

When `model()` is used:
1. Angular binds the parent's `sliderValue` to the child's `value` model input.
2. When the child calls `value.set(newVal)` or `value.update(fn)`, Angular automatically emits `newVal` through the implicit `valueChange` output.
3. The parent receives the event and updates `sliderValue`.

### With signal in parent

When the parent uses a signal (which is the typical case):

```typescript
// Parent
@Component({
  template: `<app-slider [(value)]="sliderValue" />`
})
export class ParentComponent {
  sliderValue = signal(50); // WritableSignal<number>
}
```

The desugared binding becomes:

```html
<app-slider [value]="sliderValue()" (valueChange)="sliderValue.set($event)" />
```

Angular handles this automatically. You do not write the desugared form.

### The "Change" suffix convention

The auto-generated output always follows the pattern: `{modelName}Change`. This is the same convention used for manual two-way binding since Angular 2:

| Model Name | Input Name | Output Name |
|---|---|---|
| `value` | `value` | `valueChange` |
| `checked` | `checked` | `checkedChange` |
| `selectedDate` | `selectedDate` | `selectedDateChange` |
| `score` (alias: `rating`) | `rating` | `ratingChange` |

## ModelSignal API

`model()` returns a `ModelSignal<T>`, which extends `WritableSignal<T>`. This means it has all the methods of a regular writable signal:

| Method | Signature | Description |
|---|---|---|
| `()` (call) | `(): T` | Read the current value |
| `set` | `set(value: T): void` | Set a new value |
| `update` | `update(fn: (current: T) => T): void` | Update based on the current value |
| `asReadonly` | `asReadonly(): Signal<T>` | Return a read-only view |

```typescript
@Component({ /* ... */ })
export class CounterComponent {
  count = model(0);

  increment(): void {
    this.count.update(n => n + 1);
  }

  reset(): void {
    this.count.set(0);
  }

  // Read the value
  isPositive = computed(() => this.count() > 0);
}
```

### Difference from input()

| Aspect | `input()` | `model()` |
|---|---|---|
| Return type | `InputSignal<T>` (read-only) | `ModelSignal<T>` (writable) |
| Child can modify | No | Yes (`set`, `update`) |
| Creates output | No | Yes (auto `nameChange`) |
| Two-way binding | No (one-way only) | Yes (`[()]` syntax) |
| Data flow | Parent -> Child | Parent <-> Child |
| Use case | Display data, configure behavior | User-editable state |

## Using model() with computed and effect

Because `ModelSignal` extends `WritableSignal` (and `WritableSignal` extends `Signal`), model inputs work seamlessly with all signal APIs.

### With computed

```typescript
@Component({
  selector: 'app-temperature',
  standalone: true,
  template: `
    <input type="number" [value]="celsius()" (input)="onInput($event)" />
    <p>{{ fahrenheit() }} F</p>
  `
})
export class TemperatureComponent {
  celsius = model(0);

  fahrenheit = computed(() => this.celsius() * 9 / 5 + 32);

  onInput(event: Event): void {
    this.celsius.set(+(event.target as HTMLInputElement).value);
  }
}
```

### With effect

```typescript
@Component({ /* ... */ })
export class VolumeComponent {
  volume = model(50);

  constructor() {
    effect(() => {
      console.log('Volume changed to:', this.volume());
      // Could sync to an audio API, analytics, etc.
    });
  }
}
```

### With resource

```typescript
@Component({ /* ... */ })
export class CategoryBrowserComponent {
  selectedCategory = model.required<string>();

  // Fetch items for the currently selected category
  itemsResource = httpResource<Item[]>(() =>
    `/api/categories/${this.selectedCategory()}/items`
  );
}
```

When the parent updates `selectedCategory` through two-way binding, or the child updates it via `set()`, the resource automatically refetches.

## model() Does Not Support transform

Unlike `input()`, the `model()` function does not accept a `transform` option. This is by design: transforms modify the value going in, but a model input also pushes values back out. A transform would create ambiguity about what the parent receives.

```typescript
// This does NOT work
value = model(0, { transform: numberAttribute }); // ERROR: transform not available

// Workaround: Use a computed for the transformed view
rawValue = model('');
numericValue = computed(() => Number(this.rawValue()) || 0);
```

If you need coercion, use a regular `input()` with a transform, paired with a separate `output()`.

## One-Way Binding to a Model Input

A model input can be used with one-way binding. The parent provides the value, but does not listen for changes:

```html
<!-- One-way: parent sets value, ignores child updates -->
<app-slider [value]="50" />

<!-- Two-way: parent and child stay in sync -->
<app-slider [(value)]="sliderValue" />
```

When bound one-way with `[value]="50"`, the child can still call `value.set(75)`, but the parent will not be notified. The child's local state updates, but the parent's `50` remains unchanged on re-render.

### Unbound model inputs

If the parent does not bind to the model input at all, the model signal works as a regular writable signal inside the component:

```html
<!-- No binding at all: uses the default value -->
<app-slider />
```

The child's `value` starts at its default (e.g., `0`) and can be freely modified with `set()` and `update()`.

## model() vs linkedSignal

Both `model()` and `linkedSignal` create writable signals that react to external changes, but they serve different purposes.

| Aspect | `model()` | `linkedSignal()` |
|---|---|---|
| Purpose | Two-way binding between parent and child | Local mutable copy of a signal/input |
| Exposed as output | Yes (auto `nameChange`) | No |
| Parent notified of changes | Yes (through two-way binding) | No |
| Source | Parent binding | Any signal (including inputs) |
| Use case | Form controls, UI components | Editable local state derived from input |

### When to use each

```typescript
// model(): Parent and child share the same state
@Component({ /* ... */ })
export class CheckboxComponent {
  checked = model(false); // Parent can use [(checked)]
}

// linkedSignal(): Child has its own editable copy
@Component({ /* ... */ })
export class EditableNameComponent {
  name = input.required<string>();
  editableName = linkedSignal(() => this.name()); // Resets when input changes

  save(): void {
    this.saved.emit(this.editableName()); // Manually push back
  }
  saved = output<string>();
}
```

Use `model()` when the parent should automatically receive updates. Use `linkedSignal` when the child needs a draft/editable copy that is explicitly saved.

## Legacy Two-Way Binding Pattern

Before `model()`, custom two-way binding required the `input()` + `output()` pair with the `Change` suffix convention:

```typescript
// Legacy pattern (still works, but verbose)
@Component({
  selector: 'app-legacy-slider',
  standalone: true,
  template: `<input type="range" [value]="value()" (input)="onInput($event)" />`
})
export class LegacySliderComponent {
  value = input(0);
  valueChange = output<number>();

  onInput(event: Event): void {
    const newVal = +(event.target as HTMLInputElement).value;
    this.valueChange.emit(newVal);
  }
}
```

```html
<!-- Still uses banana-in-a-box -->
<app-legacy-slider [(value)]="sliderValue" />
```

This works because Angular's desugaring convention looks for an output named `{inputName}Change`. The `model()` function automates this convention.

### With the old @Input/@Output decorators

```typescript
// Oldest pattern
@Component({ /* ... */ })
export class OldestSliderComponent {
  @Input() value = 0;
  @Output() valueChange = new EventEmitter<number>();

  onInput(event: Event): void {
    this.value = +(event.target as HTMLInputElement).value;
    this.valueChange.emit(this.value);
  }
}
```

The progression from oldest to newest:

| Era | Input | Output | Child Update Mechanism |
|---|---|---|---|
| Decorator-based | `@Input()` | `@Output() nameChange = new EventEmitter()` | Mutate property + emit |
| Function-based | `input()` | `output() nameChange` | Call output.emit() |
| Model-based | `model()` | Auto-generated | Call model.set() or model.update() |

## Common Mistakes

### 1. Using model() when input() suffices

```typescript
// BAD: model() for a display-only input
@Component({ /* ... */ })
export class BadComponent {
  title = model(''); // Why writable? The child never modifies it.
}

// GOOD: Use input() for values the child only reads
@Component({ /* ... */ })
export class GoodComponent {
  title = input('');
}
```

`model()` should only be used when the child component genuinely needs to update the value and push changes back to the parent. For one-way data flow, use `input()`.

### 2. Expecting transform support on model()

```typescript
// BAD: model() does not support transforms
disabled = model(false, { transform: booleanAttribute }); // ERROR

// GOOD: Use input() with transform if you need coercion
disabled = input(false, { transform: booleanAttribute });
```

If you need both coercion and two-way binding, restructure: use `model()` with the correct type and handle coercion in the parent, or use a regular `input()` + `output()` pair.

### 3. Deep nesting of model() bindings

```typescript
// PROBLEMATIC: model() passed through multiple levels
// GrandparentComponent -> ParentComponent -> ChildComponent

// Grandparent
template: `<app-parent [(value)]="val" />`

// Parent (forwarding)
value = model(0);
template: `<app-child [(value)]="value" />`

// Child (modifying)
value = model(0);
```

When a model signal flows through multiple levels of nesting, it becomes difficult to trace where changes originate. At more than two levels, consider using a shared service with signals instead.

### 4. Forgetting the parentheses on two-way binding syntax

```html
<!-- BAD: One-way binding only (square brackets alone) -->
<app-slider [value]="sliderValue" />

<!-- BAD: Event binding only (parentheses alone) -->
<app-slider (value)="sliderValue" />

<!-- GOOD: Two-way binding (banana-in-a-box) -->
<app-slider [(value)]="sliderValue" />
```

A common typo. `[value]` is one-way. `[(value)]` is two-way.

### 5. Using model() in a service or non-component context

`model()` can only be called in a component or directive property initializer, inside an injection context. It does not work in services, utility functions, or standalone code.

```typescript
// BAD: model() in a service
@Injectable({ providedIn: 'root' })
export class BadService {
  value = model(0); // ERROR: model() must be used in a component/directive
}

// GOOD: Use signal() in services
@Injectable({ providedIn: 'root' })
export class GoodService {
  value = signal(0);
}
```

### 6. Calling set() and emit() redundantly

When migrating from the `input()` + `output()` pattern to `model()`, developers sometimes keep both patterns:

```typescript
// BAD: Redundant after migrating to model()
@Component({ /* ... */ })
export class BadComponent {
  value = model(0);
  valueChange = output<number>(); // Redundant! model() already creates this

  onInput(event: Event): void {
    const newVal = +(event.target as HTMLInputElement).value;
    this.value.set(newVal);           // model() handles the output
    this.valueChange.emit(newVal);     // Double emission!
  }
}

// GOOD: model() handles everything
@Component({ /* ... */ })
export class GoodComponent {
  value = model(0);

  onInput(event: Event): void {
    this.value.set(+(event.target as HTMLInputElement).value);
  }
}
```

## Best Practices

### 1. Use model() only for interactive, user-editable state

The canonical use cases are form controls, toggles, sliders, pickers, and similar UI components where the component exists to let the user modify a value.

### 2. Prefer input() + output() for non-symmetric communication

If the child emits an event that is semantically different from "the value changed" (e.g., `formSubmitted`, `itemDeleted`), use separate `input()` and `output()`. Model inputs are specifically for value synchronization.

### 3. Keep model inputs shallow (avoid deep nesting)

Limit model bindings to one level of parent-child communication. For deeper state sharing, use a service with signals.

### 4. Use asReadonly() to expose model values safely

If you need to pass the model's value to a child component that should not modify it:

```typescript
@Component({ /* ... */ })
export class FormComponent {
  username = model('');

  // Expose a read-only view for display-only children
  readonlyUsername = this.username.asReadonly();
}
```

```html
<app-username-display [name]="readonlyUsername()" />
```

### 5. Name model inputs after the value they represent

```typescript
// GOOD: Clear what the value is
checked = model(false);
selectedDate = model<Date | null>(null);
searchQuery = model('');

// BAD: Vague or action-oriented names
data = model<unknown>(null);     // What kind of data?
change = model('');              // Sounds like an event, not a value
```

### 6. Use computed for derived state from model inputs

Do not use `effect()` to derive values from model inputs. Use `computed()`:

```typescript
@Component({ /* ... */ })
export class PriceInputComponent {
  price = model(0);

  // GOOD: Derived state via computed
  formattedPrice = computed(() =>
    new Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' })
      .format(this.price())
  );

  isValid = computed(() => this.price() > 0 && this.price() < 10000);
}
```

## Real-World Use Cases

### Custom Toggle/Switch Component

```typescript
@Component({
  selector: 'app-switch',
  standalone: true,
  template: `
    <button
      role="switch"
      [attr.aria-checked]="checked()"
      [class.on]="checked()"
      (click)="checked.update(v => !v)"
    >
      <span class="thumb"></span>
      <span class="label">{{ checked() ? onLabel() : offLabel() }}</span>
    </button>
  `,
  styles: [`
    :host { display: inline-block; }
    button { /* switch styles */ }
    .on .thumb { transform: translateX(24px); }
  `]
})
export class SwitchComponent {
  checked = model(false);
  onLabel = input('On');
  offLabel = input('Off');
}
```

```html
<!-- Parent: two-way binding keeps parent and child in sync -->
<app-switch [(checked)]="darkMode" onLabel="Dark" offLabel="Light" />
<p>Current theme: {{ darkMode() ? 'Dark' : 'Light' }}</p>
```

### Star Rating Component

```typescript
@Component({
  selector: 'app-star-rating',
  standalone: true,
  template: `
    <div class="stars" role="radiogroup" aria-label="Rating">
      @for (star of stars(); track star) {
        <button
          role="radio"
          [attr.aria-checked]="star <= rating()"
          [class.filled]="star <= rating()"
          [class.hovered]="star <= hoveredStar()"
          (click)="rating.set(star)"
          (mouseenter)="hoveredStar.set(star)"
          (mouseleave)="hoveredStar.set(0)"
        >
          &#9733;
        </button>
      }
    </div>
    @if (showLabel()) {
      <span class="label">{{ rating() }} / {{ maxStars() }}</span>
    }
  `
})
export class StarRatingComponent {
  rating = model(0);
  maxStars = input(5);
  showLabel = input(true);

  hoveredStar = signal(0);

  stars = computed(() =>
    Array.from({ length: this.maxStars() }, (_, i) => i + 1)
  );
}
```

```html
<app-star-rating [(rating)]="productRating" maxStars="5" />
<p>You rated this product: {{ productRating() }} stars</p>
```

### Accordion Panel with Expanded State

```typescript
@Component({
  selector: 'app-accordion-panel',
  standalone: true,
  template: `
    <div class="panel" [class.expanded]="expanded()">
      <button
        class="header"
        [attr.aria-expanded]="expanded()"
        (click)="expanded.update(v => !v)"
      >
        <ng-content select="[panel-header]" />
        <span class="icon">{{ expanded() ? '-' : '+' }}</span>
      </button>
      @if (expanded()) {
        <div class="body" [@expand]>
          <ng-content />
        </div>
      }
    </div>
  `
})
export class AccordionPanelComponent {
  expanded = model(false);
}
```

```html
<!-- Parent controls which panels are open -->
<app-accordion-panel [(expanded)]="faqOpen">
  <h3 panel-header>Frequently Asked Questions</h3>
  <p>Content here...</p>
</app-accordion-panel>

<app-accordion-panel [(expanded)]="termsOpen">
  <h3 panel-header>Terms and Conditions</h3>
  <p>Content here...</p>
</app-accordion-panel>

<!-- Parent can programmatically expand all -->
<button (click)="faqOpen.set(true); termsOpen.set(true)">Expand All</button>
```

The parent retains full control over which panels are expanded, while each panel can also toggle itself. The two-way binding keeps both sides in sync without any manual `emit()` calls.

---

> **Key Takeaway:** Use `model()` when a component needs to both receive and update a value with automatic parent notification through two-way binding `[()]`; for everything else (display-only data, configuration, or semantically distinct events), stick with `input()` and `output()`.
