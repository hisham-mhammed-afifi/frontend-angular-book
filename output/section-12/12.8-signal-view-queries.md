# 12.8 Signal View Queries

> Signal view queries (`viewChild` and `viewChildren`) let you reference elements, components, and directives in a component's own template as reactive signals, replacing the `@ViewChild` and `@ViewChildren` decorators with a type-safe, lifecycle-hook-free API.

---

## What Are View Queries

A view query finds elements in the component's own template (the HTML the component renders). This is distinct from content queries, which find elements projected into the component via `<ng-content>` (covered in 12.9).

View queries let you:
- Access DOM elements (via `ElementRef`)
- Access child component instances to call their methods or read their state
- Access directive instances applied to template elements
- Access `TemplateRef` for dynamic template rendering
- Access `ViewContainerRef` for dynamic component creation

---

## viewChild - Single Element Query

`viewChild()` returns a signal containing the first matching element. If no match is found, the signal's value is `undefined`.

### By Template Reference Variable

```typescript
import { Component, viewChild, ElementRef } from '@angular/core';

@Component({
  selector: 'app-search',
  standalone: true,
  template: `<input #searchInput type="text" placeholder="Search..." />`
})
export class SearchComponent {
  // Query by template reference variable name (string)
  searchInput = viewChild<ElementRef<HTMLInputElement>>('searchInput');
  // Type: Signal<ElementRef<HTMLInputElement> | undefined>

  focus(): void {
    this.searchInput()?.nativeElement.focus();
  }
}
```

The string `'searchInput'` matches the `#searchInput` template reference variable. The generic `<ElementRef<HTMLInputElement>>` specifies the expected type.

### By Component Type

```typescript
@Component({
  selector: 'app-dashboard',
  standalone: true,
  imports: [ChartComponent],
  template: `<app-chart [data]="chartData()" />`
})
export class DashboardComponent {
  // Query by component class
  chart = viewChild(ChartComponent);
  // Type: Signal<ChartComponent | undefined>

  refreshChart(): void {
    this.chart()?.redraw();
  }
}
```

When querying by component type, Angular returns the component instance, giving you access to its public API.

### By Directive Type

```typescript
@Component({
  selector: 'app-form',
  standalone: true,
  imports: [FormsModule],
  template: `
    <form>
      <input ngModel name="email" #emailModel="ngModel" />
    </form>
  `
})
export class FormComponent {
  // Query by directive type
  emailModel = viewChild(NgModel);
  // Type: Signal<NgModel | undefined>

  isEmailValid = computed(() => this.emailModel()?.valid ?? false);
}
```

## viewChild.required - Guaranteed Match

When you know an element will always exist in the template (not hidden by `@if` or `@for`), use `viewChild.required()`. This removes `undefined` from the type and throws a runtime error if no match is found.

```typescript
@Component({
  selector: 'app-canvas-draw',
  standalone: true,
  template: `<canvas #drawCanvas width="800" height="600"></canvas>`
})
export class CanvasDrawComponent {
  // Required: canvas is always in the template
  canvas = viewChild.required<ElementRef<HTMLCanvasElement>>('drawCanvas');
  // Type: Signal<ElementRef<HTMLCanvasElement>> (no undefined)

  constructor() {
    afterNextRender(() => {
      const ctx = this.canvas().nativeElement.getContext('2d')!;
      // ctx is guaranteed to be available
      this.initDrawing(ctx);
    });
  }

  private initDrawing(ctx: CanvasRenderingContext2D): void {
    // Set up canvas drawing
  }
}
```

### When to use required vs optional

| Scenario | Use |
|---|---|
| Element is always in the template | `viewChild.required()` |
| Element is inside `@if`, `@for`, or `@switch` | `viewChild()` (optional) |
| Element is in a `@defer` block | `viewChild()` (optional) |
| Element is always present but you want to defer access | `viewChild()` (optional) |

## viewChildren - Multiple Element Query

`viewChildren()` returns a signal containing a read-only array of all matching elements. If no elements match, the signal contains an empty array (never `undefined`).

```typescript
@Component({
  selector: 'app-tab-group',
  standalone: true,
  imports: [TabComponent],
  template: `
    <div class="tabs">
      @for (tab of tabs(); track tab.id) {
        <app-tab [label]="tab.label" [content]="tab.content" />
      }
    </div>
  `
})
export class TabGroupComponent {
  tabs = input.required<TabData[]>();

  // Query all TabComponent instances
  tabComponents = viewChildren(TabComponent);
  // Type: Signal<readonly TabComponent[]>

  tabCount = computed(() => this.tabComponents().length);

  activateTab(index: number): void {
    const tabs = this.tabComponents();
    tabs.forEach((tab, i) => tab.setActive(i === index));
  }
}
```

### By template reference variable

```typescript
@Component({
  selector: 'app-gallery',
  standalone: true,
  template: `
    @for (image of images(); track image.id) {
      <img #galleryItem [src]="image.url" [alt]="image.alt" />
    }
  `
})
export class GalleryComponent {
  images = input.required<ImageData[]>();

  // All elements with #galleryItem
  galleryItems = viewChildren<ElementRef<HTMLImageElement>>('galleryItem');
  // Type: Signal<readonly ElementRef<HTMLImageElement>[]>

  scrollToImage(index: number): void {
    const items = this.galleryItems();
    if (index < items.length) {
      items[index].nativeElement.scrollIntoView({ behavior: 'smooth' });
    }
  }
}
```

### Reactivity with control flow

`viewChildren` automatically updates when the DOM changes due to `@if`, `@for`, or `@switch`. The signal reflects the current set of matching elements.

```typescript
@Component({
  selector: 'app-filter-list',
  standalone: true,
  imports: [ItemCardComponent],
  template: `
    @for (item of filteredItems(); track item.id) {
      <app-item-card [item]="item" />
    }
  `
})
export class FilterListComponent {
  allItems = input.required<Item[]>();
  filter = model('');

  filteredItems = computed(() =>
    this.allItems().filter(i => i.name.includes(this.filter()))
  );

  // This signal automatically updates as filteredItems changes
  cards = viewChildren(ItemCardComponent);
  // When filter changes -> filteredItems changes -> DOM updates -> cards signal updates

  cardCount = computed(() => this.cards().length);
}
```

## The read Option

By default, Angular returns the "natural" type of the matched element:
- A component reference variable (`#ref`) returns the component instance
- A plain HTML element reference variable (`#ref`) returns `ElementRef`
- A directive export returns the directive instance

The `read` option overrides this default, letting you ask for a specific type from the matched element.

### Reading ElementRef from a Component

```typescript
@Component({
  selector: 'app-container',
  standalone: true,
  imports: [CardComponent],
  template: `<app-card #card [data]="cardData" />`
})
export class ContainerComponent {
  // Default: returns CardComponent instance
  card = viewChild(CardComponent);
  // Type: Signal<CardComponent | undefined>

  // With read: ElementRef: returns the host DOM element of CardComponent
  cardElement = viewChild('card', { read: ElementRef });
  // Type: Signal<ElementRef | undefined>
}
```

### Reading ViewContainerRef

`ViewContainerRef` is needed for dynamic component creation. It cannot be inferred automatically and must be explicitly requested via `read`.

```typescript
@Component({
  selector: 'app-dynamic-host',
  standalone: true,
  template: `<div #outlet></div>`
})
export class DynamicHostComponent {
  // Read as ViewContainerRef for dynamic component insertion
  outlet = viewChild.required('outlet', { read: ViewContainerRef });
  // Type: Signal<ViewContainerRef>

  loadComponent(component: Type<unknown>): void {
    const vcr = this.outlet();
    vcr.clear();
    vcr.createComponent(component);
  }
}
```

### Reading TemplateRef

```typescript
@Component({
  selector: 'app-tooltip-host',
  standalone: true,
  template: `
    <ng-template #tooltipTpl>
      <div class="tooltip">{{ tooltipText() }}</div>
    </ng-template>

    <button (mouseenter)="showTooltip()" (mouseleave)="hideTooltip()">
      Hover me
    </button>
  `
})
export class TooltipHostComponent {
  tooltipText = input('Tooltip');

  tooltipTemplate = viewChild.required('tooltipTpl', { read: TemplateRef });
  // Type: Signal<TemplateRef<unknown>>
}
```

### Reading a specific directive

When an element has multiple directives, you can query for a specific one:

```typescript
@Component({
  selector: 'app-form-page',
  standalone: true,
  imports: [ReactiveFormsModule, HighlightDirective],
  template: `
    <input formControlName="email" appHighlight #emailField />
  `
})
export class FormPageComponent {
  // Read the HighlightDirective instance from the element
  highlighter = viewChild('emailField', { read: HighlightDirective });
  // Type: Signal<HighlightDirective | undefined>
}
```

### Summary of read options

| `read` Value | Returns | When to Use |
|---|---|---|
| (default / not specified) | Component, directive, or `ElementRef` | Most cases |
| `ElementRef` | `ElementRef` wrapping the DOM element | Need the raw DOM node of a component |
| `ViewContainerRef` | `ViewContainerRef` | Dynamic component creation |
| `TemplateRef` | `TemplateRef` | Dynamic template rendering |
| `SomeDirective` | Directive instance | Access a specific directive on an element |

## Locator Types

The first argument to `viewChild()` and `viewChildren()` is the locator. It can be:

### String locator

Matches template reference variables (`#name`):

```typescript
// Matches <div #myRef>
myRef = viewChild<ElementRef>('myRef');
```

### Component or Directive class locator

Matches instances of a component or directive:

```typescript
// Matches all <app-card> instances
cards = viewChildren(CardComponent);

// Matches all elements with [appTooltip]
tooltips = viewChildren(TooltipDirective);
```

### TemplateRef locator

Rarely used directly. More commonly, you query `<ng-template>` by string reference and use `read: TemplateRef`.

## Reactivity and Timing

### Signal queries update automatically

Unlike the decorator-based API where you had to wait for `ngAfterViewInit`, signal queries are reactive. The signal value updates whenever the DOM changes.

```typescript
@Component({
  selector: 'app-conditional',
  standalone: true,
  template: `
    @if (showPanel()) {
      <div #panel>Panel content</div>
    }
    <button (click)="showPanel.update(v => !v)">Toggle</button>
  `
})
export class ConditionalComponent {
  showPanel = signal(false);

  panel = viewChild<ElementRef>('panel');
  // undefined when showPanel is false
  // ElementRef when showPanel is true

  panelVisible = computed(() => this.panel() !== undefined);
}
```

### No need for ngAfterViewInit

With decorator queries, you had to wait for `ngAfterViewInit` to safely access the query result. Signal queries remove this requirement because they are reactive:

```typescript
// Legacy: had to wait for ngAfterViewInit
@Component({ /* ... */ })
export class LegacyComponent implements AfterViewInit {
  @ViewChild('canvas') canvasRef!: ElementRef<HTMLCanvasElement>;

  ngAfterViewInit(): void {
    // Safe to access here
    const ctx = this.canvasRef.nativeElement.getContext('2d');
  }
}

// Modern: use afterNextRender or effect
@Component({ /* ... */ })
export class ModernComponent {
  canvas = viewChild.required<ElementRef<HTMLCanvasElement>>('canvas');

  constructor() {
    afterNextRender(() => {
      const ctx = this.canvas().nativeElement.getContext('2d');
      // Safe: afterNextRender fires after the DOM is ready
    });
  }
}
```

### Using effect to react to query changes

```typescript
@Component({
  selector: 'app-monitor',
  standalone: true,
  imports: [SensorComponent],
  template: `
    @for (sensor of activeSensors(); track sensor.id) {
      <app-sensor [config]="sensor" />
    }
  `
})
export class MonitorComponent {
  activeSensors = input.required<SensorConfig[]>();
  sensors = viewChildren(SensorComponent);

  constructor() {
    effect(() => {
      const count = this.sensors().length;
      console.log(`Active sensor components: ${count}`);
    });
  }
}
```

## Query Boundaries

View queries only search within the component's own template. They do not:

- Pierce into child component templates
- Find elements in projected content (`<ng-content>`)
- Find elements in parent components

```typescript
@Component({
  selector: 'app-parent',
  standalone: true,
  imports: [ChildComponent],
  template: `
    <app-child />
    <div #parentDiv>Parent element</div>
  `
})
export class ParentComponent {
  // Finds #parentDiv (in this template)
  parentDiv = viewChild<ElementRef>('parentDiv'); // FOUND

  // Finds the ChildComponent instance
  child = viewChild(ChildComponent); // FOUND

  // Does NOT find elements inside ChildComponent's template
  // Those are ChildComponent's private view
}
```

To access projected content, use `contentChild` / `contentChildren` (covered in 12.9).

## Legacy @ViewChild and @ViewChildren

### Decorator syntax

```typescript
import { ViewChild, ViewChildren, QueryList, ElementRef, AfterViewInit } from '@angular/core';

@Component({ /* ... */ })
export class LegacyComponent implements AfterViewInit {
  @ViewChild('myInput') inputRef!: ElementRef;
  @ViewChild(ChildComponent) child!: ChildComponent;
  @ViewChild('myInput', { read: ViewContainerRef }) vcr!: ViewContainerRef;
  @ViewChild('myInput', { static: true }) staticInput!: ElementRef;

  @ViewChildren(ItemComponent) items!: QueryList<ItemComponent>;

  ngAfterViewInit(): void {
    // Safe to access query results here
    this.inputRef.nativeElement.focus();
    this.items.changes.subscribe(() => {
      console.log('Items changed:', this.items.length);
    });
  }
}
```

### Comparison table

| Feature | `viewChild()` / `viewChildren()` | `@ViewChild` / `@ViewChildren` |
|---|---|---|
| Return type | `Signal<T \| undefined>` / `Signal<readonly T[]>` | `T` / `QueryList<T>` |
| Reactive | Yes (signal-based) | `QueryList.changes` observable |
| Required variant | `viewChild.required()` | Not available (use `!` assertion) |
| Lifecycle hook needed | No | Yes (`ngAfterViewInit`) |
| `static` option | Not available (always dynamic) | `{ static: true }` for early access |
| `read` option | Yes | Yes |
| Status | Current, recommended | Legacy |

### The static option (legacy only)

The decorator API had a `{ static: true }` option that made the query resolve before change detection, making it available in `ngOnInit`. Signal queries do not have this option because they are always dynamically resolved. If you need to access a DOM element immediately, use `afterNextRender`.

## Migration from Decorator Queries

### Automatic schematic

```bash
ng generate @angular/core:signal-queries-migration
```

Or as part of the combined signal migration:

```bash
ng generate @angular/core:signals
```

The schematic converts:

```typescript
// Before
@ViewChild('canvas') canvas!: ElementRef;
@ViewChild(ChartComponent) chart?: ChartComponent;
@ViewChildren(TabComponent) tabs!: QueryList<TabComponent>;

// After
canvas = viewChild.required<ElementRef>('canvas');
chart = viewChild(ChartComponent);
tabs = viewChildren(TabComponent);
```

It also updates:
- References in the class to add `()` (e.g., `this.canvas` becomes `this.canvas()`)
- `ngAfterViewInit` code that accessed query results
- `QueryList.changes` subscriptions to `effect()` patterns

### Manual migration patterns

**Single required query:**

```typescript
// Before
@ViewChild('el', { static: false }) el!: ElementRef;
ngAfterViewInit() { this.el.nativeElement.focus(); }

// After
el = viewChild.required<ElementRef>('el');
constructor() {
  afterNextRender(() => { this.el().nativeElement.focus(); });
}
```

**QueryList with changes:**

```typescript
// Before
@ViewChildren(CardComponent) cards!: QueryList<CardComponent>;
ngAfterViewInit() {
  this.cards.changes.subscribe(list => {
    console.log('Cards updated:', list.length);
  });
}

// After
cards = viewChildren(CardComponent);
constructor() {
  effect(() => {
    console.log('Cards updated:', this.cards().length);
  });
}
```

**Query with read:**

```typescript
// Before
@ViewChild('container', { read: ViewContainerRef }) vcr!: ViewContainerRef;

// After
vcr = viewChild.required('container', { read: ViewContainerRef });
```

## Common Mistakes

### 1. Using viewChild.required on conditionally rendered elements

```typescript
// BAD: Element is inside @if, may not exist
@Component({
  template: `
    @if (showEditor()) {
      <textarea #editor></textarea>
    }
  `
})
export class BadComponent {
  showEditor = signal(false);
  editor = viewChild.required<ElementRef>('editor'); // Runtime error when showEditor is false!
}

// GOOD: Use optional viewChild
@Component({ /* same template */ })
export class GoodComponent {
  showEditor = signal(false);
  editor = viewChild<ElementRef>('editor'); // Signal<ElementRef | undefined>

  focusEditor(): void {
    this.editor()?.nativeElement.focus();
  }
}
```

### 2. Reading the query in the constructor

```typescript
// BAD: Query results are not available at construction time
@Component({
  template: `<canvas #canvas></canvas>`
})
export class BadComponent {
  canvas = viewChild.required<ElementRef>('canvas');

  constructor() {
    console.log(this.canvas()); // ERROR: result not yet available
  }
}

// GOOD: Use afterNextRender for DOM access
@Component({ /* same */ })
export class GoodComponent {
  canvas = viewChild.required<ElementRef>('canvas');

  constructor() {
    afterNextRender(() => {
      console.log(this.canvas()); // Safe: DOM is rendered
    });
  }
}
```

### 3. Forgetting the read option for ViewContainerRef

```typescript
// BAD: Without read, you get ElementRef, not ViewContainerRef
outlet = viewChild.required<ViewContainerRef>('outlet');
// Actually returns ElementRef despite the generic!

// GOOD: Explicitly request ViewContainerRef
outlet = viewChild.required('outlet', { read: ViewContainerRef });
```

The generic type parameter alone does not change what Angular returns. You must use `read` to override the default.

### 4. Expecting viewChildren to deduplicate

`viewChildren` emits a new array reference whenever the query is re-evaluated, even if the content has not changed. Using the signal directly in computed or effect works fine (signals deduplicate by reference equality of the array contents), but be aware that the array reference itself is new.

### 5. Trying to query across component boundaries

```typescript
// BAD: Trying to find elements inside a child component's template
@Component({
  template: `<app-child />`
})
export class ParentComponent {
  // This finds the ChildComponent instance, not elements in its template
  childInput = viewChild<ElementRef>('someInputInChild'); // undefined!
}
```

View queries are scoped to the component's own template. You cannot reach into a child component's internal DOM.

### 6. Calling viewChild outside of a component property initializer

```typescript
// BAD: Cannot call viewChild in a method
@Component({ /* ... */ })
export class BadComponent {
  getElement(): void {
    const el = viewChild('myRef'); // ERROR: must be in property initializer
  }
}

// GOOD: Declare as a class field
@Component({ /* ... */ })
export class GoodComponent {
  el = viewChild<ElementRef>('myRef');
}
```

## Best Practices

### 1. Use viewChild.required for unconditionally rendered elements

If the element is always in the template (no `@if`, `@for`, `@switch` wrapping it), use `viewChild.required()`. This eliminates null checks and makes the code clearer.

### 2. Use afterNextRender for one-time DOM initialization

When you need to initialize a third-party library or perform DOM measurements, combine `viewChild.required` with `afterNextRender`:

```typescript
chart = viewChild.required<ElementRef>('chart');

constructor() {
  afterNextRender(() => {
    this.initChart(this.chart().nativeElement);
  });
}
```

### 3. Use effect for reactive DOM synchronization

When you need to react to query changes (e.g., the number of child components changes), use `effect()`:

```typescript
items = viewChildren(ItemComponent);

constructor() {
  effect(() => {
    const count = this.items().length;
    // Update external state or perform side effects
  });
}
```

### 4. Prefer component/directive locators over string locators

String locators (`'myRef'`) are fragile. Renaming the template reference variable requires updating the query. Component/directive class locators are refactor-safe:

```typescript
// Fragile: string must match template
input = viewChild<ElementRef>('searchInput');

// Refactor-safe: tied to the class
search = viewChild(SearchComponent);
```

### 5. Use read only when the default type is insufficient

Most queries do not need `read`. Use it only when you need `ViewContainerRef`, `TemplateRef`, or a specific directive from a multi-directive element.

### 6. Avoid querying for things that should be inputs

If you find yourself querying a child component just to read its properties, consider whether the child should expose those properties via `output()` or a service instead. Direct component querying creates tight coupling.

## Real-World Use Cases

### Auto-Resizing Textarea

```typescript
@Component({
  selector: 'app-auto-resize-textarea',
  standalone: true,
  template: `
    <textarea
      #textarea
      [value]="value()"
      (input)="onInput($event)"
      [style.height.px]="height()"
    ></textarea>
  `,
  styles: [`textarea { overflow: hidden; resize: none; }`]
})
export class AutoResizeTextareaComponent {
  value = model('');
  textarea = viewChild.required<ElementRef<HTMLTextAreaElement>>('textarea');

  height = signal(40);

  constructor() {
    // Recalculate height whenever value changes
    afterRenderEffect({
      read: () => {
        const el = this.textarea().nativeElement;
        el.style.height = 'auto'; // Reset to measure scrollHeight
        this.height.set(el.scrollHeight);
      },
    });
  }

  onInput(event: Event): void {
    this.value.set((event.target as HTMLTextAreaElement).value);
  }
}
```

### Dynamic Component Loader

```typescript
@Component({
  selector: 'app-widget-host',
  standalone: true,
  template: `
    <div class="widget-container">
      <div #widgetOutlet></div>
    </div>
  `
})
export class WidgetHostComponent {
  widgetType = input.required<Type<unknown>>();
  widgetOutlet = viewChild.required('widgetOutlet', { read: ViewContainerRef });

  private currentRef: ComponentRef<unknown> | null = null;

  constructor() {
    // React to widgetType changes: destroy old, create new
    effect(() => {
      const type = this.widgetType();
      const vcr = this.widgetOutlet();

      this.currentRef?.destroy();
      vcr.clear();
      this.currentRef = vcr.createComponent(type);
    });

    inject(DestroyRef).onDestroy(() => {
      this.currentRef?.destroy();
    });
  }
}
```

```html
<!-- Parent swaps widgets dynamically -->
<app-widget-host [widgetType]="activeWidget()" />
```

### Form with Multiple Validation Indicators

```typescript
@Component({
  selector: 'app-registration-form',
  standalone: true,
  imports: [ReactiveFormsModule, ValidationIndicatorComponent],
  template: `
    <form [formGroup]="form">
      <div class="field">
        <label>Username</label>
        <input formControlName="username" />
        <app-validation-indicator [control]="form.controls.username" />
      </div>
      <div class="field">
        <label>Email</label>
        <input formControlName="email" />
        <app-validation-indicator [control]="form.controls.email" />
      </div>
      <div class="field">
        <label>Password</label>
        <input formControlName="password" type="password" />
        <app-validation-indicator [control]="form.controls.password" />
      </div>
    </form>

    <p>Fields with errors: {{ invalidFieldCount() }}</p>
  `
})
export class RegistrationFormComponent {
  private fb = inject(FormBuilder);

  form = this.fb.group({
    username: ['', Validators.required],
    email: ['', [Validators.required, Validators.email]],
    password: ['', [Validators.required, Validators.minLength(8)]],
  });

  // Query all validation indicator components
  indicators = viewChildren(ValidationIndicatorComponent);

  invalidFieldCount = computed(() =>
    this.indicators().filter(ind => ind.hasErrors()).length
  );
}
```

---

> **Key Takeaway:** Signal view queries (`viewChild` and `viewChildren`) replace `@ViewChild` and `@ViewChildren` with reactive signals that automatically update when the DOM changes, eliminating the need for `ngAfterViewInit` and `QueryList.changes` subscriptions while providing better type safety through `viewChild.required()` and the `read` option.
