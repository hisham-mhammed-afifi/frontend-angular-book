# 12.9 Signal Content Queries

> Signal content queries (`contentChild` and `contentChildren`) let you reference elements, components, and directives that a parent projects into your component via `<ng-content>`, replacing the `@ContentChild` and `@ContentChildren` decorators with reactive signals.

---

## View Queries vs Content Queries

Before diving into content queries, it is essential to understand the distinction:

| Aspect | View Query | Content Query |
|---|---|---|
| Searches in | The component's own template | Content projected by the parent |
| API | `viewChild`, `viewChildren` | `contentChild`, `contentChildren` |
| Projection mechanism | N/A | `<ng-content>` |
| Who provides the content | The component itself | The parent using the component |

A component's **view** is the template it defines. A component's **content** is whatever the parent places between its opening and closing tags.

```html
<!-- Parent template -->
<app-card>
  <!-- Everything here is "content" for app-card -->
  <h3 card-header>Title</h3>
  <p>Body text goes here.</p>
</app-card>
```

```typescript
// app-card template (view)
@Component({
  selector: 'app-card',
  template: `
    <div class="card">
      <div class="card-header">
        <ng-content select="[card-header]" />  <!-- slot for header content -->
      </div>
      <div class="card-body">
        <ng-content />  <!-- default slot for remaining content -->
      </div>
    </div>
  `
})
export class CardComponent {
  // contentChild queries find elements projected by the parent
  // viewChild queries find elements in this template (the <div>s above)
}
```

## Content Projection Recap

Content queries only make sense in the context of content projection. Here is a quick recap of the projection mechanism.

### Single-slot projection

```typescript
@Component({
  selector: 'app-panel',
  standalone: true,
  template: `
    <div class="panel">
      <ng-content />  <!-- All projected content goes here -->
    </div>
  `
})
export class PanelComponent {}
```

```html
<app-panel>
  <p>This paragraph is projected into the panel.</p>
</app-panel>
```

### Multi-slot projection with select

```typescript
@Component({
  selector: 'app-dialog',
  standalone: true,
  template: `
    <div class="dialog">
      <header>
        <ng-content select="[dialog-title]" />
      </header>
      <div class="body">
        <ng-content />  <!-- Default slot: everything not matching a select -->
      </div>
      <footer>
        <ng-content select="[dialog-actions]" />
      </footer>
    </div>
  `
})
export class DialogComponent {}
```

```html
<app-dialog>
  <h2 dialog-title>Confirm Delete</h2>
  <p>Are you sure you want to delete this item?</p>
  <div dialog-actions>
    <button>Cancel</button>
    <button>Delete</button>
  </div>
</app-dialog>
```

The `select` attribute accepts CSS selectors: element names, attribute selectors (`[attr]`), class selectors (`.class`), and combinations. Content that does not match any `select` falls into the default `<ng-content />` slot.

## contentChild - Single Projected Element

`contentChild()` returns a signal containing the first matching projected element. If no match is found, the signal's value is `undefined`.

### By Directive or Component Type

```typescript
@Component({
  selector: 'app-tab-group',
  standalone: true,
  template: `
    <div class="tab-headers">
      @if (activeTab(); as tab) {
        <span class="active-label">{{ tab.label() }}</span>
      }
    </div>
    <div class="tab-content">
      <ng-content />
    </div>
  `
})
export class TabGroupComponent {
  // Find the first projected TabComponent
  activeTab = contentChild(TabComponent);
  // Type: Signal<TabComponent | undefined>
}
```

```html
<!-- Parent projects TabComponent instances -->
<app-tab-group>
  <app-tab label="Overview">Overview content</app-tab>
  <app-tab label="Details">Details content</app-tab>
</app-tab-group>
```

### By Template Reference Variable

```typescript
@Component({
  selector: 'app-tooltip-wrapper',
  standalone: true,
  template: `
    <div class="wrapper" (mouseenter)="show()" (mouseleave)="hide()">
      <ng-content />
      @if (visible()) {
        <div class="tooltip-popup">
          <ng-content select="[tooltip-content]" />
        </div>
      }
    </div>
  `
})
export class TooltipWrapperComponent {
  // Query projected content by template reference variable
  tooltipContent = contentChild<ElementRef>('tooltipRef');
  // Type: Signal<ElementRef | undefined>

  visible = signal(false);
  show(): void { this.visible.set(true); }
  hide(): void { this.visible.set(false); }
}
```

```html
<app-tooltip-wrapper>
  <button>Hover me</button>
  <span tooltip-content #tooltipRef>Helpful information</span>
</app-tooltip-wrapper>
```

### contentChild.required

When the component requires specific projected content to function, use `contentChild.required()`:

```typescript
@Component({
  selector: 'app-sortable-list',
  standalone: true,
  template: `
    <div class="list">
      <ng-content />
    </div>
    <p>{{ items().length }} items</p>
  `
})
export class SortableListComponent {
  // At least one SortableItemDirective must be projected
  firstItem = contentChild.required(SortableItemDirective);
  // Type: Signal<SortableItemDirective> (no undefined)
}
```

If the parent does not project any matching content, Angular throws a runtime error.

## contentChildren - Multiple Projected Elements

`contentChildren()` returns a signal containing a read-only array of all matching projected elements.

```typescript
@Directive({
  selector: '[appMenuItem]',
  standalone: true,
})
export class MenuItemDirective {
  label = input.required<string>();
  icon = input<string>();
  disabled = input(false, { transform: booleanAttribute });
}

@Component({
  selector: 'app-menu',
  standalone: true,
  template: `
    <nav class="menu" role="menu">
      @for (item of menuItems(); track item.label()) {
        <button
          role="menuitem"
          [disabled]="item.disabled()"
          (click)="itemClicked.emit(item.label())"
        >
          @if (item.icon()) {
            <span class="icon">{{ item.icon() }}</span>
          }
          {{ item.label() }}
        </button>
      } @empty {
        <p>No menu items.</p>
      }
    </nav>
  `
})
export class MenuComponent {
  // Query all projected elements with the appMenuItem directive
  menuItems = contentChildren(MenuItemDirective);
  // Type: Signal<readonly MenuItemDirective[]>

  itemClicked = output<string>();

  enabledCount = computed(() =>
    this.menuItems().filter(item => !item.disabled()).length
  );
}
```

```html
<!-- Parent projects menu items -->
<app-menu (itemClicked)="onMenuAction($event)">
  <button appMenuItem label="New File" icon="+" />
  <button appMenuItem label="Open" icon="O" />
  <button appMenuItem label="Save" icon="S" />
  <button appMenuItem label="Delete" icon="X" disabled />
</app-menu>
```

### Reactivity with dynamic content

When the parent's projected content changes (items added, removed, or reordered), the `contentChildren` signal automatically updates:

```typescript
@Component({
  selector: 'app-chip-list',
  standalone: true,
  template: `
    <div class="chips">
      <ng-content />
    </div>
    <p>{{ chipCount() }} chips</p>
  `
})
export class ChipListComponent {
  chips = contentChildren(ChipComponent);
  chipCount = computed(() => this.chips().length);
}
```

```html
<!-- Parent dynamically adds/removes chips -->
<app-chip-list>
  @for (tag of selectedTags(); track tag) {
    <app-chip [label]="tag" (removed)="removeTag(tag)" />
  }
</app-chip-list>
```

As `selectedTags` changes, `@for` updates the projected content, and `chipCount` reactively updates.

## The descendants Option

This is a critical difference between `contentChild` and `contentChildren`:

| Function | Default `descendants` | Behavior |
|---|---|---|
| `contentChild()` | `true` | Searches all nested levels of projected content |
| `contentChildren()` | `false` | Searches only direct children of the host element |

### contentChildren with descendants: false (default)

```html
<!-- Parent template -->
<app-list>
  <app-item label="Direct child 1" />      <!-- FOUND (direct child) -->
  <app-item label="Direct child 2" />      <!-- FOUND (direct child) -->
  <div>
    <app-item label="Nested child" />       <!-- NOT found (nested in a div) -->
  </div>
</app-list>
```

```typescript
@Component({ /* ... */ })
export class ListComponent {
  // Default: descendants = false, only finds direct children
  items = contentChildren(ItemComponent);
  // Result: [Direct child 1, Direct child 2]
}
```

### contentChildren with descendants: true

```typescript
@Component({ /* ... */ })
export class ListComponent {
  // Opt-in: search all nested levels
  items = contentChildren(ItemComponent, { descendants: true });
  // Result: [Direct child 1, Direct child 2, Nested child]
}
```

### Why the default matters

The `descendants: false` default prevents accidentally capturing elements that are projected into deeper nested components. Consider:

```html
<app-outer-list>
  <app-item label="Outer item" />
  <app-inner-list>
    <app-item label="Inner item" />
  </app-inner-list>
</app-outer-list>
```

With `descendants: false`, `app-outer-list`'s `contentChildren(ItemComponent)` finds only "Outer item". The "Inner item" belongs to `app-inner-list`. With `descendants: true`, the outer list would also capture the inner item, which is almost certainly undesirable.

**Rule of thumb**: Use `descendants: true` only when you intentionally want to search through wrapper elements like `<div>`, `<section>`, or plain HTML containers that the parent uses for layout.

## The read Option

Like view queries, content queries support `read` to override the default return type.

```typescript
@Component({
  selector: 'app-carousel',
  standalone: true,
  template: `
    <div class="slides">
      <ng-content />
    </div>
  `
})
export class CarouselComponent {
  // Default: returns SlideComponent instances
  slides = contentChildren(SlideComponent);

  // With read: ElementRef: returns host DOM elements of SlideComponent
  slideElements = contentChildren(SlideComponent, { read: ElementRef });
  // Type: Signal<readonly ElementRef[]>

  // With read: TemplateRef from a projected ng-template
  customTemplate = contentChild('slideTpl', { read: TemplateRef });
  // Type: Signal<TemplateRef<unknown> | undefined>
}
```

Common `read` values:

| `read` Value | Returns | Use Case |
|---|---|---|
| (default) | Component/directive instance or `ElementRef` | Access component API or DOM |
| `ElementRef` | `ElementRef` wrapping the DOM element | Need raw DOM access on a component |
| `TemplateRef` | `TemplateRef` from `<ng-template>` | Dynamic template rendering |
| `SomeDirective` | Directive instance | Access a specific directive |

## Content Queries with ng-template

A powerful pattern combines content projection with `<ng-template>` to let the parent provide a custom rendering template:

```typescript
@Component({
  selector: 'app-data-list',
  standalone: true,
  template: `
    @for (item of items(); track item.id) {
      @if (itemTemplate()) {
        <ng-container
          [ngTemplateOutlet]="itemTemplate()!"
          [ngTemplateOutletContext]="{ $implicit: item, index: $index }"
        />
      } @else {
        <div class="default-item">{{ item.name }}</div>
      }
    } @empty {
      <p>No items.</p>
    }
  `,
  imports: [NgTemplateOutlet]
})
export class DataListComponent {
  items = input.required<{ id: number; name: string }[]>();

  // Query a projected ng-template
  itemTemplate = contentChild<TemplateRef<unknown>>('itemTpl');
  // Type: Signal<TemplateRef<unknown> | undefined>
}
```

```html
<!-- Parent provides a custom template -->
<app-data-list [items]="users()">
  <ng-template #itemTpl let-user let-i="index">
    <div class="custom-row">
      {{ i + 1 }}. {{ user.name }} (custom rendering)
    </div>
  </ng-template>
</app-data-list>

<!-- Or use the default rendering -->
<app-data-list [items]="users()" />
```

This pattern is widely used in component libraries for tables, lists, and grids where the consumer needs control over how each item is rendered.

## Combining Content Queries with Signals

Content queries integrate naturally with the signal ecosystem:

```typescript
@Component({
  selector: 'app-stepper',
  standalone: true,
  template: `
    <div class="step-indicators">
      @for (step of steps(); track step.label(); let i = $index) {
        <button
          [class.active]="i === activeIndex()"
          [class.completed]="i < activeIndex()"
          (click)="activeIndex.set(i)"
        >
          {{ i + 1 }}. {{ step.label() }}
        </button>
      }
    </div>
    <div class="step-body">
      <ng-content />
    </div>
    <div class="navigation">
      <button [disabled]="isFirst()" (click)="prev()">Previous</button>
      <button [disabled]="isLast()" (click)="next()">Next</button>
    </div>
  `
})
export class StepperComponent {
  steps = contentChildren(StepComponent);
  activeIndex = signal(0);

  isFirst = computed(() => this.activeIndex() === 0);
  isLast = computed(() => this.activeIndex() >= this.steps().length - 1);

  next(): void {
    if (!this.isLast()) {
      this.activeIndex.update(i => i + 1);
    }
  }

  prev(): void {
    if (!this.isFirst()) {
      this.activeIndex.update(i => i - 1);
    }
  }
}
```

```html
<app-stepper>
  <app-step label="Personal Info">...</app-step>
  <app-step label="Address">...</app-step>
  <app-step label="Payment">...</app-step>
</app-stepper>
```

The `steps` signal updates if the parent adds or removes `<app-step>` elements dynamically. The `isFirst` and `isLast` computed signals react to both `activeIndex` and `steps.length`.

## Legacy @ContentChild and @ContentChildren

### Decorator syntax

```typescript
import {
  ContentChild, ContentChildren, QueryList,
  AfterContentInit, ElementRef
} from '@angular/core';

@Component({ /* ... */ })
export class LegacyComponent implements AfterContentInit {
  @ContentChild(HeaderDirective) header?: HeaderDirective;
  @ContentChild('footerRef') footer?: ElementRef;
  @ContentChild(HeaderDirective, { read: ElementRef }) headerEl?: ElementRef;

  @ContentChildren(TabComponent) tabs!: QueryList<TabComponent>;
  @ContentChildren(ItemComponent, { descendants: true }) allItems!: QueryList<ItemComponent>;

  ngAfterContentInit(): void {
    // Safe to access content query results here
    console.log('Tabs:', this.tabs.length);
    this.tabs.changes.subscribe(() => {
      console.log('Tabs changed:', this.tabs.length);
    });
  }
}
```

### Comparison table

| Feature | `contentChild()` / `contentChildren()` | `@ContentChild` / `@ContentChildren` |
|---|---|---|
| Return type | `Signal<T \| undefined>` / `Signal<readonly T[]>` | `T` / `QueryList<T>` |
| Reactive | Yes (signal-based) | `QueryList.changes` observable |
| Required variant | `contentChild.required()` | Not available |
| Lifecycle hook needed | No | Yes (`ngAfterContentInit`) |
| `descendants` default | `contentChild`: true, `contentChildren`: false | Same |
| `read` option | Yes | Yes |
| `static` option | Not available | `{ static: true }` |
| Status | Current, recommended | Legacy |

### Migration patterns

**Single content query:**

```typescript
// Before
@ContentChild(HeaderDirective) header?: HeaderDirective;
ngAfterContentInit() {
  if (this.header) { this.processHeader(this.header); }
}

// After
header = contentChild(HeaderDirective);
constructor() {
  effect(() => {
    const h = this.header();
    if (h) { this.processHeader(h); }
  });
}
```

**QueryList with changes:**

```typescript
// Before
@ContentChildren(TabComponent) tabs!: QueryList<TabComponent>;
ngAfterContentInit() {
  this.tabs.changes.subscribe(list => this.onTabsChanged(list.toArray()));
}

// After
tabs = contentChildren(TabComponent);
constructor() {
  effect(() => {
    this.onTabsChanged([...this.tabs()]);
  });
}
```

## Common Mistakes

### 1. Confusing viewChild with contentChild

```typescript
@Component({
  selector: 'app-wrapper',
  standalone: true,
  template: `
    <div class="wrapper">
      <ng-content />  <!-- Projected content goes here -->
    </div>
  `
})
export class WrapperComponent {
  // BAD: viewChild looks in this component's template, not projected content
  projected = viewChild(SomeComponent); // Will be undefined!

  // GOOD: contentChild looks in projected content
  projected = contentChild(SomeComponent);
}
```

`viewChild` searches the component's own template (the `<div>` and `<ng-content>`). `contentChild` searches what the parent projects into `<ng-content>`.

### 2. Expecting contentChildren to find nested elements by default

```html
<app-list>
  <div class="group">
    <app-item />  <!-- NOT found with default descendants: false -->
  </div>
</app-list>
```

```typescript
// BAD: Misunderstanding the default
items = contentChildren(ItemComponent); // Empty array

// GOOD: Set descendants: true if wrapping elements are expected
items = contentChildren(ItemComponent, { descendants: true });
```

### 3. Using contentChild.required when content is optional

```typescript
// BAD: Throws if parent doesn't project a footer
footer = contentChild.required(FooterComponent);

// GOOD: Optional is safer for content that may not be provided
footer = contentChild(FooterComponent);
```

Unlike view queries where the component controls its own template, content queries depend on what the parent decides to project. Required content queries should only be used when the component truly cannot function without the projected content, and this constraint should be documented.

### 4. Querying for elements that are not projected

Content queries only find elements that the parent places between the component's tags. They do not find:
- Elements in the component's own template (use `viewChild` for those)
- Elements in a parent's template that are not projected
- Elements in sibling components

### 5. Forgetting that ng-content is not a real DOM element

```typescript
// BAD: Trying to query ng-content itself
ngContent = viewChild<ElementRef>('ng-content'); // Does not work

// ng-content is a compile-time placeholder, not a DOM element.
// Query the projected content instead, or wrap it in a container div.
```

### 6. Not documenting required projected content

When a component relies on specific projected content (e.g., a header directive or a template reference), failing to document this creates a poor developer experience. The parent has no compile-time indication of what to project.

## Best Practices

### 1. Use directives as content query selectors

Attribute directives provide the cleanest API for multi-slot content projection:

```typescript
// Define slot directives
@Directive({ selector: '[cardHeader]', standalone: true })
export class CardHeaderDirective {}

@Directive({ selector: '[cardFooter]', standalone: true })
export class CardFooterDirective {}

// Query by directive type
@Component({
  selector: 'app-card',
  standalone: true,
  template: `
    <div class="card">
      @if (header()) { <div class="header"><ng-content select="[cardHeader]" /></div> }
      <div class="body"><ng-content /></div>
      @if (footer()) { <div class="footer"><ng-content select="[cardFooter]" /></div> }
    </div>
  `
})
export class CardComponent {
  header = contentChild(CardHeaderDirective);
  footer = contentChild(CardFooterDirective);
}
```

```html
<app-card>
  <h3 cardHeader>Card Title</h3>
  <p>Card body content.</p>
  <div cardFooter>
    <button>Action</button>
  </div>
</app-card>
```

### 2. Use contentChild for optional slots, contentChild.required for mandatory ones

If the component degrades gracefully without certain content (like an optional footer), use `contentChild()`. If the component is broken without it, use `contentChild.required()` and document the requirement.

### 3. Derive state from contentChildren with computed

```typescript
tabs = contentChildren(TabComponent);

hasMultipleTabs = computed(() => this.tabs().length > 1);
activeTab = computed(() => this.tabs().find(t => t.active()) ?? this.tabs()[0]);
```

### 4. Export directives alongside the host component

When a component uses directive-based content queries, export the directives from the same file or barrel export so consumers import everything together:

```typescript
// card.component.ts
export { CardComponent, CardHeaderDirective, CardFooterDirective };
```

### 5. Use descendants: true only when needed

Keep the default (`false` for `contentChildren`) unless you explicitly expect wrapper elements. This prevents accidental capture of deeply nested content.

### 6. Prefer ng-template projection for customizable rendering

When the consumer needs to control how items are rendered, query a projected `<ng-template>` rather than forcing a specific component structure.

## Real-World Use Cases

### Accordion with Projected Panels

```typescript
@Directive({ selector: 'app-accordion-panel', standalone: true })
export class AccordionPanelDirective {
  title = input.required<string>();
  expanded = signal(false);

  toggle(): void {
    this.expanded.update(v => !v);
  }
}

@Component({
  selector: 'app-accordion',
  standalone: true,
  template: `
    <div class="accordion">
      @for (panel of panels(); track panel.title()) {
        <div class="panel" [class.open]="panel.expanded()">
          <button class="panel-header" (click)="togglePanel(panel)">
            {{ panel.title() }}
            <span>{{ panel.expanded() ? '-' : '+' }}</span>
          </button>
          @if (panel.expanded()) {
            <div class="panel-body">
              <ng-content />
            </div>
          }
        </div>
      }
    </div>
  `
})
export class AccordionComponent {
  panels = contentChildren(AccordionPanelDirective);
  multiOpen = input(false);

  togglePanel(panel: AccordionPanelDirective): void {
    if (!this.multiOpen()) {
      // Close all other panels
      this.panels().forEach(p => {
        if (p !== panel) p.expanded.set(false);
      });
    }
    panel.toggle();
  }
}
```

```html
<app-accordion>
  <app-accordion-panel title="Section 1">
    <p>Content for section 1.</p>
  </app-accordion-panel>
  <app-accordion-panel title="Section 2">
    <p>Content for section 2.</p>
  </app-accordion-panel>
  <app-accordion-panel title="Section 3">
    <p>Content for section 3.</p>
  </app-accordion-panel>
</app-accordion>
```

### Data Table with Custom Column Templates

```typescript
@Directive({ selector: '[columnDef]', standalone: true })
export class ColumnDefDirective {
  columnDef = input.required<string>();
  header = input.required<string>();
  cellTemplate = contentChild.required('cell', { read: TemplateRef });
}

@Component({
  selector: 'app-table',
  standalone: true,
  imports: [NgTemplateOutlet],
  template: `
    <table>
      <thead>
        <tr>
          @for (col of columns(); track col.columnDef()) {
            <th>{{ col.header() }}</th>
          }
        </tr>
      </thead>
      <tbody>
        @for (row of data(); track trackBy()(row)) {
          <tr>
            @for (col of columns(); track col.columnDef()) {
              <td>
                <ng-container
                  [ngTemplateOutlet]="col.cellTemplate()"
                  [ngTemplateOutletContext]="{ $implicit: row }"
                />
              </td>
            }
          </tr>
        } @empty {
          <tr>
            <td [attr.colspan]="columns().length">No data</td>
          </tr>
        }
      </tbody>
    </table>
  `
})
export class TableComponent<T> {
  data = input.required<T[]>();
  trackBy = input<(item: T) => unknown>(() => (item: T) => item);

  columns = contentChildren(ColumnDefDirective);
}
```

```html
<app-table [data]="users()" [trackBy]="trackByUserId">
  <div columnDef="name" header="Name">
    <ng-template #cell let-user>
      <strong>{{ user.name }}</strong>
    </ng-template>
  </div>
  <div columnDef="email" header="Email">
    <ng-template #cell let-user>
      <a [href]="'mailto:' + user.email">{{ user.email }}</a>
    </ng-template>
  </div>
  <div columnDef="role" header="Role">
    <ng-template #cell let-user>
      <span class="badge">{{ user.role }}</span>
    </ng-template>
  </div>
</app-table>
```

### Toolbar with Projected Actions

```typescript
@Directive({ selector: '[toolbarAction]', standalone: true })
export class ToolbarActionDirective {
  label = input.required<string>();
  icon = input<string>();
  position = input<'left' | 'right'>('left');
}

@Component({
  selector: 'app-toolbar',
  standalone: true,
  template: `
    <div class="toolbar">
      <div class="left">
        @for (action of leftActions(); track action.label()) {
          <ng-content select="[toolbarAction][position='left']" />
        }
      </div>
      <div class="title"><ng-content select="[toolbar-title]" /></div>
      <div class="right">
        @for (action of rightActions(); track action.label()) {
          <ng-content select="[toolbarAction][position='right']" />
        }
      </div>
    </div>
  `
})
export class ToolbarComponent {
  allActions = contentChildren(ToolbarActionDirective);

  leftActions = computed(() =>
    this.allActions().filter(a => a.position() === 'left')
  );

  rightActions = computed(() =>
    this.allActions().filter(a => a.position() === 'right')
  );

  actionCount = computed(() => this.allActions().length);
}
```

```html
<app-toolbar>
  <span toolbar-title>Dashboard</span>
  <button toolbarAction label="Refresh" icon="refresh" position="left">Refresh</button>
  <button toolbarAction label="Settings" icon="gear" position="right">Settings</button>
  <button toolbarAction label="Help" icon="help" position="right">Help</button>
</app-toolbar>
```

---

> **Key Takeaway:** Signal content queries (`contentChild` and `contentChildren`) let you reactively reference elements projected by the parent through `<ng-content>`, with the critical distinction that `contentChildren` defaults to `descendants: false` (only direct children) while `contentChild` defaults to `descendants: true`, and both integrate seamlessly with `computed` and `effect` to build powerful component composition APIs.
