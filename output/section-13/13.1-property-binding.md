# 13.1 Property Binding

> Property binding connects a component expression to a DOM property, keeping the rendered element in sync with your data. It is the primary mechanism for passing values from a component into the template and into child components.

---

## The `[property]` Syntax

Property binding uses square brackets around the target property name:

```html
<img [src]="imageUrl()" [alt]="imageAlt()" />
<input [value]="username()" [placeholder]="hint()" />
<button [disabled]="isSubmitting()">Save</button>
<div [hidden]="!isVisible()">Content</div>
```

The expression on the right side is evaluated against the component instance. When the expression value changes, Angular updates the DOM property.

```typescript
@Component({
  selector: 'app-profile',
  standalone: true,
  template: `
    <img [src]="avatarUrl()" [alt]="displayName()" />
    <h2 [textContent]="displayName()"></h2>
    <p [innerHTML]="bio()"></p>
    <a [href]="profileLink()">View Full Profile</a>
  `
})
export class ProfileComponent {
  avatarUrl = input<string>('/assets/default-avatar.png');
  displayName = input.required<string>();
  bio = input<string>('');
  profileLink = input<string>('#');
}
```

---

## DOM Properties vs HTML Attributes

This distinction is the most important concept to understand about property binding. Angular binds to DOM properties, not HTML attributes.

### What Is the Difference?

HTML attributes are defined in the HTML source. They initialize the DOM when the browser parses the HTML. Once the DOM is created, attributes serve as the initial values for the corresponding DOM properties.

DOM properties are the live, in-memory representation of the element. They can change at runtime. Angular manipulates DOM properties.

```html
<!-- HTML source -->
<input type="text" value="initial">
```

When the browser parses this:
1. It creates an `HTMLInputElement` DOM node.
2. Sets the DOM property `input.type = "text"`.
3. Sets the DOM property `input.value = "initial"`.
4. The HTML attribute `value="initial"` stays frozen in the source.

If the user types "hello" into the input:
- `input.value` (DOM property) is now `"hello"`.
- `input.getAttribute('value')` (HTML attribute) is still `"initial"`.

Angular's `[value]="expression"` sets the DOM property, not the attribute. This is why changes are reflected immediately in the UI.

### Practical Implications

```typescript
@Component({
  selector: 'app-input-demo',
  standalone: true,
  template: `
    <!-- This sets the DOM property "value", which is what the user sees -->
    <input [value]="currentValue()" />

    <!-- This sets the DOM property "disabled" -->
    <button [disabled]="isLoading()">Submit</button>

    <!-- This sets the DOM property "hidden" -->
    <div [hidden]="!showDetails()">Details here</div>
  `
})
export class InputDemoComponent {
  currentValue = signal('Hello');
  isLoading = signal(false);
  showDetails = signal(true);
}
```

### The Mapping Table

| HTML Attribute | DOM Property | Notes |
|---|---|---|
| `class` | `className` | Also accessible via `classList` |
| `for` | `htmlFor` | On `<label>` elements |
| `tabindex` | `tabIndex` | Capital I in the property |
| `readonly` | `readOnly` | Capital O in the property |
| `maxlength` | `maxLength` | Capital L in the property |
| `colspan` | `colSpan` | Capital S in the property |
| `rowspan` | `rowSpan` | Capital S in the property |
| `value` | `value` | 1:1 mapping |
| `src` | `src` | 1:1 mapping |
| `href` | `href` | 1:1 mapping |
| `disabled` | `disabled` | Boolean property |
| `checked` | `checked` | Boolean property |

Some attributes have no corresponding DOM property (like `colspan` in older spec interpretations, ARIA attributes, custom `data-*` attributes). For these, use attribute binding with the `attr.` prefix.

## Attribute Binding with `attr.`

When there is no DOM property for what you want to set, use attribute binding:

```html
<table>
  <tr>
    <!-- colspan has a DOM property (colSpan), but attr. also works -->
    <td [attr.colspan]="columnSpan()">Merged Cell</td>
  </tr>
</table>

<!-- ARIA attributes have no DOM properties -->
<button
  [attr.aria-label]="buttonLabel()"
  [attr.aria-expanded]="isExpanded()"
  [attr.aria-controls]="panelId()"
>
  Toggle
</button>

<!-- Role attribute -->
<ul [attr.role]="listRole()">
  <li [attr.role]="itemRole()">Item</li>
</ul>

<!-- Custom data attributes -->
<div
  [attr.data-testid]="testId()"
  [attr.data-category]="category()"
>
  Content
</div>

<!-- SVG attributes that have no DOM property -->
<svg>
  <circle
    [attr.cx]="x()"
    [attr.cy]="y()"
    [attr.r]="radius()"
    [attr.fill]="color()"
  />
</svg>
```

### Null Removes the Attribute

When an attribute binding expression evaluates to `null` or `undefined`, Angular removes the attribute from the DOM entirely using `removeAttribute()`. This is useful for conditional attributes.

```html
<!-- If ariaLabel() returns null, the aria-label attribute is removed -->
<button [attr.aria-label]="ariaLabel()">Click</button>

<!-- Conditionally add a data attribute -->
<div [attr.data-active]="isActive() ? '' : null">
  <!-- data-active is present when active, completely removed when not -->
</div>
```

This behavior is different from property binding, where setting a property to `null` just assigns `null` to the property.

## Interpolation vs Property Binding

Interpolation (`{{ }}`) and property binding (`[ ]`) often appear interchangeable, but they behave differently.

### String-Only vs Any Type

Interpolation always produces a string. Property binding preserves the original type.

```html
<!-- Interpolation: "42" (string) -->
<app-counter count="{{ total() }}"></app-counter>

<!-- Property binding: 42 (number) -->
<app-counter [count]="total()"></app-counter>
```

If the child component expects a number input, interpolation passes a string, which may cause type errors or unexpected behavior.

```typescript
// Child component
export class CounterComponent {
  count = input.required<number>();
  // With interpolation: count() is "42" (string) - type mismatch!
  // With property binding: count() is 42 (number) - correct
}
```

### Boolean Values

This is where the distinction matters most:

```html
<!-- BAD PRACTICE: Interpolation for boolean -->
<button disabled="{{ isDisabled() }}">Save</button>
<!-- This sets disabled="true" or disabled="false" -->
<!-- But in HTML, the mere PRESENCE of "disabled" disables the button -->
<!-- So disabled="false" STILL disables the button! -->

<!-- CORRECT: Property binding for boolean -->
<button [disabled]="isDisabled()">Save</button>
<!-- This sets the DOM property to true or false -->
<!-- false actually removes the disabled state -->
```

The HTML spec says that the `disabled` attribute is a boolean attribute: its presence means "disabled," regardless of its value. So `disabled="false"` still disables the element. Property binding sets the DOM property, which correctly handles `true` and `false`.

### When Interpolation Is Appropriate

Use interpolation for inserting text into the DOM:

```html
<!-- Text content: interpolation is natural -->
<h1>{{ title() }}</h1>
<p>Welcome, {{ user().name }}</p>
<span>{{ items().length }} items</span>

<!-- String attributes where the value is always a string -->
<img src="{{ imageUrl() }}" alt="{{ imageAlt() }}" />
<!-- Equivalent and also fine: -->
<img [src]="imageUrl()" [alt]="imageAlt()" />
```

For string-valued attributes like `src`, `alt`, `href`, `placeholder`, both work identically. The convention is to use property binding for consistency.

### Cannot Mix Both

```html
<!-- BAD: Cannot combine interpolation and property binding on the same attribute -->
<img [src]="{{ imageUrl() }}" />  <!-- Error -->

<!-- Pick one or the other -->
<img [src]="imageUrl()" />
<img src="{{ imageUrl() }}" />
```

## Binding to Component Inputs

The most common use of property binding is passing data to child components via their inputs:

```typescript
// Child component
@Component({
  selector: 'app-user-card',
  standalone: true,
  template: `
    <div class="card">
      <img [src]="avatarUrl()" [alt]="name()" />
      <h3>{{ name() }}</h3>
      <p>{{ role() }}</p>
      @if (showBadge()) {
        <span class="badge">{{ badgeText() }}</span>
      }
    </div>
  `
})
export class UserCardComponent {
  name = input.required<string>();
  role = input<string>('Member');
  avatarUrl = input<string>('/assets/default.png');
  showBadge = input(false);
  badgeText = input('');
}
```

```html
<!-- Parent template -->
@for (user of users(); track user.id) {
  <app-user-card
    [name]="user.name"
    [role]="user.role"
    [avatarUrl]="user.avatar"
    [showBadge]="user.isAdmin"
    [badgeText]="user.isAdmin ? 'Admin' : ''"
  />
}
```

### Static vs Dynamic Values

When the value never changes, you can use a plain attribute (no brackets):

```html
<!-- Static value: no brackets needed -->
<app-icon name="settings" />
<app-button variant="primary" />
<app-panel title="User Settings" />

<!-- Dynamic value: brackets required -->
<app-icon [name]="iconName()" />
<app-button [variant]="buttonVariant()" />
<app-panel [title]="panelTitle()" />
```

Without brackets, Angular treats the value as a literal string. With brackets, Angular evaluates it as an expression.

```html
<!-- These are different! -->
<app-counter count="42" />      <!-- count receives the STRING "42" -->
<app-counter [count]="42" />    <!-- count receives the NUMBER 42 -->
```

For string inputs, both are equivalent:

```html
<app-icon name="settings" />          <!-- String literal -->
<app-icon [name]="'settings'" />      <!-- Expression that evaluates to a string -->
<!-- Both pass the string "settings" to the name input -->
```

## Binding to Native Element Properties

### Common DOM Properties

```html
<!-- Text content -->
<p [textContent]="description()"></p>

<!-- Inner HTML (sanitized by Angular) -->
<div [innerHTML]="htmlContent()"></div>

<!-- Hidden -->
<div [hidden]="isCollapsed()">Collapsible content</div>

<!-- Tab index -->
<div [tabIndex]="isFocusable() ? 0 : -1">Interactive element</div>

<!-- Input properties -->
<input
  [type]="fieldType()"
  [value]="fieldValue()"
  [placeholder]="fieldPlaceholder()"
  [readOnly]="isReadOnly()"
  [maxLength]="maxChars()"
/>

<!-- Select -->
<select [value]="selectedOption()">
  @for (opt of options(); track opt.value) {
    <option [value]="opt.value">{{ opt.label }}</option>
  }
</select>

<!-- Anchor -->
<a [href]="link()" [target]="openInNewTab() ? '_blank' : '_self'">
  {{ linkText() }}
</a>
```

### Boolean DOM Properties

Several DOM properties are boolean. Property binding handles them correctly:

```html
<input [disabled]="isDisabled()" />
<input [readOnly]="isReadOnly()" />
<input [required]="isRequired()" />
<input [checked]="isChecked()" />
<details [open]="isOpen()">
  <summary>More info</summary>
  <p>Detail content</p>
</details>
<video [controls]="showControls()" [autoplay]="shouldAutoplay()" [muted]="isMuted()">
  <source [src]="videoUrl()" type="video/mp4" />
</video>
```

When the expression is `false`, Angular sets the property to `false`, which the browser interprets as "not disabled/required/checked." This is the correct behavior. Interpolation cannot do this.

## Binding to `innerHTML` and Security

Angular sanitizes values bound to `innerHTML` to prevent XSS attacks:

```typescript
@Component({
  selector: 'app-article',
  standalone: true,
  template: `
    <!-- Angular sanitizes this to remove dangerous tags/attributes -->
    <div [innerHTML]="articleHtml()"></div>
  `
})
export class ArticleComponent {
  articleHtml = input<string>('');
}
```

Angular's built-in sanitizer strips:
- `<script>` tags.
- Event handler attributes (`onclick`, `onerror`, etc.).
- `javascript:` URLs.
- `<style>` tags and `style` attributes with certain patterns.

If you need to bypass sanitization (for trusted content only), use `DomSanitizer`:

```typescript
import { DomSanitizer, SafeHtml } from '@angular/platform-browser';

export class TrustedContentComponent {
  private sanitizer = inject(DomSanitizer);
  rawHtml = input.required<string>();

  trustedHtml = computed<SafeHtml>(() =>
    this.sanitizer.bypassSecurityTrustHtml(this.rawHtml())
  );
}
```

```html
<div [innerHTML]="trustedHtml()"></div>
```

**Warning:** Only bypass sanitization for content you fully control. Never bypass for user-generated content.

## One-Way Data Flow

Property binding is one-way: from component to template (and from parent to child). The template cannot write back to the component through property binding.

```typescript
@Component({
  selector: 'app-demo',
  standalone: true,
  template: `
    <!-- This sets the input's value property -->
    <input [value]="name()" />

    <!-- If the user types in the input, name() does NOT update -->
    <!-- You need event binding to capture changes -->
    <input [value]="name()" (input)="name.set($event.target.value)" />
  `
})
export class DemoComponent {
  name = signal('Alice');
}
```

This is intentional. Angular enforces unidirectional data flow:
1. Data flows down through property bindings (parent to child, component to template).
2. Events flow up through event bindings (template to component, child to parent).

For two-way binding, Angular provides `[()]` (banana-in-a-box) syntax, which is covered separately.

---

## Binding to `class` and `style`

While class and style binding are covered in detail in 13.2, the property binding mechanism underlies them:

```html
<!-- Single class toggle -->
<div [class.active]="isActive()">Tab</div>

<!-- Single style property -->
<div [style.color]="textColor()">Colored text</div>
<div [style.width.px]="widthInPixels()">Sized box</div>

<!-- Entire className property -->
<div [className]="dynamicClasses()">Styled element</div>
```

These use the same `[property]` syntax but target specific sub-properties of the DOM element.

## Performance Considerations

### Expression Evaluation Frequency

Property binding expressions are evaluated on every change detection cycle. Keep them simple:

```html
<!-- GOOD: Simple signal read -->
<p [textContent]="title()"></p>

<!-- GOOD: Simple property access -->
<img [src]="user().avatar" />

<!-- BAD PRACTICE: Expensive computation in binding -->
<div [innerHTML]="markdownToHtml(content())"></div>
<!-- markdownToHtml runs on EVERY change detection cycle -->
```

Move expensive operations to computed signals:

```typescript
export class ArticleComponent {
  content = input.required<string>();

  // Computed only re-evaluates when content() changes
  renderedHtml = computed(() => this.markdownToHtml(this.content()));

  private markdownToHtml(md: string): string {
    // expensive conversion
    return converted;
  }
}
```

```html
<div [innerHTML]="renderedHtml()"></div>
```

### Signal-Based Change Detection

With signals, Angular knows exactly which bindings need to update. When you use `[property]="someSignal()"`, Angular tracks the signal dependency and only re-evaluates that specific binding when the signal changes. This is more efficient than the traditional zone-based approach where every binding was checked on every cycle.

## Common Mistakes

- **Mistake 1: Using interpolation for non-string values.**
  `count="{{ total() }}"` passes the string `"42"`, not the number `42`. Use `[count]="total()"` to preserve the type. This is especially important for boolean properties where `disabled="false"` still disables the element.

- **Mistake 2: Forgetting the brackets for dynamic values.**
  Writing `<app-widget count="total()"` passes the literal string `"total()"`, not the evaluated expression. You need `[count]="total()"` for Angular to evaluate it.

- **Mistake 3: Trying to bind to attributes that have no DOM property without `attr.`.**
  `[colspan]="span()"` may not work as expected because the DOM property is `colSpan` (capital S). For ARIA attributes, SVG attributes, and `data-*` attributes, always use `[attr.attributeName]="expression"`.

- **Mistake 4: Putting side effects in binding expressions.**
  Property binding expressions should be pure and idempotent. Do not call methods that modify state, log to the console, or trigger HTTP requests inside a `[property]` binding. Angular evaluates these on every change detection cycle.

- **Mistake 5: Mixing interpolation and property binding on the same attribute.**
  `[src]="{{ url() }}"` is invalid. Use either `[src]="url()"` or `src="{{ url() }}"`, not both.

- **Mistake 6: Binding to `innerHTML` with unsanitized user content.**
  Angular sanitizes `innerHTML` by default, but relying on this as your only defense is risky. Validate and sanitize user content on the server side as well. Never use `bypassSecurityTrustHtml` on user-provided data.

---

## Best Practices

- **Use property binding for all dynamic values.** Even when interpolation works (for strings), property binding is more consistent and avoids the boolean attribute trap.

- **Use `attr.` for ARIA, `data-*`, and SVG attributes.** These do not have corresponding DOM properties. Attribute binding with the `attr.` prefix is the correct mechanism.

- **Leverage null removal with `attr.`** When you want to conditionally remove an attribute, use attribute binding with null: `[attr.aria-expanded]="isOpen() ? 'true' : null"`. This is cleaner than conditional `@if` blocks for individual attributes.

- **Keep binding expressions simple.** A good binding expression is a signal read, a property access, or a simple comparison. Extract anything more complex into a `computed` signal.

- **Use static attributes for values that never change.** Write `name="settings"` instead of `[name]="'settings'"` when the value is a literal that will not change. This communicates intent and avoids unnecessary change detection work.

- **Prefer `[disabled]` over `[attr.disabled]`.** The `disabled` DOM property is boolean and works correctly with `true`/`false`. The `disabled` attribute is a boolean HTML attribute where presence means disabled regardless of value. Property binding is the right choice here.

## Real-World Use Cases

### Dynamic Form Field Component

A reusable form field that binds multiple native input properties dynamically based on configuration.

```typescript
@Component({
  selector: 'app-form-field',
  standalone: true,
  template: `
    <label [attr.for]="fieldId()">
      {{ label() }}
      @if (required()) {
        <span class="required">*</span>
      }
    </label>

    <input
      [id]="fieldId()"
      [type]="type()"
      [value]="value()"
      [placeholder]="placeholder()"
      [disabled]="disabled()"
      [readOnly]="readOnly()"
      [required]="required()"
      [maxLength]="maxLength()"
      [attr.aria-describedby]="helpTextId()"
      [attr.aria-invalid]="hasError() ? 'true' : null"
      [attr.autocomplete]="autocomplete()"
      (input)="onInput($event)"
      (blur)="onBlur()"
    />

    @if (helpText()) {
      <p [id]="helpTextId()" class="help-text">{{ helpText() }}</p>
    }

    @if (hasError()) {
      <p class="error-text" role="alert">{{ errorMessage() }}</p>
    }
  `
})
export class FormFieldComponent {
  label = input.required<string>();
  type = input<string>('text');
  value = input<string>('');
  placeholder = input<string>('');
  disabled = input(false);
  readOnly = input(false);
  required = input(false);
  maxLength = input<number>(524288); // browser default
  helpText = input<string>('');
  autocomplete = input<string | null>(null);
  errorMessage = input<string>('');

  hasError = computed(() => this.errorMessage().length > 0);

  private static nextId = 0;
  fieldId = computed(() => `field-${FormFieldComponent.nextId++}`);
  helpTextId = computed(() => `${this.fieldId()}-help`);

  valueChange = output<string>();
  blurred = output<void>();

  onInput(event: Event) {
    const target = event.target as HTMLInputElement;
    this.valueChange.emit(target.value);
  }

  onBlur() {
    this.blurred.emit();
  }
}
```

### Accessible Data Table with Attribute Bindings

A table component that uses attribute bindings for ARIA properties and conditional attributes.

```typescript
@Component({
  selector: 'app-sortable-table',
  standalone: true,
  template: `
    <table
      [attr.aria-label]="tableLabel()"
      [attr.aria-rowcount]="totalRows()"
    >
      <thead>
        <tr>
          @for (col of columns(); track col.key) {
            <th
              [attr.aria-sort]="getSortDirection(col.key)"
              [attr.scope]="'col'"
              [attr.data-column]="col.key"
              [style.width]="col.width ?? 'auto'"
              (click)="toggleSort(col.key)"
              [class.sortable]="col.sortable"
              role="columnheader"
            >
              {{ col.label }}
              @if (col.sortable) {
                <span class="sort-indicator"
                  [attr.aria-hidden]="'true'"
                >
                  {{ sortKey() === col.key ? (sortDir() === 'asc' ? '▲' : '▼') : '⇅' }}
                </span>
              }
            </th>
          }
        </tr>
      </thead>
      <tbody>
        @for (row of sortedRows(); track row.id; let idx = $index) {
          <tr
            [attr.aria-rowindex]="idx + 2"
            [class.selected]="selectedId() === row.id"
            (click)="selectRow(row.id)"
          >
            @for (col of columns(); track col.key) {
              <td [attr.data-label]="col.label">
                {{ row[col.key] }}
              </td>
            }
          </tr>
        }
      </tbody>
    </table>
  `
})
export class SortableTableComponent {
  columns = input.required<Column[]>();
  rows = input.required<Record<string, any>[]>();
  tableLabel = input<string>('Data table');

  sortKey = signal<string | null>(null);
  sortDir = signal<'asc' | 'desc'>('asc');
  selectedId = signal<string | null>(null);

  totalRows = computed(() => this.rows().length);

  sortedRows = computed(() => {
    const key = this.sortKey();
    const dir = this.sortDir();
    const data = [...this.rows()];
    if (!key) return data;
    return data.sort((a, b) => {
      const cmp = String(a[key]).localeCompare(String(b[key]));
      return dir === 'asc' ? cmp : -cmp;
    });
  });

  getSortDirection(key: string): string | null {
    if (this.sortKey() !== key) return null;
    return this.sortDir() === 'asc' ? 'ascending' : 'descending';
  }

  toggleSort(key: string) {
    if (this.sortKey() === key) {
      this.sortDir.update(d => d === 'asc' ? 'desc' : 'asc');
    } else {
      this.sortKey.set(key);
      this.sortDir.set('asc');
    }
  }

  selectRow(id: string) {
    this.selectedId.set(id);
  }
}

interface Column {
  key: string;
  label: string;
  sortable?: boolean;
  width?: string;
}
```

### SVG Chart with Dynamic Property and Attribute Bindings

An SVG bar chart that combines property bindings for dimensions and attribute bindings for SVG-specific attributes.

```typescript
@Component({
  selector: 'app-bar-chart',
  standalone: true,
  template: `
    <svg
      [attr.width]="width()"
      [attr.height]="height()"
      [attr.viewBox]="viewBox()"
      role="img"
      [attr.aria-label]="chartLabel()"
    >
      @for (bar of bars(); track bar.label; let i = $index) {
        <g>
          <rect
            [attr.x]="barX(i)"
            [attr.y]="barY(bar.value)"
            [attr.width]="barWidth()"
            [attr.height]="barHeight(bar.value)"
            [attr.fill]="bar.color ?? defaultColor()"
            [attr.rx]="4"
          />
          <text
            [attr.x]="barCenterX(i)"
            [attr.y]="height() - 5"
            text-anchor="middle"
            class="label"
          >
            {{ bar.label }}
          </text>
          <text
            [attr.x]="barCenterX(i)"
            [attr.y]="barY(bar.value) - 5"
            text-anchor="middle"
            class="value"
          >
            {{ bar.value }}
          </text>
        </g>
      }
    </svg>
  `,
  styles: [`
    :host { display: block; }
    .label { font-size: 12px; fill: #666; }
    .value { font-size: 11px; fill: #333; font-weight: 600; }
  `]
})
export class BarChartComponent {
  data = input.required<BarData[]>();
  width = input(400);
  height = input(250);
  defaultColor = input('#4285f4');
  chartLabel = input('Bar chart');

  private padding = 40;
  private barGap = 8;

  viewBox = computed(() => `0 0 ${this.width()} ${this.height()}`);
  bars = computed(() => this.data());

  private maxValue = computed(() =>
    Math.max(...this.data().map(d => d.value), 1)
  );

  private chartHeight = computed(() => this.height() - this.padding);

  barWidth = computed(() => {
    const count = this.data().length;
    if (count === 0) return 0;
    return (this.width() - this.padding - this.barGap * (count - 1)) / count;
  });

  barX(index: number): number {
    return this.padding / 2 + index * (this.barWidth() + this.barGap);
  }

  barY(value: number): number {
    return this.chartHeight() - (value / this.maxValue()) * this.chartHeight() + 10;
  }

  barHeight(value: number): number {
    return (value / this.maxValue()) * this.chartHeight();
  }

  barCenterX(index: number): number {
    return this.barX(index) + this.barWidth() / 2;
  }
}

interface BarData {
  label: string;
  value: number;
  color?: string;
}
```

---

> **Key Takeaway:** Property binding (`[property]="expression"`) targets DOM properties, not HTML attributes. Use it for all dynamic values to preserve types and handle booleans correctly. Use `[attr.name]` only for attributes without a DOM property counterpart (ARIA, `data-*`, SVG). Keep binding expressions simple and extract complex logic into computed signals.
