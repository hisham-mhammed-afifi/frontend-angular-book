# 13.7 Directive Composition API

> Apply multiple directives to a component or directive declaratively through `hostDirectives`, enabling reuse without inheritance.

---

## The Problem Composition Solves

Before the Directive Composition API, sharing behavior across components required one of three approaches, each with significant downsides.

### Approach 1: Duplicate Inputs

```typescript
// Bad Practice: every component duplicates the same inputs

@Component({
  selector: 'app-button',
  standalone: true,
  template: `<button [disabled]="disabled()" [style.color]="color()">
    <ng-content />
  </button>`,
})
export class ButtonComponent {
  disabled = input(false);
  color = input('inherit');
}

@Component({
  selector: 'app-toggle',
  standalone: true,
  template: `<label [style.color]="color()">
    <input type="checkbox" [disabled]="disabled()" />
  </label>`,
})
export class ToggleComponent {
  disabled = input(false);
  color = input('inherit');
}
```

Two components, same two inputs, same logic. Every new component repeats this. If the disabled logic changes, you update every component.

### Approach 2: Inheritance

```typescript
// Bad Practice: inheritance leaks irrelevant API surface

class InteractiveBase {
  disabled = input(false);
  color = input('inherit');
}

@Component({ selector: 'app-spinner', ... })
export class SpinnerComponent extends InteractiveBase {
  // Spinner inherits "disabled" and "color" even though
  // a loading spinner has no concept of being disabled.
}
```

Inheritance forces every subclass to carry the full parent API, whether relevant or not.

### Approach 3: Template-Level Directives

```html
<!-- Works, but the consumer must remember to add every directive -->
<app-button appColor appDisable [disableState]="false" color="secondary">
  Save
</app-button>
```

If the template author forgets `appDisable`, the component silently loses that behavior. Nothing enforces completeness.

---

## The `hostDirectives` Property

The Directive Composition API, introduced in Angular 15, solves all three problems. You declare directives inside a component or directive metadata, and Angular applies them to the host element automatically.

```typescript
import { Component, Directive, input } from '@angular/core';

@Directive({
  selector: '[appDisable]',
  standalone: true,
  host: {
    '[attr.disabled]': 'disableState() || null',
    '[class.is-disabled]': 'disableState()',
  },
})
export class DisableDirective {
  disableState = input(false);
}

@Directive({
  selector: '[appColor]',
  standalone: true,
  host: {
    '[style.color]': 'color()',
  },
})
export class ColorDirective {
  color = input('inherit');
}
```

Now compose them into a component:

```typescript
@Component({
  selector: 'app-toggle',
  standalone: true,
  hostDirectives: [DisableDirective, ColorDirective],
  template: `
    <label class="switch">
      <input type="checkbox" />
      <span class="slider"></span>
    </label>
  `,
})
export class ToggleComponent {}
```

Angular creates an instance of `DisableDirective` and `ColorDirective` for every `<app-toggle>` element. Their host bindings apply to the toggle's host element. No template markup needed.

### Critical Rule: Standalone Only

Every directive in `hostDirectives` must be standalone. Non-standalone directives declared in NgModules cannot participate in composition.

```typescript
// This WILL NOT compile
@Directive({ selector: '[appLegacy]' }) // no standalone: true
export class LegacyDirective {}

@Component({
  hostDirectives: [LegacyDirective], // ERROR: not standalone
})
export class SomeComponent {}
```

### Selectors Are Ignored

Angular ignores the `selector` of any directive listed in `hostDirectives`. The directive is applied unconditionally to the host, regardless of what its selector says.

```typescript
@Directive({
  selector: '[appTooltip]', // this selector is irrelevant when composed
  standalone: true,
})
export class TooltipDirective { ... }

@Component({
  selector: 'app-pin',
  hostDirectives: [TooltipDirective], // applied without [appTooltip] attribute
})
export class PinComponent {}
```

## Exposing Inputs and Outputs

By default, composed directive inputs and outputs are **hidden** from the component's public API. This is a deliberate design choice: you control what the consumer sees.

### Hidden by Default

```typescript
@Component({
  selector: 'app-card',
  standalone: true,
  hostDirectives: [ColorDirective], // color input is NOT accessible
  template: `<ng-content />`,
})
export class CardComponent {}
```

```html
<!-- ERROR: 'color' is not a known input of 'app-card' -->
<app-card [color]="'red'">Content</app-card>
```

### Explicitly Exposing Inputs and Outputs

Use the object syntax to list which inputs and outputs to expose:

```typescript
@Component({
  selector: 'app-card',
  standalone: true,
  hostDirectives: [
    {
      directive: ColorDirective,
      inputs: ['color'],       // now accessible on <app-card>
    },
    {
      directive: DisableDirective,
      inputs: ['disableState'],
    },
  ],
  template: `<ng-content />`,
})
export class CardComponent {}
```

```html
<!-- Works: both inputs are now part of the component's API -->
<app-card [color]="'blue'" [disableState]="true">Content</app-card>
```

### Exposing Outputs

Outputs follow the same pattern:

```typescript
@Directive({
  selector: '[appDraggable]',
  standalone: true,
})
export class DraggableDirective {
  dragStart = output<{ x: number; y: number }>();
  dragEnd = output<{ x: number; y: number }>();
}

@Component({
  selector: 'app-widget',
  standalone: true,
  hostDirectives: [
    {
      directive: DraggableDirective,
      outputs: ['dragStart', 'dragEnd'],
    },
  ],
  template: `<ng-content />`,
})
export class WidgetComponent {}
```

```html
<app-widget (dragStart)="onDragStart($event)" (dragEnd)="onDragEnd($event)">
  Dashboard Widget
</app-widget>
```

## Input and Output Aliasing

Directive input names might not fit the context of the host component. Aliasing renames them without modifying the directive.

```typescript
@Directive({
  selector: '[appDraggable]',
  standalone: true,
})
export class DraggableDirective {
  dragzone = input('default'); // generic name
  dropped = output<void>();
}

@Component({
  selector: 'app-pin',
  standalone: true,
  hostDirectives: [
    {
      directive: DraggableDirective,
      inputs: ['dragzone: pinBoard'],     // alias: consumer sees "pinBoard"
      outputs: ['dropped: onPinDropped'], // alias: consumer sees "onPinDropped"
    },
  ],
  template: `<img [src]="imageSrc()" />`,
})
export class PinComponent {
  imageSrc = input('assets/pin.svg');
}
```

```html
<!-- Consumer uses the aliased names -->
<app-pin [pinBoard]="'board-1'" (onPinDropped)="handleDrop()" />
```

The syntax is `'originalName: aliasName'`, identical to how Angular handles input aliasing elsewhere.

## Transitive Composition (Directives on Directives)

A directive can itself declare `hostDirectives`, creating layered composition. When that directive is composed into a component, all directives in the chain are instantiated.

```typescript
@Directive({
  selector: '[appTooltip]',
  standalone: true,
})
export class TooltipDirective {
  tooltipText = input('');
}

@Directive({
  selector: '[appHighlight]',
  standalone: true,
  host: {
    '[class.highlighted]': 'isHighlighted()',
  },
})
export class HighlightDirective {
  isHighlighted = input(false);
}

// This directive composes two others
@Directive({
  selector: '[appInteractive]',
  standalone: true,
  hostDirectives: [
    { directive: TooltipDirective, inputs: ['tooltipText'] },
    { directive: HighlightDirective, inputs: ['isHighlighted'] },
  ],
})
export class InteractiveDirective {
  // Additional logic specific to "interactive" behavior
}

// The component gets all three directives
@Component({
  selector: 'app-card',
  standalone: true,
  hostDirectives: [
    {
      directive: InteractiveDirective,
      inputs: ['tooltipText', 'isHighlighted'],
    },
  ],
  template: `<ng-content />`,
})
export class CardComponent {}
```

When Angular creates a `CardComponent`, it also creates instances of `InteractiveDirective`, `TooltipDirective`, and `HighlightDirective`. The chain is fully transparent.

### Re-exposing Inputs Through the Chain

For an input from a deeply nested directive to be accessible on the final component, it must be exposed at every level in the chain.

```typescript
// Level 1: the base directive
@Directive({ standalone: true, selector: '[appBase]' })
export class BaseDirective {
  baseColor = input('gray');
}

// Level 2: composes BaseDirective, exposes its input
@Directive({
  standalone: true,
  selector: '[appMiddle]',
  hostDirectives: [
    { directive: BaseDirective, inputs: ['baseColor'] }, // must expose here
  ],
})
export class MiddleDirective {}

// Level 3: composes MiddleDirective, must re-expose
@Component({
  selector: 'app-widget',
  standalone: true,
  hostDirectives: [
    { directive: MiddleDirective, inputs: ['baseColor'] }, // re-expose again
  ],
  template: `...`,
})
export class WidgetComponent {}
```

If you forget to expose `baseColor` at the `MiddleDirective` level, it will not be available on `WidgetComponent` even if you list it there.

## Accessing Composed Directive Instances

Inside the host component, you can inject a composed directive to read its state or call its methods.

```typescript
@Component({
  selector: 'app-panel',
  standalone: true,
  hostDirectives: [DisableDirective, ColorDirective],
  template: `
    <div class="panel-header">
      <ng-content select="[header]" />
    </div>
    <div class="panel-body">
      <ng-content />
    </div>
  `,
})
export class PanelComponent {
  private disableDir = inject(DisableDirective);
  private colorDir = inject(ColorDirective);

  isDisabled(): boolean {
    return this.disableDir.disableState();
  }

  currentColor(): string {
    return this.colorDir.color();
  }
}
```

This works because composed directives live in the same node injector as the host component. The `inject()` call resolves to the exact instance that Angular created for that host element.

### Injection Without Composition Fails

If you try to inject a directive that is NOT in `hostDirectives`, Angular throws a "No provider" error:

```typescript
// Bad Practice: injecting a directive not listed in hostDirectives
@Component({
  selector: 'app-broken',
  standalone: true,
  hostDirectives: [ColorDirective],
  template: `...`,
})
export class BrokenComponent {
  // ERROR at runtime: No provider for DisableDirective
  private disable = inject(DisableDirective);
}
```

## Lifecycle Hook Ordering

Host directives execute their lifecycle hooks **before** the host component or directive. This ordering is consistent:

1. Host directive constructors (in declaration order within `hostDirectives`)
2. Host component constructor
3. Host directive `ngOnInit` hooks
4. Host component `ngOnInit`
5. Host directive `ngAfterViewInit` hooks
6. Host component `ngAfterViewInit`

The same before-host pattern applies to all lifecycle hooks.

```typescript
@Directive({
  selector: '[appFirst]',
  standalone: true,
})
export class FirstDirective {
  constructor() {
    console.log('FirstDirective: constructor');
  }

  ngOnInit() {
    console.log('FirstDirective: ngOnInit');
  }
}

@Directive({
  selector: '[appSecond]',
  standalone: true,
})
export class SecondDirective {
  constructor() {
    console.log('SecondDirective: constructor');
  }

  ngOnInit() {
    console.log('SecondDirective: ngOnInit');
  }
}

@Component({
  selector: 'app-demo',
  standalone: true,
  hostDirectives: [FirstDirective, SecondDirective],
  template: `<p>Demo</p>`,
})
export class DemoComponent {
  constructor() {
    console.log('DemoComponent: constructor');
  }

  ngOnInit() {
    console.log('DemoComponent: ngOnInit');
  }
}

// Console output:
// FirstDirective: constructor
// SecondDirective: constructor
// DemoComponent: constructor
// FirstDirective: ngOnInit
// SecondDirective: ngOnInit
// DemoComponent: ngOnInit
```

### Host Binding Override

Because host directives bind first, the host component can override any host binding set by a composed directive:

```typescript
@Directive({
  selector: '[appBase]',
  standalone: true,
  host: { class: 'base-style' },
})
export class BaseStyleDirective {}

@Component({
  selector: 'app-card',
  standalone: true,
  hostDirectives: [BaseStyleDirective],
  host: { class: 'card-style' }, // this wins over 'base-style'
  template: `<ng-content />`,
})
export class CardComponent {}

// The host element gets class="card-style", NOT "base-style"
```

For additive bindings like `class`, both values apply. For conflicting bindings on the same property, the component wins.

---

## Components Cannot Be Host Directives

Only directives can appear in `hostDirectives`. Components are excluded.

```typescript
// This WILL NOT compile
@Component({
  selector: 'app-parent',
  standalone: true,
  hostDirectives: [SomeOtherComponent], // ERROR
  template: `...`,
})
export class ParentComponent {}
```

This is by design. Components have templates and view encapsulation, which conflicts with host directive semantics where the directive operates on the host element without its own template.

---

## Duplicate Directive Prevention

A directive can appear only once in a composition chain. Angular throws a compile-time error if the same directive appears multiple times.

```typescript
// Bad Practice: duplicate directive in the chain

@Directive({
  standalone: true,
  selector: '[appWrapper]',
  hostDirectives: [ColorDirective],
})
export class WrapperDirective {}

@Component({
  selector: 'app-widget',
  standalone: true,
  hostDirectives: [
    WrapperDirective,  // already includes ColorDirective
    ColorDirective,    // ERROR: duplicate
  ],
  template: `...`,
})
export class WidgetComponent {}
```

## Practical Patterns

### Pattern 1: Behavior Mixins

Create small, focused directives that each handle one concern, then compose them.

```typescript
@Directive({
  selector: '[appFocusable]',
  standalone: true,
  host: {
    'tabindex': '0',
    '(focus)': 'onFocus()',
    '(blur)': 'onBlur()',
    '[class.focused]': 'isFocused',
  },
})
export class FocusableDirective {
  isFocused = false;

  onFocus() {
    this.isFocused = true;
  }

  onBlur() {
    this.isFocused = false;
  }
}

@Directive({
  selector: '[appSelectable]',
  standalone: true,
  host: {
    '(click)': 'toggle()',
    '[class.selected]': 'isSelected',
    '[attr.aria-selected]': 'isSelected',
  },
})
export class SelectableDirective {
  isSelected = false;

  toggle() {
    this.isSelected = !this.isSelected;
  }
}

@Directive({
  selector: '[appHoverable]',
  standalone: true,
  host: {
    '(mouseenter)': 'isHovered = true',
    '(mouseleave)': 'isHovered = false',
    '[class.hovered]': 'isHovered',
  },
})
export class HoverableDirective {
  isHovered = false;
}

// Compose all three into an interactive list item
@Component({
  selector: 'app-list-item',
  standalone: true,
  hostDirectives: [
    FocusableDirective,
    SelectableDirective,
    HoverableDirective,
  ],
  template: `<ng-content />`,
  styles: `
    :host { display: block; padding: 8px 16px; cursor: pointer; }
    :host(.hovered) { background: var(--hover-bg, #f5f5f5); }
    :host(.focused) { outline: 2px solid var(--focus-ring, #1976d2); }
    :host(.selected) { background: var(--selected-bg, #e3f2fd); }
  `,
})
export class ListItemComponent {}
```

Usage:

```html
<app-list-item>First item</app-list-item>
<app-list-item>Second item</app-list-item>
<app-list-item>Third item</app-list-item>
```

Every list item automatically gets focus, selection, and hover behavior without any template attributes.

### Pattern 2: Dashboard Widgets

```typescript
@Directive({
  selector: '[appResizable]',
  standalone: true,
  host: {
    '(mousedown)': 'onResizeStart($event)',
    '[style.width.px]': 'width()',
    '[style.height.px]': 'height()',
  },
})
export class ResizableDirective {
  width = input(300);
  height = input(200);
  resized = output<{ width: number; height: number }>();

  onResizeStart(event: MouseEvent) {
    // resize logic
  }
}

@Directive({
  selector: '[appDraggable]',
  standalone: true,
  host: {
    '(mousedown)': 'onDragStart($event)',
    '[style.transform]': 'translateValue',
  },
})
export class DraggableDirective {
  dragStart = output<{ x: number; y: number }>();
  dragEnd = output<{ x: number; y: number }>();
  translateValue = 'translate(0, 0)';

  onDragStart(event: MouseEvent) {
    // drag logic
  }
}

@Component({
  selector: 'app-dashboard-widget',
  standalone: true,
  hostDirectives: [
    {
      directive: ResizableDirective,
      inputs: ['width', 'height'],
      outputs: ['resized'],
    },
    {
      directive: DraggableDirective,
      outputs: ['dragStart', 'dragEnd'],
    },
  ],
  template: `
    <div class="widget-header">
      <ng-content select="[header]" />
    </div>
    <div class="widget-body">
      <ng-content />
    </div>
  `,
})
export class DashboardWidgetComponent {}
```

```html
<app-dashboard-widget
  [width]="400"
  [height]="300"
  (resized)="saveLayout($event)"
  (dragEnd)="savePosition($event)"
>
  <span header>Sales Chart</span>
  <app-chart [data]="salesData" />
</app-dashboard-widget>
```

### Pattern 3: Form Field Wrapper

```typescript
@Directive({
  selector: '[appValidationStyle]',
  standalone: true,
  host: {
    '[class.is-valid]': 'valid()',
    '[class.is-invalid]': 'invalid()',
    '[class.is-touched]': 'touched()',
  },
})
export class ValidationStyleDirective {
  valid = input(false);
  invalid = input(false);
  touched = input(false);
}

@Directive({
  selector: '[appFieldLabel]',
  standalone: true,
})
export class FieldLabelDirective {
  label = input.required<string>();
  required = input(false);
}

@Component({
  selector: 'app-form-field',
  standalone: true,
  hostDirectives: [
    {
      directive: ValidationStyleDirective,
      inputs: ['valid', 'invalid', 'touched'],
    },
    {
      directive: FieldLabelDirective,
      inputs: ['label', 'required'],
    },
  ],
  template: `
    @let labelDir = fieldLabel();
    <label>
      {{ labelDir.label() }}
      @if (labelDir.required()) {
        <span class="required-marker">*</span>
      }
    </label>
    <div class="field-content">
      <ng-content />
    </div>
  `,
})
export class FormFieldComponent {
  private fieldLabel = viewChild(FieldLabelDirective);
  // Or use inject():
  // private fieldLabel = inject(FieldLabelDirective);
}
```

## Comparison: Composition vs Inheritance vs Template Directives

| Aspect | Composition (`hostDirectives`) | Inheritance (`extends`) | Template Directives |
|---|---|---|---|
| Multiple behaviors | Yes, unlimited | Single parent only | Yes, but manual |
| API control | Explicit (inputs/outputs) | Inherits everything | Implicit |
| Consumer effort | Zero (automatic) | Zero | Must remember every directive |
| Encapsulation | Strong | Weak (leaks parent API) | None |
| Performance | One instance per directive | One instance | One instance per directive |
| Type safety | Full | Full | Full |
| Composability depth | Unlimited (transitive) | Single level | N/A |

## Performance Implications

Each composed directive creates a separate class instance at the same node. For a component with three `hostDirectives`, Angular allocates four objects per component instance (the component plus three directives).

### When This Matters

```typescript
// A table row component with three composed directives
@Component({
  selector: 'app-table-row',
  standalone: true,
  hostDirectives: [
    SelectableDirective,
    HoverableDirective,
    FocusableDirective,
  ],
  template: `...`,
})
export class TableRowComponent {}
```

If you render 1,000 rows, Angular creates 4,000 objects (1,000 components + 3,000 directive instances). Compare this to a single component with inline logic: 1,000 objects.

### Mitigation Strategies

1. **Measure first.** Use Angular DevTools profiler and Chrome memory snapshots. Composition overhead is often negligible for typical UI component counts (dozens to low hundreds).

2. **Use virtual scrolling for large lists.** The CDK `ScrollingModule` limits the number of instantiated components, which also limits composed directive instances.

3. **Do not compose for trivial bindings.** A single `[class.active]="isActive"` does not need a separate directive. Composition is for reusable, non-trivial behavior.

4. **Keep composed directives lightweight.** Avoid expensive constructor logic or heavy `ngOnInit` operations in directives intended for composition.

---

## Common Mistakes

- **Forgetting to expose inputs/outputs.** The most frequent mistake. If you compose a directive but do not list its inputs in the `inputs` array, consumers cannot set them. The template compiler will report an "unknown input" error, which can be confusing if you expected the directive's inputs to pass through automatically.

- **Using non-standalone directives.** The compiler error is clear ("Host directive must be standalone"), but developers migrating from NgModule-based codebases often hit this. Migrate directives to standalone before composing them.

- **Expecting input propagation across transitive chains.** If Directive A exposes BaseDirective's input, and Component B composes Directive A, Component B must also explicitly expose that input. Each level is a separate gate.

- **Composing too many directives.** Having six or seven directives in `hostDirectives` is a code smell. It suggests the component is doing too much. Consolidate related behaviors into a single directive first, then compose that.

- **Injecting a directive not in the composition chain.** `inject(SomeDirective)` only works if `SomeDirective` is in `hostDirectives` (directly or transitively). Otherwise, Angular throws a runtime error.

- **Trying to use a component as a host directive.** Only `@Directive`-decorated classes are allowed in `hostDirectives`. Components are excluded.

---

## Best Practices

- **Prefer small, single-purpose directives.** Each directive should handle exactly one behavior: focus management, drag support, validation styling, etc. Compose them to build complex components.

- **Alias inputs to match the host component's domain language.** If you compose a `DraggableDirective` into a `PinComponent`, alias `dragzone` to `pinBoard`. The consumer should not need to know about the underlying directive.

- **Document which directives are composed.** Since composed directives are invisible in the template, add a comment or documentation listing what behaviors a component includes.

- **Use `inject()` to coordinate between composed directives.** The host component can inject any composed directive to read its state or react to its changes.

- **Test composed behavior through the host component.** Do not test composed directives in isolation when verifying composition. Write tests that exercise the component and verify the directive behavior is present.

- **Prefer composition over inheritance every time.** Angular officially recommends composition through `hostDirectives` as the primary mechanism for code reuse across components.

## Real-World Use Cases

### Accessible Button Component

A button that automatically handles focus ring visibility, disabled state, and ARIA attributes through composition:

```typescript
@Directive({
  selector: '[appFocusRing]',
  standalone: true,
  host: {
    '(focus)': 'onFocus()',
    '(blur)': 'onBlur()',
    '(mousedown)': 'onMouseDown()',
    '[class.focus-visible]': 'showFocusRing',
  },
})
export class FocusRingDirective {
  showFocusRing = false;
  private usedKeyboard = true;

  onFocus() {
    if (this.usedKeyboard) {
      this.showFocusRing = true;
    }
  }

  onBlur() {
    this.showFocusRing = false;
    this.usedKeyboard = true;
  }

  onMouseDown() {
    this.usedKeyboard = false;
  }
}

@Directive({
  selector: '[appAriaDisabled]',
  standalone: true,
  host: {
    '[attr.aria-disabled]': 'disabled()',
    '[class.is-disabled]': 'disabled()',
    '[attr.tabindex]': 'disabled() ? -1 : null',
  },
})
export class AriaDisabledDirective {
  disabled = input(false);
}

@Component({
  selector: 'app-button',
  standalone: true,
  hostDirectives: [
    FocusRingDirective,
    {
      directive: AriaDisabledDirective,
      inputs: ['disabled'],
    },
  ],
  host: {
    'role': 'button',
  },
  template: `<ng-content />`,
  styles: `
    :host {
      display: inline-flex;
      padding: 8px 16px;
      border-radius: 4px;
      cursor: pointer;
    }
    :host(.is-disabled) { opacity: 0.5; pointer-events: none; }
    :host(.focus-visible) { outline: 2px solid var(--focus-color, #1976d2); }
  `,
})
export class ButtonComponent {}
```

```html
<!-- Consumer gets accessibility for free -->
<app-button [disabled]="isSaving()">Save</app-button>
```

### Shared CMS Content Block Behaviors

A CMS editor where content blocks share editing, selection, and drag-and-drop behavior:

```typescript
@Directive({
  selector: '[appEditable]',
  standalone: true,
  host: {
    '(dblclick)': 'startEdit()',
    '[class.editing]': 'isEditing',
  },
})
export class EditableDirective {
  isEditing = false;
  editStarted = output<void>();
  editEnded = output<void>();

  startEdit() {
    this.isEditing = true;
    this.editStarted.emit();
  }

  endEdit() {
    this.isEditing = false;
    this.editEnded.emit();
  }
}

// Compose into a "block behavior" directive for transitive use
@Directive({
  selector: '[appBlockBehavior]',
  standalone: true,
  hostDirectives: [
    {
      directive: EditableDirective,
      outputs: ['editStarted', 'editEnded'],
    },
    SelectableDirective,
    {
      directive: DraggableDirective,
      outputs: ['dragEnd'],
    },
  ],
})
export class BlockBehaviorDirective {}

// Every content block type composes the bundle
@Component({
  selector: 'app-text-block',
  standalone: true,
  hostDirectives: [
    {
      directive: BlockBehaviorDirective,
      outputs: ['editStarted', 'editEnded', 'dragEnd'],
    },
  ],
  template: `<p [contentEditable]="editable.isEditing">{{ content() }}</p>`,
})
export class TextBlockComponent {
  content = input('');
  protected editable = inject(EditableDirective);
}

@Component({
  selector: 'app-image-block',
  standalone: true,
  hostDirectives: [
    {
      directive: BlockBehaviorDirective,
      outputs: ['editStarted', 'editEnded', 'dragEnd'],
    },
  ],
  template: `<img [src]="src()" [alt]="alt()" />`,
})
export class ImageBlockComponent {
  src = input.required<string>();
  alt = input('');
}
```

Both `TextBlockComponent` and `ImageBlockComponent` automatically support editing, selection, and drag behavior through a single `hostDirectives` entry.

---

> **Key Takeaway:** The Directive Composition API replaces inheritance-based code reuse with explicit, composable behavior units that apply automatically to host elements, giving you full control over the public API while keeping consumers unaware of the internal composition.
