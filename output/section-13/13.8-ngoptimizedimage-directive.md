# 13.8 `NgOptimizedImage` Directive

> A built-in directive that enforces image loading best practices, automatically handling lazy loading, responsive srcsets, preconnects, and LCP prioritization.

---

## Why Image Optimization Matters

Images are typically the largest assets on a page. A single unoptimized hero image can add seconds to your Largest Contentful Paint (LCP), which directly affects Core Web Vitals scores and user experience. Common problems include:

- Loading offscreen images eagerly, wasting bandwidth.
- Missing `width` and `height`, causing layout shifts (CLS).
- Using a single fixed-size image for all screen sizes.
- No preconnect to the image CDN, adding DNS/TLS latency.
- No `fetchpriority` hint, so the browser guesses which image matters most.

`NgOptimizedImage` solves all of these automatically when you replace `src` with `ngSrc`.

---

## Setup

Import `NgOptimizedImage` from `@angular/common` and add it to your component's `imports`.

```typescript
import { Component } from '@angular/core';
import { NgOptimizedImage } from '@angular/common';

@Component({
  selector: 'app-hero',
  standalone: true,
  imports: [NgOptimizedImage],
  template: `
    <img ngSrc="hero-banner.jpg" width="1600" height="900" priority />
  `,
})
export class HeroComponent {}
```

That single change activates all optimizations: lazy loading for non-priority images, `fetchpriority="high"` for priority images, layout shift prevention, and development-mode warnings.

## Core Attributes

### `ngSrc` (Required)

Replaces the standard `src` attribute. When using a built-in or custom image loader, `ngSrc` contains only the image path relative to your CDN base URL. Without a loader, it is the full URL or relative path.

```html
<!-- Without a loader: full URL -->
<img ngSrc="https://example.com/images/photo.jpg" width="800" height="600" />

<!-- With a loader (e.g., Cloudinary): just the path -->
<img ngSrc="products/shoe.jpg" width="800" height="600" />
```

### `width` and `height` (Required Unless `fill` Is Used)

These prevent Cumulative Layout Shift by reserving space before the image loads. They represent:

- **Responsive images:** The intrinsic (natural) dimensions of the image file.
- **Fixed-size images:** The desired rendered dimensions.

```html
<!-- Responsive: intrinsic size of the file -->
<img ngSrc="landscape.jpg" width="2400" height="1600" sizes="100vw" />

<!-- Fixed: exact rendered size -->
<img ngSrc="avatar.jpg" width="48" height="48" />
```

Angular validates these at development time. If the rendered aspect ratio deviates significantly from the declared width/height ratio, you get a console warning about visual distortion.

### `priority`

Marks an image as the Largest Contentful Paint element. Apply this to the above-the-fold hero image or the largest visible image on page load.

```html
<img ngSrc="hero.jpg" width="1600" height="900" priority />
```

What `priority` does:

| Behavior | Without `priority` | With `priority` |
|---|---|---|
| `loading` attribute | `lazy` | `eager` |
| `fetchpriority` attribute | Not set | `high` |
| `decoding` attribute | `auto` | `sync` |
| SSR preload link | Not generated | Auto-generated |

Rules for `priority`:
- Only one or two images per page should have it.
- It must be the LCP image (usually the hero banner or the largest product photo).
- Marking too many images as priority defeats the purpose and can hurt performance.

```html
<!-- Bad Practice: marking multiple images as priority -->
<img ngSrc="img1.jpg" width="200" height="200" priority />
<img ngSrc="img2.jpg" width="200" height="200" priority />
<img ngSrc="img3.jpg" width="200" height="200" priority />
<img ngSrc="img4.jpg" width="200" height="200" priority />
<!-- Only the actual LCP image should have priority -->
```

### `loading`

Controls browser loading behavior. Defaults to `lazy` for non-priority images.

```html
<!-- Explicit eager loading (prefer using priority instead) -->
<img ngSrc="above-fold.jpg" width="800" height="400" loading="eager" />

<!-- Default: lazy (no attribute needed) -->
<img ngSrc="below-fold.jpg" width="800" height="400" />
```

Accepted values: `"lazy"`, `"eager"`, `"auto"`.

### `decoding`

Controls how the browser decodes the image. Defaults to `"auto"`. Priority images auto-set to `"sync"`.

```html
<!-- Gallery thumbnails: async decoding to avoid blocking -->
<img ngSrc="gallery/thumb-1.jpg" width="300" height="200" decoding="async" />
```

Accepted values: `"auto"`, `"async"`, `"sync"`.

## Fill Mode

When you do not know the image dimensions at build time, or the image should fill its container, use `fill`. This removes the `width` and `height` requirement.

```typescript
@Component({
  selector: 'app-card',
  standalone: true,
  imports: [NgOptimizedImage],
  template: `
    <div class="card-image-wrapper">
      <img ngSrc="products/item-42.jpg" fill />
    </div>
  `,
  styles: `
    .card-image-wrapper {
      position: relative; /* REQUIRED for fill mode */
      width: 100%;
      height: 300px;
    }
    img {
      object-fit: cover; /* controls how the image fills the box */
    }
  `,
})
export class CardComponent {}
```

### Fill Mode Requirements

1. The parent element must have `position: relative`, `position: absolute`, or `position: fixed`.
2. The parent element must have a non-zero height (either explicit or from content).
3. Do not set `width` or `height` attributes when using `fill`.

```html
<!-- Bad Practice: fill without positioned parent -->
<div style="width: 100%;">
  <!-- This will look broken: the image has no height reference -->
  <img ngSrc="photo.jpg" fill />
</div>

<!-- Correct -->
<div style="position: relative; width: 100%; height: 400px;">
  <img ngSrc="photo.jpg" fill />
</div>
```

### `object-fit` Values with Fill

| `object-fit` | Behavior |
|---|---|
| `cover` | Image fills the container, cropping excess. Best for background-like images. |
| `contain` | Image fits inside the container, adding letterbox space. |
| `fill` | Image stretches to fill, potentially distorting aspect ratio. |
| `none` | Image renders at natural size, cropping if larger. |
| `scale-down` | Like `contain`, but never scales up beyond natural size. |

## Responsive Images: `sizes` and Automatic `srcset`

### How `srcset` Generation Works

When you provide a `sizes` attribute, `NgOptimizedImage` automatically generates a `srcset` with multiple widths. The browser then picks the best fit.

```html
<!-- Full-width hero: tell the browser it fills the viewport -->
<img ngSrc="hero.jpg" width="2400" height="1200" sizes="100vw" priority />

<!-- Two-column layout: 50% of viewport on desktop, 100% on mobile -->
<img
  ngSrc="product.jpg"
  width="1200"
  height="800"
  sizes="(max-width: 768px) 100vw, 50vw"
/>
```

Angular auto-prepends `"auto"` to the provided `sizes` value for better browser behavior.

### Default Breakpoints

The auto-generated `srcset` uses these widths by default:

```
16, 32, 48, 64, 96, 128, 256, 384, 640, 750, 828, 1080, 1200, 1920, 2048, 3840
```

### Customizing Breakpoints

Override the defaults through the `IMAGE_CONFIG` token:

```typescript
import { IMAGE_CONFIG } from '@angular/common';

@Component({
  selector: 'app-root',
  standalone: true,
  imports: [NgOptimizedImage],
  providers: [
    {
      provide: IMAGE_CONFIG,
      useValue: {
        breakpoints: [320, 640, 960, 1280, 1920],
      },
    },
  ],
  template: `...`,
})
export class AppComponent {}
```

Or at the application level in `app.config.ts`:

```typescript
import { ApplicationConfig } from '@angular/core';
import { IMAGE_CONFIG } from '@angular/common';

export const appConfig: ApplicationConfig = {
  providers: [
    {
      provide: IMAGE_CONFIG,
      useValue: {
        breakpoints: [320, 640, 960, 1280, 1920],
      },
    },
  ],
};
```

### Manual `srcset` with `ngSrcset`

For fine-grained control, provide explicit width or density descriptors:

```html
<!-- Width descriptors -->
<img
  ngSrc="product.jpg"
  ngSrcset="200w, 400w, 800w, 1200w"
  sizes="(max-width: 768px) 100vw, 50vw"
  width="1200"
  height="800"
/>

<!-- Density descriptors (for fixed-size images) -->
<img
  ngSrc="logo.png"
  ngSrcset="1x, 2x, 3x"
  width="200"
  height="60"
/>
```

You only provide the descriptors. The directive combines them with `ngSrc` and the image loader to produce the full URLs.

### Disabling Automatic `srcset`

```html
<img ngSrc="special.jpg" width="800" height="600" disableOptimizedSrcset />
```

Use this when you need the exact URL without any width transformations, for example with SVGs or when your CDN does not support width parameters.

## Image Placeholders

Placeholders show a low-resolution preview while the full image loads, preventing blank space.

### Automatic Placeholder (Requires a Loader)

When using a CDN loader that supports width-based resizing, Angular requests a tiny version (30px wide by default) and displays it as a blurred background.

```html
<img ngSrc="product.jpg" width="800" height="600" placeholder />
```

The placeholder is shown as a CSS `background-image` with a blur filter applied. Once the full image loads, the placeholder fades away.

### Configuring Placeholder Resolution

```typescript
providers: [
  {
    provide: IMAGE_CONFIG,
    useValue: {
      placeholderResolution: 40, // default is 30
    },
  },
],
```

### Data URL Placeholder

If you have a pre-generated Base64 image (e.g., from a build step or CMS), pass it directly:

```html
<img
  ngSrc="hero.jpg"
  width="1600"
  height="900"
  placeholder="data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAAAAEAAAAB..."
  priority
/>
```

Keep data URL placeholders under 4KB. Larger ones bloat your bundle and negate the performance benefit.

### Disabling Blur on Placeholder

```html
<img
  ngSrc="product.jpg"
  width="800"
  height="600"
  placeholder
  [placeholderConfig]="{ blur: false }"
/>
```

### Placeholder Without a Loader

If no image loader is configured, Angular cannot generate a placeholder automatically. Using the `placeholder` boolean attribute without a loader throws an error. You must provide a data URL string instead.

```html
<!-- ERROR: no loader configured, cannot auto-generate placeholder -->
<img ngSrc="photo.jpg" width="400" height="300" placeholder />

<!-- Works: explicit data URL -->
<img ngSrc="photo.jpg" width="400" height="300" placeholder="data:image/..." />
```

## Image Loaders

Loaders are functions that transform `ngSrc` + width into a full CDN URL. They are the key to unlocking CDN-specific optimizations (format negotiation, quality settings, resizing).

### Generic Loader (Default)

Without any loader configuration, `ngSrc` is used as-is for the `src` attribute. No width transformations, no CDN integration.

### Built-in CDN Loaders

Angular provides pre-configured loaders for popular CDNs:

```typescript
import {
  provideCloudflareLoader,
  provideCloudinaryLoader,
  provideImageKitLoader,
  provideImgixLoader,
  provideNetlifyLoader,
} from '@angular/common';
```

Register one in your providers:

```typescript
// app.config.ts
import { provideCloudinaryLoader } from '@angular/common';

export const appConfig: ApplicationConfig = {
  providers: [
    provideCloudinaryLoader('https://res.cloudinary.com/my-account'),
  ],
};
```

Then use just the path in templates:

```html
<!-- Cloudinary generates: https://res.cloudinary.com/my-account/image/upload/w_800/products/shoe.jpg -->
<img ngSrc="products/shoe.jpg" width="800" height="600" sizes="50vw" />
```

| Provider | CDN | Base URL Example |
|---|---|---|
| `provideCloudflareLoader` | Cloudflare Images | `https://yoursite.yourcdn.com` |
| `provideCloudinaryLoader` | Cloudinary | `https://res.cloudinary.com/your-account` |
| `provideImageKitLoader` | ImageKit | `https://ik.imagekit.io/your-id` |
| `provideImgixLoader` | Imgix | `https://your-subdomain.imgix.net` |
| `provideNetlifyLoader` | Netlify Image CDN | `https://yoursite.netlify.app` |

### CDN Transformations via `loaderParams`

Built-in loaders for Cloudinary, Cloudflare, ImageKit, and Imgix support the `transform` property inside `loaderParams`. Netlify does not support this.

```html
<!-- Cloudinary: apply grayscale and rounded corners -->
<img
  ngSrc="products/shoe.jpg"
  width="400"
  height="300"
  [loaderParams]="{ transform: 'e_grayscale,r_10' }"
/>

<!-- Object syntax also works -->
<img
  ngSrc="products/shoe.jpg"
  width="400"
  height="300"
  [loaderParams]="{ transform: { e: 'grayscale', r: 10 } }"
/>
```

### Custom Image Loader

For CDNs without a built-in loader, or for custom server setups, provide the `IMAGE_LOADER` token:

```typescript
import { IMAGE_LOADER, ImageLoaderConfig } from '@angular/common';

const myImageLoader = (config: ImageLoaderConfig): string => {
  const params: string[] = [];

  if (config.width) {
    params.push(`w=${config.width}`);
  }

  // Access custom properties from loaderParams
  if (config.loaderParams?.['quality']) {
    params.push(`q=${config.loaderParams['quality']}`);
  }

  if (config.loaderParams?.['format']) {
    params.push(`fmt=${config.loaderParams['format']}`);
  }

  const query = params.length ? `?${params.join('&')}` : '';
  return `https://cdn.example.com/images/${config.src}${query}`;
};

export const appConfig: ApplicationConfig = {
  providers: [
    {
      provide: IMAGE_LOADER,
      useValue: myImageLoader,
    },
  ],
};
```

Usage:

```html
<img
  ngSrc="hero-banner.jpg"
  width="1600"
  height="900"
  [loaderParams]="{ quality: 80, format: 'webp' }"
  priority
/>
<!-- Generates: https://cdn.example.com/images/hero-banner.jpg?w=1600&q=80&fmt=webp -->
```

### The `ImageLoaderConfig` Interface

```typescript
interface ImageLoaderConfig {
  src: string;              // the ngSrc value
  width?: number;           // requested width (from srcset generation)
  height?: number;          // requested height
  loaderParams?: {          // custom parameters from [loaderParams]
    [key: string]: any;
  };
  isPlaceholder?: boolean;  // true when generating placeholder URL
}
```

The `isPlaceholder` flag lets your loader apply different transforms for placeholder images (e.g., extreme compression, blur).

```typescript
const advancedLoader = (config: ImageLoaderConfig): string => {
  const width = config.width ?? 'auto';

  if (config.isPlaceholder) {
    return `https://cdn.example.com/${config.src}?w=${width}&blur=20&q=10`;
  }

  return `https://cdn.example.com/${config.src}?w=${width}&q=80`;
};
```

## Preconnect and Resource Hints

### Automatic Preconnect Detection

Angular uses static analysis to detect image CDN domains from configured loaders and auto-generates `<link rel="preconnect">` tags. This removes a round-trip of DNS lookup + TLS handshake before the first image request.

### Manual Preconnect

If you use a domain that Angular does not detect automatically, add it manually in `index.html`:

```html
<link rel="preconnect" href="https://cdn.example.com" />
```

### Development Warning

In dev mode, Angular warns if the LCP image's domain lacks a preconnect hint. This helps catch missing optimization before production.

### Suppressing Preconnect Warnings

If you handle preconnect outside Angular (e.g., through a CDN proxy on the same domain), suppress the warning:

```typescript
import { PRECONNECT_CHECK_BLOCKLIST } from '@angular/common';

providers: [
  {
    provide: PRECONNECT_CHECK_BLOCKLIST,
    useValue: 'https://same-origin-cdn.example.com',
  },
]
```

You can also provide an array of URLs:

```typescript
{
  provide: PRECONNECT_CHECK_BLOCKLIST,
  useValue: ['https://cdn1.example.com', 'https://cdn2.example.com'],
}
```

## SSR and Hydration

When using Angular's server-side rendering, `NgOptimizedImage` provides additional benefits:

- **Automatic `<link rel="preload">`:** For images with `priority`, Angular injects a preload link into the server-rendered HTML `<head>`. This tells the browser to start downloading the image immediately, even before the CSS and JavaScript are parsed.

- **No duplicate loading:** During hydration, Angular reuses the server-rendered `<img>` element rather than creating a new one, so the image is not re-fetched.

```html
<!-- Server-rendered output for a priority image -->
<head>
  <link rel="preload" as="image" href="https://cdn.example.com/hero.jpg" fetchpriority="high" />
</head>
<body>
  <img
    src="https://cdn.example.com/hero.jpg"
    fetchpriority="high"
    loading="eager"
    decoding="sync"
    width="1600"
    height="900"
  />
</body>
```

## Migrating Background Images to `NgOptimizedImage`

CSS `background-image` cannot be optimized by `NgOptimizedImage`. Replace them with `fill` mode images.

### Before

```css
.hero-section {
  background-image: url('/assets/hero.jpg');
  background-size: cover;
  background-position: center;
  height: 500px;
}
```

### After

```html
<div class="hero-section">
  <img ngSrc="hero.jpg" fill priority />
  <div class="hero-content">
    <h1>Welcome</h1>
  </div>
</div>
```

```css
.hero-section {
  position: relative;
  height: 500px;
}

.hero-section img {
  object-fit: cover;
  object-position: center;
  z-index: 0;
}

.hero-content {
  position: relative;
  z-index: 1;
}
```

## Development-Mode Validations

`NgOptimizedImage` runs several checks in development mode and logs warnings or errors:

| Check | Severity | What It Catches |
|---|---|---|
| Missing `width`/`height` without `fill` | Error | Layout shift risk |
| Oversized image | Warning | Downloading a file much larger than the rendered size |
| Distorted aspect ratio | Warning | `width`/`height` ratio does not match the natural ratio |
| LCP without `priority` | Warning | The largest visible image is lazy-loaded |
| Missing preconnect | Warning | The image CDN domain has no preconnect hint |
| Placeholder too large | Warning | Base64 placeholder exceeds 4KB |
| `priority` with `loading="lazy"` | Error | Conflicting directives |
| `fill` with `width`/`height` | Error | Redundant attributes in fill mode |

These checks do not run in production builds, so there is no runtime cost.

## Complete Attribute Reference

| Attribute | Type | Default | Description |
|---|---|---|---|
| `ngSrc` | `string` | Required | Image path (processed by loader) |
| `width` | `number` | Required* | Intrinsic or rendered width |
| `height` | `number` | Required* | Intrinsic or rendered height |
| `priority` | `boolean` | `false` | Marks as LCP image |
| `fill` | `boolean` | `false` | Image fills parent container |
| `sizes` | `string` | Undefined | Triggers automatic responsive srcset |
| `ngSrcset` | `string` | Undefined | Manual srcset descriptors |
| `loading` | `string` | `"lazy"` | `"lazy"`, `"eager"`, or `"auto"` |
| `decoding` | `string` | `"auto"` | `"auto"`, `"async"`, or `"sync"` |
| `placeholder` | `string \| boolean` | Undefined | Enable placeholder or provide data URL |
| `placeholderConfig` | `object` | Undefined | `{ blur: boolean }` |
| `loaderParams` | `object` | Undefined | Custom data passed to the image loader |
| `disableOptimizedSrcset` | `boolean` | `false` | Disables automatic srcset |

*Not required when `fill` is used.

## Common Mistakes

- **Not using `priority` on the LCP image.** This is the single highest-impact optimization. Angular warns in dev mode, but many developers dismiss the warning. Always check your page with Lighthouse or Chrome DevTools Performance panel to identify the LCP element and add `priority` to it.

- **Using `fill` without a positioned parent.** The image renders with `position: absolute`, so it needs a parent with `position: relative`, `absolute`, or `fixed`. Without this, the image escapes its container and overlaps other content.

- **Setting `width` and `height` to CSS rendered size instead of intrinsic size for responsive images.** If your image file is 2400x1600 but renders at 800x533, use `width="2400" height="1600"` with a `sizes` attribute. The actual rendered size is determined by CSS and the `sizes` attribute together.

- **Forgetting `sizes` for responsive images.** Without `sizes`, no `srcset` is generated. The browser downloads the full-size image regardless of viewport. Always pair responsive images with a `sizes` attribute.

- **Providing a Base64 placeholder over 4KB.** Large data URLs increase your JavaScript bundle size and slow initial parsing. Generate tiny placeholders (10-30px wide) or use the automatic placeholder feature with a CDN loader.

- **Using multiple loaders.** Angular supports one `IMAGE_LOADER` provider at a time. If you need images from multiple CDNs, write a custom loader that routes based on the `src` path.

- **Overriding `src` directly alongside `ngSrc`.** Never set both `src` and `ngSrc` on the same element. The directive controls `src` entirely.

---

## Best Practices

- **Audit LCP with Lighthouse.** Run Lighthouse in Chrome DevTools. It identifies the LCP element. If it is an image, add `priority` to that `<img>` tag.

- **Use a CDN loader in production.** The generic loader provides no resizing or format negotiation. A CDN loader automatically serves WebP/AVIF, resizes on the fly, and caches at the edge.

- **Set `sizes` on every non-fixed image.** Even a rough estimate like `"(max-width: 768px) 100vw, 50vw"` prevents the browser from downloading oversized images.

- **Prefer `fill` over manually calculating dimensions for dynamic content.** When images come from a CMS and dimensions are unknown, `fill` with `object-fit: cover` is the most reliable approach.

- **Configure `IMAGE_CONFIG` breakpoints to match your design breakpoints.** If your layout has breakpoints at 640, 1024, and 1440, include those in the breakpoints array.

- **Add preconnect hints for third-party image domains.** Even though Angular auto-detects most CDN domains, verify by checking the Network panel. Add `<link rel="preconnect">` for any domain that Angular misses.

- **Use `placeholder` for images below the fold that take time to load.** The blur-up effect gives users visual feedback and perceived performance improvement.

## Real-World Use Cases

### E-Commerce Product Gallery

```typescript
@Component({
  selector: 'app-product-gallery',
  standalone: true,
  imports: [NgOptimizedImage],
  template: `
    <!-- Hero image: LCP candidate -->
    <div class="hero-image">
      <img
        [ngSrc]="selectedImage()"
        width="1200"
        height="1200"
        sizes="(max-width: 768px) 100vw, 50vw"
        placeholder
        priority
      />
    </div>

    <!-- Thumbnail strip: lazy loaded, small fixed size -->
    <div class="thumbnails">
      @for (img of images(); track img.id) {
        <img
          [ngSrc]="img.path"
          width="80"
          height="80"
          (click)="selectImage(img.path)"
        />
      }
    </div>
  `,
  styles: `
    .hero-image { max-width: 600px; }
    .thumbnails { display: flex; gap: 8px; }
    .thumbnails img { cursor: pointer; border-radius: 4px; }
  `,
})
export class ProductGalleryComponent {
  images = input.required<{ id: string; path: string }[]>();
  selectedImage = signal('');

  constructor() {
    effect(() => {
      const imgs = this.images();
      if (imgs.length > 0 && !this.selectedImage()) {
        this.selectedImage.set(imgs[0].path);
      }
    });
  }

  selectImage(path: string) {
    this.selectedImage.set(path);
  }
}
```

### Blog Post with Mixed Content

```typescript
@Component({
  selector: 'app-blog-post',
  standalone: true,
  imports: [NgOptimizedImage],
  template: `
    <!-- Hero: full width, priority -->
    <div class="post-hero" style="position: relative; height: 400px;">
      <img [ngSrc]="post().heroImage" fill priority />
    </div>

    <article class="post-body">
      <h1>{{ post().title }}</h1>

      <!-- Inline content images: responsive, lazy -->
      @for (block of post().blocks; track block.id) {
        @if (block.type === 'text') {
          <p [innerHTML]="block.content"></p>
        } @else if (block.type === 'image') {
          <figure>
            <img
              [ngSrc]="block.src"
              [width]="block.width"
              [height]="block.height"
              sizes="(max-width: 768px) 100vw, 720px"
              placeholder
            />
            <figcaption>{{ block.caption }}</figcaption>
          </figure>
        }
      }
    </article>
  `,
  styles: `
    .post-hero img { object-fit: cover; }
    figure { margin: 24px 0; }
    figure img { border-radius: 8px; }
  `,
})
export class BlogPostComponent {
  post = input.required<BlogPost>();
}
```

### Custom Loader for a Self-Hosted Image Server

```typescript
// image-loader.ts
import { IMAGE_LOADER, ImageLoaderConfig } from '@angular/common';

export function provideCustomImageLoader(baseUrl: string) {
  return {
    provide: IMAGE_LOADER,
    useValue: (config: ImageLoaderConfig): string => {
      const width = config.width ? `&w=${config.width}` : '';
      const quality = config.loaderParams?.['quality']
        ? `&q=${config.loaderParams['quality']}`
        : '&q=80';
      const format = config.loaderParams?.['format']
        ? `&fmt=${config.loaderParams['format']}`
        : '&fmt=auto';

      // Placeholder: tiny size, heavy compression
      if (config.isPlaceholder) {
        return `${baseUrl}/resize?src=${config.src}&w=30&q=10&fmt=webp`;
      }

      return `${baseUrl}/resize?src=${config.src}${width}${quality}${format}`;
    },
  };
}

// app.config.ts
import { provideCustomImageLoader } from './image-loader';

export const appConfig: ApplicationConfig = {
  providers: [
    provideCustomImageLoader('https://images.myapp.com'),
  ],
};
```

```html
<!-- Template uses ngSrc with custom quality -->
<img
  ngSrc="uploads/user-photo-42.jpg"
  width="400"
  height="400"
  [loaderParams]="{ quality: 90, format: 'avif' }"
```
