# 13.9 `NgComponentOutlet` and `NgTemplateOutlet`

> Two directives for rendering dynamic content: `NgTemplateOutlet` stamps out template fragments with context, and `NgComponentOutlet` instantiates entire components at runtime.

---

## `ng-template` Foundations

Before diving into the outlet directives, you need to understand `<ng-template>`. It declares a template fragment that is not rendered immediately. The content inside `<ng-template>` only appears when something explicitly renders it.

```html
<p>This paragraph renders immediately.</p>

<ng-template>
  <p>This paragraph does NOT render. It is a dormant fragment.</p>
</ng-template>
```

### Getting a Reference to a Template Fragment

There are three ways to reference a template fragment.

**1. Template reference variable:**

```html
<ng-template #greeting>
  <p>Hello, world!</p>
</ng-template>
```

**2. Component query with `viewChild`:**

```typescript
@Component({
  template: `
    <ng-template #greeting>
      <p>Hello, world!</p>
    </ng-template>
  `,
})
export class AppComponent {
  greetingRef = viewChild.required<TemplateRef<unknown>>('greeting');
}
```

**3. Injection inside a directive on the `ng-template`:**

```typescript
@Directive({ selector: '[appLazy]', standalone: true })
export class LazyDirective {
  private template = inject(TemplateRef);
  private vcr = inject(ViewContainerRef);

  show() {
    this.vcr.createEmbeddedView(this.template);
  }
}
```

## `NgTemplateOutlet`

`NgTemplateOutlet` renders a template fragment at a specific location in the DOM. It is a structural directive from `@angular/common`.

### Basic Usage

```typescript
import { Component } from '@angular/core';
import { NgTemplateOutlet } from '@angular/common';

@Component({
  selector: 'app-demo',
  standalone: true,
  imports: [NgTemplateOutlet],
  template: `
    <!-- Define the fragment -->
    <ng-template #banner>
      <div class="banner">Welcome to our site!</div>
    </ng-template>

    <!-- Render it here -->
    <ng-container *ngTemplateOutlet="banner" />

    <!-- Render it again somewhere else -->
    <ng-container *ngTemplateOutlet="banner" />
  `,
})
export class DemoComponent {}
```

The fragment renders twice, each as a separate embedded view. Changes to bindings inside the fragment reflect in both instances because the bindings evaluate against the declaring component.

### Property Binding Syntax

You can also use the property binding form instead of the structural directive `*` syntax:

```html
<ng-container
  [ngTemplateOutlet]="banner"
  [ngTemplateOutletContext]="{ name: 'Angular' }"
/>
```

Both forms are equivalent. The `*` syntax is more concise for simple cases.

## Template Context with `ngTemplateOutletContext`

Context is how you pass data into a template fragment. The context is an object whose keys become available through `let-` declarations on the `<ng-template>`.

### Named Properties

```typescript
@Component({
  selector: 'app-demo',
  standalone: true,
  imports: [NgTemplateOutlet],
  template: `
    <ng-template #userCard let-name="userName" let-role="userRole">
      <div class="card">
        <h3>{{ name }}</h3>
        <span>{{ role }}</span>
      </div>
    </ng-template>

    <ng-container
      *ngTemplateOutlet="userCard; context: { userName: 'Alice', userRole: 'Admin' }"
    />

    <ng-container
      *ngTemplateOutlet="userCard; context: { userName: 'Bob', userRole: 'Editor' }"
    />
  `,
})
export class DemoComponent {}
```

`let-name="userName"` means: "Create a local variable called `name` whose value comes from the context object's `userName` property."

### The `$implicit` Convention

When the context object has a `$implicit` key, its value is available to any `let-` declaration that does not specify a property name.

```html
<ng-template let-item>
  <!-- "item" receives the $implicit value -->
  <p>{{ item }}</p>
</ng-template>

<ng-container
  *ngTemplateOutlet="myTemplate; context: { $implicit: 'Hello' }"
/>
<!-- Renders: <p>Hello</p> -->
```

You can combine `$implicit` with named properties:

```html
<ng-template let-item let-index="idx">
  <p>{{ index }}: {{ item }}</p>
</ng-template>

<ng-container
  *ngTemplateOutlet="myTemplate; context: { $implicit: 'Apple', idx: 0 }"
/>
<!-- Renders: <p>0: Apple</p> -->
```

### Dynamic Context

Context values can be signals, component properties, or any expression:

```typescript
@Component({
  standalone: true,
  imports: [NgTemplateOutlet],
  template: `
    <ng-template #status let-count="total" let-label="label">
      <span>{{ label }}: {{ count }}</span>
    </ng-template>

    <ng-container
      *ngTemplateOutlet="status; context: { total: itemCount(), label: 'Items' }"
    />
  `,
})
export class StatusComponent {
  itemCount = signal(42);
}
```

When `itemCount` updates, the rendered template updates automatically because Angular re-evaluates the context expression.

## Template Injector

The `ngTemplateOutletInjector` input controls which injector the embedded view uses for dependency injection.

```typescript
@Component({
  standalone: true,
  imports: [NgTemplateOutlet],
  template: `
    <ng-template #content>
      <app-child />
    </ng-template>

    <ng-container
      *ngTemplateOutlet="content; injector: customInjector"
    />
  `,
})
export class ParentComponent {
  customInjector = Injector.create({
    providers: [{ provide: CONFIG_TOKEN, useValue: { theme: 'dark' } }],
    parent: inject(Injector),
  });
}
```

Use the special string value `'outlet'` to inherit the injector from the outlet's location in the DOM rather than from the template's declaration location:

```html
<ng-container *ngTemplateOutlet="fragment; injector: 'outlet'" />
```

## Passing Templates as Component Inputs

A powerful pattern is accepting `TemplateRef` as a component input, allowing consumers to customize rendering.

```typescript
// table.component.ts
@Component({
  selector: 'app-data-table',
  standalone: true,
  imports: [NgTemplateOutlet],
  template: `
    <table>
      <thead>
        <tr>
          @for (col of columns(); track col) {
            <th>{{ col }}</th>
          }
        </tr>
      </thead>
      <tbody>
        @for (row of data(); track $index) {
          <tr>
            @for (col of columns(); track col) {
              <td>
                @if (cellTemplate()) {
                  <ng-container
                    *ngTemplateOutlet="cellTemplate()!; context: {
                      $implicit: row[col],
                      row: row,
                      column: col
                    }"
                  />
                } @else {
                  {{ row[col] }}
                }
              </td>
            }
          </tr>
        }
      </tbody>
    </table>
  `,
})
export class DataTableComponent {
  columns = input.required<string[]>();
  data = input.required<Record<string, any>[]>();
  cellTemplate = input<TemplateRef<any>>();
}
```

Consumer provides a custom cell template:

```html
<ng-template #customCell let-value let-row="row" let-col="column">
  @if (col === 'status') {
    <span [class]="'badge badge-' + value">{{ value }}</span>
  } @else if (col === 'avatar') {
    <img [src]="value" width="32" height="32" />
  } @else {
    {{ value }}
  }
</ng-template>

<app-data-table
  [columns]="['name', 'status', 'avatar']"
  [data]="users()"
  [cellTemplate]="customCell"
/>
```

### Using Content Queries Instead of Inputs

An alternative is using `@ContentChild` (or `contentChild` signal query) to grab a template projected through content:

```typescript
@Component({
  selector: 'app-list',
  standalone: true,
  imports: [NgTemplateOutlet],
  template: `
    <ul>
      @for (item of items(); track item.id) {
        <li>
          @if (itemTemplate()) {
            <ng-container
              *ngTemplateOutlet="itemTemplate()!; context: { $implicit: item, index: $index }"
            />
          } @else {
            {{ item.label }}
          }
        </li>
      }
    </ul>
  `,
})
export class ListComponent {
  items = input.required<{ id: string; label: string }[]>();
  itemTemplate = contentChild<TemplateRef<any>>('itemTpl');
}
```

```html
<app-list [items]="products()">
  <ng-template #itemTpl let-product let-i="index">
    <span class="index">{{ i + 1 }}.</span>
    <strong>{{ product.label }}</strong>
  </ng-template>
</app-list>
```

## `NgComponentOutlet`

`NgComponentOutlet` dynamically instantiates a component class and inserts its host view into the DOM. It is the declarative alternative to manually using `ViewContainerRef.createComponent()`.

### Basic Usage

```typescript
import { Component, Type } from '@angular/core';
import { NgComponentOutlet } from '@angular/common';

@Component({
  selector: 'app-greeter',
  standalone: true,
  template: `<p>Hello from GreeterComponent!</p>`,
})
export class GreeterComponent {}

@Component({
  selector: 'app-host',
  standalone: true,
  imports: [NgComponentOutlet],
  template: `
    <ng-container *ngComponentOutlet="componentType" />
  `,
})
export class HostComponent {
  componentType: Type<any> = GreeterComponent;
}
```

Angular creates an instance of `GreeterComponent` and inserts its template into the DOM at the outlet location.

### Setting `null` Destroys the Component

When the input becomes `null`, Angular destroys the existing component and clears the view:

```typescript
@Component({
  standalone: true,
  imports: [NgComponentOutlet],
  template: `
    <ng-container *ngComponentOutlet="activeComponent()" />
    <button (click)="clear()">Clear</button>
  `,
})
export class HostComponent {
  activeComponent = signal<Type<any> | null>(GreeterComponent);

  clear() {
    this.activeComponent.set(null); // destroys the component
  }
}
```

## Passing Inputs with `ngComponentOutletInputs`

Since Angular 16.2, you can pass an object of inputs directly. This replaced the older pattern of creating custom injectors to provide data.

```typescript
@Component({
  selector: 'app-alert',
  standalone: true,
  template: `
    <div [class]="'alert alert-' + severity()">
      <strong>{{ title() }}</strong>
      <p>{{ message() }}</p>
    </div>
  `,
})
export class AlertComponent {
  title = input('Alert');
  severity = input<'info' | 'warning' | 'error'>('info');
  message = input('');
}

@Component({
  standalone: true,
  imports: [NgComponentOutlet],
  template: `
    <ng-container
      *ngComponentOutlet="AlertComponent; inputs: alertInputs()"
    />
  `,
})
export class HostComponent {
  AlertComponent = AlertComponent;

  alertInputs = signal({
    title: 'Warning',
    severity: 'warning' as const,
    message: 'Disk space is running low.',
  });
}
```

When `alertInputs` changes, Angular updates the component's inputs. The keys in the object must match the input names on the target component.

### Before Angular 16.2 (Legacy Pattern)

Before `ngComponentOutletInputs` existed, developers had to create custom injectors:

```typescript
// Bad Practice: manual injector for passing data (pre-16.2 pattern)
const DATA_TOKEN = new InjectionToken<string>('data');

// In the host component:
customInjector = Injector.create({
  providers: [{ provide: DATA_TOKEN, useValue: 'some data' }],
  parent: inject(Injector),
});

// Template:
// <ng-container *ngComponentOutlet="comp; injector: customInjector" />

// In the dynamic component:
// data = inject(DATA_TOKEN);
```

This still works but is unnecessary for simple input passing. Use `ngComponentOutletInputs` instead.

## Accessing the Component Instance

Since Angular 19.1, `NgComponentOutlet` exposes a `componentInstance` property, accessible through a template reference variable or `viewChild` query.

```typescript
@Component({
  standalone: true,
  imports: [NgComponentOutlet],
  template: `
    <ng-container
      *ngComponentOutlet="PlayerComponent; inputs: playerInputs"
      #outlet="ngComponentOutlet"
    />
    <button (click)="callPlay()">Play</button>
  `,
})
export class HostComponent {
  PlayerComponent = PlayerComponent;
  playerInputs = { trackUrl: '/audio/song.mp3' };

  outlet = viewChild<NgComponentOutlet>('outlet');

  callPlay() {
    const player = this.outlet()?.componentInstance;
    if (player && 'play' in player) {
      (player as PlayerComponent).play();
    }
  }
}
```

`componentInstance` returns `T | null` where `T` is the component class. It is `null` when no component is rendered (the input is `null`).

## Custom Injector with `ngComponentOutletInjector`

For components that depend on injection tokens, provide a custom injector:

```typescript
import { Injector, InjectionToken } from '@angular/core';

export const THEME = new InjectionToken<string>('theme');

@Component({
  standalone: true,
  template: `<div [class]="'panel panel-' + theme">Panel</div>`,
})
export class PanelComponent {
  theme = inject(THEME);
}

@Component({
  standalone: true,
  imports: [NgComponentOutlet],
  template: `
    <ng-container *ngComponentOutlet="PanelComponent; injector: panelInjector" />
  `,
})
export class HostComponent {
  PanelComponent = PanelComponent;

  private parentInjector = inject(Injector);

  panelInjector = Injector.create({
    providers: [{ provide: THEME, useValue: 'dark' }],
    parent: this.parentInjector,
  });
}
```

## Content Projection with `ngComponentOutletContent`

The `ngComponentOutletContent` input accepts `Node[][]`, where each inner array represents a content projection slot.

```typescript
@Component({
  selector: 'app-card',
  standalone: true,
  template: `
    <div class="card-header"><ng-content /></div>
    <div class="card-body"><ng-content select="[body]" /></div>
  `,
})
export class CardComponent {}

@Component({
  standalone: true,
  imports: [NgComponentOutlet],
  template: `
    <ng-container
      *ngComponentOutlet="CardComponent; content: cardContent"
    />
  `,
})
export class HostComponent {
  CardComponent = CardComponent;

  cardContent: Node[][] = [];

  constructor() {
    // Slot 0: default content
    const header = document.createTextNode('Card Title');
    // Slot 1: [body] content
    const body = document.createTextNode('Card body text here');
    this.cardContent = [[header], [body]];
  }
}
```

Content projection with `NgComponentOutlet` requires raw DOM nodes, not Angular templates. This limits its usefulness compared to other content projection patterns. For most cases, prefer using `NgTemplateOutlet` or standard `<ng-content>` with static templates.

## Lazy Loading Components with `NgComponentOutlet`

A common pattern is loading components on demand using dynamic `import()`:

```typescript
@Component({
  standalone: true,
  imports: [NgComponentOutlet],
  template: `
    @if (chartComponent()) {
      <ng-container *ngComponentOutlet="chartComponent()!" />
    } @else {
      <button (click)="loadChart()">Show Chart</button>
    }
  `,
})
export class DashboardComponent {
  chartComponent = signal<Type<any> | null>(null);

  async loadChart() {
    const { ChartComponent } = await import('./chart/chart.component');
    this.chartComponent.set(ChartComponent);
  }
}
```

The component module is only downloaded when the user clicks the button. Angular then instantiates it and renders it through the outlet.

### Lazy Loading with Inputs

```typescript
async loadEditor() {
  const { EditorComponent } = await import('./editor/editor.component');
  this.editorComponent.set(EditorComponent);
  this.editorInputs.set({ content: this.document().body, readOnly: false });
}
```

```html
@if (editorComponent()) {
  <ng-container
    *ngComponentOutlet="editorComponent()!; inputs: editorInputs()"
  />
}
```

## Switching Between Components

`NgComponentOutlet` is ideal for scenarios where you switch between component types at runtime.

```typescript
@Component({
  standalone: true,
  imports: [NgComponentOutlet],
  template: `
    <nav>
      <button (click)="setView('list')">List View</button>
      <button (click)="setView('grid')">Grid View</button>
      <button (click)="setView('kanban')">Kanban View</button>
    </nav>

    <ng-container
      *ngComponentOutlet="activeView(); inputs: viewInputs()"
    />
  `,
})
export class ProjectBoardComponent {
  private viewMap: Record<string, Type<any>> = {
    list: ListViewComponent,
    grid: GridViewComponent,
    kanban: KanbanViewComponent,
  };

  private currentView = signal('list');

  activeView = computed(() => this.viewMap[this.currentView()]);

  viewInputs = computed(() => ({
    items: this.items(),
    filters: this.activeFilters(),
  }));

  items = signal<Task[]>([]);
  activeFilters = signal<Filter[]>([]);

  setView(view: string) {
    this.currentView.set(view);
  }
}
```

When the user clicks a view button, Angular destroys the current component and creates the new one. Inputs are passed through `ngComponentOutletInputs`.

## Comparison: `NgTemplateOutlet` vs `NgComponentOutlet`

| Aspect | `NgTemplateOutlet` | `NgComponentOutlet` |
|---|---|---|
| What it renders | A template fragment (`TemplateRef`) | A component class (`Type<T>`) |
| Data passing | Context object with `let-` bindings | `ngComponentOutletInputs` object |
| Own encapsulation | None (inherits from declaring component) | Full (own styles, change detection) |
| Lifecycle hooks | None | Full component lifecycle |
| Dependency injection | Inherits or custom via `ngTemplateOutletInjector` | Own node injector, customizable |
| Content projection | Not applicable | `ngComponentOutletContent` (raw nodes) |
| Lazy loading | Not directly (template must exist) | Yes, via dynamic `import()` |
| Access to instance | Not applicable | `componentInstance` (Angular 19.1+) |
| Use case | Template customization, render slots | Plugin systems, lazy features, view switching |

### When to Use Which

Use **`NgTemplateOutlet`** when:
- You want consumers to customize how data is rendered within your component.
- You need to render the same template fragment in multiple places.
- The template is defined in the same file or passed as a `TemplateRef` input.
- You do not need isolated change detection or styles.

Use **`NgComponentOutlet`** when:
- You need to render an entire component with its own styles, lifecycle, and encapsulation.
- You are building a plugin system where component types are determined at runtime.
- You need lazy loading of heavy components.
- Different component types share the same input contract but have different templates.

## `ViewContainerRef.createEmbeddedView` vs `NgTemplateOutlet`

`NgTemplateOutlet` is the declarative approach. The imperative alternative is `ViewContainerRef`:

```typescript
// Imperative approach
@Directive({ selector: '[appRenderTemplate]', standalone: true })
export class RenderTemplateDirective {
  private vcr = inject(ViewContainerRef);
  template = input.required<TemplateRef<any>>();
  context = input<any>({});

  constructor() {
    effect(() => {
      this.vcr.clear();
      this.vcr.createEmbeddedView(this.template(), this.context());
    });
  }
}
```

Prefer `NgTemplateOutlet` for straightforward cases. Use `ViewContainerRef` when you need fine-grained control over view creation, ordering, or multiple views in the same container.

## Common Mistakes

- **Forgetting to import the directive.** Both `NgTemplateOutlet` and `NgComponentOutlet` must be imported from `@angular/common` and added to the component's `imports` array. Without this, Angular silently ignores the directive and renders nothing.

- **Mismatching context keys and `let-` declarations.** If your context has `{ userName: 'Alice' }` but the template declares `let-name="user"`, the variable `name` is `undefined`. The key after `=` in `let-x="key"` must match a property name in the context object exactly.

- **Expecting output binding on `NgComponentOutlet`.** As of Angular 21, `NgComponentOutlet` does not support binding to component outputs. You cannot write `(someEvent)="handler()"` on the outlet. To react to events from a dynamic component, access `componentInstance` and subscribe to its outputs manually, or use a shared service.

- **Using `NgComponentOutlet` with non-standalone components without `ngComponentOutletNgModule`.** If the dynamic component is not standalone and depends on declarations from an NgModule, you must provide the module class through `ngComponentOutletNgModule`. Otherwise, the component's dependencies are unavailable.

- **Mutating the `ngComponentOutletInputs` object instead of replacing it.** Angular detects input changes by reference. If you mutate the same object, Angular does not see the change.

```typescript
// Bad Practice: mutating the existing object
this.inputs['title'] = 'New Title'; // Angular does NOT detect this

// Correct: replace with a new object
this.inputs.set({ ...this.inputs(), title: 'New Title' });
```

- **Creating `Node[][]` with Angular templates for content projection in `NgComponentOutlet`.** The `ngComponentOutletContent` input requires raw DOM `Node` objects, not `TemplateRef`. Attempting to pass template references here causes type errors or unexpected behavior.

---

## Best Practices

- **Prefer `NgTemplateOutlet` for render customization.** When building reusable components that need customizable rendering (tables, lists, dropdowns), accept `TemplateRef` inputs and render them with `NgTemplateOutlet`. This gives consumers full control without coupling to specific component types.

- **Use `NgComponentOutlet` for feature-level dynamics.** When entire features or views change at runtime (dashboard widgets, form field types, plugin systems), `NgComponentOutlet` is the right tool.

- **Always provide a fallback.** When the template or component input may be `null`, render default content:

```html
@if (customTemplate()) {
  <ng-container *ngTemplateOutlet="customTemplate()!" />
} @else {
  <span>{{ defaultText() }}</span>
}
```

- **Type your context objects.** Create an interface for template contexts to catch errors at compile time:

```typescript
interface UserCardContext {
  $implicit: User;
  index: number;
}

// In the component
userTemplate = input<TemplateRef<UserCardContext>>();
```

- **Use `computed()` for `NgComponentOutlet` inputs.** Derive the inputs object from signals so it updates reactively:

```typescript
viewInputs = computed(() => ({
  items: this.items(),
  page: this.currentPage(),
}));
```

## Real-World Use Cases

### Configurable Dashboard with Lazy Widgets

```typescript
interface WidgetConfig {
  id: string;
  loader: () => Promise<Type<any>>;
  inputs: Record<string, unknown>;
}

@Component({
  selector: 'app-dashboard',
  standalone: true,
  imports: [NgComponentOutlet],
  template: `
    <div class="dashboard-grid">
      @for (widget of loadedWidgets(); track widget.id) {
        <div class="widget-cell">
          <ng-container
            *ngComponentOutlet="widget.component; inputs: widget.inputs"
          />
        </div>
      }
    </div>
  `,
})
export class DashboardComponent {
  private widgetConfigs = input.required<WidgetConfig[]>();

  loadedWidgets = signal<{ id: string; component: Type<any>; inputs: Record<string, unknown> }[]>([]);

  constructor() {
    effect(async () => {
      const configs = this.widgetConfigs();
      const loaded = await Promise.all(
        configs.map(async (cfg) => {
          const component = await cfg.loader();
          return { id: cfg.id, component, inputs: cfg.inputs };
        })
      );
      this.loadedWidgets.set(loaded);
    });
  }
}

// Usage:
// dashboardConfig = [
//   {
//     id: 'sales',
//     loader: () => import('./widgets/sales.component').then(m => m.SalesWidgetComponent),
//     inputs: { period: 'monthly' },
//   },
//   {
//     id: 'users',
//     loader: () => import('./widgets/users.component').then(m => m.UsersWidgetComponent),
//     inputs: { limit: 10 },
//   },
// ];
```

### Reusable Dropdown with Custom Item Template

```typescript
@Component({
  selector: 'app-dropdown',
  standalone: true,
  imports: [NgTemplateOutlet],
  template: `
    <div class="dropdown" (click)="toggle()">
      <div class="dropdown-trigger">
        @if (selectedItem() && selectedTemplate()) {
          <ng-container
            *ngTemplateOutlet="selectedTemplate()!; context: { $implicit: selectedItem() }"
          />
        } @else {
          {{ selectedItem()?.label ?? placeholder() }}
        }
      </div>

      @if (isOpen()) {
        <ul class="dropdown-menu">
          @for (item of items(); track item.id) {
            <li (click)="select(item); $event.stopPropagation()">
              @if (itemTemplate()) {
                <ng-container
                  *ngTemplateOutlet="itemTemplate()!; context: { $implicit: item }"
                />
              } @else {
                {{ item.label }}
              }
            </li>
          }
        </ul>
      }
    </div>
  `,
})
export class DropdownComponent<T extends { id: string; label: string }> {
  items = input.required<T[]>();
  placeholder = input('Select...');
  itemTemplate = input<TemplateRef<{ $implicit: T }>>();
  selectedTemplate = input<TemplateRef<{ $implicit: T }>>();

  selected = output<T>();

  isOpen = signal(false);
  selectedItem = signal<T | null>(null);

  toggle() {
    this.isOpen.update((v) => !v);
  }

  select(item: T) {
    this.selectedItem.set(item);
    this.selected.emit(item);
    this.isOpen.set(false);
  }
}
```

Consumer with custom rendering:

```html
<ng-template #userItem let-user>
  <div class="user-option">
    <img [src]="user.avatar" width="24" height="24" />
    <span>{{ user.name }}</span>
    <small>{{ user.email }}</small>
  </div>
</ng-template>

<app-dropdown
  [items]="users()"
  [itemTemplate]="userItem"
  [selectedTemplate]="userItem"
  (selected)="onUserSelected($event)"
/>
```

### Dynamic Form Field Renderer

```typescript
interface FieldConfig {
  type: 'text' | 'select' | 'date' | 'checkbox';
  name: string;
  label: string;
  options?: string[];
}

@Component({
  selector: 'app-dynamic-form',
  standalone: true,
  imports: [NgComponentOutlet],
  template: `
    <form>
      @for (field of fields(); track field.name) {
        <div class="form-group">
          <label>{{ field.label }}</label>
          <ng-container
            *ngComponentOutlet="getComponent(field.type); inputs: { config: field }"
          />
        </div>
      }
    </form>
  `,
})
export class DynamicFormComponent {
  fields = input.required<FieldConfig[]>();

  private componentMap: Record<string, Type<any>> = {
    text: TextFieldComponent,
    select: SelectFieldComponent,
    date: DateFieldComponent,
    checkbox: CheckboxFieldComponent,
  };

  getComponent(type: string): Type<any> {
    return this.componentMap[type];
  }
}
```

Each field component has a `config` input and handles its own rendering, validation, and state. The form component does not need to know the implementation details of any field type.

---

> **Key Takeaway:** Use `NgTemplateOutlet` to let consumers customize how data renders within your component (render slots), and use `NgComponentOutlet` when you need to instantiate entire components dynamically at runtime (plugin systems, lazy loading, view switching).
