# 14.1 Services and `@Injectable`

> Services are classes that encapsulate reusable logic, data access, and shared state, made available to components and other services through Angular's dependency injection system.

---

## What Is a Service?

A service is any class whose purpose is to provide functionality that is not directly tied to a single component's view. Services handle:

- **Data access:** HTTP calls, localStorage, IndexedDB.
- **Business logic:** Calculations, transformations, validation rules.
- **Shared state:** Data that multiple components need to read or write.
- **Integration:** Third-party SDKs, browser APIs, WebSocket connections.

Angular does not enforce a strict definition. Any class decorated with `@Injectable()` can be a service. The convention is to name files with a `.service.ts` suffix and classes with a `Service` suffix.

---

## Creating a Service

### With the Angular CLI

```bash
ng generate service services/user
```

This generates `src/app/services/user.service.ts` and a corresponding `.spec.ts` test file.

### Manually

```typescript
import { Injectable } from '@angular/core';

@Injectable({
  providedIn: 'root',
})
export class UserService {
  private users: User[] = [];

  getAll(): User[] {
    return this.users;
  }

  getById(id: string): User | undefined {
    return this.users.find(u => u.id === id);
  }

  add(user: User): void {
    this.users.push(user);
  }
}
```

Two things make this a service:

1. The `@Injectable()` decorator tells Angular this class participates in dependency injection.
2. The `providedIn: 'root'` option registers it as a singleton available application-wide.

---

## The `@Injectable()` Decorator

`@Injectable()` marks a class so Angular's DI system can create and deliver instances of it. Without this decorator, Angular cannot resolve the class when something requests it through injection.

### When Is `@Injectable()` Required?

Every service that uses dependency injection (either as a provider or as a consumer that injects other services) needs `@Injectable()`.

```typescript
// This service injects HttpClient, so it MUST have @Injectable()
@Injectable({ providedIn: 'root' })
export class ApiService {
  private http = inject(HttpClient);

  fetchData() {
    return this.http.get('/api/data');
  }
}
```

```typescript
// Bad Practice: missing @Injectable() on a service that injects dependencies
export class BrokenService {
  private http = inject(HttpClient); // Runtime error: no provider context
}
```

### `@Injectable()` Without `providedIn`

You can use `@Injectable()` without `providedIn`. In this case, the service is not automatically registered anywhere. You must manually add it to a `providers` array.

```typescript
@Injectable() // no providedIn
export class LoggerService {
  log(message: string) {
    console.log(`[LOG] ${message}`);
  }
}

// Must be provided manually somewhere:
@Component({
  selector: 'app-dashboard',
  providers: [LoggerService], // registered here
  template: `...`,
})
export class DashboardComponent {
  private logger = inject(LoggerService);
}
```

This pattern is useful when you want a new instance per component rather than a singleton.

## `providedIn` Options

The `providedIn` property controls where Angular registers the service and how many instances exist.

### `providedIn: 'root'`

Creates a single shared instance for the entire application. This is the default and recommended option for most services.

```typescript
@Injectable({ providedIn: 'root' })
export class AuthService {
  private currentUser = signal<User | null>(null);

  isLoggedIn = computed(() => this.currentUser() !== null);

  login(user: User) {
    this.currentUser.set(user);
  }

  logout() {
    this.currentUser.set(null);
  }
}
```

Key behaviors:
- Singleton: one instance shared across all components, directives, pipes, and other services.
- Tree-shakable: if no component or service injects `AuthService`, the bundler removes it from the final build.
- No additional configuration needed: it works without adding the service to any `providers` array.

### `providedIn: 'any'`

Creates a separate instance for each lazy-loaded module boundary. All eagerly loaded components share one instance, but each lazy-loaded route gets its own.

```typescript
@Injectable({ providedIn: 'any' })
export class FeatureStateService {
  items = signal<Item[]>([]);
}
```

| Module Type | Instance |
|---|---|
| Eagerly loaded components | Share one instance |
| Lazy-loaded route A | Gets its own instance |
| Lazy-loaded route B | Gets its own instance |

This is rarely needed. It was more relevant in NgModule-based architectures. In standalone applications with route-level lazy loading, component-level providers achieve the same result more explicitly.

### `providedIn: 'platform'`

Registers the service at the platform injector level. This is only relevant when multiple Angular applications run on the same page (e.g., Angular Elements or micro-frontends).

```typescript
@Injectable({ providedIn: 'platform' })
export class SharedAnalyticsService {
  track(event: string) {
    // Shared across multiple Angular apps on the same page
  }
}
```

Most applications never need this.

### Summary Table

| `providedIn` | Scope | Instances | Tree-Shakable | Use Case |
|---|---|---|---|---|
| `'root'` | Application | 1 (singleton) | Yes | Default for most services |
| `'any'` | Module boundary | 1 per lazy module | Yes | Isolated state per feature |
| `'platform'` | Platform | 1 across apps | Yes | Multi-app pages |
| (omitted) | Manual | Depends on where provided | No | Component-scoped services |

## Providing Services Without `providedIn`

When you omit `providedIn`, you must register the service explicitly. There are several locations where you can do this.

### Application-Level Providers

In `app.config.ts`, services are available to the entire application:

```typescript
// app.config.ts
import { ApplicationConfig } from '@angular/core';
import { LoggerService } from './services/logger.service';

export const appConfig: ApplicationConfig = {
  providers: [
    LoggerService,
  ],
};
```

Services registered here are NOT tree-shakable. The bundler always includes them because they are explicitly referenced in the providers array.

### Component-Level Providers

Each component (or directive) can provide its own instance of a service:

```typescript
@Component({
  selector: 'app-editor',
  standalone: true,
  providers: [UndoService], // new instance for each EditorComponent
  template: `...`,
})
export class EditorComponent {
  private undo = inject(UndoService);
}
```

When `EditorComponent` is created, Angular creates a new `UndoService` instance. When the component is destroyed, the service instance is also destroyed. Child components of `EditorComponent` share the same `UndoService` instance unless they provide their own.

### Route-Level Providers

Services can be scoped to a route and its children:

```typescript
export const routes: Routes = [
  {
    path: 'admin',
    providers: [AdminService],
    children: [
      { path: 'users', component: UserListComponent },
      { path: 'settings', component: SettingsComponent },
    ],
  },
];
```

`AdminService` is available to `UserListComponent`, `SettingsComponent`, and all their descendants. It is NOT available outside the `/admin` route. When the user navigates away, the service instance is destroyed.

## Injecting Services

### The `inject()` Function (Preferred)

The modern, recommended way to consume services is with the `inject()` function:

```typescript
import { Component, inject } from '@angular/core';
import { UserService } from './services/user.service';

@Component({
  selector: 'app-user-list',
  standalone: true,
  template: `
    @for (user of userService.getAll(); track user.id) {
      <div>{{ user.name }}</div>
    }
  `,
})
export class UserListComponent {
  protected userService = inject(UserService);
}
```

`inject()` can be called:
- In class field initializers (most common).
- In constructor bodies.
- In factory functions that run within an injection context.

```typescript
// Field initializer
export class MyComponent {
  private auth = inject(AuthService);
  private router = inject(Router);
}

// Constructor body
export class MyComponent {
  private auth: AuthService;

  constructor() {
    this.auth = inject(AuthService);
  }
}
```

### Constructor Injection (Legacy)

The older pattern uses constructor parameters with type annotations:

```typescript
// Legacy pattern: still works but not recommended for new code
@Component({ ... })
export class UserListComponent {
  constructor(private userService: UserService) {}
}
```

This requires the `emitDecoratorMetadata` TypeScript compiler option. The `inject()` function does not need this option and is the preferred approach for all new Angular code.

### Why `inject()` Is Preferred Over Constructor Injection

| Aspect | `inject()` | Constructor injection |
|---|---|---|
| Works in field initializers | Yes | No |
| Works in functions/guards | Yes | No (needs a class) |
| Requires `emitDecoratorMetadata` | No | Yes |
| Supports `options` (optional, self, skipSelf, host) | Yes | Via parameter decorators |
| Type safety | Full | Full |
| Minification safe | Yes | Needs metadata |
| Testability | Same | Same |

## Services with Signals

Modern Angular services use signals for reactive state management. This replaces the older pattern of BehaviorSubject + Observable.

### Signal-Based Service

```typescript
@Injectable({ providedIn: 'root' })
export class CartService {
  // Private writable signals
  private items = signal<CartItem[]>([]);

  // Public read-only signals for consumers
  readonly cartItems = this.items.asReadonly();
  readonly itemCount = computed(() => this.items().length);
  readonly total = computed(() =>
    this.items().reduce((sum, item) => sum + item.price * item.quantity, 0)
  );

  addItem(product: Product, quantity = 1) {
    this.items.update(items => {
      const existing = items.find(i => i.productId === product.id);
      if (existing) {
        return items.map(i =>
          i.productId === product.id
            ? { ...i, quantity: i.quantity + quantity }
            : i
        );
      }
      return [...items, { productId: product.id, name: product.name, price: product.price, quantity }];
    });
  }

  removeItem(productId: string) {
    this.items.update(items => items.filter(i => i.productId !== productId));
  }

  clear() {
    this.items.set([]);
  }
}
```

### Consuming the Signal-Based Service in a Component

```typescript
@Component({
  selector: 'app-cart-summary',
  standalone: true,
  template: `
    <div class="cart-summary">
      <span>{{ cartService.itemCount() }} items</span>
      <span>{{ cartService.total() | currency }}</span>
    </div>
  `,
})
export class CartSummaryComponent {
  protected cartService = inject(CartService);
}
```

Because signals are reactive, the template updates automatically when cart state changes. No `async` pipe, no manual subscription management.

### The BehaviorSubject Pattern (Legacy)

Before signals, services used RxJS BehaviorSubjects:

```typescript
// Legacy pattern: still found in many codebases
@Injectable({ providedIn: 'root' })
export class CartServiceLegacy {
  private items$ = new BehaviorSubject<CartItem[]>([]);

  readonly cartItems$ = this.items$.asObservable();
  readonly itemCount$ = this.items$.pipe(map(items => items.length));
  readonly total$ = this.items$.pipe(
    map(items => items.reduce((sum, i) => sum + i.price * i.quantity, 0))
  );

  addItem(product: Product) {
    const current = this.items$.getValue();
    this.items$.next([...current, { ... }]);
  }
}
```

This still works, but signal-based services are simpler: no subscription management, no `async` pipe in templates, and direct synchronous reads.

## Service Lifecycle

### Singleton Services (`providedIn: 'root'`)

Created the first time they are injected. They live for the entire application lifetime. They are never destroyed (unless the application itself is destroyed).

```typescript
@Injectable({ providedIn: 'root' })
export class AppConfigService {
  constructor() {
    console.log('AppConfigService created'); // logs once, ever
  }
}
```

### Component-Scoped Services

Created when the component is created. Destroyed when the component is destroyed. The service can use `DestroyRef` to run cleanup logic.

```typescript
@Injectable()
export class TimerService {
  private intervalId: ReturnType<typeof setInterval> | null = null;
  private destroyRef = inject(DestroyRef);

  elapsed = signal(0);

  constructor() {
    this.intervalId = setInterval(() => {
      this.elapsed.update(n => n + 1);
    }, 1000);

    this.destroyRef.onDestroy(() => {
      if (this.intervalId) {
        clearInterval(this.intervalId);
        this.intervalId = null;
      }
    });
  }
}

@Component({
  selector: 'app-stopwatch',
  standalone: true,
  providers: [TimerService], // new instance per component
  template: `<p>{{ timer.elapsed() }}s</p>`,
})
export class StopwatchComponent {
  protected timer = inject(TimerService);
}
```

Each `<app-stopwatch>` gets its own `TimerService`. When the component is removed from the DOM, the interval is cleared.

## Service Inheritance

Services can extend other services, but this pattern should be used sparingly. Prefer composition (injecting one service into another) over inheritance.

```typescript
@Injectable({ providedIn: 'root' })
export class BaseApiService {
  protected http = inject(HttpClient);

  protected get<T>(url: string) {
    return this.http.get<T>(url);
  }

  protected post<T>(url: string, body: unknown) {
    return this.http.post<T>(url, body);
  }
}

@Injectable({ providedIn: 'root' })
export class UserApiService extends BaseApiService {
  getUsers() {
    return this.get<User[]>('/api/users');
  }

  createUser(data: CreateUserDto) {
    return this.post<User>('/api/users', data);
  }
}
```

### Composition Over Inheritance

```typescript
// Preferred: composition
@Injectable({ providedIn: 'root' })
export class UserApiService {
  private http = inject(HttpClient);

  getUsers() {
    return this.http.get<User[]>('/api/users');
  }

  createUser(data: CreateUserDto) {
    return this.http.post<User>('/api/users', data);
  }
}
```

There is rarely a good reason for a base API service class. Each service can inject `HttpClient` directly. The extra layer of abstraction adds complexity without benefit.

## Tree-Shaking and Bundle Size

### How Tree-Shaking Works with `providedIn: 'root'`

When a service uses `providedIn: 'root'`, Angular registers it through a static property on the class itself. If no code imports or injects the service, the bundler (esbuild) removes the entire class from the output.

```typescript
// This service is tree-shakable
@Injectable({ providedIn: 'root' })
export class UnusedService {
  // If nothing imports/injects this, it is removed from the bundle
}
```

### When Tree-Shaking Breaks

Providing a service in a `providers` array creates an explicit import reference. The bundler sees the reference and includes the service, even if nothing injects it.

```typescript
// Bad Practice: explicit providers array prevents tree-shaking
export const appConfig: ApplicationConfig = {
  providers: [
    UnusedService, // always in the bundle, even if nothing injects it
  ],
};
```

### The Rule

Use `providedIn: 'root'` for application-wide singletons. Only use `providers` arrays when you need component-scoped or route-scoped instances.

## Testing Services

### Testing a Simple Service

```typescript
describe('UserService', () => {
  let service: UserService;

  beforeEach(() => {
    TestBed.configureTestingModule({});
    service = TestBed.inject(UserService);
  });

  it('should add and retrieve users', () => {
    service.add({ id: '1', name: 'Alice', email: 'alice@example.com' });
    const users = service.getAll();
    expect(users).toHaveLength(1);
    expect(users[0].name).toBe('Alice');
  });

  it('should find user by id', () => {
    service.add({ id: '1', name: 'Alice', email: 'alice@example.com' });
    const user = service.getById('1');
    expect(user?.name).toBe('Alice');
  });
});
```

### Testing a Service with Dependencies

```typescript
describe('UserApiService', () => {
  let service: UserApiService;
  let httpMock: HttpTestingController;

  beforeEach(() => {
    TestBed.configureTestingModule({
      providers: [provideHttpClient(), provideHttpClientTesting()],
    });
    service = TestBed.inject(UserApiService);
    httpMock = TestBed.inject(HttpTestingController);
  });

  afterEach(() => {
    httpMock.verify(); // ensure no outstanding requests
  });

  it('should fetch users', () => {
    const mockUsers: User[] = [{ id: '1', name: 'Alice', email: 'a@b.com' }];

    service.getUsers().subscribe(users => {
      expect(users).toEqual(mockUsers);
    });

    const req = httpMock.expectOne('/api/users');
    expect(req.request.method).toBe('GET');
    req.flush(mockUsers);
  });
});
```

### Mocking Services in Component Tests

```typescript
describe('UserListComponent', () => {
  it('should display users', () => {
    const mockUserService = {
      getAll: () => [
        { id: '1', name: 'Alice' },
        { id: '2', name: 'Bob' },
      ],
    };

    TestBed.configureTestingModule({
      imports: [UserListComponent],
      providers: [
        { provide: UserService, useValue: mockUserService },
      ],

    const fixture = TestBed.createComponent(UserListComponent);
    fixture.detectChanges();

    const items = fixture.nativeElement.querySelectorAll('div');
    expect(items.length).toBe(2);
    expect(items[0].textContent).toContain('Alice');
  });
});
```

## Common Mistakes

- **Forgetting `@Injectable()` on a service that injects dependencies.** If your service uses `inject()` or has constructor parameters, it must have the `@Injectable()` decorator. Without it, Angular throws a runtime error because it cannot create the class within an injection context.

- **Providing `providedIn: 'root'` services again in component `providers`.** This creates a second instance scoped to that component, breaking the singleton pattern. If a component provides a root service, it gets its own copy, and state changes in the root instance are invisible to it:

```typescript
// Bad Practice: re-providing a root singleton
@Injectable({ providedIn: 'root' })
export class AuthService { ... }

@Component({
  providers: [AuthService], // WRONG: creates a second instance
})
export class LoginComponent {
  private auth = inject(AuthService); // this is NOT the root instance
}
```

- **Using `providers` arrays for services that should be tree-shakable.** Adding a service to `ApplicationConfig.providers` or a component's `providers` creates an explicit import that prevents the bundler from removing the service if it is unused. Use `providedIn: 'root'` for global singletons.

- **Injecting services outside an injection context.** `inject()` only works during class construction (field initializers or constructor body) or in functions Angular calls within an injection context (guards, resolvers, factory providers). Calling `inject()` in a regular method or callback throws:

```typescript
// Bad Practice: inject() outside injection context
@Injectable({ providedIn: 'root' })
export class BadService {
  getData() {
    const http = inject(HttpClient); // ERROR: not in injection context
    return http.get('/api/data');
  }
}

// Correct: inject in field initializer
@Injectable({ providedIn: 'root' })
export class GoodService {
  private http = inject(HttpClient); // injection context: field initializer

  getData() {
    return this.http.get('/api/data');
  }
}
```

- **Making services do too many things.** A service that handles authentication, user preferences, API calls, and caching violates the Single Responsibility Principle. Split into focused services: `AuthService`, `PreferencesService`, `UserApiService`, `CacheService`.

- **Exposing mutable state directly.** If a service exposes a writable signal or plain array, any consumer can modify state unpredictably. Expose read-only signals and provide methods for mutations:

```typescript
// Bad Practice: exposing writable signal
@Injectable({ providedIn: 'root' })
export class TodoService {
  todos = signal<Todo[]>([]); // any component can call todos.set([])
}

// Correct: expose readonly, provide methods
@Injectable({ providedIn: 'root' })
export class TodoService {
  private _todos = signal<Todo[]>([]);
  readonly todos = this._todos.asReadonly();

  add(todo: Todo) {
    this._todos.update(list => [...list, todo]);
  }
}
```

---

## Best Practices

- **Default to `providedIn: 'root'`.** Unless you have a specific reason for component-scoped or route-scoped instances, use `providedIn: 'root'`. It gives you a singleton, tree-shaking, and zero configuration.

- **Use `inject()` over constructor injection.** The `inject()` function works in more contexts (field initializers, functional guards, factory functions), does not require TypeScript metadata, and is the direction Angular is moving.

- **Keep services focused.** One service per responsibility. A `NotificationService` sends notifications. A `NotificationPreferencesService` manages user preferences. Do not combine them.

- **Use signals for reactive state in services.** Expose `computed()` and `asReadonly()` signals. This provides reactive updates without subscription management and integrates cleanly with signal-based templates.

- **Use `DestroyRef` for cleanup in non-singleton services.** Component-scoped services need cleanup (intervals, WebSocket connections, manual subscriptions). Inject `DestroyRef` and register cleanup callbacks.

- **Prefer composition over inheritance for services.** Inject one service into another rather than using class inheritance. Composition is more flexible, avoids tight coupling, and makes dependencies explicit.

## Real-World Use Cases

### Notification Service with Signal State

```typescript
interface AppNotification {
  id: string;
  message: string;
  type: 'success' | 'error' | 'info' | 'warning';
  timestamp: number;
}

@Injectable({ providedIn: 'root' })
export class NotificationService {
  private _notifications = signal<AppNotification[]>([]);
  readonly notifications = this._notifications.asReadonly();
  readonly unreadCount = computed(() => this._notifications().length);

  show(message: string, type: AppNotification['type'] = 'info', duration = 5000) {
    const notification: AppNotification = {
      id: crypto.randomUUID(),
      message,
      type,
      timestamp: Date.now(),
    };

    this._notifications.update(list => [...list, notification]);

    if (duration > 0) {
      setTimeout(() => this.dismiss(notification.id), duration);
    }
  }

  dismiss(id: string) {
    this._notifications.update(list => list.filter(n => n.id !== id));
  }

  clearAll() {
    this._notifications.set([]);
  }
}
```

```typescript
// Any component can inject and use it
@Component({
  selector: 'app-notification-toast',
  standalone: true,
  template: `
    @for (notification of notificationService.notifications(); track notification.id) {
      <div [class]="'toast toast-' + notification.type" (click)="notificationService.dismiss(notification.id)">
        {{ notification.message }}
      </div>
    }
  `,
})
export class NotificationToastComponent {
  protected notificationService = inject(NotificationService);
}
```

### Feature-Scoped Service with Route Providers

```typescript
@Injectable() // no providedIn: scoped to route
export class ProjectEditorService {
  private http = inject(HttpClient);
  private destroyRef = inject(DestroyRef);

  project = signal<Project | null>(null);
  isDirty = signal(false);

  load(projectId: string) {
    this.http.get<Project>(`/api/projects/${projectId}`).subscribe(p => {
      this.project.set(p);
      this.isDirty.set(false);
    });
  }

  updateField(field: keyof Project, value: unknown) {
    this.project.update(p => p ? { ...p, [field]: value } : p);
    this.isDirty.set(true);
  }

  save() {
    const p = this.project();
    if (!p) return;
    this.http.put(`/api/projects/${p.id}`, p).subscribe(() => {
      this.isDirty.set(false);
    });
  }
}

// Route configuration
export const projectRoutes: Routes = [
  {
    path: ':projectId',
    providers: [ProjectEditorService], // one instance per project route
    children: [
      { path: '', component: ProjectOverviewComponent },
      { path: 'settings', component: ProjectSettingsComponent },
      { path: 'members', component: ProjectMembersComponent },
    ],
  },
];
```

All child components share the same `ProjectEditorService` instance. When the user navigates to a different project, the old instance is destroyed and a new one is created.

### Service Coordinating Multiple Data Sources

```typescript
@Injectable({ providedIn: 'root' })
export class DashboardService {
  private userApi = inject(UserApiService);
  private orderApi = inject(OrderApiService);
  private analyticsApi = inject(AnalyticsApiService);

  dashboardData = rxResource({
    loader: () => forkJoin({
      recentUsers: this.userApi.getRecent(5),
      pendingOrders: this.orderApi.getPending(),
      todayStats: this.analyticsApi.getToday(),
    }),
  });

  readonly isLoading = this.dashboardData.isLoading;
  readonly error = this.dashboardData.error;

  readonly stats = computed(() => {
    const data = this.dashboardData.value();
    if (!data) return null;
    return {
      userCount: data.recentUsers.length,
      orderCount: data.pendingOrders.length,
      revenue: data.todayStats.revenue,
    };
  });

  refresh() {
    this.dashboardData.reload();
  }
}
```

The `DashboardService` composes three API services into a single reactive data source. Components inject `DashboardService` and bind directly to its signals without knowing about the underlying API structure.

---

> **Key Takeaway:** Use `@Injectable({ providedIn: 'root' })` for application-wide singletons with automatic tree-shaking, inject services with the `inject()` function in field initializers, expose reactive state through read-only signals and computed values, and scope services to components or routes only when you need isolated instances.
