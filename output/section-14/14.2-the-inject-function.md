# 14.2 The `inject()` Function

> The `inject()` function retrieves a dependency from the current injector, replacing constructor parameter injection with a more flexible, composable approach that works in field initializers, factory functions, and custom injection utilities.

---

## Basic Syntax

```typescript
import { inject } from '@angular/core';

@Component({
  selector: 'app-dashboard',
  standalone: true,
  template: `...`,
})
export class DashboardComponent {
  private userService = inject(UserService);
  private router = inject(Router);
}
```

`inject(token)` looks up the token in the current injector hierarchy and returns the resolved instance. If the token is not found, it throws an error (unless the `optional` flag is set).

---

## Where `inject()` Can Be Called

`inject()` only works inside an **injection context**. Angular establishes an injection context in specific situations. Calling `inject()` outside these contexts throws the `NG0203` error.

### 1. Class Field Initializers

The most common and recommended location:

```typescript
@Component({ ... })
export class ProductComponent {
  // Field initializers run during construction, inside the injection context
  private http = inject(HttpClient);
  private route = inject(ActivatedRoute);
  private productId = input.required<string>();
}
```

### 2. Constructor Body

```typescript
@Component({ ... })
export class ProductComponent {
  private http: HttpClient;
  private route: ActivatedRoute;

  constructor() {
    this.http = inject(HttpClient);
    this.route = inject(ActivatedRoute);
  }
}
```

Both field initializers and constructor body run within the same injection context. Field initializers are preferred because they are more concise.

### 3. Factory Functions in Providers

```typescript
const API_URL = new InjectionToken<string>('API_URL', {
  providedIn: 'root',
  factory: () => {
    // inject() works here because this is a factory function
    const config = inject(AppConfigService);
    return config.apiBaseUrl;
  },
});
```

### 4. `useFactory` in Provider Definitions

```typescript
providers: [
  {
    provide: LoggerService,
    useFactory: () => {
      const isDev = inject(IS_DEV_MODE);
      return isDev ? new ConsoleLogger() : new RemoteLogger(inject(HttpClient));
    },
  },
]
```

### 5. Functional Route Guards and Resolvers

Angular's functional guards and resolvers run within an injection context:

```typescript
export const authGuard: CanActivateFn = (route, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (auth.isLoggedIn()) {
    return true;
  }

  return router.createUrlTree(['/login'], {
    queryParams: { returnUrl: state.url },
  });
};
```

### 6. `afterRender` and `afterNextRender` Callbacks

```typescript
@Component({ ... })
export class ChartComponent {
  constructor() {
    afterNextRender(() => {
      const el = inject(ElementRef).nativeElement;
      // DOM is ready, initialize chart library
    });
  }
}
```

### Where `inject()` CANNOT Be Called

```typescript
@Component({ ... })
export class BrokenComponent {
  private http = inject(HttpClient); // OK: field initializer

  ngOnInit() {
    const router = inject(Router); // ERROR: NG0203 - lifecycle hooks are NOT an injection context
  }

  onClick() {
    const auth = inject(AuthService); // ERROR: NG0203 - methods are NOT an injection context
  }

  async loadData() {
    const data = await fetch('/api');
    const parser = inject(DataParser); // ERROR: NG0203 - async callbacks are NOT
  }
}
```

The rule: `inject()` must be called synchronously during class construction or inside a function that Angular explicitly runs within an injection context.

## `InjectOptions`

The second parameter to `inject()` is an options object that controls how Angular resolves the dependency.

```typescript
interface InjectOptions {
  optional?: boolean;
  self?: boolean;
  skipSelf?: boolean;
  host?: boolean;
}
```

### `optional: true`

Returns `null` instead of throwing when the token is not found:

```typescript
@Component({ ... })
export class TooltipComponent {
  // May or may not exist depending on the context
  private parentForm = inject(NgForm, { optional: true });

  ngOnInit() {
    if (this.parentForm) {
      // We are inside a form, register with it
    }
  }
}
```

Without `optional`, a missing provider throws `NullInjectorError`. With `optional: true`, you get `null`.

```typescript
// Type implications
const required = inject(SomeService);            // Type: SomeService
const optional = inject(SomeService, { optional: true }); // Type: SomeService | null
```

### `self: true`

Restricts resolution to the current component's own injector. Angular does NOT look up the hierarchy.

```typescript
@Component({
  selector: 'app-editor',
  providers: [UndoService], // provided locally
  template: `...`,
})
export class EditorComponent {
  // Only resolves if UndoService is in THIS component's providers
  private undo = inject(UndoService, { self: true });
}
```

If `UndoService` is not in `EditorComponent`'s own `providers`, Angular throws even if a parent provides it. Combine with `optional` to get `null` instead:

```typescript
private undo = inject(UndoService, { self: true, optional: true });
```

### `skipSelf: true`

Starts resolution from the parent injector, skipping the current component's own providers.

```typescript
@Component({
  selector: 'app-child',
  providers: [ConfigService], // this local provider is SKIPPED
  template: `...`,
})
export class ChildComponent {
  // Resolves ConfigService from a parent injector, ignoring local providers
  private parentConfig = inject(ConfigService, { skipSelf: true });
}
```

This is useful when a component provides its own instance of a service but also needs access to the parent's instance.

### `host: true`

Restricts resolution to injectors up to and including the host element's injector. In practice, this means the search stops at the component boundary when looking from within a directive or projected content.

```typescript
@Directive({
  selector: '[appValidation]',
  standalone: true,
})
export class ValidationDirective {
  // Resolves from the host component or its ancestors,
  // but not from the content-projecting parent
  private form = inject(FormGroupDirective, { host: true, optional: true });
}
```

The `host` option is less commonly used than `self` and `skipSelf`. It primarily matters for directives that need to distinguish between the host component's injector and the injector of a component that projects content.

### Combining Options

Options can be combined:

```typescript
// Skip the local injector, and don't throw if missing
private parentTheme = inject(ThemeService, { skipSelf: true, optional: true });

// Only check the local injector, don't throw if missing
private localCache = inject(CacheService, { self: true, optional: true });
```

Invalid combinations (e.g., `self: true` with `skipSelf: true`) are contradictory and produce unexpected results. Do not combine them.

## `inject()` vs Constructor Injection

### Constructor Injection (Legacy)

```typescript
// Legacy pattern with constructor parameters
@Component({ ... })
export class UserListComponent {
  constructor(
    private userService: UserService,
    private router: Router,
    @Optional() private analytics: AnalyticsService | null,
    @SkipSelf() private parentConfig: ConfigService,
  ) {}
}
```

### `inject()` Function (Modern)

```typescript
// Modern pattern with inject()
@Component({ ... })
export class UserListComponent {
  private userService = inject(UserService);
  private router = inject(Router);
  private analytics = inject(AnalyticsService, { optional: true });
  private parentConfig = inject(ConfigService, { skipSelf: true });
}
```

### Comparison

| Aspect | `inject()` | Constructor injection |
|---|---|---|
| Location | Field initializers, constructor body, factories | Constructor parameters only |
| Options syntax | `{ optional: true, skipSelf: true }` | `@Optional() @SkipSelf()` parameter decorators |
| TypeScript metadata | Not required | Requires `emitDecoratorMetadata` |
| Composable utilities | Yes (see custom inject functions) | No |
| Functional guards/resolvers | Yes | No (requires a class) |
| Minification | No issues | May need metadata for resolution |
| Parameter decorator deprecation | N/A | `@Inject()`, `@Optional()`, `@Self()`, `@SkipSelf()`, `@Host()` are on the deprecation path |

The Angular team has signaled that parameter decorators (`@Optional()`, `@Self()`, etc.) will be deprecated in favor of `inject()` options. New code should use `inject()` exclusively.

## Creating Custom Inject Functions

One of the most powerful features of `inject()` is composability. You can create reusable utility functions that encapsulate common injection patterns.

### Basic Custom Inject Function

```typescript
// inject-native-element.ts
import { assertInInjectionContext, ElementRef, inject } from '@angular/core';

export function injectNativeElement<T extends Element>(): T {
  assertInInjectionContext(injectNativeElement);
  return inject(ElementRef<T>).nativeElement;
}
```

Usage:

```typescript
@Component({ ... })
export class CardComponent {
  private el = injectNativeElement<HTMLDivElement>();
}

@Directive({ ... })
export class HighlightDirective {
  private el = injectNativeElement<HTMLElement>();
}
```

### `assertInInjectionContext`

Always call `assertInInjectionContext(yourFunction)` at the start of custom inject functions. It throws a clear error pointing to your function name if called outside an injection context, rather than the generic `NG0203` error.

```typescript
export function injectRouteParam(name: string): Signal<string | null> {
  assertInInjectionContext(injectRouteParam);

  const route = inject(ActivatedRoute);
  return toSignal(
    route.paramMap.pipe(map(params => params.get(name))),
    { initialValue: null }
  );
}

// Usage in a component
@Component({ ... })
export class ProductComponent {
  productId = injectRouteParam('id');
}
```

### Custom Inject Function with Cleanup

```typescript
import { assertInInjectionContext, DestroyRef, inject } from '@angular/core';

export function injectInterval(intervalMs: number): Signal<number> {
  assertInInjectionContext(injectInterval);

  const tick = signal(0);
  const destroyRef = inject(DestroyRef);

  const id = setInterval(() => {
    tick.update(n => n + 1);
  }, intervalMs);

  destroyRef.onDestroy(() => clearInterval(id));

  return tick.asReadonly();
}

// Usage
@Component({
  template: `<p>Elapsed: {{ seconds() }}s</p>`,
})
export class TimerComponent {
  seconds = injectInterval(1000);
}
```

The `DestroyRef` injection ensures the interval is cleaned up when the component is destroyed. This pattern encapsulates setup and teardown in a single function call.

### Custom Inject Function for Media Queries

```typescript
export function injectMediaQuery(query: string): Signal<boolean> {
  assertInInjectionContext(injectMediaQuery);

  const destroyRef = inject(DestroyRef);
  const mql = window.matchMedia(query);
  const matches = signal(mql.matches);

  const handler = (e: MediaQueryListEvent) => matches.set(e.matches);
  mql.addEventListener('change', handler);
  destroyRef.onDestroy(() => mql.removeEventListener('change', handler));

  return matches.asReadonly();
}

// Usage
@Component({ ... })
export class LayoutComponent {
  isMobile = injectMediaQuery('(max-width: 768px)');
  prefersDark = injectMediaQuery('(prefers-color-scheme: dark)');
}
```

### Custom Inject Function for localStorage

```typescript
export function injectLocalStorage<T>(key: string, defaultValue: T): WritableSignal<T> {
  assertInInjectionContext(injectLocalStorage);

  const destroyRef = inject(DestroyRef);

  const stored = localStorage.getItem(key);
  const initial = stored ? (JSON.parse(stored) as T) : defaultValue;
  const state = signal<T>(initial);

  // Persist changes to localStorage
  const effectRef = effect(() => {
    localStorage.setItem(key, JSON.stringify(state()));
  });

  return state;
}

// Usage
@Component({ ... })
export class SettingsComponent {
  theme = injectLocalStorage('theme', 'light');
  fontSize = injectLocalStorage('fontSize', 16);
}
```

## `runInInjectionContext`

When you need to call `inject()` outside of a constructor or field initializer, use `runInInjectionContext`. It establishes a temporary injection context using a provided injector.

```typescript
import { EnvironmentInjector, inject, runInInjectionContext } from '@angular/core';

@Injectable({ providedIn: 'root' })
export class PluginManager {
  private envInjector = inject(EnvironmentInjector);
  private plugins: Plugin[] = [];

  loadPlugin(pluginFactory: () => Plugin) {
    // pluginFactory may use inject() internally
    const plugin = runInInjectionContext(this.envInjector, pluginFactory);
    this.plugins.push(plugin);
  }
}
```

### When to Use `runInInjectionContext`

- **Dynamic plugin/extension loading** where factory functions need DI access.
- **Deferred initialization** where setup happens after construction.
- **Library code** that accepts user-provided functions that might call `inject()`.

```typescript
@Component({ ... })
export class DashboardComponent {
  private envInjector = inject(EnvironmentInjector);

  // This method runs after construction, so inject() would normally fail
  async loadWidget(widgetConfig: WidgetConfig) {
    const widget = await import(widgetConfig.modulePath);

    // Run the widget's factory in an injection context
    const instance = runInInjectionContext(this.envInjector, () => {
      return widget.createWidget(); // this factory can use inject()
    });

    this.widgets.push(instance);
  }
}
```

### Do Not Overuse `runInInjectionContext`

If you find yourself using `runInInjectionContext` frequently in application code, reconsider your architecture. Most dependencies should be injected in field initializers. `runInInjectionContext` is primarily for library authors and advanced patterns.

## `inject()` in Functional Guards and Resolvers

Functional guards and resolvers are functions that Angular calls within an injection context. This is why `inject()` works inside them without any extra setup.

### Functional Guard

```typescript
export const authGuard: CanActivateFn = (route, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (!auth.isLoggedIn()) {
    return router.createUrlTree(['/login'], {
      queryParams: { returnUrl: state.url },
    });
  }

  return true;
};

// Route configuration
{
  path: 'dashboard',
  canActivate: [authGuard],
  component: DashboardComponent,
}
```

### Functional Resolver

```typescript
export const productResolver: ResolveFn<Product> = (route) => {
  const productService = inject(ProductService);
  const id = route.paramMap.get('id')!;
  return productService.getById(id);
};

{
  path: 'product/:id',
  resolve: { product: productResolver },
  component: ProductDetailComponent,
}
```

### Composing Guards with `inject()`

```typescript
function requireRole(role: string): CanActivateFn {
  return () => {
    const auth = inject(AuthService);
    return auth.hasRole(role);
  };
}

// Usage
{
  path: 'admin',
  canActivate: [authGuard, requireRole('admin')],
  component: AdminComponent,
}
```

The factory function `requireRole` returns a guard function. When Angular calls that guard, it runs in an injection context, so `inject()` works.

## `inject()` with `DestroyRef`

`DestroyRef` is injected to register cleanup callbacks. It replaces the `ngOnDestroy` lifecycle hook for many use cases.

```typescript
@Component({ ... })
export class LiveDataComponent {
  private destroyRef = inject(DestroyRef);
  private ws: WebSocket | null = null;

  messages = signal<string[]>([]);

  constructor() {
    this.ws = new WebSocket('wss://api.example.com/live');

    this.ws.onmessage = (event) => {
      this.messages.update(msgs => [...msgs, event.data]);
    };

    this.destroyRef.onDestroy(() => {
      this.ws?.close();
      this.ws = null;
    });
  }
}
```

### `takeUntilDestroyed()` with Observables

```typescript
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';

@Component({ ... })
export class SearchComponent {
  private searchInput = new FormControl('');

  constructor() {
    this.searchInput.valueChanges.pipe(
      debounceTime(300),
      distinctUntilChanged(),
      takeUntilDestroyed(), // automatically unsubscribes on destroy
    ).subscribe(term => {
      this.performSearch(term);
    });
  }
}
```

`takeUntilDestroyed()` internally injects `DestroyRef`. It must be called within an injection context (constructor or field initializer). If called elsewhere, pass a `DestroyRef` instance explicitly:

```typescript
private destroyRef = inject(DestroyRef);

someMethod() {
  someObservable$.pipe(
    takeUntilDestroyed(this.destroyRef), // explicit DestroyRef
  ).subscribe();
}
```

## The `NG0203` Error

The most common error with `inject()`:

```
NG0203: inject() must be called from an injection context such as a constructor,
a factory function, a field initializer, or a function used with runInInjectionContext.
```

### Common Causes

**1. Calling `inject()` in a lifecycle hook:**

```typescript
// Bad Practice
ngOnInit() {
  const service = inject(SomeService); // NG0203
}
```

Fix: move to a field initializer.

**2. Calling `inject()` in an event handler or method:**

```typescript
// Bad Practice
onClick() {
  const router = inject(Router); // NG0203
}
```

Fix: inject in a field initializer and use the stored reference.

**3. Calling `inject()` after an `await`:**

```typescript
// Bad Practice
constructor() {
  this.init();
}

async init() {
  await somePromise;
  const service = inject(SomeService); // NG0203: context lost after await
}
```

Fix: inject before any `await`:

```typescript
constructor() {
  const service = inject(SomeService); // OK: synchronous, before await
  this.init(service);
}

async init(service: SomeService) {
  await somePromise;
  service.doSomething(); // use the already-injected reference
}
```

**4. Calling `inject()` in a plain function not invoked by Angular:**

```typescript
// Bad Practice
function createLogger() {
  return inject(LoggerService); // NG0203 unless called in injection context
}

// Fix: either call from field initializer or use runInInjectionContext
```

---

## Common Mistakes

- **Calling `inject()` in lifecycle hooks (`ngOnInit`, `ngAfterViewInit`, etc.).** Lifecycle hooks run after construction, outside the injection context. Always inject in field initializers and store references.

- **Calling `inject()` inside `setTimeout`, `Promise.then`, or after `await`.** The injection context is synchronous. Any asynchronous boundary breaks it. Inject everything synchronously during construction.

- **Using `inject()` in a plain utility function without `assertInInjectionContext`.** If you create a helper function that calls `inject()`, consumers get a confusing `NG0203` error unless you add `assertInInjectionContext(yourFunction)` for a clear error message.

- **Combining `self: true` and `skipSelf: true`.** These are contradictory. `self` says "only look at my injector" while `skipSelf` says "skip my injector." Together they resolve to unpredictable behavior. Use one or the other.

- **Forgetting `optional: true` for services that may not be provided.** Without `optional`, a missing provider throws `NullInjectorError` at runtime. If a dependency is conditionally provided, always use `{ optional: true }` and handle the `null` case.

- **Overusing `runInInjectionContext` in application code.** If you need it frequently, your architecture likely has issues. Most injection should happen during construction. Reserve `runInInjectionContext` for library code and plugin systems.

---

## Best Practices

- **Inject everything in field initializers.** This is the most concise and readable pattern. It keeps all dependencies visible at the top of the class.

```typescript
@Component({ ... })
export class OrderComponent {
  private orderService = inject(OrderService);
  private router = inject(Router);
  private notifier = inject(NotificationService);
}
```

- **Create custom inject functions for reusable patterns.** If multiple components need the same injection + setup logic (route params, media queries, intervals), extract it into a function that calls `assertInInjectionContext` and encapsulates the pattern.

- **Use `DestroyRef` over `ngOnDestroy`.** `DestroyRef` can be injected in services and custom inject functions, not just components. It is more composable than implementing the `OnDestroy` interface.

- **Use `takeUntilDestroyed()` for observable subscriptions.** This eliminates manual `ngOnDestroy` + `Subject` + `takeUntil` boilerplate.

- **Mark inject utilities with `assertInInjectionContext`.** Every custom inject function should start with this assertion. It turns a confusing `NG0203` into a descriptive error naming your function.

- **Prefer `{ optional: true }` over try/catch for optional dependencies.** Angular's `optional` flag is purpose-built for this. It returns `null` cleanly and communicates intent.

## Real-World Use Cases

### Custom `injectQueryParams` Utility

```typescript
export function injectQueryParams(): Signal<Params> {
  assertInInjectionContext(injectQueryParams);
  const route = inject(ActivatedRoute);
  return toSignal(route.queryParams, { initialValue: {} });
}

export function injectQueryParam(name: string): Signal<string | null> {
  assertInInjectionContext(injectQueryParam);
  const route = inject(ActivatedRoute);
  return toSignal(
    route.queryParamMap.pipe(map(params => params.get(name))),
    { initialValue: null },
  );
}

// Usage
@Component({
  template: `
    @let page = currentPage();
    <p>Page: {{ page ?? 1 }}</p>
  `,
})
export class SearchResultsComponent {
  query = injectQueryParam('q');
  currentPage = injectQueryParam('page');
}
```

### `injectDestroy` for Custom Observable Cleanup

```typescript
export function injectDestroy(): Observable<void> {
  assertInInjectionContext(injectDestroy);
  const destroyRef = inject(DestroyRef);
  const subject = new Subject<void>();

  destroyRef.onDestroy(() => {
    subject.next();
    subject.complete();
  });

  return subject.asObservable();
}

// Usage with RxJS
@Component({ ... })
export class ChatComponent {
  private destroy$ = injectDestroy();

  constructor() {
    this.webSocketService.messages$.pipe(
      takeUntil(this.destroy$),
    ).subscribe(msg => this.messages.update(list => [...list, msg]));
  }
}
```

### Feature Flag Service with Conditional Injection

```typescript
@Injectable({ providedIn: 'root' })
export class FeatureFlagService {
  private flags = signal<Record<string, boolean>>({});

  isEnabled(flag: string): boolean {
    return this.flags()[flag] ?? false;
  }
}

export function injectIfFeatureEnabled<T>(
  token: ProviderToken<T>,
  featureFlag: string,
): T | null {
  assertInInjectionContext(injectIfFeatureEnabled);
  const featureService = inject(FeatureFlagService);

  if (!featureService.isEnabled(featureFlag)) {
    return null;
  }

  return inject(token, { optional: true });
}

// Usage
@Component({ ... })
export class DashboardComponent {
  private analytics = injectIfFeatureEnabled(AnalyticsService, 'analytics');
  private experimentalChat = injectIfFeatureEnabled(ChatService, 'live-chat');

  trackEvent(name: string) {
    this.analytics?.track(name);
  }
}
```

### Composing Multiple Injections in a Factory

```typescript
interface AppContext {
  user: Signal<User | null>;
  theme: Signal<string>;
  locale: Signal<string>;
  permissions: Signal<string[]>;
}

export function injectAppContext(): AppContext {
  assertInInjectionContext(injectAppContext);

  const authService = inject(AuthService);
  const themeService = inject(ThemeService);
  const i18nService = inject(I18nService);
  const permService = inject(PermissionService);

  return {
    user: authService.currentUser,
    theme: themeService.currentTheme,
    locale: i18nService.locale,
    permissions: permService.userPermissions,
  };
}

// Usage: single call instead of four inject() calls
@Component({ ... })
export class NavbarComponent {
  ctx = injectAppContext();
  // ctx.user(), ctx.theme(), ctx.locale(), ctx.permissions()
}
```

---

> **Key Takeaway:** Use `inject()` in field initializers for all dependency injection, leverage `InjectOptions` (`optional`, `self`, `skipSelf`, `host`) to control resolution behavior, and extract reusable injection patterns into custom inject functions with `assertInInjectionContext` for clear error reporting.
