# 15.1 Router Setup

> The Angular Router maps URLs to components. Setting it up correctly with `provideRouter` and its feature functions determines how navigation, scrolling, preloading, and transitions behave across your entire application.

---

## The Standalone Router Setup

Modern Angular applications use `provideRouter` instead of `RouterModule.forRoot()`. The setup lives in the application configuration, typically in `app.config.ts`.

### Minimal Setup

```typescript
// app.config.ts
import { ApplicationConfig } from '@angular/core';
import { provideRouter } from '@angular/router';
import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
  ],
};
```

```typescript
// app.routes.ts
import { Routes } from '@angular/router';

export const routes: Routes = [
  { path: '', component: HomeComponent },
  { path: 'about', component: AboutComponent },
  { path: '**', component: NotFoundComponent },
];
```

```typescript
// main.ts
import { bootstrapApplication } from '@angular/platform-browser';
import { AppComponent } from './app.component';
import { appConfig } from './app.config';

bootstrapApplication(AppComponent, appConfig);
```

```typescript
// app.component.ts
import { Component } from '@angular/core';
import { RouterOutlet } from '@angular/router';

@Component({
  selector: 'app-root',
  imports: [RouterOutlet],
  template: `
    <header>
      <h1>My App</h1>
    </header>
    <main>
      <router-outlet />
    </main>
  `,
})
export class AppComponent {}
```

### The `provideRouter` Signature

```typescript
function provideRouter(
  routes: Routes,
  ...features: RouterFeatures[]
): EnvironmentProviders;
```

The first argument is the root route array. Additional arguments are router feature functions that configure specific behaviors.

---

## Router Feature Functions

Angular provides a set of `withXxx` functions that plug into `provideRouter`. Each enables or configures a specific router capability.

### Complete Feature List

| Feature Function | Purpose |
|-----------------|---------|
| `withComponentInputBinding()` | Binds route params, query params, data, and resolver results to component inputs |
| `withViewTransitions()` | Enables CSS View Transitions API for smooth navigation animations |
| `withPreloading(strategy)` | Preloads lazy routes in the background |
| `withInMemoryScrolling(options)` | Manages scroll position on navigation |
| `withHashLocation()` | Uses hash-based URLs (`/#/path`) instead of HTML5 `pushState` |
| `withRouterConfig(options)` | Fine-tunes router behavior (params inheritance, URL update strategy, etc.) |
| `withNavigationErrorHandler(fn)` | Custom handler for navigation errors |
| `withDebugTracing()` | Logs all router events to the console |
| `withDisabledInitialNavigation()` | Prevents the router from navigating on startup |
| `withEnabledBlockingInitialNavigation()` | Blocks rendering until initial navigation completes (useful for SSR) |

### Production-Ready Configuration

A typical production setup uses several features together:

```typescript
// app.config.ts
import { ApplicationConfig } from '@angular/core';
import {
  provideRouter,
  withComponentInputBinding,
  withViewTransitions,
  withInMemoryScrolling,
  withPreloading,
  withRouterConfig,
  withNavigationErrorHandler,
  PreloadAllModules,
} from '@angular/router';
import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(
      routes,
      withComponentInputBinding(),
      withViewTransitions({ skipInitialTransition: true }),
      withInMemoryScrolling({
        scrollPositionRestoration: 'top',
        anchorScrolling: 'enabled',
      }),
      withPreloading(PreloadAllModules),
      withRouterConfig({
        paramsInheritanceStrategy: 'always',
      }),
      withNavigationErrorHandler((error) => {
        console.error('Navigation failed:', error);
        // Could inject a service here to show a user-friendly error
      }),
    ),
  ],
};
```

## `withComponentInputBinding()`

This feature (Angular 16+) is one of the most impactful router features. It automatically binds route parameters, query parameters, static data, and resolver results to matching component inputs.

### Before (Manual)

```typescript
// Without withComponentInputBinding -- must use ActivatedRoute
@Component({ ... })
export class ProductComponent implements OnInit {
  private route = inject(ActivatedRoute);
  productId = '';

  ngOnInit() {
    this.route.paramMap.subscribe(params => {
      this.productId = params.get('id') ?? '';
    });
  }
}
```

### After (Automatic)

```typescript
// With withComponentInputBinding -- route params map to inputs automatically
@Component({
  selector: 'app-product',
  template: `<h1>Product {{ id() }}</h1>`,
})
export class ProductComponent {
  // The 'id' input is automatically bound to the ':id' route parameter
  id = input.required<string>();
}
```

### What Gets Bound

Given this route:

```typescript
{
  path: 'products/:id',
  component: ProductComponent,
  data: { breadcrumb: 'Product Detail' },
  resolve: { product: productResolver },
}
```

And a URL like `/products/42?tab=reviews`:

| Source | Input Name | Value |
|--------|-----------|-------|
| Path parameter `:id` | `id` | `'42'` |
| Query parameter `?tab=` | `tab` | `'reviews'` |
| Static route data | `breadcrumb` | `'Product Detail'` |
| Resolver result | `product` | Result of `productResolver` |

```typescript
@Component({ ... })
export class ProductComponent {
  id = input.required<string>();          // From path param
  tab = input<string>();                  // From query param
  breadcrumb = input<string>();           // From static data
  product = input<Product>();             // From resolver
}
```

### Priority Order

When multiple sources have the same name, the binding priority is:
1. Static route `data`
2. Path parameters
3. Query parameters

Resolver results are also bound under their key name from the `resolve` map.

## `withViewTransitions()`

Enables the CSS View Transitions API for smooth animated transitions between routed views.

```typescript
provideRouter(
  routes,
  withViewTransitions({
    skipInitialTransition: true, // Do not animate the first page load
    onViewTransitionCreated: (info) => {
      // Access the ViewTransition object for custom control
      console.log('Transition from:', info.from.url);
      console.log('Transition to:', info.to.url);
    },
  }),
);
```

### Custom CSS for Transitions

```css
/* Default cross-fade for all navigations */
::view-transition-old(root) {
  animation: fade-out 0.2s ease-in;
}

::view-transition-new(root) {
  animation: fade-in 0.3s ease-out;
}

@keyframes fade-out {
  from { opacity: 1; }
  to { opacity: 0; }
}

@keyframes fade-in {
  from { opacity: 0; }
  to { opacity: 1; }
}
```

> **Note:** View Transitions require browser support. As of 2025, Chrome, Edge, and Safari support this API. Firefox support is still in development. The router gracefully falls back to instant navigation in unsupported browsers.

## `withInMemoryScrolling()`

Controls scroll behavior during navigation.

```typescript
provideRouter(
  routes,
  withInMemoryScrolling({
    // Scroll to top on every forward navigation
    scrollPositionRestoration: 'top',    // 'disabled' | 'top' | 'enabled'

    // Enable scrolling to anchor elements via fragment
    anchorScrolling: 'enabled',          // 'disabled' | 'enabled'
  }),
);
```

| Option | `'disabled'` | `'top'` | `'enabled'` |
|--------|-------------|---------|-------------|
| `scrollPositionRestoration` | No scroll management | Always scroll to top | Scroll to top on forward nav, restore position on back/forward |

### Anchor Scrolling

With `anchorScrolling: 'enabled'`, fragment URLs like `/page#section2` will scroll to the element with `id="section2"`:

```typescript
// In a template
<a [routerLink]="['/docs']" fragment="installation">Go to Installation</a>

// Navigates to /docs#installation and scrolls to <h2 id="installation">
```

## `withPreloading()`

Configures how lazy-loaded routes are preloaded in the background after the initial page load.

### Built-in Strategies

```typescript
import { PreloadAllModules, NoPreloading } from '@angular/router';

// Preload all lazy routes immediately after initial load
provideRouter(routes, withPreloading(PreloadAllModules));

// No preloading (default) -- lazy routes load on demand only
provideRouter(routes, withPreloading(NoPreloading));
```

### Custom Preloading Strategy

You can build a strategy that preloads only routes marked with a `preload: true` data property:

```typescript
import { PreloadingStrategy, Route } from '@angular/router';
import { Observable, of, EMPTY } from 'rxjs';
import { Injectable } from '@angular/core';

@Injectable({ providedIn: 'root' })
export class SelectivePreloadingStrategy implements PreloadingStrategy {
  preload(route: Route, load: () => Observable<unknown>): Observable<unknown> {
    if (route.data?.['preload']) {
      return load();
    }
    return EMPTY;
  }
}

// Usage
provideRouter(routes, withPreloading(SelectivePreloadingStrategy));

// Route configuration
const routes: Routes = [
  {
    path: 'dashboard',
    loadComponent: () => import('./dashboard/dashboard.component'),
    data: { preload: true }, // This route will be preloaded
  },
  {
    path: 'admin',
    loadComponent: () => import('./admin/admin.component'),
    // No preload flag -- loads only when navigated to
  },
];
```

## `withHashLocation()`

Uses hash-based URLs instead of the default HTML5 `pushState` strategy.

```typescript
provideRouter(routes, withHashLocation());
```

| Strategy | URL Format | Server Config Required | Use Case |
|----------|-----------|----------------------|----------|
| `PathLocationStrategy` (default) | `/products/42` | Yes (URL rewriting) | Most SPAs, SSR |
| `HashLocationStrategy` | `/#/products/42` | No | Static file hosting, legacy environments |

### When to Use Hash Location

- Hosting on a static file server that cannot rewrite URLs (e.g., plain S3, GitHub Pages without 404.html trick).
- Legacy server environments where configuring URL rewriting is not possible.
- Desktop applications using Electron where URL rewriting is irrelevant.

For most deployments, the default `PathLocationStrategy` is preferred because it produces clean URLs and is required for server-side rendering.

## `withRouterConfig()`

Fine-tunes the router's internal behavior.

```typescript
provideRouter(
  routes,
  withRouterConfig({
    // How child routes inherit parent route parameters
    paramsInheritanceStrategy: 'always',  // 'emptyOnly' (default) | 'always'

    // When the URL in the browser address bar updates
    urlUpdateStrategy: 'deferred',  // 'deferred' (default) | 'eager'

    // Cancel navigation when a new navigation starts
    canceledNavigationResolution: 'computed',  // 'replace' | 'computed'

    // What to do with the URL when navigation is cancelled
    onSameUrlNavigation: 'reload',  // 'ignore' (default) | 'reload'
  }),
);
```

### `paramsInheritanceStrategy`

This is one of the most commonly needed options:

```typescript
// Route tree:
// /users/:userId/orders/:orderId

// With 'emptyOnly' (default):
// OrderComponent can only access :orderId from its own route
// It must use parent.paramMap to get :userId

// With 'always':
// OrderComponent automatically has access to both :userId and :orderId
```

**Recommendation:** Set `paramsInheritanceStrategy: 'always'` in most applications. It simplifies deeply nested routes significantly.

### `onSameUrlNavigation`

Controls what happens when navigating to the current URL:

```typescript
// 'ignore' (default): Navigation is completely ignored
// 'reload': Router events fire again, guards re-run, resolvers re-execute

// Useful for "refresh" buttons or search pages where the same URL
// should re-trigger data loading
```

## `withNavigationErrorHandler()`

Provides a centralized handler for navigation failures:

```typescript
import { RedirectCommand, Router } from '@angular/router';

provideRouter(
  routes,
  withNavigationErrorHandler((error) => {
    const router = inject(Router);

    if (error instanceof Error && error.message.includes('chunk')) {
      // Chunk loading failure -- reload the page
      window.location.reload();
      return;
    }

    // Navigate to an error page for other failures
    return new RedirectCommand(router.parseUrl('/error'));
  }),
);
```

This is especially useful for handling lazy-load chunk failures in production when a new deployment invalidates cached chunk URLs.

## `withDebugTracing()`

Logs all router events to the browser console. Use during development only:

```typescript
import { isDevMode } from '@angular/core';

const features = [
  withComponentInputBinding(),
  withInMemoryScrolling({ scrollPositionRestoration: 'top' }),
];

if (isDevMode()) {
  features.push(withDebugTracing());
}

provideRouter(routes, ...features);
```

### Router Events Logged

When enabled, you see the full navigation lifecycle:

```
NavigationStart    { id: 1, url: '/products' }
RoutesRecognized   { id: 1, url: '/products', urlAfterRedirects: '/products' }
GuardsCheckStart   { id: 1, url: '/products' }
GuardsCheckEnd     { id: 1, url: '/products', shouldActivate: true }
ResolveStart       { id: 1, url: '/products' }
ResolveEnd         { id: 1, url: '/products' }
ActivationStart    { ... }
ActivationEnd      { ... }
NavigationEnd      { id: 1, url: '/products', urlAfterRedirects: '/products' }
```

## Router Directives in Standalone Components

In standalone components, router directives must be explicitly imported.

### `RouterOutlet`

The placeholder where routed components are rendered:

```typescript
import { Component } from '@angular/core';
import { RouterOutlet } from '@angular/router';

@Component({
  selector: 'app-root',
  imports: [RouterOutlet],
  template: `
    <app-navbar />
    <router-outlet />
    <app-footer />
  `,
})
export class AppComponent {}
```

### Named Outlets

For auxiliary routes (secondary navigation areas):

```typescript
@Component({
  imports: [RouterOutlet],
  template: `
    <router-outlet />                 <!-- Primary outlet -->
    <router-outlet name="sidebar" />  <!-- Named outlet -->
  `,
})
export class LayoutComponent {}
```

```typescript
const routes: Routes = [
  { path: 'dashboard', component: DashboardComponent },
  { path: 'help', component: HelpPanelComponent, outlet: 'sidebar' },
];

// URL: /dashboard(sidebar:help)
```

### `RouterLink`

Binds clickable elements to navigation targets:

```typescript
import { Component } from '@angular/core';
import { RouterLink } from '@angular/router';

@Component({
  imports: [RouterLink],
  template: `
    <!-- String path -->
    <a routerLink="/home">Home</a>

    <!-- With path parameters -->
    <a [routerLink]="['/products', product.id]">{{ product.name }}</a>

    <!-- With query parameters and fragment -->
    <a
      [routerLink]="['/search']"
      [queryParams]="{ q: 'angular', page: 1 }"
      fragment="results"
    >
      Search
    </a>

    <!-- Relative navigation -->
    <a routerLink="../sibling">Go to Sibling</a>

    <!-- Preserve or merge query params -->
    <a
      routerLink="/products"
      queryParamsHandling="merge"
    >
      Products (keep current query params)
    </a>
  `,
})
export class NavComponent {
  product = { id: 42, name: 'Widget' };
}
```

### `queryParamsHandling`

| Value | Behavior |
|-------|----------|
| `''` (default) | Drop all existing query params |
| `'merge'` | Merge new params with existing ones |
| `'preserve'` | Keep existing params, ignore new ones |

### `RouterLinkActive`

Applies CSS classes when the linked route is active:

```typescript
import { Component } from '@angular/core';
import { RouterLink, RouterLinkActive } from '@angular/router';

@Component({
  imports: [RouterLink, RouterLinkActive],
  template: `
    <nav>
      <a
        routerLink="/home"
        routerLinkActive="active"
        [routerLinkActiveOptions]="{ exact: true }"
      >
        Home
      </a>
      <a routerLink="/products" routerLinkActive="active">Products</a>
      <a routerLink="/about" routerLinkActive="active">About</a>
    </nav>
  `,
  styles: `
    .active {
      font-weight: bold;
      border-bottom: 2px solid currentColor;
    }
  `,
})
export class NavbarComponent {}
```

### `routerLinkActiveOptions`

The `exact` option controls matching precision:

```typescript
// With { exact: true }:
// /products       -> active
// /products/42    -> NOT active

// With { exact: false } (default):
// /products       -> active
// /products/42    -> active (because /products is a prefix)
```

> **Important:** Always use `{ exact: true }` for root links (like `/` or `/home`). Otherwise, the root link will show as active on every page since `/` is a prefix of all paths.

### `RouterLinkActive` with `isActiveChange`

You can react programmatically to active state changes:

```typescript
<a
  routerLink="/dashboard"
  routerLinkActive="active"
  (isActiveChange)="onActiveChange($event)"
>
  Dashboard
</a>
```

## The `<base href>` Requirement

When using the default `PathLocationStrategy`, your `index.html` must include a `<base href>` tag:

```html
<!doctype html>
<html>
<head>
  <base href="/" />
  <!-- ... -->
</head>
<body>
  <app-root></app-root>
</body>
</html>
```

If your app is deployed to a subdirectory (e.g., `https://example.com/my-app/`), set the base href accordingly:

```html
<base href="/my-app/" />
```

Or set it at build time:

```bash
ng build --base-href /my-app/
```

Without the correct `<base href>`, the router cannot compute relative URLs, and navigation will break.

## Programmatic Navigation

The `Router` service provides methods for navigation from TypeScript code:

```typescript
import { Component, inject } from '@angular/core';
import { Router } from '@angular/router';

@Component({ ... })
export class LoginComponent {
  private router = inject(Router);

  async onLogin(credentials: Credentials) {
    const success = await this.authService.login(credentials);

    if (success) {
      // Absolute navigation
      this.router.navigate(['/dashboard']);

      // With query parameters
      this.router.navigate(['/search'], {
        queryParams: { q: 'angular' },
      });

      // Navigate by URL string
      this.router.navigateByUrl('/dashboard?welcome=true');
    }
  }
}
```

### `navigate` vs `navigateByUrl`

| Method | Input | Relative Navigation | Query Params |
|--------|-------|--------------------|--------------|
| `navigate` | Array of path segments | Supported | Via options object |
| `navigateByUrl` | Full URL string | Not supported | Embedded in the string |

```typescript
// These are equivalent:
this.router.navigate(['/products', 42], { queryParams: { tab: 'reviews' } });
this.router.navigateByUrl('/products/42?tab=reviews');
```

## Common Mistakes

### Mistake 1: Forgetting to Import Router Directives

```typescript
// BAD: RouterOutlet is not imported
@Component({
  template: `<router-outlet />`,  // Renders nothing, no error in some cases
})
export class AppComponent {}

// GOOD: Explicitly import what you use
@Component({
  imports: [RouterOutlet],
  template: `<router-outlet />`,
})
export class AppComponent {}
```

### Mistake 2: Missing the Wildcard Route

```typescript
// BAD: No catch-all route -- unmatched URLs show a blank page
const routes: Routes = [
  { path: '', component: HomeComponent },
  { path: 'about', component: AboutComponent },
];

// GOOD: Wildcard route catches everything else
const routes: Routes = [
  { path: '', component: HomeComponent },
  { path: 'about', component: AboutComponent },
  { path: '**', component: NotFoundComponent },
];
```

The wildcard `**` route must always be last because routes are matched in order.

### Mistake 3: Not Using `exact: true` on Root Links

```typescript
// BAD: Home link is always active because '/' matches every URL
<a routerLink="/" routerLinkActive="active">Home</a>

// GOOD: Exact match for root path
<a routerLink="/" routerLinkActive="active"
   [routerLinkActiveOptions]="{ exact: true }">
  Home
</a>
```

### Mistake 4: Deploying Without URL Rewrite Configuration

With `PathLocationStrategy`, navigating directly to `/products/42` (via browser refresh or direct link) sends a request to the server for `/products/42`. Without server-side URL rewriting, this returns a 404.

Fix: Configure your server to return `index.html` for all routes:

```nginx
# Nginx
location / {
  try_files $uri $uri/ /index.html;
}
```

```apache
# Apache .htaccess
RewriteEngine On
RewriteCond %{DOCUMENT_ROOT}%{REQUEST_URI} -f [OR]
RewriteCond %{DOCUMENT_ROOT}%{REQUEST_URI} -d
RewriteRule ^ - [L]
RewriteRule ^ /index.html [L]
```

### Mistake 5: Including `withDebugTracing()` in Production

```typescript
// BAD: Debug tracing in production floods the console
provideRouter(routes, withDebugTracing());

// GOOD: Only in development
const features: RouterFeatures[] = [withComponentInputBinding()];
if (isDevMode()) {
  features.push(withDebugTracing());
}
provideRouter(routes, ...features);
```

## Best Practices

### 1. Always Enable `withComponentInputBinding()`

This feature eliminates most `ActivatedRoute` subscription boilerplate. There is no downside to enabling it, and it makes routed components cleaner and more testable.

### 2. Set `paramsInheritanceStrategy: 'always'`

Unless you have a specific reason not to, `'always'` saves you from manually accessing parent route params in deeply nested routes.

### 3. Configure Scroll Restoration

Users expect scroll-to-top on forward navigation and restored scroll position on back navigation. Use `withInMemoryScrolling({ scrollPositionRestoration: 'enabled' })`.

### 4. Separate Routes into Their Own File

Keep routes in `app.routes.ts` (and feature routes in their own files). This keeps `app.config.ts` clean and makes the route tree easy to find.

### 5. Handle Navigation Errors

Use `withNavigationErrorHandler()` to catch chunk-load failures and other navigation errors gracefully rather than leaving users on a broken page.

### 6. Use `withPreloading()` Strategically

`PreloadAllModules` is appropriate for small-to-medium apps. For large apps, use a custom strategy that preloads based on user behavior or route priority.

## Real-World Use Cases

### Full Production Configuration

```typescript
// app.config.ts
import { ApplicationConfig, isDevMode } from '@angular/core';
import {
  provideRouter,
  withComponentInputBinding,
  withViewTransitions,
  withInMemoryScrolling,
  withPreloading,
  withRouterConfig,
  withNavigationErrorHandler,
  withDebugTracing,
  RouterFeatures,
  Router,
} from '@angular/router';
import { provideHttpClient, withInterceptors } from '@angular/common/http';
import { routes } from './app.routes';
import { SelectivePreloadingStrategy } from './core/selective-preloading';
import { inject } from '@angular/core';

const routerFeatures: RouterFeatures[] = [
  withComponentInputBinding(),
  withViewTransitions({ skipInitialTransition: true }),
  withInMemoryScrolling({
    scrollPositionRestoration: 'enabled',
    anchorScrolling: 'enabled',
  }),
  withPreloading(SelectivePreloadingStrategy),
  withRouterConfig({
    paramsInheritanceStrategy: 'always',
    onSameUrlNavigation: 'reload',
  }),
  withNavigationErrorHandler((error) => {
    if (error?.message?.includes('Loading chunk')) {
      // Deployment happened while user was on the app
      window.location.reload();
    }
  }),
];

if (isDevMode()) {
  routerFeatures.push(withDebugTracing());
}

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes, ...routerFeatures),
    provideHttpClient(withInterceptors([])),
  ],
};
```

```typescript
// app.component.ts
import { Component } from '@angular/core';
import { RouterOutlet, RouterLink, RouterLinkActive } from '@angular/router';

@Component({

```

### Modular Route Organization

```typescript
// app.routes.ts
import { Routes } from '@angular/router';

export const routes: Routes = [
  {
    path: '',
    loadComponent: () => import('./home/home.component').then(m => m.HomeComponent),
  },
  {
    path: 'products',
    loadChildren: () => import('./products/product.routes').then(m => m.PRODUCT_ROUTES),
  },
  {
    path: 'admin',
    loadChildren: () => import('./admin/admin.routes').then(m => m.ADMIN_ROUTES),
  },
  {
    path: '**',
    loadComponent: () => import('./not-found/not-found.component').then(m => m.NotFoundComponent),
  },
];

// products/product.routes.ts
import { Routes } from '@angular/router';

export const PRODUCT_ROUTES: Routes = [
  {
    path: '',
    loadComponent: () => import('./product-list.component').then(m => m.ProductListComponent),
  },
  {
    path: ':id',
    loadComponent: () => import('./product-detail.component').then(m => m.ProductDetailComponent),
  },
];
```

> **Key Takeaway:** Use `provideRouter` with feature functions to configure every aspect of routing behavior. At minimum, enable `withComponentInputBinding()` and `withInMemoryScrolling()` in every application, and always include a wildcard route and a navigation error handler.
