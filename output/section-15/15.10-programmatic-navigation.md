# 15.10 Programmatic Navigation

> Beyond template-driven `RouterLink`, Angular's `Router` service provides `navigate()` and `navigateByUrl()` for imperative navigation from component or service code, with fine-grained control over URL construction, history behavior, and transient state.

---

## navigate() vs navigateByUrl()

Angular provides two methods on the `Router` service for programmatic navigation. They differ in how they accept the target URL.

| Feature | `navigate()` | `navigateByUrl()` |
|---|---|---|
| Input | Array of URL segments (`any[]`) | Full URL string or `UrlTree` |
| Relative navigation | Supported (with `relativeTo`) | Not supported (always absolute) |
| Query params | Via `NavigationExtras` | Parsed from the string or via extras |
| Matrix params | Via segment objects in the array | Parsed from the string |
| Use case | Dynamic route construction, relative navigation | Known absolute URLs, deep links |

### navigate()

Accepts a "commands" array where each element becomes a URL segment:

```typescript
import { Component, inject } from '@angular/core';
import { Router } from '@angular/router';

@Component({ /* ... */ })
export class ProductListComponent {
  private router = inject(Router);

  viewProduct(id: number): void {
    // Navigates to /products/42
    this.router.navigate(['/products', id]);
  }

  goHome(): void {
    // Navigates to /
    this.router.navigate(['/']);
  }
}
```

### navigateByUrl()

Accepts a complete URL string. Always treated as absolute (leading `/` is implied if missing):

```typescript
@Component({ /* ... */ })
export class SearchComponent {
  private router = inject(Router);

  goToProduct(id: number): void {
    // Navigates to /products/42
    this.router.navigateByUrl(`/products/${id}`);
  }

  goToSearchResults(): void {
    // Navigates to /search?q=angular&sort=relevance#results
    this.router.navigateByUrl('/search?q=angular&sort=relevance#results');
  }
}
```

### Return Value

Both methods return `Promise<boolean>`:

```typescript
async saveAndNavigate(): Promise<void> {
  await this.saveData();

  const success = await this.router.navigate(['/dashboard']);

  if (success) {
    console.log('Navigation succeeded');
  } else if (success === false) {
    console.log('Navigation failed (guard rejected)');
  } else if (success === null) {
    console.log('Navigation was not performed (same URL)');
  }
}
```

| Return value | Meaning |
|---|---|
| `true` | Navigation completed successfully |
| `false` | Navigation was rejected (by a guard) |
| `null` | No navigation occurred (same URL, or superseded by another navigation) |

## The Commands Array

The first argument to `navigate()` is a "commands" array. Each element is processed sequentially to build the URL:

```typescript
// Simple path segments
this.router.navigate(['/users', 'profile', 'settings']);
// Result: /users/profile/settings

// With route parameters
this.router.navigate(['/users', userId, 'orders', orderId]);
// Result: /users/42/orders/789

// With matrix parameters (object after a segment)
this.router.navigate(['/products', { category: 'electronics', page: 2 }]);
// Result: /products;category=electronics;page=2

// Mixing segments and matrix params
this.router.navigate(['/users', userId, 'orders', { status: 'pending' }]);
// Result: /users/42/orders;status=pending
```

### Outlets in the Commands Array

Navigate to named outlets by including an `outlets` object:

```typescript
// Navigate the primary outlet to /products and the sidebar outlet to /filters
this.router.navigate([{
  outlets: {
    primary: ['products'],
    sidebar: ['filters']
  }
}]);
// Result: /products(sidebar:filters)

// Close a named outlet by setting it to null
this.router.navigate([{
  outlets: {
    sidebar: null
  }
}]);
```

## Relative Navigation

By default, `navigate()` treats the commands array as absolute when it starts with `/`. To navigate relative to the current route, use the `relativeTo` option with an `ActivatedRoute` reference.

```typescript
import { Component, inject } from '@angular/core';
import { Router, ActivatedRoute } from '@angular/router';

@Component({ /* ... */ })
export class UserDetailComponent {
  private router = inject(Router);
  private route = inject(ActivatedRoute);

  goToEdit(): void {
    // Current URL: /users/42
    // Navigates to: /users/42/edit
    this.router.navigate(['edit'], { relativeTo: this.route });
  }

  goToSibling(): void {
    // Current URL: /users/42
    // Navigates to: /users/43
    this.router.navigate(['../43'], { relativeTo: this.route });
  }

  goToParent(): void {
    // Current URL: /users/42/edit
    // Navigates to: /users/42
    this.router.navigate(['..'], { relativeTo: this.route });
  }

  goToParentSibling(): void {
    // Current URL: /users/42/edit
    // Navigates to: /users/43
    this.router.navigate(['../../43'], { relativeTo: this.route });
  }
}
```

### Bad Practice: Forgetting relativeTo

```typescript
// BAD: Without relativeTo, this navigates to /edit (absolute)
this.router.navigate(['edit']);

// GOOD: With relativeTo, this navigates to /users/42/edit
this.router.navigate(['edit'], { relativeTo: this.route });
```

Without `relativeTo`, the commands array is resolved against the root of the router. The leading segment `'edit'` becomes `/edit`, not a child of the current route.

### Relative Navigation from Services

Services do not have an `ActivatedRoute`. If you need relative navigation from a service, accept the route as a parameter:

```typescript
@Injectable({ providedIn: 'root' })
export class NavigationService {
  private router = inject(Router);

  navigateToChild(childPath: string, relativeTo: ActivatedRoute): void {
    this.router.navigate([childPath], { relativeTo });
  }
}
```

## NavigationExtras: Complete Reference

`NavigationExtras` combines two interfaces: `UrlCreationOptions` (how the URL is built) and `NavigationBehaviorOptions` (how the navigation behaves).

### UrlCreationOptions

These options control the structure of the target URL:

```typescript
interface UrlCreationOptions {
  relativeTo?: ActivatedRoute | null;
  queryParams?: Params | null;
  fragment?: string;
  queryParamsHandling?: QueryParamsHandling | null;
  preserveFragment?: boolean;
}
```

#### queryParams

```typescript
// Navigates to /search?q=angular&page=1
this.router.navigate(['/search'], {
  queryParams: { q: 'angular', page: 1 }
});
```

#### queryParamsHandling

Controls how existing query parameters interact with new ones:

```typescript
// Current URL: /search?q=angular&page=1

// Default (replace): /search?sort=name (q and page are dropped)
this.router.navigate(['/search'], {
  queryParams: { sort: 'name' }
});

// merge: /search?q=angular&page=1&sort=name
this.router.navigate(['/search'], {
  queryParams: { sort: 'name' },
  queryParamsHandling: 'merge'
});

// preserve: /search?q=angular&page=1 (new params ignored)
this.router.navigate(['/search'], {
  queryParams: { sort: 'name' },
  queryParamsHandling: 'preserve'
});
```

#### fragment

```typescript
// Navigates to /docs/setup#installation
this.router.navigate(['/docs/setup'], {
  fragment: 'installation'
});
```

#### preserveFragment

```typescript
// Current URL: /docs/setup#installation
// Navigates to /docs/advanced#installation (fragment preserved)
this.router.navigate(['/docs/advanced'], {
  preserveFragment: true
});
```

### NavigationBehaviorOptions

These options control browser history and navigation behavior:

```typescript
interface NavigationBehaviorOptions {
  skipLocationChange?: boolean;
  replaceUrl?: boolean;
  state?: { [k: string]: any };
  info?: unknown;
  browserUrl?: string | UrlTree;
  onSameUrlNavigation?: OnSameUrlNavigation;
}
```

#### skipLocationChange

Navigates without updating the browser's address bar. The URL stays at its current value, but the route tree updates normally.

```typescript
// URL remains at current value, but /modal/confirm is activated
this.router.navigate(['/modal/confirm'], {
  skipLocationChange: true
});
```

Use cases:
- Modal dialogs that should not create a history entry
- Wizard steps that you control programmatically
- Internal redirects that should be invisible to the user

**Warning:** The back button will not return to this "invisible" navigation. Use sparingly.

#### replaceUrl

Replaces the current browser history entry instead of pushing a new one. The URL updates, but pressing back skips this page.

```typescript
// After login, replace the login page in history so back button
// does not return to the login form
this.router.navigate(['/dashboard'], {
  replaceUrl: true
});
```

Use cases:
- Post-login redirects (user should not "go back" to the login form)
- Redirect chains (intermediate URLs should not clutter history)
- Normalizing URLs (e.g., redirecting `/home` to `/`)

#### skipLocationChange vs replaceUrl

| Option | URL changes? | History entry added? | Back button behavior |
|---|---|---|---|
| Neither (default) | Yes | Yes | Returns to previous URL |
| `skipLocationChange: true` | No | No | Skips this navigation entirely |
| `replaceUrl: true` | Yes | No (replaces current) | Returns to URL before the replaced one |

#### state

Passes arbitrary data through navigation without exposing it in the URL. The data is serialized into the browser's `history.state`.

```typescript
// Passing state during navigation
this.router.navigate(['/order/confirmation'], {
  state: {
    orderId: 'ORD-12345',
    fromCheckout: true
  }
});

// Reading state in the target component
@Component({ /* ... */ })
export class ConfirmationComponent {
  private router = inject(Router);

  // Read from history.state (available immediately)
  orderId = history.state?.orderId;
  fromCheckout = history.state?.fromCheckout;
}
```

**Reading state from the Router:**

```typescript
// During navigation (inside a guard or resolver)
const nav = this.router.getCurrentNavigation();
const state = nav?.extras?.state;

// After navigation completes, use history.state
const state = history.state;
```

**Limitations of `state`:**
- Must be serializable (no functions, no class instances, no circular references)
- Limited in size (browser-dependent, typically 2-16 MB)
- Lost on page reload unless the browser preserves `history.state` (most do)
- Not part of the URL, so it cannot be bookmarked or shared

#### info

Transient metadata that is NOT persisted to `history.state`. It is available only during the navigation lifecycle (in guards, resolvers, and router events) and disappears once navigation completes.

```typescript
// Pass transient info
this.router.navigate(['/products'], {
  info: { source: 'search-bar', timestamp: Date.now() }
});

// Access in a guard
const authGuard: CanActivateFn = (route, state) => {
  const router = inject(Router);
  const nav = router.getCurrentNavigation();
  const info = nav?.extras?.info as { source: string } | undefined;
  console.log('Navigation source:', info?.source);
  return true;
};
```

Use cases:
- Tracking navigation source for analytics without polluting `history.state`
- Passing context to guards/resolvers that should not survive page reload

#### browserUrl

Displays a different URL in the address bar than the route that was actually activated. Added in recent Angular versions.

```typescript
// Internally navigate to /error/not-found
// But show /products/missing-item in the address bar
this.router.navigateByUrl('/error/not-found', {
  browserUrl: '/products/missing-item'
});
```

This is useful for:
- Showing user-friendly URLs while routing to generic error pages
- Keeping the original URL intact when redirecting to fallback routes

#### onSameUrlNavigation

Overrides the global `onSameUrlNavigation` setting for a single navigation:

```typescript
// Force re-running guards and resolvers even though URL is the same
this.router.navigate(['/dashboard'], {
  onSameUrlNavigation: 'reload'
});
```

## Navigation from Guards and Resolvers

### Using RedirectCommand

The preferred way to redirect from a guard or resolver. It wraps a `UrlTree` with optional `NavigationBehaviorOptions`:

```typescript
import { inject } from '@angular/core';
import { CanActivateFn, Router, RedirectCommand } from '@angular/router';

export const authGuard: CanActivateFn = () => {
  const authService = inject(AuthService);
  const router = inject(Router);

  if (!authService.isAuthenticated()) {
    const loginUrl = router.parseUrl('/login');
    // Redirect with replaceUrl so back button skips the rejected route
    return new RedirectCommand(loginUrl, { replaceUrl: true });
  }

  return true;
};
```

### Using UrlTree (Legacy)

Before `RedirectCommand`, guards returned a `UrlTree` directly. This still works but offers no control over navigation behavior:

```typescript
// Still valid, but cannot specify replaceUrl, skipLocationChange, etc.
export const authGuard: CanActivateFn = () => {
  const authService = inject(AuthService);
  const router = inject(Router);

  if (!authService.isAuthenticated()) {
    return router.parseUrl('/login');
  }

  return true;
};
```

### Redirecting from Resolvers

```typescript
import { ResolveFn, Router, RedirectCommand } from '@angular/router';

export const productResolver: ResolveFn<Product> = (route) => {
  const productService = inject(ProductService);
  const router = inject(Router);
  const id = route.paramMap.get('id');

  const product = productService.getById(id!);

  if (!product) {
    return new RedirectCommand(router.parseUrl('/products'), {
      replaceUrl: true
    });
  }

  return product;
};
```

## Router.createUrlTree()

Build a `UrlTree` without navigating. Useful for generating URLs programmatically (for `href` attributes, clipboard copying, conditional logic, etc.):

```typescript
@Component({ /* ... */ })
export class ShareComponent {
  private router = inject(Router);
  private route = inject(ActivatedRoute);

  getShareUrl(productId: number): string {
    const tree = this.router.createUrlTree(
      ['/products', productId],
      { queryParams: { ref: 'share' } }
    );
    // Serialize to a string
    return this.router.serializeUrl(tree);
    // Returns: /products/42?ref=share
  }

  getRelativeUrl(): string {
    const tree = this.router.createUrlTree(
      ['edit'],
      { relativeTo: this.route }
    );
    return this.router.serializeUrl(tree);
  }
}
```

### Router.parseUrl()

The inverse of `serializeUrl()`. Converts a URL string into a `UrlTree`:

```typescript
const tree = this.router.parseUrl('/products/42?sort=price#reviews');

console.log(tree.root.children['primary'].segments[0].path); // 'products'
console.log(tree.queryParams['sort']);                         // 'price'
console.log(tree.fragment);                                    // 'reviews'
```

## Location Service

Angular's `Location` service (from `@angular/common`) provides lower-level URL manipulation that bypasses the router. Use it only for operations that should not trigger route resolution.

```typescript
import { Location } from '@angular/common';

@Component({ /* ... */ })
export class DetailComponent {
  private location = inject(Location);

  goBack(): void {
    this.location.back();
  }

  goForward(): void {
    this.location.forward();
  }

  // Replace the URL without triggering navigation
  updateUrl(newPath: string): void {
    this.location.replaceState(newPath);
  }

  // Read the current path
  currentPath(): string {
    return this.location.path();
  }
}
```

### When to Use Location vs Router

| Use case | Use |
|---|---|
| Navigate to a route | `Router.navigate()` or `Router.navigateByUrl()` |
| Go back in history | `Location.back()` |
| Change URL without route resolution | `Location.replaceState()` |
| Read the current URL path | `Location.path()` |
| Check if a URL matches a base path | `Location.isCurrentPathEqualTo()` |

**Bad Practice: Using `Location.go()` for navigation**

```typescript
// BAD: Bypasses the router, components do not update
this.location.go('/products');

// GOOD: Uses the router, full navigation lifecycle runs
this.router.navigateByUrl('/products');
```

`Location.go()` and `Location.replaceState()` update the browser URL but do NOT trigger route resolution. The router state, activated components, guards, and resolvers are all skipped. Only use these methods when you intentionally want to change the displayed URL without affecting the route tree.

## Navigation Timing and Lifecycle

### Waiting for Navigation to Complete

```typescript
async onSubmit(): Promise<void> {
  const saved = await this.formService.save(this.formData);

  if (saved) {
    // Wait for navigation to complete before doing cleanup
    const navigated = await this.router.navigate(['/success']);
    if (navigated) {
      this.cleanup();
    }
  }
}
```

### Canceling In-Flight Navigations

Starting a new navigation automatically cancels any in-progress navigation. The canceled navigation receives a `NavigationCancel` event with code `SupersededByNewNavigation`:

```typescript
// First navigation starts
this.router.navigate(['/page-a']);

// Immediately starting another navigation cancels the first
this.router.navigate(['/page-b']);
// Only /page-b navigation completes
```

### Preventing Duplicate Navigations

```typescript
@Component({ /* ... */ })
export class SearchComponent {
  private router = inject(Router);
  private isNavigating = false;

  async search(query: string): Promise<void> {
    if (this.isNavigating) return;

    this.isNavigating = true;
    try {
      await this.router.navigate(['/search'], {
        queryParams: { q: query }
      });
    } finally {
      this.isNavigating = false;
    }
  }
}
```

A simpler approach uses `Router.currentNavigation`:

```typescript
search(query: string): void {
  // Skip if a navigation is already in progress
  if (this.router.currentNavigation()) return;

  this.router.navigate(['/search'], {
    queryParams: { q: query }
  });
}
```

## Building Navigation Utilities

### Typed Navigation Helper

Create a type-safe navigation wrapper for your application's routes:

```typescript
// app-routes.ts
export const AppRoutes = {
  home: () => ['/'] as const,
  products: () => ['/products'] as const,
  productDetail: (id: number) => ['/products', id] as const,
  productEdit: (id: number) => ['/products', id, 'edit'] as const,
  search: (query: string) => ({
    commands: ['/search'] as const,
    extras: { queryParams: { q: query } }
  }),
  userProfile: (username: string) => ['/users', username] as const,
} as const;
```

```typescript
// navigation.service.ts
@Injectable({ providedIn: 'root' })
export class NavigationService {
  private router = inject(Router);

  navigateTo(
    routeOrConfig: any[] | { commands: any[]; extras?: NavigationExtras },
    extras?: NavigationExtras
  ): Promise<boolean> {
    if (Array.isArray(routeOrConfig)) {
      return this.router.navigate(routeOrConfig, extras);
    }
    return this.router.navigate(routeOrConfig.commands, {
      ...routeOrConfig.extras,
      ...extras
    });
  }
}
```

```typescript
// Usage in components
@Component({ /* ... */ })
export class ProductCardComponent {
  private nav = inject(NavigationService);

  viewProduct(id: number): void {
    this.nav.navigateTo(AppRoutes.productDetail(id));
  }

  editProduct(id: number): void {
    this.nav.navigateTo(AppRoutes.productEdit(id));
  }

  searchProducts(query: string): void {
    this.nav.navigateTo(AppRoutes.search(query));
  }
}
```

### Back Navigation with Fallback

```typescript
@Injectable({ providedIn: 'root' })
export class BackNavigationService {
  private location = inject(Location);
  private router = inject(Router);

  // Go back if there is history, otherwise navigate to fallback
  back(fallbackUrl: string = '/'): void {
    if (window.history.length > 1) {
      this.location.back();
    } else {
      this.router.navigateByUrl(fallbackUrl);
    }
  }
}
```

## Common Mistakes

- **Forgetting `relativeTo` for relative navigation.** Without `relativeTo`, `navigate(['edit'])` resolves to `/edit`, not `./edit` relative to the current route. Always pass `{ relativeTo: this.route }` when navigating to child or sibling routes.

- **Using `Location.go()` instead of `Router.navigate()`.** `Location.go()` changes the URL but does not trigger router navigation. Components, guards, and resolvers are not executed. Use `Router.navigate()` or `Router.navigateByUrl()` for actual route changes.

- **Not handling the navigate() return value.** `navigate()` returns a `Promise<boolean>`. If you need to perform actions after navigation completes (cleanup, logging), `await` the promise. Ignoring it can lead to race conditions.

- **Passing non-serializable data in `state`.** Functions, class instances, `Map`, `Set`, and objects with circular references cannot be serialized to `history.state`. Keep state data as plain objects with primitive values.

- **Using `skipLocationChange` for page-level navigation.** This breaks the back button and makes the URL meaningless. Reserve it for overlay-style navigations (modals, drawers) where the underlying page has not conceptually changed.

- **Hardcoding URLs as strings throughout the codebase.** Scattered string literals like `'/users/' + id + '/edit'` are fragile. Centralize route paths in a constants file or typed helper.

---

## Best Practices

- **Use `navigate()` for dynamic routes, `navigateByUrl()` for known absolute URLs.** This makes intent clear. `navigate()` with its commands array is more flexible for building URLs from variables. `navigateByUrl()` is cleaner when the full path is already a string.

- **Always specify `replaceUrl: true` for post-authentication redirects.** After login, the user should not be able to press back and return to the login form. The same applies to wizard completion pages and other one-way transitions.

- **Centralize route paths.** Define route path constants or typed helper functions in a single file. This prevents typos and makes refactoring easier.

- **Use `RedirectCommand` over `UrlTree` in guards and resolvers.** `RedirectCommand` lets you specify navigation behavior options (`replaceUrl`, `skipLocationChange`) that `UrlTree` does not support.

- **Prefer `info` over `state` for transient navigation context.** If the data only needs to live during the navigation lifecycle (guards, resolvers, events), use `info`. It avoids polluting `history.state` and is automatically discarded.

- **Use `Location.back()` for "go back" buttons.** It respects the browser's history stack. Combine it with a fallback URL for cases where there is no history (direct link or new tab).

## Real-World Use Cases

### 1. Multi-Step Wizard with History Control

```typescript
@Component({ /* ... */ })
export class WizardComponent {
  private router = inject(Router);
  private route = inject(ActivatedRoute);

  currentStep = input.required<number, string>({ transform: numberAttribute });

  nextStep(): void {
    const next = this.currentStep() + 1;
    // Replace URL so back button goes to the page before the wizard,
    // not to the previous step
    this.router.navigate(['../step', next], {
      relativeTo: this.route,
      replaceUrl: true
    });
  }

  previousStep(): void {
    const prev = this.currentStep() - 1;
    if (prev >= 1) {
      this.router.navigate(['../step', prev], {
        relativeTo: this.route,
        replaceUrl: true
      });
    }
  }

  finish(): void {
    this.router.navigate(['/wizard/complete'], {
      state: { completedSteps: this.currentStep() },
      replaceUrl: true
    });
  }
}
```

### 2. Search with URL State Synchronization

```typescript
@Component({
  selector: 'app-search',
  template: `
    <input
      [value]="query()"
      (input)="onSearchInput($event)"
      placeholder="Search..."
    />
    <app-search-results [results]="results()" />
  `
})
export class SearchComponent {
  private router = inject(Router);
  private route = inject(ActivatedRoute);
  private searchService = inject(SearchService);

  // Bound from query params via withComponentInputBinding()
  query = input<string>('');
  sort = input<string>('relevance');

  results = computed(() =>
    this.searchService.search(this.query(), this.sort())
  );

  onSearchInput(event: Event): void {
    const value = (event.target as HTMLInputElement).value;
    // Update URL without creating a new history entry for each keystroke
    this.router.navigate([], {
      relativeTo: this.route,
      queryParams: { q: value },
      queryParamsHandling: 'merge',
      replaceUrl: true   // typing should not create history entries
    });
  }

  changeSort(sort: string): void {
    // Changing sort SHOULD create a history entry
    this.router.navigate([], {
      relativeTo: this.route,
      queryParams: { sort },
      queryParamsHandling: 'merge'
      // No replaceUrl, so the user can go back to the previous sort
    });
  }
}
```

### 3. Conditional Navigation Based on User Role

```typescript
@Injectable({ providedIn: 'root' })
export class PostLoginNavigationService {
  private router = inject(Router);
  private authService = inject(AuthService);

  navigateAfterLogin(returnUrl?: string): void {
    const user = this.authService.currentUser();

    if (returnUrl && !returnUrl.startsWith('/login')) {
      // Return to the page the user originally wanted
      this.router.navigateByUrl(returnUrl, { replaceUrl: true });
      return;
    }

    // Role-based default landing page
    switch (user?.role) {
      case 'admin':
        this.router.navigate(['/admin/dashboard'], { replaceUrl: true });
        break;
      case 'manager':
        this.router.navigate(['/team/overview'], { replaceUrl: true });
        break;
      default:
        this.router.navigate(['/dashboard'], { replaceUrl: true });
        break;
    }
  }
}
```

> **Key Takeaway:** Use `navigate()` with a commands array for dynamic route construction and relative navigation, `navigateByUrl()` for known absolute URLs, and always leverage `NavigationExtras` properties like `replaceUrl`, `state`, and `info` to control history behavior and pass contextual data without polluting the URL.
