# 15.11 Route Data and Page Titles

> Angular routes carry metadata through the `data` property for static values and resolvers for dynamic values, while the `title` property and `TitleStrategy` give you built-in control over the document title on every navigation.

---

## Static Route Data

The `data` property on a `Route` object holds an arbitrary key-value map of static metadata. It does not change between navigations and is available immediately when the route activates.

```typescript
const routes: Routes = [
  {
    path: 'dashboard',
    component: DashboardComponent,
    data: {
      analyticsId: 'page-dashboard',
      requiredRole: 'admin',
      showSidebar: true,
      breadcrumb: 'Dashboard'
    }
  },
  {
    path: 'settings',
    component: SettingsComponent,
    data: {
      analyticsId: 'page-settings',
      requiredRole: 'user',
      showSidebar: false,
      breadcrumb: 'Settings'
    }
  }
];
```

### Reading Static Data with Signal Inputs

With `withComponentInputBinding()`, static route data binds directly to component inputs. The input name must match the key in the `data` object:

```typescript
// app.config.ts
provideRouter(routes, withComponentInputBinding());

// dashboard.component.ts
@Component({
  selector: 'app-dashboard',
  template: `
    @if (showSidebar()) {
      <app-sidebar />
    }
    <main>Dashboard content</main>
  `
})
export class DashboardComponent {
  // Bound from route data { showSidebar: true }
  showSidebar = input<boolean>(false);
  breadcrumb = input<string>('');
  analyticsId = input<string>('');
}
```

### Reading Static Data with ActivatedRoute

The legacy approach uses `ActivatedRoute.data` or `ActivatedRoute.snapshot.data`:

```typescript
@Component({ /* ... */ })
export class DashboardComponent {
  private route = inject(ActivatedRoute);

  // Snapshot (non-reactive, read once)
  analyticsId = this.route.snapshot.data['analyticsId'];

  // Observable (reactive, updates if data changes via resolver re-execution)
  data$ = this.route.data;
}
```

### Data Binding Priority

When `withComponentInputBinding()` is enabled and multiple sources share the same key name, the binding follows this priority order:

| Priority | Source | Example |
|---|---|---|
| 1 (highest) | Route `data` and resolver data | `data: { id: 'static' }` |
| 2 | Path parameters | `:id` in the URL |
| 3 (lowest) | Query parameters | `?id=value` |

This means if you have a route parameter `:id` and also `data: { id: 'override' }`, the component input `id` will receive `'override'`.

## Resolved Route Data

Resolvers produce dynamic data that is merged into the route's `data` object before the component activates. The resolved values are accessible through the same mechanisms as static data.

```typescript
// product.resolver.ts
export const productResolver: ResolveFn<Product> = (route) => {
  const productService = inject(ProductService);
  return productService.getById(route.paramMap.get('id')!);
};

// routes
const routes: Routes = [
  {
    path: 'products/:id',
    component: ProductDetailComponent,
    data: { breadcrumb: 'Product Detail' },     // static
    resolve: { product: productResolver }        // dynamic
  }
];

// component
@Component({ /* ... */ })
export class ProductDetailComponent {
  // Static data bound via input
  breadcrumb = input<string>('');

  // Resolved data bound via input (same mechanism)
  product = input<Product>();
}
```

Both static `data` and `resolve` data are merged into a single data object. If they share a key, the resolved value wins.

## The title Property

Every route can specify a `title` that Angular automatically sets as the document title (`document.title`) when the route activates.

### Static Titles

```typescript
const routes: Routes = [
  {
    path: '',
    component: HomeComponent,
    title: 'Home'
  },
  {
    path: 'about',
    component: AboutComponent,
    title: 'About Us'
  },
  {
    path: 'contact',
    component: ContactComponent,
    title: 'Contact'
  }
];
```

When the user navigates to `/about`, the browser tab displays "About Us".

### Dynamic Titles with ResolveFn

The `title` property also accepts a `ResolveFn<string>`. This runs like any resolver but its return value becomes the page title:

```typescript
// Dynamic title based on route parameters
const productTitleResolver: ResolveFn<string> = (route) => {
  const productService = inject(ProductService);
  const id = route.paramMap.get('id')!;
  return productService.getById(id).pipe(
    map(product => product.name)
  );
};

const routes: Routes = [
  {
    path: 'products/:id',
    component: ProductDetailComponent,
    title: productTitleResolver
  }
];
```

When the user navigates to `/products/42`, the resolver fetches the product and sets the document title to the product's name.

### Title Resolver with Existing Resolved Data

A title resolver runs in the same phase as other resolvers. However, it does not have access to data resolved by sibling resolvers in the `resolve` object because they run in parallel. If the title depends on resolved data, use a different approach:

```typescript
// BAD: Cannot access sibling resolver data
const routes: Routes = [
  {
    path: 'products/:id',
    resolve: { product: productResolver },
    title: (route) => {
      // route.data['product'] is NOT available here
      // because resolvers run in parallel
    }
  }
];

// GOOD: Fetch the data independently in the title resolver
const routes: Routes = [
  {
    path: 'products/:id',
    resolve: { product: productResolver },
    title: (route) => {
      const productService = inject(ProductService);
      return productService.getById(route.paramMap.get('id')!).pipe(
        map(p => p.name)
      );
    }
  }
];

// ALTERNATIVE: Use a shared cache so both resolvers hit the same data
```

### Title in Child Routes

The default behavior selects the title from the deepest activated primary outlet. Child route titles override parent titles:

```typescript
const routes: Routes = [
  {
    path: 'admin',
    component: AdminLayoutComponent,
    title: 'Admin',                      // used when /admin is the deepest match
    children: [
      {
        path: 'users',
        component: UserListComponent,
        title: 'Users'                   // overrides 'Admin' at /admin/users
      },
      {
        path: 'users/:id',
        component: UserDetailComponent,
        title: userTitleResolver          // overrides at /admin/users/42
      },
      {
        path: 'settings',
        component: SettingsComponent
        // No title: falls back to parent's 'Admin'
      }
    ]
  }
];
```

If a child route has no `title`, Angular walks up the route tree and uses the nearest ancestor's title.

## TitleStrategy

`TitleStrategy` is the abstract class that controls how the document title is set. Angular ships with `DefaultTitleStrategy`, which simply sets `document.title` to the deepest route's title value.

### DefaultTitleStrategy Behavior

```
Route tree: /admin/users/42
Titles:     'Admin' -> 'Users' -> 'User: Alice'
Result:     document.title = 'User: Alice'
```

It traverses the router state, finds the deepest primary outlet with a title, and sets it directly.

### Creating a Custom TitleStrategy

Override `TitleStrategy` to add prefixes, suffixes, separators, or other custom logic:

```typescript
import { Injectable, inject } from '@angular/core';
import { TitleStrategy, RouterStateSnapshot } from '@angular/router';
import { Title } from '@angular/platform-browser';

@Injectable()
export class AppTitleStrategy extends TitleStrategy {
  private title = inject(Title);

  private readonly appName = 'MyApp';

  override updateTitle(snapshot: RouterStateSnapshot): void {
    const pageTitle = this.buildTitle(snapshot);

    if (pageTitle) {
      this.title.setTitle(`${pageTitle} | ${this.appName}`);
    } else {
      this.title.setTitle(this.appName);
    }
  }
}
```

### Providing the Custom Strategy

```typescript
// app.config.ts
import { TitleStrategy } from '@angular/router';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
    { provide: TitleStrategy, useClass: AppTitleStrategy }
  ]
};
```

### TitleStrategy API

```typescript
abstract class TitleStrategy {
  // Must implement: sets the document title
  abstract updateTitle(snapshot: RouterStateSnapshot): void;

  // Traverses the state and returns the deepest title string (or undefined)
  buildTitle(snapshot: RouterStateSnapshot): string | undefined;

  // Returns the resolved title for a specific route snapshot
  getResolvedTitleForRoute(snapshot: ActivatedRouteSnapshot): any;
}
```

`buildTitle()` does the heavy lifting of walking the route tree. Your `updateTitle()` implementation typically calls `this.buildTitle(snapshot)` to get the raw title, then formats it as needed.

`getResolvedTitleForRoute()` returns the title for a specific `ActivatedRouteSnapshot`. This is useful when building breadcrumb-style titles that concatenate titles from multiple route levels.

## Advanced Title Strategies

### Breadcrumb Title Strategy

Concatenate titles from all levels of the route tree:

```typescript
@Injectable()
export class BreadcrumbTitleStrategy extends TitleStrategy {
  private title = inject(Title);
  private readonly separator = ' > ';
  private readonly appName = 'MyApp';

  override updateTitle(snapshot: RouterStateSnapshot): void {
    const titles: string[] = [];
    let route: ActivatedRouteSnapshot | null = snapshot.root;

    while (route) {
      const routeTitle = this.getResolvedTitleForRoute(route);
      if (routeTitle) {
        titles.push(routeTitle);
      }
      route = route.firstChild;
    }

    const fullTitle = titles.length > 0
      ? `${titles.join(this.separator)} | ${this.appName}`
      : this.appName;

    this.title.setTitle(fullTitle);
  }
}
```

Result for `/admin/users/42`: `Admin > Users > Alice | MyApp`

### Title Strategy with Translation

```typescript
@Injectable()
export class TranslatedTitleStrategy extends TitleStrategy {
  private title = inject(Title);
  private translateService = inject(TranslateService);
  private readonly appName = 'MyApp';

  override updateTitle(snapshot: RouterStateSnapshot): void {
    const titleKey = this.buildTitle(snapshot);

    if (titleKey) {
      // Treat the route title as a translation key
      const translated = this.translateService.instant(titleKey);
      this.title.setTitle(`${translated} | ${this.appName}`);
    } else {
      this.title.setTitle(this.appName);
    }
  }
}
```

Route configuration with translation keys:

```typescript
const routes: Routes = [
  {
    path: 'dashboard',
    component: DashboardComponent,
    title: 'nav.dashboard'          // translation key, not literal text
  },
  {
    path: 'settings',
    component: SettingsComponent,
    title: 'nav.settings'
  }
];
```

### Title Strategy with Environment Prefix

```typescript
@Injectable()
export class EnvironmentTitleStrategy extends TitleStrategy {
  private title = inject(Title);
  private env = inject(ENVIRONMENT);

  override updateTitle(snapshot: RouterStateSnapshot): void {
    const pageTitle = this.buildTitle(snapshot) ?? 'App';
    const prefix = this.env.production ? '' : `[${this.env.name}] `;
    this.title.setTitle(`${prefix}${pageTitle}`);
  }
}
```

Result in staging: `[Staging] Dashboard`
Result in production: `Dashboard`

## Static Data Patterns

### Role-Based Access Control

```typescript
const routes: Routes = [
  {
    path: 'admin',
    component: AdminComponent,
    data: { roles: ['admin'] },
    canActivate: [roleGuard]
  },
  {
    path: 'reports',
    component: ReportsComponent,
    data: { roles: ['admin', 'manager'] },
    canActivate: [roleGuard]
  }
];

// role.guard.ts
export const roleGuard: CanActivateFn = (route) => {
  const authService = inject(AuthService);
  const router = inject(Router);

  const requiredRoles = route.data['roles'] as string[];
  const userRole = authService.currentUser()?.role;

  if (requiredRoles.includes(userRole!)) {
    return true;
  }

  return new RedirectCommand(router.parseUrl('/unauthorized'));
};
```

### Animation Metadata

```typescript
const routes: Routes = [
  {
    path: 'home',
    component: HomeComponent,
    data: { animation: 'HomePage' }
  },
  {
    path: 'about',
    component: AboutComponent,
    data: { animation: 'AboutPage' }
  },
  {
    path: 'contact',
    component: ContactComponent,
    data: { animation: 'ContactPage' }
  }
];

// app.component.ts
@Component({
  selector: 'app-root',
  template: `
    <div [@routeAnimations]="getAnimationData()">
      <router-outlet />
    </div>
  `,
  animations: [routeAnimations]
})
export class AppComponent {
  private route = inject(ActivatedRoute);

  getAnimationData(): string {
    return this.route.firstChild?.snapshot.data['animation'] ?? '';
  }
}
```

### Feature Flags

```typescript
const routes: Routes = [
  {
    path: 'new-feature',
    component: NewFeatureComponent,
    data: { featureFlag: 'new-dashboard' },
    canMatch: [featureFlagGuard]
  },
  {
    path: 'new-feature',
    component: OldFeatureComponent
    // Fallback if the feature flag is off
  }
];

export const featureFlagGuard: CanMatchFn = (route) => {
  const featureService = inject(FeatureFlagService);
  const flag = route.data?.['featureFlag'] as string;
  return featureService.isEnabled(flag);
};
```

## Accessing Data in Parent and Child Routes

### Data Inheritance

By default, a child route does NOT inherit its parent's `data`. Each route has its own isolated data object.

```typescript
const routes: Routes = [
  {
    path: 'admin',
    data: { section: 'admin' },
    children: [
      {
        path: 'users',
        component: UserListComponent
        // data['section'] is NOT available here by default
      }
    ]
  }
];
```

To enable inheritance, configure `paramsInheritanceStrategy`:

```typescript
provideRouter(
  routes,
  withRouterConfig({ paramsInheritanceStrategy: 'always' })
)
```

With `'always'`, child routes inherit all `data`, `params`, and `queryParams` from their ancestors.

### Reading Parent Data Without Inheritance

If you do not want global inheritance, read parent data explicitly:

```typescript
@Component({ /* ... */ })
export class UserListComponent {
  private route = inject(ActivatedRoute);

  // Read parent's data
  section = this.route.parent?.snapshot.data['section'];
}
```

Or walk the route tree:

```typescript
function getRouteData(route: ActivatedRoute, key: string): any {
  let current: ActivatedRoute | null = route;
  while (current) {
    if (current.snapshot.data[key] !== undefined) {
      return current.snapshot.data[key];
    }
    current = current.parent;
  }
  return undefined;
}
```

## Route Data and the Title Input

The resolved `title` is also accessible as part of the route data. With `withComponentInputBinding()`, a component can receive the title through an input named `title`:

```typescript
const routes: Routes = [
  {
    path: 'products/:id',
    component: ProductDetailComponent,
    title: productTitleResolver         // resolves to 'Widget Pro'
  }
];

@Component({
  selector: 'app-product-detail',
  template: `<h1>{{ title() }}</h1>`
})
export class ProductDetailComponent {
  // Receives the resolved title value: 'Widget Pro'
  title = input<string>('');
}
```

This means the resolved title is available both as `document.title` (via `TitleStrategy`) and as a component input.

## Typed Route Data with Injection Tokens

For type safety, define route data keys as injection tokens or use a typed wrapper:

```typescript
// route-data-keys.ts
export interface AppRouteData {
  breadcrumb?: string;
  roles?: string[];
  analyticsId?: string;
  animation?: string;
  featureFlag?: string;
}

// Typed helper to read route data
export function getTypedRouteData(route: ActivatedRouteSnapshot): AppRouteData {
  return route.data as AppRouteData;
}
```

```typescript
// In a guard
export const roleGuard: CanActivateFn = (route) => {
  const data = getTypedRouteData(route);
  const requiredRoles = data.roles ?? [];
  // ...
};
```

This does not provide runtime enforcement, but gives you compile-time autocomplete and catches typos.

## Common Mistakes

- **Expecting sibling resolvers to be available in a title resolver.** The `title` resolver runs in parallel with other resolvers in the `resolve` object. You cannot access `route.data['product']` inside the title resolver because it has not resolved yet. Either duplicate the fetch or use a shared caching layer.

- **Forgetting to provide a custom TitleStrategy.** Extending `TitleStrategy` alone does nothing. You must register it: `{ provide: TitleStrategy, useClass: AppTitleStrategy }`. Without this, Angular uses `DefaultTitleStrategy`.

- **Assuming child routes inherit parent data.** By default, `paramsInheritanceStrategy` is `'emptyOnly'`, meaning data is only inherited if the child route has an empty path. Use `'always'` for full inheritance or read parent data explicitly.

- **Storing sensitive information in route data.** Route `data` is visible in DevTools and router state snapshots. Never store secrets, tokens, or sensitive user information in `data`. Use services or environment variables instead.

- **Using route data for large datasets.** The `data` object is meant for small metadata (roles, flags, IDs). Do not store large arrays or objects in it. Use resolvers or services for data fetching.

- **Not defining a title for every route.** Missing titles result in the browser tab showing the previous page's title or the HTML's `<title>` tag. This is bad for accessibility and SEO. Always set a title, even if it is a fallback in a custom `TitleStrategy`.

---

## Best Practices

- **Always define a `title` on every route.** This improves accessibility (screen readers announce the page title), SEO (crawlers use it), and user experience (browser tabs are meaningful). Use a custom `TitleStrategy` to add a consistent suffix.

- **Use `withComponentInputBinding()` to read route data.** Signal inputs are the cleanest way to consume both static `data` and resolved data. It avoids injecting `ActivatedRoute` and manually subscribing.

- **Keep static data small and metadata-focused.** Use `data` for configuration like roles, feature flags, animation identifiers, and breadcrumb labels. Fetch actual content through resolvers or services.

- **Create a custom `TitleStrategy` early.** Even a simple one that appends your app name prevents inconsistent titles. It is much easier to set up once than to retrofit later.

- **Use a shared caching layer if the title resolver needs the same data as another resolver.** Services with internal caching (or `shareReplay`) prevent duplicate HTTP calls when both the title resolver and a data resolver need the same resource.

- **Type your route data.** Define an interface for your route data keys and use a typed accessor. This catches key name typos at compile time.

## Real-World Use Cases

### 1. SEO-Friendly Dynamic Titles with Product Names

```typescript
// Product title resolver that includes category
const productPageTitle: ResolveFn<string> = (route) => {
  const productService = inject(ProductService);
  const id = route.paramMap.get('id')!;

  return productService.getById(id).pipe(
    map(product => `${product.name} - ${product.category}`)
  );
};

const routes: Routes = [
  {
    path: 'products/:id',
    component: ProductDetailComponent,
    title: productPageTitle,
    resolve: { product: productResolver }
  }
];

// Custom strategy appends the site name
@Injectable()
export class SeoTitleStrategy extends TitleStrategy {
  private title = inject(Title);
  private meta = inject(Meta);

  override updateTitle(snapshot: RouterStateSnapshot): void {
    const pageTitle = this.buildTitle(snapshot);
    const fullTitle = pageTitle
      ? `${pageTitle} | ShopName`
      : 'ShopName - Online Store';

    this.title.setTitle(fullTitle);

    // Also update Open Graph meta tag
    this.meta.updateTag({ property: 'og:title', content: fullTitle });
  }
}
```

Result: Browser tab shows "Widget Pro - Electronics | ShopName"

### 2. Analytics Tracking via Route Data

```typescript
const routes: Routes = [
  {
    path: 'checkout',
    component: CheckoutComponent,
    title: 'Checkout',
    data: {
      analyticsCategory: 'purchase',
      analyticsAction: 'checkout-start'
    }
  },
  {
    path: 'checkout/confirm',
    component: ConfirmComponent,
    title: 'Confirm Order',
    data: {
      analyticsCategory: 'purchase',
      analyticsAction: 'checkout-confirm'
    }
  }
];

// analytics.service.ts
@Injectable({ providedIn: 'root' })
export class AnalyticsService {
  private router = inject(Router);

  constructor() {
    this.router.events
      .pipe(
        filter((e): e is NavigationEnd => e instanceof NavigationEnd),
        takeUntilDestroyed()
      )
      .subscribe(() => {
        const root = this.router.routerState.root;
        const data = this.getDeepestRouteData(root);

        if (data['analyticsCategory']) {
          this.track(data['analyticsCategory'], data['analyticsAction'], data['analyticsLabel']);
        }
      });
  }

  private getDeepestRouteData(route: ActivatedRoute): Record<string, any> {
    let current = route;
    while (current.firstChild) {
      current = current.firstChild;
    }
    return current.snapshot.data;
  }

  private track(category: string, action: string, label?: string): void {
    gtag('event', action, {
      event_category: category,
      event_label: label
    });
  }
}
```

### 3. Layout Configuration via Route Data

```typescript
const routes: Routes = [
  {
    path: '',
    component: MainLayoutComponent,
    data: { layout: 'full' },
    children: [
      {
        path: 'dashboard',
        component: DashboardComponent,
        title: 'Dashboard',
        data: { layout: 'sidebar', sidebarWidth: 280 }
      },
      {
        path: 'settings',
        component: SettingsComponent,
        title: 'Settings',
        data: { layout: 'sidebar', sidebarWidth: 220 }
      },
      {
        path: 'profile',
        component: ProfileComponent,
        title: 'Profile',
        data: { layout: 'full' }
      }
    ]
  }
];

// main-layout.component.ts
@Component({
  selector: 'app-main-layout',
  template: `
    <app-header />
    <div class="content" [class.has-sidebar]="layoutConfig().layout === 'sidebar'">
      @if (layoutConfig().layout === 'sidebar') {
        <app-sidebar [width]="layoutConfig().sidebarWidth ?? 250" />
      }
      <main>
        <router-outlet />
      </main>
    </div>
  `
})
export class MainLayoutComponent {
  private router = inject(Router);
  private route = inject(ActivatedRoute);

  layoutConfig = toSignal(
    this.router.events.pipe(
      filter((e): e is NavigationEnd => e instanceof NavigationEnd),
      map(() => {
        let current = this.route;
        while (current.firstChild) {
          current = current.firstChild;
        }
        return current.snapshot.data as {
          layout: 'full' | 'sidebar';
          sidebarWidth?: number;
        };
      })
    ),
    { initialValue: { layout: 'full' as const } }
  );
}
```

> **Key Takeaway:** Use the route `data` property for small, static metadata (roles, flags, breadcrumbs) and the `title` property (with `ResolveFn<string>` for dynamic titles) on every route, backed by a custom `TitleStrategy` that adds your app name and handles missing titles consistently.
