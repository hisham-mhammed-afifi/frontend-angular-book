# 15.12 View Transitions API Integration

> Angular's router integrates with the browser's native View Transitions API to create smooth, CSS-driven animations between route changes without a single line of JavaScript animation code.

---

## How View Transitions Work

The View Transitions API is a browser-native mechanism for animating DOM changes. When a view transition starts, the browser:

1. **Captures the old state.** Takes a screenshot of the current page (or specific named elements).
2. **Executes the DOM update.** Angular's router swaps the old component out and the new component in.
3. **Captures the new state.** Takes a snapshot of the updated page.
4. **Animates between states.** The browser cross-fades from old to new using CSS pseudo-elements.

This all happens through `document.startViewTransition()`, which Angular calls automatically when `withViewTransitions()` is enabled.

### The Pseudo-Element Tree

During a transition, the browser creates a tree of pseudo-elements that you can target with CSS:

```
::view-transition
  ::view-transition-group(root)
    ::view-transition-image-pair(root)
      ::view-transition-old(root)       -- screenshot of the old state
      ::view-transition-new(root)       -- live capture of the new state
```

If you assign `view-transition-name` to specific elements, each named element gets its own group:

```
::view-transition
  ::view-transition-group(root)
    ::view-transition-image-pair(root)
      ::view-transition-old(root)
      ::view-transition-new(root)
  ::view-transition-group(hero-image)
    ::view-transition-image-pair(hero-image)
      ::view-transition-old(hero-image)
      ::view-transition-new(hero-image)
```

## Enabling View Transitions in Angular

### Standalone Application (provideRouter)

```typescript
import { provideRouter, withViewTransitions } from '@angular/router';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(
      routes,
      withViewTransitions()
    )
  ]
};
```

### NgModule Application

```typescript
@NgModule({
  imports: [
    RouterModule.forRoot(routes, {
      enableViewTransitions: true
    })
  ]
})
export class AppRoutingModule {}
```

That is all you need for a basic cross-fade between routes. The browser will fade the entire page from old to new by default.

## ViewTransitionsFeatureOptions

`withViewTransitions()` accepts an optional configuration object:

```typescript
interface ViewTransitionsFeatureOptions {
  skipInitialTransition?: boolean;
  onViewTransitionCreated?: (transitionInfo: ViewTransitionInfo) => void;
}
```

### skipInitialTransition

When the application loads for the first time, the initial route activation triggers a view transition. This can look odd because there is no "old" state to transition from. Set `skipInitialTransition: true` to disable the animation on first load:

```typescript
provideRouter(
  routes,
  withViewTransitions({
    skipInitialTransition: true
  })
)
```

### onViewTransitionCreated

A callback that runs after `document.startViewTransition()` is called. It receives a `ViewTransitionInfo` object and runs in an injection context, so you can use `inject()`.

```typescript
interface ViewTransitionInfo {
  transition: ViewTransition;          // the native ViewTransition object
  from: ActivatedRouteSnapshot;        // route being navigated away from
  to: ActivatedRouteSnapshot;          // route being navigated to
}
```

The native `ViewTransition` object has these properties:

```typescript
interface ViewTransition {
  finished: Promise<void>;             // resolves when the animation finishes
  ready: Promise<void>;                // resolves when pseudo-elements are created
  updateCallbackDone: Promise<void>;   // resolves when the DOM update callback completes
  skipTransition(): void;              // cancels the animation, applies DOM changes instantly
}
```

## Skipping Transitions Conditionally

Use `onViewTransitionCreated` to skip transitions for specific navigations:

### Skip Fragment-Only Navigation

When the user clicks an anchor link that scrolls to a section on the same page, animating the entire page is unnecessary:

```typescript
provideRouter(
  routes,
  withViewTransitions({
    onViewTransitionCreated: (info) => {
      const router = inject(Router);
      const targetUrl = router.parseUrl(router.serializeUrl(
        router.createUrlTree([], { relativeTo: undefined })
      ));

      // Skip transition if only the fragment changed
      if (info.from.url === info.to.url) {
        info.transition.skipTransition();
      }
    }
  })
)
```

### Skip Based on Route Data

```typescript
// Route config
const routes: Routes = [
  {
    path: 'modal',
    component: ModalComponent,
    data: { skipViewTransition: true }
  },
  {
    path: 'dashboard',
    component: DashboardComponent
  }
];

// Configuration
provideRouter(
  routes,
  withViewTransitions({
    onViewTransitionCreated: (info) => {
      if (info.to.data['skipViewTransition']) {
        info.transition.skipTransition();
      }
    }
  })
)
```

## CSS Customization

The default transition is a cross-fade of the entire page. You control the animation entirely through CSS pseudo-elements defined in your **global stylesheet**.

### Why Global Styles Only?

Angular's view encapsulation (Emulated by default) scopes component styles with attribute selectors. The view transition pseudo-elements exist outside the component DOM tree, so scoped styles cannot target them. All view transition CSS must go in `styles.css` (or your global stylesheet).

### Customizing the Default Transition

```css
/* styles.css */

/* Change the duration of the default cross-fade */
::view-transition-old(root),
::view-transition-new(root) {
  animation-duration: 300ms;
}
```

### Slide Transitions

```css
/* styles.css */

/* Slide the old page out to the left */
::view-transition-old(root) {
  animation: slide-out-left 300ms ease-in both;
}

/* Slide the new page in from the right */
::view-transition-new(root) {
  animation: slide-in-right 300ms ease-out both;
}

@keyframes slide-out-left {
  from { transform: translateX(0); }
  to   { transform: translateX(-100%); }
}

@keyframes slide-in-right {
  from { transform: translateX(100%); }
  to   { transform: translateX(0); }
}
```

### Fade + Scale

```css
::view-transition-old(root) {
  animation: fade-and-shrink 250ms ease-in both;
}

::view-transition-new(root) {
  animation: fade-and-grow 250ms ease-out both;
}

@keyframes fade-and-shrink {
  from { opacity: 1; transform: scale(1); }
  to   { opacity: 0; transform: scale(0.95); }
}

@keyframes fade-and-grow {
  from { opacity: 0; transform: scale(1.05); }
  to   { opacity: 1; transform: scale(1); }
}
```

### Disabling Transition for Specific Elements

If an element should not participate in the page-level transition:

```css
.no-transition {
  view-transition-name: none;
}
```

## Named View Transitions (Element-Level)

The real power of view transitions is animating individual elements between pages. For example, a product image on a list page can smoothly morph into the hero image on the detail page.

### How It Works

1. Assign the same `view-transition-name` to the element on both pages.
2. The browser matches elements by name across the old and new states.
3. Each named element gets its own transition group with independent animation.

### Static Names in CSS

```css
/* On the list page, applied to the product card image */
.product-image {
  view-transition-name: product-hero;
}

/* On the detail page, applied to the hero image */
.detail-hero-image {
  view-transition-name: product-hero;
}
```

**Problem:** If the list page has multiple product images, they would all share the same `view-transition-name`, which is invalid (names must be unique within a document).

### Dynamic Names with Style Binding

Assign unique names per element using Angular's style binding:

```typescript
// product-list.component.ts
@Component({
  selector: 'app-product-list',
  template: `
    @for (product of products(); track product.id) {
      <div class="product-card">
        <img
          [src]="product.imageUrl"
          [style.view-transition-name]="'product-image-' + product.id"
          (click)="viewProduct(product.id)"
        />
        <h3 [style.view-transition-name]="'product-name-' + product.id">
          {{ product.name }}
        </h3>
      </div>
    }
  `
})
export class ProductListComponent {
  products = input.required<Product[]>();
  private router = inject(Router);

  viewProduct(id: number): void {
    this.router.navigate(['/products', id]);
  }
}
```

```typescript
// product-detail.component.ts
@Component({
  selector: 'app-product-detail',
  template: `
    <img
      [src]="product().imageUrl"
      [style.view-transition-name]="'product-image-' + product().id"
    />
    <h1 [style.view-transition-name]="'product-name-' + product().id">
      {{ product().name }}
    </h1>
    <p>{{ product().description }}</p>
  `
})
export class ProductDetailComponent {
  product = input.required<Product>();
}
```

The browser sees `view-transition-name: product-image-42` on an element in the old page and the same name on an element in the new page. It morphs the old into the new with a smooth animation.

### CSS for Named Transitions

```css
/* styles.css */

/* Animate the product image with a custom easing */
::view-transition-group(product-image-*) {
  animation-duration: 400ms;
  animation-timing-function: cubic-bezier(0.4, 0, 0.2, 1);
}

/* The old product name fades out */
::view-transition-old(product-name-*) {
  animation: fade-out 200ms ease-in;
}

/* The new product name fades in */
::view-transition-new(product-name-*) {
  animation: fade-in 200ms ease-out 200ms;
}
```

**Note:** The wildcard `*` syntax in pseudo-element selectors for view transitions (like `::view-transition-group(*.card)`) is a newer CSS feature. Check browser support before relying on it. You may need to target specific names without wildcards.

## Directive for Dynamic View Transition Names

Encapsulate the logic for applying `view-transition-name` in a reusable directive:

```typescript
import { Directive, input, effect, ElementRef, inject } from '@angular/core';

@Directive({
  selector: '[viewTransitionName]',
  standalone: true
})
export class ViewTransitionNameDirective {
  private el = inject(ElementRef);

  viewTransitionName = input.required<string>();

  constructor() {
    effect(() => {
      this.el.nativeElement.style.viewTransitionName = this.viewTransitionName();
    });
  }
}
```

Usage:

```html
<img [viewTransitionName]="'product-image-' + product().id" [src]="product().imageUrl" />
```

### Conditional View Transition Names

Only apply the name when the element should participate in a transition:

```typescript
@Directive({
  selector: '[viewTransitionName]',
  standalone: true
})
export class ViewTransitionNameDirective {
  private el = inject(ElementRef);
  private route = inject(ActivatedRoute);

  viewTransitionName = input.required<string>();
  viewTransitionActive = input(true);

  constructor() {
    effect(() => {
      const name = this.viewTransitionActive()
        ? this.viewTransitionName()
        : '';
      this.el.nativeElement.style.viewTransitionName = name || 'none';
    });
  }
}
```

## Tracking Transition Direction

A common need is to animate differently based on navigation direction (forward vs. backward). The View Transitions API does not provide this natively, but you can track it with `onViewTransitionCreated`:

```typescript
// view-transition.service.ts
@Injectable({ providedIn: 'root' })
export class ViewTransitionService {
  direction = signal<'forward' | 'back' | 'none'>('none');
  currentTransition = signal<ViewTransition | null>(null);
}

// app.config.ts
provideRouter(
  routes,
  withViewTransitions({
    skipInitialTransition: true,
    onViewTransitionCreated: (info) => {
      const service = inject(ViewTransitionService);
      const router = inject(Router);

      // Determine direction from NavigationStart trigger
      const nav = router.getCurrentNavigation();
      const trigger = (nav?.extras as any)?.['navigationTrigger'];

      if (trigger === 'popstate') {
        service.direction.set('back');
      } else {
        service.direction.set('forward');
      }

      service.currentTransition.set(info.transition);

      // Add a CSS class to the document for direction-based styling
      document.documentElement.dataset['transitionDirection'] =
        service.direction();

      info.transition.finished.finally(() => {
        service.currentTransition.set(null);
        delete document.documentElement.dataset['transitionDirection'];
      });
    }
  })
)
```

Then in your CSS:

```css
/* styles.css */

/* Forward navigation: slide right */
[data-transition-direction="forward"]::view-transition-old(root) {
  animation: slide-out-left 300ms ease-in both;
}
[data-transition-direction="forward"]::view-transition-new(root) {
  animation: slide-in-right 300ms ease-out both;
}

/* Back navigation: slide left */
[data-transition-direction="back"]::view-transition-old(root) {
  animation: slide-out-right 300ms ease-in both;
}
[data-transition-direction="back"]::view-transition-new(root) {
  animation: slide-in-left 300ms ease-out both;
}

@keyframes slide-out-left {
  to { transform: translateX(-100%); opacity: 0; }
}
@keyframes slide-in-right {
  from { transform: translateX(100%); opacity: 0; }
}
@keyframes slide-out-right {
  to { transform: translateX(100%); opacity: 0; }
}
@keyframes slide-in-left {
  from { transform: translateX(-100%); opacity: 0; }
}
```

## Reduced Motion Support

Always respect the user's motion preferences. Users who prefer reduced motion should not see animated transitions:

```css
/* styles.css */

@media (prefers-reduced-motion: reduce) {
  ::view-transition-old(root),
  ::view-transition-new(root) {
    animation-duration: 0ms;
  }

  /* Or disable transitions entirely */
  ::view-transition-group(*),
  ::view-transition-old(*),
  ::view-transition-new(*) {
    animation: none !important;
  }
}
```

You can also skip transitions programmatically:

```typescript
provideRouter(
  routes,
  withViewTransitions({
    onViewTransitionCreated: (info) => {
      const prefersReducedMotion = window.matchMedia(
        '(prefers-reduced-motion: reduce)'
      ).matches;

      if (prefersReducedMotion) {
        info.transition.skipTransition();
      }
    }
  })
)
```

## Browser Support and Graceful Degradation

| Browser | Support |
|---|---|
| Chrome 111+ | Full support |
| Edge 111+ | Full support (Chromium) |
| Safari 18+ | Full support |
| Firefox 144+ | Full support |
| Older browsers | No support (graceful fallback) |

Angular handles this gracefully. If the browser does not support `document.startViewTransition`, the router skips the view transition and processes the navigation normally. No error is thrown, no fallback code is needed.

```typescript
// You do NOT need to write this check yourself
if ('startViewTransition' in document) {
  // Angular does this internally
}
```

## View Transitions vs Angular Animations

| Feature | View Transitions API | Angular Animations (@angular/animations) |
|---|---|---|
| Powered by | Browser-native CSS | Angular's animation DSL + Web Animations API |
| Setup | `withViewTransitions()` + CSS | `provideAnimations()` + `@trigger` in component |
| Bundle size | Zero (browser-native) | ~60KB for the animations package |
| Cross-element morphing | Built-in (via `view-transition-name`) | Not supported natively |
| Route transitions | First-class (page-level + element-level) | Requires manual setup with route data |
| Component animations | Not designed for this | First-class (enter/leave, state changes) |
| Browser support | Modern browsers only | All browsers Angular supports |
| Complexity | Low (CSS only) | High (complex DSL, trigger/state/transition) |

**Recommendation:** Use View Transitions for route animations (page-level transitions, element morphing between pages). Use Angular Animations for component-level animations (accordions, dropdown menus, toast notifications) that need to work everywhere.

## Common Mistakes

- **Defining view transition CSS in component styles.** Angular's view encapsulation prevents component-scoped styles from targeting `::view-transition-*` pseudo-elements. All view transition CSS must be in global styles (`styles.css`).

- **Duplicate `view-transition-name` values.** Every `view-transition-name` must be unique within the document at any given time. If two visible elements share the same name, the browser cannot determine which to animate and the transition breaks. Use dynamic names based on IDs.

- **Not handling the initial transition.** The first route activation triggers a view transition by default. Since there is no "old" state, this can produce an awkward fade-in from nothing. Use `skipInitialTransition: true`.

- **Ignoring `prefers-reduced-motion`.** Some users have vestibular disorders or motion sensitivity. Always provide a reduced-motion fallback that either shortens or disables the animation.

- **Expecting backwards navigation to animate differently by default.** The View Transitions API does not know about navigation direction. Without explicit direction tracking (via `onViewTransitionCreated` or `NavigationStart.navigationTrigger`), forward and back navigations use the same animation.

- **Applying `view-transition-name` to elements inside `*ngFor` / `@for` without unique names.** Each iterated element needs a unique name. Use the item's ID or index to generate it.

---

## Best Practices

- **Start with the default cross-fade.** Just calling `withViewTransitions()` with no options gives you a pleasant cross-fade for free. Add custom CSS only when the default is not sufficient.

- **Use `skipInitialTransition: true`.** Almost all applications look better without an animation on the initial page load.

- **Define animation CSS in `styles.css` (global styles).** This is a requirement, not a preference. Component-scoped styles cannot reach view transition pseudo-elements.

- **Name elements sparingly.** Only assign `view-transition-name` to elements that genuinely need individual transitions (hero images, headings). Every named element creates its own transition group, which adds overhead.

- **Provide `prefers-reduced-motion` overrides.** A single `@media` block in global styles is enough to disable or shorten all view transition animations.

- **Leverage `onViewTransitionCreated` for direction-aware animations.** Track forward vs. back navigation and apply a CSS data attribute to the document element. Use CSS selectors to vary the animation based on direction.

- **Use the `transition.finished` promise for cleanup.** If you add temporary CSS classes or data attributes during a transition, remove them in `transition.finished.finally()`.

## Real-World Use Cases

### 1. Product List to Detail Morph

The signature use case: a product card image on the list page smoothly morphs into the hero image on the detail page.

```typescript

@Component({

          <img
            [src]="product.thumbnail"

          </h3>
          <span class="price">{{ product.price | currency }}</span>

      }
    </div>
  `

export class ProductListComponent {
  products = input.required<Product[]>();

// product-detail.component.ts
@Component({
```

### 2. Dashboard with Contextual Page Transitions

Different sections of a dashboard use different transition styles:

```typescript
// Route config with transition metadata
const routes: Routes = [
  {
    path: 'dashboard',
    component: DashboardComponent,
    title: 'Dashboard',
    data: { transition: 'fade' }
  },
  {
    path: 'analytics',
    component: AnalyticsComponent,
    title: 'Analytics',
    data: { transition: 'slide-up' }
  },
  {
    path: 'settings',
    component: SettingsComponent,
    title: 'Settings',
    data: { transition: 'slide-left' }
  }
];

// app.config.ts
provideRouter(
  routes,
  withViewTransitions({
    skipInitialTransition: true,
    onViewTransitionCreated: (info) => {
      const transition = info.to.data['transition'] ?? 'fade';
      document.documentElement.dataset['transition'] = transition;

      info.transition.finished.finally(() => {
        delete document.documentElement.dataset['transition'];
      });
    }
  })
)
```

```css
/* styles.css */

/* Fade (default) */

}
[data-transition="fade"]::view-transition-new(root) {
  animation: fade-in 200ms ease-out both;
}

/* Slide up */
[data-transition="slide-up"]::view-transition-old(root) {
  animation: slide-down-out 300ms ease-in both;
}
[data-transition="slide-up"]::view-transition-new(root) {
  animation: slide-up-in 300ms ease-out both;

/* Slide left */
```

### 3. Persistent Header with Page Content Transition

Exclude the header from the page transition so it stays stable while only the content area animates:

```typescript
// app.component.ts
@Component({
  selector: 'app-root',
  template: `
    <header class="app-header">
      <nav>
        <a routerLink="/home">Home</a>
        <a routerLink="/about">About</a>
      </nav>
    </header>
    <main class="page-content">
      <router-outlet />
    </main>
  `,
  styles: [`
    /* These styles are in the component but they work because
       view-transition-name is applied to the element itself,
       not to the pseudo-elements */
  `]
})
export class AppComponent {}
```

```css
/* styles.css */

/* Give the header its own transition name so it is isolated */
.app-header {
  view-transition-name: header;
}

/* The header does not animate: it stays in place */
::view-transition-old(header),
::view-transition-new(header) {
  animation: none;
}

/* Give the content area its own name */
.page-content {
  view-transition-name: page-content;
}

/* Only the content area transitions */
::view-transition-old(page-content) {
  animation: fade-out 200ms ease-in both;
}
::view-transition-new(page-content) {
  animation: fade-in 200ms ease-out both;
}

/* Disable the root-level transition since we handle it per-section */
::view-transition-old(root),
::view-transition-new(root) {
  animation: none;
}
```

---

> **Key Takeaway:** Enable view transitions with a single `withViewTransitions()` call for instant cross-fade page transitions, then use `view-transition-name` on matched elements and CSS pseudo-element selectors (`::view-transition-old()`, `::view-transition-new()`) in global styles to create rich, element-level morphing animations between routes with zero JavaScript animation code.
