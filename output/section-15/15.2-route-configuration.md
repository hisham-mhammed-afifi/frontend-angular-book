# 15.2 Route Configuration

> The `Route` interface defines how URLs map to components, how data flows into routes, and how the application is split into lazy-loaded chunks. Mastering route configuration is essential for building scalable Angular applications.

---

## The Route Interface

Every route in Angular is an object conforming to the `Route` interface. The `Routes` type is simply an array of `Route` objects.

```typescript
import { Routes } from '@angular/router';

export const routes: Routes = [
  // Each object is a Route
];
```

### Key Properties Overview

| Property | Type | Purpose |
|----------|------|---------|
| `path` | `string` | URL segment to match |
| `pathMatch` | `'prefix' \| 'full'` | How to match the path |
| `component` | `Type<any>` | Eagerly loaded component |
| `loadComponent` | `() => Promise<Type<any>>` | Lazily loaded standalone component |
| `children` | `Routes` | Nested child routes |
| `loadChildren` | `() => Promise<Routes>` | Lazily loaded child routes |
| `redirectTo` | `string \| RedirectFunction` | Redirect target |
| `outlet` | `string` | Named outlet |
| `data` | `object` | Static route data |
| `resolve` | `object` | Dynamic data resolvers |
| `title` | `string \| ResolveFn<string>` | Page title |
| `providers` | `Provider[]` | Route-scoped DI providers |
| `canActivate` | `CanActivateFn[]` | Activation guards |
| `canActivateChild` | `CanActivateChildFn[]` | Child activation guards |
| `canDeactivate` | `CanDeactivateFn[]` | Deactivation guards |
| `canMatch` | `CanMatchFn[]` | Route matching guards |
| `runGuardsAndResolvers` | `string` | When to re-run guards/resolvers |
| `matcher` | `UrlMatcher` | Custom URL matching function |

## Path Matching

### Basic Paths

```typescript
const routes: Routes = [
  { path: '',          component: HomeComponent },        // Matches root URL
  { path: 'products',  component: ProductListComponent }, // Matches /products
  { path: 'about',     component: AboutComponent },       // Matches /about
];
```

### Path Parameters

Parameters are prefixed with `:`:

```typescript
const routes: Routes = [
  { path: 'products/:id',             component: ProductDetailComponent },
  { path: 'users/:userId/orders/:orderId', component: OrderDetailComponent },
];
// /products/42         -> id = '42'
// /users/5/orders/100  -> userId = '5', orderId = '100'
```

### Wildcard Route

The `**` path matches any URL that has not been matched by a previous route:

```typescript
const routes: Routes = [
  { path: '',        component: HomeComponent },
  { path: 'about',   component: AboutComponent },
  { path: '**',      component: NotFoundComponent }, // Must be last
];
```

### `pathMatch: 'prefix'` vs `pathMatch: 'full'`

```typescript
// pathMatch: 'prefix' (default)
// Matches if the path is a prefix of the remaining URL
{ path: 'products', component: ProductsComponent }
// Matches: /products, /products/42, /products/42/reviews

// pathMatch: 'full'
// Matches only if the ENTIRE remaining URL equals the path
{ path: 'products', component: ProductsComponent, pathMatch: 'full' }
// Matches: /products
// Does NOT match: /products/42
```

The critical rule: **always use `pathMatch: 'full'` on empty-path redirects**:

```typescript
// REQUIRED: pathMatch: 'full' on empty-path redirects
{ path: '', redirectTo: 'home', pathMatch: 'full' }

// Without pathMatch: 'full', this redirect matches EVERY URL
// because '' is a prefix of every string
```

## Component Loading Strategies

### Eager Loading with `component`

The component is included in the main bundle:

```typescript
import { HomeComponent } from './home/home.component';

const routes: Routes = [
  { path: '', component: HomeComponent },
];
```

Use for: the landing page and any route the user always visits first.

### Lazy Loading with `loadComponent`

The component is loaded on demand when the route is activated:

```typescript
const routes: Routes = [
  {
    path: 'settings',
    loadComponent: () =>
      import('./settings/settings.component').then(m => m.SettingsComponent),
  },
];
```

The dynamic `import()` creates a separate JavaScript chunk that is fetched only when the user navigates to `/settings`.

### Default Export Shorthand

If the component is the default export, the `.then()` is unnecessary:

```typescript
// settings.component.ts
@Component({ ... })
export default class SettingsComponent {}

// Route
{
  path: 'settings',
  loadComponent: () => import('./settings/settings.component'),
}
```

### Lazy Loading Child Routes with `loadChildren`

Load an entire set of child routes lazily:

```typescript
const routes: Routes = [
  {
    path: 'admin',
    loadChildren: () =>
      import('./admin/admin.routes').then(m => m.ADMIN_ROUTES),
  },
];
```

```typescript
// admin/admin.routes.ts
import { Routes } from '@angular/router';

export const ADMIN_ROUTES: Routes = [
  {
    path: '',
    loadComponent: () => import('./admin-layout.component'),
    children: [
      { path: '',        loadComponent: () => import('./dashboard.component') },
      { path: 'users',   loadComponent: () => import('./users.component') },
      { path: 'settings', loadComponent: () => import('./admin-settings.component') },
    ],
  },
];
```

## Child Routes and Nesting

Child routes render inside the parent component's `<router-outlet>`.

### Inline Children

```typescript
const routes: Routes = [
  {
    path: 'account',
    component: AccountLayoutComponent,
    children: [
      { path: '',         component: AccountOverviewComponent },
      { path: 'profile',  component: ProfileComponent },
      { path: 'security', component: SecurityComponent },
      { path: 'billing',  component: BillingComponent },
    ],
  },
];
```

```typescript
// account-layout.component.ts
@Component({
  imports: [RouterOutlet, RouterLink, RouterLinkActive],
  template: `
    <nav>
      <a routerLink="./" routerLinkActive="active"
         [routerLinkActiveOptions]="{ exact: true }">Overview</a>
      <a routerLink="profile" routerLinkActive="active">Profile</a>
      <a routerLink="security" routerLinkActive="active">Security</a>
      <a routerLink="billing" routerLinkActive="active">Billing</a>
    </nav>
    <section>
      <router-outlet />
    </section>
  `,
})
export class AccountLayoutComponent {}
```

URLs produced: `/account`, `/account/profile`, `/account/security`, `/account/billing`.

### Deeply Nested Routes

```typescript
const routes: Routes = [
  {
    path: 'shop',
    component: ShopLayoutComponent,
    children: [
      { path: '', component: ShopHomeComponent },
      {
        path: 'categories/:categoryId',
        component: CategoryComponent,
        children: [
          { path: '', component: CategoryProductsComponent },
          { path: 'products/:productId', component: ProductDetailComponent },
        ],
      },
    ],
  },
];
// /shop
// /shop/categories/electronics
// /shop/categories/electronics/products/42
```

Each level requires its own `<router-outlet>` in the parent component.

## Componentless Routes

A route without a `component` or `loadComponent` property is a componentless route. It does not render anything but serves as a grouping mechanism for child routes.

### Grouping Routes with Shared Configuration

```typescript
const routes: Routes = [
  {
    path: 'admin',
    // No component -- this route is just a grouping node
    canActivate: [isAdminGuard],
    providers: [AdminService],
    children: [
      { path: '', loadComponent: () => import('./admin/dashboard.component') },
      { path: 'users', loadComponent: () => import('./admin/users.component') },
      { path: 'logs', loadComponent: () => import('./admin/logs.component') },
    ],
  },
];
```

The `canActivate` guard and `AdminService` provider apply to all children without requiring a wrapper component.

### Empty-Path Componentless Routes for Layout Grouping

```typescript
const routes: Routes = [
  {
    path: '',
    canActivate: [authGuard],
    children: [
      { path: 'dashboard', component: DashboardComponent },
      { path: 'profile', component: ProfileComponent },
      { path: 'settings', component: SettingsComponent },
    ],
  },
  { path: 'login', component: LoginComponent },
  { path: 'register', component: RegisterComponent },
];
```

All routes under the empty-path parent require authentication. `login` and `register` do not.

### Parameter Merging

Componentless routes merge their parameters, data, and resolve results into their children:

```typescript
const routes: Routes = [
  {
    path: 'team/:teamId',
    // No component
    resolve: { team: teamResolver },
    children: [
      {
        path: 'members',
        component: MembersComponent,
        // MembersComponent has access to both :teamId and the team resolver result
      },
    ],
  },
];
```

## Redirects

### Static Redirects

```typescript
const routes: Routes = [
  { path: '', redirectTo: 'home', pathMatch: 'full' },
  { path: 'home', component: HomeComponent },

  // Redirect old URL to new URL
  { path: 'legacy-dashboard', redirectTo: 'dashboard', pathMatch: 'full' },
];
```

### Redirect Types

```typescript
// Relative redirect (appends to the current path level)
{ path: 'old', redirectTo: 'new', pathMatch: 'full' }
// /old -> /new

// Absolute redirect (starts from root)
{ path: 'old', redirectTo: '/dashboard/new', pathMatch: 'full' }
// /old -> /dashboard/new
```

### `RedirectFunction` (Angular 18+)

The `redirectTo` property can accept a function for dynamic redirects:

```typescript
import { RedirectFunction } from '@angular/router';

const routes: Routes = [
  {
    path: 'go/:target',
    redirectTo: ({ params }) => {
      // Dynamic redirect based on the route parameter
      switch (params['target']) {
        case 'docs':    return '/documentation';
        case 'support': return '/help-center';
        default:        return '/home';
      }
    },
  },
];
```

### `RedirectFunction` with Dependency Injection

`RedirectFunction` runs in an injection context, so you can use `inject()`:

```typescript
const routes: Routes = [
  {
    path: '',
    pathMatch: 'full',
    redirectTo: () => {
      const auth = inject(AuthService);
      return auth.isAuthenticated() ? '/dashboard' : '/login';
    },
  },
];
```

### `RedirectFunction` Limitations

The function receives a partial `ActivatedRouteSnapshot` because route matching is not yet complete. You get access to:
- `params` (path parameters matched so far)
- `queryParams`
- `fragment`

You do NOT have access to:
- `data`
- `resolve` results
- `component`

## Route Data

### Static Data

```typescript
const routes: Routes = [
  {
    path: 'products',
    component: ProductListComponent,
    data: {
      breadcrumb: 'Products',
      pageTitle: 'Product Catalog',
      requiresAuth: false,
    },
  },
];

// Accessing in a component (with withComponentInputBinding)
@Component({ ... })
export class ProductListComponent {
  breadcrumb = input<string>();  // Bound automatically
  pageTitle = input<string>();
}
```

### Route Title

The `title` property sets the browser tab title automatically:

```typescript
const routes: Routes = [
  {
    path: 'products',
    component: ProductListComponent,
    title: 'Products - MyApp',
  },
  {
    path: 'products/:id',
    component: ProductDetailComponent,
    title: productTitleResolver,  // Dynamic title
  },
];

// Dynamic title resolver
const productTitleResolver: ResolveFn<string> = (route) => {
  const productService = inject(ProductService);
  const id = route.paramMap.get('id')!;
  return productService.getProduct(id).pipe(
    map(product => `${product.name} - MyApp`)
  );
};
```

Angular automatically updates `document.title` when navigation completes.

### Custom Title Strategy

For app-wide title formatting:

```typescript
import { TitleStrategy } from '@angular/router';
import { Injectable } from '@angular/core';
import { Title } from '@angular/platform-browser';

@Injectable({ providedIn: 'root' })
export class AppTitleStrategy extends TitleStrategy {
  private title = inject(Title);

  override updateTitle(snapshot: RouterStateSnapshot) {
    const pageTitle = this.buildTitle(snapshot);
    this.title.setTitle(pageTitle ? `${pageTitle} | MyApp` : 'MyApp');
  }
}

// Register in app.config.ts
providers: [
  { provide: TitleStrategy, useClass: AppTitleStrategy },
]
```

## Route-Level Providers

Routes can define their own DI providers, creating a scoped `EnvironmentInjector`:

```typescript
const routes: Routes = [
  {
    path: 'editor',
    loadComponent: () => import('./editor/editor.component'),
    providers: [
      EditorStateService,
      UndoRedoService,
      { provide: SAVE_INTERVAL, useValue: 30000 },
    ],
  },
];
```

These services are:
- Created when the user navigates to `/editor`.
- Shared among the component and all its children.
- Destroyed when the user navigates away.

### Combining with `makeEnvironmentProviders`

```typescript
import { makeEnvironmentProviders } from '@angular/core';

function provideEditor() {
  return makeEnvironmentProviders([
    EditorStateService,
    UndoRedoService,
    AutoSaveService,
  ]);
}

const routes: Routes = [
  {
    path: 'editor',
    loadComponent: () => import('./editor/editor.component'),
    providers: [provideEditor()],
  },
];
```

## `canMatch` Guard

`canMatch` determines whether a route should even be considered during URL matching. If it returns `false`, the router skips the route entirely and tries the next one.

### Use Case: Feature Flags

```typescript
const isFeatureEnabled: CanMatchFn = (route) => {
  const features = inject(FeatureFlagService);
  const flag = route.data?.['featureFlag'] as string;
  return features.isEnabled(flag);
};

const routes: Routes = [
  {
    path: 'checkout',
    loadComponent: () => import('./checkout-v2/checkout.component'),
    canMatch: [isFeatureEnabled],
    data: { featureFlag: 'newCheckout' },
  },
  {
    path: 'checkout',
    loadComponent: () => import('./checkout-v1/checkout.component'),
    // Fallback: old checkout if feature flag is off
  },
];
```

### Use Case: Role-Based Route Selection

```typescript
const isAdmin: CanMatchFn = () => inject(AuthService).isAdmin();

const routes: Routes = [
  {
    path: 'dashboard',
    loadComponent: () => import('./admin-dashboard.component'),
    canMatch: [isAdmin],
  },
  {
    path: 'dashboard',
    loadComponent: () => import('./user-dashboard.component'),
    // Regular users get this component
  },
];
```

Same URL, different components based on the user's role.

## `runGuardsAndResolvers`

Controls when guards and resolvers re-execute during navigation:

```typescript
const routes: Routes = [
  {
    path: 'search',
    component: SearchComponent,
    resolve: { results: searchResolver },
    runGuardsAndResolvers: 'paramsOrQueryParamsChange',
  },
];
```

| Value | Re-runs When |
|-------|-------------|
| `'paramsChange'` (default) | Path parameters change |
| `'pathParamsChange'` | Only path params change (not matrix params) |
| `'paramsOrQueryParamsChange'` | Path params OR query params change |
| `'pathParamsOrQueryParamsChange'` | Path params OR query params change (excludes matrix) |
| `'always'` | Every navigation, even if nothing changes |

### When to Change the Default

The default `'paramsChange'` does NOT re-run resolvers when only query parameters change. For search pages or filtered lists where query params drive the data:

```typescript
// /search?q=angular -> /search?q=react
// Default: resolver does NOT re-run
// paramsOrQueryParamsChange: resolver re-runs

{
  path: 'search',
  component: SearchComponent,
  resolve: { results: searchResolver },
  runGuardsAndResolvers: 'paramsOrQueryParamsChange',
}
```

## Custom URL Matcher

For routes that cannot be expressed with standard path syntax, use a `UrlMatcher`:

```typescript
import { UrlSegment, UrlMatcher, Route, UrlSegmentGroup } from '@angular/router';

// Match URLs like /product-42, /product-abc
const productMatcher: UrlMatcher = (
  segments: UrlSegment[],
): { consumed: UrlSegment[]; posParams: Record<string, UrlSegment> } | null => {
  if (segments.length === 1) {
    const match = segments[0].path.match(/^product-(.+)$/);
    if (match) {
      return {
        consumed: segments,
        posParams: {
          id: new UrlSegment(match[1], {}),
        },
      };
    }
  }
  return null;
};

const routes: Routes = [
  {
    matcher: productMatcher,
    component: ProductComponent,
    // ProductComponent receives 'id' as a route param
  },
];
```

### Locale-Based Matcher

```typescript
const localeMatcher: UrlMatcher = (segments) => {
  if (segments.length >= 1 && /^(en|fr|de|es|ja)$/.test(segments[0].path)) {
    return {
      consumed: [segments[0]],
      posParams: {
        locale: segments[0],
      },
    };
  }
  return null;
};

const routes: Routes = [
  {
    matcher: localeMatcher,
    component: LocaleLayoutComponent,
    children: [
      { path: '', component: HomeComponent },
      { path: 'about', component: AboutComponent },
    ],
  },
  // Fallback for URLs without a locale prefix
  { path: '', redirectTo: '/en', pathMatch: 'full' },
];
```

## Route Ordering

Angular matches routes in the order they are defined. The first match wins.

```typescript
const routes: Routes = [
  // Specific routes first
  { path: 'products/new', component: NewProductComponent },
  { path: 'products/:id', component: ProductDetailComponent },
  { path: 'products',     component: ProductListComponent },

  // Generic/catch-all routes last
  { path: '**', component: NotFoundComponent },
];
```

### Order Rules

1. **Exact paths before parameterized paths**: `products/new` before `products/:id`.
2. **More specific before less specific**: `products/:id/reviews` before `products/:id`.
3. **Redirects before components** (at the same path level).
4. **Wildcard route always last**.

```typescript
// BAD: :id matches 'new' before the literal route
const routes: Routes = [
  { path: 'products/:id',  component: ProductDetailComponent },
  { path: 'products/new',  component: NewProductComponent },  // Never reached!
];

// GOOD: Literal paths first
const routes: Routes = [
  { path: 'products/new',  component: NewProductComponent },
  { path: 'products/:id',  component: ProductDetailComponent },
];
```

## Organizing Routes for Large Applications

### Feature Route Files

```
src/
  app/
    app.routes.ts              # Root routes
    products/
      product.routes.ts        # Product feature routes
    admin/
      admin.routes.ts          # Admin feature routes
    account/
      account.routes.ts        # Account feature routes
```

```typescript
// app.routes.ts
export const routes: Routes = [
  { path: '', loadComponent: () => import('./home/home.component') },
  {
    path: 'products',
    loadChildren: () => import('./products/product.routes').then(m => m.PRODUCT_ROUTES),
  },
  {
    path: 'admin',
    loadChildren: () => import('./admin/admin.routes').then(m => m.ADMIN_ROUTES),
  },
  {
    path: 'account',
    loadChildren: () => import('./account/account.routes').then(m => m.ACCOUNT_ROUTES),
  },
  { path: '**', loadComponent: () => import('./not-found/not-found.component') },
];
```

```typescript
// products/product.routes.ts
export const PRODUCT_ROUTES: Routes = [
  { path: '',    loadComponent: () => import('./product-list.component') },
  { path: 'new', loadComponent: () => import('./product-form.component') },
  { path: ':id', loadComponent: () => import('./product-detail.component') },
  { path: ':id/edit', loadComponent: () => import('./product-form.component') },
];
```

### Layout Routes Pattern

Different sections of the app can have different layouts:

```typescript
export const routes: Routes = [
  // Public layout (no sidebar, no auth required)
  {
    path: '',
    loadComponent: () => import('./layouts/public-layout.component'),
    children: [
      { path: '', loadComponent: () => import('./home/home.component') },
      { path: 'login', loadComponent: () => import('./auth/login.component') },
      { path: 'register', loadComponent: () => import('./auth/register.component') },
    ],
  },
  // Authenticated layout (sidebar, top nav, auth required)
  {
    path: '',
    loadComponent: () => import('./layouts/app-layout.component'),

    children: [
      { path: 'dashboard', loadComponent: () => import('./dashboard/dashboard.component') },
      { path: 'settings', loadComponent: () => import('./settings/settings.component') },
      {
        path: 'admin',
        loadChildren: () => import('./admin/admin.routes').then(m => m.ADMIN_ROUTES),
        canMatch: [isAdminGuard],
      },
    ],
  },
  { path: '**', loadComponent: () => import('./not-found/not-found.component') },
];
```

## Common Mistakes

### Mistake 1: Missing `pathMatch: 'full'` on Empty-Path Redirects

```typescript
// BAD: Matches every URL, causes infinite redirect loop
{ path: '', redirectTo: 'home' }

// GOOD: Only matches the exact root URL
{ path: '', redirectTo: 'home', pathMatch: 'full' }
```

### Mistake 2: Putting `pathMatch: 'full'` on Parent Routes with Children

```typescript
// BAD: No child route will ever match
{
  path: 'admin',
  component: AdminLayoutComponent,
  pathMatch: 'full',  // Requires the ENTIRE URL to be 'admin'
  children: [
    { path: 'users', component: UsersComponent },  // Never reachable!
  ],
}

// GOOD: Remove pathMatch, let children handle the rest
{
  path: 'admin',
  component: AdminLayoutComponent,
  children: [
    { path: 'users', component: UsersComponent },
  ],
}
```

### Mistake 3: Wrong Route Order

```typescript
// BAD: Wildcard catches everything before specific routes
const routes: Routes = [
  { path: '**', component: NotFoundComponent },
  { path: 'home', component: HomeComponent },     // Never reached!
];

// GOOD: Wildcard last
const routes: Routes = [
  { path: 'home', component: HomeComponent },
  { path: '**', component: NotFoundComponent },
];
```

### Mistake 4: Forgetting `<router-outlet>` in Parent Components

```typescript
// Route expects children to render, but parent has no outlet
{
  path: 'account',
  component: AccountComponent,
  children: [
    { path: 'profile', component: ProfileComponent },
  ],
}

// AccountComponent MUST contain <router-outlet /> for children to render
@Component({
  imports: [RouterOutlet],
  template: `
    <h1>Account</h1>
    <router-outlet />  <!-- Required! -->
  `,
})
export class AccountComponent {}
```

### Mistake 5: Lazy Loading Without Code Splitting

```typescript
// BAD: Import at the top means no code splitting
import { SettingsComponent } from './settings/settings.component';

const routes: Routes = [
  { path: 'settings', component: SettingsComponent }, // Already in the main bundle!
];

// GOOD: Dynamic import creates a separate chunk
const routes: Routes = [
  {
    path: 'settings',
    loadComponent: () =>
      import('./settings/settings.component').then(m => m.SettingsComponent),
  },
];
```

### Mistake 6: Not Handling the Default Child Route

```typescript
// BAD: Navigating to /account shows a blank router-outlet
{
  path: 'account',
  component: AccountLayoutComponent,
  children: [
    { path: 'profile', component: ProfileComponent },
    { path: 'settings', component: SettingsComponent },
    // No empty-path child -- /account renders nothing in the outlet
  ],
}

// GOOD: Provide a default child or redirect
{
  path: 'account',
  component: AccountLayoutComponent,
  children: [
    { path: '', redirectTo: 'profile', pathMatch: 'full' },
    { path: 'profile', component: ProfileComponent },
    { path: 'settings', component: SettingsComponent },
  ],
}
```

## Best Practices

### 1. Lazy Load Everything Except the Landing Page

Keep the initial bundle small. Eagerly load only the component the user sees first.

### 2. Use `loadComponent` Over `loadChildren` for Single Components

If a route maps to a single component with no children, `loadComponent` is simpler than creating a separate routes file for `loadChildren`.

### 3. Use Componentless Routes for Shared Guards and Providers

Group routes under a componentless parent to apply guards, resolvers, or providers once.

### 4. Set a Dynamic Title on Every Route

Users and search engines rely on `document.title`. Use static `title` strings or resolver functions to set meaningful titles.

### 5. Keep Route Files Flat

Avoid nesting route files more than two levels deep. If you have `admin.routes.ts` loading `admin-users.routes.ts` loading `admin-user-detail.routes.ts`, it is too deep. Flatten the hierarchy.

### 6. Use `canMatch` Instead of Complex Guard Logic

When the same URL should render different components based on conditions (roles, feature flags), `canMatch` is cleaner than conditional logic inside guards.

## Real-World Use Cases

### E-Commerce Application Routes

```typescript
export const routes: Routes = [
  { path: '', redirectTo: 'home', pathMatch: 'full' },
  {
    path: 'home',
    loadComponent: () => import('./home/home.component'),
    title: 'Home',
  },
  {
    path: 'products',
    loadChildren: () => import('./products/product.routes').then(m => m.PRODUCT_ROUTES),
  },
  {
    path: 'cart',
    loadComponent: () => import('./cart/cart.component'),
    title: 'Shopping Cart',
  },
  {
    path: 'checkout',
    loadComponent: () => import('./checkout/checkout.component'),
    canActivate: [authGuard],
    title: 'Checkout',
  },
  {
    path: 'orders',
    loadChildren: () => import('./orders/order.routes').then(m => m.ORDER_ROUTES),
    canActivate: [authGuard],
  },
  {
    path: 'auth',
    children: [
      { path: 'login', loadComponent: () => import('./auth/login.component'), title: 'Sign In' },
      { path: 'register', loadComponent: () => import('./auth/register.component'), title: 'Sign Up' },
      { path: 'forgot-password', loadComponent: () => import('./auth/forgot-password.component') },
    ],
  },
  { path: '**', loadComponent: () => import('./not-found/not-found.component'), title: 'Page Not Found' },
];
```

### SaaS Dashboard with Role-Based Routing

```typescript
const isAdmin: CanMatchFn = () => inject(AuthService).hasRole('admin');
const isManager: CanMatchFn = () => inject(AuthService).hasRole('manager');

export const routes: Routes = [
  {
    path: '',
    loadComponent: () => import('./layouts/dashboard-layout.component'),
    canActivate: [authGuard],
    children: [
      // Admin dashboard
      {
        path: 'dashboard',
        loadComponent: () => import('./dashboards/admin-dashboard.component'),
        canMatch: [isAdmin],
        title: 'Admin Dashboard',
      },
      // Manager dashboard
      {
        path: 'dashboard',
        loadComponent: () => import('./dashboards/manager-dashboard.component'),
        canMatch: [isManager],
        title: 'Manager Dashboard',
      },
      // Default dashboard for everyone else
      {
        path: 'dashboard',
        loadComponent: () => import('./dashboards/user-dashboard.component'),
        title: 'Dashboard',
      },
      {
        path: 'reports',
        loadChildren: () => import('./reports/report.routes').then(m => m.REPORT_ROUTES),
      },
      {
        path: 'settings',
        loadComponent: () => import('./settings/settings.component'),
        title: 'Settings',
        providers: [SettingsStateService],
      },
    ],
  },
  {
    path: 'login',
    loadComponent: () => import('./auth/login.component'),
    title: 'Sign In',
  },
  { path: '**', redirectTo: 'dashboard' },
];
```

---

> **Key Takeaway:** Route configuration determines your app's URL structure, code splitting strategy, and navigation behavior. Use `loadComponent` for lazy loading, componentless routes for shared guards/providers, `canMatch` for conditional route selection, and always put specific routes before parameterized ones and wildcard routes last.
