# 16.1 Template-Driven Forms

> Template-driven forms use directives (`ngModel`, `NgForm`, `NgModelGroup`) in the template to create and manage form controls implicitly, making them the simplest way to build forms in Angular for straightforward use cases.

---

## Setup

Template-driven forms require `FormsModule`. Import it in the component's `imports` array:

```typescript
import { Component } from '@angular/core';
import { FormsModule } from '@angular/forms';

@Component({
  selector: 'app-contact-form',
  standalone: true,
  imports: [FormsModule],
  templateUrl: './contact-form.component.html'
})
export class ContactFormComponent {
  model = {
    name: '',
    email: '',
    message: ''
  };
}
```

When `FormsModule` is imported, Angular automatically attaches an `NgForm` directive to every `<form>` element in that component's template. You do not need to add anything to the `<form>` tag.

---

## ngModel: The Core Directive

`NgModel` creates and manages a `FormControl` instance for a form element. It comes in three binding flavors:

### One-Way Binding (Template to Model)

```html
<!-- Just registers the control with the form; no initial value -->
<input name="username" ngModel />
```

The control is registered under the key `"username"` in the form's value object. The value starts as an empty string.

### One-Way Binding (Model to Template)

```html
<!-- Sets the initial value from the component; does NOT write back -->
<input name="username" [ngModel]="model.username" />
```

The input displays the value of `model.username`, but typing in the input does not update `model.username`. Changes only flow from the component to the template.

### Two-Way Binding

```html
<!-- Full two-way sync between input and component property -->
<input name="username" [(ngModel)]="model.username" />
```

Changes in the input update `model.username`, and changes to `model.username` in the component update the input. This is the most common pattern.

### The name Attribute is Required

Every `ngModel` control must have a `name` attribute (or configure `standalone` mode). Without `name`, Angular throws an error because it cannot register the control in the form group.

```html
<!-- BAD: Missing name attribute -->
<input [(ngModel)]="model.email" />
<!-- Error: ngModel on form element requires name attribute or standalone -->

<!-- GOOD -->
<input name="email" [(ngModel)]="model.email" />
```

### Standalone ngModel

If you want `ngModel` on an element that is not part of a form (or should not be registered with the parent `NgForm`), use `ngModelOptions` with `standalone: true`:

```html
<input [(ngModel)]="searchQuery" [ngModelOptions]="{ standalone: true }" />
```

This control has its own `FormControl` but does not contribute to the form's value or validity.

## NgForm: The Form Directive

`NgForm` is automatically attached to `<form>` elements when `FormsModule` is imported. It creates a top-level `FormGroup` that aggregates all `ngModel` controls within the form.

### Template Reference Variable

Access the `NgForm` instance with a template reference variable:

```html
<form #contactForm="ngForm" (ngSubmit)="onSubmit(contactForm)">
  <input name="name" [(ngModel)]="model.name" required />
  <input name="email" [(ngModel)]="model.email" required email />
  <textarea name="message" [(ngModel)]="model.message" required></textarea>

  <button type="submit" [disabled]="contactForm.invalid">Send</button>
</form>
```

### NgForm Properties

| Property | Type | Description |
|---|---|---|
| `form` | `FormGroup` | The underlying `FormGroup` instance |
| `value` | `object` | Current form value (`{ name: '...', email: '...', message: '...' }`) |
| `valid` | `boolean` | `true` if all controls pass validation |
| `invalid` | `boolean` | `true` if any control fails validation |
| `pristine` | `boolean` | `true` if no control has been modified |
| `dirty` | `boolean` | `true` if any control has been modified |
| `touched` | `boolean` | `true` if any control has been focused and blurred |
| `untouched` | `boolean` | `true` if no control has been focused and blurred |
| `submitted` | `boolean` | `true` after the form has been submitted |
| `controls` | `object` | Map of child controls |

### Accessing the Form in the Component

Use `@ViewChild` to access the form programmatically:

```typescript
import { Component, ViewChild } from '@angular/core';
import { NgForm } from '@angular/forms';

@Component({ /* ... */ })
export class ContactFormComponent {
  @ViewChild('contactForm') form!: NgForm;

  model = { name: '', email: '', message: '' };

  onSubmit(form: NgForm): void {
    if (form.valid) {
      console.log('Form value:', form.value);
      // { name: 'Alice', email: 'alice@example.com', message: 'Hello' }
    }
  }

  resetForm(): void {
    this.form.resetForm();
    // Resets all controls to pristine/untouched and clears values
  }
}
```

### resetForm() vs reset()

- `resetForm()`: Resets the form AND marks it as not submitted. Also resets all controls to pristine and untouched.
- `form.reset()`: Resets the underlying `FormGroup` but does NOT reset the submitted state.

Always prefer `resetForm()` in template-driven forms.

## Template Reference Variables for Controls

Access individual control state by exporting `ngModel`:

```html
<input
  name="email"
  [(ngModel)]="model.email"
  required
  email
  #emailCtrl="ngModel"
/>

@if (emailCtrl.invalid && emailCtrl.touched) {
  <div class="error">
    @if (emailCtrl.errors?.['required']) {
      Email is required.
    }
    @if (emailCtrl.errors?.['email']) {
      Enter a valid email address.
    }
  </div>
}
```

The `#emailCtrl="ngModel"` syntax creates a template reference variable that points to the `NgModel` directive instance. This gives you access to all control state properties: `valid`, `invalid`, `pristine`, `dirty`, `touched`, `untouched`, `errors`, `value`, `status`.

## CSS State Classes

Angular automatically adds CSS classes to form elements based on their state:

| State | Class when true | Class when false |
|---|---|---|
| Control visited (blurred) | `ng-touched` | `ng-untouched` |
| Control value changed | `ng-dirty` | `ng-pristine` |
| Control value valid | `ng-valid` | `ng-invalid` |
| Form submitted | `ng-submitted` | (none) |

Use these for visual feedback without any extra logic:

```css
/* Show a red border on invalid fields after user interaction */
input.ng-invalid.ng-touched {
  border: 2px solid #dc3545;
}

/* Show a green border on valid fields after user interaction */
input.ng-valid.ng-dirty {
  border: 2px solid #28a745;
}

/* Red border on all invalid fields after form submission */
form.ng-submitted input.ng-invalid {
  border: 2px solid #dc3545;
}
```

## Built-In Validators

Angular provides validator directives that mirror HTML5 validation attributes:

| Directive | HTML Attribute | Validates |
|---|---|---|
| `RequiredValidator` | `required` | Value is not empty |
| `MinLengthValidator` | `minlength="n"` | Value has at least `n` characters |
| `MaxLengthValidator` | `maxlength="n"` | Value has at most `n` characters |
| `PatternValidator` | `pattern="regex"` | Value matches the regex pattern |
| `EmailValidator` | `email` | Value is a valid email format |
| `MinValidator` | `min="n"` | Numeric value >= n |
| `MaxValidator` | `max="n"` | Numeric value <= n |

```html
<input
  name="username"
  [(ngModel)]="model.username"
  required
  minlength="3"
  maxlength="20"
  pattern="^[a-zA-Z0-9_]+$"
  #usernameCtrl="ngModel"
/>

@if (usernameCtrl.invalid && usernameCtrl.touched) {
  <div class="errors">
    @if (usernameCtrl.errors?.['required']) {
      <p>Username is required.</p>
    }
    @if (usernameCtrl.errors?.['minlength']) {
      <p>
        Minimum length is {{ usernameCtrl.errors?.['minlength'].requiredLength }}.
        You entered {{ usernameCtrl.errors?.['minlength'].actualLength }} characters.
      </p>
    }
    @if (usernameCtrl.errors?.['pattern']) {
      <p>Only letters, numbers, and underscores are allowed.</p>
    }
  </div>
}
```

### Dynamic Validators

Bind validator attributes to component properties to enable/disable them dynamically:

```html
<input
  name="phone"
  [(ngModel)]="model.phone"
  [required]="model.contactMethod === 'phone'"
  pattern="^\+?[0-9]{10,15}$"
/>
```

When `model.contactMethod` changes, the `required` validator is toggled automatically.

## ngModelGroup: Grouping Controls

`NgModelGroup` creates a sub-`FormGroup` within the form. It does not require a visible DOM element:

```html
<form #registrationForm="ngForm">
  <h3>Personal Information</h3>
  <ng-container ngModelGroup="personal">
    <input name="firstName" [(ngModel)]="model.firstName" required />
    <input name="lastName" [(ngModel)]="model.lastName" required />
  </ng-container>

  <h3>Address</h3>
  <ng-container ngModelGroup="address" #addressGroup="ngModelGroup">
    <input name="street" [(ngModel)]="model.street" required />
    <input name="city" [(ngModel)]="model.city" required />
    <input name="zip" [(ngModel)]="model.zip" required pattern="^\d{5}$" />
  </ng-container>

  @if (addressGroup.invalid && addressGroup.touched) {
    <p class="error">Please complete your address.</p>
  }

  <button type="submit" [disabled]="registrationForm.invalid">Register</button>
</form>
```

The form value becomes a nested object:

```json
{
  "personal": {
    "firstName": "Alice",
    "lastName": "Smith"
  },
  "address": {
    "street": "123 Main St",
    "city": "Springfield",
    "zip": "62701"
  }
}
```

`ngModelGroup` also supports validation: if any control inside the group is invalid, the group itself is invalid. You can access the group's validity through a template reference variable (`#addressGroup="ngModelGroup"`).

### Resetting a Group

```typescript
// Reset only the address fields
this.form.form.get('address')?.reset();
```

## ngModelOptions

Configure how `ngModel` behaves with the `ngModelOptions` input:

### updateOn

Controls when the model is updated and validation runs:

```html
<!-- Default: update on every input event -->
<input name="search" [(ngModel)]="query" />

<!-- Update only on blur (when the user leaves the field) -->
<input name="email" [(ngModel)]="model.email"
       [ngModelOptions]="{ updateOn: 'blur' }" />

<!-- Update only on form submit -->
<input name="name" [(ngModel)]="model.name"
       [ngModelOptions]="{ updateOn: 'submit' }" />
```

`updateOn: 'blur'` is particularly useful for fields with async validators (e.g., checking email uniqueness) to avoid firing HTTP requests on every keystroke.

### Setting updateOn at the Form Level

```html
<form [ngFormOptions]="{ updateOn: 'blur' }">
  <!-- All controls in this form update on blur by default -->
  <input name="firstName" [(ngModel)]="model.firstName" />
  <input name="lastName" [(ngModel)]="model.lastName" />

  <!-- Override: this one updates on change (default) -->
  <input name="search" [(ngModel)]="model.search"
         [ngModelOptions]="{ updateOn: 'change' }" />
</form>
```

### standalone

Exclude a control from the parent form:

```html
<form #myForm="ngForm">
  <input name="included" [(ngModel)]="model.included" />

  <!-- This control is NOT part of myForm -->
  <input [(ngModel)]="filterText" [ngModelOptions]="{ standalone: true }" />
</form>
```

## Form Submission

### Basic Submission

```html
<form #myForm="ngForm" (ngSubmit)="onSubmit(myForm)">
  <input name="name" [(ngModel)]="model.name" required />
  <button type="submit">Submit</button>
</form>
```

```typescript
onSubmit(form: NgForm): void {
  if (form.invalid) {
    // Mark all controls as touched to show validation errors
    form.form.markAllAsTouched();
    return;
  }

  // Process the form
  this.saveData(form.value);

  // Reset after successful submission
  form.resetForm();
}
```

### Showing Errors on Submit

A common pattern is to show validation errors only after the user attempts to submit. Use the `submitted` state:

```html
<form #myForm="ngForm" (ngSubmit)="onSubmit(myForm)">
  <input name="email" [(ngModel)]="model.email" required email #emailCtrl="ngModel" />

  @if (emailCtrl.invalid && (emailCtrl.touched || myForm.submitted)) {
    <div class="error">Please enter a valid email.</div>
  }

  <button type="submit">Submit</button>
</form>
```

The error shows when the field is invalid AND either the user has touched the field OR the form has been submitted.

## Custom Validators as Directives

For template-driven forms, custom validators are implemented as directives that implement the `Validator` interface:

```typescript
import { Directive, input } from '@angular/core';
import { Validator, AbstractControl, ValidationErrors, NG_VALIDATORS } from '@angular/forms';

@Directive({
  selector: '[appForbiddenName]',
  standalone: true,
  providers: [
    {
      provide: NG_VALIDATORS,
      useExisting: ForbiddenNameDirective,
      multi: true
    }
  ]
})
export class ForbiddenNameDirective implements Validator {
  appForbiddenName = input<string>('');

  validate(control: AbstractControl): ValidationErrors | null {
    const forbidden = this.appForbiddenName();
    if (!forbidden) return null;

    const nameRe = new RegExp(forbidden);
    const isForbidden = nameRe.test(control.value);
    return isForbidden ? { forbiddenName: { value: control.value } } : null;
  }
}
```

Usage in template:

```html
<input
  name="heroName"
  [(ngModel)]="model.heroName"
  [appForbiddenName]="'admin'"
  #heroNameCtrl="ngModel"
/>

@if (heroNameCtrl.errors?.['forbiddenName']) {
  <p class="error">
    "{{ heroNameCtrl.errors?.['forbiddenName'].value }}" is not allowed.
  </p>
}
```

### Reusable ValidatorFn

Extract the validation logic into a pure function so it can be reused in both template-driven and reactive forms:

```typescript
import { ValidatorFn, AbstractControl, ValidationErrors } from '@angular/forms';

// Pure validator function (reusable in reactive forms)
export function forbiddenNameValidator(nameRe: RegExp): ValidatorFn {
  return (control: AbstractControl): ValidationErrors | null => {
    const forbidden = nameRe.test(control.value);
    return forbidden ? { forbiddenName: { value: control.value } } : null;
  };
}

// Directive wrapper for template-driven forms
@Directive({
  selector: '[appForbiddenName]',
  standalone: true,
  providers: [
    { provide: NG_VALIDATORS, useExisting: ForbiddenNameDirective, multi: true }
  ]
})
export class ForbiddenNameDirective implements Validator {
  appForbiddenName = input<string>('');

  validate(control: AbstractControl): ValidationErrors | null {
    const pattern = this.appForbiddenName();
    if (!pattern) return null;
    return forbiddenNameValidator(new RegExp(pattern))(control);
  }
}
```

## Async Validators

Async validators return an `Observable` or `Promise` and are registered with `NG_ASYNC_VALIDATORS`:

```typescript
import { Directive, inject } from '@angular/core';
import {
  AsyncValidator, AbstractControl, ValidationErrors,
  NG_ASYNC_VALIDATORS
} from '@angular/forms';
import { Observable, map, catchError, of, debounceTime, switchMap, first } from 'rxjs';

@Directive({
  selector: '[appUniqueEmail]',
  standalone: true,
  providers: [
    { provide: NG_ASYNC_VALIDATORS, useExisting: UniqueEmailDirective, multi: true }
  ]
})
export class UniqueEmailDirective implements AsyncValidator {
  private userService = inject(UserService);

  validate(control: AbstractControl): Observable<ValidationErrors | null> {
    return this.userService.checkEmailAvailable(control.value).pipe(
      map(available => available ? null : { emailTaken: true }),
      catchError(() => of(null))
    );
  }
}
```

Usage with `updateOn: 'blur'` to avoid excessive server calls:

```html
<input
  name="email"
  [(ngModel)]="model.email"
  required
  email
  appUniqueEmail
  [ngModelOptions]="{ updateOn: 'blur' }"
  #emailCtrl="ngModel"
/>

@if (emailCtrl.pending) {
  <span>Checking availability...</span>
}
@if (emailCtrl.errors?.['emailTaken']) {
  <span class="error">This email is already registered.</span>
}
```

The `pending` state is `true` while the async validator is running. The control remains in `PENDING` status until the observable emits.

## Cross-Field Validation

Validate combinations of multiple fields using a validator on the `NgForm` or `NgModelGroup`:

```typescript
@Directive({
  selector: '[appPasswordMatch]',
  standalone: true,
  providers: [
    { provide: NG_VALIDATORS, useExisting: PasswordMatchDirective, multi: true }
  ]
})
export class PasswordMatchDirective implements Validator {
  validate(group: AbstractControl): ValidationErrors | null {
    const password = group.get('password')?.value;
    const confirm = group.get('confirmPassword')?.value;

    if (password && confirm && password !== confirm) {
      return { passwordMismatch: true };
    }
    return null;
  }
}
```

Apply it to `ngModelGroup`:

```html
<form #regForm="ngForm">
  <ng-container ngModelGroup="passwords" appPasswordMatch #pwGroup="ngModelGroup">
    <input name="password" type="password" [(ngModel)]="model.password" required />
    <input name="confirmPassword" type="password" [(ngModel)]="model.confirmPassword" required />

    @if (pwGroup.errors?.['passwordMismatch'] && pwGroup.touched) {
      <p class="error">Passwords do not match.</p>
    }
  </ng-container>
</form>
```

## Select, Checkbox, and Radio Controls

### Select Dropdown

```html
<select name="role" [(ngModel)]="model.role" required>
  <option value="" disabled>Select a role</option>
  @for (role of roles; track role.id) {
    <option [value]="role.id">{{ role.name }}</option>
  }
</select>
```

For binding to objects instead of strings, use `[ngValue]`:

```html
<select name="role" [(ngModel)]="model.selectedRole" required>
  <option [ngValue]="null" disabled>Select a role</option>
  @for (role of roles; track role.id) {
    <option [ngValue]="role">{{ role.name }}</option>
  }
</select>
```

`[value]` binds a string value. `[ngValue]` binds any type (objects, numbers, etc.).

### Checkboxes

Each checkbox needs a unique `name`:

```html
@for (perm of permissions; track perm.id) {
  <label>
    <input
      type="checkbox"
      [name]="'perm_' + perm.id"
      [(ngModel)]="perm.granted"
    />
    {{ perm.label }}
  </label>
}
```

### Radio Buttons

Radios in the same group share the same `name`:

```html
<label>
  <input type="radio" name="contactMethod" [(ngModel)]="model.contactMethod" value="email" />
  Email
</label>
<label>
  <input type="radio" name="contactMethod" [(ngModel)]="model.contactMethod" value="phone" />
  Phone
</label>
<label>
  <input type="radio" name="contactMethod" [(ngModel)]="model.contactMethod" value="mail" />
  Mail
</label>
```

All three radios bind to the same `model.contactMethod` property. The `value` of the selected radio becomes the model value.

## Sub-Form Components

Break large forms into reusable child components by providing the parent `ControlContainer`:

```typescript
// shared/form-view-provider.ts
import { ControlContainer, NgForm } from '@angular/forms';

export const formViewProvider = {
  provide: ControlContainer,
  useExisting: NgForm
};
```

```typescript
// address-form.component.ts
@Component({
  selector: 'app-address-form',
  standalone: true,
  imports: [FormsModule],
  viewProviders: [formViewProvider],
  template: `
    <ng-container ngModelGroup="address">
      <input name="street" [(ngModel)]="address.street" required placeholder="Street" />
      <input name="city" [(ngModel)]="address.city" required placeholder="City" />
      <input name="zip" [(ngModel)]="address.zip" required placeholder="ZIP" />
    </ng-container>
  `
})
export class AddressFormComponent {
  address = input.required<Address>();
}
```

```html
<!-- parent-form.component.html -->
<form #orderForm="ngForm" (ngSubmit)="submit(orderForm)">
  <input name="customerName" [(ngModel)]="model.customerName" required />

  <app-address-form [address]="model.shippingAddress" />

  <button type="submit" [disabled]="orderForm.invalid">Place Order</button>
</form>
```

The child component's controls (street, city, zip) are automatically registered in the parent `NgForm` under the `address` group. The parent form's validity reflects all nested controls.

## When to Use Template-Driven Forms

| Scenario | Template-Driven | Reactive |
|---|---|---|
| Simple contact forms | Preferred | Overkill |
| Login/registration forms | Good fit | Also fine |
| Complex dynamic forms | Difficult | Preferred |
| Forms with many conditional fields | Awkward | Preferred |
| Forms requiring programmatic control | Limited | Preferred |
| Inline editing (table cells) | Good fit | Also fine |
| Forms shared across multiple pages | Harder to reuse | Preferred |

Template-driven forms work best when:
- The form is simple (5-10 fields).
- Validation rules are straightforward (required, minlength, pattern).
- Two-way binding to a model object is sufficient.
- You want minimal TypeScript code for form management.

## Common Mistakes

- **Missing `name` attribute on `ngModel` controls.** Without `name`, Angular cannot register the control with the form and throws a runtime error. Every `ngModel` inside a `<form>` must have a `name` attribute.

- **Using the same `name` for multiple controls.** Each control must have a unique name within its form group. Duplicate names cause controls to share the same `FormControl` instance, leading to unexpected behavior.

- **Radio buttons with different `name` attributes.** Radios in the same group must share the same `name`. Different names create independent controls instead of a single selection group.

- **Showing errors before the user interacts.** Displaying validation errors immediately on page load is a bad user experience. Always check `touched` or `dirty` (or `submitted`) before showing errors.

- **Forgetting to call `resetForm()` after submission.** Using `form.reset()` does not reset the `submitted` state. Use `form.resetForm()` to fully reset both the form values and the submitted flag.

- **Putting view transition CSS in component styles.** Validation error styles using `.ng-invalid.ng-touched` work in component styles, but keep them consistent. Define a global set of form styling rules and reuse them.

- **Using `[value]` when you need object binding.** `[value]` always binds a string. To bind objects to `<option>` elements, use `[ngValue]`.

---

## Best Practices

- **Use `[(ngModel)]` with a model object, not individual variables.** Binding to `model.name` is cleaner than binding to standalone `name`, `email`, `message` variables. It keeps form data organized.

- **Prefer `updateOn: 'blur'` for async validators.** This prevents firing server requests on every keystroke. Set it at the form level with `[ngFormOptions]` if most fields need it.

- **Show errors only after `touched` or `submitted`.** Combine control state checks: `emailCtrl.invalid && (emailCtrl.touched || myForm.submitted)`.

- **Use `ngModelGroup` for logical grouping.** It organizes the form value into nested objects and lets you validate and reset sections independently.

- **Extract validator logic into `ValidatorFn` functions.** The directive is the template-driven wrapper. The actual logic should be a pure function that works in both template-driven and reactive forms.

- **Use `markAllAsTouched()` on submit.** When the user clicks submit with invalid fields, call `form.form.markAllAsTouched()` to reveal all validation errors at once.

## Real-World Use Cases

### 1. Simple Contact Form

```typescript
@Component({
  selector: 'app-contact',
  standalone: true,
  imports: [FormsModule],
  template: `
    <form #contactForm="ngForm" (ngSubmit)="send(contactForm)">
      <label for="name">Name</label>
      <input id="name" name="name" [(ngModel)]="model.name" required #nameCtrl="ngModel" />
      @if (nameCtrl.invalid && nameCtrl.touched) {
        <span class="error">Name is required.</span>
      }

      <label for="email">Email</label>
      <input id="email" name="email" [(ngModel)]="model.email"
             required email #emailCtrl="ngModel" />
      @if (emailCtrl.invalid && emailCtrl.touched) {
        <span class="error">Enter a valid email.</span>
      }

      <label for="message">Message</label>
      <textarea id="message" name="message" [(ngModel)]="model.message"
                required minlength="10" #msgCtrl="ngModel"></textarea>
      @if (msgCtrl.errors?.['minlength'] && msgCtrl.touched) {
        <span class="error">
          At least 10 characters required
          ({{ msgCtrl.errors?.['minlength'].actualLength }}/{{ msgCtrl.errors?.['minlength'].requiredLength }}).
        </span>
      }

      <button type="submit" [disabled]="contactForm.invalid">Send</button>
    </form>
  `
})
export class ContactComponent {
  model = { name: '', email: '', message: '' };

  send(form: NgForm): void {
    if (form.invalid) {
      form.form.markAllAsTouched();
      return;
    }
    console.log('Sending:', form.value);
    form.resetForm();
  }
}
```

### 2. Settings Form with Conditional Fields

```typescript
@Component({
  selector: 'app-notification-settings',
  standalone: true,
  imports: [FormsModule],
  template: `
    <form #settingsForm="ngForm" (ngSubmit)="save(settingsForm)">
      <h3>Notification Preferences</h3>

      <label>
        <input type="checkbox" name="emailNotifications"
               [(ngModel)]="settings.emailNotifications" />
        Receive email notifications
      </label>

      @if (settings.emailNotifications) {
        <div>
          <label for="notifEmail">Notification email</label>
          <input id="notifEmail" name="notificationEmail"
                 [(ngModel)]="settings.notificationEmail"
                 required email #notifEmailCtrl="ngModel" />
          @if (notifEmailCtrl.invalid && notifEmailCtrl.touched) {
            <span class="error">Valid email required.</span>
          }

          <label>Frequency</label>
          <select name="frequency" [(ngModel)]="settings.frequency" required>
            <option value="instant">Instant</option>
            <option value="daily">Daily digest</option>
            <option value="weekly">Weekly summary</option>
          </select>
        </div>
      }

      <label>
        <input type="checkbox" name="smsNotifications"
               [(ngModel)]="settings.smsNotifications" />
        Receive SMS notifications
      </label>

      @if (settings.smsNotifications) {
        <div>
          <label for="phone">Phone number</label>
          <input id="phone" name="phone" [(ngModel)]="settings.phone"
                 required pattern="^\+?[0-9]{10,15}$" #phoneCtrl="ngModel" />
          @if (phoneCtrl.invalid && phoneCtrl.touched) {
            <span class="error">Enter a valid phone number.</span>
          }
        </div>
      }

      <button type="submit" [disabled]="settingsForm.invalid">Save</button>
    </form>
  `
})
export class NotificationSettingsComponent {
  settings = {
    emailNotifications: false,
    notificationEmail: '',
    frequency: 'daily',
    smsNotifications: false,
    phone: ''
  };

  save(form: NgForm): void {
    if (form.valid) {
      console.log('Settings saved:', form.value);
    }
  }
}
```

### 3. Inline Edit with Standalone ngModel

```typescript
@Component({
  selector: 'app-editable-field',
  standalone: true,
  imports: [FormsModule],
  template: `
    @if (editing) {
      <input
        [(ngModel)]="editValue"
        [ngModelOptions]="{ standalone: true }"
        (keydown.enter)="save()"
        (keydown.escape)="cancel()"
        (blur)="save()"
        #inputEl
      />
    } @else {
      <span (dblclick)="startEdit()">{{ value() }}</span>
    }
  `
})
export class EditableFieldComponent {
  value = model.required<string>();

  editing = false;
  editValue = '';

  startEdit(): void {
    this.editValue = this.value();
    this.editing = true;
  }

  save(): void {
    this.value.set(this.editValue);
    this.editing = false;
  }

  cancel(): void {
    this.editing = false;
  }
}
```

---

> **Key Takeaway:** Template-driven forms use `ngModel`, `NgForm`, and `NgModelGroup` directives to manage form state implicitly in the template, making them the fastest way to build simple forms; always include a `name` attribute on every `ngModel` control and show validation errors only after user interaction (`touched` or `submitted`).
