# 16.3 Typed Reactive Forms

> Since Angular 14, reactive forms are strictly typed by default, giving you compile-time safety on form values, `setValue`/`patchValue` arguments, `valueChanges` emissions, and control access paths.

---

## How Type Inference Works

When you create a `FormControl`, TypeScript infers the value type from the initial value:

```typescript
const name = new FormControl('Alice');
// Inferred type: FormControl<string | null>

name.value;           // string | null
name.setValue('Bob');  // OK
name.setValue(42);     // ERROR: Argument of type 'number' is not assignable
```

The type includes `null` because calling `reset()` sets the value to `null`. This applies to every control created without the `nonNullable` option.

### FormGroup Type Inference

```typescript
const form = new FormGroup({
  email: new FormControl(''),
  age: new FormControl(0)
});

form.value;
// Type: { email: string | null; age: number | null }

form.value.email;     // string | null
form.value.age;       // number | null
form.value.address;   // ERROR: Property 'address' does not exist
```

TypeScript tracks the shape of the group. Accessing a non-existent property is a compile error.

### FormArray Type Inference

```typescript
const names = new FormArray([new FormControl('Alice')]);
// Type: FormArray<FormControl<string | null>>

names.value;            // (string | null)[]
names.at(0).value;      // string | null
names.push(new FormControl('Bob'));     // OK
names.push(new FormControl(42));       // ERROR
```

## The Nullability Problem

Every `FormControl` includes `null` in its type because `reset()` sets the value to `null`:

```typescript
const email = new FormControl('user@example.com');
// Type: FormControl<string | null>

email.reset();
console.log(email.value);  // null

// In templates or computed logic, you must handle null:
const upper = email.value?.toUpperCase();  // need optional chaining
```

This is correct behavior, but it adds noise when you know a control should never be null in practice.

---

## nonNullable Controls

The `nonNullable` option makes a control reset to its initial value instead of `null`. This removes `null` from the type:

```typescript
const email = new FormControl('user@example.com', { nonNullable: true });
// Type: FormControl<string>

email.value;          // string (never null)
email.reset();
console.log(email.value);  // 'user@example.com' (initial value, not null)
```

### Per-Control nonNullable

```typescript
const form = new FormGroup({
  name: new FormControl('', { nonNullable: true }),     // FormControl<string>
  bio: new FormControl<string | null>(null)              // FormControl<string | null>
});

form.value.name;  // string
form.value.bio;   // string | null
```

### When to Use nonNullable

Use `nonNullable: true` when:
- The control has a meaningful default value (empty string, `0`, `false`).
- You do not want `null` to be a valid state.
- You want `reset()` to restore the initial value, not clear it.

Keep nullable controls when:
- The control genuinely starts as `null` (e.g., an optional date picker).
- `null` is a semantically meaningful state (e.g., "no selection").

## NonNullableFormBuilder

Instead of adding `{ nonNullable: true }` to every control, use `NonNullableFormBuilder`:

```typescript
import { Component, inject } from '@angular/core';
import { NonNullableFormBuilder, Validators, ReactiveFormsModule } from '@angular/forms';

@Component({
  selector: 'app-profile',
  standalone: true,
  imports: [ReactiveFormsModule],
  templateUrl: './profile.component.html'
})
export class ProfileComponent {
  private fb = inject(NonNullableFormBuilder);

  form = this.fb.group({
    name: ['', Validators.required],           // FormControl<string>
    email: ['', [Validators.required]],        // FormControl<string>
    age: [0],                                   // FormControl<number>
    active: [true]                              // FormControl<boolean>
  });
}
```

Every control created through `NonNullableFormBuilder` has `nonNullable: true` by default. The form value type is:

```typescript
{
  name: string;
  email: string;
  age: number;
  active: boolean;
}
// No null anywhere
```

### Accessing via FormBuilder.nonNullable

You can also use the standard `FormBuilder` and access its `.nonNullable` property:

```typescript
private fb = inject(FormBuilder);

form = this.fb.nonNullable.group({
  name: [''],
  email: ['']
});
// Same as using NonNullableFormBuilder
```

## Explicit Type Annotations

### When Inference Falls Short

If a control starts with `null`, TypeScript infers `FormControl<null>`, which is too narrow:

```typescript
// BAD: inferred as FormControl<null>
const date = new FormControl(null);
date.setValue('2025-01-01');  // ERROR: 'string' is not assignable to 'null'
```

Fix by explicitly specifying the type parameter:

```typescript
// GOOD: explicitly typed
const date = new FormControl<string | null>(null);
date.setValue('2025-01-01');  // OK
date.setValue(null);          // OK
```

### When Initial Value Does Not Reflect the Full Type

```typescript
// Bad: inferred as FormControl<string | null>
// but you want to also allow number
const flexible = new FormControl('');
flexible.setValue(42);  // ERROR

// Good: explicitly typed for the union
const flexible = new FormControl<string | number>('', { nonNullable: true });
flexible.setValue(42);  // OK
flexible.setValue('hello');  // OK
```

## FormGroup with Interface

For complex forms, define a type interface that maps control names to their `FormControl` types:

```typescript
interface UserForm {
  name: FormControl<string>;
  email: FormControl<string>;
  age: FormControl<number>;
  address: FormGroup<AddressForm>;
}

interface AddressForm {
  street: FormControl<string>;
  city: FormControl<string>;
  zip: FormControl<string>;
}

@Component({ /* ... */ })
export class UserFormComponent {
  private fb = inject(NonNullableFormBuilder);

  form = new FormGroup<UserForm>({
    name: new FormControl('', { nonNullable: true, validators: Validators.required }),
    email: new FormControl('', { nonNullable: true }),
    age: new FormControl(0, { nonNullable: true }),
    address: new FormGroup<AddressForm>({
      street: new FormControl('', { nonNullable: true }),
      city: new FormControl('', { nonNullable: true }),
      zip: new FormControl('', { nonNullable: true })
    })
  });
}
```

Now TypeScript enforces the structure. Missing keys, extra keys, or wrong control types produce compile errors.

### Bad Practice: Declaring the Type Separately from Initialization

```typescript
// BAD: declaring type without initialization loses type safety
form!: FormGroup<UserForm>;

ngOnInit(): void {
  this.form = this.fb.group({ /* ... */ });
  // TypeScript cannot verify the builder output matches UserForm
}
```

```typescript
// GOOD: declare and initialize together
form = new FormGroup<UserForm>({
  // TypeScript verifies every control matches the interface
});
```

Always combine form declaration with initialization. Separating them (e.g., declaring in the class body and initializing in `ngOnInit`) breaks type inference.

## Optional Controls

Some controls may be added or removed dynamically. Mark them with `?` in the interface:

```typescript
interface LoginForm {
  email: FormControl<string>;
  password: FormControl<string>;
  rememberMe?: FormControl<boolean>;  // optional
}

const form = new FormGroup<LoginForm>({
  email: new FormControl('', { nonNullable: true }),
  password: new FormControl('', { nonNullable: true })
  // rememberMe is omitted: this is valid because it is optional
});

// Add it later
form.addControl('rememberMe', new FormControl(false, { nonNullable: true }));

// Remove it
form.removeControl('rememberMe');

// Access (may be undefined)
const remember = form.get('rememberMe');  // FormControl<boolean> | undefined
```

Without the `?`, removing a control or omitting it from the constructor would be a type error.

---

## value vs getRawValue()

The type of `form.value` accounts for disabled controls by making their properties optional (potentially `undefined`):

```typescript
const form = new FormGroup({
  name: new FormControl('Alice', { nonNullable: true }),
  email: new FormControl('alice@example.com', { nonNullable: true })
});

// All controls enabled:
form.value;
// Type: { name?: string; email?: string }
// Note: properties are OPTIONAL because any could be disabled

form.value.name;   // string | undefined
```

Why is `name` optional even though it is always present? Because if `name` were disabled, it would be excluded from `value`. TypeScript must account for that possibility.

### getRawValue() Provides the Full Type

```typescript
form.getRawValue();
// Type: { name: string; email: string }
// No optional properties, no undefined
```

`getRawValue()` always includes all controls, so the type does not need the `?` modifier.

**Rule:** Use `getRawValue()` when you need a complete, fully-typed form value (especially for API submissions). Use `value` only when you are prepared to handle potentially missing fields.

## FormRecord: Typed Dynamic Groups

`FormRecord` is for groups where the keys are not known at compile time but all controls share the same type:

```typescript
import { FormRecord, FormControl } from '@angular/forms';

// All controls are FormControl<string | null>
const tags = new FormRecord<FormControl<string | null>>({});

// Add controls dynamically
tags.addControl('frontend', new FormControl('Angular'));
tags.addControl('backend', new FormControl('Node.js'));

// Type-safe
tags.value;  // { [key: string]: string | null }

// Remove
tags.removeControl('backend');
```

### FormRecord vs FormGroup

```typescript
// FormGroup: fixed keys, known at compile time
interface ProfileForm {
  name: FormControl<string>;
  email: FormControl<string>;
}
const profile = new FormGroup<ProfileForm>({ /* ... */ });
profile.addControl('age', ctrl);  // ERROR: 'age' is not in ProfileForm

// FormRecord: dynamic keys, all controls same type
const record = new FormRecord<FormControl<boolean>>({});
record.addControl('anyKey', new FormControl(true));  // OK
record.addControl('anotherKey', new FormControl(false));  // OK
```

### NonNullable FormRecord

```typescript
const permissions = new FormRecord<FormControl<boolean>>({});
// Each control is FormControl<boolean> (nonNullable is per-control)

// Using NonNullableFormBuilder
const fb = inject(NonNullableFormBuilder);
const perms = fb.record<boolean>({
  read: true,
  write: false
});
// Type: FormRecord<FormControl<boolean>>
```

## UntypedForm* Classes: Migration Path

When Angular 14 introduced typed forms, a migration schematic renamed all existing form classes to their `Untyped` counterparts:

| Typed (new) | Untyped (legacy) |
|---|---|
| `FormControl` | `UntypedFormControl` |
| `FormGroup` | `UntypedFormGroup` |
| `FormArray` | `UntypedFormArray` |
| `FormBuilder` | `UntypedFormBuilder` |

The `Untyped*` classes behave identically to the pre-Angular 14 versions. All values are `any`.

### Migration Strategy

1. **Run the schematic.** `ng update @angular/core` renames your existing code to `Untyped*` classes.
2. **Incrementally convert.** Replace `UntypedFormGroup` with `FormGroup` one form at a time. TypeScript will immediately report type errors.
3. **Fix errors.** Add initial values, specify type parameters, or use `nonNullable` as needed.
4. **Remove all Untyped imports** when migration is complete.

```typescript
// Step 1: After migration schematic
import { UntypedFormGroup, UntypedFormControl } from '@angular/forms';

const form = new UntypedFormGroup({
  name: new UntypedFormControl('')
});

form.value.anything;  // no error (any)

// Step 2: Convert to typed
import { FormGroup, FormControl } from '@angular/forms';

const form = new FormGroup({
  name: new FormControl('', { nonNullable: true })
});

form.value.anything;  // ERROR: Property 'anything' does not exist
form.value.name;      // string | undefined (correct)
```

## Type Narrowing with get()

The `get()` method returns `AbstractControl | null`. You lose the specific control type:

```typescript
const form = new FormGroup({
  email: new FormControl('', { nonNullable: true })
});

const emailCtrl = form.get('email');
// Type: AbstractControl<string, string> | null

// Must narrow:
if (emailCtrl) {
  emailCtrl.value;  // string
}
```

### Better: Direct Property Access

```typescript
const emailCtrl = form.controls.email;
// Type: FormControl<string>
// No null check needed, fully typed
```

Direct property access through `controls` preserves the exact type. Use `get()` only when you have a dynamic path string.

### Nested Access

```typescript
interface MyForm {
  address: FormGroup<{
    city: FormControl<string>;
  }>;
}

const form = new FormGroup<MyForm>({ /* ... */ });

// Direct access preserves types at every level
form.controls.address.controls.city.value;  // string
```

## Typed FormBuilder Patterns

### fb.group() Type Inference

```typescript
const fb = inject(NonNullableFormBuilder);

// TypeScript infers the full type from the builder arguments
const form = fb.group({
  name: ['', Validators.required],    // FormControl<string>
  age: [0],                            // FormControl<number>
  active: [true]                       // FormControl<boolean>
});

// Hover over 'form' to see:
// FormGroup<{
//   name: FormControl<string>;
//   age: FormControl<number>;
//   active: FormControl<boolean>;
// }>
```

### fb.array() Type Inference

```typescript
const aliases = fb.array(['Alice', 'Bob']);
// Type: FormArray<FormControl<string>>

aliases.push(fb.control('Charlie'));  // OK
aliases.push(fb.control(42));        // ERROR
```

### fb.record() Type Inference

```typescript
const settings = fb.record({ theme: 'dark', locale: 'en' });
// Type: FormRecord<FormControl<string>>

settings.addControl('timezone', fb.control('UTC'));  // OK
settings.addControl('count', fb.control(5));         // ERROR: number not string
```

## Extracting Value Types

Angular provides utility types to extract the value type from a form:

```typescript
import { FormGroup, FormControl } from '@angular/forms';

const form = new FormGroup({
  name: new FormControl('', { nonNullable: true }),
  email: new FormControl('', { nonNullable: true }),
  age: new FormControl(0, { nonNullable: true })
});

// Extract the raw value type
type FormValue = typeof form.value;
// { name?: string; email?: string; age?: number }

// Extract the raw value type (all properties required)
type FormRawValue = ReturnType<typeof form.getRawValue>;
// { name: string; email: string; age: number }
```

Use `ReturnType<typeof form.getRawValue>` to get a type where all properties are required (matching what `getRawValue()` returns). This is useful for service method signatures:

```typescript
@Injectable({ providedIn: 'root' })
export class UserService {
  save(data: ReturnType<typeof ProfileComponent.prototype.form.getRawValue>): Observable<void> {
    // data is fully typed: { name: string; email: string; age: number }
    return this.http.post<void>('/api/users', data);
  }
}
```

A cleaner approach is to define the type separately:

```typescript
interface UserFormValue {
  name: string;
  email: string;
  age: number;
}

// Use it for both the form and the service
```

## Common Mistakes

- **Declaring form type separately from initialization.** Writing `form!: FormGroup<MyForm>` and initializing in `ngOnInit()` breaks type inference. The `FormBuilder`'s output type may not match your interface, and TypeScript cannot verify it. Always declare and initialize together.

- **Initializing a control with `null` without an explicit type.** `new FormControl(null)` infers `FormControl<null>`. You must write `new FormControl<string | null>(null)` to get the intended type.

- **Using `form.value` for API calls with disabled controls.** Disabled controls are excluded from `value`, making their properties `undefined`. Use `getRawValue()` for submissions.

- **Using `get()` instead of `controls` for typed access.** `get()` returns `AbstractControl | null`, losing the specific type. Use `form.controls.email` for full type safety.

- **Explicitly typing the form variable as `FormGroup` without a type parameter.** `FormGroup` defaults to `FormGroup<any>` when no type parameter is provided. Let TypeScript infer it, or provide the interface explicitly.

- **Forgetting `nonNullable` and then not handling `null`.** If a control is nullable (the default), every `.value` access is `T | null`. Either add `nonNullable: true` or handle `null` in your code.

---

## Best Practices

- **Let TypeScript infer the form type.** Instead of writing `FormGroup<MyInterface>`, let the builder infer it from initialization. This eliminates redundancy and keeps the type synchronized with the actual structure.

- **Use `NonNullableFormBuilder` for most forms.** The majority of form controls have meaningful defaults and should never be `null`. This eliminates null-handling noise.

- **Use `getRawValue()` for all submissions.** It provides a complete, fully-typed value regardless of which controls are disabled.

- **Access controls through `.controls` instead of `.get()`.** Direct property access preserves the exact control type and avoids null checks.

- **Define optional controls with `?` in the interface.** If you use `addControl`/`removeControl` for conditional fields, the interface must mark those controls as optional.

- **Migrate from `UntypedForm*` incrementally.** Convert one form at a time. TypeScript immediately flags type mismatches, making it safe to migrate gradually.

- **Use `FormRecord` for truly dynamic key sets.** When keys come from runtime data (user permissions, feature flags, tag lists), `FormRecord` gives you type safety without pre-declaring every key.

## Real-World Use Cases

### 1. Fully Typed Registration Form

```typescript
interface RegistrationForm {
  username: FormControl<string>;
  email: FormControl<string>;
  password: FormControl<string>;
  confirmPassword: FormControl<string>;
  acceptTerms: FormControl<boolean>;
  newsletter?: FormControl<boolean>;
}

@Component({
  selector: 'app-register',
  standalone: true,
  imports: [ReactiveFormsModule],
  template: `
    <form [formGroup]="form" (ngSubmit)="register()">
      <input formControlName="username" />
      <input formControlName="email" type="email" />
      <input formControlName="password" type="password" />
      <input formControlName="confirmPassword" type="password" />

      <label>
        <input type="checkbox" formControlName="acceptTerms" />
        I accept the terms
      </label>

      <button type="submit" [disabled]="form.invalid">Register</button>
    </form>
  `
})
export class RegisterComponent {
  form = new FormGroup<RegistrationForm>({
    username: new FormControl('', { nonNullable: true, validators: [Validators.required, Validators.minLength(3)] }),
    email: new FormControl('', { nonNullable: true, validators: [Validators.required, Validators.email] }),
    password: new FormControl('', { nonNullable: true, validators: [Validators.required, Validators.minLength(8)] }),
    confirmPassword: new FormControl('', { nonNullable: true, validators: Validators.required }),
    acceptTerms: new FormControl(false, { nonNullable: true, validators: Validators.requiredTrue })
  });

  register(): void {
    const data = this.form.getRawValue();
    // data is: { username: string; email: string; password: string;
    //            confirmPassword: string; acceptTerms: boolean }
    // No null, no undefined, fully typed
    console.log(data.username.toUpperCase());  // TypeScript knows it's a string
  }
}
```

### 2. Dynamic Permissions with FormRecord

```typescript
@Component({
  selector: 'app-permissions',
  standalone: true,
  imports: [ReactiveFormsModule, KeyValuePipe],
  template: `
    <form [formGroup]="form" (ngSubmit)="save()">
      @for (perm of form.controls | keyvalue; track perm.key) {
        <label>
          <input type="checkbox" [formControlName]="perm.key" />
          {{ perm.key }}
        </label>
      }

      <button type="submit">Save Permissions</button>
    </form>
  `
})
export class PermissionsComponent {
  private fb = inject(NonNullableFormBuilder);
  private permService = inject(PermissionService);

  // FormRecord: all controls are FormControl<boolean>
  form = new FormRecord<FormControl<boolean>>({});

  constructor() {
    // Load permissions from API
    this.permService.getAll().subscribe(permissions => {
      permissions.forEach(p => {
        this.form.addControl(
          p.name,
          new FormControl(p.granted, { nonNullable: true })
        );
      });
    });
  }

  save(): void {
    const values = this.form.getRawValue();
    // Type: { [key: string]: boolean }
    // Every value is a boolean, never null
    this.permService.update(values).subscribe();
  }
}
```

### 3. Nested Form with Type-Safe Access

```typescript
interface OrderForm {
  customer: FormGroup<{
    name: FormControl<string>;
    email: FormControl<string>;
  }>;
  items: FormArray<FormGroup<{
    product: FormControl<string>;
    quantity: FormControl<number>;
  }>>;
  notes: FormControl<string>;
}

@Component({
  selector: 'app-order',
  standalone: true,
  imports: [ReactiveFormsModule],
  template: `
    <form [formGroup]="form" (ngSubmit)="submit()">
      <div formGroupName="customer">
        <input formControlName="name" placeholder="Customer" />
        <input formControlName="email" placeholder="Email" />
      </div>

      <div formArrayName="items">
        @for (item of form.controls.items.controls; track $index) {
          <div [formGroupName]="$index">
            <input formControlName="product" />
            <input formControlName="quantity" type="number" />
            <button type="button" (click)="removeItem($index)">Remove</button>
          </div>
        }
      </div>

      <button type="button" (click)="addItem()">Add Item</button>
      <textarea formControlName="notes"></textarea>
      <button type="submit">Submit Order</button>
    </form>
  `
})
export class OrderComponent {
  form = new FormGroup<OrderForm>({
    customer: new FormGroup({
      name: new FormControl('', { nonNullable: true, validators: Validators.required }),
      email: new FormControl('', { nonNullable: true, validators: Validators.email })
    }),
    items: new FormArray([
      this.createItemGroup()
    ]),
    notes: new FormControl('', { nonNullable: true })
  });

  private createItemGroup() {
    return new FormGroup({
      product: new FormControl('', { nonNullable: true, validators: Validators.required }),
      quantity: new FormControl(1, { nonNullable: true, validators: Validators.min(1) })
    });
  }

  addItem(): void {
    this.form.controls.items.push(this.createItemGroup());
  }

  removeItem(index: number): void {
    this.form.controls.items.removeAt(index);
  }

  submit(): void {
    const data = this.form.getRawValue();
    // Fully typed nested structure:
    // {
    //   customer: { name: string; email: string };
    //   items: { product: string; quantity: number }[];
    //   notes: string;
    // }
    console.log(data.customer.name);          // string
    console.log(data.items[0].quantity);       // number
    console.log(data.items[0].quantity + 1);   // TypeScript knows it is a number
  }
}
```

---

> **Key Takeaway:** Let TypeScript infer your form types from initialization, use `NonNullableFormBuilder` to eliminate null noise, access controls through `.controls` for full type safety, and always use `getRawValue()` for submissions to get a complete, properly-typed form value.
