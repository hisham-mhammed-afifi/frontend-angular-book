# 16.8 Signal Forms (Angular 21)

> Angular 21 introduces signal-based forms that use signals instead of observables, centralized schema validation instead of per-control validators, and automatic two-way binding instead of manual CVA plumbing.

---

## What Signal Forms Replace

Signal Forms are not an evolution of Reactive Forms. They are a ground-up rethink of how forms work in Angular, built entirely on the signals primitive.

| Concept | Reactive Forms | Signal Forms |
|---------|---------------|--------------|
| Source of truth | `FormControl` / `FormGroup` tree | Writable signal holding a plain object |
| State tracking | Observables (`valueChanges`, `statusChanges`) | Signals (`value()`, `valid()`, `dirty()`) |
| Validation | Validators attached per-control | Schema function with path-based rules |
| Custom controls | `ControlValueAccessor` (4 methods + provider) | `FormValueControl` (1 signal property) |
| Type safety | Explicit with typed forms | Inferred from model automatically |
| Disabled/hidden | Imperative (`disable()`, `enable()`) | Declarative rules (`disabled()`, `hidden()`) |
| Import path | `@angular/forms` | `@angular/forms/signals` |

## Creating a Signal Form

Signal Forms follow a five-step pattern: model, form, bind, read, submit.

### Step 1: Define the Data Model

The model is a writable signal holding a plain TypeScript object:

```typescript
import { signal } from '@angular/core';

interface LoginData {
  email: string;
  password: string;
}

// The signal IS your form state
const loginModel = signal<LoginData>({
  email: '',
  password: '',
});
```

The signal must be writable. If your data comes from a readonly source (like a store selector), wrap it with `linkedSignal()`:

```typescript
// readonly signal from a store
private flight = this.store.selectSignal(selectCurrentFlight);

// Convert to writable for the form
protected flightModel = linkedSignal(() => this.flight());
```

### Step 2: Generate the Field Tree with `form()`

The `form()` function takes your model signal and returns a field tree, an object that mirrors your model's shape:

```typescript
import { form } from '@angular/forms/signals';

@Component({ /* ... */ })
export class LoginComponent {
  loginModel = signal<LoginData>({ email: '', password: '' });

  // No validation yet
  loginForm = form(this.loginModel);
  // loginForm.email -> field for email
  // loginForm.password -> field for password
}
```

With validation (schema function as second argument):

```typescript
import { form, required, email, minLength } from '@angular/forms/signals';

loginForm = form(this.loginModel, (path) => {
  required(path.email, { message: 'Email is required' });
  email(path.email, { message: 'Enter a valid email' });
  required(path.password, { message: 'Password is required' });
  minLength(path.password, 8, { message: 'At least 8 characters' });
});
```

### Step 3: Bind with the `[formField]` Directive

```typescript
import { FormField } from '@angular/forms/signals';

@Component({
  selector: 'app-login',
  standalone: true,
  imports: [FormField],
  template: `
    <form (ngSubmit)="onSubmit()">
      <label>
        Email
        <input type="email" [formField]="loginForm.email" />
      </label>

      <label>
        Password
        <input type="password" [formField]="loginForm.password" />
      </label>

      <button type="submit">Log In</button>
    </form>
  `,
})
export class LoginComponent {
  // ...
}
```

The `[formField]` directive handles:
- Two-way value synchronization
- Setting `required`, `disabled`, `readonly` attributes on the element
- Tracking `touched` state (on blur)
- Tracking `dirty` state (on value change)

### Step 4: Read Field State

Call a field as a function to get its `FieldState`:

```typescript
// In the template
@if (loginForm.email().touched() && loginForm.email().invalid()) {
  <ul>
    @for (error of loginForm.email().errors(); track error) {
      <li>{{ error.message }}</li>
    }
  </ul>
}
```

Available state signals on every field:

| Signal | Type | Description |
|--------|------|-------------|
| `value()` | `WritableSignal<T>` | Current field value |
| `valid()` | `Signal<boolean>` | All validators pass |
| `invalid()` | `Signal<boolean>` | At least one validator fails |
| `errors()` | `Signal<ValidationError[]>` | Array of error objects |
| `touched()` | `Signal<boolean>` | User has focused and blurred |
| `dirty()` | `Signal<boolean>` | User has modified the value |
| `pending()` | `Signal<boolean>` | Async validation in progress |
| `disabled()` | `Signal<boolean>` | Field is disabled |
| `readonly()` | `Signal<boolean>` | Field is read-only |
| `hidden()` | `Signal<boolean>` | Field is hidden |
| `disabledReasons()` | `Signal<Reason[]>` | Why the field is disabled |

### Step 5: Submit and Read the Model

```typescript
onSubmit(): void {
  // Read the current model value directly from the signal
  const credentials = this.loginModel();
  console.log(credentials);
  // { email: 'user@example.com', password: 'secret123' }
}
```

Or use the `submit()` utility for validation-aware submission:

```typescript
import { submit } from '@angular/forms/signals';

async onSubmit(): Promise<void> {
  await submit(this.loginForm, async (formRef) => {
    try {
      await this.authService.login(formRef().value());
      return null; // Success
    } catch (error) {
      return {
        kind: 'login_failed',
        message: 'Invalid credentials',
      };
    }
  });
}
```

The `submit()` function validates first. If validation fails, the callback is not executed. Server errors returned from the callback are added to the form's error list.

## Input Type Handling

The `[formField]` directive automatically handles type conversion based on the HTML input type:

```typescript
interface BookingData {
  name: string;        // text
  email: string;       // email
  guests: number;      // number (auto-converts string to number)
  date: string;        // date (ISO format YYYY-MM-DD)
  time: string;        // time (HH:mm)
  vip: boolean;        // checkbox
  mealPlan: string;    // radio / select
  notes: string;       // textarea
}

bookingModel = signal<BookingData>({
  name: '',
  email: '',
  guests: 1,
  date: '',
  time: '',
  vip: false,
  mealPlan: 'standard',
  notes: '',
});

bookingForm = form(this.bookingModel);
```

```html
<input type="text" [formField]="bookingForm.name" />
<input type="email" [formField]="bookingForm.email" />
<input type="number" [formField]="bookingForm.guests" />
<input type="date" [formField]="bookingForm.date" />
<input type="time" [formField]="bookingForm.time" />

<!-- Checkbox: binds to boolean -->
<input type="checkbox" [formField]="bookingForm.vip" />

<!-- Radio buttons: multiple radios with same formField share binding -->
<input type="radio" [formField]="bookingForm.mealPlan" value="standard" />
<input type="radio" [formField]="bookingForm.mealPlan" value="premium" />

<!-- Select -->
<select [formField]="bookingForm.mealPlan">
  <option value="standard">Standard</option>
  <option value="premium">Premium</option>
</select>

<!-- Textarea -->
<textarea [formField]="bookingForm.notes"></textarea>
```

No manual conversion needed. The directive knows `type="number"` means convert to `number`, `type="checkbox"` means bind to `boolean`, and so on.

## Schema Validation

All validation rules live in one place: the schema function passed as the second argument to `form()`.

### Built-in Validators

```typescript
import {
  form,
  required,
  email,
  min,
  max,
  minLength,
  maxLength,
  pattern,
} from '@angular/forms/signals';

const profileForm = form(this.profileModel, (path) => {
  required(path.name, { message: 'Name is required' });
  minLength(path.name, 2, { message: 'At least 2 characters' });
  maxLength(path.name, 50, { message: 'Max 50 characters' });

  required(path.email, { message: 'Email is required' });
  email(path.email, { message: 'Invalid email format' });

  min(path.age, 18, { message: 'Must be 18 or older' });
  max(path.age, 120, { message: 'Invalid age' });

  pattern(path.phone, /^\+\d{1,3}-\d{7,15}$/, {
    message: 'Format: +1-5551234567',
  });
});
```

Every validator accepts an optional `message` property. Error objects have a `kind` (the validator name, e.g., `"required"`, `"email"`) and optionally a `message`.

### Custom Validators

Use the `validate()` function for custom rules:

```typescript
import { validate } from '@angular/forms/signals';

const form1 = form(this.model, (path) => {
  validate(path.website, ({ value }) => {
    if (!value().startsWith('https://')) {
      return { kind: 'https', message: 'URL must start with https://' };
    }
    return null;
  });
});
```

The validator callback receives a `FieldContext` with:
- `value()`: signal for the current field value
- `valueOf(otherPath)`: read another field's value (for cross-field validation)
- `state`: reference to the field state

### Cross-Field Validation

Use `valueOf()` to reference other fields:

```typescript
const registrationForm = form(this.registrationModel, (path) => {
  required(path.password);
  required(path.confirmPassword);

  validate(path.confirmPassword, ({ value, valueOf }) => {
    if (value() !== valueOf(path.password)) {
      return { kind: 'passwordMismatch', message: 'Passwords do not match' };
    }
    return null;
  });
});
```

Cross-field validators re-execute automatically when any referenced field changes. No manual subscription management.

### Conditional Validation

Apply validators only when a condition is true:

```typescript
const orderForm = form(this.orderModel, (path) => {
  required(path.promoCode, {
    message: 'Promo code required for discounts',
    when: ({ valueOf }) => valueOf(path.applyDiscount),
  });
});
```

Or use `applyWhen` for applying entire schema blocks conditionally:

```typescript
const flightForm = form(this.flightModel, (path) => {
  required(path.from);
  required(path.to);

  // Only validate delay fields when the flight is delayed
  applyWhen(path, (ctx) => ctx.valueOf(path.delayed), (path) => {
    required(path.delayReason, { message: 'Reason required for delayed flights' });
    min(path.delayMinutes, 1, { message: 'Delay must be at least 1 minute' });
  });
});
```

### Async Validation

Use `validateHttp()` for server-side validation:

```typescript
import { validateHttp } from '@angular/forms/signals';

const signupForm = form(this.signupModel, (path) => {
  required(path.username);
  minLength(path.username, 3);

  // Runs only after sync validators pass
  validateHttp(path.username, {
    request: ({ value }) => `/api/check-username?username=${value()}`,
    onSuccess: (response) => {
      return response.taken
        ? { kind: 'usernameTaken', message: 'Username already taken' }
        : null;
    },
    onError: () => ({
      kind: 'networkError',
      message: 'Could not verify username',
    }),
  });
});
```

Async validation only runs after all synchronous validators pass. While running, `field().pending()` returns `true`.

### Standard Schema Integration (Zod, Valibot)

Signal Forms integrate with Standard Schema libraries:

```typescript
import { validateStandardSchema } from '@angular/forms/signals';
import * as z from 'zod';

const userSchema = z.object({
  email: z.string().email(),
  password: z.string().min(8),
  age: z.number().min(18),
});

const userForm = form(this.userModel, (path) => {
  validateStandardSchema(path, userSchema);
});
```

You can pass a signal as the schema for dynamic validation rules that change based on application state.

## Declarative Field State

Unlike Reactive Forms where you call `disable()`, `enable()`, or `markAsReadonly()`, Signal Forms declare state rules in the schema.

### Disabled State

```typescript
const form1 = form(this.model, (path) => {
  // Disable the delay field when the flight is not delayed
  disabled(path.delay, (ctx) => !ctx.valueOf(path.delayed));
});
```

You can provide reasons for why a field is disabled:

```typescript
disabled(path.delay, (ctx) =>
  !ctx.valueOf(path.delayed) ? 'Flight is not delayed' : false
);
```

Access the reasons in the template:

```html
@if (flightForm.delay().disabled()) {
  @for (reason of flightForm.delay().disabledReasons(); track reason) {
    <p class="hint">{{ reason.message }}</p>
  }
}
```

### Hidden State

```typescript
hidden(path.notes, () => !this.showNotes());
```

Hidden fields are excluded from validation. This prevents "fix this error on a field you can't see" situations.

### Readonly State

```typescript
readonly(path.id, () => true);
readonly(path.createdAt, () => true);
```

### Debouncing

Apply debouncing at the field or form level:

```typescript
const searchForm = form(this.searchModel, (path) => {
  debounce(path, 300); // 300ms debounce on all fields
});
```

## Nested Forms

Signal Forms handle nested objects naturally. The field tree mirrors your data shape.

### Nested Object Model

```typescript
interface Order {
  customer: {
    name: string;
    email: string;
  };
  shipping: {
    street: string;
    city: string;
    zip: string;
  };
  notes: string;
}

orderModel = signal<Order>({
  customer: { name: '', email: '' },
  shipping: { street: '', city: '', zip: '' },
  notes: '',
});

orderForm = form(this.orderModel, (path) => {
  required(path.customer.name);
  email(path.customer.email);
  required(path.shipping.street);
  required(path.shipping.city);
  required(path.shipping.zip);
});
```

Access nested fields with dot notation:

```html
<input [formField]="orderForm.customer.name" />
<input [formField]="orderForm.customer.email" />
<input [formField]="orderForm.shipping.street" />
```

### Reusable Sub-Schemas with `schema()` and `apply()`

Extract validation rules into reusable schema functions:

```typescript
import { schema, apply } from '@angular/forms/signals';

// Reusable schema for an address
const addressSchema = schema<Address>((path) => {
  required(path.street, { message: 'Street required' });
  required(path.city, { message: 'City required' });
  pattern(path.zip, /^\d{5}$/, { message: 'Invalid ZIP' });
});

// Reusable schema for a person
const personSchema = schema<Person>((path) => {
  required(path.name);
  email(path.email);
});

// Compose schemas
const orderForm = form(this.orderModel, (path) => {
  apply(path.customer, personSchema);
  apply(path.shipping, addressSchema);
  apply(path.billing, addressSchema);
});
```

### Sub-Components with FieldTree

Pass a branch of the field tree to a child component:

```typescript
import { FieldTree } from '@angular/forms/signals';

@Component({
  selector: 'app-address-form',
  standalone: true,
  imports: [FormField],
  template: `
    <input placeholder="Street" [formField]="address.street" />
    <input placeholder="City" [formField]="address.city" />
    <input placeholder="ZIP" [formField]="address.zip" />
  `,
})
export class AddressFormComponent {
  address = input.required<FieldTree<Address>>();
}
```

Parent usage:

```html
<app-address-form [address]="orderForm.shipping" />
<app-address-form [address]="orderForm.billing" />
```

No `ControlContainer`, no `viewProviders`, no `NG_VALUE_ACCESSOR`. Just pass the field tree branch as an input.

## Form Arrays

Signal Forms handle arrays using `applyEach()` in the schema and standard `@for` in the template.

### Defining an Array Form

```typescript
interface Invoice {
  customer: string;
  items: InvoiceItem[];
}

interface InvoiceItem {
  description: string;
  quantity: number;
  price: number;
}

invoiceModel = signal<Invoice>({
  customer: '',
  items: [{ description: '', quantity: 1, price: 0 }],
});

invoiceForm = form(this.invoiceModel, (path) => {
  required(path.customer);

  // Validate each item in the array
  applyEach(path.items, (item) => {
    required(item.description, { message: 'Description required' });
    min(item.quantity, 1, { message: 'At least 1' });
    min(item.price, 0.01, { message: 'Price must be positive' });
  });
});
```

### Template Binding

```html
@for (item of invoiceForm.items; track $index) {
  <div class="invoice-item">
    <input [formField]="item.description" placeholder="Description" />
    <input type="number" [formField]="item.quantity" placeholder="Qty" />
    <input type="number" [formField]="item.price" placeholder="Price" />
    <button type="button" (click)="removeItem($index)">Remove</button>
  </div>
}

<button type="button" (click)="addItem()">Add Item</button>
```

### Adding and Removing Items

Manipulate arrays by updating the model signal directly:

```typescript
addItem(): void {
  this.invoiceForm.items().value.update(items => [
    ...items,
    { description: '', quantity: 1, price: 0 },
  ]);
}

removeItem(index: number): void {
  this.invoiceForm.items().value.update(items =>
    items.filter((_, i) => i !== index)
  );
}
```

No `FormArray.push()`, no `FormArray.removeAt()`. Just update the signal.

## Programmatic Value Updates

### Setting Individual Field Values

```typescript
// Set a single field
this.orderForm.customer.name().value.set('Jane Doe');

// Update based on current value
this.orderForm.shipping.zip().value.update(zip => zip.trim());
```

### Resetting the Entire Form

```typescript
resetForm(): void {
  this.orderModel.set({
    customer: { name: '', email: '' },
    shipping: { street: '', city: '', zip: '' },
    notes: '',
  });
}
```

Since the field tree is derived from the model signal, resetting the signal resets everything: values, dirty state, and validation.

### Patching Partial Data

```typescript
// Update just the customer portion
this.orderModel.update(current => ({
  ...current,
  customer: {
    ...current.customer,
    email: 'new@email.com',
  },
}));
```

## Custom Form Controls with FormValueControl

Signal Forms replace `ControlValueAccessor` with the `FormValueControl` interface.

### Single-Value Controls

```typescript
import { FormValueControl, ValidationError } from '@angular/forms/signals';

@Component({
  selector: 'app-star-rating',
  standalone: true,
  template: `
    @for (star of [1, 2, 3, 4, 5]; track star) {
      <button
        type="button"
        [class.active]="star <= value()"
        [disabled]="disabled()"
        (click)="select(star)">
        {{ star <= value() ? '\u2605' : '\u2606' }}
      </button>
    }
    @if (invalid()) {
      @for (err of errors(); track err) {
        <span class="error">{{ err.message }}</span>
      }
    }
  `,
})
export class StarRatingComponent implements FormValueControl<number> {
  // Required: model signal for the value
  value = model(0);

  // Optional: receive form state automatically
  disabled = input(false);
  errors = input<readonly ValidationError[]>([]);
  invalid = input(false);
  touched = model(false);

  select(star: number): void {
    if (!this.disabled()) {
      this.value.set(star);
    }
  }
}
```

Usage:

```html
<app-star-rating [formField]="reviewForm.rating" />
```

No providers. No `forwardRef`. No `writeValue`. No `registerOnChange`. The `[formField]` directive detects the `value` model signal and handles everything.

### Checkbox Controls

For boolean on/off controls, implement `FormCheckboxControl`:

```typescript
import { FormCheckboxControl } from '@angular/forms/signals';

@Component({
  selector: 'app-toggle-switch',
  standalone: true,
  template: `
    <button
      type="button"
      role="switch"
      [attr.aria-checked]="checked()"
      [disabled]="disabled()"
      (click)="toggle()">
      <span class="track">
        <span class="thumb" [class.on]="checked()"></span>
      </span>
    </button>
  `,
})
export class ToggleSwitchComponent implements FormCheckboxControl {
  checked = model(false);
  disabled = input(false);

  toggle(): void {
    if (!this.disabled()) {
      this.checked.update(v => !v);
    }
  }
}
```

### Optional State Signals

Both `FormValueControl` and `FormCheckboxControl` extend `FormUiControl`, which provides optional input/model signals that the `[formField]` directive will bind if present:

| Property | Type | Direction | Purpose |
|----------|------|-----------|---------|
| `value` / `checked` | `model()` | Two-way | The control's value |
| `touched` | `model()` | Two-way | Has user interacted |
| `dirty` | `input()` | Form to control | Has value changed |
| `disabled` | `input()` | Form to control | Is control disabled |
| `readonly` | `input()` | Form to control | Is control read-only |
| `required` | `input()` | Form to control | Is control required |
| `errors` | `input()` | Form to control | Validation errors |
| `valid` | `input()` | Form to control | Is control valid |
| `invalid` | `input()` | Form to control | Is control invalid |
| `pending` | `input()` | Form to control | Async validation pending |
| `name` | `input()` | Form to control | Field name |

You only declare the signals you actually use. The directive skips any that are absent.

## CSS State Classes

Signal Forms do NOT add the traditional `.ng-valid`, `.ng-invalid`, `.ng-touched`, `.ng-dirty` classes by default. This is a deliberate break from Reactive Forms.

### Restoring Classic Classes

If you need the classic behavior (e.g., for existing CSS):

```typescript
import { provideSignalFormsConfig, NG_STATUS_CLASSES } from '@angular/forms/signals';

bootstrapApplication(AppComponent, {
  providers: [
    provideSignalFormsConfig({
      classes: NG_STATUS_CLASSES,
    }),
  ],
});
```

### Custom Class Configuration

```typescript
provideSignalFormsConfig({
  classes: {
    valid: 'is-valid',
    invalid: 'is-invalid',
    touched: 'is-touched',
    dirty: 'is-dirty',
    disabled: 'is-disabled',
  },
});
```

## Null and Undefined Handling

Signal Forms treat `undefined` as "this field does not exist." Use `null` for empty/cleared values:

```typescript
// BAD: undefined means the field doesn't exist
const model = signal({ name: undefined }); // Type error or runtime issue

// GOOD: use null for empty values
const model = signal<{ name: string | null }>({ name: null });

// GOOD: use empty string for text fields
const model = signal({ name: '' });
```

When mapping domain models to form models, convert `undefined` to defaults:

```typescript
function toFormModel(domain: DomainModel): FormModel {
  return {
    ...domain,
    notes: domain.notes ?? '',
    delay: domain.delay ?? 0,
  };
}

// Use linkedSignal for automatic conversion
protected formModel = linkedSignal(() => toFormModel(this.domainModel()));
```

## Migration from Reactive Forms

Angular provides two strategies for incremental migration.

### Top-Down: `compatForm()`

Wrap existing `FormControl` and `FormGroup` instances inside a signal form:

```typescript
import { compatForm } from '@angular/forms/signals';

// Existing reactive form control with complex async validation
const passwordControl = new FormControl('', {
  validators: [Validators.required, customPasswordValidator()],
  nonNullable: true,
});

// Mix reactive controls into a signal model
const userModel = signal({
  email: '',                // Signal-managed
  password: passwordControl, // Reactive-managed (preserved)
});

// Wrap with compatForm instead of form
const userForm = compatForm(userModel);
```

Signal fields (`email`) use `[formField]`. Reactive fields (`password`) can also use `[formField]`, with the compatibility layer syncing bidirectionally.

### Bottom-Up: `SignalFormControl`

Expose a signal-based control as a standard `FormControl` for use in existing `FormGroup` structures:

```typescript
import { SignalFormControl } from '@angular/forms/signals';

const emailControl = new SignalFormControl('', (p) => {
  required(p, { message: 'Email is required' });
  email(p, { message: 'Invalid email' });
});

// Use in an existing FormGroup
const form = new FormGroup({
  email: emailControl, // Signal-based but compatible
  name: new FormControl(''), // Traditional
});
```

### Migration Restrictions

`SignalFormControl` deliberately blocks imperative methods:

```typescript
// These will throw or be no-ops:
emailControl.addValidators(Validators.required); // NO
emailControl.disable(); // NO
emailControl.setErrors({ custom: true }); // NO

// Instead, use schema rules:
const emailControl = new SignalFormControl('', (p) => {
  required(p);
  disabled(p, () => this.isLocked());
  applyWhen(p, () => this.needsExtra(), (p) => {
    minLength(p, 5);
  });
});
```

## Common Mistakes

### Mistake 1: Using `undefined` in the model

Signal Forms reject `undefined` values. A field with value `undefined` is treated as non-existent.

```typescript
// BAD: undefined causes issues
const model = signal({ name: undefined as string | undefined });

// GOOD: use null or default values
const model = signal({ name: '' });
const model = signal<{ name: string | null }>({ name: null });
```

### Mistake 2: Trying to use imperative state methods

Developers coming from Reactive Forms instinctively reach for `.disable()`, `.enable()`, `.addValidators()`, etc. Signal Forms do not support imperative state changes.

```typescript
// BAD: imperative style
effect(() => {
  if (this.isRequired()) {
    // No way to "add" a validator at runtime
  }
});

// GOOD: declarative conditional validation
const myForm = form(this.model, (path) => {
  applyWhen(path, () => this.isRequired(), (path) => {
    required(path.name);
  });
});
```

### Mistake 3: Importing from `@angular/forms` instead of `@angular/forms/signals`

Signal Forms live in a separate entry point. Mixing imports causes confusing errors.

```typescript
// BAD: wrong import path
import { form, required } from '@angular/forms';

// GOOD: signal forms entry point
import { form, required, FormField } from '@angular/forms/signals';
```

### Mistake 4: Expecting CSS state classes by default

Signal Forms do not add `.ng-valid`, `.ng-invalid`, etc. If your existing CSS relies on these classes, nothing will style.

```typescript
// Fix: opt in via config
provideSignalFormsConfig({ classes: NG_STATUS_CLASSES });
```

### Mistake 5: Mutating the model signal's value directly

Signal updates must be immutable. Mutating the object inside the signal bypasses change detection.

```typescript
// BAD: mutation
this.model().items.push(newItem); // Signal doesn't detect this

// GOOD: immutable update
this.model.update(m => ({
  ...m,
  items: [...m.items, newItem],
}));

// Or update via the field tree
this.myForm.items().value.update(items => [...items, newItem]);
```

### Mistake 6: Using readonly signals as form models

The `form()` function requires a writable signal because it writes user changes back to the model.

```typescript
// BAD: computed signals are readonly
const model = computed(() => this.store.data());
const myForm = form(model); // Error

// GOOD: use linkedSignal to create a writable copy
const model = linkedSignal(() => this.store.data());
const myForm = form(model);
```

## Best Practices

### 1. Keep the model interface separate from UI concerns

```typescript
// Domain model: what the data looks like
interface UserProfile {
  name: string;
  email: string;
  bio: string;
  avatarUrl: string;
}

// The signal holds the domain model directly
profileModel = signal<UserProfile>({
  name: '',
  email: '',
  bio: '',
  avatarUrl: '',
});
```

No `FormControl`, no `FormGroup`, just a plain interface. Your form model and your domain model are the same thing.

### 2. Extract schemas for reuse

```typescript
// schemas/address.schema.ts
export const addressSchema = schema<Address>((path) => {
  required(path.street, { message: 'Street is required' });
  required(path.city, { message: 'City is required' });
  pattern(path.zip, /^\d{5}(-\d{4})?$/, { message: 'Invalid ZIP code' });
});

// Use in multiple forms
const orderForm = form(this.orderModel, (path) => {
  apply(path.shippingAddress, addressSchema);
  apply(path.billingAddress, addressSchema);
});
```

### 3. Use `submit()` for validation-aware submission

```typescript
async save(): Promise<void> {
  await submit(this.myForm, async (formRef) => {
    try {
      await this.api.save(formRef().value());
      this.router.navigate(['/success']);
      return null;
    } catch (err) {
      return { kind: 'server_error', message: 'Save failed. Try again.' };
    }
  });
}
```

### 4. Use `linkedSignal` for store-driven forms

```typescript
// Data comes from a store or service (readonly signal)
private userData = toSignal(this.userService.getUser());

// Create a writable copy for the form
protected formModel = linkedSignal(() => this.userData() ?? defaultUser);

// The form edits the copy, not the original
protected userForm = form(this.formModel, (path) => {
  required(path.name);
  email(path.email);
});

// On save, send the edited model back
save(): void {
  this.userService.updateUser(this.formModel());
}
```

### 5. Prefer `hidden()` over conditionally rendering fields

```typescript
// GOOD: field stays in the form, validation is skipped when hidden
const myForm = form(this.model, (path) => {
  hidden(path.promoCode, () => !this.showPromo());
  required(path.promoCode, { message: 'Promo code required' });
});

// When hidden, the required validator is automatically skipped
```

## Real-World Use Cases

### 1. Multi-Step Registration Wizard

```typescript
interface RegistrationData {
  // Step 1
  email: string;
  password: string;
  // Step 2
  firstName: string;
  lastName: string;
  dateOfBirth: string;
  // Step 3
  company: string;
  role: string;
  acceptTerms: boolean;
}

@Component({
  selector: 'app-registration',
  standalone: true,
  imports: [FormField],
  template: `
    <form (ngSubmit)="onSubmit()">
      @switch (currentStep()) {
        @case (1) {
          <h2>Account</h2>
          <input type="email" [formField]="regForm.email" placeholder="Email" />
          @if (regForm.email().touched() && regForm.email().invalid()) {
            @for (err of regForm.email().errors(); track err) {
              <p class="error">{{ err.message }}</p>
            }
          }
          <input type="password" [formField]="regForm.password" placeholder="Password" />
        }
        @case (2) {
          <h2>Personal Info</h2>
          <input [formField]="regForm.firstName" placeholder="First Name" />
          <input [formField]="regForm.lastName" placeholder="Last Name" />
          <input type="date" [formField]="regForm.dateOfBirth" />
        }
        @case (3) {
          <h2>Professional</h2>
          <input [formField]="regForm.company" placeholder="Company" />
          <input [formField]="regForm.role" placeholder="Role" />
          <label>
            <input type="checkbox" [formField]="regForm.acceptTerms" />
            I accept the terms
          </label>
        }
      }

      <div class="actions">
        @if (currentStep() > 1) {
          <button type="button" (click)="prevStep()">Back</button>
        }
        @if (currentStep() < 3) {
          <button type="button" (click)="nextStep()">Next</button>
        } @else {
          <button type="submit">Register</button>
        }
      </div>
    </form>
  `,
})
export class RegistrationComponent {
  currentStep = signal(1);

  regModel = signal<RegistrationData>({
    email: '', password: '',
    firstName: '', lastName: '', dateOfBirth: '',
    company: '', role: '', acceptTerms: false,
  });

  regForm = form(this.regModel, (path) => {
    // Step 1 validation
    required(path.email, { message: 'Email is required' });
    email(path.email, { message: 'Invalid email' });
    required(path.password, { message: 'Password is required' });
    minLength(path.password, 8, { message: 'At least 8 characters' });

    // Step 2 validation
    required(path.firstName, { message: 'First name required' });
    required(path.lastName, { message: 'Last name required' });

    // Step 3 validation
    validate(path.acceptTerms, ({ value }) =>
      !value() ? { kind: 'terms', message: 'You must accept the terms' } : null
    );
  });

  nextStep(): void {
    this.currentStep.update(s => Math.min(s + 1, 3));
  }

  prevStep(): void {
    this.currentStep.update(s => Math.max(s - 1, 1));
  }

  async onSubmit(): Promise<void> {
    await submit(this.regForm, async (formRef) => {
      await this.authService.register(formRef().value());
      return null;
    });
  }
}
```

### 2. Dynamic Invoice with Computed Totals

```typescript
interface Invoice {
  client: string;
  items: { name: string; qty: number; rate: number }[];
  taxRate: number;
  notes: string;
}

@Component({
  selector: 'app-invoice',
  standalone: true,
  imports: [FormField],
  template: `
    <input [formField]="invoiceForm.client" placeholder="Client" />

    <table>
      <thead>
        <tr>
          <th>Item</th><th>Qty</th><th>Rate</th><th>Subtotal</th><th></th>
        </tr>
      </thead>
      <tbody>
        @for (item of invoiceForm.items; track $index) {
          <tr>
            <td><input [formField]="item.name" /></td>
            <td><input type="number" [formField]="item.qty" /></td>
            <td><input type="number" [formField]="item.rate" /></td>
            <td>{{ item.qty().value() * item.rate().value() | currency }}</td>
            <td><button (click)="removeItem($index)">X</button></td>
          </tr>
        }
      </tbody>
      <tfoot>
        <tr><td colspan="3">Subtotal</td><td>{{ subtotal() | currency }}</td></tr>
        <tr>
          <td colspan="3">
            Tax (<input type="number" [formField]="invoiceForm.taxRate" style="width:4em" />%)
          </td>
          <td>{{ tax() | currency }}</td>
        </tr>
        <tr><td colspan="3"><strong>Total</strong></td><td><strong>{{ total() | currency }}</strong></td></tr>
      </tfoot>
    </table>

    <button type="button" (click)="addItem()">Add Line Item</button>
    <textarea [formField]="invoiceForm.notes" placeholder="Notes"></textarea>
  `,
})
export class InvoiceComponent {
  invoiceModel = signal<Invoice>({
    client: '',
    items: [{ name: '', qty: 1, rate: 0 }],
    taxRate: 10,
    notes: '',
  });

  invoiceForm = form(this.invoiceModel, (path) => {
    required(path.client, { message: 'Client name required' });
    applyEach(path.items, (item) => {
      required(item.name, { message: 'Item name required' });
      min(item.qty, 1, { message: 'Qty must be at least 1' });
      min(item.rate, 0, { message: 'Rate cannot be negative' });
    });
  });

  // Computed values using signals
  subtotal = computed(() => {
    const data = this.invoiceModel();
    return data.items.reduce((sum, item) => sum + item.qty * item.rate, 0);
  });

  tax = computed(() => this.subtotal() * (this.invoiceModel().taxRate / 100));
  total = computed(() => this.subtotal() + this.tax());

  addItem(): void {
    this.invoiceForm.items().value.update(items => [
      ...items,
      { name: '', qty: 1, rate: 0 },
    ]);
  }

  removeItem(index: number): void {
    this.invoiceForm.items().value.update(items =>
      items.filter((_, i) => i !== index)
    );
  }
}
```

### 3. Settings Form with Conditional Sections

```typescript
interface AppSettings {
  theme: 'light' | 'dark' | 'system';
  notifications: {
    enabled: boolean;
    email: boolean;
    push: boolean;
    digest: 'daily' | 'weekly' | 'never';
  };
  privacy: {
    profilePublic: boolean;
    showEmail: boolean;
  };
}

@Component({
  selector: 'app-settings',
  standalone: true,
  imports: [FormField],
  template: `
    <h2>Appearance</h2>
    <select [formField]="settingsForm.theme">
      <option value="light">Light</option>
      <option value="dark">Dark</option>
      <option value="system">System</option>
    </select>

    <h2>Notifications</h2>
    <label>
      <input type="checkbox" [formField]="settingsForm.notifications.enabled" />
      Enable notifications
    </label>

    <!-- These fields are disabled when notifications are off -->
    <label>
      <input type="checkbox" [formField]="settingsForm.notifications.email" />
      Email notifications
    </label>
    <label>
      <input type="checkbox" [formField]="settingsForm.notifications.push" />
      Push notifications
    </label>
    <select [formField]="settingsForm.notifications.digest">
      <option value="daily">Daily digest</option>
      <option value="weekly">Weekly digest</option>
      <option value="never">No digest</option>
    </select>

    <h2>Privacy</h2>
    <label>
      <input type="checkbox" [formField]="settingsForm.privacy.profilePublic" />
      Public profile
    </label>
    <label>
      <input type="checkbox" [formField]="settingsForm.privacy.showEmail" />
      Show email on profile
    </label>

    <button (click)="save()">Save Settings</button>
  `,
})
export class SettingsComponent {
  settingsModel = signal<AppSettings>({
    theme: 'system',
    notifications: {
      enabled: true,
      email: true,
      push: false,
      digest: 'weekly',
    },
    privacy: {
      profilePublic: true,
      showEmail: false,
    },
  });

  settingsForm = form(this.settingsModel, (path) => {
    // Disable notification sub-fields when notifications are off
    disabled(path.notifications.email, (ctx) =>
      !ctx.valueOf(path.notifications.enabled) ? 'Notifications disabled' : false
    );
    disabled(path.notifications.push, (ctx) =>
      !ctx.valueOf(path.notifications.enabled) ? 'Notifications disabled' : false
    );
    disabled(path.notifications.digest, (ctx) =>
      !ctx.valueOf(path.notifications.enabled) ? 'Notifications disabled' : false
    );

    // Show email toggle only when profile is public
    disabled(path.privacy.showEmail, (ctx) =>
      !ctx.valueOf(path.privacy.profilePublic) ? 'Profile is private' : false
    );
  });

  save(): void {
    console.log('Settings:', this.settingsModel());
  }
}
```

> **Key Takeaway:** Signal Forms replace the `FormControl`/`FormGroup` tree with a plain signal, replace per-control validators with a centralized schema, and replace `ControlValueAccessor` with a single `value` model signal, resulting in dramatically less boilerplate while gaining full type safety and fine-grained signal reactivity.
