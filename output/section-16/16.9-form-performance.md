# 16.9 Form Performance

> Understanding how Angular forms trigger change detection, validation, and DOM updates, and the techniques to keep large or complex forms fast.

---

## How Forms Impact Performance

Every keystroke in an Angular form triggers a cascade of operations:

1. The DOM `input` event fires
2. The form directive reads the new value
3. The `FormControl` updates its internal value
4. `valueChanges` emits
5. All synchronous validators run
6. `statusChanges` emits
7. The parent `FormGroup` recalculates its value and status (recursively to the root)
8. Change detection runs on the component (and potentially the entire tree)
9. The template re-evaluates bindings

For a simple login form, this is invisible. For a form with 200 controls, nested groups, complex validators, and computed displays, each keystroke can cause hundreds of milliseconds of work.

---

## The updateOn Option

The single most impactful optimization for form performance is controlling when value updates and validation occur.

### Default: updateOn: 'change'

Every keystroke triggers the full update cycle:

```typescript
// Default behavior: updates on every input event
const name = new FormControl('', {
  validators: [Validators.required, Validators.minLength(3)],
  // updateOn: 'change' is implied
});
```

### updateOn: 'blur'

Updates and validation happen only when the user leaves the field:

```typescript
const name = new FormControl('', {
  validators: [Validators.required, Validators.minLength(3)],
  updateOn: 'blur',
});
```

This eliminates validation on every keystroke. For a field with an async validator that calls an API, this is the difference between dozens of HTTP requests and one.

### updateOn: 'submit'

Updates happen only when the form is submitted:

```typescript
const form = new FormGroup({
  name: new FormControl(''),
  email: new FormControl(''),
  bio: new FormControl(''),
}, { updateOn: 'submit' });
```

Setting `updateOn` on the `FormGroup` applies it to all children. Individual controls can override with their own `updateOn`.

### Choosing the Right Strategy

| Scenario | Recommended `updateOn` | Why |
|----------|----------------------|-----|
| Real-time search/filter | `'change'` with debounce | User expects instant feedback |
| Standard text inputs | `'blur'` | Validate once, not per keystroke |
| Large forms (10+ fields) | `'blur'` or `'submit'` | Reduce total validation runs |
| Async validators (API calls) | `'blur'` minimum | Prevent request flooding |
| Simple checkbox/toggle | `'change'` | Single event, no performance concern |

### Template-Driven Equivalent

```html
<input
  [(ngModel)]="user.email"
  name="email"
  required
  email
  [ngModelOptions]="{ updateOn: 'blur' }" />
```

## Debouncing valueChanges

When you need `updateOn: 'change'` for UX reasons (like a live search), debounce the processing:

```typescript
@Component({
  selector: 'app-search',
  standalone: true,
  imports: [ReactiveFormsModule],
  template: `
    <input [formControl]="searchControl" placeholder="Search..." />
    @for (result of results(); track result.id) {
      <div>{{ result.name }}</div>
    }
  `,
})
export class SearchComponent implements OnInit, OnDestroy {
  searchControl = new FormControl('');
  results = signal<SearchResult[]>([]);
  private destroy$ = new Subject<void>();

  private searchService = inject(SearchService);

  ngOnInit(): void {
    this.searchControl.valueChanges
      .pipe(
        debounceTime(300),       // Wait 300ms after last keystroke
        distinctUntilChanged(),  // Skip if value hasn't actually changed
        filter(term => term !== null && term.length >= 2), // Min length
        switchMap(term => this.searchService.search(term!)),
        takeUntil(this.destroy$),
      )
      .subscribe(results => this.results.set(results));
  }

  ngOnDestroy(): void {
    this.destroy$.next();
  }
}
```

Key operators:
- `debounceTime(300)`: Wait for the user to stop typing
- `distinctUntilChanged()`: Skip duplicate values (e.g., type "ab", delete "b", type "b" again)
- `switchMap()`: Cancel the previous request if a new one arrives

## OnPush Change Detection

By default, Angular uses the `Default` change detection strategy, which checks every component in the tree on every event. With forms, this means every keystroke triggers change detection across the entire application.

### Applying OnPush

```typescript
@Component({
  selector: 'app-user-form',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  imports: [ReactiveFormsModule],
  template: `
    <form [formGroup]="form">
      <input formControlName="name" />
      <input formControlName="email" />
    </form>
  `,
})
export class UserFormComponent {
  form = inject(FormBuilder).group({
    name: [''],
    email: [''],
  });
}
```

With `OnPush`, the component only re-renders when:
- An `@Input()` reference changes
- An event originates from the component or its children
- An `Observable` bound with `async` pipe emits
- `ChangeDetectorRef.markForCheck()` is called explicitly

Reactive Forms work well with `OnPush` because form directives trigger events from within the component.

### The FormArray + OnPush Gotcha

When you programmatically modify a `FormArray` (push, remove, clear), the component's view may not update because no DOM event occurred:

```typescript
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,
  // ...
})
export class DynamicListComponent {
  private cdr = inject(ChangeDetectorRef);
  formArray = new FormArray<FormControl<string>>([]);

  addItem(): void {
    this.formArray.push(new FormControl(''));
    // Without this, the new control won't appear in the template
    this.cdr.markForCheck();
  }

  removeItem(index: number): void {
    this.formArray.removeAt(index);
    this.cdr.markForCheck();
  }
}
```

Why? The `formArray` reference itself does not change when you push or remove. Angular's `OnPush` check sees the same object reference and skips re-rendering.

With signals, this problem disappears because signal updates always notify the template:

```typescript
// Signal-based alternative: no markForCheck needed
items = signal<string[]>(['']);

addItem(): void {
  this.items.update(list => [...list, '']);
}
```

## emitEvent and onlySelf Options

Every `setValue`, `patchValue`, `disable`, `enable`, `markAsTouched`, and `markAsDirty` call propagates changes up the form tree and emits observables. For batch operations, this creates redundant work.

### Suppressing Events During Batch Updates

```typescript
// BAD: each call triggers valueChanges, statusChanges, and parent recalculation
this.form.get('name')!.setValue('Alice');
this.form.get('email')!.setValue('alice@example.com');
this.form.get('age')!.setValue(30);
// Result: 3 valueChanges emissions, 3 status recalculations

// GOOD: suppress events, then trigger once
this.form.get('name')!.setValue('Alice', { emitEvent: false });
this.form.get('email')!.setValue('alice@example.com', { emitEvent: false });
this.form.get('age')!.setValue(30); // Only this one emits
// Result: 1 valueChanges emission
```

Or better, use `patchValue` on the group:

```typescript
// BEST: single operation, single emission
this.form.patchValue({
  name: 'Alice',
  email: 'alice@example.com',
  age: 30,
});
```

### The onlySelf Option

By default, when a control's value or status changes, Angular recalculates the parent group, which recalculates its parent, all the way to the root. `onlySelf: true` stops this propagation:

```typescript
// Only update this control, don't recalculate parent groups
control.setValue('new value', { onlySelf: true });
control.markAsTouched({ onlySelf: true });
control.updateValueAndValidity({ onlySelf: true });
```

Use this when updating multiple controls in a loop, then manually trigger the parent update once:

```typescript
// Update 100 controls without 100 parent recalculations
for (const control of this.formArray.controls) {
  control.setValue(getDefaultValue(), { onlySelf: true, emitEvent: false });
}
// Trigger one recalculation at the end
this.formArray.updateValueAndValidity();
```

## Large FormArray Performance

FormArrays with dozens or hundreds of entries are one of the most common performance bottlenecks in Angular forms.

### Problem: DOM Bloat

Each `FormControl` bound to an input element means a DOM element with event listeners, Angular directives, and change detection bindings. 200 rows with 5 columns = 1000 active form controls.

### Strategy 1: Virtual Scrolling

Render only the visible rows:

```typescript
import { CdkVirtualScrollViewport, CdkVirtualFor } from '@angular/cdk/scrolling';

@Component({
  selector: 'app-large-table',
  standalone: true,
  imports: [ReactiveFormsModule, CdkVirtualScrollViewport, CdkVirtualFor],
  template: `
    <cdk-virtual-scroll-viewport itemSize="48" class="table-viewport">
      <div
        *cdkVirtualFor="let control of formArray.controls; trackBy: trackByIndex"
        class="row">
        <input [formControl]="control" />
      </div>
    </cdk-virtual-scroll-viewport>
  `,
  styles: [`
    .table-viewport { height: 500px; }
    .row { height: 48px; }
  `],
})
export class LargeTableComponent {
  formArray = new FormArray(
    Array.from({ length: 1000 }, () => new FormControl(''))
  );

  trackByIndex(index: number): number {
    return index;
  }
}
```

Virtual scrolling renders only ~15 rows at a time instead of 1000, reducing DOM nodes by ~98%.

### Strategy 2: Paginated Editing

Instead of rendering all rows, paginate:

```typescript
@Component({
  selector: 'app-paginated-form',
  standalone: true,
  imports: [ReactiveFormsModule],
  template: `
    @for (item of visibleItems(); track item.index) {
      <div class="row">
        <input [formControl]="item.control" />
      </div>
    }
    <div class="pagination">
      <button (click)="prevPage()" [disabled]="page() === 0">Previous</button>
      <span>Page {{ page() + 1 }} of {{ totalPages() }}</span>
      <button (click)="nextPage()" [disabled]="page() >= totalPages() - 1">Next</button>
    </div>
  `,
})
export class PaginatedFormComponent {
  pageSize = 20;
  page = signal(0);

  formArray = new FormArray(
    Array.from({ length: 500 }, (_, i) => new FormControl(`Item ${i}`))
  );

  totalPages = computed(() =>
    Math.ceil(this.formArray.length / this.pageSize)
  );

  visibleItems = computed(() => {
    const start = this.page() * this.pageSize;
    return this.formArray.controls
      .slice(start, start + this.pageSize)
      .map((control, i) => ({ control, index: start + i }));
  });

  nextPage(): void {
    this.page.update(p => p + 1);
  }

  prevPage(): void {
    this.page.update(p => p - 1);
  }
}
```

### Strategy 3: Lazy Control Creation

Only create `FormControl` instances when the user interacts with a row:

```typescript
@Component({
  selector: 'app-lazy-form',
  standalone: true,
  imports: [ReactiveFormsModule, FormsModule],
  template: `
    @for (item of items(); track item.id) {
      <div class="row">
        @if (item.editing) {
          <!-- Full FormControl only when editing -->
          <input [formControl]="getControl(item.id)" />
          <button (click)="save(item)">Save</button>
        } @else {
          <!-- Static display: no form control overhead -->
          <span (dblclick)="edit(item)">{{ item.value }}</span>
        }
      </div>
    }
  `,
})
export class LazyFormComponent {
  items = signal<{ id: number; value: string; editing: boolean }[]>(
    Array.from({ length: 500 }, (_, i) => ({
      id: i,
      value: `Item ${i}`,
      editing: false,
    }))
  );

  private controls = new Map<number, FormControl<string>>();

  getControl(id: number): FormControl<string> {
    if (!this.controls.has(id)) {
      const item = this.items().find(i => i.id === id);
      this.controls.set(id, new FormControl(item?.value ?? '', { nonNullable: true }));
    }
    return this.controls.get(id)!;
  }

  edit(item: { id: number; value: string; editing: boolean }): void {
    this.items.update(items =>
      items.map(i => ({ ...i, editing: i.id === item.id }))
    );
  }

  save(item: { id: number; value: string; editing: boolean }): void {
    const control = this.controls.get(item.id);
    if (control) {
      this.items.update(items =>
        items.map(i =>
          i.id === item.id
            ? { ...i, value: control.value, editing: false }
            : i
        )
      );
      this.controls.delete(item.id);
    }
  }
}
```

This keeps the active `FormControl` count low (only 1 at a time) while displaying hundreds of rows.

## Validator Performance

### Expensive Validators

Custom validators run on every value change by default. If a validator does heavy computation, it slows every keystroke:

```typescript
// BAD: expensive validation on every keystroke
function expensiveValidator(control: AbstractControl): ValidationErrors | null {
  const value = control.value;
  // Imagine this parses a large dataset or does complex regex
  const result = heavyComputation(value);
  return result.valid ? null : { expensive: true };
}

// Applied with default updateOn: 'change'
const control = new FormControl('', [expensiveValidator]); // Runs on every key
```

Solutions:

```typescript
// Solution 1: Move to blur
const control = new FormControl('', {
  validators: [expensiveValidator],
  updateOn: 'blur',
});

// Solution 2: Make it an async validator with debounce
function debouncedExpensiveValidator(): AsyncValidatorFn {
  return (control: AbstractControl): Observable<ValidationErrors | null> => {
    return timer(300).pipe(
      map(() => {
        const result = heavyComputation(control.value);
        return result.valid ? null : { expensive: true };
      })
    );
  };
}
```

### Validator Composition Cost

Each validator in the array runs independently. If you have 5 validators on a single control and the first one already fails, the other 4 still execute:

```typescript
// All 5 validators run even if 'required' fails
const control = new FormControl('', [
  Validators.required,    // Fails for empty string
  Validators.minLength(3), // Also runs
  Validators.pattern(/^[a-z]+$/), // Also runs
  customValidator1,        // Also runs
  customValidator2,        // Also runs
]);
```

If your validators are cheap (built-in validators are), this is not a problem. If they are expensive, combine them into a single validator that short-circuits:

```typescript
function combinedValidator(control: AbstractControl): ValidationErrors | null {
  const value = control.value;

  // Short-circuit: if empty, skip expensive checks
  if (!value) {
    return { required: true };
  }

  if (value.length < 3) {
    return { minlength: { requiredLength: 3, actualLength: value.length } };
  }

  // Only run expensive check if basic checks pass
  return expensiveCheck(value);
}
```

### Async Validator Timing

Async validators only run after all synchronous validators pass. This is already an optimization. But multiple async validators on the same control all fire simultaneously:

```typescript
const control = new FormControl('', {
  asyncValidators: [
    checkUsernameAvailability,   // API call 1
    checkUsernameNotBanned,      // API call 2
    checkUsernameNotReserved,    // API call 3
  ],
  updateOn: 'blur',
});
// All 3 API calls fire at once on blur
```

Combine them into a single async validator to reduce network requests:

```typescript
function checkUsername(http: HttpClient): AsyncValidatorFn {
  return (control: AbstractControl): Observable<ValidationErrors | null> => {
    // Single API call that checks everything server-side
    return http.get<UsernameCheck>(`/api/check-username/${control.value}`).pipe(
      map(result => {
        if (result.taken) return { usernameTaken: true };
        if (result.banned) return { usernameBanned: true };
        if (result.reserved) return { usernameReserved: true };
        return null;
      }),
      catchError(() => of(null)),
    );
  };
}
```

## Splitting Forms into Sub-Components

Large forms in a single component create a massive template that Angular must check on every change detection cycle. Breaking the form into sub-components with `OnPush` isolates change detection.

### Parent Component

```typescript
@Component({
  selector: 'app-order-form',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  imports: [ReactiveFormsModule, CustomerFormComponent, ShippingFormComponent, ItemsFormComponent],
  template: `
    <form [formGroup]="form" (ngSubmit)="submit()">
      <app-customer-form [group]="customerGroup" />
      <app-shipping-form [group]="shippingGroup" />
      <app-items-form [array]="itemsArray" />
      <button type="submit">Submit</button>
    </form>
  `,
})
export class OrderFormComponent {
  form = inject(FormBuilder).group({
    customer: inject(FormBuilder).group({
      name: ['', Validators.required],
      email: ['', [Validators.required, Validators.email]],
      phone: [''],
    }),
    shipping: inject(FormBuilder).group({
      street: ['', Validators.required],
      city: ['', Validators.required],
      state: [''],
      zip: ['', Validators.required],
    }),
    items: inject(FormBuilder).array<FormGroup>([]),
  });

  get customerGroup(): FormGroup {
    return this.form.get('customer') as FormGroup;
  }

  get shippingGroup(): FormGroup {
    return this.form.get('shipping') as FormGroup;
  }

  get itemsArray(): FormArray {
    return this.form.get('items') as FormArray;
  }

  submit(): void {
    if (this.form.valid) {
      console.log(this.form.getRawValue());
    }
  }
}
```

### Child Component

```typescript
@Component({
  selector: 'app-customer-form',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  imports: [ReactiveFormsModule],
  template: `
    <fieldset [formGroup]="group()">
      <legend>Customer</legend>
      <input formControlName="name" placeholder="Name" />
      <input formControlName="email" placeholder="Email" />
      <input formControlName="phone" placeholder="Phone" />
    </fieldset>
  `,
})
export class CustomerFormComponent {
  group = input.required<FormGroup>();
}
```

When the user types in the "Name" field inside `CustomerFormComponent`, change detection runs only inside that component (and its ancestors up the tree if needed), not in `ShippingFormComponent` or `ItemsFormComponent`.

## Signal Forms Performance Advantages

Signal Forms (Angular 21) address several performance issues inherently:

### Fine-Grained Reactivity

Signal-based updates only re-render the specific template binding that changed, not the entire component:

```typescript
// Each field is an independent signal chain
// Changing email() does NOT cause name() to re-evaluate
this.form.email().value.set('new@email.com');
// Only the template binding for email updates
```

### No Observable Overhead

Reactive Forms create `valueChanges` and `statusChanges` observables for every control, group, and array. Each observable maintains a subscriber list and emits on changes. Signal Forms replace this with signals, which have lower memory and CPU overhead.

### Declarative State

No imperative `markForCheck()` calls needed. Signals naturally notify the view:

```typescript
// Signal Forms: no ChangeDetectorRef needed
items = signal<Item[]>([]);

addItem(): void {
  this.items.update(list => [...list, newItem]);
  // Template updates automatically
}
```

### Automatic Hidden Field Optimization

Fields marked as `hidden()` in the schema skip validation entirely:

```typescript
const myForm = form(this.model, (path) => {
  hidden(path.promoCode, () => !this.showPromo());
  required(path.promoCode); // Skipped when hidden
});
```

No manual `clearValidators()` / `setValidators()` dance.

## Common Mistakes

### Mistake 1: Subscribing to valueChanges without debouncing

```typescript
// BAD: fires on every keystroke, triggers expensive work
this.form.get('search')!.valueChanges.subscribe(value => {
  this.results = this.service.search(value); // API call per keystroke
});

// GOOD: debounce and cancel previous
this.form.get('search')!.valueChanges.pipe(
  debounceTime(300),
  distinctUntilChanged(),
  switchMap(value => this.service.search(value)),
).subscribe(results => this.results = results);
```

### Mistake 2: Using Default change detection with large forms

```typescript
// BAD: every keystroke checks the entire component tree
@Component({
  // No changeDetection specified = ChangeDetectionStrategy.Default
  template: `
    <!-- 50 form fields, computed values, conditional displays -->
  `,
})
export class HugeFormComponent { }

// GOOD: isolate with OnPush
@Component({
  changeDetection: ChangeDetectionStrategy.OnPush,
  // ...
})
export class HugeFormComponent { }
```

### Mistake 3: Rebuilding FormArray controls on every change

```typescript
// BAD: destroys and recreates all controls
updateItems(items: Item[]): void {
  this.formArray.clear();
  items.forEach(item => {
    this.formArray.push(new FormGroup({
      name: new FormControl(item.name),
      price: new FormControl(item.price),
    }));
  });
}

// GOOD: update existing controls, add/remove as needed
updateItems(items: Item[]): void {
  // Remove excess controls
  while (this.formArray.length > items.length) {
    this.formArray.removeAt(this.formArray.length - 1);
  }
  // Update existing and add new
  items.forEach((item, i) => {
    if (i < this.formArray.length) {
      this.formArray.at(i).patchValue(item, { emitEvent: false });
    } else {
      this.formArray.push(new FormGroup({
        name: new FormControl(item.name),
        price: new FormControl(item.price),
      }));
    }
  });
  this.formArray.updateValueAndValidity();
}
```

### Mistake 4: Forgetting emitEvent: false in loops

```typescript
// BAD: 100 valueChanges emissions + 100 parent recalculations
for (const control of this.formArray.controls) {
  control.setValue(defaultValue);
}

// GOOD: suppress events in the loop, emit once at the end
for (const control of this.formArray.controls) {
  control.setValue(defaultValue, { emitEvent: false, onlySelf: true });
}
this.formArray.updateValueAndValidity();
```

### Mistake 5: Complex template expressions bound to form state

```typescript
// BAD: called on every change detection cycle
get totalPrice(): number {
  return this.formArray.controls.reduce((sum, group) => {
    const qty = group.get('quantity')!.value;
    const price = group.get('price')!.value;
    return sum + (qty * price);
  }, 0);
}

// GOOD: compute reactively with a signal or cache the result
total = signal(0);

ngOnInit(): void {
  this.formArray.valueChanges.pipe(
    debounceTime(100),
    takeUntilDestroyed(this.destroyRef),
  ).subscribe(values => {
    this.total.set(
      values.reduce((sum: number, item: any) =>
        sum + (item.quantity * item.price), 0)
    );
  });
}
```

## Best Practices

### 1. Default to updateOn: 'blur' for text inputs

Unless you need real-time validation feedback, `'blur'` eliminates the majority of unnecessary validation runs.

### 2. Use OnPush on every form component

There is rarely a reason for a form component to use the `Default` strategy. Form directives already trigger change detection when needed.

### 3. Break large forms into sub-components

Each sub-component with `OnPush` creates a change detection boundary. Typing in one section does not re-render other sections.

### 4. Use emitEvent: false for programmatic batch updates

When setting multiple controls programmatically, suppress intermediate events and trigger once at the end.

### 5. Combine multiple async validators into one

Each async validator is a separate asynchronous operation (usually an HTTP call). Consolidate them server-side into a single endpoint.

### 6. Use trackBy (or track in @for) with FormArrays

Prevent Angular from destroying and recreating DOM elements when the array changes:

```html
<!-- @for with track prevents full re-render -->
@for (control of formArray.controls; track $index) {
  <input [formControl]="control" />
}
```

### 7. Virtual scroll for large lists

If you have more than 50 rows of form controls, use CDK virtual scrolling to render only what is visible.

## Real-World Use Cases

### 1. Spreadsheet-Style Editable Table

A table with 200 rows and 5 editable columns, optimized with lazy control creation and virtual scrolling:

```typescript
@Component({
  selector: 'app-data-grid',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  imports: [ReactiveFormsModule, CdkVirtualScrollViewport, CdkVirtualFor],
  template: `
    <cdk-virtual-scroll-viewport itemSize="40" class="grid-viewport">
      <div
        *cdkVirtualFor="let row of rows(); trackBy: trackById"
        class="grid-row">
        @for (col of columns; track col.key) {
          <input
            [value]="row[col.key]"
            (blur)="onCellBlur(row.id, col.key, $event)"
            class="grid-cell" />
        }
      </div>
    </cdk-virtual-scroll-viewport>
  `,
  styles: [`.grid-viewport { height: 600px; } .grid-row { height: 40px; display: flex; }`],
})
export class DataGridComponent {
  rows = signal<Record<string, any>[]>([]);
  columns = [
    { key: 'name', label: 'Name' },
    { key: 'email', label: 'Email' },
    { key: 'department', label: 'Department' },
    { key: 'role', label: 'Role' },
    { key: 'salary', label: 'Salary' },
  ];

  // No FormControls at all. Use plain values + blur event.
  onCellBlur(rowId: number, colKey: string, event: Event): void {
    const newValue = (event.target as HTMLInputElement).value;
    this.rows.update(rows =>
      rows.map(row =>
        row['id'] === rowId ? { ...row, [colKey]: newValue } : row
      )
    );
  }

  trackById(_: number, row: Record<string, any>): number {
    return row['id'];
  }
}
```

For truly large datasets, skipping `FormControl` entirely and using plain DOM values with blur events is the fastest approach. You trade Angular form features (validation, dirty tracking) for raw speed.

### 2. Multi-Step Form with Lazy Validation

Only validate the current step, not the entire form:

```typescript
@Component({
  selector: 'app-wizard',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  imports: [ReactiveFormsModule],
  template: `
    @switch (step()) {
      @case (0) {
        <app-step-personal [group]="personalGroup" />
      }
      @case (1) {
        <app-step-address [group]="addressGroup" />
      }
      @case (2) {
        <app-step-payment [group]="paymentGroup" />
      }
    }
    <button (click)="next()" [disabled]="!isCurrentStepValid()">Next</button>
  `,
})
export class WizardComponent {
  step = signal(0);

  form = inject(FormBuilder).group({
    personal: inject(FormBuilder).group({
      name: ['', Validators.required],
      email: ['', [Validators.required, Validators.email]],
    }),
    address: inject(FormBuilder).group({
      street: ['', Validators.required],
      city: ['', Validators.required],
    }),
    payment: inject(FormBuilder).group({
      cardNumber: ['', Validators.required],
      expiry: ['', Validators.required],
    }),
  });

  get personalGroup(): FormGroup {
    return this.form.get('personal') as FormGroup;
  }

  get addressGroup(): FormGroup {
    return this.form.get('address') as FormGroup;
  }

  get paymentGroup(): FormGroup {
    return this.form.get('payment') as FormGroup;
  }

  private stepGroups = ['personal', 'address', 'payment'];

  isCurrentStepValid(): boolean {
    const groupName = this.stepGroups[this.step()];
    return this.form.get(groupName)!.valid;
  }

  next(): void {
    const currentGroup = this.form.get(this.stepGroups[this.step()])!;
    currentGroup.markAllAsTouched();
    if (currentGroup.valid) {
      this.step.update(s => s + 1);
    }
  }
}
```

Only the current step's sub-component is rendered and checked. Previous and future steps are not in the DOM.

### 3. Real-Time Filter Form with Debounced API Calls

```typescript
@Component({
  selector: 'app-product-filter',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  imports: [ReactiveFormsModule],
  template: `
    <div class="filters" [formGroup]="filters">
      <input formControlName="search" placeholder="Search products..." />
      <select formControlName="category">
        <option value="">All Categories</option>
        @for (cat of categories; track cat) {
          <option [value]="cat">{{ cat }}</option>
        }
      </select>
      <input type="number" formControlName="minPrice" placeholder="Min Price" />
      <input type="number" formControlName="maxPrice" placeholder="Max Price" />
    </div>
    <div class="results">
      @if (loading()) {
        <p>Loading...</p>
      }
      @for (product of products(); track product.id) {
        <app-product-card [product]="product" />
      }
    </div>
  `,
})
export class ProductFilterComponent implements OnInit {
  private destroyRef = inject(DestroyRef);
  private productService = inject(ProductService);

  categories = ['Electronics', 'Clothing', 'Books', 'Home'];
  products = signal<Product[]>([]);
  loading = signal(false);

  filters = inject(FormBuilder).group({
    search: [''],
    category: [''],
    minPrice: [null as number | null],
    maxPrice: [null as number | null],
  });

  ngOnInit(): void {
    this.filters.valueChanges
      .pipe(
        debounceTime(400),         // Wait for user to finish adjusting filters
        distinctUntilChanged(      // Deep compare to skip no-ops
          (prev, curr) => JSON.stringify(prev) === JSON.stringify(curr)
        ),
        tap(() => this.loading.set(true)),
        switchMap(filters =>
          this.productService.search(filters).pipe(
            catchError(() => of([])),
          )
        ),
        takeUntilDestroyed(this.destroyRef),
      )
      .subscribe(results => {
        this.products.set(results);
        this.loading.set(false);
      });
  }
}
```

The `debounceTime(400)` ensures that adjusting multiple filters in quick succession (search + category + price range) results in a single API call, not three.

---

> **Key Takeaway:** The biggest form performance wins come from three changes: setting `updateOn: 'blur'` to reduce validation frequency, using `OnPush` change detection on form components to limit re-rendering, and using `emitEvent: false` with `onlySelf: true` for batch programmatic updates.
