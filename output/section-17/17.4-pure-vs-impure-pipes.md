# 17.4 Pure vs Impure Pipes

> The `pure` flag on a pipe controls when Angular re-executes the `transform()` method. Understanding this distinction is critical for performance, correctness, and choosing the right approach for data transformation in templates.

---

## What Makes a Pipe Pure or Impure

Every pipe has a `pure` property in its `@Pipe` decorator. It defaults to `true`.

```typescript
// Pure pipe (default)
@Pipe({ name: 'double', standalone: true })
export class DoublePipe implements PipeTransform {
  transform(value: number): number {
    return value * 2;
  }
}

// Impure pipe
@Pipe({ name: 'filter', standalone: true, pure: false })
export class FilterPipe implements PipeTransform {
  transform<T>(items: T[], predicate: (item: T) => boolean): T[] {
    return items.filter(predicate);
  }
}
```

The difference is entirely about **when Angular calls `transform()`**, not about what the pipe does internally.

---

## Pure Pipes: How They Work

A pure pipe's `transform()` method is called **only when Angular detects a change to the input value or parameters**. The change detection rules depend on the input type.

### Primitive Values

For primitive types (`string`, `number`, `boolean`), Angular uses value comparison:

```typescript
@Pipe({ name: 'double', standalone: true })
export class DoublePipe implements PipeTransform {
  transform(value: number): number {
    console.log('transform called');
    return value * 2;
  }
}
```

```typescript
@Component({
  standalone: true,
  imports: [DoublePipe],
  template: `
    <p>{{ counter() | double }}</p>
    <button (click)="increment()">+1</button>
    <button (click)="noop()">No-op</button>
  `
})
export class CounterComponent {
  counter = signal(1);

  increment() {
    this.counter.update(c => c + 1);
  }

  noop() {
    // Triggers change detection but counter value unchanged
  }
}
```

- Clicking "+1" changes the value: `transform()` runs.
- Clicking "No-op" triggers change detection but the value is the same: `transform()` does NOT run.

### Object and Array References

For objects and arrays, Angular uses **reference equality** (`===`). It does not perform deep comparison.

```typescript
@Pipe({ name: 'sortBy', standalone: true })
export class SortByPipe implements PipeTransform {
  transform<T>(items: T[], key: keyof T): T[] {
    console.log('sortBy transform called');
    return [...items].sort((a, b) => {
      if (a[key] < b[key]) return -1;
      if (a[key] > b[key]) return 1;
      return 0;
    });
  }
}
```

```typescript
@Component({
  standalone: true,
  imports: [SortByPipe],
  template: `
    @for (user of users() | sortBy:'name'; track user.id) {
      <div>{{ user.name }}</div>
    }
  `
})
export class UserListComponent {
  users = signal<User[]>([
    { id: 1, name: 'Charlie' },
    { id: 2, name: 'Alice' },
    { id: 3, name: 'Bob' }
  ]);

  addUser(user: User) {
    // BAD: Mutating the array. Pure pipe will NOT detect this.
    // this.users().push(user);

    // GOOD: Creating a new array reference. Pure pipe WILL detect this.
    this.users.update(current => [...current, user]);
  }
}
```

This is the single most important concept about pure pipes: **mutations are invisible, only new references trigger re-evaluation**.

### Parameter Changes Also Trigger Re-evaluation

Pure pipes also re-run when any parameter changes:

```html
{{ items | slice:start():end() }}
```

If either `start()` or `end()` changes, the pipe re-evaluates even if `items` has the same reference.

### Angular's Internal Caching

Angular stores the last input value, parameter values, and the last output for each pure pipe binding. On each change detection cycle, it compares the current inputs against the cached ones:

```
// Pseudocode of Angular's pure pipe check
if (currentValue !== cachedValue ||
    currentParam1 !== cachedParam1 ||
    currentParam2 !== cachedParam2) {
  cachedOutput = pipe.transform(currentValue, currentParam1, currentParam2);
  cachedValue = currentValue;
  cachedParam1 = currentParam1;
  cachedParam2 = currentParam2;
}
return cachedOutput;
```

Internally, Angular uses `Object.is()` for the comparison (via the `bindingUpdated` function). This means `NaN === NaN` returns `true` (unlike `===`), and `+0` and `-0` are treated as different values.

This means:
- Pure pipes perform a fast `Object.is()` check per change detection cycle (cheap).
- The actual `transform()` call is skipped when inputs have not changed (the expensive part is avoided).
- Each template binding gets its own cached state stored in the component's LView data structure.

## Impure Pipes: How They Work

An impure pipe's `transform()` is called on **every change detection cycle**, regardless of whether the inputs changed.

```typescript
@Pipe({ name: 'debugLog', standalone: true, pure: false })
export class DebugLogPipe implements PipeTransform {
  private callCount = 0;

  transform(value: unknown): unknown {
    this.callCount++;
    console.log(`transform called ${this.callCount} times, value:`, value);
    return value;
  }
}
```

```html
{{ someValue | debugLog }}
```

Every time Angular runs change detection (which can be triggered by any event, timer, HTTP response, or signal change anywhere in the application), this pipe runs. In a typical application, this could mean dozens or hundreds of calls per second during user interaction.

### When Impure Pipes Run

```typescript
@Component({
  standalone: true,
  imports: [DebugLogPipe],
  template: `
    <p>{{ data | debugLog }}</p>
    <button (click)="onClick()">Click</button>
    <input (input)="onInput($event)">
  `
})
export class DemoComponent {
  data = 'hello';

  onClick() {
    // Even if data doesn't change, the impure pipe runs
    // because change detection runs
  }

  onInput(event: Event) {
    // Every keystroke triggers change detection
    // The impure pipe runs for EACH keystroke
  }
}
```

### Instance Creation

Angular creates a **separate instance** of an impure pipe for each template binding where it is used. This means each usage has its own internal state:

```html
<!-- Two separate instances of FilterPipe are created -->
<div>{{ admins | filter:isAdmin }}</div>
<div>{{ editors | filter:isEditor }}</div>
```

For pure pipes, Angular can (and does) reuse the same pipe instance within a component.

## Performance Comparison

Consider a pipe that filters a list of 1000 items:

```typescript
@Pipe({ name: 'activeOnly', standalone: true, pure: false })
export class ActiveOnlyPipe implements PipeTransform {
  transform(users: User[]): User[] {
    return users.filter(u => u.isActive);
  }
}
```

| Scenario | Pure Pipe Calls | Impure Pipe Calls |
|----------|----------------|-------------------|
| Initial render | 1 | 1 |
| Click a button (no data change) | 0 | 1 |
| Type 10 characters in an input | 0 | 10 |
| Scroll the page | 0 | Many (scroll events) |
| Timer fires every second (60s) | 0 | 60 |
| Mouse move over the component | 0 | Hundreds |

For a filter operation on 1000 items that takes 1ms, the impure pipe could consume 100ms+ per second just from mouse movements. A pure pipe with unchanged data uses 0ms.

### Profiling Pipe Performance

Add simple instrumentation to see the impact:

```typescript
@Pipe({ name: 'expensiveTransform', standalone: true, pure: false })
export class ExpensiveTransformPipe implements PipeTransform {
  transform(data: LargeDataSet): TransformedData {
    const start = performance.now();
    const result = this.doExpensiveWork(data);
    const elapsed = performance.now() - start;

    if (elapsed > 1) {
      console.warn(`Pipe took ${elapsed.toFixed(2)}ms`);
    }

    return result;
  }

  private doExpensiveWork(data: LargeDataSet): TransformedData {
    // ...
  }
}
```

## Built-in Impure Pipes

Angular has four built-in impure pipes:

### AsyncPipe (`pure: false`)

`AsyncPipe` must be impure because it subscribes to Observables and Promises, which emit values asynchronously. A pure pipe would never know when a new value arrives because the Observable reference typically stays the same.

```html
{{ data$ | async }}
```

Internally, `AsyncPipe`:
1. Subscribes to the Observable on the first `transform()` call
2. Stores the latest emitted value
3. Calls `ChangeDetectorRef.markForCheck()` when a new value arrives
4. Returns the stored value on every subsequent `transform()` call
5. Unsubscribes when the component is destroyed

### JsonPipe (`pure: false`)

`JsonPipe` is impure because it calls `JSON.stringify()` on every cycle. Since deep comparison of arbitrary objects is expensive, it re-serializes every time.

### SlicePipe (`pure: false`)

`SlicePipe` is impure because mutating the source array (via `push()` or `splice()`) does not change the array reference. To detect these mutations, it must run on every cycle.

### KeyValuePipe (`pure: false`)

`KeyValuePipe` converts objects/Maps to iterable key-value arrays. It is impure because it needs to detect property additions and removals on objects, which cannot be detected by reference comparison alone.

## Interaction with Change Detection Strategies

### Default Change Detection

With the default change detection strategy, change detection runs frequently (on every event, timer, HTTP response). Impure pipes run on every one of these cycles.

### OnPush Change Detection

With `OnPush`, change detection only runs when:
- An `@Input` reference changes
- An event handler fires within the component
- An Observable with `AsyncPipe` emits
- `ChangeDetectorRef.markForCheck()` is called
- A signal that the template reads changes

Impure pipes still run on every change detection cycle **within that component**, but `OnPush` reduces how often change detection runs in the first place:

```typescript
@Component({
  standalone: true,
  imports: [FilterPipe],
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    @for (item of items | filter:isActive; track item.id) {
      <div>{{ item.name }}</div>
    }
  `
})
export class OnPushListComponent {
  items = input.required<Item[]>();
  isActive = (item: Item) => item.active;
}
```

With `OnPush`, the impure `FilterPipe` runs only when the component's change detection runs (triggered by input changes or events within this component), not on every application-wide event.

### Signals and Pipes

With Angular's signal-based reactivity, pure pipes work naturally with signals when the signal value is passed as an argument:

```html
{{ counter() | double }}
```

When `counter` signal changes, Angular detects the new value and re-evaluates the pure pipe. No impure pipe is needed.

For signals containing objects or arrays, use immutable updates to trigger pure pipe re-evaluation:

```typescript
users = signal<User[]>([]);

addUser(user: User) {
  // Creates new array reference -> pure pipe re-evaluates
  this.users.update(list => [...list, user]);
}
```

**Important limitation:** If a pure pipe reads a signal *internally* (inside `transform()`), the pipe will NOT re-execute when that signal changes. Pure pipes only check their explicit template arguments, not signals read inside the method body:

```typescript
// BAD: Pure pipe reading a signal internally - won't re-run when locale changes
@Pipe({ name: 'formatDate', standalone: true })
export class FormatDatePipe implements PipeTransform {
  private locale = inject(LocaleService).currentLocale; // signal

  transform(date: Date): string {
    // locale() is read here, but the pipe won't re-run when it changes
    return date.toLocaleDateString(this.locale());
  }
}

// GOOD: Pass the signal value as an explicit argument
// template: {{ date | formatDate:locale() }}
@Pipe({ name: 'formatDate', standalone: true })
export class FormatDatePipe implements PipeTransform {
  transform(date: Date, locale: string): string {
    return date.toLocaleDateString(locale);
  }
}
```

There is an open Angular feature request (GitHub #56401) to make pure pipes react to signals read inside `transform()`, similar to how `OnPush` components react to signal reads in their templates. Until that lands, always pass signal values as explicit pipe arguments.

### Zoneless Angular

In zoneless Angular (`provideZonelessChangeDetection()`), change detection is triggered only by signals and explicit `markForCheck()` calls. This significantly reduces how often impure pipes run because there is no Zone.js patching DOM events to trigger unnecessary cycles.

However, impure pipes still run whenever change detection does run for their component. The recommendation in zoneless applications is even stronger: avoid impure pipes entirely and use signals with `computed()`.

## When You Genuinely Need an Impure Pipe

There are very few legitimate cases for impure pipes. Here are the real ones:

### 1. Subscribing to Async Data (AsyncPipe Pattern)

The classic use case. The pipe subscribes to an Observable or Promise and returns the latest value. The Observable reference does not change, but the emitted values do.

```typescript
// AsyncPipe is the canonical example
{{ data$ | async }}
```

In Angular 21, `toSignal()` often replaces this need.

### 2. Detecting Object/Array Mutations

When you cannot control the data source and it mutates arrays in place:

```typescript
@Pipe({ name: 'arrayLength', standalone: true, pure: false })
export class ArrayLengthPipe implements PipeTransform {
  transform(arr: unknown[] | null): number {
    return arr?.length ?? 0;
  }
}
```

This is a code smell. The better fix is to use immutable data patterns.

### 3. Time-Dependent Transformations

A pipe whose output changes over time even when the input stays the same:

```typescript
@Pipe({ name: 'timeAgo', standalone: true, pure: false })
export class TimeAgoPipe implements PipeTransform {
  transform(date: Date | string): string {
    // Output changes as time passes, even with the same input
    return formatDistanceToNow(new Date(date));
  }
}
```

Better alternative with signals:

```typescript
// Component approach
private now = signal(Date.now());
private interval = setInterval(() => this.now.set(Date.now()), 60_000);

timeAgo = computed(() => {
  const diff = this.now() - this.date().getTime();
  return this.formatDistance(diff);
});

ngOnDestroy() {
  clearInterval(this.interval);
}
```

## Alternatives to Impure Pipes

For every impure pipe use case, there is usually a better alternative:

### Replace Impure Filter Pipe with Computed Signal

```typescript
// BAD: Impure filter pipe
@Pipe({ name: 'filter', standalone: true, pure: false })
export class FilterPipe implements PipeTransform {
  transform<T>(items: T[], fn: (item: T) => boolean): T[] {
    return items.filter(fn);
  }
}
```

```html
<!-- BAD: Runs on every CD cycle -->
@for (item of items | filter:isActive; track item.id) {
  <div>{{ item.name }}</div>
}
```

```typescript
// GOOD: Computed signal, recalculates only when items() changes
items = signal<Item[]>([]);
activeItems = computed(() => this.items().filter(i => i.active));
```

```html
<!-- GOOD: Reads signal, no pipe needed -->
@for (item of activeItems(); track item.id) {
  <div>{{ item.name }}</div>
}
```

### Replace Impure Sort Pipe with Computed Signal

```typescript
// GOOD: Sort in a computed signal
sortField = signal<keyof User>('name');
sortDirection = signal<'asc' | 'desc'>('asc');

sortedUsers = computed(() => {
  const field = this.sortField();
  const dir = this.sortDirection();
  return [...this.users()].sort((a, b) => {
    const comparison = a[field] < b[field] ? -1 : a[field] > b[field] ? 1 : 0;
    return dir === 'asc' ? comparison : -comparison;
  });
});
```

### Replace AsyncPipe with toSignal()

```typescript
// Impure pipe approach
// template: {{ data$ | async }}

// Signal approach
data = toSignal(this.http.get<Data>('/api/data'), { initialValue: null });
// template: {{ data() }}
```

### Replace Impure Time-Based Pipe with Effect

```typescript
// Signal + timer approach
private destroyRef = inject(DestroyRef);
relativeTime = signal('');

constructor() {
  const updateTime = () => {
    this.relativeTime.set(this.formatTimeAgo(this.timestamp));
  };
  updateTime();
  const id = setInterval(updateTime, 60_000);
  this.destroyRef.onDestroy(() => clearInterval(id));
}
```

## Making an Impure Pipe Efficient

If you must use an impure pipe, minimize the cost of `transform()`:

### Internal Caching

Implement your own change detection inside the pipe:

```typescript
@Pipe({ name: 'filterUsers', standalone: true, pure: false })
export class FilterUsersPipe implements PipeTransform {
  private cachedInput: User[] | null = null;
  private cachedFilter: string = '';
  private cachedResult: User[] = [];

  transform(users: User[], filterTerm: string): User[] {
    // Only recompute if inputs actually changed
    if (users === this.cachedInput && filterTerm === this.cachedFilter) {
      return this.cachedResult;
    }

    this.cachedInput = users;
    this.cachedFilter = filterTerm;
    this.cachedResult = users.filter(u =>
      u.name.toLowerCase().includes(filterTerm.toLowerCase())
    );

    return this.cachedResult;
  }
}
```

This pattern makes the impure pipe behave almost like a pure pipe with deep comparison. The pipe is called on every cycle, but the expensive work only runs when inputs actually differ.

### Return the Same Reference

Returning the same array/object reference when the result has not changed prevents downstream change detection from propagating unnecessarily:

```typescript
@Pipe({ name: 'activeUsers', standalone: true, pure: false })
export class ActiveUsersPipe implements PipeTransform {
  private lastResult: User[] = [];
  private lastInput: User[] = [];

  transform(users: User[]): User[] {
    // Quick reference check
    if (users === this.lastInput) {
      return this.lastResult;
    }

    const result = users.filter(u => u.isActive);

    // Deep equality check on result
    if (this.arraysEqual(result, this.lastResult)) {
      return this.lastResult; // Same reference prevents downstream updates
    }

    this.lastInput = users;
    this.lastResult = result;
    return result;
  }

  private arraysEqual(a: User[], b: User[]): boolean {
    if (a.length !== b.length) return false;
    return a.every((item, index) => item === b[index]);
  }
}
```

## Decision Flowchart

Use this to decide between pure pipe, impure pipe, or an alternative:

```
Does the transformation depend only on its inputs?
  YES -> Does the input change by reference when data changes?
    YES -> Use a PURE PIPE
    NO  -> Can you change the data source to use immutable updates?
      YES -> Use a PURE PIPE + immutable data
      NO  -> Use a COMPUTED SIGNAL (preferred) or IMPURE PIPE with caching
  NO  -> Does it depend on time, locale, or external state?
    -> Use a COMPUTED SIGNAL with a reactive dependency
       (e.g., a timer signal, a locale signal)
```

## Side-by-Side Comparison

```typescript
// PURE: Only runs when `price` reference/value changes
@Pipe({ name: 'formatPrice', standalone: true })
export class FormatPricePipe implements PipeTransform {
  transform(price: number, currency: string = 'USD'): string {
    return new Intl.NumberFormat('en-US', {
      style: 'currency',
      currency
    }).format(price);
  }
}
```

```typescript
// IMPURE: Runs EVERY change detection cycle
@Pipe({ name: 'formatPrice', standalone: true, pure: false })
export class FormatPricePipe implements PipeTransform {
  transform(price: number, currency: string = 'USD'): string {
    // Same logic, but called hundreds of times unnecessarily
    return new Intl.NumberFormat('en-US', {
      style: 'currency',
      currency
    }).format(price);
  }
}
```

The pure version is called once when `price` changes. The impure version is called on every change detection cycle, even when `price` has not changed. For a simple formatting operation, there is zero reason to use impure here.

## Common Mistakes

- **Making a pipe impure because "it wasn't updating."** The real fix is almost always to use immutable data updates (new array/object references) instead of mutations. Switching to impure is a band-aid that causes performance problems.

- **Not implementing internal caching in impure pipes.** If you must use an impure pipe, cache the last inputs and output to avoid redundant computation on every change detection cycle.

- **Using impure filter/sort pipes in `@for` loops.** Each iteration multiplies the performance cost. With 100 items and frequent change detection, you get thousands of unnecessary `transform()` calls per second.

- **Assuming OnPush eliminates impure pipe overhead.** OnPush reduces how often change detection runs for the component, but impure pipes still run on every cycle within that component. Events, signal changes, and `markForCheck()` calls still trigger cycles.

- **Forgetting that impure pipes create separate instances per binding.** If you use the same impure pipe in three places in a template, you get three instances, each with its own state and each running independently on every cycle.

- **Creating an impure pipe to react to signal changes.** Signals trigger change detection naturally. A pure pipe on a signal value re-evaluates when the signal changes. No impure pipe is needed.

---

## Best Practices

- **Default to pure pipes.** Only use `pure: false` when you have exhausted all alternatives and have a clear, justified reason.

- **Use immutable data patterns with pure pipes.** Replace arrays with `[...old, newItem]` and objects with `{ ...old, prop: newValue }`. Signals with `update()` make this natural.

- **Prefer `computed()` signals over impure pipes for filtering, sorting, and derived state.** Computed signals recalculate only when their reactive dependencies change, not on every change detection cycle.

- **Use `toSignal()` instead of `AsyncPipe`** in Angular 21 for new code. This eliminates the most common impure pipe usage.

- **If you must use an impure pipe, implement internal caching.** Compare current inputs against cached inputs and return the cached result when nothing has changed.

- **Combine OnPush with any impure pipes** to reduce how frequently change detection runs for the component, thereby reducing how often the impure pipe executes.

- **Profile before optimizing.** Use Angular DevTools or browser performance tools to measure actual pipe execution frequency and cost before deciding to refactor.

## Real-World Use Cases

### Search Filter with Pure Pipe + Immutable Data

```typescript
@Pipe({ name: 'search', standalone: true })
export class SearchPipe implements PipeTransform {
  transform<T extends Record<string, unknown>>(
    items: T[],
    term: string,
    fields: (keyof T)[]
  ): T[] {
    if (!term || term.length < 2) return items;
    const lower = term.toLowerCase();
    return items.filter(item =>
      fields.some(field => {
        const value = item[field];
        return typeof value === 'string' && value.toLowerCase().includes(lower);
      })
    );
  }
}

@Component({
  standalone: true,
  imports: [SearchPipe],
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <input (input)="onSearch($event)">
    @for (product of products() | search:searchTerm():searchFields; track product.id) {
      <div>{{ product.name }} - {{ product.category }}</div>
    }
  `
})
export class ProductListComponent {
  searchTerm = signal('');
  searchFields: (keyof Product)[] = ['name', 'category', 'description'];

  // Products signal uses immutable updates
  products = signal<Product[]>([]);

  onSearch(event: Event) {
    this.searchTerm.set((event.target as HTMLInputElement).value);
  }
}
```

The `SearchPipe` is pure. It re-evaluates when either `products()` reference changes or `searchTerm()` changes. No impure pipe needed because signals trigger change detection and the pipe sees new parameter values.

### Computed Signal Replacing an Impure Sort Pipe

```typescript
@Component({
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <div class="sort-controls">
      <button (click)="toggleSort('name')">Name</button>
      <button (click)="toggleSort('date')">Date</button>
      <button (click)="toggleSort('price')">Price</button>
    </div>

    @for (item of sortedItems(); track item.id) {
      <div class="item">
        <span>{{ item.name }}</span>
        <span>{{ item.price | currency }}</span>
      </div>
    }
  `,
  imports: [CurrencyPipe]
})
export class SortableListComponent {
  items = input.required<Item[]>();
  sortKey = signal<keyof Item>('name');
  sortAsc = signal(true);

  // No pipe at all. Computed signal handles sorting reactively.
  sortedItems = computed(() => {
    const key = this.sortKey();
    const asc = this.sortAsc();
    return [...this.items()].sort((a, b) => {
      const cmp = a[key] < b[key] ? -1 : a[key] > b[key] ? 1 : 0;
      return asc ? cmp : -cmp;
    });
  });

  toggleSort(key: keyof Item) {
    if (this.sortKey() === key) {
      this.sortAsc.update(v => !v);
    } else {
      this.sortKey.set(key);
      this.sortAsc.set(true);
    }
  }
}
```

### Cached Impure Pipe for Legacy Mutable Data

When working with a legacy service that mutates arrays in place and you cannot change it:

```typescript
@Pipe({ name: 'activeCount', standalone: true, pure: false })
export class ActiveCountPipe implements PipeTransform {
  private lastLength = -1;
  private lastActiveCount = 0;

  transform(items: Item[]): number {
    // Quick check: if array length hasn't changed, skip recount
    if (items.length === this.lastLength) {
      return this.lastActiveCount;
    }

    this.lastLength = items.length;
    this.lastActiveCount = items.filter(i => i.active).length;
    return this.lastActiveCount;
  }
}
```

```html
<p>{{ items | activeCount }} active items</p>
```

This is a pragmatic solution when you cannot refactor the data source. The internal caching minimizes the cost of running on every change detection cycle.

---

> **Key Takeaway:** Pure pipes are Angular's default for good reason: they skip execution when inputs have not changed, making them essentially free during unchanged change detection cycles. Impure pipes run every cycle regardless, creating a performance tax that grows with application complexity. In Angular 21, `computed()` signals and `toSignal()` eliminate nearly all legitimate reasons for impure pipes.
