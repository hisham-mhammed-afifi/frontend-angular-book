# 18.1 Reactive Programming Concepts

> Reactive programming is a paradigm centered on data streams and the propagation of change. Instead of pulling data when you need it, data is pushed to you as it becomes available, and you declare how to transform and react to it.

---

## What Is Reactive Programming

Reactive programming treats everything as a stream of values over time. A stream can emit zero, one, or many values, and it can complete or error. You declare transformations on these streams, and the system propagates changes automatically.

Consider the difference between asking for data and being notified of data:

**Imperative (pull):** "Check the mailbox every 5 minutes to see if a letter arrived."

**Reactive (push):** "The mailman rings the doorbell when a letter arrives."

The most intuitive analogy is a **spreadsheet**. When you change cell A1, every cell that references A1 automatically recalculates. You declare the relationships once; the runtime handles propagation. You never manually tell B1 to update when A1 changes.

In code, this translates to:

```typescript
// Imperative: You ask for data
const data = fetchData();           // Pull: you request, you wait
const filtered = data.filter(...);  // Transform after receiving
render(filtered);                   // Then act on it

// Reactive: Data comes to you
dataStream$
  .pipe(filter(...))                // Declare transformation
  .subscribe(filtered => {          // React when data arrives
    render(filtered);
  });
```

The reactive approach inverts control. Instead of your code deciding *when* to get data, it declares *what to do* when data arrives.

## Push vs Pull Models

The distinction between push and pull is fundamental to understanding reactive programming.

### Pull Model

The consumer decides when to receive data. The consumer requests data from the producer.

```typescript
// Pull: Iterator pattern
function* numberGenerator() {
  yield 1;
  yield 2;
  yield 3;
}

const iterator = numberGenerator();
console.log(iterator.next().value); // 1 - consumer pulls
console.log(iterator.next().value); // 2 - consumer pulls again
console.log(iterator.next().value); // 3 - consumer pulls again
```

```typescript
// Pull: Function call
function getCurrentUser(): User {
  return { name: 'Alice', role: 'admin' };
}

const user = getCurrentUser(); // Consumer pulls by calling
```

| Aspect | Producer | Consumer |
|--------|----------|----------|
| Pull | Passive (waits to be asked) | Active (decides when to read) |

### Push Model

The producer decides when to send data. The consumer reacts to whatever arrives.

```typescript
// Push: Event listener
button.addEventListener('click', (event) => {
  // Consumer reacts when producer (button) pushes an event
  console.log('Clicked!', event);
});

// Push: Observable
const clicks$ = fromEvent(button, 'click');
clicks$.subscribe(event => {
  console.log('Clicked!', event);
});
```

```typescript
// Push: WebSocket
const socket = new WebSocket('wss://api.example.com/feed');
socket.onmessage = (event) => {
  // Data is pushed to you as it arrives
  console.log('Received:', event.data);
};
```

| Aspect | Producer | Consumer |
|--------|----------|----------|
| Push | Active (decides when to send) | Passive (reacts to arrivals) |

### Comparison Table

| | Single Value | Multiple Values |
|---|---|---|
| **Pull** | Function | Iterator / Generator |
| **Push** | Promise | Observable |

- A **Function** produces a single value when called (pull, single).
- A **Promise** pushes a single value when it resolves (push, single).
- An **Iterator** produces multiple values on demand (pull, multiple).
- An **Observable** pushes multiple values over time (push, multiple).

Observables fill the gap that Promises and Iterators leave: they handle multiple asynchronous values pushed over time.

## The Observer Pattern

Reactive programming builds on the Observer pattern, one of the classic Gang of Four design patterns. The pattern defines a one-to-many relationship between a subject (producer) and its observers (consumers).

```typescript
// The Observer pattern in plain TypeScript
interface Observer<T> {
  next(value: T): void;
  error(err: Error): void;
  complete(): void;
}

class Subject<T> {
  private observers: Observer<T>[] = [];

  subscribe(observer: Observer<T>): () => void {
    this.observers.push(observer);
    // Return unsubscribe function
    return () => {
      this.observers = this.observers.filter(o => o !== observer);
    };
  }

  next(value: T): void {
    this.observers.forEach(o => o.next(value));
  }

  error(err: Error): void {
    this.observers.forEach(o => o.error(err));
  }

  complete(): void {
    this.observers.forEach(o => o.complete());
  }
}
```

```typescript
// Usage
const priceUpdates = new Subject<number>();

const unsubscribe = priceUpdates.subscribe({
  next: (price) => console.log(`New price: $${price}`),
  error: (err) => console.error('Price feed error:', err),
  complete: () => console.log('Price feed closed')
});

priceUpdates.next(42.50);   // "New price: $42.50"
priceUpdates.next(43.00);   // "New price: $43.00"
priceUpdates.complete();     // "Price feed closed"
```

RxJS extends this pattern with:
- **Lazy execution:** Nothing happens until you subscribe.
- **Operators:** Composable transformations between producer and consumer.
- **Teardown logic:** Cleanup when the subscription ends.
- **Schedulers:** Control over when and how emissions are processed.

## Streams as the Core Abstraction

A stream is a sequence of ongoing events ordered in time. Every stream can emit three things:

1. **A value** (the `next` notification)
2. **An error** (the `error` notification, terminates the stream)
3. **A completion signal** (the `complete` notification, terminates the stream)

```
Timeline:  ──1──2──3──4──5──|
                              ↑ complete

Timeline:  ──A──B──C──X
                        ↑ error

Timeline:  ──●──●──●──●──●──●──●──  (ongoing, never completes)
```

### Examples of Streams in Frontend Development

| Source | Values Over Time | Completes? |
|--------|-----------------|------------|
| Button clicks | Click events | No (ongoing) |
| HTTP request | Response body | Yes (single value, then complete) |
| WebSocket | Messages | When connection closes |
| Form input | Keystrokes | No (ongoing) |
| Timer | Tick values | Optional |
| Route changes | URL segments | No (ongoing) |
| Window resize | Dimensions | No (ongoing) |
| Mouse movement | Coordinates | No (ongoing) |

Everything in a frontend application can be modeled as a stream: user interactions, server responses, timers, state changes, navigation events.

## Declarative vs Imperative Data Transformation

Reactive programming favors declarative data pipelines over imperative step-by-step instructions.

### Imperative Approach

```typescript
// Imperative: step-by-step instructions
let results: SearchResult[] = [];
let loading = false;
let error: string | null = null;

async function onSearchChange(term: string) {
  if (term.length < 3) {
    results = [];
    return;
  }

  loading = true;
  error = null;

  try {
    const response = await fetch(`/api/search?q=${term}`);
    if (!response.ok) throw new Error('Search failed');
    results = await response.json();
  } catch (e) {
    error = e.message;
    results = [];
  } finally {
    loading = false;
  }
}
```

Problems:
- Mutable state (`results`, `loading`, `error`) is scattered and manually managed.
- Race conditions: if the user types fast, earlier requests may resolve after later ones.
- No debouncing built in.
- Error and loading states require manual toggling.

### Declarative Reactive Approach

```typescript
// Reactive: declare the transformation pipeline
const searchResults$ = searchInput$.pipe(
  debounceTime(300),                          // Wait for typing pause
  filter(term => term.length >= 3),           // Ignore short terms
  distinctUntilChanged(),                     // Skip duplicates
  switchMap(term =>                           // Cancel previous request
    fromFetch(`/api/search?q=${term}`).pipe(
      switchMap(res => res.json()),
      catchError(err => of([]))               // Handle errors inline
    )
  )
);
```

Benefits:
- **Debouncing** is declarative: `debounceTime(300)`.
- **Race conditions** are handled: `switchMap` cancels the previous request when a new one starts.
- **Deduplication** is built in: `distinctUntilChanged()`.
- **Error handling** is inline: `catchError`.
- **No mutable state** to manage manually.

The reactive version reads as a description of *what* should happen, not *how* to make it happen step by step.

## Key Reactive Concepts

### Producer (Observable)

The source of data. It pushes values to consumers. In RxJS, this is an `Observable`.

```typescript
import { Observable } from 'rxjs';

const temperature$ = new Observable<number>(subscriber => {
  // This function runs when someone subscribes
  const sensor = connectToSensor();
  sensor.onReading = (temp) => subscriber.next(temp);
  sensor.onError = (err) => subscriber.error(err);

  // Cleanup when subscription ends
  return () => sensor.disconnect();
});
```

### Consumer (Observer)

The receiver of data. It defines how to react to values, errors, and completion.

```typescript
const observer = {
  next: (temp: number) => console.log(`Temperature: ${temp}C`),
  error: (err: Error) => console.error('Sensor error:', err),
  complete: () => console.log('Sensor disconnected')
};
```

### Subscription

The connection between a producer and a consumer. It represents an ongoing execution and provides a way to cancel it.

```typescript
const subscription = temperature$.subscribe(observer);

// Later: cancel the subscription (triggers cleanup)
subscription.unsubscribe();
```

### Operators

Functions that transform, filter, combine, or otherwise manipulate streams. They take an Observable as input and return a new Observable.

```typescript
import { map, filter, throttleTime } from 'rxjs';

const alerts$ = temperature$.pipe(
  filter(temp => temp > 100),      // Only high temperatures
  throttleTime(5000),              // At most one alert per 5 seconds
  map(temp => `WARNING: ${temp}C`) // Transform to alert message
);
```

Operators are composable. You chain them in a `pipe()` to build a transformation pipeline.

### Subscription Lifecycle

```
1. Subscribe     -> Producer starts emitting
2. Next (0..N)   -> Consumer receives values
3. Error OR Complete -> Stream terminates
4. Unsubscribe   -> Cleanup runs (if not already terminated)
```

An Observable can emit zero or more `next` values. It terminates with either `error` or `complete` (never both). After termination, no more values are emitted.

## Marble Diagrams

Marble diagrams are the standard visual notation for describing Observable behavior. They show values as marbles on a timeline.

### Reading Marble Diagrams

```
Source:   ──1──2──3──4──5──|
                            ↑ complete (vertical bar)

Operator: map(x => x * 10)

Output:   ──10─20─30─40─50─|
```

```
Source:   ──A──B──C──X
                      ↑ error (X)

Source:   ──●──●──●──●──●──>
                            ↑ ongoing (arrow, never completes)
```

### Symbols

| Symbol | Meaning |
|--------|---------|
| `──` | Time passing |
| `1`, `A`, `●` | Emitted value (a "marble") |
| `\|` | Completion |
| `X` or `#` | Error |
| `>` or `→` | Stream continues (no completion) |
| `^` | Subscription point |
| `!` | Unsubscription point |

### Operator Marble Diagrams

```
Source:   ──1──2──3──4──5──6──|

filter(x => x % 2 === 0)

Output:   ─────2─────4─────6──|
```

```
Source:   ──1────2────3────|

map(x => x * x)

Output:   ──1────4────9────|
```

```
Source A:  ──1────2────3────|
Source B:  ────A────B────C──|

merge(A, B)

Output:   ──1─A──2─B──3─C──|
```

### switchMap Marble Diagram

```
Source:     ──A──────B──────C──|

switchMap(x => interval(100).pipe(take(3), map(i => x + i)))

               A0─A1─A2
                     B0─B1─B2
                           C0─C1─C2─|
Output:    ──A0─A1──B0─B1──C0─C1─C2─|
                ↑ A cancelled when B arrives
                      ↑ B cancelled when C arrives
```

`switchMap` cancels the inner Observable when a new outer value arrives. This is why it solves the race condition problem in search typeaheads.

## Backpressure

Backpressure is a flow control mechanism where a slow consumer signals a fast producer to slow down. When a producer emits data faster than the consumer can process it, the system needs a strategy to cope.

In backend reactive systems (Project Reactor, RxJava), backpressure is explicit: the consumer requests a specific number of items via `Subscription.request(n)`. In frontend RxJS, backpressure is implicit through operators that control the rate of emissions.

### Backpressure Strategies in RxJS

```typescript
import {
  fromEvent, debounceTime, throttleTime, auditTime,
  sampleTime, bufferTime
} from 'rxjs';

const mouseMoves$ = fromEvent(document, 'mousemove');

// Drop: ignore events during a window (keep first)
mouseMoves$.pipe(throttleTime(100));

// Drop: ignore events during a window (keep last)
mouseMoves$.pipe(debounceTime(300));

// Sample: take the latest value at fixed intervals
mouseMoves$.pipe(sampleTime(200));

// Audit: wait for a window, then emit the latest
mouseMoves$.pipe(auditTime(100));

// Buffer: accumulate into arrays, emit periodically
mouseMoves$.pipe(bufferTime(500));
```

| Strategy | RxJS Operator | Behavior |
|----------|--------------|----------|
| Drop (first) | `throttleTime` | Emit first, ignore rest of window |
| Drop (last) | `debounceTime` | Wait for silence, emit last |
| Sample | `sampleTime` | Emit latest at fixed intervals |
| Audit | `auditTime` | After first emission, wait, then emit latest |
| Buffer | `bufferTime` | Collect into arrays, emit batch |

Backpressure does not eliminate the problem. It moves the decision about what to drop, delay, or batch to a point in the pipeline where it can be handled deliberately rather than causing uncontrolled failures.

## Hot vs Cold Observables

### Cold Observables

A cold Observable creates a new producer for each subscriber. Each subscription gets its own independent execution.

```typescript
import { Observable } from 'rxjs';

// Cold: each subscriber gets independent values
const cold$ = new Observable<number>(subscriber => {
  console.log('Producer created');
  subscriber.next(Math.random());
  subscriber.complete();
});

cold$.subscribe(v => console.log('Sub 1:', v)); // Producer created, Sub 1: 0.847
cold$.subscribe(v => console.log('Sub 2:', v)); // Producer created, Sub 2: 0.293
// Different values! Different producer instances.
```

Most RxJS creation operators produce cold Observables: `of()`, `from()`, `interval()`, `HttpClient.get()`.

### Hot Observables

A hot Observable shares a single producer among all subscribers. Values are emitted regardless of whether anyone is subscribed.

```typescript
import { fromEvent } from 'rxjs';

// Hot: all subscribers share the same event source
const clicks$ = fromEvent(document, 'click');

// Subscriber 1 joins
clicks$.subscribe(e => console.log('Sub 1:', e));

// 3 seconds later, Subscriber 2 joins
// Sub 2 does NOT receive the clicks that happened before it subscribed
setTimeout(() => {
  clicks$.subscribe(e => console.log('Sub 2:', e));
}, 3000);
```

DOM events, WebSockets, and Subjects are hot. HTTP requests and `interval()` are cold.

### Why This Matters

- **Cold** Observables are safe to subscribe to multiple times (each gets its own data).
- **Hot** Observables require awareness of timing: late subscribers miss earlier emissions.
- Converting cold to hot (multicasting) is a common pattern: `share()`, `shareReplay()`.

## Reactive Programming in Angular

Angular uses two complementary reactive systems:

### RxJS Observables (Asynchronous Streams)

RxJS handles event-driven, asynchronous data flows:

```typescript
// HTTP requests
this.http.get<User[]>('/api/users')

// Form value changes
this.searchControl.valueChanges

// Router events
this.router.events

// WebSocket messages
webSocket('wss://api.example.com/feed')
```

RxJS excels at:
- Composing multiple async operations
- Cancellation (via `switchMap`, `takeUntil`)
- Retry logic and error recovery
- Debouncing and throttling
- Combining streams from different sources

### Signals (Synchronous Reactive State)

Angular Signals handle synchronous state that changes over time:

```typescript
// Writable signal
const count = signal(0);

// Computed (derived) signal
const doubled = computed(() => count() * 2);

// Effect (side effect on signal change)
effect(() => console.log('Count is:', count()));

// Update
count.set(5);    // doubled is now 10
count.update(c => c + 1); // doubled is now 12
```

Signals excel at:
- Component state management
- Derived/computed values
- Template bindings (fine-grained change detection)
- Synchronous reactivity without subscription management

### When to Use Each

| Use Case | RxJS | Signals |
|----------|------|---------|
| HTTP requests | Yes | Via `toSignal()` |
| WebSockets | Yes | Convert with `toSignal()` |
| User input debouncing | Yes | No built-in debounce |
| Component state | Possible but verbose | Yes (preferred) |
| Derived/computed values | `combineLatest` + `map` | `computed()` |
| Complex async orchestration | Yes | No |
| Template binding | Via `AsyncPipe` or `toSignal()` | Direct `signal()` |

The two systems interoperate through `toSignal()` and `toObservable()`:

```typescript
// Observable -> Signal
const users = toSignal(this.http.get<User[]>('/api/users'), { initialValue: [] });

// Signal -> Observable
const count$ = toObservable(this.count);
```

## Why Reactive Programming Matters for Frontend

Frontend applications are inherently event-driven: user clicks, network responses, timers, animations, route changes, form inputs. Reactive programming provides a unified model for handling all of these.

### Problems Reactive Programming Solves

**1. Race conditions:**

```typescript
// Imperative: race condition if user types "abc" then "ab"
// The "abc" response may arrive after "ab" response
searchInput.addEventListener('input', async (e) => {
  const results = await search(e.target.value);
  renderResults(results); // Which results are displayed?
});

// Reactive: switchMap cancels stale requests
searchInput$.pipe(
  switchMap(term => search(term))
).subscribe(results => renderResults(results));
```

**2. Memory leaks from forgotten cleanup:**

```typescript
// Imperative: must remember to remove listener
const handler = (e) => { ... };
window.addEventListener('resize', handler);
// Easy to forget: window.removeEventListener('resize', handler);

// Reactive: subscription tracks cleanup
const sub = fromEvent(window, 'resize').subscribe(e => { ... });
sub.unsubscribe(); // Or use takeUntilDestroyed() in Angular
```

**3. Coordinating multiple async operations:**

```typescript
// Imperative: callback nesting or complex Promise chains
const user = await getUser(id);
const [orders, preferences] = await Promise.all([
  getOrders(user.id),
  getPreferences(user.id)
]);
// What if getUser fails? What if we need to retry?

// Reactive: declarative composition
const userData$ = getUserId$.pipe(
  switchMap(id => forkJoin({
    user: getUser(id),
    orders: getOrders(id),
    preferences: getPreferences(id)
  })),
  retry(3),
  catchError(err => of(defaultData))
);
```

## Common Mistakes

- **Thinking reactively means using RxJS everywhere.** Reactive programming is a mindset, not a library. Angular Signals are reactive too. Use the right tool for the right job: Signals for synchronous state, RxJS for async streams.

- **Treating Observables like Promises.** Observables are lazy (nothing happens until subscribe), can emit multiple values, and must be unsubscribed. Promises are eager, emit once, and cannot be cancelled.

- **Forgetting that cold Observables create new executions per subscriber.** Each `.subscribe()` on an HTTP Observable triggers a new request. Use `shareReplay()` or `toSignal()` to share a single execution.

- **Mixing imperative state mutation with reactive streams.** Avoid updating external variables inside `subscribe()` callbacks. Instead, compose the stream to produce the final result declaratively.

- **Not unsubscribing from long-lived Observables.** Subscriptions to ongoing streams (events, intervals, WebSockets) must be cleaned up when the component is destroyed. Use `takeUntilDestroyed()`, `AsyncPipe`, or `toSignal()`.

---

## Best Practices

- **Think in streams first.** Before writing code, identify the data sources and how they relate. Draw a marble diagram if the flow is complex.

- **Use operators to transform data declaratively.** Prefer `pipe()` with operators over imperative logic in `subscribe()` callbacks.

- **Keep `subscribe()` callbacks minimal.** The subscribe callback should trigger a side effect (render, log, navigate), not contain business logic. Business logic belongs in the operator chain.

- **Prefer `switchMap` for search and navigation patterns.** It automatically cancels previous inner Observables, preventing race conditions.

- **Use Signals for component-local state** and RxJS for event streams and async operations. Convert between them with `toSignal()` and `toObservable()` at the boundaries.

- **Learn marble diagrams.** They are the universal language for reasoning about reactive code. Every RxJS operator's behavior can be described with a marble diagram.

## Real-World Use Cases

### Autocomplete Search

```typescript
// Declarative pipeline: debounce, deduplicate, cancel stale, handle errors
const suggestions$ = this.searchControl.valueChanges.pipe(
  debounceTime(300),
  distinctUntilChanged(),
  filter(term => term.length >= 2),
  switchMap(term =>
    this.http.get<string[]>(`/api/suggest?q=${term}`).pipe(
      catchError(() => of([]))
    )
  )
);
```

This single pipeline handles debouncing, deduplication, race condition prevention, and error recovery. The imperative equivalent would require timers, flags, abort controllers, and try/catch blocks.

### Real-Time Dashboard

```typescript
// Combine multiple real-time sources into a single view model
const dashboard$ = combineLatest({
  metrics: this.metricsService.liveMetrics$,
  alerts: this.alertService.activeAlerts$,
  status: timer(0, 30000).pipe(
    switchMap(() => this.healthService.checkStatus())
  )
}).pipe(
  map(({ metrics, alerts, status }) => ({
    ...metrics,
    alertCount: alerts.length,
    criticalAlerts: alerts.filter(a => a.severity === 'critical'),
    systemHealth: status.healthy ? 'operational' : 'degraded'
  }))
);
```

### Form with Dependent Validation

```typescript
// Country selection drives available cities
const cities$ = this.countryControl.valueChanges.pipe(
  distinctUntilChanged(),
  switchMap(country =>
    this.locationService.getCities(country).pipe(
      startWith([]),   // Clear cities while loading
      catchError(() => of([]))
    )
  )
);
```

---

> **Key Takeaway:** Reactive programming inverts the flow of control: instead of pulling data and managing state imperatively, you declare transformation pipelines over push-based streams. This eliminates entire categories of bugs (race conditions, memory leaks, stale state) and makes complex async coordination composable and readable.
