# 18.10 Multicasting (`share`, `shareReplay`)

> Multicasting lets multiple subscribers share a single execution of an Observable pipeline instead of each triggering their own, saving resources and ensuring consistency.

---

## Why Multicasting Matters

By default, RxJS Observables are **cold**. Each subscription triggers a brand-new execution of the producer function. For HTTP calls, this means duplicate network requests. For WebSocket connections, it means multiple sockets. For expensive computations, it means redundant CPU work.

```typescript
// Cold Observable: each subscriber triggers a separate HTTP request
const users$ = this.http.get<User[]>('/api/users');

// Subscriber 1: fires HTTP request #1
users$.subscribe(data => this.renderTable(data));

// Subscriber 2: fires HTTP request #2 (duplicate!)
users$.subscribe(data => this.updateCount(data));
```

Multicasting converts a cold Observable into a hot one. All subscribers share a single upstream subscription, receiving the same values at the same time.

```typescript
// Hot (multicasted): one HTTP request, shared by all subscribers
const users$ = this.http.get<User[]>('/api/users').pipe(
  shareReplay(1)
);

// Both subscribers get the same data from a single request
users$.subscribe(data => this.renderTable(data));
users$.subscribe(data => this.updateCount(data));
```

## The Evolution of Multicasting in RxJS

RxJS multicasting APIs have changed significantly across versions. Understanding the history helps when reading older codebases.

| Version | API | Status |
|---------|-----|--------|
| RxJS 5 | `multicast()`, `publish()`, `refCount()`, `publishReplay()`, `publishBehavior()`, `publishLast()` | Removed in RxJS 8 |
| RxJS 6 | Same as above, plus `shareReplay()` | Deprecated in RxJS 7 |
| RxJS 7+ | `share()`, `shareReplay()`, `connectable()` | **Current API** |
| RxJS 8 | `share()`, `shareReplay()`, `connectable()` | All legacy operators removed |

**If you are starting fresh, only learn `share()`, `shareReplay()`, and `connectable()`.** Everything else is legacy.

### Legacy Migration Mapping

If you encounter deprecated operators in existing codebases, here is the migration path:

| Deprecated Pattern | Modern Replacement |
|---|---|
| `multicast(() => new Subject()) + refCount()` | `share()` |
| `publish() + refCount()` | `share({ resetOnError: false, resetOnComplete: false, resetOnRefCountZero: false })` |
| `publishReplay(N) + refCount()` | `share({ connector: () => new ReplaySubject(N), resetOnError: false, resetOnComplete: false, resetOnRefCountZero: false })` |
| `publishLast() + refCount()` | `share({ connector: () => new AsyncSubject(), resetOnError: false, resetOnComplete: false, resetOnRefCountZero: false })` |
| `publishBehavior(init) + refCount()` | `share({ connector: () => new BehaviorSubject(init), resetOnError: false, resetOnComplete: false, resetOnRefCountZero: false })` |
| `new ConnectableObservable(source, subj)` | `connectable(source, { connector: () => subj })` |

Note: The old `publishReplay(1).refCount()` reused the same `ReplaySubject` instance when refCount dropped to zero. The modern `share()` creates a **new** Subject via its factory on each reset. This is a behavioral difference to watch for during migration.

## Understanding the Fundamentals

### Unicast vs Multicast

```typescript
// UNICAST (default): Each subscriber gets its own independent execution
const random$ = new Observable<number>(subscriber => {
  const value = Math.random();
  console.log('Producer called, generated:', value);
  subscriber.next(value);
  subscriber.complete();
});

random$.subscribe(v => console.log('Sub A:', v));
// Producer called, generated: 0.472
// Sub A: 0.472

random$.subscribe(v => console.log('Sub B:', v));
// Producer called, generated: 0.891  <-- different value, separate execution
// Sub B: 0.891
```

```typescript
// MULTICAST: All subscribers share one execution
const random$ = new Observable<number>(subscriber => {
  const value = Math.random();
  console.log('Producer called, generated:', value);
  subscriber.next(value);
  subscriber.complete();
}).pipe(share());

random$.subscribe(v => console.log('Sub A:', v));
// Producer called, generated: 0.472
// Sub A: 0.472

random$.subscribe(v => console.log('Sub B:', v));
// Sub B: 0.472  <-- same value, shared execution
```

### The Subject Under the Hood

All multicasting operators work by placing a **Subject** between the source Observable and the subscribers. The Subject subscribes to the source once, and all downstream subscribers subscribe to the Subject.

```
Without multicasting:
  Source ──> Subscriber A  (execution 1)
  Source ──> Subscriber B  (execution 2)
  Source ──> Subscriber C  (execution 3)

With multicasting:
  Source ──> Subject ──> Subscriber A
                    ──> Subscriber B  (single execution, shared)
                    ──> Subscriber C
```

The type of Subject used determines the behavior:

| Subject Type | Operator | Behavior |
|--------------|----------|----------|
| `Subject` | `share()` | Late subscribers miss past values |
| `ReplaySubject(n)` | `shareReplay(n)` | Late subscribers get the last `n` values |
| `BehaviorSubject` | (custom via `share` config) | Late subscribers get the current value |
| `AsyncSubject` | (custom via `share` config) | Subscribers get only the last value on completion |

## `share()` -- The Modern Multicasting Operator

`share()` is the primary multicasting operator in RxJS 7+. It replaced all the legacy `publish*` + `refCount()` combinations.

### Basic Usage

```typescript
import { share } from 'rxjs';

const source$ = interval(1000).pipe(
  tap(n => console.log('Source emitted:', n)),
  share()
);

// First subscriber triggers the source
const subA = source$.subscribe(v => console.log('A:', v));
// Source emitted: 0
// A: 0

// Second subscriber joins, shares the same execution
// (added after 500ms, so it catches value 1 onward)
setTimeout(() => {
  const subB = source$.subscribe(v => console.log('B:', v));
}, 500);

// Source emitted: 1
// A: 1
// B: 1  <-- same value, same time
```

### `share()` Configuration (RxJS 7+)

The real power of `share()` is its configuration object, which lets you customize every aspect of the multicasting behavior.

```typescript
share({
  connector: () => new Subject(),          // What Subject to use
  resetOnError: true,                       // Reset on source error?
  resetOnComplete: true,                    // Reset on source complete?
  resetOnRefCountZero: true                 // Reset when all unsubscribe?
})
```

#### `connector`

Controls which type of Subject is used internally.

```typescript
// Replay the last value to late subscribers
const cached$ = source$.pipe(
  share({
    connector: () => new ReplaySubject(1)
  })
);

// Use BehaviorSubject for an initial value
const withDefault$ = source$.pipe(
  share({
    connector: () => new BehaviorSubject('loading')
  })
);

// Only emit the final value (like publishLast)
const final$ = source$.pipe(
  share({
    connector: () => new AsyncSubject()
  })
);
```

#### `resetOnRefCountZero`

Controls what happens when all subscribers unsubscribe (refCount drops to zero).

```typescript
// Default: true -- resets, next subscriber gets a fresh execution
const resettable$ = source$.pipe(
  share({ resetOnRefCountZero: true })
);

// false -- keeps the Subject alive, reconnects to source on next subscribe
const persistent$ = source$.pipe(
  share({ resetOnRefCountZero: false })
);

// With a delay: resets after a grace period
// Useful for HTTP caching with a TTL
const cachedWithTTL$ = source$.pipe(
  share({
    connector: () => new ReplaySubject(1),
    resetOnRefCountZero: () => timer(5000) // 5-second cache window
  })
);
```

The timer-based reset is powerful for caching patterns:

```typescript
@Injectable({ providedIn: 'root' })
export class ConfigService {
  private readonly http = inject(HttpClient);

  // Cache config for 30 seconds after last subscriber unsubscribes
  readonly config$ = this.http.get<AppConfig>('/api/config').pipe(
    share({
      connector: () => new ReplaySubject<AppConfig>(1),
      resetOnRefCountZero: () => timer(30_000),
      resetOnComplete: false,
      resetOnError: () => timer(5_000) // Retry sooner on error
    })
  );
}
```

#### `resetOnError`

Controls behavior when the source errors.

```typescript
// Default: true -- resets, next subscriber retries the source
const retryable$ = source$.pipe(
  share({ resetOnError: true })
);

// false -- error is cached, all future subscribers get the same error
const cachedError$ = source$.pipe(
  share({ resetOnError: false })
);

// With delay: wait before allowing retry
const throttledRetry$ = source$.pipe(
  share({ resetOnError: () => timer(3000) })
);
```

#### `resetOnComplete`

Controls behavior when the source completes.

```typescript
// Default: true -- resets, next subscriber re-executes the source
const fresh$ = source$.pipe(
  share({ resetOnComplete: true })
);

// false -- completion is cached, future subscribers get immediate complete
const oneShot$ = source$.pipe(
  share({ resetOnComplete: false })
);
```

### How RefCount Works

The "refCount" is the number of active subscribers. `share()` uses it to manage the lifecycle:

```
Timeline:
  Sub A subscribes    -> refCount = 1 -> subscribe to source
  Sub B subscribes    -> refCount = 2
  Sub A unsubscribes  -> refCount = 1
  Sub B unsubscribes  -> refCount = 0 -> unsubscribe from source (reset)
  Sub C subscribes    -> refCount = 1 -> re-subscribe to source (fresh)
```

This automatic lifecycle management is why `share()` replaced the old `multicast()` + `refCount()` pattern.

## `shareReplay()` -- Caching with Replay

`shareReplay()` is built on top of `share()` with a `ReplaySubject` connector. It replays the last `n` values to late subscribers.

### Basic Usage

```typescript
import { shareReplay } from 'rxjs';

const data$ = this.http.get<Data>('/api/data').pipe(
  shareReplay(1) // Cache the last emission
);

// First subscriber triggers the HTTP call
data$.subscribe(d => console.log('First:', d));

// Second subscriber (even much later) gets the cached value immediately
// No new HTTP request
setTimeout(() => {
  data$.subscribe(d => console.log('Second:', d)); // Instant replay
}, 10000);
```

### `shareReplay` Signatures

```typescript
// Shorthand: just the buffer size
shareReplay(1)

// Full config object
shareReplay({
  bufferSize: 1,
  refCount: false,       // CRITICAL: defaults to false
  windowTime: undefined  // Optional: time-based cache expiry
})
```

### The Critical `refCount` Parameter

This is the single most important thing to understand about `shareReplay`. The `refCount` parameter determines whether the inner ReplaySubject resets when all subscribers unsubscribe.

```typescript
// refCount: false (DEFAULT) -- DANGEROUS for long-lived apps
const leaky$ = source$.pipe(
  shareReplay(1) // refCount defaults to false
);

// The source subscription NEVER tears down, even with zero subscribers.
// The ReplaySubject stays in memory forever.
// This is fine for app-level singletons, but dangerous for component-level usage.
```

```typescript
// refCount: true -- SAFE for component-level usage
const safe$ = source$.pipe(
  shareReplay({ bufferSize: 1, refCount: true })
);

// When all subscribers unsubscribe, the source subscription tears down.
// Next subscriber triggers a fresh execution.
```

**Rule of thumb:**
- App-level singleton services: `shareReplay(1)` (default `refCount: false`) is fine because the service lives as long as the app.
- Component-level or feature-level: Always use `shareReplay({ bufferSize: 1, refCount: true })`.

### `windowTime` for Time-Based Cache Expiry

```typescript
// Cache expires after 30 seconds
const timedCache$ = this.http.get<Data>('/api/data').pipe(
  shareReplay({
    bufferSize: 1,
    refCount: true,
    windowTime: 30_000 // Milliseconds
  })
);

// Within 30 seconds: replays cached value
// After 30 seconds: triggers new HTTP request on next subscribe
```

### `shareReplay` vs `share` with ReplaySubject

These are nearly equivalent but have different defaults:

```typescript
// shareReplay(1) is roughly equivalent to:
share({
  connector: () => new ReplaySubject(1),
  resetOnError: false,        // Different from share() default!
  resetOnComplete: false,     // Different from share() default!
  resetOnRefCountZero: false  // Different from share() default!
});

// shareReplay({ bufferSize: 1, refCount: true }) is roughly:
share({
  connector: () => new ReplaySubject(1),
  resetOnError: false,
  resetOnComplete: false,
  resetOnRefCountZero: true   // Only this changes with refCount: true
});
```

The key differences from plain `share()`:
1. `shareReplay` defaults `resetOnRefCountZero` to `false` (share defaults to `true`)
2. `shareReplay` defaults `resetOnError` to `false` (share defaults to `true`)
3. `shareReplay` defaults `resetOnComplete` to `false` (share defaults to `true`)

This means `shareReplay` is stickier by default. Once a value, error, or completion is cached, it stays cached.

**Critical behavioral difference:** `shareReplay({ refCount: true })` keeps the same `ReplaySubject` instance (buffer preserved) when refCount drops to zero, only tearing down the source subscription. `share({ connector: () => new ReplaySubject(1) })` creates a **new** `ReplaySubject` on reset, discarding the buffer entirely. This means with `shareReplay({ refCount: true })`, a re-subscriber still gets the cached value immediately, while with `share()`, they get nothing until the source re-emits.

## `connectable()` -- Manual Connection Control

`connectable()` gives you explicit control over when the source starts emitting. Unlike `share()` which auto-connects on first subscribe, `connectable()` requires a manual `connect()` call.

```typescript
import { connectable, interval, Subject } from 'rxjs';

const source$ = interval(1000);

const multicasted$ = connectable(source$, {
  connector: () => new Subject()
});

// Subscribe does NOT trigger the source
const subA = multicasted$.subscribe(v => console.log('A:', v));
const subB = multicasted$.subscribe(v => console.log('B:', v));

// Nothing happening yet...

// Manually connect: NOW the source starts
const connection = multicasted$.connect();

// A: 0, B: 0
// A: 1, B: 1
// ...

// Manually disconnect
connection.unsubscribe();
```

### When to Use `connectable()`

Use `connectable()` when you need all subscribers to be set up before the source starts emitting. This prevents the race condition where early subscribers get values that late subscribers miss.

```typescript
// Setup phase: wire up all consumers
const events$ = connectable(
  fromEvent(document, 'click').pipe(
    map(e => ({ x: (e as MouseEvent).clientX, y: (e as MouseEvent).clientY }))
  ),
  { connector: () => new Subject() }
);

// All subscribers ready before any events flow
events$.subscribe(pos => this.updateCursor(pos));
events$.subscribe(pos => this.logAnalytics(pos));
events$.subscribe(pos => this.checkBounds(pos));

// Start
events$.connect();
```

### `connectable` with `resetOnDisconnect`

```typescript
const multicasted$ = connectable(source$, {
  connector: () => new ReplaySubject(1),
  resetOnDisconnect: true // Default: true. Resets Subject on disconnect.
});
```

## Multicasting in Angular Services

### Pattern 1: Shared HTTP Cache

The most common Angular multicasting pattern. A service caches an HTTP response so multiple components do not trigger duplicate requests.

```typescript
@Injectable({ providedIn: 'root' })
export class UserService {
  private readonly http = inject(HttpClient);

  // Single HTTP request shared across all subscribers
  // Cached indefinitely (appropriate for root-level service)
  readonly currentUser$ = this.http.get<User>('/api/me').pipe(
    shareReplay(1)
  );
}

// Component A
@Component({
  template: `<h1>Welcome, {{ user().name }}</h1>`
})
export class HeaderComponent {
  private readonly userService = inject(UserService);
  readonly user = toSignal(this.userService.currentUser$);
}

// Component B -- same service, same cached data, no duplicate request
@Component({
  template: `<img [src]="user().avatar" />`
})
export class AvatarComponent {
  private readonly userService = inject(UserService);
  readonly user = toSignal(this.userService.currentUser$);
}
```

### Pattern 2: Refreshable Cache

Sometimes you need to invalidate the cache and fetch fresh data.

```typescript
@Injectable({ providedIn: 'root' })
export class ProductService {
  private readonly http = inject(HttpClient);
  private readonly refresh$ = new Subject<void>();

  readonly products$ = this.refresh$.pipe(
    startWith(undefined), // Trigger initial load
    switchMap(() => this.http.get<Product[]>('/api/products')),
    shareReplay(1)
  );

  refresh(): void {
    this.refresh$.next();
  }
}
```

### Pattern 3: Shared WebSocket Stream

```typescript
@Injectable({ providedIn: 'root' })
export class LivePriceService {
  private readonly prices$ = webSocket<PriceUpdate>('wss://prices.example.com').pipe(
    share({
      connector: () => new ReplaySubject<PriceUpdate>(1),
      resetOnRefCountZero: () => timer(5000) // Keep connection for 5s after last unsub
    })
  );

  getPriceStream(symbol: string): Observable<PriceUpdate> {
    return this.prices$.pipe(
      filter(update => update.symbol === symbol)
    );
  }
}
```

### Pattern 4: Computed Observable with Multiple Sources

```typescript
@Injectable({ providedIn: 'root' })
export class DashboardService {
  private readonly http = inject(HttpClient);

  private readonly users$ = this.http.get<User[]>('/api/users').pipe(
    shareReplay(1)
  );

  private readonly orders$ = this.http.get<Order[]>('/api/orders').pipe(
    shareReplay(1)
  );

  // Derived stream that depends on both cached sources
  readonly dashboard$ = combineLatest([this.users$, this.orders$]).pipe(
    map(([users, orders]) => ({
      totalUsers: users.length,
      totalOrders: orders.length,
      recentOrders: orders.filter(o => isRecent(o))
    })),
    shareReplay(1)
  );
}
```

## Multicasting with Signals

Angular Signals and RxJS multicasting solve overlapping problems. Signals are inherently multicast (multiple consumers read the same value). When bridging between RxJS and Signals, understand where multicasting is already handled.

```typescript
@Injectable({ providedIn: 'root' })
export class FeatureFlagService {
  private readonly http = inject(HttpClient);

  // RxJS multicasting for the stream
  private readonly flags$ = this.http.get<FeatureFlags>('/api/flags').pipe(
    shareReplay(1)
  );

  // Signal for template consumption -- toSignal subscribes once
  readonly flags = toSignal(this.flags$, {
    initialValue: {} as FeatureFlags
  });
}
```

**Important:** `toSignal()` creates one subscription. If you call `toSignal()` on the same Observable in multiple places without `shareReplay`, each call creates a separate subscription. The `shareReplay` ensures the HTTP call only fires once.

```typescript
// BAD: Two toSignal calls on a cold Observable = two HTTP requests
@Component({ /* ... */ })
export class BadComponent {
  private readonly http = inject(HttpClient);
  private readonly data$ = this.http.get('/api/data');

  readonly dataA = toSignal(this.data$); // HTTP request #1
  readonly dataB = toSignal(this.data$); // HTTP request #2
}

// GOOD: shareReplay ensures single request
@Component({ /* ... */ })
export class GoodComponent {
  private readonly http = inject(HttpClient);
  private readonly data$ = this.http.get('/api/data').pipe(shareReplay(1));

  readonly dataA = toSignal(this.data$); // Shared
  readonly dataB = toSignal(this.data$); // Shared, replayed
}
```

## The `refCount` Lifecycle in Detail

Understanding exactly when subscriptions are created and torn down is essential to avoiding bugs.

```typescript
const source$ = interval(1000).pipe(
  tap(n => console.log(`[Source] ${n}`)),
  share() // refCount: resets on zero subscribers
);

// T=0s: Sub A subscribes. refCount: 0 -> 1. Source starts.
const subA = source$.subscribe(v => console.log('A:', v));

// T=0.5s: Sub B subscribes. refCount: 1 -> 2. Source already running.
setTimeout(() => {
  const subB = source$.subscribe(v => console.log('B:', v));

  // T=2.5s: Sub B unsubscribes. refCount: 2 -> 1. Source continues.
  setTimeout(() => subB.unsubscribe(), 2000);
}, 500);

// T=4s: Sub A unsubscribes. refCount: 1 -> 0. Source STOPS.
setTimeout(() => subA.unsubscribe(), 4000);

// T=6s: Sub C subscribes. refCount: 0 -> 1. Source RESTARTS from 0.
setTimeout(() => {
  source$.subscribe(v => console.log('C:', v));
}, 6000);
```

Output timeline:
```
T=1s:  [Source] 0  -> A: 0, B: 0
T=2s:  [Source] 1  -> A: 1, B: 1
T=2.5s: B unsubscribes
T=3s:  [Source] 2  -> A: 2
T=4s:  A unsubscribes, source stops
T=7s:  [Source] 0  -> C: 0  (restarted from 0)
```

With `shareReplay(1)`:
```typescript
const source$ = interval(1000).pipe(
  tap(n => console.log(`[Source] ${n}`)),
  shareReplay({ bufferSize: 1, refCount: true })
);

// Same lifecycle as share(), BUT:
// When Sub C subscribes at T=6s, it immediately gets the last value
// from the buffer (if the Subject hasn't reset)
```

## Debugging Multicasting Issues

### Symptom: Duplicate HTTP Requests

```typescript
// PROBLEM: shareReplay placed after the subscribe point
// Each component's async pipe creates a fresh chain
@Injectable({ providedIn: 'root' })
export class DataService {
  private readonly http = inject(HttpClient);

  getData(): Observable<Data> {
    // BAD: Every call to getData() returns a new Observable chain
    return this.http.get<Data>('/api/data').pipe(
      shareReplay(1) // This shareReplay is per-call, not shared!
    );
  }
}

// FIX: Store the shared Observable as a property
@Injectable({ providedIn: 'root' })
export class DataService {
  private readonly http = inject(HttpClient);

  // GOOD: Single Observable instance, shared across all consumers
  readonly data$ = this.http.get<Data>('/api/data').pipe(
    shareReplay(1)
  );
}
```

### Symptom: Stale Data After Error

```typescript
// PROBLEM: shareReplay caches errors by default (resetOnError: false)
const data$ = this.http.get('/api/data').pipe(
  shareReplay(1)
);

// If the first request fails, ALL future subscribers get the cached error
// even if the network has recovered.

// FIX: Use share() with explicit error reset
const data$ = this.http.get('/api/data').pipe(
  share({
    connector: () => new ReplaySubject(1),
    resetOnError: true,     // Allow retry on error
    resetOnComplete: false, // Keep cached value
    resetOnRefCountZero: false
  })
);
```

### Symptom: Memory Leak from `shareReplay`

```typescript
// PROBLEM: shareReplay(1) in a lazy-loaded feature module
// The Observable never tears down because refCount defaults to false
@Injectable({ providedIn: 'root' })
export class AnalyticsService {
  // This subscription lives forever, even after the feature is destroyed
  readonly events$ = fromEvent(document, 'click').pipe(
    map(e => ({ type: 'click', timestamp: Date.now() })),
    shareReplay(1) // refCount: false by default
  );
}

// FIX: Use refCount: true or share() with resetOnRefCountZero
readonly events$ = fromEvent(document, 'click').pipe(
  map(e => ({ type: 'click', timestamp: Date.now() })),
  shareReplay({ bufferSize: 1, refCount: true })
);
```

## Edge Cases and Gotchas

1. **Synchronous emissions and `share()`.** If the source emits synchronously (e.g., `of(1, 2, 3).pipe(share())`), a second subscriber added synchronously after the first will miss all emissions because the source completes during the first subscription before the second subscriber attaches. Use `shareReplay` if synchronous sources need to be shared.

2. **`async` pipe with `@if` and `share()`.** When Angular's `async` pipe is inside an `@if` block, the pipe unsubscribes when the element is removed from the DOM. If this is the only subscriber, refCount drops to 0 and the source resets. When the element reappears, a fresh execution triggers (e.g., a new HTTP request). Use `shareReplay({ bufferSize: 1, refCount: true })` or `share({ resetOnRefCountZero: () => timer(300) })` to add a grace period.

3. **Timer-based `resetOnRefCountZero` and silent errors.** Using `resetOnRefCountZero: () => timer(1000)` creates a window where the source is still alive but has no subscribers. If the source errors during this window, the error is swallowed because there are no subscribers to receive it.

4. **Multiple `shareReplay` in a pipe chain.** Each `shareReplay` creates its own `ReplaySubject`. Chaining them creates nested multicasting layers, which is almost never what you want.

    ```typescript
    // BAD: Two separate caching layers, confusing behavior
    const data$ = source$.pipe(
      shareReplay(1),
      map(transform),
      shareReplay(1) // Creates a second, independent ReplaySubject
    );
    ```

5. **Completion vs refCount zero are different events.** If a source completes while there are active subscribers, the completion is cached (with `resetOnComplete: false`). But if refCount drops to zero before completion, the state resets entirely and the next subscriber starts fresh.

## Common Mistakes

1. **Creating new Observable chains in methods instead of storing them as properties.** If `getData()` returns `this.http.get(...).pipe(shareReplay(1))`, every call to `getData()` creates a new pipeline with its own `shareReplay`. The fix is to assign the Observable to a class property once.

2. **Using `shareReplay(1)` without `refCount: true` in component-scoped services.** The default `refCount: false` means the inner subscription never tears down. For root-level singleton services this is fine. For anything else, use `refCount: true`.

3. **Placing `share()` before other operators.** Operators after `share()` run once per subscriber, not once for all. Put `share()` or `shareReplay()` at the end of the pipeline.

    ```typescript
    // BAD: map runs for each subscriber
    const shared$ = source$.pipe(
      share(),
      map(v => expensiveTransform(v)) // Runs N times for N subscribers
    );

    // GOOD: map runs once, result is shared
    const shared$ = source$.pipe(
      map(v => expensiveTransform(v)), // Runs once
      share()
    );
    ```

4. **Forgetting that `share()` resets on completion.** HTTP Observables complete after emitting. With `share()` (not `shareReplay`), a late subscriber triggers a new request because the Subject reset on completion.

    ```typescript
    // BAD for HTTP caching
    const data$ = this.http.get('/api/data').pipe(share());
    // Second subscriber after completion triggers another request

    // GOOD for HTTP caching
    const data$ = this.http.get('/api/data').pipe(shareReplay(1));
    // Second subscriber gets the cached value
    ```

5. **Not understanding that `shareReplay` caches errors.** By default, `resetOnError` is `false`. If the source errors, all future subscribers receive the cached error immediately without retrying.

6. **Using `share()` on already-hot Observables.** Subjects, EventEmitters, and other hot sources are already multicast. Adding `share()` adds unnecessary overhead.

7. **Subscribing to a `share()`'d Observable synchronously in multiple places expecting all to get the synchronous value.** `share()` with a plain Subject does not replay. Only the subscriber present at emission time receives the value.

---

## Best Practices

1. **Use `shareReplay(1)` for HTTP caching in root-level services.** The default `refCount: false` is acceptable for singletons that live for the app's lifetime.

2. **Use `shareReplay({ bufferSize: 1, refCount: true })` for feature-level or component-level caching.** This ensures teardown when no one is listening.

3. **Use `share()` for event streams where replay is not needed.** WebSocket messages, DOM events, and real-time data streams typically should not replay old values to late subscribers.

4. **Use `share({ resetOnRefCountZero: () => timer(ms) })` for timed caches.** This prevents the "thundering herd" problem where multiple quick subscribe/unsubscribe cycles each trigger a fresh upstream subscription.

5. **Store multicasted Observables as class properties, not in methods.** This ensures all consumers reference the same Observable instance.

6. **Prefer Signals over multicasting for synchronous state.** If you are sharing a synchronous value across components, a Signal is simpler and more efficient than `BehaviorSubject` + `shareReplay`.

7. **Place `share()`/`shareReplay()` as the last operator in the pipeline.** Everything before it runs once (shared). Everything after it runs per subscriber.

## Real-World Use Cases

### 1. API Response Cache with TTL

```typescript
@Injectable({ providedIn: 'root' })
export class CatalogService {
  private readonly http = inject(HttpClient);

  // Cache catalog for 60 seconds after last subscriber leaves
  readonly catalog$ = this.http.get<Product[]>('/api/catalog').pipe(
    share({
      connector: () => new ReplaySubject<Product[]>(1),
      resetOnRefCountZero: () => timer(60_000),
      resetOnComplete: false,
      resetOnError: () => timer(5_000) // Retry sooner on failure
    })
  );
}
```

### 2. Shared Search Results Stream

```typescript
@Injectable({ providedIn: 'root' })
export class SearchService {
  private readonly http = inject(HttpClient);
  private readonly query$ = new BehaviorSubject<string>('');

  readonly results$ = this.query$.pipe(
    debounceTime(300),
    distinctUntilChanged(),
    switchMap(query =>
      query.length < 2
        ? of([])
        : this.http.get<SearchResult[]>(`/api/search?q=${query}`)
    ),
    shareReplay(1)
  );

  search(query: string): void {
    this.query$.next(query);
  }
}
```

### 3. Permission Guard with Shared Auth State

```typescript
@Injectable({ providedIn: 'root' })
export class AuthService {
  private readonly http = inject(HttpClient);

  // Shared auth state: single request, cached, retryable on error
  readonly user$ = this.http.get<AuthUser>('/api/auth/me').pipe(
    share({
      connector: () => new ReplaySubject<AuthUser>(1),
      resetOnError: true,      // Allow retry after auth failure
      resetOnComplete: false,  // Keep cached user
      resetOnRefCountZero: false // Never tear down auth state
    })
  );

  readonly isAdmin$ = this.user$.pipe(
    map(user => user.roles.includes('admin'))
  );

  readonly permissions$ = this.user$.pipe(
    map(user => user.permissions)
  );
}

// Functional guard using the shared auth stream
export const adminGuard: CanActivateFn = () => {
  const auth = inject(AuthService);
  return auth.isAdmin$.pipe(
    take(1),
    tap(isAdmin => {
      if (!isAdmin) {
        inject(Router).navigate(['/unauthorized']);
      }
    })
  );
};
```

## Quick Reference: Choosing the Right Operator

| Scenario | Operator | Config |
|----------|----------|--------|
| Share live event stream | `share()` | Default |
| Cache HTTP response (root service) | `shareReplay(1)` | Default |
| Cache HTTP response (feature service) | `shareReplay({...})` | `{ bufferSize: 1, refCount: true }` |
| Cache with TTL | `share({...})` | `resetOnRefCountZero: () => timer(ms)` |
| Cache but retry on error | `share({...})` | `connector: () => new ReplaySubject(1), resetOnError: true` |
| All subscribers ready before start | `connectable()` | Manual `connect()` call |
| Share with initial value | `share({...})` | `connector: () => new BehaviorSubject(initial)` |

---

> **Key Takeaway:** Use `shareReplay(1)` for HTTP caching in singleton services, `shareReplay({ bufferSize: 1, refCount: true })` for component-level sharing, and `share()` with its configuration object for fine-grained control over reset behavior, error handling, and cache TTL.
