# 18.2 Observable Basics

> Observables are the core primitive of RxJS: lazy, push-based collections that can emit zero or more values over time and optionally complete or error.

---

## What Is an Observable

An Observable is a function that sets up a data source and emits values to an Observer over time. Unlike Promises, Observables are:

- **Lazy:** They do not execute until someone subscribes.
- **Multicast-capable:** They can emit zero, one, or many values.
- **Cancellable:** Subscriptions can be unsubscribed to stop execution and clean up resources.
- **Composable:** Operators transform, filter, and combine Observables into pipelines.

```typescript
import { Observable } from 'rxjs';

// An Observable is just a function that takes a subscriber
const numbers$ = new Observable<number>(subscriber => {
  subscriber.next(1);
  subscriber.next(2);
  subscriber.next(3);
  subscriber.complete();
});

// Nothing happens until we subscribe
numbers$.subscribe({
  next: value => console.log(value),     // 1, 2, 3
  complete: () => console.log('Done')    // Done
});
```

The `$` suffix is a naming convention for Observable variables. It is not required by the language but is universally used in Angular codebases to distinguish Observables from plain values.

---

## The Observer Interface

An Observer is the consumer of an Observable. It defines callbacks for three notification types:

```typescript
interface Observer<T> {
  next: (value: T) => void;
  error: (err: any) => void;
  complete: () => void;
}
```

| Notification | Purpose | Terminates stream? |
|-------------|---------|-------------------|
| `next` | Delivers a value | No |
| `error` | Delivers an error | Yes |
| `complete` | Signals no more values | Yes |

Rules:
- After `error` or `complete`, no more `next` calls are delivered.
- `error` and `complete` are mutually exclusive. A stream ends with one or neither, never both.
- All three callbacks are optional when subscribing.

### Shorthand Subscribe

You do not have to pass a full Observer object. RxJS accepts several shorthand forms:

```typescript
// Full Observer object
source$.subscribe({
  next: val => console.log(val),
  error: err => console.error(err),
  complete: () => console.log('Done')
});

// Just the next callback (most common)
source$.subscribe(val => console.log(val));

// Next and error only
source$.subscribe({
  next: val => console.log(val),
  error: err => console.error(err)
});
```

**Deprecated pattern (RxJS 7+):** Passing multiple positional arguments to `subscribe()` was deprecated in RxJS 7 and removed in RxJS 8:

```typescript
// Bad Practice: deprecated positional arguments
source$.subscribe(
  val => console.log(val),      // next
  err => console.error(err),    // error
  () => console.log('Done')     // complete
);

// Good: use the Observer object
source$.subscribe({
  next: val => console.log(val),
  error: err => console.error(err),
  complete: () => console.log('Done')
});
```

## Creating Observables

### The `new Observable` Constructor

The constructor gives you full control over what the Observable emits and when.

```typescript
import { Observable } from 'rxjs';

const timer$ = new Observable<number>(subscriber => {
  let count = 0;
  const id = setInterval(() => {
    subscriber.next(count++);
  }, 1000);

  // Teardown: runs when subscriber unsubscribes or stream completes/errors
  return () => {
    clearInterval(id);
    console.log('Interval cleared');
  };
});

const subscription = timer$.subscribe(val => console.log(val));

// Stop after 5 seconds
setTimeout(() => subscription.unsubscribe(), 5000);
// Logs: 0, 1, 2, 3, 4, then "Interval cleared"
```

The function returned from the constructor callback is the **teardown logic**. It runs when:
- The subscriber calls `unsubscribe()`
- The Observable calls `subscriber.complete()`
- The Observable calls `subscriber.error()`

### `of()` - Emit Fixed Values

Creates an Observable that emits each argument in sequence, then completes.

```typescript
import { of } from 'rxjs';

of(1, 2, 3).subscribe({
  next: v => console.log(v),     // 1, 2, 3
  complete: () => console.log('Done')  // Done
});

// Single value (common for returning default/fallback data)
of([]).subscribe(v => console.log(v)); // []

// Emits synchronously and completes immediately
```

Use `of()` when you need to wrap a static value into an Observable, often for fallback values in `catchError`:

```typescript
this.http.get('/api/data').pipe(
  catchError(() => of([])) // Return empty array on error
);
```

### `from()` - Convert to Observable

Converts an array, iterable, Promise, or another Observable into an Observable.

```typescript
import { from } from 'rxjs';

// From array: emits each element individually
from([10, 20, 30]).subscribe(v => console.log(v)); // 10, 20, 30

// From Promise: emits the resolved value, then completes
from(fetch('/api/users').then(r => r.json()))
  .subscribe(users => console.log(users));

// From string (iterable): emits each character
from('hello').subscribe(c => console.log(c)); // h, e, l, l, o

// From generator
function* fibonacci() {
  let a = 0, b = 1;
  while (true) {
    yield a;
    [a, b] = [b, a + b];
  }
}
from(fibonacci()).pipe(take(8)).subscribe(v => console.log(v));
// 0, 1, 1, 2, 3, 5, 8, 13
```

Key difference from `of()`: `from([1,2,3])` emits `1`, `2`, `3` individually. `of([1,2,3])` emits `[1,2,3]` as a single value.

### `fromEvent()` - DOM Events

Creates an Observable from DOM events.

```typescript
import { fromEvent } from 'rxjs';

// Click events
const clicks$ = fromEvent(document, 'click');
clicks$.subscribe(event => console.log('Clicked at:', event.clientX));

// Keyboard events
const keydowns$ = fromEvent<KeyboardEvent>(document, 'keydown');
keydowns$.subscribe(event => console.log('Key:', event.key));

// Input events (typed)
const input = document.querySelector('input')!;
const input$ = fromEvent<Event>(input, 'input');
input$.pipe(
  map(event => (event.target as HTMLInputElement).value)
).subscribe(value => console.log('Input:', value));
```

In Angular components, use `fromEvent` with `ElementRef` or `ViewChild`:

```typescript
@Component({
  template: `<button #btn>Click me</button>`
})
export class ClickComponent implements AfterViewInit {
  private btn = viewChild<ElementRef>('btn');
  private destroyRef = inject(DestroyRef);

  ngAfterViewInit() {
    fromEvent(this.btn()!.nativeElement, 'click').pipe(
      takeUntilDestroyed(this.destroyRef)
    ).subscribe(() => console.log('Clicked'));
  }
}
```

### `interval()` and `timer()`

```typescript
import { interval, timer } from 'rxjs';

// interval: emits sequential numbers at a fixed time interval
// Starts at 0, increments by 1 each emission
interval(1000).subscribe(n => console.log(n));
// 0 (at 1s), 1 (at 2s), 2 (at 3s), ...

// timer: emits 0 after a delay, then completes
timer(3000).subscribe(() => console.log('3 seconds passed'));

// timer with period: emits after initial delay, then at intervals
timer(2000, 500).subscribe(n => console.log(n));
// 0 (at 2s), 1 (at 2.5s), 2 (at 3s), ...
```

`interval()` never completes on its own. Always pair it with an operator like `take()`, `takeUntil()`, or `takeUntilDestroyed()` to prevent memory leaks.

### `defer()` - Lazy Creation

Creates the Observable only when someone subscribes. This is useful when the Observable creation depends on a value that changes over time or involves a side effect.

```typescript
import { defer, of } from 'rxjs';

// Without defer: timestamp is captured at creation time
const eager$ = of(Date.now());
// Both subscriptions see the SAME timestamp
eager$.subscribe(t => console.log(t)); // 1710500000000
setTimeout(() => eager$.subscribe(t => console.log(t)), 1000); // 1710500000000

// With defer: timestamp is captured at subscription time
const lazy$ = defer(() => of(Date.now()));
// Each subscription gets a FRESH timestamp
lazy$.subscribe(t => console.log(t)); // 1710500000000
setTimeout(() => lazy$.subscribe(t => console.log(t)), 1000); // 1710500001000
```

Common Angular use case: deferring a conditional Observable:

```typescript
const data$ = defer(() =>
  this.useCache ? of(this.cachedData) : this.http.get('/api/data')
);
```

### `EMPTY`, `NEVER`, and `throwError()`

Special Observables for specific scenarios:

```typescript
import { EMPTY, NEVER, throwError } from 'rxjs';

// EMPTY: completes immediately without emitting any values
EMPTY.subscribe({
  next: () => console.log('Never called'),
  complete: () => console.log('Complete') // Immediately
});

// NEVER: never emits, never completes, never errors
// Useful for testing or keeping a stream alive
NEVER.subscribe({
  next: () => console.log('Never called'),
  complete: () => console.log('Never called')
});

// throwError: emits an error immediately
throwError(() => new Error('Something broke')).subscribe({
  error: err => console.error(err.message) // "Something broke"
});
```

Note: `throwError` takes a factory function (not a direct error value) since RxJS 7. This ensures a fresh error stack trace per subscription:

```typescript
// Bad Practice: passing error directly (deprecated)
throwError(new Error('fail'));

// Good: factory function
throwError(() => new Error('fail'));
```

### `range()` and `generate()`

Two lesser-used creation operators for numeric sequences:

```typescript
import { range, generate } from 'rxjs';

// range: emit a sequence of numbers
range(1, 5).subscribe(v => console.log(v)); // 1, 2, 3, 4, 5

// generate: loop-like Observable with state, condition, and iterate
generate(
  1,                    // initial state
  x => x <= 32,        // condition
  x => x * 2           // iterate
).subscribe(v => console.log(v)); // 1, 2, 4, 8, 16, 32
```

## Subscription Lifecycle

### The Subscription Object

`subscribe()` returns a `Subscription` object that represents the ongoing execution.

```typescript
import { interval } from 'rxjs';

const subscription = interval(500).subscribe(n => console.log(n));

// The subscription is active
console.log(subscription.closed); // false

// Stop receiving values and trigger cleanup
subscription.unsubscribe();
console.log(subscription.closed); // true
```

### Composite Subscriptions

You can group subscriptions together so they can be unsubscribed at once:

```typescript
import { Subscription, interval, fromEvent } from 'rxjs';

const parent = new Subscription();

parent.add(interval(1000).subscribe(n => console.log('Timer:', n)));
parent.add(fromEvent(document, 'click').subscribe(e => console.log('Click')));

// Unsubscribe all at once
parent.unsubscribe();
```

However, in modern Angular, prefer `takeUntilDestroyed()` or `DestroyRef` over manual subscription management:

```typescript
@Component({ ... })
export class MyComponent {
  private destroyRef = inject(DestroyRef);

  constructor() {
    interval(1000).pipe(
      takeUntilDestroyed(this.destroyRef)
    ).subscribe(n => console.log(n));
  }
}
```

### Teardown and the `finalize` Operator

Teardown logic runs when a subscription ends (complete, error, or unsubscribe). Use `finalize` to add teardown logic without modifying the Observable source:

```typescript
import { interval, finalize, take } from 'rxjs';

interval(1000).pipe(
  take(3),
  finalize(() => console.log('Cleanup: stream ended'))
).subscribe({
  next: v => console.log(v),
  complete: () => console.log('Complete')
});
// 0, 1, 2, Complete, Cleanup: stream ended
```

`finalize` runs after `complete` or `error`, and also on unsubscribe. It is the operator equivalent of the teardown function returned from `new Observable()`.

## Observable vs Promise

Observables and Promises both handle asynchronous operations, but they are fundamentally different:

| Feature | Observable | Promise |
|---------|-----------|---------|
| Execution | Lazy (runs on subscribe) | Eager (runs immediately on creation) |
| Values | Zero, one, or many | Exactly one (resolve) or error (reject) |
| Cancellation | `unsubscribe()` | Not cancellable (AbortController is external) |
| Operators | Rich operator library (200+) | `.then()`, `.catch()`, `.finally()` |
| Multicasting | Configurable (cold/hot) | Always multicast (shared result) |
| Error handling | `catchError` operator, retryable | `.catch()`, not retryable |

### Laziness

```typescript
// Promise: the HTTP call fires IMMEDIATELY
const promise = fetch('/api/users'); // Request is already in flight

// Observable: nothing happens yet
const obs$ = this.http.get('/api/users'); // No request made

// Request fires NOW when we subscribe
obs$.subscribe(users => console.log(users));
```

This laziness is critical for composability. You can pass Observables around, combine them with operators, and nothing executes until the final subscription.

### Multiple Values

```typescript
// Promise: resolves once
const promise = new Promise(resolve => {
  resolve(1);
  resolve(2); // Ignored! Promise already resolved
});

// Observable: emits many values
const obs$ = new Observable(subscriber => {
  subscriber.next(1); // Delivered
  subscriber.next(2); // Delivered
  subscriber.next(3); // Delivered
  subscriber.complete();
});
```

### Cancellation

```typescript
// Observable: cancel an in-flight HTTP request
const sub = this.http.get('/api/slow-endpoint').subscribe(data => {
  console.log(data);
});

// User navigates away: cancel the request
sub.unsubscribe(); // XHR is aborted, no response processing

// Promise: cannot cancel
const promise = fetch('/api/slow-endpoint');
// There is no promise.cancel()
// The request completes even if you no longer need the result
```

### Converting Between Them

```typescript
import { from, firstValueFrom, lastValueFrom } from 'rxjs';

// Promise -> Observable
const obs$ = from(somePromise);

// Observable -> Promise (gets the first emitted value)
const value = await firstValueFrom(source$);

// Observable -> Promise (gets the last emitted value before complete)
const finalValue = await lastValueFrom(source$);
```

`firstValueFrom` and `lastValueFrom` replaced the deprecated `toPromise()` in RxJS 7. They throw an `EmptyError` if the Observable completes without emitting, unless you provide a `defaultValue`:

```typescript
const value = await firstValueFrom(EMPTY, { defaultValue: null });
// Returns null instead of throwing
```

## Cold vs Hot Observables (Revisited)

### Cold: Each Subscriber Gets Its Own Execution

```typescript
import { Observable } from 'rxjs';

const cold$ = new Observable<number>(subscriber => {
  console.log('Side effect: creating producer');
  const random = Math.random();
  subscriber.next(random);
  subscriber.complete();
});

cold$.subscribe(v => console.log('A:', v));
// Side effect: creating producer
// A: 0.7291

cold$.subscribe(v => console.log('B:', v));
// Side effect: creating producer
// B: 0.3847  <-- different value, different execution
```

Cold Observables include:
- `of()`, `from()`, `interval()`, `timer()`
- `HttpClient.get()`, `.post()`, etc.
- `new Observable()` (by default)

### Hot: Shared Producer, Existing Stream

```typescript
import { Subject } from 'rxjs';

const hot$ = new Subject<number>();

hot$.subscribe(v => console.log('A:', v));

hot$.next(1); // A: 1

hot$.subscribe(v => console.log('B:', v));

hot$.next(2); // A: 2, B: 2
// B missed the value 1 because it subscribed late
```

Hot Observables include:
- `Subject`, `BehaviorSubject`, `ReplaySubject`
- `fromEvent()` (DOM events exist regardless of subscription)
- Multicasted Observables (`share()`, `shareReplay()`)

### Making Cold Observables Hot

```typescript
import { interval, share, shareReplay } from 'rxjs';

// Cold: each subscriber triggers a new interval
const cold$ = interval(1000);

// Hot: all subscribers share one interval
const hot$ = cold$.pipe(share());

// Hot with replay: new subscribers get the last N values
const hotReplay$ = cold$.pipe(shareReplay(1));
```

This matters in Angular because subscribing to an `HttpClient` Observable multiple times triggers multiple HTTP requests:

```typescript
// Bad Practice: two HTTP requests
const users$ = this.http.get<User[]>('/api/users');
users$.subscribe(u => this.userList = u);   // Request 1
users$.subscribe(u => this.userCount = u.length); // Request 2

// Good: share the request
const users$ = this.http.get<User[]>('/api/users').pipe(
  shareReplay(1)
);
users$.subscribe(u => this.userList = u);   // Request 1
users$.subscribe(u => this.userCount = u.length); // Shared, no extra request

// Better: use toSignal for template consumption
const users = toSignal(this.http.get<User[]>('/api/users'), { initialValue: [] });
```

## Observables in Angular

### HttpClient Returns Observables

Angular's `HttpClient` returns cold Observables. Each subscription triggers a new HTTP request.

```typescript
@Component({ ... })
export class UserListComponent {
  private http = inject(HttpClient);

  // Option 1: toSignal (preferred for simple cases)
  users = toSignal(this.http.get<User[]>('/api/users'), { initialValue: [] });

  // Option 2: Observable with AsyncPipe
  users$ = this.http.get<User[]>('/api/users');
  // Template: @for (user of users$ | async; track user.id) { ... }

  // Option 3: Manual subscribe (least preferred)
  users: User[] = [];
  private destroyRef = inject(DestroyRef);

  constructor() {
    this.http.get<User[]>('/api/users').pipe(
      takeUntilDestroyed(this.destroyRef)
    ).subscribe(users => this.users = users);
  }
}
```

### Router Events Are Observables

```typescript
@Component({ ... })
export class AppComponent {
  private router = inject(Router);
  private route = inject(ActivatedRoute);

  // Route parameters
  userId = toSignal(
    this.route.paramMap.pipe(map(params => params.get('id'))),
    { initialValue: null }
  );

  // Query parameters
  search = toSignal(
    this.route.queryParamMap.pipe(map(params => params.get('q'))),
    { initialValue: '' }
  );

  // Navigation events
  isNavigating = toSignal(
    this.router.events.pipe(
      filter(e => e instanceof NavigationStart || e instanceof NavigationEnd),
      map(e => e instanceof NavigationStart)
    ),
    { initialValue: false }
  );
}
```

### Form Value Changes Are Observables

```typescript
@Component({ ... })
export class SearchComponent {
  searchControl = new FormControl('');

  results = toSignal(
    this.searchControl.valueChanges.pipe(
      debounceTime(300),
      distinctUntilChanged(),
      switchMap(term => this.searchService.search(term ?? '')),
      catchError(() => of([]))
    ),
    { initialValue: [] }
  );
}
```

### Signal Interop with `toSignal()` and `toObservable()`

Angular provides two functions to bridge Observables and Signals:

```typescript
import { toSignal, toObservable } from '@angular/core/rxjs-interop';

@Component({ ... })
export class InteropComponent {
  private http = inject(HttpClient);

  // Observable -> Signal
  // toSignal subscribes automatically and unsubscribes on destroy
  data = toSignal(this.http.get<Data>('/api/data'));
  // Type: Signal<Data | undefined> (undefined until first emission)

  // With initial value to avoid undefined
  data = toSignal(this.http.get<Data>('/api/data'), { initialValue: null });
  // Type: Signal<Data | null>

  // Signal -> Observable
  count = signal(0);
  count$ = toObservable(this.count);
  // Emits whenever count changes

  // Common pattern: signal drives an Observable pipeline
  searchTerm = signal('');
  results = toSignal(
    toObservable(this.searchTerm).pipe(
      debounceTime(300),
      switchMap(term => this.http.get(`/api/search?q=${term}`))
    ),
    { initialValue: [] }
  );
}
```

`toSignal()` options:

| Option | Effect |
|--------|--------|
| `initialValue` | Value before first emission (removes `undefined` from type) |
| `requireSync` | Asserts the Observable emits synchronously (errors if it does not) |
| `manualCleanup` | Does not auto-unsubscribe on destroy (use with caution) |
| `equal` | Custom equality function for signal updates |

If the source Observable errors, reading the resulting signal will throw that error. Handle errors in the pipeline before `toSignal()`:

```typescript
// Bad Practice: error propagates to signal, reading it throws
const data = toSignal(this.http.get('/api/data'));

// Good: catch errors before converting to signal
const data = toSignal(
  this.http.get('/api/data').pipe(
    catchError(() => of(null))
  ),
  { initialValue: null }
);
```

## Error Handling Basics

Errors terminate the Observable stream. Once an error is emitted, no more values are delivered.

```typescript
const source$ = new Observable<number>(subscriber => {
  subscriber.next(1);
  subscriber.next(2);
  subscriber.error(new Error('Something failed'));
  subscriber.next(3); // Never delivered
});

source$.subscribe({
  next: v => console.log(v),      // 1, 2
  error: err => console.error(err) // Error: Something failed
});
```

### Catching Errors in the Pipeline

```typescript
import { of, catchError } from 'rxjs';

this.http.get<User[]>('/api/users').pipe(
  catchError(err => {
    console.error('Request failed:', err);
    return of([]); // Return fallback Observable
  })
).subscribe(users => console.log(users));
```

`catchError` replaces the errored Observable with the Observable you return. The stream continues with the replacement rather than terminating.

### Retrying Failed Operations

```typescript
import { retry, timer } from 'rxjs';

this.http.get('/api/data').pipe(
  retry({
    count: 3,
    delay: (error, retryCount) => timer(retryCount * 1000) // Exponential-ish backoff
  })
).subscribe(data => console.log(data));
```

## Common Mistakes

- **Subscribing without unsubscribing.** Subscriptions to long-lived Observables (events, intervals, WebSockets, router params) leak memory if not cleaned up. Use `takeUntilDestroyed()`, `AsyncPipe`, or `toSignal()` in Angular components. HTTP requests complete naturally, so they do not leak, but it is still good practice to manage them.

- **Subscribing multiple times to cold Observables unintentionally.** Each `subscribe()` on an `HttpClient` Observable triggers a new HTTP request. Use `shareReplay(1)` to share the result, or convert to a signal with `toSignal()`.

- **Using `subscribe()` inside `subscribe()`.** Nested subscriptions are imperative and lead to pyramid-of-doom code. Use flattening operators instead:

```typescript
// Bad Practice: nested subscribe
this.route.params.subscribe(params => {
  this.http.get(`/api/users/${params['id']}`).subscribe(user => {
    this.http.get(`/api/orders/${user.orderId}`).subscribe(order => {
      console.log(order);
    });
  });
});

// Good: flat pipeline with switchMap
this.route.params.pipe(
  switchMap(params => this.http.get<User>(`/api/users/${params['id']}`)),
  switchMap(user => this.http.get<Order>(`/api/orders/${user.orderId}`)),
  takeUntilDestroyed()
).subscribe(order => console.log(order));
```

- **Ignoring errors.** An unhandled error in an Observable kills the stream silently. Always provide an `error` callback or use `catchError` in the pipeline.

- **Confusing `of([1,2,3])` with `from([1,2,3])`.** `of` emits the entire array as a single value. `from` emits each element individually. Use the one that matches your intent.

- **Using deprecated `toPromise()`.** Use `firstValueFrom()` or `lastValueFrom()` instead, which have clearer semantics and better error handling for empty Observables.

- **Returning long-lived Observables from route resolvers.** Angular's router waits for resolver Observables to complete before activating the route. If you return an Observable that never completes (like Firebase `valueChanges()`), navigation hangs. Pipe through `take(1)` or `first()` to ensure completion.

---

## Best Practices

- **Use `toSignal()` for template consumption** in Angular 21. It auto-subscribes, auto-unsubscribes, and integrates with signal-based change detection.

- **Keep `subscribe()` callbacks thin.** Side effects only (update UI, trigger navigation, log). Business logic belongs in the `pipe()` operator chain.

- **Name Observables with the `$` suffix.** This convention instantly signals to other developers that the variable is an Observable and requires subscription handling.

- **Prefer declarative streams over imperative subscriptions.** Build data pipelines with operators and let Angular's template consume them, rather than subscribing in component code and assigning to properties.

- **Use `catchError` to return fallback Observables** instead of letting errors terminate the stream. This is especially important for Observables that drive UI state.

- **Use `defer()` when Observable creation has side effects** or depends on values that change between subscription time and creation time.

- **Always clean up long-lived subscriptions.** `takeUntilDestroyed()` is the standard Angular pattern. For non-component contexts, use `DestroyRef` or manual `Subscription` management.

## Real-World Use Cases

### Polling an API

```typescript
@Component({ ... })
export class StatusComponent {
  private http = inject(HttpClient);

  // Poll every 30 seconds, starting immediately
  status = toSignal(
    timer(0, 30_000).pipe(
      switchMap(() => this.http.get<SystemStatus>('/api/status')),
      retry({ count: 3, delay: 2000 }),
      catchError(() => of({ healthy: false, message: 'Unreachable' }))
    ),
    { initialValue: { healthy: true, message: 'Checking...' } }
  );
}
```

### Debounced Search with Loading State

```typescript
@Component({ ... })
export class SearchComponent {
  searchControl = new FormControl('');
  private http = inject(HttpClient);

  private searchAction$ = this.searchControl.valueChanges.pipe(
    debounceTime(300),
    distinctUntilChanged(),
    filter((term): term is string => term !== null && term.length >= 2)
  );

  loading = signal(false);

  results = toSignal(
    this.searchAction$.pipe(
      tap(() => this.loading.set(true)),
      switchMap(term =>
        this.http.get<Result[]>(`/api/search?q=${term}`).pipe(
          catchError(() => of([])),
          finalize(() => this.loading.set(false))
        )
      )
    ),
    { initialValue: [] }
  );
}
```

### Combining Multiple Data Sources

```typescript
@Component({ ... })
export class DashboardComponent {
  private http = inject(HttpClient);
  private route = inject(ActivatedRoute);

  private projectId = toSignal(
    this.route.paramMap.pipe(map(p => p.get('projectId')!))
  );

  dashboard = toSignal(
    toObservable(computed(() => this.projectId())).pipe(
      filter((id): id is string => id != null),
      switchMap(id => forkJoin({
        project: this.http.get<Project>(`/api/projects/${id}`),
        members: this.http.get<User[]>(`/api/projects/${id}/members`),
        tasks: this.http.get<Task[]>(`/api/projects/${id}/tasks`)
      })),
      catchError(() => of(null))
    ),
    { initialValue: null }
  );
}
```

> **Key Takeaway:** Observables are lazy, cancellable, multi-value streams that form the backbone of asynchronous data handling in Angular. Use them with operators for complex async flows, convert to Signals with `toSignal()` for template consumption, and always manage subscription lifecycle to prevent memory leaks.
