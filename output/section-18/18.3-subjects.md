# 18.3 Subjects

> A Subject is both an Observable and an Observer: it can emit values to multiple subscribers and be used as a bridge between imperative code and reactive streams.

---

## What Is a Subject

A regular Observable is unicast: each subscriber gets its own independent execution. A Subject is multicast: it maintains a list of subscribers and pushes the same values to all of them simultaneously.

```typescript
import { Subject } from 'rxjs';

const subject = new Subject<number>();

// Subscriber A
subject.subscribe(v => console.log('A:', v));

// Subscriber B
subject.subscribe(v => console.log('B:', v));

subject.next(1);
// A: 1
// B: 1

subject.next(2);
// A: 2
// B: 2

subject.complete();
```

A Subject implements both interfaces:
- **Observable:** You can call `.subscribe()` on it, `.pipe()` operators through it.
- **Observer:** You can call `.next()`, `.error()`, and `.complete()` on it.

This dual nature makes Subjects the bridge between imperative "push a value" code and declarative Observable pipelines.

---

## Subject Variants

RxJS provides four types of Subjects, each with different behavior around late subscribers and value retention.

### Subject (Plain)

The base Subject. Subscribers only receive values emitted *after* they subscribe. No buffering, no replay, no initial value.

```typescript
import { Subject } from 'rxjs';

const subject = new Subject<string>();

subject.next('before'); // No one is listening, this value is lost

subject.subscribe(v => console.log('A:', v));

subject.next('hello'); // A: hello

subject.subscribe(v => console.log('B:', v));

subject.next('world');
// A: world
// B: world
// B never received "hello" because it subscribed after that emission
```

Use plain Subject when:
- You only care about future values, not past ones.
- You are modeling events (click, action, trigger) where history is irrelevant.

### BehaviorSubject

Requires an initial value and always holds a "current value." New subscribers immediately receive the current value upon subscription.

```typescript
import { BehaviorSubject } from 'rxjs';

const subject = new BehaviorSubject<number>(0); // Initial value: 0

console.log(subject.getValue()); // 0 (synchronous access)

subject.subscribe(v => console.log('A:', v)); // A: 0 (receives current value)

subject.next(1); // A: 1
subject.next(2); // A: 2

subject.subscribe(v => console.log('B:', v)); // B: 2 (receives current value)

subject.next(3);
// A: 3
// B: 3
```

Key characteristics:
- Must be initialized with a value.
- `getValue()` returns the current value synchronously (use sparingly).
- Late subscribers receive the most recent value immediately.

Use BehaviorSubject when:
- You need a "current state" that always has a value.
- New subscribers need the latest state immediately.
- You are building a simple state store in a service.

### ReplaySubject

Buffers a specified number of emissions and replays them to new subscribers. Does not require an initial value.

```typescript
import { ReplaySubject } from 'rxjs';

// Buffer the last 2 values
const subject = new ReplaySubject<string>(2);

subject.next('a');
subject.next('b');
subject.next('c');

// Late subscriber receives the last 2 buffered values
subject.subscribe(v => console.log('Late:', v));
// Late: b
// Late: c
```

```typescript
// Buffer with time window: replay values from the last 500ms
const subject = new ReplaySubject<number>(Infinity, 500);

subject.next(1); // t=0ms

setTimeout(() => {
  subject.next(2); // t=200ms
}, 200);

setTimeout(() => {
  subject.next(3); // t=600ms
}, 600);

setTimeout(() => {
  // t=700ms: subscribes, replays values from t=200ms to t=700ms
  subject.subscribe(v => console.log('Late:', v));
  // Late: 2 (within 500ms window)
  // Late: 3 (within 500ms window)
  // Value 1 has expired (emitted 700ms ago, outside 500ms window)
}, 700);
```

Constructor parameters:

| Parameter | Default | Description |
|-----------|---------|-------------|
| `bufferSize` | `Infinity` | Maximum number of values to buffer |
| `windowTime` | `Infinity` | Maximum age (ms) of buffered values |

Use ReplaySubject when:
- Late subscribers need more than just the latest value.
- You want to cache a fixed number of emissions.
- You need time-windowed replay (e.g., replay the last 5 seconds of events).

### AsyncSubject

Only emits the *last* value, and only when it completes. If it errors, no value is emitted.

```typescript
import { AsyncSubject } from 'rxjs';

const subject = new AsyncSubject<number>();

subject.subscribe(v => console.log('A:', v));

subject.next(1); // Not emitted yet
subject.next(2); // Not emitted yet
subject.next(3); // Not emitted yet

subject.complete();
// A: 3  (only the last value, emitted on complete)
```

```typescript
// If error occurs, no value is emitted
const subject = new AsyncSubject<number>();
subject.subscribe({
  next: v => console.log('Value:', v),
  error: err => console.error('Error:', err)
});

subject.next(1);
subject.error(new Error('fail'));
// Error: fail  (no value emitted)
```

Use AsyncSubject when:
- You only care about the final result of a computation.
- You are modeling a Promise-like behavior (single eventual value on completion).
- Cache a resolved value: late subscribers receive the last value even after completion.

---

## Comparison Table

| Feature | Subject | BehaviorSubject | ReplaySubject | AsyncSubject |
|---------|---------|-----------------|---------------|--------------|
| Initial value required | No | Yes | No | No |
| Late subscriber receives | Nothing | Current value | Buffered values | Last value (on complete) |
| `getValue()` | No | Yes | No | No |
| Emits on | Each `next()` | Each `next()` | Each `next()` | Only on `complete()` |
| Common Angular use | Event bus, actions | State management | Caching, shareReplay | Rare |

## Subjects in Angular Services

The most common Angular pattern for Subjects is using them inside services to share state or events across components.

### State Management with BehaviorSubject

```typescript
@Injectable({ providedIn: 'root' })
export class AuthService {
  // Private: only the service can push values
  private currentUser$ = new BehaviorSubject<User | null>(null);

  // Public: consumers can only subscribe, not push
  readonly user$ = this.currentUser$.asObservable();

  // Convenience signal for template consumption
  readonly user = toSignal(this.user$, { initialValue: null });

  login(credentials: Credentials): Observable<User> {
    return this.http.post<User>('/api/login', credentials).pipe(
      tap(user => this.currentUser$.next(user))
    );
  }

  logout(): void {
    this.currentUser$.next(null);
  }
}
```

### Event Bus with Plain Subject

```typescript
@Injectable({ providedIn: 'root' })
export class NotificationService {
  private notifications$ = new Subject<Notification>();

  // Expose as Observable
  readonly onNotification$ = this.notifications$.asObservable();

  show(message: string, type: 'success' | 'error' | 'info' = 'info'): void {
    this.notifications$.next({ message, type, timestamp: Date.now() });
  }
}
```

```typescript
// Consumer component
@Component({ ... })
export class NotificationBarComponent {
  private notificationService = inject(NotificationService);

  // Convert to signal for template
  latestNotification = toSignal(
    this.notificationService.onNotification$.pipe(
      switchMap(n => timer(5000).pipe(
        mapTo(null),
        startWith(n)
      ))
    )
  );
}
```

### Caching with ReplaySubject

```typescript
@Injectable({ providedIn: 'root' })
export class ConfigService {
  private http = inject(HttpClient);

  // ReplaySubject(1) caches the last value for late subscribers
  private config$ = new ReplaySubject<AppConfig>(1);
  private loaded = false;

  loadConfig(): Observable<AppConfig> {
    if (!this.loaded) {
      this.loaded = true;
      this.http.get<AppConfig>('/api/config').subscribe(
        config => this.config$.next(config)
      );
    }
    return this.config$.asObservable();
  }
}
```

A simpler alternative using `shareReplay`:

```typescript
@Injectable({ providedIn: 'root' })
export class ConfigService {
  private http = inject(HttpClient);

  // shareReplay(1) uses a ReplaySubject internally
  readonly config$ = this.http.get<AppConfig>('/api/config').pipe(
    shareReplay(1)
  );
}
```

## The `asObservable()` Pattern

Exposing a Subject directly allows external code to push values into it, breaking encapsulation. Use `asObservable()` to expose a read-only view.

```typescript
// Bad Practice: exposing the Subject directly
@Injectable({ providedIn: 'root' })
export class CounterService {
  count$ = new BehaviorSubject<number>(0); // Anyone can call .next()!
}

// Some rogue component:
this.counterService.count$.next(999); // Bypasses all business logic

// Good: expose as Observable
@Injectable({ providedIn: 'root' })
export class CounterService {
  private _count$ = new BehaviorSubject<number>(0);
  readonly count$ = this._count$.asObservable(); // Read-only

  increment(): void {
    this._count$.next(this._count$.getValue() + 1);
  }
}
```

## Subjects vs Signals for State Management

In Angular 21, Signals have replaced many BehaviorSubject patterns for simple state management. Here is a direct comparison:

### BehaviorSubject Pattern (Legacy)

```typescript
@Injectable({ providedIn: 'root' })
export class CartService {
  private items$ = new BehaviorSubject<CartItem[]>([]);
  readonly cartItems$ = this.items$.asObservable();
  readonly itemCount$ = this.cartItems$.pipe(map(items => items.length));
  readonly total$ = this.cartItems$.pipe(
    map(items => items.reduce((sum, item) => sum + item.price * item.quantity, 0))
  );

  addItem(item: CartItem): void {
    const current = this.items$.getValue();
    this.items$.next([...current, item]);
  }

  removeItem(id: string): void {
    const current = this.items$.getValue();
    this.items$.next(current.filter(i => i.id !== id));
  }
}
```

### Signal Pattern (Modern Angular 21)

```typescript
@Injectable({ providedIn: 'root' })
export class CartService {
  private items = signal<CartItem[]>([]);
  readonly cartItems = this.items.asReadonly();
  readonly itemCount = computed(() => this.items().length);
  readonly total = computed(() =>
    this.items().reduce((sum, item) => sum + item.price * item.quantity, 0)
  );

  addItem(item: CartItem): void {
    this.items.update(items => [...items, item]);
  }

  removeItem(id: string): void {
    this.items.update(items => items.filter(i => i.id !== id));
  }
}
```

### When to Still Use Subjects

Signals replace BehaviorSubject for most synchronous state management. However, Subjects remain necessary for:

| Scenario | Why Subjects |
|----------|-------------|
| Event streams (notifications, actions) | Signals represent current state, not events |
| Multi-value emissions over time | Signals hold a single current value |
| Integration with RxJS operators | `debounceTime`, `switchMap`, etc. |
| WebSocket message streams | Continuous async data |
| Cross-component event communication | Event bus pattern |
| Complex async orchestration | Retry, race conditions, backpressure |

**Rule of thumb:** If you are modeling *state* (a value that changes), use Signals. If you are modeling *events* (things that happen), use Subjects.

## Void Subjects for Event Signaling

When a Subject represents an event with no payload, use `Subject<void>`:

```typescript
@Injectable({ providedIn: 'root' })
export class RefreshService {
  private refresh$ = new Subject<void>();
  readonly onRefresh$ = this.refresh$.asObservable();

  triggerRefresh(): void {
    this.refresh$.next(); // No argument needed for void
  }
}
```

```typescript
// Consumer
@Component({ ... })
export class DataListComponent {
  private refreshService = inject(RefreshService);
  private http = inject(HttpClient);

  data = toSignal(
    this.refreshService.onRefresh$.pipe(
      startWith(undefined), // Trigger initial load
      switchMap(() => this.http.get<Item[]>('/api/items'))
    ),
    { initialValue: [] }
  );
}
```

## Subject Completion and Error Behavior

### Completion Stops the Subject

Once a Subject completes, it cannot emit any more values. All future subscribers receive only the completion notification.

```typescript
const subject = new Subject<number>();

subject.subscribe(v => console.log('A:', v));

subject.next(1); // A: 1
subject.complete();

subject.next(2); // Nothing happens, subject is complete

subject.subscribe({
  next: v => console.log('B:', v),
  complete: () => console.log('B: complete')
});
// B: complete  (immediately, no values)
```

### Error Terminates the Subject

An error terminates the Subject. All future subscribers receive only the error.

```typescript
const subject = new Subject<number>();

subject.subscribe({
  next: v => console.log('A:', v),
  error: err => console.error('A error:', err.message)
});

subject.next(1); // A: 1
subject.error(new Error('broken'));
// A error: broken

subject.subscribe({
  next: v => console.log('B:', v),
  error: err => console.error('B error:', err.message)
});
// B error: broken  (immediately re-emits the error)
```

### BehaviorSubject After Completion

```typescript
const subject = new BehaviorSubject<number>(0);
subject.next(42);
subject.complete();

// Late subscriber gets completion, NOT the value 42
subject.subscribe({
  next: v => console.log('Value:', v),
  complete: () => console.log('Complete')
});
// Complete  (not "Value: 42")
```

This is a common gotcha. If you need late subscribers to receive the last value even after completion, use `ReplaySubject(1)` instead.

## Multicasting with Subjects

Subjects are the underlying mechanism for RxJS multicasting operators. Understanding this connection clarifies when to use each.

### Manual Multicasting

```typescript
import { interval, Subject } from 'rxjs';

const source$ = interval(1000); // Cold: each sub gets its own interval
const subject = new Subject<number>();

// Pipe the cold Observable through the Subject
source$.subscribe(subject); // Subject subscribes to source

// Now multiple subscribers share the same interval
subject.subscribe(v => console.log('A:', v));
subject.subscribe(v => console.log('B:', v));
// A and B receive the same values at the same time
```

### Operator-Based Multicasting

RxJS provides operators that handle the Subject creation internally:

```typescript
import { interval, share, shareReplay } from 'rxjs';

// share() uses a plain Subject internally
const shared$ = interval(1000).pipe(share());

// shareReplay(1) uses a ReplaySubject(1) internally
const sharedReplay$ = interval(1000).pipe(shareReplay(1));

// connectable() for manual connect/disconnect control
import { connectable } from 'rxjs';
const connectable$ = connectable(interval(1000), {
  connector: () => new Subject()
});
connectable$.subscribe(v => console.log('A:', v));
connectable$.connect(); // Starts the source
```

| Operator | Internal Subject | Ref counting | Replay |
|----------|-----------------|--------------|--------|
| `share()` | Subject | Yes | No |
| `shareReplay(n)` | ReplaySubject(n) | Configurable | Yes (n values) |
| `connectable()` | Configurable | Manual | Depends on Subject |

### share() Behavior

`share()` creates a Subject, subscribes to the source when the first subscriber arrives, and unsubscribes from the source when the last subscriber leaves:

```typescript
const shared$ = this.http.get('/api/data').pipe(
  share()
);

shared$.subscribe(d => console.log('A:', d)); // Triggers HTTP request
shared$.subscribe(d => console.log('B:', d)); // Shares same request (if still in flight)
// After both unsubscribe, the next subscribe triggers a NEW request
```

### shareReplay() Behavior

`shareReplay(1)` keeps the last value cached even after all subscribers leave:

```typescript
const cached$ = this.http.get('/api/config').pipe(
  shareReplay(1)
);

cached$.subscribe(d => console.log('A:', d)); // Triggers HTTP request
// A unsubscribes

cached$.subscribe(d => console.log('B:', d)); // Gets cached value, no new request
```

To reset the cache when all subscribers leave, use `shareReplay({ bufferSize: 1, refCount: true })`.

## Common Mistakes

- **Exposing Subjects publicly.** Always use `asObservable()` or `asReadonly()` (for signals) to prevent external code from pushing values directly. This breaks encapsulation and makes debugging state changes nearly impossible.

- **Using `getValue()` in reactive pipelines.** `BehaviorSubject.getValue()` is a synchronous escape hatch that breaks the reactive paradigm. Use it only for imperative operations (like building a new value from the old one). Never use it inside `pipe()` or `subscribe()`.

```typescript
// Bad Practice: getValue in reactive code
source$.pipe(
  map(val => val + this.counter$.getValue()) // Breaks reactivity
);

// Good: combine streams
combineLatest([source$, this.counter$]).pipe(
  map(([val, count]) => val + count)
);
```

- **Forgetting that completed Subjects are dead.** Once `complete()` or `error()` is called, the Subject cannot emit any more values. Do not call `complete()` on a BehaviorSubject that you plan to keep using.

- **Using BehaviorSubject when Subject suffices.** If you do not need late subscribers to receive the current value, use a plain Subject. BehaviorSubject forces an initial value that may not be meaningful (e.g., `new BehaviorSubject<User>(null!)` with a type assertion).

- **Not unsubscribing from Subject-based streams.** Subjects keep references to all subscribers. If a component subscribes but never unsubscribes, the Subject holds a reference to that component, preventing garbage collection.

- **Using ReplaySubject(Infinity) without thinking.** An unbounded buffer grows forever, consuming memory. Always set a reasonable `bufferSize` or `windowTime`.

- **Placing `takeUntil` before higher-order operators.** `takeUntil` must be the last operator before `subscribe()`. If placed before `switchMap` or `mergeMap`, the inner subscriptions created by those operators will not be cleaned up:

```typescript
// Bad Practice: takeUntil before switchMap leaks inner subscriptions
source$.pipe(
  takeUntilDestroyed(),
  switchMap(id => this.http.get(`/api/${id}`)) // Inner sub may leak
).subscribe();

// Good: takeUntil/takeUntilDestroyed as the last operator
source$.pipe(
  switchMap(id => this.http.get(`/api/${id}`)),
  takeUntilDestroyed()
).subscribe();
```

- **Using `shareReplay(1)` without `refCount`.** Without `refCount: true`, the source subscription persists even after all subscribers leave, causing a memory leak. Use `shareReplay({ bufferSize: 1, refCount: true })` when the cache should reset after all subscribers unsubscribe.

## Best Practices

- **Prefer Signals over BehaviorSubject** for synchronous state in Angular 21 services and components. Signals are simpler, have no subscription management, and integrate with Angular's change detection.

- **Use the private Subject + public Observable pattern** consistently. Name the private Subject with an underscore or descriptive suffix, and expose only the Observable.

- **Use `Subject<void>` for events without data.** This makes the intent clear: the event signals that something happened, not what happened.

- **Choose the right Subject variant.** Plain Subject for events. BehaviorSubject when you need current state (and Signals are not viable). ReplaySubject for caching. AsyncSubject for final-value semantics.

- **Prefer `shareReplay(1)` over manual ReplaySubject** for caching HTTP responses or computed values. It composes better with Observable pipelines.

- **Complete Subjects when the service is destroyed** (for non-root services). Root services live for the application lifetime, but scoped services should clean up:

```typescript
@Injectable()
export class ScopedService implements OnDestroy {
  private action$ = new Subject<string>();

  ngOnDestroy(): void {
    this.action$.complete(); // Release all subscribers
  }
}
```

## Real-World Use Cases

### Action Stream Pattern

A pattern where user actions drive data loading through a Subject:

```typescript
@Component({ ... })
export class ProductListComponent {
  private http = inject(HttpClient);

  // Action streams
  private categoryFilter$ = new BehaviorSubject<string>('all');
  private sortOrder$ = new BehaviorSubject<'asc' | 'desc'>('asc');
  private refresh$ = new Subject<void>();

  // Derived data stream combining all actions
  products = toSignal(
    combineLatest({
      category: this.categoryFilter$,
      sort: this.sortOrder$,
      trigger: this.refresh$.pipe(startWith(undefined))
    }).pipe(
      debounceTime(50), // Batch rapid changes
      switchMap(({ category, sort }) =>
        this.http.get<Product[]>('/api/products', {
          params: { category, sort }
        })
      ),
      catchError(() => of([]))
    ),
    { initialValue: [] }
  );

  filterByCategory(category: string): void {
    this.categoryFilter$.next(category);
  }

  toggleSort(): void {
    const current = this.sortOrder$.getValue();
    this.sortOrder$.next(current === 'asc' ? 'desc' : 'asc');
  }

  refresh(): void {
    this.refresh$.next();
  }
}
```

### Cross-Component Communication

```typescript
// Shared service
@Injectable({ providedIn: 'root' })
export class SidebarService {
  private _open$ = new BehaviorSubject<boolean>(false);
  readonly isOpen$ = this._open$.asObservable();
  readonly isOpen = toSignal(this.isOpen$, { initialValue: false });

  toggle(): void {
    this._open$.next(!this._open$.getValue());
  }

  open(): void { this._open$.next(true); }
  close(): void { this._open$.next(false); }
}

// Header component
@Component({
  template: `<button (click)="sidebar.toggle()">Menu</button>`
})
export class HeaderComponent {
  protected sidebar = inject(SidebarService);
}

// Sidebar component
@Component({
  template: `
    @if (sidebar.isOpen()) {
      <nav class="sidebar">...</nav>
    }
  `
})
export class SidebarComponent {
  protected sidebar = inject(SidebarService);
}
```

### WebSocket Message Stream

```typescript
@Injectable({ providedIn: 'root' })
export class ChatService {
  private messages$ = new ReplaySubject<ChatMessage>(50); // Buffer last 50 messages
  private socket: WebSocket | null = null;

  connect(roomId: string): void {
    this.socket = new WebSocket(`wss://chat.example.com/rooms/${roomId}`);

    this.socket.onmessage = (event) => {
      const message: ChatMessage = JSON.parse(event.data);
      this.messages$.next(message);
    };

    this.socket.onerror = () => {
      this.messages$.error(new Error('WebSocket connection failed'));
    };
  }

  getMessages(): Observable<ChatMessage> {
    return this.messages$.asObservable();
  }

  disconnect(): void {
    this.socket?.close();
  }
}
```

```typescript
// Chat component: late-joining users see the last 50 messages
@Component({ ... })
export class ChatComponent {
  private chatService = inject(ChatService);
  private destroyRef = inject(DestroyRef);

  messages: ChatMessage[] = [];

  constructor() {
    this.chatService.getMessages().pipe(
      takeUntilDestroyed(this.destroyRef)
    ).subscribe(msg => this.messages.push(msg));
  }
}
```

---

> **Key Takeaway:** Subjects bridge imperative code and reactive streams by acting as both Observable and Observer. Use plain Subject for events, BehaviorSubject when you need current state (or prefer Signals in Angular 21), ReplaySubject for caching, and always hide the Subject behind `asObservable()` to protect encapsulation.
