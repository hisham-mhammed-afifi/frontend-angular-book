# 18.5 Transformation Operators

> Transformation operators take values emitted by an Observable and reshape, accumulate, flatten, or restructure them into new forms before passing them downstream.

---

## Overview

Transformation operators sit in the middle of a reactive pipeline, converting upstream values into something more useful for downstream consumers. They range from simple value mapping to complex higher-order Observable flattening.

| Category | Operators |
|----------|----------|
| Value mapping | `map`, `pluck` (deprecated), `mapTo` (deprecated) |
| Accumulation | `scan`, `reduce`, `mergeScan` |
| Flattening (higher-order) | `switchMap`, `mergeMap`, `concatMap`, `exhaustMap` |
| Buffering | `buffer`, `bufferTime`, `bufferCount`, `bufferWhen`, `bufferToggle` |
| Windowing | `window`, `windowTime`, `windowCount` |
| Grouping | `groupBy` |
| Sequential pairs | `pairwise` |
| Miscellaneous | `toArray`, `expand`, `timestamp`, `timeInterval` |

---

## Value Mapping

### `map(project)`

The most fundamental operator. Transforms each emitted value using a projection function.

```typescript
import { of, map } from 'rxjs';

of(1, 2, 3).pipe(
  map(x => x * 10)
).subscribe(v => console.log(v));
// 10, 20, 30
```

```
Source:  ──1──2──3──|
map(x => x * 10)
Output:  ──10─20─30─|
```

Angular use cases:

```typescript
// Extract data from HTTP response
this.http.get<ApiResponse<User[]>>('/api/users').pipe(
  map(response => response.data)
);

// Transform route params
this.route.paramMap.pipe(
  map(params => params.get('id')!)
);

// Project form values
this.form.valueChanges.pipe(
  map(values => ({ ...values, email: values.email.toLowerCase() }))
);
```

`map` is synchronous and 1:1. Each input value produces exactly one output value. For async transformations (where each input produces an Observable), use the higher-order mapping operators.

### Deprecated: `pluck` and `mapTo`

`pluck` (deprecated in RxJS 7.x, removed in 8) extracted nested properties. Use `map` instead:

```typescript
// Deprecated
source$.pipe(pluck('user', 'name'));

// Replacement
source$.pipe(map(v => v.user.name));

// With optional chaining for safety
source$.pipe(map(v => v?.user?.name));
```

`mapTo` (deprecated in RxJS 7.x, removed in 8) replaced each value with a constant. Use `map` instead:

```typescript
// Deprecated
clicks$.pipe(mapTo('clicked'));

// Replacement
clicks$.pipe(map(() => 'clicked'));
```

## Accumulation Operators

### `scan(accumulator, seed)`

Like `Array.reduce()`, but emits the intermediate accumulated value after each source emission.

```typescript
import { of, scan } from 'rxjs';

of(1, 2, 3, 4, 5).pipe(
  scan((acc, value) => acc + value, 0)
).subscribe(v => console.log(v));
// 1, 3, 6, 10, 15
```

```
Source:  ──1──2──3──4──5──|
scan((acc, v) => acc + v, 0)
Output:  ──1──3──6──10─15─|
```

`scan` is the reactive equivalent of maintaining state. Each emission carries the full accumulated state.

**Without a seed:** If you omit the seed, the first source value passes through unchanged and becomes the initial accumulator:

```typescript
of(1, 2, 3).pipe(
  scan((acc, value) => acc + value) // No seed
).subscribe(v => console.log(v));
// 1, 3, 6 (first value is passed through as-is)
```

Always provide a seed when the accumulator type differs from the source type, or when you need a predictable initial value.

Angular state management with `scan`:

```typescript
// Action-based state management
interface State {
  items: Item[];
  loading: boolean;
}

const initialState: State = { items: [], loading: false };

type Action =
  | { type: 'LOAD' }
  | { type: 'LOADED'; items: Item[] }
  | { type: 'ADD'; item: Item };

const state$ = actions$.pipe(
  scan((state: State, action: Action) => {
    switch (action.type) {
      case 'LOAD':
        return { ...state, loading: true };
      case 'LOADED':
        return { ...state, items: action.items, loading: false };
      case 'ADD':
        return { ...state, items: [...state.items, action.item] };
      default:
        return state;
    }
  }, initialState)
);
```

### `reduce(accumulator, seed)`

Like `scan`, but only emits the final accumulated value when the source completes.

```typescript
import { of, reduce } from 'rxjs';

of(1, 2, 3, 4, 5).pipe(
  reduce((acc, value) => acc + value, 0)
).subscribe(v => console.log(v));
// 15 (only the final value)
```

```
Source:  ──1──2──3──4──5──|
reduce((acc, v) => acc + v, 0)
Output:  ─────────────────15|
```

`reduce` waits for completion. If the source never completes (like `interval` or `fromEvent`), `reduce` never emits. Use `scan` for ongoing streams and `reduce` for finite sequences.

### `pairwise()`

Emits the current and previous values as a pair. The first source emission is buffered and not emitted (since there is no "previous" yet).

```typescript
import { of, pairwise } from 'rxjs';

of(1, 2, 3, 4).pipe(
  pairwise()
).subscribe(v => console.log(v));
// [1, 2], [2, 3], [3, 4]
```

```
Source:  ──1──2──3──4──|
pairwise()
Output:  ─────[1,2]─[2,3]─[3,4]─|
```

Angular use case: detecting scroll direction:

```typescript
fromEvent(window, 'scroll').pipe(
  map(() => window.scrollY),
  pairwise(),
  map(([prev, curr]) => curr > prev ? 'down' : 'up'),
  distinctUntilChanged(),
  takeUntilDestroyed()
).subscribe(direction => this.scrollDirection.set(direction));
```

Route change tracking:

```typescript
this.router.events.pipe(
  filter(e => e instanceof NavigationEnd),
  pairwise(),
  takeUntilDestroyed()
).subscribe(([prev, curr]) => {
  this.analytics.trackNavigation(prev.urlAfterRedirects, curr.urlAfterRedirects);
});
```

### `mergeScan(accumulator, seed, concurrent?)`

Like `scan`, but the accumulator returns an Observable. Each accumulated result is flattened like `mergeMap`.

```typescript
import { of, mergeScan, delay } from 'rxjs';

// Accumulate with async operations
of(1, 2, 3).pipe(
  mergeScan((acc, value) =>
    of(acc + value).pipe(delay(100)),
    0
  )
).subscribe(v => console.log(v));
// 1, 3, 6
```

Use `mergeScan` when your accumulation involves an async operation (like an HTTP request that depends on the previous result).

## Higher-Order Mapping Operators

These are the most important transformation operators. They handle the case where each source value maps to a new Observable (an "inner Observable"), and they determine how those inner Observables are managed.

### The Problem They Solve

```typescript
// Without higher-order mapping: nested subscribe (BAD)
searchInput$.subscribe(term => {
  this.http.get(`/api/search?q=${term}`).subscribe(results => {
    this.results = results;  // Race condition! Stale results may overwrite newer ones
  });
});

// With higher-order mapping: flat pipeline (GOOD)
searchInput$.pipe(
  switchMap(term => this.http.get(`/api/search?q=${term}`))
).subscribe(results => {
  this.results = results;  // Always the latest result
});
```

### `switchMap(project)`

When a new outer value arrives, **cancel the current inner Observable** and switch to the new one.

```typescript
import { interval, switchMap, timer, map, take } from 'rxjs';

interval(3000).pipe(
  switchMap(i => timer(0, 1000).pipe(
    map(j => `Outer ${i}, Inner ${j}`),
    take(5)
  ))
).subscribe(v => console.log(v));
// Outer 0, Inner 0
// Outer 0, Inner 1
// Outer 0, Inner 2  (at 3s, new outer arrives)
// Outer 1, Inner 0  (previous inner cancelled)
// Outer 1, Inner 1
// ...
```

```
Outer:  ──A──────────B──────────C──|
          switchMap(x => inner(x))
Inner A:  ──a1──a2──a3
Inner B:            ──b1──b2──b3
Inner C:                      ──c1──c2──c3──|

Output: ──a1──a2──b1──b2──c1──c2──c3──|
                  ↑ A cancelled     ↑ B cancelled
```

**Use when:** You only care about the latest value. Previous in-flight operations should be cancelled.

Angular use cases:
- Search typeahead (cancel previous search when user types)
- Route parameter changes (cancel previous data load)
- Form field changes that trigger API calls

```typescript
// Search typeahead
this.searchControl.valueChanges.pipe(
  debounceTime(300),
  distinctUntilChanged(),
  switchMap(term => this.http.get<Result[]>(`/api/search?q=${term}`).pipe(
    catchError(() => of([]))
  ))
);

// Route-driven data loading
this.route.paramMap.pipe(
  map(params => params.get('id')!),
  switchMap(id => this.http.get<Product>(`/api/products/${id}`))
);
```

### `mergeMap(project, concurrent?)`

Subscribe to all inner Observables concurrently. No cancellation. All results are merged into the output.

```typescript
import { of, mergeMap, delay } from 'rxjs';

of(1, 2, 3).pipe(
  mergeMap(x => of(x * 10).pipe(delay(x * 100)))
).subscribe(v => console.log(v));
// 10 (after 100ms), 20 (after 200ms), 30 (after 300ms)
// Order depends on inner Observable timing, not source order
```

```
Outer:  ──A──B──C──|
          mergeMap(x => inner(x))
Inner A:  ──a1──a2──a3──|
Inner B:     ──b1──b2──b3──|
Inner C:        ──c1──c2──c3──|

Output: ──a1──b1──a2──c1──b2──a3──c2──b3──c3──|
         (interleaved, all run concurrently)
```

**Use when:** All inner Observables should run concurrently and their order does not matter.

The optional `concurrent` parameter limits parallelism:

```typescript
// Process at most 3 uploads concurrently
fileList$.pipe(
  mergeMap(file => this.uploadService.upload(file), 3)
);
```

Angular use cases:
- Bulk operations (upload multiple files)
- Sending analytics events (fire and forget)
- Processing items where order does not matter

```typescript
// Bulk delete
selectedIds$.pipe(
  mergeMap(ids => ids.map(id => this.http.delete(`/api/items/${id}`))),
  mergeMap(req => req) // Flatten the array of Observables
);
```

### `concatMap(project)`

Subscribe to inner Observables one at a time, in order. Wait for each to complete before subscribing to the next.

```typescript
import { of, concatMap, delay } from 'rxjs';

of(3, 1, 2).pipe(
  concatMap(x => of(x).pipe(delay(x * 100)))
).subscribe(v => console.log(v));
// 3 (after 300ms), 1 (after 400ms), 2 (after 600ms)
// Always in source order, regardless of inner timing
```

```
Outer:  ──A──B──C──|
          concatMap(x => inner(x))
Inner A:  ──a1──a2──|
Inner B:              ──b1──b2──|
Inner C:                          ──c1──c2──|

Output: ──a1──a2──b1──b2──c1──c2──|
         (sequential, B waits for A to complete)
```

**Use when:** Order matters and operations must execute sequentially.

Angular use cases:
- Sequential API calls where each depends on the previous
- Ordered write operations
- Queue-based processing

```typescript
// Sequential saves (order matters for audit log)
saveActions$.pipe(
  concatMap(action => this.http.post('/api/save', action))
);

// Chained API calls
this.http.get<User>('/api/user').pipe(
  concatMap(user => this.http.get<Preferences>(`/api/prefs/${user.id}`)),
  concatMap(prefs => this.http.get<Theme>(`/api/themes/${prefs.themeId}`))
);
```

**Warning:** If inner Observables never complete, `concatMap` queues up all subsequent outer values indefinitely, consuming memory.

### `exhaustMap(project)`

While an inner Observable is active, **ignore** new outer values. Only subscribe to a new inner Observable after the current one completes.

```typescript
import { fromEvent, exhaustMap, timer, map } from 'rxjs';

fromEvent(button, 'click').pipe(
  exhaustMap(() => timer(2000).pipe(map(() => 'Done')))
).subscribe(v => console.log(v));
// Rapid clicks during the 2s window are ignored
// Only the first click triggers the timer
```

```
Outer:  ──A──B──C──────D──|
          exhaustMap(x => inner(x))
Inner A:  ──a1──a2──|
                      ──d1──d2──|

Output: ──a1──a2──d1──d2──|
              ↑ B and C ignored (A still active)
```

**Use when:** You want to prevent duplicate operations from rapid triggers.

Angular use cases:
- Form submission (prevent double-submit)
- Login button clicks
- Any operation that should not be retriggered while in progress

```typescript
// Prevent double form submission
submitClick$.pipe(
  exhaustMap(() =>
    this.http.post('/api/orders', this.form.value).pipe(
      catchError(err => {
        this.error.set(err.message);
        return EMPTY;
      })
    )
  )
);
```

### Comparison Table

| Operator | New outer value while inner is active | Use case |
|----------|--------------------------------------|----------|
| `switchMap` | Cancel current inner, start new | Search, route params, latest-wins |
| `mergeMap` | Run both concurrently | Bulk operations, fire-and-forget |
| `concatMap` | Queue, wait for current to finish | Sequential writes, ordered processing |
| `exhaustMap` | Ignore new outer value | Form submit, prevent double-click |

### Decision Flowchart

```
Should previous work be cancelled?
  YES -> switchMap

Should operations run in parallel?
  YES -> mergeMap (with optional concurrency limit)

Must operations preserve order?
  YES -> concatMap

Should repeated triggers be ignored?
  YES -> exhaustMap
```

## Buffering Operators

Buffering operators collect emitted values into arrays and emit the arrays based on various strategies.

### `bufferTime(timeSpan, creationInterval?, maxBufferSize?)`

Collects values into an array and emits the array at fixed time intervals.

```typescript
import { interval, bufferTime } from 'rxjs';

interval(200).pipe(
  bufferTime(1000)
).subscribe(v => console.log(v));
// [0, 1, 2, 3] (at 1s)
// [4, 5, 6, 7, 8] (at 2s)
// ...
```

Angular use case: batch analytics events:

```typescript
this.analyticsEvents$.pipe(
  bufferTime(5000),
  filter(batch => batch.length > 0),
  switchMap(batch => this.http.post('/api/analytics/batch', batch)),
  takeUntilDestroyed()
).subscribe();
```

### `bufferCount(bufferSize, startBufferEvery?)`

Collects a fixed number of values, then emits them as an array.

```typescript
import { of, bufferCount } from 'rxjs';

of(1, 2, 3, 4, 5, 6, 7).pipe(
  bufferCount(3)
).subscribe(v => console.log(v));
// [1, 2, 3], [4, 5, 6], [7]
```

With overlapping buffers:

```typescript
of(1, 2, 3, 4, 5).pipe(
  bufferCount(3, 1) // Buffer size 3, start new buffer every 1 value
).subscribe(v => console.log(v));
// [1, 2, 3], [2, 3, 4], [3, 4, 5], [4, 5], [5]
```

### `buffer(closingNotifier)`

Collects values until the closing notifier Observable emits, then emits the buffer as an array.

```typescript
import { interval, fromEvent, buffer } from 'rxjs';

const source$ = interval(300);
const click$ = fromEvent(document, 'click');

// Collect interval values, emit on each click
source$.pipe(
  buffer(click$)
).subscribe(v => console.log('Buffered:', v));
```

### `bufferWhen(closingSelector)`

Like `buffer`, but the closing notifier is created by a factory function for each buffer.

```typescript
import { interval, bufferWhen } from 'rxjs';

interval(100).pipe(
  bufferWhen(() => interval(500 + Math.random() * 500))
).subscribe(v => console.log(v));
// Random-length buffers
```

### `bufferToggle(openings, closingSelector)`

Starts buffering when the openings Observable emits, stops when the closing Observable emits.

```typescript
import { interval, bufferToggle, EMPTY } from 'rxjs';

const source$ = interval(100);
const open$ = interval(500);

source$.pipe(
  bufferToggle(open$, () => interval(300))
).subscribe(v => console.log(v));
// Opens a buffer every 500ms, closes each 300ms later
```

## Windowing Operators

Window operators are like buffer operators, but instead of collecting values into arrays, they emit nested Observables.

### `window(windowBoundaries)`

```typescript
import { interval, window, fromEvent, mergeAll, take } from 'rxjs';

const source$ = interval(200);
const click$ = fromEvent(document, 'click');

source$.pipe(
  window(click$),
  mergeAll() // Flatten the inner Observables
).subscribe(v => console.log(v));
```

### `windowTime(windowTimeSpan)`

```typescript
import { interval, windowTime, mergeAll } from 'rxjs';

interval(100).pipe(
  windowTime(500),
  map(window$ => window$.pipe(toArray())),
  mergeAll()
).subscribe(v => console.log(v));
// [0, 1, 2, 3] every 500ms
```

### `windowCount(windowSize)`

```typescript
import { of, windowCount, mergeAll } from 'rxjs';

of(1, 2, 3, 4, 5, 6).pipe(
  windowCount(3),
  mergeAll()
).subscribe(v => console.log(v));
// 1, 2, 3, 4, 5, 6 (but grouped into windows of 3)
```

Windows are useful when you need to apply different operators to each group (e.g., `count`, `reduce`, `toArray`) without buffering everything into memory first.

## Grouping

### `groupBy(keySelector, elementSelector?)`

Splits a source Observable into multiple Observables, one per unique key.

```typescript
import { of, groupBy, mergeMap, toArray } from 'rxjs';

interface LogEntry {
  level: 'info' | 'warn' | 'error';
  message: string;
}

of(
  { level: 'info', message: 'Started' },
  { level: 'error', message: 'Failed' },
  { level: 'info', message: 'Retrying' },
  { level: 'warn', message: 'Slow response' },
  { level: 'error', message: 'Timeout' }
).pipe(
  groupBy(entry => entry.level),
  mergeMap(group$ => group$.pipe(
    toArray(),
    map(entries => ({ level: group$.key, entries }))
  ))
).subscribe(v => console.log(v));
// { level: 'info', entries: [...] }
// { level: 'error', entries: [...] }
// { level: 'warn', entries: [...] }
```

```
Source:  ──I──E──I──W──E──|
groupBy(x => x.level)

Group 'info':   ──I─────I──────────|
Group 'error':  ─────E────────E──|
Group 'warn':   ──────────W──────|
```

**Warning:** Each group is a new Observable. If the source is long-lived and keys are unbounded (like user IDs), groups accumulate forever. Use the `duration` parameter or pipe groups through `take`/`takeUntil` to limit their lifetime:

```typescript
groupBy(
  item => item.userId,
  { duration: group$ => timer(60000) } // Expire groups after 60s
)
```

## Other Transformation Operators

### `toArray()`

Collects all source emissions into an array and emits the array when the source completes.

```typescript
import { of, toArray } from 'rxjs';

of(3, 1, 4, 1, 5).pipe(
  toArray()
).subscribe(v => console.log(v));
// [3, 1, 4, 1, 5]
```

Like `reduce`, `toArray` waits for completion. It will never emit if the source never completes.

### `expand(project, concurrent?)`

Recursively projects each value into an Observable, subscribing to the output and feeding it back as input. Continues until the inner Observable returns `EMPTY`.

```typescript
import { of, expand, delay, take, EMPTY } from 'rxjs';

// Fetch paginated API results
function fetchPage(page: number): Observable<{ data: Item[]; nextPage: number | null }> {
  return this.http.get<PageResponse>(`/api/items?page=${page}`);
}

fetchPage(1).pipe(
  expand(response =>
    response.nextPage != null
      ? fetchPage(response.nextPage)
      : EMPTY
  ),
  map(response => response.data),
  reduce((all, page) => [...all, ...page], [] as Item[])
).subscribe(allItems => console.log(allItems));
```

`expand` is the recursive version of `mergeMap`. Each output is fed back in as a new input. Return `EMPTY` to stop recursion.

Angular use case: traversing a tree structure via API:

```typescript
// Fetch all child nodes recursively
this.http.get<TreeNode>(`/api/nodes/${rootId}`).pipe(
  expand(node =>
    node.childIds.length > 0
      ? from(node.childIds).pipe(
          mergeMap(id => this.http.get<TreeNode>(`/api/nodes/${id}`))
        )
      : EMPTY
  ),
  toArray()
);
```

### `timestamp()` and `timeInterval()`

Add timing metadata to emissions.

```typescript
import { of, timestamp, delay, concatMap, timeInterval, interval } from 'rxjs';

// timestamp: wraps each value with the current time
of('a', 'b', 'c').pipe(
  concatMap(v => of(v).pipe(delay(1000))),
  timestamp()
).subscribe(v => console.log(v));
// { value: 'a', timestamp: 1710500001000 }
// { value: 'b', timestamp: 1710500002000 }
// { value: 'c', timestamp: 1710500003000 }

// timeInterval: wraps each value with the time since the previous emission
interval(1000).pipe(
  timeInterval()
).subscribe(v => console.log(v));
// { value: 0, interval: 1003 }  (includes execution overhead)
// { value: 1, interval: 1001 }
```

### `materialize()` and `dematerialize()`

Convert stream notifications (next, error, complete) into `Notification` objects, and back.

```typescript
import { of, materialize, dematerialize, map } from 'rxjs';

of(1, 2, 3).pipe(
  materialize()
).subscribe(v => console.log(v));
// Notification { kind: 'N', value: 1, ... }
// Notification { kind: 'N', value: 2, ... }
// Notification { kind: 'N', value: 3, ... }
// Notification { kind: 'C', ... }  (complete)
```

Use case: inspecting or modifying error notifications without terminating the stream:

```typescript
source$.pipe(
  materialize(),
  map(notification => {
    if (notification.kind === 'E') {
      console.error('Error intercepted:', notification.error);
      // Transform error into a next value
      return new Notification('N', { error: notification.error.message });
    }
    return notification;
  }),
  dematerialize()
);
```

## Common Mistakes

- **Using `mergeMap` when `switchMap` is intended.** This is the most common higher-order mapping mistake. `mergeMap` does not cancel previous inner Observables, so stale HTTP responses can overwrite newer ones. Use `switchMap` for search typeaheads, route-based loading, and any "latest wins" scenario.

- **Using `switchMap` when `concatMap` is intended.** For write operations (POST, PUT, DELETE), `switchMap` can cancel in-flight saves, causing data loss. Use `concatMap` for sequential writes or `exhaustMap` to prevent duplicate submissions.

- **Forgetting that `reduce` and `toArray` wait for completion.** These operators never emit if the source never completes. Use `scan` for ongoing streams.

- **Nested subscribes instead of higher-order mapping.** Subscribing inside a `subscribe` callback creates unmanageable subscription chains and race conditions. Always use `switchMap`, `mergeMap`, `concatMap`, or `exhaustMap`.

```typescript
// Bad Practice: nested subscribe
source$.subscribe(id => {
  this.http.get(`/api/${id}`).subscribe(data => {
    this.data = data;
  });
});

// Good: flat pipeline
source$.pipe(
  switchMap(id => this.http.get(`/api/${id}`))
).subscribe(data => this.data = data);
```

- **Not handling errors inside higher-order mapping operators.** If the inner Observable errors and you have no `catchError` inside the `switchMap`/`mergeMap`, the entire outer stream terminates:

```typescript
// Bad Practice: outer stream dies on first error
source$.pipe(
  switchMap(id => this.http.get(`/api/${id}`)) // Error here kills everything
).subscribe(data => console.log(data));

// Good: catch errors inside the inner Observable
source$.pipe(
  switchMap(id => this.http.get(`/api/${id}`).pipe(
    catchError(err => {
      console.error(err);
      return EMPTY; // Skip this emission, keep outer stream alive
    })
  ))
).subscribe(data => console.log(data));
```

- **Using `scan` with mutable state.** Mutating the accumulator breaks referential transparency and can cause subtle bugs with change detection:

```typescript
// Bad Practice: mutating the accumulator
scan((acc, item) => {
  acc.push(item); // Mutates the SAME array reference
  return acc;
}, []);

// Good: return new reference
scan((acc, item) => [...acc, item], [] as Item[]);
```

- **Unbounded `groupBy` on long-lived streams.** Without a `duration` option, `groupBy` keeps a reference to every group key forever, leaking memory.

---

## Best Practices

- **Default to `switchMap` for read operations** (GET requests, data loading) and `exhaustMap` or `concatMap` for write operations (POST, PUT, DELETE). This prevents both race conditions and duplicate submissions.

- **Always handle errors inside higher-order mapping operators.** Place `catchError` inside the inner Observable pipeline to prevent the outer stream from dying.

- **Use `scan` for reactive state accumulation** instead of maintaining mutable state outside the stream. This keeps state transitions explicit and traceable.

- **Limit `mergeMap` concurrency** when processing many items. The second argument caps the number of concurrent inner subscriptions: `mergeMap(fn, 5)`.

- **Use `bufferTime` or `bufferCount` for batching** API calls or analytics events. Batching reduces HTTP overhead and improves performance.

- **Prefer `pairwise` over manual "previous value" tracking.** Instead of storing the last value in a variable, use `pairwise()` to get `[previous, current]` pairs declaratively.

- **Convert to Signals after transformation.** In Angular 21, pipe your transformations and use `toSignal()` at the end for template consumption:

```typescript
products = toSignal(
  this.categoryFilter$.pipe(
    switchMap(cat => this.http.get<Product[]>(`/api/products?cat=${cat}`)),
    map(products => products.filter(p => p.active)),
    scan((all, batch) => [...all, ...batch], [] as Product[])
  ),
  { initialValue: [] }
);
```

## Real-World Use Cases

### Autocomplete with Debounce and Cancellation

```typescript
@Component({ ... })
export class AutocompleteComponent {
  searchControl = new FormControl('');
  private http = inject(HttpClient);

  suggestions = toSignal(
    this.searchControl.valueChanges.pipe(
      debounceTime(300),
      distinctUntilChanged(),
      filter((term): term is string => term != null && term.length >= 2),
      switchMap(term =>
        this.http.get<string[]>(`/api/suggest?q=${term}`).pipe(
          catchError(() => of([]))
        )
      )
    ),
    { initialValue: [] }
  );
}
```

### Paginated Data Loading with `expand`

```typescript
@Injectable({ providedIn: 'root' })
export class PaginatedApiService {
  private http = inject(HttpClient);

  fetchAll<T>(endpoint: string): Observable<T[]> {
    return this.fetchPage<T>(endpoint, 1).pipe(
      expand(response =>
        response.meta.nextPage
          ? this.fetchPage<T>(endpoint, response.meta.nextPage)
          : EMPTY
      ),
      map(response => response.data),
      reduce((all, page) => [...all, ...page], [] as T[])
    );
  }

  private fetchPage<T>(endpoint: string, page: number) {
    return this.http.get<PaginatedResponse<T>>(`${endpoint}?page=${page}`);
  }
}
```

### Real-Time Dashboard with Scan-Based State

```typescript
@Component({ ... })
export class MetricsDashboardComponent {
  private ws = inject(WebSocketService);

  // Accumulate metrics over time with scan
  metrics = toSignal(
    this.ws.messages$.pipe(
      filter((msg): msg is MetricEvent => msg.type === 'metric'),
      scan((state: MetricsState, event: MetricEvent) => ({
        ...state,
        [event.metric]: {
          current: event.value,
          min: Math.min(state[event.metric]?.min ?? Infinity, event.value),
          max: Math.max(state[event.metric]?.max ?? -Infinity, event.value),
          history: [...(state[event.metric]?.history ?? []).slice(-99), event.value]
        }
      }), {} as MetricsState)
    ),
    { initialValue: {} as MetricsState }
  );
}
```

---

> **Key Takeaway:** Higher-order mapping operators (`switchMap`, `mergeMap`, `concatMap`, `exhaustMap`) are the most critical transformation operators in Angular. Choose the right one based on whether you need cancellation (switchMap), concurrency (mergeMap), ordering (concatMap), or debounce-like ignoring (exhaustMap). Always catch errors inside the inner Observable to keep the outer stream alive.
