# 18.9 Higher-Order Observables and Flattening Strategies

> A higher-order Observable emits other Observables instead of plain values. Flattening strategies determine how those inner Observables are subscribed to, managed, and merged into a single output stream. Choosing the wrong strategy is the single most common source of subtle bugs in Angular applications that use RxJS.

---

## What Is a Higher-Order Observable

A regular Observable emits values like numbers, strings, or objects. A higher-order Observable emits Observables as its values. The type signature is `Observable<Observable<T>>` instead of `Observable<T>`.

```typescript
import { of, map, interval, take } from 'rxjs';

// Regular Observable: emits numbers
const numbers$ = of(1, 2, 3);

// Higher-order Observable: emits Observables
const higher$ = numbers$.pipe(
  map(n => interval(1000).pipe(take(n)))
);
// higher$ emits:
//   Observable that emits 0              (take(1))
//   Observable that emits 0, 1           (take(2))
//   Observable that emits 0, 1, 2        (take(3))
```

Subscribing to a higher-order Observable gives you Observable objects, not their values. This is never what you want.

```typescript
// Without flattening: you get Observable objects, not their values
higher$.subscribe(inner$ => {
  console.log(inner$); // Observable { ... } -- not useful
});

// With flattening: you get the values from the inner Observables
higher$.pipe(
  mergeAll() // Subscribe to each inner Observable and merge their values
).subscribe(value => {
  console.log(value); // 0, 0, 1, 0, 1, 2 (interleaved values from all inners)
});
```

---

## Why Higher-Order Observables Matter in Angular

Higher-order Observables appear constantly in Angular applications because almost every asynchronous operation returns an Observable:

```typescript
// User types a search term -> map to HTTP request -> HTTP returns Observable
this.searchControl.valueChanges.pipe(
  map(term => this.http.get(`/api/search?q=${term}`))
  // Type: Observable<Observable<Result[]>>
  // Each keystroke produces a new HTTP Observable inside the outer stream
);
```

Without flattening, you have Observables inside Observables. The flattening strategy determines:
1. **When** inner Observables are subscribed to.
2. **Whether** previous inner Observables are cancelled.
3. **How many** inner Observables can run concurrently.
4. **What happens** when a new outer value arrives while an inner Observable is active.

---

## The Nested Subscribe Antipattern

Before learning the flattening operators, understand why manual subscription fails.

```typescript
// BAD PRACTICE: nested subscribe pyramid
this.route.params.subscribe(params => {
  this.http.get(`/api/users/${params['id']}`).subscribe(user => {
    this.http.get(`/api/orders?userId=${user.id}`).subscribe(orders => {
      this.http.get(`/api/recs?orderId=${orders[0].id}`).subscribe(recs => {
        this.recommendations = recs;
      });
    });
  });
});
```

| Problem | Explanation |
|---------|-------------|
| Memory leaks | Inner subscriptions are never unsubscribed. If the outer emits again (route change), old inner subscriptions remain active. |
| Race conditions | If `params` emits a new `id` before inner requests complete, both old and new requests race to update the same property. The UI shows whichever response arrives last, not the latest one. |
| No error propagation | An error in an inner Observable does not propagate to the outer subscription. You need separate error handlers at every level. |
| Unreadable | Four levels of nesting obscure the actual data flow. |
| Untestable | You cannot marble-test nested subscribes. |

**Refactored with higher-order operators:**

```typescript
// GOOD: flat pipeline with switchMap and concatMap
this.route.params.pipe(
  map(params => params['id']),
  switchMap(id => this.http.get<User>(`/api/users/${id}`)),
  concatMap(user => this.http.get<Order[]>(`/api/orders?userId=${user.id}`)),
  concatMap(orders => this.http.get<Rec[]>(`/api/recs?orderId=${orders[0].id}`)),
  takeUntilDestroyed()
).subscribe(recs => this.recommendations.set(recs));
```

One subscription. Automatic cancellation of stale requests (via `switchMap` on route params). Error propagation works. Marble-testable.

## The Four Flattening Strategies

Every flattening operator does three things:
1. Receives a value from the outer (source) Observable.
2. Creates an inner Observable from that value (via your projection function).
3. Subscribes to the inner Observable and emits its values to the output.

The difference is what happens when a **new outer value arrives while an inner Observable is still active**.

| Strategy | Operator | On new outer emission | Active inner subscriptions |
|----------|----------|----------------------|---------------------------|
| **Switch** | `switchMap` | Unsubscribe from current inner, subscribe to new inner | At most 1 |
| **Merge** | `mergeMap` | Subscribe to new inner without touching current ones | Unlimited (configurable) |
| **Concat** | `concatMap` | Queue the new inner, subscribe when current completes | Exactly 1, with an unbounded queue |
| **Exhaust** | `exhaustMap` | Ignore the new outer value entirely | Exactly 1, no queue |

---

## switchMap: Cancel Previous, Subscribe to Latest

`switchMap` unsubscribes from the current inner Observable and subscribes to a new one every time the outer emits. Only the latest inner Observable is active at any time.

### Text Marble Diagram

```
Outer:     --a--------b--------c--------|
               \        \        \
Inner(a):       --1--2--3--4              (unsubscribed when b arrives)
Inner(b):                --5--6--7        (unsubscribed when c arrives)
Inner(c):                         --8--9--|

Output:    ----1--2----5--6------8--9----|

Values 3, 4 from Inner(a) are never emitted. Values 7 from Inner(b) is never emitted.
```

### How Cancellation Works Internally

1. The outer Observable emits value `b`.
2. `switchMap` calls `.unsubscribe()` on the subscription to `Inner(a)`.
3. The teardown function of `Inner(a)` executes. For Angular's `HttpClient`, this calls `XMLHttpRequest.abort()`, cancelling the HTTP request at the network level. You can see this in Chrome DevTools Network tab where the request shows a "cancelled" status.
4. `switchMap` calls your projection function with `b` to create `Inner(b)`.
5. `switchMap` subscribes to `Inner(b)`.

**Critical detail:** This cancellation works because Angular's `HttpClient` uses XHR internally, and XHR supports abort. If you use `fetch()` wrapped in `from()`, unsubscription does NOT cancel the network request because Promises are not cancellable. The Observable wrapper simply stops listening to the Promise, but the request completes on the server side.

### Angular Use Cases

```typescript
// 1. Type-ahead search: cancel stale searches automatically
@Component({ /* ... */ })
export class SearchComponent {
  private http = inject(HttpClient);

  searchResults = toSignal(
    toObservable(this.searchTerm).pipe(
      debounceTime(300),
      distinctUntilChanged(),
      switchMap(term =>
        term.length < 2
          ? of([])
          : this.http.get<SearchResult[]>(`/api/search?q=${term}`).pipe(
              catchError(() => of([]))
            )
      )
      // Each new search term cancels the previous HTTP request.
      // No race condition: only the result for the latest term reaches the subscriber.
    ),
    { initialValue: [] }
  );
}

// 2. Route-driven data loading: cancel when user navigates
this.route.paramMap.pipe(
  map(params => params.get('id')!),
  switchMap(id => this.http.get<Product>(`/api/products/${id}`))
);

// 3. Signal-driven Observable pipeline
toObservable(this.selectedCategory).pipe(
  switchMap(cat => this.http.get<Item[]>(`/api/items?cat=${cat}`))
);
```

### When to Use switchMap

- Type-ahead / autocomplete search
- Route parameter changes triggering data loads
- Dropdown or filter changes
- Any read operation where only the latest result matters
- Polling with user-triggered refresh (cancel current cycle, start fresh)

### switchMap Gotchas

**1. Never use switchMap for write operations (POST/PUT/DELETE)**

```typescript
// BAD PRACTICE: switchMap cancels previous saves
saveButton$.pipe(
  switchMap(data => this.http.post('/api/save', data))
  // If user clicks Save twice quickly, the first POST is aborted.
  // The server may have partially processed it (payment charged, email sent).
  // Data corruption or inconsistent state.
).subscribe();
```

**2. Promises are not cancellable**

```typescript
// switchMap unsubscribes from the Observable wrapping the Promise,
// but the Promise itself keeps executing.
source$.pipe(
  switchMap(val => fetch(`/api/${val}`).then(r => r.json()))
  // fetch() completes on the server even after switchMap "cancels" it.
  // Use HttpClient (XHR-based) for true network cancellation.
).subscribe();
```

**3. Long-lived inner Observables reset on every outer emission**

```typescript
// If the inner Observable is a WebSocket connection,
// switchMap tears it down and reconnects on every outer emission.
config$.pipe(
  switchMap(cfg => webSocketConnect(cfg.url))
  // Every config change kills and recreates the WebSocket.
  // May not be desired if config changes frequently.
).subscribe();
```

## mergeMap: Run All Concurrently

`mergeMap` (also known as `flatMap`) subscribes to every inner Observable immediately as the outer emits. All inner Observables run concurrently and their outputs are interleaved.

### Text Marble Diagram

```
Outer:     --a--------b--------c--------|
               \        \        \
Inner(a):       --1--2--3--|
Inner(b):                --4--5--6--|
Inner(c):                         --7--8--|

Output:    ----1--2--3-4--5--6--7--8----|

All inner Observables run to completion.
Output order depends on timing, not emission order.
```

No inner Observable is cancelled. No emissions are ignored. Output order is non-deterministic.

### The Concurrency Parameter

`mergeMap` accepts an optional second argument that limits concurrent inner subscriptions. This is critical for production use.

```typescript
import { from, mergeMap } from 'rxjs';

const urls = generateHundredUrls();

from(urls).pipe(
  mergeMap(
    url => this.http.get(url),
    3 // At most 3 concurrent HTTP requests at any time
    // When one completes, the next URL is immediately subscribed to
  )
).subscribe(response => this.handleResponse(response));
```

| Concurrency value | Behavior |
|-------------------|----------|
| `Infinity` (default) | All inner Observables fire at once |
| `1` | Identical to `concatMap` (sequential) |
| `n` | At most `n` inner Observables active at any time |

### Angular Use Cases

```typescript
// 1. Bulk operations with concurrency limit
from(selectedIds).pipe(
  mergeMap(id => this.http.delete(`/api/items/${id}`).pipe(
    catchError(err => of({ id, error: err })) // Don't fail the batch
  ), 5) // 5 concurrent deletes
);

// 2. Fire-and-forget analytics (order doesn't matter, each must send)
userActions$.pipe(
  mergeMap(action => this.analytics.track(action).pipe(
    catchError(() => EMPTY) // Analytics failure is non-critical
  ))
);

// 3. Parallel data enrichment
this.http.get<Order[]>('/api/orders').pipe(
  switchMap(orders =>
    from(orders).pipe(
      mergeMap(order =>
        this.http.get<User>(`/api/users/${order.userId}`).pipe(
          map(user => ({ ...order, userName: user.name }))
        ),
        5 // 5 concurrent user lookups
      ),
      toArray() // Collect all enriched orders into a single array
    )
  )
);
```

### When to Use mergeMap

- Parallel writes where each must execute (bulk delete, bulk update)
- Fire-and-forget side effects (analytics, logging)
- Batch processing where order does not matter
- Parallel data enrichment with a concurrency limit

### mergeMap Gotchas

**1. Memory leaks from long-lived inner Observables**

```typescript
// BAD PRACTICE: each outer emission creates an infinite interval
source$.pipe(
  mergeMap(val => interval(1000))
  // After 10 outer emissions, 10 intervals run simultaneously.
  // Memory and CPU usage grow without bound.
).subscribe();

// FIX: ensure inner Observables complete
source$.pipe(
  mergeMap(val => interval(1000).pipe(take(5)))
).subscribe();
```

**2. Unbounded parallelism overwhelms the server**

```typescript
// BAD PRACTICE: 10,000 simultaneous HTTP requests
from(tenThousandIds).pipe(
  mergeMap(id => this.http.get(`/api/items/${id}`))
  // Browser limits connections to ~6 per domain.
  // Server may rate-limit, return 429 errors, or crash.
).subscribe();

// GOOD: limit concurrency
from(tenThousandIds).pipe(
  mergeMap(id => this.http.get(`/api/items/${id}`), 5)
).subscribe();
```

**3. Output order is not guaranteed**

```typescript
// Request for id=3 may complete before id=1
from([1, 2, 3]).pipe(
  mergeMap(id => this.http.get(`/api/items/${id}`))
).subscribe(item => {
  // Items arrive in completion order, not emission order.
  // If you need order, use concatMap.
});
```

## concatMap: Queue and Execute Sequentially

`concatMap` subscribes to each inner Observable one at a time, in the order received. A new inner Observable is not subscribed to until the previous one completes. Values are queued in arrival order.

### Text Marble Diagram

```
Outer:     --a----b--------c--------|
               \   (queued)  (queued)
Inner(a):       --1--2--|
Inner(b):               --3--4--|        (starts after Inner(a) completes)
Inner(c):                        --5--|  (starts after Inner(b) completes)

Output:    ----1--2--3--4----5------|

Strict ordering. No parallelism. No cancellation. No values lost.
```

Even though `b` and `c` arrive while `Inner(a)` is still active, they wait in a FIFO queue.

### Angular Use Cases

```typescript
// 1. Sequential saves where order matters for consistency
saveActions$.pipe(
  concatMap(action => this.http.post('/api/save', action))
  // Save #2 starts only after Save #1 completes
);

// 2. Chained dependent API calls
this.http.get<User>('/api/me').pipe(
  concatMap(user => this.http.get<Preferences>(`/api/prefs/${user.id}`)),
  concatMap(prefs => this.http.get<Theme>(`/api/themes/${prefs.themeId}`))
);

// 3. Sequential file uploads preserving order
from(files).pipe(
  concatMap((file, index) =>
    this.uploadService.upload(file).pipe(
      tap(result => console.log(`File ${index + 1} uploaded: ${result.url}`))
    )
  ),
  toArray() // Collect all results in order
);

// 4. Multi-step wizard: each step depends on the previous
from(wizardSteps).pipe(
  concatMap((step, index) =>
    this.http.post(`/api/wizard/step/${index}`, step).pipe(
      tap(() => console.log(`Step ${index + 1} complete`)),
      catchError(err => {
        console.error(`Step ${index + 1} failed:`, err);
        return EMPTY; // Skip failed step, continue with next
      })
    )
  )
);
```

### When to Use concatMap

- Sequential writes where order matters (audit logs, ordered saves)
- Chained API calls where each depends on the previous result
- Sequential animations or transitions
- File uploads that must happen in order
- Database transactions

### concatMap Gotchas

**1. Backpressure / unbounded queue growth**

```typescript
// BAD: outer emits faster than inner completes -> queue grows indefinitely
fastSource$.pipe(
  concatMap(val => slowHttpRequest(val))
  // If source emits 1000 values/second and each request takes 2 seconds,
  // the queue grows by ~1000 entries/second.
  // Memory consumption grows linearly over time.
  // The last item processes ~2000 seconds after it was emitted.
).subscribe();
```

**2. A stalled inner Observable blocks everything**

```typescript
// If Inner(a) never completes, Inner(b) and all queued items never start
source$.pipe(
  concatMap(val => webSocketMessages$)
  // WebSocket stream never completes -> queue grows forever, nothing dequeues
).subscribe();
```

Always ensure inner Observables complete in finite time when using `concatMap`. Add `timeout()` as a safety net:

```typescript
source$.pipe(
  concatMap(val =>
    this.http.get(`/api/${val}`).pipe(
      timeout(30000), // Fail after 30 seconds
      catchError(() => EMPTY) // Skip, don't block queue
    )
  )
).subscribe();
```

**3. Slower than mergeMap for independent operations**

```typescript
// 10 requests at 200ms each:
//   concatMap: 2000ms total (sequential)
//   mergeMap:  ~200ms total (parallel)
// If operations are independent and order doesn't matter, use mergeMap.
```

## exhaustMap: Ignore While Busy

`exhaustMap` subscribes to the inner Observable and silently ignores all new outer emissions until that inner Observable completes. Once complete, it accepts the next outer emission. Dropped values are NOT queued.

### Text Marble Diagram

```
Outer:     --a--b--c-----------d--e------|
               \  (ignored)     \  (ignored)
                  (ignored)
Inner(a):       ----1--2--|
Inner(d):                       ----3--|

Output:    ------1--2----------3--------|

b and c arrive while Inner(a) is active -> permanently discarded.
e arrives while Inner(d) is active -> permanently discarded.
```

Unlike `concatMap`, values dropped by `exhaustMap` are gone forever. They are not queued.

### Angular Use Cases

```typescript
// 1. Prevent double form submission
this.submitBtn$.pipe(
  exhaustMap(() => {
    this.submitting.set(true);
    return this.http.post<Order>('/api/orders', this.form.value).pipe(
      finalize(() => this.submitting.set(false))
    );
  })
);
// First click triggers POST. Rapid subsequent clicks are ignored.
// No duplicate orders.

// 2. Login: ignore repeated clicks while authenticating
this.loginBtn$.pipe(
  exhaustMap(() =>
    this.authService.login(this.credentials).pipe(
      catchError(err => {
        this.error.set(err.message);
        return EMPTY;
      })
    )
  )
);

// 3. Refresh button: ignore while data is loading
this.refreshBtn$.pipe(
  exhaustMap(() => this.http.get<Data[]>('/api/data'))
);

// 4. Grouped exhaustMap for per-item deduplication
// Prevent the same item from being processed twice simultaneously
source$.pipe(
  groupBy(item => item.id),
  mergeMap(group$ =>
    group$.pipe(
      exhaustMap(item => this.processItem(item))
    )
  )
);
```

### When to Use exhaustMap

- Form submission buttons (prevent duplicate POSTs)
- Login/logout buttons
- Data refresh triggers
- Payment processing
- Any operation where duplicate execution causes harm

### exhaustMap Gotchas

**1. Silently drops user actions without feedback**

```typescript
// BAD: user clicks "Save" but nothing happens (request in flight)
this.saveBtn$.pipe(
  exhaustMap(() => this.http.post('/api/save', data))
).subscribe();
// User thinks button is broken.

// GOOD: provide visual feedback
this.saveBtn$.pipe(
  tap(() => this.saving.set(true)),
  exhaustMap(() =>
    this.http.post('/api/save', data).pipe(
      finalize(() => this.saving.set(false))
    )
  )
).subscribe();
// Template: <button [disabled]="saving()">{{ saving() ? 'Saving...' : 'Save' }}</button>
```

**2. Not suitable for type-ahead search**

```typescript
// BAD: user's latest input is dropped
searchInput$.pipe(
  exhaustMap(term => this.http.get(`/api/search?q=${term}`))
  // User types "ang", request fires. User types "angular".
  // "angu", "angul", "angula", "angular" are all dropped.
  // User sees results for "ang", not "angular".
).subscribe();

// GOOD: use switchMap for latest-value-wins
searchInput$.pipe(
  switchMap(term => this.http.get(`/api/search?q=${term}`))
).subscribe();
```

**3. Values are permanently lost (no queue)**

Unlike `concatMap`, which queues values for later processing, `exhaustMap` discards them. If the discarded value carried meaningful data that will never be re-emitted, that data is lost.

## Comparison Table

| Aspect | `switchMap` | `mergeMap` | `concatMap` | `exhaustMap` |
|--------|-------------|------------|-------------|--------------|
| Active inners | 1 max | Unlimited (configurable) | 1 + queue | 1, no queue |
| On new outer while busy | Cancel current, start new | Start new alongside current | Queue new, wait | Ignore new |
| Values lost? | Previous inner's remaining values | None | None | Outer values during busy |
| Order guaranteed? | Latest only | No | Yes (strict) | First-come only |
| Primary use | Reads, search, filters | Parallel bulk ops | Sequential ops | Duplicate prevention |
| Risk | Cancels writes | Server overload, memory leaks | Unbounded queue | Drops user input |

## The `*All` Operators: Flattening Without Mapping

Each flattening strategy exists in two forms:
1. **`xxxMap(fn)`:** Maps a value to an Observable, then flattens. (Most common in practice.)
2. **`xxxAll()`:** Flattens an already-existing higher-order Observable. No mapping step.

| Map + Flatten | Flatten only | Equivalence |
|---------------|-------------|-------------|
| `switchMap(fn)` | `map(fn)` + `switchAll()` | Identical behavior |
| `mergeMap(fn)` | `map(fn)` + `mergeAll()` | Identical behavior |
| `concatMap(fn)` | `map(fn)` + `concatAll()` | Identical behavior |
| `exhaustMap(fn)` | `map(fn)` + `exhaustAll()` | Identical behavior |

```typescript
// These two pipelines produce identical output:

// Using switchMap
source$.pipe(
  switchMap(id => this.http.get(`/api/${id}`))
);

// Using map + switchAll
source$.pipe(
  map(id => this.http.get(`/api/${id}`)),
  switchAll()
);
```

### When to Use the `*All` Variants

Use them when you already have a higher-order Observable and need no mapping step:

```typescript
// An external API returns Observable<Observable<Message>>
const channels$: Observable<Observable<Message>> = this.chatService.activeChannels$;

// switchAll: listen to only the latest channel
channels$.pipe(
  switchAll()
).subscribe(message => this.handleMessage(message));

// mergeAll: listen to all channels simultaneously
channels$.pipe(
  mergeAll()
).subscribe(message => this.handleMessage(message));

// concatAll: listen to channels sequentially (one finishes, next starts)
channels$.pipe(
  concatAll()
).subscribe(message => this.handleMessage(message));

// mergeAll with concurrency limit
channels$.pipe(
  mergeAll(3) // At most 3 simultaneous channel subscriptions
).subscribe(message => this.handleMessage(message));
```

## Inner Observable Lifecycle Management

Understanding exactly when inner Observables are subscribed to, when they emit, and when they are torn down is essential for debugging.

### Subscription and Teardown Timing

```typescript
import { of, delay, tap, switchMap, concatMap, mergeMap, exhaustMap } from 'rxjs';

function createInner(label: string) {
  return of(`${label}-result`).pipe(
    delay(1000),
    tap({
      subscribe: () => console.log(`  SUBSCRIBED to Inner(${label})`),
      next: v => console.log(`  EMITTED from Inner(${label}): ${v}`),
      complete: () => console.log(`  COMPLETED Inner(${label})`),
      unsubscribe: () => console.log(`  UNSUBSCRIBED from Inner(${label})`)
    })
  );
}

const source$ = of('a', 'b', 'c'); // Emits synchronously
```

**switchMap with synchronous outer:**

```
source$.pipe(switchMap(v => createInner(v))).subscribe();

// SUBSCRIBED to Inner(a)
// UNSUBSCRIBED from Inner(a)       <- b arrived, a cancelled
// SUBSCRIBED to Inner(b)
// UNSUBSCRIBED from Inner(b)       <- c arrived, b cancelled
// SUBSCRIBED to Inner(c)
// EMITTED from Inner(c): c-result  <- only c runs to completion
// COMPLETED Inner(c)
```

**concatMap with synchronous outer:**

```
source$.pipe(concatMap(v => createInner(v))).subscribe();

// SUBSCRIBED to Inner(a)
// EMITTED from Inner(a): a-result
// COMPLETED Inner(a)
// SUBSCRIBED to Inner(b)           <- b starts after a completes
// EMITTED from Inner(b): b-result
// COMPLETED Inner(b)
// SUBSCRIBED to Inner(c)           <- c starts after b completes
// EMITTED from Inner(c): c-result
// COMPLETED Inner(c)
```

**mergeMap with synchronous outer:**

```
source$.pipe(mergeMap(v => createInner(v))).subscribe();

// SUBSCRIBED to Inner(a)
// SUBSCRIBED to Inner(b)           <- all subscribe immediately
// SUBSCRIBED to Inner(c)
// (after 1 second, all three emit and complete concurrently)
// EMITTED from Inner(a): a-result
// COMPLETED Inner(a)
// EMITTED from Inner(b): b-result
// COMPLETED Inner(b)
// EMITTED from Inner(c): c-result
// COMPLETED Inner(c)
```

**exhaustMap with synchronous outer:**

```
source$.pipe(exhaustMap(v => createInner(v))).subscribe();

// SUBSCRIBED to Inner(a)
// (b is IGNORED - Inner(a) is active)
// (c is IGNORED - Inner(a) is active)
// EMITTED from Inner(a): a-result
// COMPLETED Inner(a)
```

### Teardown Behavior for HTTP Requests

When `switchMap` unsubscribes from an inner Observable backed by `HttpClient`, the teardown function runs:

```typescript
// Simplified view of what HttpClient does internally
const httpObs$ = new Observable(subscriber => {
  const xhr = new XMLHttpRequest();
  xhr.open('GET', url);
  xhr.onload = () => {
    subscriber.next(xhr.response);
    subscriber.complete();
  };
  xhr.onerror = () => subscriber.error(xhr.statusText);
  xhr.send();

  // Teardown: called when unsubscribed
  return () => {
    xhr.abort(); // Actually cancels the network request
  };
});
```

You can verify this in Chrome DevTools: switch to the Network tab, trigger a `switchMap` cancellation, and observe the request shows a "cancelled" status with a red indicator.

### Custom Teardown in Your Own Observables

```typescript
function createConnection(url: string): Observable<Message> {
  return new Observable(subscriber => {
    const ws = new WebSocket(url);
    ws.onmessage = e => subscriber.next(JSON.parse(e.data));
    ws.onerror = e => subscriber.error(e);

    // Teardown: runs on unsubscribe (e.g., when switchMap cancels)
    return () => {
      console.log(`Closing WebSocket to ${url}`);
      ws.close();
    };
  });
}

const url$ = new Subject<string>();

url$.pipe(
  switchMap(url => createConnection(url))
  // When url$ emits a new URL:
  // 1. Teardown of previous connection runs (ws.close())
  // 2. New connection opens to new URL
).subscribe();
```

## Combining Flattening Operators in a Pipeline

You can chain multiple flattening operators for complex workflows. Each operator independently manages its own inner subscriptions.

```typescript
// Route change -> load user -> load posts -> load comments for each post
this.route.params.pipe(
  // switchMap: cancel everything if route changes
  switchMap(params =>
    this.userService.getById(params['id'])
  ),
  // switchMap: cancel previous posts load if user data changes
  switchMap(user =>
    this.postService.getByUser(user.id)
  ),
  // switchMap + mergeMap: load comments for all posts in parallel
  switchMap(posts =>
    from(posts).pipe(
      mergeMap(post =>
        this.commentService.getByPost(post.id).pipe(
          map(comments => ({ ...post, comments })),
          catchError(() => of({ ...post, comments: [] }))
        ),
        3 // 3 concurrent comment requests
      ),
      toArray() // Collect all posts-with-comments into one array emission
    )
  )
).subscribe(postsWithComments => {
  this.posts.set(postsWithComments);
});
```

## Error Handling with Flattening Operators

Where you place `catchError` relative to the flattening operator determines whether the pipeline survives errors.

### Errors Inside Inner Observables (Pipeline Survives)

```typescript
// catchError inside: each inner Observable handles its own errors
// The outer pipeline continues for future emissions
source$.pipe(
  mergeMap(id =>
    this.http.get(`/api/items/${id}`).pipe(
      catchError(err => {
        console.error(`Failed for ${id}:`, err);
        return EMPTY; // Skip this item, continue with others
        // or: return of(null)         -- emit a placeholder
        // or: return of({ id, error: true }) -- emit an error marker
      })
    )
  )
).subscribe(item => {
  // Receives all successful items.
  // Failed items are either skipped (EMPTY) or replaced.
});
```

### Errors Outside (Pipeline Dies)

```typescript
// catchError outside: first error kills the entire pipeline
source$.pipe(
  mergeMap(id => this.http.get(`/api/items/${id}`)),
  catchError(err => {
    console.error('Pipeline failed:', err);
    return EMPTY; // Pipeline completes here, no future emissions
  })
).subscribe();
```

### Retry Inside Inner Observables

```typescript
source$.pipe(
  switchMap(id =>
    this.http.get(`/api/${id}`).pipe(
      retry({ count: 2, delay: 1000 }), // Retry up to 2 times with 1s delay
      catchError(() => of(null))          // Final fallback after all retries
    )
  )
).subscribe();
```

**Rule of thumb:** Always place `catchError` inside the inner Observable's pipe to keep the outer pipeline alive for future emissions.

## takeUntil Placement: A Critical Gotcha

When using `takeUntil` for component cleanup, its position relative to flattening operators is critical.

```typescript
// BAD: takeUntil BEFORE the flattening operator
source$.pipe(
  takeUntil(this.destroy$),     // Completes the outer Observable
  switchMap(v => longRunning$)  // But the inner Observable may still be running!
).subscribe();
// When destroy$ emits, the outer stops emitting new values to switchMap.
// But switchMap's currently active inner subscription is NOT affected.
// The inner Observable keeps running after the component is destroyed.

// GOOD: takeUntil AFTER the flattening operator
source$.pipe(
  switchMap(v => longRunning$),
  takeUntil(this.destroy$)      // Unsubscribes from the output Observable
).subscribe();
// When destroy$ emits, the output subscription is torn down.
// This cascades: switchMap's inner subscription is also torn down.
// Everything cleans up properly.
```

This applies to all flattening operators. Always place `takeUntil` as the **last** operator in the pipe.

### Angular 21 Preferred Pattern

```typescript
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';

@Component({ /* ... */ })
export class MyComponent {
  private destroyRef = inject(DestroyRef);

  constructor() {
    // takeUntilDestroyed() in the constructor (injection context)
    source$.pipe(
      switchMap(v => this.http.get(`/api/${v}`)),
      takeUntilDestroyed()
    ).subscribe();
  }

  // OR outside the constructor with explicit DestroyRef
  ngOnInit(): void {
    source$.pipe(
      switchMap(v => this.http.get(`/api/${v}`)),
      takeUntilDestroyed(this.destroyRef)
    ).subscribe();
  }
}
```

## Performance Considerations

| Operator | Memory | CPU | Network | Browser Connections |
|----------|--------|-----|---------|---------------------|
| `switchMap` | Low (1 active inner) | Low | Aborts cancelled requests | 1 at a time |
| `mergeMap` | High (N active inners) | High | N simultaneous requests | Up to browser limit (~6/domain) |
| `mergeMap(fn, k)` | Medium (k active inners) | Medium | k simultaneous requests | k connections |
| `concatMap` | Medium (1 active + queue in memory) | Low | 1 request at a time | 1 connection |
| `exhaustMap` | Low (1 active inner) | Low | Drops excess requests | 1 at a time |

**Browser connection limits:** Most browsers allow ~6 concurrent connections per domain. Using `mergeMap` without a concurrency limit on a large list can exhaust these slots, causing other parts of your application (images, scripts, WebSocket) to stall while waiting for a connection.

**Server-side impact:** Even if the browser queues requests, the server sees them all arrive (after connection slots free up). Rate limiting, 429 responses, or server crashes can result from unbounded `mergeMap`.

## Decision Matrix

| Scenario | Operator | Why |
|----------|----------|-----|
| Type-ahead search | `switchMap` | Only latest result matters. Cancel stale requests. |
| Route param change loading data | `switchMap` | Previous data irrelevant after navigation. |
| Dropdown filter change | `switchMap` | Only latest filter matters. |
| Form submission (POST) | `exhaustMap` | Prevent duplicate submissions. Ignore clicks while busy. |
| Login request | `exhaustMap` | One login attempt at a time. |
| Refresh button | `exhaustMap` | Ignore clicks while loading. |
| Payment processing | `exhaustMap` | Never send duplicate payment requests. |
| Saving data (each save matters) | `concatMap` | Every save executes in order. |
| Chained dependent requests | `concatMap` | Request B depends on Request A's result. |
| Sequential file uploads | `concatMap` | Upload one at a time, in order. |
| Batch downloads (order irrelevant) | `mergeMap(fn, 5)` | Parallel is faster. Limit to avoid overload. |
| Fire-and-forget analytics | `mergeMap` | Each event must send. No cancellation. |
| Bulk delete operations | `mergeMap(fn, 3)` | All must execute. Parallel with concurrency limit. |
| Parallel data enrichment | `mergeMap(fn, 5)` | Independent lookups, combined afterward. |

### Master-Detail with Route Parameters

```typescript
@Component({
  template: `
    @if (product(); as product) {
      <h2>{{ product.name }}</h2>
      <p>{{ product.description }}</p>
      <app-reviews [reviews]="reviews()" />

      <app-skeleton />
    }

export class ProductDetailComponent {
  private route = inject(ActivatedRoute);
  private http = inject(HttpClient);

  // switchMap: cancel previous product load on route change
  product = toSignal(
    this.route.paramMap.pipe(
      map(params => params.get('id')!),

        this.http.get<Product>(`/api/products/${id}`).pipe(

  reviews = toSignal(
    toObservable(this.product).pipe(
      filter((p): p is Product => p !== null),

        )
```

### Bulk Operations with Progress Tracking

```typescript
@Component({
  template: `
    @if (deleting()) {
      <p>Deleted {{ progress() }} / {{ total() }}</p>
      <progress [value]="progress()" [max]="total()" />
    }
    <button (click)="deleteSelected()" [disabled]="deleting()">
      Delete Selected
    </button>
  `
})
export class BulkDeleteComponent {
  private http = inject(HttpClient);

  progress = signal(0);
  total = signal(0);
  deleting = signal(false);

  deleteSelected(): void {
    const ids = this.selectedIds(); // string[]
    this.total.set(ids.length);
    this.progress.set(0);
    this.deleting.set(true);

    from(ids).pipe(
      mergeMap(id =>
        this.http.delete(`/api/items/${id}`).pipe(
          map(() => ({ id, success: true as const })),
          catchError(() => of({ id, success: false as const }))
        ),
        3 // 3 concurrent deletions
      ),
      tap(() => this.progress.update(p => p + 1)),
      toArray(),
      finalize(() => this.deleting.set(false))
    ).subscribe(results => {
      const failed = results.filter(r => !r.success);
      if (failed.length > 0) {
        this.notification.warn(`${failed.length} items failed to delete`);
      }
    });
  }
}
```

### Multi-Step Wizard with Sequential Saves

```typescript
@Component({ /* ... */ })
export class WizardComponent {
  private http = inject(HttpClient);
  private stepSave$ = new Subject<{ step: number; data: unknown }>();

  savingStep = signal<number | null>(null);
  completedSteps = signal(new Set<number>());
  stepError = signal<{ step: number; message: string } | null>(null);

  constructor() {
    // concatMap: save each step in strict order, wait for completion
    this.stepSave$.pipe(
      concatMap(({ step, data }) => {
        this.savingStep.set(step);
        return this.http.post(`/api/wizard/step/${step}`, data).pipe(
          tap(() => this.completedSteps.update(s => new Set([...s, step]))),
          catchError(err => {
            this.stepError.set({ step, message: err.message });
            return EMPTY; // Skip failed step, process next in queue
          }),
          finalize(() => this.savingStep.set(null))
        );
      }),
      takeUntilDestroyed()
    ).subscribe();
  }

  saveStep(step: number, data: unknown): void {
    this.stepSave$.next({ step, data });
  }
}
```

### Login with exhaustMap and Full State Management

```typescript
@Component({
  template: `
    <form (ngSubmit)="login()">
      <input [(ngModel)]="username" name="username" />
      <input [(ngModel)]="password" name="password" type="password" />
      <button type="submit" [disabled]="loggingIn()">
        {{ loggingIn() ? 'Logging in...' : 'Log In' }}
      </button>
      @if (error()) {
        <p class="error">{{ error() }}</p>
      }
    </form>
  `
})
export class LoginComponent {
  private authService = inject(AuthService);
  private router = inject(Router);
  private loginAction$ = new Subject<{ username: string; password: string }>();

  loggingIn = signal(false);
  error = signal<string | null>(null);
  username = '';
  password = '';

  constructor() {
    this.loginAction$.pipe(
      tap(() => {
        this.loggingIn.set(true);
        this.error.set(null);
      }),
      exhaustMap(credentials =>
        this.authService.login(credentials).pipe(
          catchError(err => {
            this.error.set(err.message);
            return EMPTY; // Don't propagate, let user try again
          }),
          finalize(() => this.loggingIn.set(false))
        )
      ),
      takeUntilDestroyed()
    ).subscribe(token => {
      this.authService.storeToken(token);
      this.router.navigate(['/dashboard']);
    });
  }

  login(): void {
    this.loginAction$.next({
      username: this.username,
      password: this.password
    });
  }
}
```

## Testing Flattening Operators with Marble Syntax

```typescript
import { TestScheduler } from 'rxjs/testing';

describe('Flattening operators', () => {
  let scheduler: TestScheduler;

  beforeEach(() => {
    scheduler = new TestScheduler((actual, expected) => {
      expect(actual).toEqual(expected);
    });
  });

  it('switchMap cancels previous inner on new outer emission', () => {
    scheduler.run(({ cold, expectObservable }) => {
      const outer =   cold('--a------b------|');
      const innerA =  cold('  ---1--2--3--|');
      const innerB =  cold('         ---4--5--|');
      const expected =      '-----1---4--5---|';
      // Values 2 and 3 from innerA are never emitted (cancelled when b arrives)

      const result$ = outer.pipe(
        switchMap(val => val === 'a' ? innerA : innerB)
      );

      expectObservable(result$).toBe(expected);
    });
  });

  it('concatMap queues inner Observables', () => {
    scheduler.run(({ cold, expectObservable }) => {
      const outer =   cold('--a-b---|');
      const innerA =  cold('  --1--|');
      const innerB =  cold('       --2--|');
      const expected =      '----1---2--|';
      // b is queued until innerA completes

      const result$ = outer.pipe(
        concatMap(val => val === 'a' ? innerA : innerB)
      );

      expectObservable(result$).toBe(expected);
    });
  });

  it('mergeMap runs inner Observables concurrently', () => {
    scheduler.run(({ cold, expectObservable }) => {
      const outer =   cold('--a-b---|');
      const innerA =  cold('  ----1--|');
      const innerB =  cold('    ----2--|');
      const expected =      '------1-2--|';
      // Both inners run at the same time

      const result$ = outer.pipe(
        mergeMap(val => val === 'a' ? innerA : innerB)
      );

      expectObservable(result$).toBe(expected);
    });
  });

  it('exhaustMap ignores emissions while inner is active', () => {
    scheduler.run(({ cold, expectObservable }) => {
      const outer =   cold('--a-b------c---|');
      const innerA =  cold('  ----1--|');
      const innerC =  cold('           --2--|');
      const expected =      '------1-----2--|';
      // b is ignored because innerA is still active

      const result$ = outer.pipe(
        exhaustMap(val => {
          if (val === 'a') return innerA;
          if (val === 'c') return innerC;
          return innerA; // b never reaches here
        })
      );

      expectObservable(result$).toBe(expected);
    });
  });
});
```

## Refactoring Patterns

### Pattern 1: Sequential Dependent Calls

```typescript
// BAD PRACTICE: nested subscribe pyramid
this.route.params.subscribe(params => {
  this.http.get(`/api/users/${params['id']}`).subscribe(user => {
    this.http.get(`/api/orders?userId=${user.id}`).subscribe(orders => {
      this.orders = orders;
    });
  });
});

// GOOD: flat pipeline
this.route.params.pipe(
  switchMap(params => this.http.get<User>(`/api/users/${params['id']}`)),
  concatMap(user => this.http.get<Order[]>(`/api/orders?userId=${user.id}`)),
  takeUntilDestroyed()
).subscribe(orders => this.orders.set(orders));
```

### Pattern 2: Parallel Independent Calls

```typescript
// BAD PRACTICE: sequential when it could be parallel
this.http.get('/api/user').subscribe(user => {
  this.user = user;
  this.http.get('/api/settings').subscribe(settings => {
    this.settings = settings;
  });
});

// GOOD: parallel with forkJoin
forkJoin({
  user: this.http.get<User>('/api/user'),
  settings: this.http.get<Settings>('/api/settings')
}).subscribe(({ user, settings }) => {
  this.user.set(user);
  this.settings.set(settings);
});
```

### Pattern 3: Conditional Inner Observable

```typescript
// BAD PRACTICE: conditional subscribe inside subscribe
this.trigger$.subscribe(value => {
  if (value > threshold) {
    this.http.post('/api/alert', { value }).subscribe();
  }
});

// GOOD: filter + flattening operator
this.trigger$.pipe(
  filter(value => value > threshold),
  exhaustMap(value =>
    this.http.post('/api/alert', { value }).pipe(
      catchError(() => EMPTY)
    )
  ),
  takeUntilDestroyed()
).subscribe();
```

### Pattern 4: Event Stream with Data Loading

```typescript
// BAD PRACTICE: subscribe in subscribe from event
fromEvent(button, 'click').subscribe(() => {
  this.http.get('/api/data').subscribe(data => {
    this.data = data;
  });
});

// GOOD: exhaustMap (or switchMap depending on requirements)
fromEvent(button, 'click').pipe(
  exhaustMap(() => this.http.get<Data>('/api/data')),
  takeUntilDestroyed(this.destroyRef)
).subscribe(data => this.data.set(data));
```

## Common Mistakes Summary

| Mistake | Consequence | Fix |
|---------|-------------|-----|
| `switchMap` for POST/PUT/DELETE | Write operations get cancelled mid-flight | Use `concatMap` or `exhaustMap` |
| `mergeMap` without concurrency limit | Server overload, exhaust browser connections | `mergeMap(fn, 5)` |
| `concatMap` when order is irrelevant | Needlessly slow sequential execution | Use `mergeMap` for parallel |
| `exhaustMap` for search/type-ahead | User's latest input is silently dropped | Use `switchMap` |
| Nested subscribes | Memory leaks, race conditions, unreadable code | Refactor to flattening operators |
| `takeUntil` before flattening operator | Inner Observable outlives the component | Place `takeUntil` last in pipe |
| `catchError` outside inner Observable | One failure kills the entire pipeline | Place `catchError` inside inner pipe |
| `mergeMap` with infinite inner Observables | Unbounded memory and CPU growth | Use `switchMap` or add `take`/`takeUntil` to inner |
| `switchMap` with `fetch()` / Promises | Promise keeps executing despite "cancellation" | Use `HttpClient` (XHR-based) for true cancel |
| `concatMap` with never-completing inner | Queue grows forever, pipeline appears frozen | Ensure inner Observables complete; add `timeout()` |
| No visual feedback with `exhaustMap` | Users think the button is broken | Disable button or show spinner while busy |

## Quick Reference

```
switchMap  = "Cancel previous, subscribe to latest"
             Use for: search, route params, reads where only latest matters
             Active inners: 1

mergeMap   = "Subscribe to all, run in parallel"
             Use for: batch ops, fire-and-forget, writes that all matter
             Active inners: unlimited (or limited with 2nd arg)

concatMap  = "Queue and process one at a time, in order"
             Use for: ordered operations, sequential writes, dependent chains
             Active inners: 1 + unbounded queue

exhaustMap = "Ignore new emissions while busy"
             Use for: form submit, login, refresh, prevent double execution
             Active inners: 1, no queue (values permanently dropped)

xxxAll     = Same strategies without the mapping step
             Use when you already have Observable<Observable<T>>
```
